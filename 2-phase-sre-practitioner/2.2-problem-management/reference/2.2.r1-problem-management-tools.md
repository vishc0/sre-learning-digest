# Problem Management Tools Reference

> **Practical, not marketing.** Every tool has trade-offs. The decision is not "which tool is best" — it is "which tool fits our team size, compliance requirements, and existing tooling stack." Tool fit failures cost more to unwind than tool selection takes. Read the weakness column before the strength column.

---

## SECTION 1: TOOL SELECTION CRITERIA

> These are the capabilities you need the tool to support. Map each against your candidate tools before selecting.

| Capability Needed | What to Look For in a Tool |
|---|---|
| **Problem record lifecycle management** | Configurable status workflow (New → Under Investigation → Root Cause Identified → Fix In Progress → Resolved → Closed); SLA clock per priority; audit history that cannot be edited |
| **Known Error Database (KEDB)** | Searchable by symptom (not just title); linkable to incident records; structured fields (not just free text); version history; expiry/review date enforcement |
| **SLA tracking and alerting** | Automated alerts before SLA breach (not just at breach); priority-specific SLA configuration; SLA pause/hold for approved reasons |
| **CI/CMDB linkage** | Ability to link problem records to specific configuration items; view all problems linked to a CI; CI change history visible from problem record |
| **Incident integration** | Bi-directional link between problem records and incident records; ability to see "all incidents that contributed to this problem" and "all problems linked to this incident" |
| **Reporting and metrics** | Out-of-the-box reports for backlog age, cycle time, recurrence — or the query capability to build them; scheduler for regular distribution |
| **RCA document management** | Storage or integration for RCA documents; approval workflow; versioning; linkage to problem record |
| **Change/fix tracking** | Link from problem record to change request; visibility into change status from problem record |
| **Access control** | Ability to restrict sensitive problem records (security incidents) while keeping others broadly readable; role-based permissions |
| **Search and discoverability** | Full-text search across all KEDB content; symptom-based discovery; works at 2am with minimal clicks |

## SECTION 2: TOOL PROFILES

---

### ServiceNow (ITSM)

**Problem Management Module Capabilities**:
- Full problem record lifecycle with configurable status workflow and SLA clock per priority tier
- Native Known Error Database (separate table, linked to problem records, searchable)
- CMDB-CI integration: link any problem record to one or more CIs; view the CI's problem history from the CI record
- Automated SLA breach alerting: configure threshold alerts (e.g., alert at 80% of SLA consumed)
- Workflow automation: auto-assign problem owners based on CI ownership, auto-notify stakeholders on status change
- Compliance reporting: built-in audit trail, exportable for SOC2/ISO27001 evidence packages

**Strengths**:
- Full ITIL alignment — problem management module implements the ITIL problem lifecycle without customization
- CMDB-CI linkage is the best in class — no other tool maps problem records to infrastructure topology as well
- Automated SLA alerting prevents surprises — the tool tells you before you breach, not after
- Compliance evidence packaging — audit-ready reports exist out of the box
- Approval workflows for RCA sign-off are configurable without custom code

**Weaknesses**:
- Expensive — licensing plus implementation plus admin overhead is material cost; meaningful investment for a mid-market org
- Heavy admin overhead — configuration complexity means a dedicated ServiceNow admin is not optional; without one, the tool drifts from configured state
- Poor UX for engineers — the interface was designed for ITSM practitioners, not software engineers; engineers avoid it, which means data quality suffers
- Integration complexity — integrating with modern dev tools (GitHub, PagerDuty) requires middleware or out-of-the-box connectors that often need tuning
- Upgrade risk — major version upgrades require regression testing of all customizations

**Best for**: Enterprise organizations with >200 engineers, dedicated ITSM admin, compliance-driven environment (SOC2, ISO27001, PCI-DSS, FedRAMP). Organizations that are already on ServiceNow for incident management — adding problem management to an existing ServiceNow instance is lower-cost than the initial deployment.

**Problem Management specific config notes**:
- Known Error database is a separate table (`kb_knowledge` or `problem_kb_entry` depending on version) — ensure it is linked to your problem table, not a standalone knowledge base
- SLA tables must be configured per priority tier — default SLAs are not aligned to ITIL thresholds
- Problem discovery reports require custom report configuration — out-of-the-box dashboards are limited; invest in report setup early

### Jira Service Management (Atlassian)

**Problem Management Capabilities**:
- Native ITIL-aligned problem record type (available in JSM Premium tier)
- Problem lifecycle tracking with customizable workflows
- KEDB via Confluence: known errors documented as Confluence pages, linked to problem records via Jira-Confluence integration
- RCA documentation: Confluence pages linked to Jira problem records; Confluence provides version history and approval via page restrictions
- SLA configuration: SLA timers per priority configurable; built-in SLA reports
- Jira Software integration: problem records link directly to Jira epics/stories for fix tracking; engineers see problem context in their Jira backlog

**Strengths**:
- Developer-friendly: engineers already live in Jira; problem records visible in the same tool as their sprint boards; no tool-switching for fix tracking
- Atlassian stack integration: seamless connection between JSM (operations) and Jira Software (engineering) and Confluence (knowledge) with no middleware
- Lower administrative overhead than ServiceNow: configuration is more approachable; Jira admins are more common than ServiceNow admins
- Good for engineering-led orgs: prioritization decisions are visible to both SRE and engineering in the same interface
- PagerDuty / Opsgenie integration: alert-to-incident-to-problem linking works with minimal config when on the Atlassian stack

**Weaknesses**:
- KEDB is less mature: Confluence-based KEDB lacks structured field enforcement; anyone can edit any page; no expiry enforcement without add-ons
- SLA reporting requires configuration: out-of-the-box SLA reports exist but are limited; serious SLA tracking requires custom dashboards in JSM or Eazybi
- CMDB is limited: Assets (formerly Insight) provides CMDB capability but is significantly less mature than ServiceNow's CMDB; CI-to-problem linkage is functional but not as rich
- Governance controls weaker: approval workflows for RCA sign-off require add-ons or page-level workarounds in Confluence
- Compliance evidence packaging requires manual effort: no equivalent to ServiceNow's compliance report exports

**Best for**: Engineering-led organizations with existing Jira footprint. Teams where engineering velocity and developer experience matter more than ITSM governance overhead. Organizations in the 50–500 engineer range without strict compliance mandates for ITSM tooling specifically.

**Problem Management specific config notes**:
- Use JSM Premium for native ITIL problem record type — the free tier does not support it
- Establish a Confluence space specifically for KEDB with a template page; enforce the template via a page blueprint
- Link problem record "Fix in Progress" status to a Jira Software epic using the Jira-Confluence link — engineers get visibility, SREs track progress without duplicating updates
- Configure SLA timers on the problem issue type for each priority tier before going live

### PagerDuty

**Problem Management Approach**:
PagerDuty is not a problem record management tool. It is the best tool available for **identifying problem candidates** from alert and incident patterns. Use it upstream of your problem management tool, not instead of it.

- **Service analytics**: PagerDuty aggregates incidents by service over time, showing frequency, mean time to resolve, and incident volume trends — this is your proactive problem identification feed
- **Incident clustering**: PagerDuty identifies incidents that share alert patterns, providing the "related incidents" data you need to confirm a recurring problem
- **Analytics dashboard**: Provides incident frequency by service, MTTR trends, and responder load — all inputs to problem prioritization
- **Workflow automation**: Can auto-create problem records in ServiceNow or Jira when incident count for a service exceeds a configurable threshold in a time window

**Strengths**:
- Alert-to-incident correlation is best in class — PagerDuty knows which alerts are firing and which incidents they belong to better than any ITSM tool
- Proactive problem identification: service analytics surfaces recurring patterns before a human reviews incident history
- On-call integration: when a KEDB entry exists, PagerDuty can surface it to the on-call engineer during incident response if integrated with a knowledge tool
- Incident volume trends: clear signal for frequency scoring in the priority matrix

**Weaknesses**:
- Not a problem record management tool: no problem lifecycle, no SLA tracking, no KEDB, no RCA workflow — must pair with ServiceNow or Jira for tracking
- Requires integration work: getting PagerDuty analytics to auto-create problem records requires integration configuration
- Licensing cost for analytics features: advanced analytics (PagerDuty Analytics module) is an add-on cost

**Best for**: Proactive problem identification feed. Use PagerDuty analytics as the source that tells you "this service has had 8 incidents in 30 days — open a problem record." Then manage the problem record in ServiceNow or JSM.

### Opsgenie (Atlassian)

**Problem Management Approach**:
Similar role to PagerDuty — alert aggregation and incident correlation — but Atlassian-native. Better choice when the organization is on the Atlassian stack.

- **Alert correlation**: Groups related alerts into a single incident, reducing noise and making pattern identification easier
- **Jira SM integration**: Native push of incident data to Jira Service Management problem records; less configuration than PagerDuty-to-Jira integration
- **Analytics**: Incident frequency and MTTR reporting similar to PagerDuty; useful for problem candidate identification
- **Escalation policies**: More configurable than PagerDuty at the same price point; important for problem manager notification on SLA approach

**Strengths**:
- Atlassian-native: tighter Jira SM and Confluence integration than PagerDuty
- Lower cost: Opsgenie pricing is more favorable than PagerDuty for organizations already on the Atlassian stack (included in some Atlassian bundles)
- Incident-to-problem handoff: simpler workflow for organizations already on Jira SM

**Weaknesses**:
- Analytics depth is less mature than PagerDuty: the incident analytics and trend reporting capabilities are not as advanced
- Smaller ecosystem: fewer pre-built integrations with monitoring tools compared to PagerDuty
- UI/UX: more complex to configure than PagerDuty for equivalent functionality

**Best for**: Organizations already on the Atlassian stack (Jira SW + Jira SM + Confluence) who want alert correlation feeding into Jira SM problem records. Avoids the cross-platform integration overhead of PagerDuty-to-Jira.

### Confluence (Atlassian)

**Use for KEDB when**:
- The organization needs a KEDB quickly without heavy tooling investment
- KEDB entries are primarily documentation (not transactional records requiring workflows)
- Team is already on Confluence and engineers will actually look there during incidents
- Compliance requirements do not mandate a structured ITSM-native KEDB

**Setup approach**:
- Create a dedicated Confluence space: `KEDB` or `Known Error Database`
- Create a page template using the Known Error Entry template (see `templates/known-error-entry-template.md`) — enforce via Confluence blueprint
- Tag pages by: service/CI name, symptom keywords, priority of parent problem
- Configure Confluence search to index KEDB space with higher weighting — test that symptom searches return KEDB results
- Add KEDB space to the incident response bookmarks in your runbook header

**Strengths**:
- Accessible to all: no additional licensing cost if org is on Confluence; no access barrier during incidents
- Excellent search: Confluence full-text search works well if entry titles use symptom language
- Version history: all edits tracked; can see what the entry said at any point in time
- Easy linking: Confluence pages link to Jira problem records, Jira incidents, other Confluence pages — creates a navigable knowledge graph
- Low barrier to update: engineers can update a Confluence page without ITSM training; lowers the cost of keeping KEDB current

**Weaknesses**:
- No structured field enforcement: the template is a convention, not a constraint; entries drift from template over time without governance
- No SLA tracking: no way to enforce review dates or expiry without manual process or add-ons
- No status workflow: "Active vs. Archived" is a manual label; no system-enforced state transition
- No usage logging: the KEDB entry template includes a usage log, but updating it requires discipline — no system tracks when a page is accessed during an incident
- Governance gap: anyone can archive, edit, or delete a KEDB entry — requires naming a KEDB owner and periodic audits

**Best for**: Small organizations (<50 engineers) without ITSM compliance mandates. Engineering teams that want a fast, lightweight KEDB without tool procurement overhead. Organizations using Jira SM where KEDB entries are Confluence pages linked to Jira problem records.

## SECTION 4: TOOL SELECTION DECISION GUIDE

> Match your organization profile to the recommended stack. These are starting points — adjust based on existing tooling, team preferences, and budget constraints.

### Profile A: Small Organization, No Formal Compliance Requirement

**Size**: < 50 engineers  
**Compliance**: No SOC2, ISO27001, or equivalent ITSM-specific mandates  
**Budget**: Minimal tooling investment  
**Recommended stack**: Jira Service Management + Confluence KEDB

Rationale: JSM provides sufficient problem lifecycle management with lower admin overhead than ServiceNow. Confluence KEDB gives engineers a familiar, accessible knowledge store. PagerDuty or Opsgenie (whichever is already in use) feeds problem candidates.

**What to configure before going live**:
- JSM: problem issue type with required fields and SLA timers per priority
- Confluence: KEDB space with blueprint template; review date reminder workflow (manual or Confluence automation)
- Jira integration: link problem records to Jira Software epics for fix tracking

---

### Profile B: Mid-Market, SOC2 or ISO27001 Required

**Size**: 50–300 engineers  
**Compliance**: SOC2 Type II or ISO27001 — must demonstrate formal problem management process with evidence  
**Budget**: Moderate — tool investment justified by compliance requirement  
**Recommended stack**: Jira Service Management (Premium) + PagerDuty/Opsgenie for detection, OR ServiceNow (if already deployed for incident management)

Rationale: JSM Premium with full configuration (SLA enforcement, approval workflows, audit logging) satisfies SOC2/ISO27001 at lower cost than ServiceNow for mid-market. If ServiceNow is already deployed for incident management, extend it to problem management — adding a module to an existing deployment is significantly cheaper than a new tool procurement.

**Key configuration requirements**:
- Problem records must have immutable audit trail
- SLA tracking with breach alerting must be active and configured to priority matrix targets
- RCA approval workflow must produce a datable, signable artifact (Confluence page with approver names and dates is sufficient for most compliance audits)
- Evidence export capability for auditor review must be validated before audit cycle

---

### Profile C: Enterprise, Strict ITIL Compliance

**Size**: > 300 engineers  
**Compliance**: SOC2, ISO27001, potentially PCI-DSS, FedRAMP, or industry-specific mandates  
**Budget**: Significant — ITSM is treated as enterprise infrastructure  
**Recommended stack**: ServiceNow

Rationale: At enterprise scale with strict ITIL process requirements and compliance mandates, ServiceNow's full CMDB integration, native ITIL problem lifecycle, compliance reporting, and governance controls justify the cost and admin overhead. The key investment is dedicated ServiceNow admin capacity — without it, the tool degrades rapidly.

**Critical success factors**:
- Dedicated ServiceNow admin assigned before go-live, not after configuration debt accumulates
- Problem management module configuration reviewed by a ServiceNow ITSM practitioner, not just a generalist admin
- Engineer-facing interfaces supplemented: consider building a Confluence-side facing KEDB that pulls data from ServiceNow, to give engineers search access without requiring them to navigate ServiceNow's interface

---

### Profile D: Engineering-Led Organization, Speed Priority

**Size**: Any  
**Compliance**: Lightweight or none for ITSM tooling specifically  
**Priority**: Developer experience, speed of adoption, integration with engineering workflow  
**Recommended stack**: Jira Service Management + Jira Software + PagerDuty + Confluence

Rationale: Engineers adopt tools they already use. Problem management that lives in Jira — the same interface as sprint planning — gets used. Problem management that requires logging into a separate ITSM portal with different UX conventions does not get used. Imperfect data in a tool engineers use beats perfect data in a tool they avoid.

**Trade-off to accept**: Compliance evidence packaging and governance controls are weaker. If compliance requirements grow, plan for a migration to more structured tooling — but do not let compliance requirements drive early-stage tool decisions before those requirements actually apply.

## ANTI-PATTERNS IN TOOL SELECTION

| Anti-Pattern | Why It Fails | Better Approach |
|---|---|---|
| Selecting ServiceNow because "that's the enterprise standard" without dedicated admin | Tool degrades within 6 months; engineers bypass it; data quality collapses | Only deploy ServiceNow with committed admin headcount |
| Using Confluence as KEDB with no governance | Entries become stale; on-call engineers stop trusting it; usage drops to zero | Assign a KEDB owner; quarterly audit; automated review-date reminders |
| Expecting PagerDuty to replace a problem record tool | PagerDuty identifies problems; it cannot manage them | Use PagerDuty as detection feed; use JSM or ServiceNow for tracking |
| Deploying all tools simultaneously | Integration complexity overwhelms the team; nothing works well | One tool at a time; integrate detection tool (PagerDuty) to tracking tool (JSM/ServiceNow) after each is stable independently |
| Letting engineers choose their own tooling per-team | Problem management requires cross-team data correlation; siloed tools make this impossible | One problem record system organization-wide; one KEDB; tooling diversity in dev tooling is fine, not in operational record systems |
