# Problem Management Metrics and KPIs

> **How to use this reference**: Each metric is defined precisely enough to implement. Disagreements about whether a metric is "green" or "red" are almost always disagreements about definition — this document eliminates that ambiguity. Use the Splunk query samples and ServiceNow report descriptions as starting points; adjust field names and index names for your environment.
>
> **Measurement principle**: Never report a metric you cannot trace back to a raw data source. If the measurement method section says "count records in ServiceNow," you must be able to pull that count independently. Metrics that depend on manual data entry or judgment are unreliable.

## METRIC DETAIL

---

### Metric 1: MTBI — Mean Time Between Incidents

**Definition**: The average time between occurrences of incidents linked to the same problem record. Measures how frequently a known structural issue manifests as a customer-impacting event.

**Why it matters**: A high MTBI means the problem is not actively harming customers at a high rate. A declining MTBI means the failure is accelerating — the structural condition is worsening or triggering more easily.

**Formula**:
```
MTBI (per problem) = Total observation period ÷ Number of linked incidents

Portfolio-level MTBI = Average of per-problem MTBIs across all active problems
Service-level MTBI = Average MTBI for all problems affecting a specific service
```

**Target**:
- Per-problem P1: MTBI should be increasing (recurrences decreasing) within 30 days of problem opening
- Per-problem P2: MTBI increasing or stable within 60 days
- Portfolio-level: improving quarter-over-quarter

**Measurement method**: Query ITSM for all incident records linked to each problem record. Calculate time delta between each sequential pair of incidents. Average across the set.

**Splunk query (if incidents are indexed)**:
```splunk
index=itsm sourcetype=incident_data problem_id="PR-*" status="resolved"
| eval incident_time=strptime(resolved_at, "%Y-%m-%dT%H:%M:%S")
| sort problem_id, incident_time
| streamstats window=2 range(incident_time) as delta by problem_id
| stats avg(delta) as avg_mtbi_seconds count as incident_count by problem_id
| eval avg_mtbi_hours=round(avg_mtbi_seconds/3600,1)
| table problem_id, incident_count, avg_mtbi_hours
```

**ServiceNow report**: Incident table → filter by problem record ID (not null) → group by problem record → calculate time between sequential resolution times.

| Threshold | Value | Color |
|---|---|---|
| Green | MTBI improving vs. prior 30 days OR within SLA recovery targets | Green |
| Amber | MTBI stable (no improvement) for 30 days on P1, 60 days on P2 | Amber |
| Red | MTBI declining (recurrences accelerating) on any P1 or P2 | Red |

**Metric health — what drives it to Red**:
- Workaround is ineffective or inconsistently applied — incidents keep recurring at same rate
- Fix delivery is delayed — problem stays open, recurrences continue
- Environmental conditions are changing (load increase, dependency behavior change) — problem is worsening without a fix

### Metric 2: Problem Backlog Size

**Definition**: Count of open problem records at a given point in time, broken down by priority. "Open" = any status except Closed.

**Why it matters**: Backlog size reveals program capacity. A growing P1 backlog means the program is not resolving critical issues faster than they are identified. A growing P3/P4 backlog with stable P1/P2 backlog is a low-risk signal of normal operations.

**Formula**:
```
Backlog Size = COUNT(problem records where status != 'Closed')
Segmented by: Priority (P1, P2, P3, P4)
```

**Target** (calibrate to your organization scale):
- P1: ≤ 3 open at any time (more than 3 indicates either triage quality problem or fix delivery failure)
- P2: ≤ 10 open
- P3/P4: No absolute target; track trend — growing backlog = new problems outpacing closure rate

**Measurement method**: ITSM report — problem table filtered to open status, grouped by priority. Pull weekly.

**Splunk query**:
```splunk
index=itsm sourcetype=problem_data status!="Closed"
| stats count by priority
| sort priority
```

**ServiceNow report**: Problem table → exclude Closed → group by Priority → show count.

| Threshold | Value | Color |
|---|---|---|
| Green | P1 ≤ 3, P2 ≤ 10, P3/P4 stable or decreasing | Green |
| Amber | P1 = 4–5, OR P2 > 10, OR any priority growing >20% week-over-week | Amber |
| Red | P1 > 5, OR any priority growing >50% month-over-month | Red |

**Metric health — what drives it to Red**:
- Reactive problem opening rate exceeds closure rate — every major incident spawns a problem, none get closed
- Fix delivery rate is low — problems pile up because engineering is not delivering fixes
- Triage quality is poor — too many P3/P4 elevated to P2/P1 without proper scoring

### Metric 3: Problem Backlog Age

**Definition**: Average and maximum time a problem record has been in Open status, by priority. Age starts at problem record creation date.

**Why it matters**: A large backlog is manageable if problems are aging within SLA. A small backlog with chronic age violations is a program failure — old problems are not being resolved.

**Formula**:
```
Age = (Current date - Date problem record opened)
Average Age = AVG(Age) for all open problems, grouped by priority
Maximum Age = MAX(Age) for all open problems, grouped by priority
```

**Target (SLA thresholds)**:
- P1 Average Age: ≤ 30 days
- P1 Maximum Age: ≤ 60 days (>60 days requires Service Owner escalation)
- P2 Average Age: ≤ 60 days
- P2 Maximum Age: ≤ 90 days
- P3 Average Age: ≤ 90 days
- P4 Average Age: ≤ 180 days

**Measurement method**: ITSM — for each open problem, calculate current date minus opened date. Group by priority, calculate average and max.

**Splunk query**:
```splunk
index=itsm sourcetype=problem_data status!="Closed"
| eval age_days=round((now()-strptime(opened_at,"%Y-%m-%dT%H:%M:%S"))/86400,0)
| stats avg(age_days) as avg_age max(age_days) as max_age count by priority
| sort priority
```

| Threshold | Value | Color |
|---|---|---|
| Green | Average age within SLA targets for all priorities | Green |
| Amber | Any priority average age within 20% of SLA limit | Amber |
| Red | Any priority average age exceeds SLA limit, OR any P1 > 60 days old | Red |

**Metric health — what drives it to Red**:
- Problems opened but not actively investigated — no owner activity logged
- Engineering capacity consumed by incident response, no bandwidth for fix delivery
- Root cause identification stalled — investigation in progress for > SLA period indicates the wrong method is being used or evidence is missing

### Metric 4: Fix Delivery Rate

**Definition**: Percentage of problems with committed fix delivery dates where the fix was deployed on or before the committed date. Measures execution reliability of the problem management program.

**Formula**:
```
Fix Delivery Rate = 
  (Problems with committed dates where fix deployed ≤ committed date)
  ÷
  (All problems with committed dates in the measurement period)
  × 100
```

**Target**: ≥ 85% (< 85% signals systemic delivery failure, not one-off slippage)

**Measurement method**: ITSM — problems closed in measurement period with a committed delivery date. Compare actual fix deployment date (or linked change record implementation date) against the committed date. Count on-time vs. late.

**ServiceNow report**: Problem table → Closed in date range → target_date (fix commitment) field populated → compare actual_close_date vs. target_date → % where actual ≤ target.

| Threshold | Value | Color |
|---|---|---|
| Green | ≥ 85% of committed dates met | Green |
| Amber | 70–84% | Amber |
| Red | < 70% | Red |

**Metric health — what drives it to Red**:
- Committed dates are set without engineering input — commitments are aspirational rather than grounded
- Dependencies not identified at commitment time — cross-team coordination causes delays
- P1/P2 incidents consuming engineering capacity during fix delivery sprints
- Complexity was underestimated — score 2 problems take score 4 effort

### Metric 5: RCA Completion Rate

**Definition**: Percentage of problems that required an RCA (P1 mandatory; P2 default; P3/P4 by judgment) where the RCA was completed and approved within the SLA. Measures the program's knowledge generation discipline.

**Formula**:
```
RCA Completion Rate = 
  (Problems requiring RCA with RCA completed within SLA)
  ÷
  (All problems requiring RCA closed in measurement period)
  × 100

SLA: P1 RCA complete within 5 business days of problem opening
     P2 RCA complete within 10 business days
```

**Target**: ≥ 90% within SLA (missing RCA = the same problem can and will recur)

**Measurement method**: ITSM — problems closed in period. Filter to those requiring RCA (P1/P2 or flagged P3). Check RCA status = Approved AND RCA approval date ≤ (problem open date + SLA days). Calculate %.

**Splunk query**:
```splunk
index=itsm sourcetype=problem_data status="Closed" priority IN ("P1","P2")
| eval rca_required=1
| eval rca_sla_days=case(priority=="P1",5,priority=="P2",10)
| eval opened_epoch=strptime(opened_at,"%Y-%m-%dT%H:%M:%S")
| eval rca_due=opened_epoch+(rca_sla_days*86400)
| eval rca_approved_epoch=strptime(rca_approved_at,"%Y-%m-%dT%H:%M:%S")
| eval on_time=if(rca_approved_epoch<=rca_due,1,0)
| stats sum(on_time) as on_time_count count as total by priority
| eval rate=round(on_time_count/total*100,1)
```

| Threshold | Value | Color |
|---|---|---|
| Green | ≥ 90% within SLA | Green |
| Amber | 75–89% | Amber |
| Red | < 75%, OR any P1 RCA missing at problem closure | Red |

**Metric health — what drives it to Red**:
- RCA sessions not scheduled within 48 hours of problem opening — delays compound
- Facilitator role not clearly assigned — no one owns the scheduling
- RCA participants unavailable — no backup identified, session postponed repeatedly
- Approval process creates bottleneck — approvers not reviewing on time

### Metric 6: Workaround Count

**Definition**: Number of active KEDB entries that describe a workaround but have no permanent fix deployed. These represent ongoing operational risk — each is a structural condition that requires engineer intervention every time it triggers.

**Formula**:
```
Workaround Count = COUNT(KEDB entries where status = 'Active' AND permanent_fix_status != 'Deployed')

Segmented by:
  - By priority (P1 workarounds are highest risk)
  - Age of KEDB entry (workarounds > 90 days old without fix progress are chronic)
```

**Target**:
- P1 workarounds without committed fix date: 0 (any P1 workaround must have a committed fix date)
- P2 workarounds > 90 days old without fix progress: 0
- Total active workarounds: track trend — declining is healthy; growing is a sign that fixes are not being delivered

**Measurement method**: KEDB (ServiceNow Known Error table or Confluence-based KEDB) — count entries in Active status where fix deployed flag is not set. Age by entry creation date.

| Threshold | Value | Color |
|---|---|---|
| Green | P1: all have committed fix dates. P2: no entries > 90 days without progress. Trend: stable or declining | Green |
| Amber | Any P1 workaround > 30 days without committed fix date, OR total count growing | Amber |
| Red | Any P1 workaround > 60 days without fix progress, OR P2 count growing month-over-month | Red |

**Metric health — what drives it to Red**:
- Fix delivery rate is low — problems stay open and workarounds accumulate
- Engineering prioritizes new features over problem resolution — workarounds become permanent
- Risk acceptance culture — "the workaround works, so why fix it?" is a leading indicator of a future P1 incident

### Metric 7: Recurrence Rate

**Definition**: Percentage of closed problems that re-opened or generated new related incidents within 90 days of closure. Measures whether the fix actually worked.

**Formula**:
```
Recurrence Rate = 
  (Problems closed in period that generated a new linked incident within 90 days)
  ÷
  (All problems closed in period)
  × 100
```

**Target**: ≤ 10% (>10% means either the fix was wrong, the root cause was incomplete, or a new problem was incorrectly linked to a closed record)

**Measurement method**: Problems closed in measurement period. 90 days after closure, check whether any new incidents were linked to the same CI set with similar symptoms. Count matches. This requires a 90-day lag — you are always measuring recurrence on problems closed at least 90 days ago.

**ServiceNow report**: Problem table → Closed in [date range] → check for new incidents linked to same CI(s) in the 90-day window post-closure → count problems with ≥ 1 such incident.

| Threshold | Value | Color |
|---|---|---|
| Green | ≤ 10% recurrence within 90 days | Green |
| Amber | 11–20% | Amber |
| Red | > 20%, OR any P1 problem recurs within 30 days of closure | Red |

**Metric health — what drives it to Red**:
- Root cause was misidentified — fix addressed the triggering event, not the structural condition
- Fix was incomplete — addressed one instance of the condition, not all instances (e.g., fixed one queue but not others on the same pattern)
- Corrective action was not verified — stated as complete but not confirmed to eliminate the root cause
- New environment condition triggered the same structural vulnerability — fix was correct but environment changed

### Metric 8: KEDB Match Rate

**Definition**: Percentage of incidents where the incident responder searched the KEDB, found a matching entry, and applied or confirmed the workaround. Measures whether the knowledge base is being used as an operational tool.

**Formula**:
```
KEDB Match Rate = 
  (Incidents where KEDB entry was found and referenced)
  ÷
  (All incidents where a KEDB search was performed)
  × 100
```

> **Implementation note**: This metric requires KEDB search activity to be logged. In ServiceNow, track this via the "related known errors" field on the incident record. In Jira/Confluence, require an explicit field on incident post-mortems. If you cannot measure KEDB searches, measure "incidents linked to KEDB entry on closure" as a proxy.

**Target**: ≥ 60% for incidents where a KEDB entry exists for the affected CI (if the entry is there and they are not finding it, discoverability is broken)

**Measurement method**: Incident records closed in measurement period → filter to incidents where affected CI has an active KEDB entry → count % with KEDB entry referenced.

| Threshold | Value | Color |
|---|---|---|
| Green | ≥ 60% of incidents (with applicable KEDB entries) link to the KEDB entry | Green |
| Amber | 40–59% | Amber |
| Red | < 40% | Red |

**Metric health — what drives it to Red**:
- KEDB is hard to search — entry titles use architectural language, not symptom language
- KEDB is not integrated into the incident response workflow — engineers don't check it by habit
- Alert names in KEDB don't match current alert names — KEDB entries are stale
- KEDB requires access that on-call engineers don't have during incidents

### Metric 9: Proactivity Rate

**Definition**: Percentage of problems opened without a triggering incident — identified through trending, monitoring review, or proactive pattern analysis. Measures program maturity. Reactive programs find problems after incidents. Mature programs find structural vulnerabilities before they cause incidents.

**Formula**:
```
Proactivity Rate = 
  (Problems opened with no linked triggering incident)
  ÷
  (All problems opened in measurement period)
  × 100
```

**Target**:
- Year 1 (program establishing): ≥ 10% proactive
- Year 2 (program maturing): ≥ 20% proactive
- Year 3+ (program mature): ≥ 30% proactive

A problem management program at 0% proactivity is purely reactive — it is a documentation exercise, not a prevention program.

**Measurement method**: Problem records opened in measurement period → count those where no incident ID is listed in "Related Incidents" field → calculate %.

**Splunk query**:
```splunk
index=itsm sourcetype=problem_data
| eval has_incident=if(isnull(related_incident_count) OR related_incident_count=0,"Proactive","Reactive")
| stats count by has_incident
| eventstats sum(count) as total
| eval rate=round(count/total*100,1)
| where has_incident="Proactive"
```

| Threshold | Value | Color |
|---|---|---|
| Green | Meets program maturity target for year | Green |
| Amber | Within 10 percentage points below target | Amber |
| Red | > 10 percentage points below target, OR 0% proactive for 2+ consecutive months | Red |

**Metric health — what drives it to Red**:
- No systematic trending review cadence — proactive identification requires scheduled analysis time
- Trending analysis tool not available or not used — engineers don't have a way to surface patterns
- Problem management seen as incident post-processing only — cultural gap in what the program is for
- SRE team capacity fully consumed by incident response — no capacity for proactive work

### Metric 10: Problem Resolution Cycle Time

**Definition**: Average time from problem record opened to problem record closed (permanent fix deployed and verified). Measures end-to-end throughput of the problem management process.

**Formula**:
```
Cycle Time (per problem) = Closed date - Opened date (in days)
Average Cycle Time = AVG(Cycle Time) for all problems closed in period, by priority
```

**Target**:
- P1: ≤ 30 days average (≤ 60 days for Complexity 5 with documented approval)
- P2: ≤ 60 days average
- P3: ≤ 90 days average
- P4: ≤ 180 days average

**Measurement method**: Problem records closed in measurement period. For each: subtract opened date from closed date. Average by priority.

**Splunk query**:
```splunk
index=itsm sourcetype=problem_data status="Closed"
| eval cycle_days=round((strptime(closed_at,"%Y-%m-%dT%H:%M:%S")-strptime(opened_at,"%Y-%m-%dT%H:%M:%S"))/86400,0)
| stats avg(cycle_days) as avg_cycle median(cycle_days) as p50_cycle max(cycle_days) as max_cycle count by priority
| sort priority
```

> Include both average and median in reports. Outliers (e.g., one Complexity 5 problem with 180-day cycle) skew the average significantly. Median provides a more useful picture of typical performance.

| Threshold | Value | Color |
|---|---|---|
| Green | Average cycle time within target for all priorities | Green |
| Amber | Average within 20% above target for P1/P2 | Amber |
| Red | Average exceeds target for P1 or P2, OR any P1 > 90 days open | Red |

**Metric health — what drives it to Red**:
- Root cause identification is taking too long — method mismatch or evidence gaps
- Fix delivery is delayed — engineering capacity, dependency, or commitment quality issues
- Approval and change management process creates delivery lag
- Problem records stay in "Root Cause Identified" status without engineering pickup — no backlog integration

## DASHBOARD DESIGN GUIDANCE

> These metrics are most useful when viewed together, not in isolation. A healthy program has correlated indicators. Use this correlation table to interpret anomalies.

| If This Metric Is Red | Check These Metrics | Likely Diagnosis |
|---|---|---|
| Recurrence Rate | MTBI, Fix Delivery Rate | Root cause misidentified or fix incomplete |
| Backlog Age | Fix Delivery Rate, Cycle Time | Engineering capacity or commitment quality issue |
| KEDB Match Rate | Workaround Count | KEDB discoverability problem or stale entries |
| Proactivity Rate | Backlog Size | Program purely reactive; no capacity for prevention |
| Fix Delivery Rate | Cycle Time, Backlog Age | Systematic delivery failure; escalation needed |
| RCA Completion Rate | Recurrence Rate (lagging) | Knowledge gap creating future recurrence risk |
