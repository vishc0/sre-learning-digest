# RCA Method Selector

> **Decision guide for RCA facilitators.** Choosing the wrong method wastes a session and produces a root cause statement that is either too shallow (5 Whys on a multi-factor distributed failure) or overengineered (FTA on a straightforward process failure). Match the method to the failure type before scheduling the session.

---

## DECISION FLOW

```
START: What type of failure occurred?
│
├── Single service / component failed
│   ├── Causal chain is linear (A caused B caused C)
│   │   └──> 5 WHYS
│   └── Multiple contributing factors, unclear which is root
│       └──> FISHBONE (Ishikawa)
│
├── Multiple services failed together (distributed failure)
│   ├── Two or more independent failure paths combined to cause the outcome
│   │   └──> FAULT TREE ANALYSIS (FTA)
│   └── Sequence of events across services — one triggered the next
│       └──> TIMELINE ANALYSIS + 5 WHYS per causal thread
│
├── Process / human error caused the failure
│   └── Stop before concluding "human error" — human error is always a symptom.
│       Use FISHBONE to surface the process or system condition that allowed
│       the human error to have this impact.
│       └──> FISHBONE
│
├── Proactive analysis (no failure occurred yet)
│   └── Enumerating failure modes before deployment
│       └──> FMEA
│
└── UNCLEAR — you don't know what type of failure it is yet
    └──> Build the TIMELINE first, then re-enter this flow
```

## DETAILED COMPARISON TABLE

| Method | Best For | Not Good For | Time Required | Output Format | Min Team Size | Key Strength | Key Weakness |
|---|---|---|---|---|---|---|---|
| **5 Whys** | Linear causal chains; single-service failures; process failures | Multi-factor failures; distributed systems; failures with multiple parallel causes | 1–2 hours | Causal chain document + root cause statement | 2–4 | Fast; low overhead; accessible to non-engineers | Produces shallow results if team stops at proximate cause; breaks down when there are parallel causal paths |
| **Fishbone (Ishikawa)** | Multi-factor failures; unclear root cause category; team brainstorming | Failures requiring deep quantitative analysis; very complex system interactions | 2–3 hours | Fishbone diagram + prioritized contributing factors | 4–8 | Surfaces organizational and process factors that 5 Whys misses; prevents premature convergence | Can produce a long list of contributing factors without clear primary cause; requires skilled facilitation to prioritize |
| **Fault Tree Analysis** | Complex distributed failures; safety analysis; failures with multiple independent causal paths | Simple failures; time pressure situations; teams unfamiliar with Boolean logic | 4–8 hours | Fault tree diagram + minimal cut sets | 4–6 (plus system SMEs) | Rigorous; surfaces all combinations of events that cause failure; identifies single points of failure | Time-intensive; requires systems knowledge to decompose events; output requires interpretation |
| **FMEA** | Proactive analysis of new systems before deployment | Post-incident root cause analysis (it is not designed for backward-looking analysis) | 4–16 hours | FMEA table with RPN scores and mitigation actions | 4–8 (cross-functional) | Comprehensive; forces enumeration of ALL failure modes; produces prioritized risk list | Labor intensive; not appropriate when you have a specific failure to explain; generates many low-priority items |
| **Timeline Analysis** | Any complex failure where sequence of events is unclear; always as a precursor to other methods | As a standalone method — it describes what happened, not why | 1–2 hours (initial) | Chronological event table with evidence sources | 2–3 (then expand) | Evidence-anchored; surfaces detection gaps; required foundation for FTA and multi-thread 5 Whys | Not a root cause method — must be paired with another method for WHY |

## METHOD-SPECIFIC GUIDANCE

---

### 5 Whys

**Best for**: Single service failures with a clear causal chain. Process failures where one event led to another in a linear sequence. Quick RCAs for P3/P4 problems where overhead of more complex methods is not justified.

**How to use**:
1. Start with the customer impact statement: "Customers experienced [X]."
2. Ask "Why did this happen?" — record the answer.
3. Ask "Why did [that] happen?" — record the answer.
4. Continue asking "Why?" until you reach a structural condition: a system design, configuration gap, process absence, or organizational condition that, if corrected, would prevent recurrence even if the triggering event repeated.
5. Stop when you reach a structural condition. The structural condition IS the root cause.

**Stopping rule**: You have reached the root cause when your answer is a structural condition, not a person's decision or a one-time event. The test: "If we fix this, would the failure still be possible next time?" If yes, keep going.

**Anti-pattern**: Stopping at "the engineer made an error." That is a contributing factor, not a root cause. The real question is: "What process, system, or organizational condition allowed this human error to produce this customer impact?" That answer is the root cause.

**Example**:
```
Customer impact: Payment confirmations delayed 47 minutes for 800K customers.
Why? → RabbitMQ consumer lag exceeded 500K messages on payment-events queue.
Why? → Consumer throughput was insufficient for peak batch load.
Why? → Consumer group size was set to 3; batch load during month-end requires 12+.
Why? → No load testing was performed for month-end batch scenarios.
Why? → Load testing runbook only covered standard peak, not month-end peak. (← Structural condition: process gap)

Root cause: Process gap in load testing scope — month-end batch scenarios were not included in pre-deployment testing criteria.
```

**Output**: Causal chain document (each Why → Because pair) + root cause statement in the required format.

### Fishbone (Ishikawa)

**Best for**: Failures where the contributing factors are unclear, multiple, or span different domains (technical, process, human, organizational). Particularly effective when the team has competing hypotheses about the root cause. Forces a structured exploration before converging on a single cause.

**Categories to use** (adapt to your context):
- **People**: Skills, training, communication, decision-making, workload
- **Process**: Procedures, runbooks, review gates, change controls, approval workflows
- **Technology**: System design, configuration, code quality, dependencies, tooling
- **Environment**: External dependencies, network conditions, cloud provider, vendor behavior
- **Measurement**: Monitoring gaps, alerting thresholds, observability coverage, log retention
- **Management**: Prioritization decisions, resource allocation, organizational structure, incentives

**How to use**:
1. Draw the fishbone: place the failure effect (customer impact) at the right end (the "head"). Draw a horizontal spine. Add 6 diagonal bones for each category.
2. Brainstorm: for each category, list all contributing factors the team can identify. Do not filter at this stage — capture everything. Each factor is a smaller bone.
3. Deepen: for each contributing factor, apply 5 Whys to understand the causal chain within that bone.
4. Prioritize: after all bones are populated, vote or discuss to identify the 2–3 factors that most directly enabled the failure. These are your candidate root causes.
5. Confirm: gather evidence for each candidate. The one with the strongest evidence and most direct causal link to the customer impact is the root cause.

**Facilitation tips**:
- Do not allow debate during brainstorming — all items go on the board first.
- Explicitly prompt for Process and Management factors — teams naturally gravitate to Technology. If the fishbone has nothing in Process or Management, push the team harder.
- Use dot voting (3 dots each) after brainstorming to surface the team's top candidates before evidence review.

**Output**: Fishbone diagram (can be whiteboard photo or draw.io export) + prioritized contributing factors list + root cause statement after evidence review.

### Fault Tree Analysis (FTA)

**Best for**: Complex distributed system failures. Safety-critical analysis. Failures where multiple independent conditions combined to produce the outcome. Any situation where you need to understand ALL possible causal paths, not just the path that caused this specific incident.

**Core concept**: FTA works top-down. Start with the top event (the customer impact — what you do NOT want to happen). Decompose it into the events that together cause it, using AND gates (all must occur) and OR gates (any one can cause it). Continue decomposing until you reach basic events — conditions that cannot be further decomposed and that have observable, measurable states.

**AND gate**: Both sub-events must occur for the parent event to occur. Example: "Service unavailability" requires AND ["all pods crashed" AND "load balancer health check passed incorrectly"].

**OR gate**: Either sub-event alone causes the parent event. Example: "Data corruption" is caused by OR ["write race condition" OR "cache invalidation failure" OR "replica sync gap"].

**Minimal cut sets**: The most important output of FTA. A minimal cut set is the smallest combination of basic events that, if they all occurred simultaneously, would cause the top event. Single-element minimal cut sets are single points of failure — highest priority to fix.

**How to use**:
1. Define the top event precisely: the specific customer impact, not a vague description.
2. Ask: "What immediate conditions or events directly cause this top event?" Connect with AND or OR gate.
3. For each sub-event, ask: "What directly causes this?" Continue down.
4. Stop when you reach basic events: conditions that are observable and cannot be further decomposed (e.g., "consumer lag threshold exceeded," "alert not configured," "EKS node pool exhausted").
5. Identify minimal cut sets: trace all paths from top to basic events. Any path that reaches the top through AND-only gates is a minimal cut set.
6. Prioritize corrective actions starting with single-element cut sets (single points of failure).

**Tool recommendation**: Start on a whiteboard with sticky notes (AND gate = D-shape pointing up, OR gate = curved shield shape). For documentation, use draw.io (free) with the fault tree shape library. For complex trees, Relyence or Isograph offer more structure.

**Output**: Fault tree diagram + minimal cut sets table + corrective actions targeting highest-priority cut sets first.

### FMEA (Failure Mode and Effects Analysis)

**Best for**: Proactive analysis of new systems or significant changes before deployment. Enumerating what could go wrong before it does. Use FMEA outputs to create proactive problem records targeting high-RPN failure modes.

**Not for post-incident RCA**: FMEA is forward-looking. If you have a specific failure to explain, use 5 Whys, Fishbone, or FTA. FMEA on a known failure produces a list of potential causes instead of the actual cause.

**How to use**:
1. Define the system scope: which components, interfaces, and processes are in scope.
2. Enumerate failure modes: for each component or process step, list all the ways it could fail. Think broadly — complete failure, degraded performance, incorrect output, missing output, delayed output.
3. For each failure mode, assess:
   - **Severity (S)**: How severe is the impact on the customer if this failure mode occurs? Scale 1–10 (1 = no impact, 10 = safety risk or complete service loss).
   - **Occurrence (O)**: How likely is this failure mode to occur? Scale 1–10 (1 = extremely rare, 10 = almost certain).
   - **Detection (D)**: How likely is it that current controls would detect this failure before it impacts the customer? Scale 1–10 (1 = definitely detected, 10 = no detection mechanism).
4. Calculate Risk Priority Number: **RPN = S × O × D**. Range: 1–1000.
5. Prioritize: address high-RPN items first. Also prioritize any item with S=10 regardless of RPN.
6. For each high-RPN item: define a corrective action to reduce S, O, or D. Recalculate RPN after action.

**Problem management use**: Run FMEA before launching new services or after major architectural changes. High-RPN failure modes that cannot be mitigated before launch become proactive problem records with a pre-written contributing factors list.

**Output**: FMEA table (Component/Process Step | Failure Mode | Effect | S | Cause | O | Current Controls | D | RPN | Recommended Action | Owner | RPN After Action).

## METHOD SELECTION DECISION TABLE

| Failure Type | Primary Method | Secondary Method | Avoid |
|---|---|---|---|
| Single service / single component failed, linear cause chain | 5 Whys | Timeline Analysis (as precursor) | FTA (overkill) |
| Single service, multiple contributing factors, unclear primary cause | Fishbone | 5 Whys per factor | FTA (premature for single service) |
| Multiple services failed in cascade (A triggered B triggered C) | Timeline Analysis → 5 Whys per thread | Fishbone for organizational factors | FMEA (not retrospective) |
| Multiple services failed independently at same time (fan-in failure) | FTA | Fishbone for contributing factors | 5 Whys (linear method for non-linear failure) |
| Process failure / human error caused or contributed to failure | Fishbone (forces Process + Human + Organizational categories) | 5 Whys on each bone | Stopping at "human error" as root cause |
| Intermittent failure with no reproducible trigger | Timeline Analysis → Fishbone | 5 Whys (if pattern emerges from timeline) | FTA (tree requires stable failure logic) |
| New system / major change pre-deployment analysis | FMEA | FTA for highest-risk components | 5 Whys / Fishbone (retrospective methods) |
| Unknown failure type — still investigating | Timeline Analysis first | Re-enter this table after timeline | Commit to any root cause method before timeline is built |
| P4 / low-complexity failure with obvious cause | 5 Whys (abbreviated — 3 Whys may suffice) | — | Full FMEA or FTA (not proportionate) |
| Safety-critical or compliance-critical failure | FTA (rigorous enumeration of all paths required) | FMEA (for residual risk assessment) | 5 Whys (not comprehensive enough for safety analysis) |

## FACILITATOR CHECKLIST

> **Before the session**:
- [ ] Failure type classified using decision flow above
- [ ] Method selected and documented in problem record
- [ ] Participants confirmed: engineer who responded, engineer who owns the failing system, process owner if process failure suspected
- [ ] Evidence assembled: timeline, logs, metrics, change records — do not run a session with no evidence; hypothesizing without evidence is not RCA
- [ ] Session length allocated: 5 Whys = 1.5 hours; Fishbone = 2.5 hours; FTA = 5+ hours; FMEA = half-day minimum

> **During the session**:
- [ ] Facilitator is neutral — not an investigator; the group investigates; the facilitator structures the process
- [ ] Every claim requires evidence citation — "I think" without evidence goes to the hypothesis column, not the finding column
- [ ] At least one process/organizational factor explored regardless of which method is used
- [ ] "Human error" challenged — always ask what system or process allowed this human error to have this impact

> **After the session**:
- [ ] Root cause statement written in required format before adjourning
- [ ] Every contributing factor has an evidence citation
- [ ] Evidence gaps documented
- [ ] Corrective actions drafted with named owners before adjourning (refine, but don't start from scratch later — momentum is lost)
