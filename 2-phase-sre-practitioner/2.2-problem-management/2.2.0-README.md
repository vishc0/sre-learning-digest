# Problem Management: The Complete Framework

## What This Is

Incidents are events. Problems are conditions. This framework is about eliminating the conditions.

Every organization that manages incidents at scale eventually discovers the same pattern: the same failures recur. The on-call queue fills with variations of the same alert. Engineers who joined three years apart are solving the same class of failure they inherited from the engineers who came before them. The incidents are being managed. The problems are not.

This framework converts problem management from a compliance obligation into an engineering discipline — one with defined lifecycle stages, measurable outcomes, economic accountability, and governance that connects the postmortem to the engineering calendar to the budget conversation.

This is not a theory document. It is a working framework covering:
- How to detect that a problem exists and open the record before the next incident recurs
- What a problem record contains and who is accountable for each field
- How to run a Root Cause Analysis (RCA) that produces a permanent fix, not a summary
- How to build and govern a Known Error Database (KEDB) that actually accelerates incident response
- How to prioritize problems when you have more than you can fix
- How to use trend analysis and SLO signals to find problems before they produce incidents
- How to govern the program so that problem records close with engineering outcomes, not just status changes

## The Three Failure Modes of Problem Management Programs

Most organizations fail at problem management in one of three patterns. These are distinct from incident management failures — they appear in organizations that run competent incident management but still see recurrence rates that do not improve.

**Failure Mode 1: Treating problems as slow incidents**

The incident closes. A problem record is opened — sometimes. When it is opened, it is assigned to the same on-call team that just resolved the incident, with no distinct ownership model, no different process, and no different outcome expectation. The problem record becomes a slow-moving incident ticket: engineers debug against it when they have time, it ages in the queue, and eventually it is closed because the incident recurred and a new record was opened. The fundamental category error is treating problem management as incident management with a longer SLA. It is not. Incident management is time-bounded, customer-impact-focused, and command-driven. Problem management is open-ended, analytically-driven, and systemic-fix-focused. Different timescales, different owners, different accountability structures.

**Failure Mode 2: Analysis theater — excellent postmortems, zero engineering investment**

Organizations in this failure mode run blameless RCAs, write thorough five-why analyses, produce polished postmortem documents, and identify root causes with genuine precision. Then those root causes sit in a backlog that engineering leadership never prioritizes. The RCA report exists. The Jira ticket exists. The sprint never comes. The problem record status changes from "Root Cause Known" to "Fix in Progress" on the day someone updates the date field — not the day any code ships. The result: MTBI (Mean Time Between Incidents) for known problems does not improve, because the analysis was never converted into engineering work. This failure mode is particularly dangerous because it creates the appearance of a functioning problem management program while producing none of the outcomes.

**Failure Mode 3: Fixing symptoms, not systems**

Permanent fixes are defined and delivered, but they target the proximate cause — the immediate mechanism of failure — rather than the structural conditions that created the failure environment. A service crashes under load: the fix is a memory limit increase. A deployment breaks a downstream dependency: the fix is a rollback procedure improvement. A circuit breaker fails to open: the fix is a threshold adjustment. These changes reduce the immediate failure probability without addressing the conditions that guarantee another failure of the same class: insufficient capacity planning, absent integration contract testing, circuit breaker configurations that were never validated under realistic load. The problem record closes. The underlying structural weakness remains. Six months later, a different variant of the same problem opens a new record.

This framework addresses all three. Chapters 1–5 address Failure Mode 1 by establishing distinct processes and ownership. Chapters 3 and 7 address Failure Mode 2 by connecting RCA outcomes to governance and budget conversations. Chapters 3, 5, and 8 address Failure Mode 3 by setting quality standards for what constitutes a permanent fix versus a symptom treatment.

## Key Definitions (Used Consistently Across All Files)

| Term | Precise Definition |
|------|-------------------|
| **Problem** | The underlying cause of one or more incidents; a structural condition that will produce further incidents until permanently resolved |
| **Known Error** | A problem whose root cause has been identified but whose permanent fix has not yet been deployed; a workaround exists |
| **KEDB** | Known Error Database — the authoritative registry of all known errors with their workarounds, fix plans, and review status |
| **Problem Manager** | Process owner accountable for problem record lifecycle, status tracking, SLA compliance, and governance reporting; does not debug |
| **Problem Owner** | Technical subject matter expert accountable for RCA completion and permanent fix delivery; does not manage the record |
| **Root Cause** | The structural condition — architectural, operational, or organizational — whose removal prevents recurrence of the incident class |
| **Workaround** | A procedure that reduces or eliminates customer impact from a known error without resolving its root cause |
| **Permanent Fix** | An engineering change that removes the root cause and eliminates the possibility of the same failure class recurring under the same conditions |
| **MTBI** | Mean Time Between Incidents — the average interval between occurrences of the same incident class; the primary outcome metric for problem management |
| **Problem Record** | The formal artifact tracking a single problem through its lifecycle from identification through permanent resolution or risk acceptance |
| **PIR** | Post-Incident Review — the structured review process conducted after a significant incident; the primary trigger for reactive problem management (see [Incident Management framework](../incident-management/README.md)) |
| **Reactive Problem Management** | Problem records opened in response to incidents that have already occurred; triggered by postmortem findings or incident clusters |
| **Proactive Problem Management** | Problem records opened before incidents occur, triggered by trend analysis, SLO burn rate signals, or capacity forecasts |

## Relationship to Incident Management and SRE Framework

Problem management and incident management share a boundary, not a process. The handoff is explicit: an incident closes when customer impact is zero. A problem opens when the underlying cause is not yet eliminated. These are separate events, separate records, and separate ownership chains.

**The handoff mechanism**: When an incident produces a PIR ([Post-Incident Review process](../incident-management/playbook/postmortem-playbook.md)), the PIR output includes a determination: is this failure class resolved permanently, or does a structural condition remain? If a structural condition remains, a problem record opens at PIR close. The incident team hands off to the Problem Manager. The incident record is linked to the problem record. From that point, the problem has its own SLA, its own owner, and its own governance cadence independent of the incident's resolution state.

**SRE error budget integration**: In the SRE model, every incident consumes error budget. A problem left unfixed will recur, consuming additional error budget in the future. The economic cost of a problem is therefore the sum of error budget consumed by all future recurrences until the problem is resolved — which can be calculated if MTBI and incident severity are known. This framing converts problem management from a process obligation into a measurable engineering investment decision: the cost of fixing the problem versus the cost of the error budget it will continue to consume. At T-Mobile notification platform scale (25M messages/day), a P2 problem with a 14-day MTBI at 45-minute average incident duration is a calculable ongoing cost. Directors who frame it this way get engineering investment prioritized.

**KEDB as the bridge**: The Known Error Database is the operational link between incident knowledge and future response acceleration. When an on-call engineer encounters a known error, the KEDB entry provides the workaround immediately — reducing MTTR by eliminating the diagnostic phase for that failure class. The KEDB does not exist to normalize known failures; it exists to minimize their impact while permanent fixes are in flight. A KEDB entry that remains open indefinitely without a fix ETA has failed its purpose and become a documentation of accepted dysfunction.

**SRE coined terms in this framework**: Where SRECapstone terms are used, they are referenced in brackets:
- `[EBV]` (Error Budget Velocity) drives the urgency calculation in ch05 and ch08
- `[MTBI]` is the primary outcome metric tracked in ch07 governance reporting
- `[FLI]` (Failure Locality Index) informs the proactive problem opening criteria in ch08
