# Chapter 2: Problem Record Lifecycle

---

## 1. Problem Detection Triggers

A problem record does not open itself. The detection mechanism — who sees the signal, who has the authority to open the record, and under what criteria — determines whether your problem management program catches failures before the fourth recurrence or the fourteenth.

At T-Mobile notification platform scale, with 25M messages per day and thousands of change records per month, manual problem detection is operationally impossible. An SRE manager scanning incident history for patterns will miss signals that a Splunk query running every 8 hours will catch reliably. Automated detection is not a luxury — it is the only mechanism that scales. Human judgment is applied to the analysis and the fix decision, not to the pattern recognition.

**Five detection trigger types** and how each operates:

**Trigger 1: Incident Cluster**
Definition: The same failure mode appearing in 3 or more incidents within a 30-day window for the same Configuration Item (CI) or service.
Detection mechanism: Automated query across incident management platform (ServiceNow, Jira Service Management) identifying CI + incident cause code combinations that meet the threshold. At T-Mobile: a scheduled Splunk query joining change records, alert data, and incident records on service name and failure class.
Who opens the record: Problem Manager for the owning service, auto-assigned based on service ownership CMDB entry.
Why this threshold: 3 incidents in 30 days represents a failure rate that will consume meaningful error budget over a quarter. Waiting for more recurrences is a choice to accept that consumption.

**Trigger 2: Post-Incident Review (PIR) Finding**
Definition: A PIR for a SEV1 or SEV2 identifies a structural root cause — a condition in the system design, configuration, or operational model — that requires engineering work beyond the immediate incident fix.
Detection mechanism: PIR template includes a mandatory question: "Does a structural condition remain after this incident's resolution that will produce future incidents of the same class?" If yes, the PIR facilitation guide requires a problem record to be opened before the PIR is marked complete.
Who opens the record: Problem Manager, at PIR close — not the incident team.
Note: This is the most common reactive trigger. It is also the most frequently skipped, because the incident team is exhausted and wants to close everything. The PIR process must enforce the handoff.

**Trigger 3: Proactive Monitoring Signal**
Definition: An SLO burn rate or capacity metric crosses a threshold that predicts future incidents before they occur.
Detection mechanism: SLO burn rate exceeding 2x the safe consumption rate with more than 50% of the measurement window remaining; or resource utilization (CPU, memory, queue depth, connection pool) trending toward saturation within the next 30 days based on linear projection.
Who opens the record: Problem Manager, on automated alert from observability platform.
Note: This trigger is the rarest in most organizations and should be the growth area. Proactive problem opening before a SEV occurs is the highest-leverage form of problem management.

**Trigger 4: Audit or Compliance Finding**
Definition: A compliance review, security audit, chaos engineering test, or architecture review identifies an operational gap that creates recurring risk — a single point of failure, an unvalidated failover path, a dependency without a circuit breaker.
Detection mechanism: Audit output review; chaos engineering test result analysis.
Who opens the record: Problem Manager, upon receipt of audit finding categorized as operational risk.
Note: These problems often have no associated incident history yet. They require Proactive Problem Management posture: opening a record for a condition that has not caused a customer-visible incident but has a credible path to one.

**Trigger 5: Change Correlation**
Definition: The same change window or the same type of change consistently produces incidents across multiple deployment cycles.
Detection mechanism: Change failure rate analysis — percentage of change records in a 30-day window that were followed by an incident within 4 hours. A change failure rate above 15% for a service is a problem signal.
Who opens the record: Problem Manager for the change advisory process, or the service Problem Manager.

**Trigger Decision Table**:

| Trigger Type | Detection Method | Who Opens Problem Record | Default Priority |
|-------------|-----------------|--------------------------|-----------------|
| Incident cluster (3+ in 30 days) | Automated ITSM query / Splunk | Service Problem Manager | P2 |
| SEV1 repeat (within 90 days) | PIR process gate | Service Problem Manager | P1 |
| PIR structural finding | PIR template mandatory field | Problem Manager at PIR close | P1/P2 per PIR severity |
| SLO burn rate ≥ 2x baseline | Observability platform alert | Problem Manager on call | P2 |
| Capacity saturation projection | Capacity planning model | Service Problem Manager | P3 |
| Audit / compliance finding | Audit report triage | Problem Manager + service owner | P2/P3 per risk rating |
| Change failure rate > 15% | Change analytics dashboard | Change advisory Problem Manager | P2 |

## 2. Problem Record Structure

A problem record is not a freeform incident ticket with a different label. Every field has a specific owner, a specific purpose, and — for required fields — a completion SLA. A problem record that reaches "Root Cause Known" status with empty required fields from earlier stages has failed its governance function: the downstream owner cannot make decisions without that data.

| Field | Type | Required/Optional | Owner | Purpose |
|-------|------|-------------------|-------|---------|
| Problem ID | Auto-generated | Required | System | Unique identifier referenced in all linked incidents and change records |
| Short Description | Single sentence | Required | Problem Manager | Actionable description: service + failure class + operational impact. Not a copy of the incident title. |
| Long Description | Free text | Required | Problem Manager | Symptom pattern across all triggering incidents; affected CIs; customer impact summary with scope and frequency |
| Priority | P1–P4 | Required | Problem Manager | Assigned at record creation using priority matrix; reviewed at each status transition |
| Problem Manager | Person | Required | Assigned by service ownership table | Accountable for record lifecycle, SLA compliance, and governance reporting; changes with the record if service ownership changes |
| Problem Owner | Person | Required by Investigating status | Problem Manager | Technical SME accountable for RCA completion and permanent fix delivery |
| Affected Services (CIs) | List | Required | Problem Owner | CMDB CI list for all services that exhibit the failure class or are affected by it; used for KEDB entry scoping |
| Related Incidents | Linked records | Required | Problem Manager | All incident records that triggered or corroborate this problem; must include incident IDs, dates, and impact summaries |
| Status | Enum (see §3) | Required | Problem Manager | Current lifecycle state; updated only when entry and exit criteria are met |
| Workaround | Free text or KEDB link | Required if status = Known Error | Problem Owner | Step-by-step workaround procedure or link to KEDB entry; must be operational precision, not conceptual description |
| Root Cause | Free text | Required by Root Cause Known status | Problem Owner | The structural condition whose removal prevents recurrence — not the proximate failure mechanism |
| Fix Plan | Structured text | Required by Fix in Progress status | Problem Owner | What engineering work will be done, by whom, in which sprint, with what acceptance criteria |
| Fix ETA | Date | Required by Fix in Progress status | Problem Owner + EM | Committed sprint delivery date — not a calendar aspiration. Must correspond to sprint backlog entry. |
| Review Date | Date | Required | Problem Manager | Next governance checkpoint; must not exceed the maximum SLA for current status (see §4) |
| Risk Acceptance Approver | Person | Required for Accepted Risk closure | Director minimum | Named individual who formally accepted the residual risk, with date and rationale |

## 3. Status Flow

A problem record moves through defined lifecycle states. Each state has an entry criterion (what must be true to enter it) and an exit criterion (what must be true to leave it). Moving a status field without meeting the criteria is a governance failure — it produces records whose status reflects what the team wants to be true, not what is operationally true.

```
[Identified] → [Investigating] → [Root Cause Known] → [Fix in Progress] → [Resolved] → [Closed]
                     ↑__________________________|
                     (if RCA reveals initial root cause was incomplete or incorrect)
```

**State definitions**:

**Identified**: The problem record exists. Trigger has been confirmed. Priority is assigned. Problem Manager is named. The structural condition is suspected but not yet analyzed.
- Entry criteria: Trigger threshold met; Problem ID generated; Problem Manager assigned; related incidents linked.
- Exit criteria: Problem Owner assigned; investigation scope defined; initial hypothesis documented.

**Investigating**: Active RCA underway. The Problem Owner and technical team are running the root cause analysis. The structural condition is being mapped.
- Entry criteria: Problem Owner confirmed; investigation scope and method selected; RCA session scheduled or underway.
- Exit criteria: Root cause documented with evidence; RCA report draft complete; workaround identified (if possible).

**Root Cause Known**: Root cause is documented with sufficient precision to define a fix. Workaround exists or has been declared unavailable. Fix options have been evaluated.
- Entry criteria: RCA report approved by Problem Manager; root cause statement is specific and actionable (not "service instability" or "network issue"); fix options documented.
- Exit criteria: Fix plan approved at sprint level; Fix ETA committed; engineering EM confirmed capacity.

**Fix in Progress**: Engineering work to implement the permanent fix is actively underway in a sprint.
- Entry criteria: Fix plan in sprint backlog with story points; Fix ETA confirmed; Problem Owner tracking progress.
- Exit criteria: Fix deployed to production; validation criteria met; no recurrence of incident class within defined observation window.

**Resolved**: Fix has been deployed. Observation window is running to confirm the fix was effective.
- Entry criteria: Fix deployed and change record linked; post-deployment monitoring active; no recurrence in initial observation period (72 hours for P1, 7 days for P2, 14 days for P3).
- Exit criteria: Observation window completed without recurrence; MTBI clock reset; final RCA report published.

**Closed**: Problem record permanently closed. Either permanent fix confirmed effective, or risk formally accepted.
- Entry criteria: Resolved status observation complete with no recurrence OR risk acceptance formally documented with approver named.
- Exit criteria: N/A — terminal state. Problem record is read-only.

**SLA Table — Maximum Time in Each Status Before Escalation**:

| Priority | Identified → Investigating | Investigating → Root Cause Known | Root Cause Known → Fix in Progress | Fix in Progress → Resolved | Total Fix Delivery |
|----------|---------------------------|----------------------------------|-------------------------------------|----------------------------|--------------------|
| **P1** | 4 business hours | 5 business days | 10 business days | 30 calendar days | 45 calendar days |
| **P2** | 1 business day | 10 business days | 20 business days | 60 calendar days | 90 calendar days |
| **P3** | 3 business days | 20 business days | 30 business days | 90 calendar days | 120 calendar days |
| **P4** | 5 business days | Quarterly review cycle | Quarterly review cycle | Next major release cycle | 6 months |

Escalation path when SLA is breached: Problem Manager → Director of Engineering for the owning service → VP of Platform/SRE (for P1/P2 breach at fix delivery stage).

## 4. Owner Accountability Model

Four distinct roles carry distinct accountability. The confusion of roles — which is universal in smaller organizations and still common in large ones — is one of the most reliable predictors of problem records that age without resolution.

**Problem Manager**: Owns the problem record lifecycle. Accountable for: status accuracy, SLA compliance, escalation when SLAs are at risk, governance reporting, and closing the record with a documented outcome. Does NOT perform the technical analysis. Does NOT write code. Does NOT determine root cause. The Problem Manager's job is to make sure the people who can do those things are doing them — and to escalate when they aren't.

**Problem Owner**: Owns the technical investigation and fix delivery. Accountable for: selecting the RCA method, running or commissioning the analysis, producing a root cause statement of sufficient precision to define a fix, defining the fix plan, and tracking fix delivery through deployment. Does NOT manage the record. Does NOT update status fields. Does NOT run governance meetings. The Problem Owner's job is engineering analysis and fix accountability.

**Technical Lead**: The engineer who executes the fix work. Reports to the Problem Owner during the fix delivery phase. Accountable for: implementing the fix per the agreed plan, running validation tests, and reporting progress to the Problem Owner. Does NOT make scope decisions without Problem Owner approval.

**Service Owner**: Business accountability for the affected service. Accountable for: approving risk acceptance decisions, providing business context for prioritization, and signing off on "Accepted Risk" closure. At T-Mobile scale, this is typically the product owner or service delivery manager for the platform, not the engineering lead.

**The confusion point at scale**: At organizations with small SRE teams, the Problem Manager, Problem Owner, and Technical Lead may be the same individual. This is survivable for P3 and P4 problems. For P1 and P2, role confusion creates accountability gaps: the person who is supposed to be tracking SLA compliance is the same person who is behind on the RCA because they were on call last week. At T-Mobile scale, these roles must be distinct — not because ITIL requires it, but because the accountability surfaces are genuinely different and cannot be held by one person without something being dropped.

**RACI Table — Problem Record Lifecycle**:

| Activity | Problem Manager | Problem Owner | Technical Lead | Service Owner |
|----------|----------------|---------------|----------------|---------------|
| Open problem record | R/A | I | I | I |
| Assign Problem Owner | R/A | — | — | C |
| Define investigation scope | C | R/A | C | I |
| Select RCA method | I | R/A | C | — |
| Conduct RCA | I | A | R | — |
| Document root cause | C | R/A | C | I |
| Define fix plan | I | R/A | C | I |
| Approve fix plan (sprint capacity) | I | C | — | A |
| Track fix delivery | R/A | C | R | I |
| Validate fix in production | I | R/A | R | I |
| Approve risk acceptance | C | C | — | R/A |
| Close problem record | R/A | C | — | I |
| Publish to KEDB | R/A | C | — | I |
| Governance reporting | R/A | I | — | I |

R = Responsible (does the work) | A = Accountable (owns the outcome) | C = Consulted (input required) | I = Informed (notified)

## 5. SLA for Problem Records: The Funding Signal Interpretation

SLAs for problem records serve two functions. The operational function is accountability: they define the conditions under which a problem escalates so that problems don't age silently. The strategic function — which is less commonly understood — is a funding signal.

When P1 problems consistently miss their Fix in Progress → Resolved SLA, there are three possible explanations: the fix is technically harder than the initial assessment, the engineering team is not prioritizing the problem work, or the team does not have the capacity to do the fix work alongside their other commitments. The first two are management problems. The third is a resource problem that requires a different conversation.

A Problem Manager or director who escalates an SLA breach as a process failure ("the team missed the SLA") will get a process response — an updated date, an apology, a commitment that feels less firm than the last one. A director who escalates an SLA breach as a potential funding signal ("we have 4 P1 problems in Fix in Progress with missed fix dates — I need to understand whether this is a capacity constraint or a prioritization decision") will get an honest answer about whether the team has the headroom to actually deliver the fixes they have committed to.

> **Director/VP Callout — SLA Breach as Resource Signal**
>
> A problem SLA breach is a funding signal, not just a process failure. If P1 problems consistently slip their fix dates, the team doesn't have enough capacity — that's a resource conversation, not a performance conversation. The difference matters for how you respond. A performance conversation asks "why did you miss the deadline?" A resource conversation asks "what would it take to hit the deadline? What else would need to stop?" If the answer to the second question is "we can't do this within the current team" — that is the conversation you should be having with your VP before the next SEV1 recurs, not after.

**SLA Enforcement Escalation Path**:

| Condition | Escalation Target | Escalation Content |
|-----------|------------------|--------------------|
| P1 problem in Identified > 4 hours | Problem Manager's Director | Alert: P1 unowned problem record; risk to next incident recurrence |
| P1 RCA not complete > 5 business days | VP of Platform/SRE | Status: P1 root cause unknown; KEDB cannot be updated; recurrence risk unmitigated |
| P2 fix delivery missed by > 14 days | Director of Engineering | Review: capacity vs. backlog; is this a prioritization or resourcing issue? |
| Any problem record with no status change > 30 days | Problem Manager's Director | Stale record: needs closure, risk acceptance, or re-prioritization |
| 3+ P1/P2 problems in Fix in Progress with missed ETAs simultaneously | VP of Operations/SRE | Portfolio capacity review: systemic under-resourcing signal |

## Connecting the Lifecycle to the KEDB

The problem record lifecycle does not exist in isolation. Two of the three terminal outcomes (Known Error and Permanent Fix) have direct connections to the Known Error Database.

When a problem reaches "Root Cause Known" status with no fix yet deployed, a KEDB entry is required. The KEDB entry is the operational artifact that on-call engineers use to respond faster when the incident recurs in the interim period before the fix ships. The KEDB entry must contain: the recognized failure pattern (how operators identify this known error in production), the workaround procedure (step-by-step, precise enough to execute under incident pressure at 2am), the expected outcome of the workaround (what "better" looks like after the workaround is applied), and the problem record ID so on-call engineers can track whether the permanent fix has shipped.

When the permanent fix ships and the problem record closes as Resolved, the KEDB entry is either retired (if the fix is known to eliminate the failure class entirely) or updated with the fix deployment information. An unreviewd KEDB entry for a problem that closed as "Permanently Fixed" creates false confidence in on-call teams who may apply a workaround to a condition that no longer exists.

The KEDB governance lifecycle is covered in [ch04 — Known Error Database](ch04-known-error-database.md). The connection here is the accountability handoff: the Problem Manager is responsible for creating the KEDB entry trigger when "Root Cause Known" status is reached. The Problem Owner provides the technical content. The KEDB entry becomes active immediately upon creation — not after review — because the value of a KEDB entry is highest during the period when the next incident is most likely.
