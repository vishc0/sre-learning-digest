# Problem Review Meeting Playbook
## Weekly Meeting Facilitation Guide — Director/VP of Operations Lens

---

## Meeting Purpose

**The Problem Review Meeting is a governance forum, not a technical working session.**

| This meeting IS for | This meeting is NOT for |
|--------------------|------------------------|
| Status updates — what changed since last week | Debugging — figuring out why something broke |
| Escalation decisions — who needs to act and when | RCA facilitation — that requires a separate dedicated session |
| New problem approvals — should we open a record? | Technical deep dives — architecture discussions, log analysis |
| Fix delivery accountability — is the committed date at risk? | Solution design — what should the fix be? |
| Resource and priority conflicts — what is blocking progress? | Blame attribution — whose fault is this? |

If a technical discussion starts, the facilitator's job is to surface it and redirect — not participate in it.

> **Redirect script**: "This is a useful technical question — it's outside the scope of this meeting. Can [Name] schedule a working session with [Name] and [Name] by [date]? We'll note it as an action item."

## Standing Participants

### Required Attendees

| Role | Attendance Requirement | Why Required |
|------|----------------------|--------------|
| Problem Manager | Every meeting — chair | Owns the agenda, inputs, and outputs |
| Service Owner(s) | Required for any active P1 or P2 problem they own | Accountable for fix delivery and risk acceptance |
| Engineering Lead | Required for any problem currently in "Fix in Progress" | Accountable for committed delivery dates |

### Optional but High-Value Attendees

| Role | When to Invite | Value Added |
|------|---------------|-------------|
| SRE Manager | When P1 problem is at risk of SLA breach | Provides resource unblocking authority |
| Change Manager | When problem-driven changes are in next CAB cycle | Ensures problem urgency is reflected in change prioritization |
| KEDB Owner | When multiple KEDB entries require governance decisions | Coordinates KEDB publishing and archive actions |

### Attendance Rules

- **Maximum 8 people** in the standing meeting
- Service Owners for P3 problems: attend only when their problem is on the agenda (send Problem Manager a 15-minute advance notice slot)
- Director of Operations: not a standing attendee — they receive the weekly summary email, attend only when a P1 escalation is on the agenda and they have been formally escalated to

An over-attended Problem Review Meeting becomes a status theater session where everyone listens to updates that don't affect them.

## Pre-Meeting Inputs

**Problem Manager prepares all inputs by EOD Monday** for a Tuesday meeting. Nothing should be assembled during the meeting.

### Input 1: Updated Problem Records

Pull the current state of all active problem records. For each:
- Current status (confirm it reflects the latest action — not the status from last week)
- Owner (confirmed, not theoretical)
- Next milestone and due date
- Last action taken (what happened since last meeting)

If any record status has not changed from last week and no action is logged, it is a stall — flag it for the SLA breach review section.

### Input 2: SLA Breach List

Generate or compile for P1 and P2 problems:

| Column | Content |
|--------|---------|
| Problem ID | Link to record |
| Title | Short description |
| Priority | P1 / P2 |
| SLA Milestone Type | RCA due / Fix plan due / Fix delivery due |
| Milestone Due Date | Calendar date |
| Days Overdue (or Days Remaining) | Positive = overdue, negative = days remaining |
| Last Action | What happened most recently |
| Blocking Reason | Why milestone was missed (if overdue) |
| Recommended Action | What the meeting needs to decide |

Sort: most overdue first. Color-code if your format supports it: red = overdue, yellow = due within 5 days.

### Input 3: New Problem Candidates List

Compile incidents and findings since last meeting that may qualify for problem records:

| Column | Content |
|--------|---------|
| Source (Incident ID / Trend Finding) | Link |
| CI Affected | Specific component |
| Symptom Summary | 2 sentences — what happened, what was the impact |
| Recurrence Evidence | How many times has this occurred in 30 days? |
| Proposed Priority | P1 / P2 / P3 with rationale |
| Proposed Owner | Named individual |
| Recommendation | Approve / Reject / Defer with rationale |

If a candidate clearly meets recurrence threshold, state that in the recommendation — don't make the meeting relitigate the criteria.

### Input 4: Fix Delivery Report

For all problems in "Fix in Progress":

| Column | Content |
|--------|---------|
| Problem ID | Link |
| Title | Short description |
| Engineering Lead | Named individual accountable |
| Committed Fix Date | The date agreed to in last meeting or change request |
| Current Status | On track / At risk / Overdue |
| Change Request ID | Link to CR in CAB queue |
| Blocking Reason | If at risk or overdue |
| New ETA (if applicable) | What Engineering Lead is now committing to |

## Agenda Template (50 Minutes)

### 0:00 – 0:10 — SLA Breach Review (10 Minutes)

**Facilitator goal**: for each SLA breach, get a clear escalation decision and assign next action.

For each item on the SLA breach list:

1. State the breach: "Problem [ID], [title], P[priority], [milestone] is [X days overdue]."
2. Ask the Service Owner: "What is blocking resolution? What do you need?"
3. Get a decision:
   - New committed date with reason for confidence? → Document, move on
   - Resource needed from a specific team? → Assign action item to escalating party
   - Director escalation warranted? → Confirm trigger, initiate after meeting
   - Acceptable deferral with risk acceptance? → Get Service Owner sign-off in meeting

Do not let a single item consume more than 2 minutes unless it is a P1 with active resource conflict.

### 0:10 – 0:25 — RCA Status (15 Minutes)

**For problems in "Investigating" status**:

- Is the RCA session scheduled? If yes: when?
- Is evidence collection complete?
- Any blockers: SME unavailable, log access issue, vendor holding RCA findings?
- Confirm: when will RCA be complete?

**For problems in "Root Cause Known" status**:

- Is there a fix plan?
- Is a change request in the system?
- Is the Engineering Lead aligned on the committed date?
- Is the KEDB entry published? (Workaround available to responders now?)

**Watch for**: problems that have been in "Investigating" for more than 14 days without a scheduled RCA session. That is not "investigating" — that is stalled. Treat it as a resource or priority issue and apply the escalation criteria.

### 0:25 – 0:40 — Fix Delivery Review (15 Minutes)

Work through the Fix Delivery Report.

For each problem in "Fix in Progress":

- Engineering Lead confirms: on track for committed date?
  - Yes: note, move on
  - At risk: what is the new ETA? What is blocking? Who needs to act?
  - Overdue: when did it slip? New committed date from Engineering Lead (not "we'll see") — get a date in the meeting

> **Facilitator rule**: "We'll update the ticket later" is not a meeting output. Get the date in the room so everyone hears the same commitment.

### 0:40 – 0:50 — New Problem Candidates (10 Minutes)

Work through the new candidates list. For each:

1. State the candidate and recommendation
2. Quick group vote: Approve / Reject / Defer
3. For approved: assign owner in the room (confirmed verbally by the named individual)
4. For rejected: state rationale clearly — it goes in the record
5. For deferred: state condition for reassessment and when it will be revisited

> **Facilitator rule**: this is not the place to litigate whether a problem is real. If it meets the documented threshold criteria, it is approved. If it does not, it is rejected. Criteria exist so this decision takes 30 seconds per candidate.

### Overflow Management

Any item that cannot be resolved in its time slot goes to the parking lot:

- Problem Manager notes it
- Problem Owner is responsible for resolving offline within 48 hours
- Parking lot items are reported at next week's meeting: resolved or still open?

## Decision Criteria for Common Decisions

### Approving a New Problem Record

Approve if ANY of the following:
- Recurrence threshold met: 2+ incidents in 30 days for P1/P2 CI; 3+ incidents in 90 days for P3 CI
- Customer impact warrants proactive investigation even if only one occurrence
- Trend analysis indicates the condition is worsening (even without recurrence threshold)
- Post-incident review flagged systemic condition requiring root cause investigation

Reject if ALL of the following:
- Single occurrence with no recurrence risk (no repeating conditions)
- Root cause is already known and fix is confirmed in the current change pipeline
- Impact was below threshold and no systemic condition identified

Defer (revisit in 2 weeks) if:
- Evidence is insufficient to make the determination
- Related incident is still being investigated and PIR is not complete

### Escalating to Director of Operations

| Escalation Trigger | Threshold |
|-------------------|-----------|
| P1 with no fix plan | > 10 calendar days from problem opened |
| P2 SLA breach | > 30 days past SLA milestone |
| Engineering team not engaged | RCA not started > 7 days after problem opened |
| Resource deadlock | Two teams in conflict over priority with no resolution |
| Vendor not responding | > 7 days without substantive update on P1-causing vendor problem |

When the group agrees to escalate: Problem Manager sends escalation email per the formal escalation format in `problem-manager-playbook.md` within 4 hours of the meeting. Not tomorrow. Not end of week.

### Accepting Risk on a Known Error

Conditions for acceptance:
- Fix is technically not feasible within the current planning horizon
- Fix cost exceeds approved budget and no exception is available
- Permanent fix requires architectural change that is in long-term roadmap but not current cycle

Required to accept risk:
1. Service Owner verbally confirms acceptance in the meeting
2. Risk acceptance is documented in the problem record with the specific risk statement
3. Director of Operations is notified (not asked for approval — notified)
4. KEDB entry is marked "Accepted Risk — Workaround in Place"
5. Review date is set (typically 90 days) — risk acceptance is not permanent

Do not accept risk on a P1 problem without Director-level notification. The Director does not need to approve it, but they need to know the risk is being held in the environment.

## Meeting Outputs

Problem Manager is responsible for all outputs within 24 hours of the meeting.

| Output | Owner | Deadline |
|--------|-------|----------|
| Updated problem records (status, ETA, blocking reason) | Problem Manager | 24 hours |
| New problem records opened (for approved candidates) | Problem Manager | 24 hours |
| Escalation email to Director (for escalation decisions) | Problem Manager | 4 hours |
| Action item list (owner + date for each) | Problem Manager | 24 hours |
| Meeting summary to all attendees | Problem Manager | 24 hours |

Meeting summary format (not a transcript — a decision record):

```
Problem Review Meeting Summary — [Date]

SLA BREACH DECISIONS:
- [Problem ID] [Title]: [Decision made] — [Action] — Owner: [Name] — Due: [Date]

RCA STATUS UPDATES:
- [Problem ID] [Title]: RCA session scheduled [date] / Blocked by [reason] / Root cause confirmed

FIX DELIVERY UPDATES:
- [Problem ID] [Title]: On track [date] / Slipped to [new date] — reason: [brief]

NEW PROBLEM RECORDS OPENED:
- [Problem ID] [Title] — Owner: [Name] — Priority: [P1/P2/P3]

RECORDS REJECTED:
- [Candidate description] — Reason: [brief rationale]

ESCALATIONS:
- [Problem ID] — Escalated to Director [Name] — [Date/Time]

ACTIONS:
- [Action] — Owner: [Name] — Due: [Date]

Next meeting: [Date/Time]
```

## Handling Common Derailments

### Technical Debate Breaks Out

**Signal**: Two engineers debating root cause, architecture, or solution approach

**Do not**: participate in or moderate the technical debate

**Do**: "This is an important technical question — it's outside this meeting's scope. [Name] and [Name], can you schedule 30 minutes this week to work through this? I'll note it as an action item with a 48-hour deadline."

### Required Attendee Is Absent

**If Service Owner is absent for a P1 or P2 problem they own**:
- Do not discuss that problem without them — you cannot make governance decisions without the accountable person
- Note the absence
- Problem Manager contacts them immediately after the meeting: synchronous call, not email
- If pattern of absence: escalate to SRE Manager or Director

**If Engineering Lead is absent for a problem in Fix in Progress**:
- Accept delegate only if delegate has authority to commit to a new date
- If no delegate with authority: defer the item, note absence, Engineering Lead must provide written update by COB same day

### System Access Unavailable

If the problem management system is unavailable and attendees cannot access records:

- Problem Manager should have printed or offline access to the status sheet prepared as a Monday input
- Run the meeting from the offline document — update the system after the meeting
- Never cancel or defer the meeting because of a tool outage; the decisions are more important than the system

### Someone Attempts to Re-litigate a Decision

**Signal**: "I still think this shouldn't be a problem record" (when a previous week's decision is being challenged)

**Do**: "That record was approved in a prior meeting. If you have new evidence that changes the decision criteria, please send it to me and I will add it to next week's agenda as a reconsideration item. For this meeting, we'll proceed based on the existing decision."

Do not re-open decisions in real time without new evidence. Consistency is the foundation of governance credibility.

### Meeting Runs Long

At 45 minutes if not through the agenda:

"We have 5 minutes left. We haven't covered [section]. I will handle the remaining items as follows: [decision on how to expedite or defer]. Any objections?"

Common resolutions:
- New problem candidates with clear threshold outcomes: Problem Manager decides and notifies; only edge cases come to meeting
- Fix delivery items with no blocking issues: Problem Manager collects written updates before meeting and only escalates the exceptions to the meeting
