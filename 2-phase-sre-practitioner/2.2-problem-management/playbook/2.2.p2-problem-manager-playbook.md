# Problem Manager Playbook
## Operational Reference Guide — Director/VP of Operations Lens

---

## Role Definition

### What the Problem Manager Owns

- **Problem record lifecycle** — from identification to closure, every status transition is gated by the Problem Manager
- **RCA process integrity** — ensuring RCAs are completed, are high quality, and lead to verified corrective actions
- **KEDB governance** — publishing, reviewing, and archiving known error entries
- **SLA accountability** — tracking P1/P2 problem SLAs, issuing breach notifications, escalating when needed
- **Reporting** — weekly backlog report, monthly Director report, trend metrics
- **Problem Review Meeting** — prepare inputs, chair or delegate, document outputs
- **Escalation decisions** — determining when to escalate to Director and executing that escalation
- **New problem candidate triage** — approve, reject, or defer with documented rationale

### What the Problem Manager Does NOT Own

> **Sidebar: Not Your Job**
>
> - **Root cause investigation** — you facilitate the process; engineers do the analysis
> - **Fixing anything** — you own tracking the fix; the engineering team owns delivery
> - **Running war rooms** — that is Incident Management
> - **Debugging** — you never touch systems, never write queries against production, never direct engineers on how to fix something
> - **Vendor negotiations** — you document and escalate; Vendor Management or Procurement owns the contract
> - **Change approval** — you link problem records to change requests; the CAB approves
> - **Risk ownership** — you document accepted-risk decisions; the Service Owner signs them

The Problem Manager is a process and governance role. Operational influence comes from visibility, escalation authority, and data — not from technical intervention.

## Daily Routine (30-Minute Ritual)

Run every business day, first thing. This is non-negotiable for P1/P2 environment health.

### 1. Review All Active P1 and P2 Problem Records (10 min)

For each open P1 and P2 problem:

- Has status changed in the last 24 hours?
  - Yes: verify the status transition is legitimate (correct evidence, correct approver)
  - No: is this expected (e.g., waiting on vendor fix) or a sign of stall?
- Is the SLA milestone at risk?
  - Calculate: days remaining vs. days to next milestone
  - Flag any P1 within 48 hours of SLA breach, any P2 within 5 days
- Is there a named owner actively working this?
  - No owner = escalation trigger, not a "check back tomorrow" situation

### 2. Check for New Incident Clusters Requiring Problem Records (10 min)

Pull the incident management queue or dashboard for the last 24 hours:

- Any P1 or P2 incidents closed overnight?
  - Review PIR status — is a problem candidate flagged?
- Any CI that appeared in more than one incident in the past 7 days?
  - If yes: apply recurrence threshold criteria (see ch01) — does this meet the bar for a new problem record?
- Any incident where root cause was listed as "unknown" or "under investigation"?
  - Flag for follow-up with Incident Manager — problem record may be warranted

### 3. Review Open RCA Requests for Evidence Collection Support (5 min)

For each problem in "Investigating" status:

- Has the assigned Problem Owner requested any specific evidence?
- Are there blockers to evidence collection (log retention expired, access issue, SME unavailable)?
- If RCA session is scheduled: confirm participant invites are out, pre-read is sent, room/bridge is booked

### 4. Update Problem Record Statuses Based on Overnight Progress (5 min)

- Apply any status transitions that became valid overnight (e.g., fix deployed and confirmed → move to "Fix Verified")
- Close any problems where fix has been confirmed stable for the required observation period
- Sync KEDB: if a problem closed, trigger archive process for the corresponding KEDB entry

## Weekly Cadence

### Monday (Prep Day)

**By EOD Monday**, prepare all inputs for Tuesday's Problem Review Meeting:

1. Pull updated problem records — confirm all statuses are current, not stale from last week
2. Generate SLA breach list:
   - P1 problems with any milestone overdue
   - P2 problems with any milestone >30 days overdue
   - Include: problem ID, title, breach type, days overdue, last action taken
3. Build new problem candidates list:
   - Incidents since last meeting that meet recurrence or impact threshold
   - Proposed classification (P1/P2), proposed owner
4. Compile fix delivery report:
   - All problems in "Fix in Progress"
   - Committed delivery date vs. today's date
   - On track / at risk / overdue

**SLA breach notifications**: send to Engineering Leads for every P1/P2 approaching or past SLA deadline. Use the formal escalation email format. Do not send casual Slack messages for SLA breach communication — the paper trail matters.

### Tuesday (Meeting Day)

Run Problem Review Meeting per `problem-review-meeting-playbook.md`.

### Wednesday–Thursday (Follow-Through)

- Update all problem records with decisions made in Tuesday's meeting (within 24 hours of meeting)
- Open new problem records approved in Tuesday's meeting — assign owner, set initial status to "Logged"
- Send escalation communications where decided in meeting

### Friday (Backlog Review)

- Update Problem Backlog report:
  - Total open problems by priority and status
  - Week-over-week change (new, closed, escalated)
  - Average age by priority bucket
  - Problems with no activity in past 7 days — flag for attention
- Review KEDB for entries 90+ days without update:
  - Flag as stale
  - Notify Problem Owner, require review within 5 business days

## Monthly Cadence

### First Week of Month

**Prepare monthly report for Director of Operations**:

- Open problems by priority: count, average age, SLA compliance rate
- Problems closed this month: how many, average time to closure
- RCA completion rate: problems where RCA was completed within SLA vs. total problems that reached "Investigating" status
- Fix delivery rate: % of committed fix dates met
- Recurring incident trend: incidents per known error this month vs. prior 3 months
- Top 3 systemic themes: what is the environment telling you about where structural problems concentrate?

### Mid-Month

**Fix delivery rate review** — identify teams consistently missing committed fix dates:

- Pull all problems in "Fix in Progress" or "Fix Verified" closed in the past 90 days
- Group by engineering team / service owner
- Flag any team with >30% missed fix date rate
- Document finding; include in monthly report; escalate to SRE Manager or Director if pattern persists 2+ months

### End of Month

**Archive resolved KEDB entries**:

1. Query KEDB for entries where linked problem status = "Closed — Fix Verified" AND fix has been stable for 60+ days
2. Confirm with Problem Owner: no recurrence in the observation period?
3. Archive entry: status = "Archived — Fix Deployed [date]"
4. Entry remains searchable — archival is not deletion

**Review accepted-risk problems**:

- Pull all problems with status "Accepted Risk"
- For each: is the stated risk condition still true?
  - Has the system changed in a way that changes the risk profile?
  - Has the workaround degraded?
  - Has the business context changed (new regulatory requirement, new customer segment)?
- If risk basis has changed: reopen for reassessment; notify Service Owner and Director

## Escalation Rules

### What Triggers Escalation to Director

| Condition | Threshold | Action |
|-----------|-----------|--------|
| P1 problem with no fix plan | >10 calendar days from problem opened | Escalate |
| P2 problem SLA breach | >30 days past SLA milestone | Escalate |
| Recurring incident from known error | >5 occurrences in a single calendar month | Escalate |
| Vendor missing committed dates | Fix date missed + no new commitment | Escalate |
| Engineering team blocking RCA | RCA not started >7 days after problem opened | Escalate |
| Accepted-risk problem reactivated | Risk condition has changed | Escalate |

### How to Escalate: Formal Escalation Email Format

```
To: [Director of Operations]
CC: [SRE Manager], [Service Owner], [Engineering Lead]
Subject: Problem Escalation — [Problem ID] — [Short Title] — [Escalation Reason]

Director [Name],

I am formally escalating the following problem record per our escalation criteria.

PROBLEM RECORD: [ID]
PRIORITY: [P1/P2]
OPENED: [date]
CURRENT STATUS: [status]
ESCALATION REASON: [specific criterion triggered — e.g., "No fix plan after 10 days"]

CURRENT SITUATION:
[2–3 sentences: what is broken, what has been tried, what is blocking progress]

CUSTOMER IMPACT:
[Current or potential — be specific: number of users, transaction volume affected, SLA exposure]

WHAT IS NEEDED:
[Specific decision or resource you need from the Director — be explicit]

RECOMMENDED ACTION:
[Your recommendation — what should happen next]

Next update: [date/time]

[Problem Manager Name]
Problem Management
```

**Expected response time from Director**: 4 business hours for P1 escalation, 1 business day for P2.

## Problem Record Management

### Opening a New Problem Record

Required fields at creation:

| Field | Required Value |
|-------|---------------|
| Title | Descriptive — include CI and symptom, not just "issue with X" |
| Priority | P1 / P2 / P3 per classification criteria |
| Affected CI | Specific configuration item — not "production" |
| Problem Owner | Named individual, not a team |
| Source | PIR / Trend Analysis / Proactive / Vendor |
| Source Incident(s) | Link to triggering incident record(s) |
| Description | Observed symptoms, confirmed impact, what is unknown |
| Opened Date | Today |
| Status | Logged |

Do NOT create a problem record without a named Problem Owner. "TBD" is not an owner. If you cannot identify an owner, escalate immediately — an orphaned problem record is a governance failure.

### Status Transitions: Criteria and Required Fields

| From | To | Criteria | Required Fields |
|------|-----|----------|----------------|
| Logged | Investigating | Problem Owner confirmed, RCA session scheduled or underway | RCA start date, Owner contact |
| Investigating | Root Cause Known | RCA completed, root cause statement documented, group confirmed | Root cause text, RCA completion date |
| Root Cause Known | Fix in Progress | Fix plan approved, change request linked | Change request ID, committed fix date |
| Fix in Progress | Fix Verified | Fix deployed, observation period complete, no recurrence | Fix deployment date, verification method |
| Fix Verified | Closed | All corrective actions verified, KEDB archived | Closure date, KEDB archive status |
| Any | Accepted Risk | Service Owner signed off, Director notified | Risk acceptance statement, Service Owner approval, Director notification date |
| Any | Cancelled | Duplicate, not reproducible, or scope error | Cancellation reason, reference to correct record if duplicate |

### Managing Rejected Problem Candidates

When a proposed problem record is rejected in Problem Review Meeting:

1. Record the rejection in your tracking log: problem candidate description, rejection date, rejection rationale, decision maker
2. In the source incident record: add a note — "Problem candidate reviewed [date] — rejected: [reason]"
3. Set a staleness check: if the same symptoms recur within 60 days, the rejection is automatically reopened for reconsideration
4. Never delete the candidate record — it is an audit trail

## Integration Points

### Incident Management

- **Input you receive**: problem candidates from PIR (Post-Incident Review) process
- **Your action**: review each candidate within 48 hours; approve/reject with documented rationale
- **What you return**: confirmation of problem record opened (with ID) or rejection reason
- **Escalation handoff**: when a P1 incident is still active and a pattern is already known in KEDB, you push the KEDB entry to the Incident Commander — this is real-time, not post-incident

### Change Management

- **Input you receive**: notification when a change request is linked to your problem record
- **Your action**: track change completion; when change closes, update problem record fix status
- **CAB participation**: Problem Manager should attend CAB when a change is tagged as problem-driven — you provide context on urgency and risk of deferral
- **What you return**: problem record update confirming fix delivery and observation period status

### KEDB

- **When to create**: immediately when problem reaches "Root Cause Known" AND a workaround exists
- **When to update**: after any incident where workaround was attempted; after any change to affected CI; monthly review
- **When to archive**: when problem reaches "Closed" and 60-day observation period complete
- **Ownership**: Problem Manager publishes and archives; Problem Owner maintains content accuracy
