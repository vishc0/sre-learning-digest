# KEDB Governance Playbook
## Known Error Database — Operational Guide — Director/VP of Operations Lens

---

## Why KEDB Governance Fails

KEDB fails in most organizations for one of three reasons:

1. **Creation gap** — entries are never created because no one owns the trigger
2. **Staleness** — entries are created once and never reviewed; workaround steps break as systems evolve
3. **Discoverability** — entries exist but incident responders don't know to look for them or can't find them at 2am

This playbook addresses all three. A KEDB that responders trust and actually use during incidents is worth more than any post-incident report.

## Section 1 — Creating a New KEDB Entry

### Trigger

A KEDB entry must be created when a problem record transitions to "Root Cause Known" status **AND** a workaround exists — even a partial one.

Do not wait for the permanent fix. The workaround is what incident responders need now. The fix delivers later.

### Required Inputs Before You Start

| Input | Source | Who Provides |
|-------|--------|--------------|
| Confirmed root cause statement | Approved RCA document | Problem Owner |
| Workaround steps (tested and verified) | Engineer who used them during the incident | Incident Responder / SME |
| Affected CI (specific configuration item) | CMDB | Problem Owner |
| Source problem record ID | Problem Management system | Problem Manager |
| Symptom description (2am language) | Incident record, war room notes | Incident Responder |
| System version / environment (if version-specific) | Change Management / CMDB | System SME |

> **Critical**: The workaround steps must be verified by the engineer who actually used them during the incident — not written from memory two weeks later by someone who wasn't there. Unverified workarounds in KEDB are worse than no entry — they waste response time on steps that don't work.

### Entry Creation Steps

1. **Open a new KEDB draft entry** in your service management platform (ServiceNow / Jira Service Management)

2. **Write the symptom description** — use incident responder language, not architect language:
   - Wrong: "Asynchronous message processing subsystem exhibits elevated queue accumulation due to consumer thread pool saturation"
   - Right: "Message queue depth rising rapidly; consumers appear stopped; alerts for [queue-name] firing; RabbitMQ management UI shows 0 consumer throughput"
   - Test: would an SRE who has never touched this system understand what they are looking at at 2am? If not, rewrite.

3. **Write the root cause summary** — copy the confirmed root cause statement from the RCA. Do not paraphrase. Accuracy over brevity here.

4. **Write the workaround steps** — numbered, sequential, with specific commands or UI navigation:
   ```
   1. [Verify symptom]: Check queue depth for [queue-name] in RabbitMQ management UI (https://[host]/api/queues). Confirm depth > threshold and consumers = 0.
   2. [Confirm scope]: Check whether issue is isolated to this queue or affects all queues on this vhost.
   3. [Apply workaround]: Restart [service-name] consumer pods: kubectl rollout restart deployment/[name] -n [namespace]
   4. [Verify recovery]: Monitor queue depth for 5 minutes post-restart. Should drop below 1,000 within 3 minutes if workaround is effective.
   5. [If workaround fails]: Escalate to [SME name/team] — queue may require manual message requeue or dead-letter investigation.
   ```

5. **Complete required KEDB fields**:

   | Field | Content |
   |-------|---------|
   | Title | Short, searchable — include CI name and failure type |
   | Symptom | 2am language (step 2 above) |
   | Root Cause Summary | From RCA (step 3 above) |
   | Workaround | Numbered steps (step 4 above) |
   | Affected CI | Specific CMDB reference |
   | Problem Record ID | Link — so reviewers can read the full RCA |
   | Fix ETA | From linked change request, or "Under Investigation" |
   | Entry Author | Your name |
   | Last Reviewed | Today |
   | Status | Draft (until published) |
   | Workaround Verified By | Name of engineer who confirmed steps work |

6. **Quality check before publishing**:

   - [ ] Has someone other than the author tested the workaround steps against the actual system?
   - [ ] Is the symptom description written in incident responder language (would they recognize this at 2am)?
   - [ ] Does every workaround step have a verification check?
   - [ ] Is the affected CI linked to the correct CMDB entry?
   - [ ] Is the source problem record linked?
   - [ ] Does the entry include what to do if the workaround fails?

7. **Publication**:
   - Problem Owner creates draft and self-reviews
   - Problem Manager reviews for quality gate completeness
   - Problem Manager publishes entry (changes status from Draft to Active)
   - Notify Service Owner and on-call team lead that new KEDB entry is available

## Section 2 — KEDB Entry Review and Update Cadence

### Monthly Review (Problem Owner)

Every Problem Owner reviews their active KEDB entries monthly. Review criteria:

- [ ] Are the workaround steps still accurate for the current system state?
- [ ] Has any change to the affected CI been deployed since the last review? If yes: re-verify workaround steps.
- [ ] Is the Fix ETA still realistic? If not: update with current estimate or mark "No ETA."
- [ ] Does the symptom description still match current system behavior (e.g., alert names may have changed)?
- [ ] Is the Problem Owner contact information current?

Update the "Last Reviewed" date after completing the review, even if no changes were made. The date is the governance signal — a stale date triggers the staleness process regardless of entry accuracy.

### Immediate Review Triggers (Out-of-Cycle)

Trigger an immediate out-of-cycle review when any of the following occur:

| Trigger | Who Initiates | Response Time |
|---------|--------------|---------------|
| Change deployed to affected CI | Change Manager notifies Problem Owner | 5 business days |
| Incident where workaround was attempted and failed | Incident Manager notifies Problem Owner | 48 hours |
| Incident where workaround succeeded but required modification | On-call SRE notifies Problem Owner | 5 business days |
| System major version upgrade (CI is affected) | Problem Manager review of change schedule | 5 business days post-upgrade |

For workaround failures: do not just update the entry — investigate why the workaround failed. This may indicate a deeper change to the underlying problem, requiring the RCA to be reopened.

### Review Checklist (Full)

```
KEDB Entry Review — [Entry ID] — [Date]

Reviewer: [Name]
Last Review Date: [Previous date]

WORKAROUND ACCURACY
[ ] Workaround steps reflect current system state
[ ] All command syntax is current (no deprecated flags, old endpoint names)
[ ] All service names, namespaces, queue names, host references are current
[ ] Verification checks in the workaround are still valid indicators of success

FIX STATUS
[ ] Fix ETA is current
[ ] If fix has been deployed: trigger archive process (see Section 5)

SYMPTOM ACCURACY
[ ] Alert names referenced in symptom description still exist
[ ] Dashboard/UI paths referenced are still valid
[ ] Threshold values mentioned still reflect actual alert thresholds

OWNERSHIP
[ ] Problem Owner is still the correct person
[ ] SME escalation path referenced is still current

RESULT: [ ] No changes needed | [ ] Updated (describe changes below) | [ ] Trigger archive | [ ] Escalate

Changes made: _______________
```

## Section 3 — Linking KEDB to Runbooks and Incident Management

### Runbook Integration

Every runbook that covers a system with active KEDB entries must have a "Known Failures" section:

```
## Known Failures

The following known errors are documented in the KEDB. Check these entries first when
symptoms match before proceeding with standard diagnostic steps.

| KEDB ID | Symptom Match | Workaround Available |
|---------|--------------|---------------------|
| KEDB-0042 | Queue consumers drop to zero, depth rising | Yes |
| KEDB-0051 | Payment API returning 503, downstream timeout | Partial |
```

Update this section whenever a new KEDB entry is created for a CI that this runbook covers. Problem Manager owns notifying the runbook owner.

### Service Management Platform Integration (ServiceNow / Jira SM)

Configure the CI relationship so that when a new incident is created with the affected CI, active KEDB entries surface automatically:

- In ServiceNow: use the Related Records tab; configure CI-to-KEDB relationship
- In Jira SM: use linked issues; configure automation rule to surface KEDB entries when CI matches
- Test this quarterly: open a test incident for a CI with an active KEDB entry; verify the entry appears without manual search

If automatic surfacing is not technically available, document in the incident triage runbook: "Before proceeding, search KEDB for [affected CI name]."

### Alerting Platform Integration (PagerDuty / Opsgenie)

Configure alert routing notes to surface KEDB entries based on alert name match:

- In PagerDuty: use the "Runbook URL" field in service configuration; link to KEDB search for that CI
- In Opsgenie: use the "Actions" field in alert routing to include KEDB search link
- Advanced: if your alerting platform supports custom alert templates, embed the KEDB ID directly in the alert body for known error patterns

### Quarterly Integration Test

Every quarter, run the following test for the top 5 highest-traffic KEDB entries:

1. Open a test incident with the affected CI
2. Verify KEDB entry surfaces in the incident UI
3. Click through to the KEDB entry
4. Verify the workaround steps are current
5. Document test result: Pass / Fail / Requires Update

Failures go into the next Problem Review Meeting as integration maintenance items.

## Section 4 — Staleness Detection and Management

### Staleness Criteria

An entry is considered stale if any of the following are true:

| Criterion | Threshold |
|-----------|-----------|
| Last review date | > 90 days ago |
| System version at entry creation vs. current | > 2 major versions behind |
| Workaround success rate (if tracked) | < 60% over last 10 incidents |
| Fix ETA date | Passed with no update |

### Staleness Response Process

1. **Detect**: weekly Problem Manager review of last-review dates; automated staleness flag if platform supports it

2. **Flag**: mark entry status as "Stale — Under Review"

3. **Notify**: email Problem Owner with:
   - Entry ID and title
   - Why it is flagged as stale (specific criterion)
   - Required action: review and update or confirm still accurate
   - Deadline: 5 business days

4. **Follow-up**: if no response in 5 business days:
   - Suspend the entry: status = "Suspended — Verify Before Use"
   - Notify on-call team lead: this entry is unreliable, confirm with SME before applying
   - Escalate to Service Owner for owner assignment correction if Problem Owner is unresponsive

5. **Suspended entry behavior**:
   - Entry remains visible in KEDB search
   - Displays banner: "VERIFY BEFORE USE — This entry has not been reviewed in 90+ days. Confirm workaround steps with [SME contact] before applying."
   - Suspended entries are tracked in Problem Manager staleness report

### Staleness Report (Monthly)

| Metric | Formula | Target |
|--------|---------|--------|
| Stale entries | Count with last review > 90 days | 0 at start of month |
| Suspended entries | Count with status = Suspended | < 5% of total entries |
| Average days since last review | Sum of days / entry count | < 45 days |
| Entries reviewed this month | Count reviewed in current month | > 30% of total entries |

Present this report in the monthly Director of Operations report. Sustained staleness above targets is a governance failure.

## Section 5 — Archiving Resolved Entries

### Archive Criteria

An entry should be archived when all three conditions are met:

1. The permanent fix has been deployed to production
2. The fix has been verified as effective (no recurrence of the known error pattern)
3. A 60-day observation period has passed with no recurrence

All three must be true. Do not archive after fix deployment alone — a fix may be incomplete, and the workaround may still be needed.

### Archive Process

1. **Problem Owner confirms** fix is deployed and observation period is complete
2. **Problem Manager verifies**: pull incident history for the affected CI for the past 60 days. Zero recurrences of the known error pattern?
3. **Archive entry**:
   - Status → "Archived — Fix Deployed [date]"
   - Add archive note: "Permanent fix deployed via Change [CR ID] on [date]. Observation period complete [date]. No recurrence confirmed."
4. **Archived entry behavior**:
   - Removed from active KEDB search results (responders do not see it in standard queries)
   - Remains searchable via explicit archive search or by entry ID
   - Referenced in original problem record and RCA document — links remain active

> **Do not delete archived entries.** An archived KEDB entry is historical evidence. It is referenced by the RCA, the problem record, and potentially by audit processes. Deletion breaks the audit trail.

### Archive Audit (Quarterly)

Every quarter:

1. Pull all archived entries from the past 12 months
2. For each: have incidents with the same symptom pattern recurred since archival?
   - Yes: reactivate the entry, reopen the problem record, initiate RCA review
   - No: confirm archive is correct; no action needed

Reactivation rate (archived entries that were reactivated) should be < 5%. A higher rate indicates fix verification was premature or fix delivery was incomplete.

## Section 6 — KEDB Quality Metrics

Track these four metrics monthly. Include in Problem Manager monthly report and Director of Operations report.

### Match Rate

**Definition**: percentage of incidents where an active KEDB entry was found for the affected CI and symptom

**Formula**: (incidents where KEDB was searched and entry found) / (total incidents) × 100

**Target**: > 40% for a mature operations environment

**Interpretation**: A match rate below 40% typically means either the KEDB is under-populated (not enough entries) or discoverability is broken (entries exist but responders don't find them). Diagnose which before treating.

### Workaround Success Rate

**Definition**: percentage of incidents where a KEDB workaround was applied and succeeded in restoring service or limiting impact

**Formula**: (incidents where workaround succeeded) / (incidents where workaround was attempted) × 100

**Target**: > 80%

**Interpretation**: Below 80% means entries are stale, workarounds are incomplete, or responders are applying workarounds to non-matching symptoms. Sample 5 failure cases and categorize: stale steps, misapplication, or incomplete workaround.

### Staleness Rate

**Definition**: percentage of active KEDB entries flagged as stale at any point during the month

**Formula**: (entries with last review > 90 days) / (total active entries) × 100

**Target**: < 10%

**Interpretation**: Above 10% means the monthly review cadence is not being followed. Escalate to Service Owners whose entries are stale.

### Archive Rate

**Definition**: percentage of closed problem records where the corresponding KEDB entry was archived within 30 days of problem closure

**Formula**: (closed problems with KEDB archived within 30 days) / (closed problems with a KEDB entry) × 100

**Target**: 100%

**Interpretation**: Any entry below 100% means a KEDB entry for a fixed error is still showing as active — responders may apply a workaround for a problem that no longer exists. Run a monthly reconciliation: problem closed → KEDB archived.

## KEDB Health Check Calendar

### Monthly Tasks (Problem Manager)

| Week | Task |
|------|------|
| Week 1 | Staleness report: flag all entries > 90 days without review; notify Problem Owners |
| Week 1 | Archive reconciliation: identify closed problems with no KEDB archive action taken |
| Week 2 | Follow up on staleness notifications: any non-responders? Escalate to Service Owner |
| Week 3 | New entry audit: entries created this month — quality gate check (were all fields completed?) |
| Week 4 | Metrics calculation: compile match rate, success rate, staleness rate, archive rate |
| Week 4 | Update monthly Director report with KEDB metrics section |

### Quarterly Tasks (Problem Manager + KEDB Owner)

| Month | Task |
|-------|------|
| Q-end Month | Integration test: verify top 5 entries surface correctly in incident management and alerting platforms |
| Q-end Month | Archive audit: review archived entries from past 12 months for recurrence |
| Q-end Month | Runbook alignment: spot-check 5 runbooks that cover CIs with active KEDB entries — is the "Known Failures" section current? |
| Q-end Month | KEDB coverage assessment: which CIs have the most incidents but zero KEDB entries? These are population gaps. |
