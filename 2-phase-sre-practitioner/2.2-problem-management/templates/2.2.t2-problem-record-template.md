# Problem Record Template

> **How to use this template**: Fill every required field before submitting for triage. Optional fields should be completed as information becomes available — do not leave them blank at closure. Inline guidance (italicized) explains what distinguishes a good entry from a low-quality one. Delete guidance text before publishing.

## PROBLEM DESCRIPTION

### Short Description _(Required)_
> **Format**: [Symptom] in [affected system/service] causing [observable impact]. Max 120 characters.
>
> **Good**: "RabbitMQ consumer lag exceeds 500K messages on payment-events queue during peak traffic"
>
> **Bad**: "RabbitMQ issues" — provides no triage context; cannot be distinguished from other RabbitMQ problems at 2am

```
[Enter short description here]
```

### Long Description _(Required)_
> **What to include**: Full technical description of the observed failure behavior. Include: what is failing, how it fails, under what conditions, what the first observable symptom is, and how it was detected. Write for a senior engineer who has no prior context.
>
> **Good**: Includes specific component names, error messages, metric values, log patterns, and detection mechanism.
>
> **Bad**: A copy of the incident summary. Problem description describes a recurring structural condition, not a single event.

```
[Enter long description here]

Detection method: [alert name / Splunk search / customer report / proactive review]
First occurrence confirmed: [date]
Most recent occurrence: [date]
Occurrence count (if known): [N times in last 30 days]
```

## IMPACT ASSESSMENT

### Customer Impact Type _(Required)_
> Select all that apply. Do not understate — err toward broader impact classification.

- [ ] Service Unavailability (customers cannot complete transaction)

- [ ] Data Integrity Risk (incorrect data shown or stored)
- [ ] Security/Privacy Exposure

### Estimated Users Affected _(Required)_

## WORKAROUND

### Workaround Available? _(Required)_
> [ ] Yes — complete all fields below
> [ ] No — state why not (technically infeasible / investigation in progress / not yet tested)
> [ ] Partial — describe limitations

### Workaround Summary _(Required if available)_
> **Write for an on-call engineer at 2am who has never seen this before.** Each step must be numbered. Do not assume familiarity with the system beyond standard SRE tools.

```
[Enter numbered workaround steps here]

1. 
2. 
3. 

Expected outcome after completing steps: [describe what normal looks like]
Time to implement: approximately [N] minutes
```

### KEDB Entry _(Required if workaround exists)_
> A KEDB entry must be created and linked within 24 hours of confirming the workaround works.

```
KEDB Entry ID: KE-XXXX
KEDB Entry URL: [link]
KEDB Entry Status: Draft / Active / Verified
```

### Workaround Risk _(Required if workaround exists)_
> **Do not skip this.** Every workaround has a risk. Stating "no risk" is almost always wrong. What could the workaround break? What must be monitored while it is active? What is the rollback if it makes the situation worse?

```
Risk: [describe what could go wrong]
Monitor for: [specific metric or alert to watch during/after workaround]
Rollback procedure: [steps if workaround makes it worse]
Risk level: High / Medium / Low
```

### RCA Status _(Required)_

> [ ] Timeline Analysis
> [ ] FMEA (for proactive problems)
> [ ] Other: [specify]
>
> _Rationale for method choice_: [why this method fit this failure type]

### Root Cause Statement _(Required once complete)_
> **Use this exact format.** This format forces you to separate the structural condition (what made the system vulnerable) from the triggering event (what activated the failure) from the failure mode (how it manifested).
>
> **Template**: "Due to [structural condition], when [triggering event] occurred, [failure mode] resulted in [customer impact]."
>
> **Good**: "Due to the absence of consumer lag alerting thresholds on the payment-events queue, when batch job submission volume increased 3× during month-end processing, RabbitMQ consumer lag exceeded queue capacity resulting in 47-minute payment notification delays for 800K customers."
>
> **Bad**: "The queue filled up because of too much traffic." — describes symptoms, not structural condition.

```
Root Cause Statement:
"Due to [structural condition], when [triggering event] occurred, [failure mode] resulted in [customer impact]."
```

### Contributing Factors _(Required once complete — minimum 3)_
> **Contributing factors are conditions that made the root cause possible or worse.** They are not the root cause itself. Each factor should have evidence. If you cannot cite evidence, label it as suspected.

## FIX PLAN

### Proposed Fix _(Required once root cause identified)_
> **Describe the permanent fix** — not the workaround. What structural change eliminates the root cause? Include the specific component being changed, the nature of the change (config / code / architecture / process), and why this fix addresses the root cause statement above.

```
Fix description: [what will change]
Component(s) affected: [list]
Fix type: Configuration / Code / Architecture / Process / Vendor Action
Why this eliminates the root cause: [explicit link back to root cause statement]
```

## CLOSURE CRITERIA

> **All required items must be checked before closing the problem record.** The Problem Manager is accountable for verifying each item. Do not close with unchecked required items — document a waiver with approval if any item is genuinely not applicable.

### Technical Closure
- [ ] **[Required]** Permanent fix deployed to production
- [ ] **[Required]** Fix verified in production — verification method: [describe how, not just "tested"]
- [ ] **[Required]** Monitoring confirmed to detect recurrence if fix fails
- [ ] **[Optional]** Regression test added to CI/CD pipeline

### Knowledge Management
- [ ] **[Required]** KEDB entry updated to "Archived — Fix Deployed" status
- [ ] **[Required]** RCA report finalized and approved
- [ ] **[Optional]** Runbook updated if operational procedure changed

### Incident Hygiene
- [ ] **[Required]** All related incident records linked to this problem record
- [ ] **[Required]** Incident records note closure of underlying problem
- [ ] **[Optional]** Post-incident review updated if applicable

### Risk Management
- [ ] **[Required if fix not deployed]** Risk acceptance documented with approver, review date, and conditions under which risk acceptance must be revisited
- [ ] **[Required if fix not deployed]** Interim monitoring in place proportional to residual risk

### Closure Notes
```
Verified by: [name, role]
Verification method: [specific — not "testing complete"]
Residual risk after closure: [None / Low / Medium — if Medium, risk acceptance required]
```
