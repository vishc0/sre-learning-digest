# Known Error Database (KEDB) Entry Template

> **Who uses this**: The on-call engineer who receives a 2am alert. Write every field for that person. Assume they know your platform's tools (Splunk, kubectl, RabbitMQ management UI) but have zero prior context on this specific failure. Architectural elegance is not the goal — operational clarity is.
>
> **Quality standard**: If an engineer cannot apply this workaround without asking anyone a question, the entry is not ready for Active status.

---

## IDENTIFICATION

| Field | Value |
|---|---|
| **Known Error ID** | KE-XXXX _(auto-generated or sequential; never reuse)_ |
| **Title** | [Symptom as the on-call engineer would describe it — not architectural cause] |
| **Parent Problem Record ID** | PR-YYYY-XXXX |
| **Date Created** | YYYY-MM-DD |
| **Created By** | [Name, Role] |
| **Date Last Reviewed** | YYYY-MM-DD |
| **Review Due Date** | YYYY-MM-DD _(max 90 days from last review; P1 errors reviewed monthly)_ |
| **Status** | Active / Suspended — Verify Before Use / Archived — Fix Deployed |

> **Title guidance**:
> - Good: "Payment service returns 504 errors for >15% of requests during peak traffic windows"
> - Bad: "RabbitMQ consumer configuration insufficient for peak load" — this is the cause, not the symptom the on-call engineer sees

**Status definitions**:
- **Active**: Workaround is verified working. Use with confidence.
- **Suspended — Verify Before Use**: Workaround worked previously but recent changes may have altered the environment. Verify steps are still valid before applying.
- **Archived — Fix Deployed**: Permanent fix is in production. Problem is closed. This entry is retained for historical reference. Do NOT apply the workaround — the underlying condition no longer exists.

### Step 1: Confirm You Are Seeing This Error

> Verify before acting. Never skip this step.

```
Action: Run the following Splunk search to confirm the error pattern:
[Search string]

Expected output: [N] events in the last 15 minutes matching [pattern]

If output matches: proceed to Step 2
If no events found: this may not be the error described here — re-check alerts and escalate to [team] if unsure
```

```
- [Specific guidance — check this log, try this alternative, escalate to this team]
```

### Step 3: [Action Name]

```
Action: [Specific action]

Expected output: [what success looks like]

If expected output is NOT seen:
- [Guidance]
```

### Step 4: Verify Resolution

> Do not close the incident until you have confirmed the customer-facing symptom is resolved, not just that the workaround steps completed.

```
Action: Verify that [specific observable indicator] has returned to normal:
- Alert [alert name] has cleared: Y / N
- Metric [metric name] has returned to [normal range]: Y / N
- Error rate in Splunk has dropped below [threshold]: Y / N

Expected: All three indicators showing normal within [N] minutes of completing steps

If not resolved within [N] minutes: proceed to Escalation section below
```

### Escalation If Workaround Does Not Work

```
Escalate to: [Team name] — [contact method (PagerDuty policy / Slack channel / phone)]
Information to provide when escalating:
1. Confirmation that you followed Steps 1–4 above
2. Output of Step 1 (paste the Splunk result)
3. Output of Step [N] (the step that did not produce expected output)
4. Current metric readings from [dashboard]
```
