## EXECUTIVE SUMMARY

> **Purpose**: Five bullets that a VP or Director can read in 90 seconds and understand: what happened, who was affected, why it happened, what is being done, and whether it will happen again. No jargon. No hedging. No "we are investigating."
>
> **Format rule**: Each bullet is one sentence. Present tense for ongoing actions; past tense for completed events.

1. **What happened**: [One sentence — service name, failure type, duration, detection method]
2. **Customer impact**: [One sentence — number affected, nature of impact, severity]
3. **Root cause headline**: [One sentence — structural condition in plain language, not technical jargon]
4. **Fix committed**: [One sentence — what change is being made, who owns it, when it will be complete]
5. **Recurrence risk**: [One sentence — likelihood of recurrence before fix is deployed, what interim protection is in place]

> **Evidence requirement**: Every row must have an evidence source. "Engineer recalls" is a valid source but low confidence — note it. Unverified events should be marked (Unverified).

> **Event types**:
> - **Detection**: first indication something was wrong

> - **Diagnosis**: investigation activity — commands run, logs checked

## CONTRIBUTING FACTORS

> **Critical distinction**: Contributing factors are conditions that made the failure possible, faster, worse, or harder to detect. They are not the root cause. List them all — even the uncomfortable ones. Organizational and process factors belong here as much as technical ones.
>
> **Each factor needs evidence.** If you believe a factor contributed but cannot prove it, mark as "Suspected" and document the evidence gap.

| # | Factor Description | Evidence | Category | Confidence | Link to Root Cause |
|---|---|---|---|---|---|
| 1 | | [Specific evidence: log entry, metric, config value] | Technical | Confirmed / Suspected | [How this factor enabled or worsened the root cause] |
| 2 | | | Process | | |
| 3 | | | Human | | |
| 4 | | | Organizational | | |
| 5 | | | Environmental | | |

**Category definitions**:
- **Technical**: System design, configuration, code, dependencies, infrastructure
- **Process**: Missing or inadequate procedures, review gates, runbooks, change controls
- **Human**: Individual decision-making, error, skill gap — always ask what process or system allowed the human error to have this impact
- **Organizational**: Team structure, ownership gaps, communication failures, capacity, incentive misalignment
- **Environmental**: External dependencies — vendor outages, cloud provider events, network conditions outside your control

> **Organizational and process factors are not optional.** A pure technical root cause statement that ignores the process conditions that allowed it to occur will produce corrective actions that only fix the symptom. If your contributing factors list has no Process or Organizational items, revisit the analysis.

## ROOT CAUSE STATEMENT

> **This is the most important section of the report.** One paragraph. Precise. Evidence-backed. If you cannot support every claim with evidence from the Evidence Inventory section, you do not have a root cause — you have a hypothesis. Return to investigation.
>
> **Required format**: "Due to [structural condition — the underlying vulnerability], when [triggering event — what activated the failure], [failure mode — how the system failed] resulted in [customer impact — what customers experienced]."
>
> The structural condition describes WHY the system was vulnerable. The triggering event describes WHAT activated the failure. Eliminating the triggering event is not a fix — it will recur. Eliminating the structural condition is the fix.

### Root Cause Statement

```
Due to [structural condition], when [triggering event] occurred, [failure mode] resulted in [customer impact].
```

## CORRECTIVE ACTIONS

> **Quality bar for corrective actions**: Each action must be specific enough that a different person could execute it without asking clarifying questions. "Improve monitoring" is not an action. "Add PagerDuty alert on RabbitMQ consumer lag > 100K messages with 5-minute evaluation window for payment-events queue" is an action.
>
> **Action types**:
> - **Prevention**: Eliminates the structural condition so the root cause cannot recur
> - **Detection**: Ensures faster identification when similar conditions occur
> - **Response**: Improves speed or effectiveness of incident response when it does occur

### Action Completeness Check
> Before finalizing this section: Does at least one action address the structural condition directly (Prevention type)? If not, the corrective action set is incomplete — you have only detection and response improvements, not a fix.

- [ ] At least one Prevention action addresses the structural condition directly
- [ ] Every evidence gap has a corresponding corrective action (Detection type)
- [ ] Every action has a named individual owner (not a team)
- [ ] Every action has a specific, measurable verification method
- [ ] No action is "review process" or "improve monitoring" without specifics
