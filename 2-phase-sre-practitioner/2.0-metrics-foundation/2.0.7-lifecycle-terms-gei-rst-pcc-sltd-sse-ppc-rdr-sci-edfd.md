# Chapter 7: Service Lifecycle Metrics — Admission, Transition, and Decommission

**Nine formulae that govern the full lifetime of a service: from first deployment through migration to safe shutdown.**

---

## Who This Chapter Is For

You have shipped production systems. You know what it costs when a rollout goes wrong at 2 AM, when a migration drags on for six months, or when you try to decommission a service and discover twelve undocumented callers three days before the cutover. These nine metrics exist to make those situations quantitative — so the decision to proceed, slow down, or stop is driven by math rather than gut feel and meeting room politics.

Each metric covers one phase of service life:

- **Growth**: GEI, RST, SLTD — admission, statistical signal, traffic events
- **Transition**: PCC, SSE, PPC — migration economics, concurrency safety, patch windows
- **Validation and Decommission**: RDR, SCI, EDFD — fix verification, shadow callers, external deadlines

The running example throughout is an e-commerce platform — a shopping cart system with authentication, payments, inventory, recommendations, and search. The numbers are concrete and consistent across all nine sections.

## Part A: Growth Terms

### Term 1 — GEI: Growth Exposure Index

**Formula:**

```
GEI = (new_deps × avg_tier_weight_of_deps) / (test_coverage × canary_hours)
```

**Threshold:** < 1.0 proceed normally | 1.0–3.0 extended canary | > 3.0 block

---

#### The Concept

Think about how a good engineering team handles adding a new engineer to the on-call rotation. Before that person carries the pager, you ask three questions: How many critical systems are they now responsible for? What training have they completed? How many hours have they shadowed before taking live calls?

A service joining your production call chain is the same problem. You are handing it a pager. The GEI answers those three questions numerically:

- **new_deps × avg_tier_weight_of_deps** — how many critical systems does this service now touch, and how critical are they? A service that calls Auth and Payment is inheriting more operational risk than one that calls only the Recommendation service. The tier weight (tw) encodes criticality — P0 dependencies cost more than P3 ones.
- **test_coverage** — how well-trained is this service? A service with 90% test coverage has demonstrated it can handle edge cases. A service at 40% coverage is still learning.
- **canary_hours** — how long has it shadowed production? An hour of canary exposure is like one on-call shift. Four hours is a week. Canary duration is the strongest lever you have on GEI.

The ratio gives you an exposure score. A low GEI means the service is well-prepared relative to the risk it is taking on. A high GEI means you are promoting someone to on-call before they are ready.

One important operational insight: GEI is not a static property of the service. It is a property of the deployment decision. The same service with the same dependencies will have a GEI of 0.88 with a four-hour canary and a GEI of 3.54 with a one-hour canary. The formula does not judge the service — it judges the deployment plan.

The denominator is where operational discipline lives. Test coverage requires weeks of engineering investment. Canary hours require only a scheduling decision. A team that understands GEI will negotiate canary window length before committing to a release timeline — because that negotiation changes the math directly and immediately.

---

#### Formula Components

```
GEI = (new_deps × avg_tier_weight_of_deps) / (test_coverage × canary_hours)

Where:
  new_deps                = count of new upstream dependencies introduced by this deployment
  avg_tier_weight_of_deps = mean(tw_i for all new deps), tw in {1.0, 0.8, 0.5, 0.3}
  test_coverage           = fraction of service code covered by automated tests (0.0–1.0)
  canary_hours            = hours the service has been in canary before the promotion decision
```

---

#### Simple Example

A new Fraud Detection service is being added to the Checkout call chain.

**Dependencies introduced:**
- Payment Vault API (P0, tw=1.0)
- ML Model Service (P1, tw=0.8)
- IP Reputation API (P2, tw=0.5)

**Step 1 — Count dependencies:**
```
new_deps = 3
```

**Step 2 — Compute average tier weight:**
```
avg_tw = (1.0 + 0.8 + 0.5) / 3
       = 2.3 / 3
       = 0.767
```

**Step 3 — Gather deployment plan parameters:**
```
test_coverage = 0.65  (65% line coverage)
canary_hours  = 4
```

**Step 4 — Compute GEI:**
```
GEI = (3 × 0.767) / (0.65 × 4)
    = 2.301 / 2.60
    = 0.885
```

**Result:** GEI = 0.885 — below 1.0. Proceed normally.

---

#### Complex Scenario: What Happens When the Canary Window Is Compressed

Same service, same dependencies, same test coverage. The release manager wants to ship Friday before a holiday weekend and proposes a one-hour canary instead of four.

**Step 4 with canary_hours = 1:**
```
GEI = (3 × 0.767) / (0.65 × 1)
    = 2.301 / 0.65
    = 3.54
```

**Result:** GEI = 3.54 — above 3.0 — BLOCK.

The release manager has turned a safe deployment into a blocked one by compressing the canary from four hours to one. The formula makes this negotiation concrete. Instead of "I feel like one hour is not enough," the SRE can say: "Cutting canary from four hours to one raises GEI from 0.885 to 3.54. Our policy blocks anything above 3.0. To deploy Friday you need either a four-hour canary window or to drop one dependency."

**Alternative path — improve test coverage to 0.85, canary = 1 hour:**
```
GEI = (3 × 0.767) / (0.85 × 1)
    = 2.301 / 0.85
    = 2.71
```

GEI = 2.71 — in the extended canary zone (1.0–3.0). Not blocked, but requires two weeks of canary monitoring rather than standard promotion. Still better than a holiday weekend block.

**Lesson:** Canary duration is the fastest GEI lever in your control. Test coverage requires engineering work measured in weeks. Adding or removing a dependency changes the numerator. Canary hours change the denominator immediately and proportionally.

---

#### Decision Gates

| GEI Value | State | Action |
|-----------|-------|--------|
| < 1.0 | Low exposure | Proceed with standard canary promotion |
| 1.0 – 3.0 | Elevated exposure | Extend canary to minimum 2 weeks, increase monitoring cadence |
| > 3.0 | High exposure | Block deployment — reduce deps, increase coverage, or extend canary first |

---

#### Best Practices

**Do:**
- Calculate GEI before committing to a canary start time — learn the number before scheduling
- Use GEI as the opening data point in release review meetings
- Track GEI trend across deployments — a team whose GEI is climbing is accumulating operational debt
- Set canary duration based on GEI, not on sprint timelines

**Don't:**
- Don't let release pressure compress canary hours without recalculating GEI
- Don't apply GEI only to new services — apply it to any deployment that introduces new upstream dependencies
- Don't treat test_coverage as a fixed number — it is the one input you can improve in advance of a release
- Don't use GEI as the only admission gate — it is a risk signal, not a substitute for design review

---

#### Guardrails

- GEI above 3.0 is a hard block unless explicitly overridden by VP-level sign-off with documented rationale
- Any override must include a mitigation plan: which risk is being accepted and why
- Services with GEI > 2.0 require a dedicated rollout runbook before canary start
- GEI must be recalculated any time the deployment plan changes (canary hours adjusted, dependency list changed, coverage updated)

### Term 2 — RST: Rollout Signal Threshold

**Formula:**

```
RST = minimum rollout% where:
      (feature_error - baseline_error) / σ_baseline ≥ 2.0
```

**Threshold:** z < 1.0 noise | z 1.0–2.0 watch only | z ≥ 2.0 valid signal — act

---

#### The Concept

Imagine you are running an A/B test on a new checkout flow. After 30 minutes you have data from five users: two experienced an error on the new flow versus zero on the old flow. Is that a real signal? Probably not — with five users, random variation can produce any result. With 50,000 users, two percent versus zero percent is very likely a real difference.

RST applies the same reasoning to service rollouts. As you incrementally increase the percentage of traffic routed to the new version, you are accumulating sample size. Early in the rollout, the error rate you observe might look alarming — but it is almost certainly noise. The formula gives you the minimum rollout percentage at which the observed difference between the new version and the baseline becomes statistically meaningful (z-score ≥ 2.0, approximately 95% confidence).

Before RST: sit on your hands. An error spike in the first one percent of rollout is not a signal — it is statistical noise from a small sample. Rolling back based on noise teaches your team to be reactive to randomness, which is one of the most expensive habits an SRE organization can develop. It trains engineers to distrust gradual rollout mechanisms and revert to big-bang deployments, which is exactly the wrong direction.

After RST: act decisively. A z-score above 2.0 means the observed degradation is unlikely to be random. Investigate. If the investigation confirms a defect, roll back immediately. If it reveals an acceptable trade-off, make a deliberate decision to continue.

The operational discipline RST enforces is simple: you must know the difference between noise and signal before you respond to data. RST gives you the line.

---

#### Formula Components

```
z = (feature_error - baseline_error) / σ_baseline

RST = minimum rollout% at which z ≥ 2.0 is sustained

Where:
  feature_error   = error rate observed in the new version cohort (%)
  baseline_error  = error rate observed in the stable version cohort (%)
  σ_baseline      = standard deviation of baseline error rate over recent stable history
  z               = z-score: standard deviations the feature error rate is above baseline
```

The RST is the rollout percentage at which your sample size is large enough that z ≥ 2.0 becomes a reliable indicator rather than noise. In practice, compute z at each rollout milestone and identify when it crosses 2.0.

---

#### Simple Example

A new Recommendation algorithm is being rolled out. Historical baseline for the Recommendation service:

```
baseline_error = 0.10%
σ_baseline     = 0.02%
```

**At 5% rollout, feature_error = 0.12%:**
```
z = (0.12 - 0.10) / 0.02
  = 0.02 / 0.02
  = 1.0
```
z = 1.0 — not valid. One standard deviation above baseline is well within normal variation. Do not act.

**At 15% rollout, feature_error = 0.16%:**
```
z = (0.16 - 0.10) / 0.02
  = 0.06 / 0.02
  = 3.0
```
z = 3.0 — valid signal. Three standard deviations above baseline at 15% rollout. This is not noise. Investigate immediately.

**RST conclusion:** For this service and error profile, the RST lies between 5% and 15% rollout. Below RST: do not react to the error rate. Above RST: the error rate is a real signal requiring action.

---

#### Complex Scenario: What Error Rate Would Be Valid at 8% Rollout?

The release manager wants to know: if the Recommendation service shows degraded error rates at 8% rollout, at what observed error rate should the team declare a valid signal?

```
Need: z ≥ 2.0
(feature_error - 0.10) / 0.02 ≥ 2.0
feature_error - 0.10 ≥ 0.04
feature_error ≥ 0.14%
```

At 8% rollout: only act if the observed error rate reaches or exceeds 0.14%. An error rate of 0.12% at 8% rollout is noise by this math. An error rate of 0.17% at 8% rollout is a valid signal — roll back or investigate.

This gives the on-call engineer a concrete number to put in their runbook: "If Recommendation v2 shows error rate above 0.14% during rollout, declare the signal valid and escalate."

---

#### Decision Gates

| z-score | State | Action |
|---------|-------|--------|
| < 1.0 | Noise | Continue rollout — do not act on this data |
| 1.0 – 2.0 | Watch zone | Increase monitoring cadence, do not change rollout, do not roll back |
| ≥ 2.0 | Valid signal | Stop rollout increment, investigate, make deliberate decision to continue or roll back |

---

#### Best Practices

**Do:**
- Compute z-score at every rollout milestone (1%, 5%, 10%, 25%, 50%, 100%)
- Document the RST percentage in the rollout runbook before starting the rollout
- Give on-call engineers the concrete error rate threshold in the runbook, not just the formula
- Maintain historical σ_baseline — accuracy of RST depends entirely on accurate baseline statistics

**Don't:**
- Don't roll back on z < 1.0 — you are responding to randomness, not reality
- Don't wait for z ≥ 3.0 before acting — 2.0 is sufficient confidence; delay increases user impact
- Don't use RST on error rates that include unrelated incidents during the rollout window
- Don't apply RST to latency data without recalculating σ from the appropriate latency distribution

---

#### Guardrails

- RST is not a substitute for SLO breach alerts — if your SLO burns faster than budget allows, roll back regardless of z-score
- σ_baseline must be computed from at least 14 days of stable baseline data
- RST calculations must be re-run if baseline conditions change during rollout (infrastructure changes, traffic profile shifts)
- Never use RST to justify continuing a rollout when absolute error rates exceed SLO thresholds, even if z < 2.0

### Term 3 — SLTD: Scaling Lead Time Delta

**Formula:**

```
SLTD = (ramp_rate_rps/min × autoscaling_lag_min) / pod_capacity_rps
```

**Units:** Pods — the number of pods that must be pre-provisioned before the traffic event begins.

---

#### The Concept

Before a major parade, city emergency services do not wait for the first ambulance call to dispatch units. They pre-position ambulances at intervals along the route, knowing the crowd will generate incidents that require faster response than dispatch-from-station allows. Autoscaling is dispatch-from-station. By the time the HPA observes elevated CPU, decides to scale, schedules a pod, pulls an image, starts the container, and passes readiness checks, three minutes have elapsed. If your traffic ramps faster than that lag allows, you have a coverage gap.

SLTD quantifies that gap in pods. It tells you exactly how many pods' worth of capacity is unserved during the autoscaling lag window — and therefore how many you must pre-provision before the traffic ramp begins.

The formula answers: "If traffic ramps at X rps per minute and autoscaling takes Y minutes to respond, how many pods' worth of capacity is unserved during that lag?" That is your pre-provisioning requirement.

This is the difference between capacity planning as a strategy (SLTD) and capacity planning as a reaction (autoscaling). Both are necessary. SLTD tells you where autoscaling alone is insufficient — which is always at the front edge of a steep ramp.

---

#### Formula Components

```
SLTD = (ramp_rate_rps/min × autoscaling_lag_min) / pod_capacity_rps

Where:
  ramp_rate_rps/min    = rate of incoming traffic increase during the ramp period (rps per minute)
  autoscaling_lag_min  = time from HPA trigger to pod ready state (minutes) — measure this empirically
  pod_capacity_rps     = maximum sustainable rps per pod at target latency (from load testing)
```

---

#### Simple Example

A regular Tuesday traffic ramp for the checkout service:

```
ramp_rate        = 50 rps/min
autoscaling_lag  = 3 min
pod_capacity     = 200 rps/pod
```

```
SLTD = (50 × 3) / 200
     = 150 / 200
     = 0.75 pods
```

SLTD = 0.75 — less than one pod. Autoscaling covers this ramp adequately. No pre-provisioning required. This is the expected outcome for normal traffic patterns. SLTD becomes critical for steep ramps.

---

#### Complex Scenario: Black Friday Checkout Ramp

Black Friday opening rush: traffic goes from near-zero to 15,000 rps over 30 minutes.

**Step 1 — Calculate ramp rate:**
```
ramp_rate = 15,000 rps / 30 min = 500 rps/min
```

**Step 2 — Gather infrastructure parameters:**
```
autoscaling_lag  = 3 min  (measured from load tests on this EKS cluster)
pod_capacity     = 200 rps/pod  (from sustained load testing at p99 < 150ms)
```

**Step 3 — Compute SLTD:**
```
SLTD = (500 × 3) / 200
     = 1,500 / 200
     = 7.5 pods
```

**Result:** SLTD = 7.5 — round up to 8 pods. Pre-provision 8 pods before the Black Friday ramp starts.

**What happens without pre-provisioning:**

The first 3 minutes of the ramp receive no new pod capacity. During those 3 minutes, traffic has ramped from 0 to 1,500 rps (3 min × 500 rps/min). If existing pods handle 200 rps each and you start with 4 pods (800 rps capacity), the gap at minute 3 is:

```
Traffic arriving at minute 3:  1,500 rps
Existing pod capacity:           800 rps
Unserved:                        700 rps
```

700 rps of checkout traffic failing for 3 minutes on Black Friday. With a conversion rate of 4.57% (OCR), that is approximately 32 transactions per second lost during the gap — not a noise event.

**With 8 pre-provisioned pods:**
```
Pre-provisioned capacity:  8 × 200 = 1,600 rps
Traffic at minute 3:               1,500 rps
Gap: 0 — fully covered
```

Autoscaling then handles everything above 1,600 rps as the ramp continues, but the critical opening window is covered.

---

#### Decision Gates

| SLTD Value | State | Action |
|------------|-------|--------|
| 0 (< 0.5) | Autoscaling sufficient | No pre-provisioning required |
| 1 – 5 pods | Moderate pre-scaling | Pre-provision before event, confirm warm-up time |
| > 5 pods | Mandatory pre-scaling | Pre-provision + dedicated monitoring during ramp + rollback plan documented |

---

#### Best Practices

**Do:**
- Measure autoscaling_lag empirically on your actual cluster — do not assume 3 minutes; measure it under realistic load
- Re-run SLTD calculations whenever pod_capacity changes (new resource limits, JVM tuning, dependency changes)
- Include SLTD calculation as a required field in traffic event runbooks
- Pre-provision pods at least 15 minutes before the event to allow full warm-up and readiness confirmation

**Don't:**
- Don't assume autoscaling handles all traffic events — that assumption breaks at steep ramp rates
- Don't calculate SLTD only for peak capacity — calculate it for the ramp rate specifically
- Don't pre-provision and then let pods sit idle without periodic health checks before the event
- Don't confuse SLTD (pods to pre-provision for the ramp window) with total capacity planning (pods needed at peak)

---

#### Guardrails

- SLTD > 10 pods requires platform team review — may indicate pod_capacity tuning opportunity
- Pre-provisioned pods must be removed after the event window closes to control cost
- SLTD assumes a linear ramp — for non-linear ramps (flash sale, viral traffic), use the maximum instantaneous ramp rate
- autoscaling_lag must be measured under realistic load conditions, not on an idle cluster

## Part B: Transition Terms

### Term 4 — PCC: Protocol Coexistence Cost

**Formula:**

```
PCC = dual_version_ops_hours/day × team_cost_rate × days
```

**Purpose:** Economic forcing function that tells you when the accumulated cost of running two protocol versions exceeds the cost of completing the migration.

---

#### The Concept

Imagine a logistics company that runs two fuel types in its fleet: diesel for the old trucks and electric for the new ones. Every day both fuel types exist, you pay for two separate supply chains: two sets of fuel contracts, two sets of mechanics, two training programs, two monitoring systems, two sets of spare parts. The rational move is to complete the transition to one fuel type as fast as operationally feasible — because every day of coexistence is a cost you pay for zero additional value.

Protocol coexistence in software systems works the same way. When Auth v1 (JWT) and Auth v2 (OAuth 2.0) both run in production, your team pays a daily operational tax: compatibility testing between versions, dual monitoring dashboards, coordination overhead when one version has an incident, documentation that must explain both paths, and on-call engineers who must understand two auth systems simultaneously.

PCC makes this cost visible and accumulating. It converts the soft pressure of "we should finish the migration soon" into a hard number: "As of day 22, we have spent $4,950 on coexistence overhead. Completing the migration would have cost $3,000. We are now underwater."

Once PCC exceeds the migration completion cost, every additional day of coexistence is economically irrational. The formula creates a forcing function for technical leadership to treat migration completion as a cost-reduction initiative rather than an operational nicety.

---

#### Formula Components

```
PCC = dual_version_ops_hours/day × team_cost_rate × days

Where:
  dual_version_ops_hours/day = hours per day spent on coexistence overhead
                               (monitoring, compatibility testing, coordination, documentation)
  team_cost_rate             = fully-loaded hourly cost per engineer hour ($)
  days                       = number of days the dual-version state has persisted or is projected to persist

Daily PCC:
  daily_PCC = dual_version_ops_hours/day × team_cost_rate

Days until budget threshold:
  days_remaining = budget_threshold / daily_PCC
```

---

#### Simple Example

Auth service migration: JWT (v1) → OAuth 2.0 (v2).

Daily overhead estimate:
- 1.0 hr: monitoring both versions, comparing dashboards
- 0.2 hr: weekly compatibility regression test (amortized to daily)
- 0.5 hr: token format translation layer maintenance
- 0.3 hr: documentation and on-call prep for new engineers

```
dual_version_ops_hours = 1.0 + 0.2 + 0.5 + 0.3 = 2.0 hours/day
team_cost_rate         = $75/hour (loaded engineer cost)
daily_PCC              = 2.0 × $75 = $150/day
```

If migration completion costs an estimated $3,000 (40 engineer hours):
```
days_to_breakeven = $3,000 / $150/day = 20 days
```

After day 20, every additional day of coexistence costs more than simply completing the migration.

---

#### Complex Scenario: Auth Migration with Measured Overhead

After two weeks of actual coexistence, the team measures real overhead — which is higher than the initial estimate:

```
dual_version_ops_hours = 3.0 hours/day
  Breakdown:
  1.0 hr: dual monitoring + alerting maintenance
  0.5 hr: token translation debugging (higher than anticipated)
  0.5 hr: on-call handoff documentation updates
  1.0 hr: cross-team coordination (downstream services calling both versions)
```

```
daily_PCC          = 3.0 × $75 = $225/day
budget_threshold   = $5,000 (set by engineering director as migration investment cap)
```

```
days_to_threshold = $5,000 / $225 = 22.2 days
```

At current overhead rate, day 22 is the forcing function deadline. The team now has a quantified argument for prioritizing migration completion over backlog items.

**Cumulative PCC at key milestones:**

| Day | Cumulative PCC | Status |
|-----|---------------|--------|
| 7   | $1,575 | Below threshold — monitoring |
| 14  | $3,150 | Migration completion cost exceeded |
| 22  | $4,950 | Approaching director's budget threshold |
| 25  | $5,625 | Over threshold — escalation required |

---

#### Decision Gates

| PCC vs. Threshold | State | Action |
|-------------------|-------|--------|
| PCC < 50% of threshold | Coexistence sustainable | Monitor, continue migration at planned pace |
| PCC approaching threshold | Cost pressure accumulating | Accelerate migration, reduce scope if needed |
| PCC > threshold | Economically irrational to continue | Force hard cutover date, accept technical risk if necessary |
| PCC growing faster than projected | Overhead underestimated | Re-measure, re-project, escalate to stakeholders |

---

#### Best Practices

**Do:**
- Measure dual-version overhead empirically after the first week — initial estimates are almost always low
- Set the PCC budget threshold before the migration starts, not during it
- Include PCC trend in weekly migration status reports — it creates urgency without drama
- Use PCC as input to sprint prioritization — coexistence cost competes directly with feature work

**Don't:**
- Don't let migrations run indefinitely without a PCC calculation — invisible cost is cost that never gets prioritized
- Don't set the budget threshold based on gut feel — anchor it to the actual migration completion estimate
- Don't calculate PCC only at project start — re-run monthly with measured actual overhead
- Don't use PCC to justify cutting corners on migration quality; use it to justify the resources needed to complete it properly

---

#### Guardrails

- PCC calculation must be reviewed by the team lead and engineering manager quarterly for any migration exceeding 60 days
- Migrations where cumulative PCC exceeds migration completion cost for more than 30 days require director-level review
- PCC does not account for risk cost of the coexistence period — security vulnerabilities, compliance exposure, and incident risk are additive to PCC
- Hard cutover decisions driven by PCC must still include a documented rollback plan

### Term 5 — SSE: Simultaneous Switchover Exposure

**Formula:**

```
SSE = (services_in_cutover / total_services) × avg(1 - SLO_margin_remaining_i)
```

**Threshold:** < 0.10 safe | 0.10–0.25 slow down | > 0.25 stop

---

#### The Concept

Imagine you need to rotate the tires on your car while it is moving at highway speed. You can do one tire at a time — risky but survivable. You cannot do all four simultaneously — there is no graceful recovery from that failure mode.

Platform migrations work the same way. When you cut services over to a new infrastructure (new Kubernetes cluster, new service mesh, new API gateway), each service is in its most vulnerable state during the cutover window. It has left the stable environment but has not yet proven stability in the new one. SLO margins are thinner. Rollback complexity is higher. Incident response is slower because engineers are operating unfamiliar systems.

SSE measures how much of your total platform is simultaneously in this vulnerable state, weighted by how close each service is to burning its error budget. A service with 80% of its SLO margin remaining can absorb an incident during cutover. A service with 5% margin remaining cannot — any failure during cutover causes a breach.

The fraction term (services_in_cutover / total_services) measures breadth of exposure. The SLO term avg(1 - SLO_margin_remaining) measures depth of risk per service. Together they give you a compound exposure score that captures both dimensions.

When SSE is high, you are combining breadth (many services at risk) with depth (each service near its SLO limit). That compound state is where migrations produce cascading failures — which is why SSE exists as a gate and not just a monitoring metric.

---

#### Formula Components

```
SSE = (services_in_cutover / total_services) × avg(1 - SLO_margin_remaining_i)

Where:
  services_in_cutover        = number of services currently in active cutover state
  total_services             = total services in scope for this migration
  SLO_margin_remaining_i     = fraction of SLO error budget remaining for service i
                               (1.0 = full budget untouched, 0.0 = budget fully exhausted)
  avg(1 - SLO_margin_remaining_i) = average SLO risk factor across all services currently in cutover
```

---

#### Simple Example

EKS cluster migration for the shopping cart platform. 12 services total in scope.

**Scenario — 4 services in active cutover simultaneously:**

Batch: Inventory, Pricing, Product Catalog, Search (all P1 services)

SLO margins at time of cutover:
```
Inventory:       30% remaining  → risk factor = 1 - 0.30 = 0.70
Pricing:         25% remaining  → risk factor = 1 - 0.25 = 0.75
Product Catalog: 40% remaining  → risk factor = 1 - 0.40 = 0.60
Search:          35% remaining  → risk factor = 1 - 0.35 = 0.65
```

```
avg(1 - margin) = (0.70 + 0.75 + 0.60 + 0.65) / 4 = 2.70 / 4 = 0.675
```

```
SSE = (4/12) × 0.675
    = 0.333 × 0.675
    = 0.225
```

SSE = 0.225 — in the slow-down zone (0.10–0.25, approaching the stop threshold of 0.25). Reduce batch size before proceeding.

---

#### Complex Scenario: Finding the Safe Batch Size

The team wants to know: how many services can be migrated simultaneously while staying in the safe zone (SSE < 0.10)?

Given average SLO risk factor of 0.675 per service (from the scenario above), solve for batch size n:

```
SSE < 0.10
(n/12) × 0.675 < 0.10
n/12 < 0.148
n < 1.78
```

Safe batch size: **1 service at a time** (serial migration).

**Verify at n=1:**
```
SSE = (1/12) × 0.675
    = 0.0833 × 0.675
    = 0.056 → SAFE
```

**Verify at n=2:**
```
SSE = (2/12) × 0.675
    = 0.1667 × 0.675
    = 0.113 → SLOW DOWN zone
```

**Conclusion:** With current SLO margins, this team must migrate services serially. Two services simultaneously pushes into the slow-down zone.

**If teams stabilize SLO margins to 60% remaining before cutover (average risk factor = 0.40):**
```
Need: (n/12) × 0.40 < 0.10
n < 3.0
```

With improved margins, a batch of 2 becomes safe. This is the operational incentive: stabilize services before cutover, not during. SSE rewards pre-migration reliability work with faster migration batches.

---

#### Decision Gates

| SSE Value | State | Action |
|-----------|-------|--------|
| < 0.10 | Safe | Proceed with current batch |
| 0.10 – 0.25 | Slow down | Reduce batch size by half, increase monitoring |
| > 0.25 | Stop | Halt all cutover activity, stabilize currently migrated services, reassess SLO margins |

---

#### Best Practices

**Do:**
- Calculate SSE before starting each migration batch, not just at the beginning of the project
- Measure SLO margin remaining at time of cutover, not at project start — margins drift over weeks
- Use SSE to negotiate the migration schedule — "we can safely batch 2 services this sprint" is a quantified position
- Prioritize stabilizing SLO margins before cutover — it increases safe batch size and reduces total migration time

**Don't:**
- Don't migrate Auth and Payment in the same batch regardless of SSE score — they require separate batches by policy
- Don't use total_services from the original project scope if some services have been descoped mid-migration
- Don't assume SSE is a one-time calculation — recompute before each new batch
- Don't ignore the avg(1 - margin) component — two services in cutover with 5% margin remaining is more dangerous than four services with 80% margin

---

#### Guardrails

- SSE > 0.25 is a hard stop — no exceptions without incident commander approval
- Services with SLO_margin_remaining < 0.10 (less than 10% budget left) must not be included in any cutover batch regardless of SSE
- Auth and Payment services must be migrated in separate batches regardless of SSE score
- SSE must be recalculated after any incident during migration — an incident drains SLO margin, which raises the SSE for the next batch

---

### Term 6 — PPC: Parallel Patch Count

**Formula:**

```
PPC = floor((remaining_window - validation_window) / avg_rollback_time_per_svc)
```

**Purpose:** Maximum number of services that can be patched in parallel while preserving the ability to fully roll back all of them within the maintenance window.

---

#### The Concept

Security patching under a maintenance window is a planning problem with one hard constraint: if the patch goes wrong, you must be able to undo everything before the window closes. The window is finite. Rollback takes time. If you patch too many services in parallel and two of them need rollback, you may run out of window before you finish rolling back.

PPC answers: given my window and my rollback time per service, how many services can I patch simultaneously while guaranteeing I can roll back all of them if needed?

Think of it like a fire evacuation drill. You have a building with 20 floors. The drill window is 30 minutes. Evacuating one floor takes 5 minutes. You can run 6 floors simultaneously (30/5) — but not all 20. If you try all 20 simultaneously and something goes wrong on floor 15, you cannot reset and try again within the 30-minute window.

The validation_window is the time reserved at the end to confirm everything is healthy before declaring success — you must protect that time even under window pressure.

The formula is conservative by design: it assumes all patched services need rollback simultaneously, which is the worst case. In practice, partial rollback is far more common, but the guarantee must hold for the worst case.

---

#### Formula Components

```
PPC = floor((remaining_window - validation_window) / avg_rollback_time_per_svc)

Where:
  remaining_window          = total maintenance window duration (minutes)
  validation_window         = time reserved for post-patch health verification before window close (minutes)
  avg_rollback_time_per_svc = average time to complete rollback for one service (minutes)
                              — includes rollback execution + health check confirmation
```

---

#### Simple Example

Zero-day vulnerability affecting all checkout services. Emergency patch tonight.

**Window parameters:**
```
remaining_window          = 240 min (4-hour maintenance window)
validation_window         = 10 min  (post-patch health check)
avg_rollback_time_per_svc = 8 min
```

**Step 1 — Compute PPC:**
```
PPC = floor((240 - 10) / 8)
    = floor(230 / 8)
    = floor(28.75)
    = 28
```

**Step 2 — Compare to services in scope:**

9 services require patching: Auth, Cart, Payment, Order Management, Inventory, Pricing, Product Catalog, Search, User Profile.

PPC = 28 ≥ 9 services — patch all simultaneously (with dependency grouping). The window is comfortable enough to roll back all 9 services if needed and still complete validation in 10 minutes.

---

#### Complex Scenario: Tight Window with Dependency Constraints

Same vulnerability, but the maintenance window is only 90 minutes (emergency weekend patch, constrained by change freeze policy).

```
remaining_window          = 90 min
validation_window         = 10 min
avg_rollback_time_per_svc = 25 min (stateful services, slower rollback)
```

**Step 1 — Compute PPC:**
```
PPC = floor((90 - 10) / 25)
    = floor(80 / 25)
    = floor(3.2)
    = 3
```

PPC = 3 — batch in groups of maximum 3 services.

**Step 2 — Apply dependency grouping rule:**

Never patch two services with direct dependencies in the same batch. If both Auth and Cart are patched simultaneously and both need rollback, the rollback order matters (Cart depends on Auth) and creates time pressure within the batch.

**Dependency-safe batching for 9 services:**

| Batch | Services | Reasoning |
|-------|---------|-----------|
| Batch 1 | Auth, Inventory, User Profile | No direct dependencies between these three |
| Batch 2 | Cart, Pricing, Product Catalog | Start only after Auth (Batch 1) is confirmed healthy |
| Batch 3 | Payment, Search, Order Management | Start only after Cart (Batch 2) is confirmed healthy |

**Time accounting:**
```
Batch 1:  25 min patch + 10 min health confirm = 35 min
Batch 2:  25 min patch + 10 min health confirm = 35 min
Batch 3:  25 min patch + 10 min final validation = 35 min
Total:    35 + 35 + 35 = 105 min
```

105 min > 90 min available window. The three-batch approach does not fit.

**Resolution options:**
1. Request window extension to 120 minutes — preferred
2. Accept that Batch 3 (Payment, Search, Order Management) patches the following night with documented risk acceptance and compensating controls (WAF rules, traffic monitoring, rate limiting on affected endpoints)
3. Use faster rollback tooling — if automated rollback can be reduced to 15 min, PPC = floor(80/15) = 5, enabling a two-batch approach

---

#### Decision Gates

| PPC vs. Total Services | State | Action |
|------------------------|-------|--------|
| PPC ≥ total services | Window is comfortable | Patch all in parallel (with dependency grouping) |
| PPC = 2 to N-1 | Batch patching required | Group by dependency tier, batch sequentially with health confirms |
| PPC = 1 | Serial patching only | Likely need window extension; patch P0 services first |
| PPC = 0 | Window too small for even one rollback | Do not start patching — negotiate window before proceeding |

---

#### Best Practices

**Do:**
- Calculate PPC before every maintenance window, not just for emergencies
- Measure avg_rollback_time empirically — the first time you do a real rollback, your estimate will be wrong
- Group services by dependency tier when creating batches — never patch two directly-dependent services simultaneously
- Build the dependency grouping rule into your patch runbook as a required step, not an optional recommendation

**Don't:**
- Don't patch Auth and Payment in the same batch — they are the most likely to need rollback and have the most downstream dependencies
- Don't use PPC = 0 as a reason to skip patching — use it as a reason to negotiate a larger window
- Don't assume rollback time is the same for stateless and stateful services — measure them separately
- Don't skip the validation_window under time pressure — it is the only way to detect a broken patch before the window closes

---

#### Guardrails

- PPC = 0 is a hard stop — do not begin patching without window renegotiation or explicit risk acceptance at VP level
- Patch batches must be ordered so that P0 services (Auth, Payment) are patched first and fully confirmed before P1/P2 services begin
- Any service in a batch that fails its health check triggers full batch rollback — do not attempt to isolate and continue with remaining services
- Post-patch validation window must be at minimum 10 minutes regardless of window pressure

## Part C: Validation and Decommission Terms

### Term 7 — RDR: Repair Displacement Ratio

**Formula:**

```
RDR = new_incident_mode_rate / old_incident_mode_rate
```

**Target at 90-day review:** < 0.10 clean fix | 0.10–0.50 partial displacement | > 0.50 problem moved | > 1.0 fix made things worse

---

#### The Concept

When a city repairs a pothole by filling it with asphalt, you check it six months later. If the pothole is gone and the road is intact, the repair worked. If the original pothole is gone but water damage has appeared in the adjacent section, the repair displaced the failure mode — the structural water issue was never addressed, it just manifested differently. If the repair made the road surface worse (subsidence, cracking), the repair created a new failure.

RDR is the 90-day pothole check for incident fixes. It compares the rate of the old failure mode after the fix to the rate of any new failure mode that emerged around the same time.

A clean fix eliminates the old mode (old rate drops to zero) and introduces no new mode (new rate = 0). RDR = 0.

The dangerous outcome is RDR between 0.10 and 0.50: partial displacement. The original problem is less frequent, but a new manifestation has appeared. The underlying root cause was not fully addressed — you patched a symptom and the system expressed the problem through a different path. Teams often close these incidents as "fixed" without realizing they have shifted the failure mode.

RDR > 1.0 is the worst outcome: the fix introduced a new failure mode that occurs more frequently than the original problem. This happens when the fix removes a constraint that was throttling a deeper issue, or when the fix introduces a new dependency that is less reliable than what it replaced.

The 90-day review window is deliberate. Many fixes appear successful at 7 and 30 days. Displaced failure modes often take 60–90 days to manifest at sufficient frequency to measure reliably.

---

#### Formula Components

```
RDR = new_incident_mode_rate / old_incident_mode_rate

Where:
  old_incident_mode_rate = rate of the original incident pattern (incidents per month)
                           measured over 3+ months pre-fix
  new_incident_mode_rate = rate of any new incident pattern that emerged after the fix
                           measured over 90 days post-fix
```

RDR requires precise incident mode definitions before the fix is implemented. If you cannot define what the original failure looks like in observable terms, you cannot measure whether it was displaced.

---

#### Simple Example

Auth service connection pool exhaustion fix.

**Pre-fix state (3-month baseline):**
```
Auth "connection pool exhaustion" incidents: 3/month
```

The fix: added PgBouncer as a connection pooling proxy in front of PostgreSQL. Auth no longer connects directly to Postgres; PgBouncer manages the pool.

**Post-fix measurement at 90 days:**
```
Auth "connection pool exhaustion" incidents: 0/month  (old mode eliminated)
New: "PgBouncer queue saturation" incidents: 0.5/month (new mode observed during write spikes)
```

**RDR calculation:**
```
RDR = 0.5 / 3.0 = 0.167
```

RDR = 0.167 — partial displacement zone (0.10–0.50). The fix eliminated Auth connection pool exhaustion completely, but introduced a less frequent new failure mode: PgBouncer queue saturation under sustained write load.

**Interpretation:** This is not a failed fix. The original problem is gone. But the underlying write load pattern that exhausted connection pools is now expressing itself as PgBouncer queue backup when write throughput spikes. The real root cause — write amplification during checkout peak — was not addressed.

**Action:** Do not reopen the original incident. Open a new, separate incident for PgBouncer queue saturation with a targeted fix: add auto-drain logic and alerting on queue depth threshold. The fix is 83% effective; the residual 17% requires a follow-up that addresses the write amplification pattern directly.

---

#### Complex Scenario: Comparing Fix Outcomes Across Four Teams

| Scenario | old_rate | new_rate | RDR | Verdict |
|----------|----------|----------|-----|---------|
| Checkout timeout fix | 4/month | 0/month | 0.0 | Clean fix — close the incident |
| Auth pool fix (PgBouncer) | 3/month | 0.5/month | 0.167 | Partial displacement — address residual |
| Payment retry storm fix | 2/month | 1.5/month | 0.75 | Significant displacement — reopen |
| Cart cache fix | 1/month | 2/month | 2.0 | Fix made it worse — rollback the fix |

For the payment retry storm case (RDR = 0.75): the team fixed the original "payment timeout during peak" by adding aggressive retries. The retries eliminated the timeouts but created a new pattern: retry storms that cause downstream service saturation. The problem shifted from "customers see timeout errors" to "entire checkout degrades during retry floods." RDR = 0.75 signals this clearly — 75% of the problem's impact moved to a new mode rather than being eliminated.

For the cart cache fix (RDR = 2.0): the fix introduced a new failure mode that occurs twice as frequently as the original problem. This is a clear signal to roll back the fix entirely and restart the root cause investigation.

---

#### Decision Gates

| RDR Value | State | Action |
|-----------|-------|--------|
| 0.0 | Complete elimination | Close incident as fully resolved |
| < 0.10 | Clean fix with acceptable residual | Close, add monitoring for residual mode |
| 0.10 – 0.50 | Partial displacement | Close original, open follow-up ticket for new mode |
| > 0.50 | Significant displacement | Reopen original incident, investigate new mode jointly |
| > 1.0 | Fix made things worse | Rollback the fix, restart root cause investigation |

---

#### Best Practices

**Do:**
- Define the incident mode precisely before implementing the fix — "connection pool exhaustion on Auth Postgres during peak checkout" not just "Auth incidents"
- Set a 90-day calendar reminder at fix deployment time for the mandatory RDR review
- Track new incident modes for 90 days after every significant infrastructure change
- Use RDR trends across the team to identify systemic patterns (e.g., fixes that consistently displace to the database layer indicate unresolved write amplification)

**Don't:**
- Don't close a fix as "resolved" at 7 or 30 days — too early to measure displacement reliably
- Don't conflate different incident modes when calculating RDR — define modes precisely before the fix, not retroactively
- Don't ignore RDR > 0.50 cases in retrospectives — these are your most valuable learning opportunities
- Don't use RDR as a blame metric — use it as a system health indicator that informs future root cause work

---

#### Guardrails

- Every P0 and P1 incident fix requires a 90-day RDR review as a mandatory postmortem step
- RDR > 0.50 at 90 days requires root cause re-analysis before the incident can be closed
- RDR > 1.0 triggers immediate review of the fix with rollback consideration
- RDR reviews must use incident data from at least 60 days post-fix — earlier data may not show displacement patterns at sufficient frequency to measure reliably

---

### Term 8 — SCI: Shadow Coupling Index

**Formula:**

```
SCI = (observed_rps - declared_rps) / total_observed_rps
```

**Requirement:** SCI = 0 AND stable for ≥ 30 days before decommission.

---

#### The Concept

Every long-lived service accumulates undocumented callers. It is as predictable as entropy. The mechanisms are consistent across every organization: a developer copies a curl command from a Confluence page written three years ago and wires it into a script that runs in production. A team discovers an undocumented internal API that returns exactly the data they need and uses it without registering the dependency. A one-off dashboard query becomes a scheduled job that runs every minute. A feature experiment that was supposed to be temporary is still calling the service 18 months later.

When you decide to decommission a service, you consult the architecture documentation and find three registered callers. You believe those are all the callers. You are wrong. The SCI tells you by how much.

SCI compares the traffic volume declared in your service dependency registry against the traffic actually observed hitting the service. Any gap is shadow coupling — callers that exist in production but not in documentation.

The operational severity of shadow coupling during decommission is asymmetric: a 2% undocumented dependency will fail hard when you pull the service, even though it appeared insignificant in the traffic summary. That 2% might be a critical reporting job that runs monthly, an on-call script that only fires during incidents, or a data pipeline that is quiet until end-of-quarter batch. You will not discover it by watching dashboards before decommission — you will discover it at 2 AM when it fails after the service is gone.

SCI = 0 means observed traffic equals declared traffic. Before you reach SCI = 0, you have not found all the callers.

---

#### Formula Components

```
SCI = (observed_rps - declared_rps) / total_observed_rps

Where:
  observed_rps   = total request rate observed hitting the service (from APM or service mesh metrics)
  declared_rps   = total request rate from registered, documented callers
                   (sum of expected rps from each entry in the dependency registry)
  total_observed_rps = same as observed_rps (normalizes the undocumented gap as a fraction)
```

SCI = 0 means all observed traffic is accounted for in documentation.
SCI > 0 means undocumented callers exist. Do not decommission.

---

#### Simple Example

Recommendation service v1 decommission check.

**Documented callers (from service registry):**
```
Product Catalog API:  120 rps (pulls recommended products for catalog pages)
Browse API:            80 rps (displays "you might also like" on browse pages)
declared_rps          = 120 + 80 = 200 rps
```

**Observed traffic (service mesh metrics, 7-day average):**
```
observed_rps = 215 rps
```

**SCI calculation:**
```
SCI = (215 - 200) / 215
    = 15 / 215
    = 0.07
```

SCI = 0.07 — BLOCKED. 15 rps of undocumented traffic exists. Decommission cannot proceed.

**Investigation:** Query access logs grouped by caller service identity. Finding: Search service (P1) is sending 15 rps to Recommendation v1 with no entry in the dependency registry. Tracing the call origin: a feature flag experiment from 14 months ago ("boost search results with recommendation scores") was never fully rolled back. The flag remains enabled for 8% of users, and the Search service continues to call Recommendation v1 for that cohort.

**Resolution path:**
1. Register Search as a caller of Recommendation v1 in the dependency registry
2. Migrate Search to Recommendation v2 (validate recommendation score parity in staging)
3. Disable the legacy feature flag or implement the equivalent behavior in v2
4. Re-verify SCI: after Search migration, observed_rps = 200, declared_rps = 200, SCI = 0
5. Observe for 30 days — confirm SCI remains stable at 0 (no new shadow callers appear)
6. Proceed with decommission

---

#### Complex Scenario: Multiple Shadow Callers at Different Frequencies

User Activity Logging API decommission:

**Documented callers:**
```
Order Management: 300 rps (order state change events)
declared_rps      = 300 rps
```

**Observed traffic (60-day window, 1-minute resolution):**
```
observed average: 347 rps
```

**SCI:**
```
SCI = (347 - 300) / 347 = 47/347 = 0.135
```

BLOCKED. 47 rps undocumented. Caller identity analysis:

```
Order Management:    300 rps  (documented — matches registry)
Analytics pipeline:   30 rps  (batch job polling every 2 minutes for event data)
Fraud Detection:      12 rps  (added 8 months ago for transaction review events)
Legacy ops script:     5 rps  (on-call tool that polls during active incident investigations)
```

Three shadow callers discovered. Each requires a separate migration path:
- Analytics pipeline: migrate to Kafka event stream consumer (proper event-driven architecture, not polling)
- Fraud Detection: register, migrate to Events v2 API, validate parity under load
- Ops script: update script to use new API, distribute updated version to all on-call engineers

**The ops script (5 rps average) is the highest-priority shadow caller despite being lowest volume.** It runs during incidents — exactly when a decommissioned dependency causes maximum compounding pain. An on-call engineer discovering a broken tool during a P0 incident is a cascading failure that RDR will not help you fix afterward.

After all three migrations: observe for 30 days with SCI = 0 before proceeding.

---

#### Decision Gates

| SCI Value | State | Action |
|-----------|-------|--------|
| > 0 | Shadow callers exist | BLOCKED — find and migrate all undocumented callers before proceeding |
| = 0 for < 30 days | Callers migrated, not yet stable | Observe — do not decommission yet |
| = 0 for ≥ 30 days | Stable and fully documented | Safe to proceed with decommission planning |

---

#### Best Practices

**Do:**
- Start SCI checks 60 days before planned decommission — not 7 days
- Use traffic percentile data and time-series, not simple averages — low-volume callers run at off-peak hours and can be invisible in hourly summaries
- Investigate caller identity through service mesh headers (caller service name), not just source IP addresses
- Look for callers with periodic patterns (hourly, daily, weekly, monthly) — they may appear rarely in short observation windows

**Don't:**
- Don't declare SCI = 0 based on a 24-hour observation window — some callers are weekly or monthly batch jobs
- Don't migrate callers without testing the migrated path under production-representative load
- Don't decommission on a Friday or before a holiday regardless of SCI status
- Don't assume that teams who registered as callers during system design also registered every call site they later added

---

#### Guardrails

- SCI = 0 observation period is minimum 30 days — non-negotiable regardless of deadline pressure
- Observation window must include at least one month-end period (end-of-month batch jobs often only appear once per month)
- Any caller discovered through SCI analysis must be formally registered in the dependency registry before migration begins, not after
- SCI analysis must use service mesh metrics or APM distributed trace data — load balancer logs alone are insufficient (they may aggregate callers behind shared infrastructure)

---

### Term 9 — EDFD: Externally Driven Forcing Date

**Formula:**

```
EDFD = today + (EBR_reserved / daily_EBV_burn_from_deprecation)
```

**Purpose:** The hard migration completion deadline imposed by third-party API deprecation, calculated from your error budget math — not from the vendor's calendar.

---

#### The Concept

Your vendors do not operate on your sprint schedule. When a payment gateway announces that its v1 API will begin returning malformed responses on a sunset date, or when an authentication provider's old OAuth endpoint starts rate-limiting deprecated clients, or when a data API begins introducing breaking schema changes six months before formal deprecation — your error budget starts burning immediately, not on the sunset date.

EDFD is the date by which your migration must be complete, calculated from how fast the deprecated API is consuming your error budget today. It is not the vendor's announced deprecation date — it is your budget math deadline, which may be significantly earlier.

The mechanism: every day the deprecated API operates in degraded mode (returning errors, introducing latency, dropping requests), it burns a fraction of your SLO error budget. You have a finite budget per month. You reserve a percentage of that budget specifically to absorb migration-period failures — so that the deprecation does not burn your entire SLO budget and eliminate your margin for other risks. That reserved amount divided by the daily burn rate gives you the number of days before you will have consumed all the budget you were willing to spend on this problem.

After EDFD, continuing to use the deprecated API means either breaching your SLO, reducing your error budget reserve for other operational risks, or accepting a formal budget suspension from your SLA contract. None of those outcomes is acceptable without explicit stakeholder sign-off.

EDFD creates a concrete engineering and business conversation: "We have 16 days to complete this migration. The migration takes 21 days at current resourcing. We need either more engineers or a formal SLA pause."

---

#### Formula Components

```
EDFD = today + (EBR_reserved / daily_EBV_burn_from_deprecation)

Where:
  EBR_reserved                      = error budget reserved for the migration period
                                      = reservation_fraction × monthly_error_budget_minutes
  daily_EBV_burn_from_deprecation   = minutes of error budget consumed per day by the deprecated API
                                      (measured from current failure rate data — not estimated)
```

---

#### Simple Example

Payment gateway v1 sunset. The gateway has already begun returning malformed responses for certain transaction types.

**Checkout journey SLA parameters:**
```
Monthly error budget = 21.6 min  (SLA = 99.95%)
```

**Step 1 — Calculate EBR_reserved:**

The team reserves 15% of checkout budget for migration period failures:
```
EBR_reserved = 0.15 × 21.6 = 3.24 min
```

**Step 2 — Measure daily EBV burn:**

From Splunk: Payment v1 API malformed response errors are causing 0.20 min/day of checkout budget consumption.
```
daily_EBV_burn = 0.20 min/day
```

**Step 3 — Calculate EDFD:**
```
EDFD = today + (3.24 / 0.20)
     = today + 16.2 days
```

**Interpretation:** The reserved budget runs out in 16 days. If the migration takes 3 weeks (21 days), the team is already behind before the migration starts.

**Decision:** Start the migration immediately. There is no planning buffer. At day 16, you will begin burning the unreserved portion of your error budget — every additional day of delay creates SLO exposure that competes with your other reliability commitments.

---

#### Complex Scenario: Burn Rate Acceleration

Same scenario, one week later. The vendor has accelerated its deprecation timeline. Malformed response rate has doubled.

**New daily EBV burn:**
```
daily_EBV_burn = 0.40 min/day  (doubled from original measurement)
```

**Remaining EBR_reserved at day 7:**
```
Burned in days 1–7 (at original rate of 0.20 min/day):
  7 × 0.20 = 1.40 min consumed

Remaining:
  3.24 - 1.40 = 1.84 min
```

**New EDFD from today (day 7):**
```
EDFD = today + (1.84 / 0.40)
     = today + 4.6 days
```

**The migration deadline has collapsed from 16 days to 4.6 days** — and the migration was already 7 days in with 21 days of total scope.

**Options that must be on the table in the same meeting:**
1. Emergency sprint: triple resourcing, reduce migration scope to critical transaction paths only, defer non-critical endpoint migration to follow-up sprint
2. Negotiate formal SLA pause with stakeholders — accept elevated checkout error rate for 2 weeks with documented customer impact acknowledgment
3. Contact vendor for a compatibility shim or extended support window at increased cost

Each option has a different cost. EDFD makes all three concrete: "In 4.6 days we exhaust our reserved budget. Here are three paths with their trade-offs. Which do we choose?"

---

#### Decision Gates

| EDFD vs. Migration Duration | State | Action |
|-----------------------------|-------|--------|
| EDFD > migration duration + 1 week | Comfortable | Begin planning in normal sprint cycles |
| EDFD ≈ migration duration (within 1 week buffer) | At risk | Start migration immediately, weekly EDFD recalculation |
| EDFD < migration duration | Past deadline math | Emergency sprint + stakeholder escalation + SLA negotiation |
| EDFD < 1 week | Critical | Drop all other work, activate incident-level response |

---

#### Best Practices

**Do:**
- Measure daily_EBV_burn from actual error data the day deprecation begins producing failures — do not wait for the formal vendor sunset date
- Recalculate EDFD weekly — burn rates change as vendors accelerate their deprecation timelines
- Set EBR_reserved before migration starts, with explicit stakeholder agreement on what fraction is acceptable to spend on third-party migration risk
- Track EDFD in migration status reports as the primary deadline metric — it is more honest than sprint velocity estimates

**Don't:**
- Don't use the vendor's official sunset date as your deadline — your deadline is budget math, which may be significantly earlier
- Don't set EBR_reserved to 100% of the monthly budget — you need error budget margin for other operational risks during the migration period
- Don't ignore burn rate acceleration signals — vendors often gradually increase deprecation pressure before hard cutover
- Don't conflate EDFD (when you run out of budget tolerance) with migration planning date (when migration is optimal to complete)

---

#### Guardrails

- EDFD must be recalculated after any change in vendor behavior (rate limit changes, new error codes, latency increases, announced acceleration of deprecation timeline)
- If EDFD is within 14 days and migration is less than 50% complete, trigger director-level escalation automatically
- EBR_reserved must not exceed 20% of monthly budget — reserving more creates false comfort and removes operational slack for other risks
- Teams must have a documented SLA pause process in place before EDFD arrives — negotiating it during the crisis is too slow and produces worse outcomes

## How These Terms Work Together

These nine metrics are not independent calculations. They form three operational clusters, each covering a distinct phase of service lifetime. Understanding the flow between clusters is how you apply them in practice.

---

### Cluster 1: Growth — Admitting and Scaling a New Service

```
New service proposed
        |
        v
[GEI] ← Admission gate: is this service ready to carry its dependency risk?
        |
        | GEI < 1.0: proceed | GEI 1-3: extended canary | GEI > 3: block
        v
[Canary rollout begins — incremental traffic increase]
        |
        v
[RST] ← Statistical signal gate: when can we trust the error data?
        |
        | z < 2.0: noise, continue | z >= 2.0: valid signal, act
        v
[Service at 100% traffic — first major traffic event approaching]
        |
        v
[SLTD] ← Traffic event gate: how many pods must exist before ramp starts?
        |
        | Pre-provision SLTD pods before event begins
        v
[Service handles traffic event successfully]
```

**The interaction:** GEI governs admission. RST governs the rollout decision window — it tells you when to act on error data and when to ignore it. SLTD governs the first major traffic event after the service reaches full production. A service can pass GEI and still require RST monitoring during gradual rollout. A service that reaches 100% traffic still needs SLTD analysis before any non-linear traffic event.

---

### Cluster 2: Transition — Migrating Infrastructure or Protocol

```
Migration initiated
        |
        v
[SLTD] ← Pre-event scaling: enough capacity before migration traffic hits?
        |
        | Pre-provision before cutover traffic ramp
        v
[Migration batches begin — services cutting over one batch at a time]
        |
        v
[SSE] ← Concurrency gate: how many services can cut over simultaneously?
        |
        | SSE < 0.10: safe | 0.10-0.25: reduce batch | > 0.25: stop
        v
[Batches complete — dual-version coexistence accumulating cost]
        |
        v
[PCC] ← Economic forcing function: when must dual-version state end?
        |
        | PCC approaching threshold: accelerate to completion
        v
[Security vulnerability discovered during migration — patch required]
        |
        v
[PPC] ← Patch window gate: how many services can be patched in parallel?
        |
        | Batch by PPC, group by dependency tier, never exceed PPC
        v
[Migration complete]
```

**The interaction:** SLTD and SSE govern the active migration period. PCC creates the economic deadline for migration completion — preventing indefinite coexistence. PPC is an overlay that can fire at any time during migration when a security event requires patching within the transition period.

### Cluster 3: Validation and Decommission — Retiring a Service Safely

```
Service marked for decommission
        |
        v
[SCI] ← Shadow caller discovery: observed traffic = declared traffic?
        |
        | SCI > 0: blocked | SCI = 0 for < 30 days: observe
        | SCI = 0 for >= 30 days: proceed
        v
[Third-party forcing function discovered — deprecated API burning budget]
        |
        v
[EDFD] ← External deadline: by when must migration complete per budget math?
        |
        | Race between EDFD and migration duration — escalate if EDFD < duration
        v
[Migration complete, service decommissioned — 90-day timer starts for all fixes applied during migration]
        |
        v
[RDR] ← Fix validation at 90 days: did the fix eliminate the problem or displace it?
        |
        | RDR < 0.10: close | 0.10-0.50: follow up | > 0.50: reopen | > 1.0: rollback
        v
[Lifecycle complete]
```

**The interaction:** SCI must reach zero and stabilize before decommission can proceed. EDFD may create a deadline conflict — if shadow callers take three weeks to migrate and EDFD is in two weeks, you have a concrete escalation scenario with a quantified cost. RDR closes the loop on every fix applied during the migration or decommission period.

### Cross-Cluster Dependency: SSE and SCI

These two metrics share a structural relationship that is easy to miss.

SSE uses (1 - SLO_margin_remaining) — services in active cutover are in their most fragile state and depleting their SLO margin. Those same services may have undocumented shadow callers (SCI > 0) sending traffic to the old version that the team has not accounted for. When you cut those services over, the shadow callers fail silently — they continue hitting the old infrastructure that no longer exists, generating errors that burn the SLO margin that SSE is trying to protect.

The operational discipline: run SCI checks on every service before it enters the SSE calculation. A service with SCI > 0 should not be included in a migration cutover batch at all.

## Chapter Summary

1. **GEI** gates service admission by weighting dependency criticality against test coverage and canary duration. Canary hours are the fastest lever — compressing canary time can turn a safe deployment into a blocked one in a single calculation. The formula judges the deployment plan, not the service.

2. **RST** prevents reactive rollbacks based on statistical noise. Before RST is reached, error rate fluctuations are normal variation. After RST, they are genuine signals requiring deliberate action. The discipline is knowing which state you are in before you respond to data.

3. **SLTD** quantifies the autoscaling coverage gap for traffic events. Autoscaling is not instantaneous — the lag between HPA trigger and pod ready state creates an unserved window during steep ramps. SLTD measures that gap in pods and tells you exactly how many to pre-provision.

4. **PCC, SSE, and PPC** govern migration economics, migration concurrency safety, and patch window safety. PCC creates economic urgency for completing migrations by making coexistence cost visible and accumulating. SSE prevents simultaneous cutover of too many services by measuring compound exposure. PPC preserves rollback capability within constrained maintenance windows through conservative parallelism limits.

5. **RDR, SCI, and EDFD** close the service lifecycle loop. SCI prevents decommissioning services that have undocumented shadow callers. EDFD converts third-party deprecation timelines into budget math deadlines that are often earlier than the vendor's announced date. RDR validates at 90 days that fixes eliminated problems rather than displacing them to new failure modes.

**Exercise 7.1 — Shadow Caller Discovery (SCI)**

The Notification service is scheduled for decommission. Based on architecture documentation, the only registered caller is Order Management (sends order confirmation and shipping notification triggers).

60-day observed traffic data:
```
Order Management:  500 rps  (documented in registry)
User Profile:       80 rps  (not in registry — sends "account activity" notifications)
Reviews:            20 rps  (not in registry — sends "review reminder" notifications)
Total observed:    600 rps
```

Calculate SCI. What must happen before decommission can proceed? Identify the risks of the two shadow callers and explain which should be migrated first and why.

*Answer guide:*
```
declared_rps  = 500
observed_rps  = 600
SCI = (600 - 500) / 600 = 100/600 = 0.167
```

BLOCKED (SCI > 0). Both User Profile and Reviews must be registered as callers, their dependencies migrated to the replacement service, and SCI must reach 0 and remain stable for 30 days before decommission proceeds.

Migration priority: User Profile (80 rps) should be migrated first because it is higher volume and higher user visibility. Reviews (20 rps) appears lower risk but may include time-sensitive review reminder jobs with external deadlines. Both shadow callers are discovered, registered, and migrated before the 30-day SCI = 0 observation window begins.

**Exercise 7.2 — Rollout Signal Validity (RST)**

Search v2 is rolling out. Historical baseline for the Search service:
```
baseline_error = 0.08%
σ_baseline     = 0.015%
```

At 8% rollout, Search v2 shows an error rate of 0.11%.

*Part A:* Is this a statistically valid signal? Calculate z-score and state your conclusion.

*Part B:* At 8% rollout, what is the minimum error rate that constitutes a valid signal (z ≥ 2.0)?

*Answer guide:*

Part A:
```
z = (0.11 - 0.08) / 0.015
  = 0.03 / 0.015
  = 2.0
```
z = 2.0 — exactly at the valid signal boundary. This is a borderline case. Treat it as a valid signal: investigate the error type, but do not immediately roll back without confirming the errors are user-visible failures rather than internal retries or monitoring artifacts. At z = 2.0, there is approximately 95% confidence the difference is real.

Part B:
```
Need z ≥ 2.0:
feature_error ≥ baseline_error + (2.0 × σ_baseline)
feature_error ≥ 0.08 + (2.0 × 0.015)
feature_error ≥ 0.08 + 0.03
feature_error ≥ 0.11%
```

At 8% rollout: any error rate at or above 0.11% is a statistically valid signal. Below 0.11%: treat as noise and continue rollout without action.

**Exercise 7.3 (Advanced) — External Migration Deadline (EDFD)**

Browse journey SLA: 99.5%, monthly error budget = 216 min/month.

A third-party inventory data API that the Browse journey depends on has been deprecated. The deprecated API is already producing failures.

Parameters:
```
Budget reservation for migration: 10% of monthly budget
Current daily EBV burn from deprecated API: 0.05 min/day
Migration duration: 30 days
```

*Part A:* Calculate EDFD. How many days until the reserved budget is exhausted?

*Part B:* Must you start the migration immediately? What is your buffer in days?

*Part C:* The vendor accelerates deprecation. Daily burn increases to 0.15 min/day. Recalculate EDFD from today. Does the urgency change? At what burn rate would EDFD fall within the migration duration?

*Answer guide:*

Part A:
```
EBR_reserved = 0.10 × 216 = 21.6 min
EDFD = today + (21.6 / 0.05)
     = today + 432 days
```

432 days until the reserved budget is exhausted at current burn rate.

Part B:
Migration takes 30 days. EDFD is 432 days out. Buffer = 402 days. No immediate urgency — begin planning in normal sprint cycles and start migration within the next 30 days to maintain buffer.

Part C:
Burn rate triples to 0.15 min/day:
```
EDFD = today + (21.6 / 0.15)
     = today + 144 days
```

EDFD drops from 432 to 144 days — still comfortable relative to a 30-day migration (114-day buffer). However, the trend is the critical signal: if burn rate triples again from 0.15 to 0.45 min/day:

```
EDFD = today + (21.6 / 0.45) = today + 48 days
Buffer = 48 - 30 = 18 days
```

18-day buffer with a 30-day migration leaves no room for scope creep or delays. Monitor burn rate weekly. Trigger escalation protocol if it crosses 0.30 min/day.

Burn rate at which EDFD equals migration duration (30 days):
```
21.6 / burn_rate = 30
burn_rate = 21.6 / 30 = 0.72 min/day
```

If the deprecated API begins burning 0.72 min/day or more, start the migration immediately — the reserved budget runs out before the migration can complete.
