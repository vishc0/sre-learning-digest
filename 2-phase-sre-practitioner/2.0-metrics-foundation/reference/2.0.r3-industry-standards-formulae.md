# Industry Standards and Best Practices — Expressed as Formulae Using Coined Terms

## Purpose

Every major SRE framework — Google SRE Book, DORA, AWS Well-Architected, ITIL, Chaos Engineering, NIST patch standards — defines best practices in qualitative language. "Set realistic SLOs." "Minimize blast radius." "Patch critical vulnerabilities within 24 hours." These are correct principles that fail in practice because they cannot be measured, enforced, or prioritized without a quantitative framework.

This document takes the most important best practices from each industry standard and re-expresses them as precise formulae using the 25 coined terms from this framework. The result: every best practice becomes a threshold, a gate, or a calculation — not an aspiration.

**Structure per standard**: source principle (quoted) → traditional interpretation → formula using coined terms → operational threshold → how to measure it.

## Part 1 — Google SRE (Google SRE Book + Workbook)

### 1.1 Error Budget Policy

**Source principle**: *"The error budget provides a clear, objective metric that determines how unreliable the service is allowed to be within a single quarter."* — Google SRE Book, Chapter 3

**Traditional interpretation**: stop feature deployments when error budget is exhausted.

**Formula using coined terms**:
```
deploy_allowed = (budget_consumed / total_budget) < (1 - EBR(MRI))
```

Where `EBR(MRI) = min(MRI × 0.05, 0.80)` is the fraction that must remain in reserve before any change.

A deployment is permitted only when:
1. The consumed fraction is below the EBR threshold for the change's MRI, AND
2. EBV has been ≤ 1.3 for the past 24 hours (no active burn spike)

**Combined gate formula**:
```
deploy_gate = [budget_consumed < (1 - EBR)] AND [EBV_24h_avg ≤ 1.3]
```

**Operational threshold**: if either condition fails, the change is held until the next budget window (or until EBV normalizes).

**Measurement**: automated pre-flight check in CI/CD pipeline computes EBR(MRI) from the service registry, queries error budget consumption from the SLO platform, and blocks or approves the deployment before artifact is promoted.

### 1.2 Toil 50% Ceiling

**Source principle**: *"SRE teams should spend no more than 50 percent of their time on ops work."* — Google SRE Book, Chapter 5

**Traditional interpretation**: measure time spent on toil vs. engineering; intervene if toil exceeds 50%.

**Formula using coined terms**:

The 50% ceiling is a stock measure. TAF (Toil Amplification Factor) is the flow measure that predicts *when* the ceiling will be breached:

```
time_to_ceiling_months = log(0.50 / current_toil_fraction) / log(1 + (TAF - 1) × quarterly_traffic_growth_rate)
```

**Interpretation**: if current toil fraction is 35% and TAF = 1.4 with 25% quarterly traffic growth, the ceiling is breached in:
```
log(0.50/0.35) / log(1 + (1.4-1) × 0.25) = log(1.43) / log(1.10) = 0.355 / 0.0953 = 3.7 quarters
```

The team breaches the toil ceiling in under 4 quarters without intervention.

**Derived action threshold**: if `time_to_ceiling < 2 quarters` → automation sprint is mandatory, not optional.

**Toil investment formula**:
```
required_automation_reduction = current_toil_fraction - (0.45 / (1 + TAF_target × growth_rate))
```

Where `TAF_target = 0.5` (the automation goal: toil grows at half the rate of traffic growth).

### 1.3 Burn Rate Alerting

**Source principle**: *"Alert on burn rates [rather than raw error rates] to give better signal about SLO adherence."* — Google SRE Workbook, Chapter 5

**Traditional burn rate formula**: `burn_rate = error_rate / (1 - SLO)`
- Fast burn alert: burn_rate > 14.4 for 1h (exhausts 2% of 30-day budget in 1h)
- Slow burn alert: burn_rate > 1 for 6h

**Formula using coined terms — the EBV enhancement**:

Standard burn rate cannot distinguish "budget burning because reliability degraded" from "budget burning because traffic grew." EBV resolves this:

```
fast_burn_alert = [burn_rate > 14.4 for 1h] AND [EBV > 2.0 for 1h]
slow_burn_alert = [burn_rate > 1.0 for 6h] AND [EBV > 1.3 for 6h]
traffic_growth_alert = [burn_rate > 1.0] AND [EBV < 1.0] → investigate capacity, not reliability
```

If burn rate is elevated but EBV < 1.0: traffic grew faster than errors — reliability is actually improving. Do not page. Investigate capacity instead.

If burn rate is elevated and EBV > 1.5: reliability degraded independent of traffic growth — page immediately.

**Alert precision formula**:
```
APR_improvement_factor = (alerts_with_EBV_filter_actionable) / (alerts_without_EBV_filter_actionable)
```

Adding EBV filtering to burn rate alerts typically improves APR by 20–40 percentage points (reduces traffic-spike false positives that dominate the noise category in standard burn rate alerting).

### 1.4 Optimal SLO Setting

**Source principle**: *"The SLO should be set at the level that would make a typical customer unhappy if it fell below it — not at the maximum achievable."* — Google SRE Book, Chapter 4

**Traditional interpretation**: set SLO based on what customers care about, not what the system can achieve.

**Formula using coined terms — two-constraint SLO validation**:

An SLO is valid only when it satisfies both constraints simultaneously:

**Constraint 1 — MTBI-achievability** (can the system actually sustain this SLO?):
```
achievable_SLO = MTBI / (MTBI + MTTR)
```

If `achievable_SLO < declared_SLO`: the SLO is aspirational. The system will breach it based on historical incident frequency alone.

**Constraint 2 — SCS-coherence** (does this SLO compound to produce the customer SLA?):
```
SCS = per_service_SLO ^ CD / customer_SLA
SCS must be in [0.95, 1.10]
```

- SCS < 0.95: SLO is too weak — customer SLA is mathematically unsupported
- SCS > 1.10: SLO is too strong — over-investing in reliability beyond what the customer requires

**Combined SLO validity formula**:
```
SLO_valid = (achievable_SLO ≥ declared_SLO) AND (SCS ∈ [0.95, 1.10])
```

**The SLO gap**:
```
SLO_gap = achievable_SLO - declared_SLO
```

If SLO_gap < 0: the gap must be closed by reducing incident frequency (improving MTBI) or improving recovery speed (reducing MTTR) before the SLO can be legitimately declared.

### 1.5 On-Call Health Standard

**Source principle**: *"The primary on-call engineer should handle at most eight incidents per shift."* — Google SRE Book, Chapter 11

**Traditional interpretation**: track incident count per on-call shift; intervene if > 8.

**Formula using coined terms**:

The number of actionable pages per shift is a function of MTBI, shift length, and APR:
```
actionable_pages_per_shift = (shift_duration_hours / MTBI_hours) × APR
```

Where:
- `shift_duration_hours / MTBI_hours` = expected incidents per shift (raw)
- `× APR` = fraction that are actionable (the rest are noise, but they still interrupt)

**Two-part on-call health formula**:
```
interrupt_pages_per_shift = (shift_duration / MTBI) × 1.0  [all alerts interrupt, not just actionable ones]
actionable_pages_per_shift = (shift_duration / MTBI) × APR  [only actionable ones require response]
```

**Target derivation**:
- Google standard: ≤ 8 actionable incidents per 12-hour shift
- This requires: `(12 / MTBI) × APR ≤ 8` → `MTBI ≥ 12 × APR / 8`
- At APR = 0.80: `MTBI ≥ 1.2 hours` (MTBI must be at least 1.2 hours between actionable incidents)
- At APR = 0.30 (alert fatigue zone): `MTBI ≥ 0.45 hours` — which sounds good, but total interrupt count (not actionable) is `12 / 0.45 = 26.7 total alerts` — on-call burns out from noise even though only 8 are actionable

**The hidden on-call cost of low APR**:
```
total_interruptions_per_shift = shift_duration / MTBI_hours  [independent of APR]
cognitive_load_factor = total_interruptions / actionable_interruptions = 1 / APR
```

At APR = 0.30: cognitive load is 3.3× — for every actionable alert, the on-call engineer processes 2.3 noise alerts. Even with ≤ 8 actionable incidents, total cognitive interruptions may be 26.

**Standard**: APR must be > 0.70 AND total interruptions per shift < 15 (not just actionable ≤ 8).

### 1.6 Production Readiness Review (PRR) Gate

**Source principle**: *"Before a service is allowed to handle production traffic, it must pass a Production Readiness Review."* — Google SRE Book, Chapter 32

**Traditional interpretation**: checklist-based review covering deployment, monitoring, capacity, and incident response.

**Formula using coined terms — quantitative PRR score**:

A service passes PRR if and only if all 5 conditions are satisfied:

```
PRR_pass = (GEI ≤ 3)          [new service risk is manageable]
         × (OCR_service ≥ 0.80)  [all 4 golden signals instrumented]
         × (SCS ∈ [0.95, 1.10])  [SLO is coherent with consumer SLA]
         × (LBH_compliance = 1)  [service p99 ≤ allocated LBH]
         × (RV ≤ 30 min)          [rollback demonstrable within 30 minutes]
```

Each condition is binary: 1 (pass) or 0 (fail). PRR fails if any factor is 0.

**Canary duration from GEI**:
```
mandatory_canary_hours = max(24, GEI × 24)
```

A service with GEI = 2 must run canary for 48 hours minimum before PRR is signed off for full traffic.

**Post-PRR monitoring window**:
```
monitoring_window_days = max(7, GEI × 7)
```

During this window, the service is on "reliability watch" — SRE reviews metrics daily and can pull the service from production without a CAB review if reliability falls below RIR.

## Part 2 — DORA Four Key Metrics

### 2.1 Deployment Frequency — Safety-Adjusted

**Source principle**: *"Elite performers deploy to production multiple times per day."* — DORA State of DevOps Report

**Traditional interpretation**: higher deployment frequency = better organizational performance.

**Formula using coined terms — safety-adjusted deployment frequency**:

Raw deployment frequency is only safe when MRI and SCV are controlled:
```
safe_deployment_frequency = desired_frequency × max(0, 1 - (MRI / 10)) × max(0, 1 - SCV_normalized)
```

Where `SCV_normalized = SCV / SCV_baseline` (1.0 = typical deployment, > 1.0 = higher risk than normal).

**Interpretation**:
- At MRI = 0 and SCV = baseline: safe to deploy at desired frequency
- At MRI = 5: safe deployment frequency is 50% of desired (deploy half as often as planned)
- At MRI = 10: safe deployment frequency = 0 (change is blocked until MRI is reduced)

**DORA elite requires**: MRI < 2 AND SCV_normalized < 1.5 to support multiple-daily deploys without elevated incident risk.

---

### 2.2 Lead Time for Changes — Framework-Aware

**Source principle**: *"Lead time for changes (from code committed to running in production): Elite = less than one hour."* — DORA

**Traditional measurement**: git commit timestamp to production deploy timestamp.

**Formula using coined terms — effective lead time including safety windows**:
```
effective_lead_time = pipeline_time + GEI_canary_extension + RST_observation_window
```

Where:
- `GEI_canary_extension = max(0, (GEI - 1) × 24h)` — extra canary time for new/risky services
- `RST_observation_window = max(0, time to reach RST traffic percentage × observation_hours)`

**DORA elite achievable only when**:
- GEI < 1.0 (service is well-tested with short canary)
- RST is reached within the first 1% rollout (high-traffic service where signal is fast)
- Pipeline automation eliminates all manual gates below CAB threshold

**Design principle**: lead time and safety are in tension. Reducing lead time below the RST observation window means deploying before the error rate signal is statistically valid. Elite teams solve this by increasing traffic volume on canary (more traffic → RST reached faster) rather than shortening the observation window.

---

### 2.3 Change Failure Rate — Formula-Derived Ceiling

**Source principle**: *"Change failure rate: Elite = 0–15%."* — DORA

**Traditional measurement**: fraction of deployments causing degradation requiring immediate rollback or hotfix.

**Formula using coined terms**:
```
change_failure_rate = SCV × base_failure_coefficient
```

Where `base_failure_coefficient` is calibrated from historical data (typically 0.05–0.20 depending on test maturity).

**SCV ceiling for target CFR**:
```
SCV_ceiling = target_CFR / base_failure_coefficient
```

If target CFR = 0.15 and base_failure_coefficient = 0.10:
```
SCV_ceiling = 0.15 / 0.10 = 1.5
```

Services with SCV > 1.5 will exceed the 15% CFR target. Either reduce deployment frequency, increase test coverage (lower base_failure_coefficient), or reduce change surface area.

**Composite formula**:
```
CFR_predicted = deployment_frequency × mean_CSAI × change_failure_rate_per_change × (1 / test_coverage)
```

This shows that CFR is reducible by any of four levers: deploy less often, smaller changes, better testing, or better change surface control — all captured in SCV.

---

### 2.4 MTTR — Floor Defined by Framework

**Source principle**: *"Time to restore service: Elite = less than one hour."* — DORA

**Traditional measurement**: time from incident start to service restored.

**Formula using coined terms — MTTR decomposition**:
```
MTTR = MTTD + diagnosis_time + fix_time + validation_time
```

Where:
- `MTTD ≥ 0` (detection time, dependent on APR and OCR)
- `diagnosis_time ≥ f(OCR, FLI)` — lower OCR means longer diagnosis (can't see what's wrong); lower FLI means more services to diagnose
- `fix_time = deployment_rollback_time = RV`
- `validation_time = canary_observation_after_fix`

**MTTR floor formula**:
```
MTTR_floor = (1/APR_signal_quality × 5min) + (1/OCR_weighted × 10min) + RV + validation_time
```

At APR = 0.80 and OCR = 0.95: `MTTR_floor ≈ 6min + 11min + RV + 5min = 22min + RV`

For DORA elite (< 60 min): `RV ≤ 38 minutes` at these instrumentation levels.

At APR = 0.30 and OCR = 0.45 (shopping cart current state): `MTTR_floor ≈ 17min + 22min + RV + 5min = 44min + RV`

For DORA elite: `RV ≤ 16 minutes` — a very tight rollback constraint given poor observability.

**Insight**: improving APR and OCR directly reduces the minimum achievable MTTR, independent of how fast the fix is implemented.

## Part 3 — AWS Well-Architected Framework (Reliability Pillar)

### 3.1 Automatic Recovery from Failure

**Source principle**: *"Automatically recover from failure. Monitor for KPIs and configure systems to trigger an automated response when a threshold is breached."* — AWS Well-Architected Reliability Pillar

**Formula using coined terms**:

Automatic recovery has two components: failure containment (circuit breakers) and capacity recovery (autoscaling).

**Containment effectiveness**:
```
containment_score = FLI  [fraction of failures that self-contain without human action]
```

**Scale recovery effectiveness**:
```
scale_recovery_score = max(0, 1 - SLTD / available_headroom)
```

Where `SLTD > 0` means a lag exists; `SLTD = 0` means autoscaling catches up exactly.

**Combined automatic recovery score**:
```
auto_recovery_effectiveness = FLI × (1 - max(0, SLTD / available_headroom))
```

**Target**: `auto_recovery_effectiveness > 0.85`

At FLI = 0.72 (current) and SLTD = 1.94 pods (Black Friday): auto_recovery_effectiveness varies by scenario. During normal operations, SLTD ≈ 0 → score = 0.72. Below target; circuit breaker improvement needed.

---

### 3.2 Test Recovery Procedures

**Source principle**: *"Test recovery procedures. Use automation to simulate different failures or to recreate scenarios that led to failures before."* — AWS Well-Architected

**Formula using coined terms — GameDay success criteria**:

A GameDay experiment is valid and successful when:
```
gameday_valid = (FLI_pre ≥ 0.85) AND (EBV_pre ∈ [0.8, 1.2]) AND (APR > 0.60)
```

(Steady state must hold before injection)

```
gameday_success = (FLI_post ≥ FLI_pre × 0.90) AND (CC_injected ≤ CC_expected)
```

(Failures contained within 10% of pre-game baseline; no unexpected cascade)

**RDR applied to GameDay**:
```
RDR_gameday = cascades_observed / failures_injected
```

Target: `RDR_gameday < 0.10` — at most 10% of injected failures cascade to dependent services.

**GameDay frequency formula**:
```
gameday_frequency_days = 90 / max(CC_max, 1)
```

Higher CC services (more contagious failures) require more frequent GameDays to validate circuit breaker effectiveness. A service with CC = 5 should have a GameDay every 18 days. A service with CC = 0.5 can have GameDays every 180 days.

---

### 3.3 Stop Guessing Capacity

**Source principle**: *"Stop guessing capacity. Maintain the optimal level to satisfy demand without over-provisioning."* — AWS Well-Architected

**Formula using coined terms**:

Capacity planning replaces guessing with SLTD-based pre-scaling:

```
required_pre_scale_pods = ceil(SLTD × expected_ramp_rate / pod_capacity_rps)
```

```
optimal_CBI = 1 + (SLTD / pod_capacity_rps) + safety_buffer
```

Where `safety_buffer = 0.2` (20% headroom above what SLTD demands).

**Capacity plan completeness criteria**:
```
capacity_plan_complete = (CBI_peak ≥ 1.5) AND (SLTD ≤ 0 after pre-scaling)
```

SLTD ≤ 0 after pre-scaling means autoscaling will catch up before demand exceeds capacity — even on the fastest ramp the system has historically experienced.

**Over-provisioning formula**:
```
over_provision_cost = (actual_capacity - CBI_required × peak_demand) × cost_per_unit
```

Optimal provisioning minimizes over_provision_cost while keeping SLTD ≤ 0.

---

### 3.4 Manage Change in Automation

**Source principle**: *"Make changes using automation to reduce the risk of introducing unexpected outcomes."* — AWS Well-Architected

**Formula using coined terms**:

Automation readiness is the combined effect of APR (signals are trustworthy enough to gate automation) and the fraction of changes that are automated:
```
automation_readiness = APR × (1 - manual_change_fraction)
```

**Target**: `automation_readiness ≥ 0.70`

At APR = 0.31 (current state) and 40% manual changes: `automation_readiness = 0.31 × 0.60 = 0.186` — far below target. The system cannot safely automate deployments when more than 30% of alerts are noise (APR = 0.31) because the automated gate will false-positive or false-negative on 70% of signals.

**Implication**: APR must reach 0.70 before automated deployment gates are trustworthy. This is why alert redesign (APR improvement) is a prerequisite for deployment automation, not a parallel initiative.

## Part 4 — ITIL / ITSM Standards

### 4.1 Change Advisory Board — MRI-Based Trigger

**Source principle**: *"The CAB is responsible for assessing changes for potential impacts and risk before authorizing them."* — ITIL 4

**Traditional trigger**: organizational definition (usually "significant changes" requiring VP approval).

**Formula using coined terms — objective CAB trigger**:
```
standard_CR      → MRI < 1  [SRE auto-approve; notify on-call]
CAB_review       → 1 ≤ MRI < 3  [2 approvers: SRE lead + EM]
VP_approval      → 3 ≤ MRI < 8  [VP Engineering + SRE lead + CAB]
architecture_review_required → MRI ≥ 8  [BLOCK until pre-migration remediation complete]
```

**CAB time allocation formula**:
```
CAB_review_minutes = base_review_time × (MRI - 1) × complexity_factor
```

Where `base_review_time = 15 minutes` and `complexity_factor = 1 + (CSD / 10)`.

High-MRI, high-CSD changes get proportionally longer CAB review time — their approval is not a rubber stamp but a proportional investment in pre-authorization scrutiny.

---

### 4.2 Incident Classification — BRI-Governed Severity

**Source principle**: *"Incidents shall be categorized by their actual or potential impact on business operations."* — ITIL 4

**Formula using coined terms**:
```
SEV1 → BRI > 0.70 OR BRI_revenue > 0.50
SEV2 → BRI 0.30–0.70
SEV3 → BRI 0.10–0.30
SEV4 → BRI < 0.10
```

**Revenue-weighted BRI for payment journey**:
```
BRI_revenue = GMV_loss_rate / baseline_GMV_per_minute
```

This produces counterintuitive but correct severity classifications:
- Payment Service 40% error rate → BRI_revenue = 0.40 → SEV2 (despite low user-count BRI = 0.05)
- Product Catalog 100% outage → BRI = 0.18 (most users browsing, informational only) → SEV3

**Cascade severity escalation formula**:
```
effective_SEV = min(base_SEV, SEV_from_CC)
```

Where `SEV_from_CC`: if CC > 3 → escalate by one SEV level (cascade multiplies impact).

---

### 4.3 Problem Management — RCR and RDR Lifecycle

**Source principle**: *"Problem management aims to reduce the likelihood and impact of incidents by identifying actual and potential causes."* — ITIL 4

**Formula using coined terms — full Problem lifecycle**:

**Problem record creation threshold**:
```
create_problem_record = (same_root_cause_in ≥ 3 postmortems) OR (RCR > 0.50)
```

**Problem priority scoring**:
```
problem_priority_score = RCR × DG(affected_service) × (1 / MTBI)
```

High RCR (expensive incidents), high DG (widely impactful service), low MTBI (frequent) → highest priority.

**Problem resolution criteria**:
```
problem_resolved = (RDR < 0.10 for 90 days) AND (MTBI_new ≥ 2 × MTBI_original)
```

The fix must both not displace the problem (RDR < 0.10) and demonstrably double the time between incidents (MTBI improvement).

**Known error status** (ITIL term: fix deferred but workaround documented):
```
known_error_status = (RCR < 0.50) AND (runbook_exists = true)
```

If the fix costs more than the incidents, document the workaround and set `known_error_status`. Revisit when RCR crosses 0.50 due to increasing incident frequency.

## Part 5 — Chaos Engineering (Chaos Monkey, Gremlin, LitmusChaos Principles)

### 5.1 Steady-State Hypothesis

**Source principle**: *"Define a measurable, normal behavior of the system before injecting chaos."* — Principles of Chaos Engineering

**Formula using coined terms**:
```
steady_state = {
  FLI > 0.85,           // failures are self-contained
  EBV ∈ [0.8, 1.2],    // error rate is proportional to traffic (neither burning nor recovering)
  APR > 0.60,           // alert system is trustworthy
  SCS ∈ [0.95, 1.10]   // SLO coherence holds
}
```

An experiment is valid only if `steady_state = true` at the start. Injecting failures into a system that is already in a degraded state (EBV > 1.2 or FLI < 0.85) produces results that cannot be attributed to the injection alone.

**Steady-state restoration time**:
```
restoration_time = max(MTTR_recent, 2 × MTBI_observation_window)
```

After any real incident, wait at least this long before running a chaos experiment — the system needs to demonstrate sustained steady state, not just momentary recovery.

---

### 5.2 Blast Radius Minimization

**Source principle**: *"Minimize blast radius. Start small and build confidence over time."* — Principles of Chaos Engineering

**Formula using coined terms**:
```
max_experiment_BRI ≤ 0.10   [maximum 10% customer impact during any experiment]
max_experiment_CC ≤ 2.0     [maximum 2 expected cascades from injected failure]
```

**Experiment approval gate**:
```
experiment_approved = (predicted_BRI ≤ 0.10) AND (CC_target_service ≤ 2.0)
```

For services with CC > 2.0: the experiment must be scoped to a single AZ or instance (not the full service) to keep predicted_BRI within bounds.

**Progressive experiment schedule** based on FLI improvement:
```
experiment_scope_pct = min(100%, FLI × 100%)
```

At FLI = 0.72: max experiment scope = 72% of the service's pods (keeping 28% healthy guarantees BRI stays below 0.10 for most services).

---

### 5.3 Hypothesis-Driven Experiments

**Source principle**: *"Run experiments in production with real users, not just in staging with synthetic traffic."* — Principles of Chaos Engineering

**Formula using coined terms — RDR as the experimental validation metric**:

Every chaos experiment tests a specific hypothesis about resilience. The hypothesis is validated using RDR:

```
hypothesis = "circuit breaker on Cart → Auth call prevents checkout cascade when Auth fails"
experiment = inject Auth Service failure for 5 minutes
expected: RDR_experiment < 0.10 (checkout failure rate ≤ 10% of Auth failure rate)
actual: measure RDR_experiment = checkout_failures / auth_failures
```

If `RDR_experiment > expected`: hypothesis rejected → circuit breaker is not working as expected → fix circuit breaker.

This turns every chaos experiment into a precise RDR measurement, connecting chaos engineering directly to the framework's effectiveness validation.

## Part 6 — NIST / Security Standards

### 6.1 Vulnerability Patch SLAs (CVSS-Derived)

**Source principle** (CISA / NIST SSDF): Critical vulnerabilities (CVSS ≥ 9.0) must be patched within 24 hours. High vulnerabilities (CVSS ≥ 7.0) within 72 hours.

**Formula using coined terms — PPC-based feasibility check**:
```
patch_feasible = PPC × SLA_window_hours ≥ affected_services × avg_patch_hours
```

If `patch_feasible = false`: the patch SLA cannot be met by sequenced deployment. Options:
1. Increase PPC by pre-staging the patch artifact before the CVE is public (reduces `avg_patch_hours`)
2. Pre-position rollback snapshots (reduces `avg_rollback_hours`, which increases `PPC`)
3. Accept SLA breach and document compensating controls

**Patch readiness formula** (run quarterly as a drill):
```
patch_readiness_score = PPC / total_services × SLA_window_hours / avg_patch_hours
```

Target: `patch_readiness_score ≥ 1.5` (comfortable margin). Below 1.0 means the organization cannot meet the CVSS critical patch SLA with current tooling.

---

### 6.2 Security Change Management

**Source principle**: *"Require security review for changes affecting authentication, authorization, data handling, and network topology."* — ISO 27001 / SOC 2

**Formula using coined terms — security-augmented MRI**:
```
MRI_security = MRI × security_multiplier
```

Where `security_multiplier`:
- Auth Service change: × 2.0 (authentication is the blast radius amplifier)
- Payment Service change: × 1.8 (financial data, PCI scope)
- Network topology change: × 1.5 (blast radius is entire system)
- Data schema change affecting PII: × 1.5
- All other: × 1.0

**Security gate formula**:
```
security_review_required = MRI_security > 2.0 OR security_multiplier > 1.0
```

Any change to Auth Service automatically requires security review regardless of MRI (because Auth DG = 18 means any compromise has maximum blast radius).

## Part 7 — SRE Framework-Derived Composite Formulae

These are new composite formulae derived from combining multiple coined terms into unified reliability scores. They have no direct industry standard source but represent the synthesis of multiple standards applied simultaneously.

### 7.1 Reliability Investment Score (RIS)

**Purpose**: rank reliability investments across the backlog by return on investment, adjusted for implementation risk.

```
RIS = (RCR × OL) / MRI
```

- High RCR: incident cost significantly exceeds fix cost (strong ROI)
- High OL: the investment efficiently converts engineering time into reliability improvement
- Low MRI: the investment itself is low-risk to implement

**Sorted backlog example**:

| Investment | RCR | OL | MRI | RIS | Priority |
| --- | --- | --- | --- | --- | --- |
| Kafka watchdog | 1.2 | 18 | 0.5 | 43.2 | **1st** |
| Auth pod scaling | 0.8 | 12 | 1.0 | 9.6 | **2nd** |
| Alert redesign | 0.6 | 8.3 | 0.8 | 6.2 | **3rd** |
| Inventory CB | 0.4 | 6 | 1.5 | 1.6 | **4th** |
| Auth cert hot-reload | 0.6 | 4 | 2.0 | 1.2 | **5th** |

RIS provides a single sortable number that replaces the subjective "this feels important" prioritization that dominates most reliability backlogs.

---

### 7.2 System Reliability Maturity Index (SRMI)

**Purpose**: a single number representing the overall operational maturity of the system, measuring completeness of reliability practices.

```
SRMI = (FLI × OCR_weighted × (APR / 0.85)) / (TAF × CC_max_normalized)
```

Where:
- `CC_max_normalized = CC_highest_service / 10` (normalized to 0–1 scale, assuming CC ceiling of 10)
- `(APR / 0.85)` scales APR to 1.0 at the "good" threshold (caps at 1.18 for APR = 1.0)

**Shopping cart current SRMI**:
```
SRMI = (0.72 × 0.457 × (0.31/0.85)) / (0.62 × (10.8/10))
     = (0.72 × 0.457 × 0.365) / (0.62 × 1.08)
     = 0.120 / 0.670
     = 0.179
```

**Shopping cart target SRMI** (12-month):
```
SRMI = (0.92 × 0.95 × (0.80/0.85)) / (0.50 × (3.0/10))
     = (0.92 × 0.95 × 0.941) / (0.50 × 0.30)
     = 0.823 / 0.15
     = 5.49
```

SRMI improvement from 0.18 to 5.49 (30× improvement) represents the distance between current state and mature SRE practice. This is the single number that answers "how mature is our SRE program?" for executive reporting.

### 7.3 Deployment Safety Score (DSS)

**Purpose**: before any deployment, compute a composite safety score that captures all deployment-time risk factors.

```
DSS = (1 - SCV_normalized) × (1 - MRI/10) × APR × (budget_remaining / total_budget)
```

Where `SCV_normalized = SCV / SCV_90th_percentile_for_this_service_type`.

**DSS thresholds**:

| DSS | Deployment recommendation |
| --- | --- |
| DSS > 0.70 | Deploy with standard canary |
| DSS 0.40–0.70 | Deploy with extended canary; SRE monitoring required |
| DSS 0.20–0.40 | Deploy in maintenance window only; SRE on-call bridge |
| DSS < 0.20 | Block deployment; reduce MRI or improve APR before proceeding |

**Automated gate**: DSS is computed in CI/CD for every deployment. The pipeline blocks at DSS < 0.20, requires SRE approval at DSS < 0.40, and auto-approves at DSS > 0.70.

### 7.4 Journey Reliability Coherence Score (JRCS)

**Purpose**: a per-journey composite that measures whether all reliability properties are simultaneously coherent for a specific customer journey.

```
JRCS(journey) = SCS × FLI × (OCR_coverage_on_journey_path / 1.0) × (1 - CC_max_on_path / 10)
```

Where:
- `SCS`: SLO coherence for this journey
- `FLI`: system-level failure locality
- `OCR_coverage_on_journey_path`: weighted observability for services on this specific journey's call chain
- `CC_max_on_path`: highest CC among services on this journey's call chain

**Shopping cart Checkout JRCS** (current):
```
JRCS(checkout) = 1.0 × 0.72 × 0.60 × (1 - 10.8/10)
               = 1.0 × 0.72 × 0.60 × (-0.08)
               = negative → CASCADE RISK EXCEEDS SAFETY THRESHOLD
```

Auth's CC = 10.8 makes the JRCS term `(1 - 10.8/10)` negative — meaning cascade risk on the checkout path exceeds any achievable reliability from other factors. The circuit breaker gap on Auth's dependents must be fixed before JRCS can be computed meaningfully.

**Target Checkout JRCS** (after circuit breaker additions reduce Auth CC to 2.0):
```
JRCS(checkout) = 1.0 × 0.92 × 0.95 × (1 - 2.0/10)
               = 1.0 × 0.92 × 0.95 × 0.80
               = 0.699  → Good (target ≥ 0.60)
```
