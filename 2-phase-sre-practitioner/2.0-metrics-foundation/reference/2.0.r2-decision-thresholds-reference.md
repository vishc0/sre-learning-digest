# Decision Thresholds Reference — Complete Gate Reference

All gates, thresholds, and action triggers in one place. Open this during incidents, CAB calls, and postmortems.

---

## 1. SEVERITY DECLARATION (BRI)

```
BRI = (affected_users/total) × journey_criticality × tier_weight_of_failing_service
```

| BRI | Severity | Who to Call | Time Constraint |
|-----|----------|-------------|----------------|
| > 0.70 | SEV1 | Incident commander + VP Eng + Customer Success + all service owners in chain | Acknowledge in 5 min, bridge in 10 min |
| 0.30 – 0.70 | SEV2 | SRE lead + engineering team + service owner | Acknowledge in 15 min, bridge in 30 min |
| 0.10 – 0.30 | SEV3 | On-call SRE + service team | Acknowledge in 30 min, update in 1 hr |
| < 0.10 | SEV4 | Service team only | Next business day |

**Journey criticality reference:** Checkout = 1.0, Add-to-Cart = 1.0, Browse = 0.6, Notification = 0.3, Reviews = 0.2

**Escalation trigger:** If BRI rises during incident (more users affected as cascade spreads), re-declare severity immediately. Do not wait for next update cycle.

## 2. CHANGE REQUEST APPROVAL TIER (MRI)

```
MRI = DG × CD × (1 - FLI) × (2 - OCR_weighted)
```

| MRI | Approval Tier | Requirements |
|-----|--------------|-------------|
| < 1 | Notify | Post to #changes channel. No approval required. |
| 1 – 3 | CAB | 2 approvers from: SRE lead, service owner, peer engineer. 48hr review window. |
| 3 – 8 | VP Engineering | Written justification + risk mitigation plan. 72hr review. |
| ≥ 8 | BLOCKED | Cannot proceed regardless of business pressure. Fix system state first. |

**What lowers MRI:** Improve FLI (circuit breakers) and OCR (instrumentation). These are the only levers.

**MRI formula sensitivity:**
- FLI: 0.72 → 0.90 cuts (1-FLI) from 0.28 to 0.10 — a 64% reduction in MRI
- OCR: 0.457 → 0.70 cuts (2-OCR) from 1.543 to 1.30 — a 16% reduction in MRI
- Combined: MRI drops ~68%. Most blocked changes become VP-approvable after FLI+OCR fixes.

**Shopping cart MRI examples at current system state (FLI=0.72, OCR=0.457):**

| Service | DG | Journey CD | MRI | Approval |
|---------|-----|-----------|-----|---------|
| Auth | 18 | 5 | 38.9 | BLOCKED |
| Cart | 10 | 5 | 21.6 | BLOCKED |
| Inventory | 8 | 5 | 17.3 | BLOCKED |
| Search | 7 | 3 | 9.1 | BLOCKED |
| Product Catalog | 6 | 3 | 7.8 | VP Eng |
| Pricing | 4 | 3 | 5.2 | VP Eng |
| Notification | 3 | 1 | 1.3 | CAB |

## 3. BUDGET GATE (EBR)

```
EBR = min(MRI × 0.05, 0.80)
Gate: EBR must be ≤ budget_remaining_fraction
```

| Condition | Decision |
|-----------|---------|
| EBR ≤ budget remaining | Budget gate passes — check other gates |
| EBR > budget remaining | BLOCKED by budget — defer to next month |
| Budget remaining < 5% | Freeze all non-emergency changes regardless of EBR |

**EBR quick lookup:**

| MRI | EBR | Budget needed |
|-----|-----|--------------|
| 1 | 5% | Need ≥ 5% remaining |
| 2 | 10% | Need ≥ 10% remaining |
| 4 | 20% | Need ≥ 20% remaining |
| 8 | 40% | Need ≥ 40% remaining (but MRI=8 is blocked anyway) |
| 16+ | 80% (capped) | Moot — blocked by MRI tier |

**Budget strategy:** High-MRI changes must be scheduled at the start of the budget month (day 1–5) when 100% budget is available.

## 4. WINDOW SIZING (RV)

```
deployment_window ≥ 3 × rollback_time (RV)
```

| Condition | Decision |
|-----------|---------|
| window ≥ 3×RV | Window is sufficient |
| window < 3×RV | INSUFFICIENT — extend window or reduce RV |
| window ≥ 3.5×RV | Recommended — includes decision buffer |

**Why 3×RV:**
- 1×RV: Time to detect the problem (alerts must fire, SRE must see them)
- 1×RV: Time to assess and make rollback decision
- 1×RV: Time to execute rollback

**Window sizing examples:**

| RV | Min Window | Recommended |
|----|-----------|-------------|
| 10 min | 30 min | 35 min |
| 15 min | 45 min | 52 min |
| 22 min | 66 min | 77 min → use 90 min |
| 35 min | 105 min | 122 min → use 120 min |
| 45 min | 135 min | 158 min → use 3 hr window |

**Data migration adds time:** For schema migrations with data backfill, add backfill_time to RV before computing minimum window.

## 5. DEPLOYMENT MONITORING (DSS)

```
DSS = (1 - SCV_norm) × (1 - MRI/10) × APR × budget_remaining_fraction
      SCV_norm = min(SCV/5, 1.0)
```

| DSS | Monitoring Level | SRE Requirements |
|-----|-----------------|-----------------|
| > 0.70 | Standard | Normal pipeline. Automated rollback sufficient. |
| 0.40 – 0.70 | SRE bridge | SRE on bridge during deploy. Manual rollback authorized. |
| 0.20 – 0.40 | Maintenance window | Scheduled window. SRE + service owner present. Rollback tested. |
| < 0.20 | BLOCK | Do not deploy. Improve DSS inputs first. |

**DSS sensitivity — what moves it most:**

| Input worsens | DSS impact |
|--------------|-----------|
| APR drops 0.80 → 0.31 | DSS drops 61% |
| Budget drops 100% → 30% | DSS drops 70% |
| MRI rises 2 → 8 | DSS drops 75% |
| SCV rises 0 → 1.0 | DSS drops 20% |

APR and remaining budget are the fastest levers. Fix alert quality before planning deploys in the second half of the month.

## 6. INVESTMENT PRIORITIZATION (RCR)

```
RCR = (response_hours × frequency_per_year) / fix_hours
```

| RCR | Decision | Action |
|-----|---------|--------|
| > 2.0 | Fix immediately | Current sprint. Authorize overtime if needed. |
| 0.50 – 2.0 | Fix | Next sprint. Prioritize by RCR descending. |
| 0.10 – 0.50 | Evaluate | Fix if: natural refactor opportunity, or blocking other work |
| < 0.10 | Defer | Document. Revisit when frequency or fix cost changes. |

**RCR interpretation:** RCR = 1.0 means the toil costs exactly as many hours per year as the fix. Every year you defer is another year's worth of engineering hours lost.

**RIS for investment ranking when multiple options compete:**
```
RIS = (RCR × OL) / MRI
Higher RIS = prioritize first
```

## 7. ALERT QUALITY (APR)

```
APR = actionable_alerts / total_alerts
```

| APR | Status | Action Required |
|-----|--------|----------------|
| > 0.80 | Healthy | Maintain. Review quarterly. |
| 0.60 – 0.80 | Degrading | Alert audit within 2 weeks. Identify and suppress top noise sources. |
| 0.40 – 0.60 | Poor | Alert redesign sprint. Suspend lowest-value rules immediately. |
| < 0.40 | Critical | On-call team is operating in noise. Emergency alert reduction. Remove rules until APR > 0.60. |

**To reach APR target:** You cannot add more actionable alerts. You must eliminate non-actionable ones.
```
required_total = actionable_count / 0.80
alerts_to_eliminate = current_total - required_total
```

**Alert elimination process:**
1. Pull 30-day alert history
2. Identify rules that fired but produced no SRE action > 80% of the time
3. Delete or convert to dashboard metric (not paging)
4. Raise thresholds on remaining rules until they only fire when action is required

## 8. CASCADE RISK (CC)

```
CC = Σ[P(dep_fails | X_fails) × tier_weight(dep)]
     P = 0.9 (no CB) | 0.5 (partial CB) | 0 (full CB)
```

| CC | Risk Level | Circuit Breaker Requirement |
|----|-----------|---------------------------|
| < 2 | Low | Monitor. No immediate CB required. |
| 2 – 5 | Moderate | Plan CBs for highest-P callers in next quarter. |
| 5 – 10 | High | Circuit breakers required on P0 caller paths this sprint. |
| > 10 | Critical | JRCS will be negative. Emergency CB deployment. Freeze dependent changes. |

**CB priority:** Target callers with highest `P × tier_weight` contribution first. One CB on a P0 caller (P=0.9, tw=1.0) removes 0.9 from CC — equivalent to 1.8 CBs on P2 callers.

**CC and JRCS interaction:** JRCS = ... × (1 - CC_max/10). When CC_max > 10, (1 - CC_max/10) < 0, making JRCS negative regardless of all other metrics. The only fix is reducing CC_max below 10.

**Shopping cart CC_max=10.8 (Auth):** Three CBs on Cart→Auth, Payment→Auth, Order Mgmt→Auth reduce CC_max from 10.8 to ~1.2. Cost: 40 engineering hours. JRCS goes from negative to ~0.55.

## 9. SYSTEM MATURITY (SRMI)

```
SRMI = (FLI × OCR × APR/0.85) / (TAF × CC_max/10)
```

| SRMI | Maturity Level | What It Means | Investment Priority |
|------|---------------|--------------|-------------------|
| > 0.80 | Mature | System is production-ready for traffic events | Maintenance mode. Focus on new capabilities. |
| 0.50 – 0.80 | Developing | Most practices in place. Known gaps. | Close remaining FLI/OCR gaps. |
| 0.30 – 0.50 | Early | Foundation built but significant risk remains | FLI and CC_max are primary targets. |
| < 0.30 | Immature | System is not safe for major traffic events | All metrics need improvement. Start with CC/FLI. |

**Shopping cart SRMI = 0.179 (immature).** Breakdown:
- Numerator drag: APR=0.31 (below 0.85 baseline → ratio 0.365 pulls number down)
- Denominator amplification: CC_max=10.8 (CC_max/10 = 1.08, amplifies denominator)
- Both must improve simultaneously for SRMI to move materially

**Path to SRMI > 0.50 for shopping cart:**
1. Deploy CBs on Auth → CC_max drops to ~1.2 → denominator drops from 0.670 to 0.62×0.12=0.074
2. Fix alert noise → APR rises to 0.80 → APR/0.85 = 0.94 (vs current 0.365)
3. Add observability → OCR rises to 0.70 → numerator improves
4. Result: SRMI = (0.90×0.70×0.94)/(0.62×0.12) = 0.592/0.074 = 7.99 → mature

## 10. NEW SERVICE GATE (GEI)

```
GEI = (new_deps × avg_tier_weight) / (test_coverage × canary_hours)
```

| GEI | Risk Level | Canary Requirement |
|-----|-----------|-------------------|
| < 1.0 | Low | Minimum 6 hours. Standard canary. |
| 1.0 – 3.0 | Moderate | Minimum 24 hours. SRE monitored. Rollback tested. |
| > 3.0 | High | Minimum 72 hours or BLOCK until test coverage improves. |

**GEI levers:**
- Increase `test_coverage` (run more integration tests before canary)
- Increase `canary_hours` (longer observation window)
- Reduce `new_deps` (integrate fewer external dependencies per release)
- Reduce `avg_tier_weight` (onboard P2/P3 deps before P0 deps)

**Do not shorten canary hours to meet a deadline.** Shorter canary raises GEI and increases integration risk. If under time pressure: reduce scope (fewer new deps per release), not observation time.

## 11. DECOMMISSION GATE (SCI)

```
SCI = (observed_rps - declared_rps) / total_observed_rps
      Must = 0.0 before decommissioning
```

| SCI | Status | Action |
|-----|--------|--------|
| 0.0 for ≥ 30 days | CLEAR | Proceed with decommission |
| 0.0 for < 30 days | WAIT | Continue observation to full 30-day window |
| > 0.0 | BLOCKED | Identify all callers. Migrate or notify. Restart 30-day clock. |

**30-day minimum rationale:** Monthly batch jobs, invoicing cycles, and reporting pipelines may only call a service once per month. A 7-day or 14-day observation window will miss them. 30 days covers one full monthly traffic cycle.

**Unknown caller protocol:** If a caller cannot be identified within 5 business days:
1. Enable full request logging (caller IP, user-agent, trace headers)
2. Correlate against service mesh registration
3. Check API gateway access logs
4. If still unidentified after 10 days: halt decommission, escalate to architecture review
5. Never decommission with SCI > 0 regardless of deadline pressure

**Post-decommission:** Retain infrastructure for 14 days with monitoring. Set alert for any traffic to decommissioned endpoints. Any traffic detected = restore immediately and restart investigation.

## 12. ROLLOUT SIGNAL (RST)

```
RST = rollout_percentage where:
      (feature_error_rate - baseline_error_rate) / σ_baseline ≥ 2
```

| z-score | Action |
|---------|--------|
| < 1.0 | Continue rollout — signal within normal variation |
| 1.0 – 2.0 | Watch closely — elevated but not statistically significant |
| ≥ 2.0 | STOP rollout — statistically significant regression detected |
| ≥ 3.0 | ROLLBACK immediately — strong signal of regression |

**Statistical significance means:** A z-score of 2 means there is a 95% probability the elevation is real (not noise). At z=3, probability rises to 99.7%.

**Minimum sample size before RST is meaningful:** Need at least 1,000 requests at the canary percentage before computing RST. At low traffic rates, wait for sufficient sample accumulation before acting on z-score.

**Feature flag rollout sequence:** 1% → 5% → 10% → 25% → 50% → 100%. Compute RST at each stage. If RST triggers at 5%, the blast radius of the regression is contained to 5% of users.

## 13. TOIL HEALTH (TAF)

```
TAF = toil_growth_rate / traffic_growth_rate
```

| TAF | Status | Investment Priority |
|-----|--------|-------------------|
| < 0.50 | Excellent | Automation ahead of growth. Team is scaling. |
| 0.50 – 0.80 | Healthy | Automation keeping pace. Monitor quarterly. |
| 0.80 – 1.0 | Warning | Toil approaching parity with growth. Automation sprint needed. |
| 1.0 – 1.5 | Concerning | Toil growing faster than traffic. Team will fall behind at current trajectory. |
| > 1.5 | Critical | Toil is a scaling emergency. Automation investment authorized immediately. |

**TAF decomposition:** Identify which toil category drives growth using weighted contribution:
```
overall_TAF_driver = (fraction_A × growth_A) + (fraction_B × growth_B) + ...
```
Eliminate the highest-contributing category first. In most systems, alert noise is the dominant driver (see APR).

**TAF and headcount:** A team with TAF=1.0 requires linear headcount growth to match traffic growth. A team with TAF=0.50 can double traffic without adding headcount. This is the quantitative case for automation investment in headcount planning discussions.

## 14. JOURNEY HEALTH (JRCS)

```
JRCS = SCS × FLI × OCR_path × (1 - CC_max/10)
```

| JRCS | Status | Customer Experience Risk |
|------|--------|------------------------|
| > 0.70 | Excellent | Journey is structurally sound. Low customer impact risk. |
| 0.40 – 0.70 | Healthy | Journey is safe but has observable gaps. Monitor. |
| 0.10 – 0.40 | At Risk | Journey has structural vulnerabilities. Improve weakest metric. |
| 0 – 0.10 | Marginal | Journey is barely positive. One metric degradation tips it negative. |
| < 0 | UNSAFE | CC_max > 10 OR compound of all metrics too low. Emergency action. |

**Negative JRCS interpretation:** If (1 - CC_max/10) is negative (CC_max > 10), no improvement to SCS, FLI, or OCR can make JRCS positive. The only fix is reducing CC_max below 10. This means: deploy circuit breakers on the highest-CC service before any other reliability investment delivers JRCS benefit.

**JRCS improvement path for shopping cart checkout (current: -0.035):**

| Step | Change | New JRCS |
|------|--------|---------|
| Current | CC_max=10.8, FLI=0.72, OCR=0.60 | -0.035 |
| After CBs on Auth | CC_max=1.2 | +0.418 |
| After alert redesign | APR=0.80 (indirect — APR affects DSS not JRCS directly) | +0.418 |
| After OCR improvement | OCR_path=0.80 | +0.557 |
| After FLI improvement | FLI=0.90 | +0.696 |

Each step compounds. Circuit breakers are the prerequisite — all other improvements deliver zero JRCS benefit until CC_max drops below 10.

## GATE DECISION QUICK GUIDE

When a change request arrives, run these checks in order. Stop at first BLOCK.

```
1. MRI gate:    MRI = DG × CD × (1-FLI) × (2-OCR)
                If ≥ 8 → BLOCK. Done.

2. Budget gate: EBR = min(MRI×0.05, 0.80)
                If EBR > budget_remaining → BLOCK. Defer to next month.

3. Window gate: Is window ≥ 3 × RV?
                If no → BLOCK. Extend window or reduce rollback time.

4. DSS gate:    DSS = (1-SCV_norm)×(1-MRI/10)×APR×budget_remaining
                If < 0.20 → BLOCK.
                If 0.20–0.40 → Maintenance window required.
                If 0.40–0.70 → SRE bridge required.
                If > 0.70 → Standard pipeline.

5. All gates pass → proceed with approval tier from MRI table.
```

When an incident is declared, run these checks in order:

```
1. BRI:  BRI = (affected/total) × journey_criticality × tier_weight
         Maps to SEV1/2/3/4 → triggers correct response.

2. CC:   Are circuit breakers preventing cascade spread?
         If CC_max > 5 and cascading: open CBs manually if available.

3. FLI:  Track FLI as incident unfolds.
         FLI = contained / total events.
         If FLI < 0.50, cascade is active — containment is the first priority.

4. EBV:  After incident resolves, compute new EBV and EDFD.
         If EDFD < end of month: freeze non-critical changes.
```
