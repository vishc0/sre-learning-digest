# Chapter 2: Structural Terms — Call Depth, SLO Coherence Score, and Latency Budget Per Hop

## Learning Objectives

By the end of this chapter, you will be able to:
- Define Call Depth (CD) and correctly count synchronous hops in a call chain, excluding asynchronous calls
- Calculate the required per-service SLO for any customer SLA and call depth using the CD formula
- Compute the SLO Coherence Score (SCS) and interpret whether a system's declared SLOs are mathematically coherent with its customer SLA
- Calculate the Latency Budget Per Hop (LBH) and use it to set per-service p99 latency targets
- Apply all three structural terms together to evaluate a proposed architectural change before it reaches production

---

## The Problem This Solves

Chapter 1 showed you that compound reliability is mathematically worse than any individual service's reliability, and that the gap between declared SLOs and actual customer experience can be large, invisible, and damaging. But knowing the problem exists is not enough. You need a way to *measure* the structural risk before incidents occur — to look at an architecture diagram and say with precision: "This system cannot deliver on its customer SLA as designed," or "This proposed change will require renegotiating three service SLOs."

Without these measurements, architectural decisions about microservice decomposition are made on intuition: "One more service seems fine," or "The team says they can hit 99.99% — let's put them in the checkout chain." With these measurements, architectural decisions become engineering conversations with specific numbers, specific thresholds, and specific recommended actions.

The three terms in this chapter — CD, SCS, and LBH — are the first vocabulary of structural reliability. They describe the *shape* of the system before any failure occurs. They answer three questions: How many hops does a request take? Do the declared SLOs add up to the customer SLA? And how much latency does each service actually have budget for?

These are not operational metrics you collect at runtime. They are architectural properties you derive from the design. And because they are architectural, they can — and must — be evaluated during design review, before code is written, before services are deployed.

---

## CD — Call Depth

### The Concept

Call Depth counts the number of sequential, synchronous hops a customer request must traverse to complete a journey. Think of it as the number of times your request is transferred to another department before it comes back with an answer.

You have experienced this in support center interactions. You call tech support. The first agent cannot help you and transfers you to Tier 2. Tier 2 cannot resolve it and escalates to a specialist. The specialist needs to consult the backend team. Each transfer is a hop — and each hop adds two risks: the risk that the receiving department is unavailable, and the time you spend waiting on hold while the transfer happens. In a synchronous call chain, your request is "on hold" at every hop. The customer cannot receive a response until every hop has completed and the response has traveled back through the entire chain.

This is exactly what happens in a microservice checkout. When a customer clicks "Place Order," the request flows: API Gateway receives it → calls Auth to validate the session → Auth responds → API Gateway calls Cart to retrieve items → Cart responds → API Gateway calls Payment to charge the card → Payment responds → API Gateway calls Order Management to create the order → Order Management responds → API Gateway returns confirmation to the customer.

That is five hops. Every hop must succeed. Every hop adds latency. CD = 5.

**The critical boundary: synchronous vs. asynchronous.**

CD counts only *synchronous, blocking* calls — calls where the caller waits for a response before it can proceed. Asynchronous calls — publishing to a Kafka topic, sending to a RabbitMQ exchange, emitting an event — do not count toward CD. The caller does not wait. If the consumer fails, the message stays in the queue and can be retried without breaking the customer-facing journey.

This distinction has architectural implications. A team proposing to add a new service to a synchronous call chain is increasing CD. A team proposing to convert a synchronous call to an asynchronous message is *decreasing* CD. Both changes affect the compound reliability math, in opposite directions. CD makes that effect visible and quantifiable.

**Counting correctly:**

Count from the first service the customer request touches to the last service whose response is required for the customer to receive a result. Include every synchronous service-to-service call in the critical path. Exclude:
- Parallel fan-out calls where any one response suffices
- Fire-and-forget async publishes
- Database reads (these are internal to the service — the service is one hop, not one hop per database query)
- Cache reads (same reasoning)

If a service makes two parallel synchronous calls and waits for both, count that as one hop (not two) — the depth of the chain does not increase, though the failure probability for that hop does.

### Formula

```
CD = count of sequential synchronous service-to-service hops in the critical path

Required per-service SLO = customer_SLA ^ (1 / CD)
```

**Components:**

- `customer_SLA`: The end-to-end availability target for the customer journey, expressed as a decimal (e.g., 99.95% = 0.9995)
- `CD`: The call depth of the journey — count of synchronous hops
- `(1 / CD)`: The exponent that distributes the SLA budget equally across all hops
- `customer_SLA ^ (1 / CD)`: The minimum per-service SLO that each service in the chain must meet for the compound availability to equal the customer SLA

**Intuition behind the formula:** If compound availability = per_service_SLO^CD, then to find the per-service SLO that produces a given compound availability, you take the CD-th root of the customer SLA. This is the algebraic inverse of compound probability.

### Simple Example

**Journey:** Checkout  
**Customer SLA:** 99.95% (0.9995)  
**CD:** 5

```
Required per-service SLO = 0.9995 ^ (1/5)
                         = 0.9995 ^ 0.2
                         = 0.99990
                         = 99.990%
```

Each service in the checkout chain must maintain 99.990% availability for the compound availability to equal the customer SLA of 99.95%. This is the *minimum bar* — if any service falls below 99.990%, the customer SLA is mathematically broken.

**Sanity check:** Verify the inverse.
```
0.99990^5 = 0.9995 = 99.95% ✓
```

### Simple Example: Browse Journey

**Journey:** Browse  
**Customer SLA:** 99.5% (0.995)  
**CD:** 2

```
Required per-service SLO = 0.995 ^ (1/2)
                         = 0.995 ^ 0.5
                         = 0.99750
                         = 99.75%
```

Browse is a much shallower chain with a more relaxed SLA. Each service only needs to deliver 99.75% availability — substantially easier than the 99.990% required for checkout. This is the correct calibration: services on deeper, more critical chains need tighter SLOs. Services on shallow, lower-stakes chains can have relaxed SLOs without compromising the customer promise.

### Complex Scenario: The Fraud Detection Insertion

**Setup:** The security team has proposed inserting a new "Fraud Detection" microservice into the checkout call chain between Payment and Order Management. The current checkout flow:

```
API Gateway → Auth → Cart → Payment → Order Management
(CD = 5)
```

Proposed flow after insertion:
```
API Gateway → Auth → Cart → Payment → Fraud Detection → Order Management
(CD = 6)
```

The Fraud Detection service is a third-party ML scoring API. Its vendor SLA is 99.97%. The security team believes this is "good enough" and is eager to ship.

**Step 1: What does CD change from 5 to 6 mean for the required per-service SLO?**

Before insertion (CD = 5):
```
Required per-service SLO = 0.9995 ^ (1/5) = 99.990%
```

After insertion (CD = 6):
```
Required per-service SLO = 0.9995 ^ (1/6) = 0.9995 ^ 0.1667
                         = 99.992%
```

Each *existing* service now needs to deliver 99.992% instead of 99.990% — a tighter target that no existing service was designed to meet.

**Step 2: What does the Fraud Detection service actually contribute?**

Fraud Detection's vendor SLA is 99.97%. The required per-service SLO at CD=6 is 99.992%. Fraud Detection misses the requirement by 22 basis points. That means even if every other service performs perfectly at its declared SLO, Fraud Detection alone will break the customer SLA.

**Step 3: What is the new compound availability if Fraud Detection enters at its vendor SLA?**

```
Compound availability = 0.9999 × 0.9999 × 0.9999 × 0.9999 × 0.9997 × 0.9999
                      = (0.9999^5) × 0.9997
                      = 0.99950 × 0.9997
                      = 0.99920
                      = 99.920%
```

Customer SLA: 99.95%. New compound availability: 99.920%. The gap has grown from 33 basis points to 30 basis points against the SLA — but now there is an additional structural deficit because the per-service requirement has tightened for all existing services.

More precisely: the current actual checkout availability is 99.62%. Adding Fraud Detection at 99.97% multiplies that:

```
New actual compound = 0.9962 × 0.9997 = 0.9959 = 99.59%
```

The insertion makes a bad situation measurably worse.

**SRE's recommendation:** Do not insert Fraud Detection as a synchronous hop in the checkout chain at CD=6. Two alternatives:

1. **Async pattern**: Run fraud scoring asynchronously after order creation. The checkout completes synchronously. The order is held in a "pending fraud review" state for 30-60 seconds while the async fraud score arrives. If the score triggers a hold, the order is paused before fulfillment. This keeps CD=5 and removes Fraud Detection from the compound reliability calculation entirely.

2. **Pre-auth pattern**: Run fraud scoring as a parallel call during the Cart validation step (before Payment), using a timeout with a safe default (approve if score service is unavailable). This keeps the synchronous critical path at CD=5 and degrades gracefully under Fraud Detection failure.

This is what CD makes visible: not just a technical count, but an architectural decision gate with quantified consequences.

### Decision Gates

| CD Value | Interpretation | Required Action |
|----------|---------------|-----------------|
| CD ≤ 2   | Shallow chain. Compound risk is minimal. SLO requirements are relaxed. | Monitor standard metrics. No structural concern. |
| CD = 3   | Manageable depth. Compound risk is present but bounded. | Verify per-service SLOs derived from CD formula. Document the chain. |
| CD = 4–5 | Watch closely. Compound risk is significant. Per-service SLO requirements are tight (often 99.99%+). | Mandatory SLO derivation review. Every proposed addition requires SCS and LBH evaluation. |
| CD = 6   | Architectural review required. Tight per-service SLOs become difficult to meet. Each additional hop is a compounding incident risk. | Evaluate async alternatives for any non-critical path calls. Require sign-off from SRE architect before adding services. |
| CD > 6   | High-risk zone. The architecture is structurally fragile. The required per-service SLOs may be impossible to achieve sustainably. | Immediate architectural decomposition or async conversion initiative. Escalate to VP Engineering with compound availability projection. |

### Best Practices

**Do:**
- Count CD during design review, before services are built or integrated
- Recompute CD every time a new service is proposed for an existing chain
- Document the CD of every customer journey in the service catalog
- Use CD as a gate in architecture review: any change that increases CD on a critical journey requires SCS and LBH evaluation before approval
- Consider async conversion as the primary mitigation when CD exceeds 5 on a journey with a tight SLA

**Don't:**
- Count database queries or cache reads as hops — these are internal to the service
- Count async publishes, Kafka writes, or fire-and-forget calls as hops
- Assume that "one more service" is fine without doing the compound math
- Let teams self-certify that their service "won't be in the critical path" without architectural verification

### Guardrails

Set the following architectural gates in your design review process:

| Trigger | Action |
|---------|--------|
| CD increases on a P0 journey (Checkout, Payment) | Mandatory SCS recalculation before approval |
| CD reaches 5 on any journey | Alert to SRE architect for review |
| CD reaches 6 on any journey | Block deployment pending architectural review |
| CD > 6 on any journey | Escalation required to VP Engineering |

These are process gates, not runtime alerts. CD is an architectural property — it changes when the design changes, not when traffic increases.

## SCS — SLO Coherence Score

### The Concept

The SLO Coherence Score is a single number that answers the question: "Given the call depth of this journey and the declared SLOs of the participating services, is the customer SLA mathematically achievable?"

Think of it as a budget audit. Imagine you are a CFO reviewing a project's financial plan. Each team has submitted a spending forecast. You add up all the forecasts and compare them to the total budget. If the sum of forecasts exceeds the budget, the plan is incoherent — it cannot be executed as written, regardless of how confident each team is in their own number. SCS is that audit for reliability.

The target range for SCS is 0.95–1.10. A score in this range means the declared SLOs are coherent with the customer SLA — not too tight (which would indicate over-engineering), and not too loose (which would indicate the SLA will be missed even if all services hit their targets exactly). A score below 0.95 is a structural alarm: even perfect execution cannot achieve the customer SLA. A score above 1.10 means the declared SLOs are more stringent than necessary — the organization is over-investing in reliability relative to what the customer SLA requires.

The SCS is computed entirely from declared values — not from observed metrics. This is intentional. You want to know if the *plan* is coherent before you start measuring performance against it. If the declared SLOs compound to something less than the customer SLA, no amount of operational excellence can close the gap. The architecture must change.

### Formula

```
SCS = (per_service_SLO ^ CD) / customer_SLA

Target range: 0.95 ≤ SCS ≤ 1.10
```

**Components:**

- `per_service_SLO`: The SLO declared by each service (assumed uniform here; we address non-uniform SLOs in the Complex Scenario below)
- `CD`: The call depth of the journey
- `per_service_SLO ^ CD`: The compound availability of the chain if every service meets its declared SLO exactly
- `customer_SLA`: The end-to-end availability target, expressed as a decimal
- `SCS`: The ratio of compound availability to the customer SLA

**Interpreting the ratio:**
- SCS = 1.00: The declared SLOs compound to exactly the customer SLA. No headroom, no over-engineering. Theoretically perfect, but fragile — any deviation from declared SLOs breaks the customer promise.
- SCS = 1.05: The declared SLOs compound to 5% above the customer SLA. Reasonable headroom for operational variance.
- SCS = 0.90: The declared SLOs compound to 10% *below* the customer SLA. The customer SLA is impossible to achieve even with perfect execution.
- SCS = 1.20: The declared SLOs compound to 20% above the customer SLA. Services are over-engineered relative to business requirements.

**For non-uniform SLOs** (different services declaring different SLOs):

```
SCS = (product of all per-service declared SLOs) / customer_SLA
    = (SLO_1 × SLO_2 × SLO_3 × ... × SLO_n) / customer_SLA
```

### Simple Example: Checkout at CD = 5

**All five services declare 99.99% (0.9999).**  
**Customer SLA:** 99.95% (0.9995)

```
SCS = 0.9999^5 / 0.9995
    = 0.99950 / 0.9995
    = 1.0000
```

SCS = 1.00. The declared SLOs compound to exactly the customer SLA. This is coherent but has zero headroom. If any service degrades by even a fraction of a basis point, the customer SLA is broken.

**Practical note:** A target SCS of 1.05 (5% headroom) is healthier than exactly 1.00. It allows for minor operational variance without immediately violating the customer SLA. Achieved by having services declare SLOs slightly above the minimum required — for example, 99.991% instead of 99.990% at CD=5.

### Complex Scenario: The CD = 6 Problem — Fraud Detection Revisited

Continuing the Fraud Detection insertion scenario. The proposed new chain at CD=6 includes:
- Auth: declares 99.990% (0.99990)
- Cart: declares 99.99% (0.9999)
- Payment: declares 99.99% (0.9999)
- Fraud Detection: vendor SLA 99.97% (0.9997)
- Order Management: declares 99.99% (0.9999)

Plus API Gateway (infrastructure, not counted in per-service SLOs).

**Customer SLA:** 99.95% (0.9995)

**Step 1: Compute the compound availability of the proposed chain.**

```
Compound = 0.99990 × 0.9999 × 0.9999 × 0.9997 × 0.9999
         = 0.99990 × 0.9999 × 0.9999 × 0.9997 × 0.9999
```

Working through:
```
0.99990 × 0.9999 = 0.99980
0.99980 × 0.9999 = 0.99970
0.99970 × 0.9997 = 0.99940
0.99940 × 0.9999 = 0.99930
```

Compound availability of declared SLOs = 99.930%.

**Step 2: Compute SCS.**

```
SCS = 0.99930 / 0.9995
    = 0.9988
```

SCS = 0.9988. This is below the floor of 0.95? No — 0.9988 is above 0.95 but the key finding is that the compound availability (99.930%) is *less than* the customer SLA (99.95%). The SCS below 1.0 tells you the same thing: the declared SLOs do not add up to the customer SLA.

**Reading SCS correctly:**
- SCS < 1.00 means the compound of declared SLOs falls short of the customer SLA
- SCS = 0.9988 means the declared SLOs produce a compound availability that is 0.12 percentage points below the customer SLA
- The customer SLA will be missed even if every service hits its declared SLO perfectly

**Step 3: What does this mean concretely?**

| Metric | Value |
|--------|-------|
| Customer SLA | 99.95% |
| Compound availability (declared SLOs) | 99.93% |
| SCS | 0.9988 (below 1.0 — incoherent) |
| Gap | −0.02 percentage points (structural deficit) |
| Required Fraud Detection SLO to restore coherence | 99.992% |
| Fraud Detection actual vendor SLA | 99.97% |
| Deficit | 22 basis points — impossible to close without architectural change |

**Recommendation:** Fraud Detection cannot enter the synchronous checkout chain at its current vendor SLA. SCS quantifies this with precision before a single line of code is written.

### A Second SCS Scenario: Browse Journey Extension

The Browse journey currently has CD=2 (API Gateway → Product Catalog → Search). The team proposes adding a "Personalization" service:

New chain: API Gateway → Product Catalog → Personalization → Search (CD=3)

Both Product Catalog and Search declare 99.99% (0.9999).  
Personalization is a new service — what SLO must it declare?  
Browse customer SLA: 99.5% (0.995).

**Step 1: Required per-service SLO at CD=3.**
```
Required per-service SLO = 0.995 ^ (1/3)
                         = 0.995 ^ 0.333
                         = 0.99833
                         = 99.833%
```

Personalization only needs to declare 99.833% to keep the chain coherent. Product Catalog and Search at 99.99% are already well above this requirement.

**Step 2: Verify SCS if Personalization declares 99.90%.**
```
SCS = (0.9999 × 0.9990 × 0.9999) / 0.995
    = 0.9988 / 0.995
    = 1.0038
```

SCS = 1.0038. Within the target range (0.95–1.10). The declaration of 99.90% for Personalization produces a coherent system with a small positive headroom.

### Decision Gates

| SCS Value | Interpretation | Required Action |
|-----------|---------------|-----------------|
| SCS < 0.90 | Severe structural deficit. The customer SLA is unachievable regardless of operational quality. | Halt deployment. Architectural redesign required. Escalate to VP Engineering. |
| 0.90 ≤ SCS < 0.95 | Significant structural deficit. SLA will be missed under normal operational variance. | Architecture review required. Identify which services must increase their declared SLOs or which calls can be made async. |
| 0.95 ≤ SCS < 1.00 | Marginal deficit. The compound SLA is close to but below target. Small operational variances will break the customer SLA. | Tighten one or more service SLOs before proceeding. No new services may be added to the chain. |
| 1.00 ≤ SCS ≤ 1.10 | Coherent and healthy. The declared SLOs support the customer SLA with operational headroom. | Monitor. This is the target zone. |
| 1.10 < SCS ≤ 1.20 | Slightly over-engineered. Services are more reliable than the customer SLA requires. | Acceptable. Consider whether SLO headroom is intentional (shared chain) or can be relaxed to reduce cost. |
| SCS > 1.20 | Over-engineered. Significant investment in reliability that the customer SLA does not require. | Review per-service SLOs for potential right-sizing. Redirect reliability investment to higher-risk journeys. |

### Best Practices

**Do:**
- Compute SCS at design review for every customer journey before any service is added or modified
- Treat SCS < 1.00 as a hard blocker — a system with SCS < 1.00 structurally cannot deliver on its customer SLA
- Recompute SCS whenever CD changes, whenever a service renegotiates its SLO, or whenever the customer SLA is updated
- Track SCS over time — a healthy system should maintain stable SCS in the 1.00–1.10 range
- Use SCS to right-size SLOs: if SCS > 1.15, some services may be able to safely lower their declared SLOs

**Don't:**
- Treat SCS as a runtime metric — it is an architectural property computed from declared values
- Assume SCS = 1.00 is ideal; always aim for 1.02–1.08 to allow operational headroom
- Compute SCS using observed availability; use declared SLOs (observed availability is handled by other terms in the framework)
- Allow a new service to enter a chain without computing the resulting SCS first

### Guardrails

| Condition | Alert Type | Action |
|-----------|-----------|--------|
| SCS < 0.95 (any journey) | P0 architectural alert | Block deployment pipeline. Mandatory review within 24 hours. |
| SCS < 1.00 (P0 journey — Checkout, Payment) | P1 architectural alert | Architecture review within 1 week. No additional services may be added to the chain. |
| SCS drops below 1.00 due to SLO renegotiation | Ticket | Route to SRE architect for impact assessment. |
| SCS > 1.20 (any journey) | Info | Flag in quarterly reliability review for potential SLO right-sizing. |

## LBH — Latency Budget Per Hop

### The Concept

Latency Budget Per Hop distributes the end-to-end latency SLA across the services in the call chain, giving each service a specific latency target it must meet for the overall p99 to be achievable.

The analogy is project budget allocation. A project has a total budget of $1 million. Five teams will execute the work. If you hand each team a checkbook with no spending limit, you will overspend. If you give each team an equal share of $200,000, each team has a clear constraint and the total is controlled. LBH is that budget allocation — but for milliseconds instead of dollars.

Without LBH, latency targets are set ad hoc. Each service team decides independently what their p99 should be — often based on what they think is achievable, not on what the end-to-end customer experience requires. The result is a checkout that takes 3.2 seconds at p99 even though the customer-facing SLA is 2 seconds. Nobody's service is "broken" — but the sum of all individually "acceptable" latencies exceeds the customer expectation.

LBH makes latency a first-class constraint, derived from the customer SLA and distributed across the chain with precision. It tells every service team: "You have 370ms. Not 500ms. Not 200ms. 370ms. Everything else in the chain needs the rest."

**Why infra overhead matters:**

Not all of the end-to-end latency budget is available for application-level work. Network transit, TLS handshake, DNS resolution, load balancer processing, and Kubernetes kube-proxy overhead consume a portion of the budget before a single line of application code runs. This is the `infra_overhead` term in the formula. It must be subtracted before dividing the remaining budget across application hops.

In a cloud-native environment running on EKS within a single region, infra overhead is typically 100–200ms for a 5-hop chain (roughly 20–40ms per hop for network and service mesh overhead). This must be measured for your specific environment — do not assume zero.

### Formula

```
LBH = (end_to_end_p99 - infra_overhead) / CD
```

**Components:**

- `end_to_end_p99`: The customer-facing latency target at the 99th percentile, in milliseconds. Derived from the customer SLA or product requirement.
- `infra_overhead`: The total latency consumed by infrastructure (network transit, TLS, DNS, load balancer, service mesh), measured empirically. In milliseconds.
- `(end_to_end_p99 - infra_overhead)`: The "application budget" — what remains for services to actually do work
- `CD`: Call depth — the number of hops to divide the application budget across
- `LBH`: Each service's individual p99 latency target, in milliseconds

**Important assumption:** LBH assumes equal allocation across hops. In practice, you may allocate unequally — Payment might get a larger budget because it calls an external payment processor. Chapter 4 covers weighted latency allocation. For structural analysis, equal distribution is the baseline.

### Simple Example: Checkout at CD = 5

**End-to-end p99 target:** 2000ms (2 seconds — a common checkout latency SLA)  
**Infra overhead:** 150ms (measured for this EKS environment)  
**CD:** 5

```
LBH = (2000 - 150) / 5
    = 1850 / 5
    = 370ms
```

Every service in the checkout chain must complete its work — including any database reads, cache lookups, and external API calls — within 370ms at p99. This is the constraint. If Payment needs to call an external payment processor that has a p99 of 600ms, that alone exceeds the budget. The team must either: (a) cache the result, (b) negotiate a faster SLA with the processor, or (c) escalate for a LBH allocation adjustment (which requires compensating by reducing other services' allocations).

**Sanity check:**
```
Service budget: 370ms × 5 = 1850ms
Infra overhead: 150ms
Total: 2000ms ✓
```

### Simple Example: Browse at CD = 2

**End-to-end p99 target:** 800ms  
**Infra overhead:** 60ms (shallower chain, less overhead)  
**CD:** 2

```
LBH = (800 - 60) / 2
    = 740 / 2
    = 370ms
```

Browse has the same per-hop budget as checkout by coincidence — same math, different inputs. Each service on the Browse path has 370ms to serve a catalog response at p99.

### Complex Scenario: Fraud Detection Insertion Changes LBH for All Existing Services

Continuing the Fraud Detection scenario. Before insertion, the checkout LBH was:

```
LBH_before = (2000 - 150) / 5 = 370ms per hop
```

After inserting Fraud Detection, CD becomes 6:

```
LBH_after = (2000 - 150) / 6
           = 1850 / 6
           = 308ms per hop
```

**What this means for existing services:**

Every service that was previously allocated 370ms now has 308ms — a reduction of 62ms. For services like Cart (which reads from a Redis cache and a PostgreSQL database) and Order Management (which writes to multiple tables), a 62ms reduction may require significant re-engineering: query optimization, caching improvements, or async read patterns.

**Comprehensive impact analysis of Fraud Detection insertion:**

| Metric | Before (CD=5) | After (CD=6) | Change |
|--------|--------------|--------------|--------|
| CD | 5 | 6 | +1 |
| Required per-service SLO | 99.990% | 99.992% | Tighter by 2 bps |
| SCS (using declared SLOs) | 1.0000 | 0.9988 | Below 1.0 — incoherent |
| LBH per hop | 370ms | 308ms | −62ms per hop |
| Compound availability (declared) | 99.950% | 99.930% | −20 bps |
| Actual compound availability | ~99.62% | ~99.59% | −3 bps (Fraud Det. at 99.97%) |

**SRE recommendation (restated with LBH data):**

The insertion fails all three structural tests:
1. CD rises to 6 — into the architectural review zone
2. SCS falls to 0.9988 — below 1.00, structurally incoherent with the customer SLA
3. LBH drops by 62ms per hop — requiring re-engineering of 5 existing services to stay within budget

Do not insert synchronously. Proceed with async fraud scoring after order creation, or parallel pre-payment scoring with safe defaults.

### Decision Gates

| LBH Value | Interpretation | Required Action |
|-----------|---------------|-----------------|
| LBH ≥ 500ms | Generous budget. Services have substantial headroom for database I/O, cache misses, and external calls. | Normal development practices. Monitor for creep. |
| 300ms ≤ LBH < 500ms | Reasonable budget. Most services can operate comfortably, but external API calls require careful management. | Document external call timeouts. Cache aggressively where possible. |
| 200ms ≤ LBH < 300ms | Tight budget. Requires active latency management. Cache reads must dominate. Database queries must be indexed and bounded. | Mandatory p99 tracking in dashboards. Latency regression testing required before deploys. |
| 100ms ≤ LBH < 200ms | Very tight. Synchronous external calls are nearly impossible within budget. Database patterns must be highly optimized. | Architecture review. Consider async patterns or caching layers to absorb external latency. |
| LBH < 100ms | Extreme. Essentially limits services to in-memory operations with no external I/O on the critical path. | Escalate. Either the end-to-end p99 target must be relaxed, CD must be reduced, or the service architecture must be fundamentally redesigned. |

### Best Practices

**Do:**
- Measure infra overhead empirically for each environment — do not assume a universal value
- Recompute LBH whenever CD changes or the end-to-end p99 target is renegotiated
- Use LBH as the input to each service team's performance SLO — "your service must respond within 370ms at p99"
- Alert when a service's observed p99 exceeds its LBH for more than 5 minutes (latency budget breach)
- Consider weighted LBH allocation for services with known external dependencies (see Chapter 4)

**Don't:**
- Set per-service latency targets without computing LBH first
- Allow infra overhead to be estimated — measure it with synthetic tracing in the actual environment
- Assume LBH is static — it changes with CD and with updates to the end-to-end p99 target
- Apply a single LBH to all journeys — each journey has its own p99 target and its own CD

### Guardrails

Set the following in your observability platform (Grafana/Prometheus):

| Alert | Threshold | Severity | Action |
|-------|-----------|----------|--------|
| Service p99 exceeds LBH | p99 > LBH for 5+ minutes | Warning | Page on-call. Investigate query performance or cache hit rate. |
| Service p99 exceeds LBH by >50% | p99 > 1.5 × LBH for 5+ minutes | Critical | Immediate incident. This service is blowing the latency budget for all downstream consumers. |
| Service p99 exceeds LBH at >0.1% of requests | Sustained | Ticket | SRE review. This is a slow leak that will manifest as customer-visible latency degradation under load. |

**Prometheus query pattern for LBH breach detection:**

```promql
histogram_quantile(0.99,
  rate(http_request_duration_seconds_bucket{service="payment"}[5m])
) * 1000 > 370
```

Replace `370` with the computed LBH for that service in milliseconds.

## How These Terms Work Together

CD, SCS, and LBH are not independent measurements. They are three angles on the same underlying structure:

**CD tells you the shape.** How many hops? How deep is the chain? What is the architectural risk profile?

**SCS tells you the coherence.** Do the declared SLOs add up? Is the customer promise mathematically achievable, or is the organization committing to something the architecture cannot deliver?

**LBH tells you the budget.** How much latency does each service have? Are the per-service performance targets derived from the customer requirement, or set ad hoc?

The three terms compound in their consequences. When a new service is proposed for a synchronous call chain:
- CD increases by 1
- SCS must be recomputed (it will decrease, potentially below 1.00)
- LBH decreases for every existing service in the chain
- Per-service SLO requirements tighten

A single architectural decision — "let's add one more service" — has ripple effects across all three dimensions. Without this vocabulary, each effect is invisible until a production incident surfaces it. With this vocabulary, every effect is computable from the design document.

**Connecting to later chapters:**

- Chapter 3 introduces MTBI (Mean Time Between Incidents) and MTTR — the operational data that lets you compute *actual* availability versus *declared* SLO, identifying where the SCS gap comes from at the service level
- Chapter 4 introduces Dependency Graph complexity and weighted reliability scoring — building on CD to account for non-uniform service criticality
- Chapter 5 introduces the SLO Drift Metric — tracking the delta between declared and observed availability over time, quantifying SLO theater at scale

The structural terms in this chapter (CD, SCS, LBH) describe the *designed* system. The operational terms in Chapters 3 and 4 describe the *observed* system. The framework is complete only when both views are held simultaneously.

## Chapter Summary

- **Call Depth (CD)** counts the synchronous hops in a customer journey. The required per-service SLO is `customer_SLA^(1/CD)`. Adding one hop to a 5-hop chain at 99.95% SLA tightens the per-service requirement from 99.990% to 99.992% — and reduces LBH for every existing service.

- **SLO Coherence Score (SCS)** is `(compound_declared_SLO) / customer_SLA`. Target range: 0.95–1.10. SCS below 1.00 means the customer SLA is architecturally unachievable. SCS must be computed at design time, not discovered at incident time.

- **Latency Budget Per Hop (LBH)** is `(end_to_end_p99 - infra_overhead) / CD`. It distributes the latency SLA across the chain. Adding one hop shrinks LBH for every existing service — a hidden cost of architectural complexity that LBH makes explicit.

- **The three terms are interdependent.** Any architectural change that affects CD propagates through SCS (coherence) and LBH (latency) simultaneously. Evaluate all three before approving any change to a synchronous call chain.

- **Structural terms are design-time tools.** They are computed from architecture diagrams and declared SLOs, not from runtime metrics. Their value is in preventing problems before code is written, not diagnosing them after incidents occur.

## Check Your Understanding

**Question 1:** A new microservice gateway is being added to the Browse journey. The journey currently passes through 2 services (CD=2). After the addition, the chain will be: API Gateway → Product Catalog → Content Enrichment → Search. The Browse SLA is 99.5% (0.995). Currently, Product Catalog and Search both declare 99.99%. What is the new CD? What SLO must Content Enrichment declare to maintain SCS within the target range? If the end-to-end p99 target is 800ms with 60ms infra overhead, what is the new LBH for each service?

> *Reference calculation:*
> ```
> New CD = 3 (API Gateway not counted; three service hops)
>
> Required per-service SLO = 0.995^(1/3) = 99.833%
>
> SCS check if Content Enrichment declares 99.90%:
>   Compound = 0.9999 × 0.9990 × 0.9999 = 0.9988
>   SCS = 0.9988 / 0.995 = 1.0038 ✓ (within 0.95–1.10)
>
> LBH = (800ms - 60ms) / 3 = 246.7ms ≈ 247ms per hop
> ```
> Content Enrichment must respond within 247ms at p99. Product Catalog and Search, previously allocated (800-60)/2 = 370ms, now each have 247ms — a reduction of 123ms. This may require optimization of existing services.

**Question 2:** The Order Tracking journey (SLA=99.5%, CD=2) is being redesigned to add real-time inventory status. The new chain will have CD=3. What is the required per-service SLO at CD=3? The Inventory Service has a MTBI of 45 days and MTTR of 30 minutes. What is the Inventory Service's MTBI-implied availability, and does it meet the requirement?

> *Reference calculation:*
> ```
> Required per-service SLO at CD=3:
>   = 0.995^(1/3)
>   = 99.833%
>
> Inventory MTBI-implied availability:
>   Downtime per cycle = MTTR = 30 minutes
>   Cycle length = MTBI = 45 days × 1440 minutes/day = 64,800 minutes
>   Unavailability = 30 / 64,800 = 0.000463
>   Availability = 1 - 0.000463 = 99.954%
> ```
> Inventory's MTBI-implied availability is 99.954%, which is well above the required 99.833%. The Inventory Service meets the structural requirement. However, note that its declared SLO and its MTBI-implied availability should be compared — if the declared SLO is lower than 99.954%, the service may be under-declaring. If the declared SLO is higher, SLO theater is present.

**Exercise 2.3 (Advanced):** You are presenting to VP Engineering. The team wants to add 3 new microservices to the checkout journey over the next quarter. Call depth would rise from CD=5 to CD=8. Customer SLA is 99.95% (0.9995). What is the required per-service SLO at CD=8? Is this achievable? What is your recommendation?

> *Reference calculation:*
> ```
> Required per-service SLO at CD=8:
>   = 0.9995^(1/8)
>   = 0.9995^0.125
>   = 0.999938
>   = 99.9938%
>
> Current LBH at CD=5: (2000 - 150) / 5 = 370ms
> New LBH at CD=8:     (2000 - 150) / 8 = 231ms per hop
> LBH reduction: 370ms → 231ms — a 37% reduction for every existing service
> ```
>
> **Recommendation for VP Engineering (Executive Brief format):**
>
> **Bottom line:** Do not approve CD=8 for the checkout journey. The required per-service SLO of 99.9938% is at the edge of what can be reliably achieved at scale, and the LBH reduction to 231ms will require significant re-engineering of existing services.
>
> **Why:** At 99.9938%, services are permitted roughly 32 minutes of downtime per *year*. At that tier, infrastructure incidents (Kubernetes node evictions, EKS control plane maintenance, network blips) alone can consume the entire error budget. This SLO is not impossible, but it requires dedicated reliability engineering, chaos engineering programs, and active investment in every service in the chain — for *three new services* whose operational maturity is unproven.
>
> **Alternative:** Evaluate each of the three proposed services for async conversion. If any of them serve non-blocking functions (notifications, analytics, fraud scoring, recommendation logging), convert them to async event consumers and keep checkout at CD=5 or CD=6. If all three must be synchronous, require each team to demonstrate 99.9938% availability in a staging environment under load before any enters the production checkout chain.
>
> **Risk:** Approving CD=8 without this analysis will result in a checkout compound availability that the current actual performance (~99.62%) makes even worse. The three new services, unless they immediately achieve 99.9938%, will degrade the customer experience further before any operational improvements occur.
