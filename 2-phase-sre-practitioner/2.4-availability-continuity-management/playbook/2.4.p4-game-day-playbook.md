# Game Day Playbook

## Purpose

Game Days inject real failure into production or production-equivalent environments to validate that the system and team actually behave as documented — not just as assumed. Unlike tabletop exercises, Game Days produce empirical evidence: actual RTO contribution times, actual detection speeds, actual blast radius.

This playbook governs Game Day design, authorization, execution, and post-game reporting.

---

## Game Day Philosophy

**Fail small, learn fast.** Start with the smallest failure that meaningfully tests the hypothesis. The goal is a controlled, informative experiment — not a demonstration of how bad things can get.

**Abort criteria are not optional.** If a Game Day causes uncontrolled damage, it was not a Game Day — it was an incident that should have been an incident. Abort criteria defined in advance, and enforced without exception, are what separate controlled chaos engineering from recklessness.

**Run in production or learn nothing useful.** A Game Day that runs in staging tells you about staging. Production systems have different traffic patterns, data volumes, external integrations, and operational context. Staging-only Game Days produce staging-only confidence.

## Authorization Requirements

| Service Tier | Authorization | Advance Notice | Communication Required |
|-------------|---------------|---------------|------------------------|
| Tier 1 | VP of Operations approval + Change Advisory Board notification | Minimum 2 weeks | Customer notification if impact possible; on-call team briefing |
| Tier 2 | Director of Operations approval | Minimum 1 week | On-call team briefing; business unit notification |
| Tier 3 | Engineering lead approval | 48 hours | On-call team briefing |

**Never schedule Game Days during**:
- Peak business hours for customer-facing Tier 1 services
- Periods when the availability budget is within 20% of exhaustion
- The 2 weeks before or after a major release or infrastructure change
- On-call rotation transitions (the incoming on-call should be fully ramped before being tested)

## Pre-Game Readiness Check (Day Before)

Complete this checklist the day before the Game Day. If any item is not green, postpone.

**Team readiness**
- [ ] All critical on-call engineers are confirmed available and briefed on the exercise
- [ ] DR Activation Lead is confirmed and available
- [ ] Communications Lead is confirmed and available (for Tier 1 exercises)
- [ ] Abort authority is identified and reachable (person who can call abort)
- [ ] Observing stakeholders have been briefed on what they will see

**Environment readiness**
- [ ] Monitoring dashboards are confirmed working and displaying correct metrics
- [ ] Alerting is confirmed working (send a test alert and confirm receipt)
- [ ] All runbooks are accessible and current version confirmed
- [ ] Recovery procedure is documented step-by-step and has been reviewed
- [ ] Rollback procedure is documented and confirmed executable
- [ ] DR environment (if applicable) is confirmed healthy

**Coordination readiness**
- [ ] War room bridge established and confirmed functional
- [ ] Customer support team briefed (for Tier 1: advise to expect possible elevated contact volume)
- [ ] Change management notification filed per [Change Management] process
- [ ] External monitoring probes confirmed active (to detect user-visible impact)
- [ ] Comms templates ready (in case impact is greater than expected)

**Abort criteria confirmed and agreed by all participants**
- [ ] Abort criteria are written down and acknowledged by abort authority
- [ ] Everyone knows how to call abort and what happens when abort is called

## Hypothesis Definition

Every Game Day begins with a testable hypothesis. Without a hypothesis, the exercise has no pass/fail criteria and produces no actionable findings.

**Hypothesis format**:
```
If [failure condition], then [expected system behavior] will occur within [expected time].
Evidence: [what we will observe in monitoring to confirm the hypothesis].
```

Examples:
```
Hypothesis 1:
If we terminate all pods in the Notification API deployment in us-east-1a,
then K8s pod anti-affinity rules will reschedule pods to us-east-1b and us-east-1c 
within 90 seconds with no customer-visible error rate increase.
Evidence: Pod count recovers to N on new nodes; error rate remains < 0.1% in Datadog.

Hypothesis 2:
If we simulate a database primary failure by revoking write access,
then the database HA failover will promote the replica and reconnect within 3 minutes,
and the Notification API will resume processing within 5 minutes of reconnection.
Evidence: Write errors clear in Datadog; message throughput returns to baseline;
replica lag monitor shows 0 on promoted primary.
```

Write hypotheses before the Game Day. The gap between what you hypothesize and what you observe is where the learning is.

## Failure Injection Catalog

### Kubernetes Failures

| Failure | Injection Method | What It Tests |
|---------|-----------------|---------------|
| Node failure | AWS: terminate EC2 instance. K8s: `kubectl drain node <name> --force` | Pod rescheduling, PDB behavior, autoscaler replacement |
| Pod kill | `kubectl delete pod <name>` (or all pods in deployment) | Restart policy, liveness probe behavior |
| AZ loss simulation | Remove subnet from load balancer target group; restrict security group to deny traffic from AZ | Multi-AZ failover, pod redistribution |
| Container CPU saturation | Run stress tool in container (`kubectl exec` + `stress-ng`) | HPA response, throttling behavior |
| Container memory saturation | Inject memory consumer | OOMKill behavior, restart behavior |
| Network policy restriction | Apply NetworkPolicy that blocks inter-service communication | Circuit breaker behavior, error propagation |

### Database Failures

| Failure | Injection Method | What It Tests |
|---------|-----------------|---------------|
| Read replica lag spike | Temporarily pause replication; resume | Application behavior under stale read data |
| Connection exhaustion | Open max connections from test process | Connection pool behavior, queue depth |
| Slow query injection | Execute long-running query on shared table | Lock contention, timeout behavior |
| Primary failure simulation | RDS: promote read replica manually | Failover automation, application reconnection |

### Network Failures

| Failure | Injection Method | What It Tests |
|---------|-----------------|---------------|
| Latency injection | `tc netem delay 200ms 50ms` on target host | Timeout handling, retry behavior |
| Packet loss | `tc netem loss 10%` on target host | Retry storms, circuit breaker activation |
| DNS timeout | Modify /etc/resolv.conf timeout settings temporarily | DNS fallback behavior |
| Load balancer target removal | Deregister instance from target group | Health check deregistration behavior |

### Application Failures

| Failure | Injection Method | What It Tests |
|---------|-----------------|---------------|
| Dependency timeout | Introduce artificial delay in dependency response | Client timeout handling, fallback path |
| Error injection | Inject 500 responses from a dependency (using a proxy like Toxiproxy) | Circuit breaker, retry, error propagation |
| Credentials expiry | Rotate a credential without updating the application | Secret management, error handling |

## Execution Protocol

### Minute 0: Confirm Go/No-Go

Abort authority explicitly states "Go" or "No-Go" based on readiness check status. If No-Go, postpone with documented reason.

### Minute 0–5: Baseline Capture

Before injecting any failure, capture current state:
- Error rate (baseline)
- P50, P95, P99 latency (baseline)
- Message throughput (baseline)
- Active pod/instance count (baseline)

Record all baseline values. These are the "before" against which you measure "after."

### Minute 5+: Failure Injection

Inject the failure using the documented method. Record exact timestamp. Call out verbally to the war room: "Failure injected at [timestamp]."

### During Injection: Observation

Observer role actively monitors:
- How long until automated monitoring detects the failure?
- How long until the first alert fires?
- How long until automated recovery activates (if applicable)?
- What is the customer-visible error rate?
- Is the blast radius within the planned boundary?

If blast radius exceeds planned boundary at any point → **ABORT**. See abort procedure.

### Recovery Observation

After injection, either:
- **Automated recovery occurs** — observe and measure time; do not intervene
- **Automated recovery does not occur** — execute the manual recovery runbook; measure time per step

Compare actual behavior against the hypothesis. Record every deviation.

### Failure Termination

At the pre-defined exercise time limit (or when the hypothesis is answered), end the failure injection:
- Restore the injected failure (restart terminated instances, restore network rules, etc.)
- Confirm systems return to baseline
- Monitor for 15 minutes post-restoration

---

## Abort Procedure

If abort criteria are met at any point:

1. **Abort authority calls "ABORT"** — verbally and in the war room channel
2. Stop all failure injection immediately
3. Execute the pre-documented rollback procedure
4. Confirm systems are recovering
5. Keep war room open until all systems return to baseline + 30 minutes
6. If rollback does not resolve the issue within 15 minutes → transition to incident mode via [Incident Management]
7. Document: what triggered abort, at what time, in what state the system was at abort

Abort is not failure. Abort is the exercise working correctly.

## Post-Game Report (Publish Within 5 Business Days)

### Report Sections

**1. Exercise summary**
- Hypotheses tested
- Failure type and injection method
- Duration of exercise
- Who participated

**2. Hypothesis results**

| Hypothesis | Expected Behavior | Actual Behavior | Result | Gap |
|-----------|-------------------|-----------------|--------|-----|
| Pod failure recovery | 90 seconds | 2 minutes 40 seconds | Failed | Anti-affinity constraint prevented scheduling in one AZ; new node provision required |
| Database failover | 3 min failover + 5 min reconnect | 4 min failover + 12 min reconnect | Failed | Application retry timeout too short; reconnection failed first attempt |

**3. Metrics observed**

| Metric | Baseline | During Failure | Recovery Time | Target |
|--------|----------|---------------|---------------|--------|
| Error rate | 0.02% | 8.4% (peak) | 2:40 min | < 1 min |
| P99 Latency | 120ms | 3400ms (peak) | 2:10 min | < 90 sec |
| Message throughput | 10,000/min | 0 (during failure) | 3:15 min | < 2 min |

**4. Findings and action items**
(Same format as tabletop exercise findings — owner, due date, verification method)

**5. Blast radius assessment**
- Did failure stay within the planned boundary?
- Were any unexpected services affected?
- Was customer-visible impact within the planned tolerance?

**6. Next Game Day recommendation**
- What scenario to test next (based on what was found)
- When (suggested timing given current availability budget)
- What must be fixed before the next exercise (prerequisites)
