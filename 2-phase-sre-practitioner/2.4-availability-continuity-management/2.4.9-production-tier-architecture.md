# Chapter 09 — Production Tier Architecture

> **Audience**: Director/VP of Operations candidates. This chapter covers how to design the infrastructure investment allocation model for a production environment — what tier classification is, how to govern it, and what each tier's infrastructure must look like.

---

## Why Tier Classification Exists

The instinct of engineering teams is to treat all production applications as equally important. This instinct is organizationally sensible (nobody wants their application called "not critical") and operationally wrong. Building 99.99% infrastructure for a batch reporting job that runs nightly is expensive overengineering. Running the billing transaction engine on 99.5% infrastructure is a business-continuity risk.

Tier classification is the governance mechanism that aligns infrastructure investment with business criticality. It answers the question: "How much downtime per year is this application worth purchasing resilience against?" That is a business question with a cost answer. The VP of Operations owns the answer; the architecture and engineering teams build to it.

At T-Mobile scale, the stakes are concrete. E911 routing goes down for 10 minutes and you have a federal regulatory incident. The employee-facing shift scheduling tool goes down for 4 hours and you have 200 managers using spreadsheets temporarily. These are not the same problem and they do not deserve the same infrastructure investment.

## Section 1: The Four-Tier Model

### Tier 0 — Mission Critical (99.999% — ~5 minutes/year)

**What qualifies**: real-time life-safety or financial finality applications where downtime has immediate, irreversible consequences. At T-Mobile: E911 routing, number portability database (NPAC), core network routing infrastructure. In banking: clearing and settlement systems. In insurance: fraud detection systems that block transactions.

The test for Tier 0 qualification: "If this is down for 10 minutes, does someone get hurt, does a federal regulator call us, or does the core network stop functioning?" If any answer is yes, you are in Tier 0 territory.

**Characteristics**:
- No scheduled maintenance downtime — upgrades are performed with zero-downtime techniques only (active-active failover, rolling upgrades with continuous traffic)
- Zero-RPO replication — data written to the primary is synchronously committed to all replicas before the write acknowledgment is returned to the caller. Any write that does not satisfy this contract is rejected.
- Sub-second automatic failover — the application layer must not experience a timeout on the scale that a human could observe
- No shared components with any other tier — no shared load balancers, no shared database clusters, no shared message brokers. Tier 0 infrastructure is physically and logically isolated.

**Infrastructure requirements**:
- Active-active multi-region: live traffic runs in two or more regions simultaneously. Not active-passive with fast failover — active-active with traffic split
- Synchronous database replication within the region, asynchronous with strong consistency guarantees to the secondary region
- Dedicated hardware with no multi-tenancy (no noisy neighbor risk)
- Dedicated network paths with BGP multi-homing and sub-second convergence
- Custom SLA agreements with cloud provider (if cloud-hosted) or dedicated hardware maintenance contracts

**Cost**: 5–10× Tier 1 infrastructure cost. At an organization with 500 production applications, only 3–5 should qualify as Tier 0. If you have 50 applications claiming Tier 0, your tier classification governance is broken.

**Examples at T-Mobile scale**: E911 routing gateway, number portability query engine, core IMS (IP Multimedia Subsystem) routing, billing finality engine (the component that marks a transaction as settled — not the full billing stack, just the finality component).

### Tier 1 — Business Critical (99.99% — ~52 minutes/year)

**What qualifies**: customer-facing applications that directly generate revenue or that are covered by contractual SLA commitments. If a customer can observe the application is down, and the downtime results in revenue loss or contractual liability, it is Tier 1.

At T-Mobile: customer self-service portal (myT-Mobile), activation and provisioning API, customer care platform (what agents use to serve customers), roaming gateway APIs.

**Characteristics**:
- Active-active within region: traffic distributes across multiple pods/instances across multiple AZs
- Cross-region warm standby: a second region has the application deployed and healthy, receiving asynchronous replication, able to serve traffic within 30 seconds of a regional failover decision
- Automatic failover within the region: a pod failure is handled by Kubernetes scheduling within seconds. An AZ failure is handled by load balancer health checks routing around the failed AZ within 30 seconds. A regional failure requires a manual or automated region failover decision — target < 5 minutes.
- RPO < 1 minute: at most 1 minute of data loss on a regional failure, assuming synchronous intra-region replication and async cross-region replication at sub-60-second lag

**Infrastructure requirements in detail**:

*Compute*: N+2 redundancy. If the SLO requires 10 pods to handle peak load, deploy 12. The environment can lose 2 pods (maintenance, failure) and still meet the SLO. Pods distributed across a minimum of 3 AZs with anti-affinity rules preventing co-location of pods on the same node.

*Load balancing*: active-active load balancers (two or more LBs with health check cross-monitoring). A single load balancer as the entry point is a single point of failure for a Tier 1 application — this is a common architectural gap found in audits of "Tier 1" applications that are actually designed as Tier 2.

*Database*: synchronous primary-standby replication within the AZ (or within the region for cross-AZ synchronous). Asynchronous read replica in the second AZ for DR reads. For RDS: Multi-AZ deployment with automated failover. For Aurora: Aurora Global Database with primary region and replica region. Failover time target: < 30 seconds for within-region standby promotion.

*Middleware / Message Brokers*: 
- RabbitMQ: cluster of minimum 3 nodes with quorum queues (not classic mirrored queues — quorum queues have stronger consistency guarantees). No single broker node should hold the only copy of a critical queue.
- Kafka: ISR (in-sync replicas) = 3 minimum. `acks=all` producer configuration. `min.insync.replicas=2`. A Tier 1 Kafka topic should tolerate one broker node failure with zero message loss.
- MuleSoft: clustered runtime with shared persistent object store, load balanced at the API gateway layer

*Network*: dual ISP connections at the data center or regional internet gateway level. BGP multi-homing ensures that an ISP outage fails over to the secondary ISP within BGP convergence time (typically 30–90 seconds). Dedicated network paths for Tier 1 east-west traffic — Tier 1 services should not compete with batch processing traffic for network bandwidth.

*Storage*: RAID-10 minimum for any block storage. SAN with dual-path connectivity (two independent HBAs, two independent SAN fabrics) for physical infrastructure. For cloud storage: S3 or equivalent with cross-region replication enabled.

*Monitoring*: synthetic monitoring from multiple external probes every 60 seconds. External probes run from outside the cloud provider's network (not from within the same VPC) to detect network-layer failures that internal health checks would miss. Dedicated observability stack — Tier 1 observability should not share infrastructure with the observability stack used for Tier 3 applications (a Tier 3 logging problem should not impact Tier 1 alerting).

*On-call*: 24/7 dedicated on-call with ≤ 5-minute page-to-acknowledge SLA. The 5-minute SLA is not aspirational — it is enforced. On-call engineers who repeatedly miss the SLA are rotated off the Tier 1 rotation.

**Change governance**: 
- All Tier 1 changes require either Emergency Change (for break-fix) or Normal Change with VP approval
- No changes during business hours (8am–8pm local) without explicit VP Operations approval
- Changes are always reviewed by a second engineer before execution
- Blue-green deployment only — no rolling deployments for Tier 1. Rolling deployments introduce a window where both old and new versions are simultaneously serving traffic, which can produce inconsistent behavior for stateful operations (session state, in-flight transactions).

**Deployment strategy**: blue-green. Maintain two identical environments (blue and green). Active traffic runs on blue. Deploy to green. Run smoke tests against green. When green passes: flip traffic at the load balancer level. Blue becomes the rollback target. Keep blue running for 4 hours before decommissioning — this is your rollback window.

### Tier 2 — Important (99.9% — ~8.7 hours/year)

**What qualifies**: applications that support business operations but are not directly customer-facing or revenue-generating. Internal portals, operational dashboards, partner APIs with non-critical SLAs, second-tier business processes.

**Characteristics**:
- Active-passive with hot standby: the standby instance is running and healthy but not serving traffic. Failover is automatic and targets < 5 minutes.
- RPO < 15 minutes: asynchronous replication with up to 15 minutes of acceptable data loss on a failure
- Multi-AZ preferred but not required for all components

**Infrastructure requirements**:
- Compute: N+1 redundancy. Lose 1 node and still meet SLO. Two AZs minimum.
- Database: asynchronous replication to a warm standby. Failover time < 5 minutes.
- Middleware: clustered within the region. Single-region is acceptable for Tier 2.
- On-call: 24/7 shared on-call (not dedicated to Tier 2), ≤ 15-minute acknowledge SLA
- Monitoring: internal synthetic monitoring every 5 minutes. External probes optional.

**Deployment**: rolling deployment acceptable with automated rollback trigger. If the rolling deployment causes error rate to spike above threshold, automatic rollback executes without human intervention.

**Change governance**: Normal Change process, CAB approval required. Changes can be made during business hours with change record approval. 24-hour change freeze before quarter-end or major business events.

### Tier 3 — Standard (99.5% — ~43 hours/year)

**What qualifies**: internal tools, reporting systems, batch processing applications, non-customer-facing developer tools, administrative UIs.

**Characteristics**:
- Single instance with application-level HA (application restarts automatically on failure, no infrastructure-level standby)
- RTO < 4 hours: the team must be able to restore service within 4 hours of a failure
- RPO < 4 hours: up to 4 hours of data loss is acceptable (daily backup minimum)

**Infrastructure**:
- Single AZ is acceptable
- Daily backup (automated), tested monthly
- No standby instance required
- On-call: business hours only. After-hours pages are P1 only (total outage). P2 and below wait for business hours.

**Deployment**: rolling or recreate strategy acceptable. No pre-approved change required for standard deployments; engineering team discretion.

## Section 2: N-Tier Infrastructure Design

### What N-Tier Means in Practice

N-tier in the production infrastructure context means separating concerns across layers so that failure at one layer does not cascade through adjacent layers by design, not by hope.

The standard four-tier infrastructure design for a Tier 1 application:

```
┌─────────────────────────────────────────────────────────┐
│  TIER 4: PRESENTATION                                    │
│  CDN (CloudFront/Akamai) + WAF (web application         │
│  firewall) + External Load Balancer                      │
│  → Caches static content, absorbs DDoS, terminates TLS  │
└────────────────────────┬────────────────────────────────┘
                         │ HTTPS (443), rate-limited
┌────────────────────────▼────────────────────────────────┐
│  TIER 3: APPLICATION                                     │
│  Kubernetes pods, horizontal pod autoscaler,             │
│  service mesh (Istio/Linkerd), internal load balancer    │
│  → Executes business logic, scales with load             │
└────────────────────────┬────────────────────────────────┘
                         │ Internal protocol (gRPC/HTTP)
                         │ + message queue events (async)
┌────────────────────────▼────────────────────────────────┐
│  TIER 2: MIDDLEWARE                                      │
│  Message brokers (RabbitMQ/Kafka), API gateway           │
│  (Kong/Apigee), integration services (MuleSoft),         │
│  cache (Redis/ElastiCache)                               │
│  → Decouples services, manages async flows, caches       │
└────────────────────────┬────────────────────────────────┘
                         │ JDBC/ODBC + proprietary protocols
┌────────────────────────▼────────────────────────────────┐
│  TIER 1: DATA                                            │
│  Databases (RDS/Aurora), persistent storage (EBS/EFS),   │
│  data cache (Redis primary), blob storage (S3)           │
│  → Source of truth for all persistent state              │
└─────────────────────────────────────────────────────────┘
```

### Tier Isolation Rules

Each tier has its own network boundary — subnet or security group — that enforces which traffic can enter:

- Presentation tier accepts traffic from the internet (port 443) and returns responses. No other inbound traffic.
- Application tier accepts traffic only from the presentation tier (load balancer IP range) and from other application-tier services (service mesh east-west). No direct inbound from the internet.
- Middleware tier accepts traffic only from the application tier. No direct internet access, no direct presentation-tier access.
- Data tier accepts traffic only from the application tier and middleware tier. Never from the presentation tier directly. This is a common misconfiguration — a frontend service calling the database directly because "it was faster" — and it is a Tier 1 security violation.

**Independence of scaling**: each tier scales without requiring changes to adjacent tiers. Application tier HPA triggers pod additions without any action in the middleware or data tier. Middleware tier scales its broker nodes independently. This independence requires that inter-tier interfaces are stable and documented.

### Failsafe Patterns Per Tier

**Presentation tier degradation**: CDN serves cached content when the application tier is degraded. The CDN must be configured with appropriate cache TTLs and stale-while-revalidate behavior. A customer hitting the CDN during an application tier outage sees content from the last cache refresh — not an error page. This requires that the presentation tier be designed for graceful degradation, not just availability.

**Application tier**: circuit breakers (Hystrix/Resilience4j or service mesh circuit breaker configuration) prevent cascade failures from an unhealthy middleware or data tier service from consuming all application tier threads. If the database is slow, the circuit breaker opens and the application tier returns fast errors rather than accumulating thread backlog until OOM.

**Middleware tier**: dead letter queues (DLQs) for both RabbitMQ and Kafka prevent message loss when consumers are down. A consumer that is temporarily unavailable does not cause message loss — messages accumulate in the queue with TTL policies that ensure they are not held indefinitely. Consumer lag is operationally acceptable and recoverable; message loss is not.

**Data tier**: read replicas absorb read traffic when the primary is recovering from failure. This requires that the application tier be designed to route read queries to the replica connection string, not the primary — a design decision that must be made before the system is built, not during a production incident.

## Section 3: Redundancy Patterns by Component

### Load Balancers

| Component | Redundancy Pattern | Failover Mechanism | Failure Mode if Redundancy Fails |
|-----------|------------------|-------------------|---------------------------------|
| External LB (Tier 1) | Active-active pair with ECMP routing | BGP withdrawal routes traffic to surviving LB | Complete loss of external access |
| External LB (Tier 2) | Active-passive with VIP failover | VRRP/HSRP, failover in 2–3 seconds | 2–3 second outage during failover |
| Internal LB | Kubernetes Service (kube-proxy/IPVS) | Pod endpoint list updated as pods fail | Request to failed pod returns ECONNREFUSED, client retries |
| ALB/NLB (cloud) | Managed redundancy by cloud provider | Transparent | Cloud provider regional failure |

### Application Nodes (Kubernetes)

- **Pod redundancy**: minimum 3 replicas for Tier 1 workloads. Pod Disruption Budget configured to allow at most 1 pod unavailable at any time during voluntary disruptions (rolling deployments, node drains).
- **Node redundancy**: worker nodes distributed across 3 AZs. Pod anti-affinity rules prevent all replicas from scheduling on the same node or in the same AZ.
- **Node failure**: Kubernetes reschedules failed pods to healthy nodes within 60 seconds (default pod eviction timeout — consider reducing to 30 seconds for Tier 1 workloads via `node.kubernetes.io/unreachable` toleration).
- **Cluster failure**: for Tier 0 and Tier 1, a second cluster in a second region must be ready to serve traffic. This requires externally managed DNS with health-check-based failover (Route 53 health checks with latency routing, for example).

### Message Brokers

**RabbitMQ cluster**:
- Minimum 3 nodes for quorum queue support (quorum queues require odd number of nodes and majority quorum)
- Quorum queues replicate to all nodes in the cluster — a message is considered committed when a majority of nodes acknowledge the write
- Node failure: if 2 of 3 nodes are running, quorum is maintained and the cluster continues to operate
- Network partition handling: configure `cluster_partition_handling = pause_minority` — the minority partition pauses rather than continuing to accept writes on both sides, preventing split-brain

**Kafka cluster**:
- Minimum 3 broker nodes for ISR=3 topics
- `min.insync.replicas=2` means at least 2 replicas must acknowledge a write before the producer's `acks=all` commitment is satisfied
- Broker failure: with 3 brokers and ISR=2, the cluster tolerates 1 broker failure while maintaining write availability for topics with replication factor 3
- Controller failure: KRaft mode (Kafka 3.3+) eliminates ZooKeeper dependency — controller re-election happens within seconds

### Database (Primary-Standby)

| Scenario | Failover Mechanism | Failover Time | Data Loss Risk |
|----------|------------------|--------------|---------------|
| Primary pod failure (cloud DB) | Automated standby promotion | 30–60 seconds (RDS Multi-AZ) | Zero with synchronous replication |
| AZ failure | LB routes to standby in surviving AZ | 30–60 seconds | Zero with synchronous replication |
| Region failure | Manual or automated region failover | 5–30 minutes | Bounded by async replication lag (typically < 1 minute) |
| Standby failure | Primary continues; standby rebuild begins | Immediate (primary unaffected) | Risk increases — no standby while rebuild runs |

### Cache (Redis)

**Redis Sentinel** (single region, Tier 2 pattern):
- 1 primary + 2 replicas + 3 Sentinel nodes
- Sentinel quorum detects primary failure and promotes a replica
- Failover time: 10–30 seconds
- Data loss on failover: commands acknowledged since the last async replication sync (typically < 1 second)

**Redis Cluster** (Tier 1 pattern):
- 6 nodes minimum (3 primary shards + 3 replicas, one replica per shard)
- Sharding distributes data and load across primary nodes
- A primary shard failure triggers automatic promotion of its replica
- No single point of failure — losing one primary + its replica would cause a cluster outage (2 simultaneous failures)

### API Gateway

- Tier 1: deploy the API gateway as a cluster with 3+ nodes behind a load balancer. No single gateway node handles 100% of traffic.
- Circuit breakers at the gateway level (not just at the application level): the gateway can shed load and return 503 before the condition cascades to the application tier
- Configuration replication: gateway configuration must be stored in a distributed, replicated data store — a gateway node that restarts should come back with the same routes and policies as its peers, not require manual reconfiguration

### CDN and DNS

**CDN**: CDN is inherently multi-node and geographically distributed. The resilience concern is not node failure — it is origin failover. The CDN must be configured to: (a) failover to an alternative origin if the primary origin is unhealthy, and (b) serve stale cached content with a configurable grace period during origin unavailability. Both behaviors require explicit CDN configuration — they are not defaults.

**DNS**: Route 53 (or equivalent) with health-check-based routing. Tier 1 application DNS records must have health checks attached. If the primary endpoint fails its health check, DNS automatically returns the IP of the secondary endpoint. TTL must be low enough to allow timely failover — 60 seconds is the practical minimum for operational DNS TTLs.

## Section 4: Tier Classification Governance

### Who Assigns Tiers

Tier classification is not self-service. The **Architecture Review Board (ARB)** conducts the initial classification, and the **VP of Operations** approves it. The ARB owns the classification criteria and applies them objectively — their job is to say "no" to tier inflation.

Classification is reviewed annually and triggered by:
- A new revenue dependency is established for an existing application
- A customer SLA is contractually committed against the application
- A regulatory requirement mandates specific availability (FCC requirements for E911-adjacent services)
- A post-incident learning identifies that the application was under-tiered (it caused more business impact than its tier suggested)

### The Tier Inflation Problem

Tier inflation is the single most common governance failure in enterprise tier classification. Engineering teams over-classify their applications as Tier 1 to receive increased infrastructure investment. The pressure is structural: a team whose application is Tier 2 gets shared on-call, shared infrastructure, and longer change windows. The same team re-classified at Tier 1 gets dedicated on-call, dedicated infrastructure, and faster change support. The incentive to inflate is real.

The governance gate against tier inflation is **objective classification criteria**:

| Criterion | Tier 0 | Tier 1 | Tier 2 | Tier 3 |
|-----------|--------|--------|--------|--------|
| Customer visibility | Directly serves customers in life-safety scenario | Directly serves customers; customer notices if down | Customers may eventually notice if down > 4 hours | Customers not affected |
| Revenue impact | Regulatory violation | Revenue loss begins within 15 minutes | Revenue impact after 4+ hours | No revenue impact |
| Contractual SLA | Federal/regulatory mandate | Customer SLA commitment | Internal SLA | No SLA |
| Regulatory obligation | Federal, immediate | State or federal, non-immediate | Internal compliance | None |

A team claiming Tier 1 must satisfy at least 2 of the 4 criteria. Single-criterion Tier 1 claims require VP Operations review. Teams that cannot produce evidence for any criterion default to Tier 2.

### Tier-to-Cost Mapping

The infrastructure cost multiplier is the mechanism that makes over-classification expensive. If the base infrastructure unit cost for a Tier 3 application is $10K/month:

| Tier | Multiplier | Monthly Cost | Annual Cost |
|------|-----------|-------------|-------------|
| 0 | 10× | $100K | $1.2M |
| 1 | 3× | $30K | $360K |
| 2 | 1.5× | $15K | $180K |
| 3 | 1× | $10K | $120K |

These multipliers are charged back to the owning business unit. When a VP of Engineering is asked to sign a budget line for Tier 1 vs. Tier 2 infrastructure for their application, the classification discussion becomes concrete.

### The Application Tier Registry

The registry is the authoritative record of production application classification. Every production application must have an entry. Entries without a classification are automatically treated as Tier 3 (the governance incentive to complete the classification).

**Required fields per entry**:
- Application name and CMDB ID
- Business owner (name + title)
- Technical owner (Engineering Manager)
- Current tier assignment
- Tier assignment date and ARB approval reference
- SLO commitments (availability %, RTO, RPO)
- Infrastructure design evidence (link to architecture diagram, link to IaC repository)
- Next review date (annual, or triggered by change events)
- Classification rationale (which criteria qualify this application for its tier)

### Board Reporting on Tier Management

Quarterly reporting to the Operations leadership team includes:

1. **Tier distribution**: how many applications in each tier, trend over time. An organization with a growing Tier 1 count without growing revenue is experiencing tier inflation.

2. **Tier 1 SLO actuals**: for each Tier 1 application, actual availability vs. committed SLO for the quarter. Applications that cannot sustain their Tier 1 SLO are either misconfigured (infrastructure fix required) or over-tiered (reclassification discussion required).

3. **Tier classification change log**: applications reclassified this quarter, reason, ARB decision, cost impact.

4. **Infrastructure cost by tier**: total spend per tier, % of operations budget. If Tier 3 applications are consuming 40% of the operations budget, the cost multiplier enforcement has broken down.

5. **Incidents by tier**: P1 and P2 incident counts per tier per quarter. Tier 1 applications with repeated P1 incidents are candidates for architecture review.

## Tier Model: Quick Reference

```
          Business Impact
High ─────────────────────────────── Low
  │                                    │
  │   T0        T1        T2        T3 │
  │  99.999%  99.99%   99.9%    99.5% │
  │  5min/yr  52min/yr  8.7hr/yr 43hr/yr│
  │                                    │
Low ─────────────────────────────── High
          Acceptable Downtime
```

The VP of Operations signs off on every Tier 0 and Tier 1 classification. The ARB signs off on Tier 2. Tier 3 is self-declared with team manager sign-off and annual ARB review.
