# DR Strategy Comparison Reference

A decision tool for matching DR strategy to service requirements. Use this in investment cases, architecture reviews, and board presentations.

---

## Strategy Comparison Matrix

| Dimension | Backup & Restore | Pilot Light | Warm Standby | Hot Standby | Multi-Site Active-Active |
|-----------|-----------------|------------|--------------|-------------|--------------------------|
| **RTO Range** | 8–72 hours | 1–4 hours | 15–60 minutes | 5–15 minutes | < 60 seconds |
| **RPO Range** | 1–24 hours | 15–60 minutes | 5–30 minutes | 0–5 minutes | 0–60 seconds |
| **Infrastructure Cost Multiplier** | 1.0x (storage only) | 1.1x–1.2x | 1.25x–1.5x | 1.5x–2.0x | 2.0x–3.0x |
| **Operational Cost Multiplier** | 1.0x | 1.1x | 1.2x | 1.4x | 1.6x–2.0x |

| **Testing Complexity** | Low | Medium | High | High | Very High |
| **Data Consistency Risk** | Low (backup is snapshot) | Medium (replication lag) | Medium | Low (near-sync) | High (split-brain) |

| **Minimum Viable Availability** | 99%–99.9% | 99.9% | 99.95% | 99.99% | 99.999% |

## Strategy Profiles

### Strategy 1: Backup and Restore

**Description**: Data is backed up on a schedule (daily, hourly, or transaction log backup). In a DR event, infrastructure is rebuilt and data is restored from backup.

**How failover works**: Manual. DR engineer provisions environment (via Terraform/CloudFormation), restores data from backup, reconnects integrations, validates, and redirects traffic.

**How traffic redirects**: Manual DNS update or load balancer reconfiguration after validation.

**Data consistency**: High risk of data loss equal to the backup interval. If daily backup at 2am and failure at 6pm, 16 hours of data may be lost.

**Cost breakdown**:
- Infrastructure: Backup storage only (S3, Azure Blob, tape)
- Operations: Backup monitoring, restoration testing (quarterly minimum)
- Engineering: Low — no standby to maintain

**When to use**:
- Tier 3 services (internal tools, reporting, dev/test environments)
- Services where 24-hour RTO and multi-hour RPO are acceptable
- Services with low transaction volume (restoration is fast)

**When not to use**:
- Any service with customer-facing SLA tighter than 24 hours
- Services where data loss of more than 1 hour is unacceptable
- Services that cannot rebuild cleanly from backup (complex dependency chains)

**Key failure modes**:
- Backup corruption discovered during restoration (backup was corrupt but was never restored from)
- Backup storage inaccessible (credentials expired, access policy changed, storage region unavailable)
- Infrastructure provisioning fails (IaC is outdated, region quota insufficient)
- Restoration time significantly underestimated (large database sizes, slow network)

**AWS implementation**: AWS Backup + S3 Cross-Region Replication. Restore to new EC2 instances via AMI + data restore from S3. Automate with Systems Manager automation documents.

### Strategy 2: Pilot Light

**Description**: Critical data infrastructure (database, core data) is replicated continuously to the DR site. Application infrastructure is not running — it is provisioned on demand when DR is declared.

**How failover works**: Declare DR → provision application tier in DR region (Terraform/CDK) → configure applications to connect to DR database replica → promote replica to primary → validate → redirect traffic.

**How traffic redirects**: DNS failover (Route 53 health check triggers) or manual DNS update.

**Data consistency**: Database replica is continuously updated (near-real-time for most configurations). RPO depends on replication lag at time of failure, typically 15–60 minutes for cross-region asynchronous.

**Cost breakdown**:
- Infrastructure: Database replication (cross-region read replica) + minimal network costs
- Operations: Monitoring replication health, testing provisioning automation
- Engineering: Moderate — IaC must be maintained and regularly tested

**When to use**:
- Tier 2 services where 1–4 hour RTO is acceptable
- Cloud-native services where provisioning is fully automated
- Services where database data is the primary DR concern

**When not to use**:
- When automated provisioning cannot be reliably validated (manual steps in provisioning = RTO risk)
- When the provisioning time itself would breach RTO (large application stacks with many dependencies)
- When RPO must be < 15 minutes

**Key failure modes**:
- Provisioning automation fails (IaC works in development but fails in DR region due to account limits, missing IAM permissions, or AMI not available in region)
- Configuration drift: DR region configuration is different from production (environment variables, security groups, secrets)
- Dependency unavailability: services the application needs that aren't replicated to the DR region (shared secrets, internal APIs)

**AWS implementation**: RDS cross-region read replica + CloudFormation/Terraform stored in S3 + Route 53 health check + EventBridge rule to trigger Lambda for DR notification.

### Strategy 3: Warm Standby

**Description**: A scaled-down but fully functional version of the production environment runs continuously in the DR site. It is sized for a fraction of production load but can scale up on declaration.

**How failover works**: Declare DR → trigger auto-scaling to production capacity in DR environment → validate → redirect traffic. No provisioning step — the environment exists; it just needs to be scaled.

**How traffic redirects**: Route 53 health check failover (pre-configured) or manual DNS update.

**Data consistency**: Continuous database replication. RPO typically 5–30 minutes depending on replication configuration.

**Cost breakdown**:
- Infrastructure: 25–50% of production infrastructure cost running continuously in DR region
- Operations: Monitoring both environments, testing scale-up automation, configuration parity
- Engineering: Significant — dual-environment configuration management is ongoing work

**When to use**:
- Tier 2 services where 15–60 minute RTO is required
- Services where scale-up automation is reliable and tested
- Regulated environments requiring demonstrated recovery capability

**When not to use**:
- When RTO must be < 15 minutes (scale-up takes time)
- When cost of the continuously running standby is not justified by the service tier
- When configuration parity between primary and standby is difficult to maintain

**Key failure modes**:
- Scale-up fails or takes longer than expected under DR stress
- Configuration drift between primary and standby (standby has stale config, stale container images)
- Standby environment encounters resource limits at scale (insufficient quota for scale target)
- Data replication falls behind during primary high-load periods (the primary fails while the replica is behind)

**AWS implementation**: EKS with minimal node group in DR region + RDS read replica + Auto Scaling groups pre-configured for scale + Route 53 health check + CloudWatch alarms for replication lag.

### Strategy 4: Hot Standby

**Description**: A full-scale, fully configured secondary environment runs continuously. It receives all data changes in real time (near-synchronous replication). No scale-up step — failover is traffic redirection only.

**How failover works**: Declare DR → health check triggers traffic redirect (or manual DNS update) → validate that DR environment is serving traffic correctly. No provisioning, no scale-up.

**How traffic redirects**: Route 53 health check failover (< 60 seconds) or AWS Global Accelerator endpoint failover.

**Data consistency**: Near-synchronous replication. RPO typically 0–5 minutes. For Aurora Global Database, cross-region replication lag is typically < 1 second.

**Cost breakdown**:
- Infrastructure: ~200% of production infrastructure (two full environments)
- Operations: High — two full environments to monitor, maintain, and test
- Engineering: Very high — all configuration changes must be applied to both environments; continuous parity validation

**When to use**:
- Tier 1 services where 5–15 minute RTO is required
- Services with financial, regulatory, or SLA obligations that cannot tolerate longer recovery
- When the cost of hot standby is justified by the downtime cost it prevents

**When not to use**:
- When 15–60 minute RTO is acceptable (warm standby at lower cost)
- When the 2x infrastructure cost cannot be justified in the ROI model
- When data synchronization between regions introduces unacceptable write latency

**Key failure modes**:
- Standby environment configuration drift (missed configuration updates)
- Standby validation gaps (the standby is technically up but functionally broken; discovered only at failover)
- Write amplification: every production write must be replicated; replication overhead at high write volume
- Failback complexity: restoring the primary after a DR event requires careful data reconciliation

**AWS implementation**: Aurora Global Database (< 1 second replication lag) + EKS multi-region + AWS Global Accelerator with endpoint health checks + Route 53 latency-based routing.

---

### Strategy 5: Multi-Site Active-Active

**Description**: Both sites handle live production traffic simultaneously. Load balancer or DNS distributes traffic across both sites. If one site fails, 100% of traffic routes to the surviving site without a failover step.

**How failover works**: There is no failover. The load balancer detects that one endpoint is unhealthy (via health check) and routes all traffic to the surviving endpoint. This happens within the health check interval (typically 10–30 seconds).

**How traffic redirects**: AWS Global Accelerator, Cloudflare Load Balancing, or DNS-based load balancing with very low TTL.

**Data consistency**: The hard problem. Both sites receive writes simultaneously. Conflict resolution, consistency models, and split-brain prevention are complex engineering challenges.

**Cost breakdown**:
- Infrastructure: 200–300% of single-site production (two full sites + global load balancing)
- Operations: Very high — two live environments, continuous health monitoring, data consistency monitoring
- Engineering: Extreme — active-active data consistency is a hard distributed systems problem

**When to use**:
- Five-nines services where any failover time is unacceptable
- Services with geographically distributed users where routing to nearest region is a product requirement
- When regulatory requirements mandate geographic separation for active services

**When not to use**:
- When the data layer cannot support active-active writes safely (most relational databases)
- When the engineering complexity and operational cost are not justified by the availability requirement
- When warm or hot standby achieves the required RTO at a fraction of the cost

**Key failure modes**:
- Split-brain: both sites accept writes during a network partition; data diverges
- Write conflict: two users write conflicting data to different sites simultaneously; merge resolution logic fails
- Global load balancer failure: the thing routing traffic between sites becomes a SPOF
- Operational complexity: changes must be deployed to both sites simultaneously; deployment errors are multiplied

**AWS implementation**: DynamoDB Global Tables (active-active writes, eventual consistency) + EKS in two regions + AWS Global Accelerator + Route 53 health checks. For relational data: Aurora Global Database with write forwarding (but this is logically active-passive at the data layer despite active-active application traffic).

## Strategy Selection Decision Tree

```
Start: What is the required RTO?

RTO > 8 hours → Backup & Restore (Tier 3)
         ↓
RTO 1–8 hours → Is RPO < 1 hour? 
                  Yes → Pilot Light (data replication + on-demand provisioning)
                  No  → Backup & Restore with more frequent backups
         ↓
RTO 15–60 minutes → Warm Standby
         ↓
RTO 5–15 minutes → Hot Standby
         ↓
RTO < 5 minutes → Multi-Site Active-Active
                   But first: Can the data layer support active-active writes?
                   No  → Hot Standby with very fast health check failover
                   Yes → Active-Active (with full awareness of data consistency complexity)
```
