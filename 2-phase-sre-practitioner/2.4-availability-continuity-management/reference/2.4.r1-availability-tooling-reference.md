# Availability Tooling Reference

A decision guide for the tool landscape across monitoring, DR orchestration, chaos engineering, and continuity management. Organized by use case. Includes capability comparison, cost tier, and operational fit.

---

## Category 1: Health Checking and Traffic Routing

### AWS Route 53 Health Checks + Failover Routing

**What it does**: Route 53 health checks poll an endpoint (HTTP, HTTPS, TCP, or a CloudWatch alarm). When a health check fails a configured threshold (e.g., 3 of 5 checks fail), Route 53 updates the DNS record to route to a failover endpoint.

**Failover speed**: Health check interval: 10 or 30 seconds. Failure threshold: 1–10 checks. Minimum failover time: ~30 seconds (10-sec interval × 3 failure threshold). DNS TTL must be set low (60 seconds) for clients to respect the update.

**Capabilities**:
- HTTP/HTTPS endpoint health checking with custom headers
- TCP health checking for non-HTTP services
- CloudWatch alarm–based health checks (checks a metric, not an endpoint)
- String matching in health check response body
- Multi-region failover routing policies
- Latency-based routing (route to lowest-latency region)
- Calculated health checks (composite of multiple child checks)

**Cost**: ~$0.50/month per health check endpoint. Very low cost.

**Limitation**: DNS TTL is a floor on failover time. Route 53 can update the record in 30 seconds, but clients that cached the old IP will continue sending traffic to the failed endpoint until their TTL expires. Global load balancer approaches (AWS Global Accelerator) bypass this limitation.

**Best for**: Multi-region failover routing, public endpoint health checks, simple active-passive DNS failover.

---

### AWS Global Accelerator

**What it does**: Routes traffic through AWS's global network (not the public internet) to the nearest healthy endpoint. Failover to a secondary endpoint occurs in the AWS network layer — no DNS TTL limitation.

**Failover speed**: ~ 30 seconds (health check based; no DNS TTL constraint). Clients do not need to re-resolve DNS — AWS routes the traffic differently at the network layer.

**Capabilities**:
- Static anycast IP addresses (clients always connect to the same IP; AWS routes from there)
- Health check-based endpoint failover without DNS propagation delay
- Traffic dial: send X% of traffic to primary, Y% to secondary (for blue/green or gradual failover)
- Endpoint groups per region with weights
- DDoS protection via AWS Shield Standard (included)

**Cost**: ~$0.025/hour per accelerator + data transfer charges. Higher than Route 53 but eliminates DNS TTL failover delay.

**Best for**: Mission-critical Tier 1 services where DNS TTL failover delay is unacceptable. Applications with global users requiring lowest latency path.

---

### Azure Traffic Manager

**What it does**: DNS-based global traffic routing with health checks. Equivalent to Route 53 with failover routing policies.

**Routing methods**: Priority (active-passive), Weighted, Performance (latency-based), Geographic, Subnet, Multivalue.

**Failover speed**: DNS-based; subject to TTL limitations. Typical failover time 30–60 seconds plus TTL.

**Best for**: Azure-hosted services requiring global failover and traffic distribution.

---

### Cloudflare Load Balancing

**What it does**: Cloudflare's load balancer operates at the network edge — health checks run from Cloudflare's global network, and failover happens at the Cloudflare layer, not at the DNS layer. No DNS TTL limitation for failover.

**Capabilities**:
- Active health checks from multiple Cloudflare locations (detects regional failure, not just endpoint failure)
- Steering policies: off, geo, random, dynamic latency
- Session affinity (sticky sessions)
- Fallback pool (last-resort destination)
- Origin health check with customizable interval and threshold

**Cost**: Starts at $5/month (Enterprise Load Balancing); pro features require Business/Enterprise plan.

**Best for**: Services behind Cloudflare CDN that need fast failover without DNS TTL impact. Consumer-facing services with existing Cloudflare deployment.

## Category 2: Disaster Recovery Orchestration

### AWS Elastic Disaster Recovery (CloudEndure successor)

**What it does**: Continuous block-level replication of source servers (on-premises, AWS, or other cloud) to a lightweight staging area in AWS. In a DR event, converts staging to full-scale recovery instances in minutes.

**RTO**: Typically 5–20 minutes (staging to full scale)  
**RPO**: Seconds (continuous block replication)

**Capabilities**:
- Agent-installed on source servers; OS-agnostic
- Continuous replication to lightweight staging (minimal cost during standby)
- Non-disruptive DR drills (test failover without interrupting replication)
- Point-in-time recovery (restore to any point in the last 90 days)
- Automated launch settings per server

**Cost**: ~$0.028/hour per replicated server during replication + EC2 costs at launch.

**Best for**: Lift-and-shift DR for on-premises servers to AWS. Legacy application DR without cloud-native refactoring. Hybrid environments with mix of on-premises and AWS.

---

### Azure Site Recovery

**What it does**: Replication and orchestrated failover for Azure VMs, on-premises VMware/Hyper-V VMs, and physical servers. Provides recovery plans with sequenced failover across multiple services.

**RTO**: 15–60 minutes for Azure-to-Azure; varies for on-premises-to-Azure  
**RPO**: As low as 30 seconds for Azure-to-Azure

**Capabilities**:
- Replication to Azure or to a secondary Azure region
- Recovery plans with ordering (which services start first; dependency sequencing)
- Test failover to isolated network (validates DR without impacting production)
- Automated health checks and failover triggers
- Integration with Azure Monitor for health monitoring
- Network mapping (maps VNets in source to VNets in target region)

**Cost**: ~$25/month per replicated Azure VM; lower for on-premises VMs.

**Best for**: Azure-primary workloads requiring tested, orchestrated failover. Regulated environments requiring demonstrable DR capability (ASR provides test failover evidence). On-premises to Azure DR migration.

---

### Veeam Backup & Replication

**What it does**: Enterprise backup, replication, and recovery platform. Supports VMware, Hyper-V, AWS, Azure, GCP, and physical servers.

**Capabilities**:
- Instant VM recovery (restore running VM in minutes from backup)
- CDP (Continuous Data Protection) for near-zero RPO on VMware
- Replication to secondary site or cloud
- Backup verification (automated restoration testing)
- Immutable backup to S3 Object Lock / hardened Linux repository

**Cost**: Per-socket or per-workload licensing; significant cost for large environments.

**Best for**: Large enterprise environments with heterogeneous infrastructure. Organizations needing immutable backup for ransomware protection. VMware-heavy environments.

## Category 3: Synthetic Monitoring and Probing

### Datadog Synthetic Monitoring

**What it does**: Runs simulated user transactions from Datadog's global network of probes. Detects availability failures before or without customer reports.

**Capabilities**:
- Browser tests: Selenium-like scripts that simulate user journeys (login, checkout, form submission)
- API tests: HTTP calls that validate response code, response body, response time
- Multi-step API tests: sequences of API calls that test a workflow
- Private locations: run probes from inside your network (tests internal services)
- Alert on test failure with configurable retry and notification

**Test types by use case**:
- "Is the service reachable at all?" → Simple HTTP uptime check (API test, 1 step)
- "Can a user complete a login?" → Browser test or multi-step API test
- "Is the DR environment functional?" → Run the same synthetic test targeting the DR endpoint

**Cost**: Starts at $5/test/month for API tests; $12/test/month for browser tests. Significant at scale (dozens of tests).

**Best for**: SLA monitoring with audit evidence. DR validation (run synthetics against DR endpoint during DR test). Customer-journey availability monitoring.

---

### Pingdom / Uptime Robot (Simpler Alternatives)

For organizations not needing full Datadog integration, simpler uptime monitoring tools:

**Pingdom**: HTTP/S, TCP, DNS, SMTP monitoring from 100+ global locations. Alert on downtime, response time, and certificate expiry. RUM (real user monitoring) add-on. Cost: $15–$85/month.

**Uptime Robot**: Free tier (50 monitors, 5-minute checks). Paid for 1-minute checks and advanced features. Good for Tier 3 service monitoring. Cost: Free to ~$60/month.

## Category 4: Chaos Engineering

### AWS Fault Injection Simulator (FIS)

**What it does**: AWS-native chaos engineering service. Injects failures into AWS resources: EC2 instances, EKS pods, RDS databases, ECS tasks, network conditions.

**Capabilities**:
- Experiment templates: define target resources, fault type, and stop conditions
- Stop conditions: automatically stop the experiment if a CloudWatch alarm fires (blast radius control)
- Native integration with AWS services — no agent required
- Experiment history and results logging to CloudWatch
- Pause/resume experiments

**Fault types available**:
- EC2: terminate instances, send stop/reboot, inject CPU stress or network blackhole
- EKS: terminate pods or nodes (via SSM)
- RDS: reboot with failover (triggers Multi-AZ failover)
- Network: inject latency or packet loss between VPCs or to internet
- Systems Manager: run arbitrary commands on managed instances

**Cost**: Pay per experiment action hour. Very low — typically < $1 per experiment.

**Best for**: AWS-native environments. Teams beginning chaos engineering who want controlled, auditable experiments. Pre-configured Game Days without custom tooling.

---

### Gremlin

**What it does**: Enterprise chaos engineering platform. Broader fault type library than AWS FIS. Works across cloud providers, on-premises, and Kubernetes.

**Capabilities**:
- State attacks: kill, shutdown, time travel (change system clock)
- Network attacks: blackhole, latency, packet loss, packet corruption, bandwidth limit
- Resource attacks: CPU, memory, disk, IO
- Application attacks (limited): requires agent or library integration
- Kubernetes integration: target pods, nodes, namespaces
- Scenarios: multi-step attack sequences
- Attack history and impact reporting

**Cost**: $10K–$50K+/year (enterprise). Free tier is very limited.

**Best for**: Organizations with dedicated SRE teams running regular Game Days. Multi-cloud or hybrid environments where AWS FIS doesn't reach. Advanced fault type requirements.

---

### Chaos Monkey (Netflix OSS)

**What it does**: Randomly terminates EC2 instances in production. The original chaos engineering tool. Simple, opinionated: if your services cannot tolerate random instance termination, you have a reliability problem.

**Capabilities**: Limited to EC2 instance termination (the original tool). Various forks and extensions add more fault types.

**Cost**: Open source; operational cost of running the tool.

**Best for**: Teams that want to enforce "always-on" resilience — Chaos Monkey can run continuously in background, not just during scheduled Game Days. Simple N+1 resilience validation.

## Category 5: Incident Alerting and On-Call (Cross-reference with [Incident Management])

### PagerDuty

**What it does**: On-call scheduling, alert routing, and incident management platform. Integrates with monitoring tools to route alerts to the right on-call engineer.

**DR-relevant capabilities**:
- Escalation policies: if on-call engineer doesn't acknowledge within X minutes, escalate to next level
- Status page integration: auto-update status page on incident creation
- Response plays: pre-defined response sequences triggered by specific alert types (including "trigger DR activation" play)
- Stakeholder notifications: notify non-technical stakeholders about major incidents
- Runbook attachments: attach runbook links to alert routing rules

**Reliability for DR**: PagerDuty is a critical vendor for on-call operations. If PagerDuty is unavailable during a DR event, your alerting chain fails. Maintain a fallback notification method (SMS broadcast list, secondary alerting tool) and test it. Cross-reference [ch05 — Supplier Continuity].

---

### OpsGenie (Atlassian)

**What it does**: Alternative to PagerDuty for on-call and incident alerting. Tighter integration with Jira and Confluence.

**DR-relevant capabilities**: Similar to PagerDuty. On-call scheduling, escalation policies, alert routing, heartbeat monitoring (if a service stops sending heartbeats, alert fires).

---

## Category 6: DR Plan and BCP Documentation

### Fusion Framework / Riskonnect / Archer (Enterprise GRC)

**What they do**: Governance, Risk, and Compliance platforms that include BCP/DR planning modules. Manage DR plans, BIA data, test results, and audit evidence in a structured system.

**Capabilities**:
- BIA data collection and scoring
- DR plan authoring with templates
- Test scheduling and results tracking
- Regulatory compliance mapping
- Executive dashboard and reporting

**Cost**: Enterprise licensing; $50K–$500K+/year depending on modules and scale.

**Best for**: Large enterprises with formal GRC programs and regulatory audit obligations. Organizations where DR plans must be managed centrally across hundreds of services.

**Alternative for smaller teams**: SharePoint/Confluence with structured templates (this framework's templates serve this purpose). GitHub repository with Markdown DR plans (version-controlled, accessible).

## Tooling Selection by Service Tier

| Capability | Tier 1 (Mission-Critical) | Tier 2 (Business-Critical) | Tier 3 (Standard) |
|-----------|--------------------------|---------------------------|-------------------|
| Health checking | AWS Global Accelerator + Route 53 | Route 53 health checks | Route 53 or Uptime Robot |
| Synthetic monitoring | Datadog Synthetics (multi-step) | Datadog Synthetics (API test) | Uptime Robot or Pingdom |
| DR orchestration | AWS Elastic DR or Azure Site Recovery | AWS FIS + manual runbooks | Manual runbooks |
| Chaos engineering | Gremlin (quarterly) + AWS FIS (monthly) | AWS FIS (semi-annual) | AWS FIS (annual) |
| On-call alerting | PagerDuty + SMS fallback | PagerDuty | PagerDuty or OpsGenie |
| DR plan management | GRC platform or structured Git repo | Structured Git repo or Confluence | Confluence or SharePoint |
