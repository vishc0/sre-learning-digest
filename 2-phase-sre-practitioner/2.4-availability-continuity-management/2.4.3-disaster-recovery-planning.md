# ch03 — Disaster Recovery Planning

## DR vs. HA: The Distinction That Matters

High availability keeps you running through failures. Disaster recovery gets you running again after failures that HA cannot absorb.

This distinction is operationally critical because the two disciplines have different activation criteria, different decision authorities, and different cost profiles.

**HA activates automatically** — no human decision required. When a node fails in an N+1 cluster, traffic redistributes. The SRE team is notified but does not need to initiate anything. HA is always on.

**DR activates by declaration** — a human authority decides that the current failure is a DR scenario, not a recoverable incident, and activates the DR plan. DR is a managed transition with defined steps, a defined sequence, and defined validation gates before service is restored to users.

The failure to understand this distinction produces the most common DR error: the team fights the failure as an incident for 3–6 hours before someone recognizes it as a DR scenario and declares. Those 3–6 hours are burned on the wrong operating model.

**Rule**: Define your DR declaration criteria before the disaster. The declaration should require no judgment in the moment — it should be deterministic based on observable conditions.

## DR Scenario Taxonomy

Not all disasters are equal. Each scenario type has different implications for recovery architecture and recovery time.

### Tier A: Infrastructure Failure (Single Component)

**What it is**: A single server, storage array, or network device fails. HA mechanisms should absorb this. If they do not, it may indicate an SPOF that was not accounted for in the HA design.

**DR implication**: Usually not a DR scenario — this is an incident. DR declaration is inappropriate unless the component failure triggers a cascade that HA cannot absorb.

**Recovery time**: Minutes to hours with HA. Hours without.

### Tier B: Data Center or Availability Zone Failure

**What it is**: An entire physical facility or cloud AZ becomes unavailable — power failure, cooling failure, network isolation, physical disaster.

**DR implication**: If the service is deployed across multiple AZs (standard for Tier 1 and Tier 2 services), this is absorbed by HA. If the service is single-AZ (legacy or cost-constrained), this triggers a DR event.

**Recovery time**: Minutes (multi-AZ HA) to hours (DR failover to secondary site).

### Tier C: Regional Failure

**What it is**: An entire cloud region or geographic area becomes unavailable. AWS us-east-1 has experienced multiple partial failures. Major natural disasters, regional power grid failures, or large-scale network events fall here.

**DR implication**: Tier 1 services with geographic redundancy absorb this. All other services fail — this is a DR event requiring failover to a second region.

**Recovery time**: Minutes (hot standby with geo redundancy) to hours (warm standby activation).

### Tier D: Data Corruption or Data Loss

**What it is**: Data in a database, file system, or data store is corrupted or deleted — by software bug, operator error, ransomware encryption, or cascading failure.

**DR implication**: This scenario is the hardest. HA mechanisms replicate the corruption to the replica before anyone realizes the corruption has occurred. The secondary copy is as corrupt as the primary.

**Recovery time**: Hours to days. Point-in-time recovery from backup. Data validation before restoration. The window of data loss depends on when the last clean backup was taken and how quickly the corruption was detected.

**T-Mobile context**: For the notification platform, a corrupt routing configuration applied to the database could affect message delivery for all 25M daily messages before detection. The RPO in this scenario is measured from the last pre-corruption backup — which must be tested.

### Tier E: Ransomware or Malicious Encryption

**What it is**: Ransomware encrypts production data and, often, backup data. Attackers increasingly target backup systems specifically to maximize leverage.

**DR implication**: Standard DR plans do not account for ransomware because standard DR assumes backups are clean. Ransomware recovery requires: (1) confirmed clean backup (isolated, immutable, air-gapped); (2) forensic containment of infected systems before restoration; (3) rebuilt infrastructure before connecting restored data.

**Recovery time**: Days to weeks. IBM Security 2023 data: average ransomware recovery time is 23 days. Organizations without air-gapped backups average 47 days.

**Director decision**: Does your DR plan include a ransomware scenario? Does it include isolated backup storage? If the answer to either is no, this is a critical gap.

### Tier F: Multi-Region or Multi-Cloud Failure

**What it is**: Simultaneous failure of primary and secondary regions, or a shared dependency failure that affects both (a global CDN failure, a DNS provider outage, a shared authentication provider failure).

**DR implication**: Catastrophic. No automated failover resolves this — both targets are down. Recovery requires manual intervention, potentially including activating tertiary arrangements or accepting an extended outage.

**Recovery time**: Hours to days. This scenario tests whether your BCP addresses the situation where DR itself has failed.

## DR Strategy Selection

Five DR strategies, in ascending order of cost and descending order of recovery time:

### Strategy 1: Backup and Restore

**How it works**: Regular backups are taken and stored. In a DR event, the environment is rebuilt and data is restored from backup.

**RTO**: Hours to days (dependent on backup restoration time and environment rebuild time)
**RPO**: Hours to days (dependent on backup frequency)
**Cost**: Lowest — pay only for storage. No standby infrastructure.
**Use case**: Tier 3 services where hours-long RTO is acceptable. Development environments. Data archiving.
**Risk**: RTO and RPO are theoretical until tested. Most organizations discover their actual RTO is 3–5x their estimate on first test.

### Strategy 2: Pilot Light

**How it works**: Core infrastructure (database, critical data) is replicated to the DR site and kept current. Application infrastructure is not provisioned — it is spun up only when DR is declared.

**RTO**: 1–4 hours (time to provision application tier and configure it)
**RPO**: Minutes (database replication runs continuously; lag depends on sync mode)
**Cost**: Low — pay for data replication and minimal standby infrastructure. Application tier is provisioned on-demand.
**Use case**: Tier 2 services where 2–4 hour RTO is acceptable. Cloud-native applications where provisioning is automated.
**Risk**: Application provisioning under DR stress is error-prone. Automation must be tested. Provisioning dependencies (container images, configuration secrets, network configuration) must be validated in the DR environment before they are needed.

### Strategy 3: Warm Standby

**How it works**: A scaled-down but fully functional version of the production environment runs continuously in the DR site. It is too small to handle production load, but it can be scaled up on DR declaration.

**RTO**: 15–60 minutes (time to scale up the standby environment and redirect traffic)
**RPO**: Seconds to minutes (continuous data replication)
**Cost**: Medium — pay for scaled-down standby infrastructure (typically 25–50% of production capacity)
**Use case**: Tier 2 services where 30–60 minute RTO is required. Business-critical applications with moderate budgets.
**Risk**: Scale-up under DR stress may not complete successfully if automation is untested. The warm standby may not have current configuration if drift has occurred between primary and secondary.

### Strategy 4: Hot Standby

**How it works**: A full-scale, fully configured secondary environment runs continuously. It receives all data changes in real time (synchronous or near-synchronous replication). Failover redirects traffic without any provisioning step.

**RTO**: Minutes (failover and health validation only)
**RPO**: Seconds (synchronous replication) or minutes (asynchronous)
**Cost**: High — pay for two full production environments running continuously (minus any traffic on the standby)
**Use case**: Tier 1 services where 5–15 minute RTO is required. Regulated environments with strict recovery commitments.
**Risk**: Cost is significant. Standby drift if configuration management is not synchronized. Validation that the standby is truly ready (not just running but correct) must be automated.

### Strategy 5: Multi-Site Active-Active

**How it works**: Both sites handle live traffic simultaneously. No failover step is needed — traffic is redistributed from the failed site to the surviving site by the load balancer or DNS health check.

**RTO**: Seconds (no failover step; traffic redistribution only)
**RPO**: Near-zero (synchronous data replication or shared data layer)
**Cost**: Very high — two full production environments, both serving traffic, with synchronous data replication adding write latency
**Use case**: Five-nines services. Mission-critical applications where any failover time is unacceptable. Carrier-grade infrastructure.
**Risk**: Data consistency between active sites is the hard problem. Split-brain. Write conflict resolution. The engineering investment is substantial. Do not deploy active-active without solving the data layer first.

## RTO/RPO Requirements: Derivation from Business Requirements

The most common error in DR planning: RTO and RPO targets are set by engineering based on what the architecture can achieve, rather than by the business based on what it needs.

The correct direction is business → architecture:

1. Business defines MTPD (Maximum Tolerable Period of Disruption) — see [ch04]
2. MTPD sets the ceiling for RTO (RTO must be less than MTPD)
3. Business defines acceptable data loss (converted to RPO)
4. Engineering designs an architecture that achieves the required RTO/RPO
5. Engineering presents the cost
6. Business accepts the cost or negotiates the target

When engineering presents the target ("our DR architecture gives us 4-hour RTO"), it is inverted — the architecture is defining what the business will accept rather than the business defining what the business needs. This produces chronic misalignment discovered during actual disasters.

## DR Architecture Documentation

A DR plan that exists only in someone's memory is not a DR plan. It is a risk.

### Required DR Documentation Set

**1. Service inventory and tier classification** — every service, its tier, its RTO/RPO target, its DR strategy, and who owns it.

**2. Network architecture diagram** — current state, DR state. How traffic flows in each state. DNS configuration, load balancer configuration, CDN configuration in DR mode.

**3. Recovery runbooks** — step-by-step instructions for each DR scenario type. Every step must include the expected output so the operator knows whether it succeeded.

**4. Data recovery procedures** — how to restore from backup, how to validate data integrity, how to handle replication resync.

**5. Contact list** — who is called when DR is declared, in what order, using what channel. Include escalation paths for unreachable contacts. Include vendor support contacts (cloud provider, networking vendor, database vendor) with account numbers and support tier.

**6. Validation checklist** — before declaring DR complete and restoring user traffic, these checks must pass. This is the gate that prevents declaring recovery complete when services are technically up but functionally broken.

**7. Communication plan** — what customers, regulators, and internal stakeholders are told, when, and by whom. Cross-reference [ch05 — BCP].

### Documentation Ownership and Review Cadence

| Document | Owner | Review Cadence | Update Triggers |
|----------|-------|---------------|-----------------|
| Service inventory | VP of Operations | Quarterly | New service launch, tier reclassification |
| DR runbooks | SRE team (per service) | Semi-annual | Architecture change, DR test findings |
| Network architecture | Infrastructure team | After each significant change | Major infrastructure change |
| Contact list | Operations Manager | Monthly | Personnel changes, vendor changes |
| Validation checklist | SRE lead | After each DR test | Failed validation steps in tests |

## The DR Gap Analysis

The gap analysis compares current recovery capability against business requirements. This is the document that drives DR investment decisions.

### Gap Analysis Format

| Service | Tier | Required RTO | Required RPO | Current RTO (tested) | Current RPO (tested) | RTO Gap | RPO Gap | Gap Risk | Investment Required |
|---------|------|-------------|-------------|---------------------|---------------------|---------|---------|----------|---------------------|
| Payment API | 1 | 15 min | 0 | 45 min | 5 min | 30 min | 5 min | Critical | Hot Standby upgrade: $400K/yr |
| Notification Platform | 2 | 2 hr | 30 min | 3 hr | 1 hr | 1 hr | 30 min | High | Warm Standby config fix: $50K/yr |
| Internal CRM | 3 | 24 hr | 4 hr | 20 hr | 2 hr | None | None | None | — |

The gap analysis is the investment request document. The "Investment Required" column translates directly into a capital or operational budget line item.

**When to run the gap analysis**: After every DR test. After every significant architecture change. At the annual budget cycle. After a real DR event.

## Cloud DR Patterns

### AWS Cross-Region DR

AWS provides native tools for cross-region DR at each strategy level:

| Strategy | AWS Services | Notes |
|----------|-------------|-------|
| Backup & Restore | AWS Backup, S3 Cross-Region Replication | S3 CRR has near-real-time replication; RDS automated backups are regional |
| Pilot Light | CloudFormation/Terraform (IaC provisioning), RDS read replica promoted to primary | IaC must be tested in DR region before it is needed |
| Warm Standby | EKS cluster (minimal node count), RDS multi-region read replica | Auto Scaling must be tested; scaling speed is a constraint |
| Hot Standby | Multi-region RDS cluster (Aurora Global), EKS in both regions, Route 53 health check failover | Aurora Global has < 1 second replication lag; typical RPO of seconds |
| Active-Active | AWS Global Accelerator, DynamoDB Global Tables, Aurora Global with write forwarding | DynamoDB Global Tables provides active-active write capability |

**AWS Route 53 health check failover**: Route 53 health checks poll your primary endpoint. If the health check fails, Route 53 updates the DNS record to point to your secondary endpoint. Failover completes in approximately 60–90 seconds (health check failure threshold + DNS propagation time). DNS TTL must be set low (60 seconds) for this to be effective.

### Azure Site Recovery

Azure Site Recovery (ASR) provides automated replication and orchestrated failover for VMs and Azure services. Key capabilities:
- Continuous replication of Azure VMs, on-premises VMs, and physical servers to Azure
- Recovery plans with sequencing — define which services recover first
- Test failover capability: can validate DR in an isolated network without affecting production
- RTO typically 15–60 minutes for Azure-to-Azure scenarios

**T-Mobile context**: For hybrid environments mixing on-premises and Azure, ASR can replicate on-premises infrastructure to Azure as the DR target — eliminating the need for a physical secondary data center.

### Multi-Cloud DR

Multi-cloud DR uses a different cloud provider as the DR target. Motivations:
- Eliminating cloud provider as a single point of failure
- Regulatory requirement for provider diversity
- Geographic coverage in regions where secondary provider is stronger

**Challenges**: Cloud-native services (managed databases, serverless compute, cloud load balancers) are not portable between providers. Multi-cloud DR requires an abstraction layer (Kubernetes, Terraform) or explicit parallel implementations for both clouds. The operational complexity is significantly higher. Consider whether the risk of a single-provider failure justifies the cost and complexity.

**Decision rule**: Multi-cloud DR is appropriate when regulatory requirements mandate it or when the expected cost of a single-provider outage at your scale exceeds the incremental cost of multi-cloud architecture. For most organizations, multi-region within a single provider is sufficient.
