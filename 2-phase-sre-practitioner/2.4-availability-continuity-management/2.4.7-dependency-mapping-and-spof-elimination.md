# ch07 — Dependency Mapping and SPOF Elimination

## Why Dependency Maps Exist

During the 2021 Facebook outage, a routine BGP configuration change withdrew Facebook's own IP prefixes from the global internet. Simultaneously, the internal systems Facebook engineers used to diagnose and fix the problem also went down — because those internal systems depended on the same external network that was no longer reachable. The physical access control systems at Facebook's data centers used the same authentication infrastructure, which was also offline. Engineers who drove to the data center could not get in.

The outage lasted 6 hours. The root cause was a configuration change in a system that appeared, from the documentation, to be isolated from the services it actually took down.

The lesson is not about BGP. The lesson is about undocumented dependencies — the connections between systems that exist in production but not in architecture diagrams.

A dependency map is the document that closes this gap. It is the authoritative record of what depends on what, built from observation rather than design.

## Service Dependency Map: Why You Need It, How to Build It

### What a Dependency Map Is

A dependency map documents every component that a service requires to function, and every service that depends on that component. It answers:
- "If X fails, what else fails?"
- "What does X depend on that might not be obvious?"

A dependency map is built from:
- **Traffic observation** (what the service actually calls at runtime) — not what documentation says it calls
- **Configuration files** (what endpoints, databases, and external services are configured)
- **Incident postmortems** (what unexpected dependencies were discovered during past failures)
- **Engineering interviews** (what team members know that isn't documented)

### Building a Service Dependency Map

**Step 1: Start with the service inventory**
Every service in scope must be catalogued with: owner, tier, primary function, and initial list of known dependencies.

**Step 2: Instrument for dependency discovery**
Distributed tracing (OpenTelemetry, Jaeger, AWS X-Ray) captures actual runtime call graphs. Service mesh (Istio, Linkerd) logs every inter-service call. Run dependency discovery in production — staging traffic patterns are not representative.

**Step 3: Map upstream and downstream**
For each service:
- **Upstream dependencies** (what this service calls): databases, caches, APIs, message queues, external vendors, cloud services
- **Downstream dependents** (what calls this service): every service or system that sends requests to this service

**Step 4: Identify implicit dependencies**
Implicit dependencies are the dangerous ones. Common hidden dependencies:
- Shared infrastructure services (DNS, NTP, container registries, artifact stores)
- Shared operational tooling (logging pipelines, monitoring systems, CI/CD)
- Shared authentication (if Okta is down, everything that requires authentication is down)
- Shared configuration stores (if your config server is down, services that require dynamic config at startup cannot start)

**Step 5: Validate with incident history**
Review the last 12 months of incident postmortems. Every "unexpected dependency" found during an incident should appear in the map. If it does not, the map is incomplete.

### Dependency Map Format

For each service, document:

```
Service: Notification API
Tier: 2 (Business-Critical)
Owner: Platform SRE Team
RTO: 2 hours | RPO: 30 minutes

UPSTREAM DEPENDENCIES:
  - RabbitMQ (message queue) — synchronous; failure = message delivery stops
  - MySQL (routing database) — synchronous; failure = API cannot process requests
  - Redis (session cache) — synchronous; failure = authentication fails
  - Okta (identity provider) — synchronous; failure = all authenticated requests fail
  - AWS SQS (dead letter queue) — asynchronous; failure = failed messages not preserved
  - Datadog (metrics) — asynchronous; failure = monitoring blind, operations continue
  - Splunk (log drain) — asynchronous; failure = log gaps, operations continue

DOWNSTREAM DEPENDENTS:
  - Enterprise Customer A — direct API consumer
  - Enterprise Customer B — direct API consumer
  - Delivery Status Service — consumes delivery receipts
  - Billing Platform — consumes usage events

IMPLICIT DEPENDENCIES:
  - Route53 (DNS) — failure = API unreachable by name
  - ACM (TLS certificates) — failure = HTTPS connections fail
  - Secrets Manager (credential rotation) — failure at startup = service cannot start
  - Container registry (ECR) — failure at deployment = new pods cannot pull image

FAILURE PROPAGATION:
  - RabbitMQ failure → Notification API stops processing → Enterprise customer SLA breach in < 30 min
  - Okta failure → All authenticated calls fail → Notification API unavailable to customers
  - DNS failure → API unreachable by name → Complete outage (no DNS, no traffic)
```

### Dependency Map Review Cadence

| Trigger | Action |
|---------|--------|
| Quarterly | Full review of Tier 1 service maps; spot-check Tier 2 |
| Architecture change (new service, new integration) | Update affected maps within 2 weeks of change |
| Incident revealing unknown dependency | Update maps within 24 hours; post-incident requirement |
| New vendor onboarded | Add vendor to dependent service maps |
| Annual DR test | Validate maps are current before test design |

## SPOF Identification: The Systematic Process

A Single Point of Failure (SPOF) is any component whose failure causes a Tier 1 or Tier 2 service to become completely unavailable, where no automatic redundancy activates.

### SPOF Identification Dimensions

**Infrastructure SPOFs**
- Single server or instance without redundancy
- Single storage array or volume
- Single power supply or power path
- Single data center or availability zone for a service that requires regional HA

**Network SPOFs**
- Single network path (no redundant routing)
- Single DNS provider
- Single load balancer (no HA pair)
- Single CDN provider with no fallback
- Single TLS certificate with no automated renewal

**Application SPOFs**
- Single instance of a stateful service (no replica)
- Single-region database with no standby
- Shared configuration store with no redundancy
- External API with no fallback path

**Data SPOFs**
- Single backup copy (no verified secondary copy)
- Backup stored in the same facility as primary data
- Backup encryption keys managed by same system as data
- No point-in-time recovery capability for critical databases

**People SPOFs**
- Single engineer with operational knowledge of a critical system
- Single person with vendor escalation credentials
- Single person who can approve production changes in a specific domain
- Single person who maintains the DR plan

**Vendor SPOFs**
- Single vendor providing a critical service with no pre-qualified alternate
- Vendor with a contractual SLA but no demonstrated DR capability
- Vendor whose outage your SLA does not protect against

## SPOF Risk Scoring

Score each identified SPOF using three dimensions:

**Probability of failure** (P):
- Low (L): Component fails less than once per year historically; vendor has 99.99%+ SLA
- Medium (M): Component fails 1–3 times per year; vendor has 99.9%–99.99% SLA
- High (H): Component fails more than 3 times per year; vendor availability is unknown or below 99.9%

**Impact of failure** (I):
- Low: Degraded service; customers experience slower response but service continues
- Medium: Partial outage; specific functions or customer segments lose service
- Critical: Complete outage; Tier 1 or Tier 2 service is fully unavailable

**Exposure time** (E): Average time from failure to detection + time to activate redundancy or restore manually:
- Short (< 5 minutes): Automated detection and automated failover
- Medium (5–30 minutes): Automated detection, manual failover
- Long (> 30 minutes): Manual detection required, manual recovery

**SPOF Risk Score**: P × I × E (high values = highest priority)

| SPOF | Probability | Impact | Exposure | Risk Score | Priority |
|------|-------------|--------|----------|-----------|----------|
| Single DNS provider | Medium | Critical | Long | High | P1 |
| Single RabbitMQ broker (no HA) | Medium | Critical | Medium | High | P1 |
| Senior DBA (only person who can restore from backup) | Low | Critical | Long | Medium | P2 |
| Single MPLS circuit to HQ | Low | Medium | Medium | Medium | P2 |
| Single monitoring agent on host | Low | Low | Short | Low | P3 |

## SPOF Remediation Prioritization

Not all SPOFs can be eliminated simultaneously. Prioritize based on cost vs. risk reduction:

### Cost vs. Risk Reduction Matrix

| SPOF | Remediation Cost | Risk Reduction | Priority |
|------|-----------------|----------------|---------|
| Add DNS secondary provider | Low ($5K/yr) | Critical | Fund immediately |
| RabbitMQ HA cluster | Medium ($40K/yr infra + 2 weeks eng) | Critical | Fund this quarter |
| DBA cross-training | Medium (40 hrs engineer time) | Critical | Schedule this quarter |
| Redundant MPLS circuit | High ($60K/yr) | Medium | Evaluate with business case |
| Active-Active database | Very high ($200K/yr) | High for Tier 1 | Only if Tier 1 requires it |

**Rule**: Any SPOF with Critical impact that can be eliminated for less than 10% of the annual cost of the failure it prevents should be funded without requiring a formal business case. The math is self-evident.

## Vendor SPOF: The Governance Problem

Vendor SPOFs are the hardest category to address because they are outside the organization's direct control. No amount of internal architecture can compensate for a critical vendor outage if there is no alternate path.

### The Vendor SPOF Problem

Organizations routinely rely on vendors for:
- **Identity (Okta, Azure AD)**: If the identity provider is down, authentication fails across all services simultaneously. This turns a vendor outage into a complete internal outage.
- **Observability (Datadog, PagerDuty, Splunk)**: If the monitoring platform is down during a DR event, you are flying blind during the most critical operational moment.
- **DNS (Route 53, Cloudflare)**: DNS infrastructure failure is an outage for every service behind it.
- **Message brokers (Confluent Cloud, AWS MSK)**: Managed queue infrastructure outage affects all producers and consumers.

### Vendor SPOF Governance Approach

For each critical vendor (Tier 1 service dependency):
1. **Review their historical SLA performance** — published incident history, not stated SLA
2. **Request evidence of their DR capability** — when did they last test? what was the result?
3. **Assess whether a vendor failure triggers your regulatory notification requirement** — if yes, the vendor is a Tier 1 risk
4. **Identify a technical fallback path** — can you route around them even temporarily?
5. **Negotiate vendor-to-vendor SLA alignment** — if they breach, how does that flow through your customer SLA?

**The contractual reality**: Vendor SLAs typically cap liability at the value of the contract. If a vendor outage costs you $2M in customer SLA credits and the vendor contract is worth $50K/year, the SLA credit from the vendor covers a fraction of your loss. Vendor SLAs are risk disclosure documents, not loss prevention mechanisms.

## People SPOF: The Bus Factor

The "bus factor" measures how many people would need to be simultaneously unavailable for critical knowledge or function to be lost. A bus factor of 1 is a SPOF.

### Identifying People SPOFs

Ask for each critical operational function:
- Who can perform this function today?
- If that person is unavailable for 2 weeks, who else can perform it?
- Is the procedure documented in enough detail for someone to perform it without the primary person?

### Measuring Bus Factor

| Function | Primary Owner | Secondary Owner | Documented? | Bus Factor |
|----------|--------------|-----------------|-------------|-----------|
| DR failover execution | Lead SRE | None | Partially | 1 (SPOF) |
| Database backup restoration | DBA | None | No | 1 (SPOF) |
| FCC regulatory notification | Regulatory VP | Legal Counsel | Yes (template) | 2 |
| Production change approval | CTO | VP of Engineering | Yes (policy) | 2 |
| Vendor escalation (AWS) | Platform Lead | SR-1 account team | Yes (contacts) | 3 |

Any function with Bus Factor = 1 is a people SPOF. Remediation requires documentation + cross-training + validation.

**The documentation rule**: A procedure documented well enough for someone to execute independently — without calling the primary owner — reduces the bus factor by making the procedure itself the durable knowledge holder.

## The "Known Unknown" Problem

The most dangerous dependencies are ones that exist but are not known to the team managing the service.

Categories of known unknowns:
- **Legacy integrations**: Old systems that call a service via an undocumented API path that was never officially supported
- **Batch jobs**: Scheduled processes that run at 2am and nobody documented them because they "always just worked"
- **Monitoring agents**: An external monitoring system that depends on port access or API endpoints not listed in the service specification
- **Emergency access paths**: A backdoor or break-glass access path that bypasses the normal authentication system and hasn't been tested in 3 years

The known-unknown problem is solved by:
1. **Runtime dependency discovery** (tracing, mesh telemetry) — see what actually calls your service in production
2. **Incident-driven discovery** — every incident that reveals an unexpected dependency gets mapped immediately
3. **Quarterly dependency review** — ask "what has changed that might create new dependencies we haven't mapped?"
