# SLM Tooling Reference

## Overview

No single tool covers the full SLM requirement. The production SLM stack integrates three functional layers: measurement and tracking (SLI/SLO), lifecycle management (SLA/breach/reporting), and customer-facing communication (status, incidents). This reference covers the four primary tools and their roles in each layer.

---

## Tool 1: ServiceNow SLM Module

### Role in the SLM Stack

ServiceNow is the system of record for SLA lifecycle management: SLA definitions, breach records, credit calculations, service report workflow, and SLM program reporting. It is the commercial and governance layer of the stack.

### Core Capabilities

**SLA Definition Management**:
- Store and version SLA agreements (availability target, credit schedule, measurement methodology)
- Link SLAs to CI (Configuration Item) records in the CMDB — connecting the SLA to the service it covers
- Automate SLA timer tracking: start/pause/stop based on incident state changes

**SLA Availability Tracking**:
- Calculate SLA compliance from incident data: when an incident is open and causes an SLA impact, ServiceNow tracks the duration and compares against the SLA commitment
- Generate SLA breach alerts when availability is at risk
- Calculate credits based on the configured credit schedule

**Breach Record Management**:
- Create and track breach records through the lifecycle (declared → notified → credit issued → remediation committed → closed)
- Attach breach records to customer account records
- Generate breach history reports by service and customer

**Service Report Generation**:
- Pull availability data from incident records to generate monthly service reports
- Route service reports through approval workflow (VP review before customer delivery)
- Track service report delivery history

### Gaps and Limitations

- **Not a real-time measurement tool**: ServiceNow tracks SLA compliance based on incident records, not continuous SLI measurement. If an incident is not opened in ServiceNow, the SLA clock does not start — which means SLA tracking is only as accurate as incident management discipline.
- **Integration required for SLI data**: The SLI measurement (from Datadog or similar) must be fed into ServiceNow. This integration is typically custom-built and requires ongoing maintenance.
- **Customer portal is limited**: ServiceNow's customer-facing portal capability requires significant customization to deliver a clean, self-service experience. Most organizations use a separate tool (Statuspage) for customer-facing status.
- **SLO tracking is not native**: ServiceNow manages SLAs (commercial agreements). Managing SLOs (internal engineering targets) with error budget burn rates requires Datadog or Grafana integration.

### VP-Relevant Features

- **SLA dashboard**: View all active SLAs with current compliance status, at-risk services, and breach records
- **Escalation workflow**: Automated escalation notifications when SLA compliance drops below defined thresholds
- **Service report approvals**: VP can review and approve service reports before they go to customers
- **Breach record audit trail**: Complete history of every breach, credit, and remediation commitment per customer

## Tool 2: Datadog SLO Tracking

### Role in the SLM Stack

Datadog is the engineering and operations layer: it measures SLIs in real time, tracks SLO compliance, calculates error budget burn rates, and provides the monitoring data that feeds the ServiceNow SLM layer.

### Core Capabilities

**SLO Definition and Tracking**:
- Define SLOs from existing monitors (request success rate, latency, custom business metrics)
- Track SLO compliance over configurable time windows (7-day, 28-day, 30-day, 90-day)
- Calculate error budget: remaining budget, burn rate, projected exhaustion date

**Two SLO Types in Datadog**:

| Type | How It Works | Best For |
|------|-------------|---------|
| **Monitor-based** | SLO is met when a monitor is in "OK" state | Synthetic probes, uptime checks |
| **Metric-based** | SLO is calculated from a ratio of "good" to "total" events | Request success rate, latency percentile |

**Error Budget Burn Rate Alerts**:
- 7-day and 30-day burn rate alerts: notify when the service is consuming error budget faster than sustainable
- Burn rate formula: `(Error Budget Consumed in N Days) / (Error Budget Allowed in N Days)`
- Alert when burn rate > 2× for 7 days, or burn rate > 14× for 1 hour (Datadog standard thresholds)

**SLO Dashboard**:
- Per-service SLO status, history, and error budget remaining
- Multi-service portfolio view for SRE leads and VP
- Historical compliance data for service reporting

### Gaps and Limitations

- **Commercial layer not included**: Datadog does not manage SLA documents, credit calculations, breach records, or service reports. The gap between Datadog's SLO data and ServiceNow's SLA tracking must be bridged by integration or manual process.
- **Customer-facing access requires configuration**: Sharing SLO dashboards with customers requires Datadog organizational access configuration. Most organizations use Statuspage rather than direct Datadog access for customers.
- **Dependency tracking is limited**: Compound SLO calculation (aggregating SLOs across dependent services) is not natively supported — it requires custom dashboards or external calculation.
- **Historical data retention**: Default retention varies by plan. Long-term SLO trend analysis (12+ months) may require extended retention configuration.

### VP-Relevant Features

- **Portfolio SLO dashboard**: Single view of all services' SLO status and error budget
- **Burn rate alerts**: VP receives alert when a service's burn rate indicates imminent SLA risk
- **Trend views**: Week-over-week and month-over-month SLO performance comparison

## Tool 3: Grafana (with Prometheus or InfluxDB)

### Role in the SLM Stack

Grafana serves as the flexible visualization layer when the organization's metrics live in Prometheus, InfluxDB, or other open-source time-series databases. It is particularly common in Kubernetes-native environments (T-Mobile/EKS context) where Prometheus is the default metrics system.

### Core Capabilities

**SLO Dashboards**:
- Build custom SLO panels from Prometheus metrics: request success rate, latency histograms, error rates
- Multi-service dashboard with SLO compliance indicators
- Error budget burn rate visualization using PromQL queries

**Prometheus SLO Recording Rules**:
SLO compliance is calculated using Prometheus recording rules that pre-aggregate the SLI data:

```yaml
# Example: Availability SLO recording rule
- record: job:http_request_success:ratio_rate5m
  expr: |
    sum(rate(http_requests_total{status=~"2..|3..|4.."}[5m]))
    /
    sum(rate(http_requests_total[5m]))
```

Multi-window burn rate alerts using Prometheus alerting rules:

```yaml
# Example: SLO burn rate alert
- alert: HighErrorBudgetBurnRate
  expr: |
    (
      job:http_request_success:ratio_rate1h < (1 - 14.4*0.001)
    and
      job:http_request_success:ratio_rate5m < (1 - 14.4*0.001)
    )
  for: 2m
  annotations:
    summary: "High error budget burn rate for {{ $labels.job }}"
```

**Multi-AZ and Service Chain Views**:
- Grafana dashboards can visualize availability across multiple availability zones or regions
- Service dependency graphs can be built using Grafana's node graph panel

### Gaps and Limitations

- **Requires engineering investment to build and maintain**: Grafana dashboards do not build themselves. SLO dashboards require SRE time to build the PromQL queries, configure the panels, and maintain them as the service architecture changes.
- **No built-in SLA management**: Grafana is a visualization tool. It has no concept of SLA commitments, credit schedules, or breach records.
- **Customer access requires security configuration**: Exposing Grafana dashboards to external customers requires authentication and authorization configuration that many organizations do not have in place.

### VP-Relevant Context

Grafana is primarily an engineering tool. The VP's relationship with Grafana is indirect: it powers the VP dashboard through a Grafana snapshot or embedded panel in a custom dashboard. The VP should understand that Grafana's data is only as reliable as the Prometheus scrape configuration — a misconfigured scrape will produce incorrect SLO calculations.

## Tool 4: Atlassian Statuspage

### Role in the SLM Stack

Statuspage is the customer-facing communication layer: it provides the public (or customer-specific) view of current service status, incident history, and scheduled maintenance. It is the "glass door" between the organization's operations and the customer's awareness.

### Core Capabilities

**Component Status Management**:
- Define service components (API, dashboard, notification delivery, etc.)
- Set status per component: Operational / Degraded Performance / Partial Outage / Major Outage
- Update status manually or via API/integration with monitoring tools

**Incident Communication**:
- Create incident records with customer-facing descriptions
- Post updates during incidents (can be set to notify subscribers at each update)
- Mark incidents as resolved with resolution summary

**Scheduled Maintenance Announcements**:
- Create maintenance windows in advance (customers receive advance notice if subscribed)
- Automatically set components to "Maintenance" status during the window
- Send pre-maintenance and post-maintenance notifications

**Historical Uptime Display**:
- Optionally display 90-day historical uptime per component
- This is configurable — organizations can choose to show or hide historical uptime data
- Historical data is based on status updates, not independent measurement

**Customer Subscription**:
- Customers subscribe to receive notifications (email, webhook, Slack)
- Can subscribe to specific components rather than all service updates
- This fulfills the SLA commitment to proactively notify customers of incidents

### Gaps and Limitations

- **Not a measurement tool**: Statuspage status is set by humans (or via monitoring integration). It reflects what the team reports, not independent measurement. If the team is slow to update status, customers see a delayed picture.
- **Historical uptime is not SLA compliance data**: The 90-day uptime shown on Statuspage is based on status updates, which may not match the precise SLI measurement used for SLA compliance calculation. Do not cite Statuspage uptime percentages in breach negotiations without cross-referencing against the authoritative SLI system.
- **No credit management**: Statuspage has no concept of SLA credits or breach records.
- **Limited customization for enterprise accounts**: Statuspage serves a public or semi-public audience. For enterprise customers who want a private, account-specific view, Statuspage requires separate page configuration.

### VP-Relevant Features

- **Update approval workflow**: Some Statuspage plans allow approval workflow for status updates — requiring VP or Director approval before a customer-facing incident update is posted during SLA-impacting events
- **Subscriber management**: The VP can confirm that enterprise customers are subscribed to receive proactive notifications, fulfilling the SLA's breach notification commitment
- **Post-incident summary**: Statuspage's incident resolution summary is the public post-incident communication — the VP should review and approve the language for any Gold-tier service incident

## Tool Integration Architecture

The integrated SLM stack:

```
[Service Instrumentation]
(Application code, Prometheus metrics, Datadog agents)
         ↓ SLI data (request counts, error counts, latency)
[Datadog / Grafana]
(SLO tracking, error budget calculation, engineering dashboards)
         ↓ Availability data (via integration or manual reporting)
[ServiceNow SLM]
(SLA records, breach management, credit calculation, service reports)
         ↓ Incident creation (via ITSM integration)
[Statuspage]
(Customer-facing status, incident communication, subscriber notifications)
```

Each integration point is a potential failure mode:
- Datadog → ServiceNow: if this integration breaks, ServiceNow's SLA compliance tracking becomes inaccurate
- ServiceNow → Statuspage: if incident records in ServiceNow don't trigger Statuspage updates, customer communication lags
- Monitoring alert → Statuspage: if monitoring integration to Statuspage breaks, status updates require manual action

The SLM Manager owns the integration health. The VP should ask quarterly: "Are all integrations operational and validated?"

## Tooling Maturity Progression

| Maturity Level | Tooling State | VP Action |
|---------------|--------------|-----------|
| Level 1 (Ad-hoc) | Spreadsheets for SLA tracking; manual status page updates; no SLO tooling | Fund ServiceNow SLM module and Datadog SLO configuration |
| Level 2 (Defined) | ServiceNow for SLA records; Datadog for SLO; Statuspage for customers | Build integrations between tools; create VP dashboard |
| Level 3 (Governed) | Full integration stack operational; automated breach alerts; VP dashboard live | Invest in customer-facing portal; API access to SLO data |
| Level 4 (Predictive) | ML-enhanced burn rate prediction; real-time compound SLO modeling; customer self-service | Publish SLO data externally; benchmark against industry |
