# Service Catalog Governance Playbook

## Purpose and Scope

The service catalog is the operational backbone of the SLM program. This playbook defines the processes for maintaining it: onboarding new services, reviewing existing entries, and decommissioning services that are no longer active. An unmaintained catalog is worse than no catalog — it sends incident responders and the VP to wrong owners, wrong runbooks, and wrong SLOs.

**Owner**: SLM Manager (operations); VP of Operations (governance)
**Catalog system**: ServiceNow CMDB / Backstage / OpsLevel (whichever is the system of record)
**Mandatory update triggers**: New service to production, owner change, dependency change, SLO change, SLA change, decommission

---

## Process 1: New Service Onboarding

### When This Process Runs

Every time a new service is being deployed to production and will be maintained by the operations team, it must have a catalog entry before it goes live. No service enters production without:
- A catalog entry with all required fields populated
- An approved SLO
- An on-call rotation assigned
- A runbook published and reviewed

This is a production readiness gate — not a post-launch task.

### Step 1.1: Service Owner Completes the Initial Draft

The engineering team's service owner completes the [slo-definition-template.md] and submits to the SLM Manager 30 days before the planned production date.

Required at submission:
- Service name (canonical, lowercase-kebab-case)
- Proposed service tier (Gold/Silver/Bronze)
- Proposed SLO (availability target, latency target if applicable)
- SLI measurement methodology
- Known dependencies (services this service will depend on)
- On-call rotation (must be named and staffed)
- Runbook location (link; runbook must be published before this step completes)

### Step 1.2: SRE Lead Reviews the Submission

Within 5 business days:
- [ ] Verify SLI measurement methodology is technically valid (can actually be measured from existing infrastructure)
- [ ] Verify the proposed SLO is realistic given the architecture and dependency chain
- [ ] Calculate compound SLO including all listed dependencies
- [ ] If SLO is too aggressive relative to architecture: flag and return to Service Owner for revision
- [ ] Confirm on-call rotation has sufficient staffing (minimum 5 people, escalation chain defined)
- [ ] Confirm runbook has been tested (a runbook that has never been run is not a runbook)

### Step 1.3: VP Approves Tier Assignment and SLO (Gold and Silver)

For Gold or Silver tier services:
- [ ] VP reviews proposed tier and SLO
- [ ] VP confirms SLO provides adequate buffer above any planned SLA commitment
- [ ] VP approves or requests revision
- [ ] If service will be covered by an existing customer SLA: Legal must review whether the new service needs to be added to the SLA scope via amendment

For Bronze tier: Engineering Manager approval is sufficient.

### Step 1.4: Create the Catalog Entry

SLM Manager creates the catalog entry with all fields populated:

- Service Name
- Service Owner (named individual)
- Business Owner (named executive)
- Service Tier
- SLO (availability, latency if applicable)
- SLI Method
- Error Budget (calculated from SLO)
- Dependencies
- Depended On By (populated as other services document this dependency)
- On-Call Rotation
- Runbook Location
- SLA Backed (Yes/No; if Yes: which SLA)
- Last SLO Review (date of initial approval)
- Next SLO Review (set for 90 days from go-live)
- Status: Active

### Step 1.5: Notify Stakeholders

- [ ] Notify the Account Manager if this service will be covered by any existing SLA
- [ ] Notify the on-call rotation members that the service is now live and they are accountable
- [ ] Notify the SLM Review Board at the next quarterly meeting (new services added since last meeting)

## Process 2: Quarterly Catalog Audit

### When This Process Runs

Quarterly, timed to run 2 weeks before the SLM Review Board meeting so the audit results can be presented at the board.

### Step 2.1: Export the Full Catalog

SLM Manager exports all active catalog entries. This is the working document for the audit.

### Step 2.2: Verify Every Required Field

For each entry, check:

| Field | Verification Required |
|-------|-----------------------|
| Service Owner | Is this person still with the organization? Still in this role? |
| Business Owner | Is this person still the appropriate business owner? |
| On-Call Rotation | Is the rotation staffed? (Check PagerDuty/OpsGenie for current staffing) |
| Runbook Location | Does the link resolve? Was the runbook updated in the last 12 months? |
| SLO | Does this match the approved SLO definition document? |
| Dependencies | Are there new dependencies that aren't listed? Were any removed? |
| Next SLO Review | Is this date in the future? If past-due: flag |
| SLA Backed | Does this match the current SLA library? Have any SLAs expired? |
| Status | Active? If the service was deprecated or decommissioned, was the catalog updated? |

### Step 2.3: Generate the Audit Report

Audit report contains:
- Number of entries reviewed
- Number of entries with no issues
- Issues identified (by field type): stale owner, broken runbook link, past-due SLO review, etc.
- Services with no SLO review in > 6 months (flag for VP attention)
- Services listed as SLA-backed where the associated SLA has expired or is approaching expiration
- New entries since the last audit

### Step 2.4: Remediate Issues

For each issue:
- Assign to the relevant Service Owner or SLM Manager for resolution
- Set a due date (within 30 days for all non-critical issues; immediate for stale service owners on Gold/Silver services)
- Track completion

### Step 2.5: Present to SLM Review Board

The audit summary is a standing agenda item at the SLM Review Board. The VP reviews:
- Trend: is catalog health improving or degrading?
- High-priority issues: any stale owners on SLA-backed services?
- SLO review backlog: any SLOs that have not been reviewed in > 6 months?

## Process 3: Service Owner Change

When a service owner changes (departure, role change, restructuring), the catalog must be updated within 5 business days.

### Step 3.1: Identify the New Owner

The incoming owner must:
- Understand the service architecture and operational characteristics
- Be enrolled in the on-call rotation (or have a deputy arrangement in place)
- Accept accountability for the SLO

The VP must confirm the new owner for Gold and Silver tier services. A team cannot be "the owner" — a named individual must be accountable.

### Step 3.2: Conduct a Handoff Review

Within 10 business days of the owner change, the SLM Manager facilitates a handoff review:
- Outgoing owner (if available) walks through the service with the incoming owner
- Review: recent incidents, open action items, current SLO status, on-call runbook currency
- Confirm incoming owner has access to monitoring dashboards, PagerDuty rotation, and runbook systems

### Step 3.3: Update the Catalog

- Update Service Owner field
- Record the date and reason for the change in the audit log
- Notify any relevant escalation chains (incident escalation lists, customer-facing escalation contacts)

## Process 4: Dependency Change

When a service adds or removes a significant dependency (another service, a database, an external API), the catalog must be updated and the compound SLO must be recalculated.

### Step 4.1: Trigger

Dependency changes are identified through the change management process. [Cross-reference: change-release-management/] Any change that modifies a service's dependency graph must include a catalog update as a required change step.

### Step 4.2: Update the Catalog

- Add or remove the dependency in the service's catalog entry
- Update the "Depended On By" field for the dependency service (add or remove the dependent service)

### Step 4.3: Recalculate Compound SLO

For any SLA-backed service:
- [ ] Recalculate compound SLO with the new dependency configuration
- [ ] If compound SLO is now within 0.05% of the SLA commitment: flag to VP immediately
- [ ] If compound SLO is now below the SLA commitment: this is an urgent SLM governance issue — escalate to VP and trigger an immediate SLO review

## Process 5: Service Decommission

When a service is being retired, the catalog must be updated before the service is taken offline — not after.

### Step 5.1: Pre-Decommission Checklist

30 days before planned decommission:
- [ ] Confirm no customer SLA currently covers this service; if it does, SLA must be amended before decommission
- [ ] Confirm no other services depend on this service; if they do, their catalog entries must be updated
- [ ] Confirm on-call rotation for this service will be disbanded or reassigned
- [ ] Notify any internal teams with OLAs that reference this service

10 days before planned decommission:
- [ ] Update catalog entry status to "Deprecated"
- [ ] Add a decommission date field with the planned date
- [ ] Notify the SLM Manager to remove this service from the next SLO review cycle

### Step 5.2: On Decommission Day

After the service is confirmed offline:
- [ ] Update catalog entry status to "Sunset"
- [ ] Remove from active SLO tracking
- [ ] Archive the catalog entry (do not delete — retain for SLA dispute reference)
- [ ] Remove from on-call rotation systems

### Step 5.3: SLA Amendment

If the decommissioned service was covered by a customer SLA:
- The SLA must be amended to remove the service from scope
- Legal must draft the amendment
- The amendment must be signed by both parties before the service is taken offline
- Taking a service offline that is still in SLA scope creates an immediate breach

## Catalog Health Metrics (Reported Quarterly)

| Metric | Target | Red Threshold |
|--------|--------|--------------|
| % of services with current Service Owner | 100% | < 95% |
| % of Gold/Silver services with SLO review in last 90 days | 100% | < 90% |
| % of runbook links resolving and dated < 12 months | 95% | < 80% |
| % of SLA-backed services with compound SLO > SLA + 0.02% | 100% | < 100% |
| Days between dependency changes and catalog updates | < 10 days | > 30 days |
| % of new services with catalog entry at go-live | 100% | < 100% |
