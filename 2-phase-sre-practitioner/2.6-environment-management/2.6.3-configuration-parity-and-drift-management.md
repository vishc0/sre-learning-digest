# Chapter 3: Configuration Parity and Drift Management

## The Parity Imperative

Configuration parity is not a nice-to-have. It is the mechanism by which testing produces valid predictions about production behavior. Every configuration deviation between a non-production environment and production is a test invalidity — a test that proves something about an environment that does not exist.

This chapter defines what parity means across each configuration layer, how drift accumulates and is detected, and how Infrastructure-as-Code enforces parity as a structural guarantee rather than a manual discipline.

---

## What Configuration Parity Means Across Each Layer

### Layer 1: Infrastructure

Infrastructure parity means non-production environments use the same compute families, storage classes, and network topologies as production — scaled appropriately.

| Infrastructure Parameter | Production Example | Non-Prod Acceptable | Non-Prod Not Acceptable |
|-------------------------|-------------------|--------------------|-----------------------|
| Instance family | m5.2xlarge (compute-optimized) | m5.large (same family) | t3.micro (burstable — different behavior) |
| Storage type | gp3 NVMe SSD, 3000 IOPS | gp3, 1000 IOPS (lower throughput, same type) | gp2 (different IOPS scaling model) |
| Network topology | VPC with private subnets, NLB | Same VPC design, smaller CIDR | Public subnets with security group allow-all |
| Node count | 20 EKS worker nodes | 5 worker nodes (same instance family) | 1 node (single-node removes failure isolation) |
| Availability Zones | Multi-AZ (3 AZs) | Single-AZ acceptable for Dev/QA; Multi-AZ required for Staging | Single-AZ for Staging |

**Why instance family matters:** A burstable instance (t3) can absorb load spikes that a fixed-performance instance (m5) cannot. Performance tests on t3 produce results that do not predict m5 behavior in production.

### Layer 2: Middleware Configuration

This is the highest-risk configuration layer for T-Mobile-scale delivery organizations. RabbitMQ, Kafka, and TIBCO configuration in non-prod must match production — not just "be present."

| Middleware Parameter | Production | Non-Prod Must Match | Drift Consequence |
|---------------------|-----------|--------------------|--------------------|
| RabbitMQ queue durable flag | true | true | Queue survives broker restart in prod; loses messages on restart in non-prod |
| RabbitMQ prefetch count | 50 | 50 | Consumer throughput behavior differs |
| RabbitMQ dead-letter exchange | Configured | Configured (same DLX name) | Dead-letter routing fails in prod if not tested |
| Kafka partition count | 24 | 24 | Consumer group rebalancing behavior differs |
| Kafka retention.ms | 604800000 (7 days) | 86400000 acceptable (1 day) | Data retention behavior differs; acceptable for cost |
| Kafka max.message.bytes | 10485760 (10MB) | 10485760 | Message size limits must be identical |
| TIBCO EMS queue max depth | 50000 | 50000 | Back-pressure behavior differs |
| Connection pool size | 20 connections | 10 connections acceptable | Throughput ceiling differs; must be documented |

**Director Governance Gate**: Middleware configuration for QA and above tiers must be version-controlled and reviewed as part of environment validation. Any middleware configuration difference between non-prod and production must be documented and approved.

### Layer 3: Application Configuration

Application configuration (env vars, config maps, feature flags, batch sizes, cache TTLs) must be managed through a single source of truth with environment-specific overrides that are explicitly declared and reviewed.

**The configuration source of truth model:**

```
config/
├── base/           ← shared across all environments
│   ├── application.yaml
│   └── middleware.yaml
├── dev/            ← dev-only overrides
├── integration/    ← integration-only overrides
├── qa/             ← qa-only overrides
├── performance/    ← performance-only overrides
├── uat/            ← uat-only overrides
├── staging/        ← staging overrides (should be minimal — staging mirrors prod)
└── production/     ← production values
```

**Governance rule**: Any value in `qa/` or `staging/` that differs from `production/` must have a documented justification. "We don't know why it's different" is not a justification — that is a drift incident.

### Layer 4: Data

Data configuration parity is the hardest to achieve and the most frequently ignored. See ch05 for full test data management governance. The parity requirements at this layer:

| Data Parameter | Production | Non-Prod Requirement | Notes |
|---------------|-----------|---------------------|-------|
| Record count | 10M customers | ≥ 1M for QA, ≥ 5M for Performance | Pagination and aggregation behavior requires representative volume |
| Data shape | All production record variants | All variants must be represented | Edge cases (null fields, max-length strings, special characters) must exist in test data |
| Referential integrity | Enforced | Enforced | Tests that pass on broken references fail in prod |
| PII fields | Real values | Masked — format-preserving | SSN format preserved, value changed; phone format preserved, value changed |
| Data freshness | Live | Refreshed per environment SLA | See ch05 refresh cadence |

## Configuration Drift: How Environments Silently Diverge

Configuration drift is the process by which a once-accurate non-production environment becomes inaccurate over time without any single intentional act. It accumulates through:

1. **Manual hotfixes applied to production but not non-prod**: A production incident requires a config change. The change is applied to production. The post-incident ticket to apply the same change to non-prod is filed but never completed.

2. **New service dependencies added to production without updating non-prod**: Service A now calls Service B in production. Non-prod Service A still points to a stub because nobody updated the environment.

3. **Infrastructure upgrades applied selectively**: Production EKS is upgraded from 1.28 to 1.29. Non-prod clusters are "on the backlog" for 6 months.

4. **Middleware version drift**: Production RabbitMQ is upgraded to 3.12. Non-prod runs 3.10. Behavior differences in the new version are never validated.

5. **Configuration override accumulation**: Small "just for now" overrides accumulate in environment-specific configuration files. Nobody removes them. After 12 months, the environment-specific files contain dozens of deviations, none documented.

**The drift accumulation curve**: Configuration drift is not linear. The first 30 days after environment creation, drift is minimal. After 90 days without active management, drift is moderate. After 12 months, a non-production environment may share less than 60% of its configuration with production — despite being called "production-like."

## Drift Detection: Automated Tooling

Drift detection must be automated. Manual comparison of environment configurations is not reliable at scale.

### Terraform Drift Detection

```bash
# Daily drift detection run — compare current infrastructure state to Terraform-declared state
terraform plan -out=drift-check.tfplan

# If plan shows non-zero changes: drift detected
# Output must be captured and reported to environment owner + platform team
```

**Drift detection pipeline (automated, daily):**

```
Schedule: Daily 6am UTC
→ terraform plan (per environment)
→ Parse plan output: count resource changes
→ If changes > 0:
   → Create JIRA ticket: "Drift Detected: [Environment Name] — [N] resources differ"
   → Assign to environment owner
   → SLA: resolved within 2 business days (QA/Integration), 24h (Staging)
→ Report: drift incident count per environment per week
```

### Configuration Scanning

For application configuration beyond Terraform scope:

| Tool | What It Detects | Implementation |
|------|----------------|----------------|
| Conftest + OPA | Policy violations in Kubernetes manifests, Helm values | Pre-deployment gate in CI pipeline |
| Checkov | Terraform misconfiguration against security and parity policies | Pre-plan CI gate |
| AWS Config Rules | Infrastructure configuration drift in real time | Managed rule: `DESIRED_INSTANCE_TYPE` per environment |
| Custom environment diff script | Application config values that differ from production baseline | Python script comparing config maps across namespaces |

### Environment Diff Reports

Monthly environment diff report: automated comparison of each environment's declared configuration against production. Produced as a structured report:

```
Environment: QA-East
Report Date: [date]
Configuration Source: Terraform state + Helm values + ConfigMaps

--- Infrastructure ---
Instance type: MATCH (m5.large vs m5.2xlarge — documented scale delta)
Storage class: MATCH (gp3)
Network topology: MATCH
EKS version: DRIFT DETECTED (1.27 vs 1.29 in production)

--- Middleware ---
RabbitMQ prefetch count: MATCH (50)
RabbitMQ DLX config: MATCH
Kafka partition count: DRIFT DETECTED (12 vs 24 in production)

--- Application Config ---
Batch size: MATCH (500)
Cache TTL: DRIFT DETECTED (300s vs 60s in production) — no documented justification

Action required: 3 drift items → JIRA tickets created
```

## The "Works in QA, Fails in Prod" Root Cause Framework

Every "works in QA, fails in prod" incident must be classified and used to improve configuration governance:

| Classification | Root Cause | Prevention Action |
|---------------|-----------|-------------------|
| Config value drift | A configuration value in QA differed from production | Add configuration value to drift detection scan |
| Data volume difference | Production data volume triggered behavior not seen in QA | Increase test data volume; add to performance environment test criteria |
| Missing dependency | A service/queue/topic exists in production but not in QA | Update environment provisioning template; add dependency to service catalog |
| Middleware config delta | Middleware setting differed between environments | Add middleware setting to configuration parity checklist |
| Infrastructure behavior difference | Instance type or storage class produced different behavior | Add infrastructure parity requirement to environment SLA |
| Feature flag state | Feature flag was on in production but off in QA | Add feature flag state to environment promotion checklist |

**What the VP measures**: "Works in QA, fails in prod" incident count per quarter. Target: 0. Trend reduction of 25% quarter-over-quarter if baseline > 0.

## Infrastructure-as-Code for Non-Production

The single most effective control for configuration parity is Infrastructure-as-Code applied consistently to non-production environments with the same rigor as production.

**The consistency guarantee**: If production and non-production environments are both declared from the same Terraform module with environment-specific variable files, the configuration structure is identical. Deviations are explicit (variable overrides) and reviewable.

**Terraform module structure for multi-environment environments:**

```
modules/
└── application-environment/
    ├── main.tf           ← environment infrastructure definition
    ├── variables.tf      ← all configurable parameters
    └── outputs.tf        ← exported values

environments/
├── dev/
│   ├── main.tf           ← calls module
│   └── terraform.tfvars  ← dev-specific values
├── qa/
│   ├── main.tf
│   └── terraform.tfvars  ← qa-specific values
├── performance/
│   ├── main.tf
│   └── terraform.tfvars
├── staging/
│   ├── main.tf
│   └── terraform.tfvars  ← minimal deviations from production
└── production/
    ├── main.tf
    └── terraform.tfvars  ← production values
```

**Governance requirement**: Any variable in `qa/terraform.tfvars` that differs from `production/terraform.tfvars` must have a comment with the justification. Undocumented deviations are drift.

**Helm chart parallel**: Same principle applies to Kubernetes application deployment. A single Helm chart with environment-specific values files. Every `values.qa.yaml` override must have a documented justification.

## Environment Promotion Gates: Configuration Validation

Before software can promote from one environment tier to the next, configuration validation must pass. This is not application testing — it is environment configuration validation.

**Promotion gate: Integration → QA**

| Check | Validation | Pass Criteria |
|-------|-----------|--------------|
| Middleware config parity | Compare queue/topic configuration to production baseline | Zero undocumented deviations |
| Infrastructure drift | `terraform plan` shows zero undocumented changes | No undocumented drift |
| Dependency availability | All service dependencies present and healthy | All health checks pass |
| Data freshness | Test data refreshed within SLA window | Last refresh < [tier SLA] hours ago |

**Promotion gate: Staging → Production**

| Check | Validation | Pass Criteria |
|-------|-----------|--------------|
| Full configuration parity | Diff of Staging vs Production config | Zero deviations (exceptions approved at VP level) |
| Secrets rotation current | All Vault secrets rotated within policy window | Pass |
| Feature flag state match | Production feature flag state matches Staging test state | Pass |
| Performance test passed | Performance test results within SLA thresholds | Pass |
| Security scan passed | SAST/DAST/dependency scan | No Critical, no High unmitigated |

## Secrets Management Across Environments

Environment-specific secrets must be managed through a centralized secrets manager (HashiCorp Vault) with environment-scoped access control. Plaintext secrets in environment variable files or ConfigMaps are a compliance violation at any tier.

**Vault path structure:**

```
secret/
├── dev/
│   └── <service-name>/   ← dev service secrets
├── integration/
│   └── <service-name>/
├── qa/
│   └── <service-name>/
├── performance/
│   └── <service-name>/
├── uat/
│   └── <service-name>/
├── staging/
│   └── <service-name>/
└── production/
    └── <service-name>/
```

**Access control rule**: A service's Kubernetes service account in the `qa` namespace has read access to `secret/qa/<service-name>` only. It cannot read `secret/production/<service-name>`. Vault policies enforce this boundary.

**Secret rotation governance**: Non-production secrets must be rotated on the same schedule as production secrets. Unrotated non-production secrets are a compliance exposure if they share values with production secrets (which they should not, but audit must verify).

## Environment Configuration Change Process

Not all environments require the same change control rigor. The change process scales with environment tier.

| Tier | Change Control Level | Approval Required | Notification Required |
|------|---------------------|-------------------|-----------------------|
| Dev | None | None | None |
| Feature Branch | None | None | None |
| Integration | Lightweight | Tech lead | Dev team |
| QA | Standard | QA lead + Tech lead | QA team + release manager |
| Performance | Standard + scheduling | Performance lead + SRE | Performance team |
| UAT | Standard | Product owner + Tech lead | Business stakeholders + QA team |
| Staging | Full CAB process | Change Advisory Board | All delivery teams |
| Production | Full CAB + release gate | Change Advisory Board + VP SRE | All stakeholders |

**Director Governance Gate**: Any change to a Staging environment configuration without CAB approval is a process violation. Staging environment change control is enforced identically to production change control.
