# Environment Incident Playbook

## Purpose

Response framework for environment incidents — situations where a non-production environment is unavailable, degraded, or producing incorrect results, blocking one or more delivery teams.

**Audience**: Environment owners, platform SRE on-call, delivery team leads.

---

## What Is an Environment Incident

An environment incident is any condition that prevents a delivery team from using an environment for its intended purpose. This includes:
- Environment completely unavailable (all services down)
- Environment partially degraded (some services down, blocking test execution)
- Data corruption (test data in a state that invalidates test results)
- Configuration corruption (environment configuration has changed unexpectedly)
- Performance degradation (environment too slow to execute test suite in reasonable time)

**What is NOT an environment incident:**
- A failing deployment that the team triggered (that is a deployment issue, not an environment incident)
- A test failure caused by a code defect (that is a defect, not an environment incident)
- Planned maintenance within a communicated maintenance window

---

## Severity Classification for Environment Incidents

| Severity | Condition | Response SLA |
|----------|-----------|-------------|
| P1 | Staging environment unavailable; active release blocked | 15-minute response; 2-hour MTTR target |
| P1 | UAT environment unavailable during active UAT window | 15-minute response; 2-hour MTTR target |
| P1 | Performance environment unavailable during booked test window | 15-minute response; 1-hour MTTR target |
| P2 | QA environment unavailable; sprint testing blocked | 30-minute response; 4-hour MTTR target |
| P2 | Integration environment unavailable; multiple teams blocked | 30-minute response; 4-hour MTTR target |
| P3 | Dev/shared dev environment unavailable | 2-hour response; next business day MTTR target |
| P3 | Environment degraded but usable | 4-hour response; 2-business-day MTTR target |

## Phase 1: Detection and Initial Triage (0–15 minutes)

### Detection Sources

| Source | How to Detect |
|--------|-------------|
| Automated monitoring | Prometheus alert fires → Slack #env-incidents channel |
| Team-reported | Team posts in #env-incidents or pages environment owner |
| Proactive check | Environment health dashboard shows red |

### Initial Triage (< 5 minutes)

When an environment incident is reported or detected:

**Step 1**: Open the environment's health dashboard (Grafana environment overview)

**Step 2**: Run the 5-question triage:
1. Which environment? (name, tier)
2. What is unavailable? (all services? specific services? data? performance?)
3. Who is blocked? (which teams, which test cycles)
4. When did it start? (check monitoring timeline)
5. Has anything changed recently? (recent deployments, data refreshes, maintenance)

**Step 3**: Classify severity (see table above)

**Step 4**: Page appropriate responders:
- P1: Page platform SRE on-call immediately
- P2: Notify platform SRE on-call via Slack; page if no response in 15 minutes
- P3: Assign to environment owner; no immediate page

---

## Phase 2: Communication to Affected Teams (Within 15 minutes of detection)

Do NOT wait for root cause before communicating. Teams need to know their environment is impacted so they can make scheduling decisions.

**Initial communication template:**

```
[ENV INCIDENT] [Environment Name] — [Tier] — [Severity]

Status: INVESTIGATING

Impact: [Which teams are blocked; what test activity is suspended]
Started: [approximate time]
Responders: [names]

Current understanding: [what we know so far — even "unknown cause, investigating" is better than silence]

Next update: [time — every 30 minutes for P1, every 60 minutes for P2]

Workaround: [if any — e.g., "Team can use integration environment for smoke tests while QA is down"]
```

**Communication channels:**
- Slack: #env-incidents (all severities)
- Slack: #[team channel] for each blocked team (P1/P2)
- Email: [environment owner distribution list] (P1 only)
- Status page update: P1 only; update environment status to "Degraded" or "Unavailable"

## Phase 3: Diagnosis (15–60 minutes)

### Diagnostic Funnel

Start at the infrastructure layer and work up. Do not assume application-layer cause until infrastructure is cleared.

**Layer 1: Kubernetes / Infrastructure**

```bash
# Check node health
kubectl get nodes -o wide

# Check namespace status
kubectl get pods -n ${ENVIRONMENT_NAME}
kubectl describe pods -n ${ENVIRONMENT_NAME} | grep -A 5 "Events:"

# Check resource pressure
kubectl top nodes
kubectl top pods -n ${ENVIRONMENT_NAME}

# Check events in namespace
kubectl get events -n ${ENVIRONMENT_NAME} --sort-by='.lastTimestamp' | tail -20
```

**Layer 2: Application Health**

```bash
# Check service endpoints
for SERVICE in $(kubectl get svc -n ${ENVIRONMENT_NAME} -o name); do
  kubectl exec -n ${ENVIRONMENT_NAME} debug-pod -- curl -s http://${SERVICE}/health
done

# Check application logs
kubectl logs -n ${ENVIRONMENT_NAME} -l app=${SERVICE_NAME} --tail=100 | grep -i "error\|exception\|fatal"
```

**Layer 3: Data Layer**

```bash
# Check database connectivity
kubectl exec -n ${ENVIRONMENT_NAME} ${DB_CLIENT_POD} -- \
  mysql -h ${DB_HOST} -u ${DB_USER} -p${DB_PASS} -e "SELECT 1;"

# Check record counts (basic data integrity)
./scripts/validate-data.sh --environment ${ENVIRONMENT_NAME} --quick
```

**Layer 4: Configuration**

```bash
# Check for unexpected configuration changes (compare to last known good)
kubectl get configmaps -n ${ENVIRONMENT_NAME} -o yaml > /tmp/current-config.yaml
diff /tmp/current-config.yaml configs/last-known-good/${ENVIRONMENT_NAME}.yaml

# Check Vault secret availability
vault kv list secret/${TIER}/${ENVIRONMENT_NAME}
```

**Layer 5: Recent Changes**

```bash
# What changed in the last 24 hours?
kubectl rollout history deployment -n ${ENVIRONMENT_NAME}
git log --since="24 hours ago" -- environments/${TIER}/${ENVIRONMENT_NAME}/
```

## Phase 4: Remediation

### Common Remediation Actions by Root Cause

| Root Cause | Remediation Action | Validation |
|-----------|-------------------|-----------|
| Pod crashloop (OOM) | Increase memory limit; restart pod | `kubectl get pods -n ENV` — all Running |
| Pod crashloop (startup failure) | Check logs; fix config or image; redeploy | `kubectl get pods -n ENV` — all Running |
| Node pressure (CPU/memory) | Cordon and drain affected node; reschedule | `kubectl get nodes` — all Ready |
| Data corruption | Roll back to last known good data snapshot | `./scripts/validate-data.sh` — PASS |
| Configuration drift (unexpected change) | Revert to IaC-declared configuration; `terraform apply` | `./scripts/environment-diff.sh` — no deviations |
| Networking failure (DNS) | Check CoreDNS pod health; restart if unhealthy | DNS resolution test passes |
| Secrets unavailable (Vault) | Check Vault cluster health; renew expired tokens | Service starts and can read secrets |
| Database unavailable | Restart database pod; check PVC health | DB connection test passes |
| Helm chart deployment failed mid-apply | `helm rollback` to previous release | All services healthy |

### Escalation Decision

If root cause not identified within:
- **P1**: 30 minutes → escalate to platform SRE lead + VP SRE
- **P2**: 2 hours → escalate to platform SRE lead
- **P3**: Next business day → schedule with platform team

If remediation will take > MTTR target:
- Communicate revised ETA to all affected teams
- Consider whether a workaround environment can be stood up faster than remediating the current one

## Phase 5: Recovery Validation

Before declaring the environment recovered:

```bash
# Full health check suite
./scripts/env-health-check.sh --environment ${ENVIRONMENT_NAME}

# Data integrity check
./scripts/validate-data.sh --environment ${ENVIRONMENT_NAME}

# Service-to-service smoke test
./scripts/integration-smoke-test.sh --environment ${ENVIRONMENT_NAME}
```

All checks must pass before declaring recovery. If any check fails, continue remediation.

**Recovery communication:**

```
[ENV RESOLVED] [Environment Name] — [Tier]

Status: RECOVERED

Impact duration: [start time] → [end time] ([total duration])
Root cause: [brief summary]
Resolution: [what was done]

Environment is ready for use.

RCA: Will be completed within [SLA based on severity]
JIRA: [link to RCA ticket]
```

## Phase 6: Root Cause Analysis

**RCA SLA:**

| Severity | RCA Due |
|----------|---------|
| P1 | Within 48 hours of resolution |
| P2 | Within 5 business days |
| P3 | Within 10 business days |

**Minimum RCA content:**

| Section | Required Content |
|---------|----------------|
| Timeline | Detection → triage → diagnosis → remediation → recovery with timestamps |
| Root cause | Specific, not "environment issue" — e.g., "Memory limit set to 512Mi in Helm chart; service OOMKilled under load" |
| Impact | Which teams blocked; how many test hours lost; any release impact |
| Detection gap | Why was this not caught by monitoring before teams were impacted? |
| Prevention action | Specific change that prevents recurrence; JIRA ticket with owner and due date |
| Detection improvement | Monitoring or alerting change that would detect this faster next time |

**RCA output**: JIRA ticket with all sections completed. Linked to environment incident ticket. Prevention action tracked to completion.

---

## Lessons Learned Tracking

Monthly environment incident review:
- Count incidents by tier, by root cause category
- Identify recurring root causes (same cause > once = systemic issue)
- Review prevention action completion rate from prior month's RCAs
- Present trend to VP SRE and engineering directors

**Director Governance Gate**: Any root cause that produces > 2 incidents in a quarter requires a systemic remediation plan (not a per-incident fix). The environment manager presents the plan at the quarterly governance review.
