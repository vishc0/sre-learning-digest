# Environment Provisioning Playbook

## Purpose

Step-by-step execution guide for the environment provisioning process: from environment request submission through validation and team handoff. This playbook covers both the automated pipeline flow and the manual verification gates.

**Audience**: Platform team engineers, environment managers, delivery team leads requesting environments.

---

## Prerequisites

Before executing this playbook, confirm:
- [ ] Environment request template completed (all required fields)
- [ ] Requestor has been authenticated in the Internal Developer Portal (IDP)
- [ ] Requesting team has available budget (cost allocation check passed)
- [ ] Required environment tier is available (no current capacity freeze)
- [ ] For UAT/Staging: change window is open and approved

---

## Phase 1: Request Validation (0–2 minutes, automated)

**Automated checks run on request submission:**

| Check | Pass Criteria | Fail Action |
|-------|--------------|------------|
| Required fields complete | All mandatory fields in request template populated | Return to requestor with field list |
| Tier authorization | Requestor role authorized for requested tier | Reject with escalation path |
| Team budget available | Current month spend < 90% of team budget | Alert; continue for < 100%; block at 100% |
| Cluster capacity | Requested compute within available cluster capacity | Queue request; notify requestor of estimated wait |
| Naming convention | Environment name matches `<tier>-<team>-<purpose>` pattern | Auto-correct or reject |
| Compliance scope declared | PCI/HIPAA/None field populated | Reject until declared |

**If all checks pass**: Request enters provisioning queue. Requestor receives confirmation notification with estimated ready time.

**If any check fails**: Request rejected with specific failure reason. Requestor notified within 2 minutes.

## Phase 2: Approval (0 minutes for automated tiers; up to 1 hour for UAT+)

| Tier | Approval Required | Approver | SLA |
|------|-----------------|---------|-----|
| Feature Branch | None — auto-approved | N/A | 0 min |
| Dev | None — auto-approved | N/A | 0 min |
| Integration | None — auto-approved | N/A | 0 min |
| QA | None — auto-approved | N/A | 0 min |
| Performance | Platform SRE capacity check | Platform SRE on-call | < 15 min |
| UAT | Product owner (via IDP) + Platform team | Product owner + Platform SRE | < 1 hour |
| Staging | Change Advisory Board pre-approval | CAB | < 4 hours (next CAB window) |

**Approval notification**: Approvers receive notification via Slack + email. Approval or rejection via IDP portal or Slack slash command (`/env approve <request-id>`).

## Phase 3: Infrastructure Provisioning (5–15 minutes, automated)

Executed by the provisioning pipeline (GitHub Actions / Tekton) after approval.

### Step 3.1: Namespace Creation

```bash
# Kubernetes namespace creation
kubectl create namespace ${ENVIRONMENT_NAME}

# Apply resource quotas (from tier template)
kubectl apply -f quotas/${TIER}-quota.yaml -n ${ENVIRONMENT_NAME}

# Apply network policies (from tier template)
kubectl apply -f network-policies/${TIER}-netpol.yaml -n ${ENVIRONMENT_NAME}

# Apply RBAC (team-scoped roles)
kubectl apply -f rbac/${TEAM}-roles.yaml -n ${ENVIRONMENT_NAME}
```

**Verification**: `kubectl get namespace ${ENVIRONMENT_NAME}` — status must be `Active`.

### Step 3.2: Infrastructure Provisioning (Terraform)

```bash
cd environments/${TIER}/${ENVIRONMENT_NAME}
terraform init -backend-config="key=${ENVIRONMENT_NAME}/terraform.tfstate"
terraform apply -var-file="${TIER}.tfvars" -var="environment_name=${ENVIRONMENT_NAME}" -auto-approve
```

**Verification**: Terraform apply exits with code 0. All resources in `terraform show` show as `tainted: false`.

**On failure**: Terraform plan captured, error logged, provisioning ticket created, requestor notified. Platform on-call paged if failure is infrastructure-layer.

### Step 3.3: Vault Secret Provisioning

```bash
# Create environment-scoped secret path
vault secrets enable -path=secret/${TIER}/${ENVIRONMENT_NAME} kv-v2

# Seed environment secrets from template
./scripts/seed-vault-secrets.sh --tier ${TIER} --environment ${ENVIRONMENT_NAME} --team ${TEAM}

# Bind Kubernetes service account to Vault policy
vault write auth/kubernetes/role/${ENVIRONMENT_NAME} \
  bound_service_account_names=default \
  bound_service_account_namespaces=${ENVIRONMENT_NAME} \
  policies=${TIER}-${TEAM}-policy \
  ttl=1h
```

**Verification**: `vault kv list secret/${TIER}/${ENVIRONMENT_NAME}` returns expected secret keys.

### Step 3.4: Service Deployment (Helm)

```bash
# Deploy all services from request's service list
for SERVICE in ${SERVICE_LIST}; do
  helm install ${SERVICE} charts/${SERVICE} \
    -n ${ENVIRONMENT_NAME} \
    -f values/${SERVICE}/values.${TIER}.yaml \
    --set global.environment=${ENVIRONMENT_NAME} \
    --set global.team=${TEAM} \
    --wait --timeout=5m
done
```

**Verification**: `kubectl get pods -n ${ENVIRONMENT_NAME}` — all pods in `Running` state. No pods in `CrashLoopBackOff` or `Error`.

## Phase 4: Data Provisioning (5–20 minutes, automated)

### For synthetic data (Feature Branch, Dev, Integration):

```bash
./scripts/seed-synthetic-data.sh \
  --environment ${ENVIRONMENT_NAME} \
  --tier ${TIER} \
  --record-count ${SYNTHETIC_RECORD_COUNT}
```

**Expected output**: `Synthetic data seeded: [N] customers, [N] transactions, [N] reference records`

### For masked production snapshot (QA, UAT, Performance, Staging):

```bash
# Pull latest masked snapshot from S3
aws s3 cp s3://masked-snapshots/${SERVICE}/latest.sql.gz /tmp/data-restore/

# Restore to environment database
./scripts/restore-snapshot.sh \
  --environment ${ENVIRONMENT_NAME} \
  --snapshot /tmp/data-restore/latest.sql.gz \
  --database-host ${DB_HOST}
```

**Expected output**: `Restore complete: [N] records loaded. Integrity check: PASS`

**Verification**:
```bash
./scripts/validate-data.sh --environment ${ENVIRONMENT_NAME}
# Expected: All record counts within 5% of expected; referential integrity check PASS
```

## Phase 5: Configuration Parity Validation (2–5 minutes, automated)

```bash
# Run environment diff against production baseline
./scripts/environment-diff.sh \
  --environment ${ENVIRONMENT_NAME} \
  --baseline production \
  --tier ${TIER}
```

**Expected output**: Diff report with no undocumented deviations.

**Pass criteria**:
- Infrastructure diff: zero undocumented resource differences
- Middleware diff: zero configuration value differences (documented scale differences only)
- Application config diff: all deviations from production have documented justification in `values.${TIER}.yaml`

**On drift detection**:
- < 3 minor deviations (documented): Continue; deviations noted in provisioning ticket
- Any undocumented deviation: Block handoff; notify environment manager; create drift ticket

---

## Phase 6: Health Validation (2–5 minutes, automated)

```bash
# Run environment health check suite
./scripts/env-health-check.sh --environment ${ENVIRONMENT_NAME}
```

**Health checks executed:**

| Check | Command | Pass Criteria |
|-------|---------|--------------|
| All pods running | `kubectl get pods -n ${ENV}` | 0 pods not in Running state |
| Service endpoints reachable | HTTP GET /health on each service | HTTP 200 response |
| Database connectivity | DB connection test per service | All pass |
| Message broker connectivity | Queue/topic health check | All queues/topics exist and accessible |
| Service-to-service communication | Integration smoke test | All inter-service calls succeed |
| Data integrity | Record count validation | All counts within 5% of expected |

**On health check failure**:
- Document failing check and output
- Page platform on-call if infrastructure-layer
- Notify requestor of delay with estimated resolution time
- Do NOT proceed to handoff with failing health checks

## Phase 8: Post-Provisioning Registration (automated)

Environment automatically registered in the environment inventory:

```bash
./scripts/register-environment.sh \
  --name ${ENVIRONMENT_NAME} \
  --tier ${TIER} \
  --owner ${PRIMARY_OWNER} \
  --backup-owner ${BACKUP_OWNER} \
  --decommission-date ${DECOMMISSION_DATE} \
  --cost-center ${COST_CENTER} \
  --compliance-scope ${COMPLIANCE_SCOPE}
```

**Verification**: Environment appears in IDP environment catalog within 5 minutes.
