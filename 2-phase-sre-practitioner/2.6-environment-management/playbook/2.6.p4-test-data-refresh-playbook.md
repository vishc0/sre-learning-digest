# Test Data Refresh Playbook

## Purpose

Step-by-step execution guide for refreshing test data in non-production environments from production snapshots through the anonymization pipeline. Covers both the automated nightly pipeline and the on-demand refresh triggered by environment owners.

**Audience**: Data engineering on-call, environment owners, platform SRE.

**Compliance requirement**: This playbook must be followed for every non-prod data refresh. Deviations require data governance approval. Shortcuts that skip masking validation are a compliance incident.

---

## Prerequisites

Before executing a data refresh:
- [ ] Environment owner has confirmed it is safe to refresh (no active test execution in progress)
- [ ] Data refresh change notification sent to all teams using the environment (minimum 4 hours notice for QA; 24 hours for UAT)
- [ ] Rollback point confirmed: current data state can be restored if refresh fails
- [ ] Masking pipeline last-successful-run timestamp is < 24 hours old
- [ ] Target environment's compliance scope documented (PCI / HIPAA / None) — determines masking rule set

---

## Phase 1: Production Snapshot (Automated Nightly — 2:00 AM UTC)

This phase is fully automated. Environment owners do not execute this phase; they consume its output.

### Step 1.1: Snapshot Creation

```bash
# RDS snapshot (for relational databases)
aws rds create-db-snapshot \
  --db-instance-identifier prod-primary \
  --db-snapshot-identifier snap-$(date +%Y%m%d) \
  --tags Key=Purpose,Value=masking-pipeline

# DynamoDB export (for NoSQL tables)
aws dynamodb export-table-to-point-in-time \
  --table-arn arn:aws:dynamodb:us-east-1:${ACCOUNT}:table/prod-events \
  --s3-bucket prod-snapshots-encrypted \
  --s3-prefix events/$(date +%Y%m%d)/

# Cassandra snapshot (if applicable)
nodetool snapshot -t snap-$(date +%Y%m%d) keyspace_name
```

### Step 1.2: Snapshot Export to S3 (Encrypted)

```bash
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier masking-pipeline-restore-$(date +%Y%m%d) \
  --db-snapshot-identifier snap-$(date +%Y%m%d) \
  --db-instance-class db.r5.large \
  --no-publicly-accessible

# Wait for restore to complete (~15-20 minutes)
aws rds wait db-instance-available \
  --db-instance-identifier masking-pipeline-restore-$(date +%Y%m%d)
```

**Monitoring**: CloudWatch alert if snapshot creation fails. Data engineering on-call paged. Previous day's snapshot used as fallback.

## Phase 2: Masking Pipeline Execution (Automated — Runs After Phase 1)

### Step 2.1: PII Detection Audit (Presidio Scan)

Before masking, scan for known PII patterns to establish a baseline of what must be masked:

```bash
python3 scripts/presidio-scan.py \
  --source masking-pipeline-restore-$(date +%Y%m%d) \
  --output /tmp/pii-audit-$(date +%Y%m%d).json \
  --rules configs/pii-rules-${COMPLIANCE_SCOPE}.yaml
```

**Expected output**: JSON report listing all columns containing detected PII entities, by table and column name.

**Action on unexpected PII detected in previously unmasked column**: Stop pipeline; alert data governance team; do not proceed until masking rule is added for the new column.

### Step 2.2: Apply Masking Transformations (AWS Glue / Spark job)

```bash
# Submit masking job
aws glue start-job-run \
  --job-name pii-masking-pipeline \
  --arguments \
    --source-db=masking-pipeline-restore-$(date +%Y%m%d) \
    --target-bucket=s3://masked-snapshots/ \
    --masking-config=configs/masking-rules-${COMPLIANCE_SCOPE}.yaml \
    --run-date=$(date +%Y%m%d)
```

**Masking rules applied (example):**

| Table | Column | Masking Method | Rule |
|-------|--------|---------------|------|
| customers | ssn | FPE (format-preserving encryption) | Valid SSN format, unique per customer |
| customers | phone | FPE | Valid 10-digit US format, non-assigned range |
| customers | email | Substitution | `test-{customer_id}@testdomain.example.com` |
| customers | first_name | Substitution | Random from approved name list |
| customers | last_name | Substitution | Random from approved name list |
| billing | credit_card_pan | FPE | Luhn-valid, non-real BIN |
| billing | billing_address | Substitution | Synthetic address, valid format |
| transactions | account_number | FPE | Same length, same check digit format |
| events | ip_address | Substitution | 192.0.2.x (RFC 5737 documentation range) |

**Expected duration**: 45–90 minutes depending on data volume.

### Step 2.3: Post-Masking PII Validation (Presidio Scan on Masked Output)

```bash
python3 scripts/presidio-scan.py \
  --source s3://masked-snapshots/$(date +%Y%m%d)/ \
  --output /tmp/pii-validation-$(date +%Y%m%d).json \
  --rules configs/pii-rules-${COMPLIANCE_SCOPE}.yaml \
  --expected-findings 0
```

**Pass criteria**: Zero PII entities detected in masked output.

**On any PII finding in masked output**:
1. STOP — do not proceed to distribution
2. Identify which column was not masked correctly
3. Update masking rule
4. Re-run masking pipeline from Step 2.2
5. Alert data governance team of finding
6. Document in compliance log

### Step 2.4: Data Integrity Validation

```bash
python3 scripts/data-integrity-check.py \
  --source s3://masked-snapshots/$(date +%Y%m%d)/ \
  --expected-counts configs/expected-record-counts.yaml \
  --tolerance 0.05
```

**Checks performed:**
- Record count per table within 5% of expected (based on production yesterday)
- Referential integrity: no orphaned foreign keys
- Non-null constraints: required fields are populated
- Format validation: masked values pass format checks (phone format, email format, etc.)

**Pass criteria**: All checks pass. Snapshot marked as `validated` in the snapshot registry.

## Phase 3: On-Demand Non-Prod Data Refresh (Triggered by Environment Owner)

### Step 3.1: Pre-Refresh Verification

Environment owner confirms before triggering refresh:

```bash
# Check no active test runs against the environment
./scripts/check-active-tests.sh --environment ${ENVIRONMENT_NAME}
# Expected: "No active test sessions detected"

# Confirm masking pipeline succeeded today
aws glue get-job-runs --job-name pii-masking-pipeline \
  --query 'JobRuns[0].{Status: JobRunState, Completed: CompletedOn}' \
  --output table
# Expected: Status=SUCCEEDED, CompletedOn=today
```

### Step 3.2: Create Rollback Point

```bash
# Export current environment database state before overwriting
./scripts/export-env-data.sh \
  --environment ${ENVIRONMENT_NAME} \
  --output s3://env-rollback-snapshots/${ENVIRONMENT_NAME}/pre-refresh-$(date +%Y%m%d-%H%M%S)/

echo "Rollback snapshot created. If refresh fails, run:"
echo "./scripts/restore-rollback.sh --environment ${ENVIRONMENT_NAME} --snapshot pre-refresh-$(date +%Y%m%d-%H%M%S)"
```

### Step 3.3: Load Masked Snapshot

```bash
# Pull masked snapshot from S3
aws s3 sync \
  s3://masked-snapshots/latest/ \
  /tmp/data-restore/${ENVIRONMENT_NAME}/ \
  --sse aws:kms

# Restore to environment database
./scripts/restore-snapshot.sh \
  --environment ${ENVIRONMENT_NAME} \
  --snapshot-dir /tmp/data-restore/${ENVIRONMENT_NAME}/ \
  --database-host ${DB_HOST} \
  --database-name ${DB_NAME}
```

**Expected output:**

```
Restoring snapshot to qa-east-payments...
  Dropping existing tables... Done
  Creating schema... Done
  Loading customers (2,341,892 records)... Done [4m 12s]
  Loading transactions (18,923,445 records)... Done [12m 34s]
  Loading reference data (47,230 records)... Done [0m 45s]
  Creating indexes... Done [2m 10s]
Restore complete. Total: 21,312,567 records in 19m 41s
```

### Step 3.4: Post-Refresh Validation

```bash
# Run data integrity check on restored environment
./scripts/validate-data.sh --environment ${ENVIRONMENT_NAME} --full

# Run PII spot-check on restored data
python3 scripts/presidio-scan.py \
  --source ${ENVIRONMENT_NAME} \
  --sample-size 10000 \
  --expected-findings 0
```

**Pass criteria:**
- Record counts within 5% of expected
- All referential integrity checks pass
- PII scan on sample returns zero findings

**On failure**: Execute rollback from Step 3.2 snapshot. Notify environment owner. Create data engineering ticket.

### Step 3.5: Post-Refresh Notification

```bash
# Notify environment owner and all team subscribers
./scripts/notify-data-refresh-complete.sh \
  --environment ${ENVIRONMENT_NAME} \
  --record-count ${TOTAL_RECORDS} \
  --snapshot-date ${SNAPSHOT_DATE} \
  --duration ${REFRESH_DURATION_MINUTES}
```

**Notification content:**

```
Data refresh complete: [ENVIRONMENT_NAME]

Snapshot date: [date]
Records loaded: [N]
Refresh duration: [N] minutes
Data integrity: PASS
PII validation: PASS (0 findings)

Environment is ready for testing.
```

## Rollback Procedure

If data refresh fails at any step after Step 3.2:

```bash
# Identify most recent pre-refresh snapshot
aws s3 ls s3://env-rollback-snapshots/${ENVIRONMENT_NAME}/ | sort | tail -5

# Restore from rollback snapshot
./scripts/restore-rollback.sh \
  --environment ${ENVIRONMENT_NAME} \
  --snapshot ${ROLLBACK_SNAPSHOT_ID}

# Validate rollback
./scripts/validate-data.sh --environment ${ENVIRONMENT_NAME} --quick
```

## Compliance Log

Every data refresh must be logged in the compliance tracking system:

```bash
./scripts/log-data-refresh.sh \
  --environment ${ENVIRONMENT_NAME} \
  --snapshot-date ${SNAPSHOT_DATE} \
  --masking-pipeline-run-id ${GLUE_RUN_ID} \
  --pii-validation-status PASS \
  --record-count ${TOTAL_RECORDS} \
  --executed-by ${EXECUTOR_NAME} \
  --trigger automated|on-demand
```

Compliance log entries are retained for 7 years per audit requirements.
