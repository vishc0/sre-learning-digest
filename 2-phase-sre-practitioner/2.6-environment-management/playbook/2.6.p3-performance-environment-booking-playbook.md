# Performance Environment Booking Playbook

## Purpose

Process guide for reserving, configuring, executing, and archiving results from the performance environment. The performance environment is a shared, limited resource. This playbook governs its use to prevent test result contamination, configuration disputes, and capacity conflicts.

**Audience**: Performance engineers, SRE leads, delivery team leads requesting performance validation.

---

## Performance Environment Reservation

### Reservation Request Requirements

A performance environment reservation must be submitted at least 5 business days in advance (except emergency reservations).

**Required information:**

| Field | Required | Details |
|-------|----------|---------|
| Team name | Yes | Team requesting the environment |
| Use case | Yes | What are you testing? One sentence |
| Test type | Yes | Load / Stress / Soak / Spike / Chaos / Baseline |
| Start time | Yes | UTC; must fall within allowed scheduling windows |
| Duration | Yes | Minimum 4 hours; maximum 72 hours (soak test requires SRE approval) |
| Services under test | Yes | List of services that will receive load |
| Expected load profile | Yes | Peak RPS or msg/sec; ramp-up duration; steady-state duration |
| Load generator(s) | Yes | Which tool (k6, JMeter, Gatling, Locust); from which network location |
| Data requirements | Yes | Standard masked snapshot (latest) OR specific versioned snapshot |
| Configuration requirements | Yes | Any deviations from standard configuration (requires pre-approval) |
| Results storage path | Yes | Where will results be archived |
| Cancellation commitment | Yes | Agree to cancel via portal ≥ 2 hours before start if no longer needed |

**Reservation via IDP portal or Slack**: `/env book perf --type load --start 2026-07-10T14:00:00Z --duration 8h --team payments`

### Scheduling Windows

Performance tests may only run during approved scheduling windows to prevent calendar conflicts and ensure platform team coverage:

| Window | Days | UTC Hours | Notes |
|--------|------|----------|-------|
| Standard | Monday–Friday | 14:00–22:00 UTC | Platform SRE on-call coverage |
| Extended | Saturday | 10:00–22:00 UTC | On-call coverage; requires 7-day notice |
| Soak test | Any | 22:00 UTC start | 24–72 hour tests; platform SRE approval required |
| Emergency | Any | Any | VP SRE approval + 48-hour notice |

### Reservation Confirmation

Within 24 hours of submission, reservation team receives:
- Confirmation of time slot
- Confirmation that performance environment configuration is current (parity validated)
- Pre-test checklist
- Load generator access credentials for test window
- Results storage path pre-created

## Pre-Test Checklist (Must Complete Before Test Start)

All items must be verified and acknowledged ≥ 1 hour before test start.

### Configuration Parity Verification

```bash
# Confirm performance environment matches production
./scripts/environment-diff.sh \
  --environment perf-global \
  --baseline production \
  --tier performance

# Expected: Zero undocumented deviations
# Any deviation found within 48 hours of test must be resolved or test must be postponed
```

**Configuration items requiring explicit verification:**

| Item | Verify Command | Expected Result |
|------|---------------|----------------|
| JVM heap settings | `kubectl get pod -n perf-global <service> -o yaml \| grep -A2 JAVA_OPTS` | Matches production |
| Connection pool size | `kubectl get configmap -n perf-global <service>-config -o yaml` | Matches production |
| Kafka partition count | `kafka-topics.sh --describe --topic <topic>` | Matches production |
| RabbitMQ prefetch | `rabbitmqctl list_consumers` | Matches production |
| Database connection pool | Check HikariCP metrics endpoint | Matches production |

### Data Volume Verification

```bash
# Verify data volume meets minimum requirements
./scripts/validate-data.sh --environment perf-global --volume-check

# Expected:
# customers: 5,234,892 (≥ 50% of production: 10,469,784) ✓
# transactions: 89,234,512 (≥ 50% of production: 178,469,025) ✓
# reference_data: 47,230 (= 100% of production: 47,230) ✓
```

If data volume is below minimum: Trigger data refresh (see [Test Data Refresh Playbook](test-data-refresh-playbook.md)). This requires 2–3 hours. Factor into test scheduling.

### Network Isolation Verification

```bash
# Confirm performance environment VPC is not routing to production
aws ec2 describe-route-tables --filters "Name=vpc-id,Values=${PERF_VPC_ID}" \
  --query 'RouteTables[*].Routes[*].{Destination: DestinationCidrBlock, Target: GatewayId}'

# Confirm no VPC peering to production VPC
aws ec2 describe-vpc-peering-connections \
  --filters "Name=accepter-vpc-info.vpc-id,Values=${PERF_VPC_ID}"
# Expected: empty result (no peering to production)
```

### Load Generator Readiness

```bash
# Validate load generator connectivity to performance environment
curl -s https://perf-gateway.internal/health
# Expected: HTTP 200

# Validate load generator can inject traffic at expected rate (ramp test)
k6 run --vus 10 --duration 30s scripts/ramp-test.js
# Expected: Zero errors at low load; confirms connectivity
```

## During Test Execution

### Test Monitoring Requirements

During active performance test execution, a designated test monitor must be watching:

**Monitor responsibilities:**
- Watch the performance environment Grafana dashboard (auto-populated for test)
- Watch load generator output for error rate anomalies
- Watch application logs for exceptions or error spikes
- Watch infrastructure metrics for node pressure or storage saturation

**Stop-test criteria (abort the test run immediately if any of these occur):**

| Condition | Action |
|-----------|--------|
| Error rate > 5% of requests | Stop test; investigate; do not continue under error |
| Any pod enters CrashLoopBackOff | Stop test; debug; restart if cause is identified; resume |
| Database connection pool exhaustion | Stop test; environment may need configuration correction |
| Node CPU > 95% for > 5 minutes | Stop test; risk of cascading failure that corrupts results |
| Out-of-memory kill on any pod | Stop test; heap configuration may need adjustment |
| Production traffic detected in performance environment | Stop test immediately; escalate to network engineering |

### Maintaining Test Isolation During Execution

**No changes allowed while test is running:**
- No deployments to any service in the performance environment
- No configuration changes to any service
- No data refreshes
- No other teams' tests in the performance environment (exclusive reservation)

**Enforcement**: Deployment pipeline checks performance environment booking calendar. Deployment is blocked if active test is registered.

## Post-Test: Results Archival

Results must be archived within 2 hours of test completion.

### Step 1: Export Raw Results

```bash
# Export load generator results
k6 run --out json=/tmp/results/raw-results.json scripts/load-test.js

# Export Grafana dashboard data for test window
./scripts/export-grafana-data.sh \
  --dashboard performance-environment \
  --start ${TEST_START_TIME} \
  --end ${TEST_END_TIME} \
  --output /tmp/results/grafana-export/
```

### Step 2: Generate Summary Report

```bash
./scripts/generate-perf-report.sh \
  --raw-results /tmp/results/raw-results.json \
  --grafana-data /tmp/results/grafana-export/ \
  --sla-thresholds configs/sla-thresholds/${SERVICE}.yaml \
  --output /tmp/results/summary-report.md
```

**Summary report must include:**

| Metric | Value Captured | SLA Threshold | Pass/Fail |
|--------|---------------|--------------|----------|
| p50 latency (ms) | [measured] | < [threshold] | P/F |
| p95 latency (ms) | [measured] | < [threshold] | P/F |
| p99 latency (ms) | [measured] | < [threshold] | P/F |
| Error rate (%) | [measured] | < 0.1% | P/F |
| Peak RPS achieved | [measured] | ≥ [target] | P/F |
| Message throughput (msg/sec) | [measured] | ≥ [target] | P/F |
| CPU utilization at peak | [measured] | < 80% | P/F |
| Memory utilization at peak | [measured] | < 85% | P/F |

### Step 3: Trend Delta Calculation

```bash
# Compare to previous test results for same service
./scripts/calculate-trend-delta.sh \
  --current /tmp/results/summary-report.md \
  --previous s3://performance-results/${SERVICE}/latest/summary-report.md \
  --output /tmp/results/trend-delta.md
```

**Trend delta flags:**
- p99 latency increased > 10% vs. previous: flag for investigation
- Error rate increased > 0.05% vs. previous: flag for investigation
- Throughput ceiling decreased > 10% vs. previous: flag for investigation

### Step 4: Archive to S3

```bash
aws s3 sync /tmp/results/ \
  s3://performance-results/${SERVICE}/${TEST_DATE}-${TEST_TYPE}/ \
  --sse aws:kms

# Update "latest" pointer
aws s3 sync /tmp/results/ \
  s3://performance-results/${SERVICE}/latest/ \
  --sse aws:kms --delete
```

### Step 5: Post-Test Cleanup

```bash
# Scale down performance environment services after test
kubectl scale deployment --all --replicas=0 -n perf-global

# Clear performance test data (prevent reuse of test-state-polluted data)
./scripts/clear-test-state.sh --environment perf-global

# Release reservation in booking system
./scripts/release-perf-reservation.sh --reservation-id ${RESERVATION_ID}
```

**Post-test notification:**

```
Performance Test Complete: [Service Name] — [Test Type]

Result: [PASS / FAIL]
Duration: [actual test duration]
Peak load achieved: [RPS or msg/sec]

SLA Results:
  p50 latency: [N]ms (threshold: [N]ms) [PASS/FAIL]
  p95 latency: [N]ms (threshold: [N]ms) [PASS/FAIL]
  p99 latency: [N]ms (threshold: [N]ms) [PASS/FAIL]
  Error rate: [N]% (threshold: 0.1%) [PASS/FAIL]

Trend vs. previous: [+N% / -N% / no significant change]
Results archived: s3://performance-results/[path]

Release gate status: [APPROVED for production / BLOCKED — performance criteria not met]
```

## Emergency Reservation Process

When a production incident reveals a performance characteristic that must be validated before the next release:

1. **Request**: Team lead submits emergency reservation request to VP SRE via Slack with justification
2. **Approval**: VP SRE approves within 2 hours; notifies platform SRE on-call
3. **Slot**: Platform SRE assigns earliest available slot (may interrupt scheduled maintenance windows with VP SRE approval)
4. **Execution**: Standard pre-test checklist still required; no exceptions on configuration parity or data volume
5. **Results**: Archived within 2 hours; included in production incident RCA

Emergency reservations do not bypass the pre-test checklist. A performance test without configuration parity verification is not a performance test.
