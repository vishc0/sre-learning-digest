# Chapter 4: Environment Lifecycle Management

## The Lifecycle

Every environment has a lifecycle: it is requested, provisioned, used, maintained, and eventually decommissioned. Without lifecycle governance, environments accumulate unbounded — provisioned on demand, never decommissioned, drifting further from production with each passing sprint. The environment inventory becomes an untended garden that consumes compute budget and creates delivery risk.

This chapter governs the complete lifecycle from request through decommission.

---

## Environment Request Process

### Who Can Request an Environment

| Requestor | Can Request | Requires Approval From |
|-----------|------------|------------------------|
| Dev team lead | Dev, Feature Branch, Integration | None (self-service) |
| QA lead | QA | Platform team (capacity check) |
| Performance engineering lead | Performance | Platform team + SRE (scheduling) |
| Product owner | UAT | Platform team + SRE (data refresh) |
| Release manager | Staging | SRE (change-controlled) |
| VP/Director | Any | N/A |

**Self-service for lower tiers**: Dev, Feature Branch, and Integration environments must be self-service — no human approval required. Any process that requires human approval for these tiers creates a bottleneck that teams will work around.

### What a Request Must Specify

Before environment provisioning begins, the request must capture:

| Field | Required | Notes |
|-------|----------|-------|
| Environment tier | Yes | From defined taxonomy |
| Purpose | Yes | One sentence: what will this environment validate? |
| Duration | Yes | Start date, expected end date, hard-deadline date (for decommission scheduling) |
| Service list | Yes | All services that must be deployed; version or branch |
| Configuration requirements | Yes | Deviations from standard template (must be justified) |
| Data requirements | Yes | Synthetic, anonymized production, or specific fixture dataset |
| Team owner | Yes | Named primary owner + backup owner |
| Compute size | Yes | Estimated peak load for capacity planning |
| Network requirements | Conditional | External access requirements, firewall rules needed |
| Compliance requirements | Conditional | PCI scope, HIPAA scope — affects PII masking requirements |

**Template**: [Environment Request Template](../templates/environment-request-template.md)

### Provisioning SLA (From Request Submission to Environment Ready)

| Tier | Automated Self-Service SLA | With Manual Approval SLA |
|------|--------------------------|--------------------------|
| Feature Branch (Ephemeral) | < 5 minutes | N/A (self-service only) |
| Dev | < 30 minutes | N/A (self-service only) |
| Integration | < 30 minutes | N/A (self-service only) |
| QA | < 30 minutes automated; < 2 hours with data refresh | < 4 hours including approval |
| Performance | < 4 hours (dedicated infrastructure) | < 1 business day including scheduling |
| UAT | < 2 hours with standard data; < 4 hours with production data refresh | < 1 business day |
| Staging | < 1 hour | < 4 hours including CAB approval |

**What the VP measures**: MTTE (Mean Time to Environment) per tier. If MTTE consistently exceeds SLA, the provisioning automation investment is insufficient.

## Environment Provisioning: Automated vs. Manual

**The mandate**: Manual environment provisioning is not acceptable at scale. At T-Mobile delivery scale (hundreds of services, multiple delivery trains), manual provisioning creates a platform team bottleneck, introduces configuration errors from tribal knowledge, and cannot meet MTTE SLAs.

**Automated provisioning pipeline:**

```
Environment Request Submitted (via Internal Developer Portal)
    ↓
Validation: required fields complete, compute capacity available
    ↓
Approval (if required for tier)
    ↓
Automated provisioning triggered:
    ├── Terraform apply: VPC, subnets, EKS namespace, IAM roles
    ├── Vault secret provisioning: environment-scoped secrets created
    ├── Helm install: all services deployed from request's service list
    ├── Data provisioning: synthetic data or masked production snapshot loaded
    └── Health check: all services healthy (readiness probes passing)
    ↓
Provisioning validation:
    ├── Configuration parity check: environment diff against baseline
    ├── Service dependency check: all declared dependencies responding
    └── Data integrity check: expected record counts present
    ↓
Environment Ready notification: team owner notified with access credentials
```

**Why manual provisioning is always the wrong answer at scale:**
1. No two manually provisioned environments are identical — tribal knowledge introduces variance
2. Manual provisioning time (hours to days) is incompatible with developer productivity requirements
3. Manual provisioning creates platform team bottleneck — the ops team becomes the blocker for all delivery teams
4. Manually provisioned environments are not reproducible — you cannot destroy and rebuild with confidence

## Environment Ownership

### Ownership Model

Every environment must have:

| Role | Responsibility |
|------|---------------|
| **Primary Owner** | Accountable for environment availability, configuration, and decommission. Receives incident notifications. |
| **Backup Owner** | Covers when primary owner is unavailable. Must have equivalent access and knowledge. |
| **Platform Team Steward** | Infrastructure owner — responsible for compute health, but not for application-layer issues. |

**Governance rule**: Any environment without a named primary owner is a governance violation. Quarterly environment audit (see ch08) must identify and remediate ownerless environments within 10 business days.

### Owner Responsibilities

**Day-to-day:**
- Monitor environment health (via environment-specific dashboard)
- Approve or reject requests to deploy to the environment
- Coordinate data refreshes per environment SLA
- Communicate maintenance windows to consuming teams

**Incident response:**
- First-line responder for environment incidents
- Escalates to platform team for infrastructure-layer issues
- Files environment incident RCA within SLA

**Lifecycle:**
- Initiates decommission when environment purpose is complete
- Validates configuration before each environment is used for a new test cycle

## Environment Refresh Cadence

Environments drift from production over time. Periodic rebuild from production configuration snapshots prevents drift accumulation.

| Tier | Refresh Trigger | Refresh Content | Owner Action Required |
|------|----------------|----------------|----------------------|
| Feature Branch (Ephemeral) | Per PR — always fresh | Full rebuild from current IaC | None — automated |
| Dev | Weekly or on-demand | Rebuild from current IaC template | Coordinate with dev team on timing |
| Integration | Per sprint start | Rebuild from current IaC + latest service versions | QA lead sign-off |
| QA | Per sprint start + on-demand | Rebuild + data refresh | QA lead schedules |
| Performance | Per test cycle | Full rebuild + data refresh from production snapshot | Performance lead + SRE schedule |
| UAT | Per UAT cycle | Data refresh from masked production snapshot | Product owner triggers |
| Staging | Per release | Full rebuild from production-identical IaC | Release manager + SRE |

**Full rebuild vs. data refresh:**
- **Full rebuild**: Destroy existing environment, provision from IaC, deploy from artifact registry — eliminates all drift
- **Data refresh**: Keep infrastructure and application deployment; replace data layer from masked production snapshot — faster but does not fix infrastructure or config drift

**Recommendation**: Full rebuild for Staging before each release. Data refresh is acceptable for QA between sprints when infrastructure has not changed.

## Decommission Process

An environment that has served its purpose must be decommissioned. Zombie environments — environments that are no longer used but not destroyed — consume compute budget, create security exposure, and clutter the environment inventory.

**Decommission triggers:**
- Project has ended or been cancelled
- Environment has had no deployment or test activity in 30 days
- Hard-deadline date from original environment request has passed
- Capacity constraint requires reclamation

**Decommission process:**

```
Step 1: Notification (T-10 business days)
    → Email to primary owner, backup owner, platform team
    → "Environment [name] scheduled for decommission on [date]"
    → Include: data export options, artifact archive checklist

Step 2: Owner Confirmation (T-5 business days)
    → Owner confirms decommission OR requests extension
    → Extensions require justification and new end date
    → No response = decommission proceeds on schedule

Step 3: Data Export (T-2 business days)
    → Owner exports any test data or test results needed for record-keeping
    → Owner confirms no active test runs are in progress

Step 4: Decommission Execution
    → terraform destroy (in order: application layer → data layer → networking)
    → Vault secrets deleted (environment-scoped path)
    → DNS records removed
    → Environment removed from inventory registry

Step 5: Cost Validation
    → Verify compute costs associated with environment stop appearing in cost reporting
    → Confirm namespace deletion complete (kubectl get ns)
```

**Director Governance Gate**: Any environment more than 14 days past its declared end date without an approved extension is a governance escalation. The platform team notifies the VP. Decommission proceeds within 48 hours of escalation unless VP grants explicit extension.

## Capacity Planning for Non-Production

Non-production environments collectively consume 30–50% of total compute budget at most delivery organizations. This is not waste — it is investment in delivery quality. It must be planned, not discovered.

**Capacity allocation model:**

| Tier | % of Production Compute Reserved | Rationale |
|------|----------------------------------|-----------|
| Dev environments (all teams) | 10% | Low utilization; bursty; ephemeral |
| Integration | 15% | Moderate utilization; shared |
| QA (all active environments) | 20% | Peak during release cycles |
| Performance | 50% | Must be production-parity; only run periodically |
| UAT | 15% | Peak during UAT windows |
| Staging | 80–100% | Production-identical; active during release windows |

**Total non-prod compute reserve**: Plan for 60–80% of production compute capacity reserved for non-production, assuming the performance environment is not running 24×7. When the performance environment is active, total non-prod may exceed 100% of production compute for the duration.

**Capacity planning inputs:**
- Number of active development teams × environments per team
- Release cadence (weekly releases = staging always active; monthly = staging active 1 week/month)
- Performance test schedule (quarterly full runs vs. continuous load testing)
- Ephemeral environment peak concurrency (open PR count × resources per namespace)

## Environment Cost Allocation

Environment compute costs must be allocated to consuming teams. Without chargeback or showback, teams have no incentive to decommission unused environments or right-size their compute requests.

**Chargeback vs. showback:**

| Model | Description | When to Use |
|-------|-------------|-------------|
| Showback | Teams see their environment costs but are not charged | Initial implementation; build cost awareness without organizational friction |
| Chargeback | Teams are charged against their budget for environment compute consumed | Mature organizations; strong incentive for right-sizing and decommission |

**Cost tagging requirements:**

Every environment resource must be tagged:

```
Tags:
  Environment: qa-east-payments
  Tier: qa
  Owner: payments-team
  CostCenter: CC-12345
  Project: payments-v3
  DecommissionDate: 2026-09-30
```

**Cost allocation reporting (monthly):**

| Team | Tier | Environment Name | Compute Cost | Storage Cost | Total | % Budget |
|------|------|-----------------|-------------|-------------|-------|---------|
| Payments | QA | qa-east-payments | $1,240 | $180 | $1,420 | 14% |
| Auth | QA | qa-auth-v2 | $890 | $120 | $1,010 | 10% |
| Notifications | Performance | perf-notifications | $3,200 | $450 | $3,650 | 36% |

## Environment Inventory: The Registry

Every environment must be registered in the environment inventory. The inventory is the authoritative record of what environments exist, who owns them, their current status, and when they will be decommissioned.

**Minimum inventory fields:**

| Field | Description |
|-------|-------------|
| Environment Name | Unique identifier (e.g., `qa-east-payments-v3`) |
| Tier | From defined taxonomy |
| Purpose | One-sentence description |
| Primary Owner | Name + email |
| Backup Owner | Name + email |
| Status | Active / Inactive / Decommission Pending / Decommissioning |
| Configuration Version | Git SHA of IaC configuration |
| Last Refresh Date | When was environment last rebuilt or data refreshed |
| Decommission Date | Hard deadline for decommission |
| Cost (monthly) | Current monthly compute + storage cost |
| Compliance Scope | PCI / HIPAA / None — determines PII masking requirement |

**Template**: [Environment Inventory Template](../templates/environment-inventory-template.md)

**Inventory governance**: The inventory is reviewed quarterly (see ch08). Any environment in the inventory that has not had deployment activity in 30+ days is flagged for decommission review. Any environment missing a named owner is flagged as a governance violation.
