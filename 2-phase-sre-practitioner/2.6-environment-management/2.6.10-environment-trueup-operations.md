# Chapter 10 — Environment Trueup Operations

> **Audience**: Director/VP of Operations candidates. This chapter covers the operational mechanics of running trueups — what they are, how to execute them, what goes wrong, and how to measure them.

---

## What a Trueup Actually Is

A trueup is a controlled reset of an environment's state to a defined reference baseline. The word "trueup" is sometimes used loosely to mean "we refreshed the data." That is not a trueup. A trueup has three dimensions that must all be reset:

1. **Code version**: the services deployed in the environment match the reference baseline's service versions, not the versions that were deployed during the previous testing cycle
2. **Configuration state**: middleware topology (RabbitMQ vhosts, Kafka topics, consumer groups, MuleSoft flows), application configuration (feature flags, connection strings, environment variables), and infrastructure configuration (K8s ConfigMaps, Secrets, namespace quotas) all match the reference baseline
3. **Data state**: the environment's persistent data matches a masked, anonymized snapshot of the reference baseline's data, with referential integrity validated

If you only reset one or two of these dimensions, you do not have a trued-up environment. You have a partially refreshed environment that will produce test results of uncertain reliability.

---

## Why Environments Decay Without Trueup Discipline

An environment that has never been trued up will drift from its intended state within weeks. The drift accumulates through mechanisms that are each individually small and justifiable:

**Configuration hotfixes applied directly**: a test was blocked because a RabbitMQ queue had the wrong message TTL. Someone with environment access fixed it directly — `rabbitmqctl set_policy` — without going through the configuration management pipeline. The fix is not in the Helm chart. The next deployment that touches the environment will not include this change. The test that depended on it will fail for mysterious reasons.

**Feature flag states changed during testing**: QA enables a feature flag to test a specific scenario, then forgets to disable it. The environment now has a feature enabled that is not enabled in production. Tests run against this state and pass. Production does not have this feature enabled. Production behavior differs from what was tested.

**Data state accumulated from previous test cycles**: BUNDLE-QA-A finishes Month N testing. Some test scenarios created orders that triggered financial records. Some accounts now have transaction histories that did not exist in the production baseline. Month N+1 testing begins in the same environment. Test scenarios that assume a clean account state fail. The test team spends a day investigating before realizing the environment's data is contaminated from last month.

**Service versions pinned during troubleshooting**: a service was rolled back in the environment during Month N testing to diagnose a defect. The rollback was never reverted. Month N+1 testing begins with last month's version of that service deployed. The incompatibility surfaces as a mysterious integration failure.

This is the **environment that lied** problem. The environment appears to be configured correctly — it has all the right services deployed — but its accumulated state diverges from production in ways that make its test results unreliable predictors of production behavior. Teams that have never experienced this problem will not believe it. Teams that have experienced it will not skip trueups.

## Trueup Reference Baseline Choices

Three reference baselines are in common use. The choice depends on the environment's purpose:

| Baseline | Definition | Best For | Risk |
|----------|-----------|---------|------|
| Production snapshot | Production state immediately after the last release | BUNDLE-QA, UX-ENV, FINANCIAL-DOMAIN-ENV, PRE-PROD | Masking pipeline complexity; PCI/PII handling overhead |
| Last release tag | The git tag and configuration that was deployed to production | PROJECT-QA, DEV-INT | Slightly behind production if hotfixes were applied post-release; must include hotfix tracking |
| Sprint baseline | The designated "stable" commit from the current sprint, agreed by the team | DEV, DEV-INT mid-sprint | May not reflect integration decisions made late in sprint |

For any environment that gate-validates a production release candidate, the production snapshot is the only defensible baseline. The question "does this environment behave like production?" has only one honest answer if the environment started from a production snapshot.

## The Trueup Pipeline: Step-by-Step

```
[Reference baseline selection]
         │
         ▼
[Source snapshot capture]
  - DB logical dump or physical snapshot
  - Configuration export (infra, middleware, application)
  - Service version manifest
         │
         ▼
[Anonymization / masking pipeline]
  - PII field masking (names, SSNs, account numbers)
  - PCI data replacement (card numbers → test tokens)
  - Referential integrity validation (masked data must still join correctly)
         │
         ▼
[Infrastructure reset]
  - REBUILD: tear down environment, provision from IaC from scratch
  - OR DELTA: apply only the configuration/version differences from baseline
         │
         ▼
[Configuration sync]
  - Middleware (RabbitMQ, Kafka, MuleSoft)
  - Application (feature flags, connection strings)
  - Infrastructure (K8s ConfigMaps, Secrets, HPA limits)
         │
         ▼
[Data load]
  - Masked snapshot restore
  - Referential integrity validation
  - Volume verification (row counts within tolerance of snapshot)
         │
         ▼
[Smoke test suite]
  - 20-minute functional smoke (health endpoints, critical user journeys)
  - Not full regression — full regression runs later during testing cycle
         │
         ▼
[Readiness declaration]
  - SRE engineer signs off formally
  - Environment enters standby state
  - Change management record updated
```

## Infrastructure Rebuild vs. Delta Apply

This is the most consequential technical decision in the trueup process. Both approaches work. The wrong choice for the wrong situation causes either missed drift (delta on a heavily drifted environment) or unnecessary delay (rebuild when delta would have been sufficient).

### Rebuild from Scratch

**Mechanism**: tear down the environment completely — delete the namespace, destroy the database, remove all configuration. Provision from IaC (Terraform + Helm) as if the environment had never existed. Load data from the masked snapshot.

**Advantages**:
- Eliminates all accumulated drift by construction — there is nothing to accumulate drift on after a full rebuild
- Validates the IaC itself: if provisioning fails, you know the IaC has bugs before a production deployment reveals them
- Produces a deterministic environment state — you know exactly what is in it because you built it from known inputs

**Disadvantages**:
- Takes significantly longer: full rebuild for a BUNDLE-QA environment typically runs 4–6 hours for infrastructure provisioning + 2–4 hours for data load = 6–10 hours total
- Requires robust IaC — if the Terraform modules have gaps (resources managed manually, not in code), rebuild will miss those resources
- Higher blast radius if rebuild fails mid-process: the environment is down while rebuild is in progress

**Use rebuild when**:
- The environment has not been trued up in more than 2 release cycles
- Significant IaC changes were made during the cycle (new modules, refactored resources)
- The previous cycle's environment is known to have accumulated significant drift
- The environment supports a high-stakes testing window (BUNDLE-QA, PRE-PROD)

### Delta Apply

**Mechanism**: compare the current environment state against the reference baseline. Apply only the differences — service version upgrades/downgrades, configuration changes, data changes (truncate-and-load for changed tables, not full rebuild).

**Advantages**:
- Much faster: delta apply for a BUNDLE-QA environment typically runs 2–3 hours
- Lower risk mid-process: environment is partially functional throughout, not completely down
- Practical for environments where full rebuild would be prohibitively slow

**Disadvantages**:
- Only as reliable as the drift detection: if drift exists that the delta comparison does not detect, it will not be corrected
- Requires a reliable state comparison tool — diffing two environment states is harder than it sounds, especially for middleware configuration
- Cumulative risk: each delta cycle leaves the possibility that some drift was missed

**Use delta apply when**:
- The previous trueup was a full rebuild (low baseline drift)
- The time window for trueup is constrained (mid-cycle emergency trueup, DEV-INT monthly trueup)
- The environment is DEV-INT or PROJECT-QA (lower stakes than BUNDLE-QA)
- You have high confidence in your drift detection tooling

**Decision rule**: BUNDLE-QA and PRE-PROD always get full rebuild for the first trueup of each release cycle. Subsequent mid-cycle trueups (defect-driven) can use delta apply. DEV-INT and PROJECT-QA default to delta apply unless drift is suspected.

## Configuration Trueup: The Part People Skip

Data trueup gets the most attention. Configuration trueup gets the least. This is backwards. Configuration drift is harder to detect than data drift and produces subtler, more dangerous failures.

### Middleware Configuration

**RabbitMQ**: vhosts, exchanges, queues, bindings, policies (HA mirrors, TTLs, dead-letter routing), and user permissions must all match the reference baseline. A common drift pattern: a queue that was created during incident response in a previous cycle (to temporarily reroute messages) still exists in the environment. The next cycle's test creates messages that route to this queue unexpectedly.

Export the RabbitMQ definition file from the reference baseline: `rabbitmqctl export_definitions`. Apply it to the target environment: `rabbitmqctl import_definitions`. Verify by diffing the exported definitions before and after.

**Kafka**: topic configurations (partition count, replication factor, retention policy, cleanup policy), consumer group offsets (reset to beginning or specific offset depending on test scenario requirements), ACLs. Consumer group offset state in particular accumulates between cycles — if consumer groups from Month N testing are not reset, Month N+1 tests may pick up from the wrong offset.

**MuleSoft / Integration Layer**: flow deployment versions, property file values, connector configurations, API endpoint registrations. MuleSoft flows deployed during testing but not part of the release baseline create particularly dangerous drift because they may intercept message traffic silently.

### Application Configuration

Feature flags are the highest-risk application configuration item. Use a configuration-as-code approach where feature flag state is declared in a file committed to the repository. Trueup resets feature flags from the declared baseline file, not from whatever state the previous testing cycle left them in.

Connection strings, environment variables, and JVM system properties should be managed as K8s Secrets and ConfigMaps with their values sourced from a secrets manager (Vault, AWS Secrets Manager). Trueup for these: sync the K8s Secret and ConfigMap manifests from the reference baseline's IaC repository. Restart affected pods after sync to ensure no in-memory cached configuration survives.

## Data Trueup: The Mechanics

**Snapshot capture**: logical dump (pg_dump for PostgreSQL, mysqldump for MySQL) is preferred over physical snapshot for portability and ability to apply masking at the field level. Physical snapshot (EBS snapshot, RDS snapshot) is faster but requires masking at a different layer and is harder to validate.

**PII masking pipeline**: the masking pipeline must be:
1. Deterministic — the same source value always produces the same masked value (so that JOINs on masked values still work across tables)
2. Format-preserving — masked phone numbers look like phone numbers, masked account numbers have the right length and check digit validity
3. Referential-integrity-preserving — if customer_id 12345 in the customers table is masked to 99001, all references to customer_id 12345 in orders, transactions, and support_tickets must also become 99001

After masking, run referential integrity validation: check all foreign key relationships still resolve. A masking pipeline that breaks referential integrity produces test data that causes spurious failures unrelated to the release under test.

**Volume verification**: the loaded data should be within 5% of the snapshot's row counts by table. Significant deviation indicates a load failure. For very large tables (billions of rows), a 10% sample load is acceptable for functional testing — document this decision explicitly so test teams know the data is not full volume.

## BUNDLE-QA A/B Trueup Schedule: Detailed Calendar

The trueup schedule must be designed so that BUNDLE-QA-B is always ready before BUNDLE-QA-A completes its testing window. This is a scheduling constraint, not an aspiration.

```
Month N Timeline:

Day 1:    Production release completes at 2am
          BUNDLE-QA-B trueup triggered automatically (post-release hook)
          SRE on-call initiates trueup pipeline

Day 1–2:  BUNDLE-QA-B: production snapshot captured and masked (runs overnight)

Day 2–3:  BUNDLE-QA-B: infrastructure rebuild (runs overnight if full rebuild)

Day 3:    BUNDLE-QA-B: configuration sync and data load

Day 4:    BUNDLE-QA-B: smoke test validation
          Target: BUNDLE-QA-B declared READY by end of Day 4

Day 5:    BUNDLE-QA-B in STANDBY state — trueup complete, buffer day
          BUNDLE-QA-A: Month N+1 bundle begins loading from PROJECT-QA

Days 6–19: BUNDLE-QA-A: active regression testing for Month N+1 bundle
            (14-day testing window)
            BUNDLE-QA-B: remains in standby (no changes permitted)

Day 20:   BUNDLE-QA-A: regression complete, sign-offs collected
          Month N+1 bundle advances to UAT

Day 21:   BUNDLE-QA-B trueup begins for Month N+2
          (14-day cycle repeats with A and B swapped)
```

**Minimum trueup lead time**: the trueup must complete at least 1 business day before the next bundle testing window opens. This buffer exists to handle trueup failures without immediately impacting the release schedule.

**If trueup takes longer than expected**: the SRE lead is notified when trueup duration exceeds the plan by 50% (e.g., expected 4-day trueup is not complete by Day 6). VP Operations is notified when trueup is not complete by the last business day before the testing window opens.

## Trueup Validation Checklist

Before issuing a readiness declaration, the following must all pass:

```
Infrastructure Validation
[ ] All services deployed at reference baseline versions (verified against version manifest)
[ ] No legacy pods from previous cycle still running (namespace clean)
[ ] Resource limits and HPA configs match baseline
[ ] Network policies applied and validated (cross-namespace connectivity as expected)

Middleware Validation
[ ] RabbitMQ: vhost count, exchange count, queue count match baseline definition export
[ ] RabbitMQ: HA mirror policies applied to all queues requiring mirroring
[ ] Kafka: topic list, partition counts, replication factors match baseline
[ ] Kafka: consumer group offsets reset to designated starting position
[ ] MuleSoft: deployed flow versions match reference baseline
[ ] API Gateway: endpoint registrations match baseline

Application Configuration Validation
[ ] Feature flags at baseline state (all, not just changed ones — full state comparison)
[ ] Connection strings pointing to environment-local services (not production!)
[ ] Environment variable values match baseline ConfigMap manifest

Data Validation
[ ] Row counts within 5% of snapshot for all critical tables
[ ] Referential integrity check: zero orphaned foreign keys
[ ] Masking validation: PII fields contain masked values (sample check)
[ ] No data from previous testing cycle in the environment (truncation confirmed)

Smoke Test Validation
[ ] Health endpoints: all services return 200 within SLA
[ ] Critical user journey: primary business flow completes end-to-end
[ ] Message flow: publish a test message, confirm consumer receives it
[ ] Database connectivity: read and write transactions complete successfully
[ ] Cache connectivity: Redis SET and GET succeed
```

Checklist is executed by two SRE engineers (one executes, one validates). Both sign the readiness declaration.

## Trueup Failure Handling

A trueup failure is a pipeline blocker. The failure must be diagnosed, not retried blindly.

**Failure classification**:

| Failure Type | Symptom | Response |
|-------------|---------|---------|
| Infrastructure provisioning failure | Terraform apply errors, pods not starting | Fix IaC bug, re-run provisioning stage only |
| Masking pipeline failure | Data load errors, referential integrity violations | Fix masking pipeline, re-run from snapshot capture |
| Configuration sync failure | Middleware not matching baseline | Diagnose config diff, re-run sync stage |
| Smoke test failure | Health checks failing post-trueup | Triage as environment incident — do not declare ready |
| Data volume failure | Row counts significantly below expected | Investigate load failure, re-run data load stage |

**Escalation path**:
1. SRE engineer discovers failure and diagnoses root cause
2. If root cause not identified within 2 hours: SRE Lead engaged
3. If trueup will not complete within 50% of SLA: VP Operations notified with impact assessment and options
4. If trueup failure will delay release schedule: Program Manager and Release Manager notified

**When to restart the full trueup vs. resume from failure point**: restart the full trueup if the failure occurred in the snapshot capture or masking stages. Resume from the failure stage if the failure occurred in configuration sync, data load, or smoke test — assuming the earlier stages completed cleanly and their state is still valid.

## Trueup Metrics

Measure these to demonstrate operational excellence in environment management:

| Metric | Definition | Target | Measurement Cadence |
|--------|-----------|--------|-------------------|
| Trueup duration | Time from trueup initiation to readiness declaration | Within tier SLA | Per trueup |
| Trueup success rate | Percentage of trueups that complete without requiring restart | ≥ 95% | Monthly |
| Drift detection rate | Percentage of delta-apply trueups that detect configuration drift | Track trend (high drift rate = signal to switch to rebuilds) | Monthly |
| Time from production release to B-ready | Calendar hours from production release to BUNDLE-QA-B readiness declaration | ≤ 5 business days | Per release cycle |
| Smoke test pass rate on first attempt | Percentage of post-trueup smoke tests that pass without environment fixes | ≥ 90% | Monthly |
| Defect-driven trueup frequency | Number of emergency trueups triggered per release cycle by environment state issues | Target: zero | Monthly |

A rising defect-driven trueup frequency is the leading indicator that your environment pipeline has a drift problem. It should be reviewed in every SRE operations review as a pipeline health metric.
