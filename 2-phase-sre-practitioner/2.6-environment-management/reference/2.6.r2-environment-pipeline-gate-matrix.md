## Master Gate Matrix

| Station | Environment | Purpose | Gate Criteria (ALL must pass) | Sign-off Authority | Gate Type | SLA | Promotes To |
|---------|-------------|---------|-------------------------------|-------------------|-----------|-----|-------------|
| 1 | DEV | Developer experimentation | (1) Unit tests ≥ 95% pass rate, zero P0 failures (2) Code review approved by ≥ 1 peer (3) SAST scan: no HIGH or CRITICAL unresolved (4) PR merged to sprint branch | Developer (self-service) | Advisory | Best-effort | DEV-INT |
| 2 | DEV-INT | Cross-team integration testing | (1) Integration test suite pass: contract tests, API tests, message flow tests (2) No P1/P2 integration defects open (3) DEV-INT stable (no P1 incidents) for 24 hours prior to promotion | Engineering Manager | Blocking | 95% uptime business hours; 4hr P1 restore | PROJECT-QA |
| 3 | PROJECT-QA | Per-project functional QA | (1) Functional test pass rate ≥ 98% (2) All P1/P2 defects resolved or waivered (waiver: Eng Manager + Product Owner sign) (3) QA Test Lead formal sign-off (4) No unresolved service dependency issues | QA Lead + Engineering Manager | Blocking | 98% during test window; 2hr restore | BUNDLE-QA |
| 4 | BUNDLE-QA | Full bundle regression (A/B rotation) | (1) Full regression suite pass (all features, not just new) (2) Performance smoke test within 20% of PERF-ENV baseline (3) All parallel domain sign-offs received: UX-ENV, FINANCIAL-DOMAIN-ENV, PERF-ENV (4) Zero unresolved P1/P2 defects | Release Manager | Blocking | 99% during 2-week test window; immediate VP escalation on any P1 | UAT |
| 5 | UX-ENV | UX/accessibility/A-B validation | (1) Automated a11y scan: zero WCAG AA violations on in-scope pages (2) UX journeys validated for all release-scope user flows (3) A/B test results reviewed and signed (4) No blocking UX defects open | UX Lead + Product Owner | Blocking (parallel with BUNDLE-QA) | 98% during test window | BUNDLE-QA aggregate gate |
| 6 | FINANCIAL-DOMAIN-ENV | Financial/regulatory/compliance testing | (1) All financial calculations validated against expected output set (2) Regulatory scenario coverage: 100% of mandated scenarios executed (3) GL reconciliation: zero unreconciled entries (4) Payment flow: authorization, settlement, reversal paths all pass (5) Audit log of all test scenarios complete | Finance Manager + Compliance Officer | Blocking (parallel with BUNDLE-QA) | 99% during test window | BUNDLE-QA aggregate gate |
| 7 | UAT | Business acceptance validation | (1) All business process journeys validated by domain BAs (2) P1 defects: none open (3) P2 defects: none open OR VP Operations + Business Owner joint waiver with remediation timeline (4) All business domain owners signed | Business Owner (per domain) + Program Manager | Blocking (sequential — after BUNDLE-QA+UX+FIN) | 98% during UAT window; 2hr restore | PRE-PROD |
| 8 | PERF-ENV | Continuous performance gate | (1) 48-hour soak test completed without interruption (2) p95 latency within 15% of established baseline for full soak duration (3) TPS throughput within 10% of established baseline (4) Memory growth rate flat (no leak signature detected) (5) GC pause frequency increase ≤ 20% from baseline (6) Zero pod restarts (OOMKilled, CrashLoopBackOff) during soak | Performance Engineering Lead + Platform SRE Lead | Blocking (parallel with BUNDLE-QA; result collected before BUNDLE-QA sign-off) | 99.9% continuous (PERF-ENV runs 24/7) | PRE-PROD (evidence required) |
| 9 | PRE-PROD | Deployment rehearsal + final go/no-go | (1) CAB approval on change record (2) All upstream sign-off evidence attached to change record (BUNDLE-QA, UX, FIN, UAT, PERF) (3) Deployment completed using production automation — no manual steps (4) Rollback procedure rehearsed and timed (5) Smoke test: all health endpoints 200, primary journeys functional (6) On-call team briefed and confirmed available | VP Operations or delegated Release Manager | Blocking | Production-parity maintained continuously; 4hr delta-apply trueup SLA | PRODUCTION |
| 10 | PRODUCTION | Live service delivery | N/A — terminal promotion station | N/A | N/A | Tier-dependent SLO (T0: 99.999%; T1: 99.99%; T2: 99.9%; T3: 99.5%) | POST-PROD VALIDATION |
| 11 | POST-PROD | Release stability confirmation | (1) Smoke test pass: 0–15 min post-deployment (2) Canary metrics clean: 15–60 min (error rate ≤ 2× baseline, latency p95 ≤ 20% above baseline) (3) 4-hour monitoring watch: no P1 indicators (4) No downstream services reporting unexpected behavior | On-call Engineer (execution) + Release Manager (closure) | Time-bound gate — change record held open until all criteria met | N/A — time-bound activity (24hr max) | Change record closure |

## Parallel vs. Sequential Gate Map

```
DEV  ──►  DEV-INT  ──►  PROJECT-QA(A)  ─┐
                         PROJECT-QA(B)  ─┤
                         PROJECT-QA(N)  ─┘
                                         │ (all promoting projects)
                                         ▼
                                    BUNDLE-QA ◄──────────────────────┐
                                         │                            │
                    ┌────────────────────┼──────────────────┐         │
                    ▼                    ▼                   ▼         │
                 UX-ENV          FINANCIAL-DOMAIN-ENV    PERF-ENV      │
              (parallel)              (parallel)         (parallel,    │
                    │                    │               continuous)   │
                    └────────────────────┘                             │
                                         │ (ALL THREE must sign off)   │
                                         ▼                             │
                                        UAT  ◄──── SEQUENTIAL GATE    │
                                         │                             │
                                         ▼                             │
                                     PRE-PROD                          │
                                         │                             │
                                         ▼                             │
                                    PRODUCTION ──────────────────────►─┘
                                         │          (next release cycle)
                                         ▼
                                  POST-PROD VALIDATION
                                         │
                                         ▼
                                  Change Record Closure
```

**Key rules**:
- PROJECT-QA instances are parallel to each other — multiple projects promote to BUNDLE-QA independently
- UX-ENV, FINANCIAL-DOMAIN-ENV, and PERF-ENV run in parallel with BUNDLE-QA (not after it)
- UAT is strictly sequential — it requires all three parallel domain gates to complete
- PERF-ENV is continuous — its gate is satisfied by collecting the evidence, not by sequencing after BUNDLE-QA

## Monthly Release Cycle Trueup Calendar

The following calendar assumes a standard 28-day release cycle with production release on Day 1. Adjust dates based on your organization's actual cycle length.

```
┌────────────────────────────────────────────────────────────────────────────────┐
│ MONTHLY RELEASE CYCLE CALENDAR                                                 │
├─────────┬──────────────────────────────────────────────────────────────────────┤
│ Day 1   │ PRODUCTION RELEASE (2am deployment window)                          │
│         │ → BUNDLE-QA-B trueup triggered (post-release automation hook)       │
│         │ → DEV-INT monthly trueup triggered                                  │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 1–2 │ BUNDLE-QA-B: Production snapshot capture + masking pipeline          │
│         │ DEV-INT: delta-apply trueup running (4hr target)                    │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 2   │ DEV-INT trueup complete → DEV-INT open for sprint work              │
│         │ BUNDLE-QA-B: infrastructure rebuild running overnight               │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 3   │ BUNDLE-QA-B: configuration sync + data load                         │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 4   │ BUNDLE-QA-B: smoke test validation                                  │
│         │ TARGET: BUNDLE-QA-B READY DECLARATION by end of Day 4               │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 5   │ BUNDLE-QA-B in STANDBY (buffer day — trueup complete)               │
│         │ PROJECT-QA environments promoting completed projects to BUNDLE-QA-A  │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 6   │ BUNDLE-QA-A: Month N+1 bundle testing BEGINS                        │
│         │ UX-ENV: parallel UX testing begins                                  │
│         │ FINANCIAL-DOMAIN-ENV: parallel financial testing begins              │
│         │ PERF-ENV: release candidate deployed for 48hr soak                  │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 7   │ PROJECT-QA: new sprint — new PROJECT-QA instances provisioned       │
│         │   for Month N+2 projects                                             │
│         │ PERF-ENV: 48hr soak in progress                                     │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 8   │ PERF-ENV: 48hr soak complete → Performance gate evidence collected  │
│         │ BUNDLE-QA-A: regression testing continues (Day 3 of 14-day window)  │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Days    │ BUNDLE-QA-A: active regression (full 14-day window)                 │
│ 9–18   │ UX-ENV: UX/a11y validation in progress                              │
│         │ FINANCIAL-DOMAIN-ENV: financial scenario testing in progress        │
│         │ BUNDLE-QA-B: STANDBY — no changes, awaiting next cycle             │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 18  │ TARGET: UX-ENV sign-off complete                                    │
│         │ TARGET: FINANCIAL-DOMAIN-ENV sign-off complete                      │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 19  │ BUNDLE-QA-A: final regression pass, all defect resolutions          │
│         │ TARGET: BUNDLE-QA-A sign-off (with all domain evidence)             │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 20  │ UAT BEGINS (sequential — all prior gates complete)                  │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 22  │ TARGET: UAT sign-off complete                                        │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 23  │ PRE-PROD: deployment rehearsal, runbook dry-run, rollback rehearsal  │
│         │ CAB: change record submitted                                         │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 24  │ CAB: change record review and approval                               │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 25  │ PRE-PROD: final smoke test, on-call briefing, go/no-go call         │
│         │ BUNDLE-QA-A trueup: begins (now the inactive environment)            │
├─────────┼──────────────────────────────────────────────────────────────────────┤
│ Day 28  │ PRODUCTION RELEASE (2am deployment window)                          │
│         │ Day 1 of Month N+2 cycle — calendar repeats                         │
└─────────┴──────────────────────────────────────────────────────────────────────┘
```

**Trueup lead-time constraint**: the inactive BUNDLE-QA environment (B or A) must be declared READY at least 1 business day before the active environment's testing window opens. In the calendar above, BUNDLE-QA-B must be ready by Day 4 (or Day 5 at latest with the buffer day), and BUNDLE-QA-A testing opens on Day 6.

## Environment Configuration Parity Checklist

For each environment, the following table defines what must match production exactly vs. what is permitted to differ.

| Configuration Dimension | PRE-PROD | BUNDLE-QA | PROJECT-QA | DEV-INT | DEV |
|------------------------|----------|-----------|------------|---------|-----|
| **Service versions** | Must match production release candidate exactly | Must match release candidate | Must match project candidate tag | Current sprint HEAD | Developer branch |
| **Instance types / hardware class** | Must match production (same EC2 family, size) | Same family, may scale to 50% size | Scaled to 25% | Minimal sizing acceptable | Minimal |
| **JVM settings** (heap, GC algo, thread pools) | Must match production | Must match production | Should match (documented deviation acceptable) | Best-effort | Not required |
| **Database engine and version** | Must match production | Must match production | Must match production | Must match production | May use compatible version |
| **Connection pool sizes** | Must match production | Must match production | Scaled (50% acceptable) | Not required to match | Not required |
| **RabbitMQ vhost and queue structure** | Must match production | Must match production (from trueup) | Must match production | Must match production | Mock acceptable |
| **Kafka topic config** (partitions, replication, retention) | Must match production | Must match production (from trueup) | Must match production | Must match production | Single-node acceptable |
| **MuleSoft flow versions** | Must match release candidate | Must match release candidate | Must match project candidate | Sprint HEAD | Mock acceptable |
| **Feature flags** | Production baseline state | Production baseline state + test overrides documented | Project-specific overrides on top of baseline | Sprint baseline | Developer discretion |
| **K8s ConfigMaps** | Must match production | Must match production (from trueup) | Must match production with documented overrides | Must match baseline | Developer discretion |
| **K8s Secrets** (values) | Must match production (via secrets manager sync) | Environment-specific secrets (not production credentials) | Environment-specific secrets | Environment-specific secrets | Local/mock secrets |
| **Network security groups** | Must match production | Must match production | Relaxed for test tooling access | Relaxed | Developer machine access |
| **TLS certificates** | Production certificate or production-class cert | Valid cert (not self-signed for external tests) | Self-signed acceptable | Self-signed acceptable | Self-signed acceptable |
| **WAF rules** | Must match production | Must match production | Reduced rule set acceptable | WAF optional | WAF not required |
| **Data volume** | Full production snapshot (masked) | Full production snapshot (masked) | Subset (representative sample, documented) | Reduced synthetic dataset | Minimal seed data |
| **PII masking** | Required | Required | Required | Required | Required |
| **Monitoring / alerting config** | Must match production alerting rules | Must have environment-specific alerting | Basic alerting | Team notification only | Developer notification only |
| **Log retention** | 90 days (matches production) | 30 days | 14 days | 7 days | Session only |
| **Backup policy** | Daily (matches production) | Not required — replaced by trueup | Not required | Not required | Not required |
| **DNS / hostname** | Environment-specific subdomain (`preprod.internal`) | Environment-specific (`bundle-qa-a.internal`) | Project-specific (`qa-payments.internal`) | `dev-int.internal` | `dev-{username}.local` |
| **External integrations** | Production endpoints (read-only testing) or stub | Stub / sandbox endpoints | Stub / sandbox endpoints | Stub | Mock |

**Key governance rules from the parity matrix**:

1. **Secrets are never shared between environments**. PRE-PROD does not use production database passwords. It uses production-class infrastructure with environment-specific credentials. Any environment using production credentials is a security incident.

2. **JVM and connection pool settings must match in BUNDLE-QA**. Tests that pass in a BUNDLE-QA with undersized JVM settings and then fail in production due to GC pressure are a parity failure, not a code defect.

3. **Feature flag state must be explicitly declared and documented for every environment above DEV-INT**. Ad-hoc feature flag changes during testing are the primary source of environment configuration drift.

4. **External integration endpoints**: BUNDLE-QA and below use sandbox/stub endpoints. PRE-PROD may use production read-only endpoints for smoke testing, but never writes to production systems. This requires explicit approval from the system owner of each external integration used in PRE-PROD.

## Trueup Schedule by Environment Type

| Environment | Trigger | Baseline | Full Rebuild or Delta | Target Duration | Max Duration | Who Initiates | Who Declares Ready |
|-------------|---------|----------|----------------------|----------------|-------------|--------------|-------------------|
| BUNDLE-QA (inactive) | Post-production release (automatic) | Production snapshot (masked) | Full rebuild | 4 calendar days | 5 business days | Automated hook + SRE Lead | SRE Lead (2-engineer sign-off) |
| PRE-PROD | Before each release (continuous parity) | Production current state | Delta apply | 4 hours | 8 hours | Release Manager triggers, SRE executes | SRE on-call |
| DEV-INT | First Monday post-production release | Last release tag | Delta apply | 4 hours | 8 hours | SRE automated schedule | SRE on-call |
| PROJECT-QA | Project start (new project instance) | DEV-INT current baseline | Delta apply (new namespace) | 8 hours | 16 hours | Engineering Manager request | SRE on-call |
| UX-ENV | Monthly (aligned with BUNDLE-QA cycle) | Production snapshot (masked) | Delta apply | 8 hours | 16 hours | SRE automated schedule | UX Environment Owner + SRE |
| FINANCIAL-DOMAIN-ENV | Monthly + regulatory calendar triggers | Compliance-approved test data set | Full rebuild | 1 business day | 2 business days | Finance IT + SRE | Finance IT Manager + SRE |
