# Environment Tooling Reference

**Purpose**: Tool landscape for environment provisioning, configuration management, and governance. Maps each tool to its capability, integration point, and fit for the environment management program.

---

## Provisioning Tools

### Terraform (HashiCorp)

**Category**: Infrastructure provisioning (IaC)  
**Primary use**: Provisioning VPCs, EKS clusters, RDS instances, IAM roles, S3 buckets, and all infrastructure-layer environment resources.

| Capability | Notes |
|-----------|-------|
| Multi-environment provisioning | Workspace or directory-per-environment model; shared modules |
| State management | Remote state in S3 + DynamoDB locking; per-environment state file |
| Drift detection | `terraform plan` shows infrastructure drift; integrate into daily CI run |
| Provider support | AWS, Azure, GCP, Kubernetes, Helm, Vault — single tool for multi-layer provisioning |
| Secret management | Vault provider for secret creation; never store secrets in `.tfvars` |

**Strengths**: Industry standard; large module library; strong state management; native drift detection.  
**Limitations**: Does not manage Kubernetes application deployments well (use Helm for that); state file can become bottleneck if not partitioned correctly.

**Environment management integration**: Terraform manages the infrastructure layer of each environment. Every environment has its own state file. A shared `modules/application-environment` module enforces configuration structure. Daily `terraform plan` run per environment for drift detection.

---

### Helm (Kubernetes Package Manager)

**Category**: Kubernetes application deployment  
**Primary use**: Deploying application services to environment namespaces with environment-specific configuration.

| Capability | Notes |
|-----------|-------|
| Environment-specific values | `values.<tier>.yaml` per environment tier; `values.override.yaml` for specific environment |
| Rollback | `helm rollback <release> <version>` — one-command rollback to previous chart version |
| Release history | `helm history <release>` — full deployment history per release |
| Templating | Shared chart; environment differences expressed as values, not separate charts |
| Secrets | Use external-secrets operator to pull from Vault; never store secrets in Helm values |

**Strengths**: Environment-specific values files cleanly separate what varies across environments; rollback is reliable.  
**Limitations**: Does not self-heal (that's Kubernetes); chart upgrades must be managed; values files can accumulate drift if not audited.

---

### ArgoCD (GitOps Continuous Delivery)

**Category**: GitOps-based application deployment  
**Primary use**: Continuous delivery of Helm charts to environments; self-healing application state; environment configuration drift detection at the application layer.

| Capability | Notes |
|-----------|-------|
| GitOps model | Declared state in Git; ArgoCD reconciles cluster state to Git state |
| Self-healing | Detects and reverts manual changes to cluster resources (configuration drift prevention) |
| Multi-environment | ApplicationSet resource manages deployments across all environments from single definition |
| Sync status | Dashboard shows which environments are in-sync or out-of-sync with Git |
| Rollback | Revert Git commit; ArgoCD syncs automatically |

**Strengths**: Self-healing prevents manual configuration drift; GitOps model provides full audit trail; ApplicationSet scales across many environments.  
**Limitations**: Requires Git as the source of truth — any change outside Git is reverted; learning curve for teams used to direct kubectl apply.

**Environment management integration**: ArgoCD manages application-layer parity. Any configuration change made directly to the cluster (bypassing Git) is detected and reverted within 3 minutes. This is the primary control for application-layer drift prevention.

---

### Crossplane

**Category**: Kubernetes-native infrastructure provisioning  
**Primary use**: Provision cloud infrastructure (VPCs, databases, message brokers) as Kubernetes custom resources; enable platform teams to expose environment provisioning as self-service Kubernetes APIs.

| Capability | Notes |
|-----------|-------|
| Composition | Define composite resources (e.g., "PostgreSQLDatabase") that map to multiple cloud resources |
| Self-service | Delivery teams create a Kubernetes resource to request infrastructure; Crossplane provisions it |
| Multi-cloud | AWS, Azure, GCP providers; single API surface across clouds |
| GitOps compatible | Works with ArgoCD; declared in Git; reconciled continuously |

**Strengths**: Enables true self-service infrastructure provisioning without requiring teams to write Terraform; platform team defines Compositions that encode approved patterns; delivery teams consume without touching cloud provider APIs.  
**Limitations**: Younger ecosystem than Terraform; debugging provider issues is complex; requires platform team expertise in Composition authoring.

**Environment management integration**: Crossplane is the implementation primitive for self-service environment provisioning in mature platform engineering organizations. Delivery teams submit a `CompositeEnvironment` resource; Crossplane provisions the VPC, database, and message broker; Helm deploys the applications. End-to-end in < 15 minutes.

## Developer Portal and Catalog Tools

### Backstage (Spotify/CNCF)

**Category**: Internal Developer Portal (IDP)  
**Primary use**: Self-service environment provisioning UI; environment catalog; cost visibility; onboarding guide.

| Capability | Notes |
|-----------|-------|
| Software catalog | Inventory of all services, environments, and owners |
| Templates (scaffolder) | Self-service forms for environment provisioning; calls Terraform/Crossplane/Helm |
| TechDocs | Inline documentation for each environment template |
| Plugins | Cost plugin (infracost), Kubernetes plugin (pod status), ArgoCD plugin (sync status) |
| Search | Find any environment, owner, or runbook |

**Strengths**: Best-in-class IDP; large plugin ecosystem; strong community; CNCF graduated project.  
**Limitations**: Requires engineering investment to set up and maintain; plugin quality varies; not plug-and-play.

**Environment management integration**: Backstage is the primary interface for delivery teams. Environment request forms live in Backstage Scaffolder. Environment status (health, cost, owner) surfaces in the Software Catalog. New engineer onboarding includes a Backstage tutorial as day-1 step.

## Configuration Management and Drift Detection Tools

### Conftest + OPA (Open Policy Agent)

**Category**: Policy-as-Code for configuration validation  
**Primary use**: Validate Kubernetes manifests and Helm values against environment governance policies before deployment.

| Policy | Example |
|--------|---------|
| No plaintext secrets in ConfigMaps | Reject any ConfigMap with values matching common secret patterns |
| Resource limits required | Reject any Deployment without CPU and memory limits |
| Image tag must not be `latest` | Reject any image reference without a pinned version tag |
| PII in environment variable names | Warn on env vars named `PASSWORD`, `SECRET`, `TOKEN` without Vault reference |

**Integration point**: Pre-deployment gate in CI pipeline. `conftest test` runs on every PR against the environment's manifests.

---

### Checkov (Bridgecrew / Prisma Cloud)

**Category**: Static analysis for IaC  
**Primary use**: Scan Terraform files for security misconfigurations and compliance violations before `terraform apply`.

| Check Category | Example |
|---------------|---------|
| S3 bucket security | No public access; encryption enabled; versioning enabled |
| RDS security | Encryption at rest; no public accessibility; backup retention ≥ 7 days |
| EKS security | API server not public; node groups not using default VPC; secrets encryption enabled |
| IAM | No wildcard actions; no inline policies; MFA required for console access |

**Integration point**: CI/CD pipeline pre-apply gate. Failures block Terraform apply.

---

### AWS Config + Config Rules

**Category**: Continuous infrastructure compliance monitoring  
**Primary use**: Real-time detection of infrastructure drift in AWS-managed resources.

| Rule | What It Detects |
|------|----------------|
| `DESIRED_INSTANCE_TYPE` | Instance type changed from approved list |
| `EC2_INSTANCE_PROFILE_ATTACHED` | EC2 instances without instance profiles |
| `ENCRYPTED_VOLUMES` | Unencrypted EBS volumes |
| `RDS_STORAGE_ENCRYPTED` | Unencrypted RDS instances |
| Custom rule | Database in non-prod not tagged with compliance scope |

**Integration point**: AWS Config findings automatically create JIRA tickets via EventBridge → Lambda → JIRA API. Environment owner and platform team alerted within 15 minutes of drift detection.

## Kubernetes-Specific Tools

### Rancher

**Category**: Multi-cluster Kubernetes management  
**Primary use**: Manage multiple EKS clusters (one per major tier or set of tiers) from a single control plane.

| Capability | Use Case |
|-----------|---------|
| Cluster catalog | Inventory all non-prod and prod clusters |
| Namespace management | Create and manage namespaces across clusters |
| RBAC management | Assign team access to namespaces across clusters |
| Monitoring integration | Prometheus/Grafana integration per cluster |

**Strengths**: Simplifies multi-cluster operations; unified RBAC across clusters; useful at T-Mobile scale with multiple EKS clusters.

---

### Loft / vCluster

**Category**: Virtual cluster / lightweight environment provisioning  
**Primary use**: Create fully isolated Kubernetes environments inside a single EKS cluster using virtual clusters.

| Capability | Notes |
|-----------|-------|
| Virtual clusters | Each team gets a virtual Kubernetes cluster inside a shared physical cluster |
| Full isolation | Separate API server, control plane components — stronger isolation than namespace |
| Cost efficiency | 10–20x more cost-effective than separate EKS clusters per environment |
| Ephemeral | Create and destroy virtual clusters on-demand in < 60 seconds |

**When to prefer vCluster over namespaces**: Teams that need to install cluster-level resources (CRDs, cluster-scoped RBAC) in their environment; teams with conflicting CRD versions.

## Cost Management Tools

### Infracost

**Category**: Infrastructure cost estimation  
**Primary use**: Estimate cost of new environments before provisioning; surface cost impact of configuration changes.

**Integration**: Runs in CI pipeline on every Terraform PR. Posts cost estimate as PR comment. Flags changes that increase estimated cost by > 20%.

### Kubecost

**Category**: Kubernetes cost allocation  
**Primary use**: Allocate Kubernetes compute cost to namespaces, teams, and environments.

**Integration**: Deployed to each EKS cluster. Feeds cost data to Backstage cost plugin and monthly environment cost report.

## Tool Selection Decision Matrix

| Need | Primary Tool | Alternative |
|------|-------------|-------------|
| Infrastructure provisioning | Terraform | Crossplane (for self-service) |
| Application deployment | Helm + ArgoCD | Helm standalone (without GitOps) |
| Self-service IDP | Backstage | Port, Cortex (commercial) |
| Policy enforcement | Conftest + OPA | Kyverno (Kubernetes-native) |
| Drift detection (infra) | Terraform plan + AWS Config | Spacelift (commercial) |
| Drift detection (app) | ArgoCD sync status | Flux (alternative GitOps) |
| Ephemeral environments | Kubernetes namespaces | vCluster (stronger isolation) |
| Cost allocation | Kubecost | AWS Cost Explorer + tags |
| Secret management | HashiCorp Vault | AWS Secrets Manager (simpler) |
