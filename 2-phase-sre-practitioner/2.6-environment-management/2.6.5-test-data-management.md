# Chapter 5: Test Data Management

## Why Test Data Is the Hardest Part of Environment Management

Compute is easy to provision. Configuration can be codified. Test data is different: it must be realistic enough to trigger production-representative behavior, compliant enough to prevent PII exposure, fresh enough to reflect current production patterns, and stable enough to support repeatable test execution. These requirements are in tension with each other.

Most organizations solve test data by doing one of three wrong things:
1. Using production data directly in non-prod (PII violation, compliance exposure)
2. Using a synthetic dataset that has not been updated in 18 months (stale data debt)
3. Using each developer's personal fixtures (non-representative, non-repeatable)

None of these works at scale. This chapter defines the governance model for test data that is realistic, compliant, fresh, and reproducible.

---

## Test Data Categories

| Category | Description | Use Cases | Compliance Risk |
|----------|-------------|-----------|-----------------|
| Synthetic (generated) | Programmatically generated records that match production schema | Unit tests, feature branch environments, contract tests | Low — no real data |
| Anonymized production | Production data with PII fields masked using format-preserving techniques | QA, UAT, Performance — where representative data distribution is required | Medium — masking quality determines compliance |
| Static fixtures | Hand-crafted datasets for specific test scenarios | Edge case testing, regression test anchors | Low if no real PII |
| Dynamic data | Data created by test execution itself | Integration tests that exercise write paths | Low — test-generated |

**Decision rule**:
- Feature branch, integration, dev: synthetic data
- QA: anonymized production snapshot (refreshed per sprint)
- Performance: anonymized production snapshot at representative volume (≥ 50% production record count)
- UAT: anonymized production snapshot (freshest available — business users validate real data shapes)
- Staging: anonymized production snapshot or production data in-place (if staging IS production and blue-green is the model)

## PII/Sensitive Data in Test Environments: The Compliance Problem

Unmasked PII in a non-production environment is a compliance violation under:
- **GDPR**: Personal data of EU residents may not be used for testing without explicit consent (which no consumer has given for testing)
- **CCPA**: California consumer personal information has access and deletion rights that apply regardless of environment
- **PCI-DSS**: Cardholder data (PANs, CVVs) may not exist in non-production environments in unmasked form
- **HIPAA**: Protected health information (PHI) requires the same controls in all environments, making non-prod PHI nearly impossible to comply with

**At T-Mobile**: Customer PII (MSISDN/phone numbers, account numbers, SSN fragments, billing addresses, device IMEIs) must be masked before entering any non-production environment. This is non-negotiable compliance requirement, not a best practice.

**The compliance exposure of production data in non-prod:**
- Non-prod environments have wider access than production (developers access non-prod directly; production access is restricted)
- Non-prod environments are not monitored at production security levels
- Non-prod data is not subject to production-level backup and deletion compliance
- A production data breach via a non-prod environment is indistinguishable from a production breach in its regulatory consequences

## Data Masking and Anonymization

### What Must Be Masked

| Data Type | Masking Requirement | Format-Preserving? |
|-----------|--------------------|--------------------|
| Social Security Number (SSN) | Replace with synthetic SSN (same format) | Yes — format must be valid |
| Phone number (MSISDN) | Replace with non-assigned number range | Yes — 10-digit format preserved |
| Email address | Replace with `<synthetic-id>@testdomain.example.com` | Partial — domain always test domain |
| Credit card PAN | Replace with synthetic PAN (Luhn-valid) | Yes — passes format validation |
| Name (first/last) | Replace with synthetic name from approved name pool | No — format not required |
| Physical address | Replace with synthetic address | Partial — city/state/ZIP must be valid format |
| Account number | Replace with synthetic account number (same length, same check digit format) | Yes |
| Device IMEI | Replace with synthetic IMEI (Luhn-valid) | Yes |
| IP address | Replace with RFC 5737 documentation range (192.0.2.0/24) | No — just consistent within dataset |

### Format-Preserving Encryption (FPE)

Format-preserving encryption allows masked values to pass format validation (regex checks, Luhn algorithm, length checks) while containing no real PII. This is critical for test data that must pass application-level validation.

Example: A test payment processor that validates Luhn algorithm on credit card numbers must receive a masked PAN that passes the Luhn check. FPE achieves this.

### Anonymization Approaches

| Approach | Description | When to Use | Limitation |
|----------|-------------|-------------|-----------|
| Substitution | Replace PII field with synthetic value | Most PII fields | Relationship between fields may break (name no longer matches address) |
| Shuffling | Randomize values within the column across rows | Non-linked PII fields | Must not apply to fields that are referenced across tables |
| Generalization | Replace specific value with less-specific (exact age → age range) | Analytics datasets | Reduces data utility for testing |
| Nullification | Set PII field to null | Fields not required for testing | Breaks tests that expect non-null values |
| FPE | Encrypt while preserving format | Fields that must pass format validation | Requires key management |

**Tool landscape:**

| Tool | Capability | Integration Point |
|------|-----------|-------------------|
| AWS Glue DataBrew | Point-and-click data masking transforms | S3-based pipeline |
| Faker (Python) | Synthetic data generation | CI pipeline scripts |
| Presidio (Microsoft) | NLP-based PII detection + masking | Pre-masking audit |
| DataVeil | Enterprise database masking | Database-to-database masking pipeline |
| Delphix | Complete data masking + provisioning platform | Enterprise; high cost |
| Custom Python (Faker + SparkSQL) | Organization-specific masking rules | Mid-market option for custom PII schemas |

## Test Data Provisioning: How Teams Get Data

**Target model: self-service with guardrails**

Teams should be able to request a test data refresh without filing a ticket with the data engineering team. The request goes through the data provisioning pipeline, which applies masking rules, validates compliance, and delivers the dataset to the environment.

**Self-service data provisioning portal:**

```
Developer requests data refresh for QA environment:
    → Select environment: qa-east-payments
    → Select dataset: payments-full (standard masked production snapshot)
    → Select date: latest available (refreshed nightly)
    → Submit

Pipeline:
    → Check: is requester authorized for this environment? (RBAC)
    → Check: is masked snapshot available for requested date? 
    → Load masked snapshot into environment database
    → Run data integrity validation (record counts, referential integrity)
    → Notify requester: "Data refresh complete. 2.3M records loaded."
```

**Refresh cadence by tier:**

| Tier | Data Refresh Cadence | Trigger |
|------|---------------------|---------|
| Feature Branch | Synthetic data — no refresh needed | N/A |
| Integration | Weekly (automated) | Monday 6am UTC |
| QA | Per sprint start + on-demand | Sprint planning event or developer request |
| Performance | Per test cycle | Performance test schedule |
| UAT | Per UAT cycle (typically per release) | Product owner trigger |
| Staging | Per release | Release manager trigger |

## Test Data Versioning

Some tests require a specific data state to be repeatable. Regression tests that depend on known-good data states cannot be run against a freshly refreshed dataset that has different record counts or changed data distributions.

**Versioned dataset model:**

```
data-snapshots/
├── payments/
│   ├── 2026-Q1-baseline.sql.gz      ← Q1 regression anchor
│   ├── 2026-Q2-baseline.sql.gz      ← Q2 regression anchor
│   └── latest.sql.gz                ← most recent masked snapshot
├── auth/
│   ├── 2026-Q1-baseline.sql.gz
│   └── latest.sql.gz
└── notifications/
    └── latest.sql.gz
```

**Versioned snapshot governance:**
- Quarterly snapshots retained for 12 months (regression test anchor)
- Weekly snapshots retained for 4 weeks (recent regression)
- Latest snapshot refreshed nightly from production masking pipeline
- Snapshot metadata: record counts, masking run ID, masking policy version, source production date

## Database Snapshot Strategy: The Production → Non-Prod Pipeline

```
Production Database (RDS/Aurora/Cassandra/MongoDB)
    ↓
[Daily 2am UTC] Automated snapshot → S3 (encrypted, same-region)
    ↓
Masking Pipeline (AWS Glue / custom Spark job)
    ├── Load snapshot into isolated processing environment
    ├── Apply masking rules (per data classification policy)
    ├── Validate: zero unmasked PII fields (Presidio scan)
    ├── Validate: record counts within 5% of production (volume check)
    └── Export masked snapshot to S3 (non-prod access only)
    ↓
Non-Prod Data Store
    ↓
Environment data refresh (on-demand via self-service portal)
```

**Masking pipeline SLA**: Masked snapshot available by 6am UTC daily. If masking pipeline fails: alert to data engineering on-call; fallback to previous day's masked snapshot (declared in status page update to environment owners).

**Access control for snapshots:**
- Production S3 snapshot bucket: accessible only by masking pipeline IAM role
- Non-prod masked snapshot bucket: accessible by non-prod environment IAM roles + data engineering team
- Production data never directly accessible from non-prod environment

## Data Debt: Test Environments With Stale Data

Data debt in test environments is the accumulation of stale, unrepresentative, or incorrect test data that no longer reflects production behavior. It has the same consequence as configuration drift: tests that pass against data debt do not predict production behavior.

**Data debt indicators:**

| Indicator | Measurement | Threshold |
|-----------|------------|-----------|
| Data age | Days since last refresh | > 30 days for QA = data debt |
| Record count divergence | Non-prod record count vs. production | < 10% of production = data debt for Performance |
| Missing data variants | Production data patterns not in non-prod | Any production edge case not represented |
| Broken referential integrity | Foreign key violations in non-prod data | Any violation = data debt |
| Unmasked PII detected | Presidio scan finding in non-prod | Any finding = compliance incident, not just debt |

**Data debt quantification:**
- Count environments with data older than SLA → cost of delayed test confidence
- Count "data-caused test failures" per quarter → engineering cost of debugging data issues vs. code issues
- Count environments that trigger "data environment" as cause in production incident RCA → revenue impact

**Director Governance Gate**: Data age exceeding refresh SLA must be reported in the weekly environment health dashboard. Any QA environment with data older than 30 days is a P2 governance issue. Any non-prod environment with unmasked PII is a P0 compliance incident.
