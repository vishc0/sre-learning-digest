# Chapter 6: Performance Environment Governance

## Why Performance Environments Are Special

Every other non-production tier tests a qualitative question: does the software meet its functional requirements? The performance environment tests a quantitative question: does the software meet its performance requirements under realistic load?

This distinction creates a different class of governance requirements. A QA environment that differs from production in middleware configuration will produce functionally incorrect test results. A performance environment that differs from production in any measurable way — instance type, JVM heap, connection pool size, data volume, network topology — will produce performance test results that are quantitatively meaningless for production prediction.

A performance test run against an environment that is not production-parity is not a performance test. It is a performance test of a different system.

## Performance Environment Configuration Requirements

The performance environment must match production in every dimension that affects performance behavior. This is a stricter standard than any other non-production tier.

### Compute

| Parameter | Production | Performance Environment | Variance Allowed |
|-----------|-----------|------------------------|-----------------|
| Instance family | m5.2xlarge | m5.2xlarge | None — same family, same generation required |
| Instance count | 20 worker nodes | 10 worker nodes (50% scale) | Scale reduction acceptable; family change not |
| CPU allocation (JVM) | 4 vCPU | 4 vCPU | None — per-pod CPU limit must be identical |
| Memory allocation (JVM) | 8Gi | 8Gi | None — heap behavior is not linear; smaller heap changes GC patterns |
| Storage IOPS | 3000 IOPS gp3 | 3000 IOPS gp3 | None — disk I/O bound workloads will show different behavior |
| Network throughput | 10Gbps | 10Gbps | None for NIC; acceptable to reduce node count |

**Why JVM heap cannot differ**: JVM garbage collection patterns are non-linear with heap size. A service configured with 4GB heap in the performance environment but 8GB heap in production will show GC pause patterns that are meaningless for production prediction. The GC algorithm (G1GC, ZGC) may behave differently at different heap sizes for the same workload.

**Why instance family cannot differ**: Burstable instances (t3/t2) accumulate CPU credits during idle periods and burst above their baseline. Fixed-performance instances (m5/c5/r5) do not. A performance test on a burstable instance during its credit accumulation phase will show performance that is impossible in production when credits are exhausted.

### Middleware Configuration

All middleware (RabbitMQ, Kafka, TIBCO) must be configured identically to production — not just "similar."

| Middleware Parameter | Production | Performance Env | Consequence of Delta |
|---------------------|-----------|-----------------|---------------------|
| RabbitMQ prefetch count | 50 | 50 | Consumer throughput ceiling differs |
| RabbitMQ queue max-length | 100000 | 100000 | Back-pressure behavior differs |
| RabbitMQ connection pool per service | 20 | 20 | Concurrency behavior differs |
| Kafka partition count per topic | 24 | 24 | Consumer parallelism differs |
| Kafka linger.ms (producer) | 5ms | 5ms | Producer batching behavior differs |
| Kafka max.in.flight.requests | 5 | 5 | Ordering and throughput differ |
| Thread pool sizes (app-level) | 200 threads | 200 threads | Concurrency and queue behavior differ |
| Database connection pool size | 20 connections | 20 connections | Database contention behavior differs |
| HTTP client connection timeout | 5000ms | 5000ms | Timeout behavior under load differs |

**Director Governance Gate**: Performance environment middleware configuration must be validated against production via automated diff before each performance test cycle. Any undocumented delta invalidates the test results and blocks performance sign-off.

### Data Volume Requirements

| Data Requirement | Performance Environment Target | Minimum Acceptable |
|-----------------|------------------------------|-------------------|
| Customer records | 100% of production | 50% — below 50% risks pagination/aggregation behavior differences |
| Transaction history | 12 months (same as production) | 6 months minimum |
| Reference data (products, plans) | 100% | 100% — reference data is small; no excuse for less |
| Time-series/event data | 90-day window | 30-day minimum |
| Index characteristics | Same index structure + statistics | Same — different statistics change query plan behavior |

**Why data volume cannot be significantly reduced**: Many performance failures are data-volume-dependent. A query that returns 100 rows from 10K records in 50ms may return 50,000 rows from 10M records in 45 seconds. This failure mode will not appear in a performance environment with 1% of production data.

Database query optimizer behavior is also data-volume-dependent. Query plans change with cardinality estimates. A query plan that is efficient on 50K records may full-scan on 10M records because the optimizer's cost estimates flip.

## Performance Environment Provisioning

### Dedicated vs. Shared

| Model | Description | Recommended For |
|-------|-------------|----------------|
| Dedicated | Performance environment infrastructure is exclusively allocated to performance testing | T-Mobile scale: recommended. Performance tests cannot share infrastructure with other test tiers. |
| Shared (scheduled) | Infrastructure is shared with other test tiers; scheduled windows for performance exclusive use | Small organizations with infrequent performance testing |
| On-demand (cloud burst) | Performance environment spun up in cloud on demand; destroyed after test | Cost-effective for infrequent tests; configuration parity risk if not fully automated |

**At T-Mobile scale**: A dedicated performance environment is the correct model. Shared environments introduce noise from other teams' activity, making performance test results non-reproducible.

### Reservation and Scheduling System

The performance environment is a shared, limited resource. Multiple teams competing for it without a scheduling system create:
- Test result contamination (two teams' tests running simultaneously)
- Resource starvation (one team's test consumes all available capacity)
- No accountability for environment state between tests

**Reservation model:**

```
Performance Environment Calendar:
- Reservation unit: 4-hour block (minimum) or 24-hour block (full test suite)
- Lead time: 5 business days minimum for standard reservation
- Emergency reservation: approved by Platform SRE + VP SRE, 48-hour lead time
- Maximum continuous reservation: 72 hours (soak test exception requires SRE approval)

Reservation requirements:
- Team name + use case
- Test type (load / stress / soak / spike / chaos)
- Expected load profile (RPS, concurrent users, message throughput)
- Data requirements (which dataset, which version)
- Configuration requirements (any deviations from standard config must be pre-approved)
- Results storage location (where will results archive go)
```

**Playbook**: [Performance Environment Booking Playbook](../playbook/performance-environment-booking-playbook.md)

## Performance Test Types and Environment Requirements

Different performance test types have different environment requirements. Not all tests require the full performance environment.

| Test Type | Purpose | Environment Requirement | Duration |
|-----------|---------|------------------------|---------|
| **Load test** | Validate behavior at expected production load | Production-parity config + ≥ 50% data volume | 1–4 hours |
| **Stress test** | Find the system's breaking point | Production-parity config + ≥ 50% data volume | 2–8 hours |
| **Soak test** | Validate stability under sustained load (memory leaks, connection pool exhaustion) | Production-parity config + ≥ 50% data volume | 24–72 hours |
| **Spike test** | Validate behavior under sudden traffic spikes | Production-parity config; data volume less critical | 1–2 hours |
| **Chaos test** | Validate resilience when components fail under load | Production-parity + chaos engineering tools (Chaos Monkey, LitmusChaos) | 2–4 hours |
| **Baseline test** | Establish or verify performance baseline after a change | Production-parity required; abbreviated data volume acceptable | 30–60 minutes |

**What the VP measures**: Performance test pass rate (within SLA thresholds) per release cycle. Tests that do not have production-parity environments have asterisks on their results — they are noted in the release decision as "performance validated under non-parity conditions."

## Performance Environment Access Control

The performance environment must be isolated from:
1. **Accidental production traffic**: DNS misconfiguration that routes production traffic to the performance environment would corrupt test results and potentially corrupt performance test data
2. **Accidental developer access during test**: A developer deploying a new service version mid-test invalidates the test run
3. **External test tool access**: Load generators must be internal — external load generator IPs must not be blocked by production-side network ACLs during test

**Access controls:**

| Access Type | Control |
|------------|---------|
| Network ingress from production | Blocked — performance environment VPC has no peering to production VPC |
| Developer deployment during test | Deployment pipeline checks performance environment booking calendar; blocks deployment if test is active |
| Load generator access | Dedicated load generator subnet with performance environment security group rules |
| Data access (read) | Authorized by environment booking record; removed after test window |
| Configuration changes during test | Blocked by change control — any change during active test invalidates results |

## Performance Environment Maintenance

As production evolves, the performance environment must evolve in lockstep. A performance environment that is 6 months behind production is not testing production behavior.

**Maintenance triggers:**

| Production Change | Performance Environment Action | Lead Time |
|------------------|-------------------------------|-----------|
| Infrastructure upgrade (EKS version, instance type) | Apply same upgrade to performance environment | Within 2 sprints |
| Middleware upgrade (Kafka/RabbitMQ version) | Apply same upgrade + validate configuration parity | Within 2 sprints |
| New service added to production | Add service to performance environment deployment | Before first performance test post-launch |
| Database schema change | Apply schema migration + data volume refresh | Before next test cycle |
| New external dependency added | Add dependency stub/integration to performance environment | Before next test cycle |
| JVM configuration change in production | Apply change to performance environment | Before next test cycle |

**Maintenance governance**: Performance environment maintenance is tracked in the same backlog as production infrastructure work. It is not optional. A performance environment that is out of sync with production fails its governance gate and its test results are flagged as non-authoritative.

## Performance Test Result Archive

Performance test results must be retained and organized for trend analysis. A single performance test result tells you how the system performed. A 12-month archive of performance test results tells you whether performance is improving, stable, or degrading over time.

**Archive structure:**

```
performance-results/
├── <service-name>/
│   ├── <YYYY>-<MM>-<DD>-<test-type>/
│   │   ├── test-config.yaml          ← Load profile, environment config, tool config
│   │   ├── environment-parity.json   ← Diff of performance env vs. production at test time
│   │   ├── raw-results/              ← Tool-specific output (JMeter JTL, Gatling reports, k6 JSON)
│   │   ├── summary-report.md         ← Human-readable: RPS, p50/p95/p99 latency, error rate, SLA pass/fail
│   │   └── trend-delta.md            ← Comparison to previous test: better/worse/stable
│   └── baseline.yaml                 ← Current accepted baseline for this service
```

**Retention policy:**
- Raw results: 12 months
- Summary reports: 36 months (performance trend analysis)
- Baseline files: indefinite (governs current release sign-off criteria)

**Trend analysis**: Quarterly performance trend report compares p99 latency, error rate, and throughput ceiling across the last 4 test cycles for each service. Degrading trend = mandatory performance investigation before next release.
