# Chapter 09 — Enterprise Environment Pipeline Architecture

> **Audience**: Director/VP of Operations candidates. This chapter assumes you have managed release pipelines, not that you are learning what one is.

---

## Why a Pipeline Model, Not a Gate Model

The instinct at most organizations is to treat environments as checkboxes — "it went through QA, it went through UAT, ship it." That instinct produces brittle releases, mystery defects in production, and environments that lie to you. The pipeline model treats each environment as a **station with a contract**: specific inputs it accepts, specific outputs it produces, and a defined promotion gate that must pass before the artifact advances.

At T-Mobile scale — thousands of microservices, multiple domain teams, monthly release bundles carrying 60–100 changes — the pipeline model is not an aspiration. It is the only way to maintain release velocity without accepting systemic instability.

The pipeline described in this chapter has eleven stations. Each station is described with: purpose, infrastructure characteristics, governance, promotion gate criteria, sign-off authority, and SLA. The gate matrix at the end of this chapter consolidates all eleven into a single reference table.

---

## Station 1: DEV (Individual Developer Environments)

**Purpose**: The DEV environment is the developer's laboratory. Its job is to enable fast iteration — code, run, break, fix — without any shared blast radius. Nothing that happens in a DEV environment should matter to any other team.

**Characteristics**:
- Ephemeral by design: created per developer, per branch, or per pull request — never permanently allocated
- Spin-up time target: under 30 minutes from `git checkout` to running environment. If it takes longer, developers will stop using it and start sharing DEV-INT, which destroys DEV-INT's stability
- Infrastructure footprint: minimal — single-pod deployments, in-memory databases or embedded DB acceptable, mock dependencies for external services
- Dependency handling: real integrations with downstream services are optional and discouraged. Use WireMock, contract mocks, or locally containerized stubs. The goal is isolation, not fidelity
- Provisioning: Helm chart with a developer values override file + a `make dev-up` wrapper that handles cluster context switching. Terraform if the dev environment includes cloud resources (S3 buckets, SNS topics), but scoped to a developer-specific namespace prefix

**Who owns it**: The developer. Central SRE does not monitor, page on, or restore DEV environments. If your environment is broken, you rebuild it.

**Promotion gate to DEV-INT**:
1. Unit test suite passes (≥ 95% pass rate, zero P0 test failures)
2. Code review approved by at least one peer engineer
3. Static analysis scan completed (SAST), no HIGH or CRITICAL findings unresolved
4. Pull request merged to sprint branch — not just approved, merged

**SLA**: None. Best-effort. The central SRE platform ensures the scaffolding works (Helm chart, base images, CI pipeline). Anything above the platform layer is the developer's problem.

## Station 2: DEV-INT (Development Integration Environment)

**Purpose**: DEV-INT is where individual developer work collides with reality for the first time. Its job is to surface cross-service integration failures early — before they contaminate QA. It is a shared environment representing the current sprint's HEAD across all teams working on the same release bundle.

**The core challenge with DEV-INT**: multiple teams are deploying simultaneously. Team A's service breaks Team B's integration test at 2pm on a Tuesday, and nobody knows whose fault it is. This is the "everyone's breaking it" problem, and it is not a failure of process — it is the expected behavior of a shared integration environment during active development. The goal is not to eliminate it; it is to contain it.

**Governance mechanisms that make DEV-INT usable**:
- **Integration freeze windows**: in the 48 hours before a DEV-INT regression test run, no new deployments are allowed. This stabilizes the environment long enough to get a meaningful test result.
- **Service ownership labeling**: every deployment carries team ownership metadata. When an integration test fails, the routing to the responsible team is automatic — not a 30-minute Slack thread about who owns the failing service.
- **Canary deployment to DEV-INT**: new service versions are deployed to 1 pod first, not full replacement. Integration test suite runs against the canary. If it passes, full deployment proceeds.

**Characteristics**:
- Shared across all teams contributing to the current sprint
- All services deployed at current sprint HEAD — not release-tagged, not pinned to a previous stable version
- Semi-stable: breakage is expected but must be restored within 4 hours during business hours
- Data: synthetic dataset refreshed weekly from a masked production snapshot. Not representative at volume, but representative at shape.

**Promotion gate to PROJECT-QA**:
1. Integration test suite passes: contract tests, API tests, message flow tests (Kafka consumer→producer round-trip, RabbitMQ queue drain with expected consumer behavior)
2. No P1 or P2 integration defects open against the promoting service
3. DEV-INT has been stable (no P1 incidents) for the 24 hours preceding promotion

**SLA**: 95% uptime during business hours (8am–8pm local team time). 4-hour restoration SLA for P1 incidents. On-call rotation covers DEV-INT but it is shared rotation, not dedicated.

## Station 3: PROJECT-QA (Per-Project Functional QA Environments)

**Purpose**: PROJECT-QA is where a specific project or feature bundle gets isolated functional testing. The key word is isolated — PROJECT-QA environments are namespaced to a project. At any given time there may be QA-PROJ-PAYMENTS, QA-PROJ-AUTH, QA-PROJ-NETWORK-OPS all running simultaneously, each at a different maturity stage, each with a different sign-off status.

**Why parallel PROJECT-QA environments are non-negotiable**: without them, all projects compete for the same QA environment. A blocking P1 defect in the payments project holds up auth testing. A flaky test in network-ops restarts the QA clock for everyone. Serialization of QA across projects is one of the most common causes of release velocity collapse at enterprise scale.

**Provisioning model**: each PROJECT-QA environment is provisioned from the shared baseline using project-specific values overrides:
```
base-values.yaml           (shared across all PROJECT-QA instances)
  └── project-payments-overrides.yaml   (project-specific config, feature flags, test data set)
  └── project-auth-overrides.yaml
  └── project-network-ops-overrides.yaml
```

The base values are managed by SRE. The override files are owned by the project team and reviewed by SRE before provisioning.

**Configuration**: project-specific feature flags in enabled state, project-specific test data loaded, service versions pinned to the project's candidate release tag (not HEAD — HEAD is DEV-INT's domain).

**Promotion gate to BUNDLE-QA**:
1. Functional test pass rate ≥ 98% (not 100% — flaky test tolerance exists, but it is documented and tracked)
2. All P1 and P2 defects resolved or waivered with documented Risk Acceptance signed by Engineering Manager and Product Owner
3. QA test lead sign-off — the test lead must explicitly declare the project ready, not just observe that tests passed
4. No unresolved service dependency issues (downstream mocks replaced by real services, or dependency explicitly waivered)

**SLA**: 98% uptime during the project's active testing window. 2-hour restoration SLA. PROJECT-QA incidents that breach 2 hours automatically escalate to the SRE manager on-call.

## Station 4: BUNDLE-QA (Monthly Release Bundle — Alternating A/B with Trueup)

**Purpose**: BUNDLE-QA is the first environment where the complete monthly release bundle is tested as a unit — all projects, all changes, together. This is the full integrated regression validation. It is the most complex environment to operate correctly, which is why it is the environment most likely to accumulate drift and lie to you.

### The A/B Architecture

BUNDLE-QA is never a single environment. It is always two: BUNDLE-QA-A and BUNDLE-QA-B. These alternate:

```
Month N testing:   BUNDLE-QA-A is ACTIVE (testing Month N bundle)
                   BUNDLE-QA-B is being TRUED UP to production state for Month N+1

Month N+1 testing: BUNDLE-QA-B is ACTIVE (testing Month N+1 bundle)
                   BUNDLE-QA-A is being TRUED UP to production state for Month N+2
```

This design solves three problems simultaneously:
1. **No downtime between release cycles**: when Month N testing completes, Month N+1 testing can begin immediately on the already-prepared B environment
2. **No state contamination**: Month N's test artifacts, defect data loads, and environment mutations do not pollute Month N+1 testing. B was trued up from production, not inherited from A
3. **Trueup time is not on the critical path**: because B's trueup runs while A is actively testing, the trueup's duration does not delay the release cycle — as long as B is ready before A's testing completes

### What Trueup Means for BUNDLE-QA

Trueup is not a refresh. It is a controlled reset to a known reference baseline. For BUNDLE-QA, the reference baseline is the production state immediately after the previous release:
1. Production snapshot captured (database dump, configuration export, service versions)
2. PII/PCI masking pipeline applied to the data snapshot
3. BUNDLE-QA-B infrastructure rebuilt or delta-applied from baseline (see ch10 for rebuild vs. delta decision)
4. Configuration synced: RabbitMQ vhosts and queues, Kafka topics and consumer groups, MuleSoft flows, application feature flags, K8s ConfigMaps and Secrets
5. Masked data loaded and referential integrity validated
6. Smoke test suite run against the trued-up environment
7. "Ready" declaration issued by SRE — this is a formal gate, not an informal check

**Trueup schedule**: trueup for the B environment must complete within the first 5 business days after production release. This gives a comfortable buffer before the next bundle's testing window opens (typically 2 weeks into the month).

**Trueup failure escalation**: if the trueup cannot complete within 5 business days, the VP of Operations is notified. Options are: extend the trueup window (delays bundle testing start), use the previous cycle's environment with known drift (documented risk), or defer the release (most expensive option).

### BUNDLE-QA Promotion Gate

The gate from BUNDLE-QA is the most comprehensive gate in the pipeline:
1. Full regression suite pass — not just new feature tests, everything. New features are tested in PROJECT-QA. BUNDLE-QA is about proving nothing existing broke.
2. Performance smoke test within 20% of established performance baseline (PERF-ENV owns the baseline; BUNDLE-QA runs a 4-hour abbreviated smoke version, not the full 48-hour soak)
3. All domain sign-offs received — UX, Finance, and Business must all have signed off on their respective environment results before BUNDLE-QA promotion is allowed. This makes BUNDLE-QA the aggregation point for all parallel sign-off streams.
4. Zero unresolved P1 or P2 defects in BUNDLE-QA regression

**SLA**: 99% uptime during the bundle testing window (typically the last 2 weeks of the month). Any P1 incident in BUNDLE-QA during this window goes immediately to VP Operations — not after a 2-hour restoration timeout, immediately.

## Station 5: UX-ENV (User Experience Validation Environment)

**Purpose**: UX-ENV exists because general QA does not catch UX failures. A functional test that asserts "the checkout page loaded" does not assert "the checkout page is navigable by a screen reader user" or "the redesigned layout has a 12% lower abandonment rate than the baseline." These require dedicated environments with browser automation, accessibility tooling, and real UX practitioners in the loop.

**Characteristics**:
- Front-end focused: browser automation infrastructure (Selenium Grid or Playwright cluster) sized for parallel browser sessions
- A/B testing capable: traffic splitting at the load balancer level to serve variant UX to different sessions
- Accessibility testing integrated: automated a11y scanning (axe-core) runs on every page in the release scope
- User data requirement: representative user persona data loaded — not volume production data, but shape-accurate data that allows realistic UX journeys

**Who uses it**: UX designers, product managers, accessibility specialists, and UX researchers. Not QA engineers running functional tests — that happened in PROJECT-QA.

**Run in parallel with BUNDLE-QA**: UX-ENV and BUNDLE-QA are not sequential. While BUNDLE-QA is running full regression, UX-ENV is running UX validation. Both must pass before UAT begins. The parallel run typically saves 1–2 weeks of calendar time compared to a sequential model.

**Sign-off authority**: UX Lead + Product Owner. Neither alone is sufficient — UX Lead validates the experience, Product Owner validates alignment with product requirements.

---

## Station 6: FINANCIAL-DOMAIN-ENV (Financial Domain Testing Environment)

**Purpose**: Financial testing cannot share infrastructure with general QA. The reasons are operational (specific regulatory data sets required), compliance (PCI DSS mandates separation of cardholder data environments), and organizational (Finance and Compliance own the sign-off and need their own space to run it without being blocked by QA scheduling).

**What is tested here**:
- Financial calculation accuracy: GL entry generation, billing cycle calculations, refund processing, proration logic
- Regulatory scenario coverage: specific scenarios mandated by applicable regulations (state tax calculations, GAAP revenue recognition edge cases)
- Payment flow end-to-end: authorization → settlement → reconciliation, including reversal paths and chargeback flows
- GL reconciliation: system-generated journal entries reconciled against expected entries for the test scenario set

**Data governance**: financial test data carries PCI implications. The test data set is:
- Generated synthetically (not masked from production cards/accounts), or
- Masked through an approved PCI-compliant masking pipeline with explicit Finance-IT sign-off on the masking methodology
- Audit log maintained for all test data access — who ran what scenario, when, with what test account numbers

**Runs in parallel with BUNDLE-QA and UX-ENV**: all three parallel streams must complete before UAT is allowed to begin.

**Sign-off authority**: Finance Manager + Compliance Officer. Both must sign. Missing either signature is a release blocker.

## Station 7: UAT / Business Validation

**Purpose**: UAT validates that the release meets business requirements — not technical specifications, business requirements. The distinction matters. A system can pass every technical test and still fail to support the business process it was built for.

**UAT is sequentially gated**: UAT does not begin until BUNDLE-QA, UX-ENV, and FINANCIAL-DOMAIN-ENV have all received sign-off. This is the one sequential gate in the pipeline's parallel section. The reasoning: business stakeholders' time is expensive and limited. Running UAT against a release that still has open UX or financial defects wastes that time and erodes confidence in the process.

**Who runs UAT**: business analysts, domain SMEs, and in some cases actual internal end users — the people who will use the system, not the people who built it. UAT scripts are written by BAs, not QA engineers.

**Characteristics**:
- Stable environment: UAT has a change freeze once testing begins. No deployments during active UAT unless a blocker defect requires a fix
- Representative business data: test data that matches the volume and complexity of real business operations, not just happy-path synthetic data
- Accessible to non-technical users: no command-line access required, no VPN knowledge required, browser-based access with single sign-on

**UAT defect classification and release impact**:

| Severity | Definition | Release Impact |
|----------|-----------|----------------|
| P1 | Business process cannot complete | Release blocker — no exceptions |
| P2 | Business process completes with unacceptable workaround | Release blocker, can be waivered with VP Operations + Business Owner joint approval and documented remediation timeline |
| P3 | Known defect with acceptable workaround | Release proceeds with defect documented in release notes and remediation ticket committed |
| P4 | Cosmetic or edge case | Release proceeds, tracked in backlog |

**Sign-off authority**: Business Owner per domain + Program Manager. The Business Owner signs off on the business process; the Program Manager signs off on the completeness of testing.

## Station 8: PERF-ENV (Performance Environment — Continuous Gate)

**Purpose**: PERF-ENV is the only environment in the pipeline that runs 24/7 regardless of release cycle. Its job is continuous performance baseline maintenance and release performance validation. It is the canary for slow-burn failures — memory leaks, GC pressure accumulation, query plan degradation — that are invisible in any test run shorter than 24 hours.

**Why continuous load matters**: a 30-minute load test will not find a memory leak that adds 2MB/hour to heap usage. A 4-hour load test will find it only if the leak is aggressive. A 48-hour soak test will find leaks that grow 500KB/hour — the kind that cause 3am production incidents two weeks after release.

**Configuration parity requirement**: PERF-ENV must match production in:
- JVM settings (heap sizes, GC algorithm, thread pool sizes)
- Connection pool sizes (database, MQ, HTTP client)
- Hardware class (same EC2 instance family, same memory:CPU ratio)
- Data volume rationale (same row counts, same queue depths, same cache population levels)

PERF-ENV does not need to be as large as production — it needs to be proportionally equivalent. A 10% scale PERF-ENV is acceptable if everything is proportionally reduced. A PERF-ENV running on under-powered instances produces meaningless baselines.

**Performance gate for a release candidate**:
1. Deploy release candidate to PERF-ENV
2. Run 48-hour soak test (continuous load, not burst load)
3. p95 latency must remain within 15% of established baseline for the duration
4. TPS throughput must remain within 10% of established baseline
5. Memory growth rate must be flat — a monotonically increasing heap is a leak, regardless of whether it triggers OOM within the test window
6. GC pause frequency must not increase > 20% from baseline
7. No pod restarts (OOMKilled, crash loops) during the soak period

If any criterion fails: **the release is blocked**. Not deferred pending analysis. Blocked until the performance root cause is identified and either fixed or risk-accepted with explicit Performance Engineering Lead + VP Operations sign-off.

**Sign-off authority**: Performance Engineering Lead + Platform SRE Lead.

**Cross-reference**: [Performance Engineering] — PERF-ENV baseline methodology and alert thresholds are defined in the Performance Engineering framework.

## Station 9: PRE-PROD (Pre-Production / Staging)

**Purpose**: PRE-PROD is the dress rehearsal. Its job is not to find defects — defects should be found in BUNDLE-QA. Its job is to validate that the deployment itself works, that runbooks are accurate, and that the on-call team is ready to support the production deployment.

**Production identity requirement**: PRE-PROD must be identical to production in every operationally significant way:
- Same infrastructure class (same EC2 instance types, same RDS instance class)
- Same configuration (same environment variables, same feature flag states as production baseline)
- Same network topology (same security group rules, same routing tables, same WAF rules)
- Same security controls (same IAM policies, same certificate configurations, same secrets management paths)

If PRE-PROD differs from production in any of these dimensions, the PRE-PROD validation is not a reliable predictor of production behavior. This is the most commonly violated requirement in enterprise environments — PRE-PROD drifts from production within weeks of each release because it is treated as a lower-priority environment.

**What happens in PRE-PROD**:
1. Deployment using production deployment automation — exact same Terraform plan, exact same Helm values, exact same deployment pipeline. If the pipeline fails in PRE-PROD, the release is blocked.
2. Runbook dry-run: on-call engineers execute the deployment runbook step by step, documenting any gaps or inaccuracies
3. Rollback rehearsal: rollback procedure executed in PRE-PROD to validate it works and to time it (rollback time is a key input to go/no-go decisions)
4. On-call readiness check: the engineers who will own the production deployment are briefed, dashboards are confirmed working, alert routing is validated

**Final go/no-go gate**:
1. Change Management CAB approval (the formal change record is approved)
2. All domain sign-off evidence collected and attached to the change record (BUNDLE-QA, UX-ENV, FINANCIAL-DOMAIN-ENV, UAT, PERF-ENV sign-offs)
3. Rollback plan documented and rehearsed
4. On-call team confirmed available and briefed
5. Maintenance window confirmed with all stakeholders

**Sign-off authority**: VP of Operations or delegated Release Manager. The Release Manager can sign off on routine releases; VP-level sign-off is required for changes with cross-domain impact or elevated risk.

## Station 11: Post-Production Validation

**Purpose**: deployment to production is not the end of the pipeline. Post-production validation is a defined monitoring and validation period that does not close until the release is confirmed stable.

**Timeline and checkpoints**:

| Time Window | Activity | Trigger for Rollback |
|-------------|----------|---------------------|
| 0–15 min | Smoke test execution — critical user journeys, health endpoints | Any P1 smoke test failure |
| 15–60 min | Canary monitoring — error rate, latency, saturation metrics | Error rate > 2× pre-deployment baseline OR latency p95 > 20% above baseline |
| 1–4 hr | Extended monitoring watch — synthetic monitoring, SLO burn rate | SLO burn rate indicates 24-hour budget will exhaust in < 12 hours |
| 4–24 hr | Final monitoring watch before change closure | Any P1 incident traced to the release |

**Automatic rollback trigger**: any P1 indicator firing within 4 hours of deployment triggers automatic rollback without requiring human approval. The on-call engineer executes; the post-mortem happens after stability is restored, not during.

**Change closure**: the change record is closed when:
1. All smoke tests pass
2. 4 hours of clean monitoring (no P1 or P2 indicators)
3. On-call engineer explicitly signs off in the change record
4. Downstream service owners confirm no unexpected behavior in their systems

## Trueup Schedule Design

### What Triggers a Trueup

Trueups are triggered by three conditions, in decreasing frequency:

1. **Scheduled (monthly)**: after each production release, the inactive BUNDLE-QA environment begins its trueup. This is the primary driver.
2. **Post-production release**: DEV-INT and PROJECT-QA environments receive a monthly scheduled trueup timed to the release cycle — typically the first Monday after production release.
3. **Defect-driven**: when a defect is traced to environment state divergence from production ("the BUNDLE-QA environment was running a deprecated config value that doesn't exist in production"), an emergency trueup is triggered for the affected environment.

### Trueup Steps for BUNDLE-QA

```
Production release completes
        │
        ▼
1. Production snapshot captured
   (DB dump, config export, service version manifest)
        │
        ▼
2. PII/PCI masking pipeline applied
   (field-level masking, referential integrity preserved)
        │
        ▼
3. Inactive BUNDLE-QA environment (B or A) rebuilt
   (infrastructure rebuild OR delta apply — see ch10)
        │
        ▼
4. Configuration sync
   (RabbitMQ vhosts/queues, Kafka topics/CGs, feature flags, K8s configs)
        │
        ▼
5. Masked data loaded
   (referential integrity validation, volume verification)
        │
        ▼
6. Smoke test suite executed
   (20-minute smoke, not full regression — enough to confirm environment is functional)
        │
        ▼
7. Ready declaration issued by SRE
   (formal gate — environment enters standby state, awaiting next bundle)
```

### Trueup Calendar: BUNDLE-QA Monthly Cycle

```
Week 1 (Days 1–5):   Production release (Day 1)
                     BUNDLE-QA-B trueup begins immediately (Day 1)
                     Target: B trueup complete by Day 5 (end of week 1)

Week 2 (Days 6–10):  BUNDLE-QA-A begins Month N+1 bundle testing
                     BUNDLE-QA-B in standby (trueup complete, waiting)
                     PROJECT-QA environments promoting to BUNDLE-QA-A

Weeks 3–4 (Days 11–20): BUNDLE-QA-A active regression testing window
                          UX-ENV and FINANCIAL-DOMAIN-ENV running in parallel

Day 21–22:           BUNDLE-QA-A sign-off + UAT begins
Day 23–25:           PRE-PROD validation + CAB
Day 28:              Production release
Day 29:              BUNDLE-QA-A trueup begins (it is now the inactive environment)
```

### Trueup Calendar for Other Environments

| Environment | Trueup Frequency | Trueup Reference Baseline | Lead Time Required |
|-------------|-----------------|--------------------------|-------------------|
| DEV-INT | Monthly (first Monday post-release) | Last sprint's release tag | 4 hours |
| PROJECT-QA | Per-project start | Current DEV-INT baseline | 8 hours |
| UX-ENV | Monthly (aligned with BUNDLE-QA cycle) | Production snapshot | 8 hours |
| FINANCIAL-DOMAIN-ENV | Monthly + regulatory calendar | Compliance-approved test data set | 1 business day |
| PRE-PROD | Before each release (continuous parity maintenance) | Production current state | 4 hours delta-apply |

### Trueup Failure: What Happens When the Next Bundle Is Starting

If the BUNDLE-QA-B trueup has not completed when BUNDLE-QA-A finishes testing and the next bundle needs to begin, three options exist — none of them good:

| Option | What It Means | Who Decides | Risk |
|--------|--------------|-------------|------|
| Extend trueup window | Delay bundle testing start by the trueup shortfall | VP Operations | Delays release date if not recovered |
| Use environment with known drift | Document the drift, accept the risk, begin testing | VP Operations + Risk sign-off | Test results may not reflect production behavior |
| Emergency rebuild | Accelerate trueup by adding resources, running steps in parallel | SRE Lead + Engineering Manager | Higher failure rate for rushed trueups |

The correct answer depends on how much drift exists and how critical the affected configuration dimensions are. A trueup that is 90% complete with only data volume not yet validated is a different risk than a trueup where configuration sync has not yet run.

## Parallel vs. Sequential Gate Map

Understanding which gates are parallel and which are sequential is essential for release cycle planning. The most common planning mistake is assuming all gates are sequential — this adds 4–6 weeks to a release cycle that can be compressed to 3 weeks with correct parallelism.

```
DEV → DEV-INT → PROJECT-QA (multiple, parallel per project)
                      │
                      ▼
               BUNDLE-QA (sequential — all PROJECT-QA must complete first)
                      │
              ┌───────┼───────┐
              ▼       ▼       ▼
           UX-ENV  FIN-ENV  PERF-ENV    ← all three run in parallel with BUNDLE-QA
              │       │       │
              └───────┼───────┘
                      │ (all three must sign off)
                      ▼
                    UAT         ← sequential — UAT cannot begin until all three parallel gates complete
                      │
                      ▼
                  PRE-PROD
                      │
                      ▼
                 PRODUCTION
                      │
                      ▼
              POST-PROD VALIDATION
```

**What the parallel block buys you**: UX-ENV, FINANCIAL-DOMAIN-ENV, and PERF-ENV validation run simultaneously with BUNDLE-QA regression. In a sequential model, these add 3–4 weeks. In a parallel model, they add zero calendar time if they complete before BUNDLE-QA regression concludes.

**The UAT gate**: UAT is deliberately sequential. Business stakeholders validate against a release that has been confirmed by all technical and specialist gates. Running UAT in parallel with BUNDLE-QA wastes business stakeholder time when defects in BUNDLE-QA will invalidate UAT results.
