# Chapter 8: Governance, Reporting, and Cost

## The Governance Model

Environment governance without reporting is process theater. The program exists on paper but its health is unknown. Reporting without governance is data collection without accountability. This chapter defines both: the governance structure that owns the environment program, and the reporting cadence that makes program health visible.

At VP/Director level, environment governance is a **delivery risk management** function. The governance question is not "do we have an environment program?" but "is our environment program preventing delivery failures?"

---

## Environment Manager Role

Every environment management program requires a named accountable owner at the program level. This is the Environment Manager role — not an individual environment owner (each environment has its own owner per ch04), but the program-level governance owner.

**Environment Manager responsibilities:**

| Responsibility | Cadence | Deliverable |
|---------------|---------|-------------|
| Environment inventory audit | Quarterly | Audit report: active vs. zombie, missing owners, drift incidents |
| SLA compliance review | Monthly | SLA actuals vs. targets by tier |
| Cost allocation review | Monthly | Per-team cost report; budget exceptions |
| Configuration parity review | Monthly | Drift incidents, open drift tickets, trend |
| Compliance audit support | Annual + on-demand | Evidence: PII masking attestation, access control report, audit log export |
| Decommission enforcement | Ongoing | Enforce decommission timeline; escalate non-compliance |
| Template catalog maintenance | Quarterly | Review and update environment templates |
| Tooling evaluation | Bi-annual | Assess new environment provisioning tooling |

**Escalation path:**
- Environment SLA breach → Environment Manager → VP SRE → Engineering VP
- Compliance incident (unmasked PII in non-prod) → Environment Manager → CISO → Legal
- Budget overrun → Environment Manager → Engineering Director → VP Engineering
- Zombie environment exceeding decommission deadline → Environment Manager → VP SRE

## Environment Inventory Audit: Quarterly Review

The quarterly environment inventory audit is the primary governance control that prevents environment debt accumulation.

**Audit scope**: Every environment in the inventory register.

**Audit criteria:**

| Check | Pass | Fail Action |
|-------|------|------------|
| Named primary owner (active employee) | Owner on record, currently employed | Assign new owner within 10 business days |
| Named backup owner | Backup on record | Assign backup within 10 business days |
| Last deployment activity | Within past 30 days | Flag for decommission review |
| Last data refresh | Within tier SLA | Create data refresh ticket; P2 priority |
| Configuration version (IaC commit) | Within 2 sprint versions of production | Create drift remediation ticket |
| Compliance scope documented | PCI/HIPAA/None recorded | Update inventory; notify data governance team |
| Decommission date defined | Hard date on record | Request from owner within 5 business days |
| Cost tag compliance | All required tags present | Auto-remediate via tagging policy; alert owner |

**Audit output report format:**

```
Environment Inventory Audit — Q2 2026
Audit Date: [date]
Auditor: [Environment Manager name]

Summary:
  Total environments: 127
  Active (pass all criteria): 89 (70%)
  Needs attention (1–2 criteria fail): 28 (22%)
  Critical (3+ criteria fail or zombie): 10 (8%)

Zombie environments (no activity > 30 days): 7
  → Decommission initiated for all 7; owners notified

Missing owners: 4
  → Escalated to team leads; 10-day resolution SLA

Drift incidents this quarter: 12
  → 8 resolved; 4 open (tracked in JIRA)

Open compliance issues: 0
  → All non-prod environments passed PII masking audit

Next audit: [date]
```

## Environment Cost Reporting

Non-production environment cost is a significant line item in the engineering budget. Left unmonitored, it grows faster than the delivery organization.

**Monthly cost report structure:**

| Report Section | Content | Audience |
|---------------|---------|---------|
| Total non-prod spend | Month-over-month comparison; YTD trend | VP Engineering, CFO |
| Per-tier breakdown | Cost by environment tier | Engineering Director |
| Per-team breakdown | Cost by consuming team; budget utilization | Engineering managers |
| Zombie cost | Compute cost attributable to zombie environments | VP Engineering (accountability) |
| Savings from ephemeral model | Cost avoided vs. dedicated QA per-team model | VP Engineering (ROI) |
| Cost trend vs. delivery growth | Non-prod cost / deployments ratio | VP Engineering (efficiency) |

**Non-prod cost benchmark (target ratios):**

| Ratio | Definition | Healthy Range | Warning | Action Required |
|-------|-----------|--------------|---------|-----------------|
| Non-prod / Production compute cost | Total non-prod compute cost ÷ production compute cost | 40–60% | > 80% | Zombie audit + rightsizing |
| Zombie cost / Total non-prod cost | Idle environment cost ÷ total non-prod cost | < 5% | 5–15% | Decommission sweep |
| Ephemeral cost / Per-team QA cost | Cost of ephemeral model vs. traditional per-team dedicated QA | < 50% of traditional model | N/A | Ephemeral model is always more cost-effective |
| MTTE investment / MTTE improvement | Platform team hours invested in automation vs. MTTE improvement | Positive ROI | Zero improvement | Re-evaluate automation approach |

**Savings quantification from ephemeral environments:**

```
Traditional model (per-team dedicated QA environments):
  20 delivery teams × $5,000/month/team = $100,000/month

Ephemeral model (shared cluster + on-demand namespaces):
  Cluster base cost: $15,000/month
  Ephemeral namespace compute (peak 200 concurrent): $20,000/month
  Total: $35,000/month

Monthly savings: $65,000
Annual savings: $780,000
```

## Compliance Reporting

Environment management has compliance reporting obligations at PII-regulated organizations (PCI, HIPAA, GDPR, CCPA).

**Annual compliance evidence requirements:**

| Evidence Item | Produced By | Retention |
|--------------|-------------|-----------|
| PII masking attestation | Data engineering team; verified by audit | 7 years |
| Masking pipeline run logs | Automated; S3 bucket with WORM policy | 7 years |
| Access control report | IAM/Vault access logs; per environment | 3 years |
| Audit log of non-prod data access | CloudTrail + Vault audit log | 3 years |
| Environment inventory at quarter-end | Exported from inventory system | 3 years |
| Drift incident log | JIRA export | 3 years |
| Decommission log | Record of environments destroyed, data deleted | 3 years |

**Ongoing compliance controls:**

| Control | Frequency | Owner |
|---------|-----------|-------|
| Presidio PII scan of non-prod environments | Weekly (automated) | Data engineering |
| Access control review — who can access which environment | Quarterly | Environment Manager + Security |
| Masking pipeline validation — sample record audit | Monthly | Data governance team |
| Vault secret access log review | Quarterly | Security |
| Non-prod → prod network isolation verification | Quarterly | SRE / Network engineering |

**Director Governance Gate**: Any Presidio scan finding unmasked PII in a non-production environment is a P0 compliance incident. It triggers immediate data refresh from the masking pipeline, incident response, and notification to the compliance team. The environment is locked (no new deployments) until the refresh is validated.

## Environment SLA Reporting

Environment SLA performance is reported monthly. Breaches are escalated immediately.

**Monthly SLA dashboard:**

| Tier | Uptime Target | Actual | SLA Met? | Incidents | MTTR |
|------|--------------|--------|----------|-----------|------|
| Integration | 95% | 97.3% | ✓ | 1 | 42 min |
| QA | 97% | 95.8% | ✗ — breach | 3 | 2.1 hrs |
| Performance | 99% (during test windows) | 99.4% | ✓ | 0 | — |
| UAT | 99% | 99.2% | ✓ | 1 | 18 min |
| Staging | 99.5% | 99.7% | ✓ | 0 | — |

**For any SLA breach:**

| Required Action | SLA |
|----------------|-----|
| Incident RCA completed | Within 5 business days of incident |
| Prevention action identified | Within RCA |
| Prevention action ticket created | Within 2 days of RCA |
| Prevention action completed | Within 2 sprints |
| Monthly trend review | At next monthly SLA review |

**Reporting audience**: Monthly SLA report goes to VP Engineering, Engineering Directors, and Release Management. Persistent SLA breaches (2+ consecutive months below target for same tier) require a VP-level improvement plan.

## Capacity Utilization Reporting

Non-production environments frequently have low utilization — teams provision environments sized for peak but run at a fraction of peak most of the time.

**Utilization reporting (monthly):**

| Environment | Reserved CPU | Peak CPU Used | Avg CPU Used | Utilization % | Action |
|-------------|-------------|--------------|-------------|--------------|--------|
| qa-payments | 32 CPU | 24 CPU | 8 CPU | 25% avg | Rightsize to 16 CPU |
| qa-auth | 32 CPU | 30 CPU | 22 CPU | 69% avg | No action |
| perf-notifications | 64 CPU | 62 CPU | 55 CPU (during test) | 86% during test | No action |

**Rightsizing rule**: Any environment with < 40% average CPU utilization over 30 days is a rightsizing candidate. Rightsizing recommendation goes to environment owner; owner must accept or provide justification.

**Expected savings from systematic rightsizing**: 15–25% reduction in non-prod compute cost at organizations without active utilization monitoring.

## Key Performance Indicators

**Primary KPIs for environment management program:**

| KPI | Definition | Target | Measurement |
|-----|-----------|--------|-------------|
| MTTE (Mean Time to Environment) | Time from request to environment ready | < 30 min (QA); < 5 min (ephemeral) | Automated — provisioning pipeline logs |
| Environment-Caused Release Delay | Hours of release pipeline blocked by environment issues | 0 hours/quarter | Release management tracking |
| Configuration Drift Incidents | Count of environments with detected drift per quarter | < 5/quarter; 0 for Staging | Automated drift detection reports |
| Zombie Environment Rate | % of inventory with no activity > 30 days | < 3% | Quarterly audit |
| PII Exposure Incidents | Count of unmasked PII findings in non-prod | 0 | Weekly Presidio scan |
| Environment SLA Compliance Rate | % of environments meeting uptime target | > 97% overall | Monthly SLA report |
| Environment Utilization Rate | Avg CPU utilization across non-prod fleet | > 50% avg | Monthly compute report |
| Non-Prod Cost as % of Production | Non-prod compute cost ÷ production compute cost | 40–60% | Monthly cost report |
| Developer Environment Satisfaction | Quarterly survey score | ≥ 8/10 | Quarterly survey |
| DTTFE (Dev Time to First Environment) | Time for new engineer to have working environment | < 4 hours | Onboarding tracking |

**KPI trend reporting cadence:**

| Cadence | Audience | Content |
|---------|---------|---------|
| Weekly | Platform team | MTTE, SLA breaches, open incidents, provisioning failures |
| Monthly | Engineering Directors, VP SRE | All KPIs; cost report; compliance status |
| Quarterly | VP Engineering, CTO | Trend analysis; capacity plan; investment needs; audit results |
| Annual | CISO, Legal, CTO | Compliance audit; environment program maturity assessment |
