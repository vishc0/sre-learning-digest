# Chapter 7: Self-Service and Developer Experience

## The Self-Service Imperative

At T-Mobile delivery scale — hundreds of services, multiple delivery trains, continuous deployment — the traditional model of "file a ticket and wait for ops to provision an environment" is a delivery anti-pattern. It creates a platform team bottleneck, suppresses release frequency, and incentivizes teams to work around governance (sharing environments, skipping tiers, deploying directly to staging).

Self-service environment provisioning is not a developer convenience feature. It is a delivery velocity requirement and a governance mechanism. Paradoxically, giving teams the ability to provision environments themselves — within guardrails — produces better governance outcomes than controlling access through a bottleneck.

When teams can get an environment in 30 minutes, they use the right tier for each stage. When they must wait 5 days, they use whatever environment they already have access to.

## Platform Engineering Model for Environments

The platform engineering model reframes the platform team's role from **gatekeeper** to **product team**. The platform team's customers are the delivery teams. The platform team's product is the environment provisioning capability.

**What this means in practice:**

| Traditional Ops Model | Platform Engineering Model |
|----------------------|---------------------------|
| File a ticket; ops provisions the environment | Self-service portal; team provisions the environment within guardrails |
| Ops team owns all environment knowledge | Runbook documentation + environment templates; teams are self-sufficient |
| Ops approval for any environment change | Policy-as-Code enforces guardrails automatically; no human approval for lower tiers |
| Platform team is the bottleneck | Platform team builds the rails; teams run on them |
| Environment provisioning measured in days | Environment provisioning measured in minutes |

**Internal Developer Portal (IDP) for environments**: The environment provisioning capability is surfaced through an Internal Developer Portal — Backstage is the common open-source implementation. Teams interact with the portal to request, manage, and decommission environments without filing tickets.

**IDP environment capabilities:**

| Capability | Description |
|-----------|-------------|
| Environment catalog | Browse all available environment templates |
| Self-service provisioning | One-click or API-driven environment creation from template |
| Environment status dashboard | View all team environments, health, age, cost |
| Environment inventory | Register new environments, update owner information |
| Decommission request | Self-service environment decommission with data export option |
| Cost tracker | Real-time and historical environment cost per team |

## Kubernetes Namespace Environments

Kubernetes namespaces are the implementation primitive for modern, self-service environment management. A namespace provides:
- **Compute isolation**: Resource quotas prevent one team's environment from consuming cluster capacity needed by others
- **Network isolation**: NetworkPolicies restrict traffic between namespaces (environment isolation without separate VPCs)
- **Identity isolation**: Service accounts are namespace-scoped; a service in namespace `qa-payments` cannot access secrets intended for `qa-auth`
- **Configuration isolation**: ConfigMaps and Secrets are namespace-scoped

**Namespace naming convention:**

```
<tier>-<team>-<feature/release>
Examples:
  feature-payments-pr-2341        ← ephemeral feature branch environment
  integration-payments-main       ← integration environment for payments team
  qa-payments-v3-sprint-45        ← QA environment for payments v3, sprint 45
  perf-notifications-q2-2026      ← performance environment for notifications team
  uat-auth-release-2026-q2        ← UAT environment for auth release
  staging-global                  ← single staging environment (prod-parity)
```

**Namespace resource quotas (per tier):**

| Tier | CPU Limit | Memory Limit | PVC Storage | Notes |
|------|----------|-------------|-------------|-------|
| Feature Branch | 4 CPU | 8Gi | 10Gi | Small — ephemeral |
| Dev | 8 CPU | 16Gi | 50Gi | Individual team |
| Integration | 16 CPU | 32Gi | 100Gi | Shared services |
| QA | 32 CPU | 64Gi | 200Gi | Full test suite |
| Performance | No namespace quota — dedicated cluster | N/A | N/A | Dedicated nodes |
| UAT | 16 CPU | 32Gi | 100Gi | Business validation |
| Staging | No namespace quota — dedicated cluster | N/A | N/A | Prod-identical |

## Environment Request SLA: The 30-Minute Target

**The target**: A delivery team should be able to provision a fully working environment within 30 minutes of submitting their request, for all tiers below Performance.

This is not aspirational. It is achievable with the right automation investment. The 30-minute target breaks down as:

| Phase | Time Budget |
|-------|-------------|
| Request submission validation (automated) | < 1 minute |
| Approval (automated for lower tiers; human only for UAT+) | 0 (lower tiers) / < 1 hour (UAT+) |
| Terraform/Helm provisioning | 5–15 minutes |
| Data provisioning | 5–10 minutes (synthetic) / 10–20 minutes (masked snapshot) |
| Health check validation | 2–5 minutes |
| Notification to requestor | < 1 minute |

**What breaks the 30-minute target:**
- Manual approval steps in the provisioning pipeline (eliminate for lower tiers)
- Data provisioning not pre-staged (stage masked snapshots nightly; don't generate on demand)
- Terraform state locking conflicts (use per-environment state files, not a shared state file)
- Docker image pull latency (pre-pull images to nodes on schedule; use image pull policy `IfNotPresent`)
- Sequential provisioning steps (parallelize: Terraform and data provisioning can run simultaneously)

**What the VP measures**: MTTE by tier. Reported monthly. If QA environment MTTE is consistently > 2 hours, the automation investment is under-resourced. If ephemeral MTTE is > 10 minutes, CI pipeline is the bottleneck.

## Environment Templates: Standardization as a Service

Delivery teams should not design their environment from scratch. They should select from a catalog of pre-approved, pre-tested environment templates that encode the platform team's knowledge of correct configuration.

**Template structure:**

```
environment-templates/
├── java-microservice/
│   ├── template.yaml          ← Backstage template definition
│   ├── helm/                  ← Helm chart values.yaml per tier
│   ├── terraform/             ← Terraform module inputs for this template
│   └── README.md              ← What this template provisions, what to customize
├── nodejs-api/
│   └── ...
├── kafka-consumer/
│   └── ...
├── rabbitmq-worker/
│   └── ...
└── full-stack/
    └── ...                    ← Multi-service template: frontend + API + worker + DB
```

**Template governance:**
- Templates are owned by the platform team
- New templates are reviewed by SRE + Security before being added to catalog
- Templates are version-controlled; breaking changes require new major version
- Teams can propose template modifications via PR — platform team reviews

**Template customization model**: Teams customize environment size (CPU/memory) and service-specific configuration. They cannot customize security controls, network policies, or PII masking settings — those are template-enforced.

## Developer Onboarding to Environments: Day 1 Target

A new engineer joining a delivery team should have a working development environment within their first 4 hours. If it takes longer, the environment specification is incomplete.

**Day 1 environment onboarding checklist:**

```
□ Laptop bootstrapped (company-standard toolchain: kubectl, helm, AWS CLI, docker)
□ Access granted to team's namespace in shared development cluster
□ Local environment spec (Makefile/devcontainer) cloned and executed successfully
□ Can reach shared integration environment (HTTP health check)
□ Can observe team's services in Grafana/Splunk
□ Can trigger a test environment provisioning (ephemeral namespace)
□ Access to team's Vault path for dev secrets
□ First successful test run in ephemeral environment
```

**What blocks Day 1 targets:**
- Access provisioning requires human approval (automate via SCIM/IDP group membership)
- Local environment spec is outdated or missing (platform team to audit quarterly)
- Secrets not pre-seeded in dev Vault path (provisioning script should seed on namespace creation)
- No documentation for "how to run the test suite" (platform team onboarding runbook requirement)

**Director Governance Gate**: Developer time-to-first-working-environment (DTTFE) is a platform team KPI. Target: < 4 hours for day 1. If DTTFE consistently exceeds 4 hours, the onboarding process is blocking delivery team ramp-up.

## Environment Capacity Limits: Cost Guardrails

Self-service provisioning without capacity limits creates unbounded cloud spend. Teams will provision large environments "just in case" and forget to decommission them. Namespace resource quotas are the technical guardrail; cost budgets are the governance guardrail.

**Per-team environment budget:**

| Team Size | Monthly Environment Budget | Enforcement |
|-----------|--------------------------|-------------|
| < 5 engineers | $2,000/month non-prod compute | Alert at 80%; block new provisioning at 100% |
| 5–15 engineers | $5,000/month | Alert at 80%; notify VP at 90%; block at 110% |
| 15+ engineers | $10,000/month | Alert at 80%; notify VP at 90%; block at 110% |

**Budget overrides**: Teams that exceed budget must request an exception with justification (quarterly planning deliverable, major release window). Override approved by engineering director.

**Automatic cost controls:**
- Environments idle for 7 consecutive days: automatic scale-to-zero (deployments scaled to 0 replicas; namespace preserved)
- Environments idle for 30 days: decommission notification initiated
- Ephemeral environments with PR closed > 1 hour: automatic namespace deletion

## Developer Feedback Loop: MTTE as a DORA-Adjacent Metric

DORA metrics (Deployment Frequency, Lead Time, Change Failure Rate, MTTR) measure production delivery performance. MTTE measures the platform team's contribution to delivery velocity — the infrastructure on which DORA metrics are built.

**MTTE correlation with DORA metrics:**

| MTTE State | Effect on DORA Metrics |
|-----------|----------------------|
| MTTE < 10 min (ephemeral) | Deployment Frequency enabled — teams can validate every PR |
| MTTE < 30 min (QA) | Lead Time reduced — test cycle can start without environment wait |
| MTTE > 2 hours (any tier) | Lead Time increases — environment wait adds directly to cycle time |
| MTTE > 1 day (any tier) | Teams bypass governance — Deployment Frequency metric is artificial (no QA gate) |

**Reporting MTTE to leadership:**

```
Monthly Platform Engineering Report — Environment Provisioning KPIs

MTTE by Tier (Month over Month):
  Feature Branch (ephemeral): 4.2 min → 3.8 min  ✓ (target: < 5 min)
  QA:                         28 min → 31 min    ⚠ (target: < 30 min — trending wrong)
  UAT:                        1.8 hours → 1.4 hours ✓ (target: < 2 hours)

Environment Requests This Month: 847
  Fulfilled within SLA: 94.2%
  Missed SLA: 5.8% (49 requests)
  Top miss reason: QA data provisioning timeout (23 requests) — ticket: PLAT-4521

Team Satisfaction Score (quarterly survey): 7.8/10 (↑ from 7.1)
```
