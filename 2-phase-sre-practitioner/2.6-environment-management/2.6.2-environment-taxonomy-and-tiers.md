# Chapter 2: Environment Taxonomy and Tiers

## The Tier Model

Every environment tier exists to validate a specific question. When the question is unclear, the tier's validation purpose fails. When multiple questions are collapsed into one tier, the tier becomes a bottleneck and its governance degrades.

This chapter defines each tier with precision: what question it answers, who consumes it, what configuration requirements it has, and what SLA it carries.

---

## Development Environments

### Local Development
**Purpose**: Rapid individual iteration before sharing code with the team.
**Owner**: Individual engineer.
**Provisioning**: Self-managed by the engineer; bootstrapped from project Makefile or dev container spec (`.devcontainer/`).
**Configuration**: Minimal — typically mocked external dependencies, local database, no integration with downstream services.
**SLA**: None — owned and maintained by the engineer.

**Governance requirement**: The development environment specification (dependencies, versions, configuration) must be codified so a new engineer can bootstrap a working environment on Day 1 without tribal knowledge. If it takes more than 30 minutes for a new engineer to get a working local environment, the specification is incomplete.

### Shared Development
**Purpose**: Multi-service integration during active development, before services are stable enough for Integration tier.
**Owner**: Dev team lead or platform team.
**Provisioning**: Automated — deployed via CI pipeline on merge to `develop` branch.
**Configuration**: Lightweight production-like; shared services (message brokers, databases) are real but scaled down.
**SLA**: Best effort; 8×5 support hours; no guaranteed uptime.

**Common failure mode**: Shared dev environments become permanent and teams treat them as QA. They accumulate configuration debt. They are never rebuilt. They drift silently.

## Feature Branch Environments (Ephemeral)

**Purpose**: Validate a single feature or PR in isolation before merging to a shared branch.
**Owner**: CI/CD pipeline — no human owner.
**Provisioning**: Automatically created when a PR is opened; destroyed when PR is merged or closed.
**Configuration**: Production-like subset — the services relevant to the PR, plus stubs/mocks for non-relevant dependencies.
**SLA**: Ephemeral — exists only for the lifetime of the PR.

**Kubernetes implementation**: Feature branch environments are implemented as Kubernetes namespaces. A GitHub Actions / Tekton pipeline creates the namespace, deploys the Helm chart with environment-specific values, runs automated tests, and destroys the namespace on PR close.

```
PR Opened → Namespace Created (ns: feature-<branch-name>) → Helm Deploy → Tests Run → PR Merged → Namespace Deleted
```

**What the VP measures**: Time from PR open to environment ready. Target: < 5 minutes. If teams are waiting 30+ minutes for a feature environment, CI/CD investment is insufficient.

## Integration Environments

**Purpose**: Validate service-to-service contracts — that Service A and Service B can communicate correctly when deployed together with production-representative configuration.
**Owner**: Platform team or integration test team.
**Provisioning**: Stable deployment — updated when a service passes feature-branch testing.
**Configuration**: All services deployed; production-representative middleware configuration (queue names, topic configurations, connection parameters).
**SLA**: 12×5 (business hours + 4 hours coverage); 95% uptime target; P2 response on outage.

**Configuration requirements:**
- Message broker (Kafka/RabbitMQ) must use production topic/queue names and partition counts
- Service discovery configuration must match production naming
- TLS/mTLS must be configured (not disabled "because it's integration")
- Timeout and retry configurations must match production values

**What breaks when integration config differs from production**: Contract tests pass but production traffic reveals an undeclared dependency, a mismatched message schema version, or a timeout setting that is too short for production-representative load.

---

## QA / Test Environments

**Purpose**: Functional testing, regression testing, and acceptance testing. Answer the question: "Does the software meet its functional requirements?"
**Owner**: QA team lead, with platform team for infrastructure.
**Provisioning**: Stable deployment; refreshed from Integration tier on successful integration test gate.
**Configuration**: Full service deployment; production-representative config except scale (compute sized for test load, not production load).
**SLA**: 16×5 (extended business hours); 97% uptime target; P2 response on outage; P1 response during active release test cycles.

**Stability requirements:**
- Configuration changes to QA require change request and team notification
- Data refreshes are scheduled and communicated (not ad hoc)
- Deployments to QA follow a defined deployment window — not during active test execution
- Shared QA environments require a booking system for test execution windows

**Director Governance Gate**: QA environment availability directly constrains release cadence. QA SLA breaches are reported at the release review. Any QA environment incident during a release test cycle escalates to P1.

---

## Performance Environments

**Purpose**: Validate system behavior under load. Answer the questions: "Does the software perform acceptably under expected production load?" and "Where are the system's performance limits?"
**Owner**: Performance engineering team, with SRE for infrastructure.
**Provisioning**: Dedicated infrastructure — not shared with other test tiers.
**Configuration**: **Production-identical**: same instance types, same JVM heap settings, same connection pool sizes, same middleware configuration, same network topology. Scale may differ (20–50% of production data volume is acceptable; less is not).
**SLA**: Available on-demand with a reservation system; 24×7 during performance test execution windows; P1 response during active test.

**Why performance environments must mirror production — exactly:**
A performance test run against an environment with half the JVM heap will show GC pressure that does not exist in production. It will also miss GC pressure that exists in production but is not observable in a smaller heap. The performance test is testing the environment configuration, not the software.

Every configuration delta between the performance environment and production is a variable that invalidates the test results.

**What the VP measures**: Ratio of performance-environment configuration matches to production — target 100% for JVM settings, connection pools, middleware config, and network topology. Data volume target: ≥ 50% of production data volume.

---

## UAT Environments

**Purpose**: Business user validation. Answer the question: "Does the software meet business requirements as understood by the business stakeholder, with production-representative data?"
**Owner**: Business analysis team or product management, with platform team for infrastructure.
**Provisioning**: Stable; refreshed from QA tier after QA sign-off.
**Configuration**: Production-representative; data must be masked production data — realistic enough to validate business logic, compliant with PII requirements.
**SLA**: 12×5 during business hours; 99% uptime target during UAT sign-off windows; P1 response during UAT execution.

**Data requirements:**
- Data must reflect actual production data shapes (not synthetic stubs)
- PII must be masked (see ch05 — Test Data Management)
- Data must be refreshed before each UAT cycle — stale data produces invalid business validation
- Data volume should be sufficient to trigger pagination, batch processing, and aggregation behavior

**Access controls:** Business users in UAT have read access to data and write access only to test scenarios. No admin access. Audit logging enabled.

---

## Staging Environments

**Purpose**: Final pre-production gate. Answer the question: "Is this software safe to deploy to production?" Staging must be as close to production as possible — it is the last validation layer before customer impact.
**Owner**: SRE / Release Engineering — treated with production-level rigor.
**Provisioning**: Production-identical infrastructure; deployed from the same artifacts that will be deployed to production.
**Configuration**: Production-identical. Exceptions require documented approval and risk assessment.
**SLA**: 99.5% uptime; 24×7 support; P1 response; change control required for any configuration change.

**Staging is different from all other non-prod tiers in one critical way**: a failure in Staging that was not seen in QA or UAT is a signal that production will also fail. Staging failures must be treated as production-risk signals, not just "staging issues."

**Deployment to Staging must use**:
- The identical deployment artifact as production
- The identical deployment procedure (same Helm chart, same Terraform config, same feature flag state)
- The same secrets management (Vault — not plaintext environment variables)
- A smoke test suite that mirrors production monitoring alerts

**Director Governance Gate**: No artifact may promote from Staging to Production without explicit Staging sign-off. Staging environment incidents delay the release. Any Staging configuration deviation from production requires VP-level approval.

## Ephemeral Environments: The Kubernetes Namespace Model

Ephemeral environments are the highest-leverage investment in developer productivity for modern Kubernetes-native delivery organizations. They eliminate the shared-environment contention problem by giving every developer their own isolated environment on demand.

**Architecture:**
```
GitHub PR Opened
        ↓
CI Pipeline Triggered
        ↓
kubectl create namespace feature-<pr-number>
        ↓
helm install <chart> -n feature-<pr-number> -f values.feature.yaml
        ↓
Automated tests run against namespace endpoint
        ↓
Test results posted to PR
        ↓
PR Merged or Closed → kubectl delete namespace feature-<pr-number>
```

**Cost control:** Namespace resource quotas prevent any single ephemeral environment from consuming unbounded compute.

**Configuration model:** Ephemeral environments inherit production configuration with two overrides: (1) downstream dependency endpoints point to stubs or shared integration services, (2) data is synthetic or a small masked extract.

**What the VP measures:** MTTE for ephemeral environments (target: < 5 minutes). Environment-per-PR adoption rate across delivery teams. Cost per ephemeral environment.

## Environment SLA Matrix

| Tier | Uptime Target | Supported Hours | Incident Response Priority | Refresh Cadence | Provisioning SLA |
|------|--------------|-----------------|---------------------------|-----------------|-----------------|
| Local Dev | None | N/A | N/A | N/A | N/A |
| Feature Branch (Ephemeral) | Ephemeral | Lifetime of PR | None — self-healing | Per-PR | < 5 minutes |
| Shared Dev | 90% | 8×5 | P3 | Weekly or on-demand | < 2 hours |
| Integration | 95% | 12×5 | P2 | Per successful build | < 1 hour |
| QA | 97% | 16×5 | P2 (P1 during release cycle) | Per sprint start | < 30 minutes |
| Performance | 99% during test windows | On reservation | P1 during test | Per test cycle | < 4 hours (dedicated) |
| UAT | 99% | 12×5 | P1 during UAT window | Per UAT cycle | < 2 hours |
| Staging | 99.5% | 24×7 | P1 | Per release | < 1 hour |
| Production | 99.9%–99.99% | 24×7 | P0/P1 | N/A | N/A |
