# Change Risk Matrix

---

## 1. How to Use This Matrix

The matrix produces a **Risk Tier** — a process designation that determines which review path a change follows, how much lead time is required, and who must approve.

**Three-step process**:

1. Score **Impact** (P1–P4) using Section 2 below. Impact reflects the worst-case consequence if the change fails during or after implementation.
2. Score **Likelihood** (High/Medium/Low) using Section 3 below. Likelihood reflects the probability that the change does not execute as planned.
3. Find your cell in Section 4. That cell is your pre-mitigation Risk Tier.

Then apply Section 6 (Mitigating Controls) to determine if the tier shifts down.

**Scoring rule**: When in doubt, score up. The cost of over-reviewing a change is a delayed deployment. The cost of under-reviewing a P1 change is a production incident. Score conservatively.

---

## 2. Impact Scoring Rubric

Impact reflects the consequence to customers, revenue, and compliance if the change fails or behaves unexpectedly after deployment.

**Score based on the worst-case failure mode — not the expected outcome.**

---

### P1 — Critical

**At least one of the following is true**:
- More than 10% of total customers would be directly affected (error, outage, data issue)
- Revenue impact exceeds $1M per hour of failure duration
- Regulatory or legal violation is possible (CPNI, PCI, E911, TCPA, state privacy law)
- Emergency services infrastructure (E911, FirstNet) is in the blast radius
- Customer financial data, authentication credentials, or account security is in scope

**T-Mobile scale examples**:
- Change to the authentication gateway (IAM service) — all 115M subscribers authenticate through this path
- Change to the billing calculation engine during billing cycle — revenue impact is immediate and affects all postpaid accounts
- Change to network element configuration that could affect E911 call routing
- Certificate rotation on the API gateway without a tested rollback — impacts all mobile app sessions
- Schema migration on the subscriber master record table — all downstream services read this

**What this means operationally**: A P1 failure at T-Mobile scale would trigger Sev1 declaration, executive escalation, and potential regulatory notification. Every P1 change gets explicit attention at this score.

---

### P2 — High

**At least one of the following is true**:
- 1–10% of customers would be directly affected
- Revenue impact is $100K–$1M per hour
- SLA breach with contractual consequence is possible (enterprise or B2B customers)
- A critical customer journey (checkout, plan upgrade, device activation) is degraded but not fully broken

**T-Mobile scale examples**:
- Change to the prepaid account management service — affects ~30M prepaid subscribers
- Deployment to the device activation API — device activations fail for customers at retail during the window
- Change to the payment processing integration — payment failures affect customers in the checkout flow
- Update to the wholesale/MVNO API — downstream MVNO partners cannot activate lines during the window
- Network slice configuration change for enterprise customers — B2B SLAs are at risk

**What this means operationally**: P2 changes warrant full CAB review, a tested rollback, a dedicated maintenance window, and stakeholder notification. P2 failures are serious but do not automatically trigger executive escalation unless they extend beyond the window.

---

### P3 — Medium

**At least one of the following is true**:
- Less than 1% of customers would be directly affected
- Revenue impact is below $100K per hour
- Impact is limited to an internal SLA with no contractual consequence
- The failure mode is degraded performance rather than hard failure (increased latency, partial feature unavailability)

**T-Mobile scale examples**:
- Change to the customer care internal portal — affects call center agents, not end customers directly
- Update to a non-critical notification microservice — some push notifications delayed, not dropped
- Deployment to the A/B testing platform — experiment data may be affected, no customer journey broken
- Configuration change in a dev-facing internal API — only engineering teams are consumers
- Update to a reporting pipeline — dashboards are stale, no real-time customer impact

**What this means operationally**: P3 changes still require documented rollback and peer review. The difference from P2 is the review rigor — Standard CAB agenda vs. extended review.

---

### P4 — Low

**All of the following are true**:
- No customer is directly affected under any failure scenario
- Change is confined to non-production environments, internal tooling, or developer infrastructure
- Revenue impact is zero (no production path is touched)

**T-Mobile scale examples**:
- Adding a new metric to the engineering observability dashboard
- Updating a CI/CD pipeline script that doesn't affect production artifacts
- Deploying a feature behind a disabled feature flag (no traffic routing yet)
- Configuration change in the development environment Kubernetes cluster
- Updating documentation in a code repository

**What this means operationally**: P4 changes still benefit from documentation and peer review for audit trail purposes, but they are not CAB items. Many P4 changes qualify as Standard Changes after review.

## 3. Likelihood of Failure Scoring Rubric

Likelihood reflects how probable it is that the change execution deviates from the plan — either the implementation fails, the rollback is needed, or the behavior in production differs from the staging expectation.

---

### High Likelihood

**Three or more of the following are true**:
- The change introduces new code that has not been exercised in a production-equivalent environment
- The staging environment does not closely mirror production data volumes, traffic patterns, or dependencies
- Rollback requires coordination across two or more teams or involves a schema migration reversal
- The change operator has not executed this type of change before on this system
- Three or more external dependencies are in the execution path (external APIs, third-party services, partner integrations)
- The service has had a change failure or production incident in the past 30 days
- The change mutates persistent data (writes to database, moves messages, modifies configuration state)

**What High Likelihood means**: There is a meaningful probability that the change will not proceed as written. CAB should scrutinize the rollback plan as carefully as the implementation plan.

---

### Medium Likelihood

**Mostly Medium factors apply; no more than one High factor is present**:
- The code path being modified has been tested in staging but staging has known gaps from production
- Rollback is a 2–3 step process executable by a single team within 15 minutes
- The change operator has executed similar changes but not this exact one on this service
- One or two external dependencies are in the path
- The service has had no incidents in the past 30 days but has had changes recently

**What Medium Likelihood means**: The change is well-understood but not risk-free. Standard CAB review and a tested rollback are sufficient.

---

### Low Likelihood

**Mostly Low factors apply; zero or one Medium factor is present**:
- The change is configuration-only with no code change (a value change, a flag flip, a timeout adjustment)
- The staging environment is production-equivalent and the test was run within 72 hours
- Rollback is a single-command revert executable by the change operator alone in under 5 minutes
- The change operator has executed this exact change type on this service multiple times without incident
- No external dependencies in the execution path
- The service has had no incidents and no change failures in the past 90 days

**What Low Likelihood means**: The change is well-understood, well-tested, and has a fast reliable rollback. Risk tier can often be reduced one level with the right mitigating controls.

## 4. Risk Tier Matrix

| | High Likelihood | Medium Likelihood | Low Likelihood |
|---|---|---|---|
| **P1 Impact** | Emergency Review | Extended CAB | Standard CAB |
| **P2 Impact** | Extended CAB | Standard CAB | Standard CAB |
| **P3 Impact** | Standard CAB | Standard Change Candidate | Standard Change Candidate |
| **P4 Impact** | Standard CAB | Standard Change Candidate | Standard Change Candidate |

---

## 5. What Each Risk Tier Requires

### Standard Change Candidate

**Meaning**: The change type is pre-approved. Individual instances do not require CAB review if they match the approved pattern exactly.

**Requirements to be a Standard Change**:
- The change type has been executed successfully at least 3 times in production with no deviations
- The blast radius is fully understood and documented in the Standard Change catalog entry
- A tested rollback procedure exists and has been validated
- The change has been nominated and approved by CAB for the Standard Change catalog

**Lead time**: None — Standard Changes can be executed any time outside of change freeze periods.

**Approval**: Single peer review confirming the change matches the catalog pattern.

**Examples**: Rotating a non-critical certificate using the standard rotation runbook; scaling a deployment from N to N+2 replicas with no configuration changes; updating a non-critical npm dependency to a patch version.

---

### Normal CAB (Standard CAB)

**Meaning**: The change requires review by the Change Advisory Board at a scheduled CAB meeting.

**Requirements**:
- RFC submitted at least 48 hours before CAB meeting (72 hours for P1/P2)
- All RFC sections complete — no blank fields in risk assessment, rollback, or test evidence sections
- Peer review completed and documented in the RFC
- Architect or security review completed if required by RFC Section 10

**Lead time**: 2–5 business days (submission deadline + next scheduled CAB + approval notification time)

**Approval**: CAB quorum. Majority approval. Change owner presents; does not vote.

**Implementation**: Scheduled maintenance window, typically off-peak hours.

---

### Extended CAB

**Meaning**: The change requires dedicated CAB review time beyond the standard agenda slot. P1 impact or High likelihood requires deeper scrutiny than a standard CAB slot provides.

**Additional requirements beyond Standard CAB**:
- RFC submitted at least 5 business days before the implementation window
- Architecture review is mandatory regardless of the RFC Section 10 checkboxes
- Rollback must be tested in staging and evidence provided — not just documented
- Change owner presents a verbal walkthrough at CAB; CAB may request a pre-CAB dry-run
- Domain SMEs from all services in the blast radius must review and sign off

**Lead time**: 5–10 business days

**Approval**: Full CAB panel. Documented conditions if approved conditionally.

---

### Emergency Review

**Meaning**: The change cannot wait for Normal process. Retroactive documentation is required.

**Eligibility**: Active production incident requiring this change as mitigation, confirmed security vulnerability with active exploitation, or regulatory deadline with documented legal consequence.

**Approval process**: Three-person verbal authorization (CAB Chair or delegate + domain SME + additional stakeholder). Documented in the Emergency Change Record.

**Post-implementation requirement**: PIR completed within 24 hours. If the emergency was not justified, that is documented and the change owner is counseled.

## 6. How Mitigating Controls Shift the Tier

Controls reduce the effective Likelihood Score. They do not reduce the Impact Score — a P1 impact remains P1 regardless of how good the rollback is.

**The logic**: A feature flag does not reduce the damage if the change fails. It makes it faster and simpler to stop the damage. That reduces Likelihood, not Impact.

| Control | Likelihood Reduction |
|---------|---------------------|
| Feature flag with instant rollback (flag flip = revert) | High → Medium, OR Medium → Low |
| Canary deployment (< 5% initial traffic, monitored 30 min before full rollout) | High → Medium, OR Medium → Low |
| Blue-green deployment (full parallel environment, rollback is traffic switch < 2 min) | High → Medium, OR Medium → Low |
| Automated rollback triggered by health check failure | Medium → Low (removes human reaction time from rollback path) |
| Rollback tested in staging within 72 hours | Confirms Low Likelihood for rollback complexity factor |
| Previous successful production deployments of this artifact (3+) | Reduces operator experience factor from Medium to Low |

**Combining controls**: Feature flag AND automated rollback together can reduce High → Low in one step if both are confirmed active. Document both in the RFC.

**Controls that do NOT reduce likelihood**:
- A rollback plan that has not been tested
- A "staging environment" that is not production-equivalent in traffic or data volume
- A feature flag that requires a code deploy to toggle (that is not a feature flag)

## 7. Special Cases: Override Rules

These rules apply regardless of the Impact × Likelihood matrix score.

### Schema Changes Always Start at P1

Database schema changes — table creation, column addition, column modification, index changes, constraint changes — always start at P1 Impact regardless of how many customers traverse the changed table. The reason: schema changes are difficult to reverse, can affect all services that read from the database, and have non-obvious blast radius through ORM layers and caching.

A schema change on a table used only by an internal tool still gets P1 treatment in impact scoring because the reversal path is complex enough to warrant it.

**Exception**: Additive-only schema changes (adding a nullable column with no default, adding an index on an existing column) may be scored P2 if: (a) the table is not in the read path of any customer-facing service, and (b) the change has been validated with no lock contention in staging.

### Shared Infrastructure Changes: +1 Impact Tier

Changes to shared infrastructure — API gateways, service meshes, DNS, load balancers, identity providers, shared databases, message brokers — automatically receive a one-tier increase in Impact Score.

If the base assessment is P3, it becomes P2. If the base assessment is P2, it becomes P1.

**Reason**: Shared infrastructure blast radius is always underestimated at time of RFC writing. The +1 tier is a forcing function to apply Extended CAB scrutiny.

**Applies to**: Kubernetes control plane changes, Istio/service mesh policy changes, RabbitMQ or Kafka cluster configuration changes, API gateway routing rule changes, shared Redis or ElasticSearch cluster changes, IAM or LDAP configuration changes, shared certificate authority changes.

### Certificate and Secret Rotations: High Likelihood by Default

Certificate rotations, API key rotations, and secret rotations are scored High Likelihood by default unless:
- The rotation procedure has been tested end-to-end in staging within 14 days, AND
- The previous rotation of this specific certificate/secret was completed without incident

**Reason**: Rotation failures have a specific failure mode — the new credential is deployed but services still using the old credential begin failing, and the blast radius expands over time as each service's connection pool refreshes. This is a well-documented failure pattern that warrants explicit mitigation (overlapping validity windows, canary rotation, automated rollback).
