# Change Metrics Definitions and Calculation Reference

---

## How to Use This Reference

Each metric entry follows the same structure:

- **Definition** — what the metric measures, in plain language
- **Formula** — exact calculation with variable definitions
- **Data Sources** — where to get the inputs
- **Measurement Frequency** — how often to calculate and report
- **Director Review Cadence** — when this metric belongs on your review agenda
- **Common Pitfalls** — how this metric gets measured incorrectly and what the distortion looks like
- **T-Mobile Context Example** — what this metric looks like at scale

---

## 1. Deployment Frequency (DF)

**Definition**: The rate at which the organization successfully deploys code changes to production. The primary DORA velocity metric. High deployment frequency correlates with smaller batch sizes, faster feedback, and lower change failure rates in Elite-performing organizations.

**Formula**:
```
DF (per service, per period) = count(successful production deployments) / days in period

DF (organization) = median(DF across all services)
```
Use median, not mean. A single service with hundreds of deployments per day will distort an organizational mean.

**Data Sources**:
- CI/CD platform: deployment events filtered to `environment=production` AND `status=success`
- Change management tool: RFC records with `status=Implemented` and `change_type=Code` or `change_type=Config`
- Requires: consistent environment tagging in your deployment pipeline; without it, staging deployments are indistinguishable from production deployments

**Measurement Frequency**: Per service, weekly. Organizational aggregate, monthly.

**Director Review Cadence**: Monthly. Review team-level distribution, not just the aggregate. A high organizational DF masked by two high-velocity teams and ten slow ones is a team health signal, not a success metric.

**Common Pitfalls**:
- Counting rollbacks as deployments — a rollback is a recovery action, not a deployment. Tag rollback events separately in your pipeline.
- Counting failed deployments — a deployment that errored before completing is not a deployment.
- Counting only CI/CD pipeline deployments while manual deployments happen in parallel — dark deployments outside the pipeline are invisible to DF and are a process gap.
- Using commits instead of deployments — a commit merged to main that sits in a release branch for three weeks is not a deployment.

**T-Mobile Context Example**: A team deploying the customer-facing account management portal monthly has DF = 0.033/day. A team deploying a backend microservice with automated canary deployment multiple times per week has DF = 3–5/day. The organizational median across ~200 services might be 0.5–1.0/day, placing T-Mobile's engineering organization in the High tier for this metric at scale.

## 2. Lead Time for Changes (LTC)

**Definition**: The elapsed time from a code change being merged to the main branch to that change successfully running in production. Measures the speed of the entire delivery pipeline — review, testing, staging, approval, and deployment. DORA velocity metric.

**Formula**:
```
LTC (per change) = production_deployment_timestamp − merge_to_main_timestamp

LTC (period) = median(LTC across all changes deployed in period)
```
Report as median, not mean. Emergency changes deployed in 30 minutes will pull the mean down and mask the true lead time for normal changes.

**Data Sources**:
- Git platform: merge timestamp for each commit SHA
- CI/CD platform: production deployment timestamp for each commit SHA
- Requires: tracing a commit SHA from merge event through pipeline stages to production deployment. Most modern CI/CD systems (GitLab, GitHub Actions) provide this natively if configured.

**Measurement Frequency**: Per deployment event (continuous). Report monthly median.

**Director Review Cadence**: Monthly. Break down by change tier:
- Standard Changes: LTC should be under 2 days (peer review + execution time)
- Normal CAB: LTC should be under 7 days (RFC submission + CAB meeting + window scheduling)
- Emergency Changes: LTC should be under 4 hours
- If Normal CAB LTC is consistently over 10 days, CAB scheduling frequency or RFC queue management is the bottleneck.

**Common Pitfalls**:
- Measuring from "change complete" rather than "merge to main" — the time between code complete and merge (code review, pull request approval) belongs in LTC. Including it correctly shows the full pipeline cost.
- Measuring only successful changes — failed deployments that are eventually redeployed have longer LTCs; they must be included to show the true delivery pace.
- Reporting mean instead of median — a single emergency change deployed in 1 hour during a month of 14-day average LTC will pull the mean down significantly.

**T-Mobile Context Example**: If a developer merges a notification service change on Monday morning and it deploys to production Friday after the Thursday CAB meeting, LTC = 4 days. If that same team later implements a Standard Change catalog entry for routine notification service configuration updates, LTC drops to less than 1 day — they no longer wait for CAB. That LTC improvement shows directly in the metric, quantifying the value of the Standard Change program.

## 3. Change Failure Rate (CFR)

**Definition**: The percentage of production deployments that result in a service degradation requiring a hotfix, rollback, or incident response. The primary DORA stability metric. CFR measures change quality — how often do changes that reach production cause harm?

**Formula**:
```
CFR = (count(deployments requiring remediation) / count(total production deployments)) × 100

Where "requiring remediation" = any deployment that results in:
  - A rollback being executed
  - A hotfix deployment within 24 hours
  - An incident record opened with root cause attributed to the deployment
```

**Data Sources**:
- Change management tool: post-implementation section of RFC — `rollback_required=Yes`
- Incident management tool: incidents where `root_cause_category=Change` AND `incident_opened_within=24_hours_of_deployment`
- Requires: disciplined incident root cause categorization; if RCA is not consistently done, CFR is understated

**Measurement Frequency**: Per deployment event (continuous). Report monthly.

**Director Review Cadence**: Monthly, broken down by team and service. Quarterly trend. Any team above 15% CFR (above the High/Medium boundary) should have a dedicated quality review in that quarter. CFR above 30% is a systemic quality problem requiring immediate intervention.

**Common Pitfalls**:
- Not counting rollbacks as failures — a rollback is by definition a failed change. If it is not counted in CFR, the metric is wrong.
- Only counting incidents that were formally declared — low-severity issues that engineers fix manually without opening an incident are invisible to CFR. These are the "dark matter" of change quality.
- Attributing too broadly — not every incident during a post-deployment window is caused by the deployment. Require evidence of causation in RCA, not just temporal correlation.
- Attributing too narrowly — teams that declare incidents "unrelated to the change" when the change was indeed the cause are gaming CFR. PIR review should catch this.

**T-Mobile Context Example**: If a team deploys 40 times per month and 4 of those deployments require rollback or hotfix, CFR = 10% (High tier). If 3 of those failures are traced to a single service dependency that the team did not test in staging, the corrective action is clear: add that dependency to the staging test coverage. Tracking CFR at the team level makes this visible; rolling it into an organizational aggregate hides it.

## 4. Mean Time to Restore — Change-Induced (MTTR-C)

**Definition**: The median elapsed time from detection of a change-induced service degradation to full restoration of service for affected customers. Distinct from all-cause MTTR. Change-induced MTTR reflects the effectiveness of the rollback process and incident response for change failures specifically.

**Formula**:
```
MTTR-C (per incident) = restoration_timestamp − detection_timestamp

Where:
  detection_timestamp = first alert fire or monitoring detection, not incident ticket open time
  restoration_timestamp = customer-facing service returned to normal, not incident ticket closed time

MTTR-C (period) = median(MTTR-C across all change-induced incidents in period)
```

**Data Sources**:
- Monitoring/alerting system: alert fire timestamp
- Incident management tool: incident open, resolve, and root cause fields
- Change management tool: rollback_required field linked to incident record
- Requires: linking incident records to the RFC that caused them; requires both systems to share a common identifier (RFC ID or deployment tag)

**Measurement Frequency**: Per incident event. Monthly aggregate.

**Director Review Cadence**: Monthly. Track two sub-metrics:
- MTTR-C for rollback-resolved incidents (should be under 1 hour for High tier)
- MTTR-C for hotfix-resolved incidents (will be longer; this is not necessarily a problem)
- If rollback MTTR-C is over 1 hour, the rollback process has gaps: either the rollback plan is incorrect, the rollback owner was not available, or the rollback procedure took longer than planned.

**Common Pitfalls**:
- Using ticket open time instead of alert time — a 20-minute delay from alert to incident creation inflates MTTR by 20 minutes. Use the monitoring system timestamp.
- Using ticket close time instead of service restoration — tickets often close after a post-incident review hours or days after service was restored. Use the monitoring data to confirm when service returned to baseline.
- Including non-change-induced incidents — hardware failures, dependency outages, and load anomalies have different MTTR drivers than change failures. Keep them separate.

**T-Mobile Context Example**: A P2 change failure that triggers rollback. Alert fires at 02:15. Rollback owner executes rollback at 02:22. Service restored at 02:31. Incident ticket closed at 09:00 after shift change. MTTR-C = 16 minutes (alert to restoration). The ticket close time of 09:00 is irrelevant to MTTR-C — what matters is when customers were unaffected.

## 5. Emergency Change Rate (ECR)

**Definition**: The percentage of all changes that are executed through the Emergency Change process rather than the Normal or Standard process. ECR is a process health metric. High ECR indicates that either the Normal process is too slow (driving false emergency declarations) or that change quality is low (driving genuine emergencies from incidents).

**Formula**:
```
ECR = (count(Emergency Changes in period) / count(all changes in period)) × 100
```

**Target ranges**:
- Elite / High performing: ECR < 5%
- Acceptable: ECR 5–10%
- Concerning: ECR 10–15% — investigate root cause
- Critical: ECR > 15% — the Emergency process is being used as a workaround for a broken Normal process

**Data Sources**:
- Change management tool: change type = Emergency; count by period
- Emergency Change Records: category (A/B/C) breakdown

**Measurement Frequency**: Monthly.

**Director Review Cadence**: Monthly. Break down by category:
- Category A (incident mitigation): expected and acceptable; focus on whether these incidents could have been prevented
- Category B (security): expected; track CVSS scores to understand the security posture trend
- Category C (regulatory): should be near zero; recurring regulatory emergencies indicate a planning failure
- A spike in Category A ECR means more incidents are requiring changes as mitigation — that is a quality signal, not just a process signal

**Common Pitfalls**:
- High ECR that looks like urgency but is actually process avoidance — teams declare emergency to avoid a 5-day CAB cycle. This shows up when the emergency declaration criteria (incident, exploitation, deadline) are not consistently verified.
- Not tracking ECR by team — an organization average of 8% ECR could mask one team at 35% ECR. Team-level visibility is required.

**T-Mobile Context Example**: 300 changes in a month. 24 are emergency. ECR = 8%. Investigation reveals 18 of the 24 are Category A (incident mitigation). Of those, 14 were caused by changes in the prior month that failed or had unexpected side effects. That trace — emergency changes caused by prior normal changes — is the signal for a CFR improvement program.

## 6. Standard Change Rate (SCR)

**Definition**: The percentage of all changes that are executed as pre-approved Standard Changes, requiring no CAB review. SCR measures the maturity of the Standard Change catalog and the effectiveness of the change program in identifying and pre-approving routine, low-risk changes.

**Formula**:
```
SCR = (count(Standard Changes in period) / count(all changes in period)) × 100
```

**Target ranges** (organization-maturity dependent):
- New program: SCR 10–20% (catalog is new; few change types have enough successful history)
- Maturing program: SCR 30–50% (catalog is growing; routine deployments are pre-approved)
- Mature program: SCR 50–70% (routine infrastructure and application changes are pre-approved; CAB focuses on novel and high-risk changes)

**Data Sources**: Change management tool: change type = Standard; count by period.

**Measurement Frequency**: Monthly.

**Director Review Cadence**: Quarterly. Ask: which change types are candidates for Standard Change catalog addition? Analyze Normal CAB changes with zero deviations over the past 6 months — those are Standard Change candidates.

**Common Pitfalls**:
- High SCR that hides poor quality — if Standard Changes are failing at high rates, the catalog entries are inadequately defined. Track CFR broken down by change type (Standard, Normal, Emergency). Standard Changes should have the lowest CFR; if they do not, catalog entries need revision.
- Not reviewing the catalog annually — a Standard Change approved two years ago may no longer be appropriate given service changes. Annual catalog review is required.

## 7. CAB Decision Metrics: Approval Rate, Deferral Rate, Rejection Rate

**Definition**: Distribution of CAB decisions across the three non-emergency outcomes. These metrics measure CAB decision-making quality, RFC submission quality, and whether the review process is functioning as a quality gate or a rubber stamp.

**Formulas**:
```
Approval Rate = (count(Approved) / count(all CAB decisions)) × 100
Conditional Approval Rate = (count(Approved with Conditions) / count(all CAB decisions)) × 100
Deferral Rate = (count(Deferred) / count(all CAB decisions)) × 100
Rejection Rate = (count(Rejected) / count(all CAB decisions)) × 100
```

**Target ranges**:
- Approval Rate 70–85%: CAB is approving most well-prepared changes
- Conditional Approval Rate 10–20%: CAB is finding real gaps and requiring remediation
- Deferral Rate 5–10%: RFC was submitted without sufficient evidence; returned for more work
- Rejection Rate < 5%: outright rejection should be rare; most problems should be caught in peer review before CAB

**Data Sources**: Change management tool: CAB decision field on RFC records.

**Measurement Frequency**: Monthly (per CAB cycle).

**Director Review Cadence**: Monthly. High deferral rate is a signal that RFC submission quality is low — either authors do not understand requirements or the RFC template is unclear. High rejection rate is a signal that the peer review process before CAB is not working.

**Common Pitfalls**:
- A 95%+ approval rate is not a success metric — it means CAB is rubber-stamping. A healthy CAB should find gaps in 15–30% of submissions.
- Not tracking deferral reasons — "needs more test evidence" and "rollback plan incomplete" are different deferrals with different corrective actions.

## 8. RFC Submission Compliance

**Definition**: The percentage of changes submitted through the RFC process within the required lead time before the implementation window. Measures process adherence and process accessibility.

**Formula**:
```
Submission Compliance = (count(RFCs submitted on time) / count(all production changes)) × 100

Where "on time" = RFC submitted at least 48 hours before CAB meeting (72 hours for P1/P2)

All production changes includes changes discovered post-facto through incident investigation
```

**Target**: 95%+. Below 90% indicates either process avoidance or that the RFC process is not accessible enough (too complex, too slow, too much friction).

**Data Sources**: Change management tool: RFC submission date vs. CAB meeting date vs. implementation window date. Incident records: changes discovered through post-incident investigation that had no RFC.

**Measurement Frequency**: Monthly.

**Director Review Cadence**: Quarterly. Track dark changes (changes executed without RFC) by team. Teams with recurring dark changes need coaching; if the pattern persists, investigate whether the RFC process itself is the barrier.

## 9. PIR Completion Rate

**Definition**: The percentage of changes requiring a Post-Implementation Review where the PIR was completed and closed within the defined SLA. Measures organizational learning discipline.

**SLA definitions**:
- P1/P2 changes: PIR completed within 5 business days
- Rollback-required changes: PIR completed within 5 business days
- Emergency Changes: PIR completed within 24 hours (Category A/B) or 48 hours (Category C)
- Failure outcome: PIR completed within 3 business days

**Formula**:
```
PIR Completion Rate = (count(PIRs completed within SLA) / count(PIRs required)) × 100
```

**Target**: 90%+. Below 80% indicates that PIR is seen as overhead rather than value — requires cultural intervention, not just process enforcement.

**Data Sources**: Change management tool: PIR record linked to RFC, PIR close date vs. SLA deadline.

**Measurement Frequency**: Monthly.

**Director Review Cadence**: Monthly. A declining PIR completion rate is an early warning signal for organizational learning breakdown — it precedes an increase in repeat failures.

**Common Pitfalls**:
- Counting PIR created vs. PIR completed and signed off — a PIR that is opened and never finished is not a completed PIR.
- Not tracking quality of completed PIRs — a PIR with "no deviations, change was successful" on a change that had a 30-minute rollback is a false completion. Spot-check PIR quality quarterly.

## 10. Change-to-Incident Correlation Rate (CICR)

**Definition**: The percentage of all incidents where the root cause investigation identified a recent production change as a contributing or primary cause. CICR is the primary linkage metric between change management and incident management.

**Formula**:
```
CICR = (count(incidents where root_cause includes recent change) / count(all incidents)) × 100

Where "recent change" = production deployment in the 24 hours prior to incident detection
```

**Target ranges**:
- Below 20%: Most incidents are not change-induced; focus is on reliability and dependency stability
- 20–40%: Normal range for active engineering organizations
- Above 40%: Change quality is a primary driver of incident volume; CFR improvement program is required

**Data Sources**: Incident management tool: root cause category field, linked RFC record, incident timestamp vs. deployment timestamp.

**Measurement Frequency**: Monthly.

**Director Review Cadence**: Monthly. CICR trend is more important than point-in-time value. A rising CICR during a period of increasing deployment frequency signals that deployment quality is not keeping pace with deployment velocity. A falling CICR during the same period indicates the quality controls (testing, canary, rollback) are working.

**Common Pitfalls**:
- Not doing root cause analysis consistently — if RCA is optional or superficial, change-induced incidents are mis-categorized and CICR is understated. Under-stated CICR hides a quality problem.
- False attribution — not all incidents during a post-deployment window are caused by the deployment. Require evidence in the RCA, not just temporal proximity.

**T-Mobile Context Example**: 150 incidents in a month. 45 have root cause attributed to a production change. CICR = 30%. A deeper cut shows 30 of the 45 are from three services that all have CFR above 20%. Those three teams are the corrective action target, not the organization as a whole.

## 11. Change Quality Score (CQS)

**Definition**: The distribution of change outcomes across the three quality categories (Success, Partial Success, Failure) by team and service. CQS is the primary leading indicator for change program health and the basis for team-level quality conversations.

**Categories** (from the PIR template):
- **Success**: Implemented as planned; no unexpected deviations; customer impact within bounds; no rollback; no post-change incident
- **Partial Success**: Deviations occurred but change was completed; impact within bounds; minor post-change issues resolved within 24 hours
- **Failure**: Rollback required, OR impact exceeded estimated bounds, OR post-change incident attributed to change, OR change not completable within maintenance window

**Formula** (report as distribution, not single number):
```
CQS Success Rate = (count(Success outcomes) / count(all completed changes)) × 100
CQS Partial Rate = (count(Partial Success outcomes) / count(all completed changes)) × 100
CQS Failure Rate = (count(Failure outcomes) / count(all completed changes)) × 100
```

**Target distribution**:
- Success: > 70%
- Partial Success: 15–25%
- Failure: < 10% (correlates with CFR target of < 15% for High tier)

**Data Sources**: Change management tool: post-implementation outcome field on RFC records; PIR quality score field.

**Measurement Frequency**: Monthly.

**Director Review Cadence**: Monthly by team. Quarterly trend. Teams with Failure rate above 15% get a dedicated quality review. The pattern of failure reasons (from PIR root cause field) drives the corrective action — planning gaps get RFC template improvements, execution gaps get runbook improvements, environment gaps get staging investment.

**Common Pitfalls**:
- Not closing RFCs with outcome fields — if 30% of RFCs are left in "Implemented" status without a final outcome recorded, CQS is based on a biased sample (teams that do PIRs tend to be more mature; incomplete RFCs cluster at lower-quality teams).
- Treating Partial Success as equivalent to Success — a change that required 90 minutes of unplanned steps beyond the RFC plan is not a success; it is a planning gap that should drive RFC quality improvement.
