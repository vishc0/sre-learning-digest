# Change & Release Management — Navigation Guide

## What This Is

This framework covers Change Management and Release Management as a unified engineering governance discipline. They are not separate departments with separate meetings. They are two halves of the same system:

- **Change Management** is the governance system — who decides, what risk is acceptable, what approval is required, what gets logged.
- **Release Management** is the engineering system — what artifact is moving, through what pipeline, using what deployment strategy, with what rollback.

One without the other fails. A release pipeline with no change governance is a production incident waiting to happen. Change governance without release engineering rigor is bureaucratic theater — paperwork that does not reduce risk because it has no connection to how software actually moves.

This framework is written for SRE leaders targeting Director and VP of Operations roles. It assumes deep operational experience (middleware, Kubernetes, production incident command) and focuses on the governance and engineering decisions that matter at leadership level — not the mechanics of filling out tickets.

## How to Navigate

### Scenario 1 — Emergency change happening right now

You do not have time to read chapter documents. Go directly to:

1. `playbook/emergency-change-playbook.md` — declaration criteria, compressed review steps, minimum approvals
2. `templates/emergency-change-template.md` — the abbreviated RFC to complete within 4 hours of execution
3. `ch06-emergency-change-process.md` — read this after the event, during PIR prep

One immediate check: does this actually qualify as an emergency? An emergency change is an incident-driven mitigation, a confirmed-exploit security patch, or a hard regulatory deadline. It is not a planned change that ran out of time. If you are unsure, read the declaration criteria in the playbook before proceeding.

---

### Scenario 2 — Preparing a CAB submission

Your team has a Normal change coming up and needs to submit an RFC. Sequence:

1. `templates/change-risk-assessment-template.md` — score the change before you write anything else; the risk score determines CAB review depth
2. `reference/change-risk-matrix.md` — reference grid for impact × likelihood scoring
3. `templates/rfc-template.md` — the RFC document; your risk assessment output populates the risk section
4. `ch04-change-planning-and-impact-assessment.md` — if you need to map blast radius or plan the maintenance window
5. `playbook/rfc-submission-playbook.md` — submission SLA, CAB meeting schedule, what happens after submission

Key discipline: the blast radius assessment comes before the RFC, not inside it. You cannot write a credible rollback plan if you do not know what services depend on the component you are changing.

### Scenario 3 — Designing the program as Director/VP

You are inheriting or building a change and release management program. Read in this order:

1. `ch01-foundations-change-vs-release.md` — the foundational taxonomy and DORA context; sets the vocabulary
2. `ch02-change-types-and-risk-classification.md` — how to build the risk framework your teams will use
3. `ch05-release-governance-and-dora.md` — how to instrument release pipelines and measure what matters
4. `ch03-change-advisory-board.md` — how to structure a CAB that adds risk reduction rather than process overhead
5. `ch07-change-metrics-and-reporting.md` — what to present to your executive stakeholders
6. `ch08-post-implementation-review.md` — how to close the feedback loop so the program improves over time

Your first action in any new role: pull your DORA baseline. Deployment Frequency and Change Failure Rate for the trailing 6 months. That data tells you which failure mode your organization is in before you have read a single runbook.

## Three Failure Modes

Change and release management programs fail in three distinct patterns. Recognizing which one you are in determines the correct intervention.

### Failure Mode 1 — Change Control as Bureaucratic Theater

**What it looks like:** CAB meetings where no one has read the RFC before the meeting. Approvals require 10 business days for a configuration change. The Standard Change catalog has not been updated in two years. Emergency change percentage exceeds 40% because teams learned that the emergency path is faster. Change tickets are filled out retroactively — the change happened on Friday, the ticket is created Monday. PIRs are never scheduled.

**Why it happens:** The change process was designed to create audit evidence, not to reduce risk. Leadership tracks change volume and SLA compliance. No one tracks change failure rate. The process overhead is real but the risk reduction is theater.

**What breaks:** A catastrophic change goes through the same rubber-stamp process as every other change. The process provides no signal that this one is different. The postmortem finds the CAB approved a change with a missing rollback plan and a blast radius no one had mapped.

**The fix:** Make risk the organizing principle, not process compliance. Track Change Failure Rate per team per quarter. Require blast radius assessment before RFC submission, not inside it. Grow the Standard Change catalog so CAB time concentrates on genuinely risky changes.

---

### Failure Mode 2 — Engineering Velocity Without Governance

**What it looks like:** Teams deploy continuously with no change record. There is no RFC process. No rollback plan is documented because "we can always redeploy." Blast radius is unknown because no service dependency map exists. The first time anyone thinks about risk is when the monitoring fires. Changes made six hours ago are not traceable when an incident starts.

**Why it happens:** The organization optimized for deployment frequency as the sole metric. DORA was read selectively — the velocity metrics without the stability metrics. Engineers experienced a slow, bureaucratic process somewhere else and dismantled it without replacing it.

**What breaks:** A change touches shared infrastructure. The blast radius is an order of magnitude larger than anyone expected. The incident takes 4 hours instead of 45 minutes because no one can identify which change caused it. The rollback takes an additional 2 hours because it was never tested.

**The fix:** Instrument first. You cannot improve blast radius assessment without a service dependency map. Build the Standard Change catalog immediately — it creates change records without adding process friction. Require rollback plans to be tested, not just documented.

---

### Failure Mode 3 — Governance Without Feedback Loops

**What it looks like:** Changes are approved and logged correctly. PIRs are scheduled but canceled. Failure data from failed changes never feeds back into the risk classification model. The Standard Change catalog never grows — every change that was Normal two years ago is still Normal today. CAB reviews the same change types at the same depth year after year, regardless of track record.

**Why it happens:** The program was built as a compliance artifact. The front-end (RFC submission, CAB review) works. The back-end (PIR, catalog update, metric review) was never built. Process owners measure inputs (changes submitted, CABs held) not outcomes (change failure rate, standard change %).

**What breaks:** The program does not improve. Change Failure Rate does not decline. CAB becomes a bottleneck because the Standard Change catalog never grows to absorb low-risk repeatable work. Teams lose confidence in the process and route around it.

**The fix:** The quarterly CAB catalog review is not optional — it is the core feedback loop. Every PIR must ask: should this change type become Standard? Every change failure must update the risk matrix. Track Standard Change % quarterly and set a target.

## Key Definitions

| Term | Precise Definition |
|------|--------------------|
| **Change** | Any addition, modification, or removal of a configuration item that could affect services in production. Includes code, configuration, schema, infrastructure, and dependencies. |
| **Release** | A versioned, immutable artifact — container image, Helm chart, IaC module, database migration script — approved for deployment. Releases happen in the CI pipeline. |
| **Deployment** | The act of placing a release into a target environment. May be phased: canary, blue-green, rolling. One release may produce many deployments. Deployments happen in the CD pipeline. |
| **RFC (Request for Change)** | The formal document describing a planned change. Equivalent to a pull request for production — it describes the what, why, risk, rollback, and approval chain. |
| **CAB (Change Advisory Board)** | The governance body that reviews Normal changes for risk. Not an approval committee — a risk review panel. Its purpose is to surface risks the submitter missed, not to grant permission. |
| **PIR (Post-Implementation Review)** | Structured review after a change is implemented, regardless of outcome. The planned-change equivalent of a postmortem. Required after every Emergency change; recommended after every Normal change that touched P1 or P2 services. |
| **Standard Change** | A pre-approved, repeatable change with a documented procedure, defined success/failure criteria, and tested rollback. No CAB review required per instance. CAB approves the change type, not each execution. |
| **Normal Change** | A planned change that requires RFC submission and CAB review before implementation. The default classification for any change that is not pre-approved (Standard) or time-critical (Emergency). |
| **Emergency Change** | A change required to resolve an active production incident, address a confirmed security exploit, or meet a hard regulatory deadline. Bypasses standard CAB cycle. Requires compressed review and mandatory post-hoc PIR within 24 hours. |

## Relationship to Other Frameworks

**Change Management ↔ Incident Management**

Changes are the single largest source of production incidents. Industry data consistently shows 70-80% of incidents are caused by recent changes. The blast radius concept used in incident triage is identical to the blast radius assessment required before a Normal change. The difference is timing: pre-change blast radius assessment is a risk reduction tool; post-incident blast radius assessment is a scope determination tool.

Failed changes that cause incidents generate Incident records. The incident postmortem and the change PIR are parallel documents. For significant incidents caused by a change, the PIR and postmortem should be co-authored — they address the same event from two angles.

**Change Management ↔ Problem Management**

A failed change that triggers an incident may reveal an underlying systemic weakness — a problem record. Example: a database schema migration fails because the migration toolchain does not validate backward compatibility. The immediate fix (roll back the schema) is the incident resolution. The problem record captures "migration toolchain lacks backward-compatibility validation." The problem fix may result in a new Standard Change procedure (validated, scripted schema migrations).

Problem Management's known error database should inform CAB risk assessments. If a known error exists for a component that a proposed change touches, the CAB review depth should increase.

**Change Management ↔ DORA Metrics**

DORA's four key metrics are the executive scorecard for this entire framework:

| DORA Metric | What It Measures | Change/Release Connection |
|-------------|-----------------|--------------------------|
| **Deployment Frequency** | How often code reaches production | Constrained by Normal change lead time and CAB throughput |
| **Lead Time for Changes** | Time from code commit to production | Constrained by RFC process, CAB schedule, maintenance window availability |
| **Change Failure Rate** | % of changes causing degradation or rollback | The primary output metric of this entire framework |
| **MTTR** | Time to restore service after failure | Constrained by rollback plan quality and blast radius clarity |

Deployment Frequency and Lead Time measure velocity. Change Failure Rate and MTTR measure stability. A healthy change program improves all four simultaneously — not by choosing speed over safety, but by automating Standard Changes and reducing blast radius so each deployment is lower risk.

A Director who monitors only velocity metrics will eventually produce a catastrophic change and not understand why. A Director who monitors only stability metrics will strangle deployment frequency and lose engineering talent. Watch all four.
