# Incident Management: The Complete Framework

## What This Is

Production incidents are the most expensive events in software operations. They are also the most information-rich. This framework converts both dimensions — the cost and the intelligence — into a systematic practice.

This is not a theory document. It is a working framework covering:
- What to do in the first five minutes of an incident
- How to quantify customer, financial, and business impact in real time
- How to decompose cascading failures while the failure is still happening
- Which RCA framework to apply to which type of failure
- How to implement fixes at every level (code, data, traffic, infrastructure)
- How to govern the post-incident process so action items actually close
- What rules each role follows — from on-call SRE through the executive sponsor

## The Three Failure Modes of Incident Management Programs

Most organizations fail at incident management in one of three patterns:

**Failure Mode 1: Technical competence without command structure**
Engineers know their systems deeply but have no defined IC role, no scribe, and no communication protocol. The result: multiple engineers push conflicting changes, stakeholders receive contradictory updates, and the timeline reconstructed in the postmortem is fiction.

**Failure Mode 2: Process compliance without learning culture**
Organizations run postmortems but they are blame sessions in disguise, or box-checking exercises that produce vague action items ("improve monitoring") that are never tracked to completion. The same incidents recur because the postmortem is theater.

**Failure Mode 3: Learning culture without engineering investment**
The team identifies root causes accurately and writes excellent postmortems. Engineering leadership does not fund the remediation. The blameless culture covers for an org that will not invest in systemic fixes. Incident rates plateau or worsen over time.

This framework addresses all three. Chapters 1–6 address Failure Mode 1. Chapters 7–9 address Failure Mode 2. Chapter 10 addresses Failure Mode 3.

## One Number to Know Before You Start

**MTTR is not what you think it is.**

Most organizations measure MTTR as time-to-service-restoration. What they are actually measuring is time-to-page-closure — a number that reflects when someone closed the ticket, not when the customer's experience was restored.

The gap between "service green" and "customer experience restored" includes: cache invalidation, CDN propagation, session re-establishment, retry backoff clearing, and manual customer-service remediation. In high-severity incidents, this gap can be hours.

This framework uses MTTR precisely: it ends when customer-facing impact is zero, verified — not when the monitoring dashboard turns green.

## Key Definitions (Used Consistently Across All Files)

| Term | Precise Definition |
|------|-------------------|
| **Incident** | Any unplanned interruption or degradation of service quality that violates an SLO or SLA, requires coordinated response, or causes customer-visible impact |
| **Incident Commander (IC)** | Single accountable decision-maker for the duration of the incident; does not debug |
| **MTTA** | Mean Time to Acknowledge — from alert fire to human acknowledgment |
| **MTTD** | Mean Time to Detect — from actual failure onset to alert fire |
| **MTTR** | Mean Time to Restore — from incident declaration to customer impact = zero |
| **MTTF** | Mean Time to Failure — from last recovery to next failure; inverse is incident frequency |
| **PIR** | Post-Incident Review (preferred term over "postmortem" in blameless cultures) |
| **Blast radius** | The set of services, journeys, and customers affected by a failure at a given point in time |
| **Cascading failure** | A failure that propagates beyond its origin service due to missing isolation mechanisms |
| **Mitigation** | Action that reduces customer impact without necessarily resolving root cause |
| **Resolution** | Action that eliminates the root cause; customer impact = zero after resolution |
| **Runbook** | Step-by-step operational procedure for a known failure mode with expected output at each step |
| **Playbook** | Scenario-based response framework for a category of incident, broader than a runbook |
