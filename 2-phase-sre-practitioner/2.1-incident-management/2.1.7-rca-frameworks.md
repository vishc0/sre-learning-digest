# Chapter 7: Root Cause Analysis Frameworks

> **Audience**: Principal SRE / SRE Manager — practitioner-level reference. Assumes deep operational experience with distributed systems, Kubernetes, messaging platforms (Kafka, RabbitMQ), and observability tooling (Splunk). This chapter is a working reference, not a survey.

---

## Part 1: Why RCA Frameworks Matter

### The Human Error Trap

Left to their own devices, postmortem RCA processes converge on the same finding: **human error**. Someone deployed without testing. Someone didn't follow the runbook. Someone missed the alert.

This finding is almost always technically accurate and operationally useless.

Human error as a root cause has zero actionability. You cannot fix "humans make mistakes." What you can fix is the system that made the mistake easy to make, hard to catch, and catastrophic when it landed. The moment a postmortem accepts "human error" as a root cause, the systemic gap that allowed the error to have impact goes unaddressed — and the incident recurs.

**RCA frameworks exist to prevent this.** They are forcing functions that push investigation past the human action to the system conditions that made the human action consequential.

> Analogy: When a plane crashes because a pilot misread an instrument, the NTSB does not write "pilot error" and close the file. They ask: why was the instrument ambiguous? Why did the cockpit design allow misreading? Why did CRM procedures not surface the discrepancy? The human action is the start of the investigation, not the end.

---

### The CDR Model: Contributing → Direct → Root

Before selecting a framework, you need a shared vocabulary for causal layers. The CDR model provides it.

**Contributing Cause**: A condition that created the environment where failure was possible. These are pre-existing systemic gaps. They are almost never what caused the incident today, but they are why the incident had the impact it did.

*Example*: No circuit breakers on the Auth service's dependency on the User DB. This didn't cause anything — until something else did.

**Direct Cause**: The proximate trigger. The thing that actually moved the system from degraded to failing. This is what engineers find first and often mistake for the root cause.

*Example*: User DB latency spiked from 8ms to 4,200ms during a batch analytics job that was misconfigured to run against production.

**Root Cause**: The systemic gap that allowed the direct cause to have the impact it did. Fixing the root cause prevents a recurrence — not just of this specific trigger, but of the entire class of failure.

*Example*: No capacity isolation between analytical and transactional workloads on User DB, and no observability to detect DB degradation before downstream timeouts propagated.

**The critical test**: If you fix only the direct cause, you have closed this specific incident. If you fix the root cause, you have closed the class of incident. RCA frameworks are tools for finding the root cause, not just the direct cause.

---

### Framework Selection Overview

The right framework depends on the failure mode. A single-service timeout and a cross-datacenter cascade with three simultaneous contributing causes require fundamentally different analytical tools.

| Failure Type | Primary Framework | Secondary |
|---|---|---|
| Single-service failure with clear trigger | 5 Whys | CDR |
| Multi-service cascade | Apollo / Cause Map | Fishbone (6S) |
| Human error as contributing factor | 5 Whys + Fishbone (Staff) | STAMP (control structure) |
| Recurring failure (same mode, different causes each time) | STAMP | FTA |
| Pre-launch risk assessment | FMEA | FTA |
| Third-party dependency failure | Fishbone (Suppliers) | 5 Whys |
| Data corruption / integrity failure | Apollo | 5 Whys |

The rest of this chapter covers each framework in depth: what it is, how it works, where it fails, and how to apply it to real incidents.

## Part 2: 5 Whys

### Origin and Intent

5 Whys comes from the Toyota Production System, developed by Taiichi Ohno in the 1950s as part of lean manufacturing. The original application was physical manufacturing defects — a stamping machine produces bad parts, and the goal is to find why at the system level, not just stop the machine.

The technique was popularized in software by the DevOps movement, and it's now the most widely used (and widely abused) RCA technique in the industry.

**Core premise**: Any failure has a direct cause. That cause has a cause. That cause has a cause. If you keep asking "why" rigorously, you will eventually reach a point where the answer is a systemic gap — something about the system's design, standards, or processes that allowed this chain to form. That systemic gap is the root cause.

The number five is not sacred. Ohno observed that most manufacturing defects reached the systemic level in about five iterations. In complex software systems, it often takes more. The technique ends when you reach a systemic gap, not when you hit a count.

---

### Mechanics

**Start with the observable symptom as the first statement**, not as the first question. The symptom must be specific and factual.

Wrong start: "Why did the service go down?"
Right start: "Checkout API returned 503 errors for 94% of requests from 14:07 to 14:34 UTC."

Then ask why at each layer. Each answer must be validated against evidence — logs, metrics, traces — before you ask the next why. 5 Whys is not a brainstorming technique. Every node in the chain requires evidence.

---

### Worked Example: Authentication Service Timeout Cascade

**Incident**: Checkout service returned HTTP 503 to users for 27 minutes during peak traffic. 41,000 users affected. $230K estimated revenue impact.

**Symptom statement**: Checkout API returned 503 at 94% error rate from 14:07 to 14:34 UTC.

---

**Why 1: Why did Checkout return 503?**

Answer: Checkout calls Auth service synchronously to validate session tokens. Auth service calls were timing out after the 5-second client-side timeout configured in Checkout's HTTP client.

Evidence: Checkout application logs show `AuthServiceClient: timeout after 5000ms` at 14:07:03. Splunk query: `index=checkout-prod "AuthServiceClient: timeout"` shows 41,200 occurrences between 14:07 and 14:34.

**Why 2: Why did Auth service calls time out?**

Answer: Auth service was processing requests slowly because its database connection pool was exhausted. New requests to Auth waited for a connection, hit the 5-second timeout before a connection was available, and returned errors.

Evidence: Auth service metrics (Prometheus) show `db_connection_pool_active` at 100/100 (pool exhausted) from 14:06:52. Auth service logs show `HikariPool: Timeout waiting for connection from pool` starting 14:06:58.

**Why 3: Why was the connection pool exhausted?**

Answer: A configuration change deployed at 13:58 UTC introduced a new session validation query that performed a full table scan on the `user_sessions` table instead of using the indexed `session_token` column. Each query took 8-45 seconds instead of the normal 2-5ms. Connections were held for the duration of each query, exhausting the pool.

Evidence: Auth service DB slow query log shows queries starting at 14:02:11 with execution time >8000ms. Git diff on config PR #4471 (merged 13:55 UTC) shows query template change. `EXPLAIN` output on new query shows full table scan (12.3M rows examined).

**Why 4: Why did the long-running queries from the config change not get caught before production?**

Answer: Auth DB has no slow query alerting configured. The staging environment uses a database with 50,000 rows (instead of 12.3M in production), so the full table scan completed in ~90ms in staging — well within the acceptable response time threshold. No performance test was run against production-scale data before the config change was merged.

Evidence: Auth DB CloudWatch metrics show no slow query alarm configured. Staging DB row count: `SELECT COUNT(*) FROM user_sessions` = 48,231. Production row count = 12,349,087. Auth team's deployment checklist has no performance validation step.

**Why 5: Why was there no slow query alerting on Auth DB, and why does staging not reflect production data scale?**

Answer: No organizational standard requires database dependency observability for services. Each team decides its own monitoring. The Auth team's observability coverage was designed around application-level metrics (latency, error rate) but does not extend to database-level health indicators. There is also no standard for staging data scale or performance testing requirements before configuration changes to query logic.

Evidence: SRE standards document (v2.1, last updated 18 months ago) has no mention of database-layer observability requirements. Auth team's runbook has no database monitoring section. Staging environment provisioning script uses a fixed seed of 50K rows.

---

**Root Cause** (CDR Model):

- **Root Cause**: No organizational standard for database dependency observability. Each team makes its own monitoring decisions, creating undetected gaps.
- **Contributing Cause 1**: Staging environment does not reflect production data scale, making performance regressions invisible in pre-production validation.
- **Contributing Cause 2**: No performance testing requirement for configuration changes that modify query logic.
- **Direct Cause**: Config PR #4471 introduced a full table scan query that degraded under production data volume.

---

### 5 Whys Strengths

- **Fast**: A skilled team can complete a 5 Whys in 30-45 minutes for linear failures.
- **Accessible**: Requires no specialized training. Any engineer can participate.
- **Linear failures**: When a failure has one primary causal chain, 5 Whys maps it cleanly.
- **Evidence discipline**: The requirement to validate each "why" with evidence prevents storytelling and opinion from dominating the postmortem.

---

### 5 Whys Failure Modes

**Failure Mode 1 — The Branching Problem**

5 Whys assumes a linear chain. Most production incidents are not linear. Different engineers running 5 Whys on the same incident often reach different root causes — not because one is wrong, but because they followed different branches. When this happens, you're not doing 5 Whys wrong; you've hit the limit of the tool. Switch to Apollo or Fishbone.

**Failure Mode 2 — Premature Stop at Human Action**

The most common failure mode: the chain reaches a human action ("engineer deployed without running performance tests") and stops. This is where 5 Whys is most often misapplied. The human action is itself a why. Why did the engineer deploy without running performance tests? Because there was no standard requiring it. That is the systemic gap.

A well-facilitated postmortem keeps asking why past any human action until it reaches a process, standard, tooling, or design gap. Human actions are never root causes — they are proximate events that expose systemic gaps.

**Failure Mode 3 — Concurrent Causes**

5 Whys handles one causal chain. In a cascade failure where three things went wrong simultaneously — a config change, a traffic spike, and a dependency degradation — 5 Whys on any single chain misses the others. The incident happened because all three were present at the same time. Use Apollo or Fishbone for concurrent cause analysis.

---

### How to Run 5 Whys Correctly

1. **Designate a facilitator** who is not the on-call engineer. The on-call engineer has motivated reasoning; they want the root cause to be the thing they already fixed.
2. **Write every answer on a shared document or whiteboard** before asking the next why. Do not do this verbally — verbal 5 Whys drift.
3. **Validate each answer with evidence** before proceeding. If you cannot validate, you have a hypothesis, not a finding.
4. **Branch when you hit multiple causes**. Note the branch, continue the primary chain, then return and work the secondary chain.
5. **Stop at the systemic gap**, not at the first satisfying answer. The test: "If we fix this, does it prevent this class of failure, or just this specific trigger?"

## Part 3: Fishbone / Ishikawa Diagram

### Origin and Framework

The Fishbone diagram (also called the Ishikawa diagram or cause-and-effect diagram) was developed by Kaoru Ishikawa in the 1960s for quality management in Japanese manufacturing. The visual metaphor is a fish skeleton: the "head" is the effect (the failure), and the "bones" are categories of causes branching off a central spine.

The original manufacturing framework organized causes into six categories called the 6M: Man, Machine, Material, Method, Measurement, and Mother Nature (environment). For software systems, this doesn't translate directly — "material" and "mother nature" are awkward fits. The 6S adaptation is more useful.

---

### The 6S Framework for Software Systems

**Software** — Defects and gaps in application logic:
- Code bugs (race conditions, off-by-one errors, improper error handling)
- Untested code paths triggered by unexpected input
- Memory leaks or resource exhaustion under load
- Incorrect library versions or dependency conflicts
- Missing retry logic, missing circuit breakers, missing timeouts

**System** — Infrastructure and platform:
- Compute capacity limits (CPU, memory, disk)
- Network partitions, packet loss, routing failures
- Kubernetes node failures, pod scheduling failures
- Cloud provider incidents (AZ outage, service quota exhaustion)
- Container runtime failures, storage I/O saturation

**Schema** — Data model and contract changes:
- Database migration side effects (lock contention, index rebuild blocking)
- API contract breaks (field removal, type changes, required field additions)
- Message schema changes breaking consumers (Kafka schema registry, Avro evolution)
- Data volume growth that invalidates previous performance assumptions

**Staff** — Human factors:
- Manual steps performed incorrectly (wrong target environment, wrong scope)
- Knowledge gaps (engineer unfamiliar with system behavior under failure)
- Inadequate runbooks (missing steps, outdated information)
- Process gaps (no peer review, no change approval, no rollback plan)
- Fatigue or pressure-induced errors (3am deploy, production pressure)

**Suppliers** — Third-party dependencies:
- SaaS provider outages (payment processor, identity provider, SMS gateway)
- CDN failures or misconfiguration
- DNS provider failures
- External API degradation (rate limiting, breaking changes, latency spikes)
- Certificate expiry on external service

**Specifications** — Requirements and design gaps:
- Missing SLO definition (no agreed threshold for what constitutes failure)
- Unclear service ownership (no team accountable for behavior)
- Missing non-functional requirements (no scale requirement in design doc)
- Incomplete failure mode specification (design assumed dependency is always available)
- Insufficient acceptance criteria in user stories

---

---

### When to Use Fishbone

- **Complex incidents with multiple contributing causes** — if you can count more than two contributing factors, Fishbone is more useful than 5 Whys
- **Cross-team failures** — the 6S categories create structured space for each team to identify causes in their domain without territorial conflict
- **When 5 Whys branches immediately** — if the first "why" has two legitimate answers, you've already outgrown 5 Whys; start with Fishbone
- **First hour of a postmortem** — Fishbone is an excellent brainstorming structure before you have complete evidence. Let teams populate their categories, then validate.

---

### Limitation: Fishbone Produces a Map, Not a Priority List

The output of a Fishbone diagram is a comprehensive map of contributing causes. It does not tell you which ones to fix first, which ones had the most impact, or which ones are already fixed by mitigations already in place.

**Pair Fishbone with criticality scoring**:

For each cause identified:
- **Impact**: if this cause were absent, how much would failure severity have been reduced? (H/M/L)
- **Likelihood of recurrence**: how likely is this cause to be present in the next incident? (H/M/L)
- **Fix complexity**: how hard is it to address? (H/M/L — inverse; Low complexity = quick fix)

Prioritize High Impact + High Recurrence causes regardless of fix complexity. Address Low Impact + Low Recurrence only if the fix is trivial.

## Part 4: FMEA (Failure Mode and Effects Analysis)

### Origin

FMEA was formalized in US Military Procedure MIL-P-1629 in 1949, originally for analyzing reliability of military equipment. NASA adapted it for the Apollo program — every subsystem was required to enumerate its failure modes before any human flew on it. The automotive industry (SAE J1739) and medical device industry (ISO 13485) have formalized FMEA requirements. Amazon and Google have internalized it into their pre-launch review processes, though they use different names.

**FMEA is fundamentally a pre-incident tool.** It is applied before launch, before major changes, and before migrations — not during active incidents. Its value is catching failure modes that have not yet occurred in production.

---

### Structure

For each component or subsystem in scope, you enumerate:

1. **Function** — what does this component do?
2. **Failure Mode** — in what ways can it fail to perform that function?
3. **Effect** — what is the impact of this failure mode on the system and on users?
4. **Severity (S)** — how bad is the effect? Scale: 1 (negligible) to 10 (catastrophic/safety-critical)
5. **Cause** — what could cause this failure mode?
6. **Occurrence (O)** — how often is this failure mode likely to occur? Scale: 1 (almost never) to 10 (almost certain)
7. **Current Controls** — what mechanisms currently detect or prevent this failure mode?
8. **Detectability (D)** — how likely is the current control to catch this before customer impact? Scale: 1 (always caught) to 10 (never detected)
9. **Risk Priority Number (RPN)** = S × O × D

---

### RPN Thresholds and Action Rules

| RPN Range | Classification | Required Action |
|---|---|---|
| > 200 | Critical | Immediate action before launch/change proceeds |
| 100 – 200 | High | Sprint-level priority; must be addressed within one sprint |
| 50 – 100 | Medium | Backlog with target date; must not recur |
| < 50 | Low | Monitor; address opportunistically |

**Important caveat on RPN arithmetic**: RPN is a relative ranking tool within a single FMEA session, not an absolute risk number. An RPN of 200 in a payment processing system means something different from an RPN of 200 in a logging pipeline. Use RPN to prioritize within scope, not to compare across systems.

---

### Worked Example: Payment Service Pre-Launch FMEA (Excerpt)

| Component | Failure Mode | Effect | S | Cause | O | Controls | D | RPN | Action |
|---|---|---|---|---|---|---|---|---|---|
| Stripe API Client | API returns 502 | Payment fail | 8 | Stripe degradation | 3 | Retry logic | 5 | 120 | Add circuit breaker; configure fallback processor |
| Connection Pool | Pool exhaustion | All requests queue/fail | 9 | Long-running txns + no pool limit | 4 | Pool size config | 7 | 252 | Add slow query alerting; pool timeout alert |
| Session Token Validation | Full table scan | Latency spike cascade | 8 | Missing index on query | 2 | Code review | 6 | 96 | Add query plan review to pre-launch checklist |
| DLQ | DLQ fills silently | Silent message loss | 9 | No DLQ depth alarm | 5 | None | 10 | 450 | Add DLQ depth alarm before launch |
| Fallback Processor | Not configured | Single provider dependency | 8 | Design gap | 8 | None | 10 | 640 | Add fallback before launch (blocker) |

In this excerpt, the DLQ alarm gap (RPN 450) and fallback processor gap (RPN 640) are launch blockers. The slow query detection gap (RPN 96) is a sprint-level item.

---

### Amazon's Operational Readiness Review (ORR)

Amazon's ORR is the industry's most formalized FMEA-equivalent. Before any new service launches or any significant change is deployed, teams must complete an ORR that includes:

- Enumeration of all failure modes (structured exactly like FMEA)
- Mitigation documented for each failure mode rated High or Critical
- Rollback plan documented and tested
- Runbook for each high-severity failure mode
- Alarm coverage verified for each failure mode where detection is possible

The ORR is reviewed by a senior engineer outside the team. Services that fail ORR are not launched. This is the operational culture equivalent of FMEA at scale.

**For T-Mobile platform context**: An ORR-equivalent for the notification platform would enumerate failure modes for each dependency (Kafka consumer lag, APNS connection limits, SMS gateway rate limits, device token expiry rates) and require mitigation for each before a major release. Currently, this analysis typically happens reactively, after incidents.

---

### FMEA Limitation: Known Failure Modes Only

FMEA is only as good as the team's imagination. It catches failure modes that you can enumerate. It does not surface unknown unknowns — failure modes that have never occurred, that emerge from unexpected interactions, or that arise from external changes (new traffic patterns, new adversary techniques, new cloud provider behaviors).

**Complement FMEA with**:
- **Chaos Engineering** (Chaos Monkey, Gremlin, Litmus): inject known failure modes in production to validate that controls work
- **Game Days**: simulate complex failure scenarios against real systems with real teams
- **Retrospective FMEA**: after any novel incident, add the new failure mode to the FMEA and evaluate whether your current controls would have caught it

## Part 5: Apollo Root Cause Analysis (Apollo Method)

### Origin and Key Insight

The Apollo Method was developed by Dean Gano in 1999, drawing on reliability engineering practices from aerospace and nuclear industries. It is sometimes called Cause Mapping (TapRooT uses a similar approach).

**The key difference from 5 Whys**: 5 Whys is a chain. Apollo is a tree.

In a 5 Whys chain, each effect has exactly one cause. In Apollo, each effect can have multiple simultaneous causes, and each cause can have multiple contributing conditions. This is far more accurate to how complex systems actually fail.

The single-chain illusion of 5 Whys leads teams to fix one cause and consider the incident resolved. Apollo's tree structure requires teams to address all necessary and sufficient causes — because the incident happened at the intersection of multiple conditions, and any one of them remaining unaddressed is enough to allow recurrence.

---

### Structure

**Root Node**: The undesired event (stated precisely with metrics).

**Three questions at each node**:
1. What caused this? (Identify all causes — not just the most obvious one)
2. Are there other causes? (Specifically probe for additional causes before moving on)
3. What is the evidence? (Every cause requires supporting evidence)

**Logical connectives**:
- **AND**: this cause AND this other cause were both necessary for the effect to occur
- **OR**: this cause OR this other cause is sufficient to produce the effect (either alone would have caused it)

**Terminal nodes** (stop drilling when you reach one):
- A node that is outside the scope of control (external, uncontrollable)
- A node where fixing it prevents the class of failure (systemic gap)
- A node where the evidence trail runs cold and cannot be verified

---

### Worked Example: Notification Platform Cascade

**Incident**: Push notification delivery dropped to 12% of normal volume for 45 minutes. Affects iOS and Android. SMS fallback not activated.

```
EFFECT: Push notification delivery at 12% for 45 min [AND]
    │
    ├── CAUSE A: APNS connection pool exhausted [AND]
    │       ├── CAUSE A1: Retry storm from upstream — 400K retry attempts/min
    │       │       └── CAUSE A1a: Consumer retry logic uses no exponential backoff
    │       │               Evidence: Consumer code; fixed 100ms retry interval confirmed
    │       └── CAUSE A2: APNS connection limit (1000 persistent connections) reached
    │               Evidence: APNS error logs "Connection refused — limit exceeded"
    │               └── CAUSE A2a: No connection pool circuit breaker configured
    │                       Evidence: Consumer service config; no CB setting present
    │
    ├── CAUSE B: Kafka consumer lag grew to 18 minutes [AND]
    │       ├── CAUSE B1: Consumer throughput dropped due to APNS connection saturation
    │       │       Evidence: Consumer metrics; throughput fell from 2.3K/s to 280/s at 14:08
    │       └── CAUSE B2: No consumer lag alarm configured
    │               Evidence: Alerting config; no lag alarm on notification-dispatch topic
    │
    └── CAUSE C: SMS fallback not activated [AND]
            ├── CAUSE C1: Runbook requires manual IC decision to activate fallback
            │       Evidence: Runbook v2.3, Step 14: "IC to authorize SMS fallback"
            ├── CAUSE C2: IC not paged until 14:31 (24 minutes into incident)
            │       Evidence: PagerDuty escalation log
            └── CAUSE C3: SMS fallback activation requires manual config change (no automated trigger)
                    Evidence: Fallback design document; no automated threshold defined
```

**Apollo output**: Three independent cause clusters. If any one of them were absent:
- Without Cause A: retry storm would not have saturated APNS
- Without Cause B: consumer lag would have been visible and triggered alarm
- Without Cause C: fallback would have activated within minutes

The incident required all three. Fixing only the retry storm (the "obvious" direct cause) leaves the system vulnerable to the next APNS saturation event — which will again have no alarm and no auto-fallback.

**Action items from Apollo tree** (one per terminal node):
1. Implement exponential backoff with jitter in consumer retry logic (A1a)
2. Configure APNS connection pool circuit breaker (A2a)
3. Add Kafka consumer lag alarm at 5-minute threshold (B2)
4. Add automated SMS fallback trigger at defined APNS failure threshold (C3)
5. Review IC escalation path — 24-minute delay before IC paged is unacceptable (C2)

---

### When to Use Apollo

- **Major incidents with multiple simultaneous contributing causes**
- **Cross-service cascade failures** — any incident where the postmortem needs to cover more than two teams
- **When 5 Whys produces different root causes from different participants** — that's a signal you need Apollo's tree structure
- **Data corruption and integrity failures** — these almost always have multiple necessary conditions

Apollo is slower than 5 Whys and requires a skilled facilitator. Budget 2-4 hours for a complete Apollo analysis of a major incident. It is not appropriate for routine incidents.

## Part 6: Fault Tree Analysis (FTA)

### Structure and Logic

Fault Tree Analysis is a top-down, deductive analytical technique. You start from an undesired top-level event and decompose it into combinations of lower-level events using boolean logic gates.

**AND gate (⊓)**: The top event occurs only if ALL inputs to the gate occur simultaneously. AND gates represent failure modes that require multiple conditions to align. They are harder to trigger (good) but harder to prevent (you must fix all branches).

**OR gate (⊔)**: The top event occurs if ANY input to the gate occurs. OR gates represent failure modes where any single condition is sufficient. They are easier to trigger (bad) but easier to prevent (fixing any single branch eliminates the path).

**Basic events** (terminal nodes): Individual failure modes at the component level, with associated probabilities if quantitative analysis is needed.

**Inhibit gate**: A special AND gate where one input is a conditional event — the failure only propagates if a condition is true (e.g., "maintenance mode disabled").

---

### Example: Payment Service Availability Fault Tree (Simplified)

```
                    [Payment Service Unavailable]
                              ⊔ (OR)
                 ┌────────────┴────────────────┐
                 │                             │
        [Application Failure]          [Infrastructure Failure]
              ⊔ (OR)                        ⊔ (OR)
         ┌────┴────┐                   ┌────┴────┐
         │         │                   │         │
    [Code Bug]  [Dependency]        [Node     [Network
                  Failure           Failure]   Partition]
                  ⊔ (OR)             ⊓ (AND)
              ┌───┴───┐         ┌────┴────┐
              │       │         │         │
          [Stripe  [DB        [No Auto-  [Zone
          Outage]  Failure]   Healing]   Outage]
```

In this tree:
- Payment service is unavailable if EITHER application OR infrastructure fails (OR gate at top)
- Node failure only causes service unavailability if BOTH the node fails AND auto-healing is disabled (AND gate)
- This means: fixing auto-healing (re-enabling it) makes node failure a non-issue, even without fixing the node failure probability itself

---

### Quantitative FTA

When component failure probabilities are known (from historical data or vendor SLAs), FTA can quantify the probability of the top-level event.

For an OR gate: P(top) = 1 - ∏(1 - P(input_i))
For an AND gate: P(top) = ∏(P(input_i))

This allows engineers to calculate service availability probability from component reliability data, identify which component has the highest contribution to top-event probability, and evaluate the ROI of different investments (fixing component A vs. component B).

**Practical application**: If Stripe SLA is 99.9% (0.1% failure probability) and your fallback processor SLA is 99.95%, the AND gate probability of both failing simultaneously is 0.001 × 0.0005 = 0.0000005 — effectively zero. This quantifies the value of adding a fallback processor.

---

### When to Use FTA

- **Safety-critical systems** where failure probability must be quantified (not just qualified)
- **Pre-launch design analysis** — evaluate architecture before a system is built
- **When you need to prioritize investments** — quantitative FTA shows which component improvements have the most impact on overall availability
- **Recurring failure mode analysis** — when the same type of failure keeps occurring despite fixes, FTA can reveal that multiple independent paths lead to the same failure

**What FTA is not good for**: Building an FTA during an active incident is impractical. It requires significant upfront knowledge of component behavior and failure probabilities. Use it in post-incident or pre-launch contexts.

## Part 7: STAMP/STPA (Systems Theoretic Accident Model and Process)

### Origin and Motivation

STAMP was developed by Nancy Leveson at MIT in 2002, building on her earlier work on software safety (the book *Safeware*, 1995). STPA (Systems Theoretic Process Analysis) is the hazard analysis technique derived from STAMP.

STAMP was designed because Leveson observed that traditional cause-effect chain analysis (5 Whys, FTA) was insufficient for modern sociotechnical systems. As systems grew more complex, automated, and tightly coupled, incidents increasingly arose not from component failures but from interactions between components that individually behaved correctly.

**The key insight of STAMP**: Accidents and failures in complex systems are not just component failures. They are failures of the system's control structure — the mechanisms by which the system enforces its safety constraints and coordinates behavior across components.

In a STAMP model, every system has:
- **Losses**: outcomes the system must prevent (service unavailability, data corruption, security breach)
- **Hazards**: system states that lead to losses (DB connection pool exhausted, auth token expired, message queue full)
- **Safety constraints**: rules that prevent hazards from becoming losses (max connection wait time, token refresh before expiry, DLQ alarm)
- **Control structure**: the mechanisms that enforce safety constraints (automated alerts, circuit breakers, deployment gates, runbooks, escalation policies)

When an incident recurs despite fixes, STAMP's explanation is usually: the control structure has a gap. The specific trigger changed, but the underlying control structure that should have prevented the trigger from causing impact was never repaired.

---

### STPA Process

**Step 1 — Identify Losses**
What outcomes must the system prevent? Be specific.
- Loss 1: Users unable to complete payment transactions
- Loss 2: Notification messages delivered with >15 minute delay
- Loss 3: Personally identifiable data exposed to unauthorized parties

**Step 2 — Identify Hazards**
What system states could lead to each loss?
- Hazard H1: Payment service error rate >5% for >60 seconds (→ Loss 1)
- Hazard H2: Notification consumer lag >10 minutes on any partition (→ Loss 2)
- Hazard H3: Auth token validation service returning 200 for expired tokens (→ Loss 3)

**Step 3 — Identify Safety Constraints**
What rules must hold to prevent each hazard?
- SC1: Payment service error rate alarm must trigger at 5% threshold with <2 min detection
- SC2: Consumer lag alarm must trigger at 10-minute threshold with automatic SMS fallback at 15 minutes
- SC3: Auth service must validate token expiry against current timestamp, not cached timestamp

**Step 4 — Model the Control Structure**
Map the control hierarchy: who/what enforces each safety constraint?

```
[Engineering Standards] → defines → [Deployment Gates] → enforces → [Code Review Checklist]
[Monitoring Platform] → enforces → [Alert Rules] → triggers → [On-Call Page]
[On-Call Engineer] → executes → [Runbook] → operates → [Production System]
[Incident Commander] → authorizes → [Escalation / Fallback] → operates → [Secondary System]
```

**Step 5 — Identify Unsafe Control Actions**
For each controller in the structure, what control actions could create hazards?

Examples:
- Alert rule: **Not providing control** (alert rule not configured → hazard never detected)
- Deployment gate: **Providing control at wrong time** (deployment gate bypassed during incident → config change introduced during peak → amplifies existing failure)
- Runbook: **Providing incorrect control** (runbook step has wrong threshold → on-call takes wrong action)
- IC escalation: **Too late** (escalation path requires manager approval before fallback → adds 20 minutes to MTTR)

**Step 6 — Identify Loss Scenarios**
What specific causal factors could lead to each unsafe control action?

This produces the actionable list of control structure improvements.

---

### Why STAMP Matters for SRE

Most SRE practice focuses on the technical layer: improve observability, add circuit breakers, fix the query. STAMP addresses the organizational and process layer: the control structures that coordinate human behavior and automated behavior during normal operations and failures.

**When to apply STAMP**: When a category of incident (database connection exhaustion, notification delivery failures, auth service cascades) keeps recurring — different specific triggers each time, different teams involved, but the same type of failure — STAMP explains why. The control structure has a gap. The alert fires too late, or the escalation path is too slow, or the runbook is wrong, or the deployment gate doesn't check the right thing. Fixing individual incidents without fixing the control structure guarantees recurrence.

**T-Mobile notification platform application**: If APNS delivery failures have occurred three times in 18 months with different technical root causes each time, the STAMP question is: what control structure failure allowed each trigger to become a customer-visible failure? The answer is usually: the control structure (alarms, escalation, runbooks, fallback automation) is insufficient for the class of failure, regardless of the specific trigger.

## Part 8: Framework Selection Guide

### Decision Table

| Failure Type | Primary Framework | Secondary | Rationale |
|---|---|---|---|
| Single-service failure, clear trigger | 5 Whys | CDR | Linear chain; fast to complete; one primary cause |
| Multi-service cascade | Apollo / Cause Map | Fishbone (6S) | Multiple simultaneous causes; needs tree structure |
| Human error as contributing factor | 5 Whys + Fishbone (Staff) | STAMP | 5 Whys to find systemic gap; Fishbone to check other categories; STAMP if recurrent |
| Recurring failure (same mode, different causes each time) | STAMP | FTA | Control structure gap; individual cause fixes insufficient |
| Pre-launch risk assessment | FMEA | FTA | Enumerate unknown failure modes before they occur |
| Third-party dependency failure | Fishbone (Suppliers) | 5 Whys | Dependency failures have multiple contributing system-side factors |
| Data corruption / integrity failure | Apollo | 5 Whys | Data integrity failures almost always have multiple necessary conditions |
| Safety-critical system design | FTA | FMEA | Quantifiable failure probability required |
| Complex sociotechnical system (automation + humans) | STAMP | Apollo | Component-level analysis insufficient; control structure analysis required |

### Practical Sequencing for a Major Incident Postmortem

1. **Within 24 hours of incident**: Run a rapid 5 Whys to identify the primary causal chain. This is the immediate postmortem output — fast, actionable, and good enough for the first set of action items.

2. **Within 72 hours**: If the 5 Whys branches, or if multiple teams were involved, run a Fishbone (6S) mapping session. This ensures all contributing cause categories are examined, not just the one the primary team focused on.

3. **Within 1 week**: If the incident was major (P1/P2), build an Apollo tree. Map all necessary conditions. Ensure every terminal node has an action item.

4. **After three recurrences of the same failure class**: Run a STAMP analysis on the control structure. Stop fixing individual incidents and fix the system that allows them to recur.

5. **Before the next major launch or migration**: Run FMEA on the components in scope. Use the incident history to populate occurrence and detectability scores with real data.

## Summary

| Framework | Origin | Mode | Best For | Limitation |
|---|---|---|---|---|
| 5 Whys | Toyota, 1950s | Post-incident | Linear failures, fast analysis | Branches badly; stops at human error |
| Fishbone / 6S | Ishikawa, 1960s | Post-incident | Multi-cause mapping, cross-team | Produces map, not priority list |
| FMEA | US Military, 1949 | Pre-incident | Launch risk assessment | Misses unknown unknowns |
| Apollo | Gano, 1999 | Post-incident | Complex cascades, multi-cause | Slow; requires skilled facilitator |
| FTA | Aerospace, 1960s | Pre/Post | Safety-critical, quantifiable risk | Impractical during active incident |
| STAMP/STPA | Leveson/MIT, 2002 | Post-incident | Recurrent failure classes, control structure gaps | Requires system-level modeling skill |

The frameworks are not competitors. They are tools for different jobs. A mature SRE practice uses all of them, selecting based on failure type and time horizon.
