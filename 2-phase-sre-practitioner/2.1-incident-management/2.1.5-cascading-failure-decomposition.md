## Part 1: What Cascading Failures Are

### Definition

A **cascading failure** is a failure that propagates beyond its origin service due to the absence of isolation mechanisms. The origin service fails or degrades; that degradation is transmitted — through synchronous call chains, shared thread pools, or retry storms — to services that had no fault of their own.

The word "cascade" is doing real work here: it implies directionality (propagation follows dependency edges), amplification (each hop can multiply the load on the next), and a failure to contain (something that should have been a local fault becomes a systemic one).

### The Home Circuit Breaker Analogy

Your home's electrical panel has individual breakers for each circuit — kitchen, bedroom, garage. If the garage circuit shorts, the breaker trips. You lose garage power. The bedroom lights stay on.

A home without breakers — or one where every circuit is wired in series — loses power everywhere when one circuit faults. Every appliance in the house goes dark because there was no isolation point.

Service architectures without circuit breakers behave exactly like houses without breakers. Service B depends on service A. Service A faults. B is waiting. B's threads are blocked. New requests to B queue up, then overflow. B goes down. Service C was depending on B. C goes down. The original fault in A has now consumed three services. If you had tripped a circuit breaker at the B→A edge the moment A started misbehaving, B would have served a degraded response and C would never have felt the fault.

Circuit breakers are the breaker panel of distributed systems. This chapter is about what happens when you don't have them — and how to decompose the damage when they fail to trip.

### The Three Cascade Patterns

#### Pattern 1: Cascade (Linear Propagation)

The foundational pattern. A failure propagates linearly down the call chain.

```
Service A fails
  → Service B calls A, waits, exhausts connection pool, B fails
    → Service C calls B, waits, exhausts connection pool, C fails
      → Service D calls C ... (continues)
```

Characteristics:
- Propagation is bounded by the call chain depth
- Each hop introduces latency before failure is detected
- The error signal at C looks like "B is down" — the origin (A) may not be visible to the on-call engineer at C

Key diagnostic question: *Is B failing because of B, or because of what B depends on?*

#### Pattern 2: Avalanche (Self-Amplifying Degradation)

More dangerous than linear cascade because it builds before it breaks. The origin service degrades — it doesn't fail outright, it slows down.

```
Service A degrades (high latency, not errors)
  → Upstream callers timeout waiting on A
    → Callers retry (each retry = additional load on A)
      → A's load increases → A degrades further
        → More timeouts → more retries → A fails completely
          → Linear cascade begins
```

Characteristics:
- The trigger is latency, not an error code — alert thresholds tuned to error rates miss this
- Retry behavior is the amplification mechanism; without jitter, retries synchronize
- The system is in a self-reinforcing feedback loop before anyone knows there is an incident
- By the time you declare an incident, A has already failed

The avalanche pattern is why **p99 latency alerts matter more than error rate alerts** for early detection. You want to catch A's degradation before it becomes A's failure.

#### Pattern 3: Thundering Herd

A single event causes simultaneous load from many clients against a shared dependency. Unlike the avalanche, it does not require progressive degradation — it is a step function: one event, then immediate overload.

Common triggers:
- Cache invalidation or cache restart (all clients miss simultaneously, all call origin)
- Service restart after downtime (all queued clients reconnect at once)
- Scheduled batch jobs all firing at the same second (cron alignment)
- Viral event spike (external traffic surge against an unshielded entry point)

```
Cache node restarts
  → 1,000 clients get cache miss simultaneously
    → All 1,000 call origin database simultaneously
      → Database connection pool exhausted
        → Database fails → services depending on DB cascade
```

Characteristics:
- Load is correlated across many clients — it looks like a DDoS from the inside
- The triggering event (cache restart) looks innocuous
- Jitter in client retry intervals is the primary mitigation; probabilistic cache warm-up is the structural fix

## Part 2: Failure Propagation Mechanics

### Dependency Graph Traversal

Every service in your system is a node; every call between services is a directed edge. When a failure occurs at node A, the blast propagates along outbound edges (to services A calls) and is felt by inbound edges (services that call A).

During an incident, you need to mentally (or literally) traverse this graph:

- **Downstream from origin**: What does A call? What do those services call? This is where data corruption risk lives — if A was mid-transaction when it failed.
- **Upstream from origin**: Who calls A? This is where customer impact propagates. If the payment service calls A, and A is down, payment is down.

In Kubernetes environments, Kiali (Istio service mesh) and Datadog APM generate live dependency graphs. In Splunk-instrumented environments, you can reconstruct the graph from trace IDs:

```splunk
index=app_traces earliest=-30m
| where trace_id IN [<suspect_trace_ids>]
| table _time, service, downstream_service, status_code, duration_ms
| sort _time
```

The graph you need is not the architectural diagram on the wiki — it is the live call graph at the time of the incident. Services that are supposed to be decoupled sometimes aren't at runtime.

### Synchronous vs. Asynchronous Coupling

**Synchronous coupling** (REST, gRPC): failure propagates immediately. When A fails, B's in-flight request fails in the same moment. B can circuit-break, but if it doesn't, B fails with A. Latency from A becomes latency in B's response time. There is no buffer.

**Asynchronous coupling** (RabbitMQ, Kafka, SQS): the producer and consumer are decoupled in time. If A fails, B's queue accumulates messages but B does not immediately fail. This is the buffer. However:

- If A is the *consumer* and it falls behind, the queue depth grows. Memory pressure builds on the broker. If the queue is unbounded, the broker runs out of memory. Asynchronous coupling deferred the cascade, it did not eliminate it.
- If consumers restart and immediately process a large backlog, they can overwhelm A's dependencies downstream — a thundering herd triggered by queue drain.

In RabbitMQ terms: a dead-letter queue (DLQ) that fills up during an outage is not a success — it is deferred failure. You must have a plan for draining it at a controlled rate post-recovery. See ch06, Part 6 for queue drain strategy.

### Connection Pool Exhaustion: The Primary Cascade Mechanism

This is the physical mechanism behind most cascade failures. Understanding it prevents blind debugging.

A connection pool is a bounded set of threads or sockets available for a service to make outbound calls. Say B's pool has 50 connections to A.

Scenario:
1. A slows down. Each connection to A is held longer than normal.
2. B's 50 pool slots are all occupied waiting on slow A responses.
3. New requests to B arrive. They need a connection to A. The pool is exhausted. They queue.
4. The queue overflows (or callers timeout at B's interface waiting for a slot).
5. B's own response times spike — not because B's code is broken, but because B cannot call A.
6. C, which calls B, sees the same pattern. C's pool to B exhausts. C fails.

The cascade was never really A→B→C. It was: "A was slow, and no one had set a short enough timeout or a connection limit to shed load."

Key parameters that determine whether pool exhaustion cascades:
- **Connection pool size**: smaller pools exhaust faster; larger pools delay the cascade but make it worse when it arrives
- **Connection timeout**: how long B waits for A before releasing the connection back to the pool
- **Request timeout**: how long B waits for A to respond before returning an error to its caller
- **Queue depth**: how many pending requests pile up before B starts rejecting

If you see your service's thread pool at 100% utilization and latency spiking, but your own code has not changed, connection pool exhaustion from a slow downstream is the first hypothesis.

### Thread Pool Exhaustion and the Bulkhead Pattern

Hystrix and Resilience4j implement the **bulkhead pattern** — named after the watertight compartments in a ship's hull. If one compartment floods, the bulkhead keeps the rest dry.

In thread pool terms: instead of one global thread pool shared across all downstream dependencies, each downstream gets its own thread pool. If the pool for A exhausts, only calls to A are affected. Calls to C, D, and E continue with their own pools.

```
Without bulkhead:
  [Global thread pool: 200 threads]
    → A (slow): eats 200 threads, pool exhausted
    → B: cannot get threads, fails
    → C: cannot get threads, fails

With bulkhead:
  [Thread pool for A: 50 threads]  ← exhausts, but isolated
  [Thread pool for B: 50 threads]  ← unaffected
  [Thread pool for C: 50 threads]  ← unaffected
  [Thread pool for default: 50 threads]  ← unaffected
```

Configuring bulkheads is a pre-incident architecture decision. During an incident, you cannot add bulkheads — but you can trip circuit breakers, which achieves the same isolation effect reactively.

### Retry Storm Amplification

Retries are good engineering practice — until they are not. When a service fails and clients retry, the aggregate load on the recovering service can exceed what it saw before failure.

**The math**:
- 100 clients each retry 3 times = 300 total requests during recovery
- Without jitter: all 100 clients waited the same 5-second interval → all 100 fire simultaneously → 300 requests in a 1-second window
- With jitter: clients are spread across a 5-10 second window → 30 requests/second → manageable

Exponential backoff without jitter is nearly as bad as fixed-interval retry for thundering herd scenarios:

```
Without jitter (exponential):
  t=0:    100 clients fail
  t=1s:   100 clients retry (synchronized)
  t=2s:   100 clients retry (synchronized)
  t=4s:   100 clients retry (synchronized)
  → Every retry wave hits simultaneously

With jitter (exponential + uniform random):
  t=0-1s: ~33 clients retry
  t=1-3s: ~33 clients retry
  t=3-7s: ~33 clients retry
  → Load spread, service has time to recover
```

When you are in a retry storm during an incident, rate limiting at the origin service is the emergency mitigation. The structural fix is jitter in client retry logic.

## Part 3: Decomposing a Cascade During an Incident

The following procedure is for real-time use during an active incident. It is sequential — do not skip steps.

### Step 1: Identify the Origin Service

The origin service is the **first service to show errors** — not the first service to be noticed by the on-call engineer. These are frequently different. The service that pages you is often a downstream consumer; the origin is upstream.

How to find the origin:
1. Sort services by first-error timestamp across your observability platform
2. Look for the service whose error rate spiked *before* others' error rates spiked
3. Check if the origin service's errors correlate with a deployment, config change, or infrastructure event

In Splunk:
```splunk
index=app_errors earliest=-1h
| stats min(_time) as first_error, count as error_count by service
| sort first_error
| table service, first_error, error_count
```

The service at the top of this table is your origin candidate. Validate by checking what it calls (downstream) and what called it (upstream) at the time of `first_error`.

### Step 2: Build the Live Dependency Graph

At incident time, the graph you need is the *runtime* graph, not the wiki diagram. Two ways to build it fast:

**From service mesh telemetry** (Istio/Linkerd): your mesh already has live call data. Pull the service topology from Kiali or equivalent.

**From application traces** (Splunk/Datadog APM): look up the trace IDs for failing requests and expand the trace to see the full call chain:

```splunk
index=app_traces service="origin-service" status="error" earliest=-15m
| stats count by trace_id, downstream_service
| sort -count
```

This tells you which downstream services are involved in failed traces — that is your live dependency graph for the failing path.

### Step 3: Check Correlated Error Rate Spikes

Once you have the dependency graph, check whether all upstream callers of the origin are spiking simultaneously. If yes, the cascade is active and the origin is confirmed. If only one caller is spiking, the origin may actually be in that caller.

```splunk
index=app_errors earliest=-30m
| timechart span=1m count by service
```

Look for the moment where multiple services' error rates crossed their threshold within the same 1-2 minute window. That window is when the cascade crossed the isolation boundary (or failed to).

### Step 4: Identify Cascade Direction

- **Downstream cascade**: the origin service is failing because something *it* depends on failed. Fix the dependency.
- **Upstream cascade**: the origin is failing and spreading to its callers. Isolate the origin from callers.

In practice, you will often have both simultaneously — the origin is failing because of its dependency (downstream), and it is propagating to its callers (upstream). Fix downstream first to stop the origin. Trip circuit breakers upstream to stop propagation while you fix downstream.

### Step 5: Calculate the Cascade Coefficient (CC)

The **Cascade Coefficient** quantifies the expected spread risk for each service in the graph. It determines where to focus isolation actions.

```
CC = Σ [ P(cascade_i) × tier_weight_i ]

Where:
  P(cascade) = probability that a failure at node i propagates to the next tier
    = 0.9  if no circuit breaker exists on this edge
    = 0.5  if circuit breaker exists but is partial (no fallback, or misconfigured)
    = 0.0  if circuit breaker is fully implemented with fallback

  tier_weight = service tier weight of the downstream service (from SRECapstone)
    P0 = 1.0  (auth, payment, checkout — customer-facing, revenue-critical)
    P1 = 0.8  (inventory, pricing, search — important but non-checkout-path)
    P2 = 0.5  (recommendations, notifications — supporting functionality)
    P3 = 0.3  (reviews, analytics, reporting — peripheral)
```

**Example**: A payment service (P0) has no circuit breaker on its call to the fraud-check service (P1), which in turn calls a rules engine (P3) with a partial circuit breaker.

```
CC = (0.9 × 1.0) + (0.5 × 0.8) + (0.0 × 0.3)
CC = 0.90 + 0.40 + 0.00
CC = 1.3
```

CC of 1.3 is acceptable risk. For context: the Auth service in the SRE reference system has CC_max = 10.8 — it calls 12 downstream services including Payment, Cart, and Order Mgmt (all P0, tw=1.0) with no circuit breakers, making each dependency a 0.9 × 1.0 = 0.90 contribution.

**CC Thresholds**:
- CC < 2.0: Acceptable risk. Monitor during incidents.
- CC 2.0–5.0: Elevated risk. Requires monitoring and standby isolation actions.
- CC > 5.0: Red-line. Requires immediate circuit breaker investment. Do not deploy to this service without mitigation plan.
- CC > 10.0: Architectural emergency. Any single failure can propagate system-wide. Blocks new feature deployments.

### Step 6: Prioritize Isolation Actions

Trip circuit breakers in CC-descending order. Highest CC contributor first. The goal is to stop cascade propagation before you focus on root cause.

Isolation actions (covered in detail in Part 6):
- Trip circuit breaker at the edge with highest CC
- Apply rate limiting to protect the origin from retry amplification
- Shift traffic away from the failing region if CC > 5.0 on multiple nodes

## Part 4: Blast Radius Assessment

Blast radius is not a feeling. It is a calculated index that tells you the proportion of customer impact and how fast it is growing.

### Blast Radius Index (BRI)

```
BRI = (affected_users / total_users) × journey_criticality × tier_weight

Where:
  affected_users    = unique users experiencing errors or degradation
  total_users       = total active users in the same window
  journey_criticality: 1.0 (revenue-critical: checkout, auth, payment)
                       0.7 (important: search, account)
                       0.5 (supporting: recommendations, notifications)
                       0.3 (peripheral: reviews, analytics)
  tier_weight (P-tier of the failing service):
                       P0 = 1.0 | P1 = 0.8 | P2 = 0.5 | P3 = 0.3
```

**Example**: 8,000 of 50,000 active users cannot complete checkout. Checkout is a core journey (1.0). Payment service is P0 (tier_weight = 1.0).

```
BRI = (8,000 / 50,000) × 1.0 × 1.0
BRI = 0.16 × 1.0 × 1.0
BRI = 0.16
```

BRI = 0.16 → SEV3 by BRI alone (threshold: 0.10–0.30). Apply the multi-dimension rule: a checkout outage for 16% of users also generates significant revenue exposure. Revenue impact at $1,597/min (example system) clears the SEV1 threshold ($10K/min is breached within 7 minutes). Classify by the highest dimension — **declare SEV1.**

Track BRI every 5 minutes during an active cascade — rising BRI means blast radius is expanding and isolation is not yet effective. Stable or falling BRI means containment actions are working.

### Dependency Gravity (DG)

DG is your **pre-incident risk score** for a service — computed during architecture review, not during an incident. It answers: "If this service fails, how much of the system does it take with it?"

```
DG = Σ [ tier_weight_direct × 1.0 ] + Σ [ tier_weight_transitive × 0.5 ]

Where:
  direct = services that directly call this service
  transitive = services that call those callers (two hops away)
  tier_weight = same scale as above
```

High DG services are your blast radius multipliers. They should have the most robust isolation mechanisms in your architecture — circuit breakers, bulkheads, and explicit degradation paths.

### Failure Locality Index (FLI)

FLI measures your containment effectiveness in real time:

```
FLI = contained_failures / total_failures

Where:
  contained_failures = failures that did not propagate beyond the origin service
  total_failures = all service failures observed in the incident window
```

- FLI = 1.0: perfect containment, all failures isolated at origin
- FLI = 0.5: half of failures propagated — cascade is partially active
- FLI < 0.25: cascade is dominating — isolation mechanisms are failing

Target: FLI > 0.90 at all times. During an active cascade, monitor FLI every 5 minutes. Rising FLI means your circuit breaker trips are working. Falling FLI means the cascade is outpacing your isolation actions.

### Building the Failure Dependency Tree in Real Time

In Kubernetes with a service mesh (Istio/Linkerd), you have live topology data. Use Kiali's graph view filtered to the incident namespace, set time window to the last 15 minutes, and enable error rate overlay.

In Splunk without a service mesh:

```splunk
index=app_traces earliest=-15m status="error"
| eval hop=coalesce(downstream_service, "terminal")
| stats count as error_count by service, hop
| sort -error_count
| where error_count > 50
```

This gives you the call pairs with the highest error volume — the active edges in your failure tree. Sort by error count to find the highest-volume propagation paths. Those are the edges to circuit-break first.

## Part 5: Damage Assessment Framework

### Four Damage Dimensions

During a cascade, damage accumulates across four independent dimensions. Each requires a different remediation track.

**Technical Damage**
- Services down (measured by health check failures, pod restarts, error rate)
- Queue backlog (RabbitMQ message depth, Kafka consumer lag)
- Resource exhaustion (CPU, memory, connection pools, file descriptors)
- Data inconsistency (transactions in flight when origin failed — partial writes)

**Customer Damage**
- Transactions failed (revenue-impacting if checkout, payment, fulfillment)
- Sessions dropped (authenticated users logged out)
- Data unavailable (read operations failing — customers cannot see order history, balance, etc.)
- Feature degradation (features silently returning defaults instead of real data)

**Data Damage**
- Records corrupted (partial writes during failure window — referential integrity broken)
- Transactions incomplete (payment charged but order not created, or vice versa)
- Audit log gaps (if the audit service was part of the cascade)
- Replay required (events dropped from queues that were not persisted)

**Financial Damage**
- Direct revenue lost (failed transactions not recovered)
- SLA credits owed (per contractual terms — know your SLA thresholds before incidents)
- Regulatory notification triggers (PCI, HIPAA, GDPR have mandatory notification timelines — know when you cross them)
- Remediation cost (engineering hours, customer service escalations, refunds)

### The Damage Timeline

Damage accumulation is not linear. It follows an exponential curve after the first 30 minutes.

```
0-5 min:   Damage localized. Isolation is cheapest here. Most customers haven't noticed.
5-15 min:  Blast radius expanding. Error rate rising. First customer support contacts arriving.
15-30 min: Cascade fully active if not isolated. SLA clock running. Retry storms amplifying.
30-60 min: Regulatory clock may have started. Downstream system state corruption accumulating.
60+ min:   Customer churn risk, data recovery required, incident report obligations triggered.
```

This is why IC posture matters at the 5-minute mark. The instinct to diagnose before acting is expensive. Mitigation first. Diagnosis in parallel. See ch06 for the mitigation-first principle.

## Part 6: Isolation Actions

### Circuit Breaker Trip

**What**: Stop the failing call. Return a fallback response (cached data, default, or explicit error) instead of waiting for the downstream service.

**When**: Any edge in your dependency graph where CC > 2.0 and the downstream service is in error or high-latency state.

**How**: In Resilience4j, circuit breakers can be tripped programmatically via actuator endpoints or configured for automatic trip on failure rate threshold:

```yaml
resilience4j.circuitbreaker:
  instances:
    fraud-check:
      slidingWindowSize: 10
      failureRateThreshold: 50        # trip at 50% failure rate
      waitDurationInOpenState: 30s    # stay open for 30 seconds
      permittedNumberOfCallsInHalfOpenState: 3
```

**Effect**: Immediate. Callers of the circuit-broken service stop waiting. Thread pools release. BRI stabilizes. FLI rises.

### Traffic Shift

**What**: Redirect traffic away from the failing region, cluster, or service instance.

**When**: Geographic or zonal failure. One availability zone is degraded; others are healthy.

**How**: In Kubernetes, adjust the weighted routing in the Istio VirtualService or at the ingress level. In AWS, use Route 53 weighted routing or Application Load Balancer target group weights.

**Caution**: Validate that the destination zone has capacity to absorb shifted traffic before shifting. A traffic shift onto an under-provisioned zone creates a new thundering herd.

### Rate Limiting

**What**: Cap the request rate accepted by the origin service to prevent retry storm amplification from overwhelming recovery.

**When**: Origin service is recovering; you know it will succeed if request rate is controlled. Retry storm is detected (error rate from clients is higher than baseline request rate suggests).

**How**: Nginx or Envoy rate limiting at the ingress. Application-level rate limiting via token bucket (Guava RateLimiter, Resilience4j RateLimiter).

**Effect**: Slows recovery slightly (some requests are shed), but prevents re-failure. The recovering service stabilizes rather than being overwhelmed by retries.

### Queue Draining vs. Queue Pausing

These are not interchangeable decisions.

**Pause the queue** when:
- The consumer is the failing service (draining would overwhelm the recovering consumer)
- Message order matters and you cannot afford to process out-of-sequence
- You do not know the state of the downstream and processing now could cause data corruption

**Drain the queue** when:
- The consumer has recovered and the backlog is bounded
- Messages are idempotent (safe to retry)
- You will drain at a controlled rate (not all at once — use consumer prefetch limits)

**Discard (dead-letter)** when:
- Messages have exceeded their TTL
- Processing stale messages would cause incorrect outcomes (price calculations with expired data, time-sensitive operations)
- The backlog is so large that draining would take longer than the business impact warrants

### Feature Flag Disables

**What**: Turn off a feature that calls a failing downstream service. All users see a "feature unavailable" state rather than an error.

**When**: The feature is non-critical and the failing downstream is unique to that feature (no other features call it).

**How**: LaunchDarkly, Split.io, or your internal flag system. This is a 30-second mitigation — much faster than a code rollback.

**Caution**: Confirm the flag disable does not break upstream state expectations. A feature that normally writes to a database — if disabled — may leave records in an inconsistent state if the disable happens mid-transaction.

### Graceful Degradation

**What**: Serve a cached or default response instead of calling the failing service. Customer sees degraded data (e.g., yesterday's recommendation list, cached account balance), not an error.

**Better than**: Serving an error page. An error stops the customer journey. Degraded data lets the customer continue (possibly with reduced trust).

**Implementation**: Your fallback must be pre-built. You cannot gracefully degrade at incident time if there is no degraded mode implemented. This is a pre-incident architecture decision.

## Part 7: Post-Cascade Analysis

### Reconstructing the Cascade Timeline from Logs

The post-incident reconstruction requires precise timestamps across services. Splunk is well-suited for this.

**Step 1**: Identify the error onset per service:

```splunk
index=app_errors earliest=-2h latest=now
| where status_code >= 500
| bin _time span=1m
| stats count as error_rate by _time, service
| where error_rate > 10
| sort _time, service
```

**Step 2**: Identify deployment or config events in the same window:

```splunk
index=deploy_events OR index=config_changes earliest=-2h latest=now
| table _time, service, change_type, change_author, environment
| sort _time
```

**Step 3**: Overlay latency with error rate to detect avalanche pattern:

```splunk
index=app_traces service="origin-service" earliest=-2h latest=now
| bin _time span=1m
| stats avg(duration_ms) as avg_latency, count as req_count,
        sum(eval(if(status="error",1,0))) as errors by _time
| eval error_rate=errors/req_count*100
| table _time, avg_latency, req_count, error_rate
```

Compare the `avg_latency` column against `error_rate`. In an avalanche pattern, latency rises several minutes before error rate spikes.

### Dependency Surface Area (DSA)

DSA is your structural risk metric for long-term blast radius reduction planning. Compute post-incident to understand architectural exposure.

```
DSA = Σ[ 1.0 × tier_weight_direct ] + Σ[ 0.5 × tier_weight_transitive ]

Where:
  direct = services the analyzed service directly depends on
  transitive = services those dependencies depend on
```

Higher DSA means broader exposure to external failures. Services with high DSA should be prioritized for circuit breaker implementation and dependency reduction in the next sprint.

### Blast Radius Reduction Roadmap

Post-incident, use these metrics to prioritize architectural improvements:

| Metric | Current | Target | Intervention |
|--------|---------|--------|-------------|
| FLI | Track actual | > 0.90 | Circuit breakers on all Tier 1 edges |
| CC_max | Track actual | < 5.0 | Bulkhead pattern for high-CC services |
| DG (highest) | Track actual | Reduce by 30% | Eliminate transitive synchronous dependencies |
| BRI at T+15min | Track actual | < 0.10 | Traffic shift automation, canary guardrails |

These targets should be tracked in your quarterly SRE roadmap. The CC and FLI calculations above give you the inputs. The interventions — circuit breakers, bulkheads, graceful degradation — are the outputs.

The single most impactful change in most architectures: **implement circuit breakers with real fallbacks on every Tier 1 to Tier 2 synchronous call**. Not the skeleton implementation with no fallback logic. Actual fallbacks that serve degraded but functional responses.

## Summary: Cascade Decomposition Checklist

Use this during an active incident:

- [ ] First error timestamp identified per service — origin found
- [ ] Live dependency graph built from traces or service mesh
- [ ] Correlated error spikes confirmed across callers
- [ ] Cascade direction identified (upstream / downstream / both)
- [ ] CC calculated for top 5 edges — prioritized by score
- [ ] FLI being tracked every 5 minutes
- [ ] BRI being tracked every 5 minutes
- [ ] Circuit breakers tripped on CC > 2.0 edges
- [ ] Rate limiting applied if retry storm detected
- [ ] Queue status assessed — pause or drain decision made

See [ch06-recovery-planning.md](ch06-recovery-planning.md) for what to do once the cascade is contained.
