> **Audience**: Principal SRE / SRE Manager — practitioner-level reference. Assumes deep experience with Kubernetes/EKS, Kafka/RabbitMQ, Splunk, distributed systems, and production incident command. This chapter is an operational reference, not theory.

## Part 1: Fix Taxonomy Overview

### The Spectrum from Mitigation to Prevention

Fixes are not binary. The instinct during a production incident is to think in terms of "broken vs. fixed" — but that framing obscures the most important operational question: **what level of fix are you actually implementing?**

Every fix exists somewhere on this spectrum:

```
MITIGATION ──────────────────────────────────────────── PREVENTION
    │                    │                    │                │
Stops the          Corrects the          Eliminates       Makes the failure
bleeding now       immediate             root cause       mode impossible or
(does not fix      cause (service                         detectable before
root cause)        restored)                              impact
    │                    │                    │                │
Example:         Example:             Example:         Example:
Rate limit       Roll back bad        Add circuit      Add pre-deploy FMEA
incoming         deployment           breaker          for dependency config
traffic                               on dependency
```

Understanding where your fix sits on this spectrum determines:
- **Time to implement**: mitigation in minutes; prevention in sprints
- **Risk profile**: mitigation is low-risk; infrastructure fixes can be high-risk
- **Durability**: mitigation degrades (workarounds rot); prevention is durable
- **Authorization required**: data fixes require IC authorization; config fixes require logging

**During an active incident**, you implement mitigations first to stop customer impact, then work toward resolution. Prevention happens post-incident, in the cold path.

**The trap**: Teams under pressure treat mitigations as resolutions and never implement the root cause fix. The incident closes, the postmortem action items age in Jira, and the incident recurs in six months. The fix taxonomy is a forcing function against this.

---

### Six Fix Types

The taxonomy organizes fixes into six types based on what layer of the system they change. Each type has distinct implementation mechanics, time-to-effect, validation approach, risk profile, and authorization requirements.

| Fix Type | Layer Changed | Typical Time-to-Effect | Risk Profile | IC Auth Required |
|---|---|---|---|---|
| Code Fix | Application logic | Minutes (rollback) to hours (fix-forward) | Medium | For hotfix during incident |
| Configuration Fix | Operational parameters | Seconds to minutes | Low-Medium | Logging required |
| Traffic Fix | Request routing | Seconds to minutes | Medium | For major traffic shifts |
| Data Fix | Data state | Minutes to hours | HIGH | Always |
| Infrastructure Fix | Compute/network/storage | Minutes to hours | HIGH | Always |
| Dependency Fix | Third-party services | Variable (vendor dependent) | Medium | For fallback activation |

## Fix Type 1: Code Fix

### What It Is

A code fix changes application logic — the actual behavior of a service. Code fixes are the most familiar fix type, but during an active incident they are also among the riskiest because you are deploying new code to a system that is already behaving abnormally.

**Rule**: A code fix is only appropriate during an active incident when (a) the root cause is confirmed to be in the new deployment, and (b) rollback is not faster or not possible. In all other cases, stabilize first, then fix.

---

### Code Fix Subtypes

**Rollback to Known-Good Artifact**

The fastest code fix and the lowest risk. When a deployment is the confirmed direct cause of the incident, rolling back to the previous artifact restores the previous behavior without requiring any new code.

Pre-conditions for rollback:
- The previous artifact is available in the artifact repository (ECR, Artifactory, Nexus)
- The previous artifact is compatible with the current database schema (schema migrations may block rollback)
- The deployment was recent enough that rolling back does not create configuration drift

Rollback process:
```bash
# Kubernetes: roll back to previous deployment revision
kubectl rollout undo deployment/payment-service -n production

# Verify rollout status
kubectl rollout status deployment/payment-service -n production

# Confirm pod image version
kubectl get pods -n production -l app=payment-service -o jsonpath='{.items[*].spec.containers[0].image}'
```

Time-to-effect: 2-5 minutes for a typical Kubernetes rollout. Pods are replaced one at a time (rolling update), so partial rollback begins within seconds.

**Schema migration caveat**: If the new deployment ran a database migration, rolling back the code does not roll back the schema. Assess whether the old code is compatible with the new schema before rolling back. If not, rollback is not safe — you must fix-forward.

---

**Fix-Forward with Canary**

When rollback is not possible (schema migration ran, or rollback would cause data loss), or when the fix is simple and well-understood, fix-forward with canary is the correct approach.

**Canary deployment process**:
1. Develop and review the fix (abbreviated review — correctness check only, not full review)
2. Deploy to staging and run smoke tests (abbreviated — focus on the failure path)
3. Deploy to production at 1% of traffic (canary pod or weighted routing)
4. Monitor: error rate, latency P95/P99, business metrics for 5-10 minutes
5. If metrics are stable: ramp to 10%, then 50%, then 100%
6. If metrics degrade at any stage: roll back the canary

**Time-to-deploy benchmarks** (what a mature organization should achieve):
- Simple one-line fix: 20-30 minutes from identify to canary in production
- Complex logic fix: 45-90 minutes from identify to canary in production
- If you cannot canary-deploy a fix in under 2 hours, your deployment pipeline is a reliability risk

**Canary validation metrics** (what to watch during ramp):
- Error rate (the specific error type from the incident)
- Latency P95 and P99 (regression check)
- Downstream service error rates (the fix may shift load)
- Business transaction completion rate (ultimate correctness check)

---

**Feature Flag as Code-Path Switch**

A feature flag is not a code fix — it is a code-path switch. But it achieves the same outcome (disabling bad behavior) faster than any deployment, and it is fully reversible in milliseconds.

When the bad code path is behind a feature flag:
1. Toggle the flag to disable the bad path (LaunchDarkly, Unleash, or Parameter Store boolean)
2. Service reads the flag on the next request cycle (typically <100ms propagation)
3. Bad behavior stops without any deployment

When to use feature flags as incident mitigation:
- New feature is causing failures (flag it off)
- New algorithm is producing wrong results (flag to old algorithm)
- Third-party integration is failing (flag to bypass integration)

Feature flags are not a substitute for fixing the underlying code. They are a mitigation that buys time. The flag must be resolved (either the code is fixed and the flag is removed, or the feature is permanently disabled and the dead code is cleaned up) within the next sprint.

---

**Guard Rail: Hotfix Authorization**

No hotfix deployment should occur during an active incident without explicit Incident Commander authorization. The IC must explicitly open the "fix-forward track" because:

- A new deployment during an active incident adds a second variable to an already unstable system
- If the hotfix itself has a bug, it compounds the incident
- The IC needs to track that a deployment is in progress when making other decisions (traffic shifts, rollbacks)

**Log entry format** when IC authorizes fix-forward:
```
[14:47 UTC] IC Authorization: Fix-forward track open. Deploy payment-service v2.3.1-hotfix.
Authorized by: [IC name]. Expected effect: eliminate null pointer in token validation.
Canary at 1% traffic. Ramp to 100% only after IC confirmation.
```

## Fix Type 2: Configuration Fix

### What It Is

A configuration fix changes operational parameters without changing application code. Configuration fixes are fast, often reversible, and do not require a deployment pipeline — which makes them attractive during incidents. They are also harder to validate than code changes because they bypass the normal review pipeline.

---

### Configuration Fix Subtypes

**Kubernetes ConfigMap / Environment Variable Changes**

ConfigMap changes in Kubernetes take effect differently depending on how the application reads configuration:
- If the application reads ConfigMap values at startup: pod restart required
- If the application watches ConfigMap for changes (using projected volumes or a config-watching library): change takes effect within seconds, no restart needed

```bash
# Edit ConfigMap directly (use with caution — bypasses GitOps)
kubectl edit configmap payment-service-config -n production

# Alternative: apply from file (preferred — maintains audit trail)
kubectl apply -f payment-service-config-patched.yaml

# Verify ConfigMap change
kubectl get configmap payment-service-config -n production -o yaml

# If restart is required
kubectl rollout restart deployment/payment-service -n production
```

**Decision log entry required** for every ConfigMap change during an incident:
```
[14:52 UTC] Config Change: payment-service ConfigMap.
Changed: connection_pool_max_size from 50 to 100.
Changed by: [engineer name].
Expected effect: relieve connection pool exhaustion under current load.
Pre-change value: 50. Post-change value: 100.
Rollback: set back to 50 if error rate does not improve within 5 minutes.
```

---

**Feature Flag Toggle (LaunchDarkly / Unleash)**

Feature flag changes are the fastest and most controlled configuration fix. They have millisecond propagation, instant rollback, and a complete audit trail built in.

Key characteristics:
- **Propagation time**: <500ms for LaunchDarkly, <1 second for Unleash (polling interval dependent)
- **Rollback**: toggle back to previous state; effect reverses within propagation time
- **Scope**: can target specific user segments, environments, or percentage of traffic
- **Audit trail**: flag change log includes actor, timestamp, previous value, and new value

Use feature flag toggles as the first-choice mitigation when the failure is in a flagged code path. They are safer than any other fix type because they are fully reversible with no deployment pipeline involved.

---

**Parameter Store / Secrets Manager Updates**

AWS Parameter Store and Secrets Manager updates require the application to re-read the value to take effect. Unlike ConfigMaps with projected volumes, there is no push mechanism — the application must poll.

Time-to-effect = application polling interval (commonly 30 seconds to 5 minutes). Check your application's polling configuration before relying on this for an active incident mitigation.

For secrets rotation: Secrets Manager has a dedicated rotation mechanism. Do not manually update secrets during an active incident unless you have confirmed that the application handles dynamic secret reload without restart.

---

**DNS TTL Manipulation**

DNS changes are a common traffic-shifting mechanism, but DNS TTL is the most commonly overlooked constraint. If the current TTL is 300 seconds (5 minutes), a DNS change will take up to 5 minutes to propagate to all clients — and clients that cached the old record will continue hitting the old target until their TTL expires.

**Procedure for planned DNS traffic shift**:
1. Before the change window, lower TTL to 60 seconds. Wait 24 hours for caches to expire at the old TTL.
2. Execute the DNS change.
3. Effect propagates in <60 seconds.
4. Restore original TTL after shift is confirmed stable.

**During an active incident**: if you need a DNS change and the current TTL is high, you cannot fast-path this. Accept the propagation delay, or use a different routing mechanism (load balancer weighted routing, service mesh policy) that has no TTL constraint.

---

### Configuration Fix Guard Rail

Configuration changes are fast but they bypass the code review pipeline. Every configuration change during or around an incident must be:

1. **Logged** with pre-value, post-value, expected effect, and rollback trigger
2. **Tracked** as a decision in the incident timeline
3. **Validated** by observing the specific metric that the change was intended to improve, within a defined window (typically 3-5 minutes)
4. **Reverted if no improvement** within the validation window — do not leave a config change in place if it did not produce the expected effect

## Fix Type 3: Traffic Fix

### What It Is

A traffic fix changes where requests are routed without changing application code or configuration. Traffic fixes are used to shed load from a degraded component, activate standby capacity, or control which users are exposed to a degraded service.

---

### Traffic Fix Subtypes

**Weighted Routing at Load Balancer**

Most cloud load balancers (ALB, NLB, Nginx, HAProxy) support weighted routing — split traffic in a defined ratio across multiple targets.

Use cases:
- **Canary validation**: route 1% of traffic to new version; 99% to stable version
- **Gradual failover**: shift traffic from degraded region to healthy region in 10% increments
- **Load shedding**: route 30% of traffic to overflow capacity while primary recovers

ALB weighted target group example:
```
Target Group A (primary, degraded): 30% weight
Target Group B (overflow, healthy): 70% weight
```

Time-to-effect: seconds (no DNS TTL). Weighted routing changes take effect on new connections immediately.

**Risk**: Weighted routing exposes the receiving target group to shifted load. If the receiving group is undersized, shifting traffic to it can cause a secondary incident. Always validate receiving capacity before shifting.

---

**Service Mesh Traffic Policies (Istio / Envoy)**

A service mesh gives you per-service, per-route traffic control without touching load balancer configuration. This is the most granular traffic control tool available in a Kubernetes environment.

**Circuit breaking** (Istio DestinationRule):
```yaml
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: payment-service-circuit-breaker
spec:
  host: payment-service
  trafficPolicy:
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 10s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
```

This ejects any upstream pod that returns 5 consecutive 5xx errors within 10 seconds, for a 30-second cooling period. During the incident, this prevents traffic from being sent to degraded pods, forcing it to healthy pods.

**Timeout override** (incident mitigation):
```yaml
trafficPolicy:
  connectionPool:
    http:
      http1MaxPendingRequests: 100
      http2MaxRequests: 1000
  outlierDetection:
    ...
  # Reduce timeout during incident to shed slow requests faster
  # (normal timeout: 30s; incident timeout: 5s)
```

---

**Rate Limiting and Throttling**

Rate limiting protects the origin from retry storm amplification — one of the most common ways an incident escalates from bad to catastrophic. When a service degrades, clients retry. If retry logic lacks exponential backoff and jitter, each retry wave amplifies the load on the degrading service.

Rate limiting intercepts this at the edge:
- **Global rate limit**: cap total RPS to the origin at the load balancer or API gateway
- **Per-client rate limit**: cap RPS per client/IP to prevent any single client from monopolizing capacity
- **Per-endpoint rate limit**: protect expensive endpoints while leaving lightweight endpoints unrestricted

During an active incident, applying a rate limit at the edge can stop the retry storm amplification and allow the origin to recover, even before the root cause is addressed.

---

**CDN Cache Invalidation**

When a bug deploys incorrect data to users, CDN cache invalidation removes the stale responses so that the next request fetches a corrected response from origin.

Cache invalidation characteristics:
- **Speed**: CloudFront invalidation propagates globally in 10-30 seconds for individual paths
- **Scope**: Invalidate specific paths (precise) or wildcard patterns (broad)
- **Cost**: CloudFront charges for invalidation requests beyond 1,000/month; bulk invalidations are expensive

Invalidation use cases during incidents:
- Incorrect page rendered from CDN cache after a bad deploy
- Stale API response cached at edge while origin is returning corrected data
- Security-sensitive data cached longer than intended TTL

---

### Traffic Fix Guard Rail

Always validate the receiving system's capacity before shifting traffic to it. A warm standby that has not handled traffic in months may not actually be warm — its connection pools may be cold, its JVM heap may not be warmed, its local caches may be empty.

**Pre-shift validation checklist**:
- [ ] Receiving system is healthy (health check passing)
- [ ] Receiving system has confirmed capacity for the expected traffic volume
- [ ] Receiving system's dependencies are healthy
- [ ] Traffic shift procedure has been tested (even a tabletop test counts)
- [ ] Rollback procedure is documented (how to shift traffic back, and how fast)

## Fix Type 4: Data Fix

### What It Is

A data fix corrects the state of data affected by the incident — messages that failed to process, records that were corrupted, transactions that completed only partially, or data that was lost and needs to be recovered. Data fixes are the highest-risk fix type because the data is the source of truth. A wrong data fix can compound the corruption, create new inconsistencies, or destroy the audit trail.

**Rule**: No data fix executes without a dry run, IC authorization, database engineer co-approval, and a documented rollback plan.

---

### Data Fix Subtypes

**Idempotent Replay**

The safest data fix. Re-run failed transactions through the processing pipeline. The key pre-condition is that the processing pipeline is idempotent — processing the same message twice produces the same result as processing it once (no double-charges, no duplicate records, no double-sends).

Idempotency is implemented via idempotency keys: a unique identifier per transaction that the processing system checks before executing. If the key has already been processed, the system returns the cached result and takes no action.

When idempotent replay is available:
1. Identify the set of failed transactions (by timestamp range, error type, or explicit list)
2. Re-submit them to the processing pipeline
3. The pipeline handles deduplication automatically
4. Verify completion counts match expected counts

This is why idempotency keys are one of the highest-ROI engineering investments for any transaction processing system. Without them, replay is dangerous. With them, replay is the fastest and safest data recovery operation.

---

**Dead Letter Queue (DLQ) Reprocessing**

Messages that failed processing during the incident are typically routed to a DLQ (either in Kafka, RabbitMQ, SQS, or a custom implementation). DLQ reprocessing replays these messages through the corrected processing pipeline after the root cause is fixed.

**Kafka DLQ reprocessing approach**:

Option A — Consumer group offset reset (rewind the original topic):
```bash
# Stop the consumer group
kafka-consumer-groups.sh --bootstrap-server kafka:9092 \
  --group notification-dispatch-consumer --reset-offsets \
  --to-datetime 2026-06-30T14:00:00.000 \
  --topic notification-dispatch \
  --execute

# Restart the consumer to replay from the reset offset
kubectl rollout restart deployment/notification-dispatch-consumer -n production
```

This approach replays all messages from the reset timestamp, not just failed ones. Use only when:
- Processing is idempotent (already-processed messages are safely re-processed)
- You know that all messages from the timestamp forward need reprocessing

Option B — DLQ topic replay (targeted):
```bash
# Run a DLQ replay job that reads from the DLQ topic and re-publishes to the main topic
# This preserves the original message content and only re-processes messages that failed
kubectl apply -f jobs/dlq-replay-job.yaml -n production
```

**RabbitMQ DLQ reprocessing**:
```python
# Connect to RabbitMQ and re-queue messages from the dead letter exchange
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters('rabbitmq'))
channel = connection.channel()

# Move messages from DLQ back to main queue
# Always do this with a count limit — do not re-queue unbounded
method_frame, header_frame, body = channel.basic_get('payments.dlq')
while method_frame and messages_requeued < max_requeue_count:
    channel.basic_publish(
        exchange='payments',
        routing_key='payment.process',
        body=body,
        properties=header_frame
    )
    channel.basic_ack(method_frame.delivery_tag)
    messages_requeued += 1
    method_frame, header_frame, body = channel.basic_get('payments.dlq')
```

---

**Database Point-in-Time Recovery (PITR)**

PITR restores the database to its state at a specific timestamp before the incident. This is the nuclear option for data fixes: it is complete, it is reliable, and it destroys everything that happened after the recovery point.

Use PITR when:
- Data corruption is widespread and cannot be targeted by row-level corrections
- The corruption was introduced by a schema migration that cannot be reversed
- The data loss is so severe that starting from a clean pre-incident state is the only viable path

PITR trade-offs:

| Factor | Impact |
|---|---|
| Data loss window | All data written after the recovery point timestamp is lost |
| Recovery time | Minutes to hours depending on DB size and recovery mechanism |
| Dependency impact | All services that wrote data after recovery point need to reconcile |
| Irreversibility | PITR cannot be undone — the recovery point overwrites current state |

**Before executing PITR**:
1. Confirm the recovery point timestamp — the last known-good state before the corruption began
2. Calculate the data loss window — what transactions will be lost?
3. Get IC authorization (and typically engineering leadership authorization for large windows)
4. Communicate to affected teams — every service that has a dependency on the database needs to know the recovery point so they can reconcile their own state
5. Document: recovery point timestamp, person who authorized, data loss scope, teams notified

---

**Manual Correction Scripts**

When the corrupted data is specific and bounded, targeted row-level corrections are safer and faster than PITR. The script approach allows surgical precision — you fix exactly the affected rows and leave all other data intact.

**Every manual correction script must follow this template**:

```sql
-- INCIDENT: [ticket number and description]
-- AUTHORED: [engineer name], [date]
-- REVIEWED: [DB engineer name]
-- IC AUTHORIZATION: [IC name], [timestamp]
-- SCOPE: [how many rows affected, which tables, which conditions]
-- DRY RUN: Execute SELECT version first; verify row count matches expected before UPDATE

-- STEP 1: DRY RUN — run this SELECT and verify row count
SELECT COUNT(*), order_status, payment_status
FROM orders
WHERE created_at BETWEEN '2026-06-30 14:07:00' AND '2026-06-30 14:34:00'
  AND payment_status = 'FAILED'
  AND payment_processor_error = 'TIMEOUT'
  AND order_status = 'PENDING'
GROUP BY order_status, payment_status;
-- Expected: approximately 1,240 rows based on incident error count

-- STEP 2: Execute UPDATE only after dry run count is confirmed
-- BEGIN TRANSACTION;

UPDATE orders
SET 
  payment_status = 'RETRY_ELIGIBLE',
  updated_at = NOW(),
  updated_by = 'incident-fix-TICK-4471'
WHERE created_at BETWEEN '2026-06-30 14:07:00' AND '2026-06-30 14:34:00'
  AND payment_status = 'FAILED'
  AND payment_processor_error = 'TIMEOUT'
  AND order_status = 'PENDING';

-- STEP 3: Verify row count matches expected before committing
SELECT COUNT(*) FROM orders WHERE updated_by = 'incident-fix-TICK-4471';
-- Should match dry run count

-- COMMIT; -- Uncomment only after row count verified
-- ROLLBACK; -- Use this if count is wrong
```

**Guard rails**:
- All data fix scripts require a `WHERE` clause with explicit scope boundaries
- Row count must be verified at dry run before any `UPDATE` or `DELETE` executes
- Always use a transaction so you can roll back if the row count is wrong
- Tag every updated row with a fix identifier (e.g., `updated_by = 'incident-fix-TICK-4471'`) for audit trail and rollback identification

---

**Data Reconciliation**

After any data fix, replay operation, or database restore, the data state across systems needs to be verified for consistency. Reconciliation compares the source of truth against derived systems and fills identified gaps.

**Reconciliation process**:
1. **Define the source of truth**: which system holds the authoritative record?
2. **Query both systems** for the affected time range and entity set
3. **Compare**: what exists in source-of-truth but not in derived system? What exists in derived system but not in source-of-truth?
4. **Classify discrepancies**: missing (need to replay), duplicate (need to deduplicate), corrupted (need to correct)
5. **Apply compensating actions**: re-send events, delete duplicates, apply corrections
6. **Verify**: re-run the comparison query; expect zero discrepancies

For financial systems, reconciliation is a ledger comparison: sum of all debits must equal sum of all credits for the affected transaction set. Any imbalance represents an orphaned or corrupted transaction.

---

### Data Fix Guard Rails Summary

| Gate | What to Check | Who Authorizes |
|---|---|---|
| Dry run | Row count matches expected scope | Executing engineer + DB engineer |
| Scope verification | No unintended rows in WHERE clause | DB engineer |
| Rollback plan | Explicitly documented before execution | IC |
| IC authorization | Explicit approval in decision log | IC |
| DB engineer co-approval | Schema knowledge verified | Senior DBA / DB engineer |
| Post-fix verification | Reconciliation confirms expected state | QA or on-call engineer |
| Audit trail | Every changed row tagged with fix identifier | Automated (in script) |

## Fix Type 5: Infrastructure Fix

### What It Is

An infrastructure fix changes the underlying compute, network, or storage layer. Infrastructure fixes have the widest blast radius of any fix type — a change to a node affects all workloads on that node; a network change can affect all traffic traversing that path.

**Rule**: Infrastructure fixes require explicit IC authorization. Always check what else runs on the target infrastructure before executing.

---

### Infrastructure Fix Subtypes

**Kubernetes Node Operations**

When a node is degraded (high CPU, memory pressure, disk full, kernel panic), drain and replace it:

```bash
# Step 1: Cordon the node (prevent new pods from scheduling)
kubectl cordon node-name

# Step 2: Drain the node (evict all pods gracefully)
kubectl drain node-name --ignore-daemonsets --delete-emptydir-data --grace-period=60

# Step 3: Check what's running on the node before draining
kubectl get pods --all-namespaces --field-selector spec.nodeName=node-name

# Step 4: Terminate the node (EC2 termination triggers auto-replacement in managed node group)
aws ec2 terminate-instances --instance-ids i-xxxxxxxxxxxxxxxxx

# Step 5: Verify new node joins and becomes Ready
kubectl get nodes --watch
```

**Before draining**: `kubectl get pods --all-namespaces --field-selector spec.nodeName=node-name` shows everything on the node. Verify that evicting these pods will not reduce the service below its minimum replica count. If the node has a critical singleton (e.g., a pod with no replica), draining it will cause an outage.

---

**Pod Restart and Deployment Rollout**

The least-invasive infrastructure intervention: restart pods to clear transient state (memory leak, stuck connection, corrupted in-memory cache).

```bash
# Restart all pods in a deployment (rolling restart — no downtime)
kubectl rollout restart deployment/payment-service -n production

# Watch rollout progress
kubectl rollout status deployment/payment-service -n production

# Restart a specific pod (immediate; new pod scheduled by Deployment controller)
kubectl delete pod payment-service-7d9f5b4c6-xk2m9 -n production
```

Rolling restart replaces pods one at a time, maintaining the configured replica count throughout. This is safe for stateless services. For stateful services, check whether the StatefulSet's `podManagementPolicy` is `OrderedReady` (sequential, slow) or `Parallel` (concurrent, faster but all pods restart simultaneously).

---

**Horizontal Scaling**

During a capacity incident, the fastest mitigation is horizontal scaling — add more pods to distribute the load.

```bash
# Scale immediately
kubectl scale deployment payment-service --replicas=20 -n production

# Verify pods are starting
kubectl get pods -n production -l app=payment-service --watch

# Check HPA (if HPA is configured, it may fight your manual scale)
kubectl get hpa payment-service-hpa -n production

# If HPA is fighting: temporarily patch the HPA max replicas
kubectl patch hpa payment-service-hpa -n production \
  -p '{"spec":{"maxReplicas":30}}'
```

**HPA conflict warning**: If a Horizontal Pod Autoscaler is configured, it will override manual scale-down decisions within its reconciliation loop. If you scale to 20 replicas but HPA has maxReplicas=10, HPA will scale you back down. During an incident, check HPA configuration and either patch the maxReplicas or disable HPA temporarily (with IC authorization).

---

**Storage Recovery**

Storage incidents: disk full, I/O saturation, volume mount failure.

```bash
# Check disk usage on nodes
kubectl debug node/node-name -it --image=busybox -- df -h

# Identify top disk consumers (run inside node debug pod)
du -sh /var/log/pods/* | sort -rh | head -20

# Force log rotation / cleanup on node
kubectl exec -n kube-system $(kubectl get pod -n kube-system -l app=fluentd -o name | head -1) -- logrotate -f /etc/logrotate.conf
```

For PersistentVolumeClaim issues:
```bash
# Check PVC status
kubectl get pvc -n production

# Describe PVC for events
kubectl describe pvc payment-data-pvc -n production

# For volume resize (if supported by storage class)
kubectl patch pvc payment-data-pvc -n production \
  -p '{"spec":{"resources":{"requests":{"storage":"100Gi"}}}}'
```

---

**Network Path Remediation**

Security group rule changes, VPC routing adjustments, and NACLs require elevated AWS permissions and careful scope validation.

```bash
# Add an ingress rule to a security group (example: allow traffic from new load balancer)
aws ec2 authorize-security-group-ingress \
  --group-id sg-xxxxxxxxxx \
  --protocol tcp \
  --port 8080 \
  --source-group sg-yyyyyyyyyy

# Verify the rule was applied
aws ec2 describe-security-groups --group-ids sg-xxxxxxxxxx \
  --query 'SecurityGroups[0].IpPermissions'
```

**Blast radius check for network changes**: A security group may be associated with multiple resources. Check all associations before modifying rules:
```bash
aws ec2 describe-network-interfaces \
  --filters Name=group-id,Values=sg-xxxxxxxxxx \
  --query 'NetworkInterfaces[*].[NetworkInterfaceId,Description,Status]'
```

---

### Infrastructure Fix Guard Rails

1. **Check what else runs on the target** before any node drain, security group change, or storage operation. Infrastructure changes have wider blast radius than application changes.
2. **IC authorization required** for any infrastructure change during an active incident. Log the authorization with timestamp, action, scope, and expected effect.
3. **Pre-change verification**: take a snapshot of current state (node list, pod count, network rules) before applying changes, so you can verify what changed and restore if needed.
4. **Staged application**: for multi-node operations, change one node at a time and verify before proceeding.

## Fix Type 6: Dependency / Third-Party Fix

### What It Is

A dependency fix addresses failures in third-party services — payment processors, SMS gateways, CDNs, DNS providers, identity providers, and cloud services that you do not control. The response to a third-party failure is fundamentally different from a self-inflicted failure: you cannot fix the dependency itself; you can only control your system's behavior in the presence of the dependency failure.

---

### Dependency Fix Subtypes

**Vendor Escalation**

When the dependency failure is the primary cause and your fallback options are limited, vendor escalation is the path to resolution. Know your escalation path before the incident.

**Escalation path components**:
- Status page URL (Stripe: status.stripe.com; AWS: health.aws.amazon.com)
- Support portal with P1 escalation path
- Emergency phone contact (enterprise contracts typically include this)
- Named TAM (Technical Account Manager) for cloud providers
- Slack/Teams channel for enterprise support (Stripe, Twilio, Datadog all offer this)

Document these in your runbooks. During an active incident is the wrong time to find the vendor's emergency contact number.

**SLA for response** (realistic expectations):
- AWS P1 (Business Support): 1 hour response
- AWS P1 (Enterprise Support): 15 minutes response
- Stripe (enterprise): 30 minutes
- Twilio (enterprise): 30-60 minutes

During the escalation wait, pursue fallback options in parallel.

---

**Fallback to Secondary Provider**

The gold standard for dependency resilience: two providers for every critical third-party service. When the primary fails, route to secondary.

Common dual-provider patterns:
- SMS delivery: Twilio primary, AWS SNS secondary
- Payment processing: Stripe primary, Braintree secondary
- Email: SendGrid primary, SES secondary
- CDN: CloudFront primary, Fastly secondary

**Failover trigger**: define the threshold for automatic or semi-automatic failover in your runbook:
```
If Stripe error rate exceeds 10% for >2 minutes:
1. IC authorizes SMS fallback activation
2. Engineer toggles payment_processor_secondary feature flag to TRUE
3. Traffic shifts to Braintree; monitor Braintree error rate
4. Stripe escalation ticket remains open
5. Return to Stripe when Stripe error rate returns to <1% for 5 minutes
```

---

**Synthetic Bypass (Feature Disable)**

When the dependency failure affects a non-critical feature path, disable the feature rather than expose users to errors. This is the application of the "graceful degradation" principle.

Examples:
- Recommendations service (ML-powered) depends on a failing ML inference endpoint → disable recommendations, show static fallback recommendations
- Address validation depends on a failing postal service API → disable real-time validation, accept all addresses, validate asynchronously later
- Fraud scoring depends on a failing risk API → apply conservative static rules, flag for manual review

**Implementation**: feature flag toggle or configuration change to bypass the dependency. The user experience degrades (they see a less sophisticated feature), but the transaction completes.

---

**Caching Expired Responses**

When a dependency is intermittently available, extending the cache TTL on the last-known-good response buys time:
- CDN edge cache: increase origin TTL header
- Application-level cache: extend TTL on last-successful API response
- DNS resolver cache: modern resolvers honor minimum TTL even when authoritative TTL expires

**Risk**: serving stale data. Only appropriate for data that changes infrequently (configuration, reference data, product catalog). Never appropriate for financial data, auth tokens, or real-time pricing.

---

**Circuit Breaker for External Dependency**

A circuit breaker is not a fire-and-forget configuration — it is a defense against one dependency's failure cascading into your service's failure. When the circuit is open, calls to the dependency return a pre-configured default (fallback response, cached response, or error with user-friendly message) without making a network call.

Circuit breaker states:
- **Closed** (normal): requests pass through to dependency
- **Open** (dependency degraded): requests return fallback without touching dependency; dependency is not called
- **Half-Open** (recovery probe): periodic probe request sent to dependency; if successful, circuit closes; if failed, circuit stays open

Resilience4j configuration example (Java / Kotlin):
```yaml
resilience4j:
  circuitbreaker:
    instances:
      stripe-client:
        slidingWindowSize: 10
        failureRateThreshold: 50
        waitDurationInOpenState: 30s
        permittedNumberOfCallsInHalfOpenState: 3
        automaticTransitionFromOpenToHalfOpenEnabled: true
```

This configuration opens the circuit when 50% of the last 10 calls fail, keeps it open for 30 seconds, then sends 3 probe requests to test recovery.

## Part 2: Reconciliation and Retroactive Recovery

### What Reconciliation Is

Reconciliation is the process of verifying that data state is consistent across systems after a recovery operation, and repairing any identified discrepancies. It is the final step of any data fix, and the only way to confirm that the fix was complete.

Reconciliation is not optional. Every recovery operation that touches data introduces the risk of partial completion, ordering errors, or missed scope. Reconciliation is the verification mechanism.

**When reconciliation is mandatory**:
- After any data fix (manual correction script, bulk update)
- After any replay operation (DLQ reprocessing, Kafka offset reset)
- After any database restore (PITR or snapshot restore)
- After any distributed transaction that failed mid-flight during the incident
- After any failover that involved dual writes or distributed state

---

### Idempotency as the Foundation of Reconcilability

The primary engineering investment that makes reconciliation safe and fast is **idempotency keys**.

An idempotency key is a unique identifier for a logical operation. When the processing system receives a message with an idempotency key it has already processed, it returns the cached result without re-executing the operation. This makes replay safe: re-sending 10,000 messages through an idempotent pipeline produces the same outcome as sending each message exactly once.

Without idempotency keys:
- Replay may double-charge customers
- Replay may create duplicate records
- Replay may send duplicate notifications
- Reconciliation becomes a manual comparison of every record

With idempotency keys:
- Replay is safe by construction
- Reconciliation becomes a count comparison: expected records vs. actual records
- Data recovery is a matter of re-running the pipeline, not manually patching data

**If your transaction processing pipeline does not have idempotency keys, adding them is the highest-priority engineering investment for reliability. Not circuit breakers, not observability — idempotency keys first.**

---

### Event Sourcing and Kafka Replay

Kafka enables rewindable replay by design. Messages are retained on the topic for a configurable retention period (commonly 7-30 days). Any consumer can be rewound to any offset within the retention window.

**Replay for incident recovery**:

```bash
# Find the offset for a specific timestamp
kafka-run-class.sh kafka.tools.GetOffsetShell \
  --bootstrap-server kafka:9092 \
  --topic payment-events \
  --time 1751286000000  # Unix timestamp in milliseconds (14:00:00 UTC)

# Reset consumer group to that offset
kafka-consumer-groups.sh \
  --bootstrap-server kafka:9092 \
  --group payment-processor \
  --reset-offsets \
  --to-offset [offset-from-above] \
  --topic payment-events \
  --execute

# Verify the consumer group is at the new offset
kafka-consumer-groups.sh \
  --bootstrap-server kafka:9092 \
  --group payment-processor \
  --describe
```

**Pre-conditions for Kafka replay**:
- Consumer processing is idempotent (re-processing already-processed messages is safe)
- The replay timestamp is before the first corrupted message (include a buffer)
- The consumer lag from the replay point will not overwhelm downstream systems

For Kafka-based event sourcing (event store as source of truth), replay is the native recovery mechanism. Services rebuild their state by replaying the event log from the recovery point. This is why event sourcing is the most resilient data architecture for distributed systems.

---

### Ledger Reconciliation for Financial Systems

Financial systems have a specific reconciliation requirement: **double-entry bookkeeping**. Every debit must have a corresponding credit. If the incident caused transactions to fail mid-flight, some debits may exist without their corresponding credits, or vice versa.

**Reconciliation process for financial systems**:

1. **Define the affected transaction set**: all transactions with `created_at` within the incident window that have status `FAILED`, `PENDING`, or `PROCESSING`

2. **Check balance for each transaction**:
```sql
-- Find transactions with mismatched debit/credit
SELECT 
  transaction_id,
  SUM(CASE WHEN entry_type = 'DEBIT' THEN amount ELSE 0 END) as total_debits,
  SUM(CASE WHEN entry_type = 'CREDIT' THEN amount ELSE 0 END) as total_credits,
  ABS(SUM(CASE WHEN entry_type = 'DEBIT' THEN amount ELSE -amount END)) as imbalance
FROM ledger_entries
WHERE transaction_id IN (
  SELECT transaction_id FROM transactions
  WHERE created_at BETWEEN '2026-06-30 14:07:00' AND '2026-06-30 14:34:00'
    AND status IN ('FAILED', 'PENDING', 'PROCESSING')
)
GROUP BY transaction_id
HAVING imbalance > 0
ORDER BY imbalance DESC;
```

3. **Classify each imbalanced transaction**:
   - Debit exists, no credit: payment was charged but not recorded as received → apply compensating credit or reverse the debit
   - Credit exists, no debit: payment was recorded as received but not charged → apply compensating debit or void the credit
   - Both exist but amounts mismatch: data corruption → investigate individually

4. **Apply compensating transactions** (do not delete or modify existing ledger entries — that destroys the audit trail):
```sql
-- Apply a compensating credit for an orphaned debit
INSERT INTO ledger_entries (
  transaction_id, entry_type, amount, created_at, created_by, note
) VALUES (
  'txn_1234567890',
  'CREDIT',
  99.99,
  NOW(),
  'incident-reconciliation-TICK-4471',
  'Compensating entry for orphaned debit from incident 2026-06-30 14:07-14:34'
);
```

### Communicating Retroactive Data Corrections to Customers

When an incident resulted in incorrect data being presented to customers (wrong balance, duplicate charge, missing order), customers must be notified. The communication must be:

**What to say**:
- What happened (brief, non-technical)
- What was incorrect and what the correct value is
- What action the customer needs to take (usually: none)
- What you are doing to prevent recurrence (brief)

**What not to say**:
- Internal system names, service names, or error codes
- Root cause details that could be a security concern
- Speculation about data that is not yet confirmed

**Timing**:
- Notify within 24 hours of confirmed data correction for payment/financial impacts
- Notify within 48 hours for non-financial data impacts
- Do not notify until the correction is confirmed complete — partial correction communications create confusion

**Audit trail preservation during customer communication**: every customer notification triggered by a data correction must reference the incident ticket. This creates the link between the customer-visible event and the internal record for compliance and legal purposes.

---

### Audit Trail Requirements for Reconciliation

Every reconciliation action must be logged with:

| Field | Content |
|---|---|
| Timestamp | UTC timestamp of the action |
| Actor | Engineer who performed the action (or system job ID if automated) |
| Action | What was done (row count, operation type, tables/topics affected) |
| Scope | Exact WHERE clause or offset range |
| Pre-state | Count or hash of data before the action |
| Post-state | Count or hash of data after the action |
| Verification | How it was confirmed correct (reconciliation query, count check) |
| Ticket reference | Incident ticket or Jira issue ID |

This audit trail is the evidence that the recovery was complete and correct. It is also the input to the postmortem, the compliance report, and any regulatory disclosure.

## Part 3: Fix Implementation Decision Gates

### Which Fix Type to Apply

The decision of which fix type to apply depends on the failure mode. During an active incident, you typically need to work through multiple fix types in sequence — a traffic fix first to shed load and stop customer impact, then a code or configuration fix to address the direct cause, then a data fix post-incident to reconcile affected records.

**Decision matrix**:

| Failure Mode | First Response Fix | Root Cause Fix | Post-Incident |
|---|---|---|---|
| Bad code deployment causing errors | Rollback (Code) | Canary fix-forward (Code) | FMEA update; add test coverage |
| DB connection pool exhausted | Scale down traffic (Traffic) | Pool size increase (Config); slow query alert (Config) | Add slow query alerting standard |
| Third-party API failure | Circuit breaker open (Config); feature disable (Config) | Fallback provider (Dependency) | Add secondary provider |
| Message processing backlog | Consumer scale-up (Infra) | Consumer lag alarm (Config) | Review scaling policy |
| Data corruption from bug | Stop the corrupt flow (Code/Config) | Fix the bug (Code) | Full reconciliation (Data) |
| Node hardware failure | Drain node (Infra); re-schedule pods (Infra) | Auto-healing review (Infra) | Node replacement; capacity review |
| DDoS / traffic spike | Rate limit (Traffic); CDN shield (Traffic) | WAF rule update (Config) | Review rate limit thresholds |

---

### Fix Sequencing During Active Incident

The principle is always: **fastest mitigation first, then systemic fix**.

```
[INCIDENT DETECTED]
        ↓
[MITIGATION PHASE] ← Goal: stop customer impact. Time budget: <15 minutes
  • Traffic fix (shed load, activate fallback)
  • Config fix (disable failing feature, adjust throttle)
  • Rollback if confirmed deployment caused failure
        ↓
[STABILIZATION] ← Goal: confirm customer impact stopped. Verify metrics.
        ↓
[RESOLUTION PHASE] ← Goal: fix the direct cause. Time budget: varies by complexity
  • Code fix (if mitigation was a workaround, fix the underlying bug)
  • Data fix (reconcile affected records from incident window)
        ↓
[VERIFICATION] ← Goal: confirm system is healthy. Monitor for 30 minutes.
        ↓
[POST-INCIDENT] ← Goal: prevent recurrence. Time budget: next sprint.
  • Root cause fix (FMEA, observability gap, design gap)
  • Prevention (add alarm, add circuit breaker, add fallback)
```

Do not skip the verification phase. Many secondary incidents occur because the team assumed stability after the direct cause was fixed, without monitoring long enough to confirm that the system had truly recovered.

### The Dry-Run Rule

**Every data fix and every infrastructure change that cannot be instantly reversed requires a dry-run validation before execution.**

A dry run is not a rehearsal in staging. A dry run is running the scoped query or command against production in read-only mode, confirming the scope and expected effect, and then proceeding with execution.

For data fixes:
```
Dry run = SELECT with the same WHERE clause as the UPDATE
Validate = row count matches expected scope
Proceed = only after count confirmed
```

For infrastructure changes:
```
Dry run = kubectl drain --dry-run=client node-name
Validate = output shows which pods would be evicted; confirm no critical singletons
Proceed = kubectl drain without --dry-run
```

For Terraform (post-incident remediation):
```
Dry run = terraform plan
Validate = plan shows only expected changes; no surprise resource replacements
Proceed = terraform apply only after plan reviewed
```

**No exceptions to the dry-run rule for data fixes.** The authorization process for a data fix includes dry-run validation as a required step. IC authorization is granted on the basis of a confirmed dry-run scope, not on the basis of the engineer's estimate of what will be affected.

## Summary

### Fix Type Reference Card

| Fix Type | When | Fastest Path | Highest Risk | Key Guard Rail |
|---|---|---|---|---|
| Code | Bad code causing failure | Rollback | Hotfix during incident introduces second variable | IC must open fix-forward track |
| Configuration | Parameter adjustment needed | Feature flag toggle | Config change bypasses code review | Log pre/post value; define rollback trigger |
| Traffic | Load shedding, failover | Load balancer weight change | Secondary may not be warm | Validate secondary capacity before shifting |
| Data | Corrupted/missing records | Idempotent replay from DLQ | Wrong data fix compounds corruption | Dry run + DB engineer + IC auth; always |
| Infrastructure | Node/network/storage failure | Pod restart | Node drain affects all workloads on node | Check all workloads on target before touching |
| Dependency | Third-party service failure | Feature flag disable | Fallback provider may have different behavior | Pre-test fallback path; know escalation contacts |
