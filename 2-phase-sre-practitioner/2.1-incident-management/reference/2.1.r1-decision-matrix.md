# Incident Decision Matrix — Quick Reference
**Purpose**: One-page decision reference for ICs and on-call engineers. Use during active incidents. No prose — scan and decide.

---

## 1. SEVERITY CLASSIFICATION MATRIX

Classify by the **highest** dimension that matches. When in doubt, go higher — you can downgrade, you cannot undo under-response.

| Impact Dimension        | SEV4                     | SEV3                        | SEV2                            | SEV1                              |
|-------------------------|--------------------------|-----------------------------|---------------------------------|-----------------------------------|
| Users affected          | <1% or internal only     | 1–10% of users              | 10–50% of users                 | >50% of users OR named enterprise |
| Revenue impact/min      | <$500/min                | $500–$1,000/min             | $1,000–$10,000/min              | >$10,000/min                      |
| Error rate              | <0.5% above baseline     | 0.5–2% above baseline       | 2–10% above baseline            | >10% above baseline               |
| Journey criticality     | P2 only (support/admin)  | P1 journey degraded         | P0 journey degraded             | P0 journey down or P1+P2 compound |
| SLA breach risk         | No breach possible       | Approaching monthly limit   | Single customer breach likely   | Multiple customers breaching now  |
| Regulatory exposure     | None                     | None                        | Possible (audit finding)        | Confirmed (PCI/GDPR/HIPAA)        |
| Spread risk             | Isolated / single service| Contained, no spread risk   | Adjacent services at risk       | Active cascade in progress        |
| BRI score               | <0.10                    | 0.10–0.30                   | 0.30–0.70                       | >0.70                             |

## 3. ROLLBACK vs. FIX-FORWARD DECISION MATRIX

| Signal                                              | Decision           | Rationale                                          |
|-----------------------------------------------------|--------------------|----------------------------------------------------|
| Deploy in last 60 min + error rate spike matches    | **Rollback**       | High confidence deploy is cause; rollback is fastest and lowest risk |
| Deploy in last 60 min + error rate spike does NOT match timing | **Investigate** | Don't rollback if causality unconfirmed |
| No deploy in last 60 min                            | **Fix-Forward**    | Rollback won't help; root cause is elsewhere       |
| Deploy > 60 min ago + cascading failure now          | **Fix-Forward**    | Too much latency; the deploy is probably not the trigger |
| Database migration was deployed                     | **Fix-Forward (with care)** | Rollback may not be possible or may corrupt data |
| Multiple services affected simultaneously            | **Fix-Forward + Isolate** | Rollback of one service won't stop a cascade |
| Rollback attempted and failed or error persists     | **Fix-Forward**    | Rollback is not the solution; stop retrying it     |
| Fix requires code change > 30 min to build/deploy  | **Mitigate first** (circuit breaker, scale, feature flag) then fix forward | MTTR over diagnosis |

**Rollback authorization**:
- Config-only rollback: on-call can execute independently
- Code rollback: IC authorization required
- Database rollback: Engineering Director + DBA required

## 5. BRI → SEV MAPPING

`BRI = (affected_users / total_users) × journey_criticality × tier_weight_of_failing_service`

Tier weights (from SRECapstone framework): P0 = 1.0 | P1 = 0.8 | P2 = 0.5 | P3 = 0.3
Journey criticality: 1.0 for revenue-critical (checkout, payment, auth) | 0.7 for P1 journeys | 0.5 for P2 journeys | 0.3 for P3/internal

| BRI Score  | SEV Level | Worked Example                                                     |
|------------|-----------|--------------------------------------------------------------------|
| > 0.70     | SEV1      | 94% checkout error rate: (0.94) × (1.0) × (1.0 P0) = 0.94 → SEV1 |
| 0.30–0.70  | SEV2      | 50% of users on auth service: (0.50) × (0.7) × (1.0 P0) = 0.35 → SEV2 |
| 0.10–0.30  | SEV3      | 35% of users on P2 recommendation: (0.35) × (0.5) × (0.5 P2) = 0.088 → SEV4; but 35% on P1: (0.35) × (0.7) × (0.8 P1) = 0.196 → SEV3 |
| < 0.10     | SEV4      | <10% on P2 journey; internal tooling degraded                      |

**Override rule**: If any named enterprise account is fully down, minimum SEV2 regardless of BRI score.
**Multi-dimension rule**: Always classify by the highest dimension. BRI at SEV3 does not prevent SEV2 if revenue exposure is in the SEV2 range.

## 8. RECOVERY STRATEGY SELECTION GUIDE

| RTO Tier   | Target RTO    | Strategy                    | When to Use                                                    |
|------------|---------------|-----------------------------|----------------------------------------------------------------|
| Tier 0     | <5 minutes    | Automated failover / rollback | Pre-built; triggered by alert; no human decision needed       |
| Tier 1     | 5–30 minutes  | Rollback + runbook-driven   | Deploy-caused; well-understood failure mode; runbook exists    |
| Tier 1     | 5–30 minutes  | Feature flag / kill switch  | Feature-specific; flag infrastructure healthy                  |
| Tier 2     | 30–120 minutes| Fix-forward with hotfix     | No rollback path; small targeted code fix possible             |
| Tier 2     | 30–120 minutes| Scale out + shed load       | Capacity failure; infrastructure can absorb more replicas      |
| Tier 3     | 2–8 hours     | Full region failover to DR  | Primary region unrecoverable; DR tested and ready              |
| Tier 3     | 2–8 hours     | Data restore from backup    | Data corruption or loss; backup recovery tested                |
| Tier 4     | 8–24 hours    | Full rebuild                | Last resort; DR not ready; backup restore not viable           |

**Strategy selection principle**: Always choose the highest tier your RTO allows. A Tier 3 strategy attempted when Tier 1 was available adds hours to MTTR unnecessarily.
