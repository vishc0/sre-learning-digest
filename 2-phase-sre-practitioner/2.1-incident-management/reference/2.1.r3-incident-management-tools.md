# Incident Management Tools Reference
**Purpose**: Quick-reference map of the software tooling ecosystem for incident management — what each tool does, which phase it serves, and where to learn more.  
**How to use**: Use this to understand the landscape. Your org will not use all of these. Knowing what category a tool belongs to — and what it replaces — is the interview-level knowledge.

---

## The Tool Landscape by Phase

| Phase | Tool Category | Purpose |
|-------|--------------|---------|
| Detect | Observability / Monitoring | Surface the signal |
| Detect | Alert / On-Call | Route the signal to a human |
| Declare + Triage | Incident Platform | Spin up the response structure |
| Communicate | Status Page | Inform customers |
| Communicate | Communication | Bridge and async coordination |
| Mitigate | Runbook Automation | Execute known-safe remediations |
| Resolve | ITSM / Ticketing | Track work and escalations |
| Learn | Postmortem / RCA | Analyze and improve |
| Improve | Chaos Engineering | Test the blast radius before an incident does |

## Category 1 — Alert and On-Call

These tools own the pager. They receive alert signals from monitoring, route them to the right person based on schedules and escalation policies, and track acknowledgment.

| Tool | One-liner | Key Differentiator | Learn More |
|------|-----------|--------------------|------------|
| **PagerDuty** | Industry-standard on-call platform — schedules, escalation policies, alert routing | Most enterprise adoption; best third-party integrations; acquired Jeli for postmortem | pagerduty.com/resources |
| **Opsgenie** (Atlassian) | On-call alerting integrated into the Atlassian stack (Jira, Confluence, Statuspage) | Best choice if already in the Atlassian ecosystem | atlassian.com/software/opsgenie |
| **Splunk On-Call** (formerly VictorOps) | On-call platform with strong Splunk and log correlation integration | Native Splunk integration makes it natural for Splunk-heavy shops | victorops.com / splunk.com/on-call |
| **xMatters** | Incident notification and workflow automation at enterprise scale | Strongest at complex multi-team notification workflows | xmatters.com |
| **Grafana OnCall** | Open-source on-call alerting, self-hostable, integrates with Grafana stack | Free tier; good choice for Grafana/Prometheus shops | grafana.com/products/oncall |

## Category 2 — Observability and Monitoring

These tools generate the signals that on-call tools consume. The difference between good and poor detection is almost entirely here.

| Tool | One-liner | Key Differentiator | Learn More |
|------|-----------|--------------------|------------|
| **Datadog** | Full-stack observability: APM, infrastructure, logs, synthetics, dashboards | Best-in-class correlation across traces, metrics, logs in one platform | datadoghq.com/blog |
| **New Relic** | APM and observability with a consumption-based pricing model | Strong APM depth; open telemetry support; good for cost-conscious teams | newrelic.com/blog |
| **Dynatrace** | AI-driven observability with automatic topology mapping and anomaly detection | Davis AI engine auto-identifies root cause candidates without manual correlation | dynatrace.com/resource |
| **Grafana + Prometheus** | Open-source metrics, dashboards, and alerting stack | Most popular open-source combination; self-hosted, highly customizable | grafana.com/docs, prometheus.io/docs |
| **Splunk** | Log aggregation, search, and alerting at enterprise scale | Unmatched log search depth; SIEM and incident correlation; native to telecom and regulated industries | docs.splunk.com |
| **Honeycomb** | Observability built for high-cardinality event data and exploratory debugging | Best tool for debugging novel incidents where you don't know what to query | honeycomb.io/resources |
| **AWS CloudWatch** | Native AWS monitoring: metrics, logs, alarms, dashboards, synthetics | Zero-friction for AWS-hosted services; no agent deployment for managed services | docs.aws.amazon.com/cloudwatch |

## Category 3 — Incident Platforms (End-to-End)

These tools wrap the entire incident lifecycle: channel creation, role assignment, timeline capture, task tracking, and postmortem generation. They reduce the manual ceremony of spinning up a response.

| Tool | One-liner | Key Differentiator | Learn More |
|------|-----------|--------------------|------------|
| **FireHydrant** | Incident workflow automation: auto-creates Slack channels, pages responders, tracks timeline | Strong runbook integration; automatic retrospective generation; good free tier | firehydrant.com/resources |
| **Rootly** | Slack-native incident management — declare, track, and close incidents from within Slack | Best Slack integration; auto-generates PIR drafts from incident channel history | rootly.com/blog |
| **incident.io** | Modern incident platform with Slack bot, status pages, and postmortem workflows | Clean UX; popular in tech companies; strong analytics on incident patterns | incident.io/blog |
| **Blameless** | SRE platform combining incident management, SLO tracking, and postmortem facilitation | Strong SLO and error budget integration; reliability analytics built in | blameless.com/resources |
| **PagerDuty Operations Cloud** | PagerDuty's expanded incident management layer beyond on-call routing | Best for orgs already using PagerDuty; unified alert-to-resolution workflow | pagerduty.com/platform |

## Category 4 — Status Pages

These tools communicate incident status to external customers and internal stakeholders. They decouple customer communication from your internal response.

| Tool | One-liner | Key Differentiator | Learn More |
|------|-----------|--------------------|------------|
| **Statuspage** (Atlassian) | Industry-standard hosted status page with component-level status and subscriber notifications | Most widely recognized; integrates with PagerDuty and Opsgenie | atlassian.com/software/statuspage |
| **Better Uptime** | Status page + uptime monitoring + incident communication in one tool | Cleaner UI than Statuspage; built-in monitoring means no separate synthetic tool needed | betteruptime.com |
| **Instatus** | Lightweight, fast status page with a generous free tier | Simplest setup; good for teams that need a status page and nothing else | instatus.com |
| **Cachet** | Open-source, self-hosted status page | Full control over data; no vendor dependency; requires your own hosting | cachethq.io / github.com/cachethq/cachet |

## Category 5 — Communication and War Room

The bridge. These tools host the live incident call and async coordination channel.

| Tool | One-liner | Key Differentiator | Learn More |
|------|-----------|--------------------|------------|
| **Slack** | Async-first incident coordination — incident channels, incident bots, integrations | Most common; incident.io, Rootly, and FireHydrant all build on top of Slack | slack.com/intl/en-us/resources |
| **Microsoft Teams** | Enterprise communication hub used in regulated industries and large enterprises | Required in many enterprise/government/telecom environments; SharePoint integration | learn.microsoft.com/en-us/microsoftteams |
| **Zoom** | Video bridge for live incident calls | Reliable; breakout rooms for parallel tracks; broad adoption | zoom.us/resources |
| **PagerDuty Response Mobilizer** | Auto-creates video conference room when an incident is declared | Eliminates "what bridge URL do we use?" — removes one step from the first 5 minutes | pagerduty.com/platform/incident-response |

## Category 6 — Postmortem and RCA Tooling

These tools facilitate timeline reconstruction, root cause analysis, and action item tracking after the incident.

| Tool | One-liner | Key Differentiator | Learn More |
|------|-----------|--------------------|------------|
| **Jeli** (now PagerDuty) | Incident learning platform — automated timeline from Slack + deep interview facilitation | Best-in-class for complex incidents; learns from incident channel history automatically | jeli.io / pagerduty.com/jeli |
| **Blameless PIR** | Postmortem module built into the Blameless SRE platform | Tight integration with SLO data; action item tracking with SLO impact scoring | blameless.com/reliability-workflow |
| **Confluence** | General-purpose wiki used for postmortem documents in most orgs | Most teams already have it; zero marginal cost; free-form but requires discipline | atlassian.com/software/confluence |
| **Google Docs** | Unstructured but universally accessible postmortem storage | Good for small teams; no structure enforcement; easy for external stakeholders | docs.google.com |

## Category 7 — Runbook Automation

These tools execute runbooks as code — automated remediation steps that a responder or system can trigger rather than execute manually step by step.

| Tool | One-liner | Key Differentiator | Learn More |
|------|-----------|--------------------|------------|
| **PagerDuty Process Automation** (formerly Rundeck) | Runbook automation platform — execute predefined operational jobs on demand or automatically | Most mature; widely used for triggered remediations like cache flush, pod restart, failover | rundeck.com / pagerduty.com/platform |
| **StackStorm** | Event-driven automation (if-this-then-that for ops) — triggers runbooks on alert conditions | Open-source; most powerful for complex multi-step automated responses | stackstorm.com / github.com/StackStorm |
| **AWS Systems Manager (SSM) Run Command** | Execute runbooks on EC2 and ECS instances at scale via AWS automation documents | Native for AWS shops; no agent install on managed instances; integrates with CloudWatch alarms | docs.aws.amazon.com/systems-manager |
| **Ansible** | Infrastructure automation that doubles as runbook execution at scale | Best for fleet-wide remediations (patching N servers, rotating credentials across instances) | docs.ansible.com |

## Category 8 — ITSM and Ticketing

Where incidents are tracked as formal records, especially in change management and compliance-heavy environments.

| Tool | One-liner | Key Differentiator | Learn More |
|------|-----------|--------------------|------------|
| **Jira Service Management** | ITSM platform in the Atlassian stack — incident tickets, change requests, problem records | Best Jira integration; natural choice for software teams already using Jira | atlassian.com/software/jira/service-management |
| **ServiceNow** | Enterprise ITSM platform — ITIL-compliant incident, problem, and change management | Industry standard in large enterprises, telecom, financial services; heavy but powerful | servicenow.com/products/itsm.html |
| **Freshservice** | Mid-market ITSM with a cleaner UX than ServiceNow | Easier to configure; good for 500–5,000 seat orgs that want ITSM without ServiceNow complexity | freshservice.com |

## Category 9 — Chaos Engineering

These tools test your system's resilience by injecting controlled failures before incidents do it for you. Game Days and failure injection are now a Category 1 SRE practice.

| Tool | One-liner | Key Differentiator | Learn More |
|------|-----------|--------------------|------------|
| **Gremlin** | Commercial chaos engineering platform — inject CPU, memory, network, and state failures | Safest for production use; fine-grained targeting; attack library with 100+ failure types | gremlin.com/community/tutorials |
| **Chaos Monkey** (Netflix OSS) | Randomly terminates EC2 instances in production to ensure services handle failure | The original; simpler than Gremlin; validates K8s/EC2 failover only | github.com/Netflix/chaosmonkey |
| **AWS Fault Injection Simulator (FIS)** | Native AWS chaos engineering — inject failures into EC2, ECS, EKS, RDS, and more | No additional tooling; directly integrated with CloudWatch; good for AWS-native shops | docs.aws.amazon.com/fis |
| **LitmusChaos** | Open-source chaos engineering for Kubernetes | CNCF project; Kubernetes-native; strong for EKS/GKE/AKS workloads | litmuschaos.io |
| **Chaos Toolkit** | Open-source, framework-agnostic chaos experiments as code | Write experiments in JSON/YAML; works across cloud providers; extensible | chaostoolkit.org |

## How These Categories Map to the Incident Lifecycle

```
DETECT           → Observability (Datadog, Splunk, Grafana) → Alert/On-Call (PagerDuty)
                                                                      ↓
DECLARE          → Incident Platform (FireHydrant, Rootly) → Communication (Slack, Zoom)
                                                                      ↓
TRIAGE           → Observability (trace correlation, dashboard deep-dive)
                                                                      ↓
MITIGATE         → Runbook Automation (Rundeck, SSM) → Communication (Slack bridge)
                                                                      ↓
COMMUNICATE      → Status Page (Statuspage, Better Uptime) → Communication (Teams/Slack)
                                                                      ↓
RESOLVE          → ITSM (Jira SM, ServiceNow) → Incident Platform (close + handoff)
                                                                      ↓
LEARN            → Postmortem (Jeli, Blameless) → ITSM (action item tracking)
                                                                      ↓
IMPROVE          → Chaos Engineering (Gremlin, AWS FIS) → Observability (new alerts)
```
