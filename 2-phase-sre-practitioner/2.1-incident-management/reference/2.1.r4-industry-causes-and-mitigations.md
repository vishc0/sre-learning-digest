# Industry Root Cause Taxonomy and Mitigation Patterns
**Purpose**: Reference library of root cause categories, real-world patterns, mitigation investments, and detection improvements. Use during PIR root cause analysis and during proactive reliability reviews.

---

## HOW TO USE THIS DOCUMENT

During a PIR:
1. Identify which category best fits your direct cause
2. Check the sub-patterns — does yours match a known pattern?
3. Use the mitigation column to validate that your action items address the right layer
4. Use the detection improvement column to ensure your MTTD improvement targets the correct signal

---

## CATEGORY 1 — DEPLOYMENT FAILURES

**Description**: Changes to code, configuration, dependencies, or infrastructure cause regression or unexpected behavior.

### Sub-Patterns

| Sub-Pattern               | Description                                                              | Real-World Examples                                     |
|---------------------------|--------------------------------------------------------------------------|---------------------------------------------------------|
| Bad code / logic error    | New code introduces a bug not caught in testing                          | Knight Capital 2012: erroneous order routing code wiped $440M in 45 min |
| Config drift              | Production config diverges from repo / IaC; manual override not tracked | Wrong pool_size in auth-service deploy (INC-20451)      |
| Dependency version mismatch| Transitive dependency upgrade changes behavior                          | Log4Shell cascade: upgrading one lib changed JNDI behavior across 100+ services |
| Untested deploy path      | Code works in staging but fails in prod due to data volume or traffic    | Feature flag enabled 100% in prod; only 5% tested in staging |
| Schema migration failure  | Database schema change fails mid-migration leaving partial state         | PostgreSQL migration on 50M-row table: lock timeout causing half-migrated table |
| Missing rollback path     | Deploy was applied but rollback not verified; rollback itself fails      | Blue-green swap succeeded; green has a bug; blue already torn down |

### RPN Score Range
Severity: 7–9 | Occurrence: 8 | Detectability: 3–5 → **RPN: 168–360** (highest category)

### Primary Mitigation Investment
- Progressive delivery: canary → 10% → 50% → 100% with automated error rate gate at each stage
- Deployment freeze windows: no deploys during peak traffic or before major business events
- Automated rollback: if error rate rises > X% within Y minutes of deploy, auto-rollback (no human required)
- Config as code + diff review: all config changes must go through PR with second-engineer review

### Detection Improvement
- Deploy event markers overlaid on all key metrics dashboards (latency, error rate, pool utilization)
- SLO burn rate alert (1h window) catches regressions within 1–2 minutes vs. 3–5 minute threshold alerts
- Correlation alert: "new deploy AND error rate rising" triggers faster than error rate alone

### Prevention
- Define and test rollback path for every deploy before it ships to production
- Run load tests in staging against production-scale data volumes (not staging-scale)
- Chaos engineering: inject failures in staging to verify graceful degradation

## CATEGORY 2 — CAPACITY FAILURES

**Description**: Service cannot handle the volume or resource demands placed on it, causing degradation or outage.

### Sub-Patterns

| Sub-Pattern               | Description                                                              | Real-World Examples                                    |
|---------------------------|--------------------------------------------------------------------------|--------------------------------------------------------|
| Traffic spike             | Sudden volume increase beyond designed capacity                          | Twitter "Slashdot effect" on product launches; World Cup streaming peaks |
| Autoscaling lag           | Autoscaler triggers too slowly; services degrade before new pods are ready | Black Friday: autoscaler fires at 70% CPU; pods take 4 min to ready; 6 min of degradation |
| Resource exhaustion       | CPU, memory, file descriptors, or connections exhausted                  | OOM kill loop: pod hits memory limit → OOM kill → restart → OOM kill |
| Connection pool exhaustion| Thread pool or DB connection pool saturated; requests queue → timeout    | INC-20451: auth-service pool 100/100 during Friday peak |
| Noisy neighbor            | One tenant's load degrades resources shared with others                  | Multi-tenant Kubernetes: one namespace consuming all node CPU; other ns starved |
| Thundering herd           | Cache miss after expiry causes all requests to hit origin simultaneously  | CDN cache invalidation at midnight triggers 10× DB load spike |

### RPN Score Range
Severity: 6–9 | Occurrence: 6 | Detectability: 4–6 → **RPN: 144–324**

### Primary Mitigation Investment
- Capacity planning: quarterly review of headroom against 2× peak; auto-scale tested under synthetic load
- Pre-provisioned surge capacity: keep 20% headroom above peak, not above average
- Circuit breakers: shed load gracefully when capacity is saturated rather than queuing infinitely
- Cache stampede prevention: probabilistic early cache refresh (jitter + pre-warm)

### Detection Improvement
- Alert on saturation metrics (connection pool %, memory %, CPU %) at 80% — not 100% (which is too late)
- Predictive alerts: if growth rate of metric X is trending to breach Y within Z minutes, fire now
- Autoscaler lead time testing: measure how long scale-out takes; set alert threshold to fire before degradation begins

### Prevention
- Load test every major feature at 2× expected peak before launch
- Set resource quotas with namespace isolation in Kubernetes
- Cache warm-up procedure before planned traffic events

## CATEGORY 3 — NETWORK AND INFRASTRUCTURE FAILURES

**Description**: Network routing, DNS, hardware, or cloud provider events prevent services from communicating or functioning.

### Sub-Patterns

| Sub-Pattern               | Description                                                              | Real-World Examples                                    |
|---------------------------|--------------------------------------------------------------------------|--------------------------------------------------------|
| BGP misroute              | Routing table error sends traffic to wrong destination or black-holes it | Cloudflare 2019: BGP leak from Verizon black-holed significant internet traffic |
| DNS failure               | DNS resolution fails or returns wrong records                            | Fastly 2021: misconfigured DNS propagation took down 85% of internet in 1 minute |
| Hardware fault            | Physical server, NIC, or storage device failure                          | AWS EC2 hypervisor failure causing instance loss in single AZ |
| Cloud provider event      | Cloud provider infrastructure outage affecting customer workloads         | AWS us-east-1 2021: kinesis outage cascaded to IAM, preventing EC2 start |
| Certificate expiry        | TLS certificate expires; all HTTPS traffic fails                         | Microsoft Azure 2022: expired cert caused Teams/Outlook outage; LinkedIn 2021 |
| NTP drift                 | Clock skew between services causes auth token validation failures         | Distributed system: JWT "iat" claim in the future causes token rejections |

### RPN Score Range
Severity: 7–10 | Occurrence: 4 | Detectability: 2–4 → **RPN: 112–320**

### Primary Mitigation Investment
- Multi-region / multi-AZ deployment with automatic failover: single zone loss cannot take down the service
- DNS TTL management: low TTL during changes (60s) so rollbacks propagate fast
- Certificate management automation: auto-renew all certs 30 days before expiry; alert at 60 days, 30 days, 7 days

### Detection Improvement
- Synthetic monitoring from external probes (not from within the cluster — confirms user-path health)
- BGP monitoring services (BGPmon, Cloudflare Radar) with webhook alerts
- DNS resolution monitoring every 60 seconds from multiple geographic vantage points
- Certificate expiry monitoring: every certificate in inventory, checked daily

### Prevention
- Certificate inventory automation: no cert should be manually managed; all in cert-manager or equivalent
- Chaos engineering: AZ failure drills quarterly; verify failover works before you need it
- Multi-provider DNS (Route 53 + Cloudflare) for critical paths

## CATEGORY 4 — DATA AND DATABASE FAILURES

**Description**: Failures in data storage, query execution, or data integrity that affect service functionality.

### Sub-Patterns

| Sub-Pattern               | Description                                                              | Real-World Examples                                    |
|---------------------------|--------------------------------------------------------------------------|--------------------------------------------------------|
| Runaway query             | Missing index or bad query plan causes full table scan; DB CPU spikes    | N+1 query in ORM: 1 request generates 500 queries; DB overwhelmed |
| Lock contention           | Long-running transaction holds row/table locks; others queue → timeout   | Postgres UPDATE without index on 50M rows: holds lock for 8 min; all writes queue |
| Migration failure         | Schema migration fails mid-execution; table in partial state             | ALTER TABLE on large table times out; half of rows migrated; foreign key broken |
| Data corruption           | Bit flips, failed writes, or software bugs leave data in invalid state   | MongoDB 2019: journaling disabled for performance; power loss caused journal corruption |
| Replication lag           | Read replicas fall behind primary; stale reads cause inconsistency       | E-commerce: order placed on primary; confirmation page reads replica (0s lag → 5s lag → stale) |
| Backup failure (silent)   | Backups appear to complete but are not restorable                        | GitLab 2017: deleted production DB; 5 backup methods — 4 had never been tested for restore |

### RPN Score Range
Severity: 7–10 | Occurrence: 5 | Detectability: 3–5 → **RPN: 105–250**

### Primary Mitigation Investment
- Query analysis in CI: EXPLAIN ANALYZE on all ORM-generated queries before merge
- Migration zero-downtime patterns: expand-then-contract; never block writes with lock-holding migrations
- Backup testing: restore drill monthly; automated restore validation weekly

### Detection Improvement
- Slow query log alerting: queries > P95 threshold trigger alert immediately
- Replication lag monitoring: alert when replica lag exceeds 5 seconds
- Read/write split monitoring: if reads are returning stale data, alert before users see it

### Prevention
- Database change management: every migration peer-reviewed; test on production-scale snapshot in staging
- Point-in-time recovery (PITR) enabled for all production databases
- Read-your-own-writes consistency pattern for critical user-facing flows

## CATEGORY 5 — THIRD-PARTY DEPENDENCY FAILURES

**Description**: External services, APIs, or infrastructure components outside your control fail or change behavior.

### Sub-Patterns

| Sub-Pattern               | Description                                                              | Real-World Examples                                    |
|---------------------------|--------------------------------------------------------------------------|--------------------------------------------------------|
| CDN / edge outage         | CDN provider failure makes static assets or API gateway unreachable      | Fastly June 2021: config change took CDN offline for 49 minutes |
| Payment provider failure  | Payment gateway returns errors; checkout cannot complete                 | Stripe degraded 2019: 99% of payment attempts failing for 60 minutes |
| Certificate expiry (3P)   | Third-party certificate expires; API calls to their endpoint fail        | Root CA expiry (Let's Encrypt DST Root CA X3 expiry 2021) broke millions of clients |
| API deprecation / breaking change | Third-party API changes behavior without notice or breaks clients | Twitter API v1 sunset: apps relying on v1 stop working overnight |
| SaaS dependency failure   | PagerDuty, GitHub, Datadog, or Okta outage affects operational capability | GitHub Actions outage: CI/CD pipelines blocked; deploys halted |
| DNS provider failure      | Upstream DNS provider fails; all services depending on DNS resolution fail | Dyn DDoS attack 2016: DNS provider taken offline; Amazon, Twitter, Netflix unreachable |

### RPN Score Range
Severity: 6–9 | Occurrence: 4 | Detectability: 3–5 → **RPN: 72–180**

### Primary Mitigation Investment
- Vendor health monitoring: subscribe to every provider's status page; wire to Slack
- Circuit breakers on ALL third-party calls: define fallback behavior before the outage, not during
- Multi-vendor strategy for critical dependencies: two payment providers, two CDNs

### Detection Improvement
- External dependency health dashboards: synthetic calls to each third-party endpoint every 60s
- Dependency failure runbooks: pre-written for each external dependency before the outage happens

### Prevention
- Third-party API contract testing: test against their sandbox; catch breaking changes before prod traffic
- Map and classify every external dependency: what is the fallback if this fails? Document it.
- SLA pass-through analysis: if your SLA is 99.9% but your payment provider's SLA is 99.5%, your SLA is 99.5%

## CATEGORY 6 — SECURITY-TRIGGERED INCIDENTS

**Description**: Malicious activity or security events that cause operational impact or require emergency response.

### Sub-Patterns

| Sub-Pattern               | Description                                                              | Real-World Examples                                    |
|---------------------------|--------------------------------------------------------------------------|--------------------------------------------------------|
| DDoS attack               | Volumetric traffic flood exhausts network or compute capacity            | GitHub 2018: largest recorded DDoS at 1.35Tbps (Memcached amplification) |
| Credential stuffing       | Automated login attempts with stolen credentials; auth system overwhelmed| Auth service CPU spikes to 100% as bot traffic at 10× normal volume |
| Compromised API key       | Leaked key used to abuse services; runaway API usage or data exfil       | AWS key in public GitHub repo: cryptomining spun up at $50K/day |
| Ransomware / destructive attack | Malware encrypts or destroys data; services unable to start        | NotPetya 2017: Maersk's entire global network encrypted in hours |
| Certificate compromise    | Private key leaked; man-in-the-middle possible; cert must be revoked/replaced | DigiNotar 2011: CA compromised; all certs revoked; CA went bankrupt |
| Insider threat            | Malicious or negligent internal action causes data loss or disruption    | GitLab 2017: DBA accidentally deleted production database (negligent, not malicious) |

### RPN Score Range
Severity: 8–10 | Occurrence: 3–5 | Detectability: 2–4 → **RPN: 96–200**

### Primary Mitigation Investment
- DDoS mitigation: CDN-layer DDoS protection (Cloudflare, AWS Shield) as first line; rate limiting as second
- Secrets management: HashiCorp Vault or AWS Secrets Manager; zero hardcoded secrets; rotation automated
- Zero-trust access: no implicit trust based on network location; every service-to-service call authenticated

### Detection Improvement
- WAF rule alerting: fire immediately on credential stuffing patterns (>N failed logins/min from distinct IPs)
- API key usage anomaly detection: alert on sudden usage spike from a key (10× above baseline)
- SIEM correlation: cross-source detection of multi-stage attack patterns

### Prevention
- Secret scanning in CI: block commits with API keys, passwords, or private keys
- Penetration testing annually; bug bounty for continuous coverage
- Incident response plan for security events: separate runbook; pre-authorized response actions

## CATEGORY 7 — HUMAN ERROR PATTERNS

**Description**: Operational mistakes, miscommunication, or cognitive errors that cause or worsen incidents.

### Sub-Patterns

| Sub-Pattern               | Description                                                              | Real-World Examples                                    |
|---------------------------|--------------------------------------------------------------------------|--------------------------------------------------------|
| Right command, wrong environment | Correct command executed in production when staging was intended   | `kubectl delete deployment app -n production` instead of `-n staging` |
| Manual override not reversed | Emergency override applied; not removed; causes problems later       | Max replica count raised to 200 during incident; left there; monthly bill triples |
| Config copy-paste error   | Dev/staging config value pasted into production config                   | INC-20451: pool_size=100 from dev env copied to prod deploy |
| Change without review     | Solo change to critical system without peer verification                 | Single engineer changes firewall rule; service becomes unreachable |
| Documentation-reality gap | Runbook describes old behavior; engineer follows it; makes things worse  | Runbook says "restart pod X" — pod X was renamed 3 months ago; restarting wrong pod |
| Toil-induced error        | Repeated manual process → fatigue → mistake                             | 3 AM manual log rotation on 50 servers → wrong command on server #37 |

### RPN Score Range
Severity: 5–9 | Occurrence: 7 | Detectability: 2–4 → **RPN: 98–252**

### Primary Mitigation Investment
- Blast radius reduction: production access requires explicit per-session authorization; kubectl contexts require confirmation
- Pre-execution review: any destructive command must be copy-reviewed by a second person before executing
- Automation of repetitive operations: toil that causes errors should be automated, not optimized manually

### Detection Improvement
- Audit logging on all production actions: every kubectl command, every console action, timestamped and attributed
- Change management correlation: production changes require a linked ticket; unauthorized changes alert immediately

### Prevention
- Production/staging namespace segregation that requires active context switch confirmation
- "Baseball bat" protection: destructive commands require two-factor confirmation in the CLI tool
- Runbook accuracy testing: quarterly walkthrough of all P0 runbooks to verify steps match current system

## CHRONOLOGICAL RISK PATTERNS

Understanding when incidents are most likely to occur is as important as understanding why.

### High-Risk Time Windows

| Window                         | Why It's High-Risk                                                                | Mitigation                                        |
|--------------------------------|-----------------------------------------------------------------------------------|---------------------------------------------------|
| Monday morning (08:00–10:00)   | Weekend changes deployed without full monitoring; traffic ramp catches latent bugs | No deploys before 10 AM Monday; on-call heightened alert |
| Post-holiday restart           | Services cold-started; connection pools not warmed; caches empty → cache stampede | Staged traffic return; warm-up period before full traffic |
| End of quarter                 | Engineers rushing feature deploys before deadline; review shortcuts               | Deploy freeze last 2 business days of quarter     |
| Friday afternoon               | Deploys before the weekend; on-call degraded over weekend                         | Deploy freeze after 14:00 Friday (or after 12:00 for SEV1-prone services) |
| After annual leave / long weekend | Certificates, keys, or cron jobs that expire during leave go unnoticed          | Automated expiry monitoring; handoff protocol     |
| Peak business events           | Black Friday, Cyber Monday, product launches, marketing campaigns                 | Pre-event capacity review; deploy freeze; war room |

### Certificate Expiry: The Predictable Annual Incident

Certificate expiry is the only incident that is 100% preventable and 100% predictable — yet remains a top-10 cause of outages industry-wide.

**Why it keeps happening**:
1. Certificate managed manually → no automated alert → engineer who set it up leaves → no handoff
2. Alert exists but fires 7 days before expiry → engineer sees it → "I'll do it Monday" → Monday is too late
3. Auto-renewal configured → renewal fails silently (ACME challenge fails due to firewall change) → not caught

**Prevention pattern**:
- All certificates in inventory (automated discovery)
- Auto-renewal with auto-renewal failure alerting (not just expiry alerting)
- Manual alert ladder: 90 days / 60 days / 30 days / 7 days / 1 day
- Quarterly audit: confirm all certs are auto-renewing and all alerts are active

## FMEA RPN SUMMARY

| Category                      | RPN Range  | Primary Investment                          |
|-------------------------------|------------|---------------------------------------------|
| Deployment failures           | 168–360    | Progressive delivery + automated rollback   |
| Human error                   | 98–252     | Blast radius reduction + audit logging      |
| Capacity failures             | 144–324    | Saturation alerting at 80% + circuit breakers |
| Data / database failures      | 105–250    | Migration patterns + backup restore testing |
| Security-triggered            | 96–200     | DDoS protection + secrets management        |
| Network / infrastructure      | 112–320    | Multi-AZ + certificate automation           |
| Third-party dependency        | 72–180     | Vendor monitoring + circuit breakers        |

**Highest investment priority**: Deployment failures and capacity failures account for ~60% of SEV1/SEV2 incidents in most organizations. Start there.
