# Executive SITREP Playbook

> Governs preparation and delivery of Situation Reports (SITREPs) to executive stakeholders during sustained incidents.
> Owner: Incident Commander. Drafter: Comms Lead. Maximum prep time per SITREP: 3 minutes.

---

## Section 1 — SITREP Cadence Rules

| Severity | First SITREP | Subsequent Cadence | Mitigation SITREP | Resolution SITREP |
|---|---|---|---|---|
| **SEV1** | T+30min | Every 30 minutes until customer impact = 0 | Immediately when impact reaches zero | When root cause is addressed and incident closed |
| **SEV2 with executive exposure** | T+60min | Every 60 minutes until customer impact = 0 | Immediately when impact reaches zero | When root cause is addressed and incident closed |
| **SEV2 no executive exposure** | IC discretion | IC discretion | IC discretion | IC discretion |

**Hard rules:**

- [ ] Never let a SITREP go out more than 60 minutes late. If a full update is not ready, send a "still working" ping: `[HH:MM TZ] SEV[X] — [Service] — Still investigating. Next update at [HH:MM].`
- [ ] One SITREP to all executive recipients simultaneously. Never send different content to different people.
- [ ] The Comms Lead drafts. The IC approves. Nobody else edits the SITREP content.
- [ ] SITREPs go out even when the news is bad — especially when the news is bad.

---

## Section 2 — SITREP Format (4-Line Rule)

The SITREP has exactly 4 lines. No preamble. No signature. No technical context block.

```
[HH:MM TZ] SEV[X] — [Service] — IC: [Name]
Impact: [N customers / which journeys / revenue rate] — [growing / stable / improving]
Status: [What is being done right now — one sentence, present tense]
ETA: [Specific time, or "No ETA — next update at HH:MM TZ"]
```

---

### Worked Examples

**Example 1 — Declaration SITREP (T+30)**

```
[14:32 PT] SEV1 — Checkout Service — IC: Priya Mehta
Impact: ~40,000 customers unable to complete purchase — all payment methods failing — stable, not growing
Status: Engineering has identified a failed certificate renewal as the likely cause and is applying the fix now
ETA: Mitigation expected by 15:00 PT — next update at 15:00 PT if not resolved
```

---

**Example 2 — T+30 Update (no change in status)**

```
[15:00 PT] SEV1 — Checkout Service — IC: Priya Mehta
Impact: ~40,000 customers still unable to complete purchase — stable
Status: Certificate replacement is deployed to 2 of 6 regions; rolling forward, no regression observed
ETA: Full mitigation expected by 15:30 PT — next update at 15:30 PT
```

---

**Example 3 — Pivot SITREP (approach changed)**

```
[15:30 PT] SEV1 — Checkout Service — IC: Priya Mehta
Impact: ~40,000 customers still affected — certificate fix did not fully resolve; payment gateway latency remains elevated
Status: Rolling back certificate change; pursuing alternate mitigation via payment gateway failover to secondary provider
ETA: No ETA on this approach — next update at 16:00 PT
```

---

**Example 4 — Mitigation SITREP**

```
[16:08 PT] SEV1 — Checkout Service — IC: Priya Mehta
Impact: Customer impact is ZERO as of 16:06 PT — all payment journeys processing normally
Status: Failover to secondary payment gateway is holding; primary gateway investigation continues separately
ETA: Incident declared mitigated — PIR scheduled for 2026-07-03, results shared within 5 business days
```

---

**Example 5 — Resolution SITREP**

```
[18:45 PT] SEV1 — Checkout Service — IC: Priya Mehta
Impact: Zero — no customer-facing impact since 16:06 PT
Status: Root cause confirmed (expired wildcard cert, automation failure); permanent fix deployed; monitoring nominal
ETA: Incident closed. PIR with action items delivered to this group by 2026-07-08.
```

## Section 3 — Verbal Delivery (Bridge Briefing)

When an executive joins the incident bridge, the IC takes **90 seconds maximum**. Use this script structure:

```
Situation:  "[Service] has been degraded since [time]. [N] customers are affected. [Impact description in one sentence.]"

Action:     "We are currently [what the team is doing right now]. We changed our approach at [time] because [one-sentence reason if applicable]."

Request:    "[State the decision needed as a yes/no question, or say 'No decision needed from you — I'm keeping you informed.']"
```

**Bridge rules:**

- [ ] IC speaks. Comms Lead takes notes. Engineers do not address executives directly unless asked.
- [ ] If an executive asks a technical question, IC answers at the highest level of abstraction that is still accurate.
- [ ] If the answer requires more than 30 seconds, say: "I'll follow up after the bridge — I don't want to pull attention from the live fix."
- [ ] The "Request" section is the most important part. If you need a decision, ask it as a yes/no question: "Do I have your approval to initiate the SLA credit process?" — not "We're thinking about SLA credits."
- [ ] If no decision is needed, say so explicitly: "No action needed from you — I'll update you at [HH:MM]."

## Section 4 — What Executives Actually Need to Decide

**Decisions that ARE executive-level:**

| Decision | Why It's Executive-Level |
|---|---|
| Deliberately take the service down (voluntary outage to prevent worse impact) | Public-facing revenue and reputational risk |
| Initiate Disaster Recovery (failover to DR region/site) | Cost, data risk, and recovery time commitment |
| Trigger SLA credit process for affected customers | Financial and contractual authority |
| Approve emergency vendor spend above the IC's authority limit | Budget authorization |
| Authorize a public statement, status page post, or press response | External communications authority |
| Extend an incident beyond normal staffing (overnight rotation authorization) | Resource and cost authority |

**Decisions that are NOT executive-level — IC has full authority:**

| Decision | Why It Stays with IC |
|---|---|
| Roll back a deployment | Operational; IC is accountable for mitigation |
| Page additional engineers or SMEs | Resourcing within the incident team |
| Escalate within engineering (e.g., bring in the DB team) | Operational escalation path |
| Execute steps in an approved runbook | Pre-authorized procedure |
| Adjust monitoring thresholds during an incident | Operational tuning |
| Declare severity level changes | IC's operational call |

Do not ask executives to weigh in on items in the second table. It wastes bridge time and erodes confidence in the IC.

## Section 5 — SITREP Anti-Patterns

| Anti-Pattern | Why It Fails | What to Do Instead |
|---|---|---|
| "It's complicated" | Executives interpret this as the IC not understanding the problem | Simplify: "We don't have root cause yet. We do have a working mitigation." |
| Embedding a question inside the SITREP text | Questions buried in updates get missed or misread | Call the decision out directly: ask it on the bridge or in a separate DM |
| Sending a SITREP with no ETA and no next-update time | Creates an information vacuum; executives will fill it with assumptions | Always close with either a specific ETA or an explicit next-update time |
| Technical jargon without translation | Executives cannot act on what they cannot understand | One plain-English sentence after any technical term: "cert renewal (the credential that authenticates our service) failed" |
| Going silent when things are going poorly | This is when stakeholders are most anxious; silence reads as loss of control | Increase cadence under uncertainty, not decrease it; a "still working" ping takes 30 seconds |
| Sending different SITREPs to different executives | Creates conflicting narratives; executives compare notes | One message, all recipients, every time |
| Asking for a decision that is not theirs to make | Wastes bridge time; signals the IC is not in control | Review Section 4 before every bridge — know what you own |
| Updating the SITREP format mid-incident | Recipients start comparing formats instead of reading content | Lock the format at SEV declaration; never change it mid-incident |

## Section 6 — SITREP Log Template

Log every SITREP in the incident channel. One row per send. Do not edit prior rows.

| # | Time (TZ) | Recipient(s) | Delivery Method | Key Message (one line) | Next Update Scheduled |
|---|---|---|---|---|---|
| 1 | | | Slack DM / Bridge verbal / Email | | |
| 2 | | | | | |
| 3 | | | | | |
| 4 | | | | | |
| 5 (Mitigation) | | | | Impact = 0 as of [time] | PIR date |
| 6 (Resolution) | | | | Incident closed | PIR delivery date |

**Log owner**: Comms Lead
**Log location**: Incident Slack channel, pinned message
**Retention**: Log is included as an appendix in the PIR document
