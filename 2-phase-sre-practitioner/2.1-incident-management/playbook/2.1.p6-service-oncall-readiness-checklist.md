# Service On-Call Readiness Checklist

> Pre-flight gate before any service enters the production on-call rotation.
> A failed gate is not a blocker — it is a remediation assignment with a deadline.

## Gate 1 — Observability

> On-call responders cannot debug what they cannot see. Every item here must exist and be verified before paging a human being.

- [ ] Service emits request rate, error rate, and latency (RED metrics) — dashboard link: ___
- [ ] Service emits saturation metrics: CPU, memory, connection pool depth, queue depth
- [ ] All P0/P1 customer journeys have synthetic monitors running every 60 seconds
- [ ] Log verbosity is appropriate: INFO for normal ops, ERROR for actionable events — no DEBUG in production
- [ ] Distributed tracing is enabled for all cross-service calls
- [ ] All dashboards are accessible without VPN

**Gaps**: _(list any unchecked items and owner)_

## Gate 2 — Alerting

> Alerts that fire too often get ignored. Alerts without runbooks create responder paralysis. Both are worse than no alert.

- [ ] Alert exists for each RED metric exceeding SLO thresholds
- [ ] Every alert has a runbook link in its annotation
- [ ] Alert thresholds are calibrated — no alert fires more than 3 times/week without documented investigation
- [ ] On-call team receives alerts via PagerDuty — not email or Slack alone
- [ ] After-hours alert routing is tested (someone received a page in a staging fire drill or equivalent)
- [ ] Alert-to-runbook mapping is documented

**Gate 2 Result**: Pass / Fail

## Gate 3 — Runbooks

> A runbook that has never been tested by someone other than its author is a hypothesis, not a procedure.

- [ ] A runbook exists for every alert that pages on-call
- [ ] Each runbook includes: symptom, diagnosis steps, mitigation options, rollback procedure
- [ ] Runbooks are accessible without VPN and without production access
- [ ] Runbooks have been tested by a team member who did NOT write them
- [ ] Runbook last-updated date is within 90 days

**Gate 3 Result**: Pass / Fail

## Gate 4 — Operational Contacts

> At 2am, the on-call responder needs to know who to call. This gate ensures that list exists before the incident.

- [ ] Primary on-call and secondary on-call are defined in PagerDuty with current schedules
- [ ] Service owner (escalation path above on-call) is documented with contact method
- [ ] Upstream dependency owners are documented — who to engage if the database team, messaging layer, or auth service is involved
- [ ] Downstream service owners are documented — who must be notified if this service degrades
- [ ] Vendor escalation contacts are documented (mark N/A if no vendor dependencies)

**Gate 4 Result**: Pass / Fail

## Gate 5 — Deployment and Rollback

> If you cannot roll back in under 10 minutes, you cannot mitigate a bad deploy under pressure. Test it before you need it.

- [ ] Service has a one-command rollback procedure — command is documented in the runbook
- [ ] Rollback has been executed successfully in staging within the last 30 days
- [ ] Deployment pipeline includes a canary or staged rollout gate before full production promotion
- [ ] Feature flags exist for any risky new functionality
- [ ] Deployment history is visible in the monitoring dashboard as deployment markers on graphs

**Gate 5 Result**: Pass / Fail

## Gate 6 — SLO and Error Budget

> An SLO set without historical data is a guess. A guess is not an on-call contract.

- [ ] SLO is defined and documented — target, measurement method, dashboard link: ___
- [ ] Error budget policy is filled out and approved — link: ___
- [ ] Error budget dashboard is accessible without production access
- [ ] SLO has been baselined against at least 30 days of historical data — not set aspirationally

## Gate 7 — Incident History

> Inheriting a service with unresolved PIR actions is inheriting a known debt. Document it or close it — never hide it.

- [ ] All prior PIR action items are resolved, or have assigned owners and due dates within the current sprint
- [ ] Known flaky behaviors are documented (example: "service restarts every ~10 days — monitored, root cause ticket: ___")
- [ ] Chaos or Game Day test performed within the last 6 months — or explicitly marked N/A for new services with explanation

## Failure Handling

**If any gate fails:**

1. The service does **not** enter the production on-call rotation.
2. The reviewer issues a remediation plan listing each failed item, assigned owner, and due date.
3. The remediation SLA is **2 weeks** for critical gaps (Gates 1, 2, 3). Gate 4–7 gaps may be tracked as P2 with a 30-day SLA at reviewer discretion.
4. Once all gaps are resolved, a re-review is scheduled. Only the failed gates need re-evaluation.
5. The service enters on-call rotation only after the reviewer signs off on all gates.

**Critical gates (automatic block regardless of other pass/fail status):**
- Gate 1: any RED metric missing
- Gate 2: any alert without a runbook link
- Gate 3: any runbook that has not been independently tested
- Gate 5: rollback procedure undocumented or untested
