# Action Item Tracker Template
**Purpose**: Long-term tracking of incident-driven action items from PIR to closure. A single source of truth across all open incidents.

---

## PER-ACTION-ITEM FORMAT

Use one row per action item in the master tracker. Fields are non-negotiable — incomplete rows are treated as open regardless of stated status.

| Field              | Description                                                                          |
|--------------------|--------------------------------------------------------------------------------------|
| **ID**             | Format: `AI-[INC number]-[sequence]`. Example: `AI-20451-001`                       |
| **Incident**       | INC number + one-line description. Example: `INC-20451 — Auth pool exhaustion`       |
| **Action**         | Specific, concrete action. Not "improve monitoring" — "Add Datadog alert on auth-service pool > 80%" |
| **Owner**          | Single named person. Not a team. If team, name the DRI.                             |
| **Due Date**       | Specific date. Not "sprint" or "Q4."                                                |
| **Priority**       | P0 / P1 / P2 / P3 (see Priority Scoring below)                                     |
| **Status**         | Open / In Progress / Blocked / Done / Cancelled                                     |
| **Verification**   | How we know it's done — specific, measurable criterion. Not "engineer confirms complete." |
| **Blocked By**     | What is blocking — ticket number or dependency. Blank if unblocked.                 |

## PRIORITY SCORING

Priority is set at PIR, reviewed at each weekly standup, and can only be lowered (not raised) by the owning engineering manager.

| Priority | Definition                                                                 | Expected Completion |
|----------|----------------------------------------------------------------------------|---------------------|
| **P0**   | Directly prevents recurrence of a SEV1 or SEV2 failure mode               | Within 7 days of PIR|
| **P1**   | Reduces blast radius or cascading impact if the failure recurs             | Within 30 days      |
| **P2**   | Improves detection speed (MTTD) or response speed (MTTR)                  | Within 60 days      |
| **P3**   | Operational improvement — reduces toil, improves observability, tech debt  | Within 90 days      |

**Priority override**: Any P0 item that has not been completed within its SLA requires a written exception from the Engineering Director, with an updated completion date and risk acceptance statement.

## STALENESS PROTOCOL

Applies automatically to all items regardless of stated status.

| Age          | Trigger                        | Reviewer           | Required Action                                                |
|--------------|--------------------------------|--------------------|----------------------------------------------------------------|
| > 30 days open | Any item open > 30 calendar days | Incident Commander | IC reviews at weekly standup; must get updated date or escalate |
| > 60 days open | Any item open > 60 calendar days | Engineering Manager | EM reviews; either accepts risk in writing or closes item       |
| > 90 days open | Any item open > 90 calendar days | Director of Engineering | Director reviews; item is either completed, formally cancelled with rationale, or escalated to VP |

**Staleness does not mean automatic cancellation.** It means the item must be actively reviewed and a decision made. Silence is not an acceptable response.

## CLOSURE CRITERIA BY ACTION TYPE

An action item is **Done** only when the verification criterion is met and confirmed — not when the engineer says it is done.

| Action Type                       | Closure Requires                                                                  |
|-----------------------------------|-----------------------------------------------------------------------------------|
| **Alert added**                   | Alert is active in production; APR (Alert Precision Rate) validated over minimum 2 weeks; linked to PagerDuty policy |
| **Runbook written / updated**     | Runbook reviewed by a second SRE who was not involved in writing it; linked to relevant alert; tested in a game day or walkthrough |
| **Code change deployed**          | Change deployed to production AND verified via the runbook's verification section; no regression in error rate |
| **Config / infrastructure change**| Change deployed; load tested in staging to confirm behavior under peak traffic; documented in runbook |
| **Process change**                | Policy documented in engineering wiki; communicated to all teams affected; compliance spot-checked by EM |
| **Architectural change**          | Design reviewed; implemented; tested in staging; deployed to production; post-deploy health check completed |
| **Training / documentation**      | Document published; relevant team members acknowledge read; dated confirmation recorded |

## WEEKLY REVIEW TEMPLATE (5-MINUTE STANDUP FORMAT)

Run weekly in the SRE team meeting. IC or SRE lead facilitates. Takes no more than 5 minutes if done consistently.

```
Weekly Incident Action Item Review
Date: [YYYY-MM-DD]
Facilitator: [name]

1. OVERDUE (30+ days) — [N items]
   For each:
   - AI-XXXXX-NNN: [action] — [owner] — [new date or escalate?]

2. DUE THIS WEEK — [N items]
   For each:
   - AI-XXXXX-NNN: [action] — [owner] — [on track / at risk?]

3. STATUS CHANGES SINCE LAST WEEK
   - [AI-ID]: [old status] → [new status] — [brief reason]

4. NEW ITEMS (from incidents since last review)
   - [AI-ID]: [action] — [owner] — [due date] — [priority]

5. BLOCKERS
   - [AI-ID]: blocked by [what] — [who owns unblocking?]

Next review: [date]
```
