**Principle**: Every step must have an expected output. If the output doesn't match, say where to go next.

# COMPLETED EXAMPLE — Auth Service: Connection Pool Exhaustion

## RUNBOOK HEADER

```
Service       : auth-service
Failure Mode  : Connection Pool Exhaustion
Owner         : Platform Engineering (on-call: PagerDuty policy "auth-oncall")
Last Reviewed : 2025-11-21
Linked Alerts : Datadog Monitor: "auth-service: connection pool utilization > 80%" (Monitor ID: 8812234)
                PagerDuty: P1 escalation at >95% utilization for 2 min
Severity Floor: SEV2 (escalate to SEV1 if checkout error rate exceeds 10%)
Related Runbooks: auth-service-pod-crashloop.md | checkout-service-degraded.md
```

---

## SYMPTOMS

### What the alert looks like
> Datadog alert: "auth-service db_connection_pool_utilization > 80% for 2 consecutive minutes" OR PagerDuty P1 fires when utilization exceeds 95% for 2 minutes. Secondary trigger: checkout-service error rate P1 (checkout-service errors come downstream of this failure mode within 1–3 minutes).

### What dashboards show
> - **Datadog > auth-service overview**: `db_connection_pool_active` climbing toward `db_connection_pool_max`; `auth_token_validation_latency_p99` rising above 500ms
> - **Datadog > checkout-service**: `checkout_payment_error_rate` rising (follows auth pool by ~90 seconds)
> - **GitLab deploy timeline** overlay in Datadog: look for a deploy within the last 30 minutes

### What users experience
> Checkout page shows "Payment processing failed. Please try again." for approximately 68% of checkout attempts. Login itself is unaffected for non-checkout flows.

### Blast radius indicators
> - checkout-service: hard failure on payment calls → customers cannot complete purchases
> - order-service: new order creation blocked (depends on auth validation)
> - If pool stays at 100% for >10 minutes: auth latency affects API gateway → broader login degradation

---

## IMMEDIATE ACTIONS — FIRST 5 MINUTES

**Step 1**: Check current connection pool utilization.
```bash
kubectl exec -n auth $(kubectl get pods -n auth -l app=auth-service -o name | head -1) \
  -- curl -s localhost:8080/metrics | grep db_connection_pool
```
Expected output:
```
db_connection_pool_active 98
db_connection_pool_max 100
```
If `db_connection_pool_active / db_connection_pool_max` > 0.80 → this runbook applies. Continue.
If ratio < 0.50 → this is a different failure mode. See [auth-service-pod-crashloop.md] or check auth-service logs directly.

**Step 2**: Check for a recent deploy.
```bash
kubectl rollout history deployment/auth-service -n auth | tail -5
```
Expected output: list of recent revisions with timestamps.
If a deploy happened within the last 60 minutes → **high probability this is the cause.** Go to Mitigation Option A (Rollback).
If no recent deploy → go to Investigation Step I-2.

**Step 3**: Declare incident.
> If pool utilization > 95% AND checkout error rate > 5%: declare SEV1.
> If pool utilization 80–95% AND checkout error rate < 5%: declare SEV2.
> Post to #incidents: "Investigating auth-service connection pool exhaustion. INC-XXXXX open."

**Step 4**: Pull the last 30 minutes of auth-service error logs.
```bash
kubectl logs -n auth -l app=auth-service --since=30m | grep -E "ERROR|pool|timeout|exhausted" | tail -50
```
Expected output: repeated `connection pool exhausted` or `timeout waiting for connection` errors.

**Step 5**: Check if checkout-service is in degraded state.
```bash
curl -s https://internal.checkout-service.svc/health | jq '.dependencies.auth'
```
Expected output: `"status": "degraded"` or `"status": "timeout"`

---

## INVESTIGATION STEPS

### Step I-1: Confirm pool exhaustion root cause

```bash
kubectl exec -n auth $(kubectl get pods -n auth -l app=auth-service -o name | head -1) \
  -- curl -s localhost:8080/metrics | grep -E "pool|connection|thread"
```
Expected output: `db_connection_pool_active` equals `db_connection_pool_max`. If they are equal and both non-zero, pool exhaustion is confirmed.
→ If max is 100 or below → this is likely a config issue. Go to **I-2**.
→ If max is 500+ and pool is still exhausted → traffic surge. Go to **Mitigation Option B (Scale Out)**.

### Step I-2: Compare current config against expected

```bash
kubectl get deployment auth-service -n auth -o jsonpath='{.spec.template.spec.containers[0].env}' \
  | jq '.[] | select(.name=="DB_CONNECTION_POOL_SIZE")'
```
Expected output: `{"name": "DB_CONNECTION_POOL_SIZE", "value": "500"}` (correct)
If value is `100` or lower → config error from recent deploy. **Go to Mitigation Option A (Rollback).**
If value is correct (500) → pool is exhausted under legitimate load. Go to **I-3**.

### Step I-3: Check traffic volume vs. baseline

In Datadog: auth-service > auth_token_validations_per_second (current vs. 1-week avg at same hour).
→ If current RPS > 3× baseline → traffic spike. Go to **Mitigation Option B (Scale Out)**.
→ If RPS is normal → a specific query or client is holding connections open. Escalate to auth team (Lena Schmidt / auth-oncall).

### Step I-4: Check per-client connection distribution

```bash
kubectl exec -n auth $(kubectl get pods -n auth -l app=auth-service -o name | head -1) \
  -- curl -s localhost:8080/metrics | grep db_connection_pool_by_client
```
Expected output: `db_connection_pool_by_client{client="checkout-service"} 98` — shows which client is consuming the pool.
If one client holds >80% of connections → likely a checkout-service bug (not releasing connections). Escalate to Diego Ruiz / checkout team.

---

## MITIGATION OPTIONS

### Option A — Rollback Deploy (Fastest — use if recent deploy is suspected cause)
**Time to take effect**: 8–12 minutes (rollback + drain)
**Risk**: Low
**Authorization required**: On-call can execute independently for config-only deploys. IC sign-off required if rollback includes code changes.

When to use: A deploy happened within the last 60 minutes AND pool size config is wrong.

```bash
# Identify current and previous revision
kubectl rollout history deployment/auth-service -n auth

# Roll back to previous revision
kubectl rollout undo deployment/auth-service -n auth

# Monitor rollout
kubectl rollout status deployment/auth-service -n auth --timeout=5m
```
Expected output: `deployment.apps/auth-service successfully rolled out`

```bash
# Verify pool size returned to expected value
kubectl exec -n auth $(kubectl get pods -n auth -l app=auth-service -o name | head -1) \
  -- curl -s localhost:8080/metrics | grep db_connection_pool_max
```
Expected output: `db_connection_pool_max 500`

**Verification**: Monitor `db_connection_pool_utilization` in Datadog. Expect to see it drop from 100% within 2–3 minutes of rollback. Checkout error rate will begin recovering after pool drains (expect 5–30 minutes for full recovery as active connections close naturally).

**Warning**: The pool does NOT immediately drop to 0% after rollback. Active requests hold connections until they complete. Do not declare the incident resolved until pool utilization has been stable below 60% for 5 minutes.

---

### Option B — Emergency Scale Out
**Time to take effect**: 3–5 minutes
**Risk**: Low–Medium (cost spike; verify instance limits)
**Authorization required**: On-call can execute independently up to 3× baseline replicas. IC sign-off for >3×.

When to use: No recent deploy AND traffic is 3× above normal. Pool exhaustion is due to volume, not config.

```bash
# Check current replica count
kubectl get deployment auth-service -n auth

# Scale to 3× current (replace N with 3× current count)
kubectl scale deployment auth-service -n auth --replicas=N

# Monitor pod startup
kubectl rollout status deployment/auth-service -n auth --timeout=5m
```
Expected output: New pods reach `Running` state. Pool pressure distributed across more instances.

```bash
# Verify improvement
kubectl exec -n auth $(kubectl get pods -n auth -l app=auth-service -o name | head -1) \
  -- curl -s localhost:8080/metrics | grep db_connection_pool_utilization
```
Expected: utilization drops below 60%.

---

### Option C — Enable Circuit Breaker (Protective Mode)
**Time to take effect**: Immediate
**Risk**: Medium (checkout will return graceful errors rather than hanging — customer experience degrades intentionally)
**Authorization required**: IC sign-off required.

When to use: Options A and B have not resolved the issue within 20 minutes; pool is still at 100%; checkout is still experiencing hard failures. Enabling the circuit breaker causes checkout to return a friendly "try again" message rather than hanging.

```bash
# Set circuit breaker flag via feature flag service
curl -X POST https://internal.featureflags.svc/flags/auth_circuit_breaker \
  -H "Content-Type: application/json" \
  -d '{"enabled": true, "threshold_pct": 50, "reset_after_sec": 30}'
```
Expected output: `{"status": "updated", "flag": "auth_circuit_breaker", "enabled": true}`

This trades a hard payment failure (~69% of requests) for a graceful "service temporarily unavailable" message. Customer impact is similar but the user experience is better and retries are managed.
