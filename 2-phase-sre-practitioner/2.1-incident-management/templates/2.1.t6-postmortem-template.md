# Post-Incident Review (PIR) Template
**Purpose**: Structured blameless retrospective. Goal is systemic learning, not individual accountability.
**Timing**: SEV1 within 48h; SEV2 within 5 business days; SEV3 within 2 weeks.

---

# COMPLETED EXAMPLE — Auth Service Timeout Cascade (INC-20451)

## HEADER

```
Incident ID   : INC-20451
Date of PIR   : 2025-11-16
Severity      : SEV1
Duration      : 1h 04m (14:29 PST to 15:33 PST)
IC            : Priya Mehta
PIR Facilitator: Kenji Watanabe (Senior SRE — not on incident)
Participants  : Priya Mehta (IC), Diego Ruiz (TL), Anika Patel (Scribe),
                Sam Torres (Comms), Marcus Webb (Customer Success),
                Lena Schmidt (Auth Team), Raj Iyer (Platform Engineering)
Document owner: Kenji Watanabe
Status        : Final
```

---

## EXECUTIVE SUMMARY

> At 14:29 PST on 2025-11-14, a configuration change deployed to auth-service at 14:15 PST reduced the connection pool limit from 500 to 100, which under normal Friday-afternoon traffic caused pool exhaustion within 14 minutes, resulting in auth token validation timeouts that checkout-service surfaced as hard payment failures rather than retryable errors.
> The incident affected ~12,200 consumer checkout sessions (~68% of active sessions), generated approximately 463,600 customer minutes of downtime, and caused an estimated $127,000 in lost revenue over 64 minutes.
> The configuration change was rolled back at 15:21 PST and checkout success rate returned to baseline by 15:33 PST; all enterprise/B2B customers were unaffected.

## ROOT CAUSE ANALYSIS

**Framework used**: 5 Whys + CDR

### CDR Output

| Layer              | Finding                                                                                    |
|--------------------|--------------------------------------------------------------------------------------------|
| **Direct Cause**   | auth-service connection pool exhausted under normal traffic load                           |
| **Contributing Causes** | (1) Deploy #4421 reduced pool_size from 500 to 100 — untested value; (2) checkout-service treated auth timeout as non-retryable hard failure; (3) No alert on connection pool utilization > 80% |
| **Root Cause**     | Config changes to shared infrastructure services bypass performance validation; pool_size is not tested as part of auth-service CI pipeline |

### 5 Whys Chain

1. Why did checkout fail for 68% of users? → Auth token validation timed out
2. Why did auth validation time out? → Auth-service connection pool was exhausted
3. Why was the pool exhausted? → pool_size config was set to 100; Friday traffic saturated it
4. Why was pool_size set to 100? → Deploy #4421 changed it — this was a copy-paste error from a dev environment config
5. Why did a dev-environment value reach production? → **Auth-service has no CI test for connection pool sizing; config changes bypass load validation**

### Contributing Factor Summary

| Factor                                                   | Category  | Weight       |
|----------------------------------------------------------|-----------|--------------|
| No CI test for connection pool capacity                  | Technical | Primary      |
| Checkout-service: auth timeout treated as hard failure   | Technical | Contributing |
| No alert on pool utilization > 80%                       | Process   | Contributing |
| Config change not flagged for review by platform team    | Process   | Contributing |

## DETECTION GAP

```
MTTD : 3 minutes
  Impact start   : 14:29 PST
  Alert fired    : 14:32 PST (PagerDuty P1 on checkout error rate)
  Gap            : 3 minutes
```

**Why didn't we detect it sooner?**
> The checkout error rate alert fires after 3 consecutive minutes above threshold — this is intentional to reduce false positives, but means 3 minutes of customer impact before the first alert. No upstream alert existed on auth-service connection pool utilization.

**What would have reduced MTTD?**
- Alert on auth-service connection pool utilization > 80% (would have fired at 14:28 — before checkout degraded)
- SLO burn rate alert (1h burn rate) would fire within 1 minute of impact rather than 3

---

## RESPONSE GAP

```
MTTR : 64 minutes
  Impact start          : 14:29 PST
  Customer impact = 0   : 15:33 PST
  Gap                   : 64 minutes

Breakdown:
  Detection lag         : 3 min  (14:29 → 14:32)
  Acknowledgement       : 5 min  (14:32 → 14:37 declaration)
  Diagnosis             : 5 min  (14:37 → 14:42 hypothesis)
  Decision + rollback   : 11 min (14:42 → 14:53 rollback initiated)
  Rollback execution    : 11 min (14:53 → 15:04 rollback complete)
  Connection drain      : 29 min (15:04 → 15:33 full recovery)
```

**Why did MTTR take as long as it did?**
> The rollback was faster than expected. The 29-minute connection drain was the dominant factor — active requests held connections open even after pool_size was restored, and checkout-service did not have a circuit breaker to shed load during recovery. This was a natural drain time, not an operational error.

**What would have reduced MTTR?**
- Circuit breaker on checkout → auth calls (would have begun recovery 15–20 minutes faster during drain)
- Pre-built runbook for "auth connection pool exhaustion" (diagnosis would have been 2 min, not 5)
- Automated rollback trigger on pool utilization > 95% (would have fired at 14:28, mitigation ~20 min earlier)

---

## WHAT WENT WELL

- Diego identified the deploy-to-pool correlation in 5 minutes using Datadog's deployment markers overlaid on the pool metric — this was the right tool and it worked.
- Role assignments were complete within 2 minutes of declaration; all four roles answered the bridge within 3 minutes. Comms and Scribe freed IC and TL to focus entirely on diagnosis.
- Sam posted a clear status page update within 9 minutes of declaration, ahead of social media escalation — held reputational impact at "Medium" rather than "High."
- The GitLab rollback workflow executed without issues; no manual intervention was needed in the deployment pipeline.
- Marcus Webb identified that B2B customers were unaffected within 10 minutes and proactively reached out to the top 10 enterprise accounts — zero enterprise escalations during the incident.

---

## WHAT DIDN'T GO WELL

- No alert existed on auth-service connection pool utilization — the first signal was a checkout error rate spike downstream, not the root cause upstream. We were diagnosing symptoms, not the cause.
- checkout-service has no circuit breaker for auth calls — it treated a transient resource exhaustion as a permanent failure, preventing graceful degradation. A retry with exponential backoff would have reduced customer impact by ~30%.
- The config change in deploy #4421 was not reviewed by a second engineer — single-approval deploys to shared infrastructure services are a risk pattern we need to close.
- The 29-minute recovery drain was not anticipated during diagnosis — the team expected recovery within 5 minutes of rollback completion. Incorrect mental model led to premature status page "resolving" message that had to be walked back.
- No runbook existed for "auth connection pool exhaustion" — Diego diagnosed from first principles, which worked, but added ~3 minutes to MTTR.
