## Section 2 — Budget Milestones and Actions

Budget remaining is calculated as: `(remaining_budget_minutes / total_budget_minutes) * 100`

| Budget Remaining | Status | Required Actions | Owner |
|---|---|---|---|
| **> 75%** | Normal | No action required. Standard on-call cadence. | On-call SRE |
| **50% – 75%** | Monitoring | Flag status at weekly SRE sync. Verify burn rate is not accelerating. No sprint changes required. | SRE Partner |
| **25% – 50%** | Elevated | Engineering Manager notified within 24 hours. At least one reliability work item added to sprint backlog at P1 priority. Burn rate reviewed at each sprint planning. | EM + SRE |
| **0% – 25%** | Critical | Feature freeze on this service (see Section 3). VP Engineering notified. Reliability sprint activated. Daily budget review at standup. | EM + VP Eng + SRE Manager |
| **0% (Exhausted)** | Policy Halt | All non-critical feature work halted by policy. Incident review triggered within 48 hours. No new feature deployments until budget is restored above 10%. | EM + VP Eng + SRE Manager |

## Section 3 — Feature Freeze Definition

A feature freeze triggered by error budget exhaustion is **not a complete deployment halt**. The following rules apply precisely.

### Blocked at 0% Budget

- [ ] No new user-facing features may be deployed
- [ ] No schema migrations that increase risk surface
- [ ] No third-party integrations or dependency upgrades unless security-mandated
- [ ] No configuration changes that alter request routing or traffic shaping
- [ ] No A/B test activations or flag rollouts for net-new functionality

### Still Permitted at 0% Budget

- [ ] Emergency fixes for active incidents (IC must approve each deployment)
- [ ] Security patches (any severity — these bypass the freeze without exception)
- [ ] Reliability improvements directly targeting the SLO (the goal of the freeze)
- [ ] Observability-only changes (adding metrics, dashboards, alerts — no functional change)
- [ ] Rollbacks to a previously stable version

### Freeze Lift Criteria

The freeze lifts automatically when the rolling 30-day budget is restored above **10%**. The SRE partner confirms this in the error budget tracker and notifies the Engineering Manager. No approval ceremony required — the policy gates on the metric, not a meeting.

## Section 4 — Budget Recovery Tracking

### Incident Attribution

Every incident that consumes error budget must be recorded in the budget tracker with the following fields:

| Field | Required |
|---|---|
| Incident ID | Yes |
| Incident start time (UTC) | Yes |
| Incident end time (UTC) | Yes |
| Duration (minutes) | Yes |
| SLO impact (% of budget consumed) | Yes |
| Root cause category | Yes |
| Caused by external dependency? | Yes — see credit-back rules below |

### External Dependency Credit-Back

If an incident's root cause is confirmed to be a failure of an **external dependency outside the team's control** (e.g., cloud provider outage, upstream API failure), the Engineering Manager may request a credit-back of the consumed budget.

Credit-back requires:
1. Post-incident review confirms external root cause (not shared fault)
2. The external provider acknowledges the incident in their status page or support ticket
3. SRE Manager approves the credit in the budget tracker
4. The credit is noted in the next quarterly policy review

Credit-back does not restore budget retroactively — it adjusts the denominator for the current rolling window calculation.

## Section 5 — Exception Process

A policy exception allows a team to deploy features or skip a freeze requirement when the business case overrides reliability risk.

### Who May Request

Engineering Manager for the service, with Product Manager co-sign.

### Approval Required

- [ ] Engineering Director (skip-level above EM)
- [ ] SRE Manager

Both approvals are required. Either approver may reject.

### Exception Request Format

Submit to `#sre-exceptions` Slack channel with the following fields:

```
Service:            [name]
Current budget:     [X% remaining]
Requested action:   [what you want to deploy]
Business justification: [specific, time-bound reason — not "because product wants it"]
Risk assessment:    [what could go wrong; what's the mitigation]
Duration:           [how long the exception applies — max 7 days]
Rollback plan:      [exact steps if the deployment makes the budget worse]
```

### Exception Constraints

- Maximum duration: **7 calendar days** per exception
- The exception is automatically void if the budget drops below the level at time of approval
- Exceptions are logged in the quarterly policy review
- Back-to-back exceptions without a reliability sprint in between are not permitted

---

## Section 6 — Policy Review Cadence

This policy is reviewed **quarterly**, or immediately after any of the following triggers:

- The budget is exhausted in two consecutive rolling windows
- The SLO target is changed
- The service architecture changes in a way that affects what "good" means
- A post-incident review recommends tightening thresholds

### What to Evaluate at Each Review

- [ ] Is the SLO target still calibrated to actual customer expectations?
- [ ] Are the milestone thresholds triggering the right behaviors (too early / too late)?
- [ ] Are the action owners still correct?
- [ ] Have any credit-back requests been adjudicated and documented?
- [ ] Have any exceptions been granted — were they justified in retrospect?
- [ ] Should the SLO be tightened or relaxed based on rolling-window history?

### Review Owner

SRE Manager facilitates. Engineering Manager for the service is required attendee. Product Manager optional.

| **SLO Target** | 99.95% availability |
| **Error Budget** | 0.05% = 21.9 minutes/month |
| **Measurement Window** | Rolling 30 days |
| **Policy Owner** | Priya Nair, Payments Engineering Manager |
| **SRE Owner** | Marcus Webb, Payments SRE |
| **Last Reviewed** | 2026-04-01 |
| **Next Review Due** | 2026-07-01 |

**SLI Definition**: A request is "good" if the `/v2/payments` endpoint returns HTTP 200 or 201 within 500ms. Any 5xx response or timeout is a bad request. 4xx responses are excluded (client errors, not service failures).

**Metric**: `sum(rate(http_requests_total{service="payments-api", status_code=~"2.."}[5m])) / sum(rate(http_requests_total{service="payments-api"}[5m]))`

**Budget arithmetic**:
```
SLO: 99.95%
Budget: 0.05% of 30-day window
= 0.0005 * 43,200 minutes
= 21.9 minutes/month
= ~43 seconds/day average allowance
```

**Milestone thresholds at this budget**:

| Budget Remaining | Minutes Remaining | Trigger |
|---|---|---|
| > 75% | > 16.4 min | Normal |
| 50% – 75% | 10.9 – 16.4 min | Flag at weekly sync |
| 25% – 50% | 5.5 – 10.9 min | P1 reliability backlog item |
| 0% – 25% | 0 – 5.5 min | Feature freeze + daily review |
| 0% | 0 min | Policy halt + incident review |

**Exception history (Q1 2026)**: One exception granted 2026-03-12 for PCI compliance tokenization rollout. Approved by Director Chen + SRE Manager Webb. Duration 5 days. Budget at time of exception: 8.2 min remaining. No additional budget was consumed during the exception window. Closed clean.
