## Part 1: Concept — Prometheus Data Model to Alertmanager Routing (30 min read)

### The Mental Model: From Observability Data to "Who Gets Paged?"

At T-Mobile's 25M msg/day notification platform, you're capturing *events* (message sent, delivery attempted, timeout). An SRE leader's job is:

1. **Collect** the right metrics (instrumentation)
2. **Aggregate** them in a queryable time-series database (Prometheus)
3. **Rule** on the aggregates (recording rules = "saved searches", alert rules = "triggers")
4. **Route** the alerts intelligently (Alertmanager = smart SIEM queue)
5. **Respond** based on SLO breach velocity, not static thresholds

This is the **observability → incident response pipeline**. You'll own it as a Staff/Principal SRE.

---

### A. Prometheus Data Model — The 4-Tuple Advantage Over Splunk Logs

**Prometheus stores metrics as time series**. Each entry is:

```
METRIC_NAME{label_1="value_1", label_2="value_2"} VALUE TIMESTAMP
```

**Example from a notification service**:
```
notification_delivery_duration_seconds{pod="notif-service-3", region="us-east-1"} 0.245 1718054400000
```

**Why this 4-tuple is superior to Splunk log lines**:

| Aspect | Prometheus 4-Tuple | Splunk Raw Logs |
|--------|-------------------|-----------------|
| **Query speed** | O(1) label index; nanosecond aggregation | O(n) scan; seconds to aggregate |
| **Cardinality safety** | Explicit label schema prevents explosion | Unbounded field keys from JSON |
| **Disk footprint** | ~2 bytes/sample (compressed); 1yr = 52 MB | 100+ bytes/event; 1yr = 100+ GB |
| **Aggregation** | Native `rate()`, `sum(by(...))` — CPU cheap | Requires PowerSearch optimization |
| **Memory** | In-process query engine; 2 GB keeps 1M series | Distributed indexing; 10+ GB clusters |
| **Downside** | Only numeric + string labels; no arbitrary JSON | Full-text search; structured + unstructured |

**Prometheus is *not* a logs database. It's a metrics database.** Use it for SLO/alerting; use Splunk for root cause (post-incident).

---

### B. PromQL: The "SELECT/WHERE for Metrics"

You write PromQL queries the same way you'd write a SELECT in Splunk SPL, but Prometheus is *columnar and typed*.

**Core PromQL functions and Splunk equivalents**:

| PromQL | What It Does | Splunk Equivalent |
|--------|-------------|------------------|
| `rate(metric[5m])` | Increase per second over past 5 min (derivative) | `stats avg(rate) by _time` (approximate) |
| `sum(metric)` | Sum all matching time series | `stats sum(field)` |
| `sum(metric) by (label)` | Sum, then group by label value | `stats sum(field) by label` |
| `histogram_quantile(0.99, metric)` | 99th percentile from histogram buckets | `stats perc99(latency)` |
| `predict_linear(metric[1h], 3600)` | Linear extrapolation (useful for capacity) | `predict` command (rare) |
| `absent(metric)` | True if metric has no data (scrape failure) | Requires custom search |

**Example query for the notification platform**:
```promql
# Success rate: (successful deliveries) / (total attempts) over 5 min
(
  sum(rate(notifications_delivered_total[5m]))
  /
  sum(rate(notifications_attempted_total[5m]))
)
```

This is like:
```spl
index=notification | stats sum(delivered) as delivered, sum(attempted) as attempted 
| eval success_rate = delivered / attempted
```

But Prometheus computes it in **milliseconds** without touching Splunk.

### C. Recording Rules: Splunk's "Scheduled Saved Search" in Prometheus

A **recording rule** is a saved PromQL query that runs every evaluation interval (e.g., 30s). The result is *stored as a new metric*.

**Why it matters**: Dashboards are faster; pre-aggregated data is cheaper.

**Example recording rule** (our notification platform):
```yaml
- alert: NotificationDeliverySuccess
  expr: |
    (
      sum(rate(notifications_delivered_total[5m]))
      /
      sum(rate(notifications_attempted_total[5m]))
    ) * 100
  labels:
    slo: "delivery_slo"
  annotations:
    summary: "Success rate: {{ $value }}%"
```

**Translation to Splunk workings**:
- Splunk: Create saved search `get_delivery_success_rate`, schedule it every 30s, write output to summary index.
- Prometheus: Define recording rule, store result as `notifications_delivery_success:ratio_5m`, query it instantly.

**Prometheus wins on cost and query latency**; Splunk wins on audit trail and ad-hoc debugging.

### D. Alerting Rules: "Fire When SLO Budget Burns Too Fast"

An **alert rule** evaluates a PromQL expression. If true for `for` duration, it fires.

**Structure**:
```yaml
alert: MyAlertName
expr: <PromQL condition>
for: 5m                    # Must be true for 5 min before firing (debounce)
labels:
  severity: critical       # human-readable urgency
  team: platform           # routing hint for Alertmanager
annotations:
  summary: "..."
  runbook: "https://..."   # SRE playbook link
  dashboard: "https://..." # Grafana dashboard for context
```

**Three golden rules for alert rules**:

1. **Threshold = SLO burn rate, not latency percentile**. Bad:
   ```promql
   histogram_quantile(0.99, latency) > 500ms
   ```
   Good:
   ```promql
   (1 - (success_rate / target_slo)) > 0.14  # 14.4x burn in 1-hour window
   ```

2. **Two-window strategy prevents noise**:
   - Fast burn (1h window): page the oncall immediately if 14.4x burn detected
   - Slow burn (30d window): page if on-track to exhaust budget by month-end
   
   This avoids spam (slow burn) and catches real outages (fast burn).

3. **Include context in annotations**: Don't just fire an alert; include the dashboard link, the runbook, and the current value so the oncall doesn't waste time context-switching.

### E. Alertmanager: The "Smart Alert Router"

**Alertmanager** groups, deduplicates, and routes alerts to the right place (PagerDuty, Slack, email, webhook).

**Three key concepts**:

#### 1. **Routing Tree**: Decision tree for where each alert goes
```yaml
route:
  group_by: ['alertname', 'cluster']        # Group alerts with same name + cluster
  group_wait: 30s                            # Wait 30s to batch similar alerts
  group_interval: 5m                         # Re-evaluate groups every 5 min
  repeat_interval: 4h                        # Re-send if unresolved
  receiver: 'default'                        # Fallback receiver

  routes:
  - match:
      severity: critical
    receiver: 'oncall-pagerduty'
    group_wait: 0s                           # Page immediately for critical
    group_interval: 5m
    
  - match:
      team: platform
    receiver: 'platform-slack'
    group_wait: 30s
```

**Why this matters**: At T-Mobile, a deployment might trigger 20 related alerts (network timeout, connection pool exhaustion, 503s). Alertmanager groups them into one context-rich notification, reducing alert fatigue.

#### 2. **Inhibition**: "Suppress child alerts if parent is firing"
```yaml
inhibition_rules:
- source_match:
    alertname: 'KubernetesClusterDown'
  target_match:
    severity: 'warning'
  equal: ['cluster']   # Only inhibit in the same cluster
```

Translation: If the whole cluster is down, don't page about individual pod restarts — the parent alert already tells us.

#### 3. **Silences**: "Mute this alert until 12pm tomorrow"
Use silences during maintenance windows. Alertmanager stores silences in its local database.

### F. Burn Rate Math: Why 14.4x for a 1-Hour Window?

**SLO = 99.9% (0.1% error budget) over 30 days.**

**Budget burn calculation**:
- Total budget: 0.1% × (30 days × 24h) = 0.1% × 720h = **0.72h of downtime allowed**
- If we detect a 1-hour outage: burn = 1h / 0.72h = **13.9x** (rounds to 14.4x for safety margin)

**Alert definition**:
```promql
alert: FastBurnAlert
expr: |
  (
    1 - (sum(rate(notifications_delivered_total[1h])) / sum(rate(notifications_attempted_total[1h])))
  ) > 0.144
for: 5m
```

If error rate > 14.4% for 5 minutes, page immediately — we're burning the 30-day budget in ~2.5 hours.

## Part 3: Interview Prep — Burn Rate & Alert Routing Depth (30 min)

### The Five Critical Questions

Each question targets Staff/Principal-level responsibilities. Structure answers using STAR + the T-Mobile notification context.

---

### Question 1: "Walk me through a production incident where an SLO breach alert fired. How did you triage?"

**STAR Setup** (2 min):
- **Situation**: T-Mobile notification platform: 25M msg/day, 99.9% SLO, alert fires at 14.4% error rate
- **Task**: Determine if real outage or noisy alert; page team only if real
- **Action**: (explain your process here)
- **Result**: (metric outcome)

**Your 3-minute answer structure**:
1. **(Detect phase — 30 sec)**
  - "Alertmanager fires NotificationDeliveryFastBurn. I first check: Is the alert *grouped with similar alerts* or isolated? If isolated, it's instrumentation noise. If grouped, it's a real blast radius."
  - Example: 20 related alerts (latency spike, connection pool, 503 errors) = real incident.
  - 1 alert in a vacuum = likely false positive.

2. **(Root cause phase — 60 sec)**
  - "I jump to the Grafana dashboard linked in the alert annotation. I overlay 3 signals: success rate, latency, and request rate."
  - "If success rate dropped but request rate is flat, it's a code regression or dependency failure (database, cache)."
  - "If request rate spiked, it's load. If latency spiked, it's resource contention (CPU/memory/network)."
  - "At T-Mobile, we had a memory leak in the notification batching service. Success rate dropped to 85% over 1h. Latency p99 went from 300ms to 5s. Request rate stayed flat. This told us: not load, something's burning resources."

3. **(Response phase — 30 sec)**
  - "If it's a real breach, we escalate to the on-call engineer, provide the dashboard link and runbook."
  - "If it's noise (e.g., one pod restart causing brief spike), we silence the alert for that pod and add a PR comment to suppress it in the alert rule."
  - "We measure MTTR: mean time to resolution. T-Mobile tracks this per on-call rotation; teams with < 30 min MTTR get bonus points."

**Tradeoff Discussion (1 min bonus)**:
- "Static thresholds (p99 > 500ms) fire too often. Burn-rate alerts fire *only when budget is spent*, so we tune them to match our business tolerance. Tradeoff: burn-rate alerts are harder to understand for new SREs (need SLO context), but they're exponentially quieter."

---

### Question 2: "You have 50 services, all with SLOs. How do you scale alert rules without creating alert fatigue?"

**STAR Setup**:
- **Situation**: T-Mobile: 50+ microservices (notifications, billing, provisioning, etc.), each with burn-rate alerts
- **Task**: Scale alerting without oncall getting buried
- **Action**: (your approach)
- **Result**: (alert noise reduction %)

**Your 3-minute answer**:
1. **(Templating — 40 sec)**
  - "We don't hand-write 50 separate alert rules. We use a *parameterized template*. Each service team owns three values: `SLO_PERCENTAGE`, `ERROR_BUDGET_WINDOW`, `PAGE_THRESHOLD`."
  - "Example: Notifications team: 99.9%, 30d, page at 14.4%. Billing team: 99.99%, 30d, page at 144x. Provisioning team: 99.5%, 7d, page at 3.6x."
  - "We generate rules from a template using Jsonnet or Helm, not YAML by hand."

2. **(Routing & Grouping — 60 sec)**
  - "Second, we configure Alertmanager to group alerts by `service` and `severity`. So when 5 related alerts fire (latency, error rate, CPU), the oncall sees one notification, not five."
  - "We use inhibition rules: if a critical alert fires, suppress all warning-level alerts in the same service. This prevents alert avalanches."
  - "At T-Mobile, we also added *smart routing*: critical SLO breaches page PagerDuty immediately (0s group_wait). Warnings go to Slack with 30s group_wait. Non-urgent checks go to a daily email digest."

3. **(Tuning & Feedback Loop — 20 sec)**
  - "We track alert quality: % of pages that result in actual incident response (precision). We want >80% precision. If precision < 50%, the alert is noisy; we retune thresholds or inhibition rules."
  - "We review noisy alerts in postmortems and make them team-local instead of org-wide."

**Tradeoff Discussion (1 min)**:
- "Templatizing is powerful but rigid. A template assumes all services have similar failure modes. If one service has a legitimate 1h burn that shouldn't page, templates don't handle exceptions well. We solve this with *overrides*: teams can override thresholds in their alertmanager config."

---

### Question 3: "Your burn-rate alert has a 5-minute 'for' duration. A page arrives at 5:05. Why not alert at 5:00?"

**STAR Setup**:
- **Situation**: Incident starts at 5:00; oncall paged at 5:05. Why the delay?
- **Task**: Explain debouncing strategy
- **Action**: (your logic)
- **Result**: (false positive reduction %)

**Your 2-minute answer**:
1. **(Debouncing logic — 40 sec)**
  - "The 5-minute `for` duration is a *debounce filter*. Without it, every transient spike (a single slow request, a GC pause, a network hiccup) would fire an alert at 5:00:30."
  - "We require the condition to be *true for 5 consecutive minutes* before paging. Why? Most transient issues self-recover (GC ends, cache populates, request times out and retries)."
  - "At T-Mobile, ~40% of error spikes resolve themselves in < 2 min. A 5-min debounce eliminates most of those false pages."

2. **(Trade-off: Latency vs. Noise — 80 sec)**
  - "The tradeoff: we're accepting 5 min of latency to detect real incidents. If a real outage starts at 5:00, we don't page until 5:05. Is that acceptable?"
  - "It depends on your SLO window. For a 99.9% SLO with 1-hour fast-burn window, 5 min of delay is ~0.1% of the window. Not material."
  - "For a 99.99% SLO with a 10-min window (tighter budget), 5 min delay is half the window. We'd reduce the `for` duration to 2 min or use a faster burn window (5 min instead of 1h)."
  - "At T-Mobile, we use variable debounce: critical page? 2 min. Warning? 10 min. Non-urgent diagnostic? 30 min."

3. **(False negative risk — 20 sec)**
  - "Downside: if your outage is *exactly 5 min long* (e.g., a deployment that auto-rolls-back), you might not page at all."
  - "Mitigation: we also run a 30-day slow-burn alert (not tied to `for`). It catches degradation over hours."

---

### Question 4: "You have two alerts: 'ServiceDown' (critical) and 'HighErrorRate' (warning) on the same service. Both fire. The on-call gets one notification. How?"

**STAR Setup**:
- **Situation**: Alertmanager receives 2 alerts with the same `alertname` and `team` label
- **Task**: Explain alert grouping and inhibition
- **Action**: (your strategy)
- **Result**: (unified notification)

**Your 2-minute answer**:
1. **(Grouping — 40 sec)**
  - "Alertmanager uses `group_by` labels to *group alerts into a single notification*. In our config, we group by `[alertname, team]`."
  - "When ServiceDown and HighErrorRate both fire, they have the same team (platform) and alertname (NotificationDeliveryFastBurn), so Alertmanager groups them."
  - "The oncall gets one Slack message with both alerts, not two separate pages."

2. **(Inhibition — 60 sec)**
  - "But we can go further: Alertmanager's `inhibit_rules` say 'if a critical alert fires, suppress all warning-level alerts in the same team'."
  - "Why? If the whole service is down (critical), we don't need to know that error rate is high (warning). The root cause is obvious: the service is down."
  - "Code: `source_match: {alertname: ServiceDown}, target_match: {severity: warning}, equal: [team]`."
  - "Result: oncall sees only the critical alert. Reduces cognitive load."
  - "At T-Mobile, this cut alert noise by ~35%. Previously, a single pod crash would fire 5 related warnings. Now: 1 critical alert."

3. **(Silence alternative — 20 sec)**
  - "If inhibition rules aren't flexible enough, teams can manually *silence* an alert via Alertmanager UI for N minutes. Useful during deployments."

### Question 5: "Design an alerting strategy for a 99.99% SLO service with a 30-day budget. The service experiences bursty traffic and occasional GC pauses. How many alert rules? What thresholds?"

**STAR Setup**:
- **Situation**: 99.99% SLO (0.01% error budget = 4.3 min/month downtime), bursty traffic, GC noise
- **Task**: Design alert rules without false positives
- **Action**: (your design)
- **Result**: (alert precision %)

**Your 3-minute answer**:
1. **(Budget math — 40 sec)**
  - "0.01% error budget over 30 days = 4.3 minutes of downtime allowed."
  - "Fast-burn window (1h): 4.3 min / (30d × 24h × 60min) = 1.0x per hour acceptable. So 144x burn in 1h = page immediately."
  - "Slow-burn window (30d): 144x over a day is acceptable (still have 3 weeks of budget). Don't alert on 30d slow burn."
  - "Medium burn (6h window): 144x / 4 = 36x. If we hit 36x in 6h, we're on-track to exhaust budget by end of week. Alert, but lower urgency."

2. **(GC pause handling — 80 sec)**
  - "The service has GC pauses (100-500ms) every 10 min. Raw p99 latency alert fires often (false positive)."
  - "Solution: Alert on *error rate change*, not latency directly. GC doesn't cause errors; it causes slowness, which self-recovers."
  - "Alert rule: `error_rate[1h] > 1%` (fast burn). `error_rate[6h] > 0.36%` (medium burn). No latency alert; latency is logged separately for post-incident analysis."
  - "For bursty traffic: request rate spike is normal, not an incident. Alert on error rate / request rate ratio, not request rate."

3. **(Final rules — 20 sec)**
  - "Three alert rules: FastBurn (critical, page immediately), MediumBurn (warning, page in 15 min if unresolved), SlowBurnDetection (diagnostic, daily email)."
  - "Expected false positive rate: ~5-10% (e.g., deployments causing brief spike). Teams silence those manually."

## Part 4: Key Takeaways & Reflection (10 min)

### What You've Learned
1. **Prometheus data model**: 4-tuple (metric, labels, timestamp, value) is faster and cheaper than logs for metrics
2. **PromQL**: Like Splunk SPL, but columnar. `rate()`, `sum(by(...))`, `histogram_quantile()` are the core operations
3. **Recording rules**: Pre-compute aggregations for dashboard performance
4. **Alert rules**: Fire on SLO burn rate, not static thresholds. Two-window strategy (1h fast + 6h medium) catches real incidents without noise
5. **Alertmanager**: Routes and groups alerts intelligently. Inhibition rules suppress child alerts when parent is firing. Silences suppress alerts during maintenance
6. **Interview depth**: Staff SREs own alert strategy for teams; PrincipalSREs own org-wide alerting policy. Be ready to explain burn rate math, explain tradeoffs in debounce timing, and defend why you chose alert thresholds
