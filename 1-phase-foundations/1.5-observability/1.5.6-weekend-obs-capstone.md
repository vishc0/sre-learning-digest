## Analogy

Think of your T-Mobile notification platform like a toll highway. The SLO is the contractual promise: 99.9% of cars pass through without error. The error budget is the maintenance window baked into that contract — you are allowed 43.8 minutes of failure per month before you breach the SLA. Burn rate alerts are the highway patrol: the fast-burn alert (14x) is a squad car that responds when 14% of the monthly budget is burning in 1 hour (emergency response). The slow-burn alert (6x) is a drone patrol that spots a slow leak before it becomes a crisis. Grafana is the traffic control center dashboard. The OTel Collector is the sensor network embedded in every lane — it collects, enriches, and routes telemetry to multiple backends simultaneously (CloudWatch for AWS-native alerting, Prometheus for rich SLO math).

## 2-Minute Version

The architecture has three layers. The collection layer is the OTel Collector DaemonSet — one pod per node, configured to receive OTLP from application pods and scrape Prometheus endpoints. It exports to two backends: ADOT pipeline to CloudWatch (for AWS-native alerting and cost dashboards) and Prometheus remote-write (for SLO math). The alerting layer uses Sloth to generate multi-window, multi-burn-rate alert rules from a human-readable SLO spec. Sloth outputs PrometheusRule CRDs that Prometheus Operator picks up automatically — no manual Prometheus config editing. The visualization layer is Grafana with four panels: success rate time series, p99 latency histogram, error budget burn gauge, and DORA deployment frequency bar chart. The RUNBOOK.md maps each alert to a response playbook anchored to your T-Mobile MART framework.

## 5-Minute Version

**Why dual export matters**: CloudWatch is required for AWS-native tooling (PagerDuty integrations, Cost Explorer correlation, Security Hub). Prometheus is required for SLO math — CloudWatch cannot natively compute multi-window burn rates. Running both from a single OTel Collector avoids double-instrumentation and gives you a single configuration control plane for all telemetry routing.

**Why Sloth instead of hand-written alerts**: The multi-window, multi-burn-rate alerting specification (from the Google SRE Workbook) requires 4 alert rules per SLO (2 windows x 2 burn rates). Hand-writing these is error-prone. Sloth takes one YAML spec and generates all 4 rules with the correct PromQL, plus SLO metadata labels that Grafana can use for automatic dashboard population. This is GitOps-friendly: SLO changes are a PR, not a Prometheus config edit.

**Failure modes to know**: (1) OTel Collector memory pressure — if the DaemonSet is not resource-limited, it will OOM-kill on high-cardinality metric bursts; set `memory_limiter` processor. (2) Prometheus cardinality explosion — notification platform metrics often have `user_id` or `message_id` as labels; these must be dropped at the OTel Collector before export. (3) Sloth-generated rules assume Prometheus recording rules are evaluated every 60s — if your Prometheus evaluation interval is longer, burn rate math drifts.

**Tradeoff — DaemonSet vs. Sidecar**: DaemonSet (one collector per node) is operationally simpler and resource-efficient. Sidecar (one collector per pod) gives per-pod isolation and is required if pods have different trust boundaries. For the notification platform, DaemonSet is correct — all pods are in the same trust boundary.

## Comprehension Check

**Question 1**: Your notification service is burning error budget at 14x the normal rate for 45 minutes. Should the fast-burn alert have fired? Why or why not, using the burn rate math?

**Question 2**: A teammate says "let's just use CloudWatch for everything — we already pay for it." What is the one capability that CloudWatch cannot provide that justifies running Prometheus alongside it?

**Question 3**: What is the risk of putting `user_id` as a Prometheus metric label on the notification platform, and where in the OTel pipeline do you mitigate it?

---

## ANSWER KEY — Comprehension Check

**Answer 1**: Yes, the fast-burn alert should fire. Fast-burn threshold = 14x burn rate sustained for 1 hour. At 14x for 45 minutes, you have consumed 14 * (45/60) / (30 * 24) = 14 * 0.75 / 720 = 1.46% of the 30-day error budget in one window. The alert fires when the 1-hour burn rate window shows >= 14x — by 45 minutes in, the 1h rolling window already reflects this rate, so the alert fires before the full hour elapses (typically within 5-10 minutes of sustained 14x burn, depending on scrape interval).

**Answer 2**: Multi-window, multi-burn-rate SLO alerting using PromQL recording rules. CloudWatch metric math cannot compute the rolling-window ratio `(1 - success_rate) / error_rate_budget` across two time windows simultaneously with the precision required for burn rate alerts. Prometheus recording rules pre-compute these ratios at scrape time, enabling sub-minute alert evaluation latency.

**Answer 3**: If `user_id` has high cardinality (millions of unique values), Prometheus creates one time series per unique label value, leading to cardinality explosion — memory exhaustion, slow queries, and Prometheus OOM. Mitigation: drop or hash the `user_id` label using the `attributes` processor in the OTel Collector pipeline, before the metrics are exported to Prometheus. Keep `user_id` only in traces (where cardinality is handled differently via sampling).

### Step 9: Deploy OTel Collector as DaemonSet with dual export

```bash
cat > ~/Training/observability/otel-collector/otel-collector-configmap.yaml << 'EOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: otel-collector-config
  namespace: observability
data:
  config.yaml: |
    # ============================================================
    # RECEIVERS: Define where the collector ingests data from
    # ============================================================
    receivers:
      # OTLP receiver: application pods send metrics/traces here
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317   # gRPC port for OTLP
          http:
            endpoint: 0.0.0.0:4318   # HTTP port for OTLP

      # Prometheus receiver: scrape metrics from pods with prometheus.io/scrape annotations
      prometheus:
        config:
          scrape_configs:
            - job_name: 'notification-service'
              kubernetes_sd_configs:
                - role: pod
                  namespaces:
                    names: ['default', 'notification']
              relabel_configs:
                # Only scrape pods with the annotation prometheus.io/scrape: "true"
                - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
                  action: keep
                  regex: true
                - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_port]
                  action: replace
                  target_label: __address__
                  regex: (.+)
                  replacement: ${1}

      # Host metrics: CPU, memory, disk, network per node
      hostmetrics:
        collection_interval: 30s
        scrapers:
          cpu: {}
          memory: {}
          disk: {}
          network: {}

    # ============================================================
    # PROCESSORS: Transform data before export
    # ============================================================
    processors:
      # Memory limiter: CRITICAL — prevents OOM on high-cardinality bursts
      # Think of this as a circuit breaker on the collector's memory usage
      memory_limiter:
        check_interval: 1s
        limit_mib: 400        # Hard ceiling — drops data above this to protect the node
        spike_limit_mib: 100  # Headroom for burst spikes

      # Batch processor: groups data for efficient export (reduces API call overhead)
      batch:
        timeout: 10s
        send_batch_size: 1000

      # Attributes processor: DROP high-cardinality labels before Prometheus export
      # user_id and message_id would create millions of time series — cardinality explosion
      attributes/drop_high_cardinality:
        actions:
          - key: user_id
            action: delete
          - key: message_id
            action: delete
          - key: request_id
            action: delete

      # Resource detector: adds EKS cluster metadata to all telemetry
      resourcedetection:
        detectors: [eks, ec2]
        timeout: 2s

    # ============================================================
    # EXPORTERS: Define where processed data goes
    # ============================================================
    exporters:
      # CloudWatch exporter: AWS-native metrics store
      # Used for: billing dashboards, PagerDuty integration, Security Hub correlation
      awscloudwatch:
        region: us-east-1
        namespace: NotificationPlatform    # CloudWatch metrics namespace
        log_group_name: /otel/notification-platform
        dimension_rollup_option: "ZeroAndSingleDimensionRollup"

      # Prometheus remote-write exporter: sends to kube-prometheus-stack's Prometheus
      # Used for: SLO math, burn rate alerts, Grafana dashboards
      prometheusremotewrite:
        endpoint: "http://kube-prometheus-stack-prometheus.observability.svc.cluster.local:9090/api/v1/write"
        tls:
          insecure: true    # Internal cluster traffic — TLS optional in training
        resource_to_telemetry_conversion:
          enabled: true     # Converts OTel resource attributes to Prometheus labels

      # Debug exporter: logs telemetry to collector stdout (training only — disable in prod)
      debug:
        verbosity: basic

    # ============================================================
    # SERVICE: Wire receivers → processors → exporters into pipelines
    # ============================================================
    service:
      pipelines:
        # Metrics pipeline: full dual-export path
        metrics:
          receivers: [otlp, prometheus, hostmetrics]
          processors: [memory_limiter, attributes/drop_high_cardinality, resourcedetection, batch]
          exporters: [awscloudwatch, prometheusremotewrite, debug]

      telemetry:
        logs:
          level: info    # Collector's own logs — use 'debug' when diagnosing pipeline issues
EOF

kubectl apply -f ~/Training/observability/otel-collector/otel-collector-configmap.yaml
```

**EXPECTED_OUTPUT**:
```
configmap/otel-collector-config created
```

**WHY**: The ConfigMap is the OTel Collector's routing brain — every receiver, processor, and exporter is declared here. This is the primary artifact you will tune in production.

### Step 10: Deploy OTel Collector DaemonSet

```bash
cat > ~/Training/observability/otel-collector/otel-collector-daemonset.yaml << 'EOF'
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: adot-collector
  namespace: observability
  labels:
    app: adot-collector
    component: observability
spec:
  selector:
    matchLabels:
      app: adot-collector
  template:
    metadata:
      labels:
        app: adot-collector
    spec:
      # Use the IRSA-annotated service account — this is what grants CloudWatch access
      serviceAccountName: adot-collector

      # Allow the collector to read host-level metrics
      hostNetwork: false

      containers:
        - name: adot-collector
          # AWS Distro for OTel — AWS-tested build with CloudWatch exporter included
          image: public.ecr.aws/aws-observability/aws-otel-collector:v0.40.0
          args:
            - "--config=/conf/config.yaml"
          
          # Resource limits: CRITICAL for DaemonSet — must not starve application pods
          resources:
            requests:
              cpu: "100m"      # 0.1 CPU core at minimum
              memory: "256Mi"
            limits:
              cpu: "500m"      # 0.5 CPU core maximum — prevents noisy-neighbor issues
              memory: "512Mi"  # Must be > memory_limiter limit_mib (400Mi) to avoid OOM

          # Port declarations for OTLP receivers
          ports:
            - containerPort: 4317   # OTLP gRPC
              protocol: TCP
            - containerPort: 4318   # OTLP HTTP
              protocol: TCP

          # Mount the ConfigMap as a file inside the container
          volumeMounts:
            - name: otel-config
              mountPath: /conf

          # Health check: if the collector hangs, Kubernetes restarts it
          livenessProbe:
            httpGet:
              path: /
              port: 13133    # ADOT built-in health check endpoint
            initialDelaySeconds: 10
            periodSeconds: 30

          readinessProbe:
            httpGet:
              path: /
              port: 13133
            initialDelaySeconds: 5
            periodSeconds: 10

          # Environment variable for IRSA token — injected by EKS automatically
          env:
            - name: AWS_REGION
              value: "us-east-1"
            - name: K8S_NODE_NAME
              valueFrom:
                fieldRef:
                  fieldPath: spec.nodeName

      volumes:
        - name: otel-config
          configMap:
            name: otel-collector-config
            items:
              - key: config.yaml
                path: config.yaml

      # Tolerate all node taints so the DaemonSet runs on every node including control plane
      tolerations:
        - operator: "Exists"
          effect: "NoSchedule"
EOF

kubectl apply -f ~/Training/observability/otel-collector/otel-collector-daemonset.yaml
```

```bash
  - name: "notification-delivery-success-rate"
    description: >
      99.9% of notification delivery attempts must succeed within a 28-day rolling window.
      A failed delivery is any HTTP 5xx, timeout, or downstream provider rejection.
      This SLO directly maps to the customer-facing SLA in the T-Mobile platform contract.

    # The SLO target and window
    objective:
      value: 99.9           # 99.9% success rate = 0.1% error rate allowed
      window: 28d           # 28-day rolling window = 40.32 minutes error budget per 28 days

    # SLI definition: how Prometheus measures the error rate
    # These PromQL expressions query the notification service metrics
    sli:
      # Raw error ratio: (bad events) / (total events)
      # Replace these metric names with your actual app metrics in production
      events:
        error_query: >
          sum(rate(notification_delivery_requests_total{status=~"5..",job="notification-service"}[{{.window}}]))
        total_query: >
          sum(rate(notification_delivery_requests_total{job="notification-service"}[{{.window}}]))

      name: NotificationDeliverySuccessRate

      # Annotations appear in PagerDuty/Slack alert body

        runbook_url: "https://github.com/YOUR_GITHUB/notification-platform-observability/blob/main/RUNBOOK.md"

        severity: "critical"

        service: "notification-platform"
```

## Key Engineering Decisions and Tradeoffs

**1. DaemonSet vs. Sidecar for OTel Collector**  
Chose DaemonSet: one collector per node serves all pods on that node. Lower overhead, simpler config, adequate for a trust-homogeneous notification platform. Sidecar would be required only if pods had different trust boundaries (e.g., multi-tenant with per-customer data isolation requirements).

**2. Sloth for alert generation vs. hand-written PromQL**  
Sloth generates correct multi-window burn rate rules from a single declarative spec. Hand-writing 4 recording rules + 4 alert rules with correct label selectors and window math introduces subtle errors. Sloth is the SRE community standard (used at Spotify, Grafana Labs, SoundCloud).

**3. Dual export (CloudWatch + Prometheus) vs. Prometheus-only**  
CloudWatch is required for AWS operational tooling (Security Hub, Cost Explorer, PagerDuty native integration). Prometheus is required for SLO math (multi-window burn rates). Running both from a single OTel pipeline costs ~$0.01/1000 metrics vs. the operational risk of AWS-native alerting gaps.

## Tradeoffs and Failure Modes

**Tradeoff 1: OTel Collector DaemonSet resource limits**  
Setting the memory limit too low causes the collector to drop metrics during cardinality bursts, creating gaps in your SLO data exactly when you need it most (during incidents). Setting it too high risks starving application pods. The correct sizing formula: `limit_mib = (expected_metrics_per_second * avg_metric_size_bytes * batch_timeout_seconds * 3) / 1000000`. For the notification platform at 25M msg/day with ~100 metric series, 512Mi is safe. Always set the OTel `memory_limiter` processor limit to 20% below the Kubernetes memory limit to allow graceful shedding before Kubernetes OOM-kills the pod.

**Tradeoff 2: Multi-window burn rate alerts — sensitivity vs. false positives**  
The AND condition (both short-window AND long-window must exceed threshold) is conservative — it adds latency to alert firing (you will not be paged for a 30-second spike). The alternative (OR condition) pages faster but causes alert storms during transient failures that self-heal. For the notification platform, AND is correct: a genuine sustained burn rate at 14x for 5+ minutes represents thousands of failed messages — the 2-minute `for` clause catches it quickly enough. The tradeoff point changes for latency-sensitive financial transaction systems where a 2-minute window is too slow.

**Tradeoff 3: Sloth-managed vs. hand-written PrometheusRule CRDs**  
Sloth generates rules from the SLO spec and marks them with `app.kubernetes.io/managed-by: sloth`. The tradeoff is that you cannot manually edit the generated CRDs — Sloth will overwrite them on the next `generate` run. The governance implication: the SLO YAML (`notification-delivery-slo.yaml`) is the source of truth, and it must be in Git with PR review. Any SLO target change (e.g., raising from 99.9% to 99.95%) requires a PR, a Sloth regeneration, and a reapply to the cluster. This is a feature, not a bug — it prevents undocumented SLO drift that has caused compliance failures in regulated industries (financial services, healthcare).
