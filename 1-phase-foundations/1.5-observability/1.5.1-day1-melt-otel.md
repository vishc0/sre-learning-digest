## Analogy

**Think of your T-Mobile notification platform.**

You have RabbitMQ sitting in the middle of your architecture. Producers (upstream services) publish messages. Consumers (downstream processors) subscribe. The critical insight: you can add a second consumer — say, an audit log service — without touching the producer code at all. RabbitMQ is the router; AMQP is the protocol both sides speak.

**OpenTelemetry Collector is the exact same pattern for telemetry data.**

Your application services are the producers. They emit telemetry (spans, metrics, logs) using the OTLP protocol. The OTel Collector sits in the middle. On the output side, you can fan out to Splunk HEC AND Prometheus AND Jaeger AND an S3 file exporter — simultaneously — without changing a single line of application code. OTLP is to telemetry what AMQP is to messaging: a protocol that any sender and any receiver can speak, making the broker (Collector) the only thing you swap when you change vendors.

When your team moved from one Splunk index strategy to another, you changed the routing rules in Splunk — not the application. Same mental model. Change the Collector pipeline config, not the app.

## OTel Collector Architecture: The RabbitMQ Analogy (Detailed)

You know RabbitMQ intimately. OTel Collector follows **the exact same topology**:

```
┌─────────────────────────────────────────────────────────────┐
│ OTel Collector Pipeline (like RabbitMQ broker topology)    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Receivers (like RabbitMQ input exchanges)                 │
│  ├── OTLP (protocol buffer — like AMQP)                    │
│  ├── Prometheus scrape (like polling queue depth)          │
│  └── Jaeger (like consuming from legacy trace queue)       │
│           │                                                │
│           ↓                                                │
│  Processors (like Shovel exchange + message transform)     │
│  ├── Batch (buffer spans before export)                    │
│  ├── Attributes (add customer_id to every span)            │
│  ├── Sampling (tail-based: keep if error_status=true)      │
│  └── Span processor (rename fields, drop high-cardinality)  │
│           │                                                │
│           ↓                                                │
│  Exporters (like RabbitMQ consumer bindings)               │
│  ├── OTLP gRPC (like AMQP to Kafka)                        │
│  ├── Jaeger (legacy backend compatibility)                 │
│  ├── Prometheus remote write (like Cassandra sink)         │
│  └── File exporter (dev/local debugging)                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Why this analogy works**:
- **Receivers** = RabbitMQ *input* exchanges (where messages enter)
- **Processors** = Shovel plugin (intercept, transform, route)
- **Exporters** = RabbitMQ *output* bindings (where messages exit)
- **Pipelines** = Exchange → queue → binding chain (data flow DAG)
- **Batching** = RabbitMQ prefetch (buffer before ack)

Just as you'd tune RabbitMQ prefetch to balance throughput vs. memory, you tune OTel batch processor to balance latency vs. export overhead.

## 30-Second Version

MELT is a four-signal observability framework: Metrics (aggregated numbers over time), Events (discrete moments that matter), Logs (timestamped records of what happened), Traces (the end-to-end call chain across services). You already use all four in your MART framework at T-Mobile — this is a rename, not a new concept. OpenTelemetry is the CNCF standard that gives you one SDK and one protocol (OTLP) to emit all four signals, plus a Collector that routes them to any backend without touching application code. It solves vendor lock-in the same way RabbitMQ solves consumer lock-in.

## 2-Minute Version

### MELT — Why Four Signals?

Each signal captures something the others cannot reconstruct.

**Metrics** are pre-aggregated numbers sampled over time: CPU at 87%, 25,000 msg/sec, error rate 0.3%. They are cheap to store and fast to alert on, but they throw away context. When CPU hits 87%, a metric tells you something is wrong — it cannot tell you which request caused it.

**Events** are discrete, timestamped occurrences: a deployment fired, a circuit breaker opened, a threshold crossed. Events are the "what changed" signal. In incident command, when you're triangulating a customer impact window, events give you the timeline of changes correlated against the impact. AppDynamics health rule violations are events.

**Logs** are timestamped, structured records of what the code did: request in, validation passed, downstream call made, response sent. Logs are the narrative. Splunk is your primary log platform today. The challenge with logs is volume — a 25M msg/day platform generates enormous log cardinality, which is why you index selectively in MART Reporting.

**Traces** are the cross-service call graph for a single request: Service A called Service B called Service C; B took 230ms, C returned an error. Traces are the "which path did this specific request take" signal — irreplaceable for diagnosing latency in microservice architectures. When a notification fails to deliver on your platform, a trace tells you exactly which hop introduced the delay or error, without you having to manually correlate log timestamps across five services.

### OpenTelemetry — Why It Exists

Before OTel, you faced a choice: use vendor SDK (AppDynamics agent, Datadog tracer) and get locked in, or use an open standard (OpenTracing for traces, OpenCensus for metrics) and face two incompatible APIs. In 2019, the CNCF merged both projects into OpenTelemetry — one API, one SDK, one wire protocol (OTLP). Now you instrument once and route telemetry anywhere: Splunk, Prometheus, Jaeger, Tempo, Lightstep — by changing Collector config, not application code.

### OTel Collector Pipeline

The Collector has three stages:

1. **Receivers** — listen for incoming telemetry. OTLP receiver on gRPC :4317 and HTTP :4318. Can also receive Prometheus scrapes, Jaeger spans, Zipkin spans.
2. **Processors** — transform, filter, enrich. Batch processor buffers spans before export (reduces TCP overhead). Attributes processor can add `environment: production` to every span. Tail sampling processor can drop 99% of healthy traces and keep 100% of errored ones.
3. **Exporters** — send to backends. Splunk HEC exporter, Prometheus exporter, OTLP exporter to another Collector, file exporter for debugging.

A pipeline wires these together: `receivers: [otlp] → processors: [batch] → exporters: [splunk, prometheus]`.

## 5-Minute Version

### What Each Signal Misses

Understanding MELT means understanding the gaps each signal has — because interviewers test this.

**Metrics miss cardinality.** You cannot store a separate time series per user ID or per request ID — you'd have millions of series. Metrics answer "what is the aggregate state" but not "what happened to this specific request." This is the core reason traces exist: they carry per-request context that metrics structurally cannot.

**Events miss duration.** An event says "circuit breaker opened at 14:32:07." It does not tell you how long the upstream was degraded before that, or what the p99 latency was in the 60 seconds before the event. Events give you moments; metrics give you shapes over time.

**Logs miss correlation across services.** Each service writes its own logs. To reconstruct a cross-service failure in a distributed system, you must manually correlate timestamps and request IDs across log streams — which is what Splunk correlation searches do, but it requires that every service consistently logs the same correlation ID. Traces solve this structurally by embedding the trace ID in every span at instrumentation time, so the correlation is automatic.

**Traces miss aggregate patterns.** A trace shows you one request's path. If you want to know "what percentage of requests to Service B have a slow database call," you need aggregated metrics or log analytics — not individual traces. Tail sampling compounds this: if you only keep 1% of traces, you lose statistical validity for aggregate analysis.

### Why MART and MELT Are the Same Framework

T-Mobile's MART framework (Monitoring → Alerting → Reporting → Troubleshooting) maps directly onto MELT:

| MART Stage | MELT Signal | What It Answers |
|---|---|---|
| Monitoring | Metrics | Is the system healthy right now? |
| Alerting | Events | Did something cross a threshold or change? |
| Reporting | Logs | What did the system do? (audit, compliance, postmortem) |
| Troubleshooting | Traces | Why did this specific request fail? |

This is a direct translation, not a conceptual leap. You already govern all four signals at T-Mobile — the interview insight is articulating *why* each signal exists and *what structural capability it provides that the others cannot.*

### OTel Collector Design Decisions

**Auto-instrumentation vs manual instrumentation.** Auto-instrumentation (via Java agent, Python `opentelemetry-instrument` wrapper) captures framework-level spans — HTTP requests in, HTTP requests out, database calls — without code changes. This is the "AppDynamics agent" model. It's fast to deploy but captures only what the framework exposes. Manual instrumentation means adding `tracer.start_span("validate_notification")` in your business logic — you get the spans that matter for your specific domain, but every developer must write them and keep them maintained. Most mature teams use both: auto for the framework skeleton, manual for the business-critical operations.

**Collector topology: DaemonSet vs Deployment vs Sidecar.** This is a common Staff/Principal interview question:
- *DaemonSet*: one Collector pod per node. Low resource overhead per service; all pods on a node share one Collector. Risk: a misbehaving Collector on a node affects all services on that node.
- *Deployment*: a pool of Collectors as a shared service. Easy to scale independently; tail sampling requires this topology (all spans for a trace must land on the same Collector). Risk: network hop from pod to Collector; shared fate.
- *Sidecar*: a Collector container in every pod. Maximum isolation; useful for strict multi-tenant environments. Risk: high resource overhead at scale (200 services = 200 Collector processes).
- *Recommended*: DaemonSet for agent-mode (forward to a central Deployment); central Deployment for tail sampling and fan-out. This is the "agent + gateway" topology used at hyperscalers.

**Head-based vs tail-based sampling.** Head-based sampling decides at trace start whether to record a trace — before you know if it will be interesting. It's computationally cheap (the SDK just flips a coin per-request) but wastes quota on healthy traces while possibly dropping the one rare error trace. Tail-based sampling collects all spans, buffers them, and decides after the trace completes — keeping 100% of slow or errored traces and dropping most healthy ones. The cost is memory: you must buffer spans for every in-flight trace, which at high throughput requires careful sizing and a stateful Collector deployment.

### Fan-Out: The Killer OTel Feature for Migrations

The reason OTel Collector matters for your job search is that it enables zero-downtime migrations. Current state: you have Splunk HEC receiving telemetry from your notification platform. You want to evaluate Grafana Tempo for traces. Without OTel: you either dual-instrument the app (expensive, error-prone) or do a flag-day migration (risky). With OTel: you add a second exporter to the Collector config — one line of YAML — and Tempo starts receiving the same traces Splunk already receives. You run both in parallel, validate Tempo, then remove the Splunk exporter. The application never changes.

## Director/VP Lens

### How a Leader Governs This vs How an IC Implements It

**What an IC does:**
- Writes OTel instrumentation in service code (manual spans)
- Configures Collector pipelines for their team's services
- Debugs sampling configuration when trace volume exceeds budget
- Chooses between auto and manual instrumentation for a new service

**What a Director/VP SRE does:**

*Governance*: Defines the organization-wide telemetry contract — which signals are mandatory (all services must emit traces with trace ID in logs), which are optional, and what the SLA is for telemetry availability (e.g., traces must be queryable within 60 seconds of request completion).

*Vendor strategy*: OTel's value is not technical — it's strategic. A Director frames it as: "We instrument once. We evaluate vendors without migration cost. We negotiate from a position where switching Splunk to Grafana Cloud is a config change, not a 6-month project." This is a $500K/year leverage point in enterprise vendor negotiations.

*Cost governance*: Telemetry volume is a cost driver. At 25M msg/day, unsampled traces are financially unsustainable. A Director sets sampling budgets: "We keep 100% of error traces, 10% of slow traces (p99 > SLO), 1% of healthy traces." This becomes an SLO-adjacent policy, not an engineering judgment call.

*Hiring and capability building*: Ensures the team is not dependent on a single vendor's proprietary SDK. If AppDynamics is replaced, the OTel instrumentation survives. This is a platform engineering decision, not just an observability decision.

*Board/exec communication*: Translates MELT to business outcomes — "Our mean time to detect dropped notifications is 47 seconds because we have trace-to-log correlation. The industry benchmark is 4 minutes. That gap is worth $X in reduced customer escalations."

**The AI angle (expected at Director level):** AI-assisted root cause analysis (Datadog Watchdog, Splunk ITSI, Dynatrace Davis) works by correlating MELT signals automatically. A Director's job is to ensure telemetry data quality is high enough for AI to produce reliable correlations — garbage in, garbage out. OTel's semantic conventions (standardized span attribute names) are what make AI RCA trustworthy across a heterogeneous service fleet.

---

## ANSWER KEY — Comprehension Check

**Q1 Answer:** The missing signal is **Traces**. Logs tell you each service received or rejected a message (narrative per service). Metrics tell you the aggregate error rate per service. But neither tells you *which specific request path* the dropped messages followed — i.e., which combination of Service A → Service B → Service C is correlated with the 0.1% drop rate. Traces carry a trace ID that links every hop a specific message took, allowing you to filter "show me all traces where the final outcome was a drop" and see which service in the call chain returned an error. Without traces, you are manually correlating log timestamps across 40 services — possible in Splunk with transaction searches, but O(services²) investigative effort vs. O(1) with a trace ID.

**Q2 Answer:** The objection is **vendor lock-in**. Datadog APM SDK is proprietary — if you instrument 40 services with it and later move to Grafana Cloud, you must re-instrument all 40 services, which is a multi-quarter engineering project and a migration risk. The OTel alternative: instrument all 40 services with the OpenTelemetry SDK (vendor-neutral, CNCF-governed) and run an OTel Collector that currently exports to Datadog. When you move to Grafana Cloud in 18 months, you add a Tempo exporter to the Collector config and remove the Datadog exporter. Application code does not change. The migration cost is one YAML file, not 40 service code changes.

**Q3 Answer:** Tail-based sampling must see **all spans for a complete trace before making the keep/drop decision**, which means it must buffer every in-flight span in memory until the root span closes — requiring a single, stateful process that receives all spans for a given trace ID. Head-based sampling makes the decision at the very first span (the root span), before any child spans exist, so each SDK or Collector instance can independently flip a coin per trace ID without needing to see the full trace — making it safe to run on any stateless, distributed node like a DaemonSet pod.

### Step 2: Write the OTel Collector configuration

**WHY:** This file is the heart of the lab. It defines what the Collector receives (OTLP over gRPC and HTTP), how it processes data (batch buffering to reduce TCP overhead), and where it sends data (Prometheus metrics endpoint + verbose logging to stdout). The logging exporter is your "visibility into the pipe" — it lets you see raw span data before you have a real backend.

```bash
cat > ~/otel-lab/otel-collector-config.yaml << 'EOF'
# OTel Collector Configuration
# Think of this file as your RabbitMQ topology definition:
# - receivers = queue bindings (where data enters)
# - processors = shovel rules / message transformations
# - exporters = downstream consumers
# - pipelines = the routing table that wires them together

receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317   # High-throughput binary; use for internal services
      http:
        endpoint: 0.0.0.0:4318   # JSON over HTTP; easier to traverse firewalls

processors:
  batch:
    # Buffer spans/metrics/logs before exporting.
    # WHY: Without batching, every span triggers a separate TCP connection to the backend.
    # At 25M msg/day this would saturate connection limits instantly.
    timeout: 1s           # Flush every 1 second even if batch isn't full
    send_batch_size: 1024 # Or flush when 1024 items are buffered

  memory_limiter:
    # Prevent the Collector from OOM-killing itself under load.
    # This is the operational safety net — set it before you need it.
    check_interval: 1s
    limit_mib: 400        # Hard limit: refuse new data above 400 MiB
    spike_limit_mib: 80   # Soft limit: start dropping if spike exceeds this

exporters:
  # Exporter 1: Prometheus metrics endpoint
  # WHY: Exposes OTel Collector's own internal metrics (spans received, export errors)
  # as a Prometheus scrape target. This is how you monitor your monitoring pipeline.
  prometheus:
    endpoint: "0.0.0.0:8889"
    namespace: otelcol

  # Exporter 2: Logging to stdout
  # WHY: The "printf debugging" of OTel. Prints every span/metric/log the Collector
  # receives, in human-readable form. Essential for lab validation and production debugging.
  # In production, use verbosity: normal. Use detailed only in dev/debug.
  logging:
    verbosity: detailed

  # Exporter 3: File exporter (added in Step 8)
  # Writes OTLP JSON to a file on disk — useful for offline analysis and CI test fixtures.
  file:
    path: /tmp/otel-output.json

service:
  pipelines:
    # Traces pipeline: receives OTLP traces, batches them, writes to logging + file
    traces:
      receivers: [otlp]
      processors: [memory_limiter, batch]
      exporters: [logging, file]

    # Metrics pipeline: receives OTLP metrics, exposes them via Prometheus scrape
    metrics:
      receivers: [otlp]
      processors: [memory_limiter, batch]
      exporters: [logging, prometheus]

    # Logs pipeline: receives OTLP logs, writes to logging exporter
    logs:
      receivers: [otlp]
      processors: [memory_limiter, batch]
      exporters: [logging]
EOF
```


### Step 3: Write docker-compose.yml

**WHY:** Docker Compose lets you define the full local environment (OTel Collector + Prometheus) as code and start/stop it with one command. This is Infrastructure as Code at the local lab level — the same declarative model as Terraform, but for local containers.

Now write the trace-sending script:

```bash
cat > ~/otel-lab/send_trace.py << 'EOF'
"""
Simulates a simplified version of your notification platform:
  - Parent span: "process_notification" (the top-level operation)
  - Child span: "send_to_downstream" (a downstream service call)

This is manual instrumentation — you are explicitly creating spans
with business-meaningful names, rather than relying on an auto-instrumentation
agent to create framework-level spans.
"""

from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.resources import Resource
import time

# --- Step 1: Define the resource (who is sending this telemetry?)
# In production, resource attributes identify the service in your service catalog.
resource = Resource.create({
    "service.name": "notification-processor",  # Semantic convention: always set this
    "service.version": "1.2.0",
    "deployment.environment": "lab",
})

# --- Step 2: Create a TracerProvider with OTLP exporter
# This is the SDK initialization — do this once at application startup.

    # The trace context (trace ID + parent span ID) is automatically propagated
    # to the child span — this is the W3C TraceContext standard.
    with tracer.start_as_current_span("send_to_downstream") as child_span:

# Without an explicit shutdown, the script exits before the spans are sent.
```

## Lab ANSWER KEY

**Problem: `docker compose up` fails — otelcol container exits immediately.**
```bash
docker compose logs otelcol
```
Look for: `Error in config file` followed by a YAML path. 99% of the time this is a YAML indentation error. The OTel Collector config uses 2-space indentation throughout. Tabs are not valid YAML.

**Problem: Python script exits with `StatusCode.UNAVAILABLE` or `Failed to export spans`.**
Cause: The Collector is not yet ready, or the port is not reachable.
```bash
# Confirm the Collector is listening
docker compose ps otelcol   # Should show "Up"
# Confirm port is reachable from WSL2
nc -zv localhost 4317
```
If port is not reachable: check Docker Desktop → Settings → WSL Integration is enabled. Ports published in docker-compose.yml are accessible from WSL2 via `localhost` only when WSL2 integration is active.

**Problem: `otelcol_receiver_accepted_spans` shows 0 after running the Python script.**
Cause: The `provider.shutdown()` call at the end of the script was not reached (script errored before it). Check the Python script output for exceptions. Also confirm the OTLP exporter endpoint matches: `localhost:4317` not `127.0.0.1:4317` (either works, but be consistent).

**Problem: `/tmp/otel-output.json` does not exist after Step 8.**
Cause: The file exporter may not have flushed yet (batch processor delay), or the `/tmp` mount did not bind correctly on your system.
```bash
# Check if the file was written inside the container instead
docker exec otelcol ls /tmp/
```
If you see `otel-output.json` inside the container, the `/tmp:/tmp` bind mount did not work. Fix: change the docker-compose.yml volume mount to `- ./output:/tmp` and look for the file in `~/otel-lab/output/otel-output.json`.

**Problem: Prometheus UI at localhost:9090 shows otelcol target as "DOWN".**
```bash
# Check Prometheus can resolve the otelcol hostname
docker exec prometheus wget -qO- http://otelcol:8889/metrics | head -5
```
If this fails, both containers may not be on the same Docker network. Verify the `networks: otel-net` block exists in both services in docker-compose.yml.

## Interview Questions

### Q1: "What is the MELT framework and how does it differ from the traditional 3-pillar model (metrics, logs, traces)?"

**What they are really asking:** Do you understand *why* Events exist as a distinct signal, or do you think MELT is just "3 pillars with a new acronym"?

**Answer framework:**
The 3-pillar model (metrics, logs, traces) was popularized by Charity Majors / honeycomb.io around 2017 and is accurate as far as it goes. MELT adds Events as an explicit first-class signal. The distinction matters in practice: Events are discrete, timestamped, high-priority occurrences — a deployment completion, a circuit breaker state change, a configuration push — that are neither metrics (they're not aggregated samples) nor logs (they're not narrative records of code execution). Events are change markers. In incident command, you correlate Events against your Metrics and Traces to answer "what changed right before the impact started?" Without Events as a distinct signal type, you're manually hunting for that information in log searches or deployment pipelines. In the Splunk context, MELT maps directly to MART: Monitoring (M), Alerting (E), Reporting (L), Troubleshooting (T) — the same four concerns with different names.

---

### Q2: "Why did the CNCF consolidate OpenTracing and OpenCensus into OpenTelemetry? What problem does that solve for an SRE?"

**What they are really asking:** Do you understand the ecosystem history, or did you just learn OTel in isolation?

**Answer framework:**
By 2018, distributed tracing had two incompatible open standards: OpenTracing (CNCF, API-only, adopted by Jaeger and Zipkin) and OpenCensus (Google, API + SDK, adopted by many Google-origin projects). A library author who wanted to support both had to write two separate instrumentation paths. An SRE managing a polyglot microservice fleet faced a choice: standardize on one standard and exclude libraries that only support the other, or instrument twice. The CNCF merged both projects into OpenTelemetry in 2019, delivered a stable tracing API in 2021 and metrics API in 2022. For an SRE, the value is consolidation: one SDK that every service framework (Spring Boot, Flask, Express, Go net/http) will eventually support natively, one wire protocol (OTLP) that every backend will eventually accept, and one Collector that routes everything without per-backend SDK changes. The practical impact: when your organization evaluates a new APM vendor, the switching cost is a Collector config change, not a re-instrumentation project.

---

### Q3: "Your team uses Splunk HEC today. How do you migrate to OTel Collector without a rip-and-replace during a production system?"

**What they are really asking:** Can you design a zero-downtime migration? Do you understand the OTel Collector's fan-out capability?

**Answer framework:**
The migration has four phases, none of which require application downtime:

Phase 1 — Introduce the Collector as a passthrough: Deploy OTel Collector as a DaemonSet (agent mode). Configure it with a Splunk HEC exporter pointing to your existing Splunk infrastructure. Route application logs and traces through the Collector — Splunk sees no change, application sees no change. This phase validates that the Collector pipeline works without modifying the data path.

Phase 2 — Fan-out validation: Add a second exporter to the same pipeline — your new target backend (e.g., Grafana Tempo for traces). Both Splunk and Tempo receive identical data. Run in parallel for 2-4 weeks; validate that Tempo receives complete, accurate traces. This is the key OTel value: one data stream, two destinations, no producer code change.

Phase 3 — Gradual cutover: Switch dashboards and alerts to the new backend service-by-service. Remove the Splunk exporter from individual pipelines as each service validates clean on the new backend.

Phase 4 — Splunk exporter removal: Once all services are validated, remove the Splunk HEC exporter from the Collector config. The Collector and all application code remain unchanged.

At T-Mobile, I would run this alongside our Splunk MART framework — the MART dashboards continue operating from Splunk throughout the migration; we add OTel-native dashboards incrementally and transfer on-call runbooks only after the new stack is proven.

---

### Q4: "What is the difference between auto-instrumentation and manual instrumentation? When does each approach break down?"

**What they are really asking:** Can you make the right architectural choice for a mixed fleet, not just name two approaches?

**Answer framework:**
Auto-instrumentation uses a language agent (Java `-javaagent`, Python `opentelemetry-instrument` wrapper) to hook into framework internals at runtime — capturing inbound HTTP request spans, outbound HTTP call spans, database query spans, and message broker publish/consume spans automatically. No application code changes. The failure mode is coverage gaps: auto-instrumentation captures what the framework exposes. Business logic that does not go through a standard framework boundary — a custom RabbitMQ consumer that processes messages in a non-standard way, an in-memory cache lookup, a calculation step that takes 2 seconds — generates no spans. You see the framework-level skeleton but not the business-level detail.

Manual instrumentation adds `tracer.start_span("validate_notification")` explicitly in application code. You get the spans that matter for your domain: message validation duration, downstream retry count, enrichment service latency. The failure mode is maintenance burden: developers must add spans to every new code path, and spans rot over time as code evolves without updating span names or attributes. It also requires every developer on the team to understand the OTel API.

At scale (200 microservices), the right answer is both: auto-instrumentation for the framework skeleton deployed uniformly via a base container image or admission webhook, manual instrumentation for the 5-10 business-critical operations per service that appear in SLO calculations. This keeps the per-developer burden low while ensuring your most important operations are observable.

---

### Q5: "Design an OTel Collector deployment for an EKS cluster with 200 services. What topology do you choose and why?"

**What they are really asking:** Can you make an architecture decision with real tradeoffs, not just recite topology names?

**Answer framework:**
I would use the agent + gateway topology:

**Layer 1 — DaemonSet agents:** One OTel Collector per node, running as a DaemonSet. Each agent receives OTLP from all pods on its node (pods send to `localhost:4317` via the node's host network). Agents are configured with head-based sampling and batch processing. They forward spans to the gateway tier over OTLP/gRPC. This layer handles volume reduction close to the source — preventing a spike in one service from flooding the network.

**Layer 2 — Deployment gateway:** A horizontally scaled Deployment of 3-5 Collector replicas behind a Kubernetes Service. The gateway handles: tail-based sampling (all spans for a trace must land on the same gateway pod — use consistent hashing by trace ID via the loadbalancing exporter at the agent tier), fan-out to multiple backends (Splunk HEC, Prometheus remote write, Grafana Tempo), attribute enrichment (adding cluster name, namespace, account ID to every span), and secrets handling (Splunk HEC token stored in Kubernetes Secret, mounted as environment variable in the Collector pod).

**Why not sidecar at 200 services:** 200 services × average 3 replicas = 600 Collector processes. At 50 MiB each, that's 30 GiB of memory overhead just for the observability layer. Unacceptable. DaemonSet amortizes the Collector cost across all pods on a node.

**Failure mode to call out:** At 200 services and 25M events/day, tail sampling buffer sizing is the critical operational variable. A 5-second trace window at 10K traces/second requires buffering 50K in-flight traces in memory simultaneously. Size your gateway Collector pods with 4 GiB RAM minimum and set a memory_limiter processor with appropriate spike limits.

## Tradeoffs and Failure Modes

### Tradeoff 1: Auto-instrumentation vs Manual Instrumentation

**Speed vs Precision**

| Dimension | Auto-instrumentation | Manual instrumentation |
|---|---|---|
| Time to first trace | Minutes (attach agent, restart service) | Days to weeks (code review, merge, deploy cycle) |
| Coverage | Framework boundaries only (HTTP in/out, DB, MQ) | Any code path, with business-meaningful names |
| Maintenance burden | Near-zero (agent updates are SDK releases) | High — every new code path needs explicit span authoring |
| SLO alignment | Low — framework spans don't map to SLI boundaries | High — you name spans exactly as your SLIs are defined |
| Vendor migration | Clean — SDK is vendor-neutral | Clean — same SDK, different exporter config |

**When auto-instrumentation breaks down:** Large message batch operations processed in a single framework-level span (one RabbitMQ `basicConsume` callback that processes 500 messages) appear as one span with misleading duration and no per-message context. You need manual spans inside the callback to see per-message latency.

**When manual instrumentation breaks down:** Teams under sprint pressure skip adding spans to new code paths. Within 6 months, your trace coverage degrades silently — you don't know what you're missing. Governance fix: require span coverage as part of service graduation (golden path enforcement via admission webhook or PR checklist).

---

### Tradeoff 2: DaemonSet vs Deployment vs Sidecar Collector Topology

**Resource cost vs isolation vs simplicity**

| Topology | Memory overhead | Blast radius | Tail sampling | Best for |
|---|---|---|---|---|
| DaemonSet | Low (1 per node) | Node-wide | Not native (no cross-pod span aggregation) | Agent/forward mode; high-volume clusters |
| Deployment | Medium (N replicas, shared) | Cluster-wide per replica | Yes (with loadbalancing exporter routing by trace ID) | Central gateway; tail sampling; multi-backend fan-out |
| Sidecar | High (1 per pod) | Single pod | No | Strict multi-tenant isolation; regulated workloads |

**Failure mode — DaemonSet:** A misconfigured Collector (e.g., infinite retry loop to a dead backend) consumes excessive CPU on the node, degrading all pods on that node. Mitigation: memory_limiter processor with conservative limits; health check readiness probe.

**Failure mode — Deployment (tail sampling):** A gateway pod restart loses its in-memory span buffer — all in-flight traces being held for tail sampling decision are dropped. Mitigation: use a persistent buffer (e.g., file-based queue) and configure exporters with retry-on-failure. Accept that tail sampling has a small inherent data loss window on pod restart.

**Recommended production pattern:** DaemonSet agents (no tail sampling, just batching + head sampling) → forward to Deployment gateway (tail sampling + fan-out). This gives you volume reduction at the edge and intelligence at the center, which is the same pattern as edge computing in CDN architecture.

---

### Tradeoff 3: Head-based vs Tail-based Sampling

**Cost vs Completeness**

| Dimension | Head-based | Tail-based |
|---|---|---|
| Decision point | Trace start (before outcome is known) | Trace completion (after all spans arrive) |
| Memory cost | Negligible — SDK coin-flip, no buffering | High — must buffer all in-flight spans |
| Captures all errors | No — an errored trace sampled at 1% is dropped 99% of the time | Yes — policy: keep 100% of traces with error spans |
| Captures all slow requests | No — a slow trace sampled at 1% is dropped 99% of the time | Yes — policy: keep 100% of traces with latency > SLO threshold |
| Complexity | Low — one config value (sample_ratio: 0.01) | High — stateful Collector required, trace ID routing, buffer sizing |
| Cost predictability | High — traffic × ratio = storage cost, deterministic | Low — error rate spikes → storage spikes |

**When head-based sampling is sufficient:** Your services have uniformly interesting error rates (errors are not rare events) and your trace backend is cheap enough to store 10% of all traffic. Also correct when you have no tail sampling infrastructure and need any sampling immediately.

**When tail-based sampling is mandatory:** You are measuring SLO compliance from traces (Google SRE golden signals via tracing). A P1 incident involving a rare error pattern (0.01% error rate) that was head-sampled away is a postmortem finding. Any system where "we might have dropped the evidence" is unacceptable requires tail-based sampling for error and slow-request traces.

**The hybrid approach (what hyperscalers actually use):** Head-based sampling at 1% for healthy traces (volume control) + tail-based sampling at 100% for error and latency-outlier traces (SLO compliance). This requires the Collector to first make a probabilistic head decision, then override that decision when a span arrives with an error attribute. The OTel Collector's tail sampling processor supports this with a combination of `probabilistic` and `status_code` sampling policies.

## Real-World Gotchas: Learn From Production Mistakes

### Gotcha 1: Trace Context Propagation Across Service Boundaries

**Mistake**: Instrumented service A with OTel. Calls service B. But trace ID didn't propagate → service B started new traces.

**Root cause**: Service A sent HTTP `traceparent` header (W3C standard). Service B's old instrumentation didn't read it.

**Fix**: Ensure all services (and middleware) propagate trace context. Use W3C Trace Context standard or Jaeger propagator. Test with:
```bash
curl -H "traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-a2fb4a1d1a96d312-01" http://service-b:8080/notify
```

**At T-Mobile**: We run a "trace propagation audit": weekly grep for missing `traceparent` headers in service mesh config. Caught 3 services silently dropping context.

---

### Gotcha 2: Cardinality Explosion on "Just for Debugging" Metrics

**Mistake**: Engineer added metric for "number of retries per request ID" to debug transient failures. Request ID has 1M unique values/day.

**Root cause**: Cardinality = 1M. After 30 days, Prometheus had 30M series. Cost exploded. Postgres queries to Prometheus took 30s (timeouts).

**Fix**: Use tags (`retry_count {bucket: "<50 retries"}`) instead of dimensions. Or move to logs: `log.structured(retries=47, request_id=XYZ, ...)` → queryable in Splunk, not a TSDB dimension.

**Lesson**: Every dimension is a cost lever. Default = low-cardinality dimensions only. High-cardinality data = logs or trace attributes.

---

### Gotcha 3: OTel Collector as New SPOF

**Mistake**: Deployed single Collector pod. Network blip dropped 50% of traces. Alerting went silent for 30 min.

**Root cause**: Apps emit OTLP → Collector. Collector down = no export. If apps can't reach Collector, they block on OTLP call (5s timeout x every span = 500ms per request slower).

**Fix**: (1) Run 3+ Collector replicas with load balancer. (2) Use Collector in "best effort" mode (apps don't wait for ack; timeout = drop and continue). (3) Have fallback: if Collector down, apps write to local file; drain file asynchronously.

**At T-Mobile**: Collector high-availability is non-negotiable. We use an anti-pattern: apps emit OTLP to a **sidecar Collector first** (always up), sidecar buffers and emits to aggregator. Sidecar down = pod restarts (isolation). Aggregator down = graceful backpressure (sidecars queue locally).

---

### Gotcha 4: Unintended PII in Traces/Metrics

**Mistake**: Engineer added `phone_number` to span attributes for debugging. Traces went to Datadog. Datadog was breached. PII exposed.

**Root cause**: No PII scrubbing in OTel pipeline. No data governance checks on what leaves the cluster.

**Fix**: (1) OTel Collector processor to drop PII attributes (regex on attribute keys: drop if matches `phone|email|ssn|card`). (2) Require code review for any span attribute that touches customer data. (3) Audit pipeline: run monthly to detect new PII.

**At T-Mobile**: We use a sidecar processor that scrubs PII before any export. Config:
```yaml
processors:
  attributes:
    actions:
      - key: phone_number
        action: delete
      - key: customer_email
        action: delete
```
