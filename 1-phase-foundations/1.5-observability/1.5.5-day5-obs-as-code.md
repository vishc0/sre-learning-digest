# Observability as Code and the SPACE Framework | Week 5 Day 5

## Why This Matters

Directors at cloud-native organizations are being asked to justify engineering investment in observability tooling — and the answer is no longer "we have dashboards." Interviewers at Staff/Principal level will probe whether you can *generate* SLOs from code, *automate* alert provisioning, and *translate* engineering health metrics into language a CFO understands. The SPACE framework is the 2025 replacement for DORA-only conversations: FAANG engineering leaders now cite it in design reviews and performance calibrations. Owning this topic end-to-end is a differentiator that separates operational SRE managers from platform engineering leaders.

---

## Analogy

Think about how T-Mobile's notification platform was governed during a major release cycle. You had:

- **DORA metrics** = the dashboard your incident command team watches during a go-live: how often are we deploying, are failures going up, how fast are we recovering?
- **SPACE framework** = the weekly ops review your VP runs: not just "did the system work" but "are engineers burning out, is toil killing velocity, is cross-team communication breaking down?"
- **Observability as code** = the equivalent of your TIBCO monitoring config being checked into source control instead of living in someone's brain or a shared drive. When you commit your SLO definition the same way you commit a Helm chart, it becomes reviewable, auditable, reproducible.

The SPACE framework is what happens when a VP of Engineering says: "DORA tells me about deployments. But I need to know if my teams are *healthy*." SPACE adds the human system to the technical system.

## 30-Second Version

Observability as code means your SLOs, dashboards, and alerts live in Git — they are reviewed, versioned, and deployed like application code. DORA measures deployment pipeline health across four numeric keys. SPACE extends DORA with human-system signals: satisfaction, performance, activity, communication, and efficiency. Directors use DORA to talk to engineering teams and CFOs; they use SPACE to talk to CTOs and People teams. Your Splunk-to-OTel migration strategy is fan-out: run both simultaneously, compare signal fidelity, then cut over — never rip-and-replace a production observability stack.

## 2-Minute Version

### Observability as Code: The Three Artifacts

**1. SLOs defined in YAML (Pyrra/Sloth)**

Both Pyrra and Sloth are Kubernetes-native tools that accept a custom resource definition (CRD) in YAML and emit Prometheus recording rules + alerting rules. You write:

```yaml
# This is a Pyrra SLO definition — it generates ~20 lines of PromQL automatically
apiVersion: pyrra.dev/v1alpha1
kind: ServiceLevelObjective
metadata:
  name: notification-api-availability
spec:
  target: "99.9"   # 99.9% availability
  window: 30d
  indicator:
    ratio:
      errors:
        metric: http_requests_total{job="notification-api", code=~"5.."}
      total:
        metric: http_requests_total{job="notification-api"}
```

Pyrra reads this, generates Prometheus rules, and creates a dashboard. You commit the YAML, CI/CD applies it.

**2. Grafana Dashboards as Code (JSONNET/Grafonnet)**

Grafana's native format is JSON (~500 lines for a moderate dashboard). JSONNET lets you write it as a template with variables and compile it to JSON. Grafonnet is a JSONNET library with Grafana-specific helpers. The workflow: write `.jsonnet` → compile with `jsonnet` CLI → POST JSON to Grafana API.

**3. Alerts in Terraform (Grafana provider)**

```hcl
resource "grafana_alert_rule" "error_budget_burn" {
  name      = "Error Budget Burn Rate > 5x"
  folder_uid = grafana_folder.slo.uid
  ...
}
```

### DORA Four Keys (Measurement)

| Key | What It Measures | How to Measure |
|---|---|---|
| Deployment Frequency | How often you ship to prod | Count prod deployments per day/week from CI/CD logs |
| Lead Time for Changes | Commit to prod deploy time | `deploy_timestamp - first_commit_timestamp` per PR |
| Change Failure Rate | % of deploys that caused an incident | `incidents_caused_by_deploy / total_deploys * 100` |
| MTTR | Mean time to restore after failure | `incident_resolved_at - incident_started_at` (average) |

### SPACE Framework (2021, extended 2025)

Developed by Nicole Forsgren (DORA co-creator) + GitHub researchers. Adds dimensions DORA misses:

| Dimension | What It Captures | Example Signal |
|---|---|---|
| **S**atisfaction | Developer experience, burnout risk | Survey NPS, retention rates, "toil hours" |
| **P**erformance | Quality and reliability of output | Defect escape rate, SLO attainment |
| **A**ctivity | Volume of work done | PRs merged, deployments, incidents handled |
| **C**ommunication/Collaboration | Cross-team effectiveness | PR review latency, doc freshness, on-call handoff quality |
| **E**fficiency | Flow and waste reduction | Build times, wait times, toil reduction YoY |

The key insight: **Activity metrics without Satisfaction metrics lie.** High deployment frequency with high burnout is not a healthy system.

## 5-Minute Version

### Why Observability as Code Is Not Optional at Staff Level

When a dashboard lives in someone's Grafana folder, it dies when they leave. When an alert lives in a Slack conversation, it never gets reviewed in postmortems. Observability as code solves three real problems:

1. **Auditability**: "When did this SLO change from 99.5% to 99.9%?" → `git log`
2. **Reproducibility**: Spin up a new environment and get the same dashboards automatically via GitOps
3. **Review**: SLO changes go through PR review — just like application code changes

**Sloth vs. Pyrra tradeoff**: Sloth is more flexible (works outside K8s, supports multiple backends). Pyrra is tighter K8s integration (operator model, auto-dashboard generation, UI). For a team already on EKS with Prometheus operator: Pyrra. For polyglot environments: Sloth.

### DORA Benchmarks (2024 State of DevOps Report)

| Level | Deploy Freq | Lead Time | CFR | MTTR |
|---|---|---|---|---|
| Elite | Multiple/day | < 1 hour | < 5% | < 1 hour |
| High | 1/week–1/day | 1 day–1 week | 5–10% | < 1 day |
| Medium | 1/month–1/week | 1 week–1 month | 10–15% | < 1 week |
| Low | < 1/month | > 6 months | > 15% | > 6 months |

T-Mobile notification platform context: 25M msg/day, 15-person team → you should be able to claim High/Elite on Deployment Frequency and MTTR. That's interview-ready evidence.

### SPACE in Practice: The 2025 Extension

The 2025 research extension (Forsgren, Murphy-Hill, et al.) added AI-augmented work measurement. Two new signals:
- **AI acceleration index**: How much faster are tasks completed with Copilot vs. without? (measurable via PR cycle time segmentation)
- **Cognitive load proxy**: Incidents per engineer per week as a burnout signal (not just system health)

### Director Framing: Which Metrics for Which Audience

| Audience | Metrics They Care About | Language to Use |
|---|---|---|
| Engineering Team | DORA four keys, SLO attainment, error budget remaining | "We're burning error budget 3x on Tuesdays — deployment window problem" |
| CTO | SPACE (all five dimensions), DORA trajectory | "Our Efficiency score is Low — toil is at 40% of eng time, blocking feature velocity" |
| CFO | MTTR cost, Change Failure Rate → incident cost, Lead Time → time-to-revenue | "Each P1 costs ~$45K in eng hours + revenue impact. CFR improvement from 12% to 6% saved $380K last quarter" |
| Board | Reliability as revenue protection, NPS correlation to uptime | "99.95% availability = $X protected ARR based on churn analysis" |

### Splunk → OTel Migration: Fan-Out Strategy

**Never rip-and-replace a production observability stack.** The fan-out pattern:

```
Application
    │
    ├──► OTel Collector ──► Splunk HEC (existing)
    │                  └──► Prometheus/Tempo/Loki (new)
    │
    └──► Legacy Splunk forwarder (failsafe)
```

Phase 1 (Weeks 1-4): Add OTel SDK alongside existing instrumentation. Send to both destinations. Compare signal fidelity — do alert thresholds fire at the same time?

Phase 2 (Weeks 5-8): Migrate dashboards one service at a time. Keep Splunk as source-of-truth until parity is confirmed.

Phase 3 (Weeks 9-12): Cut over alerting. Keep Splunk for compliance/long-retention. OTel stack becomes operational truth.

**Risk**: OTel SDK adds ~2-5ms latency on instrumentation initialization. Validate in staging before prod rollout. Profile the hot path.

**T-Mobile context**: Your notification platform sends 25M messages/day. Fan-out means 2x telemetry volume during transition — capacity-plan the collector fleet before starting. Horizontal scaling of the OTel collector is cheap (stateless, scales linearly).

## Director/VP Lens

**How a Director governs this:**

- Sets the SLO contract: "Notification delivery SLO is 99.9% over 30 days — this is the business commitment, not the team preference."
- Requires all SLO definitions to go through PR review with at least one other team's sign-off (API team + consumer team both approve the contract).
- Reviews DORA trajectory monthly — not individual contributor velocity, but systemic pipeline health.
- Uses SPACE to catch burnout signals before attrition — "We're High on Activity and Low on Satisfaction — that's a 6-month attrition leading indicator."
- Presents MTTR cost to CFO in dollar terms: "Our current MTTR is 47 minutes. If we hit Elite (< 1 hour — we're close), we avoid ~2 P1s per quarter that cost $45K each."

**How an IC implements this:**

- Writes the Pyrra/Sloth YAML, opens PR, applies via GitOps.
- Maintains the JSONNET dashboard templates and the Terraform alert rules.
- Instruments services with OTel SDKs using the fan-out pattern.
- Owns the DORA data pipeline: CI/CD events → metrics store → Grafana.

**The distinction interviewers probe**: A Director should be able to explain WHY the SLO target was chosen (business risk tolerance, not engineering preference) and how it connects to customer retention. An IC should be able to explain HOW the Prometheus recording rules are generated from the Pyrra CRD.

---

## ANSWER KEY — Comprehension Check

**A1.** The SPACE framework covers what DORA misses. Specifically, the **Satisfaction** dimension captures burnout, toil experience, and developer NPS — things that don't show up in deployment frequency. High Activity + Low Satisfaction is a known leading indicator of team attrition. (Bonus: the **Efficiency** dimension covers toil reduction specifically — wait times, manual steps, cognitive overhead.)

**A2.** Rip-and-replace breaks observability continuity. During the cutover window, if the Prometheus alert has a misconfigured threshold or the metric name differs slightly, you have a blind spot in production. The correct approach is **fan-out**: run both Splunk and Prometheus alerts simultaneously, validate they fire at the same time on the same events, then deprecate the Splunk alert only after signal parity is confirmed (typically 2-4 weeks of parallel operation).

**A3.** 99.5% availability over 30 days means the allowed downtime is:
- 30 days × 24 hours × 60 minutes = 43,200 minutes total
- 0.5% of 43,200 = 216 minutes = **3.6 hours** allowed downtime
- You had 5 hours (300 minutes) of downtime
- 300 > 216 → **Yes, you breached the SLO.** Error budget is exhausted and overdrawn by 84 minutes.

### Step 3: Write a JSONNET Dashboard Template

```bash
cat > ~/observability-lab/dashboards/notification-api.jsonnet << 'EOF'
// notification-api.jsonnet
// Grafonnet-style dashboard template for T-Mobile Notification API
// This compiles to Grafana's native JSON format

// --- Helper function: create a time series panel ---
local timeSeriesPanel(title, targets, gridPos) = {
  type: "timeseries",
  title: title,
  gridPos: gridPos,
  datasource: { type: "prometheus", uid: "prometheus" },
  fieldConfig: {
    defaults: {
      color: { mode: "palette-classic" },
      unit: "percentunit",
    },
  },
  options: {
    legend: { calcs: ["mean", "max"], displayMode: "table" },
  },
  targets: targets,
};

// --- Helper: Prometheus target ---
local promTarget(expr, legendFormat) = {
  expr: expr,
  legendFormat: legendFormat,
  datasource: { type: "prometheus", uid: "prometheus" },
};

// --- Service name variable (parameterized — change this for other services) ---
local service = "notification-api";

// --- Dashboard definition ---
{
  uid: "notification-api-slo-v1",
  title: "Notification API — SLO Dashboard",
  description: "Auto-generated from JSONNET. Do not edit in UI — edit notification-api.jsonnet",
  tags: ["slo", "notification", "auto-generated"],
  timezone: "browser",
  refresh: "30s",
  schemaVersion: 38,
  time: { from: "now-1h", to: "now" },

  // Template variables (dropdown selectors in Grafana UI)
  templating: {
    list: [
      {
        name: "env",
        type: "custom",
        query: "prod,staging,dev",
        current: { value: "prod", text: "prod" },
      }
    ]
  },

  // Panels array — each panel is a visual block on the dashboard
  panels: [

    // Panel 1: Availability (error rate inverted)
    timeSeriesPanel(
      title="Availability % (30m window)",
      targets=[
        promTarget(
          expr='1 - (rate(http_requests_total{job="' + service + '",code=~"5.."}[30m]) / rate(http_requests_total{job="' + service + '"}[30m]))',
          legendFormat="Availability"
        )
      ],
      gridPos={ x: 0, y: 0, w: 12, h: 8 }
    ),

    // Panel 2: Request rate
    timeSeriesPanel(
      title="Request Rate (req/sec)",
      targets=[
        promTarget(
          expr='rate(http_requests_total{job="' + service + '"}[5m])',
          legendFormat="{{code}}"
        )
      ],
      gridPos={ x: 12, y: 0, w: 12, h: 8 }
    ) + {
      // Override: request rate uses requests/sec not percent
      fieldConfig+: { defaults+: { unit: "reqps" } }
    },

    // Panel 3: P99 Latency
    timeSeriesPanel(
      title="P99 Latency (ms)",
      targets=[
        promTarget(
          expr='histogram_quantile(0.99, rate(http_request_duration_seconds_bucket{job="' + service + '"}[5m])) * 1000',
          legendFormat="P99"
        )
      ],
      gridPos={ x: 0, y: 8, w: 24, h: 8 }
    ) + {
      fieldConfig+: { defaults+: { unit: "ms" } }
    },

  ],
}
EOF

echo "JSONNET template written"
cat ~/observability-lab/dashboards/notification-api.jsonnet | wc -l
```

### Step 5: Wrap the JSON for Grafana API Import Format

Grafana's dashboard import API requires the JSON to be wrapped in a specific envelope. This step creates the API-ready payload.

```bash
cd ~/observability-lab/dashboards

# Grafana API requires: {"dashboard": {...}, "overwrite": true, "folderId": 0}
# jq wraps our compiled dashboard JSON into this envelope
jq '{
  "dashboard": .,
  "overwrite": true,
  "folderId": 0,
  "message": "Auto-deployed from JSONNET template — notification-api.jsonnet"
}' notification-api-dashboard.json > notification-api-api-payload.json

echo "=== API payload top-level keys ==="
jq 'keys' notification-api-api-payload.json

echo "=== Commit message field ==="
jq '.message' notification-api-api-payload.json
```

**EXPECTED_OUTPUT**:
```

  "dashboard",
  "folderId",
  "message",
  "overwrite"
]

"Auto-deployed from JSONNET template — notification-api.jsonnet"
```

WHY: The `message` field appears in Grafana's dashboard version history — this is your GitOps audit trail. A Director reviewing "who changed the SLO dashboard and when" can trace it back to a Git commit via this message.

### Step 7: Define a Pyrra SLO in YAML

```bash
cat > ~/observability-lab/slos/notification-delivery-slo.yaml << 'EOF'
# notification-delivery-slo.yaml
# Pyrra ServiceLevelObjective CRD definition
# Apply with: kubectl apply -f notification-delivery-slo.yaml
# Pyrra operator reads this and generates Prometheus recording rules + alerting rules

apiVersion: pyrra.dev/v1alpha1
kind: ServiceLevelObjective
metadata:
  name: notification-delivery-availability
  namespace: monitoring
  labels:
    team: notification-platform
    tier: critical
    owner: sre-team
  annotations:
    description: "99.9% of notification delivery API requests must succeed over a 30-day window"
    runbook: "https://wiki.internal/runbooks/notification-api-p1"
    slack-alert-channel: "#sre-alerts-critical"
spec:
  # The SLO target: 99.9% = 0.1% error budget
  # 30 days * 24h * 60m = 43,200 minutes total
  # 0.1% of 43,200 = 43.2 minutes allowed downtime per month
  target: "99.9"
  window: 30d

  # What "good" means for this service
  indicator:
    ratio:
      # Numerator: error requests (5xx responses from notification API)
      errors:
        metric: http_requests_total{
          job="notification-api",
          code=~"5.."
        }
      # Denominator: all requests
      total:
        metric: http_requests_total{
          job="notification-api"
        }

---
# Second SLO: Latency (P99 must be under 500ms for 95% of requests)
apiVersion: pyrra.dev/v1alpha1
kind: ServiceLevelObjective
metadata:
  name: notification-delivery-latency
  namespace: monitoring
  labels:
    team: notification-platform
    tier: critical
spec:
  # 95% of requests complete within the latency threshold
  target: "95"
  window: 30d

  indicator:
    latency:
      # Success = request completed within 500ms
      success:
        metric: http_request_duration_seconds_bucket{
          job="notification-api",
          le="0.5"
        }
      # Total = all requests (any bucket)
      total:
        metric: http_request_duration_seconds_count{
          job="notification-api"
        }
EOF

echo "=== SLO YAML written ==="
echo ""
echo "=== SLO names defined ==="
grep "name:" ~/observability-lab/slos/notification-delivery-slo.yaml

echo ""
echo "=== SLO targets ==="
grep "target:" ~/observability-lab/slos/notification-delivery-slo.yaml

echo ""
echo "=== Line count ==="
wc -l ~/observability-lab/slos/notification-delivery-slo.yaml
```

### Step 8: Simulate Pyrra Output — Generate Prometheus Rules

Pyrra normally runs as a K8s operator and generates PrometheusRule CRDs. This step simulates what Pyrra emits, so you understand what the automation produces.

```bash
cat > ~/observability-lab/slos/generated-prometheus-rules.yaml << 'EOF'
# AUTO-GENERATED by Pyrra operator — DO NOT EDIT MANUALLY
# Source: notification-delivery-slo.yaml
# Generated at: 2026-06-11T10:00:00Z

apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: pyrra-notification-delivery-availability
  namespace: monitoring
  labels:
    app.kubernetes.io/managed-by: pyrra
    pyrra.dev/slo: notification-delivery-availability
spec:
  groups:

  # Recording rules: pre-compute expensive PromQL so alerts are fast
  - name: pyrra-notification-delivery-availability.recording
    interval: 30s
    rules:

    # Error ratio over 5-minute window (short burn rate)
    - record: pyrra:notification_delivery_availability:ratio_rate5m
      expr: |
        sum(rate(http_requests_total{job="notification-api",code=~"5.."}[5m]))
        /
        sum(rate(http_requests_total{job="notification-api"}[5m]))

    # Error ratio over 30-minute window
    - record: pyrra:notification_delivery_availability:ratio_rate30m
      expr: |
        sum(rate(http_requests_total{job="notification-api",code=~"5.."}[30m]))
        /
        sum(rate(http_requests_total{job="notification-api"}[30m]))

    # Error ratio over 1-hour window
    - record: pyrra:notification_delivery_availability:ratio_rate1h
      expr: |
        sum(rate(http_requests_total{job="notification-api",code=~"5.."}[1h]))
        /
        sum(rate(http_requests_total{job="notification-api"}[1h]))

    # Error ratio over 6-hour window (trend detection)
    - record: pyrra:notification_delivery_availability:ratio_rate6h
      expr: |
        sum(rate(http_requests_total{job="notification-api",code=~"5.."}[6h]))
        /
        sum(rate(http_requests_total{job="notification-api"}[6h]))

  - name: pyrra-notification-delivery-availability.alerting
```

```bash
def calculate_deployment_frequency(deployments, window_days=30):
    """
    Deployment Frequency = total deployments / window in days
    Elite: multiple times/day
    High: 1/day to 1/week
    """
    total = len(deployments)
    per_day = total / window_days
    per_week = per_day * 7

    if per_day >= 1:
        level = "Elite" if per_day > 1 else "High"
    elif per_week >= 1:
        level = "High"
    else:
        level = "Medium"

    return {
        "total_deployments": total,
        "window_days": window_days,
        "deployments_per_day": round(per_day, 2),
        "deployments_per_week": round(per_week, 1),
        "dora_level": level,
    }


def calculate_lead_time(deployments):
    """
    Lead Time for Changes = deploy_timestamp - first_commit_timestamp
    Elite: < 1 hour
    High: 1 day to 1 week
    """
    lead_times_hours = []
    for deploy_ts, commit_ts, _ in deployments:
        delta = deploy_ts - commit_ts
        hours = delta.total_seconds() / 3600
        lead_times_hours.append(hours)

    avg_hours = mean(lead_times_hours)
    max_hours = max(lead_times_hours)
    min_hours = min(lead_times_hours)

    if avg_hours < 1:
        level = "Elite"
    elif avg_hours < 24:
        level = "High"
    elif avg_hours < 168:  # 1 week
        level = "Medium"
    else:
        level = "Low"

    return {
        "avg_lead_time_hours": round(avg_hours, 2),
        "max_lead_time_hours": round(max_hours, 2),
        "min_lead_time_hours": round(min_hours, 2),
        "dora_level": level,
    }


def calculate_change_failure_rate(deployments):
    """
    Change Failure Rate = deploys that caused incidents / total deploys * 100
    Elite/High: < 15%
    Medium: 15-30%
    Low: > 30%
    """
    total = len(deployments)
    failures = sum(1 for _, _, caused_incident in deployments if caused_incident)
    cfr_percent = (failures / total) * 100

    if cfr_percent < 5:
        level = "Elite"
    elif cfr_percent < 15:
        level = "High"
    elif cfr_percent < 30:
        level = "Medium"
    else:
        level = "Low"

    return {
        "total_deployments": total,
        "failed_deployments": failures,
        "change_failure_rate_percent": round(cfr_percent, 1),
        "dora_level": level,
    }


def calculate_mttr(incidents):
    """
    MTTR = mean(resolved_at - opened_at) for deploy-caused incidents
    Elite: < 1 hour
    High: < 1 day
    """
    # DORA MTTR counts only service-impacting incidents (here: all, regardless of cause)
    # NOTE: some teams count all incidents; Google/DORA counts deploy-caused. We show both.
    all_mttr_minutes = []
    deploy_caused_mttr_minutes = []

    for opened, resolved, caused_by_deploy in incidents:
        minutes = (resolved - opened).total_seconds() / 60
        all_mttr_minutes.append(minutes)
        if caused_by_deploy:
            deploy_caused_mttr_minutes.append(minutes)

    avg_all = mean(all_mttr_minutes)
    avg_deploy_caused = mean(deploy_caused_mttr_minutes) if deploy_caused_mttr_minutes else 0

    def level(minutes):
        if minutes < 60:
            return "Elite"
        elif minutes < 1440:  # 1 day
            return "High"
        elif minutes < 10080:  # 1 week
            return "Medium"
        else:
            return "Low"

    return {
        "total_incidents": len(incidents),
        "deploy_caused_incidents": len(deploy_caused_mttr_minutes),
        "mttr_all_incidents_minutes": round(avg_all, 1),
        "mttr_deploy_caused_minutes": round(avg_deploy_caused, 1),
        "mttr_all_hours": round(avg_all / 60, 2),
        "dora_level": level(avg_all),
    }
```

## Tradeoffs and Failure Modes

### Tradeoff 1: Sloth vs. Pyrra vs. Hand-written PromQL

**Sloth** generates recording rules and multi-window alerts from a simple YAML spec. Works outside Kubernetes. Flexible. Requires you to manually create the Grafana dashboard.

**Pyrra** is a Kubernetes operator — it generates rules AND creates a Grafana dashboard AND exposes an SLO overview UI. Tightly coupled to K8s.

**Hand-written PromQL** gives maximum flexibility but requires deep PromQL knowledge and produces rules no one reviews because they are "too complex to change."

**Interview-ready answer**: "We chose Pyrra because we're fully on EKS and the operator pattern matches our GitOps workflow. If we had services outside K8s, I'd use Sloth for those and keep Pyrra for K8s services. The worst outcome is hand-written PromQL that lives in a forgotten ConfigMap and gets copy-pasted incorrectly."

### Tradeoff 2: DORA as Team Health Proxy is Incomplete

DORA measures the deployment pipeline. It does NOT measure: code quality, security posture, team morale, customer satisfaction, or the correctness of what was deployed. A team that ships broken features fast has excellent DORA scores and a terrible product.

**The failure mode**: Organizations that optimize for DORA scores instead of outcomes. Teams start making small, trivial commits to inflate deployment frequency. Lead time is gamed by splitting tickets. Change failure rate is gamed by not linking incidents to deploys.

**Director mitigation**: DORA is a health check, not a performance evaluation. Never tie compensation or performance reviews to DORA numbers — you will get the metric, not the behavior.

### Tradeoff 3: Fan-Out Migration Doubles Telemetry Cost

Running Splunk and OTel simultaneously during migration doubles your telemetry storage and ingestion costs for 90-180 days. For a 25M message/day platform, this is significant.

**The failure mode**: Finance pushes to shorten the parallel operation window to reduce cost, creating pressure to cut over before parity is confirmed. This is how you get a missed alert at 2am three weeks into the new stack.

**Director mitigation**: Quantify the cost upfront — "parallel operation costs $X/month for 3 months = $3X total." Compare that to "one missed P1 during blind cutover = $45K in incident response plus potential SLA breach." The math almost always favors the parallel operation. Present the tradeoff explicitly to leadership before the migration starts, not during.
