# OTel Collector on EKS — Distributed Tracing in Production | Week 5 Day 3

## Why This Matters

At T-Mobile scale (25M messages/day), a single user request traverses 8–15 microservices before a notification lands on a phone. When that request is slow or fails, Splunk logs from one service tell you what happened there — but not why it happened or which upstream call caused it. Distributed tracing stitches the entire journey into one queryable timeline, cutting MTTR from hours to minutes. Every Director/VP SRE interview for cloud-native orgs will ask: "How do you observe cross-service latency?" — distributed tracing with OTel is the expected answer.

---

## Analogy

Think of distributed tracing like **airline baggage tracking**, but for requests.

When you check a bag at T-Mobile's DFW hub (the frontend service), it gets a baggage tag — that tag number is the **Trace ID**. Every conveyor belt, sorter, and handler that touches your bag scans it and records: "Bag [trace-id] was at Station [span-id] from T+0ms to T+120ms." When your bag gets lost at the Denver transfer (the slow backend call), you pull up that one tag number and see the entire journey — not just Denver's log.

**The OTel Collector is the baggage-tag reader infrastructure** — installed at every airport (every node or service), collecting all the scan events (spans), and shipping them to a central tracking system (Jaeger/X-Ray/Grafana Tempo).

Incident command parallel: a **Trace ID = a P1 incident ticket number**. Every team that touches the incident puts their update on that ticket. Without the ticket number propagated between teams, you get parallel war rooms that can't compare notes.

## Vocabulary Translation Table

| Their Term | Your Existing Mental Model |
|---|---|
| **Trace** | A single end-to-end RabbitMQ message journey — from publish to ack, through every queue and consumer |
| **Span** | One hop in that journey — e.g., "time spent in the MQ consumer processing" |
| **Trace ID** | P1 incident bridge number — all teams use it to tie their updates together |
| **Span ID** | Each team's JIRA subtask under the parent incident |
| **Parent Span** | The incident commander's bridge call — owns the overall duration |
| **Child Span** | Each team's remediation subtask — nested under the parent |
| **W3C TraceContext header** | The paging bridge number printed on every escalation email — it travels with the message so the next handler knows which incident this belongs to |
| **OTel Collector** | Splunk Heavy Forwarder — sits on the host, receives telemetry from apps, batches and ships it downstream |
| **ADOT** | AWS's branded fork of the Splunk UF — same binary, pre-configured for AWS backends (X-Ray, CloudWatch) with AWS auth baked in |
| **DaemonSet** | One Splunk UF per server — same pattern, one collector per K8s node |
| **Sidecar** | Dedicated Splunk UF per application container — higher isolation, higher cost |
| **Head-based sampling** | Deciding at the start of an incident whether to create a full timeline — fast but you might skip the "boring-looking" P1 that turns catastrophic |
| **Tail-based sampling** | Waiting until the incident is over before deciding whether to archive the full timeline — catches all P1s but needs a temporary buffer for every incident in flight |
| **Cardinality explosion** | Like adding `user_id` as a Splunk field index — every unique user becomes a unique index entry; at 25M users you've just created 25M metric series and a $500K/month bill |
| **Exemplar** | A hyperlink from a Splunk dashboard panel directly to the raw log/trace that caused that spike — metrics point to evidence |
| **Auto-instrumentation** | AppDynamics agent injection — the app doesn't need code changes; the agent wraps JVM/Node calls automatically |

## 30-Second Version

OpenTelemetry is a vendor-neutral standard for collecting traces, metrics, and logs. The OTel Collector is a standalone binary that receives telemetry from your apps, processes it (filters, batches, adds metadata), and exports it to backends like Jaeger, Prometheus, or AWS X-Ray. ADOT is AWS's supported distribution of OTel — same protocol, plus pre-built IAM auth and X-Ray exporters. Distributed tracing works by attaching a Trace ID to the first request and propagating it through every downstream call via HTTP headers, so the entire request journey can be reassembled into a single timeline.

---

## 2-Minute Version

### How a Distributed Trace Is Built

1. **Ingress/frontend service** receives an HTTP request. OTel SDK generates a `Trace ID` (128-bit UUID) and a root `Span ID`. It records: start time, service name, operation name.

2. **When frontend calls backend**, OTel injects the Trace ID + Span ID into the outgoing HTTP headers — specifically the `traceparent` header (W3C TraceContext standard):
   ```
   traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
                   ^  ^----trace id (128bit)----^  ^span id^  ^flags
   ```

3. **Backend service** reads `traceparent` from incoming headers. It creates a *child span* with the same Trace ID but a new Span ID, and marks the frontend's span as the parent.

4. **OTel Collector** on each node receives spans from the apps (via OTLP protocol, typically gRPC port 4317), batches them, and exports to the backend (Jaeger, Tempo, X-Ray).

5. **Trace backend** reassembles all spans sharing a Trace ID into a waterfall timeline — frontend (0-500ms) → backend DB call (100-450ms) → cache hit (200-210ms).

### Why the Collector Exists (vs. direct export from app)

Without a collector, each app SDK must: buffer spans, handle backpressure, implement retry logic, manage auth tokens for the backend. This puts operational complexity inside application code. The Collector extracts all of that. It also lets you change backends (Jaeger → Grafana Tempo → X-Ray) without redeploying your apps — just reconfigure the Collector's exporter section.

## 5-Minute Version

### ADOT vs. Upstream OTel Collector

The upstream OTel Collector is a community binary with 50+ receivers, processors, and exporters. ADOT (AWS Distro for OpenTelemetry) is AWS's opinionated distribution:

| Feature | Upstream OTel Collector | ADOT |
|---|---|---|
| AWS X-Ray exporter | Community-maintained | AWS-maintained, tested on EKS |
| IAM auth (IRSA) | Manual config required | Baked into EKS add-on |
| EKS add-on lifecycle | Not available | `aws eks create-addon --addon-name adot` |
| CloudWatch EMF exporter | Community | AWS-maintained |
| Support | Community GitHub issues | AWS Support tickets |
| Update cadence | Faster, less stable | Slower, AWS-tested |

**When to choose ADOT**: You're on EKS and want AWS-managed lifecycle (upgrades via `eks update-addon`), IRSA-based auth without manual secret management, and AWS Support coverage. **When to stay upstream**: You need the latest collector features, you're multi-cloud, or you have a specific processor/receiver that ADOT hasn't included yet.

### DaemonSet vs. Deployment vs. Sidecar

**DaemonSet (one collector per node)**
- Pattern: Same as Splunk UF per server — every node runs one collector.
- Pros: No per-pod overhead, shared CPU/memory for telemetry collection, simple scaling (scales with nodes).
- Cons: Can't do tail-based sampling (requires seeing all spans for a trace — which are scattered across nodes), collector restart affects all pods on that node simultaneously.
- Use when: Head-based sampling, high pod density per node, want low overhead.

**Deployment (centralized collectors, N replicas)**
- Pattern: Dedicated collector pool — 2-5 replicas behind a ClusterIP Service.
- Pros: Can do tail-based sampling (all spans route to same pool), horizontal scaling independent of nodes, easier resource tuning.
- Cons: Network hop from every pod to the collector pool, becomes a single point of failure if undersized.
- Use when: Tail-based sampling required (error capture), lower pod count, centralized processing.

**Sidecar (one collector per pod)**
- Pattern: AppDynamics agent pattern — collector container in every pod's spec.
- Pros: Maximum isolation (one app's cardinality explosion can't kill another's collector), no network hop, granular per-service config.
- Cons: 3x–10x the resource cost, operational complexity (every deployment carries collector config), upgrade = rolling restart of every deployment.
- Use when: Strict multi-tenant isolation, compliance requirement for data not leaving pod boundary, or per-service sampling requirements differ dramatically.

### Head-Based vs. Tail-Based Sampling

**Head-based sampling**: At trace start, flip a coin (e.g., sample 10% of traces). Decision made before the trace completes.
- Pro: Zero buffering needed, minimal collector memory, 10% cost = 10% storage.
- Con: The 90% you dropped may contain your only P1 trace. A request that looks normal at T+0 may 500 at T+5s — you already decided not to sample it.
- Use: High-volume baseline traffic where errors are rare and well-understood.

**Tail-based sampling**: Buffer all spans for a trace until the trace is complete (typically 30s window), then decide whether to keep it based on rules (keep if error, keep if >2s, keep 1% of healthy traces).
- Pro: Guaranteed capture of all errors and slow traces — exactly what you want for incident investigation.
- Con: Requires buffering ALL in-flight traces in collector memory. At 25M messages/day, a 30s window holds ~8,600 concurrent traces. At 50 spans/trace × 1KB/span = 430MB minimum buffer, more realistically 2–5GB.
- Use: When you need "always capture errors" SLA. Requires Deployment topology (not DaemonSet) because all spans for a trace must reach the same collector.

**Real-world hybrid**: Head-sample at 5% for baseline, then use tail-sampling rules to always keep error traces and >p99 latency traces. This is what mature orgs run.

### Cardinality Explosion — Why It Destroys Your Observability Budget

Prometheus stores metrics as time series. A time series is uniquely identified by its **label set**: `{service="frontend", method="POST", status="200"}`.

If you add a label `user_id`, and you have 1 million users, that single counter becomes **1 million separate time series**. Prometheus memory grows linearly with time series count. At 1M series × 50 bytes/series × 60 data points/hour = 3GB/hour just for one metric.

**High-cardinality labels to NEVER put on metrics**:
- `user_id`, `customer_id`, `account_id`
- `request_id`, `trace_id`, `session_id`
- `pod_name` (use `deployment` instead — pod names are ephemeral)
- `ip_address`
- Free-text fields (error messages, SQL queries)

**Rule**: If the label can take more than ~1,000 unique values in production, it does NOT belong on a metric. It belongs in a trace span attribute instead — traces are designed for high-cardinality per-request data.

### Exemplars: Bridging Metrics and Traces

An Exemplar is a sample data point attached to a metric that includes the Trace ID of the request that generated that specific measurement. In Grafana: you see a latency histogram spike → click the diamond marker on the spike → browser opens the specific trace that caused that spike.

Implementation: Prometheus 2.43+ stores exemplars natively. OTel SDK automatically injects `trace_id` and `span_id` into Prometheus exemplars when both OTel and Prometheus instrumentation are active. In Grafana, enable "Exemplars" on the data source and link it to your Jaeger/Tempo datasource.

## Director/VP Lens

### How an IC implements this:
- Writes OTel Collector DaemonSet YAML, configures receivers/processors/exporters
- Tunes batch processor settings (batch size, timeout) to balance latency vs. throughput
- Debugs span drops by watching `otelcol_processor_dropped_spans` metric
- Writes sampling rules in tail-sampling processor config

### How a Director/VP governs this:

**Platform decisions** (you make once, team executes):
1. **Single collector topology per org** — DaemonSet for low-overhead clusters, Deployment for error-capture-critical services. Document this as a golden path, not a per-team decision.
2. **Cardinality budget** — Set a policy: "No metric label with cardinality >1,000. Violations caught in CI via `promtool check rules`." This prevents the $500K/month Prometheus incident.
3. **Sampling SLA** — "All 5xx traces must be captured" is a reliability contract. Tail-sampling must be provisioned to meet this. Budget the 2-5GB collector memory per cluster.
4. **Trace retention policy** — Traces are expensive to store. 7-day hot storage (Jaeger in-memory or Tempo S3), 30-day cold. Metrics live 13 months. Logs live 90 days. These numbers map to your observability budget.

**Interview answer frame (Director level)**: "At T-Mobile, we had per-team instrumentation with no standards — eight teams, eight exporters, four backends. I chartered the observability platform team to define a golden path: ADOT DaemonSet, tail-sampling with error-always-keep policy, cardinality lint in CI, and Grafana as the single pane. We reduced MTTD on cross-service incidents by 60% in one quarter."

**AI-augmented observability** (Staff/Principal lens for 2025-2026 interviews):
- LLMs can now summarize a 50-span trace into plain English: "The slowdown originated in the auth-service Redis timeout at span 7, causing cascading retries in the API gateway."
- AWS Bedrock + CloudWatch integration allows natural language queries: "Show me all traces slower than 2 seconds that ended in a 5xx last hour."
- Your governance role: Define which trace data can be sent to LLM APIs (PII scrubbing policy), and establish SLO for AI-assisted triage (MTTD target with AI assist vs. without).

## Comprehension Check

Answer before proceeding to the lab. You need 2/3 to unlock.

**Q1**: Your frontend service calls three downstream services. A user reports their request took 8 seconds. You have logs from all three services but they each say "completed successfully." How does distributed tracing help you find the problem, and what specific artifact would you look for first?

**Q2**: Your SRE team is debating sampling strategy. Service A is your payment processor — every failed transaction must be traceable. Service B is your CDN health-check endpoint — 10,000 requests/minute, almost all 200 OK. What sampling strategy would you choose for each, and why?

**Q3**: A junior engineer adds `request_id` as a Prometheus label on your API latency histogram. Your cluster has 50,000 requests/minute. What is the operational consequence within 24 hours, and what is the correct place to store request-level identifiers?

---

## ANSWER KEY — Comprehension Check

**A1**: Distributed tracing helps because logs only show what happened within one service — they don't show the *time spent waiting* for downstream calls. With a Trace ID, you pull the waterfall view and immediately see which span has the longest duration. The artifact to look for first is the **span duration breakdown** — specifically, which child span consumed most of the 8 seconds. You'd look for the `traceparent` header in the original request or query your trace backend for traces >5s in that time window. If logs all say "successful," the 8 seconds is almost certainly in network wait time or a slow DB/cache call — visible in traces, invisible in application logs.

**A2**:
- **Payment processor (Service A)**: Tail-based sampling with rule: "always keep traces with status=error OR duration>2s, sample 1% of healthy traces." This guarantees every failed transaction has a full trace for audit and debugging. The cost of missing a payment failure trace exceeds the infrastructure cost of the buffer.
- **CDN health-check (Service B)**: Head-based sampling at 0.1% (one in 1,000). These requests are homogeneous, low-risk, and the traces are not diagnostic — you already know what a healthy health check looks like. Sampling all of them wastes storage and creates noise.
- Key insight: Sampling strategy is a business risk decision, not a technical preference.

**A3**: With 50,000 requests/minute, each request gets a unique `request_id`. After 24 hours (72M requests), Prometheus has created **72 million unique time series** for that one histogram. This causes: Prometheus OOM crash (typical at ~5-10M active series on standard hardware), scrape timeouts, dashboards fail to load, and alerting becomes unreliable. The storage cost would be in the hundreds of GB/day. The correct place for `request_id` is as a **span attribute** in the trace — traces are designed for high-cardinality, per-request data. Prometheus metrics are designed for low-cardinality aggregates.

Create the Collector configuration. This will:

```bash
# Now deploy the OTel Collector as a DaemonSet using the ADOT CRD
kubectl apply -f - <<'EOF'
apiVersion: opentelemetry.io/v1alpha1
kind: OpenTelemetryCollector
metadata:
  name: adot-daemonset
  namespace: observability
spec:
  # DaemonSet mode = one collector pod per node
  mode: daemonset

  # Use the IRSA-enabled service account we created
  serviceAccount: adot-collector

  # Resource limits for the collector sidecar on each node
  resources:
    requests:
      memory: "128Mi"
      cpu: "50m"
    limits:
      memory: "512Mi"
      cpu: "500m"

  config: |
    receivers:
      # OTLP receiver: apps send spans here via gRPC
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317   # All interfaces, port 4317
          http:
            endpoint: 0.0.0.0:4318   # HTTP fallback

    processors:
      # Batch: accumulate spans and send in bulk (reduces network calls)
      batch:
        timeout: 5s          # Wait up to 5s before flushing
        send_batch_size: 256 # Or flush when 256 spans accumulated

      # Memory limiter: prevent OOM if app sends spike of spans
      memory_limiter:
        check_interval: 1s
        limit_mib: 400
        spike_limit_mib: 100

      # K8s attributes: adds pod name, namespace, node to every span
      # This is the "enrichment" step - same concept as Splunk field extraction
      k8sattributes:
        auth_type: serviceAccount
        passthrough: false
        extract:
          metadata:
            - k8s.pod.name
            - k8s.namespace.name
            - k8s.node.name
            - k8s.deployment.name

    exporters:
      # Send to local Jaeger for visualization
      otlp/jaeger:
        endpoint: jaeger.observability.svc.cluster.local:14250
        tls:
          insecure: true  # Lab only — production uses TLS

      # Debug exporter: prints spans to collector logs (lab troubleshooting)
      debug:
        verbosity: detailed

    service:
      pipelines:
        traces:
          receivers: [otlp]
          processors: [memory_limiter, k8sattributes, batch]
          exporters: [otlp/jaeger, debug]
EOF
```

**EXPECTED_OUTPUT**:
```
opentelemetrycollector.opentelemetry.io/adot-daemonset created
```

### Step 3: Deploy Sample 2-Service App with OTel Auto-Instrumentation

We'll deploy:
- **frontend** service: Python Flask app, calls backend
- **backend** service: Python Flask app, simulates work with variable latency

Auto-instrumentation means: no OTel code in the app. The OTel Operator injects an init-container that wraps the Python process.

```bash
# Enable OTel auto-instrumentation for Python in our app namespace
kubectl create namespace demo-app

# Create an Instrumentation CRD - tells the operator HOW to instrument Python apps
kubectl apply -f - <<'EOF'
apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: python-instrumentation
  namespace: demo-app
spec:
  # Which OTel Collector to send traces to
  # The DaemonSet exposes a port on the node - use the node IP
  exporter:
    endpoint: http://$(OTEL_EXPORTER_OTLP_ENDPOINT)

  propagators:
    - tracecontext   # W3C TraceContext - the traceparent header standard
    - baggage

  sampler:
    type: parentbased_traceidratio
    argument: "1.0"  # Sample 100% for the lab (production: 0.1 for 10%)

  python:
    env:
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        valueFrom:
          fieldRef:
            fieldPath: status.hostIP  # Node IP - where DaemonSet collector listens
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: "http://$(OTEL_EXPORTER_OTLP_ENDPOINT):4317"
EOF
```

**EXPECTED_OUTPUT**:
```
namespace/demo-app created
instrumentation.opentelemetry.io/python-instrumentation created
```

```bash
# Deploy the backend service first
kubectl apply -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend
  namespace: demo-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: backend
  template:
    metadata:
      labels:
        app: backend
      annotations:
        # THIS annotation triggers OTel auto-instrumentation injection
        instrumentation.opentelemetry.io/inject-python: "python-instrumentation"
    spec:
      containers:
      - name: backend
        image: python:3.11-slim
        command: ["/bin/sh", "-c"]
        args:
          - |

**EXPECTED_OUTPUT**:
```
=== WRONG: High-cardinality label (request_id) ===
Number of time series for http_requests_with_id_total BEFORE:
0

10

## Lab ANSWER KEY

**Step 0 fails — "cluster already exists"**:
Run `kubectl get nodes` first. If you already have a working cluster from a previous lab, skip Step 0 and set `export CLUSTER_NAME=<your-existing-cluster-name>`. Use `eksctl get clusters --region us-east-1` to list existing clusters.

**Step 1 fails — "OIDC provider already associated"**:
This is not an error — just a warning that it was done in a previous lab. Continue to the `create iamserviceaccount` step.

**Step 1 fails — ADOT add-on fails with "DEGRADED" status**:
```bash
aws eks describe-addon --cluster-name $CLUSTER_NAME --addon-name adot --region $REGION
```
Look at `addon.health.issues`. Common cause: insufficient node resources. Check `kubectl describe nodes` for `Allocatable` CPU/memory. The ADOT operator needs ~200Mi RAM.

**Step 2 — DaemonSet pods stuck in "Pending"**:
```bash
kubectl describe pod -n observability -l app.kubernetes.io/component=opentelemetry-collector
```
Look for "Insufficient memory" or "Insufficient cpu" events. Reduce `resources.limits.memory` in the CRD to `256Mi` and reapply.

**Step 3 — App pods stuck in "Init:Error"**:
The OTel auto-instrumentation init container failed. Check:
```bash
kubectl logs -n demo-app <pod-name> -c opentelemetry-auto-instrumentation
```
Common fix: Instrumentation CRD endpoint misconfiguration. Verify the `hostIP` field path is correct: `status.hostIP`.

**Step 4 — No traces appearing in Jaeger**:
1. Check collector logs: `kubectl logs -n observability -l app.kubernetes.io/component=opentelemetry-collector`
2. Look for "Failed to export traces" — often an endpoint connectivity issue
3. Verify Jaeger service is reachable: `kubectl exec -n observability <collector-pod> -- wget -q http://jaeger.observability.svc.cluster.local:14250 -O -`
4. Alternative: Use the debug exporter to verify spans are arriving at the collector: `kubectl logs -n observability <collector-pod> | grep "InstrumentationScope"`

**Step 6 — Cardinality numbers don't match expected**:
If you see fewer time series than expected, check that you ran the traffic-generating loops completely. The Python app must be responding (check with `curl -s http://localhost:9090/health`). If the port-forward dropped, re-establish it.

**TEARDOWN — "cluster not found" during delete**:
The cluster was already deleted or used a different name. Verify with `eksctl get clusters --region us-east-1`. Check the AWS Console EKS page for any orphaned CloudFormation stacks: look for stacks named `eksctl-*` and delete them manually if present.

## Tradeoffs & Failure Modes

### 1. Tail-Based Sampling Reliability vs. Memory Cost
**The tradeoff**: Tail-based sampling guarantees error trace capture — critical for compliance and incident investigation. But it requires buffering all in-flight traces, which scales with both request volume AND trace duration. A service with 30-second traces that handles 10,000 req/min needs 5 million span-slots in memory at any time. Under-provision the buffer and the collector starts dropping the oldest spans — exactly the long-running transactions (like payment processing) you most need to capture.

**Failure mode**: Collector OOM during a traffic spike. The spike causes high latency → longer traces → more spans in buffer → buffer full → collector drops spans or crashes. You lose traces precisely when you need them most — during an incident.

**How to prevent**: Size the buffer for your p99 trace duration, not average. Use `memory_limiter` processor to gracefully drop new traces before OOMing (losing new incoming data is better than crashing and losing everything). Set up an alert on `otelcol_processor_dropped_spans > 0`.

### 2. Auto-Instrumentation Convenience vs. Observability Depth
**The tradeoff**: Auto-instrumentation (AppDynamics-style injection) gets you traces in 30 minutes with zero code changes — invaluable for a 100-service org that can't afford to modify every app. But auto-instrumentation only traces at the framework level (HTTP calls, DB queries) — it knows nothing about your business logic. "Checkout service called payment service" is visible; "checkout service calculated tax using rule X" is not.

**Failure mode**: During an incident, you know the latency is in the checkout service's `calculateTotal()` method — but the trace shows only "checkout-service: POST /checkout, 3.2s" with no breakdown. Without custom spans inside `calculateTotal()`, you've hit the limit of auto-instrumentation.

**How to prevent**: Use auto-instrumentation as the baseline. Add manual custom spans for business-critical paths (payment calculation, fraud check, inventory reservation). Frame this as "instrumentation by business capability priority" — the 20% of code paths that matter for 80% of incidents get manual spans.

### 3. Trace Context Propagation Failures in Async Systems
**The tradeoff**: The W3C TraceContext header works perfectly for synchronous HTTP calls. In async systems (Kafka, RabbitMQ, SQS), the `traceparent` must be propagated as a message header — but many existing producers/consumers were written before OTel and don't propagate it. The result: traces break at every async boundary, giving you fragmented traces rather than end-to-end visibility.

**Failure mode**: You deploy OTel across all services. Your trace shows frontend → API gateway → order-service (5ms). But order-service publishes to RabbitMQ, and the notification service that consumes the message has no parent trace. You see two disconnected traces instead of one end-to-end trace. The slow 8-second notification delay is invisible.

**How to prevent**: Explicitly instrument every message producer/consumer that crosses an async boundary. Add `traceparent` propagation to your RabbitMQ/Kafka producer wrapper libraries — do this once in the shared library, enforce its use via internal package policy. For legacy consumers you can't modify, accept the broken trace context and use correlation IDs in log fields as a fallback bridge between trace segments.
