## Analogy (Middleware First)

**Simple LangChain chain = a RabbitMQ message pipeline**
You declare: input → transform → output. Messages flow in sequence. No branching. No memory of what happened before. This is fine for 80% of use cases — just like a simple topic-exchange in RabbitMQ handles most notification flows without needing a saga orchestrator.

**LangGraph = Apache Airflow for AI**
When you need conditional routing ("if the retrieval confidence is low, re-query with a broader prompt"), retry logic with state checkpointing ("what step did we fail on?"), and loop detection ("are we stuck?") — that is Airflow-level orchestration. LangGraph is exactly this: a directed graph where nodes are LLM calls or tool invocations, edges are conditional transitions, and state is persisted across steps.

**LlamaIndex = the indexing and query engine layer only**
Think of it as Elasticsearch + query DSL for unstructured document data. You use LlamaIndex when the primary problem is "I have 10,000 PDFs and I need to make them queryable by an LLM." It handles chunking strategy, embedding storage, retrieval ranking, and re-ranking. LangChain and LangGraph can call LlamaIndex as a retrieval tool — they are not competitors, they complement each other.

**Plain Python = bash script**
When you have one LLM call, no branching, no retrieval — just `client.messages.create(...)` and parse the response. This is a bash one-liner. Do not import LangChain for a bash one-liner. The framework has overhead: import time (~3–5 seconds), hidden abstraction layers, framework-version lock-in, and debugging complexity.

## 30-Second Version

Three tools, three jobs:
- **LangChain LCEL**: Build sequential LLM pipelines fast. Use when steps are fixed and linear.
- **LlamaIndex**: Make large document collections queryable. Use when the problem is retrieval.
- **LangGraph**: Build stateful, branching, retrying AI workflows. Use when you need Airflow-style control flow over LLM calls.

Rule of thumb: Start with plain Python. Add LangChain when you need composability. Add LangGraph when you need state and branching. Add LlamaIndex when your retrieval logic is the hardest part.

## 2-Minute Version

### LangChain LCEL

LCEL (LangChain Expression Language) uses the pipe `|` operator to compose *Runnables*. Every component — a prompt template, an LLM call, an output parser — is a Runnable with a `.invoke()` method. Chaining them with `|` creates a new Runnable that runs them sequentially.

```python
chain = prompt | llm | output_parser
result = chain.invoke({"question": "What is our p99 latency?"})
```

This is composable, lazy (doesn't execute until `.invoke()`), and supports streaming and async out of the box. The failure mode is that error messages become confusing — you get an exception somewhere in the chain and the stack trace is framework-internal.

### LlamaIndex

LlamaIndex's core objects are:
1. **Documents** — raw text or structured data you load
2. **Index** — the processed, chunked, embedded representation stored in a vector DB
3. **Query Engine** — accepts a natural language query, retrieves relevant chunks, synthesizes an answer

LlamaIndex handles the retrieval problem better than LangChain's retriever abstraction — it has more chunking strategies (sentence window, hierarchical node parser), re-ranking, and hybrid search (vector + keyword). Use it when your document corpus is complex.

### LangGraph

LangGraph is a graph execution engine built on top of LangChain. You define:
- **State**: a TypedDict that flows through the graph (like a saga's shared context)
- **Nodes**: Python functions that receive state, do work, return updated state
- **Edges**: routes between nodes — can be conditional (`if state["confidence"] < 0.7 → re_retrieve`)

LangGraph maintains a checkpointer (SQLite or Postgres) so a workflow can be paused, inspected, and resumed. This is critical for human-in-the-loop workflows and for debugging failed runs.

### When to Use Each

| Scenario | Use |
|---|---|
| Single LLM call, parse response | Plain Python |
| Sequential: prompt → LLM → parse → transform | LangChain LCEL |
| RAG over large corpus | LlamaIndex (retriever) + LangChain (chain) |
| Conditional retry, branching decisions, loops | LangGraph |
| Human approval step mid-workflow | LangGraph with interrupt |
| Multi-agent coordination | LangGraph multi-agent pattern |

## 5-Minute Version (Tradeoffs, Failure Modes, Design Decisions)

### LangChain LCEL — Deep

LCEL's strength is composability with streaming. The `.stream()` method works automatically across the full chain — the LLM token stream flows through each Runnable and arrives at the caller as it's generated. This is table stakes for any production chatbot interface.

**Hidden complexity**: LangChain's callback system runs on every invoke/stream call. Callbacks are how tracing tools (LangSmith) and custom logging hooks attach. If you add a callback that blocks, it slows every call. If it throws an exception and you haven't wrapped it, it can crash your chain.

**Version instability**: LangChain has been the most rapidly evolving Python library in the AI space. `langchain`, `langchain-core`, `langchain-community`, `langchain-openai`, `langchain-anthropic` are separate packages. Version mismatches cause import errors that are non-obvious. Pin all versions in `requirements.txt` and review release notes before upgrading.

**Retry behavior**: LangChain's `with_retry()` wraps a Runnable with exponential backoff. It is silent by default — retries do not surface to your logging unless you hook into callbacks. In production, this means a rate-limit event looks like slow response time, not a retry storm. SRE implication: instrument the callback layer or use LangSmith tracing from day one.

### LlamaIndex — Deep

LlamaIndex's **node parser** (chunking) strategy is the most consequential retrieval decision you make:
- **SentenceSplitter**: Respects sentence boundaries. Good for prose documents.
- **SentenceWindowNodeParser**: Stores surrounding sentences as metadata. The embedding represents a small chunk, but retrieved context includes neighbors. Improves answer quality on dense technical docs.
- **HierarchicalNodeParser**: Creates parent/child chunks. Query against small chunks, but return the parent for context. Best for structured documents (manuals, runbooks).

Wrong chunking strategy is the #1 cause of "the RAG answers are terrible" problems. This is an operational tuning problem — exactly like Splunk index bucket sizing affects search performance.

**Re-ranking**: After vector retrieval returns top-N chunks, a re-ranker model (Cohere Rerank, BGE re-ranker) scores them by relevance to the query. This is a second-pass filter. Cost: adds ~100–200ms latency. Benefit: dramatically reduces hallucination from low-relevance chunks. Trade this off against latency SLO.

### LangGraph — Deep

LangGraph's state machine model has three operational implications:

1. **Infinite loop detection**: LangGraph supports a `recursion_limit` (default: 25). If your conditional edges have a bug and the graph cycles, it throws `GraphRecursionError`. Always set this explicitly and alert on it.

2. **Checkpointer durability**: The default in-memory checkpointer loses state on restart. In production, use `SqliteSaver` (single-node) or `PostgresSaver` (multi-node). Without a checkpointer, a failed mid-graph execution cannot be resumed — you start over and re-spend tokens.

3. **Human-in-the-loop interrupt**: LangGraph supports `interrupt_before` and `interrupt_after` on any node. The graph pauses, serializes state to the checkpointer, and waits. A human reviews, approves or edits, and the graph resumes. This is the architecture behind "approve this action before the AI takes it" — a critical safety pattern for agentic systems. SRE leadership should mandate this for any workflow that takes external actions (writes to DB, sends API calls, modifies infrastructure).

### Over-Engineering Warning

The biggest failure mode is using LangGraph for a bash pipeline. If your workflow is:
1. Load user query
2. Retrieve 3 documents
3. Generate answer
4. Return

That is four sequential steps with no branching, no retry, no state. Use LangChain LCEL. Adding LangGraph adds: graph compilation overhead, state serialization, checkpoint I/O, and 200–400ms cold-start time. Do not pay that cost unless you need conditional routing or state recovery.

### AI's Impact on This Domain

GitHub Copilot and Claude can generate LCEL chains and LangGraph graphs from a description in seconds. The IC task shifts from *writing* the framework glue to *auditing* it: Are retries bounded? Is state checkpointed? Are token costs tracked per workflow run? Does the retrieval latency have an SLO? Principal SRE engineers own the observability and governance layer over AI pipelines — the framework code is increasingly AI-generated.

**Q1.** Your team built a customer support bot. The flow is: receive question → retrieve 3 FAQ docs → generate answer → return. There is no branching, no retry logic, no state. A junior engineer wants to use LangGraph because "it's more powerful." What do you tell them, and what do you recommend instead?

**Q2.** A LangChain chain is making OpenAI API calls. During load testing, you notice that response times spike to 45 seconds occasionally, but no errors appear in your application logs. What is the most likely cause, and what instrumentation change would make this visible?

**Q3.** You are asked to design an AI pipeline that: (a) retrieves internal runbook documents, (b) generates a remediation plan, (c) asks a human to approve the plan, and (d) executes the approved steps via API calls. Which framework(s) do you use for each step, and why?

---

## ANSWER KEY — Comprehension Check

**A1.** LangGraph adds meaningful overhead (state serialization, checkpoint I/O, graph compilation) with no benefit for a linear four-step workflow with no branching or retry requirements. Recommend LangChain LCEL: `retriever | prompt | llm | output_parser`. This is idiomatic, composable, supports streaming, and is significantly simpler to debug. Use LangGraph when you need conditional edges or state recovery — not for every AI pipeline by default.

**A2.** LangChain's `with_retry()` wrapper is silently retrying rate-limit (429) or transient (5xx) errors from the OpenAI API. The retries add latency but succeed eventually — so the application sees slow responses, not errors. The fix: add a callback handler (or use LangSmith tracing) that logs retry events with the error type, retry count, and delay. Also add explicit `.with_retry(stop_after_attempt=3)` and alert when retry count > 0 in production.

**A3.**
- **Step (a) — Retrieval**: LlamaIndex query engine over an indexed runbook corpus. Use `SentenceWindowNodeParser` for dense operational docs. LlamaIndex's retrieval quality and re-ranking options outperform LangChain's built-in retrievers for complex document corpora.
- **Step (b) — Generation**: LangChain LCEL chain: `retrieved_context | prompt_template | llm | plan_parser`.
- **Step (c) — Human approval**: LangGraph with `interrupt_before` on the execution node. State is checkpointed to PostgreSQL. The human reviews the serialized plan state, approves or edits, and resumes the graph. This is the only framework that supports pause-and-resume with state persistence.
- **Step (d) — Execution**: LangGraph node that calls external APIs as LLM tools. The node is wrapped in a try/except that updates state with success/failure per API call, enabling partial retry without re-running the approval step.

### Step 3 — Build the LangChain RAG Chain (LCEL)

```bash
cat > ~/training/week10-day1-orchestration/rag_chain.py << 'EOF'
"""
LangChain LCEL RAG Chain over SRE Runbooks
Architecture: load docs -> split -> embed -> store in ChromaDB -> retrieve -> generate

This is the SIMPLE version: linear pipeline, no state, no branching.
Use this for 80% of RAG use cases.
"""

import os
import time
from pathlib import Path

# LangChain document loading and splitting
from langchain_community.document_loaders import DirectoryLoader, TextLoader
from langchain_text_splitters import MarkdownHeaderTextSplitter, RecursiveCharacterTextSplitter

# LangChain embeddings and vector store
from langchain_community.vectorstores import Chroma
from langchain_community.embeddings import HuggingFaceEmbeddings  # FREE - runs locally

# LangChain LLM and chain components
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# ----- STEP 1: Load documents -----
print("=" * 60)
print("STEP 1: Loading runbook documents")
print("=" * 60)

runbook_dir = Path.home() / "training" / "week10-day1-orchestration" / "runbooks"
loader = DirectoryLoader(
    str(runbook_dir),
    glob="**/*.md",
    loader_cls=TextLoader,
    loader_kwargs={"encoding": "utf-8"}
)
raw_docs = loader.load()
print(f"Loaded {len(raw_docs)} documents")
for doc in raw_docs:
    print(f"  - {doc.metadata['source'].split('/')[-1]} ({len(doc.page_content)} chars)")

# ----- STEP 2: Split documents into chunks -----
print("\n" + "=" * 60)
print("STEP 2: Splitting documents into chunks")
print("=" * 60)

# RecursiveCharacterTextSplitter respects paragraph/sentence boundaries
# chunk_size=500 chars ~ 125 tokens — small enough for precise retrieval
# chunk_overlap=50 chars — prevents losing context at chunk boundaries
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n## ", "\n### ", "\n\n", "\n", " "]  # respect markdown headers first
)
chunks = splitter.split_documents(raw_docs)
print(f"Split into {len(chunks)} chunks")
print(f"Average chunk size: {sum(len(c.page_content) for c in chunks) // len(chunks)} chars")
print("Sample chunk (first):")
print(f"  Content: {chunks[0].page_content[:200]}...")
print(f"  Metadata: {chunks[0].metadata}")

# ----- STEP 3: Embed and store in ChromaDB -----
print("\n" + "=" * 60)
print("STEP 3: Creating embeddings and storing in ChromaDB")
print("=" * 60)

# HuggingFaceEmbeddings runs LOCALLY — no API cost for embedding
# Model: all-MiniLM-L6-v2 is 22MB, fast, good quality for English operational text
print("Loading embedding model (downloads ~22MB on first run)...")
t0 = time.time()
embeddings = HuggingFaceEmbeddings(
    model_name="all-MiniLM-L6-v2",
    model_kwargs={"device": "cpu"}
)

# Store in ChromaDB — persists to disk so we don't re-embed on every run
persist_dir = str(Path.home() / "training" / "week10-day1-orchestration" / "chromadb")
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory=persist_dir,
    collection_name="sre_runbooks"
)
t1 = time.time()
print(f"Embedded and stored {len(chunks)} chunks in {t1-t0:.1f}s")
print(f"ChromaDB persisted to: {persist_dir}")

# ----- STEP 4: Build retriever -----
print("\n" + "=" * 60)
print("STEP 4: Configuring retriever")
print("=" * 60)

# k=3: retrieve top 3 most semantically similar chunks
# score_threshold=0.3: reject chunks with similarity < 30% (prevents irrelevant context)
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 3}
)
print("Retriever configured: similarity search, top 3 chunks")

# Test retrieval before building full chain
test_query = "pod is restarting constantly"
test_results = retriever.invoke(test_query)
print(f"\nTest retrieval for: '{test_query}'")
print(f"Retrieved {len(test_results)} chunks:")
for i, doc in enumerate(test_results):
    source = doc.metadata['source'].split('/')[-1]
    print(f"  [{i+1}] {source}: {doc.page_content[:100]}...")

# ----- STEP 5: Build LCEL chain -----
print("\n" + "=" * 60)
print("STEP 5: Building LCEL RAG chain")
print("=" * 60)

# System prompt: frames the LLM as an SRE assistant with specific response format
# WHY this prompt structure: role + context injection + output constraints
# reduces hallucination and keeps answers operational
SYSTEM_PROMPT = """You are an SRE assistant. You help on-call engineers diagnose and remediate incidents.

Use ONLY the context provided below from our internal runbooks. Do not add information not present in the context.
If the context does not contain sufficient information to answer, say: "I don't have a runbook for this — escalate to the relevant team."

Context from runbooks:
{context}
"""

USER_PROMPT = "Question: {question}\n\nProvide: (1) Likely cause, (2) Immediate diagnosis steps, (3) Remediation action."

prompt = ChatPromptTemplate.from_messages([
    ("system", SYSTEM_PROMPT),
    ("human", USER_PROMPT)
])

# Claude claude-haiku-3: fastest, cheapest Claude model — ideal for RAG where most cost is retrieval, not generation
llm = ChatAnthropic(
    model="claude-haiku-3-20240307",
    temperature=0,        # deterministic — we want consistent operational guidance
    max_tokens=512,       # runbook answers should be concise
    timeout=30            # fail fast — don't wait more than 30s for LLM response
)

# Helper to format retrieved docs into a single context string
def format_docs(docs):
    return "\n\n---\n\n".join([
        f"[Source: {doc.metadata['source'].split('/')[-1]}]\n{doc.page_content}"
        for doc in docs
    ])

# LCEL chain using pipe operator:
# RunnablePassthrough() passes the original question through unchanged
# retriever gets question, returns docs
# format_docs converts doc list to string
# prompt fills template with context + question
# llm generates response
# StrOutputParser extracts text from AIMessage
chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

print("Chain assembled: retriever | format_docs | prompt | llm | parser")
print("Chain type:", type(chain).__name__)

# ----- STEP 6: Run test queries -----
print("\n" + "=" * 60)
print("STEP 6: Running test queries")
print("=" * 60)

test_questions = [
    "My Cassandra p99 latency is spiking to 800ms. What do I check?",
    "RabbitMQ queue has 15000 messages and consumers are not processing. What do I do?",
    "Several pods are in CrashLoopBackOff. Walk me through the diagnosis."
]

for i, question in enumerate(test_questions, 1):
    print(f"\n--- Query {i} ---")
    print(f"Q: {question}")
    t0 = time.time()
    answer = chain.invoke(question)
```

### Step 4 — Convert to LangGraph with Conditional Retry Logic

Now we add operational intelligence: if the retrieved context has low relevance, re-query with a broader search before answering. This requires state and conditional branching — LangGraph's domain.

```bash
cat > ~/training/week10-day1-orchestration/langgraph_rag.py << 'EOF'
"""
LangGraph RAG Workflow with Conditional Retry
Architecture: retrieve -> grade_relevance -> [retry if low confidence] -> generate

This demonstrates:
1. State management across nodes
2. Conditional edges (branch on relevance score)
3. Loop with recursion limit (prevent infinite retry)
4. State inspection after completion

WHY LangGraph here (not LCEL):
- We need to DECIDE whether to retry based on retrieval quality
- We need to TRACK how many retries we've done
- We need to persist state for debugging failed runs
These three requirements = LangGraph territory
"""

import os
import time
from typing import TypedDict, List, Annotated
from pathlib import Path

# LangGraph core
from langgraph.graph import StateGraph, END
from langgraph.graph.message import add_messages

# LangChain components (reused from previous step)
from langchain_community.vectorstores import Chroma
from langchain_community.embeddings import HuggingFaceEmbeddings
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.documents import Document

# ============================================================
# STATE DEFINITION
# TypedDict defines the "saga log" — everything the graph knows
# about this workflow run at any point in time
# ============================================================
class RAGState(TypedDict):
    question: str                    # original user question (immutable)
    search_query: str                # current search query (may be broadened on retry)
    retrieved_docs: List[Document]   # documents retrieved in this iteration
    retrieval_score: float           # average relevance score (0.0 - 1.0)
    retry_count: int                 # how many times we've retried retrieval
    answer: str                      # final generated answer
    status: str                      # workflow status for observability

MAX_RETRIES = 2
RELEVANCE_THRESHOLD = 0.35  # below this, retry with broader query

# ============================================================
# INITIALIZE SHARED COMPONENTS
# (same vector store as Step 3 — reusing persisted ChromaDB)
# ============================================================
print("Initializing components...")
embeddings = HuggingFaceEmbeddings(
    model_name="all-MiniLM-L6-v2",
    model_kwargs={"device": "cpu"}
)
persist_dir = str(Path.home() / "training" / "week10-day1-orchestration" / "chromadb")
vectorstore = Chroma(
    persist_directory=persist_dir,
    embedding_function=embeddings,
    collection_name="sre_runbooks"
)
llm = ChatAnthropic(
    model="claude-haiku-3-20240307",
    temperature=0,
    max_tokens=512,
    timeout=30
)
print("Components initialized.\n")

# ============================================================
# NODE DEFINITIONS
# Each node is a Python function: takes State, returns updated State
# Think of each node as a stage in an Airflow DAG
# ============================================================

def retrieve_node(state: RAGState) -> RAGState:
    """
    Node 1: Retrieve relevant documents.
    Uses state["search_query"] — this may be the original question or a broadened query.
    Returns docs with their relevance scores.
    """
    print(f"  [retrieve] Query: '{state['search_query']}' (retry #{state['retry_count']})")
    
    # similarity_search_with_relevance_scores returns (doc, score) tuples
    # score is cosine similarity: 1.0 = identical, 0.0 = unrelated
    results = vectorstore.similarity_search_with_relevance_scores(
        state["search_query"],
        k=3
    )
    
    docs = [doc for doc, score in results]
    scores = [score for doc, score in results]
    avg_score = sum(scores) / len(scores) if scores else 0.0
    
    print(f"  [retrieve] Retrieved {len(docs)} docs, avg relevance score: {avg_score:.3f}")
    for i, (doc, score) in enumerate(results):
        source = doc.metadata['source'].split('/')[-1]
        print(f"    [{i+1}] {source}: score={score:.3f}, preview='{doc.page_content[:60]}...'")
    
    return {
        **state,
        "retrieved_docs": docs,
        "retrieval_score": avg_score,
        "status": "retrieved"
    }

def grade_relevance_node(state: RAGState) -> RAGState:
    """
    Node 2: Assess whether retrieved docs are relevant enough to answer.
    This is the decision point — determines whether we retry or proceed.
    
    In production this could use an LLM grader for more nuanced scoring.
    Here we use the embedding similarity score as a proxy.
    """
    score = state["retrieval_score"]
    retry_count = state["retry_count"]
    
    print(f"  [grade] Relevance score: {score:.3f}, threshold: {RELEVANCE_THRESHOLD}, retries used: {retry_count}/{MAX_RETRIES}")
    
    if score < RELEVANCE_THRESHOLD and retry_count < MAX_RETRIES:
        print(f"  [grade] Score below threshold — will retry with broader query")
        status = "needs_retry"
    elif score < RELEVANCE_THRESHOLD and retry_count >= MAX_RETRIES:
        print(f"  [grade] Score below threshold but retry limit reached — proceeding with best available")
        status = "low_confidence_answer"
    else:
        print(f"  [grade] Score above threshold — proceeding to generation")
        status = "ready_to_generate"
    
    return {**state, "status": status}

def broaden_query_node(state: RAGState) -> RAGState:
    """
    Node 3: Broaden the search query for retry.
    Simple heuristic: strip specific error messages, keep operational keywords.
    In production: use an LLM to rephrase the query.
    """
    original = state["question"]
    retry_count = state["retry_count"] + 1
    
    # Broadening strategy: remove specific values, add operational context keywords
    # Example: "p99 latency 800ms" -> "latency performance degradation diagnosis"
    broaden_prompt = ChatPromptTemplate.from_messages([
        ("system", "You are an SRE query rewriter. Rewrite the query to be broader and more general, focusing on the operational category (performance, availability, capacity, etc.). Return only the rewritten query, no explanation."),
        ("human", "Original query: {query}\nRewritten query:")
    ])
    
    rewrite_chain = broaden_prompt | llm | StrOutputParser()
    broader_query = rewrite_chain.invoke({"query": original})
    broader_query = broader_query.strip()
    
    print(f"  [broaden] Original: '{original}'")
    print(f"  [broaden] Broader:  '{broader_query}'")
    
    return {
        **state,
        "search_query": broader_query,
        "retry_count": retry_count,
        "status": "retrying"
    }

def generate_node(state: RAGState) -> RAGState:
    """
    Node 4: Generate final answer from retrieved context.
    Includes confidence indicator based on retrieval score.
    """
    print(f"  [generate] Generating answer with {len(state['retrieved_docs'])} docs (confidence: {state['retrieval_score']:.3f})")
    
    # Format context with source attribution
    context = "\n\n---\n\n".join([
        f"[Source: {doc.metadata['source'].split('/')[-1]}]\n{doc.page_content}"
        for doc in state["retrieved_docs"]
    ])
    
    confidence_note = ""
    if state["status"] == "low_confidence_answer":
        confidence_note = "\n\nNOTE: Retrieval confidence is low. This answer may not fully match your situation — escalate to the relevant team if unsure."
    
    prompt = ChatPromptTemplate.from_messages([
        ("system", f"""You are an SRE assistant. Answer using ONLY the provided runbook context.
Context:
{context}"""),
        ("human", "Question: {question}\n\nProvide: (1) Likely cause, (2) Immediate steps, (3) Remediation.")
    ])
    
    chain = prompt | llm | StrOutputParser()
    answer = chain.invoke({"question": state["question"]})
    answer = answer + confidence_note
    
    return {**state, "answer": answer, "status": "complete"}

# ============================================================
# GRAPH ASSEMBLY
# This is where we define the control flow — the "Airflow DAG"
# ============================================================

def route_after_grading(state: RAGState) -> str:
    """
    Conditional edge function: returns the name of the next node.
    This is the branching logic — like a conditional trigger in Airflow.
    """
    if state["status"] == "needs_retry":
        return "broaden_query"  # -> broaden_query_node
    else:
        return "generate"       # -> generate_node (covers both ready and low_confidence)

# Build the graph
print("Assembling LangGraph workflow...")
workflow = StateGraph(RAGState)

# Add nodes (each is a named processing stage)
workflow.add_node("retrieve", retrieve_node)
workflow.add_node("grade_relevance", grade_relevance_node)
workflow.add_node("broaden_query", broaden_query_node)
workflow.add_node("generate", generate_node)

# Define edges (the routing logic)
workflow.set_entry_point("retrieve")                           # start here
workflow.add_edge("retrieve", "grade_relevance")              # always grade after retrieve
workflow.add_conditional_edges(                                # branch after grading
    "grade_relevance",
    route_after_grading,
    {
        "broaden_query": "broaden_query",   # low score + retries remaining
        "generate": "generate"              # good score OR retry limit reached
    }
)
workflow.add_edge("broaden_query", "retrieve")                # retry: broaden -> retrieve -> grade again
workflow.add_edge("generate", END)                            # generation is terminal

# Compile the graph (validates edges, prepares for execution)
graph = workflow.compile()
print("Graph compiled successfully.\n")

# ============================================================
# VISUALIZE THE GRAPH STRUCTURE (text representation)
# ============================================================
print("Graph structure:")
print("  START -> retrieve -> grade_relevance")
print("             grade_relevance -> [if needs_retry] -> broaden_query -> retrieve (loop)")
print("             grade_relevance -> [if ready/low_confidence] -> generate -> END")
print()
```

## Lab ANSWER KEY (Diagnostics Per Step)

### Step 0 — If package install fails:
- **Symptom**: `ERROR: Could not find a version that satisfies the requirement langchain==0.3.25`
- **Cause**: Pinned version may have been yanked or your pip is stale
- **Fix**: `pip install --upgrade pip`, then retry. If still failing, remove version pin and use latest: `pip install langchain langchain-core langchain-anthropic langgraph`
- **Lesson**: Always test version pins in CI before committing them to a shared requirements file.

### Step 1 — If API key check shows 0 characters:
- **Symptom**: `API key length: 0 characters`
- **Cause**: `export` command was run in a different shell session (state does not persist across WSL2 terminal sessions unless added to `~/.bashrc`)
- **Fix**: `echo 'export ANTHROPIC_API_KEY="sk-ant-..."' >> ~/.bashrc && source ~/.bashrc`

### Step 2 — If files show wrong line count:
- **Symptom**: Files exist but are smaller than expected
- **Cause**: Heredoc (`<< 'EOF'`) syntax issue in copy-paste — terminal may have interpreted special characters
- **Fix**: Copy each heredoc block individually, verify with `wc -l ~/training/week10-day1-orchestration/runbooks/*.md`

### Step 3 — If embedding model download is slow or fails:
- **Symptom**: `OSError: [Errno 110] Connection timed out` during HuggingFace model download
- **Cause**: HuggingFace Hub is blocked on your network, or rate-limited
- **Fix**: Set `TRANSFORMERS_OFFLINE=1` and download model separately: `python3 -c "from sentence_transformers import SentenceTransformer; SentenceTransformer('all-MiniLM-L6-v2')"`

### Step 3 — If retrieval returns wrong documents:
- **Symptom**: "pod is restarting constantly" retrieves Cassandra docs instead of EKS docs
- **Cause**: ChromaDB collection may have been created with wrong documents from a previous run
- **Fix**: Delete the persisted collection and recreate:
  ```bash
  rm -rf ~/training/week10-day1-orchestration/chromadb
  python3 rag_chain.py  # will recreate from scratch
  ```

### Step 4 — If LangGraph raises `GraphRecursionError`:
- **Symptom**: `langgraph.errors.GraphRecursionError: Recursion limit of 10 reached`
- **Cause**: The conditional edge logic is looping more than expected — likely the relevance threshold is too high for your test data
- **Fix**: Temporarily lower `RELEVANCE_THRESHOLD` to `0.15` to observe graph completing, then tune upward

### Step 4 — If `broaden_query` node produces the same query:
- **Symptom**: Retry count increments but search query is identical to original
- **Cause**: LLM with `temperature=0` is deterministic — identical prompt produces identical output
- **Fix**: This is a design limitation. In production, add examples to the rewrite prompt or use `temperature=0.3` for the rewrite call only.

## STAR Anchor (T-Mobile Framing Per Question)

**Q1 — LangGraph forcing function:**
"At T-Mobile, our notification platform processes 25M messages/day. When we explored AI-assisted runbook lookup during incident response, the first version was a simple chain — question in, answer out. We hit the architectural limit when we needed conditional behavior: if the retrieval confidence was low, we needed to broaden the query before answering rather than give a low-quality response to an on-call engineer at 2am. That conditional branch on real-time retrieval scoring — with state preserved across the retry loop — is exactly what pushed us toward LangGraph. The trigger is always: do I have branching logic that depends on runtime data? If yes, the bash pipeline model breaks down."

**Q2 — Diagnosing unexpected API spend:**
"First question I ask: is LangSmith or any callback-based tracing enabled? That is the LangChain equivalent of Splunk indexing — without it, you are flying blind. If not, I add a `BaseCallbackHandler` subclass that logs `on_llm_start` and `on_llm_end` events with token counts to our existing Splunk pipeline immediately. Then I look at token counts per invocation path — a retry loop without a bound will show up as token counts 2-3x higher than expected on the same query pattern. I also check whether `with_retry()` is configured anywhere without an explicit stop condition — silent retries are the #1 cause of unexpected spend in LangChain pipelines."

**Q3 — Guardrails for agentic kubectl execution:**
"I would not let any agent execute kubectl commands in production without three controls in place. One: a LangGraph `interrupt_before` on every action node — the plan is shown to the on-call engineer, they approve or reject, then execution proceeds. This is non-negotiable for any external action. Two: a read-only dry-run mode that executes `kubectl --dry-run=client` and shows the diff before actual execution. Three: scope restriction via a Kubernetes service account with RBAC limited to the specific namespace and resource types the agent is allowed to touch — not cluster-admin. These are not optional guardrails, they are the go-live criteria."

**Q4 — SLO on a RAG pipeline:**
"I frame it as three SLIs. Availability: the pipeline returns a response within 30 seconds, 99.5% of requests. Retrieval quality: the retrieval relevance score exceeds 0.35 (our empirically-tuned threshold) on 90% of queries — below that we count it as a degraded response. Answer relevance: we run a lightweight LLM-as-judge evaluation on a random 1% sample and flag answers that score below 3/5 — this feeds into a monthly error budget review, not a real-time alert. The error budget is consumed by: retrieval failures, LLM timeouts, and low-quality-answer samples. We review the burn rate weekly — same mechanics as our notification platform SLO governance."

**Q5 — AI framework dependency governance:**
"Same pattern we use for any high-velocity dependency in a 15-person team: a platform team owns a curated, pinned set of approved versions in a shared `pyproject.toml` base configuration. Engineers can use any version in their local experiments, but production deployments pull from the approved baseline. When a new LangChain major version drops, the platform team validates breaking changes in a sandbox environment, updates the baseline with migration notes, and issues an ADR. We treat it exactly like a Kubernetes minor version upgrade — you do not roll it out the week it drops. The AI-specific addition: we also review token cost implications of new model versions before updating `ChatAnthropic` or `ChatOpenAI` model names in the baseline."

## Tradeoffs and Failure Modes (Top 3)

### 1. Framework Abstraction Hides Failure Signals

**Tradeoff**: LangChain's composable Runnables make building pipelines fast. But when a chain fails, the stack trace is deep in framework internals — `langchain_core.runnables.base.RunnableSequence._invoke()` — not your code. Debugging takes 3x longer than equivalent plain Python.

**Failure mode**: Silent retry storms. `with_retry()` retries by default on any exception, including rate limits. Without callback instrumentation, a 429 rate-limit event looks like p95 latency spiking from 2s to 45s. No errors in your application logs. No alerts fire. Your API bill is 5x expected at end of month.

**Mitigation**: Enable LangSmith tracing from day one, even in development. Add explicit `stop_after_attempt=3` on all retry configurations. Set `max_tokens` on all LLM calls to prevent runaway token consumption.

---

### 2. LangGraph State Explosion in Long-Running Workflows

**Tradeoff**: LangGraph's persistent state enables powerful recovery and human-in-the-loop patterns. But state grows with every retry loop iteration — retrieved docs, rewritten queries, intermediate scores — all serialized to the checkpointer.

**Failure mode**: In a workflow that retries 10 times with 3 documents per retrieval, the serialized state can exceed 500KB. With a PostgreSQL checkpointer under load, state writes become a bottleneck. Observed symptom: LangGraph node execution times spike from 200ms to 2s as state size grows.

**Mitigation**: Keep state lean. Store only what is needed for routing decisions and final answer generation. Use IDs/references instead of full document content in state. Set `recursion_limit` conservatively (10 for most workflows). Implement state compaction for long-running agentic loops.

---

### 3. LlamaIndex vs LangChain Retriever Lock-In

**Tradeoff**: LlamaIndex and LangChain both have retriever abstractions that are not directly interchangeable. If you build your retrieval layer in LlamaIndex and your chain layer in LangChain, you have two dependency trees, two versioning risks, and two debugging surfaces.

**Failure mode**: LlamaIndex releases a breaking change to their query engine API. Your LangChain chain now throws `AttributeError: 'VectorStoreIndex' object has no attribute 'as_retriever'`. You have to upgrade both libraries simultaneously, which may have conflicting `pydantic` version requirements — a notoriously painful resolution in the Python AI ecosystem.

**Mitigation**: Define an internal retriever interface (abstract base class with `.retrieve(query: str) -> List[Document]`) that wraps whichever library you use. Your LangChain chains call your interface, not LlamaIndex directly. Swap the underlying implementation without changing the chain code. This is the strangler fig pattern applied to AI dependencies.
