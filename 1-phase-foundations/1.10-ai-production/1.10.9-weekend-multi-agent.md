## Analogy

Think of this as your **incident command structure, automated**.

In a real Sev1, you have:
- **Incident Commander** (receives page, coordinates) → this is the Orchestrator Agent
- **Subject Matter Experts** (someone checks the runbook, someone checks pod status, someone drafts the incident ticket) → these are the Tools
- **Call recorder + timeline** → this is Langfuse tracing
- **Legal/compliance reviewer who scrubs PII before the postmortem is published** → this is Bedrock Guardrails

The orchestrator does not do the work itself. It routes, waits for inputs, synthesizes, and produces the artifact. That is exactly what this agent does.

In RabbitMQ terms: the alert is the message, the orchestrator is the consumer routing to bound queues, each tool is a worker queue, and the incident report is the processed ack that exits the system.

## 2-Minute Version

The agent runs a ReAct loop: it receives an alert payload, decides which tools to call in what order, calls them, and synthesizes the results. Tool 1 pulls matching runbook context from a Bedrock Knowledge Base you built in Week 9 (or a mock if that is not available). Tool 2 returns a realistic JSON blob of pod statuses, mimicking what `kubectl get pods -o json` returns. Tool 3 takes all gathered context and formats a structured Markdown incident report.

Langfuse wraps every tool call as a span — you can see the full trace in a dashboard exactly like AppDynamics shows a distributed transaction. Bedrock Guardrails sits on the output path: if the report accidentally contains an IP address, email, or other PII (simulated adversarial test), Guardrails redacts it before it exits.

The deliverable is a GitHub repo with five files: `agent.py`, `tools.py`, `guardrails_config.json`, trace screenshots, and `PORTFOLIO_README.md`.

## 5-Minute Version (Tradeoffs, Failure Modes, Design Decisions)

**Why Claude via Bedrock instead of OpenAI?** AWS Bedrock keeps data within your VPC boundary — critical for any enterprise dealing with incident data that may contain customer identifiers. At T-Mobile scale, that is not optional.

**Why Langfuse instead of native CloudWatch?** Langfuse gives you LLM-specific observability: token counts, latency per span, prompt versions, and cost per trace. CloudWatch sees HTTP calls but does not understand that a 3-second latency spike happened inside a tool call during a chain-of-thought step. For a Director presenting AI operational costs to a CFO, Langfuse dashboards are the artifact.

**Why mock kubectl instead of real kubectl?** Portfolio hygiene. A recruiter running your repo should not need a live cluster. The mock returns realistic JSON so the agent behavior is identical. In production, you swap the mock for a real subprocess call or Kubernetes Python client — one line change.

**Failure modes to know for interviews**:
- **Tool timeout during incident**: the orchestrator must have a timeout + fallback ("runbook not found — escalate to human") — you will implement this
- **Hallucinated runbook steps**: Guardrails alone does not fix this — you need grounding (RAG from KB) which is why the KB tool exists
- **Runaway agent loop**: ReAct agents can loop if tool outputs do not satisfy the stopping condition — max_iterations cap is your circuit breaker, equivalent to a watchdog timer

**Design decision — why structured output over free text?** A Markdown report with defined fields (severity, affected_pods, root_cause_hypothesis, immediate_actions) can be machine-parsed for PagerDuty ticket creation, Jira auto-ticket, or SLO impact calculation. Free text cannot. This is the difference between a demo and a production system.

- **AI augmentation vs. AI replacement framing**: position this to leadership as "on-call engineer gets a pre-structured incident brief in 30 seconds instead of 5 minutes of manual triage" — not "AI replaces on-call." That framing gets budget approved.

## Comprehension Check

Answer 2 of 3 before starting the lab.

**Q1**: In the ReAct agent pattern, what is the difference between a "thought" step and a "action" step, and why does that distinction matter for debugging a stuck agent?

**Q2**: You deploy this agent and it produces incident reports with customer phone numbers visible. Bedrock Guardrails is configured. What are the two most likely reasons Guardrails failed to redact the PII?

**Q3**: A recruiter asks: "Why did you use a Knowledge Base tool instead of just putting all your runbooks in the system prompt?" Give the 60-second answer that demonstrates production-readiness thinking.

---

## ANSWER KEY — Comprehension Check

**A1**: A "thought" step is the model's internal reasoning (what tool should I call, why) — it is not an external action, no tool is invoked, no cost beyond tokens. An "action" step is the actual tool call — it has latency, can fail, and returns external data. When debugging a stuck agent, you check: is it looping on thought steps without reaching an action (model confused, prompt issue), or is it reaching action steps but tools are returning errors (infrastructure issue). These require completely different fixes. Langfuse shows you which is happening.

**A2**: Two most likely causes: (1) Guardrails PII detection was not applied to the correct content block — it may be configured on the input path only, not the output/tool-result path. You must apply it to the final synthesized output before it is written to the report. (2) The phone number format did not match Guardrails' built-in PII regex — e.g., international format `+1-206-555-0100` vs. the pattern Guardrails was trained on. You need to add a custom regex entity for your specific PII patterns.

**A3**: "Two reasons. First, context window cost and reliability — if I have 200 runbooks, putting them all in the system prompt costs ~$0.50 per invocation and pushes me toward the context limit. The Knowledge Base does semantic retrieval and returns only the 2–3 relevant runbooks for this specific alert. Second, and more important: runbooks change. If I hardcode them in the prompt, every runbook update requires a code deploy. With a Knowledge Base, an SRE updates the S3 source document and the KB re-indexes automatically — no code change. That is the production operations model."

            root_cause_hypothesis=tool_input["root_cause_hypothesis"],
            immediate_actions=tool_input["immediate_actions"],
            escalate=tool_input["escalate"]
        )
    
    else:
        result = f"ERROR: Unknown tool '{tool_name}'. Available tools: query_knowledge_base, get_pod_status, format_incident_report"
    
    print(f"  [TOOL RESULT] {result[:200]}...")  # First 200 chars to avoid terminal flood
    return result

def run_sre_agent(alert: dict, guardrail_id: str = None, guardrail_version: str = "DRAFT") -> str:
    """
    Run the SRE triage agent on a given alert.
    
    Args:
        alert: Dict containing alert details (name, namespace, deployment, description)
        guardrail_id: Optional Bedrock Guardrail ID for PII filtering (Module 3)
        guardrail_version: Guardrail version to apply
    
    Returns:
        Final incident report string
    """
    
    # Initialize Bedrock client
    # WHY us-east-1: Bedrock Claude models have highest availability here
    bedrock = boto3.client("bedrock-runtime", region_name=os.getenv("AWS_REGION", "us-east-1"))
    model_id = os.getenv("BEDROCK_MODEL_ID", "anthropic.claude-3-haiku-20240307-v1:0")
    
    # System prompt — establishes the agent's role and constraints
    # WHY this structure: clear role + explicit workflow + output format = 
    # fewer hallucinations and more predictable tool-calling behavior
    system_prompt = """You are an SRE triage agent. Your job is to investigate Kubernetes alerts
and produce structured incident reports.

WORKFLOW — follow this exact sequence:
1. Query the knowledge base FIRST to understand the recommended response for this alert type
2. Get pod status to see the current state of affected pods
3. Analyze both results and form a root cause hypothesis
4. Call format_incident_report with all gathered information

CONSTRAINTS:
- Always call all three tools before responding
- Never guess pod status — always call get_pod_status
- Set escalate=true if: restart count > 10, severity is P1/P2, or OOMKilled
- Root cause hypothesis must reference specific evidence from tool outputs
- Do NOT include any PII (IP addresses, emails, usernames) in the report

When you have called all three tools and received the formatted report, output the report text
and nothing else."""

    # Format the initial user message from the alert payload
    # WHY alert as JSON: gives the model structured data it can extract fields from
    initial_message = f"""New Kubernetes alert received. Triage immediately.

Alert details:
{json.dumps(alert, indent=2)}

Begin triage workflow now."""

    # Initialize conversation — this is the messages array for Converse API
    # WHY this format: Bedrock Converse API requires alternating user/assistant turns
    messages = [{"role": "user", "content": [{"text": initial_message}]}]
    
    # Build the converse call kwargs
    # Guardrails are added optionally — same base call with optional config
    converse_kwargs = {
        "modelId": model_id,
        "system": [{"text": system_prompt}],
        "messages": messages,
        "toolConfig": {"tools": TOOL_DEFINITIONS},
        "inferenceConfig": {
            "maxTokens": 4096,
            "temperature": 0.1  # Low temperature = more deterministic tool calling
        }
    }
    
    # Add guardrails if configured (Module 3 populates this)
    if guardrail_id:
        converse_kwargs["guardrailConfig"] = {
            "guardrailIdentifier": guardrail_id,
            "guardrailVersion": guardrail_version,
            "trace": "enabled"  # Shows what Guardrails blocked/modified in response
        }
        print(f"[GUARDRAIL] Applied guardrail: {guardrail_id} v{guardrail_version}")
    
    print(f"\n[AGENT] Starting triage for alert: {alert.get('alert_name', 'unknown')}")
    print(f"[AGENT] Model: {model_id}")
    print("-" * 60)
    
    # ---------------------------------------------------------------------------
    # ReAct loop — Observe → Think → Act → repeat until done
    # ---------------------------------------------------------------------------
    # This is the heart of the agent. We keep looping as long as the model
    # wants to call tools. When it produces a final text response, we stop.
    # WHY loop: a single Converse call only does one round of tool use.
    # Real agent behavior requires iterating until the model is "done thinking."
    # ---------------------------------------------------------------------------
    
    max_iterations = 10  # Circuit breaker — prevents runaway loops
    iteration = 0
    final_report = None
    
    while iteration < max_iterations:
        iteration += 1
        print(f"\n[ITERATION {iteration}] Calling Bedrock...")
        
        response = bedrock.converse(**converse_kwargs)
        
        stop_reason = response["stopReason"]
        output_message = response["output"]["message"]
        usage = response.get("usage", {})
        
        print(f"[ITERATION {iteration}] Stop reason: {stop_reason} | "
              f"Input tokens: {usage.get('inputTokens', '?')} | "
              f"Output tokens: {usage.get('outputTokens', '?')}")
        
        # Add model's response to conversation history
        # WHY: Converse API is stateless — we maintain the full conversation
        # and send it back each time, just like a chat API
        messages.append(output_message)
        
        if stop_reason == "tool_use":
            # Model wants to call one or more tools
            # Extract all tool use blocks from the response
            tool_results = []
            
            for content_block in output_message["content"]:
                if content_block.get("type") == "tool_use":
                    tool_use_id = content_block["toolUseId"]
                    tool_name = content_block["name"]
                    tool_input = content_block["input"]
                    
                    # Execute the tool
                    tool_result_text = dispatch_tool(tool_name, tool_input)
                    
                    # Build the tool result in Converse API format
                    tool_results.append({
                        "toolUseId": tool_use_id,
                        "content": [{"text": tool_result_text}]
                    })
            
            # Add all tool results as a user message
            # WHY user role: Bedrock Converse requires tool results in user turn
            messages.append({
                "role": "user",
                "content": [
                    {"toolResult": tr} for tr in tool_results
                ]
            })
            
            # Update messages in kwargs and loop again
            converse_kwargs["messages"] = messages
        
        elif stop_reason == "end_turn":
            # Model produced a final text response — extract it
            for content_block in output_message["content"]:
                if content_block.get("type") == "text" or "text" in content_block:
                    text = content_block.get("text", "")
                    if text:
                        final_report = text
                        break
            
            print(f"\n[AGENT] Triage complete after {iteration} iteration(s)")
            break
        
        elif stop_reason == "guardrail_intervened":
            # Guardrails blocked the output — this is expected in adversarial tests
            print("[GUARDRAIL] Output was blocked or modified by Guardrails")
            # The response still contains a message — extract whatever passed through
            for content_block in output_message.get("content", []):
                if "text" in content_block:
                    final_report = content_block["text"]
                    break
            if not final_report:
                final_report = "[BLOCKED BY GUARDRAILS] Output contained policy violations."
            break
        
        else:
            print(f"[AGENT] Unexpected stop reason: {stop_reason}")
            break
    
    if not final_report:
        final_report = f"[ERROR] Agent did not produce a final report after {max_iterations} iterations."
    
    return final_report

## Key Design Decisions

**Why Bedrock over OpenAI API?**  
Data sovereignty. Incident reports contain operational details that must stay within AWS VPC boundary. This is a non-negotiable requirement in regulated industries (telecom, banking, health). Bedrock keeps all inference within your AWS account — no data leaves to third-party endpoints.

**Why Langfuse for observability?**  
Langfuse provides LLM-specific telemetry that CloudWatch cannot: token counts per span, cost per trace, prompt version tracking, and latency breakdown across tool calls. For a Director presenting AI operational costs to a CFO, Langfuse dashboards are the artifact. A single trace shows: 4 Bedrock calls, 3 tool executions, ~7,200 total tokens, ~$0.000034 per alert triage.

**Why mock kubectl instead of live cluster?**  
Portfolio reproducibility. Any engineer cloning this repo can run it without cluster access. The mock returns the same JSON structure as real kubectl output. Swapping in real kubectl is a one-function change in `tools.py`.

**Why structured output (Markdown with defined fields) instead of free text?**  
Machine parseability. A report with fields `severity`, `affected_pods`, `immediate_actions`, `escalate` can be ingested by PagerDuty, Jira, or a Slack bot without NLP. Free text cannot. This is the difference between a demo and a production reliability platform.

## Lab ANSWER KEY — Diagnostics per Step

**MODULE 1 failures**:
- `ModelNotReadyException` on first Bedrock call → Go to AWS Console → Bedrock → Model Access → Enable Claude 3 Haiku. Takes 2-5 minutes.
- `HereditaryTokenError` or malformed JSON from Bedrock → Temperature is set to 0.1. If tool calls come back malformed, set to 0.0 and retry.
- Agent loops and never hits `end_turn` → The model is not calling `format_incident_report` last. Add to system prompt: "You MUST call format_incident_report as your FINAL tool call."
- `AttributeError: 'dict' has no attribute 'get'` on content_block → The Converse API changed content block format in newer boto3 versions. Check `content_block` structure with `print(type(content_block), content_block.keys())`.

**MODULE 2 failures**:
- Langfuse `AuthenticationError` → Keys in .env are wrong. Go to Langfuse dashboard → Settings → API Keys → regenerate and paste exactly (no trailing spaces).
- Traces appear in dashboard but spans are missing → Missing `langfuse.flush()` at end. The script exits before async batch sends. Flush is already in `agent_traced.py` — verify it did not get cut off.
- Trace shows but all spans are flat (no nesting) → `parent_span` was not passed to `dispatch_tool_traced`. Verify the function signature matches the call.

**MODULE 3 failures**:
- `ValidationException` on `create-guardrail` → Guardrail policy JSON is malformed. Copy the exact JSON from Step 3.1 — do not modify field names.
- PII test passes but email still appears in output → Guardrail applied to input only. Verify `guardrailConfig` is in `converse_kwargs` before the final `bedrock.converse()` call. The guardrail must be on the OUTPUT path.
- Prompt injection test does not get blocked → The topic definition needs more examples. Add more adversarial phrases to the `examples` array in the topic policy and re-create the guardrail.

## Tradeoffs and Failure Modes

**Tradeoff 1 — Agent accuracy vs. latency**
More iterations = better reasoning but higher latency and cost. Claude 3 Haiku completes in ~4 iterations and ~8 seconds. Claude 3.5 Sonnet does it in 3 iterations with better reasoning but costs 5x more and runs slower. For P2/P3 alerts: Haiku. For P1 incidents requiring complex diagnosis: escalate to Sonnet. Design pattern: severity-based model routing.

**Tradeoff 2 — Mock tools vs. real integration**
Mock tools guarantee portfolio reproducibility and fast iteration during development. Real tool integration (actual KB, real kubectl) adds operational dependencies — if the K8s API is unreachable during an incident, your triage agent fails at the worst possible moment. Mitigation: circuit breaker per tool, cached fallback responses, graceful degradation to "human triage required."

**Tradeoff 3 — Guardrails completeness vs. false positives**
Aggressive PII matching blocks legitimate technical content. A pod named `john-smith-service` could trigger name-entity detection and get anonymized in the report. You tune this by testing with your actual alert corpus before production rollout. The principle: start with high sensitivity, reduce based on observed false positive rate. Never start with low sensitivity and increase — you will ship PII before you know it is a problem.
