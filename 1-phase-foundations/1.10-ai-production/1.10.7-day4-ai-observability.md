# AI Observability — Tracing, Cost, Evaluation | Week 10 Day 4

## Why This Matters (Interview Relevance + Business Impact)

**Interview relevance**: At Databricks, Snowflake, AWS, and any company running AI in production, the Staff/Principal SRE question has shifted from "how do you monitor your Kubernetes pods?" to "how do you know your AI pipeline is working correctly?" Interviewers probe: token cost attribution, latency SLOs for inference, hallucination detection, and A/B prompt testing governance. You need the same rigor you apply to your 25M msg/day notification platform — applied to LLM call chains.

**Business impact**:
- A production RAG pipeline with no tracing is a black box. When answers degrade (hallucinations spike, latency doubles), you have no MTTD path — you find out from a ticket, not a dashboard.
- Token cost without attribution is a budget nightmare. At $0.003/1K tokens (GPT-4o), an unoptimized prompt hitting 10K tokens 10,000 times/day = $300/day vs an optimized 3K-token prompt = $90/day. You cannot optimize what you cannot see.
- RAGAS scores operationalized as SLOs give you an error budget for AI quality — the same construct you use for service reliability.
- Director level: LLM observability is the governance layer that makes AI deployable in regulated industries (telecom, banking, insurance). Without it, legal and compliance will block AI in production.

**Why now**: LLMOps is the fastest-moving SRE subdomain in 2025-2026. Every Director/VP SRE job description at AI-forward companies now lists "LLM observability" or "AI platform reliability" as a requirement.

## Analogy — MELT for Your AI Layer

You already know MELT (Metrics, Events, Logs, Traces) from Week 5. You run this on your notification platform — RabbitMQ queue depth is a metric, message delivery is an event, error logs from consumers are logs, and a distributed trace connects producer → broker → consumer → database in one timeline.

AI observability is the exact same stack — applied one layer up:

```
Traditional Service (MELT):           AI Pipeline (AI Observability):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Metric: HTTP latency p95              Metric: LLM inference latency p95
Metric: Request error rate            Metric: Hallucination rate (RAGAS)
Event: Deployment pushed              Event: Prompt version changed (A/B)
Log: Exception stack trace            Log: Full prompt + completion text
Trace: API → Service → DB             Trace: User Q → Retrieval → LLM → Answer

Alert: p95 > 500ms                    Alert: faithfulness score < 0.7
SLO: 99.9% success rate               SLO: 95% answers cite correct source
Error Budget: 43.8 min/month          Error Budget: 5% hallucination budget
```

**Langfuse = AppDynamics for your LLM calls.**

When you ran AppDynamics on your notification platform, it gave you:
- A transaction trace showing every hop a message took
- Time spent in each service
- SQL query breakdown
- Error rates by service tier

Langfuse does the same thing for LLM pipelines:
- A trace showing every LLM call, retrieval step, and tool use
- Tokens consumed + cost at each step
- Latency breakdown (retrieval vs generation vs total)
- Quality scores per response

The mental model shift: your "service" is now a language model call. Your "database query" is now a vector similarity search. The observability discipline is unchanged — only the data source is new.

## 30-Second Version

AI observability = instrument every LLM call to capture: input prompt, output completion, token count (cost), latency, and quality score. Tools: Langfuse (open-source, best for RAG), LangSmith (LangChain-native), Arize (enterprise ML+LLM). Quality measurement: RAGAS framework — four scores (faithfulness, answer relevancy, context precision, context recall) that tell you if your RAG pipeline is actually working. Hallucination detection: LLM-as-judge pattern — a second LLM evaluates whether the first LLM's answer is grounded in the retrieved context. A/B prompt testing: route N% of traffic to Prompt Version B, compare RAGAS scores and cost/query.

## 2-Minute Version

**The three instrumentation layers:**

**Layer 1 — Trace/Span (what happened)**: Every LLM call becomes a span in a distributed trace. The span captures: model name, prompt text, completion text, start time, duration, token counts. A RAG pipeline trace has child spans: retrieval span (which chunks were found, similarity scores) + generation span (LLM call itself). You can see the full chain in one timeline — identical to AppDynamics Business Transaction view.

**Layer 2 — Cost attribution (what it cost)**: Token pricing is public (e.g., GPT-4o: $2.50/1M input tokens, $10.00/1M output tokens as of 2025). Langfuse calculates cost per generation automatically when you configure the model price. Roll up: cost per trace → cost per user session → cost per feature → cost per day. This is the showback/chargeback model for AI — the same construct you use for cloud cost attribution.

**Layer 3 — Quality scores (was it right)**: RAGAS evaluates your RAG pipeline on four dimensions. Faithfulness (did the answer stay within the retrieved context?), Answer Relevancy (did it answer the actual question?), Context Precision (was retrieval precise?), Context Recall (was retrieval complete?). These become your SLIs. Set SLOs: faithfulness > 0.8, answer relevancy > 0.75. Alert when scores decay below threshold — same as alerting on error rate spike.

**Tool selection**:
- Langfuse: Best default for RAG/LangChain/open-source. Free cloud tier, self-hostable for data-sensitive workloads. Best dashboard.
- LangSmith: Natural if you are already on LangChain, but vendor lock-in.
- Arize Phoenix: Better for organizations running both traditional ML models and LLMs — unified platform.

**A/B prompt testing**: Change one variable (prompt wording, temperature, model), route a percentage of traffic to the variant, compare RAGAS scores and cost. Works exactly like a feature flag canary — the feature being tested is a prompt string.

### OTel Spans for LLM Calls

OpenTelemetry is the open standard (you know this from Week 5). The OpenTelemetry Semantic Conventions project (GenAI SIG) defines standard attribute names for LLM spans:

```
Span Name: gen_ai.chat
Attributes:
  gen_ai.system = "openai"              # or "anthropic", "bedrock"
  gen_ai.request.model = "gpt-4o"
  gen_ai.request.max_tokens = 1024
  gen_ai.usage.input_tokens = 847
  gen_ai.usage.output_tokens = 203
  gen_ai.response.finish_reason = "stop"
  gen_ai.operation.name = "chat"
  server.address = "api.openai.com"
  http.response.status_code = 200
```

These are the same OTLP spans you export to your OTel Collector. Add a semantic conventions-aware exporter (Langfuse, Arize, or your Tempo/Jaeger instance) and LLM calls become first-class distributed traces in your existing observability stack — no separate tooling required at the infrastructure level.

### Token Cost Per Trace: The Math

```
Example RAG query trace:
  Retrieval: no LLM call (vector DB query only)
  Generation:
    System prompt:    500 tokens input  ($0.00125)
    Context chunks:  1,200 tokens input  ($0.00300)
    User question:    50 tokens input  ($0.000125)
    LLM answer:      200 tokens output  ($0.002000)
    ─────────────────────────────────────────────
    Total per query: $0.005625  (~half a cent)
    At 1,000 queries/day: $5.63/day → $169/month
    
    Optimization opportunity: reduce context chunks from 1,200 to 600
    (tighter retrieval precision) → cuts cost by $1.50/month per 1K daily queries
```

This is your capacity planning model. When the product team says "expand to 10,000 queries/day," you already know the cost trajectory before the first line of code is written.

### RAGAS Metrics: The Math Behind Each Score

**Faithfulness** (most critical for hallucination detection):
```
Faithfulness = (statements in answer supported by context) / (total statements in answer)

Example answer: "The deployment failed because the readiness probe timeout was 1s.
                 You should increase it to 10s per the runbook."
Statements: 2
Supported by retrieved context: 2
Faithfulness = 2/2 = 1.0  (perfect)

Hallucinated answer: "The deployment failed because of a kernel bug in Linux 5.15."
Supported by retrieved context: 0
Faithfulness = 0/2 = 0.0  (complete hallucination)
```

**Answer Relevancy**: Reverse-engineer the original question from the answer. If the LLM can reconstruct a question similar to the original, the answer was on-topic. Uses embedding similarity.

**Context Precision**: Of the N chunks retrieved, how many were actually useful? High retrieval but low precision = your vector search is surfacing noise. Tune `k` (number of chunks) or similarity threshold.

**Context Recall**: Were the chunks that contained the real answer actually retrieved? Low recall = your embeddings are missing semantic coverage. Solution: improve chunking strategy or embedding model.

### LLM-as-Judge Pattern

```
Step 1: User asks question Q, RAG answers with A, retrieved context C
Step 2: Send to judge LLM:
  "Given context: {C}
   Question: {Q}
   Answer: {A}
   
   Score faithfulness 0-1: Does the answer contain ONLY information 
   from the context? Return JSON: {faithfulness: 0.X, reasoning: '...'}"
Step 3: Store judge score in Langfuse as evaluation metric
Step 4: Alert if rolling 100-query average faithfulness < 0.75
```

The judge LLM can be a cheaper/faster model (GPT-4o-mini at $0.15/1M tokens) evaluating a more expensive model's outputs (GPT-4o at $2.50/1M tokens). The evaluation overhead is ~10-20% of your total LLM cost but gives you continuous quality monitoring.

**Failure mode**: Judge LLM can also hallucinate — especially when context is ambiguous. Mitigate by using high-temperature=0 for judge calls and cross-checking with rule-based heuristics (e.g., answer mentions facts not in context string).

### Hallucination Detection Patterns

| Pattern | Mechanism | Sensitivity | Cost |
|---|---|---|---|
| LLM-as-Judge | Second LLM scores faithfulness | High | Medium (~10% overhead) |
| Source citation check | Does answer cite retrieved chunk IDs? | Medium | Low (string match) |
| Grounding score | Embedding similarity: answer vs context | Medium | Low (no LLM call) |
| Self-consistency | Ask same Q 3x, different seeds; compare answers | High | High (3x inference cost) |
| Knowledge boundary probe | Ask about facts NOT in context; answer should say "I don't know" | Targeted | Low |

For production RAG at T-Mobile scale: run LLM-as-judge asynchronously (out of the critical path), grounding score synchronously as a fast pre-filter. Alert on 5-minute rolling window faithfulness score drop.

### A/B Prompt Testing Architecture

```
User Request
     │
     ▼
Feature Flag Service  ─── 90% ──► Prompt Version A (production)
     │                             │
     └─────── 10% ──────────────► Prompt Version B (candidate)
                                   │
                        Both paths ▼
                    ┌────────────────────────────┐
                    │ Langfuse Experiment Tracker│
                    │  - RAGAS scores per variant│
                    │  - Cost per query          │
                    │  - Latency p50/p95         │
                    └────────────────────────────┘
```

Promotion criteria (same logic as canary promotion):
- Statistical significance: minimum 200 queries per variant
- Faithfulness B >= Faithfulness A (no quality regression)
- Cost/query B <= Cost/query A * 1.1 (no more than 10% cost increase)
- p95 latency B <= p95 latency A * 1.2 (no more than 20% latency regression)

### p50/p95 Latency Targets for LLM Inference

| Model | Typical p50 | Typical p95 | What drives it |
|---|---|---|---|
| GPT-4o-mini | 800ms | 2.5s | Cheap, fast, OpenAI shared infra |
| GPT-4o | 1.5s | 5s | Higher quality, heavier model |
| Claude Sonnet 3.5 | 1.2s | 4s | Anthropic infra load-dependent |
| Bedrock Claude | 1.5s | 6s | VPC endpoint + model routing overhead |
| Local Ollama (llama3) | 2-10s | 15-30s | CPU/GPU of your machine |

RAG pipeline total p95 budget: allocate across three phases:
- Vector retrieval: < 50ms (pgvector or Pinecone)
- LLM generation: model p95 (above)
- Total pipeline SLO: model p95 + 100ms overhead (retrieval + serialization)

## Director/VP Lens

**As a Director/VP, you govern AI observability — you do not instrument it.**

**What you establish:**
- LLM observability as a non-negotiable gate in the AI deployment checklist — same as "no deploy without runbook" for services
- Token cost attribution tied to teams/features in cost reporting — prevents AI cost sprawl (the equivalent of untagged S3 buckets)
- Quality SLOs with error budgets for AI pipelines — faithfulness < 0.75 triggers the same escalation as error rate SLO breach
- Data governance policy: where can prompt/completion data live? (Regulated industries: must be self-hosted, not SaaS)
- A/B prompt testing process: same PR/review flow as code — prompt changes are deployments, not ad-hoc edits

**What your ICs implement:**
- Langfuse instrumentation in LangChain or direct API calls
- RAGAS evaluation harness in CI pipeline (run on golden question set per deploy)
- Cost dashboards in Grafana or Langfuse cloud
- LLM-as-judge async evaluation pipeline

**The governance question in interviews:** "How do you prevent AI quality from silently degrading in production?"

Answer: Operationalize RAGAS scores as SLIs. Define error budget for hallucination rate. Build async LLM-as-judge evaluation into the response pipeline. Dashboard faithfulness/cost/latency as first-class platform metrics alongside your existing MELT dashboard. Treat a prompt change as a deployment — route through canary, measure for 24 hours, promote or rollback.

**AI changes this domain:** AI observability is itself becoming AI-augmented. Tools like Arize and Langfuse are adding anomaly detection on RAGAS scores (flagging when a prompt version causes a quality distribution shift) — the same pattern as your Splunk MLTK anomaly detection, applied to LLM quality metrics.

## ANSWER KEY — Comprehension Check

**A1.** Most likely cause: the updated system prompt added instructions that encouraged the LLM to elaborate beyond the retrieved context (e.g., "Be thorough and comprehensive" without "only use the provided context"). The LLM started adding general knowledge not grounded in retrieved chunks — faithfulness measures exactly this.

Immediate diagnostic step: Pull the Langfuse trace for a low-faithfulness generation. Side-by-side compare the retrieved context chunks and the answer. Find the specific sentences in the answer that have no corresponding source in context. That tells you whether it's a prompt instruction problem (LLM ignoring context boundary) or a retrieval problem (right context was not retrieved). Then diff the prompt versions in Langfuse experiment view to isolate the change.

**A2.** ROI framing:
- Incident MTTR reduction: If AI runbook assistant saves 15 minutes per P1 (you query it 3 times per incident, 5 minutes saved each) and you run 20 P1s/month: 20 × 15min × (Sev1 engineer hourly rate $150/hr) = $750/month in eng cost saved.
- $750 saved vs $300 cost = 2.5x ROI before factoring in customer impact reduction.
- Observability data needed: average queries per incident (from Langfuse traces), query success rate (faithfulness > 0.8 = "answered correctly"), time-to-answer distribution (p95 latency per trace), cost-per-incident (roll up trace costs per session tagged with incident ID).
- Frame it as: "We instrument the AI the same way we instrument our services — cost per transaction, quality SLO, latency SLO. The business case is verifiable from the data."

**A3.** Promotion criteria for GPT-4o-mini canary:
1. Minimum sample: 200 queries per variant (statistical significance for proportion tests at 95% confidence, ~5% relative effect size)
2. Quality gate: faithfulness of mini variant >= faithfulness of GPT-4o variant * 0.95 (allow up to 5% quality degradation — validate with your team if this is acceptable for the specific use case)
3. Answer relevancy: same threshold — mini variant within 5% of GPT-4o
4. Latency: p95 latency of mini variant <= 4s (mini is typically faster so this should pass easily)
5. Run for minimum 24 hours to catch time-of-day traffic patterns
6. Automatic rollback trigger: if faithfulness drops below absolute 0.70 at any point during canary

### Step 3 — Build the RAG Pipeline with Langfuse Instrumentation

```bash
cat > rag_pipeline.py << 'PYEOF'
"""
SRE Runbook RAG Pipeline — with Langfuse Observability
Week 10 Day 4 Lab

What this does:
  1. Builds a local FAISS vector store from sample SRE runbook text
  2. Instruments every LLM call and retrieval step with Langfuse traces

# Load environment variables from .env file
load_dotenv()

from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import FAISS
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_core.documents import Document
from langchain.chains import RetrievalQA
from langfuse.callback import CallbackHandler

# ── Langfuse callback handler ────────────────────────────────────────────────
# This single handler intercepts all LangChain calls and sends traces to
# Langfuse. It captures: prompt text, completion text, token counts, latency.
# Think of it like adding an AppDynamics agent — zero code change to business
# logic, full instrumentation via callback injection.

    public_key=os.environ["LANGFUSE_PUBLIC_KEY"],
    secret_key=os.environ["LANGFUSE_SECRET_KEY"],
    host=os.environ["LANGFUSE_HOST"],

    user_id="vishweshwar-lab",         # For per-user cost attribution

# ── Sample SRE runbook data ──────────────────────────────────────────────────
# In Week 9 this was your S3 bucket of real runbooks.
# For lab simplicity, we embed the content inline.
RUNBOOK_TEXT = """
# Kubernetes Pod CrashLoopBackOff Runbook

## Symptoms
Pod status shows CrashLoopBackOff in kubectl get pods output.
The pod restarts repeatedly with increasing backoff intervals.

## Diagnostic Steps
1. Run: kubectl logs <pod-name> --previous
   This shows logs from the previous (crashed) container instance.
2. Run: kubectl describe pod <pod-name>
   Check the Events section for OOMKilled or Error exit codes.
3. If exit code is 137, the container was killed by OOM (Out of Memory).
   Solution: increase memory limits in the deployment spec.
4. If exit code is 1, check application logs for startup errors.
   Common cause: missing environment variables or misconfigured ConfigMap.
5. Check readiness probe configuration. A timeout too short (< 5s) causes
   the pod to be killed before the application finishes initialization.
   Solution: set initialDelaySeconds to 30 and timeoutSeconds to 10.

## Resolution
For OOM: kubectl edit deployment <name> and increase resources.limits.memory
For config errors: kubectl describe configmap <name> to verify all keys exist
For probe timeout: increase initialDelaySeconds in the deployment spec

# High RabbitMQ Queue Depth Runbook

## Symptoms
Queue depth exceeds 10,000 messages. Consumer lag growing. Message age
in queue exceeds 30 seconds. Alerts firing from Splunk MART dashboard.

## Diagnostic Steps
1. Check consumer count: rabbitmqctl list_queues name messages consumers
   If consumers = 0, the consumer service is down. Check pod status.
2. Check consumer health: kubectl get pods -l app=notification-consumer
3. If consumers are running but not processing: check for consumer-side
   connection errors. Common cause: credentials rotated but consumer
   pods not restarted to pick up new secrets from Vault.
4. Check dead-letter queue (DLQ) depth. If DLQ is growing, messages are
   being rejected. Check consumer logs for deserialization errors.
5. Temporary relief: scale consumer deployment:
   kubectl scale deployment notification-consumer --replicas=10

## Resolution
Consumer down: kubectl rollout restart deployment/notification-consumer
Credentials issue: restart pods after Vault secret rotation is confirmed
DLQ growth: inspect sample DLQ messages for schema mismatch errors

# SLO Breach — Error Budget Exhaustion Runbook

## Symptoms
Error budget burn rate alert fires. 30-day rolling error budget below 20%.
Availability SLO at risk of breach within 72 hours at current burn rate.

## Immediate Actions
1. Declare error budget freeze: no non-critical deployments until budget recovers.
   Notify engineering leads via #platform-sre channel.
2. Identify burn rate source: check Grafana SLO dashboard, filter by service.
3. If single service driving burn: escalate to owning team. SRE assists with
   diagnostic but owning team is responsible for fix.
4. If platform-wide: initiate Sev2 incident process.

## Error Budget Math
Daily burn rate = (error events in 24h) / (total events in 24h) = current error rate
Acceptable daily burn = (1 - SLO target) / 30 days
If current burn > 3x acceptable: Sev2 escalation
If current burn > 10x acceptable: Sev1 escalation (imminent SLO breach)
"""

# ── Build vector store ───────────────────────────────────────────────────────
print("Building FAISS vector store from runbook text...")

# Split the runbook into chunks
# Why RecursiveCharacterTextSplitter: it tries to split on paragraph
# boundaries first, then sentence boundaries — preserves semantic units.
# chunk_size=500: ~375 words per chunk. chunk_overlap=50: prevents context
# from being cut off at chunk boundary (same as having a sliding window
# in your Splunk search).
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
docs = splitter.create_documents([RUNBOOK_TEXT])
print(f"Created {len(docs)} document chunks")

# Create embeddings and vector store
```

```bash
cat > hallucination_detector.py << 'PYEOF'
"""
LLM-as-Judge Hallucination Detector
Week 10 Day 4 Lab — Step 8

What this does:
  Uses a second LLM call (the "judge") to score whether each answer
  from the RAG pipeline is grounded in the retrieved context.
  This is asynchronous quality monitoring — runs after the user
  gets their answer, does not add to response latency.

In production: run this as a background job, score rolling batches
of 100 traces every 5 minutes, alert if average drops below 0.75.
"""

import json
import os
from dotenv import load_dotenv

load_dotenv()

from openai import OpenAI
from langfuse import Langfuse

client = OpenAI()
langfuse = Langfuse(
    public_key=os.environ["LANGFUSE_PUBLIC_KEY"],
    secret_key=os.environ["LANGFUSE_SECRET_KEY"],
    host=os.environ["LANGFUSE_HOST"],
)

# Load our 5 results
with open("rag_results.json") as f:
    results = json.load(f)

JUDGE_PROMPT = """You are an AI quality evaluator for an SRE runbook assistant.
Your job is to determine whether an answer is grounded in the provided context.

CONTEXT (retrieved runbook chunks):
{context}

QUESTION:
{question}

ANSWER:
{answer}

Evaluate: Does the answer contain ONLY information that can be found in or
reasonably inferred from the context above? Does it avoid adding facts not
present in the context?

Respond with JSON only:
{{
  "faithfulness_score": <float 0.0-1.0>,
  "grounded_statements": <int — count of answer statements supported by context>,
  "total_statements": <int — count of all factual statements in answer>,
  "verdict": "<GROUNDED|PARTIALLY_GROUNDED|HALLUCINATED>",
  "reasoning": "<one sentence explanation>"
}}"""

print("Running LLM-as-Judge evaluation on 5 RAG responses...\n")
print(f"{'Q#':<4} {'Faithfulness':>12} {'Verdict':<22} Reasoning")
print("-"*75)

scores = []
for i, result in enumerate(results, 1):
    context = "\n---\n".join(result["contexts"])
    
    # Judge LLM call — temperature=0 for deterministic, reproducible scores
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        temperature=0,
        messages=[{
            "role": "user",
            "content": JUDGE_PROMPT.format(
                context=context[:2000],   # Truncate to fit context window
                question=result["question"],
                answer=result["answer"]
            )
        }]
    )
    
    # Parse judge response
    import ast
    judge_output = json.loads(response.choices[0].message.content)

    print(f"Q{i:<3} {score:>12.3f} {judge_output['verdict']:<22} "
          f"{judge_output['reasoning'][:40]}...")
    
    # Log judge score back to Langfuse as a named score
    # In production, look up the actual trace_id from Langfuse traces API
    # For lab, we log as a standalone score event
    langfuse.score(
        name="llm_judge_faithfulness",
        value=score,
        comment=judge_output["reasoning"],
        data_type="NUMERIC",
    )

avg_score = sum(scores) / len(scores)
print("-"*75)
print(f"Average LLM-judge faithfulness: {avg_score:.3f}")
print(f"SLO status (threshold 0.75):    {'PASS' if avg_score >= 0.75 else 'FAIL — ALERT'}")

langfuse.flush()
print("\nJudge scores logged to Langfuse.")
```

## Lab ANSWER KEY — Diagnostics Per Step

**Step 3 fails: `ImportError: cannot import name 'CallbackHandler' from 'langfuse'`**
Cause: Langfuse version < 2.0 or wrong package installed.
Fix: `pip install --upgrade langfuse`. Verify with `pip show langfuse | grep Version` — needs 2.x.

**Step 4 fails: `AuthenticationError: Incorrect API key`**
Cause: OpenAI API key not set or wrong value in `.env`.
Fix: `echo $OPENAI_API_KEY` to verify it loaded. Recheck `.env` file for typos. Ensure `load_dotenv()` is called before any OpenAI client initialization.

**Step 4 fails: Traces not appearing in Langfuse dashboard after script completes**
Cause: Missing `langfuse_handler.flush()` at end of script. Langfuse batches traces and sends them asynchronously — without flush(), the script exits before the batch is sent.
Fix: Ensure `langfuse_handler.flush()` is the last call before script exit.

**Step 6 fails: `ValueError: Dataset must have column 'ground_truth'`**
Cause: RAGAS version mismatch. RAGAS 0.2.x uses `reference` instead of `ground_truth`.
Fix: Check `pip show ragas | grep Version`. If 0.2.x: rename `ground_truth` key to `reference` in the dataset dict. If 0.1.x: `ground_truth` is correct.

**Step 6: faithfulness score = 0.0 for all questions**
Cause: RAGAS evaluation LLM is returning malformed JSON that RAGAS cannot parse.
Fix: Set `RAGAS_TIMEOUT=120` environment variable. Try `temperature=0.1` instead of `0` for the evaluator LLM. Check OpenAI API rate limits — RAGAS makes many parallel calls.

**Step 8: `json.JSONDecodeError` parsing judge output**
Cause: Judge LLM returned markdown code fences around the JSON (```json ... ```).
Fix: Strip markdown fences before parsing:
```python
raw = response.choices[0].message.content
raw = raw.replace("```json", "").replace("```", "").strip()
judge_output = json.loads(raw)
```

## Tradeoffs and Failure Modes

**1. LLM-as-judge circular dependency**
Tradeoff: The judge LLM evaluating your production LLM can have the same failure modes as the production LLM — hallucination, bias, cost. You are using AI to evaluate AI.
Mitigation: Use a different model family for judging than production (e.g., OpenAI as judge for Anthropic production model). Run rule-based heuristics (citation checking, length anomaly detection) as fast pre-filters before the judge LLM. For high-stakes evaluation, reserve human review for the bottom 5% of scores.
Interview framing: "LLM-as-judge is a scalable quality signal, not a ground truth. We treat it the same way we treat synthetic monitoring — useful signal, confirmed by real user feedback on a sample."

**2. RAGAS score gaming vs real quality**
Tradeoff: Optimizing a RAG pipeline for RAGAS scores can improve metrics while degrading practical usefulness. A pipeline that only retrieves and parrots back context chunks will get perfect faithfulness but may give unusable, overly literal answers.
Mitigation: RAGAS is necessary but not sufficient. Complement with: human evaluation on a quarterly golden set, production user acceptance signals (thumbs up/down), and qualitative review of the lowest-scored traces by an SRE domain expert.
Interview framing: "RAGAS gives us automated quality gates. We treat it like code coverage — 80% is the gate, but it doesn't guarantee good software. Quarterly human review keeps the automation honest."

**3. Observability data as a compliance liability**
Tradeoff: Full prompt/completion logging in Langfuse captures user queries and LLM responses in full text. In telecom and banking, these logs may contain PII, account numbers, or regulated data — creating a new compliance surface.
Mitigation: For regulated industries, self-host Langfuse on EKS within the production VPC. Implement input/output scrubbing (regex or ML-based PII detection) before the trace is sent to Langfuse. Define data retention policy — 30-day rolling deletion of raw prompt/completion text, aggregate metrics retained indefinitely.
Interview framing: "Observability and data governance are in tension for AI systems. The solution is self-hosted observability infrastructure with the same data classification controls applied to AI traces as we apply to application logs. At T-Mobile this means: all AI traces stay in our AWS account, PII scrubbed at the exporter level, retained for 30 days matching our log retention policy."
