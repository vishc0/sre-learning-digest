## Analogy

Think of Terraform modules like **TIBCO BusinessWorks shared process definitions** — reusable, parameterized, version-controlled units that teams consume without rewriting from scratch. The remote state backend (S3 + DynamoDB) is your **config store + distributed lock manager**: it ensures two engineers cannot simultaneously deploy conflicting changes, the same way your RabbitMQ broker serializes competing consumers on a single queue. Workspaces are like **PCF org/space separation** — same app definition, different environment targets, isolated namespaces, no config bleed.

## 2-Minute Version

The project structure mirrors how real platform engineering teams organize Terraform at scale. Modules encapsulate resource definitions with documented inputs and outputs — consumers wire them together in `main.tf` without duplicating code. Remote state stored in S3 means any team member (or CI pipeline) picks up the current state instead of running from a stale local file. DynamoDB provides optimistic locking: before writing new state, Terraform acquires a lock record; if another process holds it, the second run fails fast rather than corrupting state.

Workspaces give you environment isolation in a single codebase. The `terraform workspace select dev` command switches the active state file — Terraform reads a different state, applies the same code, and produces independently-tracked infrastructure. Variable files (`terraform.tfvars`, `prod.tfvars`) inject environment-specific values so the same module produces a 1-day retention SQS queue in dev and a 14-day retention queue in prod.

## 5-Minute Version

**Tradeoff 1 — Workspaces vs. directory-per-environment**: Workspaces share the same backend bucket with path-segregated state files. Directory-per-environment (a la Terragrunt) gives harder blast-radius isolation but increases boilerplate. At notification-platform scale, the workspace approach works for a single team; at 50-engineer org scale, directories + Terragrunt or Terraform Cloud become necessary to enforce who can `apply` which environment.

**Tradeoff 2 — Module granularity**: Fine-grained modules (one resource per module) maximize reuse but create deep dependency trees that slow `plan/apply` cycles. Coarse modules (one module = one logical service boundary, e.g., "notification-queue-stack") are faster to apply but harder to reuse across unrelated services. The sweet spot is domain-bounded modules with 3-8 resources each.

**Tradeoff 3 — Drift management**: Terraform detects drift but does not auto-heal — it requires human intent (`terraform apply`). This is correct behavior for production but means your monitoring must surface drift continuously, not just at apply time. Tools like Atlantis (GitOps for Terraform), Driftctl, or HashiCorp Sentinel close this gap at scale.

**Failure modes**:
- State lock not released after a failed apply (common when CI runner dies mid-run) — `terraform force-unlock <lock-id>` recovers, but requires manual intervention and audit
- Partial apply leaves resources in inconsistent state — always use `-target` sparingly; prefer full module applies
- Workspace confusion: engineer applies to `prod` thinking they are in `dev` — mitigate with workspace name in prompt, CI enforcement, and branch-to-workspace mapping policies

## Director/VP Lens

**IC implements**: Writes modules, manages variable files, runs plan/apply, handles lock conflicts, reviews plans before approving PRs.

**Director/VP governs**:
- Mandates remote state for all teams — no local state files in CI
- Sets module library standards: every module must have `variables.tf` with descriptions, `outputs.tf` with all resource IDs exported, README with usage example
- Enforces Sentinel/OPA policy: no S3 bucket without encryption, no SQS queue without DLQ, no IAM role with wildcard `*` actions
- Owns the "who can apply to prod" decision: human approval gate in CI, break-glass runbook for emergencies
- Tracks drift SLA: any unreconciled drift in prod older than 48 hours is an incident
- Frames IaC investment to executives as: "we eliminated 3 hours of manual setup per deployment, reduced configuration audit time from 2 days to 15 minutes, and can now prove to SOC 2 auditors exactly what changed and who approved it"

## Comprehension Check

Answer 2 of 3 to unlock the lab:

1. **Why does Terraform need a DynamoDB table when state is already stored in S3?**

2. **If you run `terraform workspace select prod` and then `terraform apply`, what happens to the dev environment resources you applied earlier?**

3. **A colleague runs `terraform apply` directly in the AWS console for an SQS queue that Terraform manages. Next time your CI pipeline runs `terraform plan`, what will Terraform report, and what is the safest recovery action?**

---

## ANSWER KEY — Comprehension Check

**Q1**: S3 stores state but does not provide atomic locking — two simultaneous `terraform apply` runs could both read state, compute different plans, and write conflicting state files, corrupting infrastructure. DynamoDB provides a conditional write (lock record with a unique ID) that serializes access: only the process holding the lock can write state. This is the same problem RabbitMQ's exclusive consumer flag solves — prevent two workers from processing the same message simultaneously.

**Q2**: Nothing happens to dev resources. Each workspace maintains its own state file at a separate path in the S3 bucket (e.g., `env:/prod/terraform.tfstate` vs `env:/dev/terraform.tfstate`). Applying to prod reads and writes only the prod state file — dev state is untouched. The underlying AWS resources are independent.

**Q3**: Terraform will report a **diff/drift** — the actual SQS queue attributes in AWS do not match the declared state. The safest recovery is: (a) review the `terraform plan` output to understand exactly what changed, (b) if the manual change was wrong, run `terraform apply` to reconcile back to declared state, (c) if the manual change was intentional, update the `.tf` files to match, commit, and then `terraform apply` — never run `terraform apply` to reconcile without first understanding what will change.

#### Step 2: Write the remote state bootstrap (backend.tf)

First, you need to create the S3 bucket and DynamoDB table that will hold your Terraform state. This is a chicken-and-egg problem — you cannot use a Terraform backend before it exists. Standard practice: create the backend resources manually (once) via CLI, then configure Terraform to use them.

```bash
# Replace YOUR_ACCOUNT_ID with your actual AWS account ID (from sts get-caller-identity above)
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
BUCKET_NAME="tf-state-notification-${ACCOUNT_ID}-us-east-1"
LOCK_TABLE="tf-locks-notification"

echo "Bucket: $BUCKET_NAME"
echo "Table:  $LOCK_TABLE"

# Create S3 bucket for state storage
aws s3api create-bucket \
  --bucket "$BUCKET_NAME" \
  --region us-east-1

# Enable versioning (so state history is preserved — rollback capability)
aws s3api put-bucket-versioning \
  --bucket "$BUCKET_NAME" \
  --versioning-configuration Status=Enabled

# Block all public access (state files contain sensitive resource IDs)
aws s3api put-public-access-block \
  --bucket "$BUCKET_NAME" \
  --public-access-block-configuration \
    "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

# Enable server-side encryption
aws s3api put-bucket-encryption \
  --bucket "$BUCKET_NAME" \
  --server-side-encryption-configuration '{
    "Rules": [{
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "AES256"
      }
    }]
  }'

# Create DynamoDB table for state locking
aws dynamodb create-table \
  --table-name "$LOCK_TABLE" \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --region us-east-1
```

EXPECTED_OUTPUT (S3 bucket):
```json
{
    "Location": "/tf-state-notification-123456789012-us-east-1"
}
```

EXPECTED_OUTPUT (DynamoDB table):
```json
{
    "TableDescription": {
        "TableName": "tf-locks-notification",
        "TableStatus": "CREATING",
        "BillingModeSummary": {
            "BillingMode": "PAY_PER_REQUEST"
        }
    }
}
```

WHY: This is the bootstrap — the state infrastructure must exist before `terraform init` can configure the backend. In real orgs, this bootstrap is often a separate "platform-infra" repo maintained by the platform team.

#### Step 4: Write the S3 Bucket module

```bash
# --- modules/s3-bucket/variables.tf ---
cat > ~/terraform-notification-platform/modules/s3-bucket/variables.tf << 'EOF'
# Input parameters for the S3 bucket module.
# Every variable has a description — this is the module's "API documentation."

variable "bucket_name" {
  description = "Name of the S3 bucket. Must be globally unique across all AWS accounts."
  type        = string
}

variable "environment" {
  description = "Environment label (dev/staging/prod). Used in tags for cost attribution."
  type        = string
}

variable "retention_days" {
  description = "Number of days before objects expire. Dev: 1, Prod: 90."
  type        = number
  default     = 30
}

variable "enable_versioning" {
  description = "Enable S3 object versioning. Always true in prod for audit trails."
  type        = bool
  default     = false
}
EOF

# --- modules/s3-bucket/main.tf ---
cat > ~/terraform-notification-platform/modules/s3-bucket/main.tf << 'EOF'
# S3 Bucket Module — reusable bucket with encryption, lifecycle, and tagging
# Analogy: this is a TIBCO shared process — consumers instantiate it with their parameters

resource "aws_s3_bucket" "this" {
  # The bucket name comes from the calling module — no hardcoding here
  bucket = var.bucket_name

  tags = {
    Environment = var.environment
    ManagedBy   = "terraform"
    Platform    = "notification"
  }
}

# Block all public access — notification platform buckets are never public
resource "aws_s3_bucket_public_access_block" "this" {
  bucket = aws_s3_bucket.this.id

  block_public_acls       = true
  ignore_public_acls      = true
  block_public_policy     = true
  restrict_public_buckets = true
}

# Server-side encryption — always on, even in dev
resource "aws_s3_bucket_server_side_encryption_configuration" "this" {
  bucket = aws_s3_bucket.this.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

# Versioning — controlled by variable, always on in prod
resource "aws_s3_bucket_versioning" "this" {
  bucket = aws_s3_bucket.this.id

  versioning_configuration {
    # Conditional expression: if enable_versioning is true, use "Enabled"; else "Disabled"
    status = var.enable_versioning ? "Enabled" : "Disabled"
  }
}

# Lifecycle rule — expire objects after retention_days to control storage cost
resource "aws_s3_bucket_lifecycle_configuration" "this" {
  bucket = aws_s3_bucket.this.id

  rule {
    id     = "expire-old-objects"
    status = "Enabled"

    expiration {
      days = var.retention_days
    }
  }
}
EOF

# --- modules/s3-bucket/outputs.tf ---
cat > ~/terraform-notification-platform/modules/s3-bucket/outputs.tf << 'EOF'
# Outputs are the module's published interface.
# Downstream modules reference these values instead of hardcoding resource IDs.

output "bucket_id" {
  description = "The name/ID of the bucket (same value, S3 uses name as ID)"
  value       = aws_s3_bucket.this.id
}

output "bucket_arn" {
  description = "The ARN of the bucket — used in IAM policy documents"
  value       = aws_s3_bucket.this.arn
}
EOF

echo "S3 module files written."
ls ~/terraform-notification-platform/modules/s3-bucket/
```

EXPECTED_OUTPUT:
```
S3 module files written.
main.tf  outputs.tf  variables.tf
```

WHY: Three-file module structure (main/variables/outputs) is the Terraform community standard. It separates concerns: inputs, logic, and published interface — the same separation you'd enforce in any shared library.

#### Step 5: Write the SQS Queue module

```bash
# --- modules/sqs-queue/variables.tf ---
cat > ~/terraform-notification-platform/modules/sqs-queue/variables.tf << 'EOF'
variable "queue_name" {
  description = "Name of the SQS queue. Suffix .fifo for FIFO queues."
  type        = string
}

variable "environment" {
  description = "Environment label (dev/staging/prod)."
  type        = string
}

variable "message_retention_seconds" {
  description = "How long SQS retains unprocessed messages. Dev: 3600 (1hr), Prod: 1209600 (14 days)."
  type        = number
  default     = 86400
}

variable "visibility_timeout_seconds" {
  description = "How long a message is hidden after a consumer reads it. Match your worker processing time + buffer."
  type        = number
  default     = 30
}

variable "max_receive_count" {
  description = "Number of times a message can be received before moving to DLQ."
  type        = number
  default     = 3
}
EOF

# --- modules/sqs-queue/main.tf ---
cat > ~/terraform-notification-platform/modules/sqs-queue/main.tf << 'EOF'
# SQS Queue Module — notification queue with Dead Letter Queue
# Analogy: main queue = RabbitMQ work queue; DLQ = error exchange/dead letter exchange

# Dead Letter Queue — receives messages that failed processing max_receive_count times
resource "aws_sqs_queue" "dlq" {
  # DLQ name is derived from the main queue name — keeps naming consistent
  name                      = "${var.queue_name}-dlq"
  message_retention_seconds = 1209600  # 14 days — DLQ always retains long for investigation

  tags = {
    Environment = var.environment
    ManagedBy   = "terraform"
    Platform    = "notification"
    Role        = "dead-letter-queue"
  }
}

# Main notification queue
resource "aws_sqs_queue" "main" {
  name                       = var.queue_name
  message_retention_seconds  = var.message_retention_seconds
  visibility_timeout_seconds = var.visibility_timeout_seconds

  # Wire DLQ: after max_receive_count failed attempts, message moves to DLQ
  redrive_policy = jsonencode({
    deadLetterTargetArn = aws_sqs_queue.dlq.arn
    maxReceiveCount     = var.max_receive_count
  })

  tags = {
    Environment = var.environment
    ManagedBy   = "terraform"
    Platform    = "notification"
    Role        = "main-queue"
  }
}
EOF

# --- modules/sqs-queue/outputs.tf ---
cat > ~/terraform-notification-platform/modules/sqs-queue/outputs.tf << 'EOF'
output "queue_url" {
  description = "The URL of the main SQS queue — used by producers/consumers to send/receive messages"
  value       = aws_sqs_queue.main.url
}

output "queue_arn" {
  description = "ARN of the main queue — used in IAM policies and event source mappings"
  value       = aws_sqs_queue.main.arn
}

output "dlq_url" {
  description = "URL of the Dead Letter Queue — wire to your monitoring alert"
  value       = aws_sqs_queue.dlq.url
}

output "dlq_arn" {
  description = "ARN of the Dead Letter Queue"
  value       = aws_sqs_queue.dlq.arn
}
EOF

echo "SQS module files written."
ls ~/terraform-notification-platform/modules/sqs-queue/
```

EXPECTED_OUTPUT:
```
SQS module files written.
main.tf  outputs.tf  variables.tf
```

WHY: The DLQ is created inside the module — every queue provisioned through this module automatically gets a DLQ. This is a module-enforced policy. You cannot forget the DLQ because the module does not allow it.

#### Step 6: Write the IAM Role module

```bash
# --- modules/iam-role/variables.tf ---
cat > ~/terraform-notification-platform/modules/iam-role/variables.tf << 'EOF'
variable "role_name" {
  description = "Name of the IAM role."
  type        = string
}

variable "environment" {
  description = "Environment label."
  type        = string
}

variable "trusted_service" {
  description = "AWS service principal allowed to assume this role (e.g., 'lambda.amazonaws.com', 'ec2.amazonaws.com')."
  type        = string
}

variable "policy_arns" {
  description = "List of managed policy ARNs to attach. Keep this minimal — least privilege."
  type        = list(string)
  default     = []
}
EOF

# --- modules/iam-role/main.tf ---
cat > ~/terraform-notification-platform/modules/iam-role/main.tf << 'EOF'
# IAM Role Module — creates a role with trust policy and attaches managed policies
# Analogy: trust policy = "who is allowed to use this service account"
#          attached policies = "what that service account is allowed to do"

# The trust policy (assume role policy) defines WHO can use this role
data "aws_iam_policy_document" "assume_role" {
  statement {
    effect  = "Allow"
    actions = ["sts:AssumeRole"]

    principals {
      type        = "Service"
      # The trusted_service variable determines which AWS service can assume this role
      identifiers = [var.trusted_service]
    }
  }
}

# Create the IAM role with the trust policy
resource "aws_iam_role" "this" {
  name               = var.role_name
  assume_role_policy = data.aws_iam_policy_document.assume_role.json

  tags = {
    Environment = var.environment
    ManagedBy   = "terraform"
    Platform    = "notification"
  }
}

# Attach each managed policy in the list to the role
# count + element pattern: one attachment resource per policy ARN
resource "aws_iam_role_policy_attachment" "this" {
  # Create one attachment per policy ARN in the list
  count = length(var.policy_arns)

  role       = aws_iam_role.this.name
  policy_arn = var.policy_arns[count.index]
}
EOF

# --- modules/iam-role/outputs.tf ---
cat > ~/terraform-notification-platform/modules/iam-role/outputs.tf << 'EOF'
output "role_arn" {
  description = "ARN of the IAM role — use this in resource policies and IRSA annotations"
  value       = aws_iam_role.this.arn
}

output "role_name" {
  description = "Name of the IAM role — use this to attach additional inline policies if needed"
  value       = aws_iam_role.this.name
}
EOF

echo "IAM module files written."
ls ~/terraform-notification-platform/modules/iam-role/
```

EXPECTED_OUTPUT:
```
IAM module files written.
main.tf  outputs.tf  variables.tf
```

WHY: The `count` + list pattern for policy attachments is a common Terraform idiom. It means you can attach 0, 1, or 10 policies to a role using the same module call — no module changes needed.

  retention_days    = var.notification_bucket_retention_days
  enable_versioning = var.enable_versioning

## STAR Anchor

"At T-Mobile, I manage the IaC for our 25M msg/day notification platform. When I took over the team, infrastructure was provisioned by hand with no version control. I introduced a module-based Terraform structure — separate modules for queues, buckets, and IAM roles — and enforced remote state with DynamoDB locking. The result was that our quarterly compliance audit, which used to take 2 days of manual console review, became a 15-minute `terraform show` report. The Situation was manual drift and audit risk; the Task was to build governed IaC; the Action was modular design with remote state; the Result was measurable audit time reduction and zero configuration drift incidents in the following two quarters."

**Q3 — Lock recovery (T-Mobile framing)**:
"We hit this exact scenario during a Kafka schema migration. Our GitLab CI runner lost connectivity mid-apply, and the DynamoDB lock record was never released. The runbook is: (1) confirm the lock holder is actually dead — `terraform force-unlock` is irreversible, (2) get the lock ID from the DynamoDB table, (3) run `terraform force-unlock <lock-id>` — this removes the lock record. The risk is that if the original process is still running (not dead), you now have two concurrent applies. We mitigated this by adding a dead-man switch: CI jobs write a heartbeat to DynamoDB every 30 seconds; if the heartbeat stops and no lock release is written, a Lambda cleans the lock after 5 minutes."

**Q4 — Governance model**:
"For the notification platform, I implemented a three-gate model: (1) every Terraform change goes through a PR with a `terraform plan` output attached, reviewed by a senior SRE; (2) applies to staging are automated on merge to main; (3) applies to prod require a second human approval in our CI tool (we use GitLab's environment approval gates). We also added OPA/Conftest policies that run in CI to block any SQS queue without a DLQ, any S3 bucket without encryption, and any IAM role with wildcard actions — these fail the plan before a human even reviews it."

## Tradeoffs and Failure Modes

**Tradeoff 1 — Shared state vs. per-team state files**:
A single state file for all infrastructure is simple to start but becomes a chokepoint — teams block each other during applies, and a corrupted state file is a global outage. Per-team state files scale better but require cross-state data sharing via Terraform remote state data sources (`terraform_remote_state`) or Parameter Store. For 15 teams, the correct answer is one state file per service per environment, with a shared "platform" state for baseline networking. The interview answer: "state file boundary should match blast radius boundary."

**Tradeoff 2 — Workspaces vs. directory-per-environment**:
Workspaces are fast to set up and require no code duplication. But they share the same module code, same provider config, and same backend — which means a misconfigured workspace variable can accidentally apply prod settings to dev or vice versa. Directory-per-environment (with Terragrunt or symlinks) gives harder isolation: separate backend configs, separate state, separate CI pipelines. At startup scale, workspaces are acceptable. At enterprise scale (100+ engineers), directories + Terragrunt is the standard.

**Tradeoff 3 — Auto-approve in CI vs. manual approval gates**:
`-auto-approve` in CI enables full automation — fast feedback, no human bottleneck. The failure mode is that a bad plan gets applied to prod without human review, which has caused multi-hour outages at scale (Cloudflare, Fastly incidents have IaC misconfig root causes). The governance model: auto-approve for dev only; staging requires passing a plan review check; prod requires an explicit approval in the CI interface. The `APPLY_APPROVAL` environment variable pattern in Atlantis or GitLab environments operationalizes this without slowing down the non-prod feedback loop.

## Confidence Check — State Locking Race Condition

Can you explain this without notes?

**The scenario**: Two engineers both run `terraform apply` at exactly the same time against the same state file.

**What happens without locking**: Both processes read the current state (say, 5 resources). Both compute a plan. Both write their results. The second writer overwrites the first. Now your state file says 10 resources exist but only 7 do — or worse, both tried to create the same resource with different configurations. State is corrupted.

**What DynamoDB locking does**: Before writing state, Terraform calls `PutItem` on the DynamoDB lock table with a condition: `attribute_not_exists(LockID)`. This is an atomic conditional write — DynamoDB guarantees only one writer succeeds. The loser gets a `ConditionalCheckFailedException`, which Terraform surfaces as "Error acquiring the state lock." The winner proceeds. The loser retries or fails cleanly.

**The race condition that can still occur**: If the winner crashes after acquiring the lock but before releasing it, the lock record stays in DynamoDB forever. No other process can acquire the lock. This is the "orphaned lock" problem — fixed with `terraform force-unlock <lock-id>`, which requires human judgment to verify the original process is truly dead before running.

**Director framing**: "State locking is a distributed consensus problem solved with optimistic locking on DynamoDB. The business risk is either double-applies (without locking) or indefinite CI paralysis (with an orphaned lock). The runbook for lock recovery is a Tier 2 operational document — it requires human judgment, not automation, because force-unlocking a live process destroys infrastructure integrity."

If you can explain this in three minutes without notes, you are at confidence level 4 on state management — meeting the interview threshold for Staff/Principal SRE roles.
