# Terraform Fundamentals — What IaC Is and Why State Matters | Week 1 Day 1

## Why This Matters

Every SRE Director interview at a cloud-native company will ask how you govern infrastructure repeatability and drift. Terraform is the industry-standard answer — it appears in 78% of DevSecOps/SRE job postings at the Staff+ level. If you cannot explain state management and why it matters for team safety, you will not clear the infrastructure round at AWS, Databricks, or Snowflake.

## Analogy

Think about how you manage RabbitMQ topology at T-Mobile: exchanges, queues, bindings — if someone manually creates a queue in production that does not exist in your deployment scripts, you have drift. The next automated deploy might delete it, or worse, you will not know it exists until a consumer breaks at 2am.

Terraform solves the exact same problem for cloud infrastructure. Instead of RabbitMQ queues, the resources are S3 buckets, IAM roles, VPCs, and EKS clusters. The **state file** is Terraform's mental model of what it deployed — like a live inventory of every resource it owns. If the state and reality diverge, Terraform makes dangerous decisions.

Another frame: your incident command process has a single source of truth — the incident ticket, not five people's Slack DMs. Terraform state is that single source of truth for infrastructure. Lose it, corrupt it, or let two people write to it simultaneously, and you have an infrastructure incident.

30-second version: Terraform is a desired-state enforcer, not a template. You declare what infrastructure should exist. Terraform figures out how to make reality match that declaration — and tracks the gap every time you run it.

2-minute version:

You already know this pattern — it's exactly how Kubernetes works.

Concept	Kubernetes	Terraform
Desired state file	deployment.yaml (3 replicas)	main.tf (1 VPC, 2 subnets)
Actual state	Running pods in the cluster	Real AWS resources
State store	etcd	terraform.tfstate file
Reconciliation trigger	Continuous (control loop)	On-demand (terraform apply)
Drift detection	K8s notices a pod died → restarts it	terraform plan shows what diverged
The .tf files are not templates. Templates say "here's a recipe, run it once." Terraform says "here is the truth about what should exist — I will make reality match it, and I will track every resource I own."

The critical difference from CloudFormation or ARM templates: the state file. This is what makes Terraform IaC rather than just a provisioning script. The state file records exactly what Terraform created. When you run terraform plan next time, it compares three things:


.tf files          ← what you want
    +
tfstate file       ← what Terraform last knew it created
    =
Plan output        ← delta to get from current reality to desired state
If someone manually deleted a subnet in the AWS console, terraform plan will say: "I need to create that subnet." If you delete it from your .tf file, terraform plan will say: "I need to destroy that subnet." Reality always gets pulled toward the .tf declaration.

5-minute version — where the analogy breaks and what to watch for:

Where it's NOT like Kubernetes: K8s has a continuous reconciliation loop — the controller watches constantly and fixes drift automatically. Terraform only reconciles when you run it. If someone manually deletes a resource at 2am, Terraform does nothing until you next run terraform plan or terraform apply. This is called drift and it's a real operational problem in teams where people also have console access.

The state file is the source of pain: The state file is what Terraform uses to understand what it owns. Three problems you'll hit:

State drift: Manual console changes that Terraform doesn't know about. Detected with terraform plan, reconciled with terraform apply.
State corruption: Two people run terraform apply simultaneously → state file conflict. Solved with remote state (S3 + DynamoDB lock — you'll build this in Week 1 Day 1).
State import: You have existing AWS resources not created by Terraform. You have to terraform import them to bring them under Terraform's management.
Variables make it look like a template — but it isn't:


variable "environment" { default = "dev" }

resource "aws_s3_bucket" "state" {
  bucket = "autotrader-tfstate-${var.environment}"
}
This looks like a template with ${var.environment} as a placeholder. But the variable isn't just text substitution — it determines which real AWS resource Terraform is responsible for. environment = "prod" makes Terraform manage a different S3 bucket entirely with its own state tracking.

The mental model that sticks:

Terraform is a CMDB with enforcement. Your .tf files are the CMDB entries for what should exist. terraform apply is the remediation engine that makes AWS match the CMDB. terraform plan is the compliance audit that shows you the gap.

You've run Splunk alerts → automated remediation pipelines. Terraform is that same pattern but for infrastructure provisioning: desired state defined once, actioned on demand, drift surfaced explicitly.

## 30-Second Version

Terraform is a tool that lets you describe your entire cloud infrastructure in text files, then automatically creates, updates, or deletes resources to match that description. It keeps a **state file** that tracks everything it deployed. The core workflow is four commands: `init` (download plugins), `plan` (show what will change), `apply` (make it so), `destroy` (tear it down). State files must never be stored in Git because they contain secrets and because two people writing to the same file simultaneously will corrupt it — instead, use S3 with a DynamoDB lock table.

---

## 2-Minute Version

### What Terraform Actually Does

Terraform reads your `.tf` files, compares what you described against what its state file says already exists, then calls cloud provider APIs to close the gap. This is called **reconciliation** — the same loop your Kubernetes controllers run, just triggered manually rather than continuously.

The workflow has four phases:

1. **init** — Downloads the AWS provider plugin (the adapter that knows how to call AWS APIs). This is a one-time setup per working directory.
2. **plan** — Reads your `.tf` files, reads the state file, calls AWS to refresh current state, then prints a diff: `+` means create, `-` means destroy, `~` means modify in-place.
3. **apply** — Executes the plan. Terraform calls AWS APIs, waits for resources to be ready, then writes the results into the state file.
4. **destroy** — The inverse of apply. Terraform reads the state file and deletes everything it knows about.

### Why State Is the Dangerous Part

The state file (`terraform.tfstate`) is a JSON document that maps your resource names to their real AWS IDs. Example: your `.tf` file says `aws_s3_bucket.training_bucket` — the state file records that this corresponds to AWS bucket `my-training-bucket-abc123`.

Without the state file, Terraform has no memory. It cannot know whether that S3 bucket already exists or needs to be created. It would try to create it again, fail, and leave you debugging.

State file risks:
- **Secrets leak**: State often contains resource attributes including passwords, private keys, and connection strings in plaintext
- **Concurrent writes**: Two engineers running `apply` simultaneously will corrupt the state file — last write wins, and the loser's changes are invisible to Terraform
- **Git history**: Secrets checked into Git live forever, even after deletion

The solution is a **remote backend**: store state in S3 (encrypted, versioned) and use a DynamoDB table as a lock. The lock works like a mutex — only one `apply` runs at a time, and if it crashes, the lock entry can be manually released.

## 5-Minute Version

### HCL Syntax — The Four Building Blocks You Use Every Day

**Provider block** — tells Terraform which cloud adapter to load:
```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"   # accept 5.x but not 6.x
    }
  }
}

provider "aws" {
  region = "us-east-1"
}
```

**Resource block** — declares one cloud object:
```hcl
resource "aws_s3_bucket" "training_bucket" {
  bucket = "vishwesh-terraform-training-2026"
  # format: resource_type . local_name
  # resource_type maps to an AWS API
  # local_name is your internal reference — never reaches AWS
}
```

**Variable block** — makes your code reusable:
```hcl
variable "bucket_name" {
  description = "Name for the S3 training bucket"
  type        = string
  default     = "vishwesh-terraform-training-2026"
}
# Reference it as: var.bucket_name
```

**Output block** — exports a value for other modules or humans to read:
```hcl
output "bucket_arn" {
  description = "ARN of the created bucket"
  value       = aws_s3_bucket.training_bucket.arn
  # format: resource_type . local_name . attribute
}
```

**Locals block** — computes derived values to avoid repetition:
```hcl
locals {
  common_tags = {
    Environment = "training"
    Owner       = "vishweshwar"
    Week        = "1"
  }
}
# Reference as: local.common_tags
```

### Tradeoffs — What You Must Know for Director-Level Interviews

**Terraform vs. CloudFormation**: Terraform is cloud-agnostic (same tool for AWS + GCP + Azure), has a much larger module ecosystem, and has better state management UX. CloudFormation has tighter AWS integration, native stack rollback, and no state file to manage (AWS manages it). Most cloud-native orgs choose Terraform for multi-cloud or tool consistency reasons.

**Terraform vs. Pulumi**: Pulumi uses real programming languages (Python, TypeScript) — loops, conditionals, and unit tests work naturally. Terraform HCL is a DSL that needs workarounds for loops (`count`, `for_each`). For teams with strong developers, Pulumi wins on expressiveness. For ops teams, HCL is simpler to read in a PR review.

**The state problem at scale**: In a large organization, a single state file for all infrastructure becomes a blast radius risk — one bad apply can destroy unrelated resources. Solution: **workspace separation** (dev/staging/prod state files are separate) and **module boundaries** (each team owns their state). This is a governance decision, not a technical one — it belongs in your IaC operating model.

**Drift detection**: Terraform does not continuously watch for drift like a Kubernetes controller does. You must run `terraform plan` on a schedule (typically a CI/CD pipeline nightly job) to surface drift. Tools like Atlantis or Spacelift automate this.

**Failure modes to know**:
- State file corruption from concurrent apply: mitigated by DynamoDB locking
- Partial apply (apply fails mid-run): state records what succeeded; next apply retries the rest — usually recovers cleanly
- State file diverges from reality (someone manually deleted a resource in AWS console): `terraform import` to resync, or `terraform state rm` to forget the resource

## Director / VP Lens

**How an IC implements this**: Writes `.tf` files, runs `plan` and `apply` locally or in a CI job, debugs state issues, manages module versions.

**How you govern this as a Director/VP**:

1. **IaC policy**: No manual resource creation in production AWS — all changes go through Terraform PRs with required reviewer approval. This is enforced via AWS SCPs (Service Control Policies) that deny console-based resource creation outside of break-glass procedures.

2. **State access control**: S3 bucket for state is locked down to a CI/CD service role and a break-glass IAM group. Individual engineers do not have write access to the production state bucket. This is the infrastructure equivalent of your change freeze policy.

3. **Module governance**: Your team maintains a private module registry (Terraform Cloud or a Git-based one). Teams consume approved modules — they do not write raw `aws_` resources directly. This is your golden path: safe defaults baked in, security controls pre-applied.

4. **Drift as an SLI**: You instrument your nightly `terraform plan` output and alert on plan diffs that exceed a threshold. Drift is a reliability risk — a manually created resource has no backup, no tagging, no monitoring, and no runbook.

5. **AI dimension**: GitHub Copilot and tools like HashiCorp's AI Assistant can generate Terraform modules, but generated code must go through the same PR review and `tfsec`/`checkov` policy scan that human-written code does. The review gate is the control, not the authorship.

**Q1**: You run `terraform apply` successfully and create an S3 bucket. A colleague then manually deletes that bucket from the AWS console. You run `terraform plan` — what does Terraform say, and why?

**Q2**: Why is it dangerous to store `terraform.tfstate` in a Git repository, even in a private repo?

**Q3**: What is the purpose of the DynamoDB table in the S3 remote backend setup? What specific failure scenario does it prevent?

---

## ANSWER KEY — Comprehension Check

**A1**: Terraform will show a plan to **create** the bucket again (shown as `+`). This is because Terraform's state file still records that the bucket exists — but when Terraform refreshes state (reads current AWS reality), it sees the bucket is gone. The diff between state and reality triggers a recreate. This is Terraform detecting drift and self-healing it on the next apply.

**A2**: Two reasons. First, the state file contains **sensitive data in plaintext** — database passwords, private keys, and API tokens appear as resource attributes. Git history preserves these forever, even after deletion. Second, if two engineers have local copies of the state file and both run `apply`, the last `push` wins — the other engineer's state is silently overwritten, creating a divergence between Terraform's belief and AWS reality. This divergence can cause Terraform to recreate resources that already exist, or worse, fail to destroy resources it no longer knows about.

**A3**: The DynamoDB table is a **distributed lock** (mutex). It prevents two engineers — or two CI/CD pipeline runs — from running `terraform apply` against the same state file simultaneously. Without the lock, both processes could read the same state, make conflicting changes, and then both write back — last write wins and corrupts the state. With the lock, the second `apply` waits (or fails fast with a lock error) until the first completes. The specific failure scenario prevented: concurrent apply during a deployment — for example, an automated nightly apply running at the same time a developer manually applies a hotfix.

### Step 3: Create Your Lab Directory and Write main.tf

```bash
# Create a clean working directory for this lab
mkdir -p ~/terraform-labs/week1-day1
cd ~/terraform-labs/week1-day1
```

Now create the Terraform configuration file. This is the declaration of what you want AWS to build:

```bash
cat > main.tf << 'EOF'
# ============================================================
# TERRAFORM CONFIGURATION BLOCK
# Declares which providers we need and their version constraints.
# This is like specifying your dependency versions in package.json.
# ============================================================
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"   # Official AWS adapter from HashiCorp registry
      version = "~> 5.0"          # Accept any 5.x version, reject 6.x (breaking changes)
    }
  }
  required_version = ">= 1.5.0"  # Minimum Terraform CLI version for this config
}

# ============================================================
# PROVIDER BLOCK
# Configures the AWS adapter — which region to operate in.
# Think of this as your "connection string" to AWS.
# ============================================================
provider "aws" {
  region = "us-east-1"
  # Credentials come from environment variables or ~/.aws/credentials
  # Terraform inherits them from your AWS CLI config automatically
}

# ============================================================
# LOCALS BLOCK
# Derived/computed values — avoids repeating the same string 
# in multiple places. Like eval in Splunk — compute once, reference many times.
# ============================================================
locals {
  # S3 bucket names must be globally unique across all AWS accounts.
  # We append a random suffix to avoid collisions.
  bucket_name = "vishwesh-tf-training-${random_id.suffix.hex}"

  # Tags to apply to every resource — this is your inventory metadata.
  # In production, tags drive cost allocation, alerting routing, and CMDB sync.
  common_tags = {
    Environment = "training"
    Owner       = "vishweshwar"
    Week        = "week1"
    ManagedBy   = "terraform"
  }
}

# ============================================================
# RESOURCE: random_id
# Generates a random hex string to ensure our S3 bucket name is unique.
# S3 bucket names are global — "my-bucket" is almost certainly taken.
# ============================================================
resource "random_id" "suffix" {
  byte_length = 4  # Produces an 8-character hex string, e.g. "a3f91bc2"
}

# ============================================================
# RESOURCE: aws_s3_bucket
# Declares one S3 bucket. Terraform will call the AWS API to create this.
# Format: resource "resource_type" "local_name" { ... }
# resource_type = the AWS service object type (maps to an API call)
# local_name    = YOUR internal reference — never sent to AWS
# ============================================================
resource "aws_s3_bucket" "training_bucket" {
  bucket = local.bucket_name  # The actual AWS bucket name — must be globally unique

  tags = local.common_tags
}

# ============================================================
# RESOURCE: aws_s3_bucket_versioning
# Enables versioning on the bucket — S3 keeps all versions of every object.
# Separate resource block because AWS treats this as a separate API call.
# This pattern (main resource + configuration resources) is common in AWS provider.
# ============================================================
resource "aws_s3_bucket_versioning" "training_bucket_versioning" {
  bucket = aws_s3_bucket.training_bucket.id  # Reference the bucket we just declared above
                                              # Format: resource_type.local_name.attribute

  versioning_configuration {
    status = "Enabled"
  }
}

# ============================================================
# OUTPUT BLOCK
# Exports values after apply — so you can see what was created.
# Like a pipeline artifact: downstream steps (or you) can consume these.
# ============================================================
output "bucket_name" {
  description = "The globally unique name of the created S3 bucket"
  value       = aws_s3_bucket.training_bucket.bucket
}

output "bucket_arn" {
  description = "The ARN (Amazon Resource Name) — the globally unique ID for this resource"
  value       = aws_s3_bucket.training_bucket.arn
}

output "bucket_region" {
  description = "The AWS region where the bucket was created"
  value       = aws_s3_bucket.training_bucket.region
}
EOF
```

Verify the file was created correctly:

```bash
cat main.tf
```

**EXPECTED_OUTPUT**: The full contents of the file you just wrote, with all comments and resource blocks visible.

**WHY**: This single file contains the full desired state of your infrastructure. Terraform will read this and determine what API calls to make. The comments explain intent — this is production discipline: IaC without comments is a runbook without steps.

## Lab ANSWER KEY

**Step 1 fails (auth error)**: Run `aws configure list` to see what credentials are active. Run `aws configure` to re-enter your access key and secret. Verify the IAM user has `AmazonS3FullAccess` or `AdministratorAccess`.

**Step 2 fails (apt error on WSL2)**: Try `sudo apt-get update --fix-missing` first. If HashiCorp GPG fails, verify your WSL2 Ubuntu version with `lsb_release -a` — it should be 20.04 or 22.04.

**Step 4 (init) fails with "no provider"**: Delete `.terraform/` directory and run `terraform init` again. This happens if a partial download was interrupted.

**Step 5 (plan) fails with "Error: No valid credential sources found"**: Your AWS CLI credentials are not visible to Terraform. Run `export AWS_PROFILE=default` (or whatever profile name you configured) and try again.

**Step 6 (apply) fails with "BucketAlreadyExists"**: The random_id was not unique enough — this is rare. Delete `terraform.tfstate`, run `terraform destroy` if anything was partially created, then run `apply` again to generate a new random ID.

**Step 6 (apply) fails with "AccessDenied"**: Your `training-admin` IAM user is missing S3 permissions. Verify in AWS IAM console that `AdministratorAccess` is attached.

**Step 7 — state file is empty or missing**: This means `apply` did not complete successfully. Check the error output from the apply step.

**Step 9 (destroy) fails with "BucketNotEmpty"**: If you manually uploaded files to the bucket in AWS console, S3 will refuse deletion. Run `aws s3 rm s3://BUCKET-NAME --recursive` to empty it first, then retry `terraform destroy`.

## Tradeoffs and Failure Modes

**1. State file as a single point of failure**
The state file is simultaneously the most critical artifact in your infrastructure and the most vulnerable. If your S3 backend bucket is accidentally deleted (or the IAM role loses access), your entire infrastructure becomes unmanaged — resources exist in AWS but Terraform has no knowledge of them. Mitigation: enable S3 versioning on the state bucket (yes, Terraform managing the bucket that holds its own state is a chicken-and-egg problem you solve by bootstrapping manually), and set object lock retention. Director implication: treat the state bucket as Tier 1 infrastructure — same backup and access controls as your production database.

**2. Terraform is not a continuous control loop — it is a one-shot executor**
Unlike Kubernetes controllers that continuously reconcile desired state, Terraform only reconciles when you run it. A developer who logs into the AWS console and deletes a security group at 2pm will not trigger any Terraform response. That drift lives until the next `terraform plan` is run. At scale, this means your infrastructure is only as good as your plan-frequency discipline. Mitigation: run `terraform plan` in a scheduled CI job and alert on non-empty plans. This is the IaC equivalent of a nightly configuration compliance scan (which you likely already have a concept of from your incident governance work).

**3. The module versioning trap**
Terraform modules are like shared libraries. If you pin to `source = "git::https://github.com/your-org/modules.git?ref=main"` without a version tag, every `terraform init` pulls the latest commit — a breaking change in the module silently breaks every consumer on their next init. If you do not pin at all, you get this for free on upgrade. Mitigation: pin modules to a specific git tag or semantic version. Enforce this via a policy-as-code check (tflint, checkov) in your CI pipeline. Director implication: your module registry governance policy must require tagged releases and a changelog — same discipline as your application library releases.
