# Terraform Drift Detection and Import | Week 1 Day 4

## Why This Matters

In production SRE environments, configuration drift is one of the most insidious failure modes — it erodes the contract between your declared intent and actual system state, making runbooks unreliable and postmortems harder to write. Interviewers at Staff/Principal level always ask "how do you prevent configuration drift?" because it's a proxy question for your IaC maturity. Understanding `terraform import` is equally critical because no greenfield team ever starts clean — you will always inherit manually-provisioned resources that need to come under management.

## Analogy

Think of a T-Mobile network operations center. Your NOC runbook says: "Circuit breaker CB-14 is in CLOSED state, rated for 20-amp load." During a 2am incident, an engineer bypasses the change process and physically flips CB-14 to a lower threshold to prevent cascading failure. Incident over. But now your runbook is wrong. Next time someone follows the runbook, they provision for 20 amps — and the breaker trips again at 15 because reality diverged from documentation.

This is drift. Your Terraform state file is the runbook. Your AWS account is the circuit box. Terraform plan is the engineer who walks the floor comparing the runbook to the physical world and marks every discrepancy in red.

`terraform import` is the process of retroactively writing a runbook entry for a circuit breaker that was installed without paperwork — you're bringing an undocumented device under official configuration management.

## Vocabulary Translation Table

| Their Term | Your Existing Mental Model |
|---|---|
| Drift | Config delta between your NOC runbook and actual live state |

| `terraform plan` | The pre-change diff — like a change impact analysis (CIA) before a maintenance window |
| `terraform refresh` | Terraform re-reads real infra and updates its internal state record — like a CMDB sync job |
| `terraform import` | Registering an undocumented asset into your CMDB — reverse-discovery |

| `terraform state rm` | Removing a CMDB entry while leaving the actual hardware running |
| Desired state | What your `.tf` files declare — the approved architecture diagram |
| Actual state | What AWS actually has right now — the live environment |
| State file | `terraform.tfstate` — the delta tracker between desired and actual |
| Reconciliation | Running `terraform apply` to bring actual state back to desired state |

## 30-Second Version

Drift is when your live infrastructure no longer matches what Terraform declared. It happens when engineers make manual changes — AWS Console, direct CLI calls, or other automation tools — without going through Terraform. `terraform plan` detects drift by comparing your `.tf` files against the state file and then against real AWS. `terraform import` is the escape hatch for resources that already exist and need to be adopted into Terraform management without destroying and recreating them.

## 2-Minute Version

**How drift happens:**

1. **Console cowboy problem**: Engineer logs into AWS Console during an incident and modifies a security group rule. The change is never committed to Terraform code. `terraform.tfstate` still reflects the old rule.

2. **Failed partial apply**: Terraform apply starts, creates resource A, then fails before creating resource B. State file now shows A as managed but something about A's configuration may already be out of spec if the error caused partial resource mutation.

3. **External automation**: Another tool — a Lambda function, an Ansible playbook, a competing team's Terraform — modifies a resource Terraform also manages.

4. **Resource replacement by AWS**: Auto Scaling, Aurora failover, Fargate task replacement — these can change resource identifiers or attributes that Terraform doesn't expect.

**How Terraform detects drift:**

When you run `terraform plan`, Terraform does two comparisons:
- Step 1: Read your `.tf` config files (desired state)
- Step 2: Read the state file (last known state)
- Step 3: Call AWS APIs to read actual current state (real state)
- Step 4: Produce a diff between desired state and real state

If real state differs from state file records, that's drift. The plan output shows `~` (update in-place) or `-/+` (destroy and recreate).

**`terraform refresh`** is the operation that performs Step 3 separately — it syncs the state file to match real AWS, without applying any changes. Warning: this can cause your next `plan` to show drift you didn't know about, which is useful for discovery.

**`terraform import`** bridges the gap for resources created outside Terraform. You write a resource block in your `.tf` file, then run `import` to associate that block with the existing AWS resource ID. Terraform writes a state entry for it. After import, `terraform plan` should show no changes (if your `.tf` exactly matches what's in AWS).

## 5-Minute Version

**The drift lifecycle and its failure modes:**

Drift is not just a configuration management annoyance — it has operational blast radius:

- **Security drift**: Someone opens port 22 in a security group for a debug session and forgets to close it. Terraform doesn't know. Your SOC scan finds it six weeks later. The diff between your IaC and reality is now a compliance finding.
- **Cost drift**: Someone doubles an RDS instance class for a performance test and forgets to downscale. Terraform shows the old instance type. Your FinOps review catches it at month end.
- **Reliability drift**: A load balancer health check threshold was manually changed. Your Terraform declares 3 unhealthy threshold. Actual is 2. During an incident recovery, `terraform apply` silently changes it back to 3 — and now a degraded service that was surviving at 2 fails health checks.

**The import workflow — where engineers get tripped up:**

The import process is deceptively simple in concept but has two sharp edges:

1. **You must write the resource block first.** Unlike discovery tools (AWS Config, Terraformer), `terraform import` does not generate `.tf` code for you. You write the block, then import. If your block is wrong, your next `plan` will show a diff — Terraform will try to "fix" the resource to match your declaration. This can cause unintended changes.

2. **Import is not idempotent for complex resources.** Resources with nested blocks (security group rules, IAM policy documents, ECS task definitions) often require you to understand the exact AWS data model to write a `.tf` block that results in zero-diff after import.

**`terraform state mv` vs `terraform state rm` — the operator's scalpel:**

- `state mv`: Use when you refactor code (rename a module, move a resource to a different module). The resource keeps running. State tracking moves to the new path. Without this, Terraform would destroy the old resource and create a new one.
- `state rm`: Use when you want to stop managing a resource with Terraform but keep it running. Example: a shared resource that will now be managed by another team's Terraform workspace. After `state rm`, Terraform loses track of it — any future `plan` won't touch it.

**Drift prevention — the mature posture:**

Reactive drift detection (running `plan` and finding surprises) is junior-level. Senior posture:

1. **IAM guardrails**: Attach an IAM policy to your engineers' console roles that denies `ec2:AuthorizeSecurityGroupIngress`, `s3:PutBucketPolicy`, etc. — the mutation APIs on IaC-managed resources. This forces all changes through Terraform. The policy uses resource tags: `"Condition": {"StringEquals": {"aws:ResourceTag/ManagedBy": "terraform"}}`.

2. **Continuous drift detection in CI/CD**: Schedule `terraform plan` as a nightly job. Alert on non-empty plans. This is the canary in the coal mine — a plan that should be empty but isn't means someone bypassed the process.

3. **Drift-as-code with Driftctl/Terragrunt**: Purpose-built tools that scan your AWS account and report resources not tracked by any Terraform state. Finds the dark matter — resources Terraform doesn't even know exist.

4. **Tagging governance**: Every IaC-managed resource gets `ManagedBy = terraform`, `Owner = team-name`, `TerraformWorkspace = prod`. Resources without these tags are suspect.

## Director/VP Lens

**IC implementation focus:**
- Writes the `terraform import` blocks
- Runs `terraform plan` and interprets drift output
- Adds `terraform plan -detailed-exitcode` to CI pipeline (exits 2 if there are changes = drift alert)
- Writes the import helper script for batch-importing legacy resources

**Director/VP governance focus:**
- Defines the policy: "No production changes without a Terraform PR" — enforced via IAM deny policies on IaC-tagged resources, not just process
- Establishes the drift SLA: any non-empty nightly plan triggers a P3 ticket to the owning team within 24 hours
- Requires drift remediation as part of incident postmortems: "Was this resource drifted? Why wasn't it caught?"
- Owns the IaC adoption roadmap: what percentage of infra is under Terraform management? Track it as a platform metric monthly
- Frames the business case for drift prevention to leadership: "Every manually changed resource is an undocumented dependency that extends our MTTR by X minutes because the next engineer debugging it doesn't know it was changed"

**AI change to this domain:**
Tools like GitHub Copilot can now generate `terraform import` blocks from AWS resource ARNs. Terraformer and OpenTofu's import generation features are moving toward auto-generating `.tf` code from imported state — reducing the expert barrier significantly. As a Director, you want your team using these tools but governed: AI-generated import blocks still require human review because hallucinated attributes cause silent drift post-import.

## Comprehension Check

Answer these before proceeding to the lab. You need 2/3 correct.

**Q1.** An engineer opens port 3306 on a security group via the AWS Console during a database incident. No Terraform code is changed. Two days later, a teammate runs `terraform plan`. What will the plan output show, and which Terraform concept does this illustrate?

**Q2.** You have an S3 bucket that was created manually six months ago. You want to bring it under Terraform management. What is the correct sequence of steps, and what is the risk if your `.tf` resource block doesn't exactly match the bucket's actual configuration?

**Q3.** What is the difference between `terraform state rm` and deleting the resource block from your `.tf` file? What happens to the actual AWS resource in each case?

---

## ANSWER KEY — Comprehension Check

**A1.** The plan will show a `~` (update in-place) change on the security group — specifically, it will show the port 3306 ingress rule as a resource to be REMOVED (because Terraform's declared state doesn't include it). This illustrates **drift**: the actual AWS state diverged from Terraform's desired state. Terraform detected it by comparing its state file (which never had port 3306) against the real security group (which now has it). Running `terraform apply` at this point would close port 3306.

**A2.** Correct sequence:
1. Write the `resource "aws_s3_bucket" "example" {}` block in your `.tf` file
2. Run `terraform import aws_s3_bucket.example <bucket-name>`
3. Run `terraform plan` — it should show no changes if your block matches reality
4. If plan shows changes, update your `.tf` block to match actual config until plan is clean

Risk: If your `.tf` block declares attributes that differ from the actual bucket (e.g., you declared `versioning { enabled = false }` but the bucket has versioning enabled), `terraform apply` will change the bucket to match your declaration — potentially disabling versioning on a live bucket with data.

**A3.**
- `terraform state rm resource.name`: Removes the resource from Terraform's state tracking. The actual AWS resource **keeps running** untouched. Terraform will no longer manage it. If the resource block still exists in `.tf`, next `plan` will try to CREATE a new resource (it no longer knows the old one exists).
- Deleting the resource block from `.tf` (without `state rm`): On next `terraform plan`, Terraform will show the resource as to be DESTROYED. Running `apply` will delete the actual AWS resource.
- Key distinction: `state rm` = "stop tracking this, leave it alone." Removing the block = "I want this gone."

### Step 4: Simulate drift — manual change via AWS CLI

```bash
# Capture the security group ID from Terraform output
SG_ID=$(terraform output -raw security_group_id)
echo "Security Group ID: $SG_ID"

# SIMULATE: An engineer opens port 8080 via AWS CLI during an "incident"
# This is the drift-creating action — bypasses Terraform entirely
aws ec2 authorize-security-group-ingress \
  --group-id "$SG_ID" \
  --protocol tcp \
  --port 8080 \
  --cidr "0.0.0.0/0" \
  --region us-east-1

echo "Manual change applied — port 8080 now open (not in Terraform code)"
```

**EXPECTED_OUTPUT:**
```

{
    "Return": true,
    "SecurityGroupRules": [
        {
            "SecurityGroupRuleId": "sgr-0abcdef1234567890",
            "GroupId": "sg-0123456789abcdef0",
            "IsEgress": false,
```

### Step 8: Write the Terraform resource block for the existing bucket

```bash
# Get the bucket name again if needed
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
BUCKET_NAME="terraform-import-lab-${ACCOUNT_ID}"

# Add the S3 bucket resource block to main.tf
# IMPORTANT: You must write the block BEFORE running import
cat >> main.tf << EOF

# S3 bucket that was manually created — being brought under Terraform management
# The bucket name must EXACTLY match the existing bucket
resource "aws_s3_bucket" "imported" {
  bucket = "${BUCKET_NAME}"

  tags = {
    Name        = "terraform-import-lab"
    ManagedBy   = "terraform"
    Environment = "training"
    ImportedAt  = "week1-day4"
  }
}

# Versioning is a separate resource block in AWS provider v4+ 
resource "aws_s3_bucket_versioning" "imported" {
  bucket = aws_s3_bucket.imported.id

  versioning_configuration {
    status = "Enabled"
  }
}

output "imported_bucket_name" {
  value       = aws_s3_bucket.imported.bucket
  description = "Name of the imported S3 bucket"
}
EOF

echo "Resource blocks added to main.tf"
```

**EXPECTED_OUTPUT:**
```
Resource blocks added to main.tf
```

**WHY**: Terraform import requires the resource block to already exist. The block is the "intended state declaration" — import will associate the existing AWS resource with this block. If the block doesn't exist, import fails with "resource not found in configuration."

## Lab ANSWER KEY

**Step 3 issue: `terraform plan` shows error about default VPC not found**
- Cause: Your AWS account may not have a default VPC (some accounts delete it)
- Fix: Add a `vpc_id` variable pointing to an existing VPC, or create a VPC resource. Run `aws ec2 describe-vpcs --region us-east-1` to find available VPCs.

**Step 5 issue: `terraform plan` shows "No changes" after manual drift**
- Cause: Terraform's AWS provider cached the old state and didn't re-read from AWS
- Fix: Run `terraform refresh` first to force-sync state with AWS, then run `terraform plan`

**Step 9 issue: import fails with "ResourceNotFoundException"**
- Cause: The bucket name in the import command doesn't match the actual bucket name
- Fix: Run `aws s3 ls | grep terraform-import-lab` to confirm exact bucket name, then re-run import

**Step 10 issue: plan shows changes after import (force_destroy, acceleration, etc.)**
- These are S3 bucket attributes with defaults that differ between AWS and Terraform's provider defaults
- Fix: Add these explicit blocks to your `aws_s3_bucket.imported` resource to match what AWS has:
  ```hcl
  force_destroy = false
  ```
- For acceleration status: add `resource "aws_s3_bucket_accelerate_configuration"` block
- Run `terraform plan` after each addition until you reach "No changes"
- Rule of thumb: when plan shows `+` (adding attributes), you're missing those declarations in your `.tf`

**Step 12 issue: destroy fails with "BucketNotEmpty"**
- Cause: S3 buckets must be empty before deletion. Add `force_destroy = true` to the bucket resource, then run `terraform apply` followed by `terraform destroy`

## STAR Anchor

**Q1 — Legacy import at T-Mobile:**
"In my role managing the SRE team for T-Mobile's notification platform, we inherited a Cassandra and Redis estate that predated our IaC adoption. My approach was: first, audit with AWS Config and resource tagging reports to inventory all unmanaged resources — we had 340 resources in scope. Then prioritize by blast radius: data stores last (highest risk of destroy on bad import), networking first (security groups, route tables — lower risk). I ran Terraformer to generate draft resource blocks, then hand-reviewed and cleaned up each one. The highest risk we hit was a legacy S3 bucket used for platform configs that had a complex bucket policy — the imported `.tf` block didn't match exactly, and a careless apply would have wiped the policy. We caught it in plan review. Key lesson: always treat a non-empty plan post-import as a red flag, never auto-approve it."

**Q2 — 47 resource diffs on nightly plan:**
"First, I don't panic — I page the on-call team and establish a P2 incident bridge. The immediate question is: did any of these resources change in the last 24 hours? I check AWS CloudTrail for mutation API calls on those resources, filtered for calls NOT originating from the Terraform CI role. If I find manual changes, I identify who made them and what the business justification was. If it was incident response, I validate the changes are safe and commit them to Terraform code. If it was unauthorized, that's a security event. Second question: is a `terraform apply` safe right now? The answer is almost certainly no until I've reviewed all 47 diffs. I stage the apply using `-target` to reconcile clean ones first, leaving risky ones for human review."

## Tradeoffs and Failure Modes

**1. `terraform refresh` before `terraform plan` — useful but dangerous**

Running `terraform refresh` syncs your state file to match real AWS before you plan. This sounds helpful but has a sharp edge: if a resource was accidentally deleted outside Terraform, `refresh` will update the state to say "this resource doesn't exist" — and the next `plan` will show it as "to be created." If you then `apply`, Terraform creates a new resource. But if the deleted resource had data (RDS, S3, EBS), that data is already gone. In Terraform 0.15+, refresh behavior is built into `plan` by default (`-refresh=true`). In some automation scenarios, teams use `-refresh=false` to skip it for speed — but this means plan may not detect drift that happened after your last apply.

**2. Import generates state, not code — the silent mismatch trap**

`terraform import` writes to `terraform.tfstate` but generates zero `.tf` code. If your hand-written `.tf` block doesn't perfectly declare all the resource's attributes, `terraform plan` will show changes, and `apply` will mutate your resource. The insidious case is attributes that have AWS-side defaults — Terraform's provider default may differ from what AWS set. For example, S3 bucket ownership controls, default encryption, public access blocks — these are often set by AWS account-level policies but not visible in basic `.tf` blocks. Every import must be followed by a plan review before any apply is permitted.

**3. State file as a single point of failure — the statefile corruption scenario**

If you're using local state (file on disk) and multiple team members run Terraform concurrently, state file corruption is a real risk. One apply overwrites another's state changes. The fix is remote state with locking (S3 + DynamoDB, which you set up in Day 2/3). But even with remote state, state corruption can happen from network interruptions mid-apply. Recovery path: `terraform state` commands let you surgically repair state, but you need to understand the pre-corruption state from CloudTrail. This is a Staff-level skill — being able to reconstruct a state file from CloudTrail API call history is a rare capability that interviewers remember.
