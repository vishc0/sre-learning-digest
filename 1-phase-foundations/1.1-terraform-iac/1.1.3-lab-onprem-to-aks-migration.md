# Lab: Migrate an On-Prem (TKG/TKE) App to AKS — End-to-End

> Scenario: An app is running on on-prem Tanzu/TKG. You need to stand up an AKS environment, create a namespace, deploy the app via **Terraform + GitHub Actions + Argo CD**, attach a database, and harden it. This doc walks you through it step by step with real code you can paste and run.

**Estimated time**: ~6–8 hours hands-on. Do it over 2 evenings.

---

   2) GH Actions runs `terraform plan/apply`  →  provisions infra

   4) Argo CD (running in AKS) detects change  →  syncs to `app-prod` namespace

   7) App Gateway terminates TLS, WAF inspects, routes to pod via AGIC

### TKG (on-prem Tanzu) → AKS deltas to keep in mind

| Concern | On-prem TKG | AKS |
|---|---|---|
| Control plane | You run it (etcd, API server) | Managed by Azure (free SLA, paid SLA) |
| Networking | Antrea / Calico, NSX-T | Azure CNI Overlay (recommended), kube-proxy |
| Ingress | Contour / NSX ALB | App Gateway (AGIC) or NGINX |
| Identity | LDAP/AD via dex | Microsoft Entra ID + Workload Identity |
| Secrets | Vault / sealed-secrets | Key Vault + CSI Secret Store driver |
| Storage | vSphere CSI | Azure Disk / Files CSI |
| Image registry | Harbor | ACR |
| RBAC | K8s RBAC bound to LDAP groups | K8s RBAC + AAD groups via Entra integration |
| Backup | Velero to S3/NFS | Velero to Azure Blob, or AKS Backup |
| Cost | CapEx VMware | OpEx Azure |

### Bootstrap Terraform state storage

State must live somewhere safe and locked. Create it manually **once** (chicken-and-egg problem):

```powershell
$RG="rg-tfstate-eus"
$SA="tfstate$(Get-Random -Maximum 99999)"
$LOC="eastus"

az group create -n $RG -l $LOC
az storage account create -n $SA -g $RG -l $LOC --sku Standard_LRS `
  --encryption-services blob --min-tls-version TLS1_2 `
  --allow-blob-public-access false
az storage container create -n tfstate --account-name $SA
Write-Host "Save this storage account name: $SA"
```

### Set up OIDC federation from GitHub Actions to Azure (no secrets!)

This replaces the old "service principal + client secret in GitHub Secrets" pattern.

```powershell
# Create an Azure AD app + service principal
$APP=$(az ad app create --display-name "gh-actions-aks-lab" --query appId -o tsv)
$SP=$(az ad sp create --id $APP --query id -o tsv)

# Give it Contributor on the subscription (tighten later for prod)
$SUB=$(az account show --query id -o tsv)
az role assignment create --role Contributor --assignee $APP --scope "/subscriptions/$SUB"
az role assignment create --role "User Access Administrator" --assignee $APP --scope "/subscriptions/$SUB"

# Create the federated credential — trusts GitHub's OIDC token for a specific repo + branch
# Replace OWNER/REPO with your GitHub org/repo
$REPO="YOUR_GH_USER/aks-migration-lab"

az ad app federated-credential create --id $APP --parameters (@{
  name="github-main"
  issuer="https://token.actions.githubusercontent.com"
  subject="repo:$REPO`:ref:refs/heads/main"
  audiences=@("api://AzureADTokenExchange")
} | ConvertTo-Json -Compress)

# Also allow PR runs (for plan-on-PR)
az ad app federated-credential create --id $APP --parameters (@{
  name="github-pr"
  issuer="https://token.actions.githubusercontent.com"
  subject="repo:$REPO`:pull_request"
  audiences=@("api://AzureADTokenExchange")
} | ConvertTo-Json -Compress)

Write-Host "AZURE_CLIENT_ID = $APP"
Write-Host "AZURE_TENANT_ID = $(az account show --query tenantId -o tsv)"
Write-Host "AZURE_SUBSCRIPTION_ID = $SUB"
```

Store those three values as **GitHub repo variables** (Settings → Secrets and variables → Actions → Variables tab — not Secrets, since they're not sensitive when OIDC is used).

## Part 3 — Terraform: build the platform

### `terraform/backend.tf`

```hcl
terraform {
  required_version = ">= 1.6.0"
  required_providers {
    azurerm    = { source = "hashicorp/azurerm",    version = "~> 4.0" }
    azuread    = { source = "hashicorp/azuread",    version = "~> 3.0" }
    kubernetes = { source = "hashicorp/kubernetes", version = "~> 2.30" }
    helm       = { source = "hashicorp/helm",       version = "~> 2.15" }
  }
  backend "azurerm" {
    resource_group_name  = "rg-tfstate-eus"
    storage_account_name = "tfstateXXXXX"   # paste yours
    container_name       = "tfstate"
    key                  = "aks-lab.tfstate"
    use_oidc             = true
  }
}
```

### `terraform/policy.tf` — guardrails

## Part 8 — Database integration: how it actually flows

This is the part where most candidates stumble in interview. Walk through it out loud:

1. **Terraform** provisions Postgres Flexible Server in `subnet-db` with `public_network_access_enabled = false`. DNS is registered to `<server>.postgres.database.azure.com` in a private DNS zone linked to the VNet.
2. **Terraform** generates a strong password (or you pass one in via `TF_VAR_pg_admin_password`) and writes it to **Key Vault**.
3. **Terraform** creates a User-Assigned Managed Identity (`id-app-todo-prod`) and grants it `Key Vault Secrets User` on the vault.
4. **Terraform** creates a **Federated Identity Credential** that says: "this UAMI trusts AKS's OIDC issuer for the `system:serviceaccount:app-prod:todo-sa` subject."
5. **Argo CD** deploys the K8s ServiceAccount, annotated with the UAMI's client-id and `azure.workload.identity/use: "true"`.
6. When the pod starts, the **Workload Identity webhook** (an AKS add-on) projects a service account token into the pod at `/var/run/secrets/azure/tokens/azure-identity-token`.
7. The pod calls Azure AD's token endpoint, swaps the SA token for an **AAD access token** scoped to Key Vault.
8. The **Secret Store CSI driver** uses that token to fetch `pg-admin-password` from Key Vault and projects it as a K8s `Secret` (and/or mounts it as a file).
9. The pod reads `PG_PASSWORD` env var, opens a connection to `pg-todo-prod.postgres.database.azure.com` — DNS resolves to the **private IP** in `subnet-db` because of the private DNS zone link.
10. Traffic stays inside the VNet. Postgres NSG only accepts from `subnet-aks-nodes` (you'd add that NSG rule for defense in depth).

**No secret ever sits in source control. No service principal credentials. No public IP.**

### Better: skip the password entirely with AAD auth to Postgres

Production-grade: enable **Azure AD authentication on Postgres Flexible Server**, then the same UAMI can connect with an AAD token — no DB password at all. Set this up next as Lab 7-bis.

## Part 9 — Layered Security

Defense in depth — every layer below assumes the one above failed.

### Layer 1: Identity (the first wall)
- **Microsoft Entra ID** is the single source of truth for human + workload identity
- **AKS local accounts disabled** (`local_account_disabled = true`) — no `kubectl` access without AAD login
- **Azure RBAC for Kubernetes Authorization** enabled — K8s RBAC bindings reference AAD groups
- **Workload Identity** for pods → no static credentials
- **PIM** (Privileged Identity Management) for human admins → just-in-time elevation
- **Conditional Access**: require MFA + compliant device for any subscription Contributor role

### Layer 2: Network
- AKS is a **private cluster** — API server has no public IP
- ACR, Key Vault, Postgres all use **Private Endpoints** — no public surface
- **NSGs** on each subnet:
  - `subnet-aks-nodes`: allow egress to ACR/KV/PG private endpoints, deny internet by default
  - `subnet-pe`: deny all except from AKS subnet
  - `subnet-db`: only AKS subnet → 5432
- **Azure Firewall** (or NVA) in a hub VNet for egress filtering — FQDN allowlist for `*.docker.io`, `*.github.com`, etc.
- **App Gateway WAF v2** in front of everything public — OWASP CRS 3.2, custom rules per app
- Microsoft Defender for Cloud — alerts on suspicious egress, port scans, anomalies

### Layer 3: Cluster (Kubernetes layer)
- **Pod Security Standards: restricted** enforced via namespace labels (you saw it in `namespace.yaml`)
- **Azure Policy add-on for AKS** — enforces:
  - no privileged containers
  - no `hostNetwork: true`
  - images only from approved registries (`acrtodoprod.azurecr.io/*`)
  - readonly root filesystem
- **Network Policies** (Cilium) — namespace isolation; `app-prod` can talk to db egress only
- **AAD groups → K8s RBAC** mapping example:

```yaml
# manifests/rbac/app-prod-rbac.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-prod-developers
  namespace: app-prod
subjects:
  - kind: Group
    apiGroup: rbac.authorization.k8s.io
    name: <AAD_GROUP_OBJECT_ID_FOR_TODO_DEVS>
roleRef:
  kind: ClusterRole
  name: edit             # built-in: edit anything in namespace except RBAC
  apiGroup: rbac.authorization.k8s.io
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-prod-sre
  namespace: app-prod
subjects:
  - kind: Group
    apiGroup: rbac.authorization.k8s.io
    name: <AAD_GROUP_OBJECT_ID_FOR_SRE>
roleRef:
  kind: ClusterRole
  name: admin
  apiGroup: rbac.authorization.k8s.io
```

This is the **RBAC story** worth memorizing for interview:
- *Cluster-level RBAC*: AAD groups bound via `ClusterRoleBinding` — SRE leads get `cluster-admin`, app devs get **nothing at cluster scope**
- *Namespace-level RBAC*: AAD groups bound via `RoleBinding` per namespace — devs get `edit` only in their own namespace
- *Workload identity*: pods authenticate as UAMIs via federated credentials, never with embedded keys

### Layer 4: Image / supply chain
- **Trivy scan** in CI blocks CRITICAL/HIGH vulns from merging
- **ACR Tasks** can re-scan stored images nightly
- **Notary / Cosign** (optional next step) — sign images, verify signature with Kyverno/Gatekeeper before admit
- **Image pull policy**: `IfNotPresent` with **digest pinning** in prod (`@sha256:...`), not just tags

### Layer 5: Runtime
- **Microsoft Defender for Containers** — runtime anomaly detection, file integrity monitoring
- **Falco** (optional) for K8s-native runtime alerts
- Audit logs from AKS API server → Log Analytics → KQL alert rule on suspicious verbs

### Layer 6: Data
- **Postgres**: TLS required (`require_secure_transport = ON`), AAD auth, audit logging
- **Key Vault**: purge protection on, soft delete on, RBAC mode (not access policies)
- **At-rest encryption**: managed keys by default; **CMK** (customer-managed keys) for regulated data
- **Backup**: AKS Backup add-on + Postgres automated backups (PITR up to 35 days)

### Load Balancer story (interview gold)

You'll get asked "how does traffic actually reach a pod." Walk through it like this:

1. User hits `https://todo.example.com` → public DNS resolves to **App Gateway's public IP**
2. **App Gateway** terminates TLS (cert from Key Vault), **WAF** inspects request
3. App Gateway routes to a backend pool — the pool is **dynamically managed by AGIC** (Application Gateway Ingress Controller running in AKS)
4. AGIC sees the K8s `Ingress` resource and programs App Gateway's backend pool with **pod IPs directly** (because Azure CNI gives pods VNet-routable IPs)
5. App Gateway sends the request over the VNet directly to the pod, **bypassing the Service** entirely
6. Pod responds, back through App Gateway, back to user

Compare/contrast for interview:
| Option | Layer | Use case | Cost |
|---|---|---|---|
| `Service: LoadBalancer` | L4 | Quick exposure, internal LB | Cheapest (uses Azure SLB) |
| **NGINX Ingress** + internal LB | L7 | Self-managed, more flexible | LB + nodes |
| **AGIC + App Gateway** | L7 + WAF | Public ingress with WAF | $$$ but managed WAF |
| **Front Door** + AGIC | Global L7 | Multi-region, edge caching | $$$$ |
| **Service Mesh (Istio)** | L7 in-cluster | Service-to-service mTLS | Operational overhead |

## Part 11 — Failure scenarios to drill (essential interview prep)

For each, practice describing the symptom → check order → root cause out loud.

1. **Pod stuck in `CreateContainerConfigError`** → CSI Secret Store can't fetch from KV → check UAMI role assignment on KV → check federated credential subject matches SA → check `kubectl describe pod` for the actual error
2. **Pod `CrashLoopBackOff` with "FATAL: password authentication failed"** → secret didn't mount, or wrong KV reference → check `kubectl get secret pg-secret -o yaml`
3. **App can't reach Postgres** → DNS resolves to public IP? VNet DNS link missing → `nslookup` from pod → fix private DNS zone link
4. **`ImagePullBackOff`** → AKS kubelet identity missing AcrPull → or ACR firewall blocking → `az acr check-health`
5. **Argo CD `OutOfSync` but desired state looks right** → mutating webhook (Azure Policy) is adding labels → exclude those labels from Argo's diff
6. **App Gateway 502** → AGIC health probe failing → check `/healthz` endpoint, check NSG, check pod port
