> Target stack from the JD: **Azure, AKS, Terraform, GitHub Actions, Argo/Helm, Docker, JFrog, Grafana, Networking, InfoSec, Redis**.

## Part 1 — Concept Map: read in this order

### A. Azure foundations (1–2 days)
- **Subscriptions, Management Groups, Resource Groups, Tenants** — how the hierarchy maps to blast radius and IAM
- **Azure Resource Manager (ARM)** — every API call goes through it; understand `providers`, `locations`, `tags`, `locks`
- **Regions & Availability Zones** — paired regions, zone-redundant vs zonal services
- **Azure RBAC vs Azure AD roles** — control plane vs identity plane, the #1 confusion point
- **Managed Identities (System-assigned vs User-assigned)** — replaces secrets in code

📖 Read: Microsoft Learn "Azure Fundamentals" AZ-900 path (skim, don't certify), then jump to **AZ-104** networking + identity modules.

### B. Networking (2–3 days — heaviest gap area)
- **VNets, Subnets, NSGs, ASGs, Route Tables (UDRs)**
- **Service Endpoints vs Private Endpoints vs Private Link** — know when to use which
- **Azure Firewall vs NSG vs Application Gateway WAF** — overlap is real, interview gold
- **ExpressRoute, VPN Gateway, VNet Peering, Hub-Spoke topology**
- **Azure DNS — private zones, link to VNets**
- **Front Door vs App Gateway vs Load Balancer vs Traffic Manager** — the four-way comparison comes up constantly

### C. Compute + AKS (3–4 days — your strongest leverage)
- **AKS architecture** — managed control plane, node pools, system vs user pools, virtual nodes
- **AKS networking modes**: Kubenet vs Azure CNI vs Azure CNI Overlay vs Cilium — pick one and know it cold
- **AKS identity**: AAD Pod Identity (deprecated) → **Workload Identity** (current), how it federates to AAD
- **Ingress**: Application Gateway Ingress Controller (AGIC) vs NGINX vs Istio
- **Cluster autoscaler + KEDA + HPA + VPA** — the four scaling axes
- **Node image upgrades, surge upgrades, maintenance windows**
- **Azure Container Registry (ACR)** — geo-replication, content trust, ACR Tasks

### D. Storage + Data (1–2 days)
- **Storage Account types** (Blob, Files, Queue, Table) + access tiers + redundancy (LRS/ZRS/GRS/RA-GRS)
- **Azure Disks** — Premium SSD v2, Ultra Disk, when each matters for AKS PVCs
- **Azure Key Vault** — secrets/keys/certs, soft-delete, purge protection, **CSI Secret Store Driver** for AKS
- **Azure Cache for Redis** — Basic/Standard/Premium/Enterprise tiers, geo-replication, persistence

### E. Observability (1 day)
- **Azure Monitor → Log Analytics Workspace → Container Insights**
- **Managed Prometheus + Managed Grafana** — Azure's newer offering, the JD calls out Grafana explicitly
- **Diagnostic Settings** — every resource emits to LA/Storage/Event Hub; pattern repeats everywhere
- **Application Insights** — distributed tracing, dependency map

### F. Terraform on Azure (2 days — JD mandatory)
- **AzureRM provider basics, `azurerm_*` resources**
- **Remote state on Azure Storage + state locking via blob lease**
- **Service Principal vs Managed Identity vs OIDC federation from GitHub Actions** — modern pattern is OIDC, no secrets
- **Modules, workspaces, `for_each` vs `count`, `dynamic` blocks**
- **`terraform import`, `moved` blocks, `removed` blocks** — interview asks how you handle drift
- **Sentinel / OPA / `tfsec` / `checkov` / `terraform-compliance`** — policy as code
- The **AzAPI provider** — fallback for resources AzureRM doesn't support yet

### G. GitHub Actions for IaC + AKS (1–2 days)
- **OIDC federation to Azure** (`azure/login@v2` with `client-id`, `tenant-id`, `subscription-id`)
- **Reusable workflows, composite actions, environments + required reviewers**
- **`terraform plan` on PR, `apply` on merge** — the standard gate pattern
- **`actions/cache`, matrix builds, concurrency groups**
- **GitHub Environments** for prod approval gates
- **Self-hosted runners on AKS** (Actions Runner Controller / ARC) — likely how LexisNexis runs it

### H. Security (1–2 days — JD emphasizes this)
- **Microsoft Defender for Cloud** (formerly Security Center) — CSPM + CWP
- **Azure Policy** — built-in initiatives like CIS, NIST 800-53
- **Microsoft Entra ID** (renamed from Azure AD) — Conditional Access, PIM, Identity Protection
- **Key Vault + Managed Identity pattern** — eliminate secrets in code
- **AKS-specific**: Pod Security Standards, Azure Policy add-on for AKS, image scanning in ACR, **Microsoft Defender for Containers**

### I. Argo + Helm (1 day — you may already know)
- **Argo CD** app-of-apps pattern, sync waves, hooks, sync windows
- **Helm** — chart structure, `values.yaml` overlays, `helm template` for debugging, library charts
- **Kustomize vs Helm** — when to use which (or both via Helm post-renderer)

## Part 2 — Hands-on Labs ("Live Sessions")

Goal: get your hands dirty so you can talk about it in interview from experience, not theory. Sign up for the free $200 Azure credit if you don't have a sub.

### Lab 1: Bootstrap an Azure landing zone with Terraform (2–3 hrs)
- Create: 1 Resource Group, 1 VNet with 3 subnets (system, app, data), NSGs, Storage Account for TF state
- Use **OIDC from GitHub Actions** to authenticate — no long-lived secrets
- **Deliverable for talking points**: "I bootstrapped state, federated identity, and a hub VNet from scratch with `azurerm` provider using OIDC"

### Lab 2: Deploy AKS with Terraform + Workload Identity (3–4 hrs)
- AKS with Azure CNI Overlay, system + user node pool, cluster autoscaler enabled
- Enable Workload Identity + OIDC issuer
- Deploy a sample pod that pulls a secret from Key Vault via federated identity (no K8s Secret object!)
- **Talking point**: "I wired Workload Identity end-to-end; pods authenticate to Key Vault without any static secret"

### Lab 3: Argo CD on AKS, GitOps deploy (2 hrs)
- Install Argo CD via Helm chart, expose with internal LoadBalancer
- App-of-apps pointing at a GitHub repo with 3 sample microservices
- Add a sync wave so an init job runs first
- **Talking point**: "I run Argo CD with app-of-apps and sync waves — same pattern you'd use here for the migration"

### Lab 4: GitHub Actions pipeline with `plan` gate (2 hrs)
- PR triggers `terraform plan`, posts plan as PR comment
- Merge to main triggers `apply` with required reviewer via GitHub Environment
- Use OIDC, not service principal secrets
- **Talking point**: "My pipeline blocks apply behind a human approval gate, no Azure secrets ever leave GitHub"

### Lab 5: Observability stack (2–3 hrs)
- Enable Container Insights on AKS → Log Analytics
- Stand up **Azure Managed Prometheus + Managed Grafana**
- Import the AKS dashboards, add a custom panel for pod restart rate
- Wire an alert rule that fires on high pod restarts to a webhook
- **Talking point**: "I run Managed Prom + Grafana — same as what's in the JD"

### Lab 6: Redis + private endpoint (1–2 hrs)
- Azure Cache for Redis Premium, deploy with **Private Endpoint** into your VNet
- Connect from an AKS pod, verify DNS resolves to private IP
- Try the same with a public access path and see how Private Link changes it
- **Talking point**: "I know the Redis Premium vs Enterprise tradeoff and how Private Link locks down the data plane"

### Lab 7: Break-glass drill (1 hr)
- Intentionally break something: corrupt a TF state, drift a resource manually in the portal, delete a node pool
- Recover it: `terraform import`, `terraform state rm`, `moved` block
- **Talking point**: "Drift happens; here's how I recover state without `apply`-ing destructive changes"

## Part 5 — Platform-Team Dynamics (read this carefully)

**Intel**: Core SRE is in talks with platform teams to help product teams migrate to and operate on Azure. That makes this a **platform-engineering-flavored SRE Lead** role, not a pure ops role. Your Core SRE team is effectively a paved-road provider — they don't just run infra, they enable other teams to consume it safely. This is **directly transferable** from on-prem K8s platform maintenance.

### How to reframe your background
Don't say "I did platform maintenance on-prem." Say:
> "I ran the platform layer that hundreds of developers built on — I owned the K8s control plane, the ingress story, the secrets pattern, the upgrade path, and the support burden when teams hit edges. The skill set I'm bringing is platform-as-product thinking, not just operations: build paved roads, get out of teams' way, but enforce the guardrails that keep the org safe."

That sentence puts you ahead of pure ops-background candidates.

### Platform-engineering concepts to bring up in interview
- **Paved road / golden path**: a sanctioned, supported, opinionated way to do the common thing. Off-road allowed, but unsupported.
- **Internal Developer Platform (IDP)**: Backstage, or Backstage-equivalent — service catalog, scaffolders, tech docs. Even a "well-organized GitHub org with templates" counts.
- **Self-service**: teams shouldn't file tickets to get a namespace, a database, a secret. They should run a workflow / fill out a form / merge a PR.
- **T-shirt sized offerings**: "small / medium / large AKS cluster" — limit choice surface to keep platform manageable.
- **Module registry**: central Terraform modules with versioning, semantic releases, deprecation policy.
- **Cluster API / Crossplane / IaC abstractions**: layers above raw Terraform that platform teams provide.
- **Support model**: tier 1 (docs + chat), tier 2 (office hours), tier 3 (escalation to platform on-call). Without this, the team becomes a ticket queue.
- **Cost showback / chargeback**: tag enforcement, per-team cost dashboards. Platform teams that ignore cost get defunded.

### Migration-specific framings
LexisNexis wants migration help. Frame migrations as **factory work**, not one-offs:
- **Assessment**: app inventory, dependency graph, data gravity, compliance class, blast radius
- **Wave planning**: simple/stateless first, stateful + regulated last, dependent apps together
- **Patterns over projects**: build 4–5 migration patterns (lift-shift to VM, containerize + AKS, replatform stateful DB, retire SaaS-replace, refactor) — then assign each app to a pattern
- **Migration factory**: dedicated squad of 2–3 SREs + 1 cloud architect, repeatable runbooks, post-migration handoff to product team
- **Day-2 ownership**: who runs it after migration? Platform team for the substrate, product team for the app. Define the line in a RACI.

### Questions Sebastian probably has on his mind (and how to volunteer answers)
1. *"Can this person mentor the team without me babysitting?"* → cite specific examples of mentoring patterns: pairing rotations, design review office hours, "lunch and learn" tech talks.
2. *"Can they actually do the Terraform / Azure work or are they all-leadership?"* → bring concrete IaC stories: "I owned the K8s module set for X — here's how we versioned and rolled out breaking changes."
3. *"Will they own up when things break?"* → have one postmortem story ready where YOU were a contributor to the incident.
4. *"Will they push back on me when I'm wrong?"* → answer the "disagree with manager" question with a real example.
5. *"Can they handle US + India coordination?"* → talk about written async culture, doc-first decisions, overlap-hour standups, rotation of meeting times to share the burden.

### Red flag answers to avoid
- "I'd start by understanding the team and then making a plan." → Too vague, no signal.
- "I'd put everything in Terraform." → They already know that; *how* is the question.
- "I'd hire more people." → Lead-as-bottleneck framing. Bad.
- "Whatever you need, I'm flexible." → Sounds weak; they want opinions.

## Part 6 — Lead-Level Question Bank (what Sebastian will actually ask)

Domain 10 in Part 3 had surface-level lead questions. **This section is what Sebastian — a hiring manager who just inherited a team and is hiring his right hand — will actually grill on.** Each item has three parts:

- **Q** — the question, as he'd phrase it
- **🎯 What he's really probing** — the subtext (lead candidates fail when they miss this)
- **⚠️ The trap** — the bad answer most candidates give
- **✅ Model answer skeleton** — what to actually say

Practice these out loud. The wording matters as much as the content.

---

### People & Team Leadership

**Q1.** "Tell me about your management style."
- 🎯 He wants a real style, not a textbook one. He's filtering for self-awareness.
- ⚠️ Trap: "I adapt to each person" — true but empty, gives no signal.
- ✅ "Default is high-trust, low-interrupt. I set context and outcomes, then get out of the way. Where I lean in is design reviews and postmortems — those are leverage points. With newer engineers I pair more; with senior ones I mostly defend their time. I do weekly 1:1s, and the agenda is theirs first, mine second."

**Q2.** "Walk me through your first 30, 60, and 90 days if you got this role."
- 🎯 This is the single most predictable Lead question. If you fumble it, you're out.
- ⚠️ Trap: "I'd listen first" — true but generic. He wants a plan.
- ✅ "**Days 1–30**: 1:1s with every report and every adjacent team lead, read the postmortems and last 6 months of incidents, map the current IaC coverage, identify the top 3 sources of toil. No big changes. **Days 30–60**: pick one paved-road win — probably a reusable Terraform module or a self-service pattern — and ship it with the team so I'm earning credibility through delivery, not org charts. Start writing the team charter and on-call rotation principles. **Days 60–90**: present a 6-month roadmap to you and Joshua with the migration sequencing, platform-investment areas, and where I need budget or headcount. By day 90 the team should feel less reactive."

**Q3.** "One of your engineers is underperforming. How do you handle it?"
- 🎯 Will you avoid the hard conversation? Will you escalate too fast?
- ⚠️ Trap: jumping straight to PIP. Or being too soft ("more mentorship").
- ✅ "First, define what underperforming means concretely — outputs missed, behaviors, peer feedback. Then a private, direct 1:1: 'here's what I'm seeing, here's the gap, here's what good looks like in 30 days.' Document it. Pair them with a strong peer for one project. Re-check at 30 days. If no movement, formal performance plan with HR — but no surprises by then. The thing I won't do is carry someone for six months and resent them; that's worse for everyone."

**Q4.** "How do you give critical feedback?"
- 🎯 Specifics, immediacy, and whether you actually do it.
- ⚠️ Trap: "I use the SBI framework" (Situation-Behavior-Impact) — fine, but recite-y.
- ✅ "In the moment when possible, in private always. I name the specific behavior, the impact, and ask what's driving it before I prescribe. I don't sandwich it — that's how senior engineers stop listening. Then I follow up the next week to close the loop, otherwise they don't know if it landed."

**Q5.** "How do you handle a rockstar engineer who's toxic to the team?"
- 🎯 Whether you'll protect culture over output. Big signal for hiring managers.
- ⚠️ Trap: "I'd coach them" — too soft. "I'd fire them" — too hot without trying.
- ✅ "Talent doesn't earn an exemption from being a decent teammate. Direct conversation: 'You're excellent technically, and the team is going around you because of the way you engage. That's costing us more than your output adds.' Give them a real chance to change with specific behaviors. If no change in 60 days, they're out — even if the project hurts. I've seen one toxic senior cost two good juniors. Math doesn't work."

**Q6.** "How do you hire?"
- 🎯 Whether you have a real bar and process.
- ✅ "I screen for three things: technical depth that's not surface, evidence of operating under pressure (incidents, on-call), and how they talk about teammates. I do a system design + a debugging exercise — debugging shows you more than design. References are non-negotiable for senior hires; I call back-channels too. Diversity-of-experience matters more to me than diversity-of-pedigree."

**Q7.** "How do you grow your team?"
- 🎯 Are you a coach or a delegator?
- ✅ "Growth is project-shaped, not training-shaped. I give engineers projects one notch above where they're comfortable, with a safety net of design review and pairing. I write growth plans in 1:1s — one technical area, one collab area, one leadership area per quarter. Internal tech talks are a forcing function. Conferences and certs are nice but secondary."

### Strategy, Roadmap & Prioritization

**Q8.** "How would you prioritize between firefighting and project work when both are urgent?"
- 🎯 Are you mature about toil math? Do you respect operational pressure?
- ⚠️ Trap: "We'd reserve 50% for project work" — sounds rehearsed.
- ✅ "Operational health is a precondition, not a competing priority. If the team is burning, project work won't happen anyway. I'd track toil quantitatively for two sprints, then propose a target — say, no more than 50% of team time on toil, escalating if we breach. The interesting question isn't the ratio — it's whether the toil is structural or a backlog. Structural toil needs a project; backlog toil just needs runway."

**Q9.** "We have a 12-month migration to Azure. How do you sequence it?"
- 🎯 The exact problem he's hiring you for.
- ⚠️ Trap: "I'd do simple apps first." Yes, but go deeper.
- ✅ "Three lanes in parallel. **Lane 1 — platform**: landing zone, AKS reference architecture, Terraform module library, GitHub Actions templates. Done by month 3 or nothing else lands safely. **Lane 2 — pilot migrations**: 2-3 low-risk stateless apps in months 2-4, used to harden the patterns and the runbooks. **Lane 3 — factory at scale**: months 5-12, batching apps by pattern. Stateful + regulated apps go last. Each app gets a migration playbook and an explicit day-2 owner *before* we start. The blocker isn't usually compute — it's identity, data, and DNS. I'd over-invest there early."

**Q10.** "You have headcount for two more SREs. What roles do you hire?"
- 🎯 Whether you think about team composition strategically.
- ✅ "Depends on what's missing. If the team is heavy on K8s but light on networking/security, I hire one network-leaning SRE who can be the Azure Firewall / Private Link expert. The second I'd hire toolchain-leaning — someone who lives in GitHub Actions, Argo, ACR — because that's the leverage layer. Two clones of the existing skills would be a waste."

**Q11.** "Build vs buy: a vendor is offering an SaaS observability tool for $200K/year that we could replicate with managed Grafana for cheaper. What do you do?"
- 🎯 Pragmatism vs purism.
- ✅ "Cheapest is rarely the right answer. I'd model fully-loaded cost: SRE hours to build and maintain, on-call burden, opportunity cost vs the migration. If our self-built version costs $300K-equivalent in engineering time over the year and delays the migration, the SaaS wins. If we already have the expertise and the operational story, in-house wins. Default lean: buy unless it's core differentiation."

**Q12.** "You disagree with a roadmap decision Joshua (my boss) is pushing. How do you handle it?"
- 🎯 Will you push back productively, or just nod?
- ⚠️ Trap: "I'd respect the chain of command." Wet noodle answer.
- ✅ "I'd bring my disagreement to you first — privately, with my reasoning and data. If you align with me, you carry it up. If you align with Joshua, I want to understand why; sometimes there's context I'm missing. Once a decision is made, even if I disagree, I commit and lead the team to execute. What I won't do is publicly undermine it or quietly slow-walk it — that's how trust dies."

### Cross-functional & Stakeholder

**Q13.** "InfoSec wants to mandate a control that'll slow your team down 30%. Walk me through how you handle it."
- 🎯 Will you fight productively, or capitulate, or stonewall?
- ✅ "First understand what risk it's mitigating and whether they have data on it — sometimes the 'mandate' is one person's preference dressed up. If the risk is real, the conversation isn't 'no,' it's 'yes, and here's the cost — let's find the cheapest version that closes the risk.' Bring engineering data: 'this control adds 4 hours per deploy across 12 teams, that's X dollars/year — is there an automated equivalent?' Most security folks respect that framing. The teams who say 'no' lose the war eventually."

**Q14.** "A product team wants to skip the paved road and run their own cluster. How do you respond?"
- 🎯 Will you be a gatekeeper or a platform leader?
- ✅ "I'd ask why first. Sometimes the paved road has a real gap — that's free product feedback I should act on. If the gap is real, fix the paved road. If they just want autonomy for autonomy's sake, the answer is: you can go off-road, but you own the day-2 yourself — patching, security, incidents, audit. Most teams don't actually want that when they see the bill. The ones who do are usually fine running it themselves."

**Q15.** "How do you partner with the upstream platform team if there is one?"
- 🎯 Direct hit on the platform-engineering angle.
- ✅ "Treat them as a vendor we're a major customer of. Quarterly roadmap alignment so their landing zone evolves to fit our use cases. Joint design reviews on shared abstractions. Clear escalation path when their platform blocks us — but with patience because their constraints are real. I'd never reinvent what they already provide unless they're genuinely failing us, and even then I'd over-document why I'm forking."

### Operational & On-Call

**Q16.** "What's your philosophy on on-call?"
- 🎯 Whether you've thought about it or just inherited a rotation.
- ✅ "On-call is a team sport, not a punishment. Three principles: (1) page only on customer-impacting symptoms, not predictions — false pages destroy trust; (2) every page generates either a fix or an actionable runbook within 48 hours, otherwise the same page will fire next week; (3) on-call workload is a leading indicator — if it's growing, the platform is degrading. I'd track page volume per shift as an SLI for the platform itself."

**Q17.** "Describe your ideal postmortem culture."
- 🎯 Buzzword check: do you know what blameless actually means?
- ✅ "Blameless doesn't mean accountability-less. We name what happened, including human actions, but we ask what made the action reasonable in the moment — usually a missing guardrail, missing context, or bad UX. Action items get owners and dates; postmortems without follow-through are theater. I'd run one big monthly review with the team to look at trends across postmortems — that's where the real systemic issues surface."

**Q18.** "Your team missed an SLO and the product owner is furious. How do you handle the meeting?"
- 🎯 Composure, accountability, and steering the conversation.
- ✅ "Show up with facts, not excuses. Here's what happened, here's the impact, here's the root cause, here's what we're changing. Don't relitigate the SLO unless the meeting calms — that conversation is for later. I'd own the miss publicly; I don't throw engineers under the bus even when the immediate cause was a mistake. Internally, separate conversation about what we learned."

### Decision-making & Judgment

**Q19.** "Give me an example of a technical decision you made that turned out to be wrong."
- 🎯 Self-awareness. Have a real one ready.
- ⚠️ Trap: humblebrag ("I cared too much about quality"). Or no example, which is worse.
- ✅ A real one with: what you decided, why it seemed right, what went wrong, what you'd do differently. The "what you'd do differently" is the answer he's grading.

**Q20.** "When do you escalate to me?"
- 🎯 Whether you'll surprise him or partner with him.
- ✅ "Three triggers: anything that puts a commitment to another team at risk and I can't unblock myself; anything that needs budget, headcount, or org-level alignment; and anything that's politically sensitive — I'd rather you hear it from me at a low temperature than from someone else when it's already hot. I won't escalate for technical decisions I'm equipped to make, that's why you hired me."

**Q21.** "What does technical debt look like to you, and when do you pay it down?"
- 🎯 Whether you treat debt religiously or pragmatically.
- ✅ "Debt is only debt if it's slowing us down or about to. Code we'll never touch isn't debt, it's just old. I pay down debt opportunistically when we're already in the area, and explicitly when it's blocking the roadmap. I don't do 'tech debt sprints' as a ritual — usually they're a sign the team has been deferring conversations they should be having weekly."

### Self-awareness & Fit

**Q22.** "What kind of manager do you work best under?"
- 🎯 He's asking if **he** is your kind of manager. Be honest.
- ✅ "Someone who sets clear outcomes and gets out of the way, but I can bring problems to. I don't need cheerleading and I don't want micromanagement — what I value is a manager who'll go to bat for the team upward and who'll tell me directly when I'm off. I bring problems early; I expect the same."

**Q23.** "What kind of work do you not want to do?"
- 🎯 He's checking if you'll be miserable in this role.
- ⚠️ Trap: "I'll do anything." Sounds desperate.
- ✅ "Pure project management without technical depth — running Gantt charts isn't where I add value. I want to stay close to architecture and code review even as I lead. The other thing I'm cautious about is being a permanent escalation bottleneck — I'll do it short-term but I'll be working to build the team so it doesn't need me for every decision."

**Q24.** "What's a weakness you're actively working on?"
- 🎯 Not a trick question — a sincere one. Have a real answer.
- ✅ Pick a real one (e.g., "I tend to want to dig into the technical detail myself instead of letting the team work it; I've been forcing myself to ask 'what do you think' first"). Avoid fake weaknesses.

**Q25.** "Why this role, why now?"
- 🎯 Are you running toward this, or away from something?
- ✅ Have a forward-looking answer. Frame it as: you've operated platforms on-prem at scale, the next step is owning that platform charter at cloud scale and leading the team that delivers it. LexisNexis Risk Solutions has the migration challenge and team scale where that experience compounds. Stay positive about your current/past employer.

### The closer Sebastian might end on

**Q26.** "What questions do you have for me that you haven't already asked?"
- 🎯 This is your last chance to show seniority.
- ✅ Land one of these (pick ones not yet asked):
  - "What's the one thing the team would tell me about you that you wouldn't tell me yourself?"
  - "What's success for *you* in the next 12 months — what does Joshua want from your org?"
  - "Where do you see the biggest gap between what the company says it values and what it actually rewards?"

That last one is the strongest. It signals you're thinking about the role as a partnership, not a transaction. Use it.

### Two questions you should expect from Joshua (the skip-level) if he's in the loop

**JQ1.** "How do you measure the success of a platform team?"
- ✅ "Three things: developer time-to-first-deploy, the percentage of changes that flow through the paved road, and the cost-per-workload. Bonus: NPS from the teams we serve. If those are trending right, the rest follows."

## Closing your gap honestly

You haven't run Azure in anger — own it, and pivot to your strengths:
> "I've operated K8s platforms on-prem at scale — I know the upgrade pain, the multi-tenant traps, the support model, the paved-road game. What I'm building now is the Azure-specific translation: AKS managed control plane, Workload Identity, Azure CNI, Terraform AzureRM, GitHub Actions OIDC. The platform-engineering muscle transfers directly; the API surface is what I'm internalizing."

That sentence + Labs 1–7 done + Part 5 framings = credible Lead candidate. Don't fake breadth you don't have; the interviewer will smell it. Lean hard on the platform-team angle — that's your unfair advantage given what you just heard about their direction.
