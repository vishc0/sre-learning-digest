# Terraform Modules — Reusable Infrastructure Components | Week 1 Day 3

## Why This Matters

In a 25M msg/day platform, you don't hand-configure each RabbitMQ exchange from scratch — you have a template with parameters. Terraform modules do the same thing for infrastructure: define the pattern once, instantiate it with different configs for dev, staging, and prod. Interviewers at Staff/Principal level probe whether you can distinguish *library thinking* (reusable, versioned, owned) from *copy-paste IaC* (the thing that causes prod/dev drift). Modules are the answer to "how do you enforce consistency across 50 microservice repos without becoming a bottleneck?"

## Analogy

Think of a RabbitMQ exchange configuration template in your notification platform. You have a canonical exchange definition — name, type (topic/direct), durability, dead-letter routing — that you fill in with different parameters for each service team. The template itself lives in a shared config repo. Each team calls it with their own values. If you need to add a new parameter (say, `max_priority`), you update the template once and all callers benefit on their next deployment. A Terraform module is exactly that: a parameterized infrastructure template that lives in one place and gets called (instantiated) wherever needed, with different variable values each time.

## Vocabulary Translation Table

| Their Term | Your Existing Mental Model |
|---|---|
| Module | RabbitMQ exchange config template — define once, invoke with params |
| Root module | Your top-level `main.tf` — the "runbook" that orchestrates everything |
| Child module | A reusable sub-template called by the root — like a shared TIBCO adapter |
| `source` argument | The pointer to where the template lives — like a Maven dependency coordinate |
| `variables.tf` | The parameter list the template exposes — what callers can customize |
| `outputs.tf` | The return values the template surfaces — like an API response payload |
| Module registry | Terraform's npm/Maven Central — versioned, public or private module repo |
| Module versioning | `?ref=v1.2.0` on a Git URL — same as pinning a library version in pom.xml |
| Input variable | A named parameter — like a Spring Boot `@Value` injection point |
| Output value | A named result — like a REST endpoint returning a resource ID after creation |

## 30-Second Version

A Terraform module is a folder containing `.tf` files that define infrastructure with input variables and output values. Your root `main.tf` calls child modules by pointing at them with a `source` argument and passing in variable values — like calling a function with arguments. This makes you write the S3 bucket pattern once, then stamp out a dev bucket and a prod bucket with two `module` blocks instead of duplicating 40 lines of HCL.

## 2-Minute Version

Every directory containing `.tf` files is technically a module. The **root module** is where you run `terraform apply` — it's your entry point. **Child modules** are called from the root using the `module` block:

```hcl
module "dev_bucket" {
  source      = "./modules/s3-bucket"   # path to the module folder
  bucket_name = "my-app-dev-2026"
  environment = "dev"
  enable_versioning = false
}
```

Inside `./modules/s3-bucket/`, three files do the work:

- `variables.tf` — declares what parameters the module accepts (name, type, default, description)
- `main.tf` — the actual resource definitions, referencing `var.bucket_name` etc.
- `outputs.tf` — declares what values the module returns (bucket ARN, bucket ID)

When the root calls the module twice with different variable sets, Terraform creates two separate resource trees under the hood — each with its own state entries. Outputs from a child module flow back to the root as `module.dev_bucket.bucket_arn`.

The Terraform Registry (registry.terraform.io) hosts community and verified modules — you can source them directly with a version pin:

```hcl
source  = "terraform-aws-modules/s3-bucket/aws"
version = "3.15.1"
```

This is equivalent to a versioned Maven dependency — reproducible, auditable, and safe to upgrade on your schedule.

## 5-Minute Version

**The full mechanism:**

Terraform's `init` command resolves module sources — downloading remote modules into `.terraform/modules/` and symlinking local ones. This is why you must re-run `terraform init` after adding a new module source. The module's internal resources get namespaced in state as `module.<name>.<resource_type>.<resource_name>`, which is how Terraform tracks two S3 buckets from the same module without colliding.

**Tradeoff 1 — Abstraction vs. debuggability:** A well-designed module hides complexity. A poorly-designed module hides *too much* — when a plan shows `module.prod_bucket.aws_s3_bucket.this` with a cryptic change, engineers can't tell what's happening without reading the module source. Rule of thumb: expose variables for anything that will differ between environments; hardcode only your organizational standards (e.g., always enable server-side encryption).

**Tradeoff 2 — Module granularity:** One module per AWS service (s3-bucket, rds-instance, iam-role) is too fine-grained — callers must wire them together manually and composition becomes boilerplate. One module per application tier (webapp-stack = S3 + CloudFront + Route53) is too coarse — you lose flexibility. Target: one module per *infrastructure concern* that a team owns independently.

**Tradeoff 3 — Versioning discipline:** Unversioned modules (sourced from a local path or un-pinned Git branch) cause the "works on my machine" problem at scale. If the shared module changes and a team runs `terraform apply` without re-pinning, they silently pick up the change. In a multi-team environment, version your modules with Git tags (`v1.0.0`, `v1.1.0`) and require teams to explicitly bump versions — same discipline as library upgrades.

**When NOT to use modules (the premature abstraction anti-pattern):** If you're writing infrastructure for exactly one environment that will never be replicated, a module adds complexity with no payoff. The signal to create a module is: "I'm about to copy-paste this block with minor edits." That's the moment to abstract — not before. Many Terraform codebases have deeply nested module hierarchies where a flat structure would be simpler and equally correct.

**Failure modes:**

- **Circular dependencies**: Module A outputs a value that Module B needs; Module B outputs a value that Module A needs. Terraform cannot resolve this — restructure so the shared value lives in a third module or the root.
- **State coupling**: If two teams call the same module in the same Terraform workspace, a `terraform destroy` by one team removes shared infrastructure. Module calls should be scoped to a single ownership boundary.
- **Output explosion**: Returning 20 outputs from a module to satisfy every possible caller need leads to a leaky abstraction. Surface only what callers legitimately need; let callers use data sources for the rest.

## Director/VP Lens

**As an IC:** You write the module, declare sensible defaults, document variable descriptions, add validation blocks (`validation { condition = ... }`) so callers get clear error messages, and publish it to your internal module registry with a changelog.

**As a Director/VP:** You establish the module governance model — which team owns shared modules (usually Platform Engineering), what the review/release process looks like (PR + semantic versioning + CODEOWNERS), and how you prevent module sprawl (too many overlapping modules with no clear ownership). You also set the policy: "all production resources must use approved modules from the internal registry — no inline resource blocks for security-sensitive resources like S3, IAM, KMS." This is how you scale IaC discipline across 20 squads without a central review bottleneck. You also instrument module adoption: which teams are on old module versions, what's the blast radius of a module upgrade, and is the Platform team a help desk or a product team?

**AI lens:** LLMs accelerate module authoring (generating variable declarations, validation blocks, outputs) but introduce hallucination risk in provider-specific argument names. The governance model matters more as AI-generated IaC proliferates — human review gates at the module registry are the control plane.

**A1.** Add a second `module` block for prod with `enable_versioning = true` (and a different `bucket_name`). The module itself does not change — you're passing a different value to the same input variable.   bucket_name       = "my-app-prod-2026"
```hcl
  environment       = "prod"
  enable_versioning = true
}
```

**A2.** Run `terraform init`. This command resolves and downloads module sources into `.terraform/modules/`. You must re-run it any time you add a new `source` reference — Terraform does not auto-fetch on `plan` or `apply`.

**A3.** The failure mode is **over-abstraction / composition explosion**: callers must wire together dozens of single-resource modules manually, producing more boilerplate than if they'd written the resources inline. The better heuristic is: create a module when you're about to copy-paste a block of related resources with minor parameter differences. The unit of abstraction should be an *infrastructure concern* (e.g., "an S3 bucket with our org's standard security posture"), not a single resource type.

### Step 2: Write the child module — variables.tf

```bash
cat > ~/terraform-training/week1-day3/modules/s3-bucket/variables.tf << 'EOF'
# variables.tf — declares what parameters callers must (or can) pass in
# Think of this as the method signature: what arguments does this template accept?

variable "bucket_name" {
  description = "Globally unique name for the S3 bucket. Must be lowercase, no underscores."
  type        = string

  # Validation block: catch bad input early with a clear error message
  # instead of a cryptic AWS API error
  validation {
    condition     = can(regex("^[a-z0-9][a-z0-9-]{1,61}[a-z0-9]$", var.bucket_name))
    error_message = "Bucket name must be 3-63 chars, lowercase letters/numbers/hyphens, no underscores."
  }
}

variable "environment" {
  description = "Deployment environment tag — used for cost allocation and resource identification."
  type        = string

  # Constrain to known-good values — prevents typos like 'Prod' or 'PROD' from
  # creating untagged resources that escape cost governance
  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "environment must be one of: dev, staging, prod."
  }
}

variable "enable_versioning" {
  description = "Whether to enable S3 object versioning. Recommended true for prod."
  type        = bool
  default     = false   # Safe default: off. Callers opt-in for prod.
}

variable "tags" {
  description = "Additional tags to merge onto the bucket. Base tags are always applied."
  type        = map(string)
  default     = {}      # Empty map: optional, callers add extra tags if needed
}
EOF
```

**EXPECTED_OUTPUT:**
```
# No output. Verify:
cat ~/terraform-training/week1-day3/modules/s3-bucket/variables.tf
# Should print the file contents exactly as written above.
```

**WHY:** Declaring variables with descriptions and validation blocks is the difference between a module that's safe to use at scale and one that produces mysterious errors. The validation block on `environment` is a real governance control — it prevents uncategorized resources from appearing in your cost explorer.

### Step 3: Write the child module — main.tf

```bash
cat > ~/terraform-training/week1-day3/modules/s3-bucket/main.tf << 'EOF'
# main.tf — the actual resource definitions for this module
# This file only runs in the context of whoever calls this module.
# It never runs standalone.

# Merge the caller's extra tags with our mandatory baseline tags.
# locals{} are computed values within this module — not exposed to callers.
locals {
  # Every bucket gets these tags regardless of what the caller passes.
  # This enforces org-wide tagging standards without relying on caller discipline.
  base_tags = {
    Environment = var.environment
    ManagedBy   = "terraform"
    Module      = "s3-bucket"
  }

  # merge(): right-side keys win on collision.
  # Callers can override base tags if they have a strong reason — but they have to be explicit.
  all_tags = merge(local.base_tags, var.tags)
}

# The S3 bucket resource itself.
# aws_s3_bucket creates the bucket; a separate resource controls versioning (AWS API separation).
resource "aws_s3_bucket" "this" {
  # "this" is a Terraform convention for the primary resource in a single-purpose module.
  # It makes the resource address predictable: module.<name>.aws_s3_bucket.this
  bucket = var.bucket_name
  tags   = local.all_tags
}

# Block all public access — hardcoded, not a variable.
# This is an org standard: no S3 bucket should ever be public.
# If a team needs public access, they talk to the security team — not just flip a variable.
resource "aws_s3_bucket_public_access_block" "this" {
  bucket = aws_s3_bucket.this.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Versioning is a separate resource in the AWS provider (v4+).
# The status is conditional on the variable — "Enabled" or "Suspended".
resource "aws_s3_bucket_versioning" "this" {
  bucket = aws_s3_bucket.this.id

  versioning_configuration {
    # Ternary: if enable_versioning is true → "Enabled", else → "Suspended"
    status = var.enable_versioning ? "Enabled" : "Suspended"
  }
}
EOF
```

**EXPECTED_OUTPUT:**
```
# No output. Verify:
cat ~/terraform-training/week1-day3/modules/s3-bucket/main.tf
# Should print the file exactly as written above.
```

**WHY:** Notice `aws_s3_bucket_public_access_block` is hardcoded — this is intentional. Some security controls should not be variables; making them variables means a caller can accidentally (or intentionally) set them to false. A module enforces your security posture by design, not by convention.

### Step 4: Write the child module — outputs.tf

```bash
cat > ~/terraform-training/week1-day3/modules/s3-bucket/outputs.tf << 'EOF'
# outputs.tf — what values this module surfaces back to its caller
# Think of this as the API response: after creation, what does the caller need to know?

output "bucket_id" {
  description = "The name of the S3 bucket (same as the bucket_name variable)."
  value       = aws_s3_bucket.this.id
}

output "bucket_arn" {
  description = "The ARN of the S3 bucket. Use this when granting IAM permissions."
  value       = aws_s3_bucket.this.arn
}

output "bucket_domain_name" {
  description = "The bucket domain name for constructing S3 URLs."
  value       = aws_s3_bucket.this.bucket_domain_name
}

output "versioning_status" {
  description = "The current versioning status of the bucket: Enabled or Suspended."
  value       = aws_s3_bucket_versioning.this.versioning_configuration[0].status
}
EOF
```

**EXPECTED_OUTPUT:**
```
# No output. Verify:
cat ~/terraform-training/week1-day3/modules/s3-bucket/outputs.tf
# Should print the file exactly.
```

**WHY:** The `bucket_arn` output is the most commonly needed — you'll use it in IAM policy documents to grant specific Lambda functions or ECS tasks access to exactly this bucket. Without the output, callers would have to reconstruct the ARN string manually, which is error-prone.

```bash

cat > ~/terraform-training/week1-day3/main.tf << EOF
# Root module — this is where you run terraform init/plan/apply
# It calls the s3-bucket child module twice with different configs.

terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"   # ~> means: >= 5.0.0 and < 6.0.0 (minor updates allowed, major blocked)
    }
  }

  # Remote state backend — reuse the S3+DynamoDB you created on Day 2.
  # If you haven't done Day 2 yet, comment out this entire backend block
  # and Terraform will use local state (a terraform.tfstate file).
  # backend "s3" {
  #   bucket         = "tf-state-training-${ACCOUNT_ID}"
  #   key            = "week1/day3/terraform.tfstate"
  #   region         = "us-east-1"
  #   dynamodb_table = "tf-state-lock"
  #   encrypt        = true
  # }
}

provider "aws" {
  region = "us-east-1"

  # Enforce that all resources get a Project tag — belt-and-suspenders on top of module tags.
  default_tags {
    tags = {
      Project = "sre-training"
      Owner   = "vishweshwar"
    }
  }
}

# --- DEV BUCKET ---
# Low-cost config: no versioning, minimal cost impact.
module "dev_bucket" {
  source = "./modules/s3-bucket"   # relative path to the child module directory

  bucket_name       = "sre-training-dev-${ACCOUNT_ID}"   # globally unique: includes account ID
  environment       = "dev"
  enable_versioning = false

  tags = {
    CostCenter = "training-dev"
  }
}

# --- PROD BUCKET ---
# Production config: versioning enabled for data durability.
module "prod_bucket" {
  source = "./modules/s3-bucket"   # same module, different variable values

  bucket_name       = "sre-training-prod-${ACCOUNT_ID}"
  environment       = "prod"
  enable_versioning = true

  tags = {
    CostCenter    = "training-prod"
    DataRetention = "90-days"
  }
}

# Surface the outputs from both module calls so they appear in terraform output
output "dev_bucket_arn" {
  description = "ARN of the dev bucket"
  value       = module.dev_bucket.bucket_arn
}

output "prod_bucket_arn" {
  description = "ARN of the prod bucket"
  value       = module.prod_bucket.bucket_arn
}

output "dev_versioning_status" {
  description = "Versioning on dev bucket"
  value       = module.dev_bucket.versioning_status
}

output "prod_versioning_status" {
  description = "Versioning on prod bucket"
  value       = module.prod_bucket.versioning_status
}
EOF

echo "Root main.tf written successfully."
```

## Lab ANSWER KEY

**If `terraform init` fails with "Module not installed":**
- Check that `./modules/s3-bucket/` directory exists with all three `.tf` files
- Run `ls ~/terraform-training/week1-day3/modules/s3-bucket/`
- Must show: `main.tf  outputs.tf  variables.tf`

**If `terraform plan` fails with "Invalid value for variable":**
- The validation block on `environment` is rejecting a value
- Check your `main.tf` — environment must be exactly `"dev"`, `"staging"`, or `"prod"` (lowercase)

**If `terraform plan` fails with "BucketAlreadyExists" or "BucketAlreadyOwnedByYou":**
- S3 bucket names are globally unique
- Add a unique suffix to your bucket names: `"sre-training-dev-${ACCOUNT_ID}-v2"`
- If you own the bucket (BucketAlreadyOwnedByYou), run `aws s3 rb s3://BUCKET_NAME` to delete it first

**If `terraform plan` shows only 2 resources instead of 6:**
- You may be missing the `aws_s3_bucket_public_access_block` and `aws_s3_bucket_versioning` resources in `modules/s3-bucket/main.tf`
- Re-run Step 3 to overwrite the file

**If `terraform apply` fails with "AccessDenied":**
- Your IAM training-admin user may not have `s3:CreateBucket` permission
- Verify: `aws sts get-caller-identity` — confirms you're authenticated as training-admin
- Check IAM user has `AdministratorAccess` policy attached

**If `terraform destroy` fails with "BucketNotEmpty":**
- S3 buckets can only be destroyed if empty
- Empty the bucket first: `aws s3 rm s3://BUCKET_NAME --recursive`
- Then re-run `terraform destroy`

**If module outputs show `null` instead of values:**
- The output variable name in `outputs.tf` must exactly match what's referenced in `main.tf`
- Run `terraform refresh` then `terraform output` again

## STAR Anchor

**T-Mobile framing for each question:**

**Q1 (multi-region module design):** "At T-Mobile, our 25M msg/day notification platform had to handle regional failover. When we moved to IaC, I designed our EKS node group module to accept a `regions` list variable, with a `for_each` loop creating identical infrastructure in each region. We hardcoded the security group rules for PCI compliance but exposed instance type, min/max node count, and Spot vs. On-Demand as variables. The lesson: hardcode your compliance boundaries, parameterize your operational knobs."

**Q2 (module governance):** "I implemented a Platform Engineering model where the SRE team owned shared modules in a central repo with protected main branch. Module releases required a PR with semantic version tag. We had a Slack bot that alerted teams when their pinned version was more than one minor version behind the latest. This reduced 'surprise breaks' from 3 per quarter to zero in 18 months."

**Q3 (refactoring strategy):** "I'd start with a strangler fig approach — identify the top 3 most-duplicated resource patterns (usually S3, IAM roles, and security groups), extract those into modules first. Don't touch the rest. Once the team sees the pattern working, they pull in more. The mistake is trying to refactor everything at once — you get a 6-week PR that nobody can review."

**Q4 (output vs. data source):** "Module outputs are for resources you own and created in this Terraform run. Data sources are for resources that exist independently — created by another team, another workspace, or manually. In our platform, the VPC was owned by the networking team's workspace. We used a `aws_vpc` data source (filtered by tag) to get the VPC ID, not an output — because we didn't create the VPC."

**Q5 (driving adoption):** "I'd run a dependency audit to show each team their current pinned version vs. the patched version, automated via a CI check on each team's repo. Set a 2-week SLA with escalation path for critical security fixes. Make it easy: provide a one-line PR the team can merge. Track it like an incident — daily standup until 100% adoption. You treat a supply-chain security patch like a P1."

## Tradeoffs & Failure Modes

**1. Module rename = destroy + recreate (the silent data loss trap)**

Terraform tracks resources by their address in state: `module.dev_bucket.aws_s3_bucket.this`. If you rename the module call from `dev_bucket` to `development_bucket`, Terraform sees the old address as "to destroy" and the new address as "to create." For stateful resources (RDS, ElastiCache, S3 with data), this is a data loss event. **The fix:** use `terraform state mv` to rename the state address before applying, OR use `moved {}` blocks (Terraform 1.1+) which are declarative and reviewable in PRs. This is a Staff-level interview question in disguise — "what happens if you rename a module?"

**2. Module version drift in multi-team environments**

Teams that pin to `~> 1.0` (allow any minor version) get automatic updates on the next `terraform init -upgrade`. This is convenient until a minor version introduces a behavioral change (AWS provider semantics shift frequently). The safe pattern: pin to exact versions (`version = "1.2.3"`) in production workspaces. Use renovate-bot or Dependabot to generate PRs for version bumps — you get controlled updates with a PR audit trail. The tradeoff: slightly more toil vs. no surprises in production.

**3. Over-abstraction kills visibility in the blast radius analysis**

When an incident happens and you need to answer "what Terraform resources are affected by this module?" — deeply nested modules (module calling module calling module) make the blast radius analysis a maze. A `terraform plan` that shows `module.platform.module.networking.module.vpc.aws_subnet.private[0]` tells an on-call engineer almost nothing at 2am. Rule: limit module nesting depth to 2 levels maximum. If you need a third level, you're probably solving an organizational problem with code — the real fix is team topology.
