# Terraform for Teams — Governance, Workspaces, and Policy | Week 1 Day 5

## Why This Matters

At T-Mobile scale — 15 engineers, multiple environments, live 25M msg/day traffic — a single broken Terraform apply can cascade into a P1. Senior SRE/DevSecOps interviewers probe this area hard because it separates people who've used Terraform solo from people who've governed it across teams. Every Staff+ role cares about: how do you prevent blast radius, enforce guardrails, and still ship fast?

## Analogy

Think of Terraform workspaces like **RabbitMQ virtual hosts**. One broker, multiple isolated environments — messages (state) in `dev` vhost cannot see or affect messages (state) in `prod` vhost. Same Terraform code, different state files. If a consumer crashes the `dev` vhost, `prod` is untouched.

Variable precedence is like **incident command priority**: CLI flags override everything (Incident Commander speaks last, overrides all), tfvars files are the runbook (standard ops), environment variables are the on-call policy (always present in the background). Higher authority always wins.

Policy-as-code (Conftest/OPA) is your **change advisory board (CAB) automated**: instead of a committee approving every Terraform plan on Friday, a policy engine checks the plan diff against rules and blocks the merge automatically. Governance scales horizontally.

## 30-Second Version

Terraform workspaces isolate state per environment so a dev experiment can't corrupt prod. Variable precedence gives you a clean override ladder — defaults in code, environment specifics in tfvars files, one-off overrides via CLI. Sensitive variables mask secrets in plan output. `fmt` + `validate` + `tflint` are your shift-left quality gate before a plan even runs. Atlantis closes the loop: plan happens in the PR where peers review it; apply only fires on merge. This stack lets one platform team govern dozens of squads without a bottleneck.

## 2-Minute Version

**Workspaces** — each workspace gets its own state file in the backend. `terraform workspace new dev` creates a new isolated state. Your code can branch on `terraform.workspace` to size resources differently per environment (e.g., `t3.micro` in dev, `m5.large` in prod). The limitation: workspaces share the same backend bucket and the same codebase — for true blast-radius isolation between prod and non-prod, most mature teams use **separate AWS accounts + separate S3 state buckets** per environment. Workspaces are best for lightweight environment isolation within the same account.

**Variable precedence** (lowest to highest):
1. Default values in `variable` blocks (in code)
2. `terraform.tfvars` or `*.auto.tfvars` (auto-loaded)
3. `-var-file=custom.tfvars` (explicit file flag)
4. `TF_VAR_<name>` environment variables
5. `-var="name=value"` CLI flag (highest — beats everything)

**Sensitive variables** — adding `sensitive = true` to a variable declaration tells Terraform to redact the value in all plan and apply output. The value still flows to resources; it just doesn't print. This is defense-in-depth for CI logs — not a replacement for a real secret manager.

**Quality gates** — `fmt` enforces canonical style (deterministic diffs in git). `validate` checks that your HCL is syntactically correct and all required variables have types. `tflint` goes deeper: it catches AWS-specific issues like invalid instance types, missing tags, and deprecated resource arguments. Run all three in CI before `plan`.

**Atlantis** — a self-hosted webhook server. When a PR is opened, Atlantis runs `terraform plan` and posts the diff as a PR comment. Reviewers see exactly what will change before approving. On merge, Atlantis runs `terraform apply`. No human has direct terminal access to run apply — the git workflow IS the deployment workflow. This is the GitOps model applied to infrastructure.

## 5-Minute Version

**Workspaces vs. separate accounts — the real tradeoff**

Workspaces are operationally simple but have a shared blast radius at the IAM and backend level. If your CI pipeline has a bug that corrupts the S3 state backend, all workspaces are affected. The `terraform.workspace` variable in code creates branching logic that is hard to test and easy to get wrong. Teams that outgrow workspaces typically adopt the **"three-repo, three-account" model**: one Terraform root module per environment, separate AWS accounts, separate CI pipelines. Harder to maintain, but you can grant prod-write access to exactly three people and dev-write access to everyone.

**Variable precedence failure modes**

The most common team incident: someone sets `TF_VAR_db_password` in their shell for local testing, forgets it's set, and later runs `terraform apply` in a different context — their local env var silently overrides the tfvars file. The plan looks right (sensitive vars are masked), the apply fires, and the wrong value is used. Fix: in CI/CD pipelines, explicitly unset all `TF_VAR_*` except the ones you intend. Never rely on inherited shell state.

**Sensitive variable limitations**

`sensitive = true` only masks output in the Terraform CLI and state file display. The actual value IS stored in plaintext in the `.tfstate` file if you use local state. This is why sensitive variables must always pair with encrypted remote state (S3 + KMS encryption at rest). Better still: don't put secrets in Terraform variables at all. Use the `aws_secretsmanager_secret_version` data source to pull secrets at apply time, keeping them out of tfvars entirely.

**tflint vs. validate — when each catches what**

`terraform validate` runs without cloud credentials — it only checks HCL schema. It will not catch: wrong AMI IDs, non-existent instance types, or deprecated arguments. `tflint` uses provider-aware plugins (e.g., `tflint-ruleset-aws`) and catches: `aws_instance` using an instance type that doesn't exist in the target region, `aws_s3_bucket` missing required tags per your policy, deprecated `aws_alb` instead of `aws_lb`. Run both — they complement each other.

**Atlantis governance model — the Director play**

The power of Atlantis is not technical — it's organizational. When every Terraform apply goes through a PR, you get: a full audit trail in git, peer review on infrastructure changes, no "cowboy applies" from laptops, and the ability to require CODEOWNERS approval for prod changes. This lets a platform team of 3 govern infrastructure for 30 application teams. The alternative — granting everyone CLI access to the Terraform backend and trusting process — always degrades under incident pressure.

**Conftest/OPA preview**

Before a PR is merged, Conftest takes the `terraform plan -out=plan.tfplan && terraform show -json plan.tfplan` JSON output and runs it through OPA policy files (`.rego`). Example policy: "no S3 bucket may be created with `acl = public-read`." If the plan JSON contains that, the policy fails, the CI check fails, the PR cannot merge. This happens without a human reviewer needing to know the rule — the machine enforces it consistently. Week 2 goes deep on OPA/Conftest.

## Director/VP Lens

**IC (Individual Contributor) view**: "I run `terraform workspace select prod` and then `terraform apply`. I make sure my tfvars file is correct."

**Director/VP view**: "I own the governance model that makes every one of my 10 teams' Terraform workflows safe, auditable, and self-service. My decisions:
- **Backend isolation strategy**: Do we use workspaces (cheap, shared blast radius) or separate accounts (expensive, true isolation)? For our risk profile with PCI/HIPAA data, separate accounts.
- **Who can apply to prod?**: Nobody's laptop. Only the CI pipeline after PR approval from a CODEOWNERS member. Atlantis enforces this mechanically, not culturally.
- **Secrets posture**: No secrets in tfvars, ever. All sensitive config pulled from Secrets Manager at apply time. Audit trail in CloudTrail.
- **Policy gate**: Conftest/OPA policies maintained in a central `platform-policies` repo. Each app team's pipeline pulls from it. One team maintains rules; all teams inherit them automatically.
- **Drift detection**: Scheduled `terraform plan` runs in CI (not just on PR) alert me when AWS console one-offs have drifted from IaC. I review the drift report weekly — same discipline as my SLO error budget review.
- **AI acceleration**: Copilot can write first-draft Terraform modules, but every AI-generated plan still runs through the same Conftest gate. AI speeds authoring; governance remains human-validated."

How a Director avoids becoming a bottleneck: they build the railway, not the trains. Platform team owns the rails (module registry, backend config, policy repo, Atlantis). App teams own the trains (their modules, their PRs). Director reviews policy exceptions, not individual PRs.

## Comprehension Check

**Question 1**: You have a `variable "db_password"` in your Terraform code. You set `TF_VAR_db_password=secret123` in your shell AND pass `-var="db_password=override456"` on the CLI. What value does Terraform use, and why?

**Question 2**: A teammate runs `terraform workspace select prod` on their laptop and then runs `terraform apply`. What is wrong with this workflow from a governance perspective, and what would you change?

**Question 3**: `terraform validate` passes cleanly on your code. Is the code safe to apply to production? What does validate NOT check?

## ANSWER KEY — Comprehension Check

**Answer 1**: Terraform uses `override456` from the `-var` CLI flag. CLI flags have the highest precedence in the variable ladder — they override all other sources including environment variables, auto-loaded tfvars, and explicit tfvar files. The env var `TF_VAR_db_password` is effectively ignored when a `-var` flag for the same variable is present.

**Answer 2**: The problem is that a human has direct terminal access to run `terraform apply` against the prod workspace from their laptop. Governance issues: (a) no peer review of the plan before apply, (b) no audit trail in git, (c) the laptop's environment variables or credentials may differ from the canonical CI environment, (d) if the engineer has the wrong AWS credentials loaded, they may apply to the wrong account. The fix: remove direct apply access. Only the CI/CD pipeline (Atlantis or equivalent) runs apply. Humans open PRs; the pipeline runs plan and posts output for review; apply fires on merge.

**Answer 3**: No — `terraform validate` only checks HCL syntax and schema correctness. It does NOT check: (a) whether referenced AMI IDs exist, (b) whether the instance type is valid in the target region, (c) whether IAM roles or security groups referenced as data sources actually exist in AWS, (d) whether your changes comply with organizational policy (e.g., no public S3 buckets). Validate runs entirely offline without cloud credentials. You still need `terraform plan` (with real credentials) to catch resource-level errors, and tflint + Conftest to catch policy and best-practice violations.

## Lab — Terraform Workspaces, Variables, Sensitive Values, and Linting

**Prerequisites**: WSL2 Ubuntu, Terraform CLI installed, AWS CLI configured as `training-admin`.

**What we're building**: A minimal Terraform config that demonstrates workspace isolation, variable precedence, sensitive variable masking, and code quality tools. We will NOT provision real AWS resources in this lab (no cost) — we use a `null_resource` to make the variable flow visible without spending money. The workspace mechanics and variable behavior are identical whether you're managing S3 buckets or EC2 instances.

---

### Step 1: Create the lab directory and base config

```bash
# Create a clean workspace for today's lab
mkdir -p ~/terraform-labs/week1-day5 && cd ~/terraform-labs/week1-day5

# Create the main Terraform file
cat > main.tf << 'EOF'
terraform {
  required_version = ">= 1.0"
  required_providers {
    null = {
      source  = "hashicorp/null"
      version = "~> 3.0"
    }
  }
}

# This variable has a default — lowest precedence
variable "environment" {
  description = "Deployment environment (dev/staging/prod)"
  type        = string
  default     = "dev"
}

# This variable has NO default — must be supplied
variable "app_instance_type" {
  description = "EC2 instance type for the app tier"
  type        = string
}

# Sensitive variable — will be redacted in plan/apply output
variable "db_password" {
  description = "Database master password"
  type        = string
  sensitive   = true
}

# Outputs — note db_password output is also automatically sensitive
output "current_workspace" {
  value       = terraform.workspace
  description = "Active Terraform workspace"
}

output "environment_var" {
  value       = var.environment
  description = "Value of the environment variable"
}

output "instance_type" {
  value       = var.app_instance_type
  description = "Instance type in use"
}

output "db_password_length" {
  value       = length(var.db_password)
  description = "Password length (not the password itself — proves it was received)"
  sensitive   = false
}

# null_resource lets us see variable values flowing through without creating real infra
resource "null_resource" "demo" {
  triggers = {
    workspace    = terraform.workspace
    environment  = var.environment
    instance_type = var.app_instance_type
  }
}
EOF

echo "main.tf created"
cat main.tf | wc -l
```

**EXPECTED_OUTPUT**:
```
main.tf created
52
```

**WHY**: The `null_resource` is a zero-cost Terraform resource that proves variables flow correctly through a plan and apply without provisioning anything in AWS. `terraform.workspace` is a built-in Terraform variable that returns the active workspace name — we'll use this to verify workspace isolation.

---

### Step 2: Create tfvars files for each environment

```bash
# Default tfvars — will be auto-loaded if named terraform.tfvars
cat > terraform.tfvars << 'EOF'
# This file is auto-loaded by Terraform every run
# Think of it as your standard runbook defaults
environment       = "dev"
app_instance_type = "t3.micro"
db_password       = "default-dev-password-change-me"
EOF

# Prod-specific override file — NOT auto-loaded, must be passed with -var-file
cat > prod.tfvars << 'EOF'
# Production overrides — deliberately separate file
# Reviewed by platform team before use
environment       = "prod"
app_instance_type = "m5.large"
db_password       = "prod-secure-password-vault-managed"
EOF

echo "tfvars files created:"
ls -la *.tfvars
```

**EXPECTED_OUTPUT**:
```
tfvars files created:
-rw-r--r-- 1 youruser youruser 189 Jun 11 10:00 prod.tfvars
-rw-r--r-- 1 youruser youruser 178 Jun 11 10:00 terraform.tfvars
```

**WHY**: Splitting dev defaults into `terraform.tfvars` (auto-loaded) and `prod.tfvars` (explicit flag required) creates a friction point before prod — you have to consciously pass `-var-file=prod.tfvars`. This prevents accidental prod applies with dev settings.


### Step 5: Verify workspace isolation with terraform plan

```bash
# We're currently in prod workspace — plan with terraform.tfvars (dev defaults)
# This shows a MISMATCH: prod workspace, dev tfvars
terraform plan -var-file=terraform.tfvars

echo "---"
echo "Active workspace: $(terraform workspace show)"
```

**EXPECTED_OUTPUT**:
```
Terraform used the selected providers to generate the following execution
plan. Resource actions are indicated by the following symbols:
  + create

Terraform will perform the following actions:

  # null_resource.demo will be created
  + resource "null_resource" "demo" {
      + id       = (known after apply)
      + triggers = {
          + "environment"   = "dev"
          + "instance_type" = "t3.micro"

        }
    }
```

Key observation: `workspace = "prod"` (from `terraform.workspace`) but `environment = "dev"` (from `terraform.tfvars`). **WHY**: This step deliberately demonstrates the mismatch. Seeing it in a plan output is how you build the intuition to catch it in code review.

### Step 6: Override with CLI -var flag — observe precedence

```bash
# Switch to dev workspace
terraform workspace select dev

# Plan using terraform.tfvars BUT override environment with CLI flag
# CLI flag wins — this is the highest precedence
terraform plan \
  -var="environment=FORCE-OVERRIDDEN-BY-CLI" \
  -var-file=terraform.tfvars

# Note: db_password is supplied by terraform.tfvars, not shown in output
```

**EXPECTED_OUTPUT**:
```
  # null_resource.demo will be created
  + resource "null_resource" "demo" {
      + id       = (known after apply)
      + triggers = {
          + "environment"   = "FORCE-OVERRIDDEN-BY-CLI"
          + "instance_type" = "t3.micro"
          + "workspace"     = "dev"
        }
    }
```

`environment` shows `FORCE-OVERRIDDEN-BY-CLI` even though `terraform.tfvars` sets `environment = "dev"`. The `-var` flag won.

**WHY**: This is the variable precedence proof. In CI/CD pipelines, the `-var` or `-var-file` flags passed by Atlantis/GitHub Actions are what actually matter — local tfvars files are secondary. Knowing this hierarchy prevents surprises when CI output doesn't match local plan output.



Notice: `db_password` does NOT appear in the triggers block (it's not in `null_resource.triggers`) and does NOT appear in output. Now add it to triggers to force redaction visibility:

```bash
# Quick test: try to output the sensitive value directly
terraform plan -var-file=prod.tfvars -json 2>/dev/null | python3 -c "
```

**EXPECTED_OUTPUT**:

Or if python3 parsing doesn't match exactly:
```

```

## Lab ANSWER KEY

**Step 3 — init fails with "no required providers"**:
Check that `main.tf` has the `terraform { required_providers { null = ... } }` block. Run `cat main.tf | grep -A5 required_providers`. If missing, the heredoc may have failed — recreate the file.

**Step 4 — workspace new fails with "workspace already exists"**:
Run `terraform workspace list` — if `dev` or `prod` already appears, use `terraform workspace select dev` instead of `new`.

**Step 5 — plan shows no changes instead of 1 resource**:
If you're in `prod` workspace and a prior apply ran, state already has the resource. Run `terraform destroy -var-file=terraform.tfvars` first, then re-plan.

**Step 7 — python3 command fails**:
The JSON plan parsing is optional. The core proof is that `db_password` never appears in the human-readable plan output. If the python3 command errors, skip it — the grep step before already showed no password in output.

**Step 8 — terraform fmt shows no diff**:
Your editor may have auto-formatted on save. Verify by running `cat -A badly_formatted.tf` to check for literal spaces. If the file is already formatted, the test still passed — fmt is idempotent.

**Step 10 — tflint --init fails**:
May be a network issue. Try: `tflint --init --config /dev/null` to skip plugin download and run base rules only. Alternatively, check `~/.tflint.hcl` for proxy settings if behind a corporate firewall.

**Step 11 — workspace delete fails with "workspace is not empty"**:
If you ran `terraform apply` during the lab, the workspace has state. Run `terraform destroy -var-file=terraform.tfvars` (or `prod.tfvars` for prod workspace) first, then delete.

## STAR Anchor

**Q1 — State management for 30 engineers**:

*Situation*: At T-Mobile, our notification platform SRE team had Terraform configs spread across 4 repos with no consistent state backend strategy. After a staging apply accidentally modified a shared security group used by prod services, we audited the blast radius.

*Task*: Design a state isolation model that let squads move fast without stepping on each other.

*Action*: I implemented a three-tier model: (1) shared networking/IAM state in a dedicated `platform-infra` repo owned by my team with strict CODEOWNERS, (2) per-squad application state in separate S3 prefixes with per-squad IAM roles — no squad could assume another's role, (3) environment isolation via separate AWS accounts for non-prod vs. prod. We evaluated workspaces but rejected them for blast-radius reasons — if the S3 backend or KMS key had an issue, all workspaces would be affected simultaneously.

*Result*: Zero cross-squad state corruption incidents in the following 12 months. Onboarding new squads took 4 hours vs. 2 days previously because we had a documented module to provision their backend and IAM role.

**Q4 — Policy enforcement at scale**:

*Situation*: After a third-party security audit flagged two public S3 buckets in our notification platform, I needed a solution that didn't rely on engineers remembering rules.

*Task*: Enforce security policies on Terraform at the PR level, not in post-deploy scanning.

*Action*: Introduced Conftest with OPA policies in a central `platform-policies` repo. Wrote five initial policies covering S3 ACLs, EKS node group encryption, and IAM wildcard denials. Integrated into our GitHub Actions pipeline — every Terraform PR ran `conftest test` against the plan JSON before plan output was even posted for review.

*Result*: Policy violations were caught in under 2 minutes of PR open. No policy-violating infrastructure reached staging in the 6 months after rollout. The security audit follow-up found zero new findings in the Terraform-managed inventory.

## Tradeoffs and Failure Modes

**Tradeoff 1: Workspaces vs. Separate Accounts**

Workspaces are easy to set up and keep a single codebase. But they share backend infrastructure, IAM credentials, and blast radius. A misconfigured CI pipeline that corrupts the S3 backend affects all workspaces. Separate AWS accounts with separate state buckets give true isolation — prod credentials can be restricted to a dedicated role that only the production Atlantis pipeline can assume — but they require more setup and drift management between environments. The answer depends on risk tolerance: financial services, healthcare, and large-scale consumer platforms should use separate accounts. Internal tooling and small teams can use workspaces safely.

**Tradeoff 2: Sensitive Variables vs. External Secret Managers**

`sensitive = true` protects CLI output but does nothing about state file storage. The value is still in plaintext in `.tfstate` unless you use encrypted remote state with KMS. Better: use Vault or AWS Secrets Manager as data sources inside Terraform (`data "aws_secretsmanager_secret_version"`) — Terraform never stores the secret in state at all, it just uses the ARN reference. The tradeoff: you now have a dependency on Secrets Manager availability during every apply. If Secrets Manager is degraded, Terraform apply fails even for changes unrelated to secrets. Mitigation: use `lifecycle { ignore_changes }` on sensitive fields where appropriate, or split secrets retrieval from resource provisioning.

**Tradeoff 3: Atlantis Self-Hosted vs. Terraform Cloud/Enterprise**

Atlantis is free, self-hosted, and integrates with any git provider. The cost is operational: you run the Atlantis server, manage its availability, and handle upgrades. If Atlantis goes down, no one can apply. Terraform Cloud (HCP Terraform) offloads this operational burden, adds a policy engine (Sentinel), and provides a web UI for state inspection — but it costs money per resource under management and introduces a third-party dependency in your deployment pipeline. For a team of 10-30, Atlantis is usually the right choice. Above 50 engineers with strict compliance requirements, HCP Terraform Enterprise becomes operationally cheaper than running your own Atlantis fleet. The failure mode to watch: Atlantis PR locks — if a PR is open and Atlantis is holding a plan lock, other PRs to the same workspace queue up. Under incident conditions, this lock can block emergency fixes. Always implement a `terraform force-unlock` runbook and ensure your Atlantis deployment has a documented lock-release SOP.
