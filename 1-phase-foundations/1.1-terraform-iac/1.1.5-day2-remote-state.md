# Terraform Remote State — S3 Backend + DynamoDB Locking | Week 1 Day 2

## Why This Matters

In any team environment, local Terraform state is a production incident waiting to happen — two engineers running `terraform apply` simultaneously will corrupt state, causing infrastructure drift that is painful and sometimes irreversible to fix. Remote state with locking is the foundational pattern that makes Terraform safe for team use, and every IaC interview at Staff/Principal level will probe whether you understand *why* locking exists, not just how to configure it. Getting this wrong at a $100M ARR company means a multi-hour outage from state corruption.

## Analogy

Think of Terraform state like a **RabbitMQ message queue** that only one consumer should process at a time. If two consumers (engineers) grab the same message (state file) simultaneously and both try to write back their processed version, you end up with split-brain — one engineer's changes overwrite the other's, and your queue (infrastructure) is now in an unknown state.

DynamoDB locking is the **mutex lock** on that queue. Before any consumer touches the message, it writes a lock entry to a coordination store (DynamoDB). If the lock is already held, the second consumer backs off. The lock releases only after the first consumer successfully commits. This is identical to how distributed lock managers work in your RabbitMQ/TIBCO world — advisory locks before touching shared state.

S3 is the **durable message store** — versioned, encrypted, cross-region replicable. It holds the authoritative state file the way your message broker holds the canonical message payload.

## 30-Second Version

Terraform stores a map of every resource it manages in a state file. By default this lives on your laptop — useless for teams. You move it to S3 (shared, versioned, durable) and add DynamoDB as a lock table so only one engineer can modify state at a time. This one change turns a single-player tool into a team-safe platform. Every serious Terraform deployment uses this pattern or a managed equivalent (Terraform Cloud).

## 2-Minute Version

When you run `terraform apply`, Terraform does three things in sequence:

1. **Reads current state** — what does Terraform think currently exists?
2. **Calls AWS APIs** — what actually exists right now?
3. **Computes diff** — plan the changes needed to close the gap, then execute them and write the new state.

If two engineers both do step 1 at the same time, they each get a copy of the state. Engineer A applies changes (creating an S3 bucket), writes back state with that bucket listed. Engineer B applies their changes (deleting a security group), writes back their stale copy of state — which no longer knows about Engineer A's new bucket. Terraform has now lost track of that bucket. It exists in AWS but not in state — this is **orphaned infrastructure**, which means it will never be cleaned up, may have open security groups, and is invisible to future plans.

DynamoDB locking inserts a record in a table (a `LockID` row) before any write operation. The record contains the operator's machine name, timestamp, and operation. If a second `apply` tries to insert the same `LockID`, DynamoDB's conditional write rejects it — the second operator sees a clear error: `Error: Error acquiring the state lock`. They wait. When the first apply finishes, it deletes the lock record. The second apply retries and succeeds.

S3 versioning is a parallel safety net — even if state is accidentally overwritten, you can roll back to the previous version through the AWS console or `aws s3api list-object-versions`.

## 5-Minute Version

**Design decisions you'll be asked about:**

*Why S3 and not Git?* Git state is an anti-pattern that's surprisingly common at small companies. Git has no locking mechanism for binary files. Two engineers with concurrent branches each apply from their branch, push conflicting state, and merge conflicts in JSON state files are nearly impossible to resolve manually. S3 + DynamoDB gives you atomic writes with locking. Terraform Cloud/Enterprise gives you the same plus audit logs and RBAC, which you need at scale.

*Why DynamoDB specifically?* DynamoDB's conditional writes (`ConditionExpression: "attribute_not_exists(LockID)"`) provide atomic compare-and-swap with zero additional infrastructure. You don't need Redis, Zookeeper, or a relational DB for this — DynamoDB's single-key operations are exactly the right tool. It's also serverless, so there's no lock server to maintain.

*What happens when a lock is abandoned?* If an engineer's laptop dies mid-apply, the lock record stays in DynamoDB forever. The team is now blocked. Recovery: `terraform force-unlock <LOCK_ID>` — but this requires human judgment. Before running force-unlock, you must verify the original apply is truly dead (not still running somewhere). Running force-unlock on an active apply causes exactly the corruption you were trying to prevent. This is why runbook discipline matters — force-unlock is a break-glass operation requiring a second pair of eyes.

*State encryption:* S3 backend supports server-side encryption (`encrypt = true` in backend config). This encrypts the state file at rest using your KMS key. Critical for any state that contains secrets (database passwords, API keys). Many engineers don't realize Terraform state contains plaintext sensitive outputs — this is a common security audit finding.

*State file versioning:* Enable S3 versioning on your state bucket so you can roll back bad state. Also configure a lifecycle policy to expire old versions after 90 days — otherwise state history accumulates indefinitely.

**Top failure modes:**
1. **Lock held by dead process** — operator needs runbook for force-unlock with approval gate
2. **State bucket deleted accidentally** — without versioning enabled, this is catastrophic data loss
3. **IAM permissions too broad** — state bucket should have bucket policy restricting access to Terraform roles only; state contains secrets

## Director/VP Lens

**IC implementation view:** Configure the backend block, create the S3 bucket and DynamoDB table via CLI or Terraform bootstrap, run `terraform init`, validate locking works.

**Director/VP governance view:** You don't configure backends — you define the *standard*. Your platform team publishes a blessed Terraform module for backend bootstrapping, so every team gets versioning, encryption, access logging, and lifecycle policies automatically. You set policy: no team's Terraform runs without remote state. You measure compliance via `terraform state pull` checks in CI — if state is local, the pipeline rejects the PR. You own the incident response runbook for force-unlock — it's a two-engineer break-glass operation logged in your change management system. You present state hygiene to your CISO during SOC2 or ISO27001 audits because state files contain secrets and you need to demonstrate encryption-at-rest.

**AI augmentation angle:** LLM-based IaC review tools (Checkov, Trivy with IaC mode, Sentinel policies in TFC) can scan your Terraform configs in CI for backend misconfiguration — no encryption flag, no versioning, wrong region — before any human reviews. As a Director, you wire these into your golden path pipeline so misconfigurations never reach code review.

## ANSWER KEY — Comprehension Check

**A1 — State corruption walkthrough:**
Both engineers read the same state file simultaneously (e.g., state shows 5 resources). Engineer A applies a change adding an S3 bucket (6 resources), writes state back with 6 entries. Engineer B, who read the old 5-resource state, applies their change deleting a security group (4 resources), writes back state with 4 entries — overwriting Engineer A's 6-resource state. Terraform now believes there are 4 resources. The S3 bucket Engineer A created exists in AWS but is gone from state — it is now orphaned infrastructure. Future `terraform plan` will not show it, future `terraform destroy` will not delete it. It becomes invisible to IaC management, potentially a security and cost liability.

**A2 — Force-unlock recovery:**
Step 1: Confirm the lock-holder's apply is truly dead. Call the engineer — verify their terminal session is gone and they're not still applying from another window. Check CloudTrail for in-progress API calls from their IAM principal if uncertain. Step 2: Get the Lock ID from the DynamoDB table directly (`aws dynamodb scan --table-name terraform-locks`) or from the error message the stuck operator sees. Step 3: Have a second engineer (ideally you, as the on-call lead) run `terraform force-unlock <LOCK_ID>` — two-engineer rule prevents one person making a unilateral call. Step 4: Log the unlock action in your change management system (ServiceNow/Jira). Step 5: Have the unblocked operator run `terraform plan` first (read-only, no state write) to verify state integrity before any new apply. Reason for two people: force-unlock on an active apply causes state corruption — the decision requires a second judgment that the apply is genuinely dead.

**A3 — Why not Git for state:**
Git has no file-level mutex for binary content. If two branches each contain a modified `terraform.tfstate`, Git's merge will produce a corrupt JSON file — there is no semantic merge for state. Even if only one engineer applies at a time, state files contain secrets (database passwords, API keys, certificate private keys) in plaintext. Committing state to Git means every engineer with repo access can read your production database password — and it's in git history forever. S3 + DynamoDB gives you locking, encryption-at-rest, versioned rollback, and access control that Git cannot provide.

### Step 2: Create the S3 Bucket for State Storage

Replace `YOUR_ACCOUNT_ID` with the 12-digit account number from Step 1.

```bash
# Set a variable for bucket name — must be globally unique, so we use account ID
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
BUCKET_NAME="terraform-state-training-${ACCOUNT_ID}"

echo "Bucket name will be: ${BUCKET_NAME}"

# Create the bucket
aws s3api create-bucket \
  --bucket "${BUCKET_NAME}" \
  --region us-east-1

# Enable versioning — critical for rollback capability
aws s3api put-bucket-versioning \
  --bucket "${BUCKET_NAME}" \
  --versioning-configuration Status=Enabled

# Enable server-side encryption — state files contain secrets
aws s3api put-bucket-encryption \
  --bucket "${BUCKET_NAME}" \
  --server-side-encryption-configuration '{
    "Rules": [{
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "AES256"
      }
    }]
  }'

# Block all public access — state must never be public
aws s3api put-public-access-block \
  --bucket "${BUCKET_NAME}" \
  --public-access-block-configuration \
    "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

echo "S3 bucket created and hardened: ${BUCKET_NAME}"
```

**EXPECTED_OUTPUT:**
```
Bucket name will be: terraform-state-training-123456789012

# create-bucket returns:
{
    "Location": "/terraform-state-training-123456789012"
}

# put-bucket-versioning, put-bucket-encryption, put-public-access-block return:
(no output — HTTP 200 with empty body is success)

S3 bucket created and hardened: terraform-state-training-123456789012
```

**WHY:** You're not just creating a bucket — you're hardening it. Versioning enables rollback. Encryption protects secrets in state. Public access block prevents accidental exposure. These four commands are what separates "I configured a bucket" from "I configured a production-grade state backend."

### Step 3: Create the DynamoDB Table for State Locking

```bash
# Create the lock table
# PAY_PER_REQUEST billing = no provisioned throughput needed, ~$0 at training volume
aws dynamodb create-table \
  --table-name "terraform-state-locks" \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --region us-east-1

# Wait for table to become ACTIVE before proceeding
aws dynamodb wait table-exists --table-name "terraform-state-locks"

# Verify table status
aws dynamodb describe-table \
  --table-name "terraform-state-locks" \
  --query "Table.TableStatus" \
  --output text
```

**EXPECTED_OUTPUT:**
```
# create-table returns:
{
    "TableDescription": {
        "TableName": "terraform-state-locks",
        "TableStatus": "CREATING",
        "KeySchema": [{"AttributeName": "LockID", "KeyType": "HASH"}],
        ...
    }
}

# wait table-exists: (no output — completes when table is ACTIVE, ~10-15 seconds)

# describe-table query:
ACTIVE
```

**WHY:** The `LockID` attribute is the single key Terraform writes when it holds a lock. `PAY_PER_REQUEST` means you pay per read/write operation rather than reserving capacity — for a lock table that gets a handful of writes per day, this is effectively free and requires zero capacity planning.

### Step 6: Add S3 Backend Configuration and Migrate State

```bash
cd ~/terraform-labs/week1-day2

# Retrieve your account ID
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)

# Rewrite main.tf — add the backend block inside the terraform {} stanza
# IMPORTANT: The backend block must be inside terraform {}, before required_providers
cat > main.tf << EOF
# main.tf — Week 1 Day 2: Remote State Lab (AFTER MIGRATION)

terraform {
  required_version = ">= 1.5.0"

  # REMOTE BACKEND CONFIGURATION
  # This moves state from local disk to S3 with DynamoDB locking
  backend "s3" {
    bucket         = "terraform-state-training-${ACCOUNT_ID}"  # State storage bucket
    key            = "week1/day2/terraform.tfstate"             # Path within bucket
    region         = "us-east-1"                               # Must match bucket region
    encrypt        = true                                      # Encrypt state at rest
    dynamodb_table = "terraform-state-locks"                   # Lock table name
  }

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
}

resource "aws_s3_bucket" "training_test" {
  bucket = "tf-training-test-${ACCOUNT_ID}"

  tags = {
    Name        = "Terraform Training Test Bucket"
    Environment = "training"
    ManagedBy   = "terraform"
    Week        = "1"
    Day         = "2"
  }
}

data "aws_caller_identity" "current" {}

output "test_bucket_name" {
  value       = aws_s3_bucket.training_test.bucket
  description = "The name of the test bucket managed by this Terraform config"
}
EOF

echo "main.tf updated with backend config. Running terraform init to migrate..."

# Run terraform init — this triggers state migration
# Terraform detects the new backend block and asks if you want to copy existing state
terraform init
```

### Step 9: Simulate Lock Contention (Observing Lock Behavior)

This step shows you what the lock table looks like during an apply and what another engineer sees when a lock is held.

```bash
cd ~/terraform-labs/week1-day2

# First, let's look at the DynamoDB table while NO lock is held — should be empty
aws dynamodb scan \
  --table-name terraform-state-locks \
  --query "Items" \
  --output json
```

**EXPECTED_OUTPUT (no lock held):**
```
[]
```

Now simulate a lock being held. We'll write a lock record manually to the DynamoDB table the same way Terraform does, then attempt to run `terraform plan` so you see the error.

```bash
# Write a fake lock record to simulate another engineer holding the lock
# This is exactly the structure Terraform writes when it acquires a lock
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
FAKE_LOCK_ID="abc123-fake-lock-$(date +%s)"

aws dynamodb put-item \
  --table-name terraform-state-locks \
  --item "{
    \"LockID\": {\"S\": \"terraform-state-training-${ACCOUNT_ID}/week1/day2/terraform.tfstate\"},
    \"Info\": {\"S\": \"{\\\"ID\\\":\\\"${FAKE_LOCK_ID}\\\",\\\"Operation\\\":\\\"OperationTypeApply\\\",\\\"Who\\\":\\\"engineer2@laptop\\\",\\\"Version\\\":\\\"1.7.0\\\",\\\"Created\\\":\\\"2026-06-11T10:00:00.000Z\\\"}\"}
  }"

echo "Fake lock written. Scanning table to verify:"
aws dynamodb scan \
```

## Lab ANSWER KEY

**If Step 1 fails (`aws sts get-caller-identity` errors):**
- `Unable to locate credentials` → Run `aws configure` and re-enter access key, secret key, region `us-east-1`, output `json`
- `InvalidClientTokenId` → Your access key is wrong or the IAM user `training-admin` doesn't exist; re-create from AWS console
- Network timeout → Check WSL2 network connectivity: `curl https://aws.amazon.com`

**If Step 2 fails (bucket creation):**
- `BucketAlreadyExists` or `BucketAlreadyOwnedByYou` → Either you ran this before or someone else has this bucket name. Try with a suffix: `terraform-state-training-${ACCOUNT_ID}-v2`
- `IllegalLocationConstraintException` → You're trying to use a `--create-bucket-configuration` flag for us-east-1 that doesn't need it. Remove `--create-bucket-configuration` if you added one — us-east-1 is the default region and doesn't take a LocationConstraint.

**If Step 3 fails (DynamoDB table creation):**
- `ResourceInUseException: Table already exists` → Run `aws dynamodb delete-table --table-name terraform-state-locks` and wait 30 seconds before recreating
- `AccessDeniedException` → Your `training-admin` IAM user is missing `dynamodb:CreateTable` permission. Verify the policy in IAM console.

**If Step 5 fails (`terraform init`):**
- `Error: Failed to get existing workspaces` → AWS credentials issue. Run `aws s3 ls` to verify connectivity.
- `Error: No valid credential sources found` → Terraform inherits AWS credentials from environment. Confirm `AWS_PROFILE` or `~/.aws/credentials` is configured.
- Provider download errors → Check internet connectivity from WSL2. Try `curl https://registry.terraform.io`.

**If Step 6 migration fails:**
- `Error: Failed to persist state to backend` → S3 bucket permissions issue. Verify `training-admin` has `s3:PutObject` on the state bucket.
- `Error initializing the backend: the state migration failed` → Re-run `terraform init` — occasionally a transient S3 error. If it persists, check bucket policy.
- Migration prompt doesn't appear → The backend block may have a syntax error. Run `terraform validate` to check.

**If Step 9 lock simulation doesn't produce the expected error:**
- The DynamoDB `put-item` key path must exactly match the S3 key path Terraform uses. Verify the LockID value matches your actual bucket name and state key path.
- Try running `terraform apply` instead of `terraform plan` — some versions only lock on writes.

**If teardown Step 2 (bucket deletion) fails:**
- `BucketNotEmpty` error despite running the versioned delete script → Run `aws s3 rm s3://terraform-state-training-${ACCOUNT_ID} --recursive` as a backup, then retry the bucket deletion.
- Python3 script fails → Manually list versions in AWS Console → S3 → your bucket → "Show versions" toggle → select all → delete, then delete the bucket.

## STAR Anchor

Use these T-Mobile framings to ground your interview answers:

**Q1 (State management strategy):**
"At T-Mobile, our notification platform team scaled from 2 to 8 engineers contributing to Terraform. We had one state corruption incident in 2023 where two pipeline jobs ran simultaneously during a deployment window — we lost 4 hours recovering state manually. After that, I drove adoption of remote state with DynamoDB locking as a team standard. We codified it as a bootstrapped module all new projects must use before their first PR merges. The forcing function was a pipeline check: if your backend block points anywhere other than our standard state bucket prefix, the PR is rejected automatically."

**Q2 (State recovery):**
"S3 versioning means the state file has a version history. First I'd run `terraform plan` with the previous state version to verify it reflects the correct pre-destroy state — you do this by fetching the version from S3 and saving it locally, then running `terraform plan` against it before pushing it back. Once I confirm the state version is correct, I restore it with `aws s3api copy-object` to promote the old version. Then I run `terraform plan` to see the diff between what's in AWS and what state now says — this shows me what needs to be re-created. Critically, I'd stop the team from running any apply until the state is validated."

**Q3 (Force-unlock decision):**
"Two-person rule: I never force-unlock without a second engineer confirming the original operation is dead. My diagnostic checklist: (1) identify the lock holder from the error message — who and what pipeline job. (2) Check CloudTrail for API activity from that identity in the last hour — if there are recent `CreateBucket`, `PutObject`, or similar calls, the apply may still be running. (3) Check CI/CD system — is the job marked failed, running, or timed out? (4) Only after confirming dead: `terraform force-unlock <LOCK_ID>` and log it in our incident system."

**Q4 (Governance at scale):**
"At T-Mobile with a 15-person SRE team, I define the standard and the platform team enforces it. Golden path: we publish a `terraform-bootstrap` module that creates the S3 bucket, DynamoDB table, and IAM policies in a single call. Every new Terraform project starts by calling that module. We use Terraform workspaces to namespace state by environment (dev/staging/prod). IAM policies scope state access: developers can read state in all environments but can only write to dev. CI/CD service accounts write to staging and prod. The state bucket has a resource policy that denies all access except from our Terraform role ARNs."

**Q5 (Security implications):**
"Terraform state is a secret store whether you treat it that way or not. Concrete example: if you use the `aws_db_instance` resource with a `password` argument, that password is stored in plaintext in state — I've seen this in audits where state was stored in Git and the database credential was in git history for two years. Controls I apply: (1) S3 SSE with CMK (KMS key we control), (2) S3 bucket policy denying access from outside our VPC and our Terraform roles, (3) CloudTrail logging on the state bucket so we have audit records of every state read/write, (4) Terraform code review standard: sensitive outputs are marked `sensitive = true` which redacts them from CLI output but does NOT remove them from state — you still need the encryption. For truly sensitive values, the pattern is: store the secret in Secrets Manager, reference it by ARN in Terraform, never pass the plaintext value through Terraform at all."

## Tradeoffs and Failure Modes

**1. Lock abandonment vs. team velocity**

*The tradeoff:* The longer your lock TTL (or the more complex your apply), the longer the team is blocked if the lock holder's process dies. A 45-minute EKS cluster apply that dies at minute 30 locks out your whole team for the remaining 15 minutes plus however long it takes someone to notice and force-unlock.

*The failure mode:* Teams that experience this start passing `-lock=false` to "get unblocked fast." This is the path to state corruption. The correct fix is: shorter, more granular Terraform configs (one state file per service/module, not a monolith), and a clear force-unlock runbook with a fast escalation path.

*Interview framing:* "This is a design problem, not a locking problem. If one apply is blocking your whole team, your state is monolithic. We solved this at T-Mobile by decomposing our single platform state into 7 service-scoped states. Each service's changes lock only that service's state. Maximum blast radius of a stuck lock went from 8 engineers blocked to 2."

**2. State as a source of truth vs. infrastructure drift**

*The tradeoff:* State is Terraform's memory, not AWS's. If someone manually changes an AWS resource outside of Terraform (console, CLI, another tool), state drifts from reality. The next `terraform plan` will show a diff you didn't expect. Teams that treat state as the source of truth get confused; teams that understand state is a cache of the last known configuration handle drift correctly.

*The failure mode:* Drift-induced plan surprises cause engineers to start adding `lifecycle { ignore_changes = [...] }` blocks to suppress the noise. These ignore blocks accumulate until Terraform is no longer managing those attributes at all — you've partially abandoned IaC without realizing it.

*Interview framing:* "We run `terraform plan` in read-only mode on a schedule — every night, against every environment — and alert on unexpected diffs. This catches drift before it becomes a production incident. It's our IaC equivalent of config drift detection. When drift is found, the runbook is: understand why it happened, make the Terraform config authoritative, apply, document what manual change was made and by whom."

**3. Single backend account vs. multi-account architecture**

*The tradeoff:* Centralizing state in one account (your "tooling" or "management" account) gives you a single place to govern, audit, and secure Terraform state. But it creates a dependency: if the management account is inaccessible (outage, lockout, billing suspension), no environment's Terraform can run.

*The failure mode:* Organisations that don't think about this get caught during billing disputes or account suspensions — they discover that their entire IaC operational capability is blocked because state is unreachable. This is a P1 incident with no Terraform-based resolution path.

*Interview framing:* "Cross-account state access uses IAM role assumption — the Terraform execution role in the dev/staging/prod accounts assumes a read/write role in the management account to access state. This pattern lets you centralize state while keeping execution scoped to each environment's account. The resilience play is: for truly critical environments, use a regional replica of the state bucket so a single region outage doesn't block operations. AWS S3 Cross-Region Replication + a Route53 failover DNS alias for the backend endpoint handles this pattern."
