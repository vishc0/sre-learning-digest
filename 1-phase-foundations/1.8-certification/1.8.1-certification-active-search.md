**Q1. You run `terraform plan` and see a `~` (tilde) symbol next to a resource. What does this indicate?**

A) The resource will be destroyed and recreated
B) The resource will be updated in-place
C) The resource has been manually modified outside Terraform
D) The resource is being moved to a different module

**Answer: B**
The tilde `~` means an in-place update — Terraform can modify the existing resource without destroying it. The `-/+` symbol means destroy-and-recreate. The `!` symbol (in newer versions) indicates a resource in a tainted state. Option C describes drift, which shows as a plan diff, not a symbol type.

---

**Q2. What is the correct order of the Terraform core workflow?**

A) plan → init → apply → validate
B) init → validate → plan → apply
C) init → plan → apply → destroy
D) validate → init → plan → apply

**Answer: B**
`init` downloads providers and sets up the backend. `validate` checks HCL syntax without calling provider APIs. `plan` shows what will change. `apply` makes it so. Destroy is not a required step — it is an optional teardown. The exam treats `validate` as a step between `init` and `plan`.

---

**Q3. You have a Terraform state file stored locally. Two engineers run `terraform apply` simultaneously. What is the most likely outcome?**

A) Terraform detects the conflict and refuses to apply for the second engineer
B) The second apply waits until the first completes
C) Both applies succeed, with one overwriting the other's state changes
D) Terraform merges the two state files automatically

**Answer: C**
Local state has no locking mechanism. Both applies read the same state file, make their changes, and write back — the last write wins, silently discarding the other engineer's changes. This is why remote state + DynamoDB locking is foundational. Options A and B require a lock store (DynamoDB). Option D does not exist — state files cannot be merged.

**Q4. Which of the following is the correct way to reference an output from a child module named `network` in your root module?**

A) `var.network.vpc_id`
B) `network.vpc_id`
C) `module.network.vpc_id`
D) `output.network.vpc_id`

**Answer: C**
The syntax is always `module.<module_name>.<output_name>`. The `module.` prefix is required. `var.` refers to input variables. `output.` is not a valid reference prefix in HCL. Option B would only work if `network` was a data source or resource.

**Q5. What does `terraform refresh` do?**

A) Updates the Terraform CLI to the latest version
B) Downloads updated provider plugins
C) Reconciles state file with real infrastructure without making changes
D) Applies pending changes from the plan file

**Answer: C**
`terraform refresh` queries all real infrastructure and updates the state file to match reality — without changing anything in AWS/GCP/etc. It is useful after manual changes to resync state before planning. Note: in Terraform 0.15+, `refresh` behavior is integrated into `plan` by default via the `-refresh=true` flag. The standalone `refresh` command still exists but is less commonly used.

---

**Q6. You want to create 3 identical EC2 instances using a single `resource` block. Which meta-argument should you use?**

A) `for_each`
B) `count`
C) `loop`
D) `repeat`

**Answer: B**
`count` creates N identical copies of a resource, indexed as `resource[0]`, `resource[1]`, etc. `for_each` is better when each instance has different configuration (takes a map or set). `loop` and `repeat` do not exist in Terraform. Use `count` when instances are identical, `for_each` when they are distinct.

**Q7. Which of the following backend types supports state locking natively?**

A) local
B) S3 (with DynamoDB configured)
C) HTTP
D) Consul

**Answer: B and D** (multi-select)
The S3 backend supports locking via a separately configured DynamoDB table. Consul backend has native locking built in. The local backend has no locking. The HTTP backend supports optional locking via `lock_address`/`unlock_address` configuration but it is not native — the server must implement the locking protocol.

*Note: If this appears as single-select, the exam answer is B (most common interview context).*

**Q8. What file should you NEVER commit to version control in a Terraform project?**

A) `main.tf`
B) `variables.tf`
C) `terraform.tfvars`
D) `outputs.tf`

**Answer: C**
`terraform.tfvars` often contains secrets: AWS credentials, database passwords, API keys. It should be in `.gitignore`. The `.tf` files (main, variables, outputs) are exactly what you DO commit — they define infrastructure as code. Many security audit findings at mid-size companies are from committed `.tfvars` files discovered in git history.

**Q9. You run `terraform destroy`. What happens to the Terraform state file?**

A) The state file is deleted
B) The state file is archived to S3 automatically
C) The state file is emptied — it remains but contains no resources
D) The state file is unchanged — you must delete it manually

**Answer: C**
After `terraform destroy`, all resources are removed and the state file is updated to reflect zero resources — it is empty but the file itself remains. This is correct behavior: the backend configuration is preserved so you can run `terraform apply` again later to recreate resources. The state file is not deleted.

**Q10. Which of the following correctly describes a Terraform workspace?**

A) A separate Terraform Cloud organization
B) An isolated state file for the same configuration — allows dev/staging/prod from one codebase
C) A folder containing Terraform modules
D) A provider configuration block

**Answer: B**
Workspaces give you separate state files from the same Terraform configuration directory. `terraform workspace new staging` creates a new state file. `terraform workspace select production` switches to the production state. The same `.tf` files apply to all workspaces — you parameterize with `terraform.workspace` variable. This is not the same as separate directories (which is the more common pattern at scale, but workspaces are the exam answer).

**Q11. What is the purpose of `terraform init -upgrade`?**

A) Upgrades the Terraform CLI binary
B) Forces re-download of provider plugins to the latest allowed version within version constraints
C) Migrates the state file to a newer format
D) Initializes a new workspace

**Answer: B**
`-upgrade` re-downloads providers and modules, respecting the version constraints in your `required_providers` block, but fetching the latest version within those constraints. Without `-upgrade`, `init` reuses cached providers. Use this when you want to pick up a new minor version of the AWS provider, for example.

**Q12. A resource block has `lifecycle { prevent_destroy = true }`. What happens when you run `terraform destroy`?**

A) The destroy proceeds but the resource is recreated immediately
B) Terraform errors out and refuses to destroy that resource
C) The resource is destroyed after a 60-second warning delay
D) The resource is moved to a quarantine workspace

**Answer: B**
`prevent_destroy = true` causes Terraform to error with "Error: Instance cannot be destroyed" when any operation would destroy that resource, including explicit `terraform destroy`. This is commonly used on production state buckets and databases. The only way to destroy a `prevent_destroy` resource is to first remove that lifecycle block from the code and re-apply.

**Q13. What is the difference between `terraform taint` and `terraform apply -replace`?**

A) `taint` marks a resource for replacement; `-replace` is the modern equivalent
B) `taint` destroys immediately; `-replace` schedules for next apply
C) They are identical — same command, different syntax versions
D) `taint` works for modules; `-replace` works for individual resources

**Answer: A**
`terraform taint <resource>` is the legacy command (deprecated in 0.15.2) that marks a resource as tainted so it is destroyed and recreated on next apply. `terraform apply -replace=<resource>` is the modern equivalent introduced in 0.15.2 that accomplishes the same thing in a single operation. The exam may ask about both — know that `-replace` is current best practice.

**Q14. You need to use data from an existing S3 bucket that Terraform did not create. Which block do you use?**

A) `resource "aws_s3_bucket" "existing" {}`
B) `data "aws_s3_bucket" "existing" {}`
C) `import "aws_s3_bucket" "existing" {}`
D) `ref "aws_s3_bucket" "existing" {}`

**Answer: B**
`data` sources read existing resources without managing them. Terraform will query AWS for the bucket's current state and make attributes available as `data.aws_s3_bucket.existing.arn`. A `resource` block would try to create a new bucket. `import` is a command (not a block in older versions) used to bring existing resources under management. In Terraform 1.5+, `import` blocks exist, but for reading-only, `data` is correct.

**Q15. What does the `depends_on` meta-argument do?**

A) Creates an explicit dependency between resources regardless of implicit reference
B) Sets the execution order for provider initialization
C) Defines which module a resource depends on
D) Specifies which variables must be set before plan

**Answer: A**
Terraform normally infers dependencies from references (`resource.b` uses `resource.a.id` → Terraform knows to create A before B). `depends_on` adds explicit ordering when there is no reference-based dependency — for example, an IAM policy must exist before an EC2 instance that uses it, even if the instance does not directly reference the policy ARN. Use sparingly — it can hide architectural problems.

**Q16. Which command shows you the current state without making any changes?**

A) `terraform show`
B) `terraform state list`
C) `terraform output`
D) All of the above, serving different purposes

**Answer: D**
- `terraform show` — displays the full current state or a saved plan file in human-readable format
- `terraform state list` — lists all resource addresses managed in state (just the names)
- `terraform output` — displays only the output values defined in `outputs.tf`

All three are read-only and make no changes. The exam may ask you to distinguish them.

**Q17. What is a Terraform provider?**

A) A managed service that hosts Terraform state
B) A plugin that translates HCL resource blocks into API calls for a specific platform
C) The company that created Terraform
D) A module registry for sharing configurations

**Answer: B**
Providers are plugins — written in Go — that know how to talk to a specific API (AWS, Azure, GCP, Kubernetes, Datadog, etc.). When you declare `provider "aws" {}`, Terraform downloads the AWS provider plugin during `init`. The provider translates your `resource "aws_s3_bucket"` block into the correct S3 API calls. HashiCorp maintains official providers; anyone can publish community providers to the registry.

**Q18. You have `required_version = ">= 1.0, < 2.0"` in your `terraform {}` block. What does this enforce?**

A) The AWS provider must be version 1.x
B) The Terraform CLI running this configuration must be between 1.0 and 2.0
C) The state file format must be version 1.x
D) Modules sourced from the registry must be version 1.x

**Answer: B**
`required_version` constrains the Terraform CLI binary version — not providers, not modules. If a team member runs Terraform 0.15 against a codebase requiring `>= 1.0`, Terraform errors out immediately. This prevents subtle behavior differences between team members using different CLI versions. Provider versions are constrained separately in `required_providers`.

**Q19. What is the purpose of the `.terraform.lock.hcl` file?**

A) Prevents other engineers from running Terraform concurrently
B) Records the exact provider versions selected during `init` so all team members use the same versions
C) Stores the Terraform state for the local backend
D) Encrypts sensitive variables at rest

**Answer: B**
The lock file records the exact provider versions and their checksums selected during `init`. Committing this file to git ensures every team member and every CI pipeline uses identical provider versions — preventing "works on my machine" provider drift. If you want to upgrade a provider, run `terraform init -upgrade` which updates the lock file. This file should always be committed to version control.

**Q20. Which of the following is NOT a valid Terraform variable type?**

A) `string`
B) `list(string)`
C) `map(number)`
D) `array(string)`

**Answer: D**
`array` is not a Terraform type. The correct collection types are `list` (ordered, allows duplicates), `set` (unordered, no duplicates), `map` (key-value pairs), `object` (structured with named attributes), and `tuple` (fixed-length sequence of mixed types). Common trap: developers from Python/JavaScript expect `array` — the Terraform equivalent is `list`.

## 10 Flashcard Pairs

Study these as flip-card drill: read the term, say the definition out loud, flip to check.

| Term | Definition |
|---|---|
| `terraform init` | Downloads providers, initializes backend, prepares working directory — must run before any other command or after any backend/provider config change |
| `terraform plan -out=tfplan` | Generates an execution plan and saves it to a file — use `terraform apply tfplan` to apply exactly that plan without re-computing |
| Remote backend | Configuration pointing Terraform state at a remote store (S3, Terraform Cloud, Consul) instead of the local filesystem — required for team use |
| DynamoDB lock table | A DynamoDB table with a `LockID` string partition key — Terraform writes a conditional record before any state write; if the record already exists, the apply is blocked |
| `count` vs `for_each` | `count` creates N identical copies indexed by integer; `for_each` creates one copy per map key or set element, indexed by key — prefer `for_each` when instances need different identities |
| `terraform import` | Brings an existing real resource under Terraform management — writes it into state without creating it; you still need to write the matching HCL resource block manually |
| `lifecycle { create_before_destroy = true }` | Forces Terraform to create the replacement resource before destroying the old one — critical for zero-downtime deployments of resources that cannot have a gap (e.g., TLS certificates, load balancer targets) |
| Data source | Read-only reference to an existing resource not managed by Terraform — declared with `data` block, accessed as `data.<type>.<name>.<attribute>` |
| Module source types | Local path (`./modules/x`), Git URL (`git::https://...`), Terraform Registry (`hashicorp/consul/aws`) — each behaves differently for versioning and caching |
| Sensitive variable | `variable "db_password" { sensitive = true }` — masks the value in plan and apply output; the value is still stored plaintext in state unless state encryption is configured |

## Common Wrong Answers and Why They Are Wrong

**"Terraform Cloud is required for team use."**
Wrong. S3 + DynamoDB gives you remote state and locking for free-tier cost. Terraform Cloud adds a GUI, RBAC audit logs, Sentinel policy-as-code, and a managed runner — valuable at scale but not required. The exam does not assume Terraform Cloud.

**"You should use `terraform refresh` before every plan."**
Outdated. In Terraform 1.x, `plan` runs a refresh automatically (`-refresh=true` is the default). Standalone `terraform refresh` is rarely needed and has edge cases where it can overwrite correct state with stale provider data. Prefer `terraform plan -refresh-only` for explicit refresh behavior.

**"Modules must come from the Terraform Registry."**
Wrong. Modules can be sourced from local paths, Git repositories, GitHub, Bitbucket, S3, and GCS — not just the public registry. Private module registries (in Terraform Cloud/Enterprise) are also valid. Exam questions about module sources will probe all of these.

**"`terraform apply` always prompts for confirmation."**
Wrong. `terraform apply -auto-approve` skips the prompt. This is valid in CI pipelines but dangerous in manual sessions. Exam distinguishes interactive vs automated workflows.

**"The state file is just a backup — you can delete it and re-run apply."**
Catastrophically wrong. The state file is Terraform's only record of what it manages. Deleting state means Terraform thinks nothing exists — your next `apply` will try to create everything from scratch, creating duplicate resources or failing on uniqueness constraints. State files must be protected, versioned, and backed up. This is a security audit finding when teams treat state as disposable.

## 5 Must-Practice Task Types (Week 3 Content)

### Task Type 1: RBAC — Create ServiceAccount + Role + RoleBinding

**Why this appears:** RBAC is in the Security domain (12% of CKA). Expect 1–2 RBAC tasks.

**The task will look like:** "Create a ServiceAccount named `monitoring` in namespace `ops`. Create a Role that allows `get`, `list`, `watch` on `pods`. Bind the Role to the ServiceAccount."

```bash
# Create the namespace if it does not exist
kubectl create namespace ops

# Create the ServiceAccount
kubectl create serviceaccount monitoring -n ops

# Create the Role (imperative — no YAML needed)
kubectl create role pod-reader \
  --verb=get,list,watch \
  --resource=pods \
  -n ops

# Create the RoleBinding
kubectl create rolebinding monitoring-pod-reader \
  --role=pod-reader \
  --serviceaccount=ops:monitoring \
  -n ops

# Verify
kubectl auth can-i get pods \
  --as=system:serviceaccount:ops:monitoring \
  -n ops
# EXPECTED OUTPUT: yes
```

**Why this syntax matters:** `--serviceaccount` takes `namespace:name` format — the colon separator is the most common mistake. Verify with `kubectl auth can-i` immediately after every RBAC task.

### Task Type 2: Network Policy — Restrict Pod-to-Pod Traffic

**Why this appears:** Network policies are in the Security domain and frequently appear in CKA.

**The task will look like:** "Create a NetworkPolicy in namespace `prod` that allows only pods with label `app=frontend` to connect to pods with label `app=backend` on port 8080."

```bash
# NetworkPolicy requires YAML — use kubectl create with --dry-run to scaffold
kubectl create networkpolicy backend-allow-frontend \
  --dry-run=client -o yaml > netpol.yaml

# Edit the YAML — the scaffold is incomplete; you need to add podSelector and ingress rules
# Paste this complete YAML into netpol.yaml:
cat <<'EOF' > netpol.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-allow-frontend
  namespace: prod
spec:
  podSelector:
    matchLabels:
      app: backend          # This policy applies TO backend pods
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend     # Allow FROM frontend pods only
    ports:
    - protocol: TCP
      port: 8080
EOF

kubectl apply -f netpol.yaml

# Verify
kubectl get networkpolicy -n prod
```

**Exam trap:** `podSelector: {}` (empty) means "select ALL pods in namespace" — easy to accidentally write if you are copying from docs. Always double-check your selector labels match exactly what the task specifies.

### Task Type 3: Node Troubleshooting — Uncordon + Diagnose NotReady

**Why this appears:** Cluster maintenance is a high-weight domain (11% of CKA).

**The task will look like:** "Node `node03` is in NotReady state. Investigate and fix it."

```bash
# Step 1: Check node status
kubectl get nodes
# Look for STATUS: NotReady

# Step 2: Describe the node for events
kubectl describe node node03
# Look for: Conditions section — particularly KubeletNotReady or NetworkPlugin

# Step 3: SSH into the node
ssh node03

# Step 4: Check kubelet status
sudo systemctl status kubelet
# If it shows 'failed' or 'inactive':
sudo systemctl start kubelet
sudo systemctl enable kubelet

# Step 5: Check kubelet logs for root cause
sudo journalctl -u kubelet -n 50 --no-pager
# Common causes: certificate expired, config file missing, container runtime not running

# Step 6: Check container runtime
sudo systemctl status containerd  # or docker, cri-o depending on cluster
sudo systemctl start containerd

# Step 7: Exit and verify from control plane
exit
kubectl get nodes
# EXPECTED: node03 STATUS: Ready
```

**Exam trap:** The kubelet config file path (`/var/lib/kubelet/config.yaml`) is sometimes wrong or missing. The journalctl output will tell you exactly which file it cannot find.

### Task Type 4: Persistent Volume + PVC Binding

**Why this appears:** Storage is a domain (10% of CKA). PV/PVC binding questions are predictable.

**The task will look like:** "Create a PersistentVolume of 500Mi using hostPath `/data/pv1`. Create a PVC requesting 200Mi. Create a pod that mounts this PVC at `/mnt/data`."

```bash
# PersistentVolume (cluster-scoped — no namespace)
cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-task
spec:
  capacity:
    storage: 500Mi
  accessModes:
    - ReadWriteOnce
  hostPath:
    path: /data/pv1
EOF

# PersistentVolumeClaim (namespace-scoped)
cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pvc-task
  namespace: default
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 200Mi
EOF

# Verify binding (PVC status should show Bound, not Pending)
kubectl get pvc pvc-task
# EXPECTED: STATUS: Bound   VOLUME: pv-task

# Create pod that uses the PVC
kubectl run pv-pod --image=nginx --dry-run=client -o yaml > pod.yaml
# Edit pod.yaml to add volumes and volumeMounts — the imperative command cannot do this
cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: pv-pod
spec:
  containers:
  - name: pv-pod
    image: nginx
    volumeMounts:
    - mountPath: /mnt/data
      name: task-volume
  volumes:
  - name: task-volume
    persistentVolumeClaim:
      claimName: pvc-task
EOF

kubectl get pod pv-pod
# EXPECTED: STATUS: Running
```

### Task Type 5: Cluster Upgrade — Control Plane Node

**Why this appears:** Cluster maintenance domain; `kubeadm upgrade` is a guaranteed task type.

**The task will look like:** "Upgrade the control plane node from 1.28.x to 1.29.0."

```bash
# On the control plane node — SSH in first
ssh controlplane

# Step 1: Update kubeadm (must upgrade kubeadm first, always)
sudo apt-mark unhold kubeadm
sudo apt-get update
sudo apt-get install -y kubeadm=1.29.0-00
sudo apt-mark hold kubeadm

# Step 2: Verify the upgrade plan
sudo kubeadm upgrade plan
# EXPECTED: Shows upgrade path and confirms 1.29.0 is available

# Step 3: Apply the upgrade
sudo kubeadm upgrade apply v1.29.0
# This upgrades: apiserver, controller-manager, scheduler, kube-proxy, CoreDNS

# Step 4: Upgrade kubelet and kubectl
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=1.29.0-00 kubectl=1.29.0-00
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload
sudo systemctl restart kubelet

# Step 5: Verify from control plane
exit

# EXPECTED: controlplane VERSION: v1.29.0

# Worker node upgrade is a SEPARATE task — drain first, then repeat on worker node
```

**Q1. A retail company wants to build a customer service chatbot that understands natural language queries and maintains conversation context. Which AWS service is the BEST fit?**

A) Amazon Comprehend
B) Amazon Lex
C) Amazon Kendra
D) Amazon Polly

**Answer: B**
Lex is purpose-built for conversational interfaces — it handles intents (what the user wants to do), slots (the parameters needed), and conversation context. Kendra is a search engine, not a chatbot. Comprehend analyzes text, does not drive conversation. Polly converts text to speech (output only).

**Q2. An insurance company wants to extract specific fields (policy number, claim amount, date) from scanned PDF claim forms. Which service should they use?**

A) Amazon Rekognition
B) Amazon Comprehend
C) Amazon Textract
D) Amazon Transcribe

**Answer: C**
Textract is designed specifically for structured data extraction from documents — it understands forms and tables, not just raw text. Comprehend does NLP on unstructured text. Rekognition does image/video analysis (not document structure extraction). Transcribe converts audio to text.

**Q3. What is retrieval-augmented generation (RAG)?**

A) A technique for fine-tuning foundation models on domain-specific data
B) A pattern where the model retrieves relevant documents at inference time to ground its responses in current, specific data
C) A method for compressing large language models to run on edge devices
D) An AWS service for managing model training pipelines

**Answer: B**
RAG combines a retrieval step (search a knowledge base for relevant context) with generation (pass that context + the user's question to an LLM). This lets a foundation model answer questions about your internal documents without retraining. Amazon Bedrock Knowledge Bases implements RAG. Fine-tuning (option A) updates model weights — more expensive, better for style/format adaptation. RAG is better for knowledge currency.

---

**Q4. A company needs to moderate user-uploaded images to detect explicit content. Which service handles this?**

A) Amazon Comprehend
B) Amazon Textract
C) Amazon Rekognition
D) Amazon Bedrock

**Answer: C**
Rekognition has a specific `DetectModerationLabels` API that identifies explicit, suggestive, and violent content in images and video. Comprehend handles text. Textract handles document structure. Bedrock can be used for image generation/analysis via foundation models but content moderation is a direct Rekognition use case.

**Q5. Which of the following BEST describes the difference between a foundation model and a traditional ML model?**

A) Foundation models are smaller and faster; traditional ML models are larger and more accurate
B) Foundation models are pre-trained on massive datasets and can be adapted to many tasks; traditional ML models are trained for a specific task from scratch
C) Foundation models require labeled training data; traditional ML models do not
D) Foundation models run only on AWS; traditional ML models can run anywhere

**Answer: B**
Foundation models (GPT-4, Claude, Llama, Stable Diffusion) are trained on internet-scale data and develop general capabilities that transfer across tasks. Traditional ML models (a churn prediction model, a fraud detection classifier) are trained on task-specific labeled data and work only for that task. Fine-tuning a foundation model adapts it without starting from scratch — this is the key economic advantage.

**Q6. A developer wants to use Claude 3 Sonnet via API without managing any infrastructure. Which AWS service provides this?**

A) Amazon SageMaker
B) Amazon Bedrock
C) Amazon EC2 with a GPU instance
D) Amazon ECS

**Answer: B**
Bedrock is the fully managed API access layer for foundation models including Anthropic Claude, Meta Llama, Mistral, Amazon Titan, and others. No infrastructure to manage — you call an API. SageMaker is for training/deploying your own models. EC2 GPU would require you to host the model yourself.

**Q7. What is "prompt engineering"?**

A) The process of training a language model on domain-specific data
B) The practice of designing input text (prompts) to elicit desired outputs from a language model without changing model weights
C) A software engineering discipline for building AI applications
D) The technique of compressing prompts to reduce token count

**Answer: B**
Prompt engineering is the craft of writing, structuring, and iterating on the input to a language model to get the desired output — without modifying the model itself. Techniques include few-shot prompting (providing examples), chain-of-thought (asking the model to reason step-by-step), role assignment ("You are an expert in..."), and output constraints. It requires no ML expertise, which is why it appears in a practitioner-level exam.

**Q8. A company wants to detect anomalies in financial transactions using historical labeled data (normal vs. fraudulent). What type of machine learning is this?**

A) Unsupervised learning
B) Reinforcement learning
C) Supervised learning
D) Transfer learning

**Answer: C**
Supervised learning uses labeled training data — you provide examples of "normal" and "fraudulent" transactions with labels, and the model learns to classify new transactions. Unsupervised learning finds patterns in unlabeled data (clustering, anomaly detection without labeled anomalies). Reinforcement learning trains via reward signals. Transfer learning fine-tunes a pre-trained model — not a classification type on its own.

**Q9. Which Amazon Bedrock feature allows you to connect a foundation model to your company's proprietary documents for Q&A?**

A) Model fine-tuning
B) Bedrock Knowledge Bases
C) Bedrock Agents
D) Bedrock Model Evaluation

**Answer: B**
Bedrock Knowledge Bases implements RAG — it indexes your documents (S3, Confluence, Salesforce, etc.) into a vector store and retrieves relevant chunks at inference time to ground model responses. Fine-tuning updates model weights. Agents orchestrate multi-step tasks using tools. Model Evaluation compares model outputs.

**Q10. What is a key risk of AI model bias?**

A) The model takes too long to respond
B) The model produces outputs that systematically disadvantage certain demographic groups due to skewed training data
C) The model uses too much GPU memory
D) The model answers correctly but with low confidence scores

**Answer: B**
Bias in ML arises when training data over-represents or under-represents certain groups — the model then makes systematically worse predictions for underrepresented groups. For example, a hiring algorithm trained on historical data from a male-dominated industry will score male candidates higher, perpetuating historical inequality. Responsible AI practices (fairness audits, diverse training data, bias detection tools) address this.

**Q11. A company wants to transcribe customer service call recordings and then analyze the sentiment of each call. Which combination of services should they use?**

A) Amazon Polly → Amazon Comprehend
B) Amazon Transcribe → Amazon Comprehend
C) Amazon Lex → Amazon Rekognition
D) Amazon Transcribe → Amazon Textract

**Answer: B**
Transcribe converts audio to text (speech-to-text). Comprehend then analyzes that text for sentiment (positive/negative/neutral/mixed), key phrases, and entities. This pipeline is a documented AWS reference architecture for call center analytics. Polly is text-to-speech (the reverse direction). Textract handles documents, not transcribed text.

**Q12. What does "model hallucination" mean in the context of large language models?**

A) The model processes visual inputs incorrectly
B) The model generates confident-sounding but factually incorrect or fabricated content
C) The model takes too long to generate a response due to high load
D) The model refuses to answer certain questions

**Answer: B**
Hallucination is when an LLM generates plausible-sounding text that is factually wrong — fabricating citations, events, names, or data. It happens because LLMs predict likely token sequences, not retrieve verified facts. RAG and grounding techniques reduce hallucination by providing source documents. This is a critical responsible AI concept.

---

**Q13. Which AWS service would you use to build a product recommendation engine for an e-commerce site?**

A) Amazon Forecast
B) Amazon Personalize
C) Amazon Comprehend
D) Amazon Kendra

**Answer: B**
Personalize is purpose-built for recommendation and personalization — it learns from user interaction data (clicks, purchases, views) and generates real-time recommendations like "customers also viewed" or "top picks for you." Forecast is for time-series prediction (demand forecasting, inventory). Comprehend does NLP. Kendra does document search.

**Q14. What is the primary purpose of an AI guardrail in Amazon Bedrock?**

A) To limit the number of API calls a model can make
B) To restrict model outputs based on content policies — filtering harmful, off-topic, or sensitive content
C) To reduce the cost of model inference
D) To enforce IAM permissions on Bedrock API calls

**Answer: B**
Bedrock Guardrails allow you to define content filters (hate speech, violence, PII, topic restrictions) applied to both inputs and outputs before they reach your application. This is how enterprise customers implement responsible AI policies — preventing the model from discussing competitors, generating inappropriate content, or leaking PII. IAM controls access; Guardrails controls content.

**Q15. A healthcare company wants to extract patient information (name, DOB, diagnosis) from clinical notes stored as unstructured text. Which service combination is most appropriate?**

A) Amazon Textract → Amazon Rekognition
B) Amazon Comprehend Medical → Amazon S3
C) Amazon Lex → Amazon Transcribe
D) Amazon Kendra → Amazon Bedrock

**Answer: B**
Amazon Comprehend Medical is a specialized variant of Comprehend designed specifically for clinical text — it extracts medical entities (medications, dosages, diagnoses, procedures, anatomy) and can detect PHI (Protected Health Information). The results are stored in S3 for downstream processing. Textract extracts structured data from scanned forms, not free-text clinical notes.

## STAR Anchor (T-Mobile Framing)

**Q1 — Module library:** "At T-Mobile I governed IaC standards across 4 SRE-owned platforms plus 6 app teams consuming our infrastructure. We had copy-paste Terraform that caused a prod/dev config drift incident — a dev team's local change propagated to prod because they had forked the config. I led the design of a shared module library in our internal GitLab, versioned by semantic tags, with a golden path doc. App teams call the module, they do not own it. Drift incidents dropped to zero in the following quarter."

**Q2 — CKA strategy:** "I assess operational candidates on time-to-resolution under pressure, not just knowledge. If someone needs to look up syntax for every kubectl command during a simulated incident, their cognitive load is consumed by the tool, not the problem. I use a similar lens in hiring: I ask candidates to walk me through a live debug scenario — the ones who need to Google every command are not yet at Staff level for our team."

**Q3 — Bedrock architecture review:** "When my team evaluated a GenAI feature for our notification platform, the questions I drove were: What data touches the model — does it include PII? What are the latency SLOs — Bedrock adds P99 tail latency we had to budget. Who reviews model outputs before they reach customers? What is the rollback path if the model starts hallucinating? We built a human-review gate for the first 30 days. That is the Director question — not 'can we call the API?' but 'what breaks when it fails?'"

**Q4 — Search diagnosis:** "8 screens and 2 second rounds with no offers in 3 months means the problem is in conversion, not pipeline generation. I would review my second-round performance specifically — what questions were asked, how did I answer, what did the debrief feedback say? At Director level, the failure mode is usually technical depth in one area (often live system design) or insufficient executive presence. I would schedule 3 mock loops with people who have done Director loops at FAANG and ask for brutal calibration."

**Q5 — RAG vs fine-tuning:** "For an internal SRE knowledge base, I would choose RAG in almost all cases. Our runbooks change every time we do a major migration — fine-tuning a model on a snapshot of runbooks from 6 months ago means the model is confidently wrong about your current environment. RAG retrieves from the live knowledge base at query time. Fine-tuning makes sense when you need the model to change its OUTPUT format or writing style — for example, generating incident postmortems in our specific template. Not for knowledge retrieval."

## Tradeoffs and Failure Modes

**1. Cert-chasing without lab depth**

Failure mode: You pass Terraform Associate but cannot answer "what happens if the DynamoDB lock table is deleted mid-apply?" in an interview. Interviewers at Director level know the difference between cert knowledge and operational depth — they probe one level below the exam questions. Fix: Every cert question in this document has a "why" explanation. Learn the why, not just the answer.

**2. Broad application without depth per company**

Failure mode: You apply to 20 companies simultaneously, but your cover note is generic, your resume is not tailored, and you do not know the company's engineering blog. At Director level, hiring managers Google you before every screen. If your LinkedIn and application materials do not reflect knowledge of their specific technical environment (Adobe AJO, Capital One Nessie, etc.), you lose the warmth advantage. Fix: Maximum 3 active applications per week; research each one before applying.

**3. Passive job search during active sprint**

Failure mode: Applying and waiting. Director searches at FAANG take 60–120 days end-to-end. If you only apply and wait, your pipeline dries up in week 4. Fix: Maintain parallel tracks — Phase 1 startup outreach (faster cycles, 30–45 days) running simultaneously with Phase 2 FAANG applications (longer cycles). The startup offer creates urgency with FAANG. Competing timelines are the only real leverage you have as a candidate.
