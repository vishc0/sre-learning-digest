# IRSA — IAM Roles for Service Accounts | Week 3 Day 4

## Why This Matters

Every company running workloads on EKS has faced the "credentials in pods" problem — developers hard-code AWS keys in environment variables or Kubernetes Secrets, those keys get leaked, and suddenly you have a breach investigation instead of a deploy. IRSA eliminates that attack surface entirely by making AWS credential issuance cryptographic and identity-bound, not secret-bound. At a Director level, IRSA is the non-negotiable baseline before any EKS workload touches production data — interviewers at FAANG and growth-stage companies will probe whether you govern this at platform-policy level, not just know the mechanics.

---

## Analogy

Think of IRSA exactly like **Vault AppRole** — a mechanism your team already understands operationally.

In Vault AppRole:
- The application proves its identity using a `role_id` + `secret_id`
- Vault issues a short-lived token
- That token has a policy attached — it can only do what the policy says
- The token expires; the app must renew

In IRSA:
- The pod proves its identity using a **projected ServiceAccount token** (a signed JWT issued by the cluster's OIDC provider)
- AWS STS issues short-lived IAM credentials (15-minute expiry by default, auto-refreshed by the SDK)
- Those credentials have an IAM role attached — scoped to exactly one ServiceAccount in one namespace
- No static secrets anywhere — the OIDC JWT *is* the proof of identity

The difference from Vault AppRole: **you don't manage the secret_id**. The Kubernetes control plane mints the JWT automatically when the pod starts. The AWS SDK inside the pod handles the STS exchange invisibly. Your developers write `boto3.client('s3')` — that's it. The credential plumbing is fully automatic.

## Vocabulary Translation Table

| Their Term | Your Existing Mental Model |
|---|---|
| OIDC Provider | A trust anchor — like configuring LDAP as an identity source. Tells AWS "I trust JWTs signed by this cluster." |
| ServiceAccount | Kubernetes identity for a pod — like a Linux service user (`svc-kafka`, `svc-splunk`). One per workload. |
| Projected ServiceAccount Token | The JWT credential auto-mounted into the pod. Like a Kerberos ticket — time-limited, signed, unforgeable. |
| AssumeRoleWithWebIdentity | The STS API call that exchanges the JWT for real IAM credentials. Like Vault `auth/approle/login`. |
| Trust Policy | IAM's "who is allowed to assume this role" — like a Vault policy `bound_service_account_names`. |
| IRSA | The full end-to-end pattern. Short for: OIDC + ServiceAccount annotation + token projection + STS exchange. |
| Node IAM Role | The EC2 instance profile — the blast radius problem. All pods on a node share it. IRSA replaces this need. |
| `eks.amazonaws.com/role-arn` annotation | The label you put on the ServiceAccount to link it to an IAM role. One annotation, one role. |

## 30-Second Version

IRSA lets a Kubernetes pod assume an IAM role without storing any AWS credentials. The cluster acts as an OIDC identity provider. When a pod starts, Kubernetes injects a signed JWT (a projected token) into the pod's filesystem. The AWS SDK automatically presents that JWT to STS, which issues short-lived IAM credentials scoped to the role you attached to that pod's ServiceAccount. No keys, no Secrets, no rotation scripts — cryptographic identity all the way down.

## 2-Minute Version

**The Problem IRSA Solves**

Before IRSA, your options for getting AWS credentials into a pod were bad:
1. Hard-code `AWS_ACCESS_KEY_ID` in a Deployment manifest — credentials leak into git, CI logs, kubectl describe output
2. Store them in a Kubernetes Secret — Secrets are base64 (not encrypted), readable by anyone with `get secrets` RBAC permission
3. Use the node's EC2 instance profile — the nuclear option: every pod on that node inherits the same role. A compromised nginx pod gets your S3 access. Blast radius = entire node.

**How IRSA Works — The Flow**

```
Pod starts
  └─> Kubernetes mounts projected JWT at /var/run/secrets/eks.amazonaws.com/serviceaccount/token
        └─> AWS SDK detects IRSA environment variables
              └─> SDK calls STS AssumeRoleWithWebIdentity(JWT, RoleARN)
                    └─> STS validates JWT signature against OIDC provider
                          └─> Trust policy checks: does this ServiceAccount match the condition?
                                └─> STS issues temporary credentials (15-min expiry)
                                      └─> SDK caches and auto-refreshes credentials
                                            └─> Your app code calls S3/DynamoDB/etc normally
```

**The Two Configuration Pieces**

1. **OIDC Provider**: Registered once per cluster in IAM. Tells AWS "cluster X at this URL is a trusted identity source."

2. **IAM Role Trust Policy**: Locks down who can assume the role. The condition checks:
   - The OIDC issuer URL (which cluster)
   - `sub` claim in the JWT: `system:serviceaccount:<namespace>:<serviceaccount-name>` (which pod identity)

This means even if an attacker steals a JWT from pod A, they cannot assume the role scoped to pod B — the `sub` claim won't match the trust policy condition.

## 5-Minute Version

**Design Decisions and Tradeoffs**

**1. Per-ServiceAccount Granularity vs. Operational Overhead**

IRSA gives you least-privilege at the pod identity level. A `payment-processor` pod gets S3 read on the payments bucket only. A `log-shipper` pod gets CloudWatch PutLogEvents only. This is the right security posture.

The tradeoff: you now have N IAM roles to manage (one per distinct workload identity). At scale — say, 50 microservices across 10 namespaces — this is real IAM sprawl. Governance answer: manage IAM roles for IRSA via Terraform modules with a consistent naming convention (`eks-<cluster>-<namespace>-<serviceaccount>`), and enforce it through a policy-as-code check in CI. Don't let developers create these ad hoc.

**2. OIDC Token Expiry and SDK Behavior**

The projected token has a configurable TTL (default 86400 seconds, but `eksctl` sets it to 86400s; in practice STS credentials expire in 15 min and the SDK refreshes them). The AWS SDK (boto3, the Java SDK, Go SDK) handles this refresh automatically via the `WebIdentityTokenFileCredentialsProvider`. Older SDK versions do not support IRSA — this is a gotcha on legacy apps. If you see `NoCredentialProviders` errors in pods that have IRSA configured, the first thing to check is SDK version.

**3. Node Role vs. IRSA — The Blast Radius Argument**

The node EC2 instance profile still exists — Kubernetes itself (kubelet, VPC CNI plugin) needs it to function. The governance rule: the node role should have **only** what the node needs to operate (ECR pull, VPC CNI, CloudWatch agent). Nothing application-specific. All app-level AWS access goes through IRSA. At audit time, you can prove per-workload AWS permissions through ServiceAccount annotations + IAM role trust policies. The node role is infrastructure; IRSA is workload identity. They serve different purposes and must not be conflated.

**4. Namespace Scoping as a Security Boundary**

The trust policy condition `system:serviceaccount:prod:payment-processor` means even if a developer in the `dev` namespace creates a ServiceAccount named `payment-processor`, they cannot assume the production role. The namespace is part of the identity. This makes namespace isolation a meaningful security boundary — another argument for strict namespace-per-team governance.

**5. What IRSA Does NOT Protect Against**

- A compromised pod can still use its own credentials maliciously within the scope of its role
- IRSA doesn't prevent lateral movement within the pod's permitted AWS permissions
- If you annotate a ServiceAccount too broadly (e.g., S3 full access on all buckets), IRSA doesn't help you — IAM policy quality still matters
- The OIDC private key lives in the EKS control plane — you trust AWS to protect it. For highly regulated environments, this is a compliance discussion.

## Director/VP Lens

**How a Leader Governs IRSA**

As Director/VP SRE, you don't configure trust policies — but you set the policy that makes IRSA mandatory and measurable.

**Policy posture you establish:**
1. *No pod runs with AWS credentials in environment variables or Secrets.* This is a P0 security finding, auto-flagged in CI via a `kubectl` audit or OPA Gatekeeper policy.
2. *All IRSA roles are defined in Terraform, reviewed in PR, and follow the naming convention.* No manual IAM role creation for EKS workloads.
3. *The node IAM role has a maximum permission list — anything beyond that must have a documented exception.* Reviewed quarterly.
4. *ServiceAccount IRSA annotations are audited monthly* — compare annotations in-cluster against IAM roles that have the cluster OIDC in their trust policy. Orphaned roles = wasted permissions = audit finding.

**How you assess IC readiness:**
Ask: "How would you detect if a pod was using node role instead of IRSA?" If they can't answer (look at CloudTrail `userAgent` — node-role calls appear as `aws-sdk-*` from EC2 IP, IRSA calls show `sts.amazonaws.com` assumed-role ARN), they haven't operated this in prod.

**Interview framing (Director level):**
"At T-Mobile we had 40+ microservices on EKS, each needing different S3 and DynamoDB access. I established IRSA as the platform standard — defined the Terraform module for IRSA role creation, embedded it in our golden path CI template, and made node-role-for-app-access a P2 security finding in our quarterly posture review. IRSA adoption went from 20% to 100% in two quarters."

## Comprehension Check

Answer these before unlocking the lab. You need 2 of 3 correct.

**Q1.** A developer tells you: "My pod can't access S3. I annotated the ServiceAccount with the role ARN but it's still failing." You look at the pod and see `AWS_ACCESS_KEY_ID` set as an environment variable in the Deployment spec. What's happening, and what should you do?

**Q2.** Explain why the IRSA trust policy condition `StringEquals: "oidc.eks.us-east-1.amazonaws.com/id/EXAMXXXXXX:sub": "system:serviceaccount:prod:payment-svc"` prevents a developer in the `staging` namespace from assuming the role — even if they create a ServiceAccount also named `payment-svc`.

**Q3.** You're doing a security audit. You find an IAM role with this trust policy condition: `"system:serviceaccount:*:*"` for your cluster's OIDC. What's the risk, and what's the remediation?

---

## ANSWER KEY — Comprehension Check

**A1.** The explicit `AWS_ACCESS_KEY_ID` environment variable takes precedence over IRSA in the AWS SDK credential provider chain. The SDK finds the static key first and never reaches the IRSA token. The annotation is correct but the static credential is overriding it. Fix: remove `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` from the Deployment spec entirely. The IRSA env vars (`AWS_ROLE_ARN`, `AWS_WEB_IDENTITY_TOKEN_FILE`) set automatically by the EKS pod identity webhook will then work. Also check for these credentials in Kubernetes Secrets mounted as volumes.

**A2.** The `sub` claim in the projected JWT is `system:serviceaccount:<namespace>:<serviceaccount-name>`. When the `staging` namespace's `payment-svc` ServiceAccount calls STS, the JWT's `sub` will be `system:serviceaccount:staging:payment-svc`. The trust policy condition requires `system:serviceaccount:prod:payment-svc` — the namespace `staging` does not match `prod`, so STS rejects the AssumeRole call with `AccessDenied`. The namespace is a cryptographically enforced part of the identity.

**A3.** This is a wildcard trust: any ServiceAccount in any namespace on this cluster can assume this role. It completely defeats IRSA's least-privilege purpose — any compromised pod, in any namespace, can assume this role and access whatever it permits. Remediation: immediately restrict the trust policy to the specific namespace and ServiceAccount that legitimately need this role (`system:serviceaccount:<namespace>:<specific-sa>`). Audit CloudTrail for `AssumeRoleWithWebIdentity` calls to this role to determine if it has been exploited. File it as a P1 security finding if the role has broad permissions.

### Step 3: Associate the OIDC provider with your EKS cluster

This is the "register the trust anchor" step — you're telling AWS IAM: "trust identity tokens issued by this EKS cluster."

```bash
# First, check if OIDC provider already exists for this cluster
aws eks describe-cluster \
  --name $CLUSTER_NAME \
  --region $AWS_REGION \
  --query "cluster.identity.oidc.issuer" \
  --output text
```

**EXPECTED_OUTPUT:**
```
https://oidc.eks.us-east-1.amazonaws.com/id/EXAMPL1234567890ABCDEF
```
Note the ID at the end (after `/id/`) — this is your OIDC issuer ID. Save it:

```bash
export OIDC_ID=$(aws eks describe-cluster \
  --name $CLUSTER_NAME \
  --region $AWS_REGION \
  --query "cluster.identity.oidc.issuer" \
  --output text | cut -d '/' -f 5)

echo "OIDC ID: $OIDC_ID"

# Check if OIDC provider is already registered in IAM
aws iam list-open-id-connect-providers | grep $OIDC_ID
```

If the grep returns nothing, the OIDC provider is NOT yet registered. Register it:

```bash
eksctl utils associate-iam-oidc-provider \
  --cluster $CLUSTER_NAME \
  --region $AWS_REGION \
  --approve
```

**EXPECTED_OUTPUT (if registering):**
```
2024-xx-xx xx:xx:xx [ℹ]  will create IAM Open ID Connect provider for cluster "training-cluster" in "us-east-1"
2024-xx-xx xx:xx:xx [✔]  created IAM Open ID Connect provider for cluster "training-cluster" in "us-east-1"
```

Verify it's now registered:
```bash
aws iam list-open-id-connect-providers | grep $OIDC_ID
```

**EXPECTED_OUTPUT:**
```
"Arn": "arn:aws:iam::123456789012:oidc-provider/oidc.eks.us-east-1.amazonaws.com/id/EXAMPL1234567890ABCDEF"
```

**WHY**: Without this registration, STS will refuse all `AssumeRoleWithWebIdentity` calls from this cluster because it doesn't recognize the JWT issuer. This is the trust anchor — done once per cluster, usually at cluster creation time.

### Step 4: Create the IAM role with a scoped trust policy

This is the most important step — the trust policy is what makes IRSA least-privilege.

```bash
# Build the trust policy document — this is what controls who can assume this role
cat > /tmp/irsa-trust-policy.json << EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::${AWS_ACCOUNT_ID}:oidc-provider/oidc.eks.${AWS_REGION}.amazonaws.com/id/${OIDC_ID}"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "oidc.eks.${AWS_REGION}.amazonaws.com/id/${OIDC_ID}:sub": "system:serviceaccount:${NAMESPACE}:${SERVICE_ACCOUNT_NAME}",
          "oidc.eks.${AWS_REGION}.amazonaws.com/id/${OIDC_ID}:aud": "sts.amazonaws.com"
        }
      }
    }
  ]
}
EOF

# Verify the file looks right — check that your real values were substituted
cat /tmp/irsa-trust-policy.json
```

**EXPECTED_OUTPUT** (with your real account/OIDC values substituted):
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::123456789012:oidc-provider/oidc.eks.us-east-1.amazonaws.com/id/EXAMPL1234567890ABCDEF"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "oidc.eks.us-east-1.amazonaws.com/id/EXAMPL1234567890ABCDEF:sub": "system:serviceaccount:irsa-lab:s3-reader-sa",
          "oidc.eks.us-east-1.amazonaws.com/id/EXAMPL1234567890ABCDEF:aud": "sts.amazonaws.com"
        }
      }
    }
  ]
}
```

Now create the IAM role:
```bash
aws iam create-role \
  --role-name $IAM_ROLE_NAME \
  --assume-role-policy-document file:///tmp/irsa-trust-policy.json \
  --description "IRSA role: read-only access to irsa-lab-allowed S3 bucket"
```

**EXPECTED_OUTPUT:**
```json
{
    "Role": {
        "Path": "/",
        "RoleName": "eks-irsa-lab-s3-reader",
        "RoleId": "AROA...",
        "Arn": "arn:aws:iam::123456789012:role/eks-irsa-lab-s3-reader",
        "CreateDate": "2024-...",
        "AssumeRolePolicyDocument": { ... }
    }
}
```

Save the role ARN:
```bash
export IAM_ROLE_ARN=$(aws iam get-role \
  --role-name $IAM_ROLE_NAME \
  --query "Role.Arn" \
  --output text)
echo "Role ARN: $IAM_ROLE_ARN"
```

**EXPECTED_OUTPUT:**
```
Role ARN: arn:aws:iam::123456789012:role/eks-irsa-lab-s3-reader
```

**WHY**: The `sub` condition in the trust policy is the lock. It says: this role can ONLY be assumed by a JWT whose subject is exactly `system:serviceaccount:irsa-lab:s3-reader-sa`. Any other ServiceAccount, in any other namespace, gets denied at STS — before they ever reach the IAM permission policy.

### Step 5: Create and attach the S3 permission policy

The trust policy controls *who can assume* the role. The permission policy controls *what the role can do*.

```bash
# Create a permissions policy: ListBucket + GetObject on the allowed bucket ONLY
cat > /tmp/irsa-s3-policy.json << EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket"
      ],
      "Resource": "arn:aws:s3:::${ALLOWED_BUCKET}"
    },
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::${ALLOWED_BUCKET}/*"
    }
  ]
}
EOF

# Create the policy in IAM
aws iam create-policy \
  --policy-name "eks-irsa-lab-s3-readonly" \
  --policy-document file:///tmp/irsa-s3-policy.json \
  --description "Read-only access to irsa-lab-allowed bucket"
```

**EXPECTED_OUTPUT:**
```json
{
    "Policy": {
        "PolicyName": "eks-irsa-lab-s3-readonly",
        "PolicyId": "ANPA...",
        "Arn": "arn:aws:iam::123456789012:policy/eks-irsa-lab-s3-readonly",
        "Path": "/",
        "DefaultVersionId": "v1",
        "AttachmentCount": 0,
        "IsAttachable": true
    }
}
```

```bash
# Attach the policy to the role
aws iam attach-role-policy \
  --role-name $IAM_ROLE_NAME \
  --policy-arn "arn:aws:iam::${AWS_ACCOUNT_ID}:policy/eks-irsa-lab-s3-readonly"

echo "Policy attached to role"
```

**EXPECTED_OUTPUT:**
```
Policy attached to role
```
(no output from the attach command itself = success)

**WHY**: The separation of trust policy (identity) and permission policy (capability) is IAM's core design. You can reuse the permission policy across different roles, and you can change what a role can do without changing who can assume it.

### Step 6: Create the namespace and annotated ServiceAccount

```bash
# Create the namespace
kubectl create namespace $NAMESPACE

# Create the ServiceAccount with the IRSA annotation
# The annotation is how Kubernetes links this SA identity to the IAM role
kubectl create serviceaccount $SERVICE_ACCOUNT_NAME \
  --namespace $NAMESPACE

# Annotate it with the IAM role ARN
kubectl annotate serviceaccount $SERVICE_ACCOUNT_NAME \
  --namespace $NAMESPACE \
  eks.amazonaws.com/role-arn=$IAM_ROLE_ARN

# Verify the annotation
kubectl describe serviceaccount $SERVICE_ACCOUNT_NAME \
  --namespace $NAMESPACE
```

**EXPECTED_OUTPUT:**
```
Name:                s3-reader-sa
Namespace:           irsa-lab
Labels:              <none>
Annotations:         eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/eks-irsa-lab-s3-reader
Image pull secrets:  <none>
Mountable secrets:   <none>
Tokens:              <none>
Events:              <none>
```

**WHY**: The annotation is the link. When a pod using this ServiceAccount starts, the EKS Pod Identity Webhook reads this annotation and automatically injects `AWS_ROLE_ARN` and `AWS_WEB_IDENTITY_TOKEN_FILE` environment variables into the pod — your app code never needs to know any of this.

### Step 10: Verify the denied bucket is inaccessible (least privilege proof)

```bash
# Try to list the denied bucket — should fail with AccessDenied
kubectl exec irsa-test-pod \
  --namespace $NAMESPACE \
  -- aws s3 ls s3://${DENIED_BUCKET}
```

**EXPECTED_OUTPUT:**
```
An error occurred (AccessDenied) when calling the ListObjectsV2 operation: Access Denied
```

The command will exit with a non-zero code. This is exactly what you want.

```bash
# Also try to list ALL S3 buckets — should also fail
kubectl exec irsa-test-pod \
  --namespace $NAMESPACE \
  -- aws s3 ls
```

**EXPECTED_OUTPUT:**
```
An error occurred (AccessDenied) when calling the ListBuckets operation: Access Denied
```

**WHY**: The IAM permission policy only grants access to `arn:aws:s3:::irsa-lab-allowed-<account-id>`. The `s3:ListAllMyBuckets` permission was never granted, and access to the denied bucket was never granted. This is least-privilege working exactly as designed. This output is what you show in a security audit to prove IRSA is effective.

### Step 11: (Bonus) Inspect the projected JWT token

This makes the cryptographic identity concrete — you can actually read the JWT that the pod uses.

```bash
# Read the JWT from inside the pod
kubectl exec irsa-test-pod \
  --namespace $NAMESPACE \
  -- cat /var/run/secrets/eks.amazonaws.com/serviceaccount/token
```

**EXPECTED_OUTPUT:** A long base64-encoded string like:
```
eyJhbGciOiJSUzI1NiIsImtpZCI6Ii4uLiJ9.eyJhdWQiOlsic3RzLmFtYXpvbmF3cy5jb20iXSwiZXhwIjox...
```

Now decode it to see the claims (from your WSL2 terminal, not inside the pod):
```bash

# Decode the JWT payload (no signature verification — just reading the claims)
echo $TOKEN | cut -d'.' -f2 | base64 -d 2>/dev/null | python3 -m json.tool
```

**EXPECTED_OUTPUT:**
```json

    "aud": [
        "sts.amazonaws.com"

    "iat": 1234567890,
```

## Lab ANSWER KEY — Diagnostics

**Step 3 — OIDC association fails:**
```
Error: could not retrieve cluster endpoint: ...
```
Cluster isn't reachable. Check: `aws eks describe-cluster --name $CLUSTER_NAME --region $AWS_REGION`. If `Status` is not `ACTIVE`, wait or recreate the cluster.

**Step 7 — Pod stuck in `Pending`:**
```bash
kubectl describe pod irsa-test-pod --namespace $NAMESPACE
```
Look at `Events:`. Common cause: node capacity — check `kubectl get nodes`. If no nodes, the nodegroup may have scaled to 0; check Auto Scaling Group in AWS Console.

**Step 8 — `get-caller-identity` returns EC2 instance profile (not IRSA role):**
```
"Arn": "arn:aws:sts::123456789012:assumed-role/eksctl-training-cluster-nodegroup.../i-..."
```
The pod is using the node role instead of IRSA. Root causes to check in order:
1. `kubectl describe pod irsa-test-pod -n $NAMESPACE | grep -A5 "Environment"` — are `AWS_ROLE_ARN` and `AWS_WEB_IDENTITY_TOKEN_FILE` present? If not, the webhook didn't fire.
2. Check the ServiceAccount annotation exists: `kubectl describe sa $SERVICE_ACCOUNT_NAME -n $NAMESPACE`
3. Verify the EKS pod identity webhook is running: `kubectl get pods -n kube-system | grep pod-identity-webhook`
4. If webhook is missing, re-enable OIDC: `eksctl utils associate-iam-oidc-provider --cluster $CLUSTER_NAME --region $AWS_REGION --approve` and recreate the pod.

**Step 8 — `get-caller-identity` returns AccessDenied:**
```
An error occurred (AccessDenied) when calling the AssumeRoleWithWebIdentity operation
```
Trust policy mismatch. Check:
1. The OIDC ID in the trust policy matches `$OIDC_ID` exactly: `aws iam get-role --role-name $IAM_ROLE_NAME --query "Role.AssumeRolePolicyDocument"`
2. The `sub` condition matches exactly `system:serviceaccount:<namespace>:<sa-name>` — namespace and SA name are case-sensitive
3. The `aud` condition is `sts.amazonaws.com` (not `https://sts.amazonaws.com`)

**Step 9 — S3 `ls` returns `NoSuchBucket`:**
The bucket name variable wasn't set. Rerun Step 0 and Step 2 to restore variables and recreate the bucket.

**Step 10 — Denied bucket access SUCCEEDS (wrong result):**
The IAM permission policy is too broad. Check: `aws iam list-attached-role-policies --role-name $IAM_ROLE_NAME` and then inspect the policy. If it contains `"Resource": "*"` or wildcards covering the denied bucket, the policy needs to be tightened to only the allowed bucket ARN.

## STAR Anchor

Frame each answer with T-Mobile experience:

**Q1 (IRSA flow):**
"In my current role managing the T-Mobile notification platform on EKS, I led the adoption of IRSA as our standard for AWS credential management. Let me walk through the flow as we use it: [technical flow]. Before IRSA, we had three teams storing credentials in Kubernetes Secrets — I identified this during a quarterly security posture review and led the migration."

**Q2 (Node role vs IRSA):**
"At T-Mobile we use EKS managed nodegroups. The node role is scoped to what the node infrastructure needs: ECR pull for container images, VPC CNI for pod networking, CloudWatch agent for node metrics. Nothing else. Every application-layer AWS call goes through IRSA. I enforced this as a platform standard with an OPA Gatekeeper policy that flags any Deployment not using an IRSA-annotated ServiceAccount."

**Q3 (Migration to 100%):**
"This is a change management problem as much as a technical one. Situation: my team found 40% of workloads on node role during a security audit. Task: migrate to IRSA without disruption. Action: I created a Terraform module for IRSA role creation, added it to our CI golden path, published a self-service runbook, and set a 90-day deadline with weekly tracking. For the stubborn cases I did 1:1 sessions with team leads. Result: 100% in 11 weeks. The key was making IRSA the path of least resistance, not just a mandate."

**Q4 (CloudTrail detection):**
"In CloudTrail, an IRSA call shows `userIdentity.type` as `AssumedRole` with the ARN containing your IRSA role name (`eks-<cluster>-<namespace>-<sa>`). A node role call shows `userIdentity.sessionContext.sessionIssuer.arn` pointing to the EC2 instance profile role. I built a Splunk search on our CloudTrail S3 export to flag any S3 calls where the source ARN pattern matched our node role but the request originated from a pod IP (VPC flow logs correlation). That gave us the list of non-compliant workloads."

**Q5 (Shared role anti-pattern):**
"I'd explain it this way: a shared IRSA role makes the blast radius of any one compromised pod equal to the blast radius of all pods. If your payment processor and your static asset server share a role, and the asset server gets compromised through a dependency vulnerability, the attacker now has payment processor permissions. IRSA's entire value is granularity. The operational overhead is real — manage it through Terraform modules and a naming convention, not by collapsing roles. I've had this conversation with developers at T-Mobile; the answer is always: show them the compliance risk and give them the module that makes it easy."

## Tradeoffs & Failure Modes

**Tradeoff 1: Granularity vs. IAM Role Sprawl**

IRSA done right means one IAM role per distinct workload identity. At scale (50+ microservices), this creates IAM sprawl — hundreds of roles, hard to audit by hand. The mitigation is Terraform module enforcement with consistent naming (`eks-<cluster>-<namespace>-<sa>`) and tagging, combined with a quarterly automated audit that cross-references ServiceAccount annotations in-cluster against IAM role trust policies. Roles with no matching ServiceAccount are orphaned and should be deleted. Roles with overly broad trust policies (`system:serviceaccount:*:*`) are P1 findings.

**Tradeoff 2: SDK Version Dependency**

IRSA requires AWS SDK support for `WebIdentityTokenFileCredentialsProvider`. Older SDK versions (boto3 < 1.9.x, AWS Java SDK v1 < 1.11.x) do not support this and will silently fall back to the node role or return `NoCredentialProviders`. In legacy Java microservices — common in enterprise EKS migrations — this is a real production footgun. Mitigation: SDK version is a hard gate in your container image CI pipeline, enforced via a policy scan (Trivy or Grype can detect known-old SDK versions as a vulnerability).

**Tradeoff 3: Trust Policy Misconfiguration is Silent Until Exploit**

A trust policy with `StringLike` instead of `StringEquals`, or a wildcard in the namespace position, is functionally correct — the role works for the intended ServiceAccount — but dangerously permissive. You won't know until an auditor or attacker finds it. Mitigation: use OPA/Conftest to validate IRSA trust policies as part of the Terraform plan CI check. The policy rule: any trust policy with `*` in the `sub` condition is a build failure. This is where "policy as code" earns its keep — a human code review will miss this; an automated rule won't.
