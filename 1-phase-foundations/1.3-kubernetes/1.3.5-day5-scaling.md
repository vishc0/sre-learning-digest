## Why This Matters

Autoscaling is where 80% of production Kubernetes reliability incidents originate — not because teams don't configure it, but because they configure it wrong and discover the gap during a traffic spike at 2 AM. For a Director/VP SRE role, this topic has three interview dimensions: do you understand the mechanism well enough to debug scaling failures, can you architect a node pool strategy for cost efficiency, and can you connect autoscaling to SLO burn rate (if your pods don't scale in time, your error budget bleeds). Your T-Mobile 25M msg/day platform makes KEDA's event-driven model particularly interview-relevant — you can speak from operational experience about queue-depth-driven scaling rather than theoretical examples.

## Analogy

Think of your RabbitMQ notification platform as a highway interchange. HPA is the traffic engineer who watches cars per minute on the on-ramp and opens or closes lanes (pods) based on what they see — but they need to count cars for 30-60 seconds before acting, so by the time lanes open, the backup already exists. KEDA is a smarter system that watches the queue at the toll booth directly: the moment 500 messages pile up, it opens lanes before the backup forms. Karpenter is the highway authority that decides whether to open a new interchange segment (provision a new EC2 node) based on what car types are waiting — it can grab a cheaper toll road (Spot instance) if only leisure traffic (batch jobs) is queued, not ambulances (latency-sensitive services).

## Mental Model: The Scaling Stack

Think of autoscaling like traffic control on a highway:

- **HPA** (Horizontal Pod Autoscaler): Add lanes when traffic rises — reactive, based on car-density sensors.
- **KEDA** (Kubernetes Event-Driven Autoscaling): Open express lanes based on *queue depth* signals (e.g., messages in RabbitMQ) — proactive.
- **VPA** (Vertical Pod Autoscaler): Widen each lane to let cars go faster — right-sizes CPU/memory per pod, not pod *count*.
- **Karpenter**: Add entire highway sections (nodes) when lanes are full — smarter bin-packing than Cluster Autoscaler.

**Context**: At T-Mobile, a 25M msg/day platform = ~290 msg/sec average, but 2-minute traffic spikes hit 500K msg/sec. A 4-minute scale lag means 1.5M messages queue before pod relief arrives. KEDA + Karpenter cuts that to ~167K queued, a 90% latency improvement.

## 30-Second Version

Kubernetes autoscaling has three layers: HPA scales pod count based on CPU/memory or custom metrics; VPA adjusts pod resource requests over time; Karpenter provisions and deprovisions nodes based on what pods actually need. The classic 4-minute lag problem (HPA takes 3 min to react + Cluster Autoscaler takes 1-2 min to provision a node = 4-6 min before traffic is actually absorbed) is solved by KEDA for pod scaling and Karpenter for node provisioning. In 2025, the production answer is KEDA + Karpenter for latency-sensitive workloads, with HPA reserved for simple CPU-bound stateless services.

## 2-Minute Version

**HPA Mechanics**

HPA runs a control loop every 15 seconds (configurable). It queries the Metrics Server (or a custom metrics adapter) for the current value of a metric, compares it to the target, and calculates desired replicas using:

```
desiredReplicas = ceil(currentReplicas * (currentMetricValue / targetMetricValue))
```

For CPU at 80% target with 4 pods running at 160% utilization: `ceil(4 * (160/80)) = 8 pods`.

**Critical rule**: This formula only works if **pods declare `resources.requests`**. Without it, HPA cannot compute utilization percentage:

```yaml
# WRONG — HPA cannot scale this
spec:
  containers:
  - name: worker
    image: myapp:v1
    # NO resources block

# CORRECT — HPA can now calculate (actualCPU / requestedCPU)
spec:
  containers:
  - name: worker
    image: myapp:v1
    resources:
      requests:
        cpu: "100m"
        memory: "256Mi"
      limits:
        cpu: "500m"
        memory: "512Mi"
```

The lag problem: Metrics Server aggregates every 60 seconds. HPA polls every 15 seconds but evaluates a 3-sample window before scaling up (to avoid thrashing). That's already 3-5 minutes before pods are created. Then the new pods need to pull images and reach Ready state — add 30-90 seconds. You're at 4-6 minutes total. During a traffic spike on your notification platform, that's 4-6 minutes of message queue buildup.

**KEDA Mechanics**

KEDA runs a separate controller that polls your external trigger source (RabbitMQ queue depth, Kafka consumer lag, SQS queue length) directly. It extends the HPA API — KEDA creates an HPA object on your behalf, backed by external metrics. When queue depth is 0, KEDA can scale the deployment to 0 replicas entirely (scale-to-zero). When the first message arrives, KEDA scales to 1 before HPA's polling cycle would even notice.

**Critical insight**: KEDA doesn't replace HPA. It *creates* HPA objects dynamically:

```
KEDA ScaledObject (custom resource)
       ↓
    KEDA controller watches it
       ↓
    KEDA queries source (RabbitMQ, Kafka, SQS, etc.)
       ↓
    KEDA creates/updates an HPA object
       ↓
    Standard HPA controller scales the Deployment
```

**Karpenter vs Cluster Autoscaler**

Cluster Autoscaler (CA) works with Auto Scaling Groups — it watches for pending pods, picks a matching ASG, and asks AWS to add a node. Problems: it can only add node types pre-defined in the ASG, it doesn't bin-pack (it adds a whole node even if you only need 0.5 CPU), and it takes 2-3 minutes to provision.

Karpenter talks directly to EC2 Fleet API. It reads pending pod requirements (CPU, memory, GPU, architecture, topology) and provisions the exact instance type that fits. It bin-packs multiple pods onto single nodes, supports Spot interruption handling natively, and provisions in 30-60 seconds instead of 2-3 minutes.

## 5-Minute Version

**The 4-Minute Problem in Detail**

For a 25M msg/day platform, assume you have spiky traffic patterns — a batch job fires 500K messages in 2 minutes. Here's the timeline with vanilla HPA + Cluster Autoscaler:

- T+0: Spike begins. Queue starts growing.
- T+60: Metrics Server aggregates CPU across your consumer pods. Utilization is 300%.
- T+75: HPA polls, sees first elevated reading. Waits for confirmation window.
- T+150: HPA has 3 confirmatory readings. Calculates desired replicas: 3x current. Creates 6 new pending pods.
- T+160: Cluster Autoscaler notices pending pods. Picks ASG. Calls EC2.
- T+280: EC2 node joins cluster. Pods schedule. Image pull begins.
- T+340: Pods reach Running. Consumer lag starts shrinking.
- Result: ~6 minutes of unbounded queue growth. At 500K msg/2min, that's 1.5M messages queued during the lag.

**KEDA Solution**

With KEDA watching RabbitMQ directly:
- T+0: Spike begins.
- T+10: KEDA polls RabbitMQ API. Sees queue depth > threshold (e.g., 100 messages). Scales deployment from 2 to 8 replicas immediately.
- T+40: New pods reach Ready. Consuming begins.
- Result: 40 seconds, not 6 minutes. Maximum queue buildup: ~167K messages instead of 1.5M.

**In numbers at T-Mobile**:
- HPA alone: 500K msg/sec spike → detect at +3 min → queue depth = 1.5M messages
- KEDA + RabbitMQ queue scaler: detects queue length at +30 sec → scale at +1 min → queue depth = 167K messages
- **Result**: 90% reduction in queued messages, 10x lower message latency during spikes

**VPA: When to Use It**

VPA is not a runtime scaler in production — it cannot resize a running pod (it must evict and reschedule). Use VPA in recommendation mode only: let it observe your pods for 2-4 weeks, read the recommendations, and bake them into your deployment manifests. VPA is a right-sizing tool, not a traffic-response tool. Do NOT run HPA and VPA in resource-adjustment mode simultaneously on the same deployment — they will fight each other.

**The HPA + VPA Conflict (Eviction Storm)**

**Problem**: If HPA scales pods *while* VPA is updating resource requests, chaos ensues.

**Scenario**:
1. 10-pod Deployment with 256Mi request per pod = 2.56 Gi total
2. HPA scales to 20 pods during spike
3. VPA simultaneously updates requests from 256Mi to 512Mi per pod (based on week-old data)
4. Scheduler now sees: 20 pods × 512Mi = 10.24 Gi requested, but only 3 Gi available
5. **Result**: HPA-triggered pods get evicted as "unschedulable", pods keep thrashing

**Solution**: Use VPA in "off" or "recommendation" mode for dynamic workloads; apply recommendations manually during maintenance windows.

**Node Pool Strategy**

Production clusters need segregated node pools because different workloads have different requirements:

| Pool Name | Instance Type | Purpose | Spot? |
|---|---|---|---|
| system | m5.large (on-demand) | kube-system, monitoring, ingress | No |
| app-general | m5.xlarge mixed | Business logic pods | Mixed (70% Spot) |
| batch | c5.2xlarge Spot | One-off jobs, CI runners | Yes (100%) |
| memory-intensive | r5.xlarge | Cassandra clients, cache-heavy | No |
| gpu | g4dn.xlarge | ML inference | Spot OK |

**Extended Node Pool Strategy at Scale**:

| Pool Type | Instance Types | Spot? | Use Case | T-Mobile Example |
|-----------|---|---|---|---|
| **Baseline** | t3.large, t3.xlarge | Yes (50%) | General app workloads | Order processing, logs pipeline |
| **Memory-heavy** | r5.xlarge, r5.2xlarge | Yes (30%) | Cache, in-memory databases | Redis, Elasticsearch |
| **CPU-bound** | c5.2xlarge, c5.4xlarge | No | Compute-intensive batch | ML inference, analytics |
| **GPU** | g4dn.xlarge | Yes (if available) | ML/video | Model serving, transcoding |
| **Scale-to-zero** | t3.nano (1) | No | Cost baseline, health checks | Config server, monitoring agent |

Karpenter NodePool objects encode these constraints. Pods use nodeSelector or nodeAffinity to target pools. This prevents a batch job spike from stealing capacity from your latency-sensitive notification consumers.

**Karpenter NodePool Resource Definition**:

```yaml
apiVersion: karpenter.sh/v1beta1
kind: NodePool
metadata:
  name: default
spec:
  template:
    metadata:
      labels:
        workload-type: general
    spec:
      requirements:
      - key: kubernetes.io/arch
        operator: In
        values: ["amd64"]
      - key: karpenter.sh/capacity-type
        operator: In
        values: ["on-demand", "spot"]
      - key: node.kubernetes.io/instance-type
        operator: In
        values: ["t3.medium", "t3.large", "m5.large", "m5.xlarge"]
      nodeClassRef:
        name: default
  limits:
    cpu: 1000
    memory: 1000Gi
  consolidation:
    expireAfter: 30d
    consolidateAfter: 30s
---
apiVersion: karpenter.k8s.aws/v1beta1
kind: EC2NodeClass
metadata:
  name: default
spec:
  subnetSelector:
    karpenter.sh/discovery: "true"
  securityGroupSelector:
    karpenter.sh/discovery: "true"
  iamInstanceProfile: "KarpenterNodeInstanceProfile"
  tags:
    ManagedBy: "karpenter"
```

**Tradeoff: Scale-to-Zero Cold Start**

Scale-to-zero sounds great but has a cost. When KEDA scales your RabbitMQ consumer to 0 and a new message arrives, KEDA must: scale deployment to 1, Karpenter must provision a node if none is available, pod must schedule, image must pull (if not cached), JVM must warm up (if Java). For a Kotlin/Java service, that cold start can be 45-90 seconds. For a notification platform with SLA commitments, this is unacceptable.

**The Cold Start Problem at scale**:
- Image pull: 10-30 seconds
- Container start: 2-5 seconds
- App warmup (connection pools, cache hydration): 5-30 seconds
- **Total**: 20-60 seconds, depending on image size and app startup time

Solution: use `minReplicaCount: 1` in KEDA's ScaledObject during business hours, allow scale-to-zero only during overnight maintenance windows.

**Production scale-to-zero pattern with cron overlay**:

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: batch-processor
spec:
  scaleTargetRef:
    name: batch-processor
  minReplicaCount: 0      # Scale to zero during off-peak
  maxReplicaCount: 100
  triggers:
  - type: rabbitmq
    metadata:
      queueName: "batch-jobs"
      value: "20"
  fallback:
    failureThreshold: 5
    replicas: 2           # If metrics unavailable, keep 2 pods running
---
# Cron trigger: maintain 5 replicas 8am-6pm weekdays (business hours)
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: batch-processor-business-hours
spec:
  scaleTargetRef:
    name: batch-processor
  minReplicaCount: 5
  maxReplicaCount: 100
  triggers:
  - type: cron
    metadata:
      timezone: America/Chicago
      start: 0 8 * * 1-5   # 8am Monday-Friday
      end: 0 18 * * 1-5    # 6pm Monday-Friday
      desiredNumberOfReplicas: "5"
```

**Failure Mode: Metric Source Unavailability**

If your custom metrics adapter goes down (Prometheus crashloops, KEDA loses connection to RabbitMQ), HPA falls back to its last known metric or refuses to scale. KEDA has a `fallbackReplicas` setting for exactly this — if the trigger source is unavailable for N polls, scale to a safe minimum instead of staying at zero. Always configure this.

## Director/VP Lens

**IC Implementation**: Configure HPA manifests with correct resource requests (HPA is useless if pods have no resource requests set), install KEDA with Helm, write ScaledObjects per queue, tune `pollingInterval` and `cooldownPeriod` based on message processing time.

**Director Governance**:
- Require resource requests and limits on ALL production pods as a deployment gate (admission webhook or OPA policy). Without this, HPA has nothing to calculate against — this is a top cause of misconfigured autoscaling.
- Establish autoscaling SLOs: "Pod scale-up must complete within 60 seconds of trigger" — measure this with KEDA metrics and alert on violations.
- Node pool strategy is a cost governance decision as much as a reliability decision. Own it at the team level, not per-service. Set a policy: batch workloads must use Spot pools; Spot usage below 40% of total compute triggers a cost review.
- Karpenter's consolidation feature (bin packing running nodes) can trigger pod evictions. Ensure PodDisruptionBudgets are defined for all stateful and critical services before enabling consolidation. This is a governance gap many teams miss when migrating from Cluster Autoscaler.
- Run chaos experiments against autoscaling annually: simulate a 10x traffic spike in staging with real KEDA + Karpenter and measure actual scale-up time. Compare against SLO. Document in runbook.

**AI Lens**: In 2025-2026, Karpenter's ML-assisted disruption scoring and AWS's Spot placement score API allow predictive Spot selection (choosing AZs with lowest interruption probability). This is the beginning of AI-driven infrastructure placement — a Principal SRE should know this exists and be piloting it.

**Q1**: Your notification service has HPA configured with CPU target of 70%. During a spike, you notice HPA shows `TARGETS: 250%/70%` but pod count hasn't increased yet. What are the two most likely causes of this delay, and how do you distinguish between them?

## ANSWER KEY — Comprehension Check

**A1**: Two causes: (1) HPA's confirmation window — HPA requires 3 consecutive metric readings above threshold before scaling up (to prevent thrashing); the 15-second poll interval means up to 45 seconds of observation time before action. (2) The scale-up stabilization window (`--horizontal-pod-autoscaler-upscale-stabilization`, default 0s, but scale-down is 300s) — confirm by running `kubectl describe hpa <name>` and checking "Events" section. If events show "SuccessfulRescale" is absent, look at the "unable to scale" reason. To distinguish: `kubectl describe hpa` shows the last scale event time and reason. If it shows "DesiredReplicas: 8, CurrentReplicas: 2" with no recent event, it's in the observation window. If it shows "ScalingLimited: True", you've hit maxReplicas.

**A2**: VPA in auto mode works by evicting a pod and rescheduling it with new resource requests. HPA watches pod count. When VPA evicts a pod, HPA counts the temporary reduction in replicas as a scale event. HPA may scale UP to compensate for the evicted pod, then VPA evicts the new pod to resize it, triggering another HPA reaction — an eviction storm. The correct pattern: run VPA in recommendation mode only (`updateMode: "Off"`). Read VPA recommendations weekly. Apply them to the deployment manifest manually. Use HPA for pod count management. If you need dynamic memory adjustment, use VPA for memory limits only and HPA for replicas — set `resourcePolicy` in VPA to exclude CPU so HPA and VPA don't overlap on the same resource.

**A3**: Business case: Cluster Autoscaler has two cost problems. First, it adds nodes in ASG-defined increments — if your ASG uses m5.xlarge (4 CPU) and a pod needs 0.25 CPU, CA adds a full node, you waste 3.75 CPU. Over 50 nodes, that's 15-20% wasted compute. At $0.19/hr per m5.xlarge, that's ~$700/month in waste for a mid-size cluster. Karpenter bins 8-12 small pods onto one node. Second, CA requires pre-defined Spot ASGs per instance type — if your chosen Spot type is interrupted, CA has no fallback. Karpenter uses EC2 Fleet with `price-capacity-optimized`, automatically diversifying across 15+ instance types, reducing Spot interruption rate from ~8% to ~2%. Reliability: CA node provisioning takes 2-3 minutes; Karpenter takes 30-60 seconds. For a notification platform, that 90-second difference during a spike is directly measurable as error budget burn.

### Step 2: Create a namespace and deploy a CPU-Bound Test Application

```bash
# Create namespace
kubectl create namespace scaling-lab
kubectl config set-context --current --namespace=scaling-lab

# Create the deployment manifest
cat <<'EOF' > /tmp/php-apache.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: php-apache
  namespace: scaling-lab
spec:
  selector:
    matchLabels:
      run: php-apache
  replicas: 1
  template:
    metadata:
      labels:
        run: php-apache
    spec:
      containers:
      - name: php-apache
        # This image runs a simple CPU-intensive PHP server
        image: registry.k8s.io/hpa-example
        ports:
        - containerPort: 80
        resources:
          # CRITICAL: HPA needs resource requests to calculate utilization %
          # Without requests, HPA cannot compute "current % of target"
          requests:
            cpu: 200m      # 200 millicores = 0.2 vCPU
          limits:
            cpu: 500m      # Hard cap at 0.5 vCPU
---
apiVersion: v1
kind: Service
metadata:
  name: php-apache
  namespace: scaling-lab
spec:
  ports:
  - port: 80
  selector:
    run: php-apache
EOF

kubectl apply -f /tmp/php-apache.yaml

# Wait for pod to be Running
kubectl rollout status deployment/php-apache -n scaling-lab
```

**EXPECTED_OUTPUT**:
```
deployment.apps/php-apache created
service/php-apache created
deployment "php-apache" successfully rolled out
```

**WHY**: The `requests.cpu: 200m` line is not optional — it's the denominator in HPA's utilization calculation. A pod consuming 150m CPU with a 200m request = 75% utilization. With no request set, HPA shows `<unknown>` and refuses to scale.

### Step 3: Create the HPA Object

```bash
# Create HPA targeting 50% CPU utilization
# Scale between 1 and 10 replicas
cat <<'EOF' | kubectl apply -f -
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: php-apache
  namespace: scaling-lab
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: php-apache
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50
EOF

# Verify HPA was created
kubectl get hpa php-apache -n scaling-lab
```

**EXPECTED_OUTPUT** (within 30 seconds of creation):
```
NAME         REFERENCE               TARGETS   MINPODS   MAXPODS   REPLICAS   AGE
php-apache   Deployment/php-apache   0%/50%    1         10        1          15s
```

Note: It may show `<unknown>/50%` for the first 60 seconds while Metrics Server collects initial data. This is normal.

**WHY**: The HPA object watches the php-apache Deployment. When current CPU % exceeds 50%, it will scale up. When it drops significantly below 50%, it will scale down (with a 5-minute cooldown to prevent thrashing).

### Step 10: Create the KEDA ScaledObject for RabbitMQ

The ScaledObject is the core KEDA configuration. It tells KEDA: watch THIS trigger, scale THIS deployment, using THESE thresholds.

```bash
cat <<'EOF' > /tmp/keda-scaledobject.yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: rabbitmq-scaledobject
  namespace: scaling-lab
spec:
  # Which deployment to scale
  scaleTargetRef:
    name: rabbitmq-consumer

  # Scale between 0 and 10 replicas
  minReplicaCount: 0
  maxReplicaCount: 10

  # Wait 30s before scaling down (tune per workload SLA)
  cooldownPeriod: 30

  # Poll the trigger every 5 seconds (much faster than HPA's 15s)
  pollingInterval: 5

  # If RabbitMQ is unreachable, scale to 1 (don't stay at zero during outages)
  fallback:
    failureThreshold: 3
    replicas: 1

  triggers:
  - type: rabbitmq
    metadata:
      # Connection details for RabbitMQ management API
      host: "http://admin:training123@rabbitmq.scaling-lab.svc.cluster.local:15672"
      # Which queue to monitor
      queueName: "notifications"
      # Scale up 1 pod for every 10 messages waiting in the queue
      # 10 messages waiting = 1 pod; 50 messages = 5 pods
      queueLength: "10"
      # Use the management HTTP API (not AMQP protocol)
      protocol: "http"
EOF

kubectl apply -f /tmp/keda-scaledobject.yaml

# Verify ScaledObject was created and KEDA accepted it
kubectl get scaledobject rabbitmq-scaledobject -n scaling-lab
```

**EXPECTED_OUTPUT**:
```
NAME                      SCALETARGETKIND      SCALETARGETNAME     MIN   MAX   TRIGGERS    AUTHENTICATION   READY   ACTIVE   FALLBACK   PAUSED    AGE
rabbitmq-scaledobject     apps/v1.Deployment   rabbitmq-consumer   0     10    rabbitmq                     True    False    False      Unknown   15s
```

Key columns:
- `READY: True` — KEDA successfully connected to RabbitMQ and can read metrics
- `ACTIVE: False` — No messages in queue, deployment is at 0 replicas
- `FALLBACK: False` — RabbitMQ is reachable, no fallback triggered

**WHY**: The `queueLength: "10"` parameter is the key tuning knob. It means "desired messages per consumer pod." With 50 messages queued, KEDA calculates 50/10 = 5 desired pods. This is directly equivalent to your T-Mobile capacity planning metric: messages per consumer throughput.

## Tradeoffs and Failure Modes

**1. Scale-to-Zero Cold Start vs. SLO Compliance**

| Scenario | Strategy | Cost Saving | Cold Start Latency | Best For |
|----------|----------|-------------|---|---|
| **Always-hot** | minReplicaCount=2 | $0 | 0s (already running) | Customer-facing APIs, SLA-critical |
| **Cron warm** | Cron trigger 5 min before peak | 80% savings | 35s (pods warming up) | Batch jobs, predictable spikes |
| **Scale-to-zero** | minReplicaCount=0, cold start | 95% savings | 60s (image pull + app start) | Dev/test, occasional on-demand reports |

The tradeoff: scale-to-zero saves compute cost (0 replicas = 0 pod cost) but introduces a cold start latency when the first message arrives. For a p99 SLO < 500ms, that first message will violate SLO. Resolution: use `minReplicaCount: 1` during business hours via KEDA's scheduled scaling (`type: cron`), allow scale-to-zero overnight. This captures 60% of the cost savings while maintaining SLO during operational hours. Interview answer: "Always tie scaling configuration to SLO math, not just cost intuition."

**2. Karpenter Consolidation vs. Stateful Workload Disruption**

Karpenter's consolidation mode continuously bin-packs nodes by evicting pods and rescheduling them to free up underutilized nodes. For stateless pods with PodDisruptionBudgets, this is safe. For stateful pods (Cassandra, Redis, RabbitMQ itself), unexpected evictions can cause quorum loss or data unavailability.

| Workload Type | Consolidation Safe? | Recommendation |
|---|---|---|
| Stateless APIs | Yes | Use Karpenter, high bin-packing |
| Databases (single pod) | No | Exclude from consolidation (tolerations) |
| Kafka brokers | Conditional | Use PodDisruptionBudget, allow 1 drain at a time |
| Caches (Redis) | Conditional | Use high-availability mode (Redis Cluster), multi-pod |

The failure mode: a team enables consolidation without PDBs on their stateful set, Karpenter evicts two of three Cassandra pods simultaneously (because no PDB limits simultaneous disruptions), and the ring loses majority quorum. Mitigation: require PDBs as a deployment gate; configure Karpenter's `disruption.budgets` at the NodePool level as a backstop.

**3. Custom Metrics Adapter as Single Point of Failure**

HPA with custom metrics (or KEDA) depends on an external metric source being available. If your Prometheus adapter crashes, or KEDA loses connectivity to RabbitMQ, HPA falls back to the last known metric value or enters a degraded state. In the worst case, if pods are at scale-to-zero and the metric source is unavailable, KEDA's fallback replicas kick in — but if `fallback` is not configured, the deployment stays at 0 indefinitely while messages accumulate. This is a silent failure: no errors, no alerts, just growing queue depth.

Mitigation:
1. Always run metrics-server as a DaemonSet with anti-affinity (multiple nodes)
2. For KEDA, use multiple replicas: `replicas: 2` in Helm values
3. Use fallback minReplicaCount: if metrics unavailable, maintain at least 2 pods
4. Monitor metrics adapter availability as a critical metric
5. Alert on queue depth independent of autoscaling (so queue growth always pages you regardless of whether autoscaling is functioning)

## Confidence Check: 3-Question Quiz

**Q1**: You have a Deployment with `resources.requests: {cpu: "100m"}` and a 200m CPU limit. HPA targets 50% utilization. How many pods needed if actual CPU usage is 75m?

- A) Cannot calculate — no target metric defined
- B) 1.5 pods → rounds to 2 pods (75m / 50m target)
- C) Depends on container limit
- **Answer**: B. Formula: `desiredReplicas = ceil(currentMetric / targetMetric) = ceil((75m / 100m requests) / 0.50) = ceil(1.5) = 2`. Limits are ceilings for safety, not scaling triggers.

**Q2**: KEDA watches a RabbitMQ queue and creates an HPA with `value: "10"` (10 msgs per pod). Queue depth is 150. How many pods will KEDA scale to?

- A) 15 pods (150 / 10)
- B) 10 pods (hard limit)
- C) Depends on maxReplicaCount setting
- **Answer**: C (and A if maxReplicaCount ≥ 15). KEDA computes desired replicas = `ceil(queue_depth / value)`. Without maxReplicaCount, answer is 15. If maxReplicaCount: 10, answer is 10.

**Q3**: During a 5-second cloud provider outage, your metrics-server pod is evicted, restarted on another node. What happens to HPA?

- A) HPA immediately scales to maxReplicas
- B) HPA continues scaling based on last known metrics (cached)
- C) HPA stops polling, pods remain at current replica count
- **Answer**: C. HPA reads metrics-server API. If API unavailable, HPA doesn't scale. Pods stay frozen at current count. If load rises, pods don't scale up (latency spike). This is why you need fallback minReplicaCount and metrics-server redundancy.
