# Secure Multi-Tenant EKS Cluster | Week 3 Weekend

## Why This Matters

Multi-tenancy on Kubernetes is one of the top 3 questions at Staff/Principal interviews because it exposes whether you understand the full security surface: network, IAM, compute quotas, and blast radius containment. At T-Mobile scale (25M msg/day), a misconfigured RBAC or missing NetworkPolicy is a P1 waiting to happen — getting this right is the difference between a platform engineers trust and one they route around. Interviewers use this to separate "ran kubectl at a company" from "designed a platform other teams build on."

---

## Analogy

Think of a multi-tenant EKS cluster like a shared office building with separate suites.

- The **building** = the EKS cluster (shared control plane, shared nodes)
- Each **tenant suite** = a Kubernetes namespace (locked door, separate mailbox)
- The **keycard system** = RBAC (only your badge opens your suite)
- The **fire suppression zones** = NetworkPolicy (a fire in Suite A cannot spread to Suite B through shared HVAC)
- The **IRSA role** = a suite-specific master key that opens only the AWS resources that tenant rented (their S3 bucket, their SQS queue) — the building manager cannot use one tenant's key to access another's AWS storage room

When a tenant's pod runs, it presents its ServiceAccount token to AWS STS. AWS checks: "Does this Kubernetes identity map to this IAM role?" If yes, only that role's permissions apply. Tenant A's pod literally cannot call Tenant B's AWS API even if someone misconfigures kubectl RBAC — the IAM layer is a second independent lock.

## 30-Second Version

Multi-tenant EKS means multiple teams share one cluster with hard isolation guarantees. The four pillars are: namespace boundaries (blast radius), RBAC (who can touch what), NetworkPolicy (pod-to-pod firewall), and IRSA (each tenant's AWS identity is independent). Done correctly, Tenant A cannot read Tenant B's secrets, consume Tenant B's compute quota, reach Tenant B's pods on the network, or call Tenant B's AWS services — even if a dev makes a mistake in their own namespace.

## 2-Minute Version

**Why not just give each team a separate cluster?**

Cost and operational overhead. A dedicated EKS cluster costs $0.10/hr for the control plane alone, plus EC2 nodes. Ten teams = ten control planes to patch, ten VPCs to manage, ten sets of ALB ingress controllers. The platform team's toil scales linearly with team count. Shared cluster with namespace isolation gives you 80% of the security isolation at 20% of the operational cost — the remaining 20% of isolation (node-level) is addressed by dedicated node groups per tenant if the threat model demands it.

**The four isolation layers in this lab:**

1. **Namespace** — Kubernetes API boundary. A `kubectl get pods` in namespace `notification-core` returns nothing from `notification-analytics`. Most misconfigurations are caught here.

2. **RBAC** — `Role` defines what verbs (get, list, create) are allowed on what resources (pods, secrets) within a namespace. `RoleBinding` says "this ServiceAccount or group gets this Role." A developer bound to `notification-core` namespace gets zero API access to `notification-analytics` namespace — Kubernetes API server enforces this before any request hits etcd.

3. **NetworkPolicy** — By default, all pods can talk to all pods across namespaces. NetworkPolicy is an explicit allowlist at the CNI layer (not iptables — the CNI plugin, e.g., Calico, enforces it). You must install a NetworkPolicy-capable CNI (EKS uses VPC CNI with Calico or Cilium overlay for policy enforcement). This lab uses Calico.

4. **IRSA** — Each namespace gets a dedicated Kubernetes ServiceAccount annotated with an IAM role ARN. The EKS OIDC provider issues signed JWTs that AWS STS validates. The IAM role's trust policy says "only tokens from this specific ServiceAccount in this specific namespace can assume me." Tenant A's pod cannot assume Tenant B's IAM role even if RBAC were misconfigured — the JWT subject claim would not match.

## 5-Minute Version

**Tradeoffs of namespace-based vs. cluster-per-tenant isolation:**

| Dimension | Shared Cluster (Namespace) | Dedicated Cluster |
|---|---|---|
| Cost | Low — shared control plane, shared nodes | High — $0.10/hr per control plane x N teams |
| Blast radius | Medium — noisy neighbor on nodes, shared etcd | Low — complete isolation |
| Ops toil | Low — one cluster to patch | High — N clusters to patch, upgrade, monitor |
| Compliance fit | Works for PCI DSS with NetworkPolicy + PSS | Required for some FedRAMP boundaries |
| Node-level isolation | No (unless dedicated node groups) | Yes |
| Multi-cluster tooling | Not needed | Requires Fleet/ArgoCD multi-cluster |

**Failure modes to know for interviews:**

1. **RBAC ClusterRole leak**: A developer creates a `ClusterRoleBinding` instead of `RoleBinding` — now their service account has cluster-wide access. Solution: OPA/Gatekeeper policy that blocks `ClusterRoleBinding` creation by non-admins.

2. **NetworkPolicy gap**: CNI plugin is VPC CNI without Calico. NetworkPolicy objects exist in etcd but are never enforced — pods can talk cross-namespace and no one knows. Solution: verify policy enforcement with `kubectl exec` + `curl` from Tenant A pod to Tenant B service, confirm it's dropped.

3. **IRSA misconfiguration — overly broad trust policy**: IAM role trust policy uses `StringLike` with wildcard on the namespace condition. Tenant B's pod can assume Tenant A's role. Solution: use `StringEquals` on the full subject: `system:serviceaccount:notification-core:core-sa`.

4. **ResourceQuota bypass via DaemonSet**: DaemonSets in a namespace count against ResourceQuota for requests/limits but the pod count can exceed `pods` quota if quota is not set. A rogue DaemonSet can consume all node capacity. Solution: set `pods` count in ResourceQuota AND restrict DaemonSet creation via RBAC (only cluster-admins can create DaemonSets).

5. **PodSecurity admission gap**: Namespace label missing or set to `warn` instead of `enforce`. Privileged pods run, can escape to node filesystem. Solution: enforce `restricted` or `baseline` at namespace creation time via LimitRange + PSS label, validated by OPA.

**Design decision record for ARCHITECTURE.md:**
The choice in this lab is namespace isolation (not cluster-per-tenant) because: (a) notification-core and notification-analytics are owned by the same org, (b) neither handles PCI/PHI data that requires hard node boundary, (c) the platform team is 15 engineers managing a high-throughput system — operational simplicity wins. The tradeoff accepted: noisy neighbor risk on nodes is mitigated by ResourceQuota + LimitRange. If a third tenant with PCI scope is onboarded, they get a dedicated node group with taints/tolerations.

**How a VP governs this vs. how an IC implements it:**

| Dimension | IC (you, hands-on) | Director/VP (you, in 18 months) |
|---|---|---|
| Daily concern | "Does this NetworkPolicy block the right ports?" | "Do we have a tenant onboarding SLA? Is it 2 days or 2 weeks?" |

| Security posture | Apply PodSecurityStandards label | Set policy: all new namespaces must pass CIS Benchmark before first deployment — automated gate |
| Incident scope | Debug why pod A can reach pod B | "What is our blast radius if one tenant is compromised? Do we have network egress logging?" |
| Team scaling | Configure one tenant module | "The tenant Terraform module is the golden path — no team provisions their own namespace outside it" |
| AI leverage | Copilot assists writing Terraform | "AI-generated tenant configs must pass OPA policy gates before merge — we audit, not approve manually" |

**The org design answer (for "design a multi-tenant cluster for 10 teams"):**

Three-tier RBAC model: cluster-admins (platform SRE only), namespace-admins (per-team tech lead), developers (per-team). Tenant onboarding is a Terraform PR to the tenant module — reviewed by platform SRE, merged by CI/CD. No manual kubectl for onboarding. Cost allocation via namespace labels + Kubecost or AWS Cost Allocation Tags. Quarterly RBAC audit: automated report of all ClusterRoleBindings, any non-platform principal with cluster-wide access triggers a ticket.

## Comprehension Check

Answer these before starting the lab. You need 2 of 3 correct to proceed.

**Q1.** A developer on the `notification-analytics` team runs `kubectl get secrets -n notification-core`. Explain what happens at each enforcement layer (API server, RBAC, NetworkPolicy) and which layer stops the request.

**Q2.** Your security team says "we need to ensure Tenant A's pods cannot call Tenant B's AWS S3 bucket even if a developer hard-codes Tenant B's bucket name in their app." IRSA is configured correctly. Will IRSA alone prevent this? Explain why or why not, and what additional control you would add.

**Q3.** You provision a new namespace for a third tenant and forget to apply a NetworkPolicy. The namespace has RBAC configured correctly. What is the security gap, and what is the blast radius if one of that tenant's pods is compromised?

---

## ANSWER KEY — Comprehension Check

**A1.** The request hits the Kubernetes API server first. The API server runs the authorization chain: authentication (who is this user?) → RBAC authorization (are they allowed `get secrets` in `notification-core`?). The developer's RoleBinding is in `notification-analytics` namespace only — they have zero permissions in `notification-core`. The RBAC layer returns `403 Forbidden` before the request reaches etcd. NetworkPolicy is irrelevant here because this is an API server request (kubectl → kube-apiserver), not pod-to-pod network traffic. NetworkPolicy only applies to pod network traffic.

**A2.** IRSA alone will prevent the S3 call IF the IAM role for `notification-analytics` does not have `s3:GetObject` permission on Tenant A's bucket. The IAM policy on the role is the enforcement point — not the bucket name in the app code. If the IAM role is correctly scoped (only Tenant B's bucket ARN), then even with the hard-coded bucket name, the AWS STS call returns `AccessDenied`. However, IRSA does not prevent the pod from making the API call attempt — it just fails at AWS. Additional control: S3 bucket policy on Tenant A's bucket with an explicit `Deny` for any principal that is not `notification-core`'s IAM role. Defense in depth: deny at both the role policy and the resource policy.

**A3.** The security gap: all pods in that namespace can send and receive traffic to/from any pod in any other namespace in the cluster. Kubernetes default is allow-all. RBAC controls API access (kubectl operations) but has zero effect on pod-to-pod network traffic. Blast radius if one pod is compromised: the attacker has full network reachability to every pod in the cluster — can probe internal services, reach databases, hit the Kubernetes API server from inside the cluster network, attempt to reach cloud metadata endpoint (169.254.169.254) to steal node IAM credentials (mitigated by IMDSv2 hop limit but still a vector). The fix: apply a default-deny-all NetworkPolicy to the namespace immediately on creation, then add explicit allow rules for required traffic.

## MODULE 1 — Provision EKS + Tenant Configs (60 min)

### Step 2: Write the EKS cluster Terraform module

```bash
# Create the EKS cluster module — this provisions the cluster and OIDC provider
cat > ~/training/week3-weekend/terraform/eks-cluster/main.tf << 'EOF'
# ============================================================
# eks-cluster/main.tf
# Provisions: EKS cluster, managed node group, OIDC provider
# Why: OIDC provider is the trust bridge between Kubernetes
#      ServiceAccounts and AWS IAM — required for IRSA.
# ============================================================

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

# ------------------------------------------------------------------
# DATA SOURCES — look up existing AWS account/region context
# ------------------------------------------------------------------

data "aws_caller_identity" "current" {}
# Why: We need the account ID to construct IAM ARNs later

data "aws_region" "current" {}
# Why: Region-aware resource naming without hardcoding

# ------------------------------------------------------------------
# VPC — minimal VPC with public+private subnets for EKS
# We use the terraform-aws-modules/vpc module to avoid 200 lines
# of subnet/route-table boilerplate.
# ------------------------------------------------------------------

module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"

  name = "${var.cluster_name}-vpc"
  cidr = "10.0.0.0/16"

  azs             = ["${data.aws_region.current.name}a", "${data.aws_region.current.name}b"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24"]

  # NAT gateway lets private-subnet nodes reach the internet
  # (for pulling ECR images, calling AWS APIs)
  enable_nat_gateway   = true
  single_nat_gateway   = true   # Cost optimization: one NAT for training

  # These tags are REQUIRED by EKS to discover subnets for load balancers
  public_subnet_tags = {
    "kubernetes.io/role/elb" = "1"
  }
  private_subnet_tags = {
    "kubernetes.io/role/internal-elb" = "1"
  }

  tags = var.tags
}

# ------------------------------------------------------------------
# EKS CLUSTER — managed control plane
# ------------------------------------------------------------------

module "eks" {
  source  = "terraform-aws-modules/eks/aws"
  version = "~> 20.0"

  cluster_name    = var.cluster_name
  cluster_version = "1.29"

  # Private endpoint: nodes talk to API server inside VPC
  # Public endpoint: you can run kubectl from your laptop
  cluster_endpoint_public_access  = true
  cluster_endpoint_private_access = true

  vpc_id     = module.vpc.vpc_id
  subnet_ids = module.vpc.private_subnets

  # OIDC provider — this is what makes IRSA possible.
  # When enabled, EKS creates an OIDC issuer URL.
  # AWS IAM can then trust JWTs signed by that issuer.
  enable_irsa = true

  # Managed node group — EKS manages the EC2 lifecycle
  eks_managed_node_groups = {
    shared = {
      instance_types = ["t3.medium"]   # 2 vCPU, 4GB — fits training budget
      min_size       = 2               # 2 nodes minimum for HA during lab
      max_size       = 4
      desired_size   = 2

      # Why t3.medium: large enough to run 4-6 pods per node,
      # small enough to stay in free tier adjacent territory
    }
  }

  # Allow kubectl from your WSL2 IP (or open for training — lock down in prod)
  cluster_additional_security_group_ids = []

  tags = var.tags
}

# ------------------------------------------------------------------
# OUTPUTS — values the tenant module and manifests will reference
# ------------------------------------------------------------------

output "cluster_name" {
  value = module.eks.cluster_name
}

output "cluster_endpoint" {
  value = module.eks.cluster_endpoint
}

output "oidc_provider_arn" {
  value = module.eks.oidc_provider_arn
  # Why: Tenant module uses this ARN in IAM role trust policies
}

output "oidc_provider_url" {
  value = module.eks.cluster_oidc_issuer_url
}

output "account_id" {
  value = data.aws_caller_identity.current.account_id
}

output "region" {
  value = data.aws_region.current.name
}
EOF
```

### Step 3: Write the Terraform tenant module

```bash
cat > ~/training/week3-weekend/terraform/tenant/main.tf << 'EOF'
# ============================================================
# tenant/main.tf
# Provisions: namespace, RBAC, ResourceQuota, LimitRange,
#             NetworkPolicy, IRSA IAM role
# This module is called ONCE PER TENANT — same code, different vars.
# Why: Golden path — every tenant gets identical security posture.
# ============================================================

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
    kubernetes = {
      source  = "hashicorp/kubernetes"
      version = "~> 2.0"
    }
  }
}

# ------------------------------------------------------------------
# NAMESPACE — the isolation boundary
# PodSecurity label enforces the restricted standard:
#   - no privileged containers
#   - no host network/PID
#   - read-only root filesystem (enforced)
# ------------------------------------------------------------------

resource "kubernetes_namespace" "tenant" {
  metadata {
    name = var.tenant_name

    labels = {
      # PodSecurity admission — "restricted" blocks privilege escalation
      # "enforce" = reject the pod; "warn" = allow but warn; "audit" = log only
      "pod-security.kubernetes.io/enforce" = "baseline"
      # Using "baseline" not "restricted" because notification apps
      # typically need some capabilities (e.g., bind to port < 1024 in some configs)
      # Change to "restricted" if your app supports it

      "app.kubernetes.io/managed-by" = "terraform"
      "tenant"                        = var.tenant_name
    }

    annotations = {
      "cost-center" = var.cost_center
      # Why: Platform team uses these annotations for showback/chargeback reports
    }
  }
}

# ------------------------------------------------------------------
# RESOURCE QUOTA — circuit breaker on compute consumption
# Tenant cannot consume more than this even if nodes have capacity.
# Why: One misconfigured deployment cannot starve other tenants.
# ------------------------------------------------------------------

resource "kubernetes_resource_quota" "tenant" {
  metadata {
    name      = "${var.tenant_name}-quota"
    namespace = kubernetes_namespace.tenant.metadata[0].name
  }

  spec {
    hard = {
      "requests.cpu"    = var.quota_cpu_requests
      "requests.memory" = var.quota_memory_requests
      "limits.cpu"      = var.quota_cpu_limits
      "limits.memory"   = var.quota_memory_limits
      "pods"            = var.quota_pod_count
      # Why pod count: prevents DaemonSet abuse from exhausting node capacity
    }
  }
}

# ------------------------------------------------------------------
# LIMIT RANGE — default QoS for pods that don't set their own limits
# Why: Without this, a pod with no resource requests gets Burstable QoS
#      and is the first evicted when nodes are under pressure.
# ------------------------------------------------------------------

resource "kubernetes_limit_range" "tenant" {
  metadata {
    name      = "${var.tenant_name}-limits"
    namespace = kubernetes_namespace.tenant.metadata[0].name
  }

  spec {
    limit {
      type = "Container"
      default = {
        cpu    = "200m"    # Default limit if pod doesn't specify
        memory = "256Mi"
      }
      default_request = {
        cpu    = "100m"    # Default request — what scheduler uses for placement
        memory = "128Mi"
      }
    }
  }
}

# ------------------------------------------------------------------
# RBAC — Role (what's allowed) + RoleBinding (who gets it)
# Developer role: read/write their own namespace resources
# No ClusterRole — no cross-namespace access
# ------------------------------------------------------------------

resource "kubernetes_role" "developer" {
  metadata {
    name      = "${var.tenant_name}-developer"
    namespace = kubernetes_namespace.tenant.metadata[0].name
  }

  rule {
    api_groups = ["", "apps", "batch"]
    resources  = ["pods", "deployments", "services", "configmaps", "jobs"]
    verbs      = ["get", "list", "watch", "create", "update", "patch"]
    # No "delete" — deletions require tech lead approval (ops discipline)
  }

  rule {
    api_groups = [""]
    resources  = ["pods/log", "pods/exec"]
    verbs      = ["get", "list"]
    # pods/exec = kubectl exec — read-only (get/list) blocks interactive shells
    # In prod: remove exec entirely, use ephemeral debug containers with approval
  }

  rule {
    api_groups = [""]
    resources  = ["secrets"]
    verbs      = ["get", "list"]
    # Secrets readable but not writable — secrets are managed by Vault/IRSA
    # In prod: remove secrets entirely, use CSI secrets driver
  }
}

resource "kubernetes_role_binding" "developer" {
  metadata {
    name      = "${var.tenant_name}-developer-binding"
    namespace = kubernetes_namespace.tenant.metadata[0].name
  }

  role_ref {
    api_group = "rbac.authorization.k8s.io"
    kind      = "Role"
    name      = kubernetes_role.developer.metadata[0].name
  }

  # Bind to the tenant's dedicated group — map your IdP groups here
  subject {
    kind      = "Group"
    name      = "${var.tenant_name}-developers"
    api_group = "rbac.authorization.k8s.io"
  }
}

# ------------------------------------------------------------------
# NETWORK POLICY — explicit allowlist; default deny everything
# This is the pod-level firewall.
# Step 1: deny all ingress and egress by default
# Step 2: allow specific traffic
# ------------------------------------------------------------------

# Rule 1: Default deny all ingress to this namespace
resource "kubernetes_network_policy" "default_deny_ingress" {
  metadata {
    name      = "default-deny-ingress"
    namespace = kubernetes_namespace.tenant.metadata[0].name
  }

  spec {
    pod_selector {}   # Empty selector = applies to ALL pods in namespace

    policy_types = ["Ingress"]
    # No ingress rules = deny all ingress
  }
}

# Rule 2: Default deny all egress from this namespace
resource "kubernetes_network_policy" "default_deny_egress" {
  metadata {
    name      = "default-deny-egress"
    namespace = kubernetes_namespace.tenant.metadata[0].name
  }

  spec {
    pod_selector {}
    policy_types = ["Egress"]
    # No egress rules = deny all egress
  }
}

# Rule 3: Allow pods within the SAME namespace to talk to each other
resource "kubernetes_network_policy" "allow_same_namespace" {
  metadata {
    name      = "allow-same-namespace"
    namespace = kubernetes_namespace.tenant.metadata[0].name
  }

  spec {
    pod_selector {}
    policy_types = ["Ingress", "Egress"]

    ingress {
      from {
        # Only allow traffic FROM pods in the same namespace
        pod_selector {}
        # No namespaceSelector = same namespace only
      }
    }

    egress {
      to {
        pod_selector {}
      }
    }
  }
}

# Rule 4: Allow egress to kube-dns (port 53) — pods need DNS resolution
resource "kubernetes_network_policy" "allow_dns" {
  metadata {
    name      = "allow-dns-egress"
    namespace = kubernetes_namespace.tenant.metadata[0].name
  }

  spec {
    pod_selector {}
    policy_types = ["Egress"]

    egress {
      ports {
        protocol = "UDP"
        port     = "53"
      }
      ports {
        protocol = "TCP"
        port     = "53"
      }
      # No `to` selector = allows DNS egress to any destination
      # In prod: scope to kube-dns namespace only
    }
  }
}

# Rule 5: Allow egress to AWS APIs (HTTPS port 443)
# Why: IRSA needs pods to call sts.amazonaws.com and service-specific endpoints
resource "kubernetes_network_policy" "allow_aws_egress" {
  metadata {
    name      = "allow-aws-api-egress"
    namespace = kubernetes_namespace.tenant.metadata[0].name
  }

  spec {
    pod_selector {}
    policy_types = ["Egress"]

    egress {
      ports {
        protocol = "TCP"
        port     = "443"
      }
      # Allows HTTPS to any IP — in prod, scope to VPC endpoint IPs or
      # use AWS VPC endpoints to keep traffic off public internet
    }
  }
}

# ------------------------------------------------------------------
# SERVICE ACCOUNT — the Kubernetes identity for tenant pods
# Annotated with the IAM role ARN — this is what triggers IRSA
# ------------------------------------------------------------------

resource "kubernetes_service_account" "tenant" {
  metadata {
    name      = "${var.tenant_name}-sa"
    namespace = kubernetes_namespace.tenant.metadata[0].name

    annotations = {
      # This annotation is the IRSA trigger.
      # EKS mutating webhook sees this and injects AWS_* env vars into pods
      # that reference this ServiceAccount.
      "eks.amazonaws.com/role-arn" = aws_iam_role.tenant.arn
    }
  }
}

# ------------------------------------------------------------------
# IRSA IAM ROLE — tenant's AWS identity
# Trust policy: ONLY tokens from THIS ServiceAccount in THIS namespace
# ------------------------------------------------------------------

resource "aws_iam_role" "tenant" {
  name = "${var.cluster_name}-${var.tenant_name}-irsa"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Principal = {
          Federated = var.oidc_provider_arn
        }
        Action = "sts:AssumeRoleWithWebIdentity"
        Condition = {
          StringEquals = {
```

### Step 10: Verify NetworkPolicy blocks cross-namespace pod traffic

```bash
# Get the analytics service ClusterIP
ANALYTICS_SVC_IP=$(kubectl get service notification-analytics-svc \
  -n notification-analytics \
  -o jsonpath='{.spec.clusterIP}')

echo "Analytics service IP: $ANALYTICS_SVC_IP"

# Try to reach notification-analytics service FROM notification-core pod
# This should time out because NetworkPolicy blocks cross-namespace ingress
kubectl exec -n notification-core $CORE_POD -- \
  sh -c "timeout 5 curl -s http://$ANALYTICS_SVC_IP:8080 || echo 'CONNECTION BLOCKED - NetworkPolicy working'"
```

**EXPECTED_OUTPUT**:
```
CONNECTION BLOCKED - NetworkPolicy working
```

(The curl times out after 5 seconds and the fallback message prints.)

**WHY**: The `default-deny-ingress` NetworkPolicy in `notification-analytics` drops all packets from outside the namespace. No response means the NetworkPolicy is enforced at the CNI layer — traffic never reached the analytics pod. This is the separation that RBAC alone cannot provide.

**EXPECTED_OUTPUT**:

### Step 17: ARCHITECTURE.md — Multi-tenancy decision record

```bash
cat > ~/training/week3-weekend/ARCHITECTURE.md << 'EOF'
# Multi-Tenancy Architecture Decision Record
# Notification Platform — EKS Cluster Design
# Author: Vishweshwar Chippa | Week 3 Weekend Lab

## Decision: Namespace Isolation vs. Cluster-per-Tenant

### Status: Accepted

### Context

The notification platform serves two internal tenants:
- **notification-core**: Real-time message dispatch, 25M msgs/day, latency-sensitive
- **notification-analytics**: Async event processing, batch workloads, not latency-critical

Both are owned by the same SRE team. Neither processes PCI, PHI, or FedRAMP-scoped data.
The platform team manages 15 engineers across SRE and development functions.

### Decision

**Namespace isolation within a shared EKS cluster**, not separate clusters per tenant.

### Rationale

| Factor | Namespace Isolation | Cluster-per-Tenant |
|---|---|---|
| Control plane cost | $0.10/hr (shared) | $0.20/hr (2x control planes) |
| Operational toil | 1 cluster to upgrade | 2 clusters to upgrade |
| Blast radius (app layer) | Isolated via RBAC + NetworkPolicy | Fully isolated |
| Blast radius (node layer) | Shared nodes (mitigated by ResourceQuota) | Fully isolated |
| Node-level isolation | Not required (same org, same trust boundary) | Required only for PCI/FedRAMP |
| CI/CD complexity | Single kubeconfig | Two kubeconfigs, two ArgoCD targets |
| Time to onboard new tenant | 1 Terraform PR | New cluster + VPC + peering |

### Risks Accepted

1. **Noisy neighbor on nodes**: Mitigated by ResourceQuota (CPU/memory ceiling per tenant) and LimitRange (per-pod defaults). Monitored via namespace-scoped Prometheus metrics.

2. **Shared etcd**: All tenant objects stored in one etcd. A quota abuse (e.g., 10,000 ConfigMaps) could impact API server latency. Mitigated by ResourceQuota `count/configmaps` limit (not set in v1 of this config — TODO for v2).

3. **NetworkPolicy enforcement gap**: VPC CNI without Calico provides no NetworkPolicy enforcement. This config assumes Calico or Cilium is installed. Verified in CI with a NetworkPolicy audit job.

### Escalation Trigger

If a third tenant onboards that requires PCI DSS scope or FedRAMP boundary:
- Provision a dedicated node group with `taint: pci-scope=true:NoSchedule`
- Restrict that namespace to PCI node group via `tolerations`
- Or provision a separate cluster — the `eks-cluster` module is designed to be called multiple times

### Isolation Layers Implemented

1. **Namespace boundary** — API-level blast radius containment
2. **RBAC** — `Role` + `RoleBinding` per namespace; no `ClusterRole` for tenant principals
3. **ResourceQuota** — CPU/memory/pod ceiling per namespace
4. **LimitRange** — default QoS; prevents Burstable-class pods from being first-evicted
5. **NetworkPolicy** — default-deny-all + explicit allowlist; CNI-layer enforcement
6. **PodSecurity admission** — `baseline` standard enforced at namespace level
7. **IRSA** — per-tenant IAM role with `StringEquals` condition on ServiceAccount subject

### What This Does NOT Provide

- Node-level isolation (shared EC2 instances)
- Kernel-level isolation (shared Linux kernel on each node)
- Protection against container escape exploits that reach the node (mitigated by IMDSv2)

EOF
```

**WHY**: Decision records are a Staff/Principal-level artifact. They show interviewers that you think about tradeoffs, not just implementation. This document is also the core of your portfolio GitHub repo for this capstone.

## Tradeoffs and Failure Modes

**1. Namespace isolation vs. cluster isolation — the compliance cliff**

Namespace isolation works until it doesn't. For most internal workloads, it is the right choice. The failure mode is gradual scope creep: you start with internal analytics workloads, then someone puts a PCI-adjacent service in the same cluster because "it's just reading anonymized data." Six months later you are in a compliance audit explaining why your PCI-scoped service shares a Linux kernel with a non-PCI workload. The mitigation is a written decision record (your ARCHITECTURE.md) with an explicit escalation trigger — any PCI/FedRAMP/HIPAA workload requires a separate node group or separate cluster, enforced as a policy gate in the onboarding Terraform module.

**2. NetworkPolicy enforcement requires CNI cooperation — it is not automatic**

This is the most dangerous gap in Kubernetes security that organizations consistently miss. NetworkPolicy objects are stored in the Kubernetes API (etcd) regardless of whether your CNI can enforce them. If you create a NetworkPolicy on a cluster running VPC CNI without Calico, the API accepts the object, `kubectl get networkpolicy` shows it, and you believe you are secure. But the CNI never enforces it — every pod can reach every other pod. The detection: periodically run a cross-namespace connectivity check from CI/CD. If the check succeeds, your NetworkPolicies are not enforced. The fix: install Calico or Cilium as a NetworkPolicy engine, or use Amazon VPC CNI Network Policy feature (GA in EKS 1.25+).

**3. IRSA StringLike wildcard — the silent over-privilege**

The most common IRSA misconfiguration is using `StringLike` instead of `StringEquals` in the trust policy condition, often with a wildcard: `"system:serviceaccount:*:notification-core-sa"`. This looks like it scopes to the ServiceAccount name but allows ANY namespace to claim the role by creating a ServiceAccount with the same name. A developer who can create namespaces and ServiceAccounts can now assume any IRSA role that uses a wildcard condition. The fix is `StringEquals` on the full subject string including the namespace: `"system:serviceaccount:notification-core:notification-core-sa"`. Audit all IRSA trust policies quarterly with: `aws iam get-role --role-name <name> --query 'Role.AssumeRolePolicyDocument'` and grep for `StringLike` — treat every instance as a finding.
