# Admission Webhooks + Pod Security | Week 3 Day 3

## Why This Matters

Every request to your Kubernetes API server passes through an admission control pipeline before anything gets written to etcd — this is the last enforcement gate before workloads run. Admission webhooks are how platform teams enforce security policy at scale without modifying application code, and Pod Security Standards replaced the deprecated PodSecurityPolicy (PSP) in K8s 1.25. Senior SRE/DevSecOps interviewers at Director-level specifically probe webhook failure modes because a misconfigured webhook can take down an entire cluster.

---

## Analogy

Think of the Kubernetes API server as an airport security checkpoint. When a passenger (API request) wants to board (get written to etcd and become a running pod):

1. **MutatingAdmissionWebhook** = The TSA agent who can *modify* your boarding pass — add tags, stamp it, attach a sidecar bag you didn't pack yourself (Istio envoy injection, Vault agent injection).
2. **ValidatingAdmissionWebhook** = The gate agent who *checks* your boarding pass and either lets you through or turns you away — no modifications, just approve or deny.
3. **Pod Security Standards** = The airline's published rule book (Privileged / Baseline / Restricted tiers) — the gate agent checks your ticket against this rulebook.
4. **OPA Gatekeeper** = A compliance officer standing *next to* the gate agent with a full policy manual — if the rule book doesn't cover it, Gatekeeper does.

The critical operational insight: if the checkpoint itself goes down (webhook server unavailable), the `failurePolicy` decides whether the airport shuts down or waves everyone through.

## Vocabulary Translation Table

| Their Term | Your Existing Mental Model |
|---|---|
| Admission Controller | Message filter/interceptor in a MOM system — every message passes through before processing |
| MutatingAdmissionWebhook | TIBCO BusinessWorks transformer — modifies the payload in flight |
| ValidatingAdmissionWebhook | RabbitMQ shovel policy check — accepts or rejects the message |
| Webhook server | Your own microservice that the API server calls (HTTP/HTTPS) |
| AdmissionReview | The JSON envelope the API server sends to your webhook — like a JMS message body |
| failurePolicy: Fail | Circuit breaker OPEN — deny everything if the policy service is down |
| failurePolicy: Ignore | Circuit breaker CLOSED bypass — let everything through if policy service is down |
| Pod Security Standards (PSS) | ITIL service tier classifications — Privileged = emergency break-glass, Restricted = hardened prod |
| PodSecurityPolicy (PSP) | The deprecated predecessor — like the old LDAP auth method you replaced with OIDC |
| OPA Gatekeeper | Policy-as-code engine that plugs into the admission webhook pipeline |
| ConstraintTemplate | The policy template (like a Splunk saved search pattern) |
| Constraint | An instantiation of the template against a specific scope (like a scheduled search run) |
| Namespace label | The control plane knob that activates Pod Security enforcement |

## 30-Second Version

Every Kubernetes API request runs through two webhook phases before commit: mutating (can change the object) then validating (can only approve or reject). Pod Security Standards enforce pod hardening at the namespace level using three built-in tiers — Privileged, Baseline, Restricted. OPA Gatekeeper extends this with custom policies you write in Rego, enforced as Kubernetes custom resources. The failure mode that kills clusters: a webhook with `failurePolicy: Fail` whose server goes down blocks ALL object creation in scope — including the DaemonSets trying to restart it.

## 2-Minute Version

### The Admission Control Pipeline

When you run `kubectl apply -f pod.yaml`, the API server processes the request in this exact order:

```
kubectl apply
    │
    ▼
Authentication (who are you?)
    │
    ▼
Authorization (RBAC — are you allowed?)
    │
    ▼
Admission Controllers (built-in: LimitRanger, ResourceQuota, etc.)
    │
    ▼
MutatingAdmissionWebhooks  ← your custom webhooks run here (in order)
    │   (can ADD/CHANGE fields — sidecar injection happens here)
    ▼
Object Schema Validation
    │
    ▼
ValidatingAdmissionWebhooks ← your custom webhooks run here (in order)
    │   (READ ONLY — approve or deny, cannot change)
    ▼
Written to etcd → Pod scheduled
```

Mutation runs BEFORE validation by design — otherwise you'd be validating an object that mutating hooks haven't finished shaping yet.

### Pod Security Standards (PSS) — the three tiers

PSP was deprecated in K8s 1.21 and removed in 1.25 because it was complex, required RBAC coupling, and had a confusing UX. PSS replaced it with a simpler namespace-label model:

| Tier | Who it's for | What it blocks |
|---|---|---|
| **Privileged** | System namespaces (kube-system, monitoring) | Nothing — fully open |
| **Baseline** | General workloads | HostNetwork, hostPID, hostIPC, privileged containers, hostPath volumes |
| **Restricted** | Hardened production, PCI/HIPAA, security-sensitive | Everything in Baseline + requires non-root, drops all capabilities, enforces seccomp |

You activate PSS by labeling the namespace:

```bash
# Enforce Restricted (deny non-compliant pods)
kubectl label namespace my-app pod-security.kubernetes.io/enforce=restricted

# Warn but allow (good for migration)
kubectl label namespace my-app pod-security.kubernetes.io/warn=restricted

# Audit to logs but allow (useful for discovery)
kubectl label namespace my-app pod-security.kubernetes.io/audit=restricted
```

Three modes — enforce, warn, audit — can all be active simultaneously. This is the migration path: audit → warn → enforce.

### OPA Gatekeeper

Gatekeeper installs as a ValidatingAdmissionWebhook. You write policy in two CRDs:

- **ConstraintTemplate**: defines a Rego policy and declares the CRD schema for instances of it
- **Constraint**: an instance that says "apply this template to these resources in these namespaces"

```
ConstraintTemplate (K8sRequireRegistryPrefix)
    └── Constraint (require-registry-prod)
            └── scope: namespaces=[prod, staging], resource=pods
```

## 5-Minute Version

### Webhook Registration Deep Dive

A `MutatingWebhookConfiguration` or `ValidatingWebhookConfiguration` is a cluster-scoped object that tells the API server: "for these resource types, in these namespaces, call this HTTPS endpoint and send me the AdmissionReview payload."

Key fields in the webhook config:

```yaml
webhooks:
  - name: sidecar-injector.istio.io
    admissionReviewVersions: ["v1", "v1beta1"]
    clientConfig:
      service:
        name: istiod
        namespace: istio-system
        path: /inject
      caBundle: <base64 TLS CA>       # webhook server must present valid cert
    rules:
      - operations: ["CREATE"]
        apiGroups: [""]
        apiVersions: ["v1"]
        resources: ["pods"]
    namespaceSelector:                 # only apply in namespaces with this label
      matchLabels:
        istio-injection: enabled
    failurePolicy: Fail                # ← THIS IS THE CRITICAL FIELD
    sideEffects: None
    timeoutSeconds: 5                  # default 10, max 30
```

### failurePolicy: The Design Decision That Ends Careers

| failurePolicy | Webhook server down | Behavior |
|---|---|---|
| `Fail` | Cannot contact webhook | Request DENIED — safe but availability risk |
| `Ignore` | Cannot contact webhook | Request ALLOWED — available but security risk |

**Operational rule of thumb**:
- Security-critical webhooks (image policy enforcement, OPA Gatekeeper): `Fail` + high-availability webhook deployment (3 replicas, PodDisruptionBudget)
- Non-critical mutation (add labels, annotations): `Ignore` — don't let a label-stamping webhook block pod creation

**The self-inflicted outage pattern**: You deploy Gatekeeper with `failurePolicy: Fail`. Gatekeeper crashes. Now no pods can be created — including the Gatekeeper pods trying to restart themselves. Cluster is wedged. Prevention: `namespaceSelector` that excludes `kube-system` and the Gatekeeper namespace, always.

### Why PSP Was Deprecated

PSP's fundamental design flaw: the policy was activated by RBAC, not by namespace. If a ServiceAccount had `use` permission on a permissive PSP *and* a restrictive PSP, the permissive one won. This made it nearly impossible to reason about which policy was actually enforced. Teams routinely had `privileged` PSPs active in prod because the RBAC was misconfigured. PSS's namespace-label model is explicit, auditable, and doesn't require RBAC coupling.

### Common Webhook Use Cases

| Use Case | Type | Who provides the webhook server |
|---|---|---|
| Istio sidecar injection | Mutating | istiod (Istio control plane) |
| Vault agent injection | Mutating | vault-agent-injector (HashiCorp) |
| OPA Gatekeeper policy | Validating | gatekeeper-controller-manager |
| Image policy enforcement | Validating | Gatekeeper or custom webhook |
| Resource defaults (LimitRange) | Mutating | Built-in admission controller |
| Namespace resource quotas | Validating | Built-in admission controller |
| Cert-manager TLS injection | Mutating | cert-manager webhook |

### Gatekeeper: ConstraintTemplate Anatomy

```rego
# The Rego policy — runs inside Gatekeeper's OPA engine
package k8srequireregistryprefix

violation[{"msg": msg}] {
    container := input.review.object.spec.containers[_]
    not startswith(container.image, input.parameters.prefix)
    msg := sprintf("Image '%v' must start with registry prefix '%v'", 
                   [container.image, input.parameters.prefix])
}
```

The `input.review.object` is the Kubernetes object being admitted — the same JSON the API server sends in the AdmissionReview. `input.parameters` comes from the Constraint instance, letting you parameterize policies without rewriting templates.

### Tradeoffs: PSS vs. OPA Gatekeeper

| Dimension | Pod Security Standards | OPA Gatekeeper |
|---|---|---|
| Setup complexity | Low — just label namespaces | High — install CRDs, webhook, write Rego |
| Policy expressiveness | Fixed tiers only | Arbitrary policy — any field, any condition |
| Auditability | Namespace label = clear intent | Constraint CRDs = GitOps-friendly |
| Performance overhead | Near-zero (built-in) | 5-20ms webhook latency per admission |
| Dry-run / migration | warn + audit modes | Audit-only mode available |
| Custom messages | No | Yes — custom violation messages |
| Recommendation | Default for all clusters | Layer on top for org-specific policy |

## Director/VP Lens

**How a Director governs admission webhooks:**

1. **Policy as code in Git** — every ConstraintTemplate and Constraint lives in a GitOps repo. Changes go through PR review, not `kubectl apply` directly. This is your audit trail for compliance.
2. **Namespace-tier labeling as a platform contract** — define in your Internal Developer Platform (IDP): prod namespaces MUST have `enforce=restricted`, staging gets `warn=restricted`, dev gets `warn=baseline`. Backstage scaffolder enforces this at namespace creation.
3. **Webhook availability = cluster availability** — every security webhook must be HA (3 replicas, PDB, multi-AZ). Track webhook admission latency in your SLO dashboard — if P99 > 500ms, investigate.
4. **PSP migration** — if you inherited a cluster still running PSP (K8s < 1.25), this is a security audit finding. The migration path is PSP → PSS enforce=audit → PSS enforce=warn → PSS enforce=restricted over 2-4 sprints.

**How an IC implements it:**

- Write ConstraintTemplates locally, test with `gator test` CLI before applying to cluster
- Always set `namespaceSelector` to exclude kube-system and the Gatekeeper namespace
- Use `enforcementAction: dryrun` first, then `warn`, then `deny` — never jump to deny on a new policy
- Monitor Gatekeeper audit results: `kubectl get constraints` shows violation counts

**AI/Automation angle:** AI-assisted policy generation is emerging — tools like Kubescape and Datree can auto-generate Gatekeeper policies from your existing workloads. As a Director, this is how you close the "we don't know what our pods are doing" gap quickly — run a 48-hour audit sweep before enforcing anything.

## Comprehension Check

Answer 2 of these 3 correctly to unlock the lab:

**Q1**: A MutatingAdmissionWebhook is registered for all pod CREATE operations with `failurePolicy: Fail`. The webhook server pod crashes. What happens to all subsequent pod CREATE requests, including the webhook server's own restart?

**Q2**: Your cluster is on Kubernetes 1.26. A developer complains their pod is being denied with "violates PodSecurity enforce: restricted." They set `securityContext.runAsNonRoot: true` but it's still failing. What is the most likely missing field?

**Q3**: You have a ConstraintTemplate with `enforcementAction: deny` that blocks images without a registry prefix. A developer deploys from `docker.io/nginx:latest` — the pod is created successfully. What are the two most likely causes?

---

## ANSWER KEY — Comprehension Check

**A1**: All subsequent pod CREATE requests are DENIED — including the webhook server's own RestartPolicy pod recreation. The cluster is wedged. The fix requires temporarily deleting or patching the MutatingWebhookConfiguration to remove the broken webhook, then redeploying the server. Prevention: `namespaceSelector` excluding the webhook's own namespace + HA deployment.

**A2**: The most likely missing field is `seccompProfile`. Restricted tier requires ALL of:
- `runAsNonRoot: true`
- `allowPrivilegeEscalation: false`
- `capabilities.drop: ["ALL"]`
- `seccompProfile.type: RuntimeDefault` (or `Localhost`)

Without `seccompProfile`, the pod fails Restricted even with runAsNonRoot set.

**A3**: Two most likely causes:
1. The Constraint has `enforcementAction: dryrun` or `warn` instead of `deny` — violations are logged but not blocked.
2. The Constraint's `match.namespaces` or `match.kinds` selector doesn't cover the namespace or pod resource — the policy doesn't apply to that object.

### Step 4: Try to deploy a privileged pod — observe denial

```bash
# Write a deliberately non-compliant pod spec
cat <<'EOF' > /tmp/privileged-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: bad-pod
  namespace: pss-test
spec:
  containers:
  - name: bad-container
    image: nginx:latest
    securityContext:
      privileged: true          # VIOLATION: Restricted forbids privileged
      runAsUser: 0              # VIOLATION: Restricted forbids root

```

**EXPECTED_OUTPUT**:
```
Error from server (Forbidden): error when creating "/tmp/privileged-pod.yaml":
```

### Step 9: Create a ConstraintTemplate — deny images without registry prefix

```bash
cat <<'EOF' > /tmp/require-registry-template.yaml
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8srequireregistryprefix
  annotations:
    description: "Requires container images to use an approved registry prefix"
spec:
  crd:
    spec:
      names:
        kind: K8sRequireRegistryPrefix
      validation:
        # Schema for the 'parameters' field in Constraints
        openAPIV3Schema:
          type: object
          properties:
            allowedPrefixes:
              type: array
              items:
                type: string
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8srequireregistryprefix

        # Collect all violations across all containers
        violation[{"msg": msg}] {
          # Iterate over all containers (including initContainers via union below)
          container := input_containers[_]
          # Check if ANY allowed prefix matches
          not any_prefix_matches(container.image)
          msg := sprintf(
            "Container '%v' uses image '%v' which does not start with an allowed registry prefix. Allowed: %v",
            [container.name, container.image, input.parameters.allowedPrefixes]
          )
        }

        # Helper: check if image starts with any allowed prefix
        any_prefix_matches(image) {
          prefix := input.parameters.allowedPrefixes[_]
          startswith(image, prefix)
        }

        # Helper: combine containers and initContainers into one list
        input_containers[container] {
          container := input.review.object.spec.containers[_]
        }
        input_containers[container] {
          container := input.review.object.spec.initContainers[_]
        }
EOF

kubectl apply -f /tmp/require-registry-template.yaml

# Wait for the CRD to be created
kubectl wait --for=condition=established crd/k8srequireregistryprefixes.constraints.gatekeeper.sh --timeout=60s
```

**EXPECTED_OUTPUT**:
```
constrainttemplate.templates.gatekeeper.sh/k8srequireregistryprefix created

customresourcedefinition.apiextensions.k8s.io/k8srequireregistryprefixes.constraints.gatekeeper.sh condition met
```

**WHY**: The ConstraintTemplate does two things: defines the Rego policy logic AND creates a new CRD (`K8sRequireRegistryPrefix`) that Constraint instances will use. Until this CRD exists, you cannot create Constraints.

### Step 10: Create a Constraint — apply the policy to a test namespace

```bash
# First create a namespace for Gatekeeper tests (without PSS labels — keep it simple)
kubectl create namespace gk-test

cat <<'EOF' > /tmp/require-registry-constraint.yaml
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequireRegistryPrefix
metadata:
  name: require-approved-registry
spec:
  enforcementAction: deny          # actively block — use 'dryrun' to audit first
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
    namespaces:
      - gk-test                    # only enforce in this namespace
  parameters:
    allowedPrefixes:
      - "public.ecr.aws/"          # AWS public ECR
      - "123456789012.dkr.ecr.us-east-1.amazonaws.com/"  # your private ECR
      - "registry.k8s.io/"         # official K8s images
      - "quay.io/prometheus"       # approved Prometheus images
EOF

kubectl apply -f /tmp/require-registry-constraint.yaml

# Wait for constraint to be ready
sleep 10
kubectl get k8srequireregistryprefix require-approved-registry
```

**EXPECTED_OUTPUT**:
```
namespace/gk-test created

k8srequireregistryprefixes.constraints.gatekeeper.sh/require-approved-registry created

NAME                        ENFORCEMENT-ACTION   TOTAL-VIOLATIONS
require-approved-registry   deny                 0
```

**WHY**: `enforcementAction: deny` means violations block admission. `TOTAL-VIOLATIONS: 0` means no existing pods in `gk-test` violate it (namespace was just created). The constraint only applies to the `gk-test` namespace — other namespaces are unaffected.

## STAR Anchor

**Q1 — Architecture deep dive:**
> "In my role at T-Mobile managing the 25M msg/day notification platform on EKS, I reviewed every admission controller registered on our cluster as part of our quarterly security baseline review. We found three webhooks with `failurePolicy: Fail` that had no HA backing — including one from a deprecated add-on that was half-uninstalled. I mapped the full admission pipeline, documented which webhooks ran at which phase, and we replaced the old add-on webhook with an OPA Gatekeeper constraint. The walkthrough I developed became the onboarding doc for new engineers joining the platform team."

**Q2 — Failure modes / incident response:**
> "This directly maps to an incident pattern I've managed — a critical middleware component going down and the recovery tool also being blocked. The immediate remediation is to patch the MutatingWebhookConfiguration to set `failurePolicy: Ignore` as a short-term break-glass — this restores cluster function at the cost of bypassing the control temporarily. Simultaneously, you fix and redeploy the webhook server. Then restore `failurePolicy: Fail` and verify. The permanent fix is HA deployment with PDB and a runbook for this exact scenario."

**Q3 — PSP deprecation:**
> "PSP's fundamental flaw was RBAC coupling — policy activation was controlled by whether a ServiceAccount had `use` permission on the PSP object, not by namespace or workload characteristics. In practice, teams would have both a restrictive and a permissive PSP, and the permissive one would win because of how RBAC evaluation worked. I've seen audit findings where privileged containers were running in production because someone had granted the wrong ServiceAccount access to the wrong PSP. Pod Security Standards solve this by making the policy explicit at the namespace level — there's no RBAC to misconfigure."

**Q4 — Migration strategy:**
> "I'd run this as a three-sprint migration. Sprint 1: apply `audit=restricted` to all namespaces — zero blast radius, just log violations. Sprint 2: after two weeks of audit data, fix the top violations by workload type, then apply `warn=restricted` — developers see warnings but nothing breaks. Sprint 3: resolve remaining violations, then apply `enforce=restricted` namespace by namespace starting with lowest-risk, ending with prod. At T-Mobile, we ran a similar migration for security hardening across 30+ namespaces and used a similar audit-first approach to avoid surprise production incidents."

**Q5 — Platform engineering / golden path:**
> "In our IDP using Backstage scaffolder templates, namespace creation goes through a template that automatically applies the correct PSS labels based on the environment field (dev/staging/prod maps to baseline/warn-restricted/enforce-restricted). Gatekeeper constraints are applied via ArgoCD ApplicationSets that target namespaces by label — any new namespace with `env=prod` automatically gets the full production constraint set synced from Git. The developer never touches RBAC or security policy — they pick an environment in the Backstage UI and the platform handles the rest."

## Tradeoffs & Failure Modes

### Tradeoff 1: failurePolicy Fail vs. Ignore — Security vs. Availability

**The tension**: `Fail` is the secure default — if you can't check policy, deny the request. But a webhook server outage now means cluster-wide denial of all object creation in scope, including incident response pods, monitoring, and the webhook server's own recovery.

**The right answer**: Use `Fail` for security-critical webhooks (image policy, Gatekeeper), but invest in HA: 3 replicas, PodDisruptionBudget, `minAvailable: 2`, multi-AZ placement. Always set `namespaceSelector` to exclude kube-system and the webhook's own namespace. Document break-glass procedure: `kubectl patch validatingwebhookconfiguration <name> --type=json -p='[{"op":"replace","path":"/webhooks/0/failurePolicy","value":"Ignore"}]'`

**Interview signal**: A candidate who only says "use Fail for security" without mentioning the HA requirement and the self-exclusion pattern hasn't operated this in production.

---

### Tradeoff 2: Pod Security Standards vs. OPA Gatekeeper — Simplicity vs. Expressiveness

**The tension**: PSS is built-in, zero-overhead, and covers 80% of hardening needs with just namespace labels. OPA Gatekeeper can express any policy but adds latency (5-20ms per admission), requires Rego knowledge, and needs its own HA deployment.

**The right answer**: PSS as the baseline for all namespaces — it's free and catches the most critical issues (privileged containers, host namespace access). Layer Gatekeeper on top for org-specific policy (image registry enforcement, required labels, resource limits mandates). Do NOT use Gatekeeper to re-implement what PSS already does — that's double maintenance.

**The failure mode to avoid**: Using `enforcementAction: deny` on a new Gatekeeper policy without a prior `dryrun` phase. You can lock out all pods from a namespace instantly if your Rego has an unexpected match pattern.

---

### Tradeoff 3: Admission-time enforcement vs. Runtime enforcement — Shift-Left vs. Defense-in-Depth

**The tension**: Admission webhooks only fire at CREATE/UPDATE time. A policy you add today doesn't retroactively apply to already-running pods. An attacker who gains exec access to a running pod bypasses admission entirely. Admission enforcement is not runtime security.

**The right answer**: Admission control is shift-left prevention — stop bad configs from ever running. It must be paired with runtime enforcement (Falco, AWS Security Hub findings, Kubernetes Audit logging to SIEM/Splunk) for defense-in-depth. Gatekeeper's audit mode bridges the gap for existing resources — it periodically scans and reports violations in running workloads even if it can't retroactively deny them.

**Interview signal**: At Director level, the expected answer is "admission control is one layer — what's your runtime security layer?" Candidates who treat admission webhooks as their complete security solution reveal they haven't operated under a real threat model.
