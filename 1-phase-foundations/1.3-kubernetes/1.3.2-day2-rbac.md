# Kubernetes RBAC — Who Can Do What | Week 3 Day 2

## Why This Matters

In production Kubernetes clusters, the blast radius of a compromised or misconfigured workload is determined almost entirely by its RBAC permissions. At T-Mobile scale — a 25M msg/day notification platform — a single over-privileged ServiceAccount can give an attacker cluster-admin access from a single pod breach. Interviewers at Staff/Principal level will probe whether you can design least-privilege identity models, audit them, and explain the failure modes of the defaults.

## Analogy

Think of RBAC like your T-Mobile incident command access model. When a P1 fires, the on-call SRE gets a time-boxed role — they can read dashboards, restart services, but cannot delete production databases or approve infra changes. That role assignment is temporary and scoped: it applies only to the systems in scope for that incident.

Kubernetes RBAC works the same way:
- The **Subject** is like the on-call person (a ServiceAccount, User, or Group)
- The **Role** is like the incident command runbook defining what actions are allowed
- The **RoleBinding** is the page that says "this person is on-call for this system right now"
- The **Namespace** is the blast radius boundary — like saying "you're on-call for the payments namespace only, not the entire org"

## Vocabulary Translation Table

| Their Term | Your Existing Mental Model |
|---|---|
| ServiceAccount | Bot/service identity in IAM — like a "machine user" in your LDAP/AD |
| Role | A permission policy document — like an IAM policy or CyberArk role |
| RoleBinding | The assignment linking a person to a policy — like "user X has role Y in system Z" |
| ClusterRole | An org-wide IAM policy (no namespace boundary) |
| ClusterRoleBinding | Org-wide role assignment — like granting someone global read-only in your SIEM |
| Subject | The identity being granted access — human user, service, or group |
| Verb | HTTP method equivalent — get, list, watch, create, delete |
| Resource | The Kubernetes object type — pods, secrets, configmaps, deployments |
| Aggregated ClusterRole | A "parent role" that automatically inherits rules from labeled child roles — like a composite LDAP group |
| `kubectl auth can-i` | Your "check my access" command — equivalent to `aws iam simulate-principal-policy` |

## 30-Second Version

Kubernetes RBAC answers one question: "Can Subject X perform Verb Y on Resource Z in Namespace N?" Every workload (pod) runs as a ServiceAccount. By default, all pods in a namespace share the `default` ServiceAccount, which often has no permissions — but the dangerous trap is clusters where legacy configuration or Helm charts gave `default` elevated access. You define what a ServiceAccount can do using Role + RoleBinding (namespace-scoped) or ClusterRole + ClusterRoleBinding (cluster-wide). The principle of least privilege means: give exactly the verbs on exactly the resources in exactly the namespace that the workload needs — nothing more.

## 2-Minute Version

**The RBAC Chain**

```
Subject (ServiceAccount / User / Group)
    |
    v
RoleBinding (links subject to role, scoped to a namespace)
    |
    v
Role (list of rules: which verbs on which resources)
    |
    v
API Server decision: ALLOW or DENY
```

**Role vs ClusterRole**

A `Role` exists inside a namespace. It can only grant access to resources inside that same namespace. A `ClusterRole` has no namespace — it can grant access to:
1. Cluster-scoped resources (Nodes, PersistentVolumes, Namespaces themselves)
2. Resources across ALL namespaces (when bound via ClusterRoleBinding)
3. Resources in a SPECIFIC namespace (when bound via a regular RoleBinding — this is intentional and useful)

**The last point is subtle and frequently tested in interviews**: You can bind a ClusterRole with a namespaced RoleBinding. This means "use this reusable cluster-level role definition, but only apply it in namespace X." This avoids duplicating Role definitions across namespaces.

**ServiceAccount as Pod Identity**

Every pod spec has a `serviceAccountName` field. If you don't set it, Kubernetes assigns the `default` ServiceAccount in that namespace. The API server mounts the ServiceAccount's JWT token automatically at `/var/run/secrets/kubernetes.io/serviceaccount/token` inside every pod. This token is what the pod uses to authenticate to the Kubernetes API — so if you exec into a pod and run `curl` against the API server, you're calling as that ServiceAccount identity.

**The Default ServiceAccount Trap**

In older clusters or clusters using Helm charts with sloppy defaults, the `default` ServiceAccount in a namespace may have been granted elevated permissions (sometimes cluster-admin via a ClusterRoleBinding). An attacker who compromises any pod in that namespace can then call the Kubernetes API with cluster-admin rights. Always audit: `kubectl get clusterrolebindings -o wide | grep default`.

## 5-Minute Version

**Aggregated ClusterRoles**

Kubernetes ships with built-in ClusterRoles: `cluster-admin`, `admin`, `edit`, `view`. These are "aggregated" roles — they're composed by selecting other ClusterRoles that have a specific label. For example, any ClusterRole with the label `rbac.authorization.k8s.io/aggregate-to-view: "true"` automatically gets its rules merged into the built-in `view` ClusterRole.

This is useful for operators/controllers: when you install a CRD (Custom Resource Definition), you create a ClusterRole for that CRD's verbs and label it to aggregate into the existing built-in roles. Users who already have `view` automatically get view access to your new CRD resources — no manual RoleBinding updates needed.

**Design Decisions and Tradeoffs**

| Decision | Option A | Option B | When to use A | When to use B |
|---|---|---|---|---|
| Scope of identity | One ServiceAccount per workload | Shared default ServiceAccount | Production, security-sensitive | Dev/local prototyping only |
| Role scope | Role (namespaced) | ClusterRole + RoleBinding | Single-namespace services | Reusable roles across namespaces |
| Binding scope | RoleBinding | ClusterRoleBinding | Grant access in ONE namespace | Grant access in ALL namespaces |
| Token projection | Auto-mounted token | Projected ServiceAccount Token (IRSA) | In-cluster API access only | AWS API access from pods (IRSA pattern) |

**Failure Modes**

1. **Over-broad verbs**: Granting `*` on `*` "for now" — attackers can list secrets, exec into pods, delete deployments. This is audit failure mode #1.

2. **ClusterRoleBinding to default ServiceAccount**: One Helm chart install does `kubectl create clusterrolebinding --clusterrole=cluster-admin --serviceaccount=my-ns:default` and every pod in that namespace is now cluster-admin. This has caused real production incidents.

3. **RBAC works at the API layer, not at the data layer**: RBAC controls who can GET a Secret object from the Kubernetes API. It does NOT control whether the secret value is encrypted at rest in etcd, or whether the pod can read the file on disk after it's mounted. Encryption-at-rest and audit logging are separate controls.

4. **Watch verb is often forgotten**: `list` lets you enumerate all objects. `watch` lets you subscribe to changes — which means you can exfiltrate secrets over time by watching for new Secret objects being created. Granting `list` without `watch` is a partial control — both should be considered together.

**Auditing RBAC**

Three tools every SRE should know:

1. `kubectl auth can-i <verb> <resource> --as=system:serviceaccount:<ns>:<sa>` — impersonate a ServiceAccount and check a specific permission.
2. `kubectl auth can-i --list --as=system:serviceaccount:<ns>:<sa>` — dump ALL permissions for a ServiceAccount.
3. `rbac-lookup` (open source CLI) — human-readable "who has access to what" across the whole cluster.

## Director/VP Lens

**How an IC implements this**: Writes Role and RoleBinding YAML, creates dedicated ServiceAccounts per workload, tests with `kubectl auth can-i`, integrates RBAC review into PR checklists for new service deployments.

**How a Director governs this**:
- Sets policy: "Every new service MUST have its own ServiceAccount — no use of `default`" — enforced via OPA/Gatekeeper admission webhook (Week 2 topic, reinforced here).
- Mandates RBAC audit in quarterly security reviews — uses `rbac-lookup` output as artifact.
- Defines "RBAC escalation" runbook: if a ServiceAccount needs elevated permissions temporarily (incident response), there's a break-glass process with audit trail, not a standing ClusterRoleBinding.
- Tracks "RBAC coverage" as a security SLO metric: what percentage of workloads use dedicated ServiceAccounts vs. `default`?
- AI angle: LLM-assisted RBAC policy generation is emerging (e.g., Permit.io, Styra DAS) — understand the trust boundaries before adopting AI-generated RBAC.

**Interview frame**: "At T-Mobile, I govern RBAC at the platform level — our golden path template for new services includes a dedicated ServiceAccount and a Role scoped to the exact ConfigMaps and Secrets that service owns. We enforce this via an admission webhook that rejects pods with `serviceAccountName: default` in production namespaces. ICs implement the YAML; I define the policy that makes the secure path the easy path."

## Comprehension Check

Answer these before unlocking the lab. You need 2/3 to proceed.

**Q1**: A developer says "my pod needs to read Secrets from namespace `payments` AND list Nodes in the cluster." You need to write the minimal RBAC for this. What combination of Role/ClusterRole + Binding do you use, and why can't you use a single Role for both?

**Q2**: Your security audit finds: `kubectl get clusterrolebindings | grep default` returns a binding that grants `cluster-admin` to `system:serviceaccount:monitoring:default`. What is the blast radius of this misconfiguration, and what is your immediate remediation step?

**Q3**: A teammate says "I'll just give the app's ServiceAccount `get`, `list`, `watch` on Secrets — that's read-only, so it's safe." What's the flaw in this reasoning?

---

## ANSWER KEY — Comprehension Check

**A1**: You need TWO bindings:
- A **Role** in namespace `payments` granting `get`/`list` on `secrets`, bound with a **RoleBinding** to the ServiceAccount in `payments`.
- A **ClusterRole** granting `list` on `nodes`, bound with a **ClusterRoleBinding** to the ServiceAccount.
You cannot use a single Role because Roles are namespace-scoped and `nodes` is a cluster-scoped resource — it doesn't belong to any namespace. ClusterRole is the only object that can express permissions on cluster-scoped resources.

**A2**: Blast radius: Any pod in namespace `monitoring` that uses the `default` ServiceAccount has full cluster-admin rights — it can read all secrets in all namespaces, delete any workload, create new ClusterRoleBindings to escalate further. Immediate remediation: `kubectl delete clusterrolebinding <binding-name>`. Then audit: which pods in `monitoring` are using the `default` ServiceAccount and whether any were compromised. Longer term: ensure monitoring tools (Prometheus, etc.) have their own dedicated ServiceAccount with only the permissions they need.

**A3**: The flaw is scope creep and the watch verb. `get`/`list`/`watch` on Secrets in a namespace means the ServiceAccount can read EVERY secret in that namespace — not just the one secret it needs. Least privilege means specifying `resourceNames: [my-specific-secret]` in the Role rules to restrict access to only named resources. Also, `watch` allows subscribing to secret change events, which can be used to exfiltrate secrets as they rotate — this verb should be granted only if the workload genuinely needs change notification.

## Lab — Kubernetes RBAC: Notification Service Identity

**Pre-requisite**: A running Kubernetes cluster accessible via `kubectl`. You can use a local kind/minikube cluster OR your EKS cluster (if running from Week 3 Day 1 lab). If using kind:

```bash
# Install kind if not present
curl -Lo ./kind https://kind.sigs.k8s.io/dl/v0.22.0/kind-linux-amd64
chmod +x ./kind
sudo mv ./kind /usr/local/bin/kind

# Create a cluster
kind create cluster --name rbac-lab
```

**Scenario**: You're deploying a `notification-service` pod that needs to read ConfigMaps (for feature flags) and Secrets (for API keys) in the `notifications` namespace. It must NOT be able to create pods, list nodes, or touch resources in other namespaces.

### Step 1: Create the notifications namespace

```bash
kubectl create namespace notifications
```

**EXPECTED_OUTPUT**:
```
namespace/notifications created
```

**WHY**: Namespaces are the blast radius boundary for Role objects. Everything in this lab lives in `notifications`.

### Step 2: Create a dedicated ServiceAccount for the notification service

```bash
kubectl create serviceaccount notification-service \
  --namespace notifications
```

**EXPECTED_OUTPUT**:
```
serviceaccount/notification-service created
```

**WHY**: This is the pod identity. Giving it a unique name means RBAC policies can be scoped exactly to this workload — not shared with every other pod in the namespace.

---

### Step 3: Verify the ServiceAccount exists and inspect it

```bash
kubectl get serviceaccount notification-service \
  --namespace notifications \
  -o yaml
```

**EXPECTED_OUTPUT** (abbreviated):
```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: notification-service
  namespace: notifications
  uid: <some-uid>
  creationTimestamp: "<timestamp>"
```

**WHY**: Confirms the SA was created. Note: in Kubernetes 1.24+, tokens are no longer auto-created as Secrets — they're projected tokens with expiry. You'll see no `secrets:` block referencing a token Secret here.

### Step 4: Create sample ConfigMap and Secret to test against

```bash
kubectl create configmap feature-flags \
  --namespace notifications \
  --from-literal=enable_sms=true \
  --from-literal=enable_push=true

kubectl create secret generic api-keys \

  --from-literal=firebase_key=fake-firebase-key
```

**EXPECTED_OUTPUT**:
```
configmap/feature-flags created
secret/api-keys created
```

**WHY**: Creates the actual resources the notification service needs to read. These simulate real operational config.

---

### Step 5: Create the Role — read-only access to ConfigMaps and Secrets

```bash
cat <<'EOF' | kubectl apply -f -
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: notification-reader
  namespace: notifications
rules:
  # Rule 1: Read ConfigMaps
  - apiGroups: [""]          # core API group — where ConfigMaps, Secrets, Pods live
    resources: ["configmaps"]
    verbs: ["get", "list", "watch"]
  # Rule 2: Read specific Secrets only — not ALL secrets in the namespace
  - apiGroups: [""]
    resources: ["secrets"]
    resourceNames: ["api-keys"]   # LEAST PRIVILEGE: only this named secret
    verbs: ["get"]
EOF
```

**EXPECTED_OUTPUT**:
```
role.rbac.authorization.k8s.io/notification-reader created
```

**WHY**: Two rules in one Role. The ConfigMap rule is broad (all configmaps in namespace). The Secret rule is intentionally narrow — `resourceNames` restricts to only the `api-keys` secret. This is the least-privilege pattern that most candidates miss in interviews.

---

### Step 6: Verify the Role rules look correct

```bash
kubectl describe role notification-reader --namespace notifications
```

**EXPECTED_OUTPUT**:
```
Name:         notification-reader
Labels:       <none>
Annotations:  <none>
PolicyRule:
  Resources   Non-Resource URLs  Resource Names  Verbs
  ---------   -----------------  --------------  -----
  configmaps  []                 []              [get list watch]
  secrets     []                 [api-keys]      [get]
```

**WHY**: The `describe` output is the human-readable audit view. In an interview or security review, this is what you'd screenshot as evidence of least-privilege policy.

---

### Step 7: Create the RoleBinding — link ServiceAccount to Role

```bash
cat <<'EOF' | kubectl apply -f -
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: notification-service-binding
  namespace: notifications
subjects:
  # The identity being granted access
  - kind: ServiceAccount
    name: notification-service
    namespace: notifications
roleRef:
  # The role being granted
  kind: Role
  name: notification-reader
  apiGroup: rbac.authorization.k8s.io
EOF
```

**EXPECTED_OUTPUT**:
```
rolebinding.rbac.authorization.k8s.io/notification-service-binding created
```

**WHY**: The RoleBinding is the "assignment" record. Without this, the ServiceAccount exists but has zero permissions. The Role exists but is applied to nobody. This is the glue.

---

### Step 8: Verify the RoleBinding

```bash
kubectl describe rolebinding notification-service-binding \
  --namespace notifications
```

**EXPECTED_OUTPUT**:
```
Name:         notification-service-binding
Labels:       <none>
Annotations:  <none>
Role:
  Kind:  Role
  Name:  notification-reader
Subjects:

  ----            ----                  ---------
  ServiceAccount  notification-service  notifications
```

**WHY**: Confirms the chain: ServiceAccount `notification-service` → Role `notification-reader`. This is the audit artifact you'd show in a SOC2 or security review.

---

### Step 9: Use kubectl auth can-i to verify permissions BEFORE deploying a pod

```bash
# Should be YES
kubectl auth can-i get configmaps \
  --namespace notifications \
  --as=system:serviceaccount:notifications:notification-service

# Should be YES
kubectl auth can-i get secrets \
  --namespace notifications \
  --as=system:serviceaccount:notifications:notification-service

# Should be NO — cannot create pods
kubectl auth can-i create pods \
  --namespace notifications \
  --as=system:serviceaccount:notifications:notification-service

# Should be NO — cluster-scoped resource
kubectl auth can-i list nodes \
  --as=system:serviceaccount:notifications:notification-service
```

**EXPECTED_OUTPUT**:
```
yes
yes
no
no
```

**WHY**: `kubectl auth can-i --as=` is your pre-flight check. Run this BEFORE deploying pods. This impersonates the ServiceAccount and queries the authorization layer directly — no pod needed. This is the tool you use in code reviews and security audits.

### Step 10: Get the full permission list for the ServiceAccount

```bash
kubectl auth can-i --list \
  --namespace notifications \
  --as=system:serviceaccount:notifications:notification-service
```

**EXPECTED_OUTPUT** (abbreviated — you'll see lines like):
```

configmaps                                      []                  []               [get list watch]

selfsubjectaccessreviews.authorization.k8s.io   []                  []               [create]

```

**WHY**: The `--list` flag dumps everything the identity can do. Use this for complete permission audits. The `selfsubjectaccessreviews` and `selfsubjectrulesreviews` entries are always present — they're system-granted to let any subject check its own permissions.

### Step 11: Deploy a pod using the notification-service ServiceAccount

```bash
cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: notification-worker
  namespace: notifications
spec:
  serviceAccountName: notification-service   # USE our dedicated SA, not default
  containers:
    - name: worker
      image: curlimages/curl:latest           # Small image with curl — useful for API testing
      command: ["sleep", "3600"]             # Keep pod running for exec
      env:
        - name: KUBERNETES_SERVICE_HOST      # Explicit — helps with API server URL construction
          value: "kubernetes.default.svc"
EOF
```

**EXPECTED_OUTPUT**:
```
pod/notification-worker created
```

**WHY**: The `serviceAccountName` field is what links this pod to its identity. Kubernetes will automatically mount a projected token for `notification-service` into the pod at `/var/run/secrets/kubernetes.io/serviceaccount/token`.

### Step 13: Exec into the pod and verify the ServiceAccount token exists

```bash
kubectl exec -it notification-worker \
  --namespace notifications \
  -- sh -c "ls /var/run/secrets/kubernetes.io/serviceaccount/"
```

**EXPECTED_OUTPUT**:
```
ca.crt     namespace  token
```

**WHY**: Three files are always mounted: `ca.crt` (cluster CA cert for TLS verification), `namespace` (the namespace name as a string), `token` (the JWT for authenticating to the K8s API). This is how in-cluster workloads authenticate — they read this token file.

### Step 14: From inside the pod — read the ConfigMap (should SUCCEED)

```bash
kubectl exec -it notification-worker \
  --namespace notifications \
  -- sh -c '
TOKEN=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)
CACERT=/var/run/secrets/kubernetes.io/serviceaccount/ca.crt
NAMESPACE=$(cat /var/run/secrets/kubernetes.io/serviceaccount/namespace)
APISERVER=https://kubernetes.default.svc

curl -s \
  --cacert $CACERT \
  -H "Authorization: Bearer $TOKEN" \
  "$APISERVER/api/v1/namespaces/$NAMESPACE/configmaps/feature-flags"
'
```

**EXPECTED_OUTPUT** (abbreviated JSON):
```json
{
  "kind": "ConfigMap",
  "apiVersion": "v1",
  "metadata": {
    "name": "feature-flags",
    "namespace": "notifications",
    ...
  },
  "data": {
    "enable_push": "true",
    "enable_sms": "true"
  }
}
```

**WHY**: This is the pod authenticating to the Kubernetes API using its mounted JWT token. The ConfigMap read succeeds because the Role grants `get` on `configmaps`. This is exactly what your application SDK (e.g., the Kubernetes Go client or Python `kubernetes` library) does under the hood.

### Step 15: From inside the pod — attempt to CREATE a pod (should FAIL)

```bash
kubectl exec -it notification-worker \

  --cacert $CACERT \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -X POST \
  "$APISERVER/api/v1/namespaces/$NAMESPACE/pods" \
  -d "{\"apiVersion\":\"v1\",\"kind\":\"Pod\",\"metadata\":{\"name\":\"evil-pod\"},\"spec\":{\"containers\":[{\"name\":\"c\",\"image\":\"nginx\"}]}}"

```

**EXPECTED_OUTPUT**:
```json
{
  "kind": "Status",
  "apiVersion": "v1",
  "metadata": {},
  "status": "Failure",

  "details": {
    "kind": "pods"
  },
  "code": 403
}
```

**WHY**: The 403 Forbidden with the exact ServiceAccount name in the message confirms RBAC is working. The error message `User "system:serviceaccount:notifications:notification-service"` shows how Kubernetes identifies ServiceAccounts in its audit log — this is the string you'll see in security incident reports.

## Lab ANSWER KEY

**Step 12 — Pod stuck in ImagePullBackOff**:
The cluster can't pull `curlimages/curl`. If you're on kind with no internet access:
```bash
docker pull curlimages/curl:latest
kind load docker-image curlimages/curl:latest --name rbac-lab
```
Then delete and recreate the pod.

**Step 13 — Directory `/var/run/secrets/kubernetes.io/serviceaccount/` is empty or missing**:
The pod may be running as a non-default user that can't read the mount. Try `kubectl exec -it notification-worker --namespace notifications -- ls -la /var/run/secrets/kubernetes.io/serviceaccount/` to check permissions. This is rare with standard images.

**Step 14 — curl returns `curl: (6) Could not resolve host: kubernetes.default.svc`**:
DNS is not working inside the pod. Check CoreDNS: `kubectl get pods -n kube-system | grep coredns`. If CoreDNS pods are down, restart them: `kubectl rollout restart deployment/coredns -n kube-system`.

**Step 14 — Returns 401 instead of the ConfigMap**:
Token is expired or malformed. Check the token file exists: `kubectl exec notification-worker -n notifications -- cat /var/run/secrets/kubernetes.io/serviceaccount/token | cut -c1-50`. If it shows `null` or is empty, the ServiceAccount token projection failed — delete and recreate the pod.

**Step 15 — Returns 200 instead of 403 (pod creation succeeded)**:
Your cluster has a permissive admission controller or the RBAC RoleBinding was applied to the wrong namespace. Check: `kubectl auth can-i create pods --namespace notifications --as=system:serviceaccount:notifications:notification-service`. If this returns `yes`, re-examine your Role YAML — the `verbs` field may have accidentally included `create`. Delete and reapply the Role from Step 5.

**Step 9 — `kubectl auth can-i` returns `yes` for `list nodes`**:
The `default` ClusterRoleBinding for `system:authenticated` in your cluster may grant node listing. This is a cluster configuration issue, not your RBAC. Check: `kubectl get clusterrolebindings -o yaml | grep -A5 system:authenticated`. This is actually a finding — note it in your audit.

## STAR Anchor

**Q1 — Multi-tenant RBAC design**:
"Situation: At T-Mobile, our notification platform serves multiple internal teams — SMS, push, email — each with their own configurations and API keys that must not cross-leak. Task: Design isolation without separate clusters (cost and ops overhead). Action: Implemented namespace-per-team with dedicated ServiceAccounts, Role per service type, and an admission webhook that rejects pods not matching the ServiceAccount naming convention for that namespace. Result: Zero cross-namespace secret access incidents in 18 months; passed SOC2 audit without RBAC findings."

**Q2 — Incident response for over-privileged SA**:
"Situation: Similar pattern — discovered during a Vault integration project that a monitoring tool's SA had cluster-wide secret read via a stale binding. Task: Assess blast radius, remediate, and prevent recurrence. Action: Immediately deleted the ClusterRoleBinding, ran `kubectl auth can-i --list` across all SAs in affected namespaces, reviewed API server audit logs for anomalous secret reads in the prior 30 days. Result: No evidence of exploitation; added RBAC audit to monthly security rotation."

**Q3 — ClusterRole vs Role**:
"At T-Mobile, our Prometheus scraping ServiceAccount needed the same read permissions across 12 application namespaces. Using 12 identical Roles would mean 12 places to update when we added a new CRD. We created one ClusterRole for scraping permissions and bound it with a RoleBinding in each namespace — so it only applies per-namespace but the rule definition is maintained in one place."

**Q4 — 15-minute debug**:
"First command: `kubectl get rolebindings -n <new-ns> -o wide` — look for the ServiceAccount in the `SERVICEACCOUNTS` column. Third command: `kubectl describe role <role-name> -n <new-ns>` — check if `resourceNames` is scoping to a secret that doesn't exist in the new namespace after migration."

**Q5 — Governance at scale**:

OPA/Gatekeeper enforces at admission time that no pod uses `default` SA in prod namespaces. Quarterly, I run `rbac-lookup` cluster-wide and review the output with the security team — any SA with cluster-scoped bindings needs a documented justification."

## Tradeoffs & Failure Modes

**Tradeoff 1: Least privilege vs. operational velocity**

The tightest RBAC (one Role per workload, `resourceNames` scoping for every Secret) is the most secure but creates significant operational friction — every new secret, every new configmap requires a Role update. The practical balance: use `resourceNames` for high-sensitivity secrets (API keys, certs), allow namespace-wide read for lower-sensitivity configs (feature flags). Define this policy in your security runbook so it's a decision framework, not case-by-case.

**Tradeoff 2: ClusterRoleBinding reuse vs. audit clarity**

Reusing a ClusterRole across many namespaces via individual RoleBindings reduces duplication but makes the audit graph more complex — `rbac-lookup` returns a sprawling output. Some teams prefer namespace-specific Roles for audit clarity even at the cost of duplication. Tooling like Styra DAS or `kubectl-who-can` helps navigate the graph. The right answer depends on your compliance posture — SOC2/ISO27001 auditors prefer traceability over elegance.

**Failure Mode 3: RBAC covers the API layer but not the data layer**

This is the most dangerous misunderstanding. RBAC preventing `get secret` via the API does NOT mean the secret value is protected if:
- The secret was already mounted as an environment variable in a pod (the env var is accessible to anyone who can exec into the pod)
- etcd is not encrypted at rest (secrets stored in plaintext in etcd are readable by anyone with etcd access)
- A sidecar or init container with broader permissions exfiltrates the value before RBAC applies

Defense-in-depth requires: RBAC + etcd encryption at rest + audit logging + no secrets as env vars (use volume mounts with `readOnly: true`) + Vault/AWS Secrets Manager for the most sensitive values.
