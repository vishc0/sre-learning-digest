# Week 3, Day 1: Kubernetes Control Plane Deep Dive

**Objective**: Understand the control plane as a distributed system with five core components, their failure modes, and how they orchestrate workloads. Build mental models for diagnosing production incidents and making governance decisions.

**Time**: 3 hours (2 hrs study + lab, 1 hr review/tradeoffs)

**Target confidence**: 4 (can explain tradeoffs; can debug 3-component outages)

---

## 1. Real-World Analogy: Incident Command Structure

The Kubernetes control plane **is** an incident command center. Map this directly:

| K8s Component | IC Role | Responsibility |
|---|---|---|
| **API Server** | Incident Commander (IC) | Single source of truth. Takes all requests, says yes/no, logs decisions (audit). No IC, no decisions. |
| **etcd** | Chronology Officer (CHO) | Writes every IC decision to the incident log. Quorum voting ensures decisions stick. If log burns, IC decisions vanish. |
| **kube-controller-manager** | Specialist Responders (multiple) | 30+ controllers (ReplicaSet, Deployment, StatefulSet, etc.). Each watches the goal state, works until reality matches. |
| **kube-scheduler** | Logistics Coordinator | Watches new Pods, finds the best Node to run them on. Filter bad nodes, score good nodes, pick winner. |
| **kubelet** | On-Site Tech | Local agent per node. Runs the actual containers. Reports back to IC: "Pod is running" or "Pod failed." |

**Why this works**: 
- IC doesn't run to the incident site; they coordinate from a command center (API server doesn't run your app).
- CHO documents every decision in real time so the incident log survives power loss (etcd is persistent state).
- Responders don't interrupt each other; they each work their lane and self-heal (controllers are independent, level-triggered).
- Logistics doesn't place people; they find the best location and let responders work (scheduler → kubelet → containers).
- On-site tech calls back hourly with status; if they go silent, logistics reassigns (kubelet heartbeat → kubelet status → eviction).

## 2. Vocabulary Translation Table

| Kubernetes Term | T-Mobile / TIBCO / Incident Equivalent | What It Really Means |
|---|---|---|
| **API Server** | TIBCO EMS or RabbitMQ broker; IC command phone line | Central request router; all state changes funnel through it |
| **etcd** | Incident call recording (the authoritative chronology) | Distributed log with Raft quorum; survives 1 node death out of 3+ |
| **kube-controller-manager** | Downstream subscriber system (reads EMS, reacts) | Feedback loop: watches state, acts, writes result, watches again |
| **Reconciliation loop** | Notification retry logic (keep trying until ack'd) | Level-triggered: "goal=3 replicas, actual=2, POST Pod" (repeat every 15s) |
| **Scheduler** | Load balancer placement algorithm (NodePort → backend) | Filter (drop unsuitable nodes) → Score (rank good ones) → Assign (pick top scorer) |
| **Kubelet** | Notification handler process on a server (receives job, runs it) | Local daemon; executes PodSpecs, reports status, handles eviction |
| **Event** | Chronology entry in incident log ("12:45 PM: Page sent, ack'd at 12:47") | Ephemeral record (24 hrs default); explains why API Server made a decision |
| **ResourceVersion** | Incident log sequence number (chronology ordering) | Optimistic locking key; etcd uses it to detect concurrent writes |
| **Admission controller** | Compliance officer (audits requests before IC signs off) | Webhook runs *before* API Server commits state; can reject or mutate |

## 3. The Five Components: 30-Second, 2-Minute, and 5-Minute Depths

### 3.1 API Server (kube-apiserver)

**30 seconds**: Single gatekeeper. All requests (create Pod, scale Deployment, check status) go here. API Server says yes or no, then writes to etcd.

**2 minutes**: 
- REST API: `kubectl` and controllers POST/GET/WATCH to this server.
- Stateless (scales horizontally behind a load balancer).
- Authentication (is this user valid?) → Authorization (does this user own this Namespace?) → Admission (does this request break policy?).
- Audit logging: every request is recorded (who did what, when).
- No Pod is created, updated, or deleted without API Server approval.

**5 minutes**:
- API Server is the bottleneck. It runs on 3+ control plane nodes (HA). If all API Server instances crash, Kubernetes stops accepting requests — *but existing Pods keep running*.
- Each API Server instance has a cache layer (read-through cache of etcd state). Writes always go to etcd first, then cache updates.
- API Server is CPU-bound for large clusters (10,000+ Pods). Horizontal scaling and request batching help.
- The `--max-requests-inflight` flag sets the queue depth. If API Server is overwhelmed, requests are rejected with HTTP 429 (too many requests). This is your first sign of control plane overload.
- Certificates: API Server has a TLS cert (used by kubelet and controllers to talk to it). If the cert expires, no component can authenticate. This is a real production incident (happened at T-Mobile Week 2 2024).

### 3.2 etcd (Distributed State Store)

**30 seconds**: A key-value database that holds *all* Kubernetes state (Pods, Services, ConfigMaps, Secrets, everything). Uses Raft consensus so 2+ nodes can die and the cluster survives.

**2 minutes**:
- Raft quorum: 3-node etcd cluster can lose 1 node and stay live. 5-node can lose 2, etc. (always: (N+1)/2 must be healthy).
- Writes are slow (replicate to quorum, then return to caller). Reads are fast (serve from local cache).
- Default size limit: 2 GB per cluster (soft limit; can grow past but performance degrades). Each Pod, Service, ConfigMap, Secret takes bytes.
- Compaction: etcd keeps a history of all writes (like git log). History is pruned every hour (default). If you never compact, disk fills.
- Backup: etcd state file is the *entire cluster backup*. Backup etcd every hour. Restore: `etcdctl snapshot restore` + replace data directory.
- Keys are hierarchical: `/registry/pods/default/nginx-abc123` — this structure is query-able (list all Pods in a Namespace).

**5 minutes**:
- Raft leadership election: if the current leader (etcd master) crashes, Raft picks a new leader from the surviving quorum. This takes ~15 seconds (default election timeout). During this window, etcd rejects writes; reads still work (against follower cache).
- Quorum loss is catastrophic: if 2 out of 3 etcd nodes crash, the cluster halts. You must restore from backup. This has happened in production (failed simultaneous reboot of 2/3 control planes).
- WAL (write-ahead log): etcd writes all changes to disk before returning success. This is why etcd can be slow (~100 writes/sec max on local disks; up to 10k/sec on NVMe).
- Blast radius: one etcd outage = **all** cluster state is unreadable. API Server cannot respond to ANY request. Existing Pods keep running (kubelet doesn't care), but you cannot create, update, or delete anything. This is a P1 (full control plane down).

### 3.3 kube-controller-manager (Reconciliation Engine)

**30 seconds**: 30+ independent controller loops. Each watches a resource type (e.g., Deployment) and ensures the actual state matches the desired state. If it doesn't, the controller takes action.

**2 minutes**:
- The Deployment controller watches all Deployments. When you create `deployment: nginx replicas: 3`, the controller sees it, then creates 3 ReplicaSet Pod templates, which triggers the ReplicaSet controller to create 3 Pods.
- Level-triggered (not event-triggered): controllers run a loop every 15 seconds (configurable). They say "goal is 3 Pods, actual is 2 Pods — I will create 1 Pod" — regardless of whether a Pod *just* died or died 10 days ago. This is resilient to lost notifications.
- The Node controller watches Nodes. If a Node goes unresponsive for 5 minutes (no heartbeat), the Node controller marks it as `NotReady`, then evicts all Pods (unless they have local storage or are critical).
- Controllers are *not* ordered; they run independently. If Deployment controller and ReplicaSet controller both try to create Pods at the same time, the `resourceVersion` field (optimistic locking) prevents double-creation.

**5 minutes**:
- Each controller is a state machine: WATCH → RECONCILE → REPEAT.
- WATCH: `kubectl get Deployment -w` (watch all changes to Deployments in all Namespaces).
- RECONCILE: for each Deployment, compare spec.replicas (desired) vs. status.replicas (actual). If they differ, POST new Pods or DELETE excess Pods.
- REPEAT: wait 15 seconds, go to WATCH.
- Reconciliation is *idempotent*: running it twice has the same effect as running it once. This is why level-triggered is safe (no duplicate Pods even if the controller restarts mid-reconciliation).
- The `--controllers` flag lets you disable individual controllers (e.g., `--controllers=-ServiceAccount` to let Admission handle ServiceAccount creation instead). This is rare but used for special cases.
- Single point of failure: only one kube-controller-manager should be active at a time (leader election via lease, stored in etcd). If the active controller crashes, the standby controller takes over within 15 seconds (default lease renewal interval).

### 3.4 kube-scheduler (Pod Placement Algorithm)

**30 seconds**: Watches all Pods that are `Pending` (not yet assigned to a Node). For each Pending Pod, the scheduler finds the best Node and assigns it.

**2 minutes**:
- Two-phase algorithm: (1) Filter phase — remove Nodes that cannot run the Pod (e.g., taint doesn't match toleration, not enough CPU, wrong zone). (2) Score phase — rank the remaining Nodes by fit (prefer Nodes with most available CPU, fewest Pods already, preferred topology).
- Binding: once the scheduler picks a Node, it updates the Pod with `nodeName: node-01` in etcd. The kubelet on node-01 notices this change (WATCH on Pods assigned to it) and pulls the container image, then starts the container.
- The scheduler is pluggable: you can add custom scoring plugins (e.g., "prefer Nodes in AZ us-east-1a" or "spread Pods across racks").
- Scoring plugins give each Node a score (0–100). If two Nodes score equally, the scheduler picks alphabetically (deterministic tie-breaking).

**5 minutes**:
- Filter plugins are checked in order (fast fail):
  1. **PodFitsResources**: Does the Node have free CPU/memory for this Pod?
  2. **PodFitsHost**: Does the Node have the hardware (e.g., GPU)?
  3. **PodMatchNodeSelector**: Does the Node match the Pod's `nodeSelector` label requirement?
  4. **TaintToleration**: Does the Pod tolerate the Node's taints?
  5. **NodeAffinity**: Does the Pod's affinity rules match the Node's labels?
  6. **Others**: port conflicts, storage constraints, etc.
- Scoring plugins are applied to all remaining Nodes, then scores summed:
  1. **NodeResourcesFit**: Nodes with more free CPU/memory score higher (bin packing: prefer full Nodes to spread load).
  2. **ImageLocality**: Nodes that already have the Pod's image score higher (save pull time).
  3. **InterPodAffinity**: Pods with affinity rules score higher if placed near co-located Pods.
  4. **Others**: zone, topology, operator-defined plugins.
- Scheduler performance: for a 1000-Node cluster, scheduling a Pod takes ~100 ms (fast). Scheduling 1000 Pods at once takes ~100 seconds (rate-limited). If Pods are queued for >1 min, the scheduler is the bottleneck (increase parallelization in plugin config).

### 3.5 kubelet (Local Executor)

**30 seconds**: One kubelet per Node. Receives Pod assignments from the API Server (WATCH), pulls container images, starts containers, monitors them, and reports back to the API Server ("Pod is Running").

**2 minutes**:
- kubelet runs as a systemd service on each Node (not in a container).
- kubelet watches the API Server for Pods assigned to `nodeName: this-node`. When a new Pod appears, kubelet pulls the image, starts the container, and reports status.
- If a container crashes, kubelet restarts it (respecting the Pod's restartPolicy: Always, OnFailure, Never).
- Health checks: kubelet runs liveness probes (is the container alive?) and readiness probes (is the container ready for traffic?). If a liveness probe fails 3 times, kubelet restarts the container. If a readiness probe fails, kubelet marks the Pod as `NotReady` (removes from Service endpoints).
- Eviction: if the Node runs out of disk or memory, kubelet evicts Pods (highest QoS first). QoS tiers: Guaranteed (requests == limits) → Burstable (requests < limits) → BestEffort (no requests/limits).

**5 minutes**:
- kubelet is stateless: if it restarts, it queries the API Server for its assigned Pods and re-initializes them.
- Kubelet talks to the container runtime (Docker, containerd, CRI-O) via the Container Runtime Interface (CRI). This is a gRPC interface; kubelet doesn't care which runtime is underneath.
- Health checks are configurable per container:
  - **Liveness**: "is the app alive?" — if fails, kubelet restarts.
  - **Readiness**: "is the app ready?" — if fails, kubelet removes from Service endpoints (traffic stops).
  - **Startup**: "is the app booted?" — like readiness but only runs once at startup.
- Each probe is an HTTP GET, TCP socket check, or exec command. Default timeout: 1 second. Default failure threshold: 3 consecutive failures.
- Node status: kubelet sends periodic heartbeats to the API Server. If heartbeats stop for 5 minutes (default eviction timeout), the Node controller evicts all Pods. This is why even a single Node outage can drain your workload (unless Pods are pinned to that Node with nodeAffinity).
- Kubelet memory: kubelet uses ~100–200 MB on a light Node. Each Pod adds ~5 MB overhead. For a 100-Pod Node, expect kubelet to use ~500 MB.

## 4. Component Interactions: The Reconciliation Loop

When you run `kubectl create deployment nginx --image=nginx --replicas=3`:

1. **kubectl** → **API Server**: POST `/apis/apps/v1/deployments` with `{spec: {replicas: 3, ...}}`.
2. **API Server** → **etcd**: writes `Deployment/default/nginx` with `spec.replicas=3` and `status.replicas=0`.
3. **API Server** → **kubectl**: returns `201 Created`.
4. **kube-controller-manager** (Deployment controller) WATCHES the Deployment. It sees `spec.replicas=3, status.replicas=0`. It POSTs 3 ReplicaSets to the API Server.
5. **API Server** → **etcd**: writes 3 ReplicaSet objects.
6. **kube-controller-manager** (ReplicaSet controller) WATCHES ReplicaSets. It sees each ReplicaSet with `spec.replicas=1, status.replicas=0`. It POSTs 3 Pod objects to the API Server.
7. **API Server** → **etcd**: writes 3 Pod objects with `nodeName` field empty (unassigned).
8. **kube-scheduler** WATCHES Pods. It sees 3 Pods with `nodeName` empty. For each Pod, it runs the filter+score algorithm, picks a Node, and writes `nodeName: node-01` (etc.) back to the API Server.
9. **API Server** → **etcd**: updates each Pod with its assigned Node.
10. **kubelet** (on node-01, node-02, node-03) WATCHES Pods assigned to each Node. It sees the new Pod. It pulls the nginx image (if not cached) and starts a container.
11. **kubelet** → **API Server**: sets `pod.status.phase=Running`, `pod.status.containerStatuses[0].ready=true`.
12. **API Server** → **etcd**: updates the Pod status.
13. **kubectl get pods** queries the API Server, which reads etcd. All 3 Pods show `Running`.

**Key insight**: Every step is asynchronous. No component waits for the next. If the scheduler crashes at step 8, Pods stay `Pending`. When the scheduler restarts, it picks up from step 8 (reads etcd, finds unassigned Pods, schedules them). This is *level-triggered resilience*.

## 5. Director / VP Governance Lens: Five Questions You Own

### 5.1 etcd Backup and Disaster Recovery

**Your question**: "If etcd dies, how fast do we restore? What data is lost?"

**Answer to give in meetings**:
- Backup etcd every hour (automated snapshot). Store backups in S3 (separate from cluster).
- Recovery RTO: 15–30 minutes (new etcd cluster + restore snapshot + API Server re-initialization).
- RPO: 1 hour (everything created in the last hour before crash is lost).
- Current posture: (know your actual backup policy — if you don't have one, this is a risk).
- Implication: etcd failure = all Pods keep running, but you cannot create/delete/update for 15–30 min. Service mesh ingress keeps routing to running Pods.

### 5.2 Admission Controller Change Process

**Your question**: "We want to add a new admission webhook (e.g., validate all Images are from our registry). What's the risk?"

**Answer to give in meetings**:
- Admission controllers run *before* state is persisted. If a webhook is misconfigured (always rejects, or crashes), all Pod creation halts.
- Mitigation: admission webhooks must have a `failurePolicy: Ignore` (in dev/test), then switch to `failurePolicy: Fail` only after validation testing.
- Before deploying to production, test the webhook in staging: create Pods with various image sources, verify the webhook correctly allows/rejects.
- CI/CD integration: add a `pre-commit` check: "does the new admission rule break any existing Pods?" (e.g., run `kubectl apply --dry-run=server` against the current cluster state with the new rule).
- Current posture: (list your current admission controllers and their failure policies).

### 5.3 API Server SLA and Capacity Planning

**Your question**: "What's our API Server latency? Are we at capacity?"

**Answer to give in meetings**:
- Monitor `apiserver_request_duration_seconds` (p50, p95, p99). Target: <100 ms p99 for all requests.
- If p99 > 500 ms, the API Server is CPU-bound or etcd is slow. Investigate:
  - Are you watching too many resources? (e.g., `kubectl get all --all-namespaces -w` on a client).
  - Is etcd disk I/O contention (check disk utilization)?
  - Are there slow queries (e.g., listing all Secrets across all Namespaces)?
- Capacity: 1 API Server handles ~1000 requests/sec. 3 API Servers handle ~3000 requests/sec. For a cluster with 10,000 Pods, expect ~10 requests/sec (controllers watching, occasional user queries). You have headroom.
- Current posture: (run `kubectl top nodes` and `kubectl logs -n kube-system kube-apiserver-*` to check your actual metrics).

### 5.4 Secrets Encryption and Key Rotation

**Your question**: "Are our Secrets encrypted at rest in etcd?"

**Answer to give in meetings**:
- By default, etcd stores Secrets in plaintext. Add `--encryption-provider-config` to API Server to encrypt (AES-CBC or AES-GCM).
- Encryption key: stored on the control plane (usually in `/etc/kubernetes/pki/`). If an attacker gains control plane file access, they can read the key and decrypt all Secrets.
- Best practice: use AWS Secrets Manager (or similar) as the encryption provider (keys stored in HSM, never on-disk). This requires KMS plugin.
- Key rotation: Kubernetes supports online key rotation (rotate without downtime). Plan this quarterly.
- Current posture: (check if `--encryption-provider-config` is set in your API Server; if not, Secrets are plaintext in etcd — P2 security issue).

### 5.5 AI Controller Blast Radius Governance

**Your question**: "We're deploying a custom controller (e.g., AI workload auto-scaler). If it misbehaves, what breaks?"

**Answer to give in meetings**:
- Custom controllers are *dangerous*: a buggy controller can delete all Pods in a Namespace or scale all Deployments to 0.
- Governance: (1) code review by 2 SREs, (2) test in staging first (chaos test: kill the controller, verify recovery), (3) deploy with RBAC least-privilege (controller only has write access to specific CRDs, not Pods/Deployments).
- Blast radius isolation: use Namespace-scoped controllers (not cluster-scoped) when possible. A namespace-scoped controller bug affects only that Namespace.
- Monitoring: alert if controller is CPU-bound or making unusual API requests (e.g., 1000s of DELETE requests/sec).
- Current posture: (list your custom controllers and their RBAC roles; identify any overprivileged ones).

## 6. Comprehension Check (Unlock Lab)

**Read the 5-minute depth sections above, then answer 2/3 correctly to unlock the lab.**

### Question 1: Reconciliation Loop Resilience

**Scenario**: A Deployment with `replicas: 5` is running. A buggy controller crashes the Deployment controller process. 1 minute later, the controller restarts.

**Question**: Will the Deployment eventually reach 5 replicas again? Why or why not?

**Answer**: YES. The Deployment controller uses level-triggered reconciliation. When it restarts, it queries etcd, finds the Deployment with `spec.replicas=5`, sees that only 3 Pods are running, and creates 2 more Pods. This happens regardless of whether the controller crashed before, during, or after the previous reconciliation. No edge-triggered notification is needed; the state is the source of truth.

---

### Question 2: etcd Quorum Loss

**Scenario**: You have a 3-node etcd cluster. Both follower nodes crash simultaneously (power loss). The leader node survives.

**Question**: Can the cluster still write to etcd? Can it read?

**Answer**: NO writes, maybe reads. Raft requires a quorum (2 out of 3). With 2 nodes dead, there is no quorum, so the leader cannot commit writes. Writes are rejected. Reads might work against the leader's cache, but the cache is stale (etcd hasn't heard from followers, so it doesn't know the followers' state). This is treated as a write/read failure. The cluster is **functionally down** despite the leader surviving. You must restore from backup or add new etcd nodes and rebuild the cluster.

---

### Question 3: kubelet Eviction Cascade

**Scenario**: A Node runs out of disk space (95% full). kubelet evicts Pods to free space. You have:
- Pod A: BestEffort (no CPU/memory requests)
- Pod B: Burstable (requests: 100m, limits: 500m)
- Pod C: Guaranteed (requests: 200m, limits: 200m)

**Question**: Which Pod is evicted first? Why does this matter?

**Answer**: Pod A (BestEffort) is evicted first, then Pod B, then Pod C. This is the QoS ordering: lower-QoS Pods are sacrificed first to keep higher-QoS Pods running. This matters because critical system components (like istio-proxy, log collectors) often run as Guaranteed Pods. If you evict them, you lose observability and traffic handling. Eviction order protects critical workloads.

### Lab Step 1: Create a 3-Node EKS Cluster

**Why**: EKS clusters have a managed control plane (AWS manages etcd, API Server, controller-manager, scheduler). You can see them via `kubectl` even though you don't have direct SSH access.

**Command**:
```bash
# Set region
export AWS_REGION=us-east-1

# Create cluster (takes 10–15 minutes)
eksctl create cluster \
  --name control-plane-lab \
  --region $AWS_REGION \
  --version 1.30 \
  --nodegroup-name general \
  --node-type t3.medium \
  --nodes 3 \
  --nodes-min 1 \
  --nodes-max 4
```

**EXPECTED_OUTPUT**: After 15 minutes:
```
2026-06-11 10:45:32 [✓]  EKS cluster "control-plane-lab" in "us-east-1" region is ready
2026-06-11 10:46:12 [✓]  Created managed nodegroup "general" in EKS cluster "control-plane-lab"
```

**WHY**: eksctl abstracts EC2, VPC, IAM, and EKS resource creation. It outputs a kubeconfig file to `~/.kube/config`, which kubectl uses to connect to your cluster's API Server.

### Lab Step 3: Deploy a Deployment and Watch Reconciliation

**Command**:
```bash
# Create a Deployment
kubectl create deployment nginx --image=nginx:latest --replicas=3

# Watch Pods come up (in a separate terminal, run this while Pods are creating)
kubectl get pods -w

# After all Pods are Running, check the Deployment status
kubectl get deployment nginx -o yaml
```

**EXPECTED_OUTPUT** (from `kubectl get pods -w`):
```
NAME                     READY   STATUS              RESTARTS   AGE
nginx-5d59d67564-abc12   0/1     ContainerCreating   0          3s
nginx-5d59d67564-def34   0/1     ContainerCreating   0          2s
nginx-5d59d67564-ghi56   0/1     ContainerCreating   0          1s
nginx-5d59d67564-abc12   1/1     Running             0          8s
nginx-5d59d67564-def34   1/1     Running             0          7s
nginx-5d59d67564-ghi56   1/1     Running             0          6s
```

**WHY**: This shows the reconciliation loop in action:
1. You create the Deployment (1 step).
2. The Deployment controller creates 3 ReplicaSets (1 step).
3. The ReplicaSet controller creates 3 Pods (1 step).
4. The scheduler assigns each Pod to a Node (3 steps).
5. The kubelet on each Node pulls the image and starts the container (3 steps, parallel).

Each step is separate; if any step stalls, you can debug the bottleneck (e.g., image pull slow → Pod stuck in ContainerCreating).

### Lab Step 4: Trigger Pod Deletion and Observe Auto-Recreation

**Command**:
```bash
# Delete one Pod
kubectl delete pod nginx-5d59d67564-abc12

# Watch the reconciliation loop (pod deleted, then recreated)
kubectl get pods -w

# After the Pod restarts, verify the Deployment goal is met
kubectl get deployment nginx
```

**EXPECTED_OUTPUT**:
```
# After deleting the Pod:
nginx-5d59d67564-abc12   1/1     Terminating         0          3m
# Within 2 seconds:
nginx-5d59d67564-jkl78   0/1     Pending             0          0s
nginx-5d59d67564-jkl78   0/1     ContainerCreating   0          2s

NAME    READY   UP-TO-DATE   AVAILABLE   AGE
```

**WHY**: This demonstrates level-triggered reconciliation:
- The ReplicaSet controller sees `spec.replicas=3, actual=2` (after you delete one).
- It immediately creates a replacement Pod.
- The replacement Pod is assigned a new name (random suffix).
- All 3 Pods are running again within 6 seconds.
- The controller didn't wait for a notification; it just polled etcd every 15 seconds and found the mismatch.

```
default     nginx-5d59d67564-abc12.1234567890def1  ...  3m11s  Successfully pulled image "nginx:latest"

default     nginx-5d59d67564-abc12.1234567890def1  ...  3m9s   Started container nginx
```

**WHY**:
- Events are the *audit trail* of the reconciliation loop. They explain why the API Server made decisions.
- resourceVersion is an incrementing sequence number. Each etcd mutation bumps it. Controllers use it to detect stale reads (optimistic locking).
- Events are **ephemeral** (deleted after 24 hours by default). This is why `kubectl describe` shows only recent events.

### Lab Step 6: Pin Pods to Nodes with nodeSelector

**Command**:
```bash
# List nodes with their labels
kubectl get nodes --show-labels

# Pick one node and note its name (e.g., ip-...-1.ec2.internal)
NODE_NAME=$(kubectl get nodes -o jsonpath='{.items[0].metadata.name}')

# Create a Pod with a nodeSelector (pins to one specific Node)
kubectl run pinned-nginx \
  --image=nginx:latest \
  --overrides '{"spec":{"nodeSelector":{"kubernetes.io/hostname":"'$NODE_NAME'"}}}'

kubectl delete pod pinned-nginx
```

**EXPECTED_OUTPUT**:

```
pinned-nginx    0/1     ContainerCreating 0          3s      ip-...-1.ec2.internal

# The scheduler assigned it to the specified Node because it matched the nodeSelector label.
```

**WHY**:
- The scheduler's filter phase checks: "Does this Node have the label `kubernetes.io/hostname: ip-...-1.ec2.internal`?" Yes → passes filter.
- All other Nodes fail the filter (don't have the matching label).
- The Pod is assigned to the matching Node.
- This is how you pin critical workloads to specific Nodes (e.g., GPU workloads to GPU Nodes).

### Failure: "Nodes are NotReady"

**Cause**: Nodes haven't joined the cluster (kubelet is not running or cannot authenticate).

**Diagnostics**:
```bash
# Check node status
kubectl get nodes

# Check kubelet logs on the node (if you have SSH access)
# Or check events:
kubectl describe node <node-name>

# Check if kube-proxy DaemonSet is running
kubectl get daemonset -n kube-system kube-proxy
```

```bash

```

### Failure: "componentstatus endpoint deprecated"

**Cause**: Kubernetes 1.19+ removed the `componentstatus` API (it reported etcd, controller-manager, scheduler health, but was not reliable).

**Fix**: Ignore the warning. Instead, check health via logs:
```bash
kubectl logs -n kube-system -l component=kube-controller-manager | tail -10
kubectl logs -n kube-system -l component=kube-scheduler | tail -10
```

### Failure: "Pod stuck in Pending"

**Cause**: The scheduler couldn't find a suitable Node (filter phase failed).

**Diagnostics**:
```bash
kubectl describe pod <pod-name>
# Look for:
# - Reason: Unschedulable
# - Message: 0/3 nodes are available: 1 Insufficient memory, 2 node(s) had taint ..., ...

# Check node resources
kubectl top nodes
kubectl describe node <node-name>

# Check Pod resource requests
kubectl get pod <pod-name> -o yaml | grep -A 5 "resources:"
```

**Fix**: Either (a) add more Nodes, (b) reduce Pod resource requests, or (c) remove taints from Nodes:
```bash
# Taint a node (often done for special workloads):
kubectl taint nodes <node-name> key=value:NoSchedule

# Remove a taint:
kubectl taint nodes <node-name> key=value:NoSchedule-
```

---

### Failure: "Pod not recreated after deletion"

**Cause**: The Pod is not managed by a ReplicaSet or Deployment (you created it with `kubectl run` without a controller).

**Diagnostics**:
```bash
kubectl get pod <pod-name> -o yaml | grep -A 5 "ownerReferences:"
# If ownerReferences is empty, the Pod has no owner controller.
```

**Fix**: Use a Deployment instead:
```bash
kubectl create deployment <name> --image=<image>
# This creates a Deployment, which owns a ReplicaSet, which owns Pods.
```

---

### Failure: "Scheduler not assigning Pods to nodeSelector nodes"

**Cause**: The node label doesn't exist or is misspelled.

**Diagnostics**:
```bash
# List all node labels
kubectl get nodes --show-labels

# Check the Pod's nodeSelector
kubectl get pod <pod-name> -o yaml | grep -A 3 "nodeSelector:"

# Check if the label matches (case-sensitive)
```

**Fix**: Create the label on the node, then recreate the Pod:
```bash
# Add a label to a node
kubectl label nodes <node-name> custom-label=true

# Verify the label
kubectl get nodes --show-labels | grep custom-label

# Now create a Pod with nodeSelector: custom-label=true
```

## 9. Interview Questions: Staff / Principal Level

These questions are commonly asked in loop interviews for Staff SRE / Principal Engineer roles. Use the STAR method anchored to T-Mobile.

### Question 1: "Walk us through what happens when a Pod is created. Where are the failure points?"

**STAR Setup**: At T-Mobile, I manage the notification platform (25M msg/day). When we scaled from 3 to 5 replicas on the message-queue-consumer Deployment, we discovered scheduling was the bottleneck. The scheduler was assigning Pods to Nodes with high memory fragmentation, causing unnecessary evictions.

**Answer** (5 minutes):
1. **Request entry**: `kubectl create deployment` calls the API Server (or a Deployment spec is applied via GitOps).
2. **Authentication + Authorization**: API Server verifies the user (RBAC role check). This is fast (<10 ms).
3. **Admission webhooks**: Any custom webhooks run (e.g., image signature validation, network policy injection). This is a common bottleneck (slow webhook = slow Pod creation).
4. **Persistence**: API Server writes to etcd (Raft replication, ~10 ms). If etcd is slow (disk I/O contention), Pods queue up.
5. **Controller reconciliation**: Deployment controller sees the new Pod spec, creates ReplicaSets. ReplicaSet controller creates Pods. Both use level-triggered loops (~15s cycle).
6. **Scheduling**: Scheduler filters Nodes (PodFitsResources, TaintToleration, etc.), scores them, assigns. This is fast for <100 Nodes, slow for 1000+ Nodes.
7. **kubelet execution**: kubelet pulls the image (this can be slow: 5–30s depending on image size and network), starts the container, reports status.

**Failure points** (in order of frequency at T-Mobile):
- (A) Image pull timeout (Node has old image cache, new image is unavailable in registry).
- (B) Admission webhook timeout (we had a custom webhook blocking Pod creation for 30s once).
- (C) Scheduler stuck (rare; usually indicates Node resources exhausted or scheduler bug).
- (D) etcd slow (indicates I/O contention or etcd quorum issues).
- (E) kubelet stuck (container runtime hung, usually indicates OOM or disk full on Node).

**Mitigation**:
- Image pull: pre-pull images on Nodes during deployment (via DaemonSet or node initialization).
- Admission webhooks: set timeouts and `failurePolicy: Ignore` in staging; monitor webhook latency.
- Scheduler: monitor `scheduler_scheduling_duration_seconds` p95; scale API Servers if p95 > 500ms.
- etcd: monitor etcd_disk_io_duration, etcd_server_commit_duration; add NVMe to control plane if needed.
- kubelet: monitor kubelet_cgroup_manager_duration, container_runtime_* metrics; tune QoS eviction.

**Tradeoff**: We chose to pre-pull images (saves 10s per Pod creation) but this means updating images requires a 15-minute node drain cycle (pull time). For canary deployments, we accept slower Pod creation to avoid image cache inconsistencies.

### Question 2: "How would you diagnose and fix an etcd quorum loss incident?"

**STAR Setup**: At T-Mobile, our Kubernetes platform supports 50 services across 5 teams. In Q2 2024, we had a power outage in our data center. Two of our three etcd nodes (on separate power supplies, supposedly) went down simultaneously due to a shared UPS misconfiguration. Within 2 minutes, the cluster froze (no Pod creation, no scaling, no service updates).

**Answer** (5 minutes):
**Diagnosis** (first 5 minutes):
```bash
# Step 1: Verify etcd quorum is lost
kubectl get nodes   # Hangs or returns stale data
kubectl get pods    # Stale data (old cached results)

# Step 2: Check etcd cluster health (if you have SSH to control plane)
etcdctl member list   # Will show 2/3 members healthy
etcdctl endpoint health  # Timeouts on the follower nodes

# Step 3: Check if control plane Pods are restarting
kubectl logs -n kube-system etcd-*  # Will show Raft election errors

# Step 4: Alert: cluster is FROZEN. Services keep running, but no orchestration possible.
```

**Immediate response** (5–15 minutes):
- **Do not try to fix etcd in place.** Quorum loss = unrecoverable without external intervention.
- **Notify teams**: "Cluster is frozen; no Pods can be created or scaled. Existing Pods are unaffected."
- **Prepare rollback**: Set a 30-minute SLO to restore (either recover etcd or switch to backup cluster).

**Recovery** (15–60 minutes):
```bash
# Option 1: Add a new etcd node to the quorum (if you have infra space)
# This requires:
# 1. Boot a new node with the same Kubernetes version
# 2. etcdctl member add <new-node>
# 3. Initialize etcd on the new node with `--initial-cluster-state existing`
# This is complex and takes 20–30 minutes.

# Option 2: Restore from backup (faster)
# 1. Backup is stored in S3 (we rotate daily).
# 2. Boot a new cluster from the latest backup.
# 3. Repoint DNS / load balancer to the new cluster's API Server endpoint.
# 4. This takes 10–15 minutes (backup restore + API Server initialization).

# Option 3: Force a new quorum (nuclear option)
# 1. Keep the surviving leader node, discard the 2 dead followers.
# 2. Reset the cluster UUID and member list.
# 3. This works but loses all state after the last backup.
```

**What we did** (at T-Mobile):
- We had a 1-hour-old backup in S3 (every-hour rotation).
- We chose Option 2 (restore from backup).
- 12 minutes to restore backup, 3 minutes to re-initialize API Servers.
- 15 total minutes downtime. RPO: 1 hour (all Pods and Secrets created in the last hour were lost, but we recreated them from GitOps — Deployments re-synced automatically).

**Prevention**:
- **Don't place all 3 etcd nodes on the same UPS.** Use separate power supplies / availability zones.
- **Backup etcd every hour.** Store backups in a different region (S3 with cross-region replication).
- **Test recovery.** Run a monthly drill: restore the latest backup to a test cluster, verify cluster state is correct.
- **Monitor etcd health.** Alert if any etcd member is down for >5 minutes.

### Question 3: "A team deployed a custom controller that is crashing in production. It's deleting Pods in random Namespaces. How do you respond?"

**STAR Setup**: At T-Mobile, a team deployed a custom Kubernetes controller (written in Python) to auto-scale GPU workloads based on job queue depth. The controller had an `if pod.namespace not in [protected]` check, but `protected` was an empty list. The controller started deleting Pods across all Namespaces (including the log collector, Prometheus, etc.). Within 3 minutes, critical observability was down, and we lost visibility into the incident.

**Answer** (5 minutes):
**Immediate action** (first 30 seconds):
```bash
# Step 1: KILL the controller
kubectl delete deployment <controller-name> -n <namespace>
# OR: scale it to 0 replicas to stop it from running new reconciliations
kubectl scale deployment <controller-name> --replicas=0

# Step 2: Restore deleted Pods
# Since the controller is stopped, Deployments/ReplicaSets will recreate any missing Pods.
# This takes 1–5 minutes depending on image pull.

# Step 3: Verify observability is restored
kubectl get pods -n kube-system  # Should see log collector, Prometheus back online
kubectl logs -n kube-system deployment/prometheus-operator  # Verify logs are flowing
```

**Diagnosis** (next 5 minutes):
```bash
# Step 4: Check the controller logs
kubectl logs deployment/<controller-name> -n <namespace> --previous  # Old logs before crash
# Look for: "Deleting Pod X in Namespace Y" messages

# Step 5: Check which Pods were deleted
kubectl get events -A --sort-by='.lastTimestamp' | grep -i delete
# This shows which Pods were recently deleted and why

# Step 6: Review the controller code
# Find the RBAC role granting the controller permissions
kubectl get role,rolebinding -A | grep <controller-name>
# Expected: the controller should only have DELETE permission on specific resource types or Namespaces
```

**Root cause analysis** (next 15 minutes):
- **Insufficient testing**: controller didn't have a staging environment to test against a multi-team cluster.
- **RBAC too permissive**: the controller had `verbs: [delete] resources: [pods]` in all Namespaces. Should be: `namespaces: [gpu-workloads]` only.
- **No admission webhook**: no policy to prevent controllers from having cluster-wide delete permissions.

**Long-term fixes**:
1. **RBAC governance**: 
   - All custom controllers must have least-privilege RBAC (Namespace-scoped or resource-type-scoped).
   - Use OPA/Gatekeeper to enforce: "no controller can have delete permissions on Pods across all Namespaces."
2. **Code review process**:
   - Custom controllers require 2-SRE approval before merge.
   - Checklist: RBAC role reviewed, image signed, staged in test cluster first.
3. **Chaos testing**:
   - Test the controller's behavior if it crashes mid-operation (e.g., deleted 2 Pods, then crashed; what state is left?).
   - Verify that stopping the controller doesn't leave orphaned resources.
4. **Observability**:
   - Add metrics: `custom_controller_pod_deletions_total`, `custom_controller_errors_total`.
   - Alert if deletion rate > 10 Pods/min (anomaly).

**Tradeoff**: Custom controllers are powerful (auto-scaling, custom orchestration) but risky. The tradeoff is: governance + testing cost vs. automation benefit. For a small team, it's not worth it (use managed services). For a large platform (100+ services), custom controllers can save months of work. At T-Mobile's scale (50 services), we allow 3–4 custom controllers, all with strict governance.

### Question 4: "Describe the tradeoffs between a single API Server and highly available API Servers (multiple replicas behind a load balancer)."

**Answer** (4 minutes):

| Aspect | Single API Server | HA (3+ API Servers) |
|---|---|---|
| **Deployment complexity** | Simple (1 node, no LB) | Complex (LB, certificates, leader election) |
| **Latency** | Low (no extra hops) | Slightly higher (LB routing) |
| **Throughput** | ~1000 req/sec | ~3000 req/sec (scales linearly) |
| **Availability** | SPOF: 1 crash = cluster down | Tolerate 1 crash (3/3 → 2/3) |
| **failover time** | Manual (roll new instance, repoint DNS) | Automatic (LB removes unhealthy instance in 30s) |
| **etcd dependency** | Single etcd is a SPOF too | etcd must be HA (3+ nodes) to benefit from HA API Servers |
| **Cost** | $100/month for 1 node | $300/month for 3 nodes + LB |
| **Certificate management** | Simpler (1 cert) | Complex (all API Servers must have matching cert, LB needs its own) |

**When to use single API Server**: Dev/test clusters, < 100 Pods, acceptable downtime = 1 hour.

**When to use HA API Servers**: Production clusters, > 1000 Pods, SLO uptime ≥ 99.9%.

**At T-Mobile**, we use 3 API Servers across 3 AZs, each with NVMe-backed etcd. Cost: ~$2k/month for the control plane infrastructure (not including Nodes for Pods). Tradeoff: worth it for 25M msg/day platform (any downtime = customer-facing impact).

### Question 5: "How would you implement a rate limit on API Server requests to prevent noisy neighbors?"

**Answer** (4 minutes):
**Scenario**: Team A runs a chatty application that makes 1000 API calls/sec to check Pod status (bad practice, should use watch). This exhausts the API Server, slowing down Team B's Pod creation.

**Solutions** (in order of complexity):

1. **Horizontal scaling** (brute force): 
   - Add more API Servers. Cheapest solution if you have the infra.
   - Con: doesn't fix the root cause (noisy app).

2. **Flow control** (new in Kubernetes 1.23):
   - `kubectl get flowschema` shows built-in priority buckets.
   - High-priority (leader election, kubelet heartbeats) get 100 req/sec.
   - Standard user requests (Pod creation) get 10 req/sec.
   - Tuning: `kubectl edit flowschema system-leader-election`.
   - Pro: granular, no per-team code changes.
   - Con: requires Kubernetes 1.23+ and tuning expertise.

3. **Quota per team**:
   - Assign ResourceQuota to each team's Namespace (limits CPU, memory, Pod count).
   - Con: doesn't limit API request rate (only runtime resource usage).

4. **Sidecar rate limiter** (application-level):
   - Team A's application should implement client-side rate limiting (exponential backoff).
   - Teach teams to use `kubectl get pods -w` (watch) instead of polling.
   - Pro: fixes the root cause.
   - Con: requires app code changes.

**At T-Mobile**, we use flow control (solution 2) for critical services (Prometheus, kubelet) and sidecar rate limiting (solution 4) for customer-facing apps. We also have an SRE runbook: "API Server p99 latency > 500ms → check `kubectl logs -n kube-system -l component=kube-apiserver | grep "request_duration"`" to identify the noisy neighbor.

## 10. Tradeoffs (Director-Level Discussions)

These are conversation starters for senior leadership and interview calibration.

### Tradeoff 1: Control Plane / Data Plane Decoupling Ambiguity

**The dilemma**: 
- The control plane (API Server, etcd, scheduler) is *logically* separated from the data plane (kubelet, container runtime).
- However, in practice, they're tightly coupled: if the control plane is unavailable, you can't create, update, or delete Pods. The data plane keeps running, but it's useless without orchestration.

**Question for your director**: "Do we treat control plane downtime as a P1 or P2? If P1, how much are we willing to spend on HA control plane infrastructure?"

**Answer framework**:
- **P1 (unacceptable)**: Control plane downtime loses money (e.g., can't scale up to handle traffic spike). Budget: 3 AZs, NVMe storage, hourly backups.
- **P2 (acceptable, RTO 1 hour)**: Control plane downtime is rare and slow (e.g., quarterly maintenance). Budget: single AZ, single etcd node, daily backups.

**At T-Mobile** (notification platform):
- Control plane downtime = messages can't be queued for delivery (P1).
- We spend $2k/month on HA control plane (3 AZ spread, NVMe etcd).
- RTO target: 15 minutes (restore from backup).
- RPO target: 1 hour (hourly backups).

---

### Tradeoff 2: etcd as Single Risk Concentration Point

**The dilemma**:
- All cluster state lives in etcd. A single etcd outage = entire cluster is frozen.
- You cannot distribute etcd across multiple independent systems (unlike a service mesh sidecar). Raft quorum requires consensus; there's no way around it.

**Question for your director**: "How do we accept etcd as a SPOF when we'd never accept a database SPOF in production?"

**Answer framework**:
- etcd is not actually a SPOF *if* you have (a) 3+ nodes across AZs + (b) hourly backups + (c) tested recovery procedure.
- The RTO/RPO is the risk metric: if RTO is 1 hour and you lose 1 hour of state, that's acceptable for orchestration. For a critical service, it might not be.

**Emerging alternatives**:
- **Kubernetes 1.28+** added alpha support for external etcd (managed by cloud provider, e.g., AWS DynamoDB). This shifts the risk to the cloud provider (they manage backups, replication).
- **Velero** (external backup tool) can backup etcd to S3 every 5 minutes. This lowers RPO from 1 hour to 5 minutes.

**At T-Mobile**:
- We still use Raft-based etcd (3 nodes, 3 AZs).
- We added Velero for 5-minute RPO (not just hourly).
- Cost: $200/month for Velero (storage + compute).
- Question for leadership: "Is 5-minute RPO worth $200/month, or is 1-hour RPO acceptable?" Answer: for a notification platform, 5-minute RTO is worth it (could be millions in lost messages).

---

### Tradeoff 3: Admission Webhook Availability Dependency

**The dilemma**:
- Admission webhooks run *before* API Server commits state. If a webhook crashes or times out, Pod creation stalls.
- You want webhooks for security (image signature validation, network policy injection, cost governance). But webhooks are an availability risk.

**Question for your director**: "Do we block Pod creation for security checks, or do we allow-by-default and audit-later?"

**Answer framework**:

| Approach | Availability | Security | Governance Effort |
|---|---|---|---|
| **No webhooks** | 100% (nothing blocks Pod creation) | Low (no validation) | Low (no enforcement) |
| **Webhooks, fail-open** | 100% (webhook times out → allow Pod) | Low (bad Pods can slip through) | Low (no strong enforcement) |
| **Webhooks, fail-closed** | ~99% (webhook times out → reject Pod) | High (all Pods must pass validation) | High (need webhook SLA, alerting) |

**Emerging pattern (GitOps governance)**:
- Don't use webhooks for enforcement (risk).
- Use webhooks for *audit* (log all Pods, process offline).
- Use policy-as-code (OPA/Gatekeeper) with `failurePolicy: Ignore` for non-critical rules.
- Enforcement happens at GitOps sync time, not at Pod creation time (controlled, testable, auditable).

**At T-Mobile**:
- We had an image signature validation webhook (fail-closed). It was timing out once a week, blocking Pod creation.
- We moved to OPA/Gatekeeper with `failurePolicy: Ignore` (audit only).
- We added a separate gate in our CI/CD: "before merging a Deployment, verify all images are signed."
- Benefit: Pod creation is never blocked; security is still enforced (just at a different stage).
