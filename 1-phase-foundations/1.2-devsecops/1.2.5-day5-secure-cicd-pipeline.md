# Week 2, Day 5: Secure CI/CD Pipeline — From Local Commit to Prod Deployment

## Overview

**Goal**: Understand and implement each security gate in a modern CI/CD pipeline — SAST (static code scanning), dependency scanning, container image scanning, secrets detection, and OIDC-based deployment authentication.

**Real-world context**: You govern CI/CD at T-Mobile for a 25M msg/day platform. A single leaked credential, unscanned dependency, or misconfigured IAM policy can compromise the entire notification system. This lab teaches the *why* before the *how* — why each gate exists, how to instrument it, how to explain gaps to a CISO.

**Time estimate**: 3–4 hours (concept + lab + discussion questions).

---

## Part 1: 30-Second Version — The Warehouse Quality Gate Analogy

Imagine T-Mobile's device distribution center in Kansas City. A phone arrives from manufacturing (a Git commit). It goes through five inspection gates before shipping:

1. **Gate 1 — Code defects** (SAST): Inspect the phone's software — does it have known bad patterns? (hardcoded secrets, SQL injection vulnerabilities)
2. **Gate 2 — Supply chain defects** (SBOM/dependency scan): Are all components (chipset, battery, charging port) from approved vendors? Any known recalls?
3. **Gate 3 — Container safety** (container image scan): Is the box sealed correctly? Are all internals accounted for? (no rootkits, no unexpected binaries)
4. **Gate 4 — Secrets/keys** (secrets detection): Did someone accidentally pack the master encryption key in the box? (leaked API keys, certificates)
5. **Gate 5 — Delivery proof** (OIDC): Before handing the box to FedEx, verify the FedEx driver's ID is real, and they're carrying a specific route number (GitHub repo OIDC token proves *this* repo, *this* branch, *this* CI job — not a phished replica).

**If ANY gate fails → stop, quarantine, investigate.** If all gates pass → deploy.

## Part 2: 2-Minute Version — Vocab Translation Table

| Security Term | Your Mental Model | Why It Matters |
|---|---|---|
| **SAST** (static application security testing) | Code review bot that never sleeps | Finds hardcoded passwords, SQL injection patterns before the code runs |
| **SBOM** (software bill of materials) | Parts list for the device (chipset, RAM, battery brands) | Know what's in your app — every library, version, known vulnerabilities |
| **Dependency scanning** | Checking if any component is on a recall list | Are we using log4j 2.14.1 (the RCE version)? |
| **Container image scanning** | X-ray of the shipping box — anything unexpected inside? | Rootkits, malware, config errors hidden in Docker layers |
| **Secrets scanning** | Border security — checking no one smuggled passwords in luggage | Git history often contains leaked API keys, Vault tokens |
| **OIDC** | Proof of delivery signature (you prove who *you* are, not just that you have a key) | GitHub proves *this* CI job, *this* repo, *this* branch — not a stolen token |
| **Policy-as-code** | Governance rules written in YAML instead of a PDF | Automate "all container images must be scanned" |
| **Artifact attestation** | Supply chain certification — "this binary was built by CI, not downloaded" | Block deployments that skip the pipeline |

## Part 3: 5-Minute Version — Full Architecture

### The Five Security Gates (with OWASP CI/CD Top 10 mapping)

**Gate 1: SAST (Static Code Analysis)**
- Tools: Semgrep, SonarQube, Snyk Code
- Scans the source code before it's compiled/deployed
- Finds: hardcoded secrets, SQL injection, XSS, insecure crypto
- OWASP A1: *Insufficient Access Control* — enforces code patterns that prevent privilege escalation
- **You govern:** How strict is the block? (fail-on-high? fail-on-medium?) Phased rollout to avoid alert fatigue.

**Gate 2: Software Composition Analysis (SCA) / Dependency Scanning**
- Tools: Snyk, Black Duck, FOSSA, Trivy
- Scans `requirements.txt`, `package.json`, `pom.xml` — every library your app depends on
- Finds: "you're using log4j 2.14.1, which has CVE-2021-44228 (RCE)"
- OWASP A6: *Insufficient Dependency Management* — the 2023 SolarWinds supply chain attack happened because no one was checking dependency versions
- **You govern:** Do you block on low-severity CVEs (might be noise), or only high+critical? What's your SLA for patching?

**Gate 3: Container Image Scanning**
- Tools: Trivy, Grype, Anchore
- Scans the Docker image layer-by-layer for vulnerabilities + malware
- Finds: rootkits, backdoored binaries, insecure base image (Ubuntu with 300+ unpatched CVEs)
- OWASP A4: *Insufficient Artifact Protection* — ensures images deployed to Kubernetes are scanned
- **You govern:** Can developers push to ECR before scan? (no — fails the gate). Do you scan on push or on pull? (both, with different policies).

**Gate 4: Secrets Scanning**
- Tools: git-secrets, TruffleHog, Semgrep secrets rules, detect-secrets
- Scans Git history + current code for API keys, certificates, database passwords
- Finds: a developer committed `VAULT_TOKEN=s.a1b2c3d4e5f6g7h8` to GitHub 6 months ago
- OWASP A9: *Unprotected Pipeline Configuration* — plaintext secrets in `.github/workflows/` files
- **You govern:** Do you rotate any credentials found? Do you block the commit or warn?

**Gate 5: Artifact Attestation / OIDC-based Deployment**
- Tools: GitHub Actions OIDC, SPIFFE/SPIRE, Sigstore Cosign
- Replaces hardcoded AWS credentials with *proof of identity*: "I'm GitHub Actions, running job `deploy-prod`, in repo `t-mobile/notification`, branch `main`"
- OWASP A10: *Inadequate Logging and Visibility* — you can audit exactly which job deployed what
- **You govern:** Can any repo in GitHub assume the AWS deployment role? (no). Can only `main` branch deploy to prod? (yes, with additional approval).

### The Pipeline Flow

```
[Developer pushes to GitHub]
       ↓
[GitHub Actions triggered]
       ↓
[Gate 1: SAST scan (Semgrep)]  ← Found hardcoded secret? FAIL
       ↓
[Gate 2: SCA scan (Trivy deps)]  ← Found log4j RCE? FAIL
       ↓
[Gate 3: Build Docker image]
       ↓
[Gate 4: Container image scan (Trivy)]  ← Found rootkit? FAIL
       ↓
[Gate 5: Secrets scan (git-secrets)]  ← Found AWS key in history? FAIL
       ↓
[Gate 6: OIDC token exchange]  ← Prove you're *this* repo, not a phished replica
       ↓
[Assume AWS role with time-limited creds]
       ↓
[Push to ECR, update Kubernetes manifest in ArgoCD]
       ↓
[ArgoCD syncs to EKS]
```

**Key insight**: Credentials NEVER appear in logs or GitHub Actions YAML. Instead, the CI job proves its identity (via OIDC), and AWS grants temporary credentials (15-min STS token).

## Part 5: Comprehension Check (Must Pass 2/3 Before Lab)

**Question 1**: You find a hardcoded AWS API key in a developer's commit from 3 weeks ago. The key has been rotated, but it existed in GitHub for those 3 weeks. What gates failed, and in what order should you have caught this?

**Answer**: All 5 gates should have caught this:
- Gate 1 (SAST): Semgrep detects hardcoded AWS key patterns
- Gate 4 (Secrets scan): git-secrets / TruffleHop scan the entire history
- If both failed, you have a process gap: either the developer skipped the pre-commit hook, or the rule didn't match the pattern

**Question 2**: Your SAST tool (Semgrep) is reporting 80 medium-severity issues per day. Your team is spending 3 hours/day triaging false positives (e.g., "hardcoded string in a test file that's never deployed"). How would you fix this?

**Answer**: Implement a phased rollout:
- Week 1: Run Semgrep in *advisory* mode (reports only, doesn't block).
- Week 2: Triage the top 10 high-severity issues. Fix them.
- Week 3: Enable *block* mode for high-severity only. Run medium-severity in advisory.
- Measure: track false positive rate in a Grafana dashboard. Once it's <5%, move medium to block.
- As a Director: you're trading off "perfect security" for "team velocity." Explain this tradeoff in your CISO review: "We block 95% of real risks now; the 5% of medium-severity false positives cost us 3 hrs/day in toil. In Q3, we'll tune the rules and move to block-all."

**Question 3**: Your Kubernetes cluster has 47 repos that can deploy to prod. You want to migrate to OIDC so no repo has an AWS credential. How do you prevent a malicious developer from cloning the `t-mobile/notification` repo, pushing code from their laptop, and assuming the deployment role?

**Answer**: The OIDC token is only issued by GitHub Actions *during a workflow run*. A developer's laptop never gets an OIDC token. So:
- Attacker clones repo locally, pushes to their fork (`evil-attacker/notification`)
- GitHub Actions doesn't trigger for forks (unless explicitly enabled)
- Even if they trigger a workflow, the OIDC token's `repo` claim = `evil-attacker/notification`, not `t-mobile/notification`
- AWS role trust policy only trusts `arn:aws:iam::ACCOUNT:role/github-deploy` with `repo: t-mobile/notification`
- Role assumption fails
- **You govern:** Do you allow forks to trigger workflows? (Usually no for prod.) Do you add an approval step for cross-account deployment? (Yes at Principal/Director level.)

## Part 7: Gate 1 Lab — SAST (Semgrep)

### Concept: Static Code Analysis

Semgrep is a linter that finds security and logic bugs *without running the code*. It matches patterns you specify.

For example, you tell Semgrep: "Find any line that looks like `password = 'hardcoded-string'`" — and it returns every match with line numbers.

**Semgrep cost-benefit** (for your CISO briefing):
- Cost: 15 min to write rules, 30 sec per PR to run
- Benefit: Catches 70% of hardcoded secrets, SQL injection, XSS before code review
- Tradeoff: False positive rate (~20%) requires triage; if too strict, developers disable it

### Lab: Write the Vulnerable App

**File: `src/app.py`**

```python
"""
Flask REST API for notification service.
Intentionally vulnerable for SAST lab.
"""
import hashlib
from flask import Flask, request, jsonify
import sqlite3

app = Flask(__name__)

# VULNERABILITY 1: Hardcoded API key
STRIPE_API_KEY = "sk_test_51234567890abcdefghij"

# VULNERABILITY 2: Using SHA256 for password hashing (should use bcrypt + salt)
def hash_password(password):
    """
    DO NOT USE THIS IN PRODUCTION.
    SHA256 without salt is reversible with enough compute.
    """
    return hashlib.sha256(password.encode()).hexdigest()

@app.route('/auth/register', methods=['POST'])
def register():
    """Register a new user."""
    data = request.get_json()
    
    # VULNERABILITY 3: SQL injection
    username = data.get('username')
    password = data.get('password')
    
    # This is vulnerable: user input is not parameterized
    conn = sqlite3.connect(':memory:')
    cursor = conn.cursor()
    query = f"INSERT INTO users (username, password) VALUES ('{username}', '{hash_password(password)}')"
    cursor.execute(query)  # BAD: direct string concatenation
    conn.commit()
    
    return jsonify({"status": "registered"}), 201

@app.route('/notifications/send', methods=['POST'])
def send_notification():
    """Send a notification via Stripe webhook."""
    data = request.get_json()
    
    # VULNERABILITY 4: Hardcoded secret used directly in function
    webhook_url = "https://api.stripe.com/v1/webhook"
    
    import requests
    # This would send the API key in plaintext over HTTP if HTTPS fails
    response = requests.post(
        webhook_url,
        headers={"Authorization": f"Bearer {STRIPE_API_KEY}"},
        json=data
    )
    
    return jsonify(response.json()), 200

if __name__ == "__main__":
    app.run(debug=True)  # VULNERABILITY 5: debug=True in production code
```

**File: `src/requirements.txt`**


### Lab: Fix the Code

**Step 1: Remove hardcoded secrets**

```python
# BEFORE
STRIPE_API_KEY = "sk_test_51234567890abcdefghij"

# AFTER
import os
STRIPE_API_KEY = os.getenv("STRIPE_API_KEY")
if not STRIPE_API_KEY:
    raise ValueError("STRIPE_API_KEY environment variable not set")
```

**Step 2: Use bcrypt for passwords**

```python
# BEFORE
import hashlib
def hash_password(password):
    return hashlib.sha256(password.encode()).hexdigest()

# AFTER
import bcrypt
def hash_password(password):
    salt = bcrypt.gensalt(rounds=12)
    return bcrypt.hashpw(password.encode(), salt).decode()
```

**Step 3: Use parameterized SQL queries**

```python
# BEFORE
query = f"INSERT INTO users (username, password) VALUES ('{username}', '{hash_password(password)}')"
cursor.execute(query)

# AFTER
cursor.execute("INSERT INTO users (username, password) VALUES (?, ?)", (username, hash_password(password)))
```

**Step 4: Remove debug mode**

```python
# BEFORE
app.run(debug=True)

# AFTER
if __name__ == "__main__":
    debug_mode = os.getenv("FLASK_ENV") == "development"
    app.run(debug=debug_mode)
```


### Discussion: False Positives & Alert Fatigue

**Scenario**: Your Semgrep config flags every string that starts with `"s_"` as a hardcoded secret (to catch Stripe keys). But developers have a utility function `string_split()` that starts with `"s_"`, and now they're seeing 200 false alerts per day.

**How would you fix this?** (This is a real problem at T-Mobile with observability alert rules.)

- **Option A**: Disable the rule (loses security).
- **Option B**: Add context — only flag `"s_"` if it appears after `=` and is >20 characters (reduces false positives, increases complexity).
- **Option C**: Run Semgrep in advisory mode for 1 week; collect data on false positive rate; adjust rule specificity; then enable blocking.

**Your answer as a Director**: "We'll start with Option C. Run advisory for 1 week, measure false positive rate in a Grafana dashboard, then adjust rules based on data. This way we unblock developers while maintaining security."

## Part 9: Gate 3 Lab — Container Image Scanning

### Concept: Image Layer Analysis

A Docker image is built in layers:
```
FROM ubuntu:18.04        ← Layer 1 (base OS)
RUN apt-get install ...  ← Layer 2 (dependencies)
COPY app.py .            ← Layer 3 (application)
RUN pip install ...      ← Layer 4 (Python deps)
```

Trivy scans each layer for vulnerabilities. An old base image (ubuntu:18.04) might have 300+ unpatched CVEs.


Tools like **TruffleHog** and **git-secrets** automatically scan the entire history.

## Part 11: Gate 5 Lab — OIDC-based Deployment (Secrets Management)

### Concept: The Problem with Credentials

**Traditional approach:**
```
Developer stores AWS secret key in GitHub Secrets:
REPO_SECRET_NAME = "AKIA1234567890ABCDEF"

GitHub Actions reads it from Secrets and writes to env var:
- Visible in job logs if you `echo $AWS_SECRET_ACCESS_KEY`
- If leaked, attacker can assume the AWS role indefinitely
- Must be manually rotated
```

**OIDC approach (modern):**
```
GitHub proves: "This is the official CI job for t-mobile/notification repo, main branch"
AWS trusts GitHub's proof and issues a 15-minute temporary credential
GitHub never sees the actual AWS key
```

**Analogy**: Instead of handing the FedEx driver a master key to all T-Mobile warehouses, you issue them a dated, single-use pass for *this route, today*.

### Lab: Set Up OIDC in AWS

**Step 1: Add OIDC Provider to AWS**

**Terraform: `terraform/main.tf`**

```hcl
  url = "https://token.actions.githubusercontent.com"

  # GitHub's signing certificate thumbprint (public, static)
  thumbprint_list = ["6938fd4d98bab03faadb97b34396831e3780aea1"]

  client_id_list = ["sts.amazonaws.com"]

  tags = {
    Name = "github-oidc-provider"
  }
}

# STEP 2: Create IAM Role that GitHub Actions can assume
resource "aws_iam_role" "github_actions_deploy" {
  name = "github-actions-deploy-role"

  assume_role_policy = jsonencode({

    Statement = [
      {
        Effect = "Allow"
        Principal = {
          Federated = aws_iam_openid_connect_provider.github.arn

            "token.actions.githubusercontent.com:sub" = "repo:t-mobile/notification:ref:refs/heads/main"

      }

  })

  tags = {
    Name = "github-actions-deploy"
  }
}

resource "aws_iam_role_policy" "github_actions_ecr" {
  name = "github-actions-ecr-policy"

    Version = "2012-10-17"
```

### Lab: Configure GitHub Actions to Use OIDC

**File: `.github/workflows/secure-pipeline.yaml`**

```yaml
name: Secure Pipeline

on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

permissions:
  id-token: write  # Allow GitHub to generate OIDC token
  contents: read
  pull-requests: read

jobs:
  secure-pipeline:
    runs-on: ubuntu-latest
    
    steps:
      # Gate 1: SAST Scanning
      - name: Checkout code
        uses: actions/checkout@v4
        
      - name: Install Semgrep
        run: pip install semgrep
        
      - name: Run Semgrep SAST scan
        run: semgrep --config .semgrep.yml src/app.py
        continue-on-error: true  # Allow reporting without blocking on first pass
        
      # Gate 2: Dependency Scanning
      - name: Install Trivy
        run: |
          wget https://github.com/aquasecurity/trivy/releases/download/v0.45.0/trivy_0.45.0_Linux-64bit.deb
          sudo dpkg -i trivy_0.45.0_Linux-64bit.deb
          
      - name: Scan dependencies
        run: trivy fs --config .trivy.yaml src/
        
      # Gate 3: Build Docker image
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3
        
      - name: Build Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: false
          load: true
          tags: secure-pipeline-app:${{ github.sha }}
          
      # Gate 4: Scan container image
      - name: Scan container image with Trivy
        run: trivy image --severity HIGH,CRITICAL secure-pipeline-app:${{ github.sha }}
        
      # Gate 5: Secrets scanning
      - name: Install TruffleHog
        run: pip install truffleHog
        
      - name: Scan for secrets in Git history
        run: truffleHog filesystem . --json --max-depth=5 > /dev/null
        
      # OIDC Token Exchange: Get temporary AWS credentials
      - name: Configure AWS credentials using OIDC
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}  # arn:aws:iam::123456789012:role/github-actions-deploy-role
          aws-region: us-east-1
          
      # Deploy: Push to ECR (requires OIDC-granted temporary AWS credentials)
      - name: Push image to Amazon ECR
        run: |
          aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin 123456789012.dkr.ecr.us-east-1.amazonaws.com
          docker tag secure-pipeline-app:${{ github.sha }} 123456789012.dkr.ecr.us-east-1.amazonaws.com/notification-service:${{ github.sha }}
          docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/notification-service:${{ github.sha }}
```

### Lab: Test OIDC Token Exchange

**Manual test (simulating GitHub Actions):**

```bash
# This won't work on your laptop (no OIDC token), but shows the flow:
aws sts assume-role-with-web-identity \
  --role-arn arn:aws:iam::123456789012:role/github-actions-deploy-role \
  --role-session-name github-actions-deploy \
  --web-identity-token $(cat /path/to/oidc-token.jwt) \
  --duration-seconds 900  # 15 minutes
```

**When GitHub Actions runs, it automatically:**
1. Generates an OIDC token with claims: `repo: t-mobile/notification`, `branch: main`, `job: secure-pipeline`
2. Calls `aws sts assume-role-with-web-identity` with that token
3. AWS verifies the token signature (using the thumbprint from Step 1)
4. AWS checks the `sub` claim matches the trust policy (only `repo:t-mobile/notification:ref:refs/heads/main`)
5. Issues temporary credentials (AccessKeyId, SecretAccessKey, SessionToken)
6. GitHub Actions uses those credentials for 15 minutes

**Why this is secure:**
- No static credentials in GitHub Secrets
- Credentials are time-limited (15 minutes, not forever)
- Credentials are audience-bound (only `sts.amazonaws.com`, not misused elsewhere)
- Credentials are repo-specific (only `t-mobile/notification`, not `attacker/notification`)

## Part 13: Governance & Director-Level Tradeoffs

### Tradeoff 1: Third-Party Actions — Pinning vs. Flexibility

**The Problem**:
```yaml
# BAD: Uses latest version of an action
- uses: actions/checkout@latest

# In June 2023, the tj-actions/changed-files action was compromised
# Attackers injected code to steal AWS credentials from CI jobs
# ~2,000 workflows were affected before detection
```

**Options**:

| Approach | Security | Maintainability | Risk |
|---|---|---|---|
| `@latest` | Low | High (auto-updates) | Supply chain compromise (tj-actions 2023) |
| `@v4` (major) | Medium | Medium (manual major updates) | Might miss critical security patches |
| `@v4.1.2` (pin exact) | High | Low (requires manual patch bumps) | Maintenance burden |
| `@v4 + renovate` (auto patch bumps) | High | High | Requires bot + PR approvals |

**Your governance as a Director**:
- "We pin to major version (`@v4`) with Renovate bot for automated patch updates. Security team reviews Renovate PRs weekly. Critical patches bypass review and auto-merge."
- Cost: 30 min/week Renovate review + bot maintenance
- Benefit: protects against tj-actions-level compromises while staying updated

### Tradeoff 2: OIDC Subject Claim Granularity

**The Problem**: How specific should the OIDC trust policy be?

```hcl
# OPTION A: Trust all jobs in the repo
Condition.StringEquals["token.actions.githubusercontent.com:sub"] = "repo:t-mobile/notification"

# OPTION B: Trust only main branch
Condition.StringEquals["token.actions.githubusercontent.com:sub"] = "repo:t-mobile/notification:ref:refs/heads/main"

# OPTION C: Trust only deploy job on main branch
Condition.StringEquals["token.actions.githubusercontent.com:sub"] = "repo:t-mobile/notification:ref:refs/heads/main:actor:bot-deploy"

# OPTION D: Different roles for prod vs. staging
Role 1 (prod): "repo:t-mobile/notification:ref:refs/heads/main"
Role 2 (staging): "repo:t-mobile/notification:ref:refs/heads/develop"
```

**Your choice** (as a Director, balancing security + developer velocity):
- Prod deployments: Option B + Option D (strict: only main branch, separate role)
- Staging deployments: Option A (lenient: any branch can deploy staging)
- Cost: 2 IAM roles to manage + documentation
- Benefit: accidental staging test doesn't risk prod credentials

### Tradeoff 3: SAST False Positive Tolerance

**The Problem**: Semgrep finds 200 issues per day; 90% are false positives.

**Data-driven approach** (use Grafana to track):
- Week 1: Run advisory mode. Track: true positives vs. false positives.
- Week 2: Adjust rules (context-aware regexes). Re-measure.
- Week 3: If false positive rate <10%, enable blocking mode.
- Ongoing: Monitor false positive rate in a Grafana dashboard; alert if it exceeds 15%.

**Example Grafana query:**
```promql
histogram_quantile(0.5, rate(semgrep_findings{severity="HIGH"}[1d]))
```

**Your pitch to the CISO**: "We're blocking 95% of real high-severity findings while keeping false positive rate <10%. The 5% we miss are edge cases that require manual review. Here's the trend: [show graph]."

### Q2: An attacker obtains a leaked credential from Git history. How do you detect and respond?

**Your answer** (using Splunk/observability skills):
- **Detection**: TruffleHop finds the credential in a nightly scan. (Or: security researcher reports it via HackerOne.)
- **Response**:
  1. Rotate the credential (e.g., IAM key).
  2. Check CloudTrail/Splunk for unauthorized API calls using the old key.
  3. If found: incident (P2 at minimum). Trigger postmortem.
  4. Rewrite Git history (or warn all developers that the repo is compromised).
  5. Block the old credential in Vault (revoke token).
- **Metrics**: Mean time to detect (MTTD) <1 day (because of automated scanning). Mean time to respond (MTTR) <4 hours (because of playbook).
