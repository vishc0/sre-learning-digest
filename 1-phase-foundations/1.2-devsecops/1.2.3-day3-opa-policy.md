# Week 2, Day 3: OPA Policy-as-Code — Kubernetes + Terraform Governance

## Why This Matters

OPA (Open Policy Agent) solves a foundational DevSecOps problem: **compliance cannot be human-enforced at cloud scale**. At T-Mobile, you manage a platform touching 1000s of infrastructure requests/week across dozens of teams. Manually code-reviewing every CloudFormation template or Terraform plan for encryption, tagging, and RBAC compliance is impossible. OPA lets you **encode compliance rules once, enforce them everywhere** — S3 must have encryption, every resource must have cost-center tags, and admission to Kubernetes must reject images from untrusted registries. This is the difference between SRE Tier 1 (reactive firefighting) and Tier 3+ (governance at the gate).

---

## Real-World Analogy: The Airport Security Checkpoint

Think of OPA as the security policy engine at an airport.

**Without OPA** (manual checks): Every traveler arrives at the desk; the agent manually scans their ID, checks the current ruleset (passenger name index, no-fly list, visa requirements), and decides "allow" or "deny." If the ruleset changes, you must retrain all agents. If an agent misses a rule, a dangerous passenger boards. **Chaos at scale.**

**With OPA** (policy-as-code): The **ruleset is code** — written once, deployed everywhere. When a traveler approaches, the kiosk checks ID → background → no-fly list → visa rules **automatically** using the same policy engine. Update the rules in one repo; every kiosk reflects it immediately. Rules are version-controlled, auditable, and testable before deployment.

**Applied to infrastructure**:
- **Traveler** = Terraform plan, Kubernetes manifest, ECR image push
- **Ruleset** = Rego policies (S3 must encrypt, pods must have resource limits, images must be signed)
- **Checkpoint** = CI/CD gate (Conftest) or Kubernetes admission controller (Gatekeeper)
- **Decision** = PASS (deploy) or FAIL (reject with reason)

## Vocabulary Translation Table

| Industry Term | Your Mental Model | Why It Matters |
|---|---|---|
| **OPA (Open Policy Agent)** | Policy engine (like CyberArk for infrastructure) | Centralized decision-making engine; language-agnostic (JSON → decision) |
| **Rego** | Policy language (like Splunk SPL, but for rules) | Declarative: you say *what* compliance looks like, not *how* to enforce it |
| **Conftest** | OPA test runner for IaC files | CLI tool; integrates into CI/CD; tests Terraform plans, K8s manifests, JSON blobs |
| **Gatekeeper** | Kubernetes admission webhook using OPA | Runs *inside* K8s cluster; blocks violating manifests before they hit etcd |
| **Admission Controller** | Pre-deployment checkpoint (like firewall rule) | Kubernetes says "before you create a Pod, consult Gatekeeper—does it pass policy?" |
| **failurePolicy: Fail vs Ignore** | Hard block vs soft warning | `Fail` = violating manifest is rejected (high-stakes); `Ignore` = log and allow (audit-only) |
| **Deny Rule** | Explicit rejection rule | If condition is true, output a violation message and the entire policy fails |
| **Not Provable** | Rule cannot be satisfied with given data | In Rego: `not` means "I cannot find evidence that this is true"—critical for optional field checks |

## 30-Second Version

OPA is a policy engine that encodes compliance rules once and enforces them everywhere. Conftest lets you test Terraform/Kubernetes files against OPA policies in CI/CD. Gatekeeper deploys OPA *inside* Kubernetes to block non-compliant manifests at admission time. Write rules in Rego (a declarative language), version-control them, and violations fail the pipeline automatically.

## 2-Minute Version

OPA (Open Policy Agent) decouples **policy decisions** from **policy enforcement**. Instead of embedding "must have encryption" logic inside your Terraform modules or having humans review PRs, you write a Rego policy file that says: "For every S3 bucket in the plan, deny if `server_side_encryption_configuration` is missing."

**Two deployment patterns**:

1. **CI/CD gate (Conftest)**: Before Terraform apply, the plan is exported as JSON, fed to Conftest (OPA's CLI), checked against your policies, and rejected if any violation is found. Developer sees: "FAIL: S3 bucket must have encryption." They fix the code and retry.

2. **Kubernetes admission (Gatekeeper)**: Every manifest sent to the API server is intercepted by a Gatekeeper webhook. Gatekeeper runs OPA policies on the manifest (e.g., "pods must have resource requests"). If denied, the Pod creation is blocked. This is your **last line of defense**.

**Why it matters at your scale**: At T-Mobile, 15 SRE team members + dozens of platform tenants + 100s of AWS resources. Manual policy review is error-prone and slow. OPA lets you encode institutional knowledge ("we require Owner and CostCenter tags") once, test it locally, and deploy it to your CI/CD systems and K8s clusters. Violations are *automatic* and *consistent*.

## 5-Minute Version

### The Problem OPA Solves

Compliance at scale is a **decision-making problem**, not a code problem. Imagine you manage a Kubernetes cluster for a multi-tenant platform:

- **Rule**: Pods must have CPU/memory requests and limits.
- **Without OPA**: You write docs, send Slack messages, review PRs manually. Some teams comply, others don't. In incident, you find a runaway Pod using 100% CPU because nobody enforced the rule.
- **With OPA**: You write a Rego policy that says "deny any Pod without requests/limits," deploy it as an admission controller, and *every* Pod creation is checked automatically.

### How OPA Works (Data Flow)

```
Input (JSON)        Policy (Rego)       Engine          Output (Allowed/Denied)
─────────────       ─────────────       ──────          ───────────────────────
Terraform plan  +   S3 encryption   →   OPA     →       PASS / FAIL + reason
Kubernetes YAML +   pod limits          evaluates
CloudFormation  +   image registry       the facts
ECR image metadata  against the rules
```

OPA receives JSON input (the thing being evaluated: a Terraform plan, a K8s manifest, a user's attributes), runs it through a Rego policy, and outputs a boolean decision: **allow** (all rules pass) or **deny** (at least one rule failed, with a violation message).

### Rego: The Policy Language

Rego is **declarative**: you describe what compliance looks like, not how to check it.

Example Rego rule:
```rego
deny[msg] {
    input.resource_type == "aws_s3_bucket"
    not has_encryption(input)
    msg := "S3 bucket must have server_side_encryption_configuration"
}

has_encryption(bucket) {
    bucket.server_side_encryption_configuration[_]
}
```

**What this says**:
- For this resource, if it's an S3 bucket *and* it does NOT have encryption, output a deny message.
- `[_]` means "at least one element in the list"—if the list exists and has items, the rule is satisfied.
- If the encryption block doesn't exist, `has_encryption()` fails, so `not has_encryption()` succeeds, triggering the deny.

### Two Deployment Patterns

**1. Conftest (CI/CD Gate)**
- Developer writes Terraform code, runs `terraform plan -out=tfplan`, convert plan to JSON.
- CI/CD runs `conftest test tfplan.json -p policy/`.
- Policies are checked; if any deny rules fire, CI fails. Developer fixes and retries.
- **Advantage**: Shift-left—catch issues before they reach your AWS account.

**2. Gatekeeper (K8s Admission Control)**
- Deploy Gatekeeper (OPA + webhook) in your cluster.
- Define `ConstraintTemplate` (a Rego policy + metadata) and a `Constraint` (the rule applied to certain resources).
- When a developer applies a Kubernetes manifest, the API server calls the Gatekeeper webhook.
- Gatekeeper evaluates the manifest against policies; if denied, the Pod/Deployment is rejected.
- **Advantage**: Real-time, built-in to your platform—no escape route via manual `kubectl apply`.

### Why `not` Is Tricky in Rego

A **critical debugging moment** for newcomers: the difference between "false" and "undefined."

```rego
deny[msg] {
    not input.metadata.tags.Owner
    msg := "Owner tag is required"
}
```

**Interpretation**:
- If `input.metadata.tags.Owner` **does not exist** (undefined), the rule succeeds, triggering the deny.
- If `Owner: "alice"` exists, the rule fails (because `not` of a provable fact is false).
- If `Owner: ""` (empty string), the rule fails (empty string is a defined value, so `not` is false).

This is **not** a boolean NOT in Python. It's "cannot be proven"—a subtle but critical distinction. This trips up teams used to imperative languages.

## Director/VP Lens: Governance, Not Just Technical

As an SRE Manager, you need to understand how OPA enables **scalable governance**:

1. **Consistency across teams**: Policy is written once, enforced everywhere. One team can't accidentally use an unencrypted S3 bucket while another does.

2. **Audit trail**: Every violation is logged. You can report to compliance: "Q2: 127 attempted S3 deployments without encryption were blocked by OPA."

3. **Phased rollout**: You can deploy policies in `warn` mode first (log violations, allow deployment), then shift to `deny` (block violations) after teams adapt. This is the difference between a **failing change control** and a **successful governance rollout**.

4. **Cost control**: A policy that denies T2.Nano instances in production (too risky), or requires CostCenter tags (cost allocation), scales governance without hiring more reviewers.

5. **Hiring/onboarding**: New team members don't need to memorize tribal knowledge. The policies **are** the institutional knowledge. Audit the Rego files the same way you audit code.

### Answer Key

**Q1**: Yes, OPA will deny. The rule checks `not has_encryption(input)`. If `server_side_encryption_configuration` is omitted from the plan, `has_encryption()` is undefined (can't find the field), so `not undefined` is provable (true), triggering the deny. The developer will see: "S3 bucket must have server_side_encryption_configuration."

**Q2**: 
- **Conftest**: CLI tool; tests files (Terraform plans, K8s manifests) before deployment.
- **Gatekeeper**: Webhook; intercepts requests *inside* Kubernetes cluster; blocks non-compliant manifests at admission time.
- Use Conftest for IaC before `git push`; use Gatekeeper for runtime enforcement of K8s manifests.

**Q3**: Yes, the rule fires and the Pod is denied. `not input.metadata.labels.Owner` means "the Owner label cannot be proven to exist." If it's missing, this condition is true, so the deny message is output. This is why `not` is dangerous when applied to optional fields without careful scoping.

#### Step 3: Write S3 Encryption Policy

Create `policy/s3_encryption.rego`:

```rego
# S3 Encryption Policy
# Deny any S3 bucket without server_side_encryption_configuration

deny[msg] {
    input.resource_type == "aws_s3_bucket"
    not has_encryption(input)
    msg := sprintf("S3 bucket '%s' must have server_side_encryption_configuration", [input.name])
}

# Helper: check if encryption is configured
has_encryption(bucket) {
    bucket.server_side_encryption_configuration[_]
}
```

Save to `~/opa-lab/policy/s3_encryption.rego`.

**EXPECTED_OUTPUT**: File created with no errors.

**WHY**: 
- `deny[msg]` is a deny rule; if the condition is true, it outputs a violation message.
- `input.resource_type == "aws_s3_bucket"` filters for S3 buckets only.
- `not has_encryption(input)` triggers if the bucket lacks the encryption block.
- `[_]` means "at least one element"—ensures the list is non-empty.
- `sprintf()` formats the violation message with the bucket name.

#### Step 8: Write Tagging Policy

Create `policy/required_tags.rego`:

```rego
# Required Tags Policy
# Deny any resource without Owner, CostCenter, Environment, and Team tags

deny[msg] {
    input.resource_type == "aws_s3_bucket"
    required_tags := {"Owner", "CostCenter", "Environment", "Team"}
    missing := required_tags - object.keys(input.tags)
    count(missing) > 0

# Helper: get keys from a map
# (Rego's object.keys() is built-in)
```

**EXPECTED_OUTPUT**: File created.

**WHY**:
- `required_tags - object.keys(input.tags)` computes the set difference: tags that should exist but don't.
- `count(missing) > 0` ensures at least one tag is missing.
- This policy complements S3 encryption; together they enforce two distinct rules.

### Interview Question 3: "Walk me through a Rego policy you wrote. What tradeoffs did you make?"

**Answer** (using the S3 encryption policy from the lab):

```rego
deny[msg] {
    input.resource_type == "aws_s3_bucket"
    not has_encryption(input)
    msg := sprintf("S3 bucket '%s' must have server_side_encryption_configuration", [input.name])
}

has_encryption(bucket) {
    bucket.server_side_encryption_configuration[_]
}
```

**Tradeoff 1: Encryption algorithm strictness**
- **What I did**: Policy denies *any* bucket without the encryption block, regardless of algorithm (AES256 vs. KMS).
- **Tradeoff**: 
  - ✓ Simpler rule, easier to maintain.
  - ✗ Doesn't enforce KMS for sensitive data (medical records, PII).
- **When to tighten**: If compliance requires KMS for HIPAA, you'd add: `bucket.server_side_encryption_configuration[0].rule.apply_server_side_encryption_by_default.sse_algorithm == "aws:kms"`. But now the rule is more fragile (assumes structure matches your Terraform module output).

**Tradeoff 2: Fail-fast vs. audit-only**
- **What I did**: Policy is `deny`, which means Conftest fails the pipeline.
- **Tradeoff**:
  - ✓ Forces developers to fix before deploying—shift-left.
  - ✗ Can be annoying during rapid prototyping; may pressure teams to work around the rule.
- **Alternative**: Deploy in `warn` mode first (violations logged, deployment allowed), then shift to `deny` after validation.

**Tradeoff 3: Scoping to S3 buckets**
- **What I did**: Policy only applies to `aws_s3_bucket` resources.
- **Tradeoff**:
  - ✓ Avoids false positives on other resource types.
  - ✗ If you add `aws_s3_bucket_server_side_encryption_configuration` as a separate resource (common in modular Terraform), this policy won't catch it.
- **Fix**: Add a second deny rule: `input.resource_type == "aws_s3_bucket_server_side_encryption_configuration"` with matching checks.

### Interview Question 4: "How do Conftest and Gatekeeper differ? When would you use each?"

**Answer**:

| Aspect | Conftest | Gatekeeper |
|---|---|---|
| **When it runs** | Before `terraform apply` / during CI/CD | At Kubernetes admission time (when you `kubectl apply`) |
| **What it tests** | Terraform plans, K8s manifests, JSON blobs (any input file) | Only Kubernetes manifests |
| **Deployment location** | CI/CD system (GitHub Actions, GitLab CI) | Inside K8s cluster (webhook) |
| **Failure behavior** | Fails the pipeline; developer sees error and fixes locally | Rejects the API request; developer must fix and retry |
| **Override mechanism** | Rerun after code fix (clean path) | Exception process (e.g., `exempt_namespaces` in Gatekeeper constraint) |

**When to use Conftest**:
- Enforce IaC standards *before* they're deployed (shift-left).
- Test Terraform plans for security/tagging violations in CI/CD.
- Test CloudFormation, Helm charts, any JSON/YAML.

**When to use Gatekeeper**:
- Enforce runtime K8s compliance at the boundary (prevent bad manifests from hitting etcd).
- Catch manifests that bypassed CI/CD (e.g., developer runs `kubectl apply -f` locally).
- Enforce policies that depend on cluster state (e.g., "Pod must have NetworkPolicy in place").

**At T-Mobile**: Use Conftest for all Terraform pre-deployment. Use Gatekeeper in the cluster to catch any K8s manifests that slip through (human error, local testing).

### Interview Question 5: "How would you test your Rego policies to ensure they work?"

**Answer**:

OPA policies should be tested like code:

```bash
# 1. Unit test a single policy rule
opa test policy/s3_encryption.rego -v

# 2. Integration test: all policies against a manifest
conftest test plans/tfplan_good.json -p policy/

# 3. Regression test: create a test file with known-good and known-bad plans
# Store in policy/tests/
```

Example test file (Rego):

```rego
# tests/s3_encryption_test.rego
package main

test_deny_bucket_without_encryption {
    deny["S3 bucket 'test' must have server_side_encryption_configuration"] with input as {
        "resource_type": "aws_s3_bucket",
        "name": "test",
        "tags": {}
    }
}

test_allow_bucket_with_encryption {
    count(deny) == 0 with input as {
        "resource_type": "aws_s3_bucket",
        "name": "test",
        "server_side_encryption_configuration": [{}],
        "tags": {}
    }
}
```

Run tests:
```bash
opa test policy/ -v
```

**Best practices**:
- Test both positive (should pass) and negative (should deny) cases.
- Version-control test files in the policy repo.
- Run tests in CI/CD on every policy change.
- Use descriptive test names: `test_deny_*` and `test_allow_*`.

## 3 Tradeoffs: IC vs. Director Lens

### Tradeoff 1: Strict Policies vs. Developer Flexibility

**IC Lens**: "Write strict policies—no S3 without encryption, no Pods without resource limits."
- Pro: Highest security posture.
- Con: Developers perceive friction; may pressure leadership to disable policies.

**Director Lens**: "Phased rollout—start in warn mode, measure violations, tighten gradually."
- Pro: Builds team buy-in; policies stick long-term.
- Con: Short-term compliance violations logged but not blocked.
- **How to decide**: Audit-phase policies (warn) vs. hard-requirement policies (deny). Use deny for security-critical rules (encryption); use warn for operational best-practices (tagging) for 2 weeks.

---

### Tradeoff 2: Centralized Policies vs. Team Autonomy

**IC Lens**: "All policies are global; one Rego repo, enforced everywhere."
- Pro: Consistent compliance across all teams.
- Con: Adding a new policy requires cross-team alignment; slow change velocity.

**Director Lens**: "Baseline policies are global; teams can add stricter policies locally (opt-in)."
- Pro: Flexibility; teams own their compliance profile.
- Con: Risk of divergence; harder to audit.
- **How to decide**: Core policies (encryption, RBAC) are global and non-negotiable. Optional policies (performance, cost) are namespaced or cluster-local. Document the tier clearly in your wiki.

---

### Tradeoff 3: Shift-Left (Conftest) vs. Real-Time (Gatekeeper)

**IC Lens**: "Use Conftest in CI/CD—catch issues early, avoid wasted cloud resources."
- Pro: Developers get feedback in seconds (local test).
- Con: If a developer bypasses CI/CD (e.g., `kubectl apply` directly), non-compliant resources land in the cluster.

**Director Lens**: "Use both—Conftest for speed, Gatekeeper as the safety net."
- Pro: Layered defense—no escape route.
- Con: Requires managing two policy systems; Gatekeeper adds latency to admission (ms-level, usually acceptable).
- **How to decide**: Always use Conftest for IaC. Always use Gatekeeper in prod clusters (defense-in-depth). In dev clusters, you can skip Gatekeeper if developers self-police via Conftest.

---

## Key Takeaways

1. **OPA is the policy engine; Rego is the language**: One defines the decision-making logic, the other encodes it.
2. **Conftest (CI/CD) and Gatekeeper (K8s admission) are deployment patterns**: Use both for defense-in-depth.
3. **Phased rollout is critical**: Start in `warn` mode, gather data, shift to `deny` after teams adapt. This is how governance scales.
4. **`not` in Rego means "undefined"**: A field missing is undefined; `not undefined` is provable. This trips up imperatively-minded engineers.
5. **Policies are code**: Version-control them, test them, review them like any other code.
