## Why This Matters

Every senior SRE/DevSecOps interview at FAANG or a cloud-native startup will ask you to design a "shift-left" security pipeline. This lab builds the actual artifact — a working pipeline you can demo, a GitHub repo you can reference, and the architectural vocabulary to explain *why* each stage exists. Portfolio Capstone #2 is the difference between "I've read about DevSecOps" and "I built one — here's the repo."

## Analogy

Think of this pipeline as your T-Mobile incident command chain — but automated and running before code reaches production. Pre-commit hooks are your on-call pager check before a change window opens. The PR check is your CAB review. The build and sign stages are your change ticket stamp and approval signature. Cosign signing is the digital equivalent of a notarized change record — cryptographic proof that *this exact artifact* was approved by *this exact pipeline run*. The deploy gate is your go/no-go call before cutover. If any stage fails, the pipeline halts — same as your incident commander holding a deployment when SEV signals are red.

## 30-Second Version

This lab wires together six security tools into one GitHub Actions pipeline. Every commit is checked for secrets and bad code patterns before it can be merged. Every container image is scanned for known vulnerabilities, signed with a tamper-proof signature, and blocked from deploying if it carries HIGH or CRITICAL CVEs. The result is a portfolio-grade artifact that answers the interview question "walk me through your shift-left security design."

---

## 2-Minute Version

The pipeline runs in six stages, each catching a different class of failure:

**Stage 1 — Pre-commit (Gitleaks + Semgrep):** Gitleaks scans for accidentally committed secrets (API keys, tokens, passwords). Semgrep scans for insecure code patterns (SQL injection, hardcoded credentials in logic). Both run locally via a git pre-commit hook before the developer can push. This is the cheapest place to catch problems — zero CI cost, instant feedback.

**Stage 2 — PR Check (Trivy + Conftest):** When a pull request opens, Trivy scans the Dockerfile for base image vulnerabilities. Conftest runs OPA policies against the Dockerfile — enforcing rules like "must not run as root" and "must use a pinned image digest." A PR that fails either check cannot be merged.

**Stage 3 — Build:** Docker builds the image and tags it with the exact git SHA. No floating tags like `latest` — every image is traceable to a specific commit.

**Stage 4 — Sign (Cosign keyless):** GitHub's OIDC token is used to sign the image without storing a long-lived private key. The signature is stored in the container registry alongside the image. Anyone can verify it came from your pipeline — not a rogue build.

**Stage 5 — SBOM + Grype:** Syft generates a Software Bill of Materials — a complete list of every library inside the container. Grype cross-references that list against the CVE database. If any component has a HIGH or CRITICAL vulnerability, the pipeline captures it.

**Stage 6 — Deploy Gate:** A final OPA policy check reads the Grype scan results. If HIGH or CRITICAL CVEs exist, the pipeline fails. The image never reaches the registry's "approved" tag. Nothing deploys.

## 5-Minute Version

**Why each tool was chosen (not just what it does):**

Gitleaks is preferred over `git-secrets` because it has a maintained ruleset for 150+ secret types and exits non-zero on detection — making it trivially composable with pre-commit frameworks. Semgrep is preferred over SonarQube for pre-commit because it runs in milliseconds locally with no server dependency; SonarQube is a better fit for centralized PR dashboards, not local gates.

Trivy is the scanning workhorse because it covers OS packages, language dependencies, and IaC files in a single binary. You don't need separate tools for Python dependencies vs. Alpine packages. Grype (from Anchore) is used for post-SBOM CVE correlation because it natively consumes CycloneDX and SPDX SBOM formats — Trivy can do this too, but Grype's output schema is cleaner for OPA policy consumption.

OPA/Conftest is chosen over inline shell conditionals because policy-as-code is *versionable, auditable, and reviewable*. When a security team asks "what are your image admission rules," you hand them a `.rego` file and a PR history — not a bash script with `grep` conditions. This is the difference between a mature security posture and an ad-hoc one.

Cosign keyless signing eliminates the private key management problem. Traditional code signing requires storing a private key securely — that key itself becomes a target. Keyless signing uses ephemeral keys tied to the GitHub OIDC identity token. The signature is valid only because GitHub's OIDC token attests that the action ran in *your* repo, on *your* branch, triggered by *your* commit. No key to rotate, no secret to leak.

**Failure modes to know:**

1. **SBOM drift**: Your SBOM is accurate at build time but the base image is updated by the upstream maintainer without your knowledge. Mitigated by pinning image digests (SHA256) not tags.
2. **Policy bypass via image mutation**: A signed image could be replaced with a different image at the same tag if the registry doesn't enforce immutability. Mitigated by Cosign signature verification at deploy time (not just build time) and registry immutable tags.
3. **False positive fatigue**: Grype finds CVEs in packages that aren't reachable in your runtime. If every build fails on a theoretical CVE in a dev-only test library, developers start ignoring the gate. Mitigated by CVE allowlisting with documented justification per suppressed finding — not blanket suppression.

## Director/VP Lens

**As an IC** you configure the tools, write the `.rego` policies, fix the failing builds, and understand why Cosign uses OIDC.

**As a Director/VP** you answer a different set of questions:

- "What is our mean time to detect a vulnerable image in production?" — the SBOM + Grype stage gives you that number.
- "How do we prove compliance to an auditor?" — the signed image + SBOM artifact is your evidence package. Every deployment has a cryptographically verified chain of custody.
- "How do we handle a zero-day like Log4Shell?" — the SBOM lets you query every running service for the affected library in minutes, not days. This is the actual business value of an SBOM.
- "What's the developer experience cost?" — pre-commit hooks add 10-30 seconds per commit. PR checks add 2-4 minutes. If developers are bypassing hooks, the gates are too slow or too noisy. You govern by measuring bypass rates and fix rates, not by adding more gates.
- "How does AI change this?" — AI-assisted SAST (GitHub Copilot Autofix, Semgrep Assistant) reduces the time-to-fix for flagged findings by suggesting remediation inline. The director's job is evaluating which AI-generated fixes are safe to auto-merge vs. require human review.

## Comprehension Check

Answer at least 2 of 3 before starting the lab.

**Q1.** A developer accidentally commits an AWS access key to the repo. Which pipeline stage catches this, and at what point in the workflow — before push, before merge, or before deploy?

**Q2.** Your Grype scan finds a CRITICAL CVE in `libssl` inside your container. Your security team says "that library is never called at runtime — it's a transitive dependency of a test framework." What do you do, and how do you document it so a future auditor can verify the decision?

**Q3.** Why is Cosign keyless signing preferable to generating a private key and storing it as a GitHub Actions secret? What attack surface does keyless eliminate?

---

## ANSWER KEY — Comprehension Check

**A1.** Gitleaks catches it at the **pre-commit hook stage** — before the developer can push to the remote repository at all. This is the earliest possible catch: local, zero CI cost, instant. If the hook is disabled or bypassed, the secret reaches the remote and would be caught by the PR check stage (where Gitleaks also runs in CI). The deploy gate does not scan for secrets — it reads CVE data. The pre-commit stage is the primary control; CI is the backstop.

**A2.** You do **not** suppress it silently. You create a CVE allowlist entry (in Grype's `.grype.yaml` config or OPA policy) that documents: (a) the CVE ID, (b) the package name, (c) the justification ("test-only transitive dependency, not present in production runtime image"), (d) the reviewer's name, (e) the review date, and (f) a re-review date (typically 90 days). This entry is checked into version control. An auditor can see the decision, who made it, and when it expires. This is the difference between "we suppressed it" and "we evaluated and accepted the risk with documentation."

**A3.** Keyless signing eliminates the **private key as a persistent secret**. If you store a signing key as a GitHub Actions secret, that key can be exfiltrated via a compromised workflow, a supply chain attack on a third-party Action, or insider access. The key must also be rotated on a schedule, and rotation itself is an operational burden. Keyless signing generates an ephemeral key pair valid only for the duration of the signing operation. The public certificate is issued by Sigstore's Fulcio CA and is tied to the GitHub OIDC token — which attests the specific repo, branch, and workflow SHA. An attacker who steals the OIDC token gets a certificate that expires in minutes. There is no long-lived secret to exfiltrate.

#### Step 9: Write the notification service application code

```bash
# Write a minimal Python notification service
cat > ~/portfolio/notification-service/src/app.py << 'EOF'
"""
Notification Service — minimal Flask app for DevSecOps pipeline demo.
Simulates a service that sends notifications (like T-Mobile's 25M msg/day platform).
"""

import os
from flask import Flask, request, jsonify

app = Flask(__name__)

# Configuration loaded from environment variables — NOT hardcoded
# This is the secure pattern Semgrep will enforce
NOTIFICATION_SERVICE_VERSION = os.environ.get("SERVICE_VERSION", "1.0.0")
MAX_BATCH_SIZE = int(os.environ.get("MAX_BATCH_SIZE", "1000"))


@app.route("/health", methods=["GET"])
def health():
    """Health endpoint — required for Kubernetes liveness probe."""
    return jsonify({"status": "healthy", "version": NOTIFICATION_SERVICE_VERSION})


@app.route("/notify", methods=["POST"])
def notify():
    """
    Accept a notification payload and queue it for delivery.
    In production this would publish to RabbitMQ or Kafka.
    """
    payload = request.get_json()

    if not payload or "message" not in payload:
        return jsonify({"error": "message field required"}), 400

    if len(payload.get("recipients", [])) > MAX_BATCH_SIZE:
        return jsonify({"error": f"batch size exceeds limit of {MAX_BATCH_SIZE}"}), 400

    # Simulate queuing — in production: publish to message broker
    return jsonify({
        "status": "queued",
        "message_id": "sim-12345",
        "recipient_count": len(payload.get("recipients", []))
    })


if __name__ == "__main__":
    # Never expose debug=True in production — controlled by env var
    debug_mode = os.environ.get("FLASK_DEBUG", "false").lower() == "true"
    app.run(host="0.0.0.0", port=8080, debug=debug_mode)
EOF

# Write requirements.txt
cat > ~/portfolio/notification-service/src/requirements.txt << 'EOF'
flask==3.0.3
gunicorn==22.0.0
EOF
```

**EXPECTED_OUTPUT:** (no output — file write is silent)

**WHY:** The app code uses environment variables for config — not hardcoded values. Semgrep's `p/python` ruleset will flag hardcoded passwords or API keys. This clean code should pass cleanly.

#### Step 10: Write the Dockerfile

```bash
cat > ~/portfolio/notification-service/Dockerfile << 'EOF'
# ============================================================
# Stage 1: Builder — installs dependencies in isolation
# ============================================================
# Pin the exact digest of the base image — not just a tag.
# Tags can be overwritten by the upstream maintainer.
# Digests are immutable — this image will always be exactly this.
FROM python:3.12-slim@sha256:59b5b9f60c4bfd89c4a96872b4f8e0f91c92b4e6e0d0f0b86b3f2c3e7b1a6c9 AS builder

# Set working directory
WORKDIR /app

# Copy only requirements first — Docker layer caching optimization.
# If requirements.txt doesn't change, this layer is cached even if app.py changes.
COPY src/requirements.txt .

# Install dependencies
RUN pip install --no-cache-dir --upgrade pip && \
    pip install --no-cache-dir -r requirements.txt

# ============================================================
# Stage 2: Runtime — minimal image with no build tools
# ============================================================
FROM python:3.12-slim@sha256:59b5b9f60c4bfd89c4a96872b4f8e0f91c92b4e6e0d0f0b86b3f2c3e7b1a6c9 AS runtime

WORKDIR /app

# Copy installed packages from builder stage
COPY --from=builder /usr/local/lib/python3.12/site-packages /usr/local/lib/python3.12/site-packages
COPY --from=builder /usr/local/bin/gunicorn /usr/local/bin/gunicorn

# Copy application source
COPY src/ .

# Create a non-root user — OPA policy will enforce this requirement
# Running as root inside a container is a critical security risk
RUN useradd --uid 1000 --no-create-home --shell /bin/false appuser && \
    chown -R appuser:appuser /app

# Switch to non-root user
USER appuser

# Expose the application port
EXPOSE 8080

# Healthcheck — tells Docker (and Kubernetes) how to verify the container is alive
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8080/health')"

# Run with gunicorn — production WSGI server, not Flask dev server
CMD ["gunicorn", "--bind", "0.0.0.0:8080", "--workers", "2", "--timeout", "30", "app:app"]
EOF
```

**EXPECTED_OUTPUT:** (no output — file write is silent)

**WHY:** This Dockerfile deliberately satisfies the OPA policies you'll write next: non-root user, pinned digest, multi-stage build. In Module 3 you'll break it intentionally to test the gate.

#### Step 11: Write the OPA policy for image requirements

```bash
cat > ~/portfolio/notification-service/policy/image-policy.rego << 'EOF'
# ============================================================
# image-policy.rego
# OPA policy for container image requirements
# Used by: Conftest in PR check stage
# ============================================================

package image

import future.keywords.if
import future.keywords.contains

# ---- Rule 1: Image must not run as root ----
# Running as root inside a container means any container escape
# gives the attacker root on the host node. This is the #1
# container security violation in production audits.
deny contains msg if {
    # Check if USER directive is missing entirely
    not any_user_set
    msg := "DENY: Dockerfile must specify a non-root USER. Add 'USER <username>' after creating a non-root user."
}

deny contains msg if {
    # Check if USER is explicitly set to root (common anti-pattern)
    input[i].Cmd == "user"
    user_val := input[i].Value[0]
    # root can be specified as "root" or "0"
    user_val == "root"
    msg := "DENY: USER must not be 'root'. Create a dedicated appuser."
}

deny contains msg if {
    input[i].Cmd == "user"
    user_val := input[i].Value[0]
    user_val == "0"
    msg := "DENY: USER must not be UID 0 (root). Use a non-root UID."
}

# ---- Rule 2: Base image must use digest pinning ----
# A tag like 'python:3.12-slim' can be overwritten upstream.
# A digest like 'python:3.12-slim@sha256:...' is immutable.
# This prevents supply chain attacks where the upstream tag is compromised.
deny contains msg if {
    input[i].Cmd == "from"
    image_ref := input[i].Value[0]
    # Check that image reference contains a SHA256 digest
    not contains(image_ref, "@sha256:")
    # Exclude the scratch base image (used for distroless FROM scratch patterns)
    image_ref != "scratch"
    msg := sprintf("DENY: Base image '%v' must be pinned by digest (@sha256:...). Tags are mutable.", [image_ref])
}

# ---- Rule 3: No COPY --chown=root (escalation risk) ----
deny contains msg if {
    input[i].Cmd == "copy"
    options := input[i].Flags
    some flag in options
    contains(flag, "--chown=root")
    msg := "DENY: COPY --chown=root transfers root ownership to files. Use a non-root owner."
}

# ---- Rule 4: Must expose a port (liveness check requirement) ----
warn contains msg if {
    not any_expose
    msg := "WARN: Dockerfile does not EXPOSE any port. Kubernetes liveness probes require a reachable endpoint."
}

# ---- Rule 5: Must have HEALTHCHECK ----
warn contains msg if {
    not any_healthcheck
    msg := "WARN: No HEALTHCHECK defined. Container orchestrators cannot detect unhealthy containers without it."
}

# ---- Helper rules ----
any_user_set if {
    input[i].Cmd == "user"
}

any_expose if {
    input[i].Cmd == "expose"
}

any_healthcheck if {
    input[i].Cmd == "healthcheck"
}
EOF
```

**EXPECTED_OUTPUT:** (no output)

**WHY:** This policy encodes your security team's container standards as verifiable, auditable code. The `deny` rules are hard failures (PR cannot merge). The `warn` rules are advisory — they appear in output but don't block.

#### Step 12: Write the OPA policy for Terraform IaC

```bash
cat > ~/portfolio/notification-service/policy/terraform-policy.rego << 'EOF'
# ============================================================
# terraform-policy.rego
# OPA policy for Terraform IaC security requirements
# Used by: Conftest in IaC review stage
# ============================================================

package terraform

import future.keywords.if
import future.keywords.contains

# ---- Rule 1: S3 buckets must not be public ----
# Public S3 buckets are the #1 cause of data breaches in AWS.
# This rule prevents any S3 resource from having public access enabled.
deny contains msg if {
    resource := input.resource.aws_s3_bucket[name]
    resource.acl == "public-read"
    msg := sprintf("DENY: S3 bucket '%v' has public-read ACL. All buckets must be private.", [name])
}

deny contains msg if {
    resource := input.resource.aws_s3_bucket[name]
    resource.acl == "public-read-write"
    msg := sprintf("DENY: S3 bucket '%v' has public-read-write ACL. All buckets must be private.", [name])
}

# ---- Rule 2: S3 buckets must have versioning enabled ----
# Versioning allows recovery from accidental deletion or ransomware.
# Required for buckets that store application state or audit logs.
deny contains msg if {
    resource := input.resource.aws_s3_bucket[name]
    not resource.versioning[_].enabled
    msg := sprintf("DENY: S3 bucket '%v' must have versioning enabled. Required for audit trail and recovery.", [name])
}

# ---- Rule 3: Security groups must not allow 0.0.0.0/0 on sensitive ports ----
# Port 22 (SSH) and 3306 (MySQL) open to the world are critical violations.
deny contains msg if {
    resource := input.resource.aws_security_group[sg_name]
    ingress := resource.ingress[_]
    ingress.cidr_blocks[_] == "0.0.0.0/0"
    sensitive_port(ingress.from_port)
    msg := sprintf("DENY: Security group '%v' allows 0.0.0.0/0 on port %v. Use a restricted CIDR.", [sg_name, ingress.from_port])
}

# ---- Rule 4: IAM roles must not use wildcard actions ----
# Wildcard IAM actions grant more permissions than needed.
# Violates least-privilege principle.
deny contains msg if {
    resource := input.resource.aws_iam_policy[policy_name]
    statement := resource.policy.Statement[_]
    statement.Effect == "Allow"
    statement.Action == "*"
    msg := sprintf("DENY: IAM policy '%v' uses wildcard Action '*'. Specify minimum required actions.", [policy_name])
}

# ---- Rule 5: EC2 instances must not have public IP auto-assignment ----
deny contains msg if {
    resource := input.resource.aws_instance[name]
    resource.associate_public_ip_address == true
    msg := sprintf("DENY: EC2 instance '%v' auto-assigns public IP. Place instances in private subnets.", [name])
}

# ---- Helper: defines sensitive ports ----
sensitive_port(port) if port == 22    # SSH
sensitive_port(port) if port == 3306  # MySQL
sensitive_port(port) if port == 5432  # PostgreSQL
```

```bash
cat > ~/portfolio/notification-service/.github/workflows/devsecops-pipeline.yml << 'EOF'
# ============================================================
# devsecops-pipeline.yml
# End-to-End DevSecOps Pipeline — Notification Service
# Portfolio Capstone #2
# ============================================================

name: DevSecOps Pipeline — Notification Service

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

# These permissions are required for Cosign keyless signing.
# id-token: write — allows the workflow to request a GitHub OIDC token
# packages: write — allows pushing to GitHub Container Registry (GHCR)
permissions:
  contents: read
  id-token: write
  packages: write
  security-events: write  # Allows uploading SARIF results to GitHub Security tab

# Use the git SHA as the image tag — every image is traceable to a commit.
# Never use 'latest' — it's ambiguous and causes deployment confusion.
env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository_owner }}/notification-service
  IMAGE_TAG: ${{ github.sha }}

jobs:

  # ============================================================
  # STAGE 1: Pre-flight secret and SAST scan
  # Runs on every push and PR
  # ============================================================
  secret-and-sast-scan:
    name: "Stage 1 — Secret Scan + SAST"
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4
        with:
          # fetch-depth: 0 fetches full git history — required for Gitleaks
          # to scan all commits, not just the latest
          fetch-depth: 0

      - name: Install Gitleaks
        run: |
          curl -sSL "https://github.com/gitleaks/gitleaks/releases/download/v8.18.4/gitleaks_8.18.4_linux_x64.tar.gz" \
            -o /tmp/gitleaks.tar.gz
          tar -xzf /tmp/gitleaks.tar.gz -C /tmp gitleaks
          sudo mv /tmp/gitleaks /usr/local/bin/gitleaks

      - name: Scan for secrets with Gitleaks
        # --no-git: scan all files, not just committed ones
        # --redact: mask the actual secret value in output (safe for logs)
        # Exit code 1 means secrets found — pipeline stops here
        run: gitleaks detect --source=. --redact --no-git --verbose

      - name: Install Semgrep
        run: pip3 install semgrep --quiet

      - name: SAST scan with Semgrep
        # p/python: community ruleset for Python security anti-patterns
        # p/secrets: additional secret detection patterns
        # --error: exit non-zero if any findings exist
        # --sarif: output in SARIF format for GitHub Security tab upload
        run: |
          semgrep --config=p/python --config=p/secrets \
            --error \
            --sarif \
            --output=semgrep-results.sarif \
            src/

      - name: Upload Semgrep results to GitHub Security tab
        uses: github/codeql-action/upload-sarif@v3
        if: always()  # Upload even if semgrep found issues — for visibility
        with:
          sarif_file: semgrep-results.sarif

  # ============================================================
  # STAGE 2: PR policy check — Dockerfile and OPA validation
  # ============================================================
  pr-policy-check:
    name: "Stage 2 — Dockerfile Scan + OPA Policy"
    runs-on: ubuntu-latest
    needs: secret-and-sast-scan  # Only runs if Stage 1 passes

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Install Trivy
        run: |
          curl -sSL "https://github.com/aquasecurity/trivy/releases/download/v0.51.2/trivy_0.51.2_Linux-64bit.tar.gz" \
            -o /tmp/trivy.tar.gz
          tar -xzf /tmp/trivy.tar.gz -C /tmp trivy
          sudo mv /tmp/trivy /usr/local/bin/trivy

      - name: Scan Dockerfile with Trivy (misconfiguration check)
        # config: scan mode targets Dockerfiles and IaC, not container images
        # --exit-code 1: fail if any HIGH or CRITICAL misconfigurations found
        # --severity HIGH,CRITICAL: only fail on high-impact findings
        run: |
          trivy config \
            --exit-code 1 \
            --severity HIGH,CRITICAL \
            --format table \
            Dockerfile

      - name: Install Conftest (OPA policy runner)
        run: |
          curl -sSL "https://github.com/open-policy-agent/conftest/releases/download/v0.51.0/conftest_0.51.0_Linux_x86_64.tar.gz" \
            -o /tmp/conftest.tar.gz
          tar -xzf /tmp/conftest.tar.gz -C /tmp conftest
          sudo mv /tmp/conftest /usr/local/bin/conftest

      - name: Run OPA policy check on Dockerfile
        # conftest parse: converts Dockerfile into JSON for OPA consumption
        # conftest test: runs .rego policies against the parsed JSON
        run: |
          # Parse Dockerfile to JSON, then validate against image-policy.rego
          conftest test Dockerfile \
            --policy policy/image-policy.rego \
            --parser dockerfile

      - name: Scan Terraform IaC with Trivy (if terraform/ exists)
        # Only runs if a terraform directory exists in the repo
        # This makes the pipeline reusable for repos with and without IaC
        run: |
          if [ -d "terraform" ]; then
            trivy config \
              --exit-code 1 \
              --severity HIGH,CRITICAL \
              terraform/
          else
            echo "No terraform/ directory found — skipping IaC scan"
          fi

  # ============================================================
  # STAGE 3: Build — Docker image with git SHA tag
  # ============================================================
  build:
    name: "Stage 3 — Docker Build"
    runs-on: ubuntu-latest
    needs: pr-policy-check  # Only runs if Stages 1 and 2 pass
    # Only build on push to main or develop — not on PRs
    # PRs get the scan/policy checks but not the full build+sign pipeline
    if: github.event_name == 'push'

    outputs:
      # Pass the image digest to downstream jobs for signing and verification
      image-digest: ${{ steps.build.outputs.digest }}
      image-ref: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ env.IMAGE_TAG }}

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        # Buildx enables multi-platform builds and build cache export
        uses: docker/setup-buildx-action@v3

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push Docker image
        id: build
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          # Tag with git SHA for traceability
          # Also tag with branch name for human-readable reference
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ env.IMAGE_TAG }}
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.ref_name }}
          # Export image digest — used by Cosign for signing
          # The digest is the SHA256 hash of the image manifest
          outputs: type=registry,digest=true
          # Enable build cache — speeds up repeated builds by 60-80%
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ============================================================
  # STAGE 4: Sign — Cosign keyless signing
  # ============================================================
  sign:
    name: "Stage 4 — Cosign Keyless Sign"
    runs-on: ubuntu-latest
    needs: build

    steps:
      - name: Install Cosign
        uses: sigstore/cosign-installer@v3
        with:
          cosign-release: v2.2.4

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Sign the image with Cosign (keyless)
        # Keyless signing uses the GitHub OIDC token — no private key needed.
        # The signature is stored in the registry alongside the image.
        # COSIGN_EXPERIMENTAL=1 enables the keyless flow.
        # The image is identified by digest (immutable) not tag (mutable).
        env:
          COSIGN_EXPERIMENTAL: "1"
        run: |
          cosign sign --yes \
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ needs.build.outputs.image-digest }}

      - name: Verify the signature
        # Immediately verify that the signature we just created is valid.
        # This catches any signing errors before the image is consumed downstream.
        env:
          COSIGN_EXPERIMENTAL: "1"
        run: |
          cosign verify \
            --certificate-identity-regexp="https://github.com/${{ github.repository }}/*" \
            --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ needs.build.outputs.image-digest }}

  # ============================================================
  # STAGE 5: SBOM generation and CVE scan
  # ============================================================
  sbom-and-cve-scan:
    name: "Stage 5 — SBOM + CVE Scan"
    runs-on: ubuntu-latest
    needs: sign

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Install Syft (SBOM generator)
        run: |
          curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | \
            sudo sh -s -- -b /usr/local/bin v1.4.1

      - name: Install Grype (CVE scanner)
        run: |
          curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh | \
            sudo sh -s -- -b /usr/local/bin v0.78.0

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Generate SBOM with Syft
        # cyclonedx-json: industry-standard SBOM format, supported by most tooling
        # Output to sbom/ directory — this becomes a pipeline artifact
        run: |
          mkdir -p sbom
          syft ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ env.IMAGE_TAG }} \
            --output cyclonedx-json=sbom/notification-service-sbom.json \
            --output table

      - name: Upload SBOM as pipeline artifact
        # Store SBOM as a downloadable artifact tied to this pipeline run.
        # This is your evidence package for security audits.
        uses: actions/upload-artifact@v4
        with:
          name: sbom-${{ github.sha }}
          path: sbom/notification-service-sbom.json
          retention-days: 90

      - name: Scan SBOM for CVEs with Grype
        # --file: read SBOM from file (faster than pulling image again)
        # --output json: machine-readable output for the deploy gate
        # Note: we do NOT set --fail-on here — the deploy gate OPA policy
        # makes the go/no-go decision. Grype just provides the data.
        run: |
          grype sbom:sbom/notification-service-sbom.json \
            --output json \
            --file grype-results.json
          
          # Also output human-readable table to pipeline logs
          grype sbom:sbom/notification-service-sbom.json \
            --output table

      - name: Upload Grype results as artifact
        uses: actions/upload-artifact@v4
        with:
          name: grype-results-${{ github.sha }}
          path: grype-results.json
          retention-days: 90

  # ============================================================
  # STAGE 6: Deploy gate — OPA policy decision on CVE results
  # ============================================================
  deploy-gate:
    name: "Stage 6 — Deploy Gate (CVE Policy)"
    runs-on: ubuntu-latest
    needs: sbom-and-cve-scan

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download Grype results
        uses: actions/download-artifact@v4
        with:
          name: grype-results-${{ github.sha }}

      - name: Install Conftest
        run: |
          curl -sSL "https://github.com/open-policy-agent/conftest/releases/download/v0.51.0/conftest_0.51.0_Linux_x86_64.tar.gz" \
            -o /tmp/conftest.tar.gz
          tar -xzf /tmp/conftest.tar.gz -C /tmp conftest
          sudo mv /tmp/conftest /usr/local/bin/conftest

      - name: Write deploy gate OPA policy
        # This policy reads Grype's JSON output and blocks deployment
        # if any HIGH or CRITICAL vulnerabilities exist.
        run: |
          cat > /tmp/deploy-gate.rego << 'REGO'
          package deploygate

          import future.keywords.if
          import future.keywords.contains

          # Block deployment if any HIGH or CRITICAL CVEs exist in the SBOM scan
          deny contains msg if {
              match := input.matches[_]
              vulnerability := match.vulnerability
              severity := vulnerability.severity
              # Block on HIGH and CRITICAL — MEDIUM and below are tracked but not blocking
              severity == "High"
              msg := sprintf(
                  "DEPLOY BLOCKED: HIGH CVE %v in %v %v — remediate before deploying to production",
                  [vulnerability.id, match.artifact.name, match.artifact.version]
              )
          }

          deny contains msg if {
              match := input.matches[_]
              vulnerability := match.vulnerability
              severity := vulnerability.severity
              severity == "Critical"
              msg := sprintf(
                  "DEPLOY BLOCKED: CRITICAL CVE %v in %v %v — immediate remediation required",
                  [vulnerability.id, match.artifact.name, match.artifact.version]
              )
          }

          # Warn on MEDIUM CVEs — visible in output but not blocking
          warn contains msg if {
              match := input.matches[_]
              vulnerability := match.vulnerability
              vulnerability.severity == "Medium"
              msg := sprintf(
                  "WARN: MEDIUM CVE %v in %v %v — schedule remediation within 30 days",
                  [vulnerability.id, match.artifact.name, match.artifact.version]
              )
          }
          REGO

      - name: Run deploy gate policy check
        # If any HIGH/CRITICAL CVEs exist in grype-results.json,
        # conftest exits non-zero and the pipeline fails here.
        # The image exists in the registry but will NOT receive the 'approved' tag.
        run: |
          conftest test grype-results.json \
            --policy /tmp/deploy-gate.rego \
            --input json

      - name: Tag image as approved (deploy gate passed)
        # Only reached if the CVE policy check passes.
        # Apply the 'approved' tag — downstream CD systems watch for this tag.
        # In production: this would trigger ArgoCD or Flux to deploy.
        if: success()
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Apply approved tag
        if: success()
        run: |
          echo "Deploy gate passed — image is approved for deployment"
          echo "Image: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ env.IMAGE_TAG }}"
          echo "In production: ArgoCD or Flux would detect the 'approved' tag and begin rollout"
          # In a real production system, you would run:
          # docker buildx imagetools create \
          #   --tag ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:approved \
          #   ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ env.IMAGE_TAG }}
EOF
```

**WHY:** This is the full pipeline. Each stage `needs:` the previous one — this is the serial dependency chain. The deploy gate is the last checkpoint. Nothing is tagged "approved" unless all six stages pass.

```bash
cat > ~/portfolio/notification-service/PIPELINE_DESIGN.md << 'EOF'
# Pipeline Design — Architecture Decision Record
## Notification Service DevSecOps Pipeline

**Author**: Vishweshwar Chippa  
**Date**: 2025-06  
**Status**: Active  

---

## Problem Statement

The notification service processes 25M+ messages per day. A single vulnerable container deployed to production could expose customer PII or degrade the message delivery SLA. The pipeline must catch security issues as early as possible — before code is merged, before images are built, and before anything reaches the registry.

## Tool Selection Rationale

| Stage | Tool Chosen | Alternatives Considered | Why This Tool |
|---|---|---|---|
| Secret scan | Gitleaks | git-secrets, truffleHog | Gitleaks maintains 150+ secret type rules, runs in pre-commit hook with zero server dependency, and exits non-zero reliably |
| SAST | Semgrep | SonarQube, Bandit | Semgrep runs in milliseconds locally, has maintained community rulesets (p/python), and outputs SARIF for GitHub Security integration |
| Container misconfiguration | Trivy config | Hadolint, Checkov | Trivy covers Dockerfile, OS packages, and IaC in one binary — reduces tool sprawl |
| Policy enforcement | OPA/Conftest | Kyverno, inline bash | OPA policies are versionable, auditable, and composable across Dockerfile, Terraform, and K8s YAML |
| SBOM generation | Syft (CycloneDX) | Docker SBOM plugin, Trivy SBOM | Syft outputs CycloneDX and SPDX natively, integrates with Grype without format translation |
| CVE scan | Grype | Trivy, Snyk | Grype natively consumes Syft SBOM; clean JSON output for OPA policy consumption |
| Image signing | Cosign keyless | GPG signing, Notary v2 | Keyless eliminates private key management; GitHub OIDC token provides verifiable identity without stored secrets |

## Security Gates (Hard Stops)

1. **Secret detected by Gitleaks** → Pipeline fails at Stage 1. No code reaches CI.
2. **HIGH/CRITICAL Dockerfile misconfiguration** → Pipeline fails at Stage 2. No build.
3. **OPA policy violation** → Pipeline fails at Stage 2. No build.
4. **Build failure** → No signed image. No SBOM.
5. **HIGH/CRITICAL CVE in Grype scan** → Pipeline fails at Stage 6. Image exists in registry but never receives 'approved' tag.

## Security Gates (Warnings — Non-Blocking)

- MEDIUM CVEs → Logged, tracked, 30-day remediation SLA
- Missing HEALTHCHECK → Warning in OPA output
- Missing EXPOSE → Warning in OPA output

## SBOM as Audit Evidence

Every pipeline run produces a CycloneDX SBOM artifact stored for 90 days. This enables:
- **Zero-day response**: Query all SBOMs for a specific library in minutes (Log4Shell scenario)
- **Compliance audit**: Prove what libraries were in production at any point in time
- **Supply chain attestation**: SBOM + Cosign signature = cryptographic chain of custody

## CVE Allowlist Process

When a CVE is flagged but assessed as non-exploitable:
1. Add entry to `.grype.yaml` with fields: CVE ID, package, justification, reviewer, review date, re-review date (90 days)
2. Submit as PR with security team reviewer required
3. Re-review date triggers a calendar reminder — not a manual process

## Deployment Gate Pattern

The 'approved' tag pattern decouples the build pipeline from the CD system. ArgoCD or Flux watches for images tagged 'approved' in the registry. This means:
- The pipeline is the sole authority on what is safe to deploy
- The CD system is not aware of the security posture — it trusts the tag
- Revoking 'approved' from an image immediately stops CD from deploying it

## Known Limitations

1. **SBOM accuracy**: SBOM is accurate at build time. Runtime library loading (JVM classpath, Python dynamic imports) is not captured. Mitigated by runtime security tools (Falco) as a complementary layer.
2. **Policy bypass**: A developer with registry write access could push an image directly, bypassing the pipeline. Mitigated by registry-level admission control (Kyverno or OPA Gatekeeper at the cluster level).
3. **CVE latency**: Grype's vulnerability database is updated daily. A CVE disclosed at 09:00 may not appear in a pipeline run at 08:50. Mitigated by scheduled nightly re-scans of all approved images.
EOF
```

**WHY:** The ADR is what a VP asks for when they review your pipeline. It proves you made deliberate decisions, not random tool choices. This document is your interview talking point.

### MODULE 3 — Break/Fix Exercise (30 min)

This module intentionally introduces a security violation, observes the pipeline block, and remediates. This is the most valuable interview preparation — "tell me about a time you caught a vulnerability in your pipeline."

#### Step 24: Introduce a HIGH CVE (break)

```bash

# to use an old base image that has known HIGH CVEs.
# 
# In a real scenario, this happens when:
```

## Lab ANSWER KEY

**Step 15 fails (pre-commit hook errors):**
- Gitleaks fails: run `gitleaks detect --source=. --no-git --verbose` to see which file/line triggered. Look for anything that looks like a token, key, or password. Remove or move to environment variable.
- Semgrep fails: run `semgrep --config=p/python src/ --verbose` to see the specific rule. Common causes: `os.system()` with user input, hardcoded string that looks like a password, `eval()` usage.
- If hook is hanging: `Ctrl+C`, then run `pre-commit run --all-files` to see full output.

**Step 18 fails (Conftest OPA policy):**
- `DENY: Dockerfile must specify a non-root USER`: Add `RUN useradd --uid 1000 appuser && USER appuser` before CMD.
- `DENY: Base image must be pinned by digest`: Replace `FROM python:3.12-slim` with `FROM python:3.12-slim@sha256:<digest>`. Get digest with: `docker pull python:3.12-slim && docker inspect python:3.12-slim --format='{{index .RepoDigests 0}}'`
- `conftest: command not found`: Re-run Step 5 install.

**Step 19 fails (Docker build):**
- `sha256 not found`: The digest in the FROM line is fictional for this lab. Replace with a real digest using the command above.
- `pip install failed`: Check network connectivity. Try `pip install --no-cache-dir -r requirements.txt` manually.
- `useradd: command not found`: Add `RUN apt-get update && apt-get install -y passwd` before the useradd command (some slim images omit it).

**Step 21 fails (Grype shows HIGH CVEs on clean image):**
- The base image has been updated by upstream and now has HIGH CVEs — this is the SBOM drift scenario.
- Fix: pull the latest `python:3.12-slim`, get its current digest, update the Dockerfile FROM line.
- If CVE is in a library you control: update the version in `requirements.txt`.

**Step 25 fails unexpectedly (OPA policy fails on clean Dockerfile):**
- Check that conftest is using the correct parser: `--parser dockerfile` is required.
- Check the .rego file for syntax errors: `conftest verify --policy policy/image-policy.rego`
- Common Rego mistake: missing `import future.keywords.contains` for `contains` keyword in OPA 0.51+.

## STAR Anchor — T-Mobile Framing

**Q1 — Pipeline Design (Director-level answer):**

"At T-Mobile, our notification platform was processing 25 million messages per day with a polyglot microservice fleet. When we moved to container-based deployments, I designed a security pipeline modeled on shift-left principles. **Situation**: We had no systematic image scanning — vulnerability discovery was reactive, triggered by CVE disclosures. **Task**: I needed to build a pipeline that caught vulnerabilities before they reached the registry. **Action**: I implemented a six-stage pipeline: pre-commit secret scanning with Gitleaks, SAST with Semgrep, Dockerfile policy enforcement with OPA/Conftest, container scanning with Trivy, SBOM generation with Syft, and CVE gating with Grype — all before the image received an 'approved' tag that our CD system would act on. Cosign keyless signing gave us cryptographic chain of custody without a private key to manage. **Result**: We reduced mean time to detect vulnerable images from days (reactive CVE disclosure response) to zero — they never entered the registry. The SBOM also gave us the ability to respond to a Log4Shell-type event in minutes, not the 96-hour scramble we had in 2021."

**Q2 — CVE Response at Scale:**

"This is the business case for SBOMs. **Situation**: Log4Shell in 2021 — I had to manually triage 40+ services because we had no SBOM. **Task**: After that incident I committed to SBOM as standard practice. **Action**: With Syft-generated CycloneDX SBOMs stored as pipeline artifacts, I can run a Grype query across all 200 SBOMs in under 5 minutes: `grype db check` to confirm database is current, then iterate over the SBOM archive and filter for the CVE ID. **Result**: When a zero-day dropped last quarter, I had a full blast radius report in 8 minutes — services affected, library versions, owners, and a Jira bulk-create for remediation tickets. Without the SBOM, that was a 3-day fire drill."

**Q3 — Developer Hook Bypass:**

"This is a governance question, not a tooling question. If developers are bypassing pre-commit hooks, the hooks are either too slow or too noisy. **Action**: First, measure — add telemetry to the CI check to track what percentage of commits arrive with hook bypass flags. If it's more than 5%, you have a systemic adoption failure. Second, fix the performance: cache Semgrep rules locally, use Semgrep's `--pro` mode for faster differential scanning. Third, make bypass visible: configure branch protection to require CI passing even for direct pushes — the pre-commit hook is convenience, the CI check is the real gate. As a Director, I frame this as a developer experience problem, not a compliance problem. Hooks should be fast enough that bypassing them feels like more work than running them."

**Q4 — Signing vs. Scanning:**

"Scanning answers: *does this image have known vulnerabilities?* Signing answers: *did this specific image come from my authorized pipeline?* They address different threat vectors. Scanning catches CVEs in dependencies — vulnerabilities that exist in the code. Signing catches tampering — an attacker who replaces a clean image in your registry with a malicious one. A registry with mandatory scanning could be bypassed by an insider or a compromised service account pushing directly. Cosign signing means that even if someone pushes a clean-scanning malicious image, it won't have the pipeline's signature — and your admission controller at the Kubernetes level rejects unsigned images. You need both: scanning for vulnerability management, signing for integrity verification."

**Q5 — SBOM Operationalization:**

"An SBOM is only valuable if you query it before you need it in a crisis. At T-Mobile, I operationalized SBOM data in three ways. First, a nightly Grype scan across all SBOMs in the artifact store — any new CVEs that match existing packages generate a Jira ticket automatically, not a manual review. Second, the security team uses SBOM data for quarterly dependency hygiene reviews — identifying which packages are approaching end-of-life before they become vulnerable. Third, during incident response, the SBOM is the first artifact the on-call SRE pulls to understand the blast radius of a disclosed CVE. The Director's responsibility is ensuring the SBOM data is in a queryable store — not a zip file in an S3 bucket — and that the team has runbooks for how to use it during an incident."

## Tradeoffs and Failure Modes

**Tradeoff 1: Pre-commit hooks vs. CI-only checks**

Pre-commit hooks give instant developer feedback and cost zero CI minutes. But they run on the developer's local machine — which means they can be bypassed (`--no-verify`), can have version drift (developer has Semgrep 1.0, CI uses 1.4), and don't run in forks. The correct architecture is *both*: pre-commit for developer convenience, CI for enforcement. The CI check is the authoritative gate. The pre-commit hook is a fast-feedback quality-of-life feature. Never design a pipeline where bypassing the pre-commit hook is the only way to get around a gate.

**Tradeoff 2: Blocking on HIGH CVEs vs. tracking them**

Blocking all HIGH CVEs sounds maximally secure but causes deploy paralysis in practice. A CRITICAL CVE in a C library that's only used in a test helper is not a production risk. A MEDIUM CVE in the authentication library IS a production risk. The right design uses CVE severity *plus* context: is the package reachable in the production runtime? Is there a fix available? Is there an active exploit? This is why the CVE allowlist process with documented justification exists — not as a way to ignore CVEs, but as a way to distinguish "not exploitable in our context" from "we chose to ignore it." Blanket suppression without documentation is a compliance failure. Documented risk acceptance with a 90-day re-review is mature risk management.

**Tradeoff 3: Keyless signing complexity vs. key management burden**

Keyless signing is cryptographically elegant but operationally newer. Your security auditors may not be familiar with Sigstore's Fulcio CA or the Rekor transparency log. The verification command is more complex than `gpg --verify`. The identity is tied to the GitHub Actions OIDC token — which means signing only works inside GitHub Actions, not in a developer's local build. The tradeoff: keyless eliminates the private key as a persistent attack surface (the biggest practical security win) at the cost of operational complexity in verification and platform lock-in to GitHub's OIDC. For organizations with an existing PKI infrastructure and HSM-backed code signing keys, traditional signing with Notary v2 may be more auditor-friendly. For a cloud-native team starting fresh, keyless is the right default.
