# Observability and Telemetry Design

## Why this matters for top-paying Principal SRE roles

At Principal level, you are expected to reduce time-to-understanding during incidents. Observability is how teams answer unknown questions quickly: *What is failing, where, why now, and who is impacted?*

**Observability** — Your system's ability to tell you what's happening inside it, *without you having to guess or redeploy*. If you can ask any question about system behavior and get an answer from existing data, your system is observable.

**Telemetry** — The data your system automatically collects and ships out so you can observe it. Metrics, logs, and traces are all forms of telemetry — the raw signal that makes observability possible.

Think of it this way: **telemetry is the instrument panel**, **observability is your ability to read and trust that panel** well enough to land the plane in bad weather.

## Simple explanation

Observability combines:
- **Metrics** (numbers over time)
- **Logs** (detailed events)
- **Traces** (request path across services)

Good telemetry answers:
1. Is the user impacted?
2. Which dependency is unhealthy?
3. Did this start after a change?

Think of it like medicine:
- Metrics = vital signs
- Logs = clinical notes
- Traces = full procedure history

## Real-world examples

### Example 1: API latency spike
- Metrics show p99 latency increased.
- Trace shows delay concentrated in inventory service.
- Logs confirm DB lock waits.
- Fix targets DB query/index, not random scaling.

### Example 2: Intermittent 500s
- Error metric rises only in one region.
- Logs show token validation failures after key rotation.
- Trace correlation confirms auth hop fails first.
- Recovery: sync key distribution + add rotation safeguards.

### Example 3: Queue backlog
- Queue depth metric climbs.
- Consumer logs reveal retry storms.
- Trace spans show dependency timeout chain.
- Fix: capped retries + jitter + circuit breaker.

### Example 4: “Everything green, users unhappy”
- Infra metrics normal.
- User journey SLI reveals checkout failures for mobile app version X.
- Logging dimension by client version surfaces bad request serialization bug.

## Variations and tradeoffs

- **RED vs USE dashboards**
  - RED (Rate, Errors, Duration) is service-centric.
  - USE (Utilization, Saturation, Errors) is infra-centric.
  - Best systems combine both.

- **High-cardinality labels**
  - More debugging power.
  - Higher cost and query complexity.

- **Full tracing vs sampled tracing**
  - Full = richest visibility, expensive.
  - Sampled = cheaper, may miss edge failures.

## Best practices and guidelines

- Start with user-journey SLIs before infra-only metrics.
- Standardize telemetry schema (service name, region, env, trace ID).
- Use structured logs (JSON) with consistent fields.
- Enforce trace propagation across all services.
- Build golden dashboards per critical service.
- Alert on SLO burn-rate, not only static thresholds.
- Keep runbook links directly in alert payloads.
- Review telemetry quality quarterly (missing labels, broken traces).

## Common mistakes

- Thousands of dashboards, no ownership.
- Alerts with no actionability.
- Logs without correlation IDs.
- Averaging latency (hides tail pain).
- Not instrumenting async workflows.

## Interview Q&A (strong answers)

### Q1: What does “good observability” look like?
It means responders can identify impact, fault domain, and first mitigation path within minutes. Practically: strong service SLIs, correlated metrics/logs/traces, actionable alerts, and runbook-linked incident workflows.

### Q2: How do you design alerts?
I design alerts from user-impact and SLO burn first. Each alert must be actionable, have an owner, include context (dashboard/runbook links), and map to a clear severity policy.

### Q3: Metrics vs logs vs traces—what do you check first?
Start with metrics for scope and urgency, use traces to localize the failing path, then logs for exact error details and validation. Sequence matters under pressure.

### Q4: How do you control observability cost?
Use tiered retention, sampling policies, cardinality guardrails, and per-service ownership. Keep high-fidelity telemetry for critical paths and event windows.

## 30-day mastery plan

- **Week 1:** Define telemetry standards and tagging contract.
- **Week 2:** Build golden dashboards for 3 critical services.
- **Week 3:** Implement burn-rate alerting and alert quality review.
- **Week 4:** Run an observability game day and close discovered gaps.

By day 30, you should be able to diagnose multi-service incidents rapidly and explain telemetry strategy at leadership level.

## Log Design Principles

Logs are not console output. They are structured events that must be queryable, correlatable, and actionable under incident pressure. Design them like a schema, not like debug print statements.

### Principle 1 — Structure Every Log (Structured Logging)

Always emit JSON. Never free-text prose. Every field must be queryable.

```json
{
  "timestamp": "2026-07-09T14:23:01.342Z",
  "level": "ERROR",
  "service": "notification-dispatcher",
  "env": "prod",
  "region": "us-west-2",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "span_id": "00f067aa0ba902b7",
  "message": "DLQ threshold breached",
  "queue": "dnd-priority-high",
  "dlq_depth": 12400,
  "threshold": 10000,
  "consumer_group": "push-worker-v3"
}
```

**Why**: Splunk, Loki, CloudWatch Logs Insights, Elasticsearch — all work at field-level. Free text logs force regex hacks under pressure and kill MTTD.

---

### Principle 2 — Mandatory Context Fields (Tagging Contract)

Every log line, from every service, must carry these fields. No exceptions.

| Field | Purpose | Example |
|-------|---------|---------|
| `trace_id` | Correlate across services (W3C Trace Context standard) | `4bf92f35...` |
| `span_id` | Locate exact operation within a trace | `00f067aa...` |
| `service` | Which service emitted this | `notification-dispatcher` |
| `env` | prod / staging / dev — never mix | `prod` |
| `region` | AWS/cloud region | `us-west-2` |
| `version` | Deployed artifact version | `v3.4.1` |
| `correlation_id` | Business transaction ID (order ID, user ID, msg ID) | `notif-8823-xyz` |

**Industry term**: This is your **Telemetry Schema Contract** or **Semantic Conventions** (OpenTelemetry uses this exact term). Without it, no tool can correlate logs to traces to metrics automatically.

---

### Principle 3 — Log Levels Mean Something

| Level | When to use | Example |
|-------|------------|---------|
| `DEBUG` | Internal state — dev/staging only, never prod | Variable values, loop counts |
| `INFO` | Normal lifecycle events | Request received, message enqueued |
| `WARN` | Degraded but not broken — investigate soon | Retry attempt 2 of 3, latency p95 elevated |
| `ERROR` | Failure requiring action | DB connection lost, DLQ write failed |
| `FATAL` / `CRITICAL` | Service cannot continue — triggers page | Out of disk, cert expired, OOM |

**Anti-pattern**: Logging everything at `ERROR` level. When everything is an error, nothing is. Alerts built on `level=ERROR` become noise within a week.

---

### Principle 4 — Never Log PII or Secrets

No phone numbers, email addresses, SSNs, tokens, passwords, or credentials in logs — ever. This is both a security standard (OWASP Logging Cheat Sheet) and a regulatory requirement (GDPR, CCPA, HIPAA).

**Pattern**: Log IDs, not values.
- Log: `user_id=u-88231` — not `email=john.doe@example.com`
- Log: `token_prefix=ey...` (first 6 chars only) — not the full JWT

---

### Principle 5 — Log at System Boundaries, Not Everywhere

Log at the edge — entry point and exit point of every service call. Do not log inside every function. This keeps volume manageable and preserves signal quality.

```
[Incoming request logged] → [internal processing — no logs] → [outgoing response logged]
                                      ↓ if error occurs
                              [error log with full context]
```

**Think of it like your RabbitMQ message lifecycle**: you log when the message arrives at the queue, when it's consumed, and when it's acknowledged or dead-lettered — not every line of the consumer code.

---

### Principle 6 — Retention Tiers

Not all logs have the same value over time. Design retention by signal decay.

| Tier | Retention | Log types |
|------|-----------|-----------|
| Hot | 7–15 days | ERROR, FATAL — active incident use |
| Warm | 30–90 days | WARN, INFO — trend analysis, postmortems |
| Cold (archive) | 1–7 years | All levels — compliance, audit |

**Cost implication**: Warm/cold tiers in S3 or Glacier cost 10–100x less than hot index storage. This is a real engineering trade-off at Principal level.

## Metric Instrumentation Principles

Metrics are numbers over time. The question is: which numbers, measured how, labeled by what?

### Metric Types — Choose Correctly

| Type | Definition | Use for | Example |
|------|-----------|---------|---------|
| **Counter** | Monotonically increasing, never decreases | Requests served, errors, messages processed | `http_requests_total{status="500"}` |
| **Gauge** | Current value, can go up or down | Queue depth, active connections, memory used | `rabbitmq_queue_messages{queue="dnd-high"}` |
| **Histogram** | Distribution of values across buckets | Latency, payload size, response time | `http_request_duration_seconds_bucket{le="0.1"}` |
| **Summary** | Pre-computed percentiles at client side | p50, p95, p99 latency (when you can't use histogram) | `rpc_duration_seconds{quantile="0.99"}` |

**Critical rule**: Never average latency from raw counters. Use histograms. Averages hide tail pain — a p50 of 100ms with a p99 of 8s is a disaster disguised as good performance.

---

### Labeling Strategy — Cardinality Is a Cost

Every label dimension multiplies your time-series count. A metric with 3 labels of 10 values each = 1,000 time series. At 100 labels each with 100 values = 1 billion time series → Prometheus OOM, cost explosion.

**Safe label dimensions** (bounded, stable):
- `env` (prod/staging/dev — 3 values)
- `region` (us-east-1, us-west-2 — ~5 values)
- `status_code` (200, 404, 500 — ~10 values)
- `service` (bounded by number of services)

**Dangerous label dimensions** (unbounded, per-request):
- `user_id` — never
- `request_id` — never
- `timestamp` — never (that's what the time axis is for)

**Industry term**: This is the **cardinality problem**. High-cardinality data (per-user, per-request dimensions) belongs in **traces and logs**, not metrics.

---

### Instrumentation Approach — Push vs Pull

| Model | How it works | Tools | Best for |
|-------|-------------|-------|----------|
| **Pull (Scrape)** | Collector polls your service's `/metrics` endpoint | Prometheus | Long-running services with stable endpoints |
| **Push** | Service pushes metrics to a collector agent | StatsD, OTLP push, CloudWatch Agent | Batch jobs, lambdas, short-lived processes |
| **Sidecar/Agent** | Agent runs beside your container, collects automatically | ADOT Collector, Datadog Agent, OTel Collector | Kubernetes — preferred pattern |

**For Kubernetes (your world)**: OTel Collector as a DaemonSet pulls from pods, enriches with K8s metadata (pod name, namespace, node), and pushes to your backend (Prometheus, Grafana, Datadog). This is the **ADOT** (AWS Distro for OpenTelemetry) pattern on EKS.

---

### Naming Conventions

Follow Prometheus naming conventions — these are the de facto standard across the industry:

```
<namespace>_<subsystem>_<metric_name>_<unit>
```

| Good | Bad |
|------|-----|
| `notification_dispatcher_messages_processed_total` | `msgCount` |
| `rabbitmq_consumer_lag_messages` | `queue_backlog` |
| `http_request_duration_seconds` | `latency` |
| `jvm_heap_used_bytes` | `memUsed` |

Always suffix with unit: `_seconds`, `_bytes`, `_total`, `_ratio`. Never embed units in labels.

## Industry Frameworks and Acronyms — The Complete Map

This is what interviewers test at Principal/Director level. You need to know not just the acronym but *when each framework applies* and *what problem it solves*.

---

### Signal Frameworks — What Data to Collect

| Acronym | Stands For | What it covers | Best for |
|---------|-----------|----------------|----------|
| **MELT** | Metrics, Events, Logs, Traces | All four telemetry signal types together | Comprehensive observability posture (New Relic framework) |
| **MELTA** | MELT + Alerts | Adds alerts as a first-class signal | Teams that treat alert definitions as code (GitOps for alerts) |
| **OTel / OTEL** | OpenTelemetry | Vendor-neutral SDK + collector + protocol (OTLP) for all MELT signals | Standardizing instrumentation across polyglot services |
| **OTLP** | OpenTelemetry Protocol | Wire protocol for shipping telemetry to any backend | Replacing proprietary agents with one universal exporter |
| **W3C Trace Context** | W3C standard for trace header propagation | `traceparent` and `tracestate` HTTP headers | Making trace IDs follow a request across service boundaries |
| **eBPF** | Extended Berkeley Packet Filter | Kernel-level telemetry without code changes | Auto-instrumenting legacy services; security telemetry |

---

### Metric Methodology Frameworks — What to Measure

| Framework | Stands For | Dimensions | Designed For |
|-----------|-----------|-----------|-------------|
| **Four Golden Signals** | (Google SRE Book) | Latency, Traffic, Errors, Saturation | Any service — start here |
| **RED** | Rate, Errors, Duration | Requests per second, error rate, response time | Service-level (microservices, APIs) |
| **USE** | Utilization, Saturation, Errors | % capacity used, queue length, error count | Infrastructure resources (CPU, disk, network, queues) |
| **LETS** | Latency, Errors, Traffic, Saturation | Extended RED | Google's internal variation — similar to Four Golden Signals |
| **SPACE** | Satisfaction, Performance, Activity, Communication, Efficiency | Developer productivity | Platform engineering — measuring developer experience |

**RabbitMQ mental model**:
- USE applies to RabbitMQ itself: CPU utilization of the broker, queue saturation (depth vs. consumer rate), connection errors.
- RED applies to your consumers: message consumption rate, consumer processing errors, processing duration per message.
- Both views together = full picture of the messaging system.

---

### SLO / Reliability Frameworks

| Term | Full Form | Definition |
|------|----------|-----------|
| **SLI** | Service Level Indicator | The actual measurement: `availability = successful_requests / total_requests` |
| **SLO** | Service Level Objective | The target: "99.9% of requests succeed in <200ms over 30 days" |
| **SLA** | Service Level Agreement | The contractual commitment to a customer — breach = financial penalty |
| **Error Budget** | — | `1 - SLO` expressed as allowable downtime/errors. 99.9% SLO = 43.8 min/month error budget |
| **Burn Rate** | — | How fast you are consuming error budget. Burn rate 1 = exactly on pace. Burn rate 14 = burning 30 days of budget in ~2 days |
| **Toil** | — | Repetitive, manual, automatable operational work. Google defines toil as work that scales linearly with service growth |
| **Error Budget Policy** | — | What actions trigger when budget is consumed: freeze releases, page on-call, escalate |

---

### Incident / MTTR Metrics

| Acronym | Stands For | What it measures |
|---------|-----------|-----------------|
| **MTTD** | Mean Time to Detect | Time from failure start to first alert firing |
| **MTTA** | Mean Time to Acknowledge | Time from alert to on-call picking it up |
| **MTTR** | Mean Time to Restore | Time from failure to full service restoration |
| **MTTF** | Mean Time to Failure | Average time a system runs before failing (reliability measure) |
| **MTBF** | Mean Time Between Failures | MTTF + MTTR — the full cycle time |
| **MTBI** | Mean Time Between Incidents | Broader than MTBF — includes all incident types |

**DORA Metrics** (DevOps Research and Assessment — industry gold standard):

| DORA Metric | What it measures | Elite benchmark |
|-------------|-----------------|----------------|
| **Deployment Frequency** | How often code ships to prod | Multiple times per day |
| **Lead Time for Changes** | Commit to prod time | < 1 hour |
| **Change Failure Rate** | % deploys causing incident | < 5% |
| **MTTR** | Time to restore after failure | < 1 hour |

**How observability connects to DORA**: Good telemetry directly improves MTTD → MTTR. Better MTTR → better DORA score → competitive advantage in hiring, retention, and platform reputation.

## The 4-Impact Model — Connecting Metrics to What Actually Matters

This is the framework that separates Principal SRE thinking from IC thinking. Every metric you collect must be traceable through these four impact layers. If it isn't, it's noise.

```
Raw Signal (metric/log/trace)
        ↓
[Layer 1] Technical Impact    — Is the system healthy?
        ↓
[Layer 2] Customer Impact     — Is the user experience degraded?
        ↓
[Layer 3] Business Impact     — Is the business outcome at risk?
        ↓
[Layer 4] Financial Impact    — What is this costing us in money?
```

---

### Layer 1 — Technical Impact (System / Platform / Flow)

**What it answers**: Is the infrastructure, platform, or data flow behaving as designed?

**Signal sources**: USE framework metrics, infrastructure dashboards, queue telemetry, Kubernetes control plane metrics.

| Metric Category | Example Signals | Tools |
|----------------|----------------|-------|
| **System resources** | CPU utilization, memory pressure, disk I/O, OOM events | USE framework, node-exporter, cAdvisor |
| **Platform health** | K8s pod restarts, node pressure taints, etcd latency, API server error rate | kube-state-metrics, Prometheus |
| **Queue / flow telemetry** | RabbitMQ queue depth, consumer lag, DLQ depth, unacked messages, publish rate vs consume rate | RabbitMQ Prometheus exporter, Grafana |
| **Network / service mesh** | Connection pool exhaustion, circuit breaker trips, retry storm rate, connection refused | Istio/Envoy metrics, Linkerd |
| **Database layer** | DB lock waits, slow query rate, replication lag, connection pool saturation | DB-specific exporters |

**DND platform example**: Consumer lag on `dnd-priority-high` queue crosses 10,000 messages → technical impact detected. Queue depth gauge (USE: Saturation) is the signal.

---

### Layer 2 — Customer Impact (User Experience / Service Quality)

**What it answers**: Is a real user's experience degraded right now?

**Signal sources**: SLIs measured from user journey, RED framework at API gateway, RUM data, synthetic probes.

| Metric Category | Example Signals | SLI Definition |
|----------------|----------------|---------------|
| **Availability** | % requests that succeed | `successful_requests / total_requests > 99.9%` |
| **Latency** | p95/p99 response time | `p99 latency < 500ms for 99.5% of requests` |
| **Error rate** | % 5xx errors to end users | `5xx_errors / total < 0.1%` |
| **Notification delivery** | % notifications delivered within SLA window | `delivered_within_60s / total_sent > 99.5%` |
| **User journey success** | % checkout / login / order completions | `journey_completions / journey_starts > 98%` |
| **Throughput degradation** | Requests served vs. expected demand | `served / expected > 95%` |

**Key principle**: Customer SLIs must be measured from the *customer's perspective*, not the platform's perspective. Your service reporting 200 OK while the mobile app receives a timeout is a customer impact with no platform signal — this is the "everything green, users unhappy" failure mode.

**Framework connection**: These are your **SLIs**. The targets set on them are your **SLOs**. The gap between 100% and your SLO is your **error budget**.

---

### Layer 3 — Business Impact (Outcomes / Operations)

**What it answers**: Is the degradation affecting business operations, product outcomes, or stakeholder commitments?

**Signal sources**: Business event streams, product analytics, cross-system correlation between ops metrics and business KPIs.

| Metric Category | Example Signals | Connected to |
|----------------|----------------|-------------|
| **Transaction volume** | Orders per minute, notification sends per second | Layer 2 delivery SLI |
| **Conversion rate** | Purchase completions, sign-ups, activations | Layer 2 user journey SLI |
| **Regulatory / compliance SLA** | TCPA notification delivery window compliance | Layer 2 delivery latency SLI |
| **Feature availability** | % users able to use a product feature | Layer 1 service health + Layer 2 error rate |
| **DORA metrics** | Deployment frequency, change failure rate, MTTR | Layer 1 platform + Layer 2 availability |
| **Incident frequency** | P1/P2 incidents per week | Layer 1 + 2 combined |

**DND platform example**: 12,000-message DLQ backlog (Layer 1 — technical) → 8% of T-Mobile customers did not receive opt-out confirmations within the TCPA-required 10-second window (Layer 2 — customer) → business is now in regulatory non-compliance (Layer 3 — business).

---

### Layer 4 — Financial Impact (Cost and Revenue)

**What it answers**: What is this incident, degradation, or architectural choice costing us in actual dollars?

**Signal sources**: Cloud billing APIs, revenue event streams, on-call cost accounting, infrastructure spend dashboards.

| Metric Category | Formula / Signal | Example |
|----------------|-----------------|---------|
| **Revenue at risk** | `failed_transactions × avg_order_value` | 500 failed checkouts × $85 avg = $42,500 revenue at risk per hour |
| **Error budget cost** | `budget_burned_% × annual_SLA_penalty_clause` | Contracts with SLA credits: 99.9% miss costs X |
| **MTTR cost** | `incident_duration_hrs × (eng_cost_per_hr × headcount_on_call)` | 3-hr P1 × 6 engineers × $150/hr = $2,700 direct labor cost |
| **Infrastructure waste** | Over-provisioned resources × hourly rate | 40 idle m5.large nodes × $0.096/hr = $3.84/hr waste = $33,000/year |
| **Observability tool cost** | Ingest volume × per-GB cost | 2TB/day log ingest × $0.30/GB = $600/day = $219,000/year |
| **Toil cost** | Hours of manual work per week × engineer hourly cost | 10 hrs/week toil × 5 engineers × $100/hr = $5,000/week = $260,000/year |

**FinOps connection**: Layer 4 is where observability intersects with **FinOps** (Cloud Financial Operations). Observability tooling itself has a cost, and that cost must be justified by the MTTR reduction, incident prevention, and toil elimination it delivers.

### The 4-Impact Model in Practice — Full Trace-Through

Here is how one signal propagates through all four layers. Use this pattern in every Principal-level incident debrief and architecture review.

**Scenario**: RabbitMQ consumer throughput drops 40% on the DND priority queue.

```
Signal:  rabbitmq_queue_messages{queue="dnd-priority-high"} crosses 10,000
         consumer_rate drops from 5,200/min to 3,100/min
         
Layer 1 — Technical Impact
  Queue saturation: depth climbing, consumer lag growing
  Root signal: USE framework — Saturation dimension
  Correlated: K8s pod CPU throttling on push-worker pods (HPA not yet triggered)

Layer 2 — Customer Impact
  SLI: notification delivery within 60s drops from 99.7% → 91.2%
  8.5% of customers not receiving time-sensitive notifications
  Synthetic probe: opt-out confirmation probe failing latency SLO

Layer 3 — Business Impact
  TCPA compliance window (10s for opt-out confirmations) breached for ~68,000 messages
  Customer support tickets begin rising (leading indicator: 15-min lag)
  SLO breach: 30-day error budget consumed 23% in 2 hours (burn rate = 82×)

Layer 4 — Financial Impact
  Direct: 68,000 non-compliant messages × potential TCPA fine exposure
  Labor: P2 incident — 4 engineers × 2.5 hrs × $150/hr = $1,500 MTTR cost
  Infra: Emergency HPA scale-out adds 12 pods × $0.20/hr = $2.40/hr additional spend
  Toil: Manual DLQ replay required — 45 min engineer time
```

## Observability as Code — The Modern Standard

At Principal/Director level, observability itself is treated as a software engineering discipline. Dashboards, alert rules, SLO definitions, and synthetic probes live in Git, are reviewed in PRs, and are deployed via CI/CD.

| Artifact | As Code Tool | Format |
|---------|-------------|--------|
| Prometheus alert rules | Prometheus Operator, Alertmanager | YAML |
| Grafana dashboards | Grafonnet, Terraform Grafana provider | JSON / HCL |
| SLO definitions | Sloth, OpenSLO | YAML |
| Synthetic probes | Checkly, k6, Blackbox Exporter | JS / YAML |
| Log pipelines | Vector, Fluent Bit config | TOML / YAML |
| OTel collector config | OTel Operator for K8s | YAML |

**The discipline**: If an alert is not in Git, it doesn't exist in production. If a dashboard was built by clicking in the UI and not exported to code, it will drift, be deleted, and take undocumented knowledge with it.

## Part III — Observability in the Age of AI

---

### Why Traditional Observability Is No Longer Enough

Classic observability assumes a human reads a dashboard, forms a hypothesis, and acts. That model breaks at scale.

At 25 million messages per day — or any platform of similar scale — you generate more telemetry than any human team can process in real time. The average large platform produces hundreds of thousands of metric time series, millions of log lines per hour, and thousands of distributed traces per minute. A human watching dashboards is statistically guaranteed to miss the signal that matters.

AI does not replace observability engineering. It **raises the bar for what your telemetry must provide** so that machines can act on it without guessing.

The new model:

```
Raw Telemetry (MELT)
      ↓
[AI/ML Layer] — anomaly detection, correlation, pattern recognition, root cause inference
      ↓
Audience-Aware Output — different signal delivered to different consumers
      ↓
[Human Decision Layer] — for actions that require judgment, context, or accountability
      ↓
[Autonomous Remediation Layer] — for actions that are safe to automate
```

---

### Advanced Observability Signals the AI Layer Needs

To feed AI reliably, your telemetry must go beyond basic MELT. These are the signals that separate AI-ready platforms from human-readable ones.

| Signal Type | What it is | Why AI needs it |
|------------|-----------|-----------------|
| **Baseline / seasonality metadata** | Normal ranges per time-of-day, day-of-week, event calendar | Without this, AI fires on Black Friday traffic spikes as "anomalies" |
| **Change events (change feed)** | Deployment timestamps, config changes, feature flag flips, certificate rotations | Correlating an anomaly to its cause requires knowing what changed and when |
| **Dependency topology** | Service graph — which services call which, with latency SLOs per edge | AI needs the graph to trace blast radius and identify upstream vs. downstream cause |
| **High-cardinality event data** | Per-request trace data with full attribute sets | Columnar trace stores (Honeycomb, ClickHouse) let AI slice on any attribute combination |
| **Feedback loops** | Was the alert actionable? Was the AI root cause correct? Was the auto-remediation successful? | Models degrade without ground truth. Human feedback closes the loop |
| **Synthetic user journeys** | Continuous probes that simulate real user flows | AI needs a stable "what good looks like" baseline that doesn't depend on real user traffic |
| **Cost telemetry** | Per-service, per-request cloud spend | AI-driven FinOps needs spend signal alongside performance signal |

**AIOps** is the industry term for applying ML to these signals. Tools: Splunk MLTK, Dynatrace Davis, Datadog Watchdog, AWS DevOps Guru, BigPanda, Moogsoft.

### Audience-Aware Observability — Different Signal for Different Consumers

The most important architectural shift in AI-era observability is this: **one telemetry pipeline, multiple audience-specific views**. The same raw signal must surface differently depending on who or what is consuming it.

```
                    ┌─────────────────────────────┐
                    │     Raw Telemetry Pipeline   │
                    │  (MELT + change events +     │
                    │   topology + cost signal)    │
                    └─────────────┬───────────────┘
                                  │
          ┌───────────────────────┼────────────────────────┐
          │                       │                        │
          ▼                       ▼                        ▼
  [AI/ML Engine]         [Human Operators]        [Autonomous Systems]
  Anomaly detection      SRE on-call              Auto-remediation
  Root cause inference   Incident commanders      Autoscalers / HPA
  Forecast / capacity    Platform engineers       Circuit breakers
  Blast radius scoring   Leadership / Directors   Chaos engineering
          │                       │                        │
          ▼                       ▼                        ▼
  Scored alerts          Actionable dashboards    Trigger events
  Correlated incidents   Runbook-linked alerts    Policy thresholds
  Predicted failures     SLO burn reports         Webhooks / event bus
  Capacity forecasts     Executive summaries      GitOps reconciliation
```

### Observability Tiers by Audience

#### Tier 1 — Autonomous Systems (Machines Only, No Human in the Loop)

**Who consumes**: Kubernetes HPA, KEDA, circuit breakers, auto-remediation runbooks, event-driven autoscalers.

**What they need**: Single-metric threshold signals, low latency, deterministic — no ambiguity.

| Signal | Example | Consumed by |
|--------|---------|-------------|
| Queue depth gauge | `rabbitmq_queue_messages > 10000` | KEDA — triggers consumer pod scale-out |
| CPU utilization | `cpu_usage_ratio > 0.80` | HPA — adds pod replicas |
| Circuit breaker state | `cb_state = OPEN` | API gateway — stops routing to degraded service |
| Error rate spike | `error_rate_5min > 0.05` | Auto-rollback pipeline — triggers canary rollback |
| DLQ depth | `dlq_messages > 500` | Event bridge — fires alert + pauses publisher |

**Design requirement for this tier**: Metrics must be **low cardinality, deterministic, and sub-30-second resolution**. No AI inference here — just threshold logic. Machines need yes/no decisions.

---

#### Tier 2 — On-Call SRE / Incident Responder (Human, High Pressure, Minutes Matter)

**Who consumes**: The person who just got paged at 2am and needs to understand impact, localize the fault, and take the first action within 5 minutes.

**What they need**: Pre-correlated, context-loaded, runbook-linked, prioritized by severity. Not raw data. Not 40 dashboards. One coherent incident view.

| Signal | Requirement | Why |
|--------|------------|-----|
| Correlated alert | Single alert that already links metric → trace → log sample | Eliminates the "find the log" step under pressure |
| Blast radius map | Which services and customers are affected | First question in every P1: who is impacted? |
| Change correlation | "This anomaly started 4 min after deployment `v3.4.1`" | Most common P1 cause — deployment regression |
| Recommended runbook | Direct link to the relevant runbook from the alert | Removes cognitive load during peak stress |
| SLO burn context | "You are burning error budget at 42× — at this rate budget exhausts in 17 min" | Tells responder how urgent the mitigation window is |
| AI-generated summary | "Likely cause: DB connection pool exhaustion in payment-service following config change at 14:20" | AIOps root cause inference — responder validates, doesn't discover |

**Tools that serve this tier**: PagerDuty (with AIOps correlation), Datadog Incident Management, FireHydrant, Grafana Incident, OpsGenie.

**Design requirement**: Everything above must be **precomputed and pre-assembled before the page fires**, not assembled by the responder during the incident.

---

#### Tier 3 — Platform / SRE Engineer (Human, Non-Incident, Deep Analysis)

**Who consumes**: Engineers doing postmortems, capacity planning, reliability investment decisions, toil reduction work.

**What they need**: Deep drill-down, high-cardinality trace data, trend analysis, cross-service comparisons, cost-performance trade-offs.

| Signal | Use |
|--------|-----|
| Full distributed traces with all attributes | Root cause analysis — slice by customer segment, region, version |
| Error budget consumption trend (30/60/90 day) | Reliability investment prioritization |
| Latency histogram trends | Detect gradual degradation before SLO breach |
| Dependency health scorecards | Identify which upstream is the most common blast radius source |
| Toil metrics | Quantify manual work to justify automation investment |
| Anomaly detection history | Review AI accuracy — tune thresholds, add feedback labels |

**Tools that serve this tier**: Honeycomb (high-cardinality trace analysis), Jaeger/Tempo (distributed tracing), Grafana Explore, Splunk (your domain — MLTK for anomaly pattern analysis).

---

#### Tier 4 — Engineering Leadership / Director / VP (Human, Weekly/Monthly, Strategic)

**Who consumes**: You, your peers, your VP. Not watching dashboards — receiving synthesized reports, making investment decisions, reporting to the business.

**What they need**: Aggregated reliability health, DORA metrics, cost trends, error budget posture, SLA compliance status, team toil burden, incident trend analysis.

| Signal | Format | Cadence |
|--------|--------|---------|
| SLO compliance by service | Traffic-light scorecard | Weekly |
| Error budget remaining by service | % and days-remaining | Weekly |
| DORA metrics (all four) | Trend chart vs. industry benchmark | Monthly |
| P1/P2 incident count and MTTR trend | Up/down vs. prior month | Monthly |
| Toil percentage (% eng time on toil) | Should be < 50% per Google SRE Book | Monthly |
| Observability tool spend | $ by tool, by team | Monthly |
| Top 3 reliability risks | AI-ranked by blast radius + likelihood | Weekly |

**Tools that serve this tier**: Grafana SLO dashboards, Cortex/Thanos for long-retention aggregation, Datadog SLO reports, custom Python + Splunk report pipelines (your existing strength), executive email digests generated by AIOps platforms.

---

#### Tier 5 — Business Stakeholders / Executive Leadership (Human, Monthly/Quarterly, Non-Technical)

**Who consumes**: CPO, CFO, CTO, board-level reporting. Zero tolerance for jargon. Decision currency is dollars, customers, and risk.

**What they need**: Business outcome translation — not metrics, not percentages. Impact on customers, revenue, regulatory compliance, and competitive position.

| Technical Signal | Business Translation |
|----------------|---------------------|
| SLO breach — 99.7% availability vs. 99.9% target | 24,000 customer transactions impacted this month vs. our SLA commitment |
| p99 latency 4x above baseline | Checkout abandonment rate increased 12% during the degradation window — estimated $340K revenue impact |
| TCPA notification delivery SLI at 98.2% | 1.8% of opt-out confirmations delivered outside the regulatory 10-second window — compliance exposure |
| MTTR improved from 47 min to 18 min | Engineering response time 62% faster — 3× reduction in customer-facing downtime hours |
| Toil reduced by 30% | Equivalent of 1.5 engineer headcount freed for feature work without adding headcount |

**Design requirement**: This tier must be **generated automatically** from the same telemetry pipeline that feeds Tiers 1–4. Manual report assembly is itself toil — and it introduces translation errors. AI-generated executive narratives (Datadog's AI-generated incident summaries, Dynatrace's root cause explanations) are where AIOps is delivering real value right now.

### The Observability Maturity Model in the AI Age

Use this to assess where your platform is and where it needs to go.

| Level | Name | Characteristics | AI Readiness |
|-------|------|----------------|-------------|
| **L1** | Reactive | Alerts fire after customers call. No structured logs. No trace IDs. | None — raw data not usable |
| **L2** | Aware | Dashboards exist. Structured logs. Basic alerting. On static thresholds. | Low — data exists but no baselines |
| **L3** | Proactive | RED/USE metrics. SLOs defined. Distributed tracing. Alert-to-runbook links. | Medium — AI can correlate, not infer |
| **L4** | Predictive | Baseline-aware anomaly detection. Change correlation. Burn rate alerting. Topology-aware. | High — AI can suggest root cause |
| **L5** | Autonomous | AI-driven triage. Auto-remediation for known failure classes. Self-healing pipelines. Audience-specific signal delivery. Feedback loops. | Full — AI acts, humans govern |

**Where most enterprise platforms are**: L2–L3.
**Where Principal/Director-level SREs are expected to drive their platforms**: L4 moving to L5.
**Your DND platform (T-Mobile context)**: L3 with strong Splunk maturity. The gap to L4 is change correlation, topology-aware alerting, and SLO burn rate automation.

### Decision Making and Autonomous Remediation — When to Automate vs. Escalate

Not every signal should trigger human action, and not every action is safe to automate. The decision boundary is defined by **blast radius** and **reversibility**.

```
Signal received by AI/ML layer
           │
           ▼
    Is this pattern known?
    (seen before, labeled in feedback loop)
           │
    ┌──────┴──────┐
   YES            NO
    │              │
    ▼              ▼
Is remediation   Page Tier 2
reversible and   (on-call SRE)
bounded?         with full context
    │
  ┌─┴────┐
 YES     NO
  │       │
  ▼       ▼
Auto-    Recommend action
remediate + await human
+ notify  confirmation
  Tier 2  before executing
```

| Scenario | Blast Radius | Reversible | Action |
|---------|-------------|-----------|--------|
| Queue depth → scale consumer pods | Low — adds capacity | Yes — HPA can scale back | Autonomous (KEDA trigger) |
| High error rate → circuit breaker | Medium — stops traffic to one service | Yes — CB auto-closes | Autonomous (circuit breaker policy) |
| Canary error rate > threshold → rollback | Medium — reverts one deployment | Yes — previous version | Autonomous (Argo Rollouts policy) |
| DB connection pool exhausted | High — affects all services on that DB | No — requires investigation | Page Tier 2 + recommended runbook |
| Data corruption detected | Critical — irreversible data loss risk | No | Page Tier 2 + Tier 4 + freeze writes |
| Novel anomaly pattern — no prior label | Unknown | Unknown | Page Tier 2 + log for AI feedback |

**The governance rule at Principal/Director level**: Define the automation boundary explicitly, in writing, reviewed by leadership. Document what the system will do autonomously, what requires human approval, and what always escalates. This is your **Runbook Automation Policy** — it is a reliability and risk artifact, not just an engineering one.
