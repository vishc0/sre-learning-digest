## Why This Matters (Interview Relevance + Business Impact)

Every Director/VP SRE interview in 2025-2026 includes an AI-augmented operations question. RAG (Retrieval-Augmented Generation) is the production pattern behind every "chat with your runbooks" tool — PagerDuty Copilot, Datadog Bits AI, Grafana Assistant. Building one from scratch proves you understand the architecture, not just the product.

Business impact: A RAG system on SRE documentation reduces MTTR by giving on-call engineers instant, context-aware answers instead of searching 400-page wikis at 2am. At T-Mobile scale (25M msg/day notification platform), that is measurable dollar value per incident.

Interview payoff: You can say "I built a production RAG system on AWS Bedrock, measured retrieval quality with RAGAS metrics, and added metadata filtering to scope answers by incident domain." That sentence passes the resume screen at AWS, Databricks, and Snowflake.

---

## Analogy (Middleware First)

Think of RAG like your RabbitMQ dead-letter queue investigation workflow.

When a message fails, you do not guess — you query the DLQ, read the payload, then reason about what went wrong. RAG does the same thing for LLMs: instead of the model guessing from training data, it first queries a knowledge store (your SRE docs), retrieves the most relevant chunks, then reasons on top of real content.

The S3 bucket is your message store. The Bedrock Knowledge Base is your index (like a Splunk index over structured logs). The vector search is your `sourcetype=rabbitmq_dlq | search error_code=*` — it finds semantically similar content, not just keyword matches. Claude is your L3 engineer who reads the retrieved context and synthesizes the answer.

Without RAG: Claude answers from 2024 training data, hallucinating your internal SLO thresholds.
With RAG: Claude reads your actual `slo-definitions.md` and quotes the exact burn rate formula you wrote.

## 30-Second Version

RAG = retrieve relevant docs first, then ask the LLM to answer using only those docs. You store your SRE markdown files in S3, Bedrock chunks and indexes them as vectors, your Python script queries the index with a question, Bedrock returns relevant chunks plus a grounded Claude answer. Cost for this lab: approximately $2.

---

## 2-Minute Version

Three moving parts:

1. **Knowledge Base (KBs)**: AWS Bedrock ingests your markdown files from S3, splits them into ~300-token chunks, converts each chunk to a vector embedding (a list of numbers representing semantic meaning), and stores those vectors in a managed OpenSearch index. You never touch the vector DB directly.

2. **Retrieval**: When you ask a question, Bedrock converts your question to a vector, finds the top-K most similar document chunks (cosine similarity), and returns them as context.

3. **Augmented Generation**: Claude (via Bedrock) receives your question plus the retrieved chunks and answers using only that grounded context. If the answer is not in the docs, it says so instead of hallucinating.

The Python scripts you write are thin wrappers: `s3_upload.py` puts your docs in the right bucket prefix, `kb_query.py` calls `retrieve_and_generate()` with your question and Knowledge Base ID. Metadata filtering narrows retrieval to specific weeks or domains before the similarity search runs — like adding `index=` scope in Splunk before the search executes.

---

## 5-Minute Version (Tradeoffs, Failure Modes, Design Decisions)

**Design decision 1 — Chunk size**: Bedrock default is 300 tokens with 20% overlap. Smaller chunks give precise retrieval but lose cross-paragraph context. Larger chunks preserve context but reduce precision. For operational runbooks with step-by-step procedures, 500-token chunks with 10% overlap tend to work better. You will observe this difference in Module 3.

**Design decision 2 — Embedding model**: Bedrock offers `amazon.titan-embed-text-v2` (1536 dimensions, better for technical content) and `cohere.embed-english-v3`. Titan V2 is the right choice for SRE documentation — it handles technical terminology and acronyms better.

**Design decision 3 — Retrieval strategy**: Default is "semantic search" (pure vector similarity). You can add hybrid search (semantic + keyword BM25) for better results on queries that contain specific version numbers or error codes — exactly the kind of queries on-call engineers ask at 2am.

**Failure mode 1 — Stale index**: Your Knowledge Base is only as current as the last sync. If you update a runbook in S3 but forget to re-sync the Knowledge Base, Claude answers from outdated content. Mitigation: automate sync on S3 PutObject events via EventBridge.

**Failure mode 2 — Hallucination at retrieval boundary**: If no document chunk scores above the relevance threshold, Bedrock may still return an answer synthesized from Claude's training data. Add `citations` validation in your query script — if `citations` list is empty, surface a warning to the user.

**Failure mode 3 — Chunking breaks procedural steps**: A 5-step runbook procedure split across two chunks loses causality. Step 3 might be retrieved without step 2 context. Mitigation: use hierarchical chunking (Bedrock option) or add structured YAML front matter to your markdown files so each procedure is self-contained in one chunk.

**Director lens**: At T-Mobile scale, you would not use Bedrock Knowledge Base directly — you would build an abstraction layer (LangChain or custom) that routes queries to multiple knowledge bases (incidents, runbooks, architecture decisions) and merges results before grounding the LLM. The Bedrock KB you build today is the foundational module of that architecture.

## Comprehension Check (Pass 2/3 to Unlock Lab)

**Q1**: Your RAG system returns an answer with zero citations. What does that mean operationally, and what should your query script do in that case?

**Q2**: A new engineer asks "why not just put all the runbooks in the LLM context window instead of building a Knowledge Base?" How do you answer — what are the two main production reasons RAG wins over full-context stuffing?

**Q3**: You add a new postmortem markdown file to S3 at 3pm. An on-call engineer queries the Knowledge Base at 3:05pm. They get an answer that does not include the new postmortem. What is the most likely root cause and what is the architectural fix?

---

## ANSWER KEY — Comprehension Check

**A1**: Zero citations means no document chunk scored above the retrieval relevance threshold — Bedrock found nothing semantically close enough in your knowledge base. The answer is potentially hallucinated from Claude's training data, not grounded in your docs. Your query script should check `len(response['citations']) == 0` and either return a warning ("No source documents found — answer may not reflect your environment") or refuse to answer entirely. Operationally, this is like a DLQ query returning zero messages — the absence of data is itself signal.

**A2**: Two reasons RAG wins at production scale: (1) **Context window cost** — stuffing 400 runbooks into every query costs 100K+ input tokens per call at $0.003/1K tokens = $0.30 per question. Multiplied by 500 on-call queries/month that is $150/month for context alone, vs. $0.02/month with RAG retrieving only the 3 relevant chunks. (2) **Staleness and maintainability** — a static context payload goes stale the moment a runbook is updated. RAG retrieval always serves the current indexed version. The context window approach also hits model limits for large knowledge bases.

**A3**: Root cause: the Knowledge Base data source sync has not run since the file was uploaded. Bedrock Knowledge Bases do not auto-sync on S3 upload — sync is either manual or scheduled. The engineer is querying a stale index. Architectural fix: trigger a Knowledge Base sync via EventBridge rule on S3 `ObjectCreated` events for the docs prefix, calling the `StartIngestionJob` Bedrock API automatically. Add a CloudWatch alarm if sync latency exceeds your SLA.

# SLO Definitions and Error Budget Math

## Availability SLO
Availability = (total_minutes - downtime_minutes) / total_minutes * 100

Our notification platform SLO: 99.95% monthly availability.
Error budget = (1 - 0.9995) * 43800 minutes/month = 21.9 minutes/month.

## Latency SLO
P99 latency must remain below 500ms for message delivery.
Measured via Splunk query: `index=platform sourcetype=delivery_metrics | stats p99(latency_ms)`

## Burn Rate
Burn rate = current error rate / acceptable error rate.
1x burn = consuming error budget at exactly the SLO rate (sustainable).
14.4x burn = consuming 1 month of error budget in 2 hours (page immediately).

## Multi-Window Burn Rate Alert Thresholds
- 14.4x burn over 1 hour AND 6x burn over 6 hours → CRITICAL, page on-call
- 6x burn over 6 hours AND 3x burn over 1 day → WARNING, notify team lead
- 1x burn over 3 days → INFO, weekly review

## Error Budget Policy
When error budget is below 10%: freeze non-critical changes, root cause analysis required.
When error budget is below 5%: escalate to Director, customer impact assessment.
When error budget is exhausted: only reliability-improving changes allowed until next window.

# AWS IAM Least Privilege Patterns

## IRSA (IAM Roles for Service Accounts)
Binds a Kubernetes service account to an AWS IAM role without long-lived credentials.
Every pod using the service account inherits the IAM role permissions automatically.

## IRSA Setup Pattern
1. Create IAM OIDC provider for EKS cluster
2. Create IAM role with trust policy scoped to specific service account
3. Annotate Kubernetes service account with IAM role ARN
4. Pods using that service account get temporary credentials via projected token

## Trust Policy for IRSA
```json
{
  "Effect": "Allow",
  "Principal": {
    "Federated": "arn:aws:iam::ACCOUNT:oidc-provider/OIDC_ENDPOINT"
  },
  "Action": "sts:AssumeRoleWithWebIdentity",
  "Condition": {
    "StringEquals": {
      "OIDC_ENDPOINT:sub": "system:serviceaccount:NAMESPACE:SERVICE_ACCOUNT_NAME"
    }
  }
}
```

## Least Privilege Validation
Use IAM Access Analyzer to generate least-privilege policies from CloudTrail activity.
Run for 7 days, collect all API calls, generate policy, replace overly-broad policies.

## Common SRE Mistakes
- Using node instance profile for pod-level S3 access (all pods get same permissions)
- AdministratorAccess on CI/CD service accounts (blast radius = entire account)
- Not rotating access keys for long-running automation (use IRSA/EC2 instance profiles instead)

# Chaos Engineering for SRE

## Definition
Chaos engineering is the practice of deliberately injecting failures into a system to discover weaknesses before they cause production incidents.

## Hypothesis-Driven Experiments
Always define before running:
- Steady state: what does normal look like? (P99 latency < 500ms, error rate < 0.1%)

Managed chaos engineering service — no custom scripts needed.

## Chaos Maturity Levels
1. Run experiments in dev/staging only
2. Run in production during business hours with human oversight
3. Run in production automated with automatic abort conditions

#### Step 2.1 — Write kb_query.py

cat > ~/sre-rag-lab/kb_query.py << 'PYEOF'
#!/usr/bin/env python3
"""
kb_query.py — Query the SRE Knowledge Base using Bedrock RAG.

Architecture:
  Your question → Bedrock retrieves top-K relevant chunks from OpenSearch
  → Chunks passed as context to Claude → Claude answers grounded in your docs

WHY retrieve_and_generate() vs retrieve() alone:
  retrieve() returns raw chunks — you would need to write the prompt yourself.
  retrieve_and_generate() handles the full RAG loop: retrieve + prompt + generate.
  Use retrieve() when you want to inspect what was retrieved before generating.
"""

import boto3
import json
import os
from datetime import datetime

# ── Configuration ──────────────────────────────────────────────────────────────
KB_ID = os.environ.get("KB_ID", "REPLACE_WITH_YOUR_KB_ID")  # from kb_id.txt
MODEL_ARN = "arn:aws:bedrock:us-east-1::foundation-model/anthropic.claude-3-sonnet-20240229-v1:0"
REGION = "us-east-1"

# ── Bedrock Clients ────────────────────────────────────────────────────────────
# Two separate clients — Bedrock has two API surfaces:
# bedrock-agent-runtime: retrieval + RAG queries (runtime operations)
# bedrock-runtime: direct model invocation
bedrock_agent_runtime = boto3.client("bedrock-agent-runtime", region_name=REGION)

def query_knowledge_base(question: str, num_results: int = 3) -> dict:
    """
    Query the Knowledge Base with retrieve_and_generate.
    
    Returns dict with:
      - answer: Claude's synthesized response
      - citations: list of source document references
      - raw_response: full API response for debugging
    
    WHY num_results=3: retrieve top 3 chunks by default.
    More chunks = more context for Claude but also higher token cost.
    For operational queries, 3-5 chunks is usually optimal.
    """
    response = bedrock_agent_runtime.retrieve_and_generate(
        input={"text": question},
        retrieveAndGenerateConfiguration={
            "type": "KNOWLEDGE_BASE",
            "knowledgeBaseConfiguration": {
                "knowledgeBaseId": KB_ID,
                "modelArn": MODEL_ARN,
                "retrievalConfiguration": {
                    "vectorSearchConfiguration": {
                        "numberOfResults": num_results
                    }
                }
            }
        }
    )
    
    # Extract answer text
    answer = response["output"]["text"]
    
    # Extract citations — CRITICAL for validating answer quality
    # If citations is empty, the answer may be hallucinated
    citations = []
    for citation in response.get("citations", []):
        for ref in citation.get("retrievedReferences", []):
            source = ref.get("location", {}).get("s3Location", {}).get("uri", "unknown")
            excerpt = ref.get("content", {}).get("text", "")[:200]  # first 200 chars
            citations.append({
                "source": source,
                "excerpt": excerpt
            })
    
    return {
        "answer": answer,
        "citations": citations,
        "citation_count": len(citations),
        "raw_response": response
    }

def evaluate_answer(question: str, result: dict) -> dict:
    """
    Basic answer quality evaluation.
    
    Metrics:
    - has_citations: bool — answer is grounded in retrieved docs
    - citation_count: int — how many source chunks were used
    - answer_length: int — longer answers tend to be more complete
    - hallucination_risk: 'LOW' if citations found, 'HIGH' if not
    
    WHY: In production you would use RAGAS library for formal evaluation.
    This is a lightweight version for the lab — teaches the same concepts.
    """
    return {
        "question": question,
        "answer": result["answer"],
        "citation_count": result["citation_count"],
        "has_citations": result["citation_count"] > 0,
        "answer_length_chars": len(result["answer"]),
        "hallucination_risk": "LOW" if result["citation_count"] > 0 else "HIGH",
        "sources": [c["source"].split("/")[-1] for c in result["citations"]]  # just filenames
    }

def main():
    print(f"SRE Knowledge Base Query Session")
    print(f"  KB ID: {KB_ID}")
    print(f"  Model: claude-3-sonnet")
    print(f"  Time : {datetime.utcnow().isoformat()}Z")
    print("=" * 70)
    
    # 10 operational questions — deliberately varied by domain
    questions = [
        # Observability
        "What is the error budget for our notification platform and when should we freeze changes?",
        "How do we calculate burn rate and what are the multi-window alert thresholds?",
        # Incident Response
        "What are the triage steps when the RabbitMQ DLQ depth exceeds 1000 messages?",
        "When does a DLQ incident escalate to Sev1?",
        # Kubernetes
        "What causes HPA thrashing and how do we fix it?",
        "How do we set up HPA for queue-depth based scaling?",
        # Terraform
        "How do we handle a stuck Terraform state lock?",
        # DevSecOps
        "Which SAST tool should we use for Python services and what severity should fail the pipeline?",
        # AWS
        "What is the risk of using the node instance profile for pod-level S3 access?",
        # Chaos Engineering
        "What steady state metrics should we define before running a chaos experiment?"
    ]
    
    evaluations = []
    
    for i, question in enumerate(questions, 1):
        print(f"\nQ{i}: {question}")
        print("-" * 60)
        
        try:
            result = query_knowledge_base(question)
            eval_result = evaluate_answer(question, result)
            evaluations.append(eval_result)
            
            # Print answer (truncated for readability)
            answer_preview = result["answer"][:400] + "..." if len(result["answer"]) > 400 else result["answer"]
            print(f"Answer: {answer_preview}")
            print(f"Sources ({eval_result['citation_count']}): {', '.join(eval_result['sources'])}")
            print(f"Hallucination risk: {eval_result['hallucination_risk']}")
            
        except Exception as e:
            print(f"ERROR: {e}")
            evaluations.append({
                "question": question,
                "answer": f"ERROR: {e}",
                "citation_count": 0,
                "has_citations": False,
                "answer_length_chars": 0,
                "hallucination_risk": "ERROR",
                "sources": []
            })
    
    # ── Summary statistics ─────────────────────────────────────────────────────
    print("\n" + "=" * 70)
    print("EVALUATION SUMMARY")
    print("=" * 70)
    
    total = len(evaluations)
    with_citations = sum(1 for e in evaluations if e["has_citations"])
    avg_length = sum(e["answer_length_chars"] for e in evaluations) / total
    
    print(f"Total questions     : {total}")
    print(f"Answers with sources: {with_citations}/{total} ({with_citations/total*100:.0f}%)")
    print(f"Avg answer length   : {avg_length:.0f} chars")
    print(f"High-risk answers   : {total - with_citations} (no citations)")
    
    # ── Save evaluation results ────────────────────────────────────────────────
    output_path = os.path.expanduser("~/sre-rag-lab/evaluation_results.json")
    with open(output_path, "w") as f:
        json.dump({
            "session": datetime.utcnow().isoformat(),
            "kb_id": KB_ID,
            "summary": {
                "total_questions": total,
                "answers_with_citations": with_citations,
                "citation_rate_pct": round(with_citations/total*100, 1),
                "avg_answer_length_chars": round(avg_length)
            },
            "evaluations": evaluations
        }, f, indent=2)
    
    print(f"\nFull results saved to: {output_path}")
    print("Use these results as input to Module 3 analysis.")

if __name__ == "__main__":
    main()

WHY metadata filtering matters:
If an on-call engineer asks "how do we handle a DLQ spike?", an unfiltered
query might retrieve a Terraform chunk about S3 buckets (false positive match
on "bucket"). A domain=incident-response filter ensures only incident runbooks
are searched — same as adding 'index=sre_incidents' to your Splunk query.

The metadata keys (week, domain, doc_type) come from the YAML front matter
in each markdown file. Bedrock extracts these automatically during ingestion.
"""

import boto3
import json
import os
from datetime import datetime

KB_ID = os.environ.get("KB_ID", "REPLACE_WITH_YOUR_KB_ID")
MODEL_ARN = "arn:aws:bedrock:us-east-1::foundation-model/anthropic.claude-3-sonnet-20240229-v1:0"
REGION = "us-east-1"

bedrock_agent_runtime = boto3.client("bedrock-agent-runtime", region_name=REGION)

def query_with_filter(question: str, filters: dict = None, num_results: int = 3) -> dict:
    """
    Query with optional metadata filter.
    
    filter structure example — single field:
      {"equals": {"key": "domain", "value": "incident-response"}}
    
    filter structure example — AND logic:
      {"andAll": [
        {"equals": {"key": "domain", "value": "observability"}},
        {"equals": {"key": "week", "value": "5"}}
      ]}
    
    WHY: Metadata filters run BEFORE vector similarity search, narrowing
    the candidate pool. This is equivalent to index=X in Splunk — faster
    and more precise than post-filtering after retrieval.
    """
    retrieval_config = {
        "vectorSearchConfiguration": {
            "numberOfResults": num_results
        }
    }
    
    # Add filter only if provided
    if filters:
        retrieval_config["vectorSearchConfiguration"]["filter"] = filters
    
    response = bedrock_agent_runtime.retrieve_and_generate(
        input={"text": question},
        retrieveAndGenerateConfiguration={
            "type": "KNOWLEDGE_BASE",
            "knowledgeBaseConfiguration": {
                "knowledgeBaseId": KB_ID,
                "modelArn": MODEL_ARN,
                "retrievalConfiguration": retrieval_config
            }
        }
    )
    
    answer = response["output"]["text"]
    citations = []
    for citation in response.get("citations", []):
        for ref in citation.get("retrievedReferences", []):
            source = ref.get("location", {}).get("s3Location", {}).get("uri", "unknown")
            citations.append(source.split("/")[-1])
    
    return {
        "answer": answer[:500],  # truncated for comparison display
        "sources": citations,
        "citation_count": len(citations)
    }

def run_comparison(question: str, filter_def: dict, filter_description: str) -> None:
    """Run the same question with and without a filter, print side-by-side comparison."""
    print(f"\nQUESTION: {question}")
    print(f"FILTER  : {filter_description}")
    print("-" * 70)
    
    # Without filter
    print("WITHOUT FILTER:")
    unfiltered = query_with_filter(question)
    print(f"  Sources : {', '.join(unfiltered['sources']) or 'none'}")
    print(f"  Citations: {unfiltered['citation_count']}")
    print(f"  Answer  : {unfiltered['answer'][:300]}...")
    
    print()
    
    # With filter
    print(f"WITH FILTER ({filter_description}):")
    try:
        filtered = query_with_filter(question, filters=filter_def)
        print(f"  Sources : {', '.join(filtered['sources']) or 'none (filter excluded all docs)'}")
        print(f"  Citations: {filtered['citation_count']}")
        print(f"  Answer  : {filtered['answer'][:300]}...")
    except Exception as e:
        print(f"  ERROR: {e}")
    
    print("-" * 70)

def main():
    print("Metadata Filter Comparison")
    print(f"KB ID: {KB_ID}")
    print(f"Time : {datetime.utcnow().isoformat()}Z")
    
    # ── Comparison 1: Domain filter — observability only ──────────────────────
    run_comparison(

        filter_def={
            "equals": {"key": "domain", "value": "observability"}
        },

    )
    
    # ── Comparison 2: Domain filter — incident response only ──────────────────
    run_comparison(
        question="What are the escalation procedures for a production issue?",
        filter_def={
            "equals": {"key": "domain", "value": "incident-response"}
        },
        filter_description="domain=incident-response"
    )
    
    # ── Comparison 3: Week filter — show only Week 5 content ──────────────────
    run_comparison(
        question="How do I instrument my service for production observability?",

            "equals": {"key": "week", "value": "5"}
        },

    )
    
    # ── Comparison 4: Doc type filter — reference docs only, not runbooks ─────

            "equals": {"key": "doc_type", "value": "reference"}
        },

    )
    
    # ── Comparison 5: AND filter — week 3 kubernetes reference docs ───────────
    run_comparison(

            "andAll": [
                {"equals": {"key": "domain", "value": "kubernetes"}},
                {"equals": {"key": "doc_type", "value": "reference"}}
            ]
        },
        filter_description="domain=kubernetes AND doc_type=reference"
    )
    
    print("\n" + "=" * 70)
    print("KEY TAKEAWAYS FOR EVALUATION_RESULTS.MD")
    print("=" * 70)
    print("""

## Lab ANSWER KEY

### Diagnostic: "Ingestion job FAILED"
**Root cause 90%**: IAM role does not have s3:GetObject on the bucket. Check the inline policy ARN matches your exact bucket name. Re-run `aws iam get-role-policy --role-name BedrockKBRole-SRELab --policy-name S3ReadSREDocs` and verify the bucket ARN.

**Root cause 10%**: Model access not granted. Go to AWS Console > Bedrock > Model access, confirm Titan Embed V2 shows "Access granted."

### Diagnostic: "citation_count = 0 for all answers"
**Root cause**: Knowledge Base status is not ACTIVE, or ingestion job completed with errors. Run `aws bedrock-agent get-knowledge-base --knowledge-base-id $KB_ID` and check status. Run ingestion job again and verify `numberOfDocumentsIndexed: 10`.

### Diagnostic: "metadata filter returns no results"
**Root cause**: Bedrock extracts YAML front matter as metadata but the key names must match exactly. Your front matter uses `domain: incident-response` with a hyphen — the filter must use `"value": "incident-response"` not `"value": "incident_response"`. Check for typos using `retrieve()` (not retrieve_and_generate) and inspect raw metadata fields.

### Diagnostic: "boto3 AccessDeniedException on retrieve_and_generate"
**Root cause**: Your training-admin IAM user does not have `bedrock:RetrieveAndGenerate` permission. Add inline policy to training-admin user: `{"Effect": "Allow", "Action": ["bedrock:RetrieveAndGenerate", "bedrock:Retrieve", "bedrock:InvokeModel"], "Resource": "*"}`.

### Diagnostic: "Knowledge Base creation fails with StorageConfiguration error"
**Root cause**: The simplified `create-knowledge-base` form without explicit storage configuration works in most cases. If it fails, Bedrock may require you to create the OpenSearch Serverless collection manually first — use the AWS Console for this step (Bedrock > Knowledge Bases > Create > follow wizard).

## Tradeoffs and Failure Modes

**Tradeoff 1 — Managed KB vs Custom RAG Stack**:
Bedrock KB is 80% less code to build and operate. The cost is flexibility: chunking options are limited (fixed-size, hierarchical, semantic) versus the full LangChain ecosystem. For a team that owns both the knowledge base and the query interface, Bedrock KB wins on operational overhead. For teams needing custom chunking logic or multi-modal documents (PDFs, diagrams), LangChain + Pinecone gives full control.

**Tradeoff 2 — Retrieval precision vs recall**:
Metadata filters increase precision (fewer wrong-domain results) but decrease recall (might exclude a relevant document that has the wrong metadata tag). For on-call scenarios, precision wins — a confident wrong answer is more dangerous than "I found nothing relevant, please search manually." Production implementation: filter first, fall back to unfiltered with lower confidence signal if citation_count is zero.

**Failure mode — Documentation becomes a first-class SRE artifact**:
This is the underestimated organizational change RAG introduces. If your runbooks are poorly written, inconsistently structured, or out of date, the RAG system amplifies those problems — it gives the bad runbook confident delivery. The Director/VP SRE role shifts: you now govern documentation quality as a reliability metric. Document freshness, structured metadata standards, and runbook review cadence become platform engineering concerns, not just good practice.
