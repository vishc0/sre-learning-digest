## Analogy (Middleware First)

Think of **Azure OpenAI** as a **managed RabbitMQ cluster** — but instead of message brokering, it brokers AI inference requests.

- **OpenAI public API** = running your own open-source RabbitMQ instance in a shared datacenter. You get the service, but your data transits public internet, compliance is your problem, and you share infrastructure with anyone.
- **Azure OpenAI** = a dedicated RabbitMQ cluster inside your VNET, deployed in your compliance boundary, with Microsoft's enterprise SLA and data residency guarantees. Same underlying models, radically different operational posture.

**Managed Identity** = your RabbitMQ client authenticating via Kerberos/AD service account instead of a hardcoded password in the config file. The platform rotates credentials; your app never holds a secret.

**AKS GPU node pools** = adding specialized queue workers (high-memory, GPU-optimized) that only spin up when you have GPU-bound inference jobs — like adding dedicated high-throughput consumers for a specific message type, auto-scaled.

## 2-Minute Version

**Azure OpenAI vs OpenAI API** — same models (GPT-4o, GPT-4, DALL-E, Whisper), different operational contract:
- Data residency: your prompts and completions never leave your chosen Azure region
- VNET integration: endpoint is private, reachable only from inside your network
- Compliance: SOC 2, HIPAA, PCI DSS, FedRAMP — pre-certified
- Abuse monitoring: Azure applies content policy but you can configure custom filtering

**Managed Identity auth** — two types:
- System-assigned: identity tied to a specific Azure resource (VM, AKS pod). Deleted when resource is deleted.
- User-assigned: standalone identity you attach to multiple resources. Better for shared services.
- Your app code uses `DefaultAzureCredential` — it automatically uses whichever identity is available (Managed Identity in production, your developer login locally). Zero secrets in code.

**AKS GPU node pools** — you create a separate node pool in AKS using NC-series VM SKUs. Kubernetes scheduler routes GPU-requesting pods to these nodes using resource requests (`nvidia.com/gpu: 1`). Node pools can scale to zero when idle — you only pay for GPU time when workloads run.

**Azure AI Foundry** — Microsoft's hub for managing AI projects. Think of it as an IDP (Internal Developer Platform) for AI: model catalog, deployment management, evaluation pipelines, fine-tuning workflows, and observability in one place. Replaced the older "Azure Machine Learning Studio" and "Azure OpenAI Studio" interfaces.

**Content Safety** — separate Azure service that classifies content across four harm categories (hate, violence, sexual, self-harm) with severity levels 0-6. You can require human review above a threshold. Can be applied to both input (prompt) and output (completion).

## 5-Minute Version (Tradeoffs, Failure Modes, Design Decisions)

### Data Plane Architecture

```
Your App (AKS pod)
    │
    ▼ [Managed Identity token — no API key]
Azure VNET
    │
    ▼ [Private Endpoint]
Azure OpenAI Service (your instance)
    │
    ├── Deployment: gpt-4o-prod (quota: 100K TPM)
    ├── Deployment: gpt-4o-dev  (quota: 10K TPM)
    └── Deployment: text-embedding-ada-002
    │
    ▼ [Content Safety filter — pre/post]
Response → Your App
```

### Key Design Decisions

**1. Regional deployment strategy**
Azure OpenAI is not available in all regions, and model availability varies by region. In 2025-2026, East US and Sweden Central have the broadest model availability. For T-Mobile (US regulated telecom), East US gives you data residency + lowest latency. Multi-region active-active is possible but requires quota approval per region — plan 4-6 weeks lead time for quota increases.

**2. Quota management (TPM/RPM)**
TPM = Tokens Per Minute, RPM = Requests Per Minute. Default quotas are conservative (e.g., 30K TPM for GPT-4o). You request increases through Azure Portal — not instant, 1-3 business days. SRE implication: quota exhaustion looks like 429 errors. You need retry logic with exponential backoff AND quota monitoring dashboards. This is your RabbitMQ consumer throttle equivalent.

**3. Keyless auth vs API key**
Always prefer Managed Identity in production. API keys are a flat secret — if leaked, full access. Managed Identity is tied to a specific identity with RBAC scope. The role you grant is `Cognitive Services OpenAI User` (inference only) or `Cognitive Services OpenAI Contributor` (can also manage deployments). Principle of least privilege: your inference pods get `User`, your deployment pipeline gets `Contributor`.

**4. Private endpoint vs service endpoint**
Private endpoint = a private IP inside your VNET that routes to Azure OpenAI. The service has no public internet exposure. Service endpoint = your VNET traffic takes an optimized route to the public endpoint, but the endpoint itself remains public. For compliance (HIPAA, FedRAMP), you need private endpoint. For lower-sensitivity workloads, service endpoint is simpler.

**5. Content Safety — latency tradeoff**
Content Safety adds ~50-100ms per request for synchronous filtering. For a 25M msg/day notification platform this is relevant — if AI is in the critical path for notification generation, that latency compounds. Mitigation: async safety scanning for non-critical content, synchronous only for user-facing outputs. You can also batch-evaluate content asynchronously.

**6. AKS GPU node pools — cost shock pattern**
GPU VMs (NC-series) cost 10-50x CPU VMs. The most common failure mode is a node pool that doesn't scale to zero properly, leaving GPU nodes running idle overnight. Always configure:
- `--scale-down-delay-after-add` and `--scale-down-unneeded-time` on cluster autoscaler
- Pod disruption budgets so autoscaler can actually evict pods
- Budget alerts at $50/day threshold specifically for GPU node pools

**7. Azure AI Foundry vs raw Azure OpenAI API**
Foundry adds a project/hub abstraction layer. For team scenarios (multiple developers, model evaluations, prompt versioning), Foundry is worth the abstraction. For a single-team SRE automation use case, raw Azure OpenAI REST/SDK is simpler and has less moving parts. Rule of thumb: if you have a dedicated ML/AI team, use Foundry. If SRE is consuming AI as infrastructure, use the SDK directly.

### Failure Modes

| Failure Mode | Symptom | Mitigation |
|---|---|---|
| Quota exhaustion | 429 TooManyRequests | Retry with exponential backoff; monitor TPM consumption; request quota increase proactively |
| Managed Identity token expiry | 401 Unauthorized after hours of uptime | `DefaultAzureCredential` handles refresh automatically — if you see this, check if pod identity is configured correctly |
| Regional outage | All requests fail to one endpoint | Deploy to two regions, use Azure Front Door or application-level failover |
| Content Safety false positives | Legitimate requests blocked | Tune thresholds per use case; implement human review queue for borderline cases |
| GPU node pool won't scale to zero | Runaway GPU costs | Check autoscaler config; verify pods are actually terminating, not stuck in Pending |
| Cold start latency | First request after idle takes 30+ seconds | Keep minimum replicas > 0 for production; use KEDA for demand-based scaling with warm pool |

## Director/VP Lens

**What a Director governs vs what an IC implements:**

**IC implements:**
- `DefaultAzureCredential` in Python code
- Node pool YAML with GPU resource requests
- Content Safety threshold configuration
- Monitoring dashboards for TPM/RPM consumption

**Director governs:**
- **Quota strategy**: Which teams get which quota allocations; how quota is partitioned across prod/dev/test deployments. This is a chargeback conversation.
- **Data classification policy**: Which workloads are allowed to send what data to Azure OpenAI. PII handling, prompt logging, response retention policies.
- **Multi-region decision**: When does the business require active-active AI availability? What's the SLA commitment to internal customers?
- **Build vs buy on guardrails**: Do we build custom content filtering, use Azure Content Safety, or use a third-party (Guardrails AI, LlamaGuard)? This is a make/buy decision with compliance, cost, and vendor lock-in dimensions.
- **AI incident response**: What is the runbook when AI output causes a customer-facing incident? Who owns the postmortem? How does this differ from a service outage postmortem?
- **AI cost governance**: GPU nodes are expensive. Director sets the budget envelope and the FinOps review cadence. At T-Mobile scale, unmanaged AI spend can hit $50K/month quickly.

**The AI changes this domain framing:**
- AI infrastructure SRE is not just keeping services up — it includes **output quality** as a reliability dimension. A model hallucinating is a reliability failure even if the API returns 200.
- Observability needs to include semantic metrics (answer quality, refusal rate, latency by prompt complexity), not just infrastructure metrics.
- On-call for AI systems requires domain knowledge the traditional SRE rotation may not have — plan for embedded ML engineer in the on-call rotation.

**Q1.** Your team's Python service running on AKS needs to call Azure OpenAI. A junior engineer suggests putting the Azure OpenAI API key in a Kubernetes Secret. What is wrong with this approach, and what should you use instead?

**Q2.** You're seeing 429 errors from Azure OpenAI during peak load. The service health dashboard shows no Azure outage. What are the two most likely causes, and how do you verify which one it is?

**Q3.** A Product Manager asks: "Why are we using Azure OpenAI instead of just calling OpenAI's API directly? It's the same model and Azure is more expensive." Give a 90-second Director-level answer that addresses compliance, security, and operational posture.

---

## ANSWER KEY — Comprehension Check

**A1.** Kubernetes Secrets are base64-encoded, not encrypted by default — any pod in the namespace can read them, and they appear in etcd in near-plaintext unless etcd encryption is configured. More critically, the API key is a flat credential: leaked = full access, no audit trail tied to identity, no automatic rotation. **Use Managed Identity instead**: assign a User-assigned or System-assigned Managed Identity to the AKS pod (via Workload Identity or Pod Identity), grant that identity the `Cognitive Services OpenAI User` RBAC role on the Azure OpenAI resource, and use `DefaultAzureCredential` in Python. The platform issues and rotates the token — your code never handles a secret.

**A2.** Two most likely causes:
- **(a) TPM/RPM quota exhaustion** — you've hit your allocated Tokens Per Minute or Requests Per Minute limit. Verify by checking the Azure OpenAI resource metrics in Portal → "Provisioned Managed Utilization" or the `azure_openai_requests` metric. If you see utilization at 100%, request a quota increase.
- **(b) Retry storm amplification** — a transient spike caused 429s, your retry logic has no jitter/backoff, all retries hit simultaneously and make the 429s worse. Verify by checking the request rate pattern — you'll see a sawtooth spike pattern rather than a steady ramp. Fix: exponential backoff with jitter (`tenacity` library in Python).

**A3.** "Three reasons we choose Azure OpenAI over the public API: First, **data residency and compliance** — our prompts and completions never leave East US region. With OpenAI's public API, data transits their infrastructure under their data processing terms. For a telecom handling customer data, that's a compliance and legal review that could block the project. Second, **network security** — Azure OpenAI sits inside our VNET behind a private endpoint. No public internet exposure, no firewall exceptions needed, same security posture as our other internal services. Third, **operational posture** — we get Azure Monitor integration, the same SLA structure as our other Azure services, and Managed Identity auth which fits our existing IAM model. The marginal cost difference is real but is justified by the audit, compliance, and security simplification."

### Step 6 — Python Script: Call GPT-4o with Keyless Auth

```bash
# Create the Python script
mkdir -p ~/azure-ai-lab
cat > ~/azure-ai-lab/aoai_keyless.py << 'PYTHON_SCRIPT'
"""
Azure OpenAI Keyless Auth Demo
Authenticates using DefaultAzureCredential (no API keys)
Calls GPT-4o deployment with an SRE-relevant prompt
"""

import os
from openai import AzureOpenAI
from azure.identity import DefaultAzureCredential, get_bearer_token_provider

# ─────────────────────────────────────────────────
# CONFIGURATION — replace with your actual values
# ─────────────────────────────────────────────────
AOAI_ENDPOINT = os.environ.get("AOAI_ENDPOINT", "https://YOUR_RESOURCE.openai.azure.com/")
DEPLOYMENT_NAME = "gpt-4o-lab"  # the name you gave to your deployment, not the model name
API_VERSION = "2024-10-21"       # Azure OpenAI API version (update if needed)

# ─────────────────────────────────────────────────
# KEYLESS AUTH — this is the core pattern
# DefaultAzureCredential tries, in order:
#   1. Env vars (AZURE_CLIENT_ID, etc.)
#   2. Workload Identity (AKS pods with Managed Identity)
#   3. Managed Identity (Azure VMs, App Service)
#   4. Azure CLI session (your local dev login via `az login`)
#   5. Visual Studio / VS Code auth
# No API keys. No secrets in code.
# ─────────────────────────────────────────────────
credential = DefaultAzureCredential()

# Create a token provider for Azure OpenAI's specific scope
# "cognitive services" is the Azure OpenAI RBAC scope
token_provider = get_bearer_token_provider(
    credential,
    "https://cognitiveservices.azure.com/.default"
)

# Initialize the Azure OpenAI client with token-based auth
client = AzureOpenAI(
    azure_endpoint=AOAI_ENDPOINT,
    azure_ad_token_provider=token_provider,  # keyless — token not API key
    api_version=API_VERSION,
)

# ─────────────────────────────────────────────────
# INFERENCE CALL
# Same API shape as OpenAI — only endpoint + auth differ
# ─────────────────────────────────────────────────
def call_gpt4o(prompt: str, system_message: str = None) -> dict:
    """
    Call GPT-4o deployment and return response with token usage.
    Returns dict with content, model, and usage stats.
    """
    messages = []
    
    if system_message:
        # System message sets the AI's persona and constraints
        messages.append({"role": "system", "content": system_message})
    
    messages.append({"role": "user", "content": prompt})
    
    response = client.chat.completions.create(
        model=DEPLOYMENT_NAME,  # IMPORTANT: this is the deployment name, not "gpt-4o"
        messages=messages,
        max_tokens=500,          # cap response length to control cost
        temperature=0.7,         # 0 = deterministic, 1 = creative; 0.7 is balanced
    )
    
    return {
        "content": response.choices[0].message.content,
        "model": response.model,
        "prompt_tokens": response.usage.prompt_tokens,
        "completion_tokens": response.usage.completion_tokens,
        "total_tokens": response.usage.total_tokens,
        "finish_reason": response.choices[0].finish_reason,
    }

# ─────────────────────────────────────────────────
# TEST CALLS — SRE-relevant prompts
# ─────────────────────────────────────────────────
if __name__ == "__main__":
    print("=" * 60)
    print("Azure OpenAI Keyless Auth Test")
    print(f"Endpoint: {AOAI_ENDPOINT}")
    print(f"Deployment: {DEPLOYMENT_NAME}")
    print("=" * 60)

    # Test 1: Basic SRE question
    print("\n[TEST 1] Basic inference call...")
    result = call_gpt4o(
        prompt="In one paragraph, explain what an SLO burn rate alert is and why it matters.",
        system_message="You are a concise SRE expert. Answer in plain language, 100 words max."
    )
    print(f"Response: {result['content']}")
    print(f"Tokens used: {result['total_tokens']} (prompt: {result['prompt_tokens']}, completion: {result['completion_tokens']})")
    print(f"Model: {result['model']}")

    # Test 2: Structured output prompt (common pattern for automation)
    print("\n[TEST 2] Structured output (JSON)...")
    result2 = call_gpt4o(
        prompt="""
        A service has P99 latency of 4200ms against an SLO of 2000ms.
        Error rate is 0.8% against an SLO budget of 0.1%.
        Respond ONLY in JSON with keys: severity (P1/P2/P3), actions (list of 3 strings), estimated_resolution_minutes (integer).
        """,
        system_message="You are an SRE incident triage system. Respond only with valid JSON, no prose."
    )
    print(f"Response: {result2['content']}")
    print(f"Tokens used: {result2['total_tokens']}")
    
    print("\n[COMPLETE] Keyless auth and inference working correctly.")
PYTHON_SCRIPT

echo "Script created at ~/azure-ai-lab/aoai_keyless.py"
```

**EXPECTED_OUTPUT:**
```
Script created at ~/azure-ai-lab/aoai_keyless.py
```

**WHY:** The script demonstrates the `DefaultAzureCredential` pattern which is identical whether running locally (uses `az login`) or on AKS (uses Managed Identity). The `get_bearer_token_provider` wraps credential refresh automatically — your long-running services never hit token expiry issues. The two test prompts are deliberately SRE-relevant so you can use this script directly in portfolio demos.

Azure AI Content Safety Demo
Shows how to filter prompts BEFORE sending to Azure OpenAI
Severity: 0 (safe) through 6 (most harmful)
Categories: hate, violence, sexual, self_harm
"""

import json
import requests

CS_KEY = os.environ["CS_KEY"]

def analyze_content(text: str) -> dict:
    """
    Analyze text for harmful content.
    Returns category scores — higher = more harmful.
    You set the threshold for what to block.
    """
    url = f"{CS_ENDPOINT}/contentsafety/text:analyze?api-version=2023-10-01"

        "Content-Type": "application/json",

        "text": text,
        "categories": ["Hate", "Violence", "Sexual", "SelfHarm"],

    }
    
    response = requests.post(url, headers=headers, json=payload)
    response.raise_for_status()
    
    result = response.json()
    
    # Extract severity scores per category
    scores = {
        item["category"]: item["severity"]
        for item in result.get("categoriesAnalysis", [])

    scores = analyze_content(text)

    return max_score < threshold, scores

### Step 3 Failures

## Tradeoffs and Failure Modes (Top 3)

**1. Managed Identity vs API Key: Operational Convenience vs Security**

Managed Identity is strictly more secure — no secret to leak, automatic rotation, RBAC-scoped, full audit trail. The tradeoff is operational complexity: local development requires `az login` discipline, debugging auth failures is harder (no simple "is this the right key" test), and misconfigured pod identity in AKS can silently fail to authenticate. API keys are operationally simpler but create a secret management burden. Recommendation: always use Managed Identity in production; use environment-variable-based keys only in local development with a .env file that is gitignored.

**2. Azure OpenAI vs AWS Bedrock: Ecosystem Lock-In vs Model Diversity**

Azure OpenAI gives you GPT-4o — which as of 2025-2026 remains the reference model for enterprise AI tasks. But you're locked to OpenAI models exclusively. Bedrock gives you Claude 3.x (frequently outperforming GPT-4o on reasoning tasks), Llama 3 (open weights, lower cost), and Titan (Amazon native). The architecture risk: if OpenAI's model performance degrades or pricing increases, Azure OpenAI customers have no in-platform alternative. Multi-cloud AI architecture (Azure OpenAI + Bedrock behind an abstraction layer) is the resilient but complex approach. For most enterprises in 2025-2026: choose based on your primary cloud, not model preference — the operational integration wins.

**3. Content Safety Threshold: False Positives vs Harm Exposure**

Setting Content Safety threshold too low (blocking severity 2+) causes false positives on legitimate technical content — security research queries, medical information, fiction writing. Setting it too high (blocking only severity 6) exposes your application to borderline harmful content at severity 4. There is no universally correct threshold — it is a product and legal decision, not a technical one. The SRE/platform engineering role is to make the threshold configurable without code deployment (feature flag or config), implement a human review queue for borderline cases (severity 2-4), and instrument refusal rates so product can tune the threshold based on observed false positive rates. Building a static threshold into code is the failure mode.
