## Analogy

Think of AI models as **message brokers in a multi-queue architecture**.

- You have multiple brokers available: RabbitMQ, Kafka, ActiveMQ, Pulsar.
- Each has different throughput caps (RPM/TPM = messages/second), latency profiles, payload size limits (context window = max message size), and cost models (per-message vs. subscription).
- You don't hard-wire every producer to one broker. You put a **routing layer** (like a MQ federation or a load balancer) in front — that's **LiteLLM**.
- Prompt caching = pre-warmed queue consumer. The broker already has the routing rules loaded in memory. First message is slow; subsequent ones are fast and cheap.

Just as you'd never run a 25M msg/day notification platform on a single broker without failover routing, you don't build SRE AI tooling on a single model without a routing proxy.

## 30-Second Version

Five model families dominate SRE/platform AI use in 2026:
- **Claude (Anthropic)**: Strongest reasoning + long context; best for runbooks, postmortems, policy analysis. Has prompt caching — 90% cost reduction on repeated system prompts.
- **GPT-4o / o3 (OpenAI)**: Broadest ecosystem, tool-use mature, most third-party integrations.
- **Gemini 2.0/2.5 (Google)**: Native Google Cloud integration; 1M-2M token context; best for log/trace analysis over huge windows.
- **Grok (xAI)**: Real-time X/Twitter data access; useful for threat intel; less proven for SRE ops.
- **Llama 3.x (Meta, open weights)**: Self-hosted on your own VPC; zero data egress; compliance-friendly; lower quality ceiling than frontier models.

Route all of them through **LiteLLM** so you never hard-code an API.

### Pricing Model Types

**Per-token**: You pay separately for input tokens (your prompt) and output tokens (model's reply). Dominant model. Claude, GPT, Gemini all use this.

**Per-request**: Fixed fee per API call regardless of token count. Rare now — legacy behavior.

**Subscription + credits**: OpenAI and Anthropic both offer pre-paid credit models for predictable budgeting.

**Self-host (Llama)**: You pay GPU compute (e.g., EC2 p3.2xlarge ~$3.06/hr for ~40 tokens/sec) — convert to $/1M tokens yourself.

## 5-Minute Version

### Prompt Caching (Anthropic) — Deep Dive

This is the highest-leverage cost optimization for SRE use cases.

**How it works**: You mark the top section of your prompt (system prompt + static runbook context) with a `cache_control` block. Anthropic stores it server-side for 5 minutes (default) or up to 1 hour. Subsequent requests reuse the cache — you pay only 10% of normal input cost for cached tokens.

**SRE math**: If your system prompt is 2,000 tokens (a full runbook + SLO definition) and you call the API 100 times during an incident:
- Without caching: 100 × 2,000 × $0.000003 = $0.60 per incident
- With caching: 1 full call + 99 × 2,000 × $0.0000003 = $0.003 + $0.0594 = $0.062 per incident
- **That is 10x cheaper per incident**, at no quality cost.

**When NOT to cache**: If your system prompt changes every request (personalized context), caching gives no benefit. Use dynamic slots in the non-cached portion.

### Model Routing with LiteLLM

LiteLLM is a Python library + optional proxy server that presents a single OpenAI-compatible API surface in front of any model. You write your code once against the OpenAI SDK format, and LiteLLM translates to Claude, Gemini, or Llama under the hood.

**Routing strategies you can configure:**
- **Fallback**: Primary = Claude; if rate-limited or unavailable, fall back to GPT-4o
- **Cost-based routing**: Route to cheapest model that meets minimum quality threshold
- **Load balancing**: Round-robin across multiple API keys for the same model (multiplies effective RPM)
- **Canary routing**: 5% of requests to new model, 95% to proven model — same as your Kubernetes canary deployments

**Rate limits reality check** (approximate, verified as of mid-2025):

| Provider | Tier 1 RPM | Tier 1 TPM |
|---|---|---|
| Anthropic (Claude Sonnet) | 50 RPM | 40,000 TPM |
| OpenAI (GPT-4o) | 500 RPM | 30,000 TPM |
| Google (Gemini 2.0 Flash) | 1,000 RPM | 4,000,000 TPM |
| Together.ai (Llama 3.3 70B) | 600 RPM | 100,000 TPM |

Note: RPM/TPM limits increase with usage tiers and committed spend. Anthropic scales to 4,000 RPM at Tier 4.

### Failure Modes in AI Platform Integration

**1. Token cliff**: Your runbook query + SLO data + log excerpt exceeds context window. Model silently truncates input — you get a confident but incomplete answer. Mitigation: chunk inputs, always log token counts.

**2. Rate limit cascade**: Incident triggers 50 simultaneous AI calls (one per alert). You hit RPM cap. All 50 calls fail. MTTR spikes. Mitigation: queue AI calls through LiteLLM with retry + fallback; treat AI as non-critical path in incident response.

**3. Cost spike from runaway agents**: An agentic loop (model calls tool, tool returns data, model calls again) can iterate 50+ times on a single incident, spending $2-5 in one session. Mitigation: max_iterations cap, spend alerts via Anthropic/OpenAI dashboards, same discipline as AWS Cost Anomaly Detection.

Answer 2 of 3 to unlock the lab.

**Q1**: Your incident response tool makes 200 API calls per incident, each with a 1,500-token system prompt describing your SLO policy and runbook template. Without caching you pay $3.00/1M input tokens. With Anthropic prompt caching, cached tokens cost $0.30/1M. Roughly how much does one incident cost without caching vs. with caching?

**Q2**: You're building an AI-assisted postmortem tool. The tool needs to analyze 800K tokens of log data in a single call (production incident with dense logs). Which model is the only realistic choice from the table above, and why?

**Q3**: You deploy a LiteLLM proxy with primary=Claude, fallback=GPT-4o. During an incident, Anthropic is having a partial API outage. Describe what happens to your tool's requests and what the user experience looks like if you do NOT configure a timeout before the fallback triggers.

---

## ANSWER KEY — Comprehension Check

**A1**:
- Without caching: 200 calls × 1,500 tokens = 300,000 input tokens × ($3.00/1,000,000) = **$0.90 per incident**
- With caching: 1 full call (1,500 × $0.003 = $0.0045) + 199 cached calls (1,500 × $0.0003 × 199 = $0.0896) = **~$0.094 per incident**
- Savings: ~90% cost reduction. At 100 incidents/month, that's $90 saved vs. $9 — material at scale.

**A2**: **Gemini 2.0** (Flash or Pro). It is the only model in the comparison table with a 1M+ token context window (up to 2M tokens). Claude's 200K and GPT-4o's 128K would both need chunking strategies for 800K tokens. For log analysis where you want the model to reason across the full timeline, Gemini's window eliminates the complexity of chunking and context reassembly — though you trade off some reasoning depth vs. Claude.

**A3**: Without a timeout, LiteLLM will wait for Anthropic's slow/hanging response indefinitely (or until the default HTTP timeout, often 60-600 seconds). The user sees a spinner or no response for up to 10 minutes before the fallback to GPT-4o triggers. During an incident, a 10-minute delay for your AI summarization tool is worse than having no tool. The fix: set `timeout: 10` in LiteLLM config so fallback triggers within 10 seconds of the primary failing.

### Step 4 — Anthropic Prompt Caching

```bash
cat > ~/sre-ai-lab/step4_prompt_caching.py << 'EOF'
"""
Step 4: Anthropic prompt caching demonstration.

We make 5 calls with the same system prompt.
Call 1: cold cache (full input cost)
Calls 2-5: cache hit (10% of input cost for cached portion)

cache_control breakpoint placement rules:
- Mark the END of the static block you want cached
- Everything BEFORE the breakpoint is cached
- Everything AFTER is dynamic (charged at normal rate)
- Minimum cacheable block: 1,024 tokens (pad your system prompt if needed)
"""
import time
import os
from dotenv import load_dotenv
import anthropic

load_dotenv()

client = anthropic.Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

# Expanded system prompt — we need >= 1,024 tokens to trigger caching
# Real-world equivalent: full runbook library, SLO definitions, team context
SYSTEM_PROMPT_BLOCKS = [
    {
        "type": "text",
        "text": """You are an SRE runbook assistant for a high-volume notification platform.

PLATFORM CONTEXT:
- 25 million messages per day across SMS, push, email channels
- Stack: EKS on AWS us-east-1, RabbitMQ clusters (3-node), Cassandra (6-node ring), Redis Cluster (3 primary + 3 replica), MySQL RDS
- SLO: 99.9% delivery success rate = error budget 43.8 min/month
- Secondary SLOs: P95 latency < 2s for SMS delivery, P99 < 5s
- On-call escalation: L1 (PagerDuty auto-page) -> L2 (SRE on-call, 5-min ack) -> L3 (platform eng, 15-min ack)

RUNBOOK LIBRARY SUMMARY:
RMQ-001: Queue depth spike — check consumers first, then publisher rate, then broker memory
RMQ-002: Consumer lag — check pod restarts, OOM events, downstream dependency failures  
RMQ-003: Dead letter queue accumulation — check message TTL, consumer rejection logic, schema changes
CAS-001: Cassandra read timeout — check node health, repair status, compaction queue
CAS-002: Cassandra write timeout — check consistency level, hinted handoff, node overload
RED-001: Redis cluster failure — check slot coverage, replication lag, maxmemory policy
K8S-001: Pod CrashLoopBackOff — check resource limits, liveness probes, config map changes
K8S-002: HPA not scaling — check metrics server, custom metrics adapter, scale-down cooldown
NET-001: Increased error rate without deploy — check upstream dependencies, DNS resolution, TLS cert expiry

ESCALATION CRITERIA:
- Page L3 immediately: data loss, SLO breach > 5 min, cascading failure across 2+ services
- War room at 15 min if: no root cause identified, multiple services affected, customer-facing impact confirmed

RESPONSE FORMAT:
1. Incident classification (which runbook applies)
2. Root cause ranking (probability-ordered)
3. First 3 diagnostic commands (exact kubectl/curl/cqlsh syntax)
4. Immediate mitigation action
5. Escalation trigger condition

Be direct. Operational. No preamble. No theory.
""",
        # cache_control marks this block as cacheable
        # Everything up to this point will be stored server-side
        "cache_control": {"type": "ephemeral"}
    }
]

# Five different incident queries — same system prompt, different user messages
INCIDENT_QUERIES = [
    "Alert: RabbitMQ queue 'sms-outbound-high' depth > 50K. Deploy sms-dispatcher v2.4.1 ran 15 min ago.",
    "Alert: Cassandra P95 read latency > 800ms on keyspace 'notifications'. No recent deploys.",
    "Alert: Redis cluster reporting 2 nodes down. Slot coverage degraded.",
    "Alert: sms-dispatcher pods showing CrashLoopBackOff. 3 of 5 pods affected.",
    "Alert: Overall delivery success rate dropped to 98.7%. SLO breach in 8 minutes if not resolved.",
]

print("Running 5 API calls with prompt caching enabled...\n")
print(f"{'Call':<6} {'Query':<55} {'Latency':>8} {'In Tokens':>10} {'Cache Hit':>10} {'Cost':>10}")
print("-" * 105)

total_cost = 0.0

for i, query in enumerate(INCIDENT_QUERIES, 1):
    start = time.time()
    
    response = client.messages.create(
        model="claude-sonnet-4-5",
        max_tokens=256,
        system=SYSTEM_PROMPT_BLOCKS,  # Pass list of blocks, not a string
        messages=[{"role": "user", "content": query}]
    )
    
    elapsed = time.time() - start
    
    # Extract cache metrics from response
    cache_read = getattr(response.usage, 'cache_read_input_tokens', 0)
    cache_write = getattr(response.usage, 'cache_creation_input_tokens', 0)
    normal_input = response.usage.input_tokens - cache_read - cache_write
    
    # Cost: cache write = 125% of normal; cache read = 10% of normal; output = 100%
    cost = (
        (cache_write / 1_000_000) * 3.75 +   # 125% of $3.00 for initial cache write
        (cache_read / 1_000_000) * 0.30 +     # 10% of $3.00 for cache hits
        (normal_input / 1_000_000) * 3.00 +   # normal rate for uncached input
        (response.usage.output_tokens / 1_000_000) * 15.00
    )
    total_cost += cost
    
    cache_status = f"{cache_read}t" if cache_read > 0 else f"WRITE {cache_write}t"
    query_short = query[:52] + "..." if len(query) > 52 else query
    
    print(f"{i:<6} {query_short:<55} {elapsed:>7.2f}s {response.usage.input_tokens:>10} {cache_status:>10} ${cost:>8.6f}")

print("-" * 105)
print(f"{'Total cost for 5 calls:':<75} ${total_cost:.6f}")

# Compare: what would 5 calls cost WITHOUT caching?
# Approximate: largest system prompt ~620 tokens, 5 calls

Engineers see clean responses; PII never leaves the perimeter. Logged sanitization events for compliance audit trail.

## Tradeoffs and Failure Modes

**Tradeoff 1: Quality vs. Cost vs. Latency — no model wins all three**

Claude claude-sonnet-4-5 wins on reasoning quality and long-document tasks. GPT-4o wins on ecosystem integrations and slightly lower latency. Gemini Flash wins on cost per token and ultra-long context. Llama wins on data sovereignty. The correct answer is not "pick one" — it is "route based on task type." Incident triage = Claude (reasoning matters). Log dump analysis = Gemini (huge context). Regulated environment = Llama (data never leaves). This is a Director-level architecture decision, not a library choice.

**Tradeoff 2: Prompt caching saves cost but adds architectural coupling**

Prompt caching requires you to structure your prompts with stable prefixes. If your system prompt is highly dynamic (personalized per engineer, per incident type), caching never kicks in. The discipline of making system prompts stable and reusable — storing them in a config repo, versioning them, reviewing changes via PR — is operationally valuable beyond cost savings. It forces prompt engineering rigor. The failure mode: cache-busting changes in the system prompt (a typo fix, a format change) silently reset your cache and spike costs until the cache warms again.

**Tradeoff 3: LiteLLM proxy is a new SPOF if you don't treat it like one**

Adding a proxy improves provider portability but adds a hop. If the proxy crashes during an incident, all AI tooling goes dark. Mitigation: (1) run LiteLLM as a Kubernetes deployment with 2+ replicas behind a load balancer, (2) configure a health check endpoint (`/health`), (3) instrument proxy latency as a separate metric — so you can distinguish "Claude is slow" from "proxy is slow." Most importantly: design your incident response workflow so AI assistance is augmentation, not a hard dependency. If the AI layer fails, engineers fall back to runbooks and tribal knowledge — not a broken workflow.
