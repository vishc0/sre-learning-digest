## 30-Second Version: What Is an LLM?

**The RabbitMQ Analogy:**

Think of an LLM as a **sophisticated message consumer that predicts the next word based on all prior words it has seen**. Unlike RabbitMQ (deterministic, repeatable), an LLM's output is probabilistic — the same input message may produce different outputs depending on a `temperature` setting (think: consumer processing concurrency tuning, but for prediction randomness).

| RabbitMQ | LLM | Why It Matters |
|----------|-----|---|
| Consumes messages, outputs acked/nacked state | Consumes tokens, outputs token probabilities | Both are state machines; LLMs just have stochastic outputs |
| Dead letter queue for poison pills | Retry logic + temperature sweep for bad outputs | Both need fallback strategies |
| SLO = latency + no data loss | SLO = latency + no hallucinations (harder) | Silence ≠ correctness in LLMs |
| Throughput measured in msg/sec | Throughput measured in tokens/sec | More tokens = longer latency (non-linear) |

**Core insight**: LLMs are not AI magic. They are **prediction machines trained on patterns**. Temperature, retry, and input framing are your operational levers.

## 2-Minute Version: How Transformers Work

### The Transformer Architecture

A transformer (like Llama 3, Claude, GPT-4) has three sequential phases:

1. **Tokenizer**: Break your input text into numbered tokens (e.g., "hello world" → [15043, 1879])
   - Think: log parser / field extractor in Splunk. Same input should always tokenize the same way.
   
2. **Transformer blocks** (the magic): Each block uses **attention** to decide which prior tokens matter for predicting the next one.
   - Think: Splunk correlation search. If you see tokens [user_id, login_attempt, failed], attention learns to predict "failure_rate ↑" with high probability.
   - Formula: `Attention(Q, K, V) = softmax(Q·K^T / √d_k) · V` — but you do NOT need to memorize this. Intuition: "Which past tokens are most relevant to predicting the next token?"

3. **Output head**: Predicts the probability of each token being the next one.
   - Think: anomaly score. If temp=0 (deterministic), pick highest probability. If temp=0.7 (stochastic), sample from the distribution.

### Vocabulary Translation: Attention Mechanism

| LLM Term | Your Splunk Mental Model | Your RabbitMQ Mental Model |
|----------|-------------------------|---------------------------|
| Attention | Correlation search: "which fields matter?" | Consumer filter: "which message attributes matter?" |
| Token | Event field value | Message property |
| Context window | Splunk index retention (max lookback) | Consumer queue buffer size |
| Embedding | Field fingerprint / normalized representation | Hashed message signature |
| Temperature | Query randomness (semantic vs exact match) | Consumer concurrency (determinism vs throughput) |
| Top-p (nucleus sampling) | Top N results from search | Top N consumer threads |

## 5-Minute Version: Operational Rules for LLM Inference

### Phase 1: Tokens and Context Windows

**Input tokenization is deterministic.** The same text always produces the same token sequence.

```
Input: "What is the capital of France?"
Tokens: [1724, 338, 368, 3007, 310, 9939, 30]
Token count: 7
```

**Context window is your SLO constraint.** Llama 3 (8B) has a 8,192-token context window. Llama 3.1 (405B) has 128,000 tokens. This is *hard limit* like Cassandra max_string_length.

- **Implication for SRE**: Every request costs (input_tokens + output_tokens) × (model_cost_per_token). Larger context = higher cost. This is why RAG (Retrieval Augmented Generation) is cheaper than fine-tuning for domain-specific knowledge.

### Phase 2: Inference Tuning Parameters

| Parameter | Default | Range | Operational Meaning |
|-----------|---------|-------|---|
| `temperature` | 1.0 | 0.0–2.0 | 0.0 = deterministic (always same output); 1.0 = balanced; 2.0 = very random. Set to 0.0 for deterministic tasks (log analysis, code synthesis). |
| `top_p` (nucleus sampling) | 1.0 | 0.0–1.0 | Only sample from top 90% cumulative probability. Reduces tail hallucinations. Good for creative work. |
| `max_tokens` | model default | 1–context_window | Hard limit on output length. Always set this; prevents runaway costs. |
| `frequency_penalty` | 0.0 | -2.0–2.0 | Penalize repeated tokens. Useful for brainstorming (prevents "yes yes yes..."). Bad for factual extraction. |
| `presence_penalty` | 0.0 | -2.0–2.0 | Penalize tokens that have appeared before. Different from frequency. Good for diverse output. |

**SRE Tuning Playbook:**

- **Deterministic task** (code generation, SQL synthesis, JSON extraction): `temperature=0.0, top_p=1.0, max_tokens=1000`
- **Analytical task** (root cause analysis, brainstorming): `temperature=0.7, top_p=0.9, max_tokens=2000`
- **Creative task** (test data generation, anomaly narratives): `temperature=1.2, top_p=0.95, max_tokens=500`

### Phase 3: Cost Models

**Cloud API pricing** (e.g., Claude 3.5 Sonnet via Anthropic):
- Input: $3 per million tokens (~0.003¢ per token)
- Output: $15 per million tokens (~0.015¢ per token)
- Example: 1000-token request, 500-token response = $0.0045

**Self-hosted via Ollama (free, local):
- Hardware cost: GPU (RTX 4090, $1,600) amortized over 2 years = ~$3/month
- Operational overhead: monitoring latency, managing OOM crashes, updating models

**Decision matrix:**

| Use Case | Cloud API | Self-Hosted |
|----------|-----------|-------------|
| High volume (>10k req/day) | Expensive | Preferred (lower cost, full control) |
| Low latency required (<100ms) | Hard (network overhead) | Preferred (local) |
| Compliance (no data leaving VPC) | Impossible | Preferred |
| Development/prototyping | Preferred (easy, pay-as-you-go) | Acceptable (setup overhead) |
| Deterministic output needed | Cloud + low temp | Self-hosted + easier control |

## Training vs Inference: Why This Matters for Your Team

**Training** (offline, expensive):
- Take raw data, adjust model weights over millions of examples
- Cost: $10M–$500M for a foundation model (GPT-4, Claude scale)
- Done once per model family
- You do NOT train; you use pre-trained models

**Inference** (online, your job):
- Load pre-trained weights, run input through the model, get output
- Cost: $0.001–$0.1 per request (cloud) or $0 hardware cost amortized (self-hosted)
- Done billions of times per day at scale
- This is what your team will manage: latency, cost, fallback, observability

**Why Inference Is Hard:**
- Token generation is **sequential** (one token at a time; no parallelism within a single request)
- Latency is proportional to output length: 100-token response takes ~5x longer than 20-token response
- GPU memory is shared: more concurrent requests = slower each request (queueing effect)

## Quantization: The Missing Middle

**Problem**: Llama 3 70B full precision = 140GB VRAM required. Your RTX 4090 only has 24GB.

**Solution**: Quantization — reduce precision from float32 (32-bit) to int8 (8-bit) or int4 (4-bit).

| Quantization | Memory Required | Latency Impact | Quality Drop | Best For |
|----------|----------|---|---|---|
| Full precision (fp32) | 140GB | Baseline | 0% | Research labs, not SRE |
| Half precision (fp16) | 70GB | -5% | <1% | Baseline for most labs |
| Quantized (int8, GGUF) | 35–50GB | -10% | <2% | Self-hosted Ollama |
| Highly quantized (int4, QLoRA) | 14–20GB | -15% | 3–5% | Edge devices, latency-critical |

**Vocabulary Translation:**

| Quantization Concept | RabbitMQ Analogy | Cassandra Analogy |
|---|---|---|
| Full precision | Raw message JSON, no compression | Uncompressed SSTables |
| Quantized (int4) | Gzip-compressed message payload | Snappy-compressed column family |
| Precision loss | Lossy decompression — minor info loss | CQL response rounded to precision | Some cache misses due to changed data |

**SRE Decision**: Use GGUF quantized models via Ollama for self-hosted inference. It's the sweet spot: 8GB GPU, <200ms latency, minimal quality loss.

## How to Set SLOs for Non-Deterministic Systems

This is the staff-engineer question. Here's the exact framework:

### Problem: "Temperature = Randomness, So How Do I SLO?"

Unlike a deterministic API (Cassandra read: 50ms latency guarantee), an LLM with temperature > 0 produces different outputs. How do you SLO that?

### Answer (4-layer framework):

**Layer 1: Separate determinism from performance.**

```
SLO = P95(latency) + Uptime(availability)
NOT accuracy (accuracy is a quality metric, not an SLO)
```

Example: "95% of inference requests complete within 1 second, and the service is up 99.9% of the time."

**Layer 2: Measure hallucination independently.**

Hallucination (wrong output) ≠ SLO violation (missed latency). Track separately:

```
Error Budget 1: Latency SLO — if any request > 1s, you burn 1/100 of monthly budget
Error Budget 2: Hallucination SLO — if output lacks required fields or is nonsensical, you burn 1/1000 of monthly budget
```

**Layer 3: Temperature is deterministic under SLO.**

Set `temperature=0.0` for any SLO-critical path. Non-zero temperature is acceptable **only for non-critical paths** (suggestions, brainstorming).

```yaml
Critical paths (deterministic):
  - Log parsing: temperature=0.0
  - SQL synthesis from schema: temperature=0.0
  - Alert grouping: temperature=0.0

Non-critical paths (stochastic ok):
  - Incident narrative generation: temperature=0.7
  - Runbook suggestions: temperature=0.7
  - Test data generation: temperature=1.0
```

**Layer 4: Quantify hallucination with benchmarks.**

Before deploying, measure quality on a fixed test set:

```
Test set: 100 representative inputs
- Llama 3 8B: 87% correct → acceptable (SLI target: 90%)
- Llama 3 70B: 95% correct → stricter SLI required
```

Set SLI target *above* baseline; burn rate alerts trigger when you drop below it.

### Full Example: SLO for an LLM-Powered Alert Deduplication Engine

```
Service: LLM Alert Deduplication
Primary user: SRE on-call team

SLO (binding contract with team):
  - Latency: P95 < 2 seconds (includes LLM inference + dedup logic)
  - Availability: 99.9% uptime (model server running)
  - Accuracy: 90% correct grouping (tested weekly on fixed benchmark)

SLI (measurement):
  - Latency: histogram of inference time (CloudWatch or Prometheus)
  - Availability: model_server_up (binary metric)
  - Accuracy: (correct_groups / total_requests) rolling 7-day window

Error Budget (monthly):
  - Latency: 100 minutes × 30 days / 2,880 minutes = 1% budget
    → If you run 1M requests/day and each P95 violation costs 1 req, budget = 10k violations/month
  - Accuracy: 99.9% of 1M requests = 1k incorrect groupings allowed/month
  
Burn rate alerts:
  - Latency burns 10x budget: alert if P95 > 20s for 5 min
  - Availability burns 10x budget: alert if uptime < 99%
  - Accuracy burns 5x budget: alert if accuracy < 92% (above target)
```

**Why this works**: You've decoupled speed (SLO), reliability (SLO), and quality (SLI). Each is measured independently.

### Step 7: Measure Latency Distribution (P50/P90/P95)

Create a benchmark script to measure latency across multiple requests:

```bash
# File: benchmark.sh
#!/bin/bash

cat > /tmp/benchmark.py << 'EOF'
import json
import subprocess
import time

latencies = []
for i in range(20):
    start = time.time()
    result = subprocess.run(
        ['curl', '-s', 'http://localhost:11434/api/generate',
         '-H', 'Content-Type: application/json',

        text=True
    )

    latencies.append(elapsed)
    print(f"Request {i+1}: {elapsed:.0f}ms")

latencies.sort()
print(f"\nLatency Statistics (ms):")
print(f"  P50: {latencies[10]:.0f}ms")
print(f"  P90: {latencies[18]:.0f}ms")
print(f"  P95: {latencies[19]:.0f}ms")
EOF

python3 /tmp/benchmark.py
```

**Expected output:**
```

Request 2: 567ms
Request 3: 523ms
...
Request 20: 512ms

Latency Statistics (ms):
  P50: 540ms
  P90: 590ms
  P95: 620ms
```

**Why P50 < P90 < P95?** First request includes model warmup; subsequent requests are faster and more stable.

### Question 2: "What would you do if an LLM starts hallucinating in production?"

**Situation**: Your alert deduplication system groups 1000 alerts/day. Suddenly, it starts grouping unrelated alerts (e.g., "database down" + "user login slow").

**Answer (STAR format)**:

*Situation*: I've debugged a Splunk correlation search that started matching false positives (learned behavior on new data distribution).

*Task*: Diagnose and remediate LLM hallucination.

*Action*:
1. **Immediate (circuit breaker)**: Set a quality SLI alert.
   - Measure: (correct_groups / total_groups) last 1 hour
   - Threshold: if < 85% (normal is 93%), trigger PagerDuty
   - Action: disable LLM dedup, revert to rule-based grouping
2. **Root cause (diagnostics)**:
   - Pull last 100 hallucinating examples (alert pairs that should NOT group)
   - Check input characteristics: are they new alert types? new severity patterns?
   - Check model state: did model weights change? was a different quantization used?
   - Check prompt: did the prompt wording change (temperature? max_tokens?)?
3. **Recovery**:
   - Option A: Increase temperature from 0.7 → 0.0 (forced determinism, may reduce accuracy)
   - Option B: Add few-shot examples to prompt (e.g., "Alert A and B should NOT group because X" in prompt context)
   - Option C: Use a larger model (70B → more reasoning, fewer hallucinations) and accept latency cost
   - Option D: Add a verifier model: run output through a second model to check for contradictions
4. **Testing before re-enable**: Test on 1000 fixed examples (golden set). Only re-enable if accuracy > 91%.

*Result*: Identified that new alert type (cloud-native errors with new token patterns) confused the model. Added 20 negative examples to prompt ("CloudFront timeout" and "RDS slow query" should NOT group). Accuracy recovered to 94%. Re-enabled with 99.9% confidence.

**Why this works**:
- Shows diagnosis framework (not panic, check systematically)
- Options show understanding of LLM mechanics (temperature, few-shot, model size, verification)
- Quantifies recovery (accuracy metric + threshold)
- Operationally sound (circuit breaker + testing)

### Question 3: "How would you monitor and alert on LLM inference latency?"

**Situation**: You have Ollama running 8 concurrent replicas on EKS, serving alert dedup requests.

*Task*: Apply the same framework to LLM inference.

*Action*:
1. **Instrument the inference call**:
   ```python

   inference_latency = prometheus_client.Histogram(
   ```

   - SLI metric: `histogram_quantile(0.95, inference_latency_seconds)`
   - Burn rate: If P95 > 2s for 5 minutes, burn 1% of monthly budget

### Question 4: "What's your strategy for handling context window limits?"

**Situation**: You want to use an LLM for incident RCA, but your investigation includes 50MB of logs. Llama 3 context window is only 8,192 tokens (~32KB).

**Answer (STAR format)**:

*Situation*: Similar to Splunk's index retention limits. You can't search all 5 years of logs in a single query; you have to narrow time range.

*Task*: Design a retrieval strategy that fits large corpora into context windows.

*Action*:
1. **Option 1: RAG (Retrieval Augmented Generation)**
   - Index all logs into a vector database (e.g., Weaviate, Milvus, even simple SQLite with embeddings)
   - When RCA query arrives, retrieve top 5–10 most relevant log chunks (using semantic search)
   - Pack those into prompt: "Here are the relevant logs: ... Now explain the incident"
   - Benefit: Works with 50MB corpus; context window only needs ~10KB
   - Cost: Embedding every log takes time upfront, but amortizes across many RCA queries

2. **Option 2: Hierarchical summarization**
   - Chunk logs into 5 groups (timeline): T-5min to T, T to T+5min, etc.
   - Run LLM on each chunk independently: "Summarize events in this 5-min window"
   - Concatenate summaries (now ~5KB instead of 50MB)
   - Run LLM on summaries: "What's the root cause?"
   - Benefit: Works with any context window, simple to implement
   - Cost: Multiple LLM calls (amortizes if you cache summaries)

3. **Option 3: Hybrid (recommended)**
   - Index logs with vector embeddings (offline, once)
   - On RCA request: retrieve top 20 chunks, then chunk them into N groups (timeline)
   - Summarize each group, pack summaries into context
   - Final LLM call: "Explain root cause given these summaries + context"
   - Benefit: Combines speed (vector search) + compression (summaries) + quality (context window)

*Result*: Implemented hybrid RAG for incident RCA. Embedded all 2-week rolling logs into Weaviate (took 2 hours once). Now, RCA queries (e.g., "Why did the payment service go down?") complete in < 5 seconds: retrieval (~0.5s) + summarization (~1s) + explanation (~3s). Accuracy on historical incidents: 87% (matches human RCA in root cause).

**Why this works**:
- Shows deep understanding of context window constraints
- Offers 3 concrete options (pragmatic, not theoretical)
- Picks the best one (hybrid) with justification
- Quantifies the trade-offs (speed, cost, accuracy)
- Real numbers (2-week rolling log, 87% accuracy)

### Question 5: "What guardrails would you put on an LLM that suggests action?"

**Situation**: You're building an automated incident response system where the LLM can suggest actions (e.g., "Scale up the database cluster", "Restart the pod", "Page the on-call").

**Answer (STAR format)**:

*Situation*: At T-Mobile, I lead incident command. The worst thing is a well-intentioned but wrong action. An LLM that suggests "restart all 500 pods" instead of "restart 5 pods in zone A" is dangerous.

*Task*: Design safeguards so the LLM helps but doesn't break production.

*Action*:
1. **Layer 1: Structured output + validation**
   ```python
   # Force LLM to return JSON schema
   prompt = """
   Suggest an action to resolve this incident:
   [incident context]
   
   Respond ONLY with JSON:
   {
     "action": "scale_database_replicas" | "restart_pods" | "page_oncall" | "escalate",
     "target": "<resource_id>",
     "count": <number>,
     "confidence": 0.0–1.0,
     "reasoning": "<why>"
   }
   """
   
   # Validate output
   action_json = json.loads(llm_output)
   assert action_json["action"] in ALLOWED_ACTIONS
   assert int(action_json["count"]) < MAX_RESOURCES[action_json["action"]]
   assert float(action_json["confidence"]) > 0.7  # LOW confidence = no action
   ```

2. **Layer 2: Require human approval for destructive actions**
   ```python
   DESTRUCTIVE = {"restart_pods", "scale_down", "drain_node"}
   
   if action["action"] in DESTRUCTIVE:
       # Require human approval
       slack_message(f"LLM suggests: {action['reasoning']}\nApprove? Y/N")
       approval = wait_for_human(timeout=5m)
       if not approval:
           return {"status": "rejected", "reason": "human declined"}
   else:
       # Safe actions (page, escalate, gather_metrics) can auto-execute
       execute_action(action)
   ```

3. **Layer 3: Bounded blast radius**
   ```python
   if action["action"] == "restart_pods":
       # Max 5 pods per incident (prevent cascading failure)
       actual_count = min(action["count"], 5)
       # Drain gracefully (30 sec timeout, not SIGKILL)
       for pod in pods[:actual_count]:
           gracefully_terminate(pod, timeout=30s)
   ```

4. **Layer 4: Dry-run + observability**
   ```python
   # Always show what WOULD happen before it happens
   dry_run_output = explain_action(action)
   pagerduty.alert({
       "title": f"LLM Action: {action['action']}",
       "description": dry_run_output,
       "confidence": action["confidence"],
       "status": "executing"
   })
   
   # Post-action: measure impact
   metric = get_metric(incident.metric_name)
   before, after = metric.at_time(action_time - 1m), metric.at_time(action_time + 1m)
   improvement = (after - before) / before
   log(f"Action improved metric by {improvement*100:.1f}%")
   ```

5. **Layer 5: Fallback + rollback**
   ```python
   try:
       execute_action(action)
       wait_for_stability(timeout=2m)
   except Exception as e:
       # Automatic rollback
       log(f"Action failed: {e}")
       rollback(action)  # e.g., scale_down if scale_up caused error
       pagerduty.escalate("Automated action failed; human review required")
   ```

*Result*: Deployed "LLM Incident Helper" with these 5 layers. In 3 months:
- 150 incidents → LLM suggested actions in 120 (80%)
- 45 suggestions were destructive (restart/scale) → all 45 required human approval → human approved 40 (89%)
- 75 were safe (page, escalate) → all 75 auto-executed → 71 helped resolve incident (95% useful)
- 0 false positives that broke production
- Median time-to-resolution dropped from 8 min to 4 min when LLM helped

**Why this works**:
- Layered defense (don't rely on one safeguard)
- Code examples (concrete, not hand-wavy)
- Metrics (tells a story: 80% coverage, 89% human approval, 95% useful, 0 failures)
- Operational realism (timeouts, graceful termination, rollback)
- Acknowledges the risk (LLM is helpful but not infallible)

## Tradeoffs: Three Hard Decisions

### Tradeoff 1: Cloud API vs Self-Hosted — Cost Inversion Math

**The question**: When does self-hosting become cheaper than cloud?

**Math**:

Cloud API cost per month:
```
monthly_cost_cloud = (requests_per_day × 30) × (input_tokens + output_tokens) × price_per_token
Example: 100k requests/day × (500+100) tokens × $0.003 = $180/day = $5,400/month
```

Self-hosted cost per month:
```
monthly_cost_self_hosted = (GPU_hardware_cost / 24_months) + electricity + ops_labor
Example: ($1,600 A10G / 24 months) + $100 electricity + $200 ops = $67 + $100 + $200 = $367/month
```

**Break-even point**:
```
$367/month = (requests/day × 30) × (600 tokens) × $0.003
$367 = requests/day × 30 × 600 × 0.000003
requests/day = $367 / (30 × 600 × 0.000003) = 68,000 requests/day
```

**Decision matrix**:

| Volume | Recommendation | Why |
|--------|---|---|
| < 20k req/day | Cloud API | Cheaper + less ops overhead |
| 20k–70k req/day | Hybrid (API for peak, cache results) | Smooth cost curve |
| > 70k req/day | Self-hosted | Economies of scale favor local hardware |
| > 500k req/day | Self-hosted + multi-GPU | Cloud becomes prohibitively expensive (> $50k/month) |

**Your decision at T-Mobile**: 25M messages/day through notification platform. If 10% need LLM processing = 2.5M requests/day. Cloud cost: **$375k/month** (unacceptable). Self-hosted cluster: 5 × A100 GPUs = $20k hardware + $5k/month ops = break-even after 1 month. Decision: **Self-hosted**.

### Tradeoff 2: Context Window Size vs RAG Complexity

**The question**: Should I use a 128k context window model (Llama 3.1 405B) or a 8k model (Llama 3 8B) + RAG?

**Trade-offs**:

| Factor | 128k Context Window (Llama 3.1 405B) | 8k Context + RAG (Llama 3 8B) |
|--------|------|------|
| Input latency | Slower (more tokens to process) | Faster (fewer tokens, only top-K retrieval) |
| Output quality (complex reasoning) | Better (sees full context) | Good (sees only relevant parts) |
| GPU memory required | 400GB (cloud only, expensive) | 8GB (on-premises feasible) |
| Operational complexity | Simple (just increase max_tokens) | Complex (vector DB, embedding, retrieval logic) |
| Cost (per request) | High ($0.015 input + $0.15 output per 1k) | Low ($0.0003 input + if cached embeddings, ~$0) |
| Useful for | Strategic analysis, long-form writing | Operational tasks, QA over large corpora |

**Decision framework**:

Use **128k context** if:
- You're doing strategic analysis (e.g., "Analyze our entire codebase and recommend refactoring")
- You can afford cloud API cost ($0.15+ per request)
- Latency SLO is > 5 seconds (acceptable for async batch)

Use **8k context + RAG** if:
- You're doing operational tasks (e.g., "Did a database migration cause the outage?")
- You need sub-2-second latency (self-hosted inference only)
- You process 10k+ requests/day (RAG embedding amortizes)

**Your decision for incident RCA**: Use Llama 3 8B + RAG. Incident RCA is operational (< 2s SLO), not strategic. RAG complexity is worth it for cost and speed.

### Tradeoff 3: Temperature 0 (Deterministic) vs Temperature > 0 (Stochastic)

**The question**: Should I always set temperature=0 for reproducibility, or allow randomness?

**Trade-offs**:

| Dimension | Temperature = 0 (Deterministic) | Temperature = 0.7–1.0 (Stochastic) |
|-----------|------|------|
| Reproducibility | Same input = same output (great for testing) | Same input = different outputs (harder to debug) |
| Accuracy | High for factual tasks (e.g., code generation) | Can be higher for open-ended tasks (more diverse attempts) |
| Creativity | Low (always picks most likely token) | High (explores lower-probability paths) |
| Token efficiency | High (shorter outputs on average) | Medium (slightly longer, more exploration) |
| Production readiness | Yes (predictable behavior) | Maybe (harder to SLO) |

**Decision framework**:

Use **temperature = 0.0** for:
- SQL/JSON synthesis (deterministic output required)
- Log parsing (must match exact format)
- Code generation (no room for variation)
- Incident RCA (users expect same analysis for same logs)

Use **temperature = 0.7–1.0** for:
- Brainstorming (want multiple ideas)
- Test data generation (want variety)
- Alert narrative generation (users accept variation, want readability)
- Writing polish (want fluency, not mechanical)

**Your decision for T-Mobile**: Alert dedup uses temperature=0 (deterministic grouping). Incident narrative uses temperature=0.7 (more readable, users accept variation). Alert escalation suggestion uses temperature=0 (don't want random on-call suggestions).

## Key Takeaways

1. **LLMs are prediction machines, not magic.** Temperature, context window, and quantization are your operational levers.

2. **Inference is your job, not training.** You will manage latency, cost, and quality. Training is a research/ML engineer problem.

3. **Set SLOs separately from hallucination.** P95 latency SLO is separate from accuracy SLI. Use temperature=0 for SLO-critical paths.

4. **Break even on self-hosted around 70k req/day.** Below that, cloud is cheaper. Above that, self-hosted (Ollama, vLLM) is cheaper.

5. **RAG (retrieval) is cheaper than large context windows.** Don't use Llama 3.1 405B for operational tasks. Use Llama 3 8B + RAG instead.

6. **Treat LLM outputs like Cassandra queries: measure P95, set budgets, alert on burn rate.** Build observability from day 1.
