## Analogy (Middleware/RabbitMQ/Incident Command First)

Think about how you run messaging infrastructure at T-Mobile:

| **Messaging Analogy** | **AI Inference Equivalent** |
|---|---|
| RabbitMQ managed service (no broker management) | **Bedrock** — fully managed, you call the API, AWS runs everything |
| PCF/EKS + custom RabbitMQ cluster (you own the brokers) | **SageMaker endpoints** — AWS manages the compute lifecycle, you bring the model |
| Raw EC2 + your own RabbitMQ build | **EC2 GPU instances (g4dn/p3/p4)** — you own everything, bare metal flexibility |

Just like you wouldn't self-host RabbitMQ if SQS/SNS met your needs, you wouldn't deploy GPU EKS if Bedrock meets your latency and cost requirements.

**Knowledge Base analogy**: Bedrock Knowledge Base is like a **Splunk index with semantic search**. Instead of querying by keyword/regex, you query by meaning. The S3 bucket is your log source. The knowledge base is your indexed search layer. The difference: LLM semantics instead of TF-IDF.

**Incident Command analogy for GPU capacity**: GPU instances are like helicopter slots during a major incident — finite, expensive, must be pre-planned. You cannot autoscale a GPU fleet in 60 seconds the way you autoscale web pods.

## 30-Second Version

AWS gives you three tiers for AI inference:

1. **Bedrock** — call an API, pay per token, zero infrastructure. Use for: most production LLM workloads.
2. **SageMaker** — bring your own model, AWS manages the GPU instances. Use for: custom/fine-tuned models.
3. **EC2 GPU / EKS GPU** — bare metal control, bring everything. Use for: bleeding-edge research, regulatory air-gap, or models SageMaker won't serve.

Bedrock Knowledge Base adds RAG (document-grounded answers) on top of Bedrock in ~10 minutes of config.

## 5-Minute Version (Tradeoffs, Failure Modes, Design Decisions)

### Bedrock Deep Dive

Bedrock is AWS's managed LLM API gateway. You don't own the instances — you own the API calls.

**Architecture:**
```
Your App → Bedrock API → AWS-managed model fleet → Response
                ↕
        Bedrock Knowledge Base (optional RAG layer)
                ↕
        S3 Bucket → OpenSearch Serverless (vector index)
```

**Bedrock Knowledge Base — how it works:**
1. You point it at an S3 bucket (PDFs, text, HTML, markdown)
2. Bedrock chunking service splits documents into ~500-token segments
3. Titan Embeddings model converts each chunk to a vector
4. Vectors stored in OpenSearch Serverless (auto-provisioned)
5. Query arrives → Bedrock embeds the query → nearest-vector chunks retrieved → injected into LLM prompt → LLM answers grounded in your documents

This is RAG (Retrieval-Augmented Generation). The LLM never hallucinates from its training data because it's forced to answer from retrieved chunks.

**Bedrock failure modes:**
- Throttling: Bedrock has default quota limits (e.g., 50 RPM for Claude 3.5 Sonnet). File a quota increase before production launch.
- Cold start: First call in a session may have 2-5s latency. Warm with a scheduled ping if P99 SLO is tight.
- Content policy: Bedrock runs guardrails. A prompt that triggers a refusal looks like a 400 error — design your retry logic to distinguish this from transient errors (don't retry content blocks).
- Knowledge Base sync lag: After uploading to S3, ingestion job must run (5-30 min). Stale index = stale answers.

### SageMaker Endpoint Patterns

SageMaker sits between Bedrock (managed) and EKS GPU (self-managed):
- You provide the model artifacts (in S3) or container image
- You specify instance type (ml.g4dn.xlarge, ml.p3.2xlarge, etc.)
- SageMaker manages: auto-scaling, health checks, rolling updates, multi-AZ

**Real-time vs Batch vs Serverless:**

| Pattern | Use Case | Cold Start | Cost Model |
|---|---|---|---|
| Real-time endpoint | <1s latency, persistent traffic | None (always warm) | Per instance-hour |
| Serverless inference | Spiky/low traffic | 1-5 seconds | Per invocation |
| Async inference | Long-running jobs, >1MB payloads | None | Per instance-hour |
| Batch transform | Offline scoring on datasets | None | Per instance-hour |

### EKS GPU Infrastructure

When you need Kubernetes-native GPU serving:

**Stack layers (bottom to top):**
```
EC2 GPU Instance (g4dn/p3/p4)
    └── GPU drivers (NVIDIA kernel module)
        └── NVIDIA Container Toolkit (allows containers to see GPUs)
            └── NVIDIA Device Plugin (exposes GPU as K8s resource "nvidia.com/gpu")
                └── Your pod requests: resources.limits: nvidia.com/gpu: 1
```

**GPU Operator** (NVIDIA) automates the entire driver/toolkit/plugin stack as a single Helm chart. Without it, you'd manually install drivers on every node — which breaks with node-group rolling updates.

**Instance selection guide:**

| Instance | GPU | VRAM | Use Case |
|---|---|---|---|
| g4dn.xlarge | NVIDIA T4 | 16 GB | Inference for models up to 7B params |
| g4dn.12xlarge | 4x T4 | 64 GB | Inference for 13B-34B models |
| p3.2xlarge | V100 | 16 GB | Training small models, inference 7B |
| p3.8xlarge | 4x V100 | 64 GB | Training medium, inference 34B-70B |
| p4d.24xlarge | 8x A100 | 320 GB | Training large, inference 70B+ |
| p5.48xlarge | 8x H100 | 640 GB | Frontier model training/inference |

**EKS node group for GPU (key config differences):**
- AMI: `AmazonLinux2-GPU` or Bottlerocket GPU variant
- Taint: `nvidia.com/gpu=true:NoSchedule` — prevents non-GPU workloads from landing on expensive nodes
- Toleration: GPU pods must explicitly tolerate this taint
- Node selector: `accelerator: nvidia-tesla-t4`

### Inference Endpoint Patterns for SRE

**Pattern 1: Bedrock with retry/fallback**
```
Request → Bedrock API
    → Throttle (429)? → exponential backoff → retry up to 3x
    → Hard fail? → fallback to cached response or degraded mode
    → Success → response
```

**Pattern 2: SageMaker multi-model endpoint**
One endpoint serves multiple models. Saves cost when you have 10+ models that don't all run concurrently. SageMaker unloads cold models from GPU VRAM and loads on demand.

**Pattern 3: EKS GPU with KEDA**
KEDA (Kubernetes Event Driven Autoscaler) scales GPU pods based on queue depth (SQS, Kafka). Better than HPA for AI workloads because request processing time varies wildly — a simple CPU/memory metric doesn't represent queue saturation.

## Director/VP Lens

**IC implements:**
- Writes Bedrock API calls, configures Knowledge Base, installs GPU Operator, writes Helm values for inference workloads
- Debugs CUDA OOM errors, tunes batch size, monitors token throughput
- Writes IaC for Bedrock KB and SageMaker endpoints

**You govern as Director/VP SRE:**

1. **Build vs Buy decision framework** — Bedrock first unless there is a specific technical reason to go lower. The "we need control" argument is usually cost-avoidance theater. GPU EKS costs 10x more in engineering time than Bedrock.

2. **Cost governance** — AI spend can grow 100x faster than traditional compute. Enforce:
   - Token budget per service/team (Bedrock supports tagging per API call)
   - Bedrock guardrails as policy (prevent prompt injection from users driving up costs)
   - Model tier selection policy (Haiku for classification, Sonnet for reasoning — not Opus everywhere)

3. **SLO definition for AI workloads** — harder than traditional SLOs:
   - Latency SLO: P99 < 3s (LLMs are slow — set expectations with product)
   - Availability SLO: 99.5% (Bedrock has had outages — plan fallback)
   - Quality SLO: harder — requires automated eval (LLM-as-judge or golden set accuracy)
   - Cost SLO: $/1000 requests budget enforced via CloudWatch billing alerts

4. **AI incident patterns** — new failure modes your team needs runbooks for:
   - Model degradation (AWS silently updates a model version — responses change)
   - Prompt injection attacks driving up cost or exfiltrating data
   - Knowledge Base staleness (S3 updated but ingestion job didn't run)
   - VRAM OOM on GPU nodes causing pod crash loops
   - Quota exhaustion during traffic spikes (file quota increases proactively)

5. **AI in your team's existing tooling** — frame this as platform evolution:
   - Splunk MLTK (already using) → augment with Bedrock for natural language query translation
   - Incident runbooks → Bedrock KB over your runbook library → faster on-call lookup
   - Alert fatigue → LLM-assisted alert grouping using Bedrock + your Splunk alert data

## ANSWER KEY — Comprehension Check

**A1. Classification decision tree:**
Questions to ask first:
- What is the latency requirement? (Bedrock adds ~200-500ms network; fine-tuned BERT can do <50ms on GPU)
- Does the classification need to generalize to new categories, or is the label set fixed?
- What is the accuracy requirement? (BERT fine-tuned on your data will beat a generic LLM for narrow classification)
- What is the engineering team's ML model deployment maturity?

Decision: For a fixed-label, high-volume classification task at 10M req/day, fine-tuned BERT on SageMaker serverless will likely win on cost AND latency. Bedrock Haiku is the right choice if: the categories change frequently, you want zero model management, or the classification is nuanced enough to need semantic reasoning. EKS GPU is the wrong choice here — the operational overhead is not justified for an established classification pattern.

**A2. Bedrock Knowledge Base wrong answers — operational investigation:**
1. **Staleness**: Check when the last ingestion job ran. Go to Bedrock console → Knowledge Base → Data source → Last sync status. If S3 was updated but sync didn't run, trigger a manual sync. Check if auto-sync is configured.
2. **Chunking mismatch**: The default chunk size (512 tokens) may split critical context across chunks. A document where the answer spans two chunks will retrieve only one half. Investigate by testing with smaller chunk sizes or semantic chunking.
3. **Retrieval count too low**: Default retrieval is top-3 chunks. If the answer needs more context, increase `numberOfResults` in the RetrieveAndGenerate API call.
4. **Source document quality**: Plain-text extraction from PDFs loses table structure. Check the raw S3 source — if it's a scanned PDF, OCR quality will degrade retrieval.

**A3. GPU EKS TCO hidden costs:**
The "40% cheaper per token" calculation typically counts only instance cost. Hidden costs include:
- **Engineer time**: GPU driver management, CUDA compatibility, device plugin upgrades = 0.5-1 FTE ongoing
- **Node management**: GPU AMIs need separate update pipelines; GPU nodes can't drain like CPU nodes (long-running inference jobs block draining)
- **Capacity reservation**: GPU instances need reserved capacity (1yr or 3yr) — upfront capital commitment
- **Idle cost**: GPU nodes can't scale to zero without cold-start penalty; minimum fleet floor = cost floor
- **On-call complexity**: CUDA OOM, driver panics, and GPU health are specialized skills — training cost + incident cost
- **Quota overhead**: EC2 GPU quota requests have longer lead times than Bedrock quota increases
- **Network cost**: Data transfer in/out of GPU nodes adds up at scale
- **Security surface**: Managing GPU AMIs, kernel modules, container runtime increases CVE exposure

A complete TCO: (instance cost × utilization factor) + (engineer cost allocation) + (incident response cost) + (capacity reservation capital) + (training cost for on-call). At most organizations this makes GPU EKS 2-3x more expensive than the raw compute number suggests for workloads below 1B tokens/day.

### Step 3: Call Claude via Bedrock — Python (boto3)

**WHY**: boto3 is how your production code calls Bedrock. The Python SDK adds retry logic, credential rotation, and structured response handling that the CLI lacks.

```bash
# Install boto3 if not present
pip3 install boto3 --quiet

# Create the Python script
cat > /tmp/bedrock_call.py << 'EOF'
"""
Bedrock API caller — boto3 pattern
Demonstrates: basic invoke, token counting, latency measurement
"""
import boto3
import json
import time

# Initialize the Bedrock runtime client
# boto3 picks up credentials from ~/.aws/credentials (same as CLI)
bedrock_rt = boto3.client(
    service_name='bedrock-runtime',
    region_name='us-east-1'
)

# Build the request body — Anthropic Messages API format
# This is the same format as direct Anthropic API, just wrapped in Bedrock
request_body = {
    "anthropic_version": "bedrock-2023-05-31",  # Required by Bedrock — don't change
    "max_tokens": 300,
    "messages": [
        {
            "role": "user",
            "content": "You are an SRE expert. In 3 bullet points, what are the top SRE concerns when running LLM inference on AWS?"
        }
    ]
}

print("Calling Bedrock (Claude 3.5 Sonnet)...")
print("-" * 50)

# Measure latency
start_time = time.time()

# Invoke the model
response = bedrock_rt.invoke_model(
    modelId='anthropic.claude-3-5-sonnet-20241022-v2:0',
    contentType='application/json',
    accept='application/json',
    body=json.dumps(request_body)
)

end_time = time.time()
latency_ms = (end_time - start_time) * 1000

# Parse the response body (it comes back as a streaming body)
response_body = json.loads(response['body'].read())

# Extract the text
output_text = response_body['content'][0]['text']
input_tokens = response_body['usage']['input_tokens']
output_tokens = response_body['usage']['output_tokens']

# Calculate approximate cost (Claude 3.5 Sonnet pricing)
cost_usd = (input_tokens * 0.003 / 1000) + (output_tokens * 0.015 / 1000)

print(f"Response:\n{output_text}")
print("-" * 50)
print(f"Latency:       {latency_ms:.0f} ms")
print(f"Input tokens:  {input_tokens}")
print(f"Output tokens: {output_tokens}")
print(f"Cost:          ${cost_usd:.6f}")
EOF

python3 /tmp/bedrock_call.py
```

**EXPECTED_OUTPUT:**
```
Calling Bedrock (Claude 3.5 Sonnet)...
--------------------------------------------------
Response:
• **Latency and SLA management**: LLM inference has high and variable P99 latency (2-10s vs milliseconds for traditional APIs). SRE teams must define realistic SLOs, implement request timeouts, circuit breakers, and fallback responses to prevent cascading failures when model endpoints slow down under load.

• **Cost observability and quota governance**: Token consumption can spike unpredictably with long prompts, requiring per-service cost tagging, token budget enforcement, and proactive Bedrock quota increase requests before traffic events. Without this, a single misconfigured prompt template can generate 10x expected spend overnight.

• **Reliability architecture for non-deterministic systems**: Unlike traditional services where the same input always produces the same output, LLM responses vary. SRE runbooks must distinguish between infrastructure failures (5xx, timeout, throttle) and model behavior issues (unexpected output, content policy blocks), requiring separate alert channels and escalation paths for each type.
--------------------------------------------------
Latency:       1847 ms
Input tokens:  47
Output tokens: 187
Cost:          $0.002946
```

## Lab ANSWER KEY

### Diagnostics Per Step

**Step 1 — Model access not showing in CLI:**
- The console activation takes 2-5 minutes. Wait and retry.
- If you see `AccessDeniedException` on invoke, the model access was not saved. Return to the Bedrock console and verify the model shows as "Access granted."
- If `list-foundation-models` returns empty, check your region — Bedrock is only available in select regions. us-east-1 is the most complete.

**Step 2 — CLI invoke returns `ValidationException`:**
- The `anthropic_version` field must be exactly `"bedrock-2023-05-31"`. This is Bedrock-specific, not the Anthropic API version.
- The `messages` array format must match Anthropic Messages API spec. Check that `role` is `"user"` (not `"human"` — that's the legacy Text Completions API).
- If you get `ResourceNotFoundException`, the model ID is wrong. Re-run Step 1 to get exact model IDs.

**Step 3 — Python `invoke_model` returns empty body:**
- The response body is a `StreamingBody` object — you MUST call `.read()` on it before it can be parsed. Missing the `.read()` is the most common boto3 Bedrock bug.
- If you get `botocore.exceptions.NoCredentialsError`, run `aws configure` in WSL2 and verify credentials are set.

**Step 4 — Knowledge Base creation fails:**
- The OpenSearch Serverless collection creation requires that the Bedrock service principal has permission to create it. Using the console "Quick create" option handles this automatically.
- If you get `AccessDeniedException` on `create-knowledge-base`, check that the IAM role trust policy has `aws:SourceAccount` matching your account ID exactly.
- If the KB stays in `CREATING` state >10 min, check the Bedrock console Events tab for error details.

**Step 5 — Ingestion job completes but queries return wrong answers:**
- Check document was actually uploaded: `aws s3 ls s3://${BUCKET_NAME}/runbooks/`
- Check ingestion statistics: look for `numberOfDocumentsScanned > 0` in the ingestion job details
- If `numberOfDocumentsIndexed = 0`, the document format may not be supported. Plain .txt files are always supported. Try re-uploading as .txt if using .pdf.
- Query returning generic LLM responses (not grounded in your document): increase `numberOfResults` in the retrieve config — the relevant chunk may not be in the top 3.

**Step 6 — Latency numbers higher than expected:**
- First call in a new session is always higher (TCP connection establishment). Discard call 1 from P50 calculation.
- Latency varies significantly by time of day. During AWS peak hours (US business hours), expect 20-40% higher latency.
- If all calls are >3000ms, check your WSL2 DNS resolution — slow DNS lookup can add 500-1000ms. Test with `time aws sts get-caller-identity`.

**Q2 — LLM SLOs:**
> "Traditional SLOs are availability + latency + error rate. For LLMs we add a fourth dimension: quality. At T-Mobile I govern SLOs for the notification platform — latency is objective, failure is binary. LLM quality is neither. I'd define quality SLO as: score > 80% on a golden evaluation set (20 representative prompts with human-rated ground truth), re-scored weekly. For infrastructure SLOs: availability 99.5% (Bedrock has had outages — don't promise 99.9%), P99 latency 5s (set user expectations), error rate <0.1% excluding content policy blocks."

**Q3 — Knowledge Base staleness debug:**

**Q4 — Bedrock vs GPU EKS for fintech PII:**
> "Bedrock wins for fintech with PII. Bedrock supports AWS PrivateLink — all traffic stays within the AWS network, never hits the public internet. ## Tradeoffs and Failure Modes

### Top 3 Tradeoffs

**1. Bedrock Managed vs GPU EKS Control**

| Dimension | Bedrock | GPU EKS |
|---|---|---|
| Model selection | Catalog only (no custom models) | Any model, any version |
| Latency | 200-500ms overhead | Can optimize to bare metal |
| Cost at scale | Linear per token | Fixed fleet + engineering |
| Engineering overhead | Near zero | High (drivers, CUDA, ops) |
| Compliance | AWS responsibility | Your responsibility |
| Cold start | None | Minutes for GPU node scaling |

**Governing principle**: Bedrock is the right default. GPU EKS requires a specific, documented technical reason AND a TCO analysis showing net benefit after engineering cost.

**2. Knowledge Base Sync Frequency vs Freshness**

Bedrock KB does not automatically re-index when S3 changes. You must trigger ingestion jobs. Options:
- Manual trigger: simple, but human-dependent
- EventBridge rule: trigger ingestion when S3 object is created/modified → near-real-time (5-10 min lag)
- Scheduled ingestion: every hour/day → predictable lag

**Failure mode**: Team uploads a runbook update during an active incident, expects the KB to answer with new content, but ingestion hasn't run. Answer based on stale data makes the incident worse. **Fix**: Add ingestion job monitoring to your ops dashboard; alert if last successful sync >1 hour old.

**3. Token Cost vs Context Quality**

More context in the prompt = better answers + higher cost. Knowledge Base returns top-N chunks. If N is too small, the answer misses relevant content. If N is too large, you pay for tokens that don't improve the answer.

**Failure mode**: Setting `numberOfResults: 10` for every query doubles your KB query cost. Setting `numberOfResults: 1` causes missed context on multi-part questions.

**Fix**: Start with `numberOfResults: 3`, measure answer quality on your evaluation set, increase incrementally until quality plateaus. For simple factual lookups, 3 is usually sufficient. For complex reasoning over technical documentation, 5-7 works better.
