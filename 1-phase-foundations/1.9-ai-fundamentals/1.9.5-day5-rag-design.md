## Analogy — MART Framework Reborn

Your Splunk MART framework: when a Sev1 fires, MART surfaces **relevant** historical incidents, correlated metrics, and runbook links to the responder. You don't dump the entire Splunk index — you retrieve the 5 most relevant signals and surface them in context.

RAG is the same pattern, applied to a language model:

```
MART (Splunk):                          RAG (Bedrock):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. Alert fires                          1. Engineer asks question
2. MART queries Splunk indexes          2. RAG queries vector DB
3. Returns top 5 correlated events      3. Returns top 5 relevant chunks
4. SRE reads context + decides          4. LLM reads context + generates answer
5. Resolution logged back to MART       5. Answer can cite source runbook

MART = your existing RAG. Bedrock KB = MART with a language model at the end.
```

The chunking problem is your MART signal carving problem: too broad a time window = noise; too narrow = miss context. Same math applies to text chunks.

## 30-Second Version

RAG = retrieve relevant documents, stuff them into the LLM's context window, get a grounded answer. Three components: (1) Indexing pipeline — chunk your docs, embed them, store vectors. (2) Query pipeline — embed the question, find similar chunks, inject into prompt. (3) Generation — LLM answers using only retrieved context. Use RAG when your knowledge changes faster than you can fine-tune a model. Build on Bedrock Knowledge Bases for managed infra; use pgvector when you need SQL joins with operational metadata.

## 2-Minute Version

**The core problem RAG solves**: LLMs are trained on data with a cutoff date. Your runbooks, incident postmortems, and architecture decisions don't exist in the model's weights. Fine-tuning is expensive and slow — you'd have to retrain every time an SRE writes a new runbook. RAG externalizes the knowledge: keep it in a vector database, retrieve relevant pieces at query time, inject into the prompt.

**The pipeline in four stages**:

1. **Ingest and chunk**: Split documents into ~512-token overlapping chunks. Overlap (~50 tokens) ensures a sentence at a chunk boundary gets captured by one of the two adjacent chunks.

2. **Embed**: Run each chunk through an embedding model (text-embedding-3-small produces 1536-dim vectors). Similar meaning = nearby vectors in that 1536-dimensional space.

3. **Store**: Save vectors + original text + metadata (doc_id, source, last_updated) to a vector DB (Pinecone = managed, pgvector = SQL-integrated, Chroma = local dev).

4. **Query**: At runtime — embed the question, find top-K nearest chunks (cosine similarity), optionally rerank, inject into LLM prompt as context.

**RAG vs Fine-tuning decision rule**:
- Knowledge is factual and changes frequently → RAG
- Knowledge is behavioral (tone, format, domain vocabulary) → Fine-tuning
- Both → RAG + Fine-tuning (most enterprise cases)
- Neither → Prompt engineering alone (most startup cases)

## 5-Minute Version — Tradeoffs, Failure Modes, Design Decisions

### Chunking Strategies

**Fixed-size chunking** (512 tokens, 50-token overlap):
- Simple, deterministic, works for uniform-length docs (runbooks, ADRs)
- Failure mode: splits mid-sentence, breaks semantic units ("restart the service after **[end of chunk]** confirming the pod is healthy **[start of chunk]**")
- Use when: documents have consistent structure, speed matters, budget is tight

**Semantic chunking** (split at sentence/paragraph boundaries based on embedding distance):
- Better retrieval quality — each chunk is a coherent thought
- Slower to index (requires embedding during chunking, not just at query time)
- Failure mode: inconsistent chunk sizes cause variable context usage; large chunks may exceed context window when multiple are retrieved
- Use when: docs are narrative prose (postmortems, architecture decision records)

**Recursive character splitting** (LangChain default):
- Hierarchical: try splitting on `\n\n`, then `\n`, then ` `, then character
- Best general-purpose for mixed-format docs (Confluence pages, GitHub READMEs)
- Bedrock Knowledge Bases uses this by default

**Recommendation for SRE runbooks**: Recursive splitting with 512-token chunks, 100-token overlap. Runbooks have mixed structure (headers, bullet lists, code blocks) — recursive handles all three.

### Embedding Models

| Model | Dimensions | Cost | Best For |
|---|---|---|---|
| text-embedding-3-small | 1536 | $0.02/1M tokens | General purpose, great default |
| text-embedding-3-large | 3072 | $0.13/1M tokens | Higher accuracy, multilingual |
| Amazon Titan Embeddings V2 | 1024 | $0.02/1M tokens | AWS-native, stays in VPC, HIPAA-eligible |
| Cohere Embed v3 | 1024 | $0.10/1M tokens | Best for search-specific use cases |

For T-Mobile-scale (telecom = regulated): Titan Embeddings keeps data in your AWS account, no cross-provider data transfer, easier compliance narrative.

### Vector Database Selection

| DB | Deployment | Best For | SRE Consideration |
|---|---|---|---|
| **Pinecone** | Managed SaaS | Fast start, no infra ops | SLA is external dependency — counts as 3rd-party MTTF |
| **pgvector** | Your Postgres | SQL joins with operational metadata | Familiar ops model; backup/restore via existing RDS runbooks |
| **Chroma** | Local / self-hosted | Dev/test, small-scale | No managed HA — not production-grade at T-Mobile scale |
| **OpenSearch** | AWS managed | Hybrid search built-in, existing AWS footprint | Already know Elasticsearch ops patterns |
| **Bedrock KB** | Fully managed | Zero vector DB management | Least control, highest abstraction |

**Decision for this lab**: Bedrock Knowledge Bases (manages vector store for you — OpenSearch Serverless under the hood). For production at regulated telecom: pgvector on RDS + Titan Embeddings — all data stays in your AWS account.

### Hybrid Search — BM25 + Vector

Pure vector search misses exact-match queries ("what is the RabbitMQ connection timeout setting" — keyword match wins here).
Pure BM25 misses semantic queries ("how do I handle message backpressure" — vector wins here).

Hybrid combines both:
```
final_score = α * vector_score + (1 - α) * bm25_score
```
α = 0.7 favors semantic; α = 0.3 favors keyword. Tune based on your query distribution. OpenSearch and Pinecone both support hybrid natively.

### Reranking

First retrieval (top-20 chunks) is fast but imprecise. Reranker is a cross-encoder model that reads the question AND each chunk together, produces a relevance score, re-sorts top-20 to top-5.

Cost: adds ~100-200ms latency per query. Worth it for production — reduces hallucination by ensuring the LLM sees genuinely relevant context, not just "nearby in vector space."

Options: Cohere Rerank, Bedrock native reranking (preview), local cross-encoder (ms-marco-MiniLM).

### Failure Modes

**1. Chunk boundary hallucination**: The critical context is split across two chunks; neither is retrieved; LLM hallucinates. Fix: increase overlap, use semantic chunking, check RAGAS context recall score.

**2. Retrieval-generation mismatch**: Retriever returns correct chunk, LLM ignores it and uses training data anyway. Fix: prompt engineering ("Answer ONLY from the provided context. If unsure, say 'not in context'"). Monitor with RAGAS faithfulness score — should be >0.85.

**3. Metadata staleness**: S3 runbook updated but vector DB not re-indexed. LLM answers from stale chunk. Fix: S3 event trigger → Lambda → re-index on update. Bedrock KB supports scheduled and on-demand sync.

**4. Top-K too small**: Retrieval returns only 3 chunks; the answer requires synthesizing 5 sections. Fix: increase K to 10 at retrieval, rerank down to 5 for the LLM context.

### RAGAS Evaluation Metrics

| Metric | What It Measures | Target |
|---|---|---|
| **Faithfulness** | Is the answer grounded in retrieved context? (no hallucination) | > 0.85 |
| **Answer Relevancy** | Does the answer actually address the question? | > 0.80 |
| **Context Precision** | Are retrieved chunks actually relevant to the question? | > 0.75 |
| **Context Recall** | Did retrieval capture all necessary context? | > 0.70 |

RAGAS score = your SLO for the AI pipeline. Set burn rate alerts on faithfulness drops below 0.80.

## Director/VP Lens

**IC implements**:
- Chunking strategy, overlap tuning, embedding model selection
- Vector DB schema design (metadata fields, indexing strategy)
- Query pipeline code (embed → retrieve → rerank → generate)
- RAGAS eval harness, benchmark dataset creation

**Director governs**:
- **Build vs Buy decision**: Bedrock KB (managed, faster to value, less control) vs. self-managed LangChain pipeline (full control, 3x engineering cost). Right answer for a platform team supporting 50 engineers: Bedrock KB to start, migration path to pgvector when you need SQL joins.
- **Data residency policy**: Where do embeddings live? Does the embedding model call leave your VPC? Titan Embeddings + Bedrock KB = all AWS, cleaner compliance story for telecom.
- **Eval budget**: RAGAS requires a ground-truth dataset — who curates it? Budget 20 engineer-hours per domain to build the eval set. Without it, you have no SLO for your AI pipeline.
- **Retraining trigger policy**: Define what causes a re-index (runbook update? weekly scheduled? SLA breach?). This is an operational policy decision, not an engineering one.
- **Cost governance**: $0.02/1M embedding tokens sounds trivial. At 1000 engineers asking 50 questions/day, with 5 chunks retrieved per query = 250M tokens/month = $5/month. Cheap. But vector DB storage at Pinecone for 10M chunks = $70/month. Know your cost drivers before approving the platform.
- **AI change management**: Engineers fear "AI replacing runbooks." Frame RAG as MART 2.0 — it surfaces context to the human, the human still makes decisions. Same message you used when rolling out MART at T-Mobile.

## ANSWER KEY — Comprehension Check

**A1**: RAG. Runbooks change daily — fine-tuning requires retraining the model on each change (expensive, slow, requires GPU infrastructure). RAG only requires re-indexing the updated document (minutes, ~$0.01). Fine-tuning is appropriate for behavioral changes (teaching the model T-Mobile's incident communication tone), not factual knowledge updates. Use both if you want tone + current knowledge.

**A2**: The DLQ runbook content exists in Confluence but was either (a) not ingested into the vector DB (check your S3 sync), (b) chunked in a way that splits the DLQ section across chunks with no single chunk dense enough to surface in top-5, or (c) the embedding for "dead-letter queue" is not close enough to the query embedding. Fix: (1) verify S3 contains the DLQ runbook (check metadata filtering), (2) increase top-K from 5 to 15 for this query type, (3) add metadata tag `topic=rabbitmq-dlq` and test metadata-filtered retrieval, (4) inspect the chunk that should contain DLQ info — if it's split poorly, switch to semantic chunking for that doc.

**A3**: Honest answer: No, the question does NOT stay in your AWS account. text-embedding-3-small is an OpenAI API call — query text is sent to OpenAI's servers to produce the embedding vector. Fix: replace text-embedding-3-small with Amazon Titan Embeddings V2 — the embedding call stays within your AWS VPC. Claude on Bedrock for generation already stays in AWS. With Titan + Bedrock, the entire pipeline is AWS-internal, satisfying the CISO's data residency requirement.

### Dead-Letter Queue (DLQ) Policy
When a message is rejected or expires, RabbitMQ routes it to the dead-letter exchange (DLX).
Configure DLQ on the notification queue:
- x-dead-letter-exchange: dlx.notification
- x-message-ttl: 86400000 (24 hours in ms)

### Connection Backpressure
When consumer lag exceeds 50,000 messages, RabbitMQ activates flow control.
Symptoms: producer publish rate drops to near zero, consumer metrics show zero throughput.
Resolution steps:
1. Check consumer count: rabbitmqctl list_consumers | grep notification
2. Scale consumer pods: kubectl scale deployment notification-consumer --replicas=20 -n platform
3. Monitor queue depth in Grafana dashboard: sre-rabbitmq-overview
4. If queue depth > 500k: trigger emergency consumer scale runbook

### Connection Reset Procedure
Safe restart sequence to avoid message loss:
1. Pause producers: kubectl scale deployment notification-producer --replicas=0
2. Wait for in-flight messages to drain: watch rabbitmqctl list_queues name messages
3. Restart RabbitMQ node: systemctl restart rabbitmq-server (or pod rollout)
4. Verify: rabbitmqctl cluster_status
5. Resume producers: kubectl scale deployment notification-producer --replicas=5
EOF

# Cassandra Operations Runbook
## Version: 3.0 | Owner: Data Platform SRE | Last Updated: 2025-06

### Read Timeout Resolution
Default read timeout: 5000ms. Timeouts indicate either slow nodes or query inefficiency.

Diagnosis steps:
1. Check node status: nodetool status
2. Identify slow queries: nodetool tpstats | grep -E 'ReadStage|RequestResponseStage'
3. Check pending compactions: nodetool compactionstats
4. Review table read latency: nodetool tablestats keyspace.table | grep 'Read latency'

Resolution by cause:
- If pending compactions > 1000: increase concurrent_compactors in cassandra.yaml, restart node
- If specific partition is hot: review query patterns, consider partition key redesign
- If all nodes slow: check GC logs at /var/log/cassandra/gc.log

### Repair Procedure
Run weekly to prevent data inconsistency (anti-entropy):
```bash
# Run incremental repair on a single node (less disruptive than full repair)
nodetool repair -inc keyspace_name
# Full repair (use during maintenance windows only)
nodetool repair keyspace_name
```
Never run repair on more than one node simultaneously in a 3-node cluster.

### Node Replacement
When replacing a failed node:
1. Identify dead node token: nodetool ring | grep DOWN
2. On new node: set replace_address: <dead_node_ip> in cassandra.yaml
3. Start Cassandra on new node — it will bootstrap automatically
4. Monitor progress: nodetool netstats
5. Remove old node entry: nodetool removenode <host_id>
EOF

"""
RAG Pipeline for SRE Runbooks — Bedrock Knowledge Bases
Week 9, Day 5 Lab — Vishweshwar Chippa
"""

import boto3
import json
import os
from typing import Optional

# Initialize Bedrock clients
# bedrock-agent-runtime handles query/retrieve operations
bedrock_runtime = boto3.client("bedrock-agent-runtime", region_name="us-east-1")
# bedrock-runtime handles direct LLM inference
bedrock = boto3.client("bedrock-runtime", region_name="us-east-1")

# Your Knowledge Base ID from Step 5
KB_ID = os.environ.get("KB_ID", "ABCDE12345")  # Replace with actual KB ID


def retrieve_from_kb(
    query: str,
    top_k: int = 5,
    metadata_filter: Optional[dict] = None
) -> list[dict]:
    """
    Step 1 of RAG: Retrieve relevant chunks from the Knowledge Base.
    This is the 'R' in RAG — pure retrieval, no generation yet.
    Equivalent to MART querying Splunk for correlated events.
    """
    
    retrieve_params = {
        "knowledgeBaseId": KB_ID,
        "retrievalQuery": {"text": query},
        "retrievalConfiguration": {
            "vectorSearchConfiguration": {
                "numberOfResults": top_k,
                # HYBRID SEARCH: combines vector similarity + keyword BM25
                # "overrideSearchType": "HYBRID"  # Uncomment to enable hybrid
            }
        }
    }
    
    # Add metadata filter if provided
    # This is like Splunk index=prod before the similarity search
    if metadata_filter:
        retrieve_params["retrievalConfiguration"]["vectorSearchConfiguration"][
            "filter"
        ] = metadata_filter
    
    response = bedrock_runtime.retrieve(**retrieve_params)
    
    # Extract chunk text and metadata from response
    chunks = []
    for result in response["retrievalResults"]:
        chunks.append({
            "text": result["content"]["text"],
            "score": result["score"],  # cosine similarity score (0-1)
            "source": result.get("location", {}).get("s3Location", {}).get("uri", "unknown"),
        })
    
    return chunks


def generate_answer(query: str, context_chunks: list[dict]) -> str:
    """
    Step 2 of RAG: Generate an answer using Claude with retrieved context.
    This is the 'AG' in RAG — Augmented Generation.
    The LLM is grounded to ONLY the retrieved runbook sections.
    """
    
    # Format retrieved chunks into context string
    context_text = "\n\n---\n\n".join([
        f"[Source: {chunk['source'].split('/')[-1]} | Relevance: {chunk['score']:.2f}]\n{chunk['text']}"
        for chunk in context_chunks
    ])
    
    # Prompt engineering: instruct the model to stay grounded in context
    # "Answer ONLY from context" is critical to prevent hallucination
    prompt = f"""You are an expert SRE assistant helping engineers during incidents.

RETRIEVED RUNBOOK CONTEXT:
{context_text}

QUESTION: {query}

INSTRUCTIONS:
- Answer ONLY using information from the retrieved context above
- If the answer is not in the context, say: "This information is not in the available runbooks. Please check Confluence or escalate."
- Be specific and actionable — you are helping an engineer during a live incident
- Cite which runbook section your answer comes from

ANSWER:"""
    
    # Call Claude 3 Haiku (fast + cheap for operational queries)
    response = bedrock.invoke_model(
        modelId="anthropic.claude-3-haiku-20240307-v1:0",
        body=json.dumps({
            "anthropic_version": "bedrock-2023-05-31",
            "max_tokens": 1024,
            "messages": [{"role": "user", "content": prompt}]
        }),
        contentType="application/json",
        accept="application/json"
    )
    
    result = json.loads(response["body"].read())
    return result["content"][0]["text"]


def rag_query(query: str, top_k: int = 5, metadata_filter: Optional[dict] = None) -> dict:
    """
    Full RAG pipeline: retrieve → generate.
    Returns both the answer and the retrieved chunks (for evaluation/debugging).
    """
    print(f"\n{'='*60}")
    print(f"QUERY: {query}")
    print(f"{'='*60}")
    
    # Step 1: Retrieve
    print("\n[1/2] Retrieving relevant runbook sections...")
    chunks = retrieve_from_kb(query, top_k=top_k, metadata_filter=metadata_filter)
    
    print(f"  Retrieved {len(chunks)} chunks:")
    for i, chunk in enumerate(chunks):
        print(f"  Chunk {i+1}: score={chunk['score']:.3f} | source={chunk['source'].split('/')[-1]}")
    
    # Step 2: Generate
    print("\n[2/2] Generating answer with Claude 3 Haiku...")
    answer = generate_answer(query, chunks)
    
    print(f"\nANSWER:\n{answer}")
    
    return {
        "query": query,
        "answer": answer,
        "contexts": [c["text"] for c in chunks],
        "sources": [c["source"] for c in chunks],
        "scores": [c["score"] for c in chunks]
    }


# ──────────────────────────────────────────────
# Test queries — covering your T-Mobile domain
# ──────────────────────────────────────────────
if __name__ == "__main__":
    
    # Query 1: RabbitMQ DLQ — tests specific keyword retrieval
    result1 = rag_query(
        "How do I requeue messages from the RabbitMQ dead-letter queue back to the primary queue?"
    )
    
    # Query 2: Kubernetes CrashLoopBackOff — tests procedure retrieval
    result2 = rag_query(
        "A pod is stuck in CrashLoopBackOff due to OOMKilled. What are the steps to fix it?"
    )
    
    # Query 3: Metadata-filtered query — RabbitMQ only
    # Filter: only search in documents tagged with topic=rabbitmq
    # This is the S3 metadata we set during upload
    # Note: Bedrock KB metadata filtering uses the document metadata from S3
    result3 = rag_query(
        "What is the connection timeout setting for RabbitMQ?",
        top_k=3
        # metadata_filter example (if KB supports metadata filtering):
        # metadata_filter={"equals": {"key": "topic", "value": "rabbitmq"}}
    )
    
    # Query 4: Out-of-scope query — tests grounding (should say "not in runbooks")
    result4 = rag_query(

    )
    
    print("\n" + "="*60)
    print("All queries complete. Results ready for RAGAS evaluation.")
    print("="*60)

## Tradeoffs and Failure Modes — Top 3

**1. RAG vs Fine-tuning vs Prompt Engineering**

| Approach | When to Use | When NOT to Use |
|---|---|---|
| Prompt engineering | Knowledge is in training data; task is format/tone | Knowledge is proprietary or post-cutoff |
| RAG | Knowledge changes frequently; grounding is required; auditability matters | Knowledge is small and static (just add it to system prompt) |
| Fine-tuning | Behavioral change needed (domain vocabulary, output format, reasoning style) | Knowledge updates are frequent (must retrain each time) |
| RAG + Fine-tuning | Domain-specific behavior + current knowledge | Budget-constrained early stage |

**2. Vector DB Selection Determines Your Ops Model**

Pinecone: 3rd-party SLA dependency. When Pinecone has an outage, your RAG pipeline is down. For incident response tooling (you're using it during outages), this is a critical risk. pgvector on RDS or OpenSearch Serverless (AWS-managed) eliminates this external dependency. Decision rule: if the RAG system is in your critical path for incident response, keep the vector store in your AWS account.

**3. Chunking Strategy Determines Retrieval Quality — and is Hard to Change**

Once you've indexed 50,000 documents with 512-token fixed-size chunks, switching to semantic chunking requires re-indexing everything — 2-3 hours of compute, ~$50 in embedding costs. Invest in chunking strategy evaluation before production indexing. Benchmark 3 strategies against 20 representative queries from your SRE team. Pick the winner, then index at scale. Changing this in production is expensive and requires a migration window.
