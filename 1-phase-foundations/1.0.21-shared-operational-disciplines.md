## The Core Idea

Six operational disciplines govern how technology organizations deliver and maintain software. None of them are owned by a single team. Each team participates in all of them — with different roles, at different intensities.

```
┌─────────────────────────────────────────────────────────────────────┐
│                   Six Shared Operational Disciplines                │
│                                                                     │
│  1. Incident Management      — Respond when things break           │
│  2. Problem Management       — Prevent things from breaking again   │
│  3. Change Management        — Control how things change            │
│  4. Release Management       — Orchestrate when things ship         │
│  5. Environment Management   — Maintain where things run            │
│  6. System Reliability Mgmt  — Measure and improve how well        │
│                                things work over time                │
└─────────────────────────────────────────────────────────────────────┘
```

These six disciplines are **interlocking**. A failure in one cascades into the others:
- A change without proper Change Management creates an Incident.
- An Incident without proper Incident Management produces a poor Problem record.
- A Problem without proper Problem Management causes the same Incident again.
- Releases without proper Release Management destabilize Environments.
- Unstable Environments produce unreliable deployments and undiscoverable defects.
- All of the above show up as degraded SLOs in System Reliability Management.

**SRE's governing principle**: System Reliability Management is not the sixth discipline bolted onto the other five. It is the lens through which all five are measured, improved, and governed. SRE does not replace the other disciplines — it makes them measurable.

## How to Read the Participation Tables

Each discipline has a participation table. Read it as follows:

| Symbol | Meaning |
|--------|---------|
| **GOVERNS** | Owns the process framework, standards, and tooling for this discipline |
| **LEADS** | Executes the primary work in this discipline |
| **CONTRIBUTES** | Has a specific, defined role; must show up and do their part |
| **CONSUMES** | Receives outputs from this discipline; provides inputs when asked |
| **INFORMED** | Must be kept aware; does not actively participate |
| **—** | Not relevant to this discipline |

## Discipline 1 — Incident Management

**What it is**: The structured process for detecting, declaring, responding to, resolving, and documenting disruptions to production services.

**Why it is shared**: No single team has all the knowledge needed to resolve a production incident. Every incident involves the service that failed (product team), the platform it runs on (TKe, MW, DB), the tooling that supports response (MIM, SNOW, MSFT), and the process that governs behavior during crisis (SRE, MIM, GOV). Each team that is absent or unprepared extends MTTR.

**Who governs it**: MIM Team owns the *process*. SRE owns the *technical standard*. SNOW Team owns the *tooling*. These three must align — a process without tooling is chaos; tooling without process is theater; neither without a technical standard produces learning.

### Team Participation in Incident Management

| Team | Role | Specific Contribution |
|------|------|-----------------------|
| **L1/L2 Triage** | LEADS (first response) | Acknowledge alert within SLA; triage severity; execute known runbooks; escalate to SRE with structured handoff |
| **SRE** | LEADS (technical) | Own technical diagnosis; drive remediation; approve emergency changes; declare resolution |
| **MIM Team** | GOVERNS | Assign IC within 5 min of P1; control war room; manage comms; ensure scribe captures timeline |
| **SNOW Team** | GOVERNS (tooling) | Provide reliable incident ticketing platform; auto-create tickets from SRE alerts; track SLA timers |
| **Backend Teams** | CONTRIBUTES | Respond as SME for their service; diagnose application-layer failures; execute rollbacks for their code |
| **Front End Teams** | CONTRIBUTES | Respond as SME; diagnose client-side failures; provide RUM data during customer-facing incidents |
| **MW Team** | CONTRIBUTES | Respond as SME for messaging layer; diagnose queue failures, DLQ events, consumer lag incidents |
| **TKe Team** | CONTRIBUTES | Respond as SME for cluster issues; diagnose node pressure, pod scheduling failures, control plane events |
| **DB Team** | CONTRIBUTES | Respond as SME; diagnose lock contention, slow queries, connection pool exhaustion, replication lag |
| **Firewall Team** | CONTRIBUTES | Respond when connectivity symptoms suggest policy cause; validate rules; emergency rule changes |
| **Illumio Team** | CONTRIBUTES | Respond when traffic is blocked; validate segmentation policies; emergency policy relaxation |
| **GitLab Team** | CONTRIBUTES | Provide deployment history; execute emergency rollback via pipeline; Vault issues |
| **CYB** | CONTRIBUTES | Lead incident if security indicators present; forensics; determine if operational ↔ security boundary crossed |
| **DSEC** | CONTRIBUTES | Assess data exposure scope; breach notification obligation assessment; preserve evidence |
| **MS Office Tools** | CONTRIBUTES | Provide war room (Teams channel); backup comms if primary channel degrades |
| **GOV** | CONSUMES | Notified of P1/P2; assess regulatory notification obligations (TCPA window breach, etc.) |
| **PO** | CONSUMES | Notified of customer-impacting incidents; provides business priority context for remediation decisions |
| **LDR** | CONSUMES | Notified per escalation protocol; receives executive briefing from MIM within incident SLA |
| **Business Team** | INFORMED | Informed of incidents affecting business processes they depend on |
| **JIRA Team** | INFORMED | Incident action items flow into JIRA; JIRA availability needed for tracking |
| **QA Teams** | INFORMED | Informed of defect-escape incidents; will update test coverage post-incident |
| **PM** | INFORMED | Notified if incident creates program-level schedule risk |

**The three failure modes that extend every incident**:
1. L1/L2 holds the incident too long before escalating → SRE engages late → MTTD grows
2. SME team (DB, MW, FW, ILL) is not reachable or cannot make decisions alone → SRE waits for escalation chain → MTTR grows
3. MIM does not control the war room → SRE engineers handle technical work AND executive communication simultaneously → cognitive overload → mistakes

## Discipline 2 — Problem Management

**What it is**: The process for identifying the root causes of incidents, implementing permanent fixes, and preventing recurrence. Incident Management puts out the fire; Problem Management fireproofs the building.

**Why it is shared**: Root causes almost never live in one team's domain. A P1 caused by a queue backlog might have contributing factors in application code (Backend Teams), messaging configuration (MW Team), Kubernetes resource limits (TKe Team), and an environment parity gap (ENV Teams) that hid the issue in staging. Each team must own its contributing cause — SRE cannot fix what other teams built.

**Who governs it**: SRE owns the postmortem process. SNOW Team owns the Problem record in ITSM tooling. JIRA Team owns action item tracking. These must be connected — a postmortem with action items not tracked in JIRA is a postmortem that produces no improvement.

### Team Participation in Problem Management

| Team | Role | Specific Contribution |
|------|------|-----------------------|
| **SRE** | GOVERNS | Own and complete postmortem within 5 business days; facilitate blameless review; assign action items; track MTTR trend |
| **MIM Team** | GOVERNS (process) | Ensure Problem record created in SNOW; link incident tickets to Problem; drive action item follow-up |
| **SNOW Team** | GOVERNS (tooling) | Maintain Problem record lifecycle; link incidents to Problems; track recurrence rate |
| **JIRA Team** | GOVERNS (tracking) | Provide JIRA project configured for reliability action items with SLA tracking |
| **Backend Teams** | LEADS | Own action items for application-layer causes; implement code fixes, add retry logic, fix instrumentation gaps |
| **Front End Teams** | LEADS | Own action items for client-side causes; fix error handling, improve user journey SLIs |
| **MW Team** | LEADS | Own action items for messaging-layer causes; fix queue topology, DLQ handling, consumer retry policy |
| **TKe Team** | LEADS | Own action items for cluster-level causes; fix resource limits, HPA config, node pool settings |
| **DB Team** | LEADS | Own action items for data-layer causes; fix slow queries, index strategy, connection pool config |
| **Firewall Team** | LEADS | Own action items for network policy causes; update and test rules proactively |
| **Illumio Team** | LEADS | Own action items for segmentation policy causes; pre-approve policies for new services before deployment |
| **GIT Team** | LEADS | Own action items for pipeline/Vault/config causes; fix cert rotation automation, pipeline reliability |
| **QA Teams** | CONTRIBUTES | Update test coverage for every defect escape; add test cases for the scenario that caused the incident |
| **ENV Teams** | CONTRIBUTES | Fix environment parity gaps discovered during the incident; update environment topology to match production |
| **PERF Teams** | CONTRIBUTES | Add load test scenarios for capacity-related incident causes; update SLO stress test thresholds |
| **CYB** | CONTRIBUTES | Drive security-related action items; update detection rules; close exploit paths |
| **DSEC** | CONTRIBUTES | Drive data security action items; update DLP policies; adjust log retention if gap discovered |
| **GOV** | CONSUMES | Review postmortems for compliance-related causes; update risk register; adjust audit controls |
| **PO** | CONSUMES | Review postmortems for product-layer contributing factors; prioritize reliability fixes in backlog |
| **LDR** | CONSUMES | Monthly review of Problem Management KPIs: recurring incidents, open action items, closure rate |
| **PM** | CONTRIBUTES | Track postmortem action items in JIRA; report on action item closure rate to leadership |
| **BVT Team** | CONTRIBUTES | Add BVT test cases for any defect that escaped into production and was caught by users |

**The three failure modes that make Problem Management ineffective**:
1. Postmortems written but action items not tracked → same incident recurs → team loses trust in the process
2. Action items assigned to SRE for problems in other teams' systems → SRE cannot fix what they do not own → action items age out
3. Problem records in SNOW disconnected from action items in JIRA → no visibility into whether fixes are being implemented

## Discipline 3 — Change Management

**What it is**: The governance process that controls what changes reach production, how, when, and with what safety verification. Every change to a production system — code, configuration, network policy, database schema, security rule — is subject to Change Management.

**Why it is shared**: Every team that touches production participates in Change Management as a requester, approver, or affected party. The discipline fails when any team believes their changes are exempt — "it's just a config change," "it's a security fix, it can't wait," "it's only a comment in a script." Every undocumented change is a future incident with no correlatable cause.

**Who governs it**: RMC / CAB owns the process framework and approval governance. SNOW Team owns the change record tooling. SRE owns the production acceptance gate. GOV owns compliance-mandated change controls.

### Team Participation in Change Management

| Team | Role | Specific Contribution |
|------|------|-----------------------|
| **RMC / CAB** | GOVERNS | Operate the change advisory process; enforce change windows; manage freeze calendars; require rollback plans |
| **SNOW Team** | GOVERNS (tooling) | Provide reliable change record system; maintain change audit trail; link changes to incidents for correlation |
| **SRE** | GOVERNS (production gate) | Set reliability standards for changes; provide go/no-go on production readiness; define rollback criteria |
| **GOV** | GOVERNS (compliance) | Define compliance-mandated controls; audit change records; ensure regulatory change windows are respected |
| **CYB** | GOVERNS (security gate) | Security review and approval for all changes with security implications |
| **Bar Raiser Call** | GOVERNS (quality gate) | Quality and architecture gate for significant new services or platform changes |
| **Backend Teams** | LEADS | Raise all change requests for application code; document rollback plans; execute changes per approved window |
| **Front End Teams** | LEADS | Raise all change requests for frontend deployments; provide user journey impact assessment |
| **MW Team** | LEADS | Raise change requests for messaging topology changes; document impact on consumers and producers |
| **TKe Team** | LEADS | Raise change requests for cluster upgrades, node pool changes, policy updates; coordinate maintenance windows |
| **DB Team** | LEADS | Raise change requests for schema changes, index changes, stored procedure updates; provide backward-compatibility proof |
| **GIT Team** | LEADS | Raise change requests for pipeline changes, Vault policy updates, config pipeline modifications |
| **FW Team** | LEADS | Raise change requests for all firewall rule changes; provide staging test evidence and rollback procedure |
| **ILL Team** | LEADS | Raise change requests for Illumio policy changes; provide traffic flow validation evidence |
| **DSEC** | LEADS | Raise change requests for DLP policy enforcement changes; provide staging validation evidence |
| **QA Teams** | CONTRIBUTES | Provide test evidence required by change request; validate regression coverage for the change scope |
| **BVT Team** | CONTRIBUTES | Validate BVT coverage for new change scope; confirm BVT will detect regression after this change |
| **PERF Teams** | CONTRIBUTES | Provide load test evidence for changes affecting throughput or latency SLOs |
| **ENV Teams** | CONTRIBUTES | Validate environment parity for the change; confirm staging results are representative |
| **PO** | CONTRIBUTES | Provide business priority and business impact for change scheduling decisions |
| **LDR** | CONSUMES | Receive change activity reports; approve emergency changes above standard threshold |
| **PM** | CONTRIBUTES | Track change implementation against program schedule; flag changes that create program risk |
| **MIM** | CONSUMES | Consume change feed for incident correlation during and after change windows |
| **L12** | CONSUMES | Informed of upcoming changes that may generate alerts; update runbooks for expected change-period behavior |
| **Business Team** | INFORMED | Informed of changes affecting business-critical flows during business hours |

**The three failure modes that break Change Management**:
1. Teams bypass CAB under urgency — "we'll do the ticket after" — producing undocumented changes that become undiagnosable incidents
2. Change tickets exist but contain no rollback plan — when the change fails, the team improvises a rollback under pressure with no pre-validated procedure
3. Security and compliance teams claim change exemption — their changes carry identical reliability risk to any other change; the exemption creates a blind spot in the change event feed that observability AI depends on

## Discipline 4 — Release Management

**What it is**: The coordination discipline that governs *when* and *how* software is delivered from development through to production — managing release trains, versioning, deployment strategy, and the handoff from development to operations. Distinct from Change Management (which governs individual changes) — Release Management orchestrates the bundling, scheduling, and delivery of multiple changes as a coordinated release.

**Why it is shared**: A release involves code (product teams), pipelines (GitLab), testing (QA, BVT, PERF), environments (ENV Teams), deployment infrastructure (TKe, GIT), operational readiness (SRE, runbooks, monitoring), and business timing (PO, BIZ, PM). No single team controls all of these. An uncoordinated release is the highest-risk type of change.

**Who governs it**: RMC Team owns the release calendar and release governance process. GIT Team owns the delivery pipeline. SRE owns production acceptance and SLO validation post-release.

### Team Participation in Release Management

| Team | Role | Specific Contribution |
|------|------|-----------------------|
| **RMC Team** | GOVERNS | Own release calendar; coordinate release trains; manage versioning standards; orchestrate release go/no-go |
| **GIT Team** | GOVERNS (pipeline) | Own CI/CD pipeline that executes releases; manage artifact versioning; ensure pipeline reliability |
| **SRE** | GOVERNS (prod acceptance) | Define release acceptance criteria (SLO gate); own canary promotion decisions; own rollback authority |
| **PM** | GOVERNS (schedule) | Own program-level release schedule; track dependencies across teams; flag release blockers |
| **Backend Teams** | LEADS | Deliver release artifacts; own feature branch management; ensure release is tested before handoff |
| **Front End Teams** | LEADS | Deliver frontend release artifacts; coordinate API contract changes with backend |
| **MW Team** | LEADS | Deliver messaging configuration changes as part of release; coordinate consumer/producer compatibility |
| **QA Teams** | CONTRIBUTES | Execute release regression suite; provide release sign-off (pass/fail verdict against test plan) |
| **BVT Team** | CONTRIBUTES | Run BVT immediately post-deployment; provide first pass/fail signal within 5 minutes |
| **PERF Teams** | CONTRIBUTES | Run pre-release load test for significant releases; provide SLO stress validation report |
| **ENV Teams** | CONTRIBUTES | Ensure environments are refreshed and release-ready; coordinate environment availability for release activities |
| **TKe Team** | CONTRIBUTES | Ensure cluster has capacity for new release; pre-validate resource requests; coordinate node pool readiness |
| **DB Team** | CONTRIBUTES | Execute schema migrations as part of release; validate backward compatibility with previous version |
| **CYB** | CONTRIBUTES | Security sign-off for each release (SAST/DAST results reviewed, no open Critical findings) |
| **GOV** | CONTRIBUTES | Compliance verification for releases affecting regulated services; sign-off on change window compliance |
| **PO** | CONTRIBUTES | Business readiness sign-off; confirm downstream business teams are prepared for the release |
| **BIZ** | CONSUMES | Informed of release timing that affects business operations; provides business calendar blackout dates |
| **LDR** | CONSUMES | Informed of major releases; receives release summary with reliability posture assessment |
| **L12** | CONTRIBUTES | Updated with release-specific runbook changes; briefed on expected behavior changes post-release |
| **MIM** | CONTRIBUTES | On standby for major releases; incident command structure pre-activated for high-risk releases |
| **DSEC** | CONTRIBUTES | Review releases touching data pipelines or PII-handling code; validate data classification of new fields |
| **JIRA Team** | CONTRIBUTES | Ensure release tracking in JIRA is current; release version tagged on all completed work items |

**The three failure modes that break Release Management**:
1. Releases without a production SLO gate — the release goes to 100% traffic with no canary validation, so a bad release reaches all users before anyone knows
2. Environment drift between release staging and production — QA passes in staging, fails in production because ENV Teams and product teams did not maintain parity
3. No release readiness checklist — some teams (L12, MIM, SRE) were not briefed before the release, so when it generates alerts, responders don't know if the alerts are expected behavior or a real incident

## Discipline 5 — Environment Management

**What it is**: The operational discipline that provisions, maintains, governs, and retires all environments across the software delivery lifecycle — development, integration, QA, performance, staging, UAT, and production. Each environment is a working model of the production system at a particular fidelity level.

**Why it is shared**: Environments are consumed by every team but owned by none alone. Product teams need development environments. QA needs test environments. PERF needs load test environments. TKe owns the Kubernetes substrate. DB Team manages databases in each environment. MW Team manages messaging in each environment. SRE owns production. ENV Teams coordinate the non-production lifecycle. When any team treats their environment carelessly, it creates false confidence in other teams' work.

**Who governs it**: ENV Teams govern non-production environments. TKe governs the Kubernetes platform across all environments. SRE governs production. DB and MW govern their systems across all environments.

### Team Participation in Environment Management

| Team | Role | Specific Contribution |
|------|------|-----------------------|
| **ENV Teams** | GOVERNS (non-prod) | Provision and maintain dev, integration, QA, staging, UAT environments; manage refresh cadence; ensure parity with production configuration |
| **TKe Team** | GOVERNS (K8s substrate) | Manage Kubernetes clusters across all environments; enforce namespace isolation; manage environment-specific resource quotas |
| **SRE** | GOVERNS (production) | Own production environment reliability; define production acceptance criteria; manage production capacity; govern production access |
| **DB Team** | CONTRIBUTES | Manage database instances in each environment; maintain data masking for non-production; execute schema migrations per environment |
| **MW Team** | CONTRIBUTES | Manage messaging infrastructure (RabbitMQ, Kafka) in each environment; maintain environment-specific queue topology |
| **GIT Team** | CONTRIBUTES | Manage configuration delivery to each environment via config pipeline; manage Vault instances per environment |
| **Backend Teams** | CONTRIBUTES | Use and consume environments responsibly; report environment issues; participate in environment cleanup |
| **Front End Teams** | CONTRIBUTES | Consume environments for frontend testing; manage environment-specific API endpoint configuration |
| **QA Teams** | CONTRIBUTES | Define environment requirements for test execution; validate environment readiness before test runs; report environment-caused test failures separately from product defects |
| **PERF Teams** | CONTRIBUTES | Define performance environment requirements (production-parity sizing); coordinate environment availability for load test windows |
| **CYB** | CONTRIBUTES | Define security requirements per environment tier (production controls apply to production; reduced controls for dev with compensating controls) |
| **DSEC** | CONTRIBUTES | Govern data masking requirements for non-production environments; ensure PII is masked before data is used in test environments |
| **GOV** | CONTRIBUTES | Define compliance requirements for environment data handling (HIPAA, PCI scopes apply to which environments?) |
| **FW Team** | CONTRIBUTES | Manage firewall rules per environment; maintain environment isolation at network layer |
| **ILL Team** | CONTRIBUTES | Manage Illumio segmentation policies per environment; ensure dev/staging workloads cannot reach production data |
| **PO** | CONSUMES | Consumes staging/UAT for product acceptance; provides business sign-off on environment readiness |
| **BIZ** | CONSUMES | UAT environment is where business stakeholders validate before production; must be production-parity for decisions to be valid |
| **LDR** | INFORMED | Informed of environment strategy and cost; approves environment spend as part of platform budget |
| **PM** | CONTRIBUTES | Track environment availability against program schedule; flag environment unavailability as program risk |
| **MIM** | CONSUMES | Non-production environments used for incident reproduction; environment availability during major incidents |

**The three failure modes that break Environment Management**:
1. Non-production environments not refreshed regularly — teams test against stale data and stale configuration, then are surprised when production behaves differently
2. No production parity standards enforced — staging runs on 10% of production's compute, so performance and capacity issues are invisible until production
3. PII not masked in non-production environments — a compliance violation, a data security risk, and a liability that GOV and DSEC jointly own to prevent

## Discipline 6 — System Reliability Management

**What it is**: The discipline that continuously measures, governs, and improves the reliability of production systems against defined targets (SLOs), using structured processes (error budgets, toil tracking, chaos engineering, postmortems) and data (MELT, DORA, SLI signals) to drive systematic improvement.

**Why it is shared**: This is the discipline that every other discipline feeds into and that makes all the others measurable. System Reliability Management is not "what SRE does" — it is the operating model that the entire organization adopts to treat reliability as a first-class outcome alongside feature delivery. SRE governs it; every team delivers it.

**Who governs it**: SRE COE owns the governance framework — SLO methodology, error budget policy, reliability maturity model, toil standards, chaos program, postmortem process, observability standards. Every other team participates by fulfilling their reliability obligations within that framework.

### Team Participation in System Reliability Management

| Team | Role | Specific Contribution |
|------|------|-----------------------|
| **SRE** | GOVERNS | Own SLO framework; define error budget policy; govern observability standards; operate chaos program; produce reliability scorecards; reduce toil; drive reliability maturity |
| **Backend Teams** | LEADS | Define and own SLIs for their services; instrument telemetry; respond to SLO breaches; reduce application-layer toil; participate in chaos experiments |
| **Front End Teams** | LEADS | Define and own user journey SLIs; integrate RUM; participate in synthetic monitoring coverage; reduce frontend reliability gaps |
| **MW Team** | LEADS | Define and own SLIs for messaging layer (queue throughput, consumer lag, DLQ rate); instrument broker and consumer telemetry; participate in chaos experiments |
| **TKe Team** | LEADS | Define and own SLOs for the Kubernetes platform itself; maintain cluster health SLIs; participate in platform-level chaos experiments |
| **DB Team** | LEADS | Define and own SLIs for database layer (query latency, replication lag, connection pool utilization); instrument database telemetry |
| **GIT Team** | LEADS | Define and own SLOs for GitLab CI/CD, Vault, and config pipeline; treat these as production services with reliability obligations |
| **QA Teams** | CONTRIBUTES | Define test coverage SLIs (% of critical flows covered by automated tests); ensure test reliability does not mask product reliability |
| **PERF Teams** | CONTRIBUTES | Provide capacity baselines that feed SRE's capacity SLIs; validate SLO thresholds through load testing |
| **ENV Teams** | CONTRIBUTES | Define and own environment availability SLOs (non-production environments are services that QA and PERF consume) |
| **BVT Team** | CONTRIBUTES | BVT pass rate is a leading reliability indicator for each deployment; provide this signal to SRE's change correlation layer |
| **FW Team** | CONTRIBUTES | Reliability obligation: change advance notification rate, firewall-change-induced incident rate |
| **ILL Team** | CONTRIBUTES | Reliability obligation: policy onboarding lead time, segmentation-induced incident rate |
| **CYB** | CONTRIBUTES | Reliability obligation: security change advance notification rate, pen test coordination compliance |
| **DSEC** | CONTRIBUTES | Reliability obligation: DLP policy change advance notification rate, data security incident rate |
| **SNOW Team** | CONTRIBUTES | Define and own SNOW platform SLO; SNOW is a reliability-critical dependency during incidents |
| **MSFT Team** | CONTRIBUTES | Define and own MS Teams SLO; war room availability during P1s is a reliability concern |
| **RMC / CAB** | CONTRIBUTES | Reliability obligation: change record completeness rate, advance notification compliance rate |
| **MIM Team** | CONTRIBUTES | Reliability obligation: IC assignment time, postmortem completion rate, MTTR trend |
| **L12 Triage** | CONTRIBUTES | Reliability obligation: escalation quality score, false escalation rate, L1 resolution rate |
| **GOV** | CONTRIBUTES | Define regulatory SLO floors; provide compliance requirements with sufficient lead time; participate in reliability reviews |
| **PO** | CONTRIBUTES | Negotiate SLO targets with SRE; prioritize reliability investment in product backlog; honor error budget policy |
| **BIZ** | CONSUMES | Receive reliability reports translated to business impact; provide business event calendar for capacity planning |
| **LDR** | CONSUMES | Receive monthly reliability scorecard; approve reliability investment based on error budget posture; set SLO expectations for new products |
| **PM** | CONTRIBUTES | Track reliability improvement program delivery; ensure reliability action items are funded and scheduled |
| **JIRA Team** | CONTRIBUTES | Provide JIRA workflow supporting reliability work item types (toil tickets, SLO improvement work, chaos findings) |
| **Bar Raiser Call** | CONTRIBUTES | Reliability gate on new services; ensure SLO is defined before production; ensure observability is in place before launch |

## The Interlocking Model — How All Six Disciplines Work Together

The six disciplines form a closed loop. No discipline is independent. Each creates the conditions for the others to function.

```
                    ┌────────────────────────────────┐
                    │    SYSTEM RELIABILITY MGMT     │
                    │    (measures everything else)   │
                    │    SLOs / Error Budgets / DORA  │
                    └────────────┬───────────────────┘
                                 │ governs
          ┌──────────────────────┼──────────────────────┐
          │                      │                      │
          ▼                      ▼                      ▼
 ENVIRONMENT MGMT       CHANGE MGMT            RELEASE MGMT
 (where things run)     (how things change)    (when things ship)
          │                      │                      │
          │    feeds readiness    │    controls timing    │
          └──────────────────────┼──────────────────────┘
                                 │
                     ┌───────────┴───────────┐
                     │                       │
                     ▼                       ▼
            INCIDENT MGMT           PROBLEM MGMT
            (responds when          (prevents it from
             things break)           happening again)
                     │                       │
                     └───────────┬───────────┘
                                 │ feeds learning back into
                                 ▼
                    ┌────────────────────────────────┐
                    │    SYSTEM RELIABILITY MGMT     │
                    │    (SLO trend improves /        │
                    │     error budget restored)      │
                    └────────────────────────────────┘
```

### The Cascade: How a Failure in One Discipline Breaks All the Others

| Discipline that fails | First-order effect | Second-order effect | Reliability outcome |
|----------------------|--------------------|--------------------|--------------------|
| **Environment Mgmt** fails (staging ≠ prod) | Defects hidden in staging pass QA | Defects reach production | INCIDENT — SLO breach — error budget consumed |
| **Change Mgmt** fails (undocumented change) | Incident cause cannot be correlated | MTTR extends; postmortem is incomplete | PROBLEM MGMT fails — no root cause, same incident recurs |
| **Release Mgmt** fails (no SLO gate) | Bad canary reaches 100% traffic | Customer impact before rollback | INCIDENT — plus trust damage in Release process |
| **Incident Mgmt** fails (no IC, war room chaos) | Technical team handles comms + diagnosis simultaneously | MTTR extends 30–60 minutes | SLO breach deepens; error budget burn rate spikes |
| **Problem Mgmt** fails (action items not closed) | Same root cause exists in production | Same incident recurs within 30–90 days | Recurring INCIDENT — SLO trend degrades — leadership questions reliability investment |
| **System Reliability Mgmt** fails (no SLOs defined) | No baseline for "normal" | Incident, change, and release decisions are opinion-based, not data-based | All five other disciplines operate without a measurement framework — they cannot improve what they cannot measure |

| Discipline | Process Governor | Tooling Governor | Production Gate | Measurement Owner |
|-----------|----------------|-----------------|----------------|------------------|
| **Incident Mgmt** | MIM Team | SNOW Team | SRE (resolution authority) | SRE (MTTD/MTTR/MTTA) |
| **Problem Mgmt** | SRE | SNOW (Problem) + JIRA (actions) | SRE (postmortem sign-off) | SRE (recurrence rate, action closure rate) |
| **Change Mgmt** | RMC / CAB | SNOW Team | SRE (prod reliability gate) + CYB (security gate) | SRE (change-induced incident rate) |
| **Release Mgmt** | RMC Team | GIT Team | SRE (canary SLO gate) | SRE (DORA: deployment frequency, change failure rate) |
| **Environment Mgmt** | ENV Teams (non-prod) + SRE (prod) | TKe Team (K8s) | SRE (prod acceptance) | SRE (environment parity gap rate) |
| **System Reliability Mgmt** | SRE COE | Observability Platform | SRE (SLO compliance authority) | SRE (SLO trend, error budget, toil ratio, reliability maturity) |

## How the Six Disciplines Deliver Software to Business Teams

The business teams (Business Team, PO, Leadership) are the ultimate consumers of everything these disciplines produce. They don't manage these disciplines — they depend on them functioning correctly. When the disciplines work, software is delivered reliably, changes are safe, incidents are short, and the business can plan. When they break down, the business feels it first.


## The Single Most Important Insight

Every team in this organization is in the reliability business. They may not call it that. They may not think of themselves as reliability engineers. But the firewall rule they push without testing, the schema migration they run during peak hours, the DLP policy they switch to enforcement without notifying SRE, the postmortem action item they let age out in JIRA — each of these is a reliability decision.

**SRE's job is not to make all those decisions.** SRE's job is to make every team aware that they are already making those decisions, give them the framework to make them well, measure how well they make them, and report the results to leadership — so that reliability is a shared organizational commitment, not a team-level aspiration.

The six disciplines are how that commitment is operationalized. Together, they form the system that allows a platform serving millions of customers — 25 million messages a day, hundreds of microservices, dozens of teams — to function with predictability, transparency, and continuous improvement.

## Appendix — The IT Function in a Large Enterprise

### Why This Context Matters

Everything described in this document — SRE, Platform Engineering, DevOps, Kubernetes, observability — exists inside a much larger organizational structure: **the IT function of a large enterprise**. Understanding where software engineering and SRE sit within that broader structure is critical for Director/VP-level conversations. Executives, auditors, and board-level stakeholders think in terms of the IT function, not in terms of individual engineering teams.

In a large enterprise (T-Mobile scale — 75,000+ employees, regulated telecom, public company), the IT function is split into two distinct organizational philosophies that must coexist:

### ITIL and COBIT — The Framework Language of Enterprise IT

In a large enterprise, the six disciplines described in this document are recognized and formalized under two major frameworks. Understanding these frameworks lets you translate SRE concepts into enterprise IT language — which is what you need for Director/VP-level conversations.

#### ITIL (IT Infrastructure Library)

ITIL is the most widely adopted IT service management framework. The six disciplines in this document map directly to ITIL practices:

| Discipline (This Document) | ITIL Practice | ITIL Definition |
|---------------------------|--------------|----------------|
| Incident Management | Incident Management | Restore normal service operation as quickly as possible |
| Problem Management | Problem Management | Identify root causes; prevent recurrence |
| Change Management | Change Enablement | Maximize successful changes; minimize disruption |
| Release Management | Release Management | Plan and schedule release packages to production |
| Environment Management | Infrastructure and Platform Management | Manage infrastructure to meet agreed performance levels |
| System Reliability Management | Service Level Management + Continual Improvement | Define and monitor service levels; drive systematic improvement |

**How SRE extends ITIL**: ITIL defines *what* to do. SRE defines *how to engineer it*. ITIL's Incident Management says "restore service fast." SRE says "instrument the system so you detect the incident before the customer does, automate the first response, and reduce MTTD to under 2 minutes." SRE is ITIL with engineering rigor applied.

#### COBIT (Control Objectives for Information and Related Technologies)

COBIT is the governance framework for IT — used by auditors, boards, and CIOs. It maps to ITIL but at a control and governance level. SRE's relevance to COBIT:

| COBIT Domain | SRE Connection |
|-------------|---------------|
| **APO** (Align, Plan, Organize) | SLO strategy aligns reliability investment with business objectives |
| **BAI** (Build, Acquire, Implement) | Change Management, Release Management, Environment Management disciplines |
| **DSS** (Deliver, Service, Support) | Incident Management, Problem Management, System Reliability Management |
| **MEA** (Monitor, Evaluate, Assess) | SLO compliance reporting, DORA metrics, observability, reliability scorecards |

**Why this matters for Director/VP level**: When you present SRE practices to a CIO, board, or audit committee, framing them in COBIT language (specifically the DSS domain — Deliver, Service, Support) makes SRE legible to enterprise governance stakeholders. "We reduced MTTR by 40%" is an engineering metric. "We improved DSS02 (Incident Management) maturity from Level 2 to Level 4, reducing error budget consumption by 62%" is a governance metric that resonates with a CIO.

### The IT Function Reliability Model — End to End

This is the complete picture: from physical infrastructure at the bottom to business outcomes at the top, with all six disciplines operating as the connective tissue.

```
BUSINESS OUTCOMES (Revenue, compliance, customer satisfaction)
        ↑ measured by SLOs expressed in business terms
        │
BUSINESS APPLICATIONS (Product Engineering, Frontend, Backend)
        ↑ deployed via Release Management, governed by Change Management
        │
PLATFORM LAYER (TKe, GIT, MW, API Gateway, Observability)
        ↑ maintained via Environment Management, System Reliability Mgmt
        │
INFRASTRUCTURE LAYER (Network, DB, Storage, Compute, Datacenter)
        ↑ governed by ITIL/COBIT, maintained by IT Operations teams
        │
PHYSICAL / CLOUD LAYER (AWS, data centers, carrier networks)
        ↑ operated by cloud provider / DCOps / NOC
        │
EARTH (power grid, physical laws — the only things outside IT governance)

                    ←────── SECURITY ──────→
              (CYB and DSEC cut across every layer)

              ←── SIX DISCIPLINES ──→
   (Incident, Problem, Change, Release, Environment, Reliability
    management disciplines apply at every layer simultaneously)
```

**The Director/VP insight**: When a customer cannot complete a transaction on their mobile phone, the failure could exist at any layer of this stack. System Reliability Management is the discipline that makes every layer observable, measurable, and improvable — from the physical network at the bottom to the user journey SLI at the top. SRE does not own every layer. SRE makes every layer's health visible and holds every team's reliability participation accountable through the measurement framework.
