# Week 4, Day 1: VPC Networking Architecture & Cost Optimization

## 1. Why This Matters (Interview Context)

VPC networking is where infrastructure reliability meets cost control at scale. In your T-Mobile role, you're managing a 25M msg/day notification platform across availability zones—which means every networking decision compounds across service boundaries. Interviewers test VPC depth because:

1. **Cost visibility** — Gateway Endpoints vs Interface Endpoints cost 1000x differently; most senior engineers get this wrong in the design phase.
2. **Multi-AZ resilience** — Cross-AZ data transfer charges ($0.02/GB) can silently drain $50K+/year from an enterprise platform if NAT Gateway placement is wrong.
3. **Compliance & isolation** — Network ACLs, security groups, and transit routing must enforce zero-trust without breaking service discovery.
4. **Platform scaling patterns** — Moving from 1 cluster to 10+ clusters means choosing once between centralized (Transit Gateway) and distributed (peering) and living with it.

You will be asked: *"Walk me through a multi-AZ deployment where services talk across zones. What breaks when you add a third AZ? Which egress path do you choose and why?"*

## 2. Real-World Analogy: Data Center Networking & Your TIBCO Middleware Days

Think of a VPC as your data center's **network boundary**. Subnets are **fiber trunks to equipment racks**. Availability zones are **geographic buildings** (like T-Mobile's DC in Denver + Kansas City + Ashburn).

| Concept | Your Middleware Context | VPC Translation |
|---|---|---|
| **TIBCO Rendez-vous domains** | Isolated message buses; apps either joined or not | Subnets; EC2 instances either in subnet or not; routes define which traffic reaches which "bus" |
| **Middleware DMZ gateway** | Single entry point; all inbound traffic processed through IAM/logging | NAT Gateway or NLB; all outbound traffic leaves one funnel; cross-AZ traffic has one cost profile |
| **RabbitMQ cluster replication** | Nodes in different DCs; replication cost = WAN bandwidth | Cross-AZ EC2→EC2 traffic; $0.02/GB bi-directional; choose AZ placement to minimize flows |
| **Cassandra ring topology** | Nodes gossip within DC (free) vs across DCs ($$$) | Public IP peering (cross-AZ) costs; private IP (same-AZ) is free; VPC routing decides which path |
| **Change freeze governance** | "Don't touch networking in prod during business hours" | VPC Flow Logs + CloudWatch Alarms on routing anomalies = audit + alerting during change windows |

## 3. Vocabulary Translation Table

| AWS Term | Your Existing Model | Why It Matters |
|---|---|---|
| **VPC (Virtual Private Cloud)** | Isolated network namespace | Like a dedicated data center facility; you control all IPs, subnets, routing |
| **Subnet** | Tier of infrastructure (DMZ, app, data) | Layer of isolation; each has own NACL rules; associated with 1 AZ only |
| **Availability Zone (AZ)** | Geographically isolated data center | Network latency <1ms within AZ, 5–10ms cross-AZ; cross-AZ data transfer = $0.02/GB |
| **NAT Gateway** | DMZ proxy server (stateful firewall) | Converts private IP outbound traffic to public IP; one NAT GW per AZ needed for HA |
| **Network ACL** | Stateless firewall ruleset (L3/L4) | Processes **every packet** in/out; default-allow, you write deny rules; if broken, whole subnet breaks |
| **Security Group** | Stateful host firewall | Processes **established connections**; default-deny inbound; if broken, only that instance breaks |
| **Route Table** | Network routing policy (like `route` command on Linux) | Destination CIDR → target (IGW, NAT GW, VPC Peering, TGW); first matching route wins |
| **Internet Gateway** | DMZ entry gate | Provides public IP ↔ internet path; only targets can have public IPs are those in subnets with IGW routes |
| **VPC Peering** | Direct fiber link between two DCs | Point-to-point connectivity; no transitive routing (A→B→C doesn't work); ~10 peering connections before pain |
| **Transit Gateway (TGW)** | Central network hub (like a managed switch) | Scales peering to 1000+ connections; transitive routing works; higher cost but >5 clusters = better deal |
| **Gateway Endpoint** | Host-local proxy to AWS service | Free (for S3/DynamoDB); no internet; traffic never leaves AWS backbone |
| **Interface Endpoint (PrivateLink)** | AWS-managed VPN tunnel to service | $0.01/AZ/hr ($7.20/month/AZ minimum); use only if compliance requires private connectivity |
| **VPC Flow Logs** | Network packet capture (tcpdump) | All IP flows logged to CloudWatch; enables cost analysis + security audits |
| **Egress-only Internet Gateway** | IPv6-only DMZ gate | Like IGW but **outbound only** for IPv6; useful for Kubernetes egress policies |

## 4. The 30-Second Version

**What**: A VPC is an isolated network namespace. You define subnets (which AZ), routing (where traffic goes), and egress points (NAT GW or IGW). Cost is driven by cross-AZ data transfer ($0.02/GB).

**Why it matters**: Choosing the wrong NAT Gateway placement costs $50K+/year in cross-AZ charges. Choosing Gateway Endpoints over Interface Endpoints saves ~$100K/year.

**When you'll use it**: Every AWS architecture uses VPCs; multi-AZ systems must account for egress costs.

---

## 5. The 2-Minute Version

A **VPC** is a software-defined network boundary. Inside, you define:

1. **Subnets** — Each tied to 1 AZ. Public subnets route to Internet Gateway (IGW). Private subnets do not.
2. **Routing** — Route tables define: "traffic destined for 0.0.0.0/0 (internet) goes to NAT Gateway". Longest prefix match wins.
3. **Egress design** — Choose **one**:
   - **NAT Gateway** (per-AZ): EC2 in private subnet → NAT GW in public subnet → IGW → internet. Cost: $45/month per NAT GW + $0.045/GB processed.
   - **Gateway Endpoint** (free): EC2 in private → S3 or DynamoDB **without leaving VPC**. Uses prefix-list routes. **Zero data transfer cost**. Trade: only works for S3/DynamoDB/DynamoDB Streams/SNS.
4. **Multi-AZ implications**:
   - EC2 in us-east-1a talking to EC2 in us-east-1b = **$0.02/GB per direction** (unless both are on same network path, which they rarely are in distributed systems).
   - If all egress flows to one NAT GW in us-east-1a, all cross-AZ traffic = $0.02/GB. **This is the trap**.
   - Solution: One NAT GW per AZ. Route local AZ traffic to local NAT GW.

**Interview setup**: "Walk me through what happens when a pod in us-east-1a queries DynamoDB in us-east-1b."
- Route: pod subnet → NAT GW in local AZ → IGW → DynamoDB endpoint (actually reaches AWS internal network, not internet).
- Cross-AZ? DynamoDB is managed; traffic to it from us-east-1a hitting us-east-1b endpoint = $0.02/GB.
- Fix: Use Gateway Endpoint for DynamoDB (free, routes directly without cross-AZ hop).

## 6. The 5-Minute Deep Version

### VPC Architecture (Typical Enterprise Pattern)

```
VPC CIDR: 10.0.0.0/16

Public Subnets (IGW routes):
  us-east-1a: 10.0.1.0/24  → Route: 0.0.0.0/0 → IGW
  us-east-1b: 10.0.2.0/24  → Route: 0.0.0.0/0 → IGW

Private Subnets (App tier):
  us-east-1a: 10.0.11.0/24 → Route: 0.0.0.0/0 → NAT-GW-1a
  us-east-1b: 10.0.12.0/24 → Route: 0.0.0.0/0 → NAT-GW-1b

Private Subnets (Data tier):
  us-east-1a: 10.0.21.0/24 → Route: 0.0.0.0/0 → NAT-GW-1a (if DB requires egress)
  us-east-1b: 10.0.22.0/24 → Route: 0.0.0.0/0 → NAT-GW-1b
```

### Egress Flow Examples

**Scenario 1: App in 1a wants to call DynamoDB API**
- Without Gateway Endpoint:
  - Route: app (10.0.11.x) → NAT-GW-1a (10.0.1.x, gets EIP) → IGW → DynamoDB service IP in AWS network.
  - If DynamoDB resolves to us-east-1b endpoint IP, traffic crosses AZ = $0.02/GB charged.
  - Cost for 1GB = $0.02 (one-way); if app polls every second (bidirectional), that's $0.02 * 86400 * 365 ≈ $630K/year per app.
  
- With Gateway Endpoint (recommended):
  - Route: app (10.0.11.x) → Gateway Endpoint (prefix-list) → DynamoDB (AWS backbone, no internet).
  - Explicitly stays in us-east-1a; cross-AZ routing **does not happen**.
  - Cost: $0. Savings: $630K/year per app.

**Scenario 2: App in 1a calls S3 in same AZ**
- Without Gateway Endpoint:
  - App → NAT-GW-1a → IGW → S3 service endpoint → potentially crosses AZ inside AWS backbone.
  - Cost: $0.045/GB (NAT GW processing) + potential $0.02/GB (cross-AZ).
  
- With Gateway Endpoint:
  - App → Gateway Endpoint → S3 (local AZ).
  - Cost: $0. Savings: ~$0.07/GB per egress flow.

### Cost Multiplier: T-Mobile Notification Platform Scale

Your platform: 25M messages/day, ~290 msg/sec peak, multi-service architecture (dispatcher, validator, retry engine, analytics, etc.).

**Conservative estimate per service per day**:
- 290 msg/sec × 3 DynamoDB calls/msg (state check, update, audit) = 870 DDB calls/sec.
- 870 calls/sec × 100 bytes response = 87 KB/sec = 7.5 GB/day per DynamoDB caller.
- 3 services calling DDB = 22.5 GB/day.

**Without Gateway Endpoint (cross-AZ)**:
- 22.5 GB/day × $0.02/GB = $0.45/day = $164/year.

**At 100-service enterprise scale** (typical year 4–5 growth):
- 22.5 GB/day × 100 services / 3 = 750 GB/day of internal AWS API calls.
- 750 GB/day × $0.02/GB = $15/day = $5,475/year in avoidable cross-AZ charges.
- **Plus NAT GW costs**: 3 NAT GWs × $45/month × 12 = $1,620/year.
- **Total avoidable**: ~$7K/year from Gateway Endpoints + correct subnet placement.

At a 500-person engineering org, that's $14/person/year savings—trivial monetarily, but **massive credibility** in a cost-conscious director interview: *"We reduced avoidable cross-AZ charges by $7K/year in the first pass, which funded our observability migration."*

## 7. Director / VP Lens: Governance, Scale, & AI/Platform Trends

**As a team lead**, you govern VPC design through:

1. **Policy**: "All databases use Gateway Endpoints for managed services. Multi-AZ apps route local AZ egress to local NAT GW."
   - Saves design cycles; eliminates the "forgot to cost-model cross-AZ" bug in 80% of projects.
   
2. **Naming & automation** — NAT GW names include AZ: `nat-gw-{environment}-{az}` so devs can't accidentally use the wrong one. Route tables named by tier: `rt-{env}-private-app-{az}`.

3. **Measurement** — VPC Flow Logs → CloudWatch Insights query for "unexpected cross-AZ flows". Alert if Bytes crossing AZ exceed threshold (e.g., >100GB/day for given subnet pair).

4. **Cost attribution** — Charge back $0.02/GB cross-AZ to owning team if service doesn't justify it. "Your notification retry engine is costing us $400/month in cross-AZ data transfer. Let's redesign the queuing topology."

5. **AI/platform engineering integration** — As IDP matures:
   - Golden path default: "create app" → auto-places replicas in all 3 AZs + local NAT GW routes.
   - Backstage template enforces subnet selection; rejects private-subnets-without-NAT patterns at `terraform apply` time.
   - Cost forecast: Backstage.io can forecast AZ routing cost given service replica count + data pattern; alerts PMs if cost grows outside budget.

## 8. Comprehension Check (Must Pass 2/3 Before Lab)

**Question 1: NACL Statelessness Trap**
You create a NACL rule: "Allow inbound traffic on port 443 from 0.0.0.0/0". Your application receives HTTPS requests successfully. However, the **response** packets are blocked. Why? What do you add to fix it?

*Hint: Think "stateless" — every packet direction must have an explicit rule.*

<details>
<summary>Answer</summary>

**Why**: NACLs are stateless. Inbound rule allows client→server (client IP:ephemeral_port → server IP:443). But server→client (server IP:443 → client IP:ephemeral_port) is the **opposite direction**, which needs a separate **outbound** NACL rule.

**Fix**: Add outbound rule "Allow all traffic to 0.0.0.0/0 ports 1024–65535 (ephemeral range)". Or use the AWS default NACL which has this pre-configured.

**Interview follow-up**: "Why does this not break with security groups?" Because security groups ARE stateful—they track established connections and auto-allow responses.

**Your platform context**: This is why your incident response team never touches NACLs during production incidents; a misconfiguration breaks the entire subnet's egress, not just one instance.

</details>

**Question 2: Gateway Endpoint Cost Math**
You have 3 AZs, each with a private subnet running 10 EC2 instances. All instances query S3 for object storage (10 GB/instance/day, so 300 GB/day total). You have two egress options:

A) NAT Gateway in each public subnet, instances route 0.0.0.0/0 → NAT-GW → IGW → S3.
B) One Gateway Endpoint for S3, instances route S3 CIDR prefix-list → Gateway Endpoint.

Calculate monthly cost for each. Which do you choose? Why?

*Hint: NAT GW = $45/month + $0.045/GB. Gateway Endpoint = free for S3.*

<details>
<summary>Answer</summary>

**Option A (NAT GW)**:
- 3 NAT GWs × $45/month = $135/month.
- 300 GB/day × 30 days = 9,000 GB/month × $0.045/GB = $405/month.
- **Total: $540/month**.

**Option B (Gateway Endpoint)**:
- 1 Gateway Endpoint × $0/month (S3 is free).
- **Total: $0/month**.

**Savings**: $540/month = $6,480/year.

**Choose**: B (Gateway Endpoint).

**Why**: S3 is one of the few AWS services with free Gateway Endpoints. The trade-off is that Gateway Endpoints only work for S3, DynamoDB, SQS, SNS (limited scope); for most other services (EC2, RDS, Secrets Manager), you must use Interface Endpoints ($0.01/AZ/hour = $7.20/month minimum per AZ = $21.60/month for 3 AZs, plus $0.04/GB data). That's why multi-layered egress design exists: use Gateway Endpoint for S3/DDB, Interface Endpoint only for what requires it, NAT GW only for miscellaneous internet access.

**Your platform at T-Mobile**: If your notification platform stores 100+ GB/day in S3 (event logs, audit trails, dashboards), Gateway Endpoint is non-negotiable. Cost avoidance = $1,800/month = $21,600/year.

</details>

**Question 3: Transit Gateway vs Peering at Scale**
You're designing a multi-cluster Kubernetes environment:
- Today: 1 EKS cluster (10 nodes) in VPC-A.
- Year 1 goal: 3 clusters (separate VPCs) for workload isolation.
- Year 3 goal: 10 clusters (separate VPCs) for cost allocation and team isolation.

For every cluster pair (A↔B, A↔C, etc.), services must talk. Which connectivity model do you choose at Year 0, and why? What breaks if you choose wrong?

<details>
<summary>Answer</summary>

**Year 0 choice**: **Transit Gateway (not VPC Peering)**.

**Why**:
- VPC Peering is 1:1 connection; 10 clusters = 10 × 9 / 2 = 45 peering connections.
  - AWS limit: 50 peering connections per VPC (you hit ceiling at ~10 clusters; not scalable).
  - Manual routing in each VPC for each peer (route-table hell).
  - No transitive routing: A→B→C doesn't work; you need explicit A→C peering.
  
- Transit Gateway scales to 5,000 connections per TGW, transitive routing built-in (A→B→C works via TGW).
  - Each cluster VPC attaches to TGW once; TGW routes between all.
  - Cost: TGW = $0.05/hour (~$36/month) + $0.02/GB processed. Peering = free, but unsustainable past 10 clusters.
  - **Break-even**: TGW cost = Peering cost when you have ~5 clusters (each needs more manual routing and ops overhead); TGW pays for itself in velocity and ops.

**What breaks if you choose Peering today**:
- Year 2 reorg: "We need a new cluster for team-X." You add cluster-11. Now you need 9 new peering connections (to every existing cluster). Route tables in 10 VPCs need updates. Change window expands. You learn this lesson at 2 AM during incident.
- Year 3 multi-region: "We need clusters in eu-west-1 too." Inter-region peering (now possible) has $0.02/GB cost. TGW with inter-region attachment = same $0.02/GB but unified routing. You chose Peering in Year 0 and now you're re-architecting Kubernetes networking.

**Correct answer at interview**: "At Year 0, you don't know if you'll have 3 or 30 clusters. TGW costs $36/month upfront, scales to 5000 clusters, and reduces ops. VPC Peering is free but costs you engineering time at scale. I choose TGW because Year 3-Vishweshwar will thank Year 0-Vishweshwar."

**Your T-Mobile context**: A 25M msg/day platform is built on shared infrastructure. At Year 5, you'll have 15+ clusters across 3–4 VPCs for fault isolation and billing. Choosing TGW in Week 4 makes this inevitable without re-engineering later.

</details>

### Step 1: Create VPC and Subnets

```bash
#!/bin/bash
# Step 1: Create VPC and public/private subnets in 3 AZs

set -e
export AWS_REGION=us-east-1
export LAB_NAME="vpc-lab-$(date +%s)"
export VPC_CIDR="10.200.0.0/16"

echo "[Step 1] Creating VPC and subnets..."

# Create VPC
VPC_ID=$(aws ec2 create-vpc \
  --cidr-block $VPC_CIDR \
  --tag-specifications "ResourceType=vpc,Tags=[{Key=Name,Value=$LAB_NAME-vpc}]" \
  --region $AWS_REGION \
  --query 'Vpc.VpcId' \
  --output text)

echo "VPC created: $VPC_ID"
export VPC_ID

# Enable DNS in VPC (required for Gateway Endpoints to work)
aws ec2 modify-vpc-attribute \
  --vpc-id $VPC_ID \
  --enable-dns-hostnames \
  --region $AWS_REGION

# Create Internet Gateway
IGW_ID=$(aws ec2 create-internet-gateway \
  --tag-specifications "ResourceType=internet-gateway,Tags=[{Key=Name,Value=$LAB_NAME-igw}]" \
  --region $AWS_REGION \
  --query 'InternetGateway.InternetGatewayId' \
  --output text)

aws ec2 attach-internet-gateway \
  --vpc-id $VPC_ID \
  --internet-gateway-id $IGW_ID \
  --region $AWS_REGION

echo "Internet Gateway created and attached: $IGW_ID"
export IGW_ID

# Create subnets in 3 AZs
# Public subnets: 10.200.1.0/24 (1a), 10.200.2.0/24 (1b), 10.200.3.0/24 (1c)
# Private app subnets: 10.200.11.0/24 (1a), 10.200.12.0/24 (1b), 10.200.13.0/24 (1c)
# Private data subnets: 10.200.21.0/24 (1a), 10.200.22.0/24 (1b), 10.200.23.0/24 (1c)

declare -A PUBLIC_SUBNETS
declare -A PRIVATE_APP_SUBNETS
declare -A PRIVATE_DATA_SUBNETS
declare -a AZS=("us-east-1a" "us-east-1b" "us-east-1c")

for i in 0 1 2; do
  AZ=${AZS[$i]}
  AZ_SHORT=$(echo $AZ | sed 's/.*\(.\)$/\1/')  # Extract 'a', 'b', or 'c'
  
  # Public subnet
  PUB_SUBNET=$(aws ec2 create-subnet \
    --vpc-id $VPC_ID \
    --cidr-block "10.200.$((i+1)).0/24" \
    --availability-zone $AZ \
    --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=$LAB_NAME-public-$AZ_SHORT}]" \
    --region $AWS_REGION \
    --query 'Subnet.SubnetId' \
    --output text)
  PUBLIC_SUBNETS[$AZ]=$PUB_SUBNET
  
  # Private app subnet
  PRIV_APP_SUBNET=$(aws ec2 create-subnet \
    --vpc-id $VPC_ID \
    --cidr-block "10.200.$((i+11)).0/24" \
    --availability-zone $AZ \
    --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=$LAB_NAME-private-app-$AZ_SHORT}]" \
    --region $AWS_REGION \
    --query 'Subnet.SubnetId' \
    --output text)
  PRIVATE_APP_SUBNETS[$AZ]=$PRIV_APP_SUBNET
  
  # Private data subnet
  PRIV_DATA_SUBNET=$(aws ec2 create-subnet \
    --vpc-id $VPC_ID \
    --cidr-block "10.200.$((i+21)).0/24" \
    --availability-zone $AZ \
    --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=$LAB_NAME-private-data-$AZ_SHORT}]" \
    --region $AWS_REGION \
    --query 'Subnet.SubnetId' \
    --output text)
  PRIVATE_DATA_SUBNETS[$AZ]=$PRIV_DATA_SUBNET
  
  echo "[$AZ] Public: $PUB_SUBNET | Private App: $PRIV_APP_SUBNET | Private Data: $PRIV_DATA_SUBNET"
done

# Export for next steps
export PUBLIC_SUBNETS_JSON=$(python3 -c "import json; print(json.dumps(${!PUBLIC_SUBNETS[@]}))")
```

### Step 2: Create NAT Gateways (One Per AZ)

```bash
#!/bin/bash
# Step 2: Create NAT Gateway in each public subnet (one per AZ)

echo "[Step 2] Creating NAT Gateways..."

# We need to preserve subnet IDs from Step 1. For this lab, we'll re-fetch them.
VPC_ID="vpc-0abc12def3456ghi7"  # From Step 1 output
AWS_REGION="us-east-1"

# Get subnet IDs
PUBLIC_SUBNETS=$(aws ec2 describe-subnets \
  --filters "Name=vpc-id,Values=$VPC_ID" "Name=tag:Name,Values=vpc-lab-*-public-*" \
  --region $AWS_REGION \
  --query 'Subnets[*].[AvailabilityZone, SubnetId]' \
  --output text)

declare -A SUBNET_BY_AZ
while read -r AZ SUBNET_ID; do
  SUBNET_BY_AZ[$AZ]=$SUBNET_ID
done <<< "$PUBLIC_SUBNETS"

declare -A NAT_GW_IDS
declare -a AZS=("us-east-1a" "us-east-1b" "us-east-1c")

for AZ in "${AZS[@]}"; do
  SUBNET_ID=${SUBNET_BY_AZ[$AZ]}
  AZ_SHORT=$(echo $AZ | sed 's/.*\(.\)$/\1/')
  
  # Allocate Elastic IP for NAT Gateway
  EIP=$(aws ec2 allocate-address \
    --domain vpc \
    --tag-specifications "ResourceType=elastic-ip,Tags=[{Key=Name,Value=vpc-lab-eip-$AZ_SHORT}]" \
    --region $AWS_REGION \
    --query 'AllocationId' \
    --output text)
  
  # Create NAT Gateway
  NAT_GW=$(aws ec2 create-nat-gateway \
    --subnet-id $SUBNET_ID \
    --allocation-id $EIP \
    --tag-specifications "ResourceType=nat-gateway,Tags=[{Key=Name,Value=vpc-lab-nat-$AZ_SHORT}]" \
    --region $AWS_REGION \
    --query 'NatGateway.NatGatewayId' \
    --output text)
  
  NAT_GW_IDS[$AZ]=$NAT_GW
  echo "[$AZ] NAT Gateway created: $NAT_GW (EIP: $EIP)"
  
  # NAT Gateway takes 1-2 minutes to become available; wait here
  echo "  Waiting for NAT Gateway to become available..."
  aws ec2 wait nat-gateway-available \
    --nat-gateway-ids $NAT_GW \
    --region $AWS_REGION
  echo "  NAT Gateway is now available."
done

# Export NAT Gateway IDs for next steps
export NAT_GW_IDS_JSON=$(echo "${NAT_GW_IDS[@]}" | jq -R 'split(" ") | map(select(length > 0))')
```

### Step 3: Create Route Tables (Public & Private)

```bash
#!/bin/bash
# Step 3: Create route tables: public (→ IGW) and private (→ NAT GW per AZ)

echo "[Step 3] Creating route tables..."

VPC_ID="vpc-0abc12def3456ghi7"
IGW_ID="igw-01234567890abcdef"
AWS_REGION="us-east-1"
LAB_NAME="vpc-lab-1717948234"  # Replace with actual from Step 1

# Create public route table (routes 0.0.0.0/0 → IGW)
PUBLIC_RT=$(aws ec2 create-route-table \
  --vpc-id $VPC_ID \
  --tag-specifications "ResourceType=route-table,Tags=[{Key=Name,Value=$LAB_NAME-public-rt}]" \
  --region $AWS_REGION \
  --query 'RouteTable.RouteTableId' \
  --output text)

aws ec2 create-route \
  --route-table-id $PUBLIC_RT \
  --destination-cidr-block 0.0.0.0/0 \
  --gateway-id $IGW_ID \
  --region $AWS_REGION

echo "Public route table created: $PUBLIC_RT (0.0.0.0/0 → IGW)"
export PUBLIC_RT

# Create private route tables (one per AZ, routes 0.0.0.0/0 → local NAT GW)
declare -A NAT_GW_BY_AZ
declare -A PRIVATE_RT_BY_AZ

# Fetch NAT Gateway IDs by AZ (from AWS)
NAT_GW_DATA=$(aws ec2 describe-nat-gateways \
  --filters "Name=vpc-id,Values=$VPC_ID" \
  --region $AWS_REGION \
  --query 'NatGateways[*].[SubnetId, NatGatewayId]' \
  --output text)

# Build map of NAT GW by AZ
while read -r SUBNET_ID NAT_GW; do
  AZ=$(aws ec2 describe-subnets --subnet-ids $SUBNET_ID --region $AWS_REGION --query 'Subnets[0].AvailabilityZone' --output text)
  NAT_GW_BY_AZ[$AZ]=$NAT_GW
done <<< "$NAT_GW_DATA"

declare -a AZS=("us-east-1a" "us-east-1b" "us-east-1c")

for AZ in "${AZS[@]}"; do
  AZ_SHORT=$(echo $AZ | sed 's/.*\(.\)$/\1/')
  NAT_GW=${NAT_GW_BY_AZ[$AZ]}
  
  # Create private route table for this AZ
  PRIVATE_RT=$(aws ec2 create-route-table \
    --vpc-id $VPC_ID \
    --tag-specifications "ResourceType=route-table,Tags=[{Key=Name,Value=$LAB_NAME-private-rt-$AZ_SHORT}]" \
    --region $AWS_REGION \
    --query 'RouteTable.RouteTableId' \
    --output text)
  
  # Route 0.0.0.0/0 to local NAT Gateway
  aws ec2 create-route \
    --route-table-id $PRIVATE_RT \
    --destination-cidr-block 0.0.0.0/0 \
    --nat-gateway-id $NAT_GW \
    --region $AWS_REGION
  
  PRIVATE_RT_BY_AZ[$AZ]=$PRIVATE_RT
  echo "[$AZ] Private route table created: $PRIVATE_RT (0.0.0.0/0 → $NAT_GW)"
done

export PRIVATE_RT_BY_AZ_JSON=$(python3 -c "import json; print(json.dumps(${PRIVATE_RT_BY_AZ[@]}))")
```

### Q1: "You're architecting a multi-AZ Kubernetes cluster. Pods in us-east-1a need to call a DynamoDB table. How do you route that traffic, and what's the cost implication?"

**Basic Answer** (30 seconds):
"Pods use the VPC CNI plugin to get private IPs. They route to DynamoDB via the NAT Gateway in their AZ, then to DynamoDB endpoint. If DynamoDB endpoint is in 1b, traffic crosses AZ = $0.02/GB."

**Intermediate Answer** (2 minutes):
"Best practice: Use a Gateway Endpoint for DynamoDB (free) instead of routing through NAT GW. This keeps traffic on AWS backbone, no cross-AZ charge. Route tables use prefix-list routing:
- Route: DynamoDB prefix-list → VPC Endpoint.
- Route: 0.0.0.0/0 → NAT GW (for non-DynamoDB internet).
At T-Mobile scale (25M msg/day, 3 DDB calls/msg), that's 870 calls/sec = 7.5 GB/day. Without Gateway Endpoint: 7.5 GB/day × $0.02/GB = $0.15/day = $55/year per pod. With 10 pods calling DDB = $550/year saved. Across 5 clusters = $2,750/year."

**Advanced Answer** (5 minutes):
"The full picture: Gateway Endpoints are free for S3 and DynamoDB, but only work for those two services. For everything else (Secrets Manager, EC2, RDS, etc.), you have two choices:

1. **Interface Endpoint (PrivateLink)**: $0.01/AZ/hour minimum = $7.20/month per AZ per service. If you use Secrets Manager, EC2 API, and RDS API across 3 AZs, that's 3 services × $7.20/month × 3 AZs = $64.80/month just in PrivateLink fees. Trade: traffic never leaves VPC, compliance-friendly.

2. **NAT Gateway**: $45/month + $0.045/GB data. If your app makes 100GB/month of Secrets Manager calls, that's $45 + (100 × $0.045) = $49.50/month. Trade: traffic crosses NAT GW (stateless packet inspection), slightly higher latency, but cheaper for low-volume APIs.

The correct design: **layered egress**. Use Gateway Endpoint for S3 and DynamoDB (free). Use Interface Endpoints ONLY for services that have compliance requirements (Secrets Manager, if your policy requires private connectivity). Use NAT GW for everything else (EC2 API, CloudWatch, etc.).

At director level, I'd audit all service API calls by destination, calculate volume, and choose endpoints per-service to minimize cost. For example:
- S3: Gateway Endpoint ($0).
- DynamoDB: Gateway Endpoint ($0).
- Secrets Manager: Interface Endpoint (used at startup only, low volume, justifies $7.20/month).
- EC2 API: NAT GW (low volume, mixed payload, not worth PrivateLink fee).

This decision should be documented in your infrastructure-as-code (Terraform) so the ops team doesn't second-guess it later. Cost avoidance: ~$200K/year at enterprise scale."

### Q2: "You have a disaster recovery site in a second region (us-west-2). How do you connect VPCs across regions, and what breaks if you choose the wrong model?"

**Basic Answer** (30 seconds):
"Use Inter-Region VPC Peering or Transit Gateway with inter-region attachment. Peering is point-to-point; TGW is hub-and-spoke. Both cost $0.02/GB cross-region data transfer."

**Intermediate Answer** (2 minutes):
"If you have 2 VPCs (primary, DR) in different regions:
- **Option A: VPC Peering** — Create peering connection us-east-1-vpc ↔ us-west-2-vpc. Routes in each VPC point to the peering connection. Cost: free (no per-connection fee, but $0.02/GB data applies).
- **Option B: Transit Gateway (inter-region)** — Attach each VPC to a regional TGW; attach regional TGWs to each other via inter-region peering. Cost: $0.02/GB same (bandwidth), plus $0.05/hour per TGW per region = $36/month per TGW.

You choose TGW if:
- You have >1 VPC in each region (primary has 3 VPCs for workloads; DR has 2 VPCs for standby). TGW simplifies routing.
- You plan to add more VPCs later (TGW scales; peering becomes a mess at 10+ peering connections).

You choose Peering if:
- Exactly 2 VPCs, exactly 2 regions. Peering is simpler, fewer moving parts.

**What breaks**: If you choose Peering today with 2 VPCs, and later add a 3rd VPC (for a new team's workload), you now need 3 more peering connections (primary-new, DR-new, etc.). Someone updates a route table and forgets the DR peering route → DR traffic fails silently for 8 hours → incident at 2 AM."

**Advanced Answer** (5 minutes):
"The real decision tree:

1. **At Year 0** (today): Peering works fine if both regions have exactly 1 VPC each. TGW adds overhead ($72/month for 2 TGWs).

2. **At Year 2** (growth): You've added EKS clusters per region (3 VPCs per region now). Peering model = 3 × 3 / 2 × 2 regions = 9 peering connections minimum. Route management in 6 VPCs becomes error-prone.

3. **At Year 4** (multi-tenant): You have 10+ VPCs per region (per-team isolation, per-service isolation, compliance). Peering is now unmaintainable; you've re-architected to TGW.

The cost-benefit math:
- Peering: $0 (no monthly fee). Pain: route table sprawl, ops errors, incident response time (who owns the DR peering connection?).
- TGW: $72/month (2 regional TGWs). Benefit: unified routing, transitive connectivity, standardized CIDR planning.
- At Year 3, peering ops costs (two incidents, rework of route tables, on-call engineer time) exceed TGW cost. You should have chosen TGW at Year 0.

From a **governance / team structure** perspective:
- **Peering model**: Requires a "network architect" person who owns all peering connections and route tables. Single point of failure; that person on vacation = routing changes stall.
- **TGW model**: Networking rules are centralized in one artifact (TGW attachments, transit routes). Network team owns TGW; app teams own their VPC CIDR blocks. Cleaner separation of concerns.

I'd recommend TGW for multi-region DR because it future-proofs the decision and reduces on-call toil. If the CTO asks 'why spend $72/month?', the answer is: 'Year 3 incident prevention. When the DR site fails to connect in a real DR test, we'll save weeks of rework that costs way more than $72/month.'"

### Q3: "Walk me through a design where services in EKS need private connectivity to RDS without traversing the internet. There's a compliance requirement: no traffic leaves the AWS backbone. How do you achieve this?"

**Basic Answer** (30 seconds):
"Put RDS and EKS in the same VPC (or peered VPCs). Use security groups to allow EKS pods to connect to RDS on port 3306 (MySQL) or 5432 (Postgres). No internet needed."

**Intermediate Answer** (2 minutes):
"More detail:

1. **Same VPC** (simplest):
   - RDS DB subnet group: private subnets us-east-1a, us-east-1b, us-east-1c.
   - EKS nodes: same private subnets (or same VPC, different subnets).
   - Security group: EKS nodes SG → RDS SG (port 3306 inbound to RDS).
   - No NAT GW involved; no cross-VPC peering. Traffic stays within VPC.
   - Cost: $0 (networking). Trade: EKS and RDS share the same VPC blast radius (if security group is misconfigured, all pods can hit RDS).

2. **Separate VPCs** (better isolation):
   - EKS VPC (10.0.0.0/16), RDS VPC (10.1.0.0/16).
   - VPC Peering between them (or TGW if you have >2 VPCs).
   - Route: EKS pods → peering connection → RDS.
   - Security groups: EKS node SG → RDS SG (same logic as above).
   - Cost: Free (peering). Trade: Operational complexity (two VPCs to manage).

3. **RDS in different VPC + Interface Endpoint** (overcomplicated):
   - RDS is in VPC-B. Create VPC endpoint for RDS from VPC-A.
   - This is **not a common pattern** because RDS doesn't support Interface Endpoints directly. You'd have to proxy RDS through an EC2 instance, which defeats the purpose.
   - **Skip this option.**

Best practice for your scenario: Separate VPCs (EKS-VPC and RDS-VPC) connected via peering. Security groups isolate access. All traffic is private (no internet)."

**Advanced Answer** (5 minutes):
"The **compliance angle** is crucial here. 'No traffic leaves AWS backbone' typically means:
- No traffic goes through an Internet Gateway (IGW) or NAT Gateway.
- All traffic stays within VPC or peered/transited VPCs.
- No asymmetric routing (traffic out via one path, in via another).

Here's the full design for T-Mobile scale (25M msg/day, multi-region DR):

**Primary Region (us-east-1)**:
```
EKS-VPC (10.0.0.0/16, 3 AZs)
  ├── Public subnets (10.0.1–3.0/24) — EKS control plane (managed)
  └── Private subnets (10.0.11–13.0/24) — EKS worker nodes, pods

RDS-VPC (10.1.0.0/16, 3 AZs)
  └── Private subnets (10.1.1–3.0/24) — RDS Multi-AZ

VPC Peering: EKS-VPC ↔ RDS-VPC
  Route (EKS-VPC private RT): 10.1.0.0/16 → peering-connection
  Route (RDS-VPC private RT): 10.0.0.0/16 → peering-connection
  Security group (RDS): Inbound 3306 from EKS-VPC (10.0.0.0/16)
```

**Data flow**:
1. Pod in EKS-VPC (10.0.11.x) needs to query RDS (10.1.1.x, port 3306).
2. Pod's route table match: 10.1.1.x destined for 10.1.0.0/16 → peering-connection.
3. Peering connection carries traffic to RDS-VPC private subnet.
4. RDS security group validates: 10.0.11.x:ephemeral → 10.1.1.x:3306. Allowed.
5. RDS processes query.
6. Response: 10.1.1.x:3306 → 10.0.11.x:ephemeral (security group is stateful, allows response).
7. Traffic never touches IGW, NAT GW, or internet.

**Cost**: $0 (VPC peering is free; RDS costs are separate).

**Compliance benefits**:
- ✓ Traffic audit-able via VPC Flow Logs (all flows in CloudWatch).
- ✓ No egress to internet (IGW not involved).
- ✓ Data residency compliance: all traffic stays within VPC region.
- ✓ Network ACLs can be restrictive (explicit allow 3306 only) without affecting internet traffic.

**Scaling to multi-region** (if you have DR):
- Use Transit Gateway for transitive routing: EKS-us-east-1 ↔ TGW-us-east-1 ↔ RDS-us-east-1, plus TGW-us-east-1 ↔ TGW-us-west-2 ↔ RDS-us-west-2.
- Cost: 2 TGWs × $0.05/hr = $36/month per region. Cross-region TGW attachment incurs $0.02/GB for failover traffic.
- Benefit: Unified routing policy; easier to add new VPCs.

From an **operations** perspective, I'd automate this with Terraform:
- Module for EKS-VPC (standard config).
- Module for RDS-VPC (standard config).
- Peering module (creates connection + updates both route tables).
- This ensures every environment (dev, staging, prod) follows the same pattern. No manual routing drift."

### Q4: "Your organization is moving from a single-VPC architecture (all 50 services in one VPC) to multi-VPC (each service team gets their own VPC). How do you design the network to allow service-to-service calls while preventing accidental cross-service data access?"

**Basic Answer** (30 seconds):
"Each service team gets a VPC. Use VPC peering or Transit Gateway to allow inter-VPC traffic. Security groups / NACLs enforce access control (only allow service-A to call service-B on port 8080, not vice versa)."

**Intermediate Answer** (2 minutes):
"Design:

1. **Shared Services VPC** — Databases (RDS), caches (ElastiCache), logging (CloudWatch, Splunk), auth (Vault, Okta) run here. All service VPCs peer to it.

2. **Service VPCs** — Each service team gets their own VPC for their app + workers. Apps in service-VPC-A call Shared Services (RDS, Vault) via peering.

3. **Access control**:
   - Security group: Each VPC has a "database-access" SG. Only service pods that need DB access are tagged with this SG. Inbound rule in Shared VPC RDS SG: allow "database-access" SG from any peering connection.
   - NACL: Stricter; Layer 3 control. Example: "Allow inbound port 3306 from service-VPC-A only (10.0.0.0/16), deny from service-VPC-B (10.1.0.0/16)".

4. **Routing**:
   - Service-VPC-A route table: 10.99.0.0/16 (Shared VPC) → peering-AB.
   - Service-VPC-B route table: 10.99.0.0/16 (Shared VPC) → peering-BC.
   - Shared VPC route tables: 10.0.0.0/16 (service-A) → peering-AB, 10.1.0.0/16 (service-B) → peering-BC.

5. **Scaling challenge**: At 50 services, you have 50 × 49 / 2 = 1,225 potential peering connections if every service talks to every other service. This is unmaintainable. Instead, use a **hub-and-spoke model** (below).

**Operational risk**: A misconfigured security group in Service-VPC-A allows its pods to access Service-VPC-B's database (if they peer and routes exist). This is why you need NACLs and explicit allow-lists per VPC pair."

**Advanced Answer** (5 minutes):
"The **Hub-and-Spoke (Transit Gateway) Model**:

```
                    ┌─────────────────┐
                    │  Transit GW     │
                    │  (Central Hub)  │
                    └────────┬────────┘
           ┌────────────────┼────────────────┐
           │                │                │
      Service-VPC-A    Shared-VPC      Service-VPC-B
      (10.0.0.0/16)    (10.99.0.0/16)   (10.1.0.0/16)
```

**Why TGW beats peering at scale**:
- **Peering**: Service-A ↔ Shared. Service-B ↔ Shared. Service-A ↔ Service-B (direct, if they talk often).
  - If Service-A calls Service-C through Shared (database), traffic does NOT route through Shared automatically; you need explicit A↔Shared and C↔Shared peering, then logic in app to call Shared as intermediary.
  - At 50 services: ~50–100 peering connections. Each one is an AWS API call to maintain.
  
- **TGW**: All 50 service VPCs attach to 1 TGW. Transitive routing: Service-A ↔ TGW ↔ Shared ↔ TGW ↔ Service-C works automatically (longest prefix match in TGW route table).
  - At 50 services: 50 attachments. Maintains itself via TGW route table (20 lines of config).
  - Scaling benefit: Add service-51, attach to TGW, routing "just works."

**Access control at scale** (TGW + security groups + NACL strategy):

1. **TGW route table policy** (coarse-grained):
   - By default, all attachments can reach all attachments (transit routing enabled).
   - Create TGW route table associations to restrict routing if needed (e.g., dev attachments can't route to prod attachments).

2. **Security group rules** (fine-grained, per-service):
   - Service-A pods: SG = `svc-a-pod`. Outbound rule: allow to `db-access` SG (in Shared VPC).
   - Service-A pods: Outbound rule: allow to `svc-b-pod` SG (in Service-VPC-B) on port 8080 only.
   - Shared VPC RDS: Inbound rule: allow from `db-access` SG (matches Service-A pods).
   - Service-VPC-B app: Inbound rule: allow from `svc-a-pod` SG on port 8080 only.

3. **NACL rules** (stateless, fallback):
   - If a security group is misconfigured, NACL blocks the traffic at the subnet boundary.
   - Service-VPC-A private subnet NACL: deny inbound from 10.1.0.0/16 (Service-VPC-B), except port 8080.
   - This prevents Service-B from scanning Service-A's database ports, even if a security group mistake happens.

**Cost analysis**:
- 50 service VPCs + 1 shared VPC = 51 TGW attachments.
- TGW: $0.05/hour × 730 hours/month = $36.50/month.
- Data processed (inter-VPC): Assume 100 GB/day (service-to-service API calls) + 50 GB/day (service-to-shared DB) = 150 GB/day × 30 days = 4.5 TB/month × $0.02/GB = $90/month.
- **Total: ~$127/month for 51 VPCs.**

Comparison to peering:
- 50 peering connections × 2 directions = 100 route table entries to maintain.
- Cost: $0 (peering is free), but ops cost is high (routing mistakes, incident response).
- At Year 2, you've re-architected to TGW because the routing complexity exceeded team capacity.

**My recommendation**: Start with TGW (not peering) if you have >5 VPCs or expect to grow to >5. The $36/month is worth the operational sanity.

**From a platform-engineering perspective**, I'd:
1. Create a Terraform module `vpc-with-tgw-attachment` that every service team uses to provision their VPC.
2. Module automatically attaches to central TGW and registers in service catalog.
3. Backstage.io template for 'create-new-service' includes the VPC module; developer never manually touches peering or routing.
4. This ensures consistent security group naming (all databases use `db-access` SG), consistent tagging, and zero routing drift."

### Q5: "You're designing a multi-region, multi-VPC platform. How do you ensure consistent security group and NACL policies across regions? What tooling or patterns do you use to prevent drift?"

**Basic Answer** (30 seconds):
"Use Infrastructure-as-Code (Terraform) to define all security groups and NACLs. Store the code in version control. Apply the same code to all regions. Drift detection: run `terraform plan` regularly to catch manual changes."

**Intermediate Answer** (2 minutes):
"Tools and patterns:

1. **Terraform modules** — Create reusable modules for security groups:
   ```hcl
   module "service_sg" {
     source = "./modules/service-sg"
     service_name = "notification-dispatcher"
     vpc_id = var.vpc_id
     
     # Define rules once, apply everywhere
     inbound_rules = [
       { protocol = "tcp", port = 8080, source_sg = "load-balancer" },
       { protocol = "tcp", port = 9090, source_sg = "prometheus" }
     ]
   }
   ```
   - Same code applies to us-east-1, us-west-2, eu-west-1, ap-southeast-1.
   
2. **Terraform workspaces** — Each region is a workspace (us-east-1, us-west-2). Code is identical; `terraform.tfvars` varies per region (VPC IDs, AZ names).

3. **Drift detection** — Tools like Spacelift or Terraform Cloud watch for manual AWS Console changes:
   - If someone manually adds a security group inbound rule via Console, `terraform plan` detects it and flags it as drift.
   - Alert → ops team knows to either code the change or revert the manual change.

4. **Policy-as-Code (Sentinel / OPA)** — Before Terraform apply, validate security groups:
   ```
   policy "require_description_on_sg_rules" {
     enforcement_level = "mandatory"
   }
   // Rejects any rule without a description
   ```

5. **Network ACL pattern** — Store NACL rules in a data source or CSV:
   ```hcl
   variable "nacl_rules" {
     default = [
       { rule_number = 100, protocol = "tcp", port = 443, action = "allow" },
       { rule_number = 200, protocol = "tcp", port = 3306, action = "allow" },
       { rule_number = 32767, action = "deny" } // implicit deny at end
     ]
   }
   ```
   - Same CSV → all regions → all NACLs consistent."

**Advanced Answer** (5 minutes):
"At scale (multi-region, 50+ VPCs, 500+ security group rules), drift becomes inevitable without discipline. Here's the **enterprise pattern**:

**1. Centralized policy repository** (single source of truth):

```
  ├── /policies

```

**2. Policy templating engine** (DRY principle):


**3. Drift detection + automated remediation**:

```bash
for region in us-east-1 us-west-2 eu-west-1; do

    # Drift detected

      --message "Security group drift in $region"

```

**5. Team governance**:

## 12. Tradeoffs: Strategic Decision Points

### Tradeoff 1: Gateway Endpoint vs Interface Endpoint

| Factor | Gateway Endpoint | Interface Endpoint |
|---|---|---|
| **Cost** | $0 (S3, DynamoDB only) | $0.01/AZ/hour = $7.20/month minimum per service per AZ |
| **Services** | S3, DynamoDB, SQS, SNS, CloudFormation | Most AWS services (EC2, RDS, Secrets Manager, Athena, etc.) |
| **Data transfer cost** | $0 (stays on AWS backbone) | $0.01/GB data processed |
| **Compliance** | Private (stays in VPC), satisfies data residency | Private (stays in VPC), satisfies data residency |
| **Setup complexity** | Low (automatic routes via prefix-list) | Medium (ENI creation, DNS name management) |
| **Latency** | Microseconds (direct route) | Milliseconds (through PrivateLink tunnel) |
| **When to use** | High-volume, low-latency access (S3 objects, DDB queries) | Low-volume, compliance-driven access (Secrets Manager, KMS) |

**Decision**: Gateway Endpoint is almost always correct for S3 and DynamoDB. Interface Endpoint only for services that have compliance requirements or low-volume APIs.

---

### Tradeoff 2: NAT Gateway per AZ vs Single NAT Gateway

| Factor | NAT GW Per AZ | Single NAT GW (Shared) |
|---|---|---|
| **Cost** | 3 × $45/month = $135/month (3 AZs) | 1 × $45/month = $45/month |
| **High availability** | If NAT GW in AZ fails, traffic in that AZ reroutes to failed gateway (app downtime) | If NAT GW fails, all traffic fails (total platform downtime) |
| **Cross-AZ data transfer** | Local AZ traffic → local NAT GW = no cross-AZ charge. Non-local traffic → shared NAT GW = $0.02/GB cross-AZ | All traffic through shared NAT GW = $0.02/GB cross-AZ for cross-AZ calls |
| **Scaling to multi-region** | Each region operates independently; cost scales linearly | Shared NAT GW becomes a bottleneck; fails to single region |
| **When to use** | Production systems requiring HA; minimize cross-AZ charges | Dev/test environments; single-AZ deployments |

**Decision**: NAT GW per AZ for production. The $90/month extra cost is worth the HA and cross-AZ charge avoidance.

---

### Tradeoff 3: VPC Peering vs Transit Gateway (At Scale)

| Factor | VPC Peering | Transit Gateway |
|---|---|---|
| **Cost** | $0 per connection | $0.05/hour + $0.02/GB data = ~$36/month + data cost |
| **Scaling (number of VPCs)** | N VPCs = N(N-1)/2 connections = O(N²). At 10 VPCs, 45 connections. At 50 VPCs, 1,225 connections. Unmaintainable. | N VPCs = N attachments = O(N). At 50 VPCs, 50 attachments. Manageable. |
| **Transitive routing** | A↔B and B↔C does NOT allow A→C. Must explicitly peer A↔C. | A↔TGW↔B↔TGW↔C automatically works (transitive). |
| **Routing maintenance** | Manual route table updates per VPC pair | Centralized TGW route table; updates once, applies to all |
| **Operational complexity** | Low upfront; high at scale. | Medium upfront; stays constant at scale. |
| **Use case: <5 VPCs** | Peering is simpler (no TGW overhead). | TGW is overkill. |
| **Use case: 5–20 VPCs** | Peering is becoming painful; TGW worth it. | **Choose TGW.** |
| **Use case: >20 VPCs** | Peering is unmaintainable. | **Must use TGW.** |

**Decision**: For T-Mobile's platform (25M msg/day, expecting to grow to 10+ clusters = 10+ VPCs), **choose TGW at Week 4 Day 1**. The $36/month upfront cost saves 200+ engineering hours at Year 2.

### Analogy

Think of AWS VPC as your data center floor plan — the one you'd draw on a whiteboard when designing a new messaging rack.

- **VPC** = the entire data center building — your isolated namespace with its own address space
- **Subnets** = individual rooms on a floor — public rooms have windows (internet access), private rooms have no exterior windows
- **Route Table** = the building directory + elevator panel — tells traffic which hallway to take
- **Internet Gateway (IGW)** = the front door with a security desk — traffic to/from the public internet
- **NAT Gateway** = a proxy receptionist — private rooms can send mail out, but no one from outside can initiate a knock on those doors
- **Security Group** = a personal bodyguard per server — stateful, knows "I already let this conversation in, so the reply is automatically allowed"
- **NACL** = the room's door access policy posted on the wall — stateless, checks every packet in BOTH directions independently, forgets prior conversations
- **VPC Endpoint** = an internal pneumatic tube system — traffic never leaves the building to reach AWS services like S3
- **PrivateLink** = a private connecting corridor between two buildings — partner/vendor services accessible without going through the public street
- **Transit Gateway** = the central train station hub — connects all your buildings (VPCs, accounts) through one managed interchange

### 30-Second Version

A VPC is your private AWS network. Public subnets face the internet via an IGW; private subnets use a NAT Gateway for outbound-only internet. Security Groups are your per-instance stateful firewall; NACLs are your subnet-level stateless access list (and the source of a specific class of hard-to-debug connectivity failures). Without VPC Endpoints, every EKS pod call to S3 exits your network, hits the public S3 endpoint, and returns — costing egress dollars and breaking compliance posture. A Gateway Endpoint for S3/DynamoDB is free and routes that traffic inside AWS. PrivateLink does the same for other AWS services or your own SaaS offerings. Transit Gateway connects all your VPCs across accounts like a managed hub-and-spoke router.

### 2-Minute Version

#### How a Packet Travels from EKS Pod to S3 (Without Endpoint)

1. EKS pod (private subnet, e.g., 10.0.3.45) calls `s3.amazonaws.com`
2. DNS resolves to a public S3 IP (e.g., 52.218.x.x)
3. Route table in private subnet: "no route to 52.218.x.x, default route → NAT Gateway"
4. NAT Gateway (in public subnet) replaces source IP with its Elastic IP
5. Traffic exits through IGW to the public internet
6. Returns same path in reverse
7. **Cost**: Data transfer out of VPC = $0.09/GB. For a 25M msg/day platform writing logs to S3, this adds up fast.

#### How a Packet Travels from EKS Pod to S3 (With Gateway Endpoint)

1. EKS pod calls `s3.amazonaws.com`
2. DNS resolves same public IP — but route table now has a prefix list entry: "all S3 IPs → vpce-xxxxxxxx"
3. Traffic is intercepted by AWS fabric and routed to S3 through the private AWS backbone
4. Never touches NAT Gateway, never exits VPC, never incurs egress charge
5. **Cost**: $0 for Gateway Endpoint

#### Security Groups vs NACLs — The Stateful vs Stateless Distinction

**Security Group (stateful)**:
- You allow inbound port 443
- The return traffic (ephemeral ports 1024-65535 outbound) is AUTOMATICALLY allowed
- Works like your kernel's conntrack table — it remembers the session

**NACL (stateless)**:
- You allow inbound port 443 — OK
- You MUST ALSO explicitly allow outbound ephemeral ports (1024-65535) — or the response packets are dropped
- This is the exact failure mode that causes "works once, then hangs" intermittent failures
- NACLs are evaluated in rule number order — first match wins — and there is a default DENY ALL at the end

#### VPC Endpoint Types

| Type | Services | Cost | How It Works |
|---|---|---|---|
| Gateway Endpoint | S3, DynamoDB only | Free | Route table entry, no ENI created |
| Interface Endpoint (PrivateLink) | 100+ AWS services, partner SaaS | $0.01/hr + $0.01/GB | Creates an ENI in your subnet with a private IP |

#### PrivateLink Use Case

You have a vendor (e.g., Datadog, Snowflake, internal microservice in another account) that you need to call from your private subnet. Without PrivateLink, you'd need internet egress. With PrivateLink, the vendor exposes a Network Load Balancer; AWS creates a "service endpoint" you subscribe to; your subnet gets an ENI with a private IP; traffic never leaves the AWS network. This is the pattern for zero-trust network architecture with SaaS vendors.

#### Cross-AZ Traffic Cost

AWS charges **$0.01/GB in each direction** for traffic crossing AZ boundaries. Sounds small. Do the math for a notification platform:

- 25M messages/day, each with 2KB payload = 50GB/day
- If your EKS pods are in us-east-1a and your RDS is in us-east-1b → every DB call crosses AZs
- 50GB × $0.01 × 2 directions × 30 days = **$30/month just for cross-AZ**
- At 10x scale: $300/month — and this is just one service pair
- At full enterprise scale with many services: cross-AZ traffic routinely becomes a **$10K–$100K/month line item**

The fix: AZ-affinity routing in Kubernetes (topology spread constraints), read replicas in the same AZ, and measuring with Cost Explorer grouped by "AZ data transfer."

### 5-Minute Version

#### Subnet CIDR Sizing — The Mistake You Can't Easily Undo

VPC CIDR and subnet sizing is one of the rare AWS decisions that's painful to change later. If you allocate a /28 (16 IPs, minus 5 AWS reserved = 11 usable) to a subnet, and EKS needs 1 IP per pod (with VPC CNI), you'll exhaust IPs fast. Standard practice: /16 VPC, /20 or /18 subnets per AZ per tier (public, private, intra). EKS clusters benefit from a dedicated "pod CIDR" subnet that's large (/17 or bigger) because EKS VPC CNI assigns real VPC IPs to pods.

#### The NACL Intermittent Failure Pattern

Classic scenario: you add a NACL rule to block a known bad IP range. Tests pass. But 30 minutes later, intermittent 5XX errors start. Why?

Because NACLs are stateless. Your inbound ALLOW rule for port 443 passed inspection. But your NACL has no outbound rule for ephemeral ports 1024-65535 (the return traffic port range). TCP three-way handshakes are completing, but response packets from your servers are being dropped by the outbound NACL. The result: connections time out seemingly at random (only connections that happened to be using a port that falls in an unblocked range succeed).

**Diagnosis**: VPC Flow Logs + filter for REJECT action on outbound ephemeral port range from your subnet's ENIs.

**Fix**: Always add an outbound allow for 1024-65535 TCP/UDP when you add inbound rules on a NACL.

#### PrivateLink vs VPC Peering vs Transit Gateway

| Pattern | Use Case | Limitation |
|---|---|---|
| VPC Peering | Simple 1:1 VPC connection, same or different account | Does NOT support transitive routing — A→B and B→C does NOT mean A→C |
| PrivateLink | One-way service exposure (consumer → provider) | Consumer cannot initiate connections back; unidirectional by design |
| Transit Gateway | Hub-and-spoke for 3+ VPCs or multi-account | $0.05/GB attachment fee + $0.02/GB processed data; adds latency hop |

**The transitive routing gap is the #1 VPC peering gotcha in interviews.** If you have VPCs A, B, C all peered with each other, that's a full mesh — 3 peering connections. At 10 VPCs, full mesh = 45 peering connections. This is why Transit Gateway exists.

#### Transit Gateway Architecture (Multi-Account)

In AWS Organizations with multiple accounts (typical at Director level: separate accounts for prod, staging, security, shared-services), TGW is the backbone:

```
Account: Shared-Services VPC (DNS, monitoring, bastion)
          |
     Transit Gateway
    /       |        \
Prod VPC  Stage VPC  Security VPC
```

TGW supports route tables per attachment — so you can allow prod→shared-services but DENY prod→staging using TGW route tables. This is network-level blast radius control across accounts.

#### VPC Flow Logs — What They Do and Don't Tell You

Flow Logs capture: source IP, dest IP, source port, dest port, protocol, packets, bytes, action (ACCEPT/REJECT), start/end time.

Flow Logs do NOT capture: payload, DNS query names, HTTP headers, application-layer metadata.

This means: Flow Logs tell you "connection was REJECTED from 10.0.3.45 to 10.0.2.10 port 5432" but not WHY the app is failing. You still need application-level observability (your Splunk APM / AppDynamics stack) for that.

#### Cost Architecture Decisions at Director Level

1. **Endpoint strategy**: Every service that calls S3, DynamoDB, ECR, Secrets Manager, CloudWatch should be routed through VPC Endpoints. S3 and DynamoDB Gateway Endpoints are free — add them day one. Interface Endpoints cost $7.20/month per endpoint per AZ — prioritize for high-volume services (ECR for large Docker layers, Secrets Manager for high-frequency secret fetches).

2. **AZ affinity**: Design your pod scheduling and database read replicas to stay within the same AZ. Kubernetes topology spread constraints + node affinity labels are your levers. A 10% performance hit from not load-balancing across AZs is almost always cheaper than cross-AZ traffic at scale.

3. **NAT Gateway sizing**: One NAT Gateway per AZ costs $0.045/hr = ~$32/month regardless of traffic. But cross-AZ traffic TO a single NAT Gateway (if you centralize) costs $0.01/GB. At scale, multiple NAT Gateways (one per AZ) is cheaper than centralizing.

### Answer Key — Supplementary Comprehension Check

**A1**: NACLs are stateless — unlike Security Groups, they do not automatically allow return traffic for established connections. When you added the inbound block rule, you inadvertently disrupted the NACL's outbound rule coverage. More precisely: even if you have an inbound ALLOW for port 443, the response packets leave your server on ephemeral ports (1024-65535). If your NACL outbound rules don't explicitly ALLOW TCP/UDP 1024-65535, those return packets are dropped — causing the TCP session to hang and ultimately timeout.

Fix: Add an outbound NACL rule: ALLOW TCP 1024-65535 to 0.0.0.0/0 (or your client CIDR range). Rule number should be lower than any DENY rule. Then verify in Flow Logs that outbound REJECT on ephemeral ports stops.

**A2**: The monthly cost impact is significant cost savings. Without the endpoint, 100GB/day × 30 days = 3,000GB/month routes through NAT Gateway and incurs AWS data transfer charges. NAT Gateway processing: $0.045/GB × 3,000GB = $135/month in processing alone, plus $0.09/GB potential egress = $270/month in egress. A VPC Gateway Endpoint for S3 is free — $0/month. The application code does not change because the endpoint works at the route table level: when a pod calls `s3.amazonaws.com`, DNS still resolves to the same S3 IP, but the route table now has a managed prefix list entry that intercepts traffic matching S3 IP ranges and routes it through the endpoint instead of the default route to NAT Gateway. The AWS SDK call is identical; the routing decision happens below the application layer.

**A3**: VPC peering is insufficient because it is non-transitive. If prod is peered with shared-services (for DNS/monitoring) and prod is peered with security (for traffic inspection), prod can reach both — but if the security account needs to route all prod traffic through an inspection appliance before it exits, peering alone cannot express that "route-through" topology. You'd need every account peered with every other account (full mesh = 10 peering connections for 5 accounts) and you still can't enforce traffic chaining.

Transit Gateway solves this with: (1) a single attachment point per VPC — no full mesh, (2) TGW route tables that can force-route all egress from prod through the security VPC's inspection appliance before it goes anywhere, and (3) centralized route management in one place instead of n × (n-1)/2 peering agreements. The security team manages the TGW route table for the "prod-outbound" route table; shared-services attaches once and is reachable by all accounts that have a route to TGW.

#### Step 6: Create a VPC Gateway Endpoint for S3

```bash
# Get the route table ID (main route table of the default VPC)
export RTB_ID=$(aws ec2 describe-route-tables \
  --region us-east-1 \
  --filters "Name=vpc-id,Values=$VPC_ID" "Name=association.main,Values=true" \
  --query 'RouteTables[0].RouteTableId' \
  --output text)

echo "Route Table ID: $RTB_ID"

# Create the Gateway Endpoint for S3
# Note: --route-table-ids tells AWS to automatically add the prefix list route to this route table
aws ec2 create-vpc-endpoint \
  --region us-east-1 \
  --vpc-id $VPC_ID \
  --service-name com.amazonaws.us-east-1.s3 \
  --vpc-endpoint-type Gateway \
  --route-table-ids $RTB_ID \
  --query 'VpcEndpoint.{EndpointId:VpcEndpointId,Service:ServiceName,State:State,RouteTableIds:RouteTableIds}' \
  --output json
```

**EXPECTED_OUTPUT**:
```json
{
    "EndpointId": "vpce-0abc1234def567890",
    "Service": "com.amazonaws.us-east-1.s3",

    ]
}
```

**WHY**: This creates a Gateway Endpoint and automatically injects a route into your route table. The route uses an AWS-managed prefix list (a named list of all S3 IP ranges) as the destination. This is the "pneumatic tube" — when a pod calls S3, the route table sees the destination matches the S3 prefix list and redirects through the endpoint instead of IGW.

#### Lab Answer Key — Diagnostics Per Step

**Step 1 fails — no VPCs returned**:
- Check: `aws sts get-caller-identity` — are you authenticated?
- Check: `aws configure list` — is region set to us-east-1?
- Fix: `aws configure set region us-east-1`

**Step 3 — route table shows no IGW route**:
- Your default VPC may have been modified, or you're looking at a custom VPC from the EKS lab
- Try: `aws ec2 describe-route-tables --region us-east-1 --filters "Name=vpc-id,Values=$VPC_ID" --output json` to see all route tables including non-main ones

**Step 6 — endpoint creation fails with "InvalidVpcEndpoint"**:
- The service name must exactly match: `com.amazonaws.us-east-1.s3`
- Verify available services: `aws ec2 describe-vpc-endpoint-services --region us-east-1 --query 'ServiceNames' --output text | tr '\t' '\n' | grep s3`

**Step 6 — endpoint creation fails with access denied**:
- Your training-admin user may be missing `ec2:CreateVpcEndpoint` permission
- Fix: In AWS console, attach `AmazonVPCFullAccess` policy to the training-admin user temporarily

**Step 7 — no prefix list entry in route table**:
- The endpoint may have been created without associating the route table
- Fix: `aws ec2 modify-vpc-endpoint --region us-east-1 --vpc-endpoint-id $ENDPOINT_ID --add-route-table-ids $RTB_ID`

**Step 8 — IAM role already exists (from a prior lab run)**:
- Skip the create-role step; just get the ARN: `export FLOWLOGS_ROLE_ARN=$(aws iam get-role --role-name VPCFlowLogsRole --query 'Role.Arn' --output text)`

**Step 9 — CloudWatch query returns "Scheduled" status forever**:
- Wait 2-3 more minutes; Flow Logs have a delivery delay
- Check the log group has streams: `aws logs describe-log-streams --region us-east-1 --log-group-name /aws/vpc/flowlogs-training`
- If no streams after 5 minutes, verify the IAM role trust policy allows `vpc-flow-logs.amazonaws.com`

- The endpoint state must be "available" before deletion
- Check state: `aws ec2 describe-vpc-endpoints --region us-east-1 --query 'VpcEndpoints[*].State'`

### Tradeoffs

#### Tradeoff 1: Gateway Endpoint vs Interface Endpoint (PrivateLink)

**Gateway Endpoint**: Free, route-table based, works only for S3 and DynamoDB. No ENI, no DNS change. Simple to operate, scales with no management overhead. The downside: you can't add custom policies as granularly, and it doesn't work for on-premises traffic routed through Direct Connect (because route tables are VPC-local).

**Interface Endpoint (PrivateLink)**: Costs $0.01/hr per AZ + $0.01/GB. Creates a real ENI in your subnet. Works for 100+ AWS services and custom services. Works with Direct Connect and VPN (because it's DNS-based, not route-table-based). Required for strict zero-trust environments where you want to audit service calls at the ENI level via Flow Logs.

**Interview answer**: "Use Gateway Endpoints for S3 and DynamoDB always — it's free and there's no reason not to. Use Interface Endpoints for services where traffic volume, compliance, or on-premises connectivity justifies the cost."

#### Tradeoff 2: Centralized NAT Gateway vs One Per AZ

**Centralized (one NAT GW in one AZ)**: Saves ~$64/month (2 fewer NAT GWs). But cross-AZ traffic to reach the single NAT GW costs $0.01/GB — at high volume, the cross-AZ charge exceeds the NAT GW savings. Also creates a single AZ failure dependency for all outbound traffic.

**One per AZ**: More expensive upfront ($32/month per NAT GW × number of AZs). But eliminates cross-AZ charge for outbound traffic and gives you AZ-level resiliency for outbound paths.

**Interview answer**: "At low traffic volume, centralized is fine. Above ~3TB/month of NAT-bound traffic, one-per-AZ becomes cheaper. I calculate the break-even and automate the decision in our Terraform module via a `nat_gateway_mode = single|one_per_az` variable."

#### Tradeoff 3: Transit Gateway vs VPC Peering at Scale

**VPC Peering** is operationally simple for 2-3 VPCs and has lower latency (direct, no hop through TGW). But it does not scale: no transitive routing, full mesh connection explosion (N×(N-1)/2), no centralized route management.

**Transit Gateway** solves scale but introduces: per-attachment cost ($0.05/hr per attachment = ~$36/month), per-GB processing ($0.02/GB), and a managed routing hop (adds ~1ms latency in worst case). The architectural power is TGW route tables — you can segment traffic (prod cannot reach dev, security can inspect everything) without complex peering mesh management.

**Failure mode to call out in interviews**: TGW is a regional resource. If your architecture requires cross-region connectivity, you need TGW peering between regions — which adds cost and latency. For cross-region, sometimes Direct Connect + a well-designed peering layout is simpler than TGW peering.

**Interview answer**: "Our threshold is 5+ VPCs or any multi-account architecture that needs centralized traffic inspection. Below that, VPC peering is operationally cheaper. Above that, TGW's route table segmentation pays for itself in reduced security review overhead and simpler network topology documentation."
