# GitOps with ArgoCD — Pull Model and Why It Matters | Week 4 Day 3

## Why This Matters

At T-Mobile scale — 25M messages/day, multiple teams, multiple services — the biggest deployment risk is not the code change itself, it is the blast radius of who holds cluster credentials. Push-model CI/CD (Jenkins/GitHub Actions SSHing directly into a cluster) means your entire deployment pipeline is a credential breach away from a full cluster takeover. ArgoCD's pull model eliminates that attack surface: your CI pipeline never touches the cluster. GitOps is now the interview baseline for any Principal/Director SRE role at cloud-native companies — Argo or Flux familiarity is listed explicitly in 70%+ of Staff+ job descriptions as of 2025.

---

## Analogy

Think of ArgoCD as the **Terraform plan/apply loop — but running continuously, every 3 minutes, forever, against your Kubernetes cluster**.

Here is how your existing mental model maps:

In Terraform:
- You declare desired state in `.tf` files
- `terraform plan` detects drift between desired state and actual AWS state
- `terraform apply` reconciles — makes the real world match the declaration
- If someone manually edits an S3 bucket policy (out-of-band change), `plan` shows drift on the next run

In ArgoCD:
- You declare desired state in Git (YAML manifests or Helm charts)
- ArgoCD's **application controller** polls Git + polls the live cluster every ~3 minutes
- When it detects drift (Git != cluster), it marks the app `OutOfSync`
- With auto-sync enabled, it immediately runs the equivalent of `kubectl apply` to reconcile
- Manual cluster edits (someone `kubectl edit`ing a deployment) are detected and reverted

The second analogy is your **RabbitMQ message consumer** mental model: ArgoCD is a consumer watching two queues simultaneously (Git state, cluster state) and triggering reconciliation whenever those two queues diverge.

**Why the pull model matters for security**: In a push model, your GitHub Actions runner needs a `KUBECONFIG` with cluster-admin rights stored as a CI secret. If that CI system is compromised, the attacker has full cluster access. In the pull model, ArgoCD runs *inside* the cluster — it reaches out to GitHub to pull changes. GitHub never holds cluster credentials. The blast radius of a compromised CI system is zero for cluster access.

## 30-Second Version

ArgoCD is a Kubernetes-native GitOps controller that continuously reconciles what is declared in Git with what is running in your cluster. You push code to GitHub; ArgoCD detects the change and applies it — your CI pipeline never holds cluster credentials. When someone manually edits a resource in the cluster (drift), ArgoCD detects and optionally reverts it. The result: Git is the single source of truth, every change is auditable via `git log`, and rollback is `git revert`.

## 2-Minute Version

**How ArgoCD works mechanically:**

ArgoCD has four core components running as pods in the `argocd` namespace:

1. **API Server** — the control plane. Handles the UI, CLI (`argocd` command), and gRPC/REST API calls. Think of it as the Kubernetes API server for ArgoCD itself.

2. **Repo Server** — a stateless cache service. When a sync is needed, the repo server clones the Git repository, renders the manifests (evaluates Helm templates, applies Kustomize overlays), and returns raw YAML. It is stateless — it does not talk to the cluster directly.

3. **Application Controller** — the heart of ArgoCD. Runs the reconciliation loop: every ~3 minutes it compares live cluster state against rendered manifests from the repo server. When it detects drift, it marks the Application `OutOfSync` and optionally triggers a sync.

4. **Dex** — an optional OIDC identity provider. Lets you integrate ArgoCD's UI/CLI with your corporate SSO (Okta, Azure AD, GitHub OAuth). Same role as any OAuth2 broker.

**Core objects you will work with:**

- **Application**: The fundamental unit. Declares "watch this Git path, apply to this cluster namespace." Has a sync policy (manual or automated).
- **AppProject**: A security boundary. Restricts which source repos, destination clusters, and namespaces a group of Applications can use. Think IAM policy attached to a set of Applications.
- **ApplicationSet**: A generator that creates multiple Applications from a template. Common pattern: one ApplicationSet generates one Application per environment (dev/staging/prod) using a list or cluster generator.

**Sync policies explained:**
- `manual`: ArgoCD detects drift and tells you, but does nothing. You click "Sync" in the UI or run `argocd app sync`. Safe for production environments where humans must approve changes.
- `automated`: ArgoCD auto-syncs whenever it detects Git has changed. Add `selfHeal: true` to also revert manual cluster edits. Add `prune: true` to delete K8s resources when their manifest is removed from Git.

## 5-Minute Version

**Tradeoffs and design decisions you will be asked about:**

**When to use manual vs. automated sync:**
Production environments typically use manual sync with required approval — this gives you the GitHub PR review as your change control, plus a human click as the deployment gate. Development and staging environments use automated sync with `selfHeal: true` to keep them perpetually clean and matching main branch. The organizational pattern: dev = auto + self-heal + prune; staging = auto + self-heal, no prune (orphaned resources are a canary signal); production = manual sync, no prune (protect against accidental deletions).

**Self-heal caveat — the SRE gotcha:**
Self-heal is powerful but dangerous in incidents. During an outage, your on-call SRE `kubectl edit`s a deployment to change an environment variable as a quick mitigation. With self-heal enabled, ArgoCD reverts that change within 3 minutes. Your mitigation disappears while the incident is still active. The operational pattern: disable auto-sync on production before manual incident interventions, or use ArgoCD's "suspend auto-sync" feature on the Application during incident windows.

**Prune — another gotcha:**
If you rename a Deployment in Git (e.g., `frontend-v1` to `frontend-v2`), ArgoCD with prune enabled will delete `frontend-v1` and create `frontend-v2`. If your Helm chart or Kustomize base accidentally loses a Service manifest during a refactor, prune will delete the live Service. This causes an outage. Best practice: enable prune only after your team has disciplined Git hygiene and PR reviews that include manifest diff checks.

**App-of-apps pattern:**
When you have 20 microservices, managing 20 separate ArgoCD Applications manually does not scale. The app-of-apps pattern solves this: one parent Application points to a Git directory that contains only ArgoCD Application manifests (not actual service manifests). ArgoCD deploys the parent, which discovers and deploys all child Applications. This is your Terraform root module / child module pattern — the root module manages the other modules.

**Multi-cluster GitOps:**
ArgoCD can manage multiple clusters from one control plane. Each Application specifies a `destination.server` (cluster API endpoint). This is how platform teams at large orgs run one ArgoCD instance to govern 10+ clusters. The AppProject becomes critical here — it limits which Applications can target which clusters.

**ArgoCD vs. Flux:**
Both are CNCF-graduated GitOps engines. ArgoCD has a richer UI and AppProject RBAC model, making it more enterprise-friendly for multi-team governance. Flux is more Kubernetes-native (uses CRDs more heavily), lighter weight, and preferred for single-team or platform-engineering-native setups. Both are acceptable interview answers — ArgoCD is more commonly asked about in SRE interviews because of its UI and RBAC story.

**Failure modes:**
- Git repo unreachable: ArgoCD cannot fetch latest desired state. It continues running the last-known state. Applications show `Unknown` health. The cluster keeps running — ArgoCD is not in the critical path of existing workloads, only new deployments.
- ArgoCD itself is down: Existing workloads continue running. New GitOps deployments are blocked. This is why ArgoCD is deployed with HA (3 replicas of Application Controller) in production.
- Webhook vs polling: By default ArgoCD polls Git every 3 minutes. For faster response, configure a Git webhook to push events to ArgoCD API server on every commit. This reduces deployment lag from up to 3 minutes to under 30 seconds.

## Director/VP Lens

**How a Director governs GitOps vs. how an IC implements it:**

**IC implementation focus:**
- Installs ArgoCD, configures Applications, writes Helm charts, troubleshoots sync errors, debugs manifest rendering failures, manages repo server cache, monitors ArgoCD component health.

**Director/VP governance focus:**
- **AppProject as your deployment policy**: Every team's Applications must be scoped to an AppProject that restricts source repos (only your org's GitHub), destination namespaces (never cluster-scoped unless explicitly approved), and allowed resource kinds. This is your change control enforcement mechanism — not a Jira ticket, but a cryptographic policy.
- **Git branch strategy as your SDLC**: GitOps enforces that `main` = production. Your branching strategy (GitFlow, trunk-based) directly maps to your deployment gates. Director-level decision: does a PR merge to `main` automatically deploy to prod (continuous deployment) or gate behind a manual ArgoCD sync (continuous delivery)?
- **Audit trail governance**: `git log` on your manifests repo is your deployment audit log. Every change has an author, a timestamp, a diff, and a PR review record. Regulators (SOX, PCI) accept Git history as an audit trail — this is a compliance win you can communicate at the VP level.
- **Incident response protocol for GitOps teams**: Your runbook must explicitly address "how do we make an emergency config change when GitOps is in the loop?" The answer: use `argocd app patch` to temporarily disable auto-sync during the incident window, make the manual change, then restore auto-sync and update Git to match. This prevents the self-heal revert problem.
- **AI/automation lens**: In 2025-2026, the frontier move is ArgoCD + AI-driven PR generation. A remediation bot detects an incident, generates a Git commit with the config fix, opens a PR, and if auto-approved (low-risk change), merges and deploys — all without human intervention. As Director, you govern the risk thresholds for auto-approval, not the implementation.

## ANSWER KEY — Comprehension Check

**A1.** In the push model, GitHub Actions holds a `KUBECONFIG` (cluster credentials) as a CI secret. If GitHub Actions is compromised, the attacker can run arbitrary `kubectl` commands against your cluster. After switching to ArgoCD: ArgoCD runs *inside* the cluster and reaches *out* to GitHub to pull manifests. GitHub Actions no longer needs cluster credentials — it only writes to Git. The CI secret attack surface for cluster access is eliminated entirely.

**A2.** With `selfHeal: true`, ArgoCD's application controller will detect within 3 minutes that the live deployment (10 replicas) differs from Git (e.g., 3 replicas declared in the manifest). ArgoCD will scale the deployment back down to 3 replicas — undoing the engineer's mitigation. What the engineer should do: immediately run `argocd app patch <app-name> --patch '{"spec":{"syncPolicy":null}}' --type merge` to suspend auto-sync on that Application, or click "Disable Auto-Sync" in the ArgoCD UI, before making the manual change. After the incident is resolved, update the Git manifest to the correct replica count, re-enable auto-sync, and sync.

**A3.** **ApplicationSet** is the feature. You create one ApplicationSet with a list generator (or cluster generator) that iterates over 3 environments, templating one Application per environment. For 15 services, you would have 15 ApplicationSets (one per service) — each generating 3 Applications = 45 total Applications, but only 15 ApplicationSet objects to author and maintain. The Terraform analogy: ApplicationSet is `for_each` over a map of environments; each Application is one Terraform workspace instance. The alternative is the **app-of-apps pattern** — a parent Application that manages a Git directory full of Application manifests.

### Step 10: Create an AppProject to restrict deployment scope

AppProjects are the security governance layer. In a multi-team org, you use AppProjects to enforce which teams can deploy to which namespaces from which repositories.

```bash
# Create an AppProject YAML
cat > /tmp/argocd-appproject.yaml << 'EOF'
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: team-platform
  namespace: argocd
spec:
  description: "Platform team - restricted to platform namespaces and approved repos"
  
  # Only these Git repos are allowed as sources
  sourceRepos:
    - "https://github.com/argoproj/argocd-example-apps.git"
    - "https://github.com/YOUR_ORG/*.git"   # replace with your org
  
  # Only these cluster/namespace combinations are allowed as destinations
  destinations:
    - server: https://kubernetes.default.svc
      namespace: guestbook
    - server: https://kubernetes.default.svc
      namespace: platform-*    # wildcard: any namespace starting with platform-
  
  # Restrict which Kubernetes resource types this project can deploy
  # Prevents teams from deploying ClusterRoles, PodSecurityPolicies, etc.
  namespaceResourceWhitelist:
    - group: "apps"
      kind: "Deployment"
    - group: ""
      kind: "Service"
    - group: ""
      kind: "ConfigMap"
    - group: ""
      kind: "ServiceAccount"
  
  # Block these resource types entirely (belt-and-suspenders)
  clusterResourceBlacklist:
    - group: ""
      kind: "Namespace"
    - group: "rbac.authorization.k8s.io"
      kind: "ClusterRole"
    - group: "rbac.authorization.k8s.io"
      kind: "ClusterRoleBinding"

  # Roles within the project (RBAC for ArgoCD users)
  roles:
    - name: developer
```

### Step 11: Demonstrate app-of-apps pattern (conceptual — create the structure)

The app-of-apps pattern avoids managing dozens of individual Application objects manually. You create one parent Application that manages a directory of child Application manifests.

```bash
# Create a local directory to simulate the app-of-apps Git structure
mkdir -p /tmp/app-of-apps/apps

# Create child Application manifests (these would normally live in Git)
cat > /tmp/app-of-apps/apps/guestbook-dev.yaml << 'EOF'
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: guestbook-dev
  namespace: argocd
  # This annotation ensures child apps are deleted when parent is deleted
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: https://github.com/argoproj/argocd-example-apps.git
    targetRevision: HEAD
    path: guestbook
  destination:
    server: https://kubernetes.default.svc
    namespace: guestbook-dev
  syncPolicy:
    automated:
      selfHeal: true
      prune: false
    syncOptions:
      - CreateNamespace=true
EOF

cat > /tmp/app-of-apps/apps/guestbook-staging.yaml << 'EOF'
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: guestbook-staging
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: https://github.com/argoproj/argocd-example-apps.git
    targetRevision: HEAD
    path: guestbook
  destination:
    server: https://kubernetes.default.svc
    namespace: guestbook-staging
  syncPolicy:
    automated:
      selfHeal: true
      prune: false
    syncOptions:
      - CreateNamespace=true
EOF

# Show the directory structure
ls -la /tmp/app-of-apps/apps/
```

**EXPECTED_OUTPUT:**
```
total 16
drwxr-xr-x 2 ubuntu ubuntu 4096 Jun xx xx:xx .
drwxr-xr-x 3 ubuntu ubuntu 4096 Jun xx xx:xx ..
-rw-r--r-- 1 ubuntu ubuntu  xxx Jun xx xx:xx guestbook-dev.yaml
-rw-r--r-- 1 ubuntu ubuntu  xxx Jun xx xx:xx guestbook-staging.yaml
```

**NOTE**: In a real app-of-apps setup, this `/tmp/app-of-apps/` directory would be a Git repository. The parent Application would point to the `apps/` subdirectory in that repo. ArgoCD would then deploy `guestbook-dev.yaml` and `guestbook-staging.yaml` as Application objects — which ArgoCD then manages as regular Applications, each deploying the actual guestbook service into their respective namespaces.

**WHY**: This pattern scales GitOps to large organizations. One parent Application = one object to manage. Add a new microservice to your platform: add one YAML file to the `apps/` directory in Git, commit, and ArgoCD bootstraps the new Application automatically. Remove a service: delete the YAML from Git, ArgoCD deletes the child Application (and with prune, the deployed resources too).

## Lab ANSWER KEY

**If Step 2 pods are stuck in `Pending` or `ImagePullBackOff`:**
- Check node capacity: `kubectl get nodes` — if no nodes are Ready, your EKS node group may have scaled to zero. Run: `eksctl scale nodegroup --cluster training-cluster --name ng-1 --nodes 2 --region us-east-1`
- Check events: `kubectl describe pod <stuck-pod-name> -n argocd`

**If Step 3 returns a blank line (no password):**
- The secret may not exist yet — ArgoCD creates it during install. Wait 30 seconds and retry.
- Verify: `kubectl -n argocd get secret argocd-initial-admin-secret`

**If Step 4 browser shows "connection refused":**
- Ensure the `kubectl port-forward` command is still running in its terminal. If it exited, re-run it.
- Try http (not https) first: `http://localhost:8080` — some environments redirect

**If Step 6 `argocd login` fails with "connection refused":**
- Confirm port-forward is running: `ps aux | grep port-forward`
- Try: `argocd login 127.0.0.1:8080 --username admin --insecure --password $(kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath="{.data.password}" | base64 -d)`

**If Step 8 `argocd app diff` shows no output after manual scale:**
- ArgoCD may not have refreshed yet. Force refresh: `argocd app get guestbook --hard-refresh`
- Check if ArgoCD tracks replicas: some versions of the guestbook app use HPA which overrides replica counts. If so, manually edit the deployment spec image tag instead: `kubectl set image deployment/guestbook-ui guestbook-ui=gcr.io/heptio-images/ks-guestbook-demo:0.1 -n guestbook`

**If Step 9 self-heal does not trigger within 2 minutes:**
- Verify self-heal is set: `argocd app get guestbook | grep -i sync`
- Force a sync: `argocd app sync guestbook --force`
- Check application controller logs: `kubectl logs -n argocd -l app.kubernetes.io/name=argocd-application-controller --tail=50`

**If Step 10 AppProject fails to apply:**
- Confirm ArgoCD CRDs are installed: `kubectl get crd | grep argoproj`
- You should see `appprojects.argoproj.io` in the list

## STAR Anchor

**Q1 — Security architecture (use T-Mobile context):**
"At T-Mobile, our 25M message/day notification platform used Jenkins pipelines that stored EKS KUBECONFIG tokens as CI secrets — each pipeline had cluster-admin rights as a convenience shortcut. When we proposed migrating to ArgoCD, the security finding that got executive buy-in was this: one compromised Jenkins credential would give an attacker the ability to scale down every deployment in the cluster at 3am during peak traffic. With ArgoCD's pull model, Jenkins only writes to GitHub. The cluster credentials never leave the cluster. We eliminated that attack surface entirely for the cost of a one-week migration."

**Q2 — Self-heal incident response (operational maturity question):**
"We hit exactly this failure mode in our staging environment at T-Mobile. We had self-heal enabled and an engineer scaled up a pod count during a canary deployment test — ArgoCD reverted it, the engineer re-scaled, ArgoCD reverted again. We lost 20 minutes diagnosing a 'ghost rollback.' We solved it by adding a runbook step: any emergency manual intervention must start with `argocd app patch <app> --patch '{"spec":{"syncPolicy":null}}'` to suspend auto-sync. We also added a Slack alert whenever auto-sync is disabled — that alert becomes the 'restore auto-sync' reminder at incident close."

**Q3 — Scale governance (Director lens):**
"We used ApplicationSets with a Git directory generator. Each squad owns a directory in our platform GitOps repo. The ApplicationSet template generates one Application per directory per environment. Adding a new service: the squad creates a new directory, commits a kustomization.yaml, and ArgoCD bootstraps it automatically — no platform team ticket required. That is our golden path for new service onboarding. We also scoped each squad to an AppProject that limits their allowed namespaces and resource types, so a squad cannot accidentally deploy into another team's namespace."

## Tradeoffs & Failure Modes

**Tradeoff 1 — Self-heal is enforcement, not assistance**
Self-heal guarantees cluster state matches Git — which is exactly what you want 99% of the time. But during incidents, operational muscle memory (direct `kubectl` interventions) actively fights you. Every team running GitOps must explicitly train on the "suspend auto-sync first" protocol. The failure mode is insidious: your mitigation disappears silently, you reapply it, it disappears again, you think the cluster is misbehaving when actually ArgoCD is working exactly as designed. Director-level response: add this scenario explicitly to your incident runbook and gameday exercises.

**Tradeoff 2 — GitOps creates a deploy lag proportional to Git poll interval**
Default ArgoCD polling is every 3 minutes. A commit to `main` may not be live for up to 3 minutes. For most services this is acceptable. For emergency hotfixes, it feels unacceptable. The fix (webhook integration) reduces lag to under 30 seconds, but requires network policy changes to allow GitHub to reach your ArgoCD API server. In production, this usually means exposing ArgoCD's webhook endpoint via an ingress with IP allowlist for GitHub's IP ranges. Teams often underestimate this setup cost during initial GitOps adoption.

**Tradeoff 3 — Git history is your audit log, but Git history is mutable**
GitOps gives you a deploy audit trail via `git log`. But `git rebase` and `git force-push` can rewrite history — silently eliminating the audit record. For compliance (SOX, PCI), this is a gap. The mitigation: branch protection on `main` (no force-pushes, required PR reviews), combined with an immutable external audit log (CloudTrail, or a Git mirror to an append-only store). A Director-level interview answer includes: "GitOps gives us the audit trail, but we protect its integrity with branch policies and immutable backup mirrors — Git alone is not sufficient for compliance-grade audit."
