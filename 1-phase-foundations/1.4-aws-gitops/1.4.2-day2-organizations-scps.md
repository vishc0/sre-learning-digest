# AWS Organizations and Multi-Account Governance | Week 4 Day 2

## Why This Matters

Single-account AWS is like running all your microservices in one Kubernetes namespace with no RBAC — it works until it doesn't, and when it fails it fails catastrophically. Multi-account governance through AWS Organizations is the control plane for everything a security-conscious enterprise cares about: blast radius containment, least-privilege at the infrastructure layer, audit separation, and regulatory boundary enforcement. Every Director/VP SRE interview at a company with real compliance posture will probe whether you know why multi-account exists and how to implement guardrails that even privileged admins cannot override.

## Analogy

Think of AWS Organizations like a **franchise corporate structure** layered on top of your RabbitMQ vhost model.

- **Management Account** = Corporate HQ — sets the franchise rules, never runs production workloads directly, just like your RabbitMQ admin vhost never carries application messages
- **Organizational Units (OUs)** = Regional franchise groups — "Northeast Region" contains multiple stores, all bound by corporate policy
- **Member Accounts** = Individual franchise locations — each store (team/environment) has its own P&L (billing), its own staff (IAM users), but cannot violate the franchise agreement (SCPs)
- **SCPs** = The franchise agreement printed in permanent ink — even the store manager (account admin) cannot override "no selling alcohol in dry counties"
- **Control Tower** = The franchise onboarding kit — every new store gets the same baseline setup, audit trail, and guardrails automatically

In incident command terms: SCPs are your **pre-authorized runbook constraints** — they run before any operator action, cannot be bypassed by the operator, and are set by the Incident Commander (central security team) not the responder.

## Vocabulary Translation Table

| Their Term | Your Existing Mental Model |
|---|---|
| Management Account | The "admin vhost" in RabbitMQ — governs all; never runs application workloads |
| Organizational Unit (OU) | A namespace or team partition — groups accounts that share a policy profile |
| Member Account | An isolated blast-radius container — one team, one environment, one AWS account |
| Service Control Policy (SCP) | A global circuit breaker on top of IAM — like a network policy that blocks traffic before firewall rules run |
| Permission Boundary | An IAM ceiling — like a Kubernetes LimitRange but for IAM actions, not compute |
| Control Tower | Account vending machine with golden path embedded — like your CI/CD pipeline for new team onboarding |
| Guardrail (Preventive) | An SCP attached to every OU — denial-by-policy, cannot be worked around |
| Guardrail (Detective) | An AWS Config rule + SNS alert — tells you when someone violated a policy after the fact |
| Delegated Administrator | Giving a specific member account the ability to manage an AWS service org-wide — like granting a sub-team write access to a shared Splunk index |
| Trust Policy vs Permission Policy | Trust policy = who can assume the role (authentication); Permission policy = what they can do once inside (authorization) |

## 30-Second Version

AWS Organizations groups multiple AWS accounts under one management account. SCPs are JSON policies attached at the OU or account level that act as a ceiling — no IAM policy in a member account can grant what an SCP denies, even to the root user. Control Tower automates account creation with embedded guardrails. Permission boundaries complement SCPs by constraining what a specific IAM role can do within an account, even if the admin who created the role has higher permissions. Together these layers mean your security posture is enforced in code, not dependent on humans remembering policy.

## 2-Minute Version

**The Evaluation Order That Changes Everything**

When an AWS API call happens, the authorization engine evaluates in this order:
1. **Organization SCPs** — if there's a Deny here, stop. Full stop. No IAM policy overrides it.
2. **Resource-based policies** (like S3 bucket policies) — can grant cross-account access
3. **IAM identity policies** — the familiar Allow/Deny on users, roles, groups
4. **Permission boundaries** — if attached to a role, the effective permission is the *intersection* of the identity policy AND the boundary
5. **Session policies** — temporary credential scoping

SCPs differ from IAM policies in one critical way: **SCPs do not grant permissions**. They only constrain what CAN be granted. An account still needs IAM policies to actually allow actions — the SCP just sets the outer fence. A wildcard Allow SCP means "nothing blocked at org level," not "everything allowed."

**Why OUs Matter**

A well-designed OU structure mirrors your environments and risk profiles:
```
Root
├── Security OU          ← audit, log archive, security tooling accounts
├── Infrastructure OU    ← shared services, networking, DNS
├── Workloads OU
│   ├── Prod OU          ← tightest SCPs — no manual actions, no public S3
│   ├── Staging OU       ← medium controls
│   └── Dev OU           ← loosest controls but still region-locked
└── Sandbox OU           ← total isolation, time-limited, broad permissions
```

**Permission Boundaries — the Constraint Within an Account**

SCPs constrain at the OU/account level. Permission boundaries constrain at the IAM principal level. Use case: you want to allow developers to create IAM roles for their Lambda functions, but you don't want them to create a role with AdministratorAccess. Attach a permission boundary to all roles they create — the boundary caps the maximum permissions any role they touch can have, even if they have iam:CreateRole rights.

## 5-Minute Version

**Why Single-Account is a Red Flag**

If a company runs all environments in one AWS account, three bad things are true simultaneously:
1. A compromised developer credential can reach production data
2. An IAM misconfiguration can affect all teams simultaneously
3. Billing is opaque — no team-level cost attribution
4. (Bonus) You cannot get ISO 27001 or SOC 2 Type II certification cleanly — auditors want account-level separation for sensitive workloads

When a Director-level interviewer hears "we use one AWS account," they hear "this company has never been audited and security is aspirational, not operational."

**SCP Design Patterns**

Three SCPs every organization should have from day one:

*1. Region Lock*
```json
{
  "Effect": "Deny",
  "Action": "*",
  "Resource": "*",
  "Condition": {
    "StringNotEquals": {
      "aws:RequestedRegion": ["us-east-1"]
    }
  }
}
```
Exempts global services (IAM, Route53, CloudFront) which are not region-scoped. Without the exemption, this breaks identity operations.

*2. Deny Root User API Calls*
```json
{
  "Effect": "Deny",
  "Action": "*",
  "Resource": "*",
  "Condition": {
    "StringLike": {
      "aws:PrincipalArn": "arn:aws:iam::*:root"
    }
  }
}
```
Root credentials should never be used programmatically. This enforces it at the organization layer.

*3. Deny Unapproved Instance Types*
```json
{
  "Effect": "Deny",
  "Action": "ec2:RunInstances",
  "Resource": "arn:aws:ec2:*:*:instance/*",
  "Condition": {
    "StringNotLike": {
      "ec2:InstanceType": ["t3.*", "t3a.*", "m5.*"]
    }
  }
}
```
Prevents GPU instances or oversized families from being launched without central approval.

**Control Tower Guardrails — Two Flavors**

| Guardrail Type | Mechanism | When Violated |
|---|---|---|
| Preventive | SCP | API call denied in real time |
| Detective | AWS Config Rule | CloudWatch alarm + SNS notification after the fact |

Control Tower's value is not the guardrails themselves — any experienced engineer can write SCPs. The value is the **account factory**: every new account gets VPC baseline, CloudTrail enabled, GuardDuty enabled, centralized logging to the log archive account, and breakglass procedures — before the first engineer logs in. This is your golden path for account provisioning.

**Permission Boundaries — Design Tradeoff**

Tradeoff: permission boundaries add operational friction. Every time a developer creates an IAM role, they must attach the boundary (or you automate it via Service Control Policy + IAM conditions). The pattern that scales: use an SCP to require that all IAM roles created in developer accounts carry a specific permission boundary policy ARN. This creates a self-reinforcing guardrail — developers can still create roles, but every role they create is automatically bounded.

**Failure Modes**

1. **SCP blocks global services**: Forget to exempt IAM, Route53, CloudFront from region SCPs. Result: IAM policy changes fail, DNS breaks, certificates cannot be issued.
2. **SCP on management account**: SCPs do NOT apply to the management account. If you run workloads there (you shouldn't), you have a governance gap.
3. **OU depth trap**: AWS allows up to 5 levels of OU nesting. Deep nesting makes SCP inheritance hard to reason about — a Deny at level 2 cannot be overridden at level 4. Keep OUs shallow (2-3 levels max).

## Director/VP Lens

**How a Leader Governs This**

As a Director/VP SRE, your job is not to write SCPs — it's to define the policy architecture and ensure it maps to compliance requirements. The questions you ask:

1. *"Which SCPs are mandatory for all accounts, and which are OU-specific?"* — this is your risk tiering conversation with the CISO.
2. *"How long does it take to provision a new account?"* — if it takes more than 4 hours, you have no golden path and developers are creating shadow accounts. Control Tower account factory should be under 30 minutes.
3. *"What's the break-glass procedure when an SCP blocks legitimate production response?"* — you need a documented, audited escalation path that doesn't involve removing the SCP.
4. *"Are SCPs version-controlled?"* — if SCPs live in the AWS console UI and not in Terraform/code, they are not governed. This is a compliance finding waiting to happen.

**How an IC Implements This**

- Write SCPs in Terraform using `aws_organizations_policy` resource
- Test SCPs in a sandbox OU before attaching to production OUs
- Use AWS IAM Access Analyzer to validate that new SCPs don't block intended paths
- Use CloudTrail + Athena to audit which API calls are being denied by SCPs — this catches misconfigured SCPs before they become incidents

**AI/Automation Angle**

LLMs are being used to draft SCP policies from natural language requirements ("deny all EC2 outside us-east-1 except for disaster recovery"). The risk: LLM-generated SCPs may miss exemptions for global services. Always validate with Access Analyzer before applying to production OUs. As a Director, your governance process should require Access Analyzer sign-off as a merge gate on any SCP change.

## Comprehension Check

Answer 2 of 3 correctly to unlock the lab.

**Q1**: An IAM policy in a member account has `"Action": "*", "Effect": "Allow"` on all resources. An SCP on the parent OU has `"Action": "ec2:*", "Effect": "Deny"`. Can a user in that account launch an EC2 instance? Why or why not?

**Q2**: Your Control Tower detective guardrail fires an alert that S3 bucket ACLs are enabled in the Prod OU. A developer says "I explicitly allowed it in my IAM policy." Why does the guardrail still fire, and what's the difference in how you'd respond compared to a preventive guardrail?

**Q3**: You want developers in the Dev OU to be able to create IAM roles for Lambda functions, but you never want those roles to have S3 full access. SCPs and permission boundaries are both in your toolkit. Which tool do you use and why?

## ANSWER KEY — Comprehension Check

**A1**: No. The user cannot launch an EC2 instance. SCPs are evaluated before IAM policies. An SCP Deny is a hard ceiling — it cannot be overridden by any IAM Allow in the member account, regardless of how permissive the IAM policy is. The effective permission is the intersection of what the SCP allows and what IAM allows. Since the SCP denies ec2:*, no EC2 actions are possible even with a wildcard IAM Allow.

**A2**: The guardrail fires because detective guardrails (AWS Config rules) evaluate the actual resource state, not the IAM policy that created it. IAM policies control who can do what — they don't affect whether a Config rule flags a resource configuration as non-compliant. The developer's IAM Allow is irrelevant to the Config evaluation. Response difference: a detective guardrail means the violation already happened — your response is remediation (disable the ACL, investigate blast radius, update the Config rule to auto-remediate). A preventive guardrail (SCP) would have blocked the action before it happened — no remediation needed, just an access-denied error.

**A3**: Use a **permission boundary** on the IAM roles the developers create. SCPs work at the account/OU level — you'd need to deny S3 full access for all principals in the Dev OU, which is too broad and would also block your CI/CD pipelines and admins. Permission boundaries work at the individual role level — attach a boundary policy to every Lambda execution role that excludes s3:* or caps it to specific buckets. To enforce that developers always attach the boundary, use an SCP with a Condition that requires `iam:PermissionsBoundary` to equal a specific policy ARN whenever `iam:CreateRole` is called.

**WHY**: The `NotAction` pattern (rather than `Action`) is the correct way to write region-lock SCPs. Without the exemptions, IAM, STS, Route53, and billing console operations would all fail because they are globally scoped and don't have a `aws:RequestedRegion` context key.

### Step 5: Create the SCP in AWS Organizations

```bash
# Create the SCP — this registers it in Organizations but does NOT attach it yet
SCP_ID=$(aws organizations create-policy \
  --content file:///tmp/region-lock-scp.json \
  --name "DenyAllRegionsExceptUSEast1" \
  --description "Lab SCP: Blocks all AWS actions outside us-east-1. Exempts global services." \
  --type SERVICE_CONTROL_POLICY \
  --query 'Policy.PolicySummary.Id' \
  --output text)

echo "SCP ID: $SCP_ID"

# Verify the policy was created
aws organizations describe-policy \
  --policy-id "$SCP_ID" \
  --query 'Policy.PolicySummary.{Name:Name, Id:Id, Description:Description}' \
  --output table
```

**EXPECTED_OUTPUT**:
```
SCP ID: p-xxxxxxxxxxxx

---------------------------------------------------------------------------------------------
|                                      DescribePolicy                                       |
+----------------------------------+--------------------+----------------------------------+
|  Description                     |  Id                |  Name                            |
+----------------------------------+--------------------+----------------------------------+
|  Lab SCP: Blocks all AWS...      |  p-xxxxxxxxxxxx    |  DenyAllRegionsExceptUSEast1      |
+----------------------------------+--------------------+----------------------------------+
```

**WHY**: Creating the SCP registers it in your organization's policy library. It has zero effect until attached to an OU or account. This separation allows you to draft and review policies before they impact anyone.

### Step 9: Create a Permission Boundary policy

```bash
# Switch back to your management/training-admin credentials
unset AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN

# Write the permission boundary JSON
# This boundary caps maximum permissions to S3 read-only + CloudWatch logs write
# Even if a role has AdministratorAccess, the effective permission is this boundary
cat > /tmp/lambda-boundary.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3ReadOnly",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket",
        "s3:GetBucketLocation"
      ],
      "Resource": "*"
    },
    {
      "Sid": "AllowCloudWatchLogs",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    },
    {
      "Sid": "DenyDangerous",
      "Effect": "Deny",
      "Action": [
        "iam:*",
        "organizations:*",
        "account:*"
      ],
      "Resource": "*"
    }
  ]
}
EOF

echo "Boundary JSON written"
python3 -m json.tool /tmp/lambda-boundary.json > /dev/null && echo "JSON is valid"

# Create the boundary policy in IAM
BOUNDARY_ARN=$(aws iam create-policy \
  --policy-name "LambdaExecutionBoundary" \
  --policy-document file:///tmp/lambda-boundary.json \
  --description "Permission boundary for Lambda execution roles — caps to S3 read + CloudWatch write" \
  --query 'Policy.Arn' \
  --output text)

echo "Boundary Policy ARN: $BOUNDARY_ARN"
```

**EXPECTED_OUTPUT**:
```
Boundary JSON written
JSON is valid
Boundary Policy ARN: arn:aws:iam::123456789012:policy/LambdaExecutionBoundary
```

**WHY**: The permission boundary is now a reusable policy artifact. It does nothing until attached to an IAM role. Once attached, it acts as a maximum permissions ceiling — the effective permissions of the role become the intersection of its identity policies AND this boundary.

```bash
EOF

# Create the role WITH the permission boundary attached at creation time
# Even though we'll attach AdministratorAccess (intentionally too broad),
# the boundary will cap it
ROLE_ARN=$(aws iam create-role \
  --role-name "BoundedLambdaRole-lab" \
  --assume-role-policy-document file:///tmp/lambda-trust-policy.json \
  --permissions-boundary "$BOUNDARY_ARN" \
  --description "Lab role: has AdministratorAccess attached but bounded by LambdaExecutionBoundary" \
  --query 'Role.Arn' \
  --output text)

echo "Role ARN: $ROLE_ARN"

# Attach a deliberately broad policy — we want to show the boundary RESTRICTS it
aws iam attach-role-policy \
  --role-name "BoundedLambdaRole-lab" \
  --policy-arn "arn:aws:iam::aws:policy/AdministratorAccess"

echo "AdministratorAccess attached"
```

**EXPECTED_OUTPUT**:
```

```

**WHY**: This setup deliberately creates the scenario where a developer attaches a too-permissive policy. The next step proves the boundary prevents exploitation — even with AdministratorAccess in the identity policy, the role cannot exceed what the boundary allows.

## Lab ANSWER KEY

**Step 1 fails — "This operation is only available for the management account"**
Your CLI is not configured for the management account. Run `aws sts get-caller-identity` — if the account ID shown is NOT your management account, switch profiles: `export AWS_PROFILE=management-admin` or reconfigure `aws configure`.

**Step 1 fails — FeatureSet shows CONSOLIDATED_BILLING not ALL**
SCPs require Organizations full features. In the console: AWS Organizations > Settings > Enable All Features. This requires approval from all member accounts.

**Step 5 fails — "PolicyTypeNotEnabledException"**
SCPs are not enabled as a policy type on the root. Fix: `aws organizations enable-policy-type --root-id $ROOT_ID --policy-type SERVICE_CONTROL_POLICY`

**Step 8 — Error says "AccessDenied" but no mention of SCP**
Check that the member account is actually in the test OU: `aws organizations list-parents --child-id $MEMBER_ACCOUNT_ID`. If it's still at root, the SCP doesn't apply. Re-run Step 7.

**Step 8 — us-west-2 bucket creation succeeds instead of being denied**
The SCP may not have propagated yet (rare, takes up to 30 seconds). Also check: are you running as the management account? SCPs do NOT apply to the management account. You must be assuming a role IN the member account.

**Step 11 — simulate-principal-policy shows "allowed" for ec2:RunInstances**
The boundary policy may not have been attached at role creation. Verify: `aws iam get-role --role-name BoundedLambdaRole-lab --query 'Role.PermissionsBoundary'`. If empty, attach it: `aws iam put-role-permissions-boundary --role-name BoundedLambdaRole-lab --permissions-boundary $BOUNDARY_ARN`

**Step 12 — delete-organizational-unit fails — "OrganizationalUnitNotEmptyException"**
The member account is still in the OU. Re-run the move-account step first. An OU must be completely empty (no accounts, no child OUs) before it can be deleted.

**Q3 — SCP blocks incident response**: "We had a near-miss on T-Mobile's platform where a network policy blocked our diagnostic tooling during a P1. The lesson: your runbooks must be tested against your guardrails before incidents happen. I implemented a quarterly 'guardrail fire drill' — we simulate a P1 scenario and verify every tool in our runbook still works. **Q4 — Business case**: "I've made this argument to a VP. Single-account has three hidden costs: (1) audit cost — SOC 2 prep takes 2x longer with shared accounts, typically $50-100K in consultant time; (2) blast radius — one credential compromise can touch all environments, the average breach cost for a mid-size company is $4.5M; (3) developer velocity — without account isolation, a bad deploy in staging can affect production networking, adding toil that I've measured at 10-15 hours per engineer per month."

**Q5 — SCP vs resource policy**: "I got this question at a Hashicorp interview. Resource-based policies can grant cross-account access WITHIN what the SCP allows, but they cannot override an SCP Deny. If Account A's SCP denies s3:GetObject for all principals, no resource-based policy on any S3 bucket in Account A can grant that action to Account B. The SCP is evaluated first, and its Deny is final. This is different from VPC endpoint policies or S3 bucket policies in a same-account context, where the union of policies can create access."

## Tradeoffs and Failure Modes

**Tradeoff 1: SCP coverage vs. operational agility**

Tight SCPs reduce risk surface but increase incident response friction. A region-lock SCP means your on-call engineer cannot spin up a diagnostic instance in a secondary region during a DR scenario. The mitigation is not "loosen the SCP" — it's "build break-glass automation that temporarily expands access through an audited, time-limited IAM role." The tradeoff forces you to make DR procedures explicit rather than relying on operator ability to do anything. This is the right tradeoff for a compliance-sensitive environment, but it requires upfront investment in runbook automation.

**Tradeoff 2: OU structure depth vs. policy inheritance clarity**

AWS allows 5 levels of OU nesting. Deep hierarchies let you be precise — Prod/US-East/EKS-Cluster gets a tighter SCP than Prod/US-East. But SCP inheritance becomes hard to reason about: a principal is governed by the SCPs on every parent OU up to the root, plus the account-level SCP. An Allow at a child OU cannot override a Deny at a parent OU. Complex trees mean engineers routinely misunderstand why an action is denied. The mitigation: keep OUs to 2-3 levels max, document the policy intent at each level in code comments, and use AWS IAM Access Analyzer to validate effective permissions across the tree before any SCP change goes to production.

**Tradeoff 3: Permission boundary adoption vs. developer experience**

Mandatory permission boundaries on all developer-created roles add friction: developers must know the boundary ARN, must understand why their role "isn't working," and the boundary must be maintained as application requirements evolve. If the boundary is too tight, developers work around it by requesting direct IAM permissions from the security team, which creates a bottleneck. If it's too loose, it provides false assurance. The mitigation: treat the boundary policy as a product with a versioned ARN, maintain it in a shared Terraform module, and include a self-service exception process (PR to the module + security review) so developers can request boundary expansions without blocking delivery. The boundary exists to catch mistakes, not to make engineers file tickets for every new API call.
