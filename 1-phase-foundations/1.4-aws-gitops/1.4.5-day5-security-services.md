# AWS Security Services: GuardDuty, Security Hub, Config, CloudTrail, VPC Flow Logs | Week 4 Day 5

## Why This Matters

Every Director-level SRE interview at a cloud-native company will probe your security posture — not just "can you build it" but "how do you govern threat detection, compliance, and incident forensics across a multi-account org?" AWS native security services are the backbone of cloud security operations, and knowing how GuardDuty, Security Hub, Config, CloudTrail, and VPC Flow Logs interlock to produce an incident timeline is the difference between a competent IC answer and a Director answer. In regulated verticals (fintech, telecom, healthcare), auditors want to see these services enabled, rules-as-code for compliance, and evidence of continuous monitoring — not point-in-time scans.

## Analogy

Think of your T-Mobile notification platform's observability stack: Splunk ingests logs, AppDynamics detects anomalies, you have dashboards that aggregate signal from multiple sources, and when a P1 fires you pull Splunk to reconstruct the timeline.

AWS security services are the exact same pattern, just shifted from application observability to security observability:

| Security Service | Observability Equivalent |
|---|---|
| **GuardDuty** | AppDynamics anomaly detection — watches behavior, fires alerts on threats |
| **Security Hub** | Splunk MART dashboard — aggregates findings from all tools into one pane |
| **AWS Config** | Configuration drift detector — like comparing your deployed topology against a known-good baseline |
| **CloudTrail** | The audit log — every API call, equivalent to your RabbitMQ message audit trail |
| **VPC Flow Logs** | Network packet metadata — equivalent to your TIBCO EMS message flow records |

Just as you'd correlate AppDynamics alert + Splunk logs + RabbitMQ queue depth to diagnose a P1, here you correlate GuardDuty alert + VPC Flow Logs + CloudTrail to reconstruct a security incident.

## 30-Second Version

AWS ships five interlocking security services: GuardDuty is your behavioral threat detector (watching CloudTrail + VPC Flow Logs + DNS for malicious patterns); Security Hub is your aggregation layer that normalizes findings from GuardDuty, Inspector, Macie, and Config into one prioritized list; AWS Config is your configuration compliance engine that continuously compares resource state against rules you define as code; CloudTrail is the API audit log for every action taken in your account; and VPC Flow Logs capture network-level traffic metadata. The power is in combining them — a GuardDuty alert is the signal, VPC Flow Logs show the network path, and CloudTrail shows exactly which identity took which action, giving you a complete forensic timeline.

## 2-Minute Version

### GuardDuty: What It Detects and What It Misses

GuardDuty is a managed threat detection service. You enable it, it starts consuming:
- **CloudTrail logs** — looking for unusual API call patterns (calls from Tor, credential abuse, new IAM user immediately doing privilege escalation)
- **VPC Flow Logs** — looking for port scanning, communication with known-bad IPs, unusual data exfiltration volumes
- **Route 53 DNS logs** — looking for DNS requests to known malware C2 (command-and-control) domains

GuardDuty maintains its own threat intelligence — you do not manage the feeds. It produces **findings** categorized by type (Recon, Backdoor, CryptoCurrency, Trojan, UnauthorizedAccess) and severity (Low/Medium/High/Critical).

**What GuardDuty does NOT do:**
- It does not block anything. It detects and alerts only. Think IDS, not IPS.
- It does not see inside your applications (no APM, no code analysis).
- It does not check misconfigurations (that is Config's job).
- It does not scan container images (that is Inspector's job).
- It does not scan S3 objects for PII or sensitive data (that is Macie's job).

### Security Hub: The Aggregation Layer

Security Hub normalizes findings from GuardDuty, Inspector (vulnerability scanning), Macie (S3 data classification), AWS Config, and third-party tools into a single, prioritized list using a common format called ASFF (Amazon Security Finding Format). It also runs AWS Foundational Security Best Practices checks automatically — 200+ checks against CIS benchmarks.

Think of it as the SIEM aggregation layer that sits above your individual detectors.

### AWS Config: Compliance as Code

Config continuously records the state of your AWS resources (IAM, EC2, S3, security groups, etc.) and lets you write **rules** — assertions about what the state MUST be. When a resource drifts out of compliance, Config marks it non-compliant and can trigger automatic remediation via SSM Automation documents.

Example rules:
- `iam-user-mfa-enabled` — all IAM users must have MFA
- `s3-bucket-public-read-prohibited` — no S3 bucket may be public
- `restricted-ssh` — no security group may allow 0.0.0.0/0 on port 22

### CloudTrail: The API Audit Log

CloudTrail records every AWS API call — who called it, from where, at what time, with what parameters, and what the response was. By default it captures **management events** (CreateBucket, RunInstances, AttachRolePolicy). **Data events** (GetObject, PutObject on S3; Invoke on Lambda) are OFF by default and cost extra. This is a critical gap — if someone exfiltrates data by reading S3 objects, default CloudTrail does not record it.

### VPC Flow Logs: Network Metadata

VPC Flow Logs capture packet-level metadata at the network interface, subnet, or VPC level: source IP, destination IP, source port, destination port, protocol, bytes transferred, and ACCEPT/REJECT status. They do NOT capture packet payloads (content). They are the network equivalent of "I know these two hosts talked, for this long, and transferred this much data — but not what they said."

## 5-Minute Version — Tradeoffs, Failure Modes, and Design Decisions

### GuardDuty Architecture Decisions

**Enable or not:** GuardDuty is priced on volume of CloudTrail events and VPC Flow Log data analyzed. In a large org with 100M+ API calls/day, cost can be significant. The tradeoff: the cost of GuardDuty is always cheaper than a breach investigation. For an SRE Manager: enable it in all accounts, use AWS Organizations to do it centrally from a security account, and aggregate findings to the security account — never let individual teams manage their own GuardDuty.

**Finding triage:** GuardDuty generates noise, especially in dev accounts where people do unusual things intentionally. Suppress findings in known-safe contexts (your pentest IP range, your CI/CD runner IPs) using suppression rules rather than disabling GuardDuty. In your T-Mobile context, this is equivalent to tuning Splunk alerts to avoid alert fatigue from synthetic monitors.

**The gap nobody mentions:** GuardDuty for EKS is a separate feature (EKS Audit Log Monitoring + EKS Runtime Monitoring). It is NOT automatically included. If you run EKS and do not explicitly enable these, GuardDuty has zero visibility into your Kubernetes control plane or pod behavior.

### AWS Config — What Actually Breaks in Production

Config stores a configuration history in an S3 bucket you own. If you delete that S3 bucket, you lose your history. The Config recorder must be running continuously — if you stop it during a maintenance window, you have a gap in your audit trail. Auditors hate gaps.

**Remediation gotcha:** Auto-remediation via SSM Automation can create feedback loops. If Config flags a resource non-compliant, triggers remediation, but the remediation itself changes a property that Config evaluates, you can get an infinite remediation loop. Always test remediation automations in a dev account first.

**Conformance Packs:** AWS provides pre-packaged collections of Config rules mapped to compliance frameworks (PCI-DSS, HIPAA, NIST 800-53). Use these as your starting point rather than writing individual rules from scratch — this is the "don't reinvent the wheel" move that a Director would insist on.

### CloudTrail — The Three Gaps That Get Organizations

1. **Data events off by default** — S3 object reads/writes, Lambda invocations, DynamoDB item-level activity are all off unless you explicitly enable them. This is the most common forensic gap. After an S3 data exfiltration, teams often discover they have no log of the GetObject calls.
2. **Log integrity** — CloudTrail logs stored in S3 can be deleted or modified unless you enable log file validation (SHA-256 digest files) AND lock the S3 bucket with Object Lock in COMPLIANCE mode. Without both, an attacker who gains S3 write access can erase their tracks.
3. **Multi-region blind spots** — By default, CloudTrail is per-region. You must create an organization-level trail or enable it in every region. GuardDuty operates globally by default in a region, but if you only look at us-east-1 CloudTrail you will miss API calls made in eu-west-1.

### VPC Flow Logs — What They Miss

VPC Flow Logs do not capture:
- Traffic to/from 169.254.169.254 (EC2 instance metadata service) — this is the target of SSRF attacks that steal credentials
- DNS traffic (use Route 53 Resolver query logs for that)
- Traffic between instances in the same security group if the security group allows it — you see it from the ENI but not at VPC level
- Packet contents — you know the conversation happened, not what was said

For incident response, the winning combination is: GuardDuty alert (threat signal) + VPC Flow Logs (which IPs talked, how much data moved) + CloudTrail (what IAM identity did what API action) + Application logs (what the code did).

### The Incident Correlation Playbook

```
GuardDuty fires: UnauthorizedAccess:IAMUser/ConsoleLoginSuccess.B  
                 (console login from unusual geography)

Step 1: VPC Flow Logs → filter by the source IP from the GuardDuty finding
        → see what internal resources that IP connected to, 
          how much data transferred

Step 2: CloudTrail → filter by the IAM user/role from the finding, 
        last 24 hours → see exactly what APIs were called:
        ListBuckets, GetObject, CreateAccessKey, etc.

Step 3: Config → check current state of that IAM user: 
        MFA enabled? Access keys age? Recent policy changes?

Step 4: Security Hub → are there correlated findings from Inspector 
        (known vulnerabilities on those EC2s) or Macie 
        (sensitive data in those S3 buckets)?

Outcome: Complete incident timeline in 15 minutes, 
         not 3 hours of manual log correlation.
```

## Director/VP Lens

### How an IC implements this:
- Enables GuardDuty in one account, writes a Lambda to send findings to Slack, creates a few Config rules manually, queries CloudTrail ad-hoc when needed.
- Builds it per-account, per-team. Configuration drift across accounts is inevitable.

### How a Director/VP governs this:
- **AWS Organizations + delegated admin**: You designate a single security account as the delegated administrator for GuardDuty, Security Hub, and Config. This means you can see ALL findings from ALL member accounts from one place, without trusting individual teams to set it up correctly.
- **Baseline is non-negotiable**: Define a security baseline (minimum: GuardDuty enabled, CloudTrail enabled with integrity validation, Config enabled with S3-public and MFA rules, Security Hub enabled). Enforce this via AWS Control Tower or SCP (Service Control Policy) — if a team does not have GuardDuty enabled, they cannot disable it. SCPs are the "you cannot do this even if you are root" mechanism.
- **Findings go to SIEM, not just Security Hub**: In a mature org, Security Hub exports to a SIEM (Splunk, Datadog, Sumo Logic) where your security team triages. Security Hub is an aggregator; Splunk is where you investigate.
- **Quarterly compliance reporting**: Config Conformance Packs produce compliance scores per account. You report this to engineering leadership: "87% of resources are compliant with our security baseline; here are the 13% and their owners." This is the governance artifact auditors want.
- **Cost governance**: GuardDuty + CloudTrail data events + VPC Flow Logs + Security Hub all cost money. At Director level, you approve the budget as part of the security program, not chase cost optimization on security services. This is a "you pay for the alarm system, you don't try to negotiate the monthly fee down to zero."
- **AI augmentation**: AWS Security Hub now integrates with Amazon Bedrock for finding summarization. GuardDuty has ML-based baseline models per account. As a Director, you evaluate these AI features for your team's efficiency — fewer false positives, faster triage — not as curiosities but as toil-reduction investments.

## ANSWER KEY — Comprehension Check

**A1:** VPC Flow Logs will give you: the full network conversation metadata — which source IP scanned which destination IP, which ports were probed, how many packets/bytes, and whether the connections were ACCEPTED or REJECTED by your security groups. This tells you the scope of the scan (were all 65535 ports probed? just 22, 443, 3306?), the duration, and the target instances. What VPC Flow Logs will NOT tell you: the identity behind the scan. If the scanning EC2 instance was compromised via stolen credentials, you need CloudTrail to see which IAM identity launched that instance, when, and what other API calls that identity made (did it create new IAM users? download S3 data?). VPC Flow Logs are network forensics; CloudTrail is identity/API forensics. You need both.

**A2:** The critical gap is that **data events are off by default** in CloudTrail. Management events (CreateBucket, RunInstances, AttachRolePolicy) are logged, but S3 object-level reads and writes (GetObject, PutObject) are NOT logged unless you explicitly enable data events — and they cost extra. In an S3 data exfiltration scenario: an attacker who gains access can call GetObject on thousands of files, and none of those reads will appear in your CloudTrail unless data events were pre-enabled. After the breach, you will have no forensic record of what was exfiltrated, from which buckets, at what time. This is the most common post-incident forensic gap in AWS environments.

**A3:** You should use a **centralized Config with AWS Organizations** (aggregator account / delegated admin), NOT configure Config per-account. The right approach: enable Config organization-wide from your management account or a dedicated security account; all member accounts report configuration data to the aggregator; conformance packs and compliance rules deploy to all accounts from one place. The tradeoff: this requires AWS Organizations to be set up correctly and the security account to be the delegated admin for Config. The cost of this centralization is slightly more complex IAM setup and a single-pane S3 bucket that becomes a compliance artifact store. The benefit: you get cross-account compliance visibility, drift detection across the entire org, and a single audit report instead of 15 separate ones. The failure mode: if you only configure Config per-account without an aggregator, each team can independently tune or disable their own compliance rules — you lose the org-wide governance picture.

### Step 7: Create an S3 bucket for AWS Config delivery

```bash
# Config needs an S3 bucket to store configuration snapshots and history
CONFIG_BUCKET="aws-config-training-${ACCOUNT_ID}-$(date +%s)"
echo "Creating Config bucket: $CONFIG_BUCKET"

aws s3 mb s3://$CONFIG_BUCKET --region us-east-1

# Apply a bucket policy that allows Config to write to it
cat > /tmp/config-bucket-policy.json << EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AWSConfigBucketPermissionsCheck",
      "Effect": "Allow",
      "Principal": {
        "Service": "config.amazonaws.com"
      },
      "Action": "s3:GetBucketAcl",
      "Resource": "arn:aws:s3:::${CONFIG_BUCKET}"
    },
    {
      "Sid": "AWSConfigBucketDelivery",
      "Effect": "Allow",
      "Principal": {
        "Service": "config.amazonaws.com"
      },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::${CONFIG_BUCKET}/AWSLogs/${ACCOUNT_ID}/Config/*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": "bucket-owner-full-control"
        }
      }
    }
  ]
}
EOF

aws s3api put-bucket-policy \
  --bucket $CONFIG_BUCKET \
  --policy file:///tmp/config-bucket-policy.json

echo "Config bucket created and policy applied: $CONFIG_BUCKET"
# Save for later steps
echo "export CONFIG_BUCKET=$CONFIG_BUCKET" >> ~/.bashrc_training
```

**EXPECTED_OUTPUT:**

make_bucket: aws-config-training-123456789012-1718122456
Config bucket created and policy applied: aws-config-training-123456789012-1718122456

**WHY:** AWS Config requires an S3 bucket to store configuration snapshots. The bucket policy is the permission that allows the Config service principal to write into your bucket — without this, Config will fail to deliver.

### Step 8: Create the IAM role for AWS Config

```bash
# Config needs an IAM role to describe resources across your account
cat > /tmp/config-assume-role.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "config.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
EOF

# Create the role
aws iam create-role \
  --role-name AWSConfigRole-Training \
  --assume-role-policy-document file:///tmp/config-assume-role.json \
  --query 'Role.Arn' \
  --output text

# Attach AWS managed policy for Config
aws iam attach-role-policy \
  --role-name AWSConfigRole-Training \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWS_ConfigRole

CONFIG_ROLE_ARN="arn:aws:iam::${ACCOUNT_ID}:role/AWSConfigRole-Training"
echo "Config IAM Role ARN: $CONFIG_ROLE_ARN"
```

**EXPECTED_OUTPUT:**
```
arn:aws:iam::123456789012:role/AWSConfigRole-Training
Config IAM Role ARN: arn:aws:iam::123456789012:role/AWSConfigRole-Training
```

**WHY:** The AWS_ConfigRole managed policy grants Config the read-only permissions needed to describe every resource type in your account (EC2 instances, S3 buckets, IAM users, security groups, etc.). This is the principle of least privilege — Config only needs read access to inventory your resources.

## Lab ANSWER KEY — Diagnostics Per Step

**Step 2 — GuardDuty fails to create detector:**
- Error: "GuardDuty is already enabled" → Get existing detector ID with `aws guardduty list-detectors`
- Error: "AccessDenied" → Your IAM user needs `guardduty:CreateDetector` permission; add `AmazonGuardDutyFullAccess` managed policy to training-admin

**Step 3 — No sample findings appear:**
- Wait longer than 15 seconds; some accounts take 2-3 minutes for first findings to appear
- Run `aws guardduty list-findings --detector-id $DETECTOR_ID` without any filter to see if any IDs are returned
- Confirm $DETECTOR_ID is still set: `echo $DETECTOR_ID`

**Step 5 — Security Hub enable fails:**
- Error: "Security Hub is already enabled" → This is fine, proceed. Run `aws securityhub get-enabled-standards` directly.
- Error: "AccessDenied" → Add `SecurityHub:EnableSecurityHub` to your training-admin permissions

**Step 6 — No GuardDuty findings in Security Hub:**
- The integration may take 5-10 minutes. Re-run the query after waiting.
- Confirm GuardDuty and Security Hub are in the same region: `echo $AWS_DEFAULT_REGION`

**Step 10 — Config rule evaluation takes longer than expected:**
- Increase the sleep to 120 seconds. Config evaluation can take 1-3 minutes for IAM resources on first run.
- Manually trigger evaluation: `aws configservice start-config-rules-evaluation --config-rule-names iam-user-mfa-enabled`

**Step 14 — Athena CREATE TABLE fails with SUCCEEDED but no data:**
- The table may be correct but CloudTrail has not delivered any logs to S3 yet (logs are delivered within ~15 minutes of the first API call after enabling the trail). Wait and then run a COUNT(*) query to verify: `SELECT COUNT(*) FROM cloudtrail_logs.cloudtrail_events`
- If query errors with "HIVE_BAD_DATA" → the S3 path in LOCATION is wrong; verify with `aws s3 ls s3://$TRAIL_BUCKET/AWSLogs/$ACCOUNT_ID/CloudTrail/ --recursive` to see the actual prefix structure

**Step 15 — Query returns 0 rows:**
- CloudTrail logs to S3 are not real-time — they arrive within 5-15 minutes. Run this step after you have made several API calls (the steps themselves generate CloudTrail events). Re-run after 15 minutes.
- Check the query ran against the right database: confirm `--query-execution-context Database=cloudtrail_logs`

**Step 18 — S3 bucket delete fails:**
- `An error occurred (BucketNotEmpty)` → Add `--force` flag to rb command, or run `aws s3 rm s3://BUCKET_NAME --recursive` first
- Note the typo in the template: `s3 rb s3/$TRAIL_BUCKET` should be `s3 rb s3://$TRAIL_BUCKET` (two slashes) — fix if present

## Tradeoffs and Failure Modes

### Tradeoff 1: Detection Coverage vs. Cost

GuardDuty + CloudTrail data events + VPC Flow Logs provides excellent coverage but can reach $10,000-$50,000/month in a large enterprise with high API call volume. The temptation is to disable data events or reduce GuardDuty scope to cut costs. The failure mode: an S3 data exfiltration where the attacker reads 10 million objects via GetObject, and you have zero forensic record because data events were disabled to save $3,000/month. The correct tradeoff: apply data events selectively (only buckets tagged `DataClassification: Sensitive`), use GuardDuty in all accounts but at reduced sampling in dev, and use AWS Trusted Advisor or Cost Explorer to right-size rather than remove coverage.

### Tradeoff 2: Config Continuous Recording vs. Evaluation Lag

AWS Config records every configuration change but evaluations against rules can have a lag of minutes to hours, and some rules only evaluate on scheduled intervals (every 24 hours), not on every change. The failure mode: a developer creates a public S3 bucket, the Config rule fires 45 minutes later, and in the meantime data has been read by an external scanner. The correct tradeoff: for highest-criticality rules (public S3, SSH open to 0.0.0.0/0), supplement Config with EventBridge rules that trigger immediately on the specific API calls (PutBucketAcl, AuthorizeSecurityGroupIngress) and invoke a Lambda that either alerts or auto-remediates in under 60 seconds. Config is your compliance history; EventBridge is your real-time control.

### Tradeoff 3: Security Hub Noise vs. Signal

Security Hub with all standards enabled against a fresh AWS account will produce hundreds of findings immediately — many are informational or medium severity items that take months to remediate. The failure mode: alert fatigue where the SOC treats every Security Hub finding as noise, and a real GuardDuty HIGH severity finding gets buried. The correct tradeoff: define a triage SLA by severity (CRITICAL: 4 hours, HIGH: 24 hours, MEDIUM: 2 weeks, LOW: quarterly review), mute findings you have explicitly accepted risk on using suppression rules with a named approver and expiration date, and route CRITICAL/HIGH to PagerDuty while MEDIUM/LOW stay in Security Hub for weekly review. This is the same alert governance model you would apply to your Splunk MART dashboards — the tool is only as good as the triage discipline behind it.

## 5-Minute Version: Full Integration Walkthrough

### Scenario: "Someone Uploaded a Secret to S3"

**Hour 0 (Detection Phase)**:
1. **Config Rule** fires: "S3 bucket encryption not enabled" → Finding in Config dashboard
2. **GuardDuty** (optional, if sensitive bucket is accessed): Logs unusual API patterns

**Hour +30 min (Investigation Phase)**:
1. **Security Hub** aggregates Config finding + any IAM anomalies from GuardDuty
2. You receive on-call alert (via SNS, Slack, etc.)
3. You query **CloudTrail via Athena**: "Show me all S3 PutObject calls in the last 2 hours"

```sql
SELECT eventtime, useridentity.principalid, sourceipaddress, requestparameters
FROM cloudtrail_logs
WHERE eventname = 'PutObject'
  AND requestparameters LIKE '%bucket-name%'
  AND eventtime > '2026-06-11 12:00:00'
ORDER BY eventtime DESC;
```

4. Query returns: `trainer-admin IAM user, 10.2.3.4, 13:15 UTC, uploaded db-password.txt`

**Hour +1 (Response Phase)**:
1. Disable that IAM user's credentials (rotate secrets in Vault)
2. Run another Athena query: "What else did this user do in the last 24 hours?"
3. Document in incident report: "Root cause: IAM user laptop compromised; secret uploaded at 13:15 UTC; rotated 13:45 UTC; MTTR = 30 min"
4. Update Config Rule to enforce bucket encryption + versioning (drift prevention)

## How to Talk About This at the Leadership Level

**The Cost vs. Coverage Trade-off**:
- GuardDuty: $3/month per account → covers all threat detection. **Recommendation**: Enable in all environments (dev, staging, prod). ROI is high (one compromised credential caught = millions in breach cost avoided).
- Security Hub: $0.10 per 1M findings → can be expensive in large multi-account orgs with noisy detectors. **Recommendation**: Centralize in a "security account" (delegated admin model); suppress low-severity findings; suppress duplicate findings across child accounts.
- Config: $2 per active rule → scales quickly if you add 50 rules. **Recommendation**: Start with 5–10 critical rules (public S3, unencrypted RDS, MFA on users). Use **Conformance Packs** to bundle rules by compliance framework (CIS, PCI) rather than hand-curating.
- CloudTrail: $2/trail + S3 storage → storage grows 10–50 GB/month per account. **Recommendation**: Use S3 Lifecycle Policies (move logs to Glacier after 90 days for cost reduction).

**Delegated Admin Model** (for multi-account orgs like T-Mobile):
- Create a central **security account** (separate AWS account)
- Enable GuardDuty delegated admin in that account
- Add all other accounts as "member accounts"
- All findings (from all accounts) flow to the security account
- One team monitors; reduces operational overhead

**Alert Triage SLAs**:
| Severity | Response SLA | Who Responds |
|---|---|---|
| CRITICAL | 15 min | On-call security engineer (escalates to SRE if infrastructure-related) |
| HIGH | 1 hour | SRE on-call (security engineer in business hours) |
| MEDIUM | 4 hours | Scheduled task (next business day if after-hours) |
| LOW | 7 days | Backlog (monthly remediation sprint) |

**Three Questions for Your VP When Pitching Budget**:
1. "Do we have a breach response playbook? These services cost $500/month in a 20-account org; a single breach costs $4M+ (NIST avg). ROI is 100:1 over 2 years."
2. "Are we audit-ready? Config + CloudTrail = compliance evidence. Without it, we fail SOC2/PCI audits."
3. "Can we respond to threats in <30 min? These services cut MTTR from 4 hours to 30 min (NIST data). What's your MTTR SLO?"

**Staffing**: Dedicate 0.5 FTE SRE to review Security Hub findings weekly. Pair with security team on incident response (not siloed).

## Interview Questions (Top 5, Detailed)

1. **"Walk me through how you'd use AWS security services to respond to a breach where an IAM user's credentials were compromised."**
   - **What they're testing**: Do you understand the incident response *flow*? Can you connect multiple tools?
   - **Your answer structure (STAR)**:
     - **Situation**: T-Mobile notification platform, training-admin credentials leaked (phishing).
     - **Task**: Respond in <30 minutes, preserve forensic evidence, prevent lateral movement.
     - **Action**:
       1. GuardDuty alerts on unusual API patterns (high data exfiltration, DescribeInstances from odd IP).
       2. Check Security Hub for related findings (config drifts, new IAM users created by compromised account).
       3. Query CloudTrail Athena: all API calls by training-admin in the last 4 hours.
       4. Identify pivot: attacker created new IAM user `attacker-admin` with AdministratorAccess at 14:32 UTC.
       5. Disable compromised user immediately; rotate secrets in Vault; rotate attacker-admin credentials.
       6. Re-enable compromised user with MFA + restricted policy; run Athena query again: "Did attacker-admin do anything destructive?" (e.g., DeleteSnapshot, DeleteDB).
       7. Document findings in incident report: MTTR = 18 min (from alert to user disabled).
     - **Result**: Breach contained, forensic timeline preserved for postmortem.

2. **"Config Rules say an EC2 instance is NON_COMPLIANT with 'EC2 in VPC' rule. How would you investigate and remediate?"**
   - **What they're testing**: Do you distinguish between "Configuration drift" (how a resource is configured) vs. "Infrastructure state" (whether it should exist)?
   - **Your answer**:
     - First, describe the rule: "Config is checking if the EC2 instance is in a VPC."
     - Check the instance: `aws ec2 describe-instances --instance-ids i-xxx` → does it have a VpcId field?
     - If it doesn't (rare, old-style EC2-Classic), and the org requires VPC, **terminate and rebuild in VPC**.
     - If it does have a VPC but rule still flags NON_COMPLIANT, the rule might have a whitelist of *approved VPCs*. Check: is this instance in the approved VPC list?
     - If not approved, investigate: "Should this instance be in the approved VPC?" (Business decision / runbook).
     - Remediation: Move instance to correct VPC (complex, usually rebuild) OR update rule whitelist (if policy was wrong).
   - **Leadership lens**: This shows "I understand policy vs. reality" — SREs who conflate them make costly mistakes.

3. **"You're designing a multi-account AWS org with 50 accounts. How would you set up GuardDuty, Security Hub, and Config at scale?"**
   - **What they're testing**: Can you architect for *operational scalability*? Do you know about delegated admin?
   - **Your answer (4-minute version)**:
     - Create a dedicated **Security Account** (separate AWS account, using AWS Organizations).
     - **GuardDuty Delegated Admin**: Enable GuardDuty in security account; designate it as delegated admin; add all 50 member accounts.
     - Result: All 50 accounts' findings flow to security account; one team monitors; no duplicate alerts.
     - **Security Hub Delegated Admin**: Same pattern. Central aggregation.
     - **AWS Config Aggregator**: Create central config aggregator in security account; add all member account data.
     - Result: `describe-config-rules` in security account shows compliance across all 50 accounts.
     - **Cost**: GuardDuty $3 × 50 = $150/month. Security Hub ~$50/month. Config $2 × 200 rules = $400/month. Total ~$600/month for org-wide visibility. Breakeven: one $4M breach prevented.
     - **Alert routing**: Security Hub → SNS topic in security account → SRE on-call Slack → escalate to security team if CRITICAL.

4. **"A compliance auditor asks: 'Prove your infrastructure has not been modified without approval in the last 30 days.' How do you respond?"**
   - **What they're testing**: Do you know audit evidence vs. real-time compliance? Do you understand the difference?
   - **Your answer**:
     - Pull CloudTrail logs for the last 30 days (via Athena): filter for all API calls that modify resources (CreateSecurityGroup, ModifyDBInstance, PutBucketPolicy, etc.).
     - Run Athena query: `SELECT eventname, resources, useridentity.principalid, eventtime FROM cloudtrail_logs WHERE eventname LIKE 'Modify%' OR eventname LIKE 'Put%' OR eventname LIKE 'Create%' ORDER BY eventtime;`
     - Generate report: "These 47 changes were made by approved IAM users (deployment automation), timestamped, and logged."
     - Cross-check with Config: "On 2026-06-01, S3 bucket acl was changed to public. Config Rule flagged it as NON_COMPLIANT within 5 minutes. We remediated the same day (Config compliance timeline proves it)."
     - If any unapproved changes: escalate to security/leadership; file incident.
   - **Leadership angle**: This is how you answer: "Are we compliant?" Auditors want *evidence*, not assurance.

5. **"Your team uses Terraform to manage IAM roles. A Config Rule says a role has an inline policy that violates policy (overly permissive). Your Terraform code also has the same inline policy. What's the disagreement, and how do you resolve it?"**
   - **What they're testing**: Do you understand Config as a *reality checker* for IaC? Can you handle IaC-vs.-policy conflicts?
   - **Your answer**:
     - Config Rule is *prescriptive* (what should be); Terraform is *descriptive* (what will be created).
     - The rule says: "IAM roles must not have inline policies with `*:*` permissions" (best practice: use managed policies).
     - Your Terraform code has `inline_policy { ... policy = jsonencode({ Action = "*", Resource = "*" }) ...}` for simplicity.
     - **Decision**: Which is right? Config Rule is right (least-privilege is required by your org policy). Terraform code is wrong (overly permissive).
     - **Fix**: Refactor Terraform:
       - Move inline policy to a managed policy (e.g., `aws_iam_policy` + `aws_iam_role_policy_attachment`).
       - Tighten permissions: `Action = ["ec2:DescribeInstances", "ec2:DescribeTags"]` (only what's needed).
       - Re-apply Terraform; Config Rule re-evaluates and becomes COMPLIANT.
     - **Lesson for team**: "Config Rules are your CI/CD gate. If Terraform passes TF validate but fails Config Rule, we fix Terraform before merging. Config is the source of truth for policy."

## Trade-Offs: 3 Key Decisions

### Trade-Off 1: GuardDuty Sensitivity vs. Alert Fatigue

**The Dilemma**:
- **High sensitivity**: Catch every anomaly (best security).
- **Low sensitivity**: Fewer alerts, easier on-call experience (but miss real threats).

**T-Mobile example**:
- Your platform sees 25M messages/day. If a batch job calls `DescribeSecurityGroups` 10,000 times in parallel (legitimate), does GuardDuty flag it as "unusual API pattern"?
- If yes → alert every day → on-call burns out → stops reading alerts (alert fatigue).
- If no → miss real DDoS reconnaissance (attacker mapping your security groups).

**Resolution**:
- Use GuardDuty's **suppression rules**: "Suppress finding type X from trusted source Y (e.g., IP 10.0.0.1 = your batch job)."
- Add context to findings: "From internal IP (trusted)" vs. "From external IP (investigate immediately)".
- SLA by context: internal anomalies = 4-hour response; external = 15-minute response.

### Trade-Off 2: Config Rule Granularity vs. Maintenance Cost

**The Dilemma**:
- **50 rules**: Every possible compliance requirement. Comprehensive. Hard to maintain; false positives.
- **5 rules**: Only critical controls. Easy to maintain. Risk gaps.

**T-Mobile example**:
- Rule 1 (critical): "S3 public read prohibited" (prevent data leaks).
- Rule 2 (critical): "RDS encryption at rest enabled" (HIPAA requirement).
- Rule 3 (nice-to-have): "EBS volumes have snapshots enabled" (recovery best practice).
- Rule 4 (nice-to-have): "CloudTrail multi-region enabled" (compliance parity).

**Resolution**:
- Start with 5 critical rules (top 20% of risk).
- After 2 months, if team can handle: add 10 more.
- Use **Conformance Packs** (AWS bundles of rules for HIPAA/PCI-DSS) instead of hand-curating.
- Establish SLA: CRITICAL violations (rule 1–2) remediate in 7 days; MEDIUM violations (rule 3–4) remediate in 30 days.

### Trade-Off 3: CloudTrail Storage Cost vs. Query Speed

**The Dilemma**:
- **Full retention (10 years)**: Compliance requirement. Cost: $500+/month in large orgs (S3 storage).
- **Short retention (30 days)**: Cheap. Risk: Can't forensically analyze incidents older than 30 days.

**T-Mobile example**:
- You have a major incident on day 40. Root cause is "someone misconfigured X on day 8." CloudTrail logs from day 8 were archived to Glacier and cost $20 to retrieve + 4-hour retrieval delay.

**Resolution**:
- **Tiered retention**:
  - Days 1–90 (hot): S3 Standard (fast queries, higher cost). Cost: $5–10/month.
  - Days 91–365 (warm): S3 Intelligent-Tiering (auto-archive, slower queries). Cost: $2–5/month.
  - Days 366–2000 (cold): Glacier Deep Archive (retrieve in 24 hours, very cheap). Cost: $1/month.
- **Partition Athena tables by date**: Query only days 1–90 for incident response (<$1); archive older logs for compliance.
- **SLA for forensics**: Can always reconstruct incidents from last 90 days (your MTTR window). Older incidents require 4-hour retrieval + higher cost (escalate to leadership).
