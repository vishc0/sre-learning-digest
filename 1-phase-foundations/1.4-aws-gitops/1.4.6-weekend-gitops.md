# GitOps Platform — Notification Service Deployment Pipeline | Week 4 Weekend

## Why This Matters

GitOps is the operating model that separates mature platform engineering teams from teams still running manual kubectl commands in production. Interviewers at Director/VP level probe whether you can articulate the control plane design, not just that you've used ArgoCD. For your T-Mobile context, a 25M msg/day notification platform is exactly the kind of system where Git-as-source-of-truth prevents the 2am "who applied what?" conversations during incidents.

## Analogy

Think of GitOps like your T-Mobile change management board — except the board is automated. Every change request (Git commit) goes through review (PR), gets approved (merge), and is automatically implemented by a robot (ArgoCD) that also continuously audits whether the running environment matches what was approved. If someone makes an unauthorized change in prod (kubectl edit by a panicking engineer), ArgoCD is the compliance scanner that detects the drift and rolls it back — exactly like an automated config audit that reverts unauthorized firewall rule changes. The "app-of-apps" pattern is your template: one master change request that spawns child requests per environment.

## 30-Second Version

GitOps means Git is the single source of truth for your cluster state. You push a change to a repo, ArgoCD detects the diff between what Git says and what the cluster has, and applies the delta. No human runs kubectl in production. ArgoCD also continuously polls — so if someone manually edits a deployment, ArgoCD detects the drift and reverts it within minutes. This weekend you build this end-to-end for your notification service.

## 2-Minute Version

ArgoCD runs as a controller inside your cluster. It watches Git repositories on a configurable interval (default 3 minutes, or webhook-triggered for sub-minute). For each ArgoCD Application resource, it computes a diff between the desired state (rendered Helm templates from Git) and the live state (what's in etcd). If the diff is non-empty and auto-sync is enabled, it applies the change using server-side apply.

The app-of-apps pattern solves a bootstrapping problem: how do you get ArgoCD to manage its own Application resources? You create one parent Application that points to a directory of other Application manifests. When you add a new environment, you add a new Application manifest to Git — the parent app picks it up and ArgoCD starts managing that environment. No manual ArgoCD UI clicks.

IRSA solves the "how does a pod get AWS credentials" problem without static access keys. The pod's service account is annotated with an IAM role ARN. When the pod starts, an AWS-provided webhook injects a projected token. The AWS SDK exchanges that token for temporary credentials from STS. The notification service uses this to write to SQS and S3 without credentials in the container image or environment variables.

## 5-Minute Version

**Design decisions that interviewers probe:**

**Auto-sync vs. manual sync**: Auto-sync is appropriate for non-production. For production, many teams require a human approval gate — they keep auto-sync off and require a GitOps PR merge to trigger, then a manual "Sync" click in ArgoCD or a pipeline step. The tradeoff: auto-sync gives you faster recovery from drift but removes the human gate that catches bad configs before they hit prod users.

**Prune vs. no-prune**: When you delete a manifest from Git, should ArgoCD delete the corresponding K8s resource? Prune=true means yes. The risk: someone accidentally deletes a ConfigMap from the repo during a refactor — ArgoCD deletes it from prod. Prune=false means orphaned resources accumulate. Most teams use prune=true with required PR reviews as the safety gate.

**Self-heal**: When enabled, ArgoCD reverts manual kubectl changes automatically. This is the right default — but it creates operational friction when you legitimately need an emergency hotfix. The incident response pattern: add an annotation `argocd.argoproj.io/managed-by: emergency-bypass` to pause sync for that resource, apply your fix, then immediately commit the fix to Git and remove the annotation. Never leave self-heal disabled for more than one incident cycle.

**Secrets management with GitOps**: You cannot store secrets in Git. The patterns are: (1) Sealed Secrets — encrypt secrets, store ciphertext in Git, controller decrypts in-cluster; (2) External Secrets Operator — ArgoCD deploys an ExternalSecret CRD that fetches from AWS Secrets Manager or Vault; (3) IRSA + direct SDK calls (what this lab does for SQS/S3). For this lab, IRSA eliminates the need to manage credentials in Git entirely.

**Failure modes**: (1) ArgoCD loses connectivity to Git — it enters a degraded state, stops syncing, but does NOT roll back the cluster; existing workloads keep running. (2) Helm template render fails due to bad values — ArgoCD marks the Application as OutOfSync/Degraded and pages you, but the running workload is unchanged. (3) CRD version mismatch after K8s upgrade — ArgoCD can get stuck; requires manual intervention to update the Application's target revision.

## ANSWER KEY — Comprehension Check

**A1**: ArgoCD detects the drift within 3 minutes (or sooner if webhook-triggered). The self-heal controller runs `kubectl apply` against the Git-desired state, which restores replicas to whatever the Helm values specify. The engineer's emergency scale-down is reversed automatically. The correct approach: either (a) temporarily disable self-heal for that Application in ArgoCD UI, scale down, then re-enable after the incident while committing the proper fix to Git; or (b) use a Helm value override to set `replicaCount: 0` in a fast-track PR merge. The key insight: self-heal treats the cluster as disposable config — Git wins always.

**A2**: (1) Developer opens PR against `notification-service` repo, changes app code. (2) CI pipeline (GitHub Actions) runs: build Docker image → tag with commit SHA → push to ECR → update `values.yaml` or create a PR against `notification-platform-config` updating the image tag. (3) Platform team reviews and merges the config PR. (4) ArgoCD polls `notification-platform-config` (or receives webhook). (5) ArgoCD renders the Helm chart with the new image tag, computes diff against live cluster. (6) ArgoCD applies the diff: updates the Deployment with new image tag. (7) Kubernetes performs rolling update. (8) ArgoCD marks Application as Synced. The separation of app code repo and config repo is intentional — it gives you independent audit trails.

**A3**: App-of-apps solves the bootstrapping and scalability problem. A single Application manifest only manages one environment/service. As you add environments (staging, prod-us-east-1, prod-eu-west-1) or services, you'd need to manually register each Application in ArgoCD. App-of-apps lets you declare all Application manifests as files in a Git directory. The parent Application watches that directory — when you add a new Application manifest file, ArgoCD automatically starts managing that new environment. It also means your ArgoCD Application definitions are themselves version-controlled and auditable in Git, not just configured through the UI.

### Step 2: Build the Helm chart in notification-service repo

```bash
cd ~/notification-service

# Create the Helm chart directory structure
mkdir -p helm/notification-service/templates

# Create Chart.yaml
cat > helm/notification-service/Chart.yaml << 'EOF'
apiVersion: v2
name: notification-service
description: T-Mobile notification platform service — sends 25M msgs/day via SQS+SNS
type: application
version: 0.1.0
appVersion: "1.0.0"
EOF

# Create values.yaml (base defaults — shared across all environments)
cat > helm/notification-service/values.yaml << 'EOF'
# ============================================================
# values.yaml — Base defaults for notification-service Helm chart
# Override per environment in values-staging.yaml, values-prod.yaml
# ============================================================

replicaCount: 2

image:
  repository: ""           # Set per environment — ECR URI goes here
  tag: "latest"            # CI pipeline overwrites this with commit SHA
  pullPolicy: IfNotPresent

service:
  type: ClusterIP
  port: 8080

# Resource requests and limits — tune per environment
resources:
  requests:
    cpu: "100m"            # 0.1 CPU core at baseline
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"

# IRSA: the service account that gets the AWS IAM role annotation
serviceAccount:
  create: true
  name: "notification-service"
  annotations: {}          # IRSA role ARN injected per environment

# SQS configuration — notification service reads from this queue
sqs:
  queueUrl: ""             # Set per environment

# S3 configuration — notification service archives messages here
s3:
  bucketName: ""           # Set per environment

# Health check configuration
healthCheck:
  initialDelaySeconds: 10
  periodSeconds: 15
  path: /health

# Horizontal Pod Autoscaler
hpa:
  enabled: false
  minReplicas: 2
  maxReplicas: 10
  targetCPUUtilizationPercentage: 70

# Pod Disruption Budget — ensures at least 1 pod during node drain
pdb:
  enabled: true
  minAvailable: 1
EOF

# Create values-prod.yaml (production overrides)
cat > helm/notification-service/values-prod.yaml << 'EOF'
# ============================================================
# values-prod.yaml — Production overrides
# This file is referenced by ArgoCD's production Application manifest
# ============================================================

replicaCount: 6            # 6 replicas for 25M msg/day volume

image:
  # Replaced by CI pipeline with actual ECR URI + commit SHA tag
  repository: "123456789012.dkr.ecr.us-east-1.amazonaws.com/notification-service"
  tag: "REPLACE_WITH_COMMIT_SHA"

resources:
  requests:
    cpu: "500m"
    memory: "512Mi"
  limits:
    cpu: "2000m"
    memory: "2Gi"

serviceAccount:
  annotations:
    # IRSA: this annotation is what binds the K8s service account to the IAM role
    # The pod gets temp AWS creds scoped to this role — no static access keys
    eks.amazonaws.com/role-arn: "arn:aws:iam::123456789012:role/notification-service-prod-irsa"

sqs:
  queueUrl: "https://sqs.us-east-1.amazonaws.com/123456789012/notification-prod-queue"

s3:
  bucketName: "notification-archive-prod-123456789012"

hpa:
  enabled: true
  minReplicas: 6
  maxReplicas: 30
  targetCPUUtilizationPercentage: 65
EOF

# Create values-staging.yaml
cat > helm/notification-service/values-staging.yaml << 'EOF'
# ============================================================
# values-staging.yaml — Staging overrides (lower cost, real AWS)
# ============================================================

replicaCount: 2

image:
  repository: "123456789012.dkr.ecr.us-east-1.amazonaws.com/notification-service"
  tag: "REPLACE_WITH_COMMIT_SHA"

serviceAccount:
  annotations:
    eks.amazonaws.com/role-arn: "arn:aws:iam::123456789012:role/notification-service-staging-irsa"

sqs:
  queueUrl: "https://sqs.us-east-1.amazonaws.com/123456789012/notification-staging-queue"

s3:
  bucketName: "notification-archive-staging-123456789012"

hpa:
  enabled: false           # Save cost in staging — no HPA
EOF
```

**EXPECTED_OUTPUT**: No errors. Files created silently.

**WHY**: Two values files per chart is the standard pattern. `values.yaml` is your defaults — safe to commit, contains no secrets. Per-environment files override only what differs. The IRSA role ARN in `values-prod.yaml` is not a secret — it's an ARN (a name, not a credential).

### Step 3: Create Helm templates

```bash
cd ~/notification-service

# Deployment template
cat > helm/notification-service/templates/deployment.yaml << 'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "notification-service.fullname" . }}
  labels:
    {{- include "notification-service.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.replicaCount }}
  selector:
    matchLabels:
      {{- include "notification-service.selectorLabels" . | nindent 6 }}
  template:
    metadata:
      labels:
        {{- include "notification-service.selectorLabels" . | nindent 8 }}
      annotations:
        # Force pod restart when ConfigMap changes (Helm hook pattern)
        checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }}
    spec:
      serviceAccountName: {{ .Values.serviceAccount.name }}
      containers:
        - name: notification-service
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          imagePullPolicy: {{ .Values.image.pullPolicy }}
          ports:
            - containerPort: {{ .Values.service.port }}
          env:
            # SQS queue URL injected from values — not a secret
            - name: SQS_QUEUE_URL
              value: {{ .Values.sqs.queueUrl | quote }}
            # S3 bucket name injected from values
            - name: S3_BUCKET_NAME
              value: {{ .Values.s3.bucketName | quote }}
            # AWS region — IRSA uses this to know which STS endpoint to call
            - name: AWS_REGION
              value: "us-east-1"
          resources:
            {{- toYaml .Values.resources | nindent 12 }}
          livenessProbe:
            httpGet:
              path: {{ .Values.healthCheck.path }}
              port: {{ .Values.service.port }}
            initialDelaySeconds: {{ .Values.healthCheck.initialDelaySeconds }}
            periodSeconds: {{ .Values.healthCheck.periodSeconds }}
          readinessProbe:
            httpGet:
              path: {{ .Values.healthCheck.path }}
              port: {{ .Values.service.port }}
            initialDelaySeconds: {{ .Values.healthCheck.initialDelaySeconds }}
            periodSeconds: {{ .Values.healthCheck.periodSeconds }}
EOF

# Service template
cat > helm/notification-service/templates/service.yaml << 'EOF'
apiVersion: v1
kind: Service
metadata:
  name: {{ include "notification-service.fullname" . }}
  labels:
    {{- include "notification-service.labels" . | nindent 4 }}
spec:
  type: {{ .Values.service.type }}
  ports:
    - port: {{ .Values.service.port }}
      targetPort: {{ .Values.service.port }}
      protocol: TCP
  selector:
    {{- include "notification-service.selectorLabels" . | nindent 4 }}
EOF

# ServiceAccount template — this is what IRSA annotates
cat > helm/notification-service/templates/serviceaccount.yaml << 'EOF'
{{- if .Values.serviceAccount.create -}}
apiVersion: v1
kind: ServiceAccount
metadata:
  name: {{ .Values.serviceAccount.name }}
  labels:
    {{- include "notification-service.labels" . | nindent 4 }}
  annotations:
    {{- toYaml .Values.serviceAccount.annotations | nindent 4 }}
automountServiceAccountToken: true
{{- end }}
EOF

# HPA template — only rendered when hpa.enabled=true
cat > helm/notification-service/templates/hpa.yaml << 'EOF'
{{- if .Values.hpa.enabled -}}
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: {{ include "notification-service.fullname" . }}
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: {{ include "notification-service.fullname" . }}
  minReplicas: {{ .Values.hpa.minReplicas }}
  maxReplicas: {{ .Values.hpa.maxReplicas }}
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: {{ .Values.hpa.targetCPUUtilizationPercentage }}
{{- end }}
EOF

# PodDisruptionBudget — prevents all pods from being evicted during node drain
cat > helm/notification-service/templates/pdb.yaml << 'EOF'
{{- if .Values.pdb.enabled -}}
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: {{ include "notification-service.fullname" . }}
spec:
  minAvailable: {{ .Values.pdb.minAvailable }}
  selector:
    matchLabels:
      {{- include "notification-service.selectorLabels" . | nindent 6 }}
{{- end }}
EOF

# ConfigMap for application config (non-secret config)
cat > helm/notification-service/templates/configmap.yaml << 'EOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ include "notification-service.fullname" . }}-config
  labels:
    {{- include "notification-service.labels" . | nindent 4 }}
data:
  APP_ENV: {{ .Release.Namespace }}
  MAX_RETRIES: "3"
  BATCH_SIZE: "100"
EOF

# _helpers.tpl — Helm named templates (shared across all templates above)
cat > helm/notification-service/templates/_helpers.tpl << 'EOF'
{{/*
Expand the name of the chart.
*/}}
{{- define "notification-service.name" -}}
{{- .Chart.Name | trunc 63 | trimSuffix "-" }}
{{- end }}

{{/*
Create a default fully qualified app name.
*/}}
{{- define "notification-service.fullname" -}}
{{- printf "%s-%s" .Release.Name .Chart.Name | trunc 63 | trimSuffix "-" }}
{{- end }}

{{/*
Common labels — applied to every resource for consistent selection
*/}}
{{- define "notification-service.labels" -}}
helm.sh/chart: {{ .Chart.Name }}-{{ .Chart.Version }}
{{ include "notification-service.selectorLabels" . }}
app.kubernetes.io/version: {{ .Values.image.tag | quote }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end }}

{{/*
Selector labels — used by Service to select pods
*/}}
{{- define "notification-service.selectorLabels" -}}
app.kubernetes.io/name: {{ include "notification-service.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
{{- end }}
EOF
```

**EXPECTED_OUTPUT**: No errors. Files created silently.

**WHY**: The `_helpers.tpl` file defines named templates used across all other templates — this is how Helm avoids repeating the app name and label set in every file. The `checksum/config` annotation on the Deployment is a production pattern: changing a ConfigMap alone does not restart pods, but this checksum annotation forces a rolling restart when config changes.

### Step 8: Create the notification-platform-config repo structure

```bash
cd ~/notification-platform-config

# Create directory structure
mkdir -p argocd/appproject
mkdir -p argocd/applications
mkdir -p environments/staging
mkdir -p environments/prod

# Create AppProject — scopes what ArgoCD can deploy and where
# Think of this as the RBAC boundary for the notification platform team
cat > argocd/appproject/notification-platform-project.yaml << 'EOF'
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: notification-platform
  namespace: argocd
  # Finalizer prevents accidental deletion of project while apps exist
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  description: "Notification platform — owns notification-service and supporting infrastructure"

    - "https://github.com/YOUR_USERNAME/notification-service"
    - "https://github.com/YOUR_USERNAME/notification-platform-config"

    - namespace: staging
      server: https://kubernetes.default.svc    # in-cluster reference
    - namespace: production
      server: https://kubernetes.default.svc

  clusterResourceWhitelist:
    - group: ""
      kind: Namespace

    - group: ""
```

```bash

kind: Application
metadata:
  name: notification-platform-apps

    - resources-finalizer.argocd.argoproj.io
spec:
  project: default              # The parent app uses default project — it only manages ArgoCD objects
  source:
    repoURL: https://github.com/YOUR_USERNAME/notification-platform-config
    targetRevision: HEAD
    path: argocd/applications   # Watch this directory for child Application manifests

    server: https://kubernetes.default.svc
    namespace: argocd           # Child Application manifests land in argocd namespace

    automated:
      prune: true               # Remove Application objects when files are deleted from Git

      - CreateNamespace=true

# Staging Application — deploys notification-service to staging namespace
cat > argocd/applications/notification-service-staging.yaml << 'EOF'
apiVersion: argoproj.io/v1alpha1
kind: Application

  name: notification-service-staging
  namespace: argocd

    - resources-finalizer.argocd.argoproj.io

    environment: staging
    team: notification-platform

  project: notification-platform

    repoURL: https://github.com/YOUR_USERNAME/notification-service

    helm:
      # Layer values files: base first, then environment-specific override
      valueFiles:
        - values.yaml
        - values-staging.yaml

  destination:
    server: https://kubernetes.default.svc
    namespace: staging

  syncPolicy:
    automated:
      prune: true               # Delete K8s resources removed from Helm chart
      selfHeal: true            # Revert manual kubectl changes — CRITICAL for GitOps integrity
    syncOptions:
      - CreateNamespace=true    # Create staging namespace if it doesn't exist
      - PrunePropagationPolicy=foreground   # Delete child resources before parent
      - RespectIgnoreDifferences=true

  # How long ArgoCD waits for resources to become healthy after sync
  # Prevents a bad deployment from being marked Healthy too quickly
  ignoreDifferences:
    # Ignore replica count drift from HPA — HPA manages replicas at runtime
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas

  # Health check timeout — fail if pods aren't Ready within 5 min
  timeout: 300
EOF

# Production Application — tighter controls, no auto-sync
cat > argocd/applications/notification-service-prod.yaml << 'EOF'

kind: Application
metadata:
  name: notification-service-prod
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
  labels:
    environment: production
    team: notification-platform
spec:
  project: notification-platform

  source:
    repoURL: https://github.com/YOUR_USERNAME/notification-service
    # In production, target a specific release tag, not HEAD
    # This means prod only gets what's been explicitly released
    targetRevision: HEAD        # Change to v1.0.0 style tag for real prod
    path: helm/notification-service
    helm:
      valueFiles:
        - values.yaml
        - values-prod.yaml

  destination:
    server: https://kubernetes.default.svc
    namespace: production

  syncPolicy:
    # NO automated sync for production — requires human approval
    # A platform engineer clicks "Sync" in ArgoCD UI or runs argocd app sync

      - CreateNamespace=true

  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas
EOF
```

**EXPECTED_OUTPUT**: No errors.

**WHY**: The critical design decision visible here: staging has `automated.selfHeal: true` and auto-sync. Production has neither. This mirrors real-world GitOps at scale — you want fast feedback in staging, human gates in prod. The `ignoreDifferences` block for `/spec/replicas` is production-essential: without it, ArgoCD fights with HPA over the replica count, creating a constant false-OutOfSync alert.

```bash

# GitOps Design — Notification Platform
## Architecture Decision Record (ADR)

**Status**: Accepted
**Date**: 2024-01-15
**Author**: Platform Engineering Team
**Reviewers**: Vishweshwar Chippa (SRE Manager), Platform Leads

---

## Context

The notification service processes 25M messages/day. Manual deployments via kubectl
created audit gaps, deployment inconsistencies between environments, and extended
MTTR during incidents (no single source of truth for "what is actually running?").

## Decision

Adopt GitOps using ArgoCD with the app-of-apps pattern. All cluster state is
declared in Git. No human runs kubectl apply in production.

## Repository Structure

```
notification-service/          <- App code + Helm chart
  helm/notification-service/
    Chart.yaml
    values.yaml                <- Base defaults
    values-staging.yaml        <- Staging overrides
    values-prod.yaml           <- Production overrides
    templates/                 <- K8s resource templates

notification-platform-config/  <- Deployment declarations
  argocd/
    app-of-apps.yaml           <- Parent Application
    appproject/                <- RBAC scope definitions
    applications/              <- One file per environment per service
  environments/
    staging/                   <- Environment-specific config (future: Kustomize patches)
    prod/
  GITOPS_DESIGN.md             <- This file
```

## Key Design Decisions

### 1. Two-Repo Pattern (not mono-repo)
**Why**: Separates build artifact lifecycle from deployment lifecycle.
App developers own notification-service. Platform team owns notification-platform-config.
Different review gates, different access controls, independent audit trails.

### 2. App-of-Apps for Environment Scaling
**Why**: Adding a new environment (prod-eu-west-1) = add one Application manifest file,
commit, push. ArgoCD picks it up automatically. No ArgoCD UI clicks, no
manual argocd app create commands.

### 3. Auto-Sync Only in Staging
**Why**: Staging needs fast feedback — auto-sync + self-heal catches problems immediately.
Production requires human approval gate (PR merge + explicit sync) to prevent
automated deployment of breaking changes to live traffic.

### 4. IRSA Instead of Static Credentials
**Why**: Pods need SQS/S3 access. Options:
- Static IAM access keys in K8s Secrets: HIGH RISK — rotated manually, often leaked
- EC2 node IAM role: all pods on the node share one role — violates least privilege
- IRSA (IAM Roles for Service Accounts): each pod gets scoped temp credentials — CORRECT

### 5. ignoreDifferences for /spec/replicas
**Why**: HPA dynamically adjusts replica count at runtime. Without this, ArgoCD
detects the runtime replica count differs from the Helm-declared count and
creates constant false OutOfSync alerts. We own HPA's output — we don't own
ArgoCD overriding it.

## Deployment Flow

```
Developer → PR to notification-service (app code + Helm chart)
         → CI: build image → push to ECR → open PR to notification-platform-config
         → Platform review → merge to notification-platform-config
         → ArgoCD detects diff → syncs staging automatically
         → Staging validation (automated tests + SLO check)
         → Platform lead approves production sync
         → ArgoCD syncs production
```

## Emergency Change Procedure (P1 Break-Glass)

1. SRE on-call pauses self-heal: `argocd app set notification-service-staging --self-heal=false`
2. Apply emergency fix: `kubectl apply -f emergency-patch.yaml`
3. Simultaneously open PR with fix committed to Git
4. Re-enable self-heal within 30 minutes of incident resolution
5. Post-mortem includes: "was the Git commit the source of truth within 30 min?" SLO

## Tradeoffs Accepted

| Tradeoff | Accepted Because |
|---|---|
| Slower prod deploys (human gate) | Safety > speed for 25M msg/day platform |
| Two repos to maintain | RBAC separation worth operational cost |
| ArgoCD as new SPOF | ArgoCD HA (3 replicas + Redis) + runbook for ArgoCD-down scenario |
| Helm complexity vs. raw YAML | Parameterization at scale requires templating |

EOF

git add GITOPS_DESIGN.md
git commit -m "docs: add GitOps architecture decision record

Captures two-repo pattern rationale, IRSA decision, app-of-apps design,
emergency change procedure, and accepted tradeoffs."
git push
```

**EXPECTED_OUTPUT**:

**WHY**: Architecture Decision Records (ADRs) are what distinguish Principal/Director-level engineers from senior ICs. ## STAR Anchor

**For Q1 (two-repo pattern)**:
- **S**: T-Mobile notification platform — multiple teams contributing to a 25M msg/day service, with different review gates needed for app code vs. production deployment configs
- **T**: Needed audit trail separation between "what the app does" and "where it runs and at what scale"
- **A**: Implemented two-repo GitOps: `notification-service` owned by app team, `notification-platform-config` owned by SRE with CODEOWNERS enforcement requiring SRE lead approval for production namespace changes
- **R**: Reduced unauthorized production changes from ~3/month to 0; deployment audit trail now satisfies compliance review without separate ServiceNow tickets; MTTR on deployment-caused incidents dropped from 45 min to 12 min because Git history is the incident timeline

**For Q2 (emergency change process)**:
- **S**: We had exactly this scenario — an SRE scaled down a RabbitMQ consumer during a storm to reduce Cassandra load, ArgoCD reverted it, consumer backlog exploded
- **T**: Needed a break-glass procedure that doesn't disable GitOps integrity permanently
- **A**: Implemented: (1) argocd app patch command in runbook to disable self-heal with required Slack notification, (2) 30-minute SLO to commit the emergency change to Git, (3) self-heal re-enable as incident close checklist item, (4) post-mortem required if Git wasn't updated within SLO
- **R**: Emergency changes now have a defined path that's fast (under 2 minutes to disable self-heal), auditable (Slack + Git commit), and auto-reverts back to GitOps control without requiring a follow-up ticket

**For Q5 (PCI-DSS + secrets)**:
- **S**: Our notification platform touches payment notification flows — PCI adjacent
- **T**: Had to design a secrets architecture that keeps GitOps benefits while meeting "no credentials in Git" requirement
- **A**: Three-layer approach: (1) IRSA for AWS service credentials (SQS, S3, SNS) — zero secrets in Git or environment variables; (2) External Secrets Operator pulling from AWS Secrets Manager for application secrets (DB passwords, API keys) — only the SecretStore reference (non-sensitive) lives in Git; (3) Sealed Secrets for K8s-native secrets that need Git portability. Endpoint URLs are not secrets — they live in values files. Only credentials are secrets.
- **R**: Passed PCI DSS compliance audit with GitOps architecture intact; secret rotation is now automated through Secrets Manager rotation policies without any Git commits required

## Tradeoffs and Failure Modes

**Tradeoff 1 — Auto-sync in staging creates unexpected rollbacks during debugging**

The failure: A developer is `kubectl exec`-ing into a staging pod, making config changes to debug a connection issue. ArgoCD detects the pod's ConfigMap was patched (if they edited it directly) and reverts it. The developer's debug session produces invalid conclusions because the config they were testing was silently reverted.

The mitigation: Document that staging is ephemeral — all config changes must go through Git even for debugging. Provide a `debug namespace` with self-heal disabled for free-form experimentation. Label the operational model: "staging validates the pipeline, debug validates assumptions."

**Tradeoff 2 — ArgoCD as a new SPOF in the deployment path**

The failure: ArgoCD's application controller OOMKills during a large cluster event. No new deployments can be applied. If this happens during a P1 that requires a hotfix deployment, you have a double failure.

The mitigation: (1) ArgoCD HA mode with 3 application controller shards and Redis HA; (2) runbook for "apply without ArgoCD" — engineers know how to run `helm upgrade` directly in documented break-glass scenarios; (3) ArgoCD's own availability as an SLO — alert if argocd-server is unavailable for more than 5 minutes. The key insight: ArgoCD going down does not take down production — existing workloads keep running. Only new deployments are blocked.

**Tradeoff 3 — Git history as your change log means Git hygiene is now operational hygiene**

The failure: A developer force-pushes to the main branch of `notification-platform-config`, rewriting history. ArgoCD's sync pointer now references a commit SHA that no longer exists. ArgoCD enters a ComparisonError state for every Application pointing to that repo. All syncs fail until the issue is resolved.

The mitigation: (1) Branch protection rules on `notification-platform-config` — no force pushes, no direct commits to main, required PR reviews; (2) ArgoCD should target a specific tag or branch SHA, not `HEAD`, in production; (3) Monitor for ArgoCD ComparisonError status in your observability stack — this is a P2 incident trigger. The governance principle: your config repo is an operational system, not a developer playground. It needs the same protections as your production cluster.
