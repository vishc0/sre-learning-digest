# EKS Cost Optimization and FinOps | Week 4 Day 4

## Why This Matters

Cloud cost is no longer an engineering afterthought — at Director/VP level, you own the P&L line for infrastructure. EKS clusters routinely run 3-5x overprovisioned because teams optimize for reliability without a cost feedback loop. For a 25M msg/day platform like yours, a 20% compute reduction translates directly to hundreds of thousands of dollars annually — and the ability to present unit economics (cost per message) to a CFO is what separates SRE leaders from SRE contributors.

## Analogy

Think of EKS cost the way T-Mobile would think about network capacity billing. You pay for the radio towers (control plane — flat fee regardless of traffic), the backhaul links (data transfer between AZs — billed per GB like long-distance circuits), and the cell sites themselves (EC2 nodes — pay by the minute). Over-provisioning a cell site for a load spike that never came is money lost. **Spot instances are like off-peak spectrum leases** — you get cheap capacity that the carrier can reclaim with 2 minutes' notice, so you only run interruptible workloads on them. FinOps is your NOC for spend: you need the same alerting rigor on cost anomalies that you'd apply to P1 latency spikes.

## 30-Second Version

EKS cost has five layers: the flat control plane fee, EC2 nodes (the big one), data transfer (especially cross-AZ — the hidden killer), load balancers, and storage. The fastest wins are: (1) right-size your pod requests using Goldilocks so nodes actually fill up, (2) shift stateless workloads to Spot via Karpenter, (3) eliminate unnecessary cross-AZ traffic with topology-aware routing. For leadership, translate all of this into unit economics — "we process 25M messages/day at $X per million" — and your CFO conversation becomes a capability discussion, not a cost-cutting argument.

## 2-Minute Version

### EKS Cost Components Broken Down

**1. Control Plane: $0.10/hr (~$72/month)**
Fixed. Always running. You pay this even on an idle cluster. For a team running dev/staging/prod clusters, this is $216/month before a single pod runs. Governance action: share non-prod clusters across teams, use `eksctl delete cluster` after lab sessions.

**2. EC2 Node Groups: 70-80% of total EKS spend**
The size and count of your worker nodes. The killer mistake: over-setting pod `requests` (what Kubernetes uses to schedule) relative to actual usage. If every pod requests 2 CPU but uses 0.3, your nodes are 85% idle — you're paying for ghost capacity. VPA + Goldilocks surfaces this without disrupting production.

**3. Cross-AZ Data Transfer: $0.01/GB each direction**
Looks trivial. Is not. A notification platform sending 25M messages/day with a sidecar that calls a service in a different AZ adds up fast. Example: 1TB/day cross-AZ = $10/day = $300/month = $3,600/year — invisible until you look at VPC Flow Logs. Fix: Kubernetes topology-aware routing (`topologyKey: topology.kubernetes.io/zone`), service `externalTrafficPolicy: Local`, or AWS Load Balancer Controller with AZ affinity.

**4. Load Balancers: $0.008/hr per ALB + $0.008/LCU**
Each service of type `LoadBalancer` creates an NLB/ALB. A 50-service cluster might have 20 load balancers. Fix: use one ingress controller (AWS ALB Ingress or NGINX) with path-based routing — consolidate to 2-3 load balancers total.

**5. Spot + Karpenter: The 60-70% node cost reduction**
Spot instances are EC2 capacity that AWS reclaims with a 2-minute interruption notice when demand rises. For stateless, retryable workloads (microservices, batch jobs, notification processing), this is acceptable. Karpenter's `NodePool` lets you say: "try Spot first across 10 instance families; if unavailable, fall back to On-Demand." Diversifying across families (m5, m5a, m6i, m6a, c5) dramatically reduces the chance of mass eviction — the pool is deep.

## 5-Minute Version

### Spot Instance Safety Design

The 2-minute warning is not a guarantee of 2 minutes — it is a best-effort signal. Design principles:
- **Stateless only**: Your RabbitMQ broker is not a candidate. Your notification dispatcher service is.
- **Diversify instance families**: Never pin to one type (e.g., m5.2xlarge). Use Karpenter's `requirements` to allow 5-10 instance families — when one family gets reclaimed at scale, others are available.
- **Pod Disruption Budgets (PDB)**: Set `maxUnavailable: 1` so Spot interruptions don't drain more than one replica simultaneously.
- **Graceful shutdown hooks**: `terminationGracePeriodSeconds: 60` + a `preStop` hook that drains in-flight work before the pod dies.
- **Baseline On-Demand**: Run 1-2 On-Demand replicas per critical service as an anchor — Spot handles burst.

### Right-Sizing: Requests vs Limits vs Actual

- **Requests** = what the scheduler uses to place your pod. If requests are too high, pods won't fit on nodes even if nodes are idle. This is the #1 cause of over-provisioned clusters.
- **Limits** = the ceiling. CPU limit = throttling (pod slows down, doesn't die). Memory limit = OOM kill (pod dies and restarts). Setting memory limits too low causes phantom restarts that look like app bugs.
- **Goldilocks recommendation**: Set `requests` = P50 of actual usage. Set `limits` = P99 of actual usage. This gives you bin-packing efficiency without risking OOM at peak.

### Savings Plans vs Reserved Instances for EKS

| | Savings Plans | Reserved Instances |
|---|---|---|
| Commitment | $/hr compute spend | Specific instance type + AZ |
| Flexibility | Any instance family, size, OS | Locked to what you reserved |
| Discount | 20-40% | Up to 72% |
| Best for EKS | Yes — Karpenter changes instance types dynamically | Only for stable, predictable node types |
| Risk | Low — you shift instance types freely | High if Karpenter picks different types than you reserved |

**Director decision**: For a Karpenter-managed cluster, always use Compute Savings Plans, not Reserved Instances. Karpenter's value is picking the cheapest available instance — Reserved Instances fight against that.

### Kubecost: What It Actually Does

Kubecost reads Kubernetes resource allocation (requests, limits, actual usage via metrics-server), maps it to real AWS pricing via the AWS Cost and Usage Report (CUR), and produces per-namespace, per-label, per-deployment cost reports. It also shows:
- **Efficiency score**: actual usage / requested capacity — 40% means you're wasting 60% of what you provisioned
- **Cost per namespace**: critical for chargeback to application teams
- **Network cost attribution**: with the Kubecost network cost daemonset, it can break down cross-AZ traffic by service

Limitation: Kubecost's free tier covers single-cluster. Multi-cluster visibility (comparing dev vs prod cost trends) requires the paid tier or an OpenCost deployment aggregated into Grafana.

### FinOps for Directors: CFO Language

Never walk into a CFO meeting with "we spent $45K on Kubernetes last month." Walk in with:

**"We process 25 million notifications per day at $0.0018 per notification. That's below our $0.002 unit cost target. The proposed Spot migration reduces that to $0.0011 — a $7K/month saving at current volume — and positions us for the 40M/day roadmap without a cost increase."**

This reframes infrastructure spend as a unit economics story:
- **Unit cost** = total compute + transfer cost / messages processed
- **Efficiency ratio** = cost at current volume / cost at 2x volume (shows scalability)
- **Cost avoidance** = what you would have spent without optimization × risk-adjusted probability

Tools for this: Kubecost exports + a simple Python notebook that divides monthly cost by event volume from your application metrics. One 20-line script, run monthly, produces the slide your CFO actually wants.

**IC implementation focus:**
- Setting VPA `updateMode: Off` so recommendations don't auto-apply in production
- Writing Karpenter NodePool YAML with correct instance family spread
- Reading Kubecost namespace cost reports and flagging inefficient deployments

**Director governance focus:**
- Establishing a **FinOps ritual**: monthly cost review cadence, same rigor as SLO review
- **Chargeback model**: each team owns their namespace cost — Kubecost label taxonomy enforces this. No more "infrastructure budget" as a shared pool that nobody optimizes.
- **Cost efficiency SLO**: "EKS efficiency score must stay above 50%" — treated as a reliability metric, not a finance metric
- **Savings Plans purchase authority**: Director owns the commitment decision (1yr vs 3yr Savings Plans), calibrated against roadmap confidence. A 3yr commitment made 6 months before a platform migration is a Director-level mistake.
- **Spot adoption gate**: before migrating a service to Spot, require: (1) PDB present, (2) graceful shutdown proven in load test, (3) retry logic documented. IC delivers the proof; Director owns the risk sign-off.
- **AI angle**: LLM-based cost anomaly detection (OpenAI API call in a Python script querying Kubecost API) can flag "namespace X spent 3x more than last week" with natural language root cause suggestions — early-stage but used at Netflix and Shopify today.

## ANSWER KEY — Comprehension Check

**A1.** A 35% efficiency score means only 35% of your provisioned compute is actually being used — you're paying for 2.85x more capacity than your workloads consume. The first step is to run Goldilocks (or read the VPA recommendations directly) to see which namespaces and deployments have the widest gap between `requests` and actual usage. Then compare: the biggest-gap deployments are your highest-ROI right-sizing targets. You do NOT immediately change production requests — you gather 7-14 days of VPA data to see P50/P99 before modifying anything.

**A2.** Cassandra is a stateful, data-persistence-critical workload. A Spot interruption mid-write with insufficient replication could cause data loss or split-brain. The correct answer is: **no, not for Cassandra nodes themselves**. However, Spot is appropriate for: Cassandra read-only replicas (if your replication factor allows losing one node), Cassandra repair jobs, or the application tier that *talks to* Cassandra. Conditions for any stateful Spot consideration: RF >= 3, rolling eviction protection via PDB, tested node replacement time under 5 minutes, and write acknowledgment mode (QUORUM) that tolerates one node loss.

**A3.** Investigation steps: (1) Open AWS Cost Explorer → filter by "Data Transfer" service → compare last quarter to previous. If it's a significant line item, confirm. (2) Enable VPC Flow Logs on your VPC and query in CloudWatch Logs Insights or Athena: filter for `srcaddr` and `dstaddr` in different AZ CIDR ranges, aggregate bytes by source service tag. (3) Use `kubectl get svc -A` to find services without `externalTrafficPolicy: Local` — these bounce traffic across AZs. Fix estimate: topology-aware routing typically reduces cross-AZ traffic 40-70% for services with multiple replicas. At $0.01/GB each way, present the current monthly GB volume × $0.02 round-trip × 0.55 reduction = monthly savings. Example: 500TB/month cross-AZ = $10K/month → fix saves ~$5,500/month.

### Step 2: Deploy a Sample Over-Provisioned Application

You need something running with obviously wrong resource requests to see Goldilocks recommendations. This deployment intentionally requests way more than it uses.

```bash
# Create a namespace for our demo workload
kubectl create namespace finops-demo

# Deploy a simple nginx with over-provisioned resource requests
# This requests 500m CPU and 512Mi memory but uses almost none of that
cat <<'EOF' | kubectl apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: over-provisioned-app
  namespace: finops-demo
  labels:
    app: over-provisioned-app
    team: platform
    cost-center: infra
spec:
  replicas: 3
  selector:
    matchLabels:
      app: over-provisioned-app
  template:
    metadata:
      labels:
        app: over-provisioned-app
        team: platform
    spec:
      containers:
      - name: nginx
        image: nginx:stable
        resources:
          requests:
            cpu: "500m"       # Requesting half a CPU core
            memory: "512Mi"   # Requesting 512MB RAM
          limits:
            cpu: "1000m"      # Limit: 1 full CPU core
            memory: "1Gi"     # Limit: 1GB RAM
        ports:
        - containerPort: 80
EOF

# Wait for pods to be running
kubectl rollout status deployment/over-provisioned-app -n finops-demo --timeout=60s

# Check actual usage vs requests
kubectl top pods -n finops-demo
```

**EXPECTED_OUTPUT:**
```
deployment.apps/over-provisioned-app condition met

NAME                                    CPU(cores)   MEMORY(bytes)
over-provisioned-app-xxxxxxxxx-xxxxx    1m           3Mi
over-provisioned-app-xxxxxxxxx-xxxxx    1m           3Mi
over-provisioned-app-xxxxxxxxx-xxxxx    1m           3Mi
```

**WHY:** You requested 500m CPU per pod (1500m total for 3 pods) but the pods are each using ~1m CPU. This is the waste Goldilocks will surface. Memory requested: 512Mi each; actual: ~3Mi each. This ratio (500:1 on CPU) is exaggerated for the lab but not unusual in real organizations.

### Step 3: Install Vertical Pod Autoscaler (VPA) in Recommendation Mode

VPA is the engine. Goldilocks is the human-readable dashboard on top. You must install VPA first.

```bash
# Clone the Kubernetes autoscaler repo to get VPA install scripts
git clone https://github.com/kubernetes/autoscaler.git /tmp/autoscaler-vpa 2>/dev/null || \
  (cd /tmp/autoscaler-vpa && git pull)

cd /tmp/autoscaler-vpa/vertical-pod-autoscaler

# Install VPA components
# IMPORTANT: updateMode will be set to "Off" in our VPA objects — recommendations only, no auto-apply
./hack/vpa-up.sh

# Verify VPA components are running
kubectl get pods -n kube-system | grep vpa

# This should show three components:
# vpa-admission-controller  — intercepts pod creation to apply recommendations
# vpa-recommender           — watches actual usage and computes recommendations
# vpa-updater               — applies recommendations to running pods (we disable this)
```

**EXPECTED_OUTPUT:**
```
Cloning into '/tmp/autoscaler-vpa'...

vpa-admission-controller-xxxxxxxxx-xxxxx   1/1     Running   0          45s
vpa-recommender-xxxxxxxxx-xxxxx            1/1     Running   0          45s
vpa-updater-xxxxxxxxx-xxxxx                1/1     Running   0          45s
```

**WHY:** `updateMode: Off` is critical for production safety. VPA in auto-update mode can evict pods during business hours to resize them — that's a surprise incident waiting to happen. In recommendation mode, it watches and advises; you apply changes during maintenance windows.

### Step 4: Create VPA Objects for Your Deployments

Tell VPA to watch your over-provisioned app but NOT auto-update it.

```bash
# Create a VPA object in recommendation-only mode
cat <<'EOF' | kubectl apply -f -
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: over-provisioned-app-vpa
  namespace: finops-demo
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: over-provisioned-app
  updatePolicy:
    updateMode: "Off"    # CRITICAL: recommendations only — never auto-evict pods
  resourcePolicy:
    containerPolicies:
    - containerName: nginx
      minAllowed:
        cpu: 10m          # Never recommend below 10m CPU
        memory: 32Mi      # Never recommend below 32Mi memory
      maxAllowed:
        cpu: 2000m        # Never recommend above 2 CPU cores
        memory: 2Gi       # Never recommend above 2Gi memory
EOF

# Wait a few minutes for VPA recommender to collect data, then check recommendations
# Note: first recommendation may take 3-5 minutes
sleep 180

kubectl describe vpa over-provisioned-app-vpa -n finops-demo | grep -A 30 "Recommendation:"
```

**EXPECTED_OUTPUT (after 3-5 minutes):**
```
  Recommendation:
    Container Recommendations:
      Container Name:  nginx
      Lower Bound:
        Cpu:     10m
        Memory:  32Mi
      Target:
        Cpu:     11m        # VPA says: you only need ~11m CPU, not 500m
        Memory:  32Mi       # VPA says: you only need ~32Mi memory, not 512Mi
      Uncapped Target:
        Cpu:     11m
        Memory:  16Mi
      Upper Bound:
        Cpu:     87m
        Memory:  196Mi
```

**WHY:** VPA's "Target" is the P50 recommendation. You requested 500m CPU; VPA recommends 11m. That's a 45x reduction. At scale across 50 deployments, this level of over-provisioning means you need 45x fewer CPU cores than scheduled — the root cause of the 35% efficiency scores seen in real clusters.

### Step 7: Calculate Cross-AZ Traffic Cost Using CloudWatch

Now quantify the hidden cost of cross-AZ traffic. This uses real CloudWatch metrics from your VPC.

```bash
# First, find your VPC ID
VPC_ID=$(aws ec2 describe-vpcs \
  --filters "Name=tag:alpha.eksctl.io/cluster-name,Values=$(kubectl config current-context | cut -d'/' -f2)" \
  --query 'Vpcs[0].VpcId' \
  --output text \
  --region us-east-1)

echo "VPC ID: $VPC_ID"

# Check if VPC Flow Logs are enabled for this VPC
aws ec2 describe-flow-logs \
  --filter "Name=resource-id,Values=$VPC_ID" \
  --region us-east-1 \
  --query 'FlowLogs[*].{LogGroupName:LogGroupName,Status:FlowLogStatus}' \
  --output table

# Get cross-AZ data transfer bytes from CloudWatch (EC2 metric)
# This queries bytes transferred between AZs for the last 24 hours
aws cloudwatch get-metric-statistics \
  --namespace AWS/EC2 \
  --metric-name NetworkOut \
  --dimensions Name=VpcId,Value=$VPC_ID \
  --start-time $(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --period 86400 \
  --statistics Sum \
  --region us-east-1 \
  --output table 2>/dev/null || echo "VpcId dimension not available — using node-level metrics instead"

# Alternative: Get network stats per node (more reliable)
aws cloudwatch get-metric-statistics \
  --namespace AWS/EC2 \
  --metric-name NetworkOut \
  --start-time $(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --period 86400 \
  --statistics Sum \
  --region us-east-1 \
  --output table
```

**EXPECTED_OUTPUT:**

### Step 8: Calculate Cross-AZ Cost (Manual Calculation Script)

Run this Python script to estimate what cross-AZ traffic would cost at various traffic volumes. This is the script you'd run monthly for CFO reporting.

```bash
# Create the cost calculation script
cat <<'EOF' > /tmp/cross_az_cost_calc.py
#!/usr/bin/env python3
"""
Cross-AZ Traffic Cost Calculator
Use this monthly to estimate hidden data transfer costs.
Input: daily message volume, average message size, percentage routed cross-AZ
Output: monthly cost + unit economics (cost per message)
"""

# --- INPUTS (adjust these for your platform) ---
messages_per_day = 25_000_000          # 25M messages/day (T-Mobile notification platform)
avg_message_size_bytes = 1_500         # 1.5KB average notification payload
cross_az_percentage = 0.30             # 30% of traffic crosses AZ boundaries (conservative estimate)
aws_cross_az_rate_per_gb = 0.01        # $0.01/GB each direction (us-east-1 2024 pricing)
working_days_per_month = 30

# --- CALCULATIONS ---
daily_bytes = messages_per_day * avg_message_size_bytes
daily_gb = daily_bytes / (1024 ** 3)
cross_az_daily_gb = daily_gb * cross_az_percentage

# Data transfer is billed in BOTH directions (sender pays $0.01, receiver pays $0.01)
cross_az_daily_cost = cross_az_daily_gb * aws_cross_az_rate_per_gb * 2  # × 2 for bidirectional

monthly_cross_az_gb = cross_az_daily_gb * working_days_per_month
monthly_cost = cross_az_daily_cost * working_days_per_month

# Topology-aware routing reduces cross-AZ traffic by ~55% for services with multi-AZ replicas
savings_with_topology_routing = monthly_cost * 0.55
new_monthly_cost = monthly_cost - savings_with_topology_routing

# Unit economics
cost_per_million_messages = (monthly_cost / (messages_per_day * working_days_per_month)) * 1_000_000

print("=" * 60)
print("CROSS-AZ TRAFFIC COST ANALYSIS")
print("=" * 60)
print(f"\nTraffic Profile:")
print(f"  Daily messages:          {messages_per_day:>15,.0f}")
print(f"  Avg message size:        {avg_message_size_bytes:>15,} bytes")
print(f"  Daily data volume:       {daily_gb:>15.2f} GB")
```

### Step 9: Create a Karpenter NodePool with Spot Fallback to On-Demand

This is the production-grade Spot configuration. Karpenter selects from a wide family of instance types to maximize Spot availability.

**Prerequisites:** Karpenter must be installed on your cluster. If following the training program, this was covered in Week 3. Verify with:

```bash
kubectl get pods -n karpenter
```

If Karpenter is not installed, apply this NodePool to your default managed node group instead (the Spot preference becomes a label for future Karpenter installation).

```bash
# Create a Karpenter NodePool that prefers Spot, falls back to On-Demand
# This covers stateless workloads (APIs, dispatchers, workers — NOT databases)
cat <<'EOF' | kubectl apply -f -
apiVersion: karpenter.sh/v1beta1
kind: NodePool
metadata:
  name: spot-with-ondemand-fallback
spec:
  template:
    metadata:
      labels:
        node-lifecycle: spot-preferred    # Label nodes for pod affinity rules
        workload-type: stateless
    spec:
      nodeClassRef:
        apiVersion: karpenter.k8s.aws/v1beta1
        kind: EC2NodeClass
        name: default                     # References your EC2NodeClass (subnet/SG config)
      
      requirements:
        # Prefer Spot; fall back to On-Demand automatically when Spot is unavailable
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["spot", "on-demand"]  # Karpenter tries Spot first based on cost
        
        # Allow a broad range of instance families — this is the key to Spot availability
        # Never pin to one family — Spot pools are family-specific
        - key: node.kubernetes.io/instance-type
          operator: In
          values:
            - m5.large
            - m5.xlarge
            - m5.2xlarge
            - m5a.large      # AMD variant — same perf, often cheaper Spot
            - m5a.xlarge
            - m6i.large      # 6th gen Intel
            - m6i.xlarge
            - m6a.large      # 6th gen AMD
            - m6a.xlarge
            - c5.large       # Compute-optimized (good for CPU-bound workloads)
            - c5.xlarge
            - c6i.large
            - c6i.xlarge
        
        # Only provision in us-east-1 AZs — spread across all three for resilience
        - key: topology.kubernetes.io/zone
          operator: In
          values: ["us-east-1a", "us-east-1b", "us-east-1c"]
        
        # Only Linux AMD64 nodes
        - key: kubernetes.io/arch
          operator: In
          values: ["amd64"]
        
        - key: kubernetes.io/os
          operator: In
          values: ["linux"]
      
      # Disruption policy: how aggressively to replace nodes
      disruption:
        consolidationPolicy: WhenUnderutilized   # Remove nodes that are < 20% utilized
        consolidateAfter: 30s                    # Wait 30s of underutilization before evicting
        expireAfter: 720h                        # Replace nodes every 30 days (security patching)
  
  # Resource limits: prevent runaway Spot provisioning
  limits:
    cpu: "100"          # Never provision more than 100 vCPUs in this NodePool
    memory: 400Gi       # Never provision more than 400GiB memory
  
  weight: 100           # Higher weight = Karpenter prefers this NodePool over others

---
# Pod Disruption Budget for your workloads — REQUIRED before going to Spot
# This prevents Spot interruptions from taking down more than 1 replica at a time
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: over-provisioned-app-pdb
  namespace: finops-demo
spec:
  maxUnavailable: 1         # At most 1 pod can be unavailable at any time
  selector:
    matchLabels:
      app: over-provisioned-app
EOF

# Verify the NodePool and PDB were created
kubectl get nodepool spot-with-ondemand-fallback
kubectl get pdb -n finops-demo
```

**EXPECTED_OUTPUT:**
```
nodepool.karpenter.sh/spot-with-ondemand-fallback created
poddisruptionbudget.policy/over-provisioned-app-pdb created

NAME                           READY   AGE
spot-with-ondemand-fallback    True    10s

NAME                       MIN AVAILABLE   MAX UNAVAILABLE   ALLOWED DISRUPTIONS   AGE
over-provisioned-app-pdb   N/A             1                 2                     10s
```

**WHY:** The 12-instance-family spread is the most important part of this configuration. If you only allow `m5.2xlarge` and AWS reclaims all `m5.2xlarge` Spot capacity in `us-east-1a`, your workload has nowhere to go. With 12 family options across 3 AZs, the probability of simultaneous eviction approaches zero. The PDB is the safety net: even if Karpenter needs to drain a Spot node, it can only disrupt one pod at a time.

## Lab ANSWER KEY

**Step 1 — `kubectl top nodes` shows "error: metrics not available":**
Metrics Server takes 60-90 seconds to start collecting data. Wait 2 minutes and retry. If still failing: `kubectl logs -n kube-system deployment/metrics-server` — look for "TLS handshake error." Fix: add `--kubelet-insecure-tls` to the metrics-server args.

**Step 2 — Pods are in `Pending` state:**
Your nodes may be out of capacity after requesting 500m CPU × 3 pods = 1500m total. Check: `kubectl describe pod <pod-name> -n finops-demo` — look for "Insufficient cpu" in Events. Fix: scale your node group up by 1 node, or reduce replicas to 1 for the lab.

**Step 3 — VPA install script fails with "CustomResourceDefinition already exists":**
VPA CRDs may already be installed from a previous lab. Safe to ignore this error. Verify VPA pods are running: `kubectl get pods -n kube-system | grep vpa`.

**Step 4 — VPA shows no recommendations after 5 minutes:**
VPA Recommender needs at least a few minutes of metrics data. If after 10 minutes there are still no recommendations: `kubectl logs -n kube-system deployment/vpa-recommender` — look for "Failed to get pod container list." This usually means metrics-server is not healthy.

**Step 5 — Goldilocks dashboard pod is CrashLoopBackOff:**
Check logs: `kubectl logs -n goldilocks deployment/goldilocks-dashboard`. Common cause: VPA CRDs not found. Ensure VPA was installed successfully in Step 3. The `--set vpa.enabled=false` flag assumes you installed VPA manually.

**Step 8 — Python script shows all zeros for unit economics:**
The message volume (25M) and size (1.5KB) produce small numbers. This is correct — the platform is efficient. Change `avg_message_size_bytes` to `50000` (50KB) to see larger numbers. The script logic is correct; the outputs reflect the inputs.

**Step 9 — NodePool shows `Ready: False`:**
Karpenter NodePools require an EC2NodeClass named "default" to be present. If this is a fresh cluster without the default EC2NodeClass, create it: `kubectl get ec2nodeclass` to see what's available, then update the `nodeClassRef.name` in the NodePool to match.

**Step 9 — `nodepool` resource not found (API error):**

## Tradeoffs & Failure Modes

### 1. Spot Interruption at Scale During a Mass AWS Reclamation Event
**The risk:** AWS reclaims Spot capacity at the instance-family level across a region during peak demand periods (e.g., US East Coast 9am business traffic surge). If you used only 2-3 instance families, you could lose 60% of your Spot nodes in 2 minutes simultaneously — faster than your PDB can protect against.

**The failure mode:** Pod Disruption Budgets protect against *scheduled* disruption (Karpenter draining nodes). Spot interruption is *unscheduled* — all pods on an interrupted node die simultaneously, bypassing the PDB. A 2-replica service with both replicas on the same Spot node of the same family is a single point of failure.

**The fix:** (1) 8+ instance families in your NodePool. (2) Pod anti-affinity rule: `requiredDuringSchedulingIgnoredDuringExecution` with `topologyKey: node.kubernetes.io/instance-type` — forces replicas onto different instance types. (3) Always keep at minimum 1 On-Demand replica per critical service. This is the Director-level governance rule: "no service runs 100% Spot."

---

### 2. VPA in Auto Mode Causing Production Incidents
**The risk:** VPA `updateMode: Auto` evicts pods to resize them. It does not respect your deployment's rollout strategy — it can evict multiple pods from different deployments simultaneously, effectively causing a self-inflicted incident during business hours.

**The failure mode:** VPA updater decides three deployments need resizing at the same moment. It evicts one pod from each. If those pods land on already-saturated nodes or if the new pod requests are larger than available node capacity, pods go Pending. Alert fires: "service degraded" — root cause: automated VPA resize, not a code change or traffic spike.

**The fix:** Always use `updateMode: Off` in production. Apply VPA recommendations manually during scheduled maintenance windows. Treat the VPA target as input to your quarterly resource audit, not a live controller. The only exception: development/staging namespaces where pod restarts are acceptable.

---

### 3. Savings Plans Commitment vs. Karpenter's Dynamic Instance Selection
**The risk:** You commit to $5,000/hr Compute Savings Plans based on your current node fleet (m5.2xlarge heavy). Karpenter then shifts to cheaper c6i.xlarge instances that use Compute Savings Plans. But if the instance selection shifts to types outside your Savings Plan coverage, you get On-Demand pricing for the new instances while still paying the Savings Plan commitment — paying twice.

**The failure mode:** Compute Savings Plans cover all instance types automatically (this is less of a risk for Compute SP than EC2 Instance SP). However, EC2 Instance Reserved Instances — which lock to a specific instance type and AZ — will be completely wasted if Karpenter decides to use a different family. A common mistake: buying `m5.2xlarge` Reserved Instances because that's what your managed node groups use, then installing Karpenter which prefers `m6i.xlarge` for cost reasons. Result: unused RI capacity charged at full price.

**The fix:** For Karpenter clusters, use only **Compute Savings Plans** (not EC2 Instance Savings Plans, not Reserved Instances). Compute Savings Plans apply to any instance family/size/AZ automatically. The Director decision is: commit conservatively (cover your On-Demand baseline, not your peak), review coverage quarterly as Karpenter usage data matures. Year 1 with Karpenter: Savings Plans only. Year 2: calibrate RI coverage based on proven steady-state instance family usage.
