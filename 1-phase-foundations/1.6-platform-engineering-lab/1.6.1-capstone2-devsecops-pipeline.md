## Analogy

Think of your current T-Mobile notification platform deployment pipeline as a package moving
through a sortation facility.

**Traditional pipeline**: Package goes in one end, comes out the other. No one inspects it.
No chain of custody. If something toxic is inside, it reaches the customer.

**DevSecOps pipeline**: Every package goes through:
1. X-ray scanner at intake (SAST — scan the code before it becomes a binary)
2. Ingredient manifest required (SBOM — list every library inside)
3. Known-dangerous contents check (SCA/CVE scan — compare manifest to threat database)
4. Tamper-evident seal applied at packing (image signing — Cosign + Sigstore)
5. Customs policy gate at destination (OPA — runtime policy enforcement)
6. Chain of custody log (SLSA provenance — cryptographic record of who built what, when, where)

If any check fails, the package is rejected BEFORE it ships — not recalled from customer hands.

## Vocabulary Translation Table

| Their Term | Your Existing Mental Model |
|---|---|
| SAST (Static Analysis) | Splunk log parsing before an event fires — inspect the artifact before it runs |
| SCA (Software Composition Analysis) | Dependency audit — every RabbitMQ client lib version you depend on, checked against CVE DB |
| SBOM (Software Bill of Materials) | Ingredient list on a food package — every library, version, license in the binary |
| Image Signing (Cosign) | GPG-signing a release artifact — proves it came from your pipeline, not a compromised registry |
| OPA (Open Policy Agent) | A Rego-scripted policy engine — like Splunk alert rules, but for infrastructure decisions |
| SLSA (Supply-chain Levels for Software Artifacts) | SOC2 compliance levels, but for build pipelines — Level 0 (nothing) to Level 4 (hermetic) |
| Provenance | Chain of custody document — who built it, what source commit, what build system, when |
| Vulnerability Gate | Your Sev1 threshold in incident command — P0/P1 block deployment, P2/P3 warn and log |
| Sigstore / Rekor | A public transparency log — like a blockchain receipt for your build, tamper-evident |
| DAST (Dynamic Analysis) | Chaos testing at the HTTP layer — probe the running app for injection/auth flaws |

## 30-Second Version

DevSecOps pipelines shift security left: scan code before it becomes an artifact, scan the
artifact before it becomes a deployment, and enforce policy before deployment reaches prod.
The three pillars are: **detect** (SAST/SCA), **attest** (SBOM + signing), **enforce** (OPA + gates).

---

## 2-Minute Version

A secure CI/CD pipeline has five security checkpoints layered on top of the standard
build/test/deploy flow:

**1. Pre-build (code scan)**
SAST tools (Semgrep, CodeQL) scan source code for injection flaws, hardcoded secrets,
and insecure patterns. Secret scanning (Gitleaks, Trufflehog) ensures no API keys are
committed. This runs on EVERY pull request — not just on merge.

**2. Build (dependency scan)**
Before compiling, SCA tools (Trivy, Grype, Snyk) audit your dependency tree against
CVE databases. A Critical CVE blocks the build. High CVEs generate a warning with a
tracked exception workflow.

**3. Post-build (artifact attestation)**
After building the container image: generate an SBOM (Syft — CycloneDX format),
sign the image with Cosign (keyless, using GitHub OIDC), and publish the SLSA
provenance document. These three artifacts travel with the image forever.

**4. Policy gate (OPA)**
Before deploying to any environment, OPA Gatekeeper or a CI-embedded Rego policy
evaluates the image against rules: must be signed, must have no Critical CVEs,
must not run as root, must come from an approved registry.

**5. Runtime (DAST + drift detection)**
OWASP ZAP or Nuclei performs dynamic scanning against a staging URL. Falco or
AWS GuardDuty detects runtime anomalies post-deploy.

The full chain creates a cryptographically verifiable record: "This exact binary, built
from commit abc123, was signed by pipeline run #42, has no Critical CVEs, and was
deployed by an authorized system."

## 5-Minute Version — Tradeoffs, Failure Modes, Design Decisions

### Design Decision 1: Keyless Signing vs Key-Based Signing

**Keyless (Cosign + Sigstore OIDC)**: Uses GitHub Actions OIDC token as the signing identity.
No private key to rotate or leak. Provenance is published to Rekor transparency log (public).
Tradeoff: Requires internet access to Rekor; air-gapped environments need an internal Rekor instance.

**Key-based (Cosign with KMS)**: Private key stored in AWS KMS. Fully air-gap compatible.
Tradeoff: Key rotation lifecycle is now YOUR problem. If the key leaks, all historical signatures
are suspect until you re-sign everything.

**For this portfolio**: Use keyless. It's the cloud-native default and what FAANG interviewers expect.

### Design Decision 2: When to Block vs Warn

Blocking every High CVE will cause alert fatigue and developer revolt — the same problem you
see when Splunk alerts are miscalibrated. The pragmatic gate:

| Severity | Action | Rationale |
|---|---|---|
| Critical (CVSS 9.0+) | Hard block — pipeline fails | Exploitable in the wild; unacceptable risk |
| High (CVSS 7.0–8.9) | Block if fix available; warn if no fix | Avoids blocking when upstream hasn't patched |
| Medium (CVSS 4.0–6.9) | Warn + create Jira ticket | Track but don't stop delivery |
| Low / Info | Log only | Noise reduction |

Exception workflow: Developers can open a tracked exception (GitHub Issue + label) that
a security team member approves. Exception expires in 30 days.

### Design Decision 3: SBOM Format — CycloneDX vs SPDX

**CycloneDX** (JSON/XML): Rich vulnerability metadata, native Dependency-Track support,
better toolchain integration. Default for most SRE teams.

**SPDX**: Linux Foundation standard, required for some government/regulated contracts (NTIA mandate).

**Pragmatic answer**: Generate both. Syft can output both formats in one invocation.
For this portfolio, CycloneDX is primary; SPDX is secondary.

### Failure Mode 1: Pipeline Bypass

If developers can push directly to `main` without going through CI, the entire pipeline
is theater. Mitigation: Branch protection rules (require PR + status checks), enforce
with GitHub Rulesets or OPA Gatekeeper admission webhook.

### Failure Mode 2: SBOM Drift

The SBOM is accurate at build time but your running container accumulates OS-level CVEs
over time as the base image ages. Mitigation: Weekly scheduled re-scan of deployed images
in ECR (Amazon Inspector or Trivy cron job). Alert when a new Critical CVE appears in a
running image's SBOM.

### Failure Mode 3: OPA Policy Bypass via Registry Confusion

An attacker mirrors your image to a different registry and deploys it directly, bypassing
the signing check. Mitigation: OPA policy enforces `allowedRegistries` — only images from
`123456789.dkr.ecr.us-east-1.amazonaws.com` are admitted. Combined with IRSA, no other
registry can be pulled in the cluster.

## Director/VP Lens

### How a Director/VP governs this vs how an IC implements it

**IC implements**:
- Writes the GitHub Actions YAML, configures Trivy, wires up Cosign
- Tunes CVE thresholds, writes Rego policies
- Debugs why a specific image signature is failing verification

**Director/VP governs**:
- Mandates pipeline template adoption across all 15 teams — one golden path, not 15 snowflakes
- Sets the exception SLA: "Any Critical CVE exception requires VP approval within 24 hours"
- Reports to CISO/CTO: "X% of our workloads have signed images; Y% have current SBOMs"
- Defines the SLSA level target per workload tier (customer-facing: Level 2 minimum; internal tools: Level 1)
- Owns the quarterly supply chain risk review — presents to the board as part of SOC2/ISO27001

**AI in this domain (2025 lens)**:
- AI-assisted remediation: Snyk and GitHub Advanced Security now suggest auto-fix PRs for many CVEs
- AI policy generation: Describe a security requirement in plain English; generate Rego
- Risk: AI-generated code inherits training data vulnerabilities — SAST matters MORE, not less
- Director play: Mandate AI-assisted remediation but require human approval for any auto-merge on security-tagged PRs

**Q1**: You have a container image with one Critical CVE in `libssl` and no upstream patch is
available. Your pipeline is configured to hard-block on Critical CVEs. What do you do, and
what process do you put in place so this doesn't silently block all deployments?

**Q2**: A developer asks: "Why do we need to sign images if we already scan them for
vulnerabilities?" How do you explain the difference between scanning (what's INSIDE) and
signing (who BUILT it and where FROM)?

**Q3**: Your CTO asks you to achieve "SLSA Level 3." You currently have a standard GitHub
Actions pipeline with no hermetic builds. What is the gap between your current pipeline
(SLSA Level 2) and Level 3, and what would it cost to close?

---

## ANSWER KEY — Comprehension Check

**A1**: Create a tracked exception with a time-bound expiry (30 days maximum). Exception requires:
documented CVE ID, confirmed no upstream patch, compensating control (e.g., WAF rule blocking
the attack vector), approval from security team lead. Modify the pipeline gate to allow the build
to proceed with the exception label present, but flag the image with a `security-exception` tag.
Set a calendar reminder to re-check for a patch when the exception expires.
The process answer is as important as the technical answer — this shows operational maturity.

**A2**: Scanning tells you what ingredients are in the package. Signing tells you who sealed it
and where. A supply chain attack (SolarWinds, XZ Utils) doesn't introduce a CVE — it introduces
MALICIOUS CODE into a legitimate build. The vulnerability scanner wouldn't flag it because it's
not in the CVE database yet. Signing + provenance means you can verify: "This image was built
by pipeline run #42 on GitHub Actions, from commit abc123, and has not been modified since."
If an attacker compromises your registry and substitutes a malicious image, the signature check
fails — even if the malicious image has zero CVEs.

**A3**: SLSA Level 2 requires: version-controlled build scripts, hosted build service (GitHub Actions),
authenticated provenance. Level 3 adds: hardened build platform (GitHub Actions with no
`pull_request` trigger from forks, no self-hosted runners) and non-falsifiable provenance
(the build system generates the provenance, not the pipeline script itself). The gap is primarily
architectural: you cannot use self-hosted runners, you need GitHub's own provenance generator
(`slsa-framework/slsa-github-generator`), and you must eliminate any script step that could
forge the provenance. Cost: primarily engineering time (~2 weeks for a motivated IC) plus loss
of self-hosted runner flexibility. Level 4 (hermetic, reproducible builds) is significantly
more expensive and is typically only justified for critical infrastructure artifacts.

### Step 3 — Write the GitHub Actions pipeline (core)

This is the main artifact. Write it in sections so each block's purpose is clear.

```bash
cat > /mnt/c/Work/Training/portfolio/devsecops-pipeline/.github/workflows/devsecops-pipeline.yml << 'YAML'
# =============================================================================
# DevSecOps Pipeline — Full Supply Chain Security
# Author: Vishweshwar Chippa
# Purpose: Demonstrates SLSA Level 2 pipeline with SAST, SCA, SBOM, signing
#
# Pipeline stages:
#   1. secret-scan      — Block committed secrets before anything builds
#   2. sast             — Static code analysis (Semgrep)
#   3. build            — Build container image
#   4. sca-scan         — Scan image for CVEs (Trivy)
#   5. sbom             — Generate Software Bill of Materials (Syft/CycloneDX)
#   6. sign             — Sign image with Cosign (keyless via GitHub OIDC)
#   7. provenance       — Generate SLSA provenance attestation
#   8. policy-gate      — OPA/Conftest policy evaluation
#   9. deploy-staging   — Deploy to staging (manifest update)
#  10. dast             — Dynamic scan against staging URL (OWASP ZAP)
# =============================================================================

name: DevSecOps Pipeline

on:
  push:
    branches: [ "main", "release/*" ]
  pull_request:
    branches: [ "main" ]
  # Allow manual trigger for testing
  workflow_dispatch:

# Minimal permissions by default — grant only what each job needs
# This is SLSA Level 2 requirement: least-privilege workflow permissions
permissions:
  contents: read

env:
  # GitHub Container Registry — free, no external dependencies for portfolio demo
  REGISTRY: ghcr.io
  # IMAGE_NAME uses the GitHub repo name automatically
  IMAGE_NAME: ${{ github.repository }}

jobs:

  # ===========================================================================
  # JOB 1: Secret Scanning
  # Blocks the pipeline if any secrets/API keys are found in the commit
  # Analogous to: checking RabbitMQ message payload before routing it
  # ===========================================================================
  secret-scan:
    name: Secret Scan (Gitleaks)
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
        with:
          # Fetch full history so Gitleaks can scan all commits in the PR
          fetch-depth: 0

      - name: Run Gitleaks
        uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          # GITLEAKS_LICENSE is only required for org-level scanning
          # For public repos, the free tier covers this use case

  # ===========================================================================
  # JOB 2: Static Application Security Testing (SAST)
  # Scans source code for security vulnerabilities before compilation
  # Analogous to: Splunk search over raw logs before they hit the index
  # ===========================================================================
  sast:
    name: SAST (Semgrep)
    runs-on: ubuntu-latest
    # Run SAST in parallel with secret-scan — they don't depend on each other
    permissions:
      # Semgrep needs to post findings to the Security tab
      security-events: write
      contents: read
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Run Semgrep
        uses: returntocorp/semgrep-action@v1
        with:
          # python ruleset covers Flask-specific vulnerabilities
          # p/owasp-top-ten covers OWASP A01-A10 (injection, broken auth, etc.)
          config: >-
            p/python
            p/owasp-top-ten
            p/secrets
        env:
          SEMGREP_APP_TOKEN: ${{ secrets.SEMGREP_APP_TOKEN }}

  # ===========================================================================
  # JOB 3: Build Container Image
  # Only runs after secret-scan and sast pass — gates downstream jobs
  # ===========================================================================
  build:
    name: Build Image
    runs-on: ubuntu-latest
    needs: [ secret-scan, sast ]
    permissions:
      contents: read
      packages: write   # Required to push to GHCR
    outputs:
      # Pass the full image digest to downstream jobs
      # Using digest (sha256:...) not tag — tags are mutable, digests are immutable
      image-digest: ${{ steps.build.outputs.digest }}
      image-ref: ${{ steps.build.outputs.imageid }}
      tags: ${{ steps.meta.outputs.tags }}

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3
        # Buildx enables cache, multi-platform, and BuildKit (required for provenance)

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata (tags, labels)
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            # Tag with commit SHA for immutability
            type=sha,prefix=,suffix=,format=long
            # Tag with branch name for human readability
            type=ref,event=branch
            # Tag PRs distinctly so they don't pollute main tags
            type=ref,event=pr
            # Semantic version tags if a git tag is pushed
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}

      - name: Build and push image
        id: build
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          # Enable SBOM and provenance generation at build layer (SLSA Level 2)
          sbom: true
          provenance: true
          # Cache layers in GitHub Actions cache — speeds up rebuild
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ===========================================================================
  # JOB 4: Software Composition Analysis (SCA) — Trivy CVE Scan
  # Scans the BUILT image against CVE databases
  # Critical CVEs fail the pipeline. High CVEs with no fix warn only.
  # Analogous to: Sev1 incident gate — only Sev1 pages the on-call at 3am
  # ===========================================================================
  sca-scan:
    name: SCA / CVE Scan (Trivy)
    runs-on: ubuntu-latest
    needs: [ build ]
    permissions:
      security-events: write   # Upload SARIF to GitHub Security tab
      contents: read
    steps:
      - name: Checkout (for Trivy config)
        uses: actions/checkout@v4

      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          # Scan the image we just built, referenced by digest for immutability
          image-ref: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ needs.build.outputs.image-digest }}
          format: sarif
          output: trivy-results.sarif
          # GATE: Exit code 1 (fail pipeline) on CRITICAL severity only
          # HIGH severity generates a finding but does not fail the build
          # This matches the vulnerability gate design in the architecture doc
          exit-code: '1'
          severity: 'CRITICAL'
          # Ignore CVEs where no fix is currently available
          # Rationale: blocking on unfixable CVEs creates alert fatigue with no remediation path
          ignore-unfixed: true

      - name: Upload Trivy SARIF to GitHub Security tab
        uses: github/codeql-action/upload-sarif@v3
        if: always()   # Upload even if the scan found issues (for visibility)
        with:
          sarif_file: trivy-results.sarif

      - name: Run Trivy for full report (non-blocking, all severities)
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ needs.build.outputs.image-digest }}
          format: table
          # No exit-code here — this is a visibility scan, not a gate
          severity: 'CRITICAL,HIGH,MEDIUM'
          ignore-unfixed: false

  # ===========================================================================
  # JOB 5: SBOM Generation
  # Generates a Software Bill of Materials in CycloneDX and SPDX formats
  # Stored as a signed attestation attached to the image in the registry
  # ===========================================================================
  sbom:
    name: Generate SBOM (Syft)
    runs-on: ubuntu-latest
    needs: [ build ]
    permissions:
      contents: read
      packages: write   # Required to attach attestation to GHCR image
      id-token: write   # Required for keyless signing of the SBOM attestation
    steps:
      - name: Install Syft
        uses: anchore/sbom-action/download-syft@v0

      - name: Generate SBOM (CycloneDX JSON)
        uses: anchore/sbom-action@v0
        with:
          image: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ needs.build.outputs.image-digest }}
          format: cyclonedx-json
          output-file: sbom-cyclonedx.json
          # Upload as artifact for 90-day retention
          upload-artifact: true
          upload-artifact-retention: 90

      - name: Generate SBOM (SPDX JSON) — secondary format for compliance
        uses: anchore/sbom-action@v0
        with:
          image: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ needs.build.outputs.image-digest }}
          format: spdx-json
          output-file: sbom-spdx.json
          upload-artifact: true
          upload-artifact-retention: 90

      - name: Attach SBOM as attestation to image
        # This stores the SBOM in the registry, linked to the image digest
        # Anyone who pulls the image can retrieve and verify the SBOM
        run: |
          # Install Cosign for SBOM attestation
          curl -sL https://github.com/sigstore/cosign/releases/latest/download/cosign-linux-amd64 \
            -o /usr/local/bin/cosign && chmod +x /usr/local/bin/cosign

          # Attach SBOM as attestation (keyless — uses GitHub OIDC)
          cosign attest \
            --predicate sbom-cyclonedx.json \
            --type cyclonedx \
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ needs.build.outputs.image-digest }}
        env:
          COSIGN_EXPERIMENTAL: "1"

  # ===========================================================================
  # JOB 6: Image Signing (Cosign — Keyless)
  # Signs the image with the GitHub OIDC identity of this workflow run
  # Signature is published to Sigstore's Rekor transparency log
  # Analogous to: tamper-evident seal on a package — anyone can verify authenticity
  # ===========================================================================
  sign:
    name: Sign Image (Cosign Keyless)
    runs-on: ubuntu-latest
    needs: [ sca-scan, sbom ]
    permissions:
      contents: read
      packages: write   # Required to write signature to GHCR
      id-token: write   # REQUIRED for keyless signing — gets OIDC token from GitHub
    steps:
      - name: Install Cosign
        uses: sigstore/cosign-installer@v3
        with:
          cosign-release: 'v2.2.4'

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Sign the image
        # Keyless signing: Cosign uses the GitHub OIDC token to authenticate to Fulcio CA
        # The signature and certificate are published to Rekor (public transparency log)
        # No private key is stored anywhere — the signing identity IS the GitHub workflow
        run: |
          cosign sign \
            --yes \
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ needs.build.outputs.image-digest }}
        env:
          COSIGN_EXPERIMENTAL: "1"

      - name: Verify the signature (self-verification step)
        # Verify immediately after signing to confirm the signature is valid
        # This also serves as a smoke test for the verification command consumers will use
        run: |
          cosign verify \
            --certificate-identity-regexp="https://github.com/${{ github.repository }}/.github/workflows/devsecops-pipeline.yml" \
            --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ needs.build.outputs.image-digest }} \
            | jq '.[0] | {subject: .critical.identity, issuer: .optional.Issuer}'
        env:
          COSIGN_EXPERIMENTAL: "1"

  # ===========================================================================
  # JOB 7: SLSA Provenance Generation
  # Generates a signed SLSA provenance document: who built it, from what, when
  # This is the chain-of-custody document for the artifact
  # ===========================================================================
  provenance:
    name: Generate SLSA Provenance
    needs: [ sign ]
    # Uses the official SLSA GitHub generator — this is the SLSA Level 2 requirement
    # The generator runs in a separate, isolated workflow to prevent forgery
    uses: slsa-framework/slsa-github-generator/.github/workflows/generator_container_slsa3.yml@v2.0.0
    permissions:
      actions: read
      id-token: write
      packages: write
    with:
      image: ${{ needs.build.outputs.tags }}
      digest: ${{ needs.build.outputs.image-digest }}
      registry-username: ${{ github.actor }}
    secrets:
      registry-password: ${{ secrets.GITHUB_TOKEN }}

  # ===========================================================================
  # JOB 8: OPA Policy Gate
  # Evaluates deployment manifest against Rego policies before any deploy
  # Blocks deployment if image is not from approved registry, not signed,
  # or violates security posture requirements
  # ===========================================================================
  policy-gate:
    name: OPA Policy Gate (Conftest)
    runs-on: ubuntu-latest
    needs: [ sign ]
    steps:
      - name: Checkout (to access policy/ directory)
        uses: actions/checkout@v4

      - name: Install Conftest
```

```bash
# =============================================================================
# image-policy.rego
# OPA Rego policy for container image security requirements
#
# Evaluated by: Conftest (conftest test --policy policy/ manifest.json)
# Namespace: devsecops (matches --namespace flag in pipeline)
#
# Mental model: This is like a Splunk alert rule, but for deployment decisions.
# Instead of "alert if CPU > 80%", it's "deny deployment if image is unsigned".
# =============================================================================

package devsecops

import future.keywords.if
import future.keywords.in

# -----------------------------------------------------------------------------
# DENY RULES — any deny fires = conftest exits non-zero = pipeline fails
# Each rule produces a human-readable message for the pipeline log
# -----------------------------------------------------------------------------

# Rule 1: Image must be signed
# Unsigned images could have been tampered with between build and deploy
deny[msg] if {
    # Access the image_metadata block from the deployment manifest
    metadata := input.image_metadata
    # If signed is false or missing, deny
    not metadata.signed == true
    msg := sprintf(
        "POLICY VIOLATION: Image '%v' is not signed. All images must be signed with Cosign before deployment.",
        [input.spec.template.spec.containers[0].image]
    )
}

# Rule 2: Image must come from an approved registry
# Prevents deployment of images from untrusted or compromised registries
deny[msg] if {
    container := input.spec.template.spec.containers[_]
    # Extract registry from image reference (everything before the first /)
    image := container.image
    # Check that image starts with an allowed registry prefix
    not startswith(image, "ghcr.io/")
    not startswith(image, "123456789012.dkr.ecr.us-east-1.amazonaws.com/")
    msg := sprintf(
        "POLICY VIOLATION: Image '%v' is from an unapproved registry. Only ghcr.io and ECR images are permitted.",
        [image]
    )
}

# Rule 3: No Critical CVEs in production deployments
# Critical CVEs (CVSS 9.0+) are exploitable and must be remediated before deploy
deny[msg] if {
    metadata := input.image_metadata
    metadata.critical_cves > 0
    msg := sprintf(
        "POLICY VIOLATION: Image has %v Critical CVE(s). Critical CVEs must be remediated or have an approved exception before production deployment.",
        [metadata.critical_cves]
    )
}

# Rule 4: Containers must not run as root
# Root containers can escape the container boundary more easily
deny[msg] if {
    container := input.spec.template.spec.containers[_]
    # Check securityContext at container level
    security_context := container.securityContext
    not security_context.runAsNonRoot == true
    msg := sprintf(
        "POLICY VIOLATION: Container '%v' must set securityContext.runAsNonRoot: true.",
        [container.name]
    )
}

# Rule 5: allowPrivilegeEscalation must be false
# Prevents containers from gaining more privileges than their parent process
deny[msg] if {
    container := input.spec.template.spec.containers[_]
    security_context := container.securityContext
    not security_context.allowPrivilegeEscalation == false
    msg := sprintf(
        "POLICY VIOLATION: Container '%v' must set securityContext.allowPrivilegeEscalation: false.",
        [container.name]
    )
}

# Rule 6: Resource limits must be set
# Unlimited resources allow a single container to starve the node — DoS via resource exhaustion
deny[msg] if {
    container := input.spec.template.spec.containers[_]
    not container.resources.limits
    msg := sprintf(
        "POLICY VIOLATION: Container '%v' must have resource limits defined. Unlimited resources risk node exhaustion.",
        [container.name]
    )
}

# Rule 7: Image must be referenced by digest, not just tag
# Tags are mutable — the same tag can point to a different image after a re-push
# Digest references are immutable and ensure you deploy exactly what you scanned
deny[msg] if {
    container := input.spec.template.spec.containers[_]
    image := container.image
    # A digest reference contains "@sha256:"
    not contains(image, "@sha256:")
    msg := sprintf(
        "POLICY VIOLATION: Container '%v' image '%v' must be referenced by digest (sha256), not tag. Tags are mutable.",
        [container.name, image]
    )
}

# -----------------------------------------------------------------------------
# WARN RULES — warnings are reported but do not fail the pipeline
# These represent best practices that are encouraged but not yet mandated
# -----------------------------------------------------------------------------

warn[msg] if {
    container := input.spec.template.spec.containers[_]
    security_context := container.securityContext
    not security_context.readOnlyRootFilesystem == true
    msg := sprintf(
        "POLICY WARNING: Container '%v' should set securityContext.readOnlyRootFilesystem: true for defense in depth.",
        [container.name]
    )
}

warn[msg] if {
    metadata := input.image_metadata
    not metadata.scanned == true
    msg := "POLICY WARNING: Image has no recorded CVE scan. Ensure Trivy scan completed and results are stored."
}
REGO
```

```bash
## DevSecOps Pipeline — Portfolio Project
**Author**: Vishweshwar Chippa

## Threat Modeling Methodology

Using STRIDE (Spoofing, Tampering, Repudiation, Information Disclosure,

Attack vectors mapped to MITRE ATT&CK for Enterprise (Initial Access, Execution,

## Attack Surface Overview

```
Developer Workstation → GitHub (Source) → GitHub Actions (Build) → GHCR (Registry) → EKS (Runtime)
         |                    |                    |                    |                  |
    [Secret leak]     [Branch bypass]      [Build injection]    [Image tamper]    [Runtime escape]
```

---

## Attack Vector 1: Malicious Dependency Injection (SolarWinds Pattern)

**Description**: An attacker compromises an upstream package (e.g., a PyPI package your
`requirements.txt` depends on) and publishes a malicious version. Your pipeline pulls the
new version on next build, injecting malicious code without any developer action.

**Real examples**: SolarWinds Orion (2021), XZ Utils (2024), event-stream (npm, 2018).

**Attack flow**:
1. Attacker identifies a popular dependency in your SBOM (e.g., `flask`, `requests`)
2. Attacker compromises the maintainer account (credential stuffing or phishing)
3. Attacker publishes a malicious patch version
4. Your pipeline pins `flask>=3.0` (not pinned to exact version) and pulls the malicious version
5. Malicious code executes at container startup — data exfiltration, cryptominer, backdoor

**Likelihood**: Medium (requires upstream compromise, but happens multiple times per year)
**Impact**: Critical (malicious code runs in production with app privileges)

**Mitigations implemented in this pipeline**:

| Mitigation | Implementation | Where in Pipeline |
|---|---|---|
| Dependency pinning | `requirements.txt` uses exact versions (`flask==3.0.3`) | Dockerfile / src |
| Hash verification | `pip install --require-hashes` in production builds | Dockerfile |
| SCA scanning | Trivy scans all dependencies against CVE databases | sca-scan job |
| SBOM generation | Syft generates a manifest of all dependencies | sbom job |
| Dependency review | GitHub Dependency Review Action on PRs | PR gate |

**Residual risk**: A zero-day malicious package that is not yet in any CVE database will
pass the CVE scan. Mitigation: behavioral analysis at runtime (Falco) + SBOM drift detection.

---

## Attack Vector 2: Build System Compromise (Codecov Pattern)

**Description**: An attacker gains access to the CI/CD system itself (GitHub Actions runner,
build server, or CI configuration) and injects malicious steps into the build process.
The resulting artifact appears legitimate — correct source, correct binary — but contains
a backdoor inserted during the build phase.

**Real examples**: Codecov bash uploader (2021), CircleCI breach (2023).

**Attack flow**:
1. Attacker compromises a GitHub Actions secret (leaked in logs, misconfigured scope)
2. OR: Attacker submits a PR to a fork that modifies the pipeline YAML
3. OR: Attacker compromises a third-party GitHub Action used in the pipeline
4. Build step exfiltrates secrets, modifies the binary, or adds a malicious layer to the image
5. Malicious image passes CVE scan (no known CVE) and gets deployed

**Likelihood**: Medium (CI secrets are frequently misconfigured)
**Impact**: Critical (attacker controls the build outputs)

**Mitigations implemented in this pipeline**:

| Mitigation | Implementation | Where in Pipeline |
|---|---|---|
| Pinned action versions | All `uses: action@v4` — never `uses: action@main` | All jobs |
| Minimal permissions | `permissions: contents: read` default; scoped up per job | Workflow level |
| No self-hosted runners | GitHub-hosted runners only (SLSA Level 2 requirement) | Workflow config |
| Secret scanning | Gitleaks scans all commits | secret-scan job |
| Keyless image signing | Signing identity is the GitHub OIDC workflow, not a secret | sign job |
| PR protection | Branch protection + required status checks | GitHub settings |
| Isolated build environment | Each job runs in a fresh ephemeral runner | GitHub Actions arch |

**Residual risk**: A compromised GitHub Actions itself (GitHub infrastructure breach).
Mitigation: out of scope for a single team — rely on GitHub's own security posture and
audit logging.

---

## Attack Vector 3: Registry Tampering (Image Substitution Attack)

**Description**: An attacker gains write access to the container registry and replaces
a legitimate image with a malicious one, keeping the same tag. The deployment pipeline
deploys the malicious image, which passes tag-based checks but was never scanned or signed.

**Attack flow**:
1. Attacker compromises a registry credential (leaked PAT, misconfigured IAM policy)
2. Attacker pushes a malicious image with the same tag as the current production image
3. Deployment pipeline reads `image: myapp:latest` and deploys the malicious image
4. The malicious image was never scanned, never signed, never policy-checked
5. OPA policy check is bypassed because it ran against the ORIGINAL image, not this replacement

**Likelihood**: Low-Medium (requires registry credential compromise)
**Impact**: Critical (full application compromise in production)

**Mitigations implemented in this pipeline**:

| Mitigation | Implementation | Where in Pipeline |
|---|---|---|
| Digest-based references | Images referenced as `@sha256:...` not `:latest` | OPA policy, deploy job |
| Image signing | Cosign signature is bound to the specific digest | sign job |
| Signature verification at deploy | Admission webhook (Kyverno/OPA) verifies signature | Cluster-level |
| OPA image allowlist | Only images from approved registries are admitted | policy-gate job |
| GHCR immutable tags | GitHub Packages supports immutable image tags | Registry config |

**Residual risk**: If the cluster-level admission webhook is misconfigured or bypassed,
a tampered image could still deploy. Mitigation: audit all Kubernetes admission webhook
configurations quarterly.
```

## Vulnerability Gate Design

| Severity | CVSS Range | Action | Rationale |
|---|---|---|---|
| Critical | 9.0 – 10.0 | **Hard block** — pipeline fails | Actively exploited; no acceptable risk |
| High (fix available) | 7.0 – 8.9 | **Block** — must upgrade | Patch exists; blocking forces action |
| High (no fix) | 7.0 – 8.9 | **Warn + track exception** | No path to remediate; block creates fatigue |
| Medium | 4.0 – 6.9 | Warn + Jira ticket auto-created | Track; don't stop delivery |
| Low / Info | 0.1 – 3.9 | Log only | Noise reduction |

Exception process: Open GitHub Issue with label `security-exception`. Requires approval
from security team member. Expires in 30 days automatically (enforced by label expiry bot).

## OPA Policy Summary

**image-policy.rego** enforces:
- Image must be signed (Cosign)
- Image must come from an approved registry (GHCR or ECR)
- Zero Critical CVEs
- Container must not run as root
- `allowPrivilegeEscalation: false` required
- Resource limits must be set
- Image must be referenced by digest (not tag)

**terraform-policy.rego** enforces:
- S3 buckets cannot be public
- S3 buckets must have server-side encryption
- IAM policies cannot use wildcard `*` on all resources
- Security groups cannot open sensitive ports to 0.0.0.0/0
- EKS clusters must have private endpoint access

## Tradeoffs and Failure Modes

### Tradeoff 1: Keyless Signing Requires Internet Access to Rekor

Keyless Cosign signing requires outbound access to Sigstore's public Rekor log.
In air-gapped environments (financial services, government), this is a hard blocker.
**Solution for air-gapped**: Deploy an internal Rekor instance (Rekor is open source),
or fall back to KMS-backed key signing with AWS KMS. The pipeline supports both modes
by swapping the Cosign signing command.

### Tradeoff 2: SBOM Accuracy Degrades After Deploy

The SBOM is accurate at build time but containers accumulate OS CVEs as the base image
ages in production. A container deployed 6 months ago has an SBOM that reflects 6-month-old
package versions — new CVEs against those packages are invisible without re-scanning.
**Solution**: Integrate with Amazon Inspector (for ECR images) or Dependency-Track for
continuous SBOM monitoring. Weekly re-scan of all running image digests.

### Tradeoff 3: OPA Policy as Bottleneck

As teams add exceptions and edge cases to Rego policies, the policy repository becomes
a shared bottleneck — every team needs changes reviewed by the security team.
**Solution**: Split policies into platform-level (enforced by the security team, immutable)
and team-level (teams can add policies for their own workloads). Use OPA policy bundles
to distribute platform policies as a versioned artifact.
