# ADR 002: Dual-Export Telemetry — Splunk HEC + OpenTelemetry / Prometheus

**Status**: Accepted  
**Date**: 2024-Q2  
**Decision makers**: SRE Manager, Principal SRE, Security Architect  
**Stakeholders**: CISO office, platform SRE, infrastructure observability team

---

## Context

When migrating to EKS, the team faced a decision about the observability architecture. The existing Splunk investment was deep: the T-Mobile Splunk deployment had 7 years of notification platform data, the MART (Monitoring, Alerting, Reporting, Tuning) framework was built on Splunk saved searches and correlation rules, and the security team's SIEM capabilities were entirely Splunk-based.

However, three new requirements emerged with the EKS migration:

1. **SLO burn rate alerting**: Production-grade SLO monitoring requires fast, query-efficient time-series evaluation (burn rate = rate over a window, evaluated every 30 seconds). Splunk real-time searches at this evaluation frequency are expensive — each real-time search holds a persistent scheduled job that competes with SIEM and compliance workloads on shared Splunk infrastructure.

2. **Kubernetes-native metrics**: Kubernetes exposes its health through the Prometheus exposition format. kube-state-metrics, cAdvisor, and all Kubernetes operators (KEDA, cert-manager, ArgoCD) emit Prometheus metrics natively. Ingesting all of these through Splunk would require either the Splunk Connect for Kubernetes (which adds complexity and latency) or losing the native Kubernetes observability ecosystem.

3. **Vendor risk**: If Splunk licensing changes, is acquired, or has a pricing event that forces migration, the team needs the ability to move the SLO alerting layer without a production blind spot. A dual-export architecture means the observability contract is never held by a single vendor.

---

## Decision

Run **OpenTelemetry Collector as a sidecar on every notification platform pod**, configured with dual-export:

**Export path 1 — Splunk HEC** (Security, compliance, SIEM, MART framework):
- All application logs (structured JSON)
- Distributed traces (for cross-service correlation during incident investigation)
- Business-level metrics (message count by channel, carrier, priority)
- Retained for 90 days (compliance requirement)
- Powers MART dashboards, anomaly detection (MLTK), and security alerting

**Export path 2 — Prometheus remote_write** (SLO alerting, Kubernetes-native observability):
- All application metrics in Prometheus exposition format
- Kubernetes pod/node/HPA metrics (from kube-state-metrics)
- RabbitMQ metrics (via rabbitmq_prometheus plugin)
- Powers Grafana SLO burn rate dashboard
- Alert rules generated by Sloth → PrometheusRule CRDs
- Retained for 15 days (SLO evaluation window is 30d, but raw metrics beyond 15d are not needed — recording rules handle the long-window aggregations)

**OTel Collector config principle**: The collector runs as a sidecar (not a DaemonSet agent) so each pod's telemetry is isolated. A single bad pod cannot saturate the shared agent and cause a blind spot across all pods. The trade-off is higher memory overhead (one collector per pod vs. one per node), which was judged acceptable at the current scale.

## Consequences

**Positive**:
- SLO burn rate alerts evaluate at 30-second intervals against Prometheus without loading Splunk real-time search infrastructure.
- The MART framework and all existing Splunk dashboards continue to operate unchanged — zero migration cost for the security and reporting layer.
- If either export path has an outage, the other continues independently. During a Splunk HEC maintenance window, SLO alerting is unaffected.
- OTel SDK is vendor-neutral. Switching from Prometheus to Grafana Mimir, Thanos, or VictoriaMetrics requires only a config change to the OTel exporter — no application code changes.
- Kubernetes-native operators (KEDA, ArgoCD) are directly observable in the same Prometheus/Grafana stack without any Splunk integration work.

**Negative / Costs accepted**:
- Running one OTel collector sidecar per pod adds approximately 50MB memory and 20m CPU overhead per pod. At 25 pods across all notification services, this is ~1.25GB additional memory across the cluster. Accepted as a reasonable observability tax.
- Two observability systems require two sets of alerting rules. SLO alerts live in PrometheusRule CRDs; security and business alerts live in Splunk alert actions. Engineers need to know which system governs which alert category.
- The OTel Collector configuration (receivers, processors, exporters) must be maintained as code. This is a new operational surface — a misconfigured collector is a silent blind spot. Mitigated by collector health metrics exported to Prometheus and a "meta-alert" that fires if collector metrics stop arriving.

**Risks accepted**:
- Splunk HEC ingest failures are silent unless monitored. The OTel Collector exports a metric `otelcol_exporter_send_failed_spans` which is monitored and alerts if Splunk export is failing.
- Memory limiter configuration in the OTel Collector must be tuned carefully — an incorrect `limit_mib` setting will cause the collector to drop telemetry under high load. This was validated during load testing.

## Alternatives Considered

**Splunk-only**: Rejected for SLO alerting. Splunk scheduled search evaluation frequency (minimum 1 minute) is insufficient for fast burn rate detection (need 30-second evaluation for 1-hour burn rate windows). Real-time searches at 25M messages/day would create unacceptable load on shared Splunk infrastructure.

**Prometheus-only (migrate away from Splunk)**: Rejected. The MART framework, MLTK anomaly models, and SIEM capabilities are deeply invested in Splunk. Migrating the security and compliance layer would require a 12-month engineering effort with no reliability benefit. Splunk is the right tool for long-retention log analytics; Prometheus is the right tool for short-window metric evaluation.

**DaemonSet OTel agent instead of sidecar**: Evaluated. DaemonSet reduces per-pod overhead but introduces shared fate — a single DaemonSet agent failure affects all pods on a node. At 25M messages/day, a 30-second observability gap is unacceptable. Sidecar isolation was chosen for resilience.

---

## Review Date

Review when: OTel Collector memory overhead exceeds 5% of cluster capacity; Splunk licensing changes make dual-export cost-prohibitive; or the T-Mobile observability standard changes to a managed service (e.g., AWS Managed Prometheus + CloudWatch).
