# Chapter 05 — Application Services Layer

**Layer position**: sits above Data Services, below the Edge layer. Consumes data services and middleware; exposes APIs consumed by the Edge/API Gateway.

Cross-references: [Incident Management] [Change Management] [Service Level Management]

---

## What Belongs Here

Business logic services that implement actual business capability — not wrappers or adapters.

| Belongs Here | Does NOT Belong |
|---|---|
| OrderService, PaymentService, AuthService | Data-access wrappers (→ Data Services layer) |
| NotificationService, PricingEngine | Integration adapters, routing logic (→ Middleware/Integration) |
| FraudDetectionService, AccountService | API gateway routing and auth enforcement (→ Edge layer) |
| InventoryManagementService | Raw CRUD repositories or DAOs |

**Distinction that matters**: AccountService (business capability: manage accounts, enforce business rules) is Application Services. CustomerDataService (fetches customer records from a database) is Data Services. AccountService calls CustomerDataService.

**Monolith vs. microservices at this layer**:
- Monolith: entire Application Services layer lives inside one WAR/JAR, organized as packages or Spring modules. Communication is in-process method calls (microseconds). No network, no retries needed internally.
- Microservices: each business capability is a separate deployable service on EKS. Communication is network calls (milliseconds). Every call is a potential failure.

## Microservice Architecture at This Layer

### Service-to-Service Communication

Each application service calls:
- **Data services** (downstream storage access)
- **Middleware** (produce/consume messages via Kafka, RabbitMQ)
- **Other application services** (orchestration or choreography)

Every dependency must be declared in the service catalog. Undeclared dependencies discovered during incidents are a governance failure.

### Service Discovery on EKS

Services locate each other via Kubernetes Service DNS:

```
service-name.namespace.svc.cluster.local
```

External traffic routes through the API gateway. Internal traffic is direct service-to-service or transits the service mesh. A service must never hardcode IP addresses — always use DNS names.

### Distributed Transactions

When a business operation spans multiple services, ACID transactions do not exist across service boundaries. Options:

| Pattern | Mechanism | When to Use | Risk |
|---|---|---|---|
| Saga (choreography) | Each service publishes events; downstream services react | Loosely coupled flows | Hard to visualize; compensations complex |
| Saga (orchestration) | Coordinator service directs each step | Complex multi-step business flows | Coordinator is a single point of failure |
| Eventual consistency | Accept temporary inconsistency; reconcile later | Read-heavy, tolerance for stale data | User sees inconsistent state briefly |
| Two-phase commit (2PC) | Distributed lock across all participants | Almost never appropriate | High coupling, poor availability |

> **VP ownership decision**: no 2PC in production. Saga pattern with compensating transactions is the enterprise standard. Any team proposing 2PC in a design review is proposing an architecture that will create production incidents under load. Reject it at the design gate.

### Service Mesh on EKS (Istio/Linkerd)

The mesh handles mTLS, traffic management, circuit breaking, and retries at the infrastructure layer — not the application layer. This changes the VP governance calculus:

**Before mesh adoption**: application teams own retry logic, circuit breakers, mTLS certificates in code.
**After mesh adoption**: the platform team owns mesh config; application teams remove duplicate retry/circuit-breaker logic from code.

This requires active coordination. Application teams that leave retry logic in code after the mesh provides it create double-retry behavior — a request that fails gets retried by the app AND by the mesh, quadrupling load on a struggling downstream.

> **VP ownership decision**: define the ownership handoff explicitly when adopting a service mesh. Publish a cutover date after which in-code retries and circuit breakers in mesh-managed services are a code review rejection.

## Monolith at This Layer

| Property | Behavior | Operational Implication |
|---|---|---|
| Single deployable unit | All business logic in one process | One deploy changes everything simultaneously |
| In-process communication | Method calls, not network | No retries, no circuit breakers needed internally |
| Shared memory | Modules share heap | Memory leak in one module affects all modules |
| Blast radius | Thread pool exhaustion anywhere affects everything | No isolation; one bad actor can kill the whole app |

**The monolith blast radius problem**: a memory leak, unhandled exception, or thread pool exhaustion in any module affects the entire monolith. There is no circuit breaker protecting one module from another within the same JVM. This is the primary operational argument for extracting high-risk capabilities to isolated microservices first.

**Strangler fig migration**: incrementally extract monolith capabilities into microservices, routing traffic progressively through the API gateway. VP governance requirement: each extraction must preserve all existing SLOs and must not introduce new failure modes. A failed extraction that causes a regression is worse than leaving the monolith intact.

## Application Service SLO Ownership

[Service Level Management]

The application team owns the SLO end-to-end. SRE embeds or consults — but accountability does not transfer to SRE.

| SLO Dimension | Owner | Typical Target | Measurement Point |
|---|---|---|---|
| Availability | Application team | 99.9% | Health check + synthetic monitoring |
| Latency P99 | Application team | <500ms (user-facing), <200ms (service-to-service) | APM trace |
| Error rate | Application team | <0.5% (user-facing), <0.1% (service-to-service) | APM + structured logs |
| Throughput | Application team | Capacity plan dependent | Prometheus / Datadog |

**The dependency SLO constraint**: PaymentService's availability SLO cannot exceed the mathematical product of all its downstream service SLOs. If PaymentService calls three downstream services at 99.9% availability each, the theoretical maximum PaymentService availability is 99.9% × 99.9% × 99.9% = 99.7%. Teams that set SLOs without modeling their dependency chain are setting targets they cannot mathematically achieve.

## The Twelve-Factor App Model at This Layer

| Factor | Requirement | VP Governance Enforcement |
|---|---|---|
| Config from environment | No hardcoded config; use env vars, ConfigMaps | Secrets in Vault or AWS Secrets Manager — never in git |
| Stateless processes | No session state in instance memory | Session state lives in Redis; enables horizontal scaling |
| External backing services | Databases, brokers, caches as attached resources | Automatic reconnect on backing service restart required |
| Disposability | Start <30s; graceful SIGTERM shutdown | Missing SIGTERM handling = connection pool leak on pod restart |
| Dev/prod parity | If production runs Kafka, dev runs Kafka | Stubs hide integration bugs; not acceptable for tier-1 services |

> **VP ownership decision**: enforce dev/prod parity for all tier-1 backing services. A team running RabbitMQ in production and an in-memory stub in dev will discover integration bugs in production during incidents.

## Circuit Breaker and Resilience Patterns

### Circuit Breaker State Machine (Resilience4j)

```
[CLOSED] → failure rate > threshold → [OPEN] → wait duration → [HALF-OPEN]
   ↑                                                                  |
   └──── call succeeds ────────────────────────────────────────────── ┘
                                    |
                        call fails → [OPEN] again
```

- **CLOSED**: calls pass through; failure rate tracked
- **OPEN**: all calls return fallback immediately; no downstream calls made; backend gets recovery time
- **HALF-OPEN**: limited calls test if backend recovered

Circuit breakers prevent cascading failure. Without them, a slow downstream causes thread pool exhaustion upstream, which causes that service to appear degraded, triggering its upstream's thread pool exhaustion — cascade complete.

### Retry with Jitter

Retry failed calls with exponential backoff + random jitter. Without jitter, all retries from all instances of a service hit the downstream simultaneously (thundering herd) — amplifying the problem you are trying to solve. Jitter spreads the retry load over time.

### Bulkhead Pattern

Isolate thread pools by dependency. If calls to ServiceA are slow and exhaust their thread pool, calls to ServiceB continue on their own thread pool. One slow dependency cannot bring down the entire service.

### Timeout Configuration

Every external call must have a timeout. No timeout means a thread can hang indefinitely, exhausting the thread pool. Timeout cascade rule: each service's timeout for downstream calls must be shorter than the upstream's timeout for calls to this service.

```
User → [API Gateway: 30s timeout] → [OrderService: 25s timeout] → [PaymentService: 20s timeout]
```

> **VP ownership decision**: every application service must declare and document: circuit breaker config per downstream dependency, retry policy (including which HTTP methods are safe to retry), timeout values. These are reviewed in architecture review and production readiness gates. Absence of these documents = not eligible for production deployment.

## Database Connection Pool Ownership

The application team owns its connection pool: sizing, monitoring, and exhaustion handling. This is not infrastructure — it is application behavior.

**Pool sizing starting point**: connections = (core_count × 2) + effective_spindle_count. In practice: start at 10 connections per instance, tune based on observed utilization.

**Pool exhaustion failure cascade**:
```
All threads waiting for connection → service latency spikes → upstream callers timeout → circuit breakers open → cascading failure
```

**Monitoring requirement**: connection pool utilization is a first-class metric. Alert at >80% utilization. HikariCP exposes pool metrics natively; instrument them in Prometheus.

**EKS-specific risk**: when a pod is killed and restarted (OOM, rolling deploy, node eviction), the old connection pool is abandoned without clean closure. If the database has a short connection timeout and rolling restarts are aggressive, the DB connection limit can be exhausted. Coordinate pod restart rates and max surge settings with the DBRE team for any service with a large replica count.

## Dependency Declaration Requirement

Every application service must maintain a service dependency manifest:

| Field | Content |
|---|---|
| Service name | This service |
| Upstream dependencies | Services this service calls (sync or async) |
| Downstream consumers | Services that call this service |
| Dependency type | Sync (blocking HTTP/gRPC) or Async (message queue) |
| SLO budget consumed | Estimated availability impact of this dependency |

**Why this matters operationally**:
- **Incident blast radius**: when ServiceA degrades, the manifest immediately identifies which other services depend on it
- **Change impact analysis**: before deploying to ServiceA, the manifest shows who will be affected
- **Capacity planning**: understand load propagation through the dependency graph
- **Deprecation governance**: before removing ServiceA, verify zero active consumers via the manifest and actual traffic telemetry

**Enforcement mechanism**: service catalog tooling or service mesh telemetry (Kiali on Istio shows actual traffic topology). Declared dependencies must match observed traffic. Gaps between declared and actual are a governance finding.

## Production Readiness Gate

No application service is eligible for production deployment without:

| Artifact | Content | Owner |
|---|---|---|
| Dependency manifest | All upstream and downstream dependencies declared | Application team |
| Runbook | Top 3 failure modes with diagnosis and resolution steps | Application team |
| SLO definition | Availability, latency, error rate targets with measurement methodology | Application team + SRE |
| Circuit breaker config | Per-dependency configuration documented | Application team |
| Capacity model | Expected peak load and scaling trigger thresholds | Application team |

This is a gate, not a recommendation. Missing artifacts = no production deployment.

---

> **Incident routing implication**: application service incidents are owned by the application team. Platform SRE (or embedded SRE) provides infrastructure-level diagnosis support. When the incident reveals a dependency failure (downstream service degradation), ownership of the downstream investigation transfers to the downstream team while the application team focuses on mitigation: activate circuit breakers, serve fallback responses, disable dependent features via feature flag. The Incident Commander tracks both workstreams simultaneously. [Incident Management ch02: roles and ownership]

> **VP ownership decision**: require every application service team to maintain and publish: (1) dependency manifest, (2) runbook for top 3 failure modes, (3) SLO definition with measurement methodology. Services without all three artifacts are not eligible for production deployment. This is a hard production readiness gate — not a recommendation, not a best-effort request.
