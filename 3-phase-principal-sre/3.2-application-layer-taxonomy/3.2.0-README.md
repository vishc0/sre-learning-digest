# Application Layer Taxonomy

**Framework purpose**: Classify every system component by architectural layer so that incident routing, SLO ownership, team structure, and investment decisions are deterministic — not tribal knowledge.

---

## What This Is

Layer taxonomy answers the operational question **"who do you call?"** the moment an incident is declared. It is not a diagram exercise. It is the VP's organizational architecture instrument.

Three things follow directly from layer classification:

1. **Team structure follows layer structure.** If your middleware and application layers share an owner, you have an ungoverned boundary. Conway's Law will surface that as a 2AM routing failure.
2. **SLO ownership follows layer ownership.** You cannot assign an SLO to a service without knowing which layer it occupies. SLOs cascade downward — storage SLO must be stricter than every SLO above it.
3. **Investment decisions follow blast radius by layer.** Persistent storage has the highest blast radius; it receives the highest engineering investment. Application services have bounded blast radius; they receive embedded SRE support, not dedicated platform teams.

At T-Mobile scale — hundreds of microservices, Kubernetes service mesh, 50+ Kafka topics, 8 RabbitMQ vhosts, MuleSoft integration flows, Kong and Apigee at the edge — operating without layer taxonomy means incident routing is tribal knowledge. Tribal knowledge does not survive 3AM on a holiday weekend.

## Three Failure Modes of Ungoverned Layer Classification

### Failure Mode 1: No Layer Ownership

**Pattern**: CMDB CIs tagged "Application" with no further classification. Incidents routed to "App Team."

**Consequence**: The App team investigates for 45 minutes, determines it is a database connection issue, escalates to DBRE. DBRE spends 20 minutes determining connection pool exhaustion is in the application tier, not the database. You have lost 65–90 minutes to routing ambiguity. Across a year of incidents, this inflates MTTR by 40–60% on cross-layer failures.

**Fix**: Every CI in CMDB carries a layer tag AND a named owner. Not "App Team" — "Customer Data Service, Retail Domain team, layer: Data Services, on-call: retail-sre-primary."

---

### Failure Mode 2: Layers Classified by Technology Instead of Function

**Pattern**: The "Middleware Team" owns RabbitMQ (message broker), MuleSoft flows (integration adapters), Kafka (event streaming), and REST-to-SOAP adapters — all because they are "integration technology."

**Consequence**: These components carry fundamentally different SLO requirements. RabbitMQ broker availability is a platform SLO (affects every async consumer). A REST-to-SOAP adapter is an application-tier SLO (affects one integration flow). Merging them under one team creates SLO dilution — the team cannot maintain broker-grade on-call discipline while also triaging application-tier adapter issues. Team burnout follows within 6–12 months.

**Fix**: Classify by function, not technology. Broker infrastructure (platform SLO) → Middleware Platform SRE. Integration adapters (application SLO) → owning domain team with middleware SRE consultation.

---

### Failure Mode 3: SLOs Defined Per Service Without Layer Cascade

**Pattern**: Product team defines 99.9% availability SLO for the Order Service. The underlying PostgreSQL cluster carries a 99.5% availability SLO. The RabbitMQ broker carries a 99.7% SLO.

**Consequence**: Mathematical impossibility. If storage is available 99.5% of the time, Order Service cannot be available 99.9% of the time unless it has complete fallback paths for storage outages — which it does not. The SLO is a fiction that will be violated during the next storage maintenance window.

**Fix**: SLOs cascade downward. Persistent Storage SLO must be stricter than every Data Services SLO above it. Middleware SLO must be stricter than every Application Services SLO that depends on it. Define the cascade before defining individual service SLOs.

*Cross-reference: [Service Level Management] SLO cascade mechanics. [Change Management] risk classification by layer.*

## Key Definitions

| Layer | Classification | What Belongs Here | Typical Owner | Key Metric | Common Failure Mode |
|-------|---------------|-------------------|---------------|-----------|---------------------|
| Persistent Storage | Platform | RDBMS (PostgreSQL, Oracle), NoSQL (Cassandra, DynamoDB), object storage (S3), block storage (EBS, SAN) | DBRE / Platform SRE | IOPS, query latency P99, replication lag | Schema migration blast radius — unplanned DDL locks all consumers |
| Storage Abstraction | Platform | ORM layers (Hibernate, SQLAlchemy), connection pools (HikariCP, PgBouncer), caching tier (Redis, Memcached) | Platform SRE | Cache hit rate, connection pool utilization, eviction rate | Pool exhaustion under traffic surge — all app instances queue on DB |
| Data Services | Domain | Microservices that expose data as API: Customer Service, Inventory Service, Account Service | Domain team | Query response P99, data freshness SLA, contract version | Schema contract break on upstream change — downstream consumers silent-fail |
| Middleware / Integration | Platform | Message brokers (RabbitMQ, Kafka, TIBCO EMS), ESBs (MuleSoft), workflow engines (Temporal, Conductor), event buses | Middleware SRE | Consumer lag, queue depth, DLQ count, broker uptime | Broker failure stops all async consumers simultaneously — blast radius is total |
| Application Services | Product | Business logic microservices — order processing, payment, authentication, notification, pricing, fulfillment | Product team + embedded SRE | Error rate, latency P99, throughput RPS | Dependency failure cascade — one slow upstream degrades entire call chain |
| API Gateway / Edge | Platform | Kong, Apigee, AWS API GW, NGINX, WAF, CDN (Akamai, CloudFront), load balancers (ALB, NLB) | Platform / Network SRE | Request rate, error rate by route, latency P99, TLS cert expiry | Certificate expiry or routing misconfiguration causes total traffic failure |
| Presentation / UI | Product | Web frontends (React, Angular), mobile apps (iOS/Android), BFF services, static asset CDN | Product team | Core Web Vitals, Time to Interactive, JS error rate | Broken UI = customer-perceived outage even when all backend services are healthy |

## Relationship to Other Frameworks

**[Incident Management]**: Layer taxonomy drives blast radius routing. When a SEV1 is declared, the incident commander uses layer classification to determine the lead team within the first 5 minutes. Without taxonomy, this determination takes 20–40 minutes of bridge confusion. See ch05 of Incident Management framework for blast radius routing decision trees.

**[Change Management]**: Change risk is classified by layer. Persistent Storage changes (schema migrations, replication topology changes) carry the highest risk rating and require DBRE CAB approval + rollback validation. Edge layer changes (certificate rotation, WAF rule updates) carry medium-wide risk — low complexity but broad traffic impact. See `reference/change-risk-by-layer-reference.md`.

**[Service Level Management]**: SLO cascade is a hard mathematical constraint derived from layer structure. Storage SLO must be stricter than Data Services SLO. Middleware SLO must be stricter than Application Services SLOs that depend on async messaging. The layer taxonomy is the input to SLO design — you cannot define SLOs coherently without it.

**[Problem Management]**: Recurring incidents that resist root cause identification are frequently caused by layer boundary misclassification. A component classified in the wrong layer is monitored by the wrong team with the wrong metrics. Problem Management reviews should include a layer classification audit for any incident that recurred more than twice in 90 days.
