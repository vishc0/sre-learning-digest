# Chapter 06 — API Gateway and Edge Layer

**Layer position**: outermost controllable layer. Sits between external consumers (users, partner systems, mobile clients) and internal application services. All traffic enters through this layer.

Cross-references: [Incident Management] [Change Management] [Service Level Management]

---

## What Belongs Here

| Belongs Here | Does NOT Belong |
|---|---|
| API Gateways: Kong, Apigee, AWS API Gateway, MuleSoft Anypoint API Manager (gateway function only), Azure APIM | Internal service-to-service load balancing (→ service mesh, Application Services infrastructure) |
| Reverse proxies: NGINX, HAProxy, Envoy as edge proxy | MuleSoft integration flows with transformation logic (→ Middleware/Integration) |
| CDNs: Cloudflare, Akamai, AWS CloudFront | Message broker routing (→ Middleware/Integration) |
| Load balancers: AWS ALB/NLB, F5 BIG-IP | Application business logic of any kind (→ Application Services) |
| WAFs: AWS WAF, Cloudflare WAF, F5 ASM | |
| Rate limiters: Redis-backed, Kong rate-limit plugin, AWS API Gateway throttling | |

**Envoy distinction**: Envoy as an edge proxy (standalone, at the perimeter) belongs here. Envoy as a sidecar in a service mesh belongs to Application Services infrastructure. Same technology, different deployment context, different ownership.

## The Edge Layer's Role

### Traffic Management
Route incoming requests to the correct backend service; load balance across instances; A/B test traffic splitting; canary routing (5% of traffic to v2, 95% to v1).

### Authentication and Authorization
API key validation, OAuth2 token introspection, JWT validation, mTLS termination — all enforced **before** requests reach backend services. The edge is the enforcement point; backends trust the identity claims the edge has already validated.

### Rate Limiting
Protect backend services from overload; enforce per-consumer quotas; prevent runaway clients. A client making 10,000 RPS against a service with 2,000 RPS capacity will cause an outage without edge rate limiting. With it, excess traffic receives 429 before consuming any backend resources.

### Protocol Termination
TLS termination at edge (backends run plain HTTP internally); WebSocket upgrade handling; HTTP/2 or HTTP/3 termination with HTTP/1.1 upstream. Centralizing TLS termination at the edge simplifies certificate management — one place to rotate, monitor, and alert on certificate expiry.

### Observability Entry Point
Every request enters the system at the edge. The edge is the only vantage point that sees the **complete** request picture — including unauthenticated traffic, bot traffic, failed auth attempts, and pre-rate-limit request volume. Any observability that starts downstream of the edge misses this traffic class entirely.

## API Gateway as Reliability Control Plane

The gateway is not just routing infrastructure — it is an active reliability mechanism.

| Mechanism | How It Works | What It Prevents |
|---|---|---|
| Rate limiting | Reject excess requests with 429 before backend | Backend overload from runaway clients or traffic spikes |
| Circuit breaking (Kong, Apigee) | When backend returns >threshold errors, gateway serves fallback | Retry storms amplifying a degraded backend |
| Retry policies | Safe retries on GET only; never on non-idempotent methods without idempotency keys | Silent data corruption from double-executing state-changing calls |
| Response caching | Cache GET responses at gateway; reduce backend load | Read-heavy backend load; repeated identical queries |
| Request timeout enforcement | Terminate long-running requests at gateway before backend thread hangs | Thread pool exhaustion from slow upstream clients |

> **VP ownership decision**: rate limits must be defined for every API route before production deployment. "We'll add rate limits later" is not acceptable. A rate-limit-free API is an on-call incident waiting to happen — any traffic event, intentional or accidental, will exhaust the backend. Rate limits are a production readiness requirement, not an enhancement.

## Edge Layer as Security Perimeter

### WAF and DDoS

| Control | Purpose | Ownership |
|---|---|---|
| WAF rules (OWASP Top 10) | Block SQL injection, XSS, SSRF, path traversal | Security team owns rules; platform SRE owns availability |
| Custom WAF rules | Business-specific attack pattern blocking | Security team defines; platform team implements |
| DDoS protection | Volumetric attack absorption at CDN/WAF before reaching origin | CDN vendor + Security team |
| Bot management | Distinguish legitimate users from scrapers and credential stuffing | Security team policy; gateway platform implements |
| API key management | Key rotation, scoping, revocation | Security team policy; product team per-key ownership |
| mTLS (B2B APIs) | Client certificates for machine-to-machine APIs | Platform SRE (certificate infrastructure); consuming team |

[Change Management]: WAF rule changes are HIGH risk. A misconfigured WAF rule blocks legitimate traffic (false positive) across ALL customers simultaneously — blast radius is 100% of affected API consumers. Require canary rollout for WAF rule changes: deploy to 5% of traffic, monitor 429/403 rates for 30 minutes, then promote. Emergency WAF changes during active attacks follow a separate expedited process with Security team approval.

### Three Ownership Domains at the Edge

The edge layer has three distinct ownership scopes that must not be conflated:

| Domain | Owner | Scope |
|---|---|---|
| Gateway availability, routing, TLS | Platform SRE | Uptime, certificate rotation, load balancing config |
| WAF rules, bot management, DDoS policy | Security team | Rule definitions, threat response, compliance |
| API route definitions, rate limit policies, versioning | Product teams | What routes exist, who can access what, deprecation timelines |

> **VP ownership decision**: enforce this three-way boundary explicitly. Security teams that also make availability decisions will optimize for security at the expense of uptime (blocking legitimate traffic). Platform teams that define API routes will optimize for infrastructure, not product needs. Product teams that set their own WAF rules will create security gaps. Each team decides within its domain; cross-domain decisions require all three owners.

## SLO at the Edge Layer

[Service Level Management]

| SLO Dimension | Edge Target | Backend Target | Notes |
|---|---|---|---|
| Availability | 99.99% | 99.9% | Edge must be 1 tier stricter than any backend it fronts |
| Latency added | <10ms P99 | N/A | Gateway processing overhead only (TLS, auth, rate limit check) |
| Error rate (5xx) | <0.01% (gateway errors) | Varies per service | 4xx from auth failures and rate limiting are NOT gateway errors |
| Certificate validity | 100% (zero expiry events) | N/A | Automated certificate rotation; expiry = all users blocked |

**Why edge availability must exceed backend availability**: you cannot be more available than your front door. If the edge is 99.9% and a backend is 99.9%, the end-to-end availability is at most 99.8%. The edge is the one component whose failure affects every user simultaneously. The investment in edge availability (redundant deployments, multi-region, automated failover) is justified by this blast radius.

**Tracking edge SLO separately from backend SLO**: "The API gateway was up but OrderService was down" means the edge SLO is met but the end-to-end user experience SLO is not. Report both metrics independently. Conflating them hides which layer failed.

## API Versioning Governance

### The VP Decision Framework

| Decision | Owner | Timing |
|---|---|---|
| Breaking change requires new version | API product owner | At design review |
| Deprecation timeline (minimum 6 months) | VP sets policy | At v2 GA |
| Deprecation notification mechanism | API product owner + developer portal team | At v2 GA announcement |
| Sunset: remove v1 | API product owner, after traffic confirms zero consumers | Only when telemetry shows zero traffic, not based on notification alone |

**v1 to v2 coexistence at the gateway**: the gateway routes based on path (`/api/v1/` vs `/api/v2/`) or header (`Accept: application/vnd.company.v2+json`). Both versions are live simultaneously during the migration window.

**Deprecation signals to consumers**:
1. Developer portal announcement
2. Email notification to registered API consumers
3. `Deprecation` header in v1 responses (RFC 8594): `Deprecation: Sat, 01 Jan 2027 00:00:00 GMT`
4. `Sunset` header: `Sunset: Sat, 01 Jan 2027 00:00:00 GMT`

**The traffic telemetry requirement**: never remove a version based on notification alone. Monitor actual v1 traffic at the gateway. When observed traffic reaches zero for 30 consecutive days, v1 removal is safe. Consumers who received notifications but never migrated are real — telemetry is the source of truth, not email delivery receipts.

This is a Change Advisory Board item for any public-facing API version removal.

## Edge Layer Incident Patterns

[Incident Management]

| Incident Type | Cause | Detection | Blast Radius | Resolution Path |
|---|---|---|---|---|
| Certificate expiry | Expired TLS cert | Browser errors, monitoring alert on cert expiry date | ALL users (100% of traffic) | Renew cert → deploy → verify via synthetic |
| Rate limit misconfiguration | Limit set too low | Spike in 429 errors, user complaints | Specific API consumers | Adjust limit → deploy config → monitor |
| Routing rule error | Misconfigured path → wrong service | 404 or unexpected responses on specific routes | Specific API routes | Roll back gateway config → verify |
| WAF false positive | New rule blocks legitimate traffic | Spike in 403s, user reports | All traffic matching rule pattern | Disable offending rule → analyze → refine and redeploy |
| Upstream backend timeout | Backend slow, gateway timeout shorter than backend response time | 504 Gateway Timeout spike | Specific API routes | Increase gateway timeout OR fix backend; update timeout cascade |
| CDN origin pull failure | CDN cannot reach origin server | Cache MISS + origin errors, cache hit rate drops | All uncached content | Restore origin connectivity → purge CDN cache → monitor hit rate |

**Edge incidents are almost always SEV1**: unlike backend service incidents that affect a subset of traffic, edge layer incidents affect ALL users simultaneously regardless of the backend's health. The blast radius is always maximum. The IC must be engaged within 5 minutes of edge incident detection — do not wait for impact confirmation.

## Multi-Region Edge: GeoDNS and Anycast

### GeoDNS

Routes users to the nearest regional endpoint based on DNS resolver location. At T-Mobile national scale: US-East, US-West, and optionally Central. Requires per-region health checks — GeoDNS fails over to secondary region when primary health check fails.

Health check failure → GeoDNS update → TTL propagation (60–300 seconds) → traffic routes to secondary. During TTL propagation, some users still route to the failed region. This is the recovery time gap that multi-region capacity planning must account for.

### Anycast

BGP-based routing where the same IP address is announced from multiple data centers. Traffic flows to the topologically nearest BGP announcement point. Used by Cloudflare, Akamai, and major CDN providers. Failover is BGP withdrawal, not DNS TTL — faster than GeoDNS.

### Regional Capacity Planning

When US-East fails, US-West must absorb US-East's traffic in addition to its own. This requires N+1 capacity headroom at the regional level — each region is sized to handle 100% of traffic if the other region is completely unavailable.

| Planning Requirement | What to Define |
|---|---|
| Failover runbook per component | Which components fail over automatically; which require manual intervention |
| Degraded SLO during failover | What is the committed SLO when running on one region (latency increases for cross-country users) |
| Capacity validation | Quarterly load test each region at 200% of its normal peak to validate N+1 capacity |
| DNS TTL settings | Low enough for fast failover; high enough to avoid excessive resolver load |

> **VP ownership decision**: every edge component must have a documented regional failover runbook. "What happens to US-West when US-East fails?" must be answerable from documentation, not reconstructed during an incident. Failover runbooks are reviewed in the same quarterly readiness review as backup/recovery procedures.

---

> **Incident routing implication**: edge layer incidents route to the Platform/Network SRE team. The Platform SRE team leads the incident. Backend application teams are supporting SMEs only when the edge is confirmed healthy and the issue is a backend degradation surfacing at the edge layer. The first five minutes of every edge incident are spent on a single diagnostic question: is this an edge failure, or a healthy edge accurately reflecting a backend failure? The answer determines who leads. Platform SRE diagnoses edge health; backend teams provide service health. These run in parallel; do not sequence them.

> **VP ownership decision**: three teams own the edge layer with distinct scopes — Platform SRE (gateway availability, routing, TLS), Security (WAF rules, bot management, DDoS policy), and Product (API route definitions, rate limit policies, versioning decisions). No single team owns all three. The VP enforces the boundary to prevent scope creep: security teams do not make gateway availability decisions, platform teams do not define product API contracts, product teams do not configure security controls. Cross-domain decisions require all three owners in the room with the VP arbitrating when they conflict.
