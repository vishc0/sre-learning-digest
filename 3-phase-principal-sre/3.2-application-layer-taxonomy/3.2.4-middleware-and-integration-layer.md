## What Belongs Here (and What Does Not)

### In Scope

| Category | Examples |
|----------|---------|
| Message brokers | RabbitMQ (AMQP), Kafka (log-based streaming), TIBCO EMS (enterprise JMS), ActiveMQ, IBM MQ |
| Enterprise Service Buses | MuleSoft Anypoint Platform, IBM App Connect (formerly IIB), TIBCO BusinessWorks, WSO2 |
| Event streaming platforms | Kafka Streams, AWS Kinesis, Apache Flink (when used for integration, not analytics) |
| Integration adapters | REST/SOAP adapters that transform protocol or data format between systems |
| Workflow engines | Camunda, Temporal, Apache Airflow (when used for integration orchestration) |
| Message routers | Content-based routers, topic-based pub/sub arbiters |

### Out of Scope (Common Misclassifications)

| Misclassified Item | Correct Layer | Reason |
|--------------------|--------------|--------|
| A REST microservice calling another REST service | Application Services | Synchronous point-to-point with no broker involvement |
| A data pipeline writing to a data warehouse | Data Platform / Data Services | Storage destination is the defining characteristic |
| API Gateway | Edge Layer | Traffic ingress/auth/rate-limiting function, not integration |

## The Integration Layer's Core Role

The integration layer does one thing that no other layer does: it **decouples producers from consumers**. Every other benefit flows from that.

| Function | Mechanism | Operational Implication |
|----------|----------|------------------------|
| Async communication | Producer fires and forgets; consumer processes independently | Producer SLO is independent of consumer SLO |
| Protocol translation | SOAP → REST, JMS → AMQP, XML → JSON | Legacy systems can speak to modern services without code change |
| Content-based routing | Message headers or body determine destination queue/topic | One producer, many consumers, selective subscription |
| Backpressure absorption | Broker holds messages when consumer is slow | Consumer lag is visible; it is not invisible like a thread queue |
| Delivery semantics | At-least-once, exactly-once, at-most-once enforced at broker | The application layer does not need to re-implement delivery guarantees |

Without the integration layer, every service must know the address, protocol, and availability of every other service it talks to. That is the synchronous mesh anti-pattern — brittle at scale, catastrophic under failure.

## Middleware Classification: Synchronous vs. Asynchronous

| Type | Examples | Delivery Guarantee | State | Failure Mode | T-Mobile Use Case |
|------|---------|-------------------|-------|-------------|------------------|
| Async Message Queue | RabbitMQ, TIBCO EMS, IBM MQ | At-least-once | Stateful (persisted queue) | Backpressure, DLQ overflow | Order processing, notification dispatch |
| Async Event Stream | Kafka, Kinesis | At-least-once, exactly-once | Stateful (log-based) | Consumer lag, partition leadership election | Event sourcing, audit trail, real-time analytics feed |
| Sync Integration Adapter | MuleSoft HTTP connector, REST adapter | Request-response | Stateless | Timeout, upstream unavailability | CRM to billing sync, legacy system façade |
| ESB Orchestration | MuleSoft flow, TIBCO BusinessWorks | At-least-once (with retry) | Stateful (flow state) | Flow failure, retry exhaustion | Multi-system business process coordination |
| Workflow Engine | Camunda, Temporal | Exactly-once (idempotent) | Stateful (workflow state) | Stuck workflow, saga rollback failure | Long-running business process (order fulfillment, loan processing) |

## Why Middleware Fails Differently

Every other layer fails fast and visibly. Middleware fails **slowly and silently** — and then catastrophically.

**Backpressure accumulation**: queues accumulate depth when consumers are slow. The broker absorbs load right up until it cannot — then it either applies flow control (RabbitMQ memory alarm) or drops messages (at-most-once config). The danger window between "growing queue" and "broker overwhelmed" is the SRE's operational responsibility.

**Poison messages**: a malformed or unprocessable message causes a consumer to fail repeatedly. Without DLQ configuration, the consumer retries indefinitely, blocking the queue. With DLQ, the poison message is quarantined — but then the DLQ itself must be monitored. An unmonitored DLQ fills silently while the team believes the queue is healthy.

**Kafka consumer group rebalance**: when a consumer instance crashes, Kafka triggers a rebalance across the entire consumer group. All consumers in the group pause consumption during partition redistribution — seconds to minutes depending on group size and coordinator load. Frequent consumer crashes = frequent rebalances = systemic throughput gaps that do not appear as errors on any single service dashboard.

**RabbitMQ network partition (split-brain)**: if a cluster node loses connectivity to its peers, it may continue accepting writes. Messages written to the partitioned node may be discarded when the partition heals and the cluster resolves state via the configured partition handling strategy (`pause-minority`, `autoheal`, or `ignore`). Production must use `pause-minority` — never `ignore`.

**MuleSoft memory pressure**: each active flow execution consumes JVM heap. High concurrency without explicit threading pool limits causes OOM. On CloudHub this is managed via worker sizing; on-prem Mule workers require explicit heap and thread pool tuning. A slow upstream system holding flows open is the most common cause — the Mule worker thread pool exhausts while waiting for a response it will never get.

## SRE Ownership Model for Middleware

### Three Structural Options

| Model | Structure | Best For | Risk |
|-------|----------|---------|------|
| Option A: Dedicated Middleware SRE team | A single team owns all middleware platforms | 3+ distinct middleware platforms; tier-1 SLOs on middleware | Potential bottleneck for product team escalations |
| Option B: Embedded SRE per platform | One SRE embedded with Kafka team, one with MuleSoft team | Isolated platforms, distinct codebases, minimal cross-platform incidents | Expertise siloed; no shared on-call; hard to staff |
| Option C: Platform SRE sub-specialty | Platform SRE team with middleware as a declared sub-specialty | Mid-size orgs; fewer than 3 platforms | Middleware expertise diluted if platform scope is too broad |

### The Anti-Pattern

Middleware owned by the application team that uses it most. When RabbitMQ is "owned" by the Notifications team, every other consumer team has no escalation path. The Notifications team optimizes the broker for their own traffic patterns. When a Finance team consumer starts lagging, there is no clear DRI.

> **VP ownership decision**: middleware is platform infrastructure, not application infrastructure. It belongs on the platform team's charter with explicit on-call ownership, documented escalation paths, and SLO accountability. Product teams own their consumers, their producers, their message schemas, and their DLQ remediation. The Middleware SRE team owns broker availability, cluster health, topology change approval, and platform-level SLO reporting. This boundary must be written into team charters, not left to incident-by-incident negotiation.

## RabbitMQ Deep Dive: Architecture and Operational Patterns

### vHost Structure

vHosts are logical namespaces within a RabbitMQ cluster. They share cluster resources but are completely isolated in terms of permissions, policies, and queue namespacing.

**Rule**: never share vHosts across environments. Production queues live in the `production` vHost. Staging has its own. Each vHost has its own Splunk monitoring namespace, its own alert thresholds, and its own access control list. A misconfigured staging consumer connecting to the production vHost and consuming messages is a real failure mode — vHost isolation prevents it.

### Exchange and Queue Topology

Exchanges receive messages from producers. Bindings route messages from exchanges to queues. Consumers read from queues.

| Exchange Type | Routing Mechanism | Use Case | Operational Note |
|--------------|------------------|---------|-----------------|
| Direct | Exact routing key match | Point-to-point dispatch | Simple; no wildcard; clear trace |
| Fanout | Broadcast to all bound queues | Pub/sub where all consumers must receive every message | Adding a consumer = adding a binding; no producer change required |
| Topic | Wildcard routing key (`*` one word, `#` zero or more words) | Selective subscription (`order.#` matches all order events) | Most flexible; most complex to debug misroutes |
| Headers | Message header attribute matching | Rarely used; avoid unless routing key is insufficient | High operational complexity; avoid in new designs |

### Queue Durability Models

| Model | When to Use | Tradeoff |
|-------|------------|---------|
| Classic mirrored queues (pre-3.8) | Legacy clusters only | Synchronous mirroring to all nodes; high write amplification; not recommended |
| Quorum queues (3.8+) | All tier-1 queues | Raft-based consensus; leader election without data loss; required for critical queues |
| Classic non-mirrored | Dev/test only | No HA; data loss on node failure; never production |

> **VP ownership decision**: all tier-1 queues at T-Mobile must use quorum queues. Classic mirrored queues are end-of-life in RabbitMQ 3.13. Any queue backing a tier-1 SLO that is not a quorum queue is a compliance gap against this policy. The Middleware SRE team tracks this via quarterly queue audit; findings escalate to VP if remediation is not scheduled within one sprint.

### Cross-Broker Patterns

**Shovel**: moves messages from one queue to another, across brokers or vHosts. Use cases: cross-datacenter message replication during failover; queue draining during broker migration without service disruption. Shovels are stateless and restart-safe — if the source or destination broker is unavailable, the shovel retries.

**Federation**: logical extension of exchanges or queues across brokers without requiring full cluster membership. Use for multi-region RabbitMQ where publishers are in one region and consumers in another. Federation pulls messages on-demand — consumers only receive messages when a local consumer is available, reducing cross-region bandwidth compared to full mirroring.

### Management API Monitoring

RabbitMQ exposes a REST management API at `:15672/api/`. This is the authoritative source for broker metrics.

Key endpoints for SRE and VP dashboards:

| Endpoint | Data | Use |
|---------|------|-----|
| `/api/queues` | All queue metrics (depth, consumers, rates) | Primary operational signal |
| `/api/nodes` | Cluster node health, memory, disk | Broker infrastructure health |
| `/api/connections` | All active AMQP connections by client | Connection leak detection |
| `/api/exchanges` | Exchange definitions and message rates | Topology auditing |
| `/api/overview` | Cluster-wide message rates and node count | VP dashboard summary |

Scrape via `prometheus_rabbitmq_exporter` or a custom Splunk scripted input polling the REST API every 60 seconds.

## Kafka Deep Dive: Architecture and Operational Patterns

### Partition Leadership and ISR

Every Kafka partition has exactly one leader broker that handles all reads and writes. Follower replicas sync from the leader. The ISR (In-Sync Replicas) set contains replicas that are within `replica.lag.time.max.ms` of the leader.

Failure chain:
1. Leader broker fails
2. Kafka controller elects a new leader from ISR
3. If ISR count < `min.insync.replicas`, partition becomes unavailable to producers with `acks=all`
4. Producers receive `NotEnoughReplicas` exception
5. Tier-1 producers with `acks=all` and no local buffer begin failing

**Monitoring target**: ISR count must equal replication factor in steady state. Any deviation is an immediate alert.

### Consumer Group Coordination

Each Kafka topic partition is consumed by exactly one consumer in a given consumer group. Consumer groups share the partition assignment work.

**Rebalance trigger events**: consumer joins or leaves group; consumer misses heartbeat (crash or GC pause); partition count changes; group coordinator reassignment.

During rebalance, all consumers in the group pause. Duration: seconds to minutes depending on group size, consumer count, and coordinator load. At T-Mobile scale with large consumer groups, rebalance duration directly determines the throughput gap during any consumer incident.

**Reducing rebalance impact**:

| Technique | Mechanism | When to Use |
|-----------|----------|------------|
| Cooperative rebalancing | Incremental reassignment; only affected partitions paused | All new deployments; requires Kafka 2.4+ |
| Static membership | Fixed `group.instance.id`; rejoining consumer reclaims its prior partitions | Long-lived consumer processes; reduces rebalance on restart |
| Smaller consumer groups | Fewer members = shorter rebalance | Partition capacity allows; split large groups by function |

### Producer Acknowledgment Modes

| Mode | Guarantee | Data Loss Risk | Use Case |
|------|----------|--------------|---------|
| `acks=0` | None (fire and forget) | High | Metrics, telemetry where loss is acceptable |
| `acks=1` | Leader ACK | Loss on leader crash before replica sync | Moderate-priority events |
| `acks=all` | All ISR ACKs | None (within ISR constraints) | Tier-1 topics; financial events; audit trails |

**Default for all tier-1 Kafka topics at T-Mobile**: `acks=all` with `min.insync.replicas=2` and `replication.factor=3`. Any deviation requires documented exception approved by Middleware SRE lead.

### Log Retention and Data Lifecycle

| Policy Type | Configuration | Use Case | Governance Note |
|------------|--------------|---------|----------------|
| Time-based retention | `retention.ms` | Standard event topics | Must be set per topic based on consumer SLA; never defaulted cluster-wide |
| Size-based retention | `retention.bytes` | High-volume topics where time is secondary | Combine with time-based for predictable behavior |
| Log compaction | `cleanup.policy=compact` | CDC / event sourcing where only latest value per key matters | Compaction runs asynchronously; consumers may see intermediate states |

> **VP ownership decision**: retention policies must be set per topic, not defaulted to 7 days across the board. The Middleware SRE team owns a topic registry that documents each topic's retention policy, owner team, consumer SLA, and `acks` configuration. Topics without entries in the registry within 30 days of creation are flagged for remediation. The registry is the source of truth for compliance audits.

### Consumer Lag as the Primary Operational Signal

Consumer lag = log end offset minus committed consumer offset. It is the most actionable single metric in Kafka operations.

| Lag State | Consumer Behavior | Action |
|-----------|-----------------|--------|
| 0–100 | Consumer keeping up | Healthy; no action |
| 100–1000 | Consumer slightly behind | Monitor trend; check for consumer errors |
| >1000 sustained | Consumer cannot keep pace | Scale consumer group (up to partition count); investigate consumer error rate |
| Lag growing + consumer error rate rising | Consumer crashing and restarting | Fix consumer error first; scaling a crashing consumer makes rebalances worse |
| Lag growing + consumer error rate normal | Consumer is slow but healthy | Scale consumer group; verify no GC pressure |

Alert threshold: lag >1000 messages sustained for >5 minutes, per consumer group.

## Middleware as SPOF: Blast Radius Analysis

The integration layer's defining characteristic — that dozens of services depend on it — is also its defining risk. A broker outage is not a single-service incident.

### Failure Scenarios at T-Mobile Scale

**RabbitMQ cluster outage**:
- All async consumers across all vHosts stop receiving messages
- Queues that were persisted survive on disk (if cluster recovers) or are lost (if disk failure)
- Producers receive connection refused; producer retry behavior determines secondary blast radius
- 20–50 services may be directly affected simultaneously

**Kafka cluster outage**:
- All consumers stop; all producers with `acks=all` stop; topics are unavailable
- Kafka's distributed design (ISR, partition leadership) makes partial failure more common than total outage
- Total cluster outage is rare but catastrophic; more common failure is under-replicated partitions causing producer errors on specific topics

**MuleSoft platform outage**:
- All synchronous integration flows fail — CRM sync, billing integration, legacy system access
- Blast radius proportional to flow count and downstream system dependency
- Flows that are dependencies for user-facing APIs produce user-visible errors immediately

> **Incident routing implication**: any incident involving consumer lag, queue depth spike, message processing failure, or DLQ overflow routes to the Middleware SRE team first. The consuming application team is a supporting SME, not the lead. The diagnostic split: if broker metrics (queue depth, ISR count, node health) are abnormal, Middleware SRE leads. If broker metrics are normal but one consumer group is lagging or erroring, the consuming application team leads. This distinction — broker health vs. consumer health — is the first triage step in every middleware incident. [Incident Management] ch05: a middleware outage at this scale is a tier-1 cascading failure. The IC should pre-brief all consuming service teams before they generate tickets independently.

### Resilience Design Principles

| Principle | Implementation | Prevents |
|-----------|---------------|---------|
| Producer local buffer | Producers hold messages locally if broker is unavailable; flush on reconnect | Cascading failure from broker unavailability propagating to producers |
| Exponential backoff with jitter | Retry with increasing delay; jitter prevents thundering herd on broker recovery | Recovery stampede overwhelming a just-recovered broker |
| DLQ monitoring SLO | Any DLQ message older than 24 hours triggers alert | Silent DLQ accumulation masking processing failures |
| Consumer graceful degradation | Service degrades (returns cached/default data) when no messages received, rather than erroring | Cascading user-facing failure during broker maintenance |
| Circuit breaker on sync adapters | MuleSoft flows open circuit after N consecutive failures; stop calling failed upstream | Cascading thread exhaustion in MuleSoft when upstream is down |

Never make synchronous, user-facing API flows depend on async broker availability. If a REST endpoint requires a message to be published to RabbitMQ before returning 200, a broker outage becomes a user-facing outage. The correct design: publish to broker async after returning 202 Accepted.

## Change Risk at the Middleware Layer

Middleware topology changes are among the highest-risk changes in the stack. Many are irreversible.

| Change Type | Risk Level | Approval Required | Rollback Complexity | Blast Radius | Notes |
|------------|-----------|------------------|--------------------|--------------|----|
| Queue or exchange deletion | CRITICAL | DBRE + Middleware SRE + owning team | Impossible (message loss) | All consumers of that queue | Require 30-day deprecation window with confirmed zero traffic |
| Exchange binding change | HIGH | Middleware SRE + owning teams | Hard (requires re-binding) | All consumers on that binding | Messages may misroute during transition window |
| Kafka topic partition increase | HIGH | Middleware SRE | Not reversible | All consumers (rebalance triggered) | Consumer group rebalance; throughput gap during transition |
| Kafka consumer group offset reset | HIGH | Middleware SRE + owning team | Possible but risky | All consumers in group | Can cause duplicate processing or message skip depending on direction |
| MuleSoft flow deployment | MEDIUM | Integration team | Fast (redeploy prior version via Anypoint Runtime Manager) | All flows in that Mule application | Use blue-green deployment where possible |
| RabbitMQ policy change | MEDIUM | Middleware SRE | Fast (re-apply prior policy) | All queues in policy scope | HA policy changes trigger queue synchronization traffic |
| Kafka topic retention change | LOW | Middleware SRE | Fast | None (applies to future messages) | Reduction in retention may immediately delete messages below new threshold |
| Connection limit increase | LOW | Middleware SRE | Fast | None | May require broker configuration reload |

[Change Management]: All HIGH and CRITICAL middleware changes must go through the standard change management process with a documented rollback plan and consumer team sign-off. Topology changes (binding modifications, partition count increases) require written acknowledgment from all consumer team leads before the change window opens. An unsigned consumer team is a hard gate — the change does not proceed.

## SLO Cascade: Middleware to Application to User

[Service Level Management]: middleware SLOs are foundational. Application SLOs cannot exceed what the integration layer can guarantee.

| Layer | Example SLO | Dependency | Cascade Effect if Missed |
|-------|------------|-----------|------------------------|
| Kafka broker availability | 99.95% | Cluster health, ISR count | Consumer groups cannot consume; application processing stops |
| RabbitMQ message delivery latency | P99 <500ms from publish to consumer receipt | Broker throughput, consumer count | Downstream processing SLO missed; notifications delayed |
| MuleSoft flow error rate | <0.5% | Upstream system availability, flow logic | Sync integration failures; CRM/billing data divergence |
| Consumer lag SLO | <1000 messages, 99th percentile | Consumer throughput, producer rate | Business process delay; order processing backlog |
| DLQ remediation SLO | All DLQ messages acknowledged within 24 hours | Owning team on-call responsiveness | Silent data loss; compliance risk if messages contain financial events |

The Middleware SRE team reports broker-level SLOs. Product teams report consumer-level SLOs. The VP dashboard shows both — a healthy broker with consistently lagging consumers is a capacity and application quality problem, not a platform problem. Separating these in reporting prevents platform teams from absorbing blame for application team throughput issues.
