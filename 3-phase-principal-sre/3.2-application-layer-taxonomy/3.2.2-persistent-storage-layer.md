# Chapter 02: Persistent Storage Layer

**Layer position**: Bottom of the application stack. All other layers depend on this one.
**Cross-references**: [Incident Management] [Change Management] [Service Level Management]

---

## What Belongs Here

| Belongs | Does NOT Belong |
|---------|----------------|
| Relational: PostgreSQL, MySQL, Oracle, SQL Server | Redis, Memcached (Storage Abstraction layer) |
| NoSQL: MongoDB, Cassandra, DynamoDB, Couchbase | Application-managed SQLite (Application Services layer) |
| Object storage: S3, Azure Blob, GCS | In-process embedded stores |
| File systems: NFS, CIFS, EFS (EKS persistent volumes) | Message queues (Messaging layer) |
| Block storage: EBS, SAN volumes | Search indexes managed by app teams (Elasticsearch as app cache) |

---

## Why This Layer Gets Special Treatment

Every other layer in the stack is stateless or soft-stateful. When a Kafka consumer pod crashes, you restart it — no data lost. When a persistent storage node fails, you have a data risk event, not just a downtime event. That distinction changes every operational decision at this layer.

**Five properties that make this layer categorically different:**

**Stateful by definition.** Data survives process restarts. A crashed database pod is not the same as a crashed API pod. The process restarting is the easy part; the question is whether the data on disk is consistent.

**Highest blast radius on failure.** When this layer goes down, every consumer goes down simultaneously. No circuit breaker, no retry logic, no bulkhead protects a service whose only data source is unavailable. Application teams will flood your incident channel within 60 seconds. [Incident Management]

**Slowest recovery path.** Restore from backup: hours. Replication failover: minutes (with risk). Schema rollback: often impossible without data loss. Compare to an API pod restart: 30 seconds. Persistence layer failures have the longest tail.

**Most expensive horizontal scale.** Adding an API replica: one Kubernetes Deployment change. Adding a Kafka partition: one Terraform line. Sharding a relational database: a 6-month project involving schema redesign, application changes, and migration coordination across every consuming service.

**Connection-limited at the pinch point.** Every database engine has a hard ceiling on concurrent connections (`max_connections` in PostgreSQL, for example). Every microservice in your fleet connects through this ceiling. At T-Mobile scale with hundreds of microservices, connection exhaustion is a real failure mode — not a theoretical one.

## Persistence Tier SLO Requirements

Uniquely in the stack: **data integrity > availability**. A storage component should fail closed rather than serve corrupt or stale data. An API that returns a 503 is recoverable. An API that returns data from a corrupted page is not.

At this layer, the primary SLO dimensions are **RPO** and **RTO**, not just an availability percentage.

- **RPO (Recovery Point Objective)**: maximum acceptable data loss measured in time. "If the database failed right now, how much data are we willing to lose?"
- **RTO (Recovery Time Objective)**: maximum acceptable downtime. "How quickly must the database be back and accepting connections?"

Typical enterprise target: RPO ≤ 1 minute (requires synchronous replication), RTO ≤ 15 minutes (requires automated failover, pre-tested runbooks).

> **[Service Level Management] SLO cascade law**: the persistence layer SLO must be stricter than the SLO of every layer above it. If the database is 99.99% available, a data service built on top of it cannot commit to 99.99% — it can only commit to something weaker (99.9%), because the data service adds its own failure probability on top of storage failure probability. Violations of cascade law produce SLOs that are mathematically unachievable.

| Metric | Definition | Typical Target | Who Tracks It |
|--------|-----------|----------------|--------------|
| Availability | % time DB accepts connections | 99.99% (4.3 min/month downtime) | DBRE team |
| RPO | Max data loss on failure | ≤ 1 min (synchronous replication) | DBRE team |
| RTO | Time to restore service | ≤ 15 min | DBRE + Platform SRE |
| Query P99 latency | 99th percentile query response time | Workload-dependent; set per service | DBRE + App team |
| Replication lag | Primary-to-replica data delta | ≤ 100ms steady state | DBRE team |

## SRE Ownership Model

Three viable models. The right choice depends on organizational scale and database diversity.

**Option A — Dedicated Database Reliability Engineering (DBRE) team**
Best for: organizations with 10+ production databases, complex or diverse schemas, regulated data (PCI, HIPAA, SOX).
DBRE owns: SLOs, backup policy, replication topology, schema change governance, capacity planning, major incident command for storage events.
App teams own: their query patterns, their connection pool sizing, their ORM configuration.

**Option B — Embedded DBA within platform SRE**
Best for: smaller organizations with a standardized, homogeneous platform (all PostgreSQL on RDS, no Oracle, no Cassandra).
Works until: you hit 10+ databases with divergent schemas, or a regulated data requirement forces explicit ownership.

**Option C — Managed service + SRE wrapper**
Cloud provider owns the DB engine (RDS, Aurora, DynamoDB). SRE owns the operational wrapper: connection pooling (PgBouncer, RDS Proxy), backup validation, alerting, cost governance, parameter group management.
Appropriate for greenfield teams without DB expertise, or organizations deliberately offloading engine operations.

**The anti-pattern to avoid**: "the app team owns their database." This works until a schema migration takes down 12 services and no one owns the blast radius assessment. At T-Mobile scale — hundreds of microservices, shared domain databases, complex replication topologies — this model fails.

> **VP ownership decision**: at T-Mobile scale, the only viable model is a DBRE team with explicitly documented ownership boundaries. The DBRE team owns the storage layer SLOs. Application teams own their consumption patterns. When those two ownership domains collide (e.g., an application team's runaway query destabilizes the database), the DBRE team commands the incident and the application team is a supporting actor. Make this organizational contract explicit — it will be tested during your first major database incident.

## Backup and Recovery Ownership

| Function | Owner | Frequency | Governance |
|----------|-------|-----------|-----------|
| Backup schedule definition | DBRE team | Policy document | Annual review + change on schema/volume events |
| Backup execution automation | Platform SRE | Continuous | Alerting on backup job failure |
| Retention policy | DBRE + Compliance/Legal | Policy document | Data residency, regulatory retention requirements |
| Restore validation (automated) | DBRE team | Weekly minimum | Automated restore to isolated environment, checksum validation |
| Restore validation (full) | DBRE team | Monthly minimum | Full restore with RTO measurement, documented result |
| Quarterly restore drill | DBRE + Platform SRE | Quarterly | Documented RTO, VP sign-off on result |

**The "we have backups" trap.** Having backups and having tested restore procedures are categorically different things. Most organizations discover their restore procedure is broken during an actual disaster — wrong IAM permissions, corrupted backup, changed schema making the restore incompatible, missing dependencies. The only way to know your restore works is to run it.

> **VP ownership decision**: require quarterly restore drills with documented RTO measurement. The drill result — "we restored 2.1TB in 11 minutes 43 seconds" or "restore failed at step 4, root cause: expired snapshot copy permission" — is a VP-level metric. If your DBRE team cannot show you a restore drill result from the last 90 days, your RPO and RTO commitments are unvalidated. [Change Management]: backup schedule changes require DBRE approval; retention policy changes require Compliance sign-off.

## Replication and High Availability

| Mode | RPO | RTO | Write Latency Impact | Appropriate For |
|------|-----|-----|---------------------|----------------|
| Synchronous replication | 0 | Failover time (60–300s) | +2–5ms per write | Transactional data: orders, payments, account state |
| Asynchronous replication | > 0 (up to lag) | Failover time (60–300s) | None | Read-heavy analytics, reporting, non-financial data |
| Active-Active (multi-master) | 0 | ≈ 0 | Conflict resolution overhead | Rarely appropriate for relational; DynamoDB Global Tables is the managed exception |
| Active-Passive (hot standby) | Depends on replication mode | 60–300s automated failover | Depends on replication mode | Most common enterprise pattern |

**Kubernetes-specific caveat for EKS teams.** StatefulSets with persistent volumes provide pod-level HA, not database-level HA. When a database pod is rescheduled to a new node, the EBS/EFS volume re-attaches — but the database process must re-initialize, run recovery, and establish replication. Pod restart time on Kubernetes is not the same as RTO. Measure your actual database recovery time, not your pod startup time. These numbers diverge significantly for large databases.

## Performance Characteristics: The VP Metrics

| Metric | What It Measures | Normal Range | Alert Threshold | Response Action |
|--------|-----------------|-------------|----------------|----------------|
| IOPS | Read/write operations per second | Workload-dependent | >80% of provisioned IOPS sustained | Scale storage tier or drive query optimization |
| Query P99 latency | Worst 1% of query execution time | <100ms (OLTP) | >500ms sustained | Query analysis, index review, connection pool check |
| Connection count | Active connections vs. max_connections | <70% of max | >85% of max | Increase pool size, add read replicas, or find connection leak |
| Replication lag | Primary-to-replica data delta | <100ms | >5 seconds sustained | Investigate replica I/O, long-running transactions, network throughput |
| Table bloat (PostgreSQL) | Dead row accumulation ratio | <20% overhead | >30% overhead | VACUUM schedule review, autovacuum tuning |
| Lock wait time | Query time blocked waiting for row/table lock | Minimal | >500ms frequent | Identify blocking transaction, escalate to schema redesign review |

At T-Mobile scale, **connection count** is the metric most likely to create a surprise. With hundreds of microservices each running connection pools, a deployment that increases pool size by 10 connections across 50 services adds 500 connections to the database — potentially crossing the max_connections ceiling without any individual team realizing the aggregate effect. DBRE must own a global connection budget view.

## Change Risk at the Persistence Layer

This is the highest-risk change category in the entire application stack. [Change Management]

### Schema Migration Risk

Adding a NOT NULL column to a 50M-row table acquires an exclusive table lock for the full duration of the migration. Every query that touches that table blocks. All consumers stop. In PostgreSQL, this is an Access Exclusive Lock — the highest lock level available. No reads, no writes, from any session.

**[Change Management]: all schema migrations require:**

1. Full consumer impact analysis — which services query this table, what their peak traffic looks like
2. Migration tested in non-production at production data volume (not a subset — data volume changes execution time by orders of magnitude)
3. Rollback script written AND validated in non-production (a rollback script that has never been run is not a rollback plan)
4. Maintenance window OR zero-downtime migration strategy — not both optional, one is required
5. DBRE sign-off AND owning application team sign-off
6. Change record at the highest risk tier

### The Expand-Contract Pattern (Zero-Downtime Schema Changes)

For tables that cannot tolerate a maintenance window:

| Phase | Action | Lock Type | Consumer Impact |
|-------|--------|-----------|----------------|
| 1 — Expand | Add new column as nullable | Brief metadata lock | None |
| 2 — Backfill | Populate new column for existing rows (batched) | Row-level locks per batch | Minimal if batched slowly |
| 3 — Constrain | Add NOT NULL constraint after backfill complete | Brief metadata lock | None |
| 4 — Contract | Remove old column after consumers migrated | Brief metadata lock | None |

Four separate deployments. Weeks of elapsed time. Zero downtime. This is the correct pattern for high-traffic tables. Any shortcut produces a maintenance window.

### Index Change Risk

Adding an index in PostgreSQL without `CONCURRENTLY` acquires a lock for the build duration — same risk as a schema migration. `CREATE INDEX CONCURRENTLY` avoids the lock but doubles build time and is more I/O intensive. Index builds are resource-intensive and slow replica catch-up. Schedule index operations during low-traffic windows even with CONCURRENTLY.

### Data Migration Risk

Moving data between tables, reformatting stored values, deduplication — each requires:
- Read consistency analysis: what happens if a consumer reads during the migration?
- Idempotency: can the migration be safely re-run if interrupted?
- Volume estimate: how long does this take at production row count?

> **Incident routing implication**: any incident involving slow query times, connection pool exhaustion, replication lag, data corruption, or storage capacity alerts routes **first to DBRE**, not to the application team. The application team is a supporting role until DBRE confirms the persistence layer is not the root cause. Routing application teams to lead a database incident wastes the first 20–30 minutes of response time while they eliminate storage causes they are not equipped to diagnose.

> **VP ownership decision**: DBRE team owns all persistence layer SLOs. Application teams own their query patterns and connection pool sizing. These ownership domains are not symmetric — if a service exceeds its connection quota and degrades its own performance, that is an application team incident. If the same connection pattern degrades the database for all consumers, DBRE commands the incident and the application team is a supporting actor. Draw this line explicitly in your escalation policy before the incident happens, not during it.
