## The Layer Ownership Matrix

Every layer in the application stack must have exactly one team that owns it. Shared ownership is no ownership. The matrix below is the authoritative routing table for incidents, changes, and SLO accountability.

| Layer | Platform Owner | SLO Owner | Incident Response Lead | Change Approval Authority | On-Call Team |
|---|---|---|---|---|---|
| 1. Persistent Storage | DBRE team | DBRE team | DBRE team | DBRE lead + CAB | DBRE on-call |
| 2. Storage Abstraction | Platform SRE | App team (for app-owned cache) | Platform SRE (infra) / App team (usage) | Platform SRE | Shared on-call (platform + app) |
| 3. Data Services | Domain team | Domain team | Domain team | Domain team lead + consumer sign-off | Domain team on-call + Platform SRE backup |
| 4. Middleware / Integration | Middleware SRE | Middleware SRE (broker) / App team (consumer) | Middleware SRE | Middleware SRE lead + CAB for topology changes | Middleware SRE on-call |
| 5. Application Services | Product team | Product team | Product team + embedded SRE | Product team lead | Product team on-call + SRE backup |
| 6. API Gateway / Edge | Platform/Network SRE | Platform SRE (edge) | Platform/Network SRE | Platform SRE lead + Security | Platform SRE on-call |
| 7. Presentation / UI | Product team | Product team | Product team | Product team lead | Product team on-call |

## Cross-Layer Incident Routing

### Decision Flowchart: 5-Minute Layer Identification

When an incident is reported, route using this sequence — top to bottom, stop at the first failing layer:

**Step 1: Is the edge healthy?**
Check API gateway metrics: request rate, error rate, latency at the gateway boundary.
If edge has elevated error rate → Platform/Network SRE leads. Stop routing downstream until edge is cleared.

**Step 2: Is the application service healthy?**
Check APM traces: error rate by service, latency by service.
Which service shows the first anomaly in the call chain? That service's product team leads.

**Step 3: Is the message broker healthy?**
Check Kafka consumer lag by consumer group. Check RabbitMQ queue depth and consumer connection count.
If consumer lag is growing or queue depth is spiking → Middleware SRE assesses broker health vs. consumer health.

**Step 4: Is the data service healthy?**
Check data service APM: query latency, error rate at the data API boundary.
If data service latency is elevated → Domain team leads with DBRE support.

**Step 5: Is storage healthy?**
Check DB metrics: query execution time, connection pool utilization, replication lag.
If storage metrics are abnormal → DBRE leads.

### Symptom-to-Layer Routing Table

| Symptom | First Layer to Check | Lead Team | Supporting Teams |
|---|---|---|---|
| All users getting 500 errors | Edge (gateway error logs) | Platform SRE | App team, Security |
| Specific API endpoint slow | Application Services (distributed traces) | App team | DBRE (query check), Middleware SRE (message check) |
| Messages not being processed | Middleware (consumer lag, DLQ depth) | Middleware SRE | App team (consumer health) |
| Data returned is stale or wrong | Data Services → Persistence | Domain team | DBRE |
| UI broken but APIs return 200 | Presentation / UI | UI/Product team | Platform SRE (CDN check) |
| Database connection timeouts | Persistent Storage | DBRE | App team (connection pool check), Platform SRE |
| Replication lag growing | Persistent Storage | DBRE | Platform SRE (network check) |
| Certificate errors in clients | Edge | Platform SRE | Security |
| Elevated error rate on one service only | Application Services | Owning product team | DBRE, Middleware SRE as appropriate |

## The "It's the Database" Hypothesis

The most common wrong guess in a distributed system: when something is slow, blame the database. This reflex comes from years of monolithic architecture experience where the database WAS the architectural bottleneck. At T-Mobile scale — hundreds of microservices, service mesh, async middleware — the slowdown can originate at any layer.

**Before escalating to DBRE, validate with evidence:**

1. **Check the application trace**: Is the slow span a DB call or a service-to-service call? APM will show you the exact span. If the slow span is a gRPC call to another service, DBRE is the wrong team.
2. **Check the query execution plan**: Is the query slow because of a missing index or because of lock contention? These are different problems requiring different responses.
3. **Check the connection pool**: A 4-second timeout does not mean the query took 4 seconds. It may mean the connection pool was exhausted for 3.95 seconds before a connection was available. The query itself took 50ms. Escalating to DBRE for a connection pool exhaustion incident wastes their time and delays the fix (application tuning required, not DB tuning).

**Evidence standard for layer attribution:**

| Claim | Required Evidence | Not Sufficient |
|---|---|---|
| "The DB is slow" | Slow query log showing query execution time >threshold | Application timeout (could be pool exhaustion) |
| "The broker is down" | Cluster health API returning error or node offline | Consumer lag spike (could be consumer fault) |
| "The app service is broken" | Service error rate >SLO in APM | Gateway 500 errors (could be gateway config) |
| "The network is the issue" | Packet loss or latency at network layer, traceroute | Any timeout (timeouts are multi-layer) |

> **VP ownership decision**: require evidence-based layer attribution in every incident before routing to a specialist team. "We think it's the database" is a hypothesis, not an attribution. The DBRE team's on-call time is finite. Mis-routing a SEV2 to DBRE for a connection pool problem that the app team owns costs 20–40 minutes of incident resolution time and degrades the relationship between teams. The IC must gate team escalation on evidence, not on gut instinct. Make this a standard postmortem finding category: "mis-routed due to missing layer attribution."

## SLO Cascade: The Mathematical Constraint

A downstream layer's SLO cannot exceed the SLO of the layers it depends on, minus the failure probability introduced by the downstream layer itself.

**Mathematical form:**
```
SLO(layer N) ≤ SLO(layer N-1) × (1 - layer_overhead_N)
```

**Practical implication**: if Persistent Storage SLO is 99.99%, Data Services cannot also promise 99.99%. The Data Services layer has its own failure modes on top of storage failure modes. Every hop in the dependency chain consumes error budget.

### SLO Cascade Reference Table

| Layer | SLO Target | Budget Source | Calculation Rationale |
|---|---|---|---|
| Persistent Storage | 99.99% | Baseline | Most critical layer; all downstream budgets derive from this |
| Storage Abstraction | 99.99% | Inherits from storage | Connection pool adds minimal overhead if properly sized and monitored |
| Data Services | 99.9% | Storage 99.99% - service risk | Data service has its own failure modes (deploy failures, schema bugs, OOM) |
| Middleware / Integration | 99.9% | Independent (parallel path) | Broker HA provides its own redundancy; not in primary dependency chain of storage |
| Application Services | 99.5% | Data service 99.9% - app risk | App calls data service + middleware + other dependencies; each adds risk |
| API Gateway / Edge | 99.99% | Independent (infrastructure layer) | Must sustain availability even when app services degrade; handles partial degradation |
| Presentation / UI | 99.5% | App services + edge | User-facing experience is the end-to-end chain; error budget is the union of all upstream risks |

> **[Service Level Management]**: the SLO cascade must be validated per service dependency chain, not applied globally. A service with 5 upstream dependencies must account for each upstream's error budget in its own SLO derivation. A shared SLO target applied uniformly across all services ignores the heterogeneity of dependency depth — a leaf-node service and a core orchestration service cannot carry the same SLO without mathematical analysis.

## Layer-Based Change Risk Classification

[Change Management]: all CRITICAL and HIGH risk changes require Change Advisory Board review with a validated rollback plan demonstrated in non-production. No CRITICAL change goes to production without a tested rollback procedure.

| Layer | Change Type | Risk | Approval | Rollback | Lead Time |
|---|---|---|---|---|---|
| Persistent Storage | Schema migration (DROP/ALTER) | CRITICAL | DBRE + CAB + all consumer teams | Manual SQL reverse migration | 2 weeks minimum |
| Persistent Storage | Index addition/deletion | HIGH | DBRE | Drop added index; recreate deleted index | 1 week |
| Persistent Storage | Backup policy change | MEDIUM | DBRE + Compliance | Revert policy document | 3 days |
| Data Services | Breaking API change | HIGH | Domain lead + all consumers signed off | Maintain v1 endpoint; rollback new version | 1 week (v1 must remain live) |
| Data Services | Additive API change (new field) | LOW | Domain lead | Redeploy previous container image | Standard |
| Middleware / Integration | Queue or topic deletion | CRITICAL | Middleware SRE + CAB | Impossible to replay lost messages | 4 weeks (full deprecation cycle) |
| Middleware / Integration | Broker topology change | HIGH | Middleware SRE + all owning teams | Re-apply previous topology config | 1 week |
| Middleware / Integration | Flow or consumer deploy | MEDIUM | Integration team lead | Redeploy previous version | Standard |
| Application Services | Service deploy | LOW | App team lead | Blue-green rollback or canary abort | Standard |
| Application Services | Config change via feature flag | LOW | App team lead | Toggle flag to previous state | Immediate |
| API Gateway / Edge | Route change | MEDIUM | Platform SRE | Revert gateway config | Same day |
| API Gateway / Edge | WAF rule change | HIGH | Security + Platform SRE | Disable rule (canary first) | 3 days with canary |
| API Gateway / Edge | TLS cert rotation | LOW | Platform SRE | Previous cert (if not expired) | Standard (automated preferred) |
| Presentation / UI | Frontend deploy | LOW | Product team lead | CDN cache invalidation + redeploy previous build | Immediate |

## Platform Team vs. Product Team Split

### Platform Team owns: Layers 1, 2, 4, 6

Persistent Storage, Storage Abstraction infrastructure, Middleware/Integration, API Gateway/Edge — horizontal concerns shared by all product teams.

**Responsibilities:**
- Infrastructure provisioning, uptime, security patching, upgrade cycles for shared platforms
- Shared tooling: monitoring platforms, deployment pipelines, secrets management, service mesh
- On-call for platform availability, not for application-specific business logic

### Product Team owns: Layers 3, 5, 7

Data Services, Application Services, Presentation/UI — vertical concerns specific to business domains.

**Responsibilities:**
- Business logic, data models, API contracts, user experience
- On-call for feature-level behavior, application errors, data quality within the domain
- Embedded SRE support from Platform team for production readiness reviews

### Where Boundary Disputes Arise

| Dispute | Resolution |
|---|---|
| "Our service needs a new database" | Product team requests; Platform team (DBRE) provisions and hands off. DB instance ownership = DBRE. Schema ownership = Product team. |
| "Our Kafka consumer is lagging" | Product team diagnoses their consumer application. Middleware SRE diagnoses the broker. If broker is healthy and consumer lag is growing, it is a product team problem. |
| "The API gateway is rate-limiting us" | Product team owns the rate limit policy (what the limit should be). Platform team owns the enforcement infrastructure. Product team submits a config change request to Platform SRE. |
| "The Redis cache is evicting too aggressively" | If Redis is shared infrastructure, Platform SRE owns eviction policy. If Redis is a dedicated sidecar, the product team owns it. Layer 2 ownership is infrastructure vs. app-specific. |

## Org Design by Layer: Team Structure for Accountability

Every layer must have an identifiable team with an on-call rotation and an SLO owner. "Everyone owns it" is equivalent to no one owning it — ownership disputes surface at the worst possible time: during a SEV1.

### Recommended Team Structure at T-Mobile Scale

| Team | Size | Layers Owned | Accountability |
|---|---|---|---|
| DBRE team | 5–8 engineers | Persistent Storage (Layer 1) | All databases, backup validation, replication, schema governance |
| Platform SRE | 8–12 engineers | Storage Abstraction (Layer 2), Edge/Gateway (Layer 6), shared platform | Connection pools, CDN, API gateway, service mesh, secrets |
| Middleware SRE | 4–6 engineers | Middleware / Integration (Layer 4) | RabbitMQ, Kafka, TIBCO EMS, MuleSoft runtime, ESB |
| Domain SRE (embedded) | 1–2 per domain | Data Services (Layer 3), Application Services (Layer 5) — within domain | SLO governance, runbook ownership, production readiness within their domain |
| UI/Frontend SRE or Platform Engineer | 1–2 | Presentation / UI infrastructure (Layer 7) | CDN configuration, SSR platform, RUM instrumentation, Lighthouse CI |

> **VP ownership decision**: if you do not have a dedicated Middleware SRE team and RabbitMQ or Kafka is tier-1 infrastructure at your organization, you have a governance gap. The message broker is owned in practice by whoever is on-call when it fails — and that is almost always the wrong team. A product team engineer paged at 2am for a RabbitMQ cluster partition will attempt an application restart, not a broker recovery. Forming a Middleware SRE team is a VP-level organizational decision with direct SEV1 impact. The cost of not having one becomes visible only when the broker fails at peak traffic.

## Investment Allocation by Layer

Prioritize investment where blast radius is largest and recovery complexity is highest, not where incidents are most frequent.

| Layer | Blast Radius | Recovery Complexity | Incident Frequency | Investment Priority | Recommended Investment |
|---|---|---|---|---|---|
| Persistent Storage | CRITICAL — all consumers blocked | HIGH — hours for full restore | LOW — rare, catastrophic | #1 | DBRE team, automated backup validation, replication monitoring, DR failover runbooks, chaos engineering for failover |
| Middleware / Integration | HIGH — all async consumers blocked | MEDIUM — broker restart measured in minutes | MEDIUM | #2 | Middleware SRE team, consumer lag alerting, DLQ automation, multi-node HA, poison message tooling |
| API Gateway / Edge | HIGH-WIDE — all external users impacted | LOW — config rollback is fast | MEDIUM | #3 | Automated cert rotation, WAF canary pipeline, synthetic monitoring from multiple egress points |
| Application Services | MEDIUM — one service domain impacted | LOW — container redeploy is fast | HIGH | #4 | Embedded SRE, runbook coverage target 100%, circuit breaker standards, chaos engineering per service |
| Data Services | MEDIUM — one business domain impacted | LOW — redeploy fast | MEDIUM | #5 | Contract registry, consumer impact analysis tooling, API versioning governance |
| Presentation / UI | HIGH-WIDE — all users see it | LOW — CDN cache purge is fast | HIGH | #6 | RUM platform, Lighthouse CI in pipeline, crash reporting, feature flags for UI rollback |
| Storage Abstraction | MEDIUM — one service impacted | LOW — cache restart fast | MEDIUM | #7 | Connection pool monitoring, cache hit rate alerting, Redis/Memcached HA |

> **Incident routing implication**: the governance model exists to eliminate ambiguity on the incident bridge. When the Incident Commander asks "who owns this component?", there is exactly one correct answer. The layer ownership matrix is the IC's routing table. Every configuration item in the CMDB must map to a row in that matrix with a named team and on-call rotation. Unclassified components become unowned components the moment an incident occurs.

> **VP ownership decision**: the VP's governance responsibility is to maintain the layer ownership matrix as a living document, reviewed quarterly. When a new service is deployed, it receives a layer classification and an owner before it goes to production — this is a hard gate in the production readiness review process. When an ownership dispute arises during an incident, the VP or their delegate resolves it in real time and updates the matrix to prevent recurrence. Governance debt — unclassified services, disputed ownership — accumulates silently until the next SEV1 makes it visible to leadership.
