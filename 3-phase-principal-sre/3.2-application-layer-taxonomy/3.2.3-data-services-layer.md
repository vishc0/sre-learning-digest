# Chapter 03: Data Services Layer

**Layer position**: Directly above Persistent Storage. Directly below Application Services.
**Cross-references**: [Incident Management] [Change Management] [Service Level Management]

---

## What Belongs Here

Services that sit directly on top of a persistent store and expose data as a service via a clean API contract — not raw SQL, not shared database connections, not business logic that calls other services.

| Belongs | Does NOT Belong |
|---------|----------------|
| Customer Data Service: wraps customer_db, exposes `/customers/{id}`, `/customers/search` | Services that call OTHER data services to compose a response (Application Services layer) |
| Inventory Service: wraps inventory_db, exposes stock levels, reservations, adjustments | Services that enforce business rules or orchestrate workflows (Application Services layer) |
| Order Query Service: read model from CQRS pattern, exposes order history and status | Raw database connection strings shared across teams (anti-pattern) |
| Product Catalog Service: wraps product_db, exposes catalog search and detail | Services that own no data — only transform or route (Integration layer) |
| Account Balance Service: wraps ledger_db, exposes account state | |

**The defining test**: if you removed this service, could its consumers fall back to direct database access? The answer should be no — because they should never have had direct database access in the first place. This layer exists to make direct database access unnecessary.

---

## Why This Layer Exists

**Hides storage implementation.** Consumers do not know or care whether the backing store is PostgreSQL, DynamoDB, or Cassandra. They call the API. When the Inventory team migrates from Oracle to PostgreSQL, no consumer changes.

**Enforces data ownership.** Exactly one team owns each data store through its data service. Direct database access from other teams is forbidden. This is an organizational constraint, not just a technical one — enforce it with network policy (no external DB port access) and organizational policy (DB credentials are not shared outside the owning team).

**Enables independent evolution.** The data service team can refactor their internal schema without breaking consumers, as long as the API contract is stable. Without this layer, every schema change requires coordinating every consuming team simultaneously.

**Enables consistent data access patterns.** Authentication, authorization, audit logging, rate limiting, and caching policy are applied at the data service layer — not replicated in every consuming service.

**The shared database anti-pattern (what this layer replaces).** Multiple teams write directly to the same database. Schema changes require coordinating every team simultaneously. A single bad migration from Team A takes down Teams B, C, and D. Hotspots on one team's tables degrade performance for all teams. At T-Mobile scale, with hundreds of microservices and dozens of domain teams, shared database access creates a fan-in dependency nightmare that makes the persistence layer an organizational bottleneck rather than a technical primitive. This is non-negotiable at enterprise scale.

## The Data Ownership Problem: Who Owns the Contract?

| Ownership Domain | Owner | What They Control |
|-----------------|-------|------------------|
| API contract (schema, versioning, deprecation timeline) | Data service team | OpenAPI spec, breaking vs. additive change policy, deprecation notice period |
| SLO commitment | Data service team | Availability, latency P99, data freshness |
| Query pattern optimization | Data service team | How their API translates to storage queries |
| Consumer usage (request rates, retry logic, circuit breaker config) | Each consuming team | Their call volume, their timeout config, their fallback behavior |
| Storage layer performance | DBRE team | Index strategy, replication topology, connection limits |

The conflict that plays out repeatedly without clear ownership: the data service team wants to refactor their schema; consumers depend on the current response shape. Without a formal contract — OpenAPI spec, versioning policy, documented deprecation timeline — this becomes a political negotiation every time a change is needed. The data service team holds the data hostage to political consensus rather than operating under a governed change process.

> **VP ownership decision**: require every data service to maintain a published OpenAPI spec and a versioning policy. Version 1 of any API must be supported for a minimum of 6 months after version 2 ships. Breaking changes require a formal change request with written consumer sign-off before work begins. Additive changes (new optional fields, new endpoints) follow standard change process. This policy converts a recurring political negotiation into a governed process. [Change Management]

## SLO Ownership at the Data Services Layer

The data service SLO is separate from and weaker than the storage layer SLO. Both SLOs must exist. They are not interchangeable.

**Ownership clarity on slow response incidents:**
- Data service is slow because of a bad query written by the data service team → data service team incident
- Data service is slow because the database is overloaded (IOPS exhaustion, lock contention, connection ceiling) → DBRE incident, data service team is a supporting actor
- Both teams may be in the same incident bridge. Ownership determines who commands.

> **[Service Level Management] SLO cascade law at this layer**: the data service SLO must be weaker than the persistence layer SLO. If the underlying database commits to 99.99% availability, the data service built on top of it should commit to 99.9% or lower — because the data service adds its own failure probability (pod restarts, memory pressure, deployment failures, application bugs) on top of storage failure probability. An SLO that equals or exceeds the storage SLO is not achievable and should not be published. [Service Level Management]

| SLO Dimension | Owner | Typical Target | Measurement Tool |
|--------------|-------|---------------|----------------|
| Availability | Data service team | 99.9% | Prometheus uptime probe, Datadog synthetic |
| Query P99 latency | Data service team | <200ms (data retrieval, not business logic) | Distributed tracing (Jaeger, Zipkin, Datadog APM) |
| Data freshness | Data service team | Workload-dependent; set per service | Custom metric + Splunk alert on lag |
| Error rate | Data service team | <0.1% | APM (Datadog, New Relic), Prometheus error rate metric |
| Cache hit rate (if applicable) | Data service team | >80% for read-heavy services | Redis INFO stats, custom metric |

## Data Services Team Model

Data services are owned by **domain teams**, not platform teams. This is a governance-critical distinction.

**Domain team model (correct):**
- The team that owns the business capability owns the data service for that capability
- Customer domain team → Customer Data Service
- Order domain team → Order Query Service
- Inventory domain team → Inventory Service
- Domain team owns: data model decisions, query optimization, API contract governance, data quality standards, SLO commitments

**Platform SRE provides** (supportive role, not ownership):
- Deployment infrastructure (EKS, Helm charts, CI/CD pipelines)
- Observability tooling (Prometheus, Grafana, distributed tracing)
- On-call rotation support (paging, runbook templates, incident tooling)
- Connection pooling infrastructure (PgBouncer, RDS Proxy)

**The platform team ownership anti-pattern.** When the platform team owns data services, every data contract change requires platform team involvement. Domain teams lose accountability for data quality — "that's a platform problem." Platform team becomes a bottleneck for all data access across the organization. This is a governance failure, not a technical one. Platform teams should own the infrastructure that runs data services, not the data services themselves.

## Change Governance at the Data Services Layer

| Change Type | Risk Level | Process Required |
|------------|-----------|-----------------|
| Breaking change (remove field, change type, change semantics) | HIGH | Consumer impact analysis, API versioning, coordinated release, rollback plan, written consumer sign-off |
| Additive change (new optional field, new endpoint) | LOW | Standard change process, no consumer coordination required |
| Storage-layer schema change that surfaces through the API | MEDIUM-HIGH | Coordinated between DBRE (storage change) and data service team (API contract change) — two linked change records |
| Performance optimization (query rewrite, index addition) | LOW-MEDIUM | Standard change process; load test required if query behavior changes significantly |
| Dependency version upgrade | LOW | Standard change process; regression test suite required |

**Deployment strategy for data service releases:**
Use blue-green or canary deployment for all data service changes. This enables rollback without consumer disruption if a change introduces regression at the API contract level. A canary that routes 5% of traffic to the new version exposes contract regressions before full rollout. [Change Management]

**Consumer impact analysis process:**

1. Identify all services that call this data service (use service mesh telemetry — Istio, Linkerd — or API gateway logs)
2. Notify owning teams with change description, timeline, and impact assessment
3. For breaking changes: get explicit written sign-off from each consuming team lead
4. Plan coordinated rollout — consumer updates and data service v2 launch must be sequenced
5. Keep v1 running for the full deprecation window (minimum 6 months) while consumers migrate

At T-Mobile scale, step 1 is non-trivial. Without a service mesh or API gateway providing call graph visibility, identifying all consumers of a data service requires manual audit. This is an operational cost of not investing in observability infrastructure.

## The Data Gravity Problem

**Data gravity** is the tendency for services to accumulate data responsibilities beyond their original scope, because they already have the data — and adding a new table or endpoint to an existing service is always easier than standing up a new service.

**How it happens:**
The Customer Data Service ships with customer profile data. Six months later: add session data (we already have the customer_id). Six months later: add preference data (we need to know the customer). Six months later: add marketing flags (the marketing team asked nicely). Six months later: add compliance audit events (the audit team needed somewhere to put them).

**Result:** The Customer Data Service now has 47 endpoints, owned by one team, with 12 consuming teams. The team cannot change anything without a full-org impact analysis. Every sprint is blocked on coordinating with consumers. The service is unmaintainable.

**Detection signals** (monitor quarterly):

| Signal | Threshold | Action |
|--------|-----------|--------|
| Number of endpoints | >15–20 | Architecture review |
| Number of direct consumer teams | >5 | Architecture review |
| Number of distinct data domains served | >2 | Domain decomposition planning |
| Time to deliver a breaking change | >2 sprints | Governance audit |

**Treatment: domain decomposition.** Split the Customer Data Service into:
- Customer Profile Service (identity, contact, demographics)
- Customer Preference Service (communication preferences, UI settings)
- Customer Compliance Service (consent records, audit events, regulatory data)

Each service has a clear bounded context, a small consumer list, and a team with clear ownership accountability.

> **VP ownership decision**: monitor data service dependency fan-out on a quarterly architecture review cadence. Any data service with more than 5 direct consumer teams or more than 15 endpoints is placed on an architecture review agenda. The review output is a decision: decompose, freeze new endpoints (no additions without architecture sign-off), or accept and formally document the organizational risk. "We know this service is a monolith and we've accepted that" is a valid decision. "We didn't notice it grew to 47 endpoints" is a governance failure.

## The Data Contract Registry

At T-Mobile scale, you cannot govern what you cannot see. A Data Contract Registry is the governance instrument that makes this layer manageable.

**What the registry contains:**

| Field | Content |
|-------|---------|
| Service name | Canonical name, kebab-case |
| Owning team | Team name + team lead contact |
| API spec | Link to published OpenAPI spec (versioned) |
| Current version | v1, v2, etc. |
| Deprecated versions | Versions still running with sunset dates |
| SLO | Published availability, latency, freshness commitments |
| Consumer list | All teams consuming this service (auto-populated from service mesh data) |
| Storage dependencies | Which databases/tables this service owns |
| Data classification | PII, PCI, public, internal |

**Who maintains it:** Each data service team owns their own registry entry. A platform governance team audits completeness quarterly.

**How it's used:**
- Change impact analysis: "I need to change Customer Data Service v1 — who do I need to notify?" → registry consumer list
- Incident routing: "Customer Data Service is down — who owns it?" → registry owning team
- Architecture review: "Which services have >5 consumers?" → registry fan-out query
- Compliance audit: "Which services handle PII?" → registry data classification field

> **VP ownership decision**: establish a Data Contract Registry as a first-class governance artifact, maintained alongside the service catalog. Make registry completeness a team health metric — teams with incomplete registry entries are flagged in the quarterly SRE review. The registry is not a nice-to-have at T-Mobile scale; it is the primary instrument for change governance, incident routing, and organizational ownership clarity across hundreds of microservices.

---

> **Incident routing implication**: when a data service is slow or returning errors, the data service team leads the incident. DBRE is a supporting SME, available to rule out storage-layer causes. Consuming application teams have no incident command responsibility unless their traffic pattern is identified as the root cause (e.g., a runaway query from a consumer that bypasses rate limiting). In that case, the application team addresses their traffic pattern; the data service team continues to own service recovery. Routing the wrong team to command a data service incident wastes the first 20–30 minutes of response time. [Incident Management]

> **VP ownership decision**: establish a Data Contract Registry — a catalog of all data services, their owners, their API specs, their SLOs, and their consumer list. This registry is your governance instrument for change impact analysis, incident ownership routing, and quarterly architectural health reviews. Without it, ownership conversations during incidents are improvised. With it, the answer to "who owns this?" is a registry lookup, not a Slack poll.
