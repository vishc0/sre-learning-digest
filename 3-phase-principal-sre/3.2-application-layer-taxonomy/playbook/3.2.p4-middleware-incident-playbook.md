# Middleware Incident Playbook

**Subtitle**: RabbitMQ, Kafka, and MuleSoft — broker vs. consumer triage, DLQ analysis, poison message handling, and recovery procedures

> **Cross-references**: [Incident Management] [Change Management] [Service Level Management]

---

## 1. Purpose

Message broker incidents are the most commonly mis-diagnosed incident type in a microservices architecture. The symptom ("messages aren't processing") can be caused by the broker, the consumer, the producer, or a dependent downstream service. This playbook provides the decision tree, diagnostic commands, and recovery procedures to resolve broker incidents correctly and fast.

**Covered scenarios:**
- Consumer lag growing (Kafka or RabbitMQ)
- Dead Letter Queue (DLQ) filling up
- Poison message identification and handling
- Broker node restart (RabbitMQ)
- Kafka partition leadership issues
- MuleSoft flow failure and replay

**Lead team**: Middleware SRE on-call for all broker infrastructure issues. Product team on-call for all consumer application issues. The distinction matters — see section 2.

## 2. Initial Triage: Is It the Broker or the Consumer?

This is the most critical decision in a middleware incident. Getting it wrong costs 30+ minutes.

### Broker Health Indicators

A healthy broker does not have any of these:
- Memory alarm active (RabbitMQ: `mem_alarm = true`)
- Disk alarm active (RabbitMQ: `disk_free_alarm = true`)
- Cluster partition detected (RabbitMQ: `partitions` field non-empty in cluster status)
- ISR (in-sync replicas) count < replication factor for any Kafka partition
- Kafka broker JVM GC pause time > 500ms sustained
- Network saturation on broker nodes (>80% of interface capacity)
- Kafka leader election rate elevated (more than occasional)

**If any broker health indicator is active → Middleware SRE leads. The consumer is secondary.**

### Consumer Health Indicators

A healthy consumer does not have any of these:
- Increasing exception rate in consumer application logs
- Consumer restart rate elevated (container restarts, process crashes)
- Consumer connection drops and reconnects (visible in broker connection metrics)
- Processing time per message growing (consumer is slow, not the broker)
- Consumer group rebalancing repeatedly (instability in consumer group)

**If broker is healthy and consumer indicators are active → Product team leads for that consumer.**

### The Critical Distinction

| Scenario | Root Cause | Lead Team | Evidence |
|---|---|---|---|
| Lag growing, broker healthy, consumer throwing exceptions | Consumer application bug | Product team | Consumer error logs showing exception per message |
| Lag growing, broker healthy, consumer processing slowly | Consumer throughput insufficient (slow downstream, slow query) | Product team | Consumer processing time per message elevated |
| Lag growing, broker unhealthy (memory alarm, partition) | Broker infrastructure failure | Middleware SRE | Broker health API, cluster status |
| Lag growing, broker healthy, consumer healthy | Producer spike (more messages than consumer can keep up with) | Both (assess capacity) | Producer publish rate vs. consumer processing rate |
| DLQ filling, broker healthy | Consumer rejecting messages (exception on processing) | Product team | DLQ message headers (x-death count), consumer logs |
| All consumers stopped | Broker cluster down or network partition | Middleware SRE | Broker cluster status, network diagnostics |

## 3. Consumer Lag Investigation (Kafka)

### Step 1: Identify Which Consumer Group Is Lagging

```bash
# List all consumer groups
kafka-consumer-groups.sh \
  --bootstrap-server kafka-broker-1:9092,kafka-broker-2:9092,kafka-broker-3:9092 \
  --list

# Describe a specific consumer group — shows lag per partition
kafka-consumer-groups.sh \
  --bootstrap-server kafka-broker-1:9092,kafka-broker-2:9092,kafka-broker-3:9092 \
  --describe \
  --group <consumer-group-name>

# Output columns: TOPIC | PARTITION | CURRENT-OFFSET | LOG-END-OFFSET | LAG | CONSUMER-ID | HOST | CLIENT-ID
```

**Read the output:**
- `LAG` column: current lag per partition
- `CONSUMER-ID`: which consumer instance owns each partition
- Empty `CONSUMER-ID`: partition is unassigned — consumer group is not running or rebalancing

```bash
# Check lag across all groups — sorted by lag descending
kafka-consumer-groups.sh \
  --bootstrap-server kafka-broker-1:9092 \
  --describe --all-groups 2>/dev/null \
  | awk 'NR>1 && $6 ~ /^[0-9]+$/ {print $1, $2, $3, $6}' \
  | sort -k4 -rn \
  | head 30
# Format: GROUP TOPIC PARTITION LAG
```

### Step 2: Check Whether Lag Is Growing or Stable

**A stable lag is not an incident.** A growing lag is. Run the describe command twice, 2 minutes apart, and compare.

```bash
# Automated lag trend check — run twice and compare
kafka-consumer-groups.sh --bootstrap-server kafka-broker-1:9092 \
  --describe --group <consumer-group-name> \
  | awk 'NR>1 {sum+=$6} END {print "Total lag:", sum}'
# Run at T=0 and T=2min. If T=2min > T=0, lag is growing.
```

**Splunk: consumer lag trend over time (if Kafka metrics are shipped to Splunk)**
```
index=kafka_metrics metric_name=kafka_consumer_group_lag consumer_group=<group>
| timechart span=1m max(metric_value) by topic
```

### Step 3: Check Consumer Application Logs

```
# Splunk: consumer exception rate
index=app_logs service_name=<consumer-service-name> log_level=ERROR
| timechart span=1m count by error_type

# Splunk: consumer restart events (Kubernetes)
index=kubernetes event_reason=BackOff OR event_reason=OOMKilled container=<consumer-container>
| timechart span=5m count
```

### Step 4: Check Broker Disk and Network for Read Saturation

```bash
# Kafka broker disk usage per log directory
kafka-log-dirs.sh \
  --bootstrap-server kafka-broker-1:9092 \
  --broker-list 0,1,2 \
  --topic-list <topic-name> \
  | grep -v "^{" | python3 -c "import sys,json; data=json.load(sys.stdin); [print(b['broker'], d['logDir'], d['partitions']) for b in data['brokers'] for d in b['logDirs']]"

# Broker network I/O (from node exporter / Prometheus if available)
# Look for: NetworkProcessorAvgIdlePercent < 20% (broker is network-saturated)
```

### Decision Tree: Lag Growing

```
Lag growing?
├── YES — Consumer healthy (no errors, no restarts)?
│   ├── YES — Producer rate > consumer processing rate?
│   │   ├── YES → Scale up consumer instances. Product team action.
│   │   └── NO → Broker throughput issue? Check broker I/O. Middleware SRE assesses.
│   └── NO (consumer has errors or restarts)
│       ├── Exceptions per message → Consumer bug. Product team fixes consumer.
│       └── OOMKilled / resource exhaustion → Scale consumer resources. Product team + Platform SRE.
└── NO (lag stable, even if high)
    → Monitor. No immediate action required. Assess whether stable lag is within SLO.
```

## 4. Dead Letter Queue Analysis

### Identifying DLQ Messages (RabbitMQ)

```bash
# List all queues including DLQs
rabbitmqctl list_queues name messages consumers memory \
  | grep -i "dlq\|dead\|error"

# Get message count in a specific DLQ
rabbitmqctl list_queues name messages \
  | grep <dlq-queue-name>

# Peek at a DLQ message without consuming it (requires management plugin)
# GET /api/queues/<vhost>/<dlq-name>/get
curl -u <admin>:<password> \
  -X POST "http://rabbitmq-host:15672/api/queues/%2F/<dlq-name>/get" \
  -H "Content-Type: application/json" \
  -d '{"count":5,"ackmode":"ack_requeue_true","encoding":"auto","truncate":50000}'
```

**DLQ message headers to inspect:**
- `x-death`: array of death records — how many times the message died and why
- `x-death[0].reason`: `rejected` (consumer explicitly rejected), `expired` (TTL exceeded), `maxlen` (queue full)
- `x-death[0].queue`: which queue the message died in
- `x-death[0].exchange`: which exchange it came from
- `x-first-death-reason`: the original reason for the first death

### Triage: Poison vs. Volume DLQ

| DLQ Scenario | Indicator | Cause | Response |
|---|---|---|---|
| Poison messages | `x-death` count high (>3), same message ID dying repeatedly | Consumer cannot process this specific message (schema mismatch, data corruption, business rule violation) | Quarantine, analyze, fix consumer or producer, selective replay |
| Volume DLQ (dependency unavailable) | Many different message IDs dying once or twice, `x-death.reason = rejected` with "connection refused" or timeout | Consumer's downstream dependency is unavailable | Fix the dependency first. Then controlled replay. |
| TTL expiry flood | `x-death.reason = expired`, messages from normal queue hours ago | Consumer was down; messages expired before consumer came back | Consumer was unavailable longer than TTL. Review TTL policy. |
| Queue overflow | `x-death.reason = maxlen` | Producer outpacing consumer; queue hit max-length limit | Scale consumer or increase queue capacity. |

**Splunk: DLQ rate over time**
```
index=rabbitmq_metrics metric=messages_ready queue=*dlq*
| timechart span=5m max(metric_value) by queue
```

### Volume DLQ Replay (Dependency Now Fixed)

Do NOT replay a volume DLQ at full speed. The original consumer was behind for a reason — flooding it with DLQ messages on top of live traffic will cause a second lag event.

**Controlled replay procedure:**
1. Verify the downstream dependency is healthy (not just restarted).
2. Determine normal consumer throughput (messages/second in steady state).
3. Replay DLQ at 20-30% of normal throughput — shovel or manual re-queue with rate limiting.
4. Monitor consumer lag and error rate during replay.
5. If lag grows during replay, pause replay, allow consumer to catch up, then resume.

```bash
# RabbitMQ shovel-based replay (requires shovel plugin)
# Set up a temporary shovel from DLQ back to the original exchange at a limited rate
# Management API: PUT /api/parameters/shovel/%2F/temp-replay-shovel
curl -u admin:password -X PUT \
  "http://rabbitmq-host:15672/api/parameters/shovel/%2F/temp-replay-shovel" \
  -H "Content-Type: application/json" \
  -d '{
    "value": {
      "src-uri": "amqp://",
      "src-queue": "<dlq-name>",
      "dest-uri": "amqp://",
      "dest-exchange": "<original-exchange>",
      "src-prefetch-count": 10,
      "reconnect-delay": 5,
      "publish-properties": {}
    }
  }'
# Remove shovel when replay is complete:
# DELETE /api/parameters/shovel/%2F/temp-replay-shovel
```

## 5. Poison Message Handling

### Detection

A poison message is a message that a consumer consistently cannot process, regardless of retry count. Detection signals:
- `x-death` count header value >= 3 (configurable per-consumer dead-letter-count policy)
- Same message ID appearing in DLQ repeatedly after replay attempts
- Consumer logs showing the same exception for the same message ID

```bash
# Check x-death count in a DLQ message (from the GET output above)
# Look for: "x-death": [{"count": 5, ...}]
# count >= 3 → treat as poison
```

### Analysis

Inspect the message content:
1. **Check the message body**: is it valid JSON/XML/Avro/Protobuf? Can it be parsed?
2. **Check the routing key**: is it routing to the correct consumer?
3. **Check the message headers**: are required headers present (correlation ID, content-type)?
4. **Check consumer logs** for the exact exception when this message was processed.

Common poison message causes:
- Schema mismatch: producer deployed a new schema version; consumer has not been updated
- Data corruption: producer sent malformed data (truncated, wrong encoding)
- Business rule violation: message contains data that violates a consumer business invariant
- Missing required fields: a field the consumer expects was added as optional on the producer side

### Handling Options

| Option | When to Use | Risk |
|---|---|---|
| Discard | Data is corrupt beyond recovery; no business value | Permanent data loss — document the decision |
| Fix and replay | Schema mismatch correctable; message data is valid | Low if consumer fix is tested first |
| Quarantine for manual review | Message may have business value but cannot be auto-fixed | No data loss; requires manual intervention |

**Consumer fix before bulk replay is mandatory.** If the consumer that produced the dead letter has not been fixed, replaying will re-deadletter all the messages.

**Test replay procedure:**
1. Fix the consumer application (or the producer schema mismatch).
2. Deploy the fix to staging.
3. Replay one poison message in staging.
4. Verify the message is processed successfully (no dead letter, correct output).
5. Deploy the fix to production.
6. Replay one poison message in production.
7. Verify success.
8. Replay remaining messages.

## 6. RabbitMQ Broker Restart Procedures

### Pre-Restart Checklist

Do NOT restart without completing this checklist. An uncoordinated restart loses unacknowledged messages and surprises consumer teams.

- [ ] Notify all consumer teams: "RabbitMQ restart planned at [time]. Consumer connections will drop and reconnect."
- [ ] Check current queue depth across all queues: `rabbitmqctl list_queues name messages`
- [ ] Verify HA/quorum configuration: are critical queues quorum queues or classic HA queues?
- [ ] Document current queue depths for post-restart comparison.
- [ ] Verify other nodes in the cluster are healthy (restart only the target node; cluster absorbs the loss).
- [ ] Confirm maintenance window is open if this is a planned restart.

### Graceful Node Restart (Node Is Responsive)

```bash
# Step 1: Stop the RabbitMQ application (not the Erlang VM) — drains connections gracefully
rabbitmqctl stop_app

# Step 2: From another cluster node, verify the node is gone from the cluster
# (On a healthy node)
rabbitmqctl cluster_status
# Target node should show as "disc nodes" but not in "running_nodes"

# Step 3: Restart the RabbitMQ application
rabbitmqctl start_app

# Step 4: Verify the node has rejoined the cluster
rabbitmqctl cluster_status
# Target node should appear in "running_nodes"

# Step 5: Verify quorum queue leadership has been restored
rabbitmqctl list_queues name type leader
# Quorum queues should have a leader. If leadership is unbalanced, force rebalance:
rabbitmq-queues rebalance quorum
```

### Emergency Restart (Node Unresponsive)

```bash
# Step 1: Force remove the unresponsive node from the cluster (from a healthy node)
rabbitmqctl forget_cluster_node rabbit@<unresponsive-hostname>

# Step 2: Kill the RabbitMQ and Erlang processes on the unresponsive node (OS-level)
systemctl stop rabbitmq-server
# If systemctl doesn't work:
pkill -f rabbitmq
pkill -f beam.smp

# Step 3: Clear the Mnesia data directory if the node was in a bad state
# WARNING: only do this if the node was force-removed from the cluster
# The node will rejoin as a fresh member
rm -rf /var/lib/rabbitmq/mnesia/rabbit@<hostname>/

# Step 4: Start RabbitMQ — it will rejoin the cluster as a fresh node
systemctl start rabbitmq-server

# Step 5: Verify cluster membership and quorum queue status
rabbitmqctl cluster_status
rabbitmqctl list_queues name type leader members
```

### Post-Restart Validation

- [ ] All nodes show in `running_nodes` in cluster status
- [ ] No disk or memory alarms active
- [ ] Queue depths match pre-restart documentation (no silent message loss)
- [ ] Consumer connections have re-established (check consumer team dashboards)
- [ ] No DLQ spike in the 10 minutes following restart (would indicate message loss or consumer reconnect issues)
- [ ] Quorum queue leadership distributed across nodes (not all leaders on one node)

## 7. Kafka Partition Leadership Issues

### Detection

```bash
# Check for under-replicated partitions (ISR < replication factor)
kafka-topics.sh \
  --bootstrap-server kafka-broker-1:9092 \
  --describe \
  --under-replicated-partitions

# Check for offline partitions (no leader elected)
kafka-topics.sh \
  --bootstrap-server kafka-broker-1:9092 \
  --describe \
  --unavailable-partitions

# Check broker-level metrics for leader election rate
# In JMX: kafka.controller:type=ControllerStats,name=LeaderElectionRateAndTimeMs
# Elevated rate (>0.1/min sustained) indicates leadership instability
```

### Cause Identification

| Symptom | Likely Cause | Investigation |
|---|---|---|
| Under-replicated partitions on one broker's topics | That broker is degraded | Check broker JVM (GC pause, heap), disk I/O, network |
| Under-replicated partitions spreading across brokers | Network partition between brokers | Check inter-broker network connectivity |
| Frequent leader elections on one topic | Topic partition hot-spot (one partition overloaded) | Check producer throughput per partition |
| Offline partitions | Broker crash or all ISR members unavailable | Check broker health on all nodes that hosted the partition |

### Resolution Steps

```bash
# Step 1: Identify the problem broker
# Under-replicated partitions will show the broker ID that is lagging
kafka-topics.sh --bootstrap-server kafka-broker-1:9092 \
  --describe --under-replicated-partitions \
  | awk '{print $8}' | sort | uniq -c | sort -rn
# High count for one broker ID = that broker is the issue

# Step 2: Check broker health (SSH to the broker node)
# JVM health
jcmd <kafka-pid> VM.flags
# GC pause time (from JVM logs or JMX)

# Disk I/O (can broker serve reads?)
iostat -x 5 3

# Step 3: Fix the underlying cause (JVM GC tuning, disk replacement, network fix)
# Do NOT force a leader election until the broker is healthy — you will just elect a new leader on the same degraded broker

# Step 4: After broker is healthy, trigger preferred replica election to rebalance leaders
kafka-preferred-replicas-election.sh \
  --bootstrap-server kafka-broker-1:9092
# Kafka 2.4+: this is automatic (auto.leader.rebalance.enable=true by default)

# Step 5: Verify all partitions have ISR = replication factor
kafka-topics.sh \
  --bootstrap-server kafka-broker-1:9092 \
  --describe \
  --under-replicated-partitions
# Output should be empty (no under-replicated partitions)
```

## 8. MuleSoft Flow Failure

### Step 1: Identify the Failed Flow

Navigate to **Anypoint Monitoring → Log Management** and search for the failing flow:

```
# Anypoint Log Search (use the Anypoint Monitoring UI or API)
# Filter by: application name, log level = ERROR, time range = last 1 hour

# If logs are shipped to Splunk:
index=mulesoft app_name=<application-name> log_level=ERROR
| timechart span=1m count by flow_name
| sort -count
```

### Step 2: Diagnose the Failure

Common MuleSoft flow failure causes and their log signatures:

| Cause | Log Signature | Resolution |
|---|---|---|
| Upstream service timeout | `org.mule.runtime.api.connection.ConnectionException: Timeout` | Increase timeout configuration or fix upstream. |
| Schema/payload mismatch | `org.mule.runtime.api.transformer.TransformerException` | Check incoming payload format against DataWeave transform expectations. |
| Connection pool exhaustion | `Could not obtain connection from pool` | Increase connection pool size or fix connection leak. |
| JVM Out of Memory | `java.lang.OutOfMemoryError: Java heap space` | Increase Mule worker heap; check for memory leak in DataWeave transforms. |
| Database connection refused | `java.sql.SQLException: Connection refused` | Check DB availability, connection string, firewall rules. |
| HTTP connector 4xx/5xx from target | `Response status code: 500` | Upstream API is failing; check upstream service health. |

### Step 3: Restart the Affected Application

**Restart only the affected application, not the entire Mule worker, unless the worker itself is unhealthy (OOM, unresponsive).**

```
# Via Anypoint Runtime Manager UI:
Applications → [application name] → Actions → Restart

# Via Anypoint Platform CLI:
anypoint-cli runtime-mgr application restart <application-name>

# Via REST API (for automation):
curl -X POST \
  "https://anypoint.mulesoft.com/cloudhub/api/v2/applications/<application-name>/status" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"status": "restart"}'
```

**When to restart the entire Mule worker (not just the application):**
- JVM OOM that caused the worker process to become unstable
- Multiple applications on the same worker are all failing
- Worker is unresponsive to the Anypoint Runtime Manager

### Step 4: Replay Failed Messages

If MuleSoft flows are consuming from a VM queue or reading from an audit/reprocessing database table:

1. Verify the root cause is fixed (upstream available, schema corrected, connection pool not exhausted).
2. Identify the failed messages: check the audit table or VM queue for messages that were not successfully processed.
3. Trigger replay from the Anypoint Monitoring UI (if the flow has a replay endpoint configured) or manually re-queue from the audit table.
4. Monitor the reprocessing: watch for error rate during replay. Stop replay if errors recur.

**Splunk: MuleSoft flow error rate during replay**
```
index=mulesoft app_name=<application-name> log_level=ERROR
| timechart span=1m count by flow_name
```

> **Incident routing implication**: the first question on any middleware incident bridge is "is it the broker or the consumer?" The answer determines which team leads. Middleware SRE leads for broker infrastructure problems. Product team leads for consumer application problems. The Middleware SRE is not responsible for debugging why a consumer throws a NullPointerException when it receives a valid message — that is application code, and the product team owns it. Enforce this boundary on the incident bridge.

> **VP ownership decision**: if your organization does not have an on-call rotation that can execute a RabbitMQ quorum queue recovery or a Kafka preferred replica election at 2am, you have an operational risk. These procedures are not self-service for product teams — they require broker expertise and administrative credentials. The investment in a Middleware SRE team (4–6 engineers with 24/7 on-call coverage for tier-1 broker infrastructure) is directly calibrated against the blast radius of a broker failure at peak traffic. At T-Mobile scale, a 15-minute broker outage during peak hours has a quantifiable revenue and customer experience impact. That impact is the business case for the Middleware SRE team.
