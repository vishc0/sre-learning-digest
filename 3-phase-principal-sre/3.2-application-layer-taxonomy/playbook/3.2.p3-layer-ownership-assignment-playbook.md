# Layer Ownership Assignment Playbook

**Subtitle**: Every component gets a layer, an owner, and an on-call rotation before it reaches production

> **Cross-references**: [Incident Management] [Change Management] [Service Level Management]

---

## 1. Purpose

This playbook is the operational instrument for assigning layer classification and team ownership to every new component before it is deployed to production. It is a mandatory gate in the deployment pipeline.

**The governance contract**: no component reaches production without:
1. A layer classification from the taxonomy
2. An assigned owner team
3. A derived SLO target
4. A configured on-call rotation with a written runbook
5. A CMDB record

**Who runs this process:**
- For Platform-owned layers (1, 2, 4, 6): Platform SRE or Middleware SRE lead
- For Domain-owned layers (3, 5, 7): Domain SRE lead or domain team lead
- Disputes: escalate to VP of Operations for resolution

**When to run:**
- Every new component (service, database, queue, cache, API, UI application)
- Every time a component changes layers (e.g., a data service gains business logic and becomes an application service)
- Quarterly: audit all components for classification drift

---

## 2. The Classification Decision Tree

Answer the following questions in order. Use the first YES answer to determine the layer.

---

**Q1: Does this component store data that persists beyond process restart?**

This includes relational databases, NoSQL document stores, object storage (S3-compatible), time-series databases, file systems, and data warehouses.

- YES → Does it provide direct data access (read/write) without business logic, or does it abstract/pool access to a persistence backend?
  - Direct persistence → **Layer 1: Persistent Storage**
  - Abstracts or pools access to a persistence backend → **Layer 2: Storage Abstraction**
- NO → Continue to Q2

---

**Q2: Does this component route or buffer messages asynchronously between producers and consumers?**

This includes message brokers (RabbitMQ, Kafka, ActiveMQ), ESB platforms (TIBCO EMS, MuleSoft ESB), event streaming platforms, and async queues.

- YES → **Layer 4: Middleware / Integration**
- NO → Continue to Q3

---

**Q3: Does this component expose data from a persistent store as an API without implementing business logic?**

Key test: if you removed this component and the consumer talked directly to the database, would any business logic be lost? If no business logic would be lost, it is a data service.

- YES → **Layer 3: Data Services**
- NO → Continue to Q4

---

**Q4: Does this component implement business logic — transformation, orchestration, decision-making, or rules?**

This includes microservices with domain logic, workflow engines, rules engines, and orchestration services.

- YES → **Layer 5: Application Services**
- NO → Continue to Q5

---

**Q5: Does this component route, authenticate, rate-limit, or terminate external traffic before it reaches internal services?**

This includes API gateways (Kong, AWS API Gateway, Apigee), reverse proxies, load balancers with routing logic, WAF, and mTLS terminators.

- YES → **Layer 6: API Gateway / Edge**
- NO → Continue to Q6

---

**Q6: Does this component render a user interface for a human user?**

This includes single-page applications, server-side rendered frontends, mobile app shells, and customer-facing portals.

- YES → **Layer 7: Presentation / UI**
- NO → **Escalate.** A component that does not fit any layer classification is either a composite (split it) or a classification gap (escalate to Platform SRE lead for taxonomy update).

---

For each component, record:

## 3. Team Assignment After Classification

Once the layer is determined, assign the owning team using this reference.

| Layer | Default Owner | Condition Override |
|---|---|---|
| Layer 1: Persistent Storage | DBRE team | None — DBRE always owns infrastructure. Schema ownership may be a shared responsibility with the domain team. |
| Layer 2: Storage Abstraction | Platform SRE (shared infrastructure) | If it is a dedicated sidecar cache for a single service, the product team owns it. Document which applies. |
| Layer 3: Data Services | Domain team for that business domain | If the data service is shared across domains, Platform SRE owns it. |
| Layer 4: Middleware / Integration | Middleware SRE | Consumer application health (consumer code, consumer deployment) is the product team's responsibility. Middleware SRE owns the broker infrastructure only. |
| Layer 5: Application Services | Product team for that business domain | Embedded SRE from Platform team provides production readiness support but does not own the on-call. |
| Layer 6: API Gateway / Edge | Platform/Network SRE | Route configuration is co-owned: Platform SRE owns enforcement, product team owns policy (what routes, what rate limits). |
| Layer 7: Presentation / UI | Product team | CDN configuration and SSR infrastructure are owned by Platform SRE. The frontend application is owned by the product team. |

**Team assignment output:**

| Component | Layer | Owner Team | SLO Owner | Incident Lead | Escalation Contact |
|---|---|---|---|---|---|
| [name] | [layer] | [team] | [team] | [team] | [name, title] |

## 4. SLO Derivation

Given the layer assignment, derive the required SLO targets using the SLO cascade reference from CH08.

**SLO derivation method:**

1. Identify all upstream dependencies for this component.
2. Find the minimum SLO among all upstream dependencies.
3. Subtract the component's own estimated failure probability (deployment failures, application bugs, OOM, etc.).
4. The result is the maximum achievable SLO for this component. The target must be at or below this ceiling.

**Reference SLO targets by layer (T-Mobile defaults):**

| Layer | Default SLO Target | Minimum Acceptable | Notes |
|---|---|---|---|
| Persistent Storage | 99.99% | 99.9% | Any lower requires executive approval and risk acceptance |
| Storage Abstraction | 99.99% | 99.9% | Depends on persistence layer SLO |
| Data Services | 99.9% | 99.5% | Cannot exceed persistence layer SLO |
| Middleware / Integration | 99.9% | 99.5% | Broker HA provides its own redundancy |
| Application Services | 99.5% | 99% | Accounts for dependency chain risk |
| API Gateway / Edge | 99.99% | 99.9% | Must be high; degraded gateway impacts all users |
| Presentation / UI | 99.5% | 99% | End-to-end user experience SLO |

**SLO derivation record:**

```
Component: [name]
Layer: [layer]
Upstream dependencies and their SLOs:
  - [dependency 1]: [SLO]
  - [dependency 2]: [SLO]
Minimum upstream SLO: [value]
Component own-failure estimate: [value]
Maximum achievable SLO: [calculation]
Proposed SLO: [value]
Error budget per month: [minutes at proposed SLO]
SLO owner: [team]
Approved by: [name, date]
```

## 5. On-Call Rotation Onboarding: Checklist

A component is not production-ready until every item on this checklist is complete. The on-call team must be able to respond to an incident for this component at 2am with no assistance.

### Runbook Requirements

- [ ] Runbook exists in the team's runbook library (Confluence, PagerDuty runbooks, or equivalent)
- [ ] Runbook covers: "how to verify the component is healthy" (green path check)
- [ ] Runbook covers: "how to restart the component safely" (restart procedure)
- [ ] Runbook covers: "how to roll back the component to the previous version" (rollback procedure)
- [ ] Runbook covers: "what the DLQ looks like and how to drain it" (if async consumer)
- [ ] Runbook covers: "when to escalate and to whom" (escalation contacts by name)
- [ ] Runbook has been reviewed by at least one person who did NOT write it

### Monitoring and Alerting Requirements

- [ ] The component appears in the monitoring dashboard (service health view)
- [ ] An availability alert is configured: fires when error rate exceeds SLO budget burn rate
- [ ] A latency alert is configured: fires when p99 latency exceeds SLO threshold
- [ ] If the component is a consumer: consumer lag alert is configured (Kafka) or queue depth alert (RabbitMQ)
- [ ] If the component uses a database: connection pool utilization alert is configured
- [ ] Alerts route to the correct on-call rotation in PagerDuty (or equivalent)
- [ ] Alert severity levels match the component's SLO tier (higher SLO = lower threshold for paging)

### On-Call Rotation Requirements

- [ ] The owning team has an active on-call rotation
- [ ] At least 3 engineers are trained to respond to incidents for this component
- [ ] Each trained engineer has been walked through the runbook
- [ ] A gameday or fire drill has been conducted for this component's most likely failure mode
- [ ] The escalation path from on-call to lead to VP is documented and tested

### Pre-Production Validation

- [ ] The component has been deployed to staging and operated for at least one week
- [ ] At least one simulated failure has been tested in staging (restart, dependency failure, high load)
- [ ] Rollback procedure has been executed at least once in staging and documented
- [ ] Monitoring alerts fired correctly during the simulated failure test

## 6. CMDB Registration

Every component must have a CMDB record before production deployment. The following fields are mandatory — the CMDB record is the source of truth for governance, incident routing, and change management.

### Required CMDB Fields

| Field | Description | Example |
|---|---|---|
| Component Name | Unique, human-readable name | `user-profile-api` |
| Layer | Layer number and name from taxonomy | `3 — Data Services` |
| Business Domain | Which business domain this serves | `Identity & Access` |
| Owner Team | Team with operational accountability | `Identity SRE` |
| SLO Target | Availability SLO percentage | `99.9%` |
| SLO Owner | Team accountable for SLO | `Identity SRE` |
| On-Call Rotation | PagerDuty schedule name or equivalent | `identity-sre-oncall` |
| Runbook Link | URL to the primary runbook | `https://wiki.example.com/runbooks/user-profile-api` |
| Upstream Dependencies | Comma-separated list of CMDB component names this depends on | `user-profile-db, redis-user-cache` |
| Downstream Consumers | Comma-separated list of components that depend on this | `order-service, auth-service` |
| Change Risk Tier | CRITICAL / HIGH / MEDIUM / LOW | `MEDIUM` |
| CAB Required | Whether changes require CAB review | `No` |
| Architecture Review Date | Date of most recent architecture review | `2026-06-01` |
| Architecture Review Link | URL to architecture review document | `https://wiki.example.com/arch-reviews/user-profile-api-v2` |
| Production Deploy Date | Date first deployed to production | `2026-06-15` |

### CMDB Completeness Gate

Before closing the production readiness ticket, verify:

- [ ] CMDB record created with all mandatory fields populated
- [ ] Upstream dependencies correctly mapped (validated against architecture diagram)
- [ ] Downstream consumers notified and their CMDB records updated to list this component as an upstream dependency
- [ ] On-call rotation verified active (test page sent and acknowledged)
- [ ] Layer assignment consistent with the architecture review

> **Incident routing implication**: the CMDB is the IC's lookup table during a SEV1. An incomplete or incorrect CMDB record means the IC is routing blind. Every field in the CMDB record is an incident management asset, not an administrative requirement. Treat incomplete CMDB records as a production risk, not a documentation backlog.

> **VP ownership decision**: make CMDB completeness a hard gate in the production deployment pipeline — no merge to the production deployment branch without a linked CMDB record. This converts governance from a manual audit (quarterly, reactive) to an automated gate (per deploy, proactive). The cost of enforcement is a CI check. The cost of non-enforcement is a SEV1 where no one knows who owns the failing component.
