# Architecture Review Playbook

**Subtitle**: Layer-taxonomy-based review for new services, integrations, and quarterly audits

> **Cross-references**: [Incident Management] [Change Management] [Service Level Management]

---

## 1. Purpose and When to Use

Use this playbook to conduct a structured architecture review using the layer taxonomy defined in [CH08 — Layer Governance and Ownership Model]. The review ensures every component is classified, owned, SLO-constrained, and change-risk-rated before it reaches production.

**Use for:**
- New service deployment (required gate before production)
- New integration or middleware topology change
- Quarterly architecture review of existing services
- Post-incident architectural audit when a failure exposed a governance gap

**Not for:**
- Routine feature deploys within an existing, classified service (use standard deployment checklist)
- Hotfixes in an active SEV1 (use incident playbook; conduct architecture review post-incident)

**Who runs the review:**
- Facilitator: Domain SRE lead or Platform SRE lead (depending on layer)
- Required attendees: Owning team's lead engineer, Platform SRE representative, Security representative (if edge or storage involved)
- Optional: DBRE (if persistence layer), Middleware SRE (if broker involved)

---

## 2. Pre-Review Inputs Required

The owning team must provide these inputs before the review begins. Incomplete submissions are returned without scheduling.

| Input | Description | Owner |
|---|---|---|
| Service name and description | What the service does, its business domain | Owning team |
| Component inventory | Every component in the proposed architecture (services, DBs, caches, queues, APIs) | Owning team |
| Architecture diagram | Component diagram with data flow arrows labeled with what flows across each connection | Owning team |
| Proposed SLO targets | Availability and latency targets per component | Owning team |
| Dependency list | All upstream and downstream dependencies, with their current SLO targets | Owning team |
| Change risk classification (draft) | Owning team's self-assessment of change risk per component | Owning team |
| On-call plan | Which team and rotation will handle incidents for each component | Owning team |

**Review is blocked until all inputs are present.** The facilitator does not accept verbal descriptions of architecture in lieu of a diagram.

## 3. Phase 1: Component Classification

**Goal**: Assign every component to exactly one layer from the taxonomy.

For each component in the component inventory:

| Question | If YES → Layer |
|---|---|
| Does it persist data beyond process restart (database, object store, file system)? | Persistent Storage (Layer 1) |
| Does it abstract access to persistent storage (cache, ORM, connection pool)? | Storage Abstraction (Layer 2) |
| Does it expose data from a store as an API without business logic? | Data Services (Layer 3) |
| Does it route or buffer messages asynchronously (queue, topic, ESB)? | Middleware / Integration (Layer 4) |
| Does it implement business logic that transforms, orchestrates, or decides? | Application Services (Layer 5) |
| Does it route, authenticate, or rate-limit external traffic? | API Gateway / Edge (Layer 6) |
| Does it render UI for a human user? | Presentation / UI (Layer 7) |

**Classification output format:**

| Component | Proposed Layer | Reviewer Classification | Match? | Notes |
|---|---|---|---|---|
| user-profile-db | Persistent Storage | Persistent Storage | Yes | |
| user-profile-api | Data Services | Data Services | Yes | |
| order-service | Application Services | Application Services | Yes | |
| api-gateway-route-config | API Gateway / Edge | API Gateway / Edge | Yes | |

**Misclassification flags to watch for:**
- A component described as "data service" but containing business logic → reclassify to Application Services
- A component described as "middleware" that is actually a synchronous API adapter → reclassify to Data Services or Application Services
- A component that serves both as a cache and business logic orchestrator → split into two components; mixed-layer components create governance gaps
- Any component that cannot be classified → escalate to Platform SRE lead for decision before proceeding

## 4. Phase 2: Ownership Assignment

**Goal**: Every component has exactly one named team with on-call accountability.

Apply the ownership matrix from CH08:

| Layer | Default Owner | Exception Conditions |
|---|---|---|
| Persistent Storage | DBRE | None — DBRE always owns persistence infrastructure |
| Storage Abstraction | Platform SRE (shared) or App team (dedicated sidecar) | If the cache is deployed as a dedicated sidecar for one service, the app team owns it |
| Data Services | Domain team | — |
| Middleware / Integration | Middleware SRE | Consumer application health is owned by the product team using the broker |
| Application Services | Product team | — |
| API Gateway / Edge | Platform/Network SRE | Route configuration is a shared responsibility with the product team |
| Presentation / UI | Product team | CDN and SSR infrastructure owned by Platform SRE |

**Ownership assignment output:**

| Component | Layer | Owner Team | SLO Owner | On-Call Rotation | Gap? |
|---|---|---|---|---|---|
| user-profile-db | Persistent Storage | DBRE | DBRE | DBRE on-call | No |
| user-profile-redis | Storage Abstraction | Platform SRE | App team | Shared (platform + app) | No |
| user-profile-api | Data Services | Identity domain team | Identity domain team | Identity on-call | No |

**Gap flags:**
- Any component with no named on-call rotation → BLOCKED. Must be resolved before production approval.
- Any component where ownership is disputed between two teams → escalate to VP for resolution. Do not proceed with disputed ownership.
- Any component owned by a team that does not have an existing on-call rotation → the rotation must be established and tested before production.

## 5. Phase 3: SLO Consistency Check

**Goal**: Validate that proposed SLO targets are mathematically consistent with the dependency chain.

**Validation rule (from CH08 SLO Cascade):**
```
SLO(service) ≤ MIN(SLO(all upstream dependencies)) × (1 - service_own_failure_probability)
```

**SLO consistency check table:**

| Component | Proposed SLO | Upstream Dependencies | Upstream SLOs | Math-Valid? | Corrected SLO |
|---|---|---|---|---|---|
| order-service | 99.9% | user-profile-api (99.9%), payment-api (99.9%) | Both 99.9% | Yes — after accounting for own failures | 99.9% |
| payment-api | 99.99% | payment-db (99.99%) | 99.99% | No — payment-api has own failure modes | 99.9% |

**Common SLO violations to flag:**
- A data service with a higher SLO than its backing database (mathematically impossible)
- An application service with a 99.99% SLO backed by multiple 99.9% upstream dependencies (the joint probability is lower than any individual upstream)
- A UI component with a higher SLO than the API services it calls

**Output**: approved SLO targets per component, with the derivation rationale documented. Any SLO that cannot be validated against the cascade is rejected and must be renegotiated.

> **[Service Level Management]**: SLO targets set during architecture review are the inputs to the error budget policy. A 99.9% SLO means 43 minutes of downtime allowed per month. The team must have deployment procedures, runbooks, and on-call practices consistent with that budget.

## 6. Phase 4: Dependency Mapping

**Goal**: Identify all upstream/downstream dependencies by layer and flag cross-layer dependency risks.

**Dependency map format:**

| Component | Layer | Upstream Dependencies | Upstream Layer | Dependency Risk |
|---|---|---|---|---|
| order-service | Application Services | payment-api | Data Services | MEDIUM — synchronous call; payment-api timeout = order-service timeout |
| order-service | Application Services | order-events (Kafka topic) | Middleware | LOW — async; broker failure degrades async path only |
| order-service | Application Services | order-db | Persistent Storage | HIGH — synchronous; DB failure = service failure |

**Cross-layer dependency risk classification:**

| Dependency Type | Risk Level | Required Mitigation |
|---|---|---|
| Synchronous call to Persistent Storage | HIGH | Connection pool sizing, query timeout, circuit breaker |
| Synchronous call to another Application Service | MEDIUM | Circuit breaker, timeout, fallback behavior defined |
| Async publish to Middleware | LOW | DLQ defined, producer error handling, retry policy |
| Async consume from Middleware | MEDIUM | Consumer lag alerting, DLQ handling runbook |
| Synchronous call to External API (third-party) | HIGH | Circuit breaker mandatory, fallback, SLA contractually defined |

**Flag for review:**
- Any synchronous dependency on an external third-party API without a circuit breaker → BLOCKED
- Any component with more than 5 synchronous upstream dependencies → flag for architectural simplification discussion
- Any missing DLQ definition for an async consumer → BLOCKED

## 7. Phase 5: Change Risk Assessment

**Goal**: Assign a change risk tier to every component using the change risk reference from CH08.

Apply the CH08 Layer-Based Change Risk Classification table. For each component, identify the highest-risk change type it will undergo and classify accordingly.

**Change risk summary:**

| Component | Layer | Highest-Risk Change Type | Risk Tier | CAB Required? | Rollback Plan |
|---|---|---|---|---|---|
| user-profile-db | Persistent Storage | Schema migration | CRITICAL | Yes | Manual SQL reverse migration, tested in staging |
| user-profile-api | Data Services | Breaking API change | HIGH | No | Maintain v1 endpoint through deprecation cycle |
| order-service | Application Services | Service deploy | LOW | No | Blue-green rollback |
| api-gateway-route | API Gateway / Edge | Route change | MEDIUM | No | Config revert, same day |

**Approval requirements by risk tier:**

| Risk Tier | CAB Required | Rollback Validation | Lead Time |
|---|---|---|---|
| CRITICAL | Yes | Rollback tested in staging and documented | 2+ weeks |
| HIGH | Yes | Rollback plan documented and reviewed | 1 week |
| MEDIUM | No | Rollback procedure documented | Standard |
| LOW | No | Container image rollback is sufficient | Standard |

> **[Change Management]**: every component classified as CRITICAL or HIGH must have its CAB submission prepared as part of the architecture review, not as an afterthought before the first production deploy.

## 8. Phase 6: Single Point of Failure Review

**Goal**: Identify every component with no high availability, no fallback, and no circuit breaker.

**SPOF checklist per component:**

| Component | HA Configured? | Circuit Breaker? | Fallback Behavior Defined? | DLQ (if async)? | SPOF Risk |
|---|---|---|---|---|---|
| user-profile-db | Yes (Aurora Multi-AZ) | N/A | N/A | N/A | Low |
| user-profile-redis | Yes (Redis cluster) | N/A | Service degrades gracefully without cache | N/A | Low |
| payment-api (external) | No (third-party) | No | No | N/A | CRITICAL |
| order-events (Kafka) | Yes (3-node cluster) | N/A | N/A | Yes (configured) | Low |

**SPOF flags:**
- Any external dependency without a circuit breaker → BLOCKED. Implement circuit breaker before production approval.
- Any database without Multi-AZ or equivalent HA → HIGH risk flag. Must be resolved or accepted with documented risk.
- Any async consumer without a DLQ → BLOCKED. DLQ must be configured before production.
- Any component where failure causes complete service unavailability with no degraded mode → flag for architectural discussion.

---

## 9. Review Output

The review produces one of three outcomes:

### Approved
All phases complete, no BLOCKED items, all HIGH and CRITICAL items addressed. Service may proceed to production following standard deployment process.

### Conditional Approval
Review is complete but specific items must be resolved before production. Conditionally approved services may proceed to staging but not production.

**Conditional approval output format:**
```
Conditional Approval — [Service Name]
Date: [YYYY-MM-DD]
Reviewer: [Name]

MUST resolve before production:
- [ ] [Item] — Owner: [Team] — Due: [Date]
- [ ] [Item] — Owner: [Team] — Due: [Date]

SHOULD address within 30 days:
- [ ] [Item] — Owner: [Team]

Approved by: [Name, Title]
```

### Rejected
One or more BLOCKED items that cannot be conditionally accepted. Service is returned to the owning team for redesign.

**Rejection output format:**
```
Rejected — [Service Name]
Date: [YYYY-MM-DD]
Reviewer: [Name]

Rejection reasons:
- [BLOCKED item] — required change: [description]
- [BLOCKED item] — required change: [description]

Resubmit after: [description of required changes]
```

## 10. Governance: Sign-Off and CAB Submission

**Who signs off on the architecture review:**

| Risk Profile | Sign-Off Authority |
|---|---|
| All LOW/MEDIUM risk, no HIGH | Domain SRE lead |
| Any HIGH risk component | Platform SRE lead + Domain SRE lead |
| Any CRITICAL risk component | Director of Engineering or VP of Operations |
| External-facing service (Layer 6 or 7) | Security sign-off required in addition to SRE |

**CAB submission requirements for HIGH and CRITICAL components:**
- Architecture review document (this output) included in CAB ticket
- Rollback plan documented with step-by-step instructions
- Rollback tested in non-production and test results attached
- Monitoring and alerting configured for the new component (verified in staging)
- Runbook written and linked in the CMDB record

> **VP ownership decision**: architecture review is a governance gate, not a rubber stamp. The VP's responsibility is to ensure the gate is enforced — that no service reaches production without a completed review, and that conditional approvals have a tracked resolution date. A backlog of unresolved conditional approvals is a governance liability that accumulates until a production incident makes it visible.
