# Incident Layer Routing Playbook

**Subtitle**: Identify the layer, route to the right team, within 5 minutes

> **Cross-references**: [Incident Management] [Change Management] [Service Level Management]

---

## 1. Purpose

Use this playbook on any incident where the root-cause layer is unknown. The goal is to eliminate the two most expensive incident behaviors: routing to the wrong team and debating ownership on the bridge while the incident is active.

**When to activate**: Any SEV1 or SEV2 where the symptom is known but the owning layer is not. For SEV3 and below, use the symptom-to-layer table in section 4 and route asynchronously.

**Time target**: Layer identified and lead team paged within 5 minutes of incident declaration.

---

## 2. The 5-Minute Layer Identification Protocol

Execute these steps in order. Stop at the first step where you find an anomaly — that is your layer.

---

### Step 1: Is the edge healthy?

Check the API gateway layer first. If the edge is degraded, all downstream symptoms are secondary.

**What to check:**
- API gateway error rate (target: <0.1% baseline)
- Request rate at the gateway (look for spike or drop — both are signals)
- Gateway latency p99
- Active TLS cert validity
- WAF block rate (spike may indicate attack or misconfigured rule)

**Tools:**
```
# Splunk: Edge error rate last 15 minutes
index=api_gateway sourcetype=nginx_access
| timechart span=1m count(eval(status>=500)) as errors, count as total
| eval error_pct = errors/total*100
| where error_pct > 1

# Splunk: WAF block spike
index=waf action=block | timechart span=1m count
```

**Decision**: If gateway error rate is elevated or request rate has dropped to zero → **Platform/Network SRE leads**. Do not proceed to app layer until edge is cleared.

---

### Step 2: Is the application service healthy?

If the edge is healthy, move to the application service layer.

**What to check:**
- Error rate by service in APM (Dynatrace, Datadog, or New Relic — whichever is in use)
- Latency p50/p95/p99 by service
- Which service shows the first anomaly in the distributed trace?
- Is the anomaly on one service or multiple?

**Tools:**
```
# Splunk: Service error rate by service name
index=app_logs log_level=ERROR
| timechart span=1m count by service_name
| sort -count

# Splunk: 5xx errors with service attribution
index=app_logs status>=500
| stats count by service_name, endpoint | sort -count | head 20
```

**Decision**: Identify which service shows the first anomalous span in the distributed trace. That service's **product team leads**. If multiple services are degraded simultaneously, check middleware (Step 3) — a common upstream dependency may be failing.

---

### Step 3: Is the message broker healthy?

If application services are showing consumer-related errors or if the incident involves message processing, check the middleware layer.

**What to check:**
- **Kafka**: Consumer lag by consumer group — is it growing or stable?
- **RabbitMQ**: Queue depth by queue, consumer connection count, memory/disk alarms
- **MuleSoft**: Flow error rate in Anypoint Monitoring

**Kafka diagnostic commands:**
```bash
# List all consumer groups
kafka-consumer-groups.sh --bootstrap-server kafka-broker:9092 --list

# Check lag for a specific consumer group
kafka-consumer-groups.sh --bootstrap-server kafka-broker:9092 \
  --describe --group <consumer-group-name>
# Columns: TOPIC, PARTITION, CURRENT-OFFSET, LOG-END-OFFSET, LAG, CONSUMER-ID

# Check lag for all groups (pipe to sort by lag)
kafka-consumer-groups.sh --bootstrap-server kafka-broker:9092 \
  --describe --all-groups 2>/dev/null \
  | awk 'NR>1 && $6 != "-" {print $1, $2, $6}' \
  | sort -k3 -rn | head 20
```

**RabbitMQ diagnostic commands:**
```bash
# Cluster health
rabbitmqctl cluster_status

# Queue list with depth and consumer count
rabbitmqctl list_queues name messages consumers memory

# Check for memory/disk alarms
rabbitmqctl status | grep -A5 alarms

# Check unacknowledged messages per queue
rabbitmqctl list_queues name messages_ready messages_unacknowledged
```

**Decision**: If broker is showing alarms or partitions → **Middleware SRE leads**. If broker is healthy but consumer lag is growing → consumer application issue → **Product team leads** for that consumer.

---

### Step 4: Is the data service healthy?

If application services are calling data services and showing latency, check the data service layer.

**What to check:**
- Data service API error rate and latency in APM
- Response time at the data service boundary (not the DB query — the API call)
- Is the data service returning errors or timeouts?

**Tools:**
```
# Splunk: Data service latency
index=app_logs service_name=<data-service-name>
| timechart span=1m avg(response_time_ms) as avg_latency p99(response_time_ms) as p99_latency

# Splunk: Data service 5xx
index=app_logs service_name=<data-service-name> status>=500
| timechart span=1m count
```

**Decision**: If data service latency is elevated → **Domain team leads** with DBRE as support for query-level investigation.

---

### Step 5: Is storage healthy?

If data services are slow and the trace shows DB calls as the slow span, check the persistence layer.

**What to check:**
- DB query execution time (slow query log)
- Active connection count vs. connection pool size
- Replication lag (for read replicas)
- Lock contention or deadlock events

**DB diagnostic queries (MySQL/Aurora):**
```sql
-- Active connections and states
SELECT state, count(*) FROM information_schema.PROCESSLIST GROUP BY state;

-- Slow queries currently running
SELECT id, time, state, info FROM information_schema.PROCESSLIST
WHERE time > 5 ORDER BY time DESC;

-- Replication lag (run on replica)
SHOW SLAVE STATUS\G
-- Look at: Seconds_Behind_Master

-- Lock waits
SELECT * FROM information_schema.INNODB_LOCK_WAITS;
```

```
# Splunk: DB connection pool exhaustion signal
index=app_logs "connection timeout" OR "pool exhausted"
| timechart span=1m count by service_name
```

**Decision**: If DB query execution time is elevated in the slow query log → **DBRE leads**. If connection pool is exhausted but DB queries are fast → **App team leads** (pool sizing or leak is an application problem).

## 3. Layer-to-Team Routing Table

| Layer | Lead Team | Supporting Teams | First Diagnostic Command |
|---|---|---|---|
| Persistent Storage | DBRE on-call | App team (connection pool check), Platform SRE (network) | `SHOW SLAVE STATUS\G` or `rabbitmqctl cluster_status` |
| Storage Abstraction | Platform SRE | App team (cache usage pattern) | Redis: `redis-cli info stats` / connection pool metrics in APM |
| Data Services | Domain team on-call | DBRE (if query-level issue), App team (if consumer issue) | APM trace on data service endpoint latency |
| Middleware / Integration | Middleware SRE on-call | App team (consumer health), Platform SRE (network) | `kafka-consumer-groups.sh --describe` or `rabbitmqctl list_queues` |
| Application Services | Product team on-call | Embedded SRE, DBRE (if DB spans slow) | APM distributed trace, service error rate dashboard |
| API Gateway / Edge | Platform/Network SRE | Security (if WAF involved), App team (if route-specific) | Gateway error rate in Splunk, WAF block log |
| Presentation / UI | Product/UI team on-call | Platform SRE (CDN check) | Browser error console, RUM error rate dashboard |

## 4. Symptom-to-Layer Decision Table

| Symptom | Likely Layer | First Check | Common Misrouting |
|---|---|---|---|
| All users getting HTTP 500 | Edge or Application (upstream) | Gateway error rate first | Assuming it's "the app" without checking edge |
| One API endpoint slow, others fast | Application Services | Distributed trace for that endpoint | Assuming it's the DB before checking the trace |
| Messages not appearing in consumer | Middleware or Consumer | Kafka lag or RabbitMQ queue depth | Assuming the producer is broken |
| DLQ filling up | Middleware (consumer fault) | Consumer application logs for rejection reason | Assuming the broker is broken |
| Database timeouts in app logs | Application (pool) or Storage (query) | Compare query execution time vs. pool wait time | Automatically paging DBRE before checking pool |
| UI page blank but API returns 200 | Presentation / UI | Browser console errors, CDN cache, frontend JS errors | Escalating to backend teams |
| Replication lag growing | Persistent Storage | DB master load, network throughput | Blaming the replica |
| Cert error in clients | Edge | TLS cert expiry, gateway cert config | Assuming application code issue |
| Elevated 429 errors | Edge (rate limiting) | Rate limit config, WAF rules | Assuming the app is broken |
| Consumer group offset not advancing | Middleware or Consumer | Consumer logs first, then broker health | Always blaming the broker |
| MuleSoft flow not executing | Middleware / Integration | Anypoint Monitoring flow error log | Assuming upstream API is broken |

## 5. Escalation Path per Layer

If the first team cannot resolve or narrow the issue within 30 minutes, escalate:

| Layer | Tier 1 (0–30 min) | Tier 2 Escalation (30 min) | VP Escalation (60 min, SEV1 only) |
|---|---|---|---|
| Persistent Storage | DBRE on-call | DBRE lead + Platform SRE for infrastructure assessment | Director of Platform/Engineering |
| Storage Abstraction | Platform SRE on-call | Platform SRE lead | Director of Platform/Engineering |
| Data Services | Domain team on-call | Domain SRE lead + DBRE | Domain Director |
| Middleware / Integration | Middleware SRE on-call | Middleware SRE lead + Platform SRE | Director of Platform/Engineering |
| Application Services | Product team on-call | Product SRE lead + Middleware SRE (if async path) | Product Director |
| API Gateway / Edge | Platform/Network SRE on-call | Platform SRE lead + Security (if WAF) | Director of Platform/Engineering |
| Presentation / UI | UI/Product team on-call | Product lead | Product Director |

## 6. Common Misrouting Traps

**Trap 1: Application timeout → immediately page DBRE**
The timeout may be connection pool exhaustion (app problem). Verify the DB slow query log shows slow queries before paging DBRE. Required evidence: `avg_query_execution_time > threshold` in slow query log.

**Trap 2: Consumer lag → immediately page Middleware SRE**
Consumer lag can be caused by a slow or crashing consumer application. Check consumer logs first. If the consumer is throwing exceptions and restarting, it is a product team problem. Middleware SRE is responsible for broker health, not consumer application health.

**Trap 3: 5xx errors → immediately page the app team**
5xx errors at the edge may be gateway timeouts, WAF blocks, or upstream routing failures — none of which the app team controls. Check the gateway error log before routing to the app team.

**Trap 4: "It started after the deploy" → blame the deploy**
Correlation is not causation. A deploy may have coincided with an unrelated infrastructure failure. Check whether the failure signature matches the deploy's scope (e.g., a deploy to one service should not cause failures in unrelated services).

**Trap 5: Escalate to multiple teams simultaneously "just to be safe"**
This creates bridge noise and diffuses accountability. Route to one lead team. Let them pull in support as needed. The IC controls who is on the bridge.

---

## 7. Transferring Incident Lead Between Teams Mid-Incident

When investigation reveals the issue is in a different layer than initially suspected, transfer the lead cleanly.

**Transfer protocol:**
1. Current lead team states: "Evidence shows the root cause is in [new layer]. Transferring lead to [new team]."
2. Current lead team briefs the incoming team: timeline, what was tested, what was ruled out, current hypothesis.
3. Incoming team acknowledges: "Confirmed, [name] is now IC for this incident."
4. IC updates the incident ticket with the layer reclassification and reason.
5. Current lead team stays on bridge in support role unless the IC releases them.

**What not to do**: silently disengage, leave the bridge without acknowledgment, or allow the "who is leading" question to be ambiguous mid-incident. The IC always knows exactly one team is leading at any moment.

---

> **Incident routing implication**: this playbook is the IC's operational instrument, not a reference document. It must be accessible in under 10 seconds from any device during an incident — bookmark it in the incident management platform runbook library. Practice the 5-minute protocol in quarterly gameday exercises so the decision sequence is muscle memory, not a cognitive load during a high-stress SEV1.

> **VP ownership decision**: if routing disputes consume more than 5 minutes on any incident bridge, the governance model has a gap. Track "time to layer identification" as an incident metric in your postmortem template. A rising trend indicates either the ownership matrix is stale or the diagnostic tooling is insufficient for fast layer identification. Both are VP-level investment decisions.
