# Chapter 07 — Presentation and UI Layer

**Layer position**: outermost user-facing layer. Runs in the user's browser or on their mobile device. Consumes the Edge/API Gateway layer. Not a server-side component — it executes on the client.

Cross-references: [Incident Management] [Change Management] [Service Level Management] [Problem Management]

---

## What Belongs Here

| Belongs Here | Does NOT Belong |
|---|---|
| Web frontends: React, Angular, Vue.js, Next.js (SSR/SSG), Remix | API gateways fronting multiple backends (→ Edge layer) |
| Mobile apps: iOS (Swift/SwiftUI), Android (Kotlin/Jetpack Compose), React Native | Mobile push notification services: APNs, FCM (→ Middleware/Integration) |
| BFF (Backend for Frontend) services: Node.js or Go aggregation services | CMS backend API (→ Data Services or Application Services) |
| Static asset CDN: CloudFront, Akamai, Cloudflare Pages | Authentication provider (→ Application Services or Edge layer) |
| Server-Side Rendering infrastructure: Next.js servers, edge rendering | Kafka producers/consumers (→ Middleware/Integration) |

**BFF clarification**: a BFF is a thin server-side service owned by the UI team that aggregates API responses for a specific client type. It belongs here because it exists to serve the UI layer's specific data shape needs — it has no independent business logic. It is not shared across multiple consumer types; if it becomes shared, it has migrated to Application Services.

---

## Why the VP Cares About the UI Layer

**The availability measurement problem**: a backend with 99.99% availability that serves a broken UI is a 0% experience for the user who encounters it. Backend SLOs measured at the API layer systematically overstate system reliability. User-perceived availability is what the business sells; server-side availability is an internal metric.

**The "lights are green but users are complaining" problem**: server metrics show healthy APIs, no alerts firing — but users are reporting errors and failures. Almost always a UI layer issue:
- Broken JavaScript bundle after a release
- Failed CDN asset delivery (JS loads but CSS does not)
- Client-side rendering error not visible in server logs
- Feature flag misconfiguration rendering a broken state
- Third-party script (analytics, chat widget) blocking page render

Without UI layer observability, the operations team is blind to this entire failure class.

**The mobile crash problem**: a mobile app crash rate of >1% is a significant user experience failure that does not appear in any server-side metric. The server sees the session end and attributes it to the client gracefully closing the connection. Only mobile crash analytics surfaces this.

> **VP ownership decision**: UI layer SLOs — Core Web Vitals, crash-free rate, error-free session rate — must be owned by the product team and reviewed in the same reliability forum as backend SLOs. A reliability review that only covers backend service metrics is incomplete. Require the UI team to present Core Web Vitals trends alongside backend error rates in every reliability review meeting.

## Core Web Vitals as SLO: The User Experience Contract

[Service Level Management]

| Metric | What It Measures | Good | Needs Improvement | Poor | Monitoring Tool |
|---|---|---|---|---|---|
| LCP (Largest Contentful Paint) | When main content is visible to the user | <2.5s | 2.5–4.0s | >4.0s | Real User Monitoring (RUM) |
| INP (Interaction to Next Paint) | How quickly UI responds to user input | <200ms | 200–500ms | >500ms | RUM |
| CLS (Cumulative Layout Shift) | Visual stability — elements jumping while user reads or clicks | <0.1 | 0.1–0.25 | >0.25 | RUM |
| TTFB (Time to First Byte) | Server response time to browser's first request | <800ms | 800ms–1.8s | >1.8s | Synthetic monitoring |
| FCP (First Contentful Paint) | When first content appears | <1.8s | 1.8–3.0s | >3.0s | RUM |

**LCP is the primary user experience SLO for content pages**. Target LCP <2.5s at P75 — the 75th percentile of real users. Not the average. Averaging hides the tail experience for a quarter of your users.

**INP replaced FID (First Input Delay)** in Google's Core Web Vitals in March 2024. If your team is still measuring FID, their SLO tracking is stale. Test INP.

**CLS operational impact**: a button that moves just before the user clicks it is a CLS event. The user clicks the wrong element. This causes form submission errors, accidental purchases, account deletions. CLS is a reliability metric with real business consequence, not a cosmetic concern.

**RUM vs. synthetic monitoring**: Core Web Vitals SLOs must be measured via Real User Monitoring. Synthetic tests from a datacenter miss CDN edge variance, real-world device performance (low-end Android phones), and actual network conditions. Synthetic tests complement RUM for catching regressions in CI/CD; they do not replace RUM for SLO measurement.

## Frontend Performance: Operational Concerns

| Concern | What to Measure | Target | Detection Tool |
|---|---|---|---|
| JavaScript bundle size | Initial JS bundle (compressed) | <200KB | webpack-bundle-analyzer, Lighthouse CI |
| Time to Interactive (TTI) | When page is fully interactive, not just visible | <3.5s | Lighthouse, RUM |
| API call waterfall | Sequential API calls that could run in parallel | Zero unnecessary serialization | Browser DevTools waterfall, BFF adoption |
| Image optimization | Next-gen formats (WebP, AVIF), responsive srcset | No uncompressed images >100KB | Lighthouse, PageSpeed Insights |
| CDN cache hit rate | Static assets served from edge vs. origin | >99% cache hit rate | CDN analytics (CloudFront, Cloudflare) |

**CDN cache hit rate** is the most commonly ignored operational metric in the UI layer. Static assets — JS, CSS, images — must be served from CDN with >99% cache hit rate. A cache miss sends the request to origin: higher latency for the user, increased origin load, potential origin congestion. Monitor cache hit rate per asset type, not just aggregate.

**The TTI gap problem**: a page with fast LCP but slow TTI shows users content they cannot interact with. Users see a form, try to type, and nothing happens for 3 seconds. This is a failure mode that LCP alone does not capture. Measure both.

## BFF Pattern: Governance and Operational Model

### Why BFF Exists

A mobile app needs 5 fields from 3 different API services. Without BFF:
```
Mobile App → [OrderService: 200ms] 
           → [CustomerService: 150ms]    ← 3 serial or parallel calls over mobile network
           → [InventoryService: 180ms]
```
Over a mobile network: high per-call latency, battery consumption per connection, failure if any single call fails.

With BFF:
```
Mobile App → [Mobile BFF: 50ms] → [OrderService: 200ms]     ← BFF makes 3 calls
                                 → [CustomerService: 150ms]   ← over fast datacenter network
                                 → [InventoryService: 180ms]  ← returns 1 composed response
```

One mobile network call; all aggregation in the datacenter.

### BFF Ownership and Governance

| Question | Answer |
|---|---|
| Who owns the BFF? | The UI/product team — it is a UI concern, not a backend concern |
| Can multiple clients share a BFF? | No. Web BFF and Mobile BFF are separate. Sharing creates a bottleneck that optimizes for neither. |
| Where does the BFF deploy? | Same Kubernetes cluster as application services; platform SRE owns the deployment infrastructure |
| What is the BFF's SLO? | The aggregate of all its upstream API dependencies — it cannot exceed what it depends on |

**BFF partial response design**: the BFF must return what is available and omit what is unavailable, rather than failing the entire response when one upstream is degraded. If InventoryService is down, the BFF returns order and customer data with an `inventory_unavailable: true` flag. The UI renders with available data and shows a placeholder for inventory. This is a design requirement — partial response capability must be built in, not retrofitted during an incident.

## Frontend Deployment Governance

### Why Frontend Deployments Are Operationally Different

| Property | Frontend | Backend Service |
|---|---|---|
| State on instances | Stateless — assets on CDN | May have local state, connection pools |
| Rollback mechanism | Re-deploy previous build + CDN cache invalidation | Roll back deployment, drain connections |
| Rollback time | <2 minutes | 5–15 minutes depending on service |
| Blast radius | ALL users simultaneously — 100% | Only users of that service's traffic |
| Progressive rollout difficulty | Hard without feature flags or edge personalization | Standard canary with traffic splitting |

**The 100% blast radius problem**: a bad backend deploy affects one service. A bad frontend deploy affects every user who loads any page. Despite the fast rollback available to frontend teams, the initial blast radius is maximum. This contradiction — fast recovery, maximum initial impact — makes frontend deployment governance critical.

> **VP ownership decision**: every frontend production deployment must include: (1) Automated Lighthouse CI / Core Web Vitals regression check as a required CI gate — not optional. (2) Post-deploy smoke test: synthetic test covering the top 5 user journeys within 3 minutes of deployment. (3) Documented and rehearsed rollback procedure: CDN cache invalidation and re-deploy of previous bundle in <5 minutes. (4) Change window requirement for major releases: complete redesigns, navigation restructuring, or changes to shared UI components require a scheduled change window, not an ad-hoc deploy.

[Change Management]: frontend deployments with Core Web Vitals regressions detected in CI must be blocked automatically — not flagged for human review. The regression check is a binary gate.

## Client-Side Error Monitoring: The VP Dashboard

| Tool | What It Captures | Key Metrics | Alert Trigger |
|---|---|---|---|
| Sentry | JavaScript exceptions, unhandled promise rejections, performance spans | Error rate by release, impacted users, stack trace | Error rate >1% new errors introduced by release |
| Datadog RUM | Real user sessions, page views, Core Web Vitals, resource timings, rage clicks | Session error rate, LCP P75, rage click rate | LCP regression >500ms from prior 7-day baseline |
| New Relic Browser | Page load performance, AJAX errors, JavaScript errors | Apdex score, error count, page render time by browser | Apdex score <0.85 |
| Firebase Crashlytics | Mobile crash reports, ANRs (Android Not Responding), OOM kills | Crash-free session rate, crash rate by OS version and device | Crash-free session rate <99% |
| Google Analytics 4 | User behavior, funnel completion, bounce rate | Funnel completion rate, bounce rate by page | Funnel drop >20% from 7-day baseline |

**VP metrics for the reliability review**:
- Mobile: crash-free session rate target >99.5%
- Web: error-free session rate target >99%
- Core Web Vitals: LCP <2.5s at P75, INP <200ms at P75, CLS <0.1

**Alert on new error rate spike, not just absolute count**: a release that introduces 100 distinct new error types is more operationally concerning than an existing error type occurring at the same historical rate. Error monitoring tools that provide "new in this release" segmentation are required — not optional.

[Problem Management]: recurring UI errors that correlate with specific releases across 3 or more consecutive releases are Problem Management candidates. Pattern detection rule: same error fingerprint appearing in 3+ consecutive releases without resolution = open a Problem record and drive root cause investigation. Do not treat it as a series of isolated incidents.

## Progressive Enhancement: Reducing UI Blast Radius

### Graceful Degradation Patterns

| Backend Failure | Bad UI Behavior | Required UI Behavior |
|---|---|---|
| RecommendationService down | Blank page or 500 error | Show "No recommendations available" message |
| SearchService slow (>3s) | Spinner indefinitely | Show cached results with staleness indicator |
| PaymentService degraded | Error message, dead end | Show estimated completion time, offer callback |
| InventoryService down | "Unknown" stock count | Show "Availability unavailable — check back soon" |
| AuthService slow | Login hangs | Show progress indicator; timeout with clear error message |

**Every user-facing feature that depends on a tier-1 backend service must have a defined degraded mode**. "What does the UI show when [service] is down?" is a required question in architecture review. A feature without a defined degraded mode is not production-ready.

### Feature Flags for Operational Control

Feature flags (LaunchDarkly, Unleash) give the VP a non-deployment lever: disable a UI feature that depends on a degraded backend without triggering a deployment. During an incident, toggling off a problematic feature takes seconds. Waiting for a deployment takes minutes to hours.

Feature flag requirement: every UI feature that makes calls to a tier-2 or lower availability backend must have a kill switch implemented as a feature flag. The feature flag is tested in non-production before the feature ships.

### Offline Capability (Progressive Web Apps)

For mobile-first products, service workers cache critical data locally. Users can view cached data when the network fails. Write operations are queued and synced on reconnect.

This is not universally required — it is a design choice driven by the target user's network reliability. For T-Mobile customer-facing apps where users may be in areas with poor signal, offline capability for read operations (viewing account balance, data usage) is operationally justified.

---

> **Incident routing implication**: UI layer incidents are owned by the UI/product team. Platform SRE supports CDN and hosting infrastructure issues. Routing rule for first responders: if Core Web Vitals degrade without any backend change, it is a UI team investigation — JavaScript bundle regression, third-party script interference, CDN configuration change, browser compatibility issue. If CDN origin health degrades (origin pull failures, elevated TTFB), Platform SRE leads. If backend APIs are degraded and latency is propagating into TTI and LCP, backend teams lead their service recovery and the UI team monitors and communicates user experience impact in parallel. Do not wait for backend recovery to begin UI mitigation — partial responses, feature flag toggles, and fallback UI are immediate UI team actions.

> **VP ownership decision**: UI layer SLOs — Core Web Vitals, crash-free session rate, error-free session rate — are owned by the product team with the same accountability as backend service SLOs. If your reliability review only covers backend services, you are measuring the wrong thing. Mandate that the UI team presents Core Web Vitals P75 trends, mobile crash-free rate, and client-side error-free session rate alongside backend error rates in every reliability review. Backend green + UI red = the business is failing its users.
