# Layer Ownership Matrix — Registry Template

## Instructions
One row per production component. "Component" means any independently deployable unit tracked in your CMDB (service, broker, database, gateway, cache cluster, etc.). This is NOT a logical grouping — it is a registry of individual CIs.

Rules:
- Every production component must have exactly one row
- SLO Owner and Platform Owner may be different teams — this is expected and must be explicit
- No blank cells. If you do not know, write "UNKNOWN" and flag it as a gap
- Dependencies and Consumers must reference components by their CMDB CI name, not team names
- Review this matrix quarterly. Use the date in Last Updated to detect staleness

## Governance Note

This matrix is reviewed quarterly. Ownership reviews are conducted by the SRE Manager with domain tech leads present. Decisions made in review are recorded in the Decisions column of the quarterly governance log.

**Any component without a row in this matrix is a governance gap.** Gaps discovered during incidents must be escalated to the SRE Manager within 24 hours.

**Any incident involving a component not in this matrix must result in a new row being added before the PIR is closed.** The PIR cannot be marked Complete until the row is present and the SLO Owner has signed off.

Ownership disputes (e.g., two teams both claim or both disclaim a component) are escalated to the Engineering Director owning both teams. The default rule: whoever is on-call for the component owns its SLO, regardless of code ownership.
