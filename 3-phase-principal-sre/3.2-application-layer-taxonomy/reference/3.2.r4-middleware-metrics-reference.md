# Middleware Metrics Reference — RabbitMQ · Kafka · MuleSoft

The definitive operational reference. Carry into any incident bridge or capacity review.

---

## 1. RabbitMQ Metrics

| Metric | What It Measures | Normal Range | Warning | Critical | Splunk Query Pattern | Remediation |
|---|---|---|---|---|---|---|
| `queue_messages` | Total messages in queue (ready + unacked) | < 1,000 (healthy consumers) | > 10,000 | > 100,000 | `sourcetype=rabbitmq_exporter queue_messages queue_name=* \| timechart avg(value) by queue_name` | Check consumer count and consumer lag rate; scale consumers |
| `queue_messages_ready` | Messages waiting to be delivered (consumer-side lag) | Near 0 in steady state | > 5,000 | > 50,000 | `sourcetype=rabbitmq_exporter queue_messages_ready \| timechart avg(value) by queue_name` | Consumer processing stalled — check consumer logs; scale consumer pods |
| `queue_messages_unacked` | Messages delivered but not yet acknowledged | ≤ prefetch × consumer count | > 2× prefetch count | > 10× prefetch count | `sourcetype=rabbitmq_exporter queue_messages_unacked \| timechart avg(value) by queue_name` | Consumer processing too slow; reduce prefetch or scale consumers |
| `consumers` | Number of active consumers per queue | ≥ 1 per queue (typically 2–5 per queue at target scale) | 1 (no redundancy) | 0 (no consumer) | `sourcetype=rabbitmq_exporter consumers \| timechart avg(value) by queue_name` | 0 consumers: consumer crash — restart consumer pods immediately |
| `connections` | Total TCP connections to broker | Baseline × ±20% | > 80% of `tcp_listen_backlog` | > connection limit | `sourcetype=rabbitmq_exporter connections \| timechart avg(value)` | Connection leak — check for un-closed connections in producer/consumer code |
| `node_mem_used` | Broker node memory used (bytes) | < 40% of `vm_memory_high_watermark` | > 60% of watermark | > 80% of watermark (flow control triggers at watermark) | `sourcetype=rabbitmq_exporter node_mem_used \| eval pct=value/mem_limit*100 \| timechart avg(pct)` | Flow control imminent — reduce message accumulation; scale broker memory |
| `node_disk_free` | Free disk space on broker node | > 2× `disk_free_limit` | < 2× limit | < 1.1× limit (broker blocks all publishes) | `sourcetype=rabbitmq_exporter node_disk_free \| timechart min(value)` | CRITICAL: consumers must drain queues immediately; add disk or delete old data |
| `fd_used` | File descriptors in use | < 60% of `fd_total` | > 70% | > 85% | `sourcetype=rabbitmq_exporter fd_used \| eval pct=value/fd_total*100 \| timechart avg(pct)` | Increase OS fd limit (`ulimit -n`); check for connection leaks |
| `sockets_used` | Network sockets in use | < 60% of `sockets_total` | > 70% | > 85% | `sourcetype=rabbitmq_exporter sockets_used \| timechart avg(value)` | Same as fd_used — socket exhaustion blocks new connections |
| `channel_count` | Total AMQP channels open across all connections | < 2,000 (rule of thumb: channels = connections × 2–4) | > 5,000 | > 10,000 | `sourcetype=rabbitmq_exporter channels \| timechart avg(value)` | Channel leak in producer/consumer — each operation must close channels after use |
| `exchange_messages_published_total` | Cumulative messages published to exchanges (rate = throughput) | Baseline ± 30% | > 2× baseline (unexpected spike) | N/A (rate; watch for drops) | `sourcetype=rabbitmq_exporter exchange_messages_published_total \| streamstats current=f last(value) as prev by exchange_name \| eval rate=value-prev` | Sudden drop = producer crash; sudden spike = runaway publisher |
| `exchange_messages_confirmed_total` | Cumulative publisher confirms received (should track published) | Tracks published within < 1s lag | confirmed < published by > 5% | confirmed < published by > 20% | `sourcetype=rabbitmq_exporter \| eval confirm_lag=published-confirmed \| timechart avg(confirm_lag)` | Confirm gap = broker under pressure or network issue; check broker memory/disk first |

## 2. Kafka Metrics

| Metric | What It Measures | Normal Range | Warning | Critical | Prometheus Metric Name | Grafana/Splunk Query | Remediation |
|---|---|---|---|---|---|---|---|
| Consumer lag | Messages behind log end offset per consumer group per partition | < 1,000 in steady state | > 10,000 | > 100,000 or lag growing | `kafka_consumer_group_lag` | `kafka_consumer_group_lag{group="$group"} by (topic, partition)` | Consumer processing too slow; scale consumer instances or increase partition count |
| ISR count (In-Sync Replicas) | Number of replicas in sync with leader | = `replication.factor` (typically 3) | ISR < replication factor | ISR = 1 (single copy, no HA) | `kafka_server_replicamanager_isrshrinksrate` | `kafka_topic_partition_in_sync_replica{topic="$topic"}` | ISR shrink = follower lag; check broker disk/network; increase `replica.fetch.wait.max.ms` only if disk-bound |
| Leader election rate | Rate of partition leadership elections | ~0 in steady state | > 1/min (indicates instability) | > 5/min | `kafka_controller_controllerstats_leaderelectionrateandtimems` | `rate(kafka_controller_controllerstats_leaderelectionrateandtimems[5m])` | Frequent elections = broker instability; check broker heap, GC pauses, network timeouts |
| Bytes in per broker | Incoming throughput per broker node | Baseline ± 30%; balanced across brokers | > 70% of network NIC capacity | > 90% NIC capacity or imbalanced by > 2× | `kafka_server_brokertopicmetrics_bytesin` | `rate(kafka_server_brokertopicmetrics_bytesin[1m]) by (broker_id)` | Imbalance = hot partition; trigger preferred leader election or rebalance partitions |
| Bytes out per broker | Outgoing throughput per broker (includes replication traffic) | Baseline; note replication multiplies bytes out | > 70% NIC capacity | > 90% NIC capacity | `kafka_server_brokertopicmetrics_bytesout` | `rate(kafka_server_brokertopicmetrics_bytesout[1m]) by (broker_id)` | Same as bytes in — also check if replication factor × bytes in is driving saturation |
| Request handler idle % | % time request handler threads are idle | > 50% idle (threads not saturated) | < 30% idle | < 10% idle (threads saturated) | `kafka_server_kafkarequesthandlerpool_requesthandleravgidlepercent` | `kafka_server_kafkarequesthandlerpool_requesthandleravgidlepercent` | Thread pool saturation = add brokers or reduce partition count per broker; check for slow disk |
| Network processor idle % | % time network IO threads are idle | > 50% idle | < 30% idle | < 10% idle | `kafka_network_socketserver_networkprocessoravgidlepercent` | `kafka_network_socketserver_networkprocessoravgidlepercent` | Network processor saturation = client connection overload; tune `num.network.threads` |
| Log end offset rate (producer throughput) | Rate of increase of log end offset = producer write rate | Baseline ± 30% | Drop > 50% from baseline | Drop to 0 (producer stopped) | `kafka_log_log_logendoffset` | `rate(kafka_log_log_logendoffset[1m]) by (topic, partition)` | Sudden drop = producer crash or partition leadership issue |
| Replica fetch rate | Rate at which followers are fetching from leaders | Matches write rate; stable | > 30% behind leader write rate | Fetch stopped (ISR shrink imminent) | `kafka_server_fetcherlagmetrics_consumerlag` | `kafka_server_fetcherlagmetrics_consumerlag by (broker_id, topic)` | Follower lag = disk I/O bound or network issue on follower broker; check disk latency |
| Controller active count | Number of active Kafka controllers in the cluster | Exactly 1 | 0 (no controller — cluster stalled) | > 1 (split-brain — extremely serious) | `kafka_controller_kafkacontroller_activecontrollercount` | `kafka_controller_kafkacontroller_activecontrollercount` | 0 = trigger controller election via broker restart; > 1 = ZooKeeper/KRaft fault — immediate escalation |
| Offline partition count | Partitions with no elected leader | 0 | > 0 (any offline partition is an incident) | > 5% of total partitions | `kafka_controller_kafkacontroller_offlinepartitionscount` | `kafka_controller_kafkacontroller_offlinepartitionscount` | Identify affected topic/partition; check if broker holding partition is down; trigger leader election |
| Under-replicated partition count | Partitions where ISR < replication factor | 0 | > 0 sustained for > 2 min | > 10% of partitions | `kafka_server_replicamanager_underreplicatedpartitions` | `kafka_server_replicamanager_underreplicatedpartitions` | Identify which broker is falling behind; check disk I/O, network, broker heap |

## 3. MuleSoft / Anypoint Platform Metrics

| Metric | What It Measures | Normal Range | Warning | Critical | Monitoring Source | Remediation |
|---|---|---|---|---|---|---|
| Active flows | Number of Mule flows currently deployed and running | = deployed flow count (no variation) | Any flow in STOPPED state | Any flow in ERROR state or count drops | Anypoint Runtime Manager → Application Status | Restart affected application; check flow deployment logs for initialization error |
| Flow error rate | % of flow executions that result in an error | < 0.5% in steady state | > 1% | > 5% or sustained climb | Anypoint Monitoring → Flow Metrics | Identify highest error-rate flow; check connector call errors, payload parsing failures |
| Flow response time P99 | P99 latency for synchronous flow executions | Baseline ± 50ms | > 2× baseline | > 5× baseline or SLO breach | Anypoint Monitoring → Response Time | Check downstream connector latency (HTTP, DB); check thread pool saturation |
| Thread pool utilization | % of Mule worker threads in use | < 60% | > 75% | > 90% (near starvation) | JMX → `Threading` → `ActiveThreadCount / MaxThreadCount` | Increase `maxThreads` in domain config; investigate slow synchronous calls blocking threads |
| Memory heap used % | JVM heap utilization | < 70% | > 80% | > 90% (GC thrashing imminent) | JMX → `java.lang:type=Memory` → `HeapMemoryUsage.used` | Trigger GC analysis; check for message payload size growth; increase JVM heap `-Xmx` |
| GC pause time | Duration of JVM stop-the-world GC pauses | < 200ms (G1GC, typical) | > 500ms | > 2s (flow timeouts likely) | JMX → GarbageCollector MBean | Switch to G1GC if not already; reduce heap fragmentation; profile large object allocations |
| HTTP connector response time | Latency of outbound HTTP calls from MuleSoft flows | Baseline ± 50ms | > 2× baseline | > SLO of downstream service | Anypoint Monitoring → Connector Metrics | Check downstream service health; check DNS resolution time; check connection pool exhaustion |
| DB connector response time | Latency of database calls from MuleSoft flows | < 100ms for simple queries | > 500ms | > 2s | Anypoint Monitoring → Connector Metrics | Check query plan; check DB connection pool; check L1 replication lag |
| VM queue depth (queued messages) | Messages queued in MuleSoft in-memory VM queues | Near 0 in steady state | > 100 | > 1,000 (memory pressure) | JMX → VM Queue stats | Consumer flow not processing; check consumer flow status; VM queues are NOT persistent — messages lost on restart |
| Event processor throughput | Messages processed per second by async flows | Baseline ± 20% | > 50% drop from baseline | Drop to 0 (consumer stopped) | Anypoint Monitoring → Message Processing Rate | Check flow status; check source connectivity (Kafka/RabbitMQ connector status); check thread pool |

## 4. Splunk Search Examples — Ready to Use

### RabbitMQ: Queue depth over time by queue name
```spl
index=middleware sourcetype=rabbitmq_exporter metric_name=queue_messages
| timechart span=1m avg(value) by queue_name
| rename * as "Queue: *"
```

### Kafka: Consumer lag by consumer group over 1 hour
```spl
index=middleware sourcetype=kafka_exporter metric_name=kafka_consumer_group_lag
| where earliest=-1h
| timechart span=5m max(value) by consumer_group
| rename * as "Group: *"
```

### MuleSoft: Flow error rate by flow name (last 30 min)
```spl
index=mulesoft sourcetype=anypoint_monitoring
| where earliest=-30m
| eval is_error=if(status="ERROR" OR http_status>=500, 1, 0)
| stats count as total, sum(is_error) as errors by flow_name
| eval error_rate=round(errors/total*100, 2)
| sort -error_rate
| table flow_name, total, errors, error_rate
```

### DLQ: Message arrival rate (RabbitMQ dead letter queues)
```spl
index=middleware sourcetype=rabbitmq_exporter metric_name=queue_messages_published_total
| where queue_name LIKE "%.dlq" OR queue_name LIKE "dead%"
| streamstats current=f last(value) as prev_value by queue_name
| eval rate_per_min=value - prev_value
| where rate_per_min > 0
| timechart span=1m sum(rate_per_min) by queue_name
```

### Middleware broker restart events (RabbitMQ + Kafka)
```spl
index=middleware (sourcetype=rabbitmq OR sourcetype=kafka)
| where (message="broker started" OR message="node up" OR event_type="BROKER_RESTART")
| stats count as restart_count, values(host) as hosts by sourcetype, _time span=1h
| where restart_count > 0
| table _time, sourcetype, hosts, restart_count
| sort -_time
```

### Combined middleware health summary (all three platforms, last 15 min)
```spl
index=middleware earliest=-15m
| eval platform=case(
    sourcetype LIKE "rabbitmq*", "RabbitMQ",
    sourcetype LIKE "kafka*", "Kafka",
    sourcetype LIKE "anypoint*", "MuleSoft",
    true(), "Other")
| eval is_critical=if(
    (sourcetype LIKE "rabbitmq*" AND metric_name="queue_messages" AND value > 100000), 1,
    if((sourcetype LIKE "kafka*" AND metric_name="kafka_consumer_group_lag" AND value > 100000), 1,
    if((sourcetype LIKE "anypoint*" AND metric_name="error_rate" AND value > 5), 1, 0)))
| stats sum(is_critical) as critical_signals by platform
| sort -critical_signals
```
