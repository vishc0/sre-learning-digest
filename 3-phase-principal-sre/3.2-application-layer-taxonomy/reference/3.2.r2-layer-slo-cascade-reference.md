# SLO Cascade Reference — The Mathematics of Dependent Reliability

---

## 1. The SLO Cascade Law

Every layer in a call chain adds failure probability. Storage is the foundation — its SLO sets the ceiling for every layer above it. A storage layer at 99.9% availability means no call chain that depends on it can honestly promise better than 99.9%, before any application-layer failures are counted.

**The rule:** You cannot promise a consumer SLO that is better than the weakest dependency in the call chain. If you do, you have a math problem disguised as a contract.

This is not a theoretical concern. In practice, SLO violations at L5 are frequently discovered to be L1 failures that went undetected because L5 was promising 99.9% while its L1 dependency was actually running at 99.5%.

**The VP question this enables:** "Show me the dependency chain for your SLO. What does the cascade math say the true achievable availability is?" If the engineering team cannot answer this in under two minutes, the SLO governance is broken.

---

## 2. Mathematical Model

### Availability of a serial dependency chain

```
Available(chain) = Available(L1) × Available(L2) × ... × Available(LN)
```

Where each Available(LN) is expressed as a decimal (99.9% = 0.999).

### Implication for SLO setting

```
SLO(consumer) ≤ SLO(dependency_1) × SLO(dependency_2) × ... × SLO(dependency_N)
```

Any SLO set higher than the right-hand side is mathematically unachievable. It is a governance fiction.

### Error budget relationship

```
Error_Budget(chain) = 1 - (Available_L1 × Available_L2 × ... × Available_LN)
```

Adding a single 99.5% dependency to a chain consumes 0.5% of the total error budget, regardless of how well all other dependencies perform.

### Parallel fallback (reduces risk, does not eliminate it)

```
Available(with_fallback) = 1 - ((1 - Available_primary) × (1 - Available_fallback))
```

Two 99.5% services in parallel (active-active fallback) → 1 - (0.005 × 0.005) = 99.9975%. This is why fallbacks matter — but only if they are truly independent (different AZ, different region, no shared dependency).

## 3. The Cascade Table

| Layer | Layer Name | Recommended SLO | Budget Consumed | Notes |
|---|---|---|---|---|
| L1 | Persistent Storage | **99.99%** | 0.01% budget used | Foundation. Drives everything. Every additional 9 here is earned through HA topology (multi-AZ, synchronous replication), not hope. |
| L2 | Storage Abstraction | **99.99%** | 0.01% budget used | A well-tuned cache or connection pool adds negligible failure probability. Match L1. If cache fails open (cache miss → L1 hit), this holds. If cache fails closed (cache miss → error), it does not. |
| L3 | Data Services | **99.9%** | 0.1% budget used | Service has its own failure modes: bad deploys, query errors, pod crashes. The 0.1% gap vs L1 is your service error budget. |
| L4 | Middleware / Integration | **99.9%** | Independent budget | Middleware is typically a parallel track, not in the L1→L7 serial chain. Its SLO applies to message delivery reliability, not query latency. Kafka: 99.9% is achievable with ISR≥2. RabbitMQ: 99.9% with HA queues. |
| L5 | Application Services | **99.5%** | 0.5% budget used | Calls multiple services. Each call contributes failure probability. Budget consumed fast — this is why circuit breakers are not optional at L5. |
| L6 | API Gateway / Edge | **99.99%** | 0.01% budget used | Must be the most reliable layer. It is the front door to all other services. If it fails, every service behind it is unreachable regardless of their own availability. Treat L6 like L1. |
| L7 | Presentation / UI | **99.5%** | Inherits all upstream risk + UI-specific risk | UI accumulates all upstream risk plus its own (bad bundle deploys, CDN failures, browser compatibility). 99.5% end-to-end user experience is the honest number for most stacks. |

## 4. Worked Example — Order Completion User Journey

**The journey:** User clicks "Place Order" → Kong Gateway → OrderService → PaymentService → CustomerDataService → customer_db

### Step 1: Map the chain

```
kong-gateway (L6) → order-svc (L5) → payment-svc (L5) → customer-data-svc (L3) → customer_db (L1)
```

### Step 2: Plug in SLOs

| Component | Layer | SLO |
|---|---|---|
| kong-gateway | L6 | 99.99% |
| order-svc | L5 | 99.5% |
| payment-svc | L5 | 99.5% |
| customer-data-svc | L3 | 99.9% |
| customer_db | L1 | 99.99% |

### Step 3: Multiply the chain

```
End-to-end = 0.9999 × 0.995 × 0.995 × 0.999 × 0.9999
           = 0.9999 × 0.995 × 0.995 × 0.999 × 0.9999
           ≈ 0.9892
           ≈ 98.9%
```

### Step 4: Interpret

**If your SLA to users promises 99.9% for order placement — you have a math problem.** The cascade says 98.9% is the ceiling, assuming every component hits its SLO exactly. Any single component underperforming its SLO pushes you lower.

**To achieve 99.5% end-to-end,** you need either:
- Tighter component SLOs (e.g., L5 services at 99.9% each), OR
- Fallbacks that keep the journey alive when a component fails (e.g., cached payment methods, async order submission), OR
- Acceptance that 99.5% cannot be promised without architectural change

### Step 5: The VP framing

"Our SLA says 99.9% for order placement. Our cascade analysis says 98.9% is the theoretical maximum given current component SLOs. That's a 1% gap — meaning our SLA is a commercial promise we cannot fulfill by construction. Before we commit to the next renewal, we either need to raise component SLOs or restructure the journey with fallbacks."

## 5. SLO Consistency Checklist — VP Questions for Any SLO Review

Use these in SLO review meetings to detect cascade violations and governance gaps.

| Question | What a Bad Answer Looks Like |
|---|---|
| "What is the dependency chain for this SLO?" | "We don't track that" / blank look |
| "What is the cascade math for this SLO?" | Any number higher than the product of dependencies |
| "Which dependency has the smallest error budget?" | "We don't know" / no monitoring for that dependency |
| "What happens to this SLO when L1 is at its worst-case SLO?" | "It's fine" without doing the math |
| "Do any of your dependencies have a higher promised SLO than yours?" | Yes — this means you are promising to be more reliable than your foundation |
| "Where is the circuit breaker in this chain?" | No circuit breaker between L5 and L3 dependencies = retry storm risk |
| "What does your error budget look like YTD, and which component consumed the most?" | "We don't track by component" = cannot manage to SLO |
| "If your L3 dependency degrades to 99.5% for an hour, what does that do to your SLO?" | Unknown = no cascade awareness |

## 6. Common Cascade Violations

These are SLOs that are mathematically impossible given their stated dependencies. All real examples of governance failures found in production systems.

| Violation | Why It Is Impossible | What Usually Happens |
|---|---|---|
| L5 service promises 99.9% but calls an L5 dependency promising 99.5% | 99.9% consumer × 99.5% dependency = at best 99.4% end-to-end | Consumer misses SLO in incidents; blamed on "downstream issues" that were predictable |
| L6 gateway promises 99.99% but backend services are only 99.5% | Gateway cannot serve requests it cannot fulfill, regardless of its own uptime | Gateway is "up" but serving 5xx — uptime and availability are confused |
| L7 UI promises 99.9% but makes 5 synchronous calls each at 99.5% | 0.995^5 = 97.5% chain availability | UI "flickers" in production; blamed on the frontend team; root cause is architecture |
| L3 data service promises 99.9% but L1 database is at 99.9% with no L2 cache | L3 can be no better than L1 without caching; its own failures add to, not replace, L1 risk | L3 SLO is met only when L1 overperforms; any L1 degradation immediately breaches L3 |
| Middleware promises 99.99% but runs on single-AZ broker | Hardware failure in that AZ = 100% outage; no amount of software HA fixes physical placement | Annual disk or host failure causes multi-hour outage; SLA payments follow |
| Consumer SLO set equal to its dependency's SLO with no room for own failures | Consumer's own failure budget is 0% — any consumer-side bug immediately breaches the SLO | Every minor deploy issue triggers an SLO breach; teams game the error budget accounting |
