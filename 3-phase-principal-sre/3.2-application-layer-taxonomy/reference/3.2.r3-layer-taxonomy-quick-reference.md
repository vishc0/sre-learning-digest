# Layer Taxonomy Quick Reference — Carry Into Any Meeting

---

## Layer 1 — Persistent Storage

| | |
|---|---|
| **What belongs here** | Databases (PostgreSQL, MySQL, Oracle, Cassandra, DynamoDB), object stores (S3, GCS), data warehouses, file stores with persistence guarantees |
| **Typical owner** | Database Platform Engineering / Storage SRE |
| **Key metrics** | Replication lag (ms) · Connection pool saturation (%) · IOPS utilization (%) · Query P99 latency (ms) |
| **Common failure mode** | Replication lag → read replicas serving stale data; connection pool exhaustion cascades to all callers |
| **First diagnostic step** | `SHOW SLAVE STATUS` / replication lag metric → then connection count vs max_connections |

---

## Layer 2 — Storage Abstraction

| | |
|---|---|
| **What belongs here** | Redis/Memcached clusters, PgBouncer/ProxySQL connection pools, read replicas fronted by a proxy, CDN for static assets |
| **Typical owner** | Database Platform Engineering / Caching SRE |
| **Key metrics** | Cache hit rate (%) · Eviction rate (events/sec) · Memory utilization (%) · Connection pool wait time (ms) |
| **Common failure mode** | Cache stampede on cold start after restart; thundering herd from all callers hitting L1 simultaneously |
| **First diagnostic step** | Check eviction rate and memory utilization → then inspect keyspace expiry distribution |

---

## Layer 3 — Data Services

| | |
|---|---|
| **What belongs here** | Read-optimized REST/gRPC services that wrap a single data store with no business logic; entity APIs (CustomerService, ProductCatalogService) |
| **Typical owner** | Domain Platform Team / Data SRE |
| **Key metrics** | Request rate (req/sec) · Error rate (%) · P99 latency (ms) · DB connection pool utilization (%) |
| **Common failure mode** | N+1 queries under load; upstream DB saturation reflected as latency spike at this layer |
| **First diagnostic step** | Check error rate → trace to DB call latency → check connection pool saturation at L1/L2 |

---

## Layer 4 — Middleware / Integration

| | |
|---|---|
| **What belongs here** | RabbitMQ, Kafka, TIBCO EMS, MuleSoft/Anypoint, IBM MQ, ActiveMQ, event buses, ETL pipelines, API integration hubs |
| **Typical owner** | Messaging Platform Team / Integration SRE |
| **Key metrics** | Queue depth (messages) · Consumer lag (messages, Kafka) · Unacked messages · Consumer count · Broker memory/disk |
| **Common failure mode** | Consumer group falls behind → queue backlog → broker disk exhaustion → broker crash → all producers blocked |
| **First diagnostic step** | Check consumer lag / queue depth → check consumer count → check broker disk and memory |

---

## Layer 5 — Application Services

| | |
|---|---|
| **What belongs here** | Business logic services (OrderService, PaymentService, FraudService), microservices that orchestrate other services, batch workers |
| **Typical owner** | Product Domain Team / Application SRE |
| **Key metrics** | Error rate (%) · P99 latency (ms) · Pod CPU/memory · Circuit breaker open/closed state · Retry rate |
| **Common failure mode** | Dependency latency timeout cascade; circuit breaker open → all traffic fast-failing; retry storms amplifying downstream load |
| **First diagnostic step** | Check error rate and what errors are (5xx vs timeout) → trace to which downstream call is failing → check circuit breaker state |

---

## Layer 6 — API Gateway / Edge

| | |
|---|---|
| **What belongs here** | Kong, NGINX, AWS API Gateway, Apigee, Envoy, load balancers, WAF, CDN edge, BFF (Backend for Frontend) routing tier |
| **Typical owner** | Platform Engineering — Edge / Network SRE |
| **Key metrics** | Request rate (req/sec) · Error rate (4xx/5xx %) · P99 latency (ms) · Active connections · Rate limit rejections (429/sec) |
| **Common failure mode** | Route misconfiguration → entire API surface returns 502/504; WAF rule false positive → legitimate traffic blocked |
| **First diagnostic step** | Check 5xx rate at gateway → check upstream health check status → check route configuration for recent changes |

---

## Layer 7 — Presentation / UI

| | |
|---|---|
| **What belongs here** | React/Angular SPAs, mobile app bundles, server-side rendered pages, BFF response aggregation, CDN-delivered frontend assets |
| **Typical owner** | Frontend Engineering / Experience SRE |
| **Key metrics** | Core Web Vitals (LCP, CLS, FID) · JS error rate · Page load P75 (ms) · API call error rate from client · Apdex score |
| **Common failure mode** | Broken JS bundle deploy → blank page for all users; CDN cache poisoning; BFF API contract mismatch after backend deploy |
| **First diagnostic step** | Check JS error rate in RUM → check CDN cache status → check BFF API response contract vs frontend expectations |

## Layer → Incident Lead Mapping

| Layer | Primary Incident Lead | Escalation Path |
|---|---|---|
| L1 Persistent Storage | Database SRE on-call | → DB Platform Lead → DBA on-call |
| L2 Storage Abstraction | Database SRE on-call | → DB Platform Lead |
| L3 Data Services | Domain SRE on-call (owning domain) | → Domain Tech Lead |
| L4 Middleware / Integration | Messaging Platform SRE on-call | → Messaging Platform Lead |
| L5 Application Services | Domain SRE on-call (owning domain) | → Domain Tech Lead → Engineering Director |
| L6 API Gateway / Edge | Platform SRE on-call | → Platform Engineering Lead → Networking |
| L7 Presentation / UI | Frontend SRE on-call | → Frontend Engineering Lead |

**Cross-layer incident (symptoms in L6, root cause in L1):** Platform SRE leads the bridge. Root-cause layer owner executes the fix. Incident commander bridges until root cause is confirmed.

## Layer → Change Risk (Highest Risk Change per Layer)

| Layer | Highest Risk Change | Minimum Approval Required | Rollback Complexity |
|---|---|---|---|
| L1 Persistent Storage | Schema migration on live table | Database Change Board + Engineering Director | HIGH — may require restore from backup |
| L2 Storage Abstraction | Cache cluster topology change (add/remove node) | Platform Lead + Domain SRE sign-off | MEDIUM — flush and rebuild |
| L3 Data Services | Breaking API contract change | Domain Tech Lead + all consumer teams sign-off | HIGH — consumers must roll back too |
| L4 Middleware / Integration | Kafka consumer group offset reset | Messaging Platform Lead + consuming team | HIGH — messages may be reprocessed or skipped |
| L5 Application Services | Dependency version upgrade with API breaking change | Domain Tech Lead | MEDIUM — re-deploy previous image |
| L6 API Gateway / Edge | WAF rule modification (block mode) | Platform Engineering Director | LOW — rule disable, effect immediate |
| L7 Presentation / UI | Frontend bundle deploy (full replacement) | Frontend Tech Lead | LOW — CDN rollback to previous version |

## SLO Cascade Reminder

Storage is strictest. Every layer above inherits risk from below and adds its own.

```
L1 Persistent Storage     99.99%  ← foundation — tightest budget
L2 Storage Abstraction    99.99%  ← adds negligible overhead if well-tuned
L3 Data Services          99.9%   ← own failure modes: query errors, timeouts
L4 Middleware / Integration 99.9% ← independent chain (not always in the L1→L7 path)
L5 Application Services   99.5%   ← calls multiple services; budget consumed fast
L6 API Gateway / Edge     99.99%  ← must be most reliable (front door to all)
L7 Presentation / UI      99.5%   ← absorbs all upstream risk + UI-specific failures
```

**The cascade law:** If L5 calls L3 which calls L1, end-to-end = 99.5% × 99.9% × 99.99% ≈ 99.4%. You cannot promise more than the math allows. Always multiply the chain — never set consumer SLO higher than the weakest link.
