# Change Risk by Layer — Governance Reference

The definitive change risk and approval matrix. Use in CAB submissions, change ticket creation, and architecture reviews. Risk level drives approval requirements — do not negotiate the level without a documented architectural rationale.

---

## Layer 1 — Persistent Storage

| Change Type | Risk Level | Approval Requirements | Rollback Method | Rollback Complexity | Blast Radius | Min Lead Time | Pre-conditions | Post-validation |
|---|---|---|---|---|---|---|---|---|
| Schema migration (ALTER TABLE on live table) | **CRITICAL** | Database Change Board + Engineering Director + DBA lead | Reverse migration script pre-tested in staging; restore from backup if reverse fails | HIGH — reverse migration may not be possible if data was written in new schema format | All services reading/writing this table; data corruption risk | 5 business days | Full backup verified, staging dry-run complete, off-peak window, all consumers on compatible code first | Row count check, query latency baseline comparison, replication lag < 5s, consumer error rate < 0.1% for 30 min |
| Index addition | HIGH | Database Change Board + DBA lead | `DROP INDEX` — fast, no data risk | LOW — index drop is immediate | Write throughput degradation during index build; lock risk on PostgreSQL < 12 (use `CREATE INDEX CONCURRENTLY`) | 2 business days | `CREATE INDEX CONCURRENTLY` confirmed; staging test complete; disk space > 2× index estimated size available | Index present in `pg_indexes`; query plan using new index for target queries; no lock wait spikes |
| Index removal | HIGH | Database Change Board + DBA lead | Recreate index (`CREATE INDEX CONCURRENTLY`) | MEDIUM — may take hours on large tables | Queries relying on index will full-scan; may spike CPU and latency immediately | 2 business days | Confirm no queries depend on this index via `pg_stat_user_indexes.idx_scan` — zero scans in 30 days | Query latency baseline for known queries using this index; no P99 regression within 15 min of change |
| DB version upgrade (major) | **CRITICAL** | Database Change Board + Engineering Director + Vendor/DBA validation | Failover to pre-upgrade replica; promote previous version | HIGH — logical replication may break; extensions may be incompatible | All services connected; potential for connection protocol changes | 15 business days | Upgrade tested in staging under production load for ≥ 1 week; rollback procedure tested end-to-end; maintenance window with user notification | All connection pools reconnected; full integration test suite pass; replication lag stable < 5s |
| Replication topology change (add/remove replica, change sync mode) | **CRITICAL** | Database Change Board + Engineering Director | Restore previous replication configuration | HIGH — data lag or loss possible during topology change | Read replicas and standby promotion path affected | 5 business days | Replication health 100% before change; change tested in staging; monitoring confirmed for replication lag | ISR count (if applicable) restored; replication lag < 1s; failover test successful post-change |
| Storage expansion (add volume, resize) | MEDIUM | DBA lead + Platform Engineering | Not applicable — expansion is not reversible; partition or shrink separately | LOW — expansion is safe; only risk is misconfiguration | Minimal — brief I/O pause on some storage backends | 1 business day | Disk space < 70% triggering alert; capacity planning document attached | Disk utilization drop confirmed; no I/O errors in broker logs |
| Backup policy change (frequency, retention) | MEDIUM | DBA lead + Compliance Officer (if retention reduced) | Restore previous backup cron/policy config | LOW | RPO is affected if frequency reduced | 1 business day | New policy approved by compliance if retention period shortened; backup job tested end-to-end | First backup under new policy completed and verified restorable |
| Connection limit change (`max_connections`) | HIGH | DBA lead + all team leads of services connecting | Revert `max_connections` via pg config reload | MEDIUM — reload requires restart on some DB versions | All services; connection pool exhaustion or reduced capacity | 2 business days | Current connection utilization documented; new limit tested by simulating load | Connection pool saturation metric stable; no `FATAL: too many connections` errors |

## Layer 2 — Storage Abstraction

| Change Type | Risk Level | Approval Requirements | Rollback Method | Rollback Complexity | Blast Radius | Min Lead Time | Pre-conditions | Post-validation |
|---|---|---|---|---|---|---|---|---|
| Redis cluster topology change (add/remove node) | HIGH | Platform Engineering Lead + Domain SRE teams using cluster | Remove added node or re-add removed node; data resharding required | MEDIUM — key redistribution takes time; in-flight operations may fail | All services using this cluster; cache hit rate will drop during resharding | 2 business days | Keyspace distribution analysis complete; resharding window in off-peak; backup of AOF/RDB if persistence enabled | Cache hit rate ≥ pre-change baseline within 15 min; no increase in L1 query rate beyond expected resharding period |
| Cache invalidation policy change (TTL, eviction policy) | HIGH | Domain Tech Lead + SRE | Revert TTL/eviction config via Redis CONFIG SET (live, no restart) | LOW — config rollback is immediate | All consumers of this cache; stale data risk (TTL increase) or cache stampede risk (TTL decrease) | 1 business day | Load test with new TTL in staging; thundering herd analysis if TTL reduced significantly | Cache hit rate stable; L1 query rate not spiking beyond 20% above baseline |
| Connection pool limit change (PgBouncer/ProxySQL) | MEDIUM | Platform Engineering Lead | Revert pool_size config; reload pgbouncer.ini | LOW — reload is graceful | All services connected through this pool | 1 business day | Current utilization documented; new limit tested under load | `pgbouncer SHOW POOLS` shows new limits; no client-wait timeouts |
| Cache version upgrade (Redis major version) | HIGH | Platform Engineering Lead + Domain SRE teams | Failover to standby running previous version | MEDIUM — data compatibility check required | All services using this cluster | 5 business days | Compatibility matrix checked (client library vs. Redis version); staging upgrade test ≥ 2 days; AOF backup verified | All client connections restored; no protocol errors; hit rate restored to baseline |

## Layer 3 — Data Services

| Change Type | Risk Level | Approval Requirements | Rollback Method | Rollback Complexity | Blast Radius | Min Lead Time | Pre-conditions | Post-validation |
|---|---|---|---|---|---|---|---|---|
| Breaking API contract change (field removal, type change, endpoint deletion) | **CRITICAL** | Domain Tech Lead + ALL consuming team leads sign-off | Re-deploy previous service version; consumers must revert simultaneously | HIGH — coordinated rollback across multiple teams | Every consumer of this service; coordinated deploy required | 10 business days | All consumers updated and tested against new contract; deprecation notice issued at minimum one sprint prior; consumer acceptance test suite green | Consumer error rate < 0.1% for all known consumers; contract test suite green |
| Additive API change (new field, new endpoint, new optional parameter) | LOW | Domain Tech Lead | Re-deploy previous version | LOW — consumers ignore unknown fields (if following Postel's law) | Minimal — additive changes are backward compatible | 1 business day | OpenAPI spec updated; contract tests updated; consumers do not break on unknown fields (verified) | API documentation published; no consumer errors; new field queryable and correct |
| Service version upgrade (runtime, framework) | HIGH | Domain Tech Lead + SRE | Re-deploy previous container image (ArgoCD rollback or `kubectl rollout undo`) | LOW — image rollback is fast | All consumers of this service | 2 business days | Upgrade tested in staging under production-representative load; dependency audit complete | Error rate and latency baseline met within 5 min of deploy; no unexpected GC pressure |
| Dependency library upgrade (patch/minor) | MEDIUM | Domain Tech Lead | Re-deploy previous image | LOW | Service consumers; indirect risk from library behavior changes | 1 business day | `diff` of library changelog reviewed; no known CVEs; staging test complete | Error rate stable; no new exception types in logs |
| Query optimization change (index hint, query rewrite) | MEDIUM | DBA lead + Domain Tech Lead | Revert query code in service | MEDIUM — requires re-deploy | Consumers of affected endpoint; L1 load may change | 1 business day | EXPLAIN ANALYZE before and after in staging; no regression in non-target query paths | P99 latency of target query improved or unchanged; L1 CPU/IOPS stable |

## Layer 4 — Middleware / Integration

| Change Type | Risk Level | Approval Requirements | Rollback Method | Rollback Complexity | Blast Radius | Min Lead Time | Pre-conditions | Post-validation |
|---|---|---|---|---|---|---|---|---|
| Queue creation (RabbitMQ/ActiveMQ) | LOW | Messaging Platform Lead | Delete queue (only if empty; messages are lost otherwise) | LOW | None at creation; consuming services not yet connected | Same-day | Naming convention follows standard; DLQ companion queue created; TTL policy set if appropriate | Queue visible in management UI; policy applied; consumer can connect |
| Queue deletion (RabbitMQ) | **CRITICAL** | Messaging Platform Lead + ALL consuming team leads + Engineering Director | Cannot restore messages; no rollback | CRITICAL — messages are permanently lost | All producers and consumers of this queue; messages in-flight at deletion time are lost | 10 business days | Zero messages in queue confirmed; all consumers confirmed decommissioned; producers confirmed stopped | Queue absent from management UI; no consumer connection errors in downstream services |
| Exchange binding change (add/modify/remove routing key) | HIGH | Messaging Platform Lead + affected producer and consumer team leads | Restore previous binding configuration | MEDIUM — messages routed during the window may be mis-routed or lost | Any service producing to or consuming from this exchange | 2 business days | Routing impact analysis complete; change tested in staging with actual message payloads | Message routing verified with test publish; consumer receive rate stable |
| Kafka topic creation | LOW | Messaging Platform Lead | Delete topic (only if no data matters; messages are lost) | LOW | None at creation | Same-day | Partition count set with growth in mind (cannot reduce later); replication factor ≥ 2; retention policy set | Topic visible in Kafka admin; ISR = replication factor |
| Kafka partition increase | MEDIUM | Messaging Platform Lead + consumer team leads | Cannot reduce partition count; no rollback | N/A | Consumer groups that use key-based partitioning will see key rebalancing — message ordering guarantee broken for in-flight messages | 2 business days | Consumer teams aware of key redistribution; ordering-sensitive consumers given separate window or paused during change | New partition count visible; consumer group rebalanced; lag returning to baseline within 10 min |
| Kafka consumer group offset reset | **CRITICAL** | Messaging Platform Lead + consumer team lead + Engineering Director | No automated rollback; manually set offset back to previous value if known | HIGH — re-processing or message skipping may have already occurred | The consumer group and all business processes it drives; potential duplicate processing or data loss | 5 business days | Consumer is stopped before offset reset; reason for reset documented (skip vs. replay); replay impact on downstream systems assessed | Consumer restarted; lag metric tracking from new offset; no duplicate processing alerts firing |
| Kafka consumer group deletion | HIGH | Messaging Platform Lead + consumer team lead | Recreate group — offset history is lost; consumer restarts from `auto.offset.reset` policy | HIGH — consumer resumes from earliest or latest, not from where it stopped | The consumer group and its processing continuity | 2 business days | Consumer confirmed stopped; all in-flight processing complete; confirm group deletion is intentional (decommission vs. error) | Group absent from Kafka admin; if recreated, consuming from expected offset |
| MuleSoft flow deployment | MEDIUM | Domain Tech Lead + MuleSoft Platform Lead | Re-deploy previous application version via Anypoint Runtime Manager | LOW — previous version image available in Anypoint Exchange | All API consumers of this flow | 1 business day | Staging deploy and test complete; payload contract unchanged or change communicated to consumers | Flow status STARTED; error rate < 0.5% within 2 min of deploy; Anypoint Monitoring showing traffic |
| MuleSoft runtime upgrade (Mule 4.x minor/major) | HIGH | MuleSoft Platform Lead + Domain Tech Lead + Engineering Director | Rollback to previous runtime version in Anypoint Runtime Manager | MEDIUM — runtime change may require application repackage | All flows running on this runtime | 5 business days | All flows tested on new runtime in staging; connector compatibility matrix checked; maintenance window scheduled | All flows STARTED; no compatibility errors in runtime logs; connector response times stable |
| Broker cluster expansion (add RabbitMQ node / Kafka broker) | MEDIUM | Messaging Platform Lead | Remove added node (with caution — requires partition/queue rebalancing) | MEDIUM | Minimal during expansion; rebalancing causes brief throughput dip | 1 business day | Node provisioned and verified healthy before joining cluster; disk capacity plan confirmed | New node in cluster; ISR count stable; no partition leadership elections during expansion |
| Broker version upgrade (RabbitMQ/Kafka major) | **CRITICAL** | Messaging Platform Lead + Engineering Director + Vendor support contact | Failover to pre-upgrade broker; promote standby running previous version | HIGH — protocol changes may require client library updates simultaneously | All producers and consumers | 10 business days | Client library compatibility matrix checked; rolling upgrade tested in staging; rollback procedure tested end-to-end; maintenance window with producer/consumer team notification | All clients reconnected; no protocol errors; throughput metrics at baseline; ISR stable |

## Layer 5 — Application Services

| Change Type | Risk Level | Approval Requirements | Rollback Method | Rollback Complexity | Blast Radius | Min Lead Time | Pre-conditions | Post-validation |
|---|---|---|---|---|---|---|---|---|
| Service deploy via CI/CD (standard release) | MEDIUM | Domain Tech Lead (automated via CI gate) | `kubectl rollout undo` or ArgoCD rollback to previous sync | LOW — rollback in < 2 min | All consumers of this service | Same-day (if CI gates pass) | All automated tests pass; staging smoke test complete; on-call engineer aware | Error rate and P99 at pre-deploy baseline within 5 min; no new alert firing |
| Config change (environment variable, ConfigMap) | MEDIUM | Domain Tech Lead | Revert ConfigMap/secret and trigger rolling restart | LOW | Service consumers; potential behavior change from config | Same-day | Config change peer-reviewed; staging test complete | Service running with new config (confirm via `/config` endpoint if available); error rate stable |
| Feature flag toggle | LOW | Product Manager + Domain Tech Lead | Toggle flag back via feature flag service | LOW — immediate, no deploy | Users/traffic segments affected by flag; usually incremental rollout | Same-day | Flag targeting rules tested; rollback toggle confirmed accessible on-call | Target user segment behavior matches expected; error rate and latency stable |
| Dependency version upgrade (major with API changes) | HIGH | Domain Tech Lead + consumers of this service if behavior changes | Re-deploy previous image | LOW — image rollback | Service and its consumers if exposed behavior changes | 2 business days | Integration tests cover dependency API surface; contract tests green; no known CVEs introduced | Error rate and P99 stable; dependency call success rate ≥ pre-upgrade |
| Circuit breaker config change (threshold, timeout) | HIGH | Domain Tech Lead + SRE | Revert circuit breaker config; re-deploy | LOW | Consumers of downstream services; incorrect threshold = false trips or missing protection | 1 business day | Load test with new config in staging; downstream SLO reviewed to validate new timeout value | Circuit breaker state CLOSED under normal load; trip test in staging confirmed correct threshold |

## Layer 6 — API Gateway / Edge

| Change Type | Risk Level | Approval Requirements | Rollback Method | Rollback Complexity | Blast Radius | Min Lead Time | Pre-conditions | Post-validation |
|---|---|---|---|---|---|---|---|---|
| Route addition | LOW | Platform Engineering Lead | Delete route configuration | LOW | None — new route adds traffic path; does not affect existing routes | Same-day | Upstream service health confirmed; route config reviewed for typos; auth policy applied | Route reachable; upstream responding; auth enforced (test unauthenticated request returns 401/403) |
| Route modification (upstream change, path rewrite) | HIGH | Platform Engineering Lead + affected consumer team leads | Revert route config | LOW | All consumers using the modified route | 1 business day | Consumer teams notified; staging test with production-representative traffic; rollback tested | Traffic flowing; no 5xx spike; latency baseline met |
| Rate limit change | MEDIUM | Platform Engineering Lead + Product Manager (if affects external SLA) | Revert rate limit config | LOW — immediate | External consumers and partners subject to the limit | 1 business day | Business justification documented; partner notification sent if external SLA affected | New limit visible in gateway metrics; no false throttling of legitimate traffic (check 429 rate) |
| WAF rule addition (detect mode) | LOW | Platform Engineering Lead + Security | Disable or remove rule | LOW | None in detect mode — logs only, no blocking | Same-day | Rule tested against recent traffic sample to check false positive rate | Rule fires on expected attack pattern in test; no false positives on production traffic within 1 hour |
| WAF rule modification (block mode) | **CRITICAL** | Platform Engineering Director + Security Lead | Disable rule immediately; immediate effect | LOW — rule disable is instantaneous | All external API consumers — false positive = legitimate traffic blocked | 5 business days | False positive rate analysis against 30 days of production traffic; rule tested in detect mode for ≥ 1 week before block mode; rollback procedure rehearsed | 429/403 rate for legitimate user agents at baseline; security team confirmed attack pattern is being blocked |
| TLS certificate rotation | HIGH | Platform Engineering Lead + Security | Restore previous certificate (if rotation fails; requires private key backup) | HIGH — certificate restore requires key material; backup must be pre-verified | All TLS-terminating traffic; failure = all HTTPS traffic drops | 2 business days | New certificate validated against CA chain; private key backed up; auto-renewal tested in staging; change during off-peak | TLS handshake success rate 100%; certificate expiry date updated; no client SSL errors |
| Load balancer health check change | HIGH | Platform Engineering Lead | Revert health check config | LOW | All services behind this load balancer; incorrect health check = healthy services marked down | 1 business day | New health check endpoint confirmed present in all upstream service versions; test with manual curl | Upstream services showing healthy in load balancer; traffic distribution restored to normal |
| CDN cache policy change (TTL, cache rules) | MEDIUM | Platform Engineering Lead + Frontend Tech Lead | Revert cache policy config; invalidate cache if stale content served | MEDIUM — invalidation may be slow and costly | All CDN consumers; stale content risk (TTL increase) or origin overload (TTL decrease) | 1 business day | Origin capacity can absorb traffic increase if TTL reduced; cache invalidation cost estimated | Cache hit rate at expected new baseline; origin request rate at expected level |

## Change Risk Summary — VP Executive View

One row per layer. Highest-risk change type. Minimum approval. This is the governance floor — individual organizations may require more, never less.

| Layer | Layer Name | Highest Risk Change | Minimum Approval | Rollback Speed | Why This Is the Highest Risk |
|---|---|---|---|---|---|
| L1 | Persistent Storage | Schema migration on live table | Database Change Board + Engineering Director | Hours to days | Data corruption is irreversible; cascades to all dependent layers; restore from backup may lose hours of data |
| L2 | Storage Abstraction | Cluster topology change | Platform Lead + Domain SRE sign-off | Minutes to hours | Key redistribution causes cache miss storm → L1 saturation → cascading latency to all consumers |
| L3 | Data Services | Breaking API contract change | Tech Lead + all consuming team leads | Hours (coordinated multi-team rollback) | Consumer services cannot function on broken contract; rollback requires coordinated multi-team action |
| L4 | Middleware / Integration | Kafka consumer offset reset OR queue deletion | Messaging Platform Lead + Engineering Director | No automated rollback | Data loss or massive reprocessing; downstream duplicate events or missed events may corrupt business state |
| L5 | Application Services | Dependency upgrade with API breaking change | Domain Tech Lead | Minutes (image rollback) | Behavior changes may not be caught by tests; downstream consumers experience unexpected failures |
| L6 | API Gateway / Edge | WAF rule in block mode | Platform Engineering Director + Security Lead | Seconds (rule disable) | False positive blocks all legitimate traffic for affected route; external customers impacted immediately |
| L7 | Presentation / UI | Frontend bundle deploy (full replacement) | Frontend Tech Lead | Minutes (re-deploy + CDN invalidate) | Bad bundle = blank page for all users; highest visibility failure; recovery requires CDN propagation time |
