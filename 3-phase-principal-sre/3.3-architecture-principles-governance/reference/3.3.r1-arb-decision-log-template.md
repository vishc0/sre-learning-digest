# ARB Decision Log

**The ARB Decision Log is the authoritative record of every decision made through the Architecture Review
Board process.** It is the first place anyone should look when asking:
- "Has the ARB reviewed this technology before?"
- "What did the ARB decide about X?"
- "Is this exception already approved, or does it need to go back to the ARB?"
- "What conditions were attached to the approval of System Y?"

This log is permanent. Decisions are never removed — they are superseded by later entries if reversed or
updated. The full history of how architectural thinking has evolved is preserved here.

**Log Owner**: ARB Chair (VP Operations)
**Maintained by**: Architecture Team
**Access**: All engineers and managers. Audit and compliance teams have read access.
**Update cadence**: Within 5 business days of each ARB session.

---

## How to Search This Log

**By technology**: Use Ctrl+F / browser search for the technology name (e.g., "Kafka", "MongoDB", "OpenTelemetry").

**By system**: Search for the system or service name (e.g., "Customer Profile Service", "Order Management").

**By date range**: Entries are sorted newest-first within the year. Scan the entry headers (ARB-YYYYMMDD-NNN) for the date.

**By decision type**: Scan the **Decision** field in each entry for "Approved", "Approved with Conditions", "Rejected", or "Exception Approved."

**By ADR**: If you have an ADR number, search for "ADR-XXXX" to find the ARB entry that corresponds to it.

**For exceptions specifically**: Search for "Exception" — all exception approvals contain this word in the Decision field.

## Log Format

Each entry follows this structure:

```
### ARB-YYYYMMDD-NNN
**Title**: [Short descriptive title — technology + domain or decision type]
**Date**: YYYY-MM-DD
**Submitting Team**: [Team name and domain]
**Tier**: Tier 1 | Tier 2
**Decision**: Approved | Approved with Conditions | Rejected | Exception Approved | Deferred
**Summary**: [2-4 sentences — what was proposed, what was decided, why]
**Conditions**: [List of conditions, each with owner and deadline — N/A if no conditions]
**ADR Reference**: [ADR-NNNN — link to the finalized ADR, or "None" for rejections]
**ARB Members Present**: [Names/roles]
```

Conditions are tracked until closed. Overdue conditions are reported to the VP at each quarterly governance review.

### ARB-20260301-003
**Title**: Exception — Use of MongoDB for Customer Profile Service Document Storage
**Date**: 2026-03-01
**Submitting Team**: Customer Experience Engineering
**Tier**: Tier 1
**Decision**: Exception Approved with Conditions
**Summary**: Approved a time-bounded exception for MongoDB 7.0 LTS as the document store for the Customer Profile Service. The Customer Experience Engineering team evaluated PostgreSQL (the Adopt-ring standard) and determined that the flexible, schema-optional customer profile document structure creates unacceptable migration overhead with a fixed relational schema. The ARB reviewed the PostgreSQL evaluation methodology and agreed the schema flexibility requirement is legitimate. Exception is time-bounded to 24 months.
**Conditions**:
- [ ] Exception expires 2028-03-01. Renewal requires a new exception request with documented operational evidence (incident rate, performance data, schema evolution history) demonstrating continued necessity — Owner: CX Engineering Lead
- [ ] Full observability instrumentation required before go-live: Datadog APM for query performance, Splunk for audit logging, DLQ alerting for any async write failures — Owner: CX Engineering SRE
- [ ] MongoDB version pinned to 7.0 LTS. Any upgrade to a new major version requires ARB notification and Platform Team review of the upgrade plan — Owner: Platform Engineering
- [x] MongoDB added to Technology Radar Trial ring with exception annotation — Completed 2026-03-08
- [ ] Architecture documentation (C4 model) updated to reflect MongoDB in the Customer Profile Service — Owner: CX Engineering Lead, Due: 2026-04-01
**ADR Reference**: ADR-0039 (finalized 2026-03-15)
**Exception Expiry**: 2028-03-01
**ARB Members Present**: VP Operations, Principal Engineer (CX Domain), Data Architect, Security Architect, Platform Architect, SRE Lead

### ARB-20260115-002
**Title**: Reject — Custom Pub-Sub Implementation for Real-Time Pricing Service
**Date**: 2026-01-15
**Submitting Team**: Pricing Engineering
**Tier**: Tier 2
**Decision**: Rejected with Alternative Recommendation
**Summary**: Rejected the proposed custom pub-sub implementation for the Real-Time Pricing Service event distribution pipeline. The Pricing Engineering team proposed building a custom lightweight pub-sub mechanism citing "insufficient throughput" from RabbitMQ and "operational overhead" of Kafka. The ARB reviewed the submitted benchmarks and found they were run against a misconfigured RabbitMQ instance (publisher confirms disabled, single queue, no prefetch tuning). Corrected benchmarks reviewed by the ARB's principal engineer show RabbitMQ with priority queues meets the stated latency requirement (sub-100ms p99 at 50K events/minute). Introducing a custom pub-sub creates a novel system with no operational precedent, no community support, and no on-call expertise — a Design for Failure and Observability violation.
**Conditions**: N/A (Rejection)
**Alternative Recommendation**: Use RabbitMQ with priority queues for the stated use case. Platform Engineering will provide a reference architecture and a pre-configured RabbitMQ vhost in the staging environment. Pricing Engineering should resubmit with a RabbitMQ design if after implementation the benchmarks show RabbitMQ is genuinely insufficient at production scale.
**ADR Reference**: None — decision rejected; no ADR required
**Follow-up**: Pricing Engineering scheduled for resubmission at ARB-20260201 with a RabbitMQ-based design. Platform Engineering assigned to support the implementation.
**ARB Members Present**: VP Operations, Principal Engineer (Pricing Domain), Principal Engineer (Platform), Architecture Team Lead, SRE Lead

### ARB-20251210-005
**Title**: Approve Migration from Self-Managed Kafka EC2 to Amazon MSK
**Date**: 2025-12-10
**Submitting Team**: Platform Engineering (on behalf of all Kafka consumers)
**Tier**: Tier 1
**Decision**: Approved
**Summary**: Approved the migration of all production Kafka workloads from self-managed Kafka 2.6 on EC2 to Amazon MSK 3.6. Self-managed Kafka has accumulated three active CVEs (CVE-2024-27309 CVSS 7.5, two additional medium-severity CVEs), and the Platform Team spends approximately 4 hours per week on manual broker operations — patching, rebalancing, and capacity management. MSK eliminates broker operations overhead, provides automatic minor version patching, and enables the CVE remediation path. The migration approach (parallel consumer groups, 2-week validation window, domain-by-domain cutover) was reviewed and approved. Self-managed Kafka simultaneously moved to Hold ring on the Technology Radar.
**Conditions**:
- [x] MSK cluster provisioned and validated in non-production — Completed 2026-01-15
- [x] Schema Registry migrated to AWS Glue Schema Registry — Completed 2026-02-01
- [ ] Order Domain migration to MSK — Owner: Order Engineering Lead, Due: 2026-03-31
- [ ] Billing Domain migration to MSK — Owner: Billing Engineering Lead, Due: 2026-04-30
- [ ] EC2 Kafka brokers decommissioned after all domains migrated — Owner: Platform Engineering, Due: 2026-05-31
**ADR Reference**: ADR-0031 (finalized 2025-12-22)
**ARB Members Present**: VP Operations, Principal Engineer (Platform), Principal Engineer (Order Domain), Principal Engineer (Billing Domain), Security Architect, SRE Lead

### ARB-20251105-004
**Title**: Approve Istio Service Mesh for Production mTLS and Traffic Management
**Date**: 2025-11-05
**Submitting Team**: Platform Engineering
**Tier**: Tier 1
**Decision**: Approved with Conditions
**Summary**: Approved Istio 1.20 as the production service mesh for all EKS clusters, implementing mTLS for all intra-cluster service communication, traffic management (circuit breaking, retries, timeouts), and observability (distributed tracing via Envoy + OTEL). The proposal was motivated by the Zero Trust architecture principle — the existing implementation relied on network perimeter security with no service-to-service authentication. After three quarters of Trial across two non-production clusters, the ARB determined operational maturity was sufficient for production adoption.

- [x] mTLS enforced in STRICT mode for all production namespaces — Completed 2026-01-31

- [x] Istio runbook published (circuit breaking, mTLS debugging, traffic weight shifting) — Completed 2025-12-01
- [x] Exception: Three legacy services running on Java 8 with incompatible cert libraries are granted a 90-day exception from STRICT mTLS mode — expired 2026-02-05; all three services migrated
**ADR Reference**: ADR-0024 (finalized 2025-11-20)
**ARB Members Present**: VP Operations, Principal Engineer (Platform), Security Architect, SRE Lead, Principal Engineer (Order Domain), Principal Engineer (CX Domain)

### ARB-20250901-001
**Title**: Adopt ArgoCD as Standard GitOps Delivery Platform
**Date**: 2025-09-01
**Submitting Team**: Platform Engineering
**Tier**: Tier 1
**Decision**: Approved
**Summary**: Approved ArgoCD as the standard GitOps delivery platform for all EKS workloads, replacing manual kubectl apply workflows and Helm-driven CI/CD pipelines. The proposal standardizes deployment across all domains, enables audit-quality deployment history (every deployment traceable to a Git commit), and enforces the Immutable Infrastructure principle by removing direct cluster access from CI pipelines. ArgoCD ApplicationSets will manage multi-cluster, multi-environment deployments from a single configuration. Manual kubectl apply to any production namespace after ArgoCD adoption is a policy violation (POL-001 updated accordingly).
**Conditions**:
- [x] All Tier 1 services migrated to ArgoCD management by 2026-03-01 — Completed
- [x] Production cluster RBAC updated — production namespace write access removed from CI service accounts — Completed 2025-11-01
- [x] ArgoCD runbook published — Completed 2025-10-01
**ADR Reference**: ADR-0018 (finalized 2025-09-15)
**ARB Members Present**: VP Operations, Principal Engineer (Platform), SRE Lead, Principal Engineer (Order Domain)
