# Architecture Principles — Quick Reference

*Carry this into any architecture conversation.*
*These constraints apply to every system, every decision, every team.*
*When a proposal conflicts with a principle, the principle wins — or an exception is documented.*

---

## The 12 Foundational Principles

| # | Principle | One-Line Statement | Operational Implication | Anti-Pattern to Reject on Sight |
|---|-----------|-------------------|------------------------|--------------------------------|
| 1 | **Design for Failure** | Every component will fail; design so failures are isolated and recoverable | Blast radius is bounded by design, not by luck | Shared database connection pool across all services; one slow query starves them all |
| 2 | **Loose Coupling, High Cohesion** | Minimize inter-component dependencies; maximize intra-component consistency | A change to Service A does not require a deployment of Service B | Microservices that share a database schema — schema changes require coordinating 6 teams |
| 3 | **Defense in Depth** | Multiple independent security controls at every layer | WAF failure does not equal open access — there is still auth, network policy, and app-layer validation | "We're on the internal network so we don't need authentication" |
| 4 | **Infrastructure as Code** | All infrastructure defined, versioned, and deployed as code; zero manual provisioning | Environments are reproducible; drift is detectable; a new environment takes minutes not weeks | EKS node groups configured by hand during an incident; no one knows what was changed |
| 5 | **Immutable Infrastructure** | Deployed artifacts are never modified in place; replace rather than patch | No configuration drift; rollback is a redeployment; every environment is verifiably identical | SSH into a production pod to apply a hotfix; the fix lives in one running container and nowhere else |
| 6 | **Observability by Design** | Every system emits structured logs, metrics, and traces from day one — not after the first incident | Any production incident can be diagnosed from telemetry alone, without SSH access | "We'll add monitoring later"; first incident takes 4 hours because there are no metrics |
| 7 | **API-First** | All integrations through documented, versioned APIs; no cross-service direct database access | Service owners control their own change pace without coordinating schema migrations across teams | Analytics team queries the operational database directly — a schema change breaks their reports |
| 8 | **Zero Trust** | No network location is trusted by default; all requests authenticated and authorized regardless of source | Lateral movement in a breach is contained; a compromised internal service cannot reach everything | Internal service traffic trusted implicitly because "it's behind the firewall" |
| 9 | **Stateless Application Tier** | Application services hold no durable state; all state lives in the data tier | Pods can be killed and replaced without data loss; horizontal scaling is trivial | User session state stored in Tomcat in-memory — a restart loses every active session |
| 10 | **Eventual Consistency as Default** | Async/eventual is the default; synchronous strongly-consistent operations are the explicit, justified exception | System remains operational under partial failure; one slow service does not block the entire request chain | Synchronous call chain across 6 microservices for one user-facing action — one timeout fails everything |
| 11 | **Shift Left** | Testing, security, and observability instrumentation happen during development, not at deployment | Defects found in development cost 10× less than defects found at the CAB gate or in production | Security review happens 48 hours before a release; findings cause delays or are waived |
| 12 | **Data Sovereignty** | Each service owns its own data; no other service accesses it except through the owner's API | Schema changes, data migrations, and access controls are owned by one team — no cross-team coordination required for schema work | Shared operational database as the integration layer — any team's schema change is a cross-org incident |

## Principles in Conflict — How to Decide

> When two principles point in different directions, use this table. These are not arbitrary rules —
> each resolution reflects which failure mode is more expensive in an operational context.

| Tension | Decision Rule | Operational Reasoning | Example |
|---------|--------------|----------------------|---------|
| Design for Failure vs. Eventual Consistency | Default to eventual consistency. Choose synchronous consistency only when data loss has a direct legal or financial consequence. | An eventually consistent system degrades gracefully. A synchronously coupled system fails completely when one node slows down. | Payment confirmation: synchronous (financial consequence of duplicate or missed payment). Notification delivery: eventual. Inventory decrement: eventual with idempotency. |
| Immutable Infrastructure vs. Emergency Response | Immutable wins. The fix goes through the pipeline. | An SSH hotfix that works tonight creates an undocumented snowflake that breaks at the next deployment. The 20-minute pipeline delay is cheaper than the next incident. | Fix the container image, push through CI/CD, deploy. Never hotfix a running container. |
| API-First vs. Performance | API-First wins for cross-service access. Direct access within a service boundary is acceptable. | The performance argument for direct DB access is almost always overstated. The coupling cost is always underestimated. | Service-to-service: always API. Module-to-module within the same service: flexible. Read replicas owned by the service and exposed via an API: acceptable pattern. |
| Shift Left vs. Timeline Pressure | Shift Left wins. | A security review skipped at development costs 10× more at CAB and 100× more in production. Timeline pressure is not a risk mitigation. | Engage security in sprint planning, not 48 hours before the release. If security review finds a critical issue at CAB, the release moves — that outcome is always the team's fault, not security's. |
| Zero Trust vs. Developer Velocity | Define the trust boundary clearly. Service mesh (Istio/Linkerd) enables Zero Trust without per-request developer overhead. | mTLS within the cluster and explicit auth for all external calls achieves Zero Trust without manual credential management per call. The overhead argument is a tooling problem, not a principle problem. | mTLS via Istio within the cluster. Explicit JWT/API key auth for all calls crossing the cluster boundary. Internal services never trust the caller's claimed identity without verification. |
| Loose Coupling vs. Operational Simplicity | Coupling wins at small scale; decoupling wins at team scale. | When two teams cannot deploy independently, you have the wrong coupling level. When one team manages a system independently, tight internal coupling is acceptable. | Monolith within a single team: acceptable. Same monolith after it's owned by three teams: decompose. |

## Principle Application by System Tier

> Not every principle has equal weight for every tier. Use this as a starting screen — all principles
> apply to all tiers, but these are the non-negotiables at each tier.

| Tier | Non-Negotiable Principles | Reason |
|------|--------------------------|--------|
| **Tier 1** (Revenue-critical, customer-facing) | All 12 — no exceptions without ARB approval | Failure at Tier 1 has direct customer and revenue impact |
| **Tier 2** (Internal operational) | 1 (Failure), 3 (Defense), 4 (IaC), 6 (Observability), 8 (Zero Trust) | Operational integrity required; some coupling and consistency tradeoffs acceptable |
| **Tier 3** (Supporting tools, back-office) | 3 (Defense), 4 (IaC), 6 (Observability) | Minimum viable governance; speed prioritized over architectural purity |

## Exception Process — Summary

> Any exception to any principle at any tier requires:

1. **Written exception request** — submitted using the Architecture Review Request Template, Standards Exception section
2. **ARB review and approval** — Tier 1 exceptions always require ARB; Tier 2 may allow Architecture Team sign-off
3. **VP sign-off** — required for all Tier 1 system exceptions
4. **Time-bounded exception** — maximum 12 months; renewable with documented operational evidence
5. **Documented in ADR** — the accepted exception appears in the Principles Alignment table with "Exception Required" status and a link to the ARB Decision Log entry

**No verbal exception. No informal exception. No "we'll document it later."**

An exception that is not documented is a violation.

## ARB Red Flags — Stop the Conversation

> If any of these phrases appear in an architecture proposal or review, stop and address them before
> proceeding. These are signals that a principle is about to be violated, often with full awareness.

| Phrase Heard | Principle Violated | Response |
|-------------|-------------------|---------|
| "We'll add monitoring later" | Observability by Design (#6) | No launch without observability. Monitoring is not a post-launch activity. |
| "It's internal traffic so security doesn't apply" | Zero Trust (#8) | Zero Trust applies to all traffic regardless of source. Internal networks are not trusted. |
| "The other service can just read our database directly" | API-First (#7) + Data Sovereignty (#12) | No. All cross-service data access goes through the owner's API. Always. |
| "We'll fix the architecture after we ship" | Design for Failure (#1) + all principles | This is how design debt is born. Defer the launch, not the architecture work. |
| "Everyone's been doing it this way for years" | All principles | Tribal patterns are not standards. Document and standardize the pattern, or change it. |
| "We don't have time for an architecture review" | Governance | The ARB process exists precisely because time pressure causes the most expensive mistakes. |
| "It's just a temporary solution" | All principles | Temporary solutions become permanent at the moment they handle production traffic. |
| "We can scale it later when we need to" | Design for Failure (#1) + Stateless (#9) | Retrofitting scalability into a stateful, tightly-coupled system costs 5× designing it in. |
| "The performance overhead isn't worth the abstraction" | API-First (#7) | The coupling cost always exceeds the performance cost. Benchmark both before making this claim. |
| "We'll handle security in the next sprint" | Shift Left (#11) + Zero Trust (#8) | Security work deferred to "next sprint" is security work that never ships. |

## Quick Vocabulary — Principle-to-Pattern Mapping

> For each principle, the patterns that implement it in a T-Mobile/enterprise context.

| Principle | Implementing Patterns | Anti-Patterns |
|-----------|----------------------|---------------|
| Design for Failure | Circuit breakers (Hystrix/Resilience4j), bulkheads, retry with backoff, graceful degradation, dead-letter queues | No circuit breakers; synchronous fan-out; shared thread pools |
| Loose Coupling | Async messaging (RabbitMQ, Kafka), event-driven architecture, API contracts, semantic versioning | Shared databases, direct service-to-service synchronous calls for non-critical paths |
| Defense in Depth | WAF + network policy + mTLS + app-layer auth + data encryption at rest and in transit | Single perimeter firewall; implicit trust on internal traffic |
| Infrastructure as Code | Terraform for infrastructure, Helm for Kubernetes manifests, GitOps (ArgoCD/Flux), Atlantis for Terraform PRs | ClickOps; Ansible playbooks run manually; "golden AMI" updated by hand |
| Immutable Infrastructure | Container images, baked AMIs, Helm chart deployments, blue-green and canary deployments | In-place upgrades; configuration drift; SSH access to production nodes |
| Observability by Design | Structured JSON logs → Splunk, Prometheus metrics → Datadog, OpenTelemetry traces → Datadog APM, SLO dashboards | Unstructured logs; no distributed tracing; metrics added after incidents |
| API-First | REST/gRPC with OpenAPI/Protobuf schemas, API versioning, developer portals (Kong, Apigee), contract testing | Direct DB access, shared libraries as integration points, RPC-style tight coupling |
| Zero Trust | mTLS (Istio service mesh), JWT/OAuth2 for all callers, network policies, Vault for secrets, SPIFFE/SPIRE | Implicit network trust, hardcoded credentials, IP allowlisting as primary auth |
| Stateless Application Tier | Stateless pods, Redis for session state (if needed), database-backed state, sticky sessions avoided | Session state in JVM heap, file system state on pods, sticky load balancing |
| Eventual Consistency | Kafka or RabbitMQ for event propagation, outbox pattern, saga pattern for distributed transactions | Synchronous 2PC across services, distributed transactions, blocking cross-service calls |
| Shift Left | SAST in CI (Sonar, Semgrep), dependency scanning (Snyk/Dependabot), contract tests, threat modeling in sprint 0 | Security review at CAB, no unit tests until QA phase, observability added post-go-live |
| Data Sovereignty | Service owns its own DB/schema, event publishing for data sharing, data contracts, CQRS for read/write separation | Shared operational DB, ETL pulling directly from service DBs, no data ownership defined |
