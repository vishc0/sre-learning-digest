## 1. What a Technology Standard Is

A technology standard is an approved technology choice that teams must use — or formally request an exception — for a defined category of problem. It is not a suggestion, a preference, or a default. The mandate is the governance act.

The distinction matters operationally. Saying "Kafka is good for event streaming" is a technical opinion. Saying "Kafka is our standard for event streaming — use it or request a formal exception" is a governance statement. The first can be ignored without consequence. The second cannot, because the exception process creates accountability, creates a record, and creates a deliberate decision instead of drift.

Standards reduce decision fatigue at the team level. A team that does not have to evaluate which message broker to use — because the organization has already made that decision — ships faster than one that re-evaluates the landscape every project. The cognitive load of technology selection is non-trivial. Teams that spend three weeks comparing Kafka, RabbitMQ, Pulsar, and SQS have spent three weeks not building the system. Standards recover that time.

Standards create operational leverage at the platform level. Fewer distinct technologies means fewer distinct on-call experts required across the organization. When every event-streaming workload runs on Kafka, any senior SRE can support any team's Kafka incident — because they're all running the same thing. When five teams run five different message brokers, cross-team incident support collapses. You need five sets of runbooks, five capacity planning models, five different alert tuning approaches.

A standard is not "the best tool in the abstract." It is the best tool given: operational cost, organizational expertise depth, licensing economics, vendor viability, and strategic fit. Standards can be wrong. A standard adopted in 2020 may be the wrong choice in 2025 because the vendor was acquired, the technology stagnated, or a superior alternative reached production maturity. This is why standards have a lifecycle and a review process — not because governance is uncertain, but because technology is not static.

## 2. Standard Categories

The following table represents the current approved state. **Approved** means default — use this. **Trial/Evaluate** means use on a designated evaluation project with a reporting commitment to the ARB. **Not Approved (Hold)** means do not start new work; if you are on this technology, plan migration.

| Category | Approved | Trial / Evaluate | Not Approved (Hold) |
|---|---|---|---|
| **Languages & Runtimes** | Python 3.10+ (data, automation, scripting), Java 17+ LTS (core services), Go 1.21+ (infrastructure tooling), Node.js 20 LTS (API gateway, BFF layers) | Rust — for performance-critical, memory-safety-constrained components | PHP, Ruby, Python 2.x, Java 8 (security EOL) |
| **Messaging & Event Streaming** | Kafka (high-throughput event streaming, durable pub-sub), RabbitMQ (task queues, point-to-point async, service-to-service messaging with guaranteed delivery) | Apache Pulsar — evaluate for multi-tenant, geo-distributed use cases where Kafka operational model is limiting | Custom pub-sub implementations, ActiveMQ (existing deployments only — no expansion), homegrown queue implementations |
| **Databases — Relational** | PostgreSQL 15+ (primary RDBMS for all new services), Oracle (legacy only — no new deployments, no new schemas on existing instances without ARB approval) | CockroachDB — evaluate for geo-distributed, globally consistent RDBMS requirements | MySQL for new service deployments, SQLite in any production workload |
| **Databases — Non-Relational** | Redis (caching, session state, rate limiting), Elasticsearch / OpenSearch (search, log indexing, analytics), MongoDB (document store — approved for specific use cases with ARB sign-off on data model) | ScyllaDB — evaluate for time-series at scale where Redis TTL-based approach is insufficient | Cassandra for new deployments (organizational expertise gap — incidents cannot be owned cross-team) |
| **Container & Orchestration** | Kubernetes / EKS (all production workloads), Docker (build pipelines, local development) | AWS Fargate — evaluate for serverless container workloads where cluster management overhead is the dominant cost | Raw VM deployment for new services, docker-compose in any production or staging environment |
| **Observability** | Splunk (log aggregation, SIEM, alerting, correlation), Datadog (APM, infrastructure metrics, synthetic monitoring), Prometheus + Grafana (Kubernetes cluster and pod metrics) | OpenTelemetry — evaluate as the instrumentation standard for distributed tracing across all services | Custom logging agents deployed by individual teams, homegrown dashboards for production monitoring, rolling your own alerting outside Splunk |
| **API Gateway & Service Mesh** | Kong or AWS API Gateway (external API traffic), Istio (internal service mesh for mTLS, traffic management, circuit breaking) | Envoy standalone — for specific use cases where Istio's control plane overhead is justified to isolate | Direct service-to-service HTTP without an authentication layer between services |
| **CI/CD** | GitHub Actions (source of truth for all build and test pipelines), ArgoCD (GitOps deployment — the single deployment mechanism for Kubernetes workloads) | Tekton — evaluate for on-premises pipeline requirements where GitHub Actions is unavailable | Manual deployment scripts, Jenkins (existing pipelines are migration targets — no new Jenkins pipelines) |
| **Infrastructure as Code** | Terraform (all cloud resource provisioning), Helm (Kubernetes package management and release management), AWS (primary cloud provider), Azure (secondary — Teams, Entra ID, O365 dependencies) | Pulumi — evaluate for IaC scenarios requiring complex imperative logic that HCL cannot express cleanly | AWS CloudFormation for new infrastructure work, manual console provisioning of any resource that will be permanent |

**Reading the table**: every new service starts from the Approved column. Selecting anything in Trial requires a statement in the ADR about what will be evaluated and when findings return to the ARB. Selecting anything in Not Approved requires a formal standards exception (see Section 5).

## 3. The Technology Radar Model

The Technology Radar, originated by ThoughtWorks, is the communication mechanism that makes the standards table above legible to engineers without requiring them to read governance documents. It is a visual, self-service representation of where any technology stands.

**The four rings:**

- **Adopt**: Use this. The organization has enough operational confidence, production evidence, and expert depth. This is the default choice for new work in this category. No additional justification required.
- **Trial**: Use this on a project with the explicit intent to evaluate and report back. If you're using a Trial technology, you are running a structured experiment — not just selecting a preference. You owe the ARB an evaluation report within 90 days.
- **Assess**: Research this. Attend a conference session, read the documentation, build a throwaway proof of concept. Do not use in a project intended for production. Flag to the ARB if a proof of concept reveals something worth escalating to Trial.
- **Hold**: Do not start new work with this technology. If you are on it, plan migration. Existing deployments may continue under an exception for a defined sunset window — they do not get new features built on this technology.

**The four quadrants:**

- Languages & Frameworks
- Platforms (cloud services, databases, messaging infrastructure)
- Tools & Techniques (build tooling, testing approaches, operational practices)
- Data Management (data stores, pipelines, governance tooling)

**How the radar is updated:**

The radar is not a static artifact. It is a living governance document reviewed on a structured cadence:

- **Quarterly**: ARB reviews technology owner proposals for ring movement. Technology owners (named engineers responsible for a specific technology domain) submit proposals with evidence — production incident trends, vendor EOL notices, evaluation reports from Trial projects.
- **Annual**: Full radar refresh. All technologies are reviewed. Items that have been on Hold for more than 24 months with no migration progress trigger a formal escalation to VP.
- **Ad hoc**: Security vulnerabilities with CVE scores ≥ 9.0 can trigger immediate movement to Hold. This bypasses the quarterly cycle and is a VP-level decision with ARB notification.

**The radar as self-service governance:**

Engineers should be able to answer "where does this technology stand?" without asking the architecture team. The radar makes governance visible. When a new engineer proposes using Cassandra because they used it at their previous company, a quick radar check shows it is on Hold with a note: "expertise gap — incidents cannot be owned cross-team." That answer is given by the governance artifact, not by a meeting.

> **What the VP decides**: Which technologies are approved standards is a strategic decision, not a technical one. The CTO and VP Operations own the Adopt ring. Staff engineers own the evidence gathering. The ARB owns the promotion/demotion decision. The VP signs off on any movement in the Adopt ring.

## 4. How Standards Reduce Operational Burden

The operational leverage from standards is not theoretical. It is measurable, and it is the argument a VP makes to teams that resist standardization as "constraining innovation."

**The N+1 tools problem**: If ten teams each chose their own message broker based on individual preference or prior experience, the organization now needs ten different on-call expert pools, ten sets of runbooks, ten capacity planning models, ten alert thresholds tuned separately. At 3 AM, the incident is in Team 7's custom pub-sub implementation. No one from Teams 1-6 can help because they have never touched it. The VP is now explaining to the CTO why recovery took four hours.

**Cross-team incident support**: When a Kafka consumer lag incident occurs on any team's service, any senior SRE on any other team can parachute in and be operational within minutes. The Kafka topology is the same. The monitoring dashboards use the same labels. The runbook is the same. Consumer lag means consumer lag — the diagnosis path is identical regardless of which team owns the consumer. This is the operational dividend from standardization.

**Onboarding time**: Engineers joining a team with standardized tooling reach full productivity faster. There is a known playbook for standing up a new service, a known set of Helm charts, a known monitoring configuration. Engineers who have onboarded to one team can onboard to any team within days rather than weeks, because the underlying platform is the same.

**Audit and compliance surface**: Demonstrating security and compliance controls is dramatically easier when the technology surface is bounded. "We run these nine approved technologies, and here is how each one meets our security control requirements" is a one-day audit conversation. "It depends on the team" is a three-week audit. SOC 2, PCI-DSS, and NIST 800-53 auditors want a bounded surface. Standards create that surface.

**Procurement leverage**: Enterprise licensing negotiated for the whole organization is structurally cheaper than individual team subscriptions. Confluent platform licensing negotiated for 200 engineers at the organizational level versus each team negotiating separately — the difference is material at T-Mobile scale. This is the CFO's argument for standardization.

**Real-world evidence — T-Mobile context**: Centralizing on RabbitMQ and Kafka across the middleware layer, retiring four custom messaging implementations (two ActiveMQ clusters, one custom pub-sub service, one TIBCO EMS deployment that had outlived its support window), produced measurable results:
- On-call surface for messaging incidents reduced by approximately 30% — fewer technologies, fewer unique failure modes
- Cross-team runbook sharing became possible — any SRE could work any messaging incident
- Vendor contract consolidation reduced per-seat costs significantly
- New engineer onboarding to messaging patterns dropped from 3 weeks to under 1 week because there was one thing to learn

## 5. The Standards Exception Process

Standards are not optional, but they are not absolute. A formal exception process exists to handle legitimate cases where no approved technology meets a specific requirement. The exception process is the valve that prevents standards from becoming a blockade — but it creates accountability that prevents standards from being casually ignored.

**Step-by-step process:**

1. **Identify the need**: Team identifies a requirement that no approved technology can meet. This must be a capability gap — not a preference for a technology the team used at a previous company.

2. **Complete the exception request**: Team writes a structured request covering: (a) the specific capability requirement that approved technologies cannot meet, (b) evidence that each approved technology was evaluated, (c) the proposed alternative technology, (d) an operational readiness plan (named owner, monitoring approach, on-call coverage, runbook commitment), (e) a risk assessment, and (f) a sunset commitment if this is a bridge technology.

3. **Architecture team initial review**: The architecture team reviews the request within one week. This review checks completeness (is the operational readiness plan real?) and whether the architecture team sees an approved technology path the team missed. First-pass rejections from this review return with a specific gap identified.

4. **ARB review**: At the next scheduled ARB session, the architecture team presents the request. The proposing team attends to answer questions.

5. **VP decision**: For Tier 1 exceptions (technologies affecting multiple teams or critical systems), the VP makes the final call. For Tier 2 exceptions (single-team scope), the ARB vote is sufficient.

6. **Outcome and conditions**: Approved exceptions receive the technology into the Trial ring for the requesting team. The team commits to a 90-day evaluation report. The exception is time-bounded at 12 months — after which the team must either: demonstrate evidence for promotion to the Adopt ring, renew the exception with updated evidence, or migrate to an approved technology.

**What the exception request must contain — no exceptions to the exception process:**

- **Why no standard technology meets the requirement**: This is the hardest section to write well. "We prefer this" is not acceptable. "We require X-tier CDC capabilities that PostgreSQL's logical replication does not provide at our volume, as demonstrated by the performance test results attached" is acceptable.
- **Operational readiness plan**: Who will own this technology? How will it be monitored? Where are the dashboards? Who is the on-call contact? If there is no named owner with demonstrated expertise, the exception is denied regardless of technical merit — because a technology with no operational owner is a 3 AM liability.
- **Sunset commitment**: If this is a bridge technology while a migration is in progress, what is the migration plan and what is the target date? Exceptions without a sunset plan have a higher bar for approval.
- **Risk acknowledgment**: What happens if this decision proves wrong in 18 months? What is the rollback path? What is the migration cost?

## 6. Standards Lifecycle

Standards are not permanent. They enter, evolve, and exit the approved list on a governed lifecycle.

**Promotion path: Assess → Trial → Adopt**

Movement from Assess to Trial requires: a technology owner volunteering to sponsor the evaluation, at least one team willing to run a structured Trial, and ARB approval of the evaluation criteria and timeline.

Movement from Trial to Adopt requires: production evidence from at least two Trial projects, an operational readiness review (monitoring, runbooks, on-call training), ARB vote, and VP sign-off for languages, databases, and messaging technologies (these have the broadest operational impact).

**Deprecation path: Adopt → Hold**

Deprecation is triggered by: vendor end-of-life, unresolved security vulnerabilities with no vendor patch commitment, a superior replacement reaching Adopt status, or a demonstrated organizational expertise gap that makes operational support untenable.

Deprecation communication minimum requirements:
- 12-month deprecation window (24 months preferred for deeply embedded technologies)
- Migration guide published before deprecation announcement — teams cannot be told to migrate without a documented path
- Exception process open to teams that cannot migrate within the window — this is a formal exception, not a hallway conversation
- VP communication to engineering leadership explaining why the technology is being deprecated and what the replacement is

**The "standards graveyard" anti-pattern:**

Standards documented three years ago and never enforced are worse than no standards. Teams use technologies listed as "Not Approved" because no one checks. New services are built on Hold technologies because no one says anything during code review. This is governance theater — the appearance of governance without the substance.

The graveyard forms when:
- There is no audit mechanism (no one checks what technologies are actually in use)
- There are no consequences for ignoring standards (exception process exists but is never enforced)
- The VP treats standards as the architecture team's responsibility rather than their own

The VP's job is to close the gap between documented standards and actual practice. This means: quarterly audits of technology usage against the approved list, visible consequences for teams that bypass the exception process, and personal ownership of the enforcement culture. Standards that the VP does not enforce are preferences. Standards that the VP enforces are governance.

> **What happens without standards**: The 3 AM incident where a critical service is down and no one on the on-call rotation has ever touched the technology running it. The postmortem finding that reads "we did not have organizational expertise in this technology." This is recoverable once. It becomes a systemic failure when it describes your architecture broadly.

## Summary

| Concept | Operational Meaning |
|---|---|
| Technology standard | Approved choice — use it or formally request an exception |
| Radar ring: Adopt | Default for new work — no justification required |
| Radar ring: Trial | Use with a structured evaluation commitment and reporting obligation |
| Radar ring: Assess | Research only — not for production use |
| Radar ring: Hold | No new deployments — migrate away when practical |
| Standards exception | Formal process — not a workaround, not a hallway conversation |
| Standards lifecycle | Quarterly review — standards are revisable by design |
| VP's accountability | Close the gap between documented standards and actual practice |
