## Response Time Requirements by Risk Level

| Risk Level | Detection → Plan | Plan → Execution Complete | VP Notification |
|-----------|-----------------|--------------------------|----------------|
| Critical | 48 hours | 90 days | Required within 48 hours |
| High | 30 days | 180 days (6 months) | Required if plan not started at 30 days |
| Medium | 90 days | 12 months | Required if not in planning cycle |
| Low | Next planning cycle | 24 months | Not required |
| Exception | N/A — accepted | Per exception approval | Required at exception approval |

## Exception Approval Requirements

An EOL exception is a policy violation with explicit acceptance. The following are required before any exception is logged:

1. Written risk acceptance statement signed by VP of Operations
2. Compensating controls documented (network isolation, WAF rules, enhanced monitoring)
3. Hard expiry date — no rolling exceptions; each must be re-approved at expiry
4. Upgrade plan attached even if deferred — "upgrade path unknown" is not an acceptable exception justification
5. Quarterly re-review of all exceptions; any exception older than 6 months without progress escalates to Engineering Director

## Component Type Definitions

| Type | Examples |
|------|---------|
| Runtime | Java JDK, Python interpreter, Node.js runtime, .NET CLR |
| Dep | Library, framework, vendor SDK (Spring Boot, Kafka client, MuleSoft connector) |
| Infra | Kubernetes version, container runtime (containerd, Docker), Linux kernel |
| OS | Ubuntu, RHEL, Amazon Linux, Windows Server |

For middleware platforms (RabbitMQ, Kafka, TIBCO BW, MuleSoft runtime), use Type = Dep if it is a client library embedded in the application; use Type = Infra if it is the broker/platform the application connects to. Log both separately if both are at risk.
