# Application Health Scorecard

A per-application health scorecard that produces a composite score and lifecycle recommendation. Completable in 30 minutes from available data sources: application registry, Splunk, service mesh telemetry, and a brief team interview.

---

```
APPLICATION HEALTH SCORECARD
=============================
Review Date: ____________
Reviewer: ____________
Next Review Due: ____________ (quarterly for At Risk/Critical, semi-annual for Healthy/Watch)

SECTION 1: APPLICATION IDENTITY
Application Name: ____________
Business Owner: ____________
Technical Owner: ____________
Application Tier: [ ] T1  [ ] T2  [ ] T3  [ ] T4
Current Lifecycle Stage: [ ] Inception [ ] Active Development [ ] Mature/Stable [ ] Legacy [ ] Sunset
Hosting Platform: ____________
Technology Stack Summary: ____________
Primary Dependencies (upstream): ____________
Primary Dependencies (downstream — what calls this): ____________
Age (date of last major rewrite): ____________
Last Major Investment Date: ____________

SECTION 2: HEALTH SCORING
Score each dimension 0–20. Scoring guide in reference/technology-currency-scoring-guide.md.

DIMENSION 1: RELIABILITY (0–20)
Data source: SLO dashboard, incident tracker
- SLO compliance, trailing 90 days: ___% (target: ≥ ___%)
- P1/P2 incidents in trailing 12 months: ___ (benchmark: T1 ≤ 2, T2 ≤ 4, T3 ≤ 6)
- MTTR for last 3 incidents: ___ hours average (benchmark: T1 ≤ 2h, T2 ≤ 4h)

Reliability Score: ___ / 20
Scoring guide:
  20: SLO met, 0 P1/P2 incidents, MTTR < 1h
  15: SLO met, 1 P1/P2 incident, MTTR < 2h
  10: SLO met with exceptions, 2–3 incidents, MTTR < 4h
  5: SLO breached at least once, 3+ incidents
  0: Regular SLO breaches, 5+ incidents or MTTR > 8h

DIMENSION 2: SECURITY POSTURE (0–20)
Data source: vulnerability scanner, last pentest report, secrets management audit
- Critical CVEs unpatched: ___
- High CVEs unpatched: ___
- Last security scan date: ____________
- Last penetration test date: ____________
- Secrets management: [ ] Vault/secrets manager  [ ] Env vars  [ ] Hardcoded

Security Posture Score: ___ / 20
Scoring guide:
  20: 0 critical CVEs, 0 high CVEs, pentest < 12 months, secrets manager in use
  15: 0 critical, ≤ 3 high CVEs, pentest < 18 months
  10: 0 critical, 4–8 high CVEs, pentest < 24 months
  5: 1–2 critical CVEs with mitigation in progress, pentest > 24 months
  0: Critical CVEs unmitigated, or secrets hardcoded in code, or no pentest ever

DIMENSION 3: TECHNOLOGY CURRENCY (0–20)
Data source: dependency scan, EOL risk register
- Primary runtime version and EOL status: ____________ [ ] Current  [ ] Near-EOL  [ ] EOL
- Critical dependency EOL count: ___
- Infrastructure platform EOL status: [ ] Current  [ ] Near-EOL  [ ] EOL

Technology Currency Score: ___ / 20
(Use scoring guide from reference/technology-currency-scoring-guide.md)

DIMENSION 4: OPERATIONAL COST (0–20)
Data source: FinOps/cloud cost dashboard, SRE time tracking
- Annual infrastructure cost: $____________
- Monthly SRE hours attributable to this application: ___ hours
- Monthly SRE hours × fully-loaded rate: $____________ / month
- Total annual operational cost (infra + SRE): $____________
- Benchmark for equivalent modern service: $____________ / year
- Cost ratio (actual / benchmark): ___x

Operational Cost Score: ___ / 20
Scoring guide:
  20: cost ratio ≤ 1.0x (at or below benchmark)
  15: cost ratio 1.1–1.5x
  10: cost ratio 1.6–2.0x
  5: cost ratio 2.1–3.0x
  0: cost ratio > 3.0x (3x+ the cost of equivalent modern service)

DIMENSION 5: TEAM KNOWLEDGE (0–20)
Data source: team interview, runbook audit
- People who can independently operate this application: ___
- People who can independently resolve the top 5 incident types: ___
- Runbook coverage of known failure modes: ___%
- Architecture documentation: [ ] Complete  [ ] Partial  [ ] Outdated  [ ] None
- On-call training: [ ] Full team  [ ] 2–3 people  [ ] 1 person  [ ] Unknown

Team Knowledge Score: ___ / 20
Scoring guide:
  20: 4+ people fully capable, 100% runbook coverage, complete architecture docs
  15: 3 people capable, > 80% runbook coverage, up-to-date docs
  10: 2 people capable, 60–80% runbook coverage, partial docs
  5: 1–2 people, < 60% runbook coverage, outdated docs
  0: ≤ 1 person, no runbooks, no documentation

SECTION 3: COMPOSITE SCORE
Reliability:          ___ / 20
Security Posture:     ___ / 20
Technology Currency:  ___ / 20
Operational Cost:     ___ / 20
Team Knowledge:       ___ / 20
─────────────────────────────
TOTAL:                ___ / 100

HEALTH TIER:
[ ] Healthy (80–100)  [ ] Watch (60–79)  [ ] At Risk (40–59)  [ ] Critical (20–39)  [ ] Emergency (0–19)

SECTION 4: LIFECYCLE ASSESSMENT
Current lifecycle stage (from Section 1): ____________

Stage change recommendation:
[ ] No change
[ ] Promote to: ____________ (reason: ____________)
[ ] Demote to: ____________ (reason: ____________)

Demotion triggers:
  → Mature/Stable → Legacy: technology currency score < 10 AND change risk confirmed high
  → Legacy → Sunset: replacement proven stable (90-day SLO), all consumers migrated or migration planned

SECTION 5: RECOMMENDED ACTION

Based on composite score and lifecycle stage, select the primary recommendation:

[ ] No action required — continue monitoring
[ ] Knowledge preservation sprint required — bus factor < 2 (initiate within 30 days)
[ ] EOL upgrade required — technology currency score < 10 (plan within 60 days)
[ ] Modernization assessment required — score < 60 or legacy classification
    (see modernization-assessment-playbook.md)
[ ] Change freeze recommended — L3/L4 classification (initiate immediately, VP sign-off required)
[ ] Decommission candidate — replacement in place, zero traffic confirmed
[ ] Emergency — L4 classification, VP notification required within 4 hours

Specific actions and owners:
Action 1: ____________ | Owner: ____________ | Due: ____________
Action 2: ____________ | Owner: ____________ | Due: ____________
Action 3: ____________ | Owner: ____________ | Due: ____________

SECTION 6: SIGN-OFF
Reviewed by (Technical Owner): ____________  Date: ____________
Reviewed by (VP of Operations): ____________  Date: ____________
  (Required for At Risk / Critical / Emergency)
Next scorecard due: ____________
```
