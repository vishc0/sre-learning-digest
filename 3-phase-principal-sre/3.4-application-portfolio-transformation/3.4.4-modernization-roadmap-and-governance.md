# Chapter 4: Modernization Roadmap and Governance

## 1. What a Modernization Roadmap Is — and Is Not

A modernization roadmap is a 3-year governance document that answers: given the current portfolio state, in what order do we transform systems, at what cost, and why that sequence? It is not a project plan. A project plan is owned by an engineering team and tracks tasks, sprint velocity, and delivery milestones. A roadmap is owned by the VP or Director and tracks portfolio-level investment decisions, sequencing logic, and business outcomes.

The distinction matters because roadmaps and project plans fail for different reasons. Project plans fail because of execution: scope creep, staffing changes, technical surprises. Roadmaps fail because of governance: wrong sequencing, inadequate investment, lack of executive alignment on priorities, and the absence of a quarterly review cycle that would catch drift before it becomes catastrophic.

**Why a 3-year horizon**: significant modernizations — replacing an ESB, decomposing a monolith, re-platforming a billing system — take 12–18 months minimum when done safely, with proper traffic migration, consumer coordination, and data validation. A 3-year horizon allows planning 2–3 sequential transformation waves and gives the business a visible trajectory for when the portfolio will reach its target state.

**What the roadmap contains**:

- Current portfolio state: health scores, lifecycle stages, and legacy classification (L1–L4) for every application in scope
- Target portfolio state: what the portfolio looks like in 3 years — which systems move from Legacy to Mature, which are decommissioned, which are replatformed onto current infrastructure
- Transformation waves: the grouping of transformations into logical execution phases with dependencies between waves explicitly documented
- Per-application transformation plan: which pattern (replatform, refactor, replace, retire), estimated timeline, estimated investment, expected operational outcome
- Inter-transformation dependencies: System A must be modernized before System B can be safely transformed
- Investment model: cost per wave and total, with funding source and payback period

## 2. Sequencing Logic

The single most common roadmap failure is wrong sequencing: starting with the application that causes the most pain rather than the application that unblocks the most other transformations. Pain-first sequencing feels rational — the system that is paging at 2am deserves immediate attention. But if that system depends on a shared integration layer that has not yet been modernized, replacing the application leaves it still coupled to the legacy integration infrastructure, and you have not materially improved the architecture.

**Dependency-first sequencing**: map every application's inbound and outbound dependencies before setting the transformation sequence. The application with the most inbound dependencies — the one that the greatest number of other systems depend on — must be transformed or wrapped first. If it is transformed first, downstream systems can migrate to its new interface during their own transformations. If it is not transformed first, every downstream transformation must maintain a compatibility shim with the legacy interface, which doubles the work.

**The foundation-first principle**: integration infrastructure must exist before application decomposition begins. You cannot safely decompose a monolith if you do not have an event bus for async communication between decomposed services. Without Kafka, every service extracted from the monolith must call every other service synchronously, which recreates the tight coupling the decomposition was intended to eliminate. Without an API gateway, you have no traffic management, no rate limiting, no consumer visibility, and no facade behind which to run parallel traffic during migrations.

Build the foundation before you build on it. This is not optional and it is not a sequence you can compress.

**T-Mobile-pattern example sequence**:

```
Wave 1 — Foundation (months 1–6)
  1. Deploy Kafka (event bus) — enables async communication for future decomposition
  2. Deploy API Gateway (Kong/Apigee) — enables Strangler Vine pattern over ESB

Wave 2 — Core Integration Transformation (months 6–18)
  3. Wrap TIBCO ESB behind API Gateway — Strangler Vine begins, consumers decouple
  4. Modernize shared customer data service — highest-dependency system, all others depend on it

Wave 3 — Application Decomposition (months 12–30)
  5. Decompose monolith by domain, in dependency order (data-owning services first)
  6. Replace ESB flows behind the API Gateway, one flow group at a time

Wave 4 — Decommission (months 24–36)
  7. Decommission TIBCO ESB (after zero flows remain behind gateway)
  8. Decommission monolith (after all traffic confirmed on decomposed services)
```

Why this sequence cannot be reordered: step 5 (monolith decomposition) requires Kafka from step 1 — the decomposed services communicate async. Step 6 (ESB flow replacement) requires the API gateway from step 2 — the facade must be in place before flows can be migrated behind it. Step 4 (customer data service) must precede step 5 — services decomposed from the monolith will call the shared customer data service; if that service is still legacy during decomposition, the new services are already coupled to a legacy dependency. Every reordering creates a dependency violation that forces rework.

## 3. Dependency Mapping

Dependency mapping is the prerequisite to sequencing. Without an accurate dependency map, the sequence is a guess. The map must be built before the roadmap is written, not concurrently.

**For every application in the portfolio, document**:

- Upstream dependencies: what systems does this application call, and on what protocol (REST, gRPC, RabbitMQ exchange, TIBCO adapter, database)?
- Downstream dependencies: what systems call this application, and on what protocol?
- Data dependencies: what databases or data stores does this application own? Which other applications have direct database access (shared database pattern, a legacy anti-pattern that complicates decomposition)?
- Infrastructure dependencies: which specific RabbitMQ exchange, which TIBCO flow name, which Kafka topic, which MQ queue name?

**Tools for dependency discovery** — use all four because each discovers dependencies the others miss:

- **Service mesh telemetry** (Istio/Linkerd): shows actual TCP and HTTP traffic at the network layer, including service-to-service calls that are not documented anywhere. This is the ground truth for in-cluster communication.
- **API gateway logs**: shows who has called which API, from which service identity, in the last 90 days. Catches authorized consumers through the gateway. Misses direct service-to-service calls that bypass the gateway.
- **Application dependency scanning**: Spring Boot Actuator `/actuator/beans` endpoint, dependency injection graph inspection, configuration file analysis. Identifies declared dependencies even if they are not currently active.
- **Database connection tracing**: which services connect to which databases. Identifies the shared database anti-pattern — two or more services connecting directly to the same database, which is a hidden coupling that will force coordinated changes during modernization.

**Critical warning**: documented dependencies and actual runtime dependencies are routinely different in large organizations. Documentation reflects intent at the time of writing. Actual traffic reflects what the system does today, which has drifted from documentation over years of incremental changes. Always verify with telemetry before finalizing the dependency map.

## 4. The Modernization Investment Model

Modernization must be funded as a portfolio-level commitment, not as a series of individual project approvals. Individual project approval creates a cycle where each modernization must compete for budget against feature work every planning cycle, which means modernizations are chronically underfunded or delayed.

**The self-funding model**: modernization investment is partially self-funding through the operational savings that successful modernizations produce. Each decommissioned system releases infrastructure cost, licensing cost, and SRE time back to the budget. The released budget is recycled into the next transformation wave. This creates a flywheel: Wave 1 investments produce savings that partially fund Wave 2.

**Modeling the payback period**:

| Component | Example Values |
|-----------|---------------|
| Current annual operating cost (Run) | $1.8M (infra + licensing + SRE labor at loaded rate + incident cost) |
| Projected annual cost post-modernization | $600K |
| Annual saving | $1.2M |
| Modernization investment | $2.4M over 18 months |
| Payback period | 24 months from completion (30 months from start) |
| 5-year net benefit | $3.6M savings minus $2.4M investment = $1.2M net positive |

**The business case structure for each application transformation**:

1. Current annual cost (Run): infrastructure + licensing + SRE hours at fully loaded labor rate + estimated annual incident cost (mean incident cost × mean annual frequency)
2. Projected cost post-modernization: equivalent cloud infrastructure at current efficiency + reduced SRE labor (based on expected MTTR and incident rate improvement)
3. Investment required: engineering labor + tooling + infrastructure for parallel run period
4. Payback period: investment ÷ annual saving
5. Risk of not modernizing: annual probability of a major incident × estimated cost of that incident (downtime revenue impact + labor + customer compensation)

**Budget allocation**: the Transform budget should be a committed percentage of total engineering spend, not a residual. A mature portfolio governance model allocates:

- 70% Run: keeping existing systems operational
- 20% Grow: new product features
- 10% Transform: modernization and portfolio hygiene

In a legacy-heavy portfolio, the Transform allocation may need to reach 20–25% in the first wave to address the accumulated debt, stepping down as the portfolio health improves. Present this to the CFO as a time-limited investment with a documented payback, not as an ongoing overhead.

## 5. Milestone Governance: The Quarterly Transformation Review

A modernization roadmap without a quarterly governance cadence is a document that will be out of date within 6 months and irrelevant within 12. The quarterly transformation review is the mechanism that keeps the roadmap current and the transformations on track.

**Who attends**: VP Engineering, Director SRE, Director Platform Engineering, senior engineers leading each in-flight transformation, Finance partner.

**What is reviewed every 90 days**:

- Status of every in-flight transformation: current phase, percentage of traffic migrated to new system, open blockers, revised projected completion date
- Budget vs. actual for each in-flight transformation: cost spent to date vs. plan, projected total cost at completion vs. approved budget
- Risk register changes: new risks identified since last review, resolved risks, changes in risk ratings
- Sequence adjustments: any transformations that need to re-sequence based on new dependency information or changed business priorities
- New applications entering the transformation queue: applications that have moved to Legacy classification since the last review
- Applications ready to enter the Sunset phase: transformations where the replacement has passed the 90-day SLO threshold

**Output — four possible decisions per in-flight transformation**:

| Decision | Meaning | Documentation Required |
|----------|---------|----------------------|
| Proceed | On track, no changes | Status update in the roadmap |
| Re-sequence | Priority or order changes due to new information | Documented reason, updated timeline, stakeholder notification |
| Pause | Blocked by external dependency or resource constraint | VP approval required; documented trigger and expected resume date |
| Escalate | In trouble — timeline, budget, or technical blockers that engineering cannot resolve alone | Director-level intervention; root cause analysis within 2 weeks |

"Proceed" must be explicit, not assumed. If a transformation is not discussed at the quarterly review, it does not receive a "proceed" by default — it gets flagged for follow-up.

## 6. Transformation Risk Management

Transformations fail in predictable ways. Managing these risks is a governance function, not an engineering function.

**Data migration failure**: the replacement system does not have accurate, complete data, so consumers get wrong answers. This is the most dangerous failure mode because it may not be immediately visible — the system responds, but with incorrect data. Mitigation: dual-write period with continuous reconciliation. Both old and new systems write data simultaneously. A reconciliation job runs continuously and reports discrepancies. The traffic migration does not proceed until reconciliation shows zero discrepancies over a sustained period. A one-time cutover reconciliation check is insufficient — it validates a snapshot, not an ongoing state.

**Consumer compatibility failure**: the new system's API contract breaks consumers who were calling the legacy interface and expecting legacy behavior (including legacy error codes, response formats, and field names). Mitigation: maintain a backward-compatible API contract during the migration period, enforced by the API gateway. The gateway translates between legacy interface and new interface. New consumers use the new contract. Legacy consumers continue using the legacy contract until they can migrate at their own pace.

**Performance regression**: the new system is functionally correct but slower than the legacy system under production load, causing SLO breach after traffic migration. Mitigation: performance benchmarking as a go/no-go gate before any traffic migration begins. Canary testing at 5% of production traffic before 50%. The performance baseline comes from legacy system metrics — the new system must match or exceed it under equivalent load.

**Scope creep**: "while we're modernizing this, let's also refactor the data model, rebuild the UI, migrate to a new database, and add the three features the product team has been waiting for." This is the single most common cause of transformation stalls. Each addition feels small and reasonable in isolation. Collectively they turn a 9-month modernization into a 3-year project with no delivery. Mitigation: a hard scope control gate. Modernization scope is limited to the minimum change required to achieve the stated technical outcome — moving from legacy to current architecture while preserving existing functionality. New features and additional refactoring are separate workstreams with separate timelines and separate budgets. The scope gate is enforced by the VP, not by the engineering team, because the engineering team will be pulled toward scope expansion by reasonable-sounding requests.

## 7. The Transformation Stall Pattern

The transformation stall is a predictable failure mode: a modernization project that starts, consumes 12–24 months of engineering investment, and does not deliver because scope expands without constraint and success criteria are never clearly defined.

**How to detect a stall before it becomes a crisis**:

- No production traffic has been migrated to the new system after 6 months of work. Traffic migration is the only leading indicator that matters — everything else (architecture decisions, code written, test coverage) is input, not output.
- Scope changes are approved without corresponding timeline adjustments. If the project adds 3 months of scope, the timeline must extend 3 months. If it does not, the timeline is a fiction.
- "Almost ready" appears in status reports for more than two consecutive quarterly reviews. Define "ready" operationally: what is the exact milestone that will be reached, and what is the date?
- The team is adding features to the new system that were not in the original scope. This is the clearest signal of scope creep — the team has moved from "match legacy functionality" to "build a better product," which is a different project.

**Governance intervention when a stall is detected**:

1. **Mandatory scope freeze**: a written directive from the VP that no new requirements will be accepted into the current transformation scope. Any new requirements go into a post-modernization backlog. Enforced immediately.

2. **Traffic migration milestone as the sole accepted success metric**: the transformation is considered in-progress until a defined percentage of production traffic is running on the new system. Define the milestone (e.g., "10% of production traffic on new billing system by [date]"), and measure against it at every subsequent review. No other metric substitutes for this one.

3. **Executive escalation if the milestone is missed**: missing the traffic migration milestone triggers a root cause analysis and a recovery plan presentation at Director level within 30 days. The recovery plan must include a revised milestone date with specific engineering commitments.

4. **Staffing review**: is the team actually working on the transformation, or have key engineers been diverted to incident response on L3/L4 systems, product feature work, or other priorities? Transformation stalls are frequently a resource allocation problem disguised as a technical problem. The VP must confirm that the declared engineering capacity is actually allocated to the transformation.

## 8. Success Metrics for a Modernization

A modernization is not complete when the new system is built. It is not complete when the new system is deployed. It is complete when the legacy system is decommissioned and the operational metrics demonstrate sustained improvement.

| Metric | What It Measures | How to Calculate |
|--------|-----------------|-----------------|
| Infrastructure cost reduction | Per-unit cost improvement | Monthly infrastructure cost for equivalent transaction volume: new vs. legacy |
| SLO improvement | Error rate and latency improvement | Trailing-90-day error rate and p99 latency: new system vs. legacy baseline |
| Incident count reduction | P1/P2 incidents per month | New system monthly incident rate vs. legacy 12-month average |
| MTTR reduction | Time to recover per incident | Average MTTR per incident type: new vs. legacy |
| Deployment frequency | Change velocity improvement | Deployments per month: new system vs. legacy |
| Bus factor improvement | Organizational resilience | Minimum number of engineers who can independently operate and recover the system: new vs. legacy |
| Legacy decommissioned | Final success confirmation | Boolean — the legacy system is no longer running |

The last metric is non-negotiable. Until the legacy system is decommissioned, the organization is paying double: operating the legacy system plus the new system in parallel. The transformation ROI is negative until the legacy system is off. Decommission is not a post-project cleanup task — it is a first-class delivery milestone.
