# Run / Grow / Transform Budget Model

The Run/Grow/Transform (RGT) framework is the standard lens for portfolio-level budget allocation in technology organizations. It answers one question the CFO and board always ask: "Why does IT cost so much, and where does the money go?" RGT gives a precise, defensible answer — and more importantly, it gives a management lever for shifting spending from operational overhead to strategic investment.

---

## Definitions

### Run

All spending to keep existing applications operational at their current capability level. Run spending produces no new user-visible capability and makes no architectural change. It is the cost of the portfolio as it currently exists.

**Included in Run**:
- Infrastructure costs: compute, storage, network, cloud hosting, datacenter, managed service fees
- SRE on-call hours and incident response time
- Routine maintenance: patching, certificate renewals, capacity management
- Security patching (keeping existing systems current, not hardening them)
- Observability tooling: licensing, dashboards, alert management
- License renewals: middleware platforms, databases, APM tools
- Compliance reporting overhead attributable to existing systems

**Not included in Run**:
- New features (Grow)
- Performance improvements that users experience as new capability (Grow)
- Architecture changes, even if operationally motivated (Transform)
- Automation that reduces future Run cost (Transform — the investment is Transform; the resulting savings reduce future Run)

### Grow

All spending to add user-visible capability to existing applications. Grow produces a change that a business user or customer can see and use.

**Included in Grow**:
- New features and product enhancements
- Performance improvements that users experience (page load time, transaction throughput)
- UX changes and interface improvements
- API extensions that expose new capability to consumers
- Integrations that enable new business processes

**Not included in Grow**:
- Internal refactoring that produces no user-visible change (Transform)
- Infrastructure-only work that improves reliability but not capability (Run or Transform depending on intent)

### Transform

All spending to fundamentally change how an application is built or operated, with the goal of reducing future Run cost, increasing future Grow velocity, or managing risk. Transform produces no immediate user-visible capability — its return is realized over time as Run cost decreases and Grow velocity increases.

**Included in Transform**:
- Modernization projects: re-platforming, re-architecting, cloud migration
- ESB replacement and middleware consolidation
- Monolith decomposition and service extraction
- Database migration (schema redesign, engine change, managed service migration)
- Automation development: runbook automation, self-healing, toil elimination scripts
- Observability investment: instrumenting applications that currently have no telemetry
- Security hardening: architecture-level changes (secrets management implementation, network segmentation)
- Decommission execution: the labor cost of decommissioning a legacy application

**Not included in Transform**:
- Incremental refactoring within an existing architecture (no architectural change = Run or Grow)
- Adding a microservice that extends an existing API without changing the architecture (Grow)
- Routine security patching (Run — patching is maintenance; hardening is Transform)

## Classification Rules for Ambiguous Cases

Classification disputes are common. Use these rules to resolve them consistently.

| Scenario | Classification | Rationale |
|----------|---------------|-----------|
| Kubernetes cluster upgrade | Run | It is a maintenance obligation. The cluster must be kept current. |
| Kafka deployment enabling new event-driven architecture | Transform | It establishes a new architectural capability, not just a new feature. |
| Migrating on-prem to cloud (Lift and Shift) | Transform | A structural change to hosting, even without architecture improvement. |
| Adding a microservice that extends an existing API | Grow | New user-visible capability through an established architectural pattern. |
| Replacing a microservice with a better-designed one | Transform | Architectural improvement; no new user-visible capability. |
| Writing a runbook automation script | Transform | Eliminates future manual Run labor. |
| Running incident response for an existing system | Run | Operating the existing portfolio at its current capability level. |
| SRE engineer spending 20% of time on toil automation | Transform (20%), Run (80%) | Split by the actual activity, not by the team or headcount allocation. |

When a single activity spans categories, split the cost proportionally by time spent. Precision of ±5% is sufficient — do not over-engineer the classification at the activity level.

## Industry Benchmark Splits

These are directional benchmarks from Gartner's IT Key Metrics research. They are starting points for comparison, not targets in isolation. Your actual target depends on portfolio maturity, industry, and strategic posture.

| Organization Type | Run % | Grow % | Transform % | Notes |
|------------------|-------|--------|-------------|-------|
| Legacy-heavy (>50% legacy portfolio) | 70–80% | 15–20% | 5–10% | Insufficient Transform investment — debt spiral risk |
| Transitioning (active legacy reduction program) | 55–65% | 20–25% | 15–20% | Balanced — sustainable if Transform drives Run reduction |
| Cloud-native / modernized | 40–50% | 30–35% | 20–25% | High efficiency — Run savings enable sustained investment |
| Startup / greenfield | 20–30% | 50–60% | 10–20% | Low legacy overhead — Grow-dominated |
| Target for T-Mobile-scale enterprise | < 60% | 25–30% | > 15% | Target state for mature SRE org actively modernizing |

The "debt spiral" condition: when Run exceeds 75%, Transform investment is insufficient to reduce the Run obligations that are growing as the legacy portfolio accumulates debt. The organization runs faster to stay in place. The only exit from the debt spiral is a deliberate, funded Transform program — which requires temporary Run cost increase (dual-running during transformation) before the Run reduction is realized.

## How to Measure the Current Ratio

Engineering budget is tracked by team or cost center, not by activity type. Converting team budget to RGT requires a classification overlay.

### Step 1: Classify each team's primary work by RGT percentage

Use team lead input validated against sprint data (Jira ticket classification or equivalent). Typical classifications:

| Team Type | Run % | Grow % | Transform % |
|-----------|-------|--------|-------------|
| SRE / platform (operations-focused) | 80% | 10% | 10% |
| Application support (maintaining legacy) | 70% | 20% | 10% |
| Modernization / migration team | 10% | 0% | 90% |
| Product engineering (new features, greenfield) | 20% | 70% | 10% |
| Security engineering | 50% | 10% | 40% |
| Data engineering | 40% | 30% | 30% |

Adjust these defaults based on each team's actual sprint composition. If an SRE team spent 40% of the quarter on a toil automation initiative, their Transform percentage for that quarter is 40%, not 10%.

### Step 2: Multiply fully-loaded team cost by classification percentages

Fully-loaded cost includes: salary, benefits, employer taxes, equipment, tools, training, management overhead. A common fully-loaded multiplier for US software engineers is 1.4–1.6x base salary.

### Step 3: Add infrastructure costs (100% Run)

Cloud spend, datacenter costs, managed service fees, and software licenses are Run. They produce no new capability — they keep the current portfolio running.

### Step 4: Sum and compute percentages

### Example Calculation

| Team | Headcount | Fully-Loaded Cost | Run % | Grow % | Transform % | Run $ | Grow $ | Transform $ |
|------|-----------|-------------------|-------|--------|-------------|-------|--------|-------------|
| SRE | 10 FTE | $3.0M | 80% | 10% | 10% | $2.40M | $0.30M | $0.30M |
| App support | 5 FTE | $1.4M | 70% | 20% | 10% | $0.98M | $0.28M | $0.14M |
| Modernization | 3 FTE | $0.96M | 10% | 0% | 90% | $0.10M | $0.00M | $0.86M |
| Product eng | 8 FTE | $2.4M | 20% | 70% | 10% | $0.48M | $1.68M | $0.24M |
| Infrastructure (cloud + licenses) | — | $1.2M | 100% | 0% | 0% | $1.20M | $0.00M | $0.00M |
| **Total** | 26 FTE | **$8.96M** | | | | **$5.16M** | **$2.26M** | **$1.54M** |

**Current RGT ratio**: Run 57.6% / Grow 25.2% / Transform 17.2%

Compared to T-Mobile-scale enterprise target (< 60% / 25–30% / > 15%): this example organization is near target. Run is at 57.6% (below the 60% threshold), Grow is at 25.2% (within the 25–30% target), Transform is at 17.2% (above the 15% floor).

## How to Shift the Ratio Over Time

The target is to move Run percentage down and Transform percentage up, without reducing Grow in absolute dollars. This is achieved through three simultaneous levers — not through budget reallocation alone.

### Lever 1: Decommission legacy

Each successful decommission eliminates infrastructure cost and SRE operational overhead permanently. A decommissioned application contributes $0 to Run going forward. The freed budget — previously allocated to keeping the legacy application operational — is reallocated to Transform investment for the next modernization.

Mechanism:
```
Decommission completes
  → Infrastructure cost eliminated ($X/year)
  → SRE on-call hours freed (Y hours/month × $Z/hour = $W/year)
  → Total Run reduction: $X + $W per year
  → Reallocate to Transform: funds the next modernization
```

### Lever 2: Automation (toil elimination)

SRE automation converts recurring manual Run work into a one-time Transform investment. Every manual operational task that is automated permanently reduces the Run hours required to maintain that application.

Example: a legacy billing system requires 8 hours/month of manual log rotation, disk cleanup, and health checks. An SRE automation sprint (40 hours Transform investment) automates all of this. The 8 Run hours/month (96 hours/year) become 0 hours/month — permanently.

Payback: 40 hours invested / 96 hours saved per year = 5-month payback. Every month after month 5, the automation is generating net Run reduction.

### Lever 3: Modernization payoff

Each successfully modernized application has lower Run cost than its legacy predecessor. This is the fundamental claim of the modernization business case. If the modernization was designed and executed correctly, the post-modernization application costs less to operate (infrastructure), generates fewer incidents (SRE response hours), and deploys faster (less deployment overhead).

The savings flow back into Run budget reduction, which frees capital for the next Transform investment. This is the modernization flywheel.

### What the shift is NOT

The Run/Grow/Transform ratio shift is not achieved by budget reallocation alone — telling the CFO "we are moving 5% of budget from Run to Transform" does not work because Run obligations (on-call coverage for production systems, license renewals, infrastructure commitments) cannot be reduced by budget decision. They are reduced by decommissioning the systems that generate them.

The mechanism is: reduce obligations first, then redirect freed budget. The inverse — allocate budget to Transform while Run obligations remain unchanged — requires new incremental budget for Transform, not reallocation from Run.

### Timeline expectations

Meaningful ratio shift (5+ percentage points in Run reduction) requires 18–24 months minimum. Reasons:
- Infrastructure contracts: cloud reserved instances, datacenter leases, software licenses have terms that cannot be broken without penalty.
- Decommission takes time: even after a replacement system is stable, the 8-phase decommission process (from ch05) takes 3–6 months.
- SRE retraining: engineers freed from legacy operations need time to retrain for new platform capabilities before their capacity becomes available for Transform work.

Board-level expectations must be set at the start of any ratio-shift program: "We will begin seeing Run reduction in Q3 of next year, with measurable ratio improvement by Q4."

## How to Present the RGT Model to the Board

Use a three-slide framework. The audience is the CFO, CEO, and board members — not engineers. Every number must trace to a data source.

### Slide 1: Current State

**Visual**: stacked bar chart showing current RGT ratio vs. Gartner benchmark for your organization type.

**Narrative**: "Our current budget allocation is [X]% Run, [Y]% Grow, [Z]% Transform. Gartner benchmarks for organizations of our size and maturity in this industry show [benchmark]. The gap between our current Run percentage and the benchmark represents approximately $[dollar amount] in excess operational overhead annually. This excess is the direct result of [N] legacy applications in our portfolio that cost [2–3x] more to operate than equivalent modern services."

**Data required**: current RGT calculation (from the measurement process above), Gartner benchmark reference, dollar cost of the gap.

### Slide 2: The Trajectory

**Visual**: two lines on a 3-year time chart — Run % with and without Transform investment.

**Without investment**: Run continues at current rate or increases as legacy portfolio accumulates more debt. Typical trajectory: Run grows 2–3% annually as legacy systems degrade and require more intervention.

**With investment**: Run declines 3–5% annually as decommissions complete and modernizations reduce per-application operational cost. Crossover point: approximately 18 months after Transform investment begins.

**Narrative**: "Without a modernization investment, our Run cost will grow to [projected amount] by Year 3 as our legacy portfolio continues to degrade. With the proposed Transform investment, we project Run cost declining to [target amount] by Year 3 — a net difference of $[delta] in Year 3 operating budget, which recurs annually thereafter."

**Data required**: incident trend for legacy applications (showing increasing support cost), cost modeling from modernization business cases for each planned transformation.

### Slide 3: The Ask

**Structure**:
- Total Transform investment requested: $[amount] over [N] months
- Funding sources (be specific — executives will ask):
  - New incremental budget: $[amount]
  - Run savings from decommissions completed in the prior 12 months: $[amount]
  - SRE automation savings from toil elimination: $[amount]
- Expected outcomes by Year [N]:
  - Run % reduced from [current] to [target]
  - Annual Run savings released for Grow investment: $[amount]
  - Applications decommissioned: [N]
  - SLO improvement across portfolio: [current average] → [target]

**Narrative**: "We are requesting $[total] in Transform investment, funded from three sources: [breakdown]. If approved, we project Run percentage declining to [target] by [date], releasing $[amount] annually for product capability investment. We are not asking for indefinite commitment — we will report quarterly RGT ratio progress against these projections, and if progress stalls, we will surface it before it becomes a budget problem."

The quarterly reporting commitment closes the ask — it converts the board's concern ("we've heard this before") into a measurable accountability structure.
