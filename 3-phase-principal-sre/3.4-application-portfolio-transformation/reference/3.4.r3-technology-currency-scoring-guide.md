# Technology Currency Scoring Guide

Produces the technology currency score (0–20) used in Section 2, Dimension 3 of the Application Health Scorecard. Run this assessment quarterly for Legacy-stage applications, annually for Mature/Stable, and at any time a dependency EOL event is identified.

---

## The Three Sub-Scores

Technology currency is scored across three dimensions: runtime, dependencies, and infrastructure. Each is scored independently and then summed to produce the composite 0–20 score.

---

### Sub-score 1: Runtime Score (0–7)

The application's primary runtime(s) — the execution environment that actually runs the application code. For Java applications: the JDK. For Python: the interpreter. For Node.js: the runtime. For .NET: the CLR. For containerized applications, this is the base image language runtime, not the container runtime (that is infrastructure).

| Condition | Points |
|-----------|--------|
| All runtimes current (within active support window, not approaching EOL) | 7 |
| All runtimes in LTS/extended support, not EOL, but no longer receiving new features | 5 |
| One runtime approaching EOL (< 12 months to EOL date) | 3 |
| One runtime EOL (vendor no longer releasing patches, including security patches) | 1 |
| Multiple runtimes EOL | 0 |

If the application has multiple runtimes (e.g., Java backend and Node.js frontend), score the worst-case runtime. The application's overall risk is bounded by its least-current component.

---

### Sub-score 2: Dependency Score (0–7)

The application's libraries, frameworks, vendor SDKs, and middleware clients. Includes: Spring Boot, Hibernate, Kafka client library, RabbitMQ AMQP client, MuleSoft connectors, TIBCO adapter libraries, npm packages, Python packages.

| Condition | Points |
|-----------|--------|
| All dependencies in supported versions, 0 critical CVEs, 0 high CVEs | 7 |
| All in supported versions, ≤ 3 high CVEs (patched version available in supported release) | 5 |
| 1–2 dependencies near EOL (< 12 months), or 4–8 high CVEs | 3 |
| Any dependency EOL, or any critical CVE present (even in a supported dependency if patch is not yet applied) | 1 |
| Dependencies EOL with critical CVEs (cannot patch without upgrading the EOL component first) | 0 |

The distinction between score 1 and score 0 is whether the CVE is patchable without an EOL upgrade. If a critical CVE exists in a supported dependency version and the team simply has not applied the patch: score 1 (negligent but recoverable). If the critical CVE requires upgrading a component that is itself EOL (the fix is only in a version your runtime cannot support): score 0.

---

### Sub-score 3: Infrastructure Score (0–6)

The application's hosting infrastructure: Kubernetes version, container runtime (containerd, Docker), operating system (Ubuntu, RHEL, Amazon Linux), and any managed service versions (RDS engine version, MSK Kafka version, ElastiCache engine version).

| Condition | Points |
|-----------|--------|
| All infrastructure components within supported version window | 6 |
| All in extended support, no EOL (e.g., Ubuntu with ESM, RHEL with extended subscription) | 4 |
| One component approaching EOL (< 6 months to EOL date) | 2 |
| Any infrastructure component EOL | 0 |

Infrastructure EOL is treated more severely than runtime EOL because infrastructure vulnerabilities typically have a broader blast radius — an EOL Kubernetes version exposes every workload running on that cluster, not just one application.

---

## Composite Score and Interpretation

**Composite Technology Currency Score = Runtime Score + Dependency Score + Infrastructure Score**

Maximum: 7 + 7 + 6 = 20

| Score Range | Label | Required Action |
|-------------|-------|-----------------|
| 18–20 | Current | Technology stack is healthy. Continue monitoring. Next review at normal cadence. |
| 14–17 | Watch | Minor currency issues present. Include upgrades in next planning cycle (next quarter). |
| 8–13 | At Risk | Significant EOL exposure. Add upgrade tasks to current cycle. Plan within 60 days. |
| 0–7 | Critical | EOL components in production. Upgrade plan required within 30 days. Execution required within 90 days. |

A score of 0–7 triggers immediate entry into the EOL Risk Register at Critical or High level and requires an upgrade plan submitted to the VP of Operations within 30 days.

## How to Run a Currency Assessment

### Tools for runtime and dependency scanning

| Tool | Type | Integration | Best For |
|------|------|-------------|---------|
| OWASP Dependency Check | Open source | Maven, Gradle, npm, NuGet | CVE scanning, CI/CD integration, no cost |
| Snyk | Commercial | GitHub, GitLab, CLI, IDE | Real-time CVE database, PR-level blocking, developer workflow |
| Dependabot | Free (GitHub) | GitHub repos | Automated dependency update PRs |
| AWS Inspector | Cloud (ECR integrated) | ECR, EC2, Lambda | Container image scanning, AWS-native |
| Trivy | Open source | CLI, CI/CD, container runtime | Container and filesystem scanning, fast |

### Command reference

```bash
# Trivy — scan a container image for CVEs and EOL components
trivy image [image-name]:[tag]

# Trivy — scan a filesystem (for non-containerized applications)
trivy fs /path/to/project

# Snyk — scan a Node.js project
snyk test

# Snyk — scan a Docker image
snyk container test [image-name]:[tag]

# OWASP Dependency Check — scan a Java Maven project
dependency-check.sh --project [name] --scan . --format HTML --out ./reports

# AWS Inspector — initiate a container image scan via CLI
aws inspector2 start-code-security-scan --resource-id [ecr-repo-arn]

# Trivy — scan with JSON output for programmatic processing
trivy image --format json --output results.json [image-name]:[tag]
```

### Splunk queries for operational currency evidence

CVEs in runtime logs (if application logs runtime version on startup):
```spl
index=app_logs sourcetype=application earliest=-30d
| rex field=_raw "runtime=(?<runtime_version>[^\s]+)"
| stats count by runtime_version, host
| sort -count
```

Failed dependency resolution events (Maven/npm build failures in CI):
```spl
index=ci_logs sourcetype=jenkins earliest=-90d
| search "dependency" AND ("FAILED" OR "ERROR" OR "EOL" OR "deprecated")
| stats count by job_name, _time
| timechart span=1w count by job_name
```

### CI/CD pipeline integration

Dependency scanning must run on every build. Configuration principle:

- Critical CVEs: fail the build. A build that introduces a critical CVE must not advance to staging.
- High CVEs: fail the build unless an exception is logged in the EOL Risk Register with VP sign-off.
- Medium CVEs: warn only. Track in the EOL Risk Register; remediate in next sprint planning cycle.

EOL currency scoring is a portfolio-level quarterly process, not a per-build check. The build pipeline enforces CVE policy; the quarterly scorecard enforces currency policy.

### Integrating scan results into the scorecard

After running dependency scanning tools:

1. Export CVE findings by severity (Critical / High / Medium / Low count).
2. Look up each dependency's EOL date against this reference table and the vendor's official EOL calendar.
3. Fill in Dimension 3 inputs on the scorecard.
4. Score each sub-dimension (runtime, dependency, infrastructure) using the tables above.
5. Sum to produce the 0–20 technology currency score.
6. If score is 0–7: immediately create or update the EOL Risk Register entry. Notify VP within 48 hours if Critical CVEs are present.
