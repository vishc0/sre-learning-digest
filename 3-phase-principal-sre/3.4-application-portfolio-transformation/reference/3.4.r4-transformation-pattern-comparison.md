# Transformation Pattern Comparison

Side-by-side comparison of all 9 transformation patterns used in application portfolio modernization. Pattern selection is one of the highest-leverage decisions in a modernization program — the wrong pattern wastes 12–18 months and may leave the application in a worse state than before.

---

## Pattern Comparison Table

| Pattern | Effort | Risk During Migration | Downtime Required | Rollback Ability | Delivers Value Before Complete | When to Use | When NOT to Use |
|---------|--------|-----------------------|-------------------|-----------------|-------------------------------|-------------|----------------|
| Lift and Shift | Low | Low | Minimal (maintenance window) | Easy | No | Datacenter exit, buying runway | Architecture is the problem |
| Re-platform | Low-Med | Low-Med | Minimal | Moderate | No | Kubernetes benefits without rewrite | Badly-architected app stays badly-architected |
| Re-architect | High | High | None (if done incrementally) | Hard | Partial | Architecture is the blocker | Team lacks capacity or deadline is short |
| Strangler Fig | Med-High | Low | None | Easy at each step | Yes — each slice | Most large legacy modernizations | No decomposable API boundary |
| Strangler Vine | Med | Low | None | Easy | Yes — from day 1 | Need clean API surface immediately | Legacy has no meaningful API surface |
| Anti-Corruption Layer | Low-Med | Low | None | Easy | Yes | Legacy data model corruption risk | Simple data models with clean interfaces |
| Big Bang Migration | High | Very High | Significant | Very Hard | No | Regulatory prohibition on parallel, no API boundary | Any system where rollback must be < 4 hours |
| Parallel Run | High | Low | None | Easy | No | Financial/billing correctness proof | Ongoing operations cost is prohibitive |
| DB Migration (Expand-Contract) | Med | Low-Med | None | Moderate | Yes | Zero-downtime schema changes | Major semantic model changes |

## Pattern Detail — Operational Descriptions and T-Mobile/Middleware Examples

### Lift and Shift

**What it is**: Move the application from one infrastructure to another without changing any code, architecture, or interfaces. VMs become cloud VMs. The ESB still runs on VMs — they are just AWS VMs now instead of datacenter VMs.

**When it works**: Datacenter exit deadlines. A lease is expiring, a colocated facility is closing, or a merger requires consolidating data centers. The transformation goal is eliminating the physical footprint, not improving the architecture.

**What it does NOT do**: Solve architectural problems, reduce operational cost in the long run (VM costs on cloud can exceed datacenter if not resized), or improve reliability. The same failure modes exist; they are now in a different location.

**T-Mobile/middleware example**: A TIBCO BusinessWorks deployment running on eight on-prem VMs must be moved because the datacenter lease expires in 90 days. Engineering lifts the BusinessWorks configuration, mediator flows, and adapters onto AWS EC2 instances with matching specifications. No flow logic changes. No protocol changes. ESB still processes TIBCO Rendezvous (TRDP) messages between the BSS stack and provisioning systems. Datacenter lease ends on schedule. Architecture is unchanged.

The natural next step after lift-and-shift is to begin a Strangler Vine while the application runs on cloud infrastructure. The runway bought by the lift is used to plan and execute the real modernization.

**Risk of misuse**: Teams that perform lift-and-shift believing it is "cloud migration" without a follow-on modernization plan have relocated their technical debt, not addressed it. Cloud infrastructure costs without the operational discipline (autoscaling, reserved instances, right-sizing) often exceed the datacenter baseline.

### Re-platform

**What it is**: Move the application to a new platform (Kubernetes, cloud PaaS) by packaging it differently — typically containerization — without changing application code or architecture.

**When it works**: The application logic is sound and the architecture is acceptable. The problem is the hosting platform: no autoscaling, manual capacity management, slow rollout pipelines, high per-instance cost. Containerizing delivers Kubernetes benefits (horizontal pod autoscaling, rolling deployments, resource bin-packing) without requiring architectural change.

**What it does NOT do**: Fix a shared database problem, decompose a monolith, or improve observability beyond what the application already exposes. A synchronous REST service with a shared relational database remains synchronous with a shared database — it just runs in pods now.

**T-Mobile/middleware example**: A Spring Boot customer notification service runs on PCF (Pivotal Cloud Foundry) and is being migrated to EKS as PCF is decommissioned. The migration creates a Dockerfile from the existing JAR, writes a Helm chart defining the deployment (replicas, resource limits, liveness probes), and configures the HPA to scale on CPU > 70%. No application code changes. No database changes. No API contract changes. The application now scales horizontally on demand, deploys with zero-downtime rolling updates, and runs on shared EKS cluster capacity, reducing per-application infrastructure cost.

**Risk of misuse**: Re-platforming a badly-architected application makes it a badly-architected application that runs in Kubernetes. A monolith with a shared database that regularly causes contention now causes the same contention from a pod. The operational problems shift from "the PCF instance is undersized" to "the pod is OOMKilled" — different symptom, same root cause.

### Re-architect

**What it is**: Fundamentally change how the application is structured — decompose a monolith, break up a shared database, replace synchronous call chains with event-driven communication, introduce domain boundaries. Requires significant engineering effort and carries the highest coordination risk of any pattern.

**When it works**: Architecture is the specific blocker — the current structure prevents scaling, causes cascading failures, blocks feature delivery, or creates unacceptable blast radius. The team has capacity for a 12–18 month engagement and the application will outlive the investment horizon by at least 5 years.

**What it does NOT do**: Deliver value quickly. Re-architect projects typically go 12–18 months before production traffic moves. This makes them high-risk from a business perspective — the investment must be justified with a clear payback horizon.

**T-Mobile/middleware example**: The T-Mobile retail order management function lives inside a monolith that handles order capture, inventory check, provisioning, billing setup, and fulfillment in a single deployable. Every deployment requires coordination across all five domains. An incident in billing setup causes rollbacks that affect order capture. Re-architecting extracts order management as an independent domain service.

The extraction sequence:
1. Define domain boundary: what data is "owned" by order management vs. shared?
2. Establish event communication: order events published to Kafka topic `orders.created`, `orders.updated`, `orders.fulfilled`. Other domains subscribe rather than calling the monolith directly.
3. Split the shared database: order tables become order service's private schema. The monolith's remaining modules read order state via the order service API, not via direct DB join.
4. Route traffic: API gateway routes `POST /orders` and `GET /orders/{id}` to the new service. The monolith's order endpoints become thin proxies, then are removed.
5. Verify: 30-day SLO proof before decommissioning the monolith order module.

Timeline for a well-scoped extraction: 12–18 months. For a poorly-scoped extraction: indefinite.

**Risk of misuse**: Starting a re-architect without clear domain boundaries produces a distributed monolith — microservices that are so tightly coupled through synchronous calls or shared databases that they cannot deploy or fail independently. A distributed monolith is worse than the original: all the complexity of microservices with none of the isolation benefits.

### Strangler Fig

**What it is**: Incrementally replace a legacy system by routing individual capabilities (API endpoints, message flows, batch jobs) to a new implementation while the legacy continues handling the remaining load. The new system grows around the legacy — strangling it slice by slice — until the legacy has zero traffic and can be decommissioned.

**When it works**: The legacy system has an API surface (HTTP endpoints, queue listeners, event consumers) that can be individually identified and intercepted. The replacement can be built capability by capability rather than all at once.

**What it does NOT do**: Work when the legacy has no decomposable API boundary — a tightly coupled internal system with no external interface cannot be strangled without first wrapping it (see Strangler Vine or Anti-Corruption Layer).

**T-Mobile/middleware example**: Migrating from a TIBCO ESB integration layer to an event-driven Kafka architecture. The ESB currently handles 40 integration flows between BSS (billing/ordering systems) and OSS (provisioning/network systems).

Slice 1 (weeks 1–8): Customer profile reads. Build a new profile service backed by a read replica. Update the API gateway route for `GET /customer/{id}` to forward to the new service. Monitor for 30 days: SLO ≥ ESB baseline, error rate < ESB baseline. Confirm. ESB still handles the other 39 flows.

Slice 2 (weeks 9–16): Address updates. Build address update handler publishing to `customer.address.updated` Kafka topic. Route `PATCH /customer/{id}/address` to new service. Provisioning systems consume the Kafka event instead of polling the ESB queue. Prove for 30 days.

Continue slice by slice for all 40 flows. After the last flow is migrated and proven, the ESB has zero traffic. Decommission.

**Key governance mechanism**: Each slice requires a traffic rollout gate (see transformation-status-report-template.md). No slice advances to 100% traffic without 30-day SLO proof at canary. This turns an 18-month monolithic risk into a series of 30-day controlled experiments.

### Strangler Vine

**What it is**: Place a routing facade (API gateway, proxy) in front of the legacy system before any replacement is built. All consumers are migrated to call the facade. The facade initially forwards 100% of traffic to the legacy. As new services are built to replace individual capabilities, the facade's routing rules are updated to point those calls to the new service. The legacy is replaced behind the facade without consumers ever changing their integration point again.

**When it works**: Consumers need a clean, stable API contract immediately — they cannot tolerate the endpoint changes that happen during Strangler Fig routing updates. Also effective when the legacy has a meaningful API surface but consumers are too numerous to coordinate for multiple routing changes.

**T-Mobile/middleware example**: Adding a Kong API gateway in front of TIBCO ESB. ESB currently serves 15 upstream consumers, each with a direct integration. Every time the ESB is replaced slice by slice (Strangler Fig), each consumer must update their endpoint. With Kong:

1. Deploy Kong as the single entry point. All 15 consumers are migrated to call Kong endpoints (one-time migration effort).
2. Kong routes all calls to the ESB. From consumers' perspective: nothing has changed except the URL.
3. As ESB flows are replaced by new microservices, Kong routing rules are updated internally. Consumers are not involved.
4. The ESB receives progressively less traffic as Kong routing rules shift to new services.
5. ESB reaches zero traffic. Decommission ESB and Kong routing rules that pointed to it.

**Risk of misuse**: The Kong gateway itself becomes a single point of failure and a bottleneck if not sized and operated correctly. The facade must be production-grade from day 1 — it is not a temporary proxy.

### Anti-Corruption Layer (ACL)

**What it is**: A translation component that sits between the legacy system and the new system, converting the legacy's data model and semantics to the new system's domain model. Prevents the legacy's technical debt — its naming conventions, identifier formats, data types, and coupling assumptions — from being imported into the new system.

**When it works**: The legacy system has a data model that is technically functional but semantically corrupt relative to what the business actually means. Identifiers that are repurposed, formats that were hacks, column names that mean the opposite of what they say. Without an ACL, the new system would need to perpetuate these corruptions to interoperate.

**T-Mobile/middleware example**: The ESB integration layer uses `MSISDN` (Mobile Station Integrated Services Digital Network Number — the full 10-digit phone number) as the customer identifier throughout, stored as a string with no format validation. Routing keys, queue names, and message headers all use MSISDN as the customer correlation ID. The new customer service uses a UUID-based customer ID, because MSISDN is not stable (customers change numbers), is not unique (prepaid numbers are recycled), and is PII.

Without an ACL, the new customer service would need to accept MSISDN as an input, store the MSISDN-to-UUID mapping internally, and expose MSISDN in its outbound events. The corruption propagates.

With an ACL:
- Inbound: ACL receives ESB message with MSISDN, looks up or creates UUID mapping, forwards message to customer service with UUID.
- Outbound: ACL receives event from customer service with UUID, translates to MSISDN for legacy consumers that still require it.
- The customer service never sees an MSISDN. The ESB never sees a UUID.
- As legacy consumers are migrated to the new system, ACL outbound translation for those consumers is removed. Eventually the ACL is removed entirely.

**Risk of misuse**: ACLs that are built as permanent infrastructure (not as a transitional component) become technical debt themselves. The ACL's translation logic must be removed as soon as the consumers that required it are migrated.

### Big Bang Migration

**What it is**: Stop the legacy system, switch entirely to the new system in a planned cutover window, and do not run both systems simultaneously. All traffic moves from legacy to new in a single event, typically over a maintenance window or planned downtime.

**When it works**: Regulatory or contractual prohibition on parallel operation (double-billing risk, double-charging risk, dual-record compliance violations), or systems where there is no meaningful API boundary to enable any incremental approach. Also appropriate when the legacy system's data model is so entangled with its application logic that extraction slice by slice is technically impossible.

**What it does NOT do**: Give you rollback ability quickly. Rolling back from a Big Bang requires re-importing data accumulated in the new system back to the legacy — a process that typically takes longer than the original cutover.

**T-Mobile/middleware example**: Replacing a billing system that processes $2B/year in customer charges. State PUC (Public Utilities Commission) regulation prohibits operating two billing systems simultaneously because concurrent charge generation creates regulatory exposure for double-billing. Incremental migration is legally prohibited.

Approach:
- 12-month parallel development and integration testing phase. New billing system developed, tested against production-mirrored data, and validated against 18 months of historical billing runs.
- 3 full dress rehearsals of the cutover procedure, each timed and measured.
- 72-hour war room for the cutover weekend. All billing, SRE, and operations personnel on-site.
- Cutover sequence: freeze charge generation in legacy, export final state, import to new system, run reconciliation, validate, go live.
- 48-hour rollback capability maintained for 30 days post-cutover: legacy system preserved in read-only state, rollback script tested and ready.

**Risk of misuse**: Big Bang on any system where rollback must be achievable in under 4 hours. The rollback complexity makes this pattern unsuitable for systems where a failed cutover has immediate revenue or compliance impact beyond the cost of the maintenance window itself.

### Parallel Run

**What it is**: Run the legacy system and the new system simultaneously, processing the same inputs through both, and compare outputs. The legacy system remains authoritative — its output is what actually goes to downstream consumers — while the new system's output is validated against the legacy. Traffic moves to the new system only after output divergence reaches zero or an acceptable threshold.

**When it works**: Financial correctness, billing accuracy, or regulatory accuracy must be proven before cutover. The parallel run is the proof. Divergences between legacy and new system output are bugs to be fixed, not acceptable variance.

**What it does NOT do**: Work when the operational cost of running two systems for an extended period is prohibitive — cloud compute for two full production systems plus the comparison infrastructure is expensive, and finance will question running two billing systems for 60 days.

**T-Mobile/middleware example**: Replacing the revenue assurance system that validates all charges before billing runs. Every charge that goes to a customer bill passes through revenue assurance first. A bug in the new system that incorrectly passes or blocks charges has direct revenue and customer impact.

Parallel run approach:
- New revenue assurance system deployed alongside legacy, processing all transactions.
- Both systems receive the same transaction stream.
- A comparison service checks new system output against legacy output for every transaction.
- Any divergence is flagged, investigated, and categorized: new system is wrong (bug to fix), legacy is wrong (known issue, new system is correct), or both are acceptable (different but valid outcomes).
- After 60 days of zero unexplained divergences: confidence established. Traffic cut over to new system, legacy continues as shadow for 30 additional days post-cutover for final validation.

**Risk of misuse**: Running parallel indefinitely without a defined exit criterion. The parallel run must have a predetermined acceptance threshold (zero divergences in N days, or divergence rate < X%) and a hard cutover deadline once the threshold is met.

### DB Migration (Expand-Contract)

**What it is**: A two-phase approach to database schema changes that maintains zero downtime and full rollback ability throughout. Expand phase adds new structures without removing old ones. Contract phase removes old structures once all applications are using the new ones.

**When it works**: Renaming columns, changing data types, adding or removing constraints, splitting or merging tables — any schema change that would require application downtime if done in a single operation.

**What it does NOT do**: Handle major semantic model changes (splitting a god-table into a proper normalized schema, merging multiple schemas into one) without significant additional orchestration. Expand-contract handles structural changes; domain model rearchitecting requires re-architect pattern in combination.

**T-Mobile/middleware example**: Adding a UUID primary key to the customer table. The customer table currently uses MSISDN as the primary key — a 10-digit varchar. The business has decided UUIDs are the correct primary key because MSISDNs are recycled for prepaid accounts.

Expand phase (weeks 1–4):
1. Add `customer_uuid` column to the customer table (nullable).
2. Application code updated to write `customer_uuid` on every new customer record insert.
3. Backfill script runs to populate `customer_uuid` for all existing records (UUID generated deterministically from MSISDN for existing records, pure UUID for new records).
4. Application code updated to read from `customer_uuid` when present, fall back to MSISDN when null (during backfill window).
5. Add index on `customer_uuid`.

Contract phase (weeks 5–8, after backfill complete and verified):
1. Verify 100% of records have `customer_uuid` populated.
2. Update application to read `customer_uuid` exclusively. Remove MSISDN fallback path.
3. Add NOT NULL constraint to `customer_uuid`.
4. Update all foreign key references from other tables to use `customer_uuid`.
5. Remove MSISDN as primary key. Make `customer_uuid` the primary key.
6. MSISDN column retained for now — removed in a subsequent contract phase after dependent systems are confirmed migrated.

Rollback: at any point in the expand phase, the `customer_uuid` column can be dropped and the application reverted. After the NOT NULL constraint is added in contract phase, rollback requires a full application rollback plus constraint removal — still feasible but more complex.

**Key principle**: Never remove a column in the same deployment that adds its replacement. The expand phase and the contract phase must be separate deployments separated by a verification period. Violating this principle causes downtime.
