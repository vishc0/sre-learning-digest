# Modernization Assessment Playbook

Applications are never static. Every assessment is a snapshot of a system in motion — runtimes aging, dependencies drifting, ownership degrading. This playbook produces six outputs: a risk classification, a recommended modernization pattern, a cost estimate, a sequencing recommendation, and an ARB presentation artifact.

Run this playbook once per application entering the modernization pipeline. Reassess any application whose health score drops a tier between assessments.

---

## Phase 1: Application Baseline (Days 1–3)

Gather from the application registry, architecture diagrams, and team interviews. Do not proceed to Phase 2 until the baseline sheet is complete — gaps here propagate as errors through every downstream phase.

### What to gather

**Lifecycle and health**
- Current lifecycle stage: Active / Maintenance / Legacy / Sunset
- Current health score (from application scorecard, 0–100)
- Date of last health score assessment

**Technology stack**
- Primary language and version
- Framework(s) and version
- Runtime platform and version (JDK, Node.js, Python interpreter, .NET CLR)
- Middleware dependencies: message brokers (RabbitMQ, Kafka, TIBCO, MuleSoft), ESBs, integration platforms — version and license type
- Dependency currency: what fraction of declared dependencies are EOL or within 12 months of EOL?

**Infrastructure**
- Hosting platform: on-premise / bare metal / VMware / cloud (which provider/region) / Kubernetes (which cluster)
- Deployment model: manual / scripted / pipeline (tool name)
- Scaling approach: manual / scheduled / auto-scale (metric)
- Current resource allocation: compute (vCPUs, memory), storage

**Data**
- Datastores owned by this application: database name, type (RDBMS, NoSQL, object store, message queue), volume
- Data model complexity: simple CRUD / relational with business logic / graph / time-series
- Retention requirements: regulatory minimums, operational minimums
- Cross-system data sharing: does another application write directly to this application's database?

**Integration surface**
- Documented upstream dependencies (applications this calls)
- Documented downstream consumers (applications that call this)
- Verify both against telemetry: API gateway logs, service mesh, Splunk. Undocumented dependencies are the primary source of cutover failures.
- Message-based integrations: which queues, topics, or exchanges does this application produce and consume?

**Team**
- Team size and composition
- Knowledge concentration: how many people could perform an emergency deployment without assistance?
- Availability for a modernization workstream (percentage of capacity)
- Last major onboarding: when was the most recent person brought to independence on this system?

**Change history (trailing 12 months)**
- Deployments per month (average and trend)
- Incidents: P1 count, P2 count
- Change failure rate: failed deployments / total deployments
- Time to restore: average MTTR for P1/P2 incidents

### Output

Completed Application Baseline Sheet: one page, all fields above populated. No estimates — confirmed data only. Flag any field that could not be confirmed and note the source gap.

## Phase 2: Legacy Tax Calculation (Days 3–5)

The legacy tax is the annual cost of keeping the current system alive. It is the denominator of every business case ROI calculation. Calculate it conservatively — underestimating it weakens the case for investment.

### Infrastructure cost

Pull from cloud billing or FinOps platform:
- Annual compute cost (instances, container resource reservations)
- Annual storage cost (block, object, database)
- Annual networking cost (data transfer, load balancer hours)
- Annual license cost (middleware licenses, runtime licenses, vendor support contracts)

Total annual infrastructure cost = sum of the above.

### SRE time cost

Use a fully loaded hourly rate for senior SRE/engineer: confirm with finance. Typical range: $75–$150/hour loaded.

Categorize monthly SRE hours on this application:
- Routine operations: scheduled jobs, patching, certificate rotation, capacity management
- Incident response: time spent on incidents caused by or involving this application
- Change management: change window coordination, deployment execution, rollback events

Annual SRE time cost = (monthly hours × 12) × fully loaded hourly rate

### Incident cost

For each P1/P2 incident in the trailing 12 months:
- MTTR hours
- Responding team size (people on the bridge)
- Business impact (if quantifiable from incident record: transaction volume down, revenue/hour estimate)

Annual incident cost = sum over all incidents of: (MTTR × team size × hourly rate) + documented business impact

If business impact is not quantifiable, use: MTTR × team size × hourly rate only. Do not inflate with unverifiable estimates.

### Feature delivery velocity

Baseline metric: how long does it take to deliver a small feature (1–2 story points) from commit to production for this application vs. an equivalent modern service in your portfolio?

Express as a ratio: if the legacy system takes 3x as long as a comparable modern service, the velocity tax is real even if it is not directly monetized. Note this qualitatively — include it in the ARB narrative, not the financial calculation, unless your organization has a model for it.

### Legacy tax total

```
Annual legacy tax = infrastructure cost + SRE time cost + annual incident cost
```

This number drives the payback period calculation in Phase 3.

## Phase 3: Modernization Options Analysis (Days 5–10)

Evaluate every applicable pattern. Do not pre-select before completing this analysis — the patterns that seem wrong are often the ones that reveal the real constraints.

### Pattern evaluation template

For each pattern, complete:

| Field | Value |
|-------|-------|
| Applicable | Y / N / Conditional |
| Why applicable or not | [rationale] |
| Estimated investment | $[amount] ([N] engineering weeks × $[rate] + infrastructure delta) |
| Estimated timeline | [N] months |
| Expected annual savings post-modernization | $[amount] |
| Payback period | [investment ÷ annual savings] months |
| Risk level | Low / Medium / High |
| Primary risks | [list] |

### Pattern: Lift and Shift (Rehost)

Move the application to a new hosting platform without code changes. Typically: bare metal to cloud, or VMware to Kubernetes.

- Applicable when: the primary cost driver is infrastructure, not code or operational complexity
- Not applicable when: the application has significant manual operational procedures that will persist in the new environment
- Expected savings: primarily infrastructure cost reduction (cloud economics, right-sizing)
- Risk: lowest of all patterns; does not reduce technical debt

### Pattern: Re-platform

Move to a new runtime or managed service with minimal code changes. Example: moving a Java application from an on-premise app server to a managed Kubernetes deployment, or a TIBCO BW 5 flow to TIBCO BW 6 or MuleSoft.

- Applicable when: the hosting platform or middleware platform is the primary cost or risk driver, and the application logic is sound
- Not applicable when: the application has deep coupling to the legacy platform's proprietary APIs
- Expected savings: reduced infrastructure cost, reduced middleware licensing, some operational simplification
- Risk: low-to-medium; the re-platforming effort often surfaces hidden platform coupling

### Pattern: Re-architect (Refactor)

Restructure the application internals: decompose a monolith, replace a synchronous integration with event-driven, restructure the data model, rewrite a legacy service in a modern language.

- Applicable when: the velocity tax is the primary driver; the current architecture cannot deliver at the pace the business needs
- Not applicable when: team capacity is insufficient for concurrent operate-and-rebuild, or when the business case is primarily cost-driven (re-architect rarely pays back faster than 18–24 months)
- Expected savings: primarily velocity and incident reduction; infrastructure savings vary
- Risk: high; scope creep and parallel operation complexity are the primary failure modes

### Pattern: Strangler Fig

Incrementally route traffic from the legacy system to new services, domain by domain, until the legacy system receives no traffic and can be decommissioned.

- Applicable when: there is a clean API boundary (API gateway, reverse proxy) in front of the legacy system, and the application can be decomposed into identifiable domains
- Not applicable when: there is no API boundary and adding one would require significant legacy code changes; or when callers connect directly to the application's database
- Decomposition plan: identify domains in order of least-coupled to most-coupled; start with the outermost domain
- Risk: medium; the parallel operation period is the risk zone — data consistency between old and new system requires explicit management

### Pattern: Strangler Vine

An API facade is introduced in front of the legacy system. New capabilities are built behind the facade. Over time, the facade routes requests to new implementations while the legacy system continues to handle existing traffic it was not yet replaced for.

- Applicable when: the Strangler Fig is blocked by insufficient API boundary; Strangler Vine adds the API layer as the first step
- Not applicable when: the API boundary would require changes deep in legacy consumers (they are tightly coupled to legacy-specific contracts)
- Distinction from Strangler Fig: Strangler Vine builds the vine (API facade) first; Strangler Fig assumes the vine already exists (API gateway already present)
- Risk: medium-high; the facade itself becomes a dependency that must be kept current during the migration

### Pattern: Big Bang

Full cutover from legacy to replacement on a planned date. Legacy goes dark; new system carries all traffic.

- Applicable when: the modernization has a hard external deadline (regulatory, vendor end-of-support), no other pattern is technically feasible, and the application tier tolerates a planned maintenance window
- Not applicable when: the application is Tier 1 or 2 with no downtime tolerance, or when data reconciliation from a failed cutover would take more than 4 hours
- Risk: highest of all patterns; rollback from a failed cutover is the most expensive recovery scenario in any modernization program

## Phase 4: Pattern Selection (Days 10–12)

Apply this decision framework in order. The first applicable rule wins. If multiple rules apply, the most restrictive rule governs.

### Decision rules

1. **Downtime tolerance zero?** Rule out Big Bang unless no other pattern is technically feasible. Document the infeasibility rationale explicitly.

2. **Clean API boundary exists (API gateway or reverse proxy in front of the application)?** Strangler Fig and Strangler Vine are both eligible. Prefer Strangler Fig if the facade already exists; Strangler Vine if it must be built.

3. **Data model is the primary complexity driver?** Whichever pattern is selected must include Dual-Write or Expand-Contract as a sub-pattern for the data migration phase. This is not optional for applications with > 1M records or cross-system data sharing.

4. **Regulatory parallel operation requirement exists?** Parallel Run is required as a sub-phase of any incremental pattern. Document the regulatory requirement and the acceptance criteria.

5. **Transformation driver is primarily cost reduction?** Prefer Lift and Shift or Re-platform as a first move (immediate savings, lower risk), with Re-architect deferred to a later wave.

6. **Transformation driver is primarily velocity or capability?** Re-architect or Strangler Fig required. Cost-first patterns will not achieve the velocity goal.

### Selection rationale template

Document the pattern selection with explicit rationale. This text goes directly into the ARB artifact.

```
Selected pattern: [pattern name]

Rationale:
1. [First reason — tied to a specific decision rule above]
2. [Second reason — tied to application characteristics from Phase 1]
3. [Third reason — tied to risk or dependency findings]

Patterns ruled out:
- [Pattern]: [reason ruled out]
- [Pattern]: [reason ruled out]
```

## Phase 5: Sequencing Analysis (Days 12–15)

Where this application fits in the transformation sequence is as important as which pattern to use. A technically perfect plan that starts in the wrong order will stall because its dependencies are not ready.

### Dependency analysis

For this application, answer:

1. **What applications does this depend on that must be modernized first?** If this application integrates with a legacy system that will break the integration contract during its own modernization, this application cannot be modernized until that dependency stabilizes.

2. **What applications depend on this and cannot proceed until this is modernized?** If multiple downstream systems are blocked behind this one, this application gets sequencing priority regardless of its own health score.

3. **What foundational infrastructure must exist before this transformation can start?** Common foundational dependencies: an API gateway capable of traffic splitting (required for Strangler Fig/Vine), an event bus at the required throughput (required for event-driven re-architecture), a Kubernetes cluster with the required policy controls (required for containerized re-platform targets).

4. **Is this application a sequencing blocker for 3 or more other applications?** If yes, it moves to Wave 1 or 2 regardless of other factors.

### Dependency graph

Produce an ASCII dependency graph:

```
[upstream-dependency-A] ──> [THIS APPLICATION] ──> [downstream-consumer-1]
[upstream-dependency-B] ──/                    \──> [downstream-consumer-2]
                                                \──> [downstream-consumer-3]
```

Note which arrows represent blocking dependencies (modernization of the upstream must complete before this application's modernization begins) vs. parallel-safe dependencies (can modernize concurrently with appropriate integration contract management).

### Sequencing recommendation

| Field | Value |
|-------|-------|
| Recommended wave | Wave [N] |
| Reason for wave assignment | [sequencing rationale] |
| Dependencies that must complete first | [list] |
| Applications unblocked by completing this | [list] |
| Recommended start | Q[N] [year] |
| Recommended completion | Q[N] [year] |
| Foundational infrastructure required | [list or "none"] |

### ARB submission checklist

Before submitting the ARB artifact, confirm:

- [ ] All Phase 1 baseline fields confirmed (no estimates)
- [ ] Legacy tax calculated with documented hourly rate and source data
- [ ] All patterns evaluated, not just the recommended one
- [ ] Pattern selection rationale is specific, not generic
- [ ] Dependency graph reviewed with owning teams of dependency applications
- [ ] Payback period calculation reviewed by finance partner
- [ ] Top 3 risks reviewed with on-call lead and security
- [ ] Sequence recommendation reviewed with portfolio owner
