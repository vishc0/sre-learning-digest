# Legacy Triage Playbook

Use this playbook when a legacy system surfaces without a current owner, when a new incident reveals previously unknown legacy risk, or when beginning a portfolio audit. The goal is a risk classification, immediate risk mitigation actions, and a modernization priority recommendation — produced within 24 hours of the trigger event.

This is not a full modernization assessment. That comes later. Triage answers: how dangerous is this system right now, and what do we do in the next 24 hours?

---

## Phase 1: Discovery (Hours 0–4)

Gather from application registry, Splunk, service mesh, API gateway, and team interviews. Work in parallel across all sources — do not wait for the registry to be clean before querying telemetry. Registries lie; traffic does not.

### Identity

- Application name and any aliases (legacy systems frequently have multiple names across teams, tickets, and dashboards)
- Hosting platform and location: on-premise data center, cloud account/region, bare metal rack
- Deployment method: manual (human runs commands), scripted (scripts exist, human runs them), pipeline (automated), unknown
- Codebase location: repository URL, or mark as unknown if not found

If codebase location is unknown, check:
1. Internal source control (GitLab, GitHub Enterprise) — search by application name and known team names
2. Artifact repository — search for build artifacts, find the source reference
3. Team interview — ask the person most likely to have deployed it last

### Ownership

- Current documented owner: person name, team name, last confirmed date
- Last known active owner if current documentation is stale or the documented owner has left the organization
- Business stakeholder: who calls when this goes down? Who owns the business process this application supports? This person may not be in engineering.

If ownership is unclear: check PagerDuty/on-call schedule for which team was paged during the last incident involving this system. That team is the de facto owner regardless of what the registry says.

### Technical state

Gather with minimal access requirements. Do not wait for a full environment setup.

- Runtime and version: Java 8, TIBCO BusinessWorks 5.x, Node.js 12, Python 2.7 — specifics matter. "Old Java" is not a finding; "JDK 8u181, EOL January 2019" is.
- Framework and version: Spring Boot 1.5, BW 5.12, Express 4.16
- Last deployment date: check deployment pipeline history, change management records, or artifact timestamps
- Last dependency update date: check lockfile commit history, or Maven/npm dependency dates in the artifact

CVE assessment — run if accessible, do not block triage on it:
```
# Container image scan
trivy image [image-name]:[tag]

# Dependency scan (if source accessible)
snyk test --all-projects

# Quick NIST check for known EOL runtime
# Search: https://nvd.nist.gov/products/cpe/search
```

If scanning tools are not accessible within the 4-hour window, note "CVE assessment pending" and continue. The runtime EOL status is sufficient for the initial risk score.

### Operational state

- On-call rotation: is this application in any on-call rotation? Query PagerDuty, OpsGenie, or equivalent for alert rules referencing this application's host names, service names, or log indices.
- Who is paged when it fails: name the team and the escalation path.
- Runbook existence: does a runbook exist? Yes or No — do not assess quality during triage.
- SLO: is an SLO defined? What is the target? Is it currently being met? Pull from monitoring platform; if not instrumented, mark as "no SLO defined."
- Incident history (trailing 12 months): pull from incident management system. Count: P1 incidents, P2 incidents, total. Note any incidents where root cause was attributed to this application.

### Dependency surface

Pull from service mesh (Kiali/Linkerd), API gateway logs, and Splunk. You need three data points:

1. What does this application call? (Its upstream dependencies)
2. What calls this application? (Its downstream consumers)
3. Rough consumer count: fewer than 5 / 5–20 / more than 20

Consumer count matters for triage because it determines the blast radius and the migration complexity. A fragile system with 2 consumers is a contained risk. The same system with 40 consumers is a portfolio-level risk.

```
# Splunk: identify inbound callers from service mesh or access logs
index=[mesh-log-index] dest_service="[application-name]" earliest=-30d
| stats dc(src_workload) as unique_callers, count as total_calls by src_workload
| sort -total_calls

# API gateway: filter by target service
# (query syntax varies by gateway — Kong, Apigee, AWS API GW each differ)
# Output: caller identity, call count, last call date
```

## Phase 2: Risk Classification (Hours 4–8)

Score the system on four criteria. Score each criterion 0–3. Total score maps to a legacy classification.

Be conservative — when evidence is ambiguous, score the higher risk value. You can revise down after Phase 1 is complete; you cannot un-escalate a system that caused an outage because it was scored too low.

### Scoring criteria

**Technical currency (0–3)**

| Score | Condition |
|-------|-----------|
| 3 | Runtime(s) are end-of-life. Critical unpatched CVEs present (CVSS >= 9.0). Vendor support contract expired or non-existent. |
| 2 | Runtime(s) reaching end-of-life within 12 months. High CVEs pending remediation (CVSS 7.0–8.9). Vendor support on last year. |
| 1 | Runtimes currently supported, end-of-life > 12 months away. Some dependency currency issues but no high CVEs. |
| 0 | All runtimes and dependencies current. No outstanding CVEs of concern. Vendor support current. |

**Knowledge concentration (0–3)**

| Score | Condition |
|-------|-----------|
| 3 | Fewer than 2 people can operate and deploy this system. Codebase is effectively undocumented. No runbook or runbook is known to be wrong. |
| 2 | 2–3 people can operate it, but at least one is a single point of failure. Documentation is partial. Runbook exists but untested. |
| 1 | Small team (3–5), most members have operating knowledge. Adequate documentation. Runbook exists and has been used. |
| 0 | Multiple team members can independently deploy, operate, and troubleshoot. Complete documentation. Runbook reviewed within 12 months. |

**Change risk (0–3)**

| Score | Condition |
|-------|-----------|
| 3 | No automated tests. No deployment pipeline (manual deploy). Incident history confirms high blast radius — failures in this system have caused cascading failures in other systems. |
| 2 | Minimal test coverage (< 30%). Semi-automated deploy (scripts exist, human-run). Incident history shows coupling — some cascading failures but not systematic. |
| 1 | Some test coverage (30–70%). Automated deployment pipeline exists and is used. Blast radius appears isolated based on incident history. |
| 0 | Full test coverage (> 70%). Fully automated deployment pipeline. Independent deployment confirmed — changes to this system have not caused downstream incidents. |

**Operational burden (0–3)**

| Score | Condition |
|-------|-----------|
| 3 | More than 2 P1/P2 incidents in the trailing 12 months attributable to this system. Or: this system accounts for > 20% of the on-call team's incident response time. |
| 2 | 1–2 P1/P2 incidents in trailing 12 months. Or: 10–20% of on-call team's incident response time. |
| 1 | No P1/P2 incidents in trailing 12 months. < 10% of on-call team's incident response time. Some P3 incidents or routine operational overhead. |
| 0 | No incidents in trailing 12 months. Minimal on-call mention. Essentially self-operating within defined parameters. |

### Legacy classification

| Total score | Classification | Description |
|-------------|---------------|-------------|
| 0–3 | L1 — Stable Legacy | Low risk, manageable. Monitor and plan for eventual modernization. |
| 4–6 | L2 — Knowledge-Concentrated Legacy | Manageable risk, but bus-factor exposure. Knowledge preservation is the immediate action. |
| 7–9 | L3 — Unstable Legacy | Elevated operational risk. Change freeze and protective wrapping are immediate responses. |
| 10–12 | L4 — Critical and Fragile | System is a portfolio-level risk. Emergency response required. |

### Classification record

Document the scores with evidence:

```

```

## Phase 3: Immediate Risk Mitigation (Hours 8–24)

Act within 24 hours of completing the classification. These actions do not wait for a full modernization assessment. They are immediate stabilization measures.

### L1 — Stable Legacy

1. Document the triage finding in the application registry: classification = L1, triage date, triage lead.
2. Update lifecycle stage to "Legacy" in the application registry if not already set.
3. Add to portfolio health dashboard: classification, health score, next recommended assessment date (12 months).
4. No escalation required. No change freeze. No immediate engineering action.

### L2 — Knowledge-Concentrated Legacy

1. Assign a knowledge preservation task to the owning team within 1 week: at minimum — a deployment runbook review, an architecture decision record for the top 3 non-obvious design choices, and a "bus factor drill" where a second person deploys to staging without assistance.
2. If no owner is currently assigned: escalate to the director of the team that last owned it within 48 hours. Do not leave L2 systems without an assigned owner.
3. Flag bus factor risk in the portfolio dashboard. Label: "Knowledge Concentrated — preservation required."
4. Add to quarterly SRE capacity review: is this system consuming disproportionate on-call time from a small number of people?

### L3 — Unstable Legacy

1. Escalate to VP within 24 hours. Provide the classification record and the triage report.
2. Initiate change freeze: no changes to this application except incident remediation, approved by VP on a case-by-case basis. Notify the owning team and their director in writing.
3. Begin protective wrapping assessment: can an API facade be placed in front of this system within 30 days without touching the application itself? An API facade limits the blast radius of failures by giving you a single traffic control point. Assign an architect to assess within 5 business days.
4. Add to top-5 modernization priority list. This system enters the modernization assessment pipeline at the next planning cycle, not the next annual planning cycle.
5. Initiate a stakeholder notification to the business owner: "This system has been classified as Unstable Legacy. We have frozen changes and are assessing protective measures. Modernization planning is beginning. You will receive an update within 30 days."

### L4 — Critical and Fragile

1. VP notification within 4 hours of classification. Do not wait for the 24-hour window.
2. Director escalation within 24 hours.
3. Convene an emergency modernization pod assessment: is this system a candidate for a dedicated modernization team with executive sponsorship, bypassing the normal portfolio sequencing? The answer is yes if: (a) the system is Tier 1 or 2, and (b) the probability of a major outage within 6 months is assessed as high given the technical currency and change risk scores.
4. Immediate war gaming session (within 48 hours): what happens if this system fails for 72 hours? Participants: on-call lead, business stakeholder, transformation director. Output: documented contingency plan (manual processes, compensating controls, downstream system behavior during outage).
5. Change freeze effective immediately, pre-dating the VP call. No changes without VP approval.
6. Document the contingency plan in Confluence and link it from the incident runbook.
7. Add to the next week's executive briefing: "We have an L4 classification in the portfolio. Here is the risk, here is what we are doing, here is what executive sponsorship is needed."

## Phase 4: Modernization Priority Recommendation

Use four inputs to generate a priority recommendation. This recommendation feeds directly into the next planning cycle. It is not a commitment to a timeline — it is a position in the modernization queue.

### Inputs

1. **Legacy classification**: L1–L4 from Phase 2
2. **Consumer count**: fewer than 5 / 5–20 / more than 20 — higher consumer count = more systems blocked, higher priority
3. **Application tier**: Tier 1 and 2 rank higher than Tier 3 and 4 at the same classification
4. **Legacy tax**: from the modernization assessment if already run; if not, use a rough estimate from Phase 1 infrastructure data

### Priority matrix

| Classification | Tier 1 or 2 | Tier 3 or 4 |
|---------------|-------------|-------------|
| L4 | P0 — Emergency queue: bypass normal sequencing, executive sponsorship required | P1 — Next wave: enter the next wave planning cycle immediately |
| L3 | P1 — Next wave: enter next wave planning cycle | P2 — Current roadmap: add to current roadmap for assessment and planning |
| L2 | P2 — Current roadmap: schedule modernization assessment within 90 days | P3 — Backlog: schedule modernization assessment within 6 months |
| L1 | P3 — Backlog: monitor, assess at next annual portfolio review | P4 — Monitor: no action required beyond annual health check |

Adjust the priority upward by one level if:
- Consumer count > 20 (system is a central dependency)
- The system is blocking 3 or more planned modernizations from starting
- The legacy tax is in the top quartile of the portfolio
