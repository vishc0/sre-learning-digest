# Application Decommission Playbook

Decommission is the final phase of every transformation. It is also the phase most frequently skipped or abandoned mid-execution. An application that is 90% decommissioned — traffic gone but infrastructure and alerts still live — continues to consume SRE attention, generates ghost alerts, and carries regulatory risk on data that should have been archived.

This playbook drives decommission from approval through post-decommission confirmation. All phases are required. The order is fixed. Do not proceed to the next phase until the current phase exit criteria are met.

---

## Prerequisites

Before this playbook is triggered, confirm all of the following. If any are not met, do not begin.

- [ ] Replacement system: 90 consecutive days of SLO compliance in production confirmed, with evidence from monitoring platform
- [ ] Gate: Cutover (100%) → Decommission-pending passed and logged in the transformation governance record
- [ ] VP sign-off obtained for Tier 1 and Tier 2 applications
- [ ] Decommission entry added to the portfolio decommission calendar with target date
- [ ] Change record created: type = Decommission, tier = [application tier], application = [name], target date = [date]

---

## Phase 1: Consumer Discovery (Weeks 1–2)

Run all four discovery methods. Do not substitute one for another and do not skip any. Each method catches a different class of consumer.

- API gateway logs catch: well-behaved consumers using the documented endpoint
- Service mesh telemetry catches: internal service-to-service calls that bypass the gateway
- DNS query logs catch: consumers that look up the hostname but may use unusual ports or protocols
- Application log scanning catches: consumers that identify themselves in request headers or log patterns

A consumer missed in discovery is a consumer that breaks on decommission day.

### Step 1.1: API gateway log analysis

Query your API gateway for the trailing 90 days. Adapt the query to your gateway (Kong, Apigee, AWS API Gateway, Nginx, TIBCO Mashery):

```
Target service filter: [application-name] or [application-hostname]
Time window: last 90 days
Aggregate by: caller identity (service account, mTLS CN, API key owner), weekly call volume
Output columns: caller_id, total_calls, calls_last_week, last_call_date
Sort: by last_call_date descending
```

Flag consumers with `last_call_date` within 90 days. Flag separately consumers with `last_call_date` between 90 and 180 days ago — these are potential shadow consumers with infrequent access patterns (quarterly batch jobs, annual reporting jobs).

### Step 1.2: Service mesh telemetry

In Kiali (Istio) or Linkerd Viz:
- Navigate to the workload graph for [application-name]
- Set time window: 90 days (or maximum available window if < 90 days)
- Select: inbound traffic view
- Export: source workload name, namespace, call count, last active timestamp

For Istio via CLI if the UI is not available:

```bash
kubectl -n [namespace] exec deploy/[application-name] -- \
  curl -s http://localhost:15000/stats/prometheus \
  | grep "istio_requests_total.*destination_service=\"[application-name]\""
```

### Step 1.3: DNS query log analysis

```
# Splunk — adapt index and field names to your DNS logging configuration
index=dns_logs (query="*[application-name]*" OR query="*[application-hostname]*") earliest=-90d
| stats count as query_count, max(_time) as last_query_time by src_ip, query
| eval last_query_date=strftime(last_query_time, "%Y-%m-%d")
| sort -query_count
| table src_ip, query, query_count, last_query_date
```

For each source IP with queries in the last 90 days: resolve to hostname or service identity. IP addresses without hostname resolution require manual investigation — do not assume they are inactive.

### Step 1.4: Application log scanning

```
# Splunk — adapt index and host pattern to your environment
index=[app-log-index] host="[application-name]-*" earliest=-90d
| rex field=_raw "(?i)(caller|client|source|x-forwarded-for|user-agent)[=: ]+(?<caller_id>[^\s,\]]+)"
| stats count as call_count, max(_time) as last_call_time by caller_id
| eval last_call_date=strftime(last_call_time, "%Y-%m-%d")
| where call_count > 0
| sort -call_count
```

Also scan for: HTTP `User-Agent` headers that identify internal services, `X-Caller-Service` or equivalent internal headers, mTLS certificate common names in access logs.

### Consumer master list

Compile all four discovery method outputs into a single consumer master list:

| Consumer name | Owner | Team | Contact | Discovery source(s) | Call volume (90d) | Last call date | Shadow consumer risk |
|---------------|-------|------|---------|--------------------|--------------------|----------------|---------------------|
| ... | ... | ... | ... | Gateway / Mesh / DNS / Log | ... | YYYY-MM-DD | Y/N |

Shadow consumer risk = Y if: last call was more than 60 days ago, or call volume is very low (< 10 in 90 days), or call pattern suggests a scheduled batch job rather than real-time use.

## Phase 2: Migration Plan and Consumer Notification (Weeks 2–4)

### Step 2.1: Per-consumer migration document

For each consumer in the master list:

| Field | Value |
|-------|-------|
| Consumer name | |
| Owner name | |
| Owner contact (email + Slack) | |
| Capability used | What endpoint, operation, or data the consumer depends on |
| Migration path | Exact replacement endpoint, authentication method, payload changes if any |
| Migration complexity | Low (config change only) / Medium (code change, < 1 week) / High (code change, > 1 week) |
| Migration support contact | Name and contact for migration questions |
| Target migration complete date | Decommission date minus 30 days |

### Step 2.2: Formal decommission notice

Send to each consumer's owner. Use your organization's standard communication channel (email + Slack + Confluence page). The notice must be unambiguous about the date and the action required.

```
Subject: [Application Name] Decommission Notice — Migration Required by [date]

[Application Name] will be decommissioned on [decommission target date].

Your service [consumer name] has been identified as a consumer of [Application Name].
Action required: migrate to the replacement system before [migration deadline date = 
decommission date - 30 days].

REPLACEMENT SYSTEM
Name: [replacement system name]
Endpoint: [replacement endpoint URL or message topic/queue]
Authentication: [auth method and how to obtain credentials]
Migration guide: [link to detailed migration guide]
API differences: [link to changelog or delta document, if applicable]

SUPPORT
Migration support contact: [name], [email], [Slack handle]
Office hours for migration questions: [days/times]

ACKNOWLEDGMENT REQUIRED
Reply to this notice with your migration plan (start date, target completion date, 
assigned engineer) within 5 business days.

Questions: [contact name and email]
```

### Step 2.3: Acknowledgment tracking

Track acknowledgments in the consumer master list. Add columns:

| Consumer | Notice sent date | Acknowledgment received | Migration plan confirmed | Migration complete |
|----------|-----------------|------------------------|--------------------------|-------------------|

Escalation schedule for non-acknowledgment:
- T-7 days from acknowledgment deadline: Slack reminder to the consumer team lead
- T-14 days from acknowledgment deadline: escalate to the consumer team's engineering manager
- T-21 days from acknowledgment deadline: escalate to the consumer team's director and notify your VP

A consumer that does not acknowledge is not a consumer that can be assumed gone. It is a consumer that is not paying attention. Treat it as active until proven otherwise.

## Phase 3: Consumer Migration Support (Weeks 4–12)

Track migration status weekly. The decommission lead runs weekly check-ins with any consumer that has not confirmed migration complete.

### Migration status tracker

| Consumer | Owner | Status | Last update | Blocker | Escalated |
|----------|-------|--------|-------------|---------|-----------|
| [name] | [owner] | Not started / In progress / Complete | YYYY-MM-DD | [blocker or none] | Y/N |

Status definitions:
- **Not started**: consumer acknowledged, no engineering work begun
- **In progress**: consumer has begun code or config changes
- **Complete**: consumer has confirmed in writing that production traffic has moved to the replacement, verified by API gateway logs showing calls to replacement, not legacy

### Migration complete verification

Do not accept a self-declaration of "we're done" as migration complete. Verify:

1. API gateway logs for the replacement system show calls from the consumer after the reported migration date.
2. API gateway logs for the legacy system show zero calls from the consumer after the reported migration date (allow a 7-day window for the transition to stabilize).
3. Consumer confirms in writing (email or Slack with timestamp): "Migration of [consumer name] to [replacement] is complete as of [date]."

### Consumers that cannot complete migration by deadline

If a consumer cannot complete migration by the deadline due to a legitimate technical blocker (not a resource or priority issue):

1. Consumer's director requests an exception from your VP. The request must include: the specific technical blocker, a revised migration date, and interim mitigations (e.g., traffic routing changes, fallback logic).
2. VP grants or denies. Denial is valid — the decommission date can be enforced even over a consumer's objection if the exception is not substantively justified.
3. If granted: document the exception with the revised date. The consumer moves to the top of the weekly check-in list.

Do not proceed to Phase 4 until all consumers are at status = Complete, or have a VP-approved exception with a documented revised migration date that is before the decommission date.

## Phase 4: Traffic Drain Validation (Weeks 12–16)

Run all four discovery methods from Phase 1 again, in full. This is not a shortcut step. You are confirming that the consumer migration tracking is accurate against actual traffic data.

All four discovery methods must show zero traffic to the legacy application for 30 consecutive days before proceeding to Phase 5.

### Zero traffic confirmation dashboard

```
# Splunk: daily traffic count for the legacy application
# If this query returns any results, traffic is still flowing
index=[app-log-index] host="[application-name]-*" earliest=-30d
| timechart count span=1d
| where count > 0
```

```
# API gateway: filter for legacy application target, last 30 days
# Zero rows = zero traffic
[adapter this to your gateway query language]
target_service=[application-name] AND timestamp > [30 days ago]
```

```
# Service mesh: check for any inbound traffic to the workload
# In Kiali: workload inbound graph, last 30 days — should show no inbound edges
# In Linkerd: route table for [application-name] — all routes at 0 RPS
```

### If traffic is found after Phase 3

1. Identify the source consumer immediately. This is not optional — do not defer identification.
2. Initiate emergency consumer discovery for the new caller: contact the owning team within 24 hours, confirm they were missed in Phase 1 discovery, send them the same decommission notice with an expedited migration timeline.
3. Document how this consumer was missed in Phase 1: which discovery method should have caught them and why it did not. This feeds a retrospective improvement to the discovery process.
4. Do not proceed to Phase 5 until this new caller has migrated and the 30-day zero traffic window is reset.

## Phase 5: Data Archival and Migration (Weeks 8–16)

Run in parallel with Phases 3 and 4. Data work is typically the longest-lead item — start it early, not after traffic is fully drained.

### Step 5.1: Data inventory

Enumerate every datastore owned or written to by the application:

| Datastore | Type | Location | Volume | Retention requirement | Data owner |
|-----------|------|----------|--------|-----------------------|------------|
| [name] | RDBMS / NoSQL / Object / Queue / Cache | [host or URL] | [records or GB] | [regulatory minimum] | [name] |

Do not rely on documentation alone. Cross-reference with:
- Application configuration files: look for database connection strings, S3 bucket names, queue names
- Cloud resource tags: tag value = application name
- Network flow logs: what IP addresses or DNS names did the application connect to on database ports?

### Step 5.2: Per-datastore disposition

For each datastore, assign one of three dispositions:

**Migrate to replacement**: data is actively used by the replacement system.
- Confirm migration completeness with record count check: source count must equal destination count before the source can be considered migrated.
- Confirm data integrity with a sample validation: select N random records, compare source to destination.
- Sign-off from data owner: "Migration of [datastore] is complete and verified."

**Archive**: data is not actively used but must be retained for regulatory or audit purposes.
- Copy to cold storage (S3 Glacier, Azure Archive, tape — per your organization's policy)
- Generate archive manifest: storage location, file format, encryption method, retrieval command, retention end date
- Test retrieval immediately after archival: actually execute the retrieval command and confirm a sample of records is recoverable. Do not skip this step — an archive that has never been tested is not a working archive.
- Sign-off from data owner and compliance: "Archive of [datastore] is complete, retrievable, and meets retention requirements."

**Delete**: data has no active use and no retention requirement.
- Confirm with legal/compliance before deleting. Get confirmation in writing.
- Delete only after all other dispositions are complete and the above confirmation is obtained.

### Step 5.3: Regulatory sign-off

For any application processing regulated data (financial transactions, PII, healthcare, audit records):

- Obtain written sign-off from legal/compliance that the archival plan meets retention requirements.
- Document the sign-off with the compliance contact name, date, and reference to the relevant regulatory requirement.
- Attach to the decommission change record.

This sign-off is a gate for Phase 6. Do not begin infrastructure teardown until it is obtained for regulated data.

## Phase 6: Infrastructure Teardown (Weeks 16–18)

Teardown order is fixed. Do not deviate. Each step has a confirmation gate before proceeding.

### Teardown sequence

**Step 1: DNS**
Change DNS records for all [application-name] hostnames to NXDOMAIN, or to a redirect pointing at a "service decommissioned" status page hosted separately. Do not simply delete the DNS record — NXDOMAIN is more informative than NXDOMAIN, and a redirect allows you to log who is still querying.

Monitor DNS query logs for 48 hours after DNS removal. Any queries indicate a consumer using this hostname that was missed in Phase 1 discovery. Identify and contact them.

**Step 2: Load balancer**
Remove the application from all load balancer target groups and listener rules. Confirm no health check traffic is still flowing to application instances.

Gate: DNS query logs show zero queries for 48 hours after Step 1.

**Step 3: Application pods / instances**
Terminate application Pods (Kubernetes) or instances (VMs/EC2). For Kubernetes: delete the Deployment, then the Services, then the ConfigMaps and Secrets referencing the application. For VMs: snapshot first, then terminate.

Gate: load balancer shows zero active connections to application targets.

**Step 4: Databases**
Take a final snapshot before any database is deleted. This is mandatory, not optional — the snapshot cost is negligible and the recovery value is non-zero.
- RDS / Aurora: take a final manual snapshot, label it "pre-decommission-[date]"
- Self-hosted: pg_dump / mysqldump to S3 before dropping the database
- Delete the database after snapshot is confirmed complete

Gate: confirm with monitoring that no application is still writing to the database. Check for any scheduled jobs that may run outside of normal traffic patterns.

**Step 5: Object storage**
Confirm all objects have been either migrated to the replacement or archived. Move remaining objects to archive storage if applicable. Then delete the bucket/container. Verify the bucket/container does not exist before proceeding.

**Step 6: Queues and message brokers**
Drain and delete any application-specific queues, topics, or exchanges. Confirm consumer count is zero before deleting. For Kafka: delete the topic only after confirming consumer group offset is at the end of the partition (no unread messages).

**Step 7: Network configuration**
Remove security group rules, VPC firewall rules, or on-premise firewall rules that were created specifically for this application. Remove any dedicated subnets or VLANs if they are now empty.

**Step 8: IAM and service accounts**
Revoke and delete: service accounts, IAM roles, API keys, mTLS certificates, Vault leases, and any credentials created for the application. Rotate any shared credentials the application was using.

### Teardown gate per step

Before each step, confirm in monitoring: the component being removed shows zero traffic for > 24 hours. "Zero traffic" means: zero inbound connections (for load balancer / DNS), zero write operations (for databases and queues), zero read operations (for object storage that is not archive).

## Phase 7: Monitoring Cleanup (Weeks 16–18)

Run in parallel with Phase 6. Complete before Phase 8.

Leaving orphaned alerts active after decommission is a specific kind of technical debt: it desensitizes on-call teams through alerts that fire for a system that no longer exists, erodes trust in the alerting platform, and creates confusion during future incidents.

### Monitoring cleanup checklist

**Splunk**
- [ ] Identify dashboards: search for `title=*[application-name]*` in Splunk dashboard library. Archive, do not delete. Retain for 1 year.
- [ ] Identify saved searches: search for saved searches referencing `host=[application-name]-*` or `index=[app-specific-index]`. Disable all. Do not delete immediately — disable for 90 days, then delete.
- [ ] Identify alerts: same search scope. Disable. Do not delete for 90 days (in case an incident reveals the alert was catching something that should have been caught).
- [ ] If the application had a dedicated index: mark the index as frozen (no new data), set retention to the regulatory minimum for historical data, do not delete the index.

**Grafana / Prometheus**
- [ ] Identify dashboards: search by application name tag or annotation. Archive (export to JSON, store in repository). Then unpublish.
- [ ] Identify alert rules in Prometheus or Alertmanager: remove alert rules referencing this application's metrics labels. Push the updated rules file to the configuration repository.
- [ ] Remove recording rules if any were created specifically for this application.

**PagerDuty / OpsGenie / VictorOps**
- [ ] Remove [application-name] from all escalation policies.
- [ ] Delete or disable services created specifically for this application.
- [ ] Remove any routing rules that referenced this application's alert names.
- [ ] Confirm: no escalation policy paths lead to a deleted service.

**Synthetic monitoring**
- [ ] Disable uptime checks for all [application-name] endpoints (Pingdom, Datadog Synthetics, AWS CloudWatch Synthetics, or equivalent).
- [ ] Remove any SLA reports that reference these checks.

**Certificate monitoring**
- [ ] Remove [application-name] certificates from certificate expiry monitoring. An alert for an expired certificate on a decommissioned service is noise.

## Phase 8: Post-Decommission Confirmation (Weeks 18–20)

30-day monitoring period post-teardown. This phase is not optional — it confirms that the decommission is clean and closes the risk.

### 30-day confirmation checklist

Run at the end of the 30-day window:

- [ ] Zero incidents in the incident management system referencing [application-name] in the last 30 days
- [ ] Zero consumer complaints received about the decommissioned service
- [ ] Zero DNS queries to [application-name] DNS entries (if retained as NXDOMAIN, check query logs)
- [ ] Zero references to [application-name] in current on-call alerts (PagerDuty/OpsGenie — search alert title and body)
- [ ] Data archival retrieval test (if archived data): execute the retrieval procedure from the archive manifest and confirm records are accessible

### Closure actions

1. Close the decommission change record. Resolution note: "Application decommissioned. Infrastructure removed. Data archived per manifest [link]. 30-day monitoring period complete with no incidents or consumer issues."

2. Update the application registry:
   - Lifecycle stage: Decommissioned
   - Decommission date: [date]
   - Archive manifest link: [link]
   - Replacement system: [name and link]

3. Remove from active portfolio reporting. The application should no longer appear in portfolio health dashboards, capacity planning, or SLO compliance reports.

4. Update the portfolio decommission calendar entry: status = Complete.

5. Report to VP — this is a required deliverable, not optional:

```
APPLICATION DECOMMISSION COMPLETE

Application: [name]
Decommission date: [date]
Replacement: [replacement system name]

SAVINGS REALIZED
Infrastructure savings: $[amount]/month ($[amount]/year)
SRE capacity returned: [N] hours/month
License savings (if applicable): $[amount]/month

RISK RETIRED
CVE exposure: [N] critical CVEs no longer in scope
EOL runtime: [runtime name] version [version] removed from portfolio
Compliance scope reduction: [Y/N — describe if applicable]

DECOMMISSION METRICS
Consumers migrated: [count]
Data archived: [volume]
Time from decommission approval to completion: [N] weeks vs. [N] weeks planned

No incidents occurred during the decommission process.
[Or: One incident occurred — [brief description] — resolved by [action].]
```

The VP report closes the transformation record. The transformation started with an ARB approval and ends with this report. Both ends must exist for the governance record to be complete.
