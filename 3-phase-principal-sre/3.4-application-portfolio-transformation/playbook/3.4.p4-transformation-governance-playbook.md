# Transformation Governance Playbook

A transformation without governance is a project that starts and never finishes. This playbook defines the operating rhythm for in-flight transformations: weekly health checks, phase gate criteria, escalation triggers, and cutover decision authority.

Governance applies from the moment ARB approval is issued through the day the decommission change record is closed.

---

## Weekly Transformation Health Check

Every in-flight transformation gets a weekly health check. The transformation lead completes it by end-of-business Monday. The VP reviews the portfolio dashboard by end-of-business Tuesday.

The health check is not a status meeting. It is a structured data collection that either confirms the transformation is proceeding normally or triggers an escalation response. Do not conflate the two.

### Red flag triggers

Any single trigger requires VP notification within 24 hours. Do not wait for the next scheduled review cycle.

| Trigger | Threshold | Action |
|---------|-----------|--------|
| New system SLO compliance | < 99% for 3 consecutive days | VP notification + traffic rollback assessment |
| Data reconciliation match rate | < 99.9% (standard) or < 100% (financial/regulated) | Immediate VP notification + route traffic to 0% on new system |
| Budget overrun | > 20% of plan to date | Finance notification within 24h + VP notification |
| Phase duration overrun | > 30% of planned phase duration | Written root cause from transformation lead within 48h |
| Unresolved blocker | Not resolved within 5 business days | Escalate to VP |

## Phase Gate Criteria

Phase gates are mandatory. No phase is considered complete until exit criteria are confirmed in writing by the transformation lead. No phase is skipped.

The transformation lead documents gate passage in the transformation log with: date, gate name, exit criteria met (checklist), and authorizing signature(s).

### Gate: Design → Build

Exit criteria:
- [ ] Architecture design document reviewed and approved by architecture team
- [ ] Data migration strategy documented (if applicable)
- [ ] API contract for new system defined and reviewed by consumer team leads
- [ ] Non-functional requirements confirmed: latency targets, throughput targets, SLO targets
- [ ] Environment plan confirmed: staging environment available and matches production configuration
- [ ] Team onboarded: all engineers assigned to the transformation have completed design review

### Gate: Build → Test

Exit criteria:
- [ ] Unit test coverage >= 80% (measured and reported by CI pipeline)
- [ ] Integration tests implemented for all upstream and downstream dependency contracts
- [ ] Performance benchmark in staging: within 20% of legacy baseline on P99 latency and throughput
- [ ] Security scan passing: no critical CVEs introduced by new application code or new dependencies
- [ ] No open critical or high defects
- [ ] Code review completed for all components

### Gate: Test → Canary (10%)

Exit criteria:
- [ ] All integration tests passing in staging
- [ ] Performance benchmark in staging: within 10% of legacy baseline (tighter than Build → Test gate)
- [ ] Rollback mechanism tested and verified: traffic can be routed to 0% on new system in < 5 minutes, tested in staging
- [ ] Dashboards and alerts deployed for new system in production (do not go to production without observability)
- [ ] On-call team trained on new system: each on-call member can execute incident response procedures without assistance
- [ ] Runbook complete for new system: covers deployment, rollback, and top 5 failure scenarios
- [ ] Canary traffic percentage configured in API gateway / traffic controller: verified capable of precise 10% routing

### Gate: Canary (10%) → Progressive (50%)

Minimum soak period: 7 days at 10%.

Exit criteria — all must be sustained for the full 7-day window, not just on the assessment day:
- [ ] Error rate: <= legacy baseline for all 7 days
- [ ] P99 latency: within 20% of legacy for all 7 days
- [ ] Zero data anomalies detected (data reconciliation clean if parallel run active)
- [ ] Zero incidents caused by or primarily involving the new system during the 7-day window
- [ ] No rollbacks executed during the 7-day window

Required sign-offs:
- [ ] Transformation lead
- [ ] On-call lead (confirms team confidence in operating the new system at higher traffic)

### Gate: Progressive (50%) → Cutover (100%)

Minimum soak period: 14 days at 50%.

Exit criteria — all must be sustained for the full 14-day window:
- [ ] All Canary gate metrics sustained at 50% traffic
- [ ] Load test completed at 110% of peak projected traffic: passed without SLO violation
- [ ] No rollbacks executed during the 14-day window
- [ ] Data reconciliation match rate maintained at threshold for full 14 days (if parallel run active)

Required sign-offs:
- [ ] Transformation lead
- [ ] On-call lead
- [ ] VP Engineering (Tier 1 and Tier 2 applications)

### Gate: Cutover (100%) → Decommission-pending

Minimum soak period: 30 days at 100%.

Exit criteria:
- [ ] SLO compliance >= target for all 30 days
- [ ] Zero incidents caused by or primarily involving the new system in the last 14 days of the 30-day window
- [ ] Data reconciliation confirmed clean at end of parallel run (if applicable)
- [ ] Legacy system traffic confirmed at zero for 30 days (API gateway logs, service mesh)
- [ ] Parallel run formally closed with sign-off from data owner and compliance (if applicable)

Required sign-offs:
- [ ] Transformation lead
- [ ] VP Engineering (Tier 1 and Tier 2 applications)

## Escalation Triggers and Response Procedures

### Escalation: Transformation stall

Definition: phase duration > 30% of planned phase duration.

Response sequence:
1. Transformation lead provides written root cause analysis within 48 hours of trigger. Root cause must identify: what is blocking progress, why it was not identified in planning, and what is required to unblock.
2. VP reviews root cause. Determine category:
   - **Resource-resolvable**: additional engineering capacity, tooling access, or infrastructure can clear the blocker within a defined timeline.
   - **Plan-wrong**: the original pattern, timeline, or approach was incorrect. Resources alone will not resolve it.
3. If resource-resolvable: VP approves resource allocation. Set new milestone with a hard deadline. Add weekly VP check-in for the duration.
4. If plan-wrong: convene ARB review within 5 business days. ARB options: approve a revised plan with the same pattern, approve a pattern change, defer the transformation pending foundational work. Document the decision as an amendment to the original ARB record.

### Escalation: Budget overrun > 20%

Response sequence:
1. Finance partner notified within 24 hours of the overrun being identified.
2. Transformation lead produces a cost-to-complete estimate within 48 hours: what is the remaining investment needed to reach Decommission-pending?
3. VP decision within 5 business days:
   - Approve additional budget: update the ARB record with revised investment figure.
   - Descope: identify which components can be deferred to a follow-on effort without compromising the core transformation. Document what is deferred and when it will be addressed.
   - Pause: halt the transformation at the current phase boundary. Document restart criteria explicitly — do not pause without defining what must change before resuming.
4. Any pause exceeding 30 days requires a new ARB review before restart.

### Escalation: Repeated SLO failures on new system

Definition: new system SLO compliance falls below threshold in the same phase on two separate occasions.

Response sequence (first failure):
1. Route traffic back to 0% on the new system immediately. Do not wait for root cause analysis to start the rollback.
2. Begin root cause analysis. Target: root cause identified within 24 hours.
3. Do not re-route traffic to the new system until root cause is identified and fix is confirmed in staging.

Response sequence (second failure in same phase):
1. Route traffic to 0% immediately.
2. Convene architecture review within 5 business days. Second failure in the same phase indicates either a structural defect in the new system's design or an incorrect traffic model in staging.
3. Architecture review options: approve a fix with revised testing requirements before re-entering the phase, or recommend escalation to pattern change.
4. Document the architecture review decision in the ARB record.

## Cutover Decision Authority

Cutover is the moment all production traffic moves to the new system. It is not reversible in the same way a partial traffic shift is. Decision authority reflects this.

| Tier | Decision authority | Notice required | War room |
|------|--------------------|-----------------|----------|
| Tier 3/4 | Transformation lead | 48 hours to stakeholders | Optional |
| Tier 2 | VP Engineering approval | 5 business days to stakeholders | Recommended |
| Tier 1 | VP Engineering + Director sign-off | 10 business days to stakeholders | Required, 4 hours post-cutover |

For Tier 1 cutover:
- Formal change record: type = Major, risk = High, back-out plan documented
- War room scheduled for 4 hours post-cutover with: transformation lead, on-call lead, data owner, business stakeholder representative
- Pre-cutover checklist reviewed and signed off by each required approver independently
- Rollback plan tested in staging within 7 days of cutover date

## Parallel-Run Governance

Parallel run is the period during which both the legacy system and the new system process transactions, and outputs are compared to confirm the new system produces correct results.

### Entry criteria for parallel run

Parallel run is not a default sub-phase — it is required when:
- Data correctness is a regulatory requirement (financial transactions, healthcare data, audit records)
- The application performs complex calculations where output correctness cannot be verified through standard SLO metrics alone
- The legacy system uses undocumented business logic that must be verified by comparison, not specification

If none of these conditions apply, skip parallel run. Operating both systems in parallel is expensive and introduces its own complexity.

### Parallel-run acceptance criteria

Define acceptance criteria before entering parallel run. Acceptance criteria must be:
- Specific: "99.99% of transaction outputs match within $0.01" not "outputs are correct"
- Measurable: automated reconciliation job running on a defined schedule
- Time-bounded: acceptance criteria must be met within the parallel-run window

### Parallel-run maximum duration

90 days. This is a hard ceiling.

If acceptance criteria are not met within 90 days, the VP convenes a decision meeting. The two possible outcomes are:

1. **New system has a structural defect**: acceptance criteria are correct and the new system cannot meet them. Outcome: pause the parallel run, fix the defect, restart from the Test gate. Do not extend the parallel run — extending it treats a design problem as an operational problem.

2. **Acceptance criteria are wrong**: the criteria were set without full understanding of the legacy system's behavior (known data quality issues in the legacy system, acceptable variance in certain output types, etc.). Outcome: revise the acceptance criteria with documented rationale, obtain sign-off from the data owner and compliance, resume parallel run against revised criteria.

No parallel run extends past 90 days without explicit executive escalation and a documented decision that falls into one of the two categories above.

## Transformation Portfolio Dashboard

The VP reviews a portfolio-level dashboard weekly. Each row is one in-flight transformation.

### Dashboard fields per transformation

| Field | Description |
|-------|-------------|
| Application | Name and tier |
| Current phase | Design / Build / Test / Canary / Progressive / Cutover / Parallel-run / Decomm-pending |
| Phase health | Green (on track) / Yellow (at risk) / Red (triggered) |
| Traffic % | Current percentage on new system |
| SLO compliance | Trailing 7-day compliance % |
| Budget | Consumed vs. plan (%) |
| Phase duration | Actual vs. planned days in current phase |
| Active blockers | Count |
| Next milestone | Description and date |
| Red flags | Count of active red flag triggers |

### Portfolio dashboard thresholds

Green: all health check fields within threshold, no red flags, phase duration <= planned.

Yellow: one or more health check fields within 10% of threshold, or phase duration 15–30% over planned, or one active blocker open < 5 days.

Red: any red flag trigger active, or phase duration > 30% over planned, or two or more blockers open > 5 days.

A portfolio with more than 25% of in-flight transformations in Red status requires a VP-level portfolio review within 5 business days. Portfolio-level Red is a signal that the transformation program has a systemic problem — capacity, dependency resolution, or planning methodology — not just individual application issues.
