# Chapter 5: Application Sunsetting and Decommission

## 1. Why Sunsetting Is a VP-Level Responsibility

Every application you do not decommission is a permanent operational liability. It consumes infrastructure, occupies on-call attention, expands the security surface, and adds to the cognitive load that every engineer carries when reasoning about the system landscape. Zombie applications — systems with no active users that are still running — are the most wasteful form of technical debt because they consume resources while delivering no value.

Engineers do not have the authority to decommission applications. Decommissioning a Tier-1 or Tier-2 system requires coordinated sign-off from engineering, compliance, legal, finance, and product. Only a VP or Director has the organizational reach to drive that coordination. An engineering team that identifies a zombie application will raise it, put it in the backlog, and watch it remain running for 3 more years because nobody with authority has made the decision to act.

"We'll decommission it later" is the exact sentence that creates zombie applications. Later requires governance to arrive. Without a VP-owned decommission pipeline with a defined review cadence, later never comes.

The decommission rate is a VP-level KPI: how many applications are successfully retired per year. A healthy modernizing organization decommissions at least as many applications as it adds. Net portfolio growth — adding more applications than you decommission — is a leading indicator of zombie accumulation and future operational overhead.

## 2. Sunsetting Criteria

An application is a candidate for sunsetting when it meets these four criteria:

**Primary — replacement proven stable**: the replacement system has demonstrated 90 days of SLO compliance in production under real traffic. Not under synthetic load or in a staging environment. The 90-day requirement is not arbitrary — it ensures the replacement has been through at least one monthly billing cycle, one month-end batch job run, one seasonal traffic variation, and an adequate number of deployment cycles to confirm the operational model is sound.

**Traffic — zero active users or confirmed full migration**: all consumers have been confirmed migrated to the replacement. "Confirmed" means telemetry verification, not self-reporting by consuming teams. A consumer team that says "we've migrated" but whose service mesh traffic shows continuing calls to the legacy system has not migrated.

**Regulatory — no retention requirement mandating operation**: legal and compliance confirm that no regulation requires the system to remain operational or accessible in its current running form. This is distinct from data retention — the data can be archived while the system is decommissioned.

**Data — all operational data migrated or archived**: every dataset owned by the application has been either migrated to the replacement system with reconciliation confirmed, or archived to cold storage per the applicable retention policy. No data is left in the running application's datastore without a documented disposition plan.

**Zombie application identification**: an application that shows zero traffic for 90 consecutive days and is not in an approved shutdown-pending state automatically enters the decommission review queue. This is not an automatic decommission — it is an automatic trigger for the question "why is this still running?" The answer determines next action: proceed to decommission, document a legitimate reason for continued operation, or confirm the application is no longer needed and accelerate the formal decommission process.

## 3. The Eight-Phase Decommission Process

Do not abbreviate this process. Each phase catches failures that the previous phase does not.

### Phase 1: Consumer Discovery (Weeks 1–2)

Identify every system that calls or depends on the application. Use all four discovery methods described in section 4. Document every consumer found: consuming system name, team owner, call volume (daily average and peak), criticality of the dependency, migration path to replacement.

A consumer discovery that takes one week is probably incomplete. Consumer discovery for a Tier-1 application with a 5-year history should take 2 weeks minimum, including the time to run discovery queries across 90 days of telemetry data.

### Phase 2: Migration Plan Communication (Weeks 2–4)

Formal written notification to every discovered consumer: "Application X will be decommissioned on [target date]. Migration path: [specific steps]. Support contacts: [names and channels]. Office hours for migration assistance: [schedule]."

Minimum notice periods:
- Tier 1/2 applications: 90 days minimum
- Tier 3/4 applications: 60 days minimum

Notification must be sent to the technical lead and the business owner of each consuming team, not only to the engineering distribution list. A consuming team that misses a decommission notification because it went to an unmonitored email alias is a shadow consumer incident waiting to happen.

Acknowledgment is required. Unacknowledged notifications escalate to the consumer team's VP at the 30-day mark.

### Phase 3: Consumer Migration Support (Weeks 4–12)

Active support for consumers migrating to the replacement. Track migration status per consumer in the decommission record: not started, in-progress, validated, confirmed via telemetry. Do not rely on consumer self-reporting — confirm migration via service mesh telemetry showing zero calls to legacy endpoints from each consumer's service identity.

Do not proceed to Phase 4 until all consumers have reached "confirmed via telemetry" status or have been formally escalated (consumer team refuses to migrate and the escalation is documented and approved at VP level).

### Phase 4: Traffic Drain Validation (Weeks 12–16)

Zero traffic verification — all four data sources must show zero traffic for 30 consecutive days:

1. API gateway logs: zero requests to legacy endpoints
2. Service mesh telemetry: zero TCP/HTTP connections from any service to the legacy application
3. Application logs: zero inbound requests recorded
4. DNS query logs: zero DNS resolutions of the legacy application's hostname

If any data source shows non-zero traffic after the 30-day window, Phase 1 repeats — there is a consumer that was not previously discovered. Do not interpret remaining traffic as noise. Every call is a consumer.

### Phase 5: Data Archival and Migration (Weeks 8–16, overlaps with Phases 3–4)

Data handling follows three paths, and each dataset requires a documented decision:

- **Migrated to replacement**: data moved to the new system's datastore, with reconciliation validation confirming row count, checksum, and business logic correctness. Not "we ran the migration script" — confirmed accurate via reconciliation.
- **Archived to cold storage**: data retained for regulatory compliance, moved to cold storage (AWS Glacier, Azure Archive, equivalent) with a documented retrieval procedure and documented retention expiry date.
- **Deleted**: data that has passed its retention period or has no regulatory hold. Deletion requires sign-off from legal and compliance. Document the deletion: what data, when deleted, who authorized.

No data can be deleted from the running application's datastore without documented sign-off. No data can be "archived" without a tested retrieval procedure.

### Phase 6: Infrastructure Teardown (Weeks 16–18)

Remove compute, networking, storage, and configuration in reverse provisioning order. If the application was provisioned via Terraform or another IaC tool, the teardown is `terraform destroy` against the application's module. If it was provisioned manually, document every resource removed and the removal sequence.

Save the infrastructure configuration as a documentation artifact before teardown — not to redeploy, but as a historical record of what existed, for compliance and audit purposes.

Specific teardown checklist:
- Compute: terminate EC2 instances, EKS pods, or equivalent
- Networking: remove load balancer listeners, security group rules specific to this application, DNS records
- Storage: confirm data archival is complete, then delete EBS volumes, S3 buckets, RDS instances
- Certificates: revoke or archive TLS certificates associated with the application
- IAM: remove IAM roles and policies specific to this application (do not remove shared policies)
- Secrets: rotate and then deactivate secrets associated with this application in the secrets manager

### Phase 7: Monitoring and Alerting Cleanup (Weeks 16–18, parallel with Phase 6)

Remove or archive every monitoring artifact associated with the decommissioned application:

- Splunk dashboards and saved searches
- Datadog/Dynatrace monitors and dashboards
- PagerDuty or equivalent on-call routing rules and escalation policies
- Grafana dashboards and alert rules
- Service mesh service entries and virtual services

Leaving orphaned alerts is an operational hazard. Orphaned alerts for a decommissioned system generate noise. Noise desensitizes on-call engineers. Desensitized engineers miss real alerts. Alert hygiene is not housekeeping — it is signal quality protection.

### Phase 8: Post-Decommission Confirmation (Weeks 18–20)

30-day post-decommission monitoring period:

- Zero incidents traced to the decommissioned application
- Zero consumer complaints attributable to the decommission
- Zero shadow consumers discovered after the fact (no calls to the now-removed endpoints generating 404s from the gateway)
- No regulatory or compliance inquiries about data that should have been retained but was not

At the end of the 30-day period, close the decommission change record with a final confirmation entry. This is the audit artifact that legal, compliance, and internal audit will reference.

## 4. Consumer Discovery Methods

Use all four methods in parallel during Phase 1. Each method discovers consumers that the others miss. A consumer missed in Phase 1 is a shadow consumer incident in Phase 4.

**API gateway logs**: who has called this API in the last 90 days. Query by endpoint path and return caller service identity, call volume, and last call timestamp. Catches: authorized consumers routing through the gateway. Misses: consumers that have direct network access to the application and bypass the gateway (a legacy pattern in many T-Mobile on-prem environments), and consumers using a cached response from an intermediary rather than calling the application directly.

**Service mesh telemetry** (Istio/Linkerd): shows actual TCP and HTTP traffic at the network layer, capturing all in-cluster service-to-service calls regardless of whether they route through the API gateway. Query the mesh telemetry for inbound connections to the application's service endpoint over 90 days. Catches: direct in-cluster service-to-service calls. Misses: consumers outside the mesh — on-premises batch servers, third-party integrations, services in other clusters not in the same mesh federation.

**DNS query logs**: what systems have resolved this application's hostname over the last 90 days. This is a broad net that catches any consumer using DNS (which is nearly all of them). Query the DNS resolver logs or Route 53 query logs for the application's FQDN. Catches: any consumer using the hostname, including consumers outside the cluster. Misses: consumers using hardcoded IP addresses (a legacy anti-pattern that still exists in older on-premises integrations).

**Application log scanning**: search the application's own logs for inbound request sources. This catches consumers that all other methods miss because it reads from the application's own record of who called it.

```splunk
index=app_logs host="[application-name]-*" earliest=-90d
| rex field=_raw "(?<caller_ip>\d+\.\d+\.\d+\.\d+)"
| rex field=_raw "caller-service-id: (?<caller_service>[^\s]+)"
| stats count as call_count, max(_time) as last_call by caller_ip, caller_service
| sort -call_count
```

This query returns every unique caller by IP and service identity with call volume and last call time. The last_call timestamp identifies stale consumers (called once 85 days ago, not since) vs. active consumers (called 1000 times in the last week).

## 5. The Shadow Consumer Problem

Shadow consumers are consumers discovered after decommission has begun — or worse, after the system has been taken offline. They are the most operationally dangerous failure mode in the decommission process because they surface as production incidents: a system that depends on the decommissioned application begins failing with connection errors or 404s, and the immediate response is to restore the decommissioned application.

**How shadow consumers survive the discovery phase**:

- **Low-frequency batch jobs**: a job that runs monthly or quarterly will not appear in a 30-day traffic window. The job fails silently, nobody notices, and the next run is 3 months after decommission — after all alerts and monitoring have been cleaned up.
- **Infrastructure outside the service mesh**: batch servers, on-premises systems, ETL jobs running in a data warehouse cluster, FTP/SFTP file transfer jobs. These are not covered by Istio telemetry.
- **Vendor integrations documented only in vendor systems**: a third-party vendor that calls the application's API is documented in the vendor's integration configuration, not in the application's dependency documentation.
- **Deployed versions that are ahead of the source code registry**: the deployed binary still calls the legacy API even though the source code in the repository has been updated to call the replacement. The deployment did not complete, or a rollback is running in production.
- **File-based integrations**: consumers that pick up files from a shared directory, SFTP server, or S3 bucket rather than calling an API. These do not appear in API gateway logs or service mesh telemetry.

**Prevention**:

- Extend the traffic observation window to 90 days minimum to catch monthly and quarterly batch jobs.
- Include non-API traffic in discovery: FTP server access logs, S3 bucket access logs, shared directory access logs, MQ consumer lists (which queues does this application publish to, and who are the confirmed consumers of those queues?).
- Require written acknowledgment from vendor integration owners, not just distribution list notification.
- Query the job scheduler (Control-M, Airflow, cron on known batch servers) for any jobs that reference the decommissioned application's hostname or endpoint.

**Post-decommission shadow consumer protocol**: if a shadow consumer is discovered after the application is taken offline, restore service immediately — treat the decommissioned application as in a hold state, not deleted. Issue a P2 incident. Repeat consumer discovery with a 90-day extended window and non-API traffic sources included. Update the decommission plan with the newly discovered consumer and restart Phase 3 (migration support) for that consumer.

Never proceed to infrastructure teardown (Phase 6) until Phase 4 has been clean for 30 days. A shadow consumer discovered during Phase 4 is recoverable. A shadow consumer discovered after Phase 6 (infrastructure destroyed) is a production outage with no fast path to recovery.

## 6. Data Retention During Decommission

Regulatory data retention requirements are not optional, are not negotiable, and are not the engineering team's decision. Get legal and compliance sign-off on the data disposition plan before any data handling begins in Phase 5.

**Retention requirements by industry and data type** (verify current requirements with compliance — these are reference baselines, not authoritative):

| Industry | Data Type | Typical Retention Requirement |
|----------|-----------|-------------------------------|
| Telecom | Billing records | 7 years (US, varies by state) |
| Telecom | Call detail records | 2 years (CALEA and state law vary) |
| Telecom | 911 location data | 2 years (FCC requirement) |
| Finance | Transaction records | 7 years (SOX) |
| Finance | Customer communications | 3–7 years (varies by communication type) |
| Finance | Trade records | 7 years (SEC 17a-4) |
| Healthcare | Adult patient records | 7–10 years (varies by state) |
| Healthcare | Minor patient records | Until age of majority + 7 years (varies by state) |

**Archival strategy**: data retained for regulatory compliance goes to cold storage (AWS Glacier, Azure Archive, or on-premises equivalent) rather than remaining in the active datastore. Cold storage reduces cost by 90%+ compared to active database storage while satisfying the retention requirement.

**The retrieval procedure must be tested before the application is decommissioned**. Archiving data to Glacier and documenting "archived" in the decommission record is not compliance. Compliance means: if a regulator, auditor, or legal proceeding requires retrieval of this data within the required timeframe, can you retrieve it? Test the retrieval procedure on a representative sample before Phase 6 begins. Document the retrieval time (Glacier Expedited retrieval: 1–5 minutes; Standard: 3–5 hours; Bulk: 5–12 hours) and confirm it meets the regulatory requirement for the jurisdiction.

**Cold storage governance**: archive is not fire-and-forget. Assign an owner — the team that owned the decommissioned application retains accountability for the archive until its retention period expires. Document the retention expiry date in the decommission record. Implement automated deletion when the retention period expires, or require annual documented review and sign-off to extend. Orphaned data archives with no assigned owner and no expiry date are a compliance liability.

## 7. Sunsetting Governance

Decommissioning has a governance structure because it requires decisions and sign-offs that cross organizational boundaries. Engineering can identify that a system is ready for decommission. Engineering cannot unilaterally execute the decommission for a Tier-1 system.

**VP sign-off is required for**:

- All Tier 1/2 application decommissions — no exceptions
- Any decommission where consumer migration is not 100% confirmed via telemetry (a partial migration with a documented exception requires VP acknowledgment of the residual risk)
- Any decommission executed within 12 months of a major P1 incident on the application (increased likelihood of undiscovered dependencies or unresolved data issues)

**Formal consumer notification SLA**:

- Tier 1/2: 90-day minimum written notice with required acknowledgment
- Tier 3/4: 60-day minimum written notice with required acknowledgment
- Unacknowledged notifications: escalate to consumer team's VP at 30-day mark

**Change management record**: all decommissions require a formal change record that covers all 8 phases. The change record is not closed until Phase 8 (post-decommission confirmation) is complete. The change record is the audit artifact — it documents who made each decision, when, and with what authorization. Compliance, legal, and internal audit will ask for this record when they have questions about a decommissioned system.

## 8. The Decommission Calendar

The decommission calendar is the portfolio artifact that tracks all planned and in-flight decommissions. It is owned by the VP or Director of Engineering and reviewed quarterly at the portfolio review.

**Decommission calendar columns**:

| Column | Description |
|--------|-------------|
| Application | Name and Tier |
| Lifecycle Stage | Sunset-Pending, In-Decommission (Phase N), Complete |
| Target Decommission Date | Planned completion of Phase 8 |
| Current Phase | 1–8 |
| Consumer Count | Total discovered consumers |
| % Consumers Migrated (Telemetry) | Confirmed via service mesh, not self-reported |
| Data Disposition | Not Started / In Progress / Archived / Validated |
| Blockers | Specific blockers preventing phase completion |
| VP Sign-off Status | Pending / Approved / Not Required (Tier 3/4) |
| Actual Completion Date | Filled in when Phase 8 closes |

**Review discipline**: the decommission calendar is reviewed quarterly. Applications that have slipped their target decommission date by more than one quarter without a documented reason are escalated — meaning the blocker is surfaced at the VP level and a specific resolution plan is required.

**The decommission rate KPI**: track the number of applications successfully decommissioned per year (Phase 8 complete). Compare against the number of net new applications added to the portfolio in the same period. If decommissions are fewer than additions, the portfolio is growing and zombie accumulation risk is increasing. If decommissions equal or exceed additions, the portfolio is stable or shrinking — the correct direction for a modernizing organization.

A portfolio that has been growing without decommission for 5+ years will have a large backlog of zombie and near-zombie applications. Clearing this backlog requires a dedicated decommission program — dedicated engineering time, a program manager, and VP-level prioritization — not just adding decommissions to existing team backlogs.
