# Chapter 07: Technology Currency and EOL Governance

Technology currency is not a compliance checkbox. It is the operational foundation that determines whether your team can respond to CVEs in hours, upgrade dependencies without architectural rework, and maintain vendor escalation paths when production burns. The Director or VP who does not govern technology currency is accumulating a debt that compounds silently until it surfaces as a breach, a compliance audit finding, or a P1 with no vendor support path.

---

## 1. What Technology Currency Means

Technology currency measures how current an application's technology stack is relative to actively supported versions. An application with high currency runs on runtimes, frameworks, libraries, and infrastructure that receive vendor security patches and have active support contracts. An application with low currency runs on deprecated or end-of-life (EOL) components.

Currency is not about running the latest version. It is about staying within the vendor support window. The support window is the period during which the vendor publishes security patches for known vulnerabilities.

**Example:** Java 17 (LTS) is fully supported through September 2026 under Oracle's standard support model. Java 11 (LTS) standard support ended September 2023. Running Java 11 in production in 2025 means running a runtime with no published security patches for known JVM vulnerabilities. That is low currency — not because 11 is "old," but because it no longer receives patches.

Currency is a spectrum, not a binary. A team running Java 11 with a commercial extended support contract is in a different position than a team running Java 11 with no support agreement. Know the difference and track it explicitly.

### Why Currency Is a VP-Level Concern

**Security exposure.** EOL software receives no security patches. CVEs identified after EOL are never patched. Every CVE database query that returns a finding against an EOL component returns an unfixable finding — the only resolution is upgrading off the EOL version.

**Vendor escalation loss.** When a P1 incident involves EOL software, vendor support either refuses engagement or operates at reduced SLA. The T-Mobile SRE dealing with a RabbitMQ cluster failure on a deprecated version has no escalation path with the vendor. The incident timeline extends.

**Dependency lock-in.** EOL components block upgrades of dependent components. A service running on Mule Runtime 3.x cannot consume MuleSoft connectors built for Runtime 4.x. Upgrading the connector requires upgrading the runtime. If the runtime upgrade is a 6-month project, the connector upgrade is also a 6-month project. Cascading currency debt compounds.

**Compliance and insurance risk.** SOC 2, PCI DSS, ISO 27001, and cyber insurance frameworks increasingly require demonstrable evidence that software receives security patches. EOL components directly contradict this requirement. Audit findings and insurance claim denials are the downstream consequences.

## 2. End-of-Life Governance

EOL governance is a risk management function, not a technical housekeeping task. Track when software versions and platforms reach end of vendor support. Assign ownership. Plan upgrades before the deadline — not after.

### The EOL Risk Timeline

Manage every EOL finding against this timeline, working backward from the EOL date:

| Milestone | Activity |
|-----------|---------|
| T-12 months | EOL identified in quarterly scan. Upgrade plan drafted. Owner assigned. Budget request submitted for next fiscal year. |
| T-6 months | Upgrade in active development. Non-production environment upgraded and validated. Production cutover schedule confirmed. |
| T-3 months | Production upgrade complete. Post-upgrade stability window in progress. |
| T-0 (EOL date) | All production systems must be on supported versions. |
| T+30 days | Any remaining EOL component is a policy violation. Automatic escalation to VP. |

**The T+30 rule is not negotiable.** Running past T-0 without VP-approved exception and documented risk acceptance is a policy violation. "We're working on it" is not a risk acceptance. A written document signed by the VP stating: "We are running [component] at EOL, the risk is [X], mitigation is [Y], target remediation date is [Z]" — that is a risk acceptance.

### Exception Process

When a team cannot meet the T-0 deadline:
1. Engineering Director formally requests an EOL exception from the VP no later than T-3 months
2. Exception request includes: root cause of delay, risk assessment, specific mitigation controls active during the exception period (e.g., WAF rule to block CVE exploit vectors), remediation target date
3. VP approves or denies. Approved exceptions are tracked in the EOL risk register with the new target date
4. Security lead reviews active mitigation controls monthly during the exception period
5. Exception does not extend past 6 months without board-level risk acceptance

## 3. EOL Categories

Four categories, each with distinct risk profiles and governance responses.

### Runtime EOL

The language runtime loses vendor support. The underlying interpreter, JVM, or execution engine stops receiving security patches.

Examples: Java 11 (standard support ended Sept 2023), Python 3.8 (EOL Oct 2024), Node.js 14 (EOL Apr 2023).

Impact: CVEs identified in the runtime itself accumulate without fixes. JVM vulnerabilities, Python interpreter flaws, and V8 engine exploits all go unpatched. For a high-traffic service, runtime CVEs represent the highest-severity exposure because they affect every request.

Upgrade path: runtime upgrades typically require testing for behavioral differences in language semantics (Python 3.8 to 3.11 has deprecation changes; Java 11 to 17 has module system enforcement changes). Budget 4–8 weeks for a non-trivial service to validate runtime upgrade in non-production.

### Dependency EOL

A library or framework used by the application reaches EOL. The dependency is embedded in the application code, not the infrastructure layer.

Examples: Spring Boot 2.x (EOL Nov 2023), Log4j 1.x (EOL 2015 — and the source of Log4Shell), OpenSSL 1.0.x, Struts 2.3.x.

Impact: CVEs in dependencies are more exploitable on average than runtime CVEs because dependencies have more application-specific attack surface. Log4Shell was a dependency EOL consequence — Log4j 1.x had a known RCE vulnerability with no patch because the version was EOL.

Dependency EOL is more dangerous than runtime EOL when the dependency has internet-facing exposure — any dependency that processes user-provided input or handles HTTP requests.

Upgrade path: dependency upgrades often have breaking API changes across major versions. Spring Boot 2.x to 3.x requires Java 17 (runtime upgrade dependency) and significant changes to auto-configuration, security configuration, and Actuator endpoints. A Spring Boot 2.x to 3.x migration should be scoped as a medium transformation — 2–4 months for a well-factored service.

### Infrastructure EOL

The infrastructure platform loses vendor support. Kubernetes versions, managed service versions (RDS engine versions, ElastiCache engine versions), and container base images reach EOL.

Examples: Kubernetes 1.24 (EOL in EKS), Amazon EKS Kubernetes version support windows (typically 14 months), Ubuntu 20.04 LTS (EOL April 2025).

Impact: infrastructure security patches stop. Kubernetes CVEs at EOL versions cannot be patched without upgrading the cluster version. EKS will eventually force-upgrade clusters on EOL versions — which can mean an unplanned, uncontrolled Kubernetes version upgrade during a maintenance window if you miss the notification.

Infrastructure EOL requires coordination across teams. A Kubernetes version upgrade affects every workload running on the cluster. This is not a single-team upgrade — it is a platform event that requires coordinated validation across all tenant teams.

### Operating System EOL

The underlying OS loses vendor support. Kernel patches, glibc patches, and system library patches stop.

Examples: CentOS 7 (EOL June 2024), RHEL 7, Ubuntu 18.04.

Impact: kernel and glibc vulnerabilities go unpatched. A container workload on Ubuntu 18.04 base images carries these vulnerabilities into every image build. A VM or bare-metal system on CentOS 7 has no kernel security patches after June 2024.

OS EOL is frequently the last category that teams address because it feels like "the platform team's problem." For containers, it is explicitly the application team's problem — the base image is part of the application's dependency tree.

## 4. EOL for Middleware

Specific guidance for the middleware domain where the operational intuition runs deep.

### RabbitMQ

RabbitMQ follows a release stream model. RabbitMQ 3.10 and earlier are community-deprecated. The 3.12+ stream is the current supported branch.

Two structural changes affect RabbitMQ EOL governance:

**Classic queue mirroring is deprecated.** Classic mirrored queues — the high-availability mechanism used by most long-running RabbitMQ deployments — are deprecated in favor of quorum queues. Any deployment using classic mirrored queues is running a deprecated HA pattern, not just a deprecated version. The migration from classic mirrored to quorum queues requires application-layer changes (queue declaration parameters, consumer behavior for poison message handling) in addition to broker version upgrades.

**Broadcom acquisition of VMware.** Broadcom acquired VMware (which owned RabbitMQ commercial support) and has restructured commercial support offerings. Verify current support windows and commercial terms directly with Broadcom — do not rely on VMware-era documentation for current support commitments.

EOL risk: an organization running RabbitMQ 3.9 with classic mirrored queues carries both a version EOL finding and a deprecated HA pattern finding. Both must be addressed.

### Apache Kafka

Apache Kafka community releases have a support lifecycle of approximately 12 months after release. After that, fixes are backported only by the commercial distributions (Confluent Platform, Red Hat AMQ Streams, AWS MSK).

Know your distribution and its commercial support model:
- **Apache Kafka (community):** 12-month support window from release date
- **Confluent Platform:** extended commercial support windows, tied to Confluent subscription
- **AWS MSK:** follows Amazon's own version support schedule — check the MSK documentation for the specific Kafka version end-of-support dates

MSK managed upgrades require coordination with application teams on Kafka client library compatibility. Client library version must be compatible with the target broker version. This is not automatic.

### TIBCO BusinessWorks

TIBCO BusinessWorks 5.x is EOL. TIBCO BusinessWorks 6.x has extended support. The TIBCO acquisition by Vista Equity Partners and subsequent corporate restructuring has affected the consistency of support commitments — verify directly with the current vendor entity.

Practical position: treat all BW 5.x deployments as L3 Legacy (see ch01 lifecycle classification), schedule transformation immediately, and do not invest in BW 5.x feature development. BW 6.x deployments require a current support contract verification — do not assume continuity from pre-acquisition agreements.

Migration path from BW 5.x: the target is typically MuleSoft Anypoint Platform or Apache Camel (if the organization has internal integration engineering capability). BW 5.x to BW 6.x migration is not a safe intermediate step for most organizations — the cost is comparable to a full replatform, without the long-term platform currency benefit.

### MuleSoft Anypoint Platform

Mule Runtime 3.x is EOL. Mule Runtime 4.x is the current supported runtime. The 3.x to 4.x migration is a full rewrite — not a version upgrade. The programming model, XML configuration structure, and connector APIs changed significantly.

MuleSoft's support lifecycle is tied to the Anypoint Platform subscription. EOL on the runtime means no security patches on the integration engine itself, regardless of subscription status. Confirm your subscription terms and the specific runtime version coverage in your contract.

Organizations running Mule 3.x integrations should treat each integration as an L3 or L4 application (see ch01) and schedule transformation. The business case: Mule 4.x offers DataWeave 2.0, CloudHub 2.0 deployment, improved connector framework, and active security patching — none of which are available to a Mule 3.x deployment.

### Kong API Gateway

Kong follows semantic versioning. Each major version has a support window of approximately 2 years from release. Verify the specific version's end-of-life date at kong.io/versions — commercial Kong Enterprise support windows differ from open-source Kong.

## 5. The CVE Debt Problem

EOL software accumulates unpatched CVEs. This is the direct mechanism by which poor currency governance becomes a security incident.

### Measuring CVE Debt

Run dependency scanning against every application in the portfolio quarterly. Tools: OWASP Dependency-Check (open source), Snyk, GitHub Dependabot, AWS Inspector (for infrastructure), Trivy (for containers). For each application, aggregate:

- Count of Critical CVEs (CVSS 9.0–10.0)
- Count of High CVEs (CVSS 7.0–8.9)
- Count of CVEs where the fix requires upgrading an EOL component (not patchable in place)
- Count of CVEs that have known public exploits (CISA KEV catalog is the reference)

The last two categories are the most important for executive reporting. "We have 14 critical CVEs, 11 of which require EOL component upgrades that are not yet scheduled, and 3 of which have known public exploits in active use." That sentence drives budget approval.

### The CVE Debt Spiral

EOL component → critical CVE identified → no patch available for EOL version → upgrade required to get patch → upgrade blocked by other dependency compatibility → CVE remains open → audit finding generated → compliance exception required → management overhead compounds → upgrade gets deprioritized again.

Each iteration of this spiral is more expensive than the last. The time to escape the spiral increases as the dependency graph around the EOL component grows. Prevention costs approximately 10% of the remediation overhead — but prevention requires proactive EOL governance before the CVE is discovered.

### CVE Debt as Portfolio Risk

Aggregate CVE debt across the portfolio provides a risk score that translates directly to executive language: "We have 14 critical CVEs across 6 applications, 11 of which require EOL component upgrades not yet scheduled. The average cost of a breach exploiting a critical CVE in our industry is $4.2M. Our cyber insurance deductible is $500K. The current upgrade backlog is estimated at $800K. The risk-adjusted case for the upgrade investment is straightforward."

## 6. Technology Currency Scoring

Currency scoring provides a per-application input to the application health scorecard. Three sub-scores combine into a composite.

### Sub-Scores

**Runtime score (0–7):** Assess all language runtimes used by the application.
- 7 points: all runtimes within the active vendor support window, with active patch verification
- 4–6 points: runtimes within extended support (commercial agreement), or one runtime within 6 months of EOL with upgrade in progress
- 1–3 points: one or more runtimes in extended-only support with no upgrade plan
- 0 points: one or more runtimes EOL with no active support agreement

**Dependency score (0–7):** Assess libraries, frameworks, and third-party packages.
- 7 points: no EOL dependencies, no critical CVEs in any dependency
- 4–6 points: no EOL dependencies, high CVEs present with remediation scheduled
- 1–3 points: EOL dependencies present, no critical CVEs in EOL dependencies
- 0 points: EOL dependencies with critical CVEs, or critical CVEs with known public exploits

**Infrastructure score (0–6):** Assess Kubernetes version, base OS, managed service versions.
- 6 points: all infrastructure components within vendor support window
- 3–5 points: one component within 3 months of EOL with upgrade in progress
- 1–2 points: one component EOL with no active upgrade plan
- 0 points: multiple infrastructure components EOL

**Composite score (0–20):** Sum of the three sub-scores.

| Score Range | Currency Rating | Governance Action |
|------------|----------------|-------------------|
| 18–20 | Current | Maintain with quarterly scans |
| 14–17 | Watch | Document upgrade plan for any sub-score below 6 |
| 8–13 | At Risk | Upgrade plan required, owner assigned, timeline approved |
| 0–7 | Critical | Immediate VP notification, EOL exception or emergency upgrade required |

## 8. EOL Response Playbook

Four phases, executed in sequence.

### Phase 1: Discovery

Quarterly dependency scan of all applications in the portfolio. This is not optional or best-effort — it is a quarterly control. Every application, every quarter.

Output: EOL risk register entry for every finding. Each entry includes: application name, tier, component name, current version, EOL date, days until EOL, CVE count by severity, assigned risk level (Critical/High/Medium/Low per the timeline in Section 2).

For container-based applications: scan the container image, not just the application code. Use Trivy or Grype to scan image layers. Base OS and system library EOL findings appear at the image layer.

### Phase 2: Risk Assessment

For every Critical and High finding, assess three dimensions:

**Upgrade complexity.** Can this be done as an in-place upgrade (bump version in build file, run tests, deploy)? Or does it require architectural changes (Mule 3.x to 4.x rewrite, Spring Boot 2.x to 3.x with Java upgrade dependency)? Simple upgrades are 1–4 weeks. Architectural upgrades are 2–6 months minimum.

**Blast radius.** Is this a single-service upgrade, or does it require coordinated changes across multiple services? A Kubernetes version upgrade affects every workload on the cluster. A shared library EOL affects every service that imports it. High blast radius upgrades need portfolio-level coordination, not team-level execution.

**Regulatory exposure.** Does the EOL component sit in a regulated data path? PCI-scoped services with EOL dependencies are an automatic audit finding. HIPAA-covered systems with unpatched CVEs are a compliance violation. Regulatory exposure accelerates the urgency tier.

### Phase 3: Upgrade Planning

For each Critical and High finding with an assigned owner, produce an upgrade plan with:
- Target version and rationale (why this version, not just "the latest")
- Upgrade approach: in-place version bump, phased migration, full rewrite
- Test strategy: what must pass before production deployment
- Rollout plan: which environments, in what sequence, with what validation gate between each
- Resource requirement: engineering time estimate, infrastructure cost delta
- Timeline: milestone dates aligned to the T-12/T-6/T-3/T-0 framework

Plans are reviewed and approved by Engineering Director. Critical findings require VP awareness of the plan.

### Phase 4: Execution and Validation

Upgrade executed per plan. Validation is mandatory before the finding is closed:
- Post-upgrade dependency scan confirms EOL finding is resolved (version now within support window)
- CVE scan confirms no new Critical or High CVEs introduced by the upgrade
- SLO compliance verified for 7 days post-upgrade in production
- EOL risk register entry updated to Resolved status

A finding is not closed until all four validation steps are complete. "The upgrade was deployed" is not the same as "the finding is resolved."

## 9. Vendor Relationship Management for EOL

The best EOL governance is early warning — knowing about upcoming EOL dates 12–18 months before public announcement.

**Vendor contacts.** Maintain active relationships with technical account managers and support contacts for TIBCO, MuleSoft, RabbitMQ (Broadcom), Confluent, and any other commercial middleware vendors in your portfolio. Ask them directly: "What versions are coming up for EOL in the next 18 months? What is the recommended migration path?" Vendors with an active account relationship will tell you before the public announcement.

**Security bulletins and release notes.** Subscribe to vendor security advisories and release notes for every commercial software product in the portfolio. EOL notices appear in release notes long before they appear in sales materials. Assign ownership: someone on the team is responsible for reading the MuleSoft release notes every quarter.

**Customer advisory boards.** For T-Mobile-scale deployments, participation in vendor customer councils and advisory boards provides roadmap access. Roadmap access = EOL advance notice.

**Commercial extended support.** When the upgrade timeline is known but > 12 months out, negotiate extended support in the vendor contract. Extended support is typically 20–30% of standard license cost per year, added on top of the base subscription. The calculation: extended support cost for 18 months vs. the cost of an emergency upgrade driven by a breach or compliance finding. Extended support is almost always the cheaper buy, and it eliminates the residual security risk during the gap period.

Do not buy extended support as a substitute for planning the upgrade. Buy it to cover the gap while the upgrade is in progress.
