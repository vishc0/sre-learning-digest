# JVM Analysis Playbook

## When to Use This
Use when: GC pauses are causing p99 spikes, heap utilization is trending toward -Xmx, OOM errors are occurring, or a soak test reveals memory growth. Also use proactively during load test monitoring.

## Step 1: GC Log Collection and Analysis

**Enable GC logging** (add to JVM startup flags):
```bash
# JDK 11+ (Unified GC Logging)
-Xlog:gc*:file=/var/log/jvm/gc-%t.log:time,uptime,level,tags:filecount=5,filesize=20m

# JDK 8 (legacy)
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -XX:+PrintGCTimeStamps \
-Xloggc:/var/log/jvm/gc.log -XX:+UseGCLogFileRotation \
-XX:NumberOfGCLogFiles=5 -XX:GCLogFileSize=20M
```

In Kubernetes: mount a log volume and stream GC logs to a sidecar or use a ConfigMap to set JVM flags via `JAVA_OPTS` environment variable.

**Analyze GC log with GCEasy** (web tool, upload log file):
- Key metrics to read: Throughput (% of time NOT in GC — should be > 95%), GC Pause Time (max and p99), GC Cause (what triggered each GC), Heap Reclamation Pattern

**Interpret GCEasy output**:

| Signal | Meaning | Action |
|--------|---------|--------|
| Young GC frequency > once/second | Eden too small relative to allocation rate | Increase -Xmn or -XX:NewRatio |
| Old Gen utilization > 80% sustained | Memory leak or heap too small | Heap dump analysis |
| Full GC frequency > 1/hour | Old Gen pressure | Urgent: tune or increase heap |
| GC Pause > 500ms | G1GC not meeting pause target | Reduce -XX:MaxGCPauseMillis, reduce heap, or switch to ZGC |
| GC Throughput < 90% | GC consuming > 10% of CPU | Performance crisis — JVM is fighting GC |

**Manual GC log pattern reading**:
```
[GC (Allocation Failure) 2024-01-15T10:23:45.123+0000]
[Eden: 512M(512M)->0B(512M) Survivors: 64M->64M Heap: 1024M(2048M)->768M(2048M)]
[Times: user=0.45 sys=0.01 real=0.12 secs]
```
- `Allocation Failure` = Eden full, normal Young GC trigger
- `Eden: 512M->0B` = Eden fully reclaimed (good)
- `Heap: 1024M->768M` = 256MB freed
- `real=0.12 secs` = 120ms pause — acceptable for G1GC

**Dangerous pattern — Old Gen not shrinking**:
```
[Full GC (Ergonomics)]
[Heap: 1900M(2048M)->1850M(2048M)]
```
Old Gen went from 1900M to 1850M — reclaimed only 50MB in a Full GC. This is a memory leak. The objects are being retained by application references. Heap dump analysis is mandatory.

## Step 2: Heap Dump Capture and Analysis

**Capture heap dump** (two methods):

Method 1: Proactive capture while running
```bash
kubectl exec <pod> -- jcmd <pid> GC.heap_dump /tmp/heap.hprof
kubectl cp <namespace>/<pod>:/tmp/heap.hprof ./heap-$(date +%Y%m%d-%H%M%S).hprof
```

Method 2: Automatic on OOM (must be set in JVM flags before the OOM):
```
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/jvm/
```

**Analyze with Eclipse MAT (Memory Analyzer Tool)**:
1. Open Eclipse MAT, File > Open Heap Dump > select .hprof
2. Run "Leak Suspects Report" — MAT automatically identifies most likely leak candidates
3. Check "Dominator Tree" — shows which objects hold the most heap (by retained size)
4. Check "Histogram" — shows instance counts by class — look for unexpectedly large counts

**Reading MAT findings**:
```
Problem Suspect 1:
  2,847,291 instances of com.example.CustomerSession,
  occupied 847MB (42% of heap)
  Shortest path to GC root: via SessionManager.activeSessions (HashMap)
```

Finding: SessionManager is holding 2.8 million CustomerSession objects. The activeSessions HashMap is growing without expiry. Root cause: sessions are not being removed on logout or timeout.

Fix: add session cleanup on timeout in SessionManager. Verify TTL policy in Redis if sessions externalized.

## Step 3: Thread Dump Collection and Deadlock Detection

**Capture 3 thread dumps, 10 seconds apart**:
```bash
for i in 1 2 3; do
  kubectl exec <pod> -- jcmd <pid> Thread.print > thread-dump-${i}.txt
  echo "Captured dump $i"
  sleep 10
done
```

**Quick analysis commands**:
```bash
# Count threads by state across all dumps
grep "java.lang.Thread.State:" thread-dump-1.txt | sort | uniq -c

# Find BLOCKED threads
grep -B5 "BLOCKED" thread-dump-1.txt | grep '"' | head -20

# Check for deadlock detection
grep -A20 "Java-level deadlock" thread-dump-1.txt

# Find threads waiting on same lock (contention hotspot)
grep "waiting to lock" thread-dump-1.txt | sort | uniq -c | sort -rn | head -5
```

**Three-dump comparison**:
If the same threads appear BLOCKED across all 3 dumps — they are permanently blocked, not transiently busy. This confirms a deadlock or a long-running lock holder.

If threads appear RUNNABLE in dump 1, BLOCKED in dump 2, RUNNABLE in dump 3 — transient contention, not a deadlock. Monitor under load.

## Step 4: Tuning Hypothesis Formulation

After data collection, formulate a hypothesis before making any changes:

Hypothesis format:
```
Observation: [what the data shows — specific metric, specific value]
Hypothesis: [what is causing it — specific mechanism]
Proposed change: [specific JVM flag or code change]
Expected outcome: [measurable improvement — specific metric, expected value]
Validation method: [how to confirm the hypothesis was correct]
Rollback: [how to undo if hypothesis is wrong]
```

Example:
```
Observation: Old Gen utilization trending from 60% to 80% over 8 hours of soak test.
             Full GC every 45 minutes. MAT shows 1.2M HttpSession objects retained.

Hypothesis: Session cleanup on timeout is not firing because SessionDestroyedEvent
            listener is not wired to the session registry. Sessions accumulate
            in the in-memory store without expiry.

Proposed change: Fix SessionDestroyedEvent listener wiring in SecurityConfig.java.
                 Add session timeout log line to verify cleanup fires.

Expected outcome: Soak test heap stabilizes after 2-3 hours. Old Gen utilization
                  stays below 50% throughout 24-hour soak.

Validation: Re-run 24-hour soak test. Monitor heap with jstat -gcutil every 30 minutes.

Rollback: Revert SecurityConfig.java change. Issue does not worsen (existing bug).
```

Never tune JVM flags (Xmx, GC algorithm, GC threads) until memory leak hypothesis is ruled out. Increasing heap delays the OOM but does not fix the leak.
