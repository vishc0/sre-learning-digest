# Pre-Launch Performance Validation Playbook

## Who Runs This
PE Lead or Senior SRE with PE responsibility. Run this for every release touching a service with a performance SLO or touching a revenue-critical user journey.

## Gate Criteria

**Minimum required** to proceed to production:
- Load test PASS: p95 <= SLO target at expected peak TPS for 30 minutes
- Regression check PASS: p95 not > 10% worse than current baseline
- Error rate <= 0.1% during sustained load
- Saturation point >= 1.5x expected peak TPS
- Soak test PASS (24 hours at 70% capacity): no memory growth, no GC acceleration

**Enhanced gates** for high-risk releases (payment, auth, network provisioning):
- Spike test PASS: survives 3x expected peak for 5 minutes with graceful degradation
- Chaos + performance: survives one pod kill during peak load without p95 exceeding 2x SLO

## Test Sequence

Run tests in this order. Each test depends on the prior test passing.

**Step 1: Baseline measurement (30 minutes)**
- Environment: staging-perf [Environment Management — environment must be production-equivalent]
- Tool: k6 or Gatling
- Load: 10% of expected peak TPS, 10 minutes steady state
- Purpose: establish current-build baseline before comparative testing
- Capture: p50/p95/p99 latency, error rate, TPS achieved, JVM heap, GC frequency

```bash
k6 run --vus 10 --duration 10m \
  -e BASE_URL=https://staging-perf.internal \
  scripts/account-api-baseline.js \
  --out json=baseline-results.json
```

**Step 2: Load test (60 minutes)**
- Load: ramp from 10% to 100% of expected peak in 10 minutes, hold 50 minutes
- Monitor during test: p95 latency trend, JVM heap, GC log, thread pool queue depth, connection pool
- Fail immediately if: error rate exceeds 0.5% before reaching target TPS, or p95 exceeds 2x SLO during ramp

**Step 3: Stress test (to breaking point)**
- Load: continue ramping past expected peak in 10% increments until failure
- Record: saturation TPS, failure mode (timeout, OOM, rejection), recovery behavior
- Must achieve: saturation point >= 1.5x expected peak

**Step 4: Soak test (24 hours)**
- Load: 70% of validated capacity (from load test result), constant
- Monitor hourly: heap utilization trend, GC frequency, thread count, connection pool idle count
- Fail if: heap grows > 15% over baseline after first GC cycle, GC frequency doubles from hour 1 to hour 12

**Step 5: Spike test (30 minutes)**
- Pattern: 30 min = (5 min normal) + (5 min 3x spike) + (20 min normal)
- Observe: error rate during spike, HPA response time, recovery time to baseline p95
- Pass: error rate < 5% during spike, p95 returns to baseline within 5 minutes of spike end

## Environment Requirements (Pre-Test Checklist)

- [ ] Environment is production-equivalent: same pod count, same JVM flags, same DB instance class
- [ ] Test data: realistic dataset loaded (not empty DB), user accounts parameterized
- [ ] Monitoring active: APM connected, GC logging enabled, thread pool metrics exposed
- [ ] Database warmed: run 5-minute warmup load before measurement starts (cold cache distorts results)
- [ ] Network isolation: test environment not shared with other load tests during this run
- [ ] Baseline stored: previous build's baseline results available for regression comparison

## Escalation Path if Gate Fails

| Failure | Immediate Action | Decision Owner |
|---------|-----------------|----------------|
| p95 > SLO by < 20% | Document, identify cause, estimate fix time | PE Lead |
| p95 > SLO by 20-50% | Block release, assign optimization sprint | SRE Manager |
| p95 > SLO by > 50% | Block release, escalate to VP for timeline decision | VP Operations |
| Saturation < 1.5x peak | Block release, identify capacity gap, funding required | VP Operations + Finance |
| Soak test fails (memory leak) | Block release regardless of load test result | PE Lead — no exception |
