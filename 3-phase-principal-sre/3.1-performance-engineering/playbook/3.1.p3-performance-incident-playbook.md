# Performance Incident Playbook

## Scope
A performance incident is declared when: p95 latency exceeds SLO by >20% for >5 minutes, TPS drops unexpectedly by >15% with no intentional change, error rate exceeds 1%, or an automated alert fires on thread pool saturation, connection pool exhaustion, or GC pressure.

## Phase 1: Triage (0–5 minutes)

The three diagnostic questions that determine investigation path:

**Q1: Is it latency degradation, throughput drop, or both?**
- Latency degradation only: system is processing requests but slowly — look for bottleneck (GC, lock, slow dependency)
- Throughput drop only: requests not reaching the system — look for upstream (load balancer, network, DNS) or service crash
- Both: system is saturated — look for thread pool exhaustion, connection pool saturation, or overload

**Q2: Is it one service or many?**
- One service: localized problem — investigate that service's JVM, thread pool, dependencies
- Many services: shared dependency or infrastructure problem — DB, message broker, shared cache, network

**Q3: Did anything change in the last 2 hours?**
- Check deployment log: `kubectl rollout history deployment/<service>`
- Check config changes: GitOps history
- Check external dependencies: did upstream API degrade? Check vendor status pages.
- Check traffic: is load abnormally high? (spike test scenario vs. actual degradation)

**Immediate data collection commands** (run simultaneously):

```bash
# 1. Check pod status and recent restarts
kubectl get pods -n <namespace> -l app=<service> --sort-by='.status.containerStatuses[0].restartCount'

# 2. Check HPA status (is autoscaling firing?)
kubectl get hpa -n <namespace>

# 3. Check recent events
kubectl get events -n <namespace> --sort-by='.lastTimestamp' | tail -20

# 4. Check resource utilization (requires metrics-server)
kubectl top pods -n <namespace> -l app=<service>

# 5. Check current error rate (Splunk query)
index=<app_index> service="<service>" level=ERROR | timechart span=1m count AS errors

# 6. Check latency trend (APM or Splunk)
index=<app_index> service="<service>" | timechart span=1m p95(response_time_ms) AS p95_latency
```

## Phase 2: Data Collection (5–15 minutes)

Based on triage findings, collect the appropriate diagnostic artifacts:

### If GC pressure suspected (long p99 spikes, memory metrics high):

```bash
# Get JVM PID in container
kubectl exec <pod> -- jcmd

# Enable GC logging if not already enabled (for JVM 11+)
# Note: in production, GC logs should always be enabled. If not: add JVM flags at restart.

# Check current heap usage
kubectl exec <pod> -- jcmd <pid> VM.native_memory

# Check GC activity (quick summary)
kubectl exec <pod> -- jstat -gcutil <pid> 1000 10
# Columns: S0% S1% E% O% M% CCS% YGC YGCT FGC FGCT GCT
# E% = Eden utilization, O% = Old gen utilization, FGC = Full GC count
# Danger: O% > 80% sustained, or FGC count increasing

# Capture heap dump if OOM imminent (O% > 85%)
kubectl exec <pod> -- jcmd <pid> GC.heap_dump /tmp/heap-$(date +%Y%m%d-%H%M%S).hprof
kubectl cp <pod>:/tmp/heap-<timestamp>.hprof ./heap-dump-<pod>.hprof
```

### If thread contention suspected (high CPU, many BLOCKED threads):

```bash
# Capture thread dump (take 3 dumps 10 seconds apart for pattern analysis)
kubectl exec <pod> -- jcmd <pid> Thread.print > thread-dump-1.txt
sleep 10
kubectl exec <pod> -- jcmd <pid> Thread.print > thread-dump-2.txt
sleep 10
kubectl exec <pod> -- jcmd <pid> Thread.print > thread-dump-3.txt

# Alternative: kill -3 (sends to stdout, capture from pod logs)
kubectl exec <pod> -- kill -3 <pid>
kubectl logs <pod> --tail=500 > thread-dump-stdout.txt

# Quick blocked thread count
grep -c "BLOCKED" thread-dump-1.txt

# Find the most-blocked lock
grep -A 3 "BLOCKED" thread-dump-1.txt | grep "waiting to lock"
```

### If connection pool exhaustion suspected:

```bash
# HikariCP metrics (if exposed via Micrometer/Prometheus)
curl http://<pod-ip>:8080/actuator/metrics/hikaricp.connections.pending
curl http://<pod-ip>:8080/actuator/metrics/hikaricp.connections.active
curl http://<pod-ip>:8080/actuator/metrics/hikaricp.connections.max

# Database connection count (run on DB or via DBA)
# PostgreSQL:
SELECT count(*), state, application_name FROM pg_stat_activity GROUP BY state, application_name;

# Check for long-running queries (> 5 seconds)
SELECT pid, now() - pg_stat_activity.query_start AS duration, query
FROM pg_stat_activity
WHERE (now() - pg_stat_activity.query_start) > interval '5 seconds';
```

### If Kafka consumer lag suspected:

```bash
# Check consumer group lag
kubectl exec -n kafka <kafka-pod> -- kafka-consumer-groups.sh \
  --bootstrap-server <broker>:9092 \
  --describe --group <consumer-group>

# Columns: TOPIC, PARTITION, CURRENT-OFFSET, LOG-END-OFFSET, LAG, CONSUMER-ID
# LAG > 0 and growing = consumers not keeping up with producers
```

## Phase 3: Diagnosis

Decision tree format:

**GC pressure — GC log analysis**:
- Young GC every < 5 seconds — Eden too small (`-XX:NewRatio` or `-Xmn` needs adjustment)
- Full GC frequency > 1 per minute — Old gen pressure — memory leak or heap undersized
- GC pause > 500ms — G1GC tuning needed (`-XX:MaxGCPauseMillis=200`)
- After Full GC, heap doesn't drop — memory leak confirmed — heap dump analysis required

**Thread contention — thread dump analysis**:
- Many BLOCKED threads on same lock — lock contention hotspot — identify the lock owner, assess lock scope
- Deadlock detected — kill -9 is the only immediate fix; root cause fix in code
- All threads RUNNABLE but CPU low — threads blocking on I/O without async — I/O-bound bottleneck

**Connection pool exhaustion — immediate mitigation**:
- Pending connections > 0 — pool is fully utilized — reduce query time or increase pool temporarily
- Long-running DB queries — kill the queries, fix the query, add index
- Connection leak — service restart as immediate mitigation; fix code defect

## Phase 4: Remediation Options

| Symptom | Immediate Mitigation | Root Cause Fix |
|---------|---------------------|----------------|
| GC pressure / heap full | Restart pod (clears heap), reduce traffic | Heap dump analysis — fix memory leak |
| Thread pool exhausted | Increase thread pool size (config change + restart) or scale pods | Profile slow operations, add async, or fix downstream dependency |
| Connection pool exhausted | Kill long-running DB queries, temporarily increase pool | Fix slow queries, add DB index, add read replica |
| Kafka consumer lag | Scale consumer pods (up to partition count), increase `max.poll.records` | Profile consumer processing time, optimize business logic |
| External dependency timeout | Enable circuit breaker, add timeout, use cached/default response | Dependency team remediation, add fallback |
| Memory leak | Pod restart on rotation (keeps service up), schedule maintenance window | Heap dump analysis — fix object retention |

## Phase 5: Verification and Close

Post-remediation checks:
```bash
# Confirm p95 returning to baseline
kubectl exec <pod> -- curl http://localhost:8080/actuator/metrics/http.server.requests

# Confirm GC stable
kubectl exec <pod> -- jstat -gcutil <pid> 1000 30

# Confirm thread pool clearing
curl http://<pod-ip>:8080/actuator/metrics/executor.queue.remaining

# Confirm Kafka lag draining
kubectl exec -n kafka <kafka-pod> -- kafka-consumer-groups.sh \
  --bootstrap-server <broker>:9092 \
  --describe --group <consumer-group>
```

Declare incident closed when: p95 within SLO for 15 consecutive minutes, error rate < 0.1% for 10 consecutive minutes, no active GC pressure or thread pool saturation.
