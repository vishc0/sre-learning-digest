# Load Test Execution Playbook

## Pre-Test Checklist

- [ ] Test environment provisioned and confirmed production-equivalent [Environment Management]
- [ ] Test data loaded: parameterized user list, test accounts created, data seeded
- [ ] Load test script reviewed: correct user journey, correct think times, assertions enabled
- [ ] Monitoring active: APM dashboards open, GC logging enabled, thread pool metrics exposed
- [ ] Baseline from previous build available for regression comparison
- [ ] No concurrent load tests in same environment (will corrupt results)
- [ ] Database warmed: run 5-minute warmup at 20% target TPS before measurement
- [ ] Alert noise suppressed: notify on-call that test is running (avoid false incident pages)

## Step 1: Environment Baseline Measurement (10 minutes)

Before injecting load, capture idle-state metrics:
```bash
# JVM heap baseline (idle)
kubectl exec <pod> -- jstat -gcutil <pid> 5000 5
# Record: S0%, S1%, E% (Eden), O% (Old Gen), M% (Metaspace)

# Thread count baseline (idle)
kubectl exec <pod> -- jcmd <pid> Thread.print | grep "java.lang.Thread.State" | sort | uniq -c

# Connection pool baseline (idle)
curl -s http://<pod>:8080/actuator/metrics/hikaricp.connections.idle
```

## Step 2: Warmup Run (5–10 minutes)

Run at 20% of target TPS for 5–10 minutes:
- Purpose: warm JVM (JIT compilation), warm caches, warm connection pools
- Results from warmup are DISCARDED — they are not representative
- Confirm after warmup: heap stabilized, GC frequency normal, no errors

```bash
k6 run --vus 5 --duration 5m \
  -e BASE_URL=https://staging-perf.internal \
  scripts/journey-warmup.js
echo "Warmup complete — starting measurement run"
```

## Step 3: Ramp-Up (10–15 minutes)

Increase load from 20% to 100% of target TPS in steps:
```bash
k6 run \
  --stage 2m:10,5m:50,3m:100,50m:100 \
  -e BASE_URL=https://staging-perf.internal \
  --out json=load-test-results.json \
  scripts/account-api-load.js
# Stage breakdown: 2min ramp to 10 VU, 5min ramp to 50 VU, 3min ramp to 100 VU, hold 50min
```

During ramp-up, monitor:
- Is TPS tracking injected load? (check in k6 dashboard or Grafana)
- Is p95 stable or climbing? (should remain flat during ramp if system has headroom)
- Any errors? Error rate should be 0% during ramp to expected peak

Red flags during ramp:
- p95 climbs steeply before reaching target TPS — already hitting saturation zone
- Errors appear before target TPS — insufficient capacity — stop test, investigate

## Step 4: Steady-State Measurement (30–50 minutes)

Hold at target TPS for 30+ minutes. This is the measurement window.

Monitor every 5 minutes during this phase:
```bash
# Thread pool queue depth
curl -s http://<pod>:8080/actuator/metrics/executor.queue.remaining | jq '.measurements[0].value'

# JVM heap
kubectl exec <pod> -- jstat -gcutil <pid> 60000 1

# GC frequency
kubectl exec <pod> -- jstat -gccause <pid> 60000 1
```

Stable system signatures during steady-state:
- TPS achieved = TPS injected (within 2%)
- p95 flat or slowly rising < 10% from minute 10 to minute 40
- Heap oscillates (sawtooth pattern — fill/GC/fill/GC) but does not trend upward
- GC frequency steady, not increasing
- Thread pool queue depth at 0 or near 0

## Step 5: Result Analysis

After test completes, generate and analyze results:

```bash
# k6 results summary (if using --out json)
k6 convert --input results.json --output summary.json

# Key metrics to extract:
# http_req_duration{p(50), p(95), p(99)}
# http_req_failed (error rate)
# http_reqs (total TPS)
# iterations (total user journeys)
```

Regression comparison:
```bash
# Compare p95 against stored baseline
CURRENT_P95=$(cat load-test-results.json | jq '.metrics.http_req_duration.p95')
BASELINE_P95=$(cat baseline-results.json | jq '.metrics.http_req_duration.p95')
CHANGE_PCT=$(echo "scale=2; ($CURRENT_P95 - $BASELINE_P95) / $BASELINE_P95 * 100" | bc)
echo "p95 change vs. baseline: $CHANGE_PCT%"
# If > 10%: REGRESSION DETECTED
```

## Step 6: Report Generation

Fill in the Performance Test Report Template (templates/performance-test-report-template.md).

Required outputs:
1. Executive summary (1 page): PASS/FAIL, key metrics vs. SLO, regression status, recommendation
2. Full technical report: all percentiles, time series, resource utilization, bottlenecks
3. Baseline update (if PASS): update stored baseline with this build's results
