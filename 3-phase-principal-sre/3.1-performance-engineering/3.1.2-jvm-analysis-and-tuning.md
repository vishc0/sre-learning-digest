## JVM Memory Model

The JVM divides memory into distinct regions. Each region has its own management model, failure mode, and monitoring requirement. Confusing them under a single "memory usage" metric is one of the most common causes of missed OOM investigations.

### Heap (GC-Managed)

**Young Generation**

| Space | Role | GC Event |
|-------|------|----------|
| Eden | All new object allocations land here | Young GC on fill |
| Survivor S0/S1 | Survivors of one Eden GC are promoted here | Objects cycle between S0↔S1 each Young GC |
| Promotion | After N GC cycles (default: `-XX:MaxTenuringThreshold=15`), object moves to Old Gen | Promotion on threshold |

Most objects never leave Eden. A short-lived HTTP request object — byte arrays, response buffers, Jackson deserialization objects — allocates in Eden and dies there. This is the "generational hypothesis": most objects are temporary.

**Old Generation (Tenured)**

Long-lived objects: connection pools, caches, Spring singletons, static data, thread-local variables. This is where GC pauses become dangerous. Old Gen fills slowly over hours or days; when it does, the GC work to clean it is orders of magnitude more expensive than a Young GC.

**PermGen (JDK 7 and earlier only)**: fixed-size region for class metadata. Replaced entirely by Metaspace in JDK 8. If you're still seeing `java.lang.OutOfMemoryError: PermGen space`, the service is running JDK 7 or earlier — that is an immediate upgrade incident, not a tuning problem.

### Non-Heap (Not GC-Managed)

**Metaspace**

Class metadata, method bytecode, annotations. Grows as classes are loaded. Unbounded by default — the JVM will consume all available native memory before OOMing the process.

Set a ceiling: `-XX:MaxMetaspaceSize=512m`. Without it, a class-loader leak (common with hot-deploy frameworks, OSGI, or Groovy script execution) will consume the entire host's memory silently.

Alert signal: Metaspace size growing steadily across restarts. Root cause is almost always a class-loader leak, not a class metadata explosion.

**Code Cache**

JIT-compiled native code. Default ceiling: 240MB for server JVM. When the Code Cache fills, the JVM falls back to interpreted mode — the bytecode interpreter rather than compiled native code. Performance degradation is severe: 10×–100× slower for hot code paths.

Flag: `-XX:ReservedCodeCacheSize=512m` — increase if Code Cache is consistently hitting the ceiling.
Detection: `jstat -compiler <pid>` reports compilation activity. JVM logs will emit `CodeCache is full` warnings.

**Direct Memory (Off-Heap)**

`ByteBuffer.allocateDirect()` allocates outside the heap. Not subject to GC. Not visible in heap monitoring. This is where Netty, Kafka client, NIO-based frameworks, and Cassandra drivers maintain their I/O buffers.

A container can be OOM-killed by the Linux kernel with a perfectly healthy-looking JVM heap because the total process memory (heap + Metaspace + Code Cache + stacks + off-heap) exceeds the cgroup memory limit.

### Stack (Per-Thread)

Each Java thread has its own stack: default 512KB–1MB. For a service running 200 concurrent threads:
- 200 threads × 1MB = 200MB non-heap, non-GC-managed memory
- Not visible in heap monitoring
- StackOverflowError: stack exhausted (infinite recursion, very deep call chains)

### What Lives Where — Reference Table

| Object Type | Where Allocated | GC Managed | Monitoring Signal |
|-------------|----------------|------------|-------------------|
| `new String()`, `new ArrayList()` | Eden (Young Gen) | Yes | Eden utilization rate |
| Large objects (> G1 region size / 2) | Old Gen directly (humongous) | Yes | Humongous allocation rate |
| Class metadata | Metaspace | No (class unload only) | `jstat M%` column |
| JIT compiled code | Code Cache | No | `jstat -compiler` |
| `ByteBuffer.allocateDirect()` | Off-heap | No | `container_memory_working_set_bytes` |
| Thread stack frames | Per-thread stack | No | Thread count × stack size |
| `static` fields | Old Gen (via Class object) | Only when class unloaded | Rarely collected in practice |
| Spring singleton beans | Old Gen | Only on context shutdown | Live for application lifetime |

### Engineering Deep-Dive: Why Off-Heap Matters for Kafka

In a T-Mobile messaging service handling 25M messages/day, the Kafka Java client maintains several off-heap buffer pools that are invisible to standard JVM heap monitoring.

**Producer side** (`buffer.memory`, default 32MB):
```
kafka-producer → ByteBuffer.allocateDirect(32MB) → off-heap
```
The producer's record accumulator uses this off-heap buffer pool. Increasing `buffer.memory` to handle burst traffic directly increases process memory, not heap.

**Consumer side** — per partition, per fetch:
```
max.partition.fetch.bytes = 1MB (default)
24 partitions in the topic
→ consumer holds up to 24MB off-heap in fetch buffers
```

A service with `-Xmx4g` on a 6GB pod has:
- 4GB heap
- 300MB Metaspace
- 240MB Code Cache
- 200MB thread stacks (200 threads)
- 24MB Kafka consumer off-heap
- 32MB Kafka producer off-heap
- OS page cache and overhead: ~500MB
- **Total process: ~5.3GB** — safely within 6GB

But if someone increases the heap to `-Xmx5.5g` to "fix" an OOM without accounting for off-heap: the process now needs 6.8GB on a 6GB pod. The Linux OOM killer terminates the container. The heap dump shows 5.4GB used — operator concludes the heap was at fault and increases pod size rather than finding the root cause.

**Correct monitoring**:
```bash
# Container-level memory — includes ALL process memory
container_memory_working_set_bytes

# NOT this (includes kernel page cache, misleading):
container_memory_usage_bytes

# Direct memory usage — JVM MX bean
java.lang:type=MemoryPool,name=Direct
→ attribute: MemoryUsage.used
```

Cross-reference: [Capacity Management] — memory budget modeling must account for all four memory regions (heap, Metaspace, Code Cache, off-heap) when sizing pods. A pod memory limit derived from heap size alone will produce intermittent OOM-kill events that look like infrastructure instability rather than sizing errors.

## Garbage Collection: Stop-the-World vs. Concurrent Collectors

**Stop-the-World (STW)**: all application threads are suspended for the duration of GC. Every user request in-flight at that moment pauses. The JVM's "world" stops. This is the direct source of GC-induced latency spikes visible in p99/p999 percentiles.

**Concurrent**: GC work happens while application threads continue. Not entirely pause-free — concurrent collectors still have short STW phases (initial mark, remark) — but the bulk of the work (marking live objects, evacuation) happens without stopping the application.

### Collector Comparison

| Collector | Algorithm | STW Pause | Throughput | Best For | Avoid When |
|-----------|----------|-----------|------------|---------|-----------|
| Serial (`-XX:+UseSerialGC`) | Single-threaded mark-compact | Full GC: seconds | Very low | CLI tools only | Any server workload |
| Parallel (`-XX:+UseParallelGC`) | Multi-threaded STW | Young: 10–100ms; Full: 100ms–seconds | Highest | Batch jobs, throughput > latency | Latency-sensitive services |
| G1GC (`-XX:+UseG1GC`) | Region-based, concurrent mark, incremental evacuation | Target configurable (default 200ms) | Good | General server applications, 2GB–32GB heap | Sub-10ms p99 SLO requirements |
| ZGC (`-XX:+UseZGC`) | Concurrent, colored pointers, region-based | Always < 1ms (sub-millisecond) | 10–15% lower than G1 | Latency-critical, large heaps (> 16GB) | Throughput-bound batch |
| Shenandoah (`-XX:+UseShenandoahGC`) | Concurrent evacuation (unlike G1) | Always < 10ms | Similar to ZGC | Same as ZGC | Throughput-bound batch |

G1GC is the default from JDK 9 onward and the right choice for most T-Mobile microservices. ZGC is the right choice when your p99 SLO is below 50ms and GC pauses are measurable in your latency distribution.

### G1GC Internals (Required Knowledge)

G1GC divides the heap into equal-sized regions (1–32MB, auto-calculated based on heap size). Young Gen and Old Gen are sets of regions — not contiguous memory blocks. This is why G1GC can collect incrementally.

**GC event types**:

| Event | What Happens | Typical Duration | Log Marker |
|-------|-------------|-----------------|------------|
| Young GC | Eden + Survivor regions collected | 20–150ms (scales with heap size) | `Pause Young (Normal)` |
| Mixed GC | Young + selected Old Gen regions | 100–250ms | `Pause Young (Mixed)` |
| Concurrent Mark Cycle | Background marking of live objects | Concurrent — no STW (mostly) | `Concurrent Mark` |
| Full GC | Entire heap, single-threaded | Seconds | `Pause Full` |

**Humongous allocations**: objects larger than 50% of a G1 region size bypass Young Gen entirely and allocate directly in Old Gen. High humongous allocation rate causes Old Gen pressure without any traditional promotion path. Common sources: large JSON payloads deserialized into a single object, large byte arrays, oversized Kafka messages.

Detection:
```bash
-Xlog:gc*:stdout:time,level,tags
# Look for: "Humongous" in GC log output
# Or: jstat -gcnew shows HU (humongous) region count
```

## GC Analysis: Reading GC Logs

Enable GC logging on all production JVMs (JDK 11+):
```bash
-Xlog:gc*:file=/var/log/jvm/gc-%t.log:time,uptime,level,tags:filecount=5,filesize=20m
```

This produces up to 5 × 20MB = 100MB of rolling GC logs. Essential for post-incident analysis.

### Log Pattern Reference

**Healthy Young GC — normal operation:**
```
[2.456s][info][gc] GC(12) Pause Young (Normal) (G1 Evacuation Pause) 512M->384M(2048M) 45.234ms
```
- `Pause Young (Normal)`: expected Young GC, not triggered by emergency
- `512M->384M`: heap before → after (128MB freed)
- `(2048M)`: current heap size (not always -Xmx; heap may not be fully committed)
- `45.234ms`: STW pause — every request in-flight paused for 45ms

**Mixed GC — Old Gen being collected:**
```
[120.234s][info][gc] GC(89) Pause Young (Mixed) (G1 Evacuation Pause) 1280M->900M(2048M) 189.234ms
```
- `Pause Young (Mixed)`: G1GC reclaiming Old Gen regions — expected to be longer
- `189ms` approaching 200ms `MaxGCPauseMillis` target — acceptable but trending is the signal
- If Mixed GC pauses consistently exceed target: increase heap size or reduce Old Gen occupancy threshold (`-XX:InitiatingHeapOccupancyPercent`, default 45%)

**Danger — Full GC:**
```
[3600.123s][info][gc] GC(412) Pause Full (Ergonomics) 1950M->1850M(2048M) 4523.456ms
```
- `Pause Full`: entire heap STW — 4.5 second pause
- `1950M->1850M`: only 100MB freed — G1GC evacuated a nearly-full heap and barely made progress
- `(Ergonomics)`: G1GC triggered Full GC because it couldn't keep pace with allocation rate using concurrent collection
- **This is a memory leak signal** — 100MB freed from 2GB heap means 1850MB is still live
- **Immediate action**: heap dump capture while pod is still running

**Concurrent Mark Cycle — healthy:**
```
[45.123s][info][gc] GC(67) Concurrent Mark Cycle
[45.456s][info][gc] GC(67) Pause Remark 1024M->1024M(2048M) 12.345ms
[48.234s][info][gc] GC(67) Concurrent Mark Cycle 3111ms
```
- Mark cycle ran concurrently for ~3 seconds — no STW except the short Remark pause (12ms)
- This is the correct operating mode for G1GC

### jstat: In-Container Quick Analysis

```bash
# Get PID (assumes single Java process in container)
kubectl exec -n <namespace> <pod> -- pgrep -f java

# 5-second interval, 12 samples = 60 seconds of observation
kubectl exec -n <namespace> <pod> -- jstat -gcutil <pid> 5000 12
```

Output interpretation:
```
 S0%  S1%   E%    O%    M%   CCS%  YGC   YGCT   FGC  FGCT    GCT
   0  48.2  73.5  42.1  97.3  93.4  147  12.234    0  0.000  12.234
   0  48.2  89.1  42.1  97.3  93.4  147  12.234    0  0.000  12.234
   0  52.3   2.4  43.8  97.3  93.4  148  12.456    0  0.000  12.456
   0  52.3  18.7  43.8  97.3  93.4  148  12.456    0  0.000  12.456
```

| Column | Meaning | Healthy Pattern | Warning Pattern |
|--------|---------|----------------|----------------|
| `E%` | Eden utilization | Cycles 0→100→0 rapidly | Stays high (allocation outpacing GC) |
| `O%` | Old Gen utilization | Stable (flat) | Trending upward sample-to-sample |
| `M%` | Metaspace utilization | Stable after warmup | Growing slowly (class-loader leak) |
| `YGC` | Young GC count | Incrementing steadily | Incrementing too fast (allocation pressure) |
| `FGC` | Full GC count | 0 in steady state | Any nonzero value is an incident |
| `FGCT` | Full GC time | 0.000 | Nonzero — calculate pause duration = FGCT/FGC |

**GC throughput calculation**:
```
GC Throughput % = (1 - GCT / uptime_seconds) × 100
Target: > 95%  (less than 5% of time in GC)
Alert:  < 90%  (service is spending > 10% of time in GC)
```

**GCEasy** (gceasy.io): upload the GC log file for automated analysis. Produces: GC throughput %, max pause per event type, GC cause breakdown, heap size recommendation. Use this for post-incident GC analysis rather than manual parsing of large log files.

## Heap Sizing: -Xms, -Xmx, and the Off-Heap Risk

### Rule 1: Set -Xms = -Xmx

With different values, the JVM starts at Xms and expands incrementally to Xmx as heap demand grows. Each expansion triggers a GC and may cause a brief STW pause. Under load test conditions, this warmup behavior distorts baseline measurements — the first 2–5 minutes of results are not representative.

With identical values, the heap is committed at JVM startup. Memory is pre-allocated. GC behavior is deterministic from the first request.

The counter-argument ("smaller Xms reduces startup memory") does not apply to long-running EKS pods where the container is expected to reach full utilization under load.

### Rule 2: Container Memory Headroom

Memory budget for an 8GB pod running a Java service:

| Region | Amount | Flag / Source |
|--------|--------|---------------|
| JVM Heap (-Xmx) | 6,000MB (75%) | `-Xmx6g` |
| Metaspace | ~300MB | `-XX:MaxMetaspaceSize=512m` |
| Code Cache | ~240MB | `-XX:ReservedCodeCacheSize=512m` |
| Thread stacks | ~200MB (200 threads × 1MB) | `-Xss1m` × thread count |
| Kafka off-heap | ~56MB | `buffer.memory` + consumer fetch buffers |
| OS overhead + JVM internals | ~500MB | Not flag-controlled |
| **Total** | **~7.3GB** | Fits within 8GB |

If `-Xmx = 7.5g`: heap alone consumes 7.5GB. Adding 800MB of non-heap puts the process at 8.3GB. The Linux OOM killer terminates the container. The heap dump shows 7.4GB used. The incident report says "OOM" but the root cause is a sizing error, not a leak.

**Container support flag (mandatory for JDK 8u191+ / JDK 11+)**:
```
-XX:+UseContainerSupport
```
Without this flag on older JDK 8 builds, the JVM reads physical host memory rather than the cgroup limit. A 256GB host running 32 pods: each JVM thinks it has 256GB available. Chaos ensues.

### Rule 3: Process Memory ≠ Heap Memory

Monitor the right metric:
```bash
# Kubernetes — container-level memory including all process memory
# Use this to detect OOM risk
container_memory_working_set_bytes{container="my-service"}

# NOT this — includes kernel reclaimable page cache, overestimates risk
container_memory_usage_bytes{container="my-service"}

# Pod-level (sum of all containers including sidecars)
sum(container_memory_working_set_bytes{pod="my-service-abc123"})
```

Cross-reference: [Capacity Management] — capacity models that use heap utilization as a proxy for pod memory utilization will systematically underestimate memory requirements, leading to OOM-kill events that manifest as spurious pod restarts in incident reports.

## Memory Leak Identification

### Symptom Progression

1. `jstat O%` column trends upward over hours — Old Gen not being reclaimed by Mixed GC
2. Mixed GC frequency increases — G1GC is working harder to keep up
3. Full GC events appear — G1GC failed to keep pace with concurrent collection
4. After Full GC, `O%` drops only slightly — objects are retained by live references (not garbage)
5. GC thrashing — successive Full GCs, each freeing almost nothing, CPU at 100%
6. `java.lang.OutOfMemoryError: Java heap space` — pod restart, all in-flight requests lost

The time from Step 1 to Step 6 varies: hours to days for a slow leak, minutes for a large object cache with no eviction policy.

### Heap Dump Analysis Workflow

**Step 1: Capture heap dump before the OOM, while heap is still accessible**
```bash
# Get PID
PID=$(kubectl exec -n <ns> <pod> -- pgrep -f java)

# Trigger GC first (makes the dump more useful — dead objects collected)
kubectl exec -n <ns> <pod> -- jcmd $PID GC.run

# Capture dump
kubectl exec -n <ns> <pod> -- jcmd $PID GC.heap_dump /tmp/heap-$(date +%Y%m%d-%H%M%S).hprof

# Copy to local machine
kubectl cp <ns>/<pod>:/tmp/heap-20250616-143022.hprof ./heap-20250616-143022.hprof
```

For pods with `ExitOnOutOfMemoryError` + `HeapDumpOnOutOfMemoryError` enabled, the dump is written automatically before exit. Configure the path to a persistent volume or emit to an S3-backed init container.

**Step 2: Open in Eclipse MAT**
- File → Open Heap Dump → select .hprof
- Run: Reports → Leak Suspects Report
- This auto-identifies the largest retained memory segments and their retention paths

**Step 3: Read the Dominator Tree**
- Window → Heap Dump Details → Dominator Tree
- Sort by "Retained Heap" descending
- The top entries are objects that, if removed, would free the most memory
- Example finding:
  ```
  com.example.SessionRegistry    → retained: 847MB (41% of heap)
    └─ activeSessions (HashMap)  → retained: 847MB
         └─ 2,847,291 entries    → each: ~297 bytes
              └─ CustomerSession → CustomerPreferences + AuthToken + ...
  ```

**Step 4: Find the GC root retention path**
- Right-click on the suspect object → Path to GC Roots → Exclude weak/soft references
- This shows WHY the object cannot be collected — the chain of references from a GC root (thread, static field, JNI reference) to the leaking object
- Example path:
  ```
  Thread[http-nio-8080-exec-1] 
    → com.example.App (static field: sessionRegistry)
    → SessionRegistry.activeSessions (HashMap)
    → 2.8M CustomerSession entries
  ```
- `SessionRegistry` is a Spring singleton — always reachable from the application context. It holds 2.8M CustomerSession objects that were added on login but never removed.

**Root cause**: `activeSessions.put(sessionId, session)` called on login. No corresponding remove on logout or session timeout. Over 30 days, 2.8M sessions accumulated.

**Fix**: implement `HttpSessionDestroyedEvent` listener to call `activeSessions.remove(sessionId)`.

**Validation**: deploy fix, run 24-hour soak test, confirm `O%` is stable on `jstat` output.

### Heap Analysis Tool Comparison

| Tool | Max Practical Heap | Key Strength | Cost | Use Case |
|------|--------------------|-------------|------|----------|
| Eclipse MAT | ~32GB (64-bit mode) | Leak suspects report, dominator tree, offline analysis | Free | Production heap dump analysis |
| VisualVM | ~8GB practical | Built into JDK, live heap monitoring, heap histogram | Free | Development profiling |
| JProfiler | Any | Allocation call tree, CPU + heap, Kubernetes integration | Commercial | Deep profiling during development |

Production incident workflow: Eclipse MAT (offline, on a high-memory workstation — heap dumps are large). Development workflow: VisualVM or JProfiler.

## JVM Thread Model: Platform Threads vs. Virtual Threads

### Platform Threads (JDK 8–20 Default)

One Java thread = one OS thread. OS thread creation requires a kernel system call. Default stack: 512KB–1MB per thread. Thread blocking (waiting on DB, HTTP response, sleep) occupies the OS thread entirely — no other work can proceed on that OS thread.

Practical limits:
- 1,000–2,000 OS threads: typical safe maximum before scheduler overhead degrades throughput
- Memory: 1,000 threads × 1MB = 1GB of stack memory before any heap allocation
- Tomcat default `max-threads: 200`. HikariCP default pool size: 10. These limits exist because platform threads are expensive.

### Virtual Threads (JDK 21+ — Project Loom)

Java-managed lightweight threads. Not OS threads. The JVM maintains a pool of carrier threads (OS threads equal to CPU core count — typically 4–16). Virtual threads are mounted on carrier threads; when a virtual thread blocks on I/O, the JVM unmounts it from the carrier thread and mounts another virtual thread.

From the application code's perspective, this is transparent. Write the same blocking synchronous code. The JVM handles the interleaving.

```java
// Platform thread (JDK 8+): blocks OS thread during HTTP call
ExecutorService pool = Executors.newFixedThreadPool(200);

// Virtual thread (JDK 21+): unmounts during HTTP call, no OS thread blocked
ExecutorService pool = Executors.newVirtualThreadPerTaskExecutor();
```

**Performance comparison:**

| Scenario | Platform Threads | Virtual Threads |
|----------|-----------------|----------------|
| 1,000 concurrent requests, each waiting 100ms on DB | 1,000 OS threads blocked, ~1GB stack | 4–16 OS threads (carrier), ~10MB virtual thread overhead |
| Creating 10,000 concurrent tasks | Fails or severe degradation | Normal — 10K virtual threads is unremarkable |
| Memory for 10,000 concurrent tasks | ~10GB stacks | ~50–100MB |
| Throughput, CPU-bound workload | High | Same (CPU-bound — no I/O to interleave) |

**Compatibility caveats (test before migrating)**:
- `synchronized` blocks containing blocking I/O: the carrier thread is pinned (cannot unmount virtual thread). Replace with `ReentrantLock`.
- HikariCP 5.0.x: compatible with virtual threads. Earlier versions: not compatible.
- JMX: compatible. Some JDBC drivers (older versions): may have pinning issues.
- Spring Boot 3.2+: enables virtual threads for HTTP handlers automatically when running on JDK 21 (`spring.threads.virtual.enabled=true` or auto-detected).

**T-Mobile recommendation**: any Java service on EKS being upgraded to JDK 21 should enable virtual threads for HTTP handler thread pools as the first configuration change. Services with high I/O wait ratios (DB calls, downstream HTTP calls, Kafka consumer wait) will see the largest gains — thread pool saturation incidents should decrease significantly without any code changes.

Cross-reference: [Capacity Management] — virtual thread adoption changes the pod sizing model. Thread-pool-based capacity formulas (threads × memory per thread) no longer apply as the primary sizing driver for I/O-bound services. CPU and heap utilization become the binding constraints.

## JVM Flags for Production

All JVM-based services deployed to T-Mobile EKS must include this baseline flag set. Deviations require documented justification.

### Standard Production Baseline (G1GC)

```bash
JAVA_OPTS="\
  -Xms4g -Xmx4g \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=200 \
  -XX:MetaspaceSize=256m \
  -XX:MaxMetaspaceSize=512m \
  -XX:ReservedCodeCacheSize=512m \
  -XX:+UseContainerSupport \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/var/log/jvm/ \
  -XX:+ExitOnOutOfMemoryError \
  -Xlog:gc*:file=/var/log/jvm/gc-%t.log:time,uptime,level,tags:filecount=5,filesize=20m \
  -XX:+FlightRecorder \
  -Djava.security.egd=file:/dev/./urandom"
```

Flag rationale:

| Flag | Rationale |
|------|-----------|
| `-Xms = -Xmx` | Eliminates heap expansion GC events during warmup |
| `-XX:MaxGCPauseMillis=200` | G1GC targets 200ms — adjust down for latency-sensitive services |
| `-XX:MaxMetaspaceSize=512m` | Bounds class-loader leaks before they consume host memory |
| `-XX:+UseContainerSupport` | Reads cgroup limits, not host memory |
| `-XX:+HeapDumpOnOutOfMemoryError` | Captures state for post-mortem analysis |
| `-XX:+ExitOnOutOfMemoryError` | Fail fast — let Kubernetes restart the pod rather than running in degraded state |
| `-Xlog:gc*` | GC log for post-incident analysis |
| `-XX:+FlightRecorder` | Enables JFR profiling without restart (JDK 11+) |
| `-Djava.security.egd=...` | Prevents blocking on `/dev/random` entropy on containerized Linux |

### Latency-Critical Services (p99 SLO < 50ms): Switch to ZGC

```bash
JAVA_OPTS="\
  -Xms4g -Xmx4g \
  -XX:+UseZGC \
  -XX:SoftMaxHeapSize=3g \
  -XX:MaxMetaspaceSize=512m \
  -XX:ReservedCodeCacheSize=512m \
  -XX:+UseContainerSupport \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/var/log/jvm/ \
  -XX:+ExitOnOutOfMemoryError \
  -Xlog:gc*:file=/var/log/jvm/gc-%t.log:time,uptime,level,tags:filecount=5,filesize=20m"
```

ZGC-specific notes:
- Remove `-XX:MaxGCPauseMillis` — ZGC manages its own pause targets internally
- `-XX:SoftMaxHeapSize`: ZGC uses this as a soft ceiling, keeping headroom for concurrent collection
- Requires JDK 15+ for production-ready ZGC; JDK 21 for generational ZGC (`-XX:+ZGenerational`, default in JDK 23)
- CPU overhead: 10–15% higher than G1GC for the same workload — factor into pod CPU requests

### Large Heap Services (> 16GB): ZGC Mandatory

G1GC with > 16GB heaps produces increasingly long pause times for Full GC events. ZGC maintains sub-millisecond pauses at any heap size.

```bash
  -Xms24g -Xmx24g \
  -XX:+UseZGC \
  -XX:SoftMaxHeapSize=20g
```

## JVM Tuning Workflow

The correct sequence. Do not change flags without completing Step 1–3 first. Skipping directly to flag changes is cargo-culting — flags cannot fix code problems.

### Step 1: Establish Baseline

Before any tuning, capture:
- GC log: 30-minute window at representative load
- `jstat -gcutil <pid> 5000 360` during the 30-minute window (360 samples)
- Thread count at peak: `jstack <pid> | grep "java.lang.Thread.State" | wc -l`
- Heap utilization trend from APM (Datadog, New Relic, AppDynamics)
- p95/p99 latency from load balancer or service mesh metrics during the test

These are your baseline numbers. Every tuning change must beat them.

### Step 2: Load Test Protocol

Use a realistic traffic pattern — not synthetic uniform load. T-Mobile traffic has diurnal peaks (morning/evening), burst events (major campaign launches), and sustained background load. A 30-minute load test must cover:
- Ramp: 5 minutes to peak TPS
- Sustained: 20 minutes at peak TPS
- Cool-down: 5 minutes

Use k6, Gatling, or JMeter. Target: the p99 latency SLO as the pass/fail criterion, not average latency.

### Step 3: GC Log Analysis

Analyze the GC log from Step 2 against these thresholds:

| Signal | Healthy | Warning | Critical |
|--------|---------|---------|---------|
| GC throughput | > 98% | 95–98% | < 95% |
| Young GC pause | < 100ms | 100–200ms | > 200ms |
| Mixed GC pause | < 250ms | 250–400ms | > 400ms |
| Full GC count | 0 | — | Any |
| Old Gen post-GC | Stable | Slowly rising | Rising fast or > 80% |

If Full GC occurred: stop. Heap dump analysis before any flag changes.

### Step 4: Tuning Hypothesis (Written, Not Mental)

Before changing any flag:
```
Observation:  Old Gen (O%) trended from 42% to 61% over 30-minute test.
              Mixed GC frequency: 3 events, pauses 198ms/203ms/217ms.
              No Full GC. GC throughput: 97.2%.
Root cause:   Allocation rate is high enough that G1GC is struggling to
              reclaim Old Gen fast enough during the test window.
              Likely cause: humongous allocations bypassing Young Gen
              OR promotion rate too high due to insufficient Survivor space.
Proposed change: Increase heap from 4g to 6g to give G1GC more Old Gen
              headroom. Secondary: check for humongous allocations.
Expected outcome: Old Gen stabilizes below 50% during 30-minute test.
              Mixed GC pauses remain below 200ms.
Validation:   Run identical load test. Compare O% trend on jstat output
              and Mixed GC pause duration in GC log.
Rollback:     Revert -Xmx to 4g and redeploy.
```

This document is written before the change, stored in the incident or tuning ticket, and updated with actual results after the test.

### Step 5: One Change at a Time

Apply the change. Run the identical load test. Compare to baseline. Three outcomes:

1. **Improvement**: keep the change, document the result, identify next hypothesis
2. **No change**: the hypothesis was wrong. Roll back. Revisit the data from Step 3.
3. **Regression**: the change made things worse. Roll back immediately. Re-examine the hypothesis.

Changing multiple flags simultaneously is untestable. Never do it.

### Tuning Anti-Patterns

| Anti-Pattern | Why It Fails |
|-------------|-------------|
| Increasing -Xmx to fix OOM without heap dump analysis | Memory leaks consume any heap size eventually |
| Setting -Xmx = container limit | Leaves no room for Metaspace, Code Cache, off-heap |
| Disabling GC logging in production for performance | Removes the only evidence for post-incident analysis |
| Tuning flags without a load test baseline | No way to know if the change helped |
| Using JDK 8 Serial GC flags on JDK 11+ service | Flag semantics changed; some flags are deprecated or removed |
| Switching to ZGC without CPU budget accounting | ZGC uses 10–15% more CPU; may cause CPU throttling in pods with tight requests |

## VP Governance View

### Three JVM Health Signals on Every Service Dashboard

These three metrics should be on every service's operational dashboard. Not application-level metrics — JVM-level metrics that correlate directly to SLO risk.

| Signal | Prometheus / Datadog Metric | Target | Alert Threshold |
|--------|----------------------------|--------|----------------|
| GC Overhead % | `jvm_gc_overhead_percent` or derived from GC log | < 2% | > 5% |
| Old Gen Utilization % | `jvm_memory_used_bytes{area="heap",id="G1 Old Gen"}` / max | Stable (flat trend) | Trending upward > 2 hours |
| Full GC frequency | `jvm_gc_pause_seconds_count{cause="G1 Compaction Pause"}` | 0 (steady state) | Any nonzero in production |

### Governance Decision: When to Declare a Performance Crisis

| Condition | Immediate Action | 24-Hour Action |
|-----------|-----------------|----------------|
| Full GC > 1/hour | Pod restart rotation; increase heap as temporary mitigation | Heap dump analysis, root cause identification |
| Old Gen > 85% sustained | Same | Capacity headroom decision: scale out or fix leak |
| GC overhead > 10% | Emergency tuning session | p99 SLO breach analysis; likely already breached |
| Container OOM-kill events | Increase pod memory limit as emergency | Memory budget audit across all regions |

### JVM Upgrade as a Performance Investment

Frame JDK upgrade decisions in operational ROI, not engineering preference.

**JDK 8 → JDK 17**:
- G1GC improvements: 10–25% GC pause reduction for equivalent workloads (documented in JDK release notes and Azul benchmark suite)
- Container support improvements: accurate cgroup v2 detection
- ROI: reduced p99 latency, fewer GC-induced SLO breaches, smaller CPU headroom requirement

**JDK 17 → JDK 21**:
- Virtual threads: eliminates thread pool saturation for I/O-bound services
- Generational ZGC: sub-millisecond GC pauses without the throughput penalty of prior ZGC
- ROI frame: "Migrating account-api from JDK 11 to JDK 21 is projected to reduce p99 latency by 15–20% through virtual thread adoption. This allows us to reduce pod CPU requests by 20% (currently over-provisioned to absorb thread-pool saturation spikes), saving approximately $X/month in EKS node costs."

### GC Pressure and SLO Burn Rate

A 4.5-second Full GC pause in a service with a 99.9% availability SLO (8.77 hours/year budget):
- One Full GC event: 4.5 seconds of unavailability consumed
- At 1 Full GC/hour: 4.5 seconds/hour × 8,760 hours/year = 39,420 seconds/year = 10.95 hours burned
- Result: SLO breached within the first hour of sustained Full GC activity

This makes JVM GC health a direct input to SLO burn rate, not a separate "platform health" concern.

Cross-reference: [Availability Management] — SLO burn rate calculation for GC-induced outages follows the same error budget accounting as any other outage event. Track Full GC duration as an availability event in your SLO tooling (Nobl9, Sloth, Datadog SLOs), not just as a JVM metric.

Cross-reference: [Environment Management] — JDK version standardization across environments (dev/staging/prod) is a prerequisite for GC behavior to be reproducible across load tests. A service running JDK 8 in staging and JDK 17 in production will have different GC characteristics; load test results cannot be transferred to production capacity decisions.
