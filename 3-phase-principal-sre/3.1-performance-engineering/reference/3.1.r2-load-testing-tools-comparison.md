# Load Testing Tools Comparison

Comparison of the four primary open-source load testing tools. Use with ch06 for tool selection context.

## Summary Decision Table

| Criteria | JMeter | Gatling | k6 | Locust |
|----------|--------|---------|-----|--------|
| Script language | XML + Groovy/BeanShell | Scala DSL | JavaScript | Python |
| Learning curve | High (GUI-heavy, XML verbose) | Medium (Scala required) | Low (JavaScript) | Low-Medium (Python) |
| Performance (max TPS per node) | ~1,000–5,000 TPS (JVM overhead) | ~10,000–50,000 TPS (Akka actors) | ~10,000–30,000 TPS (Go runtime) | ~1,000–10,000 TPS (Python GIL limit) |
| Distributed execution | Manual (JMeter controller + agents) | Gatling Enterprise (paid) or manual | k6 Cloud (paid) or k6 operator (K8s) | Built-in distributed mode (master-worker) |
| CI/CD native | jmeter-maven-plugin, Jenkins | gatling-maven-plugin | k6 CLI, GitHub Actions, native | locust CLI, Docker |
| Protocol support | HTTP, WebSocket, JMS, JDBC, FTP, LDAP, SMTP | HTTP, WebSocket, JMS (enterprise) | HTTP, WebSocket, gRPC, Kafka | HTTP, WebSocket (extensible via gevent) |
| Real-time results | Basic dashboard (listener plugins) | Gatling report (post-test HTML) | stdout + Grafana integration | Web UI (live) + Prometheus |
| Cost | Free (open source) | Free (OSS) / Enterprise (paid) | Free (OSS) / Cloud (paid) | Free (open source) |
| Best for | Legacy systems (JMS, JDBC protocols), enterprise mandates | High TPS validation, code-as-test teams | Modern CI/CD pipelines, developer-owned tests | Python teams, custom behavior, rapid scripting |

## JMeter

**Architecture**: Java application with GUI. Tests defined in XML (.jmx) or via GUI. Executes tests using Java threads (one thread per virtual user).

**Strengths**:
- Widest protocol support: HTTP, HTTPS, WebSocket, JMS, JDBC, FTP, LDAP, SMTP, TCP, SOAP/XML
- Large plugin ecosystem (JMeter Plugins Manager)
- GUI for test design (useful for non-developers)
- Established in enterprise environments (often the approved tool)
- JMeter DSL (code-based, no GUI) available for modern usage

**Weaknesses**:
- Thread-per-VU model: JVM overhead limits VU count per node. 1,000 VUs ~= 1GB heap.
- XML-based .jmx files are hard to review in code review (large, unreadable diffs)
- GUI dependency creates CI/CD friction (headless mode requires careful configuration)
- Performance of JMeter itself limits test fidelity at high TPS (the tool becomes the bottleneck)

**When to use**:
- Existing .jmx test library that cannot be migrated
- JMS protocol testing (RabbitMQ via AMQP, IBM MQ, ActiveMQ) — no equivalent in other tools
- JDBC testing (database load directly, not via HTTP)
- Organization mandates JMeter as approved tool

**Command-line execution** (headless, for CI/CD):
```bash
jmeter -n -t test-plan.jmx \
  -l results.jtl \
  -e -o report-output/ \
  -Jbase_url=https://staging-perf.internal \
  -Jtarget_tps=500
```

**Distributed mode**:
- One JMeter controller node + N JMeter server (agent) nodes
- Controller sends test plan to agents, agents execute and send results back
- Requires network connectivity between controller and all agents
- Agent startup: `jmeter-server -Djava.rmi.server.hostname=<agent-ip>`

## Gatling

**Architecture**: Scala application using Akka actors. Each virtual user is an actor, not a thread. This enables very high VU counts with low resource overhead.

**Strengths**:
- Very high TPS per node (Akka actor model — non-blocking I/O)
- Tests written in Scala DSL: code reviews, version control, IDE support
- Built-in HTML report with percentile graphs, request timeline, concurrent user chart
- Strong assertion API: `.check(status.is(200))`, `.check(responseTimeInMillis.lte(500))`
- Excellent for recording → playback (Gatling Recorder)
- Gatling Enterprise adds distributed execution, CI/CD dashboards, comparison

**Weaknesses**:
- Scala requirement: non-Scala teams have a steep learning curve
- Gatling Enterprise for distributed = significant cost
- Report is post-test only (no real-time during test without Enterprise)
- Protocol support more limited than JMeter (no native JMS, JDBC)

**When to use**:
- Highest TPS simulation required on single node (10,000–50,000 TPS)
- Team is comfortable with Scala
- Post-test HTML reports are sufficient for stakeholders
- Performance-as-code is a team requirement (Gatling tests go in the source repo)

**Sample simulation structure**:
```scala
class AccountApiSimulation extends Simulation {
  val httpProtocol = http.baseUrl("https://staging-perf.internal")
  
  val scn = scenario("Account View")
    .feed(csv("users.csv").circular)
    .exec(http("login").post("/api/auth/login").body(StringBody(...)).check(status.is(200)))
    .pause(5, 15)  // think time 5–15 seconds
    .exec(http("account-view").get("/api/account/${userId}").check(status.is(200), responseTimeInMillis.lte(300)))
  
  setUp(scn.inject(
    rampUsersPerSec(1).to(100).during(10.minutes),
    constantUsersPerSec(100).during(50.minutes)
  )).assertions(
    global.responseTime.percentile3.lte(300),  // p99 <= 300ms
    global.failedRequests.percent.lte(0.1)
  )
}
```

## k6

**Architecture**: Go runtime with JavaScript scripting engine (goja). Each virtual user is a goroutine. Non-blocking I/O. Very low resource overhead.

**Strengths**:
- JavaScript scripting: widest developer familiarity
- CLI-first design: purpose-built for CI/CD
- Built-in Prometheus output: `k6 run --out prometheus=<url>`
- Thresholds (pass/fail criteria) built into script — CI/CD gate in one file
- k6 operator for Kubernetes: run distributed tests natively in EKS
- Extensions ecosystem: k6 xk6 for custom protocols
- JavaScript + TypeScript support

**Weaknesses**:
- No native JMS or JDBC support (HTTP/WebSocket/gRPC only in OSS)
- No GUI (CLI only)
- Distributed execution requires k6 Cloud (paid) or k6 operator setup
- JavaScript scripting can be surprising for engineers expecting Java semantics

**When to use**:
- CI/CD-native load testing pipeline (GitHub Actions, Jenkins)
- Kubernetes-native distributed testing (k6 operator on EKS)
- Developer-owned performance tests (in-repo JavaScript tests)
- Modern REST/gRPC services (no JMS/JDBC needed)
- T-Mobile recommendation: k6 + k6 operator on EKS for standardized PE automation

**Sample script**:
```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '10m', target: 100 },  // ramp to 100 VUs
    { duration: '50m', target: 100 },  // hold
    { duration: '5m',  target: 0 },    // ramp down
  ],
  thresholds: {
    'http_req_duration{p(95)}': ['p(95)<300'],  // p95 <= 300ms
    'http_req_failed': ['rate<0.001'],           // error rate < 0.1%
  },
};

export default function () {
  const loginRes = http.post(`${__ENV.BASE_URL}/api/auth/login`, 
    JSON.stringify({ username: `user_${__VU}`, password: 'test' }),
    { headers: { 'Content-Type': 'application/json' } }
  );
  check(loginRes, { 'login 200': r => r.status === 200 });
  
  sleep(Math.random() * 10 + 5);  // think time: 5–15 seconds
  
  const accountRes = http.get(`${__ENV.BASE_URL}/api/account/${__VU}`);
  check(accountRes, {
    'account 200': r => r.status === 200,
    'account fast': r => r.timings.duration < 300,
  });
  
  sleep(Math.random() * 15 + 10);  // think time: 10–25 seconds
}
```

**k6 operator on EKS** (distributed):
```yaml
apiVersion: k6.io/v1alpha1
kind: TestRun
metadata:
  name: account-api-load-test
spec:
  parallelism: 5        # 5 k6 pods, each runs portion of VUs
  script:
    configMap:
      name: k6-test-scripts
      file: account-api-load.js
  arguments: --vus 500 --duration 60m --out prometheus=remote_write_url
```

## Locust

**Architecture**: Python event-driven framework using gevent (coroutines). Tasks defined as Python classes and methods. Web UI provides real-time visibility.

**Strengths**:
- Python scripting: maximally flexible, can import any Python library
- Built-in distributed mode (master + workers, no external tooling)
- Real-time web UI at port 8089: live RPS, response time, error rate, user count
- Easy to write custom user behaviors: conditional flows, stateful sessions, complex auth
- pytest integration: unit test your load test scripts
- Best for custom protocols or non-standard authentication flows

**Weaknesses**:
- Python GIL limits single-core throughput: ~300–1,000 RPS per worker process
- Distributed mode requires manual management (multiple processes or containers)
- No built-in HTML report (use Locust's CSV export + external visualization)
- CPU-bound Python code is not suitable for high-TPS simulation

**When to use**:
- Python engineering teams (no Scala/JS learning curve)
- Complex user session logic that is hard to express in other tools
- Exploratory testing (live UI shows impact of parameter changes immediately)
- Services with custom protocol or authentication (OAuth PKCE, SAML, custom headers)

**Sample Locust script**:
```python
from locust import HttpUser, task, between
import random

class AccountUser(HttpUser):
    wait_time = between(5, 25)  # think time: 5–25 seconds between tasks
    
    def on_start(self):
        # Called once per VU at startup — perform login
        response = self.client.post("/api/auth/login", json={
            "username": f"user_{random.randint(1, 10000)}",
            "password": "test"
        })
        self.token = response.json()["access_token"]
    
    @task(10)  # weight 10: runs 10x more than weight-1 tasks
    def view_account(self):
        self.client.get("/api/account/me", 
                       headers={"Authorization": f"Bearer {self.token}"})
    
    @task(3)
    def check_bill(self):
        self.client.get("/api/billing/current",
                       headers={"Authorization": f"Bearer {self.token}"})
    
    @task(1)
    def make_payment(self):
        self.client.post("/api/billing/payment",
                        json={"amount": 89.99, "method": "card_on_file"},
                        headers={"Authorization": f"Bearer {self.token}"})
```

**Distributed execution**:
```bash
# Master (coordinates, shows UI)
locust -f load_test.py --master --host=https://staging-perf.internal

# Workers (execute load, report to master) — run on separate machines/pods
locust -f load_test.py --worker --master-host=<master-ip>
```

## CI/CD Integration Comparison

| Dimension | JMeter | Gatling | k6 | Locust |
|-----------|--------|---------|-----|--------|
| Pass/fail in pipeline | jtl-reporter plugin or custom script parsing .jtl | Built-in `.assertions()` in Simulation | Built-in `thresholds` in script options | Custom via --csv output + script |
| Jenkins | JMeter plugin | gatling-maven-plugin | k6 CLI step | Shell step |
| GitHub Actions | Docker-based action | Docker-based action | k6 official action (`grafana/k6-action`) | Docker Compose |
| GitLab CI | Docker | Docker | Native k6 binary | Docker |
| Kubernetes native | Not native | Not native | k6 operator (CRD) | Worker pods |
| Results export | .jtl (custom format) | HTML report + .log | JSON, CSV, InfluxDB, Prometheus | CSV, Prometheus |
| Trend dashboard | BlazeMeter, Grafana+InfluxDB | Gatling Enterprise | Grafana Cloud k6, Grafana+InfluxDB | Grafana+Prometheus |

## Recommendation for T-Mobile / EKS Context

**Primary tool: k6**
- k6 operator on EKS enables distributed load testing as a Kubernetes-native CRD
- JavaScript scripting aligns with modern developer culture
- Prometheus output integrates with existing Grafana dashboards
- CI/CD native (GitHub Actions k6 action with threshold-based pass/fail)
- No external infrastructure required beyond EKS cluster

**Secondary tool: JMeter (for JMS/messaging protocols)**
- If load testing RabbitMQ consumers via AMQP or Kafka producers via JMS
- JMeter's native JMS sampler is the most capable option for messaging protocol testing
- Run headless in Docker: `docker run --rm -v $PWD:/test justb4/jmeter -n -t /test/plan.jmx`

**Profiling complement (not a load tool, but used alongside)**:
- async-profiler: production-safe CPU/allocation/lock profiling during load tests
- Java Flight Recorder (JFR): zero-overhead always-on recording, analyze with JMC after test
