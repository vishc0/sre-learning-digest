# Performance Metrics Definitions

Complete reference for performance metrics used across this framework. Every metric includes: precise definition, measurement method, target range, and common misinterpretations.

## Latency Metrics

| Metric | Precise Definition | Measurement Method | Target Range | Common Misinterpretation |
|--------|-------------------|--------------------|--------------|--------------------------|
| **Response Time** | Total elapsed time from request initiation (client sends first byte) to response completion (client receives last byte). Includes network transit + server processing + response transmission. | Client-side measurement (load test tool, browser RUM, synthetic monitoring) | Varies by SLO | Often confused with server-side latency. Response time always >= server latency due to network. |
| **Latency (server-side)** | Time from server receiving complete request to server sending first response byte. Server processing time only — excludes network. | Server-side (APM instrumentation, framework filter) | p95 <= 200–500ms depending on service type | Often used interchangeably with response time, but they are different when network is significant. |
| **p50 (median)** | The latency value below which 50% of requests complete. Half of requests are faster than this value. | Calculated from latency histogram using percentile function | 50–100ms for interactive services | Often confused with average. p50 != average. Average is inflated by outliers; p50 is not. |
| **p95** | Latency below which 95% of requests complete. 1 in 20 requests exceeds this value. | Latency histogram, p95 function | <= 300ms for interactive; <= 500ms for API | The primary SLO target for user-facing services. Above p95: outliers. Below p95: the experience for 95% of users. |
| **p99** | Latency below which 99% of requests complete. 1 in 100 requests exceeds this value. | Latency histogram, p99 function | <= 1000ms for interactive; <= 2000ms for API | At 1,000 TPS: 10 requests/second exceed p99. At T-Mobile scale (25M/day = 289 TPS): ~3 requests/second are worst-case. |
| **p99.9 (p999)** | Latency below which 99.9% of requests complete. 1 in 1,000 exceeds this. | Latency histogram | Depends on SLA; typically 5–10x p99 | Meaningful for high-TPS services. At 10,000 TPS, p999 = 10 requests/second worst-case — still operationally significant. |
| **Max latency** | Single worst observed latency in the measurement window. | Histogram max value | Not used as SLO target | Unstable — varies with test duration. A single GC pause or cold DB query makes this spike. Use p99.9 instead. |
| **Average latency** | Arithmetic mean of all observed latencies. | Sum of all latencies / count | Not used as SLO target | Misleading — one 2000ms request among 999 fast requests raises average significantly. Masks tail problems. |

## Throughput Metrics

| Metric | Precise Definition | Measurement Method | Target Range | Common Misinterpretation |
|--------|-------------------|--------------------|--------------|--------------------------|
| **TPS (Transactions Per Second)** | Number of complete transaction cycles (request + response) per second at the measurement point. | Count of completed transactions in 1 second at the defined measurement boundary | Service-specific (see ch04) | "TPS" means different things at different layers. Always specify: gateway TPS, service TPS, DB TPS. |
| **RPS (Requests Per Second)** | HTTP requests per second. For most REST services, RPS = TPS. For GraphQL or batch endpoints, a single request may represent many logical transactions. | HTTP access log count per second | Same as TPS target | One GraphQL request with 5 resolver calls is 1 RPS but 5 "transactions" in business logic terms. |
| **Throughput** | Amount of useful work completed per unit time. Can be expressed as TPS, MB/s (data pipeline), records/sec (batch), or messages/sec (broker). | Depends on system type | Varies — measure against capacity baseline | "Throughput" without units is meaningless. Always specify what unit is being measured. |
| **Goodput** | Throughput of successful, correct transactions only (excludes retries, errors, timeouts). | (Successful responses / total time) — excludes HTTP 4xx/5xx | Should match TPS target under normal load | A system may show high TPS but 20% error rate — throughput looks fine, goodput is 20% lower. |

## Concurrency Metrics

| Metric | Precise Definition | Measurement Method | Notes |
|--------|-------------------|--------------------|-------|
| **Concurrent Users** | Number of users with an active session at a given instant. | Session store count, APM session tracking | Highest-level capacity metric. Drives session memory sizing. |
| **Active Users** | Users who submitted at least one request in the last N seconds (N = typically 5–60 seconds). | APM or application instrumentation | Lower than concurrent users — the load-generating subset. |
| **Virtual Users (VU)** | Load test simulation of a single user executing a script (including think times). | Load test tool (k6, Gatling) | VU count does NOT equal TPS. VU × actions_per_second_per_VU = TPS. |
| **Concurrency (Little's Law N)** | Average number of requests simultaneously in-flight in the system. N = lambda × W. | Derived from throughput × latency | Direct indicator of thread pool and connection pool pressure. |
| **Thread Pool Utilization** | Percentage of pool threads actively executing tasks. | active_threads / max_threads × 100 | > 80% sustained = approaching saturation. > 95% = at saturation, rejections imminent. |
| **Queue Depth** | Number of tasks waiting in thread pool queue for an available thread. | executor.queue.remaining (Actuator metric, inverted) | Any sustained queue depth > 0 indicates saturation. Queue depth = added latency. |

## Error and Reliability Metrics

| Metric | Precise Definition | Calculation | Target | Notes |
|--------|-------------------|-----------|---------|----|
| **Error Rate** | Percentage of requests resulting in an error response (HTTP 5xx, timeout, connection refused). | (error_count / total_requests) × 100 | < 0.1% at normal load; < 0.5% at peak load | HTTP 4xx are not errors in the performance sense (client error, not service error). Monitor 5xx separately from 4xx. |
| **Timeout Rate** | Percentage of requests that exceed a defined time limit and are terminated by the client or load balancer. | (timeout_count / total_requests) × 100 | < 0.01% at normal load | Distinguishable from errors — timeouts indicate the service received the request but didn't respond in time. |
| **Apdex Score** | Application Performance Index. Satisfaction score from 0–1 based on response time thresholds T (satisfied), 4T (tolerating), >4T (frustrated). Apdex = (Satisfied + 0.5 × Tolerating) / Total | APM tool calculation, or manual against latency histogram | > 0.9 = excellent; 0.7–0.9 = acceptable; < 0.7 = poor; < 0.5 = unacceptable | T threshold must be set per service. T=300ms for interactive UI; T=1000ms for background API. Wrong T makes Apdex meaningless. |
| **SLO Compliance Rate** | Percentage of requests that meet the service's defined SLO (e.g., respond within 300ms with no error). | (requests_within_SLO / total_requests) × 100 | Equal to SLO target (e.g., 99.9% if SLO is 99.9%) | More business-meaningful than raw latency. Directly ties to error budget. |
| **Error Budget** | 1 - SLO availability. The permitted failure quota per time period. | (1 - SLO_target) × time_period_in_minutes | Context-dependent | 99.9% SLO → 43.2 minutes/month error budget. A performance incident consuming 5 minutes is ~12% of monthly budget. |

## Saturation Metrics

| Metric | Precise Definition | Measurement | Danger Threshold |
|--------|-------------------|-----------|----|
| **Saturation Point (TPS)** | The TPS level at which the system transitions from the linear zone to the degradation zone — where adding load increases latency non-linearly. | Stress test: measure where p95 begins climbing steeply | System should not reach saturation at expected peak load |
| **Heap Utilization** | JVM heap used as percentage of -Xmx. | (heap_used / heap_max) × 100 via jstat or JMX | > 80% sustained: GC pressure. > 90%: imminent OOM risk. |
| **GC Overhead** | Percentage of CPU time spent in GC (not running application code). | (GC_time / wall_clock_time) × 100 via JVM GC log | > 5%: concerning. > 10%: performance crisis. > 25%: JVM is thrashing. |
| **CPU Throttling** | For containers: percentage of time the container's CPU is throttled by the kernel (cgroup limits). | Kubernetes: `container_cpu_cfs_throttled_seconds_total` / `container_cpu_cfs_periods_total` | > 25%: thread pool starvation risk. > 50%: severe performance impact. |
| **Connection Pool Pending** | Number of threads waiting for a connection from the pool (all connections in use). | HikariCP: `hikaricp.connections.pending` | > 0 for > 10 seconds: pool undersized or downstream is slow. |
| **Kafka Consumer Lag** | Number of messages in a topic partition that have been produced but not yet consumed. | `kafka-consumer-groups.sh --describe` LAG column | Depends on SLA. Growing lag = consumer cannot keep up with producer. |
| **RabbitMQ Queue Depth** | Messages in queue waiting for consumer. | RabbitMQ management API: `/api/queues/%2F/<queue_name>` `messages` field | Growing depth = consumers saturated. Trigger autoscaling (KEDA). |

## JVM-Specific Metrics

| Metric | Definition | Source | Target |
|--------|-----------|--------|--------|
| **Young GC Frequency** | Young (minor) GC events per minute | GC log, jstat (`YGC` column) | < 2/min. > 5/min: Eden too small or allocation rate too high. |
| **Full GC Frequency** | Full (major) GC events per hour | GC log, jstat (`FGC` column) | < 1/day preferred. Any Full GC during load test: investigate immediately. |
| **GC Pause Duration (p99)** | 99th percentile stop-the-world pause time | GC log pause time entries | < 200ms for G1GC. < 10ms for ZGC/Shenandoah. |
| **Old Gen Utilization** | Old generation heap used as % of Old Gen size | jstat `O%` column | < 60% steady-state. > 80% sustained: memory leak or heap undersized. |
| **Thread Count** | Total JVM threads (daemon + non-daemon) | jcmd Thread.print count, JMX ThreadMXBean | Service-specific. Stable count = healthy. Growing count = thread leak. |
| **Class Count** | Total loaded classes | JMX ClassLoadingMXBean.loadedClassCount | Should be stable after warmup. Growing class count = classloader leak (common with dynamic code generation frameworks). |

## Little's Law Derived Metrics

| Derivation | Formula | Use Case |
|------------|---------|---------|
| Required thread count for TPS target | threads = TPS_target × avg_latency_seconds | Validate thread pool sizing before load test |
| TPS ceiling from thread pool | max_TPS = thread_count / avg_latency_seconds | Identify thread pool as TPS bottleneck |
| Concurrency at TPS + latency | N = TPS × latency_seconds | Validate connection pool sizing |
| TPS limit from connection pool | max_TPS = pool_size / avg_query_seconds | Identify DB connection pool as TPS bottleneck |
