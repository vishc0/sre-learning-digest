# JVM Flags Reference — Production

Reference card for JVM startup flags used in production Java services. Organized by concern. Use with ch02 for tuning context.

## Heap Sizing

| Flag | What It Controls | Recommended Value / Rationale | Risk If Wrong |
|------|-----------------|-------------------------------|---------------|
| `-Xms<size>` | Initial heap size | Set equal to -Xmx (e.g., -Xms4g). Prevents heap resizing at startup which causes GC pressure during JVM warmup. | If Xms << Xmx: JVM starts small, spends first 5–15 minutes in frequent GC as heap expands. Distorts load test warmup results. |
| `-Xmx<size>` | Maximum heap size | Container: 75% of container memory limit. Pod with 8GB limit: -Xmx6g. Leave 25% for JVM non-heap (Metaspace, Code Cache, thread stacks, OS buffers). | Too large: OOM kill of container (Linux OOM killer kills process when container exceeds limit — JVM doesn't know container limit). Too small: frequent Full GC, OOM errors. |
| `-Xmn<size>` or `-XX:NewRatio=<n>` | Young generation size | NewRatio=2 means Old:Young = 2:1 → Young = 33% of heap. For high-allocation services (request processing), increase Young gen (NewRatio=1). | Young gen too small: frequent Young GC, objects promoted to Old Gen prematurely → Old Gen pressure. Young gen too large: less Old Gen space, longer Young GC pause. |
| `-XX:MetaspaceSize=<size>` | Initial Metaspace size | 256m for most services. Prevents Metaspace resize-and-GC events at startup. | Default (21MB) causes repeated Metaspace resize GC events during class loading. Distorts startup performance. |
| `-XX:MaxMetaspaceSize=<size>` | Metaspace ceiling | 512m. Without this, Metaspace grows unbounded → process OOM. Particularly important for services loading many classes (Spring Boot, Hibernate). | Unset: Metaspace can consume all available memory. Set too low: OOM on class loading (e.g., after dynamic proxy generation). |

## Garbage Collector Selection

| Flag | What It Controls | Recommended Value / Rationale | Risk If Wrong |
|------|-----------------|-------------------------------|---------------|
| `-XX:+UseG1GC` | Enable G1GC | Default for Java 9+. Best general-purpose GC for heap 2–32GB with pause targets. | Not wrong per se — it IS the default. Explicitly setting confirms you chose it, not inherited default. |
| `-XX:MaxGCPauseMillis=<ms>` | G1GC target pause time | 200ms default. Lower (100ms) for latency-sensitive services. G1GC adjusts region size to try to meet this target — it is a target, not a hard limit. | Too low (20ms): G1GC becomes too aggressive, increases GC frequency to stay under pause target, hurts throughput. Too high (1000ms): G1GC is permissive, allows long pauses that breach SLOs. |
| `-XX:+UseZGC` | Enable ZGC | Use for services requiring p99 latency < 10ms OR heap > 32GB. ZGC pauses are consistently < 1ms. Available JDK 15+ (production-ready). | Higher CPU overhead than G1GC (5–15%). Do not use for CPU-constrained pods. |
| `-XX:+UseShenandoahGC` | Enable Shenandoah GC | Alternative to ZGC for sub-10ms pauses. Available in OpenJDK / Red Hat builds. | Similar CPU overhead to ZGC. Not available in Oracle JDK (only OpenJDK). |
| `-XX:+UseSerialGC` | Enable Serial GC | Use ONLY for tiny heap (< 256MB), single-threaded batch jobs, or CLI tools. | Never use in a multi-threaded server — single-threaded GC causes unacceptable pause times under load. |
| `-XX:ParallelGCThreads=<n>` | GC thread count for parallel phases | Default = CPU_count / 2 for containers. For constrained pods (2-4 CPU), set explicitly to avoid GC consuming all CPU. | Too many: GC starves application threads. Default can be too high in containers where JVM sees host CPU count, not container CPU limit (fixed by `-XX:ActiveProcessorCount` or JDK 11+ container awareness). |

## GC Tuning (G1GC Specific)

| Flag | What It Controls | Recommended Value / Rationale | Risk If Wrong |
|------|-----------------|-------------------------------|---------------|
| `-XX:G1HeapRegionSize=<size>` | G1 region size (1–32MB, power of 2) | Auto-calculated from heap size. Override only if you see "humongous allocations" in GC log (objects > 50% of region size). | Regions too small: humongous objects cause immediate Old Gen promotion. Regions too large: waste. |
| `-XX:G1NewSizePercent=<pct>` | Minimum young gen as % of heap | Default 5%. Increase to 20–30% for services with high allocation rate (request-per-thread models). | Too low: very small Eden → high Young GC frequency. |
| `-XX:G1MaxNewSizePercent=<pct>` | Maximum young gen as % of heap | Default 60%. For latency-sensitive services, reduce to 30% to limit pause time (smaller Eden = less to collect per pause). | Too high: G1GC can expand Young gen to 60% of heap → long Young GC pauses. |
| `-XX:InitiatingHeapOccupancyPercent=<pct>` | Old Gen % at which concurrent GC cycle starts | Default 45%. Lower to 30% for heap-pressure-sensitive services. Starts concurrent GC earlier → avoids Full GC. | Too high: concurrent cycle starts too late → Full GC required (stop-the-world). Too low: wastes CPU on premature concurrent cycles. |

## Monitoring and Diagnostics

| Flag | What It Controls | Recommended Value / Rationale | Risk If Wrong |
|------|-----------------|-------------------------------|---------------|
| `-Xlog:gc*:file=/var/log/jvm/gc-%t.log:time,uptime,level,tags:filecount=5,filesize=20m` | Unified GC logging (JDK 11+) | Always enable in production. Captures: pause times, GC cause, heap occupancy before/after. Rotation keeps log footprint bounded (5 × 20MB = 100MB max). | Without GC logging: cannot diagnose GC-related performance incidents. This is the first thing asked for during an investigation. |
| `-XX:+HeapDumpOnOutOfMemoryError` | Auto-capture heap dump on OOM | Always enable. Without this, an OOM event produces no artifact for analysis. | Not wrong to omit, but missing this means OOM events are undiagnosable. Heap dump can be large (equal to -Xmx) — ensure disk space or use `-XX:HeapDumpPath=/tmp`. |
| `-XX:HeapDumpPath=<path>` | Heap dump location | `/var/log/jvm/` or a mounted volume. In Kubernetes: mount a PVC or use an init container to stream to S3. | Default (working directory of JVM process) may not be writable or have space. |
| `-XX:+PrintFlagsFinal` | Print all JVM flag values at startup | Add during load test setup, capture in startup log. Confirms actual flags applied (some flags override others). Remove in normal production. | If you don't know what flags are active, you cannot reason about JVM behavior. |
| `-Dcom.sun.management.jmxremote.port=9090 -Dcom.sun.management.jmxremote.ssl=false -Dcom.sun.management.jmxremote.authenticate=false` | JMX remote access | Enable in performance test environments for live JConsole/VisualVM monitoring. NOT recommended in production without SSL+auth. | In production without auth/SSL: JMX is an unauthenticated admin interface. Use Kubernetes `kubectl port-forward` + localhost JMX instead. |
| `-XX:+FlightRecorder -XX:StartFlightRecording=duration=60s,filename=/tmp/recording.jfr` | Java Flight Recorder | JDK 11+ (free, no commercial license required). Zero overhead for continuous mode. Add to production JVM for always-on diagnostics. | Low-overhead mode is truly low (< 1% CPU). Profile mode adds overhead — run briefly only. |

## Crash Handling

| Flag | What It Controls | Recommended Value / Rationale | Risk If Wrong |
|------|-----------------|-------------------------------|---------------|
| `-XX:ErrorFile=<path>` | JVM crash log (hs_err_pid.log) | `/var/log/jvm/hs_err_pid%p.log`. Default goes to process working directory. | Default location may not be accessible for collection. |
| `-XX:OnOutOfMemoryError="kill -9 %p"` | Action on OOM | For containers: force immediate pod death so Kubernetes can restart it cleanly. Without this, an OOM-afflicted JVM may limp along in a broken state. | In non-containerized environments: kill -9 terminates without cleanup. Use only in container environments where Kubernetes manages restart. |
| `-XX:+ExitOnOutOfMemoryError` | Exit on OOM (JDK 8u92+) | Cleaner alternative to `OnOutOfMemoryError="kill -9 %p"`. Triggers clean JVM shutdown on OOM. | Service stops on first OOM — may mask transient OOM spikes that would have recovered. Combine with HeapDumpOnOutOfMemoryError to capture artifacts. |

## Container-Specific Flags

| Flag | What It Controls | Recommended Value / Rationale | Risk If Wrong |
|------|-----------------|-------------------------------|---------------|
| `-XX:+UseContainerSupport` | JVM respects container CPU/memory limits | Default ON in JDK 11+. Explicitly set for JDK 8u191+. Without this, JVM sees host resources, not container limits. | Without this on JDK < 11: JVM allocates heap based on host RAM → heap exceeds container limit → OOM kill. |
| `-XX:ActiveProcessorCount=<n>` | Override CPU count for GC thread sizing | Set to container CPU limit (e.g., 2 for 2-CPU pod). JVM uses this to size GC threads, ForkJoinPool, etc. | Without this, JVM may see host CPU count → over-provision GC threads → CPU throttling in container. |
| `-XX:MaxRAMPercentage=75.0` | Set max heap as % of container memory | Alternative to explicit -Xmx for dynamic pod sizes. 75% leaves room for off-heap. | > 80%: risks container OOM kill when off-heap (Metaspace, Direct Memory, thread stacks) is added. |
