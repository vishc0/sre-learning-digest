# Thread Pool Sizing Guide

Reference formulas and worked examples for sizing thread pools across service types. Use with ch03 for tuning context.

## Core Formula

**Thread Pool Sizing Formula (I/O-bound services)**:
```
threads = CPU_cores × target_CPU_utilization × (1 + wait_time / service_time)
```

Where:
- `CPU_cores`: container CPU limit (not host CPU count — see JVM flags for container awareness)
- `target_CPU_utilization`: 0.7 for most services (leave 30% for GC, OS, jitter). Use 0.5 for latency-critical services.
- `wait_time`: average time thread is blocked on I/O per request (database call, external API, file read)
- `service_time`: average CPU-active time per request (computation, serialization, business logic)

**Quick rule of thumb**:
- Pure CPU-bound: threads = CPU_cores (no I/O factor)
- Mixed I/O (typical REST service): threads = CPU_cores × 5–15
- High I/O (mostly waiting, minimal CPU): threads = CPU_cores × 20–50

## HTTP Worker Thread Pool

**Service type**: Spring Boot / Tomcat (BIO or NIO connector), thread-per-request model.

### Sizing Example: Account API (2-CPU pod)

```
CPU cores: 2
Target utilization: 0.7
Wait time per request: 120ms (60ms DB query + 40ms external auth check + 20ms Redis)
Service time per request: 15ms (JSON parsing, business logic, serialization)

threads = 2 × 0.7 × (1 + 120/15)
        = 2 × 0.7 × 9
        = 12.6 → 13 threads

Queue capacity: 10× max threads = 130
Rejection policy: AbortPolicy → throw 503 to caller (do NOT use CallerRunsPolicy — blocks HTTP acceptor)
```

Configuration (Spring Boot `application.yml`):
```yaml
server:
  tomcat:
    threads:
      min-spare: 10
      max: 13
    accept-count: 130  # queue depth
    connection-timeout: 5000  # 5s timeout to prevent indefinite blocking
```

### Sizing Example: High-TPS API Gateway (4-CPU pod)

```
CPU cores: 4
Target utilization: 0.7
Wait time: 80ms (upstream service call)
Service time: 5ms (routing, auth validation, header manipulation)

threads = 4 × 0.7 × (1 + 80/5)
        = 4 × 0.7 × 17
        = 47.6 → 48 threads

Tomcat max: 48
Queue (accept-count): 480
```

## Async Task Executor (Spring @Async)

**Use case**: background processing, email sending, event publishing, non-blocking side effects.

### Sizing Example: Notification Service

```
CPU cores: 2
Target utilization: 0.6 (lower — async tasks compete with HTTP threads)
Wait time: 200ms (SMS gateway API call)
Service time: 10ms

threads = 2 × 0.6 × (1 + 200/10)
        = 2 × 0.6 × 21
        = 25.2 → 25 threads
```

Configuration (Spring `@EnableAsync`):
```java
@Bean
public Executor asyncExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);      // always-alive threads
    executor.setMaxPoolSize(25);       // burst capacity
    executor.setQueueCapacity(500);    // backpressure buffer
    executor.setThreadNamePrefix("async-notification-");
    executor.setRejectedExecutionHandler(new ThreadPoolExecutor.AbortPolicy());
    executor.initialize();
    return executor;
}
```

## JDBC Connection Pool (HikariCP)

**HikariCP recommended formula**:
```
connections = (core_count * 2) + effective_spindle_count
```
Where `effective_spindle_count` = number of storage spindles (use 1 for SSD, NVMe, or cloud DB).

This formula is intentionally conservative. HikariCP's research shows that more connections beyond this formula actually decrease throughput due to database-side context switching.

### Sizing Example: Microservice with 4-CPU pod against RDS PostgreSQL (SSD)

```
core_count = 4 (pod CPU limit)
spindle_count = 1 (SSD/cloud DB)

connections = (4 × 2) + 1 = 9

HikariCP config:
  maximumPoolSize: 9
  minimumIdle: 5
  connectionTimeout: 3000ms   (fail fast if pool exhausted — don't queue indefinitely)
  idleTimeout: 600000ms       (10 minutes before idle connection closed)
  maxLifetime: 1800000ms      (30 minutes — rotate connections before DB timeout)
  validationTimeout: 5000ms
```

### Multi-Service Database Connection Pressure Table

| Services | Pool Size Per Service | Pods Per Service | Total Connections to DB |
|---------|----------------------|-----------------|------------------------|
| 5 microservices | 10 | 3 pods each | 5 × 10 × 3 = 150 |
| 10 microservices | 10 | 3 pods each | 10 × 10 × 3 = 300 |
| 20 microservices | 5 (reduced) | 2 pods each | 20 × 5 × 2 = 200 |

PostgreSQL default `max_connections = 100`. All three scenarios exceed the default.

**Solutions**:
1. PgBouncer in transaction mode: multiplexes to DB, each service sees a large pool, DB sees ~20 connections
2. Reduce pool size to Hikari's recommended formula (typically 5–9 per instance)
3. Increase `max_connections` in PostgreSQL (costs shared_buffers memory per connection: ~5MB each)

## Kafka Consumer Thread Pool

**Rule**: max parallelism = number of partitions. Each partition can only be consumed by one consumer in a group.

```
consumers_per_pod = topic_partitions / service_pod_count

# Kafka consumer threads per pod
consumer_threads = consumers_per_pod

# Required TPS from consumers
required_consumer_tps = messages_per_second / consumer_threads

# Verify: required_consumer_tps ≤ consumer_processing_capacity
consumer_processing_capacity = 1000 / avg_processing_time_ms
```

### Sizing Example: T-Mobile Event Consumer (25M/day)

```
messages per day: 25,000,000
messages per second (avg): 25M / 86,400 = 289 msg/sec
peak factor (5× for launch events): 289 × 5 = 1,445 msg/sec
topic partitions: 24
service pods: 6

consumers per pod: 24 / 6 = 4 consumer threads per pod
required throughput per thread: 1,445 / 24 = 60 msg/sec
required processing time per message: 1000 / 60 = 16.7ms maximum

If actual processing time = 50ms:
  max throughput per thread: 1000 / 50 = 20 msg/sec
  total throughput: 20 × 24 = 480 msg/sec — insufficient for peak

Fix options:
  a) Increase partitions to 72 (requires topic recreation or partition expansion)
  b) Increase pods to 18 (72 msg/sec × 24 = 1,728 msg/sec > 1,445 — sufficient)
  c) Optimize processing to < 17ms per message
```

Configuration (`application.yml`):
```yaml
spring:
  kafka:
    consumer:
      max-poll-records: 50          # batch size per poll — tune for throughput vs. latency
      fetch-min-size: 1             # fetch immediately, don't wait for batch fill
      fetch-max-wait: 100ms         # max wait for fetch-min-size
    listener:
      concurrency: 4                # consumer threads per pod (= partitions/pods)
      type: batch                   # batch processing for throughput
      ack-mode: batch               # commit after successful batch
```

## RabbitMQ Consumer Thread Pool

```
consumer_threads = (messages_per_second × avg_processing_time_seconds) × safety_factor

# Then verify: consumer_threads ≤ queue channel limit per connection
# RabbitMQ default channel limit: 2047 channels per connection
```

### Sizing Example: T-Mobile Alert Consumer

```
messages per second: 500 (normal)
peak: 2,000/sec (mass alert event)
processing time: 30ms per message

Normal sizing: (500 × 0.030) × 1.5 = 22.5 → 23 consumer threads
Peak sizing:   (2000 × 0.030) × 1.5 = 90 consumer threads

Strategy: use HPA to scale pods during peak.
  Normal: 3 pods × 8 consumer threads = 24 consumers
  Peak:   KEDA scales to 12 pods × 8 consumers = 96 consumers
          (KEDA trigger: RabbitMQ queue depth > 1000 messages)
```

Configuration (`SimpleMessageListenerContainer`):
```java
container.setConcurrentConsumers(8);       // minimum consumer threads per pod
container.setMaxConcurrentConsumers(16);   // max with dynamic scaling
container.setPrefetchCount(10);            // messages buffered per consumer
container.setAcknowledgeMode(AcknowledgeMode.AUTO);
```

## Scheduled Task Thread Pool

```
scheduler_threads = concurrent_jobs + 1 (always odd — avoids task pile-up)
```

Most applications: 1–5 scheduled thread pool threads. Scheduled tasks should be short (< 1 minute) or delegate work to async executor. Long-running scheduled tasks that block the scheduler thread delay other scheduled tasks.

Configuration:
```java
@Bean
public TaskScheduler taskScheduler() {
    ThreadPoolTaskScheduler scheduler = new ThreadPoolTaskScheduler();
    scheduler.setPoolSize(5);
    scheduler.setThreadNamePrefix("scheduled-task-");
    scheduler.setAwaitTerminationSeconds(30);
    scheduler.setWaitForTasksToCompleteOnShutdown(true);
    return scheduler;
}
```

## Summary: Quick-Reference Sizing Table

| Pool Type | Formula | Typical Range | Key Constraint |
|-----------|---------|---------------|----------------|
| HTTP workers (I/O-heavy) | cores × util × (1 + wait/service) | 10–100 | Must not exceed connection pool |
| HTTP workers (CPU-bound) | ≤ CPU cores | 2–8 | CPU throttling in container |
| Async executor | cores × 0.6 × (1 + wait/service) | 5–50 | Competes with HTTP workers for CPU |
| JDBC pool (HikariCP) | (cores × 2) + spindles | 5–15 per instance | DB max_connections ceiling |
| Kafka consumers | partitions / pods | 1–12 per pod | Limited by partition count |
| RabbitMQ consumers | msg/sec × processing_sec × 1.5 | 5–100 per pod | Channel limit per connection |
| Scheduled tasks | concurrent_jobs + 1 | 1–5 | Delegate long tasks to async pool |
