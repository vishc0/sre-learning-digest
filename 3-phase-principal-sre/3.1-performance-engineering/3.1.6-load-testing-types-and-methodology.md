# Chapter 6: Load Testing Types and Methodology

## The Five Load Test Types

### Load Test (Steady-State Validation)
**Purpose**: Validate that the system meets performance SLOs at expected peak load.
**Duration**: 30–60 minutes at target TPS after ramp-up.
**Ramp-up**: 10–15 minutes to reach target load (prevents cold-start distortion).
**Pass/fail criteria**:
- p95 latency ≤ SLO target throughout test
- Error rate ≤ 0.1%
- No memory growth (heap stable, no GC acceleration mid-test)
- TPS achieved ≥ 95% of injected load (indicating no request loss)

**When**: Before every major release that touches a high-traffic service.

### Stress Test (Breaking Point Discovery)
**Purpose**: Find the saturation point and characterize failure behavior beyond it.
**Duration**: Ramp until failure, hold briefly, observe recovery.
**Ramp-up**: Continuous increase, 10% increments every 2 minutes.
**What to observe**:
- At what TPS does p95 first exceed SLO? (performance ceiling)
- At what TPS does error rate exceed 1%? (reliability ceiling)
- At what TPS does the system fully fail? (hard capacity ceiling)
- Does the system recover when load drops? (recovery behavior)
- Does it fail hard (crash, OOM) or soft (queuing, timeouts, graceful degradation)?

**Key finding**: hard capacity ceiling should be ≥ 2× expected peak. If expected peak = 400 TPS, hard failure should not occur until ≥ 800 TPS.

### Soak Test (Endurance / Stability Test)
**Purpose**: Reveal failures that only manifest over extended duration: memory leaks, connection pool exhaustion, GC accumulation, thread leaks, file descriptor leaks.
**Duration**: 2–24 hours at 60–80% of validated capacity.
**Monitoring focus**:
- JVM heap growth over time (should be stable after warmup, not monotonically increasing)
- GC frequency: increasing GC frequency over time = memory pressure building
- Thread count: should be stable (not growing) — thread leak detection
- Connection pool: idle connections returning to pool (not being leaked)
- File descriptors: `lsof -p <pid> | wc -l` should be stable

**Pass/fail criteria**:
- Heap size stable within 10% variance after first GC cycle
- p95 latency does not degrade >20% from hour 1 to hour N
- No OOM errors
- No process restart required

**T-Mobile relevance**: Any Kafka consumer that runs continuously is effectively in a 24/7 soak test. Memory leaks in consumer code cause pods to OOM-kill every few days — the classic symptom is "the pod restarts itself every 3-4 days and nobody knows why." [Availability Management] SLO error budget burns silently during these restarts.

### Spike Test (Autoscaling and Resilience Validation)
**Purpose**: Validate that the system responds correctly to sudden traffic surges.
**Pattern**: Normal load → instant 5× or 10× spike → return to normal.
**What to observe**:
- Kubernetes HPA response time: how long from spike onset to new pods ready? (typically 2–4 minutes with default HPA)
- Error rate during scale-out gap: requests that arrive before new pods are ready
- Circuit breaker behavior: does it open correctly under spike load?
- Queue buildup: RabbitMQ/Kafka message lag during spike
- Recovery behavior: does p95 return to baseline within 5 minutes of spike subsiding?

**HPA tuning insight**: default Kubernetes HPA scales on CPU with a 15-second scrape interval and a 5-minute cooldown. For sub-minute spikes (market open, scheduled batch triggers), this is too slow. Options: KEDA (event-driven autoscaling on queue depth), pre-warming (scheduled scale-up before known events), or over-provisioning for predictable spikes. See [Capacity Management] for KEDA configuration patterns.

### Chaos + Performance Test (Degradation Characterization)
**Purpose**: Understand how performance degrades (not just when it fails) when infrastructure is impaired.
**Scenarios**:
- Kill one pod during peak load: does HPA replace it? What is the latency during replacement?
- Introduce 50ms network latency to the database: how does service p99 respond?
- Throttle CPU to 50%: at what TPS does p95 cross the SLO?
- Kill Redis (session store): does the service fail fast or hang?

**T-Mobile relevance**: Kafka broker rolling restart during consumer load — does the consumer group rebalance gracefully? Rebalance pause can be 10–60 seconds depending on `max.poll.interval.ms` configuration. At 25M messages/day (~290 TPS average, higher at peak), a 60-second rebalance = 17,400 unprocessed messages building in lag.

## Test Script Design

Key elements of a realistic load test script:

**1. User journey simulation** (not endpoint hammering)
Wrong: 500 virtual users all hitting GET /api/account simultaneously
Right: 500 virtual users each executing: login → view account → check bill → make payment (with think times)

**2. Dynamic data** (not the same user ID repeated)
Wrong: all virtual users use userId=12345
Right: parameterize from a CSV of 10,000 real-like test user IDs. Static data causes unrealistic caching behavior — cache hit rates will be artificially high, masking real DB load.

**3. Realistic think times**
Include pauses between steps. Use random distribution (Gaussian or uniform), not fixed. See ch05 for think-time modeling. Omitting think times inflates VU concurrency and artificially saturates thread pools.

**4. Session correlation** (capture dynamic tokens)
Extract session token from login response and pass to subsequent requests. Tools: JMeter correlation rules, Gatling `saveAs`, k6 `check` with `response.json()`. Without this, every request after login returns 401 and the test measures error handling, not performance.

**5. Assertions** (fail the test if responses are wrong)
Check: HTTP status 200, response body contains expected field, response time < threshold. Without assertions, a load test that returns 400 errors looks like "great performance" — low CPU, fast response times, all wrong.

## Load Injection Tools

| Tool | Language | Distributed | CI/CD | Best For |
|------|----------|-------------|-------|----------|
| JMeter | GUI + XML/Groovy | Via JMeter agents | jmeter-maven-plugin | Legacy, complex protocols (JMS, JDBC) |
| Gatling | Scala DSL | Via Gatling Enterprise | gatling-maven-plugin | High TPS, code-as-test |
| k6 | JavaScript | k6 Cloud or operator | Built-in | Modern teams, CI/CD native |
| Locust | Python | Built-in distributed mode | Via Docker/K8s | Python teams, flexible custom logic |

**T-Mobile stack**: k6 + k6 operator on EKS is the recommended path. k6 operator runs distributed load generation natively inside the cluster, avoiding NAT traversal and simulating internal service-to-service traffic patterns accurately.

k6 operator deployment:
```yaml
apiVersion: k6.io/v1alpha1
kind: K6
metadata:
  name: account-api-load-test
spec:
  parallelism: 4          # 4 pods, traffic split across them
  script:
    configMap:
      name: account-api-test-script
      file: test.js
  arguments: --vus 200 --duration 30m
```

## Engineering Deep-Dive: Test Environment Fidelity

The single most common reason a load test passes in staging and fails in production: the staging environment is not representative of production.

**Critical fidelity requirements**:

| Dimension | Requirement | Failure Mode if Wrong |
|-----------|-------------|----------------------|
| Data volume | Staging DB should have production-scale row counts | Queries use different query plans on small tables; p95 in staging = 80ms, production = 800ms |
| JVM heap sizing | Staging JVM -Xmx must match production | Staging heap is 2GB, production 8GB — GC behavior is completely different |
| Thread pool sizing | Must match production configuration | Saturation point in staging is 100 TPS; production saturates at 400 TPS |
| Network topology | Service mesh, load balancer, and sidecar must be present | Staging without Istio sidecar understates latency by 3–10ms per hop |
| Downstream dependencies | Real downstream services or calibrated stubs | Stubbed downstream returning 0ms hides real latency contributions |
| Pod count / HPA config | Must match production replica count and HPA thresholds | Autoscaling behavior is untestable if HPA min/max are different |

See [Environment Management] for staging environment provisioning standards that enforce fidelity requirements.

**Data fidelity for T-Mobile**: the account database has customers with 1 line vs. customers with 25 lines on a business account. Query performance is non-linear with account complexity. Load testing with simple test accounts will not reveal the p99 behavior caused by complex accounts.

## Performance Test Report Structure

**Engineering-level report** (full detail, appendix):
- Test configuration (duration, ramp, VU count, tool, environment)
- Raw latency percentiles per endpoint (p50, p75, p95, p99, p99.9)
- Throughput over time (TPS time series)
- Error rate over time
- Resource utilization (CPU, memory, GC, thread pool, connection pool)
- Bottleneck identification with evidence
- Comparison to previous baseline (regression check, delta %)

**VP-level summary** (1 page, lead document):
```
Service: Account Management API    Test Date: [date]
Test Type: Load Test (steady-state at expected peak)

RESULT: PASS

Target: p95 ≤ 300ms at 400 TPS     Actual: p95 = 245ms at 397 TPS  PASS
Target: Error rate < 0.1%           Actual: 0.02%                   PASS
Target: Capacity > 400 TPS          Actual: saturation at 520 TPS   PASS
                                    Headroom: 30% above expected peak

Key Finding: Acceptable for launch. DB query optimization (ticket #1234)
             implemented in this release improved p95 by 18% vs. baseline.

Recommendation: APPROVE for production release.
Sign-off: [PE Lead]  [SRE Manager]  [VP Operations]
```

> **VP Governance View**: Performance test reports exist to enable a binary decision: release or hold. If the PE team cannot produce a one-page summary with a clear PASS/FAIL recommendation, the report is not usable by executive leadership. Engineering detail belongs in an appendix — not the executive view. A VP who has to read 40 pages to find the answer will stop reading performance reports. See [ch08 — PE Program Governance] for the release gate framework and sign-off chain.
