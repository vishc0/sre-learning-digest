# Chapter 3: Thread and Concurrency Analysis

## Thread Pool Fundamentals

A thread pool has four tuneable parameters:

- **Core pool size**: threads that stay alive even when idle. JVM never destroys them below this count.
- **Max pool size**: burst ceiling. Threads above core size are created when queue is full and destroyed after keepAliveTime.
- **Queue capacity**: backpressure buffer. Tasks wait here when all core threads are busy.
- **Rejection policy**: what fires when max threads are busy AND queue is full.

Key insight: a thread pool is a backpressure mechanism, not just a performance knob. The queue absorbs bursts. When the queue fills, rejection policy fires. The four Java rejection policies:

| Policy | Behavior | Operational Consequence at T-Mobile Scale |
|--------|----------|-------------------------------------------|
| `AbortPolicy` | Throws `RejectedExecutionException` | In a payment service: failed transactions. Callers see 500 or exception. |
| `CallerRunsPolicy` | Caller thread executes the task | In an API gateway: the HTTP request thread blocks executing the task. Gateway throughput collapses because Tomcat/Netty threads are occupied. |
| `DiscardPolicy` | Silently drops the task | In a Kafka consumer: silent message loss. 25M/day pipe — you lose messages with zero visibility. |
| `DiscardOldestPolicy` | Drops oldest queued task | In a notification service: acceptable (latest state wins). In an audit event service: catastrophic — chronologically earliest events are deleted. |

Thread pool lifecycle states:

| State | What's Happening |
|-------|-----------------|
| Under load | Tasks execute as submitted, queue empty. Threads at or below core size. |
| At capacity | Core threads busy, queue filling. No new threads yet. |
| At saturation | Max threads busy, queue full. Rejection policy fires. |
| After spike | Threads above core size are destroyed after `keepAliveTime` elapses. |

## Thread Pool Sizing Formula

```
threads = CPU_cores × target_CPU_utilization × (1 + wait_time / service_time)
```

This is the CPU utilization formula from Brian Goetz's *Java Concurrency in Practice*.

**Worked example — payment service on a 4-core EKS pod:**

- CPU cores: 4
- Target CPU utilization: 0.7 (30% headroom for GC, OS overhead, noisy neighbors)
- Wait time (I/O wait per request — external PSP call): 80ms
- Service time (CPU-active time per request): 20ms

```
threads = 4 × 0.7 × (1 + 80/20) = 4 × 0.7 × 5 = 14 threads
```

**CPU-bound service (image processing, encryption):**

- Wait time ≈ 0, service time = 50ms
- `threads = 4 × 0.7 × (1 + 0/50) = 2.8` → round to 3–4 threads
- Adding threads beyond CPU count hurts throughput (context switch overhead dominates)

**I/O-bound service (database calls, external API):**

- Wait time = 200ms, service time = 10ms
- `threads = 4 × 0.7 × (1 + 200/10) = 4 × 0.7 × 21 = 58.8` → 60 threads
- JDBC connection pool must match: 60 threads × 1 connection each = 60 connections minimum

**Queue sizing:**

```
queue_depth = max_threads × average_queue_wait_tolerance / average_service_time
```

Conservative starting point: 10× max_threads. Too large a queue hides saturation (requests queue silently) and causes latency to spike without the error signal that would normally trigger scaling. A shallow queue makes saturation visible early.

## Thread Dump Analysis

**How to capture:**

```bash
# Local JVM by PID
jstack <pid>

# Container-friendly: SIGQUIT sends dump to stdout (caught by log aggregator)
kill -3 <pid>

# Kubernetes: PID 1 is typical in containers
kubectl exec <pod> -n <namespace> -- jstack 1

# Arthas (attach without restart): top CPU-consuming threads
# java -jar arthas-boot.jar → attach → thread -n 3
```

**Thread states:**

| State | Meaning | Action |
|-------|---------|--------|
| `RUNNABLE` | Executing or ready to run on CPU | Normal. High RUNNABLE count = good utilization. |
| `BLOCKED` | Waiting to acquire a monitor lock held by another thread | Contention problem. Find the lock holder. |
| `WAITING` | Called `Object.wait()`, `LockSupport.park()`, or `Thread.join()` | Check what it's waiting for. Common in thread pool workers. |
| `TIMED_WAITING` | Same as WAITING but with timeout | `Thread.sleep()`, `wait(timeout)`, `parkNanos()`. |
| `TERMINATED` | Thread completed execution | Should not appear unless there is a pool leak. |

**Reading a thread dump entry:**

```
"http-nio-8080-exec-15" #42 daemon prio=5 os_prio=0 tid=0x00007f... nid=0x2a3 waiting on condition
   java.lang.Thread.State: WAITING (parking)
     at sun.misc.Unsafe.park(Native Method)
     - parking to wait for  <0x000000076b5c2d08> (a java.util.concurrent.locks.ReentrantLock$NonfairSync)
```

- **Thread name**: `http-nio-8080-exec-15` — Tomcat HTTP worker thread 15. The number tells you how many threads the pool has allocated.
- **State**: `WAITING` on a `ReentrantLock` — this thread is blocked waiting for another thread to release a lock.
- **nid**: native OS thread ID. Use `ps -eLf | grep <nid_in_hex>` to correlate with OS-level CPU stats.

**Danger signal**: if 80%+ of HTTP worker threads are `BLOCKED` or `WAITING` on the same lock address — that lock is your throughput ceiling. Every new request stacks behind it.

## Deadlock Detection

A deadlock requires three conditions: mutual exclusion (a resource can only be held by one thread), hold-and-wait (a thread holds one resource while waiting for another), and circular wait (A waits for B, B waits for A).

**Common pattern in multi-resource transactions:**

```
Thread A: holds Lock-1, waiting to acquire Lock-2
Thread B: holds Lock-2, waiting to acquire Lock-1
→ Neither thread can proceed. Both threads park forever.
```

**JVM deadlock detection in thread dump:**

```
Found one Java-level deadlock:
=============================
"Thread-A":
  waiting to lock monitor 0x00007f... (object 0x..., a com.example.ResourceB)
  which is held by "Thread-B"
"Thread-B":
  waiting to lock monitor 0x00007f... (object 0x..., a com.example.ResourceA)
  which is held by "Thread-A"
```

The JVM identifies deadlocks automatically. Search for `Found one Java-level deadlock` in the dump.

**Prevention patterns:**

- **Lock ordering**: always acquire locks in the same global order across all code paths. If every thread acquires Lock-1 before Lock-2, circular wait is impossible.
- **Try-lock with timeout**: `lock.tryLock(100, TimeUnit.MILLISECONDS)` — fail fast and retry rather than parking forever.
- **Lock-free data structures**: `ConcurrentHashMap`, `AtomicLong`, `LongAdder` — eliminate the need for explicit locking on hot paths.

## Lock Contention Analysis

**Lock types and their trade-offs:**

| Lock Type | Acquisition Cost | Features | When to Use |
|-----------|-----------------|----------|-------------|
| `synchronized` | Low | Intrinsic, JVM-managed, reentrant | Simple mutual exclusion, low contention |
| `ReentrantLock` | Low | `tryLock()`, fairness option, condition variables | Need timeout, need fairness, need multiple conditions |
| `ReadWriteLock` | Medium | Multiple concurrent readers, exclusive writer | Read-heavy shared state (caches, config) |
| `StampedLock` (Java 8+) | Lowest for reads | Optimistic read mode — no locking on read fast-path | Highest read throughput; complex API, not reentrant |

**Identifying hotspots from a thread dump:**

1. Find all threads in `BLOCKED` state
2. Identify the lock address they are waiting on (`waiting to lock monitor 0x...`)
3. Find which thread holds that lock (`- locked <0x...>`)
4. Count how many threads are queued on the same lock address
5. If more than 3 threads queue on one lock → contention hotspot requiring refactoring

**Contention resolution options:**

- **Reduce lock scope**: minimize the code inside the synchronized block. Compute outside, lock only for the data write.
- **Lock striping**: partition the data structure so different keys use different locks. `ConcurrentHashMap` does this internally — 16 segments by default (Java 7), per-bucket in Java 8+.
- **Replace with `ReadWriteLock`**: if reads outnumber writes 10:1, readers can proceed concurrently.
- **Replace shared mutable state**: use actor model (Akka), message-passing, or immutable value objects to eliminate contention entirely.

## Connection Pool Analysis

**HikariCP — key metrics to expose:**

| Metric | Healthy Range | Action if Breached |
|--------|---------------|--------------------|
| `hikaricp.connections.active` | < 80% of pool size | Scale pool or reduce query latency |
| `hikaricp.connections.pending` | 0 | Pool is undersized or DB is slow |
| `hikaricp.connections.idle` | > 0 | Pool has headroom |
| `hikaricp.connections.timeout.total` | 0 (any non-zero is an incident) | Immediate investigation |

**HikariCP pool sizing:**

HikariCP's own recommendation: `connections = (core_count × 2) + effective_spindle_count`

For a 4-core pod with SSD storage: `(4 × 2) + 1 = 9 connections`. This is deliberately small — Hikari's research shows that a smaller pool with a short queue outperforms a large pool because DB-side context switching is eliminated.

T-Mobile scale implication: 20 microservices × pool size 10 = 200 connections to one PostgreSQL instance. PostgreSQL defaults to `max_connections = 100`. This is a production outage waiting to happen. Either raise `max_connections` (with corresponding shared_buffers), deploy PgBouncer as a connection proxy, or reduce per-service pool sizes.

**RabbitMQ consumer thread sizing:**

```
consumers = (messages_per_second × average_processing_time_seconds) × safety_factor
```

At 25M messages/day = ~289 msg/sec. If average processing time = 50ms:
```
consumers = 289 × 0.050 × 1.5 = 22 consumer threads
```

If you have 4 pods: 22 / 4 = 6 consumer threads per pod.

**Kafka consumer group sizing:**

- Max parallelism is bounded by partition count. One consumer thread per partition maximum.
- `consumers_per_pod = partitions / pod_count`
- With 12 partitions across 4 pods: 3 consumer threads per pod
- Consumer lag = throughput < production rate. Diagnose with:

```bash
kafka-consumer-groups.sh --bootstrap-server <broker>:9092 \
  --group <group-id> --describe
# LAG column > 0 and growing → consumers are falling behind
```

## Async vs. Sync Threading Models

**Thread-per-request (Spring MVC / Tomcat default):**

- One OS thread blocked per HTTP request, from arrival to response
- Thread parks during any I/O (DB, HTTP call, file read)
- At 200 concurrent requests: 200 OS threads minimum — ~1MB stack each = 200MB stack memory alone
- Practical saturation point: 500–800 concurrent connections on a JVM with typical settings
- Simple to debug: one request = one stack trace = one thread

**Reactive / non-blocking (Spring WebFlux / Netty):**

- Small event loop pool: typically `2 × CPU_cores` threads (8-core = 16 event loop threads)
- Tasks decomposed into non-blocking callbacks; thread never parks on I/O
- 16 event loop threads can multiplex 10,000+ concurrent connections
- Cost: callback complexity, "callback hell," stack traces are fragmented across continuations, debugging is significantly harder
- Correct for high-concurrency I/O-bound workloads; wrong for CPU-bound work (one slow CPU task blocks the event loop)

**Virtual threads (Java 21 / Project Loom):**

- JVM-managed lightweight threads; not OS threads
- Thread-per-request programming model preserved — no callback rewrites
- I/O blocking intercepted by JVM: virtual thread unmounts from OS thread, OS thread takes another virtual thread
- One OS thread can multiplex thousands of virtual threads
- Eliminates thread-per-request scalability ceiling without reactive complexity
- Spring Boot 3.2+ enables virtual threads with a single property: `spring.threads.virtual.enabled=true`
- T-Mobile implication: Java 21 migration is a performance team priority for any service above 300 concurrent requests

**Decision table:**

| Model | Best For | Avoid When | Stack |
|-------|----------|------------|-------|
| Thread-per-request | CRUD, low-medium concurrency (<500 RPS/pod) | High concurrency, I/O-heavy | Spring MVC, Tomcat |
| Reactive | High concurrency, streaming, I/O-intensive | CPU-bound work, complex business logic | WebFlux, Netty, RxJava |
| Virtual threads | New services on Java 21+, brownfield migration from blocking | Java < 21 | Spring Boot 3.2+, JDK 21 |

## Concurrency Anti-Patterns

**Thread pool exhaustion:**
All threads busy, queue full, rejection policy fires. Callers receive errors or hang. Cascading: the caller's thread pool then backs up. Circuit breaker on the thread pool prevents cascade. Monitor `hikaricp.connections.pending` and thread pool queue depth as leading indicators. CPU and memory are lagging indicators — the pool exhausts long before CPU saturates. See [Availability Management] for how pool exhaustion triggers SLO burn.

**Thundering herd:**
All clients retry simultaneously after a brief outage. The recovering service is flooded before it is stable. Result: second outage. Prevention: exponential backoff with jitter.

```python
sleep_seconds = min(cap, base * (2 ** attempt)) + random.uniform(0, base)
# cap at 30s, base = 1s → retries spread across a window rather than synchronized
```

**Lock convoy:**
Many threads queue behind a lock whose holder does expensive work inside the critical section. Symptom: many `BLOCKED` threads, low CPU utilization despite high reported load (threads are parked, not computing). Prevention: move all expensive computation outside the lock; lock only the final state mutation.

**Thread leak:**
Threads created but never terminated, or executor not shut down on application stop. Memory consumption grows monotonically over hours. Only visible in a soak test (1+ hour sustained load). Detection: thread count graph trending upward under steady load. Fix: register a JVM shutdown hook that calls `executor.shutdown()` and `executor.awaitTermination()`.

> **VP Governance View**

> Thread pool saturation is the most common mechanism for a gradual performance degradation that escalates to a P1 incident. The failure mode is silent: queue depth grows, latency climbs, and by the time error rates appear in dashboards, the service is already saturated. APM dashboards must expose thread pool queue depth per service — not just CPU and memory. CPU at 40% while thread pool queue depth is at 90% capacity is a pre-incident condition. Tie this metric to SLO burn rate alerting [Availability Management] so the on-call team has a 20-minute warning, not a 2-minute warning. Budget conversations for vertical scaling (more cores per pod) or horizontal scaling (more pods) must reference validated thread pool sizing data, not gut feel.
