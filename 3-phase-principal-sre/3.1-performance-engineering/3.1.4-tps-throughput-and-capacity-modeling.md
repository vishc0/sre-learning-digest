# Chapter 4: TPS, Throughput, and Capacity Modeling

## TPS Definition by System Type

TPS is not a universal metric. Define it precisely for each system layer, because "we handle 500 TPS" is meaningless without specifying the measurement point.

| System Type | What Counts as One Transaction | Measurement Point |
|-------------|-------------------------------|-------------------|
| REST API | One HTTP request-response cycle | API gateway ingress or service ingress |
| Message broker (Kafka/RabbitMQ) | One message produced OR one message consumed — specify which | Producer acknowledgment or consumer commit offset |
| Database | One SQL statement execution (not one business operation) | JDBC driver or DB connection pool |
| Batch processing | One record processed | Batch job controller / ItemWriter |
| gRPC service | One RPC call-response cycle | gRPC server interceptor |
| GraphQL API | One query execution (not one field resolver) | GraphQL engine execution layer |

**T-Mobile example — measurement point multiplier:**

A single customer API request counts as:
- 1 TPS at the API gateway
- 1 TPS at the account-service
- 3 TPS at the database (3 queries: account, entitlements, billing)
- 2 TPS at the Redis cache (2 lookups)

Reporting "we handle 500 TPS" at the API gateway understates database load by 3× and cache load by 2×. Capacity planning that ignores the multiplier produces undersized data tiers.

## Little's Law

```
N = λ × W
```

Where:
- **N** = average number of requests in the system simultaneously (concurrency, in-flight count)
- **λ** = arrival rate (TPS)
- **W** = average time each request spends in the system (latency in seconds)

**Worked example:**

- p50 latency (W) = 200ms = 0.2 seconds
- Throughput (λ) = 500 TPS
- Concurrency (N) = 500 × 0.2 = **100 concurrent in-flight requests**

To serve 500 TPS at 200ms average latency, the system must hold 100 simultaneous in-flight requests. Your thread pool, connection pool, and heap must be sized for 100 concurrent units of work, not just for 500 per second.

**Corollary 1 — latency doubles, capacity doubles:**
If latency increases from 200ms to 400ms (DB slow, upstream degraded), concurrency rises from 100 to 200 to maintain the same TPS. You need 2× thread pool, 2× connection pool, 2× heap capacity for the same throughput. Latency degradation cascades into resource exhaustion.

**Corollary 2 — latency reduction is free capacity:**
If caching or query optimization reduces latency from 200ms to 100ms, concurrency drops from 100 to 50. You can double TPS without adding a single pod or thread.

**Corollary 3 — thread pool as TPS ceiling:**
```
Max_TPS = thread_count / average_latency_seconds
```
60 JDBC connection pool threads at 200ms average query time: `60 / 0.2 = 300 TPS database capacity`. If peak demand is 400 TPS, you are 25% under-provisioned at the database tier regardless of how many app pods you run.

---

> **VP Governance View**
>
> Little's Law translates thread pool configuration into a business capacity number. "We have 60 database connections at 200ms average query time = 300 TPS database capacity. Our expected peak is 400 TPS. We have a 25% capacity gap." That is a capital funding conversation that a VP can approve in 30 seconds. Without this framing, engineers ask for "more pods" without explaining why, and finance sees line items instead of risk. See [Capacity Management] for the decision framework.

## The Capacity Curve

The throughput-latency relationship has three distinct zones as load increases:

```
Latency
  |                                              /
  |                                             /
  |                                            / ← Zone 3: Saturation
  |                                   ________/
  |                         _________/  ← Zone 2: Knee (target for peak)
  |_________________________/
  |←     Zone 1: Linear    →|
  |__________________________|___________________|_____________ TPS
  0                       ~70%               ~90%          100%
                                               ↑
                                        Saturation Point
```

**Zone 1 — Linear (0% to ~70% of saturation TPS):**
Throughput increases proportionally with load. Latency remains flat or climbs slightly. The system has headroom. This is the safe operating region for average load.

**Zone 2 — Knee (70% to ~90% of saturation TPS):**
Throughput still increases but latency starts climbing non-linearly as queue depths build. A 10% increase in load produces a 30–50% increase in p95 latency. This is where the system should sit at peak load — above this, degradation is rapid and difficult to recover from without shedding load.

**Zone 3 — Saturation (>90% of saturation TPS):**
- Throughput plateaus or drops (request queuing, connection timeouts, rejection policy fires)
- Latency increases dramatically — queue wait time dominates service time
- GC pressure increases as objects accumulate in queues and response objects are not freed promptly
- Error rate rises: timeouts, connection pool exhaustion, circuit breakers open

**Operations rule:** design so that expected peak load = 70% of the measured saturation point. The 30% headroom absorbs unexpected spikes without entering Zone 3. At 25M messages/day with a saturation point of 400 msg/sec, peak design target is 280 msg/sec with auto-scaling triggers at 320 msg/sec.

## USL — Universal Scalability Law

The USL is a mathematical model that predicts throughput as a function of concurrency (number of parallel workers or nodes). It extends Amdahl's Law by adding a coherency penalty term that makes throughput eventually decrease with scale.

```
X(N) = N / (1 + σ(N−1) + κN(N−1))
```

Where:
- **X(N)** = system throughput at concurrency N
- **σ** (sigma) = contention coefficient — fraction of work that is serialized (Amdahl's serial fraction)
- **κ** (kappa) = coherency coefficient — coordination overhead between parallel workers (data sharing, cross-node invalidation)

**What the coefficients mean operationally:**

| σ value | Meaning | Example |
|---------|---------|---------|
| 0 | Perfect linear scaling | Stateless, share-nothing service |
| 0.05 | 5% serialized — max ~20× speedup | Typical well-tuned microservice |
| 0.20 | 20% serialized — max ~5× speedup | Service with a shared lock or single-threaded component |

| κ value | Meaning | Example |
|---------|---------|---------|
| 0 | No coherency overhead | No shared state between nodes |
| 0.001 | Small coherency cost | Shared cache invalidation |
| 0.01 | Significant coherency | Distributed transaction coordination |

When κ > 0, there is a concurrency level at which adding more workers reduces throughput. Adding nodes beyond this point actively degrades performance.

**Finding the throughput peak:**

```
N_peak ≈ √((1 − σ) / κ)
```

Example: σ = 0.08, κ = 0.002:
```
N_peak = √(0.92 / 0.002) = √460 ≈ 21 concurrent requests
```

Beyond N = 21, coherency overhead exceeds the gain from parallelism. Adding more Kafka consumer threads, more pods, or more DB connections beyond this point reduces throughput.

**Practical usage:**
1. Run benchmarks at N = 1, 2, 4, 8, 16 concurrent users (hold each for 5 minutes, measure TPS)
2. Fit USL curve to the data (Python: `scipy.optimize.curve_fit` on the USL equation; or Baron Schwartz's `pdq-book` tooling)
3. The fitted σ and κ tell you whether the bottleneck is contention (reduce serial sections) or coherency (reduce cross-node coordination)
4. The fitted curve predicts what happens at N = 32, 64 without running the test

## TPS Baseline Protocol

A TPS baseline answers: "What is the current measured throughput capacity of this service?"

**Measurement protocol:**

1. Run load test in a production-equivalent environment [Environment Management] — same pod size, same JVM flags, same DB, same Redis tier
2. Ramp from 10% to 100% of expected peak in 10% increments. Hold each step for 5 minutes (allow JIT compilation, connection pool warm-up, and GC to stabilize)
3. Record at each step: TPS actual, p50, p95, p99 latency, error rate, thread pool queue depth, GC pause time
4. Identify the step where p95 crosses the SLO threshold OR error rate exceeds 0.1% — whichever comes first
5. That step's TPS = performance-validated capacity

**Example baseline table — account management API:**

| Load Step | TPS Actual | p50 | p95 | p99 | Error Rate | Thread Queue | Status |
|-----------|-----------|-----|-----|-----|------------|--------------|--------|
| 100 TPS | 98 | 45ms | 120ms | 180ms | 0.01% | 0 | OK |
| 200 TPS | 197 | 48ms | 135ms | 210ms | 0.01% | 0 | OK |
| 300 TPS | 294 | 55ms | 160ms | 350ms | 0.02% | 2 | OK |
| 400 TPS | 391 | 72ms | 240ms | 890ms | 0.08% | 11 | WARN — queue building |
| 500 TPS | 487 | 110ms | 420ms | 2,100ms | 0.45% | 38 | FAIL — p95 exceeds 300ms SLO |

- **Validated capacity**: 400 TPS (p95 = 240ms < 300ms SLO, error rate < 0.1%)
- **Saturation point**: between 400 and 500 TPS
- **Peak capacity recommendation**: 280 TPS sustained peak (70% of validated capacity)
- **Auto-scale trigger**: 320 TPS (80% of validated capacity — scale before entering Zone 2 knee)

## Peak Load Modeling

```
Required_TPS = Expected_Average_TPS × Peak_Factor × Safety_Factor
```

**Peak factors by domain:**

| Domain | Typical Peak Factor | Driver |
|--------|--------------------:|--------|
| E-commerce | 10–15× | Black Friday vs. February midweek |
| Telecom billing | 3–5× | End-of-month bill cycle |
| Financial market data | 5–20× | Market open, earnings announcements |
| T-Mobile network ops | 2–4× | New device launch day, major event |
| T-Mobile account API | 3–6× | Outage recovery (all apps reconnect simultaneously) |

**T-Mobile example — account management API capacity gap:**

- Normal: 200 TPS average
- Peak factor (new iPhone launch): 4×
- Safety factor: 1.3
- Required capacity: `200 × 4 × 1.3 = 1,040 TPS`
- Current validated capacity: 400 TPS
- **Capacity gap**: 640 TPS (63% under-provisioned for peak)
- Required pod scaling: from 3 pods (400 TPS) to ~8 pods (~1,100 TPS) if scaling is linear
- Caveat: USL analysis must confirm linear scaling assumption (σ and κ coefficients)

## Horizontal vs. Vertical Scaling at the TPS Level

**Horizontal scaling (more pods/nodes):**

- Effective when: bottleneck is CPU or memory, service is stateless, no shared coordination overhead
- TPS scales approximately linearly, discounted by USL contention: 4 pods at 400 TPS each ≈ 1,400 TPS (not 1,600 — load balancer overhead, shared DB pool contention)
- Kubernetes HPA default scaling metric is CPU utilization — but CPU is a lagging indicator of TPS saturation. Thread pool queue depth is a leading indicator. Configure HPA with custom metrics:

```yaml
metrics:
- type: External
  external:
    metric:
      name: thread_pool_queue_depth
      selector:
        matchLabels:
          service: account-api
    target:
      type: AverageValue
      averageValue: "10"  # scale when avg queue depth > 10
```

**Vertical scaling (larger instance/pod):**

- Effective when: bottleneck is single-threaded computation (GC, cryptography, serialization)
- JVM benefits from a larger heap on a single node: fewer GC pauses as a fraction of total time
- Diminishing returns beyond 8–16 cores for most Java services due to GC thread contention and lock contention

**When horizontal scaling does NOT help:**

| Scenario | Why Scaling Fails | Actual Fix |
|----------|------------------|------------|
| DB connection limit hit | All pods share the same max_connections ceiling | PgBouncer, read replicas, query optimization |
| External API rate limit | All pods share the same API quota | Request coalescing, caching, quota increase |
| Cache invalidation storm | More pods = more concurrent cache misses at invalidation | Probabilistic early expiry, request coalescing |
| Stateful service without affinity | Load balancer scatters sessions → cache misses → DB load | Sticky sessions or externalize state to Redis |

---

> **VP Governance View**
>
> Horizontal scaling is the default answer to TPS capacity problems. It is also the most expensive answer and often the wrong one. Before approving compute spend, require the performance engineering team to answer three questions: (1) Is the bottleneck CPU — if yes, horizontal scaling helps; (2) Is the bottleneck I/O — if yes, fix I/O (add caching, optimize queries) before adding pods; (3) Is the bottleneck a shared resource ceiling (DB connections, API quota, cache) — if yes, horizontal scaling actively makes it worse. Every compute scaling request must include a USL analysis, a bottleneck classification, and an alternative cost comparison. See [Capacity Management] for the full governance framework.

## TPS Reporting for Executives

A performance capacity report to a Director/VP must enable a funding decision in under 60 seconds.

```
Service: Account Management API
Measurement Date: [date]
Environment: EKS staging-perf (production-equivalent pod sizing, production DB replica)

Current Validated Capacity:    400 TPS    (measured, p95 ≤ 300ms SLO)
Expected Normal Load:          200 TPS    (50% of capacity — HEALTHY)
Expected Peak Load:            800 TPS    (200% of capacity — CRITICAL GAP)

Peak Headroom:                 −50%       (we are undersized for launch events)

Recommended Action:            Scale from 3 pods to 8 pods
Estimated Capacity After:     ~1,050 TPS (confirmed via USL model, σ=0.06)
Estimated Cost Delta:          +$4,800/month in EKS compute
Risk if Not Addressed:         P1 during iPhone launch day (historical: 4× traffic spike)
                               ~30 minutes degraded service, estimated $2.1M revenue impact

Decision Required By:          [date − 3 weeks before launch]
```

The format: current state → gap → recommended action → cost → risk of inaction → decision deadline. A VP either approves the spend, accepts the risk, or asks for an alternative. No ambiguity.
