# Chapter 7: Microservice vs. Monolith Performance Operations

## Monolith Performance Profile

A monolith runs as a single JVM process. All business logic, all thread pools, all connection pools exist within one JVM boundary.

**Monolith failure modes**:

**1. JVM heap exhaustion** → OutOfMemoryError → process crash → all features down simultaneously.
- Cause: memory leak (object retention), excessive object creation under load, heap sized too small.
- Detection: heap utilization trending toward -Xmx, GC frequency increasing, GC reclaiming less each cycle.
- Signal: `jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} > 0.85` for more than 5 minutes.

**2. Shared thread pool exhaustion** → all request processing queued → latency spikes across all features.
- Cause: one slow endpoint (DB timeout, external API hang) consumes all threads.
- Detection: thread pool queue depth growing, BLOCKED threads in thread dump.
- Symptom: a slow payment endpoint makes the login endpoint slow — they share the same Tomcat thread pool. This is the classic "noisy neighbor" problem within a monolith.

**3. Database connection pool saturation** → all features blocked on DB access.
- Cause: slow queries, high concurrency, connection pool undersized.
- Detection: HikariCP `hikaricp.connections.pending > 0`, `hikaricp.connections.timeout` counter incrementing.
- HikariCP alert: `hikaricp_connections_pending > 2` for more than 60 seconds = pool is undersized for current load.

**4. Stop-the-world GC pause** → all threads suspended → latency spike for all users simultaneously.
- Detection: GC log shows long pause times; latency spikes correlate with GC events in time series.
- G1GC default target: pause < 200ms (`-XX:MaxGCPauseMillis=200`). If exceeding 500ms regularly, heap is undersized or allocation rate is too high.
- ZGC / Shenandoah: sub-millisecond pauses but higher CPU overhead. Use for latency-sensitive services where GC pauses appear in p99.

**Monolith capacity modeling**:
- Single TPS ceiling for the entire application
- Vertical scaling (larger JVM, more CPU, more RAM) is the primary scaling option
- Horizontal scaling requires stateless design + session externalization (Redis/Memcached)
- Single point of failure: OOM or process crash = 100% downtime for all users

**Monolith tuning advantages**:
- Single heap to tune, single GC to analyze
- No network latency between "services" (in-process method calls = nanoseconds)
- Connection pool centralized — easy to audit and right-size
- Thread dump captures all application threads in one artifact (`jstack <pid>` or `kill -3 <pid>`)
- Profiler attaches to one process — no distributed trace correlation needed

## Microservice Performance Profile

### Network Latency Multiplication

Each service-to-service call adds:
- Network serialization/deserialization: ~0.5–2ms per call (JSON > Protobuf by 2–5×)
- TCP/IP overhead: ~0.1–0.5ms on same EKS cluster (same AZ)
- Service mesh (Istio sidecar): ~1–5ms added per hop for mTLS + header injection + telemetry
- Load balancer routing: ~0.5–1ms (kube-proxy iptables rules)

A 10-service synchronous call chain:
```
10 hops × (2ms serialization + 3ms Istio sidecar) = 50ms minimum latency
before any business logic executes.
```

At T-Mobile scale (25M messages/day, ~290 TPS average, ~800 TPS peak): 50ms of irreducible overhead per call chain is a significant fraction of a 300ms p95 SLO budget.

### The Call Depth Problem — Compound p99

If each service has p99 = 100ms, and a user journey calls 10 services in sequence:
- Probability all 10 calls complete within 100ms = 0.99^10 = 90.4%
- That means 9.6% of journeys experience at least one tail-latency event — nearly 1 in 10 users.

Practical rule: **journey p99 ≈ N × p95 of individual services** (not N × p50). Tail latency compounds through synchronous chains.

Concrete example:
```
Service A: p99 = 50ms
Service B: p99 = 80ms
Service C: p99 = 150ms  (external partner API)

Sequential chain A → B → C:
Naive estimate:  50 + 80 + 150 = 280ms
Actual journey p99 (with tail amplification): 400–600ms
```

This is why microservice architectures demand aggressive per-service p95 targets. If the journey SLO is 500ms p95 and the chain has 5 services, each service must target ~80ms p95, not 500ms.

### Fan-Out Amplification

Service A calls Services B, C, D in parallel (`CompletableFuture.allOf` / `Promise.all`):
- Latency = max(latency_B, latency_C, latency_D) — parallel reduces total duration
- But: Service A holds a thread until ALL downstream calls complete
- If D is slow or retrying, A's thread is held. A's thread pool drains. A becomes a bottleneck even though the performance problem is in D.
- The latency of the fan-out = the latency of the **slowest** downstream call

Solution:
```java
// Set per-downstream timeout — never let one slow call drain A's thread pool
CompletableFuture<Response> dFuture = callServiceD()
    .orTimeout(200, TimeUnit.MILLISECONDS)   // Hard timeout per downstream
    .exceptionally(ex -> fallbackResponse()); // Circuit break, don't rethrow
```

## Engineering Deep-Dive: Distributed Tracing for Bottleneck Isolation

A distributed trace shows span-level latency for every service and operation in a request journey. This is the primary performance analysis tool in a microservice architecture — the equivalent of a monolith thread dump + profiler combined.

Reading a trace:
```
Root span: API Gateway [total: 450ms]
├── Service A: account-lookup [120ms]
│   ├── cache-hit: redis [5ms]
│   └── db-query: account-db [110ms]   ← HOTSPOT — 91% of A's time
├── Service B: entitlement-check [80ms]
│   └── db-query: entitlement-db [72ms]
└── Service C: audit-log-write [250ms] ← HOTSPOT — blocks user response
    └── kafka-produce: audit-topic [240ms]
```

**Findings from this trace**:
1. account-db query (110ms): needs query plan analysis, index check. `EXPLAIN ANALYZE` on the query.
2. Service C audit-log write (240ms): audit logging must not block the user response. Make it async — fire-and-forget Kafka produce with no `get()` on the future. This alone saves 240ms from the critical path.

**Trace-driven test script construction**: use production traces to identify the actual call graph. Build load test scripts that replicate the real call sequence, not an assumed one. Traces from Dynatrace/Jaeger show which services are called synchronously vs. asynchronously, which endpoints are high-frequency, and which carry the most latency.

**Key trace metrics to extract**:
- Span duration histogram: p50/p95/p99 per span — identifies outlier spans
- Error span count per service: which service contributes most failures
- Trace depth: how many synchronous hops — deeper = higher compound tail risk
- Database span time as % of total: if DB spans > 60% of request time, the bottleneck is data access, not compute

**OpenTelemetry instrumentation** (vendor-neutral):
```java
// Auto-instrumentation via Java agent (zero code change)
java -javaagent:opentelemetry-javaagent.jar \
     -Dotel.service.name=account-api \
     -Dotel.exporter.otlp.endpoint=http://collector:4317 \
     -jar account-api.jar

// Manual span for custom operations
Span span = tracer.spanBuilder("account-db-query").startSpan();
try (Scope scope = span.makeCurrent()) {
    // DB call here
} finally {
    span.end();
}
```

## Service Mesh Performance Overhead

Envoy sidecar (Istio) per-pod costs at T-Mobile EKS scale:

| Resource | Per Pod | 100-pod cluster | 500-pod cluster |
|----------|---------|-----------------|-----------------|
| CPU | 50–150m cores | 5–15 cores | 25–75 cores |
| Memory | 50–200MB | 5–20GB | 25–100GB |
| Latency added | 1–5ms/hop | — | — |

100 microservices × 200MB sidecar memory = 20GB cluster memory consumed by sidecars alone. This is infrastructure cost that must be in the [Capacity Management] model.

**mTLS latency breakdown per hop**:
- TLS session resumption (cached): ~1ms
- TLS full handshake (new connection): ~5–15ms — reason connection pooling matters in service mesh
- Header propagation (trace ID, auth headers): ~0.1ms
- Authorization policy check: ~0.5ms

**Istio optimization options**:
```yaml
# Enable session resumption — reduces TLS handshake to ~1ms from ~10ms
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: account-api
spec:
  host: account-api
  trafficPolicy:
    connectionPool:
      http:
        h2UpgradePolicy: UPGRADE     # HTTP/2 multiplexing — reduces connection count
      tcp:
        maxConnections: 100
        connectTimeout: 30ms
        tcpKeepalive:
          time: 7200s
          interval: 75s
```

- Use `PeerAuthentication` STRICT only where mTLS is a compliance requirement — PERMISSIVE reduces overhead for internal dev/test namespaces
- Right-size sidecar resource requests per service traffic level — default cluster-wide values over-allocate for low-traffic services

## Database Connection Pool Per Microservice

**The fan-out problem at the database**:
- 20 microservices, each with HikariCP pool size = 10, running 3 pods each
- Total DB connections: 20 services × 3 pods × 10 pool size = 600 connections
- PostgreSQL default `max_connections = 100`
- Result: 500 connections refused at startup or under load → services cascade-fail

**Hikari right-sizing formula** (from HikariCP documentation):
```
pool_size = ((core_count * 2) + effective_spindle_count)
```
For a 4-core DB server with SSD (effective_spindle_count = 1): pool_size = 9 per service instance. Not 20. Not 50.

**Solutions for microservice connection fan-out**:

1. **PgBouncer** (connection pooler in transaction mode):
   ```
   [Services] → 600 connections → [PgBouncer] → 20 connections → [PostgreSQL]
   ```
   Services see their own pool; DB sees a fixed small pool. Essential for microservice architectures. Overhead: ~0.1ms per query.

2. **Reduce pool size**: 5–10 per service instance is almost always sufficient. The illusion that bigger pools = better performance is false — it causes DB-side lock contention.

3. **Read replicas**: route read-heavy services to read replica — splits connection load and DB CPU.

4. **Caching layer**: Redis in front of DB absorbs repetitive reads. At T-Mobile: account data reads (view account, check bill) are highly cacheable — TTL 60s can reduce DB connection pressure by 70–80%.

**T-Mobile implication**: if any EKS service defaults to HikariCP pool size 20 and runs at 10 pods × 15 microservices, that's 3,000 connection attempts. Without PgBouncer, this is a DB-level incident during any traffic spike.

## Monolith-to-Microservice Migration: Performance Regression Risk

Common regressions introduced during decomposition:

| Regression | Cause | Magnitude | Detection |
|------------|-------|-----------|-----------|
| Latency increase | In-process calls become network calls | +2–50ms per extracted call | Distributed trace: new network spans |
| Cache miss rate increase | Shared in-memory cache becomes Redis (distributed) | +1–5ms per lookup, +cache coherence complexity | Redis hit rate metric drops |
| Transaction boundary loss | Single ACID transaction becomes saga/2PC | +10–100ms coordination overhead | Trace shows compensating transactions |
| Synchronous coupling added | New service call inserted where monolith had a function call | As above | New span in trace on extracted path |

**How to detect regressions early — strangle pattern with performance gate**:
1. Establish monolith p95 baseline per user journey **before** decomposition starts
2. Route 5% of production traffic to the new microservice via canary (Argo Rollouts / Istio traffic split)
3. Compare p95 of canary vs. monolith baseline in Dynatrace — before full cutover
4. Accept up to 10% degradation as acceptable network overhead
5. Flag >10% degradation: investigate the critical path before promoting the canary

After extracting each service: re-run full journey load test. The test must simulate the full call chain, not just the extracted service in isolation.

## API Gateway Performance

API Gateway functions and their latency costs:

| Function | Typical Latency Add | Notes |
|----------|--------------------|----|
| TLS termination | 1–3ms | Amortized with session resumption |
| JWT validation | 1–5ms | RS256 validation requires public key crypto — slower than HS256 |
| Rate limiting | 0.5–2ms | Redis-backed token bucket — Redis latency dominant |
| Request routing | 0.1–0.5ms | Regex path matching |
| Response caching | −50 to −200ms | Cache hit eliminates upstream call entirely — net negative latency |
| Request/response transform | 1–10ms | JSON transformation is CPU-intensive at high TPS |
| mTLS to upstream | 1–3ms | Per upstream call with session resumption |

Total API gateway overhead: typically 5–25ms per request. This is a fixed cost that consumes part of the end-to-end performance budget allocated to the journey. Budget: `journey_SLO - gateway_overhead = upstream_SLO_budget`.

Example:
```
Journey SLO: 500ms p95
Gateway overhead: 20ms
Upstream budget: 480ms across all service hops
```

**Gateway as performance control plane**:
- Response caching at the gateway eliminates upstream calls for GET /catalog, GET /plans, GET /coverage-map — cacheable with TTL 5–60s. Can deflect 40–70% of read traffic from upstream services.
- Rate limiting protects services from traffic spikes — the gateway absorbs surge and queues/sheds before it propagates.
- Circuit breaker at the gateway prevents cascade: if account-api p95 > 2× SLO, gateway trips, returns 503 with Retry-After header — prevents service mesh from amplifying the failure.

> **VP Governance View**: The API gateway is a performance control plane, not just a routing layer. A well-configured gateway can protect services from 90% of abnormal traffic patterns. An under-configured gateway adds latency and becomes a single bottleneck for the entire service estate. The PE team should audit gateway configuration — caching policy, rate limits, circuit breaker thresholds — in every quarterly performance review. Gateway configuration is a governance artifact, not just a DevOps concern. See [ch08 — PE Program Governance] for the quarterly review agenda.
