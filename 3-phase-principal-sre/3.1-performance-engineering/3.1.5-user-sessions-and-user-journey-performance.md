# Chapter 5: User Sessions and User Journey Performance

## User Session Definition

A session is a stateful context associated with a single user's continuous interaction with a system. The boundary conditions vary by system type:

| System Type | Session Definition | Session Boundary |
|-------------|-------------------|-----------------|
| Web application | HTTP session object with session ID cookie | Browser close, explicit logout, or idle timeout (typically 30 min) |
| Mobile API | OAuth token lifetime | Token expiry (typically 60 min) or explicit logout |
| API (machine-to-machine) | Connection lifecycle or API key scope | TCP connection close or API key rotation |
| Kafka consumer | Consumer group membership epoch | Consumer restart or partition rebalance |
| Batch job | Job execution context (Spring Batch `JobExecution`) | Job start to final step completion |

**T-Mobile example — session transaction multiplier:**

A customer checking their bill creates one web session. That session generates:
- 1 auth transaction (authentication service, JWT validation)
- 3 API calls (account service, billing service, usage service)
- 1 PDF download (document service)
- 4 Redis cache lookups (account state, entitlements)

Total: 9 backend transactions per user session. At 10,000 concurrent sessions: **90,000 concurrent backend transactions**. Capacity planning that uses session count instead of transaction count underestimates backend load by nearly 10×.

## Concurrent User vs. Active User vs. Session

| Term | Definition | Load Test Implication |
|------|------------|----------------------|
| Concurrent users | Users who have an active session at the same instant | Direct driver of thread/connection/heap capacity |
| Active users | Users performing an action within a sliding window (e.g., last 5 minutes) | The load-generating subset. Typically 10–20% of concurrent users. |
| Registered users | Total user base | Irrelevant for capacity planning |
| Sessions | Count of live session objects in server or cache | Server-side memory cost |

**Why the distinction matters:**

An e-commerce site may have 50,000 concurrent sessions but only 5,000 active users generating requests at any moment. The other 45,000 are reading, thinking, or have minimized the browser. Load testing all 50,000 as active requestors overestimates TPS capacity requirement by 10× and produces an oversized (expensive) infrastructure bill.

T-Mobile account portal: typical active-to-concurrent ratio is 12–18%. Peak (bill payment day): 25–30% as users act quickly to pay before due date.

## Think Time

Think time is the realistic pause between consecutive user actions. It represents human reading, deciding, typing, and navigating — time the server is idle waiting for the next request.

**Why it matters in load tests:**

Without think time, a load test simulates users who submit the next request the instant the server responds. This is machine behavior. A 50-virtual-user test without think time can generate the same TPS as a 500-virtual-user test with 30-second think time. The test overstates concurrency by 10× and causes teams to provision 10× more capacity than production requires.

**Think time by interaction type:**

| Interaction | Typical Think Time |
|-------------|-------------------|
| Login page → submit credentials | 5–15 seconds |
| Search results → click an item | 10–30 seconds |
| Product detail page → add to cart | 15–60 seconds |
| Cart → begin checkout | 30–120 seconds |
| Checkout form → submit payment | 30–90 seconds |
| Automated health check | 0 seconds (no human think time) |
| Mobile push notification → user action | 2–10 seconds |
| T-Mobile plan comparison → select plan | 45–180 seconds |

**Pacing in load test frameworks:**

```scala
// Gatling — fixed pause
exec(http("view-bill").get("/api/bill"))
  .pause(15)

// Gatling — random uniform (more realistic)
  .pause(10, 30)

// Gatling — random exponential (most realistic, eliminates synchronized wave)
  .pause(Duration.ofSeconds(10).toMillis.toLong, Duration.ofSeconds(30).toMillis.toLong)
```

**TPS implication of think time:**

```
Effective_TPS = concurrent_users / (avg_service_time + avg_think_time)
```

500 concurrent users, 200ms service time, 30-second think time:
```
Effective_TPS = 500 / (0.2 + 30) ≈ 16.6 TPS
```

Without think time:
```
Effective_TPS = 500 / 0.2 = 2,500 TPS
```

Same 500 virtual users. 150× difference in generated load depending on think time configuration. Always validate think time assumptions with production access logs before a load test run.

## User Journey Modeling

A user journey is a sequence of transactions representing a complete business workflow. Individual service performance metrics are insufficient — the journey's end-to-end latency is what the user experiences.

**Critical path prioritization:**

1. Revenue-generating journeys (checkout, payment, subscription activation) — performance failure = direct revenue loss
2. High-frequency journeys (login, account view, usage check) — performance failure = support volume spike
3. Compliance-required journeys (audit log export, regulatory report) — performance failure = SLA breach or regulatory risk

**Example: T-Mobile new service activation journey:**

```
Step 1: Login (auth-service)                     → target p95: 200ms
Step 2: View available plans (catalog-service)    → target p95: 300ms
Step 3: Select plan (cart-service)                → target p95: 150ms
Step 4: Enter payment info (payment-service)      → target p95: 400ms
Step 5: Submit order (order-service)              → target p95: 500ms
Step 6: Confirmation email trigger (notification) → target p95: 200ms  [async, not on critical path]

Synchronous journey budget: 200 + 300 + 150 + 400 + 500 = 1,550ms
Journey SLO: complete activation in < 2,000ms p95
Headroom: 450ms (29%)
```

**Journey failure mode analysis:**

| Failure Point | Business Impact | Observability Gap |
|--------------|-----------------|-------------------|
| Step 2 (catalog) slow or down | User cannot see plans — zero revenue, 100% abandonment | Catalog latency not in checkout funnel dashboard |
| Step 4 (payment) slow | User abandons at highest-value step — worst abandonment point | Payment p99 not tracked separately from p95 |
| Step 6 (notification) fails | User unsure if order completed — support calls, duplicate orders | Async failure invisible to frontend |

The most dangerous failure is silent: Step 6 can fail while Steps 1–5 return 200 OK. The user never receives confirmation; they call support; agents process a duplicate order. Instrument async steps separately and alert on them even though they do not affect synchronous journey latency.

## Transaction Mix (Workload Model)

Real production load is not a uniform stream of identical requests. A workload model defines the realistic distribution.

**Example workload model — T-Mobile account API at 1,000 total TPS:**

| Transaction Type | % of Load | TPS | Cacheable | Write Load |
|-----------------|-----------|-----|-----------|------------|
| Account view (read) | 55% | 550 | Yes (Redis, 5-min TTL) | None |
| Usage check (read) | 25% | 250 | Yes (Redis, 1-min TTL) | None |
| Bill payment (write) | 12% | 120 | No | High DB write, external payment gateway |
| Plan change (write) | 5% | 50 | No | Complex transaction, 3 service calls |
| Account creation (write) | 3% | 30 | No | KYC checks, provisioning, 8 service calls |

**Why workload mix matters:**

A load test using only "account view" (cacheable read) produces excellent results — Redis hit rate >95%, DB load near zero. Introduce the realistic 20% write mix and the performance profile changes entirely: DB write contention, external payment gateway latency variance, and KYC service timeouts appear. The baseline looks healthy; production fails.

Rule: always test the production transaction mix. Extract it from access logs (Splunk query: `index=prod | stats count by request_type | eventstats sum(count) as total | eval pct=round(count/total*100,1)`).

## Session Capacity Planning

**Memory capacity formula:**

```
Session_Memory = Concurrent_Sessions × Average_Session_Object_Size
```

Example:
- 100,000 concurrent sessions
- Session object: 8KB (user context, cart state, JWT, entitlements snapshot)
- Session memory: 100,000 × 8KB = **800MB**

If sessions stored in JVM heap: 800MB is dedicated to session objects before any request processing heap allocation. On a pod with 2GB heap, 40% is consumed by sessions before a single request is processed.

If sessions stored in Redis: JVM heap pressure eliminated; Redis adds 1–2ms round-trip per session access. At 500 TPS with 3 session accesses per request: 1,500 Redis calls/sec — well within a Redis Cluster's multi-million ops/sec ceiling.

**Session timeout trade-offs:**

| Timeout | Memory Ceiling | UX Impact | Best For |
|---------|---------------|-----------|----------|
| 60 minutes | High — stale sessions consume memory for up to 1 hour after user leaves | Users rarely logged out unintentionally | Low-traffic, high-trust applications |
| 30 minutes | Moderate | Minor — brief lunch break may require re-login | Standard web applications |
| 15 minutes | Low | Noticeable — users logged out mid-task | Financial / high-security contexts |

Rule: session timeout is a memory capacity lever, not just a security control. Reducing from 60 to 30 minutes can cut session memory footprint by up to 40% under typical user behavior patterns.

## Session Affinity: Sticky Sessions vs. Stateless Design

**Sticky sessions (session affinity):**

Load balancer routes a user to the same pod for all requests in a session. The ALB Ingress annotation for EKS:

```yaml
alb.ingress.kubernetes.io/target-group-attributes: stickiness.enabled=true,stickiness.lb_cookie.duration_seconds=1800
```

- Pro: no session replication or Redis round-trip; session data stays local to pod heap
- Con: uneven pod utilization — pods accumulate long-lived sessions, others are under-used; pod failure loses all sticky sessions for that pod's users simultaneously
- Performance impact: if one pod has a "hot" session concentration (users from a single geographic region mapped to one pod), its p99 spikes while peers are at 30% load

**Stateless design (externalized session):**

All session state stored in Redis. Each request routable to any pod — true load balancing.

- Con: Redis round-trip per request: +1–2ms at p50, +5ms at p99 under heavy load
- T-Mobile context: Redis Cluster with 3-node replication, sub-2ms p95 read latency — acceptable for account portal. Not acceptable for sub-10ms latency SLOs on high-frequency machine APIs.
- Benefit: pod restarts, rolling deployments, and HPA scale-down have zero impact on user sessions

**Session replication (middle ground):**

Session state replicated across pod pairs without an external store. Common in Tomcat clustering and Hazelcast.

- No Redis dependency; some redundancy against pod failure
- Con: replication overhead on every session write; intra-cluster traffic increases with pod count; not compatible with stateless pod design preferred in EKS

**Recommendation for EKS/T-Mobile context:** stateless with Redis unless the service has a p50 latency SLO below 10ms, in which case evaluate sticky sessions with pod anti-affinity rules to distribute load.

## User Journey Performance Budgets

A performance budget pre-allocates the total journey latency SLO across each step. Teams that own steps get an allocation; exceeding it requires PE team sign-off. This enforces accountability before code ships, not after a prod incident.

**Format:**

```
Journey: Bill Payment
Owner: Payments Platform Team
User-Perceived SLO: p95 ≤ 3,000ms
Measurement Point: Frontend — time from submit button to confirmation screen render

Step                             Allocated   Notes
─────────────────────────────────────────────────────────────────────
DNS + TLS + Network (client)         100ms   Fixed — device/network dependent
API Gateway                           50ms   Fixed — TAL ingress overhead
Authentication (JWT validation)       75ms   Cached validation, Redis lookup
Account validation                   200ms   Account service, cached entitlements
Payment processing (external PSP)    800ms   External — Stripe/First Data SLA is 1s p95
Fraud check (internal ML model)      400ms   Internal — GPU-backed inference service
Order confirmation write             200ms   Write to order DB + publish Kafka event
Response serialization + render      100ms   Client-side — not server controllable
─────────────────────────────────────────────────────────────────────
Committed budget:                  1,925ms   (64% of 3,000ms SLO)
Reserved headroom:                 1,075ms   (36%) — absorbs variance, unknown unknowns
```

**How to enforce budgets:**

1. Instrument every step with distributed tracing (Jaeger, X-Ray, or Dynatrace). Each span is a step.
2. Define a SLO per span in the APM tooling. Alert when a span's p95 breaches its allocated budget.
3. Any team proposing a change that increases their span's p95 by >10% must get PE team review before merge.
4. Quarterly review: compare actual span durations against allocations. Reallocate headroom from fast steps to consistently slow steps.

---

> **VP Governance View**
>
> Performance budgets must be set before development begins, not after a degradation incident. When a payment integration team ships code that consumes 1,200ms instead of the budgeted 800ms, the journey's headroom is consumed, the SLO is at risk, and other teams have to find savings they did not plan for. The VP owns the journey SLO. Service teams get budget allocations from that SLO. Budget overruns require PE team sign-off — exactly as a cost overrun requires finance sign-off. The mechanism is distributed tracing with per-span SLO alerting [Availability Management]. The governance review cadence is the quarterly capacity review [Capacity Management]. Without budget accountability at the service-team level, journey SLOs degrade incrementally until they are routinely missed. See [ch08 — PE Program Governance] for the full budget governance model including the escalation path from span alert to VP decision.
