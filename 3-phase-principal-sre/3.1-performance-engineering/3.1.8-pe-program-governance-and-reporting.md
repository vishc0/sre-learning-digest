# Chapter 8: Performance Engineering Program Governance and Reporting

## Performance Engineering Program Structure

Three structural models:

**Model 1: Dedicated PE Team**
- Centralized team of 3–8 performance engineers
- Owns all load testing infrastructure, APM tooling, baseline library
- Consulted per-release for performance validation
- Pros: deep expertise, consistent tooling, clear accountability
- Cons: bottleneck (every release waits for PE team), disconnected from development context
- Best for: regulated industries, high-TPS critical services (payments, real-time billing)

**Model 2: Embedded PE (Champion Model)**
- Performance engineer embedded in each squad (1:4 or 1:6 PE:squad ratio)
- Owned by the PE center of excellence (standards, tooling, training)
- Reports to squad for delivery, dotted-line to PE CoE for standards compliance
- Pros: close to development context, faster feedback loops
- Cons: champions deprioritized in sprint pressure; standards drift across squads
- Best for: large organizations with many product teams, where PE CoE has enforcement authority

**Model 3: Shared Service (Self-Serve)**
- PE team provides tooling, frameworks, and review gate — squads run their own tests
- PE team reviews results and gives pass/fail recommendation
- Pros: scalable, empowers development teams, PE team focuses on hard problems
- Cons: requires investment in self-service tooling and squad-level training
- Best for: mature engineering orgs with strong SRE culture

**T-Mobile recommendation**: Model 3 (Shared Service) with Model 1 elements for critical paths (payment processing, network provisioning, real-time messaging at 25M+/day). Use KEDA + k6 operator in EKS for self-service load testing — squads run tests in their own namespace, PE team owns the pass/fail gate logic and baseline library. See [Capacity Management] for how PE team feeds capacity data to the infrastructure planning cycle.

## Performance Gate in the Release Process

The performance gate is a mandatory decision point in the release pipeline. Performance test results are a binary release criterion — not a recommendation.

Where it fits in CI/CD:
```
Code commit → Unit tests → Integration tests → Build artifact →
Deploy to staging-perf [Environment Management] →
Performance gate (automated load test + regression check) →
Staging sign-off → Production deploy (canary → full rollout)
```

The gate is MANDATORY for:
- Services with performance SLOs (any service with a latency target in the SLA)
- Services with TPS-based capacity commitments in the [Capacity Management] plan
- Any service on a revenue-critical user journey (checkout, activation, payment)

**Gate configuration in CI/CD** (k6 example):
```yaml
# .github/workflows/performance-gate.yml
performance_gate:
  thresholds:
    http_req_duration:
      - "p(95)<300"      # p95 must be under 300ms
      - "p(99)<1000"     # p99 hard ceiling
    http_req_failed:
      - "rate<0.001"     # Error rate < 0.1%
    http_reqs:
      - "rate>380"       # Must sustain ≥ 380 TPS (95% of target 400 TPS)
  regression_check:
    baseline_file: baselines/account-api-baseline.json
    max_degradation_pct: 10   # p95 must not be >10% worse than stored baseline
```

**Exception process** (when gate fails but release is business-critical):
1. PE Lead documents the specific failing metric, measured value, and root cause hypothesis
2. Engineering Manager confirms the business urgency in writing
3. VP Operations approves exception with explicit risk acceptance statement
4. Fix committed to backlog with P1 classification — not someday, not next quarter
5. Exception expires automatically: fix must ship within 2 sprints; if not, the release is rolled back or the service is put in a performance watch state

Exception log is a governance artifact. Number of exceptions per quarter is a KPI reported to VP. Trend: if exceptions are increasing, the release process is broken.

## Engineering Deep-Dive: Performance Regression Detection in CI/CD

**Automated baseline comparison** — the mechanism:
1. Every service has a stored baseline: p50/p95/p99 at a standard load profile (e.g., 100 TPS for 10 minutes)
2. Every merged PR triggers a standard load test on the staging-perf environment
3. Results compared automatically against the stored baseline
4. If p95 degrades > 10% from baseline → build fails, alert sent to team Slack + ticket auto-created
5. Baseline is updated only when intentional improvement ships — approved update via PE Lead, not automatic

Baseline file format:
```json
{
  "service": "account-api",
  "version": "2.14.0",
  "test_date": "2026-06-01",
  "load_profile": { "tps": 100, "duration_min": 10 },
  "p50_ms": 45,
  "p95_ms": 198,
  "p99_ms": 420,
  "error_rate_pct": 0.01,
  "saturation_tps": 520
}
```

**Common performance regression causes and detection signals**:

| Cause | Detection Signal | Lead Time to Detect | Frequency |
|-------|-----------------|---------------------|-----------|
| Added synchronous external call | New span in trace; p95 increase proportional to external latency | Next PR | Very common |
| Database query plan change (statistics stale) | DB slow query log; p99 spike at DB span in trace | Next deploy | Common |
| Dependency version upgrade | GC pattern change; allocation rate increase in JFR | Next build | Moderate |
| Thread pool misconfiguration | Thread dump saturation at lower-than-expected load | Load test only | Moderate |
| Logging added to hot path | CPU increase; p95 increases linearly with TPS | Load test | Common |
| N+1 query introduced | DB connection pool pressure; span count increase in trace | Load test | Very common |

**N+1 query detection** — specific to T-Mobile microservice context:
A service fetches a list of 50 accounts, then makes 50 individual DB calls for account details. At 1 user: 50 extra queries, p95 = 200ms. At 400 TPS: 20,000 extra queries/second, DB connection pool exhausted, p95 = 8000ms. N+1 is invisible in unit tests and integration tests; it only appears under load.

Detection in distributed trace: look for repeated identical spans (same DB query, called N times in a loop). Dynatrace and Datadog APM can detect N+1 automatically and alert.

## Performance Budget Governance

**Who owns what**:
- Sets the budget: Principal SRE + PE Lead + Product Manager, signed off by VP Operations
- Approves exceptions: PE Lead for single-sprint overruns; VP Operations for >2-sprint overruns
- Monitors compliance: PE team, via monthly service performance scorecard
- Funds remediation: Engineering Manager — performance debt is a line item in sprint capacity, not "if we have time"

**Budget governance cycle**:

1. **Pre-development** (sprint planning): PE Lead reviews stories that touch performance-sensitive paths. Tags stories with performance risk level (Low/Medium/High). High-risk stories get explicit performance acceptance criteria:
   ```
   Acceptance Criteria (Performance):
   - p95 latency of GET /account/{id} must not degrade beyond 245ms
     (current baseline: 198ms; 10% degradation allowance = 218ms; this story
     adds an external call estimated at 20ms → must verify via load test)
   ```

2. **Mid-sprint**: if implementation shows performance risk (code review, early profiling result), PE team reviews before the story is marked Done. Prevention is cheaper than post-release remediation.

3. **Pre-release**: performance gate (automated) + PE Lead sign-off on gate report.

4. **Post-release**: compare production p95 from Dynatrace against load test result. Alert if divergence > 15%. Divergence > 15% means either: the load test environment was not representative, or production traffic patterns differ from the test model. Both are problems that must be investigated.

## Quarterly Performance Review

This is the governance event where performance engineering gets budget, surfaces risk, and demonstrates business value. It is not a technical review — it is a leadership briefing.

**Agenda (60 minutes)**:
- Service performance scorecard: current state vs. SLO, trend (20 min)
- Top 3 performance risks: specific services, quantified risk, ask (15 min)
- Performance investment asks: tooling, staffing, or infrastructure (10 min)
- Q&A (15 min)

**Service Performance Scorecard** — format for leadership consumption:

| Service | SLO Target | Prod p95 (30-day) | Trend | Saturation TPS | Peak TPS (30-day) | Headroom | Status |
|---------|-----------|-------------------|-------|----------------|-------------------|----------|--------|
| account-api | ≤300ms | 220ms | ↓ improving | 520 TPS | 380 TPS | 37% | GREEN |
| payment-api | ≤500ms | 480ms | → flat | 410 TPS | 390 TPS | 5% | RED |
| catalog-api | ≤200ms | 145ms | ↑ degrading | 800 TPS | 200 TPS | 75% | GREEN |
| notification-svc | ≤1000ms | 850ms | ↑ degrading | 150 TPS | 120 TPS | 20% | YELLOW |

Color coding:
- GREEN: p95 < 70% of SLO target AND headroom > 30%
- YELLOW: p95 70–90% of SLO target OR headroom 10–30%
- RED: p95 > 90% of SLO target OR headroom < 10%

payment-api is RED: 5% headroom at current peak. At T-Mobile's growth rate, a 5% traffic increase triggers SLO breach. This is not a future risk — it is a present operational risk.

**Top 3 Risks format**:
```
Risk 1: payment-api — SLO breach imminent
  Current: p95 = 480ms vs. SLO = 500ms. Headroom: 5% TPS. 20% TPS headroom needed.
  Likelihood: HIGH (within 6 weeks at current growth rate)
  Impact: HIGH — payment SLO breach triggers contractual penalties + customer churn
  Ask: 2-sprint engineering engagement to optimize DB query on payment confirmation path
        + PgBouncer deployment to reduce connection saturation (identified in load test)
  Cost estimate: 2 engineers × 2 sprints + $800/month PgBouncer infrastructure
```

## Board-Level Performance Reporting

What a VP/Director presents to senior leadership — three metrics that translate performance into business language:

**Metric 1: Apdex Score** (Application Performance Index)
```
Apdex = (Satisfied + Tolerating × 0.5) / Total requests

Where (T = satisfaction threshold, e.g., 300ms):
  Satisfied:  response time ≤ T
  Tolerating: T < response time ≤ 4T  (e.g., 300ms to 1200ms)
  Frustrated: response time > 4T      (e.g., > 1200ms)

Score: 1.0 = perfect, 0.85 = good, 0.70 = acceptable, <0.50 = unacceptable
```

Board-ready sentence: "Our Apdex score for the account management journey is 0.92, meaning 92% of T-Mobile customers are receiving satisfactory response times. Our target is 0.90."

**Metric 2: Core Web Vitals** (for any web-facing property)
- Largest Contentful Paint (LCP): how quickly main content loads. Target: < 2.5s. Impact: correlates with bounce rate.
- First Input Delay (FID) / Interaction to Next Paint (INP): how quickly page responds to interaction. Target: < 100ms.
- Cumulative Layout Shift (CLS): visual stability. Target: < 0.1. Poor CLS = customers click the wrong button when layout shifts.

Google uses Core Web Vitals as a search ranking signal. Poor CWV degrades organic search placement — a performance problem that manifests as reduced customer acquisition.

**Metric 3: Business impact framing**
```
"Our payment API p95 is at 480ms vs. our 500ms SLO — 4% headroom.
At current growth (8% MoM TPS increase), we breach the SLO in 6 weeks.

Estimated business impact of SLO breach:
  - Amazon research: 100ms latency increase = 1% revenue loss
  - Our payment flow adds ~200ms at breach: estimated 2% conversion degradation
  - At $X average order value × Y transactions/month: $X.Xm annual revenue risk
  - SRE on-call cost for SLO breach incidents: estimated 3 P1s/month × 8hr avg = 72 eng-hours

Mitigation cost: 2 engineers × 2 sprints + $800/month infrastructure
ROI: breakeven in week 3 of the fix sprint."
```

This framing converts a performance metric into a capital allocation decision. A VP who can present this owns the conversation in any budget discussion.

## Cost of Performance Debt

Quantifying slow systems for budget conversations:

| Cost Category | How to Measure | T-Mobile Example |
|---------------|---------------|------------------|
| Direct conversion loss | 100ms latency increase ≈ 1% conversion drop (Amazon benchmark) | 200ms p95 increase in checkout → 2% conversion drop → $X/month |
| Support cost increase | Correlate slow performance events with call center volume spikes | 5% increase in call volume at $15/call average during degraded SLO windows |
| Infrastructure over-provisioning | Compute cost to maintain SLO with unoptimized code | 30% excess EKS node count provisioned to absorb inefficient query load |
| Engineering incident cost | P1 incidents × avg duration × loaded eng cost | 3 engineers × 8hr per P1 × $150/hr loaded = $3,600 per incident |
| Customer churn correlation | 30-day churn rate following performance events vs. baseline | Segment customers who experienced degraded sessions — compare 30-day retention |

> **VP Governance View**: Performance debt is not a technical concern — it is a financial liability. When a PE team can quantify "this service's latency is costing us $X/month in lost conversions and $Y/month in excess infrastructure," the conversation shifts from engineering expense to business investment. This is how the PE program earns budget, headcount, and a seat in the architecture review. A PE team that reports only in milliseconds and percentiles will always lose a budget conversation to a team that reports in dollars. Build both views — the engineering detail and the business translation — for every quarterly review.

## PE Tooling Portfolio

| Category | Open Source | Enterprise | T-Mobile Context |
|----------|-------------|------------|-----------------|
| APM | OpenTelemetry + Jaeger | Dynatrace, Datadog, New Relic | Dynatrace (T-Mobile standard) |
| Profiling | async-profiler, JFR | JProfiler, YourKit | async-profiler for prod; JFR for comprehensive diagnostics |
| Load testing | k6, Gatling OSS, JMeter | Gatling Enterprise, BlazeMeter | k6 + k6 operator on EKS |
| Distributed tracing | Jaeger, Zipkin | Datadog APM, Dynatrace | Integrated with Dynatrace |
| Heap analysis | Eclipse MAT, VisualVM | JProfiler | Eclipse MAT for offline heap dump analysis |
| GC log analysis | GCEasy (web), GCViewer | — | GCEasy for quick turnaround |
| Capacity modeling | USL (pdq-book, R usl package) | — | Python scripts, integrates with Dynatrace TPS metrics |
| Connection pool monitoring | HikariCP JMX/Prometheus metrics | — | Prometheus + Grafana or Dynatrace |

**async-profiler** — recommended for production profiling:
```bash
# CPU flame graph — 30 seconds, safe for production
./profiler.sh -d 30 -f /tmp/profile.html <pid>

# Allocation hotspots — find what is creating the most objects
./profiler.sh -d 30 -e alloc -f /tmp/alloc.html <pid>

# Lock contention — find where threads are blocked waiting for locks
./profiler.sh -d 30 -e lock -f /tmp/lock.html <pid>

# In EKS — exec into pod
kubectl exec -n account-api <pod-name> -- \
  ./profiler.sh -d 30 -f /tmp/profile.html 1
kubectl cp account-api/<pod-name>:/tmp/profile.html ./profile.html
```
CPU overhead: < 3% — safe for brief production use on a single pod. Do not profile all pods simultaneously.

**Java Flight Recorder (JFR)** — zero-overhead, built into JDK 11+:
```bash
# Start recording — can run for hours without meaningful overhead
jcmd <pid> JFR.start duration=60s filename=/tmp/recording.jfr settings=profile

# For EKS pod
kubectl exec -n account-api <pod> -- \
  jcmd 1 JFR.start duration=60s filename=/tmp/recording.jfr settings=profile
kubectl cp account-api/<pod>:/tmp/recording.jfr ./recording.jfr
```

JFR records: GC events (pause times, allocation rates), thread states (BLOCKED, WAITING durations), method profiling (CPU hot methods), I/O events, exceptions (all thrown, not just caught). Analyze with JDK Mission Control (JMC) — GUI-based, ships with JDK.

Best for: comprehensive production diagnostics when async-profiler data points to a region but you need the full picture. JFR + async-profiler together cover 95% of JVM performance root cause scenarios.

## SRE Manager / Director Ownership of PE

What distinguishes a Director/VP who owns performance from one who delegates it:

**Delegates it**: "The PE team handles performance. I look at the scorecard in the QBR."

**Owns it**: 
- Defines performance SLOs as part of the service agreement with product — not post-hoc
- Ensures [Environment Management] provides production-representative staging before load tests run
- Ensures [Capacity Management] incorporates PE saturation data in headroom modeling
- Reviews exception log personally — not just approval but interrogation of root cause
- Translates performance metrics to business impact for budget and executive audiences
- Sets the expectation that performance debt is remediated in the same quarter it is identified

The Director/VP role in performance governance is not to know the JVM flags — it is to ensure the program has the authority, tooling, and resourcing to enforce the gate. An underfunded PE program produces theater: load tests that run against under-provisioned staging environments, gates that always pass because the bar was set to yesterday's degraded performance, and quarterly reviews that show red metrics with no budget to fix them.

> **VP Governance View**: Performance engineering is an investment with a calculable return. The ROI is: (revenue protected by SLO compliance) + (infrastructure savings from efficient code) + (incident cost averted) − (PE program cost). Make this calculation explicit in every annual planning cycle. If the PE program cannot demonstrate this ROI, either the measurement framework is wrong or the program is not effective — and both are leadership problems, not engineering problems.
