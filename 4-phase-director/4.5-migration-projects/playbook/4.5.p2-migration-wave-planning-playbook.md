# Migration Wave Planning Playbook

## Overview

This playbook governs wave planning for a large-scale migration program. It covers the full sequence from application discovery through wave kick-off. Use it at the start of each migration wave, before any cutover work begins.

A migration wave is a bounded group of applications that migrate together within a defined time window. Waves exist because migrating everything at once is unmanageable, but migrating applications in arbitrary order creates dependency failures and coordination overhead. Proper wave planning determines which applications move together, in what order, and under what conditions.

**When to use this playbook**: Beginning of each wave, after overall program sequencing has been set at the program level. Wave planning assumes the overall migration strategy, target environment, and program governance model are already in place.

---

## Prerequisites

Before starting wave planning, the following must be true:

- Overall migration program has been formally authorized (executive sponsor confirmed, budget approved)
- Target environment exists or has a confirmed build timeline
- Migration program governance model is in place (steering committee, escalation path, change management process)
- Application inventory source of record is identified (CMDB, asset management system, or agreed substitute)
- Migration readiness scoring methodology is approved by the program steering committee
- Migration leads are assigned at the program level (individuals named, not just roles)

If any prerequisite is absent, stop wave planning and escalate to the program sponsor. Proceeding without these leads to rework.

## Phase 1: Wave Scoping (Week 1-2)

### Step 1.1: Pull Application Inventory

Pull the candidate application list from all authoritative sources and consolidate into a single working inventory:

- CMDB export: filter for applications in scope of the migration program
- Asset management system: cross-reference with CMDB for gaps
- Team submissions: open a structured submission form to each application team; use this to catch applications not represented in CMDB
- Prior wave carryovers: any applications deferred from a previous wave reenter here

For each application, collect:

| Field | Source |
|-------|--------|
| Application name and ID | CMDB |
| Owning team | CMDB / team submissions |
| Technology stack (runtime, framework, DB) | Team submission |
| Hosting model (VM, container, serverless) | CMDB |
| External dependencies (APIs, databases, queues) | Team submission |
| Criticality rating | CMDB or business owner |
| Last known deployment date | CI/CD system |

Do not proceed with applications where the technology stack or owning team is unknown. Flag them as "inventory incomplete" and assign a resolution deadline.

### Step 1.2: Validate Inventory Completeness

Schedule a 30-minute working session with each application team represented in the candidate list. The goal is factual validation, not readiness assessment.

Validate:
- Is the technology stack current? (teams often run things CMDB does not reflect)
- Are all external dependencies listed? (the CMDB is frequently wrong on this)
- Is the owning team correct? (ownership changes after reorgs are commonly stale)
- Is there a planned deployment or major change coming that conflicts with the migration window?

Document all corrections in the working inventory. Any application team that does not respond within 5 business days should be flagged to the program manager for escalation. Do not carry unvalidated applications into the wave.

### Step 1.3: Apply Migration Readiness Scoring

Score each candidate application on two axes using the agreed methodology. If the program does not have an approved scoring model, use this default:

**Complexity Score (1-5)**

| Score | Criteria |
|-------|----------|
| 1 | Single tier, no external dependencies, stateless |
| 2 | Multi-tier, 1-3 external dependencies, limited state |
| 3 | Multi-tier, 4-6 dependencies, persistent state |
| 4 | Complex integration topology, shared databases, custom protocols |
| 5 | Legacy monolith, undocumented dependencies, no current test coverage |

**Urgency Score (1-5)**

| Score | Criteria |
|-------|----------|
| 1 | No business driver, can defer indefinitely |
| 2 | Soft deadline, flexibility exists |
| 3 | Contractual or compliance deadline within 12 months |
| 4 | Vendor EOL or cost pressure within 6 months |
| 5 | Hard regulatory, security, or contract deadline within 3 months |

Compute a Wave Priority Score: `Priority = Urgency + (6 - Complexity)`

This formula prioritizes urgent, simple applications over urgent, complex ones. High priority scores belong in the current wave; low scores defer to later waves.

Document the scores and rationale for each application. Scores are inputs to the wave boundary decision, not automatic assignments.

### Step 1.4: Identify the Wave Boundary

Using the priority scores, dependency map (preliminary, finalized in Phase 2), and available migration capacity, determine which applications belong in this wave.

Wave boundary principles:
- Do not exceed the migration team's sustainable capacity. Overloaded waves fail not because of technical problems but because teams cannot execute in parallel at scale.
- Applications with a Complexity Score of 5 should not lead a wave unless urgency forces it. Use them to test the process only after the process is proven.
- Keep wave size manageable: 10-20 applications is a reasonable upper bound for most organizations. Larger waves lose coherence.
- Applications that share a database, a message queue, or a critical API should be in the same wave unless there is an explicit compatibility contract.

Document the wave boundary decision with rationale. Any application excluded from this wave despite high urgency requires a written risk acceptance from the executive sponsor.

## Phase 2: Dependency Mapping (Week 2-4)

### Step 2.1: Document Application Dependencies

For each application in the wave, create a dependency record:

**Upstream dependencies** (what does this application call?):
- APIs and services
- Databases (shared or dedicated)
- Message queues and event streams
- File systems and storage
- Third-party and vendor services

**Downstream dependencies** (who calls this application?):
- Applications that consume this application's APIs
- Batch jobs that read from this application's database
- Monitoring and observability pipelines
- Any external consumers outside the migration program scope

Do not rely on documentation alone. Pull network traffic data, API gateway logs, or service mesh telemetry to validate the actual dependency graph against what teams report. Self-reported dependencies are routinely incomplete.

### Step 2.2: Identify Cross-Wave Dependencies

A cross-wave dependency exists when:
- Application A is in this wave, and
- Application A calls Application B, and
- Application B is NOT in this wave

Flag every cross-wave dependency. These are the highest-risk items in wave planning because they create a coupling between this wave and a future wave's schedule.

Create a cross-wave dependency register:

| App in Wave | Depends On | That App's Wave | Dependency Type | Risk |
|-------------|-----------|-----------------|-----------------|------|
| [App A] | [App B] | Wave 3 | Synchronous API | High |

### Step 2.3: Resolve Cross-Wave Dependencies

For each cross-wave dependency, choose one of three resolutions:

**Option 1: Pull the dependency into this wave.**
If the dependent application is small, low-risk, and the team is available, moving it into this wave eliminates the cross-wave coupling. Preferred when practical.

**Option 2: Push the dependent application to a later wave.**
If the dependency is too complex or the team unavailable, defer the dependent application to the wave in which its dependency will be resolved. Document the deferral and the business impact.

**Option 3: Establish a compatibility contract.**
If neither pull nor push is viable, agree with both teams on a compatibility contract: API versioning, backward-compatible schema changes, or a documented stub/proxy during the transition period. A compatibility contract must be written, reviewed by both teams, and signed off by the program manager. Verbal agreements are not sufficient.

No cross-wave dependency may remain unresolved when the wave enters Phase 5. An unresolved dependency is a go/no-go blocker.

### Step 2.4: Create the Dependency Graph

Produce a visual dependency map of all wave applications and their relationships. The graph should show:

- Applications in this wave (as nodes)
- Dependencies between wave applications (as directed edges, showing call direction)
- Cross-wave dependencies (as dashed edges to external nodes)
- Shared infrastructure (databases, queues) as infrastructure nodes

The dependency graph is a working document, not a final deliverable. It should be updated as validation surfaces new dependencies. Share it with all migration leads at the start of Phase 3.

## Phase 3: Sequence Design (Week 3-5)

### Step 3.1: Sequence the Migration Order

Using the dependency graph, sequence application migrations from leaf nodes to root nodes: migrate dependencies before dependents.

The principle is identical to how a build system resolves compile order. An application should not migrate until all applications it depends on have successfully migrated (or a compatibility contract is in place for those that have not).

Produce a sequenced migration schedule:

| Migration Slot | Application | Depends On (already migrated?) | Assigned Window |
|---------------|-------------|-------------------------------|-----------------|
| 1 | [App A] | None | Week 6, Tuesday |
| 2 | [App B] | App A | Week 6, Thursday |

### Step 3.2: Identify the Critical Path

The critical path is the sequence of dependent applications where a delay in any one delays the entire wave.

Mark the critical path explicitly on the dependency graph and in the migration schedule. Migration leads on the critical path receive priority support from the program team. Any delay on the critical path must be escalated to the program manager within 24 hours.

### Step 3.3: Select Canary Applications

Select 1-2 applications to be the wave's canaries: the first applications migrated, chosen to test and validate the migration process before applying it at scale.

Canary selection criteria:
- Low complexity (Complexity Score 1-2)
- Not on the critical path
- Has a technically engaged and cooperative owning team
- Covers representative technology stack (so findings generalize to other wave applications)
- Failure can be absorbed without business impact

Document why each canary was selected. Canary migration results feed directly into process adjustments before the rest of the wave proceeds.

### Step 3.4: Assign Migration Windows

Assign a specific day and time window to each application's planned cutover. Migration window criteria:

- Avoid peak business hours unless the application has zero user-facing traffic
- Avoid Friday evenings unless the application has Sunday operational coverage
- Stagger windows to avoid overtaxing the SRE and infrastructure teams
- Align database migration windows with database team capacity
- Build buffer time between sequential (dependent) migrations: do not schedule App B the same day as App A if B depends on A

Confirm each assigned window with the application team and the business stakeholder for that application. Unconfirmed windows are placeholders, not commitments.

## Phase 4: Resource and Rollback Planning (Week 4-6)

### Step 4.1: Assign Migration Leads

For each application in the wave, confirm the named migration lead. This is a specific person, not a team. Requirements:

- The migration lead understands the application's architecture
- The migration lead has authority to make cutover decisions during the migration window
- The migration lead has a confirmed backup for the assigned window
- The migration lead has read the application's migration runbook and signed off on it

Document the assignment in the wave plan. Role assignment without runbook sign-off is incomplete.

### Step 4.2: Confirm Rollback Procedures

For each application, the rollback procedure must be documented before the wave begins. The rollback document must include:

- Trigger criteria: what conditions cause rollback (be specific — error rate thresholds, not "something is wrong")
- Named rollback decision authority: who calls the rollback (one person, not a committee)
- Step-by-step rollback procedure: no ambiguity, no "contact the team"
- Time estimate to complete rollback: must be within the maintenance window
- Post-rollback validation steps: how to confirm the old environment is fully restored

Review each rollback procedure in a tabletop exercise before the wave starts. A rollback procedure that has never been walked through is not a rollback procedure.

### Step 4.3: Reserve Shared Resources

Identify all shared infrastructure resources that multiple applications in the wave will need during their migration windows:

- Network capacity (VPN, interconnects, load balancers)
- Database capacity (migration windows on shared databases)
- Cluster compute (target environment capacity for parallel migrations)
- Tooling (migration automation scripts, access credentials)

Reserve these resources for the wave duration through the appropriate capacity and change management process. Do not assume shared resources will be available on demand during the wave.

### Step 4.4: Pre-Stage Rollback Infrastructure

Rollback infrastructure must be ready before the first migration starts, not prepared reactively when a rollback is needed.

Pre-staging requirements:
- Old environment remains live and accessible during each application's cutover window
- Load balancer configurations are pre-staged to shift traffic back with a single command
- Database snapshots or replication lag is within acceptable bounds for point-in-time recovery
- Rollback scripts are tested in a pre-production equivalent environment before the wave starts

If any rollback infrastructure cannot be pre-staged, document why and obtain written risk acceptance from the Director before proceeding.

## Phase 5: Wave Gate Review and Kick-off

### Step 5.1: Wave Entry Gate Review

Conduct a formal gate review before the first migration in the wave starts. The Director chairs this review. All migration leads must attend.

Review the Wave Entry Gate Checklist (below). Any unchecked item is a blocker unless accompanied by a written risk acceptance signed by the Director.

### Step 5.2: Stakeholder Briefing

Before the first migration, brief all business stakeholders whose applications are in the wave:

- Confirm the maintenance window dates and times
- Confirm the expected impact (downtime duration, if any; degraded performance window, if applicable)
- Provide the escalation contact for business impact during the migration window
- Confirm the post-migration communication plan (who tells the business when the migration is complete and validated)

Do not assume stakeholders read wave communications. Confirm receipt. For critical applications, direct phone or video confirmation is appropriate.

### Step 5.3: Wave Kick-off Communication

Send a formal wave kick-off communication to all migration leads, SRE on-call, and the migration program team. Include:

- Wave schedule (all applications, dates, times, leads)
- Escalation path (program manager, Director, and on-call SRE contacts)
- Communication channels (bridge line, Slack channel, or equivalent)
- Rollback contact for each application
- Link to all migration runbooks

### Step 5.4: Confirm Monitoring and Alerting

Before the first migration starts, confirm:

- Target environment monitoring is active (not just provisioned — actively alerting)
- Dashboards for the new environment are accessible to all migration leads
- Alert routing is configured to notify the on-call SRE for the new environment
- Synthetic monitoring or health checks are running against the target environment's endpoints

Do not start the first migration until monitoring is confirmed operational. A migration into an unmonitored environment cannot be safely validated.

## Wave Entry Gate Checklist

The Director reviews and signs off on this checklist before the first migration begins. Any unchecked item requires written risk acceptance before proceeding.

### Inventory and Scoping
- [ ] All wave applications inventoried and documented in the wave plan
- [ ] Inventory validated with each application team (no unvalidated entries)
- [ ] Migration readiness scores documented for all applications

### Dependency Mapping
- [ ] All upstream and downstream dependencies documented for each application
- [ ] Cross-wave dependencies identified and recorded in the dependency register
- [ ] All cross-wave dependencies resolved (pull, push, or compatibility contract in place)
- [ ] Dependency graph produced and reviewed by migration leads

### Sequencing and Scheduling
- [ ] Migration order sequenced per dependency-first principle
- [ ] Critical path identified and documented
- [ ] Canary applications selected and rationale documented
- [ ] Migration windows assigned and confirmed with application teams and business stakeholders

### Resource and Rollback Readiness
- [ ] Named migration lead confirmed for each application
- [ ] Rollback procedure documented and reviewed for each application (tabletop completed)
- [ ] Shared infrastructure resources reserved for the wave duration
- [ ] Rollback infrastructure pre-staged and tested

### Target Environment Readiness
- [ ] Target environment fully provisioned (not in progress)
- [ ] Target environment validated: smoke tests and load tests complete
- [ ] Monitoring and alerting confirmed operational on target environment
- [ ] Alert routing confirmed to correct on-call team

### Communication and Governance
- [ ] Stakeholder briefing completed for all business owners with applications in the wave
- [ ] Wave kick-off communication sent to all migration leads and program team
- [ ] Escalation path documented and contacts confirmed as available
- [ ] Incident bridge or coordination channel confirmed active

Any unchecked item and the associated written risk acceptance must be attached to this document before sign-off.
