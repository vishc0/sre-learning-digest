# Chapter 02 — Migration Assessment and Discovery

## Application Portfolio Assessment

Before you can sequence a migration, you need an accurate inventory. For most enterprises, that inventory either does not exist or is wrong in ways that matter.

The CMDB is the usual starting point and the usual disappointment. CMDBs are maintained by process obligation, not operational urgency. Teams add applications when they are deployed and remove them when they are decommissioned — in theory. In practice, shadow applications get deployed without CMDB entries, decommissioned applications linger in the CMDB for years after shutdown, and the dependency relationships between applications are either absent or outdated. If you have 300 applications in your portfolio and only 180 of them are in the CMDB with accurate dependency data, your migration wave plan is wrong before it starts.

Discovery must be treated as a Phase 0 workstream with dedicated resources, a defined timeline, and a discrete deliverable (a validated application inventory with dependency maps). It is not a pre-work checklist item that teams complete alongside their normal sprint work. Teams will not prioritize CMDB cleanup when they have feature backlogs. Phase 0 requires explicit resourcing and a program manager whose only job during that phase is driving inventory completion.

The deliverable at the end of Phase 0 is not a spreadsheet. It is a validated inventory where "validated" means: an engineer who owns the application has confirmed the entry is accurate, the dependency list has been cross-checked against network flow data or service mesh telemetry, and the data has a timestamp that reflects when it was last verified. Stale data is worse than no data — it creates false confidence in the wave plan.

## Dependency Mapping — The Iceberg Problem

Every application has two classes of dependencies: declared and undiscovered.

Declared dependencies are what teams put in architecture diagrams, CMDB records, and onboarding documentation. These are the 30% that float above the waterline. Teams know about them. They are relatively easy to account for in wave planning.

Undiscovered dependencies are the 70% below the surface. They accumulate over years of operational evolution, team turnover, and the pragmatic shortcuts that engineers make when they need something to work right now and will document it later. Common forms:

- **Hardcoded IP addresses or hostnames** in configuration files, environment variables, or application code. The application is not using service discovery; it is pointing directly at infrastructure that will not exist after migration.
- **Undocumented synchronous service calls** between applications that share a network segment but have no formal integration contract. These appear in network flow logs but not in architecture diagrams.
- **Shared database schemas** where two or more applications read from or write to the same tables, often with no interface layer between them. Migrating one application without the others breaks the shared schema assumption.
- **Shared message queues** — in RabbitMQ environments, this manifests as multiple teams consuming from the same exchange or queue without a formal acknowledgment that the dependency exists. Migrating the producing application changes the message format, routing key, or exchange configuration, and breaks consumers nobody knew were downstream.
- **Batch jobs and scheduled tasks** that run on infrastructure adjacent to the application rather than in the application itself. Cron jobs on VMs, ETL jobs that pull from application databases, data exports to shared file shares — these are rarely in the CMDB and rarely appear in application architecture diagrams.

The iceberg framing matters for stakeholder communication. When a sponsor asks why discovery is taking longer than planned, the iceberg is your explanation: we scoped discovery based on declared dependencies. We are finding the undiscovered layer, which is three times larger. This is expected, not exceptional.

The tooling for surfacing undiscovered dependencies:
- **Network flow data** (Splunk, VPC flow logs, Dynatrace topology maps) — shows actual traffic patterns between services, not what teams think the patterns are
- **Service mesh telemetry** — if you are already running a service mesh, dependency maps are available programmatically
- **Database query logs** — show which application identities are connecting to which schemas
- **Message broker metrics** — RabbitMQ management API and Splunk metrics show active consumers by queue, surfacing consumers that teams may not have documented

Do not start wave planning until undiscovered dependency mapping is at least 80% complete. The cost of re-sequencing a wave because a dependency was found after wave assignment is higher than the cost of spending two more weeks on discovery.

## Migration Readiness Scoring

Not every application is equally ready to migrate, and not every application has the same urgency for migration. Scoring both dimensions gives you the raw material for wave sequencing decisions.

### Complexity Score (1–10, Low / Medium / High)

| Factor | What Raises Complexity |
|--------|----------------------|
| Dependency count | More than 10 declared dependencies, or any undiscovered dependencies found during Phase 0 |
| Data volume | Stateful application with large data sets; schema migration required |
| SLA tier | Tier 1 (highest availability requirement) — less tolerance for cutover risk |
| Team familiarity with target | Team has no prior EKS/target platform experience |
| Integration surface area | External integrations (third-party APIs, partner data feeds, regulatory systems) |
| Code age and documentation | Application is more than 5 years old with limited documentation and original team attrition |

Low (1–3): Simple stateless service, few dependencies, well-documented, team experienced with target platform.
Medium (4–6): Moderate state, 5–15 dependencies, some external integrations, team is learning target platform.
High (7–10): Stateful, many dependencies (some undiscovered), external integrations, Tier 1 SLA, low team familiarity.

### Urgency Score (1–10, Low / Medium / High)

| Factor | What Raises Urgency |
|--------|-------------------|
| Platform cost | Application running on infrastructure with high ongoing cost that migration eliminates |
| Security posture | Application running on EOL OS, unsupported runtime, or unpatched infrastructure |
| Vendor EOL | Platform or dependency reaching end-of-support on a defined date |
| Strategic alignment | Application is on a platform the organization is contractually committed to exiting |
| Regulatory pressure | Compliance requirement tied to platform modernization |

Low (1–3): Application is stable, costs are manageable, no immediate vendor or security pressure.
Medium (4–6): Moderate platform cost, approaching EOL timelines, some regulatory relevance.
High (7–10): Significant ongoing cost, active security risk, hard vendor EOL date, or regulatory deadline.

## Wave Planning Principles

Wave sequencing is a Director-level decision because it is fundamentally a risk management decision, not a technical decision. Engineers can provide the complexity scores. Business owners can provide the urgency scores. Only the Director can make the call about which risks the program is willing to absorb in which sequence.

**Wave 1: Low complexity, high urgency — build the factory.** The first wave proves the migration process, not the applications. You are learning how to run cutover windows, how to validate parity in your specific environment, how to communicate status to stakeholders, and where the gaps are in your runbooks. Choose applications where a failure is recoverable and the team will learn from it. Do not choose showpiece applications for Wave 1. Choose boring ones.

**Do not put high-traffic Tier 1 systems in Wave 1.** The blast radius of a failed Wave 1 cutover should be bounded. A Tier 1 system failure during Wave 1 does not just create an incident — it creates a confidence crisis that can stall or kill the program. Your payment processing platform goes into a later wave when your cutover muscle memory is built.

**Applications with shared dependencies belong in the same wave or adjacent waves.** If Application A and Application B share a database schema, they cannot be in Wave 1 and Wave 3 respectively without an interface layer being built or the schema being split first. Wave sequencing that ignores shared dependencies creates mid-program re-sequencing crises that delay the entire portfolio.

**The "do not touch last" rule: highest-risk, lowest-urgency applications go in the final wave.** These are the applications where the complexity is high, the urgency is low, and the blast radius of a problem is significant. If the program runs out of funding, loses executive support, or gets deprioritized by a business event, these applications remain safely on the source platform. You want the applications that are most dangerous to migrate to be last, so that program disruption leaves them untouched.

**Reserve 20% of wave capacity for discovery surprises.** Every wave will surface dependencies and complexity that Phase 0 did not find. This is not a Phase 0 failure; it is the nature of legacy application portfolios. A wave plan that allocates 100% of capacity to planned work has no buffer for the reality that dependency discovery is continuous, not front-loaded.

## The Dependency-First Sequencing Principle

Before any application is assigned to a wave, map its upstream and downstream dependencies and confirm that the wave sequence respects those dependency chains.

An application with 15 active consumers cannot be migrated until all 15 consumers have validated compatibility with the new endpoint. If you migrate the producing application in Wave 2 and three of its consumers are not scheduled until Wave 4, you have either broken Wave 4 applications or you are running the old endpoint and the new endpoint simultaneously until Wave 4 completes. Both options have cost and risk implications that must be deliberate choices, not surprises.

In messaging infrastructure, this is acute. A RabbitMQ exchange migration that changes routing key patterns or exchange topology requires all consumers to be updated before or concurrently with the producer cutover. A queue consumer that migrates in Wave 2 while the producer migrates in Wave 3 may be consuming from an exchange that no longer exists in its expected form. The dependency chain runs both directions: producers and consumers must be sequenced as a unit, not independently.

The dependency-first sequencing principle causes more wave re-sequencing than any other factor in migration programs. Surface it early. Run dependency chain analysis against every proposed wave assignment before the wave plan is finalized.

## Director's Tool: The Migration Portfolio Heat Map

The heat map is the Director's primary operational artifact during program execution. It is what you review at steering committee. It is what you use to make wave sequencing decisions. It is what you show sponsors when they ask for program status.

The heat map is not a Gantt chart. Gantt charts show time. The heat map shows portfolio state: where every application is in the migration lifecycle, what its risk profile is, and who owns it.

At a minimum, the heat map contains:

| Application | Complexity | Urgency | Wave | Owner | Status | Blocker |
|-------------|-----------|---------|------|-------|--------|---------|
| ... | L/M/H | L/M/H | 1–N | Team | Not started / Discovery / Planning / Cutover / Validation / Complete | [if blocked] |

At steering committee, the Director's questions are:
- Which applications changed status since last week, and why?
- Which applications are blocked, and at what level does the blocker need to be resolved?
- Are any wave boundaries at risk based on current velocity?
- Are there dependency chain violations in the current wave plan that need re-sequencing?

The heat map keeps those questions answerable in under 10 minutes. If your program manager cannot produce a current heat map for every steering committee, the program management function is under-resourced.

## Director's Lens

Discovery is where most migration programs under-invest. The political pressure is always to start migrating quickly — to show progress, to demonstrate that the investment is producing results. Discovery work looks like overhead. It produces a spreadsheet, not a migrated application.

The case for investing 15–20% of total program cost in discovery and dependency mapping is straightforward: every hidden dependency found during discovery costs roughly $5K–$20K to remediate in wave planning. The same dependency found during a production cutover at 2am costs multiples of that in engineering time, incident response, potential rollback, and schedule delay. Discovery is not overhead. It is risk pre-payment.

When an executive pushes back on the discovery budget, the answer is: "We can reduce the discovery investment. In exchange, I need you to accept that our wave sequencing will have higher re-sequencing risk and our cutover failure rate will be higher than if we complete discovery. I can document that trade-off for the record. Do you want me to proceed on that basis?" Most sponsors will choose to fund the discovery when the risk is framed clearly.

The resistance to funding discovery is a signal. It often means the sponsor is under schedule pressure that has not been surfaced to the program. Find out what is driving the urgency before you accept a compressed discovery timeline.
