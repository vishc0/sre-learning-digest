# Chapter 01 — Foundations and Philosophy of Migration

## What Makes Migration Different from a Feature Project

A feature project adds capability. A migration transfers existing capability to a new context while maintaining fidelity.

That distinction is not semantic. It determines how you staff the work, how you fund it, how you set success criteria, and how you communicate with stakeholders.

When you build a feature, partial delivery has value. You ship what's done, users adopt it incrementally, and the old way of doing things coexists with the new. There is no "migration complete" gate because there is no obligation to reach zero of the old thing.

When you migrate, you are obligated to reach zero. The old environment has to be decommissioned — it's the entire economic justification for the program. A PCF cluster that costs $2M per year and a parallel EKS cluster that costs $1.8M per year do not save money. They cost $3.8M. The savings materialize only when the source environment is shut down. This is why migrations are governed differently from feature programs. The finish line is not "new thing works." The finish line is "old thing is gone, and new thing has proven it can carry the full load."

The resourcing implication: migration teams need dedicated capacity. A team that is 60% on BAU and 40% on migration will deliver neither well. Teams split across operational support and migration work will consistently deprioritize migration tasks when incidents happen — and incidents always happen. If you are governing a migration program and teams are not ring-fenced for it, you have a staffing problem, not a delivery problem. Escalate it as a staffing problem.

The funding implication: migrations typically require a capital investment before delivering any return. The ROI realization is back-loaded. Sponsors who approved the program based on a 24-month savings projection will get nervous at month 12 when costs are up (dual environments running), velocity appears slow (teams learning new platforms), and the old system is still running. Director-level communication during this phase is what separates programs that complete from programs that get cancelled.

## The 6Rs + 7th R

The "6Rs" is an industry-standard migration decision framework. Know it well enough to apply it in steering committee and to explain why a specific "R" was chosen for a specific application.

**Retire** — Decommission, no replacement needed. The application has no active users, the business capability is no longer needed, or the function has already been absorbed by another system. Retire aggressively. Every application that doesn't migrate is migration cost avoided. Use when: usage data shows low or zero active consumption and no strategic dependency exists.

**Retain** — Leave in place. The application cannot be economically migrated right now — too complex, too risky, or too dependent on a decommission decision that hasn't been made yet. Retain is honest technical debt parking. It is not failure; it is sequencing discipline. Use when: migration cost exceeds platform cost savings, or the application is high-risk and the program cannot absorb the blast radius right now.

**Rehost (Lift-and-Shift)** — Move as-is to new infrastructure with minimal or no code change. Same artifact, new compute substrate. You gain platform consolidation; you gain nothing in terms of architectural improvement. Use when: the goal is purely infrastructure cost reduction and the application is stable, well-understood, and not a candidate for modernization investment.

**Replatform** — Move with targeted, bounded optimizations that the new platform enables, without restructuring the application. PCF to EKS is the canonical example: the application remains a containerized Java service; what changes is the orchestration layer, ingress pattern, secrets management, and observability integration. Code changes are minimal — Dockerfile adjustments, environment variable standardization, health endpoint conformance. Use when: you want meaningful platform capability gain without the risk and cost of refactoring.

**Refactor** — Restructure the code significantly to exploit the target platform's capabilities. This might mean decomposing a large Spring Boot monolith into smaller services, or rewriting synchronous REST calls as event-driven patterns to leverage a new Kafka backbone. Refactor carries significant development cost and risk. Use when: the application's current architecture is actively preventing the business capability you need, and the platform migration is the catalyst for the architectural work you would need to do anyway.

**Re-architect** — Redesign from scratch for the target environment. Monolith to microservices is the common pattern. This is not a migration; it is a rebuild with data migration as a sub-problem. Use when: the source application's architecture is so mismatched to the target environment that replatform or refactor would cost more than a clean build. Rare. Treat with significant skepticism when engineers propose it — "re-architect" is often an engineering preference masquerading as a technical necessity.

**Replace (7th R)** — Swap the application with a SaaS or vendor-managed equivalent. On-prem ticketing system replaced by ServiceNow. On-prem monitoring replaced by Datadog. Internal identity provider replaced by Okta. Replace eliminates the migration execution problem entirely, but introduces vendor dependency, data portability risk, and integration complexity. Use when: the application's function is commodity enough that a vendor product will meet requirements and the total cost of ownership (including integration) beats maintain-and-migrate.

## Why Migrations Fail — The Seven Root Causes

These are not theoretical. They account for the majority of migration program failures and cost overruns.

**1. Hidden dependencies discovered late.** The CMDB said the application had 12 dependencies. Post-migration discovery found 31. The 19 hidden dependencies included hardcoded IP addresses, undocumented batch jobs, shared database schemas with three other applications, and a legacy message queue that two teams had forgotten they were consuming. Discovery work that was treated as pre-work instead of a funded Phase 0 workstream is the direct cause. The fix is structural, not cultural.

**2. No rollback plan — or rollback plan never tested.** Every migration cutover should have a tested, time-boxed rollback procedure. "Tested" means the rollback was executed in a non-production environment and the team knows exactly how long it takes and what state the system is in afterward. An untested rollback is a theory. Theories fail at 2am during cutover windows.

**3. Underestimated data volume and data quality issues.** Teams consistently underestimate how long data migration takes, and consistently overestimate the quality of the data they are migrating. Schema drift between source and target, referential integrity violations, encoding inconsistencies, and data that was valid in the source system but invalid in the target's stricter schema — all of these are discovered during migration execution, not during planning. Budget 2x your initial data migration time estimate until you have run a dry migration with real production data.

**4. Cutover windows too short for actual complexity.** A cutover window is sized based on what the team thinks the migration will take. It rarely accounts for: unexpected dependency failures, rollback time, validation time, stakeholder notification time, and the cognitive overhead of executing a high-stakes procedure under time pressure. Cutover windows should be sized based on dry-run execution time, not estimation.

**5. Stakeholder fatigue — the migration fog.** In an 18-month migration program, the organizational memory of why the program started begins to erode around month 9. Sponsors rotate. Business priorities shift. New leadership asks why resources are locked into a platform migration when there are feature requests in the backlog. This is the migration fog. It is not a failure of the program; it is a failure of stakeholder communication cadence. Directors who survive the migration fog are the ones who built the communication plan at program start and maintained it with consistent, honest upward reporting throughout.

**6. Teams running migration while maintaining BAU — capacity not protected.** This is a governance failure, not a team failure. If you approved a migration program but did not approve the headcount or the backlog freeze necessary to protect migration capacity, you did not approve a migration program. You approved a wish. Teams will always prioritize the production incident over the migration task. Directors must hold the line on capacity protection or adjust the program timeline to match actual available capacity.

**7. Success criteria defined too loosely.** "Migration complete" is not a success criterion. "All application traffic shifted to EKS, source PCF environment decommissioned, 30-day SLO baseline validated against pre-migration baseline in Splunk, and all incident runbooks updated to reflect new environment" is a success criterion. Loose success criteria guarantee disputes at the finish line about whether you're actually finished.

## Migration as Project vs. Migration as Program

A single-application migration is a project. It has a defined scope, a team, a timeline, and a finish line. A tech lead or senior engineer can own it with appropriate oversight.

A platform migration affecting 100, 200, or 400 applications is a program. It has multiple workstreams running in parallel across multiple product teams, a governance structure with a steering committee, a portfolio-level risk register, wave sequencing decisions that create dependencies between workstreams, and a multi-year timeline with phase gates.

Directors govern programs. They should not be managing individual migration projects within the program. If a Director is spending steering committee time on ticket-level status, the program management structure is broken. The Director's role is:

- Holding the migration philosophy (which R strategy applies and why)
- Making wave sequencing and deprioritization calls
- Managing upward: sponsor expectations, budget adjustments, timeline negotiations
- Removing organizational blockers that program managers cannot escalate past
- Owning the decommission decision — the hardest part of any migration program because it requires organizational confidence that the new environment is fully carrying the load

## The Migration Paradox

The moment you start a migration program, operational complexity increases. This is not a warning. It is a physics statement.

Dual environments mean dual incident response surfaces. Teams learning new platforms make configuration mistakes. Observability coverage is incomplete during transition — you have partial Splunk coverage of the new environment while the old environment's dashboards are still the operational source of truth. Cutover-adjacent periods generate incident spikes as traffic shifts expose edge cases that testing didn't catch.

The incident rate goes up. The team's cognitive load goes up. The cost goes up before it comes down.

If you have not communicated this to your sponsors before the program starts, you will be explaining an "unexpected" incident spike to an already-nervous leadership team at exactly the wrong moment — when confidence in the program is lowest and the temptation to pause or cancel is highest.

Communicate the migration paradox at program kickoff. Name it. Quantify it where you can ("we expect a 20-30% increase in P2/P3 incident frequency during wave 2 and wave 3 cutover periods"). Put it in the steering committee deck. Make it boring before it happens, so when it happens, it looks like good forecasting rather than damage control.

## Director's Lens

At your level, the migration decision — which R to use, when to start, when to pause, when to stop — is more consequential than the migration execution.

Technical staff execute migrations. Directors set the migration philosophy and hold the line on it when business pressure pushes for shortcuts. The most common shortcut is compressing the cutover timeline. The second most common is skipping the discovery phase because "we already know our applications." The third is accepting a success criterion of "it works in staging" without requiring a production validation period.

Your job is to know why each of these shortcuts increases program risk, to explain it clearly to non-technical sponsors, and to make the call when a shortcut is acceptable versus when it will cost more than the time it saves.

That judgment — grounded in operational experience and communicated with executive clarity — is the core competency this framework is designed to build.
