# Chapter 04: Platform and Technology Migration

**Audience**: Director-level  
**Framing**: Platform migration is the most operationally expensive category of migration — not because any single application is hard to move, but because every application has to move. The leverage point for a Director is not the per-application migration; it is the standardization decisions made before the first application migrates that determine whether the program costs 1x or 3x.

---

## Platform Migration Defined

Platform migration is the movement of applications from one runtime substrate to another. The destination may or may not be cloud. The defining characteristic is that the way applications are built, deployed, scheduled, and operated fundamentally changes.

Examples:
- PCF → Kubernetes/EKS
- On-premises servers → cloud VMs
- Bare metal → VMware
- VMware → bare metal Kubernetes
- On-premises RabbitMQ cluster → cloud-managed Kafka

The distinguishing question from cloud migration: **What does every team have to change about how they build and operate?**

Cloud migration can be transparent to developers — some rehost migrations require no code changes at all. Platform migrations almost never are. When the substrate changes, the abstractions that platform provided disappear, and teams must re-implement them. The Director question at program start: "How many teams are affected, and what does each team have to do differently?" If the answer is "most teams, and quite a lot" — this is a high-cost platform migration regardless of how simple individual applications appear.

## PCF to Kubernetes: The Canonical Enterprise Platform Migration

This is the pattern T-Mobile executed. At the Director level, the story is not about Dockerfiles and Helm charts. It is about what PCF was doing for teams that they did not know PCF was doing — and what happens when that implicit operational infrastructure becomes explicit work.

### What PCF Abstracted

PCF was a high-abstraction PaaS. Its value was that developers pushed code and the platform handled everything else. From the platform's perspective, "everything else" was substantial:

| PCF Capability | What It Did | Kubernetes Equivalent |
|----------------|-------------|----------------------|
| Buildpacks | Detected language, built container image, managed base OS | Dockerfile (authored by team) + base image selection |
| CF Marketplace | Service binding — databases, message brokers, credentials injected automatically | Kubernetes Secrets, ConfigMaps, External Secrets Operator, or Vault |
| GoRouter | HTTP routing, TLS termination, route registration | Ingress controller (NGINX, AWS ALB, Istio) — selected, configured, maintained by platform team |
| CF Health Manager | Detected crashed apps, restarted them automatically | Liveness probes and readiness probes — defined per deployment |
| CF Autoscaler | Scaled app instances based on CPU, memory, or custom metrics | HPA with metrics server — configured per deployment |
| CF Logging | Log aggregation via Loggregator to Splunk/ELK | Log shipping agents (Fluent Bit, Fluentd) — deployed and configured per cluster |

None of this was visible to developers on PCF. They ran `cf push`. It worked.

On Kubernetes, every one of these capabilities must be explicitly provisioned, configured, and maintained. For each application team, this means a material body of new work. For the platform engineering team, it means building and maintaining the cluster-level infrastructure that makes this work manageable.

### The Migration Cost Underestimation Pattern

Migration programs budget for "containerizing applications." This is the Dockerfile work — taking an application and defining its container image. Teams can reasonably estimate this. What they consistently fail to budget:

- Kubernetes manifest authoring (Deployment, Service, Ingress, HPA, PodDisruptionBudget)
- Helm chart or Kustomize overlay design — and the decision about which approach to standardize on
- RBAC configuration per namespace
- Network policy definition
- Secret management migration — moving from CF service binding to whatever secret management model the platform team has standardized
- CI/CD pipeline rebuilds — PCF pipelines used `cf push`; Kubernetes pipelines build images, push to registry, apply manifests
- Monitoring and logging agent changes — new agents, new configurations, new Splunk indexes or dashboards
- Liveness and readiness probe design — not a copy-paste exercise; incorrect probes cause production incidents

The gap between "containerize the app" and "run the app on Kubernetes with full operational capability" is where platform migration programs consistently underestimate cost by 40-60%.

### What a Director Governs: Standardization Before Migration

The most consequential Director decision in a PCF → Kubernetes migration is made before the first application migrates: **what is the standard target pattern?**

If 50 teams each design their own Kubernetes manifests, their own Helm charts, their own secret management approach, their own ingress configuration — you get 50 different operational models. Every incident becomes harder. Every on-call rotation covers wider variance. Every new engineer has a longer onboarding. The cumulative operational cost of that variance is enormous.

The Platform Engineering team must define the standard and enforce it before teams begin migrating. This means:

- A standard Helm chart template (or Kustomize base) that application teams customize within guardrails
- A mandated secret management approach — one tool, one workflow
- A standard ingress pattern — one ingress controller, one TLS model, one cert management approach
- Standard resource request and limit profiles — not left to each team to set arbitrarily
- Standard observability configuration — log format, Splunk index structure, required labels

Teams should be able to migrate by filling in a known template, not by making architecture decisions independently. The migration factory model — covered in the final section — operationalizes this.

## Messaging Platform Migration: RabbitMQ to Kafka (or Reverse)

This is operationally real at T-Mobile scale. If you have run RabbitMQ clusters in production, you understand what the operator's experience looks like. Platform migration here means moving that operational investment — the clusters, the queue topology, the producer and consumer fleet — to a different messaging paradigm.

### Why the Migration Happens

RabbitMQ and Kafka are not equivalent tools. Migration happens when workload characteristics change in ways that exceed what the current platform handles well.

| Characteristic | RabbitMQ Strength | Kafka Strength |
|----------------|-------------------|----------------|
| Message delivery model | Broker pushes to consumer | Consumer pulls from log |
| Ordering | Per-queue FIFO with strong guarantees | Per-partition ordering |
| Throughput | Moderate (hundreds of thousands/sec with tuning) | High (millions/sec, horizontal scale) |
| Message retention | TTL-based, deleted after ack | Log retention — configurable, replayable |
| Dead letter handling | Built-in DLX/DLQ model | Manual — consumer must handle and route |
| Operational model | Queue-centric, broker-managed | Partition-centric, consumer-managed |

Migration from RabbitMQ to Kafka happens when: the organization moves from request/response or task queue patterns to event streaming; volume outgrows what RabbitMQ can sustain; replay capability becomes a product requirement (audit, event sourcing, stream processing).

Migration from Kafka to RabbitMQ happens less frequently, but occurs when: a team adopted Kafka for simple task queuing and is paying the operational complexity premium without extracting the streaming benefit; strict per-message TTL and DLQ behavior is required without custom consumer logic.

### The Dual-Write Migration Pattern

Message broker migrations cannot use a hard cutover. During the transition window, producers must write to both the old and new broker simultaneously. Consumers migrate from old to new incrementally. This is the dual-write pattern.

The sequence:

1. Stand up the new broker in production. Validate it is operational with synthetic load.
2. Modify producers to write to both brokers — same message, both destinations. This is a code change and deployment for every producer.
3. Migrate consumers from old broker to new broker, one consumer group at a time. Validate each consumer group's behavior on the new broker before proceeding.
4. When all consumer groups have migrated and been validated under production load, cut off producer writes to the old broker.
5. Drain any remaining messages from the old broker. Decommission.

The dual-write period has cost: producers are doing double the write work, the old broker is still running and must be maintained, and any message that fails on one broker requires reconciliation logic. Do not underestimate the engineering work to implement clean dual-write.

### The Ordering Risk

This is the failure mode that causes the most subtle and hard-to-detect production bugs in broker migrations.

RabbitMQ provides per-queue FIFO ordering. If a producer sends messages A, B, C in sequence to a queue, a single consumer receives them in that order. This is a broker guarantee.

Kafka provides per-partition ordering. Messages with the same partition key arrive in order within that partition. Messages across partitions arrive in non-deterministic order at the consumer level.

If a consumer application depends on strict global ordering — "process payment-update before refund-issued for the same account" — and that ordering was guaranteed by RabbitMQ's queue semantics, the migration to Kafka requires explicit partition key design to preserve it. If the partition key is wrong (or absent), the ordering guarantee disappears silently. The application continues to function. Intermittently, it produces wrong results.

These bugs are the hardest to catch in testing and the most damaging in production.

Director framing: broker migration is not a "swap the URL" migration. Budget 6-9 months for a production RabbitMQ → Kafka migration at T-Mobile scale — hundreds of queues, dozens of producers and consumers, ordering semantics that must be audited queue by queue before migration.

## Language and Runtime Migration

Java 8 → Java 17. Python 2 → Python 3. These look like internal refactors. They behave like migrations: they have a dual-run period, a compatibility validation gate, and a cutover event. And they cascade.

**The cascade effect:** Upgrading the runtime forces library upgrades. Library upgrades introduce API changes. API changes require code modifications across teams that use those libraries. A runtime upgrade in a shared library can produce a wave of changes across dozens of services that the originating team did not anticipate and is not resourced to support.

**Budget framing for Directors:**

- Java 8 → 17: plan 1-2 engineer-weeks per service, assuming reasonable test coverage. Multiply by service count. Add 20% for the unexpected dependency breaks that surface during integration testing.
- Python 2 → 3: plan 2-4 engineer-weeks per service with significant business logic. Python 2 to 3 has deeper semantic breaks (string encoding, integer division, print statements, exception handling syntax) that require actual code review and modification, not just a runtime switch.

**Sequencing strategy:** Start with services that have the fewest upstream and downstream dependencies, the highest automated test coverage, and the most available team bandwidth. These early migrations de-risk the tooling and process. They produce runbooks and lessons learned that later migrations use. They also let the platform team validate that the new runtime is stable before it carries the full production load.

## Big Bang vs. Strangler Fig for Platform Migrations

The sequencing question in platform migration: do you migrate all applications to the new platform in a defined cutover window (big bang), or do you migrate incrementally while running both platforms in parallel (strangler fig)?

**Big bang:**
- Lower long-term operational complexity — no dual-platform maintenance period
- Higher organizational risk — a failed cutover affects everything simultaneously
- Requires very high confidence in the target platform before the cutover window
- Works for small portfolios (< 20 applications) where dual-platform operational cost outweighs migration risk

**Strangler fig:**
- Lower per-migration risk — each application migration is isolated
- Higher operational cost — running two platforms simultaneously is expensive in infrastructure cost and operational attention
- Produces gradual organizational learning — early-migrating teams build skills that later waves benefit from
- The migration factory model (below) makes this approach scale

**Director's rule:** For platform migrations affecting more than 50 applications or more than $10M in annual run cost, default to strangler fig. The dual-platform cost — real and significant — is insurance against catastrophic failure. An organization that cannot absorb the cost of running two platforms in parallel for 12-18 months should question whether it is ready for the migration at all.

## The Migration Factory Model

A platform migration affecting 50+ application teams requires dedicated migration capability. If each application team handles their own migration independently — reading documentation, making architecture decisions, debugging tooling issues — the program runs slowly and inconsistently.

The migration factory is a dedicated team that owns the repeatable migration work:

- Maintains the standard Helm templates, migration runbooks, and tooling
- Executes migrations for each application team wave by wave, or works alongside teams as a migration squad
- Feeds lessons learned from early waves back into the standard
- Tracks program velocity and flags blockers to Director-level leadership

**The staffing implication:** If you do not have a migration factory, your migration cost estimate is underestimated by 40-60%. The hidden cost is the time each application team spends solving problems that the migration factory would solve once and distribute. At scale, this is the difference between a 12-month program and a 24-month program with the same outcome.

The migration factory is not a permanent team. It is funded for the duration of the platform migration program, with a clear decommission date. Define that date before the program starts.

## Director's Lens

Platform migrations require three things that most programs fail to provision adequately before launch:

**A Platform Engineering team with migration tooling capacity.** Not just the target platform operational team — a team that can build, maintain, and iterate on the migration tooling, templates, and standards. Without this, you have pushed the architecture decision cost onto every application team.

**Standardization decisions made before migration begins.** The standard target pattern — ingress controller, secret management, logging approach, manifest structure — must be decided and locked before teams start migrating. Every week of ambiguity here produces divergent implementations that cost more to operate than the migration saved.

**A realistic dual-platform cost budget.** Running both the old and new platform simultaneously for 12-18 months has real cost. Infrastructure, operational attention, on-call coverage, tooling maintenance. This cost must be explicitly funded. Programs that treat dual-platform cost as "temporary" and exclude it from migration budgets consistently discover it mid-program, when it is too late to remediate without delay or scope reduction.

The platform migration that succeeds is the one where the Director spent more time on governance, standardization, and team structure in the first 90 days than on any technical decision. The technical decisions are consequential, but they are recoverable. Organizational and structural decisions made early are very hard to unwind once 20 teams are in flight.
