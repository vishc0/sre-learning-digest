# Migration Risk Patterns Reference

Fifteen most common migration failure patterns. Each pattern has appeared in real programs. Recognize the warning signs during assessment and wave planning — not during cutover.

Director use: scan this list when reviewing wave plans, assessing team readiness, or preparing for a SteerCo risk review.

---

## Pattern Table

| # | Pattern Name | Description | Warning Signs | Mitigation |
|---|-------------|-------------|--------------|-----------|
| 1 | **Hidden Dependency Discovered at Cutover** | A dependency that does not appear in the CMDB or architecture diagrams surfaces only when traffic shifts. The dependency cannot be migrated within the cutover window. Rollback is forced. | CMDB was last updated more than 6 months ago; dependency mapping was done by interviews rather than network traffic analysis; team has low confidence in their dependency inventory | Run automated dependency discovery (network flow analysis, APM dependency maps) before assessment is finalized; validate CMDB against actual traffic; require dependency count sign-off from app team lead as a wave entry gate |
| 2 | **Underestimated Data Volume** | Data export or sync takes 10x longer than estimated, consuming the entire cutover window before traffic can shift. The window closes with migration incomplete. | Estimates made from table row counts rather than actual data size; no export dry-run completed before cutover day; large blob/object columns not accounted for; historical data included unexpectedly | Run a full data export dry-run in a lower environment at least two weeks before cutover; measure actual transfer rate under realistic network conditions; size the cutover window based on measured, not estimated, data transfer time |
| 3 | **No Rollback Plan Tested** | A rollback procedure exists in the runbook but has never been executed end-to-end. During cutover, the rollback fails or takes three times longer than estimated, causing an extended outage. | Rollback section of the runbook says "reverse the steps above"; no rollback dry-run is on the pre-cutover checklist; rollback time estimate has no empirical basis | Treat rollback as a first-class deliverable; require a rollback dry-run in a lower environment before the cutover window opens; record the actual rollback time and use it to set the rollback deadline in the cutover plan |
| 4 | **Cutover Window Too Short** | A complex migration with multiple validation steps, a data sync, and a gradual traffic shift is crammed into a 4-hour maintenance window. The team runs out of time before validation completes and is forced to make a rushed go/no-go decision. | Cutover window was set by the change calendar before the migration plan was finalized; window duration was not validated against the step-by-step runbook timing; no buffer time for unexpected issues | Build the cutover runbook step-by-step first, estimate each step's duration, add 50% buffer, then request the window; never set the window first and reverse-engineer the plan into it |
| 5 | **Stakeholder Fatigue** | An 18-month migration program loses executive attention by month 9. Budget protection weakens, headcount is reallocated to new initiatives, and the migration team is left running on goodwill and prior commitments. | Monthly SteerCo attendance dropping; sponsor replaced or significantly distracted; migration appears in portfolio reviews as a line item rather than a strategic program; business value reporting has become rote | Define business value milestones — not just technical milestones — at the start of the program; report in terms of cost saved, apps decommissioned, and platform percentage migrated rather than engineer-weeks consumed; refresh the business case narrative at the midpoint of long programs |
| 6 | **Dual-Team Capacity Drain** | Migration teams also own BAU operations for the applications they are migrating. When production incidents occur (as they always do), incident response pulls engineers off migration work. Wave 2 slips. Wave 3 slips. The program falls irreparably behind schedule. | No explicit capacity model separating BAU from migration work; migration plan assumes 100% engineer availability with no incident buffer; wave plans have no slack built in | Model BAU interrupt rate from historical data (previous 6-12 months of incident volume per team); build interrupt capacity (20-30%) into wave plans; consider ring-fencing a migration team that is explicitly off BAU rotation for the wave duration |
| 7 | **Scope Creep from R Upgrade** | A "rehost" application gradually becomes a "refactor" as engineers discover improvements they want to make during the migration. Scope expands, timeline doubles, and the business case no longer holds. | Engineers describing migration tickets as "while we're in there, we should also..."; PRs touching business logic rather than infrastructure configuration; wave timeline expanding without formal change control | Establish R selection as a governance artifact — changing from Rehost to Refactor requires a formal scope change and updated business case; scope the migration work to the minimum change set required to run on the target platform; create a separate backlog for improvements to execute after the migration lands |
| 8 | **Environment Parity Gap** | The target environment does not replicate production load, configuration differences, or network topology. Issues that only appear under production conditions are not discovered until cutover. | Target environment built from a spec sheet rather than a production configuration audit; load testing run at 20% of peak rather than 150% of peak; network security groups, firewall rules, or certificate authorities differ between source and target | Conduct a configuration diff between source and target environments before the first wave; run load tests at 150% of expected peak; include a production traffic replay step in the cutover validation plan |
| 9 | **Data Validation Skipped** | Migration is declared complete — traffic is fully shifted and the source environment is decommissioned — without validating data integrity in the target. Corruption or data loss is discovered days or weeks later, after the rollback window has closed. | Cutover checklist has no data validation steps; data validation is listed as a post-cutover task rather than a gate before declaring success; teams under time pressure skip validation to hit the deadline | Data validation (row counts, checksums, business rule spot checks) must be a gate before traffic shifts to 100%, not an optional post-cutover task; define expected row counts before cutover and compare against actual in the target before proceeding |
| 10 | **Vendor Audit Triggered** | A software vendor detects that the organization is exiting their platform (through reduced license consumption, infrastructure changes, or direct communication) and initiates a license audit mid-migration. The audit creates legal distraction and may result in an unexpected true-up payment that erodes the business case. | Vendor has audit rights in the contract; license consumption dropping significantly as the migration progresses; vendor sales team suddenly engaging more aggressively; previous audit history with this vendor | Involve legal and procurement before decommissioning any licensed software; review audit clauses in vendor contracts before publishing migration timelines externally; maintain contractual license counts during migration even if actual usage declines — do not "go dark" before the contract allows |
| 11 | **Integration Re-Wiring Underestimated** | The original migration plan assumed 5 integrations; post-kickoff discovery finds 40. The additional integration re-wiring work is not in the plan, not in the budget, and not in the timeline. The wave timeline doubles or the work is skipped, creating silent integration gaps. | Integration count in assessment based on architecture diagrams rather than verified inventory; no API gateway or ESB logs reviewed to identify actual traffic patterns; number of integrations treated as a minor detail rather than a primary complexity driver | Treat integration count as a first-class complexity driver; extract actual integration inventory from API gateway logs, message broker topic/queue lists, and network flow data — not from diagrams; complete integration inventory before wave planning, not during it |
| 12 | **Network Topology Assumption** | An application was designed for an on-premises network segment and has hardcoded IP addresses, private DNS names, or network path dependencies that do not exist in the target environment. The issue is not discovered until cutover when the application cannot reach its dependencies. | Application configuration files contain hardcoded IP addresses rather than DNS names; on-prem DNS zones are not replicated to cloud DNS; application documentation does not mention network path dependencies; no network connectivity testing was done in the target environment pre-cutover | Require a network connectivity test as part of environment readiness (T-48 hours gate); audit application configuration for hardcoded network references during assessment; validate that all required DNS zones, firewall rules, and network paths exist in target before the cutover window |
| 13 | **Timezone / Locale / Encoding Issue** | Data migration introduces encoding bugs or timezone interpretation errors that are only visible in edge-case data — non-ASCII characters, legacy date formats, or character sets that differ between source and target databases. | Source and target databases use different default character sets (e.g., latin1 vs. utf8mb4); timestamp columns stored without timezone information; application assumes a specific locale for number or currency formatting; validation only checks row counts, not data content | Include encoding and locale validation in the data validation checklist; run spot checks specifically against rows containing non-ASCII characters and edge-case date values; validate timestamp interpretation (not just storage) in the target application |
| 14 | **Key Person Dependency** | A single engineer holds the majority of migration knowledge for a wave — runbooks, tool configurations, tribal knowledge about application quirks. That person leaves, takes extended leave, or is pulled to an incident during Wave 2, and the program stalls. | One engineer's name appears in every migration ticket, runbook, and architecture document; other engineers on the team cannot independently execute a cutover; no pairing or shadowing has occurred on Wave 1 cutover events | Require a minimum of two engineers to be present at every cutover event — one leads, one shadows and documents; treat runbook quality as a migration deliverable; no wave is "done" until a second engineer can execute the next wave cutover without assistance from the first |
| 15 | **Success Criteria Never Defined** | The migration "completes" but no one agrees on what complete means. Engineering says it's done because traffic shifted. Operations says it's not done because monitoring hasn't been tuned. Finance says it's not done because savings haven't materialized. Leadership declares success and demobilizes the team before cutover validation is finished. | Migration kick-off had no discussion of success criteria; wave closure criteria are not documented; business value measurement (cost savings, decommission confirmation) is not on the program milestone list | Define success criteria per wave and per application before the program starts: minimum is (1) traffic fully shifted, (2) source environment confirmed decommissioned, (3) post-migration cost baseline validated against forecast; require explicit stakeholder sign-off on each criterion |

## Pattern Frequency by Program Phase

Directors should raise the highest-frequency risks in SteerCo at the appropriate phase.

| Program Phase | Most Common Patterns to Watch |
|--------------|------------------------------|
| Assessment and Planning | 1 (hidden dependencies), 11 (integration undercount), 4 (window sizing) |
| Wave 1 | 8 (environment parity), 3 (rollback not tested), 9 (data validation skipped) |
| Wave 2+ | 6 (capacity drain), 7 (scope creep), 14 (key person dependency) |
| Mid-program (months 9-12) | 5 (stakeholder fatigue), 2 (data volume), 10 (vendor audit) |
| Cutover events | 12 (network topology), 13 (encoding issues), 15 (success criteria) |
