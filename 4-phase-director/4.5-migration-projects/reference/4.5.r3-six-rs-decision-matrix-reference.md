# Six Rs + Replace — Decision Matrix Reference

Quick-lookup guide for selecting the correct migration strategy for a given application. Use alongside the Migration Assessment Template.

---

## Primary Decision Table

| R Name | When to Use | When NOT to Use | Typical Timeline | Relative Cost |
|--------|------------|----------------|-----------------|--------------|
| **Retire** | Application is no longer needed; usage is zero or negligible; business capability is covered elsewhere | Any time the application has active users or business-critical integrations that are not covered | Weeks (decommission only) | Lowest — net cost reduction |
| **Retain** | Migration is technically feasible but not yet viable — team bandwidth unavailable, risk too high, or dependency chain not ready | As a permanent state — Retain must have a review date; "retain forever" is not a strategy | No migration work now; schedule review in 6-12 months | Lowest near-term; risk accumulates over time |
| **Rehost** | Application works as-is on new infrastructure; goal is cost or platform consolidation, not feature improvement; team lacks cloud-native skills and speed matters more than optimization | Applications with heavy platform coupling (PCF services, on-prem file systems, vendor-specific APIs) that will break on lift; applications where lift eliminates no meaningful cost | 2-8 weeks per application | Low |
| **Replatform** | Application needs minor modernization to run well on the target platform without a full code rewrite; containerization, managed database, or configuration changes are sufficient | Applications requiring significant code changes to work correctly on target platform; applications where modernization scope keeps expanding during assessment | 4-16 weeks per application | Medium |
| **Refactor** | Application has strategic value; existing codebase is viable but not cloud-native; team has capacity for code-level changes; test coverage is sufficient to refactor safely | Applications with poor test coverage (< 30%) — refactor without tests is rewrite without safety net; applications where the business case doesn't justify the engineering investment | 3-9 months per application | Medium-High |
| **Re-architect** | Application's fundamental architecture prevents it from meeting reliability, scalability, or cost targets on any platform; strategic importance justifies redesign investment | Applications with low strategic value — re-architecture is the most expensive R and should only apply to the crown jewels; any application where a SaaS replacement exists at lower cost | 6-18 months | Highest |
| **Replace** | A commercial SaaS or COTS product covers the business function at lower total cost than maintaining or migrating the custom application | Applications with regulatory, data sovereignty, or customization requirements the SaaS cannot meet; applications where switching costs (integration re-wiring, data migration, retraining) exceed the build/maintain cost | 3-12 months (vendor selection + migration) | Variable — often medium; evaluate TCO including integration and licensing |

---

## Decision Flow

Work through this sequence in order. Stop at the first applicable decision point.

**Question 1: Is the application still needed in 2+ years?**
- NO — usage declining, business capability being sunset, or capability already covered → **RETIRE**
- YES — continue to Question 2

**Question 2: Is migration technically feasible in the next 12 months?**
- NO — dependency chain not ready, team unavailable, risk unacceptable, vendor lock-in prevents clean migration → **RETAIN** with mandatory review date (6-12 months out); document blocking conditions
- YES — continue to Question 3

**Question 3: Is there a commercial SaaS that replaces this application's function?**
- YES — and the SaaS is lower cost AND the migration risk is lower than other Rs AND customization requirements are met → **REPLACE**
- NO — or SaaS exists but doesn't meet requirements → continue to Question 4

**Question 4: What is the application's strategic value to the business?**
- LOW or MEDIUM strategic value — do not over-invest; complexity and cost ceilings apply → go to Rehost or Replatform (Question 5a)
- HIGH strategic value — the application is a competitive differentiator or core to the business model → go to Question 5b

**Question 5a (Low/Medium strategic value): What does the application need to run on the target platform?**
- Minimal changes (container packaging, config externalization, DNS/network reconfiguration) → **REPLATFORM**
- No changes needed — pure infrastructure move → **REHOST**

**Question 5b (High strategic value): What is the application's technical debt level?**
- LOW debt (good test coverage, modern patterns, maintainable code) → **REPLATFORM** — get to the new platform quickly; debt is not a blocker
- MEDIUM debt (partial test coverage, some architectural compromise, maintainable with effort) → **REFACTOR** — use the migration as a forcing function to pay down debt
- HIGH debt (low test coverage, architectural patterns that prevent scaling or safe modification, no path to incremental improvement) → **RE-ARCHITECT** if strategic value justifies cost; **RETIRE** if it does not

## R Selection by Application Type

| Application Type | Recommended R | Rationale |
|-----------------|--------------|-----------|
| Legacy monolith, low business value, stable | Rehost or Retire | Modernization cost exceeds value; get it to a cheaper platform or shut it down |
| Legacy monolith, high business value, high debt | Re-architect | Debt prevents safe incremental improvement; redesign is necessary to unlock the application's value |
| Modern microservice (container-ready) | Replatform | Already architecturally cloud-native; packaging and config changes are sufficient |
| COTS application (vendor-packaged) | Replace or Rehost | Replace if a better or cheaper COTS alternative exists; Rehost if the vendor supports the target platform and switching costs are high |
| Custom batch job, low complexity | Rehost or Replatform | Batch jobs typically have fewer runtime dependencies; lift with minimal changes |
| Data pipeline | Replatform or Refactor | Data pipelines often have strong platform coupling (Hadoop, on-prem storage); assess tool compatibility before deciding |
| COTS vendor tool (e.g., monitoring, ticketing) | Replace | Vendor tools are almost always replaceable with cloud-native or SaaS equivalents; evaluate at portfolio level |
| Home-grown internal tool, low usage | Retire or Replace | Internal tools with limited adoption should be retired if possible; replace with SaaS if the function is genuinely needed |

## R Selection by Constraint

| Constraint | Recommended R | Why |
|-----------|--------------|-----|
| Hard deadline — vendor platform EOL in < 12 months | Rehost first, then Replatform | Speed is the primary variable; get off the expiring platform first, optimize second |
| Cost pressure — must reduce infrastructure spend this fiscal year | Rehost or Retire | Rehost delivers cost reduction fastest; Retire eliminates cost entirely |
| Talent pressure — team lacks skills for complex migration | Rehost | Rehost is the lowest skill-floor migration strategy; use it when team readiness is the constraint |
| Compliance deadline — regulatory requirement has a fixed date | Replatform or Rehost | Speed and predictability matter more than optimization; do the minimum to meet the compliance target |
| M&A integration — acquired company's systems must consolidate | Replace or Replatform | Replace consolidates onto the acquiring company's standard tooling; Replatform if Replace is not viable within the integration timeline |

## Common R Mismatches (Anti-Patterns)

These are the mistakes that show up repeatedly across migration programs. Recognizing them during assessment prevents rework mid-wave.

| Situation | Wrong R (Commonly Chosen) | Why It Fails | Correct R |
|-----------|--------------------------|-------------|-----------|
| Legacy app with low test coverage and high debt; team optimistically selects Refactor | Refactor | Refactor without test coverage means every change is a risk; teams slow down, scope expands, delivery stalls | Re-architect if strategic; Retire if not |
| PCF-deployed app with heavy PCF Services dependencies (Spring Cloud Config, Service Broker); team selects Rehost | Rehost | PCF Services are not available on raw Kubernetes or EC2; the app breaks immediately post-lift | Replatform — replace PCF Services with Kubernetes-native equivalents |
| Application with genuine SaaS replacement available; team selects Refactor because they want to keep the code | Refactor | Spending 6 months refactoring a custom app that a $50/month SaaS replaces is hard to justify; accumulates political and financial debt | Replace |
| Batch job with a hard vendor EOL date; team selects Re-architect to "do it right" | Re-architect | Re-architecture takes 12-18 months; the vendor EOL date is in 6 months; the app goes unsupported before the re-architecture completes | Rehost first to meet the EOL deadline; Re-architect in the next wave if strategic value warrants it |
| Internal tool used by 3 people; team selects Replatform because they are attached to it | Replatform | Investment is wildly disproportionate to usage; Replatform still requires engineering time, testing, and ongoing maintenance | Retire — or Replace with a SaaS if the function is genuinely needed |
