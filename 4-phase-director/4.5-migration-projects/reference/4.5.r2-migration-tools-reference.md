# Migration Tools Reference

Quick-lookup for migration tooling by category and use case. Organized by program phase. Use this reference when selecting tools for a migration program, evaluating vendor proposals, or advising teams on toolchain standardization.

---

## Discovery and Portfolio Assessment

Used in Phase 1 — before wave planning begins. Goal: build an accurate inventory of what you have, what it costs, and what depends on what.

| Tool | Vendor | What It Does | Best For | Limitations |
|------|--------|-------------|---------|-------------|
| AWS Application Discovery Service | AWS | Agentless or agent-based discovery of on-prem servers; collects CPU, memory, network connections, running processes | AWS-bound migration programs; initial inventory of on-prem VMs and physical servers | AWS-centric output; does not model application-level dependencies well without agent; agentless mode has lower fidelity |
| Cloudamize | Cloudamize (acquired by Rackspace) | Agent-based discovery with cloud cost modeling; generates right-sizing recommendations and TCO comparisons | Programs that need a financial model tied to discovery data; multi-cloud comparison | Agent deployment requires coordination with server teams; accuracy of cost projections depends on workload characterization fidelity |
| Movere | Microsoft | Agentless discovery of on-prem environments; integrates directly with Azure Migrate and Azure cost modeling | Azure-bound programs; organizations with existing Microsoft EA agreements | Designed for Azure migration; limited value for AWS or GCP programs; data collection requires firewall changes in some environments |
| Turbonomic (IBM) | IBM | Application resource management and cloud cost optimization; models current resource utilization and right-sizing recommendations | Programs where cost optimization is a primary driver; identifying oversized or underutilized workloads to Retire or Rehost-and-rightsize | Requires agent or integration with hypervisor; primarily an optimization tool, not a full migration planning platform; pricing scales with environment size |

## Dependency Mapping

Used during Assessment phase. Accurate dependency maps are the primary input to wave sequencing. Manual interviews are not sufficient for large portfolios.

| Tool | Vendor | What It Does | Best For | Limitations |
|------|--------|-------------|---------|-------------|
| Dynatrace (Smartscape) | Dynatrace | Automated dependency mapping via OneAgent instrumentation; builds a real-time topology map of services, processes, hosts, and network connections | Production environments where agents can be deployed; microservice architectures with API-based integrations | Agent-based — requires deployment; maps observed traffic only (does not capture dependencies that haven't fired recently); licensing cost scales with monitored hosts |
| AWS X-Ray | AWS | Distributed tracing for applications running on AWS; maps service call chains and identifies latency contributors | Applications already running on AWS or migrating to AWS; tracing-based dependency discovery for microservices | AWS-native — limited value for on-prem discovery; requires code instrumentation or X-Ray SDK integration; does not map infrastructure-level dependencies |
| Jaeger | CNCF (OSS) | Open-source distributed tracing; maps service dependencies through trace data | Teams already using OpenTelemetry; cloud-native microservice environments; programs that prefer OSS tooling | Requires application instrumentation; no infrastructure-level mapping; operational overhead to deploy and maintain the Jaeger backend |
| ServiceNow CMDB | ServiceNow | Configuration management database; intended to be the system of record for all application and infrastructure relationships | Organizations with mature ServiceNow deployments and disciplined CI hygiene; formal change management programs | CMDB accuracy is only as good as the processes maintaining it — often the least reliable source of dependency truth in practice; requires significant ServiceNow admin effort to keep current |
| Miro / Lucidchart (manual) | Miro / Lucidchart | Collaborative diagramming for building dependency maps through facilitated workshops | Small portfolios (< 20 applications); programs where tool deployment is not feasible; validating automated maps with subject matter experts | Does not scale; maps are only as accurate as the people in the room; becomes a maintenance burden immediately after creation |

## Database Migration

The highest-risk component of most migrations. Tool selection depends on database engine pair, acceptable downtime window, and whether CDC (change data capture) is required.

| Tool | Vendor | What It Does | Best For | Limitations |
|------|--------|-------------|---------|-------------|
| AWS DMS (Database Migration Service) | AWS | Managed service for homogeneous and heterogeneous database migrations; supports continuous replication mode for near-zero downtime migrations | AWS-bound migrations; Oracle-to-Aurora, MySQL-to-RDS, SQL Server-to-RDS; ongoing replication during parallel run period | Homogeneous migrations (same engine) are well-supported; heterogeneous migrations require AWS Schema Conversion Tool (SCT) for DDL translation; LOB data types and complex stored procedures require manual intervention |
| Striim | Striim | Enterprise data integration and CDC platform; supports real-time replication with transformation across heterogeneous databases and streaming systems | Complex heterogeneous migrations where data transformation is required during migration; Oracle-to-Kafka-to-cloud patterns | Significant licensing cost; operational complexity; requires expertise to configure pipelines correctly |
| Debezium | Confluent / Red Hat (OSS) | Open-source CDC connector that streams database change events to Kafka; supports MySQL, PostgreSQL, Oracle, SQL Server, MongoDB | Programs already using Kafka as a data streaming backbone; teams that want to build custom migration pipelines; dual-write / event-sourcing patterns | Requires Kafka infrastructure; operational overhead to manage connectors; not a turnkey migration tool — needs surrounding pipeline architecture |
| Flyway | Redgate (OSS + commercial) | Database schema migration tool; version-controls DDL changes and applies them in sequence | Managing schema changes during application refactor or replatform; maintaining schema version history across environments | Not a data migration tool — manages schema (DDL), not data (DML); does not handle production data movement |
| Liquibase | Liquibase (OSS + commercial) | Database schema version control and migration tool; changelog-based approach to DDL management | Polyglot environments supporting multiple database engines; teams that prefer XML/YAML changelogs over SQL scripts | Same limitation as Flyway — schema management only; significant learning curve for teams unfamiliar with changelog-based tooling |
| pgloader | OSS | Bulk data loading tool for PostgreSQL; handles MySQL-to-PostgreSQL and SQLite-to-PostgreSQL migrations | Small-to-medium migration programs migrating to PostgreSQL; one-time bulk data moves | PostgreSQL-destination only; not suitable for continuous replication; limited support for large-scale or heterogeneous enterprise migrations |

## Application Migration (Cloud)

Used for lift-and-shift and rehost scenarios. These tools automate the physical movement of workloads from on-premises to cloud infrastructure.

| Tool | Vendor | What It Does | Best For | Limitations |
|------|--------|-------------|---------|-------------|
| AWS MGN (Application Migration Service) | AWS | Agent-based continuous block-level replication from on-prem to AWS; enables near-zero-downtime cutover by keeping the target in sync until cutover | Rehost migrations to AWS; physical and virtual servers; any OS supported by the MGN agent | AWS-destination only; agent deployment required on source servers; does not address application-level configuration changes needed for cloud operation |
| Azure Migrate | Microsoft | Unified migration hub for assessment, dependency analysis, and server migration to Azure; integrates with Movere for discovery | Azure-bound programs; organizations with existing Azure EA; Hyper-V and VMware environments | Azure-destination only; migration capabilities vary by source environment; some features require Azure Arc or Azure Site Recovery |
| Google Migrate for Compute Engine | Google | VM migration service for moving workloads to Google Compute Engine; supports AWS-to-GCP and on-prem-to-GCP | GCP-destination programs; VMware-based on-prem environments; AWS-to-GCP migrations | GCP-destination only; less mature than AWS MGN for large-scale enterprise programs |
| Velero | VMware (OSS) | Backup and restore for Kubernetes clusters and persistent volumes; supports cluster migration by backing up source and restoring to target | Kubernetes workload migration between clusters or cloud providers; disaster recovery for Kubernetes; stateful workload migration | Kubernetes-only; not designed for non-container workloads; requires storage compatibility between source and target; point-in-time backup, not continuous sync |

## Data Pipeline / ETL Migration

Used when migrating batch jobs, data pipelines, or ETL workflows between platforms.

| Tool | Vendor | What It Does | Best For | Limitations |
|------|--------|-------------|---------|-------------|
| AWS Glue | AWS | Serverless ETL service; auto-generates PySpark code from visual editor; integrates with AWS data catalog | AWS-bound data pipeline migrations; replacing on-prem Informatica or SSIS with a managed service; organizations without a dedicated data engineering team | AWS-only; serverless model has cold-start latency; complex transformations require custom PySpark code despite the visual interface |
| Apache Spark | Apache (OSS) | Distributed data processing engine; runs on Databricks, EMR, HDInsight, or self-managed clusters | Large-scale data transformations; replacing legacy Hadoop MapReduce jobs; programs with a dedicated data engineering team | Requires Spark expertise; cluster management overhead (unless using a managed service); not a migration tool — it is a processing engine used within a migration pipeline |
| dbt (data build tool) | dbt Labs (OSS + commercial) | SQL-based transformation layer; version-controls and tests data transformations | Analytics engineering workflows; replacing stored-procedure-based transformation logic with version-controlled SQL; BI layer migrations | Transformation layer only — does not move raw data; requires a target data warehouse (BigQuery, Snowflake, Redshift, Databricks) already provisioned |
| Airbyte | Airbyte (OSS + commercial) | Open-source data integration platform with 300+ connectors; moves data between sources and destinations | Replacing legacy ETL point solutions; organizations that want to consolidate on a single EL (extract-load) platform; teams comfortable with OSS operational overhead | OSS version requires self-hosting; connector quality varies; transformation capabilities are basic (use dbt alongside for transforms); commercial version pricing at scale |
| Fivetran | Fivetran | Managed EL platform; fully automated connector management and schema drift handling | Programs that need managed connectors with minimal operational overhead; analytics data migrations where engineering bandwidth is constrained | Cost scales with monthly active rows — can become expensive at high data volumes; limited customization of connector behavior; not suitable for real-time / streaming use cases |

## Message Broker Migration

Relevant for environments running RabbitMQ, TIBCO EMS, IBM MQ, or on-prem Kafka that need to migrate to cloud-native or managed messaging. This is a specialty area — broker migrations require careful sequencing because message ordering, durability guarantees, and consumer group offsets do not translate directly between platforms.

| Tool | Vendor | What It Does | Best For | Limitations |
|------|--------|-------------|---------|-------------|
| Kafka MirrorMaker 2 | Apache (OSS) | Replicates Kafka topics between clusters; supports active-active and active-passive topologies | Kafka-to-Kafka migrations (on-prem Kafka to MSK, Confluent Cloud, or Aiven); cross-region replication | Kafka-to-Kafka only; does not bridge to RabbitMQ or TIBCO; consumer group offset translation requires careful configuration; does not guarantee exactly-once semantics across clusters by default |
| RabbitMQ Shovel / Federation | RabbitMQ (OSS) | Shovel: moves messages from a source queue to a destination queue (one-way); Federation: creates loosely coupled links between brokers for pub/sub patterns | RabbitMQ-to-RabbitMQ migrations; bridging on-prem RabbitMQ to cloud-hosted RabbitMQ (CloudAMQP, AmazonMQ) during parallel run period | RabbitMQ-to-RabbitMQ only; Shovel is one-directional; Federation adds operational complexity; does not bridge to Kafka or TIBCO |
| Confluent Replicator | Confluent | Enterprise-grade Kafka topic replication; handles offset translation, schema registry sync, and topic configuration migration | Large-scale Kafka migrations where MirrorMaker 2 operational complexity is a concern; organizations already using Confluent Platform | Requires Confluent Platform license; commercial tool with corresponding cost; limited value for non-Confluent Kafka environments |
| Custom Dual-Write Pattern | Internal build | Application publishes to both old and new broker simultaneously during migration; consumers are migrated independently to the new broker; old broker is decommissioned after all consumers migrate | Migrations between incompatible broker types (e.g., TIBCO EMS to Kafka, RabbitMQ to Kafka) where no tool bridges the gap; organizations with strong engineering capability | Engineering cost to build and maintain; requires application code changes; producer and consumer migration must be carefully coordinated; does not handle in-flight messages in the old broker's queues |

## Cutover and Traffic Management

Used during the cutover event itself to shift traffic incrementally and enable rollback without downtime.

| Tool | Vendor | What It Does | Best For | Limitations |
|------|--------|-------------|---------|-------------|
| AWS Route 53 Weighted Routing | AWS | DNS-based weighted routing; distributes traffic across endpoints by percentage | Simple DNS-level traffic shifts where application-level routing is not available; Rehost migrations where the only change is the server endpoint | DNS TTL introduces lag in traffic shift — changes take minutes to propagate, not seconds; not suitable for precise incremental shifts (e.g., 5% → 25% in real time) |
| Nginx / HAProxy Weighted Upstreams | OSS | Load balancer upstream weighting; enables precise percentage-based traffic splits with immediate effect | On-prem or self-managed load balancer environments; Replatform scenarios where an existing load balancer is already in the path | Requires access to load balancer configuration; changes require a reload/restart (brief impact on long-lived connections); no built-in observability integration |
| Argo Rollouts | Argo Project (OSS) | Progressive delivery controller for Kubernetes; supports canary, blue-green, and weighted traffic shifting natively in Kubernetes | Kubernetes-based application migrations; teams using GitOps workflows; Replatform scenarios where the target is Kubernetes | Kubernetes-only; requires CRD installation and Argo Rollouts controller; integrates with service mesh (Istio, Linkerd) or Kubernetes ingress — requires compatible ingress controller |
| Flagger | Flux project (OSS) | GitOps-based progressive delivery for Kubernetes; automates canary analysis using metrics from Prometheus, Datadog, or CloudWatch | Teams using Flux for GitOps; migrations where automated rollback based on metrics is desired; Replatform to Kubernetes | Kubernetes-only; requires metrics infrastructure to be in place before use; more complex to configure than Argo Rollouts for simple use cases |
| LaunchDarkly (Feature Flags) | LaunchDarkly | Feature flag management platform; enables traffic routing decisions at the application code level rather than the infrastructure level | Application-level migrations where the new code path runs alongside the old (strangler fig pattern); Refactor and Re-architect scenarios; gradual rollout to user segments | Requires SDK integration in the application code — not an infrastructure-level tool; adds a runtime dependency on the LaunchDarkly service; cost scales with monthly active users |

## Validation and Testing

Used throughout the migration program — not just at cutover. Data validation and load testing should be gates, not afterthoughts.

| Tool | Vendor | What It Does | Best For | Limitations |
|------|--------|-------------|---------|-------------|
| Great Expectations | OSS | Data quality and validation framework; defines expectations about data shape, types, and values; generates data documentation | Post-migration data validation; ongoing data quality monitoring in the target environment; validating that source-to-target data migration is complete and correct | Python-based — requires data engineering familiarity; expectations must be written before migration (requires knowledge of expected data characteristics); not a real-time tool |
| Gatling | OSS | Load testing tool with Scala/Kotlin DSL; generates detailed performance reports; designed for high-volume HTTP workload simulation | Load testing target environments before cutover; validating that the target environment can sustain production peak traffic | Requires load test scenario development (not zero-effort); Scala DSL has a learning curve; HTTP-focused — not designed for message broker or database load testing |
| k6 | Grafana Labs (OSS + commercial) | JavaScript-based load testing tool; developer-friendly syntax; integrates with Grafana for results visualization | Teams that prefer JavaScript over Gatling's Scala; cloud-native environments already using Grafana; load testing APIs and microservices | HTTP/gRPC focused; browser-based load testing requires k6 browser extension; cloud execution at scale requires k6 Cloud subscription |
| Chaos Monkey / LitmusChaos | Netflix OSS / CNCF | Chaos engineering tools; inject controlled failures (instance termination, network delay, pod deletion) to validate resilience | Validating that the target environment fails gracefully under production conditions; pre-cutover resilience testing; post-migration steady-state validation | Requires a mature observability stack to safely run chaos experiments; inappropriate for pre-production environments without adequate safeguards; LitmusChaos is Kubernetes-only |
| Custom Reconciliation Scripts | Internal build | Purpose-built scripts that compare source and target data across specific business rules — row counts, checksum validation, business logic spot checks | Applications with complex business rules that generic data validation tools cannot model; any migration where data correctness is a compliance requirement | Engineering cost to build and maintain; scripts must be written before cutover; scope and thoroughness depend entirely on engineering investment |

## Tool Selection Guide

**When to use commercial tools**: When the program scale, timeline, or compliance requirements make operational risk of OSS tooling unacceptable. Commercial tools provide support contracts, documented upgrade paths, and vendor accountability. Choose commercial when: the migration is on a hard deadline, the team lacks the depth to troubleshoot OSS tooling under pressure, or the organization has existing enterprise agreements that reduce incremental cost.

**When to use OSS tools**: When the team has strong engineering depth, the program timeline allows for learning curves, and organizational standard is already OSS-first. OSS tools are often more flexible and avoid vendor lock-in. Choose OSS when: the tool is well-established with an active community (Debezium, Kafka, Argo Rollouts), the team has prior experience with the tool, and there is no hard dependency on vendor support.

**When to build custom**: Only when no commercial or OSS tool addresses the specific requirement — typically in message broker migrations between incompatible protocols, or data validation against highly specific business logic. Custom tooling carries a maintenance burden; document it thoroughly and plan for handover.

**Director-level standardization principle**: Standardize on 2-3 tools per category across the organization rather than allowing each migration team to select independently. Independent tool selection across waves creates: fragmented operational knowledge, incompatible observability integrations, duplicated training costs, and increased vendor management overhead. Establish a migration toolchain standard at program kickoff, codify it in the migration playbook, and require a formal exception process for teams that want to deviate. The exception process is not bureaucracy — it is the mechanism by which new tools get evaluated and adopted into the standard.
