# Chapter 5: Data Migration

## Why Data Migration Is the Hardest Migration

Code can be rewritten. Data cannot.

That is the foundational asymmetry of data migration, and it changes every risk calculation you make. When you migrate an application incorrectly, you get downtime — a bad enough outcome, but a recoverable one. When you migrate data incorrectly, you get corruption or loss. Recovery is not measured in hours; it is measured in weeks of forensic reconstruction, if recovery is possible at all.

This is why data migration programs require a dedicated engineering workstream, independent validation gates, and rollback plans that are qualitatively different from application rollback plans. The blast radius of a data migration failure is not a service outage — it is a trust deficit that outlasts the technical recovery.

Director framing: the moment you understand this asymmetry, you stop treating data migration as a sub-task of the application migration and start treating it as a parallel program with its own budget, its own gates, and its own accountability.

## ETL vs. ELT

The traditional model is ETL: Extract, Transform, Load. You pull data from the source, apply transformation logic to match the target schema, then load the result. ETL assumes you know exactly what the target schema looks like before you start moving data. That assumption is rarely valid at the beginning of a large migration.

The modern cloud-native model is ELT: Extract, Load, Transform. You move raw data into the target system first, then apply transformation logic in place. This separates two fundamentally different problems: getting the data there, and making it correct for the new schema.

ELT is increasingly preferred for cloud data migrations because solving two problems sequentially is more reliable than solving them simultaneously. You can validate data movement independently of schema compatibility. You can iterate on transformations without re-running the full extract. You can bring in data teams and application teams to work on their respective layers without blocking each other.

The tradeoff is that ELT requires sufficient compute and storage in the target environment to hold raw data alongside transformed data during the migration window. That is a cost line item to budget explicitly.

## Online vs. Offline Migration

**Offline migration** is conceptually simple. The system is taken down, data is migrated in bulk, the system is brought back up on the new platform. The source and target are not live simultaneously, which eliminates the delta problem. This pattern is appropriate for non-critical systems, systems with scheduled maintenance windows, or migrations where the business can absorb a planned outage.

**Online migration** runs while the system is live, with no downtime. It is dramatically more complex because users continue writing to the source system while data is being migrated to the target. Any record written after the initial extract must also make it to the target — this is the delta problem.

Solving the delta problem requires Change Data Capture (CDC) tooling — software that continuously reads the database transaction log and replays changes to the target in near-real-time. CDC pipelines introduce operational overhead: they must be monitored for lag, they must handle schema changes gracefully, and they add a replication failure mode that must be detected and alarmed on.

Online migration is required for high-availability systems. But "online migration" is not free complexity — it is a design choice that must be resourced, staffed, and monitored as a production workload throughout the migration window.

## The Dual-Write Pattern

For zero-downtime database migration, the dual-write pattern is the safest approach. The application is modified to write to both the old and new database simultaneously. Reads are shifted incrementally from old to new as confidence in the new system grows.

The benefits are significant: no single cutover moment where all or nothing succeeds, reads can be rolled back independently of writes, and the new database can be validated under real production write load before any reads are served from it.

The costs are real and must be understood:

- Write latency increases because the application waits for both writes (or handles partial failures explicitly)
- The application must implement divergence detection — if a write succeeds to the old system but fails to the new, that record is now out of sync
- The migration window is extended because the dual-write period can last weeks
- Code complexity is higher, and that complexity must be removed cleanly after migration completes

Director framing: dual-write is the right pattern for high-value, high-availability databases. Budget for the engineering cost of implementing it, operating it, and removing it. Skipping it to save time is how migrations create silent data divergence that surfaces months after cutover.

## Cutover Strategies

**Big bang cutover** moves all traffic from old to new at a scheduled moment. It is clean and simple to reason about, and it ends the dual-state complexity quickly. It carries the highest risk per event. It requires a tested, documented rollback plan that can be executed under pressure.

**Gradual traffic shift** moves 5% of traffic, observes, moves to 25%, observes, then 50% and 100%. Each increment is lower risk than a full cutover, but the migration duration is longer and operational complexity during the transition period is higher. Two systems must be operated simultaneously throughout.

**Feature flag cutover** embeds the migration switchover inside application feature flags. This is the most flexible approach — individual users, tenants, or transaction types can be migrated independently. It requires application-level support and a feature flag management system, but it gives you the finest-grained control over rollout and rollback.

The choice of cutover strategy is driven by blast radius. Higher blast radius means smaller increments, longer stabilization windows, and more conservative rollback triggers.

## Data Validation Framework

Data validation is the most frequently skipped step in data migration programs, and it is the step that determines whether the migration was successful. Row counts matching between source and target is not validation — it is counting. Validation means confirming that the data is correct, complete, and consistent with business rules.

A complete data validation framework covers four layers:

**Row count validation**: the target contains the same number of records as the source, segmented by entity type, date range, and partition. A match here is necessary but not sufficient.

**Checksum validation**: a hash of key fields in each record matches between source and target. This catches transformation errors, encoding issues, and truncation that row counts miss.

**Business rule validation**: application-specific invariants must hold. Account balances must sum correctly. Order totals must match line items. Referential integrity between tables must be preserved. These rules must be defined by the application team before migration begins — they cannot be invented afterward.

**Statistical validation**: the distribution of values in the target matches the source within defined tolerance. Outlier counts, null rates, and value ranges should be compared. This catches systematic transformation errors that affect entire columns rather than individual records.

Director framing: data validation is a gate, not a task. No migration wave closes until all four validation layers pass. Define the pass/fail criteria before the migration begins, and give a named engineer accountability for each validation layer.

## Expand-Contract Pattern for Schema Migration

In zero-downtime systems, database schema changes cannot be deployed simultaneously with application code changes. The expand-contract pattern solves this.

**Expand**: add the new column, table, or index alongside the old structure. Both old and new application code can run against the database simultaneously — the old code ignores the new column; the new code uses it.

**Migrate**: populate the new structure while the old structure continues to serve production. Run the data backfill as a background job, not as a blocking migration.

**Contract**: once all application instances are running new code and the old structure is confirmed unused, remove it.

This three-phase approach eliminates table locking, avoids downtime from long-running schema changes, and keeps the database in a state both old and new code can tolerate throughout the migration. It adds elapsed time to the migration, but that time is operational cost, not downtime.

## Director's Lens

Data migration requires a dedicated data engineering workstream. It cannot be absorbed by the application development team as a side task. The validation tooling, CDC pipeline operations, and rollback data restoration all require engineering capacity that competes directly with feature development. Budget it separately and staff it separately.

The most common data migration budget failure is funding the move and not the validation. Validation tooling, test data environments for validation rehearsal, and the engineering time to define and run business rule checks are not optional line items. They are the difference between a migration that is technically complete and a migration that is provably correct.
