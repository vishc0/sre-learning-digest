# Chapter 03: Cloud Migration Patterns

**Audience**: Director-level  
**Framing**: You have been through a PCF → EKS migration as a practitioner. This chapter reframes what you lived through from the top of the org chart down — the decisions that were made above you, the forces that shaped them, and how you will make those decisions when you are the one setting migration strategy.

---

## Why This Chapter Exists

Cloud migration is not a technology problem. It is a portfolio decision with technology execution. The confusion between those two levels — portfolio vs. execution — is responsible for most failed migrations.

Practitioners optimize for the application in front of them. Directors must optimize for the portfolio of 200 applications, the budget that has to last three years, and the strategic value each application delivers. The Rs — Rehost, Replatform, Refactor, Retire, Repurchase — are a portfolio categorization framework, not a technical checklist.

Your job is to set the philosophy, enforce the discipline, and protect the program from the scope explosions that derail migrations before they finish.

---

## The Six Rs: A Quick Orientation

The "6 Rs" framing (Gartner / AWS) gives a vocabulary for migration disposition:

| R | What It Means |
|---|---------------|
| **Rehost** | Move as-is to cloud infrastructure — "lift-and-shift" |
| **Replatform** | Migrate to a new runtime model with minimal code changes |
| **Refactor** | Rearchitect the application to be cloud-native |
| **Repurchase** | Replace with a SaaS equivalent |
| **Retire** | Decommission — the application is no longer needed |
| **Retain** | Leave in place — not ready or not worth moving yet |

The mistake most organizations make is treating R selection as a technical call delegated to engineering teams. It is not. R selection is a business case decision. Technical teams have a role in informing it — they understand the cost and risk of each path — but they should not be setting migration strategy unilaterally.

---

## Rehost (Lift-and-Shift): When It Is the Right Call

Rehost moves a workload from on-premises infrastructure to a cloud equivalent with minimal modification. A VM becomes a cloud instance. A containerized application running on a private registry runs instead on a cloud registry. The topology is the same. The infrastructure is different.

**When rehost makes sense:**

- Speed-to-cloud outweighs optimization. A data center contract is expiring in 90 days. There is no time to refactor. Get out first, optimize later — if later is real.
- A vendor EOL date is forcing a move. The on-premises platform is losing support. The application itself will migrate, but the first priority is removing the platform dependency.
- The application is scheduled for retirement. It will be decommissioned in 18-24 months. Refactoring it is burning money on a building you are about to demolish.

**When rehost creates debt:**

- The application has hardcoded on-premises assumptions. Specific hostnames, internal network segments, file paths tied to NFS mounts, IP addresses in config files. Rehosting transfers these assumptions to cloud with no remediation. They become time bombs.
- The workload is architecturally incompatible with cloud economics. Certain mainframe workloads are built around the I/O model and memory subsystems of mainframe hardware. Moving them to cloud VMs achieves presence in the cloud but no cloud economics.
- "Lift-and-shift now, optimize later" is organizational fiction. Later never comes unless it is contractually committed, funded, and on a roadmap. Directors must call this explicitly: if "optimize later" is not on a real roadmap, the decision is "lift-and-shift permanently." That changes the business case.

## Replatform: The PCF → EKS Pattern

Replatform is the most common migration path for organizations running mature application portfolios on PaaS. The application logic does not change. The operational substrate does. Same binary, different runtime.

**The practitioner view** — what replatform looks like from inside a migration team: you are changing how the application is deployed, scheduled, scaled, and discovered. You are not changing what the application does or how it processes requests.

**The Director view** — replatform unlocks cloud-native operational capabilities that PaaS abstracts away. Horizontal Pod Autoscaler, Cluster Autoscaler, spot instance scheduling, advanced ingress control, multi-region failover. These are not available in the same form on PCF. Replatform is the minimum viable path to accessing them without bearing a full refactor cost.

### The Replatform Complexity Underestimation Trap

When T-Mobile moved from PCF to EKS, every team doing "replatform" actually did 30-40% of a refactor — because PCF abstracted away what Kubernetes surfaces.

PCF handled: build process via Buildpacks, service binding via CF marketplace, routing via GoRouter, health management via CF Health Manager, scaling via CF scaling commands. Every one of these abstractions was invisible to developers while on PCF. Every one of them required explicit implementation work when teams arrived on Kubernetes.

The hidden replatform work that most programs underestimate:

- Buildpacks → Dockerfiles (not just a conversion — requires understanding the build environment, base images, layer caching strategy)
- CF service bindings → Kubernetes Secrets and ConfigMaps (and decisions about secret management tooling — Vault, AWS Secrets Manager, External Secrets Operator)
- CF routes → Kubernetes Ingress controllers (and decisions about which ingress controller, TLS termination, path routing rules)
- CF health checks → Kubernetes liveness and readiness probes (a subtle but important distinction with real production consequences)
- CF autoscaling → HPA with custom metrics (if the app scaled on non-CPU metrics in CF, that logic must be rebuilt)

If your migration program is budgeting for "replatform" at significantly lower cost than "refactor," audit whether teams have accounted for this hidden work. Most have not.

## Re-architect (Refactor to Cloud-Native)

Re-architect is the highest-investment migration path. It decomposes monolithic applications into microservices or rebuilds them using cloud-native patterns. Done well, it yields the best long-term operational posture. Done poorly, it becomes a multi-year product program that loses connection to its migration origins.

### The Strangler Fig Pattern

The strangler fig is the standard decomposition pattern for monoliths. Build new capabilities as microservices alongside the existing monolith. Route new traffic to the new services. Gradually replace monolith functions with service equivalents until the monolith handles no traffic and can be decommissioned.

The name comes from the strangler fig tree, which grows around a host tree, eventually replacing it. The host tree (monolith) dies slowly as the new structure takes over.

**Why Directors should be skeptical of re-architect proposals:**

Re-architect migrations expand scope uncontrollably. A migration that starts as "move the billing monolith to Kubernetes" becomes "rebuild billing as 12 microservices" within two sprints. That becomes a 3-year product program that is no longer recognizable as a migration. The org has now taken on a major product initiative disguised as infrastructure work, with migration budget, migration staffing, and migration timelines — none of which are sized for what the work actually is.

The Director intervention: if a "migration" is requiring feature changes, data model redesign, or service decomposition decisions, it has become a product program. Rename it. Fund it correctly. Staff it correctly. Do not run a product transformation on migration resources.

## The R-Matching Rule

The R selection must be driven by the application's strategic value and its remaining useful life — not by the engineering team's preference for interesting work.

| Strategic Value | Remaining Useful Life | Correct R |
|-----------------|----------------------|-----------|
| High | Long | Refactor or Re-architect |
| Medium | Medium | Replatform |
| Low | Short | Rehost or Retire |
| Any | < 18 months | Retire or Rehost only |

**The critical enforcement point:** Technical teams left to their own will consistently choose Refactor. It is the most interesting work. It produces the cleanest architecture. It lets engineers build something they are proud of. All of those are true. None of them are relevant to a low-value application with two years of useful life.

Directors must establish and enforce R-to-value alignment as a governance gate. R selection should require sign-off from the Director or above, based on a written business case that includes strategic value tier, estimated remaining useful life, and cost comparison across the viable Rs.

## Cloud Migration Anti-Patterns at the Director Level

**Migrating everything in the same wave.** "Cloud by Christmas" programs fail because the organization's capacity to migrate is not infinite. Wave-based migration — where each wave is sized to organizational change absorption capacity — consistently outperforms big-bang approaches.

**Declaring victory at infrastructure migration.** Moving 200 applications to cloud is not a cloud migration success. If applications are running in cloud with on-premises operational patterns — fixed IP assumptions, no autoscaling, no cloud cost visibility, no use of managed services — you have achieved cloud residency, not cloud benefit.

**No cloud cost governance during migration.** The dual-run period, when applications exist in both on-premises and cloud environments simultaneously, is the most expensive period of a migration. Cloud costs spike because you are paying for both. Without governance gates that define when dual-run ends, migration programs regularly run 40-60% over budget in their first year.

**Treating migration as a one-time project.** Applications keep being built. New acquisitions bring new application portfolios. Regulatory changes force application changes that have migration implications. Cloud migration is a permanent operational capability, not a three-year project. The organization that treats it as a project disbands the migration team at completion — and then pays to reconstitute it eighteen months later.

## Director's Lens

Cloud migration is a portfolio investment decision with a technology execution track. Your responsibilities at the Director level:

**Set the migration philosophy.** Which Rs are acceptable for which application tiers? Make this explicit and documented. "Tier 1 applications (revenue-critical, 5+ year life) require Replatform minimum. Tier 3 applications (internal tooling, < 2 year life) are Rehost or Retire only."

**Establish governance gates.** Who approves R assignments? Who approves exceptions? What evidence is required — a business case, an architecture review, a cost comparison? Without governance gates, every team makes their own R decision and you end up with 200 different approaches to a portfolio migration.

**Protect programs from scope expansion.** The moment a migration starts looking like a product build, surface it. Rename it. Fund it correctly. Do not let product transformation hide inside migration budgets.

**Measure outcomes, not activity.** Migration success is not "applications moved." It is cloud cost as a percentage of revenue (trending down), application availability (maintained or improved), and operational toil (reduced through cloud-native capabilities). Define these before the program starts, not after.
