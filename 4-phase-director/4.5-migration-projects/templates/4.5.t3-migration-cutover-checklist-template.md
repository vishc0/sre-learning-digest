## PRE-CUTOVER CHECKLIST — T-48 Hours

Complete this section 48 hours before the cutover window. If any item cannot be checked, the Go/No-Go poll at T-1 hour should receive a NO-GO from the responsible party.

### Environment Readiness

- [ ] Target environment fully provisioned and accessible
- [ ] Target environment load tested; capacity validated against expected peak
- [ ] All required secrets, credentials, and configs migrated to target environment's secret store
- [ ] DNS, load balancer, or service mesh routing pre-configured for cutover (traffic shift should be a single config change, not a construction task)
- [ ] Monitoring and alerting fully active on target environment — dashboards validated
- [ ] Log aggregation confirmed flowing from target environment to observability platform

### Data Readiness

- [ ] Source-to-target data sync completed (or final sync window scheduled)
- [ ] Row count validation passed — Source: ___ / Target: ___
- [ ] Checksum validation passed
- [ ] Business rule validation passed (sample queries against target return expected results)
- [ ] Data validation report saved to: [link / path]

### Rollback Readiness

This section is non-negotiable. A rollback plan that exists only on paper is not a rollback plan.

- [ ] Rollback runbook reviewed and approved by Migration Lead and SRE Lead
- [ ] Rollback infrastructure staged — not just planned, physically ready
- [ ] Rollback procedure dry-run completed in lower environment: Date ___
- [ ] Rollback Lead confirmed available for the entire cutover window
- [ ] Rollback time estimate: ___ minutes
- [ ] Rollback tested end-to-end: Yes / No (if No, explain in risk log)

### Change Management

- [ ] Change ticket approved: Ticket # ___
- [ ] No change freeze active during the cutover window (verify with CAB / change management)
- [ ] Stakeholder notification sent: sent on [date/time], to [distribution list]
- [ ] Downstream team leads notified and have confirmed awareness
- [ ] Runbook for cutover execution published to: [link]

## PRE-CUTOVER GATE — T-1 Hour (Go/No-Go Poll)

The Migration Lead polls each role for an explicit GO before the window opens. A single NO-GO pauses the cutover. Document the reason. Determine: proceed with risk accepted, or postpone.

| Role | Name | GO / NO-GO | Notes |
|------|------|-----------|-------|
| Migration Lead | | GO / NO-GO | |
| SRE On-Call | | GO / NO-GO | |
| App Team Lead | | GO / NO-GO | |
| Rollback Lead | | GO / NO-GO | |
| Stakeholder Comms Lead | | GO / NO-GO | |

**Gate outcome**: PROCEED / POSTPONE / PROCEED WITH ACCEPTED RISK

**If postponed**: Document reason, proposed new window, and what must be resolved before next attempt.

## DURING CUTOVER — Execution Checkpoints

Work top to bottom. Do not skip steps. Each traffic checkpoint is a gate. Record actual metrics at each step and make an explicit GO / NO-GO call before shifting more traffic.

### Setup

- [ ] Cutover bridge open; all required participants confirmed present
- [ ] Stakeholder channel updated: "Cutover in progress — [application name] — [start time]"
- [ ] Pre-cutover health baseline captured (attach or paste): Error rate ___ | P99 latency ___ | Throughput ___

### Traffic Shift Steps

Fill in thresholds before the window opens. Do not establish thresholds in the middle of a cutover.

**Pre-defined abort thresholds** (set before window opens):

- Business KPI: [define] ___  threshold → ROLLBACK

## ROLLBACK PROCEDURE

If any step above produces a NO-GO call:

1. Migration Lead announces rollback decision on bridge: "Initiating rollback — [time]"
2. Stakeholder channel updated: "[Application name] — rolling back — [time]"
3. Rollback Lead executes rollback runbook: [link]
4. Traffic confirmed returned to source: Error rate ___ | P99 latency ___
5. Source system health confirmed stable: GO / NOT STABLE
6. Stakeholder channel updated: "Rollback complete — [time] — investigating root cause"
7. Incident ticket opened: Ticket # ___

**Rollback completed at**: [time]
**Root cause (preliminary)**: ___
**Next steps**: [re-attempt date, required remediation before retry]

## POST-CUTOVER VALIDATION — T+1 Hour

- [ ] All validation checks pass on target environment
- [ ] Monitoring dashboards showing healthy baselines — screenshot or link: ___
- [ ] No active alerts on target environment
- [ ] Downstream systems confirmed operating normally — confirmed with [team]
- [ ] Business KPIs within expected range — actual: ___ | expected: ___
- [ ] Source (legacy) environment confirmed offline or in standby — not actively serving traffic
- [ ] Stakeholder channel updated: "Cutover complete — [application name] — [time]"
- [ ] Executive / stakeholder summary sent: Yes / No

---

## POST-CUTOVER CLOSURE — T+24 Hours

Do not close the migration until the 24-hour stability period is confirmed. Incidents that surface 6-8 hours after cutover are common — triggered by overnight batch jobs, timezone-driven load patterns, or downstream caches expiring.

- [ ] 24-hour stability confirmed — no P1/P2 incidents attributed to migration
- [ ] Rollback infrastructure decommissioned or clearly scheduled for decommission
- [ ] CMDB / configuration management updated to reflect new hosting environment
- [ ] Change ticket closed: Ticket # ___ | Closed: [date]
- [ ] Legacy environment decommission ticket opened (if not immediate): Ticket # ___
- [ ] Post-cutover retrospective scheduled: [date/time]
