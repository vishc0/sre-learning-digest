# Migration Projects — Director-Level Training Framework

## Why This Exists

Platform migrations are where Director-level careers are made or broken. The failure mode is almost never technical. RabbitMQ knows how to replicate. EKS knows how to schedule pods. Cassandra knows how to partition data. What breaks migrations is governance: fuzzy success criteria, stakeholder communication that lags reality, wave sequencing that ignores dependencies, and discovery work that was budgeted at 5% when it needed 20%. This framework is a structured body of knowledge for the Director or aspiring Director who needs to govern large-scale migrations — not execute them, govern them. The distinction matters from page one.

> **Core principle**: Migrations get worse before they get better. The moment a migration program starts, operational complexity increases: dual systems, split team attention, degraded on-call coverage, and a rising incident rate as teams run both the old and new environments simultaneously. This is not a sign of failure. It is the physics of migration. Design your stakeholder communication plan around this fact before kickoff, not after the first incident.

## How to Use This Framework

### Practitioner Reading Path
You are an SRE, tech lead, or senior engineer who needs to understand how migrations are governed and wants to operate within a well-run program.

1. Start with ch01 (philosophy) and ch02 (assessment) to understand how decisions are made upstream of your work
2. Read the relevant playbook for your current migration type (pb01, pb02, or pb03)
3. Use the templates during actual program execution
4. Reference ch04 and ch07 for cutover and data migration specifics

**Time investment**: ch01 + ch02 + one playbook = solid foundation. Read the rest as the program phases progress.

### Director-Prep Reading Path
You are preparing for a Director or VP interview, or you are stepping into a Director role and will own a migration program.

1. Read ch01 through ch03 in sequence — this is your governance foundation
2. Read ch05 (stakeholder communication) before anything else involving executives
3. Internalize t01 (portfolio heat map) and t02 (steering deck) — these are your primary operating artifacts
4. Read ch06 (risk management) and ch08 (closure) to understand what "done" looks like at program level
5. Use the playbooks as reference depth when your engineers bring technical questions to steering

**Time investment**: Full framework before taking ownership of a program. For interview prep, ch01 + ch02 + ch05 + t02 is the minimum viable set.

## Domain Anchors

Throughout this framework, examples are grounded in:
- **Replatform**: PCF (Pivotal Cloud Foundry) to EKS — the canonical mid-complexity migration for a Java/Kotlin microservices shop
- **Messaging**: RabbitMQ cluster migration and topology changes
- **Data**: Cassandra cluster migration and partition key evolution
- **Observability**: Splunk as the source-of-truth for validation, SLO measurement, and cutover confidence scoring

These are not arbitrary examples. They represent the decision patterns that transfer across migrations regardless of the specific technology stack.
