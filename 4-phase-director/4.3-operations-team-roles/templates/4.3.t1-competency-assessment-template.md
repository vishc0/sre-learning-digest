# Competency Assessment Template

> **Instructions for employee**: Complete your self-assessment independently before you see your manager's assessment. Rate each competency honestly — this is a development tool, not a performance gotcha. Inflating your ratings denies you the feedback you need. For every rating, write specific evidence: a real incident, a real runbook, a real conversation. "I'm good at this" is not evidence.

> **Instructions for manager**: Complete your assessment independently before seeing the employee's self-assessment. The goal of doing it independently is to surface gaps in perception — those gaps are the most valuable part of the discussion. Be honest. Overly generous ratings protect no one and deprive the employee of accurate development signal.

> **Discussion**: Schedule a 60-minute 1:1 to review both assessments together. Start with areas of agreement, then spend the bulk of time on meaningful rating gaps (2+ points apart). End with the development plan — do not leave the meeting without at least one agreed-upon development action with a target date.

## Rating Scale

| Rating | Meaning |
|--------|---------|
| **1 — Below expectations** | Specific gap identified for this level. A development plan is required. This is not a judgment — it's a signal that support is needed. |
| **2 — Developing** | Showing progress toward level expectations but not yet consistently there. On track if trajectory continues. |
| **3 — Meets expectations** | Solid and reliable at this level. No development gap. This is the expected rating for someone performing well in their role. A "3" is not mediocre — it means "doing the job." |
| **4 — Exceeds expectations** | Demonstrating competency at the next level up. Relevant for promotion consideration. Rare — a team of all 4s is a calibration problem. |

> **Calibration note for managers**: If more than 30% of your ratings are 4s, recalibrate against what next-level performance actually looks like in your context. A strong performer who is growing is usually a 3 with upward trajectory — not a 4 by default.

## Core Competencies

> **Note**: These competencies are written for SRE roles. For Change Manager, Release Engineer, or Platform Architect roles, modify the competency names and descriptions to match the role's core functions. The structure (rating, evidence, discussion) applies to any operations role.

### Technical Competencies

| Competency | Definition for this level | Employee Rating (1–4) | Employee Evidence (specific, not general) | Manager Rating (1–4) | Manager Evidence (specific, not general) |
|------------|--------------------------|----------------------|------------------------------------------|---------------------|------------------------------------------|
| **Incident Command** | Handles the incident severity appropriate for this level independently and reliably: declares, assembles, assigns, drives to resolution, closes, and hands off cleanly. | | | | |
| **Runbook quality** | Produces runbooks that another IC can execute accurately without asking the author for help. Runbooks are current, tested, and version-controlled. | | | | |
| **Monitoring and alerting design** | Sets up alerts that are meaningful: right threshold, right severity, right responder. Alerts fire when they should and don't fire when they shouldn't. Knows the difference between a symptom and a cause when building an alert. | | | | |
| **Root cause analysis** | Leads or substantively contributes to RCA/PIR. Can distinguish contributing factors from root causes. Produces PIR outputs that result in actionable improvements, not just timelines. | | | | |
| **Domain knowledge** | Understands the services, dependencies, failure modes, and data flows in their assigned domain. Can explain how a failure in service A propagates to service B without looking it up. | | | | |
| **Tooling depth** | Works at depth in at least one observability platform (Splunk, Datadog, New Relic, or equivalent). Writes queries, builds dashboards, sets meaningful alert thresholds. Does not rely on pre-built dashboards created by others for their own domain. | | | | |
| **Troubleshooting methodology** | Follows observe → hypothesize → test → conclude. Does not skip to fix. Can articulate why a hypothesis was wrong and what they ruled out. Avoids random change thrashing during an incident. | | | | |

### Operational Competencies

| Competency | Definition for this level | Employee Rating (1–4) | Employee Evidence | Manager Rating (1–4) | Manager Evidence |
|------------|--------------------------|----------------------|-------------------|---------------------|-----------------|
| **Escalation judgment** | Knows when to escalate vs. handle independently. Does not hold incidents too long before calling for help. Does not over-escalate routine events. Can articulate what would change their escalation decision. | | | | |
| **Communication under pressure** | Calm, structured, and clear in high-stress situations. Updates stakeholders without being asked. Gives status with three components: what we know, what we're doing, when we'll know more. Does not go silent during an incident. | | | | |
| **Change risk assessment** | Assesses the real risk of a change before executing it — not just "did I follow the checklist?" Can describe what would go wrong, how they'd detect it, and what they'd do to reverse it. | | | | |
| **Process contribution** | Improves team processes, not just executes them. Has submitted at least one process change, runbook improvement, or workflow fix in the assessment period. Treats "this is how we do it" as a starting point, not a ceiling. | | | | |

### Interpersonal Competencies

| Competency | Definition for this level | Employee Rating (1–4) | Employee Evidence | Manager Rating (1–4) | Manager Evidence |
|------------|--------------------------|----------------------|-------------------|---------------------|-----------------|
| **Mentorship and knowledge transfer** | Actively develops less experienced teammates. Pairs on incidents. Reviews runbooks written by others. Gives feedback that helps people grow, not just corrects errors. | | | | |
| **Cross-team collaboration** | Works effectively with Engineering, Product, and other Ops domains. Can disagree professionally and still get to resolution. Follows up and closes the loop without being chased. | | | | |
| **Feedback receptivity** | Acts on feedback received. Can name something they changed in the last quarter based on feedback. Does not deflect or minimize critical feedback. | | | | |

## Career Goals

> **Employee completes this section independently before the discussion.**

**12-month goal** — What do you want to be doing or able to do 12 months from now that you can't do today?

[Your answer — be specific. "Get better at incidents" is not a goal. "IC P1 incidents independently without secondary support, and facilitate my first FMEA" is a goal.]

**3-year goal** — Where do you see yourself in 3 years? IC track (going deeper technically) or management track (managing a team)? Both are valid — this conversation helps your manager create opportunities that match your direction.

[Your answer — honest is more useful than strategic. If you don't know yet, say so and describe what factors would help you decide.]

**One skill you want to develop this quarter** — be specific enough that you'd know at the end of the quarter whether you succeeded.

[Your answer — e.g., "Write and maintain a Splunk dashboard for my domain's top 5 SLIs so I can diagnose incidents without asking the SRE-Monitoring team for help."]

**One stretch assignment you want** — a project, a responsibility, a rotation, or a leadership opportunity you want to be considered for.

[Your answer — e.g., "I want to facilitate the next FMEA for the payments service. I've participated in two but never led one."]

> **Complete this section for annual assessments only. Skip for quarterly assessments.**

[ ] **Yes** — they are consistently performing at the next level. I am nominating them for the next promotion cycle.
[ ] **Within 6 months** — they are on trajectory. With continued performance and the development actions above, they will be ready.
[ ] **Not yet — development needed** — specific gaps must close before promotion is appropriate.

**If "Not yet"** — what specifically needs to change? Do not write "needs more experience." Write the observable outcomes that would change this assessment:

[Specific items — e.g., "Needs to IC 3 P1 incidents without secondary support and receive no critical escalation feedback.
