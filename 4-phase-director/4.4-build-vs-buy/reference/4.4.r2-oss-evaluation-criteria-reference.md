# OSS Evaluation Criteria — Quick Reference

**Purpose**: Structured criteria for evaluating open source projects for production adoption. Use this when OSS is on the shortlist during a build vs. buy evaluation.

**Core principle**: OSS has no license cost. It has implementation cost, maintenance cost, security patching cost, and governance risk. Evaluate those honestly.

---

## OSS Health Scoring Table

Score each criterion from 1–5. Total score out of 40 (sum of raw scores before weighting). Weighted score = raw score × weight.

| Criterion | Weight | Score 1 (Concerning) | Score 3 (Acceptable) | Score 5 (Healthy) | Where to Find Data |
|-----------|--------|---------------------|---------------------|------------------|-------------------|
| **Commit Frequency** (last 90 days) | 15% | <10 commits in 90 days; project appears dormant; last commit >3 months ago | 10–50 commits in 90 days; regular cadence; some weeks without activity | 50+ commits in 90 days; consistent weekly activity; evidence of active feature development and bug fixes | GitHub Insights → Contributors graph; filter to last 3 months |
| **Contributor Count** (unique contributors, last 12 months) | 15% | <5 unique contributors; effectively a one-person project | 5–20 unique contributors; small active community | 20+ unique contributors; contributors from multiple organizations; no single contributor accounts for >50% of commits | GitHub Insights → Contributors; check organizational affiliation of top contributors via LinkedIn |
| **Governance Model** | 20% | Single commercial entity controls all decisions; no community governance; no public roadmap | Community input exists but commercial backer retains final control; public roadmap; community voting on features | Foundation-governed (CNCF, Apache, Linux Foundation); neutral Technical Steering Committee; decisions made publicly with community input | Project's GOVERNANCE.md or CONTRIBUTING.md; foundation membership page |
| **CNCF Graduation Status** | 10% | Not in CNCF ecosystem; no foundation affiliation | CNCF Sandbox (early stage; minimal maturity requirements) or CNCF Incubating | CNCF Graduated; demonstrates sustained adoption, security processes, and community governance maturity | landscape.cncf.io; project's own documentation |
| **Commercial Backer** | 10% | No commercial backer AND low contributor count (project may be abandoned if key maintainer loses interest or changes jobs) | Startup backer with venture funding; project tied to company survival; OR large open community with no specific commercial backer | Established commercial backer (large enterprise) OR strong foundation backing with multiple commercial sponsors | Crunchbase for company health; GitHub sponsors list; project blog/press releases |
| **License Type** | 15% | AGPL, SSPL, BSL, or other copyleft/commercial-restricted license that limits internal production use or redistribution without commercial license | MPL or GPL — permissive for internal use but copyleft for distribution; review carefully for your use case | Apache 2.0 or MIT — fully permissive; use in proprietary systems, modify freely, distribute without copyleft obligation | LICENSE file in repository root |
| **Fork History** | 10% | Multiple forks exist due to governance or licensing disputes; community is fragmented across competing projects | One notable fork exists but the original project remains dominant with clear community support | No significant forks; community united around a single codebase; historical forks reintegrated | GitHub fork count (large number ≠ fragmentation; search for named forks with active development) |
| **Community Size** | 5% | <500 GitHub stars; no active discussion forum; questions on GitHub Issues go unanswered >2 weeks | 500–5,000 GitHub stars; active Slack/Discord with some response to questions; periodic blog posts or community calls | 5,000+ GitHub stars; active community Slack/Discord with fast response times; regular community calls; annual conference or track | GitHub stars; CNCF Slack channels; project website |

**Scoring interpretation**: 35–40 = strong candidate for production adoption; 25–34 = acceptable with documented risk mitigation; 15–24 = significant concerns, evaluate carefully before production commitment; <15 = high-risk, avoid for production without extraordinary justification.

## License Risk Reference

| License Name | Type | Use in Proprietary Products? | Modify and Distribute? | Lock-in Risk | Notes |
|-------------|------|------------------------------|----------------------|-------------|-------|
| **Apache 2.0** | Permissive | Yes — no restrictions | Yes — keep attribution and license | Low | Most enterprise-friendly OSS license. Used by Kubernetes, Kafka, Spark. Patent grant included — protects against patent claims from contributors. |
| **MIT** | Permissive | Yes — no restrictions | Yes — keep attribution | Low | Simplest permissive license. No patent grant (unlike Apache 2.0) — minor consideration for patent-sensitive environments. |
| **GPL v2** | Strong copyleft | Yes for internal use; distributing requires source release | Yes — but must release source of modified version under GPL | Medium | Linux kernel license. Copyleft triggers on distribution, not internal use. Widely used in infrastructure; legal review for anything distributed to customers. |
| **GPL v3** | Strong copyleft | Yes for internal use; distributing requires source release | Yes — but must release source; also includes anti-tivoization clause | Medium | Stricter than GPL v2; includes anti-tivoization (cannot use GPL v3 software in hardware that prevents user modification). Less common in infrastructure. |
| **AGPL** | Network copyleft | Internal use requires careful legal review; offering as a service may trigger copyleft | Modifications must be released if offered as a service | High | "Network copyleft" — copyleft triggers when you offer modified software as a service (SaaS), not just when you distribute. Grafana OSS is AGPL. Requires legal review before production use in SaaS products. |
| **BSL (Business Source License)** | Commercial-restricted OSS | Yes for non-production and non-competing use; production use may require commercial license | Yes with restrictions | High | Used by HashiCorp (Terraform 1.6+), MariaDB. Not OSI-approved open source. Restricts "production use" and "competing" use — both terms require careful reading. Converts to permissive license after a defined date (typically 4 years). |
| **SSPL (Server Side Public License)** | Commercial-restricted | Internal use generally permitted; offering as a service triggers copyleft of all infrastructure code | Modifications must be released including all service software if offering as a service | High | MongoDB's license (post-2018); Elastic (pre-fork to OpenSearch). Not OSI-approved. Amazon's response to Elasticsearch using SSPL was to fork as OpenSearch under Apache 2.0. Strong signal of vendor wanting to limit cloud providers. |
| **MPL (Mozilla Public License)** | Weak copyleft | Yes | Yes — file-level copyleft; modifications to MPL files must be released, but can combine with proprietary code | Low-Medium | HashiCorp's original Terraform license before BSL change. Mozilla Firefox. File-level copyleft is generally manageable in practice. |

## CNCF Maturity Levels

| Level | Criteria | What It Means in Practice | Production Adoption Risk |
|-------|---------|--------------------------|-------------------------|
| **Sandbox** | Project submitted and accepted by CNCF Technical Oversight Committee; minimal requirements met | Early-stage project. May not be production-ready. CNCF sponsorship means it passed initial criteria but does not mean it is proven at scale. Think: "interesting to watch, not yet safe to bet on." | High — use only in non-critical paths or for evaluation purposes |
| **Incubating** | Demonstrated healthy community growth, clear versioning and release process, multiple production adopters documented, security processes in place | Project has real production users who have documented their use. Community is growing. Governance is maturing. Think: "viable for production with appropriate review; know your risk tolerance." | Medium — production viable for most workloads with documented fallback plan |
| **Graduated** | Demonstrated sustained adoption at scale, mature governance process, security vulnerability disclosure process, neutral hosting under CNCF, passing security audits | Battle-tested at scale across multiple organizations. The community has converged on this project as the standard. Think: "safe default choice; the burden of proof is on NOT using this." | Low — production recommended; equivalent to evaluating mature commercial software |

**CNCF Graduated projects as of 2025 (subset)**: Kubernetes, Prometheus, Envoy, CoreDNS, Containerd, Fluentd, Jaeger, Vitess, ArgoCD, Flux, Thanos, Falco, OpenTelemetry (Collector), Argo Workflows.

## Commercial OSS Risk Indicators

Five warning signs that a commercial OSS project may change its license, fork, or significantly change behavior in ways that affect your adoption:

**1. Single commercial backer with no foundation governance**  
All code decisions go through employees of one company. No neutral Technical Steering Committee. If the company changes strategy, is acquired, or faces financial pressure, the project changes. This is the structure that preceded the Elasticsearch and HashiCorp license changes. Check: does the project have a governance document that names an independent steering committee? If not, it is at risk.

**2. Backer facing financial pressure**  
A company burning cash with a narrowing runway has incentive to monetize its OSS project. Acquisition pressure creates the same dynamic — the acquirer may not share the current team's commitment to OSS. Check: Crunchbase for funding history and recent rounds; LinkedIn for executive departures; news for financial reporting if public.

**3. Large cloud provider offering a managed version that competes with the backer**  
This is the pattern that triggered both Elasticsearch/SSPL and HashiCorp/BSL. If AWS, Azure, or GCP is offering a managed service of an OSS project whose commercial backer is also selling a managed service of the same project, the backer is under commercial pressure. The license change is a predictable response. Check: does the major cloud provider have a managed offering in this category? If yes, watch the license.

**4. License already contains commercial use restrictions**  
BSL, SSPL, and AGPL all have commercial use restrictions that are materially different from permissive licenses. If a project uses these licenses today, it has already drawn a line around commercial use — future restrictions are easier to justify than the first restriction. Review the license before adoption, not after.

**5. Project has already forked once**  
A project that has forked previously — especially due to governance or licensing disputes — has a community that has already demonstrated willingness to fracture. The original project may not be the dominant one in two years. Check: are there named active forks? How does commit activity on the fork compare to the original? (Example: OpenSearch vs. Elasticsearch; OpenTofu vs. Terraform.)

| **Terraform / OpenTofu** | Infrastructure as Code | Terraform: HashiCorp (BSL); OpenTofu: Linux Foundation | Terraform: BSL (1.6+); OpenTofu: MPL 2.0 | OpenTofu: Linux Foundation project | HashiCorp (Terraform); Linux Foundation (OpenTofu) | Terraform: High (BSL); OpenTofu: Low | For new adoption, evaluate OpenTofu as the foundation-governed alternative.
