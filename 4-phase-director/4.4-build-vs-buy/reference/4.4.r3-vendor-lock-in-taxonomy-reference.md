# Vendor Lock-in Taxonomy — Quick Reference

**Purpose**: Structured classification of all vendor lock-in types, with mitigations and contract negotiation priorities. Use during vendor evaluation (Phase 4 of exit assessment) and contract review.

---

## Lock-in Taxonomy Table

| Lock-in Type | Definition | Real-World Examples | Primary Mitigation | Exit Cost Estimate |
|-------------|-----------|--------------------|--------------------|-------------------|
| **API Lock-in** | Internal systems directly depend on vendor-proprietary APIs that have no standard equivalent. Replacing the vendor requires rewriting integrations. | Vendor-specific SDKs called directly from application code; proprietary REST APIs for data ingestion; non-standard authentication flows | Wrapper pattern: internal systems call an abstraction layer you own, not the vendor API directly | Medium — weeks to months of integration rewriting depending on depth and breadth of API usage |
| **Data Lock-in** | Your data is stored in a vendor-proprietary format or schema that requires transformation to use elsewhere. Export may be difficult, slow, or incomplete. | Vendor proprietary database schemas; encrypted data where vendor holds the keys; AI training data stored in vendor-managed format; custom metadata structures | Negotiate open-format data export in contract; test export before signing; store originals in your own systems for important data | High — data transformation engineering plus validation effort; risk of data loss or corruption during migration |
| **Operational Lock-in** | Your operations team has built workflows, dashboards, runbooks, and operational knowledge around this vendor's specific UI, CLI, and tooling. Switching requires retraining and rebuilding operational practice. | Team trained on vendor-specific monitoring UI; alerting and oncall workflows built around vendor alert formats; runbooks reference vendor-specific commands | Maintain vendor-neutral operational documentation; avoid vendor-specific tool names in runbooks where standard equivalents exist; cross-train on alternative tooling periodically | Medium — retraining time + operational instability period during transition; usually 1–3 months before team reaches equivalent operational competency |
| **People Lock-in** | Engineers or operators have invested in vendor-specific certifications, skills, or specialization that creates personal switching cost. Strong preferences form, making migration politically difficult even when economically justified. | AWS/GCP/Azure certification investment; Splunk certified power user; Salesforce administrator certification; Kubernetes vendor certifications | Track certifications as team assets, not vendor loyalty; include neutral evaluation of alternatives in certification programs; build multi-vendor skill sets deliberately | Low-Medium financially; High organizationally — human change management is often the hardest part of vendor migration |
| **Contractual Lock-in** | Legal terms in the contract prevent exit or make exit financially punishing, independent of any technical difficulty in migrating. | Multi-year contracts with large early termination fees; auto-renewal clauses with short cancellation windows; minimum spend commitments that extend past intended use period | Negotiate termination for convenience clause; limit initial term to 1 year with options; set calendar reminders for renewal windows 6 months in advance | Contract-defined — the penalty is specified in the contract. If no exit clause exists, may require legal action to exit. |
| **Ecosystem Lock-in** | Other tools and vendors in your stack depend on this vendor, creating cascading switching costs. Replacing this vendor requires changing multiple other systems. | Cloud provider that also provides your container registry, CI/CD, monitoring, and IAM — migrating any one requires migrating all; vendor that provides authentication for all other tools | Design vendor integrations with independence: avoid vendor A's tooling depending on vendor B's tooling unless there is a strong cost-benefit for the coupling | High — cascade effect means exit cost is multiplied across all dependent systems |

## Mitigation Strategy by Type

### API Lock-in
- **Abstraction wrapper** (Medium effort): Build an internal abstraction layer that your application code calls. The abstraction calls the vendor API. When you switch vendors, you update the abstraction, not every application. This is the standard pattern for preventing API lock-in going forward — it does not help if you are already deep in vendor API calls.
- **Standard protocol preference** (Low effort): During vendor evaluation, prefer vendors that implement standard protocols (OAuth2, OpenMetrics, OTLP, REST with standard auth) over vendors that require proprietary SDKs. Score this in integration complexity criterion.
- **API surface audit before signing** (Low effort): Before signing, count the number of internal systems that will call vendor APIs. This is the lock-in depth. A vendor with 20 API integration points has 3–5x the lock-in depth of a vendor with 4 integration points.

### Data Lock-in
- **Pre-signature export test** (Low effort, high value): Export a sample of your data before signing the contract. Validate that the export format is usable without transformation. If the vendor cannot or will not do this, walk away.
- **Dual-write for critical data** (High effort): For the most critical data, write to both the vendor system and your own storage simultaneously. This is expensive but eliminates data lock-in entirely for that data set.
- **Open format contractual requirement** (Low effort): Make open-format data export a contract requirement. Vendors who refuse are telling you something about their exit intentions.

### Operational Lock-in
- **Vendor-neutral documentation standard** (Low effort): Establish a documentation standard that references system behavior, not vendor-specific UI paths. "Navigate to the metrics dashboard" rather than "Click on Metrics in the [VendorName] left sidebar."
- **Periodic alternatives exposure** (Low effort): Once per year, run a 1-day workshop where the team operates a representative alternative tool. Maintains transferable skills without disrupting operations.

### People Lock-in
- **Certification portfolio policy** (Low effort): Require a mix of vendor-specific and vendor-neutral certifications (CKAD, Prometheus, OpenTelemetry) alongside vendor-specific certs. Prevents skill monoculture.
- **Internal mobility**: Rotate engineers across tooling deliberately. Engineers who have only worked with one vendor tool are a bus factor risk and a lock-in risk simultaneously.

### Contractual Lock-in
- **Non-negotiable contract terms** (see contract review template): Termination for convenience, pricing cap, and data portability SLA are not optional. They must be in the contract before signing.
- **Calendar management**: Set a 6-month reminder before every auto-renewal date. Cancellation windows are frequently 30–60 days; missing them means another year of contract.

### Ecosystem Lock-in
- **Vendor diversity by layer** (Medium effort): Intentionally avoid using the same vendor for compute, monitoring, and CI/CD. Cross-layer dependency with a single vendor maximizes ecosystem lock-in.
- **Portability-first architecture choices**: Prefer managed Kubernetes over vendor-proprietary container orchestration. Prefer OTLP-native monitoring over vendor-proprietary agents. Prefer S3-compatible object storage over vendor-proprietary blob APIs.

## Lock-in Risk Scoring Guide

Score each vendor on each dimension from 1 (minimal lock-in) to 5 (maximum lock-in).

| Score | API Lock-in | Data Lock-in | Operational Lock-in | People Lock-in |
|-------|------------|--------------|--------------------|-|
| **5 — Maximum** | Proprietary SDK required in application code; no standard API alternative; vendor recommends against abstraction layers | Proprietary data format with no export capability; vendor holds encryption keys; export not available or never tested | Vendor-specific UI, CLI, and alerting format with no equivalents; all runbooks reference vendor-specific tools | Vendor certification is primary career currency for this function; team has no experience with alternatives |
| **3 — Moderate** | Mix of standard REST and proprietary extensions; abstraction is possible but requires effort; 5–15 integration touch points | Open format export available but requires transformation step; export tested and works; some custom schema elements | Standard protocols used in backend but vendor UI is primary operational interface; retraining effort is meaningful but bounded | Team has vendor certifications; some cross-training on alternatives; migration would require explicit retraining program |
| **1 — Minimal** | Standard protocols throughout (OAuth2, OpenMetrics, OTLP, REST); no SDK required; fewer than 5 integration touch points | Open standard data format (JSON, CSV, Parquet); export tested and works without transformation; data export SLA in contract | Vendor uses industry-standard tooling (Prometheus-compatible metrics, standard log formats); team skills transfer to alternatives easily | No vendor-specific certifications required; team is tool-agnostic; vendor preference but not dependency |

**Composite lock-in risk**: Average across all four dimensions. Score above 3.5 = high exit cost vendor; plan negotiation and contract terms accordingly. Score above 4.0 = consider whether you need this vendor at all, or whether a lower-lock-in alternative exists.

## Contract Negotiation Priorities by Lock-in Type

| Lock-in Type | Primary Contract Negotiation | Reference |
|-------------|------------------------------|-----------|
| API Lock-in | Right to use vendor APIs for any integration purpose, including replacement systems, during transition period | Section 3 (Exit Clauses) of contract review template |
| Data Lock-in | Data export format in open standard; export SLA ≤30 days; no data hostage clause; data remains org property | Section 2 (Data Portability) of contract review template |
| Operational Lock-in | Access to full API/CLI documentation without NDA restriction; right to build internal tooling that wraps vendor capabilities | General contract terms; confirm no restrictions on tooling built on top of vendor platform |
| People Lock-in | Not primarily a contract issue — managed through hiring and certification policy | Not contract-related |
| Contractual Lock-in | Termination for convenience clause; annual price increase cap; early termination penalty structure; auto-renewal cancellation window | Sections 3 and 4 of contract review template |
| Ecosystem Lock-in | Vendor cannot require use of their other products as a condition of support or pricing | General contract terms; review most-favored-nation and bundling clauses |

## Case Study Summary Table

| Case | Lock-in Type(s) | Trigger Event | Impact | Lesson |
|------|----------------|---------------|--------|--------|
| **VMware / Broadcom (2023)** | Contractual + Ecosystem | Broadcom acquired VMware; eliminated perpetual licenses; moved to subscription-only bundles; pricing increased 3–10x for some customers | Organizations with deep VMware vSphere/NSX/vSAN stacks faced massive price increases or multi-year migration projects. Those with good contract terms had leverage; those without did not. | Ecosystem lock-in becomes contractual lock-in at acquisition. Test your exit cost before the acquisition happens, not after. |
| **Elastic / AWS (2021)** | API + Data | Elastic changed Elasticsearch license from Apache 2.0 to SSPL (non-OSS) citing AWS's managed offering; AWS forked the project as OpenSearch | Organizations that had built directly on Elasticsearch APIs had to evaluate: stay on Elastic (new commercial terms), migrate to OpenSearch (fork), or accept AWS's managed service. Fork-aware orgs had a path; others had vendor lock-in they hadn't quantified. | Single-vendor OSS with a cloud provider offering a competing managed service is a specific high-risk pattern. Monitor license changes in all production OSS dependencies. |
| **HashiCorp / Terraform BSL (2023)** | API + Ecosystem | HashiCorp changed Terraform's license from MPL (permissive) to BSL (Business Source License, restricting competitive use); OpenTofu fork created under Linux Foundation | Vendors building products on top of Terraform (competing with HashiCorp's commercial offerings) faced immediate licensing risk. OSS adopters faced governance uncertainty. OpenTofu provided a migration path. | Commercial OSS with a dominant single backer facing competitive pressure will eventually consider license changes. Prefer foundation-governed OSS (Linux Foundation, CNCF, Apache) over single-vendor OSS for foundational infrastructure tooling. |
| **CentOS EOL (2021)** | Operational + Ecosystem | Red Hat changed CentOS from a stable RHEL downstream to a rolling release (CentOS Stream), eliminating the free RHEL-compatible stable release | Organizations running CentOS in production faced: pay for RHEL, migrate to Rocky Linux/AlmaLinux, or accept an unsupported OS. Migration effort varied — some organizations were on thousands of CentOS servers with no documented migration path. | OS-level decisions have a 5–10 year impact horizon. Distribution EOL events are predictable — track them. Avoid placing organizational dependency on a "free downstream of a commercial product" without a contingency plan. |
| **Splunk Pricing (ongoing)** | Data + Contractual | Splunk's ingest-based pricing model makes cost directly proportional to log volume. As organizations scale, Splunk cost scales proportionally. Some organizations have seen 5–10x cost increases over 3 years without architecture changes. | Organizations with significant data volumes face material cost pressure. Exit is difficult because Splunk has deep operational lock-in (custom dashboards, search queries in SPL, oncall runbooks). Many organizations negotiate aggressive volume discounts rather than exit. | Consumption-based pricing models can appear cheap at initial scale and become expensive at production scale. Model pricing at 2x, 3x, and 5x your initial volume before signing. Negotiate a volume discount that scales — or a pricing cap. |
