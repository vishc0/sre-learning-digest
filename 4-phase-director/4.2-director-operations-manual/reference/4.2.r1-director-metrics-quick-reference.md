# Director Metrics Quick Reference

One reference card for all metrics the Director actively tracks. Organized by review cadence. When a metric goes red, the action column tells you exactly what to do — not a general guideline.

**Legend**: All targets assume a mid-size enterprise operations organization running Tier 0/1 production services at scale (equivalent to T-Mobile/enterprise SLA commitments).

---

## Daily Metrics (Review Each Morning Before Standup)

| Metric | Target | Data Source | Director Action If Red |
|--------|--------|-------------|------------------------|
| SLO Burn Rate — Tier 0/1 | Error budget consumption < 5% per day (on 30-day rolling window) | Datadog SLO dashboards | Pull up real-time dashboard; call Platform Reliability Manager directly if Tier 0 service is burning > 10%/day. Do not wait for a P1 to be declared — the burn rate IS the signal. |
| P1 Open Incidents | 0 open P1s older than 4 hours | PagerDuty / ServiceNow | If any P1 is open > 4 hours without a clear ETA to resolution: join the bridge directly, ask for a SWARM status, assess whether you need to escalate to VP or pull in additional resources. |
| Overnight Change Success Rate | ≥ 97% of changes executed overnight completed without rollback | Change management tool (ServiceNow) | Review the failed changes by 8 AM. If a failed change produced a degraded service, confirm a P1 or P2 has been filed. Verify that the failed change's post-implementation review is scheduled within 24 hours. |
| Overnight On-Call Pages | ≤ 3 pages per on-call engineer overnight (10 PM–6 AM) | PagerDuty on-call report | If any engineer received > 6 pages overnight: check if it is a known noisy alert (tune it today) or a recurring issue (check the P1 problem backlog). Do not let a single engineer absorb > 8 overnight pages in any 7-day window without intervention. |

## Weekly Metrics (Review at Weekly Ops Review — Every Monday)

| Metric | Target | Data Source | Director Action If Red |
|--------|--------|-------------|------------------------|
| MTTR — Tier 1 (P1) | ≤ 30 minutes | ServiceNow incident reports / PagerDuty | If MTTR exceeded 30 min on any P1: read the incident timeline, find where time was lost (detection, escalation, bridge formation, or remediation), assign a problem ticket with a 14-day fix target to the owner of the slow step. |
| MTTR — Tier 2 (P2) | ≤ 6 hours | ServiceNow | If MTTR > 6 hrs on > 2 P2s in the same week: review for pattern (same service? same team? same time of day?). If pattern exists, create a P1 problem ticket for root cause investigation. |
| P1 Incident Count | ≤ 2 per week | ServiceNow | If week ends with ≥ 3 P1s: run a trend check — is this a spike or a worsening trend? If two consecutive weeks ≥ 3 P1s, escalate to VP and initiate a formal incident trend review with all domain managers. |
| P2 Incident Count | ≤ 6 per week | ServiceNow | If > 8 P2s in a week: check repeat rate. If > 30% are repeats, immediately prioritize the problem backlog in that week's problem review. If < 30% repeats, it may be noise — check if new deployments are the source. |
| Change Failure Rate (weekly) | ≤ 5% | ServiceNow change records | If CFR > 5% in any single week: pull the list of failed changes, categorize by type (normal / standard / emergency). If emergency changes have a disproportionate failure rate, tighten emergency change approval criteria immediately. |
| Open P1 Problems | ≤ 3 total open | ServiceNow problem records | If > 3 open P1 problems: review the backlog in the weekly ops review. Any problem open > 14 days needs a named fix date and an owner who is accountable in writing. Escalate any problem open > 30 days to VP with a recovery plan. |
| Environment MTTE (Mean Time to Environment) | ≤ 60 minutes | ServiceNow / internal environment tracker | If MTTE > 60 min for any environment request: check if it is an infra capacity issue, a process bottleneck, or a tooling failure. Assign investigation to Platform Reliability Manager with a 48-hour turnaround on root cause. |
| SLO Compliance — Tier 1 | ≥ 99.9% (weekly average) | Datadog SLO dashboards | If any Tier 1 service drops below 99.9% in any single week: check the error budget burn rate for the month. If the month-to-date budget is > 50% consumed, freeze all non-critical changes to that service and escalate to VP. |

## Monthly Metrics (Review in Monthly Ops Report — 5th Business Day)

| Metric | Target | Data Source | Director Action If Red |
|--------|--------|-------------|------------------------|
| Tier 0 Availability | ≥ 99.95% | Datadog SLO / availability reporting | If below 99.95%: check whether SLA breach threshold was crossed. If yes, initiate customer credit process and notify VP same day. If no, document in monthly report with specific root cause and remediation date. |
| Tier 1 Availability | ≥ 99.9% | Datadog SLO / availability reporting | SLA breach triggers customer notification within 24 hours (per contract). Director notifies VP before customer notification goes out. |
| Tier 2 Availability | ≥ 99.5% | Datadog SLO / availability reporting | Review breach cause; if repeated 2 months in a row, reclassify service risk and increase monitoring coverage. |
| Tier 3 Availability | ≥ 99.0% | Datadog SLO / availability reporting | Internal services only. If breach, check if it affected external-facing services (which would change tier classification). Report in monthly ops report. |
| Error Budget Consumption Rate | ≤ 50% of monthly budget consumed by midmonth | Datadog SLO dashboards | If > 70% consumed by end of Week 3: freeze all normal changes to that service. Convene Platform Reliability Manager + Change Manager to assess whether to burn or freeze the rest of the month. |
| DORA — Deployment Frequency | ≥ 40 deploys per week (org-wide) | Deployment pipeline / JIRA | If < 30/week: identify whether it is a tooling bottleneck, a change freeze, or a team capacity issue. The fix differs by cause — do not prescribe a solution before you know the cause. |
| DORA — Lead Time for Changes | ≤ 3 days (commit to production) | Deployment pipeline / JIRA | If > 5 days: review the change pipeline for bottlenecks. Common causes: CAB delay, environment unavailability, test gate failures. Assign investigation to Change Manager with a 1-week turnaround. |
| DORA — Change Failure Rate | ≤ 5% | ServiceNow / deployment pipeline | If > 5%: see weekly action above, applied to the month. Also check whether CFR spike is correlated with higher deployment frequency — acceleration sometimes introduces fragility. |
| DORA — MTTR | ≤ 30 minutes (Tier 1) | ServiceNow / PagerDuty | If monthly average > 30 min: see weekly MTTR action. At the monthly level, also check for outlier incidents skewing the average vs. systemic MTTR problems. |
| Problem Fix Delivery Rate | ≥ 2 P1 problems resolved per month | ServiceNow problem records | If 0 P1 problems resolved in any month: this is a red signal. Convene a problem backlog review within 5 days. Identify whether the blockers are engineering capacity, unclear ownership, or prioritization failure. Escalate to VP if two consecutive months at 0 resolved. |
| SLA Breach Count | 0 per month (Tier 0/1) | ServiceNow SLA tracking / contracts | Any SLA breach triggers: (1) VP notification within 4 hours of confirmed breach, (2) customer notification per contract terms, (3) RCA completed within 5 business days, (4) credits issued per contract schedule. |
| On-Call Load — Org Average | ≤ 6 pages/engineer/week | PagerDuty on-call summary | If any team averages > 8 pages/engineer/week for the month: this team is at attrition risk. Take immediate action: shared rotation coverage, toil reduction sprint, or headcount escalation. Do not wait 90 days to address. |
| Open Headcount | 0 positions open > 90 days | HRIS / HR Business Partner | If any approved headcount is open > 90 days: meet with HRBP to identify the blocker (sourcing, compensation, leveling). If blocker is systemic, escalate to VP. Unfilled headcount directly increases on-call load on existing team members. |

## Quarterly Metrics (Review at Quarterly Business Review — with VP)

| Metric | Target | Data Source | Director Action If Red |
|--------|--------|-------------|------------------------|
| Operational Excellence Score | ≥ 75/100 (weighted composite of MTTR, SLO compliance, CFR, DORA metrics, problem backlog health) | Internal scorecard / aggregated from ServiceNow + Datadog + PagerDuty | If < 65: present a recovery plan to VP within 2 weeks of QBR. Plan must include specific metric targets, owners, and a 90-day timeline. Do not present the score without the plan. |
| Repeat Incident Rate | ≤ 10% of total incidents are repeats of a known problem | ServiceNow (incidents cross-referenced against open problem tickets) | If > 15%: the problem management process is broken. Conduct a problem backlog audit within 30 days. Every repeat incident should have a corresponding problem ticket — if it does not, that is a process failure, not just a metric failure. |
| Toil Percentage | ≤ 30% of engineering time on toil (manual, repetitive, automatable work) | Team self-reported via quarterly survey + on-call log analysis | If > 40%: initiate a toil audit with each domain manager. Identify the top 3 toil sources. Prioritize automation in the next quarter's roadmap. If toil > 50%, escalate — the team is in maintenance mode and cannot invest in reliability improvements. |
| Architecture Compliance | ≥ 90% of production services compliant with the current architecture standard | Architecture review board findings / Architecture & Transformation Lead | If < 80%: convene Architecture Lead and VP Engineering. Non-compliant services are a security, reliability, and scalability risk. Develop a remediation roadmap with named owners per service. |
| Team Health Score | ≥ 70/100 (composite of on-call load, attrition rate, PTO utilization, engagement survey, headcount fill rate) | HR engagement survey + PagerDuty + HRIS | If < 60: this is a people emergency, not a metrics problem. Schedule a Director-level review of team conditions. Identify the top 2 drivers of the score and address them directly. A low team health score is a leading indicator of an availability incident — teams under stress make mistakes. |
| Transformation Velocity | ≥ 80% of quarterly roadmap milestones completed on time | JIRA roadmap / quarterly planning tracker | If < 60% of milestones completed on time: run a retrospective on what blocked delivery. Common causes: unplanned incident load, scope creep, dependency on external teams. Adjust the next quarter's roadmap based on what you learned, not based on optimism. |
