# Chapter 1: Foundations and Philosophy

## The Core Question: Why Are These Two Things Different?

An observability platform and a SIEM (Security Information and Event Management) system can ingest the same log stream, run on the same underlying infrastructure, and be operated by teams sitting in the same building — and still serve fundamentally different purposes. Understanding that distinction is the starting point for everything in this framework.

**An observability platform** answers the question: "Is the system behaving correctly, and if not, what changed?" The audience is an SRE or a platform engineer. The query pattern is: find the anomaly, trace it to a cause, restore service.

**A SIEM** answers the question: "Is someone doing something they shouldn't be doing, and can I prove it?" The audience is a security analyst or incident responder. The query pattern is: find the actor, trace the sequence of actions, preserve evidence, determine scope.

The data is often identical. The intent, the query structure, the retention requirement, and the escalation path are completely different.

This distinction matters because organizations that fail to hold both purposes in mind end up with one of two failure modes: they build an observability platform that is useless for security investigations because evidence was never preserved, or they build a SIEM that is useless for reliability work because the data is too old, too filtered, or too normalized to diagnose latency issues.

## How Splunk Serves Both

Splunk's architecture is unusually well-suited to this dual purpose, which is one of the reasons it dominates both markets in enterprise environments. The indexing layer, the search layer, and the data ingestion pipeline are shared infrastructure. What separates the observability use case from the security use case is not the underlying platform — it is the layer built on top.

Think of it this way. Your Splunk indexers receive a log stream from your API gateway. That same log stream is simultaneously useful to:

- An SRE running a search on `p99 latency by endpoint` to diagnose a slowdown
- A SOC analyst running a search on `authentication failures by source IP` to investigate credential stuffing
- A compliance auditor querying for `all accesses to the /admin endpoint in the last 90 days`

All three queries hit the same raw data. What differs is the lens: the index routing, the field extractions emphasized, the time window, the retention policy applied to that index.

Splunk Enterprise Security (ES) extends the base platform with a security-specific data model, a correlation search engine, and a risk scoring framework. It does not replace the observability layer — it runs alongside it on the same indexers. The SRE dashboards and the SOC dashboards share infrastructure. This is the architectural fact that enables unified platform strategy.

## The SRE-SOC Relationship: Three Models

How SRE teams and SOC teams relate to each other is an organizational design decision with significant operational consequences. There are three common models.

**Model 1: Siloed**
SRE and SOC are separate teams with separate tools, separate escalation paths, and no shared workflows. The handoff occurs at escalation time, when the SRE decides something looks like a security incident and pages the SOC. This is the most common model, and it produces the highest mean time to contain. The cost shows up during incidents: the SRE has already been working the problem for 45 minutes before the SOC knows it exists, evidence has been overwritten in the process of restoring service, and the SOC analyst spends another 30 minutes getting context that the SRE already has.

**Model 2: Embedded**
A SOC analyst sits within the SRE organization, often with a dual reporting line to the CISO. This produces the fastest detection-to-containment cycle because the security context is always present in the SRE incident response room. The tradeoffs are staffing (this is a hard skill combination to find), compliance (some frameworks require independence between the team operating the infrastructure and the team auditing it), and cost. Appropriate for high-security environments — financial services, defense, healthcare.

**Model 3: Federated**
SRE and SOC maintain separate teams and separate workflows, but share a data platform and shared playbooks. The Splunk deployment is shared — same indexers, potentially separate search heads or separate indexes — but correlation searches, notable events, and investigation workflows are owned by SOC while reliability dashboards and alert policies are owned by SRE. Handoff protocols are documented and drilled. This is the right model at enterprise scale, and it is almost certainly the model that applies to a T-Mobile-scale deployment.

The federated model works because it does not require co-location or co-reporting, but it does require explicit investment in shared data standards, a documented handoff protocol, and periodic joint exercises. Without those three things, federated becomes siloed with extra steps.

## Threat-Informed Reliability

There is a concept worth internalizing called threat-informed reliability. The idea is that threat intelligence should influence where you invest in reliability.

Here is the operational logic. If a threat intelligence feed tells you that your industry is being actively targeted by a threat actor that specializes in API gateway attacks — slow-response injection, credential stuffing, token replay — then your API gateway is not only a security risk. It is a reliability risk. That gateway becoming unavailable because it is under attack is an SRE problem. The SRE response (scaling, rate limiting, circuit breaking) and the SOC response (blocking, investigation, legal) need to be coordinated, not sequential.

Threat intelligence should inform your SLO prioritization, your runbook coverage, and your redundancy investments. A service that threat actors actively target needs higher reliability investment than a service nobody knows about, all else being equal.

This reframes security posture as a reliability input, not a separate concern.

## Why SRE Programs Fail at the SOC Interface

The failure modes are consistent across organizations:

**Vocabulary mismatch.** Both SRE and SOC use MTTD (mean time to detect) and MTTR. But MTTD means different things: SREs measure time from incident onset to alert firing; SOC measures time from attack initiation to first detection of the attack. These are not the same measurement. When the two teams sit down to review metrics, they are talking past each other.

**Different escalation paths.** SRE escalates up the reliability chain. SOC escalates through security, legal, and compliance. During a security incident, both escalation paths activate simultaneously, and nobody has agreed on who is the single incident commander. Chaos ensues.

**Different tool stacks.** Despite sharing Splunk in many organizations, the day-to-day tools differ: SRE uses Grafana, PagerDuty, Jira; SOC uses Splunk ES notable event queue, case management (ServiceNow, JIRA Service Management), and threat intelligence platforms. Integration between these tool stacks is almost always manual.

**Evidence destruction in the name of remediation.** The SRE instinct during an incident is: restore service. Restart the pod. Rebuild the node. Replace the corrupted service. Each of these actions destroys forensic evidence that the SOC needs. This conflict — SRE wants to restore, SOC wants to preserve — must be resolved in the handoff protocol before an incident occurs, not during one.

## Director Frame

A Director who can run a unified SRE and SOC data strategy — same Splunk deployment, separate index strategies, shared dashboards where appropriate, and a documented federated operating model — eliminates an entire category of organizational friction.

The specific friction eliminated: the incident that starts as a reliability event, turns out to be a security event after 90 minutes of SRE investigation, and by that point has no forensic evidence left because the pods were restarted. That failure mode is not a people problem. It is an architecture and protocol problem. It is solvable at the Director level. Most Directors have not solved it.

When asked in an interview how you think about the SRE-SOC interface, the answer is: "Federated model, shared data platform, separate workflows, documented handoff. The key investment is in the protocol, not the tooling. The tooling is already shared — what's missing is the operating agreement."
