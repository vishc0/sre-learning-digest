## Purpose

This playbook defines the procedure for transitioning from an SRE-led reliability incident response to a parallel SRE + SOC security incident response. It defines the trigger conditions, the pre-handoff actions SRE must take, the handoff checklist, and the parallel workstream structure for the combined response.

This playbook does not replace incident command training. It supplements it with security-specific operating agreements that must exist before any real incident requires them.

---

## Escalation Criteria

The following three conditions trigger a security escalation. **Any one condition is sufficient.** Do not wait for multiple conditions to be confirmed before escalating.

### Trigger 1: Evidence of Unauthorized Access

Any of the following constitutes evidence:
- Authentication success events from a credential that has been reported compromised by the account owner
- Successful login from a geographic location not consistent with the user's established pattern (requires identity enrichment in Splunk)
- Successful login from a source IP appearing on a current threat intelligence feed
- A service account performing interactive logins (service accounts should not authenticate interactively)
- Access to a system or resource outside the user's or service account's established access pattern, confirmed by RBAC or IAM policy review

**Action:** Page SOC immediately. Begin pre-handoff actions while SOC is spinning up.

### Trigger 2: Evidence of Data Manipulation

Any of the following constitutes evidence:
- Write operations to a data store that is designated read-only in the system's architecture
- Bulk delete events not associated with a scheduled maintenance window or known change request
- Schema changes (DDL operations) outside a defined change window
- Configuration changes to security-relevant systems (RBAC configurations, network policies, secret store policies, firewall rules) without a corresponding approved change ticket
- Modification or deletion of audit log data

**Action:** Page SOC immediately. Do not take any remediating action until SOC has assessed the scope — remediation may overwrite evidence.

### Trigger 3: Evidence of Lateral Movement

Any of the following constitutes evidence:
- A pod or service making network connections to internal or external destinations that have no historical precedent (first-seen analysis)
- Port scanning activity originating from internal systems
- DNS queries from pods to domains that have never appeared in historical DNS logs for that workload
- A service account calling Kubernetes API resources it has no documented need to access
- Process execution on a node that is inconsistent with the node's declared workload (detected via Falco or auditd)

**Action:** Page SOC immediately. Apply network isolation to the affected pod or system without restarting it if service permits.

## Pre-Handoff SRE Actions

Complete these actions before or simultaneously with paging SOC. Time elapsed from trigger detection to SOC page should not exceed 10 minutes.

### Step 1: Document the Initial Timeline

Open the incident timeline document and record:
- Time the anomaly was first observed
- Which alert or dashboard surfaced the anomaly
- Time the trigger condition was confirmed
- Initial description of the anomaly (what is observed, not what is concluded)

### Step 2: Inventory Affected Systems

List all systems currently believed to be involved:
- Pod names and namespaces
- Node IPs
- Service accounts involved
- External IP addresses or domains involved
- Data stores that may have been accessed

Be explicit about the scope of your knowledge: "confirmed involved" vs. "potentially involved."

### Step 3: Capture Current State

Before taking any remediation action, capture state:
- `kubectl get pods -A -o wide` — current pod state across all namespaces
- `kubectl get events --sort-by=.metadata.creationTimestamp -A` — recent events
- Network connection state from affected pods (if accessible): `kubectl exec <pod> -- ss -tulpn`
- Current RBAC bindings for implicated service accounts: `kubectl get rolebindings,clusterrolebindings -A -o yaml`
- The Splunk search link that surfaces the triggering events — copy and preserve the URL

Store these in the incident evidence folder: `[S3 or shared drive path]/<incident-id>/initial-state/`

### Step 4: Assess Containment Options

Determine whether any containment action can be taken that preserves forensic evidence:
- Can a NetworkPolicy be applied to restrict the affected pod's egress without restarting it?
- Can the affected pod be removed from the service load balancer while keeping it running?
- Can traffic be routed around the affected system without touching the system itself?

If containment is possible without restarting the pod, apply containment and document the action.

If service restoration requires restarting the pod: **stop, and wait for SOC input** before restarting. If the service is fully unavailable and business impact demands immediate restart, notify SOC of the restart action before executing it, and capture a memory dump first if at all possible.

### Step 5: Page SOC

Page the SOC on-call using the defined escalation path: [insert escalation path]

Provide in the page:
- One-line description of the trigger condition observed
- Number of systems believed to be affected
- Current service status (degraded, unavailable, or fully operational)
- Link to the incident timeline document

## Parallel Workstream Structure

Once SOC is active, the incident operates in two simultaneous workstreams. Both run until independently closed.

### SRE Workstream: Service Restoration

**Owner:** SRE Lead  
**Scope:**
- Contain blast radius — route traffic around affected systems, apply network isolation
- Maintain service availability for unaffected platform components
- Provision log and system access to SOC on request (do not wait to be asked — proactively offer)
- Document all actions taken in the shared timeline with exact timestamps
- **Constraint:** Do not take any action that destroys forensic evidence without SOC awareness and approval

**Update cadence:** Post a status update in the shared incident channel every 30 minutes: current service status, actions taken since last update, next planned action.

### SOC Workstream: Security Investigation

**Owner:** SOC Lead  
**Scope:**
- Collect and preserve forensic evidence (memory dumps, filesystem snapshots, network state)
- Determine the full scope of systems affected and data potentially accessed
- Identify the initial access vector and construct the attack timeline
- Assess data exposure scope for compliance and legal notification purposes
- Manage external communication: legal, compliance, regulators, PR (do not route these through SRE)
- Direct credential rotation and eradication actions (SRE executes; SOC directs timing)

**Update cadence:** Post investigation status in the shared incident channel every 30 minutes: current scope assessment, key findings, next investigation actions.

### Incident Commander

The Incident Commander for a security incident is: **[SOC Lead by default / escalate to CISO if data breach confirmed]**

The Incident Commander makes all cross-workstream decisions, including the conflict resolution between SRE restoration actions and SOC evidence preservation requirements.

## Evidence Preservation Rules

These are absolute rules during a security incident. Violating them may compromise the investigation and create legal liability.

**Do not restart compromised pods** without SOC awareness. If restart is required for service restoration:
1. Notify SOC before restarting.
2. Capture memory dump (using `crictl` or `kubectl debug`) if time permits.
3. Document the restart action in the incident timeline with timestamp and rationale.

**Do not delete log data** for any reason during the incident. If a pipeline issue is causing log duplication or errors, fix the pipeline but preserve the source data.

**Do not rotate credentials without SOC coordination.** Credential rotation destroys the ability to trace where stolen credentials were used. SOC must direct the timing and sequence of credential rotation.

**Do not apply patches or update container images** on systems believed to be compromised. Patching overwrites exploit artifacts.

**Do not restore from backup** without SOC review of the backup. The backup may be compromised.

## Escalation Beyond SOC

### Legal Hold
Trigger: Any indication that a breach has occurred, data has been exfiltrated, or litigation may result.

Action: SOC notifies legal counsel within 1 hour of confirming a breach. Once legal hold is declared by legal counsel, all relevant data must be preserved indefinitely until legal counsel releases the hold. This overrides any scheduled data deletion or log rotation.

### Regulatory Notification
Trigger: Confirmed breach involving personal data.

| Framework | Notification Deadline | Notify Whom |
|-----------|----------------------|-------------|
| GDPR | 72 hours from awareness | Data Protection Authority |
| HIPAA | Within 60 days of discovery | HHS + affected individuals if > 500 records |
| PCI-DSS | Immediately upon discovery | Acquiring bank |
| State breach laws | Varies by state | State AG + affected residents |

SOC owns the notification decision with legal counsel. SRE provides system and data scope information to support the notification. SRE does not communicate externally about a security incident without SOC and legal approval.

### Law Enforcement
Trigger: Ransomware, confirmed nation-state activity, significant fraud.

SOC engages law enforcement with legal counsel approval. Evidence handling requirements change significantly — evidence must now meet chain-of-custody standards. SOC will provide specific instructions to SRE for handling systems that are now potential evidence in a criminal investigation.

## Post-Handoff SRE Role

After the initial handoff, SRE continues in a support role for the duration of the security incident:

- Maintain a dedicated SRE contact for SOC access requests throughout the incident
- Provision log searches, system access, and network telemetry as requested by SOC
- Continue monitoring overall platform health for secondary incidents (attackers sometimes initiate a distracting reliability incident while conducting their actual attack elsewhere)
- Participate in the post-incident review with both a reliability RCA and a coordination retrospective — what worked in the SRE-SOC handoff, what did not
