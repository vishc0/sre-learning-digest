## Purpose

This playbook provides a structured investigation procedure for suspicious Kubernetes audit log activity. It is invoked when an SRE observes unusual RBAC activity, when a Splunk ES notable event fires on a Kubernetes audit log correlation search, or when SOC escalates to SRE for platform context on a K8s audit finding.

The playbook walks through the audit log anatomy, provides a step-by-step investigation sequence, includes working SPL queries for each step, and defines escalation criteria.

---

## Triggers for This Playbook

Invoke this playbook when any of the following occur:

- A Splunk ES notable event fires for a rule targeting K8s audit logs (RBAC mutation, secret access, exec events)
- An SRE observes RBAC-related audit events outside a change window during normal platform monitoring
- SOC requests platform access to support a K8s-related security investigation
- Falco or Tetragon fires an alert that implicates a specific pod or service account
- An ESCU correlation search on `index=k8s_audit` produces output with suspicious actor patterns

## Kubernetes Audit Log Anatomy

Before investigating, understand the structure of a K8s audit event. Every event has:

**Stages:** The lifecycle stage at which the event was recorded.
- `RequestReceived`: The API server received the request. Logged before authentication/authorization.
- `ResponseStarted`: The response headers were sent. Logged for long-running requests (watches).
- `ResponseComplete`: The full response was returned. This is the most common stage for point-in-time queries.
- `Panic`: The API server panicked handling the request.

For most investigations, focus on `ResponseComplete` events — these represent actions that were both authorized and executed.

**Key fields in every audit event:**

| Field | Description | Investigation Value |
|-------|-------------|-------------------|
| `user.username` | Who made the call. Can be a human user (OIDC), service account, or system component. | Primary actor field |
| `user.groups` | Which groups the caller belongs to | Authority level of the actor |
| `userAgent` | What client was used (kubectl, client library, API call) | Identifies tooling; unusual user agents indicate scripted access |
| `verb` | What action was requested (get, list, create, update, patch, delete, watch, exec) | The action type |
| `objectRef.resource` | What resource type (pods, secrets, configmaps, roles, etc.) | The target resource type |
| `objectRef.namespace` | Which namespace | Scope of the action |
| `objectRef.name` | Specific resource name | Specific target |
| `objectRef.subresource` | Sub-resource (exec, log, portforward) | Critical for detecting interactive access |
| `responseStatus.code` | HTTP response code | 200/201 = success; 403 = denied |
| `sourceIPs` | Source IP of the request | Identifies where the request originated |
| `requestURI` | Full URI of the request | Useful for identifying unusual API paths |

## Investigation Steps

### Step 1: Establish the Time Window

The audit log investigation is only useful relative to other signals. Before running audit queries, establish:

- What time did the initial anomaly appear? (alert time, anomaly detection time)
- What is the investigation window? Typically: 2 hours before the anomaly to current time.
- Are there any correlated events? Deployment times, change tickets, other alerts in the same window?

Document the time window in the incident record before proceeding.

### Step 2: Identify the Actor

Find who was active in the audit log during the investigation window. Filter out known system components to focus on human users and non-system service accounts.

```spl
index=k8s_audit 
    earliest=[start_time] latest=[end_time]
    stage=ResponseComplete
    user.username!="system:*"
    user.username!="kubernetes-admin"
| stats count AS action_count 
    BY user.username, verb, objectRef.resource, objectRef.namespace
| sort -action_count
```

Review the top actors by action count. Anything unexpected — an unfamiliar service account name, a human user active at an unusual hour, high action counts on security-sensitive resources — is a follow-up lead.

### Step 3: Look for Privilege Escalation

Privilege escalation in Kubernetes means gaining additional access rights. The specific audit events to look for:

```spl
index=k8s_audit 
    earliest=[start_time] latest=[end_time]
    stage=ResponseComplete
    (objectRef.resource=roles OR objectRef.resource=clusterroles 
     OR objectRef.resource=rolebindings OR objectRef.resource=clusterrolebindings)
    (verb=create OR verb=update OR verb=patch OR verb=delete)
    responseStatus.code=200
| table _time, user.username, verb, objectRef.resource, objectRef.namespace, objectRef.name
| sort _time
```

Any creation or modification of RBAC resources outside of a change window is a finding. Escalate if:
- The actor is a service account (service accounts should not create RBAC bindings)
- The binding grants ClusterAdmin or wildcards on sensitive resources
- The actor is a human user who is not a platform administrator

### Step 4: Look for Secret Access

Secrets in Kubernetes contain credentials, API keys, and TLS certificates. Adversaries who have a foothold in your cluster will enumerate and read secrets.

```spl
index=k8s_audit 
    earliest=[start_time] latest=[end_time]
    stage=ResponseComplete
    objectRef.resource=secrets 
    (verb=get OR verb=list)
    responseStatus.code=200
    user.username!="system:*"
| stats count AS secret_reads, 
         values(objectRef.name) AS secrets_accessed, 
         values(objectRef.namespace) AS namespaces
    BY user.username
| sort -secret_reads
```

Normal patterns: a controller reading its own namespace's secrets, a deployment reading a specific secret it is configured to use.

Suspicious patterns: a service account reading secrets across multiple namespaces, a high count of secret reads in a short window, a service account reading secrets in namespaces it is not deployed in.

### Step 5: Look for Interactive Access to Pods

`exec` and `portforward` operations establish interactive sessions inside running pods. Legitimate uses exist (debugging), but they are rare in production and should be change-managed. Adversaries with cluster access will use `exec` to gain shell access.

```spl
index=k8s_audit 
    earliest=[start_time] latest=[end_time]
    stage=ResponseComplete
    objectRef.resource=pods
    (objectRef.subresource=exec OR objectRef.subresource=portforward)
    responseStatus.code=101
| table _time, user.username, objectRef.namespace, objectRef.name, 
         requestObject.command, sourceIPs
| sort _time
```

Review every exec event. For each:
- Was there a change ticket or maintenance window covering this action?
- Is the user account one with documented need for pod exec?
- What command was run (if captured in the audit log at RequestResponse audit level)?
- Does the pod being exec'd into have any privileged capabilities?

An exec event without a corresponding change ticket is a finding requiring explanation.

### Step 6: Correlate Audit Events with Pod Events

Suspicious audit events that precede pod behavior changes are a strong indicator of adversarial activity. Correlate the audit timeline with pod restart events, new pod creation, and resource quota changes.

```spl
index=k8s_audit OR index=k8s_events
    earliest=[start_time] latest=[end_time]
| eval source_type=if(index="k8s_audit", "audit", "event")
| eval actor=coalesce(user.username, involvedObject.name)
| eval action=coalesce(verb, reason)
| eval target=coalesce(objectRef.name, involvedObject.name)
| table _time, source_type, actor, action, target, objectRef.namespace
| sort _time
```

Look for temporal clustering: audit events from a suspicious actor followed within minutes by pod restart events or new workload creation in the same namespace.

## Splunk SPL Queries — Quick Reference

### Query 1: Actor Inventory for Investigation Window

```spl
index=k8s_audit stage=ResponseComplete earliest=-4h
    user.username!="system:*"
| stats count BY user.username, verb, objectRef.resource
| sort -count
```

### Query 2: RBAC Mutations (Privilege Escalation Detection)

```spl
index=k8s_audit stage=ResponseComplete earliest=-24h
    objectRef.resource IN (roles, clusterroles, rolebindings, clusterrolebindings)
    verb IN (create, update, patch, delete)
    responseStatus.code=200
| table _time, user.username, verb, objectRef.resource, objectRef.namespace, objectRef.name
```

### Query 3: Secret Access Pattern

```spl
index=k8s_audit stage=ResponseComplete earliest=-24h
    objectRef.resource=secrets verb IN (get, list) responseStatus.code=200
    user.username!="system:*"
| stats count AS reads, values(objectRef.namespace) AS namespaces 
    BY user.username
| sort -reads
```

### Query 4: Interactive Pod Access

```spl
index=k8s_audit stage=ResponseComplete earliest=-24h
    objectRef.subresource IN (exec, portforward) responseStatus.code=101
| table _time, user.username, objectRef.namespace, objectRef.name, sourceIPs
```

### Query 5: Service Account Activity Baseline Deviation

```spl
index=k8s_audit stage=ResponseComplete earliest=-1h
    user.username="system:serviceaccount:*"
| rex field=user.username "system:serviceaccount:(?<sa_namespace>[^:]+):(?<sa_name>.+)"
| stats count BY sa_name, sa_namespace, verb, objectRef.resource
| where NOT (sa_name IN ("known-sa-1", "known-sa-2") 
             AND objectRef.resource IN ("configmaps", "endpoints"))
| sort -count
```

## Escalation Decision

### Escalate immediately to SOC if:
- Any RBAC mutation was performed by a service account (service accounts should not create RBAC bindings)
- Secret access is occurring across namespaces the actor has no documented need to access
- Pod exec events exist without corresponding change tickets
- The actor identified is a service account belonging to a workload known to be exposed externally
- Multiple investigation steps (secret access + RBAC mutation + exec) involve the same actor

### Continue internal investigation without immediate escalation if:
- All RBAC mutations correspond to open, approved change tickets
- Secret access is limited to the actor's own namespace and within expected patterns
- Exec events are covered by maintenance windows

### Close investigation if:
- All audit events are explainable by scheduled maintenance, known deployments, or approved change activity
- No anomalous actors, no privilege escalation, no secret access outside documented patterns

## Remediation Actions

**Revoke a compromised service account:**

```bash
# Delete the service account (revokes all tokens associated with it)
kubectl delete serviceaccount <sa-name> -n <namespace>

# Recreate with minimal permissions
kubectl apply -f <sa-manifest-with-corrected-rbac>

# Audit all pods using the old service account and rotate their workloads
kubectl get pods -A -o json | jq '.items[] | select(.spec.serviceAccountName=="<sa-name>") | .metadata.name'
```

**Remove an unauthorized RBAC binding:**

```bash
# Identify the binding
kubectl get rolebindings,clusterrolebindings -A -o json | jq '... | select(.subjects[].name=="<actor>")'

# Delete with audit trail
kubectl delete rolebinding <binding-name> -n <namespace>
# or
kubectl delete clusterrolebinding <binding-name>
```

**Coordinate credential rotation with SOC before executing.**

## Audit Log Audit Policy Reference

The following audit policy captures the events needed for this investigation. Apply to the API server with `--audit-policy-file`:

```yaml
apiVersion: audit.k8s.io/v1
kind: Policy
rules:
  # Log all secret access at RequestResponse level
  - level: RequestResponse
    resources:
    - group: ""
      resources: ["secrets"]
  
  # Log all RBAC mutations
  - level: Request
    resources:
    - group: "rbac.authorization.k8s.io"
      resources: ["roles", "clusterroles", "rolebindings", "clusterrolebindings"]
    verbs: ["create", "update", "patch", "delete"]
  
  # Log all exec and portforward
  - level: Request
    resources:
    - group: ""
      resources: ["pods/exec", "pods/portforward"]
  
  # Log all service account token creates
  - level: Metadata
    resources:
    - group: ""
      resources: ["serviceaccounts/token"]
  
  # Log everything else at Metadata level
  - level: Metadata
```
