## Purpose

This playbook standardizes the process for onboarding a new log source to the shared Splunk deployment. Ad-hoc log onboarding — connecting a new source without following this process — produces CIM non-compliance (breaking security correlation searches), index sprawl (data in wrong retention tiers), compliance gaps (data not retained per applicable framework), and unknown data volume (causing unexpected Splunk license overages).

Every new log source must complete all steps in this playbook before being declared "onboarded." No exceptions.

---

## Why This Needs a Process

Splunk's flexibility is its strength and its risk. Any forwarder can send data to any index. Data will arrive, be indexed, and appear searchable. Nothing in the platform prevents a new log source from being onboarded incorrectly — no error fires, no warning appears. Silent misconfiguration is the failure mode.

The specific failure patterns this process prevents:

- **Wrong source type:** Logs arrive with an incorrect `sourcetype`, causing the wrong field extractions to be applied. The data appears but key fields are null. Security correlation searches that depend on those fields silently miss events.
- **Wrong index:** Logs land in the default `main` index with a 30-day retention policy when the applicable compliance framework requires 12 months. No alert fires. The compliance gap is discovered during an audit.
- **Missing CIM mapping:** The Technology Add-on for the source type was not installed or configured. The data cannot be joined with other sources in a datamodel-based correlation search.
- **Volume surprise:** The new source generates 5x the estimated volume, causing an unexpected spike in Splunk indexing volume and license consumption.

## Step 1: Source Classification

Before any technical work begins, answer the following classification questions:

**1.1 Is this a security-relevant log source?**

A log source is security-relevant if it records any of the following: authentication events, authorization decisions, privilege changes, network connections, system calls, file access, configuration changes to security controls, or administrative actions.

Security-relevant sources require: longer retention than observability-only sources, CIM compliance validation, routing to a security index, and notification to SOC at onboarding completion.

**1.2 Does this source satisfy a compliance requirement?**

Review the Compliance Log Inventory (see `templates/compliance-log-inventory-template.md`) to determine if the applicable frameworks require this source. If yes: the source requires compliance-grade retention (see Step 3), immutable archival storage, and documentation in the compliance log inventory.

**1.3 Source classification decision:**

| Classification | Criteria | Index Tier |
|---------------|---------|-----------|
| Observability Only | No security or compliance relevance | observability index, short retention |
| Security | Security-relevant, no specific compliance requirement | security index, 12-month retention |
| Compliance | Satisfies a specific compliance framework requirement | compliance index, framework-defined retention + immutable archival |
| Both (common) | Security-relevant AND compliance-required | compliance index with security CIM mapping |

## Step 2: Source Type and Parsing

**2.1 Does a Splunk Technology Add-on (TA) exist for this source?**

Check Splunkbase: https://splunkbase.splunk.com

Search for the vendor and product name. If a TA exists:
- Review the TA documentation for supported Splunk versions
- Verify the TA provides CIM-compliant field extractions for the relevant datamodels
- Install the TA on all search heads AND on the indexers (for index-time field extractions)

If no TA exists:
- Determine if a generic TA (e.g., Splunk Add-on for Unix and Linux for syslog sources) applies
- If custom parsing is required, build a custom TA with `props.conf` and `transforms.conf` — do not put custom configurations directly in the Splunk default install

**2.2 CIM compliance pre-check**

Identify which CIM datamodels this source should populate:

| Source Type | Primary Datamodel |
|-------------|-----------------|
| Authentication logs | Authentication |
| Network flow | Network Traffic |
| DNS logs | Network Resolution |
| Process execution | Endpoint.Processes |
| File changes | Endpoint.Filesystem |
| System changes | Change |
| IDS/IPS alerts | Intrusion Detection |
| Kubernetes audit | Change + Authentication (partial) |

Verify that the TA provides field mappings for the relevant datamodel by reviewing the TA's `eventtypes.conf` and `tags.conf`.

## Step 3: Index Routing Decision

Route the log source to the appropriate Splunk index based on the Step 1 classification.

**3.1 Naming convention:**

| Index Name | Purpose | Retention (Hot) | Retention (Total) |
|-----------|---------|----------------|-----------------|
| `obs_<source>` | Observability-only sources | 30 days | 30 days |
| `sec_<source>` | Security-relevant, non-compliance | 90 days | 12 months (SmartStore) |
| `compliance_<framework>` | Compliance-required sources | 90 days | Framework-defined (see Step 6) |

If the source serves both security and compliance purposes, route to the compliance index (more restrictive retention applies).

**3.2 Index configuration:**

Create the index in `indexes.conf` if it does not exist. Set:
- `maxTotalDataSizeMB`: based on volume estimate from Step 4
- `frozenTimePeriodInSecs`: compliance retention period in seconds
- `coldPath`: SmartStore S3 bucket path (if using SmartStore for warm tier)

## Step 4: Volume Estimation and Cost Approval

**4.1 Estimate daily ingestion volume**

Methods for estimation:
- Collect a 24-hour sample from the source system before onboarding to Splunk
- Query existing logs if the source has a local log file: `wc -c <logfile>` per day
- Use vendor documentation for typical event rates per host and estimate from your host count

Express the estimate as: GB per day.

**4.2 Splunk license impact**

At the current Splunk license rate of [$ per GB/day — fill in]:

```
New source volume (GB/day) × days in billing period × license rate = incremental monthly cost
```

**4.3 Approval gate**

| Daily Volume | Approval Required |
|-------------|-----------------|
| < 10 GB/day | Platform SRE Lead |
| 10–100 GB/day | SRE Director |
| > 100 GB/day | SRE Director + Finance approval |

Do not proceed to Step 5 without documented volume approval.

## Step 5: Parsing Validation

After initial data is flowing, validate that parsing is correct before declaring the source production-ready.

**5.1 Timestamp parsing validation**

```spl
index=<target-index> sourcetype=<new-sourcetype> earliest=-1h
| eval time_skew=abs(now() - _time)
| stats max(time_skew) AS max_skew_seconds, avg(time_skew) AS avg_skew_seconds
| where max_skew_seconds > 300
```

If `max_skew_seconds > 300`, the timestamp is not being parsed correctly. The events are being indexed with the current time rather than the event time. This makes historical queries useless. Fix `TIME_PREFIX`, `TIME_FORMAT`, and `MAX_TIMESTAMP_LOOKAHEAD` in `props.conf` before proceeding.

**5.2 Field extraction validation**

```spl
index=<target-index> sourcetype=<new-sourcetype> earliest=-1h
| stats count, dc(host), 
         count(eval(isnotnull(user))) AS has_user,
         count(eval(isnotnull(src))) AS has_src,
         count(eval(isnotnull(action))) AS has_action
```

If key fields (user, src, action for authentication sources) have null values for > 5% of events, the field extractions are not working correctly.

**5.3 CIM datamodel population validation**

```spl
| datamodel <DatamodelName> search
| search sourcetype=<new-sourcetype>
| stats count BY sourcetype
```

If count is zero, the CIM mapping is not working. Review `eventtypes.conf` and `tags.conf` in the TA.

## Step 6: Retention Policy Configuration

Configure retention in `indexes.conf` based on the compliance classification from Step 1:

| Framework | Hot Retention | Warm (SmartStore/S3) | Cold (Glacier) | Immutable Required |
|-----------|-------------|----------------------|----------------|-------------------|
| SOC 2 Type II | 90 days | 12 months | — | Yes |
| PCI-DSS v4.0 | 90 days | 9 additional months | — | Yes |
| HIPAA | 90 days | 12 months | 5 additional years | Recommended |
| FedRAMP Moderate | 90 days | 9 additional months | — | Yes |
| Observability only | 30 days | — | — | No |

**Immutable archival setup (compliance sources):**

Configure a parallel write to S3 with Object Lock at the Heavy Forwarder level. Events simultaneously stream to:
1. Splunk indexer (for search and correlation)
2. S3 bucket with Object Lock enabled (for compliance-grade immutable archival)

Verify the S3 write is active before declaring compliance onboarding complete.

---

## Step 7: Alert Coverage Review

After the source is validated and retention is configured, assess whether the new data source should trigger additional Splunk ES coverage.

**7.1 ESCU content review**

Identify ESCU content packs that apply to this new source type. Search the ESCU content library (Splunk ES > Content > Content Management) filtering by:
- Source type
- Applicable MITRE ATT&CK technique

Enable applicable ESCU content packs against your environment. Validate that they produce expected results (or zero results for clean environments — zero notables is acceptable, zero events in the search is not).

**7.2 Existing correlation search review**

Check whether any existing correlation searches reference this source type or the datamodels it populates. If so, verify that the new data is being picked up correctly by the correlation searches.

**7.3 Custom correlation search need assessment**

If this is a unique source type for which no ESCU content exists, assess whether a custom correlation search should be developed. This is a SOC decision with platform team support for SPL development.

## Step 8: Documentation and SOC Handoff

Before declaring onboarding complete, produce:

**8.1 Source documentation:**
- Source name and description
- Source type
- Collection method (forwarder, HEC, syslog)
- Index name
- Applicable compliance frameworks
- Retention policy (hot / warm / cold / immutable)
- CIM datamodels populated
- Responsible team for source maintenance
- Known parsing limitations or gaps

**8.2 Update the Compliance Log Inventory** (see `templates/compliance-log-inventory-template.md`) with the new source.

**8.3 SOC notification:**

- New source name and type
- Index name and search access instructions
- CIM datamodels populated
- Applicable ESCU content enabled
- Contact for platform questions about this source

**8.4 Onboarding sign-off checklist:**

- [ ] Source classification completed (observability / security / compliance)
- [ ] TA installed on search heads and indexers
- [ ] CIM field mapping validated (no null key fields)
- [ ] Timestamp parsing validated (skew < 300 seconds)
- [ ] Index created with correct retention policy
- [ ] Volume approved by appropriate authority
- [ ] Immutable S3 archival configured (compliance sources only)
- [ ] ESCU content reviewed and applicable rules enabled
- [ ] Compliance Log Inventory updated
- [ ] SOC notification sent
- [ ] Source documentation completed

## Common Failure Modes

| Failure | Symptom | Diagnosis | Fix |
|---------|---------|-----------|-----|
| Wrong source type | Fields are null despite events arriving | `index=<idx> | stats count by sourcetype` shows unexpected sourcetype | Correct `sourcetype` in `inputs.conf` on the forwarder |
| Timestamp parsing failure | All events appear to arrive simultaneously at current time | Check `_indextime` vs `_time` — they will match when parsing fails | Fix `TIME_PREFIX` / `TIME_FORMAT` in `props.conf` |
| Missing CIM mapping | `\| datamodel Authentication search` returns 0 for new source | `eventtypes.conf` or `tags.conf` missing in TA | Add correct eventtype definition and tags |
| Wrong index | Data in `main` or unexpected index | `index=main sourcetype=<new-type> \| stats count` returns results | Update `_TCP_ROUTING` or `index` setting in `inputs.conf` |
| Volume surprise | License alert fires | Query volume: `index=<idx> sourcetype=<type> \| stats count, sum(len(_raw)) as bytes by host` | Filter noise at forwarder before reaching Splunk; revisit volume estimate |
| Missing immutable write | Compliance gap discovered in audit | Check S3 bucket write confirmation logs at Heavy Forwarder | Verify `syslog-ng` or HEC secondary output to S3 is active |
