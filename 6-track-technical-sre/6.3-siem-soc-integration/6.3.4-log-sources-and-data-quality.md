# Chapter 4: Log Sources and Data Quality

## Why Data Quality Is the Unglamorous Prerequisite

Every SIEM project that fails does so for the same reason: the log sources are unreliable, not because the correlation searches are wrong. You can write the most sophisticated detection logic in SPL, tune risk scores precisely, and map everything to MITRE ATT&CK — and if the authentication logs have a 2% gap rate, or the Kubernetes audit logs are missing exec events because the audit policy was misconfigured, your SIEM produces false negatives. Attackers who know your logging gaps will route their activity through them.

This is structurally different from observability data quality. An observability platform can tolerate a 1% log loss rate. If 99% of your requests are represented in your latency histograms, you have an accurate performance picture. A SIEM cannot tolerate this. A missing log segment is not a quality issue — it is a finding. An auditor who sees a gap in your authentication logs for a three-hour window will assume that window was used to cover activity. The burden of proof is on you to demonstrate the gap was technical, not adversarial.

This chapter covers which log sources matter, why Kubernetes audit logs are the most underused security asset most platform teams already have, and what SIEM-grade data quality actually requires.

## The Priority Stack: Which Log Sources Matter

Not all logs are equally valuable for security purposes. Priority order:

**Tier 1 — Must have, no exceptions:**

1. **Authentication logs**: Every login, logout, failure, MFA event, SSO assertion, and token issuance. This is your north star for determining who accessed what and when. Sources: Active Directory/LDAP, Okta/PingFederate/Azure AD, application-level auth endpoints, Kubernetes service account token issuances.

2. **Kubernetes API audit logs**: Every call to the Kubernetes API server is logged with actor, verb, resource, namespace, and response code. This includes `kubectl exec`, pod creation, secret access, RBAC mutations, and configmap changes. For a Kubernetes-native platform, this is the most security-dense log source you have.

3. **API gateway logs**: Your north-south boundary. Every external request hitting your platform, including authentication status, rate limit decisions, and response codes. If an attacker is probing your API, it shows here first.

**Tier 2 — High value, collect where feasible:**

4. **Network flow logs**: TCP/UDP connection metadata (source IP, destination IP, port, bytes, duration). Not packet contents — just flow records. Sufficient for lateral movement detection and exfiltration volume analysis. Sources: VPC flow logs, Kubernetes network policy logs (Calico, Cilium), service mesh telemetry.

5. **Process execution logs**: What binaries run on your nodes. In a Kubernetes environment, this is particularly valuable because any process running outside a container runtime, or any unexpected binary inside a container, is immediately suspicious. Sources: Linux auditd, Falco, Tetragon (eBPF-based, lower overhead than auditd).

6. **DNS logs**: DNS queries from your workloads to external domains. Lateral movement and command-and-control communications often use DNS. Unusually long DNS query strings indicate DNS tunneling. Sources: CoreDNS query logs, VPC DNS resolver logs, or a dedicated DNS security solution.

**Tier 3 — Compliance-driven, collect when required:**

7. **Database audit logs**: Query logs, privilege escalation events, DDL changes. Required by PCI-DSS for cardholder data environments.

8. **Cloud provider audit logs**: AWS CloudTrail, GCP Cloud Audit Logs, Azure Activity Logs. Every IAM change, every S3 bucket policy change, every security group modification.

9. **Host-level audit logs**: Linux auditd on non-containerized systems. Syscall-level visibility for bare-metal or VM workloads.

## Kubernetes Audit Logs: The Most Underused Security Asset

Kubernetes audit logs deserve their own treatment because they are simultaneously the highest-value security data source in a Kubernetes platform and the one most likely to be misconfigured or absent.

Every call to the Kubernetes API server passes through the audit subsystem. This includes:

- `kubectl exec` into a pod (verb: `create`, resource: `pods/exec`)
- Reading a secret (verb: `get`, resource: `secrets`)
- Creating a rolebinding (verb: `create`, resource: `rolebindings`)
- Scaling a deployment (verb: `update`, resource: `deployments/scale`)
- Accessing the API server from a service account running inside a pod

The audit log records: who made the call (user or service account), what they requested (verb + resource + namespace), what the API server returned (response code, response object if configured), and when.

The critical configuration point: audit logs are only as useful as the audit policy that governs them. A default or minimal audit policy logs almost nothing useful. A security-grade audit policy logs at minimum:

- All secret access at `RequestResponse` level (so you see the response, not just the request)
- All RBAC mutations (roles, clusterroles, rolebindings, clusterrolebindings) at `Request` level
- All pod exec and portforward operations at `Request` level
- All service account token requests at `Metadata` level
- All other API calls at `Metadata` level

Without the audit policy configured correctly, you have an audit log that records that `kubectl` was run but not what it did. That is not useful for security.

## API Gateway Logs: The North-South Boundary

The API gateway is where external traffic enters your platform. It is the first place credential stuffing shows up, the first place path traversal attempts show up, and the first place anomalous request patterns show up.

Security-relevant fields in API gateway logs:

- `client_ip`: Source of the request. The field to watch for geographic anomalies and clustering.
- `user_agent`: Should match known client patterns. Unknown user agents indicate automated scanning.
- `http_method` + `request_path`: Together, these identify the endpoint being accessed. Unusual path patterns (traversal attempts, known exploit paths) surface here.
- `response_status`: 401/403 spikes indicate authentication probing. 429 spikes indicate rate limit evasion attempts. 500 spikes from a single source indicate fuzzing.
- `response_time`: Unusually slow responses can indicate injection attacks causing database delays.
- `bytes_sent`: Unusually large responses indicate bulk data access.

## SIEM-Grade Data Quality Requirements

Observability data quality and SIEM data quality have different standards. Understanding the difference is an operational discipline, not a philosophical distinction.

**Completeness:** For security purposes, you must receive all events. Sampling is acceptable in observability (trace 10% of requests for latency analysis). Sampling is not acceptable for security (if you sample authentication logs, you may miss the one authentication event that starts a breach). Every authentication event must land in Splunk.

**Integrity:** Logs must not be modifiable after ingestion. An attacker who gains access to your Splunk admin account and deletes the logs that record their activity has committed a crime and also erased the evidence. SIEM-grade integrity means write-once storage for security logs, separate from the Splunk indexer's own storage (which is modifiable by a Splunk admin). This is addressed in Chapter 6 (immutable log archival).

**Timeliness:** The maximum acceptable delay between event occurrence and availability in Splunk for security correlation is typically 5 minutes. A correlation search that runs every 5 minutes and looks at the last 10 minutes needs the data to arrive within 5 minutes or the correlation will miss events. Timeliness requirements for observability are similar, but the consequence of a few-minute delay in a latency graph is annoyance. The consequence of a few-minute delay in security log ingestion is that a correlation search misses an in-progress attack.

**Retention:** Security retention requirements are driven by compliance frameworks (see Chapter 6), not by operational needs. You keep reliability logs long enough to investigate incidents and identify trends — typically 30 to 90 days in hot storage. Security logs must be retained for 1 to 6 years depending on the framework. This drives the tiered storage architecture described below.

## Log Normalization: The CIM Imperative

A Splunk deployment that ingests logs from 15 different sources without normalization produces 15 different field names for "the username that performed the action." Some call it `user`, some call it `username`, some call it `actor`, some call it `principal`. A correlation search that tries to join authentication events from Okta with Kubernetes audit events fails because the field names do not match.

The Common Information Model (CIM) solves this. CIM defines standard field names for each category of security event: `Authentication.user`, `Authentication.src`, `Authentication.action`. Technology Add-ons (TAs) for each log source perform the field mapping — they translate `okta.actor.displayName` to `Authentication.user`, for example.

For a security correlation search to work across your entire environment, every log source it depends on must be CIM-compliant. Ingesting a log source without ensuring CIM compliance means that any ESCU correlation search referencing that source's datamodel fields will silently fail — it will run without error and produce no results, because the fields are not mapped.

CIM compliance validation query:

```spl
| datamodel Authentication search 
| where sourcetype="your-new-sourcetype" 
| stats count by Authentication.user, Authentication.src, Authentication.action
| where isnull(Authentication.user)
```

If `Authentication.user` is null for your sourcetype, the CIM mapping is incomplete.

## Director Frame

Data quality is the unglamorous prerequisite that determines whether your SIEM investment delivers value. The pattern that repeats across organizations: buy Splunk ES, install ESCU content, onboard log sources without CIM compliance validation, watch correlation searches produce zero notables, conclude that the SIEM is not working, and invest in a different SIEM tool. The problem was never the SIEM. The problem was that the data going into it was not normalized correctly.

Your platform team owns the log pipeline. That means you own data quality. The specific commitment a Director should make: no log source is considered "onboarded" until it has been validated for CIM compliance, volume-estimated, retention-configured, and handed off to SOC with documentation. That four-step checklist (see the Log Source Onboarding Playbook) is the difference between a SIEM that works and a SIEM that accumulates debt.
