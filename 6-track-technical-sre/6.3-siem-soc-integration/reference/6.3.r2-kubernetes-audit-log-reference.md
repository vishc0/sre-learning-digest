# Kubernetes Audit Log Quick Reference

## Audit Log Levels

The audit level determines how much information is captured for each event. Set per-rule in the audit policy.

| Level | What Is Captured | When to Use | Storage Impact |
|-------|----------------|------------|---------------|
| `None` | Nothing | Events you explicitly want to exclude from the audit log | None |
| `Metadata` | Request metadata only: user, timestamp, resource, verb, response code — NO request or response body | Default for most events; sufficient for access pattern analysis | Low |
| `Request` | Metadata + request body | RBAC mutations, secret reads where you want to see what was requested | Medium |
| `RequestResponse` | Metadata + request body + response body | Secrets (to see actual secret content if needed), RBAC changes | High — use selectively |

**Practical guidance:** Set secrets to `RequestResponse` so you can see what was returned. Set RBAC mutations to `Request`. Set everything else to `Metadata`. Never set `RequestResponse` globally — it will generate enormous log volumes.

## Audit Stages

Each API request generates one or more audit events, one per stage:

| Stage | When It Fires | Primary Use |
|-------|-------------|------------|
| `RequestReceived` | When the API server receives the request, before authentication or authorization | Rarely needed for security analysis; pre-auth so actor may be unauthenticated |
| `ResponseStarted` | When response headers are sent but before response body completes | Long-running requests: watches, exec sessions — fires when connection opens |
| `ResponseComplete` | When the full response has been sent | **Primary stage for investigation** — actor is authenticated, action is authorized and executed |
| `Panic` | When the API server panics handling the request | Crash investigation; rare in production |

**For most investigations:** Filter to `stage=ResponseComplete` to see only successfully executed actions.

## Key Verbs to Monitor for Security

| Verb | Meaning | Security Significance |
|------|---------|----------------------|
| `get` | Read a specific resource | Secret reads, credential access |
| `list` | List resources of a type | Enumeration/reconnaissance |
| `watch` | Subscribe to changes for a resource type | Persistent monitoring by an adversary |
| `create` | Create a new resource | Backdoor service account, malicious pod, new RBAC binding |
| `update` | Full replacement of an existing resource | Privilege escalation by replacing role definition |
| `patch` | Partial modification of an existing resource | Stealthy privilege escalation |
| `delete` | Delete a resource | Evidence destruction, removing guardrails |
| `exec` (subresource) | Open an exec session in a pod | Interactive shell access — always notable in production |
| `portforward` (subresource) | Forward a port from a pod | Direct network access to pod internals |
| `impersonate` | Act as another user or service account | Identity spoofing |

## Key Resources to Monitor

| Resource | API Group | Why It Matters |
|----------|----------|---------------|
| `secrets` | core | Credentials, API keys, TLS certs |
| `configmaps` | core | May contain sensitive configuration; sometimes misused to store secrets |
| `serviceaccounts` | core | Identities for workloads; compromise = persistent access |
| `serviceaccounts/token` | core | Token generation — watch for unexpected token minting |
| `pods/exec` | core | Interactive pod access |
| `pods/portforward` | core | Direct port access |
| `pods` | core | Privileged pod creation can be a container escape vector |
| `roles` | rbac.authorization.k8s.io | Namespace-scoped permission definitions |
| `clusterroles` | rbac.authorization.k8s.io | Cluster-wide permission definitions |
| `rolebindings` | rbac.authorization.k8s.io | Grants a role to a user/group/service account within a namespace |
| `clusterrolebindings` | rbac.authorization.k8s.io | Grants a clusterrole to a user/group/service account cluster-wide |
| `networkpolicies` | networking.k8s.io | Modification can open network access between namespaces |
| `validatingwebhookconfigurations` | admissionregistration.k8s.io | Disabling admission webhooks removes security controls |

## High-Value Audit Policy Rules

Add these rules to your Kubernetes audit policy YAML to capture security-critical events at the correct level:

```yaml
apiVersion: audit.k8s.io/v1
kind: Policy

# Omit RequestReceived stage globally — ResponseComplete is sufficient
omitStages:
  - "RequestReceived"

rules:
  # Secrets: RequestResponse to see what was returned
  - level: RequestResponse
    resources:
    - group: ""
      resources: ["secrets"]
    verbs: ["get", "list", "watch"]

  # All RBAC mutations: Request level to see what was created/changed
  - level: Request
    resources:
    - group: "rbac.authorization.k8s.io"
      resources:
        - "roles"
        - "clusterroles"
        - "rolebindings"
        - "clusterrolebindings"
    verbs: ["create", "update", "patch", "delete"]

  # Pod exec and portforward: Request level
  - level: Request
    resources:
    - group: ""
      resources: ["pods/exec", "pods/portforward", "pods/attach"]

  # Service account token creation: Metadata level
  - level: Metadata
    resources:
    - group: ""
      resources: ["serviceaccounts/token"]

  # Privileged pod creation (pods with hostPID, hostNetwork, or privileged containers)
  - level: Request
    resources:
    - group: ""
      resources: ["pods"]
    verbs: ["create", "update", "patch"]

  # Network policy modifications
  - level: Request
    resources:
    - group: "networking.k8s.io"
      resources: ["networkpolicies"]
    verbs: ["create", "update", "patch", "delete"]

  # Admission webhook modifications (security control changes)
  - level: Request
    resources:
    - group: "admissionregistration.k8s.io"
      resources:
        - "validatingwebhookconfigurations"
        - "mutatingwebhookconfigurations"
    verbs: ["create", "update", "patch", "delete"]

  # All other requests at Metadata level
  - level: Metadata
    omitStages:
      - "RequestReceived"
```

## Common Attack Patterns in K8s Audit Logs

### Pattern 1: Container Escape Indicators

```
Sequence:
1. Pod running with privileged:true or hostPID:true (verb=create, resource=pods)
2. exec into that pod (verb=create, resource=pods/exec)
3. No further pod-level activity — attacker now operating at host level
```

Detection SPL:
```spl
index=k8s_audit stage=ResponseComplete
| eval is_exec=if(objectRef.resource="pods" AND objectRef.subresource="exec", 1, 0)
| eval is_priv_pod=if(objectRef.resource="pods" AND verb="create" AND 
    match(requestObject.spec.containers{}.securityContext.privileged, "true"), 1, 0)
| stats values(eval(if(is_exec=1, objectRef.name, null()))) AS exec_targets,
         values(eval(if(is_priv_pod=1, objectRef.name, null()))) AS priv_pods
    BY user.username
| where isnotnull(exec_targets) AND isnotnull(priv_pods)
```

### Pattern 2: RBAC Escalation Path

```
Sequence:
1. Low-privilege service account begins operation (normal requests for weeks)
2. That service account suddenly creates or modifies a ClusterRoleBinding
   granting itself or another account ClusterAdmin
3. New high-privilege account begins making cluster-wide requests
```

Detection SPL:
```spl
index=k8s_audit stage=ResponseComplete
    objectRef.resource IN (clusterrolebindings, rolebindings)
    verb IN (create, update, patch)
    responseStatus.code=200
    user.username!="system:*"
| rex field=requestObject "\"name\":\\s*\"(?<granted_to>[^\"]+)\""
| table _time, user.username, verb, objectRef.resource, objectRef.namespace, 
         objectRef.name, granted_to
```

### Pattern 3: Secret Enumeration

```
Sequence:
1. Service account or user runs: kubectl get secrets -A (verb=list, resource=secrets)
2. Followed by: kubectl get secret <specific-secret-name> -n <namespace> 
   multiple times (verb=get, resource=secrets, objectRef.name=<each secret>)
```

Detection SPL:
```spl
index=k8s_audit stage=ResponseComplete
    objectRef.resource=secrets
    verb IN (get, list)
    responseStatus.code=200
| stats count AS secret_reads,
         dc(objectRef.namespace) AS namespace_count,
         dc(objectRef.name) AS distinct_secrets
    BY user.username, span=5m
| where secret_reads > 10 OR namespace_count > 3
```

### Pattern 4: Crypto-Miner Installation

```
Sequence:
1. Service account creates a pod with an unexpected image 
   (image not in organization's approved registry)
2. Pod requests excessive CPU resources (resource limits set abnormally high)
3. Pod exec or no further audit activity — miner running inside container
```

Detection SPL:
```spl
index=k8s_audit stage=ResponseComplete
    objectRef.resource=pods verb=create
    responseStatus.code=201
    user.username!="system:*"
| rex field=requestObject 
    "\"image\":\\s*\"(?<container_image>[^\"]+)\""
| where NOT match(container_image, "your-registry.example.com")
| table _time, user.username, objectRef.namespace, objectRef.name, container_image
```

## Splunk Field Mapping: K8s Audit → CIM

K8s audit logs require a custom TA or manual field mapping to populate CIM datamodels. Key field translations:

| K8s Audit Field | CIM Field | CIM Datamodel |
|----------------|-----------|--------------|
| `user.username` | `Authentication.user` / `Change.user` | Authentication, Change |
| `sourceIPs{}` | `Authentication.src` / `Change.src` | Authentication, Change |
| `objectRef.resource` | `Change.object` | Change |
| `verb` | `Change.action` | Change |
| `objectRef.namespace` | `Change.object_attrs` | Change |
| `responseStatus.code` | `Authentication.action` (200=success, 403=failure) | Authentication |
| `userAgent` | `Authentication.app` | Authentication |
| `requestURI` | (no direct CIM mapping — keep as raw field) | — |

## Sample SPL Queries — High Value Security Investigations

### Query 1: Actor inventory for investigation window

```spl
index=k8s_audit stage=ResponseComplete
    earliest=-4h
    user.username!="system:*"
    user.username!="kube-admin"
| stats count AS action_count, 
         values(verb) AS verbs_used,
         values(objectRef.resource) AS resources_touched
    BY user.username
| sort -action_count
```

### Query 2: All RBAC mutations in the last 7 days

```spl
index=k8s_audit stage=ResponseComplete earliest=-7d
    objectRef.resource IN (roles, clusterroles, rolebindings, clusterrolebindings)
    verb IN (create, update, patch, delete)
    responseStatus.code=200
| table _time, user.username, verb, objectRef.resource, 
         objectRef.namespace, objectRef.name
| sort _time
```

### Query 3: Exec events — all interactive pod access

```spl
index=k8s_audit stage=ResponseComplete
    objectRef.resource=pods
    objectRef.subresource=exec
    responseStatus.code=101
| table _time, user.username, objectRef.namespace, objectRef.name, sourceIPs
| sort -_time
```

### Query 4: Secret reads by non-system accounts

```spl
index=k8s_audit stage=ResponseComplete
    objectRef.resource=secrets
    verb IN (get, list)
    responseStatus.code=200
    user.username!="system:*"
| stats count AS reads,
         values(objectRef.namespace) AS namespaces,
         values(objectRef.name) AS secret_names
    BY user.username
| sort -reads
```

### Query 5: First-time API verb/resource combinations per service account

```spl
index=k8s_audit stage=ResponseComplete
    user.username="system:serviceaccount:*"
    earliest=-24h
| rex field=user.username "system:serviceaccount:(?<sa_namespace>[^:]+):(?<sa_name>.+)"
| eval activity_signature=sa_name + "::" + verb + "::" + objectRef.resource
| eval lookback_period=relative_time(now(), "-90d@d")
| where _time > lookback_period
```

Combined with a `first_time` lookup from MLTK or ES:

```spl
index=k8s_audit stage=ResponseComplete earliest=-24h
    user.username="system:serviceaccount:*"
| rex field=user.username "system:serviceaccount:(?<sa_ns>[^:]+):(?<sa_name>.+)"
| eval sig=sa_name + "|" + verb + "|" + objectRef.resource + "|" + objectRef.namespace
| `first_time_field(sig, 90d)`
| where first_time=1
| table _time, sa_name, sa_ns, verb, objectRef.resource, objectRef.namespace
```
