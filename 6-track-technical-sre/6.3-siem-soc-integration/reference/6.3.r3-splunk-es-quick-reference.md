# Splunk Enterprise Security Quick Reference

## Key ES Objects — One-Line Definitions

| Object | Definition |
|--------|-----------|
| **Notable Event** | A correlated alert promoted to the analyst queue after passing through a correlation search threshold; the primary analyst work item in ES |
| **Risk Score** | A numeric value accumulated on an entity (user, system, IP) by triggering risk rules; alerts fire when accumulated score crosses a threshold |
| **Assets & Identities** | Lookup tables enriching raw IPs and usernames with business context (owner, department, criticality, environment) |
| **Threat Intelligence** | External and internal indicator feeds (IP, domain, hash, email) used to enrich events and drive correlation searches |
| **Correlation Search** | A scheduled SPL search that evaluates security patterns over a time window and creates notable events when thresholds are met |
| **Data Model** | A structured abstraction layer (CIM) that normalizes disparate log sources into standard field names for cross-source correlation |
| **Risk Object** | An entity that accumulates risk scores — can be a `user`, `system`, or `ip_address` type |
| **ESCU** | Enterprise Security Content Update — Splunk's library of pre-built correlation searches mapped to MITRE ATT&CK |
| **Security Posture Dashboard** | ES home dashboard showing overall notable event status, risk scores, and key indicator counts |
| **Adaptive Response** | ES framework for triggering automated response actions (blocking an IP, running a script) when a notable event fires |

## Notable Event Lifecycle

```
Correlation Search fires
        ↓
Notable Created — status: New (appears in analyst queue)
        ↓
Analyst assigns to themselves — status: In Progress
        ↓
Analyst investigates using linked drilldown searches
        ↓
Analyst closes with disposition:
    ├── Resolved       → True positive; incident handled
    ├── False Positive → Rule fired incorrectly; tune the correlation search
    └── No Action      → True observation but benign; document and close
```

**Key fields on a notable event:**

| Field | Description |
|-------|------------|
| `rule_name` | Name of the correlation search that created it |
| `severity` | informational / low / medium / high / critical |
| `src` | Source IP or host |
| `dest` | Destination IP or host |
| `user` | Implicated user account |
| `owner` | ES analyst the notable is assigned to |
| `status` | New / In Progress / Pending / Resolved / Closed |
| `drilldown_search` | The SPL search to investigate the contributing events |
| `contrib_events` | Links to the raw events that triggered the notable |

## Risk-Based Alerting Reference

### Risk Score Formula

Risk scores are assigned by risk rules (special correlation searches that do not create notables — they add points to risk objects). The accumulated score drives a separate "risk notable" correlation search.

```
Risk Rule fires → adds risk_score to risk_object
    risk_object types: user | system | ip_address
    risk_score range: 0–100 per event (by convention)
    
Risk Notable Correlation Search: fires when sum(risk_score) > threshold
    Default threshold: 100 (highly configurable)
    Lookback window: typically 24 hours (configurable)
```

### Risk Score Accumulation Example

| Time | Event | Risk Points | Running Total for user@example.com |
|------|-------|-------------|-----------------------------------|
| 09:00 | Login from unusual country | +30 | 30 |
| 09:15 | Access to sensitive application outside hours | +40 | 70 |
| 09:22 | Failed attempt to access admin console | +35 | 105 → **NOTABLE FIRES** |

### Risk Object Types

| Type | Description | Example Values |
|------|-------------|---------------|
| `user` | Human or service account | `jsmith@example.com`, `svc-api-gateway` |
| `system` | Hostname or FQDN | `prod-node-042.cluster.local` |
| `ip_address` | IP address | `10.20.30.40`, `203.0.113.5` |

### Risk Score Accumulation SPL

```spl
| from datamodel Risk.All_Risk
| stats sum(risk_score) AS total_risk, 
         values(source) AS contributing_rules,
         values(risk_message) AS risk_messages
    BY risk_object, risk_object_type
| where total_risk > 100
| sort -total_risk
```

### Risk Score Decay

Risk scores decay over time at a configurable rate. An entity that accumulated points 30 days ago and has been clean since should not trigger a notable based on stale risk. Configure decay in ES Settings > Risk Analysis.

## ESCU Content Packs

**What ESCU is:** A curated content library distributed by Splunk that provides pre-built correlation searches, macros, and threat intelligence sources, mapped to MITRE ATT&CK techniques.

**How to enable ESCU content:**
1. ES > Content Management
2. Filter by technique, source type, or data model
3. Review prerequisites (required data sources and data models)
4. Enable with "Activate"

**MITRE ATT&CK mapping:** Each ESCU detection has a `mitre_attack_id` field. Use this to identify which ATT&CK techniques your current detection coverage addresses and which are gaps.

**Evaluating ESCU rules for your environment:**

| Check | How |
|-------|-----|
| Does the required log source exist in your environment? | `\| datamodel <datamodel> search \| stats count by sourcetype` |
| Does the rule produce expected results in your data? | Enable in audit mode first; review `search_name` hits in `summary` index |
| What is the false positive rate? | Run retrospectively: `\| savedsearch "<rule_name>" \| stats count` |
| Is the MITRE technique relevant to your threat model? | Cross-reference with your threat intelligence team |

## Critical ES-Specific SPL Patterns

### Query all new, high-severity notables

```spl
| `notable`
| search status=new (severity=high OR severity=critical)
| table _time, rule_name, src, dest, user, severity, owner
| sort -_time
```

### Drill down into a specific notable's contributing events

```spl
| `notable`
| where rule_name="Brute Force Access Behavior Detected"
| head 1
| eval drilldown=drilldown_search
| `get_event_id(event_id)`
```

### Risk score drill-down for a specific entity

```spl
| from datamodel Risk.All_Risk
| where risk_object="suspicious-service-account"
| sort -_time
| table _time, source, risk_score, risk_message, risk_object_type
```

### Threat intelligence lookup against observed IPs

```spl
| lookup threatintel_by_cidr src AS src_ip 
    OUTPUT threat_category, confidence, threat_group, threat_source
| where isnotnull(threat_category)
| stats count BY src_ip, threat_category, threat_group, confidence
| sort -count
```

### Identify correlation searches with high false positive rates

```spl
index=notable_events sourcetype=stash
| stats count AS total_notables,
         count(eval(disposition="False Positive")) AS false_positives
    BY savedsearch_name
| eval fp_rate=round(false_positives/total_notables*100, 1)
| where fp_rate > 30
| sort -fp_rate
```

## ES vs. Core Splunk: What ES Adds

| Capability | Core Splunk | Splunk ES |
|-----------|------------|-----------|
| Log ingestion and indexing | Yes | Yes (same infrastructure) |
| SPL searching | Yes | Yes |
| Alerting | Yes (threshold-based) | Yes + risk-based + sequence-based |
| Data model (CIM) | Available (requires TA configuration) | Enforced + pre-configured |
| Notable events queue | No | Yes — analyst-facing work queue |
| Risk scoring | No | Yes — entity-level accumulated scoring |
| Asset & identity enrichment | Manual (lookup tables) | Built-in framework |
| Threat intelligence | Manual | Built-in feed management |
| MITRE ATT&CK mapping | No | Yes (via ESCU) |
| Adaptive response actions | No | Yes |
| Security Posture dashboard | No | Yes |

## Common ES Configuration Pain Points

| Issue | Symptom | Fix |
|-------|---------|-----|
| **Asset/Identity list staleness** | Notable events reference outdated host owners or usernames | Automate Asset and Identity list refresh from LDAP/CMDB on a schedule |
| **False positive flood** | Correlation search produces hundreds of notables from legitimate activity | Tune correlation search with exclusion macros; increase threshold; use Risk-Based Alerting instead of direct notables |
| **Correlation search performance** | ES search head is slow; correlation searches exceed 5-minute run limit | Convert to `tstats`-based searches against accelerated datamodels; reduce lookback window; optimize SPL |
| **CIM datamodel not accelerating** | `tstats`-based correlation searches return no results | Verify datamodel acceleration is enabled and has completed: ES > Data Models > check acceleration status |
| **Risk scores not accumulating** | No risk notables firing despite suspicious activity | Verify risk rules are enabled; check that risk object field is correctly mapped; confirm the risk notable threshold search is active |
| **ESCU content not matching** | ESCU rule enabled but produces no results | Confirm the required sourcetype is present in the index and the CIM mapping is correct for that sourcetype |
