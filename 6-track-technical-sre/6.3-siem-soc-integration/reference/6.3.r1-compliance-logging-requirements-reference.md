# Compliance Logging Requirements Quick Reference

## Master Comparison Table

| Requirement | SOC 2 Type II | PCI-DSS v4.0 | HIPAA | FedRAMP Moderate |
|------------|--------------|-------------|-------|-----------------|
| **Governing body** | AICPA (SSAE 18) | PCI Security Standards Council | HHS (45 CFR Part 164) | GSA / NIST (FISMA) |
| **Who it applies to** | Service organizations handling customer data | Entities storing/processing/transmitting cardholder data | Covered entities + business associates handling ePHI | Cloud service providers serving U.S. federal agencies |
| **Events to log** | Access, changes, anomalies, admin actions | Individual access, admin actions, auth failures, audit log access, object changes (10 specific categories) | Access to ePHI, workforce activity, security incidents | All security-relevant events (NIST AU-2 defined list) |
| **Online retention** | 12 months | 3 months immediately accessible | 90 days (recommended) | 90 days |
| **Archive retention** | 12 months total | 12 months total | 6 years (documentation) | 12 months total |
| **Log integrity** | Controls over log modification; documentation required | Explicit requirement to protect from unauthorized modification and destruction (10.3) | Audit controls required; integrity implied | AU-9: protect from unauthorized access, modification, deletion |
| **Real-time monitoring** | Implied (anomaly detection required as control) | Required — daily log review OR automated monitoring meeting the intent (10.4) | Implied (security incident procedures required) | AU-6: review and analyze at defined frequency; alert on findings |
| **Log access control** | Controls required; logs accessible to auditors | Access to audit logs is itself a logged event (10.2.1.3) | Workforce access controls to log systems required | AU-9: restrict access to audit logs to authorized individuals |
| **Time synchronization** | Implied (consistent timestamps required for audit trail) | Explicit requirement: internal clocks synchronized to an accepted time source (10.6) | Not explicitly specified; best practice | AU-8: synchronize clocks; reference time source documented |
| **Audit trail availability** | Available to auditors for full audit period | Available for immediate access for 3 months; remainder within defined SLA | Available for investigation on demand | Available for immediate access (90 days); archive within 24-hour SLA |
| **Centralized log management** | Not mandated; expected in practice | Implied by monitoring requirements | Not mandated | AU-12: centralized audit record generation recommended |

## What "Immutable" Means Per Framework

Compliance frameworks use different language to describe log integrity requirements. What they require in practice:

| Framework | Language Used | Minimum Technical Implementation | Is Splunk Alone Sufficient? |
|-----------|-------------|----------------------------------|---------------------------|
| SOC 2 Type II | "Restricts access to audit logs" and "monitors for changes to log configuration" | Access-controlled storage + admin action logging | No — a Splunk admin can delete data. Add S3 Object Lock or equivalent. |
| PCI-DSS v4.0 (Req 10.3) | "Protect audit logs from destruction and unauthorized modifications" | Write-once storage (WORM); integrity checking | No — Splunk indexer storage is modifiable. S3 Object Lock required. |
| HIPAA | "Implement hardware, software, and/or procedural mechanisms" (§164.312(b)) | Audit controls including log protection; less prescriptive | Possibly — if procedural controls (admin auditing, separation of duties) are documented. WORM storage is safer. |
| FedRAMP (AU-9) | "Protect audit information and audit tools from unauthorized access, modification, and deletion" | Access controls + integrity verification; some implementations require WORM | No — requires technical controls beyond access control. S3 Object Lock recommended. |

**Recommended implementation for all frameworks:**

```
Log source → Heavy Forwarder (syslog-ng / Splunk UF)
                    ↓                    ↓
            Splunk Indexers         S3 Bucket (Object Lock enabled)
         (search/correlation)    (immutable compliance archive)
                    ↓
              SmartStore → S3
           (warm tier, searchable)
```

S3 Object Lock configuration: Use `GOVERNANCE` mode for environments where privileged users may need to delete during the retention period (e.g., legal hold release). Use `COMPLIANCE` mode for strict regulatory environments where no deletion is permissible — not even by account root.

## Retention Quick Reference

| Framework | Hot (Immediately Searchable) | Warm (Queryable, Minutes) | Cold (Archive, Hours) | Total |
|-----------|-----------------------------|--------------------------|-----------------------|-------|
| SOC 2 Type II | 90 days | Remaining 9 months | — | 12 months |
| PCI-DSS v4.0 | 90 days | Remaining 9 months | — | 12 months |
| HIPAA (raw logs) | 90 days | 12 months | 5+ additional years | 6 years |
| FedRAMP Moderate | 90 days | Remaining 9 months | — | 12 months |
| Internal observability only | 30 days | — | — | 30 days |

**Note on HIPAA 6-year requirement:** HIPAA specifies 6-year retention for security policies and documentation; raw log retention is not explicitly specified. The conservative and defensible interpretation is to retain logs for 6 years for ePHI-adjacent systems, consistent with breach investigation needs and the documentation retention rule.

## Common Compliance Audit Questions and Splunk Evidence Sources

| Audit Question | Evidence Source in Splunk | SPL Approach |
|---------------|--------------------------|-------------|
| "Who accessed system X during the audit period?" | Authentication index + K8s audit | `index=compliance_auth user=* system=X earliest=-12m@mon` |
| "Were there any unauthorized access attempts?" | ES Notable Events + auth failure logs | `\| notable \| search rule_name="*Auth*" status!=false_positive` |
| "How quickly were security incidents detected and responded to?" | ES Notable Events (create time vs. close time) | `\| notable \| eval mttd=close_time - create_time` |
| "Are administrative actions logged and monitored?" | K8s audit + CloudTrail + Linux auditd | Cross-index search on admin verbs |
| "Is access to audit logs controlled and monitored?" | Splunk internal index (`_audit`) | `index=_audit action=login \| stats count by user` |
| "Are audit logs protected from modification?" | S3 Object Lock attestation + Splunk audit | Provide S3 Object Lock configuration + Splunk `_audit` admin action log |
| "What is the evidence of daily log review?" | ES notable event review history | `\| notable \| stats count by status, owner, date_mday` showing analyst review activity |
| "Are clocks synchronized?" | NTP monitoring logs | `index=infrastructure sourcetype=ntp \| stats max(offset_ms) by host` |

## Cost Estimation Guide

Reference values for planning compliance log storage and Splunk licensing costs.

### Typical Log Volume by Source Type

| Log Source | Typical Volume | Notes |
|-----------|---------------|-------|
| Kubernetes API audit logs | 1–5 GB/day per 100 nodes | Higher with verbose audit policy; lower with `None` on noisy read events |
| Authentication logs (Okta/AD) | 0.1–1 GB/day per 10,000 users | Depends on MFA event frequency |
| API gateway logs | 5–50 GB/day | Highly dependent on traffic volume; filter 200s to reduce |
| VPC flow logs | 10–100 GB/day | Use sampling or filter to relevant traffic flows |
| Linux auditd (per node) | 1–10 GB/day per node | Depends on syscall coverage; restrict to security-relevant syscalls |
| AWS CloudTrail | 0.5–5 GB/day | Scales with API call volume |
| DNS logs (CoreDNS) | 1–20 GB/day | Depends on query rate; aggressive in Kubernetes |

### Compliance Indexing Volume Estimate

Per 1,000 Kubernetes nodes (typical large platform):

| Source | Daily Volume | Annual Volume | S3 Archival Cost (approximate) |
|--------|------------|--------------|-------------------------------|
| K8s audit logs | 10–50 GB | 3.6–18 TB | $80–$400/year (S3 Standard-IA) |
| Auth logs (federated) | 1–5 GB | 365 GB–1.8 TB | $8–$40/year |
| API gateway | 50 GB | 18 TB | $400/year |
| VPC flow | 100 GB | 36 TB | $800/year |
| CloudTrail | 5 GB | 1.8 TB | $40/year |
| **Total (rough estimate)** | **~200 GB/day** | **~70 TB/year** | **~$1,500–$2,000/year S3** |

S3 pricing used: ~$0.023/GB-month Standard, $0.0125/GB-month Standard-IA (retrieval costs additional). Glacier: ~$0.004/GB-month. These are approximate AWS us-east-1 prices; verify current pricing.

Splunk hot tier cost (Splunk licensing): at typical enterprise pricing of $50–$150/GB/day indexed, compliance log volume adds significant cost. Use SmartStore to minimize hot tier to 90 days; archive the rest to S3.

## Red Flags That Cause Compliance Findings

Ordered by frequency of audit findings across SOC 2, PCI, HIPAA, and FedRAMP:

| Red Flag | Framework(s) Impacted | Why It's a Finding | Prevention |
|----------|----------------------|-------------------|-----------|
| **Log gaps** — periods with no events for a system that should be generating logs | All | Auditor assumes the gap was used to cover malicious activity | Monitor forwarder health; alert on zero-event hosts |
| **Time sync failures** — log timestamps inconsistent across systems | PCI-DSS (explicit), FedRAMP | Event correlation is unreliable; audit trail is suspect | Monitor NTP offset on all systems; alert on drift > 1 second |
| **Admin log deletion** — evidence that Splunk admins deleted log data | PCI-DSS, FedRAMP, SOC 2 | Undermines log integrity; cannot prove what was deleted | S3 Object Lock + Splunk `_audit` log monitoring for delete actions |
| **Missing log sources** — required systems not sending to SIEM | All | Cannot demonstrate monitoring coverage | Maintain compliance log inventory; quarterly coverage audit |
| **Short retention** — logs deleted before required retention period expires | All | Cannot produce logs for the full audit period on demand | Configure index retention per framework; monitor with `dbinspect` |
| **CIM non-compliance** — logs present but fields unmapped | SOC 2, FedRAMP (implied) | Cannot demonstrate automated monitoring working correctly | CIM validation at onboarding; quarterly CIM health check |
| **No evidence of log review** — logs exist but no record of analyst review | PCI-DSS (10.4), FedRAMP (AU-6) | Cannot demonstrate controls are operating | ES notable event review history; analyst activity reports |
| **Splunk admin accounts unmonitored** — admins can act without audit trail | All | Admin actions unverifiable | Monitor Splunk `_audit` index; alert on privileged admin actions |

## Splunk Administrative Audit Monitoring

To demonstrate log integrity to auditors, monitor Splunk's own administrative actions:

```spl
index=_audit
| stats count AS admin_actions, 
         values(action) AS actions_taken
    BY user, info
| where user!="splunk-system-user"
| sort -admin_actions
```

Alert on delete actions specifically:

```spl
index=_audit action=delete
| table _time, user, info, search_id
| sort -_time
```

This search surfaces any data deletion events by Splunk administrators. The presence of delete events for compliance indexes is a compliance finding unless they are documented and authorized. Provide this search result to auditors as evidence that log deletion is monitored.
