## Reliability Response Section

*Audience: SRE. Goal: restore service availability while preserving evidence.*

### Phase 1: Detection and Initial Assessment (0–10 minutes)

**Confirm the reliability signal:**
- [ ] Verify the triggering alert is genuine (not a monitoring system false positive)
- [ ] Check SLO dashboard: which SLO is breaching? What is the current error budget remaining?
- [ ] Identify affected service components:

```spl
index=access_logs service=[service-name] earliest=-15m
| stats count AS requests, 
         sum(eval(status>=500)) AS errors, 
         avg(response_time) AS avg_latency 
    BY endpoint
| eval error_rate=round(errors/requests*100, 2)
| where error_rate > 5 OR avg_latency > 2000
```

**Determine blast radius:**
- [ ] Which downstream services are affected?
- [ ] What is the estimated user impact (percentage of requests failing)?
- [ ] Is the issue isolated to one region / AZ / namespace?

**Document in shared incident timeline:** Time of detection, initial scope, first responder name.

### Phase 2: Triage (10–20 minutes)

**Run security pattern check** before taking any remediation action:

```spl
index=access_logs service=[service-name] earliest=-30m
| eval auth_fail=if(status=401 OR status=403, 1, 0),
       new_egress=if(dest_ip!="" AND isnotnull(dest_ip), 1, 0)
| stats sum(auth_fail) AS total_auth_failures, 
         dc(src_ip) AS unique_sources,
         dc(dest_ip) AS unique_dest_ips 
    BY _time span=5m
| where total_auth_failures > 100 AND unique_sources < 10
```

If the pattern check shows high auth failures from few sources: **this may be a security incident, not a pure reliability incident.** Invoke the SRE-SOC Handoff Playbook before continuing with reliability remediation.

**If no security pattern detected, continue with reliability triage:**
- [ ] Check for recent deployments: `kubectl rollout history deployment/[service] -n [namespace]`
- [ ] Check pod health: `kubectl get pods -n [namespace] -o wide`
- [ ] Check recent Kubernetes events: `kubectl get events -n [namespace] --sort-by=.metadata.creationTimestamp`
- [ ] Check resource constraints: `kubectl top pods -n [namespace]`

### Phase 3: Containment (20–40 minutes)

Actions appropriate for reliability containment that are also evidence-safe:

- [ ] Route traffic to healthy replicas (adjust load balancer or service weights)
- [ ] Scale out the affected service (adds capacity; does not destroy existing pods)
- [ ] Apply rate limiting at the API gateway (reduces load; preserves all evidence)
- [ ] Enable circuit breaker to downstream dependency (protects service; does not modify any pod)

**Actions to avoid until SOC is consulted:**
- Restarting pods (destroys in-memory state)
- Rolling back deployments on affected pods (overwrites running state)
- Rotating secrets or credentials

### Phase 4: Restoration

- [ ] Verify that containment actions have reduced the error rate below the SLO threshold
- [ ] Monitor for 15 minutes to confirm stability before declaring the reliability incident resolved
- [ ] Update SLO dashboard and error budget tracking
- [ ] Document all actions taken in the shared incident timeline

## Security Investigation Section

*Audience: SOC. Goal: scope the security event, preserve evidence, determine attribution.*

### Phase 1: Evidence Preservation (0–15 minutes, parallel with SRE triage)

**Capture current state before SRE takes any remediation action:**
- [ ] Screenshot or export the Splunk search showing the triggering events
- [ ] Capture the notable event record from ES: notable ID, contributing events, timestamp
- [ ] Identify all pods, hosts, and service accounts involved
- [ ] Request SRE to hold off on pod restarts until evidence capture is complete

```spl
| `notable` 
| where rule_name="[triggering rule name]" 
| sort -_time 
| head 1 
| table rule_name, src, dest, user, severity, drilldown_search
```

### Phase 2: Scope Assessment (15–45 minutes)

**Determine the full scope of the incident:**

```spl
index=k8s_audit stage=ResponseComplete
    earliest=[incident_start - 1h]
    user.username!="system:*"
| stats count BY user.username, verb, objectRef.resource, objectRef.namespace
| sort -count
```

- [ ] Which systems have been accessed by the implicated actor?
- [ ] Has any data been read that should not have been accessible?
- [ ] Have any credentials or secrets been accessed?
- [ ] Is there evidence of lateral movement to additional systems?

**Threat intelligence enrichment:**

```spl
index=access_logs OR index=network_flow
    earliest=[incident_start - 1h]
    src_ip=[implicated_ip]
| lookup threatintel_by_ip src_ip AS src_ip 
    OUTPUT threat_category, threat_group, confidence
| where isnotnull(threat_category)
```

### Phase 3: Investigation Queries

[Customize these queries for the specific incident type. Include 3–5 targeted SPL queries.]

**Query template 1: Authentication pattern analysis**

```spl
index=access_logs earliest=[start] latest=[end]
| stats count AS attempts,
         dc(user) AS unique_users,
         count(eval(status=200)) AS successes,
         count(eval(status=401 OR status=403)) AS failures
    BY src_ip
| eval success_rate=round(successes/attempts*100, 2)
| where failures > 50
| sort -failures
```

**Query template 2: Data access scope**

```spl
index=[data-store-logs] earliest=[start] latest=[end]
    actor=[implicated_user]
| stats count, values(resource_accessed) AS resources, sum(bytes_returned) AS total_bytes
    BY actor, action
| sort -total_bytes
```

## Post-Incident Section

### RCA (SRE — Reliability Dimension)

To be completed within 5 business days of incident resolution.

- What reliability control failed or was absent?
- What change would have prevented the reliability impact?
- What monitoring improvement would have reduced MTTD?
- Error budget impact: [hours of burn / total budget consumed]

### Forensic Report (SOC — Security Dimension)

To be completed within 10 business days of incident resolution.

- Attack timeline (first evidence to containment)
- Initial access vector
- Systems accessed and data potentially exposed
- Attacker TTPs mapped to MITRE ATT&CK
- Detection gap analysis: why was the attack not detected earlier?
- Recommended detection improvements
