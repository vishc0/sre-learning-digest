# Chapter 2: Security Event Taxonomy for SREs

## Why SREs Need This

The first person to see a security incident is almost never a SOC analyst. It is usually an SRE watching a reliability dashboard who notices something unusual — a latency spike, an error rate jump, unexpected resource utilization — and has to make a decision: is this a reliability problem or a security problem?

Making that decision correctly in 60 seconds matters. Getting it wrong in either direction is costly. If an SRE treats a security incident as a pure reliability incident, the response actions (restarting services, rolling back deployments, scaling capacity) may destroy the evidence the SOC needs to scope the breach. If an SRE treats every anomaly as a potential security incident and pages SOC constantly, the relationship degrades and escalations get ignored.

SREs do not need to be security analysts. They need enough security vocabulary to make the escalation decision accurately — to recognize the shape of a security event when it is wearing a reliability disguise.

This chapter gives you the four threat categories that matter most from an SRE vantage point, the signals each produces in reliability dashboards, and the signals that tell the SOC something different is happening.

---

## Threat Category 1: Authentication Failures

**What SREs see in reliability dashboards:**
- Spike in HTTP 401 and 403 response codes on the API gateway
- Increased latency on authentication endpoints
- Rate limit breaches on login paths
- Error budget burn on authentication services

**What SOC sees in the same data:**
- Credential stuffing pattern: many usernames tried from a small set of source IPs, with a consistent failure rate
- Password spray pattern: one password tried against many usernames
- Geographic anomaly: authentication attempts from locations inconsistent with user history
- Velocity anomaly: authentication rate far above baseline for that hour of day

**The signal composition:** The SRE sees a load problem on the auth service. The SOC sees an adversary testing stolen credential lists. Both views are correct. Neither view alone is sufficient for the right response.

**Escalation indicator:** Authentication failure rate is 10x baseline AND source IPs are a small, unusual set (fewer distinct IPs than you would expect from legitimate user traffic). That pattern is credential stuffing, not a misconfigured client.

## Threat Category 2: Privilege Escalation

**What SREs see:**
- Unexpected process running on a node (high CPU from an unknown binary)
- New service account appearing in Kubernetes RBAC audit logs
- Resource quota breach in a namespace that was quiet
- Unexpected pod creation in a production namespace

**What SOC sees:**
- Sudo event sequence: a low-privilege account running commands as root
- RBAC mutation: a service account granted ClusterAdmin or unusual verbs on sensitive resources
- New rolebinding: existing service account suddenly has permissions it did not have yesterday
- Container escape indicator: a process in a container making syscalls inconsistent with containerized workloads

**The signal composition:** The SRE sees resource usage anomalies and unexpected workloads. The SOC sees a privilege escalation path being walked. The difference is the interpretation of the RBAC events, which most SREs do not watch in real time.

**Escalation indicator:** A service account that was previously limited to read access on its own namespace now has ClusterAdmin, OR a new rolebinding was created for a namespace outside of a change window. Either is sufficient to escalate.

---

## Threat Category 3: Lateral Movement

**What SREs see:**
- Unusual service-to-service call patterns: service A is suddenly calling service B, which it never did before
- Unexpected DNS queries: pods querying external domains they have no reason to contact
- East-west traffic increase: internal network flow volume between services is higher than baseline
- Timeout patterns in services that previously had no dependency relationship

**What SOC sees:**
- New connection pairs: IP addresses that never communicated before are now exchanging data
- Port scan pattern: a single source IP making connection attempts to many internal services
- Protocol anomaly: a service communicating on a port inconsistent with its documented behavior
- Internal reconnaissance: queries against service discovery endpoints, etcd, the Kubernetes API server from unusual service accounts

**The signal composition:** The SRE sees mysterious timeout errors and unusual traffic patterns. The SOC sees an adversary mapping the internal network after establishing a foothold. Service mesh telemetry (Istio, Linkerd) is unusually valuable here — it provides east-west visibility that most SREs already collect but rarely analyze for security patterns.

**Escalation indicator:** A pod or service account is making network connections to a destination it has never communicated with, AND that destination is either external or is a high-value internal target (database, secrets store, API server).

---

## Threat Category 4: Data Exfiltration

**What SREs see:**
- Outbound bandwidth spike: egress bytes are well above baseline, often in a consistent direction (to a specific external IP or CDN)
- Unusual object storage access: S3 bucket or GCS bucket being read by a service that does not normally access it, at high volume
- Response size anomaly: API responses are larger than typical, suggesting data is being returned in bulk
- Database query volume spike: more queries than usual against a specific table or set of tables

**What SOC sees:**
- Data transfer to unknown destination IP: egress to an IP that does not appear in the organization's known external services list
- DLP alert: data loss prevention system flagging bulk transfer of structured data
- DNS tunneling indicator: unusually long DNS queries or high query rate to a single external domain
- Uncommon user agent: API calls with user agents inconsistent with known client applications

**The signal composition:** The SRE sees an unexplained egress cost spike and possible API performance degradation. The SOC sees a data exfiltration in progress. Outbound bandwidth spikes are one of the most common SRE-facing signals for exfiltration — they show up in cost dashboards before they show up in security alerts.

**Escalation indicator:** Egress bytes to a single external destination are more than 5x the baseline for that destination for that service, AND the destination does not appear in the service's known external dependency list.

## Alert Correlation: How the Signals Compose

Individual signals are weak. Composed signals are diagnostic.

The pattern that should immediately trigger a security escalation without any further analysis:

**Latency spike + Auth failure spike + New egress destination**

This three-signal combination, occurring in the same time window, has a very specific interpretation: someone successfully authenticated (after some failures), established access to the system, and data is now leaving to somewhere new. This is a breach-in-progress pattern. The latency spike is the system working harder than usual because something is actively running on it.

SREs who watch only the latency alert will miss this. SREs who have been trained to look for correlated signals — especially auth failures followed by new egress destinations — will catch it.

A working Splunk search that composes these signals:

```spl
index=access_logs earliest=-1h
| eval auth_failure=if(status=401 OR status=403, 1, 0)
| stats sum(auth_failure) as fail_count, dc(dest_ip) as dest_count, avg(response_time) as avg_latency by src_ip
| where fail_count > 20 AND dest_count > 5 AND avg_latency > 2000
```

This is not a SOC query. This is the kind of correlation check an SRE can run in the first five minutes of investigating an anomaly to determine whether SOC needs to be in the room.

## The Recognition Vocabulary

SREs need to recognize these four patterns and make an escalation decision. They do not need to investigate the security incident. The specific vocabulary to internalize:

- **Credential stuffing**: high-volume auth failures against many usernames from few source IPs
- **RBAC mutation**: a change to roles, clusterroles, or rolebindings outside of a change window
- **Lateral movement**: new service-to-service connections not present in the historical baseline
- **Exfiltration**: unusual egress volume to an unknown or unexpected external destination

Any SRE who can recognize these four patterns in their operational data will make better escalation decisions than the median security analyst who does not understand the platform. That is the target capability level.

---

## Director Frame

Build the bridge vocabulary. Train your SRE team on the four threat categories so they can make the escalation decision in 60 seconds, not 60 minutes.

The practical implementation is not a security training program for your SREs. It is two things: first, a set of correlated Splunk alerts on your reliability dashboards that surface the composed signal patterns described in this chapter; second, a one-page escalation guide that maps signal pattern to action ("if you see X, escalate to SOC and do Y before they arrive"). 

The result is SRE teams that route security incidents faster and preserve evidence correctly, without requiring each SRE to become a security analyst. That is the Director-level solution: design the system so individual judgment calls are replaced by pattern recognition against known categories.
