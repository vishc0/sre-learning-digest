# Chapter 8: Director Decision Framework

## The Decisions That Are Yours to Own

The previous seven chapters built the operational and technical vocabulary for the SRE-SOC interface. This chapter is about the decisions that sit above the operational layer — the choices that a Director or VP makes that constrain everything below them.

Three decisions matter most: which SIEM tool (and whether to extend Splunk or introduce a separate platform), which SRE-SOC organizational model, and how to structure compliance logging to manage cost without creating gaps. Each of these is a multi-year commitment. Getting them wrong has large organizational consequences.

---

## SIEM Tool Selection: The Three-Way Decision

In most large enterprises, this is not a blank-slate decision. There is already a SIEM or partial SIEM capability, and the decision is whether to extend what exists or replace it. The evaluation criteria below assume a Splunk-dominant observability platform, which is the context most relevant for a platform engineering leader.

**Option 1: Splunk Enterprise Security**

Best for organizations already running Splunk at scale, with experienced Splunk administrators, and an established Splunk data pipeline.

Strengths: The richest ecosystem of pre-built content (ESCU), the most mature Risk-Based Alerting implementation, native integration with your existing observability data (same indexes, same search layer). For an SRE team that already writes SPL daily, the ES learning curve is low.

Weaknesses: Highest licensing cost in the category. Splunk ES licensing is separate from core Splunk and priced by data volume. In large Kubernetes environments with high log volumes, this cost is significant. Splunk's cloud-native story (Splunk Cloud, Splunk Enterprise Cloud) has improved, but organizations that have moved aggressively to cloud-native architectures sometimes find ES's architecture feels heavy.

Cost model: ES is licensed per GB/day of data in the security indexes. Expect 20-40% on top of your existing Splunk licensing if you are routing all security-relevant sources to ES.

**Option 2: Microsoft Sentinel**

Best for organizations with heavy Azure investment, Microsoft 365 and Active Directory as primary identity providers, and a preference for pay-per-GB consumption pricing.

Strengths: Native integration with Azure AD, Microsoft 365 (Exchange, Teams, SharePoint), Defender XDR. Pay-as-you-go pricing with commitment tiers — lower upfront cost than ES for modest data volumes. Microsoft Copilot for Security integration for AI-assisted investigation.

Weaknesses: Weaker integration with non-Microsoft log sources (Linux, Kubernetes, AWS) compared to Splunk ES. The Kusto Query Language (KQL) is powerful but requires a parallel skill set from teams already fluent in SPL. On-premises log sources require additional forwarding architecture. For environments running hybrid or AWS-primary infrastructure, Sentinel's native advantages diminish.

Cost model: Consumption pricing per GB ingested. Commitment tiers reduce unit cost. Microsoft offers 90-day free data retention included; additional retention billed separately.

**Option 3: CrowdStrike SIEM (Falcon LogScale)**

Best for organizations with heavy CrowdStrike endpoint detection and response (EDR) adoption, prioritizing speed of log ingestion and search, and willing to invest in a platform that is newer in the SIEM market.

Strengths: Purpose-built for high-speed log ingestion. Excellent integration with Falcon EDR data — if your SOC is already living in the Falcon console, SIEM data in the same platform reduces context switching. Competitive pricing on ingest volume.

Weaknesses: Newer in the SIEM market — the content library (equivalent to ESCU) is less mature than Splunk or Sentinel. MITRE ATT&CK coverage is growing but not as comprehensive. Less penetration in large enterprise financial services and healthcare environments where Splunk ES is the established standard.

**Decision Table:**

| Criterion | Splunk ES | Microsoft Sentinel | CrowdStrike LogScale |
|-----------|-----------|-------------------|----------------------|
| Existing Splunk investment | Best | Poor | Neutral |
| Azure/M365 primary | Neutral | Best | Neutral |
| CrowdStrike EDR primary | Neutral | Neutral | Best |
| Pre-built SIEM content | Best | Good | Developing |
| SPL skill reuse | Best | None | None |
| Cloud-native architecture | Good | Best | Good |
| Compliance audit maturity | Best | Good | Developing |
| Total cost (high volume) | Highest | Medium | Lowest |

**Recommendation for T-Mobile-style platform:** Extend Splunk to ES. The existing Splunk expertise, the existing data pipeline, the MLTK investment, and the ESCU content library make ES the clear choice unless there is a specific organizational driver (Azure-primary direction, budget constraint) that changes the calculus.

## SRE-SOC Org Model Selection

The organizational model choice has more operational impact than the tool choice. A great tool deployed inside a broken org model delivers poor results. A mediocre tool inside a well-designed org model works.

**Model A: Siloed**

Structure: SRE and SOC are separate organizations with separate leadership, separate tools (even if the underlying data platform is shared), and no standing coordination mechanism. Handoff occurs at escalation time.

Operational cost: High incident cost during security incidents. Average time to contain security incidents is high because the first 30-60 minutes are spent establishing communication and context transfer rather than responding.

When this is acceptable: When security incident frequency is low, when the security incidents that do occur are not operationally impactful, and when organizational structure makes a closer model impossible.

**Model B: Embedded**

Structure: One or more SOC analysts report to the SRE Director (or are co-located with the SRE team with a dotted line to the CISO). Security review is part of the SRE team's daily operations.

Operational benefit: Fastest detection to containment. Security context is always present in the room. Security posture review is embedded in change management and production review processes.

Tradeoffs: Hard to staff — the skill set that combines deep operational platform knowledge with security analysis capability is rare and commands a significant salary premium. Some compliance frameworks (particularly FedRAMP and SOX in financial services) require organizational independence between infrastructure operators and security auditors. This model may create compliance risk in those contexts.

When this works: High-security environments, financial services and healthcare platform teams, organizations where security incidents have previously caused significant operational impact.

**Model C: Federated**

Structure: SRE and SOC maintain separate organizations, but share a data platform, shared playbooks, and a documented coordination protocol. Shared Splunk deployment (separate search heads, potentially, but same indexers). Joint tabletop exercises quarterly. Defined handoff protocol and escalation paths.

Operational benefit: Balances coordination cost and staffing feasibility. Both teams operate efficiently in normal conditions. During security incidents, the pre-agreed protocol kicks in and the parallel workstream model activates.

Tradeoffs: The protocol and shared playbooks require active maintenance. If the SRE Director and CISO are not actively coordinating, the shared infrastructure drifts apart, the playbooks become stale, and the model degrades toward siloed.

Best for: Enterprise-scale organizations where both SRE and SOC are mature teams with stable leadership. This is the right model for a T-Mobile-scale deployment.

## Compliance Logging Cost Management: The Four Levers

Compliance logging generates cost. Managing that cost without creating compliance gaps requires four specific levers.

**Lever 1: Data volume reduction at collection.** Not every event needs to be ingested. Filter noise at the collection layer (Heavy Forwarder or log aggregator) before data reaches Splunk. For compliance purposes, the events that are not security-relevant and not required by the applicable framework can be dropped or summarized at the source. Example: HTTP 200 responses from an API gateway may not be required for compliance; HTTP 401/403/500 events are. Filter the 200s at the forwarder.

**Lever 2: Tiered storage (SmartStore).** Keep compliance data in Splunk SmartStore, which automatically tiers index buckets from hot SSD to S3 based on age. Analysts query through the same interface; Splunk handles the retrieval. S3 storage costs are roughly 5-10% of equivalent SSD storage costs. This alone reduces compliance logging infrastructure cost by 60-80% compared to keeping everything in hot storage.

**Lever 3: Index partitioning.** Route compliance data to dedicated compliance indexes with their own retention policies, separate from observability indexes. This prevents compliance data from being subject to observability retention settings (typically shorter) and prevents observability data from being retained longer than necessary due to compliance requirements. Index partitioning also simplifies audits — the auditor queries the compliance index directly, rather than searching through mixed-purpose indexes.

**Lever 4: Summary indexing for reporting.** For compliance reporting (not investigation), pre-aggregate compliance data into summary indexes using scheduled searches. Compliance reports that need to show "number of login events by day for the last 12 months" can be served from a summary index, not from scanning raw events. This reduces audit query cost and reduces the performance impact of compliance reporting on operational Splunk infrastructure.

## The Business Case for SRE-SOC Integration

When proposing the investment in integrated SRE-SOC operations to leadership:

**MTTD reduction:** Security incidents detected earlier mean shorter breach duration. Industry data consistently shows that every additional hour of undetected breach increases data exposure scope and remediation cost. Quantify this in terms of your own security incident history — how long did detection take in the last three notable incidents?

**Duplicate tooling elimination:** If SOC is running a separate SIEM because the SRE observability platform does not serve their needs, you are paying twice for log storage and processing. Consolidation to a shared Splunk deployment eliminates that duplication.

**Compliance audit pass rate:** A platform team that owns compliance logging architecture, maintains the log inventory, and can provide auditor access to structured compliance indexes will pass audits with fewer findings than a team that manages compliance logging reactively. Each audit finding that requires a remediation plan consumes engineering time and creates audit risk in subsequent years.

**SRE efficiency during security incidents:** With a documented handoff protocol, SRE teams spend less time on security incidents. The protocol defines their scope, they execute it, and they return to normal operations. Without the protocol, SRE teams are involved in security investigations for hours or days after the incident because they are the only ones with platform access and context.

## Director Interview Talking Points

The question: "How do you think about the SRE-SOC interface at enterprise scale?"

Two-minute answer structure:

"The SRE-SOC interface is fundamentally an organizational design problem, not a technology problem. My approach is a federated model: SRE and SOC as separate teams with separate workflows, but on a shared data platform. We run Splunk as the common data layer — same indexers, separate search heads for operational and security contexts, shared index strategy. The key investment is not in tools, it is in the protocol: a documented, drilled handoff procedure for incidents that cross the reliability-security boundary, a defined parallel workstream structure so SRE and SOC are not stepping on each other during a security incident, and a shared log inventory that keeps both teams aligned on what data is available and what is not.

The specific capability I have built in previous roles is extending MLTK anomaly detection models — which the SRE team already uses for reliability — to serve as threat hunting leads for the SOC. The same behavioral baselines that detect a latency anomaly also detect an exfiltration pattern. That makes your existing reliability investment do double duty on the security side without requiring a separate platform purchase."

## Director Frame

The SRE-SOC integration decision is an organizational design decision, not a technology decision. Choose the org model first, then choose the tools that support it.

The directors who get this wrong start with the tool selection. They buy Splunk ES, install it, and then discover that nobody has agreed on who escalates to whom, who the incident commander is, or whether SREs can restart compromised pods. The tooling is excellent. The process is missing. The result is that the tool is underused.

The directors who get this right start with: "What organizational model do I want, and what does that model require in terms of shared infrastructure, shared protocols, and shared practice?" Then they select tools that support the model they have chosen. In most large enterprise environments, that model is federated, and that tool is Splunk ES. But the model comes first.
