# Chapter 5: SRE-SOC Incident Handoff

## The Handoff Is Not a Handoff

The most common mistake in SRE-SOC coordination during a security incident is framing the interaction as a handoff — as if SRE finishes its work, packages up the incident, and tosses it over the wall to the SOC team. That model fails for a structural reason: service restoration and security investigation must happen simultaneously, not sequentially.

When a security incident is underway, SRE cannot stop operating. Services need to be contained, blast radius needs to be managed, and availability must be maintained where it is safe to do so. At the same time, SOC cannot wait for SRE to finish before beginning an investigation — the longer the delay before forensic evidence is collected, the more evidence is lost to normal operations (log rotation, pod restarts, cache eviction).

The correct mental model is parallel workstreams, not sequential handoff. SRE owns service restoration. SOC owns investigation. Both run simultaneously with a shared incident channel, a shared timeline document, and explicit scope agreements for each team.

---

## When an SRE Incident Becomes a Security Incident

Three trigger conditions. Any one is sufficient to activate security escalation — do not wait for multiple confirmations.

**Trigger 1: Evidence of unauthorized access.** Authentication success events from credentials that the account owner has reported as compromised, from a geographic location inconsistent with the user, from a source IP appearing on threat intelligence feeds, or from a service account making interactive logins (service accounts should not be logging in interactively).

**Trigger 2: Evidence of data manipulation.** Write operations to data stores that should be read-only in normal operations, bulk delete events, schema changes outside of a change window, or configuration changes to security-relevant systems (RBAC, network policies, secret stores) without a corresponding change ticket.

**Trigger 3: Evidence of lateral movement.** New service-to-service connections that have no precedent in historical traffic patterns, port scanning activity originating from internal systems, or process execution on nodes that is inconsistent with the workload's declared container image.

One trigger is sufficient. The instinct to gather more evidence before escalating is wrong. Escalate when you see one trigger, continue investigating while SOC spins up, and correct if the escalation proves unnecessary. The cost of an unnecessary SOC page is low. The cost of a 45-minute delay in escalation is high.

## The Vocabulary Conflict That Must Be Resolved Before an Incident

The single most operationally damaging disagreement between SRE and SOC teams is about pod and container state preservation.

**SRE instinct:** The compromised pod is causing the incident. Restart it, which will bring up a clean container from the image and restore service. This is the correct remediation action for a reliability incident.

**SOC requirement:** The running pod contains forensic evidence. Its memory state, its process list, its active network connections, and its ephemeral filesystem contain artifacts that can tell investigators exactly what happened and how. Restarting the pod destroys all of this.

These two requirements are in direct conflict, and they cannot be resolved during the incident under time pressure by people who have not agreed on the answer in advance.

The pre-agreed resolution must be documented in the SRE-SOC runbook before any incident occurs. One reasonable resolution:

- If the service is fully unavailable and the compromised pod must be restarted to restore service: capture a memory dump and a filesystem snapshot before restart, notify SOC of the restart action and provide the artifact location, restart the pod.
- If the service can be maintained at reduced capacity by routing around the compromised pod: isolate the pod (remove it from service load balancer, apply a NetworkPolicy to block its egress) and preserve it for SOC investigation.
- If there are multiple compromised pods: restart all but one; preserve one for forensic investigation.

The specific policy matters less than the fact that it has been agreed upon and documented. Agreeing during an incident is too late.

## Pre-Handoff SRE Actions

Before paging SOC, capture the following. This information should transfer at the moment of escalation, not 20 minutes later.

1. **Timeline of observed events:** When did the first anomaly appear? What was the first alert? What time did the unusual activity begin in the logs? This timeline is the foundation of the SOC's investigation.

2. **Affected systems inventory:** Which pods, which nodes, which services, which namespaces are involved? What are the affected service account names?

3. **Evidence preserved:** What has been preserved, and where? Memory dumps, pod filesystem snapshots, current network connection state, Splunk search links showing the raw events.

4. **Containment actions taken:** What has SRE already done? If you have applied NetworkPolicies to isolate a pod, tell SOC. If you have revoked a service account, tell SOC. If you have rotated a credential, tell SOC. Actions taken before SOC arrival directly affect the forensic picture.

5. **Initial hypothesis:** What does SRE think is happening? This is input to SOC's investigation, not a conclusion. "We think this is a compromised service account based on unusual RBAC mutations" is useful context.

## Handoff Checklist

At the moment of SOC escalation, transfer these ten items:

| # | Item | Where to find it |
|---|------|-----------------|
| 1 | Incident timeline (first anomaly → current time) | Shared incident document |
| 2 | List of affected workloads (pod names, namespaces, node IPs) | kubectl get pods + audit log |
| 3 | Network segments involved | VPC flow logs, Kubernetes namespace network policies |
| 4 | Data stores potentially in scope | Service dependency map + unusual access logs |
| 5 | Credentials potentially exposed | Service account names, any human credentials touched |
| 6 | Containment actions taken (with timestamps) | Incident timeline |
| 7 | Evidence preserved and its location | S3 path or Splunk search link |
| 8 | Current service availability status | SLO dashboards, health check endpoints |
| 9 | Active responders and their roles | Incident channel participant list |
| 10 | Communication bridge (incident channel link) | Slack/Teams incident channel URL |

## Parallel Workstream Setup

Once SOC is active, the incident has two parallel workstreams. This is the standard operating structure for any security-flavored incident.

**SRE workstream — Service Restoration:**
- Contain the blast radius (route traffic away from affected systems, apply network isolation)
- Maintain service availability for unaffected components
- Provision log access and system access to SOC as requested
- Document all actions taken in the shared incident timeline
- Do not take any action that would destroy forensic evidence without SOC awareness

**SOC workstream — Investigation:**
- Collect and preserve evidence (memory, filesystem, network state)
- Scope the incident (how many systems affected, what data is in scope)
- Determine the initial access vector and attack timeline
- Assess data exposure scope
- Manage external communication (legal, compliance, regulators, PR)
- Drive containment decisions that conflict with SRE's restoration actions

**Incident commander:** One person holds this role. During a security incident, the incident commander is typically from SOC (because external communication and legal implications are in scope). The SRE lead operates as the technical lead for the restoration workstream, reporting into the incident commander. This prevents the dual-commander problem where both teams are making conflicting decisions.

## Evidence Preservation Rules for SREs

Things SREs must not do during a security incident without explicit SOC awareness:

- **Do not restart a compromised pod** unless the service is fully unavailable and the restart has been discussed with SOC. Provide at minimum a memory dump before restart.
- **Do not delete log data** for any reason. If a pipeline error is causing log ingestion issues, fix the pipeline but do not delete the source logs.
- **Do not rotate credentials without SOC awareness.** Rotating a credential destroys the ability to track where that credential was used by an adversary. SOC must coordinate credential rotation as part of the eradication phase.
- **Do not apply patches or update container images** on compromised systems. Patching overwrites the exploit artifacts that allow SOC to understand what happened.
- **Do not restore from backup** without SOC reviewing the backup first. The backup may itself be compromised if the adversary had sufficient access.

## Escalation Beyond SOC

Some security incidents require escalation beyond SOC to functions outside the technology organization.

**Legal hold:** When an incident may result in litigation or regulatory inquiry, legal counsel must be notified and must declare legal hold. Once legal hold is declared, all relevant data must be preserved and cannot be deleted, modified, or overwritten. This includes Splunk data, even if it falls outside normal retention periods.

**Regulatory notification:** Incidents involving personal data exposure may trigger regulatory notification requirements. GDPR requires notification to data protection authorities within 72 hours of becoming aware of a breach. HIPAA requires notification within 60 days. PCI-DSS requires notification to the acquiring bank immediately upon discovery. These clocks start running when the incident is identified, not when investigation concludes. SOC (with legal) owns this decision, but SRE must understand the timeline.

**Law enforcement:** In cases involving criminal activity (ransomware, nation-state actors, significant fraud), SOC may engage law enforcement. This changes evidence handling requirements significantly — evidence must now meet chain-of-custody standards.

## Director Frame

The SRE-SOC handoff protocol should be documented, drilled, and tested before a real incident. This is not aspirational — it is the same discipline applied to reliability runbooks. You do not write a database failover runbook after the database fails. You write it, test it, and drill it so that the actual failover is mechanical execution rather than improvisation under pressure.

The specific deliverable a Director should own: a documented, SOC-reviewed, jointly drilled Security Incident Response Protocol that covers the three trigger conditions, the handoff checklist, the parallel workstream structure, and the evidence preservation rules. The protocol should be tested annually via a tabletop exercise where SRE and SOC work through a simulated security incident together.

Organizations that have this protocol have measurably lower MTTC (mean time to contain) for security incidents, because the first thirty minutes of a real incident are spent executing the protocol rather than figuring out who calls whom.
