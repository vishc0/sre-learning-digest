# Chapter 7: Threat Hunting with SRE Data

## The Extension You Have Not Made Yet

You already use Splunk MLTK for anomaly detection in your SRE reliability workflow. You have built models that establish normal behavior for your services — baseline call rates, typical latency distributions, expected error rate envelopes — and you alert when the observed behavior deviates beyond a threshold from that baseline.

This chapter extends that exact capability to security threat hunting. The models are the same. The data is largely the same. What changes is the investigative path you walk when the anomaly fires.

The insight worth holding: reliability anomalies and security threats often produce the same signal. A DDoS and a legitimate traffic spike look identical to a latency monitor. A crypto-miner and a legitimately resource-intensive batch job both cause CPU to spike. The difference between an SRE investigation and a threat hunting exercise is not the initial detection — it is what you do with the anomaly once you have found it.

---

## How Availability and Performance Anomalies Become Threat Hunting Leads

Not every anomaly is a security event. But security events almost always produce availability and performance anomalies. This asymmetry makes SRE monitoring data a natural source of threat hunting leads — you are already watching for deviations from normal; the question is whether you are equipped to interpret those deviations through a security lens.

**Latency spike in a downstream service** can indicate:
- A slow loris attack: the attacker is sending partial HTTP requests that hold connections open without completing them, exhausting the connection pool
- SQL injection: a malicious query is executing a much longer operation than the intended query
- A resource-constrained pod being used for mining: the legitimate service is starved of CPU

**Unusual pod restart pattern** can indicate:
- A container escape attempt where the exploit causes the container process to crash
- A crypto-miner being evicted by the resource limits system and restarting from a sidecar
- An adversary testing different exploit payloads against a target container

**Outbound bandwidth anomaly** can indicate:
- Data exfiltration in progress
- A compromised pod being used as a relay for traffic to a third target
- A botnet C2 connection maintaining persistent communication

**Auth failure spike followed by auth success** is the clearest pattern:
- The auth failure spike is credential stuffing
- The auth success is the moment a valid credential was found
- Everything that happens after the auth success is the actual breach

When MLTK flags these patterns as anomalies, the SRE instinct is to diagnose the reliability root cause. Threat hunting adds a second question to the investigation: "Is this anomaly explainable by a reliability cause, or does the security hypothesis fit the data better?"

## Behavioral Baselines as Hunting Infrastructure

A behavioral baseline is a model of what normal looks like for a given entity — a service, a user, a pod, a namespace. Deviations from baseline are hunting leads.

This is identical to the SLO baseline model. Your reliability baselines establish normal call rates, normal error rates, normal latency distributions. When you extend this to security, you add baselines for:

- **Calls per minute, by service:** Established. You already have this.
- **Source IP distribution, by service endpoint:** New for most SRE teams. Which external IPs typically call a given endpoint? What is the cardinality? A login endpoint normally sees a wide distribution of IPs. If a small cluster of IPs suddenly represents 80% of traffic, that is a hunting lead.
- **Egress byte count, by pod:** Normal egress volume for a given workload is predictable. Deviations indicate either a data-heavy operation (check if there is a corresponding change deployment) or exfiltration.
- **New destination IPs, by time window:** How many unique external IP destinations does a pod contact per hour? A pod that has contacted 3-5 external IPs for months suddenly contacting 50 external IPs in an hour is a hunting lead.
- **API call patterns, by service account:** Which API verbs does a service account call, against which resources, at what rate? A service account that has never called `GET /secrets` suddenly reading secrets is a hunting lead regardless of whether it is authorized.

## MLTK Threat Hunting Queries

These build on the same MLTK foundation you have already built for reliability anomaly detection.

**Establish a density function baseline for authentication failure rates:**

```spl
| tstats count AS auth_failures FROM datamodel=Authentication 
    WHERE Authentication.action=failure 
    BY Authentication.src, _time span=5m
| fit DensityFunction auth_failures BY Authentication.src 
    into auth_failure_baseline
```

This is the same pattern you use to baseline service call rates. The model learns what a typical authentication failure rate looks like per source IP. A source IP that suddenly generates ten times its normal failure rate is flagged as an anomaly.

**Apply the model and surface anomalies:**

```spl
| tstats count AS auth_failures FROM datamodel=Authentication 
    WHERE Authentication.action=failure 
    BY Authentication.src, _time span=5m
| apply auth_failure_baseline 
| where IsAnomaly=1 AND BoundaryRanges IS NOT NULL
| sort -auth_failures
```

**Establish a baseline for egress volume by pod:**

```spl
index=network_flow direction=outbound 
    earliest=-30d latest=-1d
| stats sum(bytes) AS egress_bytes BY pod_name, _time span=1h
| fit DensityFunction egress_bytes BY pod_name 
    into pod_egress_baseline
```

**Detect pods with anomalous egress and join with threat intelligence:**

```spl
index=network_flow direction=outbound earliest=-1h
| stats sum(bytes) AS egress_bytes, values(dest_ip) AS dest_ips BY pod_name
| apply pod_egress_baseline 
| where IsAnomaly=1
| mvexpand dest_ips
| lookup threatintel_by_cidr src AS dest_ips 
    OUTPUT threat_category, confidence
| table pod_name, egress_bytes, dest_ips, threat_category, confidence
```

**Identify service accounts with new API call patterns (first-seen analysis):**

```spl
index=k8s_audit 
    user.username!=system:* 
    user.username!="kubernetes-admin"
| eval call_signature=user.username + "|" + verb + "|" + objectRef.resource
| first_time call_signature
    [search index=k8s_audit earliest=-90d 
        | stats values(eval(verb + "|" + objectRef.resource)) AS seen_before BY user.username
        | return user.username, seen_before]
| where first_time=1
| stats count by user.username, verb, objectRef.resource
```

Simpler version using the MLTK `first_time_ever` command from the Anomaly Detection content pack:

```spl
index=k8s_audit objectRef.resource=secrets verb=get
| eval hunting_key=user.username + " accessed secrets"
| `first_time_field(hunting_key, 90d)`
| where first_time=1
| table _time, user.username, objectRef.namespace, objectRef.name
```

**MLTK Showcase App connection:** The fraud detection models in the MLTK Showcase App apply directly to insider threat detection. The `fit LocalOutlierFactor` algorithm, used in the showcase for transaction fraud, applies equally to identifying service accounts whose API call patterns deviate from their cohort's behavior.

## Proactive Hunting vs. Reactive Alert Response

Alert response is reactive: an alert fires, an analyst responds. The analyst's goal is to determine whether the alert is a true positive and if so, what to do about it.

Threat hunting is proactive: an analyst starts with a hypothesis about attacker behavior and goes looking for evidence of that behavior in the data, without waiting for an alert to fire. The hypothesis might be: "A threat actor who has compromised a service account would need to read secrets before they could use those credentials to pivot to another service. Are there any service accounts reading secrets they have never read before?"

That hypothesis does not require a correlation search to fire. It requires an analyst with access to good data, a model of normal behavior to compare against, and the query capability to test the hypothesis efficiently.

The role of the platform team in enabling threat hunting:

1. **Make the data available:** Good CIM normalization, complete log sources, accessible indexes.
2. **Make normal behavior modelable:** MLTK baselines that can be applied against any time window.
3. **Make hypothesis testing fast:** Pre-built SPL macros for common hunting queries so an analyst can test a hypothesis in minutes rather than building queries from scratch.
4. **Provide context enrichment:** Asset and identity lookups that translate raw IP addresses and service account names into business context.

The platform team does not conduct the hunt. The platform team makes the hunt possible at speed.

## Sample Hunting Hypotheses for a Large-Scale Platform

**Hypothesis 1: Pods making first-ever outbound calls to external IPs**

Rationale: A compromised pod making a C2 connection will typically contact an IP it has never contacted before. New external destinations are hunting leads.

```spl
index=network_flow direction=outbound earliest=-24h
| stats min(_time) AS first_seen BY pod_name, dest_ip
| where first_seen > relative_time(now(), "-24h")
| lookup ip_enrichment dest_ip OUTPUT geo_country, asn_org, is_known_cloud
| where is_known_cloud=0
| table pod_name, dest_ip, geo_country, asn_org, first_seen
```

**Hypothesis 2: Service accounts with unusual API call timing patterns**

Rationale: Service accounts that are being used by an adversary rather than by their intended automation will often show activity at unusual hours — when the adversary's operating timezone overlaps with their work hours, which may not match the legitimate service's operational hours.

```spl
index=k8s_audit 
    user.username!="system:*"
| eval hour_of_day=strftime(_time, "%H"), 
       day_of_week=strftime(_time, "%w")
| stats count BY user.username, hour_of_day, day_of_week
| fit DensityFunction count BY user.username 
    into sa_timing_baseline
| where IsAnomaly=1 AND (hour_of_day < 6 OR hour_of_day > 22)
```

## Director Frame

Threat hunting is expensive — it requires skilled analyst time and good tooling. Your job as the platform Director is to make the data so good and the tooling so efficient that hunters can test a hypothesis in minutes, not days.

The MLTK baselines you have already built for reliability anomaly detection are the force multiplier. A threat hunter who can run `apply pod_egress_baseline` against the last 24 hours of network flow data and immediately see which pods have anomalous egress, enriched with threat intelligence, is testing a hypothesis in five minutes that would take an hour to test against raw, unmodeled data.

The investment framing for your leadership: "We have already paid for the MLTK infrastructure to support reliability anomaly detection. Extending it to security threat hunting is an incremental investment in query design and model extension, not a new platform purchase. The marginal cost of the threat hunting capability is low because the data and the modeling infrastructure already exist."

That is a Director argument. It connects existing investment to new capability without requiring additional platform spend.
