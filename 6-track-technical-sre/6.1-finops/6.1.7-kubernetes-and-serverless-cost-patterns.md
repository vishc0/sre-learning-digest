# Chapter 7: Kubernetes and Serverless Cost Patterns

## Choosing the Right Compute Model Is a Cost Decision

When a development team asks "should we run this on EKS, Fargate, or Lambda?" they are usually asking about operational simplicity. From a FinOps perspective, they are also making a significant cost decision — one that could be a 2x–10x difference in monthly spend depending on workload characteristics.

The Director's job is to ensure that the architecture conversation includes the cost conversation. This chapter gives you the cost anatomy of each compute model and the decision framework for matching workload type to the right model.

---

## EKS Cost Anatomy

EKS is not a single line item — it is an aggregate of five separate cost components. Understanding each one is prerequisite to optimizing any of them.

### Component 1: EKS Cluster Management Fee

**Cost**: $0.10/hour = **$72/month per cluster**

This is a flat fee per cluster. It covers the managed control plane (API server, etcd, controller manager). It does not scale with node count or workload size.

**Implication**: Running 10 small clusters is more expensive in management fees than running 1 cluster with namespaces. Cluster consolidation is a FinOps lever — reduce cluster count, reduce management fee overhead.

### Component 2: EC2 Worker Nodes

**Cost**: Varies by instance type. The dominant cost component.

The EC2 cost is determined by:
- Instance type (compute vs memory vs GPU optimized)
- Pricing model (on-demand, Spot, Reserved)
- Number of nodes × hours running

For a typical 12-node m5.2xlarge cluster running on-demand: 12 × $0.384/hr × 720 hr = **$3,317/month** before any discounts.

**Optimization levers**: Right-size instance type, convert to Spot for stateless workloads, purchase Savings Plans/RIs for stable baseline.

### Component 3: EBS Volumes

**Cost**: gp3 = $0.08/GB-month; io2 = $0.125/GB-month + $0.065/IOPS-month

EBS is persistent storage attached to nodes. In Kubernetes, it appears as PersistentVolumes. Common drivers of EBS cost:
- StatefulSets with large volume requests
- Log retention on the node (write to local EBS instead of CloudWatch or S3)
- Database backups written to EBS instead of S3

**Optimization**: Audit PVC sizes vs actual usage (most PVCs are oversized). Move log and backup data to S3 (far cheaper). Consider gp3 over gp2 for all new volumes — gp3 delivers the same baseline performance at 20% lower cost.

### Component 4: Data Transfer

**Cost**: $0.09/GB for inter-AZ traffic; $0.09/GB for internet egress (first 10TB/month)

Data transfer is the hidden cost in Kubernetes. Every request that crosses an AZ boundary — pod talking to a service in another AZ, a database query to a replica in a different AZ — incurs this charge. At scale, this is not trivial.

**Example**: A service making 25 million API calls/day to a Cassandra cluster in a different AZ, with 10KB average response. That is 250GB/day of inter-AZ traffic × $0.09 = **$22.50/day = $675/month** in data transfer alone.

**Optimization levers**:
- Topology-aware service routing: instruct Kubernetes to prefer same-AZ endpoints when available. EKS Topology Aware Routing achieves this.
- Cassandra rack awareness configured to serve reads from the same AZ as the application pod
- Reduce CloudWatch log export volume (high log verbosity drives egress)

### Component 5: Load Balancers and Network

**Cost**: ALB = $16.20/month base + $0.008/LCU-hour; NLB = $16.20/month base + $0.006/NLCU-hour

Every `Service` of type `LoadBalancer` provisions an AWS load balancer. Clusters with many services create many load balancers, each with their base hourly cost.

**Optimization**: Use an Ingress controller (AWS ALB Ingress Controller or NGINX) to consolidate multiple services behind one load balancer. Replace individual `LoadBalancer` services with `ClusterIP` + Ingress for HTTP workloads.

## EKS Cost Summary Model

```
Total EKS Monthly Cost =
  $72 (management fee)
  + (EC2 node count × instance price × hours × on-demand premium factor)
  + (PVC total GB × $0.08)
  + (inter-AZ GB transferred × $0.09)
  + (load balancer count × $16.20)
  + CloudWatch log ingestion (if using Container Insights)
```

For a 12-node m5.2xlarge cluster with typical usage, the breakdown is roughly:
- EC2: 65%
- EBS: 10%
- Data transfer: 8%
- Management fee: 5%
- Load balancers: 7%
- CloudWatch: 5%

## Fargate Cost Model

Fargate removes the need to manage EC2 nodes. AWS provisions compute per pod, charges per vCPU-second and per GB-second of memory.

**Cost**: $0.04048/vCPU-hour + $0.004445/GB-hour

**Example**: A pod with 0.5 vCPU and 1GB running 720 hours/month:
- vCPU: 0.5 × $0.04048 × 720 = $14.57
- Memory: 1.0 × $0.004445 × 720 = $3.20
- **Total: $17.77/pod/month**

Equivalent EC2 cost for the same workload on a shared m5.large node (with 2 vCPU and 8GB, shared among 4 pods): approximately **$8.40/pod/month** on on-demand, **$3.20/pod/month** on Spot.

### Fargate vs EC2 Cost Comparison

| Workload Type | Fargate | EC2 On-Demand | EC2 Spot |
|--------------|---------|---------------|---------|
| Consistent, always-on | 1.8–2.5x more expensive | Baseline | 0.3–0.5x |
| Batch/burst (short duration) | Competitive | Over-provisioned | Overkill |
| Dev environments | No idle waste | Idle nodes wasteful | Often cheapest |
| Low-traffic services | No minimum charge waste | Idle EC2 costly | Best choice |

**When Fargate wins on cost**: When your workloads are burst-heavy, bursty, or intermittent. A batch job that runs for 30 minutes, 4 times a day, costs almost nothing on Fargate. On EC2, you pay for the node 24 hours even if the job runs 2 hours.

**When EC2 wins on cost**: Always-on workloads at scale. If a pod runs 100% of the time and you have many pods, EC2 (especially Spot) is almost always cheaper than Fargate.

## Lambda Cost Model

Lambda charges per invocation and per GB-second of execution duration.

**Cost**: $0.20 per 1M invocations + $0.0000166667 per GB-second

**Example**: A notification preprocessing function — 1M invocations/day, 200ms average duration, 512MB memory:
- Per invocation: $0.20 per 1M × 30M/month = $6.00
- Per GB-second: (0.5 GB × 0.2 sec) × 30M invocations × $0.0000166667 = $50.00
- **Total: ~$56/month**

Equivalent on EC2 (4x m5.large, 24/7): ~$280/month on-demand.

**Lambda wins when**: Invocation volume is moderate, functions execute quickly, and workload is spiky. The serverless model means you pay nothing when traffic is zero.

**Lambda loses when**: Functions are long-running (>15 min limit), need persistent connections (Lambda cold starts break long-lived connections), need shared state (each invocation is stateless), or volume is consistently high enough that the per-invocation pricing exceeds the equivalent EC2 cost.

## Cost Comparison Framework: Choosing the Right Model

| Workload Characteristic | Best Model | Reasoning |
|------------------------|------------|-----------|
| Always-on, high RPS | EKS + EC2 + Spot | Per-second pricing; Spot lowers cost 60–80% |
| Batch, short-lived | EKS + Fargate OR Lambda | No idle cost; burst-friendly |
| Event-driven, spiky | Lambda | Pay only for invocations |
| Stateful (DB, broker) | EKS + EC2 + RI | Stable baseline; RI discount applies |
| Dev / test workloads | EKS + Spot OR Fargate | No long-term commitment; interruptible |
| Scheduled jobs (cron) | Lambda | Zero cost when idle |
| ML training | EC2 Spot (GPU) | GPU Spot instances; preemptible for long jobs |

## The Director-Level Architecture Cost Review

When an engineering team proposes a new service, the Director-level cost review asks three questions:

1. **What is the projected steady-state cost per month?** (Model it before building it.)
2. **What is the cost scaling model?** (Does cost grow linearly with traffic? Sub-linearly? That tells you whether the architecture is cost-efficient at scale.)
3. **What commitment vehicle will cover the stable baseline?** (Every new service that will run 24/7 should have a RI or Savings Plan plan from day one.)

Embedding these questions in the architecture review process (the ADR template, the system design doc template) is how a Director builds FinOps into the engineering culture rather than applying it as a retroactive audit.

---

## Key Takeaways

- EKS cost is a five-component aggregate: management fee, EC2, EBS, data transfer, load balancers — optimize each independently
- Data transfer (inter-AZ) is the hidden cost driver in multi-AZ Kubernetes — topology-aware routing is the primary mitigation
- Fargate is cost-competitive only for burst/intermittent workloads; always-on workloads belong on EC2 (preferably Spot)
- Lambda is the cheapest model for event-driven, short-duration, spiky workloads — cost scales to zero at zero traffic
- The cost comparison table (workload type → best compute model) is a Director-level decision tool, not a developer preference discussion
- Embed cost modeling questions in the architecture review process to catch expensive design choices before they are built
