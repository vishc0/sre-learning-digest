# Reference: Kubernetes Cost Optimization Patterns

**Purpose**: 12 patterns for reducing Kubernetes cost. Quick-lookup format: pattern name → what it does → when to use it → implementation notes → expected savings.  
**Applicability**: EKS on AWS; most patterns apply to GKE and AKS with minor variations.

---

## Pattern 1: Resource Request Right-Sizing

**What it does**: Reduces CPU and memory requests to match actual workload usage, reducing the amount of node capacity reserved per pod.

**When to use**: CPU efficiency < 50% OR memory efficiency < 50% on any deployment (visible in Kubecost efficiency report).

**Implementation**:
1. Deploy VPA in recommendation mode
2. Wait 14+ days for representative data
3. Apply VPA target values + 20% headroom as new requests
4. Verify with rollout + 48-hour observation window

**Expected savings**: 15–25% EC2 reduction for typical over-provisioned clusters. Higher if requests were set without observability data.

**Risk**: Low for stateless workloads with ≥ 3 replicas. Medium for stateful or singleton deployments.

**Tooling**: VPA (included in EKS), Kubecost (provides efficiency data)

## Pattern 2: Spot Instances for Stateless Workloads

**What it does**: Replaces on-demand EC2 nodes with Spot instances for workloads that can tolerate interruption.

**When to use**: Stateless application deployments with HPA enabled and ≥ 3 replicas. Also: all non-production environments.

**Implementation**:
```yaml
# EKS managed node group with Spot
eksctl create nodegroup \
  --cluster=my-cluster \
  --name=spot-workers \
  --spot \
  --instance-types=m5.2xlarge,m5a.2xlarge,m4.2xlarge \
  --nodes-min=2 \
  --nodes-max=20

# Karpenter NodePool with Spot preference
apiVersion: karpenter.sh/v1beta1
kind: NodePool
spec:
  template:
    spec:
      requirements:
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["spot", "on-demand"]
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: karpenter.sh/capacity-type
```

**Instance type diversification**: Always specify 3+ instance types to reduce Spot interruption probability. AWS Spot interruption rate for diversified pools is typically < 5%.

**Expected savings**: 60–80% EC2 cost reduction for the Spot-eligible workload fraction. If 50% of compute moves to Spot, expect 30–40% overall EC2 cost reduction.

**Risk**: Spot interruption causes pod termination with 2-minute notice. Applications must handle graceful shutdown.

## Pattern 3: Cluster Autoscaler Tuning

**What it does**: Configures Cluster Autoscaler to scale down aggressively when nodes are underutilized.

**When to use**: Any EKS cluster with Cluster Autoscaler. Scale-down is often misconfigured or disabled.

**Key parameters to tune**:
```yaml
# cluster-autoscaler deployment args
--scale-down-enabled=true
--scale-down-utilization-threshold=0.5   # Scale down if node < 50% utilized
--scale-down-unneeded-time=10m           # Wait 10 min before scaling down
--scale-down-delay-after-add=10m         # Wait after scale-up before scaling down
--skip-nodes-with-system-pods=false      # Allow system pods to be moved (except DaemonSets)
```

**Common misconfiguration**: `--scale-down-enabled=false` left in place from a scaling issue incident. Check this first when idle cost is high.

**Expected savings**: Directly reduces node count during low-traffic periods. For clusters with daily traffic patterns (peak day, valley night), scale-down can reduce overnight EC2 cost by 40–70% for the application node group.

**Alternative**: Karpenter (preferred for new clusters) provides faster and more intelligent node provisioning and consolidation without Cluster Autoscaler.

## Pattern 4: Karpenter Node Consolidation

**What it does**: Automatically identifies nodes where running pods can be rescheduled on other nodes, drains the node, and terminates it.

**When to use**: EKS clusters running Karpenter v0.27+. This is a continuous background optimization.

**Implementation**:
```yaml
apiVersion: karpenter.sh/v1beta1
kind: NodePool
metadata:
  name: default
spec:
  disruption:
    consolidationPolicy: WhenUnderutilized
    consolidateAfter: 30s      # How long to wait before consolidating
  limits:
    cpu: "1000"
    memory: 1000Gi
```

**Expected savings**: Ongoing. Typically 10–20% additional EC2 reduction beyond right-sizing, as fragmented nodes that could not scale down are consolidated.

**Risk**: Pods are rescheduled, causing brief interruption. Ensure PodDisruptionBudgets protect critical services.

## Pattern 5: Namespace Resource Quotas

**What it does**: Prevents individual namespaces from consuming unbounded cluster resources, protecting the cluster from a single runaway deployment.

**When to use**: All production and staging namespaces. Required for any namespace with a cost SLO.

**Implementation**:
```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
  namespace: payments-prod
spec:
  hard:
    requests.cpu: "20"
    requests.memory: 40Gi
    limits.cpu: "40"
    limits.memory: 80Gi
    count/deployments.apps: "20"
    count/services: "15"
    persistentvolumeclaims: "10"
```

**Expected savings**: Preventive — avoids cost spikes from runaway deployments. Does not reduce existing cost but caps future growth.

**Risk**: Quota too tight causes legitimate deployments to fail. Set quotas at 150% of current peak namespace usage, review quarterly.

## Pattern 6: LimitRange Defaults

**What it does**: Sets default resource requests and limits for containers that do not specify them, preventing "unmetered" resource consumption.

**When to use**: Any namespace where developers might deploy without setting requests/limits.

**Implementation**:
```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: payments-prod
spec:
  limits:
  - default:
      cpu: "500m"
      memory: "512Mi"
    defaultRequest:
      cpu: "100m"
      memory: "128Mi"
    type: Container
```

**Expected savings**: Preventive. Without LimitRange, containers with no requests are allocated 0 requests — Kubecost shows them as free but they consume node capacity, appearing as idle cost.

## Pattern 7: Non-Production Environment Scheduling

**What it does**: Automatically stops (scales to zero) non-production Kubernetes nodes or deployments during off-hours.

**When to use**: Any dev, staging, or feature branch environment that does not need 24/7 uptime.

**Implementation options**:
- **Kube-downscaler**: Open source tool that scales deployments to 0 replicas on a schedule. Runs as a deployment in the cluster.
- **Node group schedule**: AWS Auto Scaling Group scheduled scaling action to reduce node group to 0 nodes on a schedule.
- **Namespace annotation**: kube-downscaler watches for namespace or deployment annotations specifying uptime schedule.

```yaml
# Annotation on namespace or deployment
annotations:
  downscaler/uptime: "Mon-Fri 08:00-19:00 America/Los_Angeles"
```

**Expected savings**: Non-prod typically runs 168 hours/week. Restricting to 55 hours/week (business hours Mon-Fri) reduces non-prod EC2 cost by ~67%. For a platform team spending $4,000/month on staging, this is ~$2,680/month in savings.

**Risk**: None for non-production. Engineers must plan for environment warm-up time if they need it outside scheduled hours (manual scale-up command).

## Pattern 8: HPA with Custom Metrics

**What it does**: Replaces CPU-based HPA with workload-appropriate metrics (queue depth, RPS) for more accurate scaling.

**When to use**: Any deployment backed by a message queue (RabbitMQ, SQS, Kafka) or with variable request rates.

**Why it saves cost**: CPU-based HPA scales out before the CPU is actually saturated, pre-emptively over-provisioning. Queue-depth HPA scales out when the queue is actually building, which is the correct signal for a consumer workload. This tighter coupling between load and replicas reduces over-provisioning during low-traffic periods.

**Implementation (RabbitMQ example)**:
- Deploy KEDA (Kubernetes Event Driven Autoscaling) — the standard mechanism for external metric HPA
- Configure ScaledObject pointing to RabbitMQ queue depth
- Set maxReplicaCount conservatively — let queue depth drive scale-out, not a high ceiling

**Expected savings**: 10–20% replica count reduction during off-peak hours for queue-driven workloads. Multiplied by Spot pricing, this is meaningful.

## Pattern 9: Topology-Aware Service Routing

**What it does**: Directs pod-to-service traffic to endpoints in the same AZ, reducing inter-AZ data transfer charges.

**When to use**: Multi-AZ EKS clusters with significant service-to-service traffic.

**Implementation**:
```yaml
apiVersion: v1
kind: Service
metadata:
  name: notification-api
  annotations:
    service.kubernetes.io/topology-mode: "auto"
spec:
  ...
```

With `auto`, Kubernetes distributes EndpointSlice hints to prefer same-AZ routing. Works best when endpoints are distributed across AZs (which they should be for HA).

**Expected savings**: 30–60% reduction in inter-AZ data transfer charges for the annotated service. For a platform with $900/month data transfer, this can save $300–540/month.

**Risk**: Traffic distribution is no longer perfectly even across AZs. If one AZ has more traffic, endpoints in that AZ receive more load. Monitor service latency and error rates per AZ after enabling.

## Pattern 10: PersistentVolume Right-Sizing and Tier Migration

**What it does**: Reduces PVC storage requests to match actual usage, and migrates appropriate volumes to lower-cost storage classes.

**When to use**: Any cluster with PersistentVolumes. PVCs are frequently over-provisioned (developer requests 100Gi "to be safe" when 20Gi is used).

**Audit command**:
```bash
# Show PVC usage vs capacity
kubectl exec -n <namespace> <pod-name> -- df -h | grep /data
# Compare against PVC size
kubectl get pvc -n <namespace>
```

**Migration gp2 → gp3**:
```bash
aws ec2 modify-volume \
  --volume-id vol-xxxxxxxxxx \
  --volume-type gp3 \
  --iops 3000 \
  --throughput 125
```
gp3 costs the same as gp2 per GB but delivers 3,000 IOPS baseline (vs gp2's variable 3 IOPS/GB) and can provision additional IOPS independently.

**Expected savings**: 20–40% storage cost reduction for PVCs over-provisioned by > 50%. gp3 migration delivers same cost with better baseline IOPS — not a cost saving but a performance improvement at no extra cost.

## Pattern 11: Load Balancer Consolidation

**What it does**: Replaces multiple `LoadBalancer`-type Services with a single Ingress controller and ALB, reducing per-LB monthly fees.

**When to use**: Any cluster with > 3 external services, each with its own `LoadBalancer` service.

**Cost math**: 10 ALBs = 10 × $16.20/month = $162/month in base fees. 1 ALB with 10 Ingress rules = $16.20/month base + marginal LCU usage.

**Implementation**: Install AWS ALB Ingress Controller (aws-load-balancer-controller). Change `type: LoadBalancer` services to `type: ClusterIP`. Add Ingress resources routing paths/hosts to ClusterIP services.

**Expected savings**: $16.20 × (N-1) per month for N consolidated load balancers. On a cluster with 10 external services: ~$146/month.

## Pattern 12: Container Image Size Optimization

**What it does**: Reduces container image sizes to decrease ECR storage costs and accelerate pod startup time (which indirectly improves autoscaling response).

**When to use**: Any image > 500MB. Most common culprit: full OS base images (ubuntu, centos) instead of minimal images (alpine, distroless).

**Quick wins**:
- Migrate from `python:3.11` (900MB+) to `python:3.11-slim` (~200MB) or `python:3.11-alpine` (~50MB)
- Use multi-stage Docker builds to exclude build tools from runtime images
- Enable ECR image lifecycle policies to delete untagged images and old image versions

**Expected savings**: ECR storage: $0.10/GB-month. For a cluster with 50 images averaging 1GB: $5/month. The real saving is faster autoscaling (smaller images pull faster, reducing cold start time from minutes to seconds).

## Quick Reference: Pattern Selection

| Situation | Start With |
|-----------|-----------|
| Cluster idle cost > 20% | Patterns 1, 3, 4 |
| EC2 spend too high, workloads stateless | Pattern 2 (Spot) |
| Staging environment always-on | Pattern 7 (scheduling) |
| Data transfer charges high | Pattern 9 (topology routing) |
| Teams deploying without resource requests | Pattern 6 (LimitRange) |
| Many external services, many LBs | Pattern 11 (LB consolidation) |
| Queue-driven workload over-scaling | Pattern 8 (custom HPA metrics) |
| Storage costs growing unexpectedly | Pattern 10 (PVC right-sizing) |
| Need governance guardrails | Pattern 5 (ResourceQuota) |
