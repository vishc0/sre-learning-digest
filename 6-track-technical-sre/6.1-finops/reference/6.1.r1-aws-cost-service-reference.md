# Reference: AWS Service Cost Drivers

**Purpose**: Quick-lookup guide for what drives cost in each major AWS service. Use this to triage cost spikes, model new workloads, and prepare for architecture reviews.  
**Format**: Service → what drives cost → what to optimize → pricing model quirks

---

## EC2 (Elastic Compute Cloud)

**Primary cost drivers**:
- Instance type and size (vCPU + memory tier)
- Pricing model: On-Demand vs Reserved vs Spot vs Savings Plan
- Hours running (billed per second, minimum 60 seconds)
- Data transfer (see Data Transfer section)

**What drives unexpected EC2 cost**:
- Autoscaler stuck in scale-out (scale-down disabled or disruption budget preventing termination)
- Idle instances not terminated after jobs complete (batch workloads, test environments)
- Over-provisioned instance sizes (m5.4xlarge running at 15% CPU)
- EBS-optimized instances with no EBS (charged for optimization capability)

**Pricing model summary**:

| Model | Discount vs On-Demand | Commitment | Best For |
|-------|----------------------|------------|---------|
| On-Demand | 0% | None | Unpredictable burst |
| Spot | 60–90% | None (interruptible) | Stateless, fault-tolerant |
| 1-Year RI (no upfront) | ~35% | 1 year | Stable baseline |
| 1-Year RI (all upfront) | ~40% | 1 year | Capital-efficient |
| 3-Year RI (all upfront) | ~60–72% | 3 years | Long-stable infrastructure |
| Compute Savings Plan | Up to 66% | 1 or 3 years | Mixed fleet, flexible |

**Key pricing quirk**: EC2 costs for EKS node groups appear under "Amazon Elastic Compute Cloud" in Cost Explorer, not under "Amazon Elastic Kubernetes Service." EKS management fee is a separate $0.10/hr line item.

## EKS (Elastic Kubernetes Service)

**Primary cost drivers**:
- Cluster management fee: $0.10/hr per cluster ($72/month) — unavoidable
- Worker node EC2 cost (dominant cost — see EC2 section)
- EBS for PersistentVolumes
- Data transfer between pods across AZs

**What drives unexpected EKS cost**:
- Multiple clusters instead of namespaces (each cluster = $72/month management fee)
- Nodes that cannot scale down due to PodDisruptionBudgets or DaemonSets
- Container Insights enabled at pod level — high-cardinality metrics are expensive
- ALB Ingress Controller creating one ALB per service instead of per-ingress

**Optimization priority order**:
1. Reduce node count via right-sizing (biggest lever)
2. Spot instances for stateless workloads
3. Savings Plans for on-demand baseline
4. Topology-aware routing to reduce inter-AZ data transfer
5. Consolidate load balancers via Ingress
6. Reduce Container Insights granularity

## RDS (Relational Database Service)

**Primary cost drivers**:
- Instance type (db.m5.large, db.r5.2xlarge, etc.) — same pricing tiers as EC2
- Multi-AZ deployment (2x single-AZ cost — always recommended for production)
- Storage: gp2 or gp3 per GB-month + IOPS for io1/io2
- Snapshot storage (beyond the free tier = provisioned storage × 1)
- Data transfer (across AZs, to internet)
- Backup retention (automated backup storage)

**What drives unexpected RDS cost**:
- io1 storage provisioned for IOPS but actual IOPS need is met by gp3
- Read replicas left running after traffic shifted away from them
- Snapshot storage accumulating without retention policy
- Multi-AZ in non-production environments (unnecessary)
- Large instance type sized for peak load with no autoscaling

**gp2 vs gp3 cost comparison**:
- gp2: $0.115/GB-month, IOPS baseline = 3 IOPS/GB (max 16,000 IOPS)
- gp3: $0.115/GB-month (same), baseline 3,000 IOPS included, additional IOPS $0.02/IOPS
- **For most RDS workloads: gp3 delivers same or better performance at same cost with independent IOPS control. Migrate gp2 → gp3.**

**RI for RDS**: RDS supports Reserved DB Instances — same 1-year/3-year commitment model as EC2 RIs. For production databases running 24/7, RDS RIs are a straightforward optimization with 35–60% savings.

## S3 (Simple Storage Service)

**Primary cost drivers**:
- Storage: varies by storage class (see table below)
- Requests: GET, PUT, LIST, DELETE — cost per request type
- Data retrieval: Glacier Instant / Flexible Retrieval retrieval fees
- Data transfer: internet egress ($0.09/GB first 10TB/month)
- Replication: cross-region or same-region replication doubles storage cost for replicated data

**Storage class pricing (us-east-1)**:

| Storage Class | Price/GB-month | Retrieval Cost | Min Duration | Best For |
|--------------|---------------|----------------|-------------|---------|
| S3 Standard | $0.023 | None | None | Active data |
| S3 Intelligent-Tiering | $0.023 (frequent) | None (frequent tier) | None | Unknown access pattern |
| S3 Standard-IA | $0.0125 | $0.01/GB | 30 days | Infrequent access |
| S3 One Zone-IA | $0.01 | $0.01/GB | 30 days | Reproducible infrequent |
| S3 Glacier Instant | $0.004 | $0.03/GB | 90 days | Archive, ms retrieval |
| S3 Glacier Flexible | $0.0036 | $0.02/GB + $0.01/1K req | 90 days | Archive, minutes retrieval |
| S3 Glacier Deep Archive | $0.00099 | $0.02/GB | 180 days | Long-term archive |

**What drives unexpected S3 cost**:
- All data in Standard class with no lifecycle policy (paying Standard rates for archival data)
- High LIST request volume from applications scanning prefixes
- Versioning enabled with no noncurrent version expiration policy (every deleted/overwritten object retained)
- Cross-region replication enabled without reviewing necessity
- Large number of multipart uploads not completed or aborted

**Quick win**: Implement S3 Lifecycle policies. Move objects not accessed in 30 days to Standard-IA, not accessed in 90 days to Glacier Instant, not accessed in 365 days to Glacier Deep Archive. For most archival use cases, this reduces S3 storage cost by 60–80%.

## Data Transfer

**Primary cost drivers**:
- Outbound to internet: $0.09/GB (first 10TB/month) — highest-cost transfer
- Inter-AZ within same region: $0.02/GB each way ($0.04/GB round trip)
- Cross-region: $0.02–0.09/GB depending on regions
- Direct Connect: $0.02/GB (traffic to AWS) — cheaper than internet for large volumes
- CloudFront origin fetch: $0.01–0.02/GB — cheaper than direct internet egress

**What drives unexpected data transfer cost**:
- Kubernetes pods in AZ-a calling services in AZ-b (inter-AZ data transfer)
- Applications sending logs to CloudWatch in a different region than the application
- Cassandra cross-AZ replication without rack awareness optimized for same-AZ reads
- API responses not cached (CDN can eliminate origin egress for cacheable content)
- S3 buckets in one region accessed by Lambda in another region

**Key optimization: Topology-Aware Routing**

In EKS 1.23+, enabling Topology Aware Routing (formerly Topology Aware Hints) instructs kube-proxy to prefer same-AZ endpoints:

```yaml
apiVersion: v1
kind: Service
metadata:
  annotations:
    service.kubernetes.io/topology-mode: "auto"
```

For a cluster with significant cross-AZ traffic, this can reduce inter-AZ data transfer by 40–70%.

## CloudWatch

**Primary cost drivers**:
- Log ingestion: $0.50/GB ingested
- Log storage: $0.03/GB-month (first 5GB free)
- Metrics: first 10 custom metrics free; $0.30/metric-month after
- Dashboards: $3.00/dashboard-month (first 3 free)
- Alarms: $0.10/alarm-month (first 10 free)
- Container Insights: additional custom metrics at standard rate — can be significant at cluster scale

**What drives unexpected CloudWatch cost**:
- Application deployed with DEBUG log level in production (10–100x log volume increase)
- High-cardinality metric dimensions (e.g., `user_id` as a CloudWatch dimension creates one metric series per user)
- Container Insights pod-level metrics on large clusters (each pod emits ~20 metrics)
- CloudTrail logs routed to CloudWatch Logs (consider S3 instead — 5x cheaper for archival)
- No log retention policy (default is Never Expire — you pay forever)

**Log retention policy recommendation**:

| Log Type | Retention |
|----------|-----------|
| Application logs (prod) | 90 days |
| Application logs (dev/staging) | 14 days |
| Audit / compliance logs | 365 days (or to S3 Glacier for longer) |
| CloudTrail logs | 30 days in CW, then S3 + Glacier |
| Container Insights | 30 days |

## Lambda

**Primary cost drivers**:
- Invocation count: $0.20 per 1M invocations (first 1M free per month)
- Duration: $0.0000166667 per GB-second
- Provisioned concurrency: $0.000004646/GB-second (reserved warm capacity)

**What drives unexpected Lambda cost**:
- Functions with high memory allocation (512MB+ when 128MB would suffice)
- Functions invoked recursively or in runaway loops (SQS message trigger with failed processing creates infinite re-invoke loop)
- Provisioned concurrency left enabled for non-critical functions
- Functions with long timeouts set to max (15 min) when typical execution is seconds — failed executions timeout fully

**Cost vs duration optimization**: Lambda pricing is linear with memory AND duration. Counterintuitively, doubling memory often halves execution time (faster CPUs at higher memory tiers), which can result in the same total cost at better performance. Test at multiple memory tiers using AWS Lambda Power Tuning.

## Quick Cost Spike Triage Table

| Service in Cost Explorer | First Thing to Check | Common Culprit |
|-------------------------|---------------------|----------------|
| EC2 | New instances, autoscaler events | Runaway job, scale-out without scale-down |
| EKS | Node count, Container Insights | New cluster, Container Insights enabled |
| RDS | New instance type, snapshot storage | Read replica left running, snapshot retention |
| S3 | Storage class, request count | Versioning without expiry, all data in Standard |
| CloudWatch | Log ingestion spike | DEBUG log level in prod, new high-cardinality metric |
| Data Transfer | Inter-AZ or internet egress | Cross-AZ pod calls, no topology-aware routing |
| Lambda | Duration increase | New high-memory function, runaway loop |
