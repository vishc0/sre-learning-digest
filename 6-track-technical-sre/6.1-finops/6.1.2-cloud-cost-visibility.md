# Chapter 2: Cloud Cost Visibility

## The Instrumentation Analogy

You would never run a production Kubernetes cluster without metrics. No CPU utilization, no memory pressure, no pod restart counts — you are flying blind. Every incident would be a mystery because you have no signal, only symptoms.

Cloud cost without tagging is the same condition. You have a bill. You do not have signal. You cannot tell which team is responsible for the spike, which service drove the S3 egress charge, or which Kubernetes namespace consumed the unexpected EC2 burst. The invoice arrives and you start a forensic investigation instead of a response.

Tagging is the instrumentation layer for cost. Tag strategy design is the equivalent of deciding what metrics to emit before you write the application. Do it late and you spend months backfilling. Do it well and cost visibility becomes a self-service capability for every engineering team.

---

## Tagging Strategy

A good tagging strategy has four properties: **consistent** (same key names everywhere), **enforced** (violations are blocked or flagged automatically), **business-aligned** (tags map to how the business reasons about cost), and **low-maintenance** (automated where possible via IaC).

### Minimum Required Tags

These are the tags that must be present on every billable resource. Without them, cost cannot be attributed.

| Tag Key | Example Values | Purpose |
|---------|---------------|---------|
| `team` | `platform`, `payments`, `ml-infra` | Primary cost owner |
| `service` | `api-gateway`, `notification-worker`, `cassandra` | Service-level attribution |
| `environment` | `prod`, `staging`, `dev` | Separate prod spend from non-prod |
| `cost-center` | `CC-1042`, `CC-2017` | Finance mapping for chargeback |
| `project` | `customer-360`, `data-pipeline-v2` | Project-level rollup for time-bounded work |

### Enforcement

Tags are worthless without enforcement. Three mechanisms:

1. **AWS Config Rules** — `required-tags` managed rule flags resources missing required tags. Does not block creation; alerts and can trigger auto-remediation (tagging via Lambda).
2. **Service Control Policies (SCPs)** — in AWS Organizations, SCPs can deny resource creation if tags are missing. This is enforcement, not advisory. Use carefully on EC2 and EKS node groups.
3. **Terraform policy** — in a Terraform-first shop, enforce tags at the module level with `default_tags` in the AWS provider. Resources provisioned outside Terraform get flagged in the Config dashboard.

### Tag Governance Metric

Track **tag coverage rate**: percentage of billable spend that has all required tags applied. Target is 95%+ at Walk maturity, 99%+ at Run. Untagged spend in the 1–5% range is acceptable only if it is accounted for (e.g., AWS-managed resources that cannot be tagged like data transfer charges).

## Cost Allocation: Showback vs Chargeback

These two models represent a governance philosophy decision that a Director must make explicitly.

**Showback**: Each team can see their cloud spend. Finance and engineering leadership can see it. But teams are not billed internally — there is no transfer of budget between departments. The mechanism of accountability is visibility and social pressure (your spend shows up in the monthly review).

**Chargeback**: Cloud spend is actually debited from each team's budget. The platform team (or a shared services function) invoices each product team for their actual consumption. Teams have a financial incentive to optimize because overspend comes directly from their budget.

| Dimension | Showback | Chargeback |
|-----------|----------|-----------|
| Team accountability | Social / visibility | Financial / budget |
| Behavioral change speed | Slow | Fast |
| Organizational friction | Low | High |
| Requires mature tagging | Yes | Yes (critical) |
| Best for | Crawl → Walk transition | Walk → Run organizations |
| Risk | Teams ignore the data | Disputes over allocation fairness |

**Director recommendation**: Start with showback. It forces tagging quality without creating political conflict. Move to chargeback only after tagging coverage exceeds 90% and teams trust the attribution model. Chargeback with poor tagging creates attribution disputes that poison the FinOps practice.

## AWS Cost Explorer: What to Actually Use

Cost Explorer is AWS's native cost analysis tool. Most teams underuse it. The high-value views for an SRE Director:

**Daily cost by service**: Set date range to last 30 days, group by Service. This is your baseline. Know the top 5 cost drivers by heart for your platform.

**EC2 usage type breakdown**: Within EC2, group by Usage Type. This separates On-Demand, Spot, Reserved, and Savings Plan usage. You need this to calculate your commitment coverage.

**Savings Plan coverage report**: Shows what percentage of your EC2 and Fargate usage is covered by Savings Plans. Below 60% coverage means you are leaving significant savings on the table.

**Rightsizing recommendations**: Cost Explorer generates rightsizing recommendations for EC2 instances based on CloudWatch utilization data. Filter to instances with >14 days of data. This is your starting point for every right-sizing campaign, not your ending point — it does not account for burst patterns or Kubernetes bin-packing effects.

**Cost anomaly detection**: Enable this. It uses ML to detect spend anomalies and sends SNS alerts. Threshold: alert at 15% above the rolling 7-day average for any monitored dimension. This is the earliest warning system you have before a runaway job becomes a billing surprise.

## Azure Cost Management (for Multi-Cloud or Migration Contexts)

The equivalent tool in Azure. Key differences from Cost Explorer:

- **Cost analysis views** map to Cost Explorer's group-by dimensions
- **Budgets** in Azure Cost Management support email + action group alerts natively (Cost Explorer anomaly detection is the AWS equivalent)
- **Azure Advisor** generates cost recommendations similar to EC2 Rightsizing Recommendations
- **Resource tags** in Azure work identically but enforcement uses Azure Policy, not SCPs

If your platform spans AWS and Azure, neither tool gives you a consolidated view. You will need a third-party tool (Apptio Cloudability, CloudHealth, or Grafana + CUR export) for unified visibility.

## Building a Cost Attribution Model

A cost attribution model is the mapping between cloud resources and the business entities (teams, services, products) that own them. It is not just tagging — it is the full data pipeline from AWS billing data to a decision-ready dashboard.

### Components

1. **Cost and Usage Report (CUR)**: AWS's detailed billing export. Export to S3, query via Athena. This is the raw data source for every serious FinOps analysis. Cost Explorer is a pre-built UI on top of a subset of this data.

2. **Tagging layer**: Applied to resources. Maps resource ARNs to business dimensions.

3. **Allocation rules**: Some costs cannot be tagged (data transfer, support, marketplace fees). These must be allocated proportionally — typically by each team's percentage of tagged EC2 spend.

4. **Dashboard**: Grafana (pulling from Athena/CUR), Kubecost (for Kubernetes), or Cost Explorer saved views. The target state: any engineering lead can answer "what did my service cost last week?" without opening a ticket.

### Attribution Model: Service-Team-Environment

The three-dimensional attribution model that works for most platform organizations:

```
Cost ($) → allocated by:
  ├── team (primary owner — maps to Slack team, on-call rotation)
  ├── service (secondary — maps to Git repo, K8s namespace)
  └── environment (prod vs non-prod — often the biggest lever for optimization)
```

Non-prod typically represents 20–40% of cloud spend. That is the first optimization target in any FinOps program because it has the lowest reliability risk: dev environments can tolerate Spot interruptions, staging can run smaller instance types, and feature environments can be stopped when not in use.

---

## Key Takeaways

- Tagging is instrumentation — design it before you need it, enforce it from day one
- Showback before chargeback — build trust in the attribution model before creating financial consequences
- AWS Cost Explorer is a starting point; the CUR + Athena pipeline is the foundation of any serious FinOps practice
- Track tag coverage rate as a team health metric alongside SLO compliance
- Non-prod spend is the lowest-risk, highest-return optimization target when starting a FinOps program
