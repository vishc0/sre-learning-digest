# Chapter 3: Kubernetes Cost Optimization

## Why Kubernetes Cost Is Structurally Different

In a traditional VM-based environment, cost attribution is straightforward: one VM, one owner, one bill. In Kubernetes, a single EC2 node runs dozens of pods from a dozen teams. The EC2 cost belongs to the node pool, not to any one service. The EBS volume might serve two different deployments. The data transfer charge aggregates across every pod that made an outbound call.

Kubernetes abstracts the infrastructure so well from developers that it also abstracts the cost. Developers stop thinking about instance types, bin packing, and memory-to-CPU ratios. That abstraction is valuable for velocity, but it creates a cost blind spot that compounds at scale.

The Kubernetes cost discipline is about restoring visibility and accountability at the pod/namespace level without removing the developer-friendly abstraction that makes Kubernetes useful.

## Namespace Cost Attribution

Namespaces are the natural unit of cost attribution in Kubernetes. In a well-structured cluster, each namespace corresponds to one team or one service. This mapping makes namespace-level cost reporting the bridge between Kubernetes abstractions and the business attribution model.

### The Namespace-to-Team Contract

Define a namespace ownership registry: a Git-managed YAML or ConfigMap that maps each namespace to a team, cost center, and environment. This becomes the source of truth for your cost allocation layer.

```yaml
# namespace-registry.yaml
namespaces:
  - name: payments-prod
    team: payments
    cost-center: CC-1042
    environment: prod
  - name: notification-worker-prod
    team: platform
    cost-center: CC-2017
    environment: prod
  - name: notification-worker-staging
    team: platform
    cost-center: CC-2017
    environment: staging
```

Every cost report joins against this registry. If a namespace is missing from the registry, it shows up as "unattributed" — which creates the social pressure to register it.

### Resource Quotas as Cost Guardrails

`ResourceQuota` objects set hard limits on CPU and memory per namespace. They function as cost guardrails: a team cannot accidentally consume unbounded resources.

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: payments-prod-quota
  namespace: payments-prod
spec:
  hard:
    requests.cpu: "40"
    requests.memory: 80Gi
    limits.cpu: "80"
    limits.memory: 160Gi
```

The Director-level framing: ResourceQuotas are the Kubernetes equivalent of budget alerts. They do not stop work; they prevent runaway consumption. Coupling them with namespace cost dashboards creates a closed feedback loop: teams see their usage against their quota and have an incentive to right-size.

## Kubecost: The Primary Cost Attribution Tool for Kubernetes

Kubecost is the de facto standard for Kubernetes cost attribution. It deploys as a pod in your cluster, reads the Kubernetes API (pod specs, node usage, namespace labels) and pricing data (AWS pricing API or Cost Explorer export), and produces per-namespace, per-deployment, and per-pod cost reports.

### What Kubecost Does

- Allocates EC2 node cost to namespaces based on actual CPU and memory requests/usage (not just limits)
- Tracks idle cost — the fraction of each node's resources that no pod requested (this is wasted money)
- Reports shared cost (system namespaces, cluster add-ons) and distributes it proportionally
- Integrates with Savings Plans and RI discounts for accurate effective pricing
- Generates right-sizing recommendations per deployment

### Key Kubecost Metrics to Watch

| Metric | What It Means | Action Threshold |
|--------|--------------|-----------------|
| **Cluster idle cost** | Fraction of EC2 cost with no pod consuming it | >20%: trigger right-sizing campaign |
| **Efficiency per namespace** | Requested resources vs used resources | <50%: over-provisioned, right-size requests |
| **Cost per deployment** | Daily or weekly cost per Kubernetes Deployment | Use as a per-team cost SLO input |
| **Cluster utilization** | CPU/memory used vs total capacity | <60%: node consolidation opportunity |

### Kubecost vs Native AWS Cost Tooling

Kubecost wins on Kubernetes attribution granularity. AWS Cost Explorer cannot see inside a Kubernetes cluster — it sees only the EC2 instance and the EKS management fee. Kubecost provides the pod-to-cost mapping that Cost Explorer cannot.

The limitation: Kubecost's cost data is only as accurate as the requests and limits set on pods. If pods have no requests set (a common anti-pattern), Kubecost defaults to a proportional allocation based on actual usage — which is less precise and harder to act on. This is why enforcing resource requests is a FinOps prerequisite, not just a resource management best practice.

## VPA and HPA Right-Sizing

Over-provisioning is the single largest source of Kubernetes waste. The typical pattern: a developer sets a deployment's CPU request to 2 cores and memory to 4Gi because they "don't know exactly what it needs." The pod runs comfortably at 0.3 cores and 1.1Gi. The extra 1.7 cores and 2.9Gi are reserved on the node but idle — you are paying for them.

### Vertical Pod Autoscaler (VPA)

VPA monitors actual CPU and memory usage for each pod and recommends (or automatically sets) resource requests based on observed behavior. It is the automated right-sizing tool built into Kubernetes.

**Recommendation mode** (preferred starting point): VPA analyzes usage and writes recommendations to the VPA object without changing the pod. Your team reviews the recommendations before applying. Use this for the first 30 days to build trust in the data.

**Auto mode**: VPA changes requests automatically. This requires pods to be restarted when requests change — acceptable for stateless workloads, requires careful planning for stateful services.

**Key caveat**: VPA and HPA conflict on CPU-based metrics. Do not run VPA in auto mode on a deployment that also has HPA scaling on CPU utilization. The scaling signals fight each other. The established pattern: VPA for vertical sizing, HPA for horizontal scaling, with HPA using custom metrics (request rate, queue depth) rather than CPU.

### Horizontal Pod Autoscaler (HPA)

HPA scales the number of pod replicas based on a metric. For a notification worker consuming RabbitMQ, the right HPA metric is queue depth, not CPU. For an API gateway, it is RPS normalized per replica. CPU-based HPA is the default but is often the wrong choice because CPU utilization is an effect, not a cause.

**Director framing on HPA**: HPA is your elasticity lever. Paired with Spot instances, it is the mechanism that lets you absorb traffic spikes cheaply and shed capacity rapidly when traffic drops. The cost impact of good HPA configuration versus none can be 15–30% of EC2 spend in a traffic-variable workload.

## Spot and Preemptible Instances

Spot instances (AWS) and Preemptible VMs (GCP) offer 60–90% discount on on-demand pricing in exchange for the possibility of interruption with 2-minute notice (AWS) or 30-second notice (GCP).

### Workloads That Are Spot-Safe

- Stateless API workers (with HPA and multiple replicas — one interruption does not cause downtime)
- Batch and ETL jobs (designed to checkpoint and resume)
- Dev and staging environments (tolerate interruption)
- Kubernetes system components running in multiple replicas (kube-dns, cluster-autoscaler)
- Canary deployments (if they fail, that is the point of the canary)

### Workloads That Are NOT Spot-Safe

- Single-replica stateful workloads (databases, Kafka brokers, Cassandra nodes without proper rack awareness)
- Anything in the critical path with no replicated failover
- Jobs that cannot checkpoint or resume (long-running non-idempotent writes)

### Mixed Node Groups (The Standard Pattern)

The production-ready Spot strategy in EKS: a mixed managed node group with on-demand base capacity + Spot expansion.

```
Node Group: platform-mixed
  On-Demand Base: 2 nodes (guaranteed capacity for system pods and critical services)
  On-Demand Percentage Above Base: 20%
  Spot Allocation: 80% of scale-out nodes
  Spot Instance Types: [m5.xlarge, m5a.xlarge, m4.xlarge] (diversify to reduce interruption probability)
```

With this configuration, your baseline capacity is protected and your burst capacity is cheap. The Cluster Autoscaler and Karpenter both support this pattern. Karpenter (AWS's replacement for Cluster Autoscaler) is generally preferred for new clusters due to faster provisioning and better Spot instance diversification.

## Node Consolidation

Over time, Kubernetes clusters develop fragmentation: many small nodes with small pods, none of which can be moved to allow a node to be drained and terminated. This is the Kubernetes equivalent of memory fragmentation in an OS — structurally wasteful.

**Karpenter Consolidation**: Karpenter (v0.27+) supports automatic node consolidation. It identifies nodes where the remaining pods can be rescheduled on other nodes, cordons and drains those nodes, and terminates the underlying instances. This is a continuous background optimization.

**Manual consolidation campaign** (for Cluster Autoscaler environments):
1. Identify underutilized nodes (Kubecost cluster idle report or `kubectl top nodes`)
2. Cordon the least-utilized node: `kubectl cordon <node>`
3. Drain it: `kubectl drain <node> --ignore-daemonsets --delete-emptydir-data`
4. Terminate the EC2 instance
5. Let Cluster Autoscaler provision a replacement at the right size

Run this quarterly as part of the FinOps governance cadence.

---

## Key Takeaways

- Namespace cost attribution is the foundation — namespaces are the unit, ownership registry is the mapping
- Kubecost provides the pod-level visibility that AWS Cost Explorer cannot
- VPA right-sizes pods; HPA scales replicas — they require coordination to avoid conflicts
- Spot instances are the highest-leverage cost lever for stateless workloads: 60–90% discount is not marginal
- Node consolidation is a quarterly discipline, not a one-time fix
- Idle cost > 20% is an immediate action signal: you are paying for capacity no workload is using
