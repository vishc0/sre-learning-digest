# Playbook: Cloud Cost Anomaly Response

**Trigger**: AWS Cost Anomaly Detection alert OR manual observation of unexpected spend spike  
**Severity**: P2 (respond within 4 hours) if > $500 over baseline; P1 (respond immediately) if > $5,000 over baseline or on track to exceed monthly budget by > 20%  
**Owner**: FinOps engineer (primary), on-call SRE (escalation)

---

## Pre-Conditions

- Cost Anomaly Detection is configured with SNS + Slack integration
- CUR exports are current (24-hour lag is normal)
- Responder has read access to AWS Cost Explorer and Athena

---

## Step 1: Triage (Target: 15 minutes)

**Goal**: Determine the scope and urgency before diving into root cause.

**1.1** Open the Cost Anomaly Detection alert. Record:
- Which monitor fired (Service / Account / Tag dimension)
- Anomalous spend amount ($)
- Time period of anomaly (start date/time)
- Expected spend vs actual spend

**1.2** Open Cost Explorer → Date range: last 7 days. Group by: Service.

Identify whether the spike is isolated to one service or distributed across many. A distributed spike suggests a cross-cutting cause (traffic spike, pricing change, organizational change). An isolated spike suggests a workload-specific cause.

**1.3** Estimate impact: Anomalous daily rate × remaining days in month. 

- If projected month-end overrun < $500: document, monitor, no immediate escalation
- If projected overrun $500–$5,000: notify team lead, begin root cause investigation
- If projected overrun > $5,000: escalate to Engineering Director immediately, begin root cause investigation concurrently

---

## Step 2: Root Cause Investigation (Target: 45 minutes)

Work through these hypotheses in order. Stop at the first confirmed cause.

### Hypothesis A: Runaway Job or Process

**Indicators**: EC2 spend spike, new or modified instances in the environment, CloudTrail showing RunInstances API calls.

**Check**:
```bash
# List EC2 instances launched in the anomaly window
aws ec2 describe-instances \
  --filters "Name=launch-time,Values=<anomaly-start-date>" \
  --query 'Reservations[].Instances[].{InstanceId:InstanceId,InstanceType:InstanceType,LaunchTime:LaunchTime,Tags:Tags}'

# Check for untagged instances (not in the expected namespace/team)
aws ec2 describe-instances \
  --filters "Name=instance-state-name,Values=running" \
  --query 'Reservations[].Instances[?!Tags[?Key==`team`]].[InstanceId,InstanceType,LaunchTime]'
```

**Check Kubernetes**: 
```bash
kubectl get nodes --all-namespaces
kubectl top nodes
# Look for nodes provisioned outside normal patterns
```

**If confirmed**: Identify the initiating event (CI pipeline, autoscaler misconfiguration, runbook error). Terminate unneeded instances. Tag all resources correctly before termination for accurate cost attribution.

### Hypothesis B: Data Transfer Explosion

**Indicators**: High data transfer line item in Cost Explorer, recent deployment or configuration change, new cross-region or internet-facing service.

**Check in Cost Explorer**: Filter by "Data Transfer" usage type. Break down by: DataTransfer-Out-Bytes, DataTransfer-Regional-Bytes. Identify the account and region.

**Common causes**:
- New service writing to S3 in a different region
- Log forwarder sending to wrong region endpoint
- Increased API call volume from external clients
- Cassandra cluster sync across AZs (configuration change)
- CloudWatch exporting to cross-region destination

**If confirmed**: Identify the traffic source (VPC Flow Logs, CloudTrail). Reduce or reroute traffic. If cross-region: evaluate whether the transfer is architecturally necessary.

### Hypothesis C: Logging or Observability Explosion

**Indicators**: CloudWatch or logging service spike. Common after a deployment that increases log verbosity or adds new metrics.

**Check**:
- Cost Explorer → Filter by CloudWatch, Kinesis, OpenSearch
- Check recent deployments for `LOG_LEVEL=DEBUG` or increased metric cardinality
- Check CloudWatch Log Groups for ingestion rate change

**Common causes**:
- Application deployed with debug logging enabled in production
- New metric with high cardinality labels (user ID as a label creates thousands of time series)
- CloudWatch Container Insights enabled on a large cluster without retention policy
- Increased CloudTrail logging scope

**If confirmed**: Reduce log level. Set log group retention policies. Remove high-cardinality labels from metrics. Container Insights: disable detailed pod-level metrics if not needed.

### Hypothesis D: Autoscaler Misconfiguration

**Indicators**: EC2 spike, HPA or Cluster Autoscaler logs showing unexpected scale-out events.

**Check**:
```bash
# HPA events
kubectl get events --all-namespaces | grep HorizontalPodAutoscaler

# Cluster Autoscaler logs
kubectl logs -n kube-system -l app=cluster-autoscaler --tail=200

# Karpenter logs
kubectl logs -n karpenter -l app=karpenter --tail=200
```

**Common causes**:
- HPA min replicas set too high after a traffic experiment
- Cluster Autoscaler scale-down disabled (--scale-down-enabled=false) left in place
- Custom metric HPA using wrong metric source (too sensitive)
- Karpenter disruption budget preventing node termination

**If confirmed**: Correct the autoscaler configuration. Review the change that caused the misconfiguration (deployment or IaC change). Confirm scale-down behavior is functioning.

### Hypothesis E: Pricing or Billing Change

**Indicators**: Cost increase distributed across services, matches a specific date (AWS billing period, RI expiration, Savings Plan expiration).

**Check**:
- RI/Savings Plan expiration: AWS Cost Explorer → Reservations → Reserved Instance Coverage → filter to recent expirations
- New service pricing: AWS pricing page for affected service
- Support plan tier change

**If confirmed**: For RI/Savings Plan expiration: immediately run purchase analysis (see ch05) and procure replacement commitment. For pricing change: evaluate workload migration to lower-cost service or region.

---

## Step 3: Remediation

Document the confirmed root cause and implement the fix. For each type:

| Root Cause | Immediate Remediation | Long-term Prevention |
|------------|----------------------|---------------------|
| Runaway job | Terminate instances; revoke credentials if security concern | EC2 launch guardrails via SCPs; require team tags |
| Data transfer | Reroute traffic; add topology-aware routing | Architecture review: cross-region data flows |
| Logging explosion | Reduce log level; set retention; revoke Container Insights if unused | Log level review in deployment checklist |
| Autoscaler misconfiguration | Restore correct configuration | IaC review for autoscaler changes; load test with cost observation |
| RI/SP expiration | Purchase replacement commitment | Expiration alerts in AWS Budgets (90-day warning) |

---

## Step 4: Communication

**Within 1 hour of anomaly confirmation**: Post to FinOps Slack channel:
```
COST ANOMALY: [brief description]
- Anomalous spend: $X above baseline  
- Root cause: [confirmed/suspected]  
- Impact: Projected $Y additional spend this month  
- Status: Remediation in progress / complete  
- Next update: [time]
```

**Within 24 hours**: Send written summary to Engineering Director and relevant team lead. Include: confirmed root cause, total cost impact, remediation taken, prevention measures.

**At next monthly FinOps review**: Present as an anomaly case study. Include timeline, detection-to-resolution time, cost impact, and prevention measure status.

**KPI**: Track mean-time-to-detect (MTTD) and mean-time-to-resolve (MTTR) for cost anomalies. Target MTTD < 24 hours; MTTR < 48 hours.

## Escalation Matrix

| Projected Monthly Overrun | Escalate To | Response Time |
|--------------------------|-------------|---------------|
| < $500 | FinOps engineer (self-resolve) | 48 hours |
| $500–$5,000 | Team lead + FinOps engineer | 4 hours |
| $5,000–$25,000 | Engineering Director | 1 hour |
| > $25,000 | VP Engineering + CFO | Immediate |
