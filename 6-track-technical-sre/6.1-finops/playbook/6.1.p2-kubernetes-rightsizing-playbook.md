# Playbook: Kubernetes Right-Sizing Campaign

**Goal**: Systematically reduce CPU and memory over-provisioning across the EKS cluster to reduce EC2 spend and improve cluster utilization  
**Frequency**: Run quarterly or when cluster idle cost exceeds 20%  
**Owner**: Platform/SRE team, with team-level engineering lead sign-off  
**Estimated duration**: 3–4 weeks end-to-end  
**Expected savings**: 15–25% EC2 cost reduction for a typical over-provisioned cluster

---

## Prerequisites

Before starting a right-sizing campaign:

- [ ] Kubecost is deployed and has at least 14 days of usage data
- [ ] VPA is deployed in recommendation mode (not auto mode unless team has validated it)
- [ ] All deployments have resource requests set (no requests = no meaningful VPA data)
- [ ] Alert runbooks exist for the targeted services (know how to respond if a resize causes a problem)
- [ ] Change freeze calendar is checked — do not start a right-sizing campaign during a major release window

---

## Phase 1: Baseline and Targeting (Week 1)

### Step 1.1: Pull Kubecost Efficiency Report

In Kubecost: Reports → Efficiency → Namespace view.

Filter: CPU efficiency < 50% OR Memory efficiency < 50%.

Export to CSV. This is your candidate list.

**Efficiency definition**:
- CPU efficiency = average CPU used / CPU requested
- Memory efficiency = average memory used / memory requested

A deployment at 25% CPU efficiency has requested 4x more CPU than it uses on average. That is wasted spend on the EC2 node.

### Step 1.2: Pull VPA Recommendations

```bash
# Get VPA recommendations for all namespaces
kubectl get vpa --all-namespaces -o json | \
  jq '.items[] | {
    name: .metadata.name,
    namespace: .metadata.namespace,
    current_cpu_request: .status.recommendation.containerRecommendations[0].target.cpu,
    recommended_cpu: .status.recommendation.containerRecommendations[0].upperBound.cpu,
    current_mem_request: .status.recommendation.containerRecommendations[0].target.memory,
    recommended_mem: .status.recommendation.containerRecommendations[0].upperBound.memory
  }'
```

VPA recommendations use three bounds:
- `lowerBound`: Minimum safe value — what the pod needs to survive
- `target`: The VPA's recommended value for steady-state
- `upperBound`: The maximum observed usage + headroom — safe ceiling

For initial right-sizing, target the `target` value. Ensure you never go below `lowerBound`.

### Step 1.3: Prioritize the Candidate List

Score each candidate:

| Score Factor | Weight | Scoring |
|-------------|--------|---------|
| Monthly cost (absolute) | High | Higher spend = higher priority |
| Efficiency gap (%) | High | Lower efficiency = higher priority |
| Risk level | High | Stateless = low risk, stateful = high risk |
| Team availability | Medium | Team in crunch = defer |

**Risk classification**:
- Low risk: Stateless deployments with ≥ 3 replicas (HPA-managed)
- Medium risk: Stateless deployments with 1–2 replicas, or services with downstream dependencies
- High risk: StatefulSets, databases, message brokers, anything with persistent state

**Start with low-risk, high-cost targets.** Do not start with high-risk targets in the first campaign — build team confidence in the process first.

### Step 1.4: Build the Campaign Plan

Create a tracking table:

| Namespace | Deployment | Current CPU Req | VPA Target CPU | Current Mem Req | VPA Target Mem | Monthly Savings Est. | Risk | Owner | Status |
|-----------|-----------|----------------|---------------|----------------|---------------|---------------------|------|-------|--------|
| payments | api-gateway | 2000m | 800m | 4Gi | 1.5Gi | $210 | Low | [name] | Pending |
| notification | msg-worker | 1000m | 350m | 2Gi | 900Mi | $95 | Low | [name] | Pending |

Get each team lead to review and sign off on the deployments in their namespace before making any changes.

## Phase 2: Measurement Baseline (Week 1, concurrent with targeting)

Before making any changes, capture the current state:

```bash
# Baseline node utilization
kubectl top nodes > baseline-nodes-$(date +%Y%m%d).txt

# Baseline pod resource usage by namespace
kubectl top pods --all-namespaces --sort-by=memory > baseline-pods-$(date +%Y%m%d).txt

# Kubecost: export current weekly cost by namespace (screenshot or CSV)
```

Also record from Cost Explorer: current 7-day EC2 spend. This is your comparison baseline post-campaign.

## Phase 3: Apply Right-Sizing Changes (Weeks 2–3)

Apply changes in batches, not all at once. Batch size: 3–5 deployments per day maximum for the first round.

### Step 3.1: For Each Deployment in the Batch

**Prepare the change** (edit the Deployment spec or Helm values):

```yaml
# Before
resources:
  requests:
    cpu: "2000m"
    memory: "4Gi"
  limits:
    cpu: "4000m"
    memory: "8Gi"

# After (using VPA target with 20% headroom above target for safety)
resources:
  requests:
    cpu: "960m"       # VPA target 800m + 20%
    memory: "1800Mi"  # VPA target 1.5Gi + 20%
  limits:
    cpu: "2000m"      # Limit at 2x request (allows burst)
    memory: "3600Mi"  # Limit at 2x request
```

**Do not cut requests to the VPA target directly.** Add 15–20% headroom above the VPA target value to absorb traffic spikes. The VPA `upperBound` already includes headroom — if using `upperBound` directly, you do not need to add more.

### Step 3.2: Apply and Monitor

```bash
# Apply the change
kubectl apply -f deployment-updated.yaml

# Watch rollout
kubectl rollout status deployment/<name> -n <namespace>

# Verify pods are healthy post-rollout
kubectl get pods -n <namespace> -l app=<name>

# Check for OOMKilled events (memory request too low)
kubectl describe pods -n <namespace> -l app=<name> | grep -A5 "OOMKilled"
```

**Observation window**: Monitor for 24–48 hours per batch. Watch:
- Pod restart rate (OOMKilled = memory request too aggressive)
- CPU throttling (CPU limit too low — check with `kubectl top pod`)
- Latency metrics (if service latency increased, CPU requests may be too low)

If any pod is OOMKilled: immediately increase the memory request by 30% from the new value. Do not wait.

### Step 3.3: Rollback Criteria

Trigger an immediate rollback to the previous resource values if:
- More than 1 OOMKilled event within 24 hours
- Pod restart rate > 1 restart per hour
- P95 latency increased > 20% from baseline
- Any downstream service begins reporting errors attributable to this service

Rollback command:
```bash
kubectl rollout undo deployment/<name> -n <namespace>
```

## Phase 4: Verify and Measure Savings (Week 4)

After all batches are applied and the observation window is clear:

### Step 4.1: Compare Node Utilization

```bash
kubectl top nodes > post-rightsizing-nodes-$(date +%Y%m%d).txt
diff baseline-nodes-*.txt post-rightsizing-nodes-*.txt
```

Target: average node CPU utilization increased (you are using more of what you pay for).

### Step 4.2: Compare Kubecost Costs

In Kubecost: compare namespace cost for the 14 days before vs 14 days after. This is your realized savings measurement.

### Step 4.3: Compare EC2 Spend in Cost Explorer

Compare the 7-day EC2 average before vs after. Account for traffic variation — normalize if there was significant traffic difference.

### Step 4.4: Check for Node Count Reduction

If cluster utilization improved significantly, the Cluster Autoscaler or Karpenter may have terminated one or more underutilized nodes. This is the multiplier effect of right-sizing: fewer pod resources required → fewer nodes required → significant EC2 cost reduction.

```bash
kubectl get nodes
# Compare to baseline node count
```

### Step 4.5: Record Results

| Metric | Before | After | Change |
|--------|--------|-------|--------|
| Cluster CPU utilization | X% | Y% | +Z% |
| Cluster memory utilization | X% | Y% | +Z% |
| Node count | N | M | -P nodes |
| Weekly EC2 spend | $X | $Y | -$Z |
| Monthly cost projection | $X | $Y | -$Z |

## Phase 5: Operationalize (Ongoing)

Right-sizing is not a one-time project. After the first campaign:

1. **Add VPA recommendation review to the monthly FinOps review agenda** — any deployment with efficiency < 50% for two consecutive months is automatically queued for right-sizing
2. **Set a cluster idle cost alert** — alert when idle cost exceeds 20% (anomaly detection or custom rule)
3. **Add resource requests to the deployment checklist** — new services must have VPA installed before going to production and must show ≥ 2 weeks of recommendation data before the first right-sizing
4. **Set ResourceQuotas per namespace** — prevents new over-provisioning from eroding the gains
