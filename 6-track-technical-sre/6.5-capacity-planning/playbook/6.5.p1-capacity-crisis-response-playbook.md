# Playbook: Capacity Crisis Response

**Version**: 1.0  
**Purpose**: Immediate response when a service is approaching or has hit capacity limits  
**Trigger**: Any resource at or above the thresholds in the table below  
**Owner**: Incident Commander (Director or senior SRE manager on-call)

---

## Trigger Thresholds by Resource Type

| Resource | Warning (Escalate to Review) | Critical (Activate This Playbook) |
|---------|-----------------------------|---------------------------------|
| CPU | >80% sustained 5 min | >95% sustained 5 min, or OOMKill-adjacent behavior |
| Memory | >85% | >95% with OOMKills occurring |
| Disk | >80% | >90%, or Cassandra disk preventing compaction |
| Database connections | >80% of pool max | At pool max with requests queuing |
| Network bandwidth | >75% | >90% sustained, or packet drops occurring |
| Cassandra disk | >65% | >75% (note: lower than general disk — see ch05) |

---

## Roles

| Role | Who | Responsibilities |
|------|-----|-----------------|
| **Incident Commander (IC)** | Director or senior SRE manager | Declares incident, assigns roles, makes demand shedding decisions, approves emergency spend |
| **Technical Lead (TL)** | Platform SRE or infrastructure lead | Executes all technical steps, reports status to IC every 10 minutes |
| **Communications Lead** | On-call SRE or manager | Notifies stakeholders (VP, product, NOC), posts updates to incident channel |
| **Database Lead** | DBA or Cassandra-expert SRE | Engaged only for disk/connection/Cassandra-specific steps |

## Phase 1 — Immediate Headroom Recovery (Steps 1-8)

These steps aim to create breathing room within 15-30 minutes without making structural changes.

**Step 1 — Confirm the trigger** *(TL, 2 min)*  
Verify the alert is real (not a monitoring pipeline issue). Check the metric in at least two sources: Splunk and the native cloud metric (CloudWatch, GCP Monitoring). Confirm which service(s) are affected and which resource type is constrained.

> Decision gate: If the alert is a monitoring false positive, escalate to monitoring team and close the incident. Do not proceed with capacity steps for a false signal.

**Step 2 — Declare the incident** *(IC, 1 min)*  
Open an incident in your incident management system. Set severity based on customer impact: P1 if customer-impacting at scale, P2 if degraded or single-region impact, P3 if internal only. Assign TL, Communications Lead.

**Step 3 — CPU saturation response** *(TL)*  
If CPU is the constrained resource:
- a. If Kubernetes: increase deployment replicas by 50% (`kubectl scale deployment <name> --replicas=<new-count>`)
- b. Verify pods schedule successfully (check for Pending pods — if Pending, cluster capacity is also constrained; jump to Phase 3 Step 14)
- c. If HPA is configured: manually set minReplicas to current replica count to prevent scale-in during the incident
- d. Identify the highest CPU consumer by process/container (`kubectl top pods -n <namespace> --sort-by=cpu`)
- e. If a specific pod is consuming disproportionate CPU: check for runaway queries, infinite loops, or unexpected traffic spikes to that instance

**Step 4 — Memory saturation response** *(TL)*  
If memory is the constrained resource:
- a. Identify OOMKill victims: `kubectl get events -n <namespace> | grep OOMKilled`
- b. Temporarily increase memory limits if you have confirmed usage is legitimate (not a leak): `kubectl set resources deployment <name> --limits=memory=<new-value>`
- c. If OOMKills are from JVM services: force a GC by issuing a `jmap -histo` or `jcmd <pid> GC.run` — this is diagnostic, not a fix
- d. Scale out (add replicas) to distribute memory load across more pods
- e. If a memory leak is suspected: roll the affected pods (`kubectl rollout restart deployment/<name>`) to recover leaked memory — this buys time, not a fix

**Step 5 — Disk saturation response** *(TL + Database Lead)*  
If disk is the constrained resource:
- a. Identify largest directories: `df -h` per node, then `du -sh /var/log/* /tmp/*` to find large consumers
- b. Log files: check if log rotation has failed. Manually compress or archive logs older than 24h if rotation policy is not enforcing.
- c. Temp files: identify and clear `/tmp` accumulation from terminated pods or abandoned jobs
- d. **For Cassandra disk full**: do NOT delete SSTable data files without DBA approval. Call the Database Lead immediately.
  - If above 20% free disk: run `nodetool compact <keyspace>` on least-full node first — this may temporarily use more disk before recovering space
  - If below 20% free disk: compaction will fail. Contact Cassandra SME. Options are limited.
- e. Old backup/snapshot artifacts: if cloud snapshots or local backup files are consuming disk, coordinate with DBA before removing

> Decision gate: If disk is below 10% free on any database node, this is a database emergency. Escalate to Database Lead and platform director immediately. Standard capacity steps may not be sufficient — emergency procurement (Phase 3) may be required in parallel.

**Step 6 — Connection pool exhaustion response** *(TL + Database Lead)*  
If connections are the constrained resource:
- a. Identify top connection holders: query the database for active connections by client IP/host
  - PostgreSQL: `SELECT client_addr, count(*) FROM pg_stat_activity GROUP BY client_addr ORDER BY count DESC`
  - Cassandra: `nodetool tpstats` for in-flight operations, driver metrics for connection pool status
- b. Identify connection-leaking services: look for services with connections that have been open for hours without activity (idle connections not being returned to the pool)
- c. Restart connection-leaking service instances (rolling restart, not all at once)
- d. If PostgreSQL: activate PgBouncer if not already in the path, or reduce max connections per client in PgBouncer config
- e. Implement per-client connection limits if carrier or partner integrations are consuming excessive connections

**Step 7 — Network saturation response** *(TL)*  
If network bandwidth is the constrained resource:
- a. Identify top network consumers: use service mesh metrics (Istio, Linkerd) or VPC flow logs to find top talkers
- b. Check for runaway retry storms: a service experiencing failures and retrying aggressively can create traffic feedback loops
- c. Implement rate limiting at the ingress layer (API Gateway, Nginx) for the top consumer services
- d. If a specific client is consuming disproportionate bandwidth: block or rate-limit at the load balancer level

**Step 8 — Validate headroom recovery** *(TL)*  
After steps 3-7: confirm the constrained metric has started declining. Target: metric below 80% within 15 minutes of completing Phase 1 steps. Report status to IC.

> Decision gate: If metric is still above 90% after Phase 1 steps, proceed to Phase 2 (demand shedding). Do not attempt Phase 1 steps a second time — if they did not work, a different approach is needed.

## Phase 2 — Demand Shedding (Steps 9-13)

Demand shedding is the deliberate reduction of traffic or workload to protect the service. It degrades the service intentionally to prevent total failure. Execute in priority order — each step reduces demand further.

> IC must explicitly authorize each demand shedding step before TL executes. These decisions have customer and business impact.

**Step 9 — Drop non-critical background jobs** *(TL, IC authorization required)*  
Identify batch jobs, async processing queues, and non-customer-facing background work. Pause or terminate these first. For Kubernetes: scale down batch job deployments to 0 replicas. For message queue consumers: reduce consumer count. This may delay non-critical processing but does not impact synchronous customer requests.

**Step 10 — Throttle low-priority API consumers** *(TL, IC authorization required)*  
Apply rate limits to API consumers below a defined priority tier. Typically: internal analytics, reporting, and monitoring integrations are throttled before customer-facing API traffic. Implement via API Gateway rate limiting rules or Nginx `limit_req`.

**Step 11 — Implement circuit breakers for downstream services** *(TL, IC authorization required)*  
If a downstream service is contributing to the capacity problem (e.g., slow responses causing connection accumulation, retry storms): activate circuit breakers to fast-fail calls to that downstream. This protects your service from inheriting a downstream's capacity problems.

**Step 12 — Activate graceful degradation modes** *(TL, IC authorization required)*  
If the service has implemented graceful degradation (feature flags, fallback modes): activate them. Examples: return cached responses instead of live data, disable non-critical UI components, serve static content instead of dynamic. This reduces compute load while preserving core functionality.

**Step 13 — Shed traffic at load balancer** *(TL, IC authorization required — highest severity)*  
Last resort: return HTTP 503 to a percentage of incoming traffic to protect the remaining capacity for serving the requests that do get through. Implement at the load balancer or API gateway level. This is visible to customers. Notify Communications Lead before executing; stakeholder notification should be in flight.

## Phase 3 — Emergency Procurement (Steps 14-17)

When Phase 1 and Phase 2 are insufficient to restore headroom, the only remaining option is adding capacity. This takes time — execute in parallel with Phase 1 and Phase 2 if the incident is severe.

**Step 14 — EKS node pool expansion** *(TL)*  
- Increase the maximum size of the affected node pool Auto Scaling Group (ASG) via AWS Console or CLI
- `aws autoscaling update-auto-scaling-group --auto-scaling-group-name <name> --max-size <new-max>`
- Alternatively: add a new node pool using Karpenter NodePool CRD with appropriate instance types
- New nodes will be available in 3-8 minutes; pods will reschedule automatically

**Step 15 — Managed database scale-up** *(Database Lead, IC authorization required)*  
For RDS/Aurora: scale up the instance class. This involves a maintenance window and potential brief downtime (seconds to minutes for Aurora, minutes for RDS). Coordinate with Communications Lead before execution.

**Step 16 — Cassandra node addition** *(Database Lead, hours-long operation)*  
Adding Cassandra nodes is a multi-hour operation (see ch05). Begin immediately if disk pressure is the trigger — do not wait for Phase 1 and Phase 2 to be exhausted. This runs in parallel.
- Launch new EC2 instances with Cassandra installed
- Configure with cluster seeds and datacenter
- Start the bootstrap process (`cassandra -f`)
- Monitor with `nodetool status` — bootstrapping node shows as UJ (Up Joining)
- Full data streaming before the node joins may take 4-8 hours on large clusters

**Step 17 — Escalate cloud spend approval** *(IC)*  
Emergency capacity additions may exceed pre-approved spend limits. Identify and engage the VP-level approver who can authorize emergency cloud infrastructure spend. For T-Mobile context: know who your VP of Platform Engineering is and have their contact accessible in the on-call runbook. Do not let a spend approval process block capacity recovery.

## Phase 4 — Stabilization and Review (Steps 18-20)

**Step 18 — Confirm headroom restored** *(TL)*  
Before declaring the incident resolved:
- Target metric is below the warning threshold (CPU <80%, disk <80%, etc.)
- No new OOMKills, no connection queue depth, no packet drops
- Application latency has returned to baseline SLO range
- Autoscaling is not actively provisioning new capacity (stable state)

**Step 19 — Restore normal operations** *(TL, IC authorization)*  
Re-enable anything suspended in Phase 2 in reverse priority order: restore traffic shedding last (after headroom confirmed), re-enable circuit breakers before restoring graceful degradation, re-enable background jobs last.

**Step 20 — File capacity action item** *(IC)*  
Before closing the incident:
- Document what capacity constraint caused the incident
- File a capacity action item with 30-day resolution deadline
- Schedule a post-mortem for within 5 business days
- Notify infrastructure lead to add this service to the priority watch list for the next monthly capacity review

> The incident is not resolved until the capacity action item is filed. A capacity incident without a capacity action item will repeat.
