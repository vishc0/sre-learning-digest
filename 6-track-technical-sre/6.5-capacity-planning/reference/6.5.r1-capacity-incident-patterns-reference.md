# Capacity Incident Patterns — Quick Reference

*During an active incident: find the failure mode that matches your symptoms, confirm with the detection signals, execute the immediate response, then file for permanent fix.*  
*This is a detection and triage reference — not a substitute for the full capacity crisis response playbook.*

---

## Pattern 1 — OOM Kill (Out of Memory)

**Symptom**: Pods restarting repeatedly. Application logs truncated (process killed mid-write). Service intermittently unavailable as pods cycle.

**Detection**:
```bash
# Find OOMKilled pods in the last hour
kubectl get events -A --field-selector reason=OOMKilling

# Check termination reason for a specific pod
kubectl describe pod <pod-name> -n <namespace> | grep -A5 "Last State"
# Look for: "Reason: OOMKilled"

# Prometheus alert query
kube_pod_container_status_last_terminated_reason{reason="OOMKilled"} == 1

# Splunk query
index=k8s sourcetype=kubernetes:events "OOMKilled" | stats count by namespace, pod_name
```

**Root cause options** (investigate in this order):
1. **Memory leak**: pod memory usage grows monotonically over time — check 24-hour memory trend per pod
2. **Under-requested memory**: pod's request is set too low; actual usage spikes above limit during load
3. **Traffic spike**: sudden increase in concurrent requests increases per-request memory consumption
4. **JVM GC pressure**: Java service hitting GC pause and temporarily needing more memory than limit allows

**Immediate response**:
- Scale out (add replicas) to reduce load per pod — this helps if under-requesting is the cause
- Temporarily increase memory limit: `kubectl set resources deployment <name> -n <ns> --limits=memory=<new-value>`
- If memory leak suspected: rolling restart to recover leaked memory (`kubectl rollout restart deployment/<name>`) — buys time, not a fix
- Do NOT increase limit without understanding why the limit was reached — increasing limits without diagnosis creates future capacity debt

**Permanent fix**:
- Identify root cause: memory leak vs. under-requesting vs. traffic-driven
- Memory leak: fix in application code; add heap profiling (JVM: `jmap -histo`, Python: `tracemalloc`, Go: pprof)
- Under-requesting: right-size memory request and limit using 90-day P95 actual usage data
- Traffic-driven: add HPA on memory metric or increase baseline replicas

**Prevention signal**: memory utilization trending toward limit in monitoring. Alert at 85% memory utilization sustained for 10+ minutes.

## Pattern 2 — CPU Throttling

**Symptom**: High latency at P99 without visible CPU saturation on dashboards. P50 latency looks fine; P99 is 5-10× worse. Error rate may appear normal. Service "feels slow" without obvious cause.

*This is the most commonly misdiagnosed capacity failure mode because CPU utilization dashboards show the pod at 50-60% — which looks healthy. The problem is that the pod's actual CPU usage exceeds its CPU limit during brief bursts, causing kernel cgroup throttling even though average utilization appears normal.*

**Detection**:
```bash
# Prometheus: CPU throttling ratio per container
# High throttling = throttled_seconds / (throttled + non-throttled) > 25%
rate(container_cpu_cfs_throttled_seconds_total[5m]) /
  (rate(container_cpu_cfs_throttled_seconds_total[5m]) + 
   rate(container_cpu_cfs_periods_total[5m]))

# Threshold: >25% throttling is concerning; >50% is severe

# Splunk query if CPU metrics are forwarded
index=metrics metric_name=container.cpu.cfs_throttled_periods
  | eval throttle_ratio = throttled / (throttled + total)
  | where throttle_ratio > 0.25

# kubectl: check CPU limit vs actual usage
kubectl top pods -n <namespace>  # Shows actual usage
kubectl get pods -n <namespace> -o json | jq '.items[].spec.containers[].resources'  # Shows limits
```

**Root cause**: CPU limit is set too low for the pod's actual burst behavior. The request may be appropriately sized for average load, but the limit is too close to (or equal to) the request, leaving no burst headroom.

**Immediate response**:
- Increase CPU limit: `kubectl set resources deployment <name> --limits=cpu=<new-value>`
- Scale out to distribute CPU load across more pods (reduces per-pod burst demand)
- For latency-sensitive services: consider removing CPU limits entirely (requests still ensure scheduling fairness; limits only cause throttling)

**Permanent fix**:
- Profile CPU usage under realistic load to establish the burst profile
- Set CPU limit to P99 of actual CPU usage (not P95) — limit should accommodate the worst-case burst, not the typical burst
- For latency-sensitive services (API servers, notification processors): strongly consider running without CPU limits and relying on CPU requests for scheduling — throttling a latency-sensitive service is often worse than the noisy neighbor risk of removing limits

**Prevention signal**: CPU throttling ratio alert in Prometheus/Datadog at >20% sustained. This is not a standard alert in most environments — you likely need to add it explicitly.

## Pattern 3 — Connection Pool Exhaustion

**Symptom**: Request timeouts with errors like "timeout waiting for connection from pool," "connection pool exhausted," "too many connections." Requests succeed when connection is obtained but queue time dominates latency. Under-load: requests succeed. At load: requests start timing out before they even reach the database.

**Detection**:
```bash
# Application-level: look for pool wait time in application metrics
# HikariCP (Java): hikaricp_connections_pending (waiting for a connection)
# SQLAlchemy: pool_size vs overflow count
# Cassandra driver: in-flight requests per connection vs max

# Database-level: PostgreSQL active connections
SELECT count(*), state FROM pg_stat_activity GROUP BY state;
# Watch for: many "idle in transaction" (connection leak signal)

# Cassandra: nodetool tpstats (requests inflight)
nodetool tpstats | grep -E "Read|Write|Native"

# Splunk: application error logs
index=app "connection pool" OR "too many connections" OR "pool exhausted"
  | timechart count by service
```

**Root cause options**:
1. **Connection leak**: connections checked out from the pool are not being returned (missing close() in error paths, unclosed transactions)
2. **Traffic spike**: sudden increase in concurrent requests exhausts the pool faster than connections complete
3. **Pool sized too small**: pool maximum was never scaled to match actual concurrent request volume
4. **Slow queries**: long-running queries hold connections longer, reducing effective pool throughput (pool slots × query duration = maximum safe request rate)
5. **Database max_connections limit hit**: the database itself has reached its connection limit; no new connections can be established regardless of pool configuration

**Immediate response**:
- Identify connection-leaking services: services with connections open much longer than typical query duration
- Restart connection-leaking service instances (rolling restart)
- For PostgreSQL at max_connections: activate PgBouncer if not already in the path; reduce per-service max_connections in pgbouncer.ini
- Throttle the highest connection consumers at the API gateway level to reduce new connection demand
- For Cassandra: reduce driver concurrent requests per connection if connections-per-host are maxed

**Permanent fix**:
- Connection leak: code audit for all database code paths that can exit without returning connections (especially exception handlers)
- Pool too small: size pool to `(max concurrent requests) / (requests per connection per second)` — add 20% buffer
- Slow queries: profile and optimize; add read replicas to distribute load
- PgBouncer: non-negotiable for PostgreSQL at scale — implement in transaction pooling mode

**Prevention signal**: Connection pool utilization alert at 80% of pool maximum. Also: alert if average connection hold time exceeds 2× the P95 query duration (indicates connection leaks).

## Pattern 4 — Disk Full

**Symptom**: Write errors in application logs ("no space left on device"). Database write failures. Pod evictions (Kubernetes evicts pods from disk-pressured nodes). Service degradation or complete failure on write paths.

**Detection**:
```bash
# Node disk pressure taint
kubectl describe node <node-name> | grep -A5 "Conditions"
# Look for: DiskPressure = True

# Disk usage per node
kubectl get nodes -o json | jq '.items[] | {name: .metadata.name, 
  disk: .status.conditions[] | select(.type=="DiskPressure")}'

# CloudWatch: NodeFilesystemAvailablePercent < 10
# Prometheus: (node_filesystem_avail_bytes / node_filesystem_size_bytes) < 0.10

# Per-directory analysis (on the node via exec or SSH)
df -h
du -sh /var/log/containers/* | sort -rh | head -20
```

**Root cause options**:
1. **Log accumulation**: log rotation failed or was not configured; container logs consuming all available disk
2. **Data growth**: legitimate data growth exceeded provisioned capacity (covered by capacity planning process)
3. **Temp file leak**: terminated pods or failed jobs leaving temp data uncleaned
4. **Cassandra TTL lag**: deleted/expired data still on disk pending compaction
5. **Old compaction artifacts**: Cassandra — old SSTables not cleaned after compaction (run `nodetool cleanup`)
6. **Core dumps**: service crashes generating large core dump files

**Immediate response (non-Cassandra)**:
- Log files: identify largest log files: `du -sh /var/log/containers/* | sort -rh | head -20`
  - Compress or archive logs older than 24h
  - Force log rotation: `logrotate -f /etc/logrotate.conf`
- Temp files: `find /tmp -mtime +1 -type f | head -20` to identify old temp files; clear with care
- Core dumps: check `/var/crash` or application core dump directories; archive or delete old dumps
- For node-level disk pressure: cordon the node (`kubectl cordon <node>`) to prevent new pod scheduling; drain non-critical pods (`kubectl drain <node> --ignore-daemonsets`)

**Immediate response (Cassandra disk)**:
- **Do NOT delete SSTable files without DBA authorization** — this causes data loss
- Check available disk: `nodetool status` shows disk per node
- If >20% free disk: run `nodetool compact <keyspace>` on the least-full nodes first
  - Warning: compaction temporarily uses more disk before reclaiming space. Only run if >20% free.
- If <20% free disk: compaction cannot safely run. Contact Cassandra SME immediately. Options are limited.
  - Emergency: if TTL data is present, run `nodetool flush` then `nodetool compact` to force TTL expiry processing
  - Add nodes (hours-long operation — begin immediately in parallel)
- Never run: `rm *.db` in Cassandra data directories. This destroys data permanently.

**Permanent fix**:
- Implement log rotation policy (logrotate, Docker log driver with `max-size` and `max-file` options)
- Kubernetes: configure pod `terminationGracePeriodSeconds` and ephemeral storage limits (`resources.limits.ephemeral-storage`)
- Storage quota per namespace: prevents one workload from exhausting node disk
- Disk growth rate monitoring: alert at 80%, action at 70% (see formulas reference)
- Cassandra: regular compaction schedule, TTL verification on tables with expire policies

**Prevention signal**: node filesystem utilization >80% alert. For Cassandra: per-node disk >60% alert (tighter threshold — see ch05).

## Pattern 5 — Network Saturation

**Symptom**: Increased latency across multiple services simultaneously. Connection timeouts between services. Packet retransmit rate increase visible in metrics. The failure is diffuse — many services seem slow rather than one specific service failing.

**Detection**:
```bash
# Node network utilization (Prometheus)
rate(node_network_transmit_bytes_total[5m]) * 8  # Convert to bits/sec

# CloudWatch: NetworkOut per EC2 instance (bytes/sec)
# Alert if sustained NetworkOut > 60% of instance baseline bandwidth

# Identify top talkers (service mesh / Istio)
kubectl exec -n istio-system <prometheus-pod> -- \
  promtool query instant \
  'topk(10, rate(istio_tcp_sent_bytes_total[5m]))'

# VPC Flow Logs (CloudWatch Logs Insights)
fields @timestamp, srcAddr, dstAddr, bytes, action
| filter action="ACCEPT"
| stats sum(bytes) as totalBytes by srcAddr, dstAddr
| sort totalBytes desc
| limit 20
```

**Root cause options**:
1. **Retry storm**: a downstream service experiencing failures causes upstream callers to retry aggressively; network traffic multiplies
2. **Intra-cluster chattiness**: two services exchanging high volumes of small messages — especially bad if deployed on different nodes (crosses node network boundary)
3. **Egress surge**: external traffic or data export generating more outbound traffic than expected
4. **Node network interface saturation**: specific node running multiple high-traffic pods; node interface becomes the bottleneck
5. **NAT gateway saturation**: all cluster egress through a shared NAT gateway; NAT gateway bandwidth limit reached

**Immediate response**:
- Identify retry storms: look for error rate increases 5-15 minutes before the network saturation began — the error spike precedes the network saturation
- Activate circuit breakers on the failing downstream (fast-fail instead of retry)
- Rate limit aggressive callers at the load balancer or service mesh level
- For NAT gateway saturation: distribute traffic across multiple NAT gateways (1 per AZ)
- For specific node saturation: cordon the node to prevent new pod scheduling, let HPA redistribute workload to other nodes

**Permanent fix**:
- Service co-location: place chatty service pairs in the same node pool using pod affinity rules
- CDN for external traffic: move static content delivery to CDN, reducing origin egress
- Exponential backoff with jitter: eliminate tight retry loops that generate retry storms
- Data locality: ensure services access databases in the same AZ to avoid cross-AZ data transfer costs and latency
- Multiple NAT gateways: one per AZ eliminates the single NAT gateway as a network bottleneck

**Prevention signal**: node NetworkOut >60% sustained alert. Retry rate alert per service: if retries/sec > 10% of total requests for >2 minutes, investigate downstream.

## Pattern 6 — Cassandra Ring Degradation

**Symptom**: Read/write latency gradually increasing (not a sudden spike). Hints accumulating (indicates writes are being buffered for unavailable replicas). nodetool status shows nodes UN (Up Normal) but with high pending task counts. Service layer sees increasing timeouts on notification delivery or state reads.

**Detection**:
```bash
# Overall ring health
nodetool status

# Thread pool backlog (pending tasks = queue depth)
nodetool tpstats
# Watch: ReadStage, MutationStage, NativeTransportRequests
# Warning: pending > 100. Critical: pending > 500.

# Pending compactions per node
nodetool compactionstats | grep "pending tasks"

# Hints stored (writes to unreachable nodes)
nodetool tpstats | grep "Hinted"
# Any non-zero hints_in_progress with hints_completed > 0 = a node was recently unavailable

# Latency by operation
nodetool cfstats <keyspace>.<table> | grep -E "Read Latency|Write Latency"

# Active repair operations (repair causes load)
nodetool netstats | grep -i repair
```

**Root cause options**:
1. **Disk pressure on multiple nodes**: nodes struggling to compact; compaction backlog → write stalls → latency increase
2. **Hot partition**: one partition key receiving disproportionate read/write traffic; the node(s) holding that partition are overloaded
3. **Recent node failure**: surviving nodes absorbing the failed node's load; hints accumulating for the failed node's data
4. **Repair running during peak traffic**: Cassandra repair streams data between nodes; heavy repair competes with operational traffic
5. **Garbage collection pauses**: JVM GC stop-the-world pauses on Cassandra nodes cause latency spikes during GC events

**Immediate response**:
- Check disk per node first: `nodetool status` shows disk usage. If any node is >70% disk, this is likely the root cause.
- If hints are accumulating: a node was recently unavailable or is falling behind. Do not add more write load until the hinted handoff completes.
- Throttle writes if hints accumulating: reduce client write rates temporarily to allow hint replay to catch up
- Identify hot partitions: if one node has dramatically higher pending tasks than others, check partition key distribution for the affected table (`nodetool toppartitions <keyspace> <table> 60`)
- **Do NOT add new nodes during active ring degradation without Cassandra SME guidance**: node addition during degradation can worsen the situation by triggering additional streaming and compaction

**Permanent fix**:
- Disk pressure root cause: add nodes (ch05 expansion process); fix data retention policies; verify compaction strategy (STCS vs LCS — LCS uses more disk but produces less compaction spikes)
- Hot partition: redesign partition key to distribute load more evenly; consider using a bucket suffix on the partition key for time-series data
- Repair schedule: schedule repairs during off-peak hours; use `-pr` (primary range) repair to reduce load; limit repair bandwidth with `nodetool setstreamthroughput`
- GC tuning: for Java 11+ Cassandra, G1GC is default. Tune `-Xmx` to 50-60% of available memory; increase `G1RSetUpdatingPauseTimePercent` if GC pauses are frequent

**Prevention signals**:
- Pending tasks (ReadStage or MutationStage) >100 for any node: alert
- Per-node disk divergence: if the highest-disk node is >10% higher than the lowest-disk node, data distribution imbalance is developing
- P99 read or write latency >50ms sustained for 5+ minutes: alert
- Hints in progress >0 sustained: indicates a node health issue that needs investigation
