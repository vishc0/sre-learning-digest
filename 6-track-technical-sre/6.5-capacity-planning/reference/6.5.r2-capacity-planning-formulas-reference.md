# Capacity Planning Formulas — Quick Reference

*Use this reference during reviews, audits, and incident response. All formulas are tool-agnostic — apply in Splunk, Excel, Python, or mental math.*

---

## Formula 1 — Utilization Rate

**Formula**: `Utilization % = (Current Usage / Provisioned Capacity) × 100`

**Example**: CPU used = 109 vCPU, provisioned = 160 vCPU  
`Utilization = (109 / 160) × 100 = 68.1%`

**When to use**: Any time you need to express how much of a provisioned resource is in use. The starting point for all capacity analysis.

**Common mistakes**:
- Using average utilization instead of P95 — average hides spikes. Use P95 for capacity decisions.
- Using total cluster CPU (including OS/daemonset overhead) instead of allocatable CPU. Use `kubectl describe node` allocatable values, not total.
- Comparing utilization across different time windows without normalizing (a 7-day metric during a campaign vs a 30-day metric at baseline are not comparable).

---

## Formula 2 — Headroom Percentage

**Formula**: `Headroom % = (Provisioned - Used) / Provisioned × 100`  
Equivalent: `Headroom % = 100 - Utilization %`

**Example**: Provisioned = 160 vCPU, Used = 109 vCPU  
`Headroom = (160 - 109) / 160 × 100 = 31.9%`

**When to use**: Monitoring dashboards, capacity reviews, RAG status calculations. Headroom is the more intuitive metric for capacity planning conversations — "we have 32% headroom" is easier to reason about than "we are at 68% utilization."

**Common mistakes**:
- Confusing scheduling headroom (based on pod requests) with compute headroom (based on actual utilization). These can differ significantly in a Kubernetes cluster.
- Treating headroom as a single number when it varies by resource type (you can have 30% CPU headroom but 8% memory headroom — memory is the binding constraint).

---

## Formula 3 — Days to Capacity (at Current Growth Rate)

**Formula**: `Days Until Threshold = (Threshold% × Provisioned - Current Used) / Daily Growth`

**Example**: Cassandra disk. Provisioned = 60 TB, Current used = 33 TB, Trigger threshold = 60% (36 TB), Daily growth = 0.2 TB/day  
`Days Until 60% = (36 - 33) / 0.2 = 15 days`

**When to use**: Converting utilization trends into a deadline — the "action date" that drives urgency in capacity reviews. This is what you put on the capacity dashboard as a calendar date, not just a percentage.

**Common mistakes**:
- Using daily average growth when growth is non-linear (use the most recent 30-day growth rate, not 90-day average, if growth is accelerating).
- Setting threshold to 100% instead of your operational threshold (70% for compute, 60% for Cassandra). You need to act before 100%, not at 100%.
- Not accounting for weekend/weekday variation in daily growth rate for write-heavy workloads.

---

## Formula 4 — Growth Rate (Period-over-Period)

**Formula**: `Growth Rate % = (Current Value - Previous Value) / Previous Value × 100`

**Example**: Disk used 30 days ago = 28 TB, disk used today = 33 TB  
`Growth Rate = (33 - 28) / 28 × 100 = 17.9% over 30 days`

**When to use**: Calculating the trend before applying it to a forecast. Compare growth rates across periods to detect acceleration or deceleration — more valuable than the absolute growth rate alone.

**Common mistakes**:
- Computing growth rate over a period that includes an anomaly (campaign, incident, migration) without flagging it as a non-representative sample.
- Using a single period's growth rate as a forecast input without checking if it is representative of the trend.

---

## Formula 5 — Compound Monthly Growth Rate (CMGR)

**Formula**: `CMGR = (Current / Initial)^(1/months) - 1`

**Example**: Disk was 20 TB 6 months ago, is 33 TB today  
`CMGR = (33 / 20)^(1/6) - 1 = 1.65^0.167 - 1 = 0.0875 = 8.75%/month`

**When to use**: Smoothing growth rate over multiple periods to reduce noise from individual month anomalies. CMGR is the right rate to use for 6-12 month capacity forecasts.

**Common mistakes**:
- Applying CMGR calculated from a high-growth early period to a mature service that has since decelerated.
- Confusing CMGR (monthly compound) with simple monthly average growth rate — they are different when growth is non-linear.

---

## Formula 6 — Required Capacity at Target Date

**Formula**: `Future Capacity Required = Current Usage × (1 + CMGR)^months`

**Example**: Current CPU used = 109 vCPU, CMGR = 8.75%/month, horizon = 6 months  
`Future CPU = 109 × (1 + 0.0875)^6 = 109 × 1.659 = 180.8 vCPU`

To maintain 30% headroom: `Provisioned needed = 180.8 / 0.70 = 258 vCPU`  
Current provisioned: 160 vCPU. Gap: 98 vCPU (approximately 12 m5.2xlarge nodes).

**When to use**: Quarterly capacity reviews to determine how much infrastructure needs to be provisioned by a specific date. This is the core forecasting formula — the output drives the capacity action log.

**Common mistakes**:
- Forgetting to apply the headroom target (divide future usage by your target utilization %, not 1.0) — this is the most common forecasting error.
- Not accounting for discrete capacity additions (you add nodes in fixed increments — round up to the nearest node count).

---

## Formula 7 — CPU Request-to-Limit Ratio

**Formula**: `Limit:Request Ratio = CPU Limit / CPU Request`

**Example**: CPU Request = 500m (0.5 vCPU), CPU Limit = 1000m (1 vCPU)  
`Ratio = 1000 / 500 = 2.0`

**Target range**: 1.0-2.0

**When to use**: Kubernetes capacity audit, right-sizing analysis. A ratio of 1.0 means no burst is allowed — the pod runs at exactly its request under load. A ratio of 2.0 allows 2× burst. Above 3.0, the gap between request and limit creates "noisy neighbor" risk on shared nodes.

**Common mistakes**:
- Setting ratio to infinity (no CPU limit) — this allows one pod to consume an entire node's CPU, starving all other workloads.
- Setting ratio to 1.0 for latency-sensitive services — this guarantees CPU throttling during any traffic spike.

---

## Formula 8 — Resource Efficiency Ratio

**Formula**: `Resource Efficiency % = (Actual Usage P95 / Requested) × 100`

**Example**: CPU requested = 2 vCPU, actual P95 usage = 1.1 vCPU  
`Efficiency = (1.1 / 2) × 100 = 55%`

**Target range**: 60-80%. Below 40% indicates over-requesting. Above 85% indicates under-requesting (risk of throttling/OOM under spikes).

**When to use**: Kubernetes capacity audit (Section 3 of the audit template). This metric identifies the "invisible waste" — capacity that is reserved by pod requests but not actually used, blocking scheduling of other workloads.

**Common mistakes**:
- Using average actual usage instead of P95 — this makes efficiency look higher than it is and misses burst behavior.
- Targeting 100% efficiency — this leaves no room for traffic variance. A service that is always at 100% efficiency is under-requested and will OOMKill or throttle under any spike.

---

## Formula 9 — N+1 Required Headroom

**Formula**: `Minimum Headroom % for N+1 = (1 / N) × 100`  
where N = current node count in the pool

**Example**: 5 nodes in a node pool. Lose 1 node: 4 nodes must absorb 1/5 = 20% more load each.  
Minimum headroom required: 20%. But target for safe N+1: 25% (with margin above minimum).

**Expanded**: For a 10-node pool: minimum N+1 headroom = 10%. Each surviving node absorbs 11% more load on a failure. If nodes are at 90% utilization, a single failure pushes survivors to 100%+ — incident.

**When to use**: Designing redundancy requirements. Also useful for quickly calculating whether a proposed utilization target is actually safe for the redundancy model of the service.

**Common mistakes**:
- Assuming N+1 is always adequate. For Tier 0 services (N+2 required), this formula understates the minimum headroom. Use `(2 / N) × 100` for N+2 requirements.
- Not applying this formula to database node pools — most teams think about N+1 for stateless services but forget it for database clusters.

## Formula 10 — Cost of Downtime

**Formula**: `Outage Cost = (Revenue/hour × Hours) + (Engineer cost/hour × Hours × Count) + SLA Penalties`

**Example**: 4-hour notification platform outage  
- Revenue at risk: $12,500/hour × 4 hours = $50,000
- Engineering cost: 8 engineers × 4 hours × $150/hour (loaded) = $4,800
- SLA penalty: $25,000
- **Total: $79,800**

**When to use**: Building the expected value argument for capacity investment proposals (ch08). Also useful in post-mortems to quantify the real cost of an incident for leadership reporting.

**Common mistakes**:
- Using salary instead of fully-loaded cost (salary × 1.4-1.7 for benefits, overhead). Use $150/hour loaded as a reasonable approximation for senior SRE/platform engineer.
- Excluding executive and communication time (VP involvement in a P1 incident is real cost).
- Not including tail costs: post-mortem time, remediation engineering, customer trust follow-up.

## Formula 11 — Reserved vs On-Demand Break-Even

**Formula**: `Break-even months = (Reserved upfront cost) / (On-demand monthly - Reserved monthly)`

**Example**: On-demand m5.2xlarge: $276/month. 1-year reserved (no upfront): $190/month. Savings: $86/month.  
If Standard Reserved with partial upfront: $950 upfront + $79/month = effective $158/month.  
Break-even vs no-upfront reserved: $950 / ($190 - $158) = 29.7 months — the break-even exceeds the 12-month commitment. No-upfront is better.

Savings Plans: commit to $X/hour spend. AWS applies up to 66% discount across instance families. No upfront required.

**When to use**: Annual cloud cost optimization review. Before converting on-demand baseline load to reserved.

**Common mistakes**:
- Reserving for burst capacity instead of baseline. Reserve only what runs 24×7×365.
- Comparing Reserved to Spot (wrong comparison). Spot is for interruptible workloads; Reserved is for guaranteed baseline. Compare Reserved to On-Demand for the same guaranteed availability.
- Ignoring instance type changes over the commitment period. Use Convertible Reserved or Savings Plans if you expect instance type changes in the next 12 months.

## Formula 12 — Cassandra Disk per Node (Capacity Planning)

**Formula**: `Required Disk per Node = (Logical Data Size × Replication Factor × Compaction Overhead) / Node Count`

**Parameters**:
- Logical data size: total unique data (before replication)
- Replication factor: typically 3 (RF=3)
- Compaction overhead: 1.25-1.35 (25-35% temporary space needed during compaction)
- Node count: current or target node count

**Example**: 20 TB logical data, RF=3, 10 nodes, compaction overhead 1.3  
`Required per node = (20 × 3 × 1.3) / 10 = 78 TB / 10 = 7.8 TB per node`

To stay below 60% disk per node, provision each node with: `7.8 / 0.60 = 13 TB per node`

**When to use**: Sizing a new Cassandra cluster or planning a ring expansion. The 60% threshold (not 70% or 80%) is Cassandra-specific — compaction requires the extra headroom.

**Common mistakes**:
- Using total raw disk provisioned instead of 60% threshold as the capacity limit — this sets you up for compaction failures.
- Forgetting compaction overhead and sizing to (logical × RF) / nodes — this understates true disk requirements by 25-35%.
- Not accounting for TTL expiry lag — deleted data is not immediately removed; it appears in SSTables until the next compaction. Peak disk usage often exceeds steady-state by 15-20%.
