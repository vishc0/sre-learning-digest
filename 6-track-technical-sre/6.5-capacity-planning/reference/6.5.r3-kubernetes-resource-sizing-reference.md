# Kubernetes Resource Sizing — Quick Reference

*Use as starting-point values. Always validate against actual usage via `kubectl top`, Prometheus, or Datadog after 2-4 weeks in production. VPA recommendations override these defaults.*

---

## Workload Type Sizing Guide

| Workload Type | CPU Request | CPU Limit | Memory Request | Memory Limit | Notes |
|---|---|---|---|---|---|
| Stateless web API (low traffic, <100 req/sec) | 100m | 500m | 128Mi | 256Mi | Scale out (HPA) for traffic increases; do not scale up |
| Stateless web API (high traffic, >100 req/sec) | 500m | 2000m | 512Mi | 1Gi | Set HPA target at 60% CPU; Limit:Request = 4:1 for burst |
| Background batch job | 1000m | 4000m | 1Gi | 4Gi | Set limits to prevent runaway jobs from starving other workloads |
| Message consumer (Kafka/RabbitMQ) | 200m | 1000m | 256Mi | 512Mi | Scale by partition count (Kafka) or queue depth, not by CPU |
| Java/JVM service (Spring Boot, etc.) | 500m | 2000m | 1Gi | 2Gi | Set `-Xmx` to 75% of memory limit (e.g., `-Xmx1536m` for 2Gi limit) |
| Go service | 200m | 1000m | 128Mi | 256Mi | Go is memory-efficient; start conservative and right-size from actuals |
| Python service (FastAPI, Flask) | 200m | 1000m | 256Mi | 512Mi | GIL limits CPU parallelism; multiple replicas preferred over higher limits |
| Node.js service | 200m | 1000m | 256Mi | 512Mi | Event loop is single-threaded; CPU limit rarely binding |
| Database sidecar / proxy (PgBouncer, Envoy DB proxy) | 100m | 500m | 64Mi | 128Mi | Minimal; proxy should not compete with main container for resources |
| Splunk forwarder (DaemonSet) | 200m | 500m | 256Mi | 512Mi | **Account for this in every node's effective allocatable calculation** |
| Datadog agent (DaemonSet) | 200m | 500m | 256Mi | 512Mi | **Account for this in every node's effective allocatable calculation** |
| AWS VPC CNI (DaemonSet) | 25m | 100m | 32Mi | 64Mi | Low overhead; do not omit from daemonset overhead calculation |
| Karpenter node termination handler (DaemonSet) | 10m | 100m | 32Mi | 128Mi | Minimal; required for graceful Spot interruption handling |
| Istio sidecar (Envoy proxy, injected) | 100m | 1000m | 128Mi | 1Gi | **At scale (100+ pods), sidecar overhead is significant — model it explicitly** |
| Cassandra node | 4000m | 8000m | 16Gi | 32Gi | Cassandra manages its own heap; set JVM `-Xmx` to 50-60% of memory limit |
| Redis (standalone) | 100m | 500m | 512Mi | 2Gi | Memory limit must exceed your largest dataset + overhead (20%) |
| Redis (cluster node) | 200m | 1000m | 1Gi | 4Gi | Set `maxmemory` in Redis config to 80% of memory limit |
| Prometheus (metrics collection) | 500m | 2000m | 2Gi | 8Gi | Memory grows with time series cardinality — watch closely |
| Grafana | 200m | 1000m | 256Mi | 1Gi | CPU spikes on dashboard load; limit prevents query queue explosion |

## Limit:Request Ratio Guide

| Ratio | Description | Right For | Watch Out For |
|-------|-------------|-----------|--------------|
| **1:1** | No burst allowed. Pod runs at exactly its request. | Predictable, steady workloads with no traffic spikes (e.g., background exporters, metrics collectors) | CPU throttling on any traffic variance — even small spikes; OOMKill on any memory growth |
| **1.5:1** | Moderate burst. Standard for most stateless services. | Well-characterized services with modest traffic variability | Verify actual peak usage does not regularly approach the limit |
| **2:1** | High burst. Good for bursty workloads with infrequent peaks. | Campaign-driven traffic (notification processors), latency-sensitive services | Noisy neighbor risk on shared nodes if many pods burst simultaneously |
| **3:1** | Very high burst. Use only for explicitly justified workloads. | Services with extreme traffic variance (e.g., event-driven batch that is idle 95% of the time) | Scheduling inefficiency — you are reserving capacity that is rarely used; prefer more replicas with lower burst |
| **∞ (no limit)** | **Anti-pattern**. Pod can consume the entire node. | Never in production | Pod starves all other workloads on its node; OOMKill of other pods; cluster instability |

## JVM Memory Sizing Pattern

*Java/JVM services need specific tuning to avoid OOMKills from GC pressure.*

```
Memory Request = Initial heap (Xms) + GC overhead buffer
Memory Limit   = Maximum heap (Xmx) + Non-heap overhead (metaspace, thread stacks, JIT code cache)

Rule of thumb:
  Memory Request = Xmx × 1.25     (25% buffer above heap for non-heap + GC pause headroom)
  Memory Limit   = Xmx × 1.50     (50% above heap — allows GC to complete without OOMKill)

Example for a service with -Xmx1g:
  Memory Request = 1.25 Gi
  Memory Limit   = 1.5 Gi

NEVER set Memory Request = Memory Limit on a JVM service:
  GC pauses temporarily use more memory than steady state.
  If limit = request, GC pressure causes OOMKill under normal load.
```

## Daemonset Overhead — Effective Allocatable Calculation

*Always subtract daemonset overhead from node capacity before calculating scheduling headroom for application workloads.*

**Example for m5.2xlarge (8 vCPU, 32 GB):**

| Component | CPU Reserved | Memory Reserved |
|-----------|-------------|----------------|
| Kubernetes system (kubelet, system-reserved) | 80m | 512Mi |
| AWS VPC CNI | 25m | 32Mi |
| Splunk forwarder | 200m | 256Mi |
| Datadog agent | 200m | 256Mi |
| Karpenter node termination handler | 10m | 32Mi |
| Istio CNI (if using Istio) | 100m | 128Mi |
| **Total DaemonSet + System Overhead** | **615m (0.615 vCPU)** | **1216Mi (~1.2 GB)** |
| **Effective Allocatable** | **7.385 vCPU** | **30.8 GB** |

*Use effective allocatable (not raw node capacity) when calculating scheduling utilization and headroom.*

## HPA Configuration Guidance

| Target Metric | Recommended Threshold | Notes |
|--------------|----------------------|-------|
| CPU utilization | 60% (launch) / 70% (steady state) | Lower at launch — uncertain traffic patterns |
| Memory utilization | 70% | Memory is slower to release — scale earlier |
| Custom metric: requests/sec | Set to P75 of current capacity per replica | More accurate than CPU for I/O-bound services |
| Custom metric: queue depth | Set at 2× normal processing lag | For message consumers; scale based on backlog |

**HPA settings for launch windows**:
```yaml
spec:
  minReplicas: 5          # Higher than normal minimum — pre-warm for launch traffic
  maxReplicas: 50         # Generous ceiling — do not constrain autoscaling at launch
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60   # More aggressive than steady-state 70%
```

## Anti-Patterns — Resource Configuration

| Anti-Pattern | Problem | Fix |
|-------------|---------|-----|
| **No resource limits set** | Pod can consume all resources on the node, starving other workloads | Always set CPU and memory limits |
| **Memory request = memory limit** | Prevents JVM/GC from bursting during collection; causes OOMKills under normal load | Set limit to at least 1.25× request for JVM; 1.1× for stateless |
| **CPU request >> actual usage** (efficiency <30%) | Wastes allocatable capacity; prevents scheduling of other pods; autoscaler provisions unnecessary nodes | Right-size to P90 actual with VPA recommendations |
| **CPU limit = CPU request** (ratio 1:1) on latency-sensitive service | Guaranteed CPU throttling on any traffic spike; P99 latency degrades while error rate looks fine | Set limit to 1.5-2× request for any user-facing service |
| **Extremely high CPU limit** (ratio >5:1) | Scheduling headroom is over-allocated; noisy neighbor risk during simultaneous burst | Cap at 2-3:1 for production workloads; justify higher ratios explicitly |
| **No liveness/readiness probes** | Kubernetes cannot detect pod failure; traffic routes to unhealthy pods | Always configure probes; tune failure thresholds appropriately |
| **Same resource config for all environments** | Dev/test pods consume production-equivalent capacity on shared clusters | Use LimitRange and ResourceQuota per namespace to enforce environment-appropriate limits |
| **Cassandra on shared nodes with latency-sensitive services** | Cassandra compaction creates I/O and CPU bursts that degrade co-located services | Dedicate node pools for Cassandra using node selectors or dedicated node pools |
