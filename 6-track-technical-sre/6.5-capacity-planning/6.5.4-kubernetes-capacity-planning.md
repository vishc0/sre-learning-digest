# Chapter 4 — Kubernetes Capacity Planning: The Cluster Is Not Infinite

## Two Capacity Planes, One Scheduling System

The most common misunderstanding about Kubernetes capacity is treating it as a single number — "the cluster has X CPUs." Kubernetes actually has two distinct capacity planes, and misunderstanding the gap between them is a primary cause of unexpected scheduling failures and false confidence.

**Cluster capacity**: the total resources available across all nodes. Node count × node size. This is what you see in your cloud console — 15 nodes × 8 vCPU = 120 vCPU of cluster capacity.

**Pod capacity**: the sum of resource requests across all scheduled pods. This is what Kubernetes uses for scheduling decisions — not actual utilization, but declared requests. A pod requesting 2 vCPU occupies 2 vCPU of scheduling capacity even if it only uses 200m at steady state.

**The gap between them** is your scheduling headroom — the unallocated capacity available for new pods, traffic bursts absorbed by HPA scaling, daemonset overhead, and the deployment headroom required to roll updates without stalling.

A cluster can be at 90% pod scheduling utilization (almost no scheduling headroom) while actual CPU utilization is only 40% — because all those pods over-requested their resources. This cluster will fail to schedule new pods despite appearing to have plenty of compute. Conversely, a cluster at 60% scheduling utilization but 85% actual CPU is running hot — autoscaling may not trigger fast enough to prevent latency degradation.

Effective Kubernetes capacity planning requires tracking both planes and the gap between them.

## Node Sizing Strategy

The node sizing decision shapes your fault tolerance, scheduling flexibility, and operational overhead.

**Fewer large nodes**:
- Better bin-packing efficiency: fewer scheduling constraints, less fragmentation
- Simpler management: fewer nodes to patch, drain, cordon
- Higher blast radius per failure: a large node failure takes more pods with it

**More small nodes**:
- Smaller blast radius: losing one node affects fewer pods
- More gradual autoscaling steps: each additional node is a smaller increment
- Higher per-node overhead: more OS processes, more kubelet, more CNI agents

**For EKS in a notification platform context**:
- CPU-bound workloads (message processing, API servers): `c5.xlarge` (4 vCPU/8GB) — high CPU-to-memory ratio matches the workload
- Memory-bound workloads (caching layers, JVM services): `m5.2xlarge` (8 vCPU/32GB) — balanced, good for mixed EKS clusters
- Cassandra nodes: `r5.2xlarge` (8 vCPU/64GB) — memory-heavy because Cassandra's read performance depends heavily on OS page cache

Run heterogeneous node pools: a CPU-optimized pool for stateless services and a memory-optimized pool for data-adjacent workloads. Use node selectors or node affinity to direct workloads to the appropriate pool. Do not mix Cassandra onto the same nodes as your API services — Cassandra's compaction creates I/O and CPU bursts that will noisy-neighbor your latency-sensitive services.

## Autoscaling: What It Does and What It Cannot Do

**Cluster Autoscaler (CA)** watches for pods that fail to schedule due to insufficient resources, then adds nodes to the cluster. It is reactive by design — it acts after scheduling fails. The provisioning delay is 2-5 minutes (time to launch an EC2 instance, join the cluster, become ready). For a traffic spike that ramps over minutes, CA can keep up. For a sudden vertical spike (a campaign send that fires immediately), CA may be too slow.

**Karpenter** is Anthropic's preferred node provisioner on AWS (now AWS-native). Unlike CA, which manages existing Auto Scaling Groups, Karpenter directly provisions instances based on pending pod requirements. It can select the optimal instance type for each pending pod, is faster than CA (sub-minute provisioning in many cases), and consolidates underutilized nodes proactively. For EKS shops on AWS, Karpenter is generally the better choice for production workloads.

**Critical limitation both share**: neither tool fixes a cluster where the baseline node pool maximum is set too low. If your Karpenter node pool has a maximum of 20 nodes and 18 are already provisioned, you have 2 nodes of autoscaling headroom. A traffic spike that requires 10 more nodes will be rejected. Your autoscaling maximum must be set above your expected peak — not your current baseline.

**The autoscaling safety valve model**: think of autoscaling as a safety valve for variance, not a substitute for capacity planning. You provision for your P75 expected demand, and autoscaling covers P75 to P99. If you provision only for P50 and expect autoscaling to cover everything above that, you are relying on a reactive system to handle events that happen faster than it can respond.

## Pod Resource Requests vs Actual Usage

The single most impactful capacity optimization activity in most Kubernetes environments is right-sizing resource requests. Over-requesting is extremely common — especially in lift-and-shift migrations where application teams set requests to the maximum they ever think they might need, rather than the operational P75-P90.

**Over-requesting consequences**:
- Pod consumes scheduling capacity it does not need
- Cluster autoscaler provisions more nodes than necessary
- You pay for nodes that are underutilized because scheduling is full but actual CPU is low
- HPA triggers late (because it scales on actual utilization, but scheduling capacity is already constrained)

**Under-requesting consequences**:
- CPU throttling (pod's actual usage exceeds its limit, gets throttled by cgroup)
- OOMKills (pod's actual memory exceeds its limit, kernel terminates it)
- Both cause latency spikes and pod restarts that degrade reliability

**Target**: set requests at P75-P90 of actual usage. Measure actual usage via `kubectl top pods` (instantaneous), Prometheus `container_cpu_usage_seconds_total` (trend), or your observability platform (Datadog, Splunk Observability).

For your DND platform: measure actual CPU utilization of notification processors over a 30-day window including campaign send events. Set requests at the P90 of that distribution. Set limits at 1.5-2× the request (allowing burst without OOM risk for CPU; for memory, set limit at 1.2-1.3× request maximum — memory is harder to burst safely).

**Vertical Pod Autoscaler (VPA)**: VPA observes actual usage and produces right-sizing recommendations. Run it in `recommend` mode first — it will tell you what requests and limits should be without making changes. Review the recommendations for 2-4 weeks before considering `auto` mode. In `auto` mode, VPA restarts pods to apply new resource specs — disruptive in production if not managed carefully.

## Cluster Capacity Reservation Requirements

You should always maintain 10-15% unallocated cluster capacity — scheduling headroom that is never filled by application workloads. This reserve covers three critical use cases.

**Daemonset overhead**: every node runs daemonsets — Splunk forwarder, Datadog agent, AWS CNI plugin, potentially Istio CNI or Karpenter node termination handler. Each daemonset pod claims scheduling capacity on every node. On a 15-node cluster, if each daemonset pod requests 200m CPU and 256Mi memory, and you run 4 daemonsets, that is 800m CPU and 1Gi memory per node consumed by infrastructure — before any application workload. This is invisible in cluster-level utilization summaries but prevents application pods from scheduling when nodes are "almost full."

**Rolling deployment headroom**: Kubernetes rolling deployments create new pods before terminating old ones. During the rollout, you are running more pods than your steady-state requires. If your cluster is at 90% scheduling utilization, a rolling deployment of a high-replica service will fail to schedule new pods — the deployment stalls, and your rollout hangs. Production deployments that cannot complete their rollout are a reliability risk: you have a partially updated service, half old version, half new, with diverging behavior.

**Node failure headroom**: when a node is cordoned (scheduled for maintenance) or lost unexpectedly, its pods must reschedule onto remaining nodes. Without scheduling headroom, those pods will be Pending — effectively down — until new nodes are provisioned (2-5 minutes). Maintaining 10-15% scheduling headroom means a single node failure results in pod rescheduling within seconds, not minutes.

## Director Takeaway

Kubernetes capacity planning is an exercise in managing two planes (cluster capacity and pod capacity) and the gap between them. Autoscaling is a variance absorber, not a capacity strategy. Right-sizing pod requests is the highest-ROI capacity optimization in most Kubernetes environments. And the 10-15% scheduling headroom reserve is not waste — it is what keeps your deployments rolling and your failure recovery fast. Own these levers as Director-level operational standards and enforce them as engineering requirements for any service onboarding to your platform.
