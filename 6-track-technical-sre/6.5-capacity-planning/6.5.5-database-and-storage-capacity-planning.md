# Chapter 5 — Database and Storage Capacity Planning: Where Capacity Failures Are Hardest to Recover

## Why Database Capacity Failures Are Uniquely Dangerous

When a stateless service runs out of capacity, the recovery path is relatively straightforward: add replicas, restart pods, scale out the deployment. The service is stateless — nothing is lost when you kill and recreate pods, and new instances start serving traffic within seconds.

Database capacity failures are a different category of problem entirely. The recovery paths are slow, risky, and operationally complex. Unlike compute, you cannot simply add a database node and expect instant relief. Cassandra requires data streaming and rebalancing. PostgreSQL read replica promotion involves failover coordination. Redis replication must fully catch up to the primary before the new replica serves reads. The consequence: database capacity failures have long, fragile recovery tails during which your service is degraded or down.

There is a second, more insidious dimension. At 95% disk on a Cassandra node, you cannot compact — and compaction is the operation that recovers disk space after deletes and TTL expiry. The worse your disk situation gets, the fewer recovery options you have. This is the inverse of compute capacity, where you can at least throttle workloads or drop non-critical jobs to buy time. With a full disk, your options collapse rapidly.

The governing principle for database capacity planning: **start the expansion process at 60% utilization, not 80%**. The time to recover from a database capacity event is measured in hours to days. You need a large enough runway to act before you lose all maneuvering room.

## Disk Growth Rate Modeling

Disk is the primary capacity constraint for most data stores. It is less elastic than compute, more catastrophic when exhausted, and growth is often non-linear (new workloads onboard, retention periods change, traffic volumes step up).

**Measurement methodology**:
1. Measure disk used per day or per week over the last 90 days from your monitoring system (Splunk, Prometheus, CloudWatch)
2. Calculate the daily growth rate: `(Current GB - 90-days-ago GB) / 90`
3. Identify inflection points — sudden increases in growth rate indicate a new workload, a retention policy change, or a data model change that is producing more writes than expected
4. Project forward: `Days Until 80% Full = (0.80 × Provisioned GB - Current GB) / Daily Growth Rate`

This "action date" — the projected date when you cross 80% — is what you put on your capacity dashboard. It is not a warning; it is a deadline. When the action date is less than 60 days out, you must have a provisioning plan in motion.

**For Cassandra specifically**: disk capacity is not the raw logical data size. Account for:
- Replication factor: RF=3 means 3 copies of every row across 3 nodes. 1TB of logical data = 3TB of raw disk across the cluster.
- Compaction overhead: during compaction, Cassandra must hold both the old SSTable files and the new compacted file simultaneously. Peak disk usage during compaction is typically 1.5× the steady-state size. A node at 65% steady-state can briefly hit 97%+ during compaction — and if disk is full, compaction cannot start, and you cannot reclaim space from deletions or TTL expiry.
- Rule of thumb: for Cassandra, never provision beyond 60% of raw disk in steady state. This is not a conservative choice — it is the minimum safe operating threshold given compaction behavior.

## Cassandra Ring Capacity Planning — DND Platform Context

Your DND notification platform runs on Cassandra for notification state, delivery tracking, and suppression tables. This is a write-heavy workload with high throughput requirements, and Cassandra's capacity planning has specific characteristics that distinguish it from relational databases.

### Key Metrics to Track Daily

| Metric | Source | Warning | Critical |
|--------|--------|---------|---------|
| Disk per node (GB used / GB provisioned) | nodetool status, CloudWatch | 55% | 65% |
| Read latency P99 per node | nodetool tpstats, Cassandra JMX | 20ms | 50ms |
| Write latency P99 per node | nodetool tpstats, Cassandra JMX | 10ms | 25ms |
| Pending compactions per node | nodetool compactionstats | 20 | 50 |
| Hints stored per node | nodetool tpstats | >0 trending up | >10K |
| Node count vs replication factor | nodetool status | RF not met | Node down |

Pending compactions above 50 per node indicate the node is falling behind on compaction — typically a disk pressure or CPU pressure signal. Hints accumulating means writes are being buffered for a node that is not responding to writes — a node that is not fully healthy or is falling behind under load.

### Capacity Trigger for Adding Nodes

Add Cassandra nodes when **average disk per node exceeds 60%** on a sustained basis (over at least a week, not just a spike). Do not wait until 70-75%. The reason is the compaction requirement: you need space for compaction to run, and compaction is what keeps disk usage from growing indefinitely after deletes and TTL expiry.

When average node disk crosses 60%, initiate the capacity expansion immediately. The process:

1. **Pre-provision new nodes**: launch new EC2 instances, install Cassandra, configure with the correct cluster name and seeds
2. **Bootstrap into the ring**: `cassandra -f` on the new node. Cassandra will automatically begin streaming data from existing nodes (this is the slow part — streaming rates depend on network bandwidth and cluster load)
3. **Monitor streaming progress**: `nodetool netstats` shows active streaming. Do not interrupt streaming in progress.
4. **Run cleanup on existing nodes**: after the new nodes are fully bootstrapped and all replicas are confirmed present (`nodetool status` shows all nodes UN — Up Normal), run `nodetool cleanup` on each original node to remove the token ranges that have moved to the new nodes. This is what actually reclaims disk space.
5. **Validate**: after cleanup, verify disk per node has decreased, latency is stable, pending compactions are not accumulating

**Timeline reality check**: a Cassandra ring expansion on a cluster with 5TB of data per node (15TB total at RF=3) can take 4-8 hours of streaming per new node at a healthy streaming rate, plus cleanup time. A full expansion of a large cluster takes 24-48 hours from decision to complete. **This means you must initiate the expansion process when you are at 60% disk, not when you are at 80%.** At 80%, you are 20-30 days from critical — not enough runway for a safe expansion.

### Director Framing for Cassandra Capacity

The lead time for a safe Cassandra capacity expansion is 4-6 weeks: 1-2 weeks to plan and get approval, 1-2 weeks to provision and validate new nodes in a staging ring, 2 weeks for production bootstrapping and cleanup at a conservative pace that does not impact production latency.

When you present Cassandra capacity investment to a VP, the lead time argument is your strongest case for early action: "If we start today at 58% disk, we have a safe runway. If we wait until 75%, the expansion will be in progress during the period of maximum risk, and any operational issue during bootstrapping could cascade into a disk-full event before recovery is complete."

## Connection Pool Capacity

Connection pool saturation is a frequently overlooked database capacity dimension. It manifests as request queuing — connections wait for an available slot in the pool — which increases latency in a way that is easy to misdiagnose as a database performance problem when it is actually a capacity problem.

**The mechanism**: most database clients maintain a pool of open connections (e.g., HikariCP for Java, SQLAlchemy pool for Python). When all connections in the pool are in use and a new request arrives, the request waits in a queue for a connection to become available. Queue depth grows → latency grows. Queue hits its max depth → requests fail with timeout.

**For Cassandra**: the DataStax Java driver maintains a connection pool per host. Default is 1 connection per host for local datacenter hosts (can be configured). Under high concurrency, you may need to increase connections-per-host. Monitor the driver's in-flight requests per connection metric — if consistently at the per-connection request limit, increase connections-per-host.

**For PostgreSQL** (if used for supplementary data): each connection is a process, consuming 5-10MB of RAM plus stack space. At 500 connections, you are using 2.5-5GB just for connection overhead — before any actual query work. PgBouncer (connection pooler) is non-negotiable at scale: it multiplexes hundreds of application connections to a small number of actual PostgreSQL connections, dramatically reducing RAM overhead and connection establishment latency.

**Capacity signal to monitor**: connections active vs pool maximum. When active connections consistently exceed 80% of pool maximum, the pool is becoming a bottleneck. Add read replicas (for read-heavy workloads) or increase pool size (if the database server can support more connections without performance degradation).

## Read/Write Replica Scaling

When read replicas approach 70% CPU utilization, the right action is to add more replicas before CPU saturation causes query latency to degrade. Do not wait until the replica is at 90% CPU — at that point, query execution times have already increased, which means each active connection holds a connection slot longer, which accelerates the connection pool exhaustion spiral.

The Director decision: build read scaling into your service tier definition. Tier 0 services should have a minimum replica count specified as part of their capacity SLA. "Database must have at least N+1 read replicas with CPU headroom below 60%" is an enforceable standard.

---

## Director Takeaway

Database capacity planning has longer lead times, harder recovery paths, and less autoscaling flexibility than compute capacity planning. For Cassandra on your DND platform, act at 60% disk — not 70%, not 80%. Build the 4-6 week expansion timeline into your capacity model. Present the lead time argument to VPs as the reason for early action. Connection pool saturation and read replica CPU are secondary signals that compound database capacity problems — monitor both. Databases are where capacity planning failures become prolonged, customer-impacting incidents. They deserve more conservative headroom targets than your stateless compute tier.
