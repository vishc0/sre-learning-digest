# Reference: On-Call Tool Comparison — PagerDuty vs Opsgenie vs VictorOps vs Grafana OnCall

*Last reviewed: 2026-07. Tool capabilities change frequently — verify current feature sets at vendor sites before making procurement decisions. Pricing models in particular shift with contract cycles.*

## Decision Guide: When to Choose Each Tool

| Scenario | Recommended Tool | Rationale |
|---------|-----------------|-----------|
| Large enterprise (200+ engineers), complex routing, strong compliance requirements | **PagerDuty Business/Enterprise** | Most mature routing, strongest enterprise SSO/SAML, broadest integration catalog |
| Mid-sized org (50-200 engineers) deeply invested in Atlassian ecosystem | **Opsgenie Standard/Enterprise** | Native Jira integration reduces friction; competitive pricing; adequate routing for most needs |
| Organization with significant Splunk investment (SIEM, ITSI) | **VictorOps (Splunk)** | Native Splunk integration eliminates integration overhead; alerts route directly without webhook configuration |
| Cloud-native org running Grafana + Prometheus + Kubernetes natively | **Grafana OnCall** | Native Prometheus/Alertmanager integration; eliminates translation layer; OSS fits open-source-forward culture |
| Small team (<20 engineers), cost-sensitive, open-source-comfortable | **Grafana OnCall (self-hosted)** | Zero per-user cost; covers essential rotation, escalation, alerting for teams willing to manage infrastructure |
| Starting fresh with no existing tool investment | **Opsgenie Standard** | Best price/feature balance for new deployments at mid-scale; grow to Enterprise if needed |

## Known Limitations by Tool

**PagerDuty:**
- Cost is highest in the category — can be prohibitive for smaller teams or budget-constrained orgs
- Schedule configuration UI, while powerful, has a steep learning curve for complex patterns
- Analytics require Business or higher tier — Professional tier reporting is limited
- No self-hosted option; SaaS dependency is a compliance consideration in some industries

**Opsgenie (Atlassian):**
- Atlassian's acquisition and integration of Opsgenie has been slow; some features lag PagerDuty by 12-18 months
- Native incident response (war room) is weaker than PagerDuty's offering at equivalent tier
- Non-Atlassian integration documentation is less comprehensive
- Mobile app phone escalation reliability has had intermittent issues in some regions (verify with references)

**VictorOps (Splunk):**
- Product development has slowed since Splunk acquisition; future roadmap is tied to Splunk's ITSI roadmap
- Best value only for organizations with existing Splunk investment — standalone pricing is not competitive
- Alert routing is less flexible than PagerDuty or Opsgenie for non-Splunk alert sources
- UI is the least modern of the four tools

**Grafana OnCall:**
- OSS self-hosted requires Kubernetes expertise to operate reliably at scale
- Mobile app is less mature; phone escalation reliability varies by deployment
- No built-in analytics — you build your own dashboards (which is fine if you are already running Grafana)
- Community support only for OSS; enterprise support available via Grafana Cloud

## Migration Considerations: Switching from PagerDuty to Opsgenie

The most common migration path (when driven by Atlassian ecosystem alignment or cost).

**What migrates cleanly:**
- On-call schedules (export PD schedule → recreate in Opsgenie; APIs on both sides)
- Escalation policies (logical equivalents exist; some conditional logic may need simplification)
- Service definitions (map PD services to Opsgenie teams)
- Alert routing rules (require manual recreation; no direct import)

**What requires redesign:**
- Alert payload format — Opsgenie alert templates differ from PD; all alert sources need updated webhook configs
- Integration configurations — each alert source (Splunk, Prometheus, CloudWatch) needs reconfigured
- Reporting and analytics — PD analytics do not export to Opsgenie; rebuild from scratch

**Migration risk:** High for teams with complex routing rules (>20 services, >10 escalation policies) or heavy PagerDuty API usage in internal tooling. Budget 4-8 weeks of engineering time for a complete migration plus parallel-run validation period. Never cut over without running both tools in parallel for at least 2 weeks.

**Migration recommendation:** Only migrate if there is a concrete driver (Atlassian contract discount, PagerDuty cost elimination, Jira integration requirement). Migrations are disruptive; the productivity cost during transition is real. "Our CTO prefers Atlassian tools" is not sufficient justification.
