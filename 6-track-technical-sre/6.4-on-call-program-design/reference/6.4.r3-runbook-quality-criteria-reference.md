# Reference: Runbook Quality Criteria and Evaluation Rubric

---

## Scoring Rubric

Score each runbook against the following 10 criteria. Each criterion is scored 0, 1, or 2.

**0** = Missing entirely
**1** = Present but incomplete or vague
**2** = Complete, specific, and immediately usable

| # | Criterion | Score 0 (Missing) | Score 1 (Partial) | Score 2 (Complete) |
|---|-----------|-------------------|-------------------|--------------------|
| 1 | **Alert linkage** | No reference to the alert that triggers this runbook | Alert name mentioned but not the exact PagerDuty alert name | Exact PagerDuty/Opsgenie alert name and service listed; alert links directly to this runbook |
| 2 | **Trigger description** | No description of what condition causes the alert to fire | General description ("when the service is slow") | Specific condition with threshold ("when p99 latency exceeds 2000ms for 5 consecutive minutes") |
| 3 | **Context / impact statement** | No description of what the service does or what breaks | General description of the service | Specific description of service function + what customer-visible impact occurs when this alert fires + downstream effects |
| 4 | **Triage steps specificity** | No triage steps, or "check the logs" level | Some specific commands but missing expected outputs | All triage steps include: exact command, expected output for healthy state, and "if output shows X, proceed to step N" branching |
| 5 | **Resolution paths** | Single resolution action with no branching | Multiple actions listed but no decision logic | Decision tree: "if triage step N shows X → resolution path A; if shows Y → resolution path B" with clear entry and exit for each |
| 6 | **Expected resolution state** | No description of what resolved looks like | "Alert auto-resolves" or similar circular definition | Specific: metric name + threshold value + dashboard panel + time window that confirms resolution (e.g., "delivery rate above 95% for 5 consecutive minutes in Splunk panel X") |
| 7 | **Escalation path** | No escalation guidance | "Escalate to the team" | Named tier or rotation (not a person), specific contact method, specific information to provide when escalating, time threshold before escalating |
| 8 | **Last tested date** | No date | Date present but >180 days ago | Date present, ≤90 days ago, with method noted (game day / real incident / manual walkthrough) |
| 9 | **Owner identified** | No owner | Team identified but no individual | Named individual with explicit ownership responsibility and acknowledgment date |
| 10 | **Related runbooks linked** | No links to other runbooks | Some related runbooks mentioned but not linked | All directly related runbooks linked (upstream cause, downstream effect, parallel condition) |

**Maximum score: 20 points**

---

## Score Interpretation

| Score | Rating | Interpretation | Action |
|-------|--------|---------------|--------|
| 19-20 | **Excellent** | Runbook is complete, specific, and immediately usable by any competent engineer | Maintain quarterly; flag for use as a template for other runbooks |
| 15-18 | **Good** | Runbook covers the essential path; minor gaps exist | Review within 60 days; assign owner to close specific gaps |
| 9-14 | **Acceptable** | Runbook provides directional guidance but requires engineer judgment to fill gaps | Do not rely on this runbook alone for P0 incidents; assign owner for improvement within 30 days |
| 5-8 | **Poor** | Runbook provides starting context but is insufficient for independent response | Treat as a starting point, not a complete guide; experienced engineer review required for any use; prioritize improvement immediately |
| 0-4 | **Unusable** | Runbook provides no reliable guidance; may cause more confusion than benefit | Do not reference as a runbook; rebuild from scratch; schedule game day to validate rebuilt version |

## Anti-Pattern Examples with Before/After Rewrites

### Anti-Pattern 1: Vague Triage

**Score: 0 — Missing** for triage specificity

```
Before:
## Triage
Check if the service is running. Look at the logs for errors.
```

**Why it fails**: "Check if the service is running" defines no method, no expected output, and no next step. "Look at the logs" defines no command, no log location, no pattern to search for, no interpretation of findings.

```
After:
## Triage

Step 1: Verify pod status
kubectl get pods -n prod -l app=notification-service
Expected output (healthy): all pods in Running state, READY column showing 1/1
If any pod shows: CrashLoopBackOff → proceed to Step 3 (crash loop path)
If any pod shows: Pending → proceed to Step 4 (scheduling failure path)
If all pods Running but alert still active → proceed to Step 2

Step 2: Check recent error rate
kubectl logs -n prod deployment/notification-service --since=10m | grep "ERROR" | wc -l
Expected (healthy): < 20 errors in a 10-minute window
If > 20 errors → classify error type:
  kubectl logs -n prod deployment/notification-service --since=10m | grep "ERROR" | awk '{print $NF}' | sort | uniq -c | sort -rn | head -5
  Common error classes:
    "connection refused to carrier-api" → proceed to Step 5 (carrier API path)
    "queue full" → proceed to Step 6 (RabbitMQ backpressure path)
    "timeout" → proceed to Step 7 (upstream latency path)
```

### Anti-Pattern 2: Restart Without Context

**Score: 1 — Partial** for resolution paths (present but missing pre-check and post-verification)

```
Before:
## Resolution
If the service is unhealthy, restart it. Monitor after restart.
```

**Why it fails**: No pre-restart check (data loss risk if in-flight messages exist). No specific restart command. "Monitor after restart" defines no monitoring method and no monitoring duration.

```
After:
## Resolution Path A: Pod restart (for CrashLoopBackOff with no in-flight messages)

Pre-restart check (REQUIRED):
kubectl exec -n prod deployment/notification-service -- \
  curl -s localhost:8080/actuator/health/inflight | jq '.inflight_count'
If inflight_count > 0: wait 60 seconds, recheck. Do NOT restart while messages are in-flight.
If inflight_count = 0: proceed with restart.

Restart command:
kubectl rollout restart deployment/notification-service -n prod

Verify restart:
kubectl rollout status deployment/notification-service -n prod
Expected: "successfully rolled out" within 3 minutes
If rollout does not complete in 3 minutes: do NOT proceed; escalate to secondary (see Escalation section)

Post-restart verification:
1. kubectl get pods -n prod -l app=notification-service
   Expected: all pods Running, READY 1/1
2. Check delivery rate in Splunk: [dashboard URL]
   Expected: delivery rate above 95% within 2 minutes of pod readiness
3. If delivery rate does not recover within 5 minutes of pod readiness → escalate

Resolution confirmed: delivery rate above 95% for 5 consecutive minutes with no new ERROR logs
```

---

### Anti-Pattern 3: Undefined Escalation

**Score: 0 — Missing** for escalation path

```
Before:
## Escalation
If you cannot resolve the issue, escalate to the team.
```

**Why it fails**: "The team" is not a contact. This does not define when to escalate (time threshold), how to escalate (contact method), or what information to provide.

```
After:
## Escalation

When to escalate:
- Triage steps complete and root cause not identified within 20 minutes of page receipt
- Root cause identified but resolution requires: database access, carrier account changes, or service account credentials
- Delivery rate below 80% (below P0 threshold — auto-escalation should trigger, but confirm)
- Any data loss confirmed or suspected

How to escalate — Tier 2 (Secondary On-Call):
- Escalate via PagerDuty secondary escalation button in the active incident
- Simultaneously post in #incident-active: "Escalating [incident ID] to secondary. [2-sentence status]. [your current hypothesis]. Steps taken: [bullet list]"
- Do not wait for secondary acknowledgment before continuing triage

Information to provide when escalating:
1. Current delivery rate (exact number from Splunk)
2. Error class identified in triage Step 2 (or "error class not yet identified")
3. Pod status (all running / N pods degraded)
4. Steps taken and results
5. Your current best hypothesis for root cause

If secondary does not acknowledge within 10 minutes → escalate to Tier 3 (on-call manager)
For Tier 3 escalation: phone call required for P0; Slack DM acceptable for P1.
Tier 3 contact: [PagerDuty manager schedule link]
```

## Freshness Scoring Add-On

Use this supplemental rubric to score runbook staleness risk. Apply after the main rubric.

| Criterion | Fresh (Low Risk) | Aging (Medium Risk) | Stale (High Risk) |
|-----------|-----------------|---------------------|------------------|
| **Last updated date** | <30 days | 30-90 days | >90 days |
| **Last tested (game day or incident)** | <60 days | 60-180 days | >180 days or never |
| **Last used in actual incident** | <30 days | 30-90 days | >90 days or never |
| **System changed since last review?** | No significant changes | Minor changes (config, not architecture) | Major changes (new dependencies, k8s migration, schema change) |

**Freshness risk score**: count the number of "High Risk" columns.
- 0 High Risk: runbook is current
- 1 High Risk: schedule review within 30 days
- 2+ High Risk: treat as potentially incorrect; do not use as sole guide; experienced engineer must verify before following

## Quick Pre-Publish Checklist for Runbook Authors

Before publishing a new runbook, confirm all 10 items:

```markdown
### Runbook Pre-Publish Checklist

- [ ] Exact PagerDuty/Opsgenie alert name recorded at top of runbook
- [ ] Alert updated to include direct link to this runbook URL
- [ ] Context section explains what the service does and what breaks (2-3 sentences minimum)
- [ ] Every triage step includes a specific command AND the expected output for a healthy system
- [ ] Resolution section has at least one decision branch ("if X, then Y; if Z, then W")
- [ ] Resolution section defines what "resolved" looks like — specific metric, not "alert clears"
- [ ] Escalation section names a rotation or tier (not a person's name) and states when and how to escalate
- [ ] Last tested date set (even if it is "published but not yet tested" — be honest)
- [ ] My name is in the "Owner" field — I accept responsibility for keeping this current
- [ ] Related runbooks linked (upstream: what could cause this; downstream: what this could trigger)

**Pre-publish review**: Have one other SRE read the runbook cold and confirm they could follow it without asking you a question. If they need to ask, the runbook is incomplete.
```
