# Playbook: Alert Reduction Campaign

## Purpose

Systematically reduce pages per engineer per week by 30-50% over one quarter, without compromising reliability signal.

**Why this target is achievable**: In most on-call programs that have never run an alert reduction campaign, 40-60% of pages are either non-actionable (auto-resolve without human action), duplicative (same condition generating multiple alerts), or stale (monitoring a changed or deprecated system). The reduction potential is real — but it requires deliberate, structured work, not ad-hoc tuning.

**Expected duration**: 8-12 weeks (one quarter)
**Sprint impact**: One dedicated sprint for Phase 1-2; incremental work in subsequent sprints for Phase 3-4

---

## Roles

| Role | Responsibility |
|------|----------------|
| **Campaign Lead** (senior SRE or SRE Manager) | Owns the campaign, maintains the alert inventory worksheet, drives Phase 1-3, reports progress |
| **Alert Owners** (service team engineers) | Provide context on individual alerts, confirm classification, sign off on changes to their service's alerts |
| **Director / VP** (sign-off authority) | Approves deletions — specifically alerts that have fired recently (within 30 days) or are P0/P1 |
| **PagerDuty/Opsgenie Admin** | Executes configuration changes; maintains audit log of what changed and when |

---

## Phase 1 — Alert Inventory

### Step 1: Export Full Active Alert Catalog
**Who**: Campaign Lead + PagerDuty/Opsgenie Admin

Export all active alerts/alert rules from your alerting stack:

From PagerDuty:
- Services → Export service list → Cross-reference with integration rules for each service
- Note: PagerDuty does not have a single "alert catalog" export — you will need to pull this per service or via the API

From Opsgenie:
- Integration List → for each integration, export rules
- Or use Opsgenie API: `GET /v2/policies` to retrieve all alert policies

From Splunk (saved search alerts):
- `| rest /servicesNS/-/-/saved/searches | search alert.track=1 | table title, alert.condition, cron_schedule, actions`

The output of this step is a raw list of every alert that can generate a page.

---

### Step 2: Tag Each Alert with Service Owner
**Who**: Campaign Lead + Team Leads

For each alert, identify:
- Service name (the application or infrastructure component being monitored)
- Team/squad that owns the service
- Named engineer who is the primary owner of this alert

Alerts with no identifiable owner are automatically flagged for review — these are orphaned alerts.

---

### Step 3: Record Last-Fired Date for Each Alert
**Who**: Campaign Lead + PagerDuty Admin

From PagerDuty: pull last incident timestamp per service/alert name
From Splunk: `index=pagerduty | stats max(_time) as last_fired by alert_name | eval days_since=round((now()-last_fired)/86400,0)`

Classify by recency:
- Fired within 30 days: active
- Fired 31-90 days ago: aging
- Fired 90+ days ago: stale candidate
- Never fired (in 90-day window): dormant candidate

---

### Step 4: Record Actionable Rate Where Known
**Who**: Campaign Lead

From the audit playbook data (if already run) or from PagerDuty incident data, record the actionable rate per alert. If this data is not available at the alert level, estimate from incident notes or mark as "unknown."

Alerts with unknown actionable rates are de-prioritized for deletion but prioritized for observation in Phase 3.

---

### Step 5: Tag Alert Type for Each Alert
**Who**: Campaign Lead + Alert Owners

For each alert, classify:

| Type | Definition |
|------|-----------|
| **Symptom** | Fires on observable customer or SLO impact (error rate, latency, availability) |
| **Cause** | Fires on an infrastructure metric that may lead to impact (CPU, memory, queue depth) |
| **Watchdog** | Informational; fires to confirm a system is alive, not to signal a problem |
| **Flapping** | Repeatedly fires and auto-resolves; engineer rarely takes action |
| **Stale** | Monitors a system that has changed, been deprecated, or been decommissioned |

Use alert owner context to classify. Alert owners should know which of their alerts are cause-based vs. symptom-based.

---

## Phase 2 — Classification

### Step 6: Apply the Classification Decision Tree
**Who**: Campaign Lead + Alert Owners (for each alert)

For each alert, answer in sequence:

```
Q1: Is this alert still monitoring an active system?
    No → CANDIDATE: DELETE (stale)
    Yes → continue

Q2: Does this alert fire on customer-observable impact (error rate, SLO breach, availability)?
    Yes → CANDIDATE: KEEP
    No → continue

Q3: Does this alert fire repeatedly for the same underlying condition as another alert?
    Yes → CANDIDATE: MERGE
    No → continue

Q4: Is the severity assigned to this alert appropriate for the actual impact?
    Too high → CANDIDATE: DOWNGRADE
    Appropriate → continue

Q5: Is the resolution to this alert always the same scripted action with no investigation needed?
    Yes → CANDIDATE: AUTOMATE
    No → CANDIDATE: KEEP (review threshold)
```

---

### Steps 7-11: Record Classification for Each Alert

Enter each classification into the alert audit worksheet template. This is the central artifact of the campaign.

For each alert classified as DELETE, MERGE, DOWNGRADE, or AUTOMATE — require written rationale from the alert owner before proceeding. The rationale must answer: "What would we miss if this alert were gone or changed?"

---

### Step 12: Director Sign-Off on Deletion List
**Who**: Director/VP

Before any deletions are executed, the campaign lead presents the deletion list to the Director for review. The Director approves or rejects each deletion candidate.

**Decision gate**: If a deletion candidate has fired within the last 30 days, the Director must explicitly approve. Do not delete recently-active alerts without leadership sign-off regardless of how obvious the case seems. Alert history is irreversible — the alert can always be recreated, but the organizational trust required to make these changes can be lost if a deletion causes a missed incident.

---

## Phase 3 — Execution

### Step 13: Execute Deletions First
**Who**: PagerDuty/Opsgenie Admin + Campaign Lead

Deletions have the highest impact on page count and are fully reversible (the alert can be recreated from the audit worksheet). Execute deletions in the first week.

**Before deleting**: screenshot the alert configuration and save to the campaign folder. This is the "undo" record.

Track page rate daily after deletions. A drop in page rate confirms the deletion was correct. No change in page rate for a deleted alert means the alert was already inactive (which is fine — it was correctly identified as stale).

---

### Step 14: Execute Merges Second
**Who**: Alert Owners + PagerDuty Admin

Merges require alert rule changes at the source (Splunk, Prometheus, CloudWatch). Work with alert owners to consolidate duplicate conditions into a single alert with a richer context payload.

Example: five separate pod-restart alerts for five pods in the same deployment → one alert with the payload "N pods in deployment/X restarted in the last 5 minutes" with the count as a field.

---

### Step 15: Execute Downgrades Third
**Who**: Alert Owners + PagerDuty Admin

Downgrade means changing the severity of the alert — typically from P1/page to P2/low-urgency or from P2/page to P3/ticket. The alert still fires; it no longer wakes anyone up at night.

Downgrade candidates: cause-based alerts that provide useful diagnostic information but do not indicate immediate customer impact. "Node CPU at 85%" is worth knowing about but does not need to wake an engineer at 3am if the service is healthy.

---

### Step 16: Execute Automation Candidates
**Who**: Senior SRE + Platform team

Automation is the highest-effort change in the campaign. Build auto-remediation only after deletions, merges, and downgrades are complete (and the page rate impact is measured). Automation that reduces pages is valuable, but it takes 3-5x longer to implement than deletion.

Minimum requirements before automating any response:
- The trigger condition and the remediation action are deterministic (same condition → same action, every time)
- The action is idempotent (running it twice produces the same result as running it once)
- A safety circuit exists (if the automated action fails or produces unexpected results, it stops and pages the engineer)

---

### Step 17: Rate-Limit Alert Changes
**Who**: Campaign Lead (enforces as a campaign rule)

Never change more than 20 alerts in a single week. Alert changes need a 7-day observation period before the next batch. The reason: if an alert deletion causes a missed incident, you need to be able to identify which deletion was responsible. Changing 50 alerts at once makes attribution impossible.

One week per batch, track page rate daily. Proceed to next batch only after confirming no missed incidents in the prior batch.

---

### Step 18: Log Every Change with Timestamp and Rationale
**Who**: PagerDuty/Opsgenie Admin

Maintain a change log:
- Alert name
- Change type (DELETE/MERGE/DOWNGRADE/AUTOMATE)
- Change date
- Rationale (from the classification step)
- Approver
- Page rate before change (7-day average)
- Page rate after change (7-day average post-change)

This log is the audit trail. If anyone questions a deletion, the log provides the rationale and approval chain.

---

## Phase 4 — Verification

### Step 19: Compare Page Rate Before and After
**Who**: Campaign Lead

Pull page rate for the 4 weeks before campaign start and for the 4 weeks after each batch of changes. Present as a trend line, not a single before/after number. Week-to-week variance in page rate is high; the trend over 4 weeks is more meaningful.

Target: 30-50% reduction in total pages/week. If the campaign achieves less than 20% reduction, review the deletion/downgrade decisions — either the campaign was too conservative or the high-volume alerts were all correctly classified as KEEP.

---

### Step 20: Verify No Incidents Were Missed
**Who**: Campaign Lead + Team Leads

For every incident that occurred during the campaign period, confirm:
- Was the incident detected by an active alert? (Yes → good)
- Was the incident detected by a customer complaint or manual discovery? (Yes → investigate whether a deleted alert would have caught it earlier)

If any incident was detected only by customer complaint during the campaign period, review the deletion list immediately. This is not necessarily a problem — the incident may have been undetectable by the alert that was deleted — but it requires explicit review before proceeding.

**Decision gate**: If a missed incident is plausibly attributable to an alert deletion, pause the campaign, restore the deleted alert, and re-evaluate the classification.

---

### Step 21: Recalculate Actionable Rate
**Who**: Campaign Lead

At campaign end, recalculate the actionable rate for the rotation. Target: improvement of 15-20 percentage points over the pre-campaign baseline. If actionable rate did not improve despite page count reduction, investigate: the remaining high-volume alerts may be driving both the page count and the low actionable rate.

---

### Step 22: Publish Results
**Who**: Campaign Lead + Director

Publish the campaign results to the team:
- Total alerts reviewed
- Total alerts deleted / downgraded / merged / automated
- Page rate before vs. after (per engineer per week)
- Actionable rate before vs. after
- Any incidents evaluated for missed detection (outcome of Step 20)

**PagerDuty query for finding stale/noisy alerts:**
Analytics → Noise Reduction Analysis → sort by "Highest Alert Volume" → filter for alerts with <30% incident creation rate (low actionable rate)

**Opsgenie query for finding stale alerts:**
Reports → Alert Reports → filter by "Resolved without being acknowledged" → sort by count descending
