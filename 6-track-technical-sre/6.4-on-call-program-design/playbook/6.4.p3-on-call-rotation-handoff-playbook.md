# Playbook: On-Call Rotation Handoff

## Purpose

Ensure the incoming on-call engineer starts their shift with complete situational awareness — knowing what is active, what is fragile, what is scheduled, and what to watch for. A handoff that takes 30 minutes and is done well prevents a 3am confusion call to the outgoing engineer.

**Timing**: Begin handoff 30 minutes before the rotation end time. Complete and send the handoff document before the rotation transitions.

**When to skip this playbook**: There is no incident, no open items, no degraded systems, and no scheduled events in the next 24 hours. In this case, send a one-line message: "Quiet shift. No open items. No upcoming events. You're up."

---

## Roles

| Role | Responsibility |
|------|----------------|
| **Outgoing Primary** | Completes the handoff document, walks the incoming primary through it |
| **Incoming Primary** | Reviews the handoff document, asks questions before the transition completes |
| **Secondary (Optional)** | Joins handoff call if there is an active incident or a complex situation |

---

## Section 1 — Open Incidents

### Step 1: List All Active Incidents
For each incident currently open in PagerDuty/Opsgenie:

```
Incident ID: [PD-XXXX or OG-XXXX]
Alert: [Exact alert name]
Service: [Service name]
Severity: [P0/P1/P2]
Current Status: [Acknowledged / Investigating / Monitoring / Waiting on external]
Summary: [2-3 sentences: what fired, what was found, current state]
Next action: [What the incoming on-call needs to do or watch for]
ETA to resolve: [If known; otherwise "unknown — monitor"]
```

### Step 2: List Acknowledged-But-Not-Resolved Alerts
These are alerts that have been silenced but not fixed — the outgoing engineer is "monitoring." The incoming engineer needs to know these exist and what the watch condition is.

```
Alert: [name]
Silenced until: [time]
Watch condition: [what would indicate it has worsened]
Resolution criteria: [what needs to happen before this can be closed]
```

### Step 3: List Incidents in Monitoring State
Incidents moved to "monitoring" are in a liminal state — believed to be resolved, but not confirmed. The incoming engineer should know the confirmation criteria.

```
Incident: [ID and name]
Monitoring since: [time]
Confirmation criteria: [specific metric or time window that confirms resolution]
Auto-resolve at: [if configured — the time the alert will auto-close if not re-triggered]
If it re-triggers: [brief escalation path or action]
```

### Step 4: Provide Context on Each Active Incident
For any incident that has been active for more than 2 hours, include: what has been tried, what has been ruled out, and any open hypotheses. The incoming engineer should not have to re-discover investigation dead ends.

## Section 2 — Deferred Work

### Step 5: Items Needing Follow-Up After Shift
Actions that were identified during the shift but deferred because they are not urgent enough to complete during on-call hours.

```
Item: [description]
Ticket: [Jira/ServiceNow ticket number]
Owner: [which team/engineer should own this]
Why deferred: [not urgent / requires change window / needs discussion]
Priority: [P1/P2/P3]
```

### Step 6: Runbooks That Need Updating
If the outgoing engineer discovered that a runbook was wrong, incomplete, or missing during the shift:

```
Alert: [name]
Runbook issue: [what was wrong or missing]
Suggested fix: [what should be added or corrected]
Ticket created: [Y/N — if yes, include ticket number]
```

### Step 7: Alerts Recommended for Review
If an alert fired and the outgoing engineer believes it should be downgraded, deleted, or modified:

```
Alert: [name]
Current severity: [P0/P1/P2]
Recommended action: [DOWNGRADE / DELETE / ADJUST THRESHOLD]
Rationale: [why — include what fired, what the actual impact was]
```

---

## Section 3 — Environment State

### Step 8: Infrastructure Changes During Shift
Any changes made to infrastructure during the shift that affect the incoming engineer's context.

```
Change: [what was done]
Service affected: [service name]
Reason: [why it was done — incident response, maintenance, etc.]
State now: [current state after the change]
Watch for: [any side effects or things to monitor post-change]
Change ticket: [if applicable]
```

### Step 9: Services in Degraded-But-Stable State
Services that are not fully healthy but are stable enough to not require immediate action.

```
Service: [name]
Degradation: [what is degraded and by how much]
Stable since: [time]
Watch threshold: [at what point this becomes an active incident again]
Owner notified: [Y/N — if yes, who and when]
```

### Step 10: Scheduled Maintenance in Next 24 Hours
Any planned work that the incoming engineer should be aware of.

```
Window: [start time – end time, with timezone]
Service: [what is being changed]
Change owner: [who to contact if something goes wrong]
Expected impact: [alerts that may fire, behavior that may appear degraded]
PagerDuty suppression: [configured Y/N — if N, incoming engineer should suppress manually]
```

### Step 11: Known Risk Events in Next 24 Hours
Scheduled deployments, traffic spike events (marketing campaigns, planned events), upstream vendor maintenance windows, or any other event that increases incident probability.

```
Event: [description]
Time: [when]
Risk: [what might go wrong]
Runbook: [link if one exists]
Owner: [who to contact if it goes sideways]
```

---

## Section 4 — Noteworthy Events

### Step 12: Unusual Observations (Even If Not Paged)
Things the outgoing engineer noticed in dashboards or logs that did not cross alert thresholds but were unusual. The incoming engineer should be aware even if no action is needed now.

### Step 13: Customer Escalations Received
Any customer-facing escalations that came through support channels during the shift — even if resolved — because they may resurface.

```
Customer/Channel: [who escalated]
Issue: [brief description]
Resolution: [what was done]
Ticket: [if one was created]
Follow-up needed: [Y/N — if yes, by whom]
```

### Step 14: Performance Anomalies in Dashboards
Any metric that is trending in a concerning direction but has not triggered an alert. The incoming engineer should watch these.

```
Metric: [name and dashboard link]
Normal range: [what is baseline]
Current value: [what the outgoing engineer observed]
Trend: [increasing / decreasing / volatile]
Alert threshold: [what value would trigger a page]
```

## Handoff Artifact

Complete the handoff using the following document structure (in Confluence, Notion, or a pinned Slack message in the team channel):

```markdown
## On-Call Handoff — [DATE] [ROTATION PERIOD START] → [ROTATION PERIOD END]
**Outgoing**: @[engineer]
**Incoming**: @[engineer]
**Secondary**: @[engineer]
**Shift end time**: [HH:MM TZ]

### Open Incidents
[From Steps 1-4]

### Deferred Work
[From Steps 5-7]

### Environment State
[From Steps 8-11]

### Noteworthy
[From Steps 12-14]

### Anything else
[Free text for anything that doesn't fit above]
```

## Slack Handoff Notification Format

Post to the team on-call channel (#oncall or #sre-handoffs) at the time of rotation:

```
:rotating_light: *On-Call Rotation Handoff*
*Outgoing*: @[name] — wrapping up shift
*Incoming*: @[name] — now primary on-call
*Secondary*: @[name]

*Shift summary*: [1-2 sentences: quiet shift / N incidents / notable event]

*Handoff doc*: [link to Confluence/Notion handoff artifact]

*One thing to watch*: [the single most important thing the incoming engineer should monitor]

Ping @[outgoing name] in the next 2 hours if you have questions — after that, they're off.
```

## Decision Gate: When to Call Instead of Document

If any of the following are true, the handoff must include a synchronous call (Slack huddle or phone), not just a written document:

- There is an active P0 or P1 incident at transition time
- A service is in a degraded state with no clear recovery path
- A deployment is in progress or scheduled within 2 hours of rotation change
- The outgoing engineer has a hypothesis about an emerging problem they have not fully investigated
- This is the incoming engineer's first on-call shift (shadow or first independent shift)

Written documentation alone is insufficient for these situations. The phone call provides the incoming engineer with the outgoing engineer's mental model — something a document cannot fully capture.
