## How to Fill In Each Column

| Column | Instructions |
|--------|-------------|
| **Alert Name** | Exact alert name as it appears in PagerDuty/Opsgenie — no paraphrasing |
| **Service Owner** | Team name and/or individual engineer who owns this alert |
| **Current Severity** | P0/P1/P2/P3 as currently configured |
| **Alert Type** | Symptom / Cause / Watchdog / Flapping / Stale (see ch03 for definitions) |
| **P90 Fire Rate** | Average pages per week over the 90-day audit window. Pull from PagerDuty analytics or Splunk. |
| **Actionable Rate %** | % of fires that required human action to resolve. If unknown, mark "unknown" — do not estimate. |
| **Last Fired** | Date of most recent page from this alert |
| **Last Actionable Fire** | Date of most recent page that required human action. "Never" if none in 90-day window. |
| **False Positive Rate %** | % of fires where no actual problem existed. Inverse of actionable rate for auto-resolving alerts. |
| **Recommended Action** | KEEP / DOWNGRADE / MERGE / AUTOMATE / DELETE |
| **Notes / Rationale** | Required for all non-KEEP recommendations. Must answer: "What would we miss if this alert were changed or gone?" |
| **Owner Confirmed** | Y/N — has the service owner reviewed and agreed with the recommendation? Required before implementation. |
| **Implemented Date** | Fill in after the change is executed in PagerDuty/Opsgenie |

## Classification Decision Guide

| Recommendation | When to Use |
|---------------|-------------|
| **KEEP** | Alert is symptom-based, actionable rate >50%, fires on real customer impact |
| **DOWNGRADE** | Alert fires on real signal but wrong severity; cause-based with <40% actionable rate; should be P3 (ticket) not P1 (page) |
| **MERGE** | Multiple alerts fire for the same underlying condition; consolidate into one rich alert |
| **AUTOMATE** | Known cause + known fix + deterministic + idempotent; human response is always the same scripted action |
| **DELETE** | Watchdog / stale / 0% actionable / monitoring a deprecated system |
