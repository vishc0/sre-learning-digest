# Chapter 5 — Runbook Design and Maintenance: Engineering the Response

## Why "Check the Logs" Is Not a Runbook

The most common runbook failure mode is instructions so generic they provide no real guidance. "Check the logs." "Restart the service if needed." "Escalate to the team if unresolved." These are not runbooks — they are notes that describe the shape of a response without actually engineering one.

A runbook is not a description of what a response looks like. It is the response, written in advance, so that the engineer executing it can do so without requiring institutional knowledge or a senior engineer on a phone call. The test: can a competent engineer who has never seen this service before follow this runbook to resolution? If not, the runbook is incomplete.

This is a high standard. It is also the right standard.

## Anatomy of a Complete Runbook

**Section 1 — Trigger**
What alert or condition activates this runbook? Include the exact alert name as it appears in PagerDuty — not a paraphrase, not a description, the exact string. Engineers arriving at a runbook page from a PagerDuty link need to confirm immediately that they have the right runbook.

Example:
```
Alert: notification-service-delivery-rate-low
PagerDuty Service: notification-platform-prod
Severity: P1
```

**Section 2 — Context**
What system is this? What does it do? What breaks when this alert fires? Write this for someone who has never worked on this service.

Example:
```
The notification service processes outbound SMS, push, and email for 25M messages/day.
When delivery rate drops below 95%, customers are not receiving time-sensitive alerts.
Downstream impact: customer support call volume spikes within 15 minutes of degradation.
```

**Section 3 — Triage Steps**
Ordered diagnostic steps with the exact commands and the expected output for a healthy system. This is the section where most runbooks fail. Vague steps waste time. Specific steps with expected outputs allow the engineer to confirm or eliminate a hypothesis in under 60 seconds.

Example (bad): "Check the delivery service logs."

Example (good):
```
Step 1: Check delivery service error rate
kubectl logs -n prod deployment/notification-delivery --since=10m | grep -c "ERROR"
Expected: < 50 in a 10-minute window
If > 50: proceed to Step 2
If < 50: delivery service is healthy — proceed to Step 5 (upstream triage)

Step 2: Identify the error class
kubectl logs -n prod deployment/notification-delivery --since=10m | grep "ERROR" | awk '{print $7}' | sort | uniq -c | sort -rn | head -10
Expected outputs and meaning:
- "connection refused" → downstream carrier API unavailable (proceed to Step 3)
- "timeout" → carrier API latency issue (proceed to Step 4)
- "queue full" → RabbitMQ backpressure (proceed to Step 7)
```

**Section 4 — Resolution Paths**
Decision tree based on what you found in triage. Each branch leads to either a resolution action or an escalation decision. Resolution paths should be numbered and cross-referenced with triage step findings.

**Section 5 — Expected Output (Resolution State)**
What does resolved look like? Not "the alert resolves" — that is circular. What specific metric, log line, or dashboard panel confirms resolution?

Example:
```
Resolution confirmed when:
- Delivery rate metric returns above 95% in Splunk dashboard "Notification Platform - Production"
- No new ERROR entries in delivery service logs for 5 consecutive minutes
- PagerDuty alert auto-resolves (configured for 5 minutes at threshold)
```

**Section 6 — Escalation**
When should the on-call engineer escalate, to whom, and with what information?

Example:
```
Escalate to secondary if:
- Triage steps complete and root cause is not identified within 20 minutes
- Root cause identified but requires database access or carrier account changes

When escalating, provide:
- Current delivery rate (from Splunk)
- Error class identified in Step 2
- Steps already taken
- Your hypothesis
```

**Section 7 — Related Runbooks**
Links to upstream, downstream, and laterally-related runbooks. Engineers in mid-incident should not need to search for adjacent runbooks.

## Runbook Freshness Indicators

Every runbook should carry three metadata fields:

- **Last tested**: date when this runbook was walked through end-to-end (during a game day, a real incident, or a tabletop). If this date is more than 90 days ago, treat the runbook as potentially stale.
- **Last used in incident**: the most recent incident number where this runbook was the response guide. If never used, mark it as untested in production.
- **Review owner**: the engineer or team responsible for keeping this runbook current. "Everyone" is not an owner.

Automated staleness flagging: run a weekly check against your runbook system (Confluence, GitHub, Notion) for pages with `last_updated` older than 90 days. Post flagged runbooks to a Slack channel with the review owner tagged. This converts runbook maintenance from an honor-based system to a visible queue.

## Runbook Anti-Patterns with Rewrites

**Anti-pattern**: "Check if the service is down."
**Why it fails**: No defined method. No expected output. Engineer has to figure out *how* to check.
**Rewrite**: "Run `kubectl get pods -n prod -l app=notification-service`. If any pods show status other than Running/1/1, the service is degraded. Count the degraded pods and proceed to Step 3."

**Anti-pattern**: "Restart the service if the issue persists."
**Why it fails**: No context on what "the issue" is. No pre-restart check to prevent data loss. No post-restart verification.
**Rewrite**: "Before restarting: confirm no active in-flight message batches by checking `kubectl exec -n prod deployment/notification-service -- curl localhost:8080/health/inflight`. If inflight count > 0, wait 60 seconds and recheck. Restart with: `kubectl rollout restart deployment/notification-service -n prod`. Post-restart: verify pods reach Running state within 3 minutes with `kubectl rollout status deployment/notification-service -n prod`."

**Anti-pattern**: "Escalate to the team if unresolved."
**Why it fails**: "The team" is not a contact. This does not define who, how, or what information to provide.
**Rewrite**: "Escalate to the platform-SRE-secondary via PagerDuty secondary escalation. Send the following in #incident-active Slack channel: [incident number], [current delivery rate], [error class from Step 2], [steps taken], [your current hypothesis]. If secondary does not acknowledge in 10 minutes, escalate to on-call manager."

## Director Framing: Runbooks as Organizational Resilience

A runbook is a knowledge transfer mechanism. The question for a Director to ask is not "do we have runbooks?" — most teams have something they call runbooks. The question is: "if your most senior on-call engineer left tomorrow, would the runbooks enable the next engineer to respond to the top-20 alerts without a phone call?"

That is your resilience test. Runbook completeness — measured against that specific standard — is a team durability metric that belongs in the same category as documentation debt and bus-factor analysis.

Quarterly: assign each SRE one runbook to audit and update as part of their sprint work. This distributes the maintenance load, keeps engineers engaged with runbook quality, and ensures that the engineers who use the runbooks are also the ones improving them.
