# Chapter 7 — On-Call Tools and Automation: The Infrastructure of Response

## Tool Architecture Overview

On-call tooling is a pipeline, not a single product. Understanding the pipeline architecture helps you evaluate tools at the right level — not "is PagerDuty better than Opsgenie?" but "where in this pipeline does each tool operate, and what is the right choice for each layer?"

```
Alert Source          Routing Layer          Notification          Incident Management
─────────────         ─────────────          ────────────          ───────────────────
Splunk        ──┐
Prometheus    ──┼──→  PagerDuty   ──────→   Phone/SMS   ──────→   Slack incident channel
CloudWatch    ──┤      Opsgenie              Push notification      Jira incident ticket
Datadog       ──┘      Grafana OnCall        Email                 Incident.io / FireHydrant
                                             Slack DM              Postmortem document
```

Each layer has a distinct function. Alert sources generate signals. The routing layer receives those signals, applies escalation policy, and determines who gets notified, through what channel, at what priority. The notification layer reaches the human. Incident management gives the response team a coordination surface.

Organizations that buy a single tool expecting it to do all of these well are usually disappointed. The routing layer tools (PagerDuty, Opsgenie) are best-in-class for routing. They are adequate for incident management but not exceptional. Dedicated incident management platforms (Incident.io, FireHydrant) add structure that standalone routing tools lack.

## PagerDuty vs Opsgenie: When to Choose Which

The detailed comparison is in the reference file. At a design-decision level, the key factors are:

**PagerDuty** is the right choice when:
- You need sophisticated, multi-condition routing rules (route by time-of-day AND severity AND service AND on-call engineer's timezone)
- You have complex escalation policies with many tiers and conditional logic
- You are a large enterprise with compliance and SSO requirements
- You are already paying for it and the migration cost exceeds the marginal benefit of switching

**Opsgenie** is the right choice when:
- Your organization is deeply invested in the Atlassian ecosystem (Jira, Confluence) — the integrations are native and require less configuration
- You want more flexible pricing at smaller scale
- You are starting from scratch and the Atlassian investment is already established

**Grafana OnCall** is right when:
- You are already running Grafana for observability and want native alert integration
- Cost is a primary constraint and open-source is acceptable
- Your team has capacity to self-host and configure

The most important decision rule: do not switch routing tools without a concrete, measurable gap in your current tool's capability. Migrations are expensive, disruptive, and rarely deliver the expected improvement on their own because the problem is almost never the tool.

## Alert Routing Design

Routing design is where most organizations underinvest. The default: one PagerDuty service, one escalation policy, everything goes to the same rotation. This works at small scale and becomes a problem as the service portfolio grows.

Mature routing design uses service-level routing with service-level escalation policies:

```
EKS-Platform-Prod service → Platform SRE rotation → platform escalation policy
Notification-Service-Prod service → App SRE rotation → application escalation policy
Data-Pipeline-Prod service → Data SRE rotation → data escalation policy
```

Each service has its own on-call team, its own escalation path, and its own contact methods. A database incident does not route to the same engineers as an API gateway latency alert.

Within each service's routing rules, apply severity-based routing:
- P0: phone call + push notification, immediate escalation timeout (10 min)
- P1: push notification + SMS, 15-minute timeout
- P2: push notification only, 30-minute timeout
- P3: low-priority notification, no phone, business hours only

## Automatic Runbook Linking

Every alert that reaches PagerDuty should include a direct URL to the runbook in the alert body. This is a configuration change at the alert source — in Splunk, add a `runbook_url` field to the alert action; in Prometheus, add a `runbook_url` annotation to the alert rule; in CloudWatch, include the URL in the SNS message template.

The operational impact: engineers arriving at a PagerDuty incident can open the runbook in one click rather than searching through Confluence. Studies from PagerDuty's State of Digital Operations (2023) show that automatic runbook linking reduces MTTA by 30-40% for alerts where runbooks exist. The mechanism is straightforward — the engineer starts the right response faster because they are not spending 5 minutes finding the right document.

For a T-Mobile notification platform processing 25M messages/day, where a P0 incident can affect thousands of customers per minute, a 5-minute reduction in time-to-start-triage has direct customer experience impact.

## Incident Auto-Declaration

When a P0 or P1 alert fires, the first 5 minutes of the incident response should not be spent setting up the response environment. That setup should happen automatically.

Auto-declaration workflow:
1. P0/P1 alert fires in Splunk → webhook to PagerDuty
2. PagerDuty triggers incident → webhook to Slack bot
3. Slack bot creates `#inc-[timestamp]-[service]` channel
4. Slack bot posts: incident title, severity, runbook link, who is on-call (primary and secondary)
5. Slack bot sends DM to on-call manager: "P0 declared: [title]. Channel: #inc-..."
6. PagerDuty incident links to Slack channel

The engineer arrives in a Slack channel that already has the context. The manager is already aware. The runbook is already linked. The incident has a coordination surface before the first triage step begins.

## Auto-Remediation Patterns

Auto-remediation is the highest-leverage on-call investment for reducing engineer burden on repetitive, known-pattern incidents. The key design principle: only automate actions that are safe and idempotent.

**Safe to automate:**
- Kubernetes pod restart after OOMKill for stateless services
- Cache invalidation for known stale-data alerts (with a rate limit — invalidate at most N times/hour)
- Scaling up a deployment when CPU/memory thresholds are crossed (scale up, not down — down requires human judgment)
- Sending a synthetic probe request to confirm service availability before paging

**Not safe to automate:**
- Database failovers — the primary/replica state transition has data integrity implications; a human should confirm
- Network configuration changes — blast radius is too large for automated action
- Any action that is difficult or impossible to reverse
- Any action that affects a downstream system without a coordination protocol with that system's team

The rule: if you would hesitate to let a new engineer do it at 3am without supervision, do not automate it.

## Director Framing: The Tool Fallacy

Directors who have inherited broken on-call programs are often presented with a tool change as the solution. "If we switch from PagerDuty to Opsgenie, the engineers will be less burned out." This is the tool fallacy — the belief that the right software configuration will fix a process and culture problem.

Tools enforce policy. They cannot create it. Buying a new routing tool and importing your existing alert catalog into it produces the same noisy alert catalog, now in a different tool, at migration cost. The engineers are still burned out. Nothing changed except the vendor invoice.

The correct sequence: design the policy first (rotation, escalation, alert standards, runbook requirements), then evaluate whether your current tooling can enforce that policy, then consider switching only if there is a specific, concrete gap that the new tool addresses and the current one cannot.

This sequence is also the answer to the interview question: "How would you approach selecting an incident management tool?" — the answer that distinguishes a Director from a Senior Engineer is leading with policy design, not tool evaluation.
