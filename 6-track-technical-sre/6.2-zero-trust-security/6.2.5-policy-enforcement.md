# Chapter 5: Policy Enforcement

## Why Human-Reviewed YAML Policies at Scale Fail

Imagine your team has 200 services across 15 namespaces. Each service has a network policy, an authorization policy, and potentially a pod security policy. That is 600+ policy documents. As services evolve, callers change, new endpoints are added, and teams create new namespaces. Every change requires a policy update.

In a manual review model, a policy change works like this: an engineer writes a YAML file, submits a PR, a senior engineer reviews it, it gets merged, someone manually applies it to the cluster. This process takes anywhere from hours to days, depending on the review queue. Under time pressure — a release, an incident, a deadline — shortcuts happen. Policies are written permissively to "unblock" something. Reviews are rubber-stamped. PERMISSIVE policies that were supposed to be temporary become permanent.

After two years of this, your policy library is a mix of well-intentioned restrictions and accumulated exceptions that nobody fully understands. The policy is a security liability masquerading as a security control.

Policy-as-code applies software engineering discipline to policy authorship: policies are written in a testable language, stored in version control, reviewed via PR with automated checks (linting, testing), deployed through the same CI/CD pipeline as application code, and audited automatically. Policy debt is managed the same way technical debt is managed — by making it visible and allocating time to address it.

## OPA: The Policy Engine

OPA (Open Policy Agent) is a CNCF graduated project that provides a general-purpose policy engine. Rather than embedding policy logic in application code or infrastructure configuration, you write policies in Rego and evaluate them centrally.

Rego is a declarative language designed specifically for policy decisions. It is not imperative (no loops, no mutation). It evaluates inputs against a set of rules and returns a decision. The mental model: you define what "allowed" looks like, and OPA evaluates whether any given input matches that definition.

A simple Rego rule — only allow pulling images from approved registries:
```rego
package kubernetes.admission

deny[msg] {
    input.request.kind.kind == "Pod"
    image := input.request.object.spec.containers[_].image
    not startswith(image, "registry.tmobile.com/")
    msg := sprintf("image '%v' is not from an approved registry", [image])
}
```

Key properties of Rego:
- **Declarative**: you describe the policy, not the evaluation algorithm
- **Testable**: OPA has a built-in test framework (`opa test`)
- **Auditable**: every policy decision can be logged with full input/output context
- **Portable**: the same OPA engine runs in Kubernetes admission, CI pipelines, and microservice authorization

## Gatekeeper: OPA as a Kubernetes Admission Controller

OPA Gatekeeper is the Kubernetes-native deployment of OPA. It runs as a validating admission controller — every resource creation and modification in Kubernetes passes through Gatekeeper before it is persisted to etcd. Gatekeeper can deny non-compliant resources.

Gatekeeper introduces two CRDs:

**ConstraintTemplate**: defines a new policy type (a custom resource) and the Rego logic that evaluates it.

```yaml
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: requiredlabels
spec:
  crd:
    spec:
      names:
        kind: RequiredLabels
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package requiredlabels
        violation[{"msg": msg}] {
          provided := {label | input.review.object.metadata.labels[label]}
          required := {label | label := input.parameters.labels[_]}
          missing := required - provided
          count(missing) > 0
          msg := sprintf("missing required labels: %v", [missing])
        }
```

**Constraint**: an instance of a ConstraintTemplate, scoped to specific resources with specific parameters.

```yaml
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: RequiredLabels
metadata:
  name: require-team-label
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Namespace"]
  parameters:
    labels: ["team", "cost-center", "environment"]
```

This combination means policy types are extensible — you define the policy shape once (ConstraintTemplate) and instantiate it many times with different parameters (Constraint). You can enforce "require specific labels" once and apply it to different namespaces with different required label sets.

## Istio AuthorizationPolicy: L7 Enforcement at the Service Mesh Layer

Istio AuthorizationPolicy enforces access control at the application (L7) layer, evaluated by the Envoy sidecar on every request. It acts after mTLS verification — the connection is already authenticated; AuthorizationPolicy determines whether the authenticated caller is authorized to make the specific request.

AuthorizationPolicy supports:

- **Source principal**: the SPIFFE ID of the calling workload (`principals: ["cluster.local/ns/payments/sa/checkout"]`)
- **Namespace**: all workloads in a given namespace
- **JWT claims**: fields from a JWT token in the request (`requestPrincipals`, `when` conditions on claim values)
- **Request attributes**: HTTP method, path, host, headers

A deny-all with explicit allow:
```yaml
# Default deny for the payments namespace
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: payments
spec: {}  # empty spec = deny all
---
# Allow checkout service to call POST /payments/charge
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-checkout-charge
  namespace: payments
spec:
  action: ALLOW
  rules:
  - from:
    - source:
        principals: ["cluster.local/ns/ecommerce/sa/checkout-service"]
    to:
    - operation:
        methods: ["POST"]
        paths: ["/payments/charge"]
```

This policy enforces at the service mesh layer that only the `checkout-service` service account in the `ecommerce` namespace can call `POST /payments/charge`. Any other workload, even if it can route to the payments service at the network layer, is denied at the proxy.

## Attribute-Based Access Control (ABAC): The Policy Model

The policy evaluation model underlying both OPA and Istio AuthorizationPolicy is ABAC — Attribute-Based Access Control. Instead of enumerating individual allowed pairs (role-based, which doesn't scale in dynamic environments), ABAC evaluates a combination of:

- **Identity attributes**: who is making the request (SPIFFE ID, namespace, service account, JWT claims)
- **Resource attributes**: what is being accessed (service, path, method, data classification)
- **Environment attributes**: when and where (time of day, source cluster, request rate)

The decision is: does this combination of attributes satisfy the policy? This model scales to dynamic environments because you write policies about attribute combinations, not about specific workload pairs. When a new service is added, it inherits policies based on its attributes (namespace, labels, service account) — you do not need to update every policy to add the new service.

## The Policy Pipeline

Policy-as-code requires a delivery pipeline, just like application code:

```
[Author: Rego/YAML]
       |
       v
[Lint: opa fmt, opa check]
       |
       v
[Unit Test: opa test ./policies/...]
       |
       v
[PR Review: peer review + policy-bot checks]
       |
       v
[Integration Test: deploy to staging, run compliance scan]
       |
       v
[Deploy: GitOps (Flux/ArgoCD) applies to cluster]
       |
       v
[Audit: OPA decision logs → Splunk]
```

The audit step is non-negotiable in Zero Trust. Every allow and deny decision from OPA and Istio AuthorizationPolicy should be logged with the full context: who, what, when, which policy, what decision. This is how you prove the policy is working — and how you detect policy misconfiguration before it becomes a security incident.

## The Director Frame: Policy Debt Is the New Technical Debt

The parallel to technical debt is exact: both accumulate silently, both become harder to remediate over time, and both require sustained engineering investment to control.

Unmaintained policy libraries exhibit the same failure modes as unmaintained code:
- Policies that reference workloads that no longer exist (dead rules that create confusion)
- Permissive exceptions that were "temporary" three years ago
- Policies written by engineers who left the team, with no documentation of intent
- Duplicate policies with conflicting logic that happen to result in the correct behavior by coincidence

The Director responsibility is to treat policy maintenance as a first-class engineering discipline:
- Policy changes go through PR review, the same as code
- Policies have owners — a team is accountable for each policy in the library
- Dead rules are pruned on a quarterly cycle
- Policy coverage is a metric on the engineering dashboard (what percentage of workloads have explicit deny-all + allow policies vs. default-permit?)

Budget engineering time for policy maintenance just as you would for dependency upgrades or toil reduction. An unmaintained policy library is not a security asset — it is a security liability that creates a false sense of control.
