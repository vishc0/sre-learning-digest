# Chapter 3: Network Zero Trust

## Why Flat Networks Are a Lateral Movement Highway

Think of a flat network like a single open floor plan in an office building with no interior doors. When the front door is breached, every room is accessible. The attacker does not need to pick multiple locks — they walk room to room freely. This is exactly how lateral movement works in a flat network: once an attacker has a foothold in one workload, the absence of internal segmentation means they can communicate to any other workload on the same network.

The Mandiant M-Trends report consistently shows that the median dwell time in a flat network (the gap between initial compromise and detection) is measured in weeks. Lateral movement happens in that window. The attacker is not racing against your perimeter — they are walking through a building with no interior doors.

Microsegmentation is the act of adding those interior doors. It restricts what can talk to what, based on workload identity and policy, so that a breach in one service cannot automatically become a breach in every service. The blast radius of any given compromise is bounded by the segmentation policy.

The operational analogy from a middleware perspective: in a well-designed RabbitMQ topology, you use separate virtual hosts (vhosts) for different application domains. A producer in one vhost cannot publish to queues in another vhost by default. You explicitly configure which applications can communicate. Network microsegmentation applies that same principle to IP traffic.

## East-West Traffic: The Forgotten Attack Surface

Most traditional network security focuses on north-south traffic: traffic entering or leaving the network from the internet (ingress/egress at the perimeter). Firewalls, WAFs, and IDS/IPS systems are predominantly oriented toward this boundary.

East-west traffic is service-to-service communication inside the cluster. In a microservices architecture, this is the majority of traffic. A single user request to a web frontend might generate 10-30 internal service calls before a response is returned. All of that traffic moves east-west.

Traditional perimeter security tools see essentially none of it. The traffic never crosses the perimeter. It is completely invisible to the firewall. This means:

- A compromised service can call any other internal service it can route to
- Credential theft inside a pod gives the attacker the same service-to-service access the pod had
- Supply chain attacks (malicious code injected into a dependency) have unconstrained lateral movement within the cluster

The Zero Trust response to east-west traffic is layered:
1. **Network layer (L3/L4)**: Kubernetes NetworkPolicy restricts which pods can initiate TCP connections to which other pods
2. **Identity layer (mTLS)**: Service mesh enforces that connections are mutually authenticated with certificates, not just routable
3. **Application layer (L7)**: Istio AuthorizationPolicy enforces which specific HTTP methods and paths are allowed, based on workload identity and request attributes

Each layer catches what the layer below cannot see. Network policy cannot inspect HTTP headers. mTLS cannot enforce path-level access control. The layers are not redundant — they are complementary.

## Kubernetes NetworkPolicy: Capability and Limits

NetworkPolicy is Kubernetes' native mechanism for L3/L4 network segmentation. A NetworkPolicy resource selects pods via label selectors and defines which ingress and egress traffic is permitted.

The default behavior without any NetworkPolicy: all pods can communicate with all other pods, regardless of namespace. This is the flat network problem in its purest form.

A default-deny policy for a namespace:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: payments
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
```

This blocks all ingress and egress for every pod in the `payments` namespace. You then add explicit allow policies on top.

**What NetworkPolicy cannot do:**

- **No L7 inspection**: NetworkPolicy operates at L3/L4. It can allow TCP port 8080 from namespace A to namespace B, but it cannot distinguish between `GET /health` and `DELETE /accounts`. L7 policy requires a service mesh.
- **No DNS-based egress**: NetworkPolicy works with IP addresses and pod selectors. You cannot write a rule like "allow egress to api.external-partner.com." You can only target IP blocks or Kubernetes-native selectors. Restricting DNS-named external endpoints requires a mesh egress gateway or a DNS-aware policy engine (Cilium's FQDN policy is an exception here).
- **No cross-cluster scope**: NetworkPolicy is cluster-scoped. A multi-cluster deployment requires cluster-level federation for consistent policy.
- **CNI dependency**: NetworkPolicy is only enforced if the CNI plugin supports it. Flannel does not. Calico, Cilium, and Weave do. Verify your CNI before relying on NetworkPolicy enforcement.

## Service Mesh Fills the Gap

Service mesh addresses the gaps NetworkPolicy leaves open. By deploying a sidecar proxy (Envoy in Istio, a lightweight proxy in Linkerd) alongside every workload, the mesh intercepts all traffic and applies:

- **Mutual TLS**: every connection is authenticated with certificates, both directions
- **L7 policy**: AuthorizationPolicy can inspect HTTP method, path, JWT claims, and source SPIFFE ID
- **Traffic telemetry**: every request is instrumented — latency, response code, source, destination, upstream cluster
- **Egress control**: mesh egress gateways can enforce that traffic to external services goes through a controlled exit point

The mesh does not replace NetworkPolicy. They operate in combination:
- NetworkPolicy: "only pods in namespace `api-gateway` can open TCP connections to pods in namespace `payments`"
- Istio AuthorizationPolicy: "of those TCP connections, only the `checkout-service` service account can call `POST /payments/charge`"

Network policy at L3/L4 reduces the attack surface. Mesh policy at L7 enforces the specific authorized operations within that reduced surface.

## ZTNA: Replacing VPN with Per-Session, Identity-Aware Access

Traditional VPN is the human-facing version of the castle-and-moat problem. Once a user authenticates to VPN, they typically receive network-level access to a wide range of internal resources — sometimes the entire internal network. VPN grants location access, not application access.

Zero Trust Network Access (ZTNA) replaces VPN with per-session, per-application access grants:

| Dimension | VPN | ZTNA |
|-----------|-----|------|
| Trust scope | Network subnet after auth | Per-application, per-session |
| Identity factors | Username + password (+ MFA) | Identity + device posture + context |
| Revocation | Session-level (disconnect the tunnel) | Per-request (policy evaluated continuously) |
| Visibility | Opaque tunnel traffic | Full request logging |
| Lateral movement | Possible — network access is broad | Eliminated — no network-level access granted |

In a ZTNA model, when an engineer needs to access a production service for incident response, they receive a time-bounded, identity-bound, single-application access grant — not VPN access to the production network. Their device posture is verified (is it enrolled in MDM? Is disk encryption on?). The access is logged with their identity. When the session ends or the time bound expires, the access is revoked.

For cloud-native platforms, ZTNA for human access combines with mTLS + SPIFFE for workload access. The result is a platform where no actor — human or machine — has implicit network-level trust. Every access is explicit, authenticated, time-bounded, and logged.

## The Director Frame: Policy at Three Layers

The Director-level answer to "how do you approach network security in Kubernetes" is not "we use NetworkPolicy." It is:

> "We enforce policy at three layers. The network layer (NetworkPolicy, or Cilium for FQDN-based egress) limits what can reach what at L3/L4. The identity and transport layer (mTLS via service mesh) ensures every connection is mutually authenticated — no workload can impersonate another. The application layer (Istio AuthorizationPolicy or OPA) enforces specific authorized operations based on workload identity and request attributes. These layers are not redundant — each catches what the layer below cannot see. Network policy is the floor, not the ceiling."

That framing demonstrates architectural depth, not just tool familiarity. The follow-up question an interviewer might ask: "what did you find hardest about rolling out that approach?" The honest answer (and the right one) is: east-west policy at the application layer requires you to enumerate all legitimate service-to-service communication patterns, which in a brownfield environment means you're discovering undocumented dependencies you didn't know existed. That discovery process is more valuable than the policy enforcement itself — it produces an accurate service dependency map for the first time.
