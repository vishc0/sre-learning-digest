# Chapter 8: Director Decision Framework

## The CISA Zero Trust Maturity Model v2.0

The CISA ZT Maturity Model (April 2023) is the reference framework federal contractors and large enterprises use to assess and communicate ZT progress. Knowing it is a signal of architectural literacy. Using it is how you run a ZT program with measurable outcomes.

The model defines five pillars and three maturity levels:

| Pillar | Traditional | Advanced | Optimal |
|--------|-------------|----------|---------|
| **Identity** | Manual user lifecycle; some MFA | Automated provisioning; MFA everywhere; risk-based auth | Continuous validation; behavioral analytics; just-in-time access; phishing-resistant MFA |
| **Devices** | Basic asset inventory; static compliance | MDM enrollment; automated remediation; device health in access decisions | Continuous compliance validation; real-time posture in every access decision |
| **Networks** | Macro-segmentation; perimeter firewall | Microsegmentation; encrypted east-west; DNS filtering | Software-defined perimeter; traffic inspection at every hop; ZTNA for all access |
| **Applications** | Manual access reviews; monolithic apps | Per-application access policies; ABAC; CI/CD security gates | Continuous authorization per-request; ML-driven anomaly detection; integrated threat intelligence |
| **Data** | Static data classification; DLP at perimeter | Automated classification; data-level access controls; encryption at rest | Automated tagging; real-time access decisions based on data sensitivity; continuous DLP |

For Director interviews: you are expected to know which level your current organization operates at for each pillar, what the gaps are, and what the roadmap looks like to advance one level per pillar over an 18-24 month horizon.

## Phased Rollout Strategy

The order in which you phase a ZT implementation matters more than the components you choose. Each phase builds the operational foundation and the trust infrastructure that the next phase depends on.

### Phase 1: Network Segmentation

**Goal**: reduce lateral movement blast radius. **Duration**: 3-6 months.

- Deploy Kubernetes NetworkPolicy with default-deny-all in highest-risk namespaces
- Map all legitimate service-to-service communication (this is harder than it sounds — most organizations discover undocumented dependencies here)
- Deploy service mesh (Istio or Linkerd) in PERMISSIVE mode — no enforcement yet, but traffic becomes visible and measurable
- Establish the observability pipeline: Istio access logs to Splunk, baseline access patterns documented

Why network first: you cannot safely enable identity-layer enforcement (STRICT mTLS, AuthorizationPolicy) until you know what the legitimate communication patterns are. The network phase is discovery as much as enforcement.

### Phase 2: Identity and mTLS

**Goal**: cryptographic identity for every workload; all internal traffic mutually authenticated. **Duration**: 4-8 months.

- Deploy SPIRE (or use Istio's built-in CA) for workload certificate issuance
- Migrate highest-risk namespaces from PERMISSIVE to STRICT mTLS one namespace at a time
- Deploy Vault Kubernetes auth for all production services — eliminate static credentials in highest-risk areas
- Enable CyberArk just-in-time access for all privileged production access (replace standing admin accounts)
- Cert expiry alerting in Splunk — mandatory before any namespace goes STRICT

Why identity second: mTLS enforcement requires that every legitimate caller is meshed and has a valid certificate before you flip to STRICT. The network phase gave you the caller inventory you need.

### Phase 3: Application Layer Policy

**Goal**: enforce least privilege at the request level, not just the connection level. **Duration**: 4-6 months.

- Deploy OPA Gatekeeper for Kubernetes admission control — enforce security policies on all new resources
- Deploy Istio AuthorizationPolicy with deny-all + explicit allow in all production namespaces
- Complete dynamic secrets migration for all production database credentials
- ABAC-based access control for all human access to production (replacing RBAC where feasible)
- Policy-as-code pipeline fully operational: all policy changes go through PR review and automated testing

### Phase 4: Data Plane

**Goal**: data classification and access control at the data layer. **Duration**: ongoing.

- Data classification for all data stores — sensitivity labels drive access policy
- Data-level encryption with customer-managed keys (BYOK in AWS KMS) for highest-sensitivity data
- DLP controls at egress gateways
- Continuous access review automation

## Build vs. Buy Decision Framework

The ZT implementation stack requires decisions across four domains. This framework structures those decisions.

### Option A: DIY (Istio + SPIRE + OPA + Vault + CyberArk)

**Profile**: maximum architectural control; requires deep operational expertise.

| Component | Tool | Operational Overhead |
|-----------|------|---------------------|
| Service mesh + mTLS | Istio | High (control plane upgrades, cert management, sidecar injection) |
| Workload identity | SPIRE | Medium (DaemonSet management, attestation plugin config) |
| Policy enforcement | OPA/Gatekeeper | Medium (Rego authoring, policy pipeline) |
| Machine secrets | Vault | Medium (cluster operations, lease management, audit log) |
| Human privileged access | CyberArk | Medium (PAM operations, safe management) |

Best for: organizations with existing SRE capability, teams already operating these tools, regulated environments requiring component-level control and auditability.

### Option B: Managed Service Mesh

AWS App Mesh, Google Traffic Director, Azure Service Fabric. Cloud provider manages the control plane; you configure the data plane.

**Advantage**: eliminates Istio control plane operations (one of the highest operational overhead items in Option A).
**Risk**: cloud lock-in; policy model is less flexible than Istio; SPIFFE integration varies.

Best for: organizations with limited mesh operations experience, single-cloud deployments, teams that prioritize managed infrastructure.

### Option C: Full ZT Platform

Zscaler Private Access, Palo Alto Prisma Access, Cloudflare Access. Comprehensive SaaS-based ZT for human access and some workload patterns.

**Advantage**: fastest time to value for human access ZTNA; SOC integration; vendor-managed infrastructure.
**Risk**: cost (per-user licensing at scale is significant); limited integration with on-premise or custom workloads; data sovereignty concerns for regulated industries.

Best for: organizations prioritizing human access ZTNA before machine ZT; hybrid environments with significant SaaS workloads; organizations without deep security engineering resources.

**The Director's decision rule**: choose the stack that your team can operate reliably, not the stack with the most features. An Istio deployment that the team cannot troubleshoot is worse than a simpler mesh that they can. Be honest about your team's current expertise and the investment required to build it.

## Common Failure Modes

These are the patterns that cause ZT programs to stall or fail. Know them so you can prevent them.

**Big-bang vs. phased**: Attempting to enable ZT controls cluster-wide simultaneously. Invariably, something breaks at scale that worked in the pilot. Always phase by namespace, team, or service tier.

**PERMISSIVE mode that never becomes STRICT**: Service mesh deployed in PERMISSIVE mode for the migration period — but the migration period never ends. Two years later, PERMISSIVE is still the mode because "we can't risk breaking anything." The enforcement never materializes. This is the most common ZT implementation failure.

**Cert rotation not automated**: Manual cert rotation is an incident waiting to happen. Certs have fixed expiry dates. Humans miss deadlines. Automate rotation from day one and test the automation — do not assume it works because SPIRE says it will rotate automatically.

**Observability bolted on last**: Teams deploy controls first, observability later. By the time observability is in place, there have already been unexplained incidents that would have been immediately obvious with logging. Observability is Phase 1, not Phase 4.

**Policy sprawl without ownership**: After 18 months, nobody knows which policies are still relevant. Policy review becomes fearful ("what if we break something?") rather than disciplined. Assign owners to every policy. Review quarterly.

## Director Talking Points: ZT ROI and CFO Framing

ZT is a security investment, and like all investments, it requires business justification. The CFO is not interested in mTLS — the CFO is interested in risk reduction and cost.

**The blast radius reduction metric**: "Our current estate has 300 services on a microsegmented network with STRICT mTLS. If an attacker compromises any one service, they can reach at most the 3-5 services that service is explicitly authorized to communicate with. Before ZT, a compromise of any one service gave them access to all 300. The blast radius is bounded at the service level, not the cluster level."

**Regulatory compliance as ZT ROI**: "PCI-DSS v4.0 requirement 7 (restrict access to cardholder data) and requirement 10 (log and monitor all access) are directly addressed by our ZT investment. Without ZT, we would need to build separate compliance controls for each requirement. With ZT, the controls are architectural — they apply consistently across all in-scope systems."

**Insurance and audit cost reduction**: Organizations with demonstrable ZT controls have seen cyber insurance premiums decrease 15-30%. ZT adoption is increasingly a factor in cyber insurance underwriting. An external audit of a ZT implementation is typically faster and less expensive than an audit of a perimeter-based environment because the controls are automated and auditable.

**The opportunity cost of not doing it**: "The median cost of a lateral movement incident that reaches a payment system is $4.2M (IBM Cost of a Data Breach 2024). The entire first phase of our ZT program — network segmentation and mTLS — costs $800K in engineering labor and $120K in tooling. The math is favorable if we prevent one meaningful incident."

## The Director Frame: Architecture, Investment, and Measurement

Your role in a ZT program is not to implement every component. It is three things:

**1. Define the architecture**: which components, which order, which trade-offs. The architecture must be coherent — the pieces must fit together and build on each other. No component islands.

**2. Phase the investment**: each phase must deliver measurable risk reduction, not just tooling deployment. The CISA maturity model gives you the measurement vocabulary. Use it in every quarterly business review.

**3. Sustain the program**: ZT is not a project. It is a capability. Policies need maintenance. Cert automation needs testing. Coverage needs measurement. The program needs a product owner — someone accountable for ZT maturity as a platform capability, not a one-time initiative.

The interview question that separates Directors from architects: "Tell me about a ZT initiative that didn't go as planned and what you learned." The answer should include: what the plan was, what the specific problem was (not a vague "it was harder than expected"), what you changed, and what you would do differently. That answer demonstrates that you have operated the program, not just designed it.
