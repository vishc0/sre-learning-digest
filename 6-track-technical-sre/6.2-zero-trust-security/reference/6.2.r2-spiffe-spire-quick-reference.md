# SPIFFE/SPIRE Quick Reference

## SPIFFE ID Format

```
spiffe://<trust-domain>/<workload-path>
```

**Components:**
- `spiffe://` — required URI scheme, always lowercase
- `<trust-domain>` — organizational boundary, analogous to a DNS domain. Usually matches your cluster or environment domain.
- `<workload-path>` — hierarchical path identifying the specific workload. Convention: `/ns/<namespace>/sa/<service-account>`

**T-Mobile-style examples:**
```
spiffe://prod.tmobile.com/ns/payments/sa/checkout-service
spiffe://prod.tmobile.com/ns/network/sa/provisioning-api
spiffe://prod.tmobile.com/ns/auth/sa/token-service
spiffe://staging.tmobile.com/ns/payments/sa/checkout-service
spiffe://prod.tmobile.com/ns/infra/sa/vault-agent
spiffe://prod.tmobile.com/ns/monitoring/sa/splunk-forwarder
```

**Key rules:**
- Trust domain uniquely identifies the administrative boundary (one cluster = one trust domain, usually)
- A workload in `prod.tmobile.com` cannot be validated by a peer in `staging.tmobile.com` unless trust federation is configured
- The path is arbitrary — the convention above is recommended but not required by the spec

## SPIRE Architecture

```
[SPIRE Server]
  - Issues SVIDs (X.509 certificates with SPIFFE IDs)
  - Maintains entry registry (which workloads get which SPIFFE IDs)
  - Has its own X.509 CA certificate (the trust anchor)
  - Runs as a StatefulSet or external service
       |
       | (Authenticated via node attestation)
       |
[SPIRE Agent] (DaemonSet — one per Kubernetes node)
  - Proves the node's identity to the SPIRE Server on startup
  - Interrogates workloads requesting SVIDs
  - Maintains a local SVID cache for each workload
  - Serves the Workload API over a Unix domain socket
       |
       | (Workload API — Unix socket: /tmp/spire-agent/public/api.sock)
       |
[Workload / Sidecar Proxy]
  - Requests SVID via the Workload API
  - Uses the SVID for mTLS (X.509-SVID) or HTTP auth (JWT-SVID)
  - SPIRE Agent handles rotation automatically; workload gets new SVID before expiry
```

## Attestation Plugin Types

### Node Attestation (How the SPIRE Agent proves the node is legitimate)

| Plugin | How It Works | Environment |
|--------|-------------|-------------|
| `k8s_psat` | Agent presents a Projected Service Account Token bound to the node; SPIRE Server validates against K8s API | Kubernetes (recommended) |
| `aws_iid` | Agent presents AWS EC2 Instance Identity Document; SPIRE Server validates with AWS API | AWS EC2 / EKS |
| `gcp_iit` | Agent presents GCP Instance Identity Token; SPIRE Server validates with GCP API | GCP GKE |
| `join_token` | One-time token pre-shared with the agent; simple but does not provide strong attestation | Testing / bootstrap only |

### Workload Attestation (How the SPIRE Agent proves a workload is legitimate before issuing an SVID)

| Plugin | How It Works | Environment |
|--------|-------------|-------------|
| `k8s` | Agent validates pod service account, namespace, and labels against K8s API | Kubernetes |
| `unix` | Attestation based on process UID/GID (for non-containerized workloads) | Linux processes |
| `docker` | Attestation based on Docker container metadata | Docker (non-K8s) |

## Key SPIRE CLI Commands

### SPIRE Server

```bash
# Show the current trust bundle (CA certificate) — share with peers for federation
spire-server bundle show \
  -socketPath /tmp/spire-server/private/api.sock

# List all registered SPIRE agents (nodes that have attested)
spire-server agent list \
  -socketPath /tmp/spire-server/private/api.sock

# Create a workload entry (register a workload to receive an SVID)
spire-server entry create \
  -socketPath /tmp/spire-server/private/api.sock \
  -parentID spiffe://prod.tmobile.com/k8s-node \
  -spiffeID spiffe://prod.tmobile.com/ns/payments/sa/checkout-service \
  -selector k8s:ns:payments \
  -selector k8s:sa:checkout-service

# List all registered entries
spire-server entry show \
  -socketPath /tmp/spire-server/private/api.sock

# Delete an entry
spire-server entry delete \
  -socketPath /tmp/spire-server/private/api.sock \
  -entryID <entry-id>

# Refresh the trust bundle (force re-issuance if bundle is expiring)
spire-server bundle refresh \
  -socketPath /tmp/spire-server/private/api.sock
```

### SPIRE Agent

```bash
# Fetch an X.509 SVID for the workload running on this node (useful for debugging)
spire-agent api fetch x509 \
  -socketPath /tmp/spire-agent/public/api.sock

# Fetch a JWT SVID
spire-agent api fetch jwt \
  -socketPath /tmp/spire-agent/public/api.sock \
  -audience api.payments.prod.tmobile.com

# Check SPIRE agent health
spire-agent healthcheck \
  -socketPath /tmp/spire-agent/public/api.sock
```

## SVID Types

| SVID Type | Format | Use Case | Expiry |
|-----------|--------|----------|--------|
| **X.509-SVID** | X.509 certificate | mTLS connections, any TLS-based protocol | Short (1-24 hours); SPIRE auto-rotates |
| **JWT-SVID** | JSON Web Token | HTTP/REST API calls where TLS cert is not the auth mechanism; service-to-service HTTP where only the sender needs to prove identity | Shorter (1-hour or less); audience-bound |

**X.509-SVID anatomy:**
- Subject: workload identifier
- Subject Alternative Name (SAN): `URI: spiffe://trust-domain/path` — this is where the SPIFFE ID lives
- Issuer: SPIRE Server's CA
- NotBefore / NotAfter: defines the validity window — SPIRE rotates before NotAfter

## Common SPIRE Errors and Diagnosis

| Error | Likely Cause | Diagnosis |
|-------|-------------|-----------|
| `code = Unavailable desc = connection error` from workload API | SPIRE Agent not running on the node, or socket path wrong | `kubectl get pods -n spire-system -o wide` — is the DaemonSet pod running on this node? |
| `no SVID found for workload` | No matching entry registered in SPIRE Server for this workload | `spire-server entry show` — verify the selector matches the pod's namespace and service account |
| `node attestation failed` | Agent cannot prove node identity to SPIRE Server — often an RBAC issue with the k8s_psat plugin | Check SPIRE Server logs for attestation error; verify SPIRE's ClusterRole has access to TokenReview API |
| `SVID has expired` and agent is healthy | SVID rotation window too close to TTL, or agent's rotation polling interval too long | Check `svid_ttl` in SPIRE Server config vs. `rotateSVIDInterval` in SPIRE Agent config |
| `trust domain mismatch` during mTLS | Two services have SVIDs from different SPIRE trust domains without federation | Configure bundle federation between the two SPIRE servers, or consolidate to a single trust domain |

## SPIRE Integration Points

| System | How It Integrates | Purpose |
|--------|-----------------|---------|
| **HashiCorp Vault** | Vault SPIFFE auth method (`vault auth enable spiffe`) | Workload presents X.509-SVID to Vault as authentication credential — eliminates the need for Kubernetes service account tokens as the auth mechanism |
| **Istio** | SPIRE replaces Istio's built-in CA; SPIRE Agent serves SVID to Istio proxy via SDS API | Istio uses SPIRE-issued SVIDs for mTLS instead of Istio's own cert management — single identity system for both mesh and non-mesh workloads |
| **Envoy** | SPIRE Agent implements Envoy's SDS (Secret Discovery Service) API | Envoy sidecars (outside of Istio) can fetch SVIDs from SPIRE Agent directly for mTLS |
| **cert-manager** | SPIRE ACME server (experimental) issues certs to cert-manager | For workloads that use cert-manager for TLS certificates — bridges cert-manager lifecycle with SPIRE identity |
