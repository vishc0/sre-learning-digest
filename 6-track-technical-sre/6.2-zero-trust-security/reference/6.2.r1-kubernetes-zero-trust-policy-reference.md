# Kubernetes Zero Trust Policy Quick Reference

## NetworkPolicy

### Default Deny All (ingress + egress)

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: payments
spec:
  podSelector: {}       # Selects all pods in the namespace
  policyTypes:
  - Ingress
  - Egress
```

**Gotcha**: This also blocks DNS. Add an explicit DNS egress rule immediately after applying this.

### Allow DNS Egress (required after default-deny-egress)

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns-egress
  namespace: payments
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - protocol: UDP
      port: 53
    - protocol: TCP
      port: 53
```

### Allow Same-Namespace Traffic

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-same-namespace
  namespace: payments
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector: {}   # Empty podSelector = all pods in this namespace
```

### Allow Specific Ingress from Another Namespace

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-checkout-ingress
  namespace: payments
spec:
  podSelector:
    matchLabels:
      app: payments-api
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: ecommerce
      podSelector:
        matchLabels:
          app: checkout-service
    ports:
    - protocol: TCP
      port: 8080
```

### Allow Specific External Egress (IP Block)

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-external-payment-gateway
  namespace: payments
spec:
  podSelector:
    matchLabels:
      app: payments-api
  policyTypes:
  - Egress
  egress:
  - to:
    - ipBlock:
        cidr: 203.0.113.0/24    # Payment gateway IP range
        except:
        - 203.0.113.128/25      # Excluded range within CIDR
    ports:
    - protocol: TCP
      port: 443
```

**Critical gotcha — NetworkPolicy is additive (OR logic):**
Multiple NetworkPolicy resources that match the same pod are combined with OR logic. If policy A allows ingress from namespace X and policy B allows ingress from namespace Y, pods matching both policies accept ingress from EITHER namespace X OR namespace Y. There is no AND between policies, and there is no explicit deny (short of having no matching allow rule at all). This means you cannot use separate policies to express "allow from X AND only if condition Y."

## Pod Security Admission (PSA)

PSA replaces PodSecurityPolicy (removed in K8s 1.25). It enforces security constraints at the pod spec level via namespace labels.

### Namespace Label Syntax

```bash
# Apply a security profile to a namespace
kubectl label namespace payments pod-security.kubernetes.io/enforce=restricted
kubectl label namespace payments pod-security.kubernetes.io/warn=restricted
kubectl label namespace payments pod-security.kubernetes.io/audit=restricted
```

Three modes: `enforce` (reject non-compliant pods), `warn` (allow but warn), `audit` (allow and log).

### Profile Comparison

| Restriction | `privileged` | `baseline` | `restricted` |
|-------------|-------------|------------|-------------|
| Privileged containers | Allowed | Blocked | Blocked |
| hostNetwork, hostPID | Allowed | Blocked | Blocked |
| hostPath volumes | Allowed | Blocked | Blocked |
| Capabilities (e.g., NET_ADMIN) | Allowed | Limited set | Drop ALL required |
| runAsNonRoot | Not required | Not required | Required |
| seccompProfile | Any | Not required | RuntimeDefault or Localhost |
| allowPrivilegeEscalation | Allowed | Not required | Must be false |

**Recommended baseline for production**: `restricted` for all application namespaces. `baseline` for system/infra namespaces where some elevated capabilities are required.

## Istio PeerAuthentication

### Mesh-Wide STRICT Mode (applies to all namespaces without a more specific policy)

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system   # istio-system = mesh-wide scope
spec:
  mtls:
    mode: STRICT
```

### Namespace-Scoped STRICT Mode

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: payments
spec:
  mtls:
    mode: STRICT
```

### PERMISSIVE Mode (migration mode — accept both mTLS and plaintext)

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: payments
spec:
  mtls:
    mode: PERMISSIVE
```

### Workload-Scoped PERMISSIVE (for targeted migration)

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: payments-api-permissive
  namespace: payments
spec:
  selector:
    matchLabels:
      app: payments-api
  mtls:
    mode: PERMISSIVE
```

**Policy hierarchy**: Workload-scoped > Namespace-scoped > Mesh-wide. A workload-scoped PERMISSIVE policy overrides a namespace-scoped STRICT policy for that workload only.

## Istio AuthorizationPolicy

### Deny All (empty spec — no rules = deny all)

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: payments
spec: {}
```

### Allow Specific Source (by SPIFFE principal)

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-checkout
  namespace: payments
spec:
  action: ALLOW
  rules:
  - from:
    - source:
        principals:
        - "cluster.local/ns/ecommerce/sa/checkout-service"
    to:
    - operation:
        methods: ["POST"]
        paths: ["/payments/charge"]
```

**Principal format**: `cluster.local/ns/<namespace>/sa/<service-account>` — `cluster.local` is the default trust domain for Istio's built-in CA. If using SPIRE, the trust domain is your SPIRE trust domain.

### JWT Claims-Based Authorization

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-admin-jwt
  namespace: payments
spec:
  action: ALLOW
  rules:
  - from:
    - source:
        requestPrincipals: ["https://auth.tmobile.com/*"]
    when:
    - key: request.auth.claims[role]
      values: ["payment-admin"]
    to:
    - operation:
        methods: ["DELETE"]
        paths: ["/payments/refund"]
```

### Allow Namespace-Wide (for less-sensitive services)

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-internal-namespace
  namespace: monitoring
spec:
  action: ALLOW
  rules:
  - from:
    - source:
        namespaces: ["payments", "ecommerce", "auth"]
```

**Critical gotcha — Istio policy evaluation order:**
`CUSTOM` → `DENY` → `ALLOW`

1. CUSTOM policies (external authorizers) are evaluated first
2. If any DENY policy matches the request, it is denied (even if an ALLOW policy also matches)
3. If any ALLOW policy matches the request, it is allowed
4. If no ALLOW policy matches and no DENY policy matches, the request is denied if any AuthorizationPolicy exists in the namespace (implicit deny)

This means: a DENY rule always wins over an ALLOW rule. Use DENY rules sparingly — a misconfigured DENY can block legitimate traffic that your ALLOW policy intends to permit.

## OPA / Gatekeeper

### ConstraintTemplate Skeleton

```yaml
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8srequiredlabels
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredLabels
      validation:
        openAPIV3Schema:
          type: object
          properties:
            labels:
              type: array
              items:
                type: string
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8srequiredlabels

        violation[{"msg": msg}] {
          provided := {label | input.review.object.metadata.labels[label]}
          required := {label | label := input.parameters.labels[_]}
          missing := required - provided
          count(missing) > 0
          msg := sprintf("missing required labels: %v", [missing])
        }
```

### Constraint Example

```yaml
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredLabels
metadata:
  name: require-team-cost-env-labels
spec:
  enforcementAction: deny     # or "warn" for audit-only mode
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Namespace"]
    excludedNamespaces:       # Exclude system namespaces
    - kube-system
    - istio-system
    - spire-system
  parameters:
    labels:
    - team
    - cost-center
    - environment
```

**enforcementAction options:**
- `deny`: Reject non-compliant resources (production enforcement)
- `warn`: Allow but return a warning (migration / audit mode)
- `dryrun`: Allow and log to Gatekeeper audit, no user-facing warning

## Common Gotchas Summary

| Gotcha | Context | Impact | Fix |
|--------|---------|--------|-----|
| NetworkPolicy is additive (OR, not AND) | NetworkPolicy | Cannot express "allow from X AND Y simultaneously required" — works as "allow from X OR allow from Y" | Design policies with this in mind; use Cilium for more expressive policy |
| Default-deny egress blocks DNS | NetworkPolicy | Pods cannot resolve service names — causes wide connectivity failures that look like the services are down | Always add DNS egress allow rule before or with default-deny egress |
| Istio DENY beats ALLOW | AuthorizationPolicy | A catch-all DENY rule blocks traffic even where an ALLOW rule matches | Use DENY sparingly; default to no DENY policies and rely on implicit deny from absent ALLOW |
| PERMISSIVE mode is not enforcement | PeerAuthentication | Traffic accepted in plaintext; a service in PERMISSIVE mode gives no security guarantee | Track PERMISSIVE namespaces and drive them to STRICT on a schedule |
| Gatekeeper `warn` mode looks like enforcement | OPA/Gatekeeper | Teams see warnings but resources are admitted — policy appears to be working when it is not enforcing | Explicitly document which policies are in warn vs. deny mode; track warn→deny migration |
| Istio policy scope: empty namespace = mesh-wide | AuthorizationPolicy | An AuthorizationPolicy with no namespace selector in `istio-system` applies to the entire mesh | Always scope AuthorizationPolicy to a specific namespace unless mesh-wide is intentional |
