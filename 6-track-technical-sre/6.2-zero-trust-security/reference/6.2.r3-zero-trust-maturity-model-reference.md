# Zero Trust Maturity Model Reference

## CISA Zero Trust Maturity Model v2.0 — Full Pillar Map

Based on CISA Zero Trust Maturity Model v2.0 (April 2023). This reference maps each maturity cell to: what it looks like in practice, the SRE tool or action that advances you to the next level, and how to measure it.

---

## Pillar 1: Identity

| Maturity Level | What It Looks Like | SRE Tool / Action to Advance | Measurement |
|---------------|-------------------|------------------------------|-------------|
| **Traditional** | Password-only auth for most users. MFA optional or partial. Manual account provisioning/deprovisioning. Service accounts share credentials. No machine identity framework. | Enforce MFA org-wide. Automate provisioning via SCIM/directory sync. Begin service account inventory. | % users with MFA enabled. Days to deprovision departed employee. |
| **Advanced** | MFA enforced for all users. SSO for most applications. Risk-based auth for sensitive operations. Service accounts use Kubernetes SA tokens or Vault auth. Some JIT access for privileged roles. | Deploy SPIFFE/SPIRE for workload identity. Implement Vault Kubernetes auth for all production services. Enable JIT access via CyberArk for all privileged accounts. Phishing-resistant MFA (FIDO2) for admin accounts. | % services using dynamic/short-lived machine credentials. % privileged access that is JIT (not standing). % admin accounts using phishing-resistant MFA. |
| **Optimal** | Continuous validation on every request — session trust is re-evaluated based on behavior, not just initial auth. Behavioral analytics detect anomalous access patterns. Zero standing privileged access (all JIT). Every machine identity is SPIFFE-compliant and short-lived. | Integrate SIEM behavioral analytics with access control (adaptive auth). SPIRE for all workloads including batch and CI/CD. Automated access review with policy-driven deprovisioning. | Mean time to detect anomalous access. % access decisions using behavioral signal. Credential TTL distribution (target: 100% < 24h). |

**Director talking point (Identity pillar)**: "The Identity pillar investment has a direct line to reducing lateral movement. Every service account that moves from a static credential to a SPIFFE-based dynamic credential reduces the window of exposure from 'indefinite' to 'hours.' We track this as a coverage metric — percentage of production services on dynamic credentials — and report it quarterly. The ROI is blast radius reduction: a stolen credential that expires in one hour cannot sustain an attack the way a static API key can."

**Common maturity assessment trap**: "We have Vault" does not equal Advanced maturity. Advanced maturity requires that the majority of production services actually use Vault for machine auth, with dynamic credentials. Vault deployed but only used by 20% of services = Traditional maturity with a better toolbox.

## Pillar 2: Devices

| Maturity Level | What It Looks Like | SRE Tool / Action to Advance | Measurement |
|---------------|-------------------|------------------------------|-------------|
| **Traditional** | Partial device inventory. No MDM or MDM not enforced. Device health not considered in access decisions. No EDR on many endpoints. Manual patch tracking. | Deploy MDM (Jamf, Intune) and enforce enrollment. Implement EDR (CrowdStrike, Defender ATP). Build device inventory with asset tagging. | MDM enrollment rate. EDR coverage rate. Mean time to patch critical CVE. |
| **Advanced** | MDM-enrolled devices only for corporate access. Device compliance check at authentication (disk encryption, OS version, MDM enrollment). EDR on all corporate endpoints. Automated patch deployment for critical CVEs. | Integrate device compliance into ZTNA access decisions (Okta Device Trust, Cloudflare Access device posture). Automate non-compliant device quarantine. | % access decisions that include a device posture check. Time to quarantine non-compliant device. |
| **Optimal** | Continuous device posture evaluation — compliance is checked on every access request, not just at login. Real-time telemetry from EDR integrated into access decisions. Automated remediation for compliance drift. | Integrate EDR telemetry with SIEM and access proxy for real-time posture-gated access. Device attestation (TPM-based) for highest-sensitivity access. | Compliance drift detection time (how quickly is a device quarantined after becoming non-compliant?). |

**Director talking point (Devices pillar)**: "The Devices pillar is where the most organizational friction occurs. Engineers resist MDM enrollment on work machines. Our approach was to tie device compliance to access scope — a non-enrolled device gets read-only access to non-production systems. Production access requires enrollment, disk encryption, and an EDR agent. The access difference creates the enrollment incentive without mandating it."

## Pillar 3: Networks

| Maturity Level | What It Looks Like | SRE Tool / Action to Advance | Measurement |
|---------------|-------------------|------------------------------|-------------|
| **Traditional** | Flat internal network. Perimeter firewall for north-south. No east-west controls. All internal traffic trusted by default. VPN for remote access. | Apply Kubernetes NetworkPolicy with default-deny in highest-risk namespaces. Begin service communication mapping. | % namespaces with default-deny NetworkPolicy. East-west traffic visibility (% of service pairs with known/documented communication). |
| **Advanced** | Microsegmentation via NetworkPolicy and/or service mesh. East-west traffic encrypted (mTLS PERMISSIVE mode across all namespaces). DNS filtering active. VPN being replaced by ZTNA for human access. Service dependency map documented. | Enable STRICT mTLS namespace by namespace starting with highest-risk. Deploy Cilium or equivalent for FQDN-based egress. Deploy ZTNA for all privileged remote access. | % namespaces with STRICT mTLS. % privileged remote access via ZTNA (not VPN). Dropped packet counts by namespace (visibility into policy effectiveness). |
| **Optimal** | All east-west traffic mutually authenticated with SPIFFE-compliant certificates. Software-defined perimeter — no VPN at all. All egress through inspected, policy-gated gateways. Network policy is code-reviewed and tested in CI. | Implement network policy-as-code pipeline with full CI test coverage. Integrate network telemetry (Cilium Hubble, Istio access logs) with SIEM for anomaly detection. | Coverage of mTLS on all meshed service pairs. Network anomaly detection alert MTTD. |

**Director talking point (Networks pillar)**: "Our service mesh rollout gave us something we did not expect: the first accurate service dependency map we had ever had. By deploying in PERMISSIVE mode first and logging all east-west traffic in Splunk, we discovered 23 service-to-service calls that the teams were not aware of — undocumented dependencies from years of organic growth. That dependency map is now the authoritative source for our AuthorizationPolicy. Enforcement became safe only because discovery came first."

## Pillar 4: Applications

| Maturity Level | What It Looks Like | SRE Tool / Action to Advance | Measurement |
|---------------|-------------------|------------------------------|-------------|
| **Traditional** | Manual access reviews, usually annual. Monolithic apps with coarse-grained authorization. No ABAC. Security scanning is ad-hoc or post-deployment. | Implement RBAC for all internal services. Add SAST/SCA to CI pipeline. Begin quarterly access reviews. | Time since last access review. % services with documented authorization model. |
| **Advanced** | Per-application access policies via AuthorizationPolicy (Istio) or OPA Gatekeeper. ABAC for sensitive operations. Security gates (SAST, DAST, dependency scanning) in CI/CD. Automated access review triggers on role changes. | Deploy AuthorizationPolicy with deny-all + explicit allow in all production namespaces. Implement OPA Gatekeeper for admission control. Integrate DAST into pre-production pipelines. | % production namespaces with explicit AuthorizationPolicy. Gatekeeper policy coverage (% of admission requests evaluated). |
| **Optimal** | Continuous per-request authorization — every API call evaluated against current policy with current identity context. ML-based anomaly detection on application access patterns. Threat intelligence integrated into policy decisions. | Implement continuous authorization middleware in highest-sensitivity services. Integrate threat intelligence feeds with OPA policy evaluation. Run chaos engineering on auth controls (verify denials work under load). | Authorization policy evaluation latency (p99). False positive rate on anomaly detection. |

**Director talking point (Applications pillar)**: "Policy-as-code was the forcing function that changed how our teams think about security. When authorization policy lives in Rego files in the same repository as the application, it gets the same review, the same testing, and the same deployment pipeline. Security review is not a gate at the end — it is built into the development cycle. The result is that policy changes are no longer feared because they are tested in CI before they ever reach production."

## Pillar 5: Data

| Maturity Level | What It Looks Like | SRE Tool / Action to Advance | Measurement |
|---------------|-------------------|------------------------------|-------------|
| **Traditional** | Informal or no data classification. Encryption at rest on most systems but not enforced. DLP at email gateway only. No data inventory. Data retention is manual. | Define a data classification framework (Public / Internal / Confidential / Restricted). Enforce encryption at rest on all production databases. Document data store inventory. | % data stores with encryption at rest. % of data assets classified. |
| **Advanced** | Automated data classification (tagging based on content patterns). Data-level access controls aligned with classification (Confidential data accessible only to specific roles). Encryption at rest with cloud provider KMS. DLP extended to egress gateways. Data retention automated. | Implement BYOK (customer-managed keys) for Restricted data. Deploy Vault's transit secrets engine for application-layer encryption of sensitive fields. Integrate data classification labels with Vault policies. | % Restricted data encrypted with customer-managed keys. % egress traffic inspected by DLP. Data classification coverage (% of data stores with assigned classification). |
| **Optimal** | Automated tagging for all new data at creation time. Real-time access decisions based on data sensitivity + user identity + context. Continuous DLP with real-time blocking. Data access fully attributable (every read logged to audit trail). | Implement real-time data access governance (immuta, Privacera, or equivalent). Integrate data sensitivity signals into Vault policy evaluation. Build a data lineage map. | Data access anomaly detection rate. Time to detect unauthorized data access attempt. % of data reads attributable to a specific identity and time. |

**Director talking point (Data pillar)**: "The Data pillar is where compliance requirements and security architecture converge most directly. PCI-DSS v4 requirement 3 (protect stored account data) and requirement 7 (restrict access) are both addressed by our data classification and Vault transit encryption work. The business case to the CFO is straightforward: the cost of a customer data breach at our scale is measured in hundreds of millions when you include regulatory fines, notification costs, and customer churn. The cost of our data pillar investment is a fraction of that."

## Key Metrics Per Pillar

| Pillar | Primary Metric | Secondary Metric | Measurement Frequency |
|--------|---------------|-----------------|----------------------|
| Identity | % production services on dynamic/short-lived credentials | Mean time to revoke compromised credential | Monthly |
| Devices | % access decisions with device posture check | Time to quarantine non-compliant device | Monthly |
| Networks | % namespaces with STRICT mTLS | East-west traffic anomaly detection MTTD | Weekly |
| Applications | % production namespaces with AuthorizationPolicy | Policy evaluation coverage (% of API calls evaluated) | Weekly |
| Data | % Restricted data with BYOK encryption | % data reads attributable to specific identity | Monthly |

## Director Interview: Maturity Assessment Traps

| Trap | Why It Is Wrong | Correct Answer |
|------|----------------|----------------|
| "We have Vault" = Identity Advanced | Vault deployed ≠ Vault adopted. Coverage is the metric. | "X% of production services use Vault with dynamic credentials. The remaining Y% are on the migration roadmap with a target date." |
| "We have a service mesh" = Networks Advanced | Mesh in PERMISSIVE mode provides telemetry, not enforcement. | "Our mesh is deployed cluster-wide. Z% of namespaces are in STRICT mode. We are progressing namespace by namespace, validating caller inventory before each flip." |
| "We did an annual access review" = Identity Optimal | Annual reviews are Traditional maturity. Optimal means continuous or automated. | "We have automated access review triggers on role change events. Full access review runs quarterly with automated deprovisioning for accounts with no recent activity." |
| "We have OPA Gatekeeper" = Applications Advanced | Gatekeeper in `warn` mode is not enforcement. Policy count is not coverage. | "Gatekeeper is deployed in `deny` mode on X namespaces covering Y% of admission requests. Z% of our ConstraintTemplates have passing CI test coverage." |
| "Our data is encrypted" = Data Advanced | Encryption at rest with provider-managed keys is Traditional. BYOK + data-level access control = Advanced. | "Production databases are encrypted at rest. Our Restricted-classification data uses customer-managed keys in AWS KMS. We are implementing application-layer field-level encryption for PAN data via Vault transit." |
