# Playbook: Unauthorized Privileged Access Incident

## Incident Type

Unauthorized or anomalous privileged access detected — a human account or machine credential has accessed production systems in a way that is not expected, not authorized, or is actively suspicious.

**Severity**: P1 if production data accessed, if credentials may be compromised, or if active lateral movement is suspected. P2 for suspicious patterns with no confirmed data access.

---

## Detection

### Path 1: CyberArk alert

CyberArk generates alerts on:
- Session initiated outside approved hours (time-based restriction violation)
- Session initiated from an unrecognized IP or device
- Account checked out that has not been used in > 90 days (dormant account activation)
- Failed authentication attempts exceeding threshold (credential stuffing or brute force)
- Concurrent sessions on the same account

Treat any CyberArk alert as P1 until the session is confirmed as legitimate. Do not wait for a human to review it — isolate first, then investigate.

### Path 2: Vault audit log anomaly in Splunk

```splunk
index=vault sourcetype=vault_audit
| search type=response
| stats count by auth.metadata.service_account_name, path
| where count > 200
| sort -count
```
Threshold of 200 reads/hour per service account is a starting signal — adjust based on your baseline.

```splunk
index=vault sourcetype=vault_audit
| search type=response request_path!="/sys/leases/renew" request_path!="/auth/*"
| eval hour=strftime(_time, "%H")
| stats count by auth.display_name, request_path, hour
| where hour < 6 OR hour > 22
| sort -count
```
Purpose: access outside business hours for paths that are not automated lease renewals.

```splunk
index=vault sourcetype=vault_audit
| search type=response error=""
| eval is_privileged=if(match(request_path, "sys/auth|sys/policy|secret/prod|pki/root"), 1, 0)
| where is_privileged=1
| table _time, auth.display_name, auth.metadata.service_account_name, request_path, remote_address
| sort _time
```
Purpose: any access to root PKI, policy management, or production secret paths — flag for immediate review.

### Path 3: SOC escalation

The SOC detects an alert from SIEM correlation (cross-system: CyberArk + Vault + Kubernetes API audit + network egress). Follow the SOC's incident ticket and use this playbook for the technical response.

## Step 1: Isolation — Revoke the Session or Token Immediately

**Do not wait for root cause analysis before isolation. Contain first.**

### Revoke a Vault token

If a Vault token is suspected to be compromised:
```bash
# Revoke the specific token
vault token revoke <token-id>

# If you only have the accessor (from audit log), use accessor revocation
vault token revoke -accessor <accessor>

# Revoke all tokens issued to a specific service account (nuclear option)
vault token revoke -mode=path auth/kubernetes/login
```

Verify revocation took effect:
```bash
vault token lookup <token-id>
# Should return "Code: 403. Errors: permission denied" after successful revocation
```

### Terminate a CyberArk session

Via CyberArk PVWA:
1. Navigate to Active Sessions
2. Locate the session by username, safe name, or target device
3. Select "Terminate Session"
4. Confirm — the session is immediately disconnected and the connection proxy drops the link

Via CyberArk API (for automated response integration):
```bash
curl -X POST "https://cyberark.tmobile.com/PasswordVault/API/LiveSessions/<sessionId>/Terminate" \
  -H "Authorization: Bearer <api-token>"
```

### Revoke a Kubernetes service account token

```bash
# Delete and recreate the service account to invalidate all issued tokens
kubectl delete serviceaccount <sa-name> -n <namespace>
kubectl create serviceaccount <sa-name> -n <namespace>

# Restart all pods using the service account to force token rotation
kubectl rollout restart deployment/<deployment-name> -n <namespace>
```

## Step 2: Audit — What Did the Session Access?

Do not proceed to remediation before completing the audit. You need to know what was accessed before you can assess the scope of the breach.

### Vault audit — what secrets were read?

```splunk
index=vault sourcetype=vault_audit
| search auth.display_name="<compromised-account>" OR auth.metadata.service_account_name="<sa-name>"
| search type=response error=""
| table _time, auth.display_name, request_path, request_operation, remote_address
| sort _time
```

Note every `request_path` that contained a read operation. These are the secrets that may have been exfiltrated.

```splunk
index=vault sourcetype=vault_audit
| search auth.display_name="<compromised-account>"
| search request_operation="read" OR request_operation="list"
| stats values(request_path) as paths_accessed by auth.display_name
```

### CyberArk — session recording review

1. In PVWA, navigate to Session Management > Recordings
2. Locate the session by account name, target machine, or session ID
3. Review the full recording — look for:
   - Commands run during the session
   - Files accessed or downloaded
   - Outbound connections initiated
   - Configuration changes made
4. Export the audit log to PDF or CSV for the incident record

### Kubernetes API audit — what cluster operations were performed?

```bash
# Pull Kubernetes audit log for the service account
kubectl get events -n <namespace> --sort-by='.lastTimestamp'

# If audit logging is enabled in the cluster (check with your platform team)
# Query the audit log for actions by the compromised service account
grep '"serviceAccountName":"<sa-name>"' /var/log/kubernetes/audit.log | \
  jq '{time: .requestReceivedTimestamp, verb: .verb, resource: .objectRef.resource, namespace: .objectRef.namespace}'
```

## Step 3: Scope Assessment

Answer these questions before moving to remediation:

1. **What was accessed?** — List all secret paths read, all systems accessed, all commands run
2. **Were any credentials exfiltrated?** — Were database passwords, API keys, or cloud credentials read from Vault?
3. **Lateral movement possible?** — Can the compromised credential be used to access other systems? What does the Vault policy allow?
4. **Other credentials compromised?** — Did the session have access to safes or Vault paths that contain credentials for other accounts? Those must also be treated as compromised.
5. **Duration** — How long was the session active before detection and isolation?

```splunk
# Correlate Vault access with network egress to detect potential data exfiltration
index=vault sourcetype=vault_audit auth.display_name="<account>"
| eval vault_time=_time
| join type=left auth.remote_address [
    search index=network_flow src_ip=* dst_port!=443 dst_port!=80 earliest=-1h
    | table src_ip, dst_ip, dst_port, bytes_out
  ]
| where bytes_out > 1000000
| table vault_time, request_path, src_ip, dst_ip, dst_port, bytes_out
```

## Step 4: Remediation

Remediation scope is determined by the audit in Step 2.

### Rotate every secret that was accessed

```bash
# For each Vault secret path that was read:
vault kv put <path>/secret value="<new-rotated-value>"

# For Vault dynamic secrets (database credentials) — revoke all leases under the affected role
vault lease revoke -prefix auth/kubernetes/<role-name>
vault lease revoke -prefix database/creds/<db-role-name>
```

### Rotate cloud credentials if IAM keys were accessed

```bash
# AWS IAM key rotation
aws iam create-access-key --user-name <username>
# Store new key in Vault or Secrets Manager
# Then delete the old key
aws iam delete-access-key --user-name <username> --access-key-id <old-key-id>
```

### Review access grants

After any privileged access incident, conduct an access review for the affected accounts:

```bash
# Vault — list all policies attached to the affected auth role
vault read auth/kubernetes/role/<role-name>

# List all entities with a specific policy
vault list identity/entity/id | xargs -I{} vault read identity/entity/id/{}
```

## Step 5: Post-Incident

### Mandatory post-incident actions

- [ ] Document complete timeline: when the anomalous access started, when it was detected, when isolation occurred, how long the exposure window was
- [ ] CyberArk policy review: was this account supposed to have access to the targeted safe? If yes, was just-in-time approval required and was it bypassed?
- [ ] Vault policy tightening: review the policy attached to the compromised credential — does it grant more access than the workload or user actually needs?
- [ ] Detection gap analysis: why was this not detected sooner? Update Splunk alert thresholds or add new correlation rules
- [ ] Access review: run a full access review for the affected namespace/safe — remove any accounts with standing access that should be JIT-only

### Splunk lookups for post-incident access review

```splunk
index=vault sourcetype=vault_audit earliest=-30d
| stats dc(request_path) as unique_paths, count as total_ops, max(_time) as last_access
  by auth.metadata.service_account_name
| eval last_access_date=strftime(last_access, "%Y-%m-%d")
| sort -total_ops
```
Purpose: identify which service accounts are most active in Vault — high activity is expected; dormant accounts with any activity are a flag.

```splunk
index=vault sourcetype=vault_audit earliest=-30d
| stats latest(_time) as last_seen by auth.metadata.service_account_name
| where last_seen < relative_time(now(), "-14d")
| eval last_seen_date=strftime(last_seen, "%Y-%m-%d")
| table auth.metadata.service_account_name, last_seen_date
```
Purpose: service accounts that have not been used in > 14 days but still have active Vault policies. These should be reviewed for deprovisioning.
