# Playbook: Policy Deny Blocking Production Traffic

## Incident Type

Istio AuthorizationPolicy or OPA/Gatekeeper deny is blocking legitimate production traffic, causing service errors or complete service unavailability.

**Severity**: P1 if payment or auth services affected; P2 for non-critical services.

---

## Detection

### How the deny surfaces

**Path 1 — Application error**: Services return 403 Forbidden or RBAC connection reset to upstream callers. The application logs show "connection refused" or HTTP 403 from a downstream service. If the deny is at the mTLS/RBAC layer, the 403 is generated by the Envoy sidecar, not the application.

**Path 2 — Splunk alert on UAEX response flags**:
```splunk
index=istio sourcetype=istio_access_log
| search response_flags="UAEX"
| stats count by source_workload, destination_workload, response_code
| where count > 10
| sort -count
```
`UAEX` = upstream authorization external failure — Istio's RBAC denied the request.

**Path 3 — Gateway-level 403**: If the deny occurs at an ingress gateway AuthorizationPolicy, the upstream caller (load balancer, API gateway) receives a 403 and surfaces it in gateway access logs.

**Path 4 — OPA/Gatekeeper admission webhook**: A deployment or pod spec fails admission. The kubectl error message includes the Gatekeeper constraint name and violation message.

---

## Triage Steps

### Step 1: Identify the affected workload pair

```bash
# What is getting denied?
kubectl logs -n istio-system -l app=istiod --tail=100 | grep -i "rbac\|denied\|unauthorized"

# Check Envoy sidecar logs for the destination service
kubectl logs <destination-pod> -c istio-proxy --tail=100 | grep -i "rbac\|denied"
```

```splunk
# Identify all denied pairs in the last 15 minutes
index=istio sourcetype=istio_access_log earliest=-15m
| search response_flags="UAEX"
| table _time, source_workload, source_namespace, destination_workload, destination_namespace, request_uri, response_code
| sort _time
```

### Step 2: Identify the policy that triggered the deny

```bash
# List all AuthorizationPolicies in the destination namespace
kubectl get authorizationpolicy -n <destination-namespace> -o yaml

# Check if a deny-all policy exists
kubectl get authorizationpolicy -n <destination-namespace> | grep deny
```

Istio policy evaluation order: CUSTOM → DENY → ALLOW. A deny-all policy with an empty `spec: {}` matches everything. If a new ALLOW policy was not added for a newly added caller, that caller will be denied.

### Step 3: Check for recent policy changes

```bash
# Git history on the policy repository
git log --oneline --since="24 hours ago" -- policies/

# Kubernetes events for recent AuthorizationPolicy changes
kubectl get events -n <namespace> --sort-by='.lastTimestamp' | tail -20
```

### Step 4: Determine if this is enforcement or misconfiguration

Ask:
- Was there a recent deployment of a new service that now needs access?
- Was there a recent policy change (new deny rule, modified allow rule)?
- Is the source workload's service account what was expected?

```bash
# Confirm the source workload's service account (is it the expected identity?)
kubectl get pod <source-pod> -n <source-namespace> -o jsonpath='{.spec.serviceAccountName}'

# Confirm what SPIFFE ID Istio assigned to the source workload
istioctl proxy-config secret <source-pod> -n <source-namespace> | grep URI
```

## Temporary Bypass Procedure

Use only when the deny is confirmed to be a misconfiguration blocking legitimate traffic, not a legitimate security enforcement.

**This procedure must be time-bounded and documented before execution.**

### Option A: Scoped PERMISSIVE mode (narrowest blast radius)

Scope PERMISSIVE mode to the specific destination workload, not the entire namespace:

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: temp-permissive-payments-api
  namespace: payments
  annotations:
    incident: "INC-XXXXXXX"
    expires: "2025-07-07T00:00:00Z"
    owner: "oncall-sre@tmobile.com"
spec:
  selector:
    matchLabels:
      app: payments-api
  mtls:
    mode: PERMISSIVE
```

Apply: `kubectl apply -f temp-permissive-payments-api.yaml`

**Set a reminder to remove this within the incident resolution window.** Do not let temporary PERMISSIVE policies survive the incident.

### Option B: Add explicit ALLOW for the blocked caller

If the source workload is legitimate but missing from the ALLOW policy:

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-checkout-to-payments-temp
  namespace: payments
  annotations:
    incident: "INC-XXXXXXX"
    expires: "2025-07-07T00:00:00Z"
spec:
  action: ALLOW
  rules:
  - from:
    - source:
        principals: ["cluster.local/ns/ecommerce/sa/checkout-service"]
```

Apply and monitor immediately. Verify traffic is flowing:
```bash
kubectl exec -n ecommerce <checkout-pod> -- curl -v http://payments-api.payments.svc.cluster.local/health
```

## Root Cause Analysis

### Was this legitimate enforcement or misconfiguration?

**Legitimate enforcement**: The blocked workload was not supposed to call the destination service. The policy is working correctly. The resolution is to fix the workload (it should not be making this call) or to get the appropriate service-to-service access formally approved and added to the policy.

**Misconfiguration — new service not added to policy**: A new microservice was deployed without updating the AuthorizationPolicy in the destination namespace. The fix is to add the new service account to the ALLOW policy and remove the temporary bypass.

**Misconfiguration — policy evaluation order**: A DENY policy was added that incorrectly matches legitimate traffic. Review the DENY policies in the namespace — any DENY rule evaluated before the ALLOW rule will block traffic even if the ALLOW rule matches.

**Misconfiguration — wrong service account**: The source workload was deployed with the wrong Kubernetes service account (or no explicit service account, defaulting to `default`). The ALLOW policy references a specific service account, but the pod is using `default`. Fix: update the pod spec to use the correct service account.

---

## Remediation

1. Fix the root cause (update the policy, fix the service account, fix the workload behavior)
2. Validate the fix in staging before applying to production
3. Remove the temporary bypass (PERMISSIVE mode or temporary ALLOW policy)
4. Verify traffic flows correctly with the permanent fix:
   ```splunk
   index=istio sourcetype=istio_access_log
   | search source_workload="<source>" destination_workload="<destination>"
   | stats count by response_code, response_flags
   ```

---

## Post-Incident Actions

- **Policy CI test**: add a test case to the policy CI pipeline that covers the specific scenario that was missed. If a new service calling an existing service was not covered, add a test for that onboarding pattern.
- **Runbook update**: if the detection or triage steps were unclear, update this playbook.
- **Onboarding process**: if the root cause was "new service deployed without updating policy," update the service onboarding checklist to require AuthorizationPolicy updates as a deployment gate.
- **Policy documentation**: add a comment block to the AuthorizationPolicy YAML documenting which services are allowed and why. Undocumented policies are the root cause of future misconfigurations.
- **Retro**: 5-minute blameless retro in the next team standup — what made this hard to diagnose, what would have helped detect it faster?
