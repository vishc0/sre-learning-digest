# Playbook: mTLS Certificate Expiry Response

## Incident Type

mTLS certificates have expired (or are approaching expiry), causing connection failures between meshed services. Automatic rotation has failed or been missed.

**Severity**: P1 if payment, auth, or high-traffic services are affected; P2 for non-critical services.

---

## Detection

### Pre-expiry detection (preferred — catch it before impact)

Scheduled Splunk alert — run every 6 hours:
```splunk
index=istio sourcetype=istio_proxy_config
| eval expiry_epoch=strptime(cert_expiry, "%Y-%m-%dT%H:%M:%SZ")
| eval days_until_expiry=round((expiry_epoch - now()) / 86400, 1)
| where days_until_expiry < 7
| table workload, namespace, cert_expiry, days_until_expiry
| sort days_until_expiry
```

Alert threshold: < 7 days remaining = Warning. < 24 hours remaining = Critical.

### Post-expiry detection (reactive)

**Symptom pattern**: Sudden widespread connectivity failures across many service pairs simultaneously. Unlike a policy misconfiguration (which typically affects a specific pair), cert expiry affects all mTLS connections to or from the expired workload(s).

```splunk
index=istio sourcetype=istio_access_log earliest=-30m
| search response_flags="UC" OR response_flags="DC"
| stats count by destination_workload, destination_namespace
| where count > 50
| sort -count
```

`UC` = upstream connection failure. `DC` = downstream connection disconnect. High counts across multiple destination_workload values simultaneously = cert lifecycle incident, not a single service issue.

Additional confirmation: check the Envoy proxy status:
```bash
istioctl proxy-status
# Look for proxies showing NOT READY or CDS/LDS/RDS out of sync
# A cert expiry event causes xDS sync failures
```

## Immediate Diagnosis

### Step 1: Determine which workloads are affected

```bash
# Check cert expiry for all proxies in a namespace
istioctl proxy-config secret -n <namespace> --all | grep -A2 "CERT CHAIN"

# For a specific pod
istioctl proxy-config secret <pod-name> -n <namespace> -o json | \
  jq '.dynamicActiveSecrets[] | select(.name == "default") | .secret.tlsCertificate.certificateChain' | \
  base64 -d | openssl x509 -noout -dates -subject
```

Look at `notAfter`. If it is in the past — the cert is expired. If it is within the next 24 hours — imminent expiry.

### Step 2: Is this a SPIRE-managed cert or a manually issued cert?

```bash
# Check if SPIRE is the CA source for this namespace
kubectl get meshconfig -n istio-system -o jsonpath='{.spec.trustDomain}'
kubectl get configmap -n spire-system spire-server -o yaml | grep ca-cert

# Check SPIRE SVID for the affected workload
kubectl exec -n spire-system <spire-agent-pod> -- \
  /opt/spire/bin/spire-agent api fetch x509 -socketPath /tmp/spire-agent/public/api.sock
```

**SPIRE-managed**: automatic rotation should have occurred. If it did not, diagnose the SPIRE agent on the affected node (see Step 3).

**Manually issued**: find the rotation process that issued the cert and diagnose why it failed.

### Step 3: Diagnose SPIRE agent health on the affected node

```bash
# Find which SPIRE agent pods are on which nodes
kubectl get pods -n spire-system -o wide | grep spire-agent

# Check the SPIRE agent on the node where the affected workload runs
kubectl logs -n spire-system <spire-agent-pod-on-affected-node> --tail=200 | grep -i "error\|warn\|expire"

# Check if SPIRE agent can reach the SPIRE server
kubectl exec -n spire-system <spire-agent-pod> -- \
  curl -k https://spire-server.spire-system.svc.cluster.local:8081/healthz
```

Common SPIRE agent failure causes:
- SPIRE agent pod restarted and has not re-attested to the server
- SPIRE server overloaded (cert issuance queue backed up)
- Network policy blocking SPIRE agent → SPIRE server communication
- SPIRE server certificate itself is expired

## Emergency Rotation

### Path A: SPIRE-managed — force re-attestation

If the SPIRE agent is healthy but the workload cert has not rotated:
```bash
# Delete the affected pod to trigger a fresh attestation cycle
kubectl delete pod <affected-pod> -n <namespace>
# SPIRE agent will re-attest the new pod and issue a fresh SVID
```

If the SPIRE agent itself is the problem:
```bash
# Restart the SPIRE agent DaemonSet pod on the affected node
kubectl delete pod <spire-agent-pod> -n spire-system
# The agent will restart, re-attest to the server, and begin serving SVIDs again
```

### Path B: SPIRE bundle refresh (trust bundle propagation failure)

If the issue is that the trust bundle (CA certificate) has expired and is not being refreshed:
```bash
# On the SPIRE server pod
kubectl exec -n spire-system <spire-server-pod> -- \
  /opt/spire/bin/spire-server bundle refresh \
  -socketPath /tmp/spire-server/private/api.sock

# Verify the new bundle is being distributed
kubectl exec -n spire-system <spire-server-pod> -- \
  /opt/spire/bin/spire-server bundle show \
  -socketPath /tmp/spire-server/private/api.sock
```

### Path C: Istio built-in CA — force cert rotation

If using Istio's built-in CA (not SPIRE):
```bash
# Restart istiod to trigger cert re-issuance
kubectl rollout restart deployment/istiod -n istio-system

# Watch istiod come back up
kubectl rollout status deployment/istiod -n istio-system

# Restart affected workloads to get fresh certs
kubectl rollout restart deployment/<affected-deployment> -n <namespace>
```

### Path D: Istio — verify cert propagation after rotation

```bash
# After SPIRE or Istiod issues new certs, verify they propagated to all proxies
istioctl proxy-status
# All proxies should show SYNCED for CDS, LDS, RDS, EDS

# Verify the new cert's expiry
istioctl proxy-config secret <pod-name> -n <namespace> -o json | \
  jq '.dynamicActiveSecrets[] | select(.name == "default") | .secret.tlsCertificate.certificateChain' | \
  base64 -d | openssl x509 -noout -dates
# notAfter should now be in the future by the expected TTL (e.g., 24 hours)
```

## Verification: Connectivity Restored

```bash
# Test service-to-service connectivity from an affected caller
kubectl exec -n <source-namespace> <source-pod> -- \
  curl -v http://<destination-service>.<destination-namespace>.svc.cluster.local/health
# Expect: HTTP 200

# Verify no more UAEX or DC flags
```

```splunk
index=istio sourcetype=istio_access_log earliest=-10m
| search response_flags="UC" OR response_flags="DC" OR response_flags="UAEX"
| stats count by destination_workload
| sort -count
```

Count should be dropping or at zero.

## Identify All Certs Expiring in Next 30 Days

Run this proactively after the incident to find the next expiry before it hits:

```splunk
index=istio sourcetype=istio_proxy_config
| eval expiry_epoch=strptime(cert_expiry, "%Y-%m-%dT%H:%M:%SZ")
| eval days_until_expiry=round((expiry_epoch - now()) / 86400, 1)
| where days_until_expiry < 30
| table workload, namespace, cert_expiry, days_until_expiry, cert_type
| sort days_until_expiry
```

Any workload with < 7 days until expiry that did not self-rotate needs investigation — the rotation automation is not working for that workload.

## Post-Incident: Fix the Automation, Not Just the Cert

The root cause of any cert expiry incident is rotation automation failure. Patching the cert resolves the incident. Finding and fixing the automation failure prevents recurrence.

**Questions to answer in the post-incident review:**

1. Why did the SPIRE agent not rotate the cert automatically?
   - Was the agent healthy on the affected node?
   - Was SPIRE server reachable from the agent?
   - Was the SVID TTL shorter than the SPIRE agent's rotation polling interval?

2. Why did the monitoring alert not fire before impact?
   - Is the Splunk cert expiry alert actually running on schedule?
   - Is the Splunk data source (Istio proxy config) being indexed correctly?
   - What is the `cert_expiry` field's data freshness? (Is it updated on every pod restart, or only at a fixed interval?)

3. Is the rotation automation tested?
   - When was the last time someone deliberately let a cert approach expiry in staging and verified that rotation fired automatically?
   - Is cert rotation automation part of the quarterly DR test?

**Action items from this incident:**
- [ ] Add cert rotation automation to quarterly DR test checklist
- [ ] Verify Splunk cert expiry alert is running and alerting correctly
- [ ] Add SPIRE agent health check to the cluster health dashboard
- [ ] Document the emergency rotation procedure (this playbook) in the team runbook and walk the on-call team through it
