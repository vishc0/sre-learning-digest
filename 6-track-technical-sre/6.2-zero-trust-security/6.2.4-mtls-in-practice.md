# Chapter 4: mTLS in Practice

## What Mutual TLS Actually Does

Standard TLS (one-way TLS) is what your browser uses when it connects to https://example.com. The server presents a certificate. Your browser validates it against a trusted CA. The server's identity is proven. Your browser's identity is not proven — the server accepts any client.

Mutual TLS (mTLS) requires both sides to present and validate certificates:

1. Client connects to server, server presents its certificate
2. Client validates server certificate against a trusted CA — confirms the server is who it claims to be
3. Server requests the client's certificate
4. Client presents its certificate (containing its SPIFFE ID or other identity claim)
5. Server validates the client certificate against the same trusted CA — confirms the client is who it claims to be
6. Both sides have cryptographically verified identities; the session proceeds

The operational significance: in a service mesh with mTLS enforced, every service-to-service connection is a mutual handshake. The payments service knows it is talking to the checkout service (not a rogue pod spoofing the checkout service's IP). The checkout service knows it is talking to the payments service (not a man-in-the-middle). Both identities are proven before any application data flows.

Without mTLS, service-to-service trust typically relies on network location (the pod is on the right subnet) or application-layer tokens (API keys, JWTs passed in headers). Both are weaker than a cryptographic certificate: network location can be spoofed if an attacker has network access, and application-layer tokens can be stolen and replayed from any location.

## Certificate Lifecycle: The Operational Burden Most Teams Underestimate

The cryptographic machinery of mTLS is straightforward. The operational challenge is certificate lifecycle management at scale.

In a Kubernetes cluster with 100 services, each service needs:
- An identity certificate (X.509, SPIFFE-compliant)
- A TTL short enough to limit exposure if compromised (typically 1-24 hours for workload certs)
- Automatic rotation before expiry
- A trust bundle (the CA certificate) distributed to every workload so they can validate peers

With short-lived certificates, this means your certificate infrastructure is issuing and rotating hundreds or thousands of certificates per day. Any failure in that rotation pipeline — SPIRE agent not reachable, mTLS CA overloaded, Kubernetes API rate limiting — results in certificate expiry events that break mTLS handshakes and take down service-to-service communication.

The failure pattern is predictable: teams deploy mTLS, it works beautifully in staging, they promote to production, three months later the cert rotation automation has a silent failure, and certificates expire in production causing a widespread connectivity incident. The post-incident finding is always the same: "we didn't have monitoring on cert expiry."

This is why cert expiry monitoring is the first observability requirement in any mTLS deployment — not the last.

## Istio mTLS: PERMISSIVE vs. STRICT Mode

Istio implements mTLS through its sidecar proxy (Envoy). Every pod with an Istio sidecar injected gets automatic mTLS for all traffic that flows through the proxy.

Istio has two modes for mTLS, controlled by the `PeerAuthentication` CRD:

**PERMISSIVE mode**: The sidecar accepts both plaintext and mTLS traffic. This is the migration mode — it lets you gradually migrate services to mTLS without breaking communication with services that have not yet been meshed. A PERMISSIVE-mode service can be called by both meshed services (mTLS) and non-meshed services (plaintext).

**STRICT mode**: The sidecar only accepts mTLS traffic. Any plaintext connection is rejected. This is the target state — it means every caller must have a valid certificate and participate in mutual authentication.

```yaml
# Namespace-scoped STRICT mode
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: payments
spec:
  mtls:
    mode: STRICT
```

The operational failure mode: enabling STRICT mode on a namespace when a non-meshed caller still needs to reach it. The non-meshed caller's connection is rejected (no client certificate to present). The symptom is a connection reset or 503 at the service level. The fix is either to mesh the caller first, or temporarily revert to PERMISSIVE while the caller is onboarded.

**How sidecar proxies handle the mTLS handshake**: The sidecar intercepts outbound traffic from your application before it leaves the pod. The application believes it is making a plaintext HTTP call. The sidecar transparently upgrades the connection to mTLS, presents the pod's SVID, validates the server's SVID, and forwards the traffic. The application code requires no modification. This is the primary UX advantage of the service mesh model — mTLS adoption does not require application changes.

## Linkerd mTLS: Automatic, Zero-Config

Linkerd takes a different philosophy: mTLS should be automatic and require zero configuration from application teams. When you inject the Linkerd proxy into a pod, mTLS is on by default for all pod-to-pod traffic within the mesh. There is no PERMISSIVE mode — either both sides are meshed (mTLS) or the connection falls through as plaintext.

Linkerd uses its own CA (built into the control plane) to issue certificates. Certificate rotation happens automatically, opaquely, without operator intervention. The operational overhead is significantly lower than Istio's more configurable (and therefore more complex) certificate management.

The trade-off:

| Dimension | Istio | Linkerd |
|-----------|-------|---------|
| mTLS default | PERMISSIVE (configurable to STRICT) | Automatic for all meshed traffic |
| Certificate management | Pluggable (can use SPIRE as external CA) | Linkerd-managed CA (simpler, less flexible) |
| SPIFFE integration | First-class SPIFFE/SPIRE support | Limited SPIFFE support |
| Configuration surface | Rich (AuthorizationPolicy, PeerAuthentication, etc.) | Minimal by design |
| Operational complexity | Higher | Lower |

For environments that already use SPIRE for workload identity (because Vault uses SPIFFE auth, for example), Istio's ability to use SPIRE as the certificate authority is significant — you have a single workload identity system, not two separate CA infrastructures.

## Debugging mTLS Failures

mTLS failures are common during initial rollout and during cert rotation events. The failure modes are distinct enough that each has a different debugging path.

**Cert expiry**: Connection resets that appear suddenly across multiple services simultaneously. Check cert expiry:
```bash
istioctl proxy-status | grep -v SYNCED
# shows proxies that are out of sync — often a signal of cert issues

openssl s_client -connect <service>:443 -showcerts 2>/dev/null | openssl x509 -noout -dates
# check the NotAfter field
```

**Trust domain mismatch**: Two services issued certificates from different CAs (or different SPIRE trust domains) cannot validate each other's certificates. The connection fails during the handshake with a certificate validation error. Symptom: specific pairs of services fail mTLS while others work fine.

**PERMISSIVE mode drift**: A service was configured STRICT, a new caller (non-meshed) is added, and the caller cannot connect. The caller needs to be meshed, or the receiving service needs PERMISSIVE mode temporarily.

**Clock skew**: X.509 certificates have NotBefore and NotAfter fields. If the system clock on a node drifts significantly, a freshly issued certificate may appear "not yet valid" to a peer whose clock is ahead, or "already expired" to a peer whose clock is behind. In Kubernetes, NTP synchronization on nodes is a prerequisite for reliable mTLS.

Splunk query to detect mTLS failures (Istio access logs):
```
index=istio sourcetype=istio_access_log
| search response_flags="UAEX" OR response_flags="DC"
| stats count by source_workload, destination_workload, response_flags
| sort -count
```

`UAEX` = upstream connection failed authentication (mTLS failure from upstream). `DC` = downstream connection disconnect. High counts on either flag during a cert rotation event indicate rotation automation failure.

## The Director Frame: mTLS Enforcement Is a Journey

The mistake most platform teams make is treating mTLS enforcement as a binary switch: "we're going to enable STRICT mode cluster-wide by Q3." Then Q3 arrives and they discover:

- 30% of services have callers that are not meshed (batch jobs, monitoring agents, legacy services)
- CI/CD pipelines that run health checks from outside the mesh fail when STRICT mode is enabled
- Cert rotation automation was never tested under load — it breaks at scale

The right framing for a Director interview: mTLS enforcement is a journey with three phases, each of which delivers incremental risk reduction:

1. **Mesh coverage**: every workload has a sidecar; mTLS is PERMISSIVE everywhere. No enforcement yet, but all traffic is visible and encrypted when the peer is meshed. Cert infrastructure is tested.

2. **Namespace-by-namespace STRICT rollout**: start with the highest-risk namespaces (payment processing, auth services, secrets management). Validate that all callers are meshed before flipping to STRICT. Build the runbook for STRICT → PERMISSIVE rollback.

3. **Cluster-wide STRICT and policy layer**: all namespaces in STRICT mode. AuthorizationPolicy deployed to restrict specific service-to-service calls. mTLS enforcement is now the foundation on which L7 policy runs.

You will spend more engineering time on cert lifecycle, caller inventory, and automation testing than on the Istio configuration itself. Budget accordingly, and plan the journey, not the destination.
