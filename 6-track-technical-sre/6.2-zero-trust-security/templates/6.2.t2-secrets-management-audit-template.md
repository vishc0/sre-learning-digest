## Part 3: Vault Policy Review

Review every Vault policy that grants access to production paths. Apply least-privilege test to each.

| # | Policy Name | Attached To (Role / Entity) | Paths Granted | Operations Granted | Least-Privilege Check | Finding |
|---|------------|---------------------------|--------------|-------------------|----------------------|---------|
| 1 | | | | read / list / create / update / delete / sudo | PASS / FAIL | |
| 2 | | | | | PASS / FAIL | |
| 3 | | | | | PASS / FAIL | |
| 4 | | | | | PASS / FAIL | |
| 5 | | | | | PASS / FAIL | |

**Least-privilege check criteria:**
- FAIL if `sudo` is granted on any path that does not require it
- FAIL if `delete` or `update` is granted when only `read` is needed
- FAIL if the path includes a wildcard (`*`) where a specific path would suffice
- FAIL if the policy has not been reviewed in > 12 months
- FAIL if the policy is attached to a service account / role that no longer exists

**Vault policy audit SPL query for evidence collection:**
```splunk
index=vault sourcetype=vault_audit
| stats dc(request_path) as unique_paths, count as total_ops
  by auth.metadata.service_account_name
| sort -total_ops
```

| 3 | | | | | Y / N | |

**Kubernetes Secret audit command:**
```bash
# List all Opaque secrets across all namespaces
kubectl get secrets --all-namespaces --field-selector type=Opaque \
  -o custom-columns="NAMESPACE:.metadata.namespace,NAME:.metadata.name,CREATED:.metadata.creationTimestamp"
```

Secrets with `creationTimestamp` > 90 days ago and no rotation evidence should be treated as long-lived credentials and listed in Part 2.
