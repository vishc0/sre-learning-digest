# Chapter 2: Identity as the New Perimeter

## Why IP Addresses Cannot Be the Identity Anchor

In a traditional data center, IP addresses were a reasonable proxy for identity. Server A at 10.1.2.50 was always Server A. It had a hostname, a fixed IP, a configuration that lived on a physical machine. You could build firewall rules around IP addresses because the IP-to-workload mapping was stable.

Kubernetes broke this model completely and deliberately.

In a Kubernetes cluster, a pod's IP address is ephemeral. When a pod is rescheduled — because a node failed, because a deployment rolled out a new version, because HPA scaled the replica count — it gets a new IP address. The IP is not the workload. The IP is a temporary routing detail assigned to a particular instance of a workload at a particular point in time.

More critically, IP addresses in multi-tenant clusters give you no signal about workload identity. Two pods on the same node with consecutive IP addresses could be running completely different services owned by different teams with different security requirements. The IP tells you where a packet is coming from. It tells you nothing about what software is running, which service account is in use, or what that workload is authorized to do.

The shift to cloud-native is a shift from infrastructure identity (IP, hostname, MAC) to workload identity (cryptographic assertions about what the workload is and what it is authorized to do). This shift is the architectural foundation of Zero Trust in Kubernetes environments.

## SPIFFE: The Standard for Workload Identity

SPIFFE (Secure Production Identity Framework For Everyone) is the CNCF standard that defines how workloads prove their identity. It emerged from the recognition that every organization was solving the same problem independently — how does Service A prove to Service B that it is who it claims to be, without relying on network location?

SPIFFE defines two things:

**1. The SPIFFE ID** — a URI that uniquely identifies a workload:

```
spiffe://<trust-domain>/<workload-path>
```

Real-world examples in a T-Mobile context:
```
spiffe://prod.tmobile.com/ns/payments/sa/checkout-service
spiffe://prod.tmobile.com/ns/network/sa/provisioning-api
spiffe://staging.tmobile.com/ns/auth/sa/token-service
```

The trust domain is your organizational boundary (analogous to a DNS domain for identity). The path encodes the workload's namespace and service account. The key property: this ID is embedded in a certificate, not derived from a network address. When the pod moves, the SPIFFE ID stays the same.

**2. The SVID (SPIFFE Verifiable Identity Document)** — the cryptographic document that carries the SPIFFE ID. Two formats:
- **X.509-SVID**: an X.509 certificate with the SPIFFE ID in the Subject Alternative Name field — used for mTLS
- **JWT-SVID**: a JSON Web Token carrying the SPIFFE ID as a claim — used for HTTP/API-layer authentication

## SPIRE: The Runtime That Issues and Manages SVIDs

SPIRE (SPIFFE Runtime Environment) is the reference implementation of the SPIFFE standard. It is the operational machinery that issues SVIDs to workloads and rotates them automatically.

SPIRE architecture:

```
[SPIRE Server]
     |
     | (node attestation — is this a legitimate Kubernetes node?)
     v
[SPIRE Agent] (runs as DaemonSet on each node)
     |
     | (workload attestation — is this a legitimate workload?)
     v
[Workload API] (Unix domain socket)
     |
     | (SVID delivery)
     v
[Your Workload / Sidecar]
```

**Node attestation**: The SPIRE Agent proves to the SPIRE Server that the node it's running on is a legitimate cluster node. In Kubernetes, this uses the `k8s_psat` (Projected Service Account Token) plugin — the agent presents a node-bound service account token that the server validates against the Kubernetes API. In AWS, this uses `aws_iid` (Instance Identity Document). The node's identity is verified before any workload on it can receive an SVID.

**Workload attestation**: The SPIRE Agent interrogates the workload requesting an SVID. For Kubernetes workloads, it validates the pod's service account, namespace, and labels against the Kubernetes API. The agent confirms the workload is who it claims to be before issuing any credentials.

The critical operational property: workloads never handle long-lived credentials. SVIDs are short-lived (typically 1-24 hours) and automatically rotated by SPIRE before expiry. There are no API keys, no static passwords, no secrets that need to be distributed to pods. The identity infrastructure handles credential lifecycle automatically.

## OIDC for Machines: Kubernetes to Vault

Kubernetes service accounts can issue OIDC (OpenID Connect) tokens — signed JWTs that assert the identity of a pod. This is the mechanism that enables Vault's Kubernetes auth method, and it is the same trust model SPIFFE/SPIRE formalizes at scale.

The flow:
1. A pod presents its Kubernetes service account JWT to Vault
2. Vault calls the Kubernetes API to verify the token is valid and belongs to the expected service account
3. Vault maps the service account to a Vault policy and issues a short-lived Vault token
4. The pod uses the Vault token to read secrets — the secrets it is explicitly authorized to read, nothing more

This flow is Zero Trust applied to secrets: the workload's identity (service account) is cryptographically asserted, verified against an authoritative source (Kubernetes API), and used to derive time-bounded access (Vault token with a TTL). No static passwords. No shared credentials. Every secret access is attributable to a specific workload identity at a specific time.

## Certificate-Based Identity vs. IP-Based Trust

The fundamental shift Zero Trust requires in networking:

| Model | Trust Anchor | Failure Mode | ZT Pillar |
|-------|-------------|-------------|-----------|
| IP-based | Network location | Lateral movement after perimeter breach | None (legacy) |
| Certificate-based | Cryptographic identity | Certificate misissuance or compromise | Identity + Devices |
| SPIFFE/SVID | Attested workload identity | Trust domain compromise | Identity + verified by attestation |

Certificate-based identity solves the IP problem: the certificate travels with the workload, not with the network address. When Service B receives a TLS connection from Service A, it validates Service A's certificate — the certificate contains Service A's SPIFFE ID, which is cryptographically bound to Service A's identity by the CA (SPIRE or Istio's built-in CA). Service B can verify not just that the connection is encrypted, but that the entity on the other end is specifically the checkout service in the payments namespace, not any other workload that happens to have a routable IP.

## The Director Frame: Define "Identity" for Every Actor in Your Platform

The most important architectural decision in a Zero Trust program is also the most often deferred: defining what "identity" means for every actor in your platform.

Your platform has at least four classes of actors that need identity:
- **Human users**: engineers, operators, on-call responders
- **Machine workloads**: services, jobs, operators, controllers
- **CI/CD pipelines**: build agents, deployment tools, test runners
- **External integrations**: third-party APIs, partners, SaaS webhooks

Most organizations have solved human identity (SSO, MFA, directory services). Almost no organization has fully solved machine identity. Machine identity is where the real Zero Trust debt lives — static API keys, long-lived service account tokens, shared credentials passed via environment variables.

Your job as a Director is to define the identity architecture for all four classes, then phase the remediation starting with the highest-risk class (typically machine workloads with production access). The tooling follows the architecture. SPIFFE/SPIRE, Vault Kubernetes auth, Istio's internal CA — these are all implementations of a single decision: we will use cryptographic workload identity, not IP addresses or static credentials, as the trust anchor for service-to-service communication.

That decision, and the organizational will to enforce it, is what a Zero Trust program actually is.
