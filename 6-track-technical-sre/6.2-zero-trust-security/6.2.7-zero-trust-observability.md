# Chapter 7: Zero Trust Observability

## Why Observability Is Not Optional

In a traditional security model, observability is a detective control — you collect logs to investigate incidents after they happen. In a Zero Trust architecture, observability serves a different and more critical function: it is how you prove that your controls are working.

Consider the alternative. You have deployed mTLS across your cluster, written AuthorizationPolicy rules, enabled OPA Gatekeeper, and migrated your highest-risk services to Vault dynamic secrets. Your CISO asks: "How do we know the controls are actually enforcing what we think they're enforcing?"

Without observability, the honest answer is: "We don't." A misconfigured PERMISSIVE mode policy that never switched to STRICT looks identical to a correctly configured STRICT policy — until an attacker exploits the gap. An OPA rule that has a logic error allows what it should deny — and without decision logging, you never know.

The audit trail is the proof. Every access decision — allow or deny, human or machine, at the network layer or the application layer — must be attributable to an identity, a time, and a policy. This is the audit requirement. Not "we log errors" — every decision.

---

## What to Log in a Zero Trust Environment

The logging surface in a ZT architecture is broader than in a traditional architecture because there are more control points generating decisions. You need logs from all of them:

**Identity and authentication events:**
- SPIRE SVID issuance and rotation (workload attestation events, new trust relationship grants)
- Kubernetes service account token creation and usage
- OIDC token issuance (Kubernetes → Vault)
- Vault: every token creation, renewal, and revocation; every authentication attempt (success and failure)
- CyberArk: session initiation, session recording events, privileged account checkout

**Policy enforcement decisions:**
- Istio: every request — source workload, destination workload, response code, response flags, duration
- OPA/Gatekeeper: admission webhook decisions — resource type, allow/deny, which rule triggered
- Istio AuthorizationPolicy: explicit deny events (RBAC deny response flags)
- NetworkPolicy: unfortunately, Kubernetes NetworkPolicy does not generate audit logs by default — you need Cilium's Hubble or a similar layer for dropped packet visibility

**Certificate lifecycle events:**
- Certificate issuance (new SVID issued to a workload)
- Certificate rotation (SVID renewed before expiry)
- Certificate expiry warnings (approaching the NotAfter date)
- Certificate validation failures (mTLS handshake failures)

**Secrets access:**
- Vault audit log: every secret read, every policy evaluation, every lease creation
- Which service account accessed which secret path, at what time, from what IP

## Detecting Anomalies in Access Patterns

Zero Trust observability is not just about recording what happened. It is about detecting when what happened deviates from what is expected. The access pattern anomalies that matter most:

**New source principal accessing a service**: Service B has never been called by the `analytics` namespace before. Today it is. Is this a legitimate new integration that was not communicated to the security team, or is it lateral movement from a compromised analytics workload?

**Unusual access times**: A batch job that normally runs at 2 AM starts making requests at 11 PM. Could be a schedule change. Could be a compromised credential being used from a different timezone.

**Cross-namespace calls that weren't there yesterday**: The service dependency map changes. A service in `payments` is now receiving calls from a namespace it never received calls from before. Policy allows it (the policy was misconfigured), but the behavior is anomalous.

**High-volume secret access**: A service account that normally reads 5 secrets per hour is suddenly reading 500 per hour. Could be a misconfigured retry loop. Could be credential harvesting.

**Privilege escalation attempts**: A Vault policy evaluation that results in a "permission denied" on a path that is significantly more privileged than the service account's normal access pattern — especially repeated failures.

These are the signals that separate a Zero Trust implementation from a Zero Trust security program. The controls are the implementation. Detecting and responding to anomalies is the program.

## Splunk for Zero Trust Event Analysis

Splunk is the right tool for ZT event correlation because the events span multiple systems (Istio, SPIRE, Vault, CyberArk, Kubernetes) and the correlations that matter require joining across those systems. A Splunk lookup that joins a Vault audit event to a Kubernetes namespace label to a CyberArk session record requires the multi-source join capability that dedicated security tools often lack.

### Vault Audit Log Analysis

Vault's audit log (enable with `vault audit enable file file_path=/var/log/vault/audit.log`) generates a JSON event for every operation. In Splunk:

```splunk
index=vault sourcetype=vault_audit
| spath input=auth.metadata output=metadata
| stats count by auth.metadata.service_account_name, path
| where count > 100
| sort -count
```
Purpose: find service accounts reading secrets at unusually high volume. A service account making 1000 secret reads/hour may have a broken retry loop — or may be exfiltrating credentials.

```splunk
index=vault sourcetype=vault_audit
| search error!=""
| stats count by auth.metadata.service_account_name, path, error
| sort -count
```
Purpose: find permission denied patterns — service accounts attempting to access paths they are not authorized for.

```splunk
index=vault sourcetype=vault_audit
| search type=response request_path="/sys/leases/revoke*"
| stats count by auth.display_name, request_path
| sort -count
```
Purpose: track manual lease revocations — unexpected revocations may indicate an incident response (or an incident).

### Istio Access Log Analysis

Istio's Envoy sidecar generates a structured access log for every request. Key fields to parse: `response_code`, `response_flags`, `upstream_cluster`, `source_workload`, `destination_workload`, `x-forwarded-client-cert`.

```splunk
index=istio sourcetype=istio_access_log
| search response_flags="UAEX"
| stats count by source_workload, destination_workload
| sort -count
```
Purpose: `UAEX` (Unauthorized External Service) indicates a request rejected by Istio's RBAC (AuthorizationPolicy). High counts point to either a policy misconfiguration or a workload attempting unauthorized access.

```splunk
index=istio sourcetype=istio_access_log
| search response_flags="DC" OR response_flags="UC"
| timechart span=5m count by destination_workload
```
Purpose: `DC` (downstream connection disconnect) and `UC` (upstream connection failure) often spike during cert rotation events or PERMISSIVE→STRICT transitions. A sudden spike across multiple destination workloads signals a cert lifecycle incident.

```splunk
index=istio sourcetype=istio_access_log
| rex field=x-forwarded-client-cert "URI=(?<client_spiffe_id>[^;]+)"
| stats dc(client_spiffe_id) as unique_callers by destination_workload
| where unique_callers > 10
```
Purpose: identify services being called by an unusually high number of distinct workloads — unexpected callers may indicate lateral movement or a misconfigured policy that should be more restrictive.

### SPIRE Audit Events

SPIRE generates audit events for workload attestation and SVID issuance. In Splunk:

```splunk
index=spire sourcetype=spire_audit
| search event_type=svid_issued
| stats count by spiffe_id, node_id
| where count > 50
```
Purpose: SVIDs are typically issued once per pod startup. High issuance counts for a single SPIFFE ID may indicate pod crash-looping (and therefore repeated attestation) or a misconfigured SPIRE entry causing repeated re-attestation.

```splunk
index=spire sourcetype=spire_audit
| search event_type=attestation_failed
| stats count by node_id, error
| sort -count
```
Purpose: failed node attestation events should be near zero. A non-zero count indicates either misconfigured attestation plugins or a node attempting to join the cluster without proper credentials.

### Certificate Expiry Monitoring

```splunk
index=istio sourcetype=istio_proxy_config
| eval expiry_epoch=strptime(cert_expiry, "%Y-%m-%dT%H:%M:%SZ")
| eval days_until_expiry=round((expiry_epoch - now()) / 86400, 1)
| where days_until_expiry < 7
| table workload, namespace, cert_expiry, days_until_expiry
| sort days_until_expiry
```
Purpose: identify workloads with certs expiring within 7 days. This should fire as a scheduled alert — not something you check reactively.

## The Audit Trail Requirement

In regulated industries (PCI-DSS v4.0, HIPAA, FedRAMP), the audit requirement is not "we log errors." It is: every access to in-scope data or systems must produce an immutable audit record that includes who, what, when, from where, and the outcome.

In a Zero Trust architecture, meeting this requirement is structurally built into the design:
- Every secret access: Vault audit log
- Every mTLS connection: Istio access log with SPIFFE client cert
- Every privileged human session: CyberArk session recording + event log
- Every policy evaluation: OPA decision log
- Every Kubernetes API mutation: Kubernetes audit log

The key operational requirement is that these logs are shipped to a centralized, immutable store (Splunk with index-time MAC, or a SIEM with WORM storage) and are retained for the required period. Logs that live only in the originating system can be tampered with or lost in a node failure.

## The Director Frame: Logs Are the Security Control

The single most important observability insight for a ZT Director interview:

**Zero Trust without observability is security theater.**

You can deploy all the right components — mTLS, OPA, SPIFFE, Vault — and still have a broken security posture if you cannot query what those components are doing. The logs are not supplemental to the controls. The logs ARE the control. Without them, you cannot:

- Prove that STRICT mode is enforced everywhere it should be
- Detect a policy misconfiguration before it is exploited
- Respond to an access anomaly in time to limit damage
- Demonstrate compliance to an auditor
- Measure the effectiveness of your ZT investments to your CISO and board

The Director-level commitment: observability runs from day one of any ZT initiative, not after the controls are deployed. The first thing you build is the logging pipeline and the baseline access pattern model. The controls are instrumented as they are deployed. By the time you enable STRICT mode in any namespace, you already know exactly what the expected access patterns look like — so deviations are immediately visible.
