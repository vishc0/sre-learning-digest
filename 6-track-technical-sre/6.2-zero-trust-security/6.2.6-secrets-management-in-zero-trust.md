# Chapter 6: Secrets Management in Zero Trust

## Building on What You Already Have

You have already implemented Vault and CyberArk in production. You understand the operational realities of both: Vault's Kubernetes auth method, lease renewal, dynamic database credentials, the audit log. CyberArk's safe model, session recording, privileged access workflows, the integration with AD. This chapter extends that operational experience into Zero Trust architecture — specifically, how secrets management fits into the broader ZT posture and where the gaps typically exist even in organizations that "have Vault."

The most common finding in a Zero Trust secrets audit at an organization that already runs Vault is not that Vault is misconfigured — it is that Vault is underutilized. Services exist that bypass Vault entirely, using environment variables with static credentials. CI/CD pipelines pass secrets via plaintext environment variables that end up in pod specs. Database passwords that were supposed to be dynamic are static because the rotation job never got built. The capability is present; the coverage is incomplete.

Your job as a Director is to close that gap systematically.

## The Zero Trust Secrets Principle

The Zero Trust principles applied to secrets management yield a clear set of requirements:

**No long-lived credentials**: Any credential that does not expire is a standing liability. A compromised long-lived credential can be used indefinitely. Database passwords, API keys, service account tokens, and SSH keys should all have TTLs — not years, but hours or days.

**No static secrets**: Secrets should not be pre-provisioned and stored. They should be generated on-demand for a specific workload, for a specific purpose, with a specific TTL. Vault dynamic secrets is the implementation of this principle.

**No secrets in environment variables**: Environment variables are logged, exported to child processes, visible in `/proc/environ`, and often captured in crash reports. They are a high-exposure channel for secrets. The Zero Trust requirement is that secrets are loaded at runtime from a secure store and never materialized as environment variables.

**No secrets in container images**: Baking credentials into images is the secrets equivalent of hardcoding passwords. Images are distributed, cached, and often stored in registries for months. Any credential in an image is a standing exposure.

## Vault Kubernetes Integration: Two Patterns

You have already operated Vault's Kubernetes auth method. In the Zero Trust architecture, there are two patterns for delivering secrets from Vault to pods, each with distinct trade-offs:

### Pattern 1: Agent Sidecar Injection

Vault Agent runs as a sidecar container alongside your application. The agent authenticates to Vault using the pod's service account token (Kubernetes auth method), retrieves secrets, writes them to a shared in-memory volume, and handles lease renewal automatically.

```
[Pod]
  ├── [App Container] ← reads /vault/secrets/config
  └── [Vault Agent Sidecar] → authenticates to Vault → writes to shared tmpfs
```

Advantages:
- Strong isolation: the app container never talks to Vault directly; the agent mediates all secret access
- Automatic renewal: the agent handles token and lease renewal without app code changes
- Template rendering: Vault Agent can render secrets into configuration file formats (config.yaml, .env-style files) using templates
- Works with any application — no SDK required in the app

Limitations:
- Additional container per pod (resource overhead, scheduling complexity)
- Sidecar failure can cascade to app failure
- Secret delivery is file-based (shared volume) — not the native Kubernetes secret lifecycle

### Pattern 2: CSI Secrets Store Driver

The Secrets Store CSI Driver integrates with the Kubernetes storage layer to mount secrets from Vault (and other providers) as read-only volumes on pods. The SecretProviderClass custom resource defines what secrets to fetch and from where.

```
[Pod]
  └── [App Container] ← mounts /mnt/secrets as a volume
                              ↑
                    [CSI Driver] → fetches from Vault on pod startup
```

Advantages:
- Native Kubernetes lifecycle: secrets mount and unmount with the pod
- No sidecar: lower resource overhead
- Kubernetes Secrets sync: the driver can create Kubernetes Secret objects for applications that require them
- Works with multiple secret providers (Vault, AWS Secrets Manager, Azure Key Vault) through the same interface

Limitations:
- Secrets are loaded at pod startup; for rotation without pod restart, you need the rotation-reconciler feature enabled
- Less isolation than the sidecar pattern — the CSI driver runs with elevated privileges on the node

**When to choose which**:
- High security, regulated environments (PCI, HIPAA scope): agent sidecar for maximum isolation and auditability
- General platform workloads where developer experience matters: CSI driver for native Kubernetes integration
- Greenfield deployments with no legacy constraints: CSI driver as the default, sidecar for specific high-sensitivity services

## Dynamic Secrets: Zero Trust Applied to Credentials

Dynamic secrets are Vault's implementation of "no long-lived credentials" for database access. Instead of a static database user/password that all instances of a service share, Vault generates a unique credential pair for each request:

1. Checkout service pod starts up, authenticates to Vault via Kubernetes auth
2. Vault generates a new PostgreSQL user and password, grants the user access to the `payments` database with a specific role
3. Vault issues the credentials to the pod with a TTL (e.g., 1 hour)
4. The pod uses the credentials to connect to the database
5. When the pod terminates, Vault revokes the credentials — the PostgreSQL user is deleted

The security properties:
- Every pod instance has unique credentials — a stolen credential from one pod cannot be used by another
- Credentials expire automatically — a compromised credential is valid only until the TTL
- Revocation is immediate and automatic — no rotation job to write, no coordination required
- Every credential issuance is logged in the Vault audit log — full traceability

The operational complexity: the application must handle credential rotation gracefully. If the pod is long-running (beyond the initial TTL), it must use Vault Agent's lease renewal capability to renew the credential before expiry. Applications that open a single database connection at startup and hold it indefinitely will fail when the credential expires. Connection pooling libraries that support credential refresh callbacks (PgBouncer with auth_query, Java's HikariCP) handle this gracefully.

## CyberArk for Privileged Access: The Human-Facing Complement

Vault is the machine-facing secrets infrastructure — it handles workload identity, dynamic credentials, and programmatic secret access. CyberArk is the human-facing PAM (Privileged Access Management) layer — it handles privileged human access to production systems.

In a Zero Trust architecture, the two are complementary:

```
[Machine Access]          [Human Access]
  └── Vault               └── CyberArk
       ├── Dynamic DB creds     ├── SSH session to prod bastion
       ├── API keys (dynamic)   ├── RDP to Windows servers
       ├── Cloud credentials    ├── Database DBA access
       └── PKI certificates     └── Privileged AD accounts
```

CyberArk enforces:
- **No standing access**: privileged access is requested, approved (just-in-time), and used — no standing admin accounts
- **Session isolation**: privileged sessions go through CyberArk's proxy, recorded for audit
- **Credential vaulting**: privileged account passwords are managed by CyberArk, rotated on schedule, never revealed to the human (they proxy the session instead)
- **Dual control**: high-risk operations can require a second approver before the session is granted

The Zero Trust integration point: CyberArk sessions should be logged to Splunk alongside Vault audit events. A privileged access session to production that is not preceded by a Jira incident ticket or a change record is an anomaly worth investigating.

## Secret Rotation: The Operational Flywheel

Rotation is the operational process of replacing a credential before it expires, without causing downtime. Vault makes machine secret rotation largely automatic for dynamic secrets. For static secrets (where dynamic generation is not feasible — some third-party APIs, for example), you need a rotation process.

The rotation pattern that prevents downtime:

1. **Version the secret**: Vault's KV v2 stores multiple versions. Write the new value as a new version.
2. **Canary deploy**: a small percentage of pods pick up the new secret version first. Validate that the new credential works.
3. **Full rotation**: roll out the new secret version to all pods.
4. **Revoke the old version**: once all pods have migrated, revoke or delete the old version.

The failure mode to avoid: simultaneous rotation across all instances at once. If the new credential is invalid (wrong permissions, wrong format), you take down every instance simultaneously. The canary approach bounds the blast radius of a rotation failure.

## The Director Frame: Close the Gap Between Capability and Coverage

Every organization that runs Vault at scale has the same gap: the capability exists, but full coverage does not. The gap is filled with:

- Old services that predate Vault adoption and still use static credentials
- New services that were built quickly and "will migrate to Vault later"
- CI/CD secrets that never got migrated from the CI platform's built-in secret store
- Kubernetes Secrets objects that duplicate credentials that Vault already manages

Your job as a Director is to close that gap systematically, prioritizing by risk:

| Priority | Secret Type | Why First |
|----------|------------|-----------|
| P1 | Database credentials with write access to production data | Maximum impact if compromised |
| P1 | Cloud provider credentials (AWS IAM keys) with production access | Blast radius = entire cloud account |
| P2 | API keys for payment or auth providers | Regulatory and financial exposure |
| P2 | SSH keys with production server access | Persistent privileged access |
| P3 | Read-only API keys, internal service tokens | Lower blast radius |

Build the inventory first (the secrets audit template in this framework helps). Measure coverage (what percentage of services use dynamic or short-lived credentials). Set a target and a timeline. The target is not "100% by Q4" — that is unrealistic in a running production environment. The target is "highest-risk credentials have < 24-hour TTL by Q4, all production database credentials are dynamic by end of H1."

Progress against a measured target is what you report to the CISO and CFO. "We have Vault" is not a metric. "87% of our production database credentials are now dynamic with < 1-hour TTL, down from 12% six months ago" is.
