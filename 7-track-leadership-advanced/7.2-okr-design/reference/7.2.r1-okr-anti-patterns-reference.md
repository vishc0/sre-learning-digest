# OKR Anti-Patterns Reference

**Audience**: Director/VP Platform Engineering interviews and team OKR review sessions.
**How to use**: Run the Quick Reference table first to identify which pattern you are looking at. Then go to the full entry for the correction. In interviews, knowing the name of the pattern and the corrected version is the Director-level answer — not just "that KR is bad."

## Anti-Pattern 1: The Milestone KR

**Description**: A project task or deliverable is written as a Key Result. Milestones are binary — done or not done — and measure output, not outcome. They answer "did we ship it?" not "did it work?" The team can hit 1.0 on a milestone KR while the underlying objective fails completely. This is the most common OKR failure mode in engineering organizations, because sprint planning naturally produces milestones, and those milestones get copied into OKR documents.

**Bad Example**:

```
Objective: Improve platform reliability for tier-1 services.

KR1: Complete implementation of distributed tracing for the payment service.
KR2: Deploy Prometheus monitoring to all Kubernetes clusters.
KR3: Finish the SLO definition document for all tier-1 services.
```

**Why It Fails**: "Complete," "deploy," and "finish" describe tasks. A team can hit 1.0 on all three and still have worsening reliability — the work was done but the outcome was never measured. At quarter end, the score says "we succeeded" and the incident data says "we didn't." Leadership cannot use the score to make decisions about the team or the platform. In an interview, presenting milestone KRs signals that you have not internalized the distinction between project delivery and outcome accountability.

**Corrected Version**:

```
Objective: Improve platform reliability for tier-1 services.

KR1: Reduce mean time to detect (MTTD) payment service failures from 8 minutes to ≤3 minutes
     (Baseline: PagerDuty incident data, last 90 days. Measurement: median MTTD at quarter end.)

KR2: Increase alert-to-incident correlation rate from 62% to ≥90% across all Kubernetes workloads
     (Baseline: Alert audit, Q2. Measurement: Correlation rate from PagerDuty + incident system.)

KR3: Achieve SLO compliance ≥95% for the 5 tier-1 services for 8 consecutive weeks
     (Baseline: Current 30-day compliance from Grafana SLO dashboard. Measurement: Sustained hold.)
```

**SRE application**: "Complete the runbook for service X" is a milestone. "Reduce Sev-1 escalation rate by 40% for service X using the new runbooks" is a KR. The runbook is the mechanism; the escalation rate is the result. Always measure the result.

## Anti-Pattern 2: The SLO-as-OKR

**Description**: An existing SLO or operational commitment is copied into an OKR as a Key Result. SLOs are contractual floors — the minimum acceptable level of service agreed between SRE and the business. OKRs are improvement goals that should push the team beyond their current state. Treating an SLO as an OKR asks the team to do exactly what they are already required to do, then score it. A score of 0.7 on this KR does not mean the team fell short of a stretch goal — it means they violated a service agreement.

**Bad Example**:

```
Objective: Maintain platform reliability for Q3.

KR1: Maintain 99.9% availability for the checkout service (per existing SLO agreement).
KR2: Keep P95 API latency under 500ms (per SLA with the Product organization).
KR3: Ensure no more than 2 Sev-1 incidents per month (per operations charter).
```

**Why It Fails**: All three are operational commitments that existed before the OKR cycle began. A 1.0 score proves only that the team did their job. A 0.5 score indicates an SLA violation — that is not a stretch goal miss, it is a performance failure that should be tracked separately. The KRs produce no information about improvement, no incentive to raise the bar, and no signal about team growth or platform investment.

**Corrected Version**:

```
Objective: Raise the reliability ceiling above current SLO commitments.

KR1: Improve checkout service availability from 99.9% to 99.95% through Q3
     (Baseline: Grafana SLO dashboard, Q2 average. Measurement: Q3 30-day rolling average.)

KR2: Reduce P95 API latency from 420ms to ≤200ms under peak load (5k RPS)
     (Baseline: APM p95 at 5k RPS, Q2. Measurement: Reproducible load test at quarter end.)

KR3: Reduce Sev-1 incident count from 6 per quarter to ≤2 per quarter through proactive
     failure mode elimination (not by tightening the Sev-1 definition)
     (Baseline: Incident log, Q2. Measurement: Incident count Q3, same Sev-1 criteria.)
```

**SRE application**: The 99.9% SLO is the floor you are contractually obligated to stand on. The OKR asks: where do we want to raise the ceiling? Write from the current measured state, not from the contract. The two documents serve different purposes and should never overlap.

## Anti-Pattern 3: The Activity Dump

**Description**: An objective has five, six, or seven-plus Key Results, the majority of which are tasks or project milestones rather than outcomes. This happens when teams translate their sprint backlog into OKR format instead of selecting the outcomes that matter most. It signals that no strategic prioritization happened — every item on the roadmap was included so nothing could be said to have been missed. The scoring system becomes meaningless because a team can score 0.86 by completing six of seven tasks, while the single most important outcome was the one that scored 0.

**Bad Example**:

```
Objective: Improve developer experience on the platform.

KR1: Launch the new developer portal.
KR2: Publish 20 internal documentation articles.
KR3: Run 3 platform office hours sessions.
KR4: Migrate the CLI tool to Go.
KR5: Onboard 5 teams to the new service mesh.
KR6: Update the incident response template.
KR7: Train 10 engineers on the new observability stack.
```

**Why It Fails**: Seven KRs covering seven different domains means the team has not made a choice about what success looks like. "Run 3 office hours" is an event — not a Key Result. "Publish 20 articles" is content creation — not developer experience. A team can score 6 of 7 (0.86 overall) while completely failing the objective because the six completed tasks were the easy ones. The scoring rewards activity, not outcomes.

**Corrected Version**:

```
Objective: Reduce developer friction in platform onboarding and day-to-day operations.

KR1: Reduce median time for a new microservice to reach full observability from 8 days to ≤1 day
     (Baseline: Onboarding ticket duration, ITSM, Q2. Measurement: Median at quarter end.)

KR2: Achieve DevEx survey score ≥4.0/5 on platform usability (from 3.1/5 baseline, n≥100)
     (Baseline: Q2 DevEx survey. Measurement: Q3 survey administered in week 11.)

KR3: Reduce platform-related support tickets from 60 per month to ≤15 per month
     (Baseline: ITSM ticket volume by category, Q2. Measurement: Monthly average in Q3.)
```

**Rule of thumb**: Three KRs per objective is the target. Four is acceptable. Five is a signal to cut. Seven is an activity dump. Force yourself to choose the three outcomes that, if achieved, would make everything else feel secondary.

## Anti-Pattern 4: The 100% Guarantee

**Description**: A target is set at exactly (or very close to) the current performance level, ensuring it will be hit regardless of execution quality. Sandbagging is intentional or habitual conservative commitment — writing goals you are certain to achieve rather than goals that require real effort. It is endemic in organizations where OKR scores feed performance reviews, creating an incentive to commit low and score high. A 1.0 on a sandbagged KR proves only that the team can do what they were already doing.

**Bad Example**:

```
Objective: Improve deployment reliability.

KR1: Achieve deployment success rate of ≥98% (current rate: 97.8%).
KR2: Maintain on-call pager load within existing team norms.
KR3: Complete all planned Q3 releases on schedule.
```

**Why It Fails**: KR1 requires 0.2% improvement — achievable by fixing one known defect. KR2 asks for nothing to change. KR3 is defined by the team's own planning calendar, making it self-referentially trivial. At quarter end, all three can be green and the platform is materially unchanged. No organizational value is created. In a VP review, these OKRs signal a team optimizing for score, not for outcomes.

**Corrected Version**:

```
Objective: Materially advance deployment reliability this quarter.

KR1: Increase deployment success rate from 97.8% to ≥99.5% by eliminating the top 3
     failure causes documented in the Q2 retrospective.
     (Baseline: CI/CD pipeline data, Q2. Measurement: Q3 average from pipeline telemetry.)

KR2: Reduce on-call pages per engineer per week from 94 to ≤30.
     (Baseline: PagerDuty data, Q2 average. Measurement: Q3 weekly average per engineer.)

KR3: Achieve change failure rate ≤3% (from 8%) with automated rollback covering ≥80% of services.
     (Baseline: DORA metrics, Q2. Measurement: Change failure rate + rollback coverage at quarter end.)
```

**Calibration check**: At the time of writing, what is the team's confidence in hitting this KR? For a good stretch KR, the answer should be 50–85%. If confidence is above 90%, the target is likely sandbagged. Push the target until confidence drops to 70%.

## Anti-Pattern 5: The Moon Shot

**Description**: A target is set so far beyond any realistic state of the world that the team cannot make meaningful progress toward it within the quarter. Unlike a stretch goal — which is hard but achievable with focused effort — a moon shot is disconnected from current capability, team size, technical debt, or available tooling. It demoralizes rather than motivates. When engineers look at a moon shot OKR, they rationally disengage — they know the score will be 0.2 before the quarter begins, and effort feels futile.

**Bad Example**:

```
Objective: Achieve world-class SRE maturity across the platform.

KR1: Achieve 99.999% availability across all 1,200 microservices by Q3 end.
KR2: Reduce mean MTTR to ≤2 minutes for all incident categories.
KR3: Eliminate all toil from SRE workflows by end of quarter.
```

**Why It Fails**: 99.999% for 1,200 services is years of architectural investment, not one quarter. MTTR ≤2 minutes for all incident types is physically constrained by detection time, human coordination, and diagnostic complexity — many incident types cannot biologically be resolved in 2 minutes. "Eliminate all toil" is infinite in scope and undefined in measurement. Teams faced with these targets rationally set them aside and work on real deliverables, generating a 0.2 score that tells leadership nothing.

**Corrected Version**:

```
Objective: Raise the reliability floor for our highest-risk services this quarter.

KR1: Achieve 99.95% availability for the 15 defined tier-1 services (from 99.7% baseline).
     (Baseline: Grafana SLO dashboard, Q2. Measurement: 30-day rolling average at quarter end.)

KR2: Reduce Sev-1 MTTR from 38 minutes to ≤20 minutes for the top 5 incident types by volume.
     (Baseline: Incident system MTTR, top 5 types by Q2 count. Measurement: Median MTTR at quarter end.)

KR3: Eliminate toil from the top 3 manual workflows by hour volume — verified by 70% hour reduction.
     (Baseline: Time audit week 1 of quarter. Measurement: Time audit week 12, same 3 workflows.)
```

**The tell**: A moon shot OKR is identified when the team's immediate reaction is "is this serious?" A genuine stretch goal generates problem-solving energy and uncomfortable but focused discussion about what would have to be true to hit it. A moon shot generates a resigned shrug.

## Anti-Pattern 6: The Vanity Metric

**Description**: A Key Result is anchored to a metric that looks impressive in a quarterly report but does not correlate to real outcomes for users, customers, or the engineering system. Vanity metrics are typically easy to improve without doing meaningful work — and they can improve while the actual underlying condition worsens. They create false confidence and misallocate team effort toward numbers that look good on slides.

**Bad Example**:

```
KR: Increase the number of internal documentation articles from 45 to 200 by end of Q3.
KR: Achieve 500 developer registrations on the internal developer portal.
KR: Run 12 platform demos to product engineering teams across the quarter.
```

**Why It Fails**: 200 articles does not mean engineers can find answers faster or with less SRE assistance — it means the team wrote 155 articles, which may or may not be read, accurate, or useful. 500 registrations does not mean the platform is being actively used or that self-service is happening. 12 demos does not mean adoption occurred — a team can receive 12 demos and still open 60 SRE tickets per month. All three metrics can be improved by actions that have no relationship to the actual outcomes they imply.

**Corrected Version**:

```
KR: Reduce median time-to-answer for platform how-to questions from 4 hours to ≤30 minutes,
    measured via Slack thread response time in #platform-help.
    (Baseline: Thread timestamp audit, last 90 days. Measurement: Median at quarter end.)

KR: Achieve ≥60% of registered developers completing at least one self-service workflow per quarter
    without opening an SRE ticket.
    (Baseline: Portal usage + SRE ticket correlation, Q2. Measurement: Quarterly audit.)

KR: Achieve platform Net Promoter Score ≥40 on the quarterly DevEx survey (from current 18).
    (Baseline: Q2 DevEx survey. Measurement: Q3 survey, minimum 80 respondents.)
```

**Test for vanity**: Ask "can this metric improve while the underlying problem gets worse?" If the answer is yes, the metric is vanity. Article count can increase while engineer self-sufficiency decreases. Registrations can increase while active usage decreases. A good KR fails this test — it cannot be improved without the real condition improving.

## Anti-Pattern 7: The Invisible Measure

**Description**: A Key Result specifies a direction or aspiration but provides no data source, no measurement method, and no baseline. The outcome cannot be verified at quarter end because nobody defined how to measure it at the beginning. At scoring time, invisible measures become subjective assessments that different stakeholders score differently, or they become whatever the team reports about themselves. Accountability is structurally impossible.

**Bad Example**:

```
KR: Improve system resilience through better failure mode coverage.
KR: Reduce operational complexity of the platform architecture.
KR: Increase reliability confidence across the engineering organization.
```

**Why It Fails**: "System resilience," "operational complexity," and "reliability confidence" are real things that matter — but none of them have a measurement method, a baseline, or a data source attached. At scoring time, these become conversations rather than calculations. One VP scores it 0.8 because "a lot of work happened." Another scores it 0.3 because "we still had 6 incidents." Neither answer is defensible. The KR is unverifiable and therefore unaccountable.

**Corrected Version** (the measurement method must be defined at authoring time, not at scoring time):

```
KR: Increase failure mode test coverage from 34% to ≥80% — measured as (GameDay scenarios passed)
    divided by (known failure modes in the documented failure mode register).
    (Baseline: Current register has 47 modes; 16 covered by GameDay scenarios. Measurement: Ratio at Q3 end.)

KR: Reduce mean configuration change cycle time from 6 days to ≤1 day — measured from Jira ticket
    open to confirmed production deployment.
    (Baseline: Last 20 config change tickets, Jira timestamp data. Measurement: Median at quarter end.)

KR: Achieve ≥80% of engineers reporting "high confidence in production reliability" in the
    quarterly engineering survey (n≥45, anonymous).
    (Baseline: Establish baseline in week 1 survey. Measurement: Q3 survey, week 11.)
```

**Rule**: If you cannot name the system that will produce the score at quarter end, do not write the KR. The measurement method is not a detail — it is the KR.

## Anti-Pattern 8: The Retroactive KR

**Description**: Key Results are written or substantially reworded after the quarter has ended to match what the team actually delivered — rather than reflecting what was planned at the start. This is intentional score inflation disguised as documentation. It is distinct from updating a KR mid-quarter due to a genuine priority change (which should be documented as an amendment, not a revision). Retroactive KRs destroy the planning value of OKRs completely — future quarters inherit a false baseline and a false signal about team capacity and execution reliability.

**Bad Example**:

*(Written in October, claiming to be the Q3 plan)*

```
KR: Complete migration of services A, B, and C to the new observability stack.
```

*(Reality: Services D, E, and F were the Q3 plan. A, B, and C were migrated because D, E, F were blocked.)*
*(The retroactive version shows a 1.0. The honest version would show a 0.4.)*

**Why It Fails**: A 1.0 on a retroactive KR represents work that happened — but not the work that was planned or the outcome that was intended. The team didn't hit what they committed to; they documented what they completed and called it a hit. Leadership cannot make hiring, investment, or planning decisions from this signal. Future OKR cycles inherit artificially optimistic capacity assumptions.

**Corrected Version**:

Write KRs before the quarter begins. Lock the text. Score against the original. If work changed significantly due to a genuine priority change (a major incident consumed the team, a new security requirement emerged), document the change as a retrospective note with the original score intact:

```
Q3 Actual Score: 0.4 (Services D and F blocked by vendor dependency; pivoted to A, B, C.)
Retrospective Note: Pivot was the right call given the vendor delay. Q4 plan will resume D, E, F
with the vendor dependency resolved. Capacity assumption for Q4 should account for the carry-forward.
```

**Integrity signal**: A team that scores 0.4 honestly and explains the pivot demonstrates better OKR culture than a team that scores 1.0 retroactively and explains nothing.

## Anti-Pattern 9: The Manager KR

**Description**: A Key Result is written in terms of outcomes that only the manager or senior leader can influence — budget approvals, executive alignment, headcount hiring — with no clear line of agency for the individual contributors on the team. The team becomes passive bystanders to their own OKRs. This is distinct from the Dependency Trap (Anti-Pattern 11), which involves other engineering teams. The Manager KR involves organizational levers that only the leader controls.

**Bad Example**:

```
KR: Achieve budget approval for the platform modernization program by Q3.
KR: Obtain executive alignment on the 3-year SRE roadmap from the VP organization.
KR: Complete headcount hiring to bring the team from 12 to 18 FTEs.
```

**Why It Fails**: Budget approval depends on finance, executive priorities, and the business cycle. Executive alignment depends on the Director's influence, the political climate, and competing priorities. Headcount depends on recruiting pipelines, offer acceptance rates, and HR capacity. The engineers on the team have essentially zero agency over any of these outcomes. OKRs should create shared ownership and team energy. These create passive observers who watch their score be determined by forces entirely outside their control.

**Corrected Version**:

```
KR: Deliver a platform modernization business case with quantified ROI, risk analysis, and a
    3-year roadmap, reviewed and approved at Director level by week 6.
    (This the team can build. Budget decision follows from a quality deliverable.)

KR: Achieve documented agreement from 5 product VPs on SRE service model expectations —
    via structured working sessions and written sign-off by Q3 end.
    (The team can prepare the material, run the sessions, and drive to agreement.)

KR: Reduce time-to-productive for new SRE hires from 90 days to ≤45 days, applied to any
    hires who join in Q3.
    (The team controls the onboarding experience. Headcount is a manager KR; quality is a team KR.)
```

## Anti-Pattern 10: The Compound KR

**Description**: A single Key Result contains two distinct outcomes connected by "and." The two outcomes may be related but move independently — one can succeed significantly while the other fails entirely. At scoring time, the compound structure forces an arbitrary judgment call about how to weight the two components. It also obscures which outcome drove value and which one lagged — losing the diagnostic signal that makes OKR retrospectives useful.

**Bad Example**:

```
KR: Reduce MTTR by 40% and increase deployment frequency by 3x.
KR: Achieve SLO compliance ≥99% and reduce on-call page volume by 50%.
KR: Complete the platform migration and train all engineers on the new tooling.
```

**Why It Fails**: If MTTR drops 40% but deployment frequency only increases 1.5x, what is the score? 0.5? 0.75? The compound structure requires an arbitrary weighting decision that was never made explicit. It also hides the question "why did deployment frequency only move 1.5x?" — that question is the most valuable output of a KR that missed, but the compound structure buries it under the MTTR success.

**Corrected Version**: Split into two separate KRs. Each outcome deserves its own score and its own retrospective.

```
KR1: Reduce Sev-1 MTTR from 38 minutes to ≤20 minutes across all incidents.
     (Baseline: Incident system, Q2. Measurement: Median MTTR at quarter end.)

KR2: Increase deployment frequency for the order domain from 2 per week to 5 per week.
     (Baseline: CI/CD pipeline data, Q2. Measurement: Q3 average from pipeline telemetry.)
```

**Scoring benefit**: When these are separate, a 1.0 on MTTR and 0.5 on deployment frequency produces a specific question: "what constrained deployment frequency?" That question is the foundation of the Q4 plan. The compound version produces no such question.

## Anti-Pattern 11: The Dependency Trap

**Description**: A Key Result requires a team or individual outside the OKR owner's authority to act — and that dependency is not surfaced, risk-mitigated, or reflected in the target. When the external team does not deliver, the OKR owner's score suffers for someone else's non-delivery. The OKR score now reflects another team's decisions, priorities, and capacity, not the platform team's execution quality. This is demoralizing and produces no actionable signal about performance.

**Bad Example**:

```
KR: Achieve ≥95% of microservices instrumented with distributed tracing
    (requires all app teams to integrate the SDK — 15 teams, no shared authority).

KR: Reduce mean deployment lead time to ≤2 days
    (requires 15 product teams to adopt the new CI/CD pipeline — their decision, not SRE's).

KR: Eliminate manual certificate rotation across all services
    (requires Security team to approve the automated CA integration — separate reporting chain).
```

**Why It Fails**: The SRE team may deliver the tooling perfectly. If product teams do not adopt, or Security does not approve, the score is 0.3. The KR score has become a referendum on other teams' priorities, not a measure of the platform team's work. Dependency trap KRs are common because the most impactful outcomes in platform engineering genuinely require adoption — the trap is writing the KR as if adoption is guaranteed.

**Corrected Version** (restructure to measure what the OKR owner actually controls, while driving toward the same ultimate outcome):

```
KR: Deliver distributed tracing SDK with a self-service integration guide, and achieve ≥80%
    of tier-1 product teams (8 of 10) successfully integrated by Q3 end — SRE drives adoption.
    (Team controls: SDK quality, documentation, integration support. Adoption target is achievable
    with active enablement. If 2 teams miss, the KR is still meaningful at 0.8.)

KR: Reduce lead time from 11 days to ≤2 days for the 3 product teams who opted into the
    new CI/CD pipeline pilot — measured from pilot kickoff to quarter end.
    (Scope reduced to teams with confirmed commitment. Result proves the platform works.)

KR: Complete Security team approval of the automated CA integration and deploy to ≥20
    non-legacy services by Q3 end — SRE owns the technical proposal and the deployment.
    (Dependency named and mitigated: approval process is started in week 1, not week 8.)
```

## Anti-Pattern 12: The Jargon KR

**Description**: A Key Result is written using internal acronyms, team-specific technology names, or engineering jargon that makes it impossible for a senior leader, cross-functional stakeholder, or new team member to understand whether the KR was achieved. In a VP or board-level OKR review, jargon creates accountability blind spots — stakeholders nod along without understanding what was delivered. In an interview, presenting jargon KRs signals that you communicate for technical insiders, not for organizational leaders.

**Bad Example**:

```
KR: Migrate KOTS deployments to HelmOCI with ArgoCD wave sync for all PCL-managed services
    in the APAC footprint.

KR: Achieve DAST/SAST gate enforcement at T2+ in the PCF→EKS lift-shift pipeline.

KR: Complete TIBCO EMS → Kafka MSB bridge deprecation for legacy ESB consumers in the
    billing domain.
```

**Why It Fails**: A VP of Product, a CFO, or a new engineering hire cannot verify any of these KRs without a 20-minute briefing. Accountability requires that the scorekeeper can confirm the score without being an insider. Jargon KRs also hide ambiguity — "completed" the bridge deprecation could mean many things, and the jargon prevents anyone from asking the clarifying question.

**Corrected Version** (plain English with technical identifiers in parentheses only if needed for cross-referencing):

```
KR: Migrate 100% of platform-managed services (85 total) to the Kubernetes-native deployment
    model, eliminating the legacy KOTS infrastructure and its manual deployment dependencies.
    (Baseline: 85 services on KOTS. Measurement: Deployment config audit at quarter end.)

KR: Block all deployments with critical or high security vulnerabilities from reaching
    staging — enforced automatically through the CI/CD pipeline for all cloud-migrated services.
    (Baseline: 0% of deployments auto-blocked today. Measurement: CI/CD gate enforcement rate.)

KR: Complete retirement of the legacy messaging bridge in the billing domain, reducing
    message routing infrastructure cost by ≥40% and eliminating the dual-path maintenance burden.
    (Baseline: Current infrastructure cost from cloud billing. Measurement: Cost delta at quarter end.)
```

**Test**: Read the KR aloud to a non-engineer. If they cannot tell you what "done" looks like, rewrite it.

## Anti-Pattern 13: The Moving Target

**Description**: The definition, scope, or measurement method of a Key Result changes during the quarter — either to make it easier to hit (goalpost moving backward when the team is behind) or to chase an emerging priority (goalpost moving forward when something new appears). Both destroy the accountability and retrospective learning value of OKRs. A KR that changes mid-quarter produces a score that answers a different question than the one asked at the start.

**Bad Example**:

*(Set in July, week 1)*
```
KR: Reduce deployment lead time from 11 days to ≤3 days for the payments domain.
```

*(Revised in September, week 10, because a major incident consumed the team for 3 weeks)*
```
KR: Reduce deployment lead time from 11 days to ≤7 days for the payments domain. [REVISED]
```

*(Scored 1.0 in October)*

**Why It Fails**: The revision converts a miss into a hit. The team spent a quarter and moved lead time from 11 days to 6 days — meaningful progress, but not the committed outcome. The score says "we succeeded." The data says "we moved halfway." Next quarter's plan is built on a false baseline (6 days was the ending state, but the OKR says 7 days was the target). The incident that caused the revision is now invisible to anyone reading the OKR score.

**Corrected Version**: Lock the KR definition at quarter start. If a genuine force-majeure event prevents progress — a major incident, a critical security vulnerability, a forced regulatory change — score it accurately and document the context in the retrospective. An honest 0.4 with a clear explanation is more valuable than a revised 1.0.

```
Q3 Scoring Record:
Original KR: Reduce deployment lead time from 11 days to ≤3 days.
Actual ending state: 6 days.
Score: 0.5 (methodology: linear interpolation between 11 days and 3-day target.)
Retrospective note: 3-week Sev-1 response consumed the team in weeks 6–8.
Q4 plan: Resume from 6-day baseline with the blocker eliminated. Target ≤3 days remains valid.
```

## Anti-Pattern 14: The KPI Recycler

**Description**: Existing operational KPIs — metrics the team was already tracking and reporting — are relabeled as OKRs without any additional stretch, directional intent, or improvement commitment. The OKR cycle produces no new accountability and no new behavior. The team was already tracking these metrics; calling them OKRs adds ceremony without adding ambition or learning.

**Bad Example**:

```
Objective: Maintain platform health.

KR1: Track MTTR monthly and report in the ops review. (Existing KPI — no target)
KR2: Monitor deployment success rate and address outliers. (Existing KPI — no target)
KR3: Review on-call page volume quarterly. (Existing KPI — no target)
```

**Why It Fails**: "Track," "monitor," and "review" are reporting activities, not outcomes. The team was already doing all three before the OKR cycle began. Writing them as KRs creates no new commitment, no stretch target, and no accountability for improvement. At quarter end, the team reports the same KPIs they were already reporting, and the OKR score reflects whether the reports were written — not whether the platform improved.

**Corrected Version**: Take the existing KPI baseline and write a directional improvement target with a specific endpoint:

```
Objective: Materially improve platform operational health versus Q2.

KR1: Reduce Sev-1 MTTR from 38 minutes to ≤20 minutes.
     (Baseline: Incident system, Q2 median. Measurement: Q3 median at quarter end.)

KR2: Improve deployment success rate from 97.8% to ≥99.5%.
     (Baseline: CI/CD pipeline data, Q2. Measurement: Q3 average from pipeline telemetry.)

KR3: Reduce on-call pages per engineer per week from 94 to ≤30.
     (Baseline: PagerDuty weekly average per rotation slot, Q2. Measurement: Q3 weekly average.)
```

**The distinction**: A KPI says "this is what we watch." An OKR says "this is how far we intend to move it and by when." The KPI tells you the instrument; the OKR tells you the destination.

## Anti-Pattern 15: The Copy-Paste OKR

**Description**: The same OKRs — or near-identical variants — appear in consecutive quarters with no evidence that the team analyzed what worked, what changed, or what should be different. Copy-paste OKRs signal that OKRs are being treated as a compliance exercise rather than a planning and learning tool. They also produce an invisible problem: if the team made progress in Q2, the Q3 baseline should be the Q2 ending state — not the original Q2 starting point. Copy-pasting without updating baselines means the team either didn't move the metric (and is hiding that failure) or did move it (and is sandbagging by targeting the same endpoint again).

**Bad Example**:

*(Q2 and Q3 are identical — baselines not updated)*

```
Q2 OKRs:
KR: Reduce deployment lead time from 11 days to ≤3 days.
KR: Achieve SLO compliance ≥95% for tier-1 services.
KR: Reduce alert noise by 50% (from 94 to ≤47 alerts per engineer per week).

Q3 OKRs (copied verbatim):
KR: Reduce deployment lead time from 11 days to ≤3 days.
KR: Achieve SLO compliance ≥95% for tier-1 services.
KR: Reduce alert noise by 50% (from 94 to ≤47 alerts per engineer per week).
```

**Why It Fails**: If Q2 ended with a 7-day lead time, the Q3 target of ≤3 days is a genuine stretch — but the baseline should be 7 days, not 11. Starting from 11 again means either the team is claiming they went backwards (which they aren't documenting), or they are sandbagging by targeting the same endpoint from a higher starting point. Copy-pasting also signals no retrospective happened: what did Q2 teach the team about what constrained deployment lead time? That learning should produce a different Q3 approach and a different KR.

**Corrected Version** (baselines updated to reflect Q2 ending state; at least one KR addresses a Q2 retrospective finding):

```
Q3 OKRs (based on Q2 actual ending state and retrospective findings):

KR1: Reduce deployment lead time from 7 days (Q2 ending state) to ≤3 days — completing the
     original target that was constrained by manual approval gates in Q2.
     (Q2 retrospective finding: the bottleneck was the security approval gate, now automated.)

KR2: Hold SLO compliance at ≥95% for tier-1 services for 8 consecutive weeks — Q2 achieved
     ≥95% in aggregate but with weekly dips; sustained hold is the new bar.
     (Q2 retrospective finding: compliance was 95% on average but dropped to 88% in weeks 5 and 9.)

KR3: Reduce alert volume from 52 alerts per engineer per week (Q2 ending state) to ≤30 —
     targeting the runbook automation gap identified as the root cause of residual noise in Q2.
     (Q2 retrospective finding: the top 15 remaining alerts are P2s with existing runbooks
     that could be automated; automated remediation was not in scope for Q2.)
```

**Retrospective requirement**: Before writing Q3 OKRs, the team should be able to answer three questions for each Q2 KR: (1) What was the actual ending state? (2) What constrained progress? (3) What specifically should be different in Q3 as a result? If none of the Q3 OKRs reflect these answers, the OKR cycle is not producing learning.

## Director-Level Diagnostic: OKR Review Checklist

Use this when reviewing a team's OKR draft before the quarter starts. Each row maps to one or more anti-patterns.

| Check | Question to Ask | Anti-Pattern(s) If It Fails |
|-------|----------------|------------------------------|
| Outcome vs. output | Does each KR describe a change in state — not a task completed? | #1 Milestone KR |
| SLO check | Is any KR simply restating an existing operational commitment? | #2 SLO-as-OKR |
| Count check | Does each objective have ≤4 KRs, all of which are outcomes? | #3 Activity Dump |
| Stretch calibration | Would hitting each KR require real effort? (50–85% confidence) | #4 Guarantee, #5 Moon Shot |
| Baseline present | Is there a measured starting point from a named data source? | #7 Invisible Measure, #14 KPI Recycler |
| Single metric | Does each KR measure exactly one distinct thing? | #10 Compound KR |
| Team ownership | Can the team move this without another team's decision or budget approval? | #9 Manager KR, #11 Dependency Trap |
| Layperson readable | Can a VP of Product confirm the result without a glossary? | #12 Jargon KR |
| Measurement method | Is the scoring system named and accessible today, not TBD? | #7 Invisible Measure |
