# Reference — Executive Vocabulary Translation

Quick-lookup table for converting technical terminology into executive and board-readable language. Use this when preparing any executive communication, board slide, or investment proposal. Entries are grouped by domain.

---

## Latency and Performance Metrics

| Technical Term | Executive / Board Equivalent | Context Note |
|---------------|------------------------------|--------------|
| p99 latency | Worst-case response time for 99% of customers | "The slowest 1 in 100 transactions takes X seconds" |
| p50 latency / median response time | Typical response time for the average customer | Useful for baseline; boards care more about p99 |
| p999 latency | Worst-case response time for 99.9% of customers | Use when discussing tail latency affecting VIP or enterprise customers |
| Latency SLA | Response time commitment to customers | Always pair with the contractual consequence of breach |
| Latency degradation | Service slowdown affecting customer experience | Quantify in terms of customer wait time and abandonment rate |
| Throughput | Volume of transactions processed per second/minute | "Our platform processes X transactions per minute at peak" |
| Request timeout | Failed transaction due to slow response | "Customer transaction failed because our service did not respond in time" |
| Cache hit rate | Percentage of requests answered from fast memory vs. slow database | "How often our platform avoids a slow database query" — affects cost and speed |

## Reliability and Availability Metrics

| Technical Term | Executive / Board Equivalent | Context Note |
|---------------|------------------------------|--------------|
| SLA / SLO | Reliability commitment to customers | SLA = contractual with penalties; SLO = internal target |
| Uptime / availability | Percentage of time the service is operating | "99.9% uptime means the service was unavailable 8.7 hours per year" |
| Error rate | Percentage of customer requests that fail | Always translate to customer volume: "X% error rate = Y failed transactions per hour" |
| Error budget | Acceptable failure margin before customer impact triggers | "We have consumed 40% of our quarterly failure budget with 6 weeks remaining" |
| MTTR | Average time to restore service after an outage | "Average time to fix a failure" — boards care about this for customer impact |
| MTTD | Average time to detect a failure | "How quickly we know something is wrong" — affects MTTR |
| MTTF | Average time between failures | "How often our services fail" |
| Incident rate | Number of service disruptions per time period | Compare to prior period and industry benchmark |
| Blast radius | Scope of impact when a component fails | "If this service fails, it affects X% of customer transactions" |
| Cascade failure | One failure causing multiple dependent systems to fail | "A single failure propagated to 3 downstream services, multiplying the customer impact" |
| Toil | Manual repetitive work that automation could eliminate | "Engineering time spent on manual operations instead of improvement work" |
| Runbook | Step-by-step incident response procedure | "A standardized repair manual for known failure types" |
| On-call | Engineer available to respond to incidents outside business hours | Frame as: "24/7 operational coverage" |

## Deployment and Engineering Velocity

| Technical Term | Executive / Board Equivalent | Context Note |
|---------------|------------------------------|--------------|
| Deployment frequency | How often the team ships new software to customers | DORA metric — higher is better; compare to industry benchmark |
| Lead time for changes | Time from code complete to production deployment | "Speed from engineering decision to customer-visible change" |
| Change failure rate | Percentage of deployments that cause an incident | "How often a software update breaks something" |
| Canary deployment | Limited test rollout before full release | "Releasing to 1% of customers first to catch problems before they affect everyone" |
| Blue/green deployment | Running two versions in parallel to enable instant rollback | "The ability to switch back to the previous version in seconds if something goes wrong" |
| Feature flag | Switch to turn features on/off without a deployment | "Ability to enable or disable a feature instantly for any customer segment" |
| Technical debt | Accumulated shortcuts that slow future development | "Engineering liability — shortcuts taken earlier that now slow every future change" |
| Refactoring | Restructuring existing code without changing its behavior | "Paying down technical debt — making the system easier to work with" |
| CI/CD pipeline | Automated testing and deployment process | "Our automated quality gate and delivery conveyor belt" |
| Rollback | Reverting to a prior version after a failed deployment | "Undo a software change that caused problems" |

## Infrastructure Terms

| Technical Term | Executive / Board Equivalent | Context Note |
|---------------|------------------------------|--------------|
| Kubernetes / K8s | Container orchestration platform | "The operating system that manages where and how our software runs at scale" |
| Auto-scaling | Automatically adding or removing compute capacity based on demand | "Our platform expands to handle traffic spikes and shrinks to save cost when demand drops" |
| Load balancer | Traffic distribution system | "Routes customer requests across multiple servers to prevent overload on any single one" |
| Single point of failure (SPOF) | A component whose failure takes down the whole service | "A vulnerability where one failure brings everything down" |
| Redundancy | Duplicate systems that take over if primary fails | "Backup systems that automatically activate when the primary fails" |
| Multi-region / multi-AZ | Infrastructure spread across geographic locations | "Our systems run in multiple locations so a regional outage does not affect all customers" |
| CapEx vs. OpEx | Capital expenditure (one-time) vs. operating expenditure (recurring) | "CapEx = buying a server; OpEx = paying for cloud compute monthly" |
| Cloud spend / burn | Monthly or annual cost of cloud infrastructure | Boards care about trend and unit economics, not absolute number alone |
| Technical EOL (end of life) | Software or hardware the vendor no longer supports or patches | "No longer receiving security updates — creates vulnerability exposure" |

## SRE-Specific Concepts

| Technical Term | Executive / Board Equivalent | Context Note |
|---------------|------------------------------|--------------|
| Observability | Ability to understand what is happening inside the system from external signals | "How well we can see what is happening in our platform and diagnose problems quickly" |
| Monitoring | Watching systems for known failure patterns | "Alert system — notifies us when known problems occur" |
| Alerting | Automated notification when something goes wrong | "Pager system — triggers the on-call engineer" |
| Post-incident review / PIR | Structured analysis of what caused an incident and what to change | "Formal root cause analysis with documented corrective actions" |
| Chaos engineering | Deliberately inducing failures to test resilience | "Proactively breaking things in a controlled way to find weaknesses before customers do" |
| Capacity planning | Forecasting infrastructure needs ahead of demand | "Engineering equivalent of inventory planning — ensuring we have enough headroom for growth" |
| Incident severity (Sev1/Sev2) | Classification of incident criticality and response urgency | "Sev1 = all hands, customer-visible, maximum urgency; Sev2 = significant but contained" |

## Security Terms

| Technical Term | Executive / Board Equivalent | Context Note |
|---------------|------------------------------|--------------|
| CVE (Common Vulnerability and Exposure) | Published security vulnerability with a known exploit path | "A documented security flaw; critical CVEs have known attack methods" |
| Patch management | Applying security fixes to software | "Closing known security gaps before attackers exploit them" |
| Zero-day vulnerability | Security flaw with no available fix from the vendor | "A security gap we cannot close yet — highest risk category" |
| Penetration testing | Authorized simulated attack to find vulnerabilities | "Hiring ethical hackers to find weaknesses before real attackers do" |
| SOC2 compliance | Security and operational controls audit | "Third-party verification that our security practices meet enterprise standards" |
| Data breach | Unauthorized access to customer or business data | Always quantify: records affected, notification obligations, regulatory exposure |
| DDoS attack | Overwhelming a service with fake traffic to make it unavailable | "A digital attack that floods our systems to prevent real customers from using them" |
| Blast radius (security context) | Scope of damage if a credential or system is compromised | "How much damage can be done if one account or system is breached" |
