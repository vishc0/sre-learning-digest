# Chapter 03: Cassandra Operations — Distributed Databases at Production Scale

---

## The Mental Model Shift

PostgreSQL has a primary. MySQL has a primary. Every relational database you have ever managed has a single source of truth, and your job was to protect it.

Cassandra has no primary.

That is not a limitation — it is the design. Every node in a Cassandra cluster owns a range of the token ring and is a fully capable reader and writer for that range. When the DND platform was pushing 25M messages per day, no single Cassandra node was the gatekeeper. The coordinator node — whichever node your client happened to connect to — accepted the write and routed it to the owning replicas. You did not have to know which node held which row. The ring handled it.

This changes your mental model for reliability. In PostgreSQL, "the primary is down" is a P1 incident. In Cassandra, a single node going down is expected. The cluster is designed to absorb it. The question is not *if* nodes fail — it is whether your consistency level, replication factor, and repair hygiene are configured to make node failure invisible to the application.

---

## Ring Topology: How Writes Actually Land

Every row in Cassandra has a partition key. That key is hashed to a token — a number in a fixed range. Each node owns a segment of that token range. When a write arrives at the coordinator, it looks up which nodes own the token for that partition key and forwards the write to them.

The replication factor (RF) determines how many nodes receive each write. At RF=3, three nodes hold every row. The ring is designed so those three nodes are spread across the cluster — or, with rack/datacenter awareness configured, across physical failure domains.

At the DND scale, this means a notification's delivery state is simultaneously on three nodes. The question the coordinator asks is: "How many of those three nodes need to confirm the write before I tell the application it succeeded?" That question is answered by the consistency level.

---

## Consistency Levels as Reliability Levers

This is the insight that separates operators who understand Cassandra from those who just run it.

**ONE** — The coordinator waits for a single replica to acknowledge. Fastest possible write path. The risk: a node accepts the write, acknowledges to the coordinator, and then fails before replication completes. That replica is now missing the write. If a read later hits only that node (or the node comes back without the data), you have a silent inconsistency.

**QUORUM** — Majority of replicas must acknowledge. At RF=3, that means 2 of 3 nodes must confirm. If one replica is slow or down, the write still succeeds. If two are down, the write fails — but you never had quorum anyway. This is the operational sweet spot for most production use cases. You get durability without the latency of waiting for every replica.

**ALL** — Every replica must respond. Maximum durability guarantee. Minimum tolerance for node failure. A single replica timeout fails the entire write. You would use ALL when data loss is categorically unacceptable — but you are accepting that any node issue immediately surfaces as application errors.

**The Director Frame:** Consistency level is not a database configuration choice. It is a business decision about how much data loss is acceptable in a failure scenario.

On the DND platform, the question was concrete: if a Cassandra node fails mid-write, does a notification get delivered twice, once, or not at all? Delivering twice is annoying. Delivering not at all means a regulatory notification (opt-out, quiet hours, consent) was silently dropped. Those have different risk profiles, and the consistency level for each write path should reflect that difference — not default to whatever the driver's default happens to be.

## Compaction: Why Cassandra Fights Itself During Peak Load

Cassandra uses an LSM (Log-Structured Merge) tree storage engine. Writes always append to an in-memory structure (memtable), which is periodically flushed to disk as an immutable file (SSTable). Updates and deletes are never done in-place — a delete is a tombstone, a new write that says "this key is gone."

Over time, you accumulate many SSTables per partition. Reads get slower because Cassandra must check multiple SSTables to assemble the current state of a row. Tombstones accumulate and waste space. Compaction is the background process that merges SSTables, discards expired tombstones, and produces a smaller, cleaner set of files.

The problem: compaction is I/O intensive. It reads existing SSTables from disk, merges them in memory, and writes new SSTables back. On a cluster handling 25M messages per day, compaction and write throughput compete for the same disk I/O budget.

You have watched compaction eat write headroom on the DND cluster. The symptom is write latency spiking during compaction runs, especially when multiple nodes compact simultaneously. The mitigation is compaction throttling (`compaction_throughput_mb_per_sec`) and choosing the right compaction strategy:

- **STCS (SizeTieredCompactionStrategy)**: default, groups similar-sized SSTables. Writes fast, reads slower over time. Good for write-heavy workloads.
- **LCS (LeveledCompactionStrategy)**: maintains SSTables in levels of predictable size. Reads are faster and more consistent. Write amplification is higher. Good for read-heavy workloads.
- **TWCS (TimeWindowCompactionStrategy)**: designed for time-series data where old data is rarely updated. Compacts within time windows. Ideal for append-mostly workloads — notifications, events, logs.

For DND-style notification tracking, TWCS is typically the right choice: you write notification state once, rarely update it, and queries are almost always for recent windows.

## Repair: The Operation Nobody Runs Until It's Too Late

Cassandra is eventually consistent. Replicas drift apart. A node that was down for an hour came back up, but it missed writes that happened during its outage. A network blip caused a write to land on two of three replicas but not the third. A failed write left replicas in an inconsistent state.

Anti-entropy repair (`nodetool repair`) is the mechanism that corrects this drift. It compares data across replicas using Merkle trees and synchronizes any differences. Without periodic repair, divergence accumulates. Eventually, reads return different results depending on which replica responds.

The operational risk: `nodetool repair` is expensive. It does a full read of all data in the repaired token range, computes checksums, compares across replicas, and streams any missing data. On a cluster under write load, running repair without controls can cause read latency spikes, write timeouts, and in severe cases, compaction pressure as streamed data generates new SSTables.

The safe approach:
- Run repair on a schedule — typically weekly, during the lowest-traffic window
- Use `-pr` (primary range only) so each node only repairs data it is responsible for, avoiding duplicate work
- Use incremental repair where supported (Cassandra 4.x) — tracks which SSTables have already been repaired and skips them
- Never run repair simultaneously on multiple nodes; stagger across the cluster

**The Director Frame:** When I walk into an organization running Cassandra and ask "when did you last run repair?" — the answer tells me everything about the maturity of their operations practice.

The correct answer is "last week, on a schedule, verified clean." The concerning answers are "we don't run it" (divergence is accumulating silently) or "we ran it once when we saw inconsistencies" (reactive, not preventive). Repair is not optional maintenance. It is the mechanism that makes eventual consistency eventually correct.

## Cassandra Failure Modes

| Failure | Impact | Recovery |
|---|---|---|
| Single node down (RF=3, CL=QUORUM) | Transparent to application; 2 of 3 replicas still satisfy quorum | Node restarts and bootstraps from peers via repair/streaming |
| Two nodes down (RF=3, CL=QUORUM) | Writes fail for affected token ranges; quorum unachievable | Restore nodes quickly; check repair state after |
| Multiple nodes crossing RF threshold | Data unavailability — some partitions have zero surviving replicas | Restore from backup; repair aggressively |
| Coordinator failure | Client retries to another coordinator; no data loss | Transparent if client retry is configured |
| Network partition splitting the ring | Two island clusters, each serving some writes; divergence on shared partitions | Repair after healing; potential data conflicts |

The partition scenario is the hardest. Cassandra follows AP in CAP theorem — it favors availability and partition tolerance. During a network split, both sides continue accepting writes. When the partition heals, you have conflicting versions of the same rows. Cassandra resolves these with "last write wins" (LWW) based on timestamps. If your clocks are not synchronized (NTP is non-negotiable on Cassandra clusters), LWW produces incorrect results silently.

## nodetool: The Operator's Primary Interface

| Command | What It Does | When to Use |
|---|---|---|
| `nodetool status` | Shows cluster health: node state, load, token ownership | First command in any troubleshooting session |
| `nodetool ring` | Shows token ranges and which nodes own them | Diagnosing data distribution imbalance |
| `nodetool repair` | Runs anti-entropy repair on a node | Weekly maintenance; after node recovery |
| `nodetool cleanup` | Removes data no longer owned by this node after topology change | After adding nodes and rebalancing the ring |
| `nodetool compact` | Manually triggers compaction on a keyspace/table | Before a maintenance window to reduce SSTable count |
| `nodetool decommission` | Gracefully removes a node from the ring, streaming its data to peers | Planned node removal; never kill a node without decommissioning first |
| `nodetool tpstats` | Thread pool statistics: pending, active, blocked | Identifying back-pressure and executor saturation |
| `nodetool cfstats` | Per-table metrics: read/write latency, SSTable count, bloom filter stats | Identifying slow tables and compaction-lagging keyspaces |

## Director-Level Decision Checklist

When inheriting or evaluating a Cassandra deployment, these are the questions that separate informed oversight from cargo-cult operations:

1. **Replication factor per datacenter** — is it set to 3 or higher? Is it consistent across keyspaces?
2. **Consistency level per query path** — is it documented and intentional, or is it the driver default?
3. **Repair schedule** — automated, monitored, and verified? Or ad hoc?
4. **Compaction strategy per table** — does it match the access pattern (write-heavy vs. read-heavy vs. time-series)?
5. **NTP synchronized** — confirmed on every node? Clock skew breaks LWW conflict resolution silently.
6. **Tombstone accumulation** — are there tables with high tombstone ratios causing read performance degradation?
7. **Datacenter-aware replication** — is data replicated across physical failure domains, or is RF=3 all on the same rack?

Answering these questions is the difference between running Cassandra and operating it.
