# Chapter 05: Distributed Database Patterns — Beyond Replication to True Distribution

---

## The Question Directors Need to Answer First

Every database scaling conversation starts in the wrong place. Teams show up asking "should we use CockroachDB or Cassandra?" when the real question is: *do you actually need distribution, or do you need better replication?*

These are not the same thing. Getting this distinction wrong costs months of migration effort and introduces failure modes you didn't have before.

---

## Replication vs. Distribution: The Filing Cabinet Analogy

**Replication** is having three identical copies of your entire filing cabinet. Every drawer exists on every cabinet. If one cabinet burns down, you pull from another. Reads can go to any cabinet. Writes go to one cabinet and sync to the others. You have fault tolerance and read scale. You do not have write scale — every write still goes through one primary.

**Distribution** is dividing the filing cabinet across three offices: Office A owns drawers A–H, Office B owns drawers I–P, Office C owns Q–Z. No single office has the whole dataset. Writes for "Johnson" go directly to Office B and only Office B. You now have write scale. The tradeoff: looking up records that span offices requires coordination, and that coordination has a cost.

**Why this matters operationally**: Most teams hit read bottlenecks before write bottlenecks. A Postgres primary with read replicas will absorb enormous read traffic. Before reaching for a distributed database, ask whether you have actually exhausted what replication + caching can do for you.

---

## The T-Mobile DND Context: Why Cassandra Was the Right Call

At 25M messages/day, the DND (Do Not Disturb) platform is not primarily a read problem — it is a write problem. Suppression decisions must be logged, subscription changes must be committed, and the system must handle traffic bursts at campaign launch time that look nothing like steady state.

A single Postgres primary with connection pooling and read replicas handles read scale well. It does not handle write fan-out across millions of concurrent campaign events. Cassandra was chosen because:

1. **Write throughput is the primary SLI** — Cassandra's LSM-tree (Log-Structured Merge-tree) writes are append-only, which makes them extremely fast compared to Postgres's B-tree in-place updates
2. **Multi-region active-active was a requirement** — suppression data must be served from the region closest to the originating carrier, not routed back to a primary in us-east-1
3. **The access pattern is narrow** — lookups are almost always by phone number (partition key), not arbitrary queries. Cassandra is built exactly for this pattern

The lesson: **choose the distribution model that matches your access pattern, not the one with the best marketing.**

## The Distributed Database Landscape

### CockroachDB
**Wire protocol**: Postgres-compatible  
**Consensus**: Raft per range (data is divided into 512MB ranges; each range independently achieves consensus among its replicas)  
**Isolation**: SERIALIZABLE — the strongest isolation level, meaning no dirty reads, no phantom reads, no write skew  

**When to use it**: Globally distributed applications that need ACID semantics across regions. Financial transactions, inventory systems, anything where "I need to make sure these two rows are updated atomically even if they're in different data centers" is a real requirement.

**The tradeoff you don't always hear**: Raft requires a quorum write before acknowledging commit. If your replicas are in us-east-1, eu-west-1, and ap-southeast-1, your write latency is bounded by the round-trip to the farthest quorum member. A write that takes 1ms locally now takes 80-120ms cross-region. Design your region affinity carefully.

### YugabyteDB
**Wire protocols**: Both Postgres and Cassandra (YCQL)  
**Design goal**: Lift-and-shift migration path for teams coming from either database  
**Honest assessment**: Architecturally sound, but a smaller community and ecosystem than CockroachDB or Cassandra. Newer project means fewer battle-tested production war stories. Worth evaluating for new greenfield work; approach with more caution for a critical path migration.

### PlanetScale
**Wire protocol**: MySQL-compatible  
**Architecture**: Vitess under the hood — the sharding proxy that YouTube built to scale MySQL  
**Sharding model**: Query routing layer handles the shard mapping; the application continues to speak standard MySQL  

**When to use it**: Teams with deep MySQL expertise and existing applications that need horizontal write scale without rewriting data access patterns. The Vitess layer handles the complexity; your app sees a single MySQL endpoint.

**What you give up**: Foreign key constraints are not supported across shards. Schema migrations require an online DDL workflow. If your application is heavily relational across many tables, Vitess's constraints will surface.

## Sharding Strategies: How the Data Gets Divided

| Strategy | How It Works | Best For | Failure Mode |
|----------|-------------|----------|-------------|
| **Range sharding** | Partition by ID range or date (shard 1: Jan–Mar, shard 2: Apr–Jun) | Time-series data with sequential access | Hot partitions — recent data all lands on the last shard |
| **Hash sharding** | Apply consistent hash to partition key; distribute evenly | Write-heavy workloads needing even distribution | Range queries become scatter-gather across all shards |
| **Directory sharding** | Lookup table maps keys to shards | Flexible, allows resharding without full rehash | The lookup table is now a single point of failure and a latency bottleneck |

**Cassandra uses consistent hash ring**: Each node owns a token range on the ring. Data is placed based on the hash of the partition key. Adding a node means redistributing only that node's token range, not the entire dataset — this is what makes Cassandra scale horizontally with relatively low operational cost.

## Active-Active vs. Active-Passive Multi-Region

**Active-passive**: One region handles all writes. The other region is a warm standby. Failover is manual or automated, but involves a promotion step. Your RPO (recovery point objective) is bounded by replication lag.

**Active-active**: Both regions accept writes. Conflict resolution is required when two regions modify the same record simultaneously.

Cassandra's conflict resolution model is **last-write-wins (LWW)** using timestamps. This is operationally simple but semantically dangerous if clocks drift or if your application logic requires merge semantics. For a suppression platform where the worst outcome is a duplicate send (not data corruption), LWW is acceptable. For a financial ledger, it is not.

## Director Decision Frame

> "Most organizations reach for distributed databases before they've exhausted what replication can do. When I'm evaluating this with a team, I ask three questions: What is your current write throughput per second? What is your P99 write latency requirement? And what is your access pattern — are you doing primary key lookups or arbitrary queries? If the team can't answer those in under a minute, we're not ready for a distributed database migration. That's a capacity planning gap, not a technology selection problem."

**The maturity gate before distributed databases**:
1. Have you profiled your slow queries and added appropriate indexes?
2. Have you implemented connection pooling (PgBouncer, HikariCP)?
3. Have you offloaded reads to replicas?
4. Have you evaluated a caching layer for your most frequent access patterns?
5. Have you measured write throughput and confirmed a single primary is actually the bottleneck?

If you haven't done all five, start there. Distribution is not a shortcut around operational discipline — it adds complexity on top of it.

---

## Key Takeaways for Interview Contexts

- Replication = fault tolerance and read scale. Distribution = write scale and multi-region active-active.
- Cassandra's partition key design determines your access patterns forever. This is the most important architectural decision in a Cassandra deployment.
- CockroachDB gives you ACID across regions; you pay for it in cross-region write latency.
- Hash sharding gives you even distribution; range queries become expensive. Range sharding gives you efficient scans; hot partitions become your enemy.
- At Director/VP level, the question is not "which database is best" — it is "which database matches our access pattern, our team's operational skills, and our consistency requirements?"
