# Chapter 2: PostgreSQL High Availability — Making the Workhorse Production-Grade

**Series**: Database Reliability Engineering
**Audience**: Director/VP Platform Engineering interview prep

## Why PostgreSQL and Why the Default Is Not HA

PostgreSQL is ubiquitous in platform engineering for a simple reason: it is the most capable open-source relational database that does not require a vendor relationship to operate. It handles ACID transactions, JSONB, full-text search, time-series extensions, and geospatial queries. When a team says "we run Postgres," they mean they chose correctness and flexibility over managed convenience.

The problem is that a default Postgres installation — single node, no replication, no connection pooling — is not a production database. It is a development database that happens to hold production data. It has one mode of failure: total. When the node dies, writes stop. When the disk fails, data is gone. When the application opens 500 connections under load, Postgres spawns 500 processes and falls over.

Making Postgres production-grade requires understanding three independent layers: **replication** (data durability and failover), **leader election** (automated failover orchestration), and **connection management** (scaling the connection model). Each layer solves a different failure mode. Solving one without the others is not HA — it is partial protection with hidden gaps.

## Streaming Replication and the Write-Ahead Log

Every write in PostgreSQL passes through the Write-Ahead Log (WAL) before it touches the actual data files. The WAL is an append-only sequential log: first, record what is about to happen; then, make it happen. If the server crashes before applying the change to data files, Postgres replays the WAL on restart and recovers to a consistent state.

The WAL is also the mechanism behind streaming replication. Replicas connect to the primary and consume WAL records as they are generated — the same records used for crash recovery, but streamed to another node in near real-time. The replica applies those records to its own data files, staying in sync with the primary.

**The RabbitMQ analogy**: WAL is a durable queue for the database engine itself. Every write is published to the log first. Replicas are consumers of that log. This is exactly why replicas lag under write bursts — the same reason a consumer falls behind when a producer publishes faster than the consumer can process. The primary is not slowed down by replica lag in asynchronous replication mode; the replica simply reads further behind. The operational implication: under a write surge, a replica's data is stale. If you promote that replica during or immediately after the surge, you may lose transactions that were committed on the primary but not yet applied to the replica.

**Synchronous vs. asynchronous replication** is the tradeoff lever. Synchronous replication (`synchronous_commit = on`) means the primary waits for the replica to confirm WAL receipt before acknowledging the write to the application. You get zero data loss on failover. You pay in write latency — every write waits for a network round-trip to the replica. Asynchronous replication is faster but accepts the lag risk.

For most platforms, **synchronous replication to one standby with asynchronous replication to additional standbys** is the operational sweet spot: zero data loss to the first replica, reduced write latency exposure, and additional read-capacity from async standbys.

## Patroni: Leader Election You Did Not Write

Streaming replication keeps data in sync. It does not handle the question of what happens when the primary dies. Without orchestration, someone has to manually promote a replica, update the connection string, and notify applications. At 3 AM, under pressure, this is where mistakes happen.

Patroni is the open-source answer to automated failover for Postgres. It uses a distributed consensus store — etcd, ZooKeeper, or Consul — to maintain a single source of truth about which node is the current primary. Every Patroni agent checks in with the consensus store on a heartbeat interval. If the primary misses heartbeats, the consensus store detects the failure, and Patroni elects a new leader from the available replicas.

What Patroni does that home-grown scripts do not:
- **Fencing**: Patroni ensures the old primary cannot accept writes after a failover by explicitly withdrawing its lock in the consensus store. Without fencing, a partitioned primary that believes it is still the leader will continue accepting writes — split-brain, the most dangerous database failure mode.
- **Replica sync awareness**: Patroni selects the replica with the most current WAL position to minimize data loss on promotion.
- **Automatic reintegration**: after a failed primary recovers, Patroni reattaches it as a replica rather than leaving it isolated.

**Patroni's own failure modes** are worth knowing at Director level:
- If the consensus store (etcd) becomes unavailable, Patroni will demote the primary rather than risk split-brain. This means **etcd availability is now in your Postgres availability path**. An HA Postgres cluster requires an HA etcd cluster.
- Patroni's default `ttl` (time-to-live for the leader lock) and `loop_wait` (heartbeat interval) determine failover speed. Aggressive settings give faster failover but risk spurious leader elections on network blips. Conservative settings give stability but extend RTO during real failures. This is a tunable you own.

## PgBouncer: Solving the Process-Per-Connection Problem

Postgres uses a process-per-connection model. Every client connection spawns a new OS process on the database server. At 50 connections, this is fine. At 500 connections — common under any meaningful application load — Postgres is managing 500 processes, context-switching constantly, and consuming memory proportional to the connection count. At 1000 connections, query performance degrades before queries even run.

PgBouncer is a connection pooler that sits between the application and Postgres. The application opens connections to PgBouncer; PgBouncer maintains a smaller pool of actual Postgres connections. A thousand application connections might map to 50 Postgres connections, with PgBouncer queuing and routing.

**Two pooling modes, two use cases:**

- **Session mode**: one Postgres connection is held for the life of the application session. Simplest to operate; barely reduces connection count. Only appropriate if your application explicitly manages session-level state (temporary tables, advisory locks, `SET` parameters that must persist).

- **Transaction mode**: a Postgres connection is leased to the application only for the duration of a transaction, then returned to the pool. This is where PgBouncer pays off — 1000 application connections completing short transactions comfortably share 50 Postgres connections. The tradeoff: session-level features (`SET LOCAL`, prepared statements in some configurations, `LISTEN/NOTIFY`) break or require workarounds. Most modern application frameworks work correctly in transaction mode.

The Director-level mistake to avoid: deploying PgBouncer in session mode because it was simpler, then wondering why connection exhaustion still occurs at scale. Session mode is not pooling — it is proxying.

## WAL Archival and Point-In-Time Recovery

A nightly full backup of Postgres gives you a restore target that is 24 hours stale. If the database is destroyed at 11:59 PM, you restore from midnight's backup and lose nearly a full day of transactions. For most production systems, this is unacceptable.

WAL archival closes the gap. As the primary generates WAL segments (each is 16MB by default), they are shipped to an object store — S3, GCS, or equivalent. A restore operation then becomes: restore the last full backup, then replay WAL segments forward to any chosen point in time. This is Point-In-Time Recovery (PITR).

**Why this matters operationally**: PITR is the recovery path for the failure modes that replicas do not protect against. Replicas faithfully replicate every write — including the `DROP TABLE` that a developer ran against production, and the corrupted writes from a buggy migration. Replicas give you failover protection; WAL archival gives you a way to rewind to before the mistake.

The tooling in the Postgres ecosystem — pgBackRest, Barman, WAL-G — handles WAL archival, compression, encryption, and restore orchestration. The operational requirement is not sophisticated; it is discipline: **verify the restore process on a schedule.** A WAL archive that has never been restored is an assumption, not a recovery capability.

## PostgreSQL SLOs: What to Commit To

Availability and performance commitments for a self-managed Postgres cluster should be grounded in what the architecture can actually deliver, not what sounds reasonable.

| SLO Dimension | Reasonable Target | Notes |
|---|---|---|
| Availability | 99.95% (~4.4 hours/year downtime) | Requires Patroni + at least one sync replica; maintenance windows negotiated |
| p99 read latency | < 50ms | Workload-dependent; baseline in production before committing |
| p99 write latency | < 100ms | Higher with synchronous replication; measure with replica count |
| Replication lag (async replicas) | < 30 seconds under normal load | Alert at 10 seconds; page at 60 seconds |
| RTO (primary failure) | < 60 seconds | Patroni automated failover; application reconnect logic required |
| RPO (data loss window) | Zero (sync replica) / < 30 seconds (async) | Determined by replication mode, not by aspirations |

These numbers are starting points for negotiation, not universal truth. The important discipline is that **your SLOs are measured, not estimated.** Run failover drills. Measure actual RTO. Instrument replication lag in your observability stack. An SLO written without measurement data is a fiction.

## Director Decision Frame

> **"When I inherit a team running self-managed Postgres, my first question is: have we ever run a failover drill? Not 'do we have replicas?' — because replicas that have never been promoted are not an HA solution; they are a disaster waiting for a test."**

The follow-up questions for the first 90 days:

1. **Can we promote a replica right now, in this meeting, on a staging cluster?** If the answer is no, that is the first thing that changes.
2. **When did we last restore from WAL archive?** If the answer is "never" or "we haven't set up archival yet," the backup strategy is an assumption, not a recovery plan.
3. **What is our current RTO, measured from last drill?** If no drill has run, schedule one. The number from the drill is the actual SLO, regardless of what is in the runbook.
4. **What does connection pool saturation look like, and what is the auto-remediation?** PgBouncer exhaustion under load should trigger automatic connection recycling, not a PagerDuty alert that waits for a human.

Self-managed Postgres is a capability choice — it gives you control and avoids vendor lock-in. The cost of that choice is operational ownership. A Director who inherits self-managed Postgres without a tested failover playbook, WAL archival, and instrumented SLOs has inherited technical debt, not a capability.
