# Chapter 04: Redis Reliability — When a Cache Becomes a Liability

---

## The Double-Edged Tool

Redis makes everything faster. That is exactly why it becomes dangerous.

A slow database call that takes 200ms gets a Redis cache in front of it and drops to 2ms. The application team ships it, the latency graph looks great, and within six months Redis is in front of everything — session data, rate limiters, feature flags, queue state, pub/sub channels, distributed locks, and the original cache it was added for. The system now depends on Redis for correctness, not just speed.

Then Redis fails.

A slow database degrades gracefully. Response times climb, queues back up, users see spinner screens, SREs get paged. The application is slow but it is *doing something*. When Redis fails — and fails fast, as in-memory systems do — dependent services that were never designed with a Redis-down path cascade into errors. Session stores return null. Rate limiters throw exceptions instead of failing open. Feature flags revert to defaults that nobody has tested in production. The cache miss that was supposed to fall back to the database suddenly hits 10,000 concurrent threads doing it simultaneously, and the database falls over from the thundering herd.

The pattern is always the same: Redis was added as a performance optimization and gradually became load-bearing infrastructure with no failure handling.

This chapter is about making sure that does not happen on your watch.

## Sentinel vs. Cluster: Choosing the Right Architecture

### Sentinel Mode

Sentinel is Redis's answer to primary failover in a simple topology. You run one Redis primary, one or more replicas, and a quorum of at least three Sentinel processes that monitor the cluster independently.

When the primary becomes unavailable, the Sentinels detect the failure, vote among themselves to confirm it is a genuine failure and not a network blip on one Sentinel's side, and promote a replica to primary. The application does not connect to the primary directly — it connects to a Sentinel and asks: "Where is the current primary?" Sentinel returns the address, the application connects there, and when failover happens, the application queries Sentinel again and gets the new primary's address.

The analogy maps directly to what you have run before: Sentinel is functionally equivalent to an HAProxy in front of a RabbitMQ cluster. Clients do not connect to the broker directly. They connect to the proxy, which knows where the current primary is and routes accordingly. The application is isolated from topology changes — it just follows the indirection.

Sentinel's limitations are real and worth naming:
- Replication is asynchronous. A primary can acknowledge a write and fail before the replica receives it. That write is lost. Sentinel does not prevent data loss — it minimizes downtime.
- The quorum requirement for Sentinel processes means you need three Sentinel instances minimum, each on a separate host, to avoid a split-brain scenario where two Sentinels promote two replicas simultaneously.
- Sentinel is a single-primary topology. All writes go to one node. Horizontal write scaling is not possible.

### Cluster Mode

Redis Cluster shards data across multiple primaries. The keyspace is divided into 16,384 hash slots, and each primary owns a subset of those slots. Each primary has at least one replica. When a key is written, the client hashes the key to determine which slot it belongs to, then routes the write to the primary owning that slot.

There is no single primary in cluster mode. Losing one primary node triggers promotion of its replica, but the rest of the cluster is unaffected. Write capacity scales by adding primary/replica pairs.

The operational cost is real:
- Clients must support cluster-aware routing (MOVED redirects and ASKING redirects). A dumb client that does not understand cluster topology will fail in non-obvious ways.
- Multi-key operations only work if all keys hash to the same slot. Hash tags (`{user}:session` and `{user}:cart` both hash on `user`) are the workaround, but they require deliberate key design.
- Cluster management operations — adding nodes, rebalancing slots, removing nodes — are more complex than Sentinel failover.

**The Director Frame:** Sentinel for simple caching where operational simplicity matters and write volume fits a single node. Cluster for data that must survive node loss at scale, or where write throughput has outgrown what one primary can handle.

The mistake to avoid: choosing Cluster for its fault tolerance without accounting for the client library requirements and key design constraints. Teams that deploy Redis Cluster without cluster-aware clients spend their first incident debugging MOVED errors.

## Eviction Policies: Silent Data Loss and When It Is Acceptable

When Redis reaches its configured `maxmemory` limit, it must decide what to do. The answer is set by the eviction policy, and the choice has reliability implications that most teams do not think through until something breaks.

| Policy | Behavior | Reliability Profile |
|---|---|---|
| `noeviction` | Returns errors when full; no data removed | Redis fails loudly. Your application gets errors. No silent data loss, but dependent services start throwing exceptions. |
| `allkeys-lru` | Evicts least-recently-used keys across all keys | Silent data loss. Redis removes whatever it deems cold. Acceptable for a pure cache; catastrophic if Redis is your session store or rate limiter. |
| `volatile-lru` | Evicts LRU keys only among keys with a TTL set | A reasonable policy — it only evicts keys you have declared as ephemeral. Forces good TTL hygiene. |
| `allkeys-lfu` | Evicts least-frequently-used keys | Slightly smarter than LRU for workloads with stable hot keys. Same risk profile as allkeys-lru for non-cache use cases. |
| `volatile-ttl` | Evicts keys closest to expiration first | Aggressive cache-friendly eviction. Only safe when all resident data is genuinely ephemeral. |

The reliability rule is simple: if your system's correctness depends on data surviving in Redis, eviction is not an acceptable failure mode. You need persistence, and you need the application to handle a cache miss gracefully. Eviction policy is only a safe knob for genuinely ephemeral data — data where losing it means a slightly slower response, not a broken user session or a bypassed rate limit.

Setting `maxmemory-policy` to `noeviction` for a session store is the right call, not because you want errors, but because an error is visible and alertable. Silent LRU eviction of a session key looks like a user getting logged out for no reason — a bug that generates support tickets and never makes it to the SRE queue.

## Persistence: RDB vs. AOF vs. Hybrid

Redis is an in-memory database. By default, a restart loses all data. Persistence is how you change that.

### RDB (Redis Database Snapshots)

RDB writes a point-in-time snapshot of the entire dataset to disk on a configured schedule (e.g., every 60 seconds if 1000 keys changed). Recovery from an RDB snapshot is fast — Redis loads the file and is operational in seconds.

The cost: you lose all writes since the last snapshot. A cluster that snapshots every 60 seconds and crashes at the 59-second mark loses up to 59 seconds of writes. For a pure cache, this is acceptable — the cache rebuilds from the source of truth. For a session store, 59 seconds of sessions is real user impact.

### AOF (Append-Only File)

AOF logs every write command to disk. Recovery replays the log from the beginning, which is slower than loading an RDB snapshot. The durability guarantee depends on the `appendfsync` setting:

- `always`: every write synced to disk immediately. Maximum durability. Significant write latency overhead.
- `everysec`: sync once per second. At most one second of data loss. The practical default for most production use cases.
- `no`: OS decides when to flush. Fastest, least durable — effectively no guarantee.

AOF files grow unboundedly and must be rewritten periodically (BGREWRITEAOF). Redis handles this automatically, but the rewrite is CPU and I/O intensive.

### Hybrid Mode (Redis 4.0+)

Hybrid persistence uses an RDB snapshot embedded at the start of the AOF file, followed by AOF entries since the snapshot. Recovery loads the snapshot fast and then replays only the delta. You get fast restarts with high durability.

For anything beyond a pure ephemeral cache, hybrid persistence is the right choice. The decision tree:

- Pure ephemeral cache, cache misses fall back to a database: `noeviction` or `allkeys-lru`, no persistence required.
- Session store, rate limiter, distributed lock, feature flags: hybrid persistence (`aof-use-rdb-preamble yes`), `everysec` fsync.
- Message queue or durable pub/sub: reconsider whether Redis is the right tool. These workloads belong on RabbitMQ or Kafka.

## Redis as a Reliability Risk: The "It's Just a Cache" Trap

The accumulation of Redis use cases over time follows a predictable pattern:

1. Engineering adds Redis as a cache for a slow query. Works great.
2. Engineering adds Redis for session storage because "Redis is already there."
3. Engineering adds a distributed rate limiter using Redis sorted sets.
4. Engineering adds feature flags in Redis because it is fast and easy.
5. A pub/sub channel appears for lightweight messaging.
6. Redis is now on the critical path for authentication, feature delivery, rate limiting, and messaging — with none of the failure handling that a genuine infrastructure dependency requires.

At T-Mobile scale, this is not a hypothetical. Every platform that runs long enough accumulates Redis surface area. The SRE job is to audit that surface area and answer two questions per use case:

**Question 1: What does the application do when Redis is unavailable?**

If the answer is "it throws a 500" or "it crashes" or "I don't know," that is the finding. The correct answer is a documented fallback path: cache miss falls through to the database, rate limiter fails open with logging, feature flag returns a safe default.

**Question 2: Is there a circuit breaker, or does the failure cascade?**

Without a circuit breaker, a slow Redis (not a failed Redis — a *slow* Redis) will hold connections open, exhaust thread pools, and take down the application server. The database behind the cache starts receiving every request simultaneously. The thundering herd collapses the database. The incident scope expands from "Redis is slow" to "platform is down."

A circuit breaker trips after a threshold of failures or timeouts and stops making Redis calls for a configured recovery window. During the open circuit, requests fall through to the fallback path. This is the pattern that contains a Redis incident to a Redis incident instead of letting it become a platform incident.

## Director-Level Decision Checklist

Before signing off on Redis for any new use case:

1. **Topology decision documented** — Sentinel or Cluster? Rationale recorded based on scale and failure tolerance requirements.
2. **Eviction policy matched to use case** — is data in this Redis instance genuinely ephemeral? If not, `noeviction` plus application-level fallback.
3. **Persistence configured** — hybrid persistence for anything load-bearing. RDB-only only for pure caches with documented fallback.
4. **Failure path tested** — has the team actually stopped Redis in staging and observed the application behavior? Unknown failure paths are production surprises.
5. **Circuit breaker in place** — every Redis call from a high-traffic service should go through a circuit breaker. Verify it exists and has been tested.
6. **Memory sizing with headroom** — `maxmemory` set with 20-30% headroom above expected working set. A Redis at 95% memory is one traffic spike from hitting the eviction policy.
7. **Replication lag monitored** — for Sentinel, replica lag determines the blast radius of a failover. Alert on replication lag exceeding your acceptable data loss window.

**The Director Frame:** Before I sign off on Redis for any new use case, I ask two questions: what does the application do when Redis is unavailable? And is there a circuit breaker, or does the failure cascade?

If the engineering team cannot answer both questions, the feature does not go to production. Not because Redis is unreliable — it is extremely reliable for what it is — but because the application's contract with its dependencies must be explicit before those dependencies fail in front of users.

## Common Redis Incident Patterns and Their Tells

| Incident Pattern | Observable Signal | Root Cause |
|---|---|---|
| Memory exhaustion + eviction | Unexpected cache misses, odd session behavior, increased DB load | `maxmemory` set too low; eviction policy removing load-bearing keys |
| Thundering herd on restart | DB latency spike immediately after Redis restart | No staggered warm-up; all clients miss simultaneously |
| Connection exhaustion | Application errors: "max number of clients reached" | No connection pooling; each thread opens a new connection |
| Slow log accumulation | Latency percentile growth without traffic increase | Large key operations (HGETALL on huge hashes, SMEMBERS on large sets) blocking the single-threaded event loop |
| Split-brain after partition | Two applications writing to different primaries; divergent state | Sentinel quorum too small; network partition not detected correctly |
| AOF rewrite blocking | Write latency spikes on a schedule | BGREWRITEAOF running during peak traffic; schedule it off-peak |

Redis's single-threaded event loop is both its strength (no lock contention) and its failure mode. One slow command — a blocking O(N) operation on a large data structure — blocks every other client. `KEYS *` in production is the canonical example: it blocks Redis for the full scan duration. `SCAN` is the correct replacement. Enforcing this is an operational standard, not a suggestion.
