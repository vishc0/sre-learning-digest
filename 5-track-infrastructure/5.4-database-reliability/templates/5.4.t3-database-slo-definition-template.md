## 3. SLI Definitions

> An SLI (Service Level Indicator) is the raw measurement. SLOs are the targets applied to SLIs. Define the measurement precisely before setting targets — ambiguous SLIs make SLOs unenforceable.

### 3.1 Availability SLI

| Field | Definition |
|---|---|
| **Metric name** | `[METRIC_NAME]` *(e.g., `db_availability_ratio`)* |
| **Definition** | Percentage of time the database is accepting reads and writes successfully, as measured by [MEASUREMENT_METHOD]. |
| **Measurement method** | [METHOD] *(e.g., Synthetic probe query every 30s from application tier; Prometheus `up` metric from node exporter; Datadog synthetic monitor)* |
| **Data source** | [SOURCE] *(e.g., Prometheus / Splunk / Datadog / CloudWatch)* |
| **Splunk query (if applicable)** | `[SPLUNK_QUERY]` *(e.g., `index=db_metrics sourcetype=cassandra metric_name=availability | timechart avg(value) by node`)* |
| **Success criteria** | A request is "good" when it returns a valid response within [LATENCY_MS]ms with no application-level error code. |
| **Failure criteria** | Connection refused, timeout > [TIMEOUT_MS]ms, or error codes: [LIST_ERROR_CODES]. |
| **Planned maintenance exclusions** | Maintenance windows scheduled ≥ [N] hours in advance via [CHANGE_MANAGEMENT_PROCESS] are excluded from availability calculation. Maximum [HOURS]h/month of excluded maintenance. |
| **Measurement window** | Rolling 28-day window |

### 3.2 Read Latency SLI

| Field | Definition |
|---|---|
| **Query scope** | [SCOPE] *(e.g., All SELECT queries / Read queries on `messages` keyspace only / Endpoint `/api/v2/receipts` only)* |
| **Measurement point** | [POINT] *(Application-side: from query dispatch to response received — recommended. Database-side measurements exclude network; use application-side for customer-facing SLOs.)* |
| **Metric source** | [SOURCE] *(e.g., Prometheus histogram `db_query_duration_seconds{operation="read"}`)* |
| **Excluded query types** | [EXCLUSIONS] *(e.g., Admin queries, schema migrations, ALLOW FILTERING queries — these are known outliers and tracked separately)* |

## 4. SLO Targets

> Target values represent the commitment. Alerting thresholds trigger at a burn rate that predicts you'll exhaust the error budget before the window closes — not when you've already failed.

| Metric | SLO Target | Measurement Window | Alert Threshold | Critical Threshold | Notes |
|---|---|---|---|---|---|
| **Availability** | [TARGET]% *(e.g., 99.95%)* | Rolling 28 days | [BURN_RATE] burn rate *(e.g., 5x — page if burning error budget 5x faster than baseline)* | [BURN_RATE] burn rate *(e.g., 14.4x — exhausts budget in 1h)* | Excludes planned maintenance |
| **Read Latency P50** | ≤ [MS]ms *(e.g., 10ms)* | Rolling 28 days | > [MS]ms for > 5 min | > [MS]ms for > 2 min | Application-side measurement |
| **Read Latency P99** | ≤ [MS]ms *(e.g., 50ms)* | Rolling 28 days | > [MS]ms for > 5 min | > [MS]ms for > 2 min | |
| **Read Latency P999** | ≤ [MS]ms *(e.g., 500ms)* | Rolling 28 days | > [MS]ms for > 10 min | > [MS]ms for > 5 min | Long tail; monitor but less aggressive paging |
| **Write Latency P50** | ≤ [MS]ms *(e.g., 15ms)* | Rolling 28 days | > [MS]ms for > 5 min | > [MS]ms for > 2 min | |
| **Write Latency P99** | ≤ [MS]ms *(e.g., 100ms)* | Rolling 28 days | > [MS]ms for > 5 min | > [MS]ms for > 2 min | |
| **Replication Lag** | ≤ [DURATION] *(e.g., 1s)* | Rolling 28 days | > [DURATION] for > 60s | > [DURATION] for > 120s | |
| **Connection Pool Utilization** | ≤ [%] *(e.g., 70%)* | Instantaneous | > [%] for > 5 min | > [%] for > 2 min | |
| **Backup Freshness** | ≤ [HOURS]h *(e.g., 4h)* | Continuous | > [HOURS]h since last backup | > [HOURS]h | Alert even outside business hours |

## 5. Error Budget

> Error budget = the allowed failure time that keeps you above your SLO target. Spending it on planned work is healthy. Losing it to incidents without learning is not.

### 5.1 Availability Error Budget Calculation

| Field | Value |
|---|---|
| SLO target | [TARGET]% *(e.g., 99.95%)* |
| Measurement window | 28 days = 40,320 minutes |
| **Allowed downtime per window** | `(1 - [SLO_DECIMAL]) × 40,320 minutes` *(e.g., 0.0005 × 40,320 = **20.16 minutes**)* |
| Remaining budget (current) | [MINUTES] *(updated from monitoring dashboard — link: [DASHBOARD_URL])* |
| Budget consumed MTD | [%] |

### 5.2 Error Budget Policy

| Budget Remaining | Action |
|---|---|
| > 50% | Normal operations. Risky changes are permissible with change controls. |
| 25–50% | Slow down non-essential changes. Review incident backlog. |
| 10–25% | Feature freeze on this service. Reliability work takes priority. Notify [STAKEHOLDER]. |
| < 10% | Full feature freeze. Incident review required. Escalate to [DIRECTOR/VP]. |
| 0% (exhausted) | SLO breached. Immediate post-mortem. Executive notification per [INCIDENT_POLICY]. |
