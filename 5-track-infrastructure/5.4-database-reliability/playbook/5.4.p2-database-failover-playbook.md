# PostgreSQL Primary Failure — Patroni Failover Playbook

**Version**: 1.0  
**Applies to**: PostgreSQL clusters managed by Patroni with etcd or Consul as DCS  
**Last reviewed**: 2026-07-06  
**Owner**: Database Reliability Engineering

---

## Overview

This playbook applies when the PostgreSQL primary node is **unreachable or unresponsive** and Patroni is managing the cluster. The goal is to confirm Patroni has performed automatic failover, verify the new primary is healthy, and restore application connectivity with minimum downtime.

**Do not use this playbook for**:
- Planned switchovers (use `patronictl switchover`)
- Replica failures (separate runbook)
- Patroni configuration changes
- DCS (etcd/Consul) failures — if DCS is down, Patroni will not failover

---

## Pre-conditions (Verify Before Proceeding)

Before following this playbook, confirm these conditions are met. If any fail, escalate.

| Pre-condition | How to check | What to do if it fails |
|---|---|---|
| Patroni is healthy on at least one replica | `patronictl -c /etc/patroni/patroni.yml list` | Escalate — Patroni cannot elect without quorum |
| DCS quorum is intact (etcd or Consul) | `etcdctl endpoint health` or `consul members` | Escalate — Patroni will NOT failover without DCS |
| At least one synchronous replica exists | patronictl list shows replica in `sync` state | Failover may use async replica — risk of data loss |
| Replication was current at failure | Check `pg_stat_replication` lag on old primary logs | Note lag amount — informs data loss risk assessment |

## Step 1 — Confirm Primary Failure

### 1a. Check Patroni cluster status

Run from any node that has Patroni configured:

```bash
patronictl -c /etc/patroni/patroni.yml list
```

**Expected output when primary is down**:
```
+ Cluster: postgres-prod (7890123456789012345) +---------+----+-----------+
| Member          | Host          | Role    | State   | TL | Lag in MB |
+-----------------+---------------+---------+---------+----+-----------+
| postgres-node-1 | 10.0.2.10:5432| Leader  | stopped | 5  |           |
| postgres-node-2 | 10.0.2.11:5432| Replica | running | 5  |         0 |
| postgres-node-3 | 10.0.2.12:5432| Replica | running | 5  |         2 |
+-----------------+---------------+---------+---------+----+-----------+
```

**State interpretation**:
- `stopped`: node is down — process crashed or host unreachable
- `running`: node is healthy
- `Leader`: this node is (or was) the primary
- `Replica`: standby node

### 1b. Distinguish between failure types

Run these checks to identify the root cause before Patroni acts:

```bash
# Check if the Patroni process itself is running on the primary host
# (SSH to the primary host if accessible)
ssh postgres-node-1 "systemctl status patroni"

# Check if PostgreSQL process is running (separate from Patroni)
ssh postgres-node-1 "systemctl status postgresql"

# Check Patroni logs for the failure event
ssh postgres-node-1 "journalctl -u patroni --since '10 minutes ago' --no-pager"
```

**Failure type identification**:

| Symptom | Root Cause | Implication |
|---|---|---|
| Patroni service stopped, PG running | Patroni process crash | PG may still be serving but Patroni won't failover |
| PG stopped, Patroni running | PostgreSQL crash | Patroni will detect and trigger failover |
| Both stopped | Host crash or OOM | Patroni on replicas will detect and trigger failover |
| Patroni running, PG running, but unhealthy | Network partition | Patroni may be in split-brain detection mode |

### 1c. Network partition check

If the primary host appears reachable but is being reported as down:

```bash
# From a replica, attempt to connect to the primary's PostgreSQL port
pg_isready -h 10.0.2.10 -p 5432 -U postgres

# Check Patroni API on the primary
curl http://10.0.2.10:8008/health
```

**Expected output if primary is truly down**:
```
10.0.2.10:5432 - no response
curl: (7) Failed to connect to 10.0.2.10 port 8008
```

If the primary is responding to curl but Patroni reports it as down, you may have a **network partition between nodes but not to your workstation** — do NOT manually intervene. Allow Patroni's TTL to expire and let it handle the election.

## Step 2 — Observe Patroni Automatic Failover Behavior

Patroni triggers automatic failover based on the `ttl` configuration (default: 30 seconds). After the primary's Patroni lock expires in DCS, replicas race to acquire the lock.

**Configured TTL check**:
```bash
grep ttl /etc/patroni/patroni.yml
# Expected: ttl: 30
```

### 2a. Timeline of a Patroni failover

| Time (relative) | Event |
|---|---|
| T+0 | Primary failure occurs |
| T+0 to T+30 | Patroni on replicas detects primary is missing; awaiting TTL expiry |
| T+30 | DCS lock expires; replica election begins |
| T+30 to T+35 | Winning replica acquires DCS lock, promotes itself |
| T+35 to T+45 | New primary starts accepting writes; other replicas repoint to new primary |
| T+60 | Application connections via load balancer begin routing to new primary |

**Total expected downtime**: 30–60 seconds for writes. Read-only traffic to replicas continues uninterrupted.

### 2b. Watch Patroni logs during election on a replica

```bash
journalctl -u patroni -f --no-pager
```

**Key log lines during a successful election**:
```
INFO: Lock owner: postgres-node-1; I am postgres-node-2
INFO: Leader is not responding
INFO: Lock owner: None
INFO: promoted self to leader by acquiring session lock
INFO: cleared rewind request for postgres-node-2
INFO: no action.  i am the leader with the lock
```

The critical line is `promoted self to leader`. If you do not see this within `2 × TTL` seconds (60 seconds at default config), proceed to the escalation section.

## Step 3 — Verify Failover Completion

### 3a. Confirm new primary via patronictl

```bash
patronictl -c /etc/patroni/patroni.yml list
```

**Expected output after successful failover**:
```
+ Cluster: postgres-prod (7890123456789012345) +---------+----+-----------+
| Member          | Host          | Role    | State   | TL | Lag in MB |
+-----------------+---------------+---------+---------+----+-----------+
| postgres-node-1 | 10.0.2.10:5432| Replica | stopped | 5  |           |
| postgres-node-2 | 10.0.2.11:5432| Leader  | running | 6  |           |
| postgres-node-3 | 10.0.2.12:5432| Replica | running | 6  |         0 |
+-----------------+---------------+---------+---------+----+-----------+
```

Note the Timeline (`TL`) has incremented from 5 to 6 — this confirms a failover occurred.

### 3b. Confirm new primary is not in recovery mode

Connect directly to the new primary:

```bash
psql -h 10.0.2.11 -U postgres -c "SELECT pg_is_in_recovery();"
```

**Expected output**:
```
 pg_is_in_recovery
-------------------
 f
(1 row)
```

`f` (false) = this node is the primary. If this returns `t`, the node is still a replica — do not route writes to it.

### 3c. Check replication connections on the new primary

```bash
psql -h 10.0.2.11 -U postgres -c "SELECT * FROM pg_stat_replication;"
```

**Expected output**:
```
 pid  | usesysid | usename     | application_name | client_addr | state     | sent_lsn   | write_lsn  | flush_lsn  | replay_lsn | sync_state
------+----------+-------------+------------------+-------------+-----------+------------+------------+------------+------------+-----------
 1234 | 10       | replication | postgres-node-3  | 10.0.2.12   | streaming | 0/A001234  | 0/A001234  | 0/A001234  | 0/A001200  | async
```

If `pg_stat_replication` is empty, no replicas have yet connected to the new primary. Wait 30 seconds and recheck — replicas should reconnect automatically once Patroni repoints them.

## Step 4 — Application Connection Update

### 4a. Determine connection routing method

| Method | How to identify | Action needed |
|---|---|---|
| DNS-based failover (Route53, internal DNS) | Application uses hostname like `postgres-primary.internal` | Update DNS record to point to new primary IP |
| Load balancer (HAProxy, AWS NLB) | Application uses load balancer VIP | Verify LB health checks have removed old primary, added new one |
| Patroni REST API with HAProxy | HAProxy queries `/master` endpoint on each node | Should auto-update; verify HAProxy stats page |
| PgBouncer | Application connects to PgBouncer | Update PgBouncer config and reload |
| Hardcoded primary IP | Application config file has `host=10.0.2.10` | Manual config update + application restart required |

### 4b. PgBouncer update (if applicable)

If PgBouncer is routing to the old primary:

```bash
# Check current PgBouncer config
cat /etc/pgbouncer/pgbouncer.ini | grep host

# Update the host to the new primary
sudo sed -i 's/host=10.0.2.10/host=10.0.2.11/' /etc/pgbouncer/pgbouncer.ini

# Reload PgBouncer without dropping existing connections
sudo systemctl reload pgbouncer

# Verify the change took effect
psql -p 6432 -U pgbouncer pgbouncer -c "SHOW DATABASES;"
```

**Expected output after reload**:
```
   name    |    host    | port |      database       | ...
-----------+------------+------+---------------------+----
 app_db    | 10.0.2.11  | 5432 | app_db              | ...
```

### 4c. Verify application connectivity

```bash
# Test connection through the application's connection path
psql -h <application_connection_host> -U app_user -d app_db -c "SELECT 1;"

# Check for connection errors in application logs
# Adjust index/sourcetype to match your environment
```

Splunk query to confirm application has resumed writes:
```splunk
index=app_logs sourcetype=app_log
(error="could not connect to server" OR error="connection refused" OR "FATAL" OR "PSQLException")
| timechart span=1m count
```

Verify the error count is trending to zero.

## Step 5 — Post-Failover Validation

### 5a. Test write and read operations

```bash
# Connect to the new primary and run a test write
psql -h 10.0.2.11 -U postgres -d app_db -c "
  INSERT INTO failover_validation_log (event, ts) 
  VALUES ('post_failover_test', NOW()) 
  RETURNING id, ts;
"

# Read it back to confirm durability
psql -h 10.0.2.11 -U postgres -d app_db -c "
  SELECT * FROM failover_validation_log ORDER BY ts DESC LIMIT 1;
"
```

### 5b. Check p99 latency on the new primary

A newly promoted replica may have a cold buffer cache (page cache not warmed). This can cause elevated latency for 5–20 minutes after promotion.

```bash
# Monitor query latency on the new primary
psql -h 10.0.2.11 -U postgres -c "
  SELECT 
    query,
    calls,
    round(mean_exec_time::numeric, 2) AS mean_ms,
    round(stddev_exec_time::numeric, 2) AS stddev_ms
  FROM pg_stat_statements
  ORDER BY mean_exec_time DESC
  LIMIT 10;
"
```

If latency is elevated: this is expected and typically self-resolves. If it does not improve within 20 minutes, check for missing indexes or autovacuum backlog on the new primary.

### 5c. Verify remaining replicas are replicating from the new primary

```bash
psql -h 10.0.2.11 -U postgres -c "
  SELECT 
    application_name,
    client_addr,
    state,
    sent_lsn,
    replay_lsn,
    (sent_lsn - replay_lsn) AS lag_bytes,
    sync_state
  FROM pg_stat_replication;
"
```

All remaining replicas should appear here. If a replica is missing, check its Patroni log:
```bash
ssh postgres-node-3 "journalctl -u patroni --since '5 minutes ago' --no-pager"
```

## Step 6 — Old Primary Recovery

When the original primary (postgres-node-1) comes back online, Patroni handles demotion automatically. Do NOT manually start PostgreSQL in primary mode.

### 6a. What Patroni does automatically

1. Patroni on the old primary detects it no longer holds the DCS lock
2. It runs `pg_rewind` to bring the old primary's timeline in sync with the new primary
3. It starts PostgreSQL in standby mode, replicating from the new primary
4. `patronictl list` will show the old primary as a `Replica` with a non-zero lag

### 6b. Verify the old primary rejoined as a replica

```bash
patronictl -c /etc/patroni/patroni.yml list
```

**Expected output after old primary rejoins**:
```
+ Cluster: postgres-prod (7890123456789012345) +---------+----+-----------+
| Member          | Host          | Role    | State   | TL | Lag in MB |
+-----------------+---------------+---------+---------+----+-----------+
| postgres-node-1 | 10.0.2.10:5432| Replica | running | 6  |        12 |
| postgres-node-2 | 10.0.2.11:5432| Leader  | running | 6  |           |
| postgres-node-3 | 10.0.2.12:5432| Replica | running | 6  |         0 |
+-----------------+---------------+---------+---------+----+-----------+
```

postgres-node-1 is now a `Replica` on timeline 6 (same as the new leader) with some lag. This is correct and expected.

### 6c. Split-brain check

Confirm the old primary is NOT still serving writes as a primary:

```bash
psql -h 10.0.2.10 -U postgres -c "SELECT pg_is_in_recovery();"
```

**Must return**: `t` (true = replica/recovery mode)  
**If it returns `f`**: CRITICAL split-brain condition — stop the old primary immediately and escalate.

```bash
# Emergency: stop the old primary if it returned 'f' above
ssh postgres-node-1 "sudo systemctl stop postgresql"
# Then escalate to senior DBA
```

## Escalation Criteria

| Condition | Action |
|---|---|
| Patroni has not elected a new primary within 2× TTL (default: 60s) | Manually trigger failover — see below |
| `pg_stat_replication` shows zero replicas connected 5+ minutes after failover | Escalate — replicas may be stuck in rewind |
| Old primary returns `pg_is_in_recovery() = f` after recovery | CRITICAL — stop old primary immediately, escalate |
| Data loss confirmed (transaction committed on old primary not on new) | Escalate to DBA lead + application owner |
| Application cannot connect after connection update | Escalate — may be a PgBouncer, network, or auth issue |

### Manual failover via patronictl

If Patroni fails to elect automatically:

```bash
# List candidates
patronictl -c /etc/patroni/patroni.yml list

# Manually trigger failover to the most current replica
patronictl -c /etc/patroni/patroni.yml failover postgres-prod \
  --master postgres-node-1 \
  --candidate postgres-node-2 \
  --force
```

The `--force` flag bypasses the confirmation prompt. Use only when Patroni's automatic election has definitively failed.

## Commands Reference

| Command | Purpose | Expected Output |
|---|---|---|
| `patronictl -c /etc/patroni/patroni.yml list` | Show cluster state, roles, lag | Table with Leader/Replica, State, TL, Lag |
| `patronictl failover <cluster> --candidate <node> --force` | Manually trigger failover | Confirmation + new leader shown |
| `patronictl switchover <cluster>` | Planned graceful switchover | Interactive prompt |
| `psql -c "SELECT pg_is_in_recovery();"` | Confirm primary/replica status | `f` = primary, `t` = replica |
| `psql -c "SELECT * FROM pg_stat_replication;"` | View connected replicas | Row per replica with state and lag |
| `psql -c "SELECT now() - pg_last_xact_replay_timestamp();"` | Check replica lag duration | Interval value |
| `etcdctl endpoint health` | Verify DCS health (etcd) | `is healthy` per endpoint |
| `curl http://<host>:8008/health` | Check Patroni REST API | JSON with role and state |
| `curl http://<host>:8008/master` | HAProxy health check endpoint | 200 if leader, 503 if not |

## Escalation Path

| Condition | Escalate To | Channel |
|---|---|---|
| Failover not completing | On-call DBA Lead | PagerDuty + #db-incident |
| Data loss suspected | SRE Manager + App Owner | Direct page + incident bridge |
| DCS (etcd/Consul) unhealthy | Infrastructure On-call | PagerDuty infrastructure alert |
| Split-brain detected | On-call DBA Lead (emergency) | Direct call — do not use Slack |
