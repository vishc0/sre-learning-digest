# Replication Lag Incident Response Playbook

**Version**: 1.0  
**Applies to**: PostgreSQL replication lag AND Cassandra consistency timeout rate  
**Last reviewed**: 2026-07-06  
**Owner**: Database Reliability Engineering

---

## Overview

This playbook applies when database replication lag exceeds SLO thresholds, creating a risk of stale data reaching end users.

**SLO thresholds that trigger this playbook**:

| Database | Metric | SLO Threshold | Alert Source |
|---|---|---|---|
| PostgreSQL | Replica lag duration | > 30 seconds | `pg_last_xact_replay_timestamp()` |
| PostgreSQL | Replica lag bytes | > 512 MB | `pg_stat_replication.replay_lag` |
| Cassandra | ReplicaTimeout rate | > 0.1% of requests | Splunk metric scrape |
| Cassandra | Read repair rate spike | > 5× baseline | Cassandra JMX metrics |

**This is NOT a failover scenario.** Replication lag means the primary is healthy but replicas are behind. If the primary is down, use `database-failover-playbook.md` instead.

---

## Step 1 — Initial Triage: Quantify the Lag

Before investigating root cause, establish the precise current state of lag across all replicas.

### PostgreSQL: Measure replica lag

Run from each replica:

```bash
psql -h <replica_host> -U postgres -c "
  SELECT 
    now() - pg_last_xact_replay_timestamp() AS replication_delay,
    pg_last_xact_replay_timestamp() AS last_replayed_at,
    pg_is_in_recovery() AS is_replica
  ;
"
```

**Expected output (healthy replica)**:
```
 replication_delay | last_replayed_at              | is_replica
-------------------+-------------------------------+-----------
 00:00:00.412      | 2026-07-06 14:23:45.123+00    | t
(1 row)
```

**Expected output (lagging replica)**:
```
 replication_delay | last_replayed_at              | is_replica
-------------------+-------------------------------+-----------
 00:02:47.813      | 2026-07-06 14:21:00.001+00    | t
(1 row)
```

Run from the **primary** to see all replicas at once:

```bash
psql -h <primary_host> -U postgres -c "
  SELECT 
    application_name,
    client_addr,
    state,
    write_lag,
    flush_lag,
    replay_lag,
    pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS lag_size,
    sync_state
  FROM pg_stat_replication
  ORDER BY replay_lag DESC;
"
```

**Expected output**:
```
 application_name | client_addr | state     | write_lag | flush_lag | replay_lag | lag_size | sync_state
------------------+-------------+-----------+-----------+-----------+------------+----------+-----------
 postgres-node-3  | 10.0.2.12   | streaming | 00:00:02  | 00:00:05  | 00:02:47   | 2741 MB  | async
 postgres-node-2  | 10.0.2.11   | streaming | 00:00:00  | 00:00:00  | 00:00:01   | 1 MB     | sync
```

**Key columns**:
- `replay_lag`: time between WAL generated on primary and applied on replica — the lag users experience
- `flush_lag`: time to durably write WAL to replica disk — lag before crash safety
- `lag_size`: bytes of WAL not yet replayed — tells you whether lag is growing or stable

### Cassandra: Measure consistency timeout rate

Run this Splunk search to quantify the timeout rate:

```splunk
index=cassandra_metrics metric_name=ReplicaTimeout
| timechart span=1m sum(metric_value) AS replica_timeouts
| appendcols [
    search index=cassandra_metrics metric_name=RequestsTotal
    | timechart span=1m sum(metric_value) AS total_requests
  ]
| eval timeout_pct = round((replica_timeouts / total_requests) * 100, 4)
| table _time, replica_timeouts, total_requests, timeout_pct
```

Also check for Cassandra hints (a sign that writes are being deferred due to replica unavailability):

```bash
# From any Cassandra node
nodetool tpstats | grep -E "Hint|HintedHandoff"
```

**Expected output (healthy)**:
```
Pool Name                    Active   Pending      Completed   Blocked  All time blocked
HintedHandoffDelivery             0         0              0         0                 0
```

If `Pending` or `Blocked` HintedHandoffDelivery is non-zero: hints are backed up, indicating replicas have been lagging or unavailable.

## Step 2 — Impact Assessment

Quantify who is affected and whether the impact is business-visible.

### 2a. Identify which services read from lagging replicas

```bash
# Check application connection configs for replica hostnames
grep -r "replica\|read_replica\|read-only\|slave" /etc/app/ /etc/pgbouncer/ 2>/dev/null

# Or check PgBouncer database list
psql -p 6432 -U pgbouncer pgbouncer -c "SHOW DATABASES;" | grep -v "^$"
```

Services routing reads to replicas are the blast radius. Services that connect only to the primary are unaffected.

### 2b. Determine data staleness impact

With `replay_lag` of 2 minutes 47 seconds, users reading from that replica see data as it was 2m47s ago. Map this against your application's business operations:

| Lag Duration | Typical User-Visible Impact |
|---|---|
| < 5 seconds | Invisible — within normal async replication tolerance |
| 5s – 30s | May cause stale reads in high-frequency update scenarios (e.g., feed counts, inventory) |
| 30s – 5 minutes | Noticeable to users in real-time flows (notifications, session data, order status) |
| 5 minutes+ | Likely causing user complaints; session/auth data may be severely stale |
| Growing with no plateau | Active incident — traffic routing decision required |

### 2c. Confirm whether stale data is reaching users

Check application logs for stale-data signals:

```splunk
index=app_logs
("stale" OR "out of date" OR "cache miss" OR "inconsistency" OR "version conflict")
earliest=-30m
| stats count by sourcetype, host
| sort -count
```

Check for user-facing error signals:

```splunk
index=app_logs status=400 OR status=409 OR status=500
| timechart span=2m count
| where count > threshold
```

## Step 3 — Root Cause Identification

Use this decision sequence to identify the root cause. Do not skip ahead — each check is fast and rules out a category.

### Check A: Is this a write burst on the primary?

```bash
# On the primary: check WAL generation rate
psql -h <primary_host> -U postgres -c "
  SELECT 
    pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), '0/0')) AS total_wal_generated,
    now() AS checked_at;
"

# Run twice 60 seconds apart and compare — compute WAL per second
```

Also check:
```bash
# PostgreSQL transaction rate
psql -h <primary_host> -U postgres -c "
  SELECT sum(xact_commit + xact_rollback) AS total_txns,
         sum(tup_inserted + tup_updated + tup_deleted) AS total_writes
  FROM pg_stat_database;
"
```

**Write burst signature**: lag follows write volume spikes, lag is roughly equal across all replicas, lag is growing linearly then plateauing after the burst ends.

**Action if write burst confirmed**: Wait and monitor. Self-healing. No intervention needed unless lag exceeds 10 minutes.

### Check B: Is the replica under-resourced?

```bash
# SSH to the lagging replica and check system resources
ssh <replica_host> "iostat -x 1 5"
```

**Expected output (IO-saturated replica)**:
```
Device   r/s   w/s  rkB/s  wkB/s  await  svctm  %util
nvme0n1  0.00  850  0.00   136MB   45.2   1.17  99.4
```

`%util` consistently > 90% = IO saturation — replica cannot keep up with WAL application.

```bash
# Check CPU
ssh <replica_host> "top -b -n 1 | grep -E 'Cpu|postgres'"

# Check memory pressure
ssh <replica_host> "free -m"
ssh <replica_host> "vmstat 1 5"
```

**Resource saturation signature**: lag is specific to one or a few replicas (not all), IO wait is high, other replicas at the same write volume are keeping up.

### Check C: Is a long-running transaction blocking WAL application? (PostgreSQL)

```bash
psql -h <primary_host> -U postgres -c "
  SELECT 
    pid,
    now() - pg_stat_activity.query_start AS duration,
    query,
    state,
    wait_event_type,
    wait_event
  FROM pg_stat_activity
  WHERE state != 'idle'
    AND (now() - query_start) > INTERVAL '5 minutes'
  ORDER BY duration DESC;
"
```

**Long-running transaction signature**: one or few queries running for minutes to hours; replica lag correlates with when the transaction started.

Also check for standby conflicts (Postgres-specific — replica lag caused by conflict with long-running transactions):

```bash
psql -h <replica_host> -U postgres -c "
  SELECT 
    pid,
    query_start,
    now() - query_start AS duration,
    query,
    wait_event_type,
    wait_event
  FROM pg_stat_activity
  WHERE wait_event_type = 'Lock';
"
```

If queries on the replica are locked waiting for conflicts to resolve, this is a `recovery_min_apply_delay` or `hot_standby_feedback` issue.

### Check D: Is Cassandra compaction consuming IO? (Cassandra)

```bash
# Check compaction activity
nodetool compactionstats

# Check IO utilization on the replica node
nodetool tpstats | grep -E "CompactionExecutor|ValidationExecutor"
```

**Expected output of an active compaction storm**:
```
pending tasks: 127
          compaction type  keyspace           table                completed  total    unit    progress
                 COMPACTION app_keyspace       messages             2156899   18467291 keys    11.68%
```

127 pending tasks with 11% progress on a 18M-key table = compaction is consuming most available IO.

**Compaction storm signature**: single node lag, `nodetool compactionstats` shows many pending tasks, IO utilization > 80% on that node.

## Step 4 — Mitigation Options by Root Cause

### Mitigation A: Write burst — wait and monitor

No action required if lag is stable (not growing continuously). Monitor with:

```bash
# Watch lag trend every 30 seconds
watch -n 30 "psql -h <primary_host> -U postgres -c \
  'SELECT application_name, replay_lag, pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS lag_size FROM pg_stat_replication;'"
```

If lag is still growing after 15 minutes post-burst: escalate to Check B or C.

Consider routing reads to the primary temporarily — see Step 5.

### Mitigation B: Replica under-resourced

**Option 1 — Scale up the replica (preferred)**:

```bash
# Kubernetes: update replica resource limits
kubectl edit statefulset postgres-replica -n database
# Increase: resources.requests.memory, resources.limits.memory
# Increase: storage class IOPS if using EBS gp2/gp3

# Verify pod restarted with new limits
kubectl get pods -n database -w
```

**Option 2 — Add a new replica and remove the saturated one**:

Only if scaling up is not possible quickly. Provision a new replica with sufficient resources, verify it is streaming and caught up, then remove the saturated replica from the read pool.

**Option 3 — Reduce replica write I/O load**:
- Temporarily pause non-critical read queries against the replica
- Reschedule batch jobs that run against the replica
- Check if `autovacuum` is running aggressively on the replica and throttle it

### Mitigation C: Long-running transaction (PostgreSQL)

**Identify and evaluate**:
```bash
psql -h <primary_host> -U postgres -c "
  SELECT pid, usename, application_name, state, query_start,
         now() - query_start AS duration, query
  FROM pg_stat_activity
  WHERE state != 'idle'
  ORDER BY query_start;
"
```

**Before killing**: contact the application team or query owner if identifiable. Check if it is a business-critical operation (migration, report, batch job).

**If appropriate to terminate**:
```bash
# Graceful cancel (sends SIGINT — allows cleanup)
psql -h <primary_host> -U postgres -c "SELECT pg_cancel_backend(<pid>);"

# Forceful termination (sends SIGTERM — use if cancel does not work after 60s)
psql -h <primary_host> -U postgres -c "SELECT pg_terminate_backend(<pid>);"
```

After termination, monitor lag:
```bash
psql -h <primary_host> -U postgres -c "SELECT application_name, replay_lag FROM pg_stat_replication;"
```

Lag should begin decreasing within 30 seconds of the long transaction being cleared.

### Mitigation D: Cassandra compaction storm

**Throttle compaction to reduce IO consumption**:
```bash
# Check current throughput limit
nodetool getcompactionthroughput
# Default output: Current compaction throughput: 16 MB/s

# Reduce to throttle compaction (gives more IO headroom to replication/reads)
# Note: too low (< 8 MB/s) can cause compaction to fall further behind
nodetool setcompactionthroughput 8

# Verify
nodetool getcompactionthroughput
# Expected: Current compaction throughput: 8 MB/s
```

If compaction is causing severe timeouts and must be paused entirely (short-term only):
```bash
# Stop all compaction on the affected node — USE SPARINGLY
nodetool stop COMPACTION

# Resume when IO pressure subsides
nodetool resumecompaction
```

After IO pressure drops, monitor that ReplicaTimeout rate returns to baseline.

## Step 5 — Traffic Rerouting

If lag is severe (> 5 minutes) and impacting users, route reads to a less-lagged source. This is a temporary measure — document it.

### PostgreSQL: Reroute reads to primary

If using PgBouncer with a read replica pool:

```bash
# Identify the read-replica pool name in PgBouncer
psql -p 6432 -U pgbouncer pgbouncer -c "SHOW DATABASES;"

# Pause the replica pool (redirects connections to primary pool)
psql -p 6432 -U pgbouncer pgbouncer -c "PAUSE <replica_pool_name>;"

# Verify — connections to replica pool should now go to primary
psql -p 6432 -U pgbouncer pgbouncer -c "SHOW POOLS;"
```

Log the reroute:
```
REROUTE EVENT: [timestamp] Read traffic rerouted from postgres-node-3 to primary postgres-node-2.
REASON: replica_lag = 3m12s, exceeds SLO of 30s.
EXPECTED DURATION: Until lag clears and is stable < 30s for 15 minutes.
OWNER: [your name]
```

To restore replica reads after lag clears:
```bash
psql -p 6432 -U pgbouncer pgbouncer -c "RESUME <replica_pool_name>;"
```

### Cassandra: Reroute to a less-lagged replica

Cassandra does not have a built-in "route away from this node" command at the cluster level. Options:

1. **Raise consistency level to QUORUM** — forces reads to include more replicas, reducing the probability of hitting a stale one (at the cost of higher read latency)
2. **Remove the lagging node from the token ring temporarily** — `nodetool drain` (drastic; not recommended unless node is severely degraded)
3. **Application-level**: if the application has a configured replica preference list, remove the lagging node's IP until it recovers

## Step 6 — Resolution Criteria

**Declare the incident resolved when ALL of the following are true for 15 consecutive minutes**:

| Criterion | PostgreSQL Check | Cassandra Check |
|---|---|---|
| Lag below SLO threshold | `replay_lag < '30 seconds'` in `pg_stat_replication` | ReplicaTimeout rate < 0.1% |
| Lag not growing | Two consecutive measurements showing lag decreasing or stable | HintedHandoff pending = 0 |
| No user-visible errors | Splunk error rate at baseline | Splunk error rate at baseline |
| Alerting cleared | PagerDuty alert auto-resolved or manually resolved | Same |

### Closeout steps

1. Restore any traffic rerouting changes made in Step 5
2. Remove any compaction throttling applied in Step 4 (Cassandra):
   ```bash
   nodetool setcompactionthroughput 0   # 0 = unlimited (default)
   ```
3. Write an incident timeline in the post-incident channel
4. Update monitoring thresholds if lag was caused by a newly recognized write pattern (e.g., new feature causing 3× write volume)

## Escalation Criteria

| Condition | Action |
|---|---|
| Lag growing continuously despite mitigation for > 30 minutes | Escalate to senior DBA |
| Primary performance degrading as side effect of rerouting reads | Escalate — primary may be overloaded |
| Lag on ALL replicas simultaneously growing | Suspect primary issue — check primary health, consider failover playbook |
| Data loss risk identified (sync replica lagging) | Escalate immediately — SRE Manager + DBA lead |
| Cassandra ReplicaTimeout rate > 1% (10× SLO) | Escalate — approaching quorum risk |

## Monitoring Reference

### Splunk: Replication lag trend (PostgreSQL)

```splunk
index=postgres_metrics metric_name=replication_lag_seconds host=*replica*
| timechart span=1m max(metric_value) by host
| where max > 5
```

### Splunk: Cassandra replica timeout rate

```splunk
index=cassandra_jmx metric_name=ReplicaTimeouts OR metric_name=TotalRequests
| eval metric_type = if(metric_name="ReplicaTimeouts", "timeouts", "total")
| stats sum(metric_value) AS val by _time, host, metric_type
| eval timeout_pct = if(metric_type="timeouts", val / total_requests * 100, null)
| timechart span=2m max(timeout_pct) by host
```

### Commands reference

| Command | Platform | Purpose | Expected Output |
|---|---|---|---|
| `SELECT now() - pg_last_xact_replay_timestamp()` | PostgreSQL (replica) | Current lag duration | Interval value |
| `SELECT * FROM pg_stat_replication` | PostgreSQL (primary) | All replicas and their lag | Table per replica |
| `SELECT pg_cancel_backend(<pid>)` | PostgreSQL | Cancel a blocking query | `t` on success |
| `SELECT pg_terminate_backend(<pid>)` | PostgreSQL | Kill a blocking query | `t` on success |
| `nodetool tpstats` | Cassandra | Thread pool stats + hint backlog | Table per pool |
| `nodetool compactionstats` | Cassandra | Active/pending compactions | Pending count + progress |
| `nodetool setcompactionthroughput <mb>` | Cassandra | Throttle compaction IO | Silent success |
| `nodetool getcompactionthroughput` | Cassandra | Check current throttle | Current value in MB/s |
| `nodetool stop COMPACTION` | Cassandra | Emergency compaction stop | Confirmation |
| `nodetool resumecompaction` | Cassandra | Resume compaction | Confirmation |
