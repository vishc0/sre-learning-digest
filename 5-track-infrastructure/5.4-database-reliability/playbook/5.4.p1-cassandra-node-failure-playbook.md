# Cassandra Node Failure Response Playbook

**Version**: 1.0  
**Applies to**: Apache Cassandra clusters, Kubernetes-hosted or bare-metal  
**Last reviewed**: 2026-07-06  
**Owner**: Database Reliability Engineering

---

## Overview

This playbook applies when one or more Cassandra nodes transition to **DOWN (DN)** state — meaning the node is completely unreachable by gossip. This is distinct from a slow or degraded node, which has a separate runbook.

**Do not use this playbook for**:
- Nodes that are UP but experiencing high latency
- Cassandra process restart for configuration changes
- Planned decommission or scaling operations

**Trigger sources**: PagerDuty alert `cassandra_node_down`, Splunk alert on `nodetool status` scrape showing DN, or direct Kubernetes pod failure alert.

---

## Severity Classification

| Scenario | Severity | Action |
|---|---|---|
| 1 node down, RF=3, 2+ nodes up per token range | SEV-3 | Follow this playbook — cluster is fully operational |
| 1 node down, RF=3, only 2 nodes up per token range | SEV-2 | Follow this playbook urgently — one more failure = data loss risk |
| Multiple nodes down, still above RF quorum floor | SEV-2 | Follow this playbook, escalate immediately after assessment |
| Nodes down >= floor(RF/2) + 1 per token range | **SEV-1** | **ABORT — see Abort Criteria section. Escalate now.** |
| All nodes in a DC down | SEV-1 | ABORT — escalate immediately, invoke DR plan |

**RF quorum floor formula**: `floor(RF/2) + 1` nodes must be UP for writes at QUORUM to succeed.  
Example: RF=3 requires 2 nodes UP. If only 1 is UP, QUORUM writes fail.

## Step 1 — Assess: Determine Cluster State

Run from any live Cassandra node or from a host with `nodetool` configured:

```bash
nodetool status
```

**Expected output format**:
```
Datacenter: dc1
===============
Status=Up/Down
|/ State=Normal/Leaving/Joining/Moving
--  Address         Load       Tokens  Owns (effective)  Host ID                               Rack
UN  10.0.1.10       245.67 GiB  256    33.3%             a1b2c3d4-...                          rack1
UN  10.0.1.11       251.22 GiB  256    33.4%             b2c3d4e5-...                          rack1
DN  10.0.1.12       ?           256    33.3%             c3d4e5f6-...                          rack1
```

**What to look for**:
- `DN` (Down Normal) = node is down but was previously healthy. This is the typical failure case.
- `DL` (Down Leaving) = node was in the process of decommissioning when it failed.
- `DJ` (Down Joining) = node failed during bootstrap. Different recovery path — do not follow this playbook for bootstrap failures.
- `?` in Load column = node is unreachable; nodetool cannot collect stats.

**Record the following before proceeding**:
- Hostname / IP of the DN node: `__________________`
- Number of total nodes in the datacenter: `__________________`
- Number of UP nodes remaining: `__________________`
- Replication factor configured: `__________________`
- Token ranges served by the DN node (approximate): use `nodetool ring | grep <DN_IP>`

**Check RF for each keyspace**:
```bash
nodetool describering <keyspace_name>
```

Or from `cqlsh`:
```sql
SELECT keyspace_name, replication FROM system_schema.keyspaces;
```

Look for keyspaces with `'replication_factor': '1'` — those keyspaces have **zero redundancy** and data is unavailable while that node is down.

## Step 2 — Determine Impact

### Consistency level degradation

With one node DN in a RF=3 cluster:

| Consistency Level | Status | Notes |
|---|---|---|
| ONE | Fully available | Reads and writes succeed |
| LOCAL_QUORUM (2/3) | Fully available | Still achievable with 2 UP nodes |
| QUORUM | Fully available | Same as LOCAL_QUORUM for single-DC |
| ALL | **Degraded** | Requires all nodes — will fail |
| EACH_QUORUM | Degraded if multi-DC | Depends on other DCs |

### Client error patterns to check in Splunk

Run this Splunk search to identify client-facing impact:

```splunk
index=cassandra_app sourcetype=app_log 
(error="NoHostAvailableException" OR error="UnavailableException" OR error="WriteTimeoutException" OR error="ReadTimeoutException")
| timechart span=1m count by error
| where count > 0
```

Also check driver-level errors:
```splunk
index=cassandra_driver host=* 
("connection refused" OR "Connection to" OR "All host(s) tried")
earliest=-30m
| stats count by host, error_msg
```

**Signals that indicate user-visible impact**:
- `UnavailableException`: consistency level cannot be met — users are receiving errors
- `WriteTimeoutException`: writes are timing out — data may or may not have been written
- `ReadTimeoutException`: reads are timing out — users may see missing data or errors

**If UnavailableException rate is non-zero**: escalate severity immediately, notify application teams, and move to Step 3 with urgency.

## Step 3 — Decision Tree: Restart vs. Replace

```
Is the Cassandra node a Kubernetes Pod?
├── YES → Can the pod be rescheduled/restarted?
│         ├── YES → Go to Step 4 (Pod restart)
│         └── NO → Go to Step 5 (Node replacement)
└── NO (bare metal / VM)
      ↓
Is the host OS reachable via SSH?
├── NO → Hardware failure suspected → Page infrastructure team
│         → If host recovers, go to Step 4
│         → If host does not recover within RTO → go to Step 5
└── YES
      ↓
Is the Cassandra process running?
  (check: systemctl status cassandra OR ps aux | grep cassandra)
├── NO (process crashed) → Go to Step 4 (process restart)
└── YES (process running but node is DN)
      ↓
Check Cassandra logs for clue:
  /var/log/cassandra/system.log (last 500 lines)
  Look for: OOM, disk full, gossip failure, corrupt SSTable
      ↓
├── OOM / resource exhaustion → Fix resource issue, then Step 4
├── Disk full → Free disk space (delete logs, old snapshots), then Step 4
├── Corrupt SSTable → Do NOT restart; escalate to Sr. DBA — Step 5 path likely
└── Network partition → Verify network, resolve, then Step 4
```

## Step 4 — Node Restart Attempt

### 4a. Kubernetes pod restart

```bash
# Identify the pod
kubectl get pods -n cassandra -o wide | grep <NODE_IP>

# Record the pod name
POD_NAME=<cassandra-pod-name>

# Delete the pod to trigger restart (assumes StatefulSet manages it)
kubectl delete pod $POD_NAME -n cassandra

# Watch the pod come back up
kubectl get pods -n cassandra -w
```

**Expected output during restart**:
```
NAME               READY   STATUS      RESTARTS   AGE
cassandra-2        0/1     Terminating   0         2d
cassandra-2        0/1     Pending       0         0s
cassandra-2        0/1     Init:0/1      0         3s
cassandra-2        0/1     Running       0         12s
cassandra-2        1/1     Running       0         45s
```

The pod is **not ready** until the readiness probe passes. This typically takes 30–90 seconds after the process starts, as Cassandra must complete gossip initialization.

### 4b. Bare-metal / VM process restart

```bash
# Restart the service
sudo systemctl restart cassandra

# Follow the startup log
sudo journalctl -u cassandra -f
# OR
sudo tail -f /var/log/cassandra/system.log
```

**Key log lines to confirm successful rejoining**:
```
INFO  [main] Starting up server gossip
INFO  [HANDSHAKE-/10.0.1.10] Handshake with /10.0.1.10
INFO  [main] Cassandra starting up...
INFO  [main] Not starting RPC server as requested. Use JMX (StorageService->startRPCServer()) or nodetool (enablethrift) to start it
INFO  [main] Node /10.0.1.12 state jump to NORMAL
```

The critical line is `state jump to NORMAL`. If you see `JOINING` for more than 5 minutes without progressing, check streaming with `nodetool netstats`.

### 4c. Verify cluster status after restart

```bash
nodetool status
```

**Expected output after successful recovery**:
```
Datacenter: dc1
===============
--  Address         Load       Tokens  Owns (effective)  Host ID
UN  10.0.1.10       245.67 GiB  256    33.3%             a1b2c3d4-...
UN  10.0.1.11       251.22 GiB  256    33.4%             b2c3d4e5-...
UN  10.0.1.12       251.00 GiB  256    33.3%             c3d4e5f6-...
```

All nodes show `UN` (Up Normal). If the recovered node shows `UJ` (Up Joining) for more than 10 minutes, monitor streaming with `nodetool netstats` — see Step 5b for details on monitoring streaming.

**If restart succeeds**: skip to Step 6 (post-recovery).  
**If restart fails after 2 attempts**: proceed to Step 5 (node replacement).

## Step 5 — Node Replacement (Restart Failed)

### 5a. Decommission the old node (if reachable)

Only decommission if the node is still reachable via nodetool. If it is not reachable, skip to 5b.

```bash
# Run from the FAILED node if possible, or from another node targeting the failed one
nodetool decommission

# If the node is unreachable, remove it from the ring from a healthy node
nodetool removenode <HOST_ID_OF_FAILED_NODE>
```

Get the Host ID from the `nodetool status` output recorded in Step 1.

**Expected output of `nodetool removenode`**:
```
RemovalStatus: No token removal in process
```

Then monitor completion:
```bash
nodetool netstats
```

Watch until streaming shows `0 bytes remaining`.

### 5b. Bootstrap a new node

Provision a new host (or Kubernetes pod with a fresh PVC) with the same Cassandra version. Configure `cassandra.yaml`:

```yaml
# cassandra.yaml on the NEW node
cluster_name: '<your-cluster-name>'
seed_provider:
  - class_name: org.apache.cassandra.locator.SimpleSeedProvider
    parameters:
      - seeds: "10.0.1.10,10.0.1.11"   # Use LIVE seed nodes, NOT the failed one

# If replacing a node that held specific tokens, set:
replace_address_first_boot: 10.0.1.12   # IP of the failed node
```

Start Cassandra on the new node:
```bash
sudo systemctl start cassandra
sudo journalctl -u cassandra -f
```

### 5c. Monitor bootstrap progress

From any healthy node:
```bash
nodetool netstats
```

**Expected output during bootstrap streaming**:
```
Mode: JOINING
Bootstrap 10.0.1.13/10.0.1.13
    Receiving 14 files, 12.35 GiB total. Already received 8.22 GiB
    Sending 0 files, 0 bytes total
Active Receiving: 14, Sending: 0
```

**Expected output when streaming is complete**:
```
Mode: NORMAL
Nothing happening (idle)
```

Bootstrap time depends on data volume. At 25M msg/day throughput, a node with ~250 GiB of data may take 1–3 hours to stream. Monitor every 15 minutes.

```bash
# Quick progress check — percentage streamed
nodetool netstats | grep -E "Receiving|files"
```

### 5d. Confirm new node in ring

```bash
nodetool status
```

New node should appear as `UN` with the same token ownership (~33% for a 3-node cluster).

## Step 6 — Post-Recovery

### 6a. Run repair on the recovered node

After a node rejoins (either via restart or replacement), it may have missed writes that occurred while it was down. Run repair to catch up:

```bash
# Run repair on the recovered node — this streams missed data from peers
nodetool repair -pr <keyspace_name>
```

The `-pr` flag limits repair to the primary token range of this node, avoiding redundant work across the cluster.

**Expected output**:
```
[2026-07-06 14:23:01,112] Starting repair command #1
[2026-07-06 14:23:01,114] Starting repair session
[2026-07-06 14:23:45,890] Repair session completed successfully
```

Repair duration scales with data volume. Monitor with:
```bash
nodetool compactionstats
```

### 6b. Verify cluster is fully healthy

```bash
# All nodes UP Normal
nodetool status

# Check for any pending repairs or compactions
nodetool compactionstats

# Verify keyspace consistency
nodetool describering <keyspace_name>
```

### 6c. Update capacity tracking

- Record incident in the runbook log
- Update node inventory (IP, Host ID, rack assignment)
- If a node was permanently replaced: update asset management and DNS records
- Clear any suppressed PagerDuty alerts

### 6d. Verify client error rates have returned to baseline

```splunk
index=cassandra_app sourcetype=app_log
(error="UnavailableException" OR error="WriteTimeoutException" OR error="ReadTimeoutException")
| timechart span=5m count by error
```

Confirm the error rate trend is returning to zero.

## Abort Criteria — SEV-1 Escalation

**If the number of DOWN nodes in any datacenter satisfies:**

```
down_nodes >= floor(RF / 2) + 1
```

For RF=3: 2 or more nodes down simultaneously.  
For RF=5: 3 or more nodes down simultaneously.

**Do NOT attempt repair or recovery procedures. Instead:**

1. **Page the on-call DBA lead and SRE Manager immediately** — do not attempt to fix alone
2. **Communicate to application teams**: database is in degraded mode, writes at QUORUM will fail
3. **Consider switching application to LOCAL_ONE or ONE** consistency if the application can tolerate stale reads
4. **Do not restart multiple nodes simultaneously** — this can worsen the situation
5. **Preserve logs** before any intervention: `cp /var/log/cassandra/system.log /var/log/cassandra/system.log.incident.$(date +%s)`
6. **Open a war room bridge** and begin stakeholder communication

## Commands Reference

| Command | Purpose | Expected Output |
|---|---|---|
| `nodetool status` | View all node states | Table with UN/DN per node |
| `nodetool netstats` | Monitor streaming during bootstrap or repair | Files remaining, bytes transferred |
| `nodetool repair -pr <keyspace>` | Repair missed data on recovered node | Session start/complete log |
| `nodetool decommission` | Gracefully remove a node from the ring | Status messages, streaming begins |
| `nodetool removenode <host_id>` | Remove an unreachable node from the ring | RemovalStatus confirmation |
| `nodetool compactionstats` | View compaction and repair progress | Active tasks with bytes/files |
| `nodetool setcompactionthroughput <mb>` | Throttle compaction I/O | No output (silent success) |
| `nodetool describering <keyspace>` | View token ranges and replication | Token range to endpoint mapping |

## Escalation Path

| Condition | Escalate To | Channel |
|---|---|---|
| SEV-1 (quorum loss) | On-call DBA Lead + SRE Manager | PagerDuty P1 + #incident-bridge |
| Bootstrap > 4 hours with no progress | Senior DBA | Slack #cassandra-ops |
| Data loss suspected (keyspace RF=1) | SRE Manager + App Owner | Direct page |
| Corrupt SSTable detected in logs | Senior DBA | Slack #cassandra-ops |
