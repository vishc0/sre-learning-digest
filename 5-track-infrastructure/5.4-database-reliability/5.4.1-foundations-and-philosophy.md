# Chapter 1: Why Databases Are the Hardest Reliability Problem

**Series**: Database Reliability Engineering
**Audience**: Director/VP Platform Engineering interview prep
**Level**: Principal SRE → Director transition

---

## The Core Problem: State Is Not Replaceable

Every reliability pattern you have applied to stateless services — kill the pod, reschedule it, let the load balancer drain the connection — breaks the moment you apply it to a database.

The reason is fundamental: **stateless services compute; databases remember.**

When a broken application pod is killed, no information is lost. The pod held no truth. When a database node dies mid-write, it may hold the only committed copy of a transaction that the application has already confirmed to the user. Killing and restarting does not fix that. Recovery does — and recovery takes time, requires coordination, and can fail.

This is why database reliability is categorically harder than application reliability. The SRE instinct of "make it crash-safe by making it stateless" does not apply. The database **is** the state.

The implication for a platform team: your database reliability investment is never finished. It compounds. Every recovery you have never practiced is a debt that pays out at 3 AM.

## Failure Modes Taxonomy

Not all database failures are equal. Director-level thinking requires knowing which failures are recoverable in seconds versus hours, and which ones are silent until they are catastrophic.

| Failure Type | Impact Radius | Detection Lag | Recovery Complexity |
|---|---|---|---|
| Node failure (primary down) | Full write outage until failover | Low — health checks fire quickly | Medium — Patroni/automated failover, replica promotion |
| Disk failure (data volume lost) | Total data loss on that node | Low — I/O errors surface fast | High — PITR restore from WAL archive |
| Network partition | Split-brain risk; potential dual-primary | Medium — depends on fencing timeout | High — manual arbitration often required |
| Corrupt write (silent corruption) | Unknown until read or checksum | Very High — can persist for days/weeks | Very High — requires point-in-time recovery to pre-corruption state |
| Slow query cascade | Full connection pool exhaustion; application-wide outage | Medium — query latency metrics lag the root cause | Medium — kill blocking queries, tune work_mem, connection drain |
| Connection pool exhaustion | Application writes rejected; reads may succeed | Low — errors surface immediately | Low — connection kill or PgBouncer recycle, but root cause investigation needed |
| Schema migration failure | Locked tables; writes blocked; partial schema state | Low — migration runner fails visibly | Medium — rollback plan complexity depends on migration type |

**The dangerous column is Detection Lag.** Node failure is easy — your monitoring sees it in seconds. Silent corruption is the failure mode that ends careers. A database that returns wrong data is worse than a database that returns no data, because applications continue operating against a lie.

## ACID vs BASE: An Operational Lens

Computer science describes ACID and BASE as consistency models. Operations should describe them as **guarantees you pay for** versus **bets you make on reconciliation**.

**ACID** (Atomicity, Consistency, Isolation, Durability) is a contract the database makes with the application: every transaction either fully commits or fully rolls back; committed data will survive a crash; concurrent transactions will not corrupt each other's view. You pay for this in write throughput, lock contention, and latency.

**BASE** (Basically Available, Soft state, Eventually consistent) is the model used when you need to scale writes across many nodes faster than ACID allows. The database says: "I'll accept your write now, and I'll reconcile conflicts later." You pay for this in application complexity — your code has to handle the possibility that two nodes disagree about the current state of the world.

The RabbitMQ analogy makes this concrete. RabbitMQ with `ack-required=false` is BASE: the broker accepts the message, promises eventual delivery, and may deliver it more than once. A financial transaction system — where double-posting $50,000 is a firing offense — is ACID. The same message broker with `persistent=true`, publisher confirms, and dead-letter queues is moving toward ACID guarantees. You pay in throughput and latency, but you buy correctness.

**Operational implication**: when a developer proposes switching from PostgreSQL to a distributed NoSQL store to "scale better," the Director question is: "What consistency guarantee are you giving up, and which part of the business logic will now have to handle reconciliation?" The answer is always "something" — and that something usually belongs to the platform team to operationalize.

## CAP Theorem: Every Config Is a Choice

The CAP theorem states that a distributed system cannot simultaneously guarantee Consistency, Availability, and Partition tolerance — choose two when a network partition occurs.

This is not an academic constraint. **Every distributed database configuration you write is a CAP choice**, whether you are aware of it or not.

When you configure Cassandra with `consistency=QUORUM`, you are saying: "During a partition, I prefer to refuse reads and writes rather than return stale data." That is CP — you chose Consistency over Availability.

When you configure Cassandra with `consistency=ONE`, you are saying: "During a partition, return whatever the closest node has, even if it's stale." That is AP — you chose Availability over Consistency.

Neither is wrong. Both are **decisions** — and decisions have owners. The Director-level failure is not choosing the wrong consistency level; it is deploying a distributed database without the team understanding what CAP choice was made and why.

This surfaces in incidents as: "Why did two users see different account balances for 30 seconds?" Because your configuration chose AP. If the business cannot accept that, the configuration is wrong. If the business can accept it, document the choice and build compensating controls.

## Director Decision Frame

> **"The question is not whether your database will fail. The question is whether your team has practiced the recovery, and whether your architecture limits blast radius."**

Every database reliability program starts here. Replicas you have never promoted are not an HA solution — they are a disaster waiting for its first test. PITR you have never restored is not a recovery option — it is an assumption about your tooling that has never been validated under pressure.

The Director asks three questions on inheriting a platform:

1. **When did we last run a failover drill, and what broke?**
2. **What is our actual RTO — measured, not estimated?**
3. **If the primary goes down right now, who makes the call to promote the replica, and how long does it take?**

If no one can answer questions 2 and 3 with specifics, the reliability program starts there — not with adding more replicas.
