# Database Failure Mode Matrix

> Dense operational reference. RTO estimates assume a healthy cluster at baseline with on-call runbooks in place. Actual RTO varies by data volume, team response time, and automation maturity.

---

## Cassandra

| Failure Type | Impact | Blast Radius | Detection Signal | Detection Lag | Recovery Action | RTO Estimate | Prevention |
|---|---|---|---|---|---|---|---|
| **Single node crash** | At RF=3 + QUORUM: reads and writes continue using remaining 2 replicas. At RF=3 + ALL: reads and writes fail immediately. At CL=ONE: no impact unless that node is the only replica for a partition. | Partitions where crashed node holds 1 of RF replicas. Ops using CL=ALL fail across all partitions. | `nodetool status` shows DN. Spike in `ReplicaTimeouts` and `ReadTimeouts` in Cassandra metrics. Gossip failure detection (default: 30–120s). | 30–120s (gossip failure detection window) | Restart Cassandra process if transient. If hardware failure: `nodetool removenode <host_id>` from live node, then add replacement, then repair. | 5 min (restart) to 4+ hrs (full data stream for replacement node) | RF ≥ 3. Never use CL=ALL in production. Monitor with `nodetool status` alerts. |
| **Network partition** | Nodes on either side of the partition may continue accepting writes if they can still reach quorum within their partition segment. With CL=QUORUM and RF=3, a 2/1 split means the majority side continues; minority side rejects writes. Data divergence possible if both sides reach local quorum independently. | Depends on split topology and consistency level. Worst case: silent split-brain where both sides accept conflicting writes. | `nodetool status` returns inconsistent results depending on which node you query (different views of the ring). Gossip delays. Increased `ReadRepairAttempted` as anti-entropy triggers. | Minutes (gossip propagation delay) | Restore network connectivity. Run `nodetool repair` across all nodes post-partition to reconcile diverged data. Check for `UncertainCommit` errors in application logs (CAS operations are particularly vulnerable). | Network fix: immediate. Data reconciliation: 30 min to hours depending on divergence scope. | Use CL=QUORUM or higher. Never CL=ONE for writes where consistency matters. Rack/AZ-aware placement so partitions split cleanly. |
| **Replica lag / replica failure** | Cassandra does not have traditional replication lag — it uses hinted handoff and read repair. If a replica is consistently slow or unavailable, hints accumulate on coordinator nodes. After `max_hint_window_in_ms` (default 3h), hints are dropped and that replica falls out of sync silently. | Replicas that were down > 3h will silently serve stale data once they return. Applications using CL=ONE from that replica will read inconsistent data. | `nodetool tpstats` shows `HintedHandoff` backlog. `nodetool info` shows hints stored. Metrics: `StorageProxy.HintsInProgress`. | Minutes to hours (hints accumulate silently) | Run `nodetool repair` on the recovered node immediately after it returns. If node was down > 3h, treat as if it missed all writes during that window — full repair required. | 0 min (read/write continue via other replicas) + repair time (30 min to hours) | Monitor hint accumulation. Alert on `HintsInProgress` > threshold. Automate repair trigger on node recovery. |
| **Disk full** | Cassandra will reject writes when disk usage exceeds `disk_failure_policy` threshold (default: `stop_commit_log` at 95%). Compaction fails, causing SSTable count to grow, causing read latency to spike. | All writes to tables on that node fail. Node may stop. | `df -h` on the node. Cassandra logs: `Aborting compaction ... no disk space`. Metrics: `Storage.Load` approaching disk capacity. | Minutes (compaction failure is logged immediately) | Free disk space: delete old snapshots (`nodetool clearsnapshot`), increase `compaction_throughput_mb_per_sec` to catch up, or add disk. Do NOT delete SSTables manually. | 15–60 min (if space freed quickly) | Monitor `Storage.Load` at 70%/80%/90% thresholds. Automate snapshot cleanup. Capacity plan for compaction overhead (active data × 2 for compaction space). |
| **Corrupt write** | Cassandra uses CRC checksums on commit log and SSTable files. A corrupt SSTable causes reads from that node to fail with `CorruptSSTableException`. Other replicas are unaffected. | Reads from that node that touch the corrupt SSTable. At CL=ONE from that node: read errors. At CL=QUORUM: reads succeed via other replicas. | Cassandra logs: `Corrupted: /path/to/sstable`. `nodetool scrub` output. `Storage.Exceptions` metric increments. | Seconds (logged immediately on read) | Run `nodetool scrub <keyspace> <table>` — removes corrupt rows and logs what was dropped. Then run `nodetool repair` to repopulate from other replicas. | 30–90 min (scrub + repair) | Enable commitlog checksumming (default ON in C* 2.2+). RAID or EBS replication for disk. Monitor `Storage.Exceptions`. |
| **Connection pool exhaustion** | Client driver cannot open new connections to a Cassandra coordinator. New queries fail with `NoHostAvailableException`. Existing in-flight queries complete. | All new requests from the exhausted client pool fail. Other clients unaffected unless all Cassandra ports are saturated. | Driver logs: `NoHostAvailableException`. `nodetool tpstats` NativeTransportRequests pool. TCP connection count via `ss -s` on the Cassandra node. | Seconds (client-side exception) | Increase `native_transport_max_threads` in cassandra.yaml (requires restart). Tune driver connection pool settings (connections per host, requests per connection). Identify and kill long-running queries holding connections. | 5–15 min (config + rolling restart) | Right-size driver connection pools. Monitor open connection count. Set query timeouts so hung queries self-terminate. |
| **Long-running query / lock contention** | Cassandra uses lightweight transactions (CAS) which can block subsequent CAS operations on the same partition key. Non-CAS reads/writes are not blocked by LWT. Range queries with large result sets can cause read timeouts. | CAS operations on the same partition. Large range scans can starve the ReadStage thread pool. | `nodetool tpstats` ReadStage PendingTasks growing. `nodetool proxyhistograms` p99/max diverging. Application timeout errors. | Seconds (metrics and errors appear immediately) | Identify the slow query from `system.local` or application tracing. Kill if possible (no explicit kill in Cassandra — must timeout). Tune `read_request_timeout_in_ms`. Optimize query (add indexes, reduce range). | 1–10 min per incident | Set `read_request_timeout_in_ms` and `write_request_timeout_in_ms` appropriately. Avoid full-table scans. Use paging. Monitor `proxyhistograms` p99. |
| **Schema migration failure** | DDL propagates via gossip. If gossip is degraded or a node is down, some nodes may not receive the schema update. Result: schema version mismatch. Queries to nodes with old schema may fail or return unexpected results. | Any query that touches the new column/table from a node that hasn't applied the schema update. | `nodetool describecluster` shows > 1 schema version UUID. Cassandra logs: `Schema version mismatch`. | Seconds to minutes (describecluster reflects immediately) | Wait for automatic gossip propagation (usually resolves in < 60s). If stuck: restart the lagging node, or run `nodetool resetlocalschema` on affected node. | 1–10 min (automatic) to 30 min (manual restart) | Run migrations during low-traffic windows. Always verify `describecluster` shows one schema UUID before and after migration. Never run concurrent DDL operations. |

## PostgreSQL

| Failure Type | Impact | Blast Radius | Detection Signal | Detection Lag | Recovery Action | RTO Estimate | Prevention |
|---|---|---|---|---|---|---|---|
| **Single node crash** | In a single-node setup: all reads and writes fail. In a Patroni HA setup: failover triggers automatically — see Primary/leader crash row. | All connected applications lose their connections. PgBouncer queues connections during failover. | Application connection errors. `pg_isready` returns failure. Patroni leader key disappears from etcd/Consul. PostgreSQL process not running in systemd. | Seconds (application-level); Patroni TTL: 30s (default) | Restart PostgreSQL if transient. If primary in HA: Patroni promotes a replica automatically. Update connection strings if not using PgBouncer/HAProxy. | 30s–2 min (Patroni auto-failover) | Patroni HA. PgBouncer in front of all connections. Connection string points to Patroni VIP, not individual node. |
| **Primary/leader crash** | Writes fail immediately. Reads from standby may continue if application has read-replica routing. Patroni detects leader TTL expiry and initiates election — promotes the most up-to-date standby. During failover window (30–60s default): all writes fail. | All write traffic. Read traffic if not split across replicas. | Patroni logs: `Leader key has been lost`. Application: `FATAL: connection to server lost`. `pg_is_in_recovery()` returns false on what was the primary. | Patroni TTL: 30s (configurable) | Automatic if Patroni is configured correctly. Manually: `patronictl -c /etc/patroni/patroni.yml failover <cluster>`. Verify old primary is isolated before promoting new leader (split-brain prevention). | 30–90s (automatic Patroni failover) | Patroni with etcd/Consul DCS. Monitor Patroni leader key age. Set `synchronous_mode: true` for zero data loss (at cost of write latency). |
| **Replica lag / replica failure** | Reads from lagging replica return stale data — this is a silent failure from the application's perspective if read routing does not check lag. If replica crashes: reads routed to it fail. Primary is unaffected (async replication). | Read traffic routed to the lagging/failed replica. Application-level data inconsistency for reads (phantom stale reads). | `pg_stat_replication.write_lag` or `replay_lag` growing on primary. On replica: `pg_last_xact_replay_timestamp()` delta. Replica: `pg_is_in_recovery()` returns false unexpectedly. | Lag is continuous and measurable. Failure detection: seconds (connection drop). | For lag: identify slow queries or I/O on replica, increase `max_wal_senders`, reduce query load on replica. For failure: remove replica from read pool, restart, re-synchronize with `pg_basebackup`. | Lag reduction: minutes to hours. Replica resync: 30 min to hours (depends on data size). | Alert on `replay_lag > 10s`. Read routing layer must check lag before sending reads to replica. |
| **Disk full** | PostgreSQL stops writing WAL when disk is full. All writes fail with `ERROR: could not write to file`. If WAL volume is separate, only WAL writes fail first (still catastrophic). | All write operations. If WAL disk is separate: WAL first, then checkpoint fails, then everything halts. | PostgreSQL logs: `No space left on device`. `df -h` on the PostgreSQL data directory and WAL directory. | Seconds (writes fail immediately when disk is full) | Free disk space: clear old WAL archives (`pg_archivecleanup`), remove bloated tables (VACUUM FULL if space allows), expand disk. `CHECKPOINT` after freeing space. | 15–60 min | Monitor disk at 70%/80%/90%. Separate WAL volume from data volume. Set `wal_keep_size` appropriately. Automate WAL archive cleanup. |
| **Network partition** | In Patroni: if primary loses connection to DCS (etcd/Consul) but not to application, Patroni demotes primary to prevent split-brain. Application writes fail during the split. In async replication: replica falls behind without error. | Primary write traffic during DCS partition. Replica lag accumulates silently. | Patroni logs: `Lost connection to DCS`. Primary demotes itself (writes fail). Application: connection errors. | Patroni DCS TTL: seconds to tens of seconds | Restore network. Patroni re-elects leader automatically. Verify no split-brain occurred (`pg_is_in_recovery()` on both nodes). | Network fix: immediate. Leader election: 30–90s | Patroni's DCS dependency is by design — it prevents split-brain. Ensure DCS (etcd) is in a separate failure domain from PostgreSQL nodes. |
| **Corrupt write** | PostgreSQL detects page corruption via checksums (if `data_checksums=on`, enabled at `initdb` time). Reads from corrupt pages fail with `ERROR: invalid page header`. Other pages unaffected. | Tables with corrupt data blocks. Queries that touch those blocks. | PostgreSQL logs: `invalid page in block X of relation Y`. `pg_dump` failures during backup. | Seconds (error on read) | `VACUUM FULL` the affected table (rewrites all pages). If VACUUM FULL fails, restore from backup. Run `pg_filedump` to identify scope of corruption. | 30 min to hours (restore from backup) | Enable `data_checksums` at initdb time (cannot enable post-init without `pg_checksums`). Regular pg_dump + pg_restore validation. Monitor PostgreSQL error logs. |
| **Connection pool exhaustion** | PgBouncer queue fills up. New connection attempts fail with `sorry, too many clients`. Existing connections continue. | All new database connections from application. | PgBouncer stats: `SHOW POOLS` — `cl_waiting` growing. PostgreSQL: `pg_stat_activity` count approaching `max_connections`. Application: `too many clients` errors. | Seconds (immediate when pool is full) | Kill idle connections: `SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state='idle' AND ...`. Tune PgBouncer pool size. Increase `max_connections` (requires restart). Switch PgBouncer to transaction mode if in session mode. | 5–15 min (kill idle + tune) | PgBouncer in transaction mode (multiplexes connections). Monitor `cl_waiting`. Set `idle_in_transaction_session_timeout` in PostgreSQL to auto-kill hung sessions. |
| **Long-running query / lock contention** | Long-running queries hold row-level locks that block subsequent DML on the same rows. `idle in transaction` sessions hold locks without doing work. Lock chains can cascade: A blocks B blocks C. Table-level locks (DDL, VACUUM FULL) block all access to the table. | Any transaction waiting for a lock held by the slow query. Cascade can affect entire application write path. | `pg_locks` with `granted=false`. `pg_stat_activity` shows `waiting=true` or `lock_type`. `pg_blocking_pids(pid)` shows the blocker chain. | Seconds (lock wait appears immediately) | Identify blocker: `SELECT pg_blocking_pids(pid) FROM pg_stat_activity WHERE wait_event_type='Lock'`. Kill the head of the chain: `pg_terminate_backend(<pid>)`. | 1–5 min (identify + kill) | Set `lock_timeout` and `statement_timeout` per session/role. Set `idle_in_transaction_session_timeout`. Monitor `pg_stat_activity` for long-running transactions. |
| **Schema migration failure** | Failed migration may leave the schema in a partial state — some indexes created, some columns added, transaction rolled back if run in a transaction block. If migration runs outside a transaction: partial state is committed and may require manual cleanup. | Tables touched by the migration. Application deployments that depend on the new schema may fail. | Migration tool output (Flyway/Liquibase error). `pg_stat_activity` showing the migration connection stuck. Table/index in `pg_class` with `indisvalid=false` (for failed index creation). | Seconds (migration tool reports error) | If transactional migration: rollback is automatic. If non-transactional: identify what was applied via migration history table, manually revert partial changes, mark migration as failed. | 15–60 min (manual cleanup for non-transactional failures) | Wrap all migrations in explicit transactions where possible. Test migrations in staging against production-scale data. Use `CREATE INDEX CONCURRENTLY` for large tables to avoid table locks. |

## Redis

| Failure Type | Impact | Blast Radius | Detection Signal | Detection Lag | Recovery Action | RTO Estimate | Prevention |
|---|---|---|---|---|---|---|---|
| **Single node crash** | In standalone mode: all reads and writes fail. In Sentinel: automatic failover to replica (default 30s). In Cluster: shard goes down — keys in that shard unavailable. | All keys in standalone. Keys in affected shard for Cluster. All traffic during Sentinel failover window. | Redis process not responding to PING. Sentinel logs: `+sdown` (subjectively down), then `+odown` (objectively down = quorum reached). Application: `Connection refused`. | Sentinel: 30s (default `down-after-milliseconds`) | Sentinel promotes replica automatically. Application reconnects. Verify new primary with `redis-cli -p 26379 SENTINEL masters`. | 30–60s (Sentinel failover) | Redis Sentinel (min 3 Sentinel nodes). AOF persistence for durability. Monitor via Sentinel events. |
| **Primary/leader crash** | Sentinel detects primary down after quorum. Promotes replica with least replication lag. During failover window: all writes fail. After failover: writes succeed on new primary. Data written after last sync to the promoted replica is lost (async replication). | All write traffic during failover window. Potential data loss for writes that were not yet replicated. | Sentinel: `+try-failover` event in logs. Application: write errors. `redis-cli INFO replication` shows new primary. | Sentinel TTL: 30s (configurable) | Automatic via Sentinel. Manual: `SENTINEL FAILOVER <master-name>` from Sentinel CLI. | 30–60s | Sentinel quorum ≥ 3. Set `min-replicas-to-write 1` to prevent primary accepting writes without at least one synchronized replica. |
| **Replica lag / replica failure** | If Redis is used as a cache only: replica lag is tolerable (stale reads from replica). If Redis is a durable store: replica lag = data inconsistency window = data loss exposure on primary failure. Replica failure: Sentinel removes it from pool. | Reads routed to lagging replica. Data loss exposure equals lag amount if primary fails during lag period. | `redis-cli INFO replication`: `master_repl_offset` vs `slave_repl_offset` delta. `redis_repl_backlog_size` vs bytes used. | Continuous (lag is measurable in real time) | Identify and resolve the cause (replica under load, network congestion). Trigger `REPLICAOF NO ONE` then `REPLICAOF <primary>` to force re-sync if replica is stuck. | Minutes (if cause is resolved quickly) | Monitor replication offset delta. Alert when lag > threshold. Size `repl-backlog-size` large enough to absorb short disconnections without forcing full resync. |
| **Disk full** | Redis AOF rewrite fails. If `appendonly yes` and AOF rewrite cannot complete, eventually Redis may stop accepting writes (if configured conservatively) or corrupt the AOF file. RDB snapshots (BGSAVE) fail silently — last successful snapshot is intact. | Write durability: new writes may not be persisted. If Redis crashes after disk full, recovery uses last successful snapshot. | Redis logs: `BGSAVE failed because of fork error` or `AOF rewrite failed: No space left on device`. | Seconds (logged immediately) | Free disk space. Trigger `BGSAVE` after recovery. Verify AOF integrity with `redis-check-aof`. | 15–30 min | Separate AOF/RDB persistence volume from OS volume. Monitor disk at 70%/80%. Set `stop-writes-on-bgsave-error yes` (default) so disk failures are visible rather than silent. |
| **Network partition** | In Sentinel: if primary is partitioned from Sentinels, Sentinels promote a replica. When partition heals, old primary may accept writes briefly before realizing it was demoted — those writes are lost. Redis Cluster: nodes on minority side reject writes (cluster requires majority of masters reachable). | Write traffic during partition. Data loss for writes accepted by old primary before it realizes it was demoted. | Sentinel logs: `+sdown`, `+odown`, failover events. Cluster: `CLUSTER INFO cluster_state` returns `fail` on minority side. | Sentinel: 30s (configurable) | Sentinel handles automatically. After partition heals: old primary becomes replica. Verify data consistency for any transactions during the partition window. | Network fix: immediate. Data reconciliation: minutes | `min-replicas-to-write 1`: primary refuses writes if no replica is connected (prevents losing writes in partition). |
| **Corrupt write** | Redis does not validate data on write — corruption at the Redis level is rare; more common at the AOF/RDB file level after a crash. Corrupt AOF: Redis may fail to start. Corrupt RDB: startup fails with checksum error. | Startup failure (Redis won't load corrupt persistence file). All data in memory is lost if Redis cannot start. | Redis fails to start: `Short read or OOM loading DB`. `redis-check-rdb` or `redis-check-aof` reports errors. | On startup or during AOF rewrite | Run `redis-check-aof --fix <file>` to truncate corrupt tail. Run `redis-check-rdb` for RDB. Redis will lose data after the corruption point. | 15–30 min (check + fix) | Enable AOF with `appendfsync everysec`. Enable RDB snapshots as backup. Monitor startup logs after any Redis restart. |
| **Connection pool exhaustion** | Redis hits `maxclients` limit (default 10000). New connection attempts are rejected with `ERR max number of clients reached`. Existing connections are unaffected. | All new connection attempts. Dependent on whether application can retry with backoff or fails immediately. | `redis-cli INFO clients`: `connected_clients` approaching `maxclients`. Application: `ERR max number of clients reached`. `redis_connected_clients` metric at threshold. | Seconds (immediate) | Close idle connections: identify idle clients with `CLIENT LIST`. Tune application connection pool sizing. Increase `maxclients` in redis.conf (requires restart or `CONFIG SET maxclients N`). | 5–15 min | Monitor `connected_clients` at 70%/80% of `maxclients`. Use connection pooling in application (e.g., redis-py's ConnectionPool). Set `timeout` in redis.conf to auto-close idle connections. |
| **Long-running query / lock contention** | Redis is single-threaded for command execution. A slow command (KEYS *, SMEMBERS on a large set, SORT on a large list, LRANGE with a huge range) blocks all other commands for its entire duration. No lock contention in the traditional sense — everything is sequential. | All Redis commands are blocked for the duration of the slow command. Application experiences latency spike across all Redis operations. | `redis-cli SLOWLOG GET 10` — shows commands exceeding `slowlog-log-slower-than` threshold (default 10ms). Latency spike in `redis_commands_duration_seconds`. | Milliseconds to seconds (appears in SLOWLOG after execution) | Identify via `SLOWLOG`. Kill with `CLIENT KILL ID <id>` if still running. Replace slow commands: KEYS * → SCAN; large SMEMBERS → SSCAN; add key expiry to bound set size. | Seconds (immediate kill) | Never use KEYS * in production. Use SCAN for key iteration. Monitor SLOWLOG. Set appropriate data structure size limits. |
| **Schema migration failure** | Redis has no schema. Structural changes are handled by application-level key naming conventions and data structure migration scripts. A failed migration script may leave keys in mixed formats (old and new structure coexisting). | Keys touched by the migration script. Application code that assumes the new key structure will fail against unconverted keys. | Application errors on mixed key format. Migration script output/logs. | Seconds (application errors appear immediately after deployment) | Revert application to version compatible with old key format. Replay migration with idempotent logic. Verify with `redis-cli SCAN 0 MATCH <prefix>*` to check conversion rate. | 15–60 min (dependent on data volume) | Write migration scripts idempotently. Test against production key volume. Run migrations as background jobs with progress tracking, not as atomic deploys. |

## MySQL

| Failure Type | Impact | Blast Radius | Detection Signal | Detection Lag | Recovery Action | RTO Estimate | Prevention |
|---|---|---|---|---|---|---|---|
| **Single node crash** | Standalone: all reads and writes fail. Group Replication / InnoDB Cluster: automatic failover. ProxySQL routes away from failed node. | All application traffic in standalone. Dependent on cluster size and routing tier in HA setup. | MySQL process not running. `mysqladmin status` timeout. Application: `Can't connect to MySQL server`. ProxySQL: hostgroup unhealthy. | Seconds (connection failure) | Restart MySQL. If corrupt InnoDB tablespace: restore from backup or InnoDB recovery modes (`innodb_force_recovery`). | 2–10 min (clean restart) to hours (corruption recovery) | Group Replication or InnoDB Cluster. ProxySQL for connection routing. Regular backups (mysqldump + binlog). |
| **Primary/leader crash** | Group Replication: automatic re-election. Async replication: manual `CHANGE MASTER TO` on a replica. During failover: writes fail. After failover: writes succeed on new primary. Unacknowledged writes on old primary are lost. | All write traffic during failover. Any writes not yet replicated to the new primary. | Group Replication: `performance_schema.replication_group_members` shows member offline. MySQL Router or ProxySQL logs. `SHOW SLAVE STATUS` on replicas shows `Seconds_Behind_Master=NULL`. | Group Replication: 5–10s (paxos-based detection). Async: immediate (connection loss) but failover is manual. | Group Replication: automatic. Async: identify most up-to-date replica via `SHOW SLAVE STATUS` (`Exec_Master_Log_Pos`). Promote with `STOP SLAVE; RESET SLAVE ALL; SET GLOBAL read_only=OFF`. Point other replicas to new primary. | 30s (Group Replication) to 15–30 min (manual async promotion) | Group Replication or Orchestrator for auto-failover. Semi-sync replication (`rpl_semi_sync_master_enabled`) to prevent data loss. |
| **Replica lag / replica failure** | Reads from lagging replica return stale data. If read routing does not check lag, application silently serves stale results. Replica failure: reads fail for connections routed to that replica. Primary unaffected. | Read traffic on lagging/failed replica. Data staleness proportional to lag value. | `SHOW SLAVE STATUS`: `Seconds_Behind_Master > threshold`. ProxySQL read hostgroup health. `pt-heartbeat` for more accurate lag measurement. | Continuous (measurable). Failure detection: seconds | For lag: identify slow SQL on replica (`SHOW PROCESSLIST`), reduce replica query load, check `slave_parallel_workers`. For failure: remove from ProxySQL read hostgroup, restart, re-sync. | Lag: minutes. Resync from backup: 30 min to hours. | Monitor `Seconds_Behind_Master`. Use `pt-heartbeat` (more reliable than built-in metric). `slave_parallel_workers > 1` for parallel apply. |
| **Disk full** | InnoDB stops writing redo log when disk is full. All writes fail with `ERROR 28 (HY000): Out of disk space`. MySQL may crash. Binlog writes also fail — replication breaks. | All write operations. Replication breaks and must be manually repaired after recovery. | MySQL error log: `The table ... is full` or OS-level disk full. `df -h` on the MySQL data directory (`/var/lib/mysql`). | Seconds (writes fail immediately) | Free disk space: purge old binlogs (`PURGE BINARY LOGS BEFORE DATE(NOW())`), remove temporary tables, expand disk. After space freed: verify MySQL is running, check replication status. | 15–60 min | Monitor disk at 70%/80%/90%. Set `expire_logs_days` or `binlog_expire_logs_seconds`. Separate binlog, data, and temp directories across volumes. |
| **Network partition** | In async replication: replica falls behind silently. In Group Replication (Paxos): minority side becomes `read-only` and rejects writes. ProxySQL may split-brain if it cannot reach its DCS. | Async: replication lag; no immediate write failure. Group Replication: minority side goes read-only. | Group Replication: `performance_schema.replication_group_members` shows node status. Async: `SHOW SLAVE STATUS` shows `IO_Running: No`. | Group Replication: seconds (Paxos timeout). Async: seconds to minutes. | Restore network. Group Replication: minority side rejoins automatically. Async: `START SLAVE` on replica after network restoration. | Network fix: immediate. Replication catch-up: minutes. | Group Replication for automatic partition handling. ProxySQL with DCS (etcd/Consul) for consistent routing decisions. |
| **Corrupt write** | InnoDB checksum mismatch detected on page read. Query fails with `ERROR 1712: Index ... is corrupt`. Other tables unaffected. If system tablespace is corrupt: MySQL may not start. | Tables with corrupt pages. Queries touching those pages. | MySQL error log: `Database page corruption on disk or a failed file read`. `CHECK TABLE <tablename>` returns errors. | Seconds (on read) | `REPAIR TABLE <tablename>` (MyISAM only). InnoDB: restore from backup, or use `innodb_force_recovery` mode (1–6, increasing risk) to export data and rebuild. | 30 min to hours (restore from backup) | `innodb_checksums=ON` (default). `innodb_doublewrite=ON` (default) prevents partial page writes. Regular backups with `mysqlcheck`. |
| **Connection pool exhaustion** | MySQL hits `max_connections` (default 151). New connections fail with `ERROR 1040: Too many connections`. `max_connections` includes a 1-connection root superuser reserve. Existing connections continue. | All new connection attempts from application. | `SHOW GLOBAL STATUS LIKE 'Threads_connected'` approaching `max_connections`. MySQL error log: `Too many connections`. Application: `ERROR 1040`. | Seconds (immediate) | Kill idle connections: `SELECT concat('KILL ', id, ';') FROM information_schema.processlist WHERE command='Sleep' AND time>60`. Increase `max_connections` (dynamic: `SET GLOBAL max_connections=500`). Route through ProxySQL connection pooling. | 5–15 min | ProxySQL connection pooling in front of MySQL (multiplexes application connections). Monitor `Threads_connected` vs `max_connections`. Set `wait_timeout` and `interactive_timeout` to auto-close idle connections. |
| **Long-running query / lock contention** | InnoDB row-level locks. Long-running transaction holds locks, blocking all DML on those rows. DDL (ALTER TABLE) acquires MDL (metadata lock) — blocks all reads and writes on the table. Lock wait timeout (default 50s) causes blocked queries to fail with `ERROR 1205: Lock wait timeout exceeded`. | Rows locked by the long-running transaction. Table-level for DDL. Cascade: chain of waiting transactions. | `SHOW ENGINE INNODB STATUS` — TRANSACTION section shows lock waits. `SELECT * FROM information_schema.innodb_trx` shows active transactions. `SELECT * FROM performance_schema.events_statements_current` for slow queries. | Seconds (lock wait appears immediately in InnoDB status) | Identify blocker: `information_schema.innodb_lock_waits` JOIN `innodb_trx`. Kill: `KILL <thread_id>`. For DDL locks: `gh-ost` or `pt-online-schema-change` for large table alterations. | 1–5 min (identify + kill) | `innodb_lock_wait_timeout=30` (reduce from 50s default). `lock_wait_timeout` for MDL.
