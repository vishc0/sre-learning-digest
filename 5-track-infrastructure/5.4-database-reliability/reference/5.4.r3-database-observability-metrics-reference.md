# Database Observability Metrics Reference

> Quick-lookup for on-call and capacity review. Thresholds are baselines — calibrate to your workload. Splunk SPL patterns assume metrics forwarded via JMX exporter (Cassandra), postgres_exporter (PostgreSQL), or redis_exporter (Redis).

---

## Cassandra Metrics

JMX-based. Exported via `jmx_exporter` or `metrics-reporter-config` into your metrics pipeline. Metric paths shown are JMX MBean format.

| Metric Name | What It Measures | Normal Range / Baseline | Alert Threshold | Critical Threshold | Splunk SPL Pattern |
|---|---|---|---|---|---|
| `org.apache.cassandra.metrics:type=ClientRequest,scope=Write,name=Latency` (p99) | Coordinator write latency at 99th percentile. Time from client write request to coordinator acknowledgement, including all replicas at the configured consistency level. | < 10 ms | > 50 ms | > 200 ms | `index=cassandra_metrics metric_name="ClientRequest.Write.Latency.p99" \| timechart avg(value) by host span=1m` |
| `org.apache.cassandra.metrics:type=ClientRequest,scope=Read,name=Latency` (p99) | Coordinator read latency at 99th percentile. Time from client read request to response, including replica read and any read repair. | < 5 ms | > 25 ms | > 100 ms | `index=cassandra_metrics metric_name="ClientRequest.Read.Latency.p99" \| timechart avg(value) by host span=1m` |
| `org.apache.cassandra.metrics:type=Storage,name=Exceptions` | Count of uncaught exceptions in the Cassandra JVM since last restart. Monotonically increasing. | 0 (always) | > 0 (any exception warrants investigation) | N/A — treat any value as critical | `index=cassandra_metrics metric_name="Storage.Exceptions" \| timechart max(value) by host span=5m \| where max > 0` |
| `org.apache.cassandra.metrics:type=ThreadPools,path=request,scope=<PoolName>,name=PendingTasks` | Queued work in request-handling thread pools (ReadStage, MutationStage, CounterMutationStage, etc.). A positive pending count means Cassandra cannot drain the queue at the arrival rate. | 0 at steady state. Transient spikes < 10 acceptable. | Sustained > 10 for > 2 minutes | Sustained > 50 or growing monotonically | `index=cassandra_metrics metric_name="ThreadPools.*.PendingTasks" \| timechart avg(value) by scope span=1m` |
| `org.apache.cassandra.metrics:type=DroppedMessage,scope=<MessageType>,name=Dropped` | Dropped messages by type (READ, MUTATION, RANGE_SLICE, etc.). A dropped message means a client request timed out before Cassandra could process it. The client already received a timeout error. | 0 (always) | > 0 for READ or MUTATION | N/A — any dropped read or write is an immediate incident | `index=cassandra_metrics metric_name="DroppedMessage.*.Dropped" \| timechart sum(value) by scope span=1m \| where sum > 0` |
| `org.apache.cassandra.metrics:type=Compaction,name=PendingTasks` | Number of SSTables waiting to be compacted. High values correlate with increased read latency because reads must merge more SSTables. | < 10 | > 50 | > 200 | `index=cassandra_metrics metric_name="Compaction.PendingTasks" \| timechart max(value) by host span=5m` |
| `org.apache.cassandra.metrics:type=Storage,name=Load` | Total bytes of SSTable data on this node. Use for capacity trending and rebalance detection. | Varies by deployment. Track week-over-week delta. | 80% of disk capacity | 90% of disk capacity | `index=cassandra_metrics metric_name="Storage.Load" \| timechart max(value) by host span=1h \| eval GB=max/1024/1024/1024` |
| `org.apache.cassandra.metrics:type=ClientRequest,scope=Write,name=Timeouts` | Count of write operations that exceeded `write_request_timeout_in_ms`. Distinct from Dropped — this is the coordinator reporting a timeout to the client. | 0 at steady state | Any sustained value | Monotonically increasing | `index=cassandra_metrics metric_name="ClientRequest.Write.Timeouts" \| timechart sum(value) by host span=1m` |
| `org.apache.cassandra.metrics:type=ClientRequest,scope=Read,name=Timeouts` | Count of read operations that exceeded `read_request_timeout_in_ms`. Includes both coordinator timeouts and replica-level read timeouts (read repair excluded). | 0 at steady state | Any sustained value | Monotonically increasing | `index=cassandra_metrics metric_name="ClientRequest.Read.Timeouts" \| timechart sum(value) by host span=1m` |
| `org.apache.cassandra.metrics:type=Cache,scope=KeyCache,name=HitRate` | Key cache hit rate — fraction of reads that found the partition key offset in the key cache (avoiding one SSTable seek). | > 0.85 (85%) | < 0.60 | < 0.40 | `index=cassandra_metrics metric_name="Cache.KeyCache.HitRate" \| timechart avg(value) by host span=5m` |
| `org.apache.cassandra.metrics:type=ThreadPools,path=internal,scope=HintedHandoff,name=ActiveTasks` | Active hinted handoff tasks — hints being delivered to a recovering node. Non-zero means a node was recently down and hints are being replayed. | 0 normally. Expected non-zero during node recovery. | N/A as alert; use as context during node recovery | Sustained > 0 with no node recovery in progress | `index=cassandra_metrics metric_name="ThreadPools.HintedHandoff.ActiveTasks" \| timechart max(value) by host span=1m` |
| `org.apache.cassandra.metrics:type=Table,keyspace=<ks>,scope=<table>,name=LiveSSTableCount` | Number of SSTables per table per node. High SSTable count = compaction is falling behind. Reads must merge more SSTables = higher read latency. | < 10 per table per node | > 20 | > 50 | `index=cassandra_metrics metric_name="Table.*.LiveSSTableCount" \| timechart max(value) by scope span=5m` |

**Cassandra Splunk Correlation Patterns**:

```splunk
/* Write latency spike correlated with compaction backlog */
index=cassandra_metrics (metric_name="ClientRequest.Write.Latency.p99" OR metric_name="Compaction.PendingTasks")
| timechart avg(value) by metric_name span=1m

/* Dropped messages timeline across all nodes */
index=cassandra_metrics metric_name="DroppedMessage.*.Dropped"
| eval type=replace(metric_name,"DroppedMessage\.(.*)\.Dropped","\1")
| timechart sum(value) by type span=1m

/* Thread pool backpressure heatmap */
index=cassandra_metrics metric_name="ThreadPools.*.PendingTasks"
| eval pool=replace(metric_name,"ThreadPools\.(.*)\.PendingTasks","\1")
| stats max(value) by host pool
| where max > 0
```

## PostgreSQL Metrics

Sourced from `pg_stat_*` system views via `postgres_exporter` (Prometheus) or direct SQL polling. Metric names shown are prometheus-style from `postgres_exporter`.

| Metric Name | What It Measures | Normal Range / Baseline | Alert Threshold | Critical Threshold | Splunk SPL Pattern |
|---|---|---|---|---|---|
| `pg_replication_lag` (derived from `pg_stat_replication.replay_lag`) | Time delta between WAL generated on primary and applied on replica. Measures replication staleness. A lagging replica serves stale reads and represents data loss exposure if primary fails. | < 1 s | > 10 s | > 60 s | `index=postgres_metrics metric="pg_replication_lag_seconds" \| timechart max(value) by replica_host span=1m` |
| `pg_stat_activity_count{state="idle in transaction"}` | Count of connections in `idle in transaction` state — connected, in a transaction, but not executing a query. These hold row-level locks without doing work. Lock accumulation leads to blocking chains. | 0 | > 3 connections for > 30s | > 10 connections or any for > 5 min | `index=postgres_metrics metric="pg_stat_activity_count" state="idle in transaction" \| timechart max(value) by datname span=1m` |
| `pg_stat_bgwriter_checkpoints_timed_total` vs `pg_stat_bgwriter_checkpoints_req_total` | Checkpoint frequency. `timed` checkpoints occur on schedule (controlled by `checkpoint_completion_target`). `req` checkpoints are forced because WAL filled up before the scheduled checkpoint — indicates write pressure. | Mostly `timed`. `req / total < 10%`. | `req > 20%` of total checkpoints | `req > 50%` of total checkpoints | `index=postgres_metrics (metric="pg_stat_bgwriter_checkpoints_timed_total" OR metric="pg_stat_bgwriter_checkpoints_req_total") \| timechart sum(value) by metric span=5m` |
| `pg_database_size_bytes` | Total size of each database in bytes. Track weekly delta for capacity planning. | Varies. Track trend, not absolute value. | 80% of allocated storage | 90% of allocated storage | `index=postgres_metrics metric="pg_database_size_bytes" \| timechart max(value) by datname span=1h \| eval GB=max/1024/1024/1024` |
| `pg_stat_user_tables_n_dead_tup` | Dead tuple count per table. Dead tuples accumulate after DELETE and UPDATE operations. Autovacuum should reclaim them. High dead tuple ratio means autovacuum is not keeping up — leads to table bloat and index bloat, degrading read and write performance. | < 10% of `n_live_tup` for that table | > 10% of live tuples | > 25% of live tuples OR autovacuum has not run on this table in > 1 hour | `index=postgres_metrics metric="pg_stat_user_tables_n_dead_tup" \| timechart max(value) by relname span=5m \| sort -max` |
| `pg_locks_count{granted="false"}` | Count of lock requests that are waiting (not yet granted). Persistent waiting locks indicate contention — a query is blocked behind another. Each waiting lock has a blocker that can be identified via `pg_blocking_pids()`. | 0 | > 0 sustained for > 30s | > 5 concurrent waiting locks or any for > 2 min | `index=postgres_metrics metric="pg_locks_count" granted="false" \| timechart max(value) by datname span=1m \| where max > 0` |
| `pg_stat_user_tables_seq_scan` rate (delta over time) | Rate of sequential scans per table. High seq_scan rate on large tables = missing index or query using table scan. Each seq_scan on a large table reads every row — expensive for both I/O and CPU. | 0 for large tables (> 1M rows) | Any sustained seq_scan on tables > 100K rows without a corresponding index | N/A — context-dependent | `index=postgres_metrics metric="pg_stat_user_tables_seq_scan" \| timechart sum(value) by relname span=5m \| sort -sum` |
| `pg_stat_statements_mean_exec_time_seconds` | Mean execution time of queries tracked by `pg_stat_statements`. Identifies slow queries consuming the most cumulative time. | Varies by query. Track trend per `queryid`. | Query mean > 1s (for transactional queries) | Query mean > 5s or total_time growth > 2x baseline | `index=postgres_metrics metric="pg_stat_statements_mean_exec_time_seconds" \| timechart max(value) by query span=5m \| sort -max` |
| `pg_stat_bgwriter_buffers_clean_total` vs `pg_stat_bgwriter_buffers_backend_total` | Buffers written by bgwriter vs directly by backend. Backend writes are undesirable — they happen when bgwriter cannot keep up, blocking query execution while the buffer is flushed. | Backend writes < 10% of total | Backend writes > 20% | Backend writes > 40% | `index=postgres_metrics (metric="pg_stat_bgwriter_buffers_clean_total" OR metric="pg_stat_bgwriter_buffers_backend_total") \| timechart sum(value) by metric span=5m` |
| `pg_stat_replication_pg_current_wal_lsn_diff` (primary WAL position minus replica applied position, in bytes) | Replication lag expressed in bytes (WAL bytes not yet applied to replica). Complements time-based lag — useful when primary is idle (time-based lag inflates artificially during idle periods). | < 10 MB | > 100 MB | > 1 GB | `index=postgres_metrics metric="pg_replication_lag_bytes" \| timechart max(value) by replica_host span=1m` |

**PostgreSQL Splunk Correlation Patterns**:

```splunk
/* Idle-in-transaction sessions with lock accumulation */
index=postgres_metrics
  (metric="pg_stat_activity_count" state="idle in transaction")
  OR (metric="pg_locks_count" granted="false")
| timechart max(value) by metric span=1m

/* Checkpoint pressure: required vs timed ratio */
index=postgres_metrics
  metric IN ("pg_stat_bgwriter_checkpoints_timed_total","pg_stat_bgwriter_checkpoints_req_total")
| timechart sum(value) by metric span=5m
| eval req_pct=checkpoints_req_total/(checkpoints_req_total+checkpoints_timed_total)*100

/* Dead tuple accumulation — top 10 tables */
index=postgres_metrics metric="pg_stat_user_tables_n_dead_tup"
| stats max(value) as dead_tuples by relname
| sort -dead_tuples
| head 10
```

## Redis Metrics

Sourced from `redis_exporter` (Prometheus) which wraps `INFO all` and `LATENCY HISTORY` commands. `redis-cli INFO` section names are shown for direct CLI reference.

| Metric Name | What It Measures | Normal Range / Baseline | Alert Threshold | Critical Threshold | Splunk SPL Pattern |
|---|---|---|---|---|---|
| `redis_connected_clients` (`INFO clients: connected_clients`) | Current number of client connections. Each connection consumes memory and file descriptors. Approaching `maxclients` causes new connections to be rejected. | Varies. Baseline during normal load. Track daily pattern. | > 80% of `maxclients` (default `maxclients=10000`) | > 90% of `maxclients` | `index=redis_metrics metric="redis_connected_clients" \| timechart max(value) by host span=1m` |
| `redis_blocked_clients` (`INFO clients: blocked_clients`) | Clients waiting on blocking commands (BLPOP, BRPOP, BZPOPMIN, etc.). A persistent non-zero value means producers are not filling queues fast enough — queue consumers are starved. | 0 at steady state. Transient spikes during queue drain are normal. | > 0 sustained for > 5 min | > 10 blocked clients | `index=redis_metrics metric="redis_blocked_clients" \| timechart max(value) by host span=1m \| where max > 0` |
| `redis_evicted_keys_total` (`INFO stats: evicted_keys`) | Cumulative count of keys evicted due to `maxmemory` policy. Non-zero means Redis is under memory pressure and is removing keys to stay within bounds. For Redis used as a reliable store (not just cache): any eviction is a data loss event. For cache-only Redis: eviction is expected when working set > memory. | 0 (for durable stores). Variable (for cache). | Any non-zero rate for durable stores. For cache: rate growth > 2x baseline. | Eviction rate growing monotonically for > 10 min | `index=redis_metrics metric="redis_evicted_keys_total" \| timechart max(value) by host span=5m` |
| `redis_keyspace_hits_total` / (`redis_keyspace_hits_total` + `redis_keyspace_misses_total`) | Key lookup hit rate. For Redis used as a cache: hit rate below baseline means keys are missing (evicted, expired, or not populated). Low hit rate drives upstream load increases (more cache misses fall through to the database). | > 90% for a primary cache workload | < 75% | < 50% | `index=redis_metrics (metric="redis_keyspace_hits_total" OR metric="redis_keyspace_misses_total") \| timechart sum(value) by metric span=1m \| eval hit_rate=hits/(hits+misses)` |
| `redis_repl_backlog_size` vs active backlog consumed | Replication backlog buffer size and bytes consumed. When a replica disconnects briefly, Redis keeps recent WAL data in the backlog so the replica can catch up without a full resync. If the replica's lag exceeds the backlog, a full resync (PSYNC → FULLRESYNC) is triggered — expensive for both primary and replica. | Active backlog < 50% of `repl-backlog-size` | Active > 80% of backlog size | Full resync triggered (logged: `Full resync requested by slave`) | `index=redis_metrics metric="redis_repl_backlog_histlen" \| timechart max(value) by host span=1m` |
| `redis_memory_used_bytes` / `redis_maxmemory` | Memory utilization as a fraction of `maxmemory`. Hitting `maxmemory` triggers the eviction policy. With `maxmemory-policy noeviction` (safe for durable stores): writes fail when memory is full. | < 70% | > 75% | > 90% | `index=redis_metrics (metric="redis_memory_used_bytes" OR metric="redis_maxmemory") \| timechart max(value) by metric span=5m \| eval util_pct=used/maxmemory*100` |
| `redis_commands_duration_seconds_total` / `redis_commands_processed_total` | Average command latency. Useful for detecting slow command accumulation. Redis is single-threaded — slow commands block all subsequent commands. | < 1 ms average | > 10 ms average sustained | > 100 ms average | `index=redis_metrics (metric="redis_commands_duration_seconds_total" OR metric="redis_commands_processed_total") \| timechart sum(value) by metric span=1m \| eval avg_latency_ms=(duration/commands)*1000` |
| `redis_rdb_last_bgsave_status` | Status of the last RDB snapshot. `1` = success. `0` = failure. A failed BGSAVE means the persistence snapshot is stale — recovery from a crash will use an older snapshot. | 1 (success always) | 0 (any failure) | 0 persisting for > 1 backup interval | `index=redis_metrics metric="redis_rdb_last_bgsave_status" \| timechart min(value) by host span=5m \| where min < 1` |
| `redis_replication_offset` delta between primary and replica | Bytes of replication stream that have not yet been applied to the replica. Equivalent to PostgreSQL's WAL lag in bytes. Lower latency signal than time-based lag when primary is idle. | < 1 MB | > 50 MB | > 500 MB | `index=redis_metrics metric="redis_replica_repl_offset" \| timechart max(value) by host span=1m` |
| `redis_connected_slaves` | Number of replicas currently connected to this primary. Expected = configured replica count. Drop below expected = replica has disconnected. | Equal to configured replica count | < expected count | 0 replicas connected (standalone primary with no replication) | `index=redis_metrics metric="redis_connected_slaves" \| timechart min(value) by host span=1m` |
| `redis_aof_rewrite_in_progress` | Whether an AOF rewrite is currently running (`1` = yes). AOF rewrite forks the Redis process — during fork, memory usage doubles briefly (copy-on-write). Causes latency spikes. | 0 at steady state. Expect `1` during scheduled rewrite. | `1` for > 10 min (rewrite stuck) | N/A | `index=redis_metrics metric="redis_aof_rewrite_in_progress" \| timechart max(value) by host span=1m \| where max=1` |

**Redis Splunk Correlation Patterns**:

```splunk
/* Memory pressure: utilization + eviction rate together */
index=redis_metrics
  metric IN ("redis_memory_used_bytes","redis_maxmemory","redis_evicted_keys_total")
| timechart max(value) by metric span=5m

/* Cache effectiveness: hit rate trend */
index=redis_metrics
  (metric="redis_keyspace_hits_total" OR metric="redis_keyspace_misses_total")
| eval type=if(metric="redis_keyspace_hits_total","hits","misses")
| timechart sum(value) by type span=1m
| eval hit_rate=hits/(hits+misses)*100

/* Replication health: offset lag across replicas */
index=redis_metrics metric="redis_replica_repl_offset"
| timechart max(value) by host span=1m
| sort -max

/* Slow command detection via latency average */
index=redis_metrics
  (metric="redis_commands_duration_seconds_total" OR metric="redis_commands_processed_total")
| timechart sum(value) by metric span=1m
| eval avg_latency_ms=(redis_commands_duration_seconds_total/redis_commands_processed_total)*1000
| where avg_latency_ms > 10
```

## Cross-Database Alert Correlation Reference

Use this to correlate metrics across tiers during a multi-layer incident.

| Symptom Observed | First Database Metric to Check | Second Metric to Correlate | Likely Root Cause |
|---|---|---|---|
| Application write timeouts (Cassandra tier) | `ClientRequest.Write.Latency.p99` | `Compaction.PendingTasks`, `ThreadPools.MutationStage.PendingTasks` | Compaction lag or write thread saturation |
| Application read latency spike | `ClientRequest.Read.Latency.p99` | `Table.*.LiveSSTableCount`, `Compaction.PendingTasks` | Too many SSTables per read — compaction behind |
| Postgres replica read returning stale data | `pg_replication_lag_seconds` | `pg_stat_activity_count` (idle in transaction on replica) | Long-running transaction or replica under I/O load |
| Redis cache miss rate rising | `redis_keyspace_hits_total` ratio | `redis_evicted_keys_total`, `redis_memory_used_bytes` | Memory pressure forcing evictions; working set > maxmemory |
| Redis write latency spike | `redis_commands_duration_seconds_total` avg | `redis_connected_clients`, `redis_blocked_clients` | Slow command (KEYS *, large SMEMBERS) blocking pipeline |
| Database connection errors from application | `pg_stat_activity_count` (all states) vs `max_connections` | `redis_connected_clients` vs `maxclients` | Connection pool exhaustion at DB tier — check PgBouncer/ProxySQL |
