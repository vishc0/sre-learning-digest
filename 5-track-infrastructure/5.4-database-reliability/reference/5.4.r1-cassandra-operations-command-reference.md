# Cassandra nodetool Operations Command Reference

> Quick-lookup card. Run all `nodetool` commands as the cassandra OS user or with `-u/-pw` credentials if JMX auth is enabled. Commands target the local node unless `--host` is specified.

---

## Cluster Health

### `nodetool status`

```
nodetool status [<keyspace>]
```

**What it does**: Prints the ring view — one row per node showing state, load, token ownership, host ID, and rack.

**Expected output**:
```
Datacenter: dc1
===============
Status=Up/Down
|/ State=Normal/Leaving/Joining/Moving
--  Address        Load       Tokens  Owns    Host ID   Rack
UN  10.0.1.10     250.45 GiB  256    33.3%   <uuid>    rack1
UN  10.0.1.11     248.12 GiB  256    33.4%   <uuid>    rack1
DN  10.0.1.12     251.00 GiB  256    33.3%   <uuid>    rack1
```

**State codes**:

| Code | Meaning | Operational Implication |
|------|---------|------------------------|
| UN | Up / Normal | Node is healthy and serving traffic |
| DN | Down / Normal | Node is unreachable. At RF=3 + QUORUM, reads/writes still succeed via remaining 2 nodes. Third failure = outage. |
| UJ | Up / Joining | Bootstrap in progress. Token ranges not yet fully transferred. |
| UL | Up / Leaving | Decommission in progress. Do not kill the process. |
| DL | Down / Leaving | Decommission was interrupted. Needs manual intervention. |
| UM | Up / Moving | Token range move in progress. Rare. |

**When to use**: First command during any incident, cluster health check, capacity review.

**Caution**: DN does not tell you why the node is down. Follow with `nodetool info` on the down node (if accessible) or check system logs.

---

### `nodetool ring`

```
nodetool ring [<keyspace>]
```

**What it does**: Lists every token and its owning node. Much more verbose than `status`. Shows the actual token ring layout.

**Expected output**: Long table with columns — Address, Rack, Status, State, Load, Owns, Token. Sorted by token value.

**What to look for**: Even distribution of tokens across nodes. Severe skew (one node owns 50%+) causes hotspots. With vnodes (default 256 tokens/node), distribution should be roughly equal.

**When to use**: After a node add/remove to verify token rebalancing. When investigating read/write hotspots. Compare `Owns` column across nodes — delta > 5% warrants investigation.

**Caution**: Output can be thousands of lines. Pipe to `grep <IP>` to filter by node.

---

### `nodetool info`

```
nodetool info [--host <ip>]
```

**What it does**: Single-node stats. Prints uptime, load, heap usage, off-heap memory, data center, rack, token count, exception count.

**Expected output** (key fields):
```
ID                     : <uuid>
Gossip active          : true
Thrift active          : false
Native Transport active: true
Load                   : 250.45 GiB
Uptime (seconds)       : 864000
Heap Memory (MB)       : 12288 / 16384
Data Center            : dc1
Rack                   : rack1
Exceptions             : 0
Key Cache              : entries 123456, size 245.00 MiB, capacity 256 MiB, hit rate 0.98
Row Cache              : disabled
Counter Cache          : disabled
```

**What to look for**:
- `Exceptions > 0`: Something has gone wrong at the JVM level. Investigate logs immediately.
- `Heap Memory` utilization > 75%: GC pressure risk. Check GC logs.
- `Gossip active: false`: Node is not participating in cluster membership — effectively invisible to peers.

**When to use**: Node-level health check, after restart, when `nodetool status` shows a node as suspect.

---

### `nodetool tpstats`

```
nodetool tpstats
```

**What it does**: Thread pool statistics — pending (queued but not started), active (currently executing), completed, blocked, and dropped message counts per pool.

**Expected output** (key thread pools):
```
Pool Name                    Active   Pending   Completed   Blocked   All time blocked
ReadStage                         0         0      ...           0              0
MutationStage                     0         0      ...           0              0
CounterMutationStage              0         0      ...           0              0
RequestResponseStage              0         0      ...           0              0
ReadRepairStage                   0         0      ...           0              0
...

Message type           Dropped
READ                         0
MUTATION                     0
```

**What to look for**:

| Signal | Meaning |
|--------|---------|
| `Pending > 0` (sustained) | Backpressure — Cassandra cannot process requests as fast as they arrive. Scale up or reduce load. |
| `Pending > 0` (spike) | Transient burst. Monitor for trend. |
| `Dropped > 0` on READ or MUTATION | Client timeouts are expiring before Cassandra can respond. Immediate incident signal. |
| `Blocked > 0` | Thread pool queue is full. Requests being rejected. |

**When to use**: During latency incidents, when clients report timeouts, as part of regular health checks.

---

### `nodetool netstats`

```
nodetool netstats [--host <ip>]
```

**What it does**: Shows streaming status — bytes sent/received, active streams, pending streams. Also shows gossip message counts.

**Expected output** during a bootstrap:
```
Mode: NORMAL
Not sending any streams.
Receiving 3 streams using 3 connections
    cassandra-operations-command-reference /10.0.1.10 (completed: 45.23 GiB / 80.00 GiB)
    cassandra-operations-command-reference /10.0.1.11 (completed: 43.10 GiB / 80.00 GiB)
    cassandra-operations-command-reference /10.0.1.12 (completed: 40.00 GiB / 80.00 GiB)
```

**When to use**: During bootstrap, decommission, or repair to monitor data transfer progress. If streams stall (bytes not increasing over 5+ minutes), the streaming session may be stuck.

**Caution**: High streaming throughput competes with read/write I/O. Monitor latency during repair/bootstrap.

## Repair and Anti-Entropy

### `nodetool repair`

```
nodetool repair [-full] [<keyspace>] [<table> ...]
```

**What it does**: Runs Merkle tree comparison across replicas for this node's token ranges. Synchronizes divergent data. Ensures consistency after node outages, missed writes, or clock drift.

**Expected output**: Progress logs per keyspace/table. Completes with `Repair completed successfully` or `Repair completed with errors`.

**When to use**: After recovering a DN node. Regularly scheduled (at least once per `gc_grace_seconds` — default 10 days — to prevent tombstone accumulation and zombie data resurrection).

**Caution**: Full repair reads all data, generates significant I/O and network load. Run during low-traffic windows. Use `-pr` for production-safe scheduling.

---

### `nodetool repair -pr` (Primary Range Repair)

```
nodetool repair -pr [<keyspace>] [<table> ...]
```

**What it does**: Repairs only the primary token ranges owned by this node. When run on every node in the cluster (scheduled independently), achieves full cluster repair with lower per-node impact.

**Why use this over full repair**: Each token range is repaired exactly once across the cluster rather than RF times. Reduces total repair I/O by a factor of RF.

**When to use**: Standard scheduled repair strategy in production. Run on a rolling schedule — stagger start times by node so repairs do not overlap.

**Caution**: Coordination still required. If a node crashes during repair, that node's ranges may be partially repaired.

---

### `nodetool repair -st <start_token> -et <end_token>`

```
nodetool repair -st <start_token> -et <end_token> [<keyspace>]
```

**What it does**: Repairs only the specified token range. Used for targeted recovery of a specific data slice.

**When to use**: When you know data divergence is confined to a specific partition key range. Faster than full repair when the problem scope is known.

**How to get token values**: From `nodetool ring` output or calculated from partition key using the partitioner.

---

### `nodetool describecluster`

```
nodetool describecluster
```

**What it does**: Prints cluster name, partitioner, snitch, and — critically — schema version agreement across all nodes.

**Expected output**:
```
Name: MyCluster
Snitch: org.apache.cassandra.locator.GossipingPropertyFileSnitch
DynamicEndPointSnitch: enabled
Partitioner: org.apache.cassandra.dht.Murmur3Partitioner
Schema versions:
    <uuid1>: [10.0.1.10, 10.0.1.11, 10.0.1.12]
```

**What to look for**: All nodes should report the same schema version UUID in one group. Multiple UUIDs = schema disagreement.

| State | Meaning |
|-------|---------|
| All nodes in one UUID group | Schema is consistent. Normal state. |
| Two UUID groups | Schema migration in progress (nodes propagating DDL change via gossip). Transient — resolves in seconds to minutes. |
| Stuck with two groups > 5 min | Schema migration is stuck. Check gossip health; may require node restart. |

## Compaction

### `nodetool compact`

```
nodetool compact [<keyspace>] [<table> ...]
```

**What it does**: Triggers a major compaction — merges all SSTables for the target keyspace/table into one SSTable per node. Reclaims disk space consumed by tombstones and overwrites.

**When to use**: After bulk deletes, after a large batch load, or when compaction has fallen far behind (see `compactionstats`).

**Caution**: Extremely I/O-intensive. Will compete with active write throughput. During compaction, read latency can increase due to more SSTable files being merged. Never run on a node that is already under write pressure without throttling.

---

### `nodetool compactionstats`

```
nodetool compactionstats [-H]
```

**What it does**: Shows in-progress compaction tasks — table name, bytes compacted, bytes total, progress percentage. `-H` prints human-readable sizes.

**Expected output**:
```
pending tasks: 47
                        id  compaction type  keyspace      table  completed  total    unit  progress
<uuid>                  COMPACTION           my_ks    my_table  52428800   524288000 bytes    10%
Active compaction remaining time :   0h00m14s
```

**What to look for**:
- `pending tasks` > 50: Compaction is falling behind writes. Increase `compaction_throughput_mb_per_sec` or reduce write load.
- Compaction throughput consistently saturating disk I/O: Tune `setcompactionthroughput`.
- Write latency increasing in lockstep with pending task count: Classic compaction lag pattern.

---

### `nodetool setcompactionthroughput <MB/s>`

```
nodetool setcompactionthroughput <value>
```

**What it does**: Throttles compaction I/O to the specified MB/s. Setting to `0` removes the throttle (default Cassandra behavior is `16` MB/s).

**When to use**:
- Set lower (e.g., `8`) during peak traffic windows to protect read/write latency.
- Set higher or to `0` during maintenance windows to allow compaction to catch up.

**Caution**: Setting to `0` during high write workloads risks compaction falling behind permanently. This change is not persisted — it resets on node restart. To persist, change `compaction_throughput_mb_per_sec` in `cassandra.yaml`.

---

### `nodetool getcompactionthroughput`

```
nodetool getcompactionthroughput
```

**What it does**: Returns the current in-effect compaction throughput limit.

**Expected output**: `Current compaction throughput: 16 MB/s`

**When to use**: Verify before/after a `setcompactionthroughput` change. Baseline check during incident investigation.

## Node Lifecycle

### `nodetool decommission`

```
nodetool decommission
```

**What it does**: Gracefully removes this node from the ring. Streams all data owned by this node to remaining nodes before leaving. Node transitions: UN → UL → removed.

**Expected output**: Streaming progress visible via `nodetool netstats` on the node being decommissioned. Final state: node disappears from `nodetool status`.

**When to use**: Planned scale-in — removing a healthy node from a running cluster.

**Caution**: Do NOT kill the Cassandra process during decommission. Doing so leaves the node in DL (Down/Leaving) state, requiring `nodetool removenode` to clean up. Decommission duration depends on data volume and available streaming bandwidth.

---

### `nodetool removenode <host_id>`

```
nodetool removenode <host_id>
```

**What it does**: Removes a dead node that cannot be restarted. Triggers streaming of the dead node's token ranges to other live nodes. Run from a live node, not the dead one.

**How to get `<host_id>`**: From `nodetool status` output (the UUID column). If the node no longer appears, use `nodetool status` output captured before failure or check system tables.

**When to use**: Node has crashed and cannot be recovered. Hardware failure, terminated VM, irrecoverable corruption.

**Caution**: Ensure the node is truly dead (not network-partitioned) before running. If the node is partitioned and you removenode it, you may have a split-brain scenario where the node believes it is still active.

---

### `nodetool assassinate <endpoint>`

```
nodetool assassinate <ip_address>
```

**What it does**: Force-marks a node as dead in gossip. Does NOT stream data. Does NOT rebalance. Removes the node from the ring immediately regardless of its actual state.

**When to use**: Node is stuck in a transitional gossip state (e.g., UJ, UL, UM) and cannot progress. Last resort only.

**Caution**: High risk. If the assassinated node later comes back online, it may have data that other nodes do not, causing inconsistency. Always follow with a full repair. Never use as a substitute for `decommission` or `removenode`.

## Diagnostics

### `nodetool getlogginglevels`

```
nodetool getlogginglevels
```

**What it does**: Lists current log levels per logger class. Default is INFO for most classes.

**When to use**: Baseline check before increasing verbosity for debugging.

---

### `nodetool setlogginglevel <class> <level>`

```
nodetool setlogginglevel org.apache.cassandra.db.Memtable DEBUG
```

**What it does**: Temporarily raises log verbosity for a specific class without restarting Cassandra. Change takes effect immediately.

**Levels**: TRACE, DEBUG, INFO, WARN, ERROR

**When to use**: During live debugging of a specific subsystem. Always reset to INFO after investigation — DEBUG/TRACE log volume can saturate disk quickly.

**Caution**: Not persisted across restarts. TRACE logging on a busy node can cause log I/O to become a bottleneck.

---

### `nodetool proxyhistograms`

```
nodetool proxyhistograms
```

**What it does**: Latency histograms for coordinator-level operations — reads, writes, range queries, CAS operations. Shows p50, p75, p95, p98, p99, p999, max in microseconds.

**Expected output**:
```
Percentile  Read Latency  Write Latency  Range Latency  CAS Read Latency  CAS Write Latency  View Write Latency
50%              352.00         198.00         412.00             0.00               0.00                0.00
75%              512.00         256.00         613.00             0.00               0.00                0.00
95%             1024.00         512.00        1435.00             0.00               0.00                0.00
99%             5000.00        2048.00        8192.00             0.00               0.00                0.00
```

**What to look for**: p99 read latency > 5ms or write > 10ms warrants investigation. Max latency orders of magnitude above p99 = GC pauses or I/O spikes. Compare read vs. write latency — if read is much higher, check compaction backlog and SSTable count.

**When to use**: First diagnostic command during a latency incident. Most useful single command for coordinator-level performance.

---

### `nodetool tablehistograms <keyspace> <table>`

```
nodetool tablehistograms <keyspace> <table>
```

**What it does**: Per-table latency and size histograms — local read/write latency, SSTable count per read, partition size, cell count. More granular than `proxyhistograms`.

**What to look for**:
- `SSTables per Read` > 5: Too many SSTables being merged per read. Compaction is lagging or data model is poorly designed (too many partitions per token range).
- Local read latency much higher than coordinator read latency: Replica is slower than coordinator average — node-level problem.
- Partition size in GiB range: Large partitions cause GC pressure and read timeouts.

**When to use**: After `proxyhistograms` identifies a latency problem — drill into which table is the source.
