# Chapter 2 — Server Hardware Anatomy

## Why Directors Need to Know Hardware Internals

You will not configure servers. But you will sit across from a vendor account executive with a proposed bill of materials, and you need to know if it matches your workload's requirements. Hardware misspecification at procurement is expensive — you either overpay for capability you don't use, or you under-spec and discover it 18 months into a 36-month lifecycle. This chapter gives you the interrogation vocabulary.

---

## CPU: NUMA Topology and Architecture Choice

Modern server CPUs are not uniform memory access systems. They are **NUMA (Non-Uniform Memory Access)** systems. A 2-socket server has two physical CPU dies. Each socket has its own local memory. When Socket 0's CPU accesses memory attached to Socket 1, it crosses the NUMA boundary — typically via Intel UPI (Ultra Path Interconnect) or AMD Infinity Fabric — adding 30–40% memory access latency compared to local access.

This matters operationally because:
- A Java application scheduled on Socket 0 that allocates memory from Socket 1's pool pays that penalty on every heap access
- A database with NUMA-unaware memory allocation will thrash across sockets unpredictably, producing latency spikes that look like application bugs
- Docker and Kubernetes workloads, by default, are NOT NUMA-aware — CPU pinning and memory binding require explicit configuration

To inspect NUMA topology:
```bash
numactl --hardware          # shows NUMA nodes, CPUs per node, memory per node
numastat                    # shows NUMA hit/miss rates — high "other" = NUMA thrashing
lscpu | grep NUMA           # quick node count
```

**Intel Xeon vs AMD EPYC — what to interrogate:**

| Factor | Intel Xeon (Sapphire Rapids) | AMD EPYC (Genoa/Turin) |
|--------|------------------------------|------------------------|
| Core count (per socket) | Up to 60 cores | Up to 128 cores (Turin) |
| PCIe lanes per socket | 80 (PCIe 5.0) | 128 (PCIe 5.0) — meaningful for GPU-dense configs |
| Memory channels | 8 channels (DDR5) | 12 channels (DDR5) — higher memory bandwidth |
| Memory bandwidth | ~307 GB/s | ~460 GB/s — EPYC wins for memory-intensive workloads |
| NUMA domains | 2 per socket (SNC mode) | 4 per socket (NPS mode) — more complex NUMA awareness needed |
| Ecosystem | Dominant market share, broader ISV support | Strong in HPC, growing enterprise presence |
| T-Mobile relevance | Safe default for application servers | Preferred for memory-intensive DB, HPC |

When a vendor proposes Intel where you have a memory-intensive workload, ask about EPYC. The memory bandwidth difference alone can justify the switch.

## Memory: ECC, Channels, and Interleaving

### ECC (Error-Correcting Code) Memory

All server-grade memory is ECC. The question is what ECC actually does:

- **Single-bit errors (SEBE — Single Event Bit Error)**: ECC detects and corrects transparently. The OS and application never see it. The hardware logs a corrected error in IPMI SEL. If you see repeated DIMM corrections on the same module, that DIMM is failing — replace it before it produces an uncorrectable error.
- **Multi-bit errors (MBE)**: ECC cannot correct. The hardware asserts a machine check exception (MCE). The kernel panics. The server crashes. This is why trending corrected errors matters — they predict MCEs.

RDIMM (Registered DIMM) vs LRDIMM (Load-Reduced DIMM):
- RDIMM: standard registered memory, up to 3 DIMMs per channel on most platforms
- LRDIMM: load-reduced, allows higher capacity per channel, slight latency penalty — used for maximum memory configurations (SAP HANA, in-memory databases)

### DIMM Channels and Population

A server with 8 memory channels should have DIMMs in all 8 channels. Single-channel or 4-channel population on an 8-channel platform cuts memory bandwidth by up to 50%. This is a procurement error vendors sometimes make to hit a lower price point.

Ask vendors: "Is this configuration fully populating all memory channels?" If not, you are paying for a CPU with premium memory bandwidth and running it at half bandwidth.

### Memory Interleaving

Memory interleaving distributes sequential memory accesses across multiple channels and DIMMs in round-robin fashion, improving effective bandwidth. It requires symmetric DIMM population (same capacity/speed DIMMs in all channels). Mixed DIMM sizes or partially populated channels disable interleaving.

## PCIe: Lanes, Slots, and Bifurcation

PCIe lanes are the bandwidth connection between the CPU and peripherals (GPUs, NVMe SSDs, NICs). Lanes are finite.

- A PCIe 5.0 x16 slot delivers ~128 GB/s bidirectional bandwidth
- An A100 GPU uses x16 lanes minimum; an H100 NVLink configuration uses multiple x16 slots
- A 100GbE NIC uses x16 lanes at PCIe 4.0 (~32 GB/s — adequate)
- An NVMe SSD uses x4 lanes (~7 GB/s at PCIe 4.0)

For a GPU-dense server, count your lanes:
- 4x H100 GPUs = 4x PCIe x16 = 256 lanes (AMD EPYC with 128 lanes per socket, 2 sockets = 256 lanes — tight fit)
- Add 2x NVMe boot drives (x4 each = 8 lanes) + 2x 100GbE NICs (x16 each = 32 lanes) = 296 lanes needed
- Intel Xeon at 80 lanes per socket = 160 lanes — insufficient for 4-GPU without bifurcation compromises

**Bifurcation**: splitting a physical x16 slot into multiple logical connections (e.g., x16 → 2x x8, or x16 → 4x x4). Used for NVMe U.2 backplanes. Bifurcation capability is a BIOS setting and platform-dependent — not all CPUs support all bifurcation modes.

## Storage Controllers: RAID vs HBA Pass-Through

Server storage controllers come in two modes:

**Hardware RAID controller (e.g., Dell PERC, HPE Smart Array)**:
- Controller has dedicated cache (DRAM, sometimes NVRAM-backed) and RAID processor
- OS sees a virtual disk — individual drives are invisible
- Advantages: CPU-free RAID processing, write cache acceleration, consistent performance
- Disadvantages: if the controller fails, the RAID is tied to that specific controller model; migrating to a new controller requires rebuilding

**HBA pass-through (Host Bus Adapter)**:
- Controller passes raw drives directly to the OS
- OS sees individual drives; software manages RAID (mdadm, ZFS, storage platform like Ceph)
- Advantages: drives are portable — move drives to a new server and your software RAID reconstructs; required for Ceph, ZFS, and most software-defined storage
- Disadvantages: CPU overhead for software RAID, no hardware write cache (though NVMe drives have their own cache)

**Director rule**: if you are running software-defined storage (Ceph, Rook, Longhorn), mandate HBA pass-through in your procurement specification. Hardware RAID controllers and Ceph are incompatible — Ceph requires direct disk access.

## NIC Bonding Modes

NICs are almost always configured in bonded pairs for redundancy. The bonding mode determines how traffic distributes and how failover behaves:

| Mode | Description | Load Balancing | Failover Speed | Requires Switch Config |
|------|-------------|----------------|----------------|----------------------|
| **LACP (802.3ad)** | Dynamic link aggregation, switch-side coordination | Yes (flow-based) | ~1s (LACP negotiation) | Yes (LACP on switch port) |
| **Active-Backup** | One link active, one standby | No | ~200ms | No — switch-transparent |
| **Balance-XOR** | Static hash-based distribution | Yes (hash-based) | ~200ms | No |

**For server uplinks**: LACP 802.3ad is standard when switch infrastructure supports it. Provides both redundancy and bandwidth aggregation. Two 25GbE ports bonded = 50GbE effective for traffic that distributes across flows.

**For environments where switch config is not controlled** (colo, some cloud networking): Active-Backup eliminates dependencies on switch LACP configuration, at the cost of not aggregating bandwidth.

**Failure mode awareness**: LACP failover requires LACP PDU timeout detection — typically 1 second in fast mode, 30 seconds in slow mode. If your NIC fails silently (link stays up but no traffic flows — a known phenomenon with some NICs), bonding may not detect the failure. BFD (Bidirectional Forwarding Detection) at the routing layer provides the backup detection mechanism.

## Director Frame: Interrogating a Vendor Proposal

When a vendor presents a server configuration, your five interrogation questions:

1. **NUMA topology for this workload**: Are applications pinned to NUMA nodes, or are we accepting NUMA-blind scheduling?
2. **Memory channel population**: Is every channel populated? Is this interleaved or mixed-capacity?
3. **PCIe lane count vs device requirements**: Have we enumerated lane consumption for every card in this chassis?
4. **Storage controller mode**: Is this HBA pass-through for software-defined storage, or hardware RAID — and does our storage architecture require one vs. the other?
5. **NIC bonding alignment with switch config**: Has our network team confirmed LACP is configured on the switch ports before we standardize on 802.3ad?

These questions signal hardware fluency. They also catch the spec errors that become six-figure performance problems after the hardware is racked.
