# Chapter 1 — When Bare Metal Makes Sense

## The Core Question

Bare metal is not a legacy holdover. It is a workload routing decision. The question every Director of Platform Engineering must be able to answer is: given this specific workload's resource profile, compliance constraints, and economic horizon, where does it belong — and why?

Cloud-native defaults are correct for most workloads. But "most" is not "all." The workloads that belong on bare metal have distinct characteristics, and misrouting them costs real money and performance.

---

## Workload Classes That Belong on Bare Metal

### High-Performance Computing (HPC)

HPC workloads — weather modeling, financial risk simulation, seismic analysis, telco network simulation — share a defining characteristic: tight MPI (Message Passing Interface) coupling between compute nodes. MPI jobs pass millions of small messages between nodes per second. The latency of those messages directly determines whether a 1000-node job finishes in 4 hours or 7 hours.

On a hypervisor, you add a virtualization layer between the application and the physical NIC. That layer adds latency variability (jitter), not just raw latency. MPI jobs degrade non-linearly with jitter — a single slow node in an MPI job can stall the entire job. Bare metal eliminates that variability source.

For T-Mobile, this surfaces in network simulation and capacity planning — 5G capacity modeling at scale requires tight-coupled compute that bare metal serves better than any VM tier.

### GPU Workloads (AI/ML Training and Inference)

GPU passthrough from a hypervisor to a VM is possible (SR-IOV, vGPU) but introduces overhead and management complexity. For AI/ML training — where a job runs for hours or days and saturates GPU compute continuously — the virtualization overhead compounds across the job duration. More critically, multi-GPU jobs using NVLink or NVSwitch require physical proximity and direct hardware access that virtualization cannot fully replicate.

For inference at scale (serving models in production), latency SLOs drive the same conclusion: bare metal gives you direct CUDA access, predictable PCIe bandwidth, and no co-tenant GPU contention.

At T-Mobile, AI/ML inference for network anomaly detection, fraud scoring, and customer experience prediction are workloads where bare metal GPU nodes deliver measurable SLO advantage.

### High-Throughput Databases

Certain database workloads are pathological on shared infrastructure:

- **Oracle RAC**: Real Application Clusters requires sub-millisecond inter-node communication via the Cache Fusion interconnect. Any hypervisor-introduced latency degrades cache coherency performance measurably.
- **SAP HANA**: In-memory database that can consume 6–24 TB of RAM in a single node. The memory bandwidth requirements (requiring all DIMM channels to be populated and properly interleaved) are not achievable in a VM without NUMA complications.
- **Redis/Memcached at extreme throughput**: >5M ops/sec workloads see hypervisor scheduling jitter translate directly into tail latency spikes.

These are not "we prefer bare metal" situations. They are "the vendor's support contract requires bare metal" or "the SLA cannot be met on a VM" situations.

### Network Functions (Telco NFV and Packet Processing)

This is directly in T-Mobile's domain. Virtualized Network Functions (VNFs) — virtual routers, firewalls, session border controllers, deep packet inspection — require kernel bypass networking to achieve line-rate packet processing.

Technologies like DPDK (Data Plane Development Kit) and RDMA (Remote Direct Memory Access) bypass the OS kernel entirely to get packets from NIC to application with microsecond latency. These require direct hardware access. On a hypervisor, you can use SR-IOV to present a virtual function, but you lose isolation guarantees and add management complexity.

5G core functions (UPF — User Plane Function, in particular) are workloads where T-Mobile's physical infrastructure provides genuine competitive advantage over cloud-native equivalents, specifically because of controlled latency and throughput predictability.

## TCO Crossover: When Ownership Wins

The math is straightforward. Cloud instance cost is dominated by OpEx (monthly billing). Bare metal ownership is dominated by CapEx (purchase) with lower ongoing OpEx (power, rack space, support contracts).

For a representative compute-intensive workload:

| Factor | Cloud (c5.4xlarge) | Bare Metal (2U 2-socket) |
|--------|-------------------|------------------------|
| Unit cost (monthly) | ~$600–$800 | ~$200–$300 (amortized over 36 months) |
| Setup time | Minutes | Days–weeks (procurement, racking) |
| Break-even horizon | — | ~18–24 months for steady-state workloads |
| Elasticity | Instant scale up/down | Lead time = procurement cycle (weeks–months) |
| Maintenance burden | Cloud provider's problem | Your team's problem |

**The crossover rule**: if a workload runs at >60% utilization continuously for 18+ months, bare metal TCO beats cloud. Below that utilization or below that time horizon, cloud wins on economics alone.

This is why bare metal makes sense for T-Mobile's core network functions (always-on, predictable load) but not for burst capacity (scale for Super Bowl traffic, scale down after).

## Bare Metal Cloud: The Hybrid Play

Between "own it" and "rent a VM," there is a third option: bare metal cloud. Providers offer dedicated physical servers on-demand:

- **AWS c5.metal, c6i.metal, hpc6a.48xlarge**: Physical AWS instances — no hypervisor, dedicated hardware, still in AWS ecosystem (VPC, IAM, Direct Connect)
- **Equinix Metal** (formerly Packet): Global bare metal provider, direct interconnect at Equinix colocation facilities — ideal for T-Mobile given existing Equinix presence
- **OVHcloud dedicated servers**: European provider, cost-competitive, good for regulatory workloads with EU data residency requirements

The bare metal cloud play matters for T-Mobile specifically because:
1. **Burst capacity for bare-metal-class workloads**: When a GPU training job needs 200 nodes for 72 hours, Equinix Metal provisions that without a CapEx commitment.
2. **New geography expansion**: Before building out a physical presence in a new region, validate demand with bare metal cloud first.
3. **Disaster recovery**: Bare metal cloud as DR target for workloads that require bare metal — avoids maintaining idle physical hardware in a DR site.

## Director Frame: How to Argue This in an Interview

The trap in bare metal conversations is framing it as infrastructure dogma — "we believe in bare metal" or "we believe in cloud-native." Neither is useful.

The correct frame is workload routing with economic discipline:

> "Bare metal is the right answer for three workload classes in our environment: network functions requiring kernel bypass, GPU inference workloads with tight latency SLOs, and high-throughput databases where the vendor's support model requires it. For everything else, we default to cloud-native. The decision framework is: latency sensitivity, utilization predictability, TCO horizon, and compliance constraints. We review workload placement quarterly and move workloads as those factors change."

This answer demonstrates: economic rigor, workload-specific thinking, avoidance of both cloud-native dogma and bare-metal nostalgia, and a review process that prevents drift. That is a Director-level answer.
