# Chapter 5 — Bare Metal Monitoring

## The Hardware Monitoring Gap

Most SRE organizations have robust application and OS monitoring: CPU utilization, memory usage, disk I/O, network throughput, process health. What most SRE organizations do not have is hardware monitoring — telemetry from the hardware layer beneath the OS.

This gap is consequential. A server can be perfectly healthy at the OS level (Kubernetes node shows Ready, application response times are normal) while simultaneously:
- Running at 85°C CPU temperature (normal threshold is 80°C, critical at 90°C)
- Operating with a degraded RAID array (one disk failed, parity rebuild pending, second failure = data loss)
- Accumulating DIMM corrected ECC errors (predictor of imminent multi-bit failure = kernel panic)
- Running fans at maximum RPM (sign of cooling system problem)

You will not detect any of these failures through OS or application monitoring. They require hardware-layer telemetry — IPMI sensor data, RAID controller status, and SMART disk health data.

## IPMI Sensor Data: What to Monitor

IPMI exposes sensor readings via the Sensor Data Repository (SDR). The monitoring targets that matter operationally:

### Temperature Sensors

| Sensor | Typical Threshold | Why It Matters |
|--------|------------------|----------------|
| Inlet (ambient) temp | Warning: >27°C, Critical: >35°C | High inlet = datacenter cooling problem, affects all servers in that zone |
| CPU 1/2 temp | Warning: >80°C, Critical: >90°C | CPU thermal throttling starts at ~80°C; at 90°C server may emergency shutdown |
| DIMM temps | Warning: >85°C | Memory errors increase with temperature |
| Drive temps | Warning: >50°C, Critical: >60°C | Disk failure rates increase significantly above 45°C |

**Operationally**: a cluster of servers in the same rack zone all showing elevated inlet temperatures is a datacenter cooling event, not server events. Correlation across servers is what turns individual alerts into root cause identification.

### Fan Speed Sensors

Fan sensors report in RPM. Alert on:
- Fan speed > 90% of maximum (emergency cooling mode — something is hot)
- Fan speed = 0 RPM (fan failure — single-fan failure in a redundant fan array is non-critical, but must be replaced before a second fails)
- Fan speed drop from established baseline (fan bearing degradation before full failure)

### Voltage Rail Sensors

Servers have multiple power rails (3.3V, 5V, 12V, VCORE for CPU). Voltage outside tolerance range indicates PSU degradation or power delivery component failure. Alert on any voltage sensor in non-critical or critical state.

### Power Consumption

Total server wattage (reported via IPMI) is a useful operational metric:
- Baseline wattage per server class lets you detect anomalies (a server drawing 200W more than baseline may have a stuck fan, thermal event, or hardware fault)
- Fleet-level power trending feeds datacenter capacity planning
- Power capping via IPMI is possible for emergency load management

```bash
# Collect power consumption reading
ipmitool dcmi power reading
# Output: Instantaneous power reading: 245 Watts
```

### Polling Strategy

IPMI polling at 1-minute intervals for sensor data is appropriate for most hardware. For critical systems, 30-second intervals. Higher polling frequencies risk BMC CPU saturation — BMCs are embedded processors with limited resources.

For fleet-scale polling, use **OpenIPMI** (Linux kernel IPMI driver) for local servers, or **ipmitool** via management network for remote polling. Tools like **FreeIPMI** include `ipmi-sensors` with output formats designed for ingestion into monitoring systems.

## Hardware RAID Status Monitoring

### megacli and storcli

For servers with LSI/Broadcom RAID controllers (common in Dell PERC, some HPE configurations):
```bash
# storcli (modern, replaces megacli)
storcli64 /c0 show all              # controller 0 status and all drives
storcli64 /c0/v0 show              # virtual disk 0 status
storcli64 /c0/eall/sall show       # all enclosures and drives

# Key outputs to monitor:
# Virtual Drive state: Optl (Optimal) | Dgrd (Degraded) | Pdbd (Partially Degraded) | Offln (Offline)
# Physical Drive state: Onln (Online) | Rbld (Rebuilding) | UBad (Unconfigured Bad)
# Media Error Count > 0: disk has errors, trending toward failure
# Predictive Failure flag: SMART threshold crossed, replace soon
```

### RAID State Transitions — What They Mean Operationally

```
Optimal → Degraded:   One disk failed. Array is still serving I/O from parity.
                      CRITICAL: a second disk failure means data loss.
                      Action: replace failed disk immediately. Activate hot spare.

Degraded → Rebuilding: Replacement disk is being rebuilt from parity.
                       Performance impact: rebuild I/O competes with application I/O.
                       Duration: 4–24 hours depending on disk size and I/O load.
                       Risk: during rebuild, second disk failure = data loss. Monitor closely.

Rebuilding → Optimal:  Rebuild complete. Array is fully redundant again.
                       Action: confirm rebuild completed without errors.

Optimal → Partially Degraded: In RAID 6 — one disk failed but still protected.
                               Less urgent than RAID 5 Degraded, but still requires replacement.
```

### What to Alert On

- Virtual Drive state != Optimal → **P1 alert**
- Physical Drive Predictive Failure flag set → **P2 alert** (replace within SLA before it becomes P1)
- Media Error Count increasing → **P3 alert** (trend monitoring, schedule replacement)
- Rebuild in progress → **informational** (duration tracking, second-disk-failure alert during rebuild)

## Disk Health: SMART Data

SMART (Self-Monitoring, Analysis and Reporting Technology) is embedded in every modern disk drive and SSD. The disk firmware continuously monitors its own health and exposes the data to the host.

```bash
smartctl -a /dev/sda              # full SMART data for sda
smartctl -H /dev/sda              # health summary only: PASSED or FAILED
smartctl -t short /dev/sda        # trigger 2-minute self-test
smartctl -t long /dev/sda         # trigger full surface scan self-test (hours)
```

**Key SMART attributes — what each predicts:**

| Attribute | ID | What It Means | Action Threshold |
|-----------|----|---------------|-----------------|
| Reallocated Sectors Count | 5 | Disk has remapped bad sectors to spare area. Any non-zero value is concerning. | >0: watch. >10: replace soon |
| Reported Uncorrectable Errors | 187 | Errors that could not be corrected even with ECC. Direct predictor of impending failure. | Any non-zero = replace now |
| Pending Sector Count | 197 | Sectors flagged for reallocation but not yet remapped (waiting for a successful write). I/O to these sectors may fail. | Any non-zero = urgent |
| Uncorrectable Sector Count | 198 | Sectors that failed reallocation — unreadable. | Any non-zero = immediate data protection action |
| Power-On Hours | 9 | Total hours of operation. Correlate with MTTF spec for failure probability. | Use for age trending |
| Reallocate Event Count | 196 | Number of times reallocation occurred. | >0: watch |

**For SSDs, additional attributes:**
| Attribute | What It Means |
|-----------|--------------|
| Wear Leveling Count | Remaining SSD life — starts at 100, counts down |
| Media Wearout Indicator | Similar to above, vendor-specific |
| NAND Endurance | Some SSDs report TBW (Terabytes Written) remaining |

**Automated SMART monitoring**: `smartd` daemon can run background monitoring and email/log on threshold crossing. For fleet monitoring, use a monitoring exporter (Prometheus `smartctl_exporter`) or parse `smartctl` output in your monitoring pipeline.

## Splunk for Hardware Events

T-Mobile runs Splunk. Hardware telemetry should flow into Splunk alongside application telemetry. This is the gap most SRE organizations have: hardware events in isolated tools (IPMI console, iDRAC GUI, RAID management UI) rather than correlated with application events in the same observability platform.

### Ingestion Approach

| Data Source | Ingestion Method |
|-------------|-----------------|
| IPMI SEL logs | `ipmitool sel list` → scripted to Splunk HEC or forwarded via syslog |
| SMART data | `smartctl --json` → Splunk HEC or scripted forwarding |
| RAID status | `storcli` JSON output → scripted to Splunk HEC |
| BMC syslog | Configure iDRAC/iLO to forward syslog to Splunk HF (Heavy Forwarder) |
| OS kernel hardware messages | `/var/log/messages` or systemd journal → Splunk UF (Universal Forwarder) |

### Key SPL Queries for Hardware Anomaly Detection

```splunk
-- Servers with corrected ECC errors in last 24 hours
index=hardware_events source=ipmi_sel "Correctable ECC" 
| stats count by host, dimm_slot 
| where count > 5
| sort -count

-- RAID degraded or rebuilding status across fleet
index=hardware_events source=raid_status VirtualDrive_State IN ("Dgrd", "Rbld", "Pdbd")
| table _time, host, controller, vdrive, VirtualDrive_State, FailedDrives

-- Disk SMART predictive failures
index=hardware_events source=smart_data Reallocated_Sectors_Count > 0 OR Pending_Sector_Count > 0
| eval urgency=if(Pending_Sector_Count > 0, "CRITICAL", "WARNING")
| table host, disk, Reallocated_Sectors_Count, Pending_Sector_Count, urgency

-- Temperature anomalies correlated with performance events
index=hardware_events source=ipmi_sensors sensor_type=Temperature value > 80
| join host [search index=apm_metrics response_time_p99 > 500]
| table _time, host, sensor, temp_value, response_time_p99
```

## Director Frame: Closing the Hardware Monitoring Gap

The business case for hardware monitoring is asymmetric risk mitigation. The cost of a DIMM ECC monitoring system is low — a script, a dashboard, an alert rule. The cost of a server crashing mid-shift on an unmonitored multi-bit memory error — especially if it is a database server — is an incident, potential data integrity concerns, and an RCA that surfaces the monitoring gap.

In an interview, if asked about your observability philosophy, include hardware monitoring explicitly:

> "We instrument three layers: application/service, OS/system, and hardware. Most teams stop at the OS layer. Hardware telemetry — IPMI sensors, RAID status, SMART disk health — feeds into the same Splunk platform as our application metrics. We alert on hardware anomalies the same way we alert on application anomalies. The difference is that hardware failures often give you predictive signals days or weeks before the failure — corrected ECC errors, SMART pending sectors, degraded RAID — and acting on those signals prevents incidents rather than responding to them."
