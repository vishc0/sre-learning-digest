# Chapter 3 — IPMI Out-of-Band Management

## Why Out-of-Band Management Is the Control Plane for Physical Infrastructure

Every rack of servers has two networks: the data plane (where your applications talk) and the management plane (where you talk to the hardware itself). The management plane — implemented via IPMI and its vendor extensions — is what separates a managed fleet from a room full of appliances.

Out-of-band management works because the BMC (Baseboard Management Controller) is a separate embedded computer on the server motherboard. It has its own processor, its own memory, its own network interface, and its own power supply path. It runs even when the main server is powered off, hung, or not yet installed with an OS. This is the critical capability: you can power on, power off, reset, console into, and collect sensor data from a server with zero OS involvement.

At T-Mobile's scale — datacenters with hundreds to thousands of servers — out-of-band management is not optional. It is the multiplier that makes fleet management possible without proportional headcount growth.

## The BMC and Its Vendor Implementations

The **BMC (Baseboard Management Controller)** is the hardware. The **IPMI (Intelligent Platform Management Interface)** is the protocol. Vendors implement IPMI plus proprietary extensions:

| Vendor | Product | Interface Name | Extension Features |
|--------|---------|---------------|-------------------|
| Dell | PowerEdge servers | iDRAC (Integrated Dell Remote Access Controller) | OpenManage, Redfish, GUI, mobile app, iDRAC Direct USB |
| HPE | ProLiant servers | iLO (Integrated Lights-Out) | HPE OneView integration, Agentless Management, RESTful API |
| Supermicro | Supermicro servers | IPMI (generic) + IPMI View | Basic IPMI, BIOS-level GUI, limited compared to iDRAC/iLO |
| Lenovo | ThinkSystem | XCC (XClarity Controller) | XClarity Administrator integration, Redfish |

For T-Mobile procurement decisions: iDRAC (Dell) and iLO (HPE) are the two mature enterprise management platforms. Both support Redfish (the modern REST-based successor to IPMI) and have well-established automation integrations. Supermicro's BMC is adequate but requires more manual management.

**iDRAC vs iLO — practical differences:**
- iDRAC 9/10 includes a built-in HTML5 console (no Java plugin required) and Redfish v1.0+ from the factory
- iLO 5/6 similarly offers HTML5 and Redfish, with very strong integration into HPE's Synergy and OneView platforms
- iDRAC offers "iDRAC Direct" — a physical USB port on the front that gives you in-band iDRAC access without a network connection, useful for initial provisioning
- iLO historically has a reputation for a more stable, reliable management interface; iDRAC has more features in recent generations

## IPMI Protocol: The Core Commands

IPMI is accessed via the `ipmitool` command-line utility. For remote servers, the connection uses IPMI over LAN (using the `-I lanplus` flag, which enables IPMI v2.0 with AES encryption).

### Connection Syntax
```bash
ipmitool -I lanplus -H <IPMI_IP> -U <username> -P <password> <command>
```
In scripts, prefer `-f <passwordfile>` over `-P` to avoid credentials in process tables.

### Power Management
```bash
ipmitool chassis power status     # returns: Chassis Power is on/off
ipmitool chassis power on         # powers on from soft-off
ipmitool chassis power off        # immediate hard power cut (not graceful)
ipmitool chassis power soft       # sends ACPI soft power-off signal (OS graceful shutdown)
ipmitool chassis power reset      # hard reset (like hitting the reset button)
ipmitool chassis power cycle      # power off, pause, power on
```
**Operationally**: use `power soft` when the OS is responsive. Use `power reset` when the OS is hung but you want to preserve disk integrity (RAID write cache flush timing applies). Use `power cycle` for full power rail reset — necessary for some hardware fault clearing.

### Sensor Data
```bash
ipmitool sdr list                 # all sensors: name, reading, units, state
ipmitool sdr type Temperature     # only temperature sensors
ipmitool sdr type Fan             # fan speed sensors (RPM)
ipmitool sdr type Voltage         # voltage rail sensors
ipmitool sdr type Current         # current/power sensors
```
Reading the output: each line shows sensor name, current value, unit, and status (ok, nc=non-critical, cr=critical, nr=non-recoverable). A `cr` status on a temperature sensor means the hardware is asserting a critical threshold — often triggering thermal throttling or emergency shutdown.

### System Event Log (SEL)
The SEL is the hardware audit trail. Every significant hardware event is written here: corrected memory errors, voltage excursions, fan failures, power supply events, POST codes, sensor threshold crossings.
```bash
ipmitool sel info                 # SEL capacity, entries used, timestamps
ipmitool sel list                 # all events: ID, timestamp, sensor, description
ipmitool sel list last 20         # last 20 events
ipmitool sel clear                # DANGEROUS: clears entire SEL — do after documenting
```
**Triage pattern**: when a server has a problem, read the SEL before touching anything. The SEL timestamps the event sequence — whether the memory error preceded the crash, or the power event was the root cause.

### Chassis Status and LED Control
```bash
ipmitool chassis status           # power state, fault, identify LED state
ipmitool chassis identify 30      # blink front panel LED for 30 seconds (locate server in rack)
ipmitool chassis identify 0       # turn off identify LED
```
`chassis identify` is how you tell a datacenter technician which physical server to pull. Blinking the LED = remote physical location.

### Serial Over LAN (Console)
```bash
ipmitool sol activate             # open serial console to server
```
SOL gives you a text console to the server — you see boot messages, BIOS/UEFI screens, grub, kernel messages, and login prompt. It is not a graphical remote desktop. It requires SOL to be enabled in BIOS/UEFI settings and the BMC configured to forward the serial port.

This is how you debug a server that won't boot — you see the POST messages, kernel panic output, and filesystem errors without needing physical access.

## Out-of-Band Network Design

The BMC's network interface must be on a **separate management VLAN**, not on the server's production data network. This is non-negotiable for two reasons:

1. **Security**: IPMI gives root-equivalent control over the hardware. Anyone who can reach the BMC network interface can power off your servers. This network must be access-controlled, not accessible from production or internet.
2. **Isolation**: if the production network fails (NIC failure, switch failure), you need the management network to still be reachable to diagnose and remediate. They must not share failure domains.

**Standard BMC network design:**
```
[Management Workstation / Jump Host]
    |
[Management Switch (separate from data network)]
    |
    +-- BMC/iDRAC/iLO port (dedicated NIC on each server)
    +-- BMC/iDRAC/iLO port (server 2)
    +-- BMC/iDRAC/iLO port (server N)
```

Management network access requirements:
- Static IP addresses for every BMC (BMC and DHCP is operationally unacceptable — an IP change breaks all automation)
- DNS entries for every BMC IP (idrac-server01.mgmt.example.com, ilo-server02.mgmt.example.com)
- Firewall rules: management workstations and automation systems → BMC network only; no BMC access from production VLANs
- Authentication: strong passwords or certificate-based auth; never default credentials
- Audit logging: all BMC logins and power events should be logged to a central syslog target

**Segmentation at T-Mobile**: BMC management networks should be on dedicated OOB (Out-of-Band) VLANs, with access controlled by network ACLs and ideally bastion/jump host mediated. If an attacker compromises a server's OS, they should not be able to pivot to the BMC management plane.

## Securing IPMI at Scale

IPMI has a history of serious vulnerabilities (IPMI 2.0 cipher suite 0 flaw, authentication bypass bugs). Hardening requirements:

1. **Change default credentials on every BMC at provisioning time** — automate this; never ship a BMC with factory credentials
2. **Disable IPMI v1.5** — IPMI 1.5 uses cleartext; enforce IPMI v2.0 with AES encryption (`-I lanplus`)
3. **Disable Cipher Suite 0** — a IPMI 2.0 vulnerability that allows null authentication; disable in BMC config
4. **Limit user accounts** — one service account for automation, one per admin who needs direct access; no shared accounts
5. **Rotate credentials** — BMC credentials should be in a secrets vault (Vault, CyberArk) and rotated on the same schedule as other privileged credentials
6. **Firmware patching** — BMC firmware has security patches; include BMC firmware in your patch cadence (see Chapter 4 — Provisioning Lifecycle)

## Director Frame: BMC Access is Remote Hands at Scale

The analogy for a T-Mobile middleware background: IPMI/iDRAC is to physical servers what an MQ admin console is to a message broker. It gives you control plane access regardless of whether the data plane (the OS, the application, the network) is functional.

The Director-level question is not "do we use IPMI" — the answer is always yes. The question is: "what is our BMC management architecture, how is it secured, and is our automation platform integrated with it for provisioning, firmware management, and incident response?"

An engineering team that can only manage servers by physically walking the datacenter floor is not a platform engineering team — it is a datacenter operations team. The difference is BMC automation maturity.
