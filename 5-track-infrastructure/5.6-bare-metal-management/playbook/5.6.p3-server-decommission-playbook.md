# Server Decommission Playbook

**Version**: 1.0  
**Applies to**: All bare metal servers reaching end-of-life or being removed from production  
**Owner**: Platform Engineering (technical steps) + IT Asset Management (disposition)  
**Compliance**: NIST SP 800-88 Rev 1 (data sanitization standard)  
**Escalation**: Platform Engineering Manager → Security (data wipe certification) → IT Asset Management (disposition)

---

## Decommission Triggers

A server enters decommission when any of the following applies:

| Trigger | Description | Initiating Team |
|---------|-------------|----------------|
| **Refresh cycle** | Server has reached end of standard lifecycle (4-year default) | Platform Engineering (capacity planning) |
| **Hardware failure beyond economic repair** | Repair cost exceeds replacement value; or component failure requires board-level swap not covered by support contract | SRE / Platform Engineering |
| **Workload migration complete** | All workloads successfully moved to new infrastructure; server no longer needed | Platform Engineering |
| **Facility change** | Datacenter exit, rack consolidation, colo contract change | Datacenter Operations |
| **Compliance requirement** | Regulatory or security event requiring server disposal | Security / Compliance |

**Do not begin decommission without a tracking ticket** — every server decommission must have an associated ticket recording the trigger, approval, and all completed steps.

---

## Step 1 — Decommission Authorization

**1.1 Confirm decommission approval**
- Decommission request must be approved by: Platform Engineering Manager + owner of any workloads that ran on this server
- For servers with data storage: additional approval from Data Governance / Security for data wipe authorization

**1.2 Confirm no active holds**
- Check with Legal: is this server subject to litigation hold or eDiscovery? If yes: STOP — do not decommission until hold is released
- Check with Compliance: is this server included in any active audit scope? If yes: consult with Compliance before proceeding

**1.3 Document server identity**
Record in the decommission ticket:
- Asset ID / Tag
- Hostname (all historical hostnames if server was repurposed)
- Serial number
- iDRAC/iLO/BMC MAC address and IP
- Rack location (DC, row, rack, rack unit)
- All disk serial numbers (important for certificate of destruction)
- Install date, purchase order number

---

## Step 2 — Workload Verification

This is the highest-risk step. Decommissioning a server that is still serving production traffic is a P0 incident you caused.

**2.1 Kubernetes node check**
```bash
# Is this node in any K8s cluster?
kubectl get node <hostname>
# If exists: what is running on it?
kubectl get pods -A --field-selector spec.nodeName=<hostname>
# If pods are running: workload migration is NOT complete — stop and escalate
```

**2.2 VM inventory check (if server ran VMs)**
- Log into hypervisor management platform (vCenter, oVirt, etc.)
- Confirm no VMs are registered on this host
- Check for any orphaned VM disk files on this server's storage

**2.3 Monitoring check**
- Is this server still reporting metrics to Splunk or monitoring platform?
- Is this server the target of any monitoring alert rules?
- Remove from monitoring AFTER data wipe is complete (keep monitoring to confirm server is truly idle)

**2.4 DNS and load balancer check**
- Does this server's IP appear in any DNS records? If yes: remove DNS entries before or concurrent with decommission
- Does this server's IP appear in any load balancer pool, HAProxy backend, or NGINX upstream? If yes: remove from all pools first

**2.5 Application registration check**
- Does this server appear in any service registry, configuration management, or CMDB as an active service endpoint?
- Check: Consul, Eureka, or service discovery tools; Ansible inventory; Terraform state files

**2.6 Confirmation gate**
Only proceed when ALL of the following are confirmed:
- [ ] No K8s pods scheduled on this node (or node not in any cluster)
- [ ] No VMs registered on this host
- [ ] No application traffic routing to this server
- [ ] No active DNS records pointing to this server
- [ ] No active monitoring alerts targeting this server for data collection
- [ ] Signed confirmation from workload owner that migration is complete

## Step 3 — Final Data Backup (If Required)

**3.1 Determine if backup is needed**
- Is there any data on this server that is not replicated elsewhere and should be retained?
- For a K8s worker node: typically no — stateless
- For a database server: data should already be migrated; confirm
- For a file server or storage server: confirm data is replicated before proceeding

**3.2 If backup required:**
- Take final backup to approved backup destination
- Verify backup integrity (restore test or checksum verification)
- Document backup location and retention period
- Get confirmation from data owner that backup is complete and accessible

## Step 4 — Data Wipe (NIST 800-88 Compliant)

This is a compliance-sensitive step. All storage media on the server must be sanitized before physical removal from the datacenter.

### NIST SP 800-88 Rev 1 — Sanitization Methods

| Media Type | Method | Command/Tool | NIST Classification |
|------------|--------|-------------|-------------------|
| **HDD (rotating)** | DoD 5220.22-M (7-pass overwrite) or 3-pass (Clear) | `nwipe`, `shred`, `dd` | Clear → acceptable for non-confidential; Purge for confidential |
| **SSD / NVMe** | Cryptographic Erase (ATA Secure Erase or NVMe Secure Erase) | `hdparm`, `nvme-cli`, `sedutil` | Purge — most effective for SSD |
| **Self-Encrypting Drive (SED)** | Cryptographic Erase (destroy the encryption key) | Drive-native command or `sedutil` | Purge — fastest and most thorough |
| **RAID volume** | Wipe each physical disk individually after RAID dissolution | Per disk method above | Per-disk classification |

**4.1 For HDD — 3-pass overwrite (NIST Clear)**
```bash
# Using nwipe (preferred — provides logging and certificate):
nwipe --autonuke --method=dodshort /dev/sda

# Using shred (3 passes of random data + 1 final zero pass):
shred -n 3 -z -v /dev/sda

# Using dd (single pass of zeros — minimum acceptable for non-sensitive):
dd if=/dev/zero of=/dev/sda bs=1M status=progress
```

**4.2 For SSD/NVMe — Cryptographic Erase (preferred)**
```bash
# NVMe Secure Erase:
nvme format /dev/nvme0n1 --ses=1    # User Data Erase (User data areas)
nvme format /dev/nvme0n1 --ses=2    # Cryptographic Erase (destroys encryption key)

# SATA SSD — ATA Secure Erase:
hdparm -I /dev/sda | grep -i "supported enhanced erase"  # verify supported
hdparm --security-set-pass Erase /dev/sda                # set temp password
hdparm --security-erase Erase /dev/sda                   # execute erase
hdparm --security-erase-enhanced Erase /dev/sda          # enhanced (crypto) erase
```

**4.3 For RAID volumes:**
1. Dissolve the RAID array: `storcli64 /c0/v0 delete force` (loses RAID configuration — no data recovery after this)
2. Wipe each physical disk individually
3. Verify each disk: after wipe, read-verify a sample of sectors should show zeros or random data (not original data patterns)

**4.4 BMC factory reset (erase BMC configuration including credentials):**
```bash
# Dell iDRAC:
racadm racresetcfg -f

# HPE iLO:
# iLO GUI → Security → Security Settings → iLO Factory Defaults
```

**4.5 BIOS/UEFI clear:**
- Clear Trusted Platform Module (TPM) if present: BIOS → Security → TPM → Clear
- Clear BIOS passwords and settings: reset to factory defaults

## Step 5 — Data Wipe Verification and Certificate of Destruction

**5.1 Verification**
- After wipe: boot a live OS (USB), read back a sample of sectors from each disk
- All sectors should show zeros (for overwrite methods) or unreadable (for crypto erase)
- For compliance-grade verification: use `nwipe` or `dban` — they produce a verification report

**5.2 Certificate of Destruction**
- Document: asset ID, hostname, serial number, disk serial numbers, wipe method used, date/time, operator name
- If using a third-party e-waste vendor: require their NIST 800-88-compliant certificate of destruction
- File certificate of destruction in IT Asset Management system and the decommission ticket
- Retention: keep certificates for minimum 3 years (or per your regulatory requirements)

**Template for COD:**
```
CERTIFICATE OF DATA DESTRUCTION

Date: ______________
Organization: T-Mobile / Platform Engineering
Operator: ______________

Server:
  Asset ID: ______________
  Serial Number: ______________
  Hostname: ______________
  
Storage Media Sanitized:
  Disk 1: Model/SN: ______________  Method: ______________  Verified: Y/N
  Disk 2: Model/SN: ______________  Method: ______________  Verified: Y/N
  [additional rows as needed]

Sanitization Standard: NIST SP 800-88 Rev 1
Classification: Clear / Purge [circle one]

Signature: ______________  Date: ______________
Witness: ______________    Date: ______________
```

## Step 6 — Asset Record Update (CMDB)

Before physical removal from the rack:

**6.1 Update CMDB / asset management system**
- Status: Active → Decommissioned
- Decommission date: today
- Decommission reason: [refresh / failure / workload migration / facility change]
- Data wipe completed: Yes, with COD reference number
- Disposition method: [vendor return / e-waste / internal reuse]

**6.2 Remove from provisioning system**
- Remove server from Foreman/MAAS/Cobbler — server should no longer appear as manageable host
- Remove IPMI IP from BMC management network DHCP reservations or static assignment

**6.3 Remove from DNS**
- Remove all A/PTR records for this server's management IP and data network IP
- Verify removal: `nslookup <hostname>` should return NXDOMAIN

**6.4 Remove from monitoring**
- Remove server from monitoring inventory (Prometheus scrape targets, Splunk inputs)
- Remove server from alerting rules that reference it by hostname
- Archive historical metrics data per retention policy

**6.5 Remove from Ansible/Terraform inventory**
- Remove from Ansible inventory files
- If server was in Terraform state (e.g., managed via Terraform + provisioning provider): `terraform state rm` the resource

## Step 7 — Physical Removal from Rack

Physical removal requires coordination between Platform Engineering and the datacenter technician or facilities team.

**7.1 Pre-removal checklist**
- [ ] Data wipe completed and COD obtained
- [ ] CMDB updated to Decommissioned
- [ ] All cables identified and labeled (power, data, management)
- [ ] Power circuit identified (PDU port, circuit label)
- [ ] Neighboring servers in rack will not be disturbed by removal

**7.2 Cable removal sequence**
1. Power cables last — confirm server is powered off (`ipmitool chassis power off` or PDU port off)
2. Data network cables — label the switch port before disconnecting (switch port inventory update)
3. Management network cable (iDRAC/iLO) — note the switch port for management network
4. Release from rack: slide server out, note rack unit position for future use

**7.3 Post-removal**
- Install blanking panel in vacated rack unit (prevents airflow bypass that raises temperatures for remaining servers)
- Update rack diagram to show vacated space
- Release power circuit from PDU if circuit is fully unused (datacenter capacity planning)
- Notify datacenter facilities of released power capacity (kW freed)

## Step 8 — Asset Disposition

**Option A: Return to Vendor (RMA or End-of-Life Return)**
- If under vendor return program (Dell ITAD, HPE Asset Recovery): initiate return through vendor portal
- Vendor provides return shipping label and pickup
- Vendor issues final COD for any remaining data media they handle
- Record vendor COD in decommission ticket

**Option B: Certified E-Waste Disposal**
- Vendor must be R2 (Responsible Recycling) or e-Stewards certified
- Provide vendor with COD from Step 5 or allow vendor to perform their own wipe with independent COD
- Vendor issues certificate confirming compliant disposal
- File vendor COD in decommission ticket

**Option C: Internal Reuse**
- Server moves to dev/test environment or spare pool
- Update CMDB: change location and status to appropriate environment
- Re-provision with standard dev/test OS build
- Note: internal reuse does not require NIST data wipe (data wipe is for external disposition) — but do wipe to ensure no production data is accessible in dev/test context

---

## Step 9 — License and Cost Center Release

**9.1 Software licenses**
- Are any software licenses locked to this server's hardware ID, MAC address, or serial number?
- Common cases: Oracle Database (licensed per socket), some middleware (IBM MQ, TIBCO)
- Contact license manager: release license binding, update license inventory
- For subscription licenses (per-server basis): reduce subscription count effective decommission date

**9.2 Cost center release**
- If server was allocated to a specific cost center or business unit: release the allocation
- Depreciation schedule: close depreciation on this asset as of decommission date
- Inform finance: server is removed from asset register

**9.3 Support contract adjustment**
- If server was covered by per-unit support contract: notify vendor to remove from contract
- Adjust support contract value at next renewal

## Step 10 — Final Documentation

**10.1 Close decommission ticket with:**
- All completed checklist items with dates
- COD reference number(s)
- Disposition method and confirmation
- Any deviations from standard playbook (with justification)
- Final CMDB asset status (screenshot or record ID)

**10.2 Lessons learned (for batch decommission events)**
- Any process friction points in this decommission?
- Any surprises (workload still running, unknown data, license complications)?
- Update this playbook if process improvements identified
