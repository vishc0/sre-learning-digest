# Hardware Failure Response Playbook

**Version**: 1.0  
**Applies to**: All bare metal servers in production environment  
**Owner**: Platform Engineering / SRE On-Call  
**Escalation**: Platform Engineering Manager → Director, Platform Engineering

---

## Playbook Purpose

This playbook covers the response sequence for bare metal server hardware failures from initial detection through hardware replacement, validation, and documentation. The goal is to convert hardware failures from unplanned incidents into managed maintenance events.

**Time targets:**
- Detection to triage start: <5 minutes (automated alert → on-call paged)
- Triage to failure classification: <15 minutes
- Classification to remediation action: <30 minutes
- P1 hardware failure to replacement ordered: <2 hours
- P1 hardware failure to workload restored: within hardware SLA tier (NBD, 4-hr, 24x7x4)

---

## Step 1 — Detection

**1.1 Automated alert (primary)**
- Hardware monitoring alert fires: RAID degraded, DIMM ECC error rate, SMART threshold, temperature critical, fan failure
- Alert source: Splunk alert / monitoring platform
- Alert should include: hostname, IPMI/iDRAC IP, alert type, sensor value, timestamp

**1.2 IPMI SEL event (secondary)**
- Automated SEL scraping detects new critical entries
- Events: Memory ECC, PSU failure, Fan failure, Processor IERR, POST failure

**1.3 User/application report (tertiary)**
- Service alert or user report of application degradation
- First step: correlate with hardware monitoring — is this a hardware root cause?
- If yes: this playbook. If no: application incident response playbook.

**1.4 Physical observation**
- Amber/red LED on front panel (fault indicator)
- Audible alarm (high-temperature or fan alarm)
- Datacenter technician report

**→ Proceed to Step 2 regardless of detection source.**

---

## Step 2 — Initial IPMI Triage

All triage starts with IPMI before touching anything physical. Execute these commands against the affected server's IPMI/iDRAC IP.

**2.1 Verify power state**
```bash
ipmitool -I lanplus -H <IPMI_IP> -U <user> -f <pwdfile> chassis power status
# Expected: "Chassis Power is on"
# If off: check if intentional; if unplanned, proceed to sensor check before power-on
```

**2.2 Collect current sensor readings**
```bash
ipmitool -I lanplus -H <IPMI_IP> -U <user> -f <pwdfile> sdr list
# Scan for: any sensor in nc (non-critical), cr (critical), or nr (non-recoverable) state
# Record: all sensors not in "ok" state
```

**2.3 Read the System Event Log (SEL)**
```bash
ipmitool -I lanplus -H <IPMI_IP> -U <user> -f <pwdfile> sel list
# Review all entries, oldest to newest (chronological failure progression)
# Record: event type, timestamp, sensor name, assertion/deassertion
# Document in incident ticket before clearing — NEVER clear SEL without documenting it
```

**2.4 Check chassis status and fault indicators**
```bash
ipmitool -I lanplus -H <IPMI_IP> -U <user> -f <pwdfile> chassis status
# Output includes: power state, fault LED state, identify LED state, intrusion status
```

**2.5 If server is unreachable via OS (no SSH) but IPMI is reachable:**
```bash
# Open SOL console to see what the server is showing
ipmitool -I lanplus -H <IPMI_IP> -U <user> -f <pwdfile> sol activate
# Ctrl+] to exit SOL console
```

**→ With sensor readings and SEL, proceed to Step 3.**

---

## Step 3 — Classify Failure Type

Use the SEL events and sensor readings to classify:

```
SEL shows: "Memory ECC" event
  └── DIMM failure → Step 4A

SEL shows: "Drive Fault" or RAID status degraded
  └── Storage failure → Step 4B

SEL shows: "Processor IERR" or CPU sensor critical
  └── CPU/motherboard failure → Step 4C

Sensor shows: NIC link down, NIC error rate high
  └── NIC failure → Step 4D

SEL shows: "PSU Failure" or "PSU Redundancy Lost"
  └── PSU failure → Step 4E

Sensor shows: temperature critical, fans at max RPM
  └── Thermal event → Step 4F

Server will not POST (no OS load after power on)
  └── POST failure → Step 4G
```

---

## Step 4A — Storage Failure Response

**4A.1 Check RAID status**
```bash
# For servers with storcli (LSI/Broadcom RAID):
storcli64 /c0 show all
storcli64 /c0/eall/sall show

# For HPE Smart Array:
ssacli ctrl all show status
ssacli ctrl slot=0 pd all show status
```

**4A.2 Assess severity**

| RAID State | Severity | Immediate Action |
|------------|----------|-----------------|
| Virtual Drive: Optimal | Informational | Clear false alert; document |
| Virtual Drive: Degraded | **P1** | Replace failed drive immediately |
| Virtual Drive: Rebuilding | P2 | Monitor rebuild progress; do not interrupt |
| Virtual Drive: Failed/Offline | **P1 CRITICAL** | Data protection required; do not write; escalate |
| Physical Drive: Predictive Failure | P2 | Order replacement; schedule replacement within 24h |

**4A.3 For Degraded array**
1. Identify failed drive: `storcli64 /c0/eall/sall show` — look for state "UBad" or "Offln"
2. Blink failed drive LED: `storcli64 /c0/e252/s3 set locate` (locate the physical drive)
3. Check hot spare status: `storcli64 /c0 show` — if a hot spare is configured and compatible, rebuild should have started automatically
4. If no hot spare: order replacement drive immediately (Step 6: Parts Ordering)
5. Monitor rebuild: `storcli64 /c0/v0 show` — watch rebuild percentage every 30 minutes

**4A.4 Data preservation before any intervention**
- If application is still running on degraded RAID: take application backup/snapshot if feasible
- Do NOT power cycle a server with a degraded RAID array without vendor guidance — some RAID controllers may not recover the array after unexpected power cycle

---

## Step 4B — Memory (DIMM) Failure Response

**4B.1 Identify failed DIMM from SEL**
```bash
ipmitool sel list | grep -i "ECC\|Memory\|DIMM"
# SEL will show: "Memory #0x21 | Correctable ECC error | Asserted"
# Note: sensor name maps to DIMM slot in server service manual
```

**4B.2 Assess ECC error rate**
- Isolated event (1–3 corrections over days): monitor, schedule replacement at next window
- Accelerating corrections (multiple per hour): immediate replacement
- Uncorrectable error (MBE) logged: server has already or will crash — immediate action

**4B.3 Verify DIMM slot identification**
- Cross-reference SEL sensor name with server hardware service manual
- DIMM slot labels (A1, A2, B1, etc.) correspond to physical DIMM locations documented in service manual
- Record: slot label, DIMM part number (from `dmidecode -t 17` or iDRAC GUI)

**4B.4 For uncorrectable ECC (server crashed or at risk):**
- Do NOT restart application workloads until DIMM is replaced
- If server rebooted without DIMM replacement: it may work if the error was transient, but treat as imminent failure until replaced

**4B.5 Schedule replacement**
- Order replacement DIMM (matching part number and capacity)
- Replacement requires server OS shutdown and physical DIMM swap — cannot be hot-swapped
- Schedule maintenance window for workload migration (Step 5) before DIMM replacement

---

## Step 4C — CPU/Motherboard Failure

CPU failures are rare but catastrophic when they occur. Symptoms: IERR in SEL, server crashes immediately on boot attempt, server will not POST.

1. Attempt 1 cold boot via IPMI (power cycle, not reset) — some IERR events are transient
2. If server fails to POST: attach SOL console and capture POST failure message
3. If POST reports CPU failure: this is a board-level or CPU-level replacement
4. Escalate to vendor support immediately — CPU replacement or board swap is typically next business day at minimum
5. Do not attempt OS boot on a server with confirmed CPU fault — hardware instability risks data corruption

---

## Step 4D — NIC Failure

**4D.1 Determine bonding status**
```bash
cat /proc/net/bonding/bond0    # active interface, link state of each slave
ip link show                    # interface state
ethtool <interface>             # link detected, speed, duplex
```

**4D.2 If bonding is active and one NIC has failed:**
- Server is operational on the surviving NIC
- Service impact: none (bonding handled failover), but redundancy is lost
- Severity: P2 — server is running degraded on single NIC
- Action: schedule NIC replacement at next maintenance window; do not defer more than 48 hours

**4D.3 If both NICs failed or bonding did not failover:**
- Server is likely unreachable via data network
- Verify via IPMI (management network): server is running but data network is down
- Escalate to network team: is this a NIC failure or a switch port issue?
- If NIC failure: emergency maintenance required

---

## Step 4E — PSU Failure

**4E.1 Identify PSU from SEL**
- SEL will identify PSU 1 or PSU 2 as failed
- Verify redundant PSU status: `ipmitool sdr type "Power Supply"` — confirm remaining PSU is healthy

**4E.2 Assessment**
- Single PSU failed, second PSU healthy: server is operational, redundancy lost — P2
- Both PSUs failed: server is off — P1 (but hardware is protecting itself)

**4E.3 For single PSU failure:**
- Some PSUs are hot-swappable — verify before attempting to swap under load
- Order replacement PSU (matching part number and wattage)
- If hot-swappable: PSU can be replaced without workload migration
- If not hot-swappable: requires maintenance window and workload migration

---

## Step 4F — Thermal Event

Thermal events require identifying the heat source before resolving the symptom.

1. Check inlet temperature: is this a datacenter cooling failure (all servers in zone affected) or isolated to this server?
2. Check fan speeds: are fans running at maximum? Are any fans failed (0 RPM)?
3. Check CPU and DIMM temperatures: how far above threshold?
4. If datacenter cooling: escalate to datacenter facilities team immediately — not a server problem
5. If isolated to server with all fans running: possible blocked airflow (blanking panel missing, cable obstructing airflow), failed fan, or failed thermal sensor
6. If fans failed: order replacement fans, some are hot-swappable
7. Do NOT allow server to continue running above critical temperature thresholds — thermal damage accumulates

---

## Step 4G — POST Failure

Server powers on but does not complete POST (Power-On Self-Test) — does not load OS.

1. Attach SOL console: `ipmitool sol activate`
2. Observe POST output: what component is failing to initialize?
3. Common POST failures and actions:
   - "Memory not detected": DIMM failure or DIMM unseated — try reseating (physical access required)
   - "CPU not detected": CPU failure or socket damage — vendor escalation
   - "RAID controller error": RAID controller failure or configuration corruption — vendor escalation
   - Stuck at BIOS screen: BIOS settings corrupted — restore BIOS defaults, re-apply configuration

---

## Step 5 — Data Preservation and Workload Migration

Before any hardware intervention that requires a maintenance window (DIMM replacement, non-hot-swap component):

**5.1 For Kubernetes workloads:**
```bash
kubectl cordon <node-name>       # stop new pods scheduling
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data
# Verify all workloads migrated: kubectl get pods -A --field-selector spec.nodeName=<node-name>
```

**5.2 For stateful workloads (databases):**
- Promote replica before taking down primary
- Verify replication lag is zero before failover
- Confirm application is connected to new primary

**5.3 For VMs:**
- Live migrate VMs off the affected host
- Verify migration complete before proceeding

**5.4 Snapshot/backup:**
- If workload data is on the affected server and backup is feasible: take backup now
- Do not skip this step for storage failures

---

## Step 6 — Parts Ordering and RMA

**6.1 Identify part number**
- iDRAC/iLO/BMC GUI → Hardware Inventory shows part numbers for all components
- `dmidecode` shows DIMM, CPU, drive, and system information
- Vendor support portal: identify part by server model and component type

**6.2 Open vendor support case**
- For 4-hour or 24x7x4 support: call vendor support hotline (do not use email — SLA clock starts on case open, not email send)
- Provide: server serial number, iDRAC/iLO IP, SEL log (attach file), description of failure
- Request: parts dispatch under support contract; confirm SLA (4-hour, NBD)
- Record: case number, technician name, expected dispatch time

**6.3 Track dispatch**
- Confirm parts dispatched; note tracking number
- For 4-hour: vendor technician should be on-site within 4 hours of case open
- Escalate to vendor account manager if SLA is not being met

---

## Step 7 — Post-Replacement Validation

After physical replacement:

**7.1 Boot validation**
```bash
ipmitool chassis power on        # if server was shut down
# Monitor SOL console for POST completion
ipmitool chassis status          # verify power on, no fault
```

**7.2 Sensor validation**
```bash
ipmitool sdr list                # all sensors should be "ok"
ipmitool sel list                # review any new events; new component should not generate errors
```

**7.3 OS and application validation**
- SSH to server: verify OS is up, filesystem mounted, services running
- For RAID: `storcli64 /c0 show` — virtual drive should return to Optimal
- For DIMM: `edac-util -s 0` or check `/sys/devices/system/edac/` — no errors on new DIMM
- For NIC: `ip link show` — both bond interfaces up; `cat /proc/net/bonding/bond0` — both slaves active

**7.4 Workload return**
```bash
kubectl uncordon <node-name>     # allow scheduling to resume
# Verify pods reschedule: kubectl get pods -A -o wide | grep <node-name>
```

**7.5 Monitoring confirmation**
- Confirm hardware monitoring shows healthy state for all sensors on this server
- Confirm no new alerts firing for this server 15 minutes post-return

---

## Step 8 — Documentation and Asset Record Update

**8.1 Incident ticket**
- Document: SEL log at time of failure, sensor readings, failure classification, actions taken, timeline
- Record: part replaced (part number, serial number of old and new component)
- Close incident with root cause and resolution

**8.2 CMDB/inventory update**
- Update asset record: component replaced, date, new serial number if trackable
- Update warranty/support record if component replacement affects warranty

**8.3 Trend logging**
- Log to hardware failure trend database: server ID, date, component type, failure mode
- Review failure trends quarterly: patterns indicate systemic issues (bad batch of drives, rack cooling problem, etc.)

**8.4 Spare inventory update**
- If spare was consumed (hot spare activated, spare DIMM used): reorder to replenish stock
- Spare inventory should be replenished within 5 business days of use
