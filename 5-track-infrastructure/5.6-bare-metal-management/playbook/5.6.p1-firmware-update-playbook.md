# Firmware Update Playbook

**Version**: 1.0  
**Applies to**: All bare metal servers — BMC/iDRAC/iLO, BIOS/UEFI, NIC, Storage Controller firmware  
**Owner**: Platform Engineering  
**Cadence**: Quarterly for security patches; as-needed for critical fixes; annually for non-critical updates  
**Escalation**: Platform Engineering Manager → Vendor Support (case number required before engaging)

---

## Critical Safety Rules

1. **Never update firmware during a power maintenance window** — power loss mid-update = bricked component
2. **Never apply firmware to all servers simultaneously** — always use canary → group → full fleet sequence
3. **Never skip the compatibility check** — firmware versions between BMC, BIOS, NIC, and storage controller have interdependencies
4. **Never clear the SEL before documenting it** — preserve pre-update hardware state for post-update comparison
5. **Always have a rollback plan before starting** — some firmware updates are not rollback-able; know this in advance

---

## Step 1 — Pre-Update Inventory

Before any firmware work begins, establish a complete picture of current firmware versions across the fleet.

**1.1 Collect current firmware inventory**

For Dell PowerEdge (iDRAC):
```bash
# Via RACADM (from OS):
racadm getversion

# Via ipmitool (BMC firmware only):
ipmitool bmc info | grep "Firmware Revision"

# Via Redfish API (complete firmware inventory):
curl -sk -u admin:<pass> https://<idrac-ip>/redfish/v1/UpdateService/FirmwareInventory | \
  python3 -m json.tool | grep -E '"Id"|"Version"'
```

For HPE ProLiant (iLO):
```bash
# Via iLO REST API:
curl -sk -u administrator:<pass> https://<ilo-ip>/rest/v1/systems/1/bios | \
  python3 -m json.tool | grep -E '"CurrentVersion"|"VersionString"'
```

**1.2 Inventory spreadsheet — build this before starting:**

| Hostname | iDRAC IP | BMC Current | BMC Target | BIOS Current | BIOS Target | NIC Current | NIC Target | Storage Ctrl Current | Storage Ctrl Target | Status |
|----------|----------|-------------|-----------|--------------|------------|-------------|-----------|---------------------|--------------------|----|
| server01 | 10.1.1.101 | 6.00.00.00 | 6.10.30.00 | 2.8.2 | 2.12.1 | 21.10.09 | 22.10.09 | 52.12.0 | 52.18.0 | Pending |

**1.3 Identify update urgency classification**

| Classification | Criteria | Action |
|---------------|---------|--------|
| **Critical (Security)** | CVE with CVSS ≥ 7.0, actively exploited vulnerability in BMC/BIOS/NIC | Expedited schedule; complete within 30 days |
| **High** | CVE with CVSS 4.0–6.9, functional regression fix for known issue in environment | Plan within 60 days |
| **Maintenance** | Performance improvement, compatibility update, no security relevance | Next quarterly window |

## Step 2 — Compatibility Matrix Verification

Firmware versions across components are interdependent. A BIOS update may require a minimum BMC firmware version. A NIC driver may require a minimum NIC firmware version.

**2.1 Obtain compatibility matrix from vendor**
- Dell: downloads.dell.com → server model → Compatibility Matrix / Release Notes
- HPE: support.hpe.com → Service Pack for ProLiant (SPP) → compatibility matrix
- Read the release notes for every firmware component in the update plan

**2.2 Document dependencies:**

| Component | Target Version | Requires |
|-----------|---------------|---------|
| BIOS 2.12.1 | iDRAC firmware ≥ 6.05.00 | Update BMC first |
| NIC firmware 22.10.09 | OS NIC driver ≥ 5.4 | Verify driver before NIC firmware |
| Storage Controller 52.18.0 | No dependencies noted | Can proceed after BIOS |

**2.3 Update sequencing (always in this order):**
1. BMC / iDRAC / iLO firmware (management controller first — it orchestrates other updates)
2. BIOS / UEFI
3. NIC firmware
4. Storage controller firmware

Rationale: the BMC manages the update process for other components. If BMC is outdated, it may not correctly handle BIOS or storage controller update workflows. Updating BMC first ensures the update infrastructure is current.

## Step 3 — Risk Assessment

For each server class being updated, complete this risk assessment:

**3.1 Business impact assessment**

| Server Role | Workload Criticality | Acceptable Downtime Window | Risk Level |
|-------------|---------------------|--------------------------|-----------|
| K8s worker node (N of M) | Can drain and resume | Yes — can drain first | Low |
| K8s control plane node | Critical — quorum risk | Only with etcd backup confirmed | Medium |
| Database primary | Highest — service impact | Failover first, then update | High |
| Database replica | Can failover workload | Standard window | Medium |
| Standalone server | Variable | Per workload | Variable |

**3.2 Rollback feasibility check**

| Firmware Type | Rollback Possible? | Method |
|--------------|-------------------|--------|
| BMC/iDRAC firmware | Usually yes — dual firmware bank | iDRAC GUI → Firmware → Rollback to previous |
| BIOS/UEFI | Usually yes — some vendors maintain previous version | BIOS setup utility → firmware rollback option |
| NIC firmware | Depends on vendor | Check vendor documentation |
| Storage controller | **Often NO** — check explicitly | Vendor release notes: "Downgrade not supported" is common |

For storage controller firmware where rollback is not supported: increase the canary stage validation time and require explicit sign-off before fleet-wide rollout.

## Step 4 — Staging and Canary Approach

Never update all servers simultaneously. Use this progression:

```
Canary (2–4 servers) → Rack/Group (10–15% of fleet) → Full Fleet
   |                        |                               |
   Validate 72h            Validate 48h                   Rolling update
```

**4.1 Select canary servers**
- Non-critical workloads preferred for canary (dev/test tier if available in same server class)
- If no non-critical equivalent: select servers where workload can be migrated
- Never use a database primary as canary

**4.2 Canary validation criteria before proceeding to next stage**

After updating canary servers, wait minimum 72 hours and verify:
- [ ] All sensors healthy (no new critical/non-critical states)
- [ ] No new SEL events post-update
- [ ] OS boots cleanly, all services start
- [ ] Workload performance baseline unchanged (check Splunk metrics for these hosts)
- [ ] Network connectivity stable (no link flap events)
- [ ] RAID status optimal (if storage controller firmware was updated)
- [ ] Monitoring shows no anomalies

If any canary validation criterion fails: **STOP**. Do not proceed to group stage. Investigate and resolve before continuing.

## Step 5 — Workload Migration Before Update

For any server requiring OS shutdown for firmware update:

**5.1 Kubernetes nodes**
```bash
# Cordon to stop new scheduling
kubectl cordon <node-name>

# Drain all pods (wait for completion)
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data --timeout=300s

# Verify drained
kubectl get pods -A --field-selector spec.nodeName=<node-name>
# Expected: only DaemonSet pods remain (kube-proxy, node-exporter, etc.)
```

**5.2 Database servers**
- Promote replica to primary before updating primary
- Confirm replication lag = 0 before failover
- Update the demoted primary while it is in replica role

**5.3 Virtual machine hosts**
- Live migrate VMs to other hosts in cluster
- Confirm migration complete and all VMs healthy on new host

## Step 6 — Update Execution

### Method A: Dell iDRAC (RACADM)
```bash
# Download firmware update file from Dell support portal first
# .exe (Windows installer) or .bin (Linux), or Dell Repository Manager

# Upload and update via RACADM:
racadm update -f <firmware_file.exe> -e idrac

# For BIOS:
racadm update -f <bios_file.exe> -e bios

# Monitor update status:
racadm jobqueue view
# Watch for: JobStatus: Running → Completed

# Alternatively: iDRAC GUI → Update and Rollback → upload file → apply
```

### Method B: Dell Lifecycle Controller (GUI)
- Boot to BIOS → Launch Lifecycle Controller
- Firmware Update → Local Drive / Network → select update files
- Best for: initial provisioning or recovery scenarios where OS is unavailable

### Method C: HPE iLO (RESTful API / GUI)
```bash
# Via iLO REST API:
curl -sk -X POST -u administrator:<pass> \
  -H "Content-Type: application/octet-stream" \
  --data-binary @<firmware_file.bin> \
  https://<ilo-ip>/rest/v1/UpdateService

# Via GUI: iLO → Firmware & OS Software → Update Firmware → upload file
```

### Method D: Vendor Update Catalogs (Fleet-Scale)
- **Dell OpenManage Enterprise**: push updates to server groups via catalog
- **HPE Service Pack for ProLiant (SPP)**: bootable ISO that applies all firmware from baseline catalog
- **Ansible**: using vendor Ansible modules (dellemc.openmanage, hpe.oneview) for programmatic updates

For fleet-scale updates, use vendor management platforms or Ansible. Individual server-by-server GUI updates do not scale.

## Step 7 — Validation Post-Update

After each server update, complete these validation steps before marking complete:

**7.1 Immediate post-update checks (server just rebooted)**
```bash
# Verify power state
ipmitool chassis power status

# Check all sensors — should all be "ok"
ipmitool sdr list | grep -v ok

# Review SEL for any post-update events
ipmitool sel list | tail -20

# Verify firmware version applied
ipmitool bmc info | grep "Firmware Revision"
racadm getversion  # Dell: confirms all component firmware versions
```

**7.2 OS validation**
```bash
# Verify OS boot
ssh <server> "uptime; dmesg | tail -20; journalctl -p err --since '1 hour ago'"

# Check for any hardware errors in dmesg
dmesg | grep -iE "error|fail|fault|critical"

# NIC bond status
cat /proc/net/bonding/bond0

# RAID status (if storage controller updated)
storcli64 /c0 show | grep -E "State|Health"
```

**7.3 Workload return**
```bash
# K8s: uncordon and verify pods reschedule
kubectl uncordon <node-name>
kubectl get pods -A -o wide | grep <node-name>
```

## Step 8 — Rollback Procedure

**When to rollback:**
- Server will not POST after update
- OS fails to boot after update
- Significant performance regression confirmed post-update
- Hardware alerts that did not exist pre-update

**8.1 BMC/iDRAC rollback (dual firmware bank)**
```bash
# iDRAC GUI: Update and Rollback → Rollback → select iDRAC → apply
# Or RACADM:
racadm rollback idrac
```

**8.2 BIOS rollback**
- Most Dell and HPE servers support one-version BIOS rollback
- iDRAC GUI: Update and Rollback → Rollback → select BIOS
- Some vendors prohibit BIOS downgrade — check release notes

**8.3 Storage controller rollback**
- Often NOT SUPPORTED — if release notes say "downgrade not supported," you cannot roll back
- Recovery path: vendor support case → may require controller replacement or recovery process

**8.4 If server will not POST (most severe scenario)**
- Attempt BMC reset via IPMI: `ipmitool mc reset cold`
- If BMC unresponsive: physical access required — recovery USB or serial console (see server service manual)
- Open vendor support case immediately — treat as P1; engage 24x7 support line

## Step 9 — Fleet Tracking and Documentation

**9.1 Update inventory spreadsheet**
- Mark each server "Completed" with date and applied versions
- Record any anomalies or exceptions during update

**9.2 CMDB update**
- Update firmware version fields in CMDB for every server updated

**9.3 Incident/change record**
- Close change record with completion summary
- Document any servers that could not be updated (reason, deferred timeline)

**9.4 Deferred servers**
- If a server could not be updated (workload could not be migrated, server in failed state, hardware constraint): record explicitly with owner and target date
- Do not consider the update cycle complete while servers remain on the old firmware without a documented exception

**9.5 Post-update monitoring period**
- Flag the updated server group in monitoring for 7 days post-update
- Review any hardware alerts during this period for correlation with firmware change
