# Chapter 4 — Provisioning Lifecycle

## Provisioning Automation Is the Multiplier

An engineer provisioning servers manually — configuring BIOS, installing OS via USB, running post-install scripts — can handle maybe 50–100 servers before the process becomes the bottleneck. An engineer with a mature provisioning pipeline — PXE boot, automated OS installation, configuration management handoff — manages 1,000+ servers with the same effort. The pipeline is the multiplier.

T-Mobile's scale makes this non-optional. A provisioning gap is not a minor inefficiency; it is a hard ceiling on your deployment velocity and refresh cadence.

## PXE Boot: How Automated Provisioning Starts

PXE (Preboot Execution Environment) is the mechanism that allows a server to boot from the network before an OS is installed on its local disks.

The flow:
```
1. Server powers on, BIOS/UEFI attempts PXE boot
2. Server broadcasts DHCP DISCOVER (looking for IP address)
3. DHCP server responds with IP + "next-server" (TFTP server address) + "filename" (bootloader)
4. Server downloads bootloader via TFTP
5. Bootloader runs, downloads kernel and initrd (initial ramdisk) via TFTP (or HTTP via iPXE)
6. Kernel boots, OS installer runs
7. Installer fetches kickstart/preseed/autoinstall file for unattended installation
8. OS installs, reboots from local disk
```

**What goes wrong with PXE at scale:**

- **VLAN tagging**: DHCP DISCOVER from PXE is untagged. If the server's switch port is in a data VLAN, the DHCP broadcast reaches the wrong DHCP server (or no DHCP server). Solution: configure the switch port to the provisioning VLAN for new servers, then move to the production VLAN post-provision, or use DHCP relay agents that forward to the provisioning DHCP scope.
- **DHCP scope exhaustion**: a /24 provisioning subnet only supports 254 simultaneous provisions. For large refresh events (provisioning 500 servers), scope size and DHCP lease duration must be planned.
- **TFTP reliability**: TFTP uses UDP and has no congestion control. Large bootloader files over lossy networks can fail silently. Modern provisioning moves file delivery to HTTP (iPXE solves this).
- **Boot order conflict**: if a server has an OS on local disk, it boots from disk before PXE. Provisioning systems must either configure boot order via BMC (IPMI chassis bootdev pxe) or rely on network-first boot order configured at procurement.

```bash
# Force next boot from PXE via IPMI (one-time, reverts after reboot)
ipmitool chassis bootdev pxe
# Force next boot from disk
ipmitool chassis bootdev disk
```

## iPXE: Advantages Over Classic PXE

iPXE is an open-source network boot firmware that replaces the classic PXE ROM. Advantages:

- **HTTP delivery**: downloads bootloaders and kernels via HTTP instead of TFTP — faster, reliable, supports large files
- **Scripting**: iPXE scripts can present boot menus, check conditions, chain to different boot sources based on server attributes (MAC address, serial number, data from a provisioning API)
- **HTTPS**: encrypted provisioning workflow — protects kickstart files with credentials or tokens
- **Multiple protocols**: FTP, NFS, iSCSI, AoE in addition to HTTP/HTTPS

Typical iPXE workflow for a managed fleet:
```
Server PXE boots → downloads iPXE binary via TFTP → 
iPXE calls HTTP API: "what should I boot, given my MAC address?" →
API responds with boot script based on server role → 
iPXE downloads kernel + initrd via HTTP → 
kernel boots with kickstart/preseed URL embedded
```

This allows the provisioning API to be the source of truth for server roles — MAC address to role mapping drives the entire automated installation.

## OS Provisioning Tools: Comparison

| Tool | Ecosystem | Strengths | Limitations | Best For |
|------|-----------|-----------|-------------|----------|
| **Cobbler** | Red Hat / RHEL | Tight integration with RHEL, simple DHCP/DNS/TFTP management, mature | Limited UI, requires Linux expertise to operate, aging | RHEL/CentOS fleets, teams already in Red Hat ecosystem |
| **Foreman + Katello** | Red Hat | Full lifecycle: provisioning → patch management → subscription management → reporting | Complex to set up and maintain, significant learning curve | Enterprise RHEL fleets requiring full lifecycle management |
| **MAAS (Metal as a Service)** | Canonical / Ubuntu | Cloud-like UX for bare metal — provision via API or UI, treats servers like cloud instances, excellent BMC integration | Ubuntu-centric (RHEL support is limited), requires Canonical ecosystem buy-in | Ubuntu-based bare metal fleets, Kubernetes on bare metal via MAAS + Juju |
| **Tinkerbell** | CNCF / Equinix | Cloud-native, Kubernetes-native provisioning, workflow-based | Newer, smaller community, steeper initial setup | K8s-native shops provisioning bare metal K8s nodes |

For T-Mobile context: **Foreman/Katello** if the fleet is primarily RHEL/CentOS and you need patch lifecycle management integrated. **MAAS** if you are building bare metal Kubernetes clusters on Ubuntu and want a cloud-like provisioning API.

## BIOS/UEFI: What Matters at Scale

BIOS/UEFI is the firmware that runs before the OS. For provisioning pipelines, the relevant settings:

- **Boot order**: network first (for provisioning), disk first (for production). Boot order is configurable via IPMI or the vendor's management interface without physically touching the server.
- **Secure Boot**: UEFI feature that validates bootloader signatures against a trusted key database. Prevents loading unsigned kernels/bootloaders. In provisioning pipelines, Secure Boot must be configured to trust your signed bootloader, or disabled (less secure). For K8s nodes running custom kernels (eBPF-enabled), Secure Boot policy must allow the custom kernel's signature.
- **UEFI vs Legacy BIOS**: Modern provisioning assumes UEFI. UEFI supports GPT partition tables (required for disks >2TB), network boot improvements (HTTP Boot, IPv6 PXE), and Secure Boot. Legacy BIOS provisioning is a maintenance burden on modern hardware — if you are buying new hardware, mandate UEFI and budget time to migrate provisioning scripts that assume legacy BIOS behavior.
- **BIOS performance settings**: for latency-sensitive workloads (telco NFV, low-latency databases), BIOS settings like CPU C-states, P-states, Hyper-Threading, NUMA interleaving, and hardware prefetcher must be configured explicitly. Default BIOS settings optimize for power efficiency, not performance. A latency-sensitive server with default BIOS settings is a performance problem waiting to be discovered.

## Firmware Updates: Sequencing and Risk

Firmware — BMC firmware, BIOS/UEFI, NIC firmware, storage controller firmware — must be kept current. Firmware vulnerabilities (CPU microcode, NIC firmware bugs, BMC security patches) are real and well-documented.

**Update sequencing matters**:
1. **BMC/iDRAC/iLO firmware first**: the management controller must be updated first because it orchestrates other component updates. If you update BIOS first and the BIOS update requires a specific BMC version to complete safely, you have a recovery problem.
2. **BIOS/UEFI second**: after BMC is at target version, update BIOS. BIOS updates require a server reboot to apply.
3. **NIC firmware third**: some NIC firmware updates require a cold boot (power cycle, not warm reboot) to activate. NIC firmware bugs can cause link flap, LACP instability, and silent packet drops.
4. **Storage controller firmware last**: storage controller firmware updates carry the highest risk — a failed update can make your RAID array unreadable. Always ensure backup/snapshot before storage controller firmware updates.

**Risk factors**:
- Applying the wrong firmware version (mismatched hardware revision) can brick the component
- Power loss during a firmware update = bricked server. Use UPS. Never update firmware during a power maintenance window.
- BMC firmware update failures sometimes require physical access (serial console + recovery USB) — know this before you start a fleet-wide update

## The Full Lifecycle: Procurement to Decommission

```
PROCUREMENT
  └── RFI/RFQ → vendor selection → purchase order → delivery tracking

RACK (Physical)
  └── Receive → inspect → rack → cable (data + management + power) → PDU circuit assignment

PROVISION
  └── BIOS/UEFI config → BMC IP assignment → PXE boot → OS install → config management (Ansible/Puppet) → validation

OPERATE
  └── Patch management → firmware updates → hardware monitoring → incident response → capacity planning

REFRESH PLANNING
  └── Warranty tracking → age tracking → performance benchmarking vs current-gen → CapEx planning

DECOMMISSION
  └── Workload migration → data wipe → CMDB update → physical removal → asset disposition
```

Each stage has automation opportunities. The stages that are still manual in most organizations — firmware updates and decommission data wipe — are where the most risk and labor concentrate.

## Director Frame: Provisioning Automation as a Platform Capability

The way to frame provisioning maturity in an interview:

**Level 0 (manual)**: engineers provision servers by hand. Deployment time = days per server. No repeatability, no auditability.

**Level 1 (automated OS install)**: PXE boot + kickstart/preseed. Unattended OS install. Deployment time = hours. Consistent OS baseline.

**Level 2 (lifecycle platform)**: Foreman/MAAS + configuration management. Provisioning + patch lifecycle + hardware inventory in a single platform. Deployment time = minutes. Full auditability.

**Level 3 (API-driven)**: provisioning pipeline triggered by API call — hardware requested, BMC configured, OS installed, K8s node joined — with no human steps. Deployment time = time-to-POST (hardware brings up) + install time. Fully auditable, GitOps-compatible.

T-Mobile's physical footprint warrants Level 2 at minimum, Level 3 for net-new build-outs. The business case is straightforward: Level 0 to Level 2 is a one-time investment that pays back on every server provisioning event, every refresh cycle, and every incident where you need to reprovisioning quickly.
