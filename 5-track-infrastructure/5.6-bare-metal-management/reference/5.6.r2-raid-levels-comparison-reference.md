# RAID Levels Comparison Reference

**Scope**: Hardware RAID (controller-managed) and software RAID (mdadm, ZFS RAID-Z) common to bare metal deployments  
**Audience**: SRE/Director — design decisions, incident response, vendor proposals  
**Quick Rule**: RAID 10 for databases, RAID 6 for large data volumes, JBOD for software-defined storage (Ceph, ZFS)

---

## Per-Level Deep Dive

### RAID 0 — Striped (No Fault Tolerance)

**Usable**: 100% (N × disk)  
**Fault Tolerance**: None. Any single disk failure loses all data.  
**Performance**: Best read and write of all RAID levels — all disks contribute to every I/O.

**Write hole risk**: No — RAID 0 writes are atomic per stripe; no parity to get out of sync.

**Best Use Case**:
- Scratch/temp storage where data loss is acceptable and performance is required
- Cache tier that is rebuilt from primary store on failure (Redis, Splunk warm tier if backed up)
- Never for any data that must survive hardware failure

**Avoid when**: anything requires data persistence or durability.

**SRE/Director Note**: RAID 0 appearing on a database or application server is a misconfiguration. If a vendor proposes RAID 0 for anything but explicitly scratch storage, reject it.

---

### RAID 1 — Mirrored

**Usable**: 50% (requires 2 disks minimum; scales as N/2 with additional mirrors but this is unusual)  
**Fault Tolerance**: 1 disk (and beyond — all mirrors can fail except one)  
**Performance**: Read is fast (read from either disk, can be load-balanced); write is single-disk speed (must write to all mirrors).

**Write hole risk**: No — both copies written synchronously; if one fails mid-write, the mirror is intact.

**Best Use Case**:
- Boot drives — a server's OS disk is almost always RAID 1; if a boot drive fails, the server boots from the mirror
- Small, critical datasets — /var/log, configuration files, database transaction logs
- Any situation where simplicity and fast rebuild outweigh capacity cost

**Rebuild**: fast — copy from surviving mirror to replacement. Low I/O impact on application.

**SRE/Director Note**: all boot drives should be RAID 1. Non-negotiable. A server that boots from a single disk has an unacceptable availability risk.

---

### RAID 5 — Striped with Distributed Parity

**Usable**: (N-1) disks — 3 disks = 2 usable, 8 disks = 7 usable  
**Fault Tolerance**: 1 disk only  
**Performance**: Excellent read (all disks contribute). Write has 4-I/O penalty per write (2 reads + 2 writes for parity calculation on hardware RAID without write cache).

**Write hole risk**: YES. If a server crashes during a write that spans a RAID 5 stripe:
1. Data is written to 2 disks; parity write to 3rd disk is in-flight
2. On restart: parity and data are inconsistent
3. This inconsistency is invisible until a disk fails and parity is used to reconstruct

Mitigation: hardware RAID controller with NVRAM-backed write cache (flushes atomically). ZFS RAID-Z eliminates this with copy-on-write semantics.

**Rebuild impact**: severe. To rebuild a failed disk, the controller must read ALL remaining disks and compute parity. For 8 disks with 4TB each: reads 7 × 4TB = 28TB of data. On a busy array, this takes 6–12+ hours. During this entire window: one disk failure = data loss.

**Best Use Case**:
- General-purpose storage where capacity efficiency matters more than rebuild safety
- Archival data, lower-churn workloads
- Arrays with fewer than 6 disks (larger RAID 5 arrays have longer rebuild windows and higher URE risk)

**URE (Unrecoverable Read Error) risk during rebuild**: HDDs have a 1-in-10^14 to 1-in-10^15 bit error rate. For a 4TB drive: ~1 URE per 12–120 TB read. During a RAID 5 rebuild reading 28TB, probability of a URE on the surviving disks is not negligible — another reason RAID 6 is preferred for large arrays.

**SRE/Director Note**: Do not use RAID 5 for arrays larger than 6 disks or drives larger than 2TB. The rebuild window and URE risk make it unsuitable for large modern drives. RAID 6 is the correct choice for those configurations.

---

### RAID 6 — Striped with Dual Distributed Parity

**Usable**: (N-2) disks — 4 disks = 2 usable, 8 disks = 6 usable  
**Fault Tolerance**: 2 simultaneous disk failures  
**Performance**: Excellent read. Write has higher penalty than RAID 5 (6 I/Os: 4 reads + 2 parity writes).

**Write hole risk**: YES — same risk as RAID 5, same mitigation (NVRAM cache).

**Rebuild**: even heavier than RAID 5 (must read all remaining disks for dual parity calculation). Rebuild time is longer. However, during rebuild of the first failed disk, the array can tolerate a second disk failure — this is the critical advantage.

**Best Use Case**:
- Large storage arrays (≥6 disks, large capacity per disk)
- Any array where the rebuild duration creates a meaningful risk of data loss from a second failure
- General-purpose enterprise storage where capacity efficiency is needed but RAID 10's 50% overhead is too costly

**Avoid when**: write latency is critical (6-I/O write penalty). RAID 10 is better for high-IOPS, write-intensive workloads.

**SRE/Director Note**: RAID 6 is the standard for large-array general-purpose storage in enterprise environments. If a vendor proposes RAID 5 for an 8-disk or larger configuration, push back — RAID 6 is the safer choice.

---

### RAID 10 (RAID 1+0) — Mirrored Stripes

**Usable**: 50% (N/2 × disk)  
**Fault Tolerance**: 1 disk per mirror pair — in a 4-disk RAID 10 (2 mirror pairs), you can lose 1 disk from each pair simultaneously (2 total) as long as they are from different pairs  
**Performance**: Excellent read AND write — mirroring is simpler than parity; writes go to 2 disks simultaneously without parity calculation overhead.

**Write hole risk**: None — writes are atomic mirror copies.

**Rebuild**: fast and low-impact. Like RAID 1, rebuild copies from the surviving mirror. For 4TB drives: 4TB copy at disk speed = 1–2 hours. Application I/O impact is minimal compared to RAID 5/6 rebuilds.

**Best Use Case**:
- Databases (Oracle, PostgreSQL, MySQL) — high IOPS, write-intensive, fast rebuild matters
- Transaction logs and redo logs
- Any workload where write latency and rebuild speed justify the 50% capacity cost

**Avoid when**: capacity efficiency is a primary concern. RAID 10's 50% overhead is expensive at large scale. Consider RAID 6 for large archival or warm storage.

**SRE/Director Note**: RAID 10 is the right answer for database data volumes. The 50% capacity cost is justified by: no write penalty (vs. RAID 5/6), fast rebuild (vs. RAID 5/6), and simple failure mode understanding.

---

### RAID 50 and RAID 60

**RAID 50**: RAID 0 striped across multiple RAID 5 groups  
**RAID 60**: RAID 0 striped across multiple RAID 6 groups

Used for very large disk counts (12+ disks) where a single RAID 5 or RAID 6 group would have unacceptably long rebuild times. By splitting into groups, rebuild only affects disks in one group.

**Best Use Case**:
- Disk-dense servers with 12–24+ disks
- Storage-optimized servers (backup targets, NAS, large data lakes)
- When single RAID 5 or RAID 6 rebuild time is too long

**Avoid when**: simplicity is preferred; RAID 50/60 configurations are complex to manage and recover.

---

### JBOD (Just a Bunch of Disks)

**Usable**: 100% — no RAID overhead  
**Fault Tolerance**: 0 at hardware level — software layer manages redundancy  
**Controller Requirement**: HBA pass-through (the controller passes raw disks to the OS; no RAID performed by hardware)

**Best Use Case**:
- Ceph OSD nodes — Ceph requires raw disk access; hardware RAID prevents Ceph from seeing individual drives
- ZFS — ZFS implements its own RAID-Z with copy-on-write (no write hole); hardware RAID interferes
- Kafka, Elasticsearch, Cassandra — applications with built-in replication that prefer raw disk access
- Longhorn, OpenEBS — all software-defined storage solutions that manage their own redundancy

**SRE/Director Note**: if your storage architecture is software-defined (Ceph, ZFS), the storage controller MUST be in HBA pass-through mode. Hardware RAID and software-defined storage are incompatible — hardware RAID hides individual disk health, disables smart firmware, and prevents the software from managing disks directly. This is a procurement specification item: specify HBA mode in your RFI.

## RAID Decision Cheat Sheet

```
What is the use case?
│
├── Boot drive → RAID 1 (always)
│
├── High-IOPS database (Oracle, PostgreSQL) → RAID 10
│
├── General-purpose large storage array
│   ├── ≤6 disks, ≤2TB each → RAID 5 (acceptable rebuild window)
│   └── >6 disks OR >2TB each → RAID 6 (rebuild window + URE risk)
│
├── Software-defined storage (Ceph, ZFS, Longhorn) → JBOD (HBA pass-through)
│
├── Performance-only scratch/temp (loss acceptable) → RAID 0
│
└── Disk-dense server (12+ disks, archival or NAS) → RAID 50 or RAID 60
```

## Rebuild Time Estimates by RAID Level and Disk Size

*(Estimates assume 50% I/O load during rebuild — production workloads running)*

| RAID Level | 1TB Disk | 4TB Disk | 8TB Disk | 16TB Disk |
|------------|---------|---------|---------|----------|
| RAID 1 | 30–60 min | 2–4 hours | 4–8 hours | 8–16 hours |
| RAID 5 (6-disk) | 2–4 hours | 8–16 hours | 16–32 hours | 32–64 hours |
| RAID 6 (8-disk) | 3–6 hours | 12–24 hours | 24–48 hours | 48–96 hours |
| RAID 10 (8-disk) | 30–60 min | 2–4 hours | 4–8 hours | 8–16 hours |

**Key insight**: a RAID 6 rebuild on 8×16TB drives takes 48–96 hours. During this window — for a RAID 6 configuration — one more disk failure is tolerated. But at the end of that 96-hour window, you want the rebuild complete. This is why disk replacement SLAs for large storage arrays are measured in hours, not days.
