# IPMI Command Quick Reference

**Tool**: `ipmitool` — available on most Linux distributions (`yum install ipmitool` / `apt install ipmitool`)  
**Version requirement**: ipmitool 1.8.18+ recommended  
**Protocol**: IPMI v2.0 (use `-I lanplus` for remote access — enforces AES encryption)

---

## Remote Connection Syntax

```bash
# Standard remote command template:
ipmitool -I lanplus -H <IPMI_IP> -U <username> -P <password> <command>

# Preferred in scripts (avoids password in process table):
ipmitool -I lanplus -H <IPMI_IP> -U <username> -f /path/to/pwdfile <command>

# Local (on the server itself — no network required, BMC accessed via /dev/ipmi0):
ipmitool <command>

# Local requires ipmi_si kernel module:
modprobe ipmi_si
```

## Power Management Commands

| Command | What It Does | Expected Output | When to Use |
|---------|-------------|----------------|-------------|
| `chassis power status` | Query current power state | `Chassis Power is on` or `...is off` | First check in any triage |
| `chassis power on` | Power on from soft-off state | `Chassis Power Control: Up/On` | Boot a powered-off server |
| `chassis power off` | **Hard cut power** (no graceful shutdown) | `Chassis Power Control: Down/Off` | OS is hung, no graceful option |
| `chassis power soft` | Send ACPI soft power-off (OS handles graceful shutdown) | `Chassis Power Control: Soft` | OS is responsive; use for planned power-off |
| `chassis power reset` | Hard reset (like pressing reset button) | `Chassis Power Control: Reset` | OS hung; NMI not working; need immediate reset |
| `chassis power cycle` | Power off, 1-second pause, power on | `Chassis Power Control: Cycle` | Full power rail reset; clearing some hardware faults |

**Safety note**: `power off` and `power reset` are immediate — no OS shutdown. Risk of filesystem corruption without journaling. Use `power soft` when OS is accessible.

## Boot Device Control

| Command | What It Does | Persistence |
|---------|-------------|------------|
| `chassis bootdev disk` | Next boot from local disk | One-time (reverts to BIOS boot order after) |
| `chassis bootdev pxe` | Next boot from PXE/network | One-time |
| `chassis bootdev cdrom` | Next boot from virtual CD/DVD | One-time |
| `chassis bootdev bios` | Next boot into BIOS setup | One-time |
| `chassis bootdev disk options=persistent` | Set disk as default boot device | Persistent |
| `chassis bootdev pxe options=persistent` | Set PXE as default boot device | Persistent |

```bash
# Example: force next boot from PXE, then power cycle
ipmitool -I lanplus -H 10.1.0.101 -U admin -f /etc/ipmi/pass chassis bootdev pxe
ipmitool -I lanplus -H 10.1.0.101 -U admin -f /etc/ipmi/pass chassis power cycle
```

## Sensor Data (SDR) Commands

| Command | What It Returns | How to Read Output |
|---------|---------------|-------------------|
| `sdr list` | All sensors: name, value, units, state | Each line: `<sensor name> | <value> <units> | <state>` — state: ok / nc / cr / nr |
| `sdr type Temperature` | Temperature sensors only | Same format; look for any `cr` or `nc` state |
| `sdr type Fan` | Fan speed sensors (RPM) | `0` RPM = failed fan |
| `sdr type Voltage` | Voltage rail sensors | Any non-ok state = PSU or VRM issue |
| `sdr type Current` | Current / power sensors | Shows wattage if supported |
| `sdr type "Power Supply"` | PSU presence and status | Look for: `Presence detected` and `Power Supply AC lost` events |
| `sdr elist` | Extended sensor list with additional info | More verbose than `sdr list` |
| `sdr get "<sensor name>"` | Details for a specific sensor | Full reading + threshold values |

```bash
# Example: find any sensor not in "ok" state
ipmitool -I lanplus -H 10.1.0.101 -U admin -f /pass sdr list | grep -v " ok$"

# Example: get specific CPU temperature with thresholds
ipmitool sdr get "CPU1 Temp"
# Output includes: Reading, Lower Non-Critical, Lower Critical, Upper Non-Critical, Upper Critical, Upper Non-Recoverable
```

**State codes**:
- `ok` — within normal thresholds
- `nc` — non-critical: above warning threshold, not yet critical — investigate
- `cr` — critical: above critical threshold — immediate action required; may trigger throttling or shutdown
- `nr` — non-recoverable: hardware is protecting itself — server may shut down
- `ns` — not specified: sensor present but no reading (often disabled or not installed)

## System Event Log (SEL) Commands

| Command | What It Does | Notes |
|---------|-------------|-------|
| `sel info` | SEL metadata: capacity, entries used, last add/erase timestamps | Check "# of alloc units" to see if SEL is full |
| `sel list` | Print all SEL events in format: ID, date, time, sensor, event, direction | Oldest first; read before clearing |
| `sel list last 20` | Print last 20 events | Useful for recent events in a large SEL |
| `sel elist` | Extended SEL — includes sensor type and more metadata | More verbose than `sel list` |
| `sel clear` | **IRREVERSIBLE: clears entire SEL** | Always document/export before clearing |
| `sel save /tmp/sel-$(hostname)-$(date +%Y%m%d).txt` | Save SEL to file before clearing | Best practice before any maintenance |
| `sel delete <ID>` | Delete specific SEL entry | Use when one noisy sensor floods the SEL |

**Reading a SEL entry**:
```
ID  | Date       | Time     | Sensor Name       | Event Description         | Direction
----|------------|----------|-------------------|---------------------------|----------
0x01| 04/15/2025 | 14:23:11 | Memory #0x21      | Correctable ECC error     | Asserted
0x02| 04/15/2025 | 14:23:12 | Fan1A             | Lower Critical going high | Asserted
0x03| 04/16/2025 | 09:10:04 | Power Supply #1   | Power Supply AC lost      | Asserted
```
- **Asserted**: condition occurred / started
- **Deasserted**: condition cleared / returned to normal
- If you see `Asserted` without a subsequent `Deasserted`: condition is still active

## Chassis Commands

| Command | What It Does | Notes |
|---------|-------------|-------|
| `chassis status` | Power state, fault/identify/intrusion LED state, drives, cooling | Quick overall health overview |
| `chassis identify 30` | Blink front panel LED for 30 seconds | Use to locate server in datacenter |
| `chassis identify 0` | Turn off identify LED | After locating server |
| `chassis identify` | Blink identify LED indefinitely (until `identify 0`) | |
| `chassis restart_cause` | Why did the server last restart? | Returns: power-on / hard-reset / warm-reset / watchdog / power-interrupt |

# Main Power Fault     : false

## Serial Over LAN (SOL) — Remote Console

| Command | What It Does | Notes |
|---------|-------------|-------|
| `sol activate` | Open serial console to server | Requires SOL configured in BIOS and BMC |
| `sol info` | Show SOL configuration (enabled/disabled, baud rate) | Check before trying to activate |
| `sol set enabled true` | Enable SOL (if not already enabled) | Run on local BMC, not remote |

```bash
# Open SOL console:
ipmitool -I lanplus -H 10.1.0.101 -U admin -f /pass sol activate
# Press Ctrl+] to exit SOL session (returns to ipmitool prompt)
# Type ~ followed by . to force-close if Ctrl+] doesn't work

# SOL shows: POST messages, kernel boot messages, kernel panics, login prompt
# Bandwidth: equivalent to serial console — not suitable for GUI
```

**SOL prerequisites** (configure in BIOS on initial provisioning):
- BIOS → Serial Port: enable BIOS serial redirection, set baud rate (115200 recommended)
- iDRAC/iLO: SOL enabled, baud rate matches BIOS
- OS: configure serial console in grub (`console=ttyS0,115200n8`) and systemd (`systemctl enable serial-getty@ttyS0.service`)

## BMC Management Commands

| Command | What It Does | Notes |
|---------|-------------|-------|
| `mc info` | BMC manufacturer, firmware version, IPMI version | Version confirmation |
| `mc reset cold` | Cold reset the BMC (restart BMC firmware) | Fixes BMC unresponsiveness; server continues running |
| `mc reset warm` | Warm reset the BMC | Lighter reset; use first before cold |
| `mc selftest` | Run BMC self-test | Returns: Self Test Results: passed |
| `user list` | List BMC user accounts | Audit user access |
| `user set password <id> <newpassword>` | Change user password | Credential rotation |
| `user enable <id>` / `user disable <id>` | Enable/disable user account | Access control |
| `lan print 1` | Print network configuration for channel 1 | Verify BMC IP, gateway, VLAN |
| `lan set 1 ipaddr <IP>` | Set BMC IP address | Static IP assignment |

## Dell iDRAC — RACADM Commands

RACADM is Dell's management CLI — runs on the server OS (in-band) or remotely via SSH to iDRAC.

```bash
# Run from OS (in-band):
racadm <command>

# Run remotely (out-of-band SSH to iDRAC):
ssh admin@<idrac-ip>
# Then: racadm <command>

# Run remotely (HTTPS - requires RACADM binary with -r flag):
racadm -r <idrac-ip> -u admin -p <pass> <command>
```

| RACADM Command | What It Does |
|----------------|-------------|
| `racadm getversion` | All component firmware versions (iDRAC, BIOS, NIC, RAID) |
| `racadm getsysinfo` | System information (service tag, model, memory, CPU) |
| `racadm racreset` | Reset iDRAC (equivalent to `ipmitool mc reset cold`) |
| `racadm racresetcfg -f` | Factory reset iDRAC configuration (decommission use) |
| `racadm jobqueue view` | Show pending/running firmware update jobs |
| `racadm update -f <file>` | Apply firmware update from file |
| `racadm rollback idrac` | Rollback iDRAC to previous firmware version |
| `racadm get iDRAC.NIC` | Show iDRAC NIC network configuration |
| `racadm set iDRAC.NIC.IPAddress <IP>` | Set iDRAC IP address |
| `racadm set iDRAC.Users.2.UserName admin` | Configure user account |
| `racadm sslcertdownload -t 1` | Export SSL certificate |
| `racadm serveraction powerstatus` | Equivalent to `chassis power status` |
| `racadm serveraction hardreset` | Equivalent to `chassis power reset` |

## HPE iLO — REST and CLI Commands

```bash
# SSH to iLO:
ssh administrator@<ilo-ip>

# iLO CLI commands:
```

| iLO CLI Command | What It Does |
|----------------|-------------|
| `show /system1` | System overview |
| `show /map1/firmware1` | Firmware versions |
| `reset /map1` | Reset iLO |
| `set /map1/firmware1 upgradeuri=<URL>` | Trigger firmware update from URL |

```bash
# HPE RESTful API (curl):
# Get power state:
curl -sk -u administrator:<pass> \
  https://<ilo-ip>/rest/v1/Systems/1 | python3 -m json.tool | grep PowerState

# Get firmware inventory:
curl -sk -u administrator:<pass> \
  https://<ilo-ip>/rest/v1/systems/1/bios | python3 -m json.tool | grep -E "VersionString|CurrentVersion"

# Trigger iLO reset:
curl -sk -X POST -u administrator:<pass> \
  -H "Content-Type: application/json" \
  -d '{"Action":"Reset"}' \
  https://<ilo-ip>/rest/v1/Managers/1/Actions/Manager.Reset
```

## Common Troubleshooting: When IPMI Doesn't Respond

| Symptom | Likely Cause | Action |
|---------|-------------|--------|
| `Error: Unable to establish LAN session` | Wrong IP, wrong credentials, IPMI LAN disabled | Verify IP; verify VLAN; check if BMC is on management network |
| `Could not open device at /dev/ipmi0` | ipmi_si kernel module not loaded | `modprobe ipmi_si; modprobe ipmi_devintf` |
| Command hangs / timeout | BMC unresponsive — overloaded or crashed | Try `ipmitool mc reset cold`; if local, may need physical BMC reset |
| `Insufficient privilege` | User account lacks sufficient privilege level | Check user privilege level: `ipmitool user list` — need Administrator (4) for power commands |
| SOL session "already active" | Previous SOL session not cleanly closed | `ipmitool sol deactivate` then re-activate |
| Sensor reads `ns` (not specified) | Sensor not present or disabled | Check if component is physically installed; may be a firmware quirk |
| BMC gives wrong sensor readings | BMC firmware bug; BMC cache stale | `ipmitool mc reset cold` to flush BMC state |
| IPMI unreachable after firmware update | BMC rebooting post-update | Wait 2–5 minutes; BMC reboots during firmware apply |
