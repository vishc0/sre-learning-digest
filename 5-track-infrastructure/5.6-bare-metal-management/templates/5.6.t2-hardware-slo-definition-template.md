## Purpose

This document defines hardware reliability service level objectives (SLOs) for each server class in the production environment. These targets govern: spare parts stocking levels, vendor support tier selection, monitoring coverage requirements, and escalation response times.

Hardware SLOs operate below application SLOs. If hardware SLOs are met but the application SLO is breached, the root cause is in the application or platform layer, not the hardware layer. Conversely, hardware SLO breach will propagate to application SLO breach unless application-layer redundancy absorbs the hardware failure.

## Server Class Definitions

Define each distinct server class with its characteristics and business criticality.

| Class ID | Class Name | Representative Models | Primary Workload | Business Criticality | Typical Fleet Size |
|----------|-----------|----------------------|-----------------|--------------------|--------------------|
| CLS-01 | Web / API Tier | Dell R750, Dell R650 | Kubernetes worker nodes — stateless services | High — customer-facing | 50–200 nodes |

## Per-Class Hardware Reliability Targets

### CLS-01: Web / API Tier

| Target | Value | Rationale |
|--------|-------|-----------|
| **Node MTTF target** | >20,000 hours (~2.3 years) between hardware failures per node | Supports 4-year lifecycle with acceptable failure rate |
| **Node MTTR target** | <4 hours from failure detection to node returned to service | Aligns with 4-hour support SLA; cluster absorbs single node failure |
| **Node availability target** | 99.5% per individual node | Cluster availability (99.99%) is achieved through redundancy |
| **Disk Replacement SLA** | Failed disk replaced within 24 hours of alert | RAID rebuild during this window; cluster redundancy covers |
| **Memory (DIMM) Replacement SLA** | DIMM ECC alert → replacement scheduled within 48 hours | Pre-failure replacement before MCE |
| **NIC Replacement SLA** | Failed NIC replaced within 48 hours | Bonding covers; but redundancy must be restored quickly |
| **PSU Replacement SLA** | Failed PSU replaced within 48 hours | Dual PSU covers; restore redundancy |
| **Vendor Support Tier** | 24x7 Next Business Day hardware replacement | Single node failure does not breach application SLA |
| **Spare Disk Stock** | 3% of fleet drive count, minimum 2 drives per drive type | Covers 95th percentile failure demand in 30-day replenishment cycle |
| **Spare DIMM Stock** | 1 DIMM per 100 DIMMs in fleet, minimum 2 per DIMM type | |
| **Monitoring Coverage** | 100% of nodes with IPMI monitoring; 100% with SMART monitoring | No server without hardware telemetry |

### CLS-02: Database Tier

| Target | Value | Rationale |
|--------|-------|-----------|
| **Node MTTF target** | >25,000 hours (~2.8 years) | Higher standard reflects data criticality |
| **Node MTTR target** | <2 hours from failure detection to replica promoted and primary workload restored | Database availability requires faster response |
| **Node availability target** | 99.9% per individual node (application HA handles the rest) | |
| **Disk Replacement SLA** | Failed disk replaced within **4 hours** of alert (not 24 hours) | RAID rebuild on database volumes is extended; second disk failure risk is high |
| **Memory (DIMM) Replacement SLA** | ECC correction rate >10/hour → replace within 12 hours | Database memory corruption is catastrophic |
| **NIC Replacement SLA** | Failed NIC replaced within 24 hours | Database cluster connectivity; bonding covers but restore quickly |
| **PSU Replacement SLA** | Failed PSU replaced within 24 hours | |
| **Vendor Support Tier** | **24x7x4 on-site** (technician on-site with parts within 4 hours) | Data integrity risk justifies premium support |
| **Spare Disk Stock** | 10% of fleet drive count, minimum 4 drives; pre-positioned at datacenter | Database RAID rebuild is slow; spares must be on-site |
| **Spare DIMM Stock** | 2 DIMMs per DIMM type in use for this class; on-site | |
| **Monitoring Coverage** | 100% IPMI; 100% SMART; RAID status polled every 5 minutes | Higher polling frequency for critical tier |
| **Pre-failure Action** | Any SMART "Pending Sector" or "Reallocated Sector" >0: replace within 24 hours (do not wait for failure) | |

---

### CLS-03: GPU Inference

| Target | Value | Rationale |
|--------|-------|-----------|
| **Node MTTF target** | >15,000 hours (~1.7 years) | GPU nodes run at high sustained power; higher failure rate expected |
| **Node MTTR target** | <4 hours; GPU replacement up to 24 hours | GPU replacement requires physical access and careful handling |
| **Node availability target** | 99.0% per node (inference serving has cluster redundancy) | |
| **GPU Replacement SLA** | GPU failure alert → replacement ordered same business day; installed within 48 hours | GPUs are expensive and have lead times; maintain 1 spare per 10-node cluster |
| **Disk Replacement SLA** | NVMe failure → replacement within 12 hours (model storage impact) | |
| **Vendor Support Tier** | 24x7x4 for chassis; 48hr for GPU card replacement | GPU card RMA through vendor |
| **Spare GPU Stock** | 1 spare GPU per 10 active GPU nodes (same model/memory capacity) | GPU lead times are long; order spares at procurement |
| **Spare NVMe Stock** | 2 spare NVMe per drive type in use | |
| **Monitoring Coverage** | 100% IPMI; 100% SMART on NVMe; GPU health via nvidia-smi (temperature, ECC, power consumption) | nvidia-smi reports GPU ECC errors — monitor these same as DIMM ECC |
| **GPU ECC error policy** | Single-bit GPU ECC errors: monitor and trend. Double-bit GPU ECC errors: schedule replacement within 24 hours | |

### CLS-04: Storage / Ceph Tier

| Target | Value | Rationale |
|--------|-------|-----------|
| **Node MTTF target** | >20,000 hours | Storage tier must be highly durable |
| **Node MTTR target** | <8 hours (Ceph rebalances; MTTR is for full node recovery) | Ceph tolerates node loss during rebalancing |
| **OSD Disk Replacement SLA** | Failed OSD disk: replaced within **4 hours** | Ceph begins rebalancing immediately; second OSD failure during rebalance reduces data durability |
| **Node availability target** | 99.5% per node | Ceph handles durability through replication factor |
| **Vendor Support Tier** | 24x7x4 on-site | Drive replacement speed is critical for data durability window |
| **Spare OSD Disk Stock** | 5% of total OSD disk count, minimum 4 drives; **hot spares pre-installed if chassis supports** | Ceph rebalance is I/O intensive; minimizing duration minimizes risk |
| **Journal/WAL NVMe Stock** | 2 spare NVMe drives (same model); these are the "engine" of Ceph OSD performance | |
| **Monitoring Coverage** | 100% IPMI; 100% SMART polled every 15 minutes; Ceph health via `ceph status` integrated into Splunk | |
| **Ceph HEALTH_WARN SLA** | Ceph `HEALTH_WARN` → engineer investigates within 30 minutes | |
| **Ceph HEALTH_ERR SLA** | Ceph `HEALTH_ERR` → P1 incident; engineer responds within 5 minutes | |

---

### CLS-05: Network Functions

| Target | Value | Rationale |
|--------|-------|-----------|
| **Node MTTF target** | >25,000 hours | Network functions must be extremely reliable |
| **Node MTTR target** | <1 hour; workload fails over to redundant NFV node | |
| **Node availability target** | 99.9% per node (NFV has N+1 or active-active redundancy) | |
| **NIC Replacement SLA** | SR-IOV NIC failure → replacement within **4 hours** | NIC failure directly impacts packet processing |
| **Vendor Support Tier** | **24x7x4 on-site** | Network availability requires fastest hardware response |
| **Spare NIC Stock** | 1 spare SR-IOV NIC per 5 active nodes; same model and firmware version | NIC firmware versions must match for consistent DPDK behavior |
| **Monitoring Coverage** | 100% IPMI; NIC error counters polled every 60 seconds; packet drop rate monitoring | NIC silent drop is the insidious failure mode — must monitor end-to-end |

---

### CLS-06: Dev / Test

| Target | Value | Rationale |
|--------|-------|-----------|
| **Node MTTR target** | <24 hours (next business day) | No production impact |
| **Vendor Support Tier** | NBD (Next Business Day) | Cost optimization appropriate for non-production |
| **Monitoring Coverage** | 80% minimum IPMI coverage | Some non-critical servers may not justify full monitoring overhead |
| **Spare Stock** | Draw from production spare pool (do not maintain separate dev spare stock) | |

## Escalation SLA

| Scenario | Detection → P1 Page | P1 Page → Engineer Response | Engineer Response → Action |
|----------|--------------------|-----------------------------|--------------------------|
| RAID degraded (CLS-02, CLS-04) | <5 min (automated alert) | <15 min | <30 min (confirm degradation, order part) |
| RAID failed/offline | <5 min | <5 min (immediate page) | <15 min (data protection actions) |
| GPU double-bit ECC (CLS-03) | <5 min | <30 min | <2 hours (workload impact assessment) |
| Node DOWN (any critical class) | <5 min | <15 min | <30 min (verify redundancy, initiate repair) |
| Temperature critical | <1 min | <15 min | <30 min (thermal investigation) |
| Any CLS-02 node unreachable | <2 min | <10 min | <20 min (database availability is P0) |
