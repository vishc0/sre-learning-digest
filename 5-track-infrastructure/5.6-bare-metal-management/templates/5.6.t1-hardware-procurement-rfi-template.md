## Section 1 — Engagement Context

### 1.1 Purpose of This RFI

This Request for Information (RFI) is issued by ________________ (the "Organization") to gather information from server hardware vendors to support a procurement decision for bare metal server infrastructure. This RFI does not constitute a commitment to procure and does not obligate the Organization to issue a subsequent Request for Quotation (RFQ) or Purchase Order.

Responses to this RFI will be used to:
- Assess vendor capability to meet technical and operational requirements
- Shortlist vendors for a formal RFQ process
- Establish benchmark baselines for performance and TCO comparison
- Understand vendor support infrastructure and parts availability in the Organization's datacenter locations

## Section 8 — Vendor Response Format

Vendors must address each section below. Incomplete responses will be evaluated as such.

### 8.1 Required Technical Response

1. **Proposed configuration**: complete bill of materials with part numbers for each component (CPU, memory, storage, NICs, PSU, chassis)
2. **Benchmark results**: SPECint_rate2017, STREAM, fio (per specified profiles), iperf3 — run on the exact proposed configuration (not a similar configuration). Include test methodology and date.
3. **NUMA topology diagram**: document how NUMA nodes, PCIe root complexes, and memory controllers are arranged for the proposed configuration
4. **PCIe lane accounting**: enumerate all devices and their lane consumption; total vs. available
5. **Memory interleaving confirmation**: confirm all memory channels are populated; interleaving is enabled
6. **Redfish API coverage matrix**: list all Redfish API endpoints available on the management controller

### 8.2 Required Pricing Response

| Item | Unit Price | 3-Year TCO Per Unit | Fleet Volume Price (__ units) | Notes |
|------|-----------|--------------------|-----------------------------|-------|
| Base server hardware | | | | |
| 3-year 24x7x4 support | | | | |
| Estimated power cost (3yr, at $_/kWh) | | | | |
| Total 3-year TCO per unit | | | | |

- Provide pricing for volumes: 10 units, 50 units, 100 units, 250 units
- Identify expected discount from list price at each volume tier
- Identify any applicable government, education, or enterprise discount programs

### 8.3 Required Reference Information

1. Three (3) reference customers in telecommunications, financial services, or government with deployments of 100+ servers of the proposed model. References must agree to be contacted.
2. Describe any deployments of the proposed configuration running Kubernetes at scale (50+ nodes)
3. Describe any known issues or bugs with the proposed configuration; document mitigations

### 8.4 Vendor Viability Information

1. Current-generation product roadmap for this server line (next 3 years)
2. Parts availability commitment for proposed model (years from general availability)
3. Financial stability: publicly traded companies provide last annual report; privately held companies provide revenue range and ownership structure

## Section 9 — Evaluation Criteria and Scoring

| Criterion | Weight | Scoring Basis |
|-----------|--------|--------------|
| Benchmark performance vs. requirements | 30% | Pass/fail on minimums; scored on margin above minimum |
| 3-year TCO (hardware + support + power) | 25% | Lowest TCO = highest score; proportional scoring |
| Support infrastructure and SLA compliance | 20% | Reference validation; depot proximity; SLA contractual commitment |
| Management tooling and API maturity | 15% | Redfish coverage; Ansible module quality; tool integration |
| Vendor viability and references | 10% | Reference quality; financial stability; roadmap clarity |

**Evaluation Process:**
1. RFI response completeness review — incomplete responses not evaluated further
2. Technical scoring (internal engineering team)
3. Reference calls (1 per vendor shortlisted)
4. Benchmark validation (pilot hardware, if advancing to RFQ)
5. TCO model validation (pricing confirmed via procurement team)
