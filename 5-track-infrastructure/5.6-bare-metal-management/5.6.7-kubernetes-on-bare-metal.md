# Chapter 7 — Kubernetes on Bare Metal

## Why Bare Metal K8s: The Real Cases

Running Kubernetes on bare metal is not about avoiding cloud costs as a primary motivation. It is an architectural decision driven by specific technical requirements that cannot be satisfied by managed Kubernetes (EKS, GKE, AKS) or Kubernetes on virtual infrastructure.

The legitimate reasons, in priority order:

1. **GPU workloads at scale**: direct CUDA access without hypervisor mediation. For model training jobs consuming 100% GPU for 72+ hours, the performance difference between bare metal GPU and virtual GPU (vGPU via MIG or time-slicing) is measurable. Multi-GPU topology (NVLink, NVSwitch) requires bare metal.
2. **Extreme I/O throughput**: local NVMe on a bare metal K8s node delivers 4–7 GB/s per NVMe device. Even a fast EBS volume is network-attached with shared bandwidth. For workloads that are I/O bound, local NVMe on bare metal is orders of magnitude faster.
3. **Telco network functions**: as established in Chapter 1, DPDK and kernel-bypass networking require direct hardware access. NFV workloads on bare metal K8s nodes (with SRIOV-aware CNI) deliver line-rate packet processing that no VM can match.
4. **Cost at sustained high utilization**: at T-Mobile's scale, a sustained deployment of 200+ GPU nodes for inference serving crosses the TCO crossover where owned bare metal beats cloud instances on a 3-year horizon.
5. **Regulatory/compliance**: some compliance frameworks require known hardware, no multi-tenancy at the hypervisor layer, and physical control of the machine. Bare metal satisfies these requirements; shared cloud infrastructure may not.

## Networking Challenges: The Cloud Safety Net Is Gone

The biggest operational shock when moving from EKS to bare metal K8s is networking. EKS gives you: AWS Load Balancer Controller for LoadBalancer services, VPC CNI for pod networking, Route 53 for DNS, security groups for pod-level firewalling. All of that must be replaced.

### Load Balancer Services Without a Cloud Controller

On bare metal, `kubectl expose type=LoadBalancer` results in a service that stays in `<pending>` state forever — there is no cloud controller to provision a load balancer.

**MetalLB**: the most common solution. Runs as a DaemonSet, monitors for LoadBalancer services, and either:
- **Layer 2 mode**: announces the service IP via ARP (IPv4) or NDP (IPv6). Simple, works on any network, but single-node bottleneck — the node announcing the ARP handles all traffic until failover.
- **BGP mode**: announces service IPs via BGP to upstream routers. Enables true ECMP load balancing across nodes. Requires BGP-capable network infrastructure (switch/router). Preferred for production at scale.

**kube-vip**: alternative to MetalLB, particularly good for control plane VIP (virtual IP for the API server) and service load balancing. Uses IPVS or ARP/BGP. Simpler architecture than MetalLB for control plane HA.

### CNI Selection for Bare Metal

| CNI | Mode | Performance | eBPF Support | Key Feature | Bare Metal Suitability |
|-----|------|-------------|-------------|-------------|----------------------|
| **Calico** | eBPF or iptables | eBPF mode: very high | Yes (eBPF mode) | Network policy, BGP peering | Excellent — mature, battle-tested |
| **Cilium** | eBPF native | Highest | Yes (native) | Full eBPF stack, Hubble observability, service mesh features | Excellent — best performance and observability |
| **Flannel** | VXLAN or host-gw | Moderate | No | Simple, low overhead | Adequate for small/dev clusters |
| **Weave** | Encrypted mesh | Moderate | No | Encryption built-in | Limited — not recommended for high-throughput |

**For T-Mobile bare metal K8s**: Cilium in eBPF native mode is the current best choice. Reasons: eliminates iptables (which does not scale beyond ~10,000 rules), provides built-in network observability via Hubble (flows visible in Splunk), supports BGP via GoBGP for on-prem routing, and has SR-IOV acceleration for network-function workloads.

### BGP Integration for On-Premises Routing

On-premises bare metal clusters need to integrate with your datacenter routing fabric. The pattern:
- Each K8s node speaks BGP to the top-of-rack switch
- Node announces pod CIDRs assigned to it
- Switch learns routes to pods via BGP, distributes to datacenter routing fabric
- Service IPs (via MetalLB BGP or Cilium BGP) are also announced via BGP

This eliminates overlay networking (VXLAN tunnels) — packets route directly without encapsulation. For telco NFV workloads where every microsecond matters, eliminating overlay overhead is meaningful.

## Storage: CSI Drivers and Options

Cloud-managed K8s provides storage classes backed by cloud block storage (EBS, GCP PD, Azure Disk). On bare metal, you provision your own storage.

| Storage Solution | Architecture | Performance | Complexity | Best For |
|-----------------|-------------|-------------|------------|----------|
| **Rook/Ceph** | Distributed object/block/file storage on bare metal disks | High (with NVMe) | High | Large clusters, multiple storage types needed |
| **Longhorn** | Replicated block storage, uses node local disks | Moderate | Low | Smaller clusters, simpler operations, edge |
| **OpenEBS** (Mayastor) | NVMe-oF over RDMA/TCP, local volumes | Very high | Moderate | High-performance workloads, NVMe fleets |
| **Local Path Provisioner** | Simple local disk, no replication | Highest | Very low | Performance-critical, stateless-safe, or application-level replication |
| **NFS** | Network file storage via NFS server | Moderate | Low | Shared reads, legacy workloads |

**Rook/Ceph details**: Ceph provides a complete storage platform — object (S3-compatible), block (RBD for K8s PVs), and file (CephFS for ReadWriteMany volumes). Rook is the K8s operator that manages Ceph deployment. The requirements: at least 3 dedicated storage nodes, HBA pass-through (not hardware RAID — Ceph must see raw disks), and at minimum one NVMe SSD per Ceph OSD for the journal/WAL.

The operational complexity is real: Ceph requires expertise to tune and operate. A misconfigured Ceph cluster will silently degrade, recover slowly from disk failures, and produce subtle I/O performance issues that are difficult to diagnose. Do not deploy Ceph without dedicated expertise or training.

**Local Path Provisioner + application replication**: for stateless pods with local data (Kafka brokers, Cassandra nodes, Elasticsearch nodes — all of which replicate at the application layer), local NVMe via local path provisioner gives maximum performance with minimum complexity. The storage is non-replicated at the infrastructure layer — the application handles durability.

## Node Management: Provisioning and Lifecycle

### cluster-api with Metal3 and Ironic

**Cluster API (CAPI)**: a Kubernetes extension that manages Kubernetes cluster lifecycle via Kubernetes-style declarative objects. You describe a cluster (control plane, worker nodes), and the provider reconciles to that state.

**CAPM3 (Cluster API Provider Metal3)**: the bare metal provider for CAPI. Uses **Ironic** (OpenStack bare metal provisioning service) to manage physical servers.

```
CAPI management cluster
  └── CAPM3 provider
       └── Metal3 BareMetalHost CRD → Ironic
            └── Ironic → IPMI/Redfish → physical servers
                 └── PXE boot → OS install → K8s join
```

This creates a fully declarative bare metal K8s provisioning pipeline. A BareMetalHost object represents a physical server; when a Machine object references it, Ironic provisions the server and joins it to the cluster. This is Level 3 provisioning automation (Chapter 4) applied specifically to K8s.

### Talos OS: The Immutable Node OS

Talos is an operating system designed exclusively for Kubernetes nodes. It is:
- **Immutable**: the OS is read-only. No SSH. No package manager. No shell. All configuration via API.
- **Minimal**: ~100MB OS image. Attack surface is drastically reduced.
- **Kubernetes-native**: designed from the ground up for K8s; kernel tuning, network configuration, and node management are all Kubernetes-aware.
- **API-driven**: all operations via `talosctl` — configuration, upgrades, reset, console access.

For T-Mobile bare metal K8s: Talos is the emerging preferred OS for K8s nodes because it eliminates the OS management burden (no OS patches — OS upgrades are atomic image swaps), reduces security surface area dramatically, and provides consistent node state.

### Node Maintenance Workflow

Regardless of OS, the standard maintenance pattern for a K8s node:
```bash
# 1. Cordon: prevent new pods from scheduling to this node
kubectl cordon <node-name>

# 2. Drain: evict all running pods (waits for eviction to complete)
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data

# 3. Verify: confirm node has no non-DaemonSet pods
kubectl get pods --all-namespaces --field-selector spec.nodeName=<node-name>

# 4. Perform maintenance (firmware update, hardware replacement, OS upgrade)

# 5. Uncordon: allow scheduling to resume
kubectl uncordon <node-name>
```

**Drain failure modes**: pods with `PodDisruptionBudget` constraints that would be violated by draining will block the drain. You must either adjust the PDB, wait for replicas to redistribute, or force evict (risky — use only when PDB constraints are understood).

## Differences from EKS: What You Own Now

| Concern | EKS | Bare Metal K8s |
|---------|-----|----------------|
| Control plane | AWS managed | Your team manages (etcd, API server, scheduler, controller manager) |
| etcd | AWS managed | Your team: backups, compaction, disaster recovery |
| Certificate rotation | AWS managed | Your team: kubelets, etcd, API server, controller-manager certificates (kubeadm provides tooling) |
| Upgrades | AWS orchestrates | Your team: control plane upgrade → node upgrade, sequence matters |
| Load balancer | AWS ALB/NLB auto-provisioned | MetalLB or kube-vip, manually configured |
| Storage | EBS provisioned by CSI | Rook/Ceph or local provisioner, manually configured |
| Node lifecycle | AWS manages node replacement | Your team via CAPI/Metal3 or manual reprovisioning |
| Networking | VPC CNI | Calico or Cilium, manually configured |

The operational delta is significant. Bare metal K8s requires 2–3 dedicated platform engineers with deep K8s internals knowledge who own the control plane. EKS allows those same engineers to focus on workload concerns. Only choose bare metal K8s when the workload requirements (Chapter 1) genuinely justify the control plane ownership overhead.

## Director Frame: When EKS Wins and When It Doesn't

The interview answer on bare metal K8s vs EKS:

> "EKS is the default for workloads without specific hardware requirements. It gives us managed control planes, simplified upgrades, native AWS integrations, and faster time to value. We choose bare metal K8s for three workload categories: GPU training clusters where direct CUDA access and NVLink topology matter, network function deployments requiring kernel bypass networking, and high-throughput I/O workloads where local NVMe performance is a hard requirement. The cost of bare metal K8s is control plane ownership — etcd, certificate management, upgrade orchestration. We staff that deliberately. We don't default to bare metal K8s as a cost-saving measure — the operational overhead negates the savings unless the workload volume justifies it."
