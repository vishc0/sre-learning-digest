# Chapter 2 — Block Storage and SAN: When to Buy Expensive Iron

Block storage is the workhorse of enterprise databases and virtualization platforms. SAN is its most capable and most expensive form. This chapter covers the protocols, the vendors, and the decision logic for when SAN is justified versus when it is the expensive habit of a team that hasn't reassessed in ten years.

---

## Block Storage Protocols

### iSCSI — IP-Based Block Storage
iSCSI encapsulates SCSI commands inside TCP/IP packets. Standard Ethernet. No dedicated SAN hardware required. A software initiator on the host connects to a storage target (the array) over the existing network.

**How it works**: Host runs iSCSI initiator daemon (open-iscsi on Linux). Array presents targets. Host discovers targets via iSCSI Discovery (SendTargets). Host authenticates (CHAP). Host sees a block device — formats it like local disk. Multipath I/O connects through multiple network paths for redundancy.

**Latency**: Typically 200–500 µs round-trip on a well-tuned network. Jumbo frames (MTU 9000) reduce per-packet overhead and improve throughput significantly — always configure on iSCSI VLANs.

**Throughput**: 10GbE iSCSI: ~1.1 GB/s practical per link. 25GbE: ~2.8 GB/s. LACP bonding for aggregate. Still TCP overhead — not zero-copy.

**Use iSCSI when**: Budget does not support FC infrastructure, latency requirement is >200 µs, team does not have FC skills, workload is moderate-IOPS (SQL Server standard edition, general-purpose VMs, backup targets). Most enterprise databases under 100K IOPS are fine on iSCSI.

**What breaks iSCSI**: Network congestion (TCP drops → performance cliff), misconfigured flow control, missing jumbo frames, routing iSCSI traffic through a switch with CRC errors. iSCSI lives and dies by network quality.

---

### Fibre Channel — Dedicated SAN Fabric
FC is a dedicated, lossless network purpose-built for block storage. Not IP. Host Bus Adapters (HBAs) on the server, FC switches in the fabric, FC ports on the storage array. Nothing else runs on this network.

**How it works**: HBAs connect to FC switches via fiber or copper (SFP+ transceivers). FC switches are zoned — each HBA WWN is zoned to specific storage array ports. The array presents LUNs to specific initiator WWNs via LUN masking. Host sees block devices. DM-Multipath manages multiple FC paths.

**Zoning**: Hard zoning (by physical port — most secure) vs soft zoning (by WWN — more flexible). Single-initiator-single-target zoning is best practice — each HBA zoned only to specific array ports, not the whole fabric.

**LUN masking**: Array-side access control. Even if zoning allows visibility, LUN masking controls which host can access which LUN. Defense in depth.

**Latency**: 50–150 µs end-to-end. FC is lossless (credit-based flow control, no TCP retransmit overhead). Deterministic latency under load — unlike Ethernet.

**Throughput**: 16Gb FC per port (~1.6 GB/s), 32Gb FC per port (~3.2 GB/s), 64Gb FC emerging. Trunk multiple ports per HBA for aggregate.

**Use FC when**: Workload requires deterministic sub-200 µs latency (Oracle databases, financial transaction systems, SAP HANA), team has FC expertise, infrastructure budget can support FC switches and HBAs, VMware environment requires shared block storage for vMotion (though vSAN is displacing this).

---

### NVMe-oF — NVMe Over Fabrics
NVMe-oF extends the NVMe protocol across a network — Fibre Channel (FC-NVMe), RDMA over Converged Ethernet (RoCE), or TCP (NVMe/TCP). Eliminates the SCSI layer entirely. NVMe was designed for flash; NVMe-oF brings that to networked storage.

**FC-NVMe**: Uses existing FC infrastructure, replaces SCSI-FCP with NVMe protocol on the fabric. Sub-100 µs latency. Same fabric investment, better protocol.

**RoCE (RDMA over Converged Ethernet)**: Requires lossless Ethernet (PFC + DCB/ETS configured). Delivers RDMA performance over Ethernet — very low latency, very high throughput, no CPU overhead for I/O processing (kernel bypass). Infrastructure complexity: lossless Ethernet is non-trivial to operate.

**NVMe/TCP**: NVMe protocol over standard TCP. Easier operationally than RoCE — no lossless Ethernet requirement. Slightly higher latency than RoCE but lower complexity. Where most new deployments are heading.

**IOPS**: Millions of IOPS per array with NVMe-oF. The protocol is no longer the bottleneck — flash media is.

**Use NVMe-oF when**: You're specifying new infrastructure and latency is critical. FC-NVMe is the upgrade path if you already have FC. NVMe/TCP is the path forward if you want to deprecate FC infrastructure over time.

## Enterprise SAN Vendors

### NetApp ONTAP (AFF — All Flash FAS)
The most mature enterprise NAS/SAN platform. ONTAP unified storage runs both block (iSCSI, FC, NVMe-oF) and file (NFS, SMB) from the same system.

**Strengths**: Extremely rich data services — SnapMirror (block-level replication), FlexClone (instant clones for testing), efficiency (inline dedup, compression), ONTAP S3 (on-prem object from the same array), mature SAN/NAS integration.  
**Weaknesses**: Complex — tuning ONTAP is a specialist skill. Licensing adds up. Management UI is functional but not beautiful. Not the fastest.  
**Price tier**: $$$–$$$$ (per-TB cost plus software licensing).  
**When NetApp wins**: Organizations that need unified storage (SAN + NAS from one platform), complex data management workflows, SnapMirror-based DR.

---

### Pure Storage FlashArray (//X, //C series) and FlashBlade
FlashArray targets block storage. FlashBlade targets file and object at scale.

**FlashArray strengths**: Consistently sub-1 ms latency (Pure guarantees it in contracts), simple UI (Purity OS is one of the best in the industry), always-on dedup/compression, NVMe-native, non-disruptive upgrades (hardware and software), Evergreen (controller upgrades without data migration).  
**FlashArray weaknesses**: Block only — no NAS from FlashArray (FlashBlade for that, separate system). Higher cost than some alternatives.  
**Price tier**: $$$$ but TCO argument: fewer staff hours, less tuning, Evergreen removes forklift upgrades.  
**When Pure wins**: You want simplicity over features, you need predictable latency with a contract behind it, you are moving to NVMe-first architecture, you want minimal storage administration overhead.

---

### Dell EMC PowerStore / PowerMax
**PowerStore**: Mid-range unified storage. Block and file. NVMe-native storage shelf. AppsON (runs VMs directly on the storage controller). Good performance at the price point.  
**PowerMax**: High-end block storage. Mainframe-connected. Six nines availability (99.9999%). SRDF for synchronous replication. The choice for financial institutions and telcos with non-negotiable uptime.  
**Price tier**: $$$ (PowerStore) to $$$$ (PowerMax).  
**When Dell EMC wins**: Existing Dell relationship (price leverage), mainframe environments (PowerMax), mixed workloads that benefit from a broad product family, cost negotiation leverage due to scale.

## Director Frame

SAN is the most expensive per-GB option in your portfolio. Justify it with workload requirements — not habit, not legacy procurement patterns, and not because it's what the team knows how to operate.

The questions to ask before approving SAN spend:
1. What is the IOPS requirement? Can iSCSI on 25GbE meet it?
2. What is the latency requirement? Will NAS or iSCSI get you there?
3. What is the data access pattern? Block semantics required, or would object storage work?
4. What is the total lifecycle cost — hardware, software licensing, FC switch maintenance, HBA replacement, staff hours?

**Pure FlashArray vs NetApp ONTAP**: Pure wins on simplicity and latency guarantees. NetApp wins on data services depth and unified platform. If you need SnapMirror-class replication semantics and unified NAS/SAN management, ONTAP is the answer. If you want lowest-latency block with minimal operational overhead, Pure is the answer.

At T-Mobile scale: mission-critical network functions (BSS/OSS databases, billing, lawful intercept) are legitimate SAN use cases. Developer environments, internal tooling, log storage — are not. The Director's job is to hold the line on tier routing even when teams default to "we need SAN" because that's what they're used to.
