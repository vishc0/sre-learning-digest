# Chapter 5 — Storage for Kubernetes: The Hardest Part of Stateful Operations

Storage is where most Kubernetes stateful workload failures originate. The abstractions look clean on paper — PVs, PVCs, StorageClasses — but the operational reality involves AZ topology constraints, CSI driver bugs, volume binding races, and data that is not replicated the way the team assumed it was. This chapter covers the architecture decisions and their operational consequences.

---

## The Kubernetes Storage Model

### PersistentVolumes and PersistentVolumeClaims

**PersistentVolume (PV)**: A piece of storage in the cluster. Cluster-scoped (not namespaced). Can be provisioned statically (admin creates it manually) or dynamically (StorageClass creates it on demand). A PV has a storage class, access mode, reclaim policy, and capacity.

**PersistentVolumeClaim (PVC)**: A request for storage from a pod. Namespace-scoped. States: how much storage, which access mode, which StorageClass. The cluster binds a PVC to a matching PV — one-to-one binding. Once bound, the PV is reserved for that PVC. The pod references the PVC, not the PV directly.

**StorageClass**: The template for dynamic provisioning. Specifies which CSI driver to use, what parameters to pass (disk type, IOPS, replication), and the reclaim policy. `kubectl get storageclass` — know what's available before designing stateful workloads.

**Reclaim policies**:
- `Delete`: PV and its backing storage are deleted when PVC is deleted. Default for dynamically provisioned volumes. Dangerous if a PVC is accidentally deleted.
- `Retain`: PV persists after PVC deletion. Requires manual cleanup. Use for data you cannot afford to lose accidentally.
- `Recycle`: Deprecated. Do not use.

**Binding lifecycle**: PVC created → finds matching PV (or dynamic provisioner creates one) → Bound → Pod mounts the PVC → Pod deleted → PVC stays (data persists) → PVC deleted → reclaim policy determines what happens to PV and backing storage.

---

### Access Modes — Critical to Get Right

| Mode | Abbreviation | Meaning |
|------|-------------|---------|
| ReadWriteOnce | RWO | Single node can mount read-write. Multiple pods on SAME node can use it. |
| ReadOnlyMany | ROX | Multiple nodes can mount read-only. |
| ReadWriteMany | RWX | Multiple nodes can mount read-write simultaneously. |
| ReadWriteOncePod | RWOP | Single pod only (K8s 1.22+). Stricter than RWO. |

**Why this matters operationally**: EBS (AWS block) only supports RWO. If you deploy a workload that needs RWX on EBS, it will fail when the pod reschedules to a different node. EFS (AWS NFS) supports RWX. Know the access mode of your workload before choosing storage.

## CSI — Container Storage Interface

CSI is the standardized API between Kubernetes and storage systems. Before CSI, storage drivers were compiled into the Kubernetes binary (in-tree plugins). Any storage vendor that wanted Kubernetes support had to contribute code upstream and wait for release cycles. CSI moved storage drivers out of the Kubernetes binary into separate deployments — vendors ship their own CSI driver, independently of the Kubernetes release cycle.

**CSI driver architecture**: Controller plugin (manages lifecycle: create/delete/attach/detach volumes) + Node plugin (manages local operations: mount/unmount on each node). Both deployed as DaemonSets or Deployments in the cluster.

**Evaluating a CSI driver**:
- CSI spec version supported (1.5+ is current, older versions miss features)
- GA vs alpha/beta — alpha features break between releases
- Snapshot support (VolumeSnapshot API — needed for backup automation)
- Volume expansion support (can you resize a live PV?)
- Topology awareness (does it respect AZ constraints to prevent cross-AZ volume attachment?)
- Release cadence and open GitHub issues — how quickly are bugs fixed?
- Operator experience: what breaks during node drain, upgrades, zone failures?

## Storage Options for Kubernetes Workloads

### Local Path Provisioner
Volumes backed by local disk on the Kubernetes node itself. Fastest possible storage — no network round-trip. No replication.

**Implementation**: Rancher local-path-provisioner, or `hostPath` PVs (not recommended for production — no cleanup on PVC delete). Creates a directory on the node's filesystem.  
**Access mode**: RWO only.  
**Performance**: NVMe local disk performance. 500K–1M+ IOPS if the node has NVMe.  
**Resilience**: None. Node failure = data loss. Pod reschedule to different node = data unreachable until pod returns to original node.  
**When to use**: Ephemeral scratch space, stateful applications that replicate at the application layer (etcd, Cassandra, Kafka, ZooKeeper), ML training scratch (GPU compute nodes).

---

### AWS EBS — Elastic Block Store
Default block storage for EKS. GP3 is the current standard volume type.

**CSI driver**: `aws-ebs-csi-driver` — install as EKS add-on.  
**Access mode**: RWO only. One node at a time.  
**AZ constraint**: EBS volumes are AZ-scoped. The pod must run in the same AZ as the volume. The CSI driver uses topology constraints to enforce this. If you have a 3-AZ cluster and a single EBS PV in us-east-1a, that pod can only run in us-east-1a nodes.  
**Volume types**:
- gp3: 3000 IOPS baseline, 125 MB/s, scale to 16000 IOPS/1000 MB/s independently. Default choice.
- io2: up to 64000 IOPS, sub-millisecond latency, more expensive. For database workloads with IOPS requirements.
- sc1/st1: cheap, throughput-optimized, low IOPS. For cold data, sequential reads.  
**When to use**: Most EKS stateful workloads — databases, message queues, app state. The simplest operational choice for single-pod stateful workloads on EKS.

---

### AWS EFS — Elastic File System
Managed NFS on AWS. Scales automatically. Shared across AZs.

**CSI driver**: `aws-efs-csi-driver` — install as EKS add-on.  
**Access mode**: RWX. Multiple pods on multiple nodes in multiple AZs can mount simultaneously.  
**Performance modes**: General Purpose (default, lower latency) vs Max I/O (higher throughput, higher latency).  
**Throughput modes**: Bursting (throughput scales with storage used), Provisioned (fixed throughput regardless of size), Elastic (auto-scales, new default).  
**Cost**: $0.30/GB/month — significantly more expensive than EBS on a per-GB basis, but you are not pre-provisioning.  
**When to use**: Shared configuration files, ML training checkpoints that multiple pods need to write, ReadWriteMany requirements in EKS, applications that need a shared scratch space across pod instances.  
**When NOT to use**: High-IOPS databases (EFS latency is NFS latency — not for sub-millisecond requirements).

---

### AWS S3 via Mountpoint for Amazon S3
Mounts an S3 bucket as a filesystem on Linux using FUSE. Read-optimized — sequential reads work well, random writes are not supported (no overwrite, no append).

**CSI driver**: `mountpoint-s3-csi-driver`.  
**Access mode**: ROX or RWO (with limitations). No true RWX write semantics.  
**Use case**: ML inference pods reading model weights from S3, batch processing reading input data, large static dataset access.  
**Limitations**: Not POSIX compliant. No atomic rename. No directory listing consistency guarantees. Not for databases. Not for workloads that write frequently.

---

### Ceph / Rook-Ceph
Self-hosted distributed storage deployed on Kubernetes using the Rook operator. Provides block (RBD — RADOS Block Device), file (CephFS), and object (RADOS GW, S3-compatible) from one cluster.

**Access modes**: RBD = RWO. CephFS = RWX.  
**Deployment**: Rook operator manages Ceph cluster lifecycle. OSDs (Object Storage Daemons) on each storage node. MONs (monitors) for cluster state. MGR for management.  
**Performance**: Comparable to enterprise SAN when running on NVMe with proper network (10GbE minimum, 25GbE recommended). Replication is synchronous across OSDs.  
**Operational complexity**: High. Ceph tuning (CRUSH maps, OSD placement, pool replication factor, BlueStore configuration) requires expertise. Upgrades must be managed carefully. OSD failures require monitoring and prompt replacement.  
**When to use**: On-premises Kubernetes clusters where managed storage is not available (bare metal, private cloud, edge), organizations with existing Ceph expertise, situations where you need block + file + object from a single system.  
**When NOT to use**: EKS where EBS/EFS are managed and operationally simpler. The operational cost of running Ceph is 2–3x the cost of using managed storage.

---

### Longhorn
Distributed block storage for Kubernetes. Simpler than Ceph. Replicated volumes across nodes.

**Access mode**: RWO (RWX in newer versions via NFS-based sharing).  
**Replication**: Volume replicas spread across nodes (default 3). Synchronous replication on write.  
**Snapshots and backups**: Built-in. Backup to S3-compatible object storage.  
**Operational complexity**: Lower than Ceph. Longhorn UI is clear. Upgrades are smoother.  
**When to use**: Edge clusters, smaller clusters (5–20 nodes) where Ceph is overkill, clusters that need replicated block storage without managed cloud alternatives.  
**When NOT to use**: Very high IOPS workloads (replication overhead at high IOPS is measurable), clusters large enough to justify Ceph's features.

## Stateful Workload Patterns

### Databases on Kubernetes
**Recommendation**: Local NVMe with application-level replication, not shared storage.  
- PostgreSQL with Patroni: each pod has its own local PV (RWO, local NVMe). Patroni manages streaming replication and failover. No shared storage dependency.
- MySQL/MariaDB with Group Replication or Galera: same pattern.
- Shared SAN/NAS for databases means storage is a shared failure domain. Avoid unless the database explicitly requires shared block (Oracle RAC).

### Message Queues
**RabbitMQ on Kubernetes**: Needs a PVC per pod for message persistence. RWO block storage (EBS on EKS). RabbitMQ clustering is application-layer replication — each node has its own queue data.  
**Kafka**: Same pattern. Each broker needs its own persistent block volume. Log directories per partition. High throughput — gp3 or io2 EBS, or local NVMe.

### ML Training
- **Training data**: Read from S3 (object storage) or EFS (ReadWriteMany). Do not put training data on block volumes.
- **Model checkpoints**: Write to EFS (RWX — multiple GPUs writing checkpoints) or S3 (via object store SDK).
- **Scratch space**: Local NVMe via local path provisioner. Ephemeral — lost on pod termination, which is acceptable for scratch.

## Director Frame

Storage is the hardest part of Kubernetes operations — not networking, not ingress, not autoscaling. Most stateful workload failures in Kubernetes trace back to storage: volumes that won't provision, pods stuck in Pending because the PV is in the wrong AZ, data loss from accidental PVC deletion, Ceph OSD failures that caused degraded volumes.

The Director questions for Kubernetes storage:

1. **Which workloads are stateful?** Do you have a complete inventory? Stateful workloads that are not in the inventory are risks not in the risk register.

2. **What is the reclaim policy for every production PVC?** If it is Delete, who prevents accidental PVC deletion? If it is Retain, who owns the cleanup process?

3. **Managed vs self-hosted**: EBS/EFS on EKS costs more per-GB but has zero Ceph operational burden. Ceph/Rook on bare metal is 50% cheaper per-GB and 3x more expensive in engineer-hours. Model the actual TCO for your team's size and expertise.

4. **AZ topology**: Does your team understand that EBS is AZ-scoped? Have you had a pod scheduling issue because the PV is in the wrong AZ? If not, you will.

5. **Backup strategy for PVs**: Are EBS volumes snapshotted? Is Velero or Kasten configured for backup? If you delete a namespace, what is the recovery procedure?

At T-Mobile: EKS stateful workloads should default to EBS (RWO databases, queues) and EFS (shared config, ML checkpoints). Ceph/Rook is justified for on-prem bare metal clusters where managed storage is not available. Local NVMe for high-performance databases with application replication (Cassandra, Kafka) is the right pattern over shared storage at scale.
