# Chapter 4 — Object Storage: The Cost-Efficient Tier

Object storage is not a compromise — it is the right architecture for a large class of workloads that most organizations are still serving from NAS or block storage at 5–20x the per-GB cost. Understanding object storage deeply is a cost management capability, not just a technical one.

---

## What Makes Object Storage Different

Object storage violates every assumption from the POSIX filesystem world.

**Flat namespace**: No directories. Buckets contain objects identified by a string key. The path-like appearance of `logs/2026/07/06/app.log` is a key with slashes in it — there is no directory called `logs`. List operations return keys matching a prefix.

**HTTP API, not POSIX**: GET, PUT, DELETE, HEAD. No append, no seek, no rename (rename is a copy + delete). No file locking. No O_CREAT | O_EXCL semantics. Applications must be designed for this — you cannot NFS-mount S3 and expect database semantics.

**Eventual consistency (historically)**: Write-then-read is not guaranteed to return the new value on every node immediately in classical object store designs. AWS S3 became strongly consistent in December 2020 for new object writes and deletes — but many self-hosted alternatives (older Ceph, older MinIO) are still eventually consistent per-key. Always verify the consistency model of your object store before designing around it.

**Metadata per object**: Every object carries arbitrary key-value metadata alongside the payload. System metadata (content-type, ETag, size, last-modified) plus up to 2KB user-defined metadata. This enables rich querying and lifecycle management without a separate database.

**Immutability support**: S3 Object Lock (WORM — Write Once Read Many) prevents deletion or overwrite for a defined retention period. Required for compliance retention (FINRA, telecom regulatory). Versioning keeps prior versions of every object when overwrites happen.

## The S3-Compatible Ecosystem

AWS S3 set the API standard. The rest of the ecosystem follows it.

### AWS S3
Reference implementation. 12 storage classes from S3 Standard to S3 Glacier Deep Archive. Strongly consistent. 11-nines durability. Multi-AZ erasure coding. IAM-integrated. Event notifications (Lambda triggers). Lifecycle policies. Cross-region replication. Presigned URLs for temporary access delegation.

### MinIO
Self-hosted, S3-compatible, Kubernetes-native. Distributed MinIO runs across multiple nodes with erasure coding. Single-binary deployment. Commercially supported (MinIO AGPL + commercial license). Excellent choice for on-prem object storage when you need S3 API compatibility without AWS dependency.

**Erasure coding**: MinIO uses erasure coding across storage drives. EC:4 means 4 data shards + 4 parity shards — can tolerate loss of 4 drives/nodes without data loss.  
**Performance**: MinIO consistently benchmarks at multi-hundred GB/s read/write throughput in distributed configurations.  
**T-Mobile relevance**: On-prem Splunk SmartStore cold tier, build artifact storage, test data storage without cloud egress costs.

### Ceph RADOS Gateway (RGW)
Ceph is a distributed storage system that provides block (RBD), file (CephFS), and object (RADOS GW) interfaces from a single cluster. RADOS Gateway presents S3-compatible and Swift APIs on top of the RADOS object store.

**Advantage**: One cluster, three protocols — reduces operational sprawl.  
**Disadvantage**: Operationally complex. Ceph tuning is a specialist skill. CRUSH map, OSD configuration, pool settings — all require expertise.  
**Best for**: Organizations already running Ceph for Kubernetes block storage (Rook-Ceph) who want to add object storage from the same cluster.

### Dell ECS (Elastic Cloud Storage)
Enterprise-grade on-prem object storage. S3 and Swift compatible. Designed for exabyte-scale single-namespace deployments. WORM compliance, multi-site geo-replication. Expensive. Used in large telcos for compliant long-term data storage.

## Durability — What 11 Nines Actually Means

S3 Standard's 99.999999999% durability means losing 1 object per 100 billion objects per year on average.

**How they achieve this**: Data is erasure-coded across multiple AZs within a region. An object is sliced into k data fragments and m parity fragments. The system can reconstruct the full object if any m fragments are lost. For S3, this spans at least 3 AZs — losing an entire AZ does not lose data.

**Comparison**:
- RAID-6 on a single SAN: approximately 99.9999% durability — significantly lower.
- NAS HA pair without replication: depends on RAID inside the array, but no off-box protection.
- S3 Standard-IA: same durability as Standard, lower availability (99.9% vs 99.99%).

**Practical implication**: For backup targets and long-term retention, S3-class object storage is more durable than any on-prem RAID configuration. This is the architectural argument for object storage as your backup destination.

## SRE Use Cases for Object Storage

**Log archival — Splunk SmartStore**: Splunk's SmartStore architecture stores hot/warm buckets on fast local storage (NVMe or NAS), cold/frozen buckets on S3-compatible object storage. This is the primary cost lever for Splunk at T-Mobile scale — licensing cost is per-GB ingested, storage cost is per-GB retained. Cold data on S3 Glacier costs $0.004/GB/month vs $0.025/GB/month on S3 Standard vs whatever your NAS costs per GB.

**Build artifacts and container images**: OCI registry (Harbor, ECR) backed by object storage. Build artifacts stored as S3 objects. Immutable — version by key, delete by lifecycle policy.

**Backup target**: Veeam, Commvault, Cohesity, Rubrik — all write to S3-compatible object storage. WORM/Object Lock provides protection against ransomware deletion of backups.

**ML datasets**: Training data (images, text, audio) at petabyte scale lives in S3. ML training frameworks (PyTorch DataLoader, TensorFlow data pipelines) have native S3 support. The training cluster reads data at high throughput in parallel — exactly what S3 is designed for.

**Media storage**: CDN origin, video transcoding input/output, podcast/audio files. Object storage + CDN is the correct architecture for media at scale.

**Compliance/regulatory retention**: Telecom Call Detail Records (CDRs), lawful intercept metadata, financial transaction logs — retention periods measured in years. Object storage with WORM and lifecycle expiration is the right architecture.

## Lifecycle Policies — The Cost Management Tool

Lifecycle policies are how you turn object storage from a cost center into a cost management lever.

**Tiering transition rules** (S3 example):
- S3 Standard → Standard-IA: after 30 days (objects not accessed frequently)
- Standard-IA → S3 Glacier Instant Retrieval: after 90 days
- Glacier Instant → Glacier Flexible: after 180 days
- Glacier Flexible → Glacier Deep Archive: after 365 days

**Cost gradient** (approximate):
| Class | $/GB/month | Retrieval latency | Min storage |
|-------|-----------|------------------|-------------|
| S3 Standard | $0.023 | ms | None |
| S3 Standard-IA | $0.0125 | ms | 30 days |
| S3 Glacier Instant | $0.004 | ms | 90 days |
| S3 Glacier Flexible | $0.0036 | minutes | 90 days |
| S3 Glacier Deep Archive | $0.00099 | hours | 180 days |

A 3-year-old log archive sitting in S3 Standard is a policy gap, not a requirement.

**Expiration rules**: Permanently delete objects after a defined period. Required for any data with a legal retention window — delete after the window closes, do not pay to retain data you are legally required to destroy.

**Versioning interaction**: Lifecycle rules can expire non-current versions — prevents versioning from silently accumulating infinite storage.

## Director Frame

Object storage is the cost-efficient tier — the question is not whether to use it, but what belongs here vs hot storage. Every data lifecycle that ends in "eventually archive or delete" belongs in an object storage tiering strategy.

Director-level levers:
1. **Splunk SmartStore configuration**: moving frozen buckets from NAS to object storage is a concrete budget action at T-Mobile scale.
2. **Lifecycle policy audit**: who owns the lifecycle policies on production S3 buckets? If no one, objects accumulate in Standard tier indefinitely.
3. **Versioning and Object Lock policy**: for backup buckets and compliance data, versioning + Object Lock is non-negotiable. If it is not configured, it is not protected.
4. **On-prem vs cloud object storage decision**: at what data volume does MinIO or Ceph RGW become cheaper than S3 egress + storage costs? Model it by workload.
5. **Backup architecture**: is your backup software writing to object storage? If it is writing to NAS, you are paying 5–10x for durability that object storage provides by default.
