## Tier Definitions

| Tier | Classification | Workload Examples | Storage Types |
|------|---------------|-------------------|--------------|
| Tier 1 | Mission Critical | Core network functions, BSS/OSS databases, lawful intercept, real-time billing | All-flash SAN (FC/NVMe-oF), local NVMe |
| Tier 2 | Business Important | ERP, CRM, internal APIs, observability platforms, Splunk hot/warm | Enterprise NAS (NetApp AFF), EBS io2/gp3 |
| Tier 3 | General Purpose | Internal applications, development/staging, batch processing, non-critical shared storage | Standard NAS, EBS gp3, EFS |
| Tier 4 | Archive / Cold | Log retention, CDR archive, compliance data, cold Splunk, backup | Object storage (S3, MinIO), Glacier |

## Alert Thresholds

### Latency Alerts

| Tier | Warning (p99) | Critical (p99) | Escalation |
|------|-------------|---------------|-----------|
| Tier 1 | > 1 ms | > 2 ms | Page storage team immediately |
| Tier 2 | > 20 ms | > 50 ms | Page on-call, notify app owners |
| Tier 3 | > 100 ms | > 250 ms | Ticket to storage team |
| Tier 4 | N/A (latency not SLO'd) | N/A | |

### IOPS Saturation Alerts

| Tier | Warning | Critical | Action |
|------|--------|---------|--------|
| Tier 1 | > 80% of allocated IOPS for 5 min | > 90% for 5 min | Scale out or throttle lower-priority workloads |
| Tier 2 | > 80% for 15 min | > 90% for 15 min | Notify application owner, plan expansion |
| Tier 3 | > 90% for 30 min | > 95% for 30 min | Review usage, notify owner |

### Availability Alerts

| Tier | Target | Alert on | Escalation |
|------|--------|---------|-----------|
| Tier 1 | 99.999% | Any outage > 30 seconds | Immediate P1 incident |
| Tier 2 | 99.99% | Any outage > 5 minutes | P1 if customer-impacting, else P2 |
| Tier 3 | 99.9% | Any outage > 1 hour | P2 |
| Tier 4 | 99.9% API | API unavailable > 1 hour | P3 |

|-----|-----------------|-----------|---------------|

## SLO Breach Response

| Tier | SLO Breached | Immediate Action | Escalation Path |
|------|-------------|-----------------|----------------|
| Tier 1 | Latency or availability | Page storage team + incident commander | Storage lead → Director within 15 min if not resolved |
| Tier 1 | Durability concern (disk fault) | Page storage team | Replace failed component within 4 hours |
| Tier 2 | Latency | Notify storage team, assess root cause | Escalate if not resolved in 2 hours |
| Tier 2 | Availability > 30 min | Page application owner + storage team | Director if customer-impacting |
| Tier 3 | Any SLO breach | Ticket to storage team | Escalate if blocking production workloads |

## Workload Placement Checklist

Before approving storage provisioning for a new workload, confirm tier assignment:

- [ ] Latency requirement documented (specify p50/p99 target from application owner)
- [ ] IOPS requirement documented (sustained and burst — from application owner or benchmark)
- [ ] Access mode determined (RWO / RWX / ROX — Kubernetes) or (single-host / shared — traditional)
- [ ] Retention period documented (drives tier and lifecycle policy)
- [ ] Compliance requirement assessed (WORM, audit logging, data residency)
- [ ] RPO/RTO requirement stated (drives replication and backup frequency)
- [ ] Tier assignment approved by storage team lead
- [ ] Storage provisioned to the correct tier (not "what's available" but "what's right")
