# Chapter 3 — File Storage and NAS: The Enterprise Workhorse

NAS is the most widely deployed storage type in enterprise IT. It is also the most abused — teams reach for NFS as the default answer when object storage or block storage would be the correct one. This chapter gives you the operational depth and the routing judgment to manage both.

---

## NFS — Network File System

NFS is the standard for shared file storage in Linux/Unix environments. A server exports a directory tree; clients mount it and see a POSIX filesystem. Client writes go over the network to the server.

### NFS v3 vs NFS v4

**NFS v3** (still dominant in enterprise):
- Stateless: server does not track client state. Client crashes, reconnects, continues.
- No mandatory locking (advisory only — lock state not tracked by server). Applications must implement their own locking discipline.
- AUTH_SYS (UID/GID-based) authentication — trivially spoofed in a trusted environment, but no Kerberos requirement.
- UDP or TCP transport. TCP is default and required for anything serious.
- Performance: mature, well-tuned in kernels, predictable behavior.

**NFS v4** (preferred for new deployments):
- Stateful: server tracks client state, open files, delegations. Enables stronger semantics.
- Mandatory locking built into the protocol. No more lock races that v3 leaves to applications.
- Kerberos support (sec=krb5, krb5i, krb5p) — actual authentication, data integrity, and encryption options.
- Single port (TCP 2049) — firewall-friendly vs v3's portmapper/rpcbind chaos.
- Delegations: server grants client the right to cache reads or writes — improves performance for read-heavy workloads.
- NFS v4.1 adds pNFS (parallel NFS) — clients can stripe reads/writes across multiple NFS servers simultaneously, breaking the single-server throughput ceiling.

**Practical guidance**: Run NFS v4 for new deployments when you need Kerberos security or pNFS performance. NFS v3 persists in environments with legacy clients or older NAS appliances. Most NetApp ONTAP deployments run both simultaneously.

### NFS Performance Factors
- Network bandwidth is the ceiling — 10GbE limits you to ~1.2 GB/s per client, 25GbE to ~2.8 GB/s.
- RTT dominates small-I/O performance. NFS over a WAN is painful because each metadata operation is a round trip.
- Async vs sync mount options: `async` (default) — client buffers writes, much faster, data loss risk on server crash. `sync` — every write acknowledged before returning, safe, slower. Production databases should never run on NFS with async mount unless the application handles its own durability.
- rsize/wsize: NFS read/write block size. Default 65536 on modern kernels, increase to 1048576 for high-throughput workloads.

### NFS Reliability
- NAS HA pair: active-passive (one controller serves I/O, secondary takes over on failure) or active-active (both controllers serve different volumes, cross-takeover on failure). Takeover is typically 5–30 seconds depending on vendor and configuration.
- NetApp MetroCluster: synchronous replication between two sites (up to 300 km apart with dark fiber). RPO=0 for site failure. One of the few on-prem solutions that delivers genuine synchronous DR.
- NFS mount after failover: with automatic DNS/IP failover, mounts stay alive; without it, clients must remount to the failover IP.

## SMB/CIFS — Windows File Sharing

SMB (Server Message Block) is the Windows-native file sharing protocol. CIFS is an older dialect; SMB 3.x is what modern Windows servers negotiate.

**SMB 3.0 features**: Encryption (AES-128), multichannel (multiple NICs for parallel throughput), transparent failover (cluster-aware), SMB Direct (RDMA support for low latency).

**DFS (Distributed File System)**: Microsoft's namespace aggregation. DFS namespaces present a single UNC path (\\domain\share) that routes to multiple backend servers. DFS Replication (DFSR) syncs content between servers. Enables scale-out and geographic distribution for Windows file shares.

**In enterprise**: SMB shares are typically served from Windows File Server clusters, NetApp ONTAP (which speaks both NFS and SMB natively), or Dell EMC PowerStore File. Most enterprise environments that aren't all-Windows use NetApp ONTAP as the universal NAS to serve both NFS (Linux) and SMB (Windows) from the same volume.

## Parallel Filesystems — When NAS Is Not Enough

Standard NFS has a single-server throughput ceiling. For HPC, ML training clusters, and big data processing, you need parallelism — multiple storage nodes serving I/O simultaneously.

### GPFS / IBM Spectrum Scale
IBM's enterprise parallel filesystem. Decades old. Proven at petabyte scale. Runs on Linux, AIX, Windows.

**Architecture**: Storage nodes (NSDs — Network Shared Disks) export block devices over InfiniBand or Ethernet. Clients run the GPFS client, which stripes reads and writes across all storage nodes simultaneously.  
**Scale**: Multi-petabyte single filesystem, millions of files, thousands of concurrent clients.  
**Data services**: Policy-based tiering (active to nearline to archive), storage pools, ILM.  
**Cost**: Very expensive. IBM enterprise licensing plus specialist administration.  
**When justified**: Financial time-series, seismic data processing (oil and gas), genomics sequencing, large-scale simulation workloads. Rare in telecom unless running massive ML infrastructure.

### Lustre
Open source parallel filesystem. The dominant choice in supercomputing. Used in top 500 HPC clusters, large-scale ML training environments.

**Architecture**: MDS (Metadata Server) manages the namespace. OSS (Object Storage Servers) store actual file data. Clients stripe across OSTs (Object Storage Targets). Reads/writes bypass the MDS after initial metadata lookup — pure data I/O goes directly to OSTs.  
**Scale**: Exabyte-scale filesystems in large deployments. Hundreds of GB/s aggregate throughput.  
**Cost**: Open source (no license), but hardware and operational complexity are significant.  
**When justified**: ML training scratch space (model checkpoints at 100 GB/s+), genomics pipelines, large-scale scientific computing. AWS FSx for Lustre makes this accessible without building your own.

**For T-Mobile context**: Unless you are running serious ML training infrastructure at petabyte data volumes, parallel filesystems are out of scope. Lustre matters if you're evaluating ML platform infrastructure choices. GPFS is niche.

## NAS Quotas — Operational Necessity

Shared NAS without quota enforcement is an incident waiting to happen. One runaway log writer, one misconfigured application, one developer running an unfiltered database dump — and the entire filesystem fills, taking every other tenant offline.

**Types**:
- User quotas: per-UID. Caps individual user consumption.
- Group quotas: per-GID. Caps a team or application group.
- Volume/qtree quotas (NetApp ONTAP): per-filesystem slice. Caps a specific application's allocation at the volume level — the most operationally reliable because it's enforced at the storage layer, not the OS layer.

**Soft vs hard quotas**:
- Soft: allows temporary overage, generates alert, grace period (typically 7 days), then enforced.
- Hard: blocks writes immediately at the limit.

**Director guidance**: Volume-level hard quotas with 80%/90% alerting is the right policy. No application should have unbounded write access to shared NAS. The alert at 80% is when you engage the application team — not when you hit 95% and are in crisis mode.

## Director Frame

NAS is the workhorse — it will be in your environment regardless of how much you modernize. The failure mode is not NAS itself; it is treating NAS as the answer to every storage question.

Signs you've inherited an over-NAS'd organization:
- Logs are on NFS (they should be in object storage or Splunk SmartStore)
- ML training data is on NFS (should be S3 or parallel filesystem)
- Kubernetes persistent volumes all use NFS (ReadWriteOnce workloads should use block)
- No quotas exist anywhere (someone's going to fill it, someone always does)

The Director agenda for NAS:
1. Audit: what is actually on NAS? What should be on object storage?
2. Policy: enforce quotas and per-application volume allocation
3. Routing: new workloads go through a tier decision, NAS is not the default
4. DR: is your critical NAS replicated? MetroCluster, SnapMirror, or backup? What is the RPO and who owns it?

At T-Mobile: shared NAS for Splunk warm buckets may be a valid short-term design but object storage (SmartStore) is the long-term direction for cost management. Home directories and application configs are legitimate NAS workloads. CDR data on NAS is a routing error.
