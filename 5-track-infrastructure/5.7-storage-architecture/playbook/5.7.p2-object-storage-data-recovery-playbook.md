# Playbook: Object Storage Accidental Delete Recovery

**Trigger**: Objects, prefixes, or buckets deleted unintentionally from S3 or S3-compatible object storage  
**Severity**: P1 if production data or compliance data | P2 if significant data loss | P3 if recoverable from other sources  
**Playbook owner**: Cloud/Storage team + data owner  
**Escalation**: Director immediately if compliance-sensitive data (CDRs, lawful intercept, financial records) is involved

---

## Step 1: Assess What Was Deleted

Before any recovery action — understand the scope. Recovery approach changes dramatically based on what was deleted.

```bash
# What was the deletion? Get specifics from the person who deleted or from audit logs
# CloudTrail query for S3 delete events in the last hour
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=DeleteObject \
  --start-time $(date -d '1 hour ago' -u +%Y-%m-%dT%H:%M:%SZ) \
  --region <region>

# Also check for DeleteObjects (bulk delete) and DeleteBucket
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=DeleteObjects \
  --start-time $(date -d '2 hours ago' -u +%Y-%m-%dT%H:%M:%SZ)
```

**Scope categories**:
| Scope | Recovery complexity | First action |
|-------|-------------------|-------------|
| Single object deleted | Low | Check versioning |
| Multiple objects (prefix/folder) | Medium | Check versioning + batch restore |
| Entire bucket deleted | High | Check replication destination, backup |
| Objects expired by lifecycle policy | Medium–High | Check if backup exists, check replication |
| Versioned objects permanently deleted (delete marker + version) | Very High | Check backup |

## Step 2: Is Versioning Enabled?

This is the most important question. If yes, recovery is likely fast. If no, recovery may be impossible from S3 alone.

```bash
# Check versioning status on the bucket
aws s3api get-bucket-versioning --bucket <bucket-name>
# Response: "Status": "Enabled" = versioning is on
# Response: {} or "Status": "Suspended" = versioning is off or suspended
```

---

## Step 3a: Recovery WITH Versioning Enabled

When versioning is enabled, deleting an object creates a delete marker — the object data still exists as a previous version. Recovery is a matter of removing the delete marker.

**Recovery for a single deleted object**:

```bash
# Step 1: List versions and delete markers for the object
aws s3api list-object-versions \
  --bucket <bucket-name> \
  --prefix <object-key>

# Output shows: Versions[] (actual data) and DeleteMarkers[]
# The most recent DeleteMarker is what was created by the delete
# Note the DeleteMarker's VersionId

# Step 2: Delete the delete marker (this restores the object)
aws s3api delete-object \
  --bucket <bucket-name> \
  --key <object-key> \
  --version-id <delete-marker-version-id>

# Step 3: Verify object is restored
aws s3api head-object --bucket <bucket-name> --key <object-key>
```

**Recovery for a prefix/folder (multiple objects)**:

```bash
# List all delete markers under the affected prefix
aws s3api list-object-versions \
  --bucket <bucket-name> \
  --prefix <folder-prefix/> \
  --query 'DeleteMarkers[*].[Key,VersionId]' \
  --output text > delete_markers.txt

# Review the file — confirm these are the delete markers from the accidental delete
# (check timestamp — should be within the incident window)
head -20 delete_markers.txt

# Batch remove delete markers using a loop (for moderate counts)
while IFS=$'\t' read -r key version_id; do
  echo "Restoring: $key"
  aws s3api delete-object \
    --bucket <bucket-name> \
    --key "$key" \
    --version-id "$version_id"
done < delete_markers.txt

# For large-scale restore (thousands of objects) — use AWS SDK script or S3 Batch Operations
# S3 Batch Operations can perform bulk operations on millions of objects
```

**Recovery from permanent version delete (both data and delete marker deleted)**:

If someone deleted a specific version ID of an object (not just the delete marker), the data is gone from S3 unless:
- Cross-region replication was running before the deletion
- A backup exists
- The object can be reconstructed from source systems

## Step 4: Recovery WITHOUT Versioning — Assess Options

Without versioning, S3 does not retain deleted objects. Work through these options in order:

### Option A: Check S3 Recycle Bin (S3 Object Lock / Recycle Bin)
AWS S3 Recycle Bin is a separate feature from versioning — it retains deleted objects for a configurable period.

```bash
# List items in the Recycle Bin for S3
aws rbin list-resources \
  --resource-type S3_SNAPSHOT \
  --region <region>

# If the bucket is there — restore it
aws rbin restore-resource \
  --identifier <resource-identifier> \
  --region <region>
```

### Option B: Check Cross-Region Replication (CRR) or Same-Region Replication (SRR)

```bash
# Check if replication was configured on the bucket
aws s3api get-bucket-replication --bucket <bucket-name>

# If CRR was configured, check the destination bucket
aws s3 ls s3://<destination-bucket>/<prefix>/ | head -20
aws s3api head-object --bucket <destination-bucket> --key <object-key>
```

Note: S3 replication by default replicates deletes (delete markers are replicated). If the source deleted the object and the delete was replicated, the destination may also have the delete marker. Check destination versioning — the data version may still exist in the destination even if the delete marker was replicated.

### Option C: Check Backup

```bash
# Is there a backup job that wrote this data to a separate destination?
# Common patterns:
# - Veeam/Commvault backup to a separate S3 bucket
# - AWS Backup with S3 backup enabled
# - Application-level backup to a different prefix/bucket

# Check AWS Backup recovery points
aws backup list-recovery-points-by-resource \
  --resource-arn arn:aws:s3:::<bucket-name> \
  --region <region>
```

### Option D: Check if Object Can Be Regenerated
For objects that are derived data (ETL outputs, cached results, generated reports) — can they be regenerated from the source?

Engage the data owner and application team to assess regeneration cost vs recovery cost.

## Step 5: Check if Lifecycle Policy Caused the Deletion

If objects disappeared without an explicit human delete, a lifecycle policy may have expired them.

```bash
# Check lifecycle configuration on the bucket
aws s3api get-bucket-lifecycle-configuration --bucket <bucket-name>

# Look for:
# - Expiration rules (delete after N days)
# - NoncurrentVersionExpiration (deletes old versions after N days)
# - AbortIncompleteMultipartUpload (usually safe)

# Review each rule's filter/prefix and Days value
# Cross-reference with when objects disappeared
```

If a lifecycle rule caused premature expiration:
1. Immediately disable or modify the rule that caused the issue
2. Assess whether any objects are still within the expiration window and can be restored from versioning
3. Document the incorrect rule configuration for post-incident review

```bash
# Disable a specific lifecycle rule (modify the config to remove or adjust the rule)
# Export current config
aws s3api get-bucket-lifecycle-configuration --bucket <bucket-name> > lifecycle.json
# Edit lifecycle.json to remove or fix the problematic rule
aws s3api put-bucket-lifecycle-configuration --bucket <bucket-name> --lifecycle-configuration file://lifecycle.json
```

## Step 6: Data Loss Assessment — If Unrecoverable

If recovery is not possible through any of the above options:

1. **Document exactly what was lost**: object keys, size, age, content description
2. **Assess business impact**: which systems depend on this data? What operations are affected?
3. **Determine RPO**: when was the last known-good state of this data?
4. **Escalate to data owner and Director**: with full impact assessment — do not minimize
5. **Notify compliance team if applicable**: CDRs, financial records, and lawful intercept data have regulatory notification requirements when lost

---

## Step 7: Immediate Remediation — Prevent Recurrence

As soon as recovery actions are complete (or confirmed unrecoverable), take these steps immediately:

```bash
# 1. Enable versioning if not already enabled (do this NOW)
aws s3api put-bucket-versioning \
  --bucket <bucket-name> \
  --versioning-configuration Status=Enabled

# 2. Enable S3 Recycle Bin for the bucket
# (do this via AWS console or CloudFormation — retention period recommendation: 30 days minimum)

# 3. Enable Object Lock on the bucket (for compliance-sensitive buckets)
# Note: Object Lock must be enabled at bucket creation time for most configurations
# For existing buckets, consider creating a new bucket with Object Lock and migrating data
```

---

## Step 8: Recovery Verification

After restoring objects, confirm they are intact:

```bash
# Verify object exists and has expected metadata
aws s3api head-object --bucket <bucket-name> --key <object-key>

# Spot-check content (download and verify checksum if checksum was stored)
aws s3 cp s3://<bucket-name>/<object-key> /tmp/recovered-object
md5sum /tmp/recovered-object
# Compare against stored checksum if available

# For application data — confirm the application can read and process the restored objects
```

## Prevention — Policy Requirements

After any accidental delete incident, the following should be reviewed and enforced:

| Control | Requirement | Action if missing |
|---------|------------|------------------|
| Bucket versioning | Enabled on all production buckets | Enable immediately |
| Object Lock / WORM | Enabled on compliance and backup buckets | Create protected bucket, migrate data |
| S3 Recycle Bin | 30-day retention for production buckets | Enable via AWS console |
| MFA Delete | Required for permanent version deletions on critical buckets | Enable on compliance buckets |
| Bucket policy: deny DeleteBucket | Applied to production buckets | Add SCP/bucket policy |
| CloudTrail S3 data events | Enabled for all production buckets | Enable, ship to SIEM |
| Backup verification cadence | Monthly restore test | Schedule automated restore test |

**Bucket policy to prevent DeleteBucket on production buckets**:

```json
{
  "Statement": [
    {
      "Sid": "DenyDeleteBucket",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:DeleteBucket",
      "Resource": "arn:aws:s3:::<bucket-name>",
      "Condition": {
        "StringNotEquals": {
          "aws:PrincipalArn": "arn:aws:iam::<account>:role/StorageAdminBreakGlass"
        }
      }
    }
  ]
}
```

## Post-Incident Review Agenda

Schedule within 48 hours of recovery or confirmed data loss:

1. What was deleted, when, by whom, how (console, CLI, API, automated job)?
2. Which controls were in place? Which were not?
3. What was the recovery path? How long did it take?
4. What is the RPO impact (if any)?
5. What controls are being added to prevent recurrence?
6. Who needs to be notified (compliance, leadership, customers)?
7. Which other buckets have the same vulnerability? (Audit all production buckets)
