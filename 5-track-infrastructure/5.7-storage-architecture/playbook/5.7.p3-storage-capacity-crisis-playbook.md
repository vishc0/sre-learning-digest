# Playbook: Storage Capacity Crisis Response

**Trigger**: Any storage system at or approaching 80% utilization  
**Severity escalation**: 80% = Warning | 90% = Critical | 95% = Emergency  
**Playbook owner**: Storage/Infrastructure team lead  
**Escalation contact**: Director if customer-impacting or cannot recover below 90% within 2 hours

---

## Alert Thresholds and Initial Response SLA

| Threshold | Alert Level | Response SLA | Who Responds |
|-----------|------------|--------------|--------------|
| 80% | Warning | 4 hours | Storage team engineer on call |
| 90% | Critical | 30 minutes | Storage team + application owner paged |
| 95% | Emergency | Immediate | Storage team lead + Director notified |
| 100% (full) | P1 Incident | Immediate | Incident commander engaged |

## Phase 1: Immediate Triage (First 10 Minutes)

**Step 1: Identify exactly what is full**

```bash
# For Linux/NFS/local filesystems
df -hT | sort -k6 -rn | head -20

# For specific NAS mount — check from multiple angles
df -h /mnt/storage/path
du -sh /mnt/storage/path/* 2>/dev/null | sort -rh | head -20

# For Kubernetes PVs — check PVC usage
kubectl get pvc -A | grep -v Terminating
# Then check actual usage via exec into the pod
kubectl exec -n <namespace> <pod> -- df -h /data

# For object storage (S3 bucket size via CLI)
aws s3 ls s3://bucket-name --recursive --human-readable --summarize | tail -2
```

**Step 2: Identify the rate of fill**

Do not just look at current utilization — look at the growth rate. A system at 82% that has been at 82% for three months is different from one that was at 70% yesterday.

```bash
# If monitoring is in Splunk — pull fill rate over last 24h
index=infrastructure host=<storage_host> metric_name="disk.used_percent" mount=/storage/path
| timechart span=1h avg(value) as utilization

# If using Prometheus/Grafana: query rate of change
rate(node_filesystem_avail_bytes{mountpoint="/storage/path"}[1h])
```

**Step 3: Identify largest consumers**

```bash
du -sh /mnt/storage/path/* 2>/dev/null | sort -rh | head -30

# Find files larger than 1GB written in last 48h (runaway writer)
find /mnt/storage/path -mtime -2 -size +1G -ls 2>/dev/null | sort -k7 -rn

# Find log-like directories (common culprits)
find /mnt/storage/path -name "*.log" -mtime -7 | xargs du -sc 2>/dev/null | sort -rh | head -20
```

**Decision point**: Is the fill rate active (filling now) or static (already full)?  
- Active fill: Stop the source first, then recover space. Stopping source is priority.  
- Static fill: Recover space, then identify why alert didn't fire earlier.

## Phase 2: Quick Wins for Space Recovery (Minutes 10–45)

Work through these in order — fastest and safest first.

### 2.1 Expired Snapshots
```bash
# NetApp ONTAP — list snapshots older than 7 days
# Via ONTAP CLI or System Manager
volume snapshot show -volume <vol> -older-than 7d

# AWS EBS — find old snapshots
aws ec2 describe-snapshots --owner-id self --query \
  'Snapshots[?StartTime<=`2026-01-01`].[SnapshotId,StartTime,VolumeSize]' \
  --output table
```
Delete expired snapshots after confirming they are not referenced by any clone or AMI. Snapshots on NetApp can hold significant reclaimable space if a volume had large overwrites.

### 2.2 Orphaned Kubernetes Volumes
```bash
# PVs in Released state (PVC deleted but PV remains due to Retain policy)
kubectl get pv | grep Released

# Confirm no active PVC is bound
kubectl get pvc -A | grep <pv-name>

# If truly orphaned — describe the PV to understand what it was
kubectl describe pv <pv-name>
```

### 2.3 Old Log Files
**Confirm nothing is actively reading a log before deleting it.**

```bash
# Check if any process has the file open (critical — do not delete open files on Linux)
lsof /mnt/storage/path/old.log

# If file is open by an active process, truncate instead of delete
> /mnt/storage/path/old.log  # truncates without breaking the file descriptor
# Then rotate properly — do not just delete open log files

# Find candidate log files: no access in 30 days, compressed or old rotated logs
find /mnt/storage/path/logs -name "*.log.*" -atime +30 -ls | sort -k7 -rn
```

### 2.4 Temp Files and Build Artifacts
```bash
# Common temp locations that accumulate
find /mnt/storage/path -name "*.tmp" -o -name "*.temp" | xargs du -sc 2>/dev/null | tail -1
find /mnt/storage/path -name ".nfs*" -ls  # stale NFS handles
find /tmp -mtime +7 -ls | sort -k7 -rn   # old temp files on the storage server itself
```

### 2.5 Docker/Container Image Cache (if storage hosts run containers)
```bash
docker system df           # see how much space Docker is using
docker system prune -f     # remove stopped containers and unused images (CONFIRM before running)
```

## Phase 3: Safe Deletion Checklist

Before deleting any file or volume, confirm:

- [ ] File is not open by any process (`lsof <path>`)
- [ ] File is not referenced by any running job or scheduled task (check cron, systemd timers, batch schedulers)
- [ ] For database files: no database process has it open; database is not in recovery mode
- [ ] For snapshot: not cloned by any active volume; not referenced by any backup job currently running
- [ ] For PV: PVC is deleted; no pod reference exists; data is confirmed backed up or not needed
- [ ] Deletion is confirmed with application owner or DBA for any data that is not clearly temp/log

**When uncertain: do not delete. Compress instead, or move to a quarantine directory.**

```bash
# Safe quarantine pattern — move, don't delete
mkdir /mnt/storage/quarantine-$(date +%Y%m%d)
mv /mnt/storage/path/suspicious-file /mnt/storage/quarantine-$(date +%Y%m%d)/
# Review quarantine after 7 days — delete if nothing asks for it
```

## Phase 4: Expansion Options

If quick wins do not recover sufficient space, evaluate expansion.

**Decision tree**:

```
Can you recover >10% free space within 60 minutes using Phase 2?
├── YES → Execute, monitor, notify application owners, proceed to Phase 5
└── NO → Is growth still active?
    ├── YES → Stop the source first (kill runaway process, disable log shipping, stop job)
    │         Then escalate to Director if you cannot stop it quickly
    └── NO → Choose expansion option below based on system type
```

**Expansion options by system type**:

| System | Option | Lead Time | Approval |
|--------|--------|-----------|---------|
| NAS (NetApp ONTAP) | Expand volume (if aggregate has space) | Minutes | Storage team |
| NAS (NetApp ONTAP) | Add capacity to aggregate (disk shelf) | Days–weeks | Director + procurement |
| EBS (AWS) | Modify volume size (AWS console) | Minutes (online resize) | Cloud team |
| EFS (AWS) | Automatic — no manual expansion needed | N/A | N/A |
| S3/Object | No storage limit | N/A | Review lifecycle policy instead |
| Local disk | Add EBS volume, mount, move data | 30 minutes | Cloud team |

**Emergency object storage offload for cold data**:

If hot NAS/SAN is full and contains cold data (not accessed in 30+ days), consider emergency migration:

```bash
# Identify cold files (not accessed in 30 days)
find /mnt/storage/path -atime +30 -size +100M -ls | sort -k7 -rn | head -50

# Copy to S3 (do not move until verified)
aws s3 cp /mnt/storage/path/cold-data/ s3://cold-offload-bucket/cold-data/ --recursive

# Verify copy integrity
aws s3 ls s3://cold-offload-bucket/cold-data/ --recursive | head -20

# Only after verification — create symlinks or update application config to point to S3 path
```

## Phase 5: Escalation and Notification

### When to notify storage team:
- Any storage system at 90%+ (critical) — page, do not email

### When to notify application owners:
- When their application's storage is causing the fill
- When their application will be impacted by an emergency expansion or migration

### When to notify Director:
- System has reached 95%+ and space cannot be recovered within 1 hour
- Customer-impacting service is degraded due to storage full condition
- Planned expansion requires budget approval
- Data loss has occurred or is imminent

### Notification template for Director:
```
STORAGE ALERT — [System Name] at [X]%
Time: [HH:MM]
System: [NAS/SAN/EBS/EFS — hostname or ARN]
Current: [used GB] / [total GB] ([X]%)
Fill rate: [GB/hour — "static" if not actively filling]
Impact: [which applications, which services, customer impact Y/N]
Actions taken: [what has been done]
ETA to resolution: [estimate or "unknown — escalating"]
What I need from you: [budget approval / vendor call / executive communication]
```

## Phase 6: Post-Incident and Permanent Fix

### Root cause investigation
- Why did utilization reach 80% without triggering a capacity planning action?
- Was growth monitoring in place? Was anyone acting on it?
- Was there a runaway process, or was this predictable growth?

### Permanent remediation
- [ ] Volume-level quotas enabled for all application storage (prevent runaway growth)
- [ ] Monitoring alerts tuned — 70% alert to storage team, 80% page, 90% escalation
- [ ] Growth rate dashboard visible to storage team and application owners
- [ ] Capacity planning cadence established (quarterly review of Days-to-Full projections)
- [ ] Lifecycle policies reviewed — are cold data tiers in use?

### Capacity planning update
Update the storage capacity planning template with:
- New Days-to-Full projection post-expansion
- Application team contact for next growth planning conversation
- Date of next capacity review
