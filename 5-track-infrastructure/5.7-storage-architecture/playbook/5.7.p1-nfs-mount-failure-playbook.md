# Playbook: NFS Mount Failure Response

**Trigger**: Application errors referencing filesystem paths, NFS mounts reported stale or unresponsive, new mount attempts failing, Kubernetes pods stuck in ContainerCreating with volume mount errors  
**Severity**: P1 if production application impacted | P2 if degraded | P3 if non-production  
**Playbook owner**: Infrastructure/Storage team  
**Escalation**: Storage team lead → Director if customer-impacting and not resolved in 30 minutes

---

## Symptom Identification — What Kind of Failure Is This?

Correct diagnosis before action. NFS failures present in three distinct patterns:

| Symptom | Failure type | First action |
|---------|-------------|-------------|
| Application errors: "Stale file handle", "Input/output error" | Mount went stale (NFS server rebooted or path changed) | Remount on client |
| Application hangs, no errors — process just stops | NFS mount unresponsive (soft vs hard mount behavior) | Check network, check server |
| New pod/server cannot mount | New mount failing | Network, export config, credentials |
| Kubernetes pod stuck in ContainerCreating | CSI/PVC mount issue | PV/PVC/CSI driver |

## Phase 1: Client-Side Triage

**Step 1: Verify what is mounted and what is not**

```bash
# List all NFS mounts with server, export, and mount options
mount | grep nfs
# or
df -hT | grep nfs

# Check mount options — important for troubleshooting
# Look for: hard vs soft, timeo, retrans, vers (nfs version), proto (tcp/udp)
cat /proc/mounts | grep nfs
```

**Step 2: Test if existing mounts are responsive**

```bash
# Stat a file in the mount — if it hangs, mount is unresponsive (not just slow)
# Use timeout to prevent terminal hang
timeout 10 stat /mnt/nfs/path/testfile 2>&1
echo "Exit code: $?"
# Exit 124 = timed out = mount is hung

# df on a specific mount (also hangs if mount is unresponsive)
timeout 10 df -h /mnt/nfs/path 2>&1
```

**Step 3: Detect stale file handles**

```bash
# A stale handle means the NFS server still exists but the export path changed or server rebooted
# This is the most common post-reboot symptom
ls -la /mnt/nfs/path 2>&1 | grep -i stale
# "Stale file handle" error = remount required

# Check dmesg for NFS errors
dmesg -T | grep -i nfs | tail -30
journalctl -k | grep -i nfs | tail -30
```

**Step 4: Check for hung mount processes**

```bash
# Find processes blocked on NFS I/O
ps aux | grep -E "^root.*D " | head -20
# D state (uninterruptible sleep) = waiting on I/O = NFS is likely hung

# More targeted
cat /proc/*/wchan 2>/dev/null | sort | uniq -c | sort -rn | head -10
# nfs_wait_bit_killable = NFS hung process
```

## Phase 2: Network Path Verification

**Step 1: Can the client reach the NFS server?**

```bash
# Basic connectivity
ping -c 4 <nfs-server-ip>
traceroute <nfs-server-ip>

# NFS port check (NFS v4 uses TCP 2049 only — single port)
nc -zv <nfs-server-ip> 2049

# NFS v3 also needs portmapper (111) and dynamic ports
rpcinfo -p <nfs-server-ip>
```

**Step 2: Verify VLAN and firewall path**

```bash
# Confirm client is on the correct VLAN/subnet for storage traffic
ip addr show
ip route show

# If firewall separates storage VLANs — verify rule allows TCP 2049 from client subnet to NFS server
# This must be verified with network team if basic connectivity fails
```

**Step 3: MTU and jumbo frames (if NFS traffic uses jumbo frames)**

```bash
# Check MTU on the NFS network interface
ip link show <storage-interface>
# If storage VLAN expects MTU 9000 but interface is 1500, you will see fragmentation issues

# Test with ping at large packet size (subtract 28 bytes for ICMP headers)
ping -c 4 -s 8972 -M do <nfs-server-ip>
# "Frag needed" error = MTU mismatch
```

## Phase 3: Server-Side Triage

Requires access to the NFS server or coordination with the storage team.

**Step 1: Verify NFS service status on the server**

```bash
# Linux NFS server
systemctl status nfs-server
# or
systemctl status nfs-kernel-server   # Debian/Ubuntu naming

# Check NFS service is actually exporting
showmount -e <nfs-server-ip>         # from client (if rpcbind accessible)
showmount -e localhost               # from server directly

# NFS statistics
nfsstat -s   # server-side stats
nfsstat -c   # client-side stats
```

**Step 2: Verify exports configuration**

```bash
# View current active exports
exportfs -v     # run on NFS server

# View export config file
cat /etc/exports

# Check if the specific path is still exported
exportfs -v | grep /path/to/export
```

**Step 3: Verify disk health on the NFS server**

```bash
# Check if the underlying disk the NFS server exports is healthy
df -hT   # on the NFS server — is the source filesystem full?
dmesg -T | grep -iE "error|fault|io error|ata" | tail -20

# If NAS appliance (NetApp ONTAP, Dell) — check via vendor management tools
# NetApp: check Event Management System (EMS) for disk faults
# Pure: check alerts in Pure1 or Purity management
```

**Step 4: NFS server load**

```bash
# Check NFS thread count and load
cat /proc/net/rpc/nfsd
# "th" line: number of threads, read-ahead, cache hits
# Low thread hit ratio = server is overloaded

# System load
uptime
top -bn1 | head -15
```

## Phase 4: Common Failure Modes and Resolutions

### Failure 1: NFS Server Rebooted
**Symptoms**: Stale file handles on all clients, mounts existed before the reboot  
**Verification**: Check server uptime (`uptime`) — if recently rebooted, this is the cause  
**Resolution**: Remount on all clients (see Phase 5)  
**Why it happens**: NFS server state (open file handles, locks) is lost on reboot. Clients hold stale references.

---

### Failure 2: Network Issue Between Client and Server
**Symptoms**: Mount hangs rather than stale handle errors. New mounts time out.  
**Verification**: `ping` and `nc -zv <server> 2049` fail or show packet loss  
**Resolution**: Engage network team. Verify VLAN, firewall rules, switch port status  
**Check**: Is this affecting all clients or just one? All = network/server. One = that client's network config.

---

### Failure 3: NFS Export Removed or Path Changed
**Symptoms**: Stale handle; `showmount -e <server>` does not show expected export  
**Verification**: `exportfs -v | grep <expected-path>` — if missing, export was removed  
**Resolution**: Coordinate with storage team to restore export OR update client mount to new path  
**Do not**: Blindly create a new export to a different path — confirm with storage team what happened to the data

---

### Failure 4: NFS Server Disk Full
**Symptoms**: Writes fail with "No space left on device" even though client filesystem looks fine  
**Verification**: `df -h` on NFS server shows 100% on the exported filesystem  
**Resolution**: Capacity crisis on the NFS server — invoke storage-capacity-crisis-playbook for the server  
**Immediate**: Notify application owners that writes are failing

---

### Failure 5: Authentication Failure (NFS v4 with Kerberos)
**Symptoms**: Permission denied on mount; `mount` command succeeds but files show `nobody:nobody` ownership  
**Verification**: Check Kerberos ticket validity (`klist`), check KDC connectivity, check `/etc/idmapd.conf` for correct domain  
**Resolution**: Renew Kerberos credentials, verify idmapd domain matches on client and server

## Phase 5: Remount Procedure

### For a responsive mount with stale handles:

```bash
# Step 1: Identify what is using the mount
lsof | grep /mnt/nfs/path
fuser -mv /mnt/nfs/path

# Step 2: If nothing critical is using it — unmount cleanly
umount /mnt/nfs/path

# Step 3: If unmount hangs (mount is unresponsive) — lazy unmount
umount -l /mnt/nfs/path    # lazy unmount: detaches immediately, cleans up when all file handles closed

# Step 4: Verify unmounted
mount | grep /mnt/nfs/path  # should return nothing

# Step 5: Remount
mount -t nfs4 -o rw,relatime,vers=4.1,rsize=1048576,wsize=1048576,hard,proto=tcp <server>:/export/path /mnt/nfs/path

# Step 6: Verify mount is responsive
timeout 10 ls -la /mnt/nfs/path
```

### For a hung mount (process is blocking):

```bash
# Identify PIDs using the hung mount
fuser -mv /mnt/nfs/path

# SIGTERM to application processes — allow graceful shutdown if possible
kill -15 <pid>

# If process does not exit — SIGKILL (last resort)
kill -9 <pid>

# Lazy unmount the hung filesystem
umount -l /mnt/nfs/path

# Wait for kernel to release the mount reference
# Then remount
sleep 5 && mount -t nfs4 -o ... <server>:/path /mnt/nfs/path
```

### Handling Kubernetes pods with NFS mount issues:

```bash
# Check pod events for mount error details
kubectl describe pod <pod-name> -n <namespace>
# Look for: "Unable to mount volumes", "MountVolume.SetUp failed"

# Delete and recreate pod (if deployment/statefulset manages it)
kubectl rollout restart deployment/<name> -n <namespace>
# StatefulSet pod — delete specific pod
kubectl delete pod <pod-name> -n <namespace>

# If PVC is stuck in Terminating — check for finalizers
kubectl get pvc <pvc-name> -n <namespace> -o yaml | grep finalizers
# Remove finalizer only if you are certain the volume is truly free
```

## Phase 6: Escalation to Storage Team

Provide this information when escalating:

- NFS server hostname or IP
- Export path
- Affected clients (count and hostnames)
- Symptom (stale handle, hung mount, new mount failing, permission denied)
- Output of: `showmount -e <server>`, `dmesg | grep nfs | tail -20`, `nfsstat`
- Network test results (ping, nc to port 2049)
- Whether this is affecting all clients or specific ones

---

## Phase 7: Application Impact Assessment

While resolving, assess impact in parallel:

- Which applications are affected? Query the mount path → application dependency map
- Can the application tolerate the NFS mount being offline? Stateless applications may queue requests; databases may crash
- Is data being written that could be lost if we force-unmount? Check application transaction logs
- Is there a failover or degraded mode the application can run in? Alert NOC if not

---

## Phase 8: Recovery Verification

After remount, confirm full recovery before closing the incident:

```bash
# Mount is responsive
timeout 10 stat /mnt/nfs/path/ && echo "MOUNT OK"

# Read test — can we read existing files?
timeout 10 ls -la /mnt/nfs/path/ | head -10

# Write test — can we write? (only if write permission expected)
timeout 10 touch /mnt/nfs/path/.mount-test-$(date +%s) && echo "WRITE OK"
rm -f /mnt/nfs/path/.mount-test-*

# Application health check
<application-specific health check — confirm application is serving normally>
```

## Long-Term Recommendations

**NFS timeout tuning for hard mounts**: Hard mounts never give up — processes hang indefinitely if NFS becomes unresponsive. For non-critical NFS mounts, `soft` + short `timeo` allows applications to fail fast.

```
# Hard mount (default, appropriate for production databases)
-o hard,proto=tcp,vers=4.1

# Soft mount with timeout (for non-critical, fail-fast preferred)
-o soft,timeo=30,retrans=3,proto=tcp,vers=4.1
```

**Automount vs static mounts**: `autofs` automounts NFS on demand and unmounts on idle — avoids stale handle persistence across reboots. Appropriate for home directories and infrequently accessed shares. Static `/etc/fstab` mounts are simpler to audit.

**HA NFS consideration**: If this NFS server is a single point of failure for a critical application, escalate to the Director for HA NAS evaluation (NetApp HA pair, active-active NAS cluster, or migration to EFS for cloud workloads).
