# Kubernetes Storage Options — Quick Reference

*Organized by cloud provider, then on-premises. Includes access modes, CSI drivers, and workload fit.*

---

## AWS — EKS

### EBS (Elastic Block Store)

| Volume Type | Max IOPS | Max Throughput | Min/Max Size | Access Mode | Latency | AZ-Scoped | Best For |
|-------------|---------|---------------|-------------|-------------|---------|-----------|---------|
| **gp3** | 16,000 IOPS (configurable) | 1,000 MB/s | 1 GiB – 16 TiB | RWO | 1–5 ms | Yes | Default choice — most Kubernetes stateful workloads |
| **gp2** | 3 IOPS/GiB (burst to 3,000) | 250 MB/s | 1 GiB – 16 TiB | RWO | 1–5 ms | Yes | Legacy — migrate to gp3; better perf at same cost |
| **io2** | 64,000 IOPS | 1,000 MB/s | 4 GiB – 16 TiB | RWO (io2 Block Express supports RWO only) | <1 ms | Yes | High-IOPS databases — PostgreSQL, MySQL production |
| **io1** | 64,000 IOPS | 1,000 MB/s | 4 GiB – 16 TiB | RWO | <1 ms | Yes | Legacy io, prefer io2 |
| **st1** | 500 IOPS | 500 MB/s | 125 GiB – 16 TiB | RWO | Higher | Yes | Throughput-intensive, sequential (log storage, Hadoop) |
| **sc1** | 250 IOPS | 250 MB/s | 125 GiB – 16 TiB | RWO | Higher | Yes | Cold data, infrequent access |

**CSI driver**: `aws-ebs-csi-driver` (EKS add-on — install via `eksctl` or `terraform`)  
**StorageClass parameter key**: `type: gp3` | `iops: "16000"` | `throughput: "500"`  
**Critical gotcha**: EBS is AZ-scoped. Pod must run in same AZ as PV. Use `volumeBindingMode: WaitForFirstConsumer` on StorageClass to ensure volume is provisioned in the AZ where the pod will actually run.  
**Multi-attach (io1/io2)**: Allows attachment to multiple instances simultaneously, but requires cluster-aware filesystem (OCFS2). Not standard K8s RWX behavior.

### EFS (Elastic File System)

| Attribute | Detail |
|-----------|--------|
| **Access mode** | RWX — multiple pods on multiple nodes and AZs simultaneously |
| **Throughput modes** | Bursting (scales with storage), Provisioned (fixed MB/s), Elastic (auto-scales — preferred for K8s) |
| **Performance modes** | General Purpose (default, lower latency) \| Max I/O (higher aggregate throughput, higher latency) |
| **AZ-scoped** | No — cross-AZ by default (mount targets in each AZ) |
| **CSI driver** | `aws-efs-csi-driver` (EKS add-on) |
| **Pricing** | ~$0.30/GB/month (Standard); ~$0.025/GB/month (IA) with lifecycle policy |
| **Best for** | ReadWriteMany: shared config, ML checkpoints, web content shared across pods |
| **Avoid when** | High-IOPS databases (NFS latency, not block semantics), cost-sensitive single-pod workloads (EBS is cheaper per GB) |

**StorageClass key parameters**: `fileSystemId`, `provisioningMode: efs-ap` (access point mode — recommended)

---

### FSx for Lustre

| Attribute | Detail |
|-----------|--------|
| **Protocol** | Lustre parallel filesystem |
| **Access mode** | RWX — parallel access from multiple pods |
| **Throughput** | 50–1,000 MB/s per TiB provisioned |
| **Latency** | Sub-millisecond (NVMe-backed) |
| **CSI driver** | `aws-fsx-csi-driver` |
| **S3 integration** | Natively linked to S3 — load input data from S3, export results back to S3 |
| **Best for** | ML training data at scale (read speed is critical), genomics, HPC workloads requiring parallel I/O |
| **Cost** | Expensive — justify with ML training cluster scale |

---

### FSx for NetApp ONTAP

| Attribute | Detail |
|-----------|--------|
| **Protocols** | NFS (v3/v4), SMB, iSCSI |
| **Access mode** | RWO (iSCSI), RWX (NFS/SMB) |
| **CSI driver** | Trident (NetApp) — official Astra Trident CSI driver |
| **Features** | SnapMirror (replication), FlexClone (instant clones for dev/test), dedup/compression |
| **Best for** | Organizations with NetApp expertise migrating on-prem NAS workloads to AWS, need unified file+block on AWS |
| **Cost** | High — ONTAP licensing costs plus AWS storage |

---

### S3 via Mountpoint for Amazon S3

| Attribute | Detail |
|-----------|--------|
| **Protocol** | FUSE-based filesystem interface to S3 |
| **Access mode** | Read: excellent. Write: sequential new objects only — no overwrite, no append, no rename |
| **CSI driver** | `mountpoint-s3-csi-driver` |
| **Latency** | S3 API latency (tens of milliseconds) — not suitable for IOPS-sensitive workloads |
| **Best for** | ML inference (reading model weights), batch processing reading large S3 datasets, read-heavy static content |
| **Avoid when** | Databases, frequent small writes, anything needing POSIX semantics |

### EKS Recommended CSI Add-Ons Summary

| Add-on | Use case | Install method |
|--------|---------|---------------|
| aws-ebs-csi-driver | Block storage for most stateful workloads | EKS managed add-on |
| aws-efs-csi-driver | ReadWriteMany shared file storage | EKS managed add-on |
| mountpoint-s3-csi-driver | S3-backed read-heavy volumes | EKS managed add-on |
| aws-fsx-csi-driver | HPC/ML parallel filesystem | Manual Helm install |

## Azure — AKS

### Azure Disk (Azure Managed Disks)

| SKU | Max IOPS | Max Throughput | Access Mode | Latency | Best For |
|-----|---------|---------------|-------------|---------|---------|
| **Premium SSD (P-series)** | 20,000 IOPS (P80) | 900 MB/s | RWO | <1 ms | Production databases, high-IOPS workloads |
| **Ultra Disk** | 160,000 IOPS | 4,000 MB/s | RWO | <1 ms | Highest performance — mission-critical, latency-sensitive |
| **Standard SSD** | 6,000 IOPS | 750 MB/s | RWO | <10 ms | Dev/test, general workloads |

**CSI driver**: `disk.csi.azure.com` (built into AKS)  
**AZ-scoped**: Yes — same constraint as EBS  
**Shared disk**: Ultra Disk and Premium SSD support shared (multi-attach) with cluster-aware filesystem

---

### Azure Files (SMB and NFS)

| Protocol | Access Mode | Throughput | CSI driver | Best For |
|----------|-------------|-----------|-----------|---------|
| SMB | RWX | Up to 10 GB/s (Premium) | `file.csi.azure.com` | Windows workloads, shared config |
| NFS | RWX | Up to 10 GB/s (Premium) | `file.csi.azure.com` | Linux shared filesystem |

### Azure Blob via BlobFuse

| Attribute | Detail |
|-----------|--------|
| **CSI driver** | `blob.csi.azure.com` |
| **Access mode** | RWX (NFS-like mounting of blob storage) |
| **Best for** | ML datasets, read-heavy workloads, object store content accessed as filesystem |
| **Limitations** | BlobFuse adds latency vs native blob SDK; not POSIX compliant for all operations |

## On-Premises / Bare Metal

### Ceph / Rook-Ceph

| Interface | Protocol | Access Mode | CSI Driver | Notes |
|-----------|----------|-------------|-----------|-------|
| **RBD (RADOS Block Device)** | Block | RWO | rook-ceph RBD provisioner | Replicated block storage — databases |
| **CephFS** | NFS/POSIX | RWX | rook-ceph CephFS provisioner | Shared filesystem |
| **RADOS GW** | S3 / Swift | Object | Not Kubernetes CSI — use via SDK | On-prem object storage |

**Operator**: Rook — deploys and manages Ceph on Kubernetes using CRDs  
**Minimum cluster**: 3 storage nodes with dedicated OSDs  
**Replication**: Configurable per pool (default 3x replication or erasure coding)  
**Performance**: Comparable to enterprise SAN on NVMe with 25GbE+ network  
**Operational complexity**: High — requires Ceph expertise for tuning, OSD management, CRUSH map  
**Best for**: On-prem bare metal clusters, organizations with existing Ceph expertise, need for all three protocols (block/file/object) from one system

---

### Longhorn

| Attribute | Detail |
|-----------|--------|
| **Protocol** | Block (replicated across nodes) |
| **Access mode** | RWO (RWX experimental via NFS-based sharing) |
| **Replication** | Synchronous, configurable replica count (default 3) |
| **Snapshots** | Yes — built-in, backup to S3 |
| **CSI driver** | Built-in (Longhorn manager handles CSI) |
| **Install** | Helm chart — simpler than Rook-Ceph |
| **Best for** | Edge clusters, small-to-medium bare metal clusters, clusters where Ceph is overkill |
| **Avoid when** | Very high IOPS workloads (replication overhead), large-scale clusters (>50 nodes — Ceph scales better) |

---

### Local Path Provisioner (Rancher)

| Attribute | Detail |
|-----------|--------|
| **Protocol** | Local filesystem (directory on node) |
| **Access mode** | RWO |
| **Replication** | None — node failure = data inaccessible |
| **Performance** | NVMe speed — fastest possible |
| **CSI driver** | Local path provisioner (not CSI-spec — uses hostPath under the hood) |
| **Best for** | Applications with application-layer replication (etcd, Cassandra, Kafka, ZooKeeper), GPU scratch space, ephemeral scratch |
| **Avoid when** | Any workload where data must survive node failure without application-level protection |
