# Storage Protocol Comparison — Quick Reference

*Dense lookup table. For architecture decisions — not a substitute for vendor benchmarks in your environment.*

---

## Protocol Comparison Matrix

| Protocol | Transport | Typical Latency | Throughput Ceiling | IOPS Capability | Shared Access | K8s Compatible | Cost Tier |
|----------|-----------|----------------|-------------------|----------------|--------------|----------------|-----------|
| **DAS / NVMe (local)** | PCIe | 70–150 µs | 3–7 GB/s per drive | 500K–1M+ per drive | No | Yes — local path provisioner, hostPath | Low (no network infra) |
| **iSCSI** | TCP/IP (Ethernet) | 200–500 µs | 1–12 GB/s (per link speed) | 10K–500K | Limited (one host per LUN, unless cluster FS) | Yes — iSCSI CSI drivers | Low–Medium (Ethernet only) |
| **Fibre Channel (FC)** | FC fabric (dedicated) | 50–150 µs | 1.6–6.4 GB/s per port; aggregate much higher | 100K–2M+ | Limited (cluster FS for shared) | Limited — requires HBA on nodes; most cloud K8s not FC | High (HBAs, FC switches) |
| **NVMe-oF (FC-NVMe)** | FC fabric | 30–100 µs | 3.2–12 GB/s per port | 500K–5M+ | Limited | Emerging — FC-NVMe CSI support in enterprise arrays | Very High |
| **NVMe-oF (RoCE)** | RDMA over lossless Ethernet | 10–50 µs | 25–100+ GB/s | 1M–10M+ | Limited | Yes — NVMe-oF CSI drivers emerging | High (lossless Ethernet infra) |
| **NVMe/TCP** | TCP/IP (Ethernet) | 50–200 µs | 10–50 GB/s | 500K–2M+ | Limited | Yes — nvme-tcp CSI drivers | Medium (standard Ethernet) |
| **NFS v3** | UDP or TCP | 1–5 ms | Up to ~10 GB/s (10GbE = 1.2 GB/s per client) | 10K–50K per client | Yes — ReadWriteMany | Yes — NFS CSI, in-tree (legacy) | Medium (NAS appliance) |
| **NFS v4 / v4.1** | TCP | 1–5 ms | Up to ~10 GB/s; pNFS breaks ceiling | 10K–50K per client; higher with pNFS | Yes — ReadWriteMany | Yes — NFS CSI | Medium |
| **SMB 3.0** | TCP | 1–10 ms | Up to ~10 GB/s (multichannel) | 5K–30K per client | Yes — ReadWriteMany for Windows | Limited — SMB CSI driver available | Medium |
| **S3 (HTTP/HTTPS)** | TCP (HTTP) | 10–100 ms per API call | Effectively unlimited aggregate | N/A (API call model) | Yes — unlimited concurrent | Yes — S3 CSI (Mountpoint, csi-s3) | Low at scale |

|----------|-----------|-----------------|-------------|---------------|-----------------|---------------------|-----------------|-------------|

## Infrastructure Requirements

| Protocol | Server side | Client side | Network |
|----------|------------|------------|---------|
| DAS/NVMe | None (local) | None | None |
| iSCSI | iSCSI target (software on storage array) | open-iscsi daemon, DM-Multipath | Dedicated storage VLAN, jumbo frames (MTU 9000) recommended |
| Fibre Channel | Array FC ports | HBAs on each host | FC switches, zoning, cabling |
| NVMe-oF FC | Array NVMe-oF ports | NVMe-oF capable HBAs | FC switches with NVMe-oF support |
| NVMe-oF RoCE | Array RoCE ports | RDMA-capable NICs | Lossless Ethernet (PFC + DCB configured on switches) |
| NVMe/TCP | Array NVMe/TCP target | Linux kernel 5.0+ (built-in) | Standard Ethernet — no special config |
| NFS | NFS server (NAS appliance or Linux) | NFS client (nfs-utils) | Standard Ethernet; jumbo frames help |
| SMB | Windows File Server or NAS appliance | Windows native SMB | Standard Ethernet |
| S3 | Object storage (AWS, MinIO, Ceph RGW) | AWS CLI / SDK / s3fs / Mountpoint | HTTPS (TLS 1.2+) |

## Best For / Avoid When

| Protocol | Best For | Avoid When |
|----------|---------|-----------|
| **DAS/NVMe** | Highest-IOPS databases, ML GPU scratch, Kubernetes local volumes with app-layer replication | Workloads needing shared access, VM-based workloads needing live migration |
| **iSCSI** | Block storage where FC budget is unavailable, moderate IOPS databases, virtualization (iSCSI datastores) | Latency requirements < 200 µs, high-density environments where network congestion risk is high |
| **Fibre Channel** | Mission-critical databases, high-IOPS OLTP, VMware environments requiring lowest SAN latency | New greenfield infrastructure (consider NVMe/TCP instead), budget-constrained environments |
| **NVMe-oF (FC)** | Upgrade path from FC when array supports it, sub-100 µs latency requirement with existing FC fabric | Environments without FC infrastructure (use NVMe/TCP instead) |
| **NVMe-oF (RoCE)** | Highest performance storage networking, AI/ML training clusters | Teams without lossless Ethernet expertise (operational complexity is high) |
| **NVMe/TCP** | New high-performance storage deployments, replacing FC/iSCSI long-term | Sub-50 µs requirements (RoCE wins there), environments without kernel 5.x support |
| **NFS v3/v4** | File sharing (home dirs, app configs, shared code), K8s ReadWriteMany, lift-and-shift, Splunk warm | Databases requiring POSIX locking semantics, high-IOPS workloads, anything needing < 1 ms latency |
| **SMB 3.0** | Windows file shares, Active Directory-integrated environments, DFS namespaces | Linux-only environments, high-performance workloads |
| **S3 (HTTP)** | Log archives, backups, ML training datasets, media storage, CDR retention, long-term compliance | Databases (not POSIX), anything requiring sub-second latency, frequent small random writes |

## Quick Decision Tree

```
Workload needs shared access (multiple nodes writing simultaneously)?
├── YES → NFS (RWX) or S3 (via SDK, not filesystem)
└── NO → 
    Workload needs < 1 ms latency?
    ├── YES → 
    │   Budget for FC or NVMe-oF?
    │   ├── YES → FC or NVMe-oF
    │   └── NO → Local NVMe (if replicated at app layer) or iSCSI
    └── NO →
        Workload is Kubernetes on AWS?
        ├── YES → EBS gp3 (RWO) or EFS (RWX) or S3 (read-heavy batch)
        └── NO →
            Workload is archive / log retention / backup?
            ├── YES → S3 / object storage
            └── NO → iSCSI or NFS (evaluate based on workload type)
```
