# Chapter 1 — Storage Fundamentals: Matching Workload to Tier

The most expensive storage mistake in enterprise IT is not buying the wrong vendor — it is routing workloads to the wrong tier. This chapter gives you the routing logic.

---

## The Four Storage Architectures

### DAS — Direct-Attached Storage
Storage physically connected to the server, not shared over a network. The server owns the disks. No network protocol, no shared access.

**Technology**: Local NVMe SSDs (PCIe 4.0/5.0), SAS/SATA HDDs, U.2/M.2 form factors.  
**Performance**: Highest available — NVMe latency at 70–150 µs, throughput at 3–7 GB/s per drive, millions of IOPS from NVMe-based arrays.  
**Sharing model**: None. One server, one set of disks.  
**Redundancy**: RAID within the server, or application-level replication across servers.  
**Failure mode**: Disk failure on that server affects only that server. No blast radius to other workloads.

**Use DAS when**: The workload is latency-sensitive and you control redundancy at the application layer. Database servers with synchronous replication (PostgreSQL streaming replication, MySQL Group Replication), ML/AI GPU training nodes where scratch space is ephemeral, Kubernetes nodes running local volume provisioner for stateful pods that replicate at application level (Cassandra, etcd).

---

### NAS — Network-Attached Storage
File-level storage shared over a network. Clients see a filesystem they mount. NFS for Linux, SMB/CIFS for Windows. The NAS server owns the disks and presents a filesystem abstraction.

**Technology**: Dedicated NAS appliances (NetApp ONTAP, Dell PowerStore File, Synology for smaller scale) or general-purpose Linux (Samba, NFS server).  
**Performance**: Network-bound. 10GbE gives ~1.2 GB/s per client, 25GbE gives ~3 GB/s. NAS server throughput aggregates across all clients — can saturate at the server if oversubscribed.  
**Sharing model**: Many clients, one namespace. Multiple servers can mount the same share simultaneously (ReadWriteMany in Kubernetes terms).  
**Redundancy**: NAS appliances ship as HA pairs. Active-passive or active-active. Storage RAID inside the appliance.  
**Failure mode**: NAS server failure takes all mounted clients offline simultaneously — wide blast radius.

**Use NAS when**: The workload needs shared file access — home directories, application config shared across pods, shared code repositories, NFS-backed persistent volumes in Kubernetes for ReadWriteMany use cases, lift-and-shift workloads that expect a POSIX filesystem.

---

### SAN — Storage Area Network
Block-level storage presented over a dedicated network. The host OS sees a raw block device — no filesystem built in. The host formats and mounts it like a local disk. SAN is the closest you get to "shared DAS."

**Technology**: iSCSI (over Ethernet), Fibre Channel (dedicated SAN fabric), NVMe-oF (over RDMA or TCP). Enterprise appliances: NetApp AFF, Pure FlashArray, Dell PowerMax.  
**Performance**: iSCSI: 10–100k IOPS, <500 µs latency on 25GbE. Fibre Channel: hundreds of thousands of IOPS, <100 µs. NVMe-oF: millions of IOPS, sub-50 µs.  
**Sharing model**: One LUN, typically one host (ReadWriteOnce semantics). Shared SAN access requires cluster-aware filesystems (OCFS2, VMware VMFS).  
**Redundancy**: Redundant storage controllers, redundant fabric paths (multipathing), RAID within the array.  
**Failure mode**: Storage controller failover. Path failure handled by multipath I/O software (MPIO, DM-Multipath on Linux).

**Use SAN when**: The workload requires block semantics and cannot be served by local NVMe — VMware datastores (vSAN replaces this in modern deployments), Oracle RAC databases, high-IOPS applications that need shared block (SQL Server FCI), workloads that need SAN data services (snapshots, thin provisioning, deduplication) alongside performance.

---

### Object Storage
Flat namespace storage accessed via HTTP API (GET/PUT/DELETE). No POSIX filesystem, no directories — just buckets and object keys. Designed to scale to exabytes. Eventually consistent (S3 is now strongly consistent for new writes; many self-hosted alternatives are still eventually consistent per key).

**Technology**: AWS S3, Google Cloud Storage, Azure Blob, MinIO (self-hosted S3-compatible), Ceph RADOS Gateway.  
**Performance**: High throughput at scale, high latency per object (~10–100 ms per API call vs µs for block). Designed for throughput, not latency. Parallel access scales linearly.  
**Sharing model**: Any client with credentials and network access. Unlimited concurrent access.  
**Redundancy**: Erasure coding across availability zones. AWS S3 Standard = 11 nines durability (99.999999999%).  
**Failure mode**: S3-class services: effectively none at the service level. On-prem: Ceph cluster failure, MinIO quorum loss.

**Use Object Storage when**: Durability and scale matter more than latency — backups, logs, ML training datasets, media assets, build artifacts, anything written once and read occasionally. Not for databases, not for anything requiring sub-millisecond access.

## Performance Characteristics at a Glance

| Type | Typical Latency | Throughput | IOPS Range | Shared? |
|------|----------------|------------|------------|---------|
| DAS/NVMe | 70–150 µs | 3–7 GB/s per drive | 500K–1M+ | No |
| SAN (FC/NVMe-oF) | 50–200 µs | 10–100+ GB/s aggregate | 100K–2M+ | Limited |
| SAN (iSCSI) | 200–500 µs | 1–25 GB/s | 10K–500K | Limited |
| NAS (10GbE NFS) | 1–5 ms | Up to 1.2 GB/s/client | 10K–50K | Yes |
| Object (S3) | 10–100 ms per call | Effectively unlimited aggregate | N/A | Yes |

---

## Reliability Model Per Type

**DAS**: Reliability lives at the application layer. RAID inside the server protects against disk failure; application replication (streaming replication, distributed consensus) protects against server failure. No shared dependency — one node fails, others continue.

**NAS**: Reliability lives at the storage layer — HA pairs handle controller failure. But network availability and NAS server health are shared dependencies. One NAS outage affects all clients mounting that server.

**SAN**: Reliability at both storage (redundant controllers, RAID) and fabric (dual FC fabrics, multipath) layers. SAN outage is still shared blast radius but more contained than NAS because each LUN typically serves fewer hosts.

**Object**: Highest inherent durability — erasure coding, multi-AZ replication, designed for loss of entire availability zones. S3's 11-nines durability is not marketing; it reflects the architecture.

## Cost Hierarchy

| Tier | Relative $/TB | Why |
|------|--------------|-----|
| DAS (NVMe) | Low–Medium | You own the hardware, no sharing infrastructure overhead, but NVMe drives are not cheap |
| Object (cloud) | Low at scale | Storage is cheap; API calls and egress are where costs accumulate |
| NAS | Medium | Appliance cost spread across shared users; still requires networking |
| SAN | High | Dedicated fabric (FC), enterprise appliances, per-port licensing, management overhead |
| SAN (All-Flash, NVMe-oF) | Highest | Enterprise licensing + highest-grade hardware |

## Director Frame

Storage tier routing is a cost and performance decision — not a familiarity decision. The most common failure pattern: an organization learns NAS, then NAS-es everything. Logs go to NFS instead of object storage. ML datasets land on NAS instead of S3. Temporary scratch space lands on expensive SAN instead of local NVMe.

The Director's job is to establish routing policy:
- Define what characteristics qualify a workload for each tier (IOPS, latency, sharing, durability, retention)
- Publish and enforce the policy — new workloads don't self-select their storage tier
- Regularly audit tier placement — workloads age and their requirements change
- Use cost-per-GB-per-month as the accountability metric, not just utilization

At T-Mobile scale: CDR and log data are object storage workloads. Shared Splunk warm buckets may be NAS. Kubernetes stateful databases should be local NVMe or SAN block, not NAS. The routing decision is also the cost management decision.
