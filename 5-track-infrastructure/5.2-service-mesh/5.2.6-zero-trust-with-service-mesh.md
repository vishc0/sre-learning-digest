# Chapter 6: Zero Trust with Service Mesh

## The Scale of Adoption

According to 2025 cloud-native adoption surveys, 47% of Kubernetes production clusters use a service mesh as their primary mechanism for zero trust enforcement between workloads. This is not an academic concept — it is how production security teams at companies managing regulated, high-volume platforms have solved the lateral movement problem at scale.

---

## Why the Old Security Model Failed

The traditional enterprise security model was built on a perimeter assumption: trust the internal network. If traffic originates from inside the firewall, it is trusted. Pod A can call Pod B can call Pod C without any authentication, because they are all "inside."

This assumption breaks catastrophically once an attacker has a foothold. One compromised pod can call any other pod in the cluster. It can query the user-profile-service API to exfiltrate customer data. It can push messages into the message-router queue. It can call the admin endpoint on the notification-service. The network says "you're inside, you're trusted" — there is no second gate.

This is how most cloud-native breaches propagate: not by breaking through the perimeter, but by moving **laterally** after the initial compromise. The blast radius of a single compromised pod in a traditional Kubernetes deployment is the entire cluster.

---

## The Zero Trust Principle

"Never trust, never verify" → "Never trust, always verify."

Network location means nothing. A request from IP `10.0.4.17` carries zero security value — that IP changes with every pod restart, and an attacker with network access can spoof it. **Identity** means everything. Cryptographic proof that a specific service account made a specific call to a specific endpoint.

Every service call is authenticated and authorized, every time, regardless of whether it originates from inside or outside the cluster. The perimeter is irrelevant.

---

## Layer 1: Authentication via mTLS

Authentication is covered in Chapter 3, but viewed through the Zero Trust lens: **mTLS** gives you cryptographic proof of identity at every hop. When message-router calls notification-service, the TLS handshake establishes that:

1. notification-service is talking to the genuine message-router (`spiffe://t-mobile.com/notification/message-router`), not a spoofed pod.
2. message-router is talking to the genuine notification-service, not a man-in-the-middle.

With `PeerAuthentication STRICT` mode enforced across all namespaces, any pod that cannot present a valid SPIFFE certificate is refused connection. This eliminates unauthenticated east-west traffic entirely.

---

## Layer 2: Authorization via AuthorizationPolicy

Authentication answers "who are you?" Authorization answers "what are you allowed to do?" mTLS alone is not Zero Trust — it proves identity but does not restrict what an authenticated service can call.

**AuthorizationPolicy** is the CRD that enforces access control at the API level. It evaluates: which source service identity is making the request, what HTTP method, and what path. Every other combination is denied.

### The Deny-All Baseline

The first policy you create in any namespace:

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: notification
spec:
  {}
```

An empty spec with no rules denies all traffic to all services in the namespace. From this baseline, you add explicit allow policies for legitimate traffic flows. This is the principle of least privilege: start with nothing permitted, then open exactly what is needed.

### Explicit Allow Policy

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: notification-service-allow
  namespace: notification
spec:
  selector:
    matchLabels:
      app: notification-service
  action: ALLOW
  rules:
  - from:
    - source:
        principals:
        - "cluster.local/ns/notification/sa/message-router"
    to:
    - operation:
        methods: ["POST"]
        paths: ["/api/v1/send"]
```

This policy reads precisely: "Only the message-router service account, authenticated via mTLS, may send HTTP POST requests to `/api/v1/send` on notification-service. All other callers, methods, and paths are denied."

A pod with a different service account, even if it is inside the same namespace, cannot call this endpoint. A legitimate message-router pod trying to call `/api/v1/admin` — denied. The policy is endpoint-level, not service-level.

### The RabbitMQ Analogy

RabbitMQ has a user permissions model: configure which RabbitMQ user can write to which exchange and read from which queue. `message-router-user` has write access to the `notifications.outbound` exchange. A different user has read access to the `notifications.dlq` queue. No user has access to everything by default.

**AuthorizationPolicy** is this same model applied to HTTP and gRPC traffic. "Which service identity can call which endpoint with which method." The concepts are identical — the implementation is mesh configuration instead of RabbitMQ admin console settings.

## Zero Trust Maturity Model

| Level | Capability | What It Means | Implementation |
|-------|-----------|---------------|----------------|
| 1 | mTLS everywhere | Cryptographic identity for all workloads | PeerAuthentication STRICT in all namespaces |
| 2 | AuthorizationPolicy | Fine-grained access control per endpoint | Explicit allow policies, default deny |
| 3 | Policy as code | Mesh policies in Git, reviewed and audited | GitOps with Flux or ArgoCD for mesh config |
| 4 | Behavioral verification | Anomaly detection on access patterns | Istio telemetry + ML-based detection |

Most mature production platforms operate at Level 2-3. Level 4 requires additional tooling (e.g., Falco, service-level anomaly detection) layered on top of mesh telemetry.

## Blast Radius Reduction: T-Mobile Notification Platform

The 25M message/day notification platform runs on EKS. Without Zero Trust:

- A compromised message-router pod can call user-profile-service and exfiltrate customer data for every notification recipient.
- It can call the notification-service admin API and alter routing rules.
- It can query the Cassandra REST proxy to read raw message records.
- It can call any other service in the cluster — database proxies, monitoring endpoints, internal configuration APIs.

With **Zero Trust** enforced:

- A compromised message-router pod can only POST to `/api/v1/send` on notification-service. That is the only endpoint its AuthorizationPolicy allows.
- It cannot read user profiles. It cannot call admin APIs. It cannot query Cassandra. It cannot reach services in other namespaces.
- **Blast radius**: reduced from "anything in the cluster" to "exactly the operations message-router is legitimately authorized to perform."

The attacker gets a pod with a service account and a valid SPIFFE certificate — and immediately hits a wall on every call that isn't explicitly permitted.

## Compliance: The Audit Argument

SOC 2 Type II, PCI DSS, and NIST 800-53 all require demonstrating access control over sensitive data — including intra-service access to systems that handle personally identifiable information (PII) or payment card data.

In a traditional Kubernetes deployment, answering "which services can access your user PII API?" requires:
- Reading network policies (if they exist)
- Reviewing application code for API calls
- Checking IAM roles
- Trusting documentation that may be outdated

With **policy as code** in a GitOps workflow:
- Every AuthorizationPolicy lives in a Git repository.
- Changes require a pull request with code review.
- Merged policies are automatically applied by Flux or ArgoCD.
- The Git history is a complete, auditable record of who approved access between which services, when.

When the auditor asks "how do you control which services can access the user PII endpoint?" you show them a YAML file with a PR approval history. That is machine-readable, version-controlled evidence — the strongest possible form of compliance documentation.

## Key Terms

| Term | What It Is |
|------|------------|
| **Zero Trust** | Security model where identity is always verified regardless of network location |
| **AuthorizationPolicy** | Istio CRD defining which source identity can call which endpoint with which method |
| **PeerAuthentication STRICT** | Namespace-wide mTLS enforcement — all traffic must present a valid certificate |
| **lateral movement** | An attacker's ability to move from a compromised component to other systems |
| **blast radius** | The scope of damage possible from a single compromised component |
| **policy as code** | Storing access control policies in Git with review and audit history |
