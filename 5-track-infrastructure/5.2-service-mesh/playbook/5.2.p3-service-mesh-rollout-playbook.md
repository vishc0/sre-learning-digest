# Service Mesh (Istio) Gradual Rollout Playbook

## Purpose

Step-by-step procedure for rolling Istio out across an EKS cluster namespace-by-namespace without service disruption. This is an operational runbook, not a tutorial. Follow phases in order. Do not skip the validation gate between phases.

---

## Prerequisites

- Istio version: 1.21.x (verify current stable at istio.io/latest before starting — do not use this document's version as ground truth)
- EKS cluster version: 1.28+ recommended
- `kubectl` access to cluster with cluster-admin or equivalent role
- `istioctl` installed locally at the same version as the target Istio (version mismatch causes config validation failures)
- Helm 3.x (if using helm install method)
- Change ticket opened and approved
- Rollback procedure reviewed by a second engineer before starting

---

## Phase 1 — Pre-Rollout Assessment

Do not skip this phase. Surprises discovered here take 30 minutes to resolve. Surprises discovered in Phase 3 take 3 hours and a production incident.

### 1.1 Service Inventory

```bash
# List all namespaces
kubectl get namespaces

# List all services and their ports across the cluster
kubectl get services --all-namespaces -o wide

# List all pods and check for host network usage
kubectl get pods --all-namespaces -o json | \
  jq '.items[] | {name: .metadata.name, ns: .metadata.namespace, hostNetwork: .spec.hostNetwork}'
```

### 1.2 Identify Incompatible Patterns

These patterns must be resolved before sidecar injection. Injecting into a pod with these patterns will cause failures.

**Host network pods** (`hostNetwork: true`): Envoy cannot intercept traffic for pods using the host network. Istio skips injection for these automatically, but verify:

```bash
kubectl get pods --all-namespaces -o json | \
  jq '.items[] | select(.spec.hostNetwork==true) | {name: .metadata.name, ns: .metadata.namespace}'
```

**UDP-only services**: Istio sidecars intercept TCP only. Services communicating via UDP (e.g., DNS, DHCP, custom protocols) will not be affected by mesh policies and cannot have mTLS enforced. Document these explicitly before rollout.

**Pods running as UID 1337**: Istio reserves UID 1337 for the `istio-proxy` user. If your application container runs as UID 1337, there will be a conflict:

```bash
kubectl get pods --all-namespaces -o json | \
  jq '.items[] | select(.spec.securityContext.runAsUser==1337) | {name: .metadata.name, ns: .metadata.namespace}'
```

**Init containers that modify iptables**: Istio's `istio-init` container modifies iptables to redirect traffic. A competing init container doing the same will cause unpredictable behavior. Check:

```bash
kubectl get pods --all-namespaces -o json | \
  jq '.items[] | select(.spec.initContainers != null) | {name: .metadata.name, ns: .metadata.namespace, initContainers: [.spec.initContainers[].name]}'
```

### 1.3 Document Current Baselines

Record these in your change ticket before touching anything. You need these numbers to validate that the mesh did not degrade performance.

```bash
# Resource utilization per namespace — record per-pod CPU and memory
kubectl top pods -n <namespace>

# Note: P99 latency baselines should come from your existing APM tool (Splunk, Datadog, etc.)
# Record: P50, P99, P99.9 for every service in the rollout scope
# Record: error rates (HTTP 5xx, connection errors) per service
```

## Phase 2 — Control Plane Installation

### 2.1 Install istiod

```bash
# Download istioctl matching your target version
curl -L https://istio.io/downloadIstio | ISTIO_VERSION=1.21.0 sh -
export PATH=$PWD/istio-1.21.0/bin:$PATH

# Validate pre-conditions on the cluster (checks K8s version, permissions, conflicting resources)
istioctl x precheck

# Review the output. Fix any warnings before proceeding.
# Common issues: insufficient permissions, K8s version too old, conflicting webhook configs

# Install with the default profile
# For production: review the IstioOperator spec and customize resource requests/limits
istioctl install --set profile=default -y

# Verify installation
kubectl get pods -n istio-system
# Expected:
#   istiod-<hash>                    1/1     Running
#   istio-ingressgateway-<hash>      1/1     Running
```

### 2.2 Verify istiod Health

```bash
# Check istiod logs — look for startup errors
kubectl logs -n istio-system -l app=istiod --tail=50

# Verify the mutating admission webhook is registered
# This webhook is what triggers automatic sidecar injection
kubectl get mutatingwebhookconfigurations | grep istio
# Expected: istio-sidecar-injector appears in the list

# Verify istiod metrics endpoint is healthy
kubectl exec -n istio-system deploy/istiod -- curl -s localhost:15014/metrics | grep pilot_xds_pushes
# Expected: a metric line with a count value (not empty)
```

**Healthy signal**: istiod pod Running, no ERROR lines in startup logs, webhook registered, metrics endpoint responding.

**Problem signal**: CrashLoopBackOff on istiod, "failed to list VirtualServices" in logs (RBAC issue), webhook not registered (installation failed partway).

**If istiod fails to start**: do not proceed to Phase 3. Debug the control plane first. Common fixes:
- RBAC issue: `istioctl verify-install` will show what's missing
- Existing CRD conflicts: check for a prior Istio installation: `kubectl get crd | grep istio`

## Phase 3 — Pilot Namespace

### 3.1 Choose the Pilot Namespace

Selection criteria — all three should be true:
- Lowest traffic volume among candidates (do not choose the notification service or any customer-facing service for the pilot)
- Non-critical path (internal tooling, dev environment, or a batch processing namespace)
- Contains at least 2 services that make service-to-service calls, so you can observe mTLS in action

If no development namespace exists, create a synthetic test namespace with two services that call each other. Do not pilot in production.

### 3.2 Enable Sidecar Injection

```bash
# Label the namespace — this tells the webhook to inject sidecars into new pods in this namespace
kubectl label namespace <pilot-namespace> istio-injection=enabled

# Verify the label was applied
kubectl get namespace <pilot-namespace> --show-labels

# Existing pods are NOT automatically injected — the webhook only fires on pod creation
# Force a rolling restart to re-create all pods with sidecar injection
kubectl rollout restart deployment -n <pilot-namespace>

# Wait for rollout to complete
kubectl rollout status deployment -n <pilot-namespace>

# Verify sidecars injected — each pod should now show 2/2 or higher READY count
kubectl get pods -n <pilot-namespace>
# READY column: 1/1 = no sidecar (wrong), 2/2 = app + istio-proxy (correct)

# Confirm the sidecar container is present by name
kubectl get pod <pod-name> -n <pilot-namespace> -o jsonpath='{.spec.containers[*].name}'
# Expected output includes: istio-proxy
```

### 3.3 Verify mTLS is Working

```bash
# Use istioctl to check TLS status between a source pod and destination service
istioctl authn tls-check <pod-name>.<namespace> <service-name>.<namespace>.svc.cluster.local
# Expected output: mTLS: OK
# Problem output: "CONFLICT" or "DISABLE" — indicates mode mismatch

# For a deeper look at Envoy's TLS configuration:
kubectl exec -n <pilot-namespace> <pod-name> -c istio-proxy -- \
  curl -s localhost:15000/config_dump | python3 -c "
import sys, json
d = json.load(sys.stdin)
for l in d.get('configs', []):
    if l.get('@type', '').endswith('ListenersConfigDump'):
        for entry in l.get('dynamic_listeners', []):
            print(entry.get('name'), '— TLS context present:', 'tls_context' in str(entry))
"
```

## Phase 4 — Validation Gate

Run every check in this section before proceeding to Phase 5. If any check fails, stop and execute the rollback procedure (Phase 7) before attempting to debug forward.

### 4.1 Latency Check

```bash
# Query Envoy's downstream connection timing stats
kubectl exec -n <pilot-namespace> <pod-name> -c istio-proxy -- \
  curl -s localhost:15000/stats | grep downstream_cx_length_ms

# Compare to your Phase 1 baseline
# Acceptable: P99 increase of less than 5ms due to sidecar overhead
# Investigate further if: P99 increase is greater than 10ms
# Stop and rollback if: P99 increase exceeds 20ms or error rate increases
```

### 4.2 Error Rate Check

```bash
# Review Envoy access logs for unexpected errors
kubectl logs <pod-name> -n <pilot-namespace> -c istio-proxy --tail=100

# Specifically look for 503 errors — these indicate mTLS or policy failures, not app errors
kubectl logs <pod-name> -n <pilot-namespace> -c istio-proxy | grep ' 503 '
# Expected: zero results
```

### 4.3 Validation Gate Checklist

- [ ] All pods in pilot namespace showing 2+ containers (sidecar injected)
- [ ] `istioctl authn tls-check` passing for all service pairs in the namespace
- [ ] P99 latency increase is less than 5ms compared to baseline
- [ ] Error rate unchanged from baseline (less than 0.1% increase)
- [ ] Zero 503 errors appearing in Envoy access logs
- [ ] `kubectl top pods -n <pilot-namespace>` shows resource consumption within expected range

**Gate pass**: all items checked. Proceed to Phase 5.

**Gate fail**: one or more items unchecked. Execute Phase 7 rollback. Do not proceed to Phase 5. Debug, fix the root cause, and restart from Phase 3 after resolution.

## Phase 5 — Wave Rollout

### 5.1 Wave Order

Define your waves before starting rollout. Do not improvise the order during the rollout.

| Wave | Namespaces | Soak Period |
|---|---|---|
| Wave 1 | Non-prod: dev, staging, QA | 2 weeks minimum |
| Wave 2 | Low-criticality prod: batch jobs, internal tooling, metrics pipelines | 2 weeks minimum |
| Wave 3 | High-criticality prod: notification-service, message-router, customer-facing APIs | 2 weeks minimum |

The soak period is non-negotiable. You are looking for slow-burn failures: memory leaks in sidecars, gradual cert rotation issues, cardinality explosions in Prometheus. These do not show up in the first 24 hours.

### 5.2 Per-Wave Execution

Repeat for each namespace in the wave. Do not batch multiple namespaces simultaneously — if something goes wrong, you want to know which namespace caused it.

```bash
# Label the namespace for injection
kubectl label namespace <namespace> istio-injection=enabled

# Verify label applied
kubectl get namespace <namespace> --show-labels | grep istio-injection

# Rolling restart to inject sidecars
kubectl rollout restart deployment -n <namespace>

# Wait for all deployments to finish restarting
kubectl rollout status deployment -n <namespace>

# Run the full validation gate (Phase 4 checks) against this namespace
# Before moving to the next namespace in the wave, all gate criteria must pass
```

### 5.3 Between Waves

Before starting the next wave:
- Review Envoy access logs for the completed wave for any latent 503s
- Review Prometheus metrics for unexpected cardinality increase
- Confirm resource consumption (CPU/RAM) is within planned capacity
- Get written sign-off from service owners for any Wave 2/3 namespaces before rollout starts

## Phase 6 — Strict mTLS Enforcement

Only execute Phase 6 after ALL production namespaces are in the mesh and have completed their soak period. Applying STRICT mode before all namespaces are injected will break traffic from non-mesh services to mesh services.

```bash
# Apply STRICT PeerAuthentication cluster-wide
# This means: all traffic must be mTLS. Plaintext connections are rejected.
cat <<EOF | kubectl apply -f -
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system
spec:
  mtls:
    mode: STRICT
EOF

# Immediately after applying, watch for certificate-related errors in your highest-traffic namespace
kubectl logs -n <namespace> -l app=<service> -c istio-proxy -f | grep -iE "certificate|ssl|tls|503"

# Check for any services that are now failing (non-injected clients hitting STRICT-mode servers)
kubectl logs -n <namespace> -l app=<service> -c istio-proxy | grep ' 503 ' | head -20

# If 503s appear: identify the source
# kubectl logs output will include the source IP — map it to a pod
# If source is a non-injected pod, it will fail under STRICT mode
# Resolution: inject the source pod, or add a namespace-specific PERMISSIVE exception
```

## Phase 7 — Rollback Procedure

Execute this procedure if any validation gate fails, or if a production incident is traced to mesh behavior.

```bash
# Step 1: Remove the injection label — new pods created after this point will NOT get sidecars
kubectl label namespace <namespace> istio-injection-

# Verify label removed
kubectl get namespace <namespace> --show-labels
# istio-injection should no longer appear

# Step 2: Force rolling restart — existing pods still have sidecars until restarted
kubectl rollout restart deployment -n <namespace>

# Wait for restart to complete
kubectl rollout status deployment -n <namespace>

# Step 3: Verify sidecars removed
kubectl get pods -n <namespace>
# READY column should show 1/1 (app only, no sidecar)

# Step 4: Confirm service-to-service traffic is restored
# Run smoke tests against the affected services
# Check error rates in your APM tool
```

If rolling back the entire control plane (istiod):

```bash
# Remove injection labels from all namespaces
for ns in $(kubectl get namespaces -l istio-injection=enabled -o jsonpath='{.items[*].metadata.name}'); do
  kubectl label namespace $ns istio-injection-
done

# Restart all affected deployments
for ns in $(kubectl get namespaces -o jsonpath='{.items[*].metadata.name}'); do
  kubectl rollout restart deployment -n $ns 2>/dev/null
done

# Uninstall istiod (use the same istioctl version used during install)
istioctl uninstall --purge -y

# Verify cleanup
kubectl get pods -n istio-system
# Expected: No resources found
```

## Go-Live Criteria

All items must be checked before declaring the rollout complete.

- [ ] All production namespaces labeled for injection
- [ ] All production pods showing 2+ containers (sidecar injected)
- [ ] `PeerAuthentication` STRICT applied cluster-wide in istio-system
- [ ] Prometheus is scraping Envoy metrics from all namespaces
- [ ] Grafana dashboards showing mesh golden signals (request rate, error rate, latency, saturation)
- [ ] Splunk receiving Envoy access logs from all namespaces
- [ ] Zero unresolved 503 errors in Envoy logs in the past 24 hours
- [ ] Rollback procedure documented, linked in runbooks, and tested in staging
- [ ] On-call engineers trained on: how to check xDS sync, how to read `istioctl proxy-config`, how to initiate rollback
- [ ] Certificate rotation playbook reviewed and accessible
