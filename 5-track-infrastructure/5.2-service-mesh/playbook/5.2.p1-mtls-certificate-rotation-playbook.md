# mTLS Certificate Rotation Playbook

## Purpose

Respond to mTLS certificate expiry alerts and perform emergency rotation without service disruption. This playbook covers three distinct certificate types in an Istio mesh — each has a different failure mode, urgency, and rotation procedure.

---

## Certificate Architecture in Istio

Understanding the three cert types is not optional — confusing them leads to applying the wrong fix.

### Type 1: istiod CA Certificate (Root)

The root Certificate Authority that signs all workload certificates. This cert is long-lived (10-year default lifetime). If this cert expires, istiod can no longer issue or renew workload certs. Every service in the mesh will eventually fail as their workload certs expire and cannot be renewed.

**Storage location**: `kubectl get secret -n istio-system istio-ca-secret`

**Failure blast radius**: total — all workload cert issuance stops

### Type 2: Workload Certificates

Short-lived certificates (24-hour default lifetime) issued to each Envoy sidecar. Envoy auto-requests renewal at 80% of the cert lifetime (~19 hours for a 24h cert). In normal operations, workload certs should never expire — the renewal loop is automatic.

**Storage location**: held in Envoy's memory, not in K8s secrets (no persistent storage)

**Failure blast radius**: limited to the specific pod if renewal fails; widespread if istiod CA is unhealthy

### Type 3: Ingress Gateway Certificate

The TLS certificate that external clients see when connecting to your ingress gateway. This is completely separate from internal mTLS certs. Lifecycle is managed by ACM, cert-manager, or a manual Kubernetes Secret — depending on your setup.

**Storage location**: Kubernetes Secret in istio-system namespace, or ACM-managed

**Failure blast radius**: all external traffic to the gateway fails; internal mTLS unaffected

## Phase 1 — Detection

### 1.1 Prometheus Alerts (Set These Up Proactively)

If you do not have these alerts configured, add them before you need them.

```yaml
# Alert: istiod CA cert expiring within 30 days
# This is your early warning — 30 days is enough time to plan a rotation
- alert: IstioCACertExpiringSoon
  expr: (istio_agent_cert_expiry_seconds{cert_type="ca"} - time()) / 86400 < 30
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "istiod CA cert expires in {{ $value | humanizeDuration }}"
    runbook: "mtls-certificate-rotation-playbook.md — Phase 4"

# Alert: CA cert expiring within 7 days — escalate to critical
- alert: IstioCACertCritical
  expr: (istio_agent_cert_expiry_seconds{cert_type="ca"} - time()) / 86400 < 7
  for: 1m
  labels:
    severity: critical
  annotations:
    summary: "istiod CA cert expires in {{ $value | humanizeDuration }} — immediate action required"

# Alert: workload cert CSR signing is failing
# This should never fire in normal operations — if it fires, istiod CA is sick
- alert: IstioWorkloadCertRotationFailed
  expr: increase(citadel_server_csr_signing_err_count[10m]) > 0
  for: 2m
  labels:
    severity: critical
  annotations:
    summary: "istiod failed to sign workload cert CSRs — mesh mTLS will degrade as certs expire"
```

### 1.2 Manual Certificate Inspection

Run these when you receive an alert or when investigating a suspected cert issue.

```bash
# Check istiod CA cert expiry
kubectl get secret -n istio-system istio-ca-secret \
  -o jsonpath='{.data.ca-cert\.pem}' | \
  base64 -d | \
  openssl x509 -noout -dates
# Output example:
#   notBefore=Jan  1 00:00:00 2025 GMT
#   notAfter=Jan  1 00:00:00 2035 GMT

# Check workload cert expiry for a specific pod's Envoy sidecar
istioctl proxy-config secret <pod-name>.<namespace>
# Output columns: NAME, TYPE, STATUS, VALID CERT, EXPIRE, DAYS UNTIL EXPIRE
# Look for any row where DAYS UNTIL EXPIRE is negative (expired) or zero

# Check workload cert via direct Envoy query
kubectl exec <pod-name> -n <namespace> -c istio-proxy -- \
  openssl s_client -connect <dest-service>.<namespace>.svc.cluster.local:<port> \
  -showcerts 2>/dev/null | openssl x509 -noout -dates
```

### 1.3 Error Pattern Recognition

These are the specific log strings that indicate certificate problems. Knowing these saves 20 minutes of "what is this error?" during an incident.

```
# mTLS handshake failure — cert expired, invalid, or CA mismatch
CERTIFICATE_VERIFY_FAILED
SSL_ERROR_RX_RECORD_TOO_LONG
TLS handshake failed
HANDSHAKE_FAILURE

# CSR signing failure — istiod CA is the problem, not the workload
failed to send CSR
error sending CSR
certificate sign error

# Common confusion — this is NOT a cert problem, it is an authorization policy deny
RBAC: access denied
# If you see this, skip to the incident playbook — cert rotation will not fix it
```

## Phase 2 — Triage

Use this decision tree before taking any action. Applying the wrong rotation procedure to the wrong cert type wastes time and can cause service disruption.

```
STEP 1: What error are you seeing?
  → "CERTIFICATE_VERIFY_FAILED" or "TLS handshake failed"  → Go to STEP 2
  → "failed to send CSR" or "csr signing error"            → Skip to STEP 4 (istiod CA problem)
  → "RBAC: access denied"                                  → This is not a cert issue — see incident playbook
  → External users getting TLS errors at the gateway       → Skip to Phase 5 (ingress cert)

STEP 2: Is the workload cert expired?
  → Run: istioctl proxy-config secret <pod>.<ns>
  → If DAYS UNTIL EXPIRE is negative: Go to Phase 3 (workload cert rotation)
  → If cert is valid: the cert is not expired — the CA may be the issue

STEP 3: Is istiod running and healthy?
  → Run: kubectl get pods -n istio-system
  → If istiod is CrashLoopBackOff or not Running: Go to Phase 4 (CA rotation)
  → If istiod is Running: check why auto-rotation failed (network policy? resource exhaustion?)
  → Run: kubectl logs -n istio-system -l app=istiod | grep -i "cert\|sign\|csr"

STEP 4: Is the istiod CA cert expired?
  → Run: kubectl get secret -n istio-system istio-ca-secret \
         -o jsonpath='{.data.ca-cert\.pem}' | base64 -d | openssl x509 -noout -dates
  → If notAfter is in the past: Go to Phase 4 (emergency CA rotation)
  → If not expired: istiod CA is healthy — look at pod-to-istiod network connectivity on port 15012
```

## Phase 3 — Normal Workload Cert Rotation

Workload certs are designed to auto-rotate. If auto-rotation broke, the fix is almost always: restart the pod to force Envoy to re-request a new cert from istiod.

```bash
# Option A: Rolling restart of the deployment (zero-downtime, preferred)
kubectl rollout restart deployment/<deployment-name> -n <namespace>

# Wait for rollout to complete
kubectl rollout status deployment/<deployment-name> -n <namespace>

# Verify the new cert was issued
istioctl proxy-config secret <new-pod-name>.<namespace>
# Confirm: DAYS UNTIL EXPIRE is now approximately 24 (the default lifetime)

# Option B: Force cert refresh without restarting the app container (Istio 1.15+)
# This drains Envoy listeners and forces it to reconnect to istiod, which triggers cert re-issuance
kubectl exec <pod-name> -n <namespace> -c istio-proxy -- \
  curl -X POST localhost:15000/drain_listeners
# Note: this causes a brief period where the pod does not accept new connections
# Use rolling restart (Option A) unless you have a specific reason to avoid pod recreation
```

**Root cause investigation** — if workload certs are expiring without auto-rotating, find out why before closing the ticket:

```bash
# Check if istiod port 15012 is reachable from the affected pods
# (This is the port Envoy uses to request cert renewals via xDS SDS)
kubectl exec <pod-name> -n <namespace> -c istio-proxy -- \
  curl -v --max-time 5 https://istiod.istio-system.svc.cluster.local:15012

# Check for NetworkPolicy blocking the pod → istiod traffic
kubectl get networkpolicies -n <namespace>
kubectl get networkpolicies -n istio-system

# Check istiod error log for CSR processing failures
kubectl logs -n istio-system -l app=istiod | grep -i "csr\|sign\|fail" | tail -30
```

## Phase 4 — Emergency: istiod CA Rotation

This is the highest-impact cert operation. The istiod CA cert is the root of trust for the entire mesh. Rotation requires restarting all pods in all mesh namespaces — plan for this in your change window.

**Duration estimate**: 30–60 minutes for a 500-pod cluster. Rolling restarts are sequential per deployment.

```bash
# Step 1: Rotate the istiod CA cert by restarting istiod
# istiod will generate a new CA cert on startup if the old one is expired
kubectl rollout restart deployment/istiod -n istio-system

# Step 2: Wait for istiod to restart completely
kubectl rollout status deployment/istiod -n istio-system
# Expected: "successfully rolled out"

# Step 3: Verify the new CA cert was generated
kubectl get secret -n istio-system istio-ca-secret \
  -o jsonpath='{.data.ca-cert\.pem}' | \
  base64 -d | \
  openssl x509 -noout -dates
# Confirm: notAfter is now approximately 10 years from today

# Step 4: Verify istiod is healthy with the new CA
kubectl logs -n istio-system -l app=istiod --tail=30
# Expected: startup messages, no ERROR lines
# Look for: "Istiod is ready" or similar ready message

# Step 5: Force all workload certs to be re-issued
# istiod restarting does NOT automatically push new certs to existing Envoys
# Each pod must be restarted to trigger a new CSR with the new CA
for ns in $(kubectl get namespaces -l istio-injection=enabled -o jsonpath='{.items[*].metadata.name}'); do
  echo "Restarting deployments in namespace: $ns"
  kubectl rollout restart deployment -n $ns
  # Optional: add a pause between namespaces to avoid hammering istiod simultaneously
  # sleep 30
done

# Step 6: Monitor cert issuance as pods come up
kubectl logs -n istio-system -l app=istiod -f | grep -iE "cert|sign|csr|error"
```

**Expected timeline**:
- istiod restart: 2–3 minutes
- New CA cert active: immediately after istiod is Running
- All workload certs renewed: depends on cluster size and rollout rate — 30–60 minutes for 500 pods
- During renewal: pods with old certs will continue working (they hold valid certs) until they are restarted and pick up the new CA-signed cert

**Critical**: after CA rotation, any pod that was NOT restarted will still hold a cert signed by the OLD CA. If you later switch to STRICT mTLS or any CA pinning configuration, those old-CA pods will fail. Ensure all pods are restarted within the change window.

## Phase 5 — Ingress Gateway Cert Rotation

The ingress gateway's external TLS certificate is managed separately from internal mTLS. External users see this cert — it appears in browser padlock views and must match your domain.

### If the certificate is stored as a Kubernetes Secret:

```bash
# Create the new secret (--dry-run + apply is safe for existing secrets — it replaces in place)
kubectl create secret tls ingress-tls-cert \
  --cert=new-cert.pem \
  --key=new-key.pem \
  -n istio-system \
  --dry-run=client -o yaml | kubectl apply -f -

# Restart the ingress gateway deployment to pick up the new secret
kubectl rollout restart deployment/istio-ingressgateway -n istio-system

# Wait for rollout
kubectl rollout status deployment/istio-ingressgateway -n istio-system

# Verify the new cert is being served
kubectl exec -n istio-system deploy/istio-ingressgateway -- \
  openssl s_client -connect localhost:443 2>/dev/null | openssl x509 -noout -dates
# Confirm: notAfter matches your new certificate's expiry
```

### If the certificate is managed by cert-manager:

```bash
# Check the Certificate resource status
kubectl get certificate -n istio-system
# STATUS column: look for "True" in READY column and recent RENEWAL TIME

# Check for a failed CertificateRequest
kubectl get certificaterequests -n istio-system
# STATUS: Approved/Denied/Failed

# If cert-manager is stuck, force a re-issuance by deleting the existing secret
# cert-manager will detect the missing secret and create a new CertificateRequest automatically
kubectl delete secret <cert-secret-name> -n istio-system

# Watch cert-manager logs to confirm re-issuance started
kubectl logs -n cert-manager -l app=cert-manager --tail=30 -f
```

### If the certificate is managed by AWS ACM:

The ingress gateway does not hold the cert directly — ACM handles TLS termination at the NLB/ALB layer. Check the ACM console for the certificate status. If expired:
1. Request a new certificate in ACM for the same domain
2. Complete DNS validation
3. Update the Kubernetes Service annotation to reference the new ACM certificate ARN
4. AWS Load Balancer Controller will update the NLB/ALB listener automatically — no ingress gateway restart needed

## Phase 6 — Validation

Run these checks after any cert rotation. Do not close the incident ticket until all checks pass.

### 6.1 Cluster-Wide Cert Status Sweep

```bash
# Check all injected pods for cert validity
for ns in $(kubectl get namespaces -l istio-injection=enabled -o jsonpath='{.items[*].metadata.name}'); do
  echo ""
  echo "=== Namespace: $ns ==="
  for pod in $(kubectl get pods -n $ns -o jsonpath='{.items[*].metadata.name}'); do
    result=$(istioctl proxy-config secret $pod.$ns 2>/dev/null | grep -v "^$" | tail -1)
    echo "  $pod: $result"
  done
done
# Look for any row showing DAYS UNTIL EXPIRE as negative or "EXPIRED"
```

### 6.2 Error Rate Validation

```bash
# Check for 503 errors post-rotation (cert failures manifest as 503s in Envoy)
kubectl logs -n <namespace> -l app=<service> -c istio-proxy | grep " 503 " | wc -l
# Expected: 0

# Check for TLS errors specifically
kubectl logs -n <namespace> -l app=<service> -c istio-proxy | \
  grep -iE "certificate|ssl|tls|handshake" | tail -20
# Expected: no new errors after the rotation completed
```

### 6.3 mTLS Connectivity Validation

```bash
# Verify mTLS is still working between key service pairs after cert rotation
# Run this for each critical service-to-service path in your platform
istioctl authn tls-check <pod-name>.<namespace> <service-name>.<namespace>.svc.cluster.local
# Expected: mTLS: OK
```

### 6.4 Ingress Validation (if ingress cert was rotated)

```bash
# Verify the correct cert is being served externally
echo | openssl s_client -connect <your-ingress-hostname>:443 2>/dev/null | \
  openssl x509 -noout -dates -subject
# Confirm: notAfter is the new expiry date, CN matches your domain
```

## Post-Incident Actions

After any cert-related incident is resolved:

1. **Update Prometheus alerts** if the alert fired too late (e.g., add a 60-day warning for CA certs)
2. **Check if CA cert rotation should happen proactively** — if you found the CA cert has less than 90 days remaining, schedule a rotation in the next change window rather than waiting for the critical alert
3. **Document the root cause** — why did auto-rotation fail? Was it a network policy blocking port 15012? istiod resource exhaustion? Record it in the post-incident review
4. **Verify the rotation procedure worked as documented** — if you deviated from this playbook, update the playbook to reflect reality
