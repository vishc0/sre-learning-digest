# Service Mesh Connectivity Failure Incident Playbook

## Purpose

Structured debugging process for "Service A cannot reach Service B" in an Istio mesh. Follow steps in order — do not skip ahead. Skipping steps is how you spend 90 minutes on the wrong layer.

---

## Incident Classification

Before starting the step-by-step flow: confirm this is a mesh-layer problem, not an application or Kubernetes infrastructure problem. The mesh is one of at least four possible failure layers:

1. Application bug (wrong endpoint, missing retry logic)
2. Kubernetes networking (service misconfigured, endpoint slice empty)
3. Mesh layer (Envoy config, mTLS, AuthorizationPolicy)
4. Infrastructure (EKS node network, security group, VPC routing)

```bash
# Get the exact error the caller is seeing — quote it verbatim before proceeding
kubectl logs <pod-name> -n <namespace> --tail=50

# Common mesh error signatures and their layer:
#
#   "upstream connect error or disconnect/reset before headers"
#     → Envoy upstream connection failed — mesh or network layer
#
#   "no healthy upstream"
#     → Circuit breaker open, or all endpoints unhealthy — mesh layer
#
#   "RBAC: access denied"
#     → AuthorizationPolicy blocking the request — mesh layer (auth)
#
#   "CERTIFICATE_VERIFY_FAILED"
#     → mTLS cert issue — mesh layer (certs)
#     → Go directly to Step 4 and mtls-certificate-rotation-playbook.md
#
#   "connection refused"
#     → Port/endpoint config issue — could be app or mesh layer
#
#   "dial tcp: i/o timeout"
#     → Network timeout — check NetworkPolicy and security groups (infrastructure layer)
```

## Step 1 — Is the Target Pod Running and Healthy?

This is the most common non-mesh cause of connectivity failures. Check it first.

```bash
# Check pod status
kubectl get pod -n <target-namespace> -l app=<target-service>
# Expected: STATUS = Running, READY shows all containers ready (e.g., 2/2)

# Check for recent pod events — evictions, OOMKilled, image pull failures
kubectl describe pod -n <target-namespace> <pod-name>
# Scroll to "Events" section at the bottom — look for Warning events

# Check the application container logs for startup or runtime errors
kubectl logs -n <target-namespace> <pod-name> -c <app-container-name> --tail=50

# Verify the service has endpoints (if Endpoints is empty, K8s never finds a healthy pod)
kubectl get endpoints -n <target-namespace> <service-name>
# Expected: ENDPOINTS column shows IP:port entries
# Problem: "<none>" means no pods match the service selector — check pod labels vs service selector
kubectl get service -n <target-namespace> <service-name> -o yaml | grep -A5 selector
kubectl get pod -n <target-namespace> -l app=<target-service> --show-labels
```

**Decision**:
- Pod is not Running, or Endpoints is empty → This is an application or Kubernetes issue, not a mesh issue. Resolve the pod health problem first. The mesh cannot route to a pod that does not exist.
- Pod is Running and Endpoints has entries → Proceed to Step 2.

---

## Step 2 — Does the Pod Have a Sidecar Injected?

A pod without a sidecar is invisible to mesh-level policies and mTLS. Both sides of a connection must have sidecars for the mesh to manage the traffic.

```bash
# List containers in the source pod
kubectl get pod <source-pod-name> -n <source-namespace> \
  -o jsonpath='{.spec.containers[*].name}'
# Expected output: includes "istio-proxy" alongside your app container

# List containers in the target pod
kubectl get pod <target-pod-name> -n <target-namespace> \
  -o jsonpath='{.spec.containers[*].name}'
# Expected output: includes "istio-proxy"

# Check if the namespace has the injection label
kubectl get namespace <namespace> --show-labels | grep istio-injection
# Expected: istio-injection=enabled
```

**If istio-proxy is missing from source OR target pod**:

```bash
# Check if the injection label is missing from the namespace
kubectl get namespace <namespace> --show-labels

# If label is missing: add it and force a restart
kubectl label namespace <namespace> istio-injection=enabled
kubectl rollout restart deployment -n <namespace>

# If label is present but injection still didn't happen:
# Check the MutatingAdmissionWebhook — if this is absent or misconfigured, no injection occurs
kubectl get mutatingwebhookconfigurations | grep istio
kubectl get mutatingwebhookconfigurations istio-sidecar-injector -o yaml | grep -A5 namespaceSelector

# Check if istiod webhook service is healthy
kubectl get svc -n istio-system istio-sidecar-injector
kubectl get endpoints -n istio-system istio-sidecar-injector
```

**Decision**:
- Sidecar missing from one side → Fix injection, restart the pod, then retest. If retest passes, close the incident.
- Both sides have istio-proxy → Proceed to Step 3.

---

## Step 3 — Is the Sidecar Healthy?

An injected sidecar can still be unhealthy — stuck in a bad xDS state, not connected to istiod, or failing to process config updates.

```bash
# Check Envoy readiness
kubectl exec <pod-name> -n <namespace> -c istio-proxy -- curl -s localhost:15000/ready
# Expected: LIVE
# Problem: anything other than LIVE means Envoy is not ready to handle traffic

# Check Envoy's xDS sync status with istiod
# This shows whether Envoy has received up-to-date configuration from the control plane
istioctl proxy-status <pod-name>.<namespace>
# Columns: NAME, CDS, LDS, EDS, RDS, ECDS, ISTIOD, VERSION
# Expected: all columns show SYNCED
# Problem: STALE = istiod sent a config update but Envoy hasn't acknowledged it
# Problem: NOT SENT = istiod has not pushed any config to this Envoy (usually a new pod)
# Problem: ERROR = Envoy rejected the config update

# Scan Envoy logs for errors
kubectl logs <pod-name> -n <namespace> -c istio-proxy --tail=100
# Look for: "error", "fail", "rejected", "timeout", "disconnect"
```

**If Envoy is not LIVE or shows STALE/ERROR in proxy-status**:

```bash
# Investigate istiod — the control plane may be unable to push config
kubectl get pods -n istio-system
# Check istiod is Running

kubectl logs -n istio-system -l app=istiod --tail=100
# Look for: "failed to push xDS", "connection timeout", "resource conflict"

# Check if istiod is overloaded (too many connected Envoys, resource exhaustion)
kubectl top pods -n istio-system

# If istiod looks sick: restart it
kubectl rollout restart deployment/istiod -n istio-system
kubectl rollout status deployment/istiod -n istio-system
# Then re-check proxy-status — should resolve to SYNCED within 2-3 minutes
```

**Decision**:
- Envoy not ready or stuck STALE → Fix istiod/Envoy connectivity issue. Retest after sync resolves.
- Envoy is LIVE and SYNCED → Proceed to Step 4.

## Step 4 — Is It an mTLS Certificate Error?

mTLS cert problems produce distinctive error strings. Check these first before diving into routing config.

```bash
# Scan Envoy access logs for cert error signatures
kubectl logs <pod-name> -n <namespace> -c istio-proxy | \
  grep -iE "certificate|ssl|tls|CERTIFICATE_VERIFY|handshake" | tail -20

# Check the workload cert held by the source pod
istioctl proxy-config secret <source-pod-name>.<source-namespace>
# Column: DAYS UNTIL EXPIRE — negative means expired
# Column: STATUS — "ACTIVE" is expected

# Check mTLS mode configuration between source and destination
istioctl authn tls-check <source-pod-name>.<source-namespace> \
  <dest-service>.<dest-namespace>.svc.cluster.local
# Expected output: mTLS: OK
# Problem output: "CONFLICT" means one side is STRICT and the other is PERMISSIVE
# Problem output: "DISABLE" means one or both sides have mTLS disabled
```

**Common cert error patterns**:

| Error String | Root Cause | Fix |
|---|---|---|
| `CERTIFICATE_VERIFY_FAILED` | Workload cert expired | `kubectl rollout restart deployment/<name> -n <ns>` to force cert re-issue |
| `SSL_HANDSHAKE_FAILED` | mTLS mode mismatch (STRICT vs PERMISSIVE) | Align PeerAuthentication on both sides |
| `TLS handshake failed` | CA cert mismatch (source and dest have certs from different CAs) | Usually post-CA-rotation — restart all pods to get new CA-signed certs |
| No cert in `proxy-config secret` | CSR signing failed — istiod CA unhealthy | See mtls-certificate-rotation-playbook.md Phase 4 |

**Decision**:
- Cert error confirmed → Use mtls-certificate-rotation-playbook.md to resolve. Do not proceed to Step 5 until cert issue is fixed.
- No cert errors → Proceed to Step 5.

## Step 5 — Is It an Authorization Policy Deny?

Authorization policies are intentionally silent — they deny and log, but do not send a meaningful error to the caller (the caller sees a 403 or a 503 depending on the policy). This is correct security behavior but makes debugging non-obvious.

```bash
# Look for RBAC deny in Envoy access logs — this is definitive
kubectl logs <pod-name> -n <namespace> -c istio-proxy | grep "RBAC: access denied"

# List all AuthorizationPolicies in the destination namespace
kubectl get authorizationpolicies -n <target-namespace>

# Check if there is a deny-all (default-deny) policy — common in security-hardened clusters
kubectl get authorizationpolicies -n <target-namespace> -o yaml | grep -A3 "action: DENY"

# If you find a deny-all, check whether an ALLOW rule exists for the source
kubectl get authorizationpolicies -n <target-namespace> -o yaml

# Use Istio's config analyzer to find policy issues without manual YAML reading
istioctl analyze -n <target-namespace>
# Output: lists warnings and errors in mesh config, including policy conflicts
```

**If RBAC deny is confirmed**:

```bash
# Step 1: Identify the source service account
kubectl get pod <source-pod-name> -n <source-namespace> \
  -o jsonpath='{.spec.serviceAccountName}'

# Step 2: Find the AuthorizationPolicy on the destination
kubectl get authorizationpolicies -n <target-namespace> -o yaml

# Step 3: Add the missing ALLOW rule
# The principal format is: cluster.local/ns/<namespace>/sa/<serviceaccount>
# Example fix — add the source service account to the ALLOW policy:
```

```yaml
# kubectl edit authorizationpolicy <policy-name> -n <target-namespace>
# Add under spec.rules[].from[].source.principals:
- "cluster.local/ns/<source-namespace>/sa/<source-service-account>"
```

```bash
# Step 4: Verify the fix — retest the connection
# The policy change takes effect within seconds — no restart required
```

**Decision**:
- RBAC deny confirmed and fixed → Retest. If connectivity restored, close the incident. Document the missing rule in your service's AuthorizationPolicy.
- No RBAC deny → Proceed to Step 6.

## Step 6 — Is It a Routing Misconfiguration?

VirtualService and DestinationRule misconfigurations are the most common mesh-layer bugs for teams learning Istio. These configs are YAML and case-sensitive — a single typo causes silent failures.

```bash
# Run Istio's config analyzer — this is the fastest path to routing misconfigs
istioctl analyze -n <namespace>
# Common findings:
#   "VirtualService refers to non-existent destination host"
#   "DestinationRule subset <name> does not have a matching pod"
#   "Gateway references non-existent Secret"
# These are your bugs — fix them and retest

# Inspect the VirtualService for the destination service
kubectl get virtualservice -n <namespace>
kubectl describe virtualservice <vs-name> -n <namespace>
# Check: host field matches the actual Kubernetes Service name exactly (case-sensitive)
# Check: port in route matches the port in the Service spec
# Check: subset names in route match subsets defined in DestinationRule

# Inspect the DestinationRule
kubectl get destinationrule -n <namespace> -o yaml
# Check: host matches the Kubernetes Service name
# Check: subset labels match pod labels exactly (e.g., "version: v1" vs "Version: v1" — different)
kubectl get pods -n <namespace> --show-labels
# Compare subset labels in DestinationRule against actual pod labels

# Trace Envoy's view of the destination — does Envoy know the service exists?
istioctl proxy-config listeners <source-pod-name>.<source-namespace>
# Look for a listener on the destination's port

istioctl proxy-config routes <source-pod-name>.<source-namespace>
# Look for a route entry for the destination service hostname

istioctl proxy-config clusters <source-pod-name>.<source-namespace> | grep <dest-service>
# If the destination cluster is absent: Envoy doesn't know about it
# Causes: ServiceEntry missing for external service, or VirtualService host mismatch

istioctl proxy-config endpoints <source-pod-name>.<source-namespace> | grep <dest-service>
# Check: are healthy pod IPs listed under the destination cluster?
# If endpoint list is empty: Envoy has no healthy backends to route to
```

**Common routing misconfigs and their fixes**:

| Symptom | Cause | Fix |
|---|---|---|
| Cluster missing from `proxy-config clusters` | VirtualService host doesn't match Service name | Fix the `host:` field in VirtualService |
| Endpoints empty for a subset | DestinationRule subset labels don't match pod labels | Fix labels in DestinationRule `subsets[]` or in pod spec |
| External service unreachable | No ServiceEntry for the external hostname | Create a ServiceEntry for the external domain |
| Wrong port in route | Port in DestinationRule doesn't match Service port | Align port numbers |

**Decision**:
- Routing misconfiguration found and fixed → Retest. If connectivity restored, update any related documentation for the misconfigured resource.
- No routing misconfiguration found → Proceed to Step 7.

## Step 7 — Is It a Network Issue Below the Mesh?

If all mesh-layer checks pass but traffic still fails, the problem is below the mesh. Isolate the mesh layer to confirm.

```bash
# Step 1: Temporarily set the destination namespace to PERMISSIVE mTLS
# PERMISSIVE allows both plaintext and mTLS — this lets non-mesh clients connect
# If traffic starts working in PERMISSIVE, the problem is in the mTLS layer (you missed something in Steps 4-6)
# If traffic STILL fails in PERMISSIVE, the problem is below the mesh

cat <<EOF | kubectl apply -f -
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: debug-permissive
  namespace: <target-namespace>
spec:
  mtls:
    mode: PERMISSIVE
EOF

# Step 2: Retest the connection — does it work now?

# Step 3: If still failing — check Kubernetes NetworkPolicies
kubectl get networkpolicies -n <source-namespace>
kubectl get networkpolicies -n <target-namespace>
# Check: is there an ingress policy on the target namespace that doesn't allow the source?
# NetworkPolicy is evaluated before the mesh — if NetworkPolicy denies the traffic, Envoy never sees it

# Step 4: If no NetworkPolicy issue — check EKS security groups
# In EKS, nodes communicate via VPC security groups
# Check: source node's security group allows outbound on the destination port
# Check: destination node's security group allows inbound from source node's security group
# This requires AWS Console or CLI — check EC2 security group rules for your node groups

# Step 5: Try a direct connectivity test from within the sidecar proxy
# (This bypasses the app and tests Envoy's ability to reach the destination)
kubectl exec <source-pod> -n <source-namespace> -c istio-proxy -- \
  curl -v --max-time 5 \
  <dest-service>.<dest-namespace>.svc.cluster.local:<port>/healthz

# IMPORTANT: Always clean up the debug PERMISSIVE policy when done
# Leaving it in place weakens your mTLS posture
kubectl delete peerauthentication debug-permissive -n <target-namespace>
```

**Decision tree outcome**:
- Traffic works in PERMISSIVE: Go back to Steps 4–6, you missed a cert or mTLS mode issue
- Traffic fails in PERMISSIVE, but curl from sidecar works: App-level issue (wrong endpoint, missing header)
- Traffic fails in PERMISSIVE AND curl from sidecar fails: Infrastructure layer — NetworkPolicy, security group, or VPC routing

## Escalation Criteria

Escalate to the platform/mesh team (and potentially open an Istio GitHub issue) if:

- istiod is in CrashLoopBackOff and a clean restart does not resolve it
- xDS push failures are affecting more than 50% of Envoy sidecars (visible in `istioctl proxy-status` output)
- istiod CA cert has expired — this is a multi-step rotation requiring coordinated cluster-wide pod restarts (see mtls-certificate-rotation-playbook.md Phase 4)
- Suspected Istio version bug — check `github.com/istio/istio/issues` against your Istio version before escalating internally
- Traffic routing is correct per all Istio tools but packets are still not arriving (indicates a potential Envoy data plane bug — rare but real)

## Quick Reference: istioctl Diagnostic Commands

| Command | What it shows | When to use |
|---|---|---|
| `istioctl proxy-status <pod>.<ns>` | xDS sync state for all config types | First check for Envoy config issues |
| `istioctl proxy-config listeners <pod>.<ns>` | What ports Envoy is listening on | Routing/port issues |
| `istioctl proxy-config routes <pod>.<ns>` | HTTP routing table in Envoy | VirtualService not taking effect |
| `istioctl proxy-config clusters <pod>.<ns>` | Upstream services Envoy knows | Missing ServiceEntry, wrong host |
| `istioctl proxy-config endpoints <pod>.<ns>` | Healthy pod IPs per cluster | Circuit breaker, no healthy upstream |
| `istioctl proxy-config secret <pod>.<ns>` | Cert details for the pod | mTLS cert expiry, rotation issues |
| `istioctl authn tls-check <pod>.<ns> <svc>` | mTLS mode between source and dest | mTLS CONFLICT, DISABLE |
| `istioctl analyze -n <ns>` | Config validation for the namespace | VirtualService/DestinationRule bugs |
