# Envoy Filter Patterns Reference

## How to Use This Reference
EnvoyFilter is the escape hatch — use it when Istio's VirtualService, DestinationRule, AuthorizationPolicy, and RequestAuthentication cannot express what you need. Each pattern here includes: the use case, when to reach for it, the working EnvoyFilter YAML with inline explanation, and the most common gotcha.

**Warning before using any EnvoyFilter**:
EnvoyFilter patches Envoy's internal configuration directly. Unlike VirtualService and DestinationRule, EnvoyFilter is:
- **Version-sensitive** — Envoy's internal config format changes between Istio releases. A working EnvoyFilter on Istio 1.18 may silently fail or crash Envoy on Istio 1.21.
- **Not validated before application** — istiod does minimal structural validation. A bad patch can silently fail (Envoy rejects the xDS push) or crash the Envoy process.
- **Hard to debug** — no kubectl describe output tells you whether the filter is active. Use `pilot_total_xds_rejects` in Prometheus and `istioctl proxy-config` to verify.

Default rule: if Istio has a native CRD for what you need (CORS → VirtualService `corsPolicy`, JWT → RequestAuthentication, rate limiting → consider external RLS), use the CRD. Reach for EnvoyFilter only when the CRD doesn't cover the requirement.

## Pattern 1: Local Rate Limiting

**Use case**: Protect a service from being overwhelmed by too many requests — any client, any source. "Notification-service can handle 1,000 req/sec max per pod — return 429 for anything beyond that."

**When to reach for it**:
- Protecting a downstream service from upstream retry storms
- Implementing per-service request throttling without a dedicated rate-limit service
- Complementing circuit breaking: circuit breaking handles error rates; local rate limiting handles request volume
- During an incident: you need to shed load immediately without deploying new application code

**How it works**: Envoy's Local Rate Limit filter runs inside each Envoy sidecar using a token bucket algorithm. Because the limit is per-pod (not global), the effective cluster-wide rate limit is `limit × replica count`. For a true global rate limit across all pods of a service, you need Pattern 1b — the External Rate Limit Service (RLS).

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: notification-service-local-ratelimit
  namespace: notification
spec:
  workloadSelector:
    labels:
      app: notification-service         # Target: only notification-service pods
  configPatches:
  - applyTo: HTTP_FILTER                # Patch: add a filter to the HTTP filter chain
    match:
      context: SIDECAR_INBOUND          # Apply on: inbound traffic INTO this service
      listener:
        filterChain:
          filter:
            name: "envoy.filters.network.http_connection_manager"
                                        # Target the HTTP connection manager filter chain
    patch:
      operation: INSERT_BEFORE          # Insert BEFORE the terminal router filter
      value:
        name: envoy.filters.http.local_ratelimit
        typed_config:
          "@type": type.googleapis.com/udpa.type.v1.TypedStruct
          type_url: type.googleapis.com/envoy.extensions.filters.http.local_ratelimit.v3.LocalRateLimit
          value:
            stat_prefix: local_rate_limiter
            token_bucket:
              max_tokens: 1000          # Burst capacity: 1,000 tokens maximum in the bucket
              tokens_per_fill: 1000     # Refill amount per interval
              fill_interval: 1s         # Refill every second → sustained rate: 1,000 req/sec
            filter_enabled:
              runtime_key: local_rate_limit_enabled
              default_value:
                numerator: 100
                denominator: HUNDRED    # 100% of requests subject to rate limit check
            filter_enforced:
              runtime_key: local_rate_limit_enforced
              default_value:
                numerator: 100
                denominator: HUNDRED    # 100% enforcement: no dry-run mode
            response_headers_to_add:
            - append: false
              header:
                key: x-local-rate-limit
                value: 'true'           # Attach this header to all 429 responses for visibility
```

**Gotcha 1**: Per-pod scope. With 10 notification-service pods, this allows 10,000 req/sec cluster-wide. If your limit needs to be globally enforced at 1,000 req/sec across all pods combined, this pattern is insufficient — use the Envoy Rate Limit Service (ext_authz gRPC) with a Redis backend.

**Gotcha 2**: The `filter_enabled` / `filter_enforced` split exists so you can enable dry-run mode (enabled=100%, enforced=0%) to observe rate limit hits in metrics without actually rejecting requests. Use this before enabling enforcement in production.

## Pattern 1b: Global Rate Limiting (External RLS Reference)

**Use case**: Global rate limiting across all pods of a service — "1,000 req/sec total for the entire notification-service fleet, not per pod."

**How it works**: Replace the local rate limit filter with the `ext_authz` filter pointing to an Envoy Rate Limit Service (RLS) gRPC endpoint backed by Redis. The RLS maintains a shared counter in Redis; all pod sidecars consult it on every request.

**Architecture**:
```
Client → notification-service pod sidecar
               ↓ (every request)
         Envoy ext_authz filter
               ↓ gRPC
         RLS Service (Envoy RLS implementation)
               ↓
         Redis (shared counter)
               ↓ ALLOW (200) or DENY (429)
         notification-service app container
```

This is more complex to operate (RLS + Redis become availability dependencies) but is the only correct approach for global rate limiting in a horizontally scaled service. See Pattern 6 for the ext_authz EnvoyFilter pattern.

## Pattern 2: Request and Response Header Manipulation

**Use case**: Add, modify, or remove HTTP headers on requests or responses without touching application code. "Add a correlation ID to every outbound call" or "Strip the X-Internal-Pod-ID header from responses before they reach clients."

**When to reach for it**:
- Injecting observability headers (correlation IDs, service version, trace context)
- Removing sensitive internal implementation headers from external-facing responses
- Header normalization before a legacy service expecting specific headers
- Adding static auth tokens for downstream services that require them (prefer secrets rotation over hardcoded values — use environment variable injection instead of hardcoded header values here)

**Note**: For simple static headers, VirtualService `headers` field is sufficient. Use EnvoyFilter only when you need dynamic header values from Envoy metadata, or when you need response header manipulation that VirtualService doesn't support.

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: message-router-header-injection
  namespace: notification
spec:
  workloadSelector:
    labels:
      app: message-router-service
  configPatches:
  - applyTo: HTTP_ROUTE                 # Apply at the HTTP route level (per-route headers)
    match:
      context: SIDECAR_OUTBOUND         # Apply to outbound requests FROM message-router
    patch:
      operation: MERGE
      value:
        request_headers_to_add:
        - header:
            key: x-source-service
            value: "message-router"     # Static value: identify the calling service
          append: false                 # false = replace existing header if present
                                        # true = append (allows multiple values, avoid for identity headers)
        - header:
            key: x-mesh-version
            value: "%UPSTREAM_METADATA([\"envoy.lb\", \"version\"])%"
                                        # Dynamic: reads endpoint metadata from Envoy's LB info
          append: false
        response_headers_to_remove:
        - x-internal-debug-info         # Strip this internal header before returning to caller
        - x-backend-server-id           # Don't expose which pod handled the request
```

**Gotcha 1**: HTTP/2 (used by gRPC) requires lowercase header names per spec. Envoy automatically lowercases headers for HTTP/2 traffic. If you test header injection with an HTTP/1.1 client and it works, but fails with a gRPC client, check the header case — `X-Source-Service` becomes `x-source-service` in HTTP/2, and some services handle these differently.

**Gotcha 2**: `append: false` is usually what you want for identity and routing headers. `append: true` adds an additional header value without removing the existing one — useful for list-style headers (`Via`, `X-Forwarded-For`) but wrong for single-value headers.

## Pattern 3: CORS Policy

**Use case**: Configure Cross-Origin Resource Sharing for browser-facing APIs without modifying application code.

**When to reach for it**: Browser JavaScript calling your API directly from a different origin. Not needed for service-to-service calls — CORS is a browser-enforced security mechanism, not a mesh security control.

**Important**: Istio's VirtualService has a native `corsPolicy` field. Use that first. Only reach for EnvoyFilter when you need per-route CORS that VirtualService cannot express (e.g., different CORS config for different route match conditions in the same VirtualService).

```yaml
# PREFERRED APPROACH: VirtualService native corsPolicy (simpler, Istio-validated)
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: notification-api
  namespace: notification
spec:
  hosts:
  - notification-api
  gateways:
  - notification-gateway
  http:
  - corsPolicy:
      allowOrigins:
      - exact: "https://portal.t-mobile.com"
                                        # Exact match: only this origin
      - regex: "https://.*\\.t-mobile\\.com"
                                        # Regex: any T-Mobile subdomain over HTTPS
      allowMethods:
      - GET
      - POST
      - PUT
      - DELETE
      - OPTIONS                         # OPTIONS must be included for pre-flight requests
      allowHeaders:
      - Authorization
      - Content-Type
      - X-Request-ID
      - X-Correlation-ID
      exposeHeaders:
      - X-Request-ID                    # Headers the browser is allowed to read from the response
      maxAge: "24h"                     # How long browsers can cache pre-flight responses
      allowCredentials: true            # Allow cookies and auth headers cross-origin
    route:
    - destination:
        host: notification-api
        port:
          number: 8080
```

**Gotcha 1**: `allowCredentials: true` and `allowOrigins` wildcard (`*`) are mutually exclusive. Browsers reject credentialed cross-origin requests to wildcard origins (this is a browser-enforced W3C spec requirement). Always use explicit origin lists when `allowCredentials: true`.

**Gotcha 2**: The `OPTIONS` pre-flight request must be included in `allowMethods`. Browsers send OPTIONS before any credentialed cross-origin request — if OPTIONS isn't allowed, the pre-flight fails and the actual request never sends.

## Pattern 4: JWT Authentication and Enforcement

**Use case**: Validate JWT tokens on inbound requests before they reach your service. Reject requests with missing or invalid tokens.

**When to reach for it**: API endpoints requiring a valid JWT from an identity provider (Auth0, Okta, AWS Cognito, internal token service).

**Important**: Prefer Istio's native RequestAuthentication + AuthorizationPolicy pair over EnvoyFilter for JWT. The native resources are validated, version-stable, and easier to debug. Use EnvoyFilter only for JWT scenarios requiring custom filter behavior not expressible in RequestAuthentication (e.g., reading the JWT from a non-standard location, chaining multiple JWT validators with complex fallback logic).

```yaml
# STEP 1: Validate JWT signatures and extract claims
apiVersion: security.istio.io/v1beta1
kind: RequestAuthentication
metadata:
  name: notification-api-jwt
  namespace: notification
spec:
  selector:
    matchLabels:
      app: notification-api
  jwtRules:
  - issuer: "https://auth.t-mobile.com"
                                        # JWT 'iss' claim must match exactly
    jwksUri: "https://auth.t-mobile.com/.well-known/jwks.json"
                                        # Istio fetches + caches public keys for signature validation
    audiences:
    - "notification-platform"           # JWT 'aud' claim must contain this value
    forwardOriginalToken: true          # true = pass Authorization header to the app after validation
                                        # false = Envoy validates but strips the token (safer for backends)
    fromHeaders:
    - name: Authorization
      prefix: "Bearer "                 # Standard: Authorization: Bearer <token>
```

```yaml
# STEP 2: ENFORCE rejection of requests without a valid JWT
# RequestAuthentication alone does NOT reject requests with no JWT.
# Without this AuthorizationPolicy, requests with no Authorization header pass through.
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: require-valid-jwt
  namespace: notification
spec:
  selector:
    matchLabels:
      app: notification-api
  action: ALLOW
  rules:
  - from:
    - source:
        requestPrincipals: ["*"]        # Allow any request that has a validated JWT principal
                                        # Format of requestPrincipal: "<issuer>/<subject>"
                                        # Wildcard "*" means: any issuer/subject combination,
                                        # but the JWT must have been validated by RequestAuthentication
                                        # Requests with no JWT have no requestPrincipal → not ALLOWED → rejected
```

**Gotcha**: The two-resource pattern (RequestAuthentication + AuthorizationPolicy requiring `requestPrincipals`) is widely misunderstood. New practitioners assume RequestAuthentication alone secures the endpoint. It does not. RequestAuthentication is a validator — if no token is present, there is nothing to validate, and the request flows through. The AuthorizationPolicy is the gatekeeper that enforces presence of a validated identity. This is by design (supports gradual rollout) but bites teams repeatedly.

## Pattern 5: gRPC-JSON Transcoding

**Use case**: Expose a gRPC service as a REST/JSON API without changing service code. Envoy translates HTTP/JSON requests into gRPC/protobuf and translates gRPC responses back to JSON.

**When to reach for it**:
- Browser clients that cannot call gRPC directly (browsers don't support HTTP/2 trailers, which gRPC requires)
- Legacy REST clients that need to call a gRPC service
- API gateway layer that needs to expose a single REST interface over internal gRPC services

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: grpc-json-transcoder
  namespace: notification
spec:
  workloadSelector:
    labels:
      app: notification-grpc-service
  configPatches:
  - applyTo: HTTP_FILTER
    match:
      context: SIDECAR_INBOUND          # Intercept inbound REST calls, transcode to gRPC for the app
      listener:
        filterChain:
          filter:
            name: "envoy.filters.network.http_connection_manager"
    patch:
      operation: INSERT_BEFORE          # Must be before the router filter
      value:
        name: envoy.filters.http.grpc_json_transcoder
        typed_config:
          "@type": type.googleapis.com/envoy.extensions.filters.http.grpc_json_transcoder.v3.GrpcJsonTranscoder
          proto_descriptor_bin: "<base64-encoded-proto-descriptor-set>"
                                        # Generate: protoc --descriptor_set_out=service.pb \
                                        #               --include_imports \
                                        #               --include_source_info \
                                        #               notification.proto
                                        # Encode: base64 -w0 service.pb
                                        # Consider mounting as ConfigMap volume and using proto_descriptor file path instead
          services:
          - "notification.v1.NotificationService"
                                        # Fully qualified gRPC service name from .proto file:
                                        # package notification.v1; service NotificationService {}
          print_options:
            add_whitespace: true        # Pretty-print JSON responses (disable in prod for efficiency)
            always_print_primitive_fields: true
                                        # Include fields with default values in JSON (0, "", false)
                                        # Without this, proto3 default values are omitted from JSON
            preserve_proto_field_names: false
                                        # false = camelCase JSON (e.g., "notificationId") — proto3 default
                                        # true = snake_case JSON (e.g., "notification_id") — matches .proto fields
          ignore_unknown_query_parameters: true
                                        # Don't reject requests with query params not in the proto
```

**Gotcha 1**: The proto descriptor must stay in sync with the deployed service. When you update the `.proto` file and deploy a new service version, the descriptor in the EnvoyFilter must also be updated — or transcoding breaks with `NOT_FOUND` for new endpoints or `INVALID_ARGUMENT` for changed field types. Recommend: store the descriptor as a Kubernetes ConfigMap, mount it as a file, and use `proto_descriptor` (file path) instead of `proto_descriptor_bin` (inline base64). Automate descriptor updates in your CI pipeline alongside service deployment.

**Gotcha 2**: The transcoder works at the sidecar level. If you are running the Istio Ingress Gateway and want transcoding at the edge (before traffic enters the mesh), you need to apply this EnvoyFilter with `context: GATEWAY` and `workloadSelector` targeting the gateway pod labels, not the service pod labels.

## Pattern 6: External Authorization (OPA Integration)

**Use case**: Delegate authorization decisions to an external policy engine (Open Policy Agent, a custom gRPC auth service). Every request is intercepted by Envoy and checked against OPA before being forwarded to the service.

**When to reach for it**:
- Complex authorization logic that cannot be expressed in AuthorizationPolicy (attribute-based access control, database lookups for permissions, multi-tenancy rules)
- Centralized policy enforcement across many services with a single policy repository (OPA Gatekeeper pattern)
- Compliance requirements that need a full external audit trail of every authorization decision
- Dynamic policy updates without redeploying Istio resources (OPA policy changes are hot-loaded)

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: opa-ext-authz-notification
  namespace: notification
spec:
  workloadSelector:
    labels:
      app: notification-api
  configPatches:
  - applyTo: HTTP_FILTER
    match:
      context: SIDECAR_INBOUND
      listener:
        filterChain:
          filter:
            name: "envoy.filters.network.http_connection_manager"
    patch:
      operation: INSERT_BEFORE
      value:
        name: envoy.filters.http.ext_authz
        typed_config:
          "@type": type.googleapis.com/envoy.extensions.filters.http.ext_authz.v3.ExtAuthz
          grpc_service:
            envoy_grpc:
              cluster_name: outbound|9191||opa-service.notification.svc.cluster.local
                                        # OPA sidecar or centralized OPA service endpoint
                                        # OPA must implement envoy.service.auth.v3.Authorization gRPC API
                                        # Deploy OPA with --plugin-dir envoy and --set plugins.envoy.v1.addr=:9191
            timeout: 500ms              # Hard limit: if OPA takes > 500ms, trigger failure mode
          failure_mode_allow: false     # false = DENY request if OPA is unreachable (fail-closed)
                                        # true = ALLOW request if OPA is unreachable (fail-open)
                                        # Security teams require fail-closed; availability teams want fail-open
                                        # Know your compliance requirement before setting this
          include_peer_certificate: true
                                        # Send the caller's mTLS X.509 cert DER to OPA in CheckRequest
                                        # Enables identity-aware policy: OPA can inspect the SPIFFE identity
          with_request_body:
            max_request_bytes: 8192     # Send first 8KB of request body to OPA
                                        # Required only if OPA policy inspects request body content
                                        # Increases latency and OPA load — only enable if needed
          status_on_error:
            code: 503                   # Return 503 (not 403) when OPA is unreachable
                                        # Distinguishes "auth service down" from "authorization denied"
```

**Gotcha 1**: `failure_mode_allow: false` means OPA unavailability = service unavailability. If OPA goes down (network partition, OOM, upgrade), all traffic to the protected service returns 503. This makes OPA availability part of your notification platform's SLA. Run OPA as a HA deployment (3+ replicas with anti-affinity) and monitor OPA pod availability and latency separately from application metrics.

**Gotcha 2**: Every request to the protected service now has an additional gRPC call to OPA. At 25M msg/day on the notification platform, OPA handles 290+ req/sec of auth checks. Monitor OPA latency with `ext_authz.check_call.duration_ms` histogram. If OPA latency exceeds 100ms at P99, consider deploying OPA as a sidecar per pod (rather than a centralized service) to eliminate network hop latency.

**Gotcha 3**: OPA policy changes need testing before deployment. A misconfigured Rego policy that always returns `deny` takes down the protected service. Use OPA's `--dry-run` flag and integration tests against the OPA `/v1/data` HTTP API before deploying policy changes.

## Pattern 7: Lua Filter for Custom Logic

**Use case**: Execute custom Lua scripts in Envoy's request/response processing path without writing a native Envoy filter in C++. For lightweight, one-off business logic, header transformations, or request validation that cannot be expressed in static configuration.

**When to reach for it**:
- Header transformations too dynamic for static config (e.g., compute a header value from other headers)
- Lightweight request validation (check for a required query parameter, validate a custom token format)
- Emergency behavioral hotfix without deploying new application code — Lua can be changed and applied without restarting pods (via EnvoyFilter update, which is picked up within seconds via xDS)
- Prototyping custom filter logic before investing in a native Envoy filter in C++

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: lua-correlation-id-validator
  namespace: notification
spec:
  workloadSelector:
    labels:
      app: notification-service
  configPatches:
  - applyTo: HTTP_FILTER
    match:
      context: SIDECAR_INBOUND
      listener:
        filterChain:
          filter:
            name: "envoy.filters.network.http_connection_manager"
    patch:
      operation: INSERT_BEFORE
      value:
        name: envoy.filters.http.lua
        typed_config:
          "@type": type.googleapis.com/envoy.extensions.filters.http.lua.v3.LuaPerRoute
          inline_code: |
            -- envoy_on_request: runs on every inbound request BEFORE forwarding to the app
            function envoy_on_request(request_handle)
              -- Validate required correlation ID header
              local correlation_id = request_handle:headers():get("x-correlation-id")
              
              if correlation_id == nil or correlation_id == "" then
                -- Reject immediately: return 400 with explanation
                -- After respond(), Lua execution stops and the request is not forwarded
                request_handle:respond(
                  {[":status"] = "400", ["content-type"] = "application/json"},
                  '{"error":"MISSING_HEADER","detail":"x-correlation-id is required"}'
                )
                return
              end
              
              -- Validate format: correlation ID must be a UUID (8-4-4-4-12 hex pattern)
              local uuid_pattern = "^%x%x%x%x%x%x%x%x%-%x%x%x%x%-%x%x%x%x%-%x%x%x%x%-%x%x%x%x%x%x%x%x%x%x%x%x$"
              if not string.match(correlation_id, uuid_pattern) then
                request_handle:respond(
                  {[":status"] = "400", ["content-type"] = "application/json"},
                  '{"error":"INVALID_HEADER","detail":"x-correlation-id must be a UUID"}'
                )
                return
              end
              
              -- Annotate the request for downstream observability
              request_handle:headers():add("x-processed-by-envoy", "true")
              request_handle:headers():add("x-envoy-timestamp", tostring(os.time()))
            end
            
            -- envoy_on_response: runs on every response BEFORE returning to the caller
            function envoy_on_response(response_handle)
              -- Remove internal implementation detail headers from all responses
              response_handle:headers():remove("x-internal-pod-id")
              response_handle:headers():remove("x-internal-build-hash")
            end
```

**Gotcha 1**: Lua in Envoy uses a coroutine execution model. Blocking operations — synchronous HTTP calls, file I/O, `os.execute()`, any call that doesn't return immediately — are not supported and will stall the Envoy worker thread. Stalled worker threads block ALL traffic on that pod, not just the request that triggered the stall. If you need to make an HTTP call from Lua (e.g., to look up a user's permissions), you must use Envoy's async HTTP call API (`request_handle:httpCall()`), which is asynchronous and non-blocking. This API is more complex to write but critical for production-safe Lua filters.

**Gotcha 2**: Test Lua regex patterns carefully. Lua uses its own pattern matching syntax (`%x` for hex digit, `%d` for digit), not POSIX or PCRE regex. A pattern that validates correctly in a PCRE test tool will not work as-is in Envoy Lua. Test Lua patterns in a Lua REPL (`lua -e "print(string.match(...))"`) before embedding in a filter.

**Gotcha 3**: Lua filter errors (syntax errors, runtime panics) cause Envoy to return 500 for affected requests and log errors to the Envoy container stderr. Monitor Envoy logs after applying any Lua filter: `kubectl logs <pod> -c istio-proxy | grep -i lua`.

## Pattern 8: Connection-Level TCP Filter

**Use case**: Inspect or manipulate traffic at the TCP connection level (Layer 4) before Envoy's HTTP processing. Use for non-HTTP protocols, raw TCP services, or adding connection-level metadata.

**When to reach for it**:
- TCP services (databases, message brokers, raw sockets) where HTTP filter chain doesn't apply
- Adding connection-level rate limiting or connection count limits independent of HTTP
- Injecting connection metadata for observability

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: kafka-connection-limit
  namespace: kafka
spec:
  workloadSelector:
    labels:
      app: kafka-broker
  configPatches:
  - applyTo: NETWORK_FILTER             # TCP-level filter, not HTTP filter
    match:
      context: SIDECAR_INBOUND
      listener:
        portNumber: 9092                # Kafka port
        filterChain:
          filter:
            name: "envoy.filters.network.tcp_proxy"
    patch:
      operation: INSERT_BEFORE
      value:
        name: envoy.filters.network.connection_limit
        typed_config:
          "@type": type.googleapis.com/envoy.extensions.filters.network.connection_limit.v3.ConnectionLimit
          stat_prefix: kafka_connection_limit
          max_connections: 500          # Max concurrent TCP connections to this broker pod
          delay: 1s                     # Delay new connections when at limit (instead of immediate drop)
```

**Gotcha**: TCP filter patches use `applyTo: NETWORK_FILTER` and target `envoy.filters.network.tcp_proxy`, not the HTTP connection manager. Using `applyTo: HTTP_FILTER` for a TCP service is a silent no-op — the HTTP filter chain doesn't exist for pure TCP listeners.

## EnvoyFilter Debugging Reference

When an EnvoyFilter is not applying or not working:

### Step 1: Check if istiod rejected the xDS push
```bash
# Look for xDS rejection metrics in Prometheus
kubectl exec -n istio-system deploy/istiod -- \
  curl -s localhost:15014/metrics | grep pilot_total_xds_rejects

# Non-zero value means Envoy rejected a config push from istiod
# Check istiod logs for the rejection reason
kubectl logs -n istio-system -l app=istiod --tail=100 | grep -i "rejected\|invalid\|nack"
```

### Step 2: Verify the filter is in the Envoy config dump
```bash
# List all listeners on a pod to confirm the filter chain is present
istioctl proxy-config listeners <pod-name>.<namespace> --output json

# Search the full config dump for your filter
kubectl exec <pod> -n <namespace> -c istio-proxy -- \
  curl -s localhost:15000/config_dump | \
  python3 -c "
import sys, json
d = json.load(sys.stdin)
import re
text = json.dumps(d, indent=2)
# Find your filter by name
lines = text.split('\n')
for i, line in enumerate(lines):
    if 'local_ratelimit' in line or 'ext_authz' in line or 'lua' in line:
        print(f'Line {i}: {line}')
"
```

### Step 3: Use istioctl for quick listener inspection
```bash
# Check inbound listeners on the target pod
istioctl proxy-config listeners <pod>.<namespace> \
  --port 8080 \
  --output json | python3 -m json.tool | grep -A5 "filter_name\|name"

# Check Envoy stats for your filter (rate limit, ext_authz, lua)
kubectl exec <pod> -n <namespace> -c istio-proxy -- \
  curl -s localhost:15000/stats | grep "local_rate_limit\|ext_authz\|lua"
```

### Step 4: Check Envoy proxy logs for filter-level errors
```bash
# Enable debug logging temporarily for a specific logger
kubectl exec <pod> -n <namespace> -c istio-proxy -- \
  curl -s -X POST "localhost:15000/logging?lua=debug"

# Then tail the proxy logs and reproduce the request
kubectl logs <pod> -n <namespace> -c istio-proxy -f | grep -i "lua\|rate_limit\|ext_authz\|filter"

# Reset log level to warning after debugging
kubectl exec <pod> -n <namespace> -c istio-proxy -- \
  curl -s -X POST "localhost:15000/logging?lua=warning"
```

### Step 5: Verify EnvoyFilter match context
The most common silent failure is an EnvoyFilter with the wrong `context` value:

| Traffic direction | Correct context |
|-----------------|----------------|
| Protecting a service from inbound callers | `SIDECAR_INBOUND` |
| Modifying outbound calls FROM a service | `SIDECAR_OUTBOUND` |
| Modifying traffic at the Istio ingress/egress gateway | `GATEWAY` |
| Apply everywhere | `ANY` (use sparingly — applies to gateways too) |

Applying an `INBOUND` filter with `SIDECAR_OUTBOUND` context results in a no-op with no error.

4. Run a synthetic request through each affected service and verify behavior
5. Check Envoy stats specific to each filter (rate limit counters, ext_authz check counts)
