# Istio Resource Quick Reference

## How to Use This Reference
This is a lookup table, not a reading document. When you encounter an Istio resource type in YAML, incident logs, or an interview, find it here for the one-paragraph summary plus the most common mistake teams make.

**API Groups**:
- `networking.istio.io/v1alpha3` or `v1beta1` ‚Äî traffic management resources
- `security.istio.io/v1beta1` ‚Äî security resources
- `telemetry.istio.io/v1alpha1` ‚Äî telemetry configuration
- `extensions.istio.io/v1alpha1` ‚Äî advanced Envoy customization

## Traffic Management Resources

### Gateway
- **API group**: `networking.istio.io/v1alpha3`
- **Scope**: Namespace
- **Purpose**: Configures the Istio ingress or egress gateway (a standalone Envoy proxy at the edge of the mesh). Think of it as the "listener" on the gateway pod. Separate from Kubernetes Ingress ‚Äî Gateway controls Envoy at the mesh boundary, while Ingress is a K8s-native concept that sidestep the mesh control plane.
- **Key fields**:
  - `selector`: which gateway pod this applies to (matches labels on gateway deployment, e.g., `istio: ingressgateway`)
  - `servers[].port`: protocol (HTTP, HTTPS, TLS, TCP, GRPC) and port number
  - `servers[].hosts`: which hostnames this server block applies to (e.g., `*.t-mobile.com`)
  - `servers[].tls`: TLS termination settings (SIMPLE = TLS termination at gateway, PASSTHROUGH = pass TLS through to the service unmodified)
- **Common mistake**: Configuring a Gateway without a corresponding VirtualService that binds to it. A Gateway alone opens the port but routes nothing ‚Äî you get connection refused or a 404 from Envoy because there are no route rules attached to the listener. Always pair Gateway + VirtualService.
- **Example**:
```yaml
apiVersion: networking.istio.io/v1alpha3
kind: Gateway
metadata:
  name: notification-gateway
  namespace: notification
spec:
  selector:
    istio: ingressgateway
  servers:
  - port:
      number: 443
      name: https
      protocol: HTTPS
    tls:
      mode: SIMPLE
      credentialName: notification-tls-cert    # K8s Secret with TLS cert/key
    hosts:
    - "notifications.t-mobile.com"
```

### VirtualService
- **API group**: `networking.istio.io/v1alpha3`
- **Scope**: Namespace (but can control traffic to services in other namespaces by referencing them in `hosts`)
- **Purpose**: Defines routing rules for traffic addressed to a service. Acts as a programmable routing layer on top of Kubernetes Services. Routes traffic based on host, headers, URI paths, HTTP methods, or arbitrary conditions. Also the resource where you configure retries, timeouts, fault injection, and traffic weights for canary deployments.
- **Key fields**:
  - `hosts`: the service or hostname this VS applies to. For internal mesh traffic: the K8s service short name (e.g., `notification-service`) or FQDN. For external traffic through a Gateway: the external hostname.
  - `gateways`: if this VS handles edge traffic arriving through a Gateway, list the gateway name here. Omit (or use default `mesh`) for internal service-to-service routing.
  - `http[].match`: conditions for applying this route (headers, URI prefix/exact/regex, HTTP method, query parameters)
  - `http[].route[].destination.host`: which K8s service to send matched traffic to
  - `http[].route[].destination.subset`: which DestinationRule subset (e.g., `v1`, `v2`, `canary`) to use
  - `http[].route[].weight`: percentage of traffic for this destination (all weights in a route must sum to 100)
  - `http[].retries.attempts`: how many times to retry a failed request
  - `http[].retries.retryOn`: conditions to retry on (e.g., `5xx,gateway-error,connect-failure`)
  - `http[].timeout`: per-request timeout (e.g., `10s`)
  - `http[].fault.delay`: inject artificial latency (for chaos engineering)
  - `http[].fault.abort`: inject HTTP error responses (for chaos engineering)
- **Common mistake**: Defining a subset in the VirtualService route (e.g., `subset: v2`) without a corresponding DestinationRule that declares that subset. Traffic silently fails ‚Äî Envoy receives a route to an unknown subset and returns 503. Always define subsets in DestinationRule before referencing them in VirtualService.
- **Example (canary traffic split)**:
```yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: notification-service
  namespace: notification
spec:
  hosts:
  - notification-service
  http:
  - match:
    - headers:
        x-canary-user:
          exact: "true"              # Force canary for specific header
    route:
    - destination:
        host: notification-service
        subset: v2
  - route:
    - destination:
        host: notification-service
        subset: v1
      weight: 90
    - destination:
        host: notification-service
        subset: v2
      weight: 10                     # 10% canary traffic
    retries:
      attempts: 3
      retryOn: "5xx,gateway-error"
    timeout: 5s
```

### DestinationRule
- **API group**: `networking.istio.io/v1alpha3`
- **Scope**: Namespace
- **Purpose**: Defines traffic policies applied after routing (after VirtualService has decided which destination to send to). Controls: load balancing algorithm, connection pool settings, circuit breaking (outlier detection), mTLS mode when talking to this service, and ‚Äî critically ‚Äî subset definitions (which group of pods is labeled v1, v2, canary, etc.). Without DestinationRule, you cannot use VirtualService subsets.
- **Key fields**:
  - `host`: the K8s service this DestinationRule applies to
  - `trafficPolicy.loadBalancer.simple`: ROUND_ROBIN, LEAST_CONN, RANDOM, PASSTHROUGH
  - `trafficPolicy.connectionPool.http.http1MaxPendingRequests`: max queued requests before circuit breaks
  - `trafficPolicy.connectionPool.http.http2MaxRequests`: max concurrent requests (for HTTP/2 and gRPC)
  - `trafficPolicy.connectionPool.tcp.maxConnections`: max TCP connections to this service
  - `trafficPolicy.outlierDetection.consecutiveGatewayErrors`: how many consecutive 5xx before ejecting a pod
  - `trafficPolicy.outlierDetection.interval`: how often to evaluate ejection criteria
  - `trafficPolicy.outlierDetection.baseEjectionTime`: how long to eject a failing pod
  - `trafficPolicy.outlierDetection.maxEjectionPercent`: max % of pods that can be ejected simultaneously (critical ‚Äî prevents ejecting the entire pool)
  - `trafficPolicy.tls.mode`: how Envoy talks to this destination (ISTIO_MUTUAL = mTLS using mesh certs, SIMPLE = standard TLS, DISABLE = plaintext)
  - `subsets[].name`: subset name used by VirtualService (e.g., `v1`, `v2`)
  - `subsets[].labels`: pod label selectors for this subset (e.g., `version: v1`)
- **Common mistake**: Setting `trafficPolicy.tls.mode: ISTIO_MUTUAL` on a DestinationRule targeting a service outside the mesh (no Envoy sidecar). Envoy will initiate mTLS, the destination doesn't understand it, and all traffic fails with TLS handshake errors. Use `SIMPLE` or `DISABLE` for non-mesh destinations.
- **Example (circuit breaker)**:
```yaml
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: notification-service
  namespace: notification
spec:
  host: notification-service
  trafficPolicy:
    connectionPool:
      http:
        http1MaxPendingRequests: 100
        http2MaxRequests: 1000
    outlierDetection:
      consecutiveGatewayErrors: 5      # Eject after 5 consecutive 5xx responses
      interval: 10s                    # Check every 10 seconds
      baseEjectionTime: 30s            # Eject for 30 seconds minimum
      maxEjectionPercent: 50           # Never eject more than half the pool
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
```

### ServiceEntry
- **API group**: `networking.istio.io/v1alpha3`
- **Scope**: Namespace (cluster-wide visibility when `exportTo: ["*"]` is set)
- **Purpose**: Registers an external service (outside the mesh ‚Äî a third-party API, a legacy system, an AWS RDS endpoint, a Kafka broker on a different VPC) into the mesh service registry. Without a ServiceEntry, Envoy blocks outbound traffic to unknown hosts when the mesh is in `REGISTRY_ONLY` mode. Even in `ALLOW_ANY` mode, ServiceEntry is required to apply VirtualService or DestinationRule policies to external destinations.
- **Key fields**:
  - `hosts`: the external hostname(s) to register (e.g., `api.twilio.com`, `kafka.internal.t-mobile.com`)
  - `ports`: port number and protocol of the external service (HTTP, HTTPS, TCP, TLS, GRPC, MONGO, etc.)
  - `location`: `MESH_EXTERNAL` (outside the cluster ‚Äî most common) or `MESH_INTERNAL` (inside cluster but not a K8s Service)
  - `resolution`: how Envoy resolves the host ‚Äî `DNS` (resolve at request time, most common for external), `STATIC` (fixed IP list in `endpoints`), `NONE` (pass-through, used for TCP)
  - `endpoints`: for `STATIC` resolution ‚Äî the actual IP addresses and ports
  - `exportTo`: which namespaces can see this ServiceEntry (default `*` = all namespaces)
- **Common mistake**: Forgetting to add a ServiceEntry for an external dependency when the mesh is in `REGISTRY_ONLY` mode. Services suddenly stop reaching external APIs after mesh rollout. The Envoy access log shows `BlackHoleCluster` as the upstream cluster ‚Äî this is the diagnostic signal. Traffic is being dropped because the host is not registered in the mesh registry.
- **Example**:
```yaml
apiVersion: networking.istio.io/v1alpha3
kind: ServiceEntry
metadata:
  name: twilio-sms-api
  namespace: notification
spec:
  hosts:
  - api.twilio.com
  ports:
  - number: 443
    name: https
    protocol: HTTPS
  location: MESH_EXTERNAL
  resolution: DNS
  exportTo:
  - "."                                # Only this namespace can see it
```

### Sidecar
- **API group**: `networking.istio.io/v1alpha3`
- **Scope**: Namespace
- **Purpose**: Scopes what each Envoy sidecar knows about. By default, every Envoy sidecar in a 500-service mesh downloads routing config for all 500 services ‚Äî significant memory and CPU overhead. A Sidecar resource limits which services a given workload's Envoy needs to know about. Critical for large meshes. Think of it as a firewall rules file for the Envoy config plane: it controls which service discovery entries get pushed to which Envoy instance.
- **Key fields**:
  - `workloadSelector`: which pods this Sidecar resource applies to (omit for namespace-wide)
  - `egress[].hosts`: which services this namespace's sidecars are allowed to call. Format: `<namespace>/<fqdn>` using a glob. Use `./<service>.*` for same-namespace, `<other-ns>/*` for all services in another namespace.
  - `ingress`: configures inbound listeners (usually leave default unless you need custom inbound port config)
- **Common mistake**: Forgetting to include `istio-system/*` in the egress hosts list. Envoy sidecars need to reach istiod for certificate rotation and xDS config updates. If you scope the Sidecar so tightly that istiod is unreachable, certificate renewal silently fails and mTLS breaks hours or days later when certs expire.
- **When to use**: Meshes with more than 100 services where Envoy sidecar memory consumption is a concern. Typically reduces per-sidecar memory by 40‚Äì60% in large meshes by pruning the xDS config pushed to each Envoy.
- **Example**:
```yaml
apiVersion: networking.istio.io/v1alpha3
kind: Sidecar
metadata:
  name: notification-sidecar-scope
  namespace: notification
spec:
  egress:
  - hosts:
    - "./*"                            # All services in notification namespace
    - "istio-system/*"                 # istiod, telemetry ‚Äî REQUIRED
    - "kafka/*"                        # Cross-namespace: kafka cluster
    - "monitoring/*"                   # Cross-namespace: Prometheus, Grafana
```

### EnvoyFilter
- **API group**: `networking.istio.io/v1alpha3`
- **Scope**: Namespace (or root namespace `istio-system` for mesh-wide application)
- **Purpose**: Raw Envoy configuration patches ‚Äî the escape hatch for anything Istio's higher-level resources cannot express. Can add, modify, or remove Envoy listeners, filter chains, clusters, routes, and HTTP/network filters. Use this when you need local rate limiting, Lua scripting, gRPC-JSON transcoding, external authorization, or any Envoy feature not yet surfaced in a Istio CRD.
- **Key fields**:
  - `workloadSelector`: which pods to apply this filter to (omit for namespace-wide)
  - `configPatches[].applyTo`: which part of Envoy's config to patch ‚Äî `LISTENER`, `FILTER_CHAIN`, `HTTP_FILTER`, `NETWORK_FILTER`, `CLUSTER`, `ROUTE_CONFIGURATION`, `VIRTUAL_HOST`, `HTTP_ROUTE`
  - `configPatches[].match.context`: `SIDECAR_INBOUND`, `SIDECAR_OUTBOUND`, `GATEWAY`, `ANY`
  - `configPatches[].match.listener` / `.cluster` / `.routeConfiguration`: narrow which specific Envoy config object to target
  - `configPatches[].patch.operation`: `MERGE`, `INSERT_BEFORE`, `INSERT_AFTER`, `REPLACE`, `REMOVE`, `ADD`
  - `configPatches[].patch.value`: the Envoy protobuf configuration to apply (expressed as YAML)
- **Common mistake**: EnvoyFilter is Envoy version-sensitive. Envoy's internal config format changes between Istio releases. An EnvoyFilter written for Istio 1.18 may silently fail or cause Envoy to reject the entire xDS config push on Istio 1.21. Always test EnvoyFilters in staging after every Istio upgrade. Monitor `pilot_total_xds_rejects` in Prometheus ‚Äî a spike after upgrade indicates a broken EnvoyFilter.

### WorkloadEntry
- **API group**: `networking.istio.io/v1alpha3`
- **Scope**: Namespace
- **Purpose**: Registers a non-Kubernetes workload (a VM, a bare-metal server, a legacy middleware system running outside Kubernetes) into the mesh. Makes VM traffic participate in full mesh policies ‚Äî the VM gets a SPIFFE/X.509 identity, mTLS, AuthorizationPolicy enforcement, and Envoy-based observability. The bridge between the K8s mesh world and legacy infrastructure.
- **Key fields**:
  - `address`: IP address of the VM or external workload
  - `ports`: which ports this workload exposes (name and number)
  - `labels`: labels used for routing (so DestinationRule subsets and VirtualService matches can select it alongside pods)
  - `serviceAccount`: the K8s service account identity to assign to this workload (controls what mTLS cert it receives)
  - `network`: the network name this workload is on (for multi-network mesh topologies)
- **Use case**: A legacy T-Mobile middleware system (e.g., a TIBCO broker or an RabbitMQ cluster on a VM) needs to securely call services in the mesh. Add it as a WorkloadEntry, install the Istio sidecar on the VM, and it participates in the same mTLS and AuthorizationPolicy framework as Kubernetes pods.
- **Common mistake**: Creating WorkloadEntry without also creating a ServiceEntry (or having the service registered elsewhere). The mesh doesn't know how to route *to* the VM ‚Äî WorkloadEntry registers it as a source, but the destination still needs to be discoverable.

### WorkloadGroup
- **API group**: `networking.istio.io/v1alpha3`
- **Scope**: Namespace
- **Purpose**: Template for creating WorkloadEntry resources for a fleet of VMs of the same type ‚Äî like a Kubernetes Deployment is a template for Pods. Defines common configuration (service account, labels, network, ports) that each VM instance inherits when it bootstraps and registers itself with istiod via the Istio VM auto-registration feature.
- **Key fields**:
  - `template`: the WorkloadEntry template (ports, labels, serviceAccount, network)
  - `metadata.labels`: labels applied to all WorkloadEntries auto-created from this group
  - `probe`: readiness probe configuration for the VM workloads
- **Common mistake**: Manually creating individual WorkloadEntry resources for a fleet of VMs instead of using WorkloadGroup + auto-registration. Manual WorkloadEntries become stale when VMs are replaced (new IP, same role) and require manual cleanup.

## Security Resources

### PeerAuthentication
- **API group**: `security.istio.io/v1beta1`
- **Scope**: Namespace (mesh-wide when applied to the `istio-system` namespace)
- **Purpose**: Controls how Envoy handles *incoming* connections to a workload ‚Äî whether mTLS is required (STRICT), optional (PERMISSIVE), or disabled. PERMISSIVE allows both plaintext and mTLS simultaneously ‚Äî useful during mesh rollout when not all callers have sidecars yet. STRICT rejects all plaintext ‚Äî use this as the end state after migration.
- **Key fields**:
  - `selector.matchLabels`: which pods this policy applies to (omit for namespace-wide)
  - `mtls.mode`: `STRICT` (mTLS required, plaintext rejected), `PERMISSIVE` (both allowed), `DISABLE` (mTLS disabled, plaintext only)
  - `portLevelMtls`: override mTLS mode for specific ports ‚Äî e.g., allow plaintext on port 9090 for Prometheus scraping even when the namespace is in STRICT mode
- **Common mistake**: Applying STRICT mode globally before all services in the namespace have Envoy sidecars injected. Services without sidecars still send plaintext ‚Äî their traffic gets rejected by STRICT-mode sidecars with 503 errors. The migration order is: inject sidecars ‚Üí run PERMISSIVE ‚Üí verify all traffic is mTLS via Kiali ‚Üí switch to STRICT.
- **Example**:
```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: notification
spec:
  mtls:
    mode: STRICT
  portLevelMtls:
    9090:                              # Prometheus scraping port
      mode: PERMISSIVE                 # Allow plaintext scrape from Prometheus (no sidecar)
```

### RequestAuthentication
- **API group**: `security.istio.io/v1beta1`
- **Scope**: Namespace
- **Purpose**: Validates JSON Web Tokens (JWTs) on inbound HTTP requests. When a request carries a JWT, Envoy validates its signature against the JWKS endpoint and checks `iss`, `aud`, and `exp` claims. **Critical**: RequestAuthentication alone does NOT reject requests that have no JWT ‚Äî it only validates tokens that ARE present. To reject requests without a valid JWT, pair with an AuthorizationPolicy.
- **Key fields**:
  - `selector.matchLabels`: which pods validate JWTs
  - `jwtRules[].issuer`: the expected `iss` claim value (e.g., `https://auth.t-mobile.com`)
  - `jwtRules[].jwksUri`: URL to fetch the public key set for signature validation (e.g., `https://auth.t-mobile.com/.well-known/jwks.json`)
  - `jwtRules[].audiences`: expected `aud` claim values
  - `jwtRules[].forwardOriginalToken`: whether to pass the JWT to the upstream service after validation (`true`) or strip it (`false`)
  - `jwtRules[].fromHeaders`: custom header to read the JWT from (default is `Authorization: Bearer <token>`)
- **Common mistake**: Believing RequestAuthentication alone secures the endpoint. Requests with no Authorization header are still forwarded to the service because RequestAuthentication has nothing to validate ‚Äî no JWT means no validation triggered. The `requestPrincipal` claim is simply absent. You need an AuthorizationPolicy `action: ALLOW` rule requiring `requestPrincipals: ["*"]` to actually block unauthenticated requests.

### AuthorizationPolicy
- **API group**: `security.istio.io/v1beta1`
- **Scope**: Namespace
- **Purpose**: RBAC for service-to-service traffic at Layer 7. Controls which source (by SPIFFE principal, namespace, IP block, or JWT claim) can call which destination (pod) on which port, HTTP path, or HTTP method. The primary tool for implementing zero-trust networking within the mesh. Evaluated after PeerAuthentication (mTLS) and RequestAuthentication (JWT validation).
- **Key fields**:
  - `selector.matchLabels`: which pods this policy protects
  - `action`: `ALLOW` (whitelist ‚Äî deny everything not listed), `DENY` (blacklist ‚Äî allow everything not listed), `AUDIT` (log without enforcing ‚Äî useful during rollout)
  - `rules[].from[].source.principals`: SPIFFE principal URIs of allowed callers (format: `cluster.local/ns/<namespace>/sa/<serviceaccount>`)
  - `rules[].from[].source.namespaces`: allow all services from a specific namespace (less granular than principals)
  - `rules[].from[].source.ipBlocks`: allow traffic from specific IP ranges
  - `rules[].from[].source.requestPrincipals`: JWT-based principal (format: `<issuer>/<subject>`, wildcard `*` matches any valid JWT)
  - `rules[].to[].operation.methods`: HTTP methods allowed (e.g., `["GET", "POST"]`)
  - `rules[].to[].operation.paths`: URL path prefixes or exact paths allowed (e.g., `["/api/v1/*"]`)
  - `rules[].to[].operation.ports`: ports the rule applies to
  - `rules[].when`: additional conditions (JWT claims, source labels, destination labels)
- **Common mistake**: Applying an AuthorizationPolicy with an empty `spec: {}` or an empty `rules: []` to a workload. The result depends on `action`: an ALLOW policy with no rules denies ALL traffic to that workload. Teams sometimes apply this as a "template" placeholder and accidentally take down the service.
- **Example (zero-trust service-to-service)**:
```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: notification-service-policy
  namespace: notification
spec:
  selector:
    matchLabels:
      app: notification-service
  action: ALLOW
  rules:
  - from:
    - source:
        principals:
        - "cluster.local/ns/message-router/sa/message-router-sa"
        - "cluster.local/ns/api-gateway/sa/api-gateway-sa"
    to:
    - operation:
        methods: ["POST"]
        paths: ["/api/v1/notifications/*"]
        ports: ["8080"]
```

## Telemetry & Proxy Resources

### Telemetry
- **API group**: `telemetry.istio.io/v1alpha1`
- **Scope**: Namespace (or mesh-wide when applied to the `istio-system` namespace)
- **Purpose**: Configures what telemetry signals Envoy generates and where they are sent. Replaces the older MeshConfig telemetry section and the Mixer-era telemetry resources. Controls three signal types: metrics (Prometheus), traces (Jaeger/Zipkin/OTEL), and access logs. Allows overriding mesh-wide defaults at the namespace or workload level.
- **Key fields**:
  - `selector.matchLabels`: which workloads this config applies to (omit for namespace-wide)
  - `tracing[].providers[].name`: trace backend provider name (must match a MeshConfig `extensionProvider` entry)
  - `tracing[].randomSamplingPercentage`: percentage of requests to trace (0.0‚Äì100.0)
  - `tracing[].disableSpanReporting`: disable tracing for matched workloads (useful for high-volume internal services)
  - `metrics[].providers[].name`: metrics backend (e.g., Prometheus)
  - `metrics[].overrides`: customize which metrics are reported and with what tags
  - `accessLogging[].providers[].name`: access log provider (Envoy file log, OTel, etc.)
  - `accessLogging[].disabled`: disable access logging for matched workloads (e.g., internal health check endpoints)
- **Common mistake**: Setting `randomSamplingPercentage: 100` in production. On a 25M msg/day platform, 100% trace sampling overwhelms Jaeger or Zipkin ‚Äî the backends fall behind, drop spans, and your trace store fills up in hours. Use 1‚Äì5% in production, 100% in dev/staging only. Use header-based forced sampling (`x-b3-sampled: 1`) for on-demand full tracing of specific requests.

### ProxyConfig
- **API group**: `networking.istio.io/v1beta1`
- **Scope**: Namespace (or mesh-wide when applied to the `istio-system` namespace)
- **Purpose**: Configures per-proxy Envoy settings ‚Äî worker thread concurrency, tracing backend, proxy log level, hold time during pod shutdown, and whether the app container should wait for Envoy to be ready before starting. Replaces annotation-based proxy configuration (`proxy.istio.io/config` annotation) with a proper CRD.
- **Key fields**:
  - `selector.matchLabels`: which workloads this config applies to
  - `concurrency`: number of Envoy worker threads. Default is 2. Increase to match available CPUs for high-throughput workloads (rule of thumb: set to vCPU count, max 4 unless benchmarked)
  - `tracing`: trace backend configuration (provider, TLS, custom tags)
  - `holdApplicationUntilProxyStarts`: if `true`, init container holds the app container until Envoy is ready (prevents app startup race condition)
  - `proxyMetadata`: environment variables to set in the Envoy proxy container
- **Common mistake**: Not setting `holdApplicationUntilProxyStarts: true`. The app container starts before Envoy's iptables rules are installed and the proxy is listening. Outbound connections made by the app during those first 1‚Äì2 seconds bypass the mesh entirely. This causes intermittent failures that only manifest on pod startup ‚Äî extremely difficult to reproduce and diagnose.
