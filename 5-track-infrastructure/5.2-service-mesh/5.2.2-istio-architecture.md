# Chapter 2: Istio Architecture

## System Overview

Before unpacking individual components, orient to the full picture:

```

│              CONTROL PLANE                   │
│  ┌─────────────────────────────────────┐    │
│  │              istiod                  │    │

                   │
    ┌──────────────┼──────────────┐

┌───▼───┐     ┌───▼───┐     ┌───▼───┐
│ Pod A │     │ Pod B │     │ Pod C │
```

Two distinct layers. The control plane (istiod) configures. The **data plane** (Envoy sidecars) executes. They never swap roles.

## istiod: The Brain

**istiod** is a single binary that consolidates what were once three separate Istio components: Pilot (routing config), Citadel (certificate authority), and Galley (configuration validation and distribution). Istio 1.5 merged them in 2020 to reduce operational complexity — three deployments to manage became one.

istiod does exactly three things:

**1. Routing configuration distribution**: istiod watches Kubernetes for Istio custom resources — VirtualService, DestinationRule, Gateway, ServiceEntry. When any of these change, istiod translates them from Istio's CRD schema into Envoy's native configuration format (xDS) and pushes the updated config to every affected Envoy sidecar in the mesh. Envoy never reads Kubernetes directly; it only talks to istiod.

**2. Certificate authority**: istiod acts as the mesh's CA. When a pod starts, its Envoy sidecar sends a Certificate Signing Request to istiod. istiod verifies the pod's Kubernetes service account identity, issues a short-lived X.509 certificate encoding the pod's SPIFFE identity, and returns it to the sidecar. The sidecar uses this certificate for all mTLS connections. Before the certificate expires, Envoy automatically requests a renewal. No human intervention required.

**3. Configuration validation**: istiod validates Istio CRDs as they are applied to Kubernetes via a ValidatingAdmissionWebhook. A malformed VirtualService gets rejected at apply time, not discovered when traffic starts misbehaving.

EKS analogy: istiod's relationship to Envoy sidecars is structurally identical to your EKS control plane's relationship to kubelets. The control plane (API server) holds desired state and pushes config. The kubelets (on worker nodes) receive that config and act on it. Neither the kubelet nor the Envoy sidecar makes decisions independently — they are agents of their respective control planes.

## The xDS API: Config Distribution Protocol

**xDS API** is the protocol istiod uses to push configuration to Envoy proxies. The "x" is a variable — the API is a family of related Discovery Services, each responsible for a different dimension of Envoy configuration:

| API | Full Name | What it carries |
|-----|-----------|-----------------|
| LDS | Listener Discovery Service | Which ports Envoy should listen on, and with what filter chains |
| RDS | Route Discovery Service | How to route requests received on each listener |
| CDS | Cluster Discovery Service | Which upstream services exist and their load balancing config |
| EDS | Endpoint Discovery Service | Which pod IPs are healthy for each upstream cluster |

These four together form ADS — Aggregated Discovery Service — which is how istiod actually streams all four in a single multiplexed gRPC connection per Envoy.

The Kafka analogy is precise here: istiod is the broker, Envoy sidecars are consumers, and each xDS stream is a topic subscription. When you apply a new VirtualService, istiod publishes an updated RDS message to the affected consumer group — every Envoy that handles traffic to that service. Exactly like a Kafka consumer group receiving new messages, the Envoys receive the config update and apply it in-place without restarting.

This dynamic reconfiguration is what differentiates Envoy from NGINX for mesh use cases. Envoy can receive a new routing rule, add a new upstream cluster, or rotate its mTLS certificate without dropping a single active connection.

## How Traffic Actually Flows: Inbound Request Trace

Understanding this trace is essential — it comes up in both architecture interviews and incident diagnosis.

1. **Request arrives at the pod's network interface** from another service (or from the ingress gateway).

2. **iptables intercepts** — the `istio-init` container that ran at pod startup programmed iptables REDIRECT rules into the pod's network namespace. These rules capture all inbound TCP traffic on port 15006 before it reaches the application's port. The app never sees the raw packet.

3. **Envoy receives on port 15006** — this is the inbound capture port. Envoy's listener on 15006 is the entry point for all inbound traffic destined for the pod.

4. **PeerAuthentication check** — Envoy verifies the caller's mTLS certificate. If the namespace PeerAuthentication is `STRICT`, and the incoming connection is plaintext, Envoy rejects it here. If mTLS is valid, Envoy extracts the caller's SPIFFE identity from the certificate's SAN field.

5. **AuthorizationPolicy check** — Envoy evaluates whether the caller's identity is allowed to make this call. If no matching AuthorizationPolicy exists, the default is allow (in Istio's default config). If a DENY policy matches, the request is rejected with a 403.

6. **Routing rules applied** — Envoy applies any relevant VirtualService rules: timeout enforcement, retry config, header manipulation.

7. **Forward to app container** — Envoy makes a local loopback call to the application's actual port (e.g., 8080). From the app's perspective, the request arrived from localhost.

8. **App responds** — the response flows back through Envoy on the outbound path (port 15001), which handles response header manipulation and emits telemetry for the completed request.

The **iptables REDIRECT** rules are the mechanism that makes all of this transparent. Applications do not need to know the proxy exists — the operating system network stack intercepts traffic before the app's socket sees it.

## Sidecar Injection: How Envoy Gets Into Every Pod

When a namespace has the label `istio-injection=enabled`, Istio registers a **MutatingAdmissionWebhook** with the Kubernetes API server. Every new pod creation request in that namespace passes through the webhook before the pod is scheduled.

The webhook mutates the pod spec, adding two containers:

- **`istio-init`**: an init container that runs before the application starts. Its sole job is to program iptables rules that redirect TCP traffic to the Envoy sidecar ports. It runs to completion and exits.
- **`istio-proxy`**: the Envoy sidecar container. This runs alongside the application for the lifetime of the pod. It receives its initial config from istiod when it starts and then maintains a live xDS connection to istiod for config updates.

Implication for your EKS platform: sidecar injection requires that the node's iptables is accessible from within the pod's network namespace with sufficient privileges. This works in standard EKS but requires `NET_ADMIN` capability for the init container. If your pods run with restricted security contexts (e.g., `readOnlyRootFilesystem: true`, `allowPrivilegeEscalation: false`), you need to account for this when planning a mesh rollout. Ambient mode (Istio 1.22+) eliminates the init container entirely by moving iptables interception to the node level — worth knowing for the "what's next" part of a Director conversation.

## Key Ports Reference

Knowing these cold prevents confusion during incident triage and impresses in technical interviews:

| Port | Component | Purpose |
|------|-----------|---------|
| 15000 | Envoy | Admin interface — dump config, check listeners, drain connections |
| 15001 | Envoy | Outbound traffic capture (all egress from the pod) |
| 15006 | Envoy | Inbound traffic capture (all ingress to the pod) |
| 15010 | istiod | Plaintext gRPC for xDS (dev/test only) |
| 15012 | istiod | mTLS gRPC for xDS (production) |
| 15014 | istiod | Control plane metrics (Prometheus scrape target) |
| 15020 | Envoy | Merged Prometheus metrics (app + Envoy combined) |

In an incident where a sidecar appears to be dropping traffic, `kubectl exec -it <pod> -c istio-proxy -- curl localhost:15000/config_dump` dumps the full Envoy config — listeners, routes, clusters, and endpoints — exactly as the sidecar has received it from istiod. That single command answers most "why isn't traffic getting through?" questions.
