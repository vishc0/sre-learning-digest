# Chapter 5: Observability with Istio

## L7 Telemetry for Free

The foundational premise of mesh observability: every Envoy sidecar is a telemetry collection point. Every HTTP request, every gRPC call, every TCP connection that flows between services passes through two Envoy proxies — one on the sending side, one on the receiving side. Both proxies record what happened.

The result: you get **RED metrics** (Rate, Errors, Duration), distributed traces, and structured access logs for every service-to-service call in your cluster — without modifying a single line of application code, without deploying any instrumentation library, without coordinating with development teams. The mesh observes everything because it sits in the data path.

## The Three Pillars in Mesh Context

### Pillar 1: Metrics

Every Envoy sidecar exports Prometheus metrics automatically. The three you need to know cold:

- `istio_requests_total` — request count labeled by source service, destination service, HTTP method, and response code. This gives you request rate and error rate for every service pair.
- `istio_request_duration_milliseconds` — a histogram of response times. Prometheus histograms support P50, P95, and P99 calculations. You get latency percentiles for every service pair.
- `istio_tcp_connections_opened_total` — TCP connection tracking for non-HTTP protocols.

These are **RED metrics** delivered automatically. In a traditional microservices setup without a mesh, each team instruments their own service, uses different libraries, exports metrics in different formats, and you get inconsistent coverage. One team ships excellent metrics; another ships nothing. With Istio, the floor is raised: every service gets the same telemetry on day one.

No application instrumentation is needed. The mesh sees all L7 traffic.

### Pillar 2: Distributed Tracing

Distributed tracing answers the question: "This request took 3.2 seconds end-to-end — where did the time go?" Without tracing, you know the total latency but cannot attribute it to any specific service in a chain of ten microservices.

How it works in Istio: Envoy adds `x-b3-traceid` and `x-b3-spanid` **B3 headers** to every incoming request (B3 is the header format originated by Zipkin and adopted by Jaeger and OpenTelemetry). Each Envoy sidecar along the request path reads these headers and emits a span to the configured tracing backend — Jaeger or Zipkin. The result is a trace: a waterfall diagram showing every service in the call chain, with start time and duration for each hop.

Example trace for a notification dispatch request:
```
API Gateway        → 50ms
notification-svc   → 100ms
user-profile-svc   → 2,800ms  ← bottleneck
Redis cache        → 5ms
Total              → 3,200ms (approx)
```

You can see immediately that user-profile-service is responsible for 87% of the total latency.

**The one caveat**: Envoy propagates trace headers between services automatically, but the application itself must pass those headers through from incoming to outgoing requests. If notification-service receives a request with `x-b3-traceid: abc123` and then calls user-profile-service, it must include that same header in the outgoing call. Envoy does not reach into the application to do this — the app must forward headers it did not originate. This is usually a 5-10 line change per service, and most HTTP client frameworks support automatic header propagation via middleware.

What the application does NOT need to do: generate trace IDs, create spans, or talk to Jaeger directly. Envoy handles span generation and export.

### Pillar 3: Access Logs

Envoy logs every L7 request as a structured event. This is the Splunk goldmine.

Each log entry contains: source service identity (SPIFFE URI, e.g., `spiffe://t-mobile.com/notification/message-router`), destination service, HTTP method, path, response code, bytes sent and received, response time in milliseconds, and the trace ID linking this log line to its distributed trace. Output format is configurable — set it to JSON for clean Splunk ingestion.

Compare this to what a traditional AWS ALB access log gives you: client IP, target IP, port, response code, response time. No service identity. No path-level breakdowns per microservice. No correlation to a trace ID.

## Splunk Integration

This is the section that separates an SRE who has used a mesh from one who has read about it.

**Architecture**: Envoy access logs write to stdout on each sidecar container. Fluent Bit runs as a DaemonSet on every EKS node, tails container logs, parses JSON, and ships to Splunk via **Splunk HEC** (HTTP Event Collector). No agents inside the application pods. No changes to application logging configuration.

Every HTTP request between services in your EKS cluster becomes a structured JSON event in Splunk within seconds of occurring.

**SPL queries that are now possible — and were impossible with traditional LB logs:**

Service dependency map with error rates across all service pairs:
```spl
index=mesh source=envoy
| stats count by src_service dest_service response_code
```

503 spike analysis — which destination is generating 503s and when:
```spl
index=mesh source=envoy response_code=503
| timechart span=1m count by dest_service
```

P99 latency for every service pair in the cluster — one query:
```spl
index=mesh source=envoy
| stats p99(duration_ms) by src_service dest_service
```

Service account audit — what is message-router actually calling:
```spl
index=mesh source=envoy
  src_principal="spiffe://t-mobile.com/notification/message-router"
| stats count by dest_service method path
```

This last query is powerful for security and compliance: you can prove what a specific service is calling, authenticated by cryptographic identity (SPIFFE), not by IP address. IP addresses change with pod restarts. SPIFFE identity is durable.

**Splunk alert example**: Create a saved search that fires when error rate between notification-service and user-profile-service exceeds 1% sustained for more than 2 minutes. Before the mesh, building this alert required app-level metric instrumentation. After the mesh, it is a Splunk alert on access log data that already exists.

## Kiali: Topology Visualization

**Kiali** is Istio's traffic topology tool. It queries Prometheus for the mesh metrics and renders a live graph: nodes are services, edges are call relationships, edge weight represents call rate, edge color represents error rate (green = healthy, orange = degraded, red = errors). Response time is shown on each edge.

Think of it as the RabbitMQ Management Console for service-to-service traffic. In the RabbitMQ admin UI, you watch queue depth and consumer rate graphs in real time. Kiali gives you the equivalent view for every service pair in your cluster — call rate, error rate, and P99 latency as a live, navigable graph.

Use case: a P1 alert fires. You open Kiali before opening Splunk. The topology graph shows three red edges, all pointing inward to user-profile-service. You know immediately where to focus — no log mining required to find the bottleneck service.

Kiali also shows you which traffic is mTLS-protected (lock icon on edges) versus unencrypted, surfaces misconfigured VirtualServices and DestinationRules with inline warnings, and lets you view the Envoy configuration for any specific pod without `kubectl exec`.

## Prometheus and Grafana

Istio ships with pre-built Grafana dashboards:

- **Mesh Overview**: cluster-wide request rate, error rate, and P99 latency. The 30-second view of overall mesh health.
- **Service Dashboard**: per-service RED metrics. Incoming request rate, error rate by response code, P99 latency, and outbound call breakdown.
- **Workload Dashboard**: per-pod metrics. Useful for identifying hot pods or outliers within a service.

Key panels to know for interview discussions: Request Rate (req/sec), Error Rate (% of 4xx/5xx), P99 Latency (milliseconds), Active Connections (open TCP connections to the destination), mTLS Handshake Rate (confirms zero-trust is operating).

## Key Terms

| Term | What It Is |
|------|------------|
| **RED metrics** | Rate, Errors, Duration — the three standard service health signals |
| **distributed tracing** | End-to-end request path visualization across multiple services |
| **B3 headers** | Trace propagation headers (`x-b3-traceid`, `x-b3-spanid`) used by Jaeger/Zipkin |
| **Splunk HEC** | HTTP Event Collector — the Splunk ingestion endpoint for structured log data |
| **Kiali** | Istio's live traffic topology visualization and mesh health dashboard |
| **access logs** | Per-request structured log events emitted by every Envoy sidecar |
