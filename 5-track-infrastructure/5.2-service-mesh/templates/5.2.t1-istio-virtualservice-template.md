# Istio VirtualService & DestinationRule Templates

## How to Use These Templates
Copy the relevant YAML, replace `<PLACEHOLDERS>` with your values, and apply with `kubectl apply -f`. Each inline comment explains what the field does and when to change it.

---

## Template 1: Basic VirtualService (Single Destination)

Use when: you have one version of a service and want basic routing control.

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: user-profile-service          # Name of this VirtualService resource
  namespace: notification             # Must be in same namespace as the service, OR use exportTo
spec:
  hosts:
  - user-profile-service              # The Kubernetes service name this VS applies to.
                                      # Must match exactly — case-sensitive.
                                      # For external services, use FQDN (e.g., api.stripe.com)
  http:
  - route:
    - destination:
        host: user-profile-service    # Target Kubernetes service name
        port:
          number: 8080                # Target port — must match the Service's port spec
```

**When this is enough**: most services in a mesh don't need canary splitting or fault injection. Apply this
basic VirtualService if you only need the mesh to recognize the service for observability and mTLS — even
without explicit routing rules, Istio handles this. Only add complexity (retries, subsets, faults) when
you have a concrete operational reason.

## Template 2: Traffic Splitting VirtualService (80/20 Canary)

Use when: rolling out a new version of a service gradually. Deploy v2 pods with label `version: v2`, keep
v1 pods with `version: v1`, then shift traffic incrementally.

**Typical canary progression**:
- Day 1: 95/5 (v1/v2) — watch for errors
- Day 2: 80/20 — expand if Day 1 is clean
- Day 3: 50/50 — validate at equal load
- Day 4: 0/100 — complete cutover

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: notification-service
  namespace: notification
spec:
  hosts:
  - notification-service
  http:
  - name: canary-split                # Optional: name this route for easier debugging in Kiali/Jaeger
    route:
    - destination:
        host: notification-service
        subset: v1                    # "v1" must be defined in DestinationRule below.
                                      # subset = a named group of pods with specific labels.
                                      # Think of it like a RabbitMQ queue binding — the label is
                                      # the routing key, the subset is the named consumer group.
      weight: 80                      # 80% of traffic goes to v1 (stable)
                                      # Change to: 100/0 (full rollback), 50/50 (halfway),
                                      # 0/100 (complete cutover to v2)
    - destination:
        host: notification-service
        subset: v2                    # v2 = canary pods with label version=v2
      weight: 20                      # 20% of traffic goes to v2 (canary)
                                      # Monitor: error rate, P99 latency for v2 subset
                                      # If v2 error rate > 1%, set weight to 0 (rollback)
                                      # Rollback is instant: update this file, kubectl apply -f
```

**Companion DestinationRule** (required for subsets to work):
```yaml
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: notification-service
  namespace: notification
spec:
  host: notification-service          # Must match the VirtualService hosts value exactly
  subsets:
  - name: v1                          # Must match the subset name in VirtualService
    labels:
      version: v1                     # Selects pods with label version=v1
                                      # Your pods MUST have this label before applying this DR.
                                      # Verify: kubectl get pods -n notification -l version=v1
                                      # If no pods match: Envoy returns 503 for that subset
  - name: v2
    labels:
      version: v2                     # Selects pods with label version=v2
```

**Common mistake**: creating the VirtualService with subsets before applying the DestinationRule. The
subset reference will fail (503) until the DR exists. Always apply DestinationRule first.

## Template 3: VirtualService with Retries and Timeout

Use when: you want Envoy to automatically retry failed requests and enforce a global timeout. This is
one of the most operationally impactful Istio features — it eliminates the need to implement retry logic
in application code.

**Critical**: only use retries for idempotent operations. GET requests are almost always safe to retry.
POST requests that send notifications, process payments, or write state are NOT safe without application-level
idempotency keys.

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: user-profile-service
  namespace: notification
spec:
  hosts:
  - user-profile-service
  http:
  - timeout: 10s                      # Global timeout for this route.
                                      # If user-profile-service doesn't respond in 10s, return 504.
                                      # Set this tighter than your SLO response time target.
                                      # For 25M msg/day platform: 2-3s is reasonable for profile lookups.
                                      # Analogy: this is your circuit breaker's "trip time" at the
                                      # Envoy layer — before the circuit breaker even needs to fire.
    retries:
      attempts: 3                     # Number of retry attempts after the initial request.
                                      # Total attempts = 1 (initial) + 3 (retries) = 4.
                                      # WARNING: Only use for idempotent GET requests.
                                      # NEVER retry POST /send-notification — duplicates messages!
                                      # For the notification platform: retry GET /profile, not POST /send.
      perTryTimeout: 3s               # Timeout per individual attempt.
                                      # Total worst case: 3s × 4 attempts = 12s.
                                      # Should be less than: global timeout / (attempts + 1)
                                      # If perTryTimeout × attempts > global timeout,
                                      # the global timeout fires before all retries complete.
      retryOn: "503,connect-failure,retriable-4xx"
                                      # Comma-separated conditions to retry on:
                                      # 503     = service unavailable (pod restarting, not ready)
                                      # connect-failure = Envoy couldn't reach the upstream pod
                                      # retriable-4xx   = 4xx codes considered retryable (only 409 by default)
                                      # gateway-error   = 502, 503, 504 (useful for API gateway flows)
                                      # reset           = upstream reset the connection (TCP RST)
                                      #
                                      # Do NOT add: 500 (internal server error — usually a code bug, not transient)
                                      # Do NOT add: 404 (not found — retrying won't make the resource appear)
                                      # Do NOT add: 429 (rate limited — retrying makes it worse)
    route:
    - destination:
        host: user-profile-service
        port:
          number: 8080
```

**Timeout math check** (run this before applying):
```
perTryTimeout × (attempts + 1) should be < global timeout
3s × (3 + 1) = 12s > 10s global timeout → global timeout fires first on worst case

Fix: either increase global timeout to 15s, or reduce attempts to 2:
3s × (2 + 1) = 9s < 10s → retries complete before global timeout
```

## Template 4: Header-Based Routing (A/B Testing by User Segment)

Use when: you want to route specific users (e.g., internal testers, enterprise customers) to a new
service version based on a request header. Unlike weight-based canary, this is deterministic —
the same user always gets the same version.

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: notification-service
  namespace: notification
spec:
  hosts:
  - notification-service
  http:
  # Route 1: Internal test traffic → v2
  - match:
    - headers:
        x-t-mobile-test-group:        # Custom header injected by api-gateway for internal users
          exact: "canary"             # Only this exact header value triggers this route
                                      # Other options:
                                      # prefix: "internal-" (matches "internal-team", "internal-qa")
                                      # regex: "^(canary|beta|internal)$"
    route:
    - destination:
        host: notification-service
        subset: v2                    # All "canary" header requests go to v2
      weight: 100
  
  # Route 2: All other traffic → v1 (catch-all, must be last)
  - route:
    - destination:
        host: notification-service
        subset: v1
      weight: 100
                                      # IMPORTANT: Routes are evaluated top-to-bottom.
                                      # The first matching route wins.
                                      # Always put the most specific match first.
                                      # Always put the catch-all (no match block) last.
```

**Operational note**: the api-gateway must inject the `x-t-mobile-test-group: canary` header for internal
users. Coordinate with the gateway team before deploying this pattern. If the gateway doesn't inject the
header, all traffic falls through to the catch-all v1 route — a safe failure mode.

## Template 5: VirtualService with Fault Injection (For Chaos Testing)

Use when: you want to simulate failures without actually breaking the real service. Use in non-prod
environments for chaos engineering and resilience testing. Validates that your retry policies,
circuit breakers, and fallback logic actually work.

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: user-profile-service-fault-test
  namespace: notification-staging      # ALWAYS fault-inject in staging, never in prod
spec:
  hosts:
  - user-profile-service
  http:
  # Test scenario 1: Simulate slow user-profile-service response (2s delay on 20% of requests)
  # Use this to verify: does notification-service time out correctly? Does it circuit break?
  - fault:
      delay:
        percentage:
          value: 20.0                 # 20% of requests will be delayed.
                                      # Start small (5%) and increase to find the threshold
                                      # where notification-service starts returning errors.
        fixedDelay: 2s                # 2-second artificial delay added before forwarding.
                                      # Set this > your configured timeout to test timeout handling.
                                      # For T-Mobile platform: set to 4s (> 3s perTryTimeout)
    route:
    - destination:
        host: user-profile-service
        port:
          number: 8080

  # Test scenario 2: Simulate user-profile-service returning 503 (use a separate VS for this test)
  # Uncomment this block and comment out the delay block above to switch test scenarios.
  # - fault:
  #     abort:
  #       percentage:
  #         value: 10.0               # 10% of requests get an immediate 503 response.
  #                                   # Envoy returns the error WITHOUT forwarding to the service.
  #                                   # Use to test: does circuit breaker trip at consecutiveErrors: 5?
  #                                   # Expected: after 5 consecutive 503s from a pod, that pod is ejected.
  #       httpStatus: 503
  #   route:
  #   - destination:
  #       host: user-profile-service
  #       port:
  #         number: 8080
```

**IMPORTANT operational rules for fault injection**:
1. Apply with a comment in your PR: "FAULT INJECTION — chaos test for [ticket number]"
2. Set a calendar reminder to remove after testing (max 2 hours)
3. Never apply fault injection VirtualServices to prod — a forgotten fault VS causes real outages
4. Use a separate named VS (e.g., `-fault-test` suffix) so it's clearly distinguishable
5. Validate the fault is working: watch error rates in Prometheus/Grafana before measuring resilience

## Template 6: DestinationRule with Circuit Breaking

Use when: you want Envoy to automatically eject unhealthy pods from the load balancing pool. This is
Istio's implementation of the circuit breaker pattern at the infrastructure layer — no application code
changes required.

**Analogy for Istio context**: think of `outlierDetection` as RabbitMQ's dead-letter queue mechanism
but for pod health — a pod that keeps returning errors gets "dead-lettered" (ejected) temporarily,
then tested again after a backoff period.

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: user-profile-service
  namespace: notification
spec:
  host: user-profile-service
  trafficPolicy:
    outlierDetection:
      consecutiveErrors: 5            # Eject a pod after 5 consecutive 5xx errors.
                                      # Lower = more aggressive ejection (catches problems faster,
                                      #         but flaps more on transient errors).
                                      # Higher = more tolerance for transient errors
                                      #         (pods stay in rotation longer during incidents).
                                      # For the notification platform: 5 is a reasonable default.
                                      # For payment APIs where errors are costly: consider 3.

      consecutiveGatewayErrors: 5     # Eject after 5 consecutive 502/503/504 errors.
                                      # Separate from consecutiveErrors — use when your service
                                      # returns 4xx legitimately but 5xx always means the pod is sick.

      interval: 30s                   # How often Envoy analyzes each pod's error rate.
                                      # Envoy counts errors within this sliding window.
                                      # Lower = faster detection (good), higher CPU overhead (cost).
                                      # 30s is the Istio default and works well for most cases.

      baseEjectionTime: 30s           # How long an ejected pod stays out of rotation on first ejection.
                                      # Doubles with each subsequent ejection: 30s → 60s → 120s → ...
                                      # After this time, Envoy sends a probe request to the pod.
                                      # If the probe succeeds, the pod re-enters rotation.
                                      # If it fails, the backoff doubles again.

      maxEjectionPercent: 50          # Max percentage of pods that can be ejected simultaneously.
                                      # Prevents "eject all pods" scenario during mass failure.
                                      # Example: 4 pods, maxEjectionPercent=50 → max 2 ejected.
                                      # NEVER set to 100 — you would take the entire service down.
                                      # At 50%, at least half the pods always stay in rotation.

      minHealthPercent: 30            # If fewer than 30% of pods are healthy, disable outlier detection.
                                      # Prevents a death spiral: during mass failure, stop ejecting
                                      # and send traffic to all remaining pods (cluster-is-down recovery mode).
                                      # Set to 0 to disable this safety valve (not recommended).
```

## Template 7: DestinationRule with Connection Pool Settings

Use when: you need to limit how many connections Envoy opens to a downstream service. Critical for
protecting stateful services (Cassandra, Redis, RabbitMQ) from connection exhaustion. Acts as
backpressure — when the limit is hit, Envoy returns 503 instead of overwhelming the downstream.

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: cassandra-service             # Example: protecting a Cassandra cluster
  namespace: notification
spec:
  host: cassandra-service
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100           # Max concurrent TCP connections from all Envoy sidecars
                                      # in THIS namespace to the destination service.
                                      # Match this to your Cassandra connection pool config.
                                      # If Cassandra max_connections = 200, set this to 100
                                      # to leave headroom for maintenance and Cassandra-side clients.
                                      # Exceeding this: new connections get a 503 immediately.

        connectTimeout: 3s            # TCP connection timeout.
                                      # If Cassandra doesn't accept the TCP connection in 3s, fail fast.
                                      # For local-cluster connections, 3s is generous — 1s is fine.

        tcpKeepalive:
          time: 7200s                 # Send TCP keepalive after 2 hours of idle
          interval: 75s               # Keepalive probe interval
          probes: 10                  # Number of probes before declaring connection dead
                                      # Keepalive prevents firewall/NAT devices from killing idle connections.
                                      # For T-Mobile EKS: AWS NLB drops connections idle > 350s.
                                      # Set time to 300s (5 min) to be safe.

      http:
        http1MaxPendingRequests: 50   # Max HTTP/1.1 requests queued waiting for an available connection.
                                      # Exceeding this: Envoy returns 503 immediately (fast fail).
                                      # Acts as a backpressure valve — prevents queue buildup.
                                      # Set based on acceptable latency under load.

        http2MaxRequests: 1000        # Max concurrent HTTP/2 requests (or gRPC streams).
                                      # For gRPC services (inter-service calls in many modern stacks):
                                      # this is the primary connection limit knob.

        maxRequestsPerConnection: 100 # After 100 requests on one connection, Envoy closes it
                                      # and opens a fresh connection.
                                      # Helps distribute load more evenly across pods (avoids
                                      # one lucky pod getting all long-lived connections).
                                      # Set to 1 to force reconnect every request (maximum fairness,
                                      # but higher connection overhead — only for debugging).

        maxRetries: 3                 # Max concurrent retries across all connections to this host.
                                      # Caps retry storm behavior during partial outages.
                                      # If all 10 callers simultaneously retry 3× = 30 concurrent retries
                                      # — this limit applies across all of them.
                                      # Set lower than your circuit breaker threshold.

        idleTimeout: 90s              # Close HTTP connections idle for > 90s.
                                      # Set slightly below AWS NLB idle timeout (350s) to prevent
                                      # the NLB from silently dropping connections mid-stream.
```

## Quick Reference: Key Fields Summary

| Field | Location | What It Controls | When to Change |
|-------|----------|-----------------|---------------|
| `hosts` | VirtualService | Which K8s service this VS applies to | Changing service name |
| `weight` | VirtualService route | Traffic split % between subsets | Canary progression |
| `timeout` | VirtualService http | Max time for a full request | SLO tuning |
| `attempts` | VirtualService retries | How many times to retry | Retry policy |
| `retryOn` | VirtualService retries | Which error codes trigger retry | Error budget tuning |
| `subsets` | DestinationRule | Named pod groups (by label) | Version rollout |
| `consecutiveErrors` | DestinationRule outlier | Errors before pod ejection | Sensitivity tuning |
| `maxEjectionPercent` | DestinationRule outlier | Max % pods ejected at once | Safety limit |
| `maxConnections` | DestinationRule connPool | TCP connection limit | Downstream protection |

## Interview Application: How to Answer "Explain Istio Traffic Management"

**30-second version (Director interview)**:
"VirtualService controls where traffic goes — routing rules, retries, timeouts, fault injection.
DestinationRule controls how traffic behaves once routed — which pods qualify (subsets), how many
connections to open, and when to eject unhealthy pods. Together they give you platform-layer
resilience without touching application code. In our notification platform at T-Mobile, this let us
do zero-downtime canary deployments across 25M messages/day without coordinating application releases."

**Key distinction to memorize**: VirtualService = "WHAT goes WHERE". DestinationRule = "HOW it gets there and WHICH pods qualify."
