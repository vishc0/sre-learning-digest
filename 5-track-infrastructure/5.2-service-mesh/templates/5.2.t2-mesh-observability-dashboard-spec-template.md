# Mesh Observability Dashboard Specification

## Dashboard Overview

**Name**: Service Mesh Health — Notification Platform
**Purpose**: Real-time visibility into service-to-service communication health, mTLS status, and security
policy enforcement across the Istio service mesh on EKS.
**Primary Audience**: SRE on-call, Platform Engineering team
**Secondary Audience**: Director/VP review (Row 1 summary panels), Security team (Row 4 policy panels)
**Refresh Rate**: 30 seconds (live on-call use) | 5 minutes (Director review mode)
**Time Range Default**: Last 1 hour | Options: 15m, 30m, 1h, 3h, 6h, 24h
**Data Sources**:
- Prometheus: Istio metrics (istio_requests_total, istio_request_duration_milliseconds_bucket, etc.)
- Splunk: Envoy access logs forwarded via Fluent Bit DaemonSet → Splunk HEC
**Tool**: Grafana (primary) | Splunk dashboards (security team view)

### Panel 1: Total Requests Per Second (Stat Tile — Row 1)

**Purpose**: Instant sanity check — is overall traffic volume within the expected range for this time of day?
**Type**: Single Stat / Gauge
**Unit**: requests/second
**Thresholds**: Green < 30,000 RPS | Yellow 30,000–35,000 | Red > 35,000 (unexpected spike)
**Baseline**: T-Mobile notification platform at 25M msg/day = ~289 RPS average; peaks during marketing
campaigns or morning alerts may reach 1,000–2,000 RPS. Calibrate thresholds to your actual baseline.

**PromQL**:
```promql
sum(rate(istio_requests_total{reporter="destination"}[2m]))
```

**Field explanations**:
- `reporter="destination"`: count from the receiving Envoy proxy (avoids double-counting — source and
  destination both emit this metric; pick one consistently)
- `rate(...[2m])`: per-second rate over a 2-minute window (smooth but still responsive)
- `sum(...)`: aggregate across all services, namespaces, and pods

**What it means when it spikes**: unexpected traffic surge — could be DDoS, runaway retry loop (a service
retrying at high frequency), or a new client sending at unexpectedly high rate. Correlate immediately
with Panel 2 (error rate). If error rate is also spiking, it's likely a retry storm.

**What it means when it drops suddenly**: service degradation, pod crashes, load balancer health check
failures, or legitimate traffic reduction. A drop to zero on a specific service pair (Panel 5) is
more actionable than a total RPS drop.

### Panel 2: Overall Error Rate % (Stat Tile — Row 1)

**Purpose**: Primary SLO signal. One number to rule them all — is anything broken right now?
**Type**: Single Stat
**Unit**: percentage (0–100)
**Thresholds**: Green < 0.1% | Yellow 0.1%–1% | Red > 1%

**PromQL**:
```promql
100 *
  sum(rate(istio_requests_total{reporter="destination", response_code=~"5.."}[2m]))
  /
  sum(rate(istio_requests_total{reporter="destination"}[2m]))
```

**Field explanations**:
- `response_code=~"5.."`: regex match for any 5xx status code (500, 502, 503, 504, 599)
- Division: 5xx rate / total rate = error fraction; multiply by 100 for percentage
- `reporter="destination"`: same reporter filter as Panel 1 for consistency

**Alert**: Page on-call if > 1% for > 2 consecutive minutes.
**Note**: this is overall. A small service generating 100% errors might not move this needle if it
handles low traffic. Always drill into Panel 6 (error rate by service pair) for root cause isolation.

---

### Panel 3: P99 Latency — All Services (Stat Tile — Row 1)

**Purpose**: Latency SLO signal. Catch slowdowns before users notice.
**Type**: Single Stat
**Unit**: milliseconds
**Thresholds**: Green < 200ms | Yellow 200–500ms | Red > 500ms

**PromQL**:
```promql
histogram_quantile(0.99,
  sum(rate(istio_request_duration_milliseconds_bucket{reporter="destination"}[2m]))
  by (le)
)
```

**Field explanations**:
- `histogram_quantile(0.99, ...)`: compute the 99th percentile from the histogram bucket data
- `istio_request_duration_milliseconds_bucket`: Istio's latency histogram metric (in milliseconds)
- `by (le)`: group by the "less than or equal" bucket boundary — required for histogram_quantile
- Result: the latency value below which 99% of requests complete

**Why P99 and not P50 here**: P50 (median) hides tail latency issues. For a messaging platform handling
25M msgs/day, 1% of requests = 250,000 requests/day experiencing the P99 latency. That's your worst-case
user experience signal.

### Panel 4: mTLS Coverage % (Stat Tile — Row 1)

**Purpose**: Security posture signal. What percentage of mesh traffic is mTLS-encrypted? Should always
be 100% in a correctly configured STRICT-mode mesh.
**Type**: Single Stat
**Unit**: percentage
**Thresholds**: Green = 100% | Yellow 95–99.9% | Red < 95%

**PromQL**:
```promql
100 *
  sum(rate(istio_requests_total{
    reporter="destination",
    connection_security_policy="mutual_tls"
  }[2m]))
  /
  sum(rate(istio_requests_total{reporter="destination"}[2m]))
```

**Field explanations**:
- `connection_security_policy="mutual_tls"`: Istio label on requests that used full mTLS
- Other values you may see: `"none"` (plaintext), `"unknown"` (Envoy couldn't determine)
- Any non-100% value in STRICT mode indicates a sidecar injection gap or a misconfigured service

**Alert**: Alert security team immediately if drops below 99% — indicates plaintext traffic flowing
through a mesh that should be STRICT. Common causes: a new pod deployed without a sidecar, a
legacy service added to the mesh without proper onboarding, or a PeerAuthentication misconfiguration.

**Splunk equivalent** (from Envoy access logs):
```splunk
index=mesh sourcetype=envoy_access_log namespace=notification
| eval is_mtls = if(tls_version!="" AND tls_cipher!="", 1, 0)
| stats avg(is_mtls)*100 as mtls_coverage_pct
```

### Panel 5: Request Rate by Service Pair (Time Series — Row 2)

**Purpose**: Understand which service pairs are active, at what rate, and how that rate changes over time.
Primary tool for "is this service sending traffic at all?" and "did traffic shift unexpectedly?"
**Type**: Time Series (line graph, one line per service pair)
**Unit**: requests/second
**Legend format**: `{{source_workload}} → {{destination_workload}}`

**PromQL**:
```promql
sum(
  rate(istio_requests_total{
    reporter="destination",
    destination_service_namespace="notification"
  }[2m])
) by (source_workload, destination_workload)
```

**Dashboard variable**: add a Grafana variable `$namespace` to replace `"notification"` — allows
on-call to switch between namespaces without editing queries.

**What to look for**:
- **New service pair appearing**: a previously unseen `source_workload → destination_workload` pair
  means either a new caller was added (expected, check with engineering) or unexpected access
  (investigate — could indicate a misconfigured service or unauthorized caller)
- **Existing pair dropping to zero**: the source service stopped calling the destination. Either the
  source is down, or there's a circuit breaker open, or the traffic was intentionally cut
- **Sudden rate change on one pair**: correlate with a deployment. If notification-service → user-profile
  doubles, check if notification-service pushed new code that calls profile more aggressively

**Splunk equivalent**:
```splunk
index=mesh sourcetype=envoy_access_log namespace=notification
| timechart span=1m count by src_workload, dest_workload
```

### Panel 6: Error Rate by Service Pair (Time Series — Row 2)

**Purpose**: Root cause isolation for Panel 2 alerts. Which specific service pair is generating errors?
Narrows a "something is broken" alert to "notification-service → user-profile-service is returning 503s."
**Type**: Time Series
**Unit**: percentage (0–100)
**Alert line**: horizontal reference line at 1%

**PromQL**:
```promql
100 *
  sum(
    rate(istio_requests_total{
      reporter="destination",
      response_code=~"5..",
      destination_service_namespace="notification"
    }[2m])
  ) by (source_workload, destination_workload)
  /
  sum(
    rate(istio_requests_total{
      reporter="destination",
      destination_service_namespace="notification"
    }[2m])
  ) by (source_workload, destination_workload)
```

**Reading this panel during an incident**:
1. Which pair has the highest error rate? That's your blast radius.
2. Is it one pair or many pairs calling the same destination? If multiple sources to one destination:
   the destination service is the problem. If one source to multiple destinations: the source service
   is the problem (or it's overloading its downstream).
3. When did it start? Correlate with deployment events (Grafana annotations help here).

**Splunk equivalent** (richer — includes path-level breakdown):
```splunk
index=mesh sourcetype=envoy_access_log namespace=notification
| eval error = if(response_code >= 500, 1, 0)
| timechart span=1m avg(error)*100 as error_rate_pct by src_workload, dest_workload
| where error_rate_pct > 0
```

### Panel 7: P50 / P95 / P99 Latency by Service Pair (Time Series — Row 3)

**Purpose**: Identify which specific service pairs have latency regressions, and how severe (P50 alone
vs. P95 and P99 both elevated = different diagnosis).
**Type**: Time Series (3 lines per pair: P50 dashed, P95 dotted, P99 solid)
**Unit**: milliseconds
**Legend format**: `{{source_workload}} → {{destination_workload}} P99`

**PromQL** (P99 — repeat substituting 0.50 and 0.95 for P50 and P95 panels):
```promql
histogram_quantile(0.99,
  sum(
    rate(istio_request_duration_milliseconds_bucket{
      reporter="destination",
      destination_service_namespace="notification"
    }[2m])
  ) by (le, source_workload, destination_workload)
)
```

**Latency pattern diagnosis**:

| Pattern | Likely Cause |
|---------|-------------|
| P50 normal, P99 elevated | Tail latency issue — GC pauses, hot spots, database lock contention |
| P50 and P99 both elevated equally | General slowdown — resource saturation, slow dependency |
| P50 and P99 both elevated but P99 much worse | Bimodal distribution — some requests hitting a slow code path |
| Latency spike on one pair only | That specific destination is slow — check its pod CPU/memory |
| Latency spike on all pairs to one destination | The destination service is the bottleneck |
| Latency spike on all pairs from one source | The source service is slow (high CPU) or waiting on something |

**Splunk equivalent** (for cross-referencing with access logs):
```splunk
index=mesh sourcetype=envoy_access_log namespace=notification
| stats
    p50(response_duration_ms) as p50_ms,
    p95(response_duration_ms) as p95_ms,
    p99(response_duration_ms) as p99_ms
  by src_workload, dest_workload
| sort -p99_ms
| head 20
```

### Panel 8: mTLS Handshake Failure Rate (Time Series — Row 4)

**Purpose**: Detect TLS/certificate problems before they escalate to service-wide failures.
An mTLS handshake failure means two services that should be communicating securely cannot establish
the connection at all — manifests as connection refused or 503 at the client.
**Type**: Time Series
**Unit**: failures/second
**Thresholds**: Alert if sustained > 0 for more than 1 minute

**PromQL** (Istio control plane metric — xDS config expiry, which often precedes cert problems):
```promql
sum(rate(istio_agent_pilot_xds_expired_nonce_total[2m])) by (pod)
```

**PromQL** (Envoy metric — direct TLS error count):
```promql
sum(rate(envoy_listener_ssl_handshake_error[2m])) by (pod_name)
```

**Splunk** (from Envoy access logs — TLS errors logged at Envoy level):
```splunk
index=mesh sourcetype=envoy_access_log
| search "CERTIFICATE_VERIFY_FAILED" OR "SSL_HANDSHAKE" OR "tls handshake" OR "certificate expired"
| timechart span=1m count by dest_workload
```

**Splunk** (find the specific pod and certificate involved):
```splunk
index=mesh sourcetype=envoy_access_log
| search "CERTIFICATE_VERIFY_FAILED" OR "ssl_error"
| rex field=_raw "upstream_host=(?<upstream_host>[^\s]+)"
| rex field=_raw "error=(?<tls_error>[^\]]+)"
| stats count by upstream_host, tls_error, dest_workload
| sort -count
```

**What it means when it spikes**:
- **Cert expiry**: the workload cert was not rotated (istiod rotation failed). Check:
  `kubectl logs -n istio-system deploy/istiod | grep "cert rotation"`
- **mTLS mode mismatch**: one service is STRICT, the caller has no sidecar (PERMISSIVE or no mesh).
  Run: `kubectl get peerauthentication -A`
- **Clock skew between nodes**: cert validity window checked against system time. If node clocks diverge
  > 30 seconds, certs may appear expired. Check: `chronyc tracking` on affected nodes.
- **Root cert rotation**: mesh root CA was rotated but some pods still have old root in trust bundle.
  Requires rolling restart of all pods.

### Panel 9: Certificate Expiry Countdown (Table — Row 4)

**Purpose**: Proactive cert monitoring. Know before certs expire, not after. Workload certs auto-rotate
every 24 hours in default Istio config — a countdown below 1 hour means rotation is failing.
**Type**: Table
**Refresh**: Every 5 minutes
**Columns**: Service | Namespace | Days Until Expiry | Hours Until Expiry | Status

**PromQL**:
```promql
(istio_agent_cert_expiry_seconds{} - time()) / 86400
```

This returns days until expiry. Divide by 3600 instead for hours.

**Grafana table configuration**:
- Field "Days Until Expiry": apply color threshold: > 7 (green), 1–7 (yellow), < 1 (red)
- Sort by "Days Until Expiry" ascending — most urgent at top
- Add label columns: `app`, `namespace`, `pod`

**Alert definition**:
```yaml
# Grafana alert rule (or Prometheus alerting rule)
alert: MeshCertExpiringCritical
expr: (istio_agent_cert_expiry_seconds - time()) < 86400  # < 24 hours
for: 5m
labels:
  severity: critical
  team: platform
annotations:
  summary: "Istio workload cert expiring in < 24 hours on {{ $labels.app }}"
  runbook: "https://wiki.t-mobile.com/sre/runbooks/istio-cert-renewal"
```

**What to do when a cert is near expiry**:
1. Check if istiod is healthy: `kubectl get pods -n istio-system`
2. Check istiod logs for rotation errors: `kubectl logs -n istio-system deploy/istiod | grep -i "cert\|rotation\|error"`
3. Force cert renewal by restarting the affected pod: `kubectl rollout restart deployment/<service> -n notification`
4. If istiod itself is unhealthy: this is a P1 — restart istiod and escalate to platform team

### Panel 10: Policy Deny Events (Bar Chart + Event Log — Row 4)

**Purpose**: Security monitoring — detect unauthorized service-to-service access attempts. Every deny
event means a service tried to call another service and was rejected by an AuthorizationPolicy.
**Type**: Bar chart (count per 5-minute bucket) + event table (most recent 20 denies)
**Unit**: events/minute
**Thresholds**: Alert if > 10 denies/minute sustained for > 2 minutes

**PromQL** (for time series bar chart):
```promql
sum(
  rate(istio_requests_total{
    reporter="destination",
    response_flags="RBAC_ACCESS_DENIED"
  }[2m])
) by (source_workload, destination_workload)
```

**Note on response_flags**: `RBAC_ACCESS_DENIED` is the Istio flag for AuthorizationPolicy denies.
Other flags you may see: `UH` (no healthy upstream), `UC` (upstream connection terminated),
`NR` (no route found), `RL` (rate limited).

**Splunk** (most detailed — includes source SPIFFE identity and exact path that was denied):
```splunk
index=mesh sourcetype=envoy_access_log
| search "RBAC: access denied"
| rex field=_raw "source_principal=(?<src_principal>[^\s\"]+)"
| rex field=_raw "\"path\":\"(?<uri_path>[^\"]+)\""
| stats count by src_principal, dest_workload, uri_path
| sort -count
```

**Splunk** (time series for alert correlation):
```splunk
index=mesh sourcetype=envoy_access_log
| search "RBAC: access denied"
| timechart span=5m count by dest_workload
```

**Security interpretation**:

| Deny Pattern | Likely Cause | Action |
|-------------|-------------|--------|
| New service pair suddenly denied | Recent allow policy change missed this caller | Check git history for AuthorizationPolicy changes in last 24h; add allow rule if legitimate |
| Known service pair denied intermittently | Service account mismatch after pod restart | Verify SA: `kubectl get pod <pod> -o jsonpath='{.spec.serviceAccountName}'` |
| Unknown source principal in denies | Unexpected caller — may be a misconfigured service or intrusion attempt | Investigate source IP and workload; escalate to security if source is unrecognized |
| High volume denies from one source | Retry storm hitting a denied endpoint | Fix the allow policy OR stop the retries; the deny is correct but the retry volume is the signal |

### Panel 11: Circuit Breaker Ejection Events (Time Series — Row 5)

**Purpose**: Track outlier detection ejections — Envoy ejecting unhealthy pods from the load balancing
pool. An ejection event means a pod was returning enough errors that Envoy removed it from rotation.
**Type**: Time Series (event count per minute, one line per destination service)
**Unit**: ejections/minute
**Thresholds**: Alert if > 0 sustained for > 5 minutes on the same service

**PromQL**:
```promql
sum(
  increase(envoy_cluster_outlier_detection_ejections_total[1m])
) by (cluster_name)
```

**Note**: `cluster_name` in Envoy context = the upstream service cluster (not your Kubernetes cluster name).
Format is typically: `outbound|<port>||<service-fqdn>`. Strip to just the service name with label_replace
or a Grafana transformation.

**Additional PromQL** (ejection percentage — how much of the service is currently ejected):
```promql
envoy_cluster_outlier_detection_ejections_active / envoy_cluster_membership_total * 100
```

**Splunk** (from Envoy logs when outlier detection fires):
```splunk
index=mesh sourcetype=envoy_access_log
| search "outlier_detection" OR "ejecting" OR "EjectLocalOriginConsecutiveError"
| rex field=_raw "upstream_host=(?<upstream_host>[^\s]+)"
| timechart span=5m count by upstream_host
```

**Incident response when circuit breaker fires**:
1. Identify the ejected pod: check Envoy cluster stats or Kiali service graph
2. Check ejected pod logs immediately: `kubectl logs <pod-name> -n notification --tail=100`
3. Common causes: OOM kill (pod restarting in a crash loop), database connection exhaustion
   (check Cassandra/Redis connection metrics), dependency timeout (that pod's dependency is slow)
4. Do NOT manually delete the pod immediately — the backoff gives it time to recover.
   If the pod keeps being ejected (baseEjectionTime keeps doubling), then restart it.
5. Check `maxEjectionPercent`: if multiple pods are ejected and approaching the limit, the service
   will degrade to the remaining healthy pods. This is the designed behavior — it's failing safe.

### Panel 12: Retry Rate by Service (Time Series — Row 5)

**Purpose**: High retry rates are a leading indicator of downstream service instability — catch before
the circuit breaker trips. Retries mask errors from the caller but add load to the destination.
**Type**: Time Series (one line per source→destination pair)
**Unit**: retries/second

**PromQL** (using Istio response flags for retry-related conditions):
```promql
sum(
  rate(istio_requests_total{
    reporter="source",
    response_flags=~".*RR.*|.*UR.*|.*UH.*"
  }[2m])
) by (source_workload, destination_workload)
```

**Response flag meanings**:
- `RR`: rate limit reset (connection limit hit)
- `UR`: upstream connection reset (Envoy retried due to TCP reset)
- `UH`: no healthy upstream (all pods ejected or none available)

**Splunk** (tracks retry attempt count directly from Envoy headers):
```splunk
index=mesh sourcetype=envoy_access_log namespace=notification
| eval x_envoy_attempt_count = tonumber(coalesce('x_envoy_attempt_count', "1"))
| eval is_retry = if(x_envoy_attempt_count > 1, 1, 0)
| timechart span=1m
    sum(is_retry) as retry_count,
    count as total_count
  by dest_workload
| eval retry_rate_pct = retry_count / total_count * 100
```

**Splunk** (distribution of retry attempts — how many requests needed 2 vs 3 vs 4 attempts):
```splunk
index=mesh sourcetype=envoy_access_log namespace=notification
| eval attempts = tonumber(coalesce('x_envoy_attempt_count', "1"))
| where attempts > 1
| stats count by attempts, dest_workload
| sort dest_workload, attempts
```

**What high retry rate means**:
- Retry rate high, error rate low: retries are successfully masking the failure — the caller sees
  success, but you are burning extra capacity. The destination is struggling intermittently.
- Retry rate high, error rate also high: retries are not recovering the error — destination is
  consistently failing. Expect circuit breaker ejections to follow.
- Retry rate high, latency elevated: the retries themselves are adding latency (perTryTimeout × attempts).
  Consider whether the retry policy is appropriately tuned for this service pair.

## Key Istio Metrics Reference

| Metric | Type | Labels | What It Measures |
|--------|------|--------|-----------------|
| `istio_requests_total` | Counter | reporter, source_workload, destination_workload, response_code, response_flags | Request count by status |
| `istio_request_duration_milliseconds_bucket` | Histogram | reporter, source_workload, destination_workload, le | Request latency distribution |
| `istio_request_bytes_bucket` | Histogram | reporter, source_workload, destination_workload | Request payload size |
| `istio_response_bytes_bucket` | Histogram | reporter, source_workload, destination_workload | Response payload size |
| `istio_agent_cert_expiry_seconds` | Gauge | app, namespace | Workload cert expiry timestamp |
| `envoy_cluster_outlier_detection_ejections_total` | Counter | cluster_name | Circuit breaker ejection count |
| `envoy_cluster_membership_total` | Gauge | cluster_name | Total pods in the load balancer pool |
| `envoy_listener_ssl_handshake_error` | Counter | pod_name | TLS handshake failures |

**Reporter label**: always pick one consistently for a given panel:
- `reporter="destination"`: metrics from the receiving Envoy (more reliable for error classification)
- `reporter="source"`: metrics from the sending Envoy (useful for retry and timeout tracking)
- Mixing both: double-counts traffic

## Splunk Index and Field Reference

Assumes Envoy access logs are forwarded to Splunk via Fluent Bit with these field extractions.
Configure Fluent Bit to parse Envoy's JSON access log format for clean field extraction.

| Splunk Field | Envoy Source | Description |
|-------------|-------------|-------------|
| `src_workload` | `%UPSTREAM_HOST%` parsed | Source service name |
| `dest_workload` | `%DOWNSTREAM_REMOTE_ADDRESS%` parsed | Destination service name |
| `response_code` | `%RESPONSE_CODE%` | HTTP status code |
| `response_duration_ms` | `%DURATION%` | Total request duration in ms |
| `x_envoy_attempt_count` | `%REQ(X-ENVOY-ATTEMPT-COUNT)%` | Number of retry attempts (1 = no retry) |
| `tls_version` | `%TLS_VERSION%` | TLS protocol version (TLSv1.2, TLSv1.3) |
| `tls_cipher` | `%TLS_CIPHER%` | TLS cipher suite |
| `uri_path` | `%REQ(:PATH)%` | Request path |
| `namespace` | Kubernetes metadata (Fluent Bit enrichment) | Pod namespace |

**Fluent Bit filter** (add namespace and workload name as enriched fields):
```yaml
[FILTER]
    Name kubernetes
    Match kube.*
    Kube_URL https://kubernetes.default.svc:443
    Kube_CA_File /var/run/secrets/kubernetes.io/serviceaccount/ca.crt
    Kube_Token_File /var/run/secrets/kubernetes.io/serviceaccount/token
    Merge_Log On
    K8S-Logging.Parser On
    Labels On
    Annotations Off
```

## Interview Application: How to Describe This Dashboard

**30-second version (Director/VP interview)**:
"We built a five-row mesh observability dashboard giving us instant visibility into service-to-service
health, latency, and security across the notification platform. Row 1 is executive-level: four tiles
showing total RPS, error rate, P99 latency, and mTLS coverage. The security row tracks cert expiry,
handshake failures, and policy deny events — unauthorized access attempts show up here immediately.
During a major incident, this dashboard tells you which service pair is failing, when it started,
whether it's a latency or error issue, and whether retries are masking the real blast radius — all
within the first 90 seconds of the page."

**Key numbers to know for interviews**:
- 25M messages/day = ~289 RPS average
- SLO targets: < 1% error rate, P99 < 500ms
- mTLS coverage: 100% in STRICT mode (any drop is a security event, not just a metric)
- Circuit breaker: 5 consecutive errors → pod ejection, 30s base backoff, max 50% ejected at once
