# Service Identity Register Template

## Purpose
This register maps every mesh-enrolled service to its SPIFFE identity, allowed callers, and authorization
policy. It is the source of truth for "which service can call which service" — the mesh-layer access
control list (ACL).

In a service mesh, mTLS gives every pod a cryptographic identity (a SPIFFE certificate). Authorization
policies then use those identities to enforce who can talk to whom. This register is the human-readable
companion to those machine-enforced policies — it answers "why does service A have access to service B?"
for auditors, security reviews, and incident triage.

**Analogy**: think of this as a firewall ruleset documentation, but for internal service-to-service traffic.
The mesh enforces it automatically; this register is how you audit and manage it over time.

---

## How to Use
1. Add a row for every service enrolled in the mesh (at onboarding time, not retroactively)
2. Create the corresponding AuthorizationPolicy YAML and store in your GitOps repo (same PR)
3. Review quarterly or after any service addition, removal, or service account change
4. All changes require PR approval from the security team or designated service owner
5. Keep this register in sync with actual deployed policies — drift = audit failure

## AuthorizationPolicy Templates

### Policy 0: Default Deny (Apply First — to Every Namespace)

Apply this BEFORE any allow policies. Establishes deny-all baseline. All intra-namespace and
cross-namespace traffic is blocked until explicitly allowed by a subsequent ALLOW policy.

**This is not optional.** Without default-deny, any pod in the cluster can call any service in the mesh —
mTLS authenticates the caller but does not restrict who can call. Default-deny is the enforcement layer.

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: notification             # Apply to each namespace separately — one deny-all per namespace
spec:
  {}                                  # Empty spec = deny all traffic.
                                      # No selector = applies to ALL pods in this namespace.
                                      # No action field needed — absence of allow rules means deny.
                                      #
                                      # Verification after applying:
                                      # kubectl exec <notification-pod> -n notification -- \
                                      #   curl -s user-profile-service:8080/healthz
                                      # Expected: connection refused or 403 RBAC denied.
                                      # Then add allow policies one by one, re-testing each.
```

Apply the deny-all to every namespace that contains mesh-enrolled services:
```bash
# Apply deny-all to each namespace
kubectl apply -f deny-all-notification.yaml
kubectl apply -f deny-all-ingress.yaml
kubectl apply -f deny-all-analytics.yaml

# Verify deny is active before adding allow policies
kubectl exec -n notification deploy/notification-service -c notification-service -- \
  curl -sv http://user-profile-service:8080/healthz 2>&1 | grep "< HTTP"
# Expected: 403 Forbidden (RBAC access denied)
```

### Policy 1: notification-service Allow Policy

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: notification-service-allow
  namespace: notification
spec:
  selector:
    matchLabels:
      app: notification-service       # Applies to pods with label app=notification-service
                                      # Verify pods have this label:
                                      # kubectl get pods -n notification -l app=notification-service
  action: ALLOW
  rules:
  # Rule 1: Allow message-router-service to POST notifications
  - from:
    - source:
        principals:
        - "cluster.local/ns/notification/sa/message-router-service"
                                      # SPIFFE principal in EKS short form.
                                      # "cluster.local" is the trust domain for EKS clusters.
                                      # Full SPIFFE URI: spiffe://cluster.local/ns/notification/sa/message-router-service
                                      # The caller pod MUST run under the service account "message-router-service"
                                      # in the "notification" namespace.
                                      # Verify SA: kubectl get pod <pod> -n notification -o jsonpath='{.spec.serviceAccountName}'
    to:
    - operation:
        methods: ["POST"]             # Only POST — message-router sends notifications, doesn't read
        paths: ["/api/v1/send"]       # Exact path — not /admin, not /internal/*, not /api/v2/*
        ports: ["8080"]               # Service port (string, not integer)

  # Rule 2: Allow api-gateway to trigger direct and bulk sends
  - from:
    - source:
        principals:
        - "cluster.local/ns/ingress/sa/api-gateway"
                                      # api-gateway is in the "ingress" namespace — cross-namespace
                                      # SPIFFE principal. The namespace must match where the
                                      # api-gateway pod actually runs (check with kubectl get pods -n ingress).
    to:
    - operation:
        methods: ["POST"]
        paths: ["/api/v1/send", "/api/v1/send/bulk"]
                                      # Two paths for api-gateway: single and bulk sends.
                                      # message-router only has access to single send — principle of least privilege.
        ports: ["8080"]

  # Rule 3: Health checks — allow any authenticated mesh identity
  - to:
    - operation:
        methods: ["GET"]
        paths: ["/healthz", "/readyz"]
                                      # No "from" restriction on health checks.
                                      # Any pod with a valid mesh cert can health-check.
                                      # This simplifies K8s liveness/readiness probe configuration —
                                      # probes come from the kubelet (not through Envoy), but
                                      # Prometheus scrapers come through the mesh and need this.
```

### Policy 2: user-profile-service Allow Policy

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: user-profile-service-allow
  namespace: notification
spec:
  selector:
    matchLabels:
      app: user-profile-service
  action: ALLOW
  rules:
  # Rule 1: Allow notification-service to read full user profiles
  - from:
    - source:
        principals:
        - "cluster.local/ns/notification/sa/notification-service"
                                      # notification-service needs profile data to personalize messages.
                                      # Example: fetch user's notification preferences, device tokens,
                                      # preferred language, DND schedule.
    to:
    - operation:
        methods: ["GET"]              # Read-only. notification-service NEVER modifies profiles.
                                      # If a future requirement adds writes, create a new rule — don't
                                      # change this to ["GET", "POST"] without a separate security review.
        paths: ["/api/v1/profile/*"]  # Wildcard: matches any user ID path.
                                      # /api/v1/profile/12345 ✓
                                      # /api/v1/profile/12345/devices ✓ (sub-paths also match)
                                      # /api/v1/admin ✗ (doesn't match this pattern)
        ports: ["8080"]

  # Rule 2: Allow analytics-service to read preference data only (not full profile)
  - from:
    - source:
        principals:
        - "cluster.local/ns/analytics/sa/analytics-service"
                                      # analytics-service is in a different namespace (analytics).
                                      # Cross-namespace: principal format includes the caller's namespace.
                                      # Principle of least privilege: analytics only reads preferences,
                                      # not the full profile (which may include PII like phone number).
    to:
    - operation:
        methods: ["GET"]
        paths: ["/api/v1/profile/*/preferences"]
                                      # More specific path than notification-service's access.
                                      # analytics-service cannot read /api/v1/profile/12345 (full profile) —
                                      # only /api/v1/profile/12345/preferences.
        ports: ["8080"]

  # Rule 3: Health checks
  - to:
    - operation:
        methods: ["GET"]
        paths: ["/healthz", "/readyz"]
```

## How to Discover Existing Service Identities

For services already running in the mesh before this register was established, use these commands to
discover their current SPIFFE identities and build the register retroactively.

```bash
# 1. List all service accounts in a namespace (these become the SPIFFE identities)
kubectl get serviceaccounts -n notification

# 2. Check which service account a specific pod uses
kubectl get pod <pod-name> -n notification -o jsonpath='{.spec.serviceAccountName}'

# 3. Verify a pod has an Istio sidecar (and therefore a SPIFFE cert)
kubectl get pod <pod-name> -n notification -o jsonpath='{.spec.containers[*].name}'
# Look for "istio-proxy" in the output

# 4. View the SPIFFE URI from a running pod's Envoy proxy cert
istioctl proxy-config secret <pod-name>.notification

# 5. Get the actual SPIFFE ID from the cert SAN field
kubectl exec <pod-name> -n notification -c istio-proxy -- \
  cat /var/run/secrets/istio/root-cert.pem | openssl x509 -noout -text | grep "URI:spiffe"

# 6. Discover all current cross-service calls in the mesh (from Istio telemetry)
# Run this in Prometheus to see all active service pairs in the last 24h:
# istio_requests_total{reporter="destination", destination_service_namespace="notification"}
# Group by: source_workload, destination_workload, source_principal, destination_principal

# 7. Check current AuthorizationPolicies in a namespace
kubectl get authorizationpolicies -n notification
kubectl describe authorizationpolicy notification-service-allow -n notification

# 8. Test if a specific call is allowed (from notification-service to user-profile-service)
kubectl exec -n notification deploy/notification-service -c notification-service -- \
  curl -sv http://user-profile-service:8080/api/v1/profile/test-user 2>&1 | grep "< HTTP"
# 200 = allowed, 403 = RBAC denied (check the allow policy), 503 = connection pool/circuit breaker
```

## Trust Domain Configuration

The SPIFFE trust domain is set at the Istio control plane level. Verify your trust domain before
building principals in AuthorizationPolicies — a mismatch causes silent deny.

```bash
# Check current trust domain
kubectl get configmap istio -n istio-system -o jsonpath='{.data.mesh}' | grep trustDomain

# Default EKS Istio install uses: cluster.local
# Custom domains (like t-mobile.com) require setting at istio install time:
# istioctl install --set values.global.trustDomain=t-mobile.com

# The principal format changes based on trust domain:
# cluster.local:  cluster.local/ns/<namespace>/sa/<service-account>
# t-mobile.com:   t-mobile.com/ns/<namespace>/sa/<service-account>
```

**Important**: the SPIFFE IDs in the register table use `t-mobile.com` as the trust domain (custom
production config). The AuthorizationPolicy YAML templates use `cluster.local` (default EKS). Verify
your actual trust domain and use one consistently. Mixed trust domains = policies that never match.

## PeerAuthentication: Enforcing mTLS Mode

AuthorizationPolicies control who can call whom. PeerAuthentication controls HOW they connect.
Both are required for a complete security posture.

```yaml
# Apply STRICT mTLS to the notification namespace
# This means: ALL traffic to pods in this namespace must use mTLS.
# Pods without Istio sidecars cannot send traffic to this namespace.
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: notification
spec:
  mtls:
    mode: STRICT                      # STRICT = only mTLS allowed. Plaintext connections rejected.
                                      # PERMISSIVE = mTLS and plaintext both accepted (migration mode).
                                      # Start with PERMISSIVE during mesh onboarding,
                                      # switch to STRICT once all services have sidecars.
                                      # Verify STRICT is safe: check mTLS coverage % in Grafana.
                                      # If mTLS coverage < 100%, you have services without sidecars —
                                      # STRICT mode will break them. Fix sidecar injection first.
```

**Migration sequence for moving to STRICT**:
1. Apply PERMISSIVE to namespace
2. Enable sidecar injection: `kubectl label namespace notification istio-injection=enabled`
3. Restart pods: `kubectl rollout restart deployment -n notification`
4. Verify mTLS coverage = 100% in Grafana (Panel 4 in the observability dashboard)
5. Apply STRICT mode
6. Run smoke tests on all service pairs
7. Update this register: mark "mTLS Mode" column as STRICT

## Quarterly Review Checklist

Run this checklist every 90 days. Assign to a named SRE. Sign off when complete.

```
Reviewer: _______________  Review Date: ___________  Next Review Due: ___________

Service Register Accuracy:
- [ ] All services in the register are still deployed and active
- [ ] No decommissioned services still have active AuthorizationPolicies
- [ ] All new services deployed in the quarter have entries in this register
- [ ] Service account names in register match actual K8s service accounts (kubectl get sa -n notification)

Caller Accuracy:
- [ ] All listed "Allowed Callers" are still active and still need access
- [ ] No services gained new callers in the quarter without a register entry
- [ ] Cross-namespace callers are still in the correct namespace
- [ ] No service account changes (renames, migrations) that break principal references

Policy Drift Detection:
- [ ] kubectl get authorizationpolicies -A output matches every row in this register
- [ ] No AuthorizationPolicies exist in the cluster that are NOT in this register
- [ ] PeerAuthentication STRICT mode confirmed for all production namespaces

Access Log Review:
- [ ] Reviewed Splunk for RBAC deny events in last 90 days
- [ ] Any denies > 0 investigated: either missing allow rule (update register) or unauthorized access (security incident)
- [ ] No new service pairs appearing in mesh telemetry that are not in this register

Sign-off: _______________  Date: ___________
```
