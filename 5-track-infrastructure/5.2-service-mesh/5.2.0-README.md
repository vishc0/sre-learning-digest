# Service Mesh Training Framework

This curriculum builds a Director/VP-level understanding of service mesh technology — what it solves, how it works, how to operate it, and how to make architectural decisions about it. The framework is structured in three layers: conceptual chapters that explain the "why" and "how," operational playbooks that guide real incident response, and reference material for fast lookups during design or interview prep. Each chapter uses middleware and Kubernetes analogies to anchor new concepts against what you already know from operating RabbitMQ, Kafka, and EKS at scale.

## How to Navigate This Framework

### If you have 2 hours (interview in 48 hours)

Read in this order:
1. `ch01` — foundations (the mental model you need for everything else)
2. `ch02` — Istio architecture (the most commonly tested topic)
3. `ch03` — mTLS and identity (Zero Trust framing for Director conversations)
4. `reference/service-mesh-comparison-matrix.md` — know the trade-offs cold
5. `ch08` — Director decision framework (the leadership layer)

### If you have a full week (depth preparation)

Read chapters 1–8 sequentially. Each chapter builds on the previous one. After chapter 4, read the `templates/` directory to ground the concepts in real YAML. After chapter 6, read the `playbooks/` to understand operational reality. Use the `reference/` directory throughout as a lookup resource.

## Prerequisites

This curriculum assumes you already have working knowledge of:

- **Kubernetes fundamentals**: pods, deployments, services, namespaces, CRDs, admission webhooks
- **EKS operations**: cluster networking, IAM roles for service accounts (IRSA), node groups
- **TLS basics**: what a certificate is, what a CA does, what mutual TLS means conceptually
- **Distributed systems concepts**: retries, timeouts, circuit breaking, health checks
- **Observability**: metrics, traces, logs — and how they differ

You do not need prior service mesh experience. Chapter 1 assumes zero mesh knowledge.

## How to Use Each Section

### Chapters (`ch01` through `ch08`)

Conceptual content. Read sequentially. Each chapter opens with the problem being solved before introducing the solution. Key terms are **bolded on first use**. Director-level framing is called out explicitly so you know what to emphasize in a leadership conversation vs a technical deep-dive.

### Playbooks (`playbook/`)

Operational procedures. These are not tutorials — they are step-by-step guides written as if you are handing them to your on-call engineer at 2am. Read them to understand how concepts translate into operations. In an interview, referencing a concrete operational procedure ("we would phase the mTLS migration using PERMISSIVE mode first, then gate on a cert rotation drill before flipping to STRICT") signals operational maturity.

### Templates (`templates/`)

Annotated YAML and specification templates. Every field is commented. Use these to verify you can read and reason about real Istio configurations. Do not memorize syntax — do understand what each field does and why it's there.

### Reference (`reference/`)

Fast-lookup tables. Not designed for sequential reading. Use during active design work or as a warm-up before interviews — scan the comparison matrix and CRD quick reference to refresh key facts.
