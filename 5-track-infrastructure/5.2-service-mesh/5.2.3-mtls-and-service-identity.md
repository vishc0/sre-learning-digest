# Chapter 3: mTLS and Service Identity

## Why Identity, Not Location

In a traditional network security model, you trust traffic based on where it comes from. A firewall rule allows packets from 10.0.1.0/24. A service trusts requests from IP 10.0.1.5 because that IP belongs to the payment service.

Kubernetes breaks this model completely.

Pod IPs are ephemeral. A pod dies, gets rescheduled, and comes back on a different IP — often within seconds. The IP 10.0.1.5 that was the payment service five minutes ago is now the logging agent. Network-location-based trust is meaningless in a dynamic container environment. You cannot trust "this traffic came from 10.0.1.5" because that address has no stable relationship to any particular service.

What you need is **identity**, not **location**. A cryptographic identity that travels with the workload regardless of which node it runs on, which IP it gets assigned, or how many times it gets rescheduled. This is the problem that **SPIFFE** and **mTLS** solve together.

---

## SPIFFE: Workload Identity as a Standard

**SPIFFE** (Secure Production Identity Framework for Everyone) is an open standard that defines what a workload identity looks like and how it is issued and verified. It is not a product — it is a specification, analogous to how OAuth 2.0 is a specification for authorization flows.

Every workload in a SPIFFE-compliant system gets a SPIFFE ID. The format is a URI:

```
spiffe://<trust-domain>/<path>
```

For example, on your T-Mobile notification platform:

```
spiffe://t-mobile.com/notification/message-router
spiffe://t-mobile.com/notification/delivery-service
spiffe://t-mobile.com/notification/preference-store
```

The trust domain (`t-mobile.com`) scopes the identity to your organization. The path identifies the specific workload. This SPIFFE ID is embedded in the workload's X.509 certificate as a Subject Alternative Name (SAN) field.

Now when the delivery-service receives a request from the message-router, it does not check the caller's IP. It checks the caller's certificate. The certificate's SAN contains `spiffe://t-mobile.com/notification/message-router`. That identity is cryptographically signed by the mesh CA — it cannot be forged. The IP address of the caller is irrelevant.

The RabbitMQ parallel: a SPIFFE ID is the routing key of identity. In RabbitMQ, a routing key tells the exchange what a message is and where it belongs. A SPIFFE ID tells the mesh what a workload is and who it is — regardless of the transport-level details of where it is running. Just as a routing key is embedded in the message envelope, a SPIFFE ID is embedded in the certificate envelope. Both are verified, not assumed.

## SPIRE: The Reference Implementation

**SPIRE** (SPIFFE Runtime Environment) is the reference implementation of the SPIFFE specification — the actual software that issues and manages SPIFFE-format certificates for workloads.

You do not need to run SPIRE separately when using Istio. istiod includes a built-in SPIRE-compatible CA that handles the full certificate lifecycle: issuing SPIFFE-format certificates to Envoy sidecars, managing renewals, and distributing the trust bundle (the CA root certificate) to all workloads so they can verify each other's identities.

SPIRE becomes relevant when you need cross-cluster or cross-platform identity federation — for example, when a workload in EKS needs to authenticate to a workload running in a VM outside the mesh, or when two separate Istio meshes need to trust each other's identities. In those cases, a standalone SPIRE deployment handles the trust domain federation. For a single Istio mesh, istiod's built-in CA is sufficient.

## mTLS: Both Sides Prove Who They Are

Standard TLS (the HTTPS you use in a browser) is one-directional: the client verifies the server's certificate. The server does not verify who the client is — it just serves the response.

**mTLS** (mutual TLS) is bidirectional: both the client and server present certificates and verify each other before any application data flows.

```
Normal TLS:
[Client] ──verify server cert──→ [Server]

Mutual TLS (mTLS):
[Client] ←─verify client cert──→ [Server]
         ──verify server cert──→
Both sides present and verify certificates
```

In a service mesh, mTLS between every service pair means:

- The delivery-service knows it is talking to the real preference-store (not a compromised pod that somehow got on the same network segment).
- The preference-store knows the caller is genuinely the delivery-service (not a rogue workload that spun up and started making API calls).

Neither side needs to manage API keys, shared secrets, or token exchange. The identity proof is in the certificate, and the certificate is automatically issued and rotated by istiod.

## Certificate Lifecycle in Istio

This is the sequence that happens every time a new pod joins the mesh:

1. **Pod starts.** The `istio-proxy` (Envoy) sidecar initializes.

2. **CSR generated.** Envoy generates a private key on the fly and creates a **CSR** (Certificate Signing Request) — a request asking the CA to issue a certificate for a specific identity. The CSR includes the workload's intended SPIFFE ID, derived from the pod's Kubernetes service account.

3. **Kubernetes identity verification.** Envoy sends the CSR to istiod along with the pod's Kubernetes service account token (a JWT). istiod validates the JWT against the Kubernetes API server to confirm the pod's service account identity. This is the attestation step — istiod verifies that the pod is who it claims to be before issuing a certificate.

4. **Certificate issued.** istiod signs the CSR with its CA private key and returns an **X.509** certificate containing the SPIFFE ID in the SAN field. The certificate has a short default lifetime of 24 hours.

5. **Certificate stored in memory.** Envoy stores the certificate and private key in memory — never written to disk, never stored in Kubernetes Secrets. This limits the blast radius of a compromise. A certificate extracted from a running container's memory is already nearly expired.

6. **Automatic rotation.** At 80% of the certificate lifetime (approximately 19 hours into a 24-hour cert), Envoy proactively requests a new certificate from istiod. The new certificate is loaded in place. Active mTLS connections complete normally; new connections use the new certificate. Zero downtime, zero human action.

The short-lived certificate model is the key security property. Contrast with a traditional approach where a service account credential is issued once, stored in a Secrets Manager, rotated every 90 days (if you're lucky), and potentially compromised for the entire 90-day window before anyone notices. Istio certificates expire in hours. A stolen certificate is nearly worthless by the time anyone could use it.

## PeerAuthentication: Configuring mTLS Mode

**PeerAuthentication** is the Istio CRD that controls how the mesh handles mTLS for inbound connections to a workload. It has three modes:

**STRICT mode**: All inbound connections must use mTLS. Plaintext connections are rejected at the Envoy layer before they reach the application. Use this in production once every service in the namespace is in the mesh and capable of presenting mTLS credentials.

**PERMISSIVE mode**: Envoy accepts both mTLS and plaintext connections. A service calling with valid mTLS credentials gets authenticated and the SPIFFE identity is extracted. A service calling with plaintext (e.g., a legacy service not yet in the mesh) is still allowed through, but with no identity. Use PERMISSIVE during mesh migration — it lets you add services to the mesh gradually without breaking existing callers that haven't been injected yet.

**DISABLE mode**: mTLS is turned off. Avoid in production.

Example — applying STRICT mTLS across the notification namespace:

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: notification
spec:
  mtls:
    mode: STRICT
```

A `name: default` PeerAuthentication in a namespace applies to all workloads in that namespace. You can also apply it mesh-wide by placing it in the `istio-system` namespace, or scope it to a specific workload using a `selector`.

The standard rollout pattern: deploy mesh with PERMISSIVE mode. Verify all services are injected and communicating via mTLS (check Kiali's security graph or the `istioctl` CLI). Run a cert rotation drill. Then flip to STRICT. This is a non-breaking migration — PERMISSIVE means you never have a cutover moment that breaks traffic.

## mTLS and the RabbitMQ Client Certificate Parallel

When you configure RabbitMQ for maximum client security, you set `ssl_options.verify = verify_peer` and `ssl_options.fail_if_no_peer_cert = true`. The broker refuses connections from any client that cannot present a valid certificate. You issued those client certificates from your own CA. That is mutual TLS — the broker verifies the client, and the client verifies the broker.

Istio's STRICT mode is exactly this, applied automatically to every service pair in the mesh. The difference is scale and automation: instead of you manually issuing and distributing client certificates to 40 services, istiod does it for every workload, every 24 hours, with no human intervention.

## The Foundation for Authorization

mTLS gives you **authentication** — you know who is calling. SPIFFE identity gives you a stable, cryptographically verified answer to "who is this service?"

The next layer is **authorization** — knowing who a caller is does not automatically mean they are allowed to do what they are trying to do. AuthorizationPolicy (covered in Chapter 6) builds on top of mTLS and SPIFFE identity to answer that question: the message-router-service has identity `spiffe://t-mobile.com/notification/message-router`, and yes, that identity is permitted to call the `POST /deliver` endpoint on the delivery-service.

Without mTLS providing verified identity, AuthorizationPolicy has nothing to authorize against. mTLS is the prerequisite for Zero Trust between services.

## T-Mobile Context: What This Buys You at 25M Messages/Day

On the notification platform, your message-router, delivery-service, preference-store, and retry-worker need to communicate with each other constantly. Without a service mesh, any pod that can reach those services on the right port can call them. A misconfigured service, a compromised container image, or a runaway debug tool could make API calls that look like legitimate traffic.

With mTLS in STRICT mode, only pods with certificates issued by the mesh CA — certificates that cryptographically bind to a Kubernetes service account — can initiate connections. And only the service accounts that the message-router is supposed to use can get a certificate with the message-router's SPIFFE ID. A rogue pod cannot impersonate the message-router unless it also compromises the Kubernetes service account, which is a much harder attack.

The security posture improved dramatically, and no team had to change a single line of application code to get there."
