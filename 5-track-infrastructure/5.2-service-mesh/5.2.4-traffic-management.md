# Chapter 4: Traffic Management

## The RabbitMQ Mental Model

Before touching Istio-specific YAML, anchor this to what you already know. In RabbitMQ, two constructs determine how messages flow: the **exchange** (routing rules — decides where a message goes based on routing keys, headers, or topic patterns) and the **queue binding configuration** (defines the behavior of how messages are consumed from the queue once they arrive).

Istio traffic management maps directly onto this:

- **VirtualService** = the exchange routing rules. It intercepts traffic destined for a service and makes routing decisions based on HTTP headers, URI paths, or weighted percentages. "Where does this request go?"
- **DestinationRule** = the queue binding configuration. It defines HOW traffic behaves once it reaches the destination — load balancing algorithm, connection pool limits, circuit breaking thresholds. "How does traffic behave after routing?"

Both are required. VirtualService without DestinationRule means routing works but behavior policies don't apply. DestinationRule without VirtualService means policies apply but you can't split or match traffic.

## VirtualService: Routing Rules

A **VirtualService** attaches to a Kubernetes service's hostname and intercepts all traffic destined for that service. Envoy evaluates the match rules top-to-bottom and routes to the first match.

### Traffic Splitting (Canary Deployments)

The canonical canary pattern — send a percentage of traffic to a new version while keeping the majority on the stable version:

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: notification-service
spec:
  hosts:
  - notification-service
  http:
  - route:
    - destination:
        host: notification-service
        subset: v1
      weight: 80
    - destination:
        host: notification-service
        subset: v2
      weight: 20
```

The `subset` values (`v1`, `v2`) are defined in the companion DestinationRule. Kubernetes sees one service; Envoy splits traffic based on pod labels.

### Header-Based Routing

Route specific requests to a specific version based on HTTP headers:

```yaml
http:
- match:
  - headers:
      x-version:
        exact: canary
  route:
  - destination:
      host: notification-service
      subset: v2
- route:
  - destination:
      host: notification-service
      subset: v1
```

This lets QA teams test v2 in production by setting `x-version: canary` in their requests, without exposing v2 to real user traffic.

### Retries

```yaml
retries:
  attempts: 3
  perTryTimeout: 5s
  retryOn: 503,connect-failure,retriable-4xx
```

**Critical warning**: **retries** must only be configured for idempotent operations. In the T-Mobile notification platform, retrying a read to user-profile-service is safe — if it fails and retries, the same data comes back. Retrying a "send SMS via carrier API" call is not safe — the carrier may have already processed the first attempt, and a retry sends a duplicate message to the customer. Never configure retries on anything with side effects.

### Timeouts

```yaml
http:
- timeout: 10s
  route:
  - destination:
      host: user-profile-service
      subset: v1
```

**Timeouts** are your defense against cascading slowness. If user-profile-service is degraded and takes 30 seconds to respond, every caller that lacks a timeout will hold a connection open for 30 seconds, exhausting its own connection pool, which cascades to every service above it. Set timeouts aggressively — if a service normally responds in under 500ms, a 5-second timeout is already generous.

### Fault Injection

```yaml
fault:
  delay:
    percentage:
      value: 10
    fixedDelay: 2s
  abort:
    percentage:
      value: 5
    httpStatus: 503
```

**Fault injection** is chaos engineering built into mesh configuration — no Chaos Monkey, no custom tooling. The example above: 10% of requests to this service will experience a 2-second delay; 5% will receive an immediate 503. Use this to answer: "How does notification-service behave if user-profile-service has a 2-second delay?" Run the test in a staging namespace, measure error rates and downstream impact, then remove the fault block. The application is untouched.

## DestinationRule: Behavior Policies

A **DestinationRule** defines what happens after VirtualService routing has selected a destination. It controls three things: subset definitions, load balancing, and resilience policies.

### Subset Definitions

Subsets map to pod labels. `v1` pods have `version: v1` in their labels; `v2` pods have `version: v2`. Envoy uses this to enforce the traffic split defined in VirtualService:

```yaml
subsets:
- name: v1
  labels:
    version: v1
- name: v2
  labels:
    version: v2
```

### Load Balancing

```yaml
trafficPolicy:
  loadBalancer:
    simple: LEAST_CONN
```

Options: `ROUND_ROBIN` (default), `LEAST_CONN` (send to the instance with fewest active requests — better for variable-latency services), `RANDOM`. For the notification platform, `LEAST_CONN` is usually the right choice for services with unpredictable response times.

### Circuit Breaking via outlierDetection

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: notification-service
spec:
  host: notification-service
  trafficPolicy:
    outlierDetection:
      consecutiveErrors: 5
      interval: 30s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
```

**Circuit breaking** is the electrical analogy applied to software: in electrical engineering, a circuit breaker trips when current exceeds safe limits, cutting power to protect the downstream equipment. In software, the **outlierDetection** policy monitors each pod individually. If a specific pod returns 5 consecutive 5xx errors within a 30-second window, Envoy ejects that pod from the load balancing pool for 30 seconds — traffic stops flowing to it. After 30 seconds, Envoy sends a probe request. If it succeeds, the pod is reintroduced. If not, the ejection time doubles (exponential backoff).

In RabbitMQ terms: this is equivalent to routing messages to a dead-letter exchange when a consumer fails repeatedly. The problematic consumer is isolated until it signals recovery. The difference is that outlierDetection does this automatically at the infrastructure layer — no application code change required.

`maxEjectionPercent: 50` is a critical safety valve. Even if all pods are failing, Envoy will never eject more than 50% of the pool — you always retain at least half your capacity to serve requests rather than triggering a total outage.

## Canary Deployment: The T-Mobile Use Case

The notification platform is rolling out a new message routing algorithm in v2. Traditional big-bang deployment: update all pods, monitor, roll back if broken. Risk: you discover the bug when 100% of your 25M daily messages are affected.

With **traffic splitting**:

1. Deploy v2 pods alongside v1 pods. Both registered under the same Kubernetes service.
2. VirtualService: 95% to v1, 5% to v2. Monitor error rate and P99 latency for v2 via `istio_request_duration_milliseconds` histogram. No change to app code — the traffic split is infrastructure config.
3. If clean after 24 hours: move to 80/20.
4. 48 hours clean: 50/50.
5. Final: 0/100. Delete v1 pods and remove the VirtualService split.

At no point did a developer add a feature flag to application code. The entire rollout strategy lived in two YAML files checked into Git. If v2 shows elevated error rates at step 2, roll back by changing the weight to 100/0 — no deployment, no pod restart.

## Key Terms

| Term | What It Does |
|------|-------------|
| **VirtualService** | Defines routing rules: where traffic goes based on weight, headers, or path |
| **DestinationRule** | Defines behavior policies: load balancing, circuit breaking, TLS settings |
| **traffic splitting** | Weighted routing across multiple versions for canary or blue/green deployments |
| **circuit breaking** | Automatic ejection of failing pods via outlierDetection |
| **outlierDetection** | The DestinationRule policy that implements circuit breaking |
| **fault injection** | Injecting artificial delays or errors for chaos/resilience testing |
| **retries** | Automatic retry with backoff — only safe for idempotent operations |
