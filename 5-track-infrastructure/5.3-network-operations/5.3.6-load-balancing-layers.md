# Chapter 6: Load Balancing Layers

## The Core Idea

L4 load balancing is like a traffic cop routing cars by destination city — they see the address on the windshield and point you down the right highway. They have no idea whether your car is carrying groceries or hazardous materials. L7 is like a customs inspector who reads the shipping manifest: they see what's inside, check the labels, and can route based on content — this package goes to Warehouse A, that package goes to cold storage. More information enables smarter routing, but the inspection costs something.

Understanding which layer to operate at — and why — is the difference between building a load balancer config that works and building one that survives an incident.

## L4 Load Balancing

L4 operates at the TCP/UDP layer. The load balancer sees source IP, destination IP, and port. It knows nothing about the application protocol running over that connection.

**What L4 can do:**
- Distribute connections across backend servers using round-robin, least-connections, or IP-hash algorithms
- Handle millions of concurrent connections with minimal overhead (modern implementations use kernel-bypass techniques like DPDK)
- Work with any TCP or UDP protocol — HTTP, gRPC, database connections, RabbitMQ AMQP, game server UDP, SMTP

**What L4 cannot do:**
- Inspect HTTP headers, URL paths, or cookies
- Terminate SSL (the TLS handshake is above L4)
- Route based on application-layer attributes
- Apply rate limiting by API key or user identity

**IP hash for sticky sessions** deserves specific attention. When a client's IP always hashes to the same backend, session state stored locally on that backend is always reachable. This is a pattern you encounter in stateful middleware — RabbitMQ consumer affinity, for example. The operational risk: if a backend fails, all sessions hashed to that backend are lost simultaneously, and rebalancing is uneven because the hash table shifts. Session affinity at L4 is a debt that eventually becomes an incident. The director-level preference: make backends stateless and let the session state live in Redis or a database.

## L7 Load Balancing

L7 operates at the HTTP(S) layer. The load balancer terminates the TLS connection, reads the decrypted HTTP request, and makes routing decisions based on what it sees.

**Routing capabilities at L7:**
- Path-based routing: `/api/*` → backend-api service, `/static/*` → CDN or asset server, `/auth/*` → auth service
- Header-based routing: route requests bearing `X-Beta-User: true` to canary backends
- Host-based routing: `api.example.com` and `app.example.com` handled by different backend pools on the same LB
- Cookie-based routing: sticky sessions by cookie value (more reliable than IP hash because it survives NAT and shared IPs)

**SSL termination at L7** means your backend services receive plain HTTP (or re-encrypted HTTP over an internal TLS connection if you're doing end-to-end encryption). The LB handles the full TLS handshake — certificate management, cipher negotiation, session resumption — and your backends are relieved of that overhead. This is where your certificates live and where you enforce TLS version policy.

**Additional capabilities that live at L7:**
- Rate limiting by IP, header value, or API key
- WAF (Web Application Firewall) — inspect request content for injection attacks, malformed payloads
- Circuit breaking — stop forwarding to a backend that is returning errors above a threshold
- Request/response transformation — rewrite headers, inject correlation IDs

The overhead of L7 inspection is real but usually negligible at application scale. For workloads requiring extreme throughput (millions of requests per second, sub-millisecond overhead budgets), L4 is the right layer. For everything else, the routing intelligence of L7 is worth the marginal cost.

## Tool Landscape

| Tool | Layer | Strengths | Where It Fits |
|------|-------|-----------|---------------|
| HAProxy | L4 + L7 | Extremely fast, battle-tested, fine-grained config control | Bare-metal, VM fleets, high-performance TCP workloads |
| NGINX | L7 (+ limited L4) | Web server + reverse proxy + LB in one, wide community knowledge | Kubernetes Ingress, API gateway, static serving |
| AWS ALB | L7 | Managed, native AWS integration, auto-scaling, WAF integration | AWS workloads needing HTTP routing without ops overhead |
| AWS NLB | L4 | Ultra-low latency, static IPs, preserves source IP | TCP workloads, non-HTTP protocols, PrivateLink |
| Traefik | L7 | Dynamic config via service discovery, Kubernetes-native | Kubernetes Ingress with auto cert management |

HAProxy's reputation in high-stakes environments (GitHub ran it at massive scale, Twitter used it for years) comes from its predictability under load and its granular configuration surface. It does not abstract away complexity — which is a feature, not a bug, when you need to tune behavior precisely. The cost is that config-file-driven management does not compose well with Kubernetes-native tooling.

AWS ALB's value is operational: you don't manage software versions, you don't plan for certificate rotation processes, and it scales transparently. The cost is that you are in the AWS billing model (per LCU consumed) and debugging requires CloudWatch, not a local log file.

## Kubernetes Load Balancing

Kubernetes has three distinct load balancing surfaces, each at a different layer, and conflating them is a common source of misconfiguration.

**Service (ClusterIP / NodePort / LoadBalancer)** — kube-proxy implements L4 load balancing using iptables or IPVS rules on every node. When a pod calls `http://my-service`, the request is distributed across all healthy pods backing that Service. This is L4: round-robin across pod IPs, no HTTP awareness. A LoadBalancer Service type provisions a cloud load balancer (NLB on AWS) that accepts external traffic and forwards to the Service.

**Ingress** — L7 HTTP routing inside the cluster. An Ingress object defines routing rules (host + path → backend Service). An Ingress controller (NGINX, Traefik, AWS ALB Ingress Controller) watches these objects and configures an actual load balancer accordingly. This is where you implement path-based routing, SSL termination, and rate limiting for HTTP traffic entering the cluster. One Ingress controller can consolidate routing rules for dozens of services behind a single external endpoint.

**Service Mesh (Istio, Linkerd)** — L7 inside the cluster, between services. Every pod gets a sidecar proxy (Envoy for Istio). Service-to-service traffic flows through these proxies, enabling mTLS, fine-grained circuit breaking, traffic splitting for canary deployments, and detailed observability (request latency histograms per service pair, error rates).

The director-level read on service mesh: it is often over-engineered for the team that installs it. Istio in particular carries significant operational complexity — the control plane has multiple components, config surface is large, and debugging misbehavior requires understanding Envoy internals. Teams without dedicated platform engineering capacity consistently underestimate this. Start with Ingress + well-designed health checks. Add service mesh when you have specific requirements (mTLS between services for compliance, sophisticated traffic splitting, per-service circuit breaking) that Ingress cannot satisfy.

## Health Check Design: Where Reliability Lives

Health checks are the mechanism by which a load balancer decides whether a backend is eligible to receive traffic. Poor health check design is one of the most reliable ways to turn a partial failure into a total outage.

**Shallow health check (TCP connect)** — the LB opens a TCP connection to the backend port. If it succeeds, the backend is marked healthy. Problem: the TCP port can be open while the process is completely unable to serve requests (deadlocked thread pool, corrupted state, misconfigured dependency). A process that returns HTTP 500 on every request will pass a TCP health check indefinitely.

**Deep health check (HTTP /health endpoint)** — the LB makes an HTTP GET to `/health` and expects a 200 response. The application verifies it can serve requests and responds accordingly. This is the right default. The key design question: what does "healthy" mean for your application?

**Health check poisoning** is the failure mode that makes deep health checks dangerous when designed incorrectly. If your `/health` endpoint checks all downstream dependencies — database connectivity, cache availability, message queue connection — then a database outage marks every application instance as unhealthy simultaneously. The LB removes all backends from rotation. Your application is now completely unavailable, not because the app is broken, but because one dependency is degraded.

The correct design principle: `/health` answers "can I serve HTTP requests right now." It checks local state only — is the process up, is the thread pool accepting work, is there a fatal configuration error? It does not check downstream dependencies.

Kubernetes splits this into two probes:
- **Liveness probe** — if this fails, restart the pod. Use for detecting deadlocks or fatal corruption. Should be shallow.
- **Readiness probe** — if this fails, remove the pod from Service endpoints (stop sending traffic). Use for "I'm not ready to serve" states — startup warmup, temporary overload, graceful drain during shutdown. Can check slightly more, but still avoid cascading dependency checks.

The operational pattern that works: liveness checks local process health (TCP or simple HTTP), readiness checks whether the app has completed startup and can serve its first request, and a separate `/status` or `/health/deep` endpoint (not connected to the LB health check) surfaces dependency health for observability tooling.

## Director Decision Frame

Load balancing decisions surface in three contexts at director level: cost reviews, incident retrospectives, and architecture reviews for new platforms.

In cost reviews: ALB billing by LCU can become significant at high request rates. Know your LCU consumption. Know whether consolidating Ingress controllers across teams (one per cluster vs. one per team) changes the billing profile.

In incident retrospectives: the most common LB-related incident patterns are health check misconfiguration (poisoning, as above), SSL certificate expiry on the LB termination point, and connection pool exhaustion when backends are slow and the LB keeps sending new connections. Standardize health check design as a platform policy, not a per-team decision.

In architecture reviews, the key questions:
- What layer do we need to operate at for this workload? If it's not HTTP, L4. If it's HTTP and we need path routing or SSL termination, L7.
- Are we using Kubernetes? Then we need an Ingress controller decision and a documented health check standard.
- Is the team asking for a service mesh? What specific problem are they solving, and is there a simpler solution? Ingress + circuit breaking at the application layer (Resilience4j, Hystrix) solves most of what teams install Istio for.
- What is our health check standard? Is `/health` returning only local state, or are we inadvertently building a cascading failure mechanism?

## Key Takeaways

- L4 load balances at TCP/UDP level — fast, protocol-agnostic, no HTTP awareness. Use it for non-HTTP workloads (databases, message queues, raw TCP) and extreme throughput requirements.
- L7 load balances at HTTP level — path routing, header inspection, SSL termination, WAF, rate limiting. Use it for all HTTP/gRPC traffic where routing intelligence matters.
- In Kubernetes, Service = L4 (kube-proxy), Ingress = L7 HTTP (controller required), Service Mesh = L7 service-to-service (significant operational cost).
- Service mesh is frequently over-engineered. Start with Ingress and good health checks. Add service mesh only for specific, named requirements that simpler solutions cannot satisfy.
- Health check poisoning — where `/health` checks downstream dependencies and causes mass removal from rotation during a partial outage — is one of the most operationally dangerous antipatterns in load balancer design. Standardize health check contracts as a platform policy.
