# BGP Outage Response Playbook

## Purpose
Use this playbook when traffic drops to specific prefixes, BGP sessions go down, or prefix hijacking is suspected — any event where Border Gateway Protocol failure is the root cause or prime suspect.

---

## Severity Classification

| Severity | Condition | Response Time |
|----------|-----------|---------------|
| P0 | Complete loss of external reachability / all BGP sessions down / confirmed prefix hijack | Immediate — wake the team |
| P1 | Single ISP BGP session down, partial traffic loss, suspicious route anomaly | 15 minutes |
| P2 | BGP session flapping but stable, degraded but not failing, single prefix withdrawn | 30 minutes |
| P3 | BGP route count anomaly, non-impacting route change detected | Next business hour |

---

## Prerequisites

- [ ] Access to network device console (router/switch CLI via jump host or out-of-band)
- [ ] Permission to modify BGP route policies on edge routers
- [ ] ISP NOC contact numbers for each upstream provider (stored in your runbook vault)
- [ ] Access to Splunk with `index=network` read permission
- [ ] Access to BGP monitoring tools: BGPMon, Cloudflare Radar BGP, RIPE RIS
- [ ] Looking glass access: bgp.he.net, route-views.org (no login required — public)
- [ ] RPKI/ROA dashboard access (Cloudflare RPKI Portal or your RIR's portal)

## Detection Signals

**Alert-driven entry points:**
- Network monitoring alert: "BGP session DOWN — peer AS[XXXX]"
- Synthetic monitor: external destinations unreachable from probe nodes
- Application error rate spike on Splunk: HTTP 5xx from external clients
- ISP direct notification (rare but happens)

**Splunk detection search:**
```splunk
index=network sourcetype=bgp_logs
| where event="route_withdrawn"
| stats count by prefix
| sort -count
```
Expected output: a table of prefixes with withdrawal counts. Any count > 0 is a signal. High counts on your own prefixes = serious.

**Secondary Splunk check — traffic drop correlation:**
```splunk
index=network sourcetype=netflow
| bucket _time span=5m
| stats sum(bytes) as total_bytes by _time, dst_prefix
| where dst_prefix="YOUR.PREFIX.HERE/24"
| timechart span=5m sum(total_bytes) as bytes_to_prefix
```
Expected output: a time series. A cliff edge (sudden drop to near-zero) confirms traffic impact.

## Phase 1: Immediate Response (0–15 min)

### Step 1: Confirm the scope — session down vs. routes withdrawn

These are different problems with different fixes. Do not skip this step.

On your edge router (IOS/IOS-XE syntax — adapt for your vendor):

```
show ip bgp summary
```

Expected healthy output:
```
Neighbor        V    AS MsgRcvd MsgSent   TblVer  InQ OutQ Up/Down  State/PfxRcd
203.0.113.1     4  65001   45231   44892   891234    0    0 2d03h         12450
```

**If State/PfxRcd shows `Idle`, `Active`, or `Connect` instead of a number — the session is DOWN.**
**If State/PfxRcd shows a number much lower than baseline — routes have been withdrawn.**

Know your baseline prefix counts per peer. Write them in your runbook now, before you need them.

```
show bgp neighbors 203.0.113.1
```

Look for:
- `BGP state = Established` (good) vs. `BGP state = Idle/Active` (session down)
- `Last reset:` — what caused the last reset? Read the reason carefully.
- `Prefixes received:` count — compare to your expected baseline

### Step 2: Determine if your prefixes are visible externally

Do NOT rely only on your own router. Use public looking glasses to verify external perspective.

**bgp.he.net (Hurricane Electric):**
1. Go to https://bgp.he.net
2. Enter your IP prefix in the search box (e.g., `198.51.100.0/24`)
3. Click "BGP Peer Info" — shows which ASes are announcing this prefix globally
4. If YOUR AS number is absent from all peers: your prefix is withdrawn globally

**route-views.org (public route collector):**
```
telnet route-views.org
# No password needed
route-views> show ip bgp 198.51.100.0/24
```
Expected output if reachable: BGP path attributes showing your AS in the AS_PATH
Expected output if withdrawn: `% Network not in table`

**Cloudflare Radar BGP (visual):**
- https://radar.cloudflare.com/bgp
- Search your prefix — shows global visibility and any anomalies in real time

### Step 3: Check for prefix hijacking

A hijack means someone else is announcing YOUR prefix and attracting traffic that should come to you.

Signs of a hijack:
- Your prefix IS visible in looking glasses, but the AS_PATH ends in someone else's AS number
- Traffic continues to show in your monitoring, but you're not receiving it
- BGPMon sends you a hijack alert (sign up free at bgpmon.net)

**Quick check at bgp.he.net:**
- In the BGP peer info for your prefix, look at the "Origin AS" column
- If ANY entry shows an AS number that is NOT yours or your known transit provider — treat it as a hijack

**Verify RPKI/ROA status:**
```
# Check RPKI validity of your prefix via Cloudflare
curl -s "https://rpki.cloudflare.com/api/v1/validity/YOUR-ASN/198.51.100.0/24"
```

Response will show `valid`, `invalid`, or `not-found`. If `not-found`, you have no RPKI protection. If `invalid`, your ROA is misconfigured.

## Phase 2: Diagnosis (15–30 min)

### Step 4: Session down — identify the cause

```
show bgp neighbors 203.0.113.1 | include (BGP state|Last reset|Notification|Hold time|TCP)
```

Common causes and their signatures:

| Symptom in output | Likely Cause |
|-------------------|--------------|
| `Notification Error: Hold Timer Expired` | Keepalive packets not reaching peer — check physical link, firewall rules blocking TCP 179 |
| `Notification Error: Cease/Admin Reset` | Peer deliberately reset the session — check with ISP |
| `TCP connection failed` | Layer 3 reachability to peer IP lost — check underlay routing |
| `Notification Error: Update Error` | Bad route update sent or received — check recent route policy changes |

**Check if TCP 179 is reachable to the peer:**
```
telnet 203.0.113.1 179
```
If connection hangs or refused: the path to the BGP peer itself is broken — this is a Layer 3 or firewall issue, not a BGP issue.

### Step 5: Routes withdrawn — identify which prefixes and why

```
show ip bgp regexp _65001_
```
Replace `65001` with the peer AS. Shows all routes received from that peer.

```
show ip bgp neighbors 203.0.113.1 routes
```
Shows routes being advertised TO that peer. Verify your prefixes are in this output.

If your prefixes are missing from what you're advertising: check route policy / prefix lists for accidental filtering.

```
show route-map [your-outbound-policy-name]
show ip prefix-list [your-prefix-list-name]
```

## Phase 3: Mitigation (30–60 min)

### Step 6: Fail traffic to secondary ISP (if you have one)

Option A — BGP Local Preference (prefer secondary temporarily):
```
router bgp [your-AS]
 neighbor [secondary-ISP-peer] route-map PREFER-SECONDARY in
!
route-map PREFER-SECONDARY permit 10
 set local-preference 200
```
Higher local-preference = preferred. Default is 100.

Option B — AS-PATH Prepending (make primary path less preferred to upstream):
```
route-map PREPEND-PRIMARY permit 10
 set as-path prepend [your-AS] [your-AS] [your-AS]
```
Add this outbound to your primary ISP peer. Traffic will prefer alternate paths.

Option C — Withdraw routes from failing peer temporarily:
```
neighbor 203.0.113.1 shutdown
```
This kills the session cleanly. Traffic will fail over to secondary ISP via normal BGP reconvergence.

**After any route manipulation — verify with looking glass within 2-3 minutes that external view matches your intent.**

### Step 7: Contact upstream ISP NOC

Do this in parallel with Step 6, not after.

**What to say (exact script):**
> "This is [your name] from [your company], account number [XXXX]. I am experiencing a BGP session outage on circuit [circuit-ID]. The session to our peer IP [peer-IP] in AS [peer-AS] has been [down / flapping] since approximately [timestamp with timezone]. We are seeing [describe impact: traffic loss percentage, affected prefixes]. I need emergency escalation. Can you confirm if there are any known issues on your network affecting BGP sessions in [location/region]?"

Have ready:
- Your circuit ID (on your ISP contract and usually labeled on your router interface config)
- The peer IP and AS number
- The time the issue started
- Output from `show bgp neighbors` to paste if asked

**For hijack events — escalate faster:**
> "We believe our prefix [X.X.X.X/YY] is being announced by an unauthorized AS. I need to speak with your RPKI/routing security team immediately. This is a network security incident."

---

## Phase 4: Resolution

### Step 8: Verify session restoration

```
show ip bgp summary
```
Wait for State/PfxRcd to return to expected prefix count. Do not declare resolved until you see stable counts for at least 5 minutes.

```
show ip bgp neighbors 203.0.113.1 | include (Up/Down|State|Prefixes)
```

### Step 9: Verify external reachability

Re-run looking glass checks from Phase 1, Step 2. Confirm:
- Your prefix is visible from multiple vantage points
- AS_PATH shows YOUR AS as the origin (no hijack residue)
- Route counts returned to baseline

### Step 10: Restore any failover changes made in Phase 3

Reverse any BGP prepending, local-preference changes, or peer shutdowns made during mitigation. Verify traffic distribution returns to expected split across ISPs.

## Phase 5: Post-Incident

**Within 24 hours:**
- [ ] Submit formal timeline to ISP for SLA credit if applicable (see ISP Incident Management Playbook)
- [ ] Request written RCA from ISP if session down was carrier-caused
- [ ] Review RPKI/ROA coverage for all your announced prefixes

**RPKI/ROA Coverage Review:**
1. Get the list of all prefixes you announce (from your router: `show ip bgp | begin Network`)
2. Check each at https://rpki.cloudflare.com/portal
3. Prefixes with no ROA = unprotected. Create ROAs at your RIR (ARIN, RIPE, APNIC, etc.)

**Why RPKI matters:**
RPKI (Resource Public Key Infrastructure) lets you create a Route Origin Authorization — a cryptographically signed record saying "AS [X] is authorized to announce prefix [Y]." ISPs with RPKI validation enabled will automatically reject routes from any other AS claiming your prefix. This is the only automated defense against BGP hijacking. Without it, a hijack depends entirely on ISPs manually noticing and filtering bad routes.

- [ ] Evaluate BGPMon (bgpmon.net) if not already in use — free tier available, sends email alerts on route changes
- [ ] Update baseline prefix count documentation
- [ ] Run a session retrospective if P0/P1: timeline, detection gap, mitigation speed, escalation effectiveness

## Escalation Matrix

| Condition | Escalate To | Contact Method |
|-----------|-------------|----------------|
| BGP session down > 15 min, no ISP response | ISP Level 2 / Duty Manager | Phone — ask for "duty manager" explicitly |
| Complete external reachability loss | Your on-call manager + ISP NOC | Phone (do not rely on email during an outage) |
| Confirmed prefix hijack | ISP security team + your CSIRT | Phone + incident ticket |
| Traffic failing over to secondary ISP | Secondary ISP NOC (heads-up call) | Phone — let them know traffic shift is coming |
| Outage > 2 hours | ISP Account Executive | Phone + email |
| Outage affecting SLA threshold | Your management chain | Status page update + internal Slack/Teams bridge |

## Key Decision Points

```
Is the BGP session UP or DOWN?
├── DOWN → Step 4 (diagnose session failure)
│   ├── Is TCP 179 reachable to peer IP?
│   │   ├── NO → Layer 3 / firewall issue, not BGP. Escalate to network team.
│   │   └── YES → BGP protocol issue. Check notification error. Call ISP NOC.
│   └── Call ISP NOC in parallel. Do not wait.
│
└── UP but routes withdrawn → Step 5 (diagnose route withdrawal)
    ├── Are your prefixes in your outbound route advertisement?
    │   ├── NO → Check prefix lists / route policy. Internal config change.
    │   └── YES → Peer is not accepting them. Call ISP NOC.
    └── Are external looking glasses showing your prefix?
        ├── YES with wrong origin AS → HIJACK. Escalate to security immediately.
        └── NO → Your prefix is globally withdrawn. Fail over to secondary ISP.

Do you have a secondary ISP?
├── YES → Begin failover (Step 6) while troubleshooting primary
└── NO → All focus on restoring primary + ISP NOC escalation

Outage > 60 minutes with no ISP resolution?
└── Escalate to ISP Account Executive + your management chain
```
