# DNS Failover Playbook

## Purpose
Use this playbook to execute a DNS-based failover when a primary datacenter or cloud region is degraded or unreachable, shifting traffic to a healthy secondary region via DNS record updates.

---

## Severity Classification

| Severity | Condition | Response Time |
|----------|-----------|---------------|
| P0 | Primary region completely unreachable, DNS health checks failing, customer-facing service down | Immediate — failover now |
| P1 | Primary region degraded (>5% error rate), health checks intermittent, failover decision pending | 15 minutes — assess and decide |
| P2 | Health check failures detected but service appears stable, TTL reduction pre-staging needed | 30 minutes |
| P3 | Planned maintenance requiring pre-staged DNS failover in advance | Schedule with change control |

---

## Prerequisites

- [ ] DNS provider console access (Route53, NS1, Cloudflare, etc.) with write permission on the affected zone
- [ ] Secondary region is healthy and ready to accept traffic (verify BEFORE touching DNS)
- [ ] Current TTL values documented for all critical DNS records (know your blast radius before you act)
- [ ] Splunk access with `index=network` read permission
- [ ] `dig` or `nslookup` available on your workstation or a jump host
- [ ] ISP circuit IDs and ISP NOC numbers (if failover involves changing egress path, not just origin)
- [ ] Rollback plan documented: what does "undo this" look like?

---

## Detection Signals

**Alert-driven entry points:**
- Route53 or NS1 health check alert: "Health check FAILED for primary region endpoint"
- Synthetic monitor: primary region URL returning 5xx for >N consecutive checks
- Splunk error rate alert: application error rate spike with source = primary region
- On-call page triggered by latency breach on primary region

**Splunk — watch DNS record change propagation:**
```splunk
index=network sourcetype=route53_logs
| stats count by query_name, answer
| where query_name="your-domain.example.com"
| sort -count
```
Expected output: table showing which IP answers are being returned. During failover, you'll watch the `answer` column shift from primary IP to secondary IP. This is your ground truth — not your DNS console.

**Splunk — application error rate by region:**
```splunk
index=app sourcetype=access_logs
| eval region=if(match(host,"primary-region"), "primary", "secondary")
| bucket _time span=1m
| stats count(eval(status>=500)) as errors, count as total by _time, region
| eval error_rate=round(errors/total*100,2)
| where error_rate > 5
```

## The Most Important Concept: TTL Math

**TTL (Time To Live)** is the number of seconds that DNS resolvers cache a record before re-querying your authoritative DNS server.

**If TTL is 3600 (1 hour):** After you change the record, it takes up to 60 minutes before all clients see the new answer. Clients that cached the record before your change will keep hitting the old IP until their cache expires.

**If TTL is 60 (1 minute):** After you change the record, clients see the new answer within 60 seconds.

**The failover speed you get is determined by the TTL that was in place BEFORE the outage started.** You cannot change TTL during an outage and get immediate benefit — you must wait for the old TTL to expire on cached entries first.

**Operational implication:**
- For planned failovers: lower TTL 24–48 hours in advance to 60–300 seconds. This is table stakes.
- For emergency failovers: change the record immediately (every second counts), but plan your communication timeline around the actual TTL expiry.
- Always lower TTL to 60 after an emergency failover, even if you can't use it for this event — it makes the next failover faster.

## Phase 1: Immediate Response (0–15 min)

### Step 1: Verify the secondary region is ready

Do NOT skip this. Failing over DNS to a region that isn't healthy converts a partial outage into a total outage.

```bash
# From a trusted host, hit the secondary region directly (bypass DNS)
curl -v -H "Host: your-domain.example.com" https://[SECONDARY-REGION-IP]/health
```

Expected: HTTP 200 with a healthy response body. If you get 500, 502, or timeout — the secondary is not ready. Do NOT proceed with DNS failover until this is resolved.

Also verify secondary capacity:
- Is it scaled to handle 100% of primary traffic? (Not just the 50% it normally handles in active-active)
- Are all dependencies (databases, caches, queues) reachable from secondary?

### Step 2: Document current TTL values

```bash
dig @8.8.8.8 your-domain.example.com
```

In the ANSWER SECTION, look for the number before the record type:
```
your-domain.example.com.  3600  IN  A  203.0.113.10
                           ^^^^
                       This is your TTL (in seconds)
```

Write it down. This is your maximum wait time for full propagation. Tell your incident commander this number immediately.

### Step 3: Immediately lower TTL (even if you can't use it now)

In your DNS console (Route53 example):
1. Navigate to the hosted zone for your domain
2. Find the A or CNAME record for the affected hostname
3. Change TTL from current value (e.g., 3600) to 60
4. Save — do NOT change the record value yet if you're not ready to fail over

This won't help your current outage propagation, but every second you wait delays your next failover. Do it now.

---

## Phase 2: Diagnosis (15–30 min)

### Step 4: Confirm the scope of the primary region failure

Before committing to a full DNS failover, confirm what is actually failing:

```bash
# Test primary region directly by IP (bypass DNS entirely)
curl -v --resolve your-domain.example.com:443:[PRIMARY-IP] https://your-domain.example.com/health

# If above fails, test TCP connectivity
nc -zv [PRIMARY-IP] 443
```

If direct IP access is also failing: the primary region is down at the application layer. DNS failover is appropriate.

If direct IP access succeeds: the issue may be in DNS itself, not the origin. Investigate DNS before changing records.

### Step 5: Verify secondary region health check status in your DNS provider

In Route53 — navigate to Health Checks. Find the health check for your primary endpoint.

- Status "Healthy" with endpoint unreachable = your health check is misconfigured (wrong path, wrong threshold)
- Status "Unhealthy" = health check has already triggered; if you have failover routing configured, automatic failover may have already happened

**Check if automatic DNS failover already occurred:**
```bash
dig @8.8.8.8 your-domain.example.com +short
```
If this returns your secondary IP already: automatic failover worked. Verify traffic is reaching secondary before declaring success.

---

## Phase 3: Mitigation — Execute DNS Failover (30–60 min)

### Step 6: Update the DNS record

**In Route53:**
1. Go to the hosted zone
2. Find the record set for the affected hostname
3. Change the Value from primary region IP/CNAME to secondary region IP/CNAME
4. Ensure TTL is set to 60 (you should have done this in Step 3)
5. Save the record set
6. Note the exact timestamp you made this change

**In Cloudflare:**
1. Navigate to DNS > Records
2. Find the record, click Edit
3. Change the IP address to secondary
4. Enable the orange cloud (proxy) if it was enabled — Cloudflare's cache will reflect the change quickly
5. Save

**In NS1:**
1. Navigate to DNS > Zones > your zone
2. Find the record, click the pencil icon
3. Update the answer IP
4. Set TTL to 60
5. Publish

### Step 7: Verify DNS propagation

Run these checks from multiple vantage points — Google, Cloudflare, and your authoritative server:

```bash
# Check Google's resolver
dig @8.8.8.8 your-domain.example.com +short

# Check Cloudflare's resolver
dig @1.1.1.1 your-domain.example.com +short

# Check your authoritative nameserver (find it first)
dig your-domain.example.com NS +short
# Then query it directly:
dig @[your-authoritative-ns] your-domain.example.com +short

# Full trace (shows every hop of the resolution chain)
dig +trace your-domain.example.com
```

**What to look for:**
- Google and Cloudflare should return the secondary IP within 60–120 seconds if TTL was already 60
- If TTL was 3600, Google and Cloudflare may continue returning the old IP for up to 3600 seconds — this is expected, not a bug
- Your authoritative NS should return the new IP immediately after you save

**Watch the TTL countdown:**
```bash
# Run this repeatedly — watch the TTL number count down to 0, then reset to 60
watch -n5 "dig @8.8.8.8 your-domain.example.com | grep -A1 'ANSWER SECTION'"
```

When the TTL resets to 60 on a resolver: that resolver has fetched the new record. It will now return the secondary IP.

### Step 8: Monitor traffic shift in secondary region

```splunk
index=app sourcetype=access_logs server=secondary-*
| bucket _time span=1m
| stats count by _time
| timechart span=1m count
```

You should see secondary region request counts climbing. Simultaneously:

```splunk
index=app sourcetype=access_logs server=primary-*
| bucket _time span=1m
| stats count by _time
| timechart span=1m count
```

Primary region traffic should decline as client DNS caches expire and they resolve to secondary.

---

## Phase 4: Resolution

### Step 9: Declare failover complete

Criteria for declaring failover complete:
- [ ] Secondary region is receiving traffic at or near expected volume
- [ ] Application error rate on secondary is within acceptable bounds (<1%)
- [ ] No significant traffic still flowing to primary (allow for long-TTL stragglers)
- [ ] Health checks on secondary reporting healthy

**Do NOT lower your guard.** Monitor for the full old TTL duration after failover. Long-TTL clients continue to arrive at primary for up to TTL seconds. If primary is truly down, these clients will experience errors until their cache expires.

### Step 10: Communicate resolution

Post to incident channel and status page. Include actual failover time vs. expected.

---

## Phase 5: Post-Incident

**Immediate (within 2 hours):**
- [ ] Confirm secondary region is stable under full load
- [ ] Document actual traffic shift time vs. TTL expectation — was the math right?
- [ ] Set a calendar reminder for 30 days out: SLA credit claim if primary outage was provider-caused

**Within 24 hours:**
- [ ] Review TTL policy for all critical DNS records:
  - Records serving production traffic should have TTL ≤ 300 seconds
  - Records with TTL > 3600 are a failover liability — reduce them
- [ ] Implement automated DNS health checks if not already present (Route53 health check + failover routing policy is the "self-driving" version)
- [ ] Run the failover timeline through your post-mortem:
  - When did the primary fail?
  - When did we detect it?
  - When did we make the DNS change?
  - When did 50% of traffic reach secondary? 95%?
  - Where was the gap?

**Route53 Automated Failover (the "self-driving" version):**
Instead of manual DNS changes, configure:
1. Route53 health check pointing to primary region endpoint
2. Primary record set: Route Policy = "Failover", Role = "Primary", associate health check
3. Secondary record set: Route Policy = "Failover", Role = "Secondary"
4. When health check fails, Route53 automatically stops returning the primary record

This removes the human response time from your MTTR. It only fails over as fast as your TTL allows — but it does not require anyone to wake up and log into a console.

**Facebook/Meta 2021 Reference:**
On October 4, 2021, Facebook went down for approximately 6 hours. The root cause was a BGP configuration change that withdrew Facebook's own BGP routes. Because Facebook's authoritative DNS servers were only reachable through Facebook's own BGP-announced network, DNS lookups for all Facebook properties failed globally — not just their applications, but DNS itself. The lesson for your architecture: DNS infrastructure must be reachable via network paths that are independent of your own AS or primary transit. Hosting your authoritative DNS on the same network segment as your application is a single point of failure. Use a third-party authoritative DNS provider (Route53, Cloudflare, NS1) or ensure your DNS servers have diverse connectivity independent of your application infrastructure.

## Escalation Matrix

| Condition | Escalate To | Contact Method |
|-----------|-------------|----------------|
| Secondary region health check failing before failover | Database / Application on-call | PagerDuty |
| DNS change made but no traffic shifting after 2x TTL | DNS provider support | Phone + ticket |
| Both primary and secondary regions degraded | All hands — VP Engineering | Phone + war room |
| DNS provider outage suspected | Incident Commander | Escalate; consider secondary DNS provider |
| SLA breach threshold hit | Management + legal/contracts | Email + documented ticket |

```
INCIDENT DECLARED — DNS FAILOVER IN PROGRESS
Time: [HH:MM TZ]
Severity: [P0/P1]
Primary region: [status — down / degraded]
Secondary region: [status — healthy / scaling]
Current TTL: [X seconds] — expected full propagation by [time]
IC: [name]
Next update: [15 min out]
```

**During failover (every 15 min):**
```
FAILOVER UPDATE — [HH:MM TZ]
DNS record updated at: [time]
Current TTL was: [X seconds] when change made
Traffic on secondary: [X% of expected volume]
Traffic on primary: [X% — declining]
Expected full shift by: [time]
```

**Resolution:**
```
RESOLVED — DNS FAILOVER COMPLETE [HH:MM TZ]
All traffic routing to secondary region.
Application error rate: [X%] — within acceptable bounds.
Duration of primary outage: [X] minutes.
Root cause investigation ongoing. RCA to follow.
```

## Key Decision Points

```
Is secondary region healthy and ready?
├── NO → Do NOT execute DNS failover. Fix secondary first.
│         Escalate to application on-call. Explore alternatives.
└── YES → Proceed

What is the current TTL?
├── ≤ 60 seconds → Failover will complete within ~60 seconds. Proceed immediately.
├── 61–300 seconds → Failover will complete within 5 minutes. Proceed, inform stakeholders of timeline.
├── 301–3600 seconds → Failover will complete within up to 1 hour. Proceed immediately,
│                      set stakeholder expectations, monitor closely.
└── > 3600 seconds → Significant propagation lag. Proceed with DNS change now.
                     Communicate that some clients will continue hitting primary for up to [TTL/3600] hours.
                     Consider if there are alternatives (anycast, load balancer change, etc.)

Did automatic failover already trigger?
├── YES (secondary IP already in DNS) → Verify secondary health. Monitor traffic. Do not re-change record.
└── NO → Execute manual failover per Phase 3.

After DNS change, traffic not shifting after 2x TTL?
├── Check authoritative NS is returning new record (dig @authoritative-ns)
│   ├── Authoritative shows NEW IP: resolvers are caching old record. Wait.
│   └── Authoritative shows OLD IP: your DNS change did not save. Re-check console.
└── Verify with Splunk that traffic shift is actually happening (may be faster than you think)
```
