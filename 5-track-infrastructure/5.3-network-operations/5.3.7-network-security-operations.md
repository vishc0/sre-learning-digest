# Chapter 7: Network Security Operations

## The Core Idea

Network security is the combination of two systems that you need simultaneously: physical building security and a CCTV surveillance system. Firewalls and segmentation are your building security — walls, locked doors, access card readers. You decide who can go where before anything happens. Flow data and log analysis are your CCTV system — they tell you what actually happened, when, and who was involved. Neither system alone is sufficient. A building with no cameras doesn't know a door was propped open. A camera system with no walls records the theft it can't prevent.

At Director scale, you make decisions about both layers: what gets segmented, who reviews the footage, and when the alert threshold trips an escalation.

## Firewall Rule Management

### Stateful vs. Stateless Firewalls

The fundamental divide in firewall architecture is whether the firewall tracks conversation state.

A **stateful firewall** understands that TCP is a conversation. When a client opens a connection to your web server on port 443, the firewall records that session — source IP, destination IP, port, sequence number, session state. When the server responds, the firewall recognizes this as a reply to an established session and allows it through without requiring an explicit rule for the return direction. The session table is the firewall's memory.

A **stateless firewall** has no memory. It applies rules to every packet individually, in isolation. If you want a web server to respond to clients, you need explicit rules in both directions: one allowing inbound port 443, and one allowing outbound reply traffic. Stateless firewalls are faster (no table lookups) but require more rules and are harder to reason about under load.

**AWS Translation**: Security Groups are stateful — if you allow inbound port 443, return traffic is automatically permitted. Network ACLs (NACLs) are stateless — you must explicitly allow both inbound and outbound for bidirectional communication. This is why a common mistake on AWS is: security group is correct, but the NACL blocks return traffic and the connection silently hangs.

### Rule Ordering and First-Match-Wins

Most firewalls evaluate rules top-to-bottom and stop at the first match. This means rule order is business logic, not configuration detail. A permissive rule placed above a restrictive rule eliminates the restrictive rule entirely — the packet matches the wide rule first and gets allowed before the deny rule is ever evaluated.

Implication: your most specific rules must be at the top. Deny rules for known-bad sources should precede broad allow rules. Default-deny at the bottom is the only safe baseline.

### Rule Sprawl: The Enterprise Tax

Rule sprawl is the universal enterprise firewall problem. Rules get added when a service launches. The service gets decommissioned two years later, but no one removes the rule. Multiply this across a 10-year-old firewall and you have thousands of rules, half of which apply to systems that no longer exist.

Rule sprawl creates three problems: (1) attack surface — unused rules may inadvertently permit paths to current systems, (2) audit failure — compliance reviewers find undocumented rules and flag them, (3) operational risk — a large ruleset is hard to reason about during an incident when you need to verify the path quickly.

The mitigation is a rule lifecycle process: every rule requires a service owner, an expiry review date, and a ticket reference. Rules with no owner in the last 90 days get flagged for removal review. This is a process decision — which means it's a Director decision, not an engineer decision.

### Change Management for Firewall Rules

Firewall rules are infrastructure code. Every change needs: peer review, staging validation where possible, explicit rollback documentation, and a maintenance window for production changes. A firewall rule change that breaks connectivity at 2 AM with no rollback plan is an incident, not just a mistake.

## Network Segmentation

### VLANs, VPCs, and What They Actually Do

VLANs (Virtual LANs) segment at Layer 2 — the switch level. Hosts on different VLANs cannot talk to each other unless traffic passes through a router (or Layer 3 switch) that has an explicit route between them. This is how traditional enterprise networks segment finance from HR from engineering — all on the same physical switches, but logically separated.

VPCs and subnets operate at Layer 3 — IP routing level. In AWS, a VPC is your own private address space. Subnets within a VPC can be routed to each other, but cross-VPC traffic requires VPC peering or a Transit Gateway. Public subnets have a route to an Internet Gateway; private subnets do not.

The practical difference: VLANs are a data center construct; VPCs are a cloud construct. Both achieve the same goal — contain the blast radius of a compromise by limiting which systems can talk to which.

### T-Mobile Context: Three-Plane Segmentation

Carrier networks enforce strict segmentation between three planes:

- **Signaling plane**: SS7 / Diameter / SIP — call control, location updates, authentication
- **Management plane**: network element management, SNMP, CLI access — the control surface
- **Data plane**: actual subscriber traffic (voice, data)

The rule is absolute: the data plane must never have a route to the management plane. A compromised subscriber connection must not reach the configuration interface of a network element. This is the carrier equivalent of zero-trust before zero-trust was a term.

### Zero-Trust and Microsegmentation

Traditional network security assumes that traffic inside the perimeter is trusted. Zero-trust inverts this: every request is untrusted until verified, regardless of where it originates. A compromised workstation inside your VPN is as dangerous as an external attacker — zero-trust treats them identically.

Microsegmentation takes firewall logic down to the workload level. Instead of a single perimeter firewall protecting the entire data center, each workload (pod, VM, container) has its own firewall policy. In Kubernetes, this is implemented via NetworkPolicy objects — define exactly which pods can talk to which other pods, on which ports.

The Director implication: microsegmentation shifts firewall management from a network team responsibility to a platform team responsibility. Kubernetes NetworkPolicy is written in YAML by developers and reviewed in pull requests. The network team sets policy standards; the platform team enforces them via admission controllers.

## DDoS Response

### Three Attack Types, Three Response Layers

**Volumetric DDoS** (Layer 3/4): The attacker generates enough traffic to saturate your upstream bandwidth — 100Gbps of UDP floods, Gbps of ICMP. Your server is unreachable not because it failed but because the internet pipe is full. Defense: upstream scrubbing. Cloudflare Anycast, AWS Shield Advanced, or a dedicated scrubbing center absorbs the flood before it reaches your network. You cannot solve volumetric DDoS inside your own infrastructure.

**Protocol DDoS** (Layer 4): Exploits TCP/UDP protocol behavior to exhaust server resources without requiring huge bandwidth. A SYN flood sends millions of TCP SYN packets without completing the handshake — each half-open connection consumes server state table memory until the server can accept no new connections. UDP amplification uses misconfigured servers (DNS resolvers, NTP servers) to reflect traffic at your target amplified by 10x–50x. Defense: SYN cookies (the server stops storing state for unconfirmed connections), rate limiting at the edge, upstream blocking of spoofed sources.

**Application DDoS** (Layer 7): HTTP floods that mimic legitimate user behavior — each request is valid, but the aggregate volume overwhelms your application. Harder to detect because you cannot distinguish attack from legitimate traffic at the packet level. Defense: WAF rules (rate limiting per IP, per token, per session), bot detection (TLS fingerprinting, behavioral analysis), progressive challenges (CAPTCHA for suspicious clients).

### Mitigation Layers During an Active Event

The response hierarchy is: CDN/edge absorbs first, scrubbing center filters second, your WAF handles the remainder. What gets to your origin servers should be clean traffic.

The critical operational reality: **you cannot take defensive action until you distinguish attack traffic from legitimate traffic**. Flow data is essential here. During a DDoS event, the first 10 minutes should be spent analyzing flow data — where is the traffic coming from, what destination ports, what does the source distribution look like? Blocking indiscriminately drops legitimate users. A DDoS that kills legitimate traffic is an outage you caused yourself.

## Traffic Analysis with Flow Data

### NetFlow vs. sFlow vs. Packet Capture

**NetFlow** (Cisco-origin, now IPFIX standard): network devices sample packet headers and report flow records — source IP, destination IP, ports, protocol, bytes, packets, duration, start/end timestamp. Lightweight, always-on, massive scale. The tradeoff: sampling rate matters. At 1-in-1000 sampling, you see the pattern but miss individual connections.

**sFlow**: samples actual packets rather than just headers. Better visibility, but higher overhead and cost. Used where NetFlow's metadata isn't sufficient for security analysis.

**Packet capture (pcap)**: records the full packet including payload. Complete visibility, prohibitive cost at scale. Use pcap surgically during an incident — "capture traffic on this host for the next 5 minutes" — not as continuous telemetry.

Mental model: flow data is your phone billing record (who called whom, when, for how long). Packet capture is the full audio recording of the call.

### What Flow Data Tells You (and What It Doesn't)

Flow data answers: Who is talking to whom? How much traffic? What ports? Is this pattern normal or anomalous? It is the right tool for: identifying top talkers, detecting port scans (many connections from one source to many destinations), detecting data exfiltration (large sustained outbound flows to an unfamiliar destination), and DDoS source characterization.

Flow data does not tell you: what the content of the traffic was, whether an HTTP request was malicious, whether the application-layer payload was encrypted command-and-control traffic. For that you need packet capture or application-layer logging.

## Splunk for Network Log Analysis

This is where your existing tooling gives you a genuine operational edge. Firewall logs, DNS query logs, VPC flow logs, and load balancer access logs all land in Splunk — and network security analysis in Splunk is pattern recognition at scale, which is exactly what Splunk is built for.

### Key Searches for Network Security

**Top talkers by bytes** — identifies bandwidth hogs and potential exfiltration:
```splunk
index=network sourcetype=firewall
| stats sum(bytes) as total_bytes by src_ip
| sort -total_bytes
| head 20
```

**Rejected connection spikes by destination port** — VPC flow log reconnaissance detection:
```splunk
index=network sourcetype=aws:cloudwatchlogs:vpcflow action=REJECT
| stats count by dst_port
| sort -count
```

**DNS NXDOMAIN spike** — misconfigured service or domain-generation algorithm (DGA) malware:
```splunk
index=dns sourcetype=dns rcode=NXDOMAIN
| timechart span=5m count
```
A sudden spike in NXDOMAIN responses indicates one of two things: a service is calling a decommissioned endpoint (ops problem), or a compromised host is cycling through algorithmically generated domain names trying to reach a C2 server (security problem). The Splunk alert fires the same way — you differentiate by correlating the source IP against your asset inventory.

**Firewall log correlation — unexpected source IPs hitting internal services**:
```splunk
index=network sourcetype=firewall action=deny
| lookup asset_inventory ip as src_ip OUTPUT environment as src_env
| where src_env!="internal"
| stats count by src_ip, dst_ip, dst_port
| sort -count
```

**Load balancer 4xx surge** — separating DDoS from application bugs:
```splunk
index=network sourcetype=aws:alb
| timechart span=1m count(eval(status_code>=400 AND status_code<500)) as client_errors,
             count(eval(status_code>=500)) as server_errors
```

### Director Frame: What You Define, Not Just What You Watch

As Director, your role in network log analysis is to set the policy that your team executes:

1. **Retention policy**: How long do firewall logs, VPC flow logs, and DNS logs live in Splunk? Security regulations (PCI, SOC2, HIPAA) may require 12 months. Storage cost is real — define tiers (hot: 30 days, warm: 90 days, cold archive: 12 months).

2. **Alert thresholds**: What NXDOMAIN rate is normal for your environment? What rejected connection rate is expected baseline? You cannot alert on spikes without knowing baseline. This requires a baseline-measurement sprint before alert rules are set.

3. **Ownership assignment**: Who owns network security monitoring? In most organizations this sits between Network Engineering, Security Operations, and the SRE team. Ambiguous ownership is how incidents go undetected for hours. Define it explicitly in your runbooks.

4. **Escalation paths**: When Splunk fires a DDoS alert at 3 AM, who gets called first? What is the sequence? This is a Director decision documented in an incident response plan, not something an engineer decides in the moment.

The signal that a Director-level candidate understands network security is not reciting firewall types. It is explaining why they defined retention policy X, how they determined alert threshold Y, and what organizational changes they made to ensure the right team owns response.
