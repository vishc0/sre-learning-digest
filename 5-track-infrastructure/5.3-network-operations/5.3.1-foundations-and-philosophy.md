# Chapter 1: Foundations and Philosophy

## The Core Idea

Network reliability is to application reliability what highway infrastructure is to logistics. The highway doesn't care what's in your truck — it carries everything equally. But if the highway fails, every truck stops, regardless of what they're delivering. Your application stack sits on top of a network substrate it rarely thinks about, until that substrate fails — and then everything fails at once.

This is the fundamental distinction between network reliability and application reliability: scope and invisibility. Application failures are surgical. Network failures are systemic.

---

## OSI Layers 3–7: What SREs Actually Need to Know

Forget the academic OSI model. Here is the operational version — what each layer is, what breaks there, and how it shows up in your incident queue.

| Layer | Name | What It Does | What Breaks | Incident Signature |
|-------|------|-------------|-------------|-------------------|
| **3 — Network** | IP routing, subnets, CIDR | Gets packets from A to B across networks | Route flaps, subnet misconfigurations, firewall ACL mismatches | Intermittent connectivity; some hosts reachable, others not; traceroute stops mid-path |
| **4 — Transport** | TCP/UDP, ports, sessions | Establishes reliable (TCP) or fast (UDP) delivery | TCP handshake failures, port exhaustion, RST storms, SACK issues | Connection timeouts that look like app hangs; `SYN_SENT` stuck states; connection refused on healthy pods |
| **5 — Session** | Session management, TLS negotiation | Establishes and tears down communication channels | TLS cert expiry, cipher mismatch, session exhaustion in proxies | TLS handshake errors; certificate warnings; load balancer session table full |
| **6 — Presentation** | Encoding, compression, encryption | Translates data format between systems | Encoding mismatch, compression library bugs, cert chain issues | Garbled responses; JSON parse failures on the receiver side; corrupt payloads |
| **7 — Application** | HTTP, DNS, SMTP, gRPC | Protocol-level communication | Bad headers, malformed requests, API contract violations | 4xx/5xx errors; gRPC status codes; slow HTTP responses |

**The SRE trap**: most application monitoring lives at Layer 7. You instrument HTTP status codes, response times, error rates. Layer 3 and 4 failures are often invisible to that instrumentation — until they cascade upward and look like application problems.

## The Invisibility Problem: How Network Failures Masquerade as App Failures

Packet loss does not announce itself. It shows up as latency. TCP is designed to recover from loss — it retransmits, backs off, adjusts its window. This is a feature at small scale. At incident scale, it means a 2% packet loss rate on a core link can double or triple your p99 latency before a single error fires in your APM tool.

The failure chain looks like this:

```
2% packet loss on core switch
  → TCP retransmit on every affected connection
  → Application timeout thresholds start triggering
  → Retry storms from clients amplify load
  → Application pods start OOMing under retry load
  → PagerDuty fires: "Application error rate elevated"
```

Your on-call engineer investigates the application. The application is fine. The problem is four hops upstream on a switch you can't see from Kubernetes.

This is why network observability — NetFlow, SNMP, interface error counters, BGP state — belongs in your SRE stack, not just in a separate NOC dashboard that nobody looks at during an incident.

## The Four Failure Modes: Latency, Packet Loss, Jitter, Availability

These are distinct problems that require distinct mitigations. Using the wrong diagnosis leads to the wrong fix.

**Latency** — the time it takes a packet to travel. High latency degrades user experience progressively. It is usually caused by distance (speed of light is non-negotiable), congestion (queue depth on overloaded links), or misconfigured routing (traffic taking the wrong path).

**Packet loss** — packets that never arrive. TCP handles small amounts gracefully through retransmission. At 1–2%+ loss, TCP's congestion control kicks in and throttles throughput dramatically. UDP-based protocols (DNS, some video streaming, QUIC) degrade immediately since there is no retransmission layer.

**Jitter** — variability in latency. Average latency of 20ms sounds fine. If latency oscillates between 5ms and 200ms, your p99 looks terrible and real-time applications (VoIP, streaming, gaming) break even though "average" metrics look acceptable. Jitter is often caused by traffic shaping, uneven load balancing across paths, or bufferbloat on congested links.

**Availability** — the link is simply down. This is the easiest failure mode to detect and the easiest to mitigate with redundancy. It is also the least common in mature infrastructure, which is why the other three failure modes cause more incidents per year.

## The T-Mobile Advantage: Carrier Intuition Most SREs Don't Have

Software-origin SREs typically encounter the network when it breaks. Your background is different. You've operated inside a carrier — you've seen BGP peering relationships, core routing infrastructure, RF engineering affecting packet delivery, and the operational discipline of keeping 25M+ messages per day moving through a system that has no maintenance windows.

That intuition gives you something most Director candidates don't have: you understand the network as infrastructure, not as a service. You know the difference between a network that is "up" by metrics and a network that is actually healthy. You know that carrier-grade and enterprise-grade are different operational philosophies.

In interviews, this surfaces as: you ask about blast radius before you ask about MTTR. You ask about network topology before you ask about application architecture. You already think in layers.

## Director Decision Frame

At the Director level, you are not debugging packet loss — you are making structural decisions that determine whether your platform can tolerate network failure without cascading.

**Questions you ask in design reviews:**
- What is the blast radius if the primary network path fails? Does it affect one service or all services?
- Does our monitoring detect Layer 3/4 failures before they cascade to Layer 7? Or do we find out from application alerts after the damage is done?
- Where is the network boundary between "our team's responsibility" and "networking team's responsibility"? Is that boundary well-instrumented?

**What you watch in incidents:**
- Is this failure surgical (one service, one region) or systemic (multiple services, same time)? Systemic failures almost always have a network or infrastructure root cause.
- Is latency rising before errors appear? That is the signature of network degradation, not application failure.
- Are retries amplifying the problem? Retry storms are often a network failure that the application layer is making worse.

**The platform substrate mindset**: You manage the platform that other teams build on. The network is not another service in your dependency graph — it is the substrate everything else runs on. Network reliability is a prerequisite for application reliability, not a parallel concern.

---

## Key Takeaways

- Network failures are systemic; application failures are surgical. A single network event can simultaneously impact every service on the platform.
- Packet loss manifests as latency before it manifests as errors — application monitoring will miss the early signal.
- The four failure modes (latency, packet loss, jitter, availability) are distinct and require distinct mitigations; diagnosing the wrong one wastes time.
- Layer 3 and 4 failures are invisible to Layer 7 monitoring until they cascade — network observability must be integrated into the SRE stack.
- Your carrier background gives you a substrate mindset most software-origin SREs develop late if at all — use it to ask blast-radius questions before design decisions are locked.
