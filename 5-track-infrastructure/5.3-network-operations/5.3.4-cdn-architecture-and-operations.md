# Chapter 4: CDN Architecture and Operations

## The Core Idea

Think of a CDN like building distribution warehouses near every major city instead of shipping everything from a single factory in one location. A customer in Seattle gets their package from the Seattle warehouse, not from your headquarters in Dallas. Faster delivery, lower shipping cost — but now you have a warehouse coordination problem: how do you keep every warehouse stocked with current inventory, and what happens when the factory sends out a product update?

That warehouse coordination problem is exactly what makes CDN operations interesting at director scale.

## How CDN Works: Edge to Origin

A CDN is a globally distributed network of servers called **Points of Presence (PoPs)** or edge nodes. When a user in London requests your app's assets, anycast routing sends that request to the nearest London PoP — not your origin datacenter in Virginia.

The request traverses a four-layer cache hierarchy:

```
User → Edge Node (city-level PoP) → Regional PoP → Origin Shield → Origin Server
```

**Edge node** — the first stop. Fastest. If the asset is cached here, the user never sees your origin. A cache hit at this layer costs microseconds and pennies.

**Regional PoP** — a mid-tier cache that aggregates traffic from multiple edge nodes. A cache miss at the edge node goes here before trying the origin.

**Origin shield** — a single designated intermediate cache that acts as the last line of defense before your origin. This is the most operationally important layer that most teams underinvest in.

**Origin** — your actual application servers. Every request that reaches here is one you pay full price for in compute, egress bandwidth, and latency.

## Origin Shield: The Pattern T-Mobile Scale Demands

Without an origin shield, a cache miss event looks like this: 500 edge nodes all miss simultaneously on the same URL, and all 500 fire requests directly to your origin. At T-Mobile's 25M msg/day notification platform scale, a cache stampede hitting origin directly can be an incident-creating event.

Origin shield collapses that fan-out. Instead of 500 requests hitting origin, one canonical request goes from the shield to origin, the result is cached at the shield, and all 500 edge nodes get served from that single fetch. It reduces origin load by orders of magnitude during miss events.

The tradeoff: origin shield adds one network hop, typically 20-40ms of latency. For static assets, invisible. For real-time API responses, worth considering whether CDN is even the right tool.

## Cache Invalidation: The Hardest Problem in CDN Ops

Phil Karlton's famous quote — "there are only two hard things in computer science: cache invalidation and naming things" — was written before CDN scale existed. At CDN scale, cache invalidation is genuinely hard.

**TTL expiry** — the default. Cache objects have a time-to-live header and expire naturally. Simple, but you're at the mercy of TTL values set at deployment time. A 24-hour TTL on a JavaScript bundle becomes a 24-hour incident when you push a broken deploy.

**Purge by URL** — fire a purge request to the CDN API for a specific URL. Fast, surgical, but you need to know every URL variant (query strings, locale variants, device variants). At scale this becomes a list-management problem.

**Purge by tag / surrogate key** — the most powerful pattern. Tag content at origin with a header (`Surrogate-Key: product-123 category-electronics`), and you can purge all content bearing that tag with one API call. Fastly and Cloudflare support this natively. CloudFront does not (you use invalidation paths instead).

**Instant vs. eventual purge** — Fastly's instant purge propagates to all edge nodes in ~150ms globally. CloudFront invalidations can take 5-10 minutes. This is not a minor operational difference — in an incident where wrong content is cached, the difference between 150ms and 10 minutes of blast radius is significant.

## CDN Failure Modes That Create Incidents

**Cache poisoning** — wrong content gets cached at the edge. This happens when origin returns different responses based on headers (Accept-Language, device type) and the CDN doesn't vary the cache key on those headers. One French user's response gets served to all English users. Fix: ensure `Vary` headers are correctly set and honored by the CDN config.

**Stale content after deploy** — a new JavaScript bundle ships, but the old bundle is cached at edge nodes with a 1-hour TTL. Users get a split experience: some get new frontend code calling new API contracts, some get old frontend code calling old API contracts. Fix: content-hash filenames (`app.a3f9c2.js`) make every deploy produce a unique URL — TTL becomes irrelevant because the URL itself changes.

**CDN provider outage** — the Fastly incident of June 2021 took approximately 85% of major internet sites offline for roughly one hour. GitHub, Stack Overflow, Reddit, the New York Times — all down simultaneously. Single-CDN architectures have single-CDN blast radius. Multi-CDN (Fastly + CloudFront) provides resilience but doubles the operational surface: two configs to maintain, two cache invalidation flows, two billing relationships.

**Cache stampede** — all TTLs expire simultaneously (common if you set the same TTL on everything), all edge nodes miss at once, origin absorbs the full combined load. Mitigation: TTL jitter (randomize expiry within a window), probabilistic early revalidation (Fastly calls this "request collapsing"), and origin shield as your last defense.

## CDN Cost Model

CDN billing is primarily traffic-based: you pay per GB of data transferred from edge nodes to users. The economic lever is **cache hit ratio**.

A 95% cache hit ratio means only 5% of requests reach your origin. At T-Mobile notification platform scale, moving from 90% to 95% hit ratio is not a performance tuning decision — it's a cost reduction initiative worth quantifying. The math: origin egress bandwidth costs 5-10x more than CDN egress bandwidth in most cloud configurations.

Cache hit ratio is a KPI that belongs on an engineering director's dashboard, not buried in a CDN vendor report.

## Director Decision Frame

CDN sits at the intersection of three distinct responsibilities: **reliability** (serving stale content when origin is down), **performance** (reducing latency for global users), and **cost** (shifting egress from expensive origin bandwidth to cheaper CDN bandwidth).

In design reviews, ask:
- What is our cache hit ratio target, and how do we alert when we fall below it?
- What is our purge strategy for deployment events? How long is the blast radius window if something ships wrong?
- Do we have origin shield configured? What is our origin's theoretical max load during a full cache miss event?
- Are we on single-CDN? What is our continuity plan for a Fastly-2021 scenario?
- Are we using CDN for API responses? If so, what is the Vary header strategy, and are we certain we're not serving one user's data to another?

The multi-CDN vs. single-CDN decision is a vendor management decision, not a technical one. Running two CDNs doubles operational complexity and introduces cache consistency questions between vendors. The right answer depends on your revenue impact per hour of outage weighed against the engineering cost of maintaining dual configurations.

## Key Takeaways

- CDN cache hierarchy runs edge node → regional PoP → origin shield → origin. Origin shield is the operationally critical layer that prevents miss storms from overwhelming origin at scale.
- Cache invalidation is the primary operational risk in CDN-heavy architectures. Know your CDN's purge propagation time — 150ms (Fastly instant purge) and 10 minutes (CloudFront) are not equivalent in incident response.
- Stale content and cache stampede are the two most common CDN-induced incident patterns. Content-hash filenames and TTL jitter are the standard mitigations.
- Cache hit ratio is a cost lever, not just a performance metric. A 5-point improvement in hit ratio can meaningfully reduce origin egress costs at platform scale.
- Single-CDN architectures have single-CDN blast radius. The 2021 Fastly outage is the canonical example every Director-level candidate should reference when discussing CDN resilience.
