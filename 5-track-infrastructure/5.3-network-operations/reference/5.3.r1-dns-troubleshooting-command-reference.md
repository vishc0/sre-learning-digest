# DNS Troubleshooting Command Reference

**Audience**: SRE Managers and Directors who need to run real commands during incidents or verify engineers' findings — not just read output someone else generated.

**Key principle**: DNS troubleshooting is about answering three questions: (1) Is the record correct at the authoritative source? (2) Has it propagated to the resolvers that matter? (3) Is the resolver reachable from where the problem is happening?

---

## 1. dig — The Primary Tool

`dig` (Domain Information Groper) is the standard DNS troubleshooting tool on Linux/macOS. It queries DNS servers directly and gives you structured, trustworthy output. Always prefer `dig` over `nslookup` on Unix systems.

**Install check**: `dig -v` — if not installed: `apt install dnsutils` (Debian/Ubuntu) or `yum install bind-utils` (RHEL/CentOS) or it's pre-installed on macOS.

---

### 1.1 Basic Lookup

**Purpose**: Resolve a hostname to its IP address using your system's default resolver.

```bash
dig example.com
```

**Normal output anatomy:**

```
; <<>> DiG 9.18.1 <<>> example.com
;; QUESTION SECTION:
;example.com.                   IN      A

;; ANSWER SECTION:
example.com.            3600    IN      A       93.184.216.34

;; Query time: 12 msec
;; SERVER: 10.0.0.53#53(10.0.0.53)
;; WHEN: Mon Jul 06 14:23:00 UTC 2026
;; MSG SIZE  rcvd: 56
```

**Key fields:**
- `ANSWER SECTION`: The record returned. If this section is empty, the name doesn't resolve.
- `3600`: The TTL (Time To Live) in seconds — how long this answer is cached.
- `93.184.216.34`: The IP address.
- `SERVER: 10.0.0.53`: The resolver your system used. This tells you *which* resolver answered, not the authoritative source.
- `Query time`: How long the DNS query took. > 100ms from inside a region is worth investigating.

**Abnormal output — what to look for:**
- `NXDOMAIN` in the status field = name does not exist. The record may have been deleted or the name is misspelled.
- `SERVFAIL` = the resolver could not get an answer. Often a DNSSEC validation failure or authoritative server unreachable.
- Empty ANSWER SECTION with `NOERROR` status = the name exists but has no A record (may have AAAA, CNAME, etc.).
- `Query time: 5000 msec` = resolver is slow or timing out; try querying a different resolver.

### 1.2 Query a Specific Record Type

**Purpose**: Look up a specific DNS record type rather than the default A record.

```bash
# A record (IPv4 address)
dig example.com A

# AAAA record (IPv6 address)
dig example.com AAAA

# CNAME record (alias)
dig www.example.com CNAME

# MX records (mail servers, ranked by priority)
dig example.com MX

# NS records (authoritative nameservers for the domain)
dig example.com NS

# TXT records (SPF, DKIM, domain verification, etc.)
dig example.com TXT

# SOA record (zone metadata — serial, refresh, retry, expire)
dig example.com SOA
```

**Why this matters:**
- If an app uses a CNAME but you query A, you get the final IP — not the CNAME chain. Use CNAME type to see the alias target.
- MX records have a priority number. Lower number = higher priority. If you see MX 10 and MX 20, traffic goes to the MX 10 server first.
- TXT records hold SPF rules — `v=spf1 ...`. If email is being rejected, check TXT.
- SOA serial number tells you if a zone has been updated. Two authoritative servers should have the same serial. If they differ, zone transfer is lagging.

**Abnormal output:**
- CNAME pointing to another CNAME — can create chains. Browsers follow them; some systems don't. Check with `dig +trace`.
- Missing MX record — email cannot be delivered. The zone exists but has no MX, which causes sender errors.

### 1.3 Query a Specific Nameserver

**Purpose**: Bypass your system's resolver and ask a specific nameserver directly. Essential for verifying propagation.

```bash
# Query Google's public resolver
dig @8.8.8.8 example.com

# Query Cloudflare's resolver
dig @1.1.1.1 example.com

# Query your organization's internal resolver
dig @10.0.0.53 internal-service.company.com

# Query the authoritative nameserver directly (after finding it with NS lookup)
dig @ns1.example.com example.com
```

**Why this matters:**
Your system's resolver may have a cached answer that is stale. Querying `@8.8.8.8` directly shows you what the public internet sees right now. Querying the authoritative NS directly shows you what the source of truth says — bypassing all caches.

**Pattern: authoritative vs. resolver discrepancy**
If `dig example.com` (system resolver) returns IP A, but `dig @authoritative-ns.com example.com` returns IP B, the record changed but the cache hasn't expired yet. Wait for TTL to drain, or force a cache flush on your resolver.

---

### 1.4 Trace the Full Resolution Path

**Purpose**: See every step of the DNS resolution chain — root servers → TLD nameservers → authoritative nameservers. Invaluable for diagnosing delegation failures.

```bash
dig +trace example.com
```

**Normal output pattern:**
```
.                        518400  IN      NS      a.root-servers.net.
[... list of root servers ...]

com.                     172800  IN      NS      a.gtld-servers.net.
[... list of .com TLD servers ...]

example.com.             86400   IN      NS      ns1.example.com.
example.com.             86400   IN      NS      ns2.example.com.

example.com.             3600    IN      A       93.184.216.34
```

**How to read it:**
- Each section shows a delegation step. Root servers → `.com` TLD → `example.com` authoritative.
- If the trace stops at a certain level, the problem is in the delegation from that level down.
- The nameservers shown at each level are the *delegated* servers, not necessarily the ones that have the final answer.

**Abnormal output:**
- Trace stops at root or TLD → domain not delegated. Registrar issue or domain not registered.
- Trace reaches authoritative but gets SERVFAIL → the authoritative NS is reachable but broken (zone load issue, DNSSEC misconfiguration).
- Trace reaches authoritative but answer is wrong → the zone data is wrong at the source. Fix the record there, not at the resolver.

---

### 1.5 Check TTL

**Purpose**: Find out how long until the current cached answer expires. Critical before a DNS cutover — you want TTL low before you make the change.

```bash
dig example.com
```

Read the TTL field in the ANSWER SECTION (second column). This is the remaining TTL at the resolver you queried, counting down.

```bash
# Query the authoritative server directly to see the actual configured TTL
dig @ns1.example.com example.com
```

The TTL at the authoritative server is the starting value. The TTL at a resolver is that value minus the time since the resolver cached the response.

**Pre-change TTL drain procedure:**
1. Find current TTL: `dig @authoritative-ns example.com` — note the TTL (e.g., 3600 seconds = 1 hour).
2. Reduce TTL to 60 seconds *at least* one full current-TTL period before the change (e.g., if TTL is 3600, make the change 1 hour before you need fast propagation).
3. After all caches expire, make the DNS record change.
4. After confirming the change is stable, restore TTL to 3600 or higher.

**Never change a high-TTL record and expect instant propagation.** If TTL is 3600 and you just changed the record, some resolvers will serve the old answer for up to 1 hour.

### 1.6 Short Output for Scripting

**Purpose**: Get just the IP address, suitable for scripting or quick visual check.

```bash
dig example.com +short
# Output: 93.184.216.34

dig example.com MX +short
# Output: 10 mail.example.com.

dig example.com NS +short
# Output: ns1.example.com.
#         ns2.example.com.
```

### 1.7 Reverse DNS Lookup (PTR Record)

**Purpose**: Find the hostname associated with an IP address. Used for identifying unknown IPs in logs, verifying that a server's PTR matches its A record (required for mail servers).

```bash
dig -x 8.8.8.8
# Returns: dns.google. (Google's PTR for 8.8.8.8)

dig -x 93.184.216.34
```

**Why PTR matters for email**: Mail servers check PTR records to verify that the sending IP matches the sending hostname. Missing or mismatched PTR causes email to be flagged as spam or rejected. If email deliverability issues arise, PTR is one of the first checks.

---

### 1.8 Check SOA Record

**Purpose**: Find the authoritative nameserver and check zone serial number. Used to verify that zone changes propagated across multiple authoritative nameservers.

```bash
dig example.com SOA
```

**Normal output:**
```
example.com.    3600    IN    SOA    ns1.example.com. hostmaster.example.com. (
                                    2026070601  ; Serial (YYYYMMDDNN format)
                                    3600        ; Refresh
                                    900         ; Retry
                                    604800      ; Expire
                                    300 )       ; Minimum TTL
```

**Key fields:**
- `ns1.example.com.` — the primary authoritative nameserver
- `2026070601` — the serial number. Secondary nameservers compare their serial to this. If your secondary has a lower serial, it has not picked up recent changes yet.
- `Refresh: 3600` — secondary nameservers check for zone updates every 3600 seconds

**Verifying zone propagation across authoritative nameservers:**
```bash
dig @ns1.example.com example.com SOA +short
dig @ns2.example.com example.com SOA +short
# Compare the serial numbers — they should match
```

---

### 1.9 DNSSEC Validation

**Purpose**: Verify that DNSSEC signatures are valid. Used when troubleshooting SERVFAIL errors that may be DNSSEC-related.

```bash
dig example.com +dnssec
```

Look for the `AD` (Authenticated Data) flag in the flags section:
```
;; flags: qr rd ra ad; ...
```
`ad` flag present = DNSSEC validation passed.

If you get SERVFAIL and the domain uses DNSSEC, the chain of trust may be broken. This is a common cause of mysterious "everything was fine, now it's broken" DNS failures after key rotation.

---

### 1.10 Measure DNS Response Time

**Purpose**: Quantify resolver latency. Useful for comparing resolvers or diagnosing slow DNS performance.

```bash
dig example.com | grep "Query time"
# Output: ;; Query time: 12 msec

# Compare two resolvers
dig @8.8.8.8 example.com | grep "Query time"
dig @1.1.1.1 example.com | grep "Query time"
dig @10.0.0.53 example.com | grep "Query time"
```

> 100ms from inside a datacenter = slow resolver. Check resolver load, network path to resolver, or consider switching to a closer resolver.

## 2. nslookup — Windows-Compatible Tool

`nslookup` is the standard DNS tool on Windows and is universally available. Less powerful than `dig`, but useful when you only have a Windows jump box or are helping a Windows-side engineer.

### 2.1 Basic Query

```cmd
nslookup example.com
```

**Output:**
```
Server:  dns.company.com
Address:  10.0.0.53

Non-authoritative answer:
Name:    example.com
Address:  93.184.216.34
```

"Non-authoritative answer" means the answer came from a resolver cache, not directly from the authoritative server. This is normal.

### 2.2 Query a Specific Server

```cmd
nslookup example.com 8.8.8.8
```

### 2.3 Query a Specific Record Type

```cmd
nslookup -type=MX example.com
nslookup -type=NS example.com
nslookup -type=TXT example.com
nslookup -type=SOA example.com
```

### 2.4 Reverse Lookup

```cmd
nslookup 8.8.8.8
```

### 2.5 Interactive Mode (for Multiple Queries)

```cmd
nslookup
> set type=MX
> example.com
> set type=NS
> example.com
> exit
```

## 3. Incident-Specific Recipes

These are the commands you run during an actual incident. Copy-paste-ready.

| Scenario | Command | What to Look For |
|----------|---------|-----------------|
| Is my DNS record propagated to public resolvers? | `dig @8.8.8.8 domain.com +short && dig @1.1.1.1 domain.com +short` | Same IP from both. If different, one resolver has a stale cache — wait for TTL. |
| How long until the old record expires? | `dig domain.com \| grep -A5 "ANSWER"` | TTL value (seconds). That's how long until the cached answer expires at a resolver. |
| Which NS servers are authoritative? | `dig domain.com NS +short` | List of nameservers. Use these for `@authoritative` queries. |
| Full resolution trace (where does it break?) | `dig +trace domain.com` | Watch for where the chain stops. The last successful delegation level is where the problem is. |
| Does the name even exist? | `dig nonexistent.domain.com` | `NXDOMAIN` in status field = name does not exist. `NOERROR` with empty answer = name exists but has no A record. |
| Verify failover happened (new IP serving) | `dig @8.8.8.8 domain.com +short` before and after change | IP address changed to new value. |
| Check propagation to multiple public resolvers | `for ns in 8.8.8.8 1.1.1.1 9.9.9.9 208.67.222.222; do echo "$ns: $(dig @$ns domain.com +short)"; done` | Consistent IP from all four. Inconsistency = propagation lag — wait for lowest TTL to expire. |
| Is the authoritative server returning the right answer? | `dig @$(dig domain.com NS +short \| head -1) domain.com +short` | The authoritative answer is the ground truth. If this is wrong, fixing the resolver does nothing. |
| What's the real TTL at the authoritative NS? | `dig @ns1.domain.com domain.com` | TTL in ANSWER SECTION — this is the configured TTL, not a counted-down cached value. |
| Is DNSSEC broken? | `dig domain.com +dnssec` | Look for `ad` flag and `RRSIG` records in ANSWER section. SERVFAIL + DNSSEC = broken chain of trust. |
| Find all IPs a load-balanced name returns | `for i in {1..5}; do dig domain.com +short; done` | Multiple IPs rotating = round-robin DNS. One IP = single endpoint or sticky resolver. |
| Check internal DNS resolution from a pod | `kubectl run -it --rm debug --image=busybox --restart=Never -- nslookup service-name.namespace.svc.cluster.local` | Should return ClusterIP. `NXDOMAIN` = CoreDNS issue or wrong service name. |

## 4. DNS Record Types Quick Reference

| Record Type | Full Name | Purpose | Example |
|-------------|-----------|---------|---------|
| A | Address | Maps a hostname to an IPv4 address | `api.company.com → 203.0.113.10` |
| AAAA | IPv6 Address | Maps a hostname to an IPv6 address | `api.company.com → 2001:db8::1` |
| CNAME | Canonical Name | Alias — points one name to another name | `www.company.com → company.com` |
| MX | Mail Exchange | Specifies mail servers, with priority | `company.com → 10 mail.company.com` |
| NS | Nameserver | Specifies which servers are authoritative for the zone | `company.com → ns1.provider.com` |
| TXT | Text | Free-form text — SPF, DKIM, domain verification, DMARC | `company.com → "v=spf1 include:sendgrid.net ~all"` |
| SOA | Start of Authority | Zone metadata — primary NS, serial, timers | `company.com → ns1.company.com. admin.company.com. 2026070601...` |
| PTR | Pointer | Reverse DNS — maps IP to hostname | `10.113.0.203.in-addr.arpa → api.company.com` |
| SRV | Service | Service location — protocol, host, port | `_http._tcp.company.com → 0 5 80 api.company.com` |
| CAA | Certification Authority Authorization | Controls which CAs can issue SSL certs | `company.com → 0 issue "letsencrypt.org"` |
| DKIM | (TXT record) | Email signing key — verifies sender authenticity | `mail._domainkey.company.com → "v=DKIM1; k=rsa; p=..."` |
| DMARC | (TXT record) | Email authentication policy — what to do with failed SPF/DKIM | `_dmarc.company.com → "v=DMARC1; p=reject; rua=mailto:dmarc@company.com"` |

**CNAME restriction**: A CNAME cannot coexist with other record types at the same name. You cannot have both `company.com CNAME` and `company.com MX` — this is why the root domain (`@`) cannot be a CNAME. Use an A record or ALIAS/ANAME records for the root.

## 5. Common DNS Failure Patterns

| Failure Pattern | Likely Cause | First Command | Follow-Up |
|----------------|-------------|--------------|----------|
| Everything fails suddenly | Resolver unreachable, DNS port blocked | `dig @8.8.8.8 google.com` — does external DNS work? | If yes: internal resolver is down. If no: DNS port blocked or network issue. |
| Some names fail, others work | Split-horizon DNS misconfiguration; zone not loaded | `dig internal.service.com` vs `dig @authoritative-ns internal.service.com` | Compare answers. Authoritative is source of truth. |
| Works from some locations, not others | Propagation lag (TTL not expired); regional resolver cache | `for ns in 8.8.8.8 1.1.1.1 9.9.9.9; do dig @$ns domain.com +short; done` | Different answers = propagation lag. Wait for TTL. |
| Intermittent failures | Round-robin returning a bad IP; resolver flapping | Run `dig domain.com +short` 10 times. Count distinct IPs. | If multiple IPs: check health of each IP separately. One may be down. |
| SERVFAIL for a specific domain | DNSSEC validation failure; authoritative NS unreachable | `dig domain.com +dnssec` for DNSSEC; `dig @authoritative-ns domain.com` for NS reachability | If DNSSEC: escalate to domain owner. If NS unreachable: network problem to NS. |
| NXDOMAIN for a known record | Record was deleted; wrong zone; negative TTL caching | `dig @authoritative-ns domain.com` | If authoritative returns NXDOMAIN: record is missing at source. If authoritative returns the record: resolver has cached NXDOMAIN. Wait for negative TTL. |
| DNS slow (> 100ms) from a region | Resolver overloaded; resolver far away; network congestion | `dig @10.local.resolver domain.com \| grep "Query time"` then `dig @8.8.8.8 domain.com \| grep "Query time"` | Compare times. If internal slow but external fast: internal resolver capacity issue. |
