# Network Load Balancer Comparison Matrix

**Audience**: Directors and SRE Managers making procurement and architecture decisions — not engineers writing config files.

---

## The Real Decision Framework

Choosing a load balancer is not primarily a technical decision. The technology question ("does it support WebSockets?") is almost always answered "yes" by every option in this list. The real questions are:

**Operational**: Who manages it, who's on-call for it, and what's the blast radius when it breaks?

**Financial**: Is this a capital expense (hardware), operational expense (cloud usage), or SaaS contract? How does cost scale with traffic growth?

**Organizational**: Does your team have the skills to run it? A self-managed HAProxy cluster is dramatically cheaper than AWS ALB at scale — but only if you have someone who can debug HAProxy reloads at 3am.

**Strategic**: Are you deepening cloud vendor lock-in or maintaining portability? A Cloudflare-dependent architecture is fast to build and hard to exit.

Answer these three questions before evaluating feature tables.

## When to Use Each Option

### HAProxy
**Use when:**
- You need maximum control over load balancing behavior — custom ACLs, custom health check logic, specific TCP-level behavior.
- You have high-throughput non-HTTP protocols (MySQL, PostgreSQL, RabbitMQ) that need L4 load balancing with sophisticated health checks.
- Budget is constrained and you have a strong networking/ops team that knows the tool.
- You are not in AWS/GCP and want a self-managed solution with no cloud dependency.

**Avoid when**: Your team does not have HAProxy expertise in-house. A misconfigured HAProxy is worse than a managed service. The operational overhead is real.

**Director tell**: If an engineer says "we run HAProxy for our database tier," that's a sign of a mature, ops-heavy team. If they say "we run HAProxy for our HTTP tier in AWS," ask why they're not using ALB — there may be a legitimate reason (static IP requirement, advanced ACLs), or it may be legacy.

---

### NGINX
**Use when:**
- You need an HTTP-layer load balancer and reverse proxy that your team already knows.
- You are running Kubernetes and need an Ingress controller — NGINX Ingress is the most battle-tested option with the broadest community.
- You want a single tool that handles load balancing and static file serving / API gateway behavior.
- You are migrating from an on-premises environment where NGINX is already in use.

**Avoid when**: You need WAF, DDoS protection, or global anycast — NGINX alone does not provide these. Combine with Cloudflare or a cloud WAF.

**Director tell**: NGINX Ingress is the default Kubernetes Ingress for most shops that predate the AWS Load Balancer Controller. If your team is on AWS and still using NGINX Ingress for external traffic, ask whether migrating to ALB Ingress would reduce operational overhead.

---

### AWS ALB (Application Load Balancer)
**Use when:**
- Your workloads are on AWS and you want HTTP/HTTPS load balancing with zero operational overhead.
- You need advanced routing rules (path-based, host-based, header-based, query-string-based routing).
- You are using EKS and want the AWS Load Balancer Controller to manage ALBs automatically from Kubernetes resources.
- You need native AWS service integration (Cognito for auth, WAF, Shield, ACM for certificates).

**Avoid when:** You need a static IP (ALBs use DNS, not static IPs — use NLB instead). You need non-HTTP protocols (RabbitMQ, Kafka, databases — use NLB). You need global anycast (ALB is regional).

**Cost note**: ALB pricing is per LCU-hour. At low traffic, ALB is cheaper than managing VMs. At very high traffic (sustained millions of requests/minute), ALB cost can exceed self-managed options. Run the numbers if you're at >100K req/min sustained.

---

### AWS NLB (Network Load Balancer)
**Use when:**
- You need to load balance non-HTTP TCP or UDP protocols: RabbitMQ (5672), Kafka (9092), databases, custom TCP services.
- You need a static IP address (NLB supports Elastic IPs — useful for IP allowlisting by downstream partners or firewalls).
- You need ultra-low latency — NLB operates at L4 and adds microseconds of overhead vs. milliseconds for ALB.
- You need to preserve client source IP at the backend (NLB can do this; ALB uses X-Forwarded-For headers).

**Avoid when:** You need SSL termination at the load balancer (NLB does passthrough — your backends must handle TLS). You need HTTP routing rules (use ALB for that). You need WebSocket with sticky sessions (use ALB).

**Director tell**: If your platform runs RabbitMQ or Kafka on AWS, the correct answer for load balancing those protocols is NLB, not ALB. If someone proposes ALB for Kafka, ask how they plan to handle the TCP-level protocol requirements.

---

### GCP Load Balancer (Global External LB)
**Use when:**
- Your workloads are on GCP and you want global anycast built-in — GCP's Global LB is the only major cloud-native LB with true anycast out of the box.
- You are on GKE and want native Kubernetes integration.
- You need Cloud Armor (GCP's WAF/DDoS product) integrated with your load balancer — it attaches natively to GCP LB.
- You have users globally and want traffic to be routed to the nearest GCP region automatically.

**Avoid when:** You are not on GCP (obviously). You need multi-cloud portability.

---

### Cloudflare
**Use when:**
- You want a single vendor for DNS + DDoS protection + WAF + CDN + load balancing at the edge.
- Your threat model includes volumetric DDoS — Cloudflare has the largest network capacity for DDoS mitigation.
- Your users are globally distributed and you want the lowest possible latency via anycast edge serving.
- You want to reduce infrastructure complexity: Cloudflare's edge can replace a separate CDN, WAF, DDoS scrubber, and LB layer.

**Avoid when:** You have strict data residency requirements that conflict with Cloudflare's global anycast routing. You need to terminate TLS inside your own infrastructure for compliance. Your team needs full control over routing logic that Cloudflare's policy model doesn't support.

**Director tell**: Cloudflare is the right answer for "we need DDoS protection and we need it fast." It is also a vendor concentration risk — if Cloudflare has an outage, all of your edge goes down simultaneously (this has happened). Design for this by keeping a fallback origin path that doesn't depend on Cloudflare.

---

### Fastly
**Use when:**
- You need a programmable CDN/LB where you can write custom logic (Fastly's VCL or WASM compute) to route, transform, or authenticate requests at the edge.
- You need real-time cache purging — Fastly's instant purge is genuinely faster than competitors.
- You are a media/content company or developer platform that needs edge compute capability beyond what a rules-based CDN provides.
- You need real-time log streaming to your SIEM without delay.

**Avoid when:** Your team does not have VCL or edge-compute expertise — Fastly's power comes with a learning curve. For simple HTTP load balancing, Fastly is overengineered.

**Director tell**: Fastly is the choice when engineering teams say "we need to run logic at the edge, not just cache." If the use case is pure load balancing without edge compute, Cloudflare or a cloud-native LB is simpler.

## Cost Comparison — Order of Magnitude

These are directional estimates, not quotes. Actual costs depend heavily on traffic patterns and negotiated pricing.

| Option | Cost Driver | Low Traffic (<1K req/s) | Medium Traffic (10K req/s) | High Traffic (100K+ req/s) | Predictability |
|--------|-----------|------------------------|--------------------------|---------------------------|---------------|
| HAProxy / NGINX (self-managed) | EC2 instances + labor | $200–500/mo (2× small EC2) | $500–1500/mo (larger instances, HA pair) | $2K–5K/mo (large instances, autoscaling group) + ops labor | Very predictable — fixed infra cost |
| AWS ALB | Per LCU-hour | $20–50/mo | $200–500/mo | $2K–8K/mo (can spike with traffic bursts) | Moderate — scales with traffic |
| AWS NLB | Per LCU-hour | $20–50/mo | $150–400/mo | $1K–5K/mo | Moderate — scales with traffic |
| GCP Global LB | Per rule-hour + GB | $30–80/mo | $300–800/mo | $3K–10K/mo | Moderate |
| Cloudflare (Pro/Biz/Enterprise) | Per request + flat-rate tiers | $20/mo (Pro) to $200/mo (Biz) | $200–1000/mo (Biz or negotiated enterprise) | Custom enterprise pricing — often flat-rate | High at enterprise tier (flat rate) |
| Fastly | Per request + bandwidth | $50–200/mo | $500–2000/mo | Custom enterprise pricing | High at enterprise tier |

**Key insight**: Self-managed (HAProxy/NGINX) costs scale linearly with infrastructure and are predictable, but require ops labor that often costs more than the infrastructure savings. Cloud managed (ALB/NLB) costs scale with traffic and can spike, but eliminate the ops burden. SaaS (Cloudflare/Fastly) has the most predictable costs at scale under enterprise flat-rate agreements, and include DDoS protection and CDN that you'd pay for separately otherwise.

## Director Recommendation Matrix

| Scenario | Recommended LB | Primary Rationale |
|---------|---------------|------------------|
| Pure AWS, HTTP/HTTPS workloads, want managed | **AWS ALB** | Native integration, no ops overhead, rich HTTP routing rules, ACM for cert management |
| AWS, non-HTTP protocols (RabbitMQ AMQP, Kafka, databases) | **AWS NLB** | L4 required; static IP option; preserves source IP |
| AWS, need both HTTP routing and TCP protocol support | **ALB (HTTP) + NLB (TCP)** — run both | Each handles its layer; NLB provides the TCP/static-IP capability, ALB handles HTTP tier |
| Multi-cloud or hybrid, global users | **Cloudflare or Fastly** | Global anycast, single control plane across clouds, DDoS protection included |
| Kubernetes-only platform, AWS | **NGINX Ingress + AWS NLB** | NGINX Ingress handles HTTP routing within cluster; NLB exposes NGINX to internet with static IP |
| GCP-native, global users | **GCP Global External LB** | Anycast built-in, Cloud Armor for WAF/DDoS, native GKE integration |
| DDoS-heavy threat model (financial, gaming, large consumer) | **Cloudflare** | Largest DDoS scrubbing capacity; WAF + CDN + LB in one vendor |
| Programmable edge logic needed | **Fastly** | VCL/WASM compute at edge; instant purge; real-time logging |
| T-Mobile-scale notification/messaging platform on AWS | **ALB + Fastly CDN** | ALB handles origin load balancing with EKS integration; Fastly provides global edge caching and real-time log streaming |
| Budget-constrained, strong ops team, non-HTTP protocols | **HAProxy** | Maximum control, no licensing cost, handles L4+L7; requires ops expertise |
| Legacy on-premises migration to cloud | **NGINX Ingress → ALB Ingress migration** | Start with what the team knows; migrate to cloud-native as team gains AWS experience |

## Five Questions to Ask in Any LB Architecture Review

1. **"What happens when the LB itself fails?"** — Self-managed needs HA pairs and failover. Cloud-managed and SaaS handle this internally. Know which you have.

2. **"What's the SSL termination point, and does compliance care?"** — If TLS must terminate inside your VPC for compliance, Cloudflare and Fastly (which terminate TLS at the edge) may not be acceptable. Know your compliance requirements.

3. **"Do we have a static IP requirement?"** — If downstream partners or firewalls need to allowlist your IPs, you need NLB (AWS) or a self-managed option. ALB, Cloudflare, and Fastly use DNS-based or rotating IPs.

4. **"What's the blast radius if this LB tier fails?"** — A single ALB in front of all services is a SPOF. Ask how many independent LB instances or clusters exist.

5. **"How do we do blue/green or canary deployments through this LB?"** — Every option supports it differently. The answer reveals how mature the team's deployment practice is.
