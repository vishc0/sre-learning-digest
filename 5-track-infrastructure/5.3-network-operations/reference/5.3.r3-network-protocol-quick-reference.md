# Network Protocol Quick Reference

**Audience**: Directors and SRE Managers who need to follow a technical conversation, ask smart questions, and detect when engineers are handwaving — not exam-level depth.

**How to use this**: Read the "Director Need-to-Know" column before any architecture review, incident bridge, or vendor conversation. The goal is to ask the right question, not to configure the device yourself.

---

## 1. OSI Layer Reference

Every networking conversation references these layers eventually. Know them well enough to ask: "Is this a Layer 3 routing problem or a Layer 4 connection problem? They have different on-call owners."

| Layer | Name | What It Does | Key Protocols | Failure Symptoms | On-Call Owner |
|-------|------|-------------|--------------|-----------------|--------------|
| 7 | Application | Handles the actual data the user sees. Encoding, encryption at app level, app-specific protocols. | HTTP/S, DNS, SMTP, FTP, SSH, gRPC | HTTP 5xx errors, app timeouts, cert validation errors | App team / SRE |
| 6 | Presentation | Data format translation — converts between network format and app format. Handles encryption/decryption. | TLS 1.2/1.3, SSL (legacy), JSON/XML encoding | TLS handshake failures, cert errors, encoding mismatches | SRE / Security |
| 5 | Session | Manages connection lifecycle — establishes, maintains, and terminates sessions. | NetBIOS, RPC, WebSocket (session management) | Connection resets, session timeouts, half-open connections | SRE / App team |
| 4 | Transport | End-to-end data delivery. Handles ports, segmentation, reliability (TCP) or speed (UDP). | TCP, UDP, QUIC | Connection timeouts, retransmits, port exhaustion, SYN floods | SRE / Network |
| 3 | Network | Logical addressing and routing between networks. Decides where packets go. | IP (v4/v6), BGP, OSPF, ICMP, MPLS | Routing failures, ICMP unreachable, no path to host, black holes | Network SRE |
| 2 | Data Link | Frame delivery between adjacent nodes on the same network segment. MAC addresses, VLANs. | Ethernet, 802.1Q (VLAN), ARP, LACP | CRC errors, duplex mismatch, VLAN misconfiguration, ARP failures | Network / DC ops |
| 1 | Physical | Raw bit transmission — electrical signal, light pulse, radio wave. | Fiber (single/multi-mode), Cat6/7, SFP optics, DWDM | Link down, interface flapping, high error rate, optic degradation | DC operations |

**Director pattern recognition:**

- "Is it a Layer 1-2 issue?" → Physical cabling, optics, switch port. The network team owns it. Ask for interface error counters.
- "Is it a Layer 3 issue?" → Routing. Ask "does traceroute show the expected path? Where does it stop?"
- "Is it a Layer 4 issue?" → TCP connections. Ask "are we seeing SYN_SENT or ESTABLISHED in the connection state? Are retransmits climbing?"
- "Is it a Layer 7 issue?" → Application. The error is probably in the app logs, not the network. Ask for `curl -v` output.

## 2. Routing Protocols Quick Reference

Routing protocols are how routers tell each other where to send traffic. A Director needs to know which protocol is in use and what failure modes look like — not how to configure them.

| Protocol | Type | Convergence Speed | Use Case | Director Need-to-Know |
|----------|------|------------------|---------|----------------------|
| BGP (Border Gateway Protocol) | Path-vector; policy-based | Slow (seconds to minutes) | Internet routing between autonomous systems; multi-datacenter; multi-ISP; anycast | Your most important protocol. Every internet circuit runs BGP. Multi-ISP means BGP policy determines which ISP carries your traffic. Ask "what's our BGP policy for egress traffic?" and "do we have route reflectors?" |
| OSPF (Open Shortest Path First) | Link-state; cost-based | Fast (sub-second) | Internal datacenter routing; campus networks | Used inside datacenters for internal routing. Fast convergence makes it good for internal failover. Directors typically not involved in OSPF directly, but ask "does our internal routing converge before BGP advertises us as reachable?" |
| EIGRP (Enhanced IGRP) | Cisco hybrid | Fast | Enterprise campus networks | Cisco-proprietary. Common in legacy enterprise environments. If you see it, ask "is this on a migration roadmap to OSPF or BGP?" It signals a Cisco-heavy shop. |
| IS-IS (Intermediate System) | Link-state | Fast | ISP core networks, carrier backbone | You will see this when talking to carriers about their backbone. Not typically used inside enterprise networks. Awareness-level only. |
| Static routes | N/A | Instant (no convergence) | Simple point-to-point paths; failover edge cases; default routes | Predictable but brittle. Static routes do not self-heal. Ask "where do we use static routes and do we have monitoring on those paths?" |

**BGP specifically — what a Director must understand:**

- **AS (Autonomous System)**: Your organization has an AS number (e.g., AS12345). BGP routes traffic between ASes.
- **BGP session**: A TCP connection between two routers exchanging routing information. If it drops, traffic stops.
- **BGP policy**: Rules that control which routes you accept from peers and which routes you advertise. Misconfigured policies cause route leaks (advertising routes you shouldn't) or blackholes (not advertising routes you should).
- **Multi-ISP**: Most enterprises connect to 2+ ISPs for redundancy. BGP determines which ISP is primary and how failover works.
- **Anycast**: Multiple locations advertise the same IP prefix. Traffic goes to the nearest location. Used by DNS (1.1.1.1, 8.8.8.8) and CDNs.

## 3. Transport Layer Reference

Transport is where the reliability/speed tradeoff lives. It directly affects application behavior.

| Protocol | Connection | Ordering | Reliability | Typical Latency Overhead | Use Cases | Director Need-to-Know |
|----------|-----------|---------|-------------|--------------------------|-----------|----------------------|
| TCP | Yes — 3-way handshake | Yes — sequence numbers | Yes — retransmit lost packets | ~1 RTT to establish | HTTP, databases, SSH, RabbitMQ (AMQP), Kafka | The workhorse. Reliable but adds latency for short connections. High retransmits in your metrics = packet loss on the path. |
| UDP | No | No | No — fire and forget | None | DNS queries, video streaming, VoIP, online gaming, syslog | Fast, low overhead. No retry. Applications built on UDP handle their own reliability if needed (or accept loss). DNS runs on UDP by default (falls back to TCP for large responses). |
| QUIC | Yes (logical, multiplexed) | Yes | Yes — but no head-of-line blocking | ~0.5 RTT (0-RTT resumption possible) | HTTP/3, Google services, newer cloud APIs | HTTP/3's transport. Solves TCP's head-of-line blocking problem. Runs over UDP (so it doesn't look like TCP to your firewall — check that UDP 443 is not blocked). |

**Director pattern recognition:**

- High TCP retransmit rate → packet loss somewhere on the path. Network problem, not an application problem.
- DNS failing → everything fails. DNS runs on UDP 53. Check that your firewalls allow UDP 53 to your resolvers.
- "We're seeing connection latency spikes" → check TCP handshake latency separately from data transfer latency. A spike in handshake RTT is a network problem; a spike in data transfer is application or server.

## 4. WAN and Private Connectivity Reference

How your datacenters, offices, and cloud providers connect to each other privately.

| Technology | What It Is | Cost Model | Latency | Use Case | Director Need-to-Know |
|-----------|-----------|-----------|---------|---------|----------------------|
| MPLS | Label-switched WAN — carrier-managed private network. Traffic never touches the public internet. | High — per-Mbps MRC | Predictable, low | Enterprise WAN between offices; legacy datacenter interconnects | Legacy but reliable. If your enterprise has offices connected over MPLS, it's managed by the carrier. Ask "what's our SLA for MPLS circuits and do we have a failover path?" |
| SD-WAN | Software-defined WAN — uses multiple cheaper internet links (broadband, LTE, MPLS) intelligently. Policy routes traffic across the best-available path. | Medium — hardware + licensing | Variable (depends on path selection) | Modern enterprise WAN replacing pure MPLS | Gives you multi-path flexibility. Ask "what's our SD-WAN failover policy — is it active-active or active-passive?" |
| MPLS VPN (L3VPN) | MPLS network with VRF (Virtual Routing and Forwarding) isolation — each customer gets a private routing table | High | Low | Multi-tenant enterprise WAN; carrier offering for multi-site isolation | The "private cloud" of WAN circuits. Route isolation means your traffic cannot leak to other tenants. Important for compliance. |
| AWS Direct Connect | AWS-managed physical circuit from your facility or colo to AWS region. Bypasses the public internet. | High — port hour + data transfer | 1–10ms to AWS region | AWS workloads requiring consistent bandwidth or compliance (no internet path) | If you run significant AWS workloads, ask "do we have Direct Connect or are we going over the public internet? What's the redundancy model?" Single Direct Connect is a SPOF. |
| Azure ExpressRoute | Azure equivalent of Direct Connect. Private circuit to Azure. | High | 1–10ms to Azure region | Azure workloads requiring consistent bandwidth | Same model as Direct Connect. Ask the same questions. |
| GCP Cloud Interconnect | GCP equivalent. Dedicated or Partner Interconnect options. | High | 1–10ms to GCP region | GCP workloads | Same model. Partner Interconnect uses a carrier intermediary. |
| Site-to-Site VPN (IPSec) | Encrypted tunnel over the public internet between two endpoints. | Low — internet transit only | Higher than private circuits; subject to internet congestion | Small sites; backup path for private circuits; dev/test environments | Cheap and quick to set up. Not appropriate as primary path for production traffic over ~100Mbps. Always treat as a backup or non-critical path. |

## 5. Common Ports — Director Reference

A Director needs to know the ports that matter for firewall rules, security reviews, and troubleshooting conversations.

| Port | Protocol | Service | Why a Director Cares |
|------|----------|---------|---------------------|
| 53 | UDP (primary) / TCP (fallback) | DNS | If UDP 53 is blocked, DNS resolution fails, and literally everything stops working. First thing to check in a "can't reach anything" incident. |
| 80 | TCP | HTTP | Unencrypted web. Many services still redirect here before upgrading to HTTPS. |
| 443 | TCP (HTTPS) / UDP (HTTP/3/QUIC) | HTTPS / HTTP/3 | Core encrypted web traffic. If you block UDP 443, you break HTTP/3 for modern clients. |
| 179 | TCP | BGP | Must be open between your routers and ISP peering points. If this port is blocked, BGP sessions drop and you lose internet routing. |
| 22 | TCP | SSH | Remote management of network devices and servers. Restrict to management networks only. |
| 25 / 587 / 465 | TCP | SMTP | Email. 25 for server-to-server, 587 for authenticated submission, 465 for legacy secure submission. |
| 5671 / 5672 | TCP | AMQP (RabbitMQ) | 5672 unencrypted, 5671 TLS. Core messaging bus for platforms using RabbitMQ. Firewall rules must allow this between app servers and RabbitMQ cluster nodes. |
| 9092 | TCP | Kafka | Event streaming. Kafka clients connect on 9092 (plain) or 9093 (TLS). Must be open from producers and consumers to the Kafka cluster. |
| 2181 | TCP | ZooKeeper (Kafka dependency) | Kafka coordination service. Must be open between Kafka brokers and ZooKeeper nodes. Not client-facing — internal only. |
| 6443 | TCP | Kubernetes API server | EKS control plane endpoint. kubectl and all automation that talks to the cluster uses this port. If blocked, you cannot manage your cluster. |
| 2379–2380 | TCP | etcd | Kubernetes control plane data store. Must be open between control plane nodes only. Never expose to internet. |
| 10250 | TCP | Kubelet API | kube-apiserver to kubelet communication. Must be open from control plane to worker nodes. |
| 443 | TCP | AWS API endpoints | All AWS SDK calls go to 443. If egress HTTPS is restricted by security groups, AWS API calls (S3, SQS, etc.) will fail. |
| 3306 | TCP | MySQL / Aurora | Database port. Should be restricted to app server security groups only. |
| 5432 | TCP | PostgreSQL / Aurora PG | Same principle as MySQL. Restrict to app tiers only. |
| 6379 | TCP | Redis | Cache layer. Restrict to app and backend tiers. Never expose to internet. |

## 6. Key Metrics — What to Look for on a Dashboard

When you are handed a Grafana or Datadog dashboard during an incident, know what these metrics mean.

| Metric | What It Measures | Healthy Range | Red Flag | Root Cause Direction |
|--------|-----------------|--------------|---------|---------------------|
| RTT (Round-Trip Time) | Time for a packet to travel to destination and back | Depends on distance: LAN < 1ms, same-region internet < 20ms, cross-country < 80ms | > 2× baseline for the link type | Physical path issue, congestion, routing change |
| Packet loss | Percentage of transmitted packets that do not arrive | < 0.1% | > 0.1% sustained | Physical layer degradation (optic, fiber), buffer overflow, misconfigured QoS |
| Interface utilization | Percentage of link capacity in use | < 80% sustained | > 90% sustained | Link saturation — need more capacity or traffic management |
| TCP retransmit rate | Percentage of TCP segments retransmitted due to loss or timeout | < 0.1% | > 1% | Packet loss on the path — confirms network problem, not app problem |
| BGP prefix count | Number of routes received from a BGP peer | Stable (internet full table = ~900K prefixes as of 2024) | Sudden drop (route withdrawal) or spike (route leak) | BGP misconfiguration, ISP incident, prefix filtering issue |
| DNS query latency | Time to resolve a DNS name | < 10ms from within a region for local resolvers | > 50ms or increasing trend | Resolver overload, network congestion, resolver misconfiguration |
| DNS NXDOMAIN rate | Rate of "name does not exist" responses | Low and stable | Sudden spike | Misconfigured app pointing to wrong name, DNS record deleted |
