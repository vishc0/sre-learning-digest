# Chapter 2 — VMware Stack Architecture

## The VMware Stack in One View

VMware's platform is a layered system: ESXi handles raw compute at the host level, vCenter provides centralized management above it, vSAN abstracts storage across hosts, and NSX abstracts networking across hosts. Together they form what VMware calls the **software-defined data center (SDDC)**.

```
┌─────────────────────────────────────────────────────────────────┐
│                        VMware vSphere Platform                  │
│                                                                 │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐      │
│  │   vCenter    │    │   vCenter    │    │  vCenter HA  │      │
│  │   Server     │◄───│   Plugins    │    │   Standby    │      │
│  └──────┬───────┘    └──────────────┘    └──────────────┘      │
│         │ Management Plane                                       │
│         │ (API, UI, Scheduling, Policies)                       │
│  ───────┼────────────────────────────────────────────           │
│         │ Control signals                                        │
│  ┌──────▼───────┐  ┌──────────────┐  ┌──────────────┐         │
│  │  ESXi Host 1 │  │  ESXi Host 2 │  │  ESXi Host 3 │         │
│  │  (Hypervisor)│  │  (Hypervisor)│  │  (Hypervisor)│         │
│  │  ┌────┐┌────┐│  │  ┌────┐┌────┐│  │  ┌────┐┌────┐│         │

│  │  └────┘└────┘│  │  └────┘└────┘│  │  └────┘└────┘│         │
│  └──────────────┘  └──────────────┘  └──────────────┘         │
│         │                  │                  │                  │
│  ┌──────▼──────────────────▼──────────────────▼───────┐        │
│  │                    vSAN Storage Plane               │        │
│  │         (distributed across local SSDs/HDDs)        │        │
│  └─────────────────────────────────────────────────────┘        │
│                                                                 │
│  ┌─────────────────────────────────────────────────────┐        │
│  │                   NSX Networking Plane              │        │

│  └─────────────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────────────┘
```

Each layer is independently configurable but designed to integrate. You can run ESXi without vSAN (using external shared storage). You can run ESXi + vCenter without NSX (using standard physical networking). But you cannot run vSAN or NSX without ESXi — they are dependent on the hypervisor layer.

## ESXi: The Hypervisor (Data Plane)

**ESXi** is VMware's Type 1 bare-metal hypervisor. It installs directly on physical server hardware — no underlying general-purpose OS. ESXi consumes roughly 150MB of RAM for its own operation and dedicates everything else to VMs and their resource allocation.

**Kubernetes analogy**: ESXi is the worker node. Just as a Kubernetes node runs kubelet and the container runtime (containerd, CRI-O) to host pods, ESXi runs the hypervisor kernel to host VMs. A fleet of ESXi hosts is like a Kubernetes worker node pool. Just as a kubelet handles local container lifecycle decisions without needing to call the API server for every operation, ESXi handles local VM CPU scheduling without vCenter involvement for each scheduling tick.

What ESXi manages directly and independently of vCenter:
- Physical CPU scheduling — mapping vCPUs to physical cores, enforcing CPU reservations and limits
- Physical memory allocation — including memory overcommit, ballooning, and transparent page sharing
- Physical NIC I/O — including SR-IOV passthrough and virtual switch traffic forwarding
- Local datastore access — reading and writing VM disk images (VMDK files) to local or attached storage

ESXi has a minimal command-line interface (`esxcli`, `vim-cmd`, ESXCLI shell) for emergency administration, but it is not designed to be operated directly at scale. Direct ESXi shell access is the equivalent of SSH-ing directly to a Kubernetes worker node rather than going through the API server — you do it when something is broken, not as a standard workflow.

**ESXi kernel architecture**: The ESXi kernel (VMkernel) is a purpose-built microkernel, not Linux. This is deliberate — a general-purpose OS includes far more code surface than a hypervisor needs, and each additional line of code is a potential vulnerability. VMkernel handles only what a hypervisor must: CPU scheduling, memory management, storage I/O, and network I/O. VMware world processes (VMs, management agents) run on top of VMkernel as isolated execution contexts.

## vCenter Server: The Control Plane

**vCenter Server** is VMware's centralized management plane. A single vCenter instance can manage hundreds of ESXi hosts and tens of thousands of VMs across multiple sites and datacenters within one organizational hierarchy.

**Kubernetes analogy**: vCenter is the combination of the Kubernetes API server, controller manager, and scheduler — it accepts declarative commands (via GUI, REST API, or PowerCLI), maintains the desired state of the virtualized environment, and issues instructions to ESXi hosts to execute changes. Just as the Kubernetes scheduler decides which node a pod runs on, vCenter's DRS (Distributed Resource Scheduler) decides which ESXi host a VM runs on.

What vCenter provides beyond what ESXi alone can do:

- **vMotion**: live VM migration between ESXi hosts with zero VM downtime. The VM's memory state is copied to the target host in real time while the source continues running. Once memory is synchronized, a final cutover moves execution to the new host in milliseconds. From the VM's perspective, nothing happened. From the network's perspective, the VM's MAC address moved to a new physical location and ARP caches update.
- **Storage vMotion**: live migration of a VM's disk files between datastores, without downtime. The running VM writes to the source disk while blocks are copied to the target.
- **DRS (Distributed Resource Scheduler)**: monitors CPU and memory utilization across all ESXi hosts in a cluster and automatically migrates VMs via vMotion to balance load. Operates on configurable thresholds — fully automated, partially automated (recommends but doesn't act), or manual.
- **HA (High Availability)**: monitors ESXi host heartbeats. When a host fails, HA automatically restarts its VMs on surviving hosts within the cluster. Recovery time is the VM boot time — typically 1–5 minutes for a full guest OS restart.
- **VM lifecycle management**: create, clone, snapshot, deploy from template, destroy. Snapshot chains allow point-in-time rollback but carry performance penalties when chains grow long.
- **Resource pools**: hierarchical containers that enforce CPU and memory allocations across groups of VMs. A resource pool can guarantee a minimum compute floor to a business unit's VMs regardless of cluster-wide contention.
- **Affinity and anti-affinity rules**: rules that constrain where VMs can run relative to each other. Anti-affinity rules ensure that two critical VMs (e.g., primary and secondary RabbitMQ nodes) never land on the same physical host. This is the VM equivalent of Kubernetes pod anti-affinity with `topologyKey: kubernetes.io/hostname`.
- **Role-based access control (RBAC)**: granular permissions across the object hierarchy — datacenter, cluster, host, VM. An application team can be granted power over their VMs without access to host-level configuration.

**vCenter itself runs as a virtual appliance**: The **VCSA (vCenter Server Appliance)** is a Linux-based virtual machine deployed on one of the ESXi hosts it manages (or a dedicated host). This creates a dependency: if vCenter's host fails, you lose the vCenter management plane. Existing VMs keep running — ESXi hosts operate independently once VMs are started — but you cannot manage the environment until vCenter is restored. HA for vCenter itself (vCenter HA) requires two additional vCenter instances (secondary + witness) in a cluster configuration.

## vSphere: The Platform (Not a Separate Product)

**vSphere** is the brand name for the combined platform of ESXi + vCenter together. It is not a separate installable product — when you install ESXi and vCenter and connect them, you have a vSphere environment.

The "vSphere" term is used the way "Kubernetes" is used — it refers to the platform as a whole, not any single component. Just as Kubernetes is the sum of etcd, API server, controller manager, scheduler, kubelet, and kube-proxy working together, vSphere is ESXi + vCenter + their integration protocols (vMotion protocol, HA heartbeat, DRS telemetry) working together.

## vSAN: Software-Defined Storage

**vSAN** pools the local SSDs and HDDs installed inside ESXi hosts into a distributed, shared storage cluster. Instead of requiring a dedicated SAN (Storage Area Network) array or NAS (Network Attached Storage) appliance, vSAN uses the disks that are already inside your compute servers.

**Kafka analogy**: vSAN is like Kafka's distributed log storage — each broker (ESXi host) contributes local disk, and Kafka's replication protocol (replication factor N) ensures that each partition exists on N brokers simultaneously. If one broker dies, the data remains accessible on the surviving replicas. vSAN operates identically: data is written to multiple hosts according to a **storage policy** that specifies the number of failures to tolerate (FTT — Failures To Tolerate). FTT=1 means data survives one host failure; it requires a minimum of 3 hosts for quorum.

Key vSAN concepts:

- **Failure domains**: by default, each host is a failure domain. vSAN writes data stripes and replicas across failure domains. If you define rack-level failure domains (all hosts in rack A = one domain), vSAN can survive an entire rack failure.
- **Storage policies per VM**: individual VMs can have different storage policies. A database VM might require FTT=2 (survive 2 simultaneous host failures); a test VM might run FTT=0 (no replication, maximum performance). This is analogous to setting different Kafka replication factors per topic based on criticality.
- **All-flash vs hybrid**: vSAN supports all-flash (NVMe/SSD only) for performance-optimized workloads and hybrid (SSD caching tier + HDD capacity tier) for cost-optimized deployments.
- **Deduplication and compression**: vSAN can reduce storage consumption by deduplicating identical data blocks across VMs and compressing data before writing. Similar to Kafka log compaction — reduces space without losing content.
- **vSAN Witness**: when running a 2-host cluster (stretched cluster scenario), a lightweight witness appliance provides quorum without contributing storage, preventing split-brain.

vSAN requires a minimum of 3 ESXi hosts for production use (FTT=1). All hosts in a vSAN cluster must be connected via a dedicated high-bandwidth, low-latency network (10Gbps minimum, 25Gbps recommended for all-flash).

## NSX: Software-Defined Networking

**NSX** (VMware NSX, formerly NSX-T) is VMware's software-defined networking layer. It creates an overlay network on top of the physical fabric, implementing routing, switching, firewalling, load balancing, and NAT entirely in software — without requiring those functions to be provisioned on physical network devices.

**Kubernetes analogy**: NSX is to VMware what Cilium or Calico is to Kubernetes. Just as Cilium implements network policy enforcement at the pod's virtual NIC using eBPF, NSX implements firewall rules at the VM's virtual NIC using a **Distributed Firewall (DFW)**. Just as Calico uses BGP to advertise pod routes, NSX uses **BGP or EVPN** to integrate logical networks with the physical fabric. The key difference: NSX handles both VM workloads and Kubernetes pod workloads — it is the network underlay for Tanzu (VMware's Kubernetes distribution) as well.

Key NSX capabilities:

- **Distributed Firewall (DFW)**: firewall policy is enforced at the VM's virtual NIC by a kernel module running inside each ESXi host. East-west traffic between two VMs on the same host never leaves the server — it is filtered at the source NIC before forwarding. This eliminates the bottleneck of hairpinning traffic through a centralized firewall appliance.
- **Microsegmentation**: every VM can have a security perimeter defined by identity (VM name, tag, OS type) rather than IP address. You can write a rule "allow RabbitMQ VMs to talk to Cassandra VMs on port 9042" without knowing the IP addresses in advance. This is Zero Trust east-west enforcement inside the datacenter.
- **Logical routers**: NSX implements distributed logical routers (DLR) that run as kernel modules on each ESXi host. A VM sending traffic to another subnet has that routing decision made locally on its host, not at a centralized router appliance. This dramatically reduces latency for routed east-west traffic.
- **NSX Edge**: dedicated VMs (or bare-metal nodes) that handle north-south traffic — traffic entering or leaving the datacenter. NSX Edge provides NAT, load balancing, VPN termination, and BGP peering with physical upstream routers.
- **Overlay protocol**: NSX uses **GENEVE** (Generic Network Virtualization Encapsulation) as its overlay protocol, encapsulating VM traffic inside UDP headers for transport across the physical fabric. The physical network only needs to route IP/UDP — it does not need to be VLAN-aware for tenant segmentation.

## How They Connect: The Operational Flow

**VM creation flow**:
```
Admin Request → vCenter API → DRS Scheduler → ESXi Host selected
                                                    ↓
                                             ESXi creates VM process
                                                    ↓
                               vSAN allocates virtual disk (VMDK)
                                                    ↓
                               NSX provisions virtual NIC + connects
                               VM to logical network segment
                                                    ↓
                                             VM boots
```

**Host failure flow**:
```
ESXi Host 3 loses power
        ↓
vCenter HA heartbeat timeout (default: 12 seconds)
        ↓
HA declares host failure → initiates VM restart on Host 1 and Host 2
        ↓
vSAN: data was replicated — Host 1 and Host 2 still have all blocks
        ↓
NSX: VM MAC addresses re-registered on new host virtual switches
        ↓
VMs restart (OS boot time: 1–5 min)
```

The critical observation: vSAN and NSX ensure that a VM can restart on any surviving host because storage and network are not tied to the original physical host. If VMs used local storage and physical NICs, a host failure would mean those VMs' data was gone. vSAN and NSX decouple workloads from physical host identity — the same design principle that makes Kubernetes pods portable across nodes.

## T-Mobile EKS Analogy

Your EKS control plane (API server + etcd) is to worker nodes what vCenter is to ESXi hosts. Your EKS managed node group with autoscaling is like a DRS-managed ESXi cluster — resources scale to meet demand, and the control plane makes placement decisions. The difference in design philosophy: Kubernetes is API-first and cloud-provider-integrated by design, built for ephemeral workloads at internet scale. vCenter is a management plane built for long-lived, stateful VMs in on-premises datacenters. The mental model transfers; the operational mechanics differ in exactly the ways you would expect given those different design goals.
