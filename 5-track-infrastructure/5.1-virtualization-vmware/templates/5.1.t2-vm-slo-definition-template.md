## SLO Tier Definitions

Before defining SLO targets, establish what each tier means for your organization. These tiers drive everything: hardware redundancy, DR investment, support response, and cost.

| Tier | Description | Business Impact if Down | Stakeholder | Target Availability |
|------|-------------|------------------------|-------------|---------------------|
| **Tier 0** | Mission critical — real-time revenue, safety, or regulatory obligation | Immediate revenue loss or regulatory breach | C-level notification | 99.99% |
| **Tier 1** | Business critical — core platform, customer-facing | Significant service degradation, customer impact | VP-level notification | 99.9% |
| **Tier 2** | Important — internal platform, indirect customer impact | Operations slowed, internal SLA risk | Director-level notification | 99.5% |
| **Tier 3** | Standard — dev, staging, analytics | Team productivity impact | Team lead notification | 99.0% |
| **Tier 4** | Best effort — dev/test, experiments | Individual dev impact | Self-managed | N/A (best effort) |

**Assign a tier to every VM.** VMs without an assigned tier default to Tier 3.

Performance SLOs define the acceptable range of VM resource metrics during normal operations. Breaches indicate either resource contention or application issues.

**CPU Ready Time SLO rationale**: CPU ready time >3% indicates scheduling contention on the ESXi host. For a database VM on the notification platform, >3% CPU ready translates directly to query latency spikes and potential timeout errors downstream.

## Section 3: Recovery SLO (RTO and RPO)

**RTO (Recovery Time Objective)**: Maximum time from failure detection to service restoration.
**RPO (Recovery Point Objective)**: Maximum data loss measured as time (age of last recoverable state).

| VM Name / Workload | Tier | RTO Target | RPO Target | DR Method | Replication Frequency | Failover Test Frequency |
|-------------------|------|------------|------------|-----------|----------------------|------------------------|
| tmobile-notif-db-01 | Tier 1 | 1 hour | 15 minutes | SRM + vSphere Replication | Continuous (15-min RPO) | Quarterly non-disruptive |
| [FILL IN] | | | | | | |

**DR method options**:
- **Stretched cluster**: RPO=0, RTO<5min. Very high cost. Tier 0 only.
- **SRM + vSphere Replication**: RPO 5min-4hr, RTO 30min-4hr. Mid-high cost. Tier 1.
- **SRM + daily snapshot**: RPO 24hr, RTO 1-4hr. Medium cost. Tier 2.
- **Backup-only (Veeam/Commvault)**: RPO 24hr, RTO 4-24hr. Low cost. Tier 3+.
- **No DR**: RPO = data loss possible, RTO = rebuild from scratch. Dev/test only.

## Section 4: Error Budget Calculation

**Error budget** = (1 − SLO target) × measurement period. It's the amount of downtime/error you're allowed to have while still meeting your SLO commitment.

| VM Name / Workload | SLO Target | Error Budget (monthly) | Budget Remaining (MTD) | Budget Consumed (MTD) | Status |
|-------------------|-----------|----------------------|----------------------|----------------------|--------|
| tmobile-notif-db-01 | 99.9% | 43.8 min/month | [= 43.8 − consumed] | [fill from monitoring] | [Green/Yellow/Red] |
| [FILL IN] | | | | | |

**Status thresholds**:
- **Green**: <50% of error budget consumed
- **Yellow**: 50-80% of error budget consumed — increase monitoring, review incidents
- **Red**: >80% consumed — freeze changes, focus on reliability, review SLO tier assignment

**Error budget policy**: When a Tier 1 VM's error budget reaches Red status, the on-call team should:
1. Halt non-emergency changes to that VM and its host
2. Schedule a post-incident review if not already done
3. Evaluate whether the SLO target is realistic or if it needs revision

Map your SLO targets to specific alert thresholds in your monitoring tool. This section is implementation-ready — copy these thresholds into vCenter alarms, Splunk alerts, or Datadog monitors.

| Metric | Warning Threshold | Critical Threshold | Alert Destination | SLO Impact |

## Section 6: SLO Review and Governance

**Review cycle**: Quarterly SLO review, or immediately after any SLO breach.

**SLO breach response process**:
1. **Detect**: alert fires (automated)
2. **Acknowledge**: on-call SRE within [X] minutes
3. **Incident**: open incident if duration >15 min (for Tier 1+)
4. **RCA**: root cause analysis within 5 business days of breach
5. **SLO review**: assess whether SLO target needs adjustment or infrastructure needs improvement

**SLO ownership**:
- Infrastructure team owns: availability, performance metrics, recovery time
- Application team owns: validating RTO met from their perspective, RPO adequacy
- Joint review: quarterly, with sign-off from both infrastructure and application owner
