# Hypervisor Migration Playbook: VMware to Nutanix or Proxmox

## Purpose

This playbook covers the decision gate and execution of a VMware-to-alternative hypervisor migration. It assumes the cost assessment (see `vmware-broadcom-cost-assessment-playbook.md`) has been completed, a TCO model has been built for at least two alternatives, and leadership has approved the migration direction in writing.

Do not start this playbook until you have:
- An approved migration target (Nutanix AHV or Proxmox)
- A written business case that has been signed by CTO or equivalent
- Hardware budget committed (Nutanix nodes ordered, or Proxmox-compatible hardware in procurement)
- At least one team member who has completed hands-on training on the target platform

This playbook is not a proof-of-concept guide. It is an execution guide for a production migration. Proof-of-concept work should happen before this playbook starts, during the business case phase.

---

## Phase 1: Workload Discovery and Tiering (Weeks 1–4)

**Objective**: Know every VM, its owner, its criticality, and its migration readiness before any migration begins. Moving fast without this foundation is how migrations fail at week 10 when you discover a critical database VM has no owner and no one knows what it does.

### Step 1.1 — VM Inventory Export

Use PowerCLI to export a complete, structured VM inventory from all vCenter instances. Run this in PowerShell with VMware PowerCLI installed:

```powershell
# Connect to vCenter
Connect-VIServer -Server vcenter.yourdomain.com

# Export full VM inventory
Get-VM | Select-Object `
  Name, `
  PowerState, `
  NumCpu, `
  MemoryGB, `
  @{N="DiskGB"; E={(Get-HardDisk -VM $_ | Measure-Object -Property CapacityGB -Sum).Sum}}, `
  @{N="GuestOS"; E={$_.Guest.OSFullName}}, `
  @{N="VMwareToolsStatus"; E={$_.ExtensionData.Guest.ToolsStatus}}, `
  @{N="Cluster"; E={$_.VMHost.Parent.Name}}, `
  @{N="Host"; E={$_.VMHost.Name}}, `
  @{N="Datastore"; E={($_ | Get-Datastore).Name -join ","}}, `
  @{N="Network"; E={($_ | Get-NetworkAdapter).NetworkName -join ","}}, `
  @{N="LastPoweredOn"; E={$_.ExtensionData.Runtime.BootTime}} | `
Export-Csv -Path C:\migration\vm_inventory_$(Get-Date -Format yyyyMMdd).csv -NoTypeInformation
```

Run against every vCenter instance in your environment. Merge the CSV outputs into a single master inventory. Deduplicate by VM name — linked vCenter instances may report the same VM from multiple vCenters.

**Key fields and why they matter for migration**:

| Field | Why It Matters |
|-------|---------------|
| GuestOS | Legacy OS (Win2008, RHEL 5, RHEL 6) may not have drivers for target hypervisor |
| VMwareToolsStatus | Tools must be current for live migration; outdated tools = manual driver injection |
| DiskGB | Large disks (>2TB) require extended migration windows and datastore pre-staging |
| Network | Port groups that don't exist on the target cluster will drop connectivity at cutover |
| LastPoweredOn | VMs off for 90+ days = decommission candidates; confirm before migrating |

### Step 1.2 — Tiering Assignment

Assign every VM a migration tier before starting any migration work. Tier assignment should be done by the VM owner with SRE validation, not by SRE alone.

| Tier | Criteria | Migration Approach | Acceptable Downtime |
|------|----------|--------------------|---------------------|
| **Tier A** | Stateless, non-critical, no data persistence | Cold migration or snapshot export | Hours |
| **Tier B** | Stateful but not real-time; dev/staging | Live migration with maintenance window | Minutes to 1 hour |
| **Tier C** | Production stateful (databases, message queues, caches) | Live migration, zero-downtime required | Seconds (failover) |
| **Tier D** | Legacy OS, compliance hold, specialized hardware dependency | Keep on VMware or decommission; do not migrate | N/A |
| **Tier E** | Powered off >90 days, no identified owner | Snapshot, archive, then decommission | N/A |

**Tier D handling**: Create a separate JIRA epic or project tracker for Tier D VMs. These require individual remediation plans. A Windows Server 2008 VM cannot simply be migrated to Nutanix AHV — the guest OS may not support the virtio drivers. Options: in-place OS upgrade before migration, containerization of the application, or decommission with application replacement. None of these are fast.

### Step 1.3 — Owner Identification

Establish a 2-week owner identification window immediately after VM inventory is published. Communicate clearly to all engineering teams: "Any VM without a confirmed owner in two weeks will be treated as Tier E (decommission candidate)."

Track owner identification in a shared spreadsheet or CMDB. Required fields:
- VM name
- Owner name and email
- Team / cost center
- Criticality tier (A/B/C/D/E) — proposed by SRE, confirmed by owner
- Migration contact (may differ from owner)
- Special requirements (maintenance windows, compliance requirements, vendor approval needed for OS change)

Escalate unowned VMs to engineering leadership at the end of Week 2. Do not proceed with migration waves until >95% of powered-on VMs have confirmed owners.

### Step 1.4 — Technical Readiness Checks

For each VM, run automated pre-migration checks:

```powershell
# Check VMware Tools version and status
Get-VM | Get-View | Select-Object `
  @{N="VMName"; E={$_.Name}}, `
  @{N="ToolsVersion"; E={$_.Guest.ToolsVersion}}, `
  @{N="ToolsStatus"; E={$_.Guest.ToolsStatus}}, `
  @{N="ToolsRunningStatus"; E={$_.Guest.ToolsRunningStatus}} | `
Where-Object {$_.ToolsStatus -ne "toolsOk"} | `
Export-Csv -Path C:\migration\tools_issues.csv -NoTypeInformation
```

VMs with tools status `toolsNotInstalled` or `toolsOld` require manual intervention before live migration. Plan 2–4 hours per affected VM for tools update, reboot, and validation.

## Phase 2: Target Platform Selection Decision Gate

Before any migration execution begins, produce a written decision record. This gate protects you from scope creep mid-migration and provides clear authorization for hardware procurement.

### Decision Gate Criteria

| Criteria | Nutanix AHV | Proxmox VE | Verdict Weight |
|----------|------------|------------|----------------|
| **Team expertise ramp time** | 4–8 weeks (VMware-adjacent concepts) | 6–12 weeks (Linux/KVM expertise required) | High |
| **Minimum viable cluster** | 3 nodes | 2 nodes (no quorum requirement) | Medium |
| **Migration tooling** | Nutanix Move (free, GUI-driven) | virt-v2v or manual OVF (CLI, complex) | High |
| **HA capability** | Native in Prism, automatic | Corosync + Pacemaker (manual configuration) | High |
| **Storage architecture** | Nutanix HCI (converged compute/storage) | Separate storage or local disk (not HCI) | Medium |
| **GUI management quality** | Prism Element + Prism Central (polished) | Proxmox web UI (functional, less polished) | Low |
| **Commercial support SLA** | Enterprise support available (Nutanix) | Community + Proxmox Commercial Support | High |
| **License cost** | Nutanix licensing (significant but < Broadcom) | Open source (support contract optional) | High |
| **5-year exit optionality** | Vendor lock-in to Nutanix AHV | Standard KVM; fully portable | Medium |

**Recommendation frame**: Nutanix is the right choice if your team has limited Linux/KVM expertise, you want commercial support SLA, and you are willing to accept Nutanix licensing costs (which are still typically lower than Broadcom's post-acquisition pricing). Proxmox is the right choice if your team has strong Linux skills, you are willing to invest in HA configuration effort, and cost elimination (not just cost reduction) is the primary goal.

### Go/No-Go Gate Requirements

Written documentation confirming:
- [ ] Target platform selected and rationale documented
- [ ] Hardware specification finalized (nodes, networking, storage)
- [ ] Hardware budget approved and purchase order issued
- [ ] Team training plan completed (at least 2 engineers trained on target platform)
- [ ] Migration tooling validated in lab (Nutanix Move or virt-v2v tested against sample VMs)
- [ ] CTO or Director of Engineering has signed the decision record

Do not begin pilot cluster setup without all six items confirmed.

## Phase 3: Pilot Cluster Setup (Weeks 4–8)

**Never migrate your production fleet before running a formal pilot. A pilot that fails saves you from a production migration that fails — those are not equivalent failures.**

### Pilot Scope and Selection Criteria

Pilot scope: 5–10 non-critical VMs. Selection criteria for pilot VMs:
- Not in any on-call escalation path
- No customer-facing traffic
- Development or staging environment
- Application owner is willing to participate (they will run acceptance tests)
- Workload is representative of the production population (mix of OS types, disk sizes, network requirements)

Do not pilot only the simplest VMs. If your production environment has 30% Windows Server 2019 and 40% RHEL 8, your pilot should reflect that ratio. A pilot that only migrates Ubuntu 22.04 VMs does not validate your production migration path.

### Pilot Cluster Requirements

- Minimum 3 nodes (for HA quorum on Nutanix; Proxmox can run 2 but test with 3)
- Same hardware generation as your production target — do not pilot on hardware you plan to retire
- Connected to production networking infrastructure: same VLANs, DNS servers, NTP servers, Active Directory domain (if applicable). You want to find network integration problems during the pilot, not production migration.
- Monitoring agents deployed on target cluster before the first VM arrives (Splunk UF, Prometheus node exporter, or equivalent)

### Pilot Success Criteria

Define these in writing before you start the pilot. If you define success criteria after you see the results, they will be influenced by the results.

| Success Criterion | Pass Threshold | Measurement Method |
|------------------|---------------|-------------------|
| All pilot VMs migrated without manual intervention | 100% | Nutanix Move job log review |
| Application owner functional testing passed | 100% | Signed-off test result from each owner |
| HA failover test passed | Automatic restart within 3 minutes | Simulate node failure, measure restart time |
| Network connectivity identical to VMware | Same IP, DNS resolution, throughput within 10% | ping, nslookup, iperf3 |
| Storage performance within 20% of VMware baseline | <20% degradation | FIO benchmark on VMware, repeat on Nutanix |
| Migration tooling requires no manual step per VM | 0 manual interventions for Tier A/B | Review migration job logs |

If any criterion fails, do not proceed to production waves. Diagnose, remediate, and re-run the pilot. A failed pilot criterion that you ignore will become a production migration incident.

## Phase 4: Migration Tooling Selection

### Tool Comparison

| Tool | Source | Target | Migration Type | Cost | Best For |
|------|--------|--------|---------------|------|----------|
| **Nutanix Move** | VMware ESXi | Nutanix AHV | Warm (online) migration | Free | VMware → Nutanix; handles disk, network, driver conversion automatically |
| **virt-v2v** | VMware ESXi, Xen | KVM/Proxmox, RHEL/oVirt | Cold or warm | Free (open source) | VMware → Proxmox; requires Linux expertise; driver injection manual |
| **Zerto** | VMware | Nutanix, Azure, AWS | Journal-based continuous replication | Licensed | Near-zero RPO migrations; disaster recovery; financial services workloads |
| **CloudEndure / AWS MGN** | Any source | AWS EC2 | Block-level replication | Per-server fee | VMware → AWS; when target is cloud-native |
| **Manual OVF export/import** | Any VMware | Any hypervisor | Cold only | Free | Small environments, Tier E cleanup; not suitable for production scale |

### Nutanix Move — How It Works

Nutanix Move is the recommended tool for VMware-to-AHV migration. It runs as an appliance VM deployed on the Nutanix cluster.

```
Architecture:

[VMware ESXi] ←—snapshot replication—→ [Nutanix Move Appliance] ←—write—→ [Nutanix AHV]
     |                                         |
  Source VMs                          Manages disk conversion,
                                      driver injection, network mapping,
                                      cutover orchestration
```

Migration sequence for each VM:
1. Move connects to vCenter and ESXi via VMware API
2. Move reads VM disk layout, network config, and snapshot state
3. Initial seed copy: full disk copy from VMware datastore to Nutanix storage
4. Delta sync: continuous block-level sync (online replication — VM stays running on VMware)
5. Cutover window: quiesce application, final delta sync (seconds to minutes depending on write rate)
6. Power on on Nutanix AHV with auto-injected virtio drivers
7. Move powers off VMware source VM

**Critical Nutanix Move pre-checks**:
- VMware Tools must be installed and running on source VM
- UDP port 2222 must be open from Move appliance to ESXi hosts
- HTTPS port 443 must be open from Move appliance to vCenter
- NFS mount accessible from Move appliance to Nutanix storage container

### virt-v2v for Proxmox — How It Works

virt-v2v is a command-line tool that converts VMware VMs to KVM-compatible format. It runs on a Linux host (RHEL, Fedora, or Ubuntu).

```bash
# Export a VMware VM to a local directory (cold migration — VM must be off)
virt-v2v -i vmx /path/to/vm.vmx -o local -os /path/to/output

# Export from vCenter (requires VMware credentials)
virt-v2v -ic vpx://vcenter.domain.com/Datacenter/esxi-host \
  -it vddk \
  -io vddk-libdir=/path/to/vmware-vddk \
  -io vddk-thumbprint=<esxi-thumbprint> \
  "VM Name" \
  -o local -os /output/directory

# Import converted disk to Proxmox
qm create 101 --name migrated-vm --memory 4096 --net0 virtio,bridge=vmbr0
qm importdisk 101 /output/directory/vm-disk.qcow2 local-lvm
qm set 101 --scsihw virtio-scsi-pci --scsi0 local-lvm:vm-101-disk-0
qm set 101 --boot c --bootdisk scsi0
```

**virt-v2v caveats**:
- Requires VMware VDDK (Virtual Disk Development Kit) for online migration; VDDK download requires VMware developer account
- Windows VMs require virt-v2v to inject VirtIO drivers automatically; verify post-migration that drivers loaded correctly
- virt-v2v does not handle network port group remapping — you must manually configure Proxmox network adapters post-migration
- Large VMs (>2TB disk) require extended transfer time; plan for 2–8 hours of transfer per TB over 10Gbps

## Phase 5: Wave Planning

Wave planning is the discipline that separates controlled migrations from chaotic ones. Migrations that attempt to "do all VMs" without structured waves consistently run into undiscovered dependencies, rollback confusion, and team exhaustion.

### Wave Sequencing Rules

1. **Tier E first (before any migration)**: Decommission all confirmed Tier E VMs before Wave 1 starts. This reduces your VMware footprint, reduces noise in the inventory, and reclaims storage.
2. **Tier A in Wave 1**: Stateless services — web servers, API gateways, job runners, non-critical batch VMs.
3. **Tier B in Waves 2–N**: Stateful non-critical — development databases, staging environments, QA infrastructure.
4. **Tier C last**: Production stateful workloads (databases, RabbitMQ/Kafka, Redis, core API backends). Never migrate Tier C until you have at least 2 successful waves of Tier A and Tier B behind you.
5. **Tier D never** (unless a dedicated remediation plan exists and has been executed).
6. **Never mix Tier C and Tier A in the same wave** — rollback decisions become ambiguous.

### Wave Size Guidelines

| Environment Size | VMs per Wave | Rationale |
|-----------------|-------------|-----------|
| <100 VMs total | 10–20 VMs | Team can validate all VMs within 48 hours |
| 100–500 VMs total | 20–50 VMs | Standard wave size; balance progress with control |
| 500+ VMs total | 50–100 VMs | Only if migration tooling is highly automated and rollback is tested |

Limit wave size to what your team can fully validate within 48 hours of cutover. A wave you cannot fully validate is a wave you cannot confidently close — and unclosed waves create ambiguity in the source environment (is the VMware VM a rollback target or dead weight?).

### Pre-Wave Checklist (Run Before Every Wave)

```
□ Wave VM list finalized and owner-confirmed
□ Maintenance window communicated to all VM owners (24-hour minimum notice for Tier B, 72-hour for Tier C)
□ Source VMware snapshots taken (rollback point)
□ Target cluster health verified (all nodes in normal state, storage utilization <70%)
□ Monitoring dashboards open and baseline captured for all wave VMs
□ Rollback criteria defined in writing for this wave
□ On-call engineer assigned for 48-hour hypercare post-wave
□ DNS TTL lowered to 60 seconds for any DNS-based service discovery used by wave VMs
```

### Rollback Criteria — Define Before Every Wave

Write down the conditions that trigger a full wave rollback before the migration window opens. Once you are mid-migration, rollback decisions made under pressure are wrong decisions.

Mandatory rollback triggers:
- Any Tier C VM fails post-migration health check (within the 48-hour hypercare window)
- Network connectivity failure affecting production traffic (not dev/staging)
- Data integrity check fails (file count mismatch, checksum error, database replication lag)
- Application owner rejects post-migration functional test result

Optional rollback triggers (judgment call with application owner and SRE lead):
- Performance degradation >30% versus VMware baseline
- Non-critical monitoring alert that owner cannot explain

## Phase 6: Cutover Procedure

This is the per-VM procedure for each migration within a wave. Execute this in sequence. Do not skip steps.

```
PRE-MIGRATION (24 hours before window):
  1. Notify application owner with maintenance window time and expected duration
  2. Confirm monitoring is active (both VMware and Nutanix/Proxmox dashboards visible)
  3. Lower DNS TTL for any DNS-registered services on this VM (set to 60 seconds)
  4. Take VMware snapshot (label: "pre-migration-[wave]-[date]")
  5. If using Nutanix Move: create migration plan for this VM; initiate seed copy
  6. Confirm Nutanix/Proxmox cluster storage has sufficient free space

MIGRATION WINDOW:
  7. Notify application owner: window is open
  8. Confirm Nutanix Move seed sync is complete (or initiate for cold migration)
  9. Quiesce application (stop writes): coordinate with app owner or run pre-migration script
  10. Initiate final delta sync (Nutanix Move: "Cutover" button)
  11. Wait for final sync completion (monitor Move status; Tier A: <5 min; Tier C: <2 min)
  12. Power on VM on Nutanix AHV (or Proxmox)
  13. Validate: ping, SSH or RDP, application health endpoint, log output
  14. Application owner confirms application is functional
  15. Re-enable DNS record pointing to new host (TTL propagates in 60 seconds)
  16. Notify application owner: migration complete; hypercare window begins

POST-MIGRATION (48-hour hypercare):
  17. Monitor VM performance on target platform for 48 hours
  18. Compare performance metrics to VMware baseline (CPU, memory, disk I/O, network)
  19. Application owner confirms no degradation at end of 48-hour window
  20. Delete VMware snapshot after 48-hour clean operation (not the VM — only the snapshot)
  21. Power off VMware source VM (do not delete)
  22. Mark VM as "migrated" in inventory tracking spreadsheet

POST-MIGRATION HOLD:
  23. Retain powered-off VMware source VM for 30 days
  24. After 30-day hold: delete source VMware VM
  25. Reclaim storage on VMware datastore
```

**On the 30-day hold**: this is not caution theater. Applications frequently have monthly batch jobs, monthly reports, or month-end processes that only run once per month. A VM that looks stable for 48 hours may fail a month-end job. The 30-day hold means you have not lost the rollback option when that monthly job runs for the first time post-migration.

## Phase 7: Post-Migration Validation

### Per-VM Validation Checklist

Execute this checklist for every VM after migration, before marking it complete:

```
OS and System:
  □ VM powers on and OS boots cleanly (no kernel panics, boot loops)
  □ Guest OS hostname matches expected value
  □ VMware Tools (source) replaced by target platform agent (Nutanix NTNX agent or QEMU guest agent)
  □ No unexpected kernel errors in /var/log/messages or Windows Event Viewer

Network:
  □ IP address matches expected assignment (static or same DHCP lease)
  □ Default gateway reachable (ping gateway)
  □ DNS resolution works: both forward (hostname → IP) and reverse (IP → hostname)
  □ Network throughput validated with iperf3 (compare to VMware baseline within 10%)
  □ Any firewall rules or NSX microsegmentation policies re-applied on target platform

Storage:
  □ All expected disks/volumes present and mounted
  □ File system integrity check passed (no fsck errors on Linux; chkdsk on Windows)
  □ Disk performance benchmark within 20% of VMware baseline (use fio for Linux)
  □ No data loss: file count and spot-check checksums match pre-migration snapshot

Application:
  □ Application service starts and reaches healthy state
  □ Application health endpoint returns 200 OK (or equivalent)
  □ Application logs show no unexpected errors in first 30 minutes post-start
  □ Application owner has run functional acceptance test and signed off
  □ Application metrics visible in monitoring platform (Splunk, Prometheus, etc.)

Platform Management:
  □ VM visible and managed in Prism Central (Nutanix) or Proxmox web UI
  □ HA policy applied correctly (VM will restart on node failure)
  □ Backup policy applied (if applicable)
  □ Resource reservations set appropriately (CPU, memory)
```

### Wave Closeout

A wave is not closed until all VMs in the wave have passed the per-VM validation checklist AND a wave summary document has been filed. The wave summary document captures:
- Wave number and date
- VMs migrated (list)
- VMs that failed or rolled back (list with reason)
- Issues encountered and resolution
- 48-hour performance comparison summary
- Owner sign-off confirmation
- Next wave readiness: yes/no

Wave closeout documentation becomes your migration audit trail. When the CFO asks "how is the migration going?" you can show wave summaries rather than a verbal status update.
