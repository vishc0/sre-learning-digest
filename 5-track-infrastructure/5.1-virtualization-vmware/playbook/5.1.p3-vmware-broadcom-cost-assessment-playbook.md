# VMware/Broadcom Cost Assessment Playbook

## Purpose and When to Run This Playbook

Run this playbook when any of the following conditions are true:
- Your VMware renewal is within 12 months
- You have received a renewal quote that is significantly higher than the prior year
- Leadership has asked for a formal VMware cost exposure assessment
- You are preparing a business case for platform migration

**Expected duration**: 3–4 weeks for a mid-size environment (50–500 ESXi hosts). Smaller environments (under 50 hosts) can complete in 1–2 weeks. Larger environments (500+ hosts) should plan for 6–8 weeks and assign a dedicated program manager.

**Who runs this**: SRE Manager or Director of Platform Engineering, with support from procurement, legal, and your VMware Technical Account Manager (TAM). Do not run this in isolation — procurement needs to be involved before any negotiation begins.

**Output artifacts**:
- License inventory document (Phase 1)
- Contract summary with renewal cliff dates (Phase 2)
- Bundle mapping and gap analysis (Phase 3)
- 3-year TCO delta model (Phase 4)
- Executive one-pager for CFO/CTO briefing (Phase 5)
- Negotiation position document (Phase 6)

## Phase 1: License Inventory (Week 1)

**Objective**: Know exactly what VMware products you have deployed, at what version, with what license keys, and at what utilization. You cannot negotiate what you have not measured.

### Step 1.1 — Pull from vCenter

Log into vCenter Server → Administration → Licenses. The Licenses view shows every license key assigned to the vCenter instance.

Export the license key list. For each entry, capture:
- Product name and edition (e.g., vSphere Enterprise Plus, NSX Enterprise Plus)
- License key (obfuscate for any shared documents)
- Expiration date
- Capacity (e.g., 32 CPUs licensed)
- Current usage count (e.g., 28 CPUs in use)

**What to look for**: licenses where usage count significantly exceeds capacity (overcommitted — a compliance risk) and licenses where usage count is significantly below capacity (underutilized — a negotiation opportunity).

### Step 1.2 — ESXi Host Audit

In vCenter → Hosts and Clusters → select each cluster → Hosts tab. For each host, record:

| Field | Where to Find It |
|-------|-----------------|
| Host name | Summary tab → host FQDN |
| ESXi version | Summary tab → VMware ESXi version string |
| CPU socket count | Summary tab → Processors |
| CPU core count per socket | Summary tab → or `esxcli hardware cpu global get` via SSH |
| License type applied | Summary tab → License field |
| Host state | Normal / Maintenance / Disconnected |

**Critical**: Broadcom's new licensing model bills per **core**, not per socket. If your prior-year contract was socket-based, the core count is the number that determines your new price. A 2-socket host with 32 cores per socket = 64 billable cores per host.

Repeat this for every vCenter instance in your environment. Large organizations running linked mode vCenter may have multiple independent SSO domains — audit all of them.

### Step 1.3 — NSX Inventory

If NSX is deployed: log into NSX Manager → System → Licenses. Record:
- NSX edition (NSX Data Center Advanced, Enterprise Plus, etc.)
- Number of **transport nodes** (ESXi hosts with NSX installed = one transport node each)
- Number of **edge nodes** (NSX Edge VMs providing routing/load balancing)
- Deployment type: NSX-T (new) or NSX-V (legacy, end-of-life)

**Why this matters for Broadcom bundles**: NSX Enterprise Plus is only included in VMware Cloud Foundation (VCF). If you have NSX, you are likely being quoted VCF pricing, which is the highest-cost bundle. If you are using NSX features only available in Enterprise Plus, confirm this — Basic NSX is included in VMware vSphere Foundation (VVF).

### Step 1.4 — vSAN Inventory

If vSAN is deployed: vCenter → vSAN → vSAN cluster → Summary.

Record for each vSAN cluster:
- Cluster name
- Number of hosts contributing storage
- Total raw capacity (TB)
- Total consumed capacity (TB)
- Deduplication and compression ratio (if enabled)
- vSAN edition (Standard, Advanced, Enterprise, Enterprise Plus)

**Negotiation note**: vSAN storage capacity is often dramatically oversized from initial deployment. If your actual consumed capacity is 40% of raw capacity, you have a right-sizing argument.

### Step 1.5 — Tanzu Inventory

If Tanzu is deployed: record:
- Supervisor Cluster count (vSphere with Tanzu namespaces)
- Tanzu Kubernetes Cluster (TKC) count
- TKG version (TKG 1.x vs. TKG 2.x)
- Whether you are using Tanzu Mission Control (TMC) — this is a separate subscription

### Step 1.6 — Additional Products to Audit

| Product | Where to Check | Notes |
|---------|---------------|-------|
| vRealize Operations (Aria Operations) | vROps → Administration → Licensing | Included in some VCF tiers |
| vRealize Log Insight (Aria Operations for Logs) | vRLI → Administration → Licensing | Often not counted in primary inventory |
| Site Recovery Manager (SRM) | SRM → Administration → Licensing | Not included in VVF or VCF; add-on |
| VMware HCX | HCX Manager → Administration | Often missed; add-on license |
| Carbon Black (Workspace ONE) | Separate console | If part of EUC bundle |

### Inventory Document: What to Capture

Consolidate findings into a single inventory document with these columns:

```
Product | Edition | License Key (last 8 chars) | Capacity Unit | 
Licensed Capacity | Used Capacity | Utilization % | Contract End Date | 
vCenter Instance | Notes
```

Store this document in a restricted access location. License keys and capacity data are sensitive — treat them as commercial confidential.

## Phase 2: Contract Review (Week 1–2)

**Objective**: Understand exactly when entitlements expire, what you paid historically, what is in the current contract, and what auto-renewal traps exist.

### Step 2.1 — Locate Current Contracts

Work with your procurement and legal teams. You need:
- The Master Software Agreement (MSA) with VMware/Broadcom
- All active order forms (purchase orders associated with the MSA)
- Any SaaS subscription agreements for VMware Cloud services (VMware Cloud on AWS, HCX, Aria)
- Historical invoices for the past 3 years (to establish prior-year baseline)

If your procurement team does not have these documents centrally, check with your legal team — MSAs are typically executed by legal and filed with executed contracts.

### Step 2.2 — Extract Renewal Cliff Data

For each entitlement or order form, record:

| Field | Description |
|-------|-------------|
| Contract end date | Your "renewal cliff" — the date after which you have no entitlement |
| Auto-renewal clause | Yes/No — if yes, how many days notice required to cancel? |
| Notice window deadline | The date by which you must notify Broadcom of non-renewal |
| Current annual cost | What you paid in the most recent renewal year |
| Support tier | Basic / Production / Premier / Mission Critical |
| Assigned TAM | Name and contact |

**Red flag — Auto-renewal clauses**: Broadcom's contracts increasingly include 60–90 day auto-renewal notice requirements. If you miss the notice window, you are auto-renewed at new pricing without a negotiation opportunity. The day you complete this playbook, set a calendar reminder for 90 days before your renewal cliff. Put it on your calendar and your manager's calendar.

**Red flag — Perpetual license conversion**: If your prior contract included perpetual licenses, confirm in writing how Broadcom is handling the conversion. Some perpetual customers are being moved to subscription without clear written terms for what happens to the perpetual entitlement if they cancel.

### Step 2.3 — Questions to Ask Your VMware TAM

Prepare these questions in writing before any TAM meeting. Send them in advance and request written responses — verbal answers during calls are not enforceable.

1. "What is our current entitlement mapped to the new bundle structure? Specifically, are we being quoted VVF or VCF, and what determines that?"

2. "What is the per-core price being applied to our renewal quote, and how was the core count calculated? Please provide the host-by-host breakdown."

3. "Is there a loyalty discount or migration credit available for customers who were on perpetual licenses prior to the Broadcom acquisition?"

4. "What is the minimum committed term available, and what price protection commitment comes with a 3-year versus 1-year commitment?"

5. "Are there workload exclusions or carve-outs available? For example, development/test environments billed differently than production?"

6. "What is Broadcom's commitment to the current bundle structure over the next 3 years? Is there a written price lock available?"

7. "What professional services credits are available to support migration between VMware products (e.g., from vSAN to external storage, from NSX to alternative networking)?"

Document every answer. If the TAM says they will follow up, mark it as open and track it weekly.

## Phase 3: Broadcom Bundle Mapping (Week 2)

**Objective**: Map your current installed products to the new Broadcom bundle structure and identify where you are being forced to pay for features you do not use.

### The New Bundle Structure (as of 2024)

Broadcom simplified (from their perspective) the VMware product catalog to three primary offerings:

| Bundle | What's Included | Billing Unit |
|--------|----------------|--------------|
| **VMware vSphere Foundation (VVF)** | ESXi, vCenter, vSAN, basic NSX | Per core, per year |
| **VMware Cloud Foundation (VCF)** | Everything in VVF + NSX Enterprise Plus + Aria Suite + Tanzu | Per core, per year |
| **VMware vSphere Essentials Plus** | ESXi + vCenter, small business only | Per host, per year |

The vast majority of enterprise customers are being quoted either VVF or VCF. VCF is significantly more expensive. The trigger for VCF (versus VVF) is NSX Enterprise Plus features or Tanzu.

### Mapping Exercise

For each product in your inventory, map it to the bundle it falls into:

| Your Current Product | VVF | VCF | Standalone/Add-On | You Use It? |
|---------------------|-----|-----|-------------------|-------------|
| ESXi Enterprise Plus | Yes | Yes | — | [Audit] |
| vCenter Standard | Yes | Yes | — | [Audit] |
| vSAN (Standard/Advanced) | Yes | Yes | — | [Audit] |
| NSX Data Center Advanced | Partial | Yes | — | [Audit] |
| NSX Enterprise Plus | No | Yes | — | [Audit] |
| vRealize Operations (Aria Ops) | No | Yes | Aria standalone | [Audit] |
| vRealize Log Insight (Aria Logs) | No | Yes | Aria standalone | [Audit] |
| vRealize Automation (Aria Automation) | No | VCF only | Not available standalone | [Audit] |
| Site Recovery Manager (SRM) | No | No | Add-on required | [Audit] |
| VMware HCX | No | No | Add-on required | [Audit] |
| Tanzu Kubernetes Grid (TKG) | No | Yes (partial) | Tanzu Platform (separate) | [Audit] |

### Where to Push Back on Bundle Assignment

**If you are quoted VCF but do not use NSX Enterprise Plus features**: push back. Basic NSX (L2 switching, basic routing) is VVF-included. If you are using NSX primarily for microsegmentation without advanced features like NSX Intelligence or NSX Advanced Load Balancer, confirm with your NSX admin which features are actually in use. A downgrade from VCF to VVF on a 500-core environment can represent $300–500K in annual savings.

**If you are quoted VCF because of Tanzu but have minimal TKG usage**: quantify Tanzu usage. If you have 2 TKG clusters running development workloads, the case for paying VCF pricing for your entire estate is weak. Consider whether those workloads could move to a cloud-managed Kubernetes service (EKS, AKS) at lower total cost.

**If SRM is in your entitlement**: SRM is no longer included in any bundle. It is an add-on. Verify that SRM is actively used for DR testing (quarterly at minimum). If your DR strategy has evolved beyond SRM (for example, to Zerto, Veeam, or native cloud DR), you should not be renewing SRM.

## Phase 4: Price Delta Calculation (Week 2–3)

**Objective**: Produce a defensible financial model that leadership can review, challenge, and approve. The output is a specific dollar figure with documented assumptions — not a range, not a ballpark.

### The Calculation Formula

```
New Annual Cost = (Total core count × per-core price × bundle multiplier)
                + SRM add-on (if applicable, per-VM fee)
                + HCX add-on (if applicable, per-site fee)
                + Premier Support premium (if applicable)
                + Tanzu Platform fee (if separate from VCF)

Prior Annual Cost = Sum of all VMware invoices in most recent renewal year

Annual Delta = New Annual Cost - Prior Annual Cost

3-Year Exposure = Annual Delta × 3
  (Conservative: assumes no further price increases, which is unlikely)

3-Year Exposure (with 10% annual increase) = 
  Delta Year 1 + (Delta × 1.1) + (Delta × 1.21)
```

### Price Benchmarks

These are reference ranges only. Actual prices depend on volume, geography, negotiation, and whether you have competitive alternatives. Never present these as definitive — get your actual quote from Broadcom.

| Bundle | Reference Range (per core/year) | Notes |
|--------|--------------------------------|-------|
| VVF | $80–$150/core/year | 16-core minimum per physical CPU |
| VCF | $150–$250/core/year | 16-core minimum per physical CPU |
| Uplift from VVF to VCF | ~70–100% premium | Significant variance based on volume |

**16-core minimum**: Broadcom applies a 16-core minimum per physical CPU socket, regardless of actual core count. A host with 2 physical CPUs at 12 cores each = 24 actual cores, but you are billed for 32 (2 × 16 minimum).

### Building the Model

Use a spreadsheet with one row per ESXi host. Columns:
- Host name
- Physical CPU socket count
- Actual cores per socket
- Billable cores per socket (max of actual cores and 16)
- Total billable cores for host (sockets × billable cores)
- Bundle assigned (VVF or VCF)
- Per-core price (from quote)
- Annual cost for host

Sum the annual cost column. Add add-on costs. This is your new annual cost.

Compare to your most recent VMware invoice total. The difference is your delta.

### Documenting Assumptions

Present the model with an explicit assumptions section:

```
Assumptions:
- Core count sourced from vCenter inventory export dated [DATE]
- Per-core price sourced from Broadcom renewal quote dated [DATE]  
- 16-core minimum applied per physical CPU socket per Broadcom licensing policy
- All hosts classified as [VVF/VCF] based on NSX Enterprise Plus usage audit
- SRM not included (retired from use as of [DATE])
- Prior-year cost sourced from PO [number] dated [DATE]
- No volume discount assumed beyond current quote (may be negotiable)
```

Documenting assumptions lets leadership and procurement validate your inputs without questioning your logic. If an assumption is wrong, they correct the assumption, not the model.

## Phase 5: Stakeholder Communication

**How to structure the executive briefing**:

The executive briefing should be a single page. Executives with budget authority do not read 20-page analysis documents before making a decision — they read the one-pager and ask questions.

### One-Page Executive Summary Format

```
SUBJECT: VMware/Broadcom Renewal Exposure Assessment

PREPARED BY: [Your name]
DATE: [Date]
CLASSIFICATION: Commercial Confidential

FINANCIAL SUMMARY
Current annual VMware spend:          $[X]
Projected renewal under Broadcom:     $[Y]  
Annual increase:                      $[Z] ([%] increase)
3-year exposure (at current pricing): $[3Z]
Renewal cliff date:                   [Date]
Auto-renewal notice deadline:         [Date — if applicable]

OPTIONS UNDER EVALUATION
1. Renew with negotiation         Est. 3-yr cost: $[A]    Timeline: 60 days
2. Migrate to Nutanix (on-prem)   Est. 3-yr cost: $[B]    Timeline: 18 months
3. Migrate to AWS                 Est. 3-yr cost: $[C]    Timeline: 24-36 months

DECISION NEEDED
A direction decision is needed by [DATE] to allow sufficient lead time 
before the auto-renewal notice deadline.

RECOMMENDATION
[One sentence. E.g.: "Renew for one year while executing Nutanix pilot 
to validate migration path before committing to a 3-year VMware term."]

NEXT STEPS
1. [Action] — [Owner] — [Due date]
2. [Action] — [Owner] — [Due date]
3. [Action] — [Owner] — [Due date]
```

**What not to include in the executive summary**: technical product names (NSX, vSAN, VCF), hypervisor comparisons, migration tooling details. Save those for the supporting appendix for the team that will execute.

**What to lead with**: the financial exposure and the decision deadline. Executives do not make technology decisions — they make capital allocation decisions under time pressure. Give them the number and the date.

## Phase 6: Negotiation Preparation

**Timing**: begin negotiation preparation as soon as the executive summary is approved. You need at least 60 days of active negotiation time before your renewal cliff.

### Building Your BATNA Document

**BATNA** (Best Alternative to a Negotiated Agreement) is the single most important pre-negotiation document. It answers: "If this negotiation fails, what is our fallback, and how credible is it?"

Your BATNA document should include:
- Nutanix quote: contact a Nutanix account representative and request a formal quote for your core count equivalent. Nutanix Move (migration tool) is free — include this in the cost model. Nutanix will price aggressively if Broadcom renewal is the stated competitive situation.
- Proxmox commercial support quote: if open-source is acceptable to your organization
- Cloud migration cost estimate: even a rough AWS Migration Evaluator output counts — it shows you have done the homework
- Internal migration capability assessment: "We have [X] engineers who have completed Nutanix training / AWS certification. We can execute migration in [Y] months."

The BATNA document is internal only. You share the existence of alternatives with Broadcom, not the document itself.

### Negotiation Red Flags to Watch For

| What Broadcom Does | What It Means | How to Respond |
|-------------------|---------------|----------------|
| Quotes list price | Standard opening; nobody pays list | "We need to understand the volume discount structure. Please provide the discounted pricing framework." |
| Says discounts require single-year commitment | Pressure tactic to avoid 3-year price lock | "Our preference is a 3-year commitment. We'd like to understand what a 3-year commitment unlocks." |
| Upsells bundle features you didn't ask about | Revenue expansion attempt | "We need to baseline on what we actually use today. Please provide pricing that matches our current footprint." |
| Creates artificial urgency ("price expires in 48 hours") | Quota pressure tactic | "We operate on a [X]-week procurement cycle. We need [X] weeks to complete the business case internally." |
| Proposes annual increases built into multi-year | Common — watch the fine print | "We require a fixed per-core price for the full term. No escalators." |
| Assigns new TAM mid-negotiation | Usually means your TAM escalated | Stay calm; brief the new contact completely; do not restart from zero |

### Your Non-Negotiables

Enter the negotiation knowing exactly what you must have (non-negotiables) versus what you would like (concessions you can give):

**Must have**:
- Per-core price at or below a specific target (calculate from your budget)
- No auto-escalators in a multi-year term
- Written confirmation of what products are included in the bundle
- 30-day written notice cancellation right (for any annual subscription)

**Can give**:
- Commit to 3-year term (if pricing is right)
- Provide a case study or reference call for Broadcom's sales team
- Agree to participate in a beta or early access program

Do not give concessions without receiving something in return. Every concession you make is leverage you have transferred to Broadcom.
