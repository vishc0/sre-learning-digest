# VM Capacity Crisis Playbook: Emergency Response for VMware Cluster Resource Exhaustion

## What This Playbook Covers

This playbook activates when a VMware cluster exhibits any of the following conditions:

| Trigger Condition | Threshold | Why It Matters |
|------------------|-----------|---------------|
| **CPU ready time elevated** | >15% on any host | VMs waiting for CPU scheduling slots; measurable application latency |
| **Memory ballooning active** | Any production VM | Hypervisor reclaiming VM memory; guest OS paging to disk |
| **Datastore utilization critical** | >85% utilized | <15% headroom; snapshot operations, vSAN rebuilds, and thin provisioning expansion at risk |
| **DRS thrashing** | Continuous vMotion across hosts | Cluster cannot find stable placement; workload distribution unstable |
| **VM power-on failures** | HA admission control blocking | HA failover capacity exhausted; new VMs cannot start |

**This is an incident. Activate your incident command structure.** Assign an Incident Commander, a primary investigator (likely you), and a communication owner. Do not triage this as a "capacity ticket" — resource exhaustion on production clusters causes cascading application performance degradation that spreads faster than most people expect.

The analogy: a VMware cluster under resource pressure behaves like a congested RabbitMQ exchange with no prefetch limit. Every queue (VM) tries to consume more than it can get, the broker (hypervisor) starts throttling, and latency spikes propagate through every consumer attached to that exchange. The solution is the same: reduce the load, or increase the capacity — but first, identify which queues are the problem.

## Immediate Triage (First 15 Minutes)

**Goal**: understand scope before taking action. Moving VMs before understanding the picture can make the situation worse (moving a CPU-heavy VM to the one host that still has headroom eliminates your last buffer).

### Step 1.1 — Establish Scope

In vCenter → Hosts and Clusters → select the affected cluster → Monitor tab → Resource Overview.

Answer these five questions before touching anything:

1. How many hosts are in the cluster? How many are in a normal state (not in maintenance mode, not disconnected)?
2. What is cluster-wide CPU utilization (%)? Is it sustained above 80%?
3. What is cluster-wide memory utilization (%)? Is memory balloon/swap active anywhere?
4. Is the resource pressure on all hosts or specific hosts? (Specific hosts suggest HA failover event, not organic growth.)
5. Did anything change in the last 2 hours? (New VM provisioned? Host went into maintenance? Scheduled job started?)

A cluster-wide event looks different from a single-host event. If three of four hosts are healthy and one is at 98% CPU, the cause is almost certainly that host's workload, not cluster capacity. If all four hosts are at 85%+ CPU, you have a genuine cluster-wide saturation event.

### Step 1.2 — Identify the Trigger VMs

vCenter → Cluster → Monitor → Performance → Select "CPU" metric → Sort by "CPU Usage (%)" — descending.

Identify the top 5 CPU consumers and top 5 memory consumers. These are your suspects. Note: a VM consuming 40 vCPUs at 100% is a different situation from 40 VMs each consuming 2 vCPUs at 90% — the former is a single rogue workload; the latter is organic saturation.

Via SSH on the affected ESXi host, you can get real-time CPU ready time per VM using esxtop:

```bash
# SSH to the ESXi host
ssh root@esxi-host.domain.com

# Launch esxtop
esxtop

# In esxtop:
# Press 'c' to switch to CPU view
# The %RDY column shows CPU ready time per VM
# Press 'f' to toggle fields if %RDY is not visible
# Press 'r' to sort by %RDY (descending — worst offenders at top)
# Press 'q' to quit

# %RDY interpretation:
# 0-5%:  Normal — occasional scheduling delay
# 5-10%: Warning — application may notice latency spikes
# 10-20%: Problematic — measurable performance degradation
# >20%:  Critical — significant application impact; escalate immediately
```

### Step 1.3 — Check for Host Failures

If a host recently failed and HA moved its VMs to surviving hosts, the remaining hosts absorbed the failed host's entire workload. This is the most common cause of sudden cluster saturation — it was sized for N+1 redundancy but is now running at N capacity.

```bash
# On ESXi host via SSH: check uptime and power-on duration
esxcli system stats uptime get

# Check host health sensors
esxcli hardware ipmi bmc get

# Check vSAN host status (if vSAN cluster)
esxcli vsan cluster get
```

In vCenter: check vSphere HA → Monitor → HA Summary. If "Hosts with HA errors" > 0, a host event triggered HA failover and you are now running degraded capacity.

### Step 1.4 — Check Storage

vCenter → Storage → Datastores view. Red datastores = >75% utilized (default warning threshold).

```
Datastore utilization thresholds:
  <75%:  Normal
  75-85%: Warning — reduce pace of thin provisioning and snapshots
  85-90%: Alert — take immediate action; snapshot chains will fail silently
  >90%:  Critical — stop all provisioning; snapshot operations may corrupt
```

```bash
# On ESXi host: check filesystem utilization
esxcli storage filesystem list
# Output shows: Volume Name, UUID, Mounted, Free (MB), Size (MB), Type

# For vSAN: check vSAN health from vCenter CLI
# Or SSH to host and run:
esxcli vsan storage list
```

**Storage at >85% is a different emergency from CPU saturation.** A datastore that fills completely can corrupt virtual disk files and destroy VMs. If storage is >85%, this becomes Priority 1 regardless of CPU/memory state. Stop all VM provisioning immediately and escalate.

## Quick Wins (Minutes 15–60)

Only proceed here after Step 1 triage is complete. These actions provide immediate relief without long-term commitment. Each action is reversible.

### Quick Win 1: Manual vMotion to Rebalance

If DRS is on conservative settings or specific hosts are overloaded while others have headroom, manually migrate the top CPU/memory consumers to less-loaded hosts.

In vCenter: right-click the VM → Migrate → Change compute resource only → select the target host.

Target destination criteria: CPU utilization <60%, memory utilization <70%, not already receiving DRS-initiated vMotions.

Verify after migration: esxtop on the source host should show reduced %RDY. The migrated VM should show identical or improved performance on the target host.

**Pitfall**: Do not vMotion a VM that is the source or target of active replication (Zerto, vSphere Replication, vSAN stretched cluster). Check before moving.

### Quick Win 2: Reclaim Zombie VMs

Powered-on VMs with zero application activity hold CPU reservations and consume memory — even at idle, a Windows Server VM with 16GB allocated holds those 16GB of host memory mapped to it.

```powershell
# PowerCLI: find VMs powered on with <1% CPU over last 24 hours
Get-VM | Where-Object {$_.PowerState -eq "PoweredOn"} |
  ForEach-Object {
    $vm = $_
    $stat = Get-Stat -Entity $vm -Stat cpu.usage.average `
      -Start (Get-Date).AddHours(-24) -Finish (Get-Date) -MaxSamples 100
    $avgCpu = ($stat.Value | Measure-Object -Average).Average
    [PSCustomObject]@{
      VM = $vm.Name
      AvgCPU_24h = [math]::Round($avgCpu, 2)
      MemGB = $vm.MemoryGB
      Host = $vm.VMHost.Name
    }
  } |
  Where-Object {$_.AvgCPU_24h -lt 1} |
  Sort-Object MemGB -Descending |
  Export-Csv -Path C:\migration\zombie_vm_candidates.csv -NoTypeInformation
```

**Do not power off any VM without owner confirmation.** Use the inventory from Phase 1 of the migration playbook to identify owners. Send immediate notification: "This VM has had zero CPU activity for 24 hours. We are considering powering it off to relieve cluster pressure. Confirm use or it will be powered off in 4 hours."

The goal is owner-confirmed power-offs within the current incident window. VMs you cannot get owner confirmation on in 4 hours should be moved to your lowest-priority list and addressed after the immediate crisis is resolved.

### Quick Win 3: DRS Aggressiveness Increase

If DRS is configured at Conservative or Partially Automated, it is deliberately leaving imbalances in place to avoid excessive vMotion. During a capacity crisis, you want DRS to be more aggressive.

vCenter → Cluster → Configure → vSphere DRS → Edit:
- Set Automation level: "Fully Automated"
- Set DRS aggressiveness (Migration threshold): Level 4 or 5

Level 4 (Aggressive): DRS vMotions VMs if it will improve balance by >1% (versus the default 5%). Level 5: DRS vMotions VMs for even marginal improvements.

**Document this change and set a calendar reminder to revert to prior settings within 72 hours.** Leaving DRS at Level 5 permanently causes excessive vMotion for marginal gains. This is a crisis setting, not a steady-state setting.

### Quick Win 4: Memory Reservation Audit and Balloon Response

**Memory ballooning** is the hypervisor's way of reclaiming physical memory from VMs without crashing them. The balloon driver (a VMware kernel module inside the guest OS) inflates like a balloon inside the VM's address space, forcing the guest OS to page its least-used pages to virtual disk swap. This causes the VM to behave as though it has less RAM than allocated, which manifests as application latency and increased disk I/O.

To identify which VMs are ballooning:

```bash
# In esxtop: press 'm' for memory view
# Look for the MCTLSZ column (balloon size in MB)
# Any non-zero value = active balloon
# SWPWRT = swap write rate (MB/s) — if non-zero, the VM is paging to disk
# SWPRD = swap read rate (MB/s) — non-zero = active swap reads = significant performance hit
```

For production VMs that are actively ballooning:
- Increase the VM's memory reservation (right-click VM → Edit Settings → Resources → Memory → Reservation). Set reservation to match the VM's average working set.
- This forces the hypervisor to guarantee physical memory for this VM, at the cost of reducing flex capacity for other VMs.

**Important caveat**: increasing reservations for critical VMs may cause other VMs to balloon more aggressively. This is a priority call — protect your Tier 0 services at the cost of lower-tier services.

## Short-Term Mitigation (Hours 1–24)

### Step 3.1 — Right-Size Oversized VMs

Oversized VMs are a capacity antipattern inherited from on-premises culture. When VMs were expensive to provision, teams requested large allocations "just in case." Those allocations persist even when workloads never required them.

A VM with 32 vCPUs running at 5% CPU is not saving any performance headroom — it is holding 32 CPU scheduling slots that other VMs cannot use. The vCPU count is the scheduling unit. Reducing from 32 to 8 vCPUs immediately frees scheduling bandwidth.

```powershell
# PowerCLI: find VMs with high vCPU count but low actual utilization
Get-VM | Where-Object {$_.NumCpu -ge 8} |
  ForEach-Object {
    $vm = $_
    $stat = Get-Stat -Entity $vm -Stat cpu.usage.average `
      -Start (Get-Date).AddDays(-14) -Finish (Get-Date) -MaxSamples 500
    $avgCpu = ($stat.Value | Measure-Object -Average).Average
    $peakCpu = ($stat.Value | Measure-Object -Maximum).Maximum
    [PSCustomObject]@{
      VM = $vm.Name
      vCPUs = $vm.NumCpu
      AvgCPU_14d_pct = [math]::Round($avgCpu, 1)
      PeakCPU_14d_pct = [math]::Round($peakCpu, 1)
      WastedvCPUs = [math]::Max(0, $vm.NumCpu - [math]::Ceiling($vm.NumCpu * $peakCpu / 100) - 2)
    }
  } |
  Where-Object {$_.WastedvCPUs -gt 4} |
  Sort-Object WastedvCPUs -Descending |
  Export-Csv -Path C:\migration\rightsizing_candidates.csv -NoTypeInformation
```

vCPU reduction requires a VM restart (hot-remove of vCPUs is not supported by most guest OSes). For non-critical VMs (Tier A/B), schedule the resize within the incident window. For production VMs (Tier C), right-sizing is a planned maintenance window event, not an emergency action.

### Step 3.2 — HA Admission Control Review

If VMs are failing to power on, HA admission control is the likely gating mechanism. HA admission control reserves capacity to ensure that if a host fails, the surviving hosts can absorb the workload.

vCenter → Cluster → Configure → vSphere HA → Edit → Admission control policy.

Admission control policies:
- **Cluster resource percentage** (default): reserves X% of cluster CPU and memory for HA failover
- **Host failures tolerated**: reserves capacity for N host failures
- **Dedicated failover hosts**: designates specific hosts as HA spare capacity

If admission control is blocking VM power-ons during a genuine emergency, you can temporarily disable it (vCenter → Cluster → Configure → vSphere HA → Edit → Admission control → Do not reserve failover capacity). **Document this explicitly in your incident record**. You have just removed HA protection — if a host fails now, VMs may not have capacity to restart. Set a reminder to re-enable admission control within 4 hours.

### Step 3.3 — Storage Hot Spots

If specific datastores are overloaded while others have capacity, use Storage vMotion to redistribute VMs. This migrates a VM's disk files from one datastore to another without downtime.

In vCenter: right-click VM → Migrate → Change storage → select target datastore.

Note: Storage vMotion during a capacity crisis competes for storage I/O bandwidth. If storage I/O is already saturated (high disk latency on ESXi hosts), Storage vMotion will worsen the situation. Check datastore latency before initiating Storage vMotion.

```bash
# Check datastore latency on ESXi host
esxtop
# Press 'd' for disk view
# Look for DAVG/cmd (average device latency ms)
# >20ms on spinning disk = elevated; >5ms on SSD/NVMe = elevated; act if trending up
```

## Medium-Term Resolution (Days 1–14)

### Option A: Add Capacity (Hardware)

If triage confirms this is genuine workload growth (not zombie VMs or oversizing), adding ESXi hosts is the durable fix. DRS will automatically rebalance workloads to new hosts when they join the cluster.

Process:
1. Purchase and rack hardware (delivery time 2–8 weeks for typical enterprise hardware; check existing spare inventory first)
2. Install ESXi on new hosts
3. Add hosts to vCenter cluster (vCenter → Cluster → Add Hosts)
4. Configure networking (VDS port groups must be extended to new hosts)
5. DRS begins automatic workload placement immediately

If hardware delivery takes more than 48 hours and the cluster is already in crisis, implement Quick Wins and Short-Term Mitigation measures as a bridge.

### Option B: Workload Migration to Cloud or Kubernetes

For Tier A workloads (stateless, containerizable) under CPU/memory pressure on VMware: migrating these workloads to Kubernetes (EKS, AKS, or on-premises K8s cluster) reduces VMware cluster pressure while also progressing your modernization roadmap.

Target candidates: web servers, API gateways, worker/job services, stateless microservices.

This is a 2–4 week effort minimum, not a same-day fix. Identify candidates during the incident; execute the migration as a follow-on project.

### Option C: Storage Expansion (for Datastore Pressure)

For vSAN clusters: add disk groups to existing hosts, or add hosts to expand vSAN capacity. vSAN capacity scales linearly with host additions.

For external SAN/NAS: provision additional LUNs and present to ESXi cluster. In vCenter: add as a new datastore, then use Storage vMotion to distribute VMs.

Minimum storage headroom target: 25% free capacity at all times. This accommodates snapshot operations, thin provisioning growth bursts, and vSAN rebalancing operations.

## Escalation Criteria and Stakeholder Communication

### Escalate Immediately If

- Any Tier 0 or Tier 1 VM is experiencing measurable user-facing performance degradation
- Datastore utilization reaches 90% (5% from critical threshold; snapshot operations may fail silently)
- HA admission control is blocking emergency VM restarts (protection disabled while hosts are unhealthy)
- CPU ready time on production VMs exceeds 20% for more than 10 consecutive minutes
- vSAN health alerts: resync active + node failure = potential data unavailability

### Stakeholder Communication Format

Use this template for all incident communications. Send initial communication within 15 minutes of incident declaration; update every 30 minutes until resolved.

```
Subject: [VMWARE CAPACITY ALERT] [Severity: P1/P2/P3] — [Cluster Name]

IMPACT
Services affected: [List service names or teams affected]
User impact: [None / Degraded performance / Partial outage / Full outage]
Scope: [Number of VMs affected, estimate of users/requests impacted]

CPU ready time: [X]% (threshold: 15%)
Memory ballooning: [Active on Y VMs / Not active]

Hosts in normal state: [N of M]

Status: [Known / Under investigation]
Current hypothesis: [One sentence if investigating]

ACTIONS TAKEN
[HH:MM] — [Action taken and result]
```

## Key esxcli and esxtop Command Reference

### Host Health and System State

```bash
# System uptime
esxcli system stats uptime get

# Hardware platform info
esxcli hardware platform get

# CPU info: socket count, core count, thread count
esxcli hardware cpu global get

# Memory installed on host
esxcli hardware memory get

# IPMI BMC health (hardware health sensors)
esxcli hardware ipmi bmc get

# Check BIOS/hardware event log for hardware errors
esxcli hardware ipmi sel list | tail -20
```

### VM State and Process Management

```bash
# List all running VM processes (name, PID, world ID)
esxcli vm process list

# Kill a stuck or unresponsive VM (use world ID from process list)
esxcli vm process kill --type=soft --world-id=[WORLD_ID]
# Types: soft (graceful shutdown attempt), hard (force quit), force (kernel-level kill)
# Use soft first; escalate to hard if no response in 60 seconds
```

### Storage

```bash
# Filesystem utilization (datastores, scratch, coredump)
esxcli storage filesystem list
# Key fields: Free (MB), Size (MB), calculate utilization = (Size-Free)/Size * 100

# List all storage adapters (HBAs, NICs used for storage)
esxcli storage adapter list

# List LUNs visible to host
esxcli storage core device list | head -40

# vSAN cluster information
esxcli vsan cluster get
esxcli vsan storage list
```

### Network

```bash
# List all network adapters and driver info
esxcli network nic list

# Get statistics for a specific NIC (errors, drops, throughput)
esxcli network nic stats get -n vmnic0

# List active vSwitch/VDS port groups
esxcli network vswitch standard list
esxcli network vswitch dvs vmware list
```

### esxtop Quick Reference

esxtop is the primary real-time performance tool on ESXi. SSH to the host and run `esxtop`.

| Key | View | Critical Columns |
|-----|------|-----------------|
| `c` | CPU | %RDY (CPU ready), %USED (actual), %CSTP (co-stop for SMP VMs) |
| `m` | Memory | MCTLSZ (balloon size MB), SWPWRT/SWPRD (swap write/read MB/s) |
| `d` | Disk | DAVG/cmd (avg device latency ms), KAVG/cmd (kernel latency ms), QAVG (queue depth) |
| `n` | Network | MbTX/s, MbRX/s (throughput), DRPTX/DRPRX (drops — non-zero = problem) |
| `h` | Help | All available keys |
| `q` | Quit | Exit esxtop |

**Critical thresholds**:

| Metric | Warning | Critical | Meaning |
|--------|---------|----------|---------|
| %RDY (CPU ready) | >5% | >15% | VMs waiting for CPU scheduling |
| %CSTP (co-stop) | >3% | >10% | Multi-vCPU VM scheduling contention — reduce vCPU count |
| MCTLSZ (balloon) | >0 MB | >500 MB | Active memory reclaim from VMs |
| DAVG/cmd (disk latency) | >20ms (HDD), >5ms (SSD) | >50ms | Storage path latency — investigate HBA, fabric, array |
| DRPRX/DRPTX (net drops) | >0 | Sustained | Network drops — investigate NIC, VDS config, uplink saturation |

**Capture esxtop batch output for post-incident analysis**:

```bash
# Run esxtop in batch mode: capture all metrics every 5 seconds for 10 minutes
esxtop -b -d 5 -n 120 > /tmp/esxtop_capture_$(date +%Y%m%d_%H%M%S).csv

# Copy file off host for analysis in Excel or Grafana
scp root@esxi-host:/tmp/esxtop_capture_*.csv ./
```

## Post-Incident: Capacity Sizing Review

After the immediate crisis is resolved, do not close the incident without a capacity review. The capacity crisis was a signal — it revealed that your cluster headroom assumptions were wrong, or that workload growth outpaced your planning cycle.

**Post-incident capacity review questions**:

1. What was the trigger? (HA failover, organic growth, rogue workload, scheduled job spike?)
2. What was the headroom before the trigger? (How close were we before the event?)
3. What is the current trajectory? (Plot 90-day CPU/memory utilization trend. When do we hit 85% again?)
4. What is the lead time to add capacity? (Hardware procurement, cloud burst activation)
5. What alerts, if any, fired before the crisis? (If none: fix monitoring. If some: why were they not acted on?)
6. What is the new sizing target? (Most teams target 70% average utilization with 20% headroom for growth and 10% HA failover reserve.)

Document findings in an incident postmortem. Assign concrete action items with owners and dates. The cluster will hit capacity again — the goal is to have a capacity expansion in flight before it becomes a crisis, not to respond to crises repeatedly.
