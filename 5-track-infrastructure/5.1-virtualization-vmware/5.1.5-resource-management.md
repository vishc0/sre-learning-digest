# Chapter 5: Resource Management — CPU Ready, Memory Ballooning, vMotion, HA/DRS

## Why Resource Management Is the Core Skill

In VMware, physical resources (CPU, RAM, storage) are shared across all VMs on a host. The hypervisor manages that sharing. When it does it well, you don't notice. When it does it badly — or when you've overcommitted resources — VMs slow down in ways that are subtle, non-obvious, and hard to diagnose without knowing what to look for.

Resource management failures present as application problems: timeouts, slow response, GC pauses, connection resets. The application team sees symptoms. The infrastructure team sees clean dashboards. The disconnect is because VMware resource contention is invisible to anything running inside a VM. You have to know which VMware metrics to look for before you can diagnose it correctly.

This chapter covers the four mechanisms that matter most for production stability.

---

## CPU Ready Time: The Hidden Latency Tax

**CPU ready time** is the percentage of time a VM is in "ready" state — it has work to execute but cannot, because no physical CPU is available to schedule it on. The ESXi hypervisor is a preemptive scheduler, and when more vCPUs are demanding physical CPU time than physical cores are available, VMs wait.

RabbitMQ analogy: CPU ready time is like a RabbitMQ message that has been acknowledged by the broker (ready to deliver) but the consumer is busy processing another message. The message waits in ready state. High ready time means your consumers (physical CPUs) can't keep up with demand. The queue depth (ready count) rises while throughput stalls.

How it manifests in practice: an application running inside a VM with high CPU ready time will experience latency spikes, timeouts, and GC pauses that have no cause visible inside the VM itself. The VM's internal CPU utilization might show 40%, but wall-clock time is dilated because the VM is stalled waiting for a physical CPU. A 100ms operation takes 140ms. A health check timeout fires. A circuit breaker trips. The root cause is the hypervisor scheduler, not the application.

**Thresholds that matter**:

| CPU Ready % | Status | Action |
|-------------|--------|--------|
| 0–3% | Healthy | None |
| 3–5% | Acceptable | Monitor trend |
| 5–10% | Degraded | Investigate host load, consider vMotion |
| >10% | Critical | Immediate: add capacity or reduce VM density |

These thresholds apply per VM. A host average hides the distribution — one VM at 0% and one at 20% averages to 10%, but the second VM is in crisis.

**Causes**:
- Too many vCPUs assigned to VMs (more vCPUs than physical cores = scheduling contention). This is counterintuitive: assigning more vCPUs to a VM increases its scheduler scheduling burden. A VM with 16 vCPUs requires the hypervisor to find 16 simultaneous physical cores before it can run. On a host with 32 cores, two such VMs consume the entire scheduling domain.
- Host overcommit ratio too aggressive. A 4:1 vCPU-to-pCPU overcommit is manageable if VMs don't all burst simultaneously. An 8:1 overcommit with bursty workloads creates chronic ready time.
- VMs with high CPU bursts running simultaneously. A batch job on one VM and GC on another on the same host at the same time compounds contention.

**Fix sequence**:
1. Reduce vCPU counts on VMs that don't need them — counterintuitive, but effective. A Java app that uses 4 threads doesn't need 16 vCPUs.
2. vMotion high-ready VMs to less-loaded hosts (DRS should do this automatically if configured correctly).
3. Add ESXi hosts to the cluster to increase physical capacity.
4. Review DRS aggressiveness — if ready time is high, DRS level 3 or 4 will rebalance more proactively.

## Memory Ballooning: When the Host Runs Out of RAM

**Memory ballooning** is VMware's first response to host memory pressure. When an ESXi host is running low on physical RAM, VMware installs a small driver inside each VM called the **balloon driver** (vmmemctl). Under memory pressure, vCenter instructs the balloon driver to "inflate" — it allocates memory inside the VM, which forces the VM's OS to activate its own memory pressure response (swap to disk, evict page cache).

Redis analogy: memory ballooning is like Redis `maxmemory-policy` triggering eviction under memory pressure. The VM's OS is forced to treat its own allocated memory like Redis treats evicted keys — they go to a slower tier (disk swap instead of RAM). From outside, the system is still responding. From inside, performance has degraded significantly because the cache is gone and every miss is now a disk read.

**The ballooning sequence**:

```
ESXi host RAM pressure rises
        │
        ▼
vCenter signals balloon driver in VM
        │
        ▼
Balloon driver inflates (consumes VM RAM)
        │
        ▼
VM OS detects RAM pressure
        │
        ▼
VM OS activates swap (disk I/O spikes)
        │
        ▼
Application performance degrades
(looks like a disk I/O or memory leak problem from inside the VM)
```

**Visibility**: you'll see the `balloon` memory counter rising in vCenter performance charts, combined with elevated disk I/O inside the VM. From inside the VM — looking at OS metrics, application metrics — it looks like a disk I/O problem or memory leak. Nothing in the application or OS telemetry points to the hypervisor. This is why knowing to look at vCenter balloon metrics is essential — Splunk dashboards watching OS-level metrics won't catch it.

**VMware's memory reclamation hierarchy** (escalating performance impact):

| Technique | Trigger | Performance Impact | Mechanism |
|-----------|---------|-------------------|-----------|
| **Transparent Page Sharing (TPS)** | Always active | None | Deduplicates identical memory pages across VMs |
| **Ballooning** | Low host memory | Low–Medium | Forces VM OS to manage its own memory pressure |
| **Memory Compression** | Moderate host pressure | Medium | Compresses infrequently accessed pages (CPU-intensive) |
| **Host Swap** | Severe host pressure | High | ESXi swaps VM memory to disk (hypervisor-level, very slow) |

**Memory reservations**: the fix for stateful workloads. A **memory reservation** guarantees physical RAM to a VM — the balloon driver cannot reclaim reserved memory, ever. For Cassandra, Redis, RabbitMQ, and any other JVM workload with large heap, set a reservation equal to the VM's working set. Yes, this reduces overcommit ratio across the host. That is the point — you're buying a guarantee.

```
Without reservation: VM allocated 32GB, host can balloon 20GB away → effective RAM = 12GB + disk swap
With reservation of 24GB: balloon driver cannot touch 24GB → effective RAM never drops below 24GB
```

**Director angle**: memory overcommit works when VMs don't all need their allocated RAM simultaneously. Development environments, lightly loaded VMs, bursty-but-not-simultaneous workloads — these are good candidates for overcommit. Production stateful services are not. Set reservations and document why, so the next team member doesn't remove them to "free up capacity."

## Storage vMotion: Live Disk Migration

**Storage vMotion** migrates a VM's virtual disk files (VMDK files) from one datastore to another while the VM is running. No downtime. No service interruption to the application.

Kafka analogy: storage vMotion is like reassigning a Kafka partition to a different broker while the partition is still active — traffic continues, the data migrates in the background, and the switch happens transparently once the replica is caught up. The source is demoted, the destination is promoted, and consumers never see the gap.

**How it works**:

```
Source Datastore          Target Datastore
┌─────────────┐           ┌─────────────┐
│  VM Disk    │ ──copy──► │  VM Disk    │
│  (active)   │◄──mirror──│  (building) │
└─────────────┘           └─────────────┘
       │                         │
       └─── writes go to both ───┘
                    │
            Background copy completes
                    │
            Final sync + atomic switchover
                    │
            VM now on target datastore
         (<1 second pause at switchover)
```

**Use cases where storage vMotion matters operationally**:
- **Evacuate a failing datastore**: if a storage array is showing errors or a disk group is degraded, proactively vMotion VMs off before it causes data loss — don't wait for the array to fail.
- **Rebalance storage utilization**: if one datastore is at 90% and another is at 40%, vMotion VMs to redistribute. DRS can do this automatically if Storage DRS is configured.
- **Tier migration**: move a VM from older spinning-disk storage to NVMe SSD storage as the workload grows — no maintenance window required.
- **Pre-maintenance**: evacuate VMs from a LUN before you take the storage array offline for firmware updates.

**Storage DRS (SDRS)**: the storage equivalent of DRS. Monitors datastore utilization and I/O latency, and automatically initiates storage vMotion when thresholds are breached. Useful for large environments; requires careful threshold tuning to avoid excessive migrations.

## HA and DRS: The Availability and Balance Engine

### High Availability (HA)

**HA (High Availability)** ensures VMs restart automatically when an ESXi host fails. No manual intervention required. When vCenter detects a host is down (via heartbeat failure — vCenter loses contact and the host stops updating its heartbeat datastore), it instructs surviving hosts to restart the affected VMs.

Kubernetes analogy: HA is the Kubernetes ReplicaSet controller responding to a node failure — pods are rescheduled onto healthy nodes automatically. The key difference: a container restart is typically sub-second (image cached, process starts immediately). A VM restart takes 30 seconds to several minutes — OS boot, service startup, application initialization. For latency-sensitive workloads, this is a meaningful distinction.

**HA configuration knobs that matter**:

**Admission control**: ensures the cluster always reserves enough capacity to absorb one (or more) host failures. vCenter enforces this by refusing to power on new VMs if doing so would consume the reserved failover capacity. If you have 10 hosts at 80% utilization, and admission control requires capacity for 1 host failure, VMs may be refused power-on.

```
Example: 10 hosts, each with 100GB RAM usable
Total cluster RAM: 1000GB
HA reserves 1 host worth of capacity: 100GB
Available for VMs: 900GB
At 80% utilization (800GB used), adding a new 150GB VM → rejected by admission control
```

**VM restart priority**: when a host fails and surviving hosts have limited capacity, HA restarts VMs in priority order. Critical infrastructure (database, authentication, messaging) should be Priority 1. Application services Priority 2. Dev/test or batch workloads Priority 3 or lower. Without this configured, HA restarts VMs in arbitrary order and you get app servers up before databases — guaranteed startup failures.

**HA heartbeat datastores**: vCenter uses shared datastores as a secondary heartbeat mechanism. If network connectivity to a host is lost but the host can still write to the heartbeat datastore, vCenter knows it's a network partition, not a host failure. This prevents "split-brain" false positives where HA would restart VMs that are still running.

### Distributed Resource Scheduler (DRS)

**DRS** continuously monitors CPU and memory utilization across all hosts in a cluster and vMotions VMs to rebalance load. Think of it as the Kubernetes scheduler, but running continuously post-placement rather than only at scheduling time. Kubernetes decides where to place a pod when it's created; DRS revisits placement decisions for all running VMs on an ongoing basis.

**DRS aggressiveness levels**:

| Level | Name | Behavior | Best For |
|-------|------|----------|----------|
| 1 | Conservative | vMotion only when imbalance is severe | Latency-sensitive apps, minimal disruption |
| 2 | — | Moderate threshold | Balanced environments |
| 3 | Moderate (default) | Balances performance vs. migration disruption | Most production environments |
| 4 | — | More aggressive rebalancing | Compute-heavy batch workloads |
| 5 | Aggressive | Constant rebalancing, most vMotions | Pure batch; problematic for latency-sensitive |

For notification platform workloads (RabbitMQ, Cassandra, Redis) — stateful, latency-sensitive — DRS level 2 or 3 is appropriate. Level 5 would trigger frequent vMotions that interrupt established TCP connections and create brief latency spikes.

**VM-Host affinity and anti-affinity rules**: DRS respects rules you configure for VM placement:
- **Affinity**: keep these VMs on the same host (low-latency inter-VM communication)
- **Anti-affinity**: keep these VMs on different hosts (HA spread — don't lose both RabbitMQ nodes to a single host failure)
- **VM-Host affinity**: keep this VM on this host group (licensing, NUMA, hardware-specific requirements)

Anti-affinity rules for clustered services (RabbitMQ pairs, Cassandra nodes) are essential. Without them, DRS might consolidate all cluster members onto one host for efficiency — eliminating your HA entirely.

**Combined HA+DRS behavior after a host failure**:

```
ESXi Host 3 fails
        │
        ▼
HA detects failure (heartbeat timeout)
        │
        ▼
HA restarts VMs from Host 3 on surviving hosts
(respects VM restart priority order)
        │
        ▼
Surviving hosts are now unbalanced
(Host 3's VMs distributed unevenly)
        │
        ▼
DRS evaluates cluster balance
        │
        ▼
DRS issues vMotion recommendations (or executes automatically)
to rebalance load across Hosts 1, 2, 4, 5...
```

**Director angle**: HA+DRS settings are where overcommitment decisions become visible. If you run at 85% utilization across the cluster and haven't configured HA admission control correctly, a host failure creates a cascade — surviving hosts can't absorb all VMs, some fail to restart, and you've negated your HA investment. Set steady-state utilization targets that account for failure scenarios. An 80% utilization ceiling in a 10-host cluster with 1-host failover headroom means your practical target is 72% (80% of 90% usable capacity). Build dashboards to track this — not just current utilization, but utilization relative to HA headroom.
