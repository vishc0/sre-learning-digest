# Chapter 6: Networking in VMware — vSwitches, DVS, NSX-T, Microsegmentation

## Networking Is the Hardest Part of Virtualization

Physical networking assumes static endpoints. A server lives at a MAC address, that MAC is tied to a port on a switch, and the switch knows exactly where to send frames. Virtual networking breaks this assumption: vMotion can relocate a VM to a different physical host in seconds, which means the MAC address just moved to a different physical NIC on a different switch port — and the network needs to adapt instantly without dropping a connection.

VMware's networking stack has evolved to solve this, starting with simple per-host virtual switches and culminating in NSX's full software-defined networking overlay that abstracts the physical network entirely. Understanding where you are on that spectrum — vSS, DVS, or NSX — determines what problems you can solve and what operational capabilities you have.

---

## Standard vSwitch vs Distributed Virtual Switch

**Standard vSwitch (vSS)** is the baseline virtual network switch built into every ESXi host. It operates entirely within one ESXi host — it has no awareness of other hosts in the cluster. You configure port groups, uplinks, and policies independently on each host.

Analogy: standard vSwitch is like configuring a CNI plugin (e.g., flannel) separately on each Kubernetes node without a central controller. Each node has its own network configuration. Policies set on node A don't automatically apply to node B. If a pod moves to node B (or a VM vMotions to a different host), that host needs matching configuration — and keeping configuration consistent across dozens of hosts is manual, error-prone, and operationally expensive.

**Distributed Virtual Switch (DVS/vDS)** is managed centrally from vCenter. A single DVS spans all ESXi hosts in a datacenter. Configure a port group once, and it's automatically available and enforced on every participating host. When a VM vMotions to a different host, it follows its logical port group — no reconfiguration required on the destination.

Analogy: DVS is like Kubernetes NetworkPolicy enforced by a cluster-wide CNI controller — one policy definition, consistently enforced across all nodes regardless of which node a pod lands on.

**Key differences**:

| Feature | Standard vSwitch | Distributed vSwitch |
|---------|-----------------|---------------------|
| Scope | Per ESXi host | Cluster-wide (managed from vCenter) |
| Configuration | Per-host, manual | Centralized, inherited by all hosts |
| vMotion networking | Requires identical config per host | Automatic — VM follows its port group |
| Advanced features | Basic | LACP, NetFlow, port mirroring, traffic shaping, ingress/egress rate limiting |
| Monitoring | Per-host only | Centralized visibility across all hosts |
| Requires | Included with ESXi | vSphere Enterprise Plus license |

In production environments, DVS is the operational standard. Standard vSwitch is used for: management traffic on standalone hosts, ESXi hosts not yet under vCenter management, and bootstrap configurations before vCenter is reachable. Once in production under vCenter, workload traffic belongs on DVS.

## Port Groups, Uplinks, and VLANs

**Port groups** are the logical network segments that VMs attach to. A port group defines: the VLAN it carries, the security policies applied (promiscuous mode, MAC address changes, forged transmits), traffic shaping parameters, and which uplinks carry its traffic.

Mapping to EKS concepts: port groups are like EKS security groups combined with subnets — they define both network segment membership and a baseline policy for a set of endpoints. When you assign a VM to a port group named "Production-App-VLAN100," the VM inherits all the policies on that group, just as an EKS pod assigned to a security group inherits those ingress/egress rules.

**Uplinks** are the connections between the virtual switch and the physical network adapters (pNICs) on the ESXi host. Traffic flows from VM to virtual switch to physical NIC to physical switch. DVS aggregates uplinks from all hosts — from the DVS perspective, all physical NICs on all hosts are uplinks available to the switch.

**Uplink teaming and failover**: a port group can be configured to use multiple physical NICs (active-active for load balancing, or active-standby for failover). Common policies:
- **Route based on originating virtual port ID**: deterministic, no LACP required, most common for general workloads
- **Route based on IP hash (LACP)**: requires physical switch LACP configuration, enables true active-active across multiple pNICs
- **Failover only**: one pNIC active, standby takes over on failure; used for management traffic where consistency matters more than throughput

**VLAN tagging modes**:

| Mode | Name | How It Works | When to Use |
|------|------|-------------|-------------|
| VST | Virtual Switch Tagging | DVS tags/untags frames; VM sees untagged L2 | Standard production workloads |
| VGT | Virtual Guest Tagging | VM handles 802.1Q tagging internally | NVA (network virtual appliances), firewalls-as-VMs |
| EST | External Switch Tagging | Physical switch handles tagging | Specialized setups; rarely needed |

For most deployments: use VST. Configure VLANs at the port group level. VMs see a clean Layer 2 segment and don't need to know about VLAN tagging.

## NSX: Software-Defined Networking for VMware

**NSX** (VMware NSX, formerly NSX-T) creates a complete networking stack in software, independent of physical network topology. Where DVS manages Layer 2 Ethernet switching, NSX manages Layer 2 through Layer 7 — routing, stateful firewalling, load balancing, NAT, VPN, all running in software on ESXi hosts and edge VMs.

The key architectural shift: NSX decouples the logical network from the physical underlay. Physical switches only need to provide IP connectivity between hosts (Layer 3). Everything above that — subnets, routing, security policies — is handled by NSX in an overlay.

**NSX architecture**:

```
┌────────────────────────────────────────────────────┐
│              NSX Manager (Control Plane)            │
│    Policy API + Intent-based config engine          │
│    Distributes state to ESXi hosts and edges        │
└───────────────────────┬────────────────────────────┘
                        │ REST API / gRPC
         ┌──────────────┼──────────────┐
         │              │              │
┌────────▼──────┐ ┌─────▼──────┐ ┌────▼────────────┐
│  ESXi Host 1  │ │ ESXi Host 2│ │  NSX Edge Node  │
│  TEP + DFW    │ │  TEP + DFW │ │ (N-S traffic,   │
│  Local router │ │ Local router│ │  NAT, LB, VPN)  │
└───────────────┘ └────────────┘ └─────────────────┘
        │ GENEVE overlay (UDP 6081) │
┌───────────────────────────────────────────────────┐
│          Physical Network (underlay)               │
│  Only needs routed IP between hosts — that's it    │
└───────────────────────────────────────────────────┘
```

**TEP (Tunnel Endpoint)**: each ESXi host has a dedicated IP interface for TEP. VM traffic is encapsulated in GENEVE tunnels between TEPs. From the physical network's perspective, it sees UDP traffic between host IPs — it doesn't need to know about VM MAC addresses, VLANs, or overlay subnets. This is how NSX achieves network topology independence.

**Distributed Router**: in NSX, routing between overlay segments happens on the ESXi host local to the source VM — not at a centralized router. East-west VM-to-VM traffic in different subnets gets routed at the source host and delivered directly via GENEVE tunnel to the destination host. No centralized routing choke point, no tromboning traffic through an external router.

```
VM-A (192.168.1.10) → VM-B (192.168.2.20)
Without NSX: VM-A → physical switch → physical router → physical switch → VM-B
With NSX:    VM-A → local DR (on ESXi Host 1) → GENEVE tunnel → VM-B (on ESXi Host 3)
```

This is the same principle as Kubernetes kube-proxy with eBPF — routing decisions happen locally at the source, not at a centralized hop.

**NSX Edge Nodes**: handle north-south traffic (between overlay VMs and the external network). They run as VMs themselves, providing: NAT, load balancing (NSX Advanced Load Balancer, formerly Avi), VPN termination, BGP peering with physical routers for route advertisement. Edge Nodes are where the overlay meets the physical world.

## Microsegmentation with NSX

**Microsegmentation** is the application of fine-grained security policies at the individual workload level — not at the network perimeter. Instead of "everything inside the datacenter trusts everything else after passing the perimeter firewall," every communication path between workloads must be explicitly permitted.

Traditional datacenter security model:

```
Internet → [Perimeter Firewall] → Trusted Internal Network
                                    └── VMs trust each other freely
```

NSX microsegmentation model:

```
Every VM-to-VM communication path must be explicitly allowed
         ├── Web tier → App tier on port 8080: ALLOW
         ├── App tier → DB tier on port 5432: ALLOW
         ├── Any VM → Any other VM (no rule match): DENY
         └── Lateral movement after breach: BLOCKED
```

### Distributed Firewall (DFW)

NSX's **Distributed Firewall (DFW)** is the enforcement mechanism. It operates at the virtual NIC level of each VM, inside the ESXi kernel. Rules are evaluated before traffic enters or leaves the VM's vNIC — at the first hardware-adjacent enforcement point possible.

There is no centralized choke point. Policy is defined in NSX Manager, distributed to all ESXi hosts, and enforced at scale. Adding a new VM automatically gets DFW protection — no manual firewall rule change, no VLAN reconfiguration.

Mapping to Kubernetes: DFW is like Kubernetes NetworkPolicy enforced by Cilium. Policy is defined centrally (NSX Manager / Kubernetes API), distributed to enforcement agents (ESXi kernel modules / eBPF programs on nodes), and enforced at the workload boundary — not at a perimeter box that all traffic must pass through.

**DFW policy categories** (evaluated in order):

```
Ethernet Rules (L2) — MAC-based, VLAN-based
        ↓
Emergency Rules — override everything below (incident response)
        ↓
Infrastructure Rules — DNS, NTP, AD, management traffic
        ↓
Environment Rules — segment/zone-level policies
        ↓
Application Rules — workload-specific allow rules
        ↓
Default Rule — allow or deny (default deny = Zero Trust)
```

**A practical microsegmentation policy for a 3-tier app**:

```
# Segment: Web Tier
Rule: Allow HTTPS (443) inbound from load balancer IP
Rule: Allow HTTP (8080) outbound to App Tier security group
Rule: Deny all other inbound and outbound

# Segment: App Tier
Rule: Allow HTTP (8080) inbound from Web Tier security group
Rule: Allow PostgreSQL (5432) outbound to DB Tier security group
Rule: Allow AMQP (5672) outbound to RabbitMQ security group
Rule: Deny all other inbound and outbound

# Segment: DB Tier
Rule: Allow PostgreSQL (5432) inbound from App Tier security group
Rule: Deny all other inbound and outbound
```

**Security groups in NSX** are dynamic — membership is based on VM tags, not static IP addresses. When a new App Tier VM is provisioned and tagged `env:prod | tier:app`, it automatically joins the App Tier security group and inherits all DFW rules targeting that group. No manual firewall update. No change management ticket for a new VM.

This is the operational advantage over traditional VLAN-based segmentation: policy follows the workload, not the IP address or network segment.

### Microsegmentation in Breach Scenarios

Without microsegmentation: one compromised VM can probe and reach all other VMs in the datacenter. Lateral movement is unconstrained.

With NSX DFW microsegmentation:
- Compromised web-tier VM can only reach app-tier VMs on port 8080 (its allowed path)
- Cannot reach the DB tier directly (no rule exists)
- Cannot reach other web-tier VMs (east-west within same tier is blocked unless explicitly allowed)
- Cannot reach management infrastructure (infrastructure rules block it from the web tier)

The blast radius of a compromise is bounded by the DFW policy. This is Zero Trust applied to datacenter east-west traffic.

**Director angle**: NSX microsegmentation is the defensible answer to two interview questions: "How do you implement Zero Trust in your datacenter?" and "How do you prevent lateral movement after a breach?" If your environment has flat VLANs with no microsegmentation, a compromised VM can reach everything. NSX closes that gap without requiring a network redesign — the overlay is independent of the physical VLAN topology. You add microsegmentation without recabling a single switch port.

The conversation in Director/VP interviews is not about NSX mechanics — it's about the security posture decision. "We implemented microsegmentation using NSX DFW, which reduced our blast radius from datacenter-wide to per-workload. We did it without a network redesign because NSX runs in overlay — the physical network didn't change." That's the narrative.
