# VM Troubleshooting Command Reference

Commands for diagnosing VMware ESXi issues via SSH (esxcli, esxtop, vim-cmd) and vSphere CLI. Use these during incident triage when vCenter may not give enough detail.

**Prerequisites**: SSH must be enabled on ESXi host (Enable in vCenter → Host → Configure → Security Profile → Services → SSH). SSH access should require key-based auth in production.

---

## Section 1: Host Health Commands

### System Information and Uptime

```bash
# Show ESXi version and build
esxcli system version get

# Healthy output:
#   Product: VMware ESXi
#   Version: 8.0.2
#   Build: 23305546
#   Update: 2
#   Patch: 0

# Show system uptime
esxcli system stats uptime get
# Returns uptime in microseconds — divide by 1,000,000 for seconds, then by 86400 for days
# Problem indicator: very short uptime (unexpected reboot)

# Show hardware platform info
esxcli hardware platform get
# Shows: server model, BIOS version, serial number
# Use this to confirm you're on the right physical host
```

### CPU Health

```bash
# Show physical CPU count and basic info
esxcli hardware cpu global get

# Output shows:
#   CPU Packages: 2      <- physical sockets
#   CPU Cores: 64        <- total physical cores
#   CPU Threads: 128     <- total logical threads (hyperthreading)
#   Hyperthreading Active: true

# Check CPU power state (should not be throttled in production)
esxcli hardware cpu cpulist get | head -20
# Look for: "State: C0" = active, "State: C6" = deep power save (check if intended)
```

### Memory Health

```bash
# Show total physical memory
esxcli hardware memory get

# Output:
#   Physical Memory: 549755813888 Bytes   <- 512 GB in bytes
# Formula: bytes / 1073741824 = GB

# Problem indicator: reported memory less than spec = failed DIMM (alert, replace)
```

### Overall Host Health (Hardware Sensors)

```bash
# Show hardware sensor status (fans, temps, voltages, power)
esxcli hardware sensor list

# Healthy output: all sensors show "Green" status
# Problem indicators:
#   - Status = "Red" or "Yellow" on CPU/memory/disk sensors
#   - Fan RPM = 0 (fan failure)
#   - Temperature sensors near or above threshold
```

## Section 2: VM Management Commands

### List Running VMs

```bash
# List all VMs and their power state
esxcli vm process list

# Output (running VMs only):
#   World ID: 12345
#   Display Name: production-web-01
#   Config File: /vmfs/volumes/datastore1/production-web-01/production-web-01.vmx
#   VMX Cartel ID: 12340
#   UUID: 50 00 00 xx ...
#   Process ID: 0
#   VMX Version: vmx-19

# Problem indicators:
#   - VM in your list but not in vCenter (orphaned process)
#   - VM with World ID but won't respond to vCenter commands
```

### Force-Kill a Non-Responsive VM

```bash
# Get the World ID from esxcli vm process list first
esxcli vm process kill --type=soft --world-id=12345
# soft = graceful shutdown attempt (like SIGTERM)

esxcli vm process kill --type=hard --world-id=12345
# hard = force kill (like SIGKILL) — use if soft doesn't work after 60 seconds

esxcli vm process kill --type=force --world-id=12345
# force = last resort — may leave VM in inconsistent state, datastore lock may need manual cleanup

# Problem indicator: if force kill doesn't work, the VM process is hung in kernel space
# Next step: check for open file handles, may need ESXi host reboot
```

### VM Power Operations (vim-cmd)

```bash
# List all VMs (including powered off) with their vmid
vim-cmd vmsvc/getallvms

# Output:
#   Vmid   Name              File                          Guest OS    Version
#   1      production-web-01 [datastore1] web-01/web-01.vmx centos7_64Guest vmx-19
#   2      production-db-01  [datastore1] db-01/db-01.vmx   centos8_64Guest vmx-19

# Power on VM by vmid
vim-cmd vmsvc/power.on 1

# Power off VM gracefully (guest shutdown)
vim-cmd vmsvc/power.shutdown 1

# Power off VM hard (pulls power)
vim-cmd vmsvc/power.off 1

# Get VM power state
vim-cmd vmsvc/power.getstate 1

# Get VM detailed info (CPU, memory allocation, disk)
vim-cmd vmsvc/get.summary 1
```

### Check for Datastore Locks (Stuck VM)

```bash
# If a VM won't start and shows "file locked" error:
# Find which host has the lock on the vmdk file
vmkfstools -D /vmfs/volumes/datastore1/vm-name/vm-name.vmdk

# Output shows: lock holder MAC address and world ID
# Cross-reference MAC to identify which ESXi host holds the lock
# If lock holder is a crashed host, you may need to manually unlock

# List open file handles on a datastore (run on the host suspected to hold the lock)
vsish -e get /vmkModules/vmfs3/lockStatsStatus
```

## Section 3: Network Troubleshooting

### List Network Adapters

```bash
# List all physical NICs (pNICs)
esxcli network nic list

# Healthy output:
#   Name    PCI      Driver    Admin Status  Link Status  Speed   Duplex  MAC Address
#   vmnic0  0000:xx  ixgbe     Up            Up           10000   Full    00:50:56:xx
#   vmnic1  0000:xx  ixgbe     Up            Up           10000   Full    00:50:56:xx

# Problem indicators:
#   - Link Status = "Down" (cable issue, switch port issue)
#   - Speed = 1000 instead of 10000 (negotiation failure — check cable/SFP)
#   - Admin Status = "Down" (disabled in config — check vSwitch uplink assignment)
```

### NIC Statistics and Errors

```bash
# Get statistics for a specific NIC
esxcli network nic stats get -n vmnic0

# Output includes:
#   Packets Received: 12345678
#   Packets Sent: 9876543
#   Receive Errors: 0       <- should be 0
#   Transmit Errors: 0      <- should be 0
#   Receive Dropped: 0      <- >0 indicates buffer overflow (ring buffer too small)
#   Transmit Dropped: 0     <- >0 indicates congestion

# Problem indicators:
#   - Receive Errors > 0: physical layer issue (CRC errors, bad cable/SFP)
#   - Dropped > 0 and rising: ring buffer overflow — tuning or congestion issue
```

### Virtual Network Configuration

```bash
# List virtual switches
esxcli network vswitch standard list

# List port groups
esxcli network vswitch standard portgroup list

# List VMkernel adapters (vmk0 = management, others = vMotion/vSAN/iSCSI)
esxcli network ip interface list

# Output:
#   Name  Enabled  MTU   MAC Address       Portset     Portgroup        VLAN
#   vmk0  true     1500  00:50:56:xx       vSwitch0    Management       0
#   vmk1  true     9000  00:50:56:xx       vSwitch1    vMotion          100
#   vmk2  true     9000  00:50:56:xx       vSwitch2    vSAN             200

# Problem indicator:
#   - vmk0 MTU = 9000 but switch is 1500 (jumbo frame mismatch — packets dropped silently)
#   - vMotion adapter missing (vMotion will fail)
#   - Management on VLAN 0 but switch expects tagged (management inaccessible remotely)

# Test network connectivity from ESXi host
vmkping -I vmk0 192.168.1.1          # ping via management adapter
vmkping -d -s 8972 -I vmk1 192.168.2.1  # ping with 9000-byte jumbo frames to test MTU path
```

## Section 4: Storage Commands

### List Datastores

```bash
# List all datastores visible to this host
esxcli storage filesystem list

# Healthy output:
#   Mount Point                     Volume Name  UUID            Mounted  Type  Size (MB)   Free (MB)
#   /vmfs/volumes/datastore1        datastore1   ...             true     VMFS  1048576     524288
#   /vmfs/volumes/datastore-vsan    vsanDatastore...             true     vsan  2097152     1048576

# Problem indicators:
#   - Mounted = false (datastore unmounted — VMs on it are inaccessible)
#   - Free < 15% of Size (storage filling — trigger alert at 85% used)
#   - Type = VMFS but expected vSAN (check vSAN health)
```

### Storage Adapter and Path Health

```bash
# List storage adapters (HBAs, iSCSI, NVMe)
esxcli storage adapter list

# List storage paths (multipath status)
esxcli storage nmp path list

# Healthy output (for a path):
#   Runtime Name: vmhba0:C0:T0:L0
#   State: active       <- should be "active" not "dead" or "standby"
#   Path Selection Policy Path Config: PSP_RR    <- round-robin multipath

# Problem indicator:
#   - State: dead (path failure — check cable, HBA, SAN switch)
#   - All paths in "standby" (no active path = VM disk I/O hanging)
```

### vSAN-Specific Commands

```bash
# Check vSAN cluster health from ESXi host
esxcli vsan health cluster list

# Show vSAN disk groups on this host
esxcli vsan storage list

# Check vSAN performance stats
esxcli vsan debug object list | head -20

# Problem indicators:
#   - Disk groups with "unhealthy" status
#   - Objects showing "degraded" (data protection reduced)
#   - "Absent" component count > 0 (data on a failed/offline host)
```

## Section 5: Performance Diagnostics

### esxtop — Real-Time Performance Monitor

```bash
# Launch esxtop (interactive, SSH to ESXi host)
esxtop

# Key views (press letter to switch):
# c = CPU view (default)
# m = Memory view
# d = Disk (storage) view
# n = Network view
# q = Quit

# In CPU view — critical columns:
# %USED   = percentage of CPU time the VM is actually using
# %RDY    = CPU ready time — VM is waiting for a physical CPU
#           Healthy: <3%  |  Warning: 3-5%  |  Critical: >10%
# %CSTP   = co-stop (only relevant for SMP VMs — multiple vCPUs waiting for each other)

# In Memory view — critical columns:
# MCTLSZ  = balloon size in MB (should be 0 in healthy state)
# SWCUR   = current swap usage in MB (should be 0 — any swap = severe performance impact)
# CACHEUSD = swap cache used (non-zero = under pressure)

# Run esxtop in batch mode (for logging/scripting):
esxtop -b -n 5 -d 2 > /tmp/esxtop_dump.txt
# -b = batch mode, -n 5 = 5 iterations, -d 2 = 2 second interval
# Produces CSV-like output for later analysis
```

### VM Resource Usage Snapshot

```bash
# Get CPU and memory usage for all VMs (via vim-cmd)
vim-cmd vmsvc/get.resourceconfig [vmid]

# Get VM performance counters (requires vmid from vim-cmd vmsvc/getallvms)
vim-cmd vmsvc/get.summary [vmid]
# Look for: memoryOverhead (hypervisor overhead), quickStats.guestMemoryUsage

# Check if a specific VM has CPU affinity set (affinity can cause CPU ready issues)
vim-cmd vmsvc/get.config [vmid] | grep -A3 cpuAffinity
# If cpuAffinity shows specific CPUs pinned: this reduces scheduling flexibility
# Recommendation: clear CPU affinity unless absolutely required for NUMA optimization
```

### Quick Triage Sequence (Incident Mode)

Use this sequence when you first SSH to an ESXi host during a capacity incident:

```bash
# Step 1: Is this the right host?
esxcli system version get && esxcli hardware platform get

# Step 2: How loaded is it? (Run esxtop for 30 seconds)
esxtop -b -n 15 -d 2 > /tmp/quick_triage.txt && tail -5 /tmp/quick_triage.txt

# Step 3: What VMs are running and are any in a bad state?
esxcli vm process list

# Step 4: Any storage problems?
esxcli storage filesystem list
esxcli storage nmp path list | grep -v "active"   # Show non-active paths

# Step 5: Any network link failures?
esxcli network nic list | grep -v "Up.*Up"    # Show NICs that aren't Up/Up

# Step 6: Any hardware sensor alerts?
esxcli hardware sensor list | grep -v Green    # Show non-green sensors
```

**Incident triage rule**: always run Step 1 first. In a multi-host incident, it's common to SSH to the wrong host and spend 10 minutes diagnosing a healthy one. Confirm your target before interpreting results.
