# Datacenter Power Event Response Playbook

**Scope**: Any unplanned loss of utility power, UPS alarm, or generator start event in a datacenter environment.
**Severity**: P1 — all power events are P1 until the load is stable and confirmed on known-good power.
**Owner**: On-call SRE + Facility Manager (co-incident commanders)

---

## Detection and Initial Assessment

**Alerts that fire on a power event**:
- UPS management system: "On Battery" alarm (immediate)
- PDU management system: input voltage drop alert
- Environmental monitoring: power quality alarm
- DCIM platform: generator start notification
- Network management: device reachability drops (first sign of partial power loss in some environments)

**Step 1: Confirm the event scope (2 minutes)**
1. Check UPS management system — is one UPS on battery, or multiple?
2. Check PDU monitoring — which PDUs show input voltage anomaly?
3. Check generator status — did the generator start automatically?
4. Query facility manager or NOC: is this a planned maintenance event that wasn't communicated?

**Step 2: Classify the event (3 minutes)**
- **Utility loss (full)**: All UPS units on battery, generator starting. Action: verify UPS runtime, confirm generator transfer.
- **Utility loss (partial)**: One feed (A or B) lost. Dual-fed equipment continues on surviving feed. Action: identify single-feed equipment at risk.
- **UPS fault (not utility)**: Utility present but UPS alarming. Action: check UPS bypass status, contact vendor TAC.
- **PDU/circuit fault**: Utility and UPS healthy but specific PDU alarming. Action: identify affected circuits and equipment.

---

## UPS Switchover Verification

**Step 3: Confirm UPS is carrying load (5 minutes)**
1. Log into UPS management system (not the physical UPS — use the network management card)
2. Verify: input voltage = 0 (on battery) or input = utility (utility restored)
3. Verify: output voltage = nominal (within ±5% of rated output voltage)
4. Verify: output load % — this tells you how hard the UPS is working
5. Verify: estimated runtime remaining — this is the critical number

**Runtime decision gate**:
```
Estimated runtime > 10 minutes AND generator started?
    YES → Proceed to generator transfer verification (Step 4)
    NO → Initiate priority load shedding immediately (Step 6)
```

**If UPS output is not stable**:
- Check UPS bypass status — automatic static bypass may have engaged
- Contact UPS vendor TAC (Eaton, Vertiv, APC) immediately — this number should be in your contacts, not looked up under pressure
- Do NOT attempt to reset the UPS without vendor guidance — a UPS in bypass with a failed inverter can drop load if bypass is interrupted

---

## Generator Start and Transfer Sequence

**Step 4: Verify generator status (5 minutes)**
1. Check generator monitoring panel or DCIM: generator status should show "Running" within 30 seconds of utility loss
2. Verify generator voltage and frequency are at nominal (120/208V at 60Hz for US standard)
3. Verify fuel level reading — should be 90%+ if maintenance is current; below 50% is an immediate flag
4. Verify ATS (Automatic Transfer Switch) status: should show "Generator" position within 30–60 seconds of generator achieving stable output

**Step 5: Confirm transfer complete**
1. UPS management system should show input voltage restored (now from generator via ATS)
2. UPS should transition from "On Battery" to "On Generator" or "Normal" mode (depending on vendor UI)
3. Verify output load is stable — check for load spikes as equipment completes POST cycles
4. Verify runtime remaining has reset to N/A or "Unlimited" (indicating UPS is no longer discharging)

**If generator fails to start**:
- Manual generator start procedure: send remote hands or facility engineer to the generator room immediately
- Contact generator vendor/maintenance contractor — have this number in the war room before the event, not during
- Begin load shedding immediately (Step 6) — do not wait to see if the generator self-recovers
- Escalate to facility manager for emergency fuel check if generator is running but fuel level unknown

**Transfer time note**: Standard ATS transfer is 10–30 seconds. During this window, the UPS is carrying full load. If the transfer takes longer than 60 seconds, the ATS may be faulted — escalate to facility engineering.

---

## Load Shedding Decision Tree

**Step 6: If UPS runtime is insufficient and generator is not stable (execute within the first 5–8 minutes of the event)**

Load shedding priority (shed lowest priority first):
```
Tier 1 — NEVER shed without explicit authorization:
  - Core network elements (packet gateway, IMS, signaling)
  - E911 / life safety systems
  - Active incident command infrastructure (if the DC is the incident command location)

Tier 2 — Shed only if runtime < 5 minutes remaining:
  - Customer-facing production applications (SLA-protected)
  - Core database clusters (requires coordinated graceful shutdown)
  - Storage systems (must be shut down in defined sequence)

Tier 3 — Shed as needed to protect Tier 1 and 2:
  - Internal applications (ITSM, monitoring, collaboration tools)
  - Development and test environments
  - Batch processing workloads

Tier 4 — Shed immediately when directed:
  - Unused powered-on systems (dev boxes, old test servers still running)
  - CI/CD build infrastructure
  - Non-production workloads of all types
```

**Graceful shutdown sequence for Tier 2 assets**:
1. Notify application teams of impending shutdown (even 2 minutes of notice reduces recovery time)
2. Stop inbound load balancer traffic to affected applications
3. Drain active sessions
4. Initiate OS-level shutdown on VMs (hypervisor-level shutdown, not power cut)
5. Once VMs are down, power off hypervisor hosts
6. Storage systems: flush write cache, then power down (follow vendor-specific procedure)

**Do NOT power-cycle storage directly** — unclean shutdown of storage can cause data corruption. Always follow vendor graceful shutdown procedure.

| Condition | Contact | Method | SLA |
|-----------|---------|--------|-----|
| Generator not starting | Facility Manager | Phone (not Slack) | Immediate |
| UPS runtime < 5 min, load not shed | On-call Director | Phone | Immediate |
| Utility restoration unknown (> 20 min) | Utility provider emergency line | Phone | Within 20 min |
| Generator fuel level < 25% | Fuel supplier emergency line | Phone | Within 30 min |
| ATS fault suspected | Electrical contractor (on-call) | Phone | Within 15 min |
| Colo provider power SLA breach | Colo NOC | Phone + ticket | Within 5 min |
| Customer-facing service impacted | Customer-facing incident commander | Existing incident channel | Per P1 process |

## All-Clear Criteria and Restore Sequence

**All-clear conditions (all must be true)**:
- Stable power source confirmed: either utility restored (ATS transferred back) or generator stable and fuel confirmed
- UPS input voltage stable for 5+ minutes
- All PDU input voltages confirmed nominal
- No active load shedding in progress (or sheds completed without further incidents)
- Facility manager confirms root cause is understood and contained

**Restore sequence**:
Restore in reverse priority order — Tier 4 first, Tier 1 last. Rationale: you want to absorb the startup load surge from low-priority equipment before bringing critical systems back up.
1. Restore Tier 4 systems (power on and confirm stable)
2. Restore Tier 3 systems (power on, confirm application health)
3. Restore Tier 2 systems: storage first, hypervisors second, VMs third, load balancers last
4. Verify Tier 1 systems are confirmed healthy (they should never have gone down)
5. Restore monitoring systems to full coverage
6. Confirm no lingering UPS battery runtime impact (check battery state of charge)

## Post-Event Documentation Requirements

Complete within 24 hours of event resolution:

- [ ] Timeline of events with timestamps (utility loss, generator start, transfer, resolution)
- [ ] Duration of each phase (on battery, on generator)
- [ ] Minimum UPS runtime remaining during the event
- [ ] Any load shedding: what was shed, when, by whom, at what UPS runtime
- [ ] Generator performance: start time, transfer time, fuel level at event start and end
- [ ] ATS performance: transfer time, any anomalies
- [ ] Root cause: utility failure (weather, grid event, utility fault), facility cause (ATS fault, PDU issue, UPS fault)
- [ ] Customer or SLA impact (if any): services affected, duration, recovery time
- [ ] Corrective actions: fuel contract review, generator test schedule, ATS inspection
- [ ] File completed report in the DC incident record system and share with Facility Manager and Director

**Fuel replenishment**: After any generator run of 30+ minutes, initiate a fuel replenishment request immediately — do not wait until the next scheduled fuel delivery.
