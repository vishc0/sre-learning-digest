# Datacenter Cooling Failure Response Playbook

**Scope**: Any unplanned cooling failure — CRAC/CRAH unit failure, containment breach, chiller fault, cooling infrastructure alarm — that risks elevating server inlet temperatures above safe operating thresholds.
**Severity classification**:
- Warning: Inlet temp 80–89°F (27–32°C) — monitor, investigate, prepare mitigation
- Critical: Inlet temp 90–104°F (32–40°C) — active incident, mitigation required immediately
- Emergency: Inlet temp > 104°F (40°C) — equipment at risk, begin controlled shutdown sequence

**Time sensitivity**: Cooling failures are time-critical in a different way than power events. You have a window — typically 15–45 minutes in a high-density environment — before temperatures reach the point where equipment begins self-protecting (throttling, then shutting down). The window is wider in low-density environments. Don't let the slow ramp fool you into treating this as low urgency.

---

## Temperature Threshold Triggers

**Monitoring alert thresholds** (configure in your DCIM/environmental monitoring system):

| Level | Inlet Temp | Response |
|-------|-----------|----------|
| Info | > 75°F / 24°C | Log only, no page |
| Warning | > 80°F / 27°C | Page on-call SRE, investigate |
| Critical | > 90°F / 32°C | P1 incident declared, cooling team engaged |
| Emergency | > 104°F / 40°C | Begin controlled shutdown sequence |

**ASHRAE A1/A2 equipment ratings** (most enterprise servers):
- A1 class: 50–82°F (10–28°C) inlet — older/conservative equipment
- A2 class: 50–95°F (10–35°C) inlet — most modern enterprise servers
- Operating above rated inlet temperature triggers CPU/GPU thermal throttling, shortens component life, and in sustained conditions causes irreversible hardware damage

**Where to pull temperature data**:
1. Environmental sensors: in-row sensors at rack top, middle, and bottom (cold aisle inlet side)
2. Server IPMI/iDRAC/iLO: per-server inlet temperature reported via BMC
3. DCIM platform: facility-wide temperature heatmap with trend

## Initial Triage: Classifying the Failure

**Step 1: Determine scope (5 minutes)**

Query the DCIM and environmental monitoring platform:
1. Is the temperature rise localized (1–2 racks, specific zone) or widespread (entire room, multiple rows)?
2. Are CRAC/CRAH units showing alarms? What type — compressor fault, fan fault, refrigerant pressure, high return air temp?
3. Is the chiller plant (if CRAH-based) showing any alarms?
4. Is there a containment breach — aisle doors open, blanking panels removed, overhead containment damaged?

**Localized temperature rise** → likely a containment breach or a single CRAC/CRAH unit failure in the affected zone. Mitigation can be localized.

**Widespread temperature rise** → likely a major CRAC/CRAH multiple-unit failure or chiller plant issue. Facility-wide response required.

**Step 2: Classify failure type**
- **Unit failure (single CRAC/CRAH)**: N+1 spare should have started automatically. Verify spare is running. If it is, thermal equilibrium may be maintained — monitor closely. If spare did not start, escalate to facility engineering immediately.
- **Multiple unit failure**: N+1 is consumed. Temperature will rise. Initiate emergency mitigation.
- **Chiller plant fault (CRAH system)**: All CRAH units are starved for chilled water. Temperature will rise across the entire facility. This is a facility emergency — page facility engineering and escalate.
- **Containment breach**: Doors open, blanking panels missing, containment panels damaged. Hot air recirculating to cold aisle. Fix the breach first — this may resolve the temperature rise without any equipment intervention.

---

## Immediate Mitigation Actions

**Step 3: Containment breach mitigation (if applicable) (5 minutes)**
1. Identify which aisle doors are open or missing
2. Close aisle doors immediately — verify the door latches properly (many cold aisle containment doors spring open if not latched)
3. Identify and fill any missing blanking panels — keep a stock of blanking panels on-site; if you don't have them, use temporary foam/cardboard as a stopgap (yes, really — the physics work)
4. If overhead containment panels are displaced, alert facility staff to reseat them
5. After breach is addressed, monitor temperature trend — if temperatures are not declining within 5 minutes, the breach was not the primary cause

**Step 4: CRAC/CRAH unit response (10 minutes)**
1. If a CRAC unit is alarming, check whether the N+1 spare unit has started automatically (check unit status panel or DCIM)
2. If the spare has NOT started: attempt manual start from the unit control panel; if it doesn't start, escalate to facility engineering
3. Increase setpoint temperature on remaining units if the vendor recommends it (some CRAC units run more efficiently at a slightly higher setpoint when compensating for missing capacity)
4. Check CRAC drain pan — a unit that trips on condensate overflow needs pan cleared, not just a restart

**Step 5: Portable cooling deployment (if available and temperature is rising)**
1. Location of portable/spot coolers should be documented in the facility emergency kit list
2. Place spot coolers at the open end of the hot zone cold aisle, aimed at server intakes
3. Portable cooling is a stopgap — it buys time, it doesn't solve the problem
4. Each portable cooler typically handles 1–3 tons of cooling; know the sizing and quantity available

## Graceful Shutdown Sequence

**When to initiate**: If temperature is at Critical (90°F+) and mitigation actions have not begun reducing temperature within 10 minutes — or if Emergency threshold (104°F+) is reached at any point.

**Do NOT power-cycle equipment at high temperature** — thermal shock from hot-standby to powered-off can cause additional component stress. Power down cleanly.

**Shutdown sequence by tier (highest risk shed first)**:

```
Priority 1 — Protect data integrity:
  Storage systems MUST be flushed and shut down before any compute
  that writes to them goes offline. Sequence:
    a. Stop all application writes (application-level quiesce)
    b. Flush storage write cache (storage vendor procedure)
    c. Power down storage controllers in orderly sequence

Priority 2 — Compute (by business priority, lowest priority first):
    a. Development/test VMs → graceful OS shutdown
    b. Non-production workloads → graceful OS shutdown
    c. Internal IT systems → graceful OS shutdown
    d. Customer-facing applications → drain sessions, graceful shutdown
    e. Database cluster → follow cluster shutdown procedure (not ad-hoc)

Priority 3 — Hypervisor hosts:
    After all VMs on a host are shut down, power off the hypervisor host

Priority 4 — Network infrastructure:
    Shut down ToR switches only if necessary — maintaining network
    access for management and monitoring is critical during the incident

Priority 5 — Never shut down during thermal event:
    - CRAC/CRAH units themselves (they're the solution, not the problem)
    - UPS systems
    - Environmental monitoring equipment
    - Management network infrastructure
```

## Remote Hands Engagement During Cooling Failure

**What to ask remote hands to do**:
- Check physical state of CRAC/CRAH units (unit LED status, display panel alarms, audible alarms)
- Confirm specific CRAC/CRAH unit is on or off
- Check and clear CRAC condensate drain pan
- Open or close containment aisle doors as directed
- Deploy portable cooling units to specified locations
- Visually confirm blanking panels are in place in specified racks

**What NOT to ask remote hands to do**:
- Attempt to repair or service CRAC/CRAH equipment (refrigerant handling requires certifications)
- Reset a CRAC unit that hasn't been diagnosed — this can mask a fault that will cause repeated failures
- Power off servers (except in genuine emergency with Director authorization) — remote hands should not be making priority triage decisions
- Modify thermostat or setpoint settings on CRAC units without specific guidance from a qualified engineer

## Escalation to Facility Engineering

Escalate when:
- More than one CRAC/CRAH unit has failed
- Chiller plant is alarming
- Manual start of the N+1 spare unit fails
- Temperature is still rising 10+ minutes after mitigation actions started
- Any equipment has hit Emergency threshold (104°F+)

Escalation information to provide:
1. Current temperature readings (highest rack inlet temp, affected zone)
2. Which cooling units are confirmed failed vs. operational
3. Any visible physical issues (leaks, unusual noise from CRAC units, containment damage)
4. What mitigation has already been attempted
5. Estimated time before Emergency threshold is reached if the trend continues

---

## Recovery and Return to Service

**Do not restore power until**:
- Cooling has been confirmed stable (all active CRAC/CRAH units operational, temperatures declining)
- Target temperature reached: inlet temperatures below 75°F / 24°C for at least 10 consecutive minutes
- Root cause of the cooling failure is understood (do not restore to an environment where the same failure could recur immediately)

**Restore sequence**:
1. Storage systems first (power on, verify controller health, verify volume integrity)
2. Hypervisor hosts (power on, verify host health)
3. VMs: restore from lowest-priority to highest-priority (confirm each tier is healthy before restoring next)
4. Application health checks at each tier before proceeding
5. Confirm customer-facing services restored and monitoring is clean

## Post-Event Thermal Audit

Complete within 48 hours of resolution:

- [ ] Root cause confirmed (unit failure type, chiller fault specifics, containment breach cause)
- [ ] Maximum temperatures reached (by zone, by rack) — pull from sensor logs
- [ ] Equipment impacted (any thermal shutdowns? Any SMART errors logged on storage post-event?)
- [ ] Duration of temperature exceedance above each threshold (warning, critical, emergency)
- [ ] CRAC/CRAH unit maintenance status at time of failure (was PM current? Recent repairs?)
- [ ] N+1 spare behavior: did it start automatically? If not, why?
- [ ] Portable cooling inventory: was it available? Where was it? How quickly was it deployed?
- [ ] Blanking panel audit: were there missing blanking panels in the affected zone before the event?
- [ ] Corrective actions: PM schedule, spare unit inspection, blanking panel restock, containment inspection
- [ ] SLA impact: any customer-facing services affected, duration, SLA credit liability
- [ ] Preventive measures: temperature monitoring threshold review, cooling capacity headroom review
